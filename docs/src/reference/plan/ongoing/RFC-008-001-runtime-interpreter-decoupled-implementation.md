# 解释器优先：解耦式 Runtime（DAG+调度）实现计划

> **任务**：先落地独立 Runtime（DAG/调度/错误传播/资源串行化），并让解释器接入形成可运行闭环；后续编译器再把 `@block/@eager/@auto`、`spawn`、`Result/?` 等元信息/约束“喂给”Runtime。  
> **基于 RFC**：RFC-008（Runtime 并发模型与调度器脱耦）、RFC-001（并作模型与错误处理系统）、RFC-009（所有权模型：任务边界是泄漏边界）  
> **日期**：2026-03-11（起草），2026-03-12（更新）  
> **状态**：进行中（计划）

---

## 摘要

- 优先实现“与解释器/编译器脱耦”的 Runtime：只负责任务生命周期、依赖图（DAG）管理、调度策略、错误传播与取消、资源冲突串行化；不理解字节码/AST/类型系统细节。
- 解释器/编译器只负责把“可执行单元（任务）+ 元信息（依赖/资源/策略/Span）”提交给 Runtime。
- 阶段 A 以解释器闭环为主（默认不改变现有语义）；阶段 B 再让编译器把语言语义贯通到 Runtime。

---

## 对外接口/行为变更（本计划将产生）

- 新增（或固化）一套 **Runtime 公共接口**：任务提交/依赖/等待/取消/统计、资源串行化键、错误传播轨迹/可选错误图开关。
- 解释器新增 **运行时配置入口**：选择 Embedded/Standard/Full、workers 数、是否启用 work-stealing（可延后到 CLI，仅保留程序化入口也可）。

---

## 代码落地映射（当前）

- Runtime 核心（DAG/依赖/取消/资源串行化）：`src/backends/runtime/engine.rs`
- Runtime 三层门面（Embedded/Standard/Full）：`src/backends/runtime/facade.rs`
- 解释器运行时配置（RuntimeMode + EvalStrategy）：`src/backends/interpreter/runtime.rs`
- 解释器：任务调度/驱动/force（AsyncValue + drive_dag_until）：`src/backends/interpreter/executor/executor.rs`
- 解释器：在 `Call*` 指令处注入策略（@block/@auto/@eager）与保守资源键（ffi）：`src/backends/interpreter/executor/execute.rs`

> 说明：解释器任务需要 `&mut Interpreter`，因此当前解释器采用 **Host-driven**（单线程驱动）执行就绪任务；Full runtime 的并行执行主要用于未来编译后端与运行时验证（符合 RFC-008 分层与 RFC-009 任务边界约束）。

## 阶段 A：Runtime + 解释器（先闭环，后增强）

### A1. 定义并锁定 Runtime 边界（与解释器/编译器脱耦）

**目标**
- Runtime 仅负责：任务生命周期、DAG、调度策略、错误传播与取消、资源串行化。
- Runtime 不依赖解释器/编译器内部结构；解释器/编译器只提交任务与元信息。

**验收标准**
- [x] Runtime 模块不引用解释器与编译器内部模块（以依赖边界检查为准）。
- [x] Runtime 接口可被“解释器任务”与“未来编译器/AOT 任务”共同使用。

**测试项目**
- [x] 单元测试：用最小“假任务执行器”验证 Runtime 可独立工作（不引入解释器）。

---

### A2. 解释器接入 Embedded 路径（零语义变化）

**目标**
- 不引入并发/惰性求值，先让解释器通过 Runtime 执行最基础的“立即执行”路径，作为安全底座。

**验收标准**
- [x] 现有解释器端到端用例结果与错误信息不发生变化。
- [x] 现有测试集通过。

**测试项目**
- [x] 回归测试：解释器/集成测试全量通过。
- [x] 冒烟测试：最小程序（无并发语义）输出一致。

---

### A3. Runtime：Standard Runtime（单线程 DAG 调度 + 依赖等待）

**目标**
- 实现单线程调度：构建/维护 DAG，按依赖驱动推进；孤岛 DAG 不阻塞主链；支持环依赖检测与错误返回。

**验收标准**
- [x] 线性依赖/菱形依赖/孤岛 DAG 的执行顺序与 RFC-001 描述一致。
- [x] 环依赖不挂死，能返回明确错误（包含依赖链摘要，若可则带 Span）。

**测试项目**
- [x] 单元测试：线性依赖、菱形依赖、孤岛 DAG、环依赖（2 节点环、自环）。
- [x] 单元测试：取消与清理（取消后不再推进、可验证资源释放）。

---

### A4. Runtime：Full Runtime（多线程调度 + 可选 work-stealing）

**目标**
- 增加 workers>1 的并行调度；work-stealing 作为可选增强，禁用时也能正常运行（RFC-008 核心要求）。

**验收标准**
- [x] workers=1 与 Standard 行为一致（结果与错误一致）。
- [x] workers>1 时可观测到并行执行（不以性能为验收，只以可观测并行度为验收）。
- [x] work-stealing 开/关不影响正确性。

**测试项目**
- [x] 单元/集成测试：同一任务图在 workers=1 与 workers>1 下结果一致。
- [x] 单元测试：work-stealing 开/关矩阵下无死锁、无任务丢失。

---

### A5. 解释器：惰性值与“按需 force”（支撑 @auto/@eager/@block）

**目标**
- 支持“先生成可延迟求值占位值，需用到值时再触发求值”。
- 运行时配置三种策略：
  - 默认：保持当前同步行为（等价于全局 @block）。
  - 可选：全局 @auto（最大化延迟与并发）/全局 @eager（更偏急切求值）。

**验收标准**
- [x] @auto：调用返回 `Async` 占位值；当值被消费时通过 Runtime 驱动至目标完成并取值。
- [x] @block：默认行为与现状一致（不走 DAG；不产生 `Async` 占位值）。
- [x] @eager：调用点立即驱动并取值（不留下 `Async` 占位值）。

**测试项目**
- [x] 单元测试：`@auto` 下 lazy call 被消费时触发执行并取值。
- [x] 单元测试：`@eager` 下不留下 `Async` 占位值。
- [x] 回归测试：未启用新策略时，现有测试全量通过。

---

### A6. Runtime + 解释器：任务失败的传播、合并与取消联动

**目标**
- 任务失败沿依赖边传播：依赖者失败或被取消；多任务同时失败时输出主错误与附带错误摘要。

**验收标准**
- [x] 依赖失败时，上游不会永久等待；错误可到达顶层（取消原因包含依赖错误摘要）。
- [ ] 当上游不再需要某分支结果时，孤岛任务可被取消/停止推进（至少不阻塞主流程）。

**测试项目**
- [x] 单元测试：叶子失败导致上游失败/取消；错误信息包含根因摘要。
- [ ] 单元/集成测试：并发多失败时输出含主错误与其他摘要。
- [ ] 单元测试：取消后不再继续推进被取消任务。

---

### A7. 资源冲突的最小闭环（先保守，后精细）

**目标**
- 解释器先用“保守资源键”为副作用操作打标（例如统一为同一类资源），Runtime 强制串行化确保正确性；后续编译器提供更精细资源键提升并行度（RFC-001 资源类型体系）。

**验收标准**
- [x] workers>1 下同资源键操作严格串行（顺序稳定可观测）。
- [x] 不同资源键操作允许并行（至少可观测并发推进）。

**测试项目**
- [x] 单元测试：Full runtime 下同资源键任务不会并发启动。
- [x] 单元测试：workers>1 下无资源/不同资源键任务可并发启动。

---

### A8. 无限循环与公平性（切片/让出）

**目标**
- 落地 RFC-001：单循环无调度开销、多循环自动切片；避免饥饿；仅单任务时不引入可观测开销。

**验收标准**
- [ ] 多任务场景下不会因某个长循环导致其他任务永远不运行。
- [ ] 单任务场景行为不变（结果与错误一致）。

**测试项目**
- [ ] 集成测试：两个长循环任务都能在有限步数内推进（计数器/同步点验收）。
- [ ] 回归测试：单循环程序与现状一致。

---

## 阶段 B：编译器接入（后续把语言语义“喂给”Runtime）

### B1. 语法与约束：`@block/@eager/@auto` 与 `spawn`（仅 @block 内允许）

**目标**
- 编译器提供函数/块级策略元信息；强制 `spawn` 仅在 `@block` 作用域内可用（RFC-001/008）。

**验收标准**
- [ ] 非法 `spawn` 位置在编译期报错且定位明确。
- [ ] 注解策略可稳定传递到运行时配置/元信息。

**测试项目**
- [ ] 解析/类型检查测试：合法/非法 `spawn` 场景矩阵。
- [ ] 端到端测试：注解影响运行时策略选择（可观测差异 + 结果一致）。

---

### B2. `Result[T,E]` 与 `?` 的语言层闭环（与 Runtime 错误传播对齐）

**目标**
- `Result` 真正携带 `T/E`，`?` 在编译期保证规则正确；运行时按 RFC-001 透明传播；与并发依赖错误传播保持一致。

**验收标准**
- [ ] 非 `Result` 使用 `?` 或错误类型不匹配时编译期报错。
- [ ] 并发场景下 `?` 的短路与资源串行化规则不冲突（可解释且可测试）。

**测试项目**
- [ ] 类型检查测试：`Result/?` 正确/错误用例。
- [ ] 端到端测试：并发任务失败触发 `?` 传播到顶层。

---

### B3. 错误图（可选）与内存上限（函数作用域内）

**目标**
- 实现错误传播路径可视化；错误图仅在单函数/块作用域内，带上限与开关。

**验收标准**
- [ ] 开启时输出稳定可比较的错误图摘要；关闭时无额外开销与噪音。
- [ ] 图规模受控，不随程序规模无界增长。

**测试项目**
- [ ] 快照测试：固定源码生成固定错误图摘要（文本/JSON）。
- [ ] 压力测试：大量节点时仍能在上限内输出并稳定结束。

---

## 默认假设/选择（可调整）

- 阶段 A 默认不改变现有用户语义：解释器默认等价于全局 `@block`（Embedded 路径），通过显式配置开启 Standard/Full 与 `@auto/@eager`。
- 资源系统在阶段 A 先保守保证正确性，阶段 B 再由编译器提供精细资源键提升并行度。
