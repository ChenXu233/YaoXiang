# RFC-027：编译期谓词与统一静态验证

> **参考**:
>
> - [RFC-009: 所有权模型](../accepted/009-ownership-model.md)
> - [RFC-010: 统一类型语法 - name: type = value 模型](../accepted/010-unified-type-syntax.md)
> - [RFC-011: 泛型系统设计](../accepted/011-generic-type-system.md)
> - [RFC-024：基于 spawn 块的并发模型](../accepted/024-concurrency-model.md)
>
> **取代**：[RFC-022: 霍尔逻辑静态验证支持（规约注释与规约类型）](../deprecated/022-hoare-logic-static-verification.md) — 已废弃

## 摘要

本文提出为 YaoXiang 引入**编译期谓词**作为一等公民，将所有编译期静态验证统一为一条 **Bool 求值管道**。编译期谓词不是外挂的规约注释——它就是函数。一个返回 Bool 的函数，可在类型位置使用，编译器在编译期调用它并检查返回值。类型即命题，编译期求值即证明。SMT 求解器作为编译期求值器的后端组件内置，程序员只关心"编译器能否证明条件成立"，无需学习独立的规约语言。

**核心论点**：类型检查在编译期唯一的工作就是布尔求值。类型等式、令牌冲突、依赖类型归约、编译期谓词求值、霍尔逻辑蕴含——全部是编译期 Bool 求值，共享同一条管道。

## 动机

### 为什么废弃 RFC-022？

RFC-022 将规约设计为 `//!` 注释形式：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← 这是独立于类型的注释
    //! ensures: ExistsMax(result, arr[0..n])   ← 这是独立于类型的注释
}
```

这犯了 Curry-Howard 同构的根本性错误：**将规约和类型分裂为两层**。注释不是类型。注释不参与类型检查。注释是"外部工具"的心智模型。

白皮书说得很清楚：

> "没有 `//!` 注释。没有独立的规约语言。一切都在类型系统内。"

### 当前的问题

- RFC-022 的 `//!` 注释是独立于类型系统的外挂语法
- 规约类型和普通类型是两套体系，造成概念冗余
- Debug Build 验证 / Release Build 忽略的分裂模式破坏了统一性
- SMT 求解器被定位为外部工具而非编译器标准组件
- 类型检查、借用验证、编译期谓词检查、宏展开各自走不同的路径

### 正确的心智模型

类型检查可以抽象为一个纯函数：

```
isWellTyped : Program → Bool
```

所有编译期检查——简单类型匹配、借用冲突检测、编译期谓词验证——都是这个函数的子任务。它们共享同一条求值管道，区别仅在于表达式复杂度和求值策略。

## 提案

### 1. `{}` 是求解器的验证空间：类型即断言，验证即类型检查

YaoXiang 的 `{}` 是求解器的验证空间。里面的一切都是断言，求解器保证每一项为 True。

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  求解器保证 x 是 Float，y 是 Float

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  求解器保证 data 是 Array(T)
```

**泛型是编译期谓词的特例。**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          参数在签名处          {} 内只有断言
#          求解器在编译期调用时验证 x > 0

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      参数在签名处           求解器验证 type_of(T) == Type，type_of(data) == Array(T)
```

同一个模式：`name: (params) -> Type = { 断言 }`。求解器不区分"类型断言"和"值断言"。

**循环不变式不需要单独写。变量上的类型标注就是 Floyd-Hoare 不变式。**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0   # 求解器验证：0 == sum(arr[0..0])  → True
    mut i: UpTo(arr.len) = 0     # 求解器验证：0 <= 0 <= arr.len   → True
    while i < arr.len {
        s += arr[i]  # 求解器验证：s_new == sum(arr[0..i+1])
        i += 1       # 求解器验证：i+1 <= arr.len 保持
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

编译器对循环体生成一次验证条件——归纳假设（类型标注）→ 赋值操作 → 新值是否满足类型标注。SMT 求解器证明归纳步成立后，所有迭代自动覆盖。不需要 `: decreases`，不需要 `: Invariant`，不需要归纳证明——编译器把归纳分解为每个赋值的局部 VC。

### 2. 前置/后置条件：参数类型和返回类型上的编译期谓词

放弃 RFC-022 的 `//! requires`/`//! ensures`。编译期谓词作为参数或返回的类型标注：

```yaoxiang
# 前置条件：参数类型包含编译期谓词
divide: (a: Int, b: Positive) -> Int = a / b
#                       ^^^^^^^^  求解器在调用点验证 { b > 0 }

# 后置条件：返回类型是编译期谓词
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

max: (T: Ord) -> ((arr: NonEmpty(Array(T))) -> IsMax(T, arr)) = {
    #                                           ^^^^^^^^^^^^^^
    #  IsMax(T, arr) 部分应用 arr，返回 (result: T) -> Type
    #  求解器在返回点验证 { forall j: result >= arr[j] }
    result = arr[0]
    for i in 1..arr.len {
        if arr[i] > result { result = arr[i] }
    }
    return result
}
```

### 3. 路径条件传播：运行时值的编译期验证

编译期谓词在类型标注位置使用时，参数自动填充为被标注的变量自身。当运行时值进入精化类型参数时，编译器通过路径条件收集和 SMT 蕴含判断完成验证——无需程序员显式传递证明。

#### 3.1 隐式自引用

`Positive: (x: Int) -> Type = { x > 0 }` 是一个编译期谓词构造器。当它出现在绑定位置（参数声明、变量声明、返回类型）时，编译器将绑定的变量自动填入参数：

```yaoxiang
b: Positive
// 编译器展开：b: Positive(b)
// 正格化后：b: { b > 0 }
```

不需要用户写 `b: Positive(b)`——那会变成循环语法。隐式自引用是精化类型在绑定位置的标准展开规则，编译期谓词构造器本身仍是接受 `Int` 返回 `Type` 的纯函数。

返回类型同样适用隐式自引用：

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (T: Ord) -> ((arr: Array(T)) -> Sorted(T)) = { ... }
//                                            ^^^^^^^^^^
//  Sorted(T) 部分应用 T，返回类型为 (arr: Array(T)) -> Type
//  编译器将返回值填入剩余参数 arr → 验证 Sorted(T, return_value)
```

#### 3.2 路径条件收集

当运行时值出现在条件分支中时，编译器自动收集路径条件，形成当前作用域的**假设集合**。这些假设作为编译期 Bool 求值的背景知识参与验证。

```yaoxiang
if y > 0 {
    // 编译器在此分支内自动获得假设：{ y > 0 }
    let result = divide(x, y)
    // 验证条件：(y > 0) ⇒ (y > 0)
    // SMT 求解器判 unsat（蕴含成立）→ 通过
} else {
    // 此分支假设：{ !(y > 0) }
    // 如果要调用 divide(x, y)，验证条件为 !(y > 0) ⇒ y > 0
    // SMT 判 sat（不蕴含）→ 编译错误
}
```

这不是编译器写死特殊模式——这是编译期 Bool 求值管道的自然行为。每次类型检查调用点向 SMT 发送：

```
{背景假设} ⇒ {验证目标}
```

SMT 求解器判断蕴含性。蕴含 → 通过，不蕴含 → 编译错误。背景假设来自当前程序点的路径条件。

#### 3.3 假设栈

编译器在分析控制流时，为每个基本块维护一个假设集合：

- **if-guard**：`if y > 0` → true 分支压入 `y > 0`，false 分支压入 `!(y > 0)`（如果使用了 else）
- **match 模式**：`if let Some(v) = opt` → 分支内压入 `opt == Some(v)`
- **逻辑连接**：`if x > 0 && y < 10` → 分支内压入 `x > 0` 和 `y < 10`
- **函数前置条件**：调用 `divide(a, b)` 时，`b` 必须满足 `Positive` 的证据要么来自当前假设，要么来自实参自身的精化类型标注（`b` 已被标注为 `Positive` 则其类型携带 `b > 0`）
- **赋值**：`let z = y` 时，`y` 上已有的精化条件传递到 `z`

所有假设进入编译期求值管道，发送给 SMT 求解器时翻译为 SMT-LIB 背景断言。

#### 3.4 无静态证据则编译错误

如果程序员直接写：

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

当前程序点没有 `y > 0` 的假设，实参 `y` 自身也没有 `Positive` 类型标注。验证条件为：

```
{} ⇒ { y > 0 }
```

SMT 返回 `sat`（不蕴含）→ 编译错误：

> 无法证明参数 `b` 在 `divide` 调用中满足 `Positive`。
> `y` 来自函数输入，无已证明的界。
> 考虑用 if 分支守卫调用：`if y > 0 { divide(x, y) }`。

YaoXiang 不接受运行时值直接进入精化类型参数而不提供静态证据。这不是限制——这是硬安全哲学的核心。凡编译器无法静态证明的代码，不得通过编译。

#### 3.5 与统一管道的关系

路径条件传播不是额外的机制。它是编译期 Bool 求值管道在控制流分析上的直接延伸：

| 阶段 | 职责 |
|------|------|
| 路径条件收集 | 编译器控制流分析阶段，为每个基本块标注假设集 |
| 验证条件生成 | 遇到需验证的类型约束时，合并路径条件 + 实参类型信息 |
| Bool 求值 | 编译器自身或 SMT 求解器判断蕴含性 |
| 结果 | `True` → 通过；`False` → 编译错误 + 反例；`预算耗尽` → 编译错误 + 位置 |

没有新部件。没有特殊规则。路径条件就是 Bool 求值的背景知识——和类型等式、借用约束共享同一条管道、同一套预算系统。

### 4. 编译期 Bool 求值管道

所有编译期检查共享同一条管道：

```
编译期遇到 Bool 表达式需要求值
        │
        ├── 类型等式（T1 == T2）
        │   → 编译器直接判定
        │
        ├── 令牌冲突条件（!conflicting(tokens)）
        │   → 流敏感活性分析（Dup/Linear 属性追踪）
        │
        ├── 依赖类型归约（n + m 化简）
        │   → 编译期项重写系统
        │
        ├── 编译期谓词（x > 0, forall...）
        │   → 编译器自身 + SMT 求解器
        │
        └── 霍尔逻辑蕴含式（P ⇒ Q）
            → SMT 求解器
                    │
                    ▼
             ┌──────────┐
             │ True     │  → 编译通过
             │ False    │  → 编译错误 + 反例
             │ 超预算   │  → 编译错误 + 谓词位置 + 消耗量
             └──────────┘
```

编译期 Bool 求值的结果只有三种——它们是**同一次求值操作的三个可能结果**：

```
eval_compile_time : BoolExpr → True | False | 预算耗尽
```

这不是设计选择，是停机问题的必然结论。编译器对任意 Bool 表达式求值——Rice 定理说不可判定。编译期求值器也是程序，面对的是同一个停机问题。所以结果必须有三条路：

- `True` → 停机，答案为真
- `False` → 停机，答案为假
- `预算耗尽` → 在给定资源上限内未停机

预算硬限制是停机问题的工程解。不给 knob——给了就是在问用户"你觉得你的程序会不会停机"，用户不知道，编译器也不知道。

**关键设计**：
- **三种结果同源**：`True`/`False`/`预算耗尽` 是一条求值管道的三个出口
- **预算硬限制**：步数、时间、量词实例化深度——编译器内部固定，不给 knob
- **自动策略选择**：编译器按表达式复杂度自动选择求值策略，能自己判就自己判，判不了才调 SMT
- **工程实际**：绝大多数谓词（`x > 0`、`arr.len > 0`、`0 <= idx < arr.len`）是线性算术，SMT 毫秒级返回 True/False。预算耗尽只会出现在用户写了超出可判定片段的复杂谓词时

**管道内的分层依赖**：以上求值器共享同一接口但存在求值顺序。类型等式是所有后续分析的先决条件；所有权/令牌检查依赖类型信息；精化谓词验证依赖前两层的结果。编译器按层求值，底层失败的表达式不进入上层——避免在类型错误的程序上浪费 SMT 求解预算。

```
求值顺序（同一管道，分层调度）
├── 第 0 层：类型等式（T1 == T2）
│   └── 结构合一 → 失败则后续无意义，直接返回 False
├── 第 1 层：所有权/令牌冲突
│   └── 流敏感活性分析 → 失败则内存安全不成立，直接返回 False
└── 第 2 层：精化谓词/霍尔蕴含
    └── 编译器 + SMT → False（反例）或 预算耗尽
```

每一层仍返回 `True/False/预算耗尽`，共享同一接口和同一预算系统。

### 5. 三层函数统一

| 层级 | 运行时机 | 输入 | 输出 | 示例 |
|------|----------|------|------|------|
| 值级函数 | 运行时 | 值 | 值 | `add: (a: Int, b: Int) -> Int = a + b` |
| 类型构造器 | 编译期 | 类型/值 | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| 编译期谓词 | 编译期 | 值 | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

全部使用相同的 `name: type = value` 语法。编译期谓词和类型构造器走同一条编译期求值管道——`{}` 是求解器的验证空间。

### 6. 循环：Floyd-Hoare 验证条件生成

循环不需要单独的 `: Invariant(...)` 或 `: decreases(...)` 标注。变量上的编译期谓词类型标注定义了 Floyd-Hoare 风格的断言——编译器从类型标注生成验证条件，SMT 求解器检查每个赋值是否保持类型。

核心机制：每个赋值操作对应一个 Hoare 三元组 `{P} x := e {Q}`，验证条件是 `P ⇒ Q[e/x]`。编译器对循环体生成一次验证条件——SMT 求解器证明归纳步成立后，所有迭代自动覆盖。

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0   # 验证：0 == sum(arr[0..0])  → True
    mut i: UpTo(arr.len) = 0     # 验证：0 <= 0 <= arr.len   → True
    while i < arr.len {
        # 编译器对循环体生成一次 VC：
        #
        # s += arr[i]：
        #   前提（归纳假设）：s == sum(arr[0..i])
        #   验证义务: s_new == sum(arr[0..i+1])
        #   代入 s_new = s_old + arr[i]：
        #     s_old + arr[i] == sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   SMT 求解器：线性算术，毫秒级 → True
        #
        # i += 1：
        #   前提：i < arr.len（循环条件）+ i >= 0（类型界）
        #   验证义务：i+1 >= 0 且 i+1 <= arr.len
        #   SMT：从前提直接推导 → True
        s += arr[i]
        i += 1
    }
    return s  # 此时 s: SumUpTo(arr, arr.len)，即 s == sum(arr[0..arr.len])
}
```

循环不变式就是变量上的类型标注——程序员写类型，编译器检查归纳步。编译器不需要"发现"不变式，也不需要"自动做归纳"——它把归纳证明分解为每个赋值操作的局部验证条件，交给 SMT 求解器分而治之。

### 7. 终止检查

编译期全自动。编译器能证明的循环通过，不能证明的直接报编译错误——程序员必须让编译器能自动分析循环的终止性。不给半自动标注留口子。

#### 6.1 设计原则

编译器从两处自动提取终止证明所需的信息：

1. **变量类型标注**：精化类型中的边界约束（如 `UpTo(n)` 给出上界 `n` 和下界 `0`）
2. **循环体操作**：每次迭代对变量施加的操作

编译器按优先级尝试四种度量合成策略，找到一个即停止。

#### 6.2 策略 1：线性秩函数自动合成

当变量有线性界标注时，编译器枚举候选线性度量并 SMT 验证。

```
输入：
  变量 v₁: UpTo(u₁), v₂: UpTo(u₂), ...（有上下界的变量）
  循环条件 cond
  循环体中的赋值集合

算法：
  1. 从类型标注提取每个变量的界：[low_i, high_i]
  2. 枚举候选度量：v_i, u_i - v_i, v_i - v_j, 等线性组合
  3. 对每个候选度量 m：
     - SMT 验证 m ≥ 0（从类型界推导）
     - 对循环体的每条执行路径，SMT 验证 m' < m（严格递减）
  4. 找到符合条件的线性组合 → 终止得证
```

覆盖范围：任意变量被赋值为线性表达式（`v = a·v + b`）且有界类型标注的循环。包括 `i += const`、`i -= const`、以及二分查找式的区间收缩：

```yaoxiang
# 二分查找：low = mid + 1 或 high = mid
# 度量 high - low 在两条路径上都严格递减
binary_search: (arr: Sorted(Array(Int)), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 策略 2：谓词违反计数——从目标类型自动提取度量

当循环的返回类型是 `forall` 精化谓词且循环体包含对数据的局部修改时，编译器从谓词定义中自动生成度量。

核心洞察：**用户写的规约是编译器推理的素材。** 编译器不需要内置"什么是排序"——它读 `Sorted` 的定义，从定义中自动提取度量。

```
输入：
  目标类型：Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  循环体操作：相邻元素交换

算法：
  1. 解析谓词定义：forall i in range: cond(i, arr)
  2. 自动生成度量：violation_count = |{ i | ¬cond(i, arr) }|
  3. 分析操作对度量的影响：
     - 相邻交换 arr[j], arr[j+1] = arr[j+1], arr[j]
     - 只影响索引 j-1, j, j+1 三对
     - 如果 arr[j] > arr[j+1]（违反谓词），交换后这对满足谓词
     - violation_count 至少减 1
  4. 上界：n·(n-1)/2（最大相邻逆序数），下界：0
  → 终止得证
```

```yaoxiang
sort: (arr: Array(Int)) -> Sorted(arr) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 策略 3：有界递增/递减模式

`v += const`（正常数），变量有上界类型标注 → 度量 `upper_bound - v` 每次递减 `const`，下界 0。这是策略 1 的退化情况，编译器在最前面快速处理。

#### 6.5 策略 4：乘法缩放度量模板

`v *= const`（const > 1），变量有上下界类型标注。编译器内置对数度量模板 `ceil(log_const(upper/v))`，每次乘 const 度量减 1。

```yaoxiang
mut i: Positive = 1
while i < n {
    # 编译器自动推导：度量 ceil(log₂(n/i))，每次乘 2 度量减 1
    i *= 2
}
```

#### 6.6 终止性与正确性分离

终止性证明和正确性证明是独立的：
- **终止性**：以上四种策略自动证明循环在有限步内退出
- **正确性**：循环体是否朝目标类型推进，由 SMT 求解器通过验证条件检查

两者都通过 → 编译通过。终止性得证但正确性失败 → 编译错误 + 反例。终止性无法证明 → 编译错误并指出无法分析的变量或操作。

#### 6.7 递归函数的终止检查

对需要在编译期求值的递归函数，编译器检查参数递减：

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # 编译器分析：n-1 < n → 递减 → 终止
}

# 编译期使用——编译器保证 factorial 在编译期终止
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120，编译期完成
```

| 场景 | 行为 |
|------|------|
| 编译器能分析出递归递减（如 `n-1`） | 编译期求值 |
| 不递减/无法判定递减 | 编译错误 |
| 运行时调用（非类型位置） | 不需要终止检查 |

#### 6.8 硬边界

`i = f(i)` 且 `f` 不可逆、不封闭、不保持任何单调性——数学上不可能自动证明终止。编译错误：

> 此循环无法自动证明终止。循环变量依赖于不可分析的函数 `f`。请使用可被编译器分析的迭代模式。

这不是编译器的失败。凡无法静态证明安全的代码，不得通过编译。

### 8. SMT 求解器：编译期求值器的后端组件

SMT 求解器在传统语言中是外部工具（如 F\* 调用 Z3，Dafny 调用 Z3）。在 YaoXiang 中，它是编译期求值管道的一个后端组件——仅当编译器自身无法直接判定时才被调用。

```
编译期 Bool 表达式
        │
        ├── 编译器可直接判定（简单比较、简单算术、
        │   常量折叠后的平凡公式）
        │   → 直接返回 True/False
        │
        └── 编译器无法直接判定（量词、符号变量）
            → 依赖类型预归约（factorial(5) → 120）
            → 翻译为 SMT-LIB 格式
            → 发送给 Z3/CVC5（带预算限制）
            → 返回值：unsat（成立）│ sat（反例）│ budget_exceeded
```

**求解预算——硬限制，像栈深度一样**：

| 预算维度 | 说明 | 硬上限 |
|----------|------|--------|
| 求解步数 | 项重写/归结步数 | 编译器内部固定值 |
| 时间 | 单谓词求解时间 | 编译器内部固定值 |
| 量词实例化深度 | forall/exists 展开层数 | 编译器内部固定值 |

超预算即编译错误。没有降级，没有运行时检查，没有 silent pass。错误信息精确到源码中的谓词位置，报告消耗了多少预算。

**为什么这实际可行**：工程中 95% 的实际谓词是线性算术——`x > 0`、`arr.len > 0`、`0 <= idx < arr.len`——全在可判定片段内，SMT 求解器对这类问题毫秒级返回。遇到极少数超预算的复杂谓词，用户把谓词拆简单即可。

依赖类型在 SMT 调用前做了一层预归约：`factorial(5)` 直接编译期求值得 `120`，`append([1,2], [3])` 直接求值得 `[1,2,3]`。这些确定性的值计算不消耗 SMT 预算。

程序员不需要知道 SMT 的存在。心智模型是：**编译器能证明就通过，不能就报错**——和类型检查完全一致。

### 9. 编译期谓词组合

编译期谓词是返回 Type 的函数，组合通过函数组合自然实现：

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(T, arr)
}
```

#### 9.4 循环：编译器 VC 生成

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```

## 详细设计

### 语法变化

| 之前 (RFC-022) | 之后 (本 RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | 编译期谓词作为参数类型 `(b: Positive)` |
| `//! ensures: ExistsMax(result, arr)` | 返回类型为编译期谓词 `-> IsMax(T, arr)` |
| `/*! invariant: ... !*/` | 变量上的编译期谓词类型标注——Floyd-Hoare 不变式 |
| `//! decreases: n` | 编译器全自动推导度量函数 |
| 规约是注释 | 规约是类型系统 |

### 语法

**编译期谓词没有新语法。** `{}` 是求解器的验证空间，和现有类型定义语法完全一致。编译期谓词就是返回 Type 的函数——`name: (params) -> Type = { 断言 }`。

```bnf
# 编译期谓词 = 返回 Type 的函数，{} 内为求解器断言
# 使用现有函数/类型语法，无需新增 BNF 规则
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

### 类型系统影响

- **类型宇宙**：编译期谓词位于 Type₂ 层——接受值返回 Type 的函数，与类型构造器层级相同
- **泛型交互**：编译期谓词可带泛型参数，如 `NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **所有权交互**：编译期谓词中的表达式遵守所有权规则，只能读不能写
- **类型推导**：编译期谓词的参数参与 HM 类型推导

### 运行时表示

编译期谓词在运行时**完全擦除**。`Positive: (x: Int) -> Type = { x > 0 }`——参数 `b: Positive` 在运行时的表示就是 `Int`。精化条件 `{ x > 0 }` 在编译期被 SMT 求解器验证，验证完成后不留任何运行时痕迹。

- `Positive(5)` → 运行时表示为 `Int(5)`，精化条件 `{ 5 > 0 }` 已通过，擦除
- `SumUpTo(arr, 0)` → 运行时表示为 `Int(0)`，等式 `0 == sum(arr[0..0])` 已通过，擦除
- 将编译期谓词放在类型位置（如 `f(x: Positive)`）不产生任何包装类型、不分配额外内存、不插入运行时检查

泛型擦除与精化类型擦除是同一个原则：两者都是编译期函数，在编译期求值后完全消失。Release Build 中编译期谓词零运行时开销——这是"谓词是函数"的直接推论。

**与 `ref` 的交互约束**：编译期谓词只能引用不可变借用或所有权已转移的值。引用可变借用的编译期谓词，编译器无法在编译期保证验证结果在运行时仍然成立——此类用法直接报编译错误。

### 编译器改动

1. **解析器**：编译期谓词使用标准函数语法，无需额外解析规则
2. **编译期求值管道**：统一 Bool 返回接口，自动策略选择
3. **SMT 后端**：集成 Z3/CVC5，SMT-LIB 格式翻译
4. **验证条件生成**：WP/SP 演算 + 循环不变式证明义务
5. **错误报告**：反例格式化，关联源码位置

### 向后兼容性

- ✅ 未使用编译期谓词的代码完全不变
- ✅ 编译期谓词在 Release Build 中零运行时开销
- ⚠️ RFC-022 的 `//!` 语法不再支持——但 022 从未被实现，无迁移负担

## 权衡

### 优点

- **Curry-Howard 同构完整兑现**：类型即命题，程序即证明，`name: Proposition = Proof`
- **统一性**：编译期谓词与普通函数使用完全相同的语法，无概念分裂
- **SMT 透明**：程序员不需要知道 SMT 的存在，心智模型与类型检查一致
- **渐进式采用**：可从一个编译期谓词开始，逐步增加覆盖
- **零运行时开销**：所有求值在编译期完成，Release Build 不插入任何运行时断言

### 缺点

- **编译时间**：SMT 求解增加编译时间，但预算硬限制保证上限可控
- **SMT 能力边界**：超越线性算术的复杂谓词可能超预算，用户需拆分谓词
- **学习曲线**：编写有效的编译期谓词需要一定经验
- **实现复杂度**：编译期求值管道的统一需要精心设计

### 风险缓解

- SMT 求解预算硬限制（步数/时间/实例化深度），超预算即编译错误
- 依赖类型预归约：确定性的值计算先吃掉，SMT 只啃非确定部分
- 增量验证：仅验证变更模块
- 清晰的错误信息 + 反例展示 + 预算消耗报告

## 替代方案

| 方案                             | 为什么不选择                           |
| -------------------------------- | -------------------------------------- |
| RFC-022：`//!` 注释式规约        | 规约与类型分裂，违反 Curry-Howard 同构 |
| 独立规约文件（如 CVL）           | 规约与代码分离，增加维护成本           |
| 仅运行时断言                     | 无法静态保证正确性                     |
| 外部证明助手（如 Coq）           | 学习曲线陡峭，与编译器割裂             |
| **本方案：编译期谓词作为一等公民** | ✅                                     |
