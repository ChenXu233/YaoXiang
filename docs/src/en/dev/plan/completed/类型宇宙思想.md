## Introduction: The Type Universe Philosophy — A Mental Model

The design philosophy of YaoXiang is **"Types solve everything"**. To help developers understand the roles of various entities in the type system and their relationships, we propose the **"Type Universe Philosophy"** — a layered mental model. This model categorizes concepts in the language by semantic role, unifying functions, type constructors, interfaces, value-dependent types, and other concepts, allowing developers to view the entire language from a consistent perspective.

**Important Note**: This model is an **externally presented mental model** designed to unify syntax, guide semantic highlighting, and simplify the learning curve. **The actual compiler internals** are based on traditional cumulative universe stratification (`Type₀`, `Type₁`, `Type₂`, …), ensuring logical rigor and avoiding paradoxes. The two are not in conflict — the external model does not change the compiler implementation; it is used solely for understanding and description.

---

## External Mental Model: Type Universe Layers

| Layer   | Role               | Examples                                                                 | Notes |
|---------|--------------------|----------------------------------------------------------------------|-------|
| Type-1 | Values             | `42`, `"hello"`, the `add` function itself (functions are also values) |       |
| Type0  | Meta-type keyword  | `Type` (used to annotate "this is a type")                             |       |
| Type1  | Concrete types     | `Int`, `String`, `Point` (can be used directly to declare variables)   |       |
| Type2  | Functions/type constructors/value-dependent types | `add: (Int,Int)->Int`, `List: (T: Type) -> Type`, `Vec: (n: Int) -> Type` | |
| Type3  | Interfaces/constraints | `Clone: Type = { clone: (Self) -> Self }`                            |       |
| Type4+ | Metaprogramming constructs (reserved) | Macros, code generators, etc.                                   |       |

- **Type-1** contains all runtime values, including functions themselves.
- **Type0** is the special `Type` keyword, indicating the type of types.
- **Type1** contains concrete types — what we typically mean by "data types."
- **Type2** contains all callable entities: regular functions (returning values), type constructors (returning types), and value-dependent types (returning types).
- **Type3** contains interfaces/constraints, describing a set of method signatures.
- **Type4+** is reserved for future metaprogramming features.

This stratification is based entirely on **semantic roles**, not logical orders. It helps developers intuitively understand the purpose of each symbol.

## Internal Implementation: Traditional Cumulative Universe Stratification

The compiler internally uses standard cumulative universe stratification (as in Coq/Agda):

- `Type₀`: Contains concrete types (such as `Int`) and values of those types.
- `Type₁`: Contains the types of `Type₀` (i.e., the types of types), such as `Type₀` itself, type constructors like `List : Type₀ → Type₀`, etc.
- `Type₂`: Contains the types of `Type₁`, and so on.
- Relationship: `Typeₙ : Typeₙ₊₁`, forming an infinitely ascending hierarchy, avoiding the `Type: Type` paradox.

Each type expression is internally assigned a universe level, which is automatically managed during type checking and is completely transparent to developers. There is a natural correspondence between the layers in the external mental model and the internal universe levels:

- Concrete types (Type1) typically correspond to internal `Type₀`.
- Type constructors (Type2) typically correspond to internal `Type₁` or higher (depending on the parameter universe).
- Interfaces (Type3) typically correspond to internal `Type₁` or higher.

But this correspondence is a compiler-internal detail that developers need not concern themselves with.

## Unified Syntax

Regardless of how the external mental model is layered, YaoXiang's syntax remains unified:

```
identifier : type = expression
```

Where `type` is a type expression and `expression` is a value expression. There are no dedicated keywords like `fn`, `struct`, or `trait`.

### Type Constructors vs. Functions/Variables: Core Rules

**This is a key design choice that determines the syntax disambiguation rules:**

| Syntax | Meaning | Rule |
|--------|---------|------|
| **`x: Type = ...`** | Type constructor | `: Type` explicit declaration → forced as type |
| **`f = ...`** | Function or variable | No `: Type` → HM infers as function/variable |

**Why this design?**

The `{ ... }` syntax itself is ambiguous:
- `{ x: Float, y: Float }` could be a **type literal** (record type)
- `{ a = 1 + 1 }` could be a **code block** (executing statements, returning Void)

**Disambiguation rules**:
- **With** `: Type` → force parsing as type constructor, `{ ... }` is a type literal
- **Without** `: Type` → HM actively parses `{ ... }` as a code block, inferring as function type

```yaoxiang
# ✅ Type constructor: has : Type
Point: Type = { x: Float, y: Float }

# ✅ Function: no : Type, HM infers as () -> Void
main = { println("Hello") }

# ❌ Error: no : Type, compiler cannot parse { ... } as a type
Point = { x: Float, y: Float }  // HM infers as function, not type!
```

### Syntax Examples

| Concept | Code Syntax | Description |
|---------|-------------|-------------|
| Variable | `x: Int = 42` | Has type annotation |
| Function | `add: (a: Int, b: Int) -> Int = a + b` | Has complete type |
| Function (HM inference) | `add = (a, b) => a + b` | No `: Type`, HM infers as function |
| Record type | `Point: Type = { x: Float, y: Float }` | **Must have `: Type`** |
| Interface | `Clone: Type = { clone: (Self) -> Self }` | **Must have `: Type`** |
| Type constructor | `List: (T: Type) -> Type = { data: Array[T] }` | **Must have `: Type`** |
| Generic function | `map: (T: Type, R: Type) -> ((list: List[T], f: (T) -> R) -> List[R]) = ...` | **Must have `: Type`** |
| Value-dependent type | `Vec: (n: Int) -> Type = (n) => { data: Array[n] }` | **Must have `: Type`** |
| Constrained function | `clone: (T: Clone) -> ((value: T) -> T) = (value) => value.clone()` | No `: Type`, HM infers as function |

- Function types must include parameter names, e.g., `(a: Int, b: Int) -> Int`.
- Type constructors are also functions; their type is `(T: Type) -> Type`, perfectly aligned.
- Interfaces are also types; their definition form is consistent with record types.
- **AST is unified into Binding nodes**, no distinction between fn/struct/trait keywords.
- **Semantic determination rules**:
  - With `: Type` → type constructor
  - Without `: Type` → HM infers as function/variable

## Generics System

The generics system is built on "types are also values"; all generic concepts are unified as type-level functions or constrained parametric functions.

### Type Constructors

```
List: (T: Type) -> Type = { data: Array[T] }
```

- Accepts a type parameter `T`, returns a record type.
- At application, use square brackets `List[Int]` (syntactic sugar, equivalent to `List(Int)`).

### Generic Functions

```
map: (T: Type, R: Type) -> (
    (list: List[T], f: (x: T) -> R) -> List[R]
    ) = ...
```

- The type is a curried type-level function: first accepts type parameters, then returns a value-level function type.
- At call sites, type inference can fill in `T` and `R` automatically: `map(ints, double)`.

### Type Constraints: Generic Constraints vs. Interface Objects

YaoXiang supports two ways to implement type constraints:

#### Method One: Generic Constraints (Compile-time Polymorphism)

```
Clone: Type = { clone: (Self) -> Self }
clone: (T: Clone) -> ((value: T) -> T) = (value) => value.clone()
```

- The constraint `T: Clone` means `T` must implement the `Clone` interface.
- Compile-time monomorphization, generating separate code for each concrete type.
- Zero runtime overhead (static dispatch).
- Multiple constraints can use intersection types: `(T: Clone & Debug) -> ...`

#### Method Two: Interface Objects (Runtime Polymorphism)

```
clone_obj: (value: Clone) -> Clone = (value) => value.clone()
```

- Parameter type directly writes the interface name `Clone`.
- Runtime dynamic dispatch, calling methods through a virtual table.
- Suitable for heterogeneous collections, runtime polymorphism scenarios.

| Feature | Generic Constraint `(T: Clone)` | Interface Object `value: Clone` |
|---------|----------------------------------|-------------------------------|
| Parameter type | Concrete type `T` | Interface type `Clone` |
| Return type | Concrete type `T` | Interface type `Clone` |
| Code generation | Compile-time monomorphization | Runtime dynamic dispatch |
| Performance | Zero overhead | Indirect call overhead |
| Use cases | High performance, preserve concrete types | Heterogeneous collections, runtime polymorphism |

Both methods are valid; developers can choose based on their needs.

### Value-Dependent Types and Compile-time Constants

In YaoXiang's unified design, **types can depend on any value, including the results of function calls**. This naturally requires function calls to be evaluated at compile time, and the evaluation process must be guaranteed to terminate; otherwise, the type system will fall into an infinite loop.

#### Compile-time Evaluation Rules

- **Functions have only one definition form**, no need for `constexpr` or similar markers:
  ```
  factorial: (n: Int) -> Int = {
      if n <= 1 { return 1 }
      return n * factorial(n - 1)
  }
  ```
- **Type positions force compile-time evaluation**: Any function call appearing in a type expression (such as array size, value-dependent type parameters) requires that the call be computable at compile time and must terminate:
  ```
  matrix: Array[Int, factorial(5)] = ...  # factorial(5) evaluated at compile time
  vec: Vec[factorial(3)] = ...             # Dependent type parameters also evaluated at compile time
  ```

#### Termination Checking Mechanism (Based on Hoare Logic)

To ensure safe compile-time evaluation, YaoXiang introduces **decreases annotations**, seamlessly integrated with RFC-022:

**Termination annotation for recursive functions**:

```
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n              # n strictly decreases with each recursion
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}
```

**Termination annotation for loops**:

```
sum: (arr: Array[Int, n]) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

- The `decreases` expression must be well-founded (e.g., natural numbers).
- The compiler generates termination conditions and invokes an SMT solver to prove them.
- Supports tuple measures (lexicographic descent) and user-defined well-founded types.

#### Synergy Between Compile-time Evaluation and Termination Checking

1. When the type checking phase encounters a function call in a type position, the compiler checks whether the function has a `decreases` annotation.
2. If present, the built-in evaluator computes the function's value while verifying that the decreasing condition holds along the evaluation path.
3. If no `decreases` annotation exists but the function is obviously terminating (e.g., non-recursive), it is evaluated directly; otherwise, an error is reported prompting the addition of a termination annotation.

#### Advantages

- **Safety**: Ensures that compile-time evaluation of dependent types always terminates, avoiding infinite loops in the type system.
- **Uniformity**: Termination checking and partial correctness verification share the same annotation mechanism.
- **Progressive enhancement**: Can transition from runtime checking to fully static proof gradually.

## Type4: Metaprogramming Layer

In the complete picture of the "type universe," the Type4 layer is defined as the **metaprogramming layer for executing YaoXiang code generation constraints at compile time**.

### Core Idea: Compile-time Execution, Inline Results

Type4 leverages YaoXiang's own interpreter to **execute arbitrary YaoXiang code at compile time**:
- Reuses the existing interpreter; no separate "type-level language" needed
- Compile-time evaluation; results directly inlined into the generated binary
- Zero runtime overhead by default — no reflection structures, just ordinary code

```
┌─────────────────────────────────────────┐
│  Compile-time derive                    │
│  ├── Interpreter executes derive function│
│  ├── Reads type metadata                │
│  ├── Writes trait implementations to TraitTable│
│  └── Code generation → binary           │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  Runtime                                │
│  └── Ordinary binary, no reflection, no extra overhead│
└─────────────────────────────────────────┘
```

### Implementation Mechanism

1. **Type Reflection API**: Types carry metadata (field list, method list), exposed through standard library functions
   ```yaoxiang
   // Provided by standard library
   type Type = { fields: () -> Array[Field], ... }
   ```

2. **Compile-time Evaluation**: The interpreter executes function calls at compile time; results are inlined
   ```yaoxiang
   let fields = Point.fields()  // Compile-time evaluation → inlined as constant array
   ```

3. **Constraint Writing**: The evaluation result (generated trait implementation) is written to TraitTable

### Syntax and Examples

#### Deriving Constraints: Generating `Debug` Implementation for Structs

```yaoxiang
Debug: Type = { fmt: (Self, Formatter) -> Result }

// Type reflection API (built into standard library)
type Field = { name: String, type: Type }
type Type = {
    fields: () -> Array[Field],
    implements: (trait: Type) -> Bool,
}

// derive implementation
derive_debug: (T: Type) -> Type = (T) => {
    fields = T.fields()
    for f in fields {
        if !f.type.implements(Debug) {
            return err("Field " + f.name + " does not implement Debug")
        }
    }
    // Generate method implementation (write to TraitTable)
    T.fmt(self, f: Formatter) -> Result = { ... }
}

// Usage
Point: Type = { x: Float, y: Float }
derive_debug(Point)   # Compile-time execution, generates Point's Debug implementation
```

#### Conditional Constraints

```yaoxiang
conditional: (T: Type) -> Type = (T) => {
    if T == Int { return Add[T] }
    else { return Default[T] }
}
```

#### Composite Constraints

```yaoxiang
both: (c1: Type, c2: Type) -> Type = (c1, c2) => c1 & c2
```

### Comparison with Traditional Macros

| Dimension | Traditional Macros (Rust) | YaoXiang Type4 |
|-----------|----------------------------|-----------------|
| Execution timing | Macro expansion phase | Type checking phase |
| Language capability | Text replacement | Arbitrary YaoXiang code |
| Evaluation method | Text generation | Compile-time interpreter evaluation |
| Error catching | After expansion, type checking | Immediate during evaluation |
| Runtime overhead | None | None (results inlined) |
| Composability | Poor | Good (function composition) |

| Dimension | Traditional Macros | YaoXiang Type4 |
|-----------|-------------------|-----------------|
| Operation target | Text/syntax tree | Types/constraints |
| Execution phase | Macro expansion phase | Type checking phase |
| Hygiene | Requires complex mechanisms | Naturally hygienic |
| Readability | Special macro syntax, hard to debug | Consistent with regular functions |
| Compile verification | Type checking after expansion | Directly participates in type checking |

### Why Can the Type System Now Replace Macros?

1. **Types are first-class citizens**: Types can be computed, passed, and introspected like values.
2. **Constraints are data**: Constraints at the Type3 layer can be constructed, composed, and derived.
3. **Introspection mechanism**: The type system can introspect the structure of types.
4. **Type-level functions**: Execute pure functions during type checking, generating new constraints.

This is equivalent to building a **Turing-complete, pure functional, strongly typed metaprogramming language** within the type system itself.

## Semantic Highlighting and LSP Implementation

Based on the role classification in the external mental model, LSP can provide accurate semantic highlighting for code:

| Semantic Role | External Layer | Token Type | Example |
|---------------|----------------|------------|---------|
| Variable/constant | Type-1 | `variable` | `x`, `PI` |
| Function parameter | Type-1 | `parameter` | `a` in `(a: Int)` |
| Regular function | Type2 (return value is not `Type`) | `function` | `add`, `factorial` |
| Type constructor | Type2 (return value is `Type`) | `type` | `List`, `Vec` |
| Concrete type | Type1 | `type` | `Int`, `Point` |
| Interface | Type3 | `interface` | `Clone` |
| Meta-type keyword | Type0 | `keyword` | `Type` |

The determination rules are based on information after type checking: if the symbol's type is a function type `(…) -> …`, check whether the return type is `Type` (or a specialization of it). If yes, it is a type constructor → `type`; otherwise, it is a regular function → `function`.

## Summary

- **External mental model**: The "type universe philosophy" provides developers with a unified, intuitive perspective for understanding various concepts in the language (values, types, functions, interfaces, etc.), guiding syntax design and semantic highlighting.
- **Internal implementation**: The compiler uses traditional cumulative universe stratification, ensuring the logical rigor of the type system and avoiding paradoxes, completely transparent to the external model.
- **Relationship between the two**: The external model is descriptive and does not affect the compiler implementation; the internal implementation is technical and does not expose hierarchy details externally. The two complement each other, together building a YaoXiang type system that is both powerful and easy to use.

Through this separation, YaoXiang achieves:

- **Ultimate unification** of syntax
- **Powerful expressiveness** in generics and dependent types
- **Developer-friendly** semantic hints
- **Theoretically rigorous** underlying foundations

This is the complete presentation of YaoXiang's type universe philosophy — a mental model for understanding, backed by solid traditional type theory foundations.
