# Interpreter-First: Decoupled Runtime (DAG + Scheduling) Implementation Plan

> **Task**: First land an independent Runtime (DAG/scheduling/error propagation/resource serialization), and have the interpreter connect to form a runnable loop; the compiler will later "feed" metadata/constraints like `@block/@eager/@auto`, `spawn`, `Result/?` to the Runtime.
> **Based on RFCs**: RFC-008 (Runtime Concurrency Model and Scheduler Decoupling), RFC-001 (Spawn Model and Error Handling System), RFC-009 (Ownership Model: Task Boundary is the Leakage Boundary)
> **Date**: 2026-03-11 (drafted), 2026-03-12 (updated)
> **Status**: In Progress (planned)

---

## Abstract

- Priority is given to implementing a Runtime "decoupled from the interpreter/compiler": responsible only for task lifecycle, dependency graph (DAG) management, scheduling strategy, error propagation and cancellation, and resource conflict serialization; does not understand bytecode/AST/type system details.
- The interpreter/compiler is only responsible for submitting "executable units (tasks) + metadata (dependencies/resources/strategy/Span)" to the Runtime.
- Phase A focuses on interpreter loop closure (default does not change existing semantics); Phase B will allow the compiler to connect language semantics to the Runtime.

---

## External Interface / Behavior Changes (Generated by This Plan)

- Add (or solidify) a set of **Runtime public interfaces**: task submission/dependency/wait/cancel/statistics, resource serialization keys, error propagation traces/optional error graph toggle.
- Interpreter adds **runtime configuration entry**: select Embedded/Standard/Full, workers count, whether to enable work-stealing (can be deferred to CLI, programmatic entry alone is acceptable).

---

## Code Implementation Mapping (Current)

- Runtime core (DAG/dependency/cancellation/resource serialization): `src/backends/runtime/engine.rs`
- Runtime three-tier facade (Embedded/Standard/Full): `src/backends/runtime/facade.rs`
- Interpreter runtime configuration (RuntimeMode + EvalStrategy): `src/backends/interpreter/runtime.rs`
- Interpreter: task scheduling/driver/force (AsyncValue + drive_dag_until): `src/backends/interpreter/executor/executor.rs`
- Interpreter: inject strategy at `Call*` instructions (`@block/@auto/@eager`) with conservative resource keys (FFI): `src/backends/interpreter/executor/execute.rs`

> Note: Interpreter tasks require `&mut Interpreter`, so the current interpreter adopts **Host-driven** (single-threaded driving) execution of ready tasks; Full runtime's parallel execution is primarily for future compiler backend and runtime verification (aligning with RFC-008 layering and RFC-009 task boundary constraints).

## Phase A: Runtime + Interpreter (Close Loop First, Then Enhance)

### A1. Define and Lock Runtime Boundary (Decoupled from Interpreter/Compiler)

**Goal**
- Runtime is only responsible for: task lifecycle, DAG, scheduling strategy, error propagation and cancellation, resource serialization.
- Runtime does not depend on interpreter/compiler internal structures; interpreter/compiler only submits tasks and metadata.

**Acceptance Criteria**
- [x] Runtime module does not reference interpreter and compiler internal modules (verified by dependency boundary checks).
- [x] Runtime interface can be used by both "interpreter tasks" and "future compiler/AOT tasks".

**Test Items**
- [x] Unit tests: Verify Runtime works independently with a minimal "fake task executor" (without introducing interpreter).

---

### A2. Interpreter Connects Embedded Path (Zero Semantic Changes)

**Goal**
- Do not introduce concurrency/lazy evaluation yet; have the interpreter go through the Runtime's most basic "immediate execution" path as a safe foundation.

**Acceptance Criteria**
- [x] Existing interpreter end-to-end test results and error messages remain unchanged.
- [x] Existing test suite passes.

**Test Items**
- [x] Regression tests: Full interpreter/integration tests pass.
- [x] Smoke tests: Minimal program (no concurrency semantics) outputs consistently.

---

### A3. Runtime: Standard Runtime (Single-threaded DAG Scheduling + Dependency Wait)

**Goal**
- Implement single-threaded scheduling: construct/maintain DAG, drive progress based on dependencies; isolated DAG islands do not block main chain; support cycle dependency detection and error return.

**Acceptance Criteria**
- [x] Execution order of linear dependencies/diamond dependencies/isolated DAGs is consistent with RFC-001 description.
- [x] Cycle dependencies do not deadlock; can return clear error (including dependency chain summary, with Span if possible).

**Test Items**
- [x] Unit tests: Linear dependencies, diamond dependencies, isolated DAGs, cycle dependencies (2-node cycle, self-loop).
- [x] Unit tests: Cancellation and cleanup (no further progress after cancellation, resource release can be verified).

---

### A4. Runtime: Full Runtime (Multi-threaded Scheduling + Optional Work-stealing)

**Goal**
- Add parallel scheduling with workers>1; work-stealing as optional enhancement; still runs correctly when disabled (RFC-008 core requirement).

**Acceptance Criteria**
- [x] workers=1 behavior is consistent with Standard (same results and errors).
- [x] workers>1 can observe parallel execution (not acceptance based on performance, only based on observable parallelism).
- [x] work-stealing on/off does not affect correctness.

**Test Items**
- [x] Unit/integration tests: Same task graph yields consistent results under workers=1 and workers>1.
- [x] Unit tests: No deadlock, no task loss under work-stealing on/off matrix.

---

### A5. Interpreter: Lazy Values and "On-demand Force" (Supporting @auto/@eager/@block)

**Goal**
- Support "first generate deferred-evaluable placeholder values, trigger evaluation when values are needed".
- Runtime configures three strategies:
  - Default: Maintain current synchronous behavior (equivalent to global @block).
  - Optional: Global @auto (maximize deferral and concurrency) / global @eager (more eager evaluation).

**Acceptance Criteria**
- [x] @auto: Call returns `Async` placeholder value; when value is consumed, drive to completion via Runtime and retrieve value.
- [x] @block: Default behavior consistent with current state (does not go through DAG; does not produce `Async` placeholder values).
- [x] @eager: Immediately drive and retrieve value at call site (does not leave `Async` placeholder value).

**Test Items**
- [x] Unit tests: Under @auto, lazy call triggers execution and retrieves value when consumed.
- [x] Unit tests: Under @eager, no `Async` placeholder value is left behind.
- [x] Regression tests: Existing tests fully pass when new strategies are not enabled.

---

### A6. Runtime + Interpreter: Task Failure Propagation, Merging, and Cancellation Linkage

**Goal**
- Task failure propagates along dependency edges: dependents fail or get canceled; when multiple tasks fail simultaneously, output primary error with supplementary error summary.

**Acceptance Criteria**
- [x] When a dependency fails, upstream does not wait forever; error can reach the top level (cancellation reason includes dependency error summary).
- [x] When upstream only waits for a specific target value, Runtime prioritizes driving the dependency closure; isolated tasks do not delay target completion (stop driving; subsequent optional explicit cancellation strategy).

**Test Items**
- [x] Unit tests: Leaf failure causes upstream failure/cancellation; error message includes root cause summary.
- [x] Unit tests: Multiple dependencies failing simultaneously can merge to output primary error with other summaries (cancellation reason aggregation).
- [x] Unit tests: No further progress of canceled tasks after cancellation (including dependency cancellation propagation).
- [x] Unit tests: `drive_until(Some(target))` does not execute unrelated isolated tasks (target chain priority).

---

### A7. Minimal Loop for Resource Conflicts (Conservative First, Then Refine)

**Goal**
- Interpreter first uses "conservative resource keys" to tag side-effect operations (e.g., unified as same resource type), Runtime enforces serialization to ensure correctness; compiler later provides finer-grained resource keys to improve parallelism (RFC-001 resource type system).

**Acceptance Criteria**
- [x] Under workers>1, operations with the same resource key are strictly serialized (order stable and observable).
- [x] Operations with different resource keys allow parallelism (at least observable concurrent progress).

**Test Items**
- [x] Unit tests: Under Full runtime, tasks with same resource key do not start concurrently.
- [x] Unit tests: Under workers>1, tasks with no resource/different resource keys can start concurrently.

---

### A8. Infinite Loops and Fairness (Slicing/Yielding)

**Goal**
- Implement RFC-001: No scheduling overhead for single loops, automatic slicing for multiple loops; avoid starvation; no observable overhead when only one task is running.

**Acceptance Criteria**
- [x] In multi-task scenarios, one long-running loop does not cause other tasks to never run (cooperative slicing: tasks can `yield` to give up execution).
- [x] In single-task scenarios, task can complete without slicing (executor does not `yield` when there is no contention, avoiding observable overhead).

**Test Items**
- [x] Unit tests: Two long-running tasks can both make progress within a finite number of steps (counter/round-robin sequence acceptance).
- [x] Unit tests: Single task can complete in one poll (no slicing scheduling overhead).

## Phase B: Compiler Integration (Later "Feed" Language Semantics to Runtime)

### B1. Syntax and Constraints: `@block/@eager/@auto` and `spawn` (Only @block Inside)

**Goal**
- Compiler provides function/block-level strategy metadata; enforces that `spawn` is only available within `@block` scope (RFC-001/008).

**Acceptance Criteria**
- [x] Illegal `spawn` positions report compile-time errors with clear location.
- [x] Annotation strategy can be stably passed to runtime configuration/metadata.

**Test Items**
- [x] Parse/type check tests: Legal/illegal `spawn` scenario matrix.
- [x] End-to-end tests: Annotations affect runtime strategy selection (observable difference + consistent results).

---

### B2. `Result[T,E]` and `?` Language-Level Loop Closure (Aligned with Runtime Error Propagation)

**Goal**
- `Result` truly carries `T/E`, `?` guarantees correctness at compile time; runtime propagates transparently per RFC-001; consistent with concurrent dependency error propagation.

**Acceptance Criteria**
- [x] Using `?` on non-`Result` or type mismatch reports compile-time error.
- [x] `?` short-circuit and resource serialization rules do not conflict in concurrent scenarios (explainable and testable).

**Test Items**
- [x] Type check tests: Correct/incorrect `Result/?` usage cases.
- [x] End-to-end tests: Concurrent task failure triggers `?` propagation to top level.
- [x] Unit tests: Resource serialization as control edge (ordering only, not error propagation); cancel waiting on control deps to maintain resource order (see corresponding tests in `src/backends/runtime/engine.rs`).

---

### B3. Error Graph (Optional) and Memory Cap (Within Function Scope)

**Goal**
- Implement error propagation path visualization; error graph only within single function/block scope, with cap and toggle.

**Acceptance Criteria**
- [ ] When enabled, output stable and comparable error graph summary; when disabled, no extra overhead or noise.
- [ ] Graph size is bounded, does not grow unbounded with program size.

**Test Items**
- [ ] Snapshot tests: Fixed source code generates fixed error graph summary (text/JSON).
- [ ] Stress tests: Still output within cap and terminate stably with large number of nodes.

---

## Default Assumptions / Choices (Adjustable)

- Phase A defaults to not changing existing user semantics: interpreter default is equivalent to global `@block` (Embedded path), enabling Standard/Full and `@auto/@eager` via explicit configuration.
- Resource system is conservative to ensure correctness in Phase A; compiler provides finer-grained resource keys in Phase B to improve parallelism.
