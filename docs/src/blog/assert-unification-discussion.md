# 一次关于"统一"的彻夜长谈

2026年7月11日深夜到12日凌晨，我们花了六个小时讨论一个问题——YaoXiang的`assert`到底应该是什么。结果出乎预料：我们一直以为在讨论"如何实现assert"，实际上在讨论"类型系统的地基有哪些洞"。

这篇文章不讲最终结论（结论在spec里），讲的是我们怎么一步步发现自己错了，以及每次是怎么被拽回来的。

---

## 为什么会有两个assert

YaoXiang同时在开发两个东西：

**编译期精化类型** `Assert(N > 0)`，写在类型定义里：
```
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    _assert: Assert(N > 0),   // N必须大于0，否则编译失败
}
```
编译期验证泛型参数。N是5就过，N是0就编译报错。零运行时开销。

**运行时断言** `assert(x > 0)`，写在函数体里：
```
x = read_int()
assert(x > 0)   // 检查用户输入
```
值由外部传入，编译期不知道。失败就panic。

大写的Assert，小写的assert。一个编译期，一个运行时。同一个概念，两种形态。

最自然的反应——最省事的做法——是把它们当成两个完全独立的东西：Assert是个精化类型走证明管道，assert是个20行的native函数返回空值。井水不犯河水。

但这个方案一直有种说不清的难受。它们显然有关系，是同一种语义——"我断言这个条件必须成立"——的两面，而让用户记住"这里用大写、那里用小写、这里能编译期检查、那里只能运行时"是认知债。语言设计里，每多一个"看起来相似但要分别记忆"的东西，就是在欠债。

## 我们想统一，然后撞上了一堵墙

统一的理想其实很简洁：

```
assert: (cond: Bool) -> Assert(cond)
```

`assert`返回一个精化类型`Assert(cond)`。编译期能算出cond的值，就折叠成编译检查（擦除）；编译期算不出，就留到运行时查。一个函数，两种命运。签名本身就是连接点。

但立刻有人指出一个"根本矛盾"：

> `Assert(C)`要求C在编译期可求值。`assert(x > 0)`里x是运行时值——编译期算不出`Assert(x > 0)`。按精化类型规则，这必须**编译报错**。可`assert(x > 0)`是完全正常的运行时断言。所以签名为`Assert(cond)`的统一方案不通。必须分开。

这个论点听起来无懈可击。如果精化类型碰见运行时值就报错，那你永远没法把"编译期断言"和"运行时断言"塞进同一个签名。两个东西，两种命运，分开吧。

**但它是假的。** 假在最后那句"必须编译报错"。精化类型的结果不是二值的——不是只有"证明成立"和"报错"。它是三值：Proved（证出来了）、Disproved（证伪了，带反例）、**Unproven（判不了）**。

"判不了"和"证伪了"是完全不同的状态。证伪了就该报错，永无翻身。但判不了——就是因为值运行时才知道所以判不了——恰恰是可以留到运行时去查的。编译期判不了，不是编译期的失败，是这个命题的**天然属性**。

"运行时断言"哪是什么"另一个机制"，它就是编译期判不了的命题，等到了运行时再去检查。**同一个精化类型，两种命运，命运由它能不能被编译期判定这个事实自动决定。**

假冲突一破，统一的路就通了。但我们要先补一个地基上的洞。

## 少了一个类型：Never（假 / 发散 / 不可能）

统一方案要求`assert(false)`归约到一个"不可能有值"的类型——因为"false为真"这个命题没有证明，用它做类型的地方，程序必须发散。

然后我们发现：YaoXiang根本没有这个类型。

有`Void`。但`Void`能返回默认值——调用一个返回`Void`的函数，调完代码继续走。`Void`有一个居留者。在Curry-Howard的对应里，有一个值的类型是"真命题"（有一个证明），不是"假命题"（零个证明）。**`Void`一直是"真⊤"，只是没人这么叫它。**

我们缺的是"假⊥"：一个永不可居留的类型——没有任何值能填进`x: Never = ...`的右边。`assert(false)`、`panic`、无限循环的返回类型都该是它。没有它，`assert(false)`会返回一个合法的默认void值，然后继续执行后面的代码——断言失败变成了正常流程。整个精化类型系统在那一刻逻辑崩塌。

`Never`最终被定为**公理**：它不是从任何东西推导出来的，它是类型系统的原初概念。三条性质内建于内核——零构造子、Never是所有类型的子类型（"从假可推出一切"）、作为函数返回类型时标记该函数发散永不返回。不懂这三条没关系，知道结论就行：语言必须内置一个叫`Never`的类型，它没有任何值，叫系统知道你写`assert(false)`之后那段代码根本不可能跑到。

## 我们给自己造了一堆假分类，然后自己绕进去了

补完Never之后，就该回答"判不了的时候怎么办"。这一步，我们整整浪费了两轮讨论，因为犯了一个教科书级的错误。

**第一步：拆分Unproven。**

"判不了"看起来有好几种情况。依赖运行时变量的——`x`是外部输入，编译期当然不知道。超搜索预算的——理论上能判，但算太贵放弃了。缺上下文前提的——不是命题本身的问题，是调用方没提供足够的静态信息。哥德尔不可判定的——给再多时间和预算也判不出来，理论上就不可能。

于是我们把它拆成四种，取了一堆名字，然后争论每一种该怎么处理。依赖运行时的——降级为运行时检查？超预算的——加预算还是问用户要证明？缺前提的——补前提还是报错？不可判定的——只能用户写证明吧？

**拆完发现分类错了。**

分类的轴选错了。我们按"编译器拿到判不了之后该怎么办"在分类。但"该怎么办"是推理的结果，不是分类的依据。正确的问题是：**这个命题的谓词，编译期能不能拿到一个真值？**

能拿到的——参数在泛型位，值在编译期是确定的，比如`StaticArray: (T: Type, N: Int) -> Type`里的N——进证明管道。管道给三个答案：Proved（真，擦除）、Disproved（假，编译错误永不得翻身）、Unknown（真·判不了，但这个位置编译期有能力证，所以问你要证明）。

拿不到的——参数是函数参数，值由调用方运行时传入，比如`process: (n: Int) -> ...`里的n——根本不该进证明管道。直接走运行时检查。这条路上不存在Unproven，因为不存在"证"这个问题——你不可能为"用户可能输入了一个负数"这种命题写证明，因为它根本不是恒真的。

**就两条路。四种变两种，拆出来的多余分类是自己绕自己。**

这个结论刚出来时我们还不太信。为了验证，我举了一个例子——`process: (N: Int) -> ... { assert(N > 0) }`，说这里N编译期已知但prover判不动，是"判不了+编译期已知"的新类别，该强制写证明。

**被当场推翻。** `process`是普通函数，`N`是普通函数参数——它的值来自调用方，编译期根本不知道。我把函数签名里的`N: Int`当成了"编译期已知N的值"——但在YaoXiang里，"编译期已知"精确对应的是**泛型参数位**，不是普通函数参数位。同样的写法`N: Int`，在类型构造器（`-> Type`）里是编译期常量，在普通函数里是运行时值。两个N，两个宇宙。

纠正之后，"判不了+编译期已知"的真实面孔暴露了：不是"正常人写了证明器恰好判不动的命题"，而是**有人在泛型上下文里错用了小写`assert`**——那个位置编译期就知道参数的值，不该走运行时检查。这不是一个新类别，这就是精化类型本来的规则：身处需要证明的领域，判不了就该提供证明。

**运行时逃生舱不需要设计——它天然存在。** 就是"值编译期不知道，运行时才去查"的那条路。健全（运行时真的去查了）、顺手（不写prove）、开销显式（值本来就从外部来）。不是"我们没有prover所以才需要运行时"，而是**这一类命题在编译期根本没有真假可言**——prover再强也没用，用户可能真的输入了一个不满足条件的值，命题本身就是假的，你写不出真命题的证明。运行时是唯一sound的选择，理论上必要，不是妥协。
## 每一次走错,根源都是同一件事

回头看，推翻了九个错误结论：

1. `assert`返回空值`()`——那样运行时检查的结果传不回编译期，后续代码不知道条件已成立
2. `assert`能"消解"Unproven——不对，它不参与证明，它做的是运行时实际检查
3. 运行时命题和缺上下文命题是Unproven的不同子类——不对，它们在管道入口就被分走了，压根不生Unproven
4. 需要把RFC-027改成"特化擦除"——不用，擦除的是零大小的证明令牌，运行时保留的是那次Bool检查
5. 不可变的变量直接进编译期——不对，不可变不等于是编译期已知的值（`x = read_int()`不可变，但值编译期不知道）
6. 函数参数`N: Int`在编译期已知——只有泛型参数是，普通参数的值运行时才有
7. 有prove就不用运行时——运行时输入没有恒真的命题可证明，运行时在理论上必要
8. "判不了"该默认降级为运行时检查——判不了+编译期已知就该要prove，降级的是"编译期不知道值"那一类
9. 类型级match就是归纳法——match只是分类讨论，没有归纳假设；真正的归纳需要递归+终止性检查

**九个错误里，五个根子是同一个：混淆了两个长得像但其实正交的概念。** 不可变 vs 编译期已知。证明令牌 vs 产生令牌的计算。泛型参数 vs 函数参数。分类讨论 vs 归纳法。判不了 vs 证伪了。每一对都踩过——不是粗心，是概念本身的相似性让它极易滑进去。

## 什么样的问题值得开issue集思广益

讨论中开了一个issue#156，标题是"统一的难题"。一个错误的例子和一个不存在的分类，撑起了一个"需要社区帮忙"的问题。核心矛盾是假冲突，"判不了+编译期已知"这个类压根不存在——是被一个错误例子伪装出来的。答案其实在讨论的后半段就自己浮出来了——**"统一"的方案不是折中，是塌缩。每一个"可能还有第三种情况"的地方，最后都归入了已有路径。**

不是说不能开issue求助。是说要先确认：你真的见过这个问题的真实形态吗？你举的例子真的属于你给它划的类吗？你的矛盾是真的矛盾，还是"你以为某个前提成立、其实不成立"？

很多时候，"需要社区的意见"不是答案太远，是**自己被自己造的分类绕进去了**。

## 真正学到的东西

这次讨论持续了六个小时，推翻了自己九次。最终产物不是一个新的assert API，而是一张被钉死的地基：`Never`必须存在（之前没人意识到它缺席），精化类型的结果是三值不是二值（"判不了"≠"证伪"），编译期已知 ≠ 不可变，运行时检查不是prove的弱替而是唯一sound选择，`assert(x > 0)`不是"另一个东西"而是`Assert`在值宇宙的另一种命运。assert和Assert是一个东西的两面——两者的区别不是它们是谁，是编译器在那一刻能不能算。

**最好的设计是塌缩，不是新增。** 我们没有新增"runtime assert机制"，而是发现它本来就是`Assert`的一面。没有新增"判不了的处理方案"，而是发现它已经在管道里有位置。每次你觉得自己需要"加一个特殊情况"，先问自己：这是真的新东西，还是某个已有概念你没认出来？

**分类看本质，不要看你想怎么做。** 我们一开始按"编译器该做什么"分出了四个Unproven子类——拆完发现分法是错的，因为"该做什么"是推理的结果，不是分类的依据。正确的方法是按"命题在编译期有没有真值"分——有就有，没有就没有。两类，干净，没有交叉。
