# Rust 的生命周期标注是必要的吗？

*——YaoXiang 的另一种所有权模型，以及它与 Rust 的不同选择*

---

你第一次看到这段 Rust 代码时，花了多久才真正理解？

```rust
struct Parser<'a> {
    input: &'a str,
    pos: usize,
}

impl<'a> Parser<'a> {
    fn peek(&self) -> Option<char> {
        self.input[self.pos..].chars().next()
    }

    fn advance(&mut self) -> &'a str {
        let start = self.pos;
        self.pos += 1;
        &self.input[start..self.pos]
    }
}
```

三个 `'a`。一个在结构体定义上，两个在 impl 块和方法签名里。它们说的是同一件事：`Parser` 不能活得比它借用的 `input` 更久。

这是正确的。Rust 的安全性就建立在这套机制上。

但我们写这一类代码时，心里常常有一个念头挥之不去：**这件事，编译器自己不能搞定吗？** 每次在结构体里存引用，整个类型定义连带 impl 块都要加上 `'a`。Rust 的答案是"不能"——至少在不改变借用模型的前提下不能。

**本文要探索的不是"哪个语言更好"，而是一个更窄的问题：如果改变模型本身呢？生命周期标注是类型安全的必然代价，还是一种可以被重新设计的选择？**

---

## 问题的根源：Rust 为什么需要 `'a`

Rust 的默认语义是 **Move**，和 YaoXiang 一样：

```rust
let s1 = String::from("hello");
let s2 = s1;             // Move，s1 不再可用
// println!("{s1}");     // 编译错误
```

这在两种语言中是一样的。`'a` 标注的出现，不是因为"默认语义不同"——而是因为**借用这个行为在两种语言中的处理方式根本不同**。

在 Rust 中，`&T` 和 `&mut T` 是**引用**——它们是指向数据的指针。借用一个值，就是创建一个指向它的引用。这个引用有自己的生命周期，而这个生命周期必须被借用检查器追踪。当引用跨函数边界传播（作为返回值、存入结构体）时，编译器无法在单个函数的上下文内自动推断引用能活多久——它需要程序员用 `'a` 提供这个信息。

```
Rust 的因果链：
  借用 = 创建引用（指针）
  → 引用的生命周期必须被追踪
  → 跨函数/跨结构体时编译器缺乏信息
  → 程序员用 'a 标注引用之间的关系
```

**核心矛盾不在于默认语义——而在于"借用"这个概念被建模成了什么。** Rust 把借用建模为指针+生命周期，后者需要显式标注。

但 `'a` 不只是一种负担。理论上，它也让借用关系在签名中可见。比如 `fn get_x<'a>(&'a self) -> (&'a f64, &'a Self)` 中的 `'a` 标注了返回值和 `self` 的生命周期绑定。实践中这个论点被 `'a` 的学习曲线严重折损——大部分 Rust 初学者并不把 `'a` 当作文档来读，而是当作编译器要求的咒语来抄。

当然，Rust 社区并没有停在原地。**生命周期省略规则**让绝大多数简单函数签名免于标注——`fn get_x(&self) -> &f64` 不需要写 `'a`，编译器根据"输入引用 → 输出引用"的规则自动推导。**NLL** 在 2018 版落地，让借用不再被词法作用域束缚。**Polonius** 等下一代借用检查器在探索更聪明的推断能力。

这些努力在现有模型内极大地改善了体验。但当引用需要存入结构体、从函数返回、被闭包捕获时，省略规则和 NLL 都帮不上忙——**模型本身决定了这些场景必须由程序员显式标注引用之间的关系。** YaoXiang 问的是：如果模型本身可以不一样呢？

## 换个角度：把借用建模为值而不是指针+生命周期

YaoXiang 的核心设计记录在 [RFC-009（所有权模型）](/design/rfc/accepted/009-ownership-model) 中。它没有改变默认语义（都是 Move），而是改变了**借用的本体**。

在 YaoXiang 中，`&T` 和 `&mut T` **不是指针**。它们是**零大小的编译期令牌**——类型级别的访问权限证明。借用一个值，不是创建一个指向它的指针，而是创建一个代表"我被允许访问它"的令牌值。

这个建模差异有深远后果：令牌是**值**，值的生命周期由 RAII 管理。编译器不需要追踪"引用活多久"——令牌值的作用域就是它的生命周期，这和 `Int`、`String`、任何其他类型一样。

```
YaoXiang 的因果链：
  借用 = 创建令牌值（不是指针）
  → 令牌是值，生命周期由 RAII 管理
  → 编译器已经知道值的生命周期
  → 不需要 'a 标注。不需要独立的借用检查器。
```

**当借用被建模为值，生命周期标注就不再需要——RAII 已经替你完成了。**

但这里有一个衍生的代价。在 Rust 中，你写 `&` 在调用点（`distance(&p1, &p2)`）。在 YaoXiang 中，编译器看到函数签名要求 `&Point`，按 RFC-009 的自动借用选择规则在调用点自动创建令牌，调用点变成了 `distance(p1, p2)`。代价是定义者的签名必须声明 `&`，否则会意外吃掉所有权：

```yaoxiang
# 检查一个向量三次——签名中需要三次 &
check_dimensions: (v: &Vec3) -> Bool = { ... }
check_bounds: (v: &Vec3) -> Bool = { ... }
check_normals: (v: &Vec3) -> Bool = { ... }

v = Vec3(1.0, 2.0, 3.0)
if check_dimensions(v) && check_bounds(v) && check_normals(v) {
    # v 仍然可用——编译器每次都创建了 &Vec3 令牌
}
```

Rust 中你把 `&` 放在调用点；YaoXiang 中你把 `&` 放在定义点。不是标注消失了——是标注的位置变了。

## 借用令牌：零大小的编译期证明

在 YaoXiang 中，`&T` 和 `&mut T` **不是引用**。它们是**零大小的编译期令牌**——类型级别的访问权限证明。RFC-009 第 2 节详细定义了这套系统。

```
&T     →  ReadToken(T)   — 实现 Dup，可自由复制，多个只读视图共存
&mut T →  WriteToken(T)  — 不实现 Dup（线性类型），独占访问，不能复制
```

**就这么两条规则。没有第三条。**

令牌的生命周期就是值的生命周期——遵循 RAII。没有"这个引用不能从这个函数返回"的禁令。没有"借用不能逃逸"的特殊规则。令牌是普通类型，遵守所有普通类型的规则。

### 品牌机制：编译器内部如何追踪令牌

用户从不接触这个，但理解它有助于理解为什么不需 `'a`。RFC-009 第 2.7 节定义了品牌（brand）机制：

```
用户看到的         编译器内部表示
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N 是编译期唯一整数
&mut Point     →  WriteToken(Point, #M)   // #M 是编译期唯一整数
```

当你从 `&Point` 访问字段得到 `&Float` 时，后者携带派生品牌（`#N.field_x`）。编译器能追踪到父令牌。当你把 `&mut` 令牌 Move 给另一个变量时，编译器知道原变量不再持有——这是标准的流敏感活性分析，和追踪任何线性类型值的方法完全相同。

**品牌在单态化和内联后完全消失，零运行时开销。**

但这里有一个重要的坦诚：**品牌对用户不可见，意味着品牌冲突的错误也对用户不可见。** 当编译器内部的两个品牌因为某种原因冲突时，报错信息是面向值的（"Token #3 moved"），而不是面向语义的（"你不能同时持有对 `self` 的可变引用和对其字段 `x` 的可变引用"）。Rust 的借用检查器错误经过多年打磨，已经非常接近自然语言。YaoXiang 的错误信息是否能达到同样的可解释性，是一个尚未被验证的开放问题。

## 同一个安全问题，两种提问方式

到这里，你可能会想：YaoXiang 是不是发明了什么更聪明的借用追踪算法？

没有。YaoXiang 的编译器做的事情和 Rust 一样——确保没有悬垂引用，确保不同时存在可变和不可变借用。**问题本身没变。变的是借用被建模成了什么。**

Rust 的建模：
> "借用是引用——指向数据的指针。每个引用有自己的生命周期。当引用跨边界传播时，程序员需要用 `'a` 标注它们之间的关系。"

YaoXiang 的建模：
> "借用是令牌——访问权限的编译期证明。令牌是值，值的生命周期由 RAII 管理。编译器已经知道值的生命周期，不需要 `'a`。"

**同一个安全保证。两种建模方式。用户体验天差地别。**

这不是工程上的优化——不是"更快的编译器"或"更好的错误信息"。这是**本体论层面的转换**：借用从"指针+生命周期"变成了"令牌值"。RAII 替你做完了 Rust 中 `'a` 做的工作。

但反过来，这种转换也让借用关系从类型签名中消失了。在 Rust 中，`&'a T` 在签名中显式标注了生命周期约束；在 YaoXiang 中，你需要理解令牌的品牌派生规则才能推演同样的约束——而品牌对用户不可见。

## 代码对比：四个场景

### 场景 1：函数调用——编译器自动创建令牌

```yaoxiang
# YaoXiang：编译器根据函数签名自动插入令牌
distance: (a: &Point, b: &Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    return (dx * dx + dy * dy).sqrt()
}

p1 = Point(0.0, 0.0)
p2 = Point(3.0, 4.0)
d = distance(p1, p2)   # 编译器自动创建 &Point 令牌，调用结束后自动释放
```

```rust
// Rust：调用点显式标记借用，一眼可知所有权未被消耗
fn distance(a: &Point, b: &Point) -> f64 {
    let dx = a.x - b.x;
    let dy = a.y - b.y;
    (dx * dx + dy * dy).sqrt()
}

let p1 = Point::new(0.0, 0.0);
let p2 = Point::new(3.0, 4.0);
let d = distance(&p1, &p2);  // & 标记了这里是借用而非 Move
```

YaoXiang 的调用点更干净。但读代码时，`distance(p1, p2)` 看起来就像是传值——你不知道 `p1` 在这里是被 Move 了还是被借用了，除非你去查签名。Rust 的 `&` 在这里是一份局部可见的信息。两种选择各有侧重。

### 场景 2：返回引用——令牌随值传播

```yaoxiang
# YaoXiang：令牌是值，自然可返回
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p_ref) = p.get_x()   # 令牌传播到调用方，无需标注
```

```rust
// Rust：'a 在签名中显式声明返回值与 self 的生命周期关系
impl Point {
    fn get_x<'a>(&'a self) -> (&'a f64, &'a Self) {
        (&self.x, self)
    }
}

let p = Point::new(1.0, 2.0);
let (px_ref, p_ref) = p.get_x();
```

Rust 用 `'a` 在签名中声明了返回值和 `self` 共享生命周期。YaoXiang 中同样的约束由编译器通过令牌品牌派生自动追踪——`px_ref` 的令牌从 `p` 的令牌派生，父令牌存活时子令牌有效。同一份安全保障，在代码中的可见性不同。

### 场景 3：结构体持有引用——无生命周期参数

```yaoxiang
# YaoXiang：无生命周期参数，令牌字段和普通字段没有区别
Window: Type = {
    target: Point,
    view: &Point,   # 令牌字段——从 target 派生的只读视图
}

w = Window { target: p1, view: &p1 }  # 自然创建
```

```rust
// Rust：引用字段需要生命周期参数，'a 显式约束了 Window 的存活范围
struct Window<'a> {
    target: Point,
    view: &'a Point,
}

let w = Window { target: p1, view: &p1 };
```

Rust 中，只要结构体里有一个引用，`'a` 就需要出现在结构体定义和所有 impl 块中。在 YaoXiang 里，`view: &Point` 就是一个普通的令牌字段——它的生命周期随 `Window` 实例结束。`Window<'a>` 比 `Window` 多携带了一份生命周期约束信息，但代价是标注的传播。

### 场景 4：闭包捕获引用——零成本令牌复制

```yaoxiang
# YaoXiang：&Float 是 Dup 令牌，闭包捕获 = 复制一个零大小令牌
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)   # threshold 令牌复制进闭包，零开销
}
```

```rust
// Rust：省略规则自动推断生命周期，不需要显式标注
fn filter_by_threshold(items: Vec<Point>, threshold: &f64) -> Vec<Point> {
    items.into_iter().filter(|p| p.x > *threshold).collect()
}
```

`&Float` 实现了 Dup（可复制），闭包捕获它就像捕获一个零大小的整数。[RFC-023（闭包捕获模型）](/design/rfc/accepted/023-closure-capture-model) 详细定义了这套规则：Dup 类型直接复制、非 Dup 不逃逸则自动借用、非 Dup 逃逸则 Move。和函数调用的自动借用选择**共用同一套规则**。用户零标注。

## 一个诚实的旁注：标注消失 = 信息也消失了？

Rust 的生命周期标注有一个常被提起的论点：`'a` 不只为编译器服务，它也是一份"机器可检查的契约"。比如：

```rust
fn split_at_mut<'a>(slice: &'a mut [T], mid: usize) -> (&'a mut [T], &'a mut [T]);
```

理论上，`'a` 告诉读者返回的两个切片引用同一个原始数据的不同部分。但现实中，大多数 Rust 开发者的实际体验是：**先照猫画虎地写 `'a`，等编译器报错再调整。** `'a` 的学习曲线陡峭到让它很难作为有效的文档——如果读者不理解标注的含义，标注就不是文档。

所以这个论点的实际力度有限。但公平地说，标注消失确实意味着一种在极端情况下有用的信息消失了——当借用关系足够复杂、你确实需要推演"这个引用到底来自哪里"时，Rust 的 `'a` 至少给了你一个起点。YaoXiang 中你需要追溯令牌的品牌派生链——这依赖工具链，而工具链尚未成熟。

## "令牌是值"的隐性约束：当 RAII 过于保守时

文章前面的论述有一个隐含的前提：令牌作为普通值，遵循 RAII 自动管理生命周期——这听起来很干净。但它有一个结构性的缺陷。

普通值的 RAII 规则是：**值存活直到作用域结束。** 这恰恰是 Rust 在 NLL（Non-Lexical Lifetimes）之前的问题——借用持续到整个块结束，即使你早已不再使用它。

考虑这个场景：

```yaoxiang
process: (data: &mut Data) -> Void = {
    # 第一步：创建子令牌，读取一部分数据
    header_view: &Header = data.header()    # 从 &mut Data 派生 &Header
    header_info = parse_header(header_view)
    # header_view 在这里已经不再需要了——
    # 但它是值，按照朴素的 RAII，它存活到函数结束

    # 第二步：需要 &mut Data 来修改数据
    data.modify(header_info)   # ❌ 如果编译器只做词法作用域分析，这里会失败
}
```

Rust 的 NLL 正是为了解决这个问题而引入的：借用检查器分析**最后一次使用**而不是词法作用域。在 NLL 下，`header_view` 在 `parse_header(header_view)` 之后就"死了"，后续的 `&mut` 借用合法。

YaoXiang 的方案记录在 [RFC-009a（令牌生命期 DAG 分析）](/design/rfc/draft/009a-token-lifetime-dag) 中。核心思路是：利用 `{}` 块已有的 DAG 结构——每个语句/表达式是 DAG 的一个节点，边是数据依赖。DAG 构建完成后，每个令牌的最后一个消费节点就是它的死亡点。不是"推断"令牌的生命期，是**查询** DAG：这个令牌的最后一个消费者在哪个节点？当前节点之后还有消费者吗？

```
data = vec![1, 2, 3]         # 节点 1：生产 data
view = &data                  # 节点 2：生产 ReadToken(#N)，依赖 data
x = view.total_count          # 节点 3：消费 view，生产 x
# DAG 查询：view（ReadToken(#N)）的最后一个消费节点 = 节点 3
# → ReadToken(#N) 在节点 3 之后死亡
data.push(4)                  # 节点 4：需要 WriteToken
# → ReadToken(#N) 已死 → ✅ 安全
```

与 Rust NLL 的差异在于：NLL 需要控制流图不动点迭代来处理分支汇合、循环、遮蔽等歧义。YaoXiang 的设计约束——禁止变量遮蔽、显式 `return`、`for` 循环每次迭代新建绑定、`{}` 块有明确的依赖驱动语义——已经消除了这些歧义。DAG 天然无环，不需要不动点迭代。

**但这份方案目前还是草案，尚未实现。** 当前编译器仍然在 IR 生成阶段硬编码了令牌的释放位置（函数调用后），导致上述代码在当前版本中确实编译失败。

## 冲突检测：没有借用检查器

YaoXiang 没有独立的"借用检查器"这个概念。冲突检测是类型检查器的一部分——它对令牌值做流敏感的活性分析，和它对任何线性类型做的分析完全一样。RFC-009 第 2.5 节详细描述了这套机制。

```yaoxiang
# ❌ &mut T 是线性的，不能复制
bad: (p: &mut Point) -> Void = {
    p2: &mut Point = p    # Move：WriteToken 从 p 转移到 p2
    p.x = 10.0            # 编译错误："WriteToken(#3) was moved"
}

# ✅ &T 是 Dup，可以自由复制
good: (p: &Point) -> Void = {
    p2: &Point = p        # Dup：复制只读令牌
    print(p.x)            # OK
    print(p2.x)           # OK——两个只读令牌共存
}
```

关键细节：错误信息是 `"WriteToken(#3) was moved"`——一个普通的类型错误。不是借用检查器的特殊错误码。你不需要学习两套诊断系统。

但如前面品牌机制部分所讨论的，**这种统一的好处也有代价。** 一个复杂的借用冲突（比如"你试图在持有来自同一所有者的两个可变子令牌的同时读取第三个字段"）在 Rust 中会产生一个精心措辞的借用检查错误，解释冲突的根源。在 YaoXiang 中，这个错误可能表现为"WriteToken(#7.field_x) conflicts with WriteToken(#7.field_y)"——技术上准确，但缺少对人类读者的解释。

这是 YaoXiang 编译器错误诊断尚未被验证的领域。

## `&` 在签名中的真实代价：一个具体的反例

前面提到 YaoXiang 把 `&` 从调用点移到了签名中。这里给一个更真实的场景来展示这个选择的后果。

考虑你有一个大型的 `Scene` 对象，需要在渲染管线中依次传递给多个检查器：

```rust
// Rust：每个检查器显式借用（&），scene 的所有权不被消耗
let scene = Scene::load("level1.scene");
check_collisions(&scene);     // 借用
check_visibility(&scene);     // 借用
check_lighting(&scene);       // 借用
render(&scene);               // 借用
// scene 仍然可用，可以继续使用
```

```yaoxiang
# YaoXiang：必须在每个检查器的签名中写 &Scene
# （或者在调用点显式创建令牌）

scene = Scene.load("level1.scene")
check_collisions(scene)     # 如果签名是 (&Scene) → 自动借用
check_visibility(scene)     # 如果签名是 (&Scene) → 自动借用
check_lighting(scene)       # 如果签名是 (&Scene) → 自动借用
render(scene)               # 如果签名是 (&Scene) → 自动借用
# scene 仍然可用
```

在这个场景中，YaoXiang 的调用点实际上更干净——你不需要写 `&`。但代价是**每个检查器的函数签名都必须声明 `&Scene` 参数，否则会意外吃掉 scene 的所有权。** 如果你在定义某个检查器时忘了写 `&`，第一个调用它的地方不会报错——`scene` 被 Move 了，错误会延迟到下一次使用 `scene` 时才爆发。错误信息是"scene was moved"，它指向调用点，而不是指向真正的问题来源——那个漏写了 `&` 的函数签名。

更根本的问题在于权力结构的变化。默认 Move 让 API 设计者的权力变大了，调用者的权力变小了。在 Rust 中，即使库作者写了一个不合理地要求所有权的函数签名 `fn consume(scene: Scene)`，调用者可以通过传 `&scene` 绕过——决定借用还是 Move 的权力在调用者手里。在 YaoXiang 中，这个权力在定义者手里——如果函数签名写的是 `(scene: Scene)`，调用者只能接受 Move。**一个 API 作者的疏忽会变成所有使用者的麻烦，而且没有调用侧的补救方案。**

## 跨作用域共享：`ref` 关键字

令牌不能跨任务（跨线程）使用——它们是编译期证明，不是运行时值。当你需要跨作用域共享数据时，YaoXiang 提供了 `ref` 关键字。RFC-009 第 3 节和 [RFC-024（并发模型）](/design/rfc/accepted/024-concurrency-model) 一起定义了它的完整行为。

```yaoxiang
# 跨任务共享
shared_data = ref Point(1.0, 2.0)   # 编译器通过逃逸分析自动选择 Rc 或 Arc

spawn {
    print(shared_data.x)   # 跨任务共享——编译器识别到逃逸，选 Arc
}
```

编译器自动决策：
- 不逃逸到其他 spawn 块 → `Rc`（非原子引用计数）
- 逃逸到其他 spawn 块 → `Arc`（原子引用计数，线程安全）

**循环引用处理**（RFC-009 第 3.3 节）：
- 任务内循环：静默允许（结构化并发保证任务结束时统一释放）
- 跨任务循环：lint 警告（默认 `warn`，可配置为 `deny`）

`Weak` 作为标准库类型存在，不是语言内置。大多数场景不需要它。

### 自动选择的代价：局部不可见性

`ref` 的自动选择有一个代价：**阅读代码时，你无法从局部知道 `ref` 是 Rc 还是 Arc。** 在 Rust 中，`Rc::new` 和 `Arc::new` 是显式的，你看一眼就知道线程安全特性。在 YaoXiang 中，一个 `ref` 是原子引用计数还是非原子引用计数，取决于它是否逃逸到 `spawn` 块——这可能需要追踪整个调用图才能确定。

在性能敏感的场景中，如果你依赖 `ref` 是非原子的（Rc）来达到某个延迟目标，而某天有人在代码的另一处把数据传入了 `spawn` 块，编译器会自动切换到 `Arc`。你不会收到任何警告——性能悄悄地变差了。Rust 在这一点上是显式的：`Rc` 不会跨线程，如果你需要跨线程，编译器会报错要求你改用 `Arc`。

这不是说自动选择是错的——它减少了需要做的决策。但它的确把"看见成本"的能力从程序员转移到了工具链上。

## 当 YaoXiang 让你翻白眼的时候

前面用了很多篇幅讲 YaoXiang 怎么消除 `'a`。现在说一个它让人想摔键盘的时刻。

有一次我写一个循环解析器，中间创建了一堆只读子令牌来读取数据的不同部分，后续需要可变令牌来写入解析结果：

```yaoxiang
parse_chunk: (data: &mut Data) -> Void = {
    # 第一步：创建只读子令牌，读取各段数据
    header: &Header = data.read_header()      # 从 &mut Data 派生 &Header
    magic = verify_magic(header)
    body: &Body = data.read_body()            # 从 &mut Data 派生 &Body
    payload = extract_payload(body)

    # header 和 body 在这里已经不再需要了
    # 但它们是值——默认活到函数结束

    # 第二步：需要 &mut Data 来写入
    data.write_result(payload)   # ❌ 编译失败！
    # "ReadToken(#3) and WriteToken(#3) are both live"
}
```

编译器的错误信息是令牌冲突。解决方案是手动缩短子令牌的作用域——把它们包在嵌套块里，或者提前 drop：

```yaoxiang
parse_chunk: (data: &mut Data) -> Void = {
    payload = {
        header: &Header = data.read_header()
        magic = verify_magic(header)
        body: &Body = data.read_body()
        extract_payload(body)
    }   # header 和 body 令牌在这里随块结束而释放

    data.write_result(payload)   # ✅ &mut Data 令牌现在可用了
}
```

**这一刻我确实想念 Rust 的 NLL。** 在 Rust 里，编译器会自动分析 `header` 和 `body` 的最后使用点，不需要你手动创建嵌套作用域。

好在 [RFC-009a（令牌生命期 DAG 分析）](/design/rfc/draft/009a-token-lifetime-dag) 已经提出了解决方案：利用 `{}` 块内建的数据依赖 DAG，编译器可以查询每个令牌的最后一个消费者在哪个节点——此后令牌自动死亡，不需要手动缩短作用域。YaoXiang 的设计约束（禁止变量遮蔽、显式 `return`、`for` 每次迭代新建绑定）让 DAG 天然无环，避免了 Rust NLL 的控制流图不动点迭代。不过这份方案目前还是草案，尚未实现。

当前版本的解决方法是嵌套块或提前 drop。它不是致命的——但它是真实的摩擦，每个从 Rust 转过来的人都会碰到，直到 DAG 分析落地。

## 设计迭代：RFC-009 的九次重写

这不是一个在象牙塔里想出来的设计。RFC-009 经历了九次大版本迭代：

| 版本 | 关键变化 | 为什么被推翻 |
|------|----------|------------|
| v1–v7 | 基于 Rust 的所有权模型，逐步增加消费分析、逆函数、字段级可变性 | 过度设计，复杂性失控 |
| **v8** | 删除过度设计，引入"丐版借用"——`&T`/`&mut T` 只能做参数，不能返回、不能存结构体、不能闭包捕获 | **三条"禁止逃逸"的硬编码规则。表达力严重受限** |
| **v9** | **借用令牌系统**——`&T`/`&mut T` 是普通类型，遵守普通规则 | ✅ 消除了所有特殊规则，但将复杂度下沉到了品牌追踪算法 |

v8 到 v9 的跃迁是一个设计上的改进：从三条硬编码禁令（不能返回、不能存结构体、不能闭包捕获）到零条特殊规则。v9 说"借用不是一个特殊的东西——它就是一个值，遵守所有值都遵守的规则"。

但这里需要一份坦诚：**v9 消除的是用户可见的特殊规则，而不是系统的内在复杂性。** 品牌机制、派生品牌追踪、同源令牌冲突检测——这些复杂性仍然存在，只是从用户面前移到了编译器内部。对大多数用户来说，这是纯收益——他们不需要学习特殊规则。但当一个品牌冲突错误出现时（"ReadToken(#4.x) conflicts with WriteToken(#4)"），用户面对的是一条面向编译器内部表示的错误消息，而不是面向语义的解释。

Rust 走了另一条路：把复杂性暴露给用户（`'a` 标注），但错误信息经过精心设计，能解释冲突的语义。两种哲学各有优劣——简单性在哪个层面，复杂性在哪个层面。

## 诚实的权衡：YaoXiang 失去了什么

下表总结了从 Rust 到 YaoXiang 的语义迁移中，用户获得了什么、失去了什么：

| 维度 | Rust 的做法 | YaoXiang 的做法 | YaoXiang 的代价 |
|------|------------|----------------|----------------|
| 借用标注 | `'a` 在类型签名中 | 无 | 复杂借用场景下，标注可作为追溯数据流的起点（但学习曲线折损实际可用性） |
| 借用点标注 | `&` 在调用点 | `&` 在定义点（签名），调用点自动 | 签名的 `&` 负担增加 |
| 共享所有权 | `Rc::new` / `Arc::new` 显式 | `ref` 自动选择 | 局部不可见，性能变化是隐式的 |
| 借用检查错误 | 精心措辞的语义错误 | 类型错误（令牌冲突/移动） | 复杂冲突的可解释性未被验证 |
| 标注的文档价值 | 理论上 `'a` 可读，实践中学习曲线限制了实际效用 | 依赖 IDE/LSP 按需展示令牌关系 | 复杂场景下追溯数据流需要工具链支持 |
| 词法借用问题 | NLL（控制流图不动点迭代） | DAG 查询死亡点（RFC-009a 草案） | 方案尚未实现，当前需手动缩短令牌作用域 |
| 所有权语义 | 调用者可以借用（`&`）来绕过所有权消耗 | 定义者控制所有权语义，调用者无绕过能力 | API 设计错误更难在调用侧补救 |

这些不是缺点——它们是**取舍**。每个"损失"都对应着一个"获得"。问题是：对你来说，哪些获得值得对应的损失？

## 哪些场景下 Rust 的做法更合适？

作为一个诚实的技术分析，以下场景中 Rust 的显式生命周期标注可能是更好的选择：

### 1. 复杂嵌套借用需要追溯数据流

当你面对类似这样的签名时——

```rust
fn complex<'a, 'b>(x: &'a Data, y: &'b Config) -> (&'a Output, &'b Metadata) { ... }
```

`'a` 和 `'b` 至少给了你一个起点去追溯"返回值里的 `Output` 来自 `x`，`Metadata` 来自 `y`"。在 YaoXiang 中，返回值里的令牌来自哪个输入参数，需要从函数体推断或依赖 IDE 展示。在代码审查时（比如看 GitHub diff），这部分信息是不完整的。

但这有一个前提：审查者能读懂 `'a` 和 `'b`。对于还没有跨过 Rust 生命周期学习曲线的团队成员来说，标注的存在和不存在一样没有信息量。所以这个场景更适合说"标注对能读懂它的人有用"，而不是"标注本身就是文档"。

### 2. 性能瓶颈路径上的共享所有权

当性能至关重要时，Rust 的 `Rc::new` / `Arc::new` 是显式的。你知道你引入了什么开销。YaoXiang 的 `ref` 自动选择 `Rc` 或 `Arc` 意味着一次重构（比如把代码包进 `spawn`）可能悄悄地改变了引用计数的实现——你不会得到编译器的提醒。

---

## 结论

Rust 需要生命周期标注，不是因为它默认借用（它默认 Move，和 YaoXiang 一样），而是因为它把借用建模为引用——指向数据的指针，每个引用有自己的生命周期。当引用跨函数/结构体边界传播时，编译器缺乏信息来推断引用之间的关系，`'a` 是程序员提供的这部分信息。Rust 社区通过省略规则、NLL、Polonius 等项目在现有模型内持续改善体验，成果显著。

YaoXiang 的实验是：**如果改变模型本身呢？** 把借用重新建模为令牌值——零大小的编译期权限证明——值的生命周期由 RAII 管理。当借用不再需要追踪"引用活多久"，`'a` 就不再是必须的。

**这个实验有真实的代价。** 借用关系从签名中消失了，`ref` 的自动选择让运行时成本变得不透明。令牌生命期分析（RFC-009a）还在草案阶段——在它落地之前，当令牌作用域比实际需要长时，你需要手动嵌套块来缩短它。YaoXiang 还远远不能替代 Rust——没有生态，没有生产级编译器，错误信息经常让人困惑。

但它在一个特定问题上——**生命周期标注的消除**——展示了一种可能。如果有一天，一个新手打开编辑器，写完第一个结构体和方法，从头到尾没有写过一个 `'a`，也没有被借用检查器的跨函数错误淹没——那可能是 YaoXiang 想验证的体验。

如果这个想法让你好奇，[RFC-009](/design/rfc/accepted/009-ownership-model) 是了解它的起点。设计还在迭代中，欢迎关注。
