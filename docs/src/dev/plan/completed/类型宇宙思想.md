## 引言：类型宇宙思想——一个心智模型

YaoXiang 语言的设计哲学是 **"类型解决一切"**。为了帮助开发者理解类型系统中各种实体的角色及其相互关系，我们提出 **"类型宇宙思想"**——一个分层的心智模型。该模型将语言中的概念按语义角色划分为不同层级，统一了函数、类型构造器、接口、值依赖类型等概念，让开发者能够以一致的视角看待整个语言。

**重要说明**：此模型是 **对外呈现的心智模型**，旨在统一语法、指导语义高亮、简化学习曲线。**实际的编译器内部实现**基于传统的累积宇宙分层（`Type₀, Type₁, Type₂, …`），以保证逻辑严谨性并避免悖论。两者互不冲突，外部模型不改变编译器实现，仅用于理解和描述。

---

## 外部心智模型：类型宇宙层级

| 层级   | 角色               | 示例                                                                 |
|--------|--------------------|----------------------------------------------------------------------|
| Type-1 | 值                 | `42`, `"hello"`, `add` 函数本身（函数也是值）                         |
| Type0  | 元类型关键字       | `Type`（用于标注"这是一个类型"）                                      |
| Type1  | 具体类型           | `Int`, `String`, `Point`（可直接用于声明变量）                         |
| Type2  | 函数/类型构造器/值依赖类型 | `add: (Int,Int)->Int`, `List: (T: Type) -> Type`, `Vec: (n: Int) -> Type` |
| Type3  | 接口/约束          | `Clone: Type = { clone: (Self) -> Self }`                            |
| Type4+ | 元编程构件（预留） | 宏、代码生成器等                                                     |

- **Type-1** 层包含所有运行时值，包括函数本身。
- **Type0** 层是特殊的 `Type` 关键字，用于指示类型的类型。
- **Type1** 层是具体类型，即通常意义上的"数据类型"。
- **Type2** 层是所有可调用的实体：普通函数（返回值是值）、类型构造器（返回值是类型）、值依赖类型（返回值是类型）。
- **Type3** 层是接口/约束，描述一组方法签名。
- **Type4+** 为未来元编程特性预留。

这个分层完全基于**语义角色**，而非逻辑阶数。它帮助开发者直观理解每个符号的用途。

## 内部实现：传统累积宇宙分层

编译器内部采用标准的累积宇宙分层（如 Coq/Agda）：

- `Type₀`：包含具体类型（如 `Int`）以及这些类型的值。
- `Type₁`：包含 `Type₀` 的类型（即类型的类型），如 `Type₀` 本身、类型构造器 `List : Type₀ → Type₀` 的类型等。
- `Type₂`：包含 `Type₁` 的类型，依此类推。
- 关系：`Typeₙ : Typeₙ₊₁`，形成无限上升的层级，避免 `Type: Type` 悖论。

每个类型表达式在内部都被赋予一个宇宙级别，宇宙级别在类型检查时自动管理，对开发者完全透明。外部心智模型的层级与内部宇宙级别之间存在自然的对应关系，例如：

- 具体类型（Type1）通常对应内部 `Type₀`。
- 类型构造器（Type2）通常对应内部 `Type₁` 或更高（取决于参数宇宙）。
- 接口（Type3）通常对应内部 `Type₁` 或更高。

但这种对应是编译器内部细节，开发者无需关心。

## 统一语法

无论外部心智模型如何分层，YaoXiang 的语法始终保持统一：

```
identifier : type = expression
```

其中 `type` 是类型表达式，`expression` 是值表达式。没有 `fn`、`struct`、`trait` 等专用关键字。

### 类型构造器 vs 函数/变量：核心规则

**这是一个关键的设计选择，决定了语法的歧义消除规则：**

| 写法 | 含义 | 规则 |
|------|------|------|
| **`x: Type = ...`** | 类型构造器 | `: Type` 显式声明 → 强制为类型 |
| **`f = ...`** | 函数或变量 | 无 `: Type` → HM 主动推断为函数/变量 |

**为什么这样设计？**

`{ ... }` 语法本身有歧义：
- `{ x: Float, y: Float }` 可以是**类型字面量**（记录类型）
- `{ a = 1 + 1 }` 可以是**代码块**（执行语句，返回 Void）

**消除歧义的规则**：
- **有** `: Type` → 强制解析为类型构造器，`{ ... }` 是类型字面量
- **没有** `: Type` → HM 主动将 `{ ... }` 解析为代码块，推断为函数类型

```yaoxiang
# ✅ 类型构造器：有 : Type
Point: Type = { x: Float, y: Float }

# ✅ 函数：没有 : Type，HM 推断为 () -> Void
main = { println("Hello") }

# ❌ 错误：没有 : Type，编译器无法将 { ... } 解析为类型
Point = { x: Float, y: Float }  // HM 推断为函数，不是类型！
```

### 语法示例

| 概念 | 代码写法 | 说明 |
|------|----------|------|
| 变量 | `x: Int = 42` | 有类型标注 |
| 函数 | `add: (a: Int, b: Int) -> Int = a + b` | 有完整类型 |
| 函数（HM推断） | `add = (a, b) => a + b` | 无 `: Type`，HM 推断为函数 |
| 记录类型 | `Point: Type = { x: Float, y: Float }` | **必须有 `: Type`** |
| 接口 | `Clone: Type = { clone: (Self) -> Self }` | **必须有 `: Type`** |
| 类型构造器 | `List: (T: Type) -> Type = { data: Array[T] }` | **必须有 `: Type`** |
| 泛型函数 | `map: (T: Type, R: Type) -> ((list: List[T], f: (T) -> R) -> List[R]) = ...` | **必须有 `: Type`** |
| 值依赖类型 | `Vec: (n: Int) -> Type = (n) => { data: Array[n] }` | **必须有 `: Type`** |
| 带约束的函数 | `clone: (T: Clone) -> ((value: T) -> T) = (value) => value.clone()` | 无 `: Type`，HM 推断为函数 |

- 函数类型必须包含参数名，如 `(a: Int, b: Int) -> Int`。
- 类型构造器也是函数，其类型为 `(T: Type) -> Type`，完全对齐。
- 接口也是类型，定义形式与记录类型一致。
- **AST 统一为 Binding 节点**，不再区分 fn/struct/trait 关键字。
- **语义判定规则**：
  - 有 `: Type` → 类型构造器
  - 无 `: Type` → HM 推断为函数/变量

## 泛型系统

泛型系统基于"类型也是值"构建，所有泛型概念均统一为类型级函数或带约束的参数化函数。

### 类型构造器
```
List: (T: Type) -> Type = { data: Array[T] }
```
- 接受类型参数 `T`，返回一个记录类型。
- 应用时使用方括号 `List[Int]`（语法糖，等价于 `List(Int)`）。

### 泛型函数
```
map: (T: Type, R: Type) -> (
    (list: List[T], f: (x: T) -> R) -> List[R]
    ) = ...
```
- 类型是柯里化的类型级函数：先接受类型参数，返回值级函数类型。
- 调用时可类型推导：`map(ints, double)` 自动填充 `T`、`R`。

### 类型约束：泛型约束 vs 接口对象

YaoXiang 支持两种实现类型约束的方式：

#### 方式一：泛型约束（编译期多态）
```
Clone: Type = { clone: (Self) -> Self }
clone: (T: Clone) -> ((value: T) -> T) = (value) => value.clone()
```
- 约束 `T: Clone` 表示 `T` 必须实现 `Clone` 接口。
- 编译期单态化，为每个具体类型生成独立代码。
- 零运行时开销（静态分发）。
- 多重约束可用交集类型：`(T: Clone & Debug) -> ...`

#### 方式二：接口对象（运行时多态）
```
clone_obj: (value: Clone) -> Clone = (value) => value.clone()
```
- 参数类型直接写接口名 `Clone`。
- 运行时动态分发，通过虚表调用方法。
- 适用于异构集合、运行时多态场景。

| 特性 | 泛型约束 `(T: Clone)` | 接口对象 `value: Clone` |
|------|----------------------|------------------------|
| 参数类型 | 具体类型 `T` | 接口类型 `Clone` |
| 返回值类型 | 具体类型 `T` | 接口类型 `Clone` |
| 代码生成 | 编译期单态化 | 运行时动态分发 |
| 性能 | 零开销 | 间接调用开销 |
| 适用场景 | 高性能、保留具体类型 | 异构集合、运行时多态 |

两种方式都是合法的，开发者可根据需求选择。

### 值依赖类型与编译期常量

在 YaoXiang 的统一设计下，**类型可以依赖任何值，包括函数调用的结果**。这自然要求函数调用在编译期能够求值，且必须保证求值过程必然终止，否则类型系统将陷入无限循环。

#### 编译期求值规则

- **函数只有一种定义形式**，无需 `constexpr` 等标记：
  ```
  factorial: (n: Int) -> Int = {
      if n <= 1 { return 1 }
      return n * factorial(n - 1)
  }
  ```
- **类型位置强制编译期求值**：任何出现在类型表达式中的函数调用（如数组大小、值依赖类型参数）都要求该调用在编译期可计算且必然终止：
  ```
  matrix: Array[Int, factorial(5)] = ...  # factorial(5) 在编译期求值
  vec: Vec[factorial(3)] = ...             # 依赖类型参数也在编译期求值
  ```

#### 终止检查机制（基于霍尔逻辑）

为确保编译期求值安全，YaoXiang 引入**递减度量（decreases）规约**，与 RFC-022 无缝集成：

**递归函数的终止规约**：
```
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n              # 每次递归 n 严格递减
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}
```

**循环的终止规约**：
```
sum: (arr: Array[Int, n]) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

- `decreases` 表达式必须是良基的（如自然数）。
- 编译器在生成终止条件并调用 SMT 求解器证明。
- 支持元组度量（字典序递减）和用户自定义良基类型。

#### 编译期求值与终止检查的协同

1. 类型检查阶段遇到类型位置上的函数调用时，编译器检查该函数是否标记了 `decreases` 规约。
2. 若存在，则调用内置求值器计算函数值，同时验证递减条件在求值路径上成立。
3. 若未标记 `decreases` 但函数明显可终止（如非递归），则直接求值；否则报错提示添加终止规约。

#### 优势

- **安全性**：确保依赖类型的编译期求值必然终止，避免类型系统陷入无限循环。
- **统一性**：终止检查与部分正确性验证共享同一套规约机制。
- **渐进增强**：可从运行时检查逐步过渡到完全静态证明。

## Type4：元编程层

在"类型宇宙"的完整图景中，Type4 层被定义为**在编译期执行 YaoXiang 代码生成约束**的元编程层。

### 核心思想：编译期执行，结果内联

Type4 利用 YaoXiang 自身的解释器，在**编译期执行任意 YaoXiang 代码**：
- 复用现有的解释器，不需要单独的"类型级语言"
- 编译期求值，结果直接内联到生成的二进制
- 运行时默认零开销——没有反射结构，只有普通代码

```
┌─────────────────────────────────────────┐
│  编译期 derive                          │
│  ├── 解释器执行 derive 函数              │
│  ├── 读取类型元数据                     │
│  ├── 生成 trait 实现写入 TraitTable     │
│  └── 代码生成 → 二进制                  │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  运行时                                 │
│  └── 普通二进制，无反射，无额外开销      │
└─────────────────────────────────────────┘
```

### 实现机制

1. **类型反射 API**：类型携带元数据（字段列表、方法列表），通过标准库函数暴露
   ```yaoxiang
   // 标准库提供
   type Type = { fields: () -> Array[Field], ... }
   ```

2. **编译期求值**：解释器在编译期执行函数调用，结果内联
   ```yaoxiang
   let fields = Point.fields()  // 编译期求值 → 内联为常量数组
   ```

3. **约束写入**：求值结果（生成的 trait 实现）写入 TraitTable

### 语法与示例

#### 派生约束：为结构体生成 `Debug` 实现
```yaoxiang
Debug: Type = { fmt: (Self, Formatter) -> Result }

// 类型反射 API（标准库内置）
type Field = { name: String, type: Type }
type Type = {
    fields: () -> Array[Field],
    implements: (trait: Type) -> Bool,
}

// derive 实现
derive_debug: (T: Type) -> Type = (T) => {
    fields = T.fields()
    for f in fields {
        if !f.type.implements(Debug) {
            return err("Field " + f.name + " does not implement Debug")
        }
    }
    // 生成方法实现（写入 TraitTable）
    T.fmt(self, f: Formatter) -> Result = { ... }
}

// 使用
Point: Type = { x: Float, y: Float }
derive_debug(Point)   # 编译期执行，生成 Point 的 Debug 实现
```

#### 条件约束
```yaoxiang
conditional: (T: Type) -> Type = (T) => {
    if T == Int { return Add[T] }
    else { return Default[T] }
}
```

#### 组合约束
```yaoxiang
both: (c1: Type, c2: Type) -> Type = (c1, c2) => c1 & c2
```

### 与传统宏的对比

| 维度 | 传统宏（Rust） | YaoXiang Type4 |
|------|----------------|-----------------|
| 执行时机 | 宏展开阶段 | 类型检查阶段 |
| 语言能力 | 文本替换 | 任意 YaoXiang 代码 |
| 求值方式 | 文本生成 | 编译期解释器求值 |
| 错误捕获 | 展开后类型检查 | 求值时即时捕获 |
| 运行时开销 | 无 | 无（结果内联） |
| 可组合性 | 差 | 好（函数组合） |

| 维度 | 传统宏 | YaoXiang Type4 |
|------|--------|-----------------|
| 操作对象 | 文本/语法树 | 类型/约束 |
| 执行阶段 | 宏展开阶段 | 类型检查阶段 |
| 卫生性 | 需复杂机制保证 | 天然卫生 |
| 可读性 | 宏语法特殊，难调试 | 与普通函数一致 |
| 编译验证 | 展开后才类型检查 | 直接参与类型检查 |

### 为什么类型系统现在能取代宏？

1. **类型即一等公民**：类型可以像值一样被计算、传递、内省。
2. **约束即数据**：Type3 层的约束可以被构造、组合、派生。
3. **内省机制**：类型系统能够自省类型的结构。
4. **类型级函数**：在类型检查阶段执行纯函数，生成新的约束。

这相当于在类型系统内部构建了一个**图灵完备的、纯函数的、强类型的元编程语言**。

## 语义高亮与 LSP 实现

基于外部心智模型的角色分类，LSP 可以为代码提供准确的语义高亮：

| 语义角色 | 外部层级 | 令牌类型 | 示例 |
|----------|----------|----------|------|
| 变量/常量 | Type-1 | `variable` | `x`, `PI` |
| 函数参数 | Type-1 | `parameter` | `a` in `(a: Int)` |
| 普通函数 | Type2（返回值非 `Type`） | `function` | `add`, `factorial` |
| 类型构造器 | Type2（返回值是 `Type`） | `type` | `List`, `Vec` |
| 具体类型 | Type1 | `type` | `Int`, `Point` |
| 接口 | Type3 | `interface` | `Clone` |
| 元类型关键字 | Type0 | `keyword` | `Type` |

判断规则基于类型检查后的信息：若符号类型是函数类型 `(…) -> …`，检查返回值类型是否为 `Type`（或其特化），若是则为类型构造器 → `type`，否则为普通函数 → `function`。

## 总结

- **外部心智模型**："类型宇宙思想"为开发者提供了一个统一、直观的视角来理解语言中的各种概念（值、类型、函数、接口等），指导语法设计和语义高亮。
- **内部实现**：编译器采用传统的累积宇宙分层，确保类型系统的逻辑严谨性，避免悖论，对外部模型完全透明。
- **两者关系**：外部模型是描述性的，不影响编译器实现；内部实现是技术性的，不对外暴露层级细节。二者相辅相成，共同构建了 YaoXiang 既强大又易用的类型系统。

通过这种分离，YaoXiang 实现了：

- **极致统一**的语法
- **强大表达力**的泛型与依赖类型
- **开发者友好**的语义提示
- **理论严谨**的底层基础

这就是 YaoXiang 类型宇宙思想的完整呈现——一个用于理解的心智模型，背后是坚实的传统类型理论支撑。
