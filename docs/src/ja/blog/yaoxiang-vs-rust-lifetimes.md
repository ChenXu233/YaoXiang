# `'a` をコンパイラに隠した

*——YaoXiang 所有権モデルに関する正直な覚書*

---

あなたが初めてこの Rust コードを見たとき、本当に理解するまでにどのくらいかかりましたか？

```rust
struct Parser<'a> {
    input: &'a str,
    pos: usize,
}

impl<'a> Parser<'a> {
    fn advance(&mut self) -> &'a str {
        let start = self.pos;
        self.pos += 1;
        &self.input[start..self.pos]
    }
}
```

三つの `'a`。一つは struct 上、二つは impl ブロック内。它们说的是同一件事：`Parser` は借用している `input` よりも長生きすることはできない。これは正しい。Rust の安全性はこの仕組みの上に成り立っている。

しかし、私たちがこのようなコードを書くとき、よくこう思う——**このこと、コンパイラ自身では処理できないのか？** Rust の答えは「できない」——少なくとも借用モデルを変えなければ。

**この記事は「YaoXiang がこの問題を解決した」というものではない。「YaoXiang は別の道を試している——`'a` をコンパイラに隠し、コンパイラに書かせる。どこまで来たか、未解決の難題は何か」というものだ。**

---

## Rust なぜ `'a` が必要か

Rust の `&T` と `&mut T` はポインタ——データを指すポインタである。値を借用するとは、それへの参照を作ることだ。この参照には独自のライフタイムがある。参照が関数の境界を越えて伝播する（戻り値として、構造体に格納される）とき、コンパイラは単一の関数内で参照がどのくらいの間生存できるかを推論できない——プログラマが `'a` を使って「この戻り値とこの引数はライフタイムを共有する」という情報を提供する必要がある。

Rust コミュニティはそこで止まらなかった。ライフタイム省略規則により、ほとんどの単純な関数はアノテーションを免れる。NLL は 2018 エディションで導入され、借用が字句スコープに縛られなくなった。しかし、参照を構造体に格納し、関数から返し、クロージャにキャプチャする必要があるとき——**モデル自体が、これらのシナリオではプログラマが参照間の関係をアノテーションしなければならないと規定している。**

---

## 別の見方：借用はポインタではなく、トークンである

YaoXiang の核心的な設計は [RFC-009（所有権モデル）](/design/rfc/accepted/009-ownership-model) に記録されている。デフォルトセマンティクス（すべて Move）は変えずに、変更したのは**借用の本態**である。

YaoXiang では、`&T` と `&mut T` は**ポインタではない**。これらは**ゼロサイズのコンパイル時トークン**——型レベルのアクセス権限の証明である。値を借用するとは、それへのポインタを作ることではなく、「私はそれへのアクセスを許可されている」という証明を作ることだ：

```
&T     →  データが不変であることを保証。Dup（コピー可能）を実装し、複数の読み取り専用トークンの共存が安全
&mut T →  排他的な可変性を保証。Dup を実装しない（線形）、同じ出所からは一つだけ
```

Rust では呼び出し点で `&` を書く（`distance(&p1, &p2)`）。YaoXiang では、コンパイラが関数シグネチャで `&Point` を要求するのを見て、呼び出し点で自動的にトークンを作成する——呼び出し点は `distance(p1, p2)` になる。代償として、定義側のシグネチャは `&` を宣言しなければならない。さもなければデフォルトの Move が所有権を奪ってしまう：

```yaoxiang
# シグネチャに & が必要——コンパイラが &Point を認識し、呼び出し時に自動的にトークンを作成
check_dimensions: (v: &Vec3) -> Bool = { ... }
check_bounds: (v: &Vec3) -> Bool = { ... }

v = Vec3(1.0, 2.0, 3.0)
if check_dimensions(v) && check_bounds(v) {  # 毎回自動的に &Vec3 トークンを作成
    # v は引き続き使用可能
}
```

Rust では `&` を呼び出し点に置く；YaoXiang では `&` を定義点に置く。アノテーションが消えたわけではない——アノテーションの位置が変わったのだ。

## ブランド機構：コンパイラに隠された `'a`

ユーザーはこれに触れることはない——しかし、YaoXiang が実際に何をしているかを理解するにはこれを知る必要がある。

コンパイラは内部で、各借用トークンにコンパイル時一意の番号を割り当てる：

```
ユーザーが見るもの       コンパイラ内部表現
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)
&mut Point     →  WriteToken(Point, #M)
```

`&Point` からフィールドにアクセスして `&Float` を得るとき、後者は派生ブランドを携带する：`#N.field_x`。`&mut` トークンを別の変数に Move するとき、コンパイラは元の変数がもはや保持していないことを知る——これが Move セマンティクスの基礎能力である。

**`#N` は `'a` そのものだ。** 接頭辞関係——`#N` は `#N.field_x` の接頭辞——は outlives 制約である。同じ情報だ。Rust のプログラマは `'a` を書き、YaoXiang のコンパイラは `#N` を書く。

違いはあるのは推論の成功率だけだ。Rust の省略規則と NLL は約 80% のシナリオをカバーしている。YaoXiang の賭けはこうだ：**言語設計がコンパイラによりクリーンな入力を与えれば、もっと多くをカバーできるのではないか？**

この賭けはいくつかの言語制約によって支えられている：

- **変数のシャドーイング禁止**——このスコープで `x` は一つの ID しか持たず、コンパイラは「どの x のことを言っているのか」を区別する必要がない
- **明示的な `return`**——何がブロックから逃げるかは明記されており、コンパイラは「最後の行が戻り値かどうか」を推論する必要がない
- **`for` は各イテレーションで新しいバインディングを作成**——イテレーション間の変数は互いに干渉せず、コンパイラは「前のイテレーションで何が変わったか」を追跡する必要がない

これらは単なる「規範」ではない。Java の getter/setter のような無意味な儀式とは異なり、一つひとつが**コンパイラが推論すべき情報を、プログラム内にすでに明記された情報に変換する**ものである。コンパイラは「どの変数のことか」「これがエスケープするかどうか」「ループ変数がイテレーションを越えてどう変化するか」を推測する必要がない——答えはコードの中にある。

## ブランドができること——Rust と同じこと

トークンは通常の型なので、通常の型すべての規則に従う。「参照を返せない」「参照を構造体に格納できない」「クロージャが参照をキャプチャできない」といった特別な禁止はない。**しかし Rust でも同じことができる——Rust もすべてできる。** 違いはその能力ではなく、誰がアノテーションを書くかだ。

**参照を返す——Rust のプログラマは `'a` を書き、YaoXiang のコンパイラは `#N` を書く：**

```yaoxiang
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()   # トークンは呼び出し元に伝播し、コンパイラはブランド派生チェーンを追跡
```

Rust も同じことをするが、プログラマが入力と出力をつなぐ `'a` を書く必要がある。YaoXiang ではコンパイラがブランドパスによって自動派生する——`px_ref`（`#N.field_x`）は `p`（`#N`）から派生する。同じ制約、異なる記録方法。

**構造体に参照を保持する——ライフタイムパラメータなし：**

```yaoxiang
Window: Type = {
    target: Point,
    view: &Point,   # トークンフィールド、他のフィールドと区別なし
}
```

Rust では構造体に参照フィールドがある場合、`'a` を構造体定義とすべての impl ブロックに記述する必要がある——プログラマが `Window<'a>` のライフタイム制約を明示的にアノテーションする。YaoXiang では `view: &Point` に `'a` を書かないが、ブランド番号はコンパイラ内部で同じ役割を果たす——`Window` インスタンスが破棄されると、内部のトークンも死ぬ。同じ保証、異なる可視性。

**クロージャキャプチャ——Dup トークンのゼロコストコピー：**

```yaoxiang
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)   # threshold トークンをクロージャへコピー、オーバーヘッドなし
}
```

`&Float` は Dup（コピー可能）を実装しており、クロージャがそのトークンをキャプチャするのは、ゼロサイズの整数をキャプチャするのと同じだ。関数呼び出しの自動借用と同じ規則を共有する。ユーザーアノテーションはゼロ。

## 代償：アノテーションがシグネチャから消える

Rust の `'a` にはよく言及される価値がある：ドキュメントでもある。`fn split_at_mut<'a>(slice: &'a mut [T], mid: usize) -> (&'a mut [T], &'a mut [T])`——`'a` は読者に、二つの返されるスライスが同じ元のデータを参照していることを伝える。

現実には、この主張の力は限定的だ——ほとんどの Rust 初学者は `'a` をドキュメントとして読まず、コンパイラが要求するおまじないとして写す。しかし公平に言えば、複雑な借用シナリオでは、Rust の `'a` は少なくともデータフローをたどる出発点を与えている。YaoXiang ではブランド派生チェーンを理解する必要がある——ブランドはユーザーに不可視であり、これはツールチェーンに依存するが、ツールチェーンはまだ成熟していない。

## トークン衝突検出：同じ証明パイプライン

Rust は独立した「借用チェッカー」を持つ。YaoXiang の**設計方向**は、借用衝突を型検査の証明パイプラインに統合することである（[RFC-027（コンパイル時述語と統一静的検証）](/design/rfc/accepted/027-compile-time-evaluation-types)）。

トークン衝突はホーア命題である：

```
{ 衝突する ReadToken は死んでいる } data.push(4) { WriteToken は安全に取得可能 }
```

```yaoxiang
# &mut T は線形型——Move 後は元の変数は保持しない
bad: (p: &mut Point) -> Void = {
    p2: &mut Point = p    # WriteToken は p から p2 へ移動
    p.x = 10.0            # { p が WriteToken を保持 } p.x = 10.0 { 安全 }
}                          # → p の WriteToken は Move 済み → 反証

# &T は Dup——コピー可能
good: (p: &Point) -> Void = {
    p2: &Point = p        # 読み取り専用トークンをコピー
    print(p.x)            # OK、二つの読み取り専用トークンが共存
}
```

型エラーや述語検証失敗と同じエラー報告パスを共有する。二つの診断システムを学ぶ必要はない。**しかし代償として：** 複雑なトークン衝突は、Rust では精心に言い回された借用検査エラーを生成する；YaoXiang では「WriteToken(#7.field_x) conflicts with WriteToken(#7)」のように表現される可能性がある——技術的には正確だが、ブランド番号は人間の読者にとって意味がない。エラーメッセージの解釈可能性は、まだ検証されていない領域である。

## `ref` キーワード：Rc/Arc の自動選択

トークンはタスクをまたいで共有できない（スレッドをまたげない）——それらはコンパイル時の証明であり、ランタイム値ではない。スコープをまたぐ共有には `ref` を使う：

```yaoxiang
shared_data = ref Point(1.0, 2.0)   # コンパイラの エスケープ分析が Rc または Arc を自動選択

spawn {
    print(shared_data.x)   # タスクをまたぐ → コンパイラが Arc を選択
}
```

- spawn ブロックにエスケープしない → `Rc`（非アトミック参照カウント）
- spawn ブロックにエスケープする → `Arc`（アトミック参照カウント）

代償：コードを読む際、ローカルからは `ref` が Rc なのか Arc なのかがわからない。一度のリファクタリング（コードを spawn で包む）で、参照カウントの実装が密かに変わるかもしれない——コンパイラからの通知はない。パフォーマンスの変化は暗黙的である。

## 現在の難題：RAII が粗すぎる

先ほど述べたように、トークンは値であり、ライフタイムは RAII で管理される。しかし通常の値の RAII 規則はこうだ：**値はスコープ終了まで生存する。** これはまさに NLL 以前の Rust の問題である——借用はブロックの終わりまで続き、すでに使わなくなった後でも続く。

```yaoxiang
process: (data: &mut Data) -> Void = {
    header_view: &Header = data.header()    # &mut Data から &Header を派生
    header_info = parse_header(header_view) # ← header_view の最後の使用
    # header_view は関数の終わりまで生きる必要はない——
    # しかし RAII は } まで生存させる

    data.modify(header_info)   # ❌ ReadToken がまだ「生きて」いるため、WriteToken がブロックされる
}
```

Rust の NLL は字句スコープではなく最後の使用を分析する。YaoXiang にも同じ能力が必要だ。現在取り組んでいるのは、トークン生存解析も証明パイプラインに接続することである——三層構成：

1. **高速パス**——既存の BorrowChecker を再利用（線形スキャン、IR 命令レベル）。同一基本ブロック内でトークンが使われ終わるシナリオを直接通過させる
2. **構造解析**——ブランドツリーの接頭辞マッチング（どのトークンが衝突するかの判断）+ DAG コンシューマクエリ（トークンの最後のコンシューマが現在のノードより後にあるかの判断）
3. **SMT ソルバ**——ループ条件など論理推論が必要なときのみ起動

証明パイプライン基盤（`Proved/Disproved/Unproven` の三値返却、Z3 SMT バックエンド、仮説スタック）は `src/frontend/core/typecheck/proof/` に部分的に実装されている。所有権レイヤ（`layers/ownership.rs`）はまだスケルトンで——直接 Proved を返し、実際のチェックは行っていない。今、埋め込んでいるところだ。

現在のバージョンの解決策は、手動でブロックをネストしてトークンスコープを短縮することだ：

```yaoxiang
process: (data: &mut Data) -> Void = {
    header_info = {
        header_view: &Header = data.header()
        parse_header(header_view)
    }   # header_view はブロック終了とともに解放
    data.modify(header_info)   # ✅
}
```

これは実際の摩擦だ。Rust から来た全員がこれにぶつかる。解消できるかどうかは、パイプラインが所有権レイヤにうまく接続できるかにかかっている。

## 最も難しい問題：エスケープハッチ

Rust の `'a` は単なる負担ではない——フォールバックでもある。コンパイラが推論できないとき、プログラマがライフタイム関係をアノテーションし、コンパイラが検証する。**あなたにはペンがある。**

YaoXiang のエスケープハッチは、理論上はコンパイル時**証明関数**であるべきだ（RFC-027 §4.2）：コンパイラの自動推論が失敗 → プログラマが関数を書き、戻り値の型が「トークンは衝突しない」という命題になる → コンパイラがその関数の型を検証する。しかし——

「トークンが衝突しない」の証明関数はどういう姿をしているのか？ユーザーはどうやって `WriteTokenAvailable` 型の値を構築するのか？ブランド番号 `#N` と `#N.field_x` の接頭辞関係を理解する必要があるのか？

**もし証明関数がユーザーにブランド番号の理解を求めるなら——私たちは単に `'a` の書き方を `#1` に変えただけで、何も省けていない。**

この問題にはまだ答えがない。実験全体が最も行き詰まる可能性のある場所だ。

## RFC-009 の九回の反復

これは象牙の塔で考え出された設計ではない。RFC-009 は九回の大きなバージョンを経ている：

| バージョン | 主な変更 | 覆された理由 |
|------|----------|------------|
| v1–v7 | Rust 所有権モデルに基づき、消費分析、逆関数、フィールドレベル可変性を徐々に追加 | 過剰設計、複雑性が制御不能に |
| **v8** | 「もっさい借用」——`&T`/`&mut T` は引数としてのみ使用可能、戻り値・構造体格納・クロージャキャプチャ不可 | 三つのハードコード禁止。表現力が著しく制限される |
| **v9** | 借用トークンシステム——`&T`/`&mut T` は通常の型で、通常の規則に従う | 特殊規則を排除するが、ブランド追跡をコンパイラ内部に降ろす |

v8 から v9 への跳躍が真のブレークスルーだ：三つの禁止からゼロの特殊規則へ。しかし v9 が除去したのはユーザーに見える規則であり、システムの内在的複雑性ではない——ブランド機構、派生追跡、同一出所衝突検出、これらは依然として存在し、単にコンパイラ内に移っただけだ。借用検査を証明パイプラインに統合することは一つの方向だが、実コードで通用するか、エスケープハッチをどう設計するかは、まだ検証中だ。

## 最後に

私たちは `'a` を消し去ったわけではない。`#1` は `'a` そのものだ——同じ情報、異なる位置。

実験の賭けはこうだ：言語設計の制約（シャドーイング禁止、明示的 return、`for` の新バインディング、`{}` の DAG セマンティクス）がコンパイラによりクリーンな入力を与え、ブランド推論が Rust のライフタイム省略規則が失效するほとんどのシナリオで自動的に成功するかもしれない。もし成功すれば——ユーザーは `'a` を書く必要がなく、アノテーションと省略を区別する必要がなく、借用チェッカーを学ぶ必要がない。もし失敗すれば——あるいはエスケープハッチ機構（証明関数）がユーザーにブランド番号の理解を求めるなら——それは一種の再発明にすぎない。

作業中だ。結果が出たらまた書く。
