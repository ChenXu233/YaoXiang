# 「統一」を巡る徹夜の議論

2026年7月11日の深夜から12日の未明にかけて、我々は6時間を費やして一つの問題——YaoXiangの`assert`は本来どうあるべきか——を議論した。結果は予想外だった。我々がずっと「assertをどう実装するか」を議論していたつもりが、実は「型システムの基盤にどんな穴があるか」を議論していたのだ。

この記事で最終的な結論（結論はspecにある）は語らない。語るのは、我々がどのように段階的に自分の誤りに気づき、その度にどう正されたかである。

---

## なぜ二つのassertがあるのか

YaoXiangでは二つのものを同時に開発している：

**コンパイル時精化型** `Assert(N > 0)`——型定義の中に記述する：
```
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    _assert: Assert(N > 0),   // N必须大于0，否则编译失败
}
```
コンパイル時にジェネリクス引数を検証する。Nが5なら通過、Nが0ならコンパイルエラー。実行時のオーバーヘッドはゼロ。

**実行時アサーション** `assert(x > 0)`——関数本体の中に記述する：
```
x = read_int()
assert(x > 0)   // 检查用户输入
```
値は外部から渡されるため、コンパイル時には分からない。失敗すればpanicする。

大文字のAssert、小文字のassert。一方はコンパイル時、もう一方は実行時。同じ概念の二つの姿である。

最も自然な反応——最も手間のかからない方法——は、これらを二つの完全に独立したものとして扱うことだった。Assertは精化型で証明パイプラインを通り、assertは20行のネイティブ関数で空値を返す。互いに干渉しない。

しかしこの方案には、ずっと言葉にしがたい違和感があった。両者には明らかに関係がある。同じ意味——「この条件は必ず成立すると主張する」——の二つの側面に過ぎない。それなのに、ユーザーに「ここでは大文字、あちらでは小文字、ここはコンパイル時チェック、あちらは実行時のみ」と覚えさせるのは認知的負債である。言語設計において、「似ているように見えるが別々に記憶すべきもの」が一つ増えるごとに、借金を増やしていることになる。

## 統一を試みて、壁にぶつかる

統一という理想は実はとても簡潔だ：

```
assert: (cond: Bool) -> Assert(cond)
```

`assert`は精化型`Assert(cond)`を返す。コンパイル時にcondの値が計算できれば、コンパイル時チェックに折り畳む（消去）；コンパイル時に計算できなければ、実行時チェックに残す。一つの関数、二つの運命。シグネチャそのものが接続点となる。

しかし直ちに「根本的矛盾」を指摘する者が現れた：

> `Assert(C)`はCがコンパイル時に評価可能であることを要求する。`assert(x > 0)`のxは実行時の値であり、コンパイル時には`Assert(x > 0)`を計算できない。精化型の規則に従えば、これは**コンパイルエラー**にせざるを得ない。しかし`assert(x > 0)`は完全に正常な実行時アサーションである。従ってシグネチャが`Assert(cond)`となる統一案は成り立たない。分離すべきである。

この論点は一見したところ隙がない。精化型が実行時の値に遭遇しただけでエラーになるなら、「コンパイル時アサーション」と「実行時アサーション」を同じシグネチャに押し込めることは永遠にできない。二つのもの、二つの運命、分離しよう。

**しかしこれは誤りだ。** 誤りの根源は最後の「コンパイルエラーにせざるを得ない」という部分にある。精化型の結果は二値ではない——「証明成立」と「エラー」しかないわけではない。三値である：Proved（証明された）、Disproved（反証された、反例付き）、**Unproven（判定不能）**。

「判定不能」と「反証された」は全く異なる状態である。反証されればエラーを出すべきで、覆ることはない。しかし判定不能——まさに値が実行時にしか分からないから判定不能なのだが——は、まさに実行時まで持ち越してチェックすることができる。コンパイル時に判定できないのは、コンパイル時の失敗ではなく、この命題の**自然な属性**である。

「実行時アサーション」が「別の仕組み」であるはずがない。それはコンパイル時に判定不能な命題であり、実行時まで待ってからチェックするものに過ぎない。**同じ精化型、二つの運命。運命はそれがコンパイル時に判定可能か否かという事実によって自動的に決定される。**

偽の衝突が崩れるや、統一への道が開けた。しかし先に基盤にある穴を塞ぐ必要がある。

## 不足していた型：Never（偽 / 発散 / 不可能）

統一案は`assert(false)`を「値を持ち得ない」型へと縮約することを要求する——「falseが真である」という命題には証明がなく、それを型として使う箇所では、プログラムは発散しなければならないからだ。

そして我々は気づいた——YaoXiangにはこの型が全く存在しない。

`Void`はある。しかし`Void`はデフォルト値を返せる——`Void`を返す関数を呼び出しても、コードはそのまま先に進む。`Void`には一つの居住者がいる。Curry-Howard対応の観点では、値を持つ型は「真の命題」（証明が一つ存在する）であり、「偽の命題」（証明がゼロ）ではない。**`Void`は常に「真⊤」だった。ただ誰もそう呼ばなかっただけだ。**

我々が欠いているのは「偽⊥」——永久に居住者を得られない型——`x: Never = ...`の右辺に填める値が一つも存在しない型だ。`assert(false)`、`panic`、無限ループの返り値の型は全てこれであるべきだ。これがない場合、`assert(false)`は合法的なデフォルトvoid値を返し、その後もコードは実行を続ける——アサーション失敗が正常なフローに化けてしまう。精化型システム全体がその瞬間、論理的に崩壊する。

`Never`は最終的に**公理**として位置づけられた：何物からも導出されず、型システムの原初概念である。三つの性質が内核に組み込まれている——コンストラクタがゼロ、Neverは全ての型のサブタイプである（「偽からは全てを導出できる」）、関数の返り値型として使われた場合、その関数は発散して決して返らないことを示す。この三つを理解できなくても、結論だけ知っていればよい：言語は`Never`という型を内蔵しなければならない。これは値を持たず、`assert(false)`の後のコードには絶対に到達できないことをシステムに伝える。

## 大量の偽分類を自ら生み出し、自ら迷路に迷い込む

Neverを補った後、次に答えるべきは「判定不能のときどうするか」だ。このステップで、我々は丸々二回の議論を浪費した。教科書に載るような誤りを犯したからだ。

**第一段階：Unprovenの細分化。**

「判定不能」には一見、いくつかの種類があるように見える。実行時変数に依存する場合——`x`は外部入力であり、コンパイル時に当然分からない。探索予算を超える場合——理論的には判定可能だが、計算コストが高すぎるので諦めた。文脈上の前提が欠けている場合——命題自体の問題ではなく、呼び出し側が十分な静的情報を提供していない。ゲーデル的に決定不能な場合——どれだけ時間と予算を与えても判定できず、理論上不可能。

そこで我々はこの四種類に分割し、色々と名前をつけ、それぞれをどう処理すべきかを論じた。実行時依存のもの——実行時チェックに降格させるか？予算超過のもの——予算を増やすか、ユーザーに証明を求めるか？前提欠落のもの——前提を補うか、エラーにするか？決定不能のもの——ユーザーに証明を書いてもらうしかないのか？

**分割し終えてから、分類が間違っていたことに気づいた。**

分類の軸を誤ったのだ。我々は「コンパイラが判定不能を受け取った後どうすべきか」という軸で分類していた。しかし「どうすべきか」は推論の結果であり、分類の根拠ではない。正しい問いは：**この命題の述語について、コンパイル時に真の値を取得できるか？**

取得できる場合——引数がジェネリック位にあり、コンパイル時に値が確定している場合、例えば`StaticArray: (T: Type, N: Int) -> Type`のN——証明パイプラインに投入する。パイプラインは三つの答えを返す：Proved（真、消去）、Disproved（偽、コンパイルエラー、覆ることはない）、Unknown（真の判定不能、しかしこの位置ではコンパイル時に証明可能なので、証明の提出を求める）。

取得できない場合——引数が関数の引数であり、呼び出し側から実行時に渡される値、例えば`process: (n: Int) -> ...`のn——証明パイプラインに投入すべきではない。直接、実行時チェックに進む。この経路にはUnprovenは存在しない。「証明」という問題自体が成立しないからだ——「ユーザーが負数を入力したかもしれない」というような命題について証明を書くことは不可能である。そもそも恒真ではないからだ。

**道は二つだけ。四種が二種に、分割して生み出した余分な分類は自分が自分を回しているだけだ。**

この結論が出た当初、我々はまだ完全には納得していなかった。検証のため、一つの例を挙げた——`process: (N: Int) -> ... { assert(N > 0) }`——ここでNはコンパイル時に既知だがproverが判定できず、「判定不能+コンパイル時既知」という新カテゴリーであり、証明の記述を強制すべきだと主張した。

**その場で否定された。** `process`は通常の関数であり、`N`は通常の関数の引数である——その値は呼び出し側から来るため、コンパイル時には全く分からない。私は関数のシグネチャの`N: Int`を「コンパイル時にNの値が既知」と見なしていた——しかしYaoXiangでは、「コンパイル時既知」が正確に対応するのは**ジェネリック引数位**であり、通常の関数の引数位ではない。同じ書き方`N: Int`でも、型コンストラクタ（`-> Type`）の中ではコンパイル時定数であり、通常の関数の中では実行時の値である。二つのN、二つの宇宙。

訂正した後、「判定不能+コンパイル時既知」の真の顔が露わになった：それは「普通の人々が証明器でたまたま判定できない命題」ではない——**誰かがジェネリック文脈で小文字の`assert`を誤用している**——その位置ではコンパイル時に引数の値が分かるので、実行時チェックに進むべきではない。これは新カテゴリーではなく、精化型本来の規則に他ならない：証明が必要な領域においては、判定不能なら証明を提供すべきである。

**実行時のエスケープハッチは設計不要——自然に存在する。** それが「値がコンパイル時に分からず、実行時にチェックする」という経路だ。健全（実行時に実際にチェックされる）、手軽（proveを書かなくてよい）、オーバーヘッドが明示的（値は元より外部から来る）。「proverがないから実行時が必要」ではない——**この種の命題はコンパイル時には真偽そのものが存在しない**——proverがどれほど強力でも無意味であり、ユーザーが条件を満たさない値を実際に入力する可能性があり、命題自体が偽であり得る。真の命題の証明は書けない。実行時は唯一soundな選択であり、理論上必要なのであって、妥協ではない。

## 誤りごとに、その根源は同じ

振り返ると、九つの誤った結論を覆した：

1. `assert`が空値`()`を返す——それでは実行時チェックの結果がコンパイル時に伝わらず、後続コードは条件が成立していることを知らない
2. `assert`はUnprovenを「解消」できる——誤りである、それは証明に参加せず、実行時の実際のチェックを行うだけ
3. 実行時命題と文脈欠落命題はUnprovenの異なるサブクラスである——誤りである、それらはパイプラインの入口で既に分岐されており、Unprovenが生成されることさえない
4. RFC-027を「特化消去」に変更する必要がある——不要、消去されるのはゼロサイズの証明トークンであり、実行時に保持されるのはあのBoolチェック
5. 不変な変数は直接コンパイル時に進む——誤りである、不変であることはコンパイル時に既知の値であることと等価ではない（`x = read_int()`は不変だが、値はコンパイル時に分からない）
6. 関数の引数`N: Int`はコンパイル時に既知——ジェネリック引数のみがそうで、通常の引数の値は実行時にのみ存在する
7. proveがあれば実行時は不要——実行時入力には恒真の命題がなく、実行時は理論上必要
8. 「判定不能」はデフォルトで実行時チェックに降格すべき——判定不能+コンパイル時既知ならproveが必要、降格するのは「コンパイル時に値が不明」の方
9. 型レベルのmatchは帰納法である——matchは分類議論に過ぎず、帰納仮説を持たない；真の帰納法は再帰+停止性検査を必要とする

**九つの誤りのうち、五つの根源は同じである：互いに直交する二つの似た概念を混同したこと。** 不変 vs コンパイル時既知。証明トークン vs トークンを生成する計算。ジェネリック引数 vs 関数の引数。分類議論 vs 帰納法。判定不能 vs 反証された。どの対も踏んだ——不注意ではなく、概念自体の類似性のため滑り込みやすいのだ。

## どのような問題がissueを切って広く意見を募うに値するか

議論の中でissue#156を切った。タイトルは「統一の難題」。誤った例一つと存在しない分類一つが、「コミュニティの助けが必要」な問題を支えていた。中核的矛盾は偽の衝突であり、「判定不能+コンパイル時既知」というカテゴリーは元々存在しない——誤った例によって偽装されていたに過ぎない。答えは実は議論の後半で自然に浮かび上がっていた——**「統一」の方案は妥協ではなく、縮約である。「もしかしたら第三の場合があるかもしれない」と思う場所も、最終的には全て既存の経路に帰着した。**

issueを切って助けを求めることが間違いだというのではない。確認すべきは：本当にこの問題の本当の形を見たことがあるか？挙げた例は本当に自分が割り当てたカテゴリーに属するか？お前の矛盾は真の矛盾か、それとも「ある前提が成立していると思い込んでいるが、実は成立していない」のか？

多くの場合、「コミュニティの意見が必要」というのは答えが遠すぎるからではなく、**自分が生み出した分類に自ら迷い込んでいる**からだ。

## 本当に学んだこと

今回の議論は6時間続き、我々は9回自分を覆した。最終的な成果物は新しいassert APIではなく、釘付けにされた基盤である：`Never`は存在しなければならない（以前は誰も不在に気づかなかった）、精化型の結果は二値ではなく三値である（「判定不能」≠「反証」）、コンパイル時既知 ≠ 不変、実行時チェックはproveの弱い代替ではなく唯一soundな選択である、`assert(x > 0)`は「別のもの」ではなく値宇宙における`Assert`の別の運命である。assertとAssertは一つのものの両面であり——両者の違いはそれが何であるかではなく、その時点でコンパイラが計算できるか否かである。

**最良の設計は縮約であり、追加ではない。** 我々は「runtime assert機構」を追加したのではなく、それが元々`Assert`の一面であったことに気づいた。「判定不能の処理方案」を追加したのではなく、それが元々パイプラインに位置を持っていたことに気づいた。「特別な場合を追加する必要がある」と思うたびに、まず自問しよう：これは本当に新しいものなのか、それとも既に存在する概念を認識できていないだけなのか？

**分類は本質を見るのであって、自分がどうしたいかを見てはならない。** 我々は最初「コンパイラが何をすべきか」という軸で四つのUnprovenサブクラスに分けた——分割後に分類法が誤っていたことが判明した。「何をすべきか」は推論の結果であり、分類の根拠ではないからだ。正しい方法は「命題がコンパイル時に真の値を持つか否か」で分類すること——持てば持つ、持たねば持たない。二分類、きれいで、交差しない。
