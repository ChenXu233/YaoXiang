# match 基礎

`match`はYaoXiangの最も強力な制御フロー構造です。ある**値**の形に応じて、異なる処理パスを選択することができます。他の言語の`switch`を使ったことがあるなら、`match`はそれを包括的にアップグレードしたバージョンであることが分かるでしょう。

## 基本構文

言語仕様における`match`式の定義：

```
match Expr { MatchArm+ }
MatchArm : Pattern ('|' Pattern)* ('if' Expr)? '=>' Expr ','
```

分解すると：

- `match`の後にはマッチ対象の値が続く
- `{}`の中には1つ以上の**マッチアーム**(MatchArm)を含む
- 各マッチアーム：**パターン**の後に`=>`、次に**結果式**
- 各アームはカンマで終わる

最も簡単な例：

```yaoxiang
number = 2

text = match number {
    0 => "零",
    1 => "一",
    2 => "二",
}
println(text)  # "二"
```

## matchは式である

`if`と同様に、`match`も**式**であり、値を計算します。すべてのマッチアームの戻り値の型は一致している必要があります：

```yaoxiang
score = 85

grade = match score {
    90..100 => "A",    # 範囲パターン（上級者向けの内容）
    80..89 => "B",
    70..79 => "C",
    60..69 => "D",
    _ => "F",          # ワイルドカード：残りすべてにマッチ
}
println(grade)  # "B"
```

> **注意**：範囲パターン`90..100`などは上級者向けの内容であり、[パターンマッチ上級](../pattern-matching.md)で詳しく解説します。本章ではまず基礎パターンに焦点を当てます。

## 基本パターン

### リテラルパターン

具体的な値でマッチします：

```yaoxiang
response = 404

message = match response {
    200 => "OK",
    301 => "Moved",
    404 => "Not Found",
    500 => "Server Error",
    _ => "Unknown",
}
println(message)  # "Not Found"
```

### 識別子パターン

変数名を使ってマッチした値をキャプチャします：

```yaoxiang
result: Result(Int, String) = ok(42)

description = match result {
    ok(value) => "成功、値は: " + value.to_string(),
    err(error) => "失敗、理由: " + error,
}
println(description)  # "成功、値は: 42"
```

`ok(value)`の`value`は識別子パターンであり、`ok`でラップされた実際の値をキャプチャして、`=>`の後ろの式で使用することができます。

### ワイルドカードパターン

`_`はワイルドカードで、**あらゆる値**にマッチします。通常は最後に配置し、フォールバックとして使用します：

```yaoxiang
command = "exit"

action = match command {
    "start" => "サービスを開始",
    "stop" => "サービスを停止",
    "restart" => "サービスを再起動",
    _ => "未知のコマンド: " + command,
}
println(action)  # "未知のコマンド: exit"
```

## マッチは網羅的でなければならない

YaoXiangの`match`はすべての可能な場合をカバーすることを要求します。コンパイラが一部の可能な値を見落としていることを発見した場合、直接エラーを報告します。これが`match`の安全性の体現です。

```yaoxiang
# このコードはコンパイルに失敗します
# value = true
# result = match value {
#     true => "はい",
#     # falseブランチが欠けています——コンパイルエラー！
# }

# 正しい——_をフォールバックに使用
value = true
result = match value {
    true => "はい",
    _ => "いいえ",      # _がfalseも処理されることを保証
}
```

有限の種類しかないことが明確に分かっている場合（例えば列挙型にマッチする場合）、コンパイラが各バリアントをすべてカバーしているかをチェックしてくれます。これはブランチの書き忘れによるバグを防ぐ強力な仕組みです。

## 複数パターンの組み合わせ

1つのマッチアームは複数のパターンにマッチでき、`|`で区切ります：

```yaoxiang
day = "sunday"

type = match day {
    "monday" | "tuesday" | "wednesday" | "thursday" | "friday" => "平日",
    "saturday" | "sunday" => "休日",
    _ => "無効",
}
println(type)  # "休日"
```

## マッチアームは順番に実行される

`match`は最初のアームから順にマッチを試み、**最初にマッチが成功したブランチが有効**となり、後続のものは実行されません：

```yaoxiang
number = 5

result = match number {
    _ => "その他",     # ワイルドカードはすべてにマッチし、ここでマッチします
    5 => "五",         # 決して実行されません——上でマッチ済み
}
println(result)  # "その他"
```

この特性は、**ワイルドカード`_`を最後に配置する**ことが良い習慣であることを意味します。

## まとめ

| 要点 | 説明 |
|------|------|
| 構文 | `match 値 { パターン => 式, ... }` |
| 式 | `match`は値を計算し、すべてのブランチの型は一致する |
| リテラルパターン | 具体的な値を正確にマッチ：`200 => "OK"` |
| 識別子パターン | 値を変数にキャプチャ：`ok(value) => ...` |
| ワイルドカード`_` | あらゆる値にマッチし、フォールバックとして機能 |
| 網羅性 | すべての可能性をカバーする必要があり、コンパイラがチェックする |
| 複数パターン | `パターン1 \| パターン2 => 式` |
| 順次実行 | 上から下へ、最初にマッチしたブランチが有効 |
