# match の基本

`match` は YaoXiang のもっとも強力な制御フローの構造です。**値**の形状に応じて、異なる処理パスを選択することができます。他の言語の `switch` を使ったことがあれば、`match` が全面的にアップグレードされたバージョンだと分かるでしょう。

## 基本構文

文法仕様における `match` 式の定義：

```
match Expr { MatchArm+ }
MatchArm : Pattern ('|' Pattern)* ('if' Expr)? '=>' Expr ','
```

分解して見てみましょう：

- `match` の後に、マッチ対象の値が続きます
- `{}` の中にひとつ以上の **マッチアーム**（MatchArm）を含めます
- 各マッチアームは：**パターン** の後に `=>`、続いて**結果の式**を書きます
- 各アームはカンマで終わります

もっとも簡単な例：

```yaoxiang
number = 2

text = match number {
    0 => "零",
    1 => "一",
    2 => "二",
}
print(text)  // "二"
```

## match は式である

`if` と同じように、`match` も**式**であり、値を計算します。すべてのマッチアームの戻り値の型は一致している必要があります：

```yaoxiang
score = 85

grade = match score {
    90..100 => "A",    // 範囲パターン（応用編）
    80..89 => "B",
    70..79 => "C",
    60..69 => "D",
    _ => "F",          // ワイルドカード：残りのすべてにマッチ
}
print(grade)  // "B"
```

> **注意**：範囲パターン `90..100` などは応用編の内容であり、[パターンマッチング応用](../pattern-matching.md) で詳しく解説します。本章ではまず基礎的なパターンに焦点を当てます。

## 基礎パターン

### リテラルパターン

具体的な値でマッチします：

```yaoxiang
response = 404

message = match response {
    200 => "OK",
    301 => "Moved",
    404 => "Not Found",
    500 => "Server Error",
    _ => "Unknown",
}
print(message)  // "Not Found"
```

### 識別子パターン

変数名を使って、マッチした値をキャプチャします：

```yaoxiang
result: Result(Int, String) = ok(42)

description = match result {
    ok(value) => "成功，値: " + value.to_string(),
    err(error) => "失敗，原因: " + error,
}
print(description)  // "成功，値: 42"
```

`ok(value)` の `value` は識別子パターンです。これは `ok` が包んでいる実際の値をキャプチャし、`=>` の後の式で利用することができます。

### ワイルドカードパターン

`_` はワイルドカードであり、**あらゆる値**にマッチします。通常は最後に配置し、フォールバックとして使用します：

```yaoxiang
command = "exit"

action = match command {
    "start" => "サービス起動",
    "stop" => "サービス停止",
    "restart" => "サービス再起動",
    _ => "未知のコマンド: " + command,
}
print(action)  // "未知のコマンド: exit"
```

## マッチは網羅的でなければならない

YaoXiang の `match` はすべての可能性のあるケースをカバーすることを要求します。コンパイラが一部の可能な値を漏らしていると発見した場合、直接エラーを出します。これは `match` の安全性を体现したものです。

```yaoxiang
// このコードはコンパイルに失敗します
// value = true
// result = match value {
//     true => "はい",
//     // false の分岐が欠けています——コンパイルエラー！
// }

// 正しい書き方——_ でフォールバックする
value = true
result = match value {
    true => "はい",
    _ => "いいえ",      // _ が false も処理されることを保証します
}
```

有限のケースしかないことが明確である場合（たとえば列挙型をマッチする場合）、コンパイラはすべてのバリアントがカバーされているかをチェックしてくれます。これは分岐の書き漏れによるバグを防ぐ強力な仕組みです。

## 複数パターンの組み合わせ

ひとつのマッチアームで複数のパターンにマッチさせることができます。`|` で区切ります：

```yaoxiang
day = "sunday"

type = match day {
    "monday" | "tuesday" | "wednesday" | "thursday" | "friday" => "平日",
    "saturday" | "sunday" => "休日",
    _ => "無効",
}
print(type)  // "休日"
```

## マッチアームは順に実行される

`match` は最初のアから順にマッチを試み、**最初にマッチが成功した分岐が有効になり**、それ以降は実行されません：

```yaoxiang
number = 5

result = match number {
    _ => "その他",     // ワイルドカードはすべての値にマッチ、ここでマッチします
    5 => "五",         // 決して実行されません——すでに上でマッチしているため
}
print(result)  // "その他"
```

この特性は、**ワイルドカード `_` を最後に配置する**ことが良い習慣であることを意味します。

## まとめ

| 要点 | 説明 |
|------|------|
| 構文 | `match 値 { パターン => 式, ... }` |
| 式 | `match` は値を計算し、すべての分岐の型は一致する |
| リテラルパターン | 具体的な値を厳密にマッチ：`200 => "OK"` |
| 識別子パターン | 値を変数にキャプチャ：`ok(value) => ...` |
| ワイルドカード `_` | あらゆる値にマッチし、フォールバックとなる |
| 網羅性 | すべての可能性をカバーしなければならず、コンパイラがチェックする |
| 複数パターン | `パターン1 \| パターン2 => 式` |
| 順次実行 | 上から下へ、最初にマッチした分岐が有効になる |

> **次のステップ**：本文では `match` の基本的な使い方をカバーしました。より高度なパターン（ネストしたパターン、ガード式、構造体の分解など）については [パターンマッチング応用](../pattern-matching.md) を参照してください。
