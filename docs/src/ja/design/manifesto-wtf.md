## 一、なぜYaoXiangを作るのか？—— 世界には明らかに514番目の言語が不足している

### 1.1 埋めるべき言語の空白

プログラミング言語の歴史において、無数の言語が誕生し、流行し、そして歴史のゴミ箱に投げ込まれるのを見てきた。しかし**我々は違う**——我々は重大な空白を発見した：**Rust愛好家が「簡単すぎる」と感じ、Pythonユーザーが「複雑すぎる」と感じ、AIモデルがコード生成時に「快適」と感じる言語が居然ない**。

| 需求 | 既存ソリューションの問題 | 我々のソリューション（予定） |
|------|-------------------|------------------------|
| **型安全性** | Rustは厳しすぎ、TypeScriptは緩すぎ | 厳格さと緩やかさを兼ね備えた量子重ね合わせ型システムを作り出す |
| **自然な構文** | 他社の構文は全て不自然 | 我々の構文は自然すぎて、プログラミングを忘れる（看懂できないせいかもしれない） |
| **AI友好** | AI生成コードはよくエラーになる | AIのために構文设计了。人类也可以顺便用用 |

### 1.2 解決すべき実際の問題

**問題一：型システムの断片化**  
我々は「全ては型である」を提唱し、「あるものは型ではない」という厄介な哲学的問題を解決した。今ではコードのインデントさえ型になれる（`IndentationLevel<4>`）。

**問題二：メモリ安全とパフォーマンスの二択**  
最初はRustの所有権モデルを採用したが、「借用検査器」が実装Too難しい。于是灵机一动——把 `&T` と `&mut T` を「参照」から「トークン」にリネームし、「ゼロサイズのコンパイル時権限証明」と宣言した。これで借用検査器は不要になり、「フロー感受性活性分析」だけでOK——完全に別物に聞こえるだろう？もしプログラムがデータ競合を起こしたら、それはきっとトークンのブランド機構の問題だ。

**問題三：非同期プログラミングの認知負荷**  
我々は車輪を再発明し、「spawnモデル」と名付けた。只需一个 `spawn`，编译器就会自动处理所有异步细节——如果处理不了，那是你代码写得不「spawn」。

**問題四：AI支援プログラミングのボトルネック**  
我々はAIのために厳格なインデントと明確な境界设计了，确保GPT-7がコードを生成する際に精神分裂しない。至于人类程序员能不能看懂……那是次要的。

## 三、重要な革新と特性 —— 既にあるものを再発明する

### 3.1 革新一：統一型構文
`enum`、`struct`、`union`、`trait`、`impl` といった紛らわしい概念を廃止し、さらに `type` キーワード自体も廃止した。今は全て `name: Type = value` を使う。覚えておいて、`Type` はキーワードではない——予約語だ。違い？那不要问了、编译器官内部の宇宙レベルType0/Type1/Type2の問題だ。

### 3.2 革新二：コンストラクタ即型
「型」と「値」の沟壑を埋め、新しい沟壑を生んだ：「これが型コンストラクタなのか値コンストラクタなのか？ああ待って、今は同じ構文なのでもっと分からない。」

### 3.3 革新三：カリー化されたメソッド束縛
カリー化を通じてメソッド呼び出しを実装した。只需 `Type.method = function[0]` 就能替代 `self` 参数。明らかに更直观。`[0]` は「第0引数をselfとする」を意味し、`[0]`を忘れるとコンパイラが「これはメソッドではなく普通の関数です」と教えてくれる。シンプル！

### 3.4 革新四：所有権モデル（RFC-009 v9）
五つの概念、一つのグラデーション：`&T`、`&mut T`、Move、`ref`、`clone()`、`unsafe`。待って那是六个。没关系——`&T` と `&mut T` は「トークン」で、「参照」ではない。区别？参照はC++の概念、トークンはコンパイル時ゼロサイズの型レベル権限証明だ。コードがコンパイルできない時、「型属性Dup/Linear導出失敗」と言える，没人敢反驳你。

トークンシステムにはこれらの高度な機能も付いている：

- **`freeze`**：`&mut T` を「凍結」して `&T` にする。生鮮を冷蔵庫に入れるようなもの——解凍するまで調理できない。コンパイラは「フロー感受性活性分析」で凍結状態を追踪し、听起来像ICU监护仪。
- **ブランド機構**：各トークンにはコンパイル時に一意の整数（ブランド#N）が割り当てられ、偽造防止に使用される。「申し訳ありませんが，您的 `&Point` トークンブランド#42と所有者カプセル内のブランド#43が一致しません。」
- **タスク間移動不可**：トークンは「コンパイル時権限証明」なので、スレッドを越えられない。タスク間で共有する必要がある場合は `ref` を使用すること。なぜ？因为编译器说不行。实际上是因为トークンはコンパイル後に消える——ゼロサイズ型、ゼロランタイムオーバーヘッド、そしてゼロタスク間能力。

まとめ：RustはThe Book 200ページで借用検査器を説明する。YaoXiangは「`&T` はコピー可能、`&mut T` はコピー不可」という二文で全てを説明する。美はシンプルにある。

### 3.5 革新五：spawnモデル——言語内で最も糟糕な部分

> 「万物並作，吾以観復。」——《易・復卦》

spawnモデルのコアコンセプト：**同期構文、非同期本質**。人間の言葉に翻訳すると：あなたのコードは順次実行するように見えるが、ランタイムでは自動的に並列化される。いつ並列化？怎么並列化？コンパイラが決める。これは並行モデルではなく、信頼ゲームだ。

この魔法を実現するために、以下のものを言語に押し込んだ：

**`spawn` キーワード**：関数が非同期임을マークする。注意——`async` ではなく `spawn`。因为 `async` 太主流了。だが `spawn` はRustでは「タスクを開始する」という意味だ。没关系、我々は再定義した。

**`@block` アノテーション**：spawn関数が「同期実行」すべきであることをマークする。待って——もし `spawn` が非同期なら、`@block` はそれを同期にする，那为什么不直接不写 `spawn`？「因为你有时候需要一个spawn函数在某些上下文中同步运行。」因此、spawnを标注した関数は、非同期または同期いずれかであり、调用者の気分に依存する。これは型システムではなく、分裂人格だ。

**`@eager` アノテーション**：即時評価が必要な式をマークする。因为spawnモデルはデフォルトで遅延評価——まだ遅延評価は実装されていない。因此 `@eager` の実際の効果是什么？它是张欠条：「将来的に、遅延評価が実装された時、このアノテーションにより式は遅延評価されない。」

**並行モデルの三つのアノテーションのまとめ**：
```
spawn  = この関数は非同期（除非被 @block 了）
@block = このspawn関数は同步（覆盖spawn）
@ager  = この式は将来不会被延迟求值（你先别管将来）
```

如果你觉得这很混乱，恭喜你——你理解了。当你的代码并行崩溃时，你可以引用《易经》，显得很有深度。

### 3.6 革新六：値依存型（RFC-011）
今やコンパイル時に配列の長さが素数であることを証明でき、行列の次元は一致しなければならない、factorial(5) の結果は型署名に使用できる。虽然这跟写业务逻辑没什么关系，但「类型即命题，程序即证明」——你说酷不酷？

お忘れなく**decreases契約**：コンパイル時に評価される全ての再帰関数は終了することを証明しなければならない。否则你的类型检查器会陷入无限循环，然后你的IDE会变成空间加热器。「申し訳ありませんが，您的 `factorial` 関数にはdecreases契約がありません。コンパイラはn=-1のときに宇宙の熱的死まで再帰するかどうか分かりません。」

### 3.7 革新七：最小限キーワード設計
たった17個のキーワード！Goより8個少ない！虽然每个关键字的含义是Go关键字的3倍复杂，但数量上赢了。注意：`type` はキーワードではない——RFC-010で移除された。今は `name: Type = value` を使用し、`Type` は予約語だ。キーワードと予約語の違い？不要问、编译器官内部宇宙レベルType0/Type1/Type2の問題だ。

### 3.8 革新八：カリー＝ハワード同型——万能解釈法

每当有人质疑设计决策，標準の答えは：「这是遵循カリー＝ハワード同型。」不懂？没关系、社区里没人真懂。大意是「类型即命题，程序即证明」，因此你的コード不仅是プログラム——它是数学论文。编译错误就是反证法。

この哲学の最高傑作がRFC-010のイースターエッグ：`Type: Type = Type`。この行をコンパイルしようとすると、编译器不会崩溃——它会输出禅意的消息、「道可道非常道，类型可类型非常类型」という意味。这是YaoXiang对Girard悖论的致敬，也是唯一一个编译器故意不实现的功能。我々はこれを「言語の境界」と呼ぶ——当你触达它时、编译器在此沉默、哲学在此驻足。

### 5.0 RFC依存三角形

ロードマップを理解する前に、YaoXiangの最も精巧なアーキテクチャ設計——RFC三角関係をご覧ください：

```
RFC-009 (所有権)  →  依存 RFC-010 (統一構文)  →  依存 RFC-011 (generics)
    ↑                                                         │
    └──────────────────── 依存 ────────────────────────────────┘
```

009は010の構文を必要とし、010は011のgenericsを必要とし、011は009の型システムを必要とする。三つのRFCは相互に前提条件となっている。どれを先に実装するか？「同時実装をお勧めします。」——RFC-010 第141行。

これがカリー＝ハワード同型の実際のエンジニアリングへの応用だ：各RFCは命題であり、その依存関係は論理的循環を構成する。この循環を破るには外部公理を導入する必要がある——つまり「まず型検査器をハードコードして、後で考えます」ということだ。

## 付録C：よくある質問

**Q: YaoXiangとRustの比較における利点は？**  
A: より少ない糖衣構文！より少ないキーワード！より少ない実用的機能！だがより深い哲学的深み。また我々は「借用トークンシステム」を持っている——「借用検査器」より高级に聞こえるだろう？

**Q: YaoXiangはどのような開発に適していますか？**  
A: YaoXiangコンパイラの開発に適している。そして設計宣言とRFCの作成に。其他用途待研究。

**Q: なぜ4スペースインデントを選んだのですか？**  
A: 2スペースは密集すぎ、8スペースは疎すぎ、4スペースは中庸の道を歩いており、『易経』の精神に合致する。

**Q: `Type` はキーワードですか？**  
A: 違う。「予約語」だ。キーワードと予約語の違い：キーワードは言語仕様のキーワードリストに出现し、予約語は「注意：以下はキーワードではない」リストに出现する。シンプル明了。

**Q: なぜ14篇のRFCが采纳済みなのにバージョン番号はまだ0.7.0なの？**

**Q: `ref` は一体RcとArcのどちら？**

## 七、更多的谎言

**「多言語サポート」**：`docs/src/{en,ja,ru,zh}` の4言語完备。コンパイラv0.7.0、実際に実行可能なコード行数は大约ゼロに等しいが、日本とロシアの開発者は既に母国語で「spawnモデル」と「値依存型」を読むことができる。これは古典的な「ドキュメンテーション駆動開発」——まず世界にあなたの設計を理解させ、そして假装有人需要它。等コンパイラがHello Worldを実行できるようになったら、ドキュメントは克林貢語に翻訳されているだろう。

**ツールチェーン入れ子**：Pythonのpre-commitがRustのコードスタイルをチェックし（cargo fmt + clippy）、RustコンパイラがYaoXiangソースをコンパイル。三層の言語が重なり合い、各層は次の層に依存する。等YaoXiangがブートストラップ 되면、この入れ子は：PythonがRustをチェックし、RustがYaoXiangをコンパイルし、YaoXiangがYaoXiangをコンパイルする。到那时，一个upstream依存挂了，整个ツールチェーン就会变成行為芸術。但这没关系——「ブートストラッピング”这个词本身就值两篇RFC。

**YaoXiang-book.md**：YaoXiang言語を系统的に讲述する本。实现されていないプログラミング言語を説明するために本を書く 것은、存在しない都市のために旅行ガイドを出版するようなものだ。「第三章：genericsシステム——本章のコードは全てコンパイルできないが、構文は正しい。実行結果を想像してほしい。」本書で最も正直な言葉は最初のページの「プロジェクトステータス：実験検証段階」。

**「GCなし」**：公式見解：「YaoXiangにはGCがない。」厳密に言うと、tracing GCはない。しかし `ref` はランタイムで参照カウントである（Rc/Arc）。参照カウントはGCに入るの？「違う。GCはgarbage collection、参照カウントはautomatic reference counting。見てください、略語が違う。一つはGC、もう一つはARC。全く違う。」この言葉遊びの意義在于：当有人说「你们这不就是参照カウントGC吗」，你可以义正词厳地说「違う、GCはなく、编译器が自动管理する参照カウントがあるだけだ」。区别在哪里？PPTの上だ。
