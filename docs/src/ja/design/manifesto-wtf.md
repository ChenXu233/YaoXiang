## 一，为什么創造 YaoXiang？—— 世界には明らかに514番目の言語が不足している

### 1.1 埋めるべき言語の空白

プログラミング言語の歴史の流れにおいて，无数の言語が誕生し流行し，そして歴史のゴミ箱に投げ込まれるのを見てきた。しかし**我々は違う**——我々は敏锐に気づいた，空前の空白を：**，居然没有一门语言能同时让 Rust 爱好者觉得太简单、让 Python 用户觉得太复杂、并且让 AI 模型在生成代码时感到"舒适"**。

| ニーズ | 既存のソリューションの問題点 | 我々のソリューション（予定） |
|------|-------------------|------------------------|
| **型安全性** | Rust は厳しすぎる、TypeScript は緩すぎる | 我々は厳格でありかつ緩やかな量子重ね合わせ状態型システムを作り出す | |
| **自然な構文** | 他社の構文は全て不自然 | 我々の構文は自然すぎて、プログラミングを忘れるほど（看不懂の可能性もある） | |
| **AI 友好** | AI 生成コードはよく間違える | 我々は AI のために構文を設計し、人間はとりあえず使えるようにする | |

### 1.2 解決すべき実際の問題

**問題一：型システムの断片化**  
我々は「全ては型である」を提唱し，「一部のものは型ではない」という困惑させる哲学的問題を解決した。今ではコードのインデントさえ型にできる（`IndentationLevel<4>`）。

**問題二：メモリ安全とパフォーマンスの二者択一**  
我々は Rust の所有権モデルを採用したが、あの「鬱陶しい」コンパイルエラーを取り除いた。プログラムがデータ競合を起こしたら、それはハードウェアの問題である。

**問題三：非同期プログラミングの認知負荷**  
我々は車輪の再発明を行い，「スポーンモデル」と名付けた。只需一个 `spawn`，コンパイラが自動的に全ての非同期詳細を処理する——処理できなければ、それはあなたのコードが「スポーン」的じゃないからだ。

**問題四：AI 補助プログラミングのボトルネック**  
我々は AI のために厳格なインデントと明確な境界線を設計し，GPT-7 がコードを生成する時に精神分裂を起こさないよう確保した。至于人类程序员能不能看懂……那是次要的。

### 1.3 言語の哲学的根底

YaoXiang の名前は『易経』に由来し，これにより技術議論においてオカルト buff が自带される。コードがコンパイルできない時、こう言える：「これは陰陽未調なり、卦を立てて見ん。」

## 二、核心哲学と原則 —— 疑問の余地なき聖典

### 2.1 原則一：全ては型である
**妥協不可の理由**：これで型理論で全てを説明できる、なぜプロジェクト進捗がいつも遅延するかも含む。

### 2.2 原則二：厳格な構造化
**妥協不可の理由**：4スペースインデントは宇宙の真理である。Tab を使う 사람은火星に追放されるべきだ。

### 2.3 原則三：ゼロコスト抽象化
**妥協不可の理由**：我々の抽象化レイヤーは7層あるが、「ゼロコスト」なので、パフォーマンスは手書きアセンブリと同程度のはず……理論的には。

### 2.4 原則四：デフォルトで不変
**妥協不可の理由**：可変性は万悪の根源である。変数を変更する必要があるなら、それはあなたの設計が間違っている。

### 2.5 原則五：型即ちデータ
**妥協不可の理由**：これで実行時に型をチェックでき、そして……コンパイル時に既にチェック済みだったことに気づく。

## 三、重要な革新と特性 —— 既にあるものを再発明する

### 3.1 革新一：統一型構文
我々は `enum`、`struct`、`union` といった困惑させるコンセプトを廃止し、统一用 `type`。今、只需记住 17 个关键字，而不是 18 个。**进步！**

### 3.2 革新二：コンストラクタ即ち型
「型」と「値」の溝を埋め，新的沟壑：「これは型コンストラクタか値コンストラクタか？」

### 3.3 革新三：カリー化されたメソッドバインディング
我々はカリー化によりメソッド呼び出しを実現し，`obj.method(_)` で `obj.method()` の代わりに使える。显然更直观。

### 3.4 革新四：スポーンモデル
> 「万物並作，吾以觀復。」——《易·復卦》

代码并行崩溃时、这句话を引用すれば，显得很有深度。

### 3.5 革新五：依存型サポート
今、コンパイル時に配列の長さが素数であることを証明できる。虽然这跟写业务逻辑没什么关系，但很酷。

### 3.6 革新六：極少キーワード設計
わずか17個のキーワード！Go より8個少ない！虽然每个关键字的含义是 Go 关键字的 3 倍复杂，但数量上我々が勝った。
