## 一，为什么创造 YaoXiang？—— 世界には明らかに514番目の言語が不足している

### 1.1 埋めるべき言語の空白

プログラミング言語の歴史の長い川において，无数の言語が誕生し流行し，然后扔进历史的垃圾桶堆積されていく。だが**我々は違う**——我々は敏锐地发现了一个惊天空白：**竟然没有一门语言能同时让 Rust 爱好者觉得太简单、让 Python 用户觉得太复杂、并且让 AI 模型在生成代码时感到“舒适”**。

| ニーズ | 既存のソリューションの問題点 | 我々のソリューション（予定） |
|------|-------------------|------------------------|
| **型安全性** | Rust は厳しすぎ，TypeScript は緩すぎ | 我々は厳しくて緩い量子重ね合わせ状態型システムを作り出す |
| **自然な構文** | 他社の構文はすべて不自然 | 我々の構文は自然すぎて自分がプログラミングしていることを忘れる（看不懂のかもしれない） |
| **AI フレンドリー** | AI 生成コードはよく間違う | 我々は AI のために構文を設計し，人間は顺便使った |
| **型安全** | Rust は厳しすぎ、TypeScript は緩すぎ | 厳しさと緩やかさの量子重ね合わせ状態型システムを生み出す |
| **自然な構文** | 他社の構文はすべて不自然 | 自然にすぎて自分がプログラミングしていることを忘れる（看不懂かもしれない） |
| **AI フレンドリー** | AI 生成コードはよく間違う | AI のために構文を設計し、人間は顺便利用できればいい |

### 1.2 解決する实际问题

**問題一：型システムの断片化**
我々は「一切皆型」を提唱し，「あるものは型ではない」という困扰な哲学的問題を解決した。今ではコードのインデントすら型になれる（`IndentationLevel<4>`）。

**問題二：メモリの安全性と性能の二択**
我々は Rust の所有権モデルを採用したが，あの「烦人」なコンパイルエラーを取り除いた。プログラムがデータ競合を起こすなら，それはハードウェアの問題だ。

**問題三：非同期プログラミングの認知負荷**
我々は車輪を再発明し，それに「並作モデル」という名前を付けた。只需一个 `spawn`，コンパイラは自動的にすべての非同期詳細を処理する——処理できなければ，それはあなたのコードが「並作」的でないからだ。

**問題四：AI 支援プログラミングのボトルネック**
我々は AI のために厳密なインデントと明確な境界を设计し，确保 GPT-7 がコードを生成する際に精神分裂症にならない。人間のプログラマーが理解できるかどうかは……副次的な問題だ。

### 1.3 言語の哲学的基盤

YaoXiang の名前は『易経』に由来し，これにより技術的議論において自動的に神秘学バフが付与される。コードがコンパイルできない時，「これは陰陽未調，我一卦引いてみましょう」と言える。

## 二、核心哲学と原則 —— 容赦のない聖典

### 2.1 原則一：一切皆型
**妥協できない理由**：これで型論を用いてすべてを説明できる，包含なぜプロジェクト進捗が常に遅延するかも。

### 2.2 原則二：厳格な構造化
**妥協できない理由**：4スペースインデントは宇宙の真理だ。Tab を使う人間は火星に追放されるべきだ。

### 2.3 原則三：ゼロコスト抽象化
**妥協できない理由**：我々の抽象化レイヤーは7層あるが，「ゼロコスト」なので，理論的には手書きアセンブリと同程度の性能はずだ……。

### 2.4 原則四：デフォルトで不変
**妥協できない理由**：可変性は万悪の根源だ。変数を変更する必要があるなら，それはあなたの設計が間違っている。

### 2.5 原則五：型即データ
**妥協できない理由**：これで実行時に型をチェックでき，そして……コンパイル時にすでにチェック済みだったことに気づく。

## 三、重要な革新と特性 —— 既发明済みのものを再发明

### 3.1 革新一：統一型構文
我々は `enum`、`struct`、`union` といった困惑させる概念を廃止し，`type` で統一した。只需记住 17 個关键字，不再是 18 個。**進歩！**

### 3.2 革新二：コンストラクタ即型
「型」と「値」の溝を消除し，新しい溝を作り出した：「これは型コンストラクタなのか値コンストラクタなのか？」

### 3.3 革新三：カリー化されたメソッド束縛
我々はカリー化によりメソッド呼び出しを実現し，`obj.method(_)` で `obj.method()` の代わりに使える。显然更加直观。

### 3.4 革新四：並作モデル
> 「万物並作，吾以観復。」——《易・復卦》

あなたのコードが並行的にクラッシュする時，あなたは这句话を引用でき，深みがある，显得很有深度。

### 3.5 革新五：依存型サポート
 이제编译时可以证明数组长度是否为质数。虽然これ跟业务逻辑没关系，但是很酷。

### 3.6 革新六：极简关键字设计
17 個キーワードだけ！Go より 8 個少ない！虽然各关键字的含义是 Go 关键字的 3 倍复杂，但数量上赢了。
