## 一，为什么创造 YaoXiang？

### 1.1 填补的语言空白

编程语言の歴史において、私たちは无数の優秀な言語の誕生と進化を目の当たりにしてきた：C 言語はシステムプログラミングの効率革命をもたらし、Python は誰もが学べるプログラミング体験を作り上げ、Rust はメモリ安全とパフォーマンスの両立を証明し、TypeScript は大規模フロントエンドプロジェクト保守を可能にした。しかし、今日の言語エコシステムを見渡すと、依然として明確なギャップが存在する——**以下の3つのコアニーズを同時に満たす言語は存在しない**：

| ニーズ | 既存のソリューションの問題点 |
|--------|-------------------|
| **型安全** | Rust は厳しすぎて、学習曲線が険しい；TypeScript はオプショナル型で、コンパイル時の保証を提供できない |
| **自然な構文** | Rust の構文は複雑で晦渋；Haskell の関数型は参入障壁が高い；伝統的な静的言語は冗長で煩雑 |
| **AI フレンドリー** | 既存の言語は構文の曖昧さが多く、AST が複雑、隠された動作が予測困難で、AI によるコード生成と修正の精度が制限される |

YaoXiang の誕生は、この空白を埋めるためである。私たちは信じている：**プログラミング言語は、強力でかつ親しみやすく、安全でかつ効率的で、厳密でかつ优美であるべきだ**。

### 1.2 解决的实际问题

**問題一：型システムの断片化**

現代のプログラミング言語は、型システムにおいて深刻な断片化を示している。静的型付け言語はコンパイル時の絶対的な正確さを追求するが、しばしば開発効率の犠牲となる；動的型付け言語は柔軟性を提供するが、大規模プロジェクトでは保守困難な欠点を露呈する。YaoXiang は「すべては型である」という統一抽象フレームワークを提案し、型を言語設計の主軸として贯彻し、後付けのパッチではなくする。

**問題二：メモリ安全とパフォーマンスの二者択一**

長い間、開発者はメモリ安全と実行パフォーマンスの間で艰难な選択を迫られてきた。GC（ガベージコレクション）は開発者を解放するが、遅延波动とメモリオーバーヘッドをもたらす；手動メモリ管理は効率的だが、綱渡りのように危険である。YaoXiang は Rust スタイルの奥有所有権モデルを採用し、コンパイル時にデータ競合とメモリリークを排除的同时に、ゼロコスト抽象を維持し、GC なしで高性能を実現する。

**問題三：非同期プログラミングの認知的負担**

現代応用はネットワークと並行処理离不开だが、非同期プログラミングは常に程序员の噩梦である。コールバック関数のネスト、Promise のチェーン呼び出し、async/await 構文——各方案はコードの複雑性を增加させる。YaoXiang は非同期モデルを重新設計した：関数シグネチャの後に `spawn` マークを付けるだけで、コンパイラが自動的にすべての非同期詳細を処理し、並行プログラミングを同期コードと同様に自然にする。

**問題四：AI 辅助编程のボトルネック**

AI が开发者を支援してコードを書くようになった今、言語設計の選択が非常に重要になる。曖昧な構文ルール、暗黙の型変換、複雑な構文糖——これらの人类程序员が既に慣れている特性は、AI の理解和生成の障碍となる。YaoXiang は設計当初から「AI フレンドリー」をコア目標とした：厳密なインデントルール、明確なコードブロック境界、曖昧さのない構文構造により、AI が正確に理解し、生成し、修正できるようにする。

### 1.3 语言的哲学根基

YaoXiang の名前は『易経』の「爻」と「象」に由来する。「爻」は卦象を構成する基本記号であり、陰陽の変化、動静の相생을象徴する；「象」は事物の本質の外在的表現であり、万象万物を包括する。

この哲学思想は言語設計の各 세부적 사항に体现される：

- **統一性**：爻卦の単純な記号が複雑な卦象を構成するごとく、YaoXiang は少数のコアコンセプト（型、関数、コンストラクタ）で完全なプログラミングモデルを構築する
- **階層性**：象には先天と後天の区別があるごとく、YaoXiang の型システムは明確な階層構造を持ち、原型から原型へ、値から元型へと続く
- **変化性**：陰陽の流れのごとく変化无尽であるごとく、YaoXiang は依存型をサポートし、型が値に応じて進化することを許可する
- **認識可能性**：卦象は解読でき、万物は象ることができるごとく、YaoXiang は完全な型リフレクション能力を提供し、実行時の型情報が完全に利用可能である
- **証明可能性**：卦象が事物の法則を明らかにするごとく、YaoXiang の型システムは Curry-Howard 同型に従い（型即ち命題、プログラム即ち証明）、型チェックのプロセスは論理証明の検証である

## 二、核心哲学与原则

以下の設計信条は YaoXiang の基石であり、**不可侵で、違反できない**。任何特性提案はこれらの原則の检验を経なければならない。

### 2.1 原则一：一切皆类型

YaoXiang の世界観において、型は最も高层の抽象ユニットであり、言語を貫くコアコンセプトである。

**具体的な体现**：

- **値は型のインスタンス**：`42` は `Int` 型のインスタンス、`"hello"` は `String` 型のインスタンス
- **型自体が型である**：`Type` は言語唯一の元型キーワードであり、`Int` の型は `Type` である
- **関数は型マッピング**：`add: (a: Int, b: Int) -> Int` は `Int × Int` から `Int` への型マッピングを記述する
- **モジュールは型の組み合わせ**：モジュールは関数と型を含む名前空間組み合わせである

**不可侵の理由**：統一型抽象は言語意味論を簡素化し、値と型の二元対立を消除し、型システムをコードの正しさの守護者而不是絆脚石にする。

### 2.2 原则二：严格结构化

YaoXiang の構文設計は「曖昧さのない、予測可能な、易しい解析」を追求する。

**具体的なルール**：

- **强制 4 スペースインデント**：Tab 文字の使用を禁止し、コードブロック境界が一目でわかる
- **括弧は省略不可**：関数引数は括弧が必要、リスト要素にはカンマが必要
- **コードブロックには波括弧が必要**：`if`、`while`、`for` などの制御フローは `{ }` で囲む必要がある
- **キーワード数を厳選**：17 のコアキーワードのみを維持し、構文糖の氾濫を拒否する

**不可侵の理由**：厳格な構造化は3つの主要な優位性をもたらす——（1）IDE の構文ハイライトとコードフォールドがより正確；（2）AI のコード生成と修正の精度が大幅に向上；（3）新規学習者がコード構造を迅速に理解できる。

### 2.3 原则三：零成本抽象

高层の抽象は実行時のパフォーマンスオーバーヘッドをもたらすべきではない。

**具体的な保証**：

- **単態化**：ジェネリクス関数はコンパイル時に具体バージョンに展開され、仮想テーブル参照オーバーヘッドなし
- **インライン最適化**：単純関数は自動的にインライン化され、関数呼び出しオーバーヘッドを消除
- **スタック配置優先**：小オブジェクトはデフォルトでスタック配置、ヒープ配置は必要な場合のみ使用
- **GC なし**：所有権モデルがメモリ安全を保証し、ガベージコレクタの実行時オーバーヘッドが不要

**不可侵の理由**：パフォーマンスはプログラミング言語の生存底线である。便利さを得るためにパフォーマンスを犠牲にする設計はすべて、程序员への裏切りである。

### 2.4 原则四：默认不可变

可変性は複雑性と相伴っている。YaoXiang はデフォルトで不変を選択し、コードをより推論しやすく、理解しやすくする。

**具体的なルール**：

- 変数はデフォルトで不変であり、代入後は変更できない
- 可変が必要な場合は明示的に `mut` を宣言する必要がある
- 参照はデフォルトで不変であり、可変参照には `mut` マークが必要
- 所有権の移動は元のバインディングの無効を意味する

**不可侵の理由**：不変性は並行処理安全性の基盤であり、コード可読性の保障であり、関数型プログラミングの知恵の結晶である。

### 2.5 原则五：类型即数据

型情報はコンパイル時のみ存在するべきではなく、実行時に完全に利用可能であるべきだ。

**具体的な能力**：

- 実行時型クエリ：任何値からその型情報を取得できる
- 型リフレクション：型自体を構成して操作できる
- パターン照合デコンストラクション：型コンストラクタは直接パターン照合に使用できる
- ジェネリクストレート：実行時にジェネリックパラメータの具体化型を取得できる

**不可侵の理由**：完全な型リフレクション能力はメタプログラミングの基盤であり、高性能フレームワークとツールの基石である。

## 三、关键创新与特性

YaoXiang は既存の言語の優秀な特性を吸收的同时に、以下の革新的設計を提案する。

### 3.1 创新一：统一类型语法

**従来の言語の型定義**は往往して複数のキーワードを必要とする：

```rust
// Rust
struct Point { x: f64, y: f64 }
enum Result<T, E> { Ok(T), Err(E) }
enum Color { Red, Green, Blue }
trait Drawable { fn draw(&self, s: &Surface); }
```

**YaoXiang の統一構文**：すべては `name: type = value` で、`Type` が唯一の元型キーワードである。

```yaoxiang
# === 記録型 ===

Point: Type = {
    x: Float,
    y: Float,
}

# デフォルト値を持つフィールド
Point3D: Type = {
    x: Float = 0,
    y: Float = 0,
    z: Float = 0,
}

# === ジェネリック型 ===

Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self,
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,
}

# === インターフェース（フィールドがすべて関数型の記録） ===

Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect,
}

Serializable: Type = {
    serialize: () -> String,
}

# === インターフェース実装（インターフェース名は型体内に入る） ===

Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable,
}

# === メソッド（Type.method 構文） ===

Point.draw: (self: &Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}
```

**革新的価値**：`fn`、`struct`、`enum`、`trait`、`impl` キーワードの断片がない——統一構文ですべての宣言をカバー。

### 3.2 创新二：コンストラクタ即ち型

**値構築と関数呼び出しは完全に同じ**：

```yaoxiang
# 型定義
Point: Type = { x: Float, y: Float }
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self,
}

# 値構築：関数呼び出しと同じ
p: Point = Point(3.0, 4.0)
opt: Option(Int) = Option.some(42)
none: Option(Int) = Option.none()

# パターン照合：直接デコンストラクション
match opt {
    Option.some(value) -> print(value)
    Option.none -> print("nothing")
}
```

### 3.3 创新三：カリー化されたメソッドバインディング

YaoXiang は純粋関数型設計を採用し、カリー化を通じてオブジェクトメソッド呼び出しのような構文糖を実現し、`class` と `method` キーワードを導入する必要がない。

```yaoxiang
# === 型定義 ===

Point: Type = {
    x: Float,
    y: Float,
}

# コア関数：ユークリッド距離
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    return (dx * dx + dy * dy).sqrt()
}

# メソッド構文糖バインディング（[0] は第 0 引数位置へのバインディングを示す）
Point.distance = distance[0]

# === 使用 ===

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# 2 つの呼び出し方式是完全に同値
d1 = distance(p1, p2)     # コア関数を直接呼び出す
d2 = p1.distance(p2)      # メソッド構文糖

# カリー化用法
dist_from_p1 = p1.distance  # 部分適用、第 2 引数を待つ
d3 = dist_from_p1(p2)       # 2.828
```

**革新的価値**：純粋関数型設計で隠し `self` パラメータがなく、関数は値として自由に 전달と組み合わせが可能。

### 3.4 创新四：並作モデル

> 「万物並作，吾以観復。」——《易・復卦》
>
> 並作モデルはここに寓意し、以下のプログラミングパラダイムを記述する：開発者は同期的で順序的な思考でロジックを記述し、言語ランタイムは計算ユニットを万物並作のごとく自動的に効率的に並行実行させ、最終的に統一的に協調する。

**コア3原則**：

| 原則 | 説明 |
|------|------|
| **同期構文** | 目に見える通りの順序コード |
| **並行本質** | 実行時に自動的に並行性を抽出 |
| **統一協調** | 結果は必要な時点で自動的に集約され、ロジックの正しさを保証 |

**用語体系**：

| 公式用語 | 対応構文 | 解説 |
|----------|----------|------|
| **並作関数** | `spawn (params) => body` | 並作実行に参加できる計算ユニットを定義 |
| **並作ブロック** | `spawn { a(), b() }` | 明示的に宣言された並行領域、ブロック内のタスクは並作実行 |
| **並作ループ** | `spawn for x in xs { ... }` | データ並列、ループボディがすべての要素で並作実行 |
| **並作値** | `Async(T)` | 並作中の未来値、使用時に自動的に待機 |
| **並作グラフ** | 遅延計算グラフ（DAG） | 並作が発生するステージ、依存関係と並行関係を記述 |
| **並作スケジューラ** | ランタイムタスクスケジューラ | 万物を調整し、正しいタイミングで並作させる知的中枢 |

> **詳細**： [RFC-001 並作モデル](./rfc/001-concurrent-model-error-handling.md)

```yaoxiang
# === 並作関数 ===
# spawn マーク付きの関数
fetch_data: (url: String) -> JSON spawn = {
    return HTTP.get(url).json()
}

# === 並作ブロック ===
# spawn { } 内の式は強制的に並行実行
compute_all: () -> (Int, Int, Int) spawn = {
    (a, b, c) = spawn {
        heavy_calc(1),    # タスク 1
        heavy_calc(2),    # タスク 2
        another_calc(3)   # タスク 3
    }
    return (a, b, c)
}

# === 自動待機 ===
main: () -> Void = {
    # 2 つの独立リクエストは自動的に並行実行
    users = fetch_data("https://api.example.com/users")
    posts = fetch_data("https://api.example.com/posts")

    # 待機点は結果が必要な時点で自動的に挿入
    print(users.length + posts.length)  # users と posts を自動的に待機
}
```

**スレッド安全性**：

```yaoxiang
# ref キーワードはスレッド安全性を自動的に処理（コンパイラが自動的に Rc/Arc を選択）
main: () -> Void = {
    counter = ref SafeCounter(0)

    # タスク間共有：コンパイラが自動的に Arc を選択
    spawn {
        counter.increment()
    }
    spawn {
        counter.increment()
    }
}
```

**技術文書**：

- 詳細 [RFC-001 並作モデル](./rfc/accepted/001-concurrent-model-error-handling.md)

**革新的価値**：非同期プログラミングの認知的負担がゼロになり、コード可読性は同期コードと完全に同じでありながら、高性能並列の実行効率を得る。

### 3.5 创新五：値依存型（RFC-011）

> **ステータス**：設計中、部分実装

型は値に依存でき、真の型駆動開発を実現する。

```yaoxiang
# 行列型：次元はコンパイル時に確定
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
}

# コンパイル時計算：factorial(3) = 6
vec: Vec(factorial(3)) = Vec(6)()

# コンパイル時次元検証
identity_3x3: Matrix(Float, 3, 3) = identity(Float, 3)(3)
# multiply(matrix_2x3, matrix_4x2)  # コンパイルエラー：次元が不整合
```

**革新的価値**：コンパイル時にもっと多くのエラーを捕捉し、より正確な型保証を実現。

### 3.6 创新六：极简关键字设计

YaoXiang はわずか 17 のコアキーワードのみを定義し、主要言語よりはるかに少ない：

```
pub    use    spawn
ref    mut    if     elif
else   match  while  for    return
break  continue as     in     unsafe
```

| 比較対象言語 | キーワード数 |
|----------|-----------|
| YaoXiang | **17** |
| Rust | 51+ |
| Python | 35 |
| TypeScript | 64+ |
| Go | 25 |

**革新的価値**：より低い記憶負担、より一貫した構文スタイル、より解析しやすい構文構造。

### 4.2 类型定义与函数

```yaoxiang
# 統一型構文：name: type = value

# 記録型
Point: Type = { x: Float, y: Float }

# ジェネリック型
Option: (T: Type) -> Type = {
    some: (T) -> Self,

# インターフェース型（フィールドがすべて関数の記録）
Serializable: Type = {
    serialize: () -> String,

# 関数定義
add: (a: Int, b: Int) -> Int = a + b

# ジェネリック関数
identity: (T: Type) -> ((x: T) -> T) = x
```

### 4.3 模式匹配

```yaoxiang
# パターン照合
classify: (n: Int) -> String = {
    return match n {
        0 -> "zero",
        1 -> "one",
        _ if n < 0 -> "negative",
        _ -> "positive",
    }
}

# デコンストラクションktor パターン
Point: Type = { x: Float, y: Float }
match point {
```

### 4.4 所有权模型（RFC-009 v9）

```yaoxiang
Point: Type = { x: Float, y: Float }

# デフォルト Move（ゼロコピー）
p1 = Point(1.0, 2.0)
p2 = p1              # Move、p1 はこれ以上読めない

# &T / &mut T トークン（コンパイル時ゼロオーバーヘッド）
p2.print()           # コンパイラが自動的に &Point トークンを作成
p2.shift(1.0, 1.0)  # コンパイラが自動的に &mut Point トークンを作成

# ref：共有所有（コンパイラが自動的に Rc/Arc を選択）
shared = ref p2      # スコープ間共有

# clone()：明示的なディープコピー
backup = p2.clone()

# unsafe + ポインタ：システムレベル
unsafe {
    ptr: *Point = &p2
    (*ptr).x = 0.0
}
```

**所有権梯度**：
```
&T / &mut T    Move       ref        clone()    unsafe
    |             |          |           |          |
借用トークン     デフォルト    共有所有     ディープコピー  ポインタ
ゼロコスト       ゼロコピー    自動Rc/Arc   明示的     システムレベル
```

### 4.5 错误处理

```yaoxiang
# Result 型
Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,
}

divide: (a: Float, b: Float) -> Result(Float, String) = {
    if b == 0.0 {
        return Result.err("Division by zero")
    }
    return Result.ok(a / b)
}

# match で処理
result = divide(10.0, 2.0)
match result {
    Result.ok(value) -> print(value),
    Result.err(msg) -> print("Error: ${msg}"),
}
```

## 五、路线图与待定项

### 5.1 已决定的设计决策

以下の決定は十分な議論と審阅を経たものであり、**もはや変更を受け入れない**：

| モジュール | 決定 | 説明 |
|------|------|------|
| **型システム** | すべては型 | 値、関数、モジュール、ジェネリクスはすべて型 |
| **型構文** | 統一 `name: type = value` | 1 つの宣言形式ですべてをカバー、`Type` が唯一の元型キーワード |
| **キーワード** | 17個のコアキーワード | `type`/`fn`/`struct`/`enum`/`trait`/`impl` を含む |
| **関数構文** | シグネチャ + 式 | `name: (params) -> ReturnType = body` |
| **メソッドバインディング** | RFC-004 カリー化バインディング | `Type.method = function[position]` |
| **非同期モデル** | 並作モデル | `spawn` マーク、遅延評価、自動並列 |
| **メモリ管理** | 所有権モデル（RFC-009 v9） | Move + &T/&mut T トークン + ref + clone + unsafe、GC なし |
| **ファイル即モジュール** | モジュールシステム | 各 `.yx` ファイルは1つのモジュール |
| **メイン関数** | `main: () -> Void` | プログラムエントリー |
| **スレッド安全性** | ref が自動的に Rc/Arc を選択 | コンパイラのエスケープ解析、ユーザーに見えない |

## 付録A：言語クイックリファレンス

### A.1 キーワード

| キーワード | 役割 |
|--------|------|
| `pub` | 公開エクスポート |
| `use` | モジュールインポート |
| `spawn` | 並作マーク |
| `ref` | 共有所有（コンパイラが自動的に Rc/Arc を選択） |
| `mut` | 可変変数 |
| `if/elif/else` | 条件分岐 |
| `match` | パターン照合 |
| `while/for` | ループ |
| `return/break/continue` | 制御フロー |
| `as` | 型変換 |
| `in` | メンバー検出/リスト内包表記 |
| `unsafe` | unsafe コードブロック（ポインタ） |

> **注意**：`Type`、`true`、`false`、`void` などは予約語で、キーワードではない。`type` キーワードは RFC-010 で既に削除され、統一的に `name: Type = value` 構文を使用している。

### A.2 アノテーション

| アノテーション | 役割 |
|------|------|
| `@block` | マーク后将同步执行関数 |
| `@eager` | マークは即時評価が必要 |
| `@Send` | 明示的に Send 制約滿足を宣言 |
| `@Sync` | 明示的に Sync 制約滿足を宣言 |

### A.3 原型

| 型 | 説明 | デフォルトサイズ |
|------|------|----------|
| `Void` | 空値 | 0 バイト |
| `Bool` | ブール値 | 1 バイト |
| `Int` | 符号付き整数 | 8 バイト |
| `Uint` | 符号なし整数 | 8 バイト |
| `Float` | 浮動小数点数 | 8 バイト |
| `String` | UTF-8 文字列 | 可変 |
| `Char` | Unicode 文字 | 4 バイト |
| `Bytes` | 生バイト | 可変 |

### A.4 演算子優先順位

| 優先順位 | 演算子 | 結合性 |
|--------|--------|--------|
| 1 | `()` `[]` `.` | 左から右 |
| 2 | `as` | 左から右 |

## 付録C：よくある質問

**Q: YaoXiang は Rust と比較してどのような優位性がありますか？**

A: YaoXiang は Rust のメモリ安全とゼロコスト抽象を維持するが、よりシンプルでより低い認知的負担の構文を採用している。**並作モデル**は Rust の `async/await` より簡潔である——`spawn` マークを1つ追加するだけで済み、Future と Pin の手動管理が不要である。「万物並作，吾以観復」、並行プログラミングを自然法則を記述するのと同じくらい直感的にする。**所有権モデル**（RFC-009 v9）はライフタイム注釈を持つ Move + &T/&mut T トークンを使用し、借用量検査器の代わりに型属性（Dup/Linear）を使用する。統一型構文は `enum`/`struct`/`trait`/`impl` のコンセプトの断片化を消除する。

**Q: YaoXiang はどのような開発に適していますか？**

A: システムプログラミング、应用開発、Web サービス、スクリプトツール、AI 辅助プログラミング。目標は汎用プログラミング言語になることである。

**Q: なぜ 4 スペースインデントを選択したのですか？**

A: 4 スペースは明確なコードブロックの視覚的区切りを提供し、ネストの深さ带来的混乱を減らす。これは熟慮重ねた「AI フレンドリー」設計決定である。

**Q: 1.0 バージョンはいつリリースされますか？**

A: v1.0 目標：プロダクション使用可能。リリース時間は実装進捗に依存し、詳細 [バージョニング RFC](./rfc/003-version-planning.md) を参照。

**Q: コアチームにはどのように連絡できますか？**

A: GitHub Discussions または Discord コミュニティチャネルを通じて。コアチームメンバーは定期的に返信する。
