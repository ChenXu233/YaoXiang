## 一、なぜ YaoXiang を創造するのか？

### 1.1 埋める言語の空白

プログラミング言語の歴史において、数多くの優れた言語の誕生と進化を目の当たりにしてきた：C 言語はシステムプログラミングの効率革命をもたらし、Python は誰もが学べるプログラミング体験を生み出し、Rust はメモリ安全性と性能の両立を証明し、TypeScript は大規模フロントエンドプロジェクトを保守可能にした。しかし、今日の言語エコシステムを検証したとき、依然として明白な断層地帯を発見する——**以下の三つの核心的ニーズを同時に満たす言語は一つとして存在しない**：

| ニーズ | 既存ソリューションの問題 |
|------|-------------------|
| **型安全性** | Rust は厳格すぎ、学習曲線が急峻；TypeScript はオプション型で、compile-time保証を提供できない |
| **自然な構文** | Rust の構文は複雑で難解；Haskell の関数型は敷居が高すぎる；従来の静的言語は冗長で煩雑 |
| **AI フレンドリー** | 既存言語は構文の曖昧性、AST の複雑さ、隠れた挙動の予測困難さがあり、AI によるコード生成・修正の精度が制限される |

YaoXiang の誕生は、まさにこの空白を埋めるためである。我々は信じる：**プログラミング言語は強力で親しみやすく、安全で効率的、厳密で優雅であるべきである**。

### 1.2 解決する実際的問題

**問題一：型システムの断片化**

今日のプログラミング言語は型システムにおいて深刻な断片化を見せている。静的型付き言語はcompile-timeの絶対的正確性を追求するが、往々にして開発効率を犠牲にする；動的型付き言語は柔軟性を提供するが、大規模プロジェクトでは保守困難な欠陥を露呈する。YaoXiang は「すべては型なり」という統一抽象フレームワークを提案し、型を言語設計を貫く主軸とし、後付けの补丁ではなくする。

**問題二：メモリ安全性と性能の二者択一**

長らく、開発者はメモリ安全性と実行性能の間で難しい選択を迫られてきた。GC（ガベージコレクション）は開発者を解放するが、遅延の波動とメモリオーバーヘッドをもたらす；手動メモリ管理は効率的だが、綱渡りのように危険である。YaoXiang は Rust スタイルの所有権モデルを採用し、compile-timeにデータ競合とメモリリークを排除しつつ、ゼロコスト抽象を実現し、GC なしで高性能を達成する。

**問題三：非同期プログラミングの認知負荷**

現代アプリケーションはネットワークと並行性に依存しているが、非同期プログラミングは常にプログラマーの噩梦であった。コールバック関数のネスト、Promise のチェーン呼び出し、async/await 構文——どの方式もコードの複雑性を増大させる。YaoXiang は非同期モデルを再設計した：関数シグネチャの後に `spawn` マーカーを追加するだけで、コンパイラがすべての非同期詳細を自動処理し、並行プログラミングを同期コードのように自然にする。

**問題四：AI 支援プログラミングのボトルネック**

AI が開発者のコード記述を支援するようになったとき、言語設計の選択は極めて重要になる。曖昧な構文規則、暗黙の型変換、複雑な糖衣構文——これらは人間プログラマーがすでに慣れた特性だが、AI の理解と生成の障壁となっている。YaoXiang は設計当初から「AI フレンドリー」を核心目標とする：厳格なインデント規則、明確なコードブロック境界、曖昧さのない構文構造により、AI がコードを正確に理解し、生成し、修正できるようにする。

### 1.3 言語の哲学的根基

YaoXiang の名前は『易経』の「爻」と「象」に由来する。「爻」は卦象を構成する基本記号で、陰陽変化、動静相生を象徴する；「象」は事物本質外在的表現で、万象万物を代表する。

この哲学的思想は言語設計の細部に反映されている：

- **統一性**：爻卦の単純な記号が複雑な卦象を構成するように、YaoXiang は少数の核心概念（型、関数、コンストラクタ）で完全なプログラミングモデルを構築する
- **階層性**：象に先天後天之分があるように、YaoXiang の型システムは明確な階層構造を持ち、原型から generics、値から meta typeに至る
- **変化性**：陰陽流転、変化無窮のように、YaoXiang は依存型をサポートし、型の値変化に伴う進化を可能にする
- **可識別性**：卦象が解け、万物が象を持つように、YaoXiang は完全な型 reflection 能力を提供し、runtime型情報は完全に利用可能
- **可証明性**：卦象が事物の法則を明らかにするように、YaoXiang の型システムは Curry-Howard 同型（型は命題、プログラムは証明）に従い、型検査の過程は論理的証明の検証である

## 二、核心哲学と原則

以下の設計信条は YaoXiang の礎であり、**妥協できず、违背できない**。いかなる特性提案もこれらの原則の検証を経なければならない。

### 2.1 原則一：すべては型なり

YaoXiang の世界観において、型は最高層の抽象ユニットであり、言語を貫く核心概念である。

**具体的な体現**：

- **値は型のインスタンス**：`42` は `Int` 型のインスタンス、`"hello"` は `String` 型のインスタンス
- **型自体も型である**：`Type` は言語唯一の meta type キーワードであり、`Int` の類型は `Type` である
- **関数は型写像**：`add: (a: Int, b: Int) -> Int` は `Int × Int` から `Int` への型写像を記述する
- **モジュールは型の組み合わせ**：モジュールは関数と型を含む名前空間の組み合わせである

**妥協できない理由**：統一型抽象は言語意味論を簡素化し、値と型の二元対立を消除し、型システムをコード正確性の守護者とし、绊脚石としない。

### 2.2 原則二：厳格な構造化

YaoXiang の構文設計は「曖昧さなし、予測可能、解析容易」を追求する。

**具体的規則**：

- **強制 4 スペースインデント**：Tab 文字の使用を禁止し、コードブロック境界を一目で把握
- **括弧は省略不可**：関数引数は括弧必須、リスト要素はカンマ必須
- **コードブロックには中括弧必須**：`if`、`while`、`for` などの制御フローは `{ }` で囲むことを必須とする
- **キーワード数の簡素化**：わずか 17 個の核心キーワードのみを残し、糖衣構文の泛滥を拒否

**妥協できない理由**：厳格な構造化は三つの重要な利点をもたらす——（1）IDE のシンタックスハイライトとコード折りたたみがより正確；（2）AI コード生成と修正の精度が大幅に向上；（3）新規学習者が迅速にコード構造を理解可能。

### 2.3 原則三：ゼロコスト抽象

高レベルの抽象が実行時の性能オーバーヘッドをもたらすことはない。

**具体的保証**：

- **単相化**：generics 関数は compile-time に具象バージョンに展開され、仮想テーブル検索のオーバーヘッドなし
- **インライン最適化**：単純な関数は自動インライン化され、関数呼び出しのオーバーヘッドを消除
- **スタック割り当て優先**：小オブジェクトはデフォルトでスタック割り当て、ヒープ割り当ては必要な場合のみ使用
- **GC なし**：所有権モデルがメモリ安全性を保証し、ガベージコレクターの実行時オーバーヘッド不要

**妥協できない理由**：性能はプログラミング言語の生存底线である。性能を犠牲にして利便性を得る設計はすべてプログラマーへの裏切りである。

### 2.4 原則四：デフォルトでイミュータブル

可変性は複雑性と切り離せない。YaoXiang はデフォルトでイミュータブルを選択し、コードをより推論しやすく理解しやすくする。

**具体的規則**：

- 変数はデフォルトでイミュータブル、割り当て後に再変更不可
- 可変が必要な場合は明示的に `mut` を宣言する必要がある
- 参照はデフォルトでイミュータブル、可変参照は `mut` マーカーが必要
- 所有権の移転は元の binding の失效を意味する

**妥協できない理由**：不変性は並行安全性の基礎であり、コード可読性の保障であり、関数型プログラミングの知恵の結晶である。

### 2.5 原則五：型はデータなり

型情報は compile-time だけでなく、runtime でも完全に利用可能なはずである。

**具体的能力**：

- runtime 型クエリ：すべての値はその型情報を取得可能
- 型 reflection：型自体の構築と操作が可能
- パターン照合分解：型コンストラクタは直接パターン照合に使用可能
- generics 特化：runtime で generics 引数の具象型を取得可能

**妥協できない理由**：完全な型 reflection 能力はメタプログラミングの基礎であり、高性能フレームワークとツールの礎である。

## 三、鍵となる革新と特性

YaoXiang は既存言語の優れた特性を吸収しつつ、以下の革新的な設計を提案する。

### 3.1 革新一：統一型構文

**従来言語の型定義**は往々にして複数のキーワードを必要とする：

```rust
// Rust
struct Point { x: f64, y: f64 }
enum Result<T, E> { Ok(T), Err(E) }
enum Color { Red, Green, Blue }
trait Drawable { fn draw(&self, s: &Surface); }
```

**YaoXiang の統一構文**：すべては `name: type = value`、`Type` は唯一の meta type キーワード。

```yaoxiang
# === レコード型 ===

Point: Type = {
    x: Float,
    y: Float,
}

# デフォルト値を持つフィールド
Point3D: Type = {
    x: Float = 0,
    y: Float = 0,
    z: Float = 0,
}

# === generics 型 ===

Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self,
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,
}

# === インタフェース（フィールドがすべて関数型のレコード） ===

Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect,
}

Serializable: Type = {
    serialize: () -> String,
}

# === インタフェース実装（インタフェース名は型本体内に記述） ===

Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable,
}

# === メソッド（Type.method 構文） ===

Point.draw: (self: &Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}
```

**革新的価値**：`fn`、`struct`、`enum`、`trait`、`impl` キーワードの碎片なし——統一構文がすべての宣言をカバー。

### 3.2 革新二：コンストラクタは型なり

**値構築と関数呼び出しは完全に同一**：

```yaoxiang
# 型定義
Point: Type = { x: Float, y: Float }
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self,
}

# 値構築：関数呼び出しと同一
p: Point = Point(3.0, 4.0)
opt: Option(Int) = Option.some(42)
none: Option(Int) = Option.none()

# パターン照合：直接分解
match opt {
    Option.some(value) -> print(value)
    Option.none -> print("nothing")
}
```

### 3.3 革新三：カリー化メソッド binding

YaoXiang は純粋関数型設計を採用し、カリー化によりオブジェクトメソッド呼び出しに类似した糖衣構文を実現し、`class` と `method` キーワードの導入を必要としない。

```yaoxiang
# === 型定義 ===

Point: Type = {
    x: Float,
    y: Float,
}

# 核心関数：ユークリッド距離
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    return (dx * dx + dy * dy).sqrt()
}

# メソッド糖衣構文 binding（[0] は第 0 引数位置への binding を示す）
Point.distance = distance[0]

# === 使用 ===

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# 二つの呼び出し方式は完全に等価
d1 = distance(p1, p2)     # 核心関数の直接呼び出し
d2 = p1.distance(p2)      # メソッド糖衣構文

# カリー化用法
dist_from_p1 = p1.distance  # 部分適用、第二引数を待機
d3 = dist_from_p1(p2)       # 2.828
```

**革新的価値**：純粋関数型設計、隠れた `self` 引数なし、関数は値として自由に传递と組み合わせが可能。

### 3.4 革新四：spawn モデル

> 「万物并作，吾以观復。」——『易・復卦』
>
> spawn モデルはこれに因み、一種のプログラミングパラダイムを記述する：開発者は同期・順次思考でロジックを記述し、言語ランタイムが其中的計算ユニットを万物并作のように自動的・高效的に並行実行させ、最終的に統一協同させる。

**核心三原則**：

| 原則 | 説明 |
|------|------|
| **同期構文** | 見たままの順次コード |
| **並行本質** | ランタイムが自動的に並列性を抽出 |
| **統一協同** | 結果は必要時に自動集約、論理的正確性を保証 |

**用語体系**：

| 公用語 | 対応構文 | 説明 |
|----------|----------|------|
| **spawn 関数** | `spawn (params) => body` | spawn 実行に参加可能な計算ユニットを定義 |
| **spawn ブロック** | `spawn { a(), b() }` | 明示的に宣言された並行疆域、ブロック内タスクは spawn 実行 |
| **spawn ループ** | `spawn for x in xs { ... }` | データ並列、ループ本体はすべての要素で spawn 実行 |
| **spawn 値** | `Async(T)` | spawn 中の未来値、使用時に自動待機 |
| **spawn グラフ** | 遅延計算グラフ(DAG) | spawn 発生の舞台、依存関係と並列関係を記述 |
| **spawn スケジューラ** | ランタイムタスクスケジューラ | 万物を協調し、正しい时机に spawn させる智能中枢 |

> **詳細**：[RFC-001 spawn モデル](./rfc/001-concurrent-model-error-handling.md)

```yaoxiang
# === spawn 関数 ===
# spawn マーカー付き関数
fetch_data: (url: String) -> JSON spawn = {
    return HTTP.get(url).json()
}

# === spawn ブロック ===
# spawn { } 内の式は強制的に並列実行
compute_all: () -> (Int, Int, Int) spawn = {
    (a, b, c) = spawn {
        heavy_calc(1),    # タスク 1
        heavy_calc(2),    # タスク 2
        another_calc(3)   # タスク 3
    }
    return (a, b, c)
}

# === 自動待機 ===
main: () -> Void = {
    # 二つの独立リクエストは自動的に並列実行
    users = fetch_data("https://api.example.com/users")
    posts = fetch_data("https://api.example.com/posts")

    # 待機点は結果が必要な時に自動挿入
    print(users.length + posts.length)  # users と posts を自動待機
}
```

**スレッドセーフティ**：

```yaoxiang
# ref キーワードが自動的にスレッドセーフティを処理（コンパイラが Rc/Arc を自動選択）
main: () -> Void = {
    counter = ref SafeCounter(0)

    # タスク間共有：コンパイラが Arc を自動選択
    spawn {
        counter.increment()
    }
    spawn {
        counter.increment()
    }
}
```

**技術文書**：
- 詳細 [RFC-001 spawn モデル](./rfc/accepted/001-concurrent-model-error-handling.md)

**革新的価値**：非同期プログラミングの認知負荷がゼロに、コード可読性は同期コードと完全に同一、同時に高性能並列実行効率を獲得。

### 3.5 革新五：値依存型（RFC-011）

> **ステータス**：設計中、一部実装済み

型は値に依存可能、真の型駆動開発を実現。

```yaoxiang
# 行列型：次元が compile-time に確定
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
}

# compile-time 計算：factorial(3) = 6
vec: Vec(factorial(3)) = Vec(6)()

# compile-time 次元検証
identity_3x3: Matrix(Float, 3, 3) = identity(Float, 3)(3)
# multiply(matrix_2x3, matrix_4x2)  # compile エラー：次元不一致
```

**革新的価値**：compile-time により多くのエラーを捕捉、より精确な型保証を実現。

### 3.6 革新六：極簡キーワード設計

YaoXiang はわずか 17 個の核心キーワードのみを定義、数は主流言語を大きく下回る：

```
pub    use    spawn
ref    mut    if     elif
else   match  while  for    return
break  continue as     in     unsafe
```

| 比較言語 | キーワード数 |
|----------|-----------|
| YaoXiang | **17** |
| Rust | 51+ |
| Python | 35 |
| TypeScript | 64+ |
| Go | 25 |

**革新的価値**：より低い記憶负担、より一貫した構文スタイル、より解析容易な構文構造。

### 4.4 所有権モデル（RFC-009 v9）

```yaoxiang
Point: Type = { x: Float, y: Float }

# デフォルト Move（ゼロコピー）
p1 = Point(1.0, 2.0)
p2 = p1              # Move、p1 は以降読み取り不可

# &T / &mut T トークン（compile-time ゼロコスト）
p2.print()           # コンパイラが自動的に &Point トークンを作成
p2.shift(1.0, 1.0)  # コンパイラが自動的に &mut Point トークンを作成

# ref：共有持有（コンパイラが Rc/Arc を自動選択）
shared = ref p2      # スコープ間共有

# clone()：明示的深コピー
backup = p2.clone()

# unsafe + 生ポインタ：システムレベル
unsafe {
    ptr: *Point = &p2
    (*ptr).x = 0.0
}
```

**所有権グラデーション**：
```
&T / &mut T    Move       ref        clone()    unsafe
    |             |          |           |          |
借用トークン     デフォルト  共有持有    深コピー    生ポインタ
ゼロコスト       ゼロコピー  自動Rc/Arc  明示的     システムレベル
```

### 4.5 エラー処理

```yaoxiang
# Result 型
Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,
}

divide: (a: Float, b: Float) -> Result(Float, String) = {
    if b == 0.0 {
        return Result.err("Division by zero")
    }
    return Result.ok(a / b)
}

# match での使用
result = divide(10.0, 2.0)
match result {
    Result.ok(value) -> print(value),
    Result.err(msg) -> print("Error: ${msg}"),
}
```

### 4.6 並行プログラミング（spawn モデル）

```yaoxiang
# spawn マーカーで非同期関数をマーク
fetch_api: (url: String) -> JSON spawn = {
    response = HTTP.get(url)
    return JSON.parse(response.body)
}

# 並行構築ブロック：明示的並列
process_all: () -> (JSON, JSON, JSON) spawn = {
    (a, b, c) = spawn {
        fetch_api("https://api1.com/data"),
        fetch_api("https://api2.com/data"),
        fetch_api("https://api3.com/data")
    }
    return (a, b, c)
}
```

## 五、ロードマップと未決定事項

### 5.1 既に決定された設計決定

以下の決定は充分な議論と審査を経ており、**もはや変更を受け付けない**：

| モジュール | 決定 | 説明 |
|------|------|------|
| **型システム** | すべては型なり | 値、関数、モジュール、generics すべてが型 |
| **型構文** | 統一 `name: type = value` | 一種の宣言形式がすべての状況をカバー、`Type` は唯一の meta type キーワード |
| **キーワード** | 17 個の核心キーワード | `type`/`fn`/`struct`/`enum`/`trait`/`impl` を含まず |
| **関数構文** | シグネチャ + 式 | `name: (params) -> ReturnType = body` |
| **メソッド binding** | RFC-004 カリー化 binding | `Type.method = function[position]` |
| **非同期モデル** | spawn モデル | `spawn` マーカー、遅延評価、自動並列 |
| **メモリ管理** | 所有権モデル（RFC-009 v9） | Move + &T/&mut T トークン + ref + clone + unsafe、GC なし |
| **ファイル即モジュール** | モジュールシステム | 各 `.yx` ファイルは一つのモジュール |
| **主関数** | `main: () -> Void` | プログラムエントリポイント |
| **スレッドセーフティ** | ref が Rc/Arc を自動選択 | コンパイラの escape analysis、ユーザーは無感覚 |

## 付録C：よくある質問

**Q: YaoXiang は Rust と比較して有什么优势？**

A: YaoXiang は Rust のメモリ安全性とゼロコスト抽象を保持しつつ、より簡单な構文とより低い認知负担を採用。**spawn モデル**は Rust の `async/await` より簡潔——一つの `spawn` マーカーのみ、Future と Pin の手動管理不要。「万物并作，吾以观復」、並行プログラミングを自然法則記述のように直感的に。**所有権モデル**（RFC-009 v9）は生命周期注釈を Move + &T/&mut T トークンで代替し、借用検査器を型属性（Dup/Linear）で代替。統一型構文は `enum`/`struct`/`trait`/`impl` の概念碎片を消除。

**Q: YaoXiang はどのような種類の開発に適しているか？**

A: システムプログラミング、アプリケーション開発、Web サービス、スクリプティングツール、AI 支援プログラミング。汎用プログラミング言語になることを目標とする。

**Q: なぜ 4 スペースインデントを選択？**

A: 4 スペースは明確なコードブロック視覚的分離を提供し、ネスト深度がもたらす混乱を減少する。深思熟慮された「AI フレンドリー」設計決定。

**Q: いつ 1.0 バージョンがリリースされるか？**

A: v1.0 目標：プロダクション使用可能。リリース時期は実装進度に依存、詳細 [バージョン計画 RFC](./rfc/003-version-planning.md)。
