## 一、なぜ YaoXiang を創造したのか？

### 1.1 埋めていた言語の空白

プログラミング言語の歴史において、无数の優秀な言語の誕生と進化を見てきた：C言語はシステムプログラミングの効率革命をもたらし、Pythonは誰もが学べるプログラミング体験を作り出し、Rustはメモリ安全とパフォーマンスの両立を証明し、TypeScriptは 대규모フロントエンドプロジェクトを管理可能にした。しかし、今日の言語エコシステムを見渡すと、依然として明確な断層が存在する——**以下の3つのコアニーズを同時に満たす言語が一つもない**：

| ニーズ | 既存ソリューションの問題点 |
|------|-------------------|
| **型安全性** | Rustは厳しすぎて、学習曲線が険しい；TypeScriptはオプショナル型で、コンパイル時の保証を提供できない |
| **自然な構文** | Rustの構文は複雑で晦渋；Haskellの関数型は敷居が高い；従来の静的言語は冗長で面倒 |
| **AI フレンドリー** | 既存の言語は構文の曖昧さ、ASTの複雑さ、予測困難な隠れた動作が多く、AIのコード生成と修正の正確率が制限される |

YaoXiang の誕生は、まさにこの空白を埋めるためである。私たちは信じている：**プログラミング言語は、強力で親しみやすく、安全で効率的で、厳格で優雅であるべきだ**。

### 1.2 解決すべき實際的な問題

**問題一：タイプシステムの斷片化**

今日のプログラミング言語は、タイプシステムにおいて深刻な斷片化を見せている。静的型付け言語はコンパイル時の絶対的な正確さを追求するが、往々にして開発効率を犠牲にする代償を払う。動的型付け言語は柔軟性を提供するが、大規模プロジェクトでは維持困難な欠陥が露呈する。YaoXiang は「すべては型である」という統一的抽象フレームワークを提案し、型を言語設計の主線として貫き、後付けのパッチとするのではなく、型をコードの正しさの守護者とする。

**問題二：メモリ安全とパフォーマンスの二択**

開發者たちは長い間、メモリ安全と実行パフォーマンスの間で難しい選択を迫られてきた。GC（ガベージコレクション）は開発者を解放したが、遅延波动とメモリオーバーヘッドをもたらした。手動メモリ管理は効率的だが、綱渡りのように危険を伴う。YaoXiang はRust 流の所有権モデルを採用することで、コンパイル時にデータ競合とメモリリークを消除し、ゼロコスト抽象を維持しながら、GCなしで高性能を実現する。

**問題三：非同期プログラミングの認知的負担**

现代アプリケーションはネットワークと並行処理なしに成り立たず、非同期プログラミングは常に程序员たちの悪夢であった。コールバック関数のネスト、Promiseのチェーン呼び出し、async/await構文——每一种方案はコードの複雑さを增加させた。YaoXiang は非同期モデルを重新設計した：関数シグネチャの後に `spawn` マーカーを追加するだけで、コンパイラがすべての非同期詳細を自動的に処理し、並行プログラミングを同期コードと同じくらい自然にさせる。

**問題四：AI 支援プログラミングのボトルネック**

AI が開発者のコード作成を支援し始めた今、言語設計の選択が非常に重要になった。曖昧な構文ルール、暗黙の型変換、複雑な糖衣構文——これらの人類程序员が既に慣れている特性は、AI の理解と生成の障害となっている。YaoXiang は設計当初から「AI フレンドリー」をコア目標とした：嚴格なインデントルール、明確なコードブロック境界、曖昧さのない構文構造により、AI が正確に理解し、生成し、修正できるようにした。

### 1.3 言語のphilosophy的根本

YaoXiang という名前は『易経』の「爻」と「象」に由来する。「爻」は卦を構成する基本ymbolであり、yin-yangの変化、動静の相生の象徴である。「象」は事物の本質 внешнее表現であり、万象万物、包摂一切，代表万物。

この哲学的思想は言語設計の各ディテールに體現されている：

- **統一性**：爻卦の単純なsymbolsが複雑な卦を構成するように、YaoXiang は少数のコアコンセプト（型、関数、コンストラクタ）で完全なプログラミングモデルを構築する
- **階層性**：象に先天後天の区別があるように、YaoXiang のタイプシステムは明確な階層構造を持ち、原型からジェネリクスへ、値からメタタイプへ
- **変化性**：yin-yangの流れのように変化无穷のように、YaoXiang は依存型をサポートし、型の値を伴う進化を可能にする
- **識別可能性**：卦象が解けて万物に象るように、YaoXiang は完全な型リフレクション能力を提供し、runtime 型情報が完全に利用可能
- **証明可能性**：卦象が事物の法則を明らかにするように、YaoXiang のタイプシステムはCurry-Howard同形に従い（型は命題、プログラムは証明）、型チェックのプロセスは論理証明の検証である

## 二、コアphilosophyと原則

以下の設計信条は YaoXiang の基石であり、**妥協不许、違反不许**。すべての機能提案はこの原則の检验を経なければならない。

### 2.1 原則一：すべては型である

YaoXiang の世界観において、型は最も高次の抽象ユニットであり、言語を貫くコアコンセプトである。

**具体的な體現**：

- **値は型のインスタンス**：`42` は `Int` 型のインスタンス、`"hello"` は `String` 型のインスタンス
- **型自体も型である**：メタタイプ（`type`）はすべて型の型であり、`Int` は `type` のインスタンス
- **関数は型マッピング**：`fn add(Int, Int) -> Int` は `Int × Int` から `Int` への型マッピングを記述する
- **モジュールは型の組み合わせ**：モジュールは関数と型を含む名前空間组合せ

**妥協不许の理由**：統一的型抽象は言語セマンティクスを簡素化し、値と型の二元論を消除し、タイプシステムをコードの正しさの守護者とし、障害物而不是とする。

### 2.2 原則二：厳格な構造化

YaoXiang の構文設計は「曖昧さなし、予測可能、解析容易」を追求する。

**具体的なルール**：

- **强制 4スペースインデント**：Tab文字の使用を禁止し、コードブロックの境界が一目でわかる
- **括弧は省略不可**：関数引数には括弧が必要、リスト要素にはカンマが必要
- **コードブロックには波括弧が必要**：`if`、`while`、`for` などの制御フローは `{ }` で囲む必要がある
- **キーワード数を精简**：17のコアキーワードのみを保持し、糖衣構文の泛滥を拒否

**妥協不许の理由**：厳格な構造化は3つの重要な優位性をもたらす——（1）IDEの構文ハイライトとコードフォールドがより正確；（2）AIコード生成と修正の正確率が大幅に向上；（3）新規学習者がコード構造を迅速に理解できる。

### 2.3 原則三：ゼロコスト抽象

高レベルの抽象はruntime のパフォーマンスオーバーヘッドをもたらすべきではない。

**具体的な保証**：

- **単態化**：ジェネリック関数はコンパイル時に具体的なバージョンに展開され、vtableルックアップオーバーヘッドなし
- **インライン最適化**：シンプルな関数は自動的にインライン化され、関数呼び出しオーバーヘッドを消除
- **栈割り当て優先**：小さなオブジェクトはデフォルトで栈割り当てが必要时才のみヒープ割り当てを使用
- **GCなし**：所有権モデルがメモリ安全を保証し、ガベージコレクタのruntime オーバーヘッドが不要

**妥協不许の理由**：パフォーマンスはプログラミング言語の生存底线である。便理性のためにパフォーマンスを犠牲にする設計はすべて、程序员への裏切りである。

### 2.4 原則四：デフォルト不変性

可変性と複雑性は影を伴う。YaoXiang はデフォルト不変性を選択し、コードをより推論可能で理解しやすくする。

**具体的なルール**：

- 変数はデフォルトで不変であり、代入後は変更できない
- 可変性が必要な場合は `mut` を明示的に宣言する必要がある
- 参照はデフォルトで不変であり、可変参照には `mut` マーカーが必要
- 所有権の移転は元のバインディングの無効を意味する

**妥協不许の理由**：不変性は並行安全な基礎であり、コード可読性の保証であり、関数型プログラミングの知恵の结晶である。

### 2.5 原則五：型はデータである

型情報はコンパイル時のみ存在するべきではなく、runtime に完全に利用可能であるべきである。

**具体的な能力**：

- runtime 型クエリ：任意の値からその型情報を取得できる
- 型リフレクション：型自体を構成して操作できる
- パターン照合によるデコンストラクション：型コンストラクタは直接パターン照合に使用できる
- ジェネリック特殊化：runtime でジェネリックパラメータの具体化型を取得できる

**妥協不许の理由**：完全な型リフレクション能力はメタプログラミングの基礎であり、高パフォーマンスフレームワークとツールの基石である。

## 三、重要な革新と特性

YaoXiang は既存の言語の優秀な特性を吸收しながら、以下の革新的設計を提案する。

### 3.1 革新一：統一的型構文

**従来の言語の型定義**は往往にして複数のキーワードを必要とする：

```rust
// Rust
struct Point { x: f64, y: f64 }
enum Result<T, E> { Ok(T), Err(E) }
enum Color { Red, Green, Blue }
union IntOrFloat { i: i32, f: f32 }
```

**YaoXiang の統一的構文**：`type` キーワードのみで、`{}` でデータ構造を定義し、`[]` でインターフェース型を定義する。

```yaoxiang
# === データ型（波括弧） ===

# 構造体
type Point = { x: Float, y: Float }

# enum（複数のコンストラクタ）
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# ゼロ引数コンストラクタ
type Color = { red | green | blue }

# 混合型
type Shape = { circle(Float) | rect(Float, Float) }

# === インターフェース型（角括弧） ===

# インターフェース定義：メソッドシグネチャの集合
type Serializable = [
    serialize() -> String
]

type Drawable = [
    draw(Surface) -> Void,
    bounding_box() -> Rect
]

# === ジェネリクス ===

Option: (T: Type) -> Type = { some(T) | none }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }
```

**革新的価値**：統一的型構文で `enum`/`struct`/`union`/`trait` キーワードの斷片化を消除する。

### 3.2 革新二：コンストラクタは型である

**値構築とパターン照合は完全に同一**：

```yaoxiang
# 型定義
type Point = { x: Float, y: Float }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# 値構築：関数呼び出しと同じ
p: Point = Point(3.0, 4.0)
r: Result[Int, String] = ok(42)
err_msg: Result[Int, String] = err("not found")
c: Color = green

# パターン照合：直接デコンストラクション
match point {
    Point(0.0, 0.0) -> "origin"
    Point(x, y) -> "point at ({x}, {y})"
}
```

### 3.3 革新三：カリー化されたメソッドバインディング

YaoXiang は純粋関数型設計を採用し、カリー化を通じてオブジェクトメソッド呼び出しのような糖衣構文を実現し、`class` と `method` キーワードを導入する必要がない。

```yaoxiang
# === Point.yx ===

# 型定義
type Point = { x: Float, y: Float }

# コア関数：ユークリッド距離
distance(Point, Point) -> Float = (a, b) => {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# メソッド糖衣構文バインディング
Point.distance = distance  # デフォルトで第0引数にバインディング
```

```yaoxiang
# === main.yx ===

use Point

main() -> Void = () => {
    p1 = Point(3.0, 4.0)
    p2 = Point(1.0, 2.0)

    # 2つの呼び出し方法は完全に同等
    d1 = distance(p1, p2)      # コア関数を直接呼び出す
    d2 = p1.distance(p2)       # メソッド糖衣構文

    # 関数型用法：最初の引数を事前にバインディング
    dist_from_origin = Point.distance(Point(0.0, 0.0))
    result = dist_from_origin(p1)   # 5.0

    # カリー化用法：遅延評価
    get_dist_to_p2 = p2.distance(_)
    d3 = get_dist_to_p2(p1)         # 2.828
}
```

**革新的価値**：純粋関数型設計で隠れた `self` パラメータなし、関数は値として自由に传递と組み合わせが可能。

### 3.4 革新四：並作モデル

> 「万物並作，吾以觀復。」——《易·復卦》
>
> 並作モデルはここにアイデアを得て、プログラミングパラダイムを記述する：開発者は同期、順序的な思考でロジックを描述し、言語runtime はその中の計算ユニットを万物並作のように自動、高效に並行実行させ、最終的に統一協調する。

**コア3原則**：

| 原則 | 説明 |
|------|------|
| **同期構文** | 目に見える通りの順序コード |
| **並行本質** | runtime が自動的に並列性を抽出 |
| **統一協調** | 結果は必要な時に自動的に集結し、ロジック正確性を保証 |

**用語体系**：

| 公式用語 | 対応構文 | 解説 |
|----------|----------|------|
| **並作関数** | `spawn (params) => body` | 並作実行に参加できる計算ユニットを定義 |
| **並作ブロック** | `spawn { a(), b() }` | 明示的に宣言された並行領域、ブロック内のタスクは並作実行 |
| **並作ループ** | `spawn for x in xs { ... }` | データ並列、ループ体がすべての要素で並作実行 |
| **並作値** | `Async(T)` | 並作中の未来値、使用時に自動的に待機 |
| **並作グラフ** | 遅延計算グラフ(DAG) | 並作が発生するステージ、依存関係と並列関係を描述 |
| **並作スケジューラ** | runtime タスクスケジューラ | 万物を調整し、正しいタイミングで並作させるIntelligent中枢 |

> **詳細**: [RFC-001 並作モデル](./rfc/001-concurrent-model-error-handling.md)

```yaoxiang
# === 並作関数 ===
# spawn マーカー付き関数（RFC-003 構文）
fetch_data: String -> JSON = spawn (url) => {
    HTTP.get(url).json()
}

# === 並作ブロック ===
# spawn { } 内の式は強制的に並行実行
compute_all: () -> (Int, Int, Int) = spawn () => {
    (a, b, c) = spawn {
        heavy_calc(1),    # タスク 1
        heavy_calc(2),    # タスク 2
        another_calc(3)   # タスク 3
    }
    (a, b, c)
}

# === 並作ループ ===
# spawn for マーカー付きループは自動的に並列化
parallel_sum: Int -> Int = spawn (n) => {
    total = spawn for i in 0..n {
        fibonacci(i)  # 各反復が並行実行
    }
    total
}

# === 自動待機 ===
main() -> Void = () => {
    # 2つの獨立リクエストは自動的に並行実行
    users = fetch_data("https://api.example.com/users")
    posts = fetch_data("https://api.example.com/posts")

    # 待機点は結果が必要な時に自動的に挿入
    print(users.length + posts.length)  # users と posts を自動的に待機
}
```

**スレッド安全性**：

```yaoxiang
# Send/Sync 制約がコンパイル時のスレッド安全性を保証
type SafeCounter = SafeCounter(mutex: Mutex(Int))

main: () -> Void = () => {
    counter: Arc[SafeCounter] = Arc.new(SafeCounter(Mutex.new(0)))

    # spawn は自動的に Send 制約をチェック
    spawn () => {
        guard = counter.mutex.lock()
        guard.value = guard.value + 1
    }
}
```

**技術ドキュメント**：
- 詳細 [RFC-001 並作モデル](./rfc/001-concurrent-model-error-handling.md)

**革新的価値**：非同期プログラミングの認知的負担がゼロになり、コード可読性は同期コードと完全に同じになりながら、高パフォーマンス並列の実行効率を得る。

### 3.5 革新五：依存型サポート（将来機能）

> **状態**：v1.0 後での実装を検討

型は値に依存し、真の型駆動開発を実現する。

```yaoxiang
# 固定長ベクトル（将来構文）
Vector: (T: Type, n: Int) -> Type = {
    data: [T; n]
    length: n
}

# 使用
vec: Vector[Int, 3] = Vector([1, 2, 3], 3)

# 型チェック
# vec: Vector[Int, 3] = Vector([1, 2], 2)  # コンパイルエラー！長さが一致しない
```

**革新的価値**：コンパイル時に より多くのエラーを捕获し、より精密な型保証を実現。

### 3.6 革新六：極限まで精简されたキーワード設計

YaoXiang はわずか18のコアキーワードのみを定義し、主要言語よりはるかに少ない：

```
type   pub    use    spawn
ref    mut    if     elif
else   match  while  for    return
break  continue as     in     unsafe
```

| 比較対象言語 | キーワード数 |
|----------|-----------|
| YaoXiang | **18** |
| Rust | 51+ |
| Python | 35 |
| TypeScript | 64+ |
| Go | 25 |

**革新的価値**：より低い記憶負担、より一貫性のある構文スタイル、より解析しやすい構文構造。

### 4.5 所有権モデル

```yaoxiang
type Point = { x: Float, y: Float }

# デフォルト Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p              # Move、所有権移転、p は無効

# 明示的 ref = Arc（安全共有）
shared = ref p      # Arc、スレッドセーフ

spawn(() => print(shared.x))   # ✅ 安全

# 明示的 clone() = コピー
p3 = p.clone()      # p と p3 は獨立
```

**コアルール**：
- デフォルト Move（ゼロコピー）
- 共有は `ref`（Arc）
- コピーは `clone()`
- システムレベルは `unsafe` + `*T`

### 4.7 並行プログラミング（SeamlessAsync）

```yaoxiang
# spawn マーカー付き非同期関数
fetch_api: String -> JSON spawn = (url) => {
    response = HTTP.get(url)
    JSON.parse(response.body)
}

# 並行構築ブロック：明示的並列
process_all: () -> (JSON, JSON, JSON) spawn = () => {
    (a, b, c) = spawn {
        fetch_api("https://api1.com/data"),
        fetch_api("https://api2.com/data"),
        fetch_api("https://api3.com/data")
    }
    (a, b, c)
}

# データ並列ループ
parallel_process: Int -> Int spawn = (n) => {
    total = spawn for i in 0..n {
        compute(i)
    }
    total
}

# スレッド安全性例
type ThreadSafeCounter = { value: Mutex(Int) }

main: () -> Void = () => {
    counter = ThreadSafeCounter(Mutex.new(0))

    # spawn は自動的に Send 制約をチェック
    spawn () => {
        guard = counter.value.lock()
        guard.value = guard.value + 1
    }
}
```

## 付録C：よくある質問

**Q: YaoXiang は Rust と比較してどんな優位性がありますか？**

A: YaoXiang は Rust のメモリ安全とゼロコスト抽象を保持しますが、よりシンプルな構文とより低い認知的負担を採用しています。**並作モデル**は Rust の `async/await` より簡潔で、ただ1つの `spawn` マーカーだけで済み、Future と Pin を手動管理する必要がありません。「万物並作，吾以觀復」、並行プログラミングを自然法則を描述するように直感的にします。Send/Sync 制約は同等のスレッド安全性を保証します。統一的型構文は `enum`/`struct`/`union` の概念斷片化を消除します。

**Q: YaoXiang はどんな種類の開発に適していますか？**

A: システムプログラミング、应用開発、Web サービス、スクリプトツール、AI 支援プログラミング。目標は汎用プログラミング言語になることです。

**Q: なぜ4スペースインデントを選んだのですか？**

A: 4スペースは明確なコードブロックの視覚的区切りを提供し、ネストの深さ带来的混乱减少了。これは熟慮された「AI フレンドリー」な設計上の決定です。

**Q: いつ1.0バージョンがリリースされますか？**

A: v1.0 目標：プロダクション利用可能。リリース時間は実装進捗に依存し、詳細 [バージョニング RFC](./rfc/003-version-planning.md) を参照。

**Q: コアチームに連絡する方法は？**

A: GitHub Discussions または Discord コミュニティチャネルを通じて。コアチームメンバーは定期的に返信します。
