## 一、なぜ YaoXiang を創ったのか？

### 1.1 埋めていた言語の空白

プログラミング言語の歴史において、数多くの優れた言語の誕生と進化を見てきました：C言語はシステムプログラミングに効率革命をもたらし、Pythonは誰もが学べるプログラミング体験を作り上げ、Rustはメモリ安全と性能の両立を証明し、TypeScriptは大型フロントエンドプロジェクトを保守可能にしました。しかし、今日の言語エコシステムを見渡すと、**以下の3つのコアニーズを同時に満たす言語が依然として存在しない**という明確な空白地帯があります：

| ニーズ | 既存ソリューションの問題点 |
|--------|-------------------|
| **型安全** | Rustは過度に厳格で、学習曲線が険しい；TypeScriptはオプショナル型であり、コンパイル時の保証を提供できない |
| **自然な構文** | Rustの構文は複雑で難解；Haskellの関数型は敷居が高い；従来の静的言語は冗長で面倒 |
| **AI フレンドリ** | 既存の言語は構文の曖昧さ、ASTの複雑さ、予測困難な隠れた動作が多く、AIによるコード生成と修正の精度が制限される |

YaoXiangの誕生は、この空白を埋めるためです。私たちは信じています：**プログラミング言語は、強力で親しみやすく、安全で効率的であり、厳密で優雅であるべきです**。

### 1.2 解決すべき実際の問題

**問題1：型システムの断片化**

今日のプログラミング言語は、型システムにおいて深刻な断片化を示しています。静的型付け言語はコンパイル時の絶対的な正確さを追求しますが、往々にして開発効率を犠牲にしています；動的型付け言語は柔軟性を提供しますが、大型プロジェクトでは保守困難な欠陥が露呈します。YaoXiangは「すべては型である」という統一的抽象フレームワークを提案し、型を言語設計の主線として貫き、後付けのパッチとします。

**問題2：メモリ安全と性能の二者択一**

長年にわたり、開発者はメモリ安全と実行性能の間で難しい選択を強いられてきました。GC（ガベージコレクション）は開発者を解放しますが、遅延波动とメモリオーバーヘッドをもたらします；手動メモリ管理は効率的ですが、綱渡りのように危険です。YaoXiangはRust流の所有権モデルを採用し、コンパイル時にデータ競合とメモリリークを排除すると同時に、ゼロコスト抽象を維持し、GCなしで高性能を実現します。

**問題3：非同期プログラミングの認知負荷**

現代アプリケーションはネットワークと並行処理離れず、非同期プログラミングは一直是程序员の噩梦でした。コールバックのネスト、Promiseのチェーン呼び出し、async/await構文——各方案はコードの複雑さを増大させます。YaoXiangは非同期モデルを再設計しました：関数シグネチャの後に`spawn`マークを追加するだけで、コンパイラがすべての非同期詳細を自動処理し、並行プログラミングを同期コードのように自然にします。

**問題4：AI補助プログラミングのボトルネック**

AIが開発者のコード作成を補助し始めると、言語設計の選択が至关重要となります。曖昧な構文ルール、暗黙の型変換、複雑な構文糖——これらの是人类程序员 уже освоили特性は、AIの理解と生成の障害となっています。YaoXiangは設計当初から「AIフレンドリ」をコア目標としています：厳格なインデントルール、明確なコードブロック境界、曖昧さのない構文構造により、AIが正確に理解し、生成し、修正できるようになりました。

### 1.3 言語の哲学的基盤

YaoXiangという名前は『易経』の「爻」と「象」に由来しています。「爻」は卦象を構成する基本記号であり、陰陽の変化、動静の相生を象徴します；「象」は事物の本質 внешнего表現であり、万象万物を包容します。

この哲学的思想は言語設計のあらゆる詳細に反映されています：

- **統一性**：爻卦の単純な記号が複雑な卦象を構成するように、YaoXiangは少数のコアコンセプト（型、関数、コンストラクタ）で完全なプログラミングモデルを構築します
- **階層性**：象に先天と後天の区別があるように、YaoXiangの型システムは明確な階層構造を持ち、原型からジェネクスまで、値からメタ型まで
- **変化性**：陰陽の流転、変化窮まりないように、YaoXiangは依存型をサポートし、型が値に応じて進化することを許容します
- **認識可能性**：卦象は解読可能、万物は象せるように、YaoXiangは完全な型リフレクション能力を提供し、実行時の型情報が完全に利用可能です
- **証明可能性**：卦象が事物の法則を明らかにするように、YaoXiangの型システムはCurry-Howard同型に従い（型即ち命題、プログラム即ち証明）、型チェックのプロセスが論理証明の検証となります

## 二、コア哲学と原則

以下の設計信条はYaoXiangの基石であり、**妥協不可、違反不可**です。すべての機能提案はこれらの原則の検証を経る必要があります。

### 2.1 原則一：すべては型

YaoXiangの世界観において、型は最も高次の抽象ユニットであり、言語を貫くコアコンセプトです。

**具体的な体现**：

- **値は型のインスタンスである**：`42`は`Int`型のインスタンス、`"hello"`は`String`型のインスタンス
- **型自体も型である**：メタ型（`type`）はすべて型の型であり、`Int`は`type`のインスタンス
- **関数は型マッピングである**：`fn add(Int, Int) -> Int`は`Int × Int`から`Int`への型マッピングを記述します
- **モジュールは型コンポジションである**：モジュールは関数と型を含む名前空間コンポジションです

**妥協不可の理由**：統一的型抽象は言語セマンティクスを簡素化し、値と型の二元対立を排除し、型システムをコードの正確性の守護者而非绊脚石とします。

### 2.2 原則二：厳格な構造化

YaoXiangの構文設計は「曖昧さのない、予測可能な、解析しやすい」を追求します。

**具体的なルール**：

- **強制4スペースインデント**：Tab文字の使用を禁止し、コードブロック境界が一目瞭然
- **括弧は省略不可**：関数引数は括弧が必要、リスト要素にはカンマが必要
- **コードブロックには波括弧が必要**：`if`、`while`、`for`などの制御フローは`{ }`で包裹する必要があります
- **キーワード数の精简**：17個のコアキーワードのみを維持し、構文糖の氾濫を拒否

**妥協不可の理由**：厳格な構造化は3つの重要な優位性をもたらします——（1）IDEの構文ハイライトとコードフォールドがより正確；（2）AIコード生成と修正の精度が大幅に向上；（3）新規学習者がコード構造を迅速に理解できます。

### 2.3 原則三：ゼロコスト抽象

高次の抽象は実行時の性能オーバーヘッドをもたらすべきではありません。

**具体的な保証**：

- **単態化**：ジェネリック関数はコンパイル時に具体バージョンに展開され、仮想テーブル検索オーバーヘッドなし
- **インライン最適化**：単純関数は自動インライン化され、関数呼び出しオーバーヘッドを消除
- **スタック配置優先**：小オブジェクトはデフォルトでスタック配置、ヒープ配置は必要な場合のみ使用
- **GCなし**：所有権モデルがメモリ安全を保証し、ガベージコレクタの実行時オーバーヘッドなし

**妥協不可の理由**：性能はプログラミング言語の生存底线です。便利さを得るために性能を犠牲にする設計はすべて、程序员への裏切りです。

### 2.4 原則四：デフォルト不変

可変性と複雑性は影を伴うように付き物です。YaoXiangはデフォルト不変を選択し、コードをより推論し理解しやすくします。

**具体的なルール**：

- 変数はデフォルトで不変であり、代入後は変更不可
- 可変が必要な場合は明示的に`mut`を宣言する必要がある
- 参照はデフォルトで不変であり、可変参照には`mut`マークが必要
- 所有権の移転は元のバインディングの失效を意味する

**妥協不可の理由**：不変性は並行安全な基盤であり、コード可読性の保障であり、関数型プログラミングの知恵の結晶です。

### 2.5 原則五：型即ちデータ

型情報はコンパイル時だけでなく、実行時に完全に利用可能です。

**具体的な能力**：

- 実行時型クエリ：任意の値から型情報を取得可能
- 型リフレクション：型自体を構成して操作可能
- パターン一致によるデストラクト：型コンストラクタは直接パターン一致に使用可能
- ジェネリック特化：実行時にジェネリック引数の具体化型を取得可能

**妥協不可の理由**：完全な型リフレクション能力はメタプログラミングの基盤であり、高性能フレームワークとツールの基石です。

## 三、重要なイノベーションと機能

YaoXiangは既存の言語の優れた特性を吸取りながら、以下の革新的設計を提案します。

### 3.1 イノベーション一：統一的型構文

**従来の言語の型定義**は往往にして複数のキーワードが必要です：

```rust
// Rust
struct Point { x: f64, y: f64 }
enum Result<T, E> { Ok(T), Err(E) }
enum Color { Red, Green, Blue }
union IntOrFloat { i: i32, f: f32 }
```

**YaoXiangの統一的構文**：キーワードは`type`のみで、`{}`でデータ構造を定義し、`[]`でインターフェース型を定義します。

```yaoxiang
# === データ型（波括弧） ===

# 構造体
type Point = { x: Float, y: Float }

# 列挙型（複数のコンストラクタ）
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# ゼロ引数コンストラクタ
type Color = { red | green | blue }

# 混合型
type Shape = { circle(Float) | rect(Float, Float) }

# === インターフェース型（角括弧） ===

# インターフェース定義：メソッドシグネチャの集合
type Serializable = [
    serialize() -> String
]

type Drawable = [
    draw(Surface) -> Void,
    bounding_box() -> Rect
]

# === ジェネクス ===

Option: (T: Type) -> Type = { some(T) | none }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }
```

**イノベーション価値**：統一的型構文で`enum`/`struct`/`union`/`trait`キーワードの断片化を消除。

### 3.2 イノベーション二：コンストラクタ即ち型

**値構成とパターン一致は完全に同一です**：

```yaoxiang
# 型定義
type Point = { x: Float, y: Float }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# 値構成：関数呼び出しと同じ
p: Point = Point(3.0, 4.0)
r: Result[Int, String] = ok(42)
err_msg: Result[Int, String] = err("not found")
c: Color = green

# パターン一致：直接デストラクト
match point {
    Point(0.0, 0.0) -> "origin"
    Point(x, y) -> "point at ({x}, {y})"
}
```

### 3.3 イノベーション三：カリー化されたメソッドバインディング

YaoXiangは純粋関数型設計を採用し、カリー化を通じてオブジェクトメソッド呼び出しのような糖衣構文を実現し、`class`と`method`キーワードを導入する必要がありません。

```yaoxiang
# === Point.yx ===

# 型定義
type Point = { x: Float, y: Float }

# コア関数：ユークリッド距離
distance(Point, Point) -> Float = (a, b) => {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# メソッド糖衣構文バインディング
Point.distance = distance  # デフォルトで第0引数にバインディング
```

```yaoxiang
# === main.yx ===

use Point

main() -> Void = () => {
    p1 = Point(3.0, 4.0)
    p2 = Point(1.0, 2.0)

    # 2つの呼び出し方法は完全に同値
    d1 = distance(p1, p2)      # コア関数を直接呼び出し
    d2 = p1.distance(p2)       # メソッド糖衣構文

    # 関数型用法：最初の引数を事前バインディング
    dist_from_origin = Point.distance(Point(0.0, 0.0))
    result = dist_from_origin(p1)   # 5.0

    # カリー化用法：遅延評価
    get_dist_to_p2 = p2.distance(_)
    d3 = get_dist_to_p2(p1)         # 2.828
}
```

**イノベーション価値**：純粋関数型設計で、隠れた`self`パラメータがなく、関数は値であり自由に传递と組合が可能。

### 3.4 イノベーション四：spawnモデル

> 「万物並作，吾以観復。」——《易·復卦》
>
> spawnモデルはこれに寓意し、プログラミングパラダイムを描述します：開発者は同期的、順序的な思考でロジックを記述し、言語実行時はその中の計算ユニットを万物並作のように自動的かつ効率的に並行実行させ、最終的に統一的に協調します。

**コア3原則**：

| 原則 | 説明 |
|------|------|
| **同期構文** | 見るがままの順序コード |
| **並行本質** | 実行時に自動的に並行性を抽出 |
| **統一的協調** | 結果が必要時に自動的に集結し、ロジックの正確性を保証 |

**用語体系**：

| 公式用語 | 対応構文 | 説明 |
|----------|----------|------|
| **spawn関数** | `spawn (params) => body` | 並行実行に参加できる計算ユニットを定義 |
| **spawnブロック** | `spawn { a(), b() }` | 明示的に宣言された並行領域、ブロック内タスクがspawn実行 |
| **spawnループ** | `spawn for x in xs { ... }` | データ並行、ループ体がすべての要素でspawn実行 |
| **spawn値** | `Async(T)` | spawn中の未来値、使用時に自動待機 |
| **spawnグラフ** | 遅延計算グラフ(DAG) | spawnが発生する舞台、依存関係と並行関係を描述 |
| **spawnスケジューラ** | 実行時タスクスケジューラ | 万物を調整し、正しいタイミングでそれらを並作させる知的中枢 |

> **詳細については**：[RFC-001 spawnモデル](./rfc/001-concurrent-model-error-handling.md)

```yaoxiang
# === spawn関数 ===
# spawnマーク付きの関数（RFC-003構文）
fetch_data: String -> JSON = spawn (url) => {
    HTTP.get(url).json()
}

# === spawnブロック ===
# spawn { } 内の式は強制的に並行実行
compute_all: () -> (Int, Int, Int) = spawn () => {
    (a, b, c) = spawn {
        heavy_calc(1),    # タスク 1
        heavy_calc(2),    # タスク 2
        another_calc(3)   # タスク 3
    }
    (a, b, c)
}

# === spawnループ ===
# spawn forマークのループは自動的に並行化
parallel_sum: Int -> Int = spawn (n) => {
    total = spawn for i in 0..n {
        fibonacci(i)  # 各反復が並行実行
    }
    total
}

# === 自動待機 ===
main() -> Void = () => {
    # 2つの独立リクエストは自動的に並行実行
    users = fetch_data("https://api.example.com/users")
    posts = fetch_data("https://api.example.com/posts")

    # 待機点は結果が必要時に自動的に挿入
    print(users.length + posts.length)  # usersとpostsを自動待機
}
```

**スレッド安全**：

```yaoxiang
# Send/Sync制約がコンパイル時のスレッド安全を保証
type SafeCounter = SafeCounter(mutex: Mutex(Int))

main: () -> Void = () => {
    counter: Arc[SafeCounter] = Arc.new(SafeCounter(Mutex.new(0)))

    # spawnは自動的にSend制約をチェック
    spawn () => {
        guard = counter.mutex.lock()
        guard.value = guard.value + 1
    }
}
```

**技術ドキュメント**：
- 詳細については [RFC-001 spawnモデル](./rfc/001-concurrent-model-error-handling.md)

**イノベーション価値**：非同期プログラミングの認知負荷がゼロになり、コード可読性は同期コードと完全に同一で、同時に高性能並行の実行効率を獲得。

```yaoxiang
# デフォルト Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p              # Move、所有権移転、pは失效

# 明示的なref = Arc（安全共有）
shared = ref p      # Arc、スレッド安全

spawn(() => print(shared.x))   # ✅ 安全

# 明示的なclone() = コピー
p3 = p.clone()      # pとp3は独立
```

**コアルール**：
- デフォルト Move（ゼロコピー）
- 共有には`ref`（Arc）を使用
- コピーには`clone()`を使用
- システムレベルには`unsafe` + `*T`を使用

### 4.7 並行プログラミング（SeamlessAsync）

```yaoxiang
# spawnマーク付き非同期関数
fetch_api: String -> JSON spawn = (url) => {
    response = HTTP.get(url)
    JSON.parse(response.body)
}

# 並行構築ブロック：明示的並行
process_all: () -> (JSON, JSON, JSON) spawn = () => {
    (a, b, c) = spawn {
        fetch_api("https://api1.com/data"),
        fetch_api("https://api2.com/data"),
        fetch_api("https://api3.com/data")
    }
    (a, b, c)
}

# データ並行ループ
parallel_process: Int -> Int spawn = (n) => {
    total = spawn for i in 0..n {
        compute(i)
    }
    total
}

# スレッド安全の例
type ThreadSafeCounter = { value: Mutex(Int) }

main: () -> Void = () => {
    counter = ThreadSafeCounter(Mutex.new(0))

    # spawnは自動的にSend制約をチェック
    spawn () => {
        guard = counter.value.lock()
        guard.value = guard.value + 1
    }
}
```

## 付録C：よくある質問

**Q: YaoXiangはRustと比較してどのような優位性がありますか？**

A: YaoXiangはRustのメモリ安全とゼロコスト抽象を維持しながら、よりシンプルでより低い認知負荷の構文を採用しています。**spawnモデル**はRustの`async/await`よりも簡潔です——`spawn`マークを1つ追加するだけで済み、FutureとPinの手動管理は不要。「万物並作，吾以観復」、並行プログラミングを自然法則を描述するように直感的にします。Send/Sync制約が同等のスレッド安全保証を提供します。統一的型構文で`enum`/`struct`/`union`の概念の断片化を消除。

**Q: YaoXiangはどのような类型的開発に適していますか？**

A: システムプログラミング、アプリケーション開発、Webサービス、スクリプトツール、AI補助プログラミング。汎用プログラミング言語成为することが目標です。

**Q: なぜ4スペースインデントを選んだのですか？**

A: 4スペースは明確なコードブロックの視覚的区切りを提供し、ネストの深さ带来的混乱减少了。これは熟考された「AIフレンドリ」設計判断です。

**Q: 1.0バージョンはいつリリースされますか？**

A: v1.0目標：プロダクション利用可能。リリース時期は実装進捗に依存します。詳細については [バージョン計画RFC](./rfc/003-version-planning.md) を参照。

**Q: コアチームにはどのように連絡できますか？**

A: GitHub DiscussionsまたはDiscordコミュニティチャネルを通じて連絡できます。コアチームメンバーは定期的に返信します。
