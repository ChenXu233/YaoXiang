## 一、なぜ YaoXiang を創造したのか？

### 1.1 埋めていた言語の空白

プログラミング言語の歴史において、数多くの優れた言語の誕生と進化を目の当たりにしてきた：C言語はシステムプログラミングの効率革命をもたらし、Pythonは誰もが学べるプログラミング体験を作り出し、Rustはメモリの安全とパフォーマンスの両立を証明し、TypeScriptは大規模なフロントエンドプロジェクトを維持可能にした。しかし、今日の言語エコシステムを眺めるとき，仍然发现一个明显的断层地带——**没有任何一门语言能够同时满足以下三个核心需求**：

| 必要性 | 既存ソリューションの問題点 |
|------|-------------------|
| **型安全性** | Rust は過度に厳格で、学習曲線が険しい；TypeScript はオプショナル型で、コンパイル時の保証を提供できない |
| **自然な構文** | Rust の構文は複雑で難解；Haskell の関数型は敷居が高い；従来の静的言語は冗長で烦雑 |
| **AI フレンドリー** | 既存言語の構文には曖昧さが多い、AST が複雑で、隠れた動作が予測しづらく、AI によるコード生成と修正の精度が制限される |

YaoXiang の誕生は、まさにこの空白を埋めるためである。我々は信じている：**プログラミング言語は、強力で親しみやすく、安全で効率的であり、厳密で美しいものであるべきだ**。

### 1.2 解決しようとする实际问题

**問題一：型システムの断片化**

現代のプログラミング言語は、型システムにおいて深刻な断片化を見せている。静的型付け言語はコンパイル時の絶対的な正しさを追求するが、往々にして開発効率を犠牲にする代償を払う。動的型付け言語は柔軟性を提供するが、大規模プロジェクトではメンテナンス困難な欠陥が露呈する。YaoXiang は「すべては型である」という統一抽象フレームワークを提案し、型を言語設計の主線として貫き、事後に追加されたパッチではなくする。

**問題二：メモリの安全とパフォーマンスの二択**

長い間、開発者はメモリの安全と実行パフォーマンスの間で難しい選択を強いられてきた。GC（ガベージコレクション）は開発者を解放するが、遅延の変動とメモリオーバーヘッドをもたらす。手動のメモリ管理は高效だが、綱渡りのように危険を伴う。YaoXiang は Rust スタイル的所有権モデルを採用し、コンパイル期にデータ競合とメモリリークを消除しながら、ゼロコスト抽象を維持し、GC なしで高性能を実現する。

**問題三：非同期プログラミングの認知負荷**

現代のアプリケーションはネットワークと並行処理を避けては通れないが、非同期プログラミングは常にプログラマーの悪夢であった。コールバック関数のネスト、Promise のチェーン呼び出し、async/await 構文——各ソリューションがコードの複雑性を増大させる。YaoXiang は非同期モデルを重新設計した：関数シグネチャの後に `spawn` マークを付けるだけで、コンパイラがすべての非同期詳細を自動処理し、並行プログラミングを同期コードと同じくらい自然に実現する。

**問題四：AI 支援プログラミングのボトルネック**

AI が開発者を支援してコードを書くようになった今、言語設計の選択は極めて重要になる。曖昧な構文ルール、暗黙の型変換、複雑な糖衣構文——これらは人間のプログラマーがすでに慣れている特徴だが、AI の理解と生成の障害となる。YaoXiang は設計当初から「AI フレンドリー」をコア目標とした：厳格なインデントルール、明確なコードブロック境界、曖昧さのない構文構造により、AI が正確に理解し、生成し、修正できるようにする。

### 1.3 言語の哲学的基盤

YaoXiang という名前は、易経の「爻」と「象」に由来する。「爻」は卦象を構成する基本記号であり、陰陽の変化、動静の相生の象徴である。「象」は事物の本質 внешнее проявлениеであり、万象万物を包含し包摂するものである。

この哲学的思想は言語設計のすべての細部体现在されている：

- **統一性**：爻卦のシンプルな記号が複雑な卦象を構成するように、YaoXiang は少数のコア概念（型、関数、コンストラクタ）で完全なプログラミングモデルを構築する
- **階層性**：象先后天之分があるように、YaoXiang の型システムには明確な階層構造があり、原型からジェネリックへ、値からメタ型へ
- **変化性**：陰陽流転、変化無窮であるように、YaoXiang は依存型をサポートし、型が値に応じて進化することを許容する
- **認識可能性**：卦象は解けるように、万物は象るように、YaoXiang は完全な型リフレクション能力を提供し、実行時の型情報が完全に利用可能
- **証明可能性**：卦象が事物の法則を明らかにするように、YaoXiang の型システムは Curry-Howard 同型に従い（型は命題であり、プログラムは証明である）、型検査の過程は論理証明の検証である

## 二、コア哲学と原則

以下の設計信条は YaoXiang の基石であり、**妥協せず、違反しない**。すべての機能提案は必ずこれらの原則の检验を受けなければならない。

### 2.1 原則一：すべては型

YaoXiang の世界観において、型は最も高次の抽象ユニットであり、言語を貫くコアコンセプトである。

**具体的な體現**：

- **値は型のインスタンスである**：`42` は `Int` 型のインスタンスであり、`"hello"` は `String` 型のインスタンスである
- **型自体も型である**：メタ型（`type`）はすべての型の型であり、`Int` は `type` のインスタンスである
- **関数は型マッピングである**：`fn add(Int, Int) -> Int` は `Int × Int` から `Int` への型マッピングを記述する
- **モジュールは型組合である**：モジュールは関数と型を含む名前空間组合である

**妥協できない理由**：統一型抽象は言語セマンティクスを単純化し、値と型の二元論を消除し、型システムをコードの正しさの守護者而非絆足石とする。

### 2.2 原則二：厳格な構造化

YaoXiang の構文設計は「曖昧さのない、予測可能な、易解析な」ことを追求する。

**具体的なルール**：

- **強制4スペースインデント**：タブ文字の使用を禁止し、コードブロックの境界が一目でわかる
- **括弧は省略不可**：関数引数には括弧が必要で、リスト要素にはカンマが必要
- **コードブロックには波括弧が必要**：`if`、`while`、`for` などの制御フローには `{ }` でラップする必要がある
- **キーワード数を精简**：17個のコアキーワードのみを保持し、糖衣構文の氾濫を拒否する

**妥協できない理由**：厳格な構造化は三つの重要な優位性をもたらす——（1）IDE の構文ハイライトとコードフォールドがより正確；（2）AI コード生成と修正の精度が大幅に向上；（3）新規学習者がコード構造を急速に理解できる。

### 2.3 原則三：ゼロコスト抽象

高次の抽象は実行時のパフォーマンスオーバーヘッドをもたらすべきではない。

**具体的な保証**：

- **単態化**：ジェネリック関数はコンパイル時に具体的なバージョンに展開され、仮想テーブル参照オーバーヘッドがない
- **インライン最適化**：シンプルな関数は自動的にインライン化され、関数呼び出しオーバーヘッドを消除
- **スタック分配優先**：小さなオブジェクトはデフォルトでスタック分配し、ヒープ分配は本当に必要な場合のみ使用
- **GC なし**：所有権モデルがメモリの安全を保証し、ガベージコレクタの実行時オーバーヘッドが不要

**妥協できない理由**：パフォーマンスはプログラミング言語の生存底线である。便利さを得るためにパフォーマンスを犠牲にする設計はすべて、程序员への裏切りである。

### 2.4 原則四：デフォルトで不変

可変性と複雑性は表裏一体である。YaoXiang はデフォルトで不変を選択し、コードをより推論可能で理解しやすくする。

**具体的なルール**：

- 変数はデフォルトで不変であり、代入後は変更できない
- 可変が必要な場合は `mut` を明示的に宣言する必要がある
- 参照はデフォルトで不変であり、可変参照には `mut` マークが必要
- 所有権の移転は元のバインディングが無効になることを意味する

**妥協できない理由**：不変性は並行処理の安全性の基礎であり、コード可読性の保障であり、関数型プログラミングの知恵の結晶である。

### 2.5 原則五：型はデータである

型情報はコンパイル時のみ存在するべきではなく、実行時に完全に利用可能であるべきである。

**具体的な能力**：

- 実行時型クエリ：任意の値からその型情報を取得できる
- 型リフレクション：型自体を構成して操作できる
- パターン照合による分解：型コンストラクタを直接パターン照合に使用できる
- ジェネリック特化：実行時にジェネリックパラメータの具体化型を取得できる

**妥協できない理由**：完全な型リフレクション能力はメタプログラミングの基礎であり、高性能フレームワークとツールの基石である。

## 三、主要な革新と機能

YaoXiang は既存言語の優れた特性を吸收的同时に、以下の革新的設計を提案する。

### 3.1 革新一：統一型構文

**従来の言語の型定義**は往往にして複数のキーワードを必要とする：

```rust
// Rust
struct Point { x: f64, y: f64 }
enum Result<T, E> { Ok(T), Err(E) }
enum Color { Red, Green, Blue }
union IntOrFloat { i: i32, f: f32 }
```

**YaoXiang の統一構文**：`type` キーワードのみ、`{}` でデータ構造を定義、`[]` でインターフェース型を定義。

```yaoxiang
# === データ型（波括弧） ===

# 構造体
type Point = { x: Float, y: Float }

# 列挙型（複数のコンストラクタ）
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# ゼロ引数コンストラクタ
type Color = { red | green | blue }

# 混合型
type Shape = { circle(Float) | rect(Float, Float) }

# === インターフェース型（角括弧） ===

# インターフェース定義：メソッドシグネチャの集合
type Serializable = [
    serialize() -> String
]

type Drawable = [
    draw(Surface) -> Void,
    bounding_box() -> Rect
]

# === ジェネリック ===

Option: (T: Type) -> Type = { some(T) | none }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }
```

**革新価値**：統一型構文で `enum`/`struct`/`union`/`trait` キーワードの断片化を消除する。

### 3.2 革新二：コンストラクタは型である

**値構成とパターン照合は完全に同一**：

```yaoxiang
# 型定義
type Point = { x: Float, y: Float }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# 値構成：関数呼び出しと同じ
p: Point = Point(3.0, 4.0)
r: Result[Int, String] = ok(42)
err_msg: Result[Int, String] = err("not found")
c: Color = green

# パターン照合：直接分解
match point {
    Point(0.0, 0.0) -> "origin"
    Point(x, y) -> "point at ({x}, {y})"
}
```

### 3.3 革新三：カリー化されたメソッドバインディング

YaoXiang は純粋関数型設計を採用し、カリー化を通じてオブジェクトメソッド呼び出しのような糖衣構文を実現し、`class` と `method` キーワードを導入する必要がない。

```yaoxiang
# === Point.yx ===

# 型定義
type Point = { x: Float, y: Float }

# コア関数：ユークリッド距離
distance(Point, Point) -> Float = (a, b) => {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# メソッド糖衣バインディング
Point.distance = distance  # デフォルトで第0引数にバインディング
```

```yaoxiang
# === main.yx ===

use Point

main() -> Void = () => {
    p1 = Point(3.0, 4.0)
    p2 = Point(1.0, 2.0)

    # 2つの呼び出し方式は完全に等価
    d1 = distance(p1, p2)      # コア関数を直接呼び出す
    d2 = p1.distance(p2)       # メソッド糖衣

    # 関数型的用法：最初の引数を事前にバインディング
    dist_from_origin = Point.distance(Point(0.0, 0.0))
    result = dist_from_origin(p1)   # 5.0

    # カリー化用法：遅延評価
    get_dist_to_p2 = p2.distance(_)
    d3 = get_dist_to_p2(p1)         # 2.828
}
```

**革新価値**：純粋関数型設計で、`self` パラメータの隠蔽がなく、関数は値として自由に传递と组合が可能。

### 3.4 革新四：並作モデル

> 「万物並作，吾以觀復。」——《易·復卦》
>
> 並作モデルはここに寓意し、プログラミングパラダイムを記述する：開発者は同期的で順序だった思考でロジックを記述し、言語ランタイムはその中の計算ユニットを万物並作のように自動的で効率的に並行実行させ、最終的に統一的に協調する。

**コア三原則**：

| 原則 | 説明 |
|------|------|
| **同期構文** | 目に見えるものが得られる順序コード |
| **並行本質** | 実行時に自動的に並行性を抽出 |
| **統一協調** | 結果は必要な時に自動的に集結し、ロジックの正しさを保証 |

**用語体系**：

| 公式用語 | 対応構文 | 説明 |
|----------|----------|------|
| **並作関数** | `spawn (params) => body` | 並作実行に参加できる計算ユニットを定義 |
| **並作ブロック** | `spawn { a(), b() }` | 明示的に宣言された並行領域、ブロック内のタスクは並作実行 |
| **並作ループ** | `spawn for x in xs { ... }` | データ並列、ループ体がすべての要素で並作実行 |
| **並作値** | `Async(T)` | 並作中の未来値、使用時に自動待機 |
| **並作グラフ** | 遅延計算グラフ(DAG) | 並作が発生する舞台、依存関係と並列関係を記述 |
| **並作スケジューラ** | ランタイムタスクスケジューラ | 万物を調整し、正しいタイミングで並作させるIntelligent Hub |

> **詳細については**：[RFC-001 並作モデル](./rfc/001-concurrent-model-error-handling.md)

```yaoxiang
# === 並作関数 ===
# spawn マーク付き関数（RFC-003 構文）
fetch_data: String -> JSON = spawn (url) => {
    HTTP.get(url).json()
}

# === 並作ブロック ===
# spawn { } 内の式は強制的に並行実行
compute_all: () -> (Int, Int, Int) = spawn () => {
    (a, b, c) = spawn {
        heavy_calc(1),    # タスク 1
        heavy_calc(2),    # タスク 2
        another_calc(3)   # タスク 3
    }
    (a, b, c)
}

# === 並作ループ ===
# spawn for マーク付きループは自動並列化
parallel_sum: Int -> Int = spawn (n) => {
    total = spawn for i in 0..n {
        fibonacci(i)  # 各反復を並行実行
    }
    total
}

# === 自動待機 ===
main() -> Void = () => {
    # 2つの独立リクエストは自動並行実行
    users = fetch_data("https://api.example.com/users")
    posts = fetch_data("https://api.example.com/posts")

    # 待機点は結果が必要な時に自動挿入
    print(users.length + posts.length)  # users と posts を自動待機
}
```

**スレッド安全性**：

```yaoxiang
# Send/Sync 制約はコンパイル時にスレッド安全性を保証
type SafeCounter = SafeCounter(mutex: Mutex(Int))

main: () -> Void = () => {
    counter: Arc[SafeCounter] = Arc.new(SafeCounter(Mutex.new(0)))

    # spawn は自動的に Send 制約をチェック
    spawn () => {
        guard = counter.mutex.lock()
        guard.value = guard.value + 1
    }
}
```

**技術ドキュメント**：
- 詳細については [RFC-001 並作モデル](./rfc/001-concurrent-model-error-handling.md)

**革新価値**：非同期プログラミングの認知負荷がゼロになり、コード可読性は同期コードと完全に同じになりながら、高性能並列の実装効率を得る。

### 3.5 革新五：依存型サポート（将来機能）

> **状態**：v1.0 以後に実装予定

型は値に依存でき、真の型駆動開発を実現する。

```yaoxiang
# 固定長ベクトル（将来構文）
Vector: (T: Type, n: Int) -> Type = {
    data: [T; n]
    length: n
}

# 使用
vec: Vector[Int, 3] = Vector([1, 2, 3], 3)

# 型検査
# vec: Vector[Int, 3] = Vector([1, 2], 2)  # コンパイルエラー！長さが一致しない
```

**革新価値**：コンパイル期により多くのエラーを捕获し、より精确な型保証を実現する。

### 3.6 革新六：極限まで精简されたキーワード設計

YaoXiang はわずか18個のコアキーワードのみを定義し、メインストリーム言語よりはるかに少ない：

```
type   pub    use    spawn
ref    mut    if     elif
else   match  while  for    return
break  continue as     in     unsafe
```

| 比較言語 | キーワード数 |
|----------|-----------|
| YaoXiang | **18** |
| Rust | 51+ |
| Python | 35 |
| TypeScript | 64+ |
| Go | 25 |

**革新価値**：より低い記憶負担、より一貫した構文スタイル、より解析しやすい構文構造。

### 4.4 リスト内包表記

```yaoxiang
# Python スタイルのリスト内包表記
evens = [x * 2 for x in 0..10]          # [0, 4, 8, 12, 16]
squares = [x * x for x in 1..10 if x % 2 == 1]  # [1, 9, 25, 49, 81]

# ネストした内包表記
matrix = [[i * j for j in 1..4] for i in 1..3]
# [[1, 2, 3], [2, 4, 6], [3, 6, 9]]
```

### 4.5 所有権モデル

```yaoxiang
type Point = { x: Float, y: Float }

# デフォルト Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p              # Move、所有権移転、p は無効

# 明示的 ref = Arc（安全な共有）
shared = ref p      # Arc、スレッドセーフ

spawn(() => print(shared.x))   # ✅ 安全

# 明示的 clone() = コピー
p3 = p.clone()      # p と p3 は独立
```

**コアルール**：
- デフォルト Move（ゼロコピー）
- 共有は `ref`（Arc）を使用
- コピーは `clone()` を使用
- システムレベルは `unsafe` + `*T` を使用

```yaoxiang
# Result 型
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

    }
}

# ? 演算子の使用
process: () -> Result[Int, String] = () => {
    a = read_number()?
    b = read_number()?

```

### 4.7 並行プログラミング（SeamlessAsync）

```yaoxiang
# spawn で非同期関数をマーク
fetch_api: String -> JSON spawn = (url) => {
    response = HTTP.get(url)
    JSON.parse(response.body)
}

# 並行構成ブロック：明示的並列
process_all: () -> (JSON, JSON, JSON) spawn = () => {
    (a, b, c) = spawn {
        fetch_api("https://api1.com/data"),
        fetch_api("https://api2.com/data"),
        fetch_api("https://api3.com/data")
    }
    (a, b, c)
}

# データ並列ループ
parallel_process: Int -> Int spawn = (n) => {
    total = spawn for i in 0..n {
        compute(i)
    }
    total
}

# スレッド安全性の例
type ThreadSafeCounter = { value: Mutex(Int) }

main: () -> Void = () => {
    counter = ThreadSafeCounter(Mutex.new(0))

    # spawn は自動的に Send 制約をチェック
    spawn () => {
        guard = counter.value.lock()
        guard.value = guard.value + 1
    }
}
```

## 付録C：よくある質問

**Q: YaoXiang は Rust と比較してどのような優位性がありますか？**

A: YaoXiang は Rust のメモリ安全とゼロコスト抽象を保持しますが、よりシンプルで低い認知負荷の構文を採用しています。**並作モデル**は Rust の `async/await` より簡潔です——`spawn` マークを1つ追加するだけで済み、Future と Pin を手動で管理する必要がありません。「万物並作，吾以觀復」、並行プログラミングを自然法則を記述するように直感的にします。Send/Sync 制約は同等のスレッド安全性を保証します。統一型構文は `enum`/`struct`/`union` の概念の断片化を消除します。

**Q: YaoXiang はどのような種類の開発に適していますか？**

A: システムプログラミング、应用開発、Web サービス、スクリプトツール、AI 支援プログラミング。目標は汎用プログラミング言語になることです。

**Q: なぜ4スペースインデントを選んだのですか？**

A: 4スペースは明確なコードブロックの視覚的区切りを提供し、ネスト深度の増加带来的混乱减少了。这是深思熟虑后的「AI フレンドリー」設計决定です。

**Q: 1.0 バージョンはいつリリースされますか？**

A: v1.0 目標：プロダクション利用可能。リリース時間は実装進捗に依存します。詳細については [バージョン計画 RFC](./rfc/003-version-planning.md) を参照してください。

**Q: コアチームにはどのように連絡できますか？**

A: GitHub Discussions または Discord コミュニティチャンネルを通じて。コアチームメンバーは定期的に返信します。
