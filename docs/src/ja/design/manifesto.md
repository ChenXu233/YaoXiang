## 一、なぜ YaoXiang を創造するのか？

### 1.1 埋める言語の空白

プログラミング言語の歴史において、数多くの優れた言語の誕生と進化を目の当たりにしてきた。C 言語はシステムプログラミングの効率革命をもたらし、Python は誰もが学べるプログラミング体験を創出し、Rust はメモリ安全性とパフォーマンスの両立を証明し、TypeScript は大規模フロントエンドプロジェクトを保守可能にした。しかし、今日の言語エコシステムを検証すると、依然として明白な断層地帯が存在する——**いかなる言語も以下の三つの核心的なニーズを同時に満たすことができていない**：

| ニーズ | 既存ソリューションの問題 |
|------|-------------------|
| **型安全性** | Rust は厳格すぎ、学習曲線が急峻；TypeScript はオプショナル型で、編訳時保証を提供できない |
| **自然な構文** | Rust の構文は複雑で難解；Haskell の関数型は敷居が高い；従来の静的言語は冗長で煩雑 |
| **AI フレンドリー** | 既存言語は構文の曖昧性、AST の複雑さ、予測困難な隠れた動作が多く、AI によるコード生成と修正の精度が制限される |

YaoXiang の誕生は、まさにこの空白を埋めるためである。我々は信じる：**プログラミング言語は強大でありながら親しみやすく、安全でありながら効率的であり、厳密でありながら優雅であるべきである**。

### 1.2 解決する実際的な問題

**問題一：型システムの断片化**

今日のプログラミング言語は型システムにおいて深刻な断片化を見せている。静的型付き言語は編訳時の絶対的な正しさを追求するが、しばしば開発効率を犠牲にすることで達成される；動的型付き言語は柔軟性を提供するが、大規模プロジェクトでは保守困難な欠陥を露呈する。YaoXiang は「すべては型なり」の統一抽象フレームワークを提案し、型を言語設計全体を貫く主軸とし、事後的に追加されるパッチとはしない。

**問題二：メモリ安全性とパフォーマンスの二者択一**

長い間、開発者はメモリ安全性と実行パフォーマンスの間で困難な選択を迫られてきた。GC（ガベージコレクション）は開発者を解放するものの、レイテンシの変動とメモリオーバーヘッドをもたらす；手動メモリ管理は効率的だが、綱渡り同様に危険である。YaoXiang は Rust スタイルの所有権モデルを採用し、編訳時にデータ競合とメモリリークを排除しつつ、ゼロコスト抽象を実現し、GC なしで高性能を達成する。

**問題三：非同期プログラミングの認知負荷**

現代アプリケーションはネットワークと並行処理に依存しており、非同期プログラミングは常にプログラマーにとっての悪夢であった。コールバック関数のネスト、Promise のチェーン呼び出し、async/await 構文——いずれのソリューションもコードの複雑性を増大させる。YaoXiang は非同期モデルを再設計した：関数シグネチャの後に `spawn` マーカーを追加するだけで、コンパイラがすべての非同期詳細を自動処理し、並行プログラミングを同期コードのように自然にする。

**問題四：AI 支援プログラミングのボトルネック**

AI が開発者のコード作成を支援するようになった現在、言語設計の選択は極めて重要になる。曖昧な構文ルール、暗黙的な型変換、複雑なシンタックスシュガー——これらは人間のプログラマーがすでに慣れた特性でありながら、AI の理解と生成の障害となっている。YaoXiang は設計当初から「AI フレンドリー」をコア目標とする：厳格なインデントルール、明示的なコードブロック境界、曖昧さのない構文構造により、AI がコードを正確に理解し、生成し、修正できるようにする。

### 1.3 言語の哲学的基盤

YaoXiang の名前は『易経』の「爻」と「象」に由来する。「爻」は卦象を構成する基本記号であり、陰陽の変化、動静の相生を象徴する；「象」は事物本質外在的表現であり、万象万物を代表する。

この哲学的思想は言語設計のすべての細部に体现されている：

- **統一性**：爻卦の単純な記号が複雑な卦象を構成するように、YaoXiang は少数のコア概念（型、関数、コンストラクタ）で完全なプログラミングモデルを構築する
- **階層性**：象に先天後天の区別があるように、YaoXiang の型システムは明確な階層構造を持ち、原型から generics、値から元型まで展開する
- **可変性**：陰陽が流転し変化无穷であるように、YaoXiang は依存型をサポートし、型の値による変化と evolution を許す
- **識別可能性**：卦象が解け万物に象があるように、YaoXiang は完全な型リフレクション能力を提供し、ランタイム型情報は完全に利用可能
- **証明可能性**：卦象が事物の法則を明らかにするように、YaoXiang の型システムは Curry-Howard 同型（型は命題、プログラムは証明）に従い、型検査のプロセスが論理証明の検証となる

## 二、コア哲学と原則

以下の設計信条は YaoXiang の礎石であり、**妥協不可、違反不可**である。あらゆる特性提案はこれらの原則の検証を経なければならない。

### 2.1 原則一：すべては型なり

YaoXiang の世界観において、型は最高レベルの抽象ユニットであり、言語を貫くコア概念である。

**具体的な体现**：

- **値は型のインスタンス**：`42` は `Int` 型のインスタンス、`"hello"` は `String` 型のインスタンス
- **型自体も型である**：`Type` は言語唯一の元型キーワードであり、`Int` の類型は `Type`
- **関数は型写像**：`add: (a: Int, b: Int) -> Int` は `Int × Int` から `Int` への型写像を記述する
- **モジュールは型の合成**：モジュールは関数と型を含む名前空間の合成である

**妥協不可の理由**：統一型抽象は言語意味論を簡素化し、値と型の二元対立を排除し、型システムをコード正しさの守護者とし、障害物としない。

### 2.2 原則二：厳格な構造化

YaoXiang の構文設計は「曖昧さなし、予測可能、解析容易」を追求する。

**具体的なルール**：

- **4 スペースインデント強制**：タブ文字の使用を禁止し、コードブロックの境界を一目瞭然とする
- **括弧省略不可**：関数パラメータには括弧が必須、リスト要素にはカンマが必須
- **コードブロックには波括弧が必須**：`if`、`while`、`for` などの制御フローは `{ }` で囲む必要がある
- **キーワード数精简**：17 個のコアキーワードのみ保持し、シンタックスシュガーの泛滥を拒否する

**妥協不可の理由**：厳格な構造化は三つの重要な利点をもたらす——（1）IDE のシンタックスハイライトとコードフォールディングがより正確になる；（2）AI コード生成と修正の精度が大幅に向上する；（3）新しい学習者がコード構造を迅速に理解できる。

### 2.3 原則三：ゼロコスト抽象

高レベルの抽象はランタイムパフォーマンスのオーバーヘッドをもたらすべきではない。

**具体的な保証**：

- **モノモーフィゼーション**：generics 関数はコンパイル時に具体バージョンに展開され、仮想テーブルルックアップのオーバーヘッドがない
- **インライン最適化**：単純な関数は自動的にインライン化され、関数呼び出しのオーバーヘッドが排除される
- **スタック割り当て優先**：小オブジェクトはデフォルトでスタック割り当て、ヒープ割り当ては必要な場合のみ使用
- **GC なし**：所有権モデルがメモリ安全性を保証し、ガベージコレクターのランタイムオーバーヘッドが不要

**妥協不可の理由**：パフォーマンスはプログラミング言語の生存の底线である。利便性のためにパフォーマンスを犠牲にするあらゆる設計はプログラマーへの裏切りである。

### 2.4 原則四：デフォルト不変

可変性と複雑性は切り離せない。YaoXiang はデフォルトを不変と選択し、コードを推論しやすく理解しやすくする。

**具体的なルール**：

- 変数はデフォルトで不変、割り当て後に再変更不可
- 可変が必要な場合は明示的に `mut` を宣言する必要がある
- 参照はデフォルトで不変、可変参照は `mut` マーカーが必要
- 所有権の移転は元のバインディングの失効を意味する

**妥協不可の理由**：不変性は並行安全性の基礎であり、コード可読性の保障であり、関数型プログラミングの知恵の結晶である。

### 2.5 原則五：型はデータなり

型情報は編訳期にのみ存在するべきではなく、ランタイムで完全に利用可能であるべきである。

**具体的な能力**：

- ランタイム型クエリ：あらゆる値は型情報を取得できる
- 型リフレクション：型自体を構築および操作できる
- パターンマッチ分解：型コンストラクタは直接パターンマッチングに使用できる
- generics 特化：ランタイムで generics パラメータの具体化された型を取得できる

**妥協不可の理由**：完全な型リフレクション能力はメタプログラミングの基礎であり、高性能フレームワークとツールの礎石である。

## 三、キーイノベーションと特性

YaoXiang は既存言語の優れた特性を取り入れつつ、以下の innovative な設計を提案する。

### 3.1 イノベーション一：統一型構文

**従来の言語の型定義**はしばしば複数のキーワードを必要とする：

```rust
// Rust
struct Point { x: f64, y: f64 }
enum Result<T, E> { Ok(T), Err(E) }
enum Color { Red, Green, Blue }
trait Drawable { fn draw(&self, s: &Surface); }
```

**YaoXiang の統一構文**：すべては `name: type = value`、`Type` は唯一の元型キーワード。

```yaoxiang
# === 記録型 ===

Point: Type = {
    x: Float,
    y: Float,
}

# デフォルト値を持つフィールド
Point3D: Type = {
    x: Float = 0,
    y: Float = 0,
    z: Float = 0,
}

# === generics 型 ===

Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self,
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,
}

# === インターフェース（フィールドがすべて関数型の記録） ===

Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect,
}

Serializable: Type = {
    serialize: () -> String,
}

# === インターフェース実装（インターフェース名は型体内に記述） ===

Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable,
}

# === メソッド（Type.method 構文） ===

Point.draw: (self: &Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}
```

**イノベーションの価値**：`fn`、`struct`、`enum`、`trait`、`impl` キーワードの断片なし——一つの統一構文がすべての宣言をカバーする。

### 3.2 イノベーション二：コンストラクタは型なり

**値コンストラクタと関数呼び出しは完全に同一**：

```yaoxiang
# 型定義
Point: Type = { x: Float, y: Float }
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self,
}

# 値コンストラクタ：関数呼び出しと同じ
p: Point = Point(3.0, 4.0)
opt: Option(Int) = Option.some(42)
none: Option(Int) = Option.none()

# パターンマッチング：直接分解
match opt {
    Option.some(value) -> print(value)
    Option.none -> print("nothing")
}
```

### 3.3 イノベーション三：カリー化メソッドバインディング

YaoXiang は純粋関数型設計を採用し、カリー化によりオブジェクトメソッド呼び出しに類似したシンタックスシュガーを実現し、`class` と `method` キーワードの導入を必要としない。

```yaoxiang
# === 型定義 ===

Point: Type = {
    x: Float,
    y: Float,
}

# コア関数：ユークリッド距離
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    return (dx * dx + dy * dy).sqrt()
}

# メソッド構文シュガーバインディング（[0] は第 0 パラメータ位置へのバインディングを示す）
Point.distance = distance[0]

# === 使用法 ===

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# 二つの呼び出し方法は完全に等価
d1 = distance(p1, p2)     # コア関数の直接呼び出し
d2 = p1.distance(p2)      # メソッド構文シュガー

# カリー化使用法
dist_from_p1 = p1.distance  # 部分適用、二番目の引数を待機
d3 = dist_from_p1(p2)       # 2.828
```

**イノベーションの価値**：純粋関数型設計、隠れた `self` パラメータなし、関数即値が自由に渡せて合成可能。

### 3.4 イノベーション四：並作（並作）モデル

> 「万物並作、吾以て復を観る。」——『易・復卦』
>
> 並作モデルはその意を取り、プログラミングパラダイムを記述する：開発者は同期・逐次の思考でロジックを記述し、言語ランタイムがその中の計算ユニットを万物が並作するが如く自動的かつ効率的に並行実行させ、最終的に統一協同させる。

**コア三原則**：

| 原則 | 説明 |
|------|------|
| **同期構文** | 見たものが得られたものの逐次コード |
| **並行本質** | ランタイムが自動的に並列性を抽出 |
| **統一協同** | 結果は必要なときに自動収束し、論理の正しさを保証 |

**用語体系**：

| 公用語 | 対応構文 | 説明 |
|----------|----------|------|
| **並作関数** | `spawn (params) => body` | 並作実行に参加可能な計算ユニットの定義 |
| **並作ブロック** | `spawn { a(), b() }` | 明示的に宣言された並行領域、ブロック内タスクは並作実行 |
| **並作ループ** | `spawn for x in xs { ... }` | データ並列、ループ本体はすべての要素で並作実行 |
| **並作値** | `Async(T)` | 並作中の未来値、使用時に自動待機 |
| **並作グラフ** | 遅延計算グラフ(DAG) | 並作が発生する舞台、依存関係と並列関係を記述 |
| **並作スケジューラ** | ランタイムタスクスケジューラ | 万物を協調させ、正しいタイミングで並作させるスマート中枢 |

> **詳細**：[RFC-001 並作モデル](./rfc/001-concurrent-model-error-handling.md)

```yaoxiang
# === 並作関数 ===
# spawn マーカー付き関数
fetch_data: (url: String) -> JSON spawn = {
    return HTTP.get(url).json()
}

# === 並作ブロック ===
# spawn { } 内の式は強制的に並列実行される
compute_all: () -> (Int, Int, Int) spawn = {
    (a, b, c) = spawn {
        heavy_calc(1),    # タスク 1
        heavy_calc(2),    # タスク 2
        another_calc(3)   # タスク 3
    }
    return (a, b, c)
}

# === 自動待機 ===
main: () -> Void = {
    # 二つの独立したリクエストは自動的に並列実行される
    users = fetch_data("https://api.example.com/users")
    posts = fetch_data("https://api.example.com/posts")

    # 待機点は結果が必要なときに自動挿入される
    print(users.length + posts.length)  # users と posts を自動待機
}
```

**スレッド安全性**：

```yaoxiang
# ref キーワードがスレッド安全性を自動処理（コンパイラが自動的に Rc/Arc を選択）
main: () -> Void = {
    counter = ref SafeCounter(0)

    # タスク間共有：コンパイラが自動的に Arc を選択
    spawn {
        counter.increment()
    }
    spawn {
        counter.increment()
    }
}
```

**技術ドキュメント**：
- 詳細 [RFC-001 並作モデル](./rfc/accepted/001-concurrent-model-error-handling.md)

**イノベーションの価値**：非同期プログラミングの認知負荷がゼロに、コード可読性が同期コードと完全に同じ、高性能並列実行効率を獲得。

### 3.5 イノベーション五：値依存型（RFC-011）

> **ステータス**：設計中、一部実装済み

型は値に依存でき、真の型駆動開発を実現。

```yaoxiang
# 行列型：次元は編訳時に確定
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
}

# 編訳時計算：factorial(3) = 6
vec: Vec(factorial(3)) = Vec(6)()

# 編訳時次元検証
identity_3x3: Matrix(Float, 3, 3) = identity(Float, 3)(3)
# multiply(matrix_2x3, matrix_4x2)  # 編訳エラー：次元不一致
```

**イノベーションの価値**：編訳時により多くのエラーを捕捉し、より精密な型保証を実現。

### 3.6 イノベーション六：極簡キーワード設計

YaoXiang は 17 個のコアキーワードのみ定義し、主要言語をはるかに下回る数：

```
pub    use    spawn
ref    mut    if     elif
else   match  while  for    return
break  continue as     in     unsafe
```

| 比較言語 | キーワード数 |
|----------|-----------|
| YaoXiang | **17** |
| Rust | 51+ |
| Python | 35 |
| TypeScript | 64+ |
| Go | 25 |

**イノベーションの価値**：より低い記憶負担、より一貫した構文スタイル、より解析しやすい構文構造。

### 4.4 所有権モデル（RFC-009 v9）

```yaoxiang
Point: Type = { x: Float, y: Float }

# デフォルト Move（ゼロコピー）
p1 = Point(1.0, 2.0)
p2 = p1              # Move、p1 はもう読み取れない

# &T / &mut T トークン（編訳時ゼロオーバーヘッド）
p2.print()           # コンパイラが自動的に &Point トークンを作成
p2.shift(1.0, 1.0)  # コンパイラが自動的に &mut Point トークンを作成

# ref：共有保持（コンパイラが自動的に Rc/Arc を選択）
shared = ref p2      # スコープ間共有

# clone()：明示的ディープコピー
backup = p2.clone()

# unsafe + 生ポインタ：システムレベル
unsafe {
    ptr: *Point = &p2
    (*ptr).x = 0.0
}
```

**所有権グラデーション**：
```
&T / &mut T    Move       ref        clone()    unsafe
    |             |          |           |          |
借用トークン   デフォルト  共有保持   ディープコピー  生ポインタ
ゼロコスト     ゼロコピー  自動Rc/Arc  明示的      システムレベル
```

### 4.5 エラー処理

# Result 型
Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,

divide: (a: Float, b: Float) -> Result(Float, String) = {
    if b == 0.0 {

}

### 5.1 既に決定された設計決定

以下の決定は十分な議論と審査を経たものであり、**変更を受け付けない**：

| モジュール | 決定 | 説明 |
|------|------|------|
| **型システム** | すべては型なり | 値、関数、モジュール、generics はすべて型 |
| **型構文** | 統一 `name: type = value` | 一つの宣言形式ですべてをカバー、`Type` は唯一の元型キーワード |
| **キーワード** | 17 個のコアキーワード | `type`/`fn`/`struct`/`enum`/`trait`/`impl` を含まない |
| **関数構文** | シグネチャ + 式 | `name: (params) -> ReturnType = body` |
| **メソッドバインディング** | RFC-004 カリー化バインディング | `Type.method = function[position]` |
| **非同期モデル** | 並作モデル | `spawn` マーカー、遅延評価、自動並列 |
| **メモリ管理** | 所有権モデル（RFC-009 v9） | Move + &T/&mut T トークン + ref + clone + unsafe、GC なし |
| **ファイル即モジュール** | モジュールシステム | 各 `.yx` ファイルは一つのモジュール |
| **メイン関数** | `main: () -> Void` | プログラムエントリポイント |
| **スレッド安全性** | ref が自動的に Rc/Arc を選択 | コンパイラのエスケープ分析、ユーザーに感じさせない |

## 付録C：よくある質問

**Q: YaoXiang は Rust と比較してどのような優位性がありますか？**

A: YaoXiang は Rust のメモリ安全性とゼロコスト抽象を保持しつつ、よりシンプルな構文とより低い認知負荷を採用している。**並作モデル**は Rust の `async/await` より簡潔——一つの `spawn` マーカーのみ、Future と Pin の手動管理不要。「万物並作、吾以て復を観る」、並行プログラミングを自然法則を記述するように直感的にする。**所有権モデル**（RFC-009 v9）は Move + &T/&mut T トークンでライフタイム注釈を置き換え、型属性（Dup/Linear）で借用チェッカーを置き換える。統一型構文は `enum`/`struct`/`trait`/`impl` の概念断片を排除する。

**Q: YaoXiang はどのような種類の開発に適していますか？**

A: システムプログラミング、アプリケーション開発、Web サービス、スクリプティングツール、AI 支援プログラミング。汎用プログラミング言語になることが目標。

**Q: なぜ 4 スペースインデントを選択するのですか？**

A: 4 スペースは明確なコードブロック視覚的分離を提供し、ネストの深さによる混乱を減らす。深思熟慮された「AI フレンドリー」設計決定である。

**Q: いつバージョン 1.0 がリリースされますか？**

A: v1.0 目標：プロダクション使用可能。リリース時期は実装進度に依存する、詳細 [バージョン計画 RFC](./rfc/003-version-planning.md) を参照。

**Q: コアチームに連絡するにはどうすればよいですか？**

A: GitHub Discussions または Discord コミュニティチャンネル経由。コアチームメンバーが定期的に返信する。
