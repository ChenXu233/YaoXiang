# 2006年生まれの者の言語設計観

> Rustが胎動を始めた頃、私はこの世に生を受けた。Rustが成熟した頃、私は若者として歩み始めた。そして今後の十年、我々は自分たちの世代のための言語を創造できる。

## 序論：世代間使命の継承

2006年年はRustプログラミング言語が誕生し、私もこの世に来た年である。十九年後、私がYaoXiang（爻象）を設計・実装し始めた時、これが単なる偶然の一致ではなく、世代間の使命の継承であることを認識した。

Rustは2000年代の痛点を解決した：メモリ安全、並行安全。それは、C/C++の沼地で苦しんだ一代のエンジニアたちが示した答えである。しかし、すべての世代には自分たちの問題があり、すべての世代には自分たちのツールが必要である。

本文書は技術仕様書ではなく、**世代間宣言書**である。答えるべき問いは：**なぜ我々この世代の開発者は自分たちの言語を必要とするのか？YaoXiangはいかにして我々のニーズに応えするのか？**

## 一、世代間ギャップ——なぜ既存の言語は我々に「不服水土」を感じさせるのか

### 1.1 あの「反直感的」設計

私が初めてRustを学んだ時、所有権チェッカーにさんざんな目にあった。メモリ安全の重要性を理解していたが、なぜ単純な文字列結合にこんなに面倒なライフタイム注釈が必要なのか理解できなかった。後になって気づいたのは、**Rustの設計者たちは別の時代を生きていた**ということである。

彼らの思考パターンは：

- 「メモリ安全は意図的に解決すべき問題である」
- 「並行は小心翼翼と対応すべき怪物である」
- 「型システムはエラー捕捉のためのツールである」

私の思考パターンは：

- 「メモリ安全は言語がデフォルトで提供すべきものではないのか？」
- 「並行は呼吸のように自然なことではないのか？」
- 「型システムは問題の探求のための足場になれないのか？」

これはRustへの批判ではない。Rustは出た時代に革命的なものであった。しかし**我々の世代の「デフォルト」は上一世代の「贅沢品」なのである**。

### 1.2 「空気」と「障壁」

私の世代の開發者はマルチコアCPU、クラウドネイティブ、モ바일インターネットの世界で育った。私たちにとって：

- マルチコアプロセッサは「空気」——单核の制約など経験したことがない
- 非同期プログラミングは「空気」——同期阻塞がデフォルトモデルだった時代など知らない
- 分散システムは「空気」——ローカルファーストの設計思想など知らない

プログラミング言語の教科書をめくって、著者が「なぜ並行プログラミングを勉強する必要があるのか」を大量のページを費やして説明しているのを見ると、我々の内心のOSは：**「これは自明のことではないのか？なぜ勉強する必要がある？」**

これが世代間ギャップである。**上一世代が「勉強」すべきことは我々の世代にとっては「本能」なのである**。

### 1.3 AI時代の「文盲」困境

AIプログラミングアシスタントを使い始めた時、もっと深い問題を発見した：**既存の言語の設計はAIを考慮したことがなかった**。

- 構文歧義がAIに幻觉を生じさせる
- 暗黙のルールがAIの行動推断を妨げる
- 型システムの境界が曖昧でAIが误った型提案をする

AIがPythonのリスト内包表記とC++のlambda式を混同し、Rustの`impl Trait`とTypeScriptのジェネリクスを混乱させているのを直接見た。これはAIの問題ではなく、**言語設計がAI時代に対応していない問題である**。

**並行は我々にとって特性ではなく、デフォルト状態である。** この世代にとってマルチタスクOSが「空気」であるのと同じである。

だからYaoXiangが「並作モデル」を採用した時、これは革新ではなく、**我々の本能を言語にコード化する**ことである。

```yaoxiang
# YaoXiangの並作構文：並行はデフォルトであり、明示的ではない
fetch_user(Int) -> User spawn = (id) => { ... }

main() -> Void = () => {
    user = fetch_user(1)     # 自動並列

    print(posts.title)       # posts準備完了後に自動待機
```

## 三、YaoXiangの設計応答——新生代のために設計された言語

### 3.1 すべては型である：圏論の世界観

YaoXiangの中心設計哲学は**「すべては型である」**である。これは技術的選択ではなく、**世界観の選択**である。

YaoXiangの世界において：

- 値は型のインスタンスである
- 型自体も型のインスタンスである（メタ型）
- 関数は入力型から出力型への写像である
- モジュールは型の名前空間组合せである

```yaoxiang
# 型としての値
MyList = List(Int)    # MyList は 이제 类型値である

# 依存型：型は値に依存する
type Vector[T, n: Nat] = vector(T, n)

# パターン照合型
describe_type(type) -> String = (t) => {
    match t {
        Point(x, y) -> "Point with x=" + x + ", y=" + y
        ok(value) -> "Ok value"
        _ -> "Other type"
    }
}
```

この設計は何に応えているのか？我々の世代が**数学の美しさ**を追求することに 응답している。数学を学ぶ時に触れる集合論、圏論が教えてくれる：**型は最も高次の抽象化である**。なぜこれを徹底しないのか？

### 3.2 並作モデル：並行を空気にする

YaoXiangの並作モデル（Concurrent Model）は**従来の非同期プログラミングのパラダイム颠覆**である。

従来の非同期プログラミングは：

```rust
// Rust
async fn fetch_data(url: &str) -> Result<Data, Error> {
    let response = reqwest::get(url).await?;
    response.json().await
}
```

理解する必要がある：

- `async`/`await` 構文
- `Future` trait
- `Pin` と `Unpin`
- ランタイム（tokio/async-std）
- タスクスケジューラ

YaoXiangの並作モデルは：

```yaoxiang
# 並作関数：spawnマーク一つだけ
fetch_data(String) -> JSON spawn = (url) => {
    HTTP.get(url).json()
}

# 並作ブロック：明示的並列
compute_all(Int, Int) -> (Int, Int, Int) spawn = (a, b) => {
    (x, y, z) = spawn {
        heavy_calc(a),
        heavy_calc(b),
        another_calc(a, b)
    }
    (x, y, z)
}

# 並作ループ：データ並列
parallel_sum(Int) -> Int spawn = (n) => {
    total = spawn for i in 0..n {
        fibonacci(i)
    }
    total
}
```

これは簡略化ではなく、**問題の再定義**である。従来の非同期プログラミングが問うたのは「非阻塞コードを同期コードのように見せるにはどうするか？」である。YaoXiangが問うたのは「なぜ非同期と同期に区別が必要なのか？」である。

**並行が空気になれば、構文の差異は消える。**

### 3.3 AIに優しい構文設計

YaoXiangの設計はAIコード生成のニーズを考慮している。これは「AIが理解できる」ほど浅なく、「AIが設計に参加した」ほどの深さがある。

**設計原則：**

1. **厳密に構造化され、歧義のない構文** - AIは構文歧義で幻觉を生じない
2. **ASTが明確で定位が容易** - AIはコード位置を正確に特定できる
3. **意味が明確で隠し動作がない** - AIはコードの動作を正しく推断できる
4. **コードブロック境界が明確** - AIはスコープを誤解しない
5. **型情報が完全** - AIは正しい型提案ができる

```yaoxiang
# 明示的なコードブロック境界
function_name(Params) -> ReturnType = (params) => {
    # 関数本体
}

# 括弧の省略禁止（歧義なし）
foo(T) -> T = (x) => x

# 4スペースのインデント必須（構造清晰）
if condition {
    do_something()
} else {
    do_other()
}
```

これは単なるスタイルガイドではなく、**AI協調のために設計された言語インフラストラクチャ**である。

## 四、具体的な設計決定背后的世代間思考

### 4.1 なぜ「コンストラクタ即ち型」を選択したのか？

YaoXiangの型定義は一つだけの形式：`|`で区切られたコンストラクタである。

```yaoxiang
# ゼロ引数コンストラクタ（列挙型スタイル）
type Color = red | green | blue

# 複数引数コンストラクタ（構造体スタイル）
type Point = Point(x: Float, y: Float)

# ジェネリックコンストラクタ
type Result[T, E] = ok(T) | err(E)
```

これは何に応えているのか？**型システムは統一されるべきで分裂されるべきでない**に応えている。

Javaでは`class`、`enum`、`interface`がある。Rustでは`struct`、`enum`、`trait`がある。TypeScriptでは`interface`、`type`、`class`がある。

なぜ型にこんなに多くの形式があるのか？**型は型であり、区别は値の形式にあるべきで、型の形式にあるべきではない**。

### 4.2 なぜGCを捨てて、所有権モデルを採用したのか？

YaoXiangはGCではなく、Rustスタイルの所有権モデルを採用した。

```yaoxiang
# デフォルトで不変参照
process(ref Data) -> Void = (data) => {
    # dataは読み取り専用
}

# 可変参照
modify(mut Data) -> Void = (data) => {
    # dataを変更できる
}

# 所有権の移動
consume(Data) -> Void = (data) => {
    # dataの所有権が移る
}
```

これは単なるパフォーマンス選択ではなく、**哲学的選択**である。

我々の世代は環境、资源効率を気にかけている。**「無制限なメモリ」を当然視していない**。クラウドサービスの請求書を見てきたし、一つ一つのバイトにはコストがあることを知っている。

同時に、GCの「Stop the World」一時停止に困扰したくない。流れるような用户体验、リアルタイムシステムの反応性に慣れている。

所有権モデルは我々に与える：**ゼロコスト抽象化 + 决定的パフォーマンス + メモリ安全**。

### 4.3 なぜカリー化がコア構文なのか？

YaoXiangはカリー化を通じてオブジェクトメソッド呼び出しに似た糖衣構文を実現した。

```yaoxiang
# コア関数定義
distance(Point, Point) -> Float = (a, b) => {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# メソッド糖衣構文のバインディング
Point.distance(_) = distance(self, _)

# 呼び出し方式
p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)
d1 = distance(p1, p2)     # 直接呼び出し
d2 = p1.distance(p2)      # メソッド構文
```

これは何に応えているのか？**関数型プログラミングの純粋性を欲しいながら、オブジェクト指向の直感性もまた欲しい**に応えている。

我々の世代がプログラミングを学んだ時、Pythonから始め、その後JavaScriptに触れた。`obj.method()`の呼び出し方式に慣れているが、関数型プログラミングの優雅さも欣赏している。

カリー化は両者を**同じコインの裏表**にする。

## 五、技術を越えて——世代間視点の文化的意義

### 5.1 自分たちの声が必要だ

プログラミング言語設計は長年にわたり「先輩」たちの話題領域であった。Linus Torvaldsは21歳でLinuxを開始し、Graydon HoareはRustを設計する際 이미专业的なエンジニアであった。

しかし、すべての世代には独自の洞察がある。**若者が問題を見る角度は異なる。これは欠陥ではなく、価値である。**

YaoXiangを設計する際、私はC/C++の歴史的包袱を持っていなかった。既存システムに「適応」する必要はなく、新システムを「原生的に」設計できた。

### 5.2 オープンソース協調の新範式

我々の世代が理解するオープンソース協調とは：

- メーリングリストではなく、Discordコミュニティ
- 公式ドキュメントではなく、インタラクティブなチュートリアル
- カンファレンス演讲ではなく、生配信コーディング
- 特許保護ではなく、オープン協調

YaoXiangは最初期からオープンソースである。これは理想主義のためではなく、**我々の世代が做事する方法だからである**。

### 5.3 AI原生時代の設計

現在の言語は2000年代向けに設計されている（单核、ローカル、人間記述）。YaoXiangは2030年代向けに設計されている（マルチコア、分散、人間と機械の共創）。

これは夸张ではなく、**差し迫った現実**である。

AIはプログラミングのあらゆる环节を変えている。コード生成、コードレビュー、デバッグ支援、ドキュメント作成——AIは开发者のデフォルトパートナーになりつつある。

**AIを考慮しない言語は、プリンターを考慮しないフォントデザインのようなもの——時代遅れで不器用に感じるだろう。**
