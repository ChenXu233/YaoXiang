# RFC-027：コンパイル時述語と統一静的検証

> **参考**:
>
> - [RFC-009: 所有権モデル](../accepted/009-ownership-model.md)
> - [RFC-010: 統一型構文 - name: type = value モデル](../accepted/010-unified-type-syntax.md)
> - [RFC-011: ジェネリクスシステム設計](../accepted/011-generic-type-system.md)
> - [RFC-024：スポーンブロックに基づく並行モデル](../accepted/024-concurrency-model.md)
>
> **取代**：[RFC-022: ホア論理静的検証サポート（仕様コメントと仕様型）](../deprecated/022-hoare-logic-static-verification.md) — 既に廃止

## 概要

本文では、YaoXiang に**コンパイル時述語**を第一級市民として導入し、すべてのコンパイル時静的検証を1つの **Bool 評価パイプライン**に統一することを提案する。コンパイル時述語は外部の仕様コメントではない——それは関数である。Bool を返す関数は、型位置で使用でき、コンパイラがコンパイル時に呼び出して戻り値をチェックする。型とは命題であり、コンパイル時評価とは証明である。SMT ソルバーはコンパイル時評価器のバックエンドコンポーネントとして組み込まれ、プログラマは「コンパイラが条件を証明できるか否か」のみを気にするすればよく、別の仕様言語を学習する必要はない。

**核心的な论点**：型チェックがコンパイル時に行う唯一の仕事はブール評価である。型等式、トークン衝突、依存型簡約、コンパイル時述語評価、ホア論理含意——すべてがコンパイル時の Bool 評価であり、同一のパイプラインを共有する。

## 動機

### なぜ RFC-022 を廃止するのか？

RFC-022 では、仕様を `//!` コメント形式で設計していた：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← これは型から独立したコメント
    //! ensures: ExistsMax(result, arr[0..n])   ← これは型から独立したコメント
}
```

これは Curry-Howard 同型の根本的な錯誤を犯している：**仕様と型を二層に分離する**。コメントは型ではない。コメントは型チェックに関与しない。コメントは「外部ツール」のメンタルモデルである。

白皮书明確に述べている：

> "`//!` コメントはない。独立した仕様言語はない。すべては型システム内で完結する。"

### 現在の問題点

- RFC-022 の `//!` コメントは型システムから独立した外部構文である
- 仕様型と通常型は2つの体系であり、概念的冗長を生み出している
- Debug Build 検証 / Release Build 無視の分裂パターンが統一性を壊している
- SMT ソルバーが外部ツールとして而非コンパイラの標準コンポーネントとして位置づけられている
- 型チェック、借用語检查、コンパイル時述語チェック、マクロ展開がそれぞれ異なるパスを通る

### 正しいメンタルモデル

型チェックは次のような純粋関数として抽象化できる：

```
isWellTyped : Program → Bool
```

すべてのコンパイル時チェック——単純な型照合、借用語衝突検出、コンパイル時述語検証——はすべてこの関数のサブタスクである。它们共享同一条求值管道，区别仅在于表达式复杂度和求值策略。

## 提案

### 1. `{}` はソルバーの検証空間である：型とはアサーション、検証とは型チェック

YaoXiang の `{}` はソルバーの検証空間である。里面的一切都是アサーション、ソルバー保证每一项为 True。

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  ソルバーは x が Float、y が Float を保証

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  ソルバーは data が Array(T) を保証
```

**ジェネリクスはコンパイル時述語の特例である。**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          パラメータはシグネチャに    {} 内はアサーションのみ
#          ソルバーはコンパイル時呼び出し時に x > 0 を検証

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      パラメータはシグネチャに    ソルバーは type_of(T) == Type、type_of(data) == Array(T) を検証
```

同一のパターン：`name: (params) -> Type = { アサーション }`。ソルバーは「型アサーション」と「値アサーション」を区別しない。

**ループ不変式は別途記述する必要がない。変数上の型注釈が Floyd-Hoare 不変式である。**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0   # ソルバー検証：0 == sum(arr[0..0])  → True
    mut i: UpTo(arr.len) = 0     # ソルバー検証：0 <= 0 <= arr.len   → True
    while i < arr.len {
        s += arr[i]  # ソルバー検証：s_new == sum(arr[0..i+1])
        i += 1       # ソルバー検証：i+1 <= arr.len が維持される
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

コンパイラはループボディに対して検証条件を1回生成する——帰納仮説（型注釈）→ 代入操作 → 新たな値が型注釈を満たすか。SMT ソルバーが帰納ステップの証明成功后、すべての反復が自動的にカバーされる。`: decreases` も `: Invariant` も帰納証明も不要——コンパイラは帰納を各代入の局所 VC に分解する。

### 2. 前置条件/後置条件：パラメータ型と戻り値型上のコンパイル時述語

RFC-022 の `//! requires`/`//! ensures` を捨てる。コンパイル時述語はパラメータまたは戻り値の型注釈として使用する：

```yaoxiang
# 前置条件：パラメータ型にコンパイル時述語が含まれる
divide: (a: Int, b: Positive) -> Int = a / b
#                       ^^^^^^^^  ソルバーが呼び出し点で { b > 0 } を検証

# 後置条件：戻り値型がコンパイル時述語である
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

max: (T: Ord) -> ((arr: NonEmpty(Array(T))) -> IsMax(T, arr)) = {
    #                                           ^^^^^^^^^^^^^^
    #  IsMax(T, arr) は arr を部分適用し、(result: T) -> Type を返す
    #  ソルバーが戻り位置で { forall j: result >= arr[j] } を検証
    result = arr[0]
    for i in 1..arr.len {
        if arr[i] > result { result = arr[i] }
    }
    return result
}
```

### 3. コンパイル時 Bool 評価パイプライン

すべてのコンパイル時チェックが同一のパイプラインを共有する：

```
コンパイル時に Bool 式を評価する必要がある
        │
        ├── 型等式（T1 == T2）
        │   → コンパイラが直接判定
        │
        ├── トークン衝突条件（!conflicting(tokens)）
        │   → フロー感受性活性解析（Dup/Linear 属性追跡）
        │
        ├── 依存型簡約（n + m の簡約）
        │   → コンパイル時項書き換え系
        │
        ├── コンパイル時述語（x > 0, forall...）
        │   → コンパイラ自体 + SMT ソルバー
        │
        └── ホア論理含意式（P ⇒ Q）
            → SMT ソルバー
                    │
                    ▼
             ┌──────────┐
             │ True     │  → コンパイル成功
             │ False    │  → コンパイルエラー + 反例
             │ 予算超   │  → コンパイルエラー + 述語位置 + 消費量
             └──────────┘
```

コンパイル時 Bool 評価の結果は3種類のみ——它们は**同一の評価操作の3つの可能な結果**である：

```
eval_compile_time : BoolExpr → True | False | 予算枯渇
```

これは設計選択ではなく、停止問題の必然的結論である。コンパイラが任意の Bool 式を評価する——Rice の定理により決定不能である。コンパイル時評価器もプログラムであり、同じ停止問題に立ち向かう。 поэтому результат должен иметь три направления:

- `True` → 停止、答えは真
- `False` → 停止、答えは偽
- `予算枯渇` → 与えられた資源上限内で停止しなかった

予算の厳格な制限は停止問題に対する工的解決である。ノブを与えない——与えると「あなたのプログラムは停止すると思いますか」とユーザーに問いかけることになるが、ユーザーにもコンパイラにも分からない。

**重要な設計**：
- **3種類の結果は同一來源**：`True`/`False`/`予算枯渇` は同一評価パイプラインの3つの出口
- **予算の厳格な制限**：ステップ数、時間、量化子インスタンス化深さ——コンパイラ内部で固定され、ノブは与えない
- **自動戦略選択**：コンパイラは式の複雑さに応じて自動的に評価戦略を選択、自分で判断できれば自分で判断、判断できなければ SMT を呼び出す
- **工的現実**：ほとんどの述語（`x > 0`、`arr.len > 0`、`0 <= idx < arr.len`）は線形算術であり、SMT はミリ秒単位で True/False を返す。予算枯渇はユーザーが決定可能片段を超えた複雑な述語を書いた場合にのみ発生する

**パイプライン内の階層的依存**：以上の評価器は同一インターフェースを共有するが評価順序が存在する。型等式はすべての後続分析の前提条件である；所有権/トークン チェックは型情報に依存する；精製述語検証は前两层の結果に依存する。コンパイラは層ごとに評価し、底层失败的式は上层に進まない——型誤りのあるプログラムで SMT ソルバーの予算を浪費することを避ける。

```
評価順序（同一パイプライン、階層的スケジューリング）
├── 第 0 層：型等式（T1 == T2）
│   └── 構造的单一化 → 失敗なら後続は無意味、直接 False を返す
├── 第 1 層：所有権/トークン衝突
│   └── フロー感受性活性解析 → 失敗ならメモリ安全が成立しない、直接 False を返す
└── 第 2 層：精製述語/ホア含意
    └── コンパイラ + SMT → False（反例）或いは 予算枯渇
```

各層は依然として `True/False/予算枯渇` を返し、同一インターフェースと同一予算系を共有する。

### 4. 三層関数の統一

| 階層 | 実行タイミング | 入力 | 出力 | 例 |
|------|----------|------|------|------|
| 値レベル関数 | ランタイム | 値 | 値 | `add: (a: Int, b: Int) -> Int = a + b` |
| 型構築子 | コンパイル時 | 型/値 | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| コンパイル時述語 | コンパイル時 | 値 | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

すべて同じ `name: type = value` 構文を使用する。コンパイル時述語と型構築子は同一のコンパイル時評価パイプラインを通る——`{}` はソルバーの検証空間である。

### 5. ループ：Floyd-Hoare 検証条件生成

ループには別途 `: Invariant(...)` や `: decreases(...)` 注釈は不要である。変数上のコンパイル時述語型注釈が Floyd-Hoare スタイルのアサーションを定義する——コンパイラは型注釈から検証条件を生成し、SMT ソルバーは各代入が型を維持するかチェックする。

中核的なメカニズム：各代入操作は Hoare 三つ組 `{P} x := e {Q}` に対応し、検証条件は `P ⇒ Q[e/x]` である。コンパイラはループボディに対して検証条件を1回生成する——SMT ソルバーが帰納ステップの証明成功后、すべての反復が自動的にカバーされる。

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0   # 検証：0 == sum(arr[0..0])  → True
    mut i: UpTo(arr.len) = 0     # 検証：0 <= 0 <= arr.len   → True
    while i < arr.len {
        # コンパイラがループボディに対して VC を1回生成：
        #
        # s += arr[i]：
        #   前提（帰納仮説）：s == sum(arr[0..i])
        #   検証義務: s_new == sum(arr[0..i+1])
        #   s_new = s_old + arr[i] を代入：
        #     s_old + arr[i] == sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   SMT ソルバー：線形算術、ミリ秒級 → True
        #
        # i += 1：
        #   前提：i < arr.len（ループ条件）+ i >= 0（型境界）
        #   検証義務：i+1 >= 0 且つ i+1 <= arr.len
        #   SMT：前提から直接導出 → True
        s += arr[i]
        i += 1
    }
    return s  # この時 s: SumUpTo(arr, arr.len)、つまり s == sum(arr[0..arr.len])
}
```

ループ不変式は変数上の型注釈である——プログラマが型を書き、コンパイラが帰納ステップをチェックする。コンパイラは不変式を「発見」しも「自動的に帰納」もしない——帰納証明を各代入操作の局所検証条件に分解し、SMT ソルバーに分割統治させる。

### 6. 終了チェック

コンパイル時に全自动で完了する。コンパイラが証明できるループは通過し、証明できないものは直接コンパイルエラーを報告する——プログラマはコンパイラの自動分析に従う必要がある。半自動注釈の余地を与えない。

#### 6.1 設計原則

コンパイラは2箇所から自動的に終了証明所需的情報を抽出する：

1. **変数型注釈**：精製型内の境界制約（例：`UpTo(n)` は上限 `n` と下限 `0` を与える）
2. **ループボディ操作**：各反復で変数に施加される操作

コンパイラは優先順位に従って4つの測度合成戦略を試み、1つ見つかった時点で停止する。

#### 6.2 戦略 1：線形ランク関数の自動合成

変数が線形境界注釈を持つ場合、コンパイラは候補線形測度を列挙して SMT 検証する。

```
入力：
  変数 v₁: UpTo(u₁), v₂: UpTo(u₂), ...（上下境界のある変数）
  ループ条件 cond
  ループボディ内の代入集合

アルゴリズム：
  1. 型注釈から各変数の境界を抽出：[low_i, high_i]
  2. 候補測度を列挙：v_i, u_i - v_i, v_i - v_j、等の線形結合
  3. 各候補測度 m について：
     - SMT 検証 m ≥ 0（型境界から導出）
     - ループボディの各実行パスについて、SMT 検証 m' < m（厳密減少）
  4. 条件を満たす線形結合が見つかる → 終了証明完了
```

適用範囲：変数が線形式（`v = a·v + b`）で代入され、境界型注釈を持つ任意のループ。`i += const`、`i -= const`、および二分探索型の区间収縮を含む：

```yaoxiang
# 二分探索：low = mid + 1 或いは high = mid
# 測度 high - low は両方のパスで厳密減少する
binary_search: (arr: Sorted(Array(Int)), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 戦略 2：述語違反カウント——目標型から自動的に測度を抽出

ループの戻り値型が `forall` 精製述語であり、ループボディがデータに対する局所変更を含む場合、コンパイラは述語定義から自動的に測度を生成する。

核心的な洞察：**ユーザーが書いた仕様はコンパイラの推論の素材である。** コンパイラは「ソートとは何か」を組み込む必要はない——`Sorted` の定義を読み、定義から自動的に測度を抽出する。

```
入力：
  目標型：Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  ループボディ操作：隣接要素の交換

アルゴリズム：
  1. 述語定義を解析：forall i in range: cond(i, arr)
  2. 測度を自動生成：violation_count = |{ i | ¬cond(i, arr) }|
  3. 操作が測度に与える影響を分析：
     - 隣接交換 arr[j], arr[j+1] = arr[j+1], arr[j]
     - インデックス j-1, j, j+1 の3組のみに影響
     - もし arr[j] > arr[j+1]（述語違反）なら、交換後この組は述語を満たす
     - violation_count が少なくとも1減少する
  4. 上界：n·(n-1)/2（最大隣接逆順数）、下界：0
  → 終了証明完了
```

```yaoxiang
sort: (arr: Array(Int)) -> Sorted(arr) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 戦略 3：有界増減パターン

`v += const`（正数）、変数が上限型注釈を持つ → 測度 `upper_bound - v` が毎回 `const` だけ減少、下界は 0。これは戦略 1 の退化ケースであり、コンパイラは最初で快速に処理する。

#### 6.5 戦略 4：乗法スケーリング測度テンプレート

`v *= const`（const > 1）、変数が上下界型注釈を持つ。コンパイラは対数測度テンプレート `ceil(log_const(upper/v))` を組み込み、毎回 const を掛けると測度が1減少する。

```yaoxiang
mut i: Positive = 1
while i < n {
    # コンパイラが自動導出：測度 ceil(log₂(n/i))、毎回2を掛けると測度が1減少
    i *= 2
}
```

#### 6.6 終了性と正しさの分離

終了性証明と正しさ証明は独立している：
- **終了性**：以上の4つの戦略でループが有限ステップで終了することを自動証明
- **正しさ**：ループボディが目標型に向かって進行するか否かは、SMT ソルバーが検証条件を通じてチェック

两者とも通過 → コンパイル成功。終了性証明済みだが正しさ失敗 → コンパイルエラー + 反例。終了性が証明できない → コンパイルエラー、分析できない変数または操作を示す。

#### 6.7 再帰関数の終了チェック

コンパイル時に評価する必要がある再帰関数について、コンパイラはパラメータが減少し続けるかチェックする：

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # コンパイラ分析：n-1 < n → 減少 → 終了
}

# コンパイル時使用——コンパイラは factorial がコンパイル時に終了することを保証
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120、コンパイル時に完了
```

| シナリオ | 動作 |
|------|------|
| コンパイラが再帰減少を分析できる（例：`n-1`） | コンパイル時評価 |
| 減少しない/減少の判定不可 | コンパイルエラー |
| ランタイム呼び出し（型位置ではない） | 終了チェック不要 |

#### 6.8 ハード境界

`i = f(i)` かつ `f` が不可逆、閉鎖しない、単調性を保持しない——数学的に自動終了証明は不可能である。コンパイルエラー：

> このループは自動終了証明できません。ループ変数が分析不能な関数 `f` に依存しています。コンパイラが分析できる反復パターンを使用してください。

これはコンパイラの失敗ではない。静的証明ができないコードはすべて、コンパイルを通過してはならない。

### 7. SMT ソルバー：コンパイル時評価器のバックエンドコンポーネント

SMT ソルバーは 전통的な言語에서는 外部 도구이다（例：F\* が Z3 を呼び出す、Dafny が Z3 を呼び出す）。YaoXiang では、それはコンパイル時評価パイプラインのバックエンドコンポーネントである——コンパイラ自身が直接判定できない場合にのみ呼び出される。

```
コンパイル時 Bool 式
        │
        ├── コンパイラが直接判定可能（単純比較、単純算術、
        │   定数畳み込み後の自明な公式）
        │   → 直接 True/False を返す
        │
        └── コンパイラが直接判定不能（量化子、記号変数）
            → 依存型事前簡約（factorial(5) → 120）
            → SMT-LIB 形式に翻訳
            → Z3/CVC5 に送信（予算制限付き）
            → 戻り値：unsat（成立）│ sat（反例）│ budget_exceeded
```

**求解予算——厳格な制限、スタック深さのように**：

| 予算ディメンション | 説明 | 厳格上限 |
|----------|------|--------|
| 求解ステップ数 | 項書き換え/解決ステップ数 | コンパイラ内部固定値 |
| 時間 | 単一述語の求解時間 | コンパイラ内部固定値 |
| 量化子インスタンス化深さ | forall/exists 展開層数 | コンパイラ内部固定値 |

予算超過はコンパイルエラーである。デグラデーションなし、ランタイムチェックなし、サイレントパスなし。エラーメッセージはソースコードの述語位置に正確に対応し、予算消費量を報告する。

**なぜこれが実際可能か**：工的現実として95%の実際の述語は線形算術である——`x > 0`、`arr.len > 0`、`0 <= idx < arr.len`——これらはすべて決定可能片段 내에あり、SMT ソルバー对这些問題はミリ秒単位で True/False を返す。予算超過になる複雑な述語に遭遇することは極めてまれであり、父simply述語を分割すればよい。

依存型は SMT 呼び出し前に事前簡約の一層を経る：`factorial(5)` は直接コンパイル時評価で `120` を得、`append([1,2], [3])` は直接評価で `[1,2,3]` を得る。これらの確定的な値計算は SMT 予算を消費しない。

プログラマは SMT の存在を知る必要はない。メンタルモデルは：**コンパイラが証明できれば通過、できなければエラー**——型チェックと完全に一致する。

### 8. コンパイル時述語の合成

コンパイル時述語は Type を返す関数であり、合成は関数合成で自然に実現される：

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(T, arr)
}
```

### 9. コード例

#### 9.1 安全な除算

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive) -> Int = a / b

result = divide(10, 2)   # ✅ ソルバー検証 { 2 > 0 } → True
# result = divide(10, 0)  # ❌ ソルバー検証 { 0 > 0 } → False
```

#### 9.2 配列アクセス安全

```yaoxiang
InBounds: (T: Type, idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (T: Type) -> ((arr: Array(T), idx: InBounds(T, arr)) -> T) = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ ソルバー検証 { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ ソルバー検証 { 0 <= 5 && 5 < 3 } → False
```

#### 9.3 ソート正しさ

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T)) -> Sorted(T)) = {
    result = arr.clone()
    # ... ソートアルゴリズム実装 ...
    return result
}
```

#### 9.4 ループ：コンパイラ VC 生成

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```

## 詳細な設計

### 構文の変更

| 以前 (RFC-022) | 以降 (本 RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | コンパイル時述語をパラメータ型として `(b: Positive)` |
| `//! ensures: ExistsMax(result, arr)` | 戻り値型がコンパイル時述語 `-> IsMax(T, arr)` |
| `/*! invariant: ... !*/` | 変数上のコンパイル時述語型注釈——Floyd-Hoare 不変式 |
| `//! decreases: n` | コンパイラが全自动で測度関数を導出 |
| 仕様はコメント | 仕様は型システム |

### 構文

**コンパイル時述語に新しい構文はない。** `{}` はソルバーの検証空間であり、既存の型定義構文と完全に一致する。コンパイル時述語は Type を返す関数である——`name: (params) -> Type = { アサーション }`。

```bnf
# コンパイル時述語 = Type を返す関数、{} 前はソルバーアサーション
# 既存の関数/型構文を使用、新BNF規則不要
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

### 型システムへの影響

- **型宇宙**：コンパイル時述語は Type₂ 層に位置する——値を受け取り Type を返す関数で、型構築子と同レベル
- **ジェネリクスとの相互作用**：コンパイル時述語はジェネリック パラメータを持てる（例：`NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`）
- **所有権との相互作用**：コンパイル時述語内の式は所有権規則に従い、読み取りのみ可能で書き込み不可
- **型推論**：コンパイル時述語のパラメータは HM 型推論に参加する

### ランタイム表現

コンパイル時述語はランタイムで**完全に消去**される。`Positive: (x: Int) -> Type = { x > 0 }`——パラメータ `b: Positive` のランタイム表現は `Int` である。精製条件 `{ x > 0 }` はコンパイル時に SMT ソルバーで検証され、検証完了後 런타임에 남지 않는다.

- `Positive(5)` → ランタイム表現は `Int(5)`、精製条件 `{ 5 > 0 }` は通過済み、消去
- `SumUpTo(arr, 0)` → ランタイム表現は `Int(0)`、等式 `0 == sum(arr[0..0])` は通過済み、消去
- コンパイル時述語を型位置に使用（例：`f(x: Positive)`）しても包装型は生成されず、余分なメモリ割当もランタイムチェックも挿入されない

ジェネリクス消去と精製型消去は同じ原則である：两者ともコンパイル時関数であり、コンパイル時評価後に完全に消える。Release Build におけるコンパイル時述語のゼロランタイムオーバーヘッド——これは「述語は関数である」の直接的な帰結。

**`ref` との相互作用制約**：コンパイル時述語は不変借用または所有権が移動済みの値のみを参照できる。可変借用を参照するコンパイル時述語は、コンパイラがコンパイル時に検証結果がランタイムでも有効であることを保証できない——このような使用方法は直接コンパイルエラーを報告する。

### コンパイラの改変

1. **パーサー**：コンパイル時述語は標準関数構文を使用し、追加解析規則不要
2. **コンパイル時評価パイプライン**：統一 Bool 戻りインターフェース、自動戦略選択
3. **SMT バックエンド**：Z3/CVC5 統合、SMT-LIB 形式翻訳
4. **検証条件生成**：WP/SP 計算 + ループ不変式証明義務
5. **エラー報告**：反例フォーマット化、ソース位置関連付け

### 後方互換性

- ✅ コンパイル時述語を使用していないコードは完全に不変
- ✅ コンパイル時述語は Release Build でゼロランタイムオーバーヘッド
- ⚠️ RFC-022 の `//!` 構文はサポート終了——だが 022 は実装されたことがなく、移行負担はない

## トレードオフ

### 利点

- **Curry-Howard 同型の完全な実現**：型とは命題、プログラムとは証明、`name: Proposition = Proof`
- **統一性**：コンパイル時述語と通常関数は完全に同じ構文を使用し、概念的分裂がない
- **SMT の透過性**：プログラマは SMT の存在を知る必要がなく、メンタルモデルは型チェックと一致する
- **漸進的な採用**：1つのコンパイル時述語から開始し、逐步的にカバレッジを増加できる
- **ゼロランタイムオーバーヘッド**：すべての評価がコンパイル時に完了し、Release Build にはランタイムアサーションが挿入されない

### 欠点

- **コンパイル時間**：SMT 求解が増加するコンパイル時間。ただし、予算の厳格な制限が上限を保証する
- **SMT 能力の境界**：線形算術を超える複雑な述語は予算超過の可能性がある；ユーザーは述語を分割する必要がある
- **学習曲線**：効果的なコンパイル時述語を書くには一定の経験が必要
- **実装複雑度**：コンパイル時評価パイプラインの統一には慎重な設計が必要

### リスク緩和

- SMT 求解予算の厳格な制限（ステップ数/時間/インスタンス化深さ）、予算超過はコンパイルエラー
- 依存型事前簡約：確定的な値計算 먼저 처리、SMT は非決定的部分のみ処理
- 増分検証：変更されたモジュールのみ検証
- 明確なエラーメッセージ + 反例表示 + 予算消費報告

## 代替案

| 方案 | なぜ選択しないか |
| -------------------------------- | ---------------------------------------------- |
| RFC-022：`//!` コメント式仕様 | 仕様と型の分裂、Curry-Howard 同型に違反 |
| 独立仕様ファイル（例：CVL） | 仕様とコードの分離、保守コストの増加 |
| ランタイムアサーションのみ | 静的正しさを保証できない |
| 外部証明支援ツール（例：Coq） | 学習曲線が急峻、コンパイラと分離 |

## 実装戦略

### 段階的区分

| 段階 | 内容 |
| ---------- | -------------------------------------------------------------------- |
| **段階 1** | コンパイラが Bool を返す関数を型位置に使用できることを認識。単純算術述語をサポート（`x > 0`、`arr.len > 0`） |
| **段階 2** | コンパイル時 Bool 評価パイプライン + Z3 統合。`forall`/`exists` 量化子をサポート |
| **段階 3** | ループ不変式 VC 生成 + 終了チェック（線形ランク関数 + 述語違反カウント + 有界パターン） |
| **段階 4** | 増分検証 + キャッシュ + 反例フォーマット化 + IDE サポート |

### 依存関係

- RFC-010: 統一型構文 — コンパイル時述語は `name: type = value` に 기반
- RFC-011: ジェネリクスシステム — コンパイル時述語はジェネリック パラメータを持てる
- RFC-009: 所有権モデル — コンパイル時述語の式は所有権規則に従う
