# RFC-027：コンパイル時述語と統一静的検証

> **参考**:
>
> - [RFC-009: 所有権モデル](../accepted/009-ownership-model.md)
> - [RFC-010: 統一型構文 - name: type = value モデル](../accepted/010-unified-type-syntax.md)
> - [RFC-011: ジェネリクスシステム設計](../accepted/011-generic-type-system.md)
> - [RFC-024：spawn ブロックベースの並行性モデル](../accepted/024-concurrency-model.md)
>
> **置き換え**：[RFC-022: ホーア論理による静的検証サポート（仕様注釈と仕様型）](../deprecated/022-hoare-logic-static-verification.md) — 廃止済み

## 概要

本文書では、YaoXiang に**コンパイル時述語**を一級市民として導入し、すべてのコンパイル時静的検証を単一の **Bool 評価パイプライン**に統一することを提案する。コンパイル時述語は外付けの仕様注釈ではない——それは関数そのものである。Bool を返す関数は型位置で使用でき、コンパイラはコンパイル時にそれを呼び出し、返り値をチェックする。型は命題、コンパイル時評価は証明。SMT ソルバはコンパイル時評価器の後端コンポーネントとして組み込まれ、プログラマが意識するのは「コンパイラが条件成立を証明できるかどうか」のみで、独立した仕様言語を学ぶ必要はない。

**中核的主張**：型検査がコンパイル時に行う唯一の仕事はブール評価である。型等式、トークン衝突、依存型簡約、コンパイル時述語評価、ホーア論理含意——すべてはコンパイル時 Bool 評価であり、同じパイプラインを共有する。

## 動機

### なぜ RFC-022 を廃止するのか？

RFC-022 では仕様を `//!` 注釈形式として設計していた：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← これは型と独立した注釈
    //! ensures: ExistsMax(result, arr[0..n])   ← これは型と独立した注釈
}
```

これは Curry-Howard 同型の根本的な誤りを犯している：**仕様と型を二層に分裂させている**。注釈は型ではない。注釈は型検査に参加しない。注釈は「外部ツール」のメンタルモデルである。

ホワイトペーパーは明確に述べている：

> "`//!` 注釈はない。独立した仕様言語もない。すべては型システムの中にある。"

### 現状の問題

- RFC-022 の `//!` 注釈は型システムと独立した外付け構文である
- 仕様型と通常の型は二体系であり、概念の冗長性を生む
- Debug Build 検証 / Release Build 無視の分裂パターンが統一性を破壊する
- SMT ソルバが外部ツールとして位置づけられ、コンパイラの標準コンポーネントではない
- 型検査、借用検証、コンパイル時述語検査、マクロ展開がそれぞれ異なる経路を辿る

### 正しいメンタルモデル

型検査は純粋関数として抽象化できる：

```
isWellTyped : Program → Bool
```

すべてのコンパイル時検査——単純な型マッチ、借用衝突検出、コンパイル時述語検証——はこの関数のサブタスクである。これらは同じ評価パイプラインを共有し、違いがあるのは式の複雑さと評価戦略のみである。

## 提案

### 1. `{}` はソルバの検証空間：型はassertion、検証は型検査

YaoXiang の `{}` はソルバの検証空間である。その内部のすべてがassertionであり、ソルバは各項目が True であることを保証する。

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  ソルバが x は Float、y は Float であることを保証

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  ソルバが data は Array(T) であることを保証
```

**ジェネリクスはコンパイル時述語の特殊ケースである。**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          パラメータはシグネチャ位置     {} 内にはassertion のみ
#          ソルバがコンパイル時呼び出し時に x > 0 を検証

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      パラメータはシグネチャ位置          ソルバが type_of(T) == Type、type_of(data) == Array(T) を検証
```

同じパターン：`name: (params) -> Type = { assertion }`。ソルバは「型assertion」と「値assertion」を区別しない。

**ループ不変条件は別途書く必要がない。変数上の型注釈がそのまま Floyd-Hoare 不変条件になる。**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0   # ソルバ検証：0 == sum(arr[0..0])  → True
    mut i: UpTo(arr.len) = 0     # ソルバ検証：0 <= 0 <= arr.len   → True
    while i < arr.len {
        s += arr[i]  # ソルバ検証：s_new == sum(arr[0..i+1])
        i += 1       # ソルバ検証：i+1 <= arr.len の保持
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

コンパイラはループ本体に対して一度だけ検証条件を生成する——帰納仮説（型注釈）→ 代入操作 → 新しい値が型注釈を満たすか。SMT ソルバが帰納ステップの成立を証明すれば、すべての反復が自動的にカバーされる。`: decreases` も、`: Invariant` も、帰納証明も不要——コンパイラは帰納を各代入の局所 VC に分解する。

### 2. 事前/事後条件：引数型と戻り値型上のコンパイル時述語

RFC-022 の `//! requires`/`//! ensures` を廃止する。コンパイル時述語は引数または戻りの型注釈として機能する：

```yaoxiang
# 事前条件：引数型がコンパイル時述語を含む
divide: (a: Int, b: Positive) -> Int = a / b
#                       ^^^^^^^^  ソルバが呼び出し点で { b > 0 } を検証

# 事後条件：戻り型がコンパイル時述語
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

max: (T: Ord) -> ((arr: NonEmpty(Array(T))) -> IsMax(T, arr)) = {
    #                                           ^^^^^^^^^^^^^^
    #  IsMax(T, arr) が arr を部分適用し、(result: T) -> Type を返す
    #  ソルバが戻り点で { forall j: result >= arr[j] } を検証
    result = arr[0]
    for i in 1..arr.len {
        if arr[i] > result { result = arr[i] }
    }
    return result
}
```

### 3. 経路条件の伝播：実行時値のコンパイル時検証

コンパイル時述語が型注釈位置で使用されるとき、パラメータは注釈対象の変数自身で自動補填される。実行時値が精化型パラメータに入るとき、コンパイラは経路条件の収集と SMT 含意判定によって検証を完了する——プログラマが明示的に証明を渡す必要はない。

#### 3.1 暗黙の自己参照

`Positive: (x: Int) -> Type = { x > 0 }` はコンパイル時述語コンストラクタである。それが束縛位置（引数宣言、変数宣言、戻り型）に現れるとき、コンパイラは束縛される変数を自動的にパラメータに埋める：

```yaoxiang
b: Positive
// コンパイラの展開：b: Positive(b)
// 正格化後：b: { b > 0 }
```

ユーザに `b: Positive(b)` と書かせる必要はない——それは循環構文になる。暗黙の自己参照は精化型が束縛位置に現れるときの標準的な展開ルールであり、コンパイル時述語コンストラクタ自体は依然として `Int` を受け `Type` を返す純粋関数である。

戻り型にも暗黙の自己参照は適用される：

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (T: Ord) -> ((arr: Array(T)) -> Sorted(T)) = { ... }
//                                            ^^^^^^^^^^
//  Sorted(T) が T を部分適用し、型は (arr: Array(T)) -> Type
//  コンパイラは戻り値を残りパラメータ arr に埋める → Sorted(T, return_value) を検証
```

#### 3.2 経路条件の収集

実行時値が条件分岐に現れるとき、コンパイラは自動的に経路条件を収集し、現在のスコープの**仮定集合**を形成する。これらの仮定はコンパイル時 Bool 評価の背景知識として検証に参加する。

```yaoxiang
if y > 0 {
    // コンパイラはこの分岐内で自動的に仮定を取得：{ y > 0 }
    let result = divide(x, y)
    // 検証条件：(y > 0) ⇒ (y > 0)
    // SMT ソルバが unsat と判定（含意成立）→ 通過
} else {
    // この分岐の仮定：{ !(y > 0) }
    // divide(x, y) を呼び出す場合、検証条件は !(y > 0) ⇒ y > 0
    // SMT が sat と判定（含意不成立）→ コンパイルエラー
}
```

これはコンパイラが特定のモードをハードコードしているわけではない——コンパイル時 Bool 評価パイプラインの自然な振る舞いである。型検査の各呼び出し点で SMT に送信される：

```
{背景仮定} ⇒ {検証目標}
```

SMT ソルバが含意性を判定する。含意成立 → 通過、不成立 → コンパイルエラー。背景仮定は現在のプログラム点の経路条件から来る。

#### 3.3 仮定スタック

コンパイラは制御流を解析する際、各基本ブロックに対して仮定集合を保守する：

- **if-ガード**：`if y > 0` → true 分岐に `y > 0` をプッシュ、false 分岐に `!(y > 0)` をプッシュ（else を使用する場合）
- **match パターン**：`if let Some(v) = opt` → 分岐内に `opt == Some(v)` をプッシュ
- **論理結合**：`if x > 0 && y < 10` → 分岐内に `x > 0` と `y < 10` をプッシュ
- **関数事前条件**：`divide(a, b)` を呼び出すとき、`b` が `Positive` の証拠を持つには、現在の仮定から来るか、実引数自身の精化型注釈から来なければならない（`b` がすでに `Positive` として注釈されていれば、その型が `b > 0` を運ぶ）
- **代入**：`let z = y` のとき、`y` 上の既存の精化条件が `z` に伝播する

すべての仮定がコンパイル時評価パイプラインに入り、SMT ソルバに送信される際に SMT-LIB 背景assertionとして翻訳される。

#### 3.4 静的証拠なしはコンパイルエラー

プログラマが直接以下のように書いた場合：

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

現在のプログラム点に `y > 0` の仮定がなく、実引数 `y` 自身にも `Positive` 型注釈がない。検証条件は：

```
{} ⇒ { y > 0 }
```

SMT が `sat` を返す（含意不成立）→ コンパイルエラー：

> 呼び出し点 `divide` で引数 `b` が `Positive` を満たすことを証明できない。
> `y` は関数入力に由来し、証明済みの境界がない。
> if 分岐で呼び出しをガードすることを検討：`if y > 0 { divide(x, y) }`。

YaoXiang は静的証拠を提供せずに実行時値が精化型パラメータに直接入ることを許さない。これは制約ではなく、硬直した安全性哲学の核心である。コンパイラが静的に証明できないコードは、コンパイルを通過してはならない。

#### 3.5 統一パイプラインとの関係

経路条件の伝播は追加メカニズムではない。これはコンパイル時 Bool 評価パイプラインの制御流解析への自然な延長である：

| 段階 | 責務 |
|------|------|
| 経路条件の収集 | コンパイラの制御流解析段階が各基本ブロックに仮定集合を注釈 |
| 検証条件生成 | 検証が必要な型制約に遭遇したとき、経路条件と実引数型情報を統合 |
| Bool 評価 | コンパイラ自身または SMT ソルバが含意性を判定 |
| 結果 | `True` → 通過；`False` → コンパイルエラー + 反例；`予算超過` → コンパイルエラー + 位置 |

新しい部品はない。特殊ルールもない。経路条件は Bool 評価の背景知識に過ぎない——型等式、借用制約と同じパイプライン、同じ予算システムを共有する。

### 4. コンパイル時 Bool 評価パイプライン

すべてのコンパイル時検査は同じパイプラインを共有する：

```
コンパイル時に Bool 式の評価が必要
        │
        ├── 型等式（T1 == T2）
        │   → コンパイラが直接判定
        │
        ├── トークン衝突条件（!conflicting(tokens)）
        │   → フロー依存活性解析（複製/線形属性追跡）
        │
        ├── 依存型簡約（n + m 簡約）
        │   → コンパイル時項書き換えシステム
        │
        ├── コンパイル時述語（x > 0, forall...）
        │   → コンパイラ自身 + SMT ソルバ
        │
        └── ホーア論理含意式（P ⇒ Q）
            → SMT ソルバ
                    │
                    ▼
             ┌──────────┐
             │ True     │  → コンパイル通過
             │ False    │  → コンパイルエラー + 反例
             │ 予算超過 │  → コンパイルエラー + 述語位置 + 消費量
             └──────────┘
```

コンパイル時 Bool 評価の結果は三つだけ——それらは**同じ評価操作の三つの可能な結果**である：

```
eval_compile_time : BoolExpr → True | False | 予算超過
```

これは設計選択ではなく、停止問題の必然的な帰結である。コンパイラが任意の Bool 式を評価する——Rice の定理は決定不能と言う。コンパイル時評価器もプログラムであり、同じ停止問題に直面する。したがって結果は三つの経路を持たなければならない：

- `True` → 停止、答は真
- `False` → 停止、答は偽
- `予算超過` → 与えられた資源上限内で停止しなかった

予算のハード制限は停止問題の工学的解である。ノブは提供しない——提供すれば「あなたのプログラムが停止すると思いますか」とユーザに尋ねることになり、ユーザにもコンパイラにもそれはわからない。

**主要な設計**：
- **三つの結果は同源**：`True`/`False`/`予算超過` は単一評価パイプラインの三つの出口である
- **予算のハード制限**：ステップ数、時間、量化子インスタンス化の深さ——コンパイラ内部で固定、ノブなし
- **自動戦略選択**：コンパイラは式の複雑さに応じて評価戦略を自動選択し、自分で判定できるものは自分で判定し、できない場合のみ SMT を呼ぶ
- **工学的実態**：大多数の述語（`x > 0`、`arr.len > 0`、`0 <= idx < arr.len`）は線形算術であり、SMT はミリ秒オーダーで True/False を返す。予算超過は、ユーザが決定可能フラグメントを超える複雑な述語を書いたときにのみ発生する

**パイプライン内の階層的依存**：上記の評価器は同じインターフェースを共有するが、評価順序が存在する。型等式はすべての後続解析の前提条件である；所有権/トークン検査は型情報に依存する；精化述語検証は前二層の結果に依存する。コンパイラは層ごとに評価し、下層で失敗した式は上層に渡らない——型エラーのプログラムで SMT ソルバの予算を浪費することを避ける。

```
評価順序（同一パイプライン、階層的スケジューリング）
├── 第 0 層：型等式（T1 == T2）
│   └── 構造的単一化 → 失敗なら後続は無意味、直ちに False を返す
├── 第 1 層：所有権/トークン衝突
│   └── フロー依存活性解析 → 失敗ならメモリ安全性が成立しない、直ちに False を返す
└── 第 2 層：精化述語/ホーア含意
    └── コンパイラ + SMT → False（反例）または 予算超過
```

各層は依然として `True/False/予算超過` を返し、同じインターフェースと同じ予算システムを共有する。

### 5. 三層関数の統一

| 階層 | 実行時期 | 入力 | 出力 | 例 |
|------|----------|------|------|------|
| 値レベル関数 | 実行時 | 値 | 値 | `add: (a: Int, b: Int) -> Int = a + b` |
| 型コンストラクタ | コンパイル時 | 型/値 | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| コンパイル時述語 | コンパイル時 | 値 | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

すべて同じ `name: type = value` 構文を使用する。コンパイル時述語と型コンストラクタは同じコンパイル時評価パイプラインを辿る——`{}` はソルバの検証空間である。

### 6. ループ：Floyd-Hoare 検証条件生成

ループは個別の `: Invariant(...)` または `: decreases(...)` 注釈を必要としない。変数上のコンパイル時述語型注釈が Floyd-Hoare 風のアサーションを定義する——コンパイラは型注釈から検証条件を生成し、SMT ソルバが各代入が型を保持するかをチェックする。

中核メカニズム：各代入操作は Hoare 三つ組 `{P} x := e {Q}` に対応し、検証条件は `P ⇒ Q[e/x]` である。コンパイラはループ本体に対して一度だけ検証条件を生成する——SMT ソルバが帰納ステップの成立を証明すれば、すべての反復が自動的にカバーされる。

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0   # 検証：0 == sum(arr[0..0])  → True
    mut i: UpTo(arr.len) = 0     # 検証：0 <= 0 <= arr.len   → True
    while i < arr.len {
        # コンパイラがループ本体に対して一度 VC を生成：
        #
        # s += arr[i]：
        #   前提（帰納仮説）：s == sum(arr[0..i])
        #   検証義務：s_new == sum(arr[0..i+1])
        #   s_new = s_old + arr[i] を代入：
        #     s_old + arr[i] == sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   SMT ソルバ：線形算術、ミリ秒オーダー → True
        #
        # i += 1：
        #   前提：i < arr.len（ループ条件）+ i >= 0（型の境界）
        #   検証義務：i+1 >= 0 かつ i+1 <= arr.len
        #   SMT：前提から直接導出 → True
        s += arr[i]
        i += 1
    }
    return s  # このとき s: SumUpTo(arr, arr.len)、つまり s == sum(arr[0..arr.len])
}
```

ループ不変条件は変数上の型注釈そのものである——プログラマは型を書き、コンパイラは帰納ステップを検査する。コンパイラは不変条件を「発見」する必要も、帰納を「自動実行」する必要もない——帰納証明を各代入操作の局所検証条件に分解し、SMT ソルバに分割して処理させる。

### 7. 停止性検査

完全自動のコンパイル時処理。コンパイラが証明できるループは通過し、証明できないものは直接コンパイルエラーを報告する——プログラマはループの停止性をコンパイラが自動解析できるようにしなければならない。半自動注釈用の余地は設けない。

#### 6.1 設計原則

コンパイラは二箇所から停止性証明に必要な情報を自動抽出する：

1. **変数の型注釈**：精化型中の境界制約（例：`UpTo(n)` は上界 `n` と下界 `0` を与える）
2. **ループ本体の操作**：各反復で変数に加えられる操作

コンパイラは優先度に従って四つの度量合成戦略を試し、見つかれば停止する。

#### 6.2 戦略 1：線形ランク関数の自動合成

変数に線形境界注釈があるとき、コンパイラは線形度量の候補を列挙し SMT で検証する。

```
入力：
  変数 v₁: UpTo(u₁), v₂: UpTo(u₂), ...（上下界を持つ変数）
  ループ条件 cond
  ループ本体内の代入集合

アルゴリズム：
  1. 型注釈から各変数の境界を抽出：[low_i, high_i]
  2. 度量候補を列挙：v_i, u_i - v_i, v_i - v_j などの線形結合
  3. 各候補度量 m について：
     - SMT が m ≥ 0 を検証（型境界から導出）
     - ループ本体の各実行経路について、SMT が m' < m を検証（厳密減少）
  4. 条件を満たす線形結合を発見 → 停止が証明される
```

カバー範囲：変数が線形式で代入され（`v = a·v + b`）、有界型注釈を持つ任意のループ。`i += const`、`i -= const`、および二分探索風の区間収縮を含む：

```yaoxiang
# 二分探索：low = mid + 1 または high = mid
# 度量 high - low は両経路で厳密に減少する
binary_search: (arr: Sorted(Array(Int)), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 戦略 2：述語違反カウント——目標型から自動抽出

ループの戻り型が `forall` 精化述語であり、ループ本体がデータへの局所的な変更を含む場合、コンパイラは述語定義から自動的に度量を生成する。

中核的な洞察：**ユーザが書いた仕様がコンパイラの推論の素材である**。コンパイラは「ソートとは何か」を内蔵する必要はない——`Sorted` の定義を読み、定義から自動的に度量を抽出する。

```
入力：
  目標型：Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  ループ本体の操作：隣接要素の交換

アルゴリズム：
  1. 述語定義を解析：forall i in range: cond(i, arr)
  2. 度量を自動生成：violation_count = |{ i | ¬cond(i, arr) }|
  3. 操作が度量に与える影響を解析：
     - 隣接交換 arr[j], arr[j+1] = arr[j+1], arr[j]
     - インデックス j-1, j, j+1 の三対にのみ影響
     - arr[j] > arr[j+1]（述語違反）の場合、交換後この対は述語を満たす
     - violation_count は少なくとも 1 減少
  4. 上界：n·(n-1)/2（最大隣接転倒数）、下界：0
  → 停止が証明される
```

```yaoxiang
sort: (arr: Array(Int)) -> Sorted(arr) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 戦略 3：有界増加/減少パターン

`v += const`（正の定数）、変数に上界型注釈あり → 度量 `upper_bound - v` は各回 `const` 減少、下界 0。これは戦略 1 の縮退ケースであり、コンパイラは最初に高速処理する。

#### 6.5 戦略 4：乗法スケール度量テンプレート

`v *= const`（const > 1）、変数に上下界型注釈あり。コンパイラは対数度量テンプレート `ceil(log_const(upper/v))` を内蔵し、毎回 const 倍すると度量が 1 減少する。

```yaoxiang
mut i: Positive = 1
while i < n {
    # コンパイラが自動推論：度量 ceil(log₂(n/i))、毎回 2 倍すると度量が 1 減少
    i *= 2
}
```

#### 6.6 停止性と正当性の分離

停止性証明と正当性証明は独立である：
- **停止性**：上記四つの戦略がループの有限ステップ終了を自動証明する
- **正当性**：ループ本体が目標型に向かって進むか否かは、SMT ソルバが検証条件で検査する

両方通過 → コンパイル通過。停止性は証明されたが正当性が失敗 → コンパイルエラー + 反例。停止性を証明できない → コンパイルエラー、解析不能な変数または操作を示す。

#### 6.7 再帰関数の停止性検査

コンパイル時評価が必要な再帰関数について、コンパイラは引数の減少を検査する：

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # コンパイラ解析：n-1 < n → 減少 → 停止
}

# コンパイル時使用——コンパイラは factorial のコンパイル時停止を保証
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120、コンパイル時に完了
```

| シナリオ | 振る舞い |
|------|------|
| コンパイラが再帰的減少を解析できる（例：`n-1`） | コンパイル時評価 |
| 減少しない/減少の判定不能 | コンパイルエラー |
| 実行時呼び出し（型位置でない） | 停止性検査不要 |

#### 6.8 硬直した境界

`i = f(i)` で `f` が非可逆、非閉包、単調性を保持しない——数学的に自動停止証明は不可能。コンパイルエラー：

> このループは停止性を自動証明できない。ループ変数が解析不能な関数 `f` に依存する。コンパイラが解析可能な反復パターンを使用してください。

これはコンパイラの失敗ではない。静的に安全性を証明できないコードは、コンパイルを通過してはならない。

### 8. SMT ソルバ：コンパイル時評価器の後端コンポーネント

SMT ソルバは従来言語では外部ツールである（F\* が Z3 を呼ぶ、Dafny が Z3 を呼ぶ、等）。YaoXiang では、それはコンパイル時評価パイプラインの一つの後端コンポーネントである——コンパイラ自身が直接判定できない場合にのみ呼び出される。

```
コンパイル時 Bool 式
        │
        ├── コンパイラが直接判定可能（単純な比較、単純な算術、
        │   定数畳み込み後の自明な式）
        │   → 直接 True/False を返す
        │
        └── コンパイラが直接判定不能（量化子、記号変数）
            → 依存型事前簡約（factorial(5) → 120）
            → SMT-LIB 形式に翻訳
            → Z3/CVC5 に送信（予算制限付き）
            → 戻り値：unsat（成立）│ sat（反例）│ budget_exceeded
```

**ソルバ予算——スタック深度のようにハード制限**：

| 予算次元 | 説明 | ハード上限 |
|----------|------|-----------|
| ソルバステップ数 | 項書き換え/導出ステップ数 | コンパイラ内部の固定値 |
| 時間 | 単一述語のソルバ時間 | コンパイラ内部の固定値 |
| 量化子インスタンス化深さ | forall/exists の展開層数 | コンパイラ内部の固定値 |

予算超過は即座にコンパイルエラー。降格なし、実行時チェックなし、silent pass なし。エラーメッセージはソースコード中の述語位置まで正確で、どれだけの予算を消費したかを報告する。

**なぜこれが実際に機能するか**：工学的に実際の述語の 95% は線形算術である——`x > 0`、`arr.len > 0`、`0 <= idx < arr.len`——すべて決定可能フラグメント内にあり、SMT ソルバは这类の問題に対してミリ秒オーダーで返す。予算超過の非常に稀な複雑な述語に遭遇した場合、ユーザは述語を単純に分解すればよい。

依存型は SMT 呼び出し前に一層の事前簡約を行う：`factorial(5)` は直接コンパイル時評価で `120` になり、`append([1,2], [3])` は直接評価で `[1,2,3]` になる。これらの確定的な値計算は SMT 予算を消費しない。

プログラマは SMT の存在を知る必要がない。メンタルモデルは：**コンパイラが証明できれば通過、できなければエラー**——型検査と完全に一致する。

### 9. コンパイル時述語の組み合わせ

コンパイル時述語は Type を返す関数であり、組み合わせは関数合成を通じて自然に実現される：

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(T, arr)
}
```

### 10. コード例

#### 9.1 除算の安全性

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive) -> Int = a / b

result = divide(10, 2)   # ✅ ソルバ検証 { 2 > 0 } → True
# result = divide(10, 0)  # ❌ ソルバ検証 { 0 > 0 } → False
```

#### 9.2 配列アクセスの安全性

```yaoxiang
InBounds: (T: Type, idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (T: Type) -> ((arr: Array(T), idx: InBounds(T, arr)) -> T) = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ ソルバ検証 { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ ソルバ検証 { 0 <= 5 && 5 < 3 } → False
```

#### 9.3 ソートの正当性

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T)) -> Sorted(T)) = {
    result = arr.clone()
    # ... ソートアルゴリズム実装 ...
    return result
}
```

#### 9.4 ループ：コンパイラの VC 生成

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```

## 詳細設計

### 構文変更

| 以前 (RFC-022) | 以後 (本 RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | コンパイル時述語は引数型として `(b: Positive)` |
| `//! ensures: ExistsMax(result, arr)` | 戻り型はコンパイル時述語 `-> IsMax(T, arr)` |
| `/*! invariant: ... !*/` | 変数上のコンパイル時述語型注釈——Floyd-Hoare 不変条件 |
| `//! decreases: n` | コンパイラが度量関数を完全自動導出 |
| 仕様は注釈 | 仕様は型システム |

### 構文

**コンパイル時述語に新しい構文はない。** `{}` はソルバの検証空間であり、既存の型定義構文と完全に一致する。コンパイル時述語は Type を返す関数に他ならない——`name: (params) -> Type = { assertion }`。

```bnf
# コンパイル時述語 = Type を返す関数、{} 内はソルバassertion
# 既存の関数/型構文を使用し、新たな BNF ルールは不要
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

### 型システムへの影響

- **型宇宙**：コンパイル時述語は Type₂ 層に位置する——値を受け Type を返す関数、型コンストラクタと同じ階層
- **ジェネリクスとの相互作用**：コンパイル時述語はジェネリクスパラメータを取れる、例如 `NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **所有権との相互作用**：コンパイル時述語内の式は所有権規則に従い、読み取りのみ可能、書き込み不可
- **型推論**：コンパイル時述語のパラメータは HM 型推論に参加する

### 実行時表現

コンパイル時述語は実行時に**完全に消える**。`Positive: (x: Int) -> Type = { x > 0 }`——引数 `b: Positive` の実行時表現は `Int` そのものである。精化条件 `{ x > 0 }` はコンパイル時に SMT ソルバによって検証され、検証完了後は実行時の痕跡を一切残さない。

- `Positive(5)` → 実行時表現は `Int(5)`、精化条件 `{ 5 > 0 }` は通過済み、消去
- `SumUpTo(arr, 0)` → 実行時表現は `Int(0)`、等式 `0 == sum(arr[0..0])` は通過済み、消去
- コンパイル時述語を型位置に置く（例：`f(x: Positive)`）ことは、いかなるラッパー型も生成せず、追加メモリも確保せず、実行時チェックも挿入しない

ジェネリクス消去と精化型消去は同じ原則である：両者ともコンパイル時関数であり、コンパイル時評価後に完全に消える。Release Build においてコンパイル時述語の実行時オーバーヘッドはゼロ——これは「述語は関数である」の直接的な帰結である。

**`ref` との相互作用制約**：コンパイル時述語は不変借用または所有権が既に移動された値のみを参照できる。可変借用を参照するコンパイル時述語は、コンパイラが実行時にも検証結果が成立することを保証できない——这类の用法は直接コンパイルエラーを報告する。

### コンパイラ変更

1. **パーサー**：コンパイル時述語は標準的な関数構文を使用し、追加の解析ルールは不要
2. **コンパイル時評価パイプライン**：統一された Bool 戻りインターフェース、自動戦略選択
3. **SMT 後端**：Z3/CVC5 統合、SMT-LIB 形式翻訳
4. **検証条件生成**：WP/SP 計算 + ループ不変条件証明義務
5. **エラー報告**：反例フォーマット、ソースコード位置の関連付け

### 後方互換性

- ✅ コンパイル時述語を使用しないコードは完全に不変
- ✅ コンパイル時述語は Release Build で実行時オーバーヘッドゼロ
- ⚠️ RFC-022 の `//!` 構文はサポートされない——ただし 022 は一度も実装されておらず、移行負担なし

## トレードオフ

### 利点

- **Curry-Howard 同型の完全実現**：型は命題、プログラムは証明、`name: Proposition = Proof`
- **統一性**：コンパイル時述語と通常関数は完全に同じ構文を使用し、概念分裂なし
- **SMT 透過性**：プログラマは SMT の存在を知る必要がなく、メンタルモデルは型検査と一致
- **漸進的採用**：一つのコンパイル時述語から開始し、カバレッジを徐々に拡大可能
- **実行時オーバーヘッドゼロ**：すべての評価はコンパイル時に完了、Release Build には実行時assertionを一切挿入しない

### 欠点

- **コンパイル時間**：SMT ソルバがコンパイル時間を増加させるが、予算のハード制限により上限は制御可能
- **SMT 能力の境界**：線形算術を超える複雑な述語は予算超過の可能性があり、ユーザは述語を分解する必要がある
- **学習曲線**：効果的なコンパイル時述語の記述にはある程度の経験を要する
- **実装複雑性**：コンパイル時評価パイプラインの統一には慎重な設計が必要

### リスク軽減

- SMT ソルバ予算のハード制限（ステップ数/時間/インスタンス化深さ）、予算超過は即座にコンパイルエラー
- 依存型事前簡約：確定的な値計算を先に処理し、SMT は非確定部分のみを扱う
- 増分検証：変更されたモジュールのみを検証
- 明確なエラーメッセージ + 反例表示 + 予算消費レポート

## 代替案

| 案                             | 選択しない理由                           |
| -------------------------------- | -------------------------------------- |
| RFC-022：`//!` 注釈式仕様        | 仕様と型が分裂、Curry-Howard 同型に違反 |
| 独立仕様ファイル（例：CVL）     | 仕様とコードが分離され、保守コストが増加 |
| 実行時assertionのみ              | 静的に正当性を保証できない               |
| 外部証明アシスタント（例：Coq） | 学習曲線が急峻で、コンパイラと乖離       |
| **本方案：コンパイル時述語を一級市民として** | ✅                                     |

### 段階区分

| 段階       | 内容                                                                 |
| ---------- | -------------------------------------------------------------------- |
| **段階 1** | コンパイラが Bool を返す関数を型位置で使用可能と認識。単純な算術述語（`x > 0`、`arr.len > 0`）をサポート |
