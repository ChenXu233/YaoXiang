# RFC-030: assert アサート機構

## 概要

YaoXiang に `assert` アサート機構を導入し、テストと事前条件チェックに使用する。本 RFC では 2 つの実装案——**native 関数**と**キーワード**——を提案し、それぞれのトレードオフを分析して意思決定に資する。

## 動機

### なぜこの機能が必要か？

現在の YaoXiang の E2E テストは `if` + `io.println` + `return` でアサートを模倣することしかできない：

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

この書き方には 3 つの問題がある：

1. **ボイラープレートが多い**：各アサートに 4 行必要で、テストファイルが肥大化する
2. **エラーメッセージが弱い**：文字列を手動で連結するため、ソースコードの位置情報がない
3. **合成できない**：アサートを一括登録したり、テストフレームワークに引数として渡したりできない

### 現状の問題

- 統一されたアサート機構がない
- テストコードに `if` + プリント + `return` のパターンが蔓延している
- バイトコード層には既に `Throw` 命令があるが、言語層では公開されていない

## 案A：native 関数

`assert` を native 関数として実装し、新しいキーワードを導入しない。

```yaoxiang
use std.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### 型シグネチャ

```
assert: (Bool, String) -> ()
```

### ランタイム動作

1. 第 1 引数 `condition: Bool` を評価する
2. `true` の場合、`Unit` を返す
3. `false` の場合、ランタイムパニックを発生させる：
   - `message` の内容を出力する
   - コールスタックを出力する（デバッグモード時）
   - 現在の実行を終了する

### コンパイラの改変

**parser、AST、typecheck、IR gen の改変は不要。**

`src/std/` 配下に native 関数を登録するだけでよい：

1. `src/std/assert.rs` を新規追加
2. `std.assert` native 関数を登録
3. 内部で既存のランタイムパニック機構を呼び出す

### メリット

- **構文変更なし**：parser、AST、IR、codegen の改変が不要
- **実装が簡単**：native 関数の登録コードで約 20 行
- **第一級市民**：引数として渡したり、変数に代入したり、高階関数として使用したりできる
- **段階的な強化**：将来的にコードを壊すことなくコンパイル時のインライン化最適化を追加可能

### デメリット

- **コンパイル時解析がない**：コンパイラは `assert(false)` 以降のコードに対してデッドコード削除を行えない
- **エラーメッセージが限定的**：ソースコードの位置情報はランタイムのコールスタックから取得する必要があり、キーワードほど正確ではない
- **シャドーイング可能**：ユーザーが同名の関数を定義して `assert` を上書きできてしまう
- **import が必要**：使用するには `use std.assert` の記述が必須

## 案B：キーワード

`assert` を言語のキーワードとして導入し、コンパイラが IR 層で特別に処理する。

```yaoxiang
main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### 構文

```
assert '(' Expr ',' StringLit ')'
```

`assert` はキーワードであり、`use` でのインポートは不要。

### コンパイラの改変

全工程にわたる改変が必要：

| 段階 | 改変内容 |
|------|----------|
| Lexer | `KwAssert` トークンを追加 |
| Parser | `assert(expr, "msg")` 構文を追加 |
| AST | `StmtKind::Assert { condition, message }` を追加 |
| IR Gen | Assert を条件ジャンプ + Throw にコンパイル |
| Codegen | `JmpIfNot` + `Throw` 命令列を生成 |

### メリット

- **組み込みで利用可能**：import 不要で任意の場所で直接使用可能
- **ソースコード位置の精度**：コンパイラは AST 層で assert の位置を把握しており、エラーメッセージは行・列まで正確
- **シャドーイング不可**：キーワードはユーザーコードで上書きできない
- **将来的に拡張可能**：コンパイル時解析が可能（定数フォールディングパスと連携）
- **意味が明確**：`assert` は構文層で明確な意味を持ち、誤用されない

### デメリット

- **実装コストが高い**：parser、AST、IR、codegen の全工程にわたる改変が必要
- **第一級市民ではない**：引数として渡したり、変数に代入したりできない
- **キーワードの肥大化**：キーワードを追加するたびに言語の複雑度が増す
- **拡張性が低い**：新しいシグネチャの変種（例：`assert_eq`）を追加するには構文の改変が必要

## 比較

| 観点 | 案A：関数 | 案B：キーワード |
|------|----------|----------------|
| 実装コスト | 低（約 20 行） | 高（約 200 行、4 モジュール） |
| 構文変更 | なし | あり（新規キーワード + 構文） |
| import の要否 | 必要 | 不要 |
| 第一級市民 | あり | なし |
| ソースコード位置の精度 | ランタイムコールスタック | コンパイル時に行・列まで正確 |
| シャドーイング不可 | いいえ | はい |
| コンパイル時解析 | 不可（定数フォールディングが必要） | 可（定数フォールディングと連携） |
| 拡張性 | 高（オーバーロード、高階関数） | 低（構文改変が必要） |

### コンパイル時解析の現実的な制約

案B の核心的利点——コンパイル時解析——は**定数フォールディングパス**があって初めて有効になる。つまり、コンパイラが `assert(false)` 内の `false` をコンパイル時に評価して、初めてデッドコードだと判断できる。

YaoXiang には現在、定数フォールディングパスがない。案B を採用したとしても、`assert(x > 0)` のような一般的な書き方はコンパイル時には依然として解析できない。`assert(true)` / `assert(false)` のようなリテラルのみが解析可能である。

したがって案B のコンパイル時の利点は**現段階では理論上のものに過ぎず、実質的なものではない**。
