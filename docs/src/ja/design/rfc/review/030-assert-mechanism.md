# RFC-030: assert アサーション機構

## 概要

YaoXiang に `assert` アサーション機構を導入し、テストと事前条件チェックに使用する。`assert` は YaoXiang 唯一のランタイム panic 機構であり、独立した `raise`/`throw` キーワードを必要としない。コンパイル時条件型 `Assert(C)` は別の独立した機能であり（RFC-011 §4.3 を参照）、`std.assert` 同一モジュール下に配置されるが、セマンティクスと実装は完全に分離する。

## 動機

### なぜこの機能が必要か？

現在の YaoXiang の E2E テストは `if` + `io.println` + `return` でアサーションを模擬するしかない：

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

この書き方には 3 つの問題がある：

1. **ボイラープレートが多い**：各アサーションに 4 行必要で、テストファイルが肥大化する
2. **エラーメッセージが弱い**：文字列を手動で連結しており、ソース位置情報がない
3. **合成不可**：アサーションを一括登録したり、テストフレームワークに引数として渡したりできない

### 現状の問題

- 統一されたアサーション機構がない
- テストコードに `if` + 出力 + `return` のパターンが充斥している
- バイトコード層には既に `Throw` 命令が存在するが、言語層では公開されていない
- RFC-011 でコンパイル時 `Assert(C)` 条件型が定義されているが、ランタイム `assert()` は未実装である

### 設計原則

`assert` は YaoXiang 唯一のユーザーレベルの panic 機構である。`assert(false, "msg")` は `raise` と同等であり、独立した `throw`/`raise` キーワードを必要としない。`assert` 関数自体が `if raise` の最適なラッパーである。

**新しいキーワードを導入せず、新しい構文も導入しない。すべては関数呼び出しである。**

## 案 A：native 関数

`assert` を native 関数として実装し、新しいキーワードを導入しない。

```yaoxiang
use std.assert.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### オーバーロードシグネチャ

`assert` には 2 つのオーバーロードがある：

```
// オーバーロード 1：条件アサーション
assert(cond: Bool, ?msg: String | Error)

// オーバーロード 2：Result アサーション
assert(cond: Result)
```

#### オーバーロード 1：条件アサーション `(Bool, ?String | Error)`

`Bool` + 任意のメッセージ。メッセージは `String` または `Error` 値：

```yaoxiang
assert(1 + 1 == 2)                    // メッセージなし、デフォルトの panic 情報
assert(1 + 1 == 2, "math is broken")   // 文字列メッセージ
assert(x > 0, my_error)                // Error 値を直接 panic
```

`assert(false, "msg")` は YaoXiang の `raise`/`throw` 等価体であり、独立したキーワードを必要としない。

#### オーバーロード 2：Result アサーション `(Result)`

`Result` 1 つの引数を取り、自動的に `Err` かどうかをチェックする：

### メリット

- **構文変更なし**：純粋な関数であり、新しいキーワードが不要
- **新概念なし**：既存の native 関数登録機構を再利用
- **高い拡張性**：関数オーバーロードにより複数のシグネチャを自然にサポート
- **自己文書化**：`std.assert` 名前空間自体がドキュメント

### デメリット

- なし。`assert` の型シグネチャが正しければ、コンパイラは関数の到達可能性解析でデッドコードを推測可能。追加の pass は不要。

### ランタイム動作

1. 第 1 引数 `condition: Bool` を評価する
2. `true` の場合、`Unit` を返す
3. `false` の場合、ランタイム panic を発動する：
   - `message` の内容を出力する（ある場合）
   - コールスタックを出力する（デバッグモード）
   - 現在の実行を終了する

#### 各オーバーロードの失敗時の動作

| シグネチャ | 失敗時の動作 |
|------|-----------|
| `assert(false)` | デフォルトの panic 情報 |
| `assert(false, "msg")` | 文字列メッセージを出力後 panic |
| `assert(false, error_val)` | Error 値で panic |
| `assert(Err(x))` | Err 内容を抽出して panic |

### コンパイル時 Assert との関係

`std.assert` モジュールには 2 つの完全に独立した機能が含まれる：

| 機能 | 種類 | 名称 | タイミング | 失敗時の動作 | ランタイムオーバーヘッド |
|------|------|------|------|----------|-----------|
| ランタイムアサーション | 関数 | `assert()` | ランタイム | panic + コールスタック | 常に存在 |
| コンパイル時条件型 | 型 | `Assert(C)` | コンパイル時 | コンパイルエラー | ゼロ |

```yaoxiang
use std.assert

// ランタイム assert：テスト用、ランタイムの値を検査
assert.assert(result == 42, "expected 42")

// コンパイル時 Assert：型の精化、ジェネリクス引数の検証
length: assert.Assert(N > 0)
```

両者を同一モジュールに配置するのは、どちらも「アサーション」概念の異なる表現であるためだが、実装は完全に独立している。`assert()` は `Assert(C)` を必要としないし、逆も同様である。

### コンパイラの変更

**parser、AST、typecheck、IR gen の変更は不要。**

`src/std/` 配下に native 関数の登録を追加するだけである：

1. `src/std/assert.rs` を新設
2. `std.assert.assert` と `std.assert.Assert` を登録する（後者はコンパイル時条件型で、#155 を参照）
3. 内部的には既存の `BytecodeInstr::Throw` 命令を呼び出す

### メリット

- **構文変更なし**：純粋な関数であり、新しいキーワードが不要
- **新概念なし**：既存の native 関数登録機構を再利用
- **高い拡張性**：関数シグネチャは `assert_eq` などのバリアントに拡張可能（将来）
- **自己文書化**：`std.assert` 名前空間自体がドキュメント

### デメリット

- コンパイル時に知ることができない：案 B（キーワード）と異なり、コンパイル時にデッドコード除去を行うことができない
- デバッグモードでのみコールスタックを取得可能

## 案 B：組み込みキーワード

> 既に廃止済み、歴史的記録としてのみ残す。

`assert` キーワードで実装し、コンパイル時にソース位置と定数畳み込みを取得可能。

```yaoxiang
assert(1 + 1 == 2, "math is broken")
```

### 型シグネチャ

独立した型シグネチャはない。キーワードはパーサで処理される。

### ランタイム動作

案 A と同じ。

### コンパイラの変更

parser、AST、typecheck、IR gen の変更が必要：
1. parser：`Expr::Assert` バリアントを追加
2. AST：`Expr::Assert` ノードを追加
3. typecheck：引数の型を検証
4. IR gen：`BytecodeInstr::Throw` を生成

### メリット

- コンパイル時にソース位置が分かる（debug info に依存しない）
- コンパイル時に定数畳み込み可能：`assert(true)` → 空の操作、`assert(false)` → コンパイルエラー

### デメリット

| デメリット | 影響 |
|------|-----------|
| パーサの変更が必要 | 新しい構文ノードを導入し、保守コストが増加 |
| キーワードは拡張不可 | `assert_eq` などのバリアントも依然として関数が必要 |
| コンパイル時の優位点は非実用的 | 後述の分析を参照 |

### 比較

| 観点 | 案 A（関数） | 案 B（キーワード） |
|------|---------------|-----------------|
| 実装コスト | 約 20 行 | parser + AST + typecheck + IR gen |
| 構文変更 | なし | 新しいキーワード |
| 拡張性 | 関数オーバーロード | 付随するマクロが必要 |
| ソース位置 | debug info | コンパイル時に取得可能 |
| 定数畳み込み | pass のサポートが必要 | コンパイル時に取得可能 |
| ランタイムオーバーヘッド | 関数呼び出し | 極小 |

### コンパイル時分析の現実的な制約

案 B の核心的優位点であるコンパイル時分析は、**定数畳み込み pass** がなければ効力を発揮しない。すなわち、コンパイラがコンパイル時に `assert(false)` 中の `false` を評価して、これがデッドコードであることを知る必要がある。

YaoXiang には現在定数畳み込み pass が存在しない。仮に案 B を採用したとしても、`assert(x > 0)` のような一般的な書き方はコンパイル時に解析できない。`assert(true)` / `assert(false)` のようなリテラルのみが解析可能である。

したがって案 B のコンパイル時の優位点は**現段階では理論的なものに過ぎず、実用的ではない**。

- [x] ~~`assert` は `message` を伴わない簡易形式 `assert(cond)` をサポートすべきか？~~ → **サポートする。`assert(cond, ?msg)`、message は任意。**
- [x] ~~`assert_eq`、`assert_ne` などのバリアントは必要か？~~ → **不要。YAGNI。テストフレームワークが形になってから考える。**
- [x] ~~panic 出力にソース位置を含めるべきか？~~ → 案 A は debug info（コールスタック）に依存する。
- [ ] ~~assert / Assert 統一問題~~ → **独立した議論 #156 として切り出した**。理想形は `assert: (cond: Bool) -> Assert(cond)` で両者を統一することだが、`Assert(C)` は精化型として `C` がコンパイル時に既知であることを要求し、ランタイムアサーションと衝突する。詳細は [#156](https://github.com/ChenXu233/YaoXiang/issues/156) を参照。

## 付録 A：設計議論の記録

### 2026-07-05：案 A の選択

案 A の 20 行実装が価値とコストで勝った。案 B のコンパイル時の優位点は定数畳み込み pass が現状存在しない以上、理論的なものに過ぎない。

### 2026-07-11：assert オーバーロード設計

**問題**：なぜ `assert` は `(Bool, ?String)` に統一せず、2 つのオーバーロードを必要とするのか？

**解答**：

ランタイム `assert()` は YaoXiang 唯一のユーザーレベルの panic 機構である。`assert(false, "msg")` は他の言語の `raise`/`throw` と等価である。したがって 3 つのシナリオをカバーする必要がある：
1. 条件 + 単純なメッセージ：`assert(cond, "msg")`
2. 条件 + カスタム Error：`assert(cond, my_error)`
3. Result チェック：`assert(result)` — `if is_err { panic }` の最も簡潔な書き方

Result オーバーロードの合理性は、これがエラー伝播の最短経路だからである — 「Result は Ok であるべき、さもなくば死」。`.is_ok()` を先に呼んでエラーを別途処理する必要はない。

### 2026-07-11：std.assert モジュール構造

`std.assert` には 2 つの機能が含まれる：

- `assert()`：ランタイムアサーション関数（本 RFC）
- `Assert(C)`：コンパイル時条件型（RFC-011 §4.3、#155）

両者は「アサーション」概念の異なる表現であるため、同一モジュールに配置される。名前空間として、`std.assert` は自己文書化される — 開発者が `use std.assert` を見れば、ここでアサーション関連の機能を導入していることが分かる。
