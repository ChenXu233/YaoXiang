# RFC-030: assert アサーション機構

## 概要

YaoXiang に `assert` アサーション機構を導入し、テストと事前条件のチェックに使用する。本 RFC では 2 つの実装プラン——**native 関数**と**キーワード**——を提案し、それぞれのトレードオフを分析して意思決定に資する。

## 動機

### なぜこの機能が必要なのか？

現在の YaoXiang の E2E テストは `if` + `io.println` + `return` でアサーションを模倣することしかできない：

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

この記述方法には 3 つの問題がある：

1. **ボイラープレートが多い**：各アサーションに 4 行必要で、テストファイルが肥大化する
2. **エラーメッセージが貧弱**：文字列を手動で結合しており、ソースコードの位置情報がない
3. **合成できない**：アサーションを一括登録したり、テストフレームワークに引数として渡したりできない

### 現状の問題

- 統一されたアサーション機構がない
- テストコードに `if` + 出力 + `return` のパターンが充斥している
- バイトコード層には既に `Throw` 命令があるが、言語層では公開されていない

## プラン A：native 関数

`assert` を native 関数として実装し、新たなキーワードを導入しない。

```yaoxiang
use std.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### 型シグネチャ

```
assert: (Bool, String) -> ()
```

### ランタイム動作

1. 第 1 引数 `condition: Bool` を評価する
2. `true` の場合、`Unit` を返す
3. `false` の場合、ランタイム panic を発動する：
   - `message` の内容を出力
   - （デバッグモードでは）コールスタックを出力
   - 現在の実行を終了

### コンパイラの変更点

**parser、AST、typecheck、IR gen の変更は不要。**

`src/std/` 配下で native 関数の登録を行うのみ：

1. `src/std/assert.rs` を新規追加
2. `std.assert` native 関数を登録
3. 内部で既存のランタイム panic 機構を呼び出す

### メリット

- **構文変更ゼロ**：parser、AST、IR、codegen を変更する必要がない
- **実装が単純**：約 20 行の native 関数登録コードで完了
- **第一級市民**：引数として渡したり、変数に代入したり、高階関数に使用したりできる
- **漸進的拡張**：将来コードを壊すことなくコンパイル時インライン化最適化を追加可能

### デメリット

- **コンパイル時解析なし**：コンパイラは `assert(false)` 以降のコードに対してデッドコード除去を行えない
- **エラーメッセージが制限される**：ソースコードの位置情報はランタイムのコールスタックから取得する必要があり、キーワードほど正確ではない
- **シャドーイング可能**：ユーザーが同名の関数を定義して `assert` を上書きできる
- **import が必要**：使用するには `use std.assert` を記述する必要がある

## プラン B：キーワード

`assert` を言語キーワードとして導入し、コンパイラが IR 層で特別に処理する。

```yaoxiang
main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### 構文

```
assert '(' Expr ',' StringLit ')'
```

`assert` はキーワードであり、`use` によるインポートは不要。

### コンパイラの変更点

全工程の変更が必要：

| 段階 | 変更内容 |
|------|----------|
| Lexer | `KwAssert` トークンを追加 |
| Parser | `assert(expr, "msg")` 構文を追加 |
| AST | `StmtKind::Assert { condition, message }` を追加 |
| IR Gen | Assert を条件分岐 + Throw にコンパイル |
| Codegen | `JmpIfNot` + `Throw` 命令列を生成 |

### メリット

- **組み込みで利用可能**：import が不要で、どこからでも直接使用できる
- **ソースコード位置の正確性**：コンパイラが AST 層で assert の位置を把握しており、エラーメッセージは行番号・列番号まで正確
- **シャドーイング不可**：キーワードはユーザーコードで上書きできない
- **将来の拡張性**：コンパイル時解析（定数畳み込みパスとの組み合わせ）が可能
- **セマンティクスが明確**：`assert` は構文層で明確な意味を持ち、誤用されない

### デメリット

- **実装コストが高い**：parser、AST、IR、codegen の全工程を変更する必要がある
- **第一級市民ではない**：引数として渡したり、変数に代入したりできない
- **キーワードの肥大化**：キーワードを追加するたびに言語の複雑度が増す
- **拡張性が低い**：新たなシグネチャ変種（例：`assert_eq`）の追加には構文変更が必要

## 比較

| 観点 | プラン A：関数 | プラン B：キーワード |
|------|----------------|----------------------|
| 実装コスト | 低（約 20 行） | 高（約 200 行、4 モジュール） |
| 構文変更 | なし | あり（キーワード + 構文の追加） |
| import の要否 | 必要 | 不要 |
| 第一級市民 | はい | いいえ |
| ソースコード位置の精度 | ランタイムのコールスタック | コンパイル時に行・列まで正確 |
| シャドーイング不可 | いいえ | はい |
| コンパイル時解析 | 不可（定数畳み込みが必要） | 可（定数畳み込みと組み合わせ） |
| 拡張性 | 高（オーバーロード、高階関数） | 低（構文変更が必要） |

### コンパイル時解析の現実的な制約

プラン B の核心的メリット——コンパイル時解析——は**定数畳み込みパス**があって初めて機能する。つまり、コンパイラがコンパイル時に `assert(false)` 中の `false` を評価して、これがデッドコードであると認識する必要がある。

YaoXiang には現在定数畳み込みパスがない。仮にプラン B を採用したとしても、`assert(x > 0)` のような一般的な記述はコンパイル時には解析できない。解析可能なのは `assert(true)` / `assert(false)` のようなリテラルの場合のみである。

したがって、プラン B のコンパイル時利点は**現段階では理論的であり、実用的ではない**。

## オープンな課題

- [x] ~~プラン A とプラン B のどちらを選択するか？~~ → **プラン A：native 関数**
- [ ] `assert` は `message` を伴わない簡略形式 `assert(cond)` をサポートすべきか？
- [ ] `assert_eq`、`assert_ne` などの変種が必要か？
- [x] ~~panic 出力にソースコード位置を含めるか？~~ → プラン A は debug info（コールスタック）に依存し、プラン B はコンパイル時に取得可能。プラン A を採用。後に定数畳み込みパスを追加すれば、自然にインライン化能力が得られる
