# RFC-030: assert 表明機構

## 概要

YaoXiang に `assert` 表明機構を導入し、テストおよび前提条件チェックに使用する。本 RFC では 2 つの実装案——**ネイティブ関数**と**キーワード**——を提示し、それぞれのトレードオフを分析した上で意思決定に資する。

## 動機

### なぜこの機能が必要か？

現在の YaoXiang の E2E テストは、`if` + `io.println` + `return` で表明を模倣することしかできない：

```yaoxiang
val = some_func()
if val != 42 {
    io.println("失敗: 期待値 42")
    return
}
```

この書き方には 3 つの問題がある：

1. **ボイラープレートが多い**：各表明につき 4 行必要で、テストファイルが肥大化する
2. **エラーメッセージが弱い**：手動で文字列を連結するため、ソースコード位置が欠如する
3. **合成できない**：表明を一括登録したり、テストフレームワークに引数として渡したりできない

### 現状の問題

- 統一された表明機構が存在しない
- テストコードが `if` + プリント + `return` のパターンであふれている
- バイトコード層には既に `Throw` 命令が存在するが、言語層では公開されていない

## 案 A：ネイティブ関数

`assert` をネイティブ関数として実装し、新しいキーワードを導入しない。

```yaoxiang
use std.assert

main = {
    assert(1 + 1 == 2, "計算がおかしい")
    assert(get_name() == "YaoXiang", "名前が一致しません")
}
```

### 型シグネチャ

```
assert: (Bool, String) -> ()
```

### ランタイム動作

1. 第 1 引数 `condition: Bool` を評価する
2. `true` であれば、`Unit` を返す
3. `false` であれば、ランタイムパニックを発生させる：
   - `message` の内容を出力
   - コールスタックを出力（デバッグモード時）
   - 現在の実行を終了

### コンパイラの変更

**parser、AST、typecheck、IR gen の変更は不要。**

`src/std/` 配下でのネイティブ関数登録のみが必要：

1. `src/std/assert.rs` を新規追加
2. `std.assert` ネイティブ関数を登録
3. 内部で既存のランタイムパニック機構を呼び出す

### 利点

- **構文変更ゼロ**：parser、AST、IR、codegen を変更する必要がない
- **実装が簡単**：ネイティブ関数登録コード約 20 行
- **第一級市民**：引数として渡したり、変数に代入したり、高階関数で使用したりできる
- **漸進的な拡張**：コードを壊すことなく、将来的にコンパイル時インライン化最適化を追加可能

### 欠点

- **コンパイル時解析がない**：`assert(false)` 以降のコードに対してデッドコード除去を適用できない
- **エラーメッセージが限定的**：ソースコード位置はランタイムのコールスタックから取得する必要があり、キーワードほど正確ではない
- **シャドーイング可能**：ユーザーが同名の関数を定義して `assert` を上書きできる
- **import が必要**：使用するには `use std.assert` と記述する必要がある

## 案 B：キーワード

`assert` を言語キーワードとして導入し、コンパイラが IR 層で特別に処理する。

```yaoxiang
main = {
    assert(1 + 1 == 2, "計算がおかしい")
    assert(get_name() == "YaoXiang", "名前が一致しません")
}
```

### 構文

```
assert '(' Expr ',' StringLit ')'
```

`assert` はキーワードであり、`use` によるインポートは不要。

### コンパイラの変更

全工程にわたる変更が必要：

| 段階 | 変更内容 |
|------|----------|
| Lexer | `KwAssert` トークンを追加 |
| Parser | `assert(expr, "msg")` 構文を追加 |
| AST | `StmtKind::Assert { condition, message }` を追加 |
| IR Gen | Assert を条件分岐 + Throw にコンパイル |
| Codegen | `JmpIfNot` + `Throw` 命令列を生成 |

### 利点

- **組み込みで利用可能**：import 不要、任意の場所で直接使用可能
- **正確なソースコード位置**：コンパイラは AST 層で assert の位置を把握しているため、エラーメッセージは行番号・列番号まで正確
- **シャドーイング不可**：キーワードはユーザーコードで上書きできない
- **将来的に拡張可能**：コンパイル時解析が可能（定数畳み込みパスを併用）
- **セマンティクスが明確**：`assert` は構文層で明確なセマンティクスを持ち、誤用されない

### 欠点

- **実装コストが高い**：parser、AST、IR、codegen の全工程を変更する必要がある
- **第一級市民ではない**：引数として渡したり、変数に代入したりできない
- **キーワードの肥大化**：キーワードを 1 つ追加するたびに言語の複雑度が増す
- **拡張性が低い**：新しいシグネチャ変種（`assert_eq` など）を追加するには構文変更が必要

## 比較

| 観点 | 案 A：関数 | 案 B：キーワード |
|------|-----------|------------------|
| 実装コスト | 低（~20 行） | 高（~200 行、4 モジュール） |
| 構文変更 | なし | あり（新規キーワード + 構文） |
| import の必要性 | あり | なし |
| 第一級市民 | はい | いいえ |
| ソースコード位置の精度 | ランタイムコールスタック | コンパイル時に行・列まで正確 |
| シャドーイング不可 | いいえ | はい |
| コンパイル時解析 | 不可（定数畳み込みが必要） | 可能（定数畳み込みと併用） |
| 拡張性 | 高（オーバーロード、高階関数） | 低（構文変更が必要） |

### コンパイル時解析の現実的な制約

案 B の核心的な利点——コンパイル時解析——が機能するには、**定数畳み込みパス**が必要である。すなわち、コンパイラがコンパイル時に `assert(false)` の `false` を評価して、これがデッドコードであることを認識する必要がある。

YaoXiang には現在、定数畳み込みパスが存在しない。仮に案 B を採用したとしても、`assert(x > 0)` のような一般的な記述はコンパイル時には解析できない。`assert(true)` / `assert(false)` のようなリテラルのみが解析可能である。

したがって、案 B のコンパイル時の利点は**現段階では理論上のものに過ぎず、実質的なものではない**。
