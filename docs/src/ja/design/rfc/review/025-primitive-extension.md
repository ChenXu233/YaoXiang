# RFC-025: 拡張可能なプリミティブ型メカニズム

## 要約

本文書では YaoXiang コンパイラの**拡張可能なプリミティブ型メカニズム**（`Primitive::Extension`）を定義する。外部コードがカスタムプリミティブ型をコンパイラに登録できるようにし、コンパイラがハードコーディングなしでドメイン固有の型（量子ビット、GPU バッファ、SIMD ベクトル、ハードウェアレジスタなど）をサポートできるようにする。

## 動機

### なぜこのメカニズムが必要なのか？

現在、コンパイラはすべてのプリミティブ型をハードコーディングしている：`Int`、`Float`、`String`、`Bool`、`Unit`。新しいプリミティブ型を追加するたびに、型検査器、コードジェネレータ、所有権解析器、オプティマイザなど、コンパイラのソースコードの複数箇所を修正する必要がある。

これはオープン・クローズド原則に違反している：拡張に対して開かれており、修正に対して閉じられているべきである。

### 設計の境界

```
ハードコードされたコア型（言語の基礎）: Int, Float, String, Bool, Unit
動的に拡張可能な型（ドメインプラグイン）: Primitive::Extension 経由で登録
```

コア型をハードコードするのは、コンパイラがそれらの意味論（算術演算、条件分岐、ハッシュ、比較）に深く依存しているためである。拡張型はコンパイラにとって**不透明な値**である——コンパイラはそれらのサイズ、アラインメント、所有権属性のみを知り、内部の意味論は知らない。

これは「すべての型を動的読み込みに統一する」ということではない。コア型と拡張型は異なるものである。

## 提案

### コア設計

#### 1. Extension 型の属性

各拡張プリミティブ型を登録する際、以下の属性を宣言しなければならない：

```rust
pub struct PrimitiveExtension {
    /// 型名、例如 "Qubit", "Buffer", "Vec128"
    pub name: String,

    /// バイトサイズ（固定サイズ型）
    /// None はサイズがコンパイル時に未知であることを示す（実行時に決定が必要）
    pub size: Option<usize>,

    /// アラインメント要件
    pub align: Option<usize>,

    /// 暗黙のコピーを許可するか
    /// false = Move 意味論（代入で移動、例：Qubit）
    /// true = Copy 意味論（代入でコピー、例：Vec128）
    pub is_copy: bool,

    /// 空値（zero-sized type）を許可するか
    pub allow_zst: bool,
}
```

#### 2. 登録インターフェース

```rust
// コンパイラ内部 API
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".to_string(),
    size: Some(0),           // 論理サイズ0、物理状態は量子プロセッサ上
    align: Some(1),
    is_copy: false,          // Move 意味論、no-cloning に準拠
    allow_zst: true,
});
```

登録後、`Qubit` は型システムにおいて合法なプリミティブ型となり、変数宣言、関数引数、構造体フィールドに使用できる。

#### 3. 型検査器の動作

拡張プリミティブ型は型検査において以下の規則に従う：

| シナリオ | 動作 |
|------|------|
| 変数宣言 `q: Qubit = ...` | ✅ 合法 |
| 関数引数 `fn(q: Qubit)` | ✅ 合法 |
| 構造体フィールド `{ q: Qubit }` | ✅ 合法 |
| `is_copy == false` の場合の代入 | Move 意味論、元の変数が無効化される |
| `is_copy == true` の場合の代入 | Copy 意味論、元の変数は保持される |
| 暗黙のコピー（関数の複数箇所での使用） | `is_copy` に依存する |
| 比較 `==`、`!=` | ❌ コンパイルエラー（組み込みの比較なし） |
| 算術 `+`、`-` | ❌ コンパイルエラー（組み込みの演算なし） |
| ジェネリック制約 `T: Copy` | `is_copy == true` の場合のみ満たされる |

#### 4. コードジェネレータの動作

拡張プリミティブ型はコード生成において**不透明な値**として扱われる：

- LLVM IR：対応するサイズの `{size} x i8` または構造体として生成
- 特殊な命令は生成されない——意味論はバックエンドまたはライブラリが担当
- バックエンドが特別な処理を必要とする場合（例：QIR 量子ゲート）は、バックエンド登録メカニズムを通じて実装する（本 RFC の範囲外）

### 例

#### Move 意味論の型の登録

```rust
// 量子ビット：コピー不可、サイズ0（物理状態は QPU 上）
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".into(),
    size: Some(0),
    align: Some(1),
    is_copy: false,
    allow_zst: true,
});
```

```yaoxiang
# ユーザーコード
q: Qubit = qubit(0)
q2 = q          # ❌ コンパイルエラー：Qubit は Move 型であり、q は既に無効
q = H(q)        # ✅ q を消費し、新しい q を返す
```

#### Copy 意味論の型の登録

```rust
// SIMD ベクトル：コピー可、固定サイズ
compiler.register_primitive(PrimitiveExtension {
    name: "Vec128".into(),
    size: Some(16),
    align: Some(16),
    is_copy: true,
    allow_zst: false,
});
```

```yaoxiang
# ユーザーコード
a: Vec128 = load_vec128(data)
b = a           # ✅ Copy 意味論、a は依然として有効
c = add_vec128(a, b)  # ✅ a と b の両方が使用可能
```

## 詳細設計

### コンパイラの改変

| コンポーネント | 改変内容 |
|------|------|
| 型システム | `Ty::Extension` バリアントを新設し、`PrimitiveExtension` メタデータを保持 |
| 型検査器 | 拡張型は組み込み演算の解決に参加せず、明示的な実装なしに組み込み trait 制約を満たさない |
| 所有権解析器 | `is_copy` に基づいて Move または Copy 意味論を決定 |
| コードジェネレータ | `size`/`align` に従って不透明な値を生成し、特殊な命令は生成しない |
| エラーメッセージ | 拡張型のエラーメッセージは登録時の `name` を参照する |

### FFI との関係

`Primitive::Extension` は RFC-021（FFI）と直交関係にある：

| | Primitive::Extension | FFI |
|---|---|---|
| 役割 | 新しい**型**を登録 | 外部**関数**の呼び出し |
| レイヤー | 型システム | ランタイム |
| 例 | `Qubit` は型である | `native("sin")` は関数呼び出しである |

ドメインによっては両方を必要とする場合がある：`Qubit` 型は Extension 経由で登録し、量子ゲート関数は FFI 経由で登録する。

### 後方互換性

- ✅ 完全な後方互換性
- 既存型の意味論は一切変更されない
- 拡張型は新たな機能であり、既存コードに影響しない

## トレードオフ

### 利点

- ✅ 新しいドメインごとにコンパイラのソースコードを修正する必要がない
- ✅ ドメイン専門家がコンパイラチームに依存せず独自に型を登録できる
- ✅ コア型はハードコーディングを維持し、基本型に対するコンパイラの深い最適化を犠牲にしない
- ✅ インターフェースがシンプル：1 つの構造体ですべての属性を定義

### 欠点

- ⚠️ 拡張型は組み込み演算をサポートしない——意味論を実装するには追加の関数やバックエンドメカニズムが必要
- ⚠️ デバッグ時に拡張型は不透明な値として表示され、コア型ほど直感的ではない

## 代替案

| 代替案 | 採用しない理由 |
|------|--------------|
| すべての型を動的読み込み | コア型（Int/Float/Bool）はコンパイラの深い最適化を必要とし、動的読み込みではこれらの能力が失われる |
| ドメインごとにハードコード | ドメインを追加するたびにコンパイラを修正する必要があり、拡張性がない |
| 純粋なライブラリ方式（型を登録しない） | 型システムレベルで意味論（no-cloning など）を保証できず、ランタイムチェックに限定される |

## 実装戦略

### フェーズ 1：コアインターフェース

- [ ] 型システムに `Ty::Extension` バリアントを追加
- [ ] `register_primitive` API を実装
- [ ] 型検査器を拡張して `is_copy` 意味論を処理
- [ ] コードジェネレータを拡張して不透明な値を処理
- [ ] ユニットテスト

### フェーズ 2：登録タイミング

- [ ] コンパイラの初期化時の一括登録をサポート（設定ファイルまたは builder API）
- [ ] 標準ライブラリによる事前登録をサポート（`std.primitive` モジュールが拡張型定義をエクスポート）

### 依存関係

- ハードな依存関係なし。他の RFC と独立して実装可能。

## 未解決の問題

## 設計上の決定記録

| 決定 | 決定内容 | 理由 | 日付 |
|------|------|------|------|
| コア型は動的読み込みとしない | ハードコーディングを維持 | コンパイラはコア型の意味論に深く依存しており、動化しても利益はゼロ | 2026-06-05 |
| 拡張型を不透明な値とする | 意味論を注入しない | 意味論はバックエンド/ライブラリが担当し、コンパイラは型安全性と所有権のみを保証する | 2026-06-05 |
| FFI と直交関係とする | 統合しない | 型の登録と関数の呼び出しは異なる抽象化レイヤーである | 2026-06-05 |
