# RFC-003：バージョン計画

> **状態**: 却下
> **作成者**: 晨煦
> **作成日**: 2025-01-05
> **最終更新**: 2025-01-06
> **却下日**: 2026-06-01

## ⚠️ 却下理由

**本RFCはRFCの規格に適合していないため、却下されました。**

### 問題分析

| 問題 | 説明 |
|------|------|
| **ドキュメントタイプエラー** | RFCは「なぜこのような設計なのか」という意思決定のドキュメントであり、本文档は「現在どこにいるか、どこに向かうか」というプロジェクト管理のドキュメントである |
| **設計上の意思決定が欠落** | 技術選択のトレードオフ分析がない、代替え案の比較がない、設計上の理由がない |
| **過度な計画** | 9つのバージョン（v0.1-v1.0）を計画しているが、完了したのは1.5つのみで、v0.3以降はすべて推測である |
| **維持コストが高い** | 成功指標テーブルには50もの状態があり、維持が困難である |
| **バージョン番号の混乱** | v0.8をスキップしているが、説明がない |

### 正しいドキュメントタイプ

本文書はRFCではなく、**プロジェクトロードマップ（Roadmap）**として管理すべきです。ロードマップは事実と短期計画を記録し、RFCは設計上の意思決定を記録します。

## 摘要

YaoXiangのバージョンリリース計画、v0.1からv1.0までのロードマップ。

**コア目標**：
1. **バイトコードコンパイル**：REPLと漸進的コンパイルをサポート
2. **自挙 (Bootstrap)**：YaoXiangでYaoXiangコンパイラを作成
3. **AOTコンパイル**：バイトコードからローカル機械語へのコンパイル

## 一、動機

### なぜバージョン計画が必要인가？

1. **プロジェクト管理**：目標を実行可能なマイルストーンに分解する
2. **ユーザーの期待**：ユーザーに言語の開発段階を理解してもらう
3. **リソース配分**：各段階の重点を明確にする
4. **リスク管理**：問題を早期に発見し、方向性を調整する

### コア設計上の意思決定

- **バイトコード優先**：まずインタープリタ実行を実装し、その後AOTを検討
- **漸進的デリバリー**：各バージョンに実用的な機能を含める
- **後方互換性**：v1.0前はAPIが変更される可能性があるが、事前に告知する
- **自挙による検証**：自挙により言語の表現力を証明
- **パフォーマンスの階層化**：まず動作させ、それから最適化

## 四、コンパイル戦略の三層設計

| 階層 | バージョン | 入力 | 出力 | 説明 |
|------|------|------|------|------|
| L1: バイトコード | v0.1+ | ソースコード (.yx) | バイトコード (.yxb) | VM解釈実行 |
| L2: 自挙 | v0.9+ | YaoXiangソースコード | バイトコード | 自己コンパイル |
| L3: AOT | v1.0+ | ソースコード/バイトコード | 機械語 | ネイティブパフォーマンス |

**バイトコード優先の理由**：
1. **REPLサポート**：入力コードを即座にコンパイルし、対話型開発を実現
2. **漸進的コンパイル**：単一の関数を修正只需重新编译该部分
3. **プラットフォーム非依存**：.yxbファイルはクロスプラットフォームで実行可能で対応プラットフォームのVMが必要

## 五、依存関係戦略

**短期**：Rustライブラリの呼び出しでcrates.ioを活用（Cargo寄生）

**現在の依存関係**：
- 並行処理：parking_lot, crossbeam, rayon
- データ構造：indexmap, hashbrown, smallvec
- ネットワーク：tokio
- シリアライゼーション：serde, ron

**長期**：独自の標準ライブラリとパッケージマネージャーの構築
