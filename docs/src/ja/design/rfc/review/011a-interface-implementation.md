# RFC-011a: インターフェース実装と動的ディスパッチ  

> **親 RFC**: [RFC-011: ジェネリクスシステム設計](../accepted/011-generic-type-system.md)
>
> **本 RFC は RFC-011 §2.1-2.4 のインターフェース制約部分を補足し、置換します。**

## 摘要

RFC-011 はジェネリクスシステムを定義しましたが、インターフェース実装メカニズムについては詳しく説明されていません。本ドキュメントでは以下を補足します：

1. **インターフェース宣言**：型定義に直接インターフェース名を記述し、`impl` キーワードは不要
2. **メソッド実装**：内部宣言と外部宣言の両方をサポート
3. **オーバーロード規則**：シグネチャが異なればオーバーロードを許可、同じであればエラー（オーバーライド禁止）
4. **デフォルト値**：`= value` をフィールドの直後に記述
5. **動的ディスパッチ**：コンパイル時の型収集＋インターフェースマッチング仮想テーブルなし

**コア設計**：

```yaoxiang
# インターフェース定義
Animal: Type = {
    speak: (Self) -> String,
}

# 型定義（内部宣言）
Dog: Type = {
    x: Int = 10,
    Animal,  # インターフェース宣言
    speak: (Self) -> String = "Woof",
}

# 外部宣言（オーバーロード）
Dog.speak: (Self, volume: Int) -> String = "WOOF"

# 異種コンテナ（動的ディスパッチ）
animals: List(Animal) = [Dog.new(), Cat.new()]
animals[0].speak()  # "Woof"
```

**排除した複雑さ**：
- ❌ `impl` キーワードなし
- ❌ `dyn Trait + 'a` アノテーションなし
- ❌ 仮想テーブルなし（コンパイル時型収集＋列挙型ラッパー）
- ❌ オーバーライドなし（オーバーロード規則を統一）

## 動機

### RFC-011 の不足

RFC-011 はジェネリクスシステムを定義しましたが、以下の点是詳細に説明されていません：

| 問題 | 説明 |
|------|------|
| インターフェース宣言構文 | 型がインターフェースを実装することをどのように宣言するか？ |
| メソッド実装位置 | 内部宣言か外部宣言か？ |
| オーバーロード規則 | 同名メソッドの処理方法 |
| デフォルト値構文 | フィールドにデフォルト値を設定する方法 |
| 動的ディスパッチ | 異種コンテナの実装方法 |

### 設計目標

1. **簡潔**：`impl` キーワードが不要
2. **柔軟**：メソッド実装は内部・外部どちらもサポート
3. **統一**：オーバーロード規則を一貫させる
4. **利便性**：デフォルト値構文がシンプル
5. **ゼロオーバーヘッド**：仮想テーブルなし、コンパイル時型収集

### Rust との比較

| 機能 | Rust | YaoXiang |
|------|------|----------|
| インターフェース宣言 | `impl Animal for Dog { ... }` | `Dog: Type = { Animal, ... }` |
| メソッド実装 | `impl` ブロック内 | 内部または外部 |
| オーバーロード | サポート外 | サポート（シグネチャが異なる場合） |
| デフォルト値 | `#[default]` が必要 | 直接 `= value` を記述 |
| 異種コンテナ | `Vec<Box<dyn Animal + 'a>>` | `List(Animal)` |
| 動的ディスパッチ | 仮想テーブル検索 | コンパイル時型収集 |

## 提案

### 1. インターフェース宣言

**コア規則**：型定義に直接インターフェース名を記述し、`impl` キーワードは不要。

```yaoxiang
# インターフェース定義
Animal: Type = {
    speak: (Self) -> String,
}

# 型がインターフェースを実装することを宣言
Dog: Type = {
    x: Int,
    Animal,  # インターフェース宣言
}
```

**コンパイラ処理**：
1. `Animal` がインターフェース型であることを識別
2. `Dog` が `Animal` が要求するすべてのメソッドを持つかチェック
3. 通過 → 実装証明を生成
4. 失敗 → コンパイルエラー

**糖衣構文の同値**：

```yaoxiang
Dog: Type = {
    x: Int,
    Animal,  # Animal のメソッドを展開したものに等価だが、來源マークを保持
}

# 同価（だが來源情報を保持）
Dog: Type = {
    x: Int,
    speak: (Self) -> String,  # Animal から
}
```

**來源マークが必要な理由**：
- 直接展開すると來源情報が失われる
- 來源マークは実装証明の生成に使用
- 実行時に証明を通じて正しいメソッドを見つける

### 2. メソッド実装

**コア規則**：メソッド実装は内部宣言と外部宣言の両方をサポート。

#### 2.1 内部宣言

```yaoxiang
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",  # メソッド実装は内部に記述
}
```

#### 2.2 外部宣言

```yaoxiang
Dog: Type = {
    x: Int,
    Animal,
}

# メソッド実装は外部に記述
Dog.speak: (Self) -> String = "Woof"
```

#### 2.3 混合宣言

```yaoxiang
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",  # 一部のメソッドは内部に
}

# 一部のメソッドは外部に
Dog.play: (Self) -> Void = { ... }
```

**コンパイラ処理**：
1. すべての定義を収集（内部と外部）
2. シグネチャでグループ化（オーバーロード）
3. オーバーライドをチェック（エラー）
4. インターフェース完全性をチェック
5. 実装証明を生成

### 3. オーバーロードとオーバーライド

**コア規則**：
- シグネチャが不同 → オーバーロード → 許可
- シグネチャが同じ → オーバーライド → エラー

#### 3.1 オーバーロード（許可）

```yaoxiang
# パラメータ型が異なるためオーバーロードを許可
Dog.speak: (Self) -> String = "Woof"
Dog.speak: (Self, volume: Int) -> String = "WOOF"
```

#### 3.2 オーバーライド（禁止）

```yaoxiang
# シグネチャが完全に同じなためオーバーライドを禁止
Dog.speak: (Self) -> String = "Woof"
Dog.speak: (Self) -> String = "Bark"  # ❌ エラー：オーバーライドは許可されていません
```

**エラーメッセージ**：

```
エラー：Dog.speak(Self) -> String が重複定義されています
  --> ファイル2:5:1
  |
5 | Dog.speak: (Self) -> String = "Bark"
  | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ 重複定義
  |
  --> ファイル1:3:1
  |
3 | Dog.speak: (Self) -> String = "Woof"
  | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ 最初の定義
```

#### 3.3 規則の統一

**内部宣言と外部宣言は同じオーバーロード/オーバーライド規則に従います**：

```yaoxiang
# 内部宣言
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

# 外部宣言（オーバーロード、許可）
Dog.speak: (Self, volume: Int) -> String = "WOOF"

# 外部宣言（オーバーライド、禁止）
Dog.speak: (Self) -> String = "Bark"  # ❌ エラー
```

### 4. デフォルト値

**コア規則**：フィールドの直後に `= value` を記述し、コンストラクタを省略。

```yaoxiang
Dog: Type = {
    x: Int = 10,  # デフォルト値
    y: Int = 20,  # デフォルト値
    Animal,
}
```

**コンパイラがコンストラクタを生成**：

```yaoxiang
# すべてのフィールドにデフォルト値がある → パラメータなしコンストラクタを生成
Dog.new: () -> Dog = { x: 10, y: 20 }

# 一部のフィールドにデフォルト値がある → 部分パラメータコンストラクタを生成
Dog.new: (x: Int) -> Dog = { x: x, y: 20 }
Dog.new: (y: Int) -> Dog = { x: 10, y: y }

# 全パラメータコンストラクタ
Dog.new: (x: Int, y: Int) -> Dog = { x: x, y: y }
```

**外部宣言のデフォルト値**：

```yaoxiang
Dog: Type = {
    x: Int,
    y: Int,
    Animal,
}

# 外部にデフォルト値を宣言
Dog.x: Int = 10
Dog.y: Int = 20
```

**内部宣言と同値。**

### 5. コンパイラ実装

#### 5.1 インターフェース記述子

```rust
// コンパイラ内部：インターフェース記述子
struct InterfaceDescriptor {
    name: String,
    methods: Vec<MethodSignature>,
}
```

#### 5.2 型定義

```rust
// コンパイラ内部：型定義
struct TypeDefinition {
    name: String,
    fields: Vec<Field>,
    interface_implementations: Vec<InterfaceImplementation>,
}

// インターフェース実装（來源情報を保持）
struct InterfaceImplementation {
    interface: InterfaceId,
    methods: HashMap<MethodId, FunctionBody>,
}
```

#### 5.3 実装証明

```rust
// コンパイラ内部：実装証明
struct ImplementationProof {
    type_id: TypeId,
    interface_id: InterfaceId,
    methods: Vec<MethodPointer>,
}
```

#### 5.4 コンパイルフロー

```
1. 型定義を解析し、インターフェース宣言を収集
2. すべてのメソッド定義を収集（内部と外部）
3. シグネチャでグループ化（オーバーロード）
4. オーバーライドをチェック（エラー）
5. インターフェース完全性をチェック
6. 実装証明を生成
7. 実行時に、値が実装証明を携带
```

### 6. 動的ディスパッチ

**コア設計**：コンパイル時型収集＋インターフェースマッチング、仮想テーブルなし。

#### 6.1 異種コンテナ

```yaoxiang
# インターフェース定義
Animal: Type = {
    speak: (Self) -> String,
}

# 型定義
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

Cat: Type = {
    y: Int,
    Animal,
    speak: (Self) -> String = "Meow",
}

# 異種コンテナ
animals: List(Animal) = [Dog.new(), Cat.new()]
animals[0].speak()  # "Woof"
animals[1].speak()  # "Meow"
```

#### 6.2 コンパイル時型収集

**コア戦略：所有権追跡、インクリメンタル構築。** インターフェースを実装したすべての型をコンパイル時にスキャンするのではなく、各 `List(Animal)` の**所有権操作点で**インクリメンタルに収集します：

```yaoxiang
// 構築点
animals: List(Animal) = [Dog.new()]       // AnimalGroup = { Dog(Dog) }

// append 点
animals.append(Cat.new())                  // コンパイラが append 箇所で Cat を見る → { Dog, Cat } に拡張
animals.append(Bird.new())                 // さらに { Dog, Cat, Bird } に拡張
```

**コンパイラ処理**（インクリメンタル）：

1. `List(I)` が初めて構築される箇所に遭遇 → 現在のコンパイルユニット内で既知のすべての構築型を対象とした初期列挙型を生成
2. 毎回の `append` / `push` / インデックス代入 → 値の型が既に列挙型にあるかチェック；なければ列挙型バリアントを拡張
3. 最終的な列挙型に対して単態化 `match` ディスパッチコードを生成
4. コンパイルユニット間：リンク時に各ユニットの列挙型バリアントセットをマージ

**自動生成された列挙型**：

```yaoxiang
# コンパイラが自動生成（ユーザーは認識しない）
AnimalGroup: Type = {
    Dog(Dog),
    Cat(Cat),
    Bird(Bird),    # ← append(Bird.new()) がインクリメンタル拡張をトリガー
}

# List(Animal) の内部は List(AnimalGroup) と同値
```

#### 6.3 インターフェースマッチチェック

**重要な洞察**：インターフェースマッチングはコンパイル時にチェックされ、動的にロードされたプラグインからの型也不例外です。

```yaoxiang
# プラグインシステム
plugin = load_plugin("bird.so")

# コンパイラチェック：plugin.create_bird() の戻り値型は Animal を実装する必要がある
bird: Animal = plugin.create_bird()  # コンパイル時チェック

# 異種コンテナに入れる —— append 点で列挙型拡張をトリガー
animals: List(Animal) = [Dog.new(), Cat.new()]
animals.append(bird)                 # コンパイラ：(1) bird が Animal を実装するか検証 (2) 列挙型を拡張して許可
```

**コンパイラ処理**：
1. `append` パラメータの戻り値型をチェック
2. その型が対象インターフェースを実装しているか検証
3. 通過 → 列挙型を拡張、許可
4. 失敗 → コンパイルエラー

#### 6.4 実行時ディスパッチ

**呼び出しフロー（コンパイル時列挙型 match、ImplementationProof は消去済み）：**

```
animals[0].speak()
  ↓
コンパイラが生成した match:
  match animals[0] {
    AnimalGroup.Dog(d) => d.speak(),
    AnimalGroup.Cat(c) => c.speak(),
    AnimalGroup.Bird(b) => b.speak(),
  }
```

**仮想テーブルとの比較**：

| | 仮想テーブル（Rust） | コンパイル時列挙型（YaoXiang） |
|---|---|---|
| 検索方式 | 仮想テーブルポインタ → メソッドポインタ | 列挙型 match → 直接呼び出し |
| 実行時オーバーヘッド | 1回の間接アドレッシング | 文字列比較/branch（CPU ブランチ予測で最適化可能） |
| コンパイル時生成 | 仮想テーブル | 列挙型 + match |
| ユーザーアノテーション | `dyn Trait + 'a` が必要 | 不要 |
| ImplementationProof | 該当なし | コンパイル時に消去、実行時に存在しない |

**YaoXiang の優位性**：
- ブランドアノテーションが不要
- コンパイル時の型安全
- ユーザー透過（`dyn Animal` を記述しなくてよい）
- ImplementationProof は純粋なコンパイル時概念で、実行時オーバーヘッドゼロ

#### 6.5 制限と範囲

**当期（単一コンパイルユニット内）：** 完全サポート。所有権追跡がすべての `append`/構築点をカバーし、列挙型をインクリメンタルに構築。

**コンパイルユニット間：** リンク時に各ユニットの列挙型バリアントセットをマージ。設計はリンク時単態化と同じ仕組みを共有（各ユニットが部分列挙型を生成、リンカーがマージ）。

**未サポート：** 実行時動的型（完全なダックタイピング）。型セットはコンパイル時に完全に既知である必要がある。

## トレードオフ

### 利点

1. **簡潔**：`impl` キーワードが不要
2. **柔軟**：メソッド実装は内部・外部どちらもサポート
3. **統一**：オーバーロード規則が一貫
4. **利便性**：デフォルト値構文がシンプル
5. **ゼロオーバーヘッド**：仮想テーブルなし、コンパイル時型収集
6. **型安全**：インターフェースマッチングはコンパイル時チェック
7. **ユーザー透過**：`dyn Animal + 'a` を記述しなくてよい

### 欠点

1. **制限**：実行時動的型をサポートしていない（完全なダックタイピング）
2. **コンパイル時オーバーヘッド**：各インターフェースに対して列挙型バリアントと match ディスパッチコードを生成する必要がある
3. **型セット**：コンパイル時に完全に既知である必要がある（単一コンパイルユニット内）

### 緩和措施

1. **プラグインシステム**：コンパイル時インターフェースマッチングチェックでサポート
2. **型セット**：所有権追跡、インクリメンタル構築——グローバルスキャンではなく、各 `append`/構築点で収集
3. **コンパイルユニット間**：リンク時に列挙型バリアントセットをマージ、リンク時単態化と同じ仕組みを共有

## 代替案

| 案 | 为什么不選択 |
|------|--------------|
| `impl` キーワード | 構文複雑度が増加 |
| 仮想テーブル（`dyn Trait`） | ブランドアノテーション（`'a`）が必要 |
| 完全ダックタイピング | 実行時オーバーヘッド、型不安全 |
| 列挙型ラッパー（手動） | ユーザー負担が大きい |

## RFC-009 との関係

**ブランドとインターフェース実装**：
- インターフェース実装は型レベルであり、ブランドには関係しない
- ブランドは借用証明レベル（RFC-009a）
- 両者は直交し、互いに影響しない

**動的ディスパッチとブランド**：
- 動的ディスパッチは実装証明を使用，品牌アノテーションは不要
- 実装証明はコンパイル時に生成され，実行時にゼロ検索
- `dyn Trait + 'a` の複雑さを回避

## インターフェース継承

インターフェースは別のインターフェースを含むことができます。**新しい構文を導入しません**——型がインターフェースを宣言するのと同じ構文位置に完全に準拠：

```yaoxiang
Animal: Type = {
    speak: (Self) -> String,
}

Pet: Type = {
    Animal,                       # Pet が Animal を継承 — 新しいキーワードなし
    name: (Self) -> String,
}

# Dog が Pet を実装する場合、Animal と Pet のすべてのメソッドを同時に満たす必要がある
Dog: Type = {
    x: Int,
    Pet,
    speak: (Self) -> String = "Woof",  # Animal から
    name: (Self) -> String = "Buddy",  # Pet から
}
```

**設計原則：** 継承は存在しますが、濫用は推奨しません。主な組合方式是複数のインターフェース宣言（`Dog: Type = { Animal, Pet, ... }`）です。型は直接、実装する必要があるすべてのインターフェースを宣言でき、継承ツリーを通じて表現する必要はありません。インターフェース継承は明確な「is-a」階層がある場合にのみ使用します。

**コンパイラ処理：** 継承チェーンを展開。`Pet` は `{ Animal のすべてのメソッド, name: ... }` に展開。`Dog` が `Pet` を宣言すると、コンパイラは `Dog` が `Animal` と `Pet` の両方のメソッドをすべて満たすことを検証します。

## デフォルトメソッド実装

インターフェースはメソッドのデフォルト実装を提供できます。実装型はデフォルト実装をオーバーライドするか継承するかを選択できます：

```yaoxiang
fmt: Type = {
    display: (Self) -> String,                      # 実装必須
    debug: (Self) -> String = Self.display(),       # ✅ 同一インターフェース内のメソッドを参照
    summary: (Self) -> String = f"<{Self.name}>",   # ❌ コンパイルエラー：Self.name は fmt に存在しない
}
```

**コア制約：インターフェースは上位の実装を想定できません。** デフォルトメソッドは同一インターフェース内で既に宣言されているメソッドのみを参照できます。具体的な型のフィールドや他のインターフェース、メソッドはデフォルトメソッドから可視ではありません——インターフェースは閉じた契約であり、実装型のポケットに手を伸ばすことはできません。この制約違反は**インターフェース定義時**にすぐにエラーになります。

**継承は下位の実装を想定できます：** インターフェース `Pet` が `Animal` を継承するとき、`Pet` のデフォルトメソッドは `Animal` が宣言したメソッドを使用できます——継承しているため、実装が保証されるためです。

```yaoxiang
Animal: Type = {
    speak: (Self) -> String,
}

Pet: Type = {
    Animal,                                              # 継承
    name: (Self) -> String,
    introduce: (Self) -> String = Self.name() + " says " + Self.speak(),  # ✅ speak は継承した Animal から
}
```

**コンパイル時動作：** 型がインターフェースを実装するとき、各メソッドに対して：
1. 型が提供 → 型のメソッドを使用
2. 型が提供せず、インターフェースがデフォルトを持つ → コンパイラがデフォルト実装を型にインライン化（ゼロ仮想テーブルオーバーヘッド）
3. 型が提供せず、インターフェースもデフォルトを持たない → コンパイルエラー

**設計原則：** デフォルトメソッドは `Copy`/`Clone` の自動導出機能に類似——コンパイラが必要に応じて自動生成し、ユーザーがオーバーライド可能。`virtual`/`override`/`super` キーワードを導入しません。

## 設計意思決定記録

| 意思決定 | 決定 | 理由 | 日付 |
|------|------|------|------|
| インターフェース宣言構文 | 型本体に直接インターフェース名を記述 | `impl` キーワードを排除、インターフェース宣言は型定義の自然な一部 | 2026-06-14 |
| 動的ディスパッチ | コンパイル時型収集＋自動列挙型生成 | 仮想テーブルなし、実行時ゼロ検索、ユーザー透過 | 2026-06-14 |
| 外部メソッド宣言 | サポート | 内部宣言と同等の柔軟性、コンパイラがファイル間収集を担当 | 2026-06-14 |
| オーバーライド | 禁止（同シグネチャでエラー） | オーバーライドは予測不能な動作を招く、オーバーロードですべての場合をカバー可能 | 2026-06-14 |
| インターフェース継承 | サポート、新構文なし | 型がインターフェースを宣言するのと同じ構文位置。深い継承ツリーより組み合わせ（多インターフェース宣言）を推奨 | 2026-07-03 |
| デフォルトメソッド実装 | サポート、Copy/Clone 自動導出に類似 | インターフェースがデフォルト本体を提供、コンパイラが必要に応じて型にインライン化；ユーザーはオーバーライド可能。virtual/override を導入しない | 2026-07-03 |
| デフォルトメソッド制約 | インターフェース定義時に検証：同一インターフェース内のメソッドのみ参照可能、上位の実装を想定不可 | インターフェースは閉じた契約。継承は下位の実装を想定できるが、インターフェースは実装型のフィールド/メソッドを想定できない | 2026-07-03 |
| 型収集戦略 | 所有権追跡、インクリメンタル構築——各 append/構築点で収集 | すべての実装者をグローバルスキャンするのではなく、所有権操作点でインクリメンタルに列挙型を拡張 | 2026-07-03 |
| ImplementationProof | 純粋なコンパイル時概念、実行時に消去 | 実行時は列挙型 match でディスパッチ、証明はコンパイル時検証のみに使用 | 2026-07-03 |
| コンパイルユニット間 | リンク時に各ユニットの列挙型バリアントをマージ | リンク時単態化と同じ仕組みを共有、各ユニットが部分列挙型を生成、リンカーがマージ | 2026-07-03 |
