# RFC 022: 霍尔逻辑静态検証サポート（仕様コメントと仕様型）

> **状態**: 審査中
> **著者**: 晨煦
> **作成日**: 2026-03-16
> **最終更新**: 2026-03-16

> **参照**:
> - [RFC-010: 統一型構文](./010-unified-type-syntax.md)
> - [RFC-011: ジェネリクスシステム設計](./011-generic-type-system.md)
> - [RFC-009: 所有権モデル](./009-ownership-model.md)

## 摘要

本稿では、YaoXiang 言語に**霍尔論理静的検証メカニズム**を導入することを提案する。開発者は `//!` または `/*! ... !*/` の形式で事前条件、事後条件、循環不変式をコメント内に記述できる。Debug Build で強制的な静的検証を実行し、検証に合格した場合のみ Release Build に進む。Release Build では仕様コメントを無視し（ゼロオーバーヘッド）、検証キャッシュをクリアする。仕様自体はタイプシステムの一部とみなされ、「仕様型」（例：`Requires(P)`、`Ensures(P)`）を形成し、ユーザーが拡張できる。この設計は、言語の簡潔さを維持しながら、重要コードに高い信頼性の保障を提供することを目的とし、YaoXiang の統一型モデルと完璧に統合される。

## 動機

### なぜこの機能/変更が必要か？

YaoXiang はすでに所有権モデル（RFC-009）と並行モデル（RFC-001）によりメモリ安全とスレッド安全を保証しているが、論理的正確性は依然としてテストに依存している。システムプログラミング、安全性が重要な分野（航空宇宙、金融、OSカーネルなど）では、論理的エラーが壊滅的な結果を引き起こす可能性がある。既存のソリューション（Rust の借用チェックなど）はこのようなエラーを検出できない。霍尔論理は数学的証明手段を提供するが、従来の形式検証ツールは独立した仕様言語と複雑な学習曲線を必要とすることが多い。

### 現在の問題点

- 論理的正確性はテストでのみ検証でき、コンパイル時に保証できない
- 重要システムコードが形式検証手段を欠いている
- 既存の形式検証ツールは学習曲線が急で、主流プログラミング言語と分離している

## 提案

### コア設計

私たちの目標は、**軽量で言語と一体化した**静的検証ソリューションを設計することである：

- **Debug Build で必須検証**：開発者は高信頼性が必要なモジュールに仕様を記述し、Debug Build で Release Build に進む前に検証合格を必須とする
- **構文が洗練されている**：`//!` コメントを使用するため、新しいキーワードを導入せず、エディタで構文強調できる
- **タイプシステムとの融合**：仕様は型の一部となり、型チェックに参加でき、ユーザー定義の仕様型をサポート
- **証明可能かつテスト可能**：静的証明の他に、実行時アサーションに降格でき、段階的な導入が容易

### 1. 仕様コメント構文

関数本体またはループ本体の先頭に、`//!`（単一行）または `/*! ... !*/`（複数行）を使用して仕様を記述する。

#### 1.1 統一仕様構文

仕様は YaoXiang 統一の `name: Type = expression` 構文モデルを採用しており、タイプシステムと完全に融合している：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0
    //! ensures: GreaterOrEqual(result, arr[0..n])
    //! ensures: ExistsMax(result, arr[0..n])
    // 実装...
}
```

- 仕様本质は**型宣言**であり、右辺はブール式
- 左辺は仕様型インスタンス（型パラメータ付き）
- 特別な変数 `result` で戻り値を表すことができる

#### 1.2 ループ仕様

```yaoxiang
while i < n {
    /*! invariant: Bounds[i, n] = 0 <= i <= n
                  && SumInvariant[s, arr[0..i]] !*/
    s = s + arr[i]
    i = i + 1
}
```

#### 1.3 仕様式

仕様の右辺のブール式は YaoXiang 式構文を使用し、以下をサポート：
- 算術演算、比較演算、論理演算
- 量詞：`forall i in 0..n: P(i)`、`exists i in 0..n: P(i)` — 言語組み込みの論理構成
- 関数呼び出し（純粋関数でなければならない）

### 2. 仕様タイプシステム

仕様型は本質的に YaoXiang の通常の型であり、統一構文モデルと完全に一致している。

#### 2.1 組み込み仕様型

コンパイラは次の一般的に使用される仕様型を組み込んでいる（仕様内で直接使用可能）：

```yaoxiang
// 組み込み仕様型定義
NonEmpty: (T: Type) -> Type = { len: T; len > 0 }
Positive: Type = { x: Int; x > 0 }
GreaterOrEqual: (T: Type) -> Type = { result: T, arr: Array(T); result >= arr[0] && forall i in 1..arr.len: result >= arr[i] }
Bounds: (T: Type) -> Type = { i: T, n: T; 0 <= i && i <= n }
SumInvariant: (T: Type) -> Type = { s: T, arr: Array(T); s == sum(arr[0..i]) }

// 量詞構成（言語組み込み、関数ではない）
forall: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
exists: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
```

#### 2.2 ユーザー定義仕様型

通常の型定義と完全に一致し、ユーザーは仕様型を定義できる：

```yaoxiang
// 正整数仕様を定義
Positive: Type = { x: Int; x > 0 }

// ソート済み配列仕様を定義
Sorted: (T: Ord) -> Type = {
    arr: Array(T);
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

// 最大値仕様を定義
ExistsMax: (T: Ord) -> Type = {
    result: T, arr: Array(T);
    exists i in 0..arr.len: result == arr[i]
    && forall j in 0..arr.len: result >= arr[j]
}
```

ユーザー定義仕様の使用例：

```yaoxiang
sqrt: (x: Positive) -> Float = {
    //! ensures: SquareRootResult(result, x) = result * result <= x && (result+1)*(result+1) > x
    // 実装...
}

binary_search: (T: Ord) -> ((arr: Sorted(Array(T)), key: T) -> Option(Index)) = {
    //! ensures: SearchResult(result, arr, key)
    // 実装...
}
```

仕様型は他の型一样に、ジェネリックパラメータ、型制約をサポートし、型推論に参加する。

### 3. コンパイルモード

| モード | 動作 | オプション |
|--------|------|------------|
| **Debug Build** | 仕様を解析し、検証条件を生成して SMT ソルバを呼び出して証明；検証に合格した場合のみ Release Build に進む | `yaoxiangc --debug source.yx` |
| **Release Build** | すべての `//!` コメントを無視し、コード生成しない；すべての検証キャッシュをクリア；積極的な最適化を有効化 | `yaoxiangc --release source.yx` |
| **実行時チェック** | 仕様を実行時アサートに変換し、違反時に panic | `yaoxiangc --enable-runtime-checks source.yx` |

検証モードで証明に失敗した場合、コンパイラはエラーを報告し、可能な反例（入力値など）を提供する。

### 4. 検証メカニズム

コンパイラは仕様を検証条件（Verification Conditions）に変換し、統合された SMT ソルバ（Z3 など）に送信する。検証プロセスは大まかに以下のとおり：

1. 関数の `requires` と `ensures`、ループの `invariant` を収集
2. 各ループに対してループ不変式証明義務を生成：ループ突入前に成立すること、各反復後に維持されること、ループ終了後に事後条件を蕴含すること
3. 関数本体を一階述語論理式に変換し、仕様と組み合わせて検証条件を形成
4. SMT ソルバを呼び出して充足可能性をチェック

ソルバが `unsat`（充足不能）を返した場合、仕様は成立する。そうでなければ反例を報告する。

### 5. テストとの連携

実行時チェックモードは仕様をアサートに変換し、テストに使用できる。仕様カバレッジツールと組み合わせることで、テストが仕様をどの程度カバーしているかを評価できる。将来的には、テストから自動的に候補仕様を推論する仕様マイニングツールの導入も検討できる。

### 6. エディタサポート

`//!` と `/*! ... !*/` はエディタに認識され、特別なコメントとして異なる色（紫色など）が割り当てられ、通常のコメントと区別される。言語サーバーは仕様のホバー表示、補完、検証エラー報告を提供できる。

## 詳細設計

### 構文の変更

| 変更前 | 変更後 |
|--------|--------|
| 仕様コメント構文なし | `//!` と `/*! ... !*/` 仕様コメントを許可 |

### 7.1 構文拡張

既存の構文（RFC-010）を基础上、関数本体とループ本体の先頭に0行以上の `//!` または `/*! ... !*/` コメント的出现を許可する。仕様構文は統一型構文と一致する：

```
spec_comment     ::= ('//!' spec_line) | ('/*!' spec_block '!*/')
spec_line        ::= spec_name ':' type_expr '=' expr
spec_name        ::= 'requires' | 'ensures' | 'invariant'
spec_block       ::= (spec_name ':' type_expr '=' expr ';')*
```

- 仕様本质は型宣言：`仕様名: 仕様型 = ブール式`
- `type_expr` は仕様型式であり、型パラメータを持つことができる
- `expr` は YaoXiang 式構文を使用し、量詞をサポート

### 7.2 型チェック

検証モードでは、コンパイラは仕様コメントを対応する仕様型インスタンスに変換し、関数またはループのメタデータに記録する。

### 7.3 検証条件生成

 weakest precondition または strongest postcondition 計算を採用 し、循环不変式を組み合わせ て、一階述語論理式を生成する。生成された VC は SMT-LIB 形式を使用し、外部ソルバを呼び出す。

### 7.4 エラー報告

証明に失敗した場合、ソルバはモデル（反例）を提供する可能性がある。コンパイラはこれらの反例を可読な形式（具体的な入力値など）に変換し、ユーザーのデバッグを助ける。

### 7.5 実行時チェック

`--enable-runtime-checks` モードでは、コンパイラは仕様を `assert` 文に変換する：
- `requires`：関数入口に `assert(cond)` を挿入
- `ensures`：関数のすべての復帰点の前に `assert(cond)` を挿入（`result` を実際の戻り値に置き換え）
- `invariant`：ループ本体先頭に `assert(cond)` を挿入

### 7.6 既存設計との統合

- **所有権モデル**：仕様内の式は所有権規則に従い、 읽기専用（純粋関数）でなければならず、副作用を避ける
- **ジェネリクスシステム**：仕様型はジェネリックパラメータ（例：`Requires(P)`）をサポートし、ジェネリック関数/型と組み合わせ可能
- **依存型**：仕様内の値依存型（配列の長さ `n` など）が自然に利用可能

### タイプシステムへの影響

- 仕様型は YaoXiang の通常の型であり、統一構文モデルと一致する
- コンパイラは一般的な仕様型（`Positive`、`NonEmpty`、`GreaterOrEqual` など）を組み込んでいる
- ユーザーは通常の型定義でカスタム仕様型を定義できる
- 仕様型はジェネリックパラメータを持ち、型制約をサポートする

### 実行時動作

- **Debug Build**：SMT ソルバを呼び出して静的検証を実行し、コンパイル時間が増加する；検証成功后、検証結果をキャッシュ
- **Release Build**：仕様コメントは無視され、ゼロ実行時オーバーヘッド；すべての検証キャッシュをクリア；Span キャッシュクリアなどの積極的な最適化を有効化
- **実行時チェックモード**：assert 文を生成し、実行時に違反を検出

### コンパイラ変更点

- パーサー：仕様コメント構文を認識
- セマンティック分析：仕様を収集し、仕様型に変換
- 検証バックエンド：検証条件を生成し、SMT ソルバを呼び出す
- コード生成：実行時チェックモードをサポート

### 後方互換性

- ✅ 完全な後方互換性
- 通常のコンパイルでは仕様コメントが無視され、既存コードに影響しない
- Release Build では仕様が無視され、追加オーバーヘッドなし

## トレードオフ

### 优点

- **Debug Build 検証**：Debug Build で強制検証し、論理的正確性を保証
- **構文が洗練されている**：純粋なコメントであり、新しいキーワードを導入せず、エディタに優しい
- **タイプシステムとの融合**：仕様即ち型で、拡張可能
- **段階的な導入**：実行時チェックから静的検証に徐々に移行可能
- **信頼性の向上**：テストでは検出困難な論理的エラーを捕捉可能

### 缺点

- **コンパイル時間**：検証モードによりコンパイル時間が大幅に増加する可能性
- **学習曲線**：効果的な仕様と量詞の記述方法を学習する必要がある
- **SMT ソルバの限界**：一部の複雑な性質を自動証明できない場合がある

## 代替案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 新規キーワード（`requires` など） | 構文が直感的 | 新規キーワードの導入により簡潔さが損なわれる |
| 独立した仕様ファイル（CVL など） | 仕様とコードが分離 | ファイル数が増加し、同期が困難 |
| 実行時アサートのみ | 実装が単純 | 静的保証がない |
| **本提案（コメント+仕様型）** | 簡潔さと機能のバランス | エディタサポートが必要 |

## 実装戦略

### 段階的区分

| 段階 | 内容 |
|------|------|
| **段階 1：基盤サポート** | パーサーを拡張して `//!` と `/*! ... !*/` コメントを認識し、AST ノードに接続；検証モードで仕様を収集し、単純な検証条件（算術比較のみ）を生成；Z3 ソルバを統合 |
| **段階 2：量詞サポート** | 量詞式をサポートし、SMT-LIB の `forall`/`exists` に変換；仕様の IDE 強調表示とホバー表示を提供 |
| **段階 3：最適化とツールチェーン** | 増分検証、検証済みモジュールのキャッシュ；仕様カバレッジレポート；仕様マイニングツール（テストから候補仕様を生成） |

### 依存関係

- RFC-009: 所有権モデル - 仕様式には純粋関数セマンティクスが必要
- RFC-010: 統一型構文 - 仕様タイプシステムは型システム基础上
- RFC-011: ジェネリクスシステム設計 - 仕様型はジェネリックパラメータをサポート

### リスク

1. **SMT ソルバ統合の複雑性**：Z3 などのソルバとの統合に技術的課題が生じる可能性がある
   - 緩和：成熟した Rust Z3 バインディングを使用し、サポート対象の式タイプを徐々に拡張

2. **検証失敗時のデバッグ困難**：SMT ソルバが仕様を証明できない場合、ユーザーが原因を理解しにくい
   - 緩和：明確なエラーメッセージと反例の説明を提供

3. **パフォーマンスオーバーヘッド**：検証モードによりコンパイル時間が大幅に増加する可能性がある
   - 緩和：増分検証とキャッシュメカニズムを実装

1. RFC を `docs/design/accepted/` ディレクトリに移動
2. ファイル名を説明的な名前（例：`hoare-logic-static-verification.md`）に更新
3. 状態を「正式」に更新
4. 状態を「承認済み」に更新し、承認日付を追加

### 拒否後の操作

1. `docs/design/rfc/` ディレクトリに残す
2. ファイル上部に拒否理由と日付を追加
3. 状態を「拒否済み」に更新
