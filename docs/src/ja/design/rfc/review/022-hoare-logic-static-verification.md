# RFC 022: ホーア論理静的検証サポート（仕様コメントと仕様タイプ）

> **参考**:
> - [RFC-010: 統一タイプ構文](./010-unified-type-syntax.md)
> - [RFC-011: ジェネリクスシステム設計](./011-generic-type-system.md)
> - [RFC-009: 所有権モデル](./009-ownership-model.md)

## 概要

本文は YaoXiang 言語に**ホーア論理静的検証メカニズム**を導入することを提案する。開発者は `//!` または `/*! ... !*/` の形式でコメント内に事前条件、事後条件、ループ不変量を記述できる。Debug Build では検証を強制し、Release Build に進む前に検証を通過する必要がある。Release Build では仕様コメントを無視し（ゼロオーバーヘッド）、検証キャッシュをクリアする。仕様自体はタイプシステムの一部とみなされ、「仕様タイプ」（如 `Requires(P)`、`Ensures(P)`）を形成し、ユーザーが拡張できる。この設計は、言語の簡潔さを維持しながら、重要なコードに高い信頼性の保障を提供することを目的とし、YaoXiang の統一タイUEDモデルと完璧に統合される。

## 動機

### なぜこの機能/変更が必要なのか？

YaoXiang はすでに所有権モデル（RFC-009）と並行モデル（RFC-001）によりメモリ安全とスレッド安全を保証しているが、論理的な正しさは依然としてテストに依存している。システムプログラミング、セキュリティ重要な分野（航空宇宙、金融、OSカーネルなど）では、論理エラーが壊滅的な結果を引き起こす可能性がある。既存のソリューション（Rust の借用チェックなど）はこの種のエラーを検出できない。ホーア論理は数学的証明手段を提供するが、従来の形式検証ツールは独立した仕様言語と複雑な学習曲線を必要とすることが多い。

###  現在の問題

- 論理的正しさはテストでのみ検証でき、コンパイル時に保証できない
- 重要なシステムコードが形式検証手段を欠いている
- 既存の形式検証ツールは学習曲線が急で、主流プログラミング言語と乖離している

## 提案

### コア設計

我々の目標は、**軽量で言語に統合された**静的検証ソリューションを設計することである：

- **Debug Build で検証必須**：開発者は高い信頼性を必要とするモジュールに仕様を記述し、Debug Build で Release Build に進む前に検証を強制通過させる
- **文法的に優美**：`//!` コメントを使用し、新しいキーワードを導入せず、エディタで構文強調可能被
- **タイプシステムとの融合**：仕様はタイプの一部となり、タイプ検査に参加でき、ユーザーが仕様タイプを定義可能被
- **証明可能かつテスト可能**：静的証明也可能だが、実行時アサーションに降格も可能で、段階的な採用が容易

### 1. 仕様コメント構文

関数本体またはループ本体の先頭に、`//!`（単一行）または `/*! ... !*/`（複数行）を使用して仕様を記述する。

#### 1.1 統一仕様構文

仕様は YaoXiang 統一の `name: Type = expression` 構文モデルを採用し、タイプシステムと完全に統合される：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0
    //! ensures: GreaterOrEqual(result, arr[0..n])
    //! ensures: ExistsMax(result, arr[0..n])
    // 実装...
}
```

- 仕様本質は**タイプ宣言**であり、右辺はブール式である
- 左辺は仕様タイプのインスタンス（型パラメータ付き可能被）
- 特殊な変数 `result` を使用して戻り値を表す可能被

#### 1.2 ループ仕様

```yaoxiang
while i < n {
    /*! invariant: Bounds[i, n] = 0 <= i <= n
                  && SumInvariant[s, arr[0..i]] !*/
    s = s + arr[i]
    i = i + 1
}
```

#### 1.3 仕様式

仕様右辺のブール式は YaoXiang 式構文を使用し、以下をサポート：
- 算術演算、比較演算、論理演算
- 量記号：`forall i in 0..n: P(i)`、`exists i in 0..n: P(i)` — 言語組み込みの論理構造
- 関数呼び出し（純粋関数でなければならない）

### 2. 仕様タイプシステム

仕様タイプは本質的に YaoXiang の通常のタイプであり、統一構文モデルと完全に一致する。

#### 2.1 組み込み仕様タイプ

コンパイラは以下よく使用される仕様タイプを組み込んでいる（仕様内で直接使用可能被）：

```yaoxiang
// 組み込み仕様タイプ定義
NonEmpty: (T: Type) -> Type = { len: T; len > 0 }
Positive: Type = { x: Int; x > 0 }
GreaterOrEqual: (T: Type) -> Type = { result: T, arr: Array(T); result >= arr[0] && forall i in 1..arr.len: result >= arr[i] }
Bounds: (T: Type) -> Type = { i: T, n: T; 0 <= i && i <= n }
SumInvariant: (T: Type) -> Type = { s: T, arr: Array(T); s == sum(arr[0..i]) }

// 量記号構築（言語組み込み、関数ではない）
forall: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
exists: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
```

#### 2.2 ユーザー定義仕様タイプ

通常のタイプ定義と完全に一致し、ユーザーが仕様タイプを定義可能被：

```yaoxiang
// 正整数仕様の定義
Positive: Type = { x: Int; x > 0 }

// ソート済み配列仕様の定義
Sorted: (T: Ord) -> Type = {
    arr: Array(T);
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

// 最大値仕様の定義
ExistsMax: (T: Ord) -> Type = {
    result: T, arr: Array(T);
    exists i in 0..arr.len: result == arr[i]
    && forall j in 0..arr.len: result >= arr[j]
}
```

ユーザー定義仕様の使用：

```yaoxiang
sqrt: (x: Positive) -> Float = {
    //! ensures: SquareRootResult(result, x) = result * result <= x && (result+1)*(result+1) > x
    // 実装...
}

binary_search: (T: Ord) -> ((arr: Sorted(Array(T)), key: T) -> Option(Index)) = {
    //! ensures: SearchResult(result, arr, key)
    // 実装...
}
```

仕様タイプは他のタイプ一样に、ジェネリックパラメータ、型制約をサポートし、型推論に参加可能被。

### 3. コンパイルモード

| モード | 動作 | オプション |
|------|------|------|
| **Debug Build** | 仕様を解析し、検証条件を生成し、SMTソルバーを呼び出して証明；検証通過後にのみ Release Build 可能 | `yaoxiangc --debug source.yx` |
| **Release Build** | すべての `//!` コメントを無視し、コードを生成しない；すべての検証キャッシュをクリア；アグレッシブ最適化を有効化 | `yaoxiangc --release source.yx` |
| **実行時チェック** | 仕様を実行時アサーションに変換し、違反時に panic | `yaoxiangc --enable-runtime-checks source.yx` |

検証モードでは、証明に失敗した場合、コンパイラはエラーを報告し、可能な反例（入力値など）を提供する。

### 4. 検証メカニズム

コンパイラは仕様を検証条件（Verification Conditions）に変換し、統合された SMTソルバー（例：Z3）に送信する。検証プロセスは大まかに 다음과 같다：

1. 関数の `requires` と `ensures`、ループの `invariant` を収集
2. 各ループに対してループ不変量証明義務を生成：ループ進入前に成立、各反復後に維持、ループ終了後に事後条件を意味
3. 関数本体を論理式に変換し、仕様と組み合わせて検証条件を形成
4. SMTソルバーを呼び出して充足可能性をチェック

ソルバーが `unsat`（充足不能）を返した場合、仕様は成立。否则は反例を報告する。

### 5. テストとの組み合わせ

実行時チェクモードは仕様をアサーションに変換してテストに使用可能被。仕様カバレッジツールと組み合わせることで、テストの仕様カバレッジ度を評価可能被。将来では、テストから候補仕様を自動的に推論する仕様マイニングツールの導入も検討可能被。

### 6. エディタサポート

`//!` と `/*! ... !*/` はエディタに認識されて特別なコメントとして異なる色（例：紫色）が与えられ、通常のコメントと区別可能被。言語サーバーは仕様のホバー提示、補完、検証エラー報告を提供可能被。

## 詳細設計

### 構文変更

| 以前 | 以降 |
|------|------|
| 仕様コメント構文なし | `//!` と `/*! ... !*/` 仕様コメントを許可 |

### 7.1 構文拡張

既存の構文（RFC-010）を基础上、関数本体とループ本体の先頭に0個以上の `//!` または `/*! ... !*/` コメント的出现を許可する。仕様構文は統一タイプ構文と一致する：

```
spec_comment     ::= ('//!' spec_line) | ('/*!' spec_block '!*/')
spec_line        ::= spec_name ':' type_expr '=' expr
spec_name        ::= 'requires' | 'ensures' | 'invariant'
spec_block       ::= (spec_name ':' type_expr '=' expr ';')*
```

- 仕様本質はタイプ宣言である：`仕様名: 仕様タイプ = ブール式`
- `type_expr` は型パラメータ付き可能被の仕様タイプ式
- `expr` は YaoXiang 式構文を使用し、量記号をサポート

### 7.2 タイプ検査

検証モードでは、コンパイラは仕様コメントを対応する仕様タイプインスタンスに変換し、関数またはループのメタデータに記録する。

### 7.3 検証条件生成

 weakest precondition または strongest postcondition 计算を採用し、ループ不変量と組み合わせて一階述語論理式を生成する。生成された VC は SMT-LIB 形式を使用し、外部ソルバーを呼び出す。

### 7.4 エラー報告

証明に失敗した場合、ソルバーはモデル（反例）を提供可能被。コンパイラはこれらの反例を読み取り可能な形式に変換する必要がある。例えば具体的な入力値など、ユーザーのデバッグを支援する。

### 7.5 実行時チェック

`--enable-runtime-checks` モードでは、コンパイラは仕様を `assert` 文に変換する：
- `requires`：関数入口に `assert(cond)` を挿入
- `ensures`：関数のすべての復帰点の前に `assert(cond)` を挿入（`result` は実際の戻り値に置き換え）
- `invariant`：ループ本体先に `assert(cond)` を挿入

### 7.6 既存設計との統合

- **所有権モデル**：仕様内の式は所有権ルールに従い、参照のみ可能（純粋関数）、副作用を避ける
- **ジェネリクスシステム**：仕様タイプはジェネリックパラメータ（例：`Requires(P)`）をサポートし、ジェネリック関数/タイプと組み合わせ可能被
- **依存タイプ**：仕様内の値依存タイプ（配列長 `n` など）が自然に利用可能

### タイプシステムへの影響

- 仕様タイプは YaoXiang の通常のタイプであり、統一構文モデルと一致する
- コンパイラはよく使用される仕様タイプ（`Positive`、`NonEmpty`、`GreaterOrEqual` など）を組み込んでいる
- ユーザーは通常のタイプ定義によりカスタム仕様タイプを定義可能被
- 仕様タイプはジェネリックパラメータを持ち、型制約をサポート

### 実行時動作

- **Debug Build**：SMTソルバーを呼び出して静的検証を実行し、コンパイル時間が増加；検証成功後に検証結果をキャッシュ
- **Release Build**：仕様コメントが無視され、ゼロ実行時オーバーヘッド；すべての検証キャッシュをクリア；Span キャッシュクリアなどのアグレッシブ最適化を有効化
- **実行時チェクモード**：assert 文を生成し、実行時に違反を検出

### コンパイラ変更

- パーサー：仕様コメント構文を認識
- 意味解析：仕様を収集し、仕様タイプに変換
- 検証バックエンド：検証条件を生成し、SMTソルバーを呼び出す
- コード生成：実行時チェクモードをサポート

### 後方互換性

- ✅ 完全な後方互換性
- 仕様コメントは通常コンパイルで無視され、既存のコードに影響しない
- Release Build では仕様が無視され、追加的开销なし

## トレードオフ

### メリット

- **Debug Build 検証**：Debug Build で検証を強制し、論理的正しさを確保
- **文法的に優美**：純粋なコメント、新しいキーワードなし、エディタに優しい
- **タイプシステムとの融合**：仕様即タイプ、拡張可能被
- **段階的採用**：実行時チェクから静的検証への段階的移行が可能
- **信頼性の向上**：テストでは発見困難な論理エラーを検出可能被

### デメリット

- **コンパイル時間**：検証モードはコンパイル時間を大幅に増加시킬 가능성이あ
- **学習曲線**：効果的な仕様と量記号の記述方法を学習する必要がある
- **SMTソルバーの限界**：某些複雑な性質を自動的に証明できない可能性がある

## 代替案

| 方案 | メリット | デメリット |
|------|------|------|
| 新キーワード（例：`requires`） | 構文が直感的 | 新しいキーワードを導入し、簡潔さを損なう |
| 独立した仕様ファイル（CVLなど） | 仕様とコードが分離 | ファイル数が増加し、同期が難しい |
| 実行時アサーションのみ | 実装が簡単 | 静的保証がない |
| **本方案（コメント+仕様タイプ）** | 簡潔さと機能のバランス | エディタサポートが必要 |

## 実装戦略

### 段階的区分

| 段階 | 内容 |
|------|------|
| **段階 1：基盤サポート** | パーサーを拡張し、`//!` と `/*! ... !*/` コメントを認識して AST ノードに添付；検証モードで仕様を収集し、単純な検証条件（算術比較のみ）を生成；Z3ソルバーを統合 |
| **段階 2：量記号サポート** | 量記号式をサポートし、SMT-LIB の `forall`/`exists` に変換；仕様の IDE 強調表示とホバー提示を提供 |
| **段階 3：最適化とツールチェーン** | 増分検証、キャッシュ済みモジュールの検証；仕様カバレッジレポート；仕様マイニングツール（テストから候補仕様を生成） |

### 依存関係

- RFC-009: 所有権モデル - 仕様式は純粋関数セマンティクスが必要
- RFC-010: 統一タイプ構文 - 仕様タイプシステムはタイプシステムに基づく
- RFC-011: ジェネリクスシステム設計 - 仕様タイプはジェネリックパラメータをサポート

### リスク

1. **SMTソルバー統合の複雑性**：Z3 などのソルバーの統合は技術的課題に直面する可能性がある
   - 軽減：成熟した Rust Z3 バインディングを使用し、サポートする式类型を徐々に拡張

2. **検証失敗のデバッグ困難**：SMTソルバーが仕様を証明できない場合、ユーザーは原因を理解しにくい
   - 軽減：明確なエラーメッセージと反例の説明を提供

3. **パフォーマンスオーバーヘッド**：検証モードはコンパイル時間を大幅に増加시킬 가능성이あ
   - 軽減：増分検証とキャッシュメカニズムを実装
