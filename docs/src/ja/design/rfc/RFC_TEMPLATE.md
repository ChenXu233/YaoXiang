## ライフサイクルと帰結

RFC には以下の状態遷移があります：

```
┌─────────────┐
│   ドラフト   │  ← 著者による作成
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  レビュー中  │  ← コミュニティによる議論
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│   採用済み   │    │   拒否済み   │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│   accepted/ │    │    rfc/     │
│ (正式設計)  │    │ (元の場所)  │
└─────────────┘    └─────────────┘
```

### 状態の説明

| 状態 | 場所 | 説明 |
|------|------|------|
| **ドラフト** | `docs/design/rfc/` | 著者の草稿、レビュー提出待ち |
| **レビュー中** | `docs/design/rfc/` | コミュニティの議論とフィードバックを公開中 |
| **採用済み** | `docs/design/accepted/` | 正式な設計文書となり、実装段階へ進む |
| **拒否済み** | `docs/design/rfc/` | RFC ディレクトリに保持し、状態を更新 |

### 採用後の操作

1. RFC を `docs/design/accepted/` ディレクトリに移動
2. ファイル名を説明的な名前に更新（例：`enhanced-pattern-matching.md`）
3. 状態を「正式」に更新
4. 状態を「採用済み」に更新し、採用日を追加

### 拒否後の操作

1. `docs/design/rfc/` ディレクトリに保持
2. ファイルの先頭に拒否理由と日付を追加
3. 状態を「拒否済み」に更新

### 議論確定後の操作

未解決問題について合意が形成された場合：

1. **付録Aを更新**: 議論トピックの下に「決議」を記入
2. **本文を更新**: 決定を本文に同期
3. **決定を記録**: 「付録B：設計決定の記録」に追加
4. **問題をマーク**: 「未解決問題」リストの `[x]` をチェック

---
