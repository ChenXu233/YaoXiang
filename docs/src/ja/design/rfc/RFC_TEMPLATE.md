## ライフサイクルと帰属

RFC には以下のステータス遷移があります：

```
┌─────────────┐
│  ドラフト    │  ← 著者が作成
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ レビュー中  │  ← コミュニティで討論
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  承認済み   │    │  拒否済み   │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  accepted/  │    │    rfc/     │
│ (正式設計)  │    │ (現状維持)  │
└─────────────┘    └─────────────┘
```

### ステータス説明

| ステータス | 配置場所 | 説明 |
|------|------|------|
| **ドラフト** | `docs/design/rfc/` | 著者の草稿、レビュー待ち |
| **レビュー中** | `docs/design/rfc/` | コミュニティの討論とフィードバックを公開中 |
| **承認済み** | `docs/design/accepted/` | 正式な設計ドキュメントとなり、実装段階へ |
| **拒否済み** | `docs/design/rfc/` | RFC ディレクトリに残し、ステータスを更新 |

### 承認後の操作

1. RFC を `docs/design/accepted/` ディレクトリに移動
2. ファイル名を説明的な名称に更新（例：`enhanced-pattern-matching.md`）
3. ステータスを "正式" に更新
4. ステータスを "承認済み" に更新し、承認日を追加

### 拒否後の操作

1. `docs/design/rfc/` ディレクトリに保持
2. ファイル上部に拒否理由と日付を追加
3. ステータスを "拒否済み" に更新

### 討論確定後の操作

未解決の問題について合意が形成された場合：

1. **付録Aを更新**: 討論テーマの下に「決議」を記入
2. **本文を更新**: 決定をドキュメント本文に同期
3. **決定を記録**: 「付録B：設計決定記録」に追加
4. **問題をマーク**: 「未解決の問題」リストで `[x]` にチェック

---

> **注**: RFC 番号は討論段階でのみ使用されます。承認後は番号を削除し、説明的なファイル名を使用してください。
