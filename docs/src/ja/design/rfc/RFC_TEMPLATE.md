## ライフサイクルと去向

RFC には以下のステータス遷移があります：

```
┌─────────────┐
│   草案      │  ← 作成者が作成
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  レビュー中 │  ← コミュニティ議論
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  承認済み   │    │  却下       │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│   accepted/ │    │    rfc/     │
│ (正式設計)  │    │ (元の位置を保持) │
└─────────────┘    └─────────────┘
```

### ステータス说明

| ステータス | 位置 | 説明 |
|------|------|------|
| **草案** | `docs/design/rfc/` | 作成者の下書き、レビューの提出待ち |
| **レビュー中** | `docs/design/rfc/` | コミュニティ議論とフィードバックを開始 |
| **承認済み** | `docs/design/accepted/` | 正式なデザインドキュメントになり、実装段階へ |
| **却下** | `docs/design/rfc/` | RFC ディレクトリに保持、ステータスを更新 |

### 承認後の操作

1. RFC を `docs/design/accepted/` ディレクトリに移動
2. ファイル名を説明的な名前に更新（例：`enhanced-pattern-matching.md`）
3. ステータスを「正式」に更新
4. ステータスに承認日を追加して「承認済み」に更新

### 却下後の操作

1. `docs/design/rfc/` ディレクトリに保持
2. ファイル冒頭に却下理由と日付を追加
3. ステータスを「却下」に更新

### 議論確定後の操作

が開放問題について合意が形成された場合：

1. **付録A を更新**: 議論テーマ下に「決議」を記入
2. **本文を更新**: 決定をドキュメント本文に同期
3. **意思決定を記録**:「付録B：デザイン意思決定記録」に追加
4. **問題をマーク**:「開放問題」リストで `[x]` をチェック
