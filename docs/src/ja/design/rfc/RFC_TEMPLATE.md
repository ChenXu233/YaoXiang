## ライフサイクルと行き先

RFC には以下のステータス遷移があります：

```
┌─────────────┐
│   草案      │  ← 作者が作成
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  審査中     │  ← コミュニティ議論
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  承認済み   │    │  拒否済み   │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│   accepted/ │    │    rfc/     │
│ (正式設計)  │    │ (元の位置に保持) │
└─────────────┘    └─────────────┘
```

### ステータス説明

| ステータス | 場所 | 説明 |
|------------|------|------|
| **草案** | `docs/design/rfc/` | 作者の草稿、検索待ち |
| **審査中** | `docs/design/rfc/` | コミュニティ議論とフィードバックを募集中 |
| **承認済み** | `docs/design/accepted/` | 正式設計ドキュメントになり、実装段階へ |
| **拒否済み** | `docs/design/rfc/` | RFC ディレクトリに保持、ステータスを更新 |

### 承認後の操作

1. RFC を `docs/design/accepted/` ディレクトリに移動
2. ファイル名を説明的な名前に更新（例：`enhanced-pattern-matching.md`）
3. ステータスを "正式" に更新
4. ステータスを "承認済み" に更新し、承認日を記載

### 拒否後の操作

1. `docs/design/rfc/` ディレクトリに保持
2. ファイル冒頭に拒否理由と日付を追加
3. ステータスを "拒否済み" に更新

### 議論確定後の操作

特定の開放問題について合意に達した場合：

1. **付録A を更新**: 議論テーマの下に「決議」を記入
2. **本文を更新**: 決定内容をドキュメントの本文に同期
3. **決定を記録**: 「付録B：設計決定記録」に追加
4. **問題をマーク**: 「開放問題」リストで `[x]` にチェック
