## ライフサイクルと帰結

RFC には以下のようなステータス遷移があります：

```
┌─────────────┐
│   草案      │  ← 作成者が作成
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  レビュー中 │  ← コミュニティでの議論
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  承認済み   │    │  拒否済み   │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│   accepted/ │    │    rfc/     │
│ (正式設計)  │    │ (元の位置に│
└─────────────┘    │  維持)      │
                   └─────────────┘
```

### ステータス説明

| ステータス | 場所 | 説明 |
|------|------|------|
| **草案** | `docs/design/rfc/` | 著者の下書き、レビュー提出待ち |
| **レビュー中** | `docs/design/rfc/` | コミュニティ議論とフィードバック募集中 |
| **承認済み** | `docs/design/accepted/` | 正式設計ドキュメントとなり、実装段階へ |
| **拒否済み** | `docs/design/rfc/` | RFC ディレクトリに残し、ステータスを更新 |

### 承認後の手続き

1. RFC を `docs/design/accepted/` ディレクトリに移動する
2. ファイル名を説明的な名前に更新する（例：`enhanced-pattern-matching.md`）
3. ステータスを「正式」に更新する
4. ステータスを「承認済み」に更新し、承認日を追加する

### 拒否後の手続き

1. `docs/design/rfc/` ディレクトリに残す
2. ファイル冒頭に拒否理由と日付を追加する
3. ステータスを「拒否済み」に更新する

### 議論が決着した後の手続き

開放的な問題が合意に達した場合：

1. **付録Aを更新**: 議論テーマの下に「決議」を記入する
2. **本文を更新**: 決定をドキュメントの本文に同期する
3. **意思決定を記録**: 「付録B：設計意思決定記録」に追加する
4. **問題をマーク**: 「開放的な問題」リストで `[x]` をチェックする

---

> **注**: RFC 番号は議論段階でのみ使用します。承認後は番号を削除し、説明的なファイル名を使用します。
