## ライフサイクルと歸趨

RFC には次のようなステータス遷移があります：

```
┌─────────────┐
│   草案      │  ← 作成者が作成
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  レビュー中  │  ← コミュニティの議論
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  承認済み    │    │  拒否済み    │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│   accepted/ │    │    rfc/     │
│ (正式な設計) │    │ (元の場所に │
└─────────────┘    └─────────────┘
```

### ステータスの説明

| ステータス | 場所 | 説明 |
|------|------|------|
| **草案** | `docs/design/rfc/` | 作成者の下書き、レビューの提出待ち |
| **レビュー中** | `docs/design/rfc/` | コミュニティの議論とフィードバックを開始 |
| **承認済み** | `docs/design/accepted/` | 正式な設計ドキュメントとなり、実装段階に移行 |
| **拒否済み** | `docs/design/rfc/` | RFC ディレクトリに保持、ステータスを更新 |

### 承認後の操作

1. RFC を `docs/design/accepted/` ディレクトリに移動します
2. ファイル名を説明的な名前に更新します（例：`enhanced-pattern-matching.md`）
3. ステータスを「正式」に更新します
4. ステータスに承認日を追加します

### 拒否後の操作

1. `docs/design/rfc/` ディレクトリに保持します
2. ファイルの上部に拒否理由と日付を追加します
3. ステータスを「拒否済み」に更新します

### 議論が確定した後の操作

未解決の問題について合意に達した場合は：

1. **付録A を更新**: 議論テーマの下に「結論」を記入します
2. **本文を更新**: 決定をドキュメントの本文に同期します
3. **決定を記録**: 「付録B：設計意思決定の記録」に追加します
4. **問題をマーク**: 「未解決の問題」リストで `[x]` を付けます

---

> **注**: RFC 番号は議論段階でのみ使用されます。承認後は番号を削除し、説明的なファイル名を使用します。
