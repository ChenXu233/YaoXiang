# RFC-020：動的モジュールとFFI統合

## 概要

本文書は RFC-001、008、018 を基礎として、YaoXiang の並行モデルをさらに詳細化・拡張し、**動的モジュールロード**、**外部関数インターフェース（FFI）**、および**より精细なスケジューリング最適化**といった実シナリオに対応します。中核設計には以下が含まれます：

1. **動的モジュールメタデータ契約**：同一言語で書かれた動的ライブラリに対してコンパイル時の依存記述を提供し、主プログラムが静的にDAGを構築しつつ、透明並行性を維持できるようにします。
2. **FFIスケジューリングセマンティクス**：外部関数はDAG内でデフォルトで `@block` ノードとして扱われ、アノテーションを通じて並列スケジューリングに統合可能です（FFIツールチェーンの詳細は [RFC-021](../review/021-library-driven-ffi-extension.md) を参照）。
3. **呼び出しコンテキストに基づく最適化**：静的な閾値フォールバックを置き換え、コンパイラがDAG内での関数の実際の役割（コンシューマ数、副作用など）に基づいてインライン化または独立ノードとしてのスケジューリングを智能的に判断します。
4. **制御フローとDAGのマージ機構**：Phiノードと動的展開を通じて、`if`、`loop` などの動的構造をデータフローグラフに自然に統合します。
5. **ランタイムスケジューラのメモリとパフォーマンス最適化**：ノードのライフサイクル管理、リージョン割り当て、ロックフリーキューなどの低コスト抽象化実装を明確化します。

本文書は言語仕様を完成させ、YaoXiang の並行モデルが静的プログラム全体解析と動的かつ外部とのインタラクションの両方に対応しつつ、高性能と開発体験を維持できるようにすることを目的とします。

## 動機

### 既存設計の不足

RFC-001/008/018 は洗練された透明並行モデルを構築しましたが、現実世界の要件に対しては依然として以下の盲点があります：

- **動的モジュール**：プログラムがプラグインや動的リンクライブラリをサポートする場合、主プログラムはコンパイル時にモジュール内部の呼び出し関係や依存を把握できず、グローバルDAGの構築が失敗します。
- **FFI呼び出し**：外部関数（例：Cライブラリ）は完全なブラックボックスであり、内部に並行処理、ブロッキング、副作用を含む可能性があります。これを通常のノードとして扱うと並行安全性が損なわれます。
- **小さな関数のスケジューリングオーバーヘッド**：RFC-001 で提案された「L1自動フォールバック」は静的閾値（命令数 <50）を採用しており、この暗黙的なルールにより開発者の挙動予測が困難で、複雑な呼び出しコンテキストに適応できません。
- **制御フローとDAGの融合**：`if`、`loop` などの動的構造のDAGでの表現がまだ不明確で、依存関係解析の正確性に影響する可能性があります。
- **ランタイムオーバーヘッドの制御**：DAGノード数の増加に伴い、スケジューラのメモリ管理とパフォーマンス最適化を明確化し、ボトルネック化を回避する必要があります。

### 目標

- 透明並行性の中核哲学を維持しつつ、動的モジュールとFFIに対して明確・安全・段階的なサポートを提供します。
- スケジューリング最適化を「暗黙的なグローバルルール」から「コンテキストに基づく智能的な判断」へと転換し、予測可能性とパフォーマンスを向上させます。
- DAGの動的制御フローの表現を完成させ、すべてのプログラム構造がデータフローモデルに自然に統合できることを保証します。
- スケジューラのメモリ管理とパフォーマンス最適化戦略を明確化し、低コスト抽象化を実現します。

## 提案

### 1. 動的モジュールメタデータ契約

#### 1.1 契約内容

YaoXiang でコンパイルされた各動的ライブラリ（`.yxo` / プラットフォーム固有の動的ライブラリ）には、**メタデータ記述ファイル**（`.yxmeta`）を添付する必要があります。これには以下が含まれます：

- **エクスポート関数のリスト**：各関数の完全な型シグネチャ（パラメータ、戻り値、リソースタグ）。
- **副作用タグ**：コンパイラが自動推論した `@pure` / `@io`（開発者が明示的にオーバーライド可能）。
- **リソース依存**：各パラメータがリソース型（例：`File`）かどうか、戻り値に新リソースが含まれるかどうか。
- **呼び出しグラフサマリ**（オプション）：循環依存検出のため、その関数が呼び出す可能性のある他のエクスポート関数のID。
- **所有権情報**：パラメータの所有権セマンティクス（借用/ムーブ）、戻り値の所有権。
- **並行安全性**：`Send`/`Sync` を満たすかの自動推論結果。

メタデータ形式には解析効率のためバイナリまたは構造化テキスト（例：MessagePack）を使用します。

#### 1.2 コンパイル時処理

主プログラムのコンパイル時に、動的モジュールの関数への呼び出しに遭遇すると：

1. 対応するモジュールの `.yxmeta` ファイルを読み込みます。
2. グローバルDAG内に**プレースホルダノード**を作成し、メタデータから取得した入出力依存関係、副作用タグなどを記録します。
3. プレースホルダノードは通常ノードと同様に依存関係解析に参加し、スケジューラが実行順序を事前に計画できます。

#### 1.3 ランタイムバインディング

動的モジュールのロード時：

- ランタイムで実際の関数シグネチャとメタデータの一致を検証します（バージョンミスマッチの防止）。
- プレースホルダノードを実際の関数ポインタにバインドします。

**サブグラフのスケジューリングセマンティクスについて**：動的モジュール内に独立したサブDAGが存在する場合（例：モジュール自身が並行ロジックを含む）、そのサブグラフは**独立したスケジューリングユニット**として実行されます。その境界はモジュールのエクスポート関数によって定義されます。エクスポート関数を呼び出すと、サブグラフが全体として実行開始され、その関数が返るまで継続します。サブグラフ内部のノードスケジューリングはそのサブグラフ自身のスケジューラが担当します（モジュール内部では標準スケジューラを引き続き使用可能）。ただし、サブグラフとメインDAGのインタラクションは入出力データフローのみに限定されます。メインDAG内のプレースホルダノードはサブグラフの開始と終了のみを扱い、内部のスケジューリングには介入しません。この設計により、モジュールのカプセル化を保証しつつ、メインDAGの静的整合性を維持します。

#### 1.4 安全性保証

- 動的モジュールが契約に違反する場合（例：`@pure` を主張しつつグローバル状態を変更する）、結果は開発者が責任を負います（FFIのunsafe境界と同様）。同一言語であるため、ランタイムチェック（例：メモリ隔離）で安全性を強化することは可能ですが、オーバーヘッドが増加します。
- モジュール間の循環依存：モジュールAがBを呼び出し、BがAを呼び出す場合で、メタデータに呼び出し関係が既に宣言されていれば、コンパイラが検出してエラーを報告できます。未宣言の場合、ランタイムでデッドロックが発生する可能性があり、スケジューラが検出してパニックします。

### 2. DAGにおけるFFIのスケジューリングセマンティクス

FFIの完全なツールチェーンサポート（動的ライブラリのロード、バインディング生成、型変換、メモリ所有権）は [RFC-021](../review/021-library-driven-ffi-extension.md) で定義されています。本セクションではDAGスケジューリングにおけるFFI呼び出しの動作についてのみ説明します。

#### 2.1 デフォルトのスケジューリング動作

外部関数（`native("symbol")` で宣言）はDAG内でデフォルトで **`@block` ノード** として扱われます：

- DAG並列スケジューリングに参加せず、現在のスレッドで直接同期実行されます。
- 実行中、スケジューラは内部の並行性に干渉しません。
- 戻り値は利用可能ですが、呼び出し自体は依存エッジを生成しません。

#### 2.2 オプションの並行アノテーション

開発者はアノテーションによりFFI呼び出しをDAGスケジューリングに統合できます（詳細は [RFC-021 §2.2](../review/021-library-driven-ffi-extension.md) を参照）：

- `@pure`：通常のDAGノードとして扱い、依存関係のない他のノードと並列実行可能。
- `@io`：リソース依存関係解析に参加し、同一リソースへの複数回の呼び出しを自動的にシリアライズ。

#### 2.3 スケジューラへの影響

FFIノードはスケジューラ内で通常ノードと同じ `TaskNode` 構造を使用し、違いは `effect` が `Block` としてマークされる点のみです。スケジューラは `Block` ノードに遭遇すると、並列スケジューリングをスキップして直接同期実行します。

### 3. 呼び出しコンテキストに基づく最適化

RFC-001 の「L1自動フォールバック」の静的閾値を置き換え、**コンパイラがDAG内の各呼び出しポイントの実際のコンテキストに基づいて智能的に判断**する方式に変更します。

#### 3.1 最適化判断の根拠

コンパイラは以下の各関数呼び出しノードを解析します：

- **コンシューマ数**：そのノードの結果がいくつの下流ノードで使用されるか。1の場合はインライン化候補の資格を持ちます。1より大きい場合は結果共有のため独立ノードとして保持する必要があります。
- **副作用**：ノードに `@io` 副作用がある場合、順序保証のため独立ノードとして保持する必要があります。
- **計算量推定**：命令数などのヒューリスティックを依然として参照できますが、厳格な閾値とはせず、インライン化利益の評価にのみ使用します。
- **リソース依存**：ノードがリソース変数（例：`File`）に関与し、リソース変数が上流下流間で受け渡される場合、インライン化が依存チェーンを破壊する可能性があるため、慎重な判断が必要です。

#### 3.2 インライン化操作

判断がインライン化となった場合：

- そのノードの計算ロジックを唯一下流ノードのコードに直接埋め込みます。
- DAGからそのノードを削除し、その入力を下流ノードの入力とします。
- 最終的なコード生成時、インライン化された関数は独立したスケジューリングユニットを生成しません。

#### 3.3 インライン化制限

- 再帰関数やループ本体内の呼び出しは通常インライン化せず、無限展開を防止します。
- モジュール境界をまたぐ（動的モジュール、FFI）関数はインライン化しません。
- 開発者は `@noinline` アノテーションで強制的にインライン化を禁止したり、`@forceinline` でコンパイラにインライン化を試行するよう指示したりできます。

#### 3.4 可観測性

コンパイラは最適化レポートを生成する必要があります（`--emit-optimization-report` で有効化）。これには以下の情報が含まれます：

- **各インラインポイント**：インライン化された関数名、呼び出し位置、インライン化の理由（例：「唯一のコンシューマで純粋関数」）。
- **独立ノードとして保持された理由**：例：「複数のコンシューマあり」「副作用を含む」「モジュール間の呼び出し」など。
- **判断統計**：インライン化総数、保持されたノード数。開発者が最適化効果を評価するのに役立ちます。

レポート出力形式はテキストまたはJSONで、ツールでの解析に適しています。

### 4. 制御フローとDAGの統合

#### 4.1 条件分岐（if）の処理

**Phiノード**（SSA形式を参考にする）を導入して分岐合流点を表現します：

- コンパイル時に各 `if` 式に対して以下を構築します：
  - 2つの分岐サブDAG（`then` と `else` にそれぞれ対応）。
  - 1つのPhiノード。入力には条件変数と2つの分岐の出力が含まれます。
- Phiノードのセマンティクス：条件変数の準備完了後、条件値に応じて対応する分岐の出力を自身の出力として選択します。
- ランタイムでは、Phiノードは条件変数に依存します。条件の準備完了後、動的に自身を選択された分岐の下流リストに追加し、その分岐の結果を待ちます。

DAGの例：
```
        cond
       /    \
  then DAG  else DAG
       \    /
        Phi
         |
      後続ノード
```

#### 4.2 ループ（loop/while）の処理

ループはフィードバックエッジを持つサブDAGとして扱われ、ランタイムで**オンデマンド展開**されます：

- コンパイル時にループ本体を識別し、以下を含む**ループテンプレート**を構築します：
  - 条件ノード。
  - ループ本体サブDAG。
  - 反復間で受け渡される状態変数。
- ランタイムでループ結果が必要になった時（例：ループ終了後に累積値を使用する場合）、スケジューラは反復の動的展開を開始します：
  1. 初回に条件ノードをスケジューリングし、真であれば初回反復のサブDAGをインスタンス化します。その入力には初期状態と外部変数が含まれます。
  2. 反復完了後に新しい状態を生成し、（新しい状態に依存する）条件ノードを再びスケジューリングして継続するかを判断します。
  3. 条件が偽になるまで繰り返し、最後の反復の出力がループ結果となります。

**複雑な例：ループ条件がループ本体内の更新に依存する場合**  
```yaoxiang
let mut x = 0
while x < 10 {
    x = compute(x)  // x はループ本体内で更新される
```
このパターンでは、条件ノード `x < 10` は各反復後に更新された `x` に依存します。DAG表現は以下の通りです：
- ループテンプレートには状態変数 `x` が含まれ、初期値は 0 です。
- 各反復：まず条件ノード（現在の `x` に依存）を実行し、真であれば `x = compute(x)` を実行して新しい `x` を生成し、再び条件ノードに入ります。
- ランタイムで上記の流れに従って動的展開し、条件が偽になるまで継続します。

反復間の依存関係は状態変数を通じて自然にデータフローを形成し、依存関係のある反復は自動的にシリアル化されます。依存関係のない反復（例：`map`）は並列化可能です。

#### 4.3 無限ループの特殊処理

単一の無限ループはメインDAGとして直接同期実行されます（スケジューリングオーバーヘッドなし）。複数の無限ループはバックグラウンドDAGとして、スケジューラのタイムスライスにより並行実行されます。

### 5. ランタイムスケジューラのメモリとパフォーマンス最適化

#### 5.1 ノードのライフサイクル管理

- 各ノードは**参照カウント**（アトミック変数）を維持し、その結果に依存するコンシューマ数を表します。
- ノードの実行が完了し、結果をすべての下流に受け渡した後、参照カウントがゼロになり、ノードメモリを解放できます。
- 結果値自体も参照カウント（`Arc<T>`）を採用しますが、最適化が可能です。結果が1人のコンシューマのみによって使用される場合、カウントオーバーヘッドを避けるため直接所有権をムーブします。

#### 5.2 リージョンメモリ割り当て（Arena）

動的に生成される大量の短寿命ノード（例：ループ反復）には**リージョンアロケータ**を使用します：

- 1回のループ展開に対して1つのメモリリージョンを割り当てます。
- リージョン内のノードは連続的に割り当てられ、全体を一括解放することでフラグメンテーションと解放オーバーヘッドを削減します。
- リージョン終了時、すべてのノードメモリを一括回収します。

#### 5.3 ロックフリーデータ構造

- 依存カウンタ：`AtomicUsize` を使用し、`fetch_sub` でアトミックにデクリメント。
- 準備完了キュー：Chase-Lev両端キューを採用（各スレッドローカルキュー + ワークスティーリング）し、ロック競合を削減。
- 下流リスト：作成後は読み取り専用で、並行変更を回避。

#### 5.4 適応的スケジューリング

- システムに1つの無限ループしかない場合は直接同期実行し、ゼロスケジューリングオーバーヘッドを実現します。
- タスク粒度とシステム負荷に応じて並列度を動的に調整します（例：キューの長さを監視してワーカースレッド数を調整）。

#### 5.5 低コスト抽象化の原則

すべてのスケジューリングオーバーヘッドはタスク数に比例し、各タスクの追加オーバーヘッド（作成、キューイング、依存処理）は数十ナノ秒レベルに抑えます。超細粒度タスクについては、インライン最適化（第3節参照）でスケジューリングを回避します。

## 詳細設計

### 6.1 動的モジュールメタデータ形式（ドラフト）

```rust
// メタデータファイル構造（簡略化）
struct Metadata {
    version: u32,
    functions: Vec<FuncMeta>,
}

struct FuncMeta {
    name: String,
    signature: TypeSignature,
    effects: EffectTag,      // Pure | IO | Block
    resource_params: Vec<usize>, // パラメータインデックスのリスト、リソース型のパラメータを示す
    calls: Vec<String>,       // 呼び出される他のエクスポート関数名（オプション）
    ownership: OwnershipInfo,
    send_sync: SendSync,      // Send/Sync を満たすかどうか
}
```

### 6.2 スケジューラのコアデータ構造

```rust
struct TaskNode {
    id: TaskId,
    deps: Vec<TaskId>,                // 上流依存
    remaining_deps: AtomicUsize,
    inputs: Vec<Option<Value>>,
    result: Option<Value>,
    func: Executable,
    downstream: Vec<TaskId>,           // 下流ノード（作成後は読み取り専用）
    effect: EffectTag,
    arena_id: Option<ArenaId>,         // 所属リージョン（オプション）
}

struct Scheduler {
    ready_queues: PerThreadQueue<TaskId>, // 各スレッドローカルキュー
    global_work_stealer: WorkStealer,
    arenas: ArenaAllocator,               // リージョンアロケータ
}
```

### 6.3 コンテキストに基づく最適化解析

コンパイラは MIR レベルで以下のステップを実行します：

1. グローバル呼び出しグラフとデータ依存グラフを構築します。
2. 各関数呼び出しノードについて、その出次数（コンシューマ数）を計算します。
3. 出次数が1で、関数が副作用を持たず（`@pure`）、非再帰である場合、「インライン化候補」とマークします。
4. ヒューリスティック（例：命令数）と組み合わせてインライン化利益を評価し、インライン化するか判断します。
5. インライン化時、呼び出しノードのコードを下流に埋め込み、依存関係を更新します。

### 6.4 制御フローノード表現

```rust
enum NodeKind {
    Normal(FuncId),
    Phi { cond: TaskId, then_branch: TaskId, else_branch: TaskId },
    LoopTemplate { cond: FuncId, body: FuncId, state_var: VarId },
    // ...
}
```

ランタイムの動的展開時、`LoopTemplate` は一連の `Normal` ノードインスタンスを生成します。

## トレードオフ

### 利点

- **動的モジュールの安全な統合**：メタデータ契約により、動的ライブラリがメインプログラムと並行モデルをシームレスに共有しつつ、静的DAGの整合性を維持できます。
- **FFIの段階的な統合**：開発者は外部関数に段階的にアノテーションを追加でき、安全な降下から効率的な並行性へと移行できます。
- **最適化が予測可能**：コンテキストに基づく判断が暗黙的な閾値を置き換え、動作が透明で、開発者はツールを通じて最適化を理解できます。
- **制御フローの自然な統合**：Phiノードと動的展開により、DAGがすべてのプログラム構造を表現でき、特別な構文が不要です。
- **パフォーマンスの拡張性**：リージョン割り当て、ロックフリーキューなどの設計により、スケジューラが大規模並行性に対応できます。

### 欠点

- **メタデータ契約がコンパイルの複雑性を増す**：動的ライブラリに対してメタデータの生成と解析が必要で、ツールチェーンのサポートが必要です。
- **FFIアノテーションが開発者の正確性に依存**：誤ったアノテーションがデータ競合を引き起こす可能性があり、ドキュメントとツールのヒントでリスクを低減する必要があります。
- **コンテキスト最適化解析に時間がかかる**：グローバル解析によりコンパイル時間が増加する可能性がありますが、インクリメンタルコンパイルで緩和できます。
- **動的展開がランタイムオーバーヘッドを増加させる**：ループ展開には動的なノード作成が必要ですが、リージョン割り当てで緩和できます。

## 実装戦略

### 段階的分割（優先度推奨を含む）

> **実装優先度の推奨**：初期段階で完璧な実装を追求する必要はなく、まずシンプルな方法でシステムを動作させ、その後段階的に最適化します。例えば：
> - 参照カウントは直接 `Arc` を使用できます。
> - ロックフリーキューは成熟したライブラリ（例：crossbeam の deque）を使用できます。
> - リージョン割り当てはまずシンプルな bump allocator を使用し、後に最適化できます。

#### 段階1：基本サポート（v0.7）
- [ ] FFIのデフォルト `@block` への降下を実装。
- [ ] FFI用の `@pure`、`@io` アノテーションを追加。
- [ ] リソースラッパー型（例：`File`）とその基本メソッドを実装。

#### 段階2：動的モジュールメタデータ（v0.8）
- [ ] メタデータ形式を設計し、動的ライブラリ用の `.yxmeta` を生成するようにコンパイラを変更。
- [ ] メインプログラムコンパイル時のメタデータ読み取りとプレースホルダノード作成を実装。
- [ ] ランタイムバインディング機構を実装。

#### 段階3：コンテキスト最適化（v0.9）
- [ ] 呼び出しグラフ解析を実装し、ノード出次数を計算。
- [ ] インライン化判断とコード生成サポートを追加。
- [ ] 最適化レポート出力を実装（インラインポイント、理由などを含む）。

#### 段階4：制御フローDAG融合（v0.10）
- [ ] Phiノードと条件分岐のコンパイル時表現を実装。
- [ ] ループテンプレートと動的展開のランタイムを実装。
- [ ] 無限ループのバックグラウンドスケジューリングを改善。

#### 段階5：パフォーマンス最適化（v1.0）
- [ ] リージョンアロケータを実装。
- [ ] ロックフリーキューとワークスティーリングを最適化。
