# RFC-020：動的モジュールと FFI 統合

## 摘要

本ドキュメントは RFC-001、008、018 を踏まえ、YaoXiang の並発モデルを**動的モジュールのロード**、**外部関数インターフェース（FFI）**、および**より精细なスケジューリング最適化**などの実践的なシナリオに対応するために細分化し拡張する。コア設計は以下のとおりである：

1. **動的モジュールメタデータ契約**：同一言語で書かれた動的ライブラリにコンパイル時の依存関係記述を提供し、プログラムが静的に DAG を構築できるようにしながら透明な並発を維持する。
2. **FFI スケジューリングセマンティクス**：外部関数は DAG ではデフォルトで `@block` ノードとなり、アノテーションを通じて並列スケジューリングに統合できる（FFI ツールチェーンの詳細については [RFC-021](../review/021-library-driven-ffi-extension.md) を参照）。
3. **呼び出しコンテキストに基づく最適化**：静的閾値へのフォールバックに代わり、コンパイラは関数が DAG における実際の役割（コンシューマ数、副作用など）に基づいてインライン化するか独立ノードとしてスケジューリングするかをインテリジェントに決定する。
4. **制御フローと DAG の統合メカニズム**：Phi ノードと動的展開を通じて、`if`、`loop` などの動的構造をデータフローに自然に統合する。
5. **ランタイムスケジューラのメモリとパフォーマンス最適化**：ノードのライフサイクル管理、領域アロケーション、ロックフリーキューなどの低コスト抽象化の実装を明確化する。

本ドキュメントは、YaoXiang の並発モデルが静的全プログラム分析に対応しながらも動的性质と外部相互作用を柔軟に処理し、高いパフォーマンスと開発体験を維持することを保証する言語仕様を完成させるものである。

## 動機

### 既存設計の不足

RFC-001/008/018 は優雅な透明な並発モデルを構築したが、現実世界の需求に対しては以下の盲点が存在する：

- **動的モジュール**：プログラムがプラグインや動的リンクライブラリをサポートしている場合、コンパイル時にモジュール内部の呼び出し関係と依存関係を主プログラムが知ることができず、グローバル DAG の構築に失敗する。
- **FFI 呼び出し**：外部関数（C ライブラリなど）は完全にブラックボックスであり、内部に並発、ブロッキング、副作用を含む可能性があり、単なる普通ノードとして扱えば並発安全性を破壊する。
- **小関数のスケジューリングオーバーヘッド**：RFC-001 で提案された「L1 自動フォールバック」は静的閾値（命令数 <50）を使用し、この暗黙的なルールにより開発者は動作を予測しにくく、複雑な呼び出しコンテキストに対応できない。
- **制御フローと DAG の融合**：`if`、`loop` などの動的構造の DAG における表現がまだ不明確であり、依存関係分析の正確性に影響を与える可能性がある。
- **ランタイムオーバーヘッド制御**：DAG ノード数が増加するにつれ、スケジューラのメモリ管理とパフォーマンス最適化を明確に設計する必要があり、ボトルネックになることを避ける必要がある。

### 目標

- 透明な並発というコア哲学を維持しつつ、動的モジュールと FFI に明確、安全、段階的なサポートを提供する。
- スケジューリング最適化を「暗黙的なグローバルルール」から「コンテキストベースのインテリジェントな意思決定」に转变させ、予測可能性とパフォーマンスを向上させる。
- 動的制御フローに対する DAG の表現を完成させ、すべてのプログラム構造がデータフローモデルに自然に統合できるようにする。
- スケジューラのメモリ管理とパフォーマンス最適化戦略を明確化し、低コスト抽象化を実現する。

## 提案

### 1. 動的モジュールメタデータ契約

#### 1.1 契約内容

YaoXiang でコンパイルされた各動的ライブラリ（`.yxo` / プラットフォーム固有の動的ライブラリ）には、**メタデータ記述ファイル**（`.yxmeta`）を添付する必要がある。内容包括：

- **エクスポート関数リスト**：各関数の完全な型シグネチャ（パラメータ、返回值、リソースマーク）。
- **副作用マーク**：`@pure` / `@io` はコンパイラが自動的に導出（開発者による明示的なオーバーライドも可能）。
- **リソース依存**：各パラメータがリソース型かどうか（例：`File`）、返回值が新リソースを含むかどうか。
- **呼び出しグラフサマリー**（オプション）：この関数が呼び出す可能性のある他のエクスポート関数 ID。跨モジュール循環依存の検出に使用。
- **所有権情報**：パラメータの所有権セマンティクス（借用/移動）、返回值の所有権。
- **並発安全**：`Send`/`Sync` の自動導出結果。

メタデータフォーマットはバイナリまたは構造化テキスト（MessagePack など）を使用し、パース効率を保証する。

#### 1.2 コンパイル時処理

主プログラムのコンパイル時、動的モジュール関数の呼び出しを検出すると：

1. 対応するモジュールの `.yxmeta` ファイルを読取る。
2. グローバル DAG に**プレースホルダノード**を作成し、メタデータから 얻た入出力依存、副作用マークなどを記録する。
3. プレースホルダノードは通常のノードと同様に依存関係分析に参加し、スケジューラは事前に実行順序を計画できる。

#### 1.3 ランタイムバインディング

動的モジュールのロード時：

- ランタイムは実際の関数シグネチャとメタデータの一致を検証する（バージョン不一致防止）。
- プレースホルダノードを実際の関数ポインタにバインドする。

**サブグラフスケジューリングセマンティクスについて**：動的モジュール内部に独立したサブ DAG（例如：モジュール自身が並発ロジックを含む）がある場合、そのサブグラフは**独立したスケジューリングユニット**として実行される。その境界はモジュールのエクスポート関数によって定義される：エクスポート関数を呼び出すとき、サブグラフは全体として実行を開始し、その関数が戻るまで継続する。サブグラフ内部のノードスケジューリングはサブグラフ自身のスケジューラが担当する（モジュール内部は標準スケジューラを引き続き使用可能）が、サブグラフと主 DAG の相互作用は入出力データフローに限られる——主 DAG のプレースホルダノードはサブグラフの開始と終了のみ关心し、内部スケジューリングには介入しない。この設計はモジュールのカプセル化を保証すると同時に、主 DAG が静的完全性を維持することを可能にする。

#### 1.4 安全保証

- 動的モジュールが契約に違反した場合（例：`@pure` を主張しながらグローバル状態を修正）、結果は開発者の責任となる（FFI の unsafe 境界と同樣）。ただし同一言語であるため、ランタイムチェック（例如：メモリ隔離）を通じて安全性を強化することは可能だが、オーバーヘッドが増加する。
- 跨モジュール循環依存：モジュール A が B を呼び、B が A を呼び出す場合で、メタデータに呼び出し関係が宣言されていれば、コンパイラは检测してエラー報告できる；宣言されていなければ、ランタイムでデッドロックが発生するかもしれず、スケジューラが检测して panic する。

### 2. DAG における FFI のスケジューリングセマンティクス

FFI の完全なツールチェーンサポート（動的ライブラリロード、バインディング生成、型変換、メモリ所有権）は [RFC-021](../review/021-library-driven-ffi-extension.md) で定義される。本節では、DAG スケジューリングにおける FFI 呼び出しの動作のみを記述する。

#### 2.1 デフォルトスケジューリング動作

`native("symbol")` で宣言された外部関数は、DAG ではデフォルトで **`@block` ノード**として扱われる：

- DAG 並列スケジューリングに参加せず、現在のスレッドで同期的に直接実行される。
- 実行中はスケジューラがその内部並発に介入しない。
- 返回値は使用可能だが、呼び出し自体には依存エッジが生成されない。

#### 2.2 オプショナルな並発アノテーション

開発者はアノテーションを通じて FFI 呼び出しを DAG スケジューリングに統合できる（詳細については [RFC-021 §2.2](../review/021-library-driven-ffi-extension.md) を参照）：

- `@pure`：通常の DAG ノードとして扱い、依存関係のない他のノードと並列実行可能。
- `@io`：リソース依存関係分析に参加し、同じリソースに対する複数回の呼び出しは自動的に直列化される。

#### 2.3 スケジューラへの影響

FFI ノードはスケジューラ内で通常のノードと同じ `TaskNode` 構造を使用する。違いは `effect` マークが `Block` であることのみ。スケジューラは `Block` ノードを検出すると並列スケジューリングをスキップし、直接同期実行する。

### 3. 呼び出しコンテキストに基づく最適化

RFC-001 の「L1 自動フォールバック」静的閾値に代わり、**コンパイラが各呼び出し点が DAG における実際のコンテキストに基づいてインテリジェントに決定する**方式に変更する。

#### 3.1 最適化決定の根拠

コンパイラは各関数呼び出しノードを分析する：

- **コンシューマ数**：このノードの結果を使用する下流ノードの数。1 の場合、インライン化の候補资格を持つ；1 より大きい場合、結果共有のために独立ノードとして保持する必要がある。
- **副作用**：ノードに `@io` 副作用がある場合、順序を保証するために独立ノードとして保持する必要がある。
- **計算量推定**：命令数などのヒューリスティックを引き続き参考として使用できるが、硬性的な閾値ではなく、インライン化の收益評価のみに使用する。
- **リソース依存**：ノードがリソース変数（例如：`File`）を関与し、そのリソース変数が上下游間で渡される場合、インライン化は依存チェーンを破壊するかもしれず、慎重に対処する必要がある。

#### 3.2 インライン操作

インライン化を決定した場合：

- そのノードの計算ロジックを唯一のords下游ノードのコードに直接埋め込む。
- DAG からそのノードを移除し、その入力は直接下流ノードの入力となる。
- 最終的なコード生成時、インライン化された関数は独立したスケジューリングユニットを生成しない。

#### 3.3 インライン化制限

- 再帰関数またはループ本体内の呼び出しは通常インライン化しない（無限展開防止）。
- 跨モジュール境界（動的モジュール、FFI）の関数はインライン化しない。
- 開発者は `@noinline` アノテーションを通じてインライン化を强制的に禁止したり、`@forceinline` でコンパイラにインライン化を指示したりできる。

#### 3.4 観測可能性

コンパイラは最適化レポートを生成する必要がある（`--emit-optimization-report` で有効化可能）。内容包括：

- **各インライン点**：インライン化された関数名、呼び出し位置、インライン化理由（例如：「唯一のコンシューマで純関数であるため」）。
- **独立ノードとして保持された理由**：例如：「複数のコンシューマあり」「副作用あり」「跨モジュール呼び出し」など。
- **決定統計**：インライン化総数、保持ノード数。開発者が最適化効果を評価するのを助ける。

レポート出力フォーマットはテキストまたは JSON とし、ツール解析を容易にする。

### 4. 制御フローと DAG の統合

#### 4.1 条件分岐（if）の処理

**Phi ノード**（SSA 形式を参考）を導入して分支マージポイントを表現する：

- コンパイル時に各 `if` 式に対して以下を構築する：
  - 2 つの分支サブ DAG（それぞれ `then` と `else` に対応）。
  - 条件変数と 2 つの分支の出力を入力とする Phi ノード。
- Phi ノードのセマンティクス：条件変数が準備できたら、条件値に基づいて対応する分支の出力を自分の出力として選択する。
- ランタイムでは、Phi ノードは条件変数に依存する；条件が準備できたら、条件変数に基づいて動的に自分を所望の分支の下流リストに追加し、その分支の結果を待つ。

示例 DAG：
```
        cond
       /    \
  then DAG  else DAG
       \    /
        Phi
         |
      后续ノード
```

#### 4.2 ループ（loop/while）の処理

ループはフィードバックエッジを持つサブ DAG として取り扱い、ランタイムで**オンデマンド展開**される：

- コンパイル時にループ体を識別し、**ループテンプレート**を構築する。内容包括：
  - 条件ノード。
  - ループ本体サブ DAG。
  - 迭代間渡される状態変数。
- ランタイムでループ結果が必要になった場合（例如：ループ終了後に累積値を使用）、スケジューラは動的に迭代の展開を開始する：
  1. 初回は条件ノードをスケジューリングし、真であれば最初の迭代のサブ DAG をインスタンス化し、入力には初期状態と外部変数を含む。
  2. 迭代完了後に新しい状態が生成され、条件ノードを再度スケジューリング（新状態に依存）して継続するかを决定する。
  3. 条件が偽になるまで繰り返し、最後の迭代の出力がループ結果となる。

**複雑な示例：ループ条件がループ体内部で更新されたものに依存**
```yaoxiang
let mut x = 0
while x < 10 {
    x = compute(x)  // x はループ体内で更新される
}
```
このパターンでは、条件ノード `x < 10` は各迭代後に更新された `x` に依存する。DAG での表現は以下のとおり：
- ループテンプレートには状態変数 `x` が含まれ、初期値は 0。
- 各迭代：まず条件ノードを実行（現在の `x` に依存）、真であれば `x = compute(x)` を実行して新しい `x` を生成し、条件ノードに再度入る。
- ランタイムは上記の流程で動的に展開し、条件が偽になるまで継続する。

迭代間の依存関係は状態変数を介して自然にデータフローを形成し、依存関係のある迭代は自動的に直列化され、依存関係のない迭代（例如：`map`）は並列実行可能である。

#### 4.3 無限ループの特殊処理

単一の無限ループは主 DAG として直接同期実行される（スケジューリングオーバーヘッドなし）；複数の無限ループはバックグラウンド DAG としてスケジューラのタイムスライスにより並列実行される。

### 5. ランタイムスケジューラのメモリとパフォーマンス最適化

#### 5.1 ノードライフサイクル管理

- 各ノードは**参照カウント**（アトミック変数）を維持し、その結果を依存するコンシューマの数を表す。
- ノード実行完了後に結果をすべての下流に渡した後、参照カウントがゼロになれば、そのノードのメモリは解放可能。
- 結果値自体も参照カウント（`Arc<T>`）を使用するが最適化可能：結果が単一のコンシューマのみに使用される場合、所有権を直接移動し、カウントオーバーヘッドを避ける。

#### 5.2 領域メモリアロケーション（Arena）

動的に生成された大量の短ライフサイクルノード（例如：ループ迭代）に対しては、**領域アロケータ**を使用する：

- 1 回のループ展開に対してメモリアリアを割り当てる。
- 領域内のノードは連続的に割り当てられ、整体が解放される。フラグメンテーションと解放オーバーヘッドを削減。
- 領域終了時に、すべてのノードメモリを一度に回收する。

#### 5.3 ロックフリーデータ構造

- 依存カウンタ：`AtomicUsize` を使用し、`fetch_sub` でアトミックデクリメント。
- レディキュー：Chase-Lev ダブルエンドキュー（各スレッドのローカルキュー + ワークスチールング）を採用し、ロック競合を削減。
- 下流リスト：作成後は読み取り専用になり、並発変更を避ける。

#### 5.4 アダプティブスケジューリング

- システムに単一の無限ループしかない場合、直接同期実行し、ゼロスケジューリングオーバーヘッド。
- タスク粒度とシステム负载に応じて、動的に並列度を調整（例如：キューの長さ監視を通じてワークスレッド数を調整）。

#### 5.5 低コスト抽象化原則

すべてのスケジューリングオーバーヘッドはタスク数に比例し、各タスクの追加オーバーヘッド（作成、エanqueuing、依存関係処理）は数十ナノ秒级别に制御される。超細粒度タスクに対しては、インライン最適化（第 3 節参照）を通じてスケジューリングを避ける。


### 6.2 スケジューラコアデータ構造

```rust
struct TaskNode {
    id: TaskId,
    deps: Vec<TaskId>,                // 上流依存
    remaining_deps: AtomicUsize,
    inputs: Vec<Option<Value>>,

    func: Executable,
    downstream: Vec<TaskId>,           // 下流ノード（作成後は読み取り専用）

}
```

### 6.3 コンテキストベースの最適化分析

コンパイラは MIR 層で以下のステップを実行する：

1. グローバル呼び出しグラフとデータ依存グラフを構築する。
2. 各関数呼び出しノードについて、その出次数（コンシューマ数）を計算する。
3. 出次数が 1 であり、関数が副作用なし（`@pure`）で、非再帰的であれば、「インライン化可能候補」としてマークする。
4. ヒューリスティック（例如：命令数）と組み合わせてインライン化收益を評価し、インライン化するかを決定する。
5. インライン化時、呼び出しノードのコードをその下流に埋め込み、依存関係を更新する。

### 6.4 制御フローノード表現

```rust
enum NodeKind {
    Normal(FuncId),
    Phi { cond: TaskId, then_branch: TaskId, else_branch: TaskId },
    LoopTemplate { cond: FuncId, body: FuncId, state_var: VarId },
    // ...
}
```

ランタイム動的展開時、`LoopTemplate` は一連の `Normal` ノードインスタンスを生成する。

## 权衡

### 优点

- **動的モジュールの安全な統合**：メタデータ契約により動的ライブラリが主プログラムとシームレスに並発モデルを共有でき、同時に静的 DAG の完全性を維持する。
- **FFI の段階的な統合**：開発者は外部関数に徐々にアノテーションを追加でき、安全なデグラデーションから効率的な並発への移行が可能。
- **最適化予測可能性**：コンテキストベースの決定は暗黙的な閾値に代わり、動作が透明であり、開発者はツールを通じて最適化を理解できる。
- **制御フローの自然な統合**：Phi ノードと動的展開により、DAG はすべてのプログラム構造を表現でき、特殊構文が不要。
- **パフォーマンススケーラビリティ**：領域アロケーション、ロックフリーキューなどの設計により、スケジューラが大規模並発に対応できる。

### 缺点

- **メタデータ契約によるコンパイル複雑性の増加**：動的ライブラリ用にメタデータを生成・解析する必要があり、ツールチェーンサポートが必要。
- **FFI アノテーションの開発者正確性への依存**：誤ったマークはデータ競合を引き起こすかもしれず、ドキュメントとツールヒントを通じてリスクを軽減する必要がある。
- **コンテキスト最適化分析の計算時間**：グローバル分析はコンパイル時間を増加させるかもしれず、インクリメンタルコンパイルで緩和可能。
- **動的展開によるランタイムオーバーヘッド増加**：ループ展開は動的にノードを生成する必要があり、領域アロケーションで緩和可能。

## 実装戦略

### 段階的区分（優先度推奨付き）

> **実装優先度推奨**：初期段階では完全な実装を追求する必要はなく、システム動作を開始するための単純な方案を採用し、逐步的に最適化することが可能。例如：
> - 参照カウントは直接 `Arc` を使用できる。
> - ロックフリーキューは成熟したライブラリ（例如：crossbeam の deque）を使用できる。
> - 領域アロケーションは最初は単純な bump allocator を使用し、後で最適化する。

#### 段階 1：基本サポート（v0.7）
- [ ] FFI をデフォルトで `@block` にデグレードする機能を実装。
- [ ] FFI 使用用の `@pure`、`@io` アノテーションを追加。
- [ ] リソースラップタイプ（例如：`File`）とその基本メソッドを実装。

#### 段階 2：動的モジュールメタデータ（v0.8）
- [ ] メタデータフォーマットを設計し、コンパイラが動的ライブラリ用に `.yxmeta` を生成するように修正。
- [ ] 主プログラムコンパイル時のメタデータ読み取りとプレースホルダノード作成機能を実装。
- [ ] ランタイムバインディングメカニズムを実装。

#### 段階 3：コンテキスト最適化（v0.9）
- [ ] 呼び出しグラフ分析を実装し、ノード出次数を計算。
- [ ] インライン化決定とコード生成サポートを追加。
- [ ] 最適化レポート出力（インライン点、理由などを含む）を実装。

#### 段階 4：制御フロー DAG 統合（v0.10）
- [ ] Phi ノードと条件分支のコンパイル時表現を実装。
- [ ] ループテンプレートと動的展開ランタイムを実装。
- [ ] 無限ループのバックグラウンドスケジューリングを完成させる。

#### 段階 5：パフォーマンス最適化（v1.0）
- [ ] 領域アロケータを実装。
- [ ] ロックフリーキューとワークスチールングを最適化。
- [ ] ベンチマークテストとチューンアップ。
