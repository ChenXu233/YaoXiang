# RFC-001：並作モデルとエラー処理システム

## 設計ソース

| ドキュメント | 関係 |
|------|------|
| [async-whitepaper](/src/archive/async-whitepaper) | 設計の原点、理論的基盤 |
| [language-spec](/src/design/language-spec) | 仕様の目標 |

## 摘要

YaoXiangの並作モデルを提案する：同期構文でロジックを記述し、実行時に自動的に並列実行する。コアメカニズム：三層並列アーキテクチャ + DAG依存関係分析 + Result型システム。

## クイック選択

| シナリオ | 書き方 | 説明 |
|------|------|------|
| 自動並列 | アノテーションなし（デフォルト） | 最大並列 |
| 同期待機 | `@eager` | 依存の完了を待機 |
| 完全順序 | `@block` | 非並列、デバッグ用 |
| 部分的並列 | `spawn` | @blockスコープ内の並列 |

## 動機

現在の主流言語の並列モデルは明確な欠陥がある：

| 言語 | 並列モデル | 問題点 |
|------|----------|------|
| Rust | async/await + tokio | 非同期传染、学習コストが高い |
| Go | goroutine | 型安全性がなし |
| Python | asyncio | GIL制限 |
| JavaScript | Promise/async | コールバック複雑 |

### コアな矛盾

1. **透明性 vs 制御性**：完全透明だが制御不可 vs 完全制御可能だが不透明
2. **並列 vs デバッグ可能性**：並列プログラムはデバッグが難しい vs デバッグ可能なプログラムは並列化しにくい

## 提案

### 1. 並作モデル：三層並列アーキテクチャ

> **説明**：L1/L2/L3はメンタルモデルであり、ユーザーが異なるシナリオを理解するのに役立つ。実際の実装は一套のメカニズムのみ：DAG自動分析 + アノテーション制御。

| 階層 | メンタルモデル | 構文 | 実行方式 | 並列度 |
|------|----------|------|----------|--------|
| **L1** | 並列禁止 | `@block` | 完全順序実行 | ❌ なし |
| **L2** | @block内並列 | `spawn` | @blockスコープ内の制御可能並列 | ⚠️ 一部 |
| **L3** | 完全並列 | デフォルト（アノテーションなし） | DAG自動分析 | ✅ 完全 |

#### L1: @block 同期モード

```yaoxiang
main: () -> Void @block = {
    data1 = fetch_sync("api1")
    data2 = fetch_sync("api2")
    process(data1, data2)    # 厳密な順序、非並列
}
```

#### L2: @block 内の制御可能並列

```yaoxiang
# spawnは@block関数内部でのみ使用可能
main: () -> Void @block = {
    spawn { data1 = fetch_data("api1") }
    spawn { data2 = fetch_data("api2") }
    # すべてのspawnの完了を待機（標準ライブラリ制御）
    process(data1, data2)
}
```

#### L3: 完全透明（デフォルト）

```yaoxiang
# アノテーション不要、コンパイラが自動的にDAGを分析
heavy_calc: (n: Int) -> Int = fibonacci(n)

auto_parallel: (n: Int) -> Int = {
    a = heavy_calc(1)    # 自動並列
    b = heavy_calc(2)    # 自動並列
    c = heavy_calc(3)    # 自動並列
    a + b + c            # 値が必要な時、すべての結果を待機
}
```

### 2. アノテーションの完全な比較

| 次元 | デフォルト（アノテーションなし） | `@eager` | `@block` | `spawn` |
|------|---------------|----------|----------|---------|
| **実行方式** | 自動DAG分析 | 同期で依存を待機 | 完全順序 | @block内並列 |
| **並列度** | ✅ 完全 | ⚠️ 依存順序に従う | ❌ なし | ⚠️ 一部 |
| **DAG構築** | ✅ | ✅ | ❌ | ✅ |

**選択ガイド**：
- 最大並列 → アノテーションなし（デフォルト）
- 順序付き副作用が必要 → `@eager`
- デバッグ/初心者/重要コード → `@block`
- @block内で並列が必要 → `spawn`

```yaoxiang
# デフォルト：最大並列
calc_all: () -> Int = {
    a = heavy_calc(1)    # 自動並列
    b = heavy_calc(2)    # 自動並列
    a + b
}

# @eager：同期待機
calc_seq: () -> Int @eager = {
    a = heavy_calc(1)    # 同期実行
    b = heavy_calc(2)    # 同期実行
    a + b
}

# @block：完全順序
calc_simple: () -> Int @block = {
    a = heavy_calc(1)    # 強制同期
    b = heavy_calc(2)    # 同期
    a + b
}

# spawn：@block内並列
calc_mixed: () -> Int @block = {
    spawn { heavy_calc(1) }
    spawn { heavy_calc(2) }
    heavy_calc(3)        # 同期
}
```

### 3. DAG依存関係分析

#### 3.1 コア原則：ボトムアップ実行

```
ユーザーコード（同期構文）：
    a = fetch(url0)
    b = fetch(url1)
    print(a)

コンパイル時分析（ボトムアップ）：
    print(a) は a を必要とする → fetch(url0) に依存
    fetch(url1) は誰も結果を使わない → 孤立DAG

実行時スケジューリング（リーフから開始）：
    fetch(url0) → print(a)    ← 依存チェーン、順序付き
    fetch(url1)                ← 孤立、独立並列
```

**重要な洞察**：「トップダウン」でFutureを生成するのではなく、結果から逆方向に依存関係を「ボトムアップ」で分析する。

#### 3.2 孤立DAG：独立並列

```
メイン�フロー：fetch(url0) → process → print
孤立：  fetch(url1)  ← 誰も結果を使わない、独立並列

スケジューラ：メイン�ローは依存チェーンで実行、孤立は別コアで並列
```

#### 3.3 リソース型と副作用

**コアアイデア**：リソース操作は型でマークし、DAGが自動的に依存関係を構築する。同一リソースは自動的に直列化され、異なるリソースは自動的に並列化される。

**リソース型の境界線——明確な定義**：

リソース型はコンパイラの組み込みマーク付き型である。以下の型はコンパイラによってリソースとして認識される：

| リソース型 | 説明 | コンパイラ動作 |
|----------|------|-----------|
| `FilePath` | ファイルシステムパス | 同一パス操作は自動的に直列化 |
| `HttpUrl` | HTTPエンドポイント | 同一URL操作は自動的に直列化 |
| `DBUrl` | データベース接続 | 同一接続操作は自動的に直列化 |
| `Console` | 標準出力 | すべてのConsole操作は自動的に直列化 |

ユーザー定義のリソース型は明示的にマークする必要がある：
```yaoxiang
Database: Resource              # 明示的にリソース型としてマーク
query: (Database, String) -> Result(Row, Error)
# パラメータDatabaseはResource、操作として自動的に認識
```

Resourceマーク付きでない型はコンパイラによってリソース依存関係を追跡されない。

**使用規則**：
- 変数でリソースハンドルを渡し、DAGが自動的に順序を管理
- literalを直接使用して同一リソースすることはユーザー設計の問題であり、言語の責任ではない

```yaoxiang
# ✅ 正しい：変数渡し、DAGが自動的に直列化
filename: String = "data.txt"
File.write(filename, x)
File.write(filename, y)    # DAGが直列化

# ⚠️ ユーザーの責任：literal
File.write("data.txt", x)
File.write("data.txt", y)  # 可能性：並列、ユーザーは自己責任
```

#### 3.4 無限ループ処理

```
1つのループ → 直接同期実行、ゼロスケジューリングオーバーヘッド
複数のループ → スケジューラがスライスを切り替えて、真の並列
```

### 4. Result型とエラー処理

```yaoxiang
Result: (T: Type, E: Type) -> Type = { ok: (T) -> Self, err: (E) -> Self }

# ? 演算子は透過的に伝播
process: () -> Result(Data, Error) = {
    data = fetch_data()?
    processed = transform(data)?
    save(processed)?
}
```

### 5. DAGノード設計

```rust
enum NodeKind {
    Task,      // タスクノード
    Value,     // 値ノード
    Control,   // 制御フローノード
}

struct Node {
    id: NodeId,
    kind: NodeKind,
    inputs: Vec<ValueNodeId>,   // 入力依存
    outputs: Vec<ValueNodeId>,  // 出力値
    span: Span,                 // ソースコード位置
}
```

| エッジ型 | 記号 | 意味 |
|--------|------|------|
| DataEdge | → | データ依存（値の流動） |
| ControlEdge | ● | 制御依存（順序実行） |
| SpawnEdge | ◎ | 並列入口（並列可能な起点） |

### 6. 型システム

```
Send → スレッド間転送の安全
Sync → スレッド間共有の安全
Arc(T) は Send + Sync を実装（スレッド安全参照カウント）
```

## トレードオフ

### 优点

1. **段階的な採用**：三層モデルが異なるスキルレベルに対応
2. **自然な構文**：同期コードで並列パフォーマンスを実現
3. **コンパイル時安全性**：Send/Sync制約がデータ競合を排除
4. **デバッグ可能性**：エラーグラフがエラー伝播の明確なビューを提供

### 欠点

1. **学習コスト**：DAG依存概念の理解が必要
2. **コンパイル時間**：全プログラムDAG分析は遅い可能性
3. **ツールチェーンの複雑さ**：新しいデバッグと可視化ツールが必要

## 代替案

| 方案 | なぜ選択しないか |
|------|--------------|
| 明示的なasync/awaitのみサポート | 透明な並列を実現できない |
| 完全透明な並列のみサポート | ユーザーが制御を失う |
| Go式のgoroutine | 型安全性なし、コンパイル時チェック不可 |
| L1モードのみ | 並作モデルのコア 가치를放弃 |

## 実装戦略

### 段階的区分

1. **段階1 (v0.1)**：@block同期モード、基础型
2. **段階2 (v0.2)**：FlowSchedulerスケジューラ
3. **段階3 (v0.3)**：spawnブロック、明示的並列
4. **段階4 (v0.5)**：L3完全透明、DAG自動分析
5. **段階5 (v0.6)**：エラーグラフ、グラフデバッガ
6. **段階6 (v1.0)**：本番可用 оптимизация

### 依存関係

- RFC-001は外部依存なし（基础コア）
- RFC-008（Runtime並列モデル）→ 設計完了
- RFC-011（泛型システム）→ 設計完了

### リスク

1. **DAG分析性能**：全プログラム分析はO(n²)可能性、最適化が必要
2. **ツールチェーン欠如**：デバッガをゼロから開発する必要
3. **ユーザー受容性**：透明な並列は良いドキュメントが必要
