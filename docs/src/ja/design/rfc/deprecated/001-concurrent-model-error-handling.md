## 概要

YaoXiang の並行モデルを提案する：同期構文でロジックを記述し、ランタイムが自動的に並行実行する。コアメカニズム：3層並行アーキテクチャ + DAG 依存解析 + Result 型システム。

## クイック選択

| シナリオ | 書き方 | 説明 |
|------|------|------|
| 自動並列 | 注釈なし（デフォルト） | 並行を最大化 |
| 同期待機 | `@eager` | 依存の完了を待機 |
| 完全順序 | `@block` | 並行なし、デバッグ用 |
| 部分的並行 | `spawn` | @block スコープ内の並行 |

## 動機

現在の主流言語の並行モデルには明らかな欠陥がある：

| 言語 | 並行モデル | 問題点 |
|------|----------|------|
| Rust | async/await + tokio | 非同期伝染、学習曲線が険しい |
| Go | goroutine | 型安全性がない |
| Python | asyncio | GIL の制限 |
| JavaScript | Promise/async | コールバックが複雑 |

### 中核的矛盾

1. **透明性 vs 制御可能性**：完全透明だが制御不可 vs 完全制御可能だが不透明
2. **並行 vs デバッグ可能性**：並行プログラムはデバッグが難しい vs デバッグ可能なプログラムは並行化しにくい

## 提案

### 1. 並行モデル：3層並行アーキテクチャ

> **説明**：L1/L2/L3 はメンタルモデルであり、ユーザーが異なるシナリオを理解するのに役立つ。実際の実装は1つのメカニズムのみ：DAG 自動解析 + 注釈制御。

| 階層 | メンタルモデル | 構文 | 実行方式 | 並列度 |
|------|----------|------|----------|--------|
| **L1** | 並行禁止 | `@block` | 純粋な逐次実行 | ❌ なし |
| **L2** | @block 内並行 | `spawn` | @block スコープ内の制御可能並行 | ⚠️ 一部 |
| **L3** | 完全並行 | デフォルト（注釈なし） | DAG の自動解析 | ✅ 完全 |

#### L1: @block 同期モード

```yaoxiang
main: () -> Void @block = {
    data1 = fetch_sync("api1")
    data2 = fetch_sync("api2")
    process(data1, data2)    # 厳密な順序、並行なし
}
```

#### L2: @block 内の制御可能並行

```yaoxiang
# spawn は @block 関数内部でのみ使用可能
main: () -> Void @block = {
    spawn { data1 = fetch_data("api1") }
    spawn { data2 = fetch_data("api2") }
    # すべての spawn の完了を待機（標準ライブラリが制御）
    process(data1, data2)
}
```

#### L3: 完全透明（デフォルト）

```yaoxiang
# 注釈は一切不要、コンパイラが自動的に DAG を解析
heavy_calc: (n: Int) -> Int = fibonacci(n)

auto_parallel: (n: Int) -> Int = {
    a = heavy_calc(1)    # 自動並列
    b = heavy_calc(2)    # 自動並列
    c = heavy_calc(3)    # 自動並列
    a + b + c            # 値が必要な時、すべての結果を待機
}
```

### 2. 注釈の完全比較

| 次元 | デフォルト（注釈なし） | `@eager` | `@block` | `spawn` |
|------|---------------|----------|----------|---------|
| **実行方式** | 自動 DAG 解析 | 依存の同期待機 | 純粋な逐次 | @block 内並行 |
| **並列度** | ✅ 完全 | ⚠️ 依存順序に従う | ❌ なし | ⚠️ 一部 |
| **DAG 構築** | ✅ | ✅ | ❌ | ✅ |

**選択ガイド**：
- 最大並行 → 注釈を書かない（デフォルト）
- 順序付き副作用が必要 → `@eager`
- デバッグ/初心者/重要コード → `@block`
- @block 内で並行が必要 → `spawn`

```yaoxiang
# デフォルト：並行を最大化
calc_all: () -> Int = {
    a = heavy_calc(1)    # 自動並列
    b = heavy_calc(2)    # 自動並列
    a + b
}

# @eager：同期待機
calc_seq: () -> Int @eager = {
    a = heavy_calc(1)    # 同期実行
    b = heavy_calc(2)    # 同期実行
    a + b
}

# @block：純粋な逐次
calc_simple: () -> Int @block = {
    a = heavy_calc(1)    # 強制同期
    b = heavy_calc(2)    # 同期
    a + b
}

# spawn：@block 内並行
calc_mixed: () -> Int @block = {
    spawn { heavy_calc(1) }
    spawn { heavy_calc(2) }
    heavy_calc(3)        # 同期
}
```

### 3. DAG 依存解析

#### 3.1 基本原則：ボトムアップ実行

```
ユーザーコード（同期構文）：
    a = fetch(url0)
    b = fetch(url1)
    print(a)

コンパイル時解析（ボトムアップ）：
    print(a) は a を必要とする → fetch(url0) に依存
    fetch(url1) は誰にも必要とされない → 孤立 DAG

ランタイムスケジューリング（葉から開始）：
    fetch(url0) → print(a)    ← 依存チェーン、順序付き
    fetch(url1)                ← 孤立、獨立並行
```

**重要な洞察**：「トップダウン」に Future を生成するのではなく、結果から逆方向に依存を解析する「ボトムアップ」。

#### 3.2 孤立 DAG：独立並行

```
メインチェーン：fetch(url0) → process → print
孤立：  fetch(url1)  ← 結果が必要ない、独立並行

スケジューラ：メインチェーンは依存チェーンで実行、孤立は別コアで並行
```

#### 3.3 リソース型と副作用

**基本思想**：リソース操作は型でマークされ、DAG が自動的に依存を構築。同一リソースは自動直列化、異なるリソースは自動並列化。

**リソース型の境界——明確な定義**：

リソース型はコンパイラの組み込みマーク付き型である。以下はコンパイラによってリソースとして認識される型：

| リソース型 | 説明 | コンパイラの動作 |
|----------|------|-----------|
| `FilePath` | ファイルシステムパス | 同一パスの操作は自動直列化 |
| `HttpUrl` | HTTP エンドポイント | 同一 URL の操作は自動直列化 |
| `DBUrl` | データベース接続 | 同一接続の操作は自動直列化 |
| `Console` | 標準出力 | すべての Console 操作は自動直列化 |

ユーザー定義のリソース型は明示的にマークが必要：
```yaoxiang
Database: Resource              # リソース型として明示的にマーク
query: (Database, String) -> Result(Row, Error)
# 引数 Database は Resource、操作として自動認識
```

Resource マーク付きでない型は、コンパイラによってリソース依存が追跡されない。

**使用規則**：
- 変数を通じてリソースハンドルを渡し、DAG が自動的に順序を管理
- リテラルで同一リソースを直接使用するのはユーザーの設計上の問題であり、言語の責任ではない

```yaoxiang
# ✅ 正しい：変数渡し、DAG が自動直列化
filename: String = "data.txt"
File.write(filename, x)
File.write(filename, y)    # DAG が直列化

# ⚠️ ユーザーの責任：リテラル
File.write("data.txt", x)
File.write("data.txt", y)  # 並行の可能性あり、ユーザーの責任
```

#### 3.4 無限ループ処理

```
1 つのループ → 直接同期実行、スケジューリングオーバーヘッドゼロ
複数のループ → スケジューラがスライスして切り替え、本当の意味での並行
```

### 4. Result 型とエラー処理

```yaoxiang
Result: (T: Type, E: Type) -> Type = { ok: (T) -> Self, err: (E) -> Self }

# ? 演算子は透過的に伝播
process: () -> Result(Data, Error) = {
    data = fetch_data()?
    processed = transform(data)?
    save(processed)?
}
```

### 5. DAG ノード設計

```rust
enum NodeKind {
    Task,      // タスクノード
    Value,     // 値ノード
    Control,   // 制御フローノード
}

struct Node {
    id: NodeId,
    kind: NodeKind,
    inputs: Vec<ValueNodeId>,   // 入力依存
    outputs: Vec<ValueNodeId>,  // 出力値
    span: Span,                 // ソースコード位置
}
```

| エッジ型 | 記号 | セマンティクス |
|--------|------|------|
| DataEdge | → | データ依存（値の流動） |
| ControlEdge | ● | 制御依存（順序実行） |
| SpawnEdge | ◎ | 並行入口（並列可能な起点） |

### 6. 型システム

```
Send → スレッド間安全転送可能
Sync → スレッド間安全共有可能
Arc(T) は Send + Sync を実装（スレッドセーフ参照カウント）
```

## トレードオフ

### 优点

1. **段階的採用**：3層モデルが異なるスキルレベルに対応
2. **自然な構文**：同期コードで並列性能を獲得
3. **コンパイル時安全性**：Send/Sync 制約でデータ競合を排除
4. **デバッグ可能性**：エラーグラフでエラー伝播の明確なビューを提供

### 欠点

1. **学習曲線**：DAG 依存概念の理解が必要
2. **コンパイル時間**：全プログラム DAG 解析は遅い可能性
3. **ツールチェーンの複雑さ**：全新的デバッグと可視化ツールが必要

## 代替案

| 方案 | 为什么不选择 |
|------|--------------|
| 明示的な async/await のみサポート | 透過的並行を実現できない |
| 完全透過的並行のみサポート | ユーザーが制御権を失う |
| Go 式の goroutine | 型安全性がない、コンパイル時チェック不可 |
| L1 モードのみ | 並行モデルのコア価値を放棄 |

## 実装戦略

### 段階的区分

1. **段階 1 (v0.1)**：@block 同期モード、 基本型
2. **段階 2 (v0.2)**：FlowScheduler スケジューラ
3. **段階 3 (v0.3)**：spawn ブロック、明示的並行
4. **段階 4 (v0.5)**：L3 完全透過、DAG 自動解析
5. **段階 5 (v0.6)**：エラーグラフ、グラフデバッガ
6. **段階 6 (v1.0)**：プロダクション可用最適化

### 依存関係

- RFC-001 は外部依存なし（基本コア）
- RFC-008（Runtime 並行モデル）→ 設計完了
- RFC-011（ジェネリクスシステム）→ 設計完了

### リスク

1. **DAG 解析性能**：全プログラム解析は O(n²) の可能性、最適化が必要
2. **ツールチェーンの欠如**：デバッガをゼロから開発する必要がある
3. **ユーザー受容性**：透明な並行には良いドキュメントが必要
