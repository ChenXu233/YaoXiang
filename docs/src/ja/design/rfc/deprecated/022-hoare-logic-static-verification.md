# RFC 022: ホア論理静的検証サポート（仕様コメントと仕様型）[廃止]

> **参考**:
> - [RFC-010: 統一型構文](../accepted/010-unified-type-syntax.md)
> - [RFC-011: ジェネリックシステム設計](../accepted/011-generic-type-system.md)
> - [RFC-009: 所有権モデル](../accepted/009-ownership-model.md)

## 摘要

本文は YaoXiang 言語に**ホア論理静的検証メカニズム**を導入することを提案します。開発者はコメント中で `//!` または `/*! ... !*/` の形式で事前条件、事後条件、循环不変式を記述できます。Debug Build では強制的な静的検証を行い、検証成功后でなければ Release Build へ進めません。Release Build では仕様コメントを無視し（ゼロオーバーヘッド）、検証キャッシュをクリアします。仕様は型システムの一部と見なし、「仕様型」（`Requires(P)`、`Ensures(P)` など）を形成し、用户拡張も可能です。この設計は言語の简洁さを保ちながら重要コードに高信頼性保障を提供し、YaoXiang の統一型モデルと完美に統合されます。

## 動機

### なぜこの機能/変更が必要か？

YaoXiang はすでに所有権モデル（RFC-009）と並列モデル（RFC-001）でメモリ安全とスレッド安全を保障していますが、論理的正当性は依然としてテストに依存しています。システムプログラミング、安全Critical分野（航空、宇宙、金融、OSカーネルなど）では、論理的誤りが壊滅的な結果を引き起こす可能性があります。既存の方案（Rust の借用チェックなど）はこの種の誤りを検出できません。ホア論理は数学的証明手段を提供しますが、従来の形式検証ツールは往往にして独立した仕様言語と複雑な学習コストを要求します。

### 現在の問題

- 論理的正当性はテストでのみ検証でき、コンパイル時に保証できない
- Critical系コード缺乏形式検証手段
- 既存の形式検証ツールは学習コストが高く、干流プログラミング言語と分離している

## 提案

### 核心設計

我々の目標は**軽量、言語一体型**の静的検証方案を設計することです：

- **Debug Build 必検証**：開発者は高信頼性が必要なモジュールに仕様を記述し、Debug Build時に強制検証成功后でなければ Release Build に進めません
- **構文の elegance**：`//!` コメントを使用し、新しいキーワードを導入せず、エディタ染色で識別可能
- **型システムとの融合**：仕様は型の一部となり、型検査に参加でき、用户自定义仕様型をサポート
- **証明可能とテスト可能**：静的証明にも、降级して実行時アサーションにもでき、段階的導入が容易

### 1. 仕様コメント構文

関数体または循环体の先頭に `//!`（単一行）または `/*! ... !*/`（複数行）を使用して仕様を記述します。

#### 1.1 統一仕様構文

仕様は YaoXiang 統一の `name: Type = expression` 構文モデルを採用し、型システムと完全に融合します：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0
    //! ensures: GreaterOrEqual(result, arr[0..n])
    //! ensures: ExistsMax(result, arr[0..n])
    // 実装...
}
```

- 仕様本質は**型宣言**であり、右辺はブール式です
- 左辺は仕様型インスタンス（型パラメータ付き可）
- 特殊変数 `result` で返り値を表せます

#### 1.2 循环仕様

```yaoxiang
while i < n {
    /*! invariant: Bounds[i, n] = 0 <= i <= n
                  && SumInvariant[s, arr[0..i]] !*/
    s = s + arr[i]
    i = i + 1
}
```

#### 1.3 仕様式

仕様右辺のブール式は YaoXiang 式構文を使用し、以下をサポート：
- 算術演算、比較演算、論理演算
- 量詞：`forall i in 0..n: P(i)`、`exists i in 0..n: P(i)` — 言語組み込みの論理構成
- 関数呼び出し（純粋関数でなければならない）

### 2. 仕様型システム

仕様型本质は YaoXiang の通常の型であり、統一構文モデルと完全に一致します。

#### 2.1 組み込み仕様型

コンパイラは以下に示す一般的な仕様型を組み込み済みです（仕様で直接使用可能）：

```yaoxiang
// 組み込み仕様型定義
NonEmpty: (T: Type) -> Type = { len: T; len > 0 }
Positive: Type = { x: Int; x > 0 }
GreaterOrEqual: (T: Type) -> Type = { result: T, arr: Array(T); result >= arr[0] && forall i in 1..arr.len: result >= arr[i] }
Bounds: (T: Type) -> Type = { i: T, n: T; 0 <= i && i <= n }
SumInvariant: (T: Type) -> Type = { s: T, arr: Array(T); s == sum(arr[0..i]) }

// 量詞構成（言語組み込み、非関数）
forall: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
exists: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
```

#### 2.2 ユーザー自定义仕様型

通常型定義と完全に一致し、ユーザーは自定义仕様型を定義できます：

```yaoxiang
// 正整数仕様の定義
Positive: Type = { x: Int; x > 0 }

// ソート済み配列仕様の定義
Sorted: (T: Ord) -> Type = {
    arr: Array(T);
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

// 最大値存在仕様の定義
ExistsMax: (T: Ord) -> Type = {
    result: T, arr: Array(T);
    exists i in 0..arr.len: result == arr[i]
    && forall j in 0..arr.len: result >= arr[j]
}
```

自定义仕様の使用：

```yaoxiang
sqrt: (x: Positive) -> Float = {
    //! ensures: SquareRootResult(result, x) = result * result <= x && (result+1)*(result+1) > x
    // 実装...
}

binary_search: (T: Ord) -> ((arr: Sorted(Array(T)), key: T) -> Option(Index)) = {
    //! ensures: SearchResult(result, arr, key)
    // 実装...
}
```

仕様型は他の型一样にジェネリックパラメータ、型制約をサポートし、型推論に参加できます。

### 3. コンパイルモード

| モード | 動作 | オプション |
|------|------|------|
| **Debug Build** | 仕様を解析し、検証条件を生成し、SMT求解器呼出して証明；検証成功后でなければ Release Build に進めません | `yaoxiangc --debug source.yx` |
| **Release Build** | すべての `//!` コメントを無視し、いかなるコードも生成しない；すべての検証キャッシュをクリア；積極的な最適化を有効化 | `yaoxiangc --release source.yx` |
| **実行時チェック** | 仕様を実行時アサーションに変換し、違反時に panic | `yaoxiangc --enable-runtime-checks source.yx` |

検証モードでは、証明に失敗すると、コンパイラはエラーを報告し、可能な反例（入力値など）を提供します。

### 4. 検証メカニズム

コンパイラは仕様を検証条件（Verification Conditions）に変換し、統合された SMT 求解器（Z3など）に送信します。検証過程はおおまかに以下の通りです：

1. 関数の `requires` と `ensures`、循环の `invariant` を収集
2. 各循环に対して、循環不変式証明義務を生成：循环突入前に成立、 各反復後に維持、循环終了後に事後条件を蕴含
3. 関数体を論理式に変換し、仕様と組み合わせて検証条件を形成
4. SMT 求解器を呼出して充足可能性を検査

求解器が `unsat`（充足不能）を返した場合、仕様は成立します。さもなければ反例を報告します。

### 5. テストとの結合

実行時チェックモードでは仕様をアサーションに変換し、テストに使用できます。仕様カバレッジツールと組み合わせることで、テストが仕様をどの程度カバーしているかを評価できます。将来的にはテストから候補仕様を自動推定する仕様マイニングツールも検討できます。

### 6. エディタサポート

`//!` と `/*! ... !*/` はエディタに特殊的コメントとして認識され、普通のコメントと区別するために異なる色（例：紫）で表示されます。言語サーバーは仕様のホバリングヒント、補完、検証エラー報告を提供できます。

## 詳細設計

### 構文変更

| 変更前 | 変更後 |
|------|------|
| 仕様コメント構文なし | `//!` と `/*! ... !*/` 仕様コメントを許可 |

### 7.1 構文拡張

既存の構文（RFC-010）を基础上、関数体と循环体の先頭にゼロ個以上の `//!` または `/*! ... !*/` コメント的出现を許可します。仕様構文は統一型構文と一致します：

```
spec_comment     ::= ('//!' spec_line) | ('/*!' spec_block '!*/')
spec_line        ::= spec_name ':' type_expr '=' expr
spec_name        ::= 'requires' | 'ensures' | 'invariant'
spec_block       ::= (spec_name ':' type_expr '=' expr ';')*
```

- 仕様本质は型宣言：`仕様名: 仕様型 = ブール式`
- `type_expr` は仕様型式であり、型パラメータを持てます
- `expr` は YaoXiang 式構文を使用し、量詞をサポート

### 7.2 型検査

検証モードでは、コンパイラは仕様コメントを対応する仕様型インスタンスに変換し、関数または循环のメタデータに記録します。

### 7.3 検証条件生成

weakest precondition または strongest postcondition 計算を採用し、循環不変式と組み合わせて一階述語論理式を生成します。生成された VC は SMT-LIB 形式を使用し、外部求解器を呼び出します。

### 7.4 エラー報告

証明に失敗した場合、求解器はモデル（反例）を提供する可能性があります。コンパイラはこれらの反例を読みやすい形式（具体的な入力値など）に変換し、ユーザーのデバッグを支援します。

### 7.5 実行時チェック

`--enable-runtime-checks` モードでは、コンパイラは仕様を `assert` 文に変換します：
- `requires`：関数入口に `assert(cond)` を挿入
- `ensures`：関数すべての返り点 inúmer前に `assert(cond)` を挿入し、`result` を実際の返り値に置換
- `invariant`：循环体先頭に `assert(cond)` を挿入

### 7.6 既存設計との統合

- **所有権モデル**：仕様中の式は所有権規則に従い、読めるだけで書けない（純粋関数）、副作用を避ける
- **ジェネリックシステム**：仕様型はジェネリックパラメータ（`Requires(P)` など）をサポートし、ジェネリック関数/型と組み合わせ可能
- **依存型**：仕様中の値依存型（配列の長さ `n` など）は自然に使える

### 型システムへの影響

- 仕様型は YaoXiang の通常の型であり、統一構文モデルと一致する
- コンパイラは一般的な仕様型（`Positive`、`NonEmpty`、`GreaterOrEqual` など）を組み込み済み
- ユーザーは通常の型定義一样に自定义仕様型を定義できる
- 仕様型はジェネリックパラメータを持て、型制約をサポートする

### 実行時動作

- **Debug Build**：SMT 求解器呼出して静的検証を実行し、コンパイル時間が増加；検証成功后検証結果をキャッシュ
- **Release Build**：仕様コメントは無視され、ゼロ実行時オーバーヘッド；すべての検証キャッシュをクリア；アクティブな最適化を有効化
- **実行時チェックモード**：assert 文を生成し、実行時に違反を検出

### コンパイラ改动

- パーサ：仕様コメント構文を識別
- 意味分析：仕様を収集し、仕様型に変換
- 検証バックエンド：検証条件を生成し、SMT 求解器を呼び出す
- コード生成：実行時チェックモードをサポート

### 向後互換性

- ✅ 完全な向後互換
- 仕様コメントは通常のコンパイルで無視され、既存コードに影響しない
- Release Build では仕様は無視され、追加オーバーヘッドなし

## 权衡

### メリット

- **Debug Build 検証**：Debug Build 時に強制検証され、論理的正当性を保障
- **構文の elegance**：純粋コメント形式、新しいキーワードなし、エディタに優しい
- **型システムとの融合**：仕様即ち型、拡張可能
- **段階的採用**：実行時チェックから静的検証に逐步的に移行可能
- **信頼性向上**：テストでは発見しにくい論理的誤りを検出可能

### デメリット

- **コンパイル時間**：検証モードはコンパイル時間を大幅に増加させる可能性がある
- **学習曲線**：効果的な仕様と量詞の書き方を覚える必要がある
- **SMT 求解器の限界**：一部の複雑な性子brush 自動証明不可能の場合がある

## 代替方案

| 方案 | メリット | デメリット |
|------|------|------|
| 新キーワード（`requires` など） | 構文が直感的 | 新しいキーワードの導入により简洁さが崩れる |
| 独立した仕様ファイル（CVLなど） | 仕様とコードが分離 | ファイル数が増加し、同期が困難 |
| 実行時アサーションのみ | 実装が简单 | 静的保証がない |
| **本方案（コメント+仕様型）** | 简洁さと機能のバランス | エディタサポートが必要 |

## 実装策略

### 段階划分

| 段階 | 内容 |
|------|------|
| **段階 1：基礎サポート** | パーサを拡張し、`//!` と `/*! ... !*/` コメントを識別して AST ノードに附加；検証モードで仕様を収集し、単純な検証条件（算術比較のみ）を生成；Z3 求解器を統合 |
| **段階 2：量詞サポート** | 量詞式，支持し、SMT-LIB の `forall`/`exists` に翻訳；仕様の IDE ハイライティングとホバリングヒントを提供 |
| **段階 3：最適化とツールチェーン** | 增量検証、検証済みモジュールのキャッシュ；仕様カバレッジレポート；仕様マイニングツール（テストから候補仕様を生成） |

### 依存関係

- RFC-009: 所有権モデル - 仕様式は純粋関数語義が必要
- RFC-010: 統一型構文 - 仕様型システムは型システムに基づく
- RFC-011: ジェネリックシステム設計 - 仕様型はジェネリックパラメータをサポート

### リスク

1. **SMT 求解器統合の複雑性**：Z3 などの求解器の統合は技術的課題に直面する可能性がある
   - 緩和：成熟した Rust Z3 バインディングを使用し、サポートする式の種類を逐步的に拡張

2. **検証失敗デバッグの困難性**：SMT 求解器が仕様を証明できない場合、ユーザーは原因を理解しにくい
   - 緩和：明確なエラーメッセージと反例の説明を提供

3. **性能オーバーヘッド**：検証モードはコンパイル時間を大幅に増加させる可能性がある
   - 緩和：增量検証とキャッシュメカニズムを実装

4. 状態を「受理済」に更新し、受理日を追加

2. ファイル先頭に拒否理由と日付を追加
3. 状態を「拒否済」に更新
