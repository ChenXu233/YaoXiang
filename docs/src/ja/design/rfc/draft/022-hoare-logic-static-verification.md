# RFC 022：ホーア論理による静的検証サポート（仕様コメントと仕様型）

> **ステータス**: レビュー中
> **著者**: 晨煦
> **作成日**: 2026-03-16
> **最終更新**: 2026-03-16

> **参照**:
> - [RFC-010: 統合型構文](./010-unified-type-syntax.md)
> - [RFC-011: ジェネリック型システム設計](./011-generic-type-system.md)
> - [RFC-009: 所有権モデル](./009-ownership-model.md)

## 概要

本稿では、YaoXiang 言語に**ホーア論理による静的検証メカニズム**を導入することを提案する。開発者は `//!` または `/*! ... !*/` の形式で事前条件、事後条件、ループ不変式をコメントに記述できる。Debug Build時に強制的な静的検証を行い、検証に合格した場合のみ Release Build に進む。Release Build では仕様コメントを無視し（ゼロオーバーヘッド）、検証キャッシュをクリアする。仕様自体は型システムの一部とみなされ、「仕様型」（`Requires(P)`、`Ensures(P)` など）を形成し、ユーザーが拡張可能である。この設計は、言語の簡潔さを維持しながら重要コードの高い信頼性を保証し、YaoXiang の統合型モデルとシームレスに統合することを目的とする。

## 動機

### この機能/変更なぜ必要か？

YaoXiang はすでに所有権モデル（RFC-009）と並行モデル（RFC-001）によりメモリ安全性とスレッド安全性を保証しているが、論理的な正しさはテストに依存している。宇宙、金融、OSカーネルなどのシステムプログラミング、安全重要分野では、論理エラーが壊滅的な結果を招く可能性がある。既存の解決策（Rust の借用チェックなど）はこの種のエラーを検出できない。ホーア論理は数学的証明手段を提供するが、従来の形式検証ツールは独立した仕様言語と複雑な学習曲線を必要とすることが多い。

### 現在の問題

- 論理的正しさはテストでのみ検証可能で、コンパイル時に保証できない
- 重要システムコードに形式検証手段がない
- 既存の形式検証ツールは学習曲線が急峻で、主流のプログラミング言語と乖離している

## 提案

### コア設計

私たちの目標は、**軽量で言語と一体化した**静的検証ソリューションを設計することである：

- **Debug Build 時は必須検証**：高信頼性が必要なモジュールで開発者が仕様を記述し、Debug Build 時に強制検証に合格した場合のみ Release Build に進む
- **文法が优雅**： `//!` コメントを使用。新しいキーワードを導入せず、エディタが構文強調できる
- **型システムとの融合**：仕様が型の一部となり、型検査に参加でき、ユーザー定義の仕様型をサポート
- **証明可能かつテスト可能**：静的証明もできれば、ランタイムアサーションに退化も可能で、段階的な導入が容易

### 1. 仕様コメント構文

関数本体またはループ本体の先頭に、`//!`（単一行）または `/*! ... !*/`（複数行）を使用して仕様を記述する。

#### 1.1 統合仕様構文

仕様は YaoXiang の統合的な `name: Type = expression` 構文モデルを採用しており、型システムと完全に統合されている：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0
    //! ensures: GreaterOrEqual(result, arr[0..n])
    //! ensures: ExistsMax(result, arr[0..n])
    // 実装...
}
```

- 仕様本質は**型宣言**であり、右辺はブール式
- 左辺は仕様型のインスタンス（型パラメータ付き）
- 戻り値を表す特別な変数 `result` を使用可能

#### 1.2 ループ仕様

```yaoxiang
while i < n {
    /*! invariant: Bounds[i, n] = 0 <= i <= n
                  && SumInvariant[s, arr[0..i]] !*/
    s = s + arr[i]
    i = i + 1
}
```

#### 1.3 仕様式

仕様の右辺のブール式は YaoXiang 式構文を使用し、以下をサポート：
- 算術演算、比較演算、論理演算
- 量詞：`forall i in 0..n: P(i)`、`exists i in 0..n: P(i)` — 言語組み込みの論理構成
- 関数呼び出し（純粋関数でなければならない）

### 2. 仕様型システム

仕様型は本質的に YaoXiang の通常の型であり、統合構文モデルと完全に一致する。

#### 2.1 組み込み仕様型

コンパイラは次の一般的に使用される仕様型を組み込んでいる（仕様で直接使用可能）：

```yaoxiang
// 組み込み仕様型定義
NonEmpty: (T: Type) -> Type = { len: T; len > 0 }
Positive: Type = { x: Int; x > 0 }
GreaterOrEqual: (T: Type) -> Type = { result: T, arr: Array(T); result >= arr[0] && forall i in 1..arr.len: result >= arr[i] }
Bounds: (T: Type) -> Type = { i: T, n: T; 0 <= i && i <= n }
SumInvariant: (T: Type) -> Type = { s: T, arr: Array(T); s == sum(arr[0..i]) }

// 量詞構成（言語組み込み、関数ではない）
forall: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
exists: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
```

#### 2.2 ユーザー定義仕様型

通常の型定義と完全に一致し、ユーザーは仕様型をカスタム定義できる：

```yaoxiang
// 正整数仕様の定義
Positive: Type = { x: Int; x > 0 }

// ソート済み配列仕様の定義
Sorted: (T: Ord) -> Type = {
    arr: Array(T);
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

// 最大値存在仕様の定義
ExistsMax: (T: Ord) -> Type = {
    result: T, arr: Array(T);
    exists i in 0..arr.len: result == arr[i]
    && forall j in 0..arr.len: result >= arr[j]
}
```

カスタム仕様の使用：

```yaoxiang
sqrt: (x: Positive) -> Float = {
    //! ensures: SquareRootResult(result, x) = result * result <= x && (result+1)*(result+1) > x
    // 実装...
}

binary_search: (T: Ord) -> ((arr: Sorted(Array(T)), key: T) -> Option(Index)) = {
    //! ensures: SearchResult(result, arr, key)
    // 実装...
}
```

仕様型は他の型一样に、ジェネリックパラメータ、型制約をサポートし、型推論に参加できる。

### 3. コンパイルモード

| モード | 動作 | オプション |
|------|------|------|
| **Debug Build** | 仕様を解析し、検証条件を生成し、SMT求解系を呼び出して証明。検証に合格した場合のみ Release Build に進む | `yaoxiangc --debug source.yx` |
| **Release Build** | すべての `//!` コメントを無視し、コードを生成しない。すべての検証キャッシュをクリア。積極的な最適化を有効化 | `yaoxiangc --release source.yx` |
| **ランタイムチェック** | 仕様をランタイムアサーションに変換し、違反時に panic | `yaoxiangc --enable-runtime-checks source.yx` |

検証モードで証明に失敗した場合、コンパイラはエラーを報告し、可能性のある反例（入力値など）を提供する。

### 4. 検証メカニズム

コンパイラは仕様を検証条件（Verification Conditions）に変換し、統合された SMT求解系（Z3など）に送信する。検証プロセスは概ね次のとおり：

1. 関数の `requires` と `ensures`、ループの `invariant` を収集
2. 各ループに対してループ不変式証明義務を生成：ループ突入前に成立すること、各反復後に維持されること、ループ終了後に事後条件を蕴含すること
3. 関数本体を論理式に変換し、仕様と組み合わせて検証条件を形成
4. SMT求解系を呼び出して充足可能性をチェック

求解系が `unsat`（充足不能）を返した場合、仕様は成立。さもなくば反例を報告する。

### 5. テストとの組み合わせ

ランタイムチェックモードは仕様をアサーションに変換し、テストに使用可能。仕様カバレッジツールと組み合わせることで、テストによる仕様のカバー度を評価できる。将来的には、テストから候选仕様を自動的に推論する仕様マイニングツールの導入も検討する。

### 6. エディタサポート

`//!` と `/*! ... !*/` はエディタに認識されて特別なコメントとして異なる色（紫色など）が付与され、通常のコメントと区別される。言語サーバーは仕様のホバー提示、補完、検証エラー報告を提供できる。

## 詳細設計

### 構文変更

| 変更前 | 変更後 |
|------|------|
| 仕様コメント構文なし | `//!` と `/*! ... !*/` 仕様コメントを許可 |

### 7.1 構文拡張

既存の構文（RFC-010）を拡張し、関数本体とループ本体の先頭に0個以上の `//!` または `/*! ... !*/` コメント的出现を許可する。仕様構文は統合型構文と一致する：

```
spec_comment     ::= ('//!' spec_line) | ('/*!' spec_block '!*/')
spec_line        ::= spec_name ':' type_expr '=' expr
spec_name        ::= 'requires' | 'ensures' | 'invariant'
spec_block       ::= (spec_name ':' type_expr '=' expr ';')*
```

- 仕様は本質的に型宣言である：`仕様名: 仕様型 = ブール式`
- `type_expr` は型パラメータ付きの仕様型式
- `expr` は YaoXiang 式構文を使用し、量詞をサポート

### 7.2 型検査

検証モードでは、コンパイラは仕様コメントを対応する仕様型インスタンスに変換し、関数またはループのメタデータに記録する。

### 7.3 検証条件生成

 weakest precondition または strongest postcondition 計算を使用し、ループ不変式と組み合わせて一階述語論理式を生成する。生成された VC は SMT-LIB 形式を使用し、外部求解系を呼び出す。

### 7.4 エラー報告

証明に失敗した場合、求解系はモデル（反例）を提供する可能性がある。コンパイラはこれらの反例を具体的な入力値など、読みやすい形式に変換し、ユーザーのデバッグを支援する。

### 7.5 ランタイムチェック

`--enable-runtime-checks` モードでは、コンパイラは仕様を `assert` 文に変換する：
- `requires`：関数入口に `assert(cond)` を挿入
- `ensures`：関数のすべてのり返り点の前に `assert(cond)` を挿入（`result` は実際の戻り値に置き換え）
- `invariant`：ループ本体のはじめに `assert(cond)` を挿入

### 7.6 既存設計との統合

- **所有権モデル**：仕様内の式は所有権規則に従い、読み取りのみ可能（純粋関数）、副作用を避ける
- **ジェネリックシステム**：仕様型はジェネリックパラメータ（`Requires(P)` など）をサポートし、ジェネリック関数/型と組み合わせ可能
- **依存型**：仕様内の値依存型（配列の長さ `n` など）を自然に利用可能

### 型システムへの影響

- 仕様型は YaoXiang の通常の型であり、統合構文モデルと一致する
- コンパイラは一般的に使用される仕様型（`Positive`、`NonEmpty`、`GreaterOrEqual` など）を組み込んでいる
- ユーザーは通常の型定義でカスタム仕様型を定義できる
- 仕様型はジェネリックパラメータを持ち、型制約をサポートする

### ランタイム動作

- **Debug Build**：SMT求解系を呼び出して静的検証を実行し、コンパイル時間が増加。検証成功后、検証結果をキャッシュ
- **Release Build**：仕様コメントは無視され、ランタイムオーバーヘッドゼロ。すべての検証キャッシュをクリア。Span キャッシュクリアなどの積極的な最適化を有効化
- **ランタイムチェックモード**：assert 文を生成し、ランタイムで違反を検出

### コンパイラ変更

- パーサー：仕様コメント構文を識別
- 意味解析：仕様を収集し、仕様型に変換
- 検証バックエンド：検証条件を生成し、SMT求解系を呼び出す
- コード生成：ランタイムチェックモードをサポート

### 後方互換性

- ✅ 完全な後方互換性
- 通常のコンパイルでは仕様コメントが無視され、既存コードに影響なし
- Release Build では仕様が無視され、追加オーバーヘッドなし

## トレードオフ

### 利点

- **Debug Build 検証**：Debug Build 時に強制検証され、論理的正しさが保証される
- **优雅な構文**：純粋なコメントであり、新しいキーワードがなく、エディタに優しい
- **型システムとの融合**：仕様は型であり、拡張可能
- **段階的な導入**：ランタイムチェックから静的検証への段階的移行が可能
- **信頼性の向上**：テストでは発見しにくい論理エラーを検出可能

### 欠点

- **コンパイル時間**：検証モードはコンパイル時間を大幅に増加させる可能性がある
- **学習曲線**：効果的な仕様と量詞の書き方を学ぶ必要がある
- **SMT求解系の限界**：一部の複雑な性質を自動的に証明できない可能性がある

## 代替案

| 方案 | 利点 | 欠点 |
|------|------|------|
| 新規キーワード（`requires` など） | 構文が直感的 | 新規キーワードの導入により簡潔さが損なわれる |
| 独立した仕様ファイル（CVLなど） | 仕様とコードが分離 | ファイル数が増加し、同期が困難 |
| ランタイムアサーションのみ | 実装がシンプル | 静的保証がない |
| **本提案（コメント+仕様型）** | 簡潔さと機能のバランス | エディタサポートが必要 |

## 実装戦略

### 段階的区分

| 段階 | 内容 |
|------|------|
| **段階1：基本サポート** | パーサーを拡張し、`//!` と `/*! ... !*/` コメントを識別して AST ノードに添付。検証モードで仕様を収集し、簡単な検証条件（算術比較のみ）を生成。Z3求解系を統合 |
| **段階2：量詞サポート** | 量詞式をサポートし、SMT-LIB の `forall`/`exists` に変換。仕様の IDE 強調表示とホバー提示を提供 |
| **段階3：最適化とツールチェーン** | 増分検証とキャッシュされた検証済みモジュールのサポート。仕様カバレッジレポート。仕様マイニングツール（テストから候选仕様を生成） |

### 依存関係

- RFC-009: 所有権モデル - 仕様式には純粋関数意味論が必要
- RFC-010: 統合型構文 - 仕様型システムは型システムに基づく
- RFC-011: ジェネリック型システム設計 - 仕様型はジェネリックパラメータをサポート

### リスク

1. **SMT求解系統合の複雑性**：Z3 などの求解系の統合に技術的な課題が生じる可能性がある
   - 軽減：成熟した Rust Z3 bindings を使用し、サポートする式タイプを段階的に拡張

2. **検証失敗デバッグの困難さ**：SMT求解系が仕様を証明できない場合、ユーザーが原因を理解するのが難しい場合がある
   - 軽減：明確なエラーメッセージと反例の説明を提供

3. **パフォーマンスオーバーヘッド**：検証モードはコンパイル時間を大幅に増加させる可能性がある
   - 軽減：増分検証とキャッシュメカニズムを実装
