# RFC 022: ホア論理静的検証サポート（仕様コメントと仕様型）

> **ステータス**: レビュー中
> **著者**: 晨煦
> **作成日**: 2026-03-16
> **最終更新**: 2026-03-16

> **参考**:
> - [RFC-010: 統一型構文](./010-unified-type-syntax.md)
> - [RFC-011: ジェネリック型システム設計](./011-generic-type-system.md)
> - [RFC-009: 所有権モデル](./009-ownership-model.md)

## 概要

本稿では、YaoXiang 言語に**ホア論理静的検証メカニズム**を導入することを提案する。開発者は `//!` または `/*! ... !*/` の形式でコメントに前置条件、後置条件、ループ不変式を記述できる。Debug Build では強制的な静的検証を行い、検証通过了後でなければ Release Build を行えない。Release Build では仕様コメントを無視し（ゼロオーバーヘッド）、検証キャッシュをクリアする。仕様自体は型システムの一部と見なされ、「仕様型」（`Requires(P)`、`Ensures(P)` など）を形成し、ユーザーが拡張することもできる。この設計は、言語のシンプルさを維持しながら、重要コードに高い信頼性の保証を提供することを目的とし、YaoXiang の統一型モデルと完璧に統合される。

## 動機

### この機能/変更が必要な理由

YaoXiang は既に所有権モデル（RFC-009）と並行モデル（RFC-001）によりメモリ安全とスレッド安全を保証しているが、論理的正確性は依然としてテストに依存している。システムプログラミング、安全性が重要な分野（航空宇宙、金融、OS カーネルなど）では、論理的エラーが壊滅的な結果をもたらす可能性がある。既存の解決策（Rust の借用チェックなど）はこのようなエラーを検出できない。ホア論理は数学的証明の手法を提供するが、従来の形式検証ツールは独立した仕様言語と複雑な学習曲線を必要とすることが多い。

###  現在の問題

- 論理的正確性はテストでのみ検証でき、コンパイル時に保証できない
- 重要システムコードが形式検証手段を欠いている
- 既存の形式検証ツールは学習曲線が急で、主流プログラミング言語と切り離されている

## 提案

### コア設計

私たちの目標は、**軽量で言語と一体化した**静的検証ソリューションを設計することである：

- **Debug Build 必須検証**：開発者は高い信頼性を要するモジュールに仕様を記述し、Debug Build で強制的に検証を通過しなければならない
- **洗練された構文**：`//!` コメントを使用し、新しいキーワードを導入せず、エディタの色分けで識別可能
- **型システムとの融合**：仕様が型の一部となり、型検査に参加でき、ユーザーが仕様型を拡張可能
- **証明可能かつテスト可能**：静的に証明できる他、実行時アサーションに降格も可能で、段階的な導入が容易

### 1. 仕様コメント構文

関数本体またはループ本体の先頭で、`//!`（単一行）または `/*! ... !*/`（複数行）を使用して仕様を記述する。

#### 1.1 統一仕様構文

仕様は YaoXiang の統一された `name: Type = expression` 構文モデルを採用しており、型システムと完全に統合されている：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0
    //! ensures: GreaterOrEqual(result, arr[0..n])
    //! ensures: ExistsMax(result, arr[0..n])
    // 実装...
}
```

- 仕様は本質的に**型宣言**であり、右辺はブール式である
- 左辺は仕様型のインスタンス（型パラメータ付きの場合あり）
- 戻り値を表す特別な変数 `result` を使用可能

#### 1.2  ループ仕様

```yaoxiang
while i < n {
    /*! invariant: Bounds[i, n] = 0 <= i <= n
                  && SumInvariant[s, arr[0..i]] !*/
    s = s + arr[i]
    i = i + 1
}
```

#### 1.3 仕様式

仕様右辺のブール式は YaoXiang 式構文を使用し、以下をサポート：

- 算術演算、比較演算、論理演算
- 量詞：`forall i in 0..n: P(i)`、`exists i in 0..n: P(i)` — 言語組み込みの論理構成要素
- 関数呼び出し（純関数である必要がある）

### 2. 仕様型システム

仕様型は本質的に YaoXiang の通常の型であり、统一構文モデルと完全に一貫している。

#### 2.1 組み込み仕様型

コンパイラは以下的一般的な仕様型を組み込み込んでいる（仕様内で直接使用可能）：

```yaoxiang
// 組み込み仕様型定義
NonEmpty: (T: Type) -> Type = { len: T; len > 0 }
Positive: Type = { x: Int; x > 0 }
GreaterOrEqual: (T: Type) -> Type = { result: T, arr: Array(T); result >= arr[0] && forall i in 1..arr.len: result >= arr[i] }
Bounds: (T: Type) -> Type = { i: T, n: T; 0 <= i && i <= n }
SumInvariant: (T: Type) -> Type = { s: T, arr: Array(T); s == sum(arr[0..i]) }

// 量詞構成（言語組み込み、関数ではない）
forall: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
exists: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
```

#### 2.2 ユーザー定義仕様型

通常の型定義と完全に一貫しており、ユーザーは独自の仕様型を定義できる：

```yaoxiang
// 正整数仕様の定義
Positive: Type = { x: Int; x > 0 }

// ソート済み配列仕様の定義
Sorted: (T: Ord) -> Type = {
    arr: Array(T);
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

// 最大値仕様の定義
ExistsMax: (T: Ord) -> Type = {
    result: T, arr: Array(T);
    exists i in 0..arr.len: result == arr[i]
    && forall j in 0..arr.len: result >= arr[j]
}
```

ユーザー定義仕様の使用：

```yaoxiang
sqrt: (x: Positive) -> Float = {
    //! ensures: SquareRootResult(result, x) = result * result <= x && (result+1)*(result+1) > x
    // 実装...
}

binary_search: (T: Ord) -> ((arr: Sorted(Array(T)), key: T) -> Option(Index)) = {
    //! ensures: SearchResult(result, arr, key)
    // 実装...
}
```

仕様型は他の型と同様に、ジェネリックパラメータ、型制約をサポートし、型推論に参加できる。


### 3. コンパイルモード

| モード | 動作 | オプション |
|------|------|------|
| **Debug Build** | 仕様を解析し、検証条件を生成し、SMT求解器を呼び出して証明。検証通過後でないと Release Build 不可 | `yaoxiangc --debug source.yx` |
| **Release Build** | すべての `//!` コメントを無視し、コードを生成しない。すべての検証キャッシュをクリア。積極的な最適化を有効化 | `yaoxiangc --release source.yx` |
| **実行時チェック** | 仕様を実行時アサーションに変換し、違反時に panic | `yaoxiangc --enable-runtime-checks source.yx` |

検証モードで証明に失敗した場合、コンパイラはエラーを報告し、可能な反例（入力値など）を提供する。

### 4. 検証メカニズム

コンパイラは仕様を検証条件（Verification Conditions）に変換し、統合された SMT求解器（Z3など）に送信する。検証プロセスは大まかに 다음과 같다：

1. 関数の `requires` と `ensures`、ループの `invariant` を収集
2. 各ループに対してループ不変式証明義務を生成：ループ進入前に成立、 各反復後に維持、ループ終了後に後置条件を涵意
3. 関数本体を論理式に変換し、仕様と組み合わせて検証条件を形成
4. SMT求解器を呼び出して充足可能性をチェック

求解器が `unsat`（充足不能）を返した場合、仕様は成立。さもなければ反例を報告する。

### 5. テストとの組み合わせ

実行時チェックモードは仕様をアサーションに変換し、テストに使用可能。仕様カバレッジツールと組み合わせることで、テストの仕様カバレッジ度を評価できる。将来的には、テストから仕様を自動推論する仕様マイニングツールの導入も検討できる。

### 6. エディタサポート

`//!` と `/*! ... !*/` はエディタに特殊的コメントとして認識され、通常のコメントと異なる色（紫色など）で表示できる。言語サーバーは仕様のホバー情報、补完、検証エラー報告を提供できる。

## 詳細設計

### 構文変更

| 変更前 | 変更後 |
|------|------|
| 仕様コメント構文なし | `//!` と `/*! ... !*/` 仕様コメントを許可 |

### 7.1 構文拡張

既存の構文（RFC-010）を拡張し、関数本体とループ本体の先頭に0行以上の `//!` または `/*! ... !*/` コメント的出现を許可する。仕様構文は統一型構文と一貫している：

```
spec_comment     ::= ('//!' spec_line) | ('/*!' spec_block '!*/')
spec_line        ::= spec_name ':' type_expr '=' expr
spec_name        ::= 'requires' | 'ensures' | 'invariant'
spec_block       ::= (spec_name ':' type_expr '=' expr ';')*
```

- 仕様は本質的に型宣言：`仕様名: 仕様型 = ブール式`
- `type_expr` は型パラメータ付きの仕様型式
- `expr` は YaoXiang 式構文を使用し、量詞をサポート

### 7.2 型検査

検証モードでは、コンパイラは仕様コメントを対応する仕様型インスタンスに変換し、関数またはループのメタデータに記録する。

### 7.3 検証条件生成

最弱事前条件（weakest precondition）または最強事後条件（strongest postcondition）計算を採用 し、ループ不変式と組み合わせて一階述語論理式を生成する。生成された VC は SMT-LIB 形式を使用し、外部求解器を呼び出す。

### 7.4 エラー報告

証明に失敗した場合、求解器はモデル（反例）を提供する可能性がある。コンパイラはこれらの反例を読みやすい形式に変換すべきである（具体的な入力値など），ユーザーのデバッグを支援する。

### 7.5 実行時チェック

`--enable-runtime-checks` モードでは、コンパイラは仕様を `assert` 文に変換する：
- `requires`：関数エントリポイントに `assert(cond)` を挿入
- `ensures`：関数のすべての返回点に `assert(cond)` を挿入，其中 `result` は実際の戻り値で置換
- `invariant`：ループ本体の先頭に `assert(cond)` を挿入

### 7.6 既存設計との統合

- **所有権モデル**：仕様内の式は所有権ルールに従い、読み取りのみ可能（純関数）、副作用を避ける
- **ジェネリックシステム**：仕様型はジェネリックパラメータ（`Requires(P)` など）をサポートし、ジェネリック関数/型と組み合わせ可能
- **依存型**：仕様内の値依存型（配列長 `n` など）が自然に使用可能

### 型システムへの影響

- 仕様型は YaoXiang の通常の型であり、統一構文モデルと一貫している
- コンパイラはよく使用される仕様型（`Positive`、`NonEmpty`、`GreaterOrEqual` など）を組み込み込んでいる
- ユーザーは通常の型定義を通じて独自の仕様型を定義できる
- 仕様型はジェネリックパラメータを带き、型制約をサポート

### 実行時動作

- **Debug Build**：SMT求解器を呼び出して静的検証を行い、コンパイル時間が増加。検証成功後は検証結果をキャッシュ
- **Release Build**：仕様コメントが無視され、ゼロ実行時オーバーヘッド。すべての検証キャッシュをクリア。Span キャッシュクリアなどの積極的な最適化を有効化
- **実行時チェックモード**：assert 文を生成し、実行時に違反を検出

### コンパイラ変更

- パーサー：仕様コメント構文を認識
- 意味解析：仕様を収集し、仕様型に変換
- 検証バックエンド：検証条件を生成し、SMT求解器を呼び出す
- コード生成：実行時チェックモードをサポート

### 後方互換性

- ✅ 完全な後方互換性
- 仕様コメントは通常コンパイル時に無視され、既存コードに影響しない
- Release Build で仕様が無視され、追加オーバーヘッドなし

## トレードオフ

### 利点

- **Debug Build 検証**：Debug Build で強制検証し、論理的正確性を確保
- **洗練された構文**：純粋なコメント、新しいキーワードなし、エディタに優しい
- **型システムとの融合**：仕様即型、拡張可能
- **段階的な導入**：実行時チェックから静的検証へ徐々に移行可能
- **信頼性向上**：テストでは発見しにくい論理的エラーを検出可能

### 欠点

- **コンパイル時間**：検証モードでコンパイル時間が大幅に増加する可能性
- **学習曲線**：効果的な仕様と量詞の書き方を学ぶ必要がある
- **SMT求解器の限界**：一部の複雑な性子장은 자동 증명할 수 없을 수 있음

## 代替案

| 方案 | 利点 | 欠点 |
|------|------|------|
| 新キーワード（`requires` など） | 構文が直感的 | 新しいキーワードの導入によりシンプルさが損なわれる |
| 独立した仕様ファイル（CVLなど） | 仕様とコードが分離 | ファイル数が増加し、同期が困難 |
| 実行時アサーションのみ | 実装がシンプル | 静的保証不可 |
| **本方案（コメント+仕様型）** | シンプルさと機能のバランス | エディタサポートが必要 |

## 実装戦略

### 段階的区分

| 段階 | 内容 |
|------|------|
| **段階1：基礎サポート** | パーサーを拡張し、`//!` と `/*! ... !*/` コメントを認識して AST ノードに添付。検証モードで仕様を収集し、単純な検証条件（算術比較のみ）を生成。Z3求解器を統合 |
| **段階2：量詞サポート** | 量詞式をサポートし、`forall`/`exists` として SMT-LIB に変換。仕様の IDE ハイライトとホバーサポートを提供 |
| **段階3：最適化とツールチェーン** | 増分検証とキャッシュ済みモジュールのサポート。仕様カバレッジレポート。仕様マイニングツール（テストから仕様を生成） |

### 依存関係

- RFC-009: 所有権モデル - 仕様式には純関数セマンティクスが必要
- RFC-010: 統一型構文 - 仕様型システムは型システムに基づく
- RFC-011: ジェネリックシステム設計 - 仕様型はジェネリックパラメータをサポート

### リスク

1. **SMT求解器統合の複雑さ**：Z3 などの求解器の統合に技術的課題が生じる可能性
   - 軽減策：確立された Rust Z3 バインディングを使用し、対応する式类型を段階的に拡張

2. **検証失敗デバッグの困難さ**：SMT求解器が仕様を証明できない場合、ユーザーが理由を理解しにくい
   - 軽減策：明確なエラーメッセージと反例の説明を提供

3. **パフォーマンスオーバーヘッド**：検証モードでコンパイル時間が大幅に増加する可能性
   - 軽減策：増分検証とキャッシュメカニズムを実装

1. RFC を `docs/design/accepted/` ディレクトリに移動
2. ファイル名を記述的名称に更新（`hoare-logic-static-verification.md` など）
3. ステータスを「正式」に更新
4. ステータスを「承認済み」に更新し、承認日を追加

1. `docs/design/rfc/` ディレクトリに残す
2. ファイル上部に拒否理由と日付を追加
3. ステータスを「拒否済み」に更新
