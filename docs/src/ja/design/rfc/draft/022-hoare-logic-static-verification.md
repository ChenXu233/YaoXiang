# RFC 022: 霍尔論理静的検証サポート（仕様コメントと仕様型）

> **状態**: レビュー中
> **著者**: 晨煦
> **作成日**: 2026-03-16
> **最終更新**: 2026-03-16

> **参照**:
> - [RFC-010: 統合型構文](./010-unified-type-syntax.md)
> - [RFC-011: ジェネリック型システム設計](./011-generic-type-system.md)
> - [RFC-009: 所有権モデル](./009-ownership-model.md)

## 摘要

本稿では、YaoXiang 言語に**霍尔論理（Hoare Logic）静的検証メカニズム**を導入することを提案する。開発者は `//!` または `/*! ... !*/` の形式で事前条件、事後条件、 ループ不変式をコメントに記述できる。Debug Build で強制的な静的検証を行い、検証成功后のみ Release Build を行う。Release Build では仕様コメントを無視し（ゼロオーバーヘッド）、検証キャッシュをクリアする。仕様自体は型システムの一部とみなされ、「仕様型」（例：`Requires(P)`、`Ensures(P)`）を形成し、用户が拡張可能である。この設計は、言語の簡潔さを維持しながら、重要なコードに高い信頼性の保証を提供し、YaoXiang の統合型モデルと完璧に統合することを目的とする。

## 動機

### この機能/変更がなぜ必要か？

YaoXiang はすでに所有権モデル（RFC-009）と並行モデル（RFC-001）を 통해メモリ安全とスレッド安全を保証しているが、論理的正当性はテストに依存している。システムプログラミング、安全Critical分野（宇宙、金融、OSカーネルなど）では、論理的エラーが壊滅的な結果を招く可能性がある。既存のソリューション（Rust の借用チェックなど）はこの種のエラーを検出できない。霍尔論理は数学的な証明手段を提供するが、従来の形式検証ツールは独立した仕様言語と複雑な学習曲線を必要とすることが多い。

### 現在抱えている問題

- 論理的正当性はテストでのみ検証でき、コンパイル時に保証できない
- Criticalシステムコードが形式検証手段を欠いている
- 既存の形式検証ツールは学習曲線が急で、主流プログラミング言語から切り離されている

## 提案

### 中核設計

我々の目標は、**軽量で言語と一体化した**静的検証ソリューションを設計することである：

- **Debug Build で検証必須**: 高信頼性が必要なモジュールで開発者が仕様を記述し、Debug Build 时に强制的に検証を行い、验证成功后のみ Release Build に進む
- **構文がエレガント**: `//!` コメントを使用し、新しいキーワードを導入せず、エディタの色分けで識別可能
- **型システムとの融合**: 仕様が型の一部となり、型チェックに参加でき、用户が定義した仕様型をサポート
- **証明可能かつテスト可能**: 静的に証明することも、运行时アサーションに降格することもでき、段階的な採用が可能

### 1. 仕様コメント構文

関数本体またはループ本体の先頭で、`//!`（単一行）または `/*! ... !*/`（複数行）を使用して仕様を記述する。

#### 1.1 統合仕様構文

仕様は YaoXiang の統合的な `name: Type = expression` 構文モデルを採用し、型システムと完全に統合されている：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0
    //! ensures: GreaterOrEqual(result, arr[0..n])
    //! ensures: ExistsMax(result, arr[0..n])
    // 実装...
}
```

- 仕様本质上是**型宣言**であり、右辺はブール式である
- 左辺は仕様型のインスタンス（型パラメータ付き）
- 特別な変数 `result` を使用して返回值を表すことができる

#### 1.2 ループ仕様

```yaoxiang
while i < n {
    /*! invariant: Bounds[i, n] = 0 <= i <= n
                  && SumInvariant[s, arr[0..i]] !*/
    s = s + arr[i]
    i = i + 1
}
```

#### 1.3 仕様式

仕様右辺のブール式は YaoXiang 式構文を使用し、以下をサポート：
- 算術演算、比較演算、論理演算
- 量詞：`forall i in 0..n: P(i)`、`exists i in 0..n: P(i)` — 言語組み込みの論理構成要素
- 関数呼び出し（純関数でなければならない）

### 2. 仕様型システム

仕様型は本质上 YaoXiang の通常の型であり、統合構文モデルと完全に一致する。

#### 2.1 組み込み仕様型

コンパイラは以下の一般的な仕様型を組み込んでいる（仕様で直接使用可能）：

```yaoxiang
// 組み込み仕様型定義
NonEmpty: (T: Type) -> Type = { len: T; len > 0 }
Positive: Type = { x: Int; x > 0 }
GreaterOrEqual: (T: Type) -> Type = { result: T, arr: Array(T); result >= arr[0] && forall i in 1..arr.len: result >= arr[i] }
Bounds: (T: Type) -> Type = { i: T, n: T; 0 <= i && i <= n }
SumInvariant: (T: Type) -> Type = { s: T, arr: Array(T); s == sum(arr[0..i]) }

// 量詞構成（言語組み込み、関数ではない）
forall: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
exists: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
```

#### 2.2 ユーザー定義仕様型

通常の型定義と完全に一致し、ユーザーが仕様型を定義できる：

```yaoxiang
// 正整数仕様の定義
Positive: Type = { x: Int; x > 0 }

// ソート済み配列仕様の定義
Sorted: (T: Ord) -> Type = {
    arr: Array(T);
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

// 最大値仕様の定義
ExistsMax: (T: Ord) -> Type = {
    result: T, arr: Array(T);
    exists i in 0..arr.len: result == arr[i]
    && forall j in 0..arr.len: result >= arr[j]
}
```

カスタム仕様の使用：

```yaoxiang
sqrt: (x: Positive) -> Float = {
    //! ensures: SquareRootResult(result, x) = result * result <= x && (result+1)*(result+1) > x
    // 実装...
}

binary_search: (T: Ord) -> ((arr: Sorted(Array(T)), key: T) -> Option(Index)) = {
    //! ensures: SearchResult(result, arr, key)
    // 実装...
}
```

仕様型は他の型一样に、ジェネリックパラメータ、型制約、支持し、型推論に参加できる。


### 3. コンパイルモード

| モード | 動作 | オプション |
|------|------|------|
| **Debug Build** | 仕様を解析し、検証条件を生成し、SMTソルバーを呼び出して証明；検証成功后のみ Release Build に進む | `yaoxiangc --debug source.yx` |
| **Release Build** | すべての `//!` コメントを無視し、コードを生成しない；すべての検証キャッシュをクリア；アグレッシブ最適化を有効化 | `yaoxiangc --release source.yx` |
| **Runtime Checks** | 仕様をRuntimeアサーションに変換し、違反時に panic | `yaoxiangc --enable-runtime-checks source.yx` |

検証モードでは、証明に失敗した場合、コンパイラはエラーを報告し、可能な反例（入力値など）を提供する。

### 4. 検証メカニズム

コンパイラは仕様を検証条件（Verification Conditions）に変換し、統合された SMT ソルバー（Z3 など）に送信する。検証プロセスは概ね以下の通り：

1. 関数の `requires` と `ensures`、ループの `invariant` を収集する
2. 各ループに対して、ループ不変式証明義務を生成する：ループ 진입前に成立すること、各反復後に維持されること、ループ退出後に事後条件を蕴含すること
3. 関数本体を論理式に変換し、仕様と組み合わせ、検証条件を 形成する
4. SMT ソルバーを呼び出して充足可能性をチェックする

ソルバーが `unsat`（充足不能）を返した場合、仕様は成立する。そうでなければ、反例を報告する。

### 5. テストとの組み合わせ

Runtime Checks モードでは、仕様をアサーションに変換してテストに使用できる。仕様カバレッジツールと組み合わせることで、テストによる仕様のカバー度を評価できる。将来的には、テストから自動的に候補仕様を推断する仕様マイニングツールの導入も検討できる。

### 6. エディタサポート

`//!` と `/*! ... !*/` はエディタに認識され、特殊コメントとして異なる色（紫色など）が割り当てられ、通常のコメントと区別される。Language Server は仕様のホバー補完、完全な入力補完、検証エラー報告を提供できる。

## 詳細設計

### 構文変更

| 変更前 | 変更後 |
|------|------|
| 仕様コメント構文なし | `//!` と `/*! ... !*/` 仕様コメントを許可 |

### 7.1 構文拡張

既存の構文（RFC-010）を基础上、関数本体とループ本体の先頭に0行以上の `//!` または `/*! ... !*/` コメント的出现を許可する。仕様構文は統合型構文と一致する：

```
spec_comment     ::= ('//!' spec_line) | ('/*!' spec_block '!*/')
spec_line        ::= spec_name ':' type_expr '=' expr
spec_name        ::= 'requires' | 'ensures' | 'invariant'
spec_block       ::= (spec_name ':' type_expr '=' expr ';')*
```

- 仕様本质上是型宣言：`仕様名: 仕様型 = ブール式`
- `type_expr` は仕様型式であり、型パラメータを持てる
- `expr` は YaoXiang 式構文を使用し、量詞をサポート

### 7.2 型チェック

検証モードでは、コンパイラは仕様コメントを対応する仕様型インスタンスに変換し、関数またはループのメタデータに記録する。

### 7.3 検証条件生成

 weakest precondition または strongest postcondition 計算を使用し、ループ不変式と組み合わせ、一階述語論理式を生成する。生成された VC は SMT-LIB 形式を使用し、外部ソルバーを呼び出す。

### 7.4 エラー報告

証明に失敗した場合、ソルバーはモデル（反例）を提供する可能性がある。コンパイラはこれらの反例を 읽기 쉬운形式に変換する必要があります（例：具体的な入力値），ユーザーがデバッグしやすいようにする。

### 7.5 Runtime Checks

`--enable-runtime-checks` モードでは、コンパイラは仕様を `assert` 文に変換する：
- `requires`：関数 진입前に `assert(cond)` を挿入する
- `ensures`：関数のすべての返回点に先に `assert(cond)` を挿入する（`result` は実際の返回值に置換）
- `invariant`：ループ本体 先頭に `assert(cond)` を挿入する

### 7.6 既存設計との統合

- **所有権モデル**：仕様中の式は所有権ルールに従い、読み取りのみ可能（純関数）、副作用を避ける
- **ジェネリックシステム**：仕様型はジェネリックパラメータ（例：`Requires(P)`）をサポートし、ジェネリック関数/型と組み合わせ可能
- **依存型**：仕様中の値依存型（配列の長さ `n` など）が自然に利用可能

### 型システムへの影響

- 仕様型は YaoXiang の通常の型であり、統合構文モデルと一致する
- コンパイラは一般的な仕様型（`Positive`、`NonEmpty`、`GreaterOrEqual` など）を組み込んでいる
- ユーザーは通常の型定義でカスタム仕様型を定義できる
- 仕様型はジェネリックパラメータを持て、型制約をサポートする

### 実行時動作

- **Debug Build**: SMT ソルバーを呼び出して静的検証を行い、コンパイル時間が增加する；検証成功后、検証結果をキャッシュする
- **Release Build**: 仕様コメントが無視され、Runtime オーバーヘッド为零；すべての検証キャッシュをクリア；Span キャッシュクリアなどのアグレッシブ最適化を有効化
- **Runtime Checks モード**: assert 文を生成し、Runtime で違反を検出する

### コンパイラ変更点

- パーサー：仕様コメント構文を識別する
- 意味解析：仕様を収集し、仕様型に変換する
- 検証バックエンド：検証条件を生成し、SMT ソルバーを呼び出す
- コード生成：Runtime Checks モードをサポートする

### 後方互換性

- ✅ 完全な後方互換性
- 通常のコンパイルでは仕様コメントが無視され、既存コードに影響しない
- Release Build では仕様が無視され、追加オーバーヘッドがない

## トレードオフ

### 利点

- **Debug Build 検証**: Debug Build 时に强制検証を行い、論理的正当性を保証
- **構文がエレガント**: 純粋なコメントで、新しいキーワードを導入せず、エディタに優しい
- **型システムとの融合**: 仕様即ち型で、拡張可能
- **段階的採用**: Runtime Checks から静的検証への段階的移行が可能
- **信頼性向上**: テストでは発見しにくい論理的エラーを検出可能

### 欠点

- **コンパイル時間**: 検証モードではコンパイル時間が著しく 增加する可能性がある
- **学習曲線**: 効果的な仕様と量詞の記述方法を学ぶ必要がある
- **SMT ソルバーの限界**: 一部の複雑な性質を自動的に証明できない場合がある

## 代替案

| 方案 | 利点 | 欠点 |
|------|------|------|
| 新キーワード（`requires` など） | 構文が直感的 | 新キーワードの導入により簡潔さが崩れる |
| 独立した仕様ファイル（CVL など） | 仕様とコードが分离 | ファイル数が増加し、同期が困難 |
| Runtime アサーションのみ | 実装が単純 | 静的保証がない |
| **本提案（コメント+仕様型）** | 簡潔さと機能のバランス | エディタサポートが必要 |

## 実装戦略

### 段階的計画

| 段階 | 内容 |
|------|------|
| **段階 1：基本サポート** | パーサーを拡張し、`//!` と `/*! ... !*/` コメントを識別して AST ノードに添付；検証モードで仕様を収集し、簡単な検証条件（算術比較のみ）を生成；Z3 ソルバーを統合 |
| **段階 2：量詞サポート** | 量詞式をサポートし、SMT-LIB の `forall`/`exists` に変換；仕様の IDE ハイライトとホバー補完を提供 |
| **段階 3：最適化とツールチェーン** | 増分検証、検証済みモジュールのキャッシュ；仕様カバレッジレポート；仕様マイニングツール（テストから候補仕様を生成） |

### 依存関係

- RFC-009: 所有権モデル - 仕様式には純関数セマンティクスが必要
- RFC-010: 統合型構文 - 仕様型システムは型システムに基づく
- RFC-011: ジェネリック型システム設計 - 仕様型はジェネリックパラメータをサポート

### リスク

1. **SMT ソルバー統合の複雑性**: Z3 などのソルバーの統合には技術的課題が発生する可能性がある
   - 軽減：成熟した Rust Z3 バインディングを使用し、サポート対象の式类型を段階的に拡張する

2. **検証失敗時のデバッグの困難さ**: SMT ソルバーが仕様を证明できない場合、ユーザーが理由を理解しにくい
   - 軽減：明確なエラー情報と反例の説明を提供する

3. **性能オーバーヘッド**: 検証モードではコンパイル時間が著しく 增加する可能性がある
   - 軽減：増分検証とキャッシュメカニズムを実装する
