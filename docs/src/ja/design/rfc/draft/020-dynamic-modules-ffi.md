## 摘要

本ドキュメントはRFC-001、008、018を踏まえ、YaoXiangの並発モデルを**動的モジュールロード**、**外部関数インターフェース（FFI）**、**より精细なスケジューリング最適化**といった実利用場面に対応できるよう、さらに细化・拡張したものである。核となる設計は以下の通り：

1. **動的モジュールのメタデータ契約**：同一言語で書かれた動的ライブラリに対しコンパイル時の依存記述を提供し、メインプログラムが静的DAGを構築的同时に、透明な並発を維持する。
2. **FFI分层処理**：外部コードはデフォルトで同期（`@block`）に降格され、開発者はアノテーションでメタデータを提供することで徐々に並発スケジューリングに巻き込める。外部の並発は外部言語側で管理する。
3. **呼び出しコンテキストに基づく最適化**：静的しきい値へのフォールバックに代わり、コンパイラは関数呼び出しがDAG内で果たす実際の役割（消費者数、副作用など）に基づいて、その関数をインライン展開するかどうかをインテリジェントに判断する。
4. **制御流とDAGのマージメカニズム**：`if`、`loop`などの動的構造をPhiノードと動的展開を通じて、データフローグラフに自然に融合させる。
5. **ランタイムスケジューラのメモリとパフォーマンス最適化**：ノードのライフサイクル管理、アリーナアロケーション、ロックフリースキューなどの低コスト抽象化の実装を明確化する。

本ドキュメントは、YaoXiangの並発モデルが静的全プログラム分析に対応するとともに、動的性质や外部との相互作用にも柔軟に対応でき、高パフォーマンスと開発体験を維持できることを保証するものである。

## 動機

### 既存設計の不足

RFC-001/008/018は洗練された透明な並発モデルを构筑しているが、現実感のある要件に対しては依然として盲点が存在する：

- **動的モジュール**：プログラムがプラグインや動的リンクライブラリをサポートしている場合、メインプログラムはコンパイル時にモジュール内部の呼び出し関係や依存関係を把握できず、グローバルなDAG構築が失敗する。
- **FFI呼び出し**：外部関数（Cライブラリなど）は完全なブラックボックスであり、内部に並発、ブロッキング、副作用が含まれる可能性がある。これを通常のノードと同様に扱うと並発の安全性が壊れる。
- **小関数のスケジューリングオーバーヘッド**：RFC-001で提唱された「L1自動フォールバック」は静的しきい値（命令数 <50）に基づいており、この暗黙的なルールは開発者の行動予測を困难にし、複雑な呼び出しコンテキストに対応できない。
- **制御流とDAGの融合**：`if`、`loop`などの動的構造がDAG内でどのように表現されるかが不明確であり、依存分析の正確性に 영향을 미칠 수 있다。
- **ランタイムオーバーヘッドの管理**：DAGノード数が増加するにつれ、スケジューラのメモリ管理とパフォーマンス最適化を明確化する必要がある。

### 目標

- 透明な並発の核心理念を維持しながら、動的モジュールとFFIに明確で安全、段階的なサポートを提供する。
- スケジューリング最適化を「暗黙的なグローバルルール」から「コンテキストベースのインテリジェントな判断」に转变させ、予測可能性とパフォーマンスを向上させる。
- 動的制御流れのDAG表現を完善させ、すべてのプログラム構造がデータフローモデルに自然に融合することを保証する。
- スケジューラのメモリ管理とパフォーマンス最適化戦略を明確化し、低コストな抽象化を実現する。

## 提案

### 1. 動的モジュールのメタデータ契約

#### 1.1 契約内容

YaoXiangでコンパイルされた各動的ライブラリ（`.yxo` / プラットフォーム固有の動的ライブラリ）には必ず**メタデータ記述ファイル**（`.yxmeta`）が添付され、以下の内容を含める：

- **エクスポート関数リスト**：各関数の完全な型シグネチャ（引数、戻り値、リソースタグ）。
- **副作用タグ**：コンパイラが自動導出する `@pure` / `@io`（開発者が明示的にオーバーライド也可能）。
- **リソース依存**：各引数がリソース型かどうか（例：`File`）、戻り値に新しいリソースが含まれるかどうか。
- **呼び出しグラフの要約**（オプション）：この関数が呼び出す可能性のある他のエクスポート関数のID。跨モジュール循環依存の検出に使用。
- **所有権情報**：引数の所有権セマンティクス（借用/移動）、戻り値の所有権。
- **並発安全性**：`Send`/`Sync`の自動導出結果。

メタデータフォーマットはバイナリまたは構造化テキスト（MessagePackなど）を使用し、パース効率を保証する。

#### 1.2 コンパイル時処理

メインプログラムのコンパイル時、動的モジュール関数の呼び出しに出会った場合：

1. 対応するモジュールの `.yxmeta` ファイルを読み込む。
2. グローバルDAG内に**プレースホルダノード**を作成し、メタデータから得た入出力依存、副作用タグなどを記録する。
3. プレースホルダノードは通常のノードと同様に依存分析に参加し、スケジューラは事前に実行順序を計画できる。

#### 1.3 ランタイムバインディング

動的モジュールのロード時：

- ランタイムは実際の関数シグネチャとメタデータの一致を検証する（バージョン不一致を防止）。
- プレースホルダノードを実際の関数ポインタにバインディングする。

**サブグラフスケジューリングのセマンティクスについて**：動的モジュール内に独立したサブDAGがある場合（例如ば、モジュール自体が並発ロジックを含んでいる場合）、そのサブグラフは**独立したスケジューリングユニット**として実行される。その境界はモジュールのエクスポート関数によって定義される：エクスポート関数を呼び出すとき、サブグラフは全体として実行を開始し、その関数が戻るまで続く。サブグラフ内部のノードスケジューリングはサブグラフ自身のスケジューラが担当する（モジュール内部では標準スケジューラを継続使用可能）が、サブグラフとメインDAGのやり取りは入出力データフローのみに限定される——メインDAG内のプレースホルダノードはサブグラフの開始と終了のみを気にし、内部スケジューリングには介入しない。この設計により、モジュールのカプセル化が保証される的同时に、メインDAGは静的な完全性を维持する。

#### 1.4 安全保证

- 動的モジュールが契約に違反した場合（例如 `@pure` を宣言していながらグローバルな状態を修正した場合）、結果は開発者の責任となる（FFIのunsafe境界 подобно）。ただし同一言語であるため、実行時チェック（例如メモリ隔離）によって安全性を強化することは可能だが、オーバーヘッドが増加する。
- 跨モジュール循環依存：モジュールAがBを呼び、BがAを呼び、かつメタデータに関数呼び出し関係が宣言されている場合、コンパイラは检测してエラーにできる；宣言されていない場合、実行時にデッドロックが発生する可能性がある。

### 2. FFI分层処理

#### 2.1 默认降格

`extern` を通じて導入された外部関数はデフォルトで `@block` 同期呼び出しとして扱われる：

- 呼び出しはDAGスケジューリングに入らず、現在のスレッドで直接実行される。
- 実行期間中にYaoXiangスケジューラはその内部の並発に介入しない。
- 戻り値は使用可能だが、呼び出し自体は依存分析に参加しない。

> **⚠️ 開発者への注意**：FFI呼び出しはデフォルトでDAGスケジューリングに参加しない。因此为其内部に存在する並発（例如Cライブラリが開始するスレッド）は完全に外部言語側で管理され、YaoXiangは追踪できない。FFI呼び出しを並発スケジューリングに巻き込みたい場合、アノテーションでメタデータを提供する必要がある（2.2節を参照）。「同期コードだから自動並発する」という假设は하지 마라——FFI呼び出しは自動的に並列化されない。

#### 2.2 オプションメタデータ注釈

開発者はアノテーションを通じてより精确なメタデータを提供でき、FFI呼び出しを徐々に並発スケジューリングに巻き込める：

```yaoxiang
@pure   // 純粋関数、副作用なし、自由なスケジューリング可能
extern "C" fn sin(x: Double) -> Double

@io     // I/O関連、リソース依存で直列化が必要
extern "C" fn write(fd: i32, buf: *u8, count: usize) -> isize

@block  // 明示的な同期宣言（デフォルト、省略可能）
extern "C" fn old_api() -> i32
```

- `@pure` でマークされ、パラメータがリソース変数を含まない場合、通常のDAGノードとして扱われ、並列実行が可能。
- `@io` でマークされ、パラメータにリソース型が含まれる場合（例：ラップされた `File`）、自動的に依存関係が確立される。

#### 2.3 リソースラッパー

標準ライブラリでは、一般的なFFIリソース（ファイルディスクリプタ、データベース接続など）にラッパー型を提供し、生のリソースをYaoXiangリソース型にラップして、DAG依存分析を活用することを推奨する。例如：

```yaoxiang
struct File { fd: i32 }  // リソース型としてマーク
impl File {
    @io
    fn write(self: &mut File, data: &[u8]) -> Result { ... }
}
```

これにより、同じ `File` 変数への複数の `write` 呼び出しは自動的に直列化される。

#### 2.4 外部の並発管理

FFI関数が内部でスレッドや非同期操作を開始した場合、それは完全に外部言語側の責任となる。YaoXiangは呼び出しの開始と終了のみを気にし、内部詳細は追踪しない。開発者はこれらの並発操作がYaoXiangのメモリモデルと衝突しないことを確認する必要がある（たとえばコールバック時の所有権とSend制約に注意）。

### 3. 呼び出しコンテキストに基づく最適化

RFC-001の「L1自動フォールバック」静的しきい値に代わり、**コンパイラが各呼び出し地点のDAGにおける実際のコンテキストに基づいてインテリジェントに判断**する。

#### 3.1 最適化判断の根拠

コンパイラは各関数呼び出しノードを分析する：

- **消費者数**：そのノードの結果が何個のダウンストリームノードで使用されるか。1つであればインライン展開の候補となり、1つより多ければ結果共有のために独立ノードとして保持する必要がある。
- **副作用**：ノードが `@io` 副作用を持つ場合、順序を保証するために独立ノードとして保持する必要がある。
- **計算量の見積もり**：命令数などのヒューリスティック依旧是参照可能だが，硬性しきい値としてではなく、インライン展開の収益評価に使用する。
- **リソース依存**：ノードがリソース変数（例：`File`）を扱い、そのリソース変数がアップストリームとダウンストリーム間で渡される場合、インライン展開は依存チェーンを壊す可能性があり、慎重する必要がある。

#### 3.2 インライン展開

インライン展開と判断された場合：

- そのノードの計算ロジックを唯一のダウンストリームノードのコードに直接埋め込む。
- DAGからそのノードを移除し、入力は直接ダウンストリームノードの入力となる。
- 最終的なコード生成時、インライン展開された関数は独立したスケジューリングユニットを生成しない。

#### 3.3 インライン展開の制限

- 再帰関数またはループ内の呼び出しは通常インライン展開しない（無限展開を防止）。
- モジュール境界（動的モジュール、FFI）を跨ぐ関数はインライン展開しない。
- 開発者は `@noinline` アノテーションでインライン展開を明示的に禁止したり、`@forceinline` でコンパイラにインライン展開を試みるようヒントを与えられる。

#### 3.4 観察可能性

コンパイラは最適化レポートを生成する（`--emit-optimization-report` で有効化可能）：

- **各インライン展開点**：インライン展開された関数名、呼び出し位置、インライン展開の理由（例：「唯一の消費者かつ純粋関数」）。
- **独立ノードとして保持の理由**：「複数の消費者あり」「副作用あり」「跨モジュール呼び出し」など。
- **判断統計**：インライン展開総数、保持ノード数、最適化効果の評価に使用。

レポートの出力フォーマットはテキストまたはJSONとし、ツールによる解析を可能にする。

### 4. 制御流とDAGのマージ

#### 4.1 条件分岐（if）の処理

分岐マージ点を表すために **Phiノード**（SSA形式）を導入する：

- コンパイル時に各 `if` 式に対して以下を構築する：
  - 2つの分岐サブDAG（それぞれ `then` と `else` に対応）。
  - Phiノード。その入力には条件変数と2つの分岐の出力が含まれる。
- Phiノードのセマンティクス：条件変数が準備できたら、条件値に基づいて対応する分岐の出力を選択的就是自己的出力。
- ランタイム時、Phiノードは条件変数に依存する；条件が準備できたら、自身を選択した分岐のダウンストリームリストに動的に追加し、その分岐の結果を待つ。

DAG示例：
```
        cond
       /    \
  then DAG  else DAG
       \    /
        Phi
         |
      后续ノード
```

#### 4.2 ループ（loop/while）の処理

ループはフィードバックエッジ付きのサブDAGとして扱い、ランタイム時に**必要に応じて展開**する：

- コンパイル時にループ体を識別し、**ループテンプレート**を構築する：
  - 条件ノード。
  - ループボディサブDAG。
  - イテレーション間で 전달される状態変数。
- ランタイム時、ループ結果が必要な場合（例如ループ終了後に累積値を使用）、スケジューラは動的にイテレーションの展開を開始する：
  1. 初めて条件ノードをスケジューリングし、真であれば最初のイテレーションのサブDAGをインスタンス化。その入力には初期状態と外部変数が含まれる。
  2. イテレーション完了後に新しい状態が生成され、条件ノードを再度スケジューリング（新状態に依存）して継続するかどうかを決定する。
  3. 条件が偽になるまで繰り返し、最後のイテレーションの出力がループ結果となる。

**複雑な示例：ループ条件がループボディ内部の更新に依存**
```yaoxiang
let mut x = 0
while x < 10 {
    x = compute(x)  // xはループボディ内で更新される
}
```
このパターンでは、条件ノード `x < 10` は各イテレーション後に更新される `x` に依存する。DAGは以下の通り：
- ループテンプレートには状態変数 `x` が含まれ、初期値は0。
- 各イテレーション：首先条件ノードを実行（現在の `x` に依存）、真であれば `x = compute(x)` を実行して新しい `x` を生成し、然后再度条件ノードに戻る。
- ランタイムは上述のプロセスで動的に展開し、条件が偽になるまで続ける。

イテレーション間の依存は状態変数を通じて自然にデータフローを形成し、依存関係のあるイテレーションは自動的に直列化され、依存関係のないイテレーションは並列可能（例：`map`）。

#### 4.3 無限ループの特別な処理

单个の無限ループはメインDAGとして直接同期実行（スケジューリングオーバーヘッドなし）；複数の無限ループはバックグラウンドDAGとして、スケジューラのタイムスライスで並列実行される。

### 5. ランタイムスケジューラのメモリとパフォーマンス最適化

#### 5.1 ノードのライフサイクル管理

- 各ノードは**参照カウンタ**（アトミック変数）を维持し、その結果を依存する消費者の数を表す。
- ノードが完了して結果をすべてのダウンストリームに渡した後、参照カウンタがゼロになれば、そのノードのメモリを解放できる。
- 結果値自体も参照カウンタを使用（`Arc<T>`）するが、优化可能：結果が1つの消費者のみ使用する場合、所有権を直接移動し、カウンタオーバーヘッドを避ける。

#### 5.2 アリーナメモリアロケーション（Arena）

動的に生成された大量の短寿命ノード（例如ループイテレーション）に対して、**アリーナアロケータ**を使用：

- 1回のループ展開に対してメモリアリーナを割り当てる。
- アリーナ内のノードは連続的に割り当てられ、全体で一度に解放されるため、フラグメンテーションと解放オーバーヘッドが减少する。
- アリーナ終了時、すべてのノードメモリを一度に回收する。

#### 5.3 ロックフリーデータ構造

- 依存カウンタ：`AtomicUsize` を使用し、`fetch_sub` でアトミックに递减。
- レディキュー：Chase-Lev 双方向キュー（各スレッドのローカルキュー + ワークスティール）を使用し、ロック競合を减少。
- ダウンストリームリスト：作成後は読み取り専用のため、并发修改を避ける。

#### 5.4 アダプティブスケジューリング

- システムに無限ループが1つしかない場合、直接同期実行し、ゼロスケジューリングオーバーヘッド。
- タスク粒度とシステム负载に基づき、パラメータ度を動的に調整（例如キュー長を監視してワーカースレッド数を調整）。

#### 5.5 低コスト抽象化の原則

すべてのスケジューリングオーバーヘッドはタスク数に比例し、各タスクの追加オーバーヘッド（作成、エンキュー、依存処理）は数十ナノ秒级别に抑制する。超细粒度タスクについては、インライン展開最適化（セクション3）を 통해スケジューリングを避ける。

## 詳細設計

### 6.1 動的モジュールのメタデータフォーマット（草案）

```rust
// メタデータファイル構造（簡略化）
struct Metadata {
    version: u32,
    functions: Vec<FuncMeta>,
}

struct FuncMeta {
    name: String,
    signature: TypeSignature,
    effects: EffectTag,      // Pure | IO | Block
    resource_params: Vec<usize>, // 引数インデックスリスト，哪些引数がリソース型か
    calls: Vec<String>,       // 呼び出す他のエクスポート関数名（オプション）
    ownership: OwnershipInfo,
    send_sync: SendSync,      // Send/Sync滿足是否
}
```

### 6.2 スケジューラのコアデータ構造

```rust
struct TaskNode {
    id: TaskId,
    deps: Vec<TaskId>,                // アップストリーム依存
    remaining_deps: AtomicUsize,
    inputs: Vec<Option<Value>>,
    result: Option<Value>,
    func: Executable,
    downstream: Vec<TaskId>,           // ダウンストリームノード（作成後は読み取り専用）
    effect: EffectTag,
    arena_id: Option<ArenaId>,         // 所属アリーナ（オプション）
}

struct Scheduler {
    ready_queues: PerThreadQueue<TaskId>, // 各スレッドのローカルキュー
    global_work_stealer: WorkStealer,
    arenas: ArenaAllocator,               // アリーナアロケータ
}
```

### 6.3 コンテキストベースの最適化分析

コンパイラはMIR層で以下のステップを実行する：

1. グローバル呼び出しグラフとデータ依存グラフを構築する。
2. 各関数呼び出しノードに対して、その出次数（消費者数）を計算する。
3. 出次数が1で、関数が副作用なし（`@pure`）で、非再帰的であれば、「インライン展開候補」としてマークする。
4. ヒューリスティック（例如命令数）と組み合わせてインライン展開の収益を評価し、インライン展開を決定する。
5. インライン展開時、呼び出しノードのコードをそのダウンストリームに埋め込み、依存関係を更新する。

### 6.4 制御流れノードの表現

```rust
enum NodeKind {
    Normal(FuncId),
    Phi { cond: TaskId, then_branch: TaskId, else_branch: TaskId },
    LoopTemplate { cond: FuncId, body: FuncId, state_var: VarId },
    // ...
}
```

ランタイム動的展開時、`LoopTemplate` は一連の `Normal` ノードインスタンスを生成する。

## 权衡

### メリット

- **動的モジュールの 안전한統合**：メタデータ契約により、動的ライブラリはメインプログラムとシームレスに並発モデルを共有でき、同時に静的DAGの完全性を維持する。
- **FFIの段階的な統合**：開発者は外部関数に徐々にアノテーションを追加でき、安全な降格から効率的な並発への移行が可能。
- **最適化の結果の予測可能性**：コンテキストベースの判断により暗黙的なしきい値を置き換え、動作が透明になり、開発者はツールで最適化を理解できる。
- **制御流れの自然な融合**：Phiノードと動的展開により、DAGはすべてのプログラム構造を表現でき、特殊構文が不要。
- **パフォーマンスのスケーラビリティ**：アリーナアロケーション、ロックフリーキューなどの設計により、スケジューラは大規模な並発に対応できる。

### デメリット

- **メタデータ契約增加的コンパイルの複雑さ**：動的ライブラリに対してメタデータを生成・パースする必要がありツールチェーンの対応が必要。
- **FFIアノテーション开发者の正しさに依存**：誤ったラベル付けはデータ競合を引き起こす可能性があり、ドキュメントとツールヒントでリスクを軽減する必要がある。
- **コンテキスト最適化分析の時間**：グローバル分析はコンパイル時間を增加させる可能性があるが、增量コンパイルで缓解可能。
- **動的展開增加的ランタイムオーバーヘッド**：ループ展開は動的にノードを作成する必要があるが、アリーナアロケーションで缓解可能。

## 実装戦略

### フェーズ划分（含 PRIORITY提案）

> **実装PRIORITY提案**：初期段階では完璧な実装を追求せず、简单な手でシステムを動作させ、逐渐的に最適化することが望ましい。例如：
> - 参照カウンタは直接 `Arc` を使用する。
> - ロックフリーキューは成熟したライブラリ（例：crossbeamのdeque）を使用。
> - アリーナアロケーションは最初は単純なbump allocatorを使用し、後で最適化する。

#### フェーズ1：基本サポート（v0.7）
- [ ] FFIのデフォルト降格を `@block` として実装する。
- [ ] FFIが使用する `@pure`、`@io` アノテーションを追加する。
- [ ] リソースラッパー型（例：`File`）とその基本メソッドを実装する。

#### フェーズ2：動的モジュールのメタデータ（v0.8）
- [ ] メタデータフォーマットを設計し、コンパイラが動的ライブラリ用に `.yxmeta` を生成するよう修正する。
- [ ] メインプログラムのコンパイル時にメタデータを読み込み、プレースホルダノードを作成する機構を実装する。
- [ ] ランタイムバインディング機構を実装する。

#### フェーズ3：コンテキスト最適化（v0.9）
- [ ] 呼び出しグラフ分析を実装し、ノードの出次数を計算する。
- [ ] インライン展開判断とコード生成サポートを追加する。
- [ ] 最適化レポート出力（インライン展開点、理由など）を実装する。

#### フェーズ4：制御流れDAG融合（v0.10）
- [ ] Phiノードと条件分岐のコンパイル時表現を実装する。
- [ ] ループテンプレートと動的展開ランタイムを実装する。
- [ ] 無限ループのバックグラウンドスケジューリングを完善する。

#### フェーズ5：パフォーマンス最適化（v1.0）
- [ ] アリーナアロケータを実装する。
- [ ] ロックフリーキューとワークスティールを最適化する。
- [ ] ベンチマークテストと 튜닝。
