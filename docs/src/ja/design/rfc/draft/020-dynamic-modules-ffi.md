# RFC-020：動的モジュールとFFI統合

> **状態**: 草案

## 摘要

本文書はRFC-001、008、018に基づき、YaoXiangの並行モデルをさらなる微細化と拡張を行い、**動的モジュールのロード**、**外部関数インターフェース（FFI）**、および**より精细なスケジューリング最適化**などの現実的なシナリオに対応することを目指す。コア設計は以下のとおりである：

1. **動的モジュールメタデータ契約**：同一言語で書かれた動的ライブラリに対してコンパイル時の依存関係記述を提供し、メイン�プログラムが透明な並行性を維持しながら静的にDAGを構築できるようにする。
2. **FFIレイヤー処理**：外部コードはデフォルトで同期（`@block`）に降格され、開発者はアノテーションを通じてメタデータを提供することで徐々に並行スケジューリングに組み込むことができ、外部の並行性は外部言語が自行管理する。
3. **呼び出しコンテキストベースの最適化**：静的閾値フォールバックに取って代わり、コンパイラは関数dagにおける実際の役割（消費者数、副作用など）に基づいてインライン化するか独立ノードとしてスケジューリングするかをインテリジェントに決定する。
4. **制御フローとDAGの統合メカニズム**：`if`、`loop`などの動的構造をPhiノードと動的展開を通じてデータフローグラフに自然に溶け込ませる。
5. **ランタイムスケジューラのメモリとパフォーマンス最適化**：ノードのライフサイクル管理、エリア割り当て、ロックフリーキューなど低コスト抽象化の実装を明確化する。

本文書は言語仕様を完善化し、YaoXiangの並行モデルが静的全プログラム分析に対応しながらも動的特性と外部 상호作用を柔軟に処理でき、パフォーマンスと開発体験を両立させることを目指す。

## 動機

### 既存設計の不足

RFC-001/008/018は優雅な透明な並行モデルを構築したが、現実世界のニーズに直面すると以下のBlind Spotが存在する：

- **動的モジュール**：プログラムがプラグインや動的リンクライブラリをサポートしている場合、メインペログラムはコンパイル時にモジュール内部の呼び出し関係と依存関係を知ることができず、グローバルDAG構築が失敗する。
- **FFI呼び出し**：外部関数（Cライブラリなど）は完全にブラックボックスであり、内部に並行性、ブロッキング、副作用が含まれる可能性があり、普通ノードとして扱うと並行安全性が破られる。
- **小関数のスケジューリングオーバーヘッド**：RFC-001で提案された"L1自動フォールバック"は静的閾値（命令数<50）を採用しており、この暗黙的なルールにより開発者は行動を予測しにくく、複雑な呼び出しコンテキストに対応できない。
- **制御フローとDAGの融合**：`if`、`loop`などの動的構造のDAGにおける表現がまだ明確ではなく、依存性分析の正確性に影響を与える可能性がある。
- **ランタイムオーバーヘッド制御**：DAGノード数が増加するにつれ、スケジューラのメモリ管理とパフォーマンス最適化を明確化する必要がある。ボトルネックを避けるため。

### 目標

- 透明な並行性のコア哲学を維持しながら、動的モジュールとFFIに明確で安全、段階的なサポートを提供する。
- スケジューリング最適化を「暗黙的なグローバルルール」から「コンテキストベースのインテリジェントな決定」に转变し、予測可能性とパフォーマンスを向上させる。
- DAGの動的制御フロー表現を完善し、すべてのプログラム構造がデータフローモデルに自然に溶け込めるようにする。
- スケジューラのメモリ管理とパフォーマンス最適化戦略を明確化し、低コスト抽象化を実現する。

## 提案

### 1. 動的モジュールメタデータ契約

#### 1.1 契約内容

YaoXiangでコンパイルされた各動的ライブラリ（`.yxo` / プラットフォーム固有の動的ライブラリ）には、**メタデータ記述ファイル**（`.yxmeta`）を添付しなければならない。内容包括：

- **エクスポート関数リスト**：各関数の完全な型シグネチャ（引数、戻り値、リソースタグ）。
- **副作用タグ**：コンパイラが自動導出する `@pure` / `@io`（開発者も明示的にオーバーライド可能）。
- **リソース依存**：各引数がリソースタイプかどうか（例：`File`）、戻り値に新リソースが含まれるかどうか。
- **呼び出しグラフサマリー**（オプション）：この関数が呼び出す可能性のある他のエクスポート関数のID。モジュール間循環依存検出用。
- **所有権情報**：引数の所有権セマンティクス（借用/移動）、戻り値の所有権。
- **並行安全性**：`Send`/`Sync`の自動導出結果。

メタデータフォーマットはバイナリまたは構造化テキスト（MessagePackなど）を採用し、解析効率を保証する。

#### 1.2 コンパイル時処理

メインペログラムのコンパイル時、動的モジュール関数の呼び出し箇所に遭遇した場合：

1. 対応モジュールの `.yxmeta` ファイルを読み込む。
2. グローバルDAGに**プレースホルダノード**を作成し、メタデータから 얻た入出力依存、副作用タグなどを記録する。
3. プレースホルダノードは普通ノードと同様に依存性分析に参加し、スケジューラは事前に実行順序を計画できる。

#### 1.3 ランタイムバインディング

動的モジュールのロード時：

- ランタイムは実際の関数シグネチャとメタデータの一致を検証する（バージョン不一致防止）。
- プレースホルダノードと実際の関数ポインタをバインディングする。

**サブグラフスケジューリングセマンティクスについて**：動的モジュール内に独立したサブDAGがある場合（例：モジュール自体が並行ロジックを含む）、そのサブグラフは**独立したスケジューリングユニット**として実行される。その境界はモジュールのエクスポート関数によって定義される：エクスポート関数を呼び出すとき、サブグラフは全体として実行を開始し、その関数が戻るまで続く。サブグラフ内部のノードスケジューリングはサブグラフ自身のスケジューラが担当する（モジュール内部では引き続き標準スケジューラを使用可能）が、サブグラフとメインDAGのやり取りは入出力データフローのみに限定される——メインDAG内のプレースホルダノードはサブグラフの開始と終了のみを関心とし、内部スケジューリングには介入しない。この設計により、モジュールのカプセル化が保証される同時に、メインDAGの静的な完全性が維持される。

#### 1.4 安全性の保証

- 動的モジュールが契約に違反した場合（例：`@pure`と主張しながらグローバル状態を修正）、結果は開発者の責任となる（FFIのunsafe境界と同様）。しかし同言語であるため、ランタイムチェック（メモリ隔離など）で安全性を強化することは可能だが、オーバーヘッドが増加する。
- モジュール間循環依存：モジュールAがBを呼び、BがまたAを呼び、元データに呼び出し関係が宣言されている場合、コンパイラは検出可能でエラーとなる；宣言されていない場合、ランタイムでデッドロックが発生する可能性がある。スケジューラが検出しpanicする。

### 2. FFIレイヤー処理

#### 2.1 デフォルト降格

外部関数（`extern` を通じて導入）はデフォルトで `@block` 同期呼び出しとして扱われる：

- 呼び出しはDAGスケジューリングに参加せず、現在のスレッドで直接実行される。
- 実行中 YaoXiang スケジューラはその内部並行性に介入しない。
- 戻り値は使用可能だが、呼び出し自体は依存性分析に参加しない。

> **⚠️ 開発者への注意**：FFI呼び出しはデフォルトでDAGスケジューリングに参加しないため、内部に存在する並行性（Cライブラリが開始するスレッドなど）は完全に外部言語が管理し、YaoXiangは追跡できない。FFI呼び出しを並行スケジューリングに組み込むしたい場合は、アノテーションを通じてメタデータを提供する必要がある（2.2節参照）。「同期コードが自動的に並行する」と想定しないでください——FFI呼び出しは自動的に並列化されません。

#### 2.2 オプションのメタデータアノテーション

開発者はアノテーションを通じてより正確なメタデータを提供し、FFI呼び出しを徐々に並行スケジューリングに組み込むことができる：

```yaoxiang
@pure   // 純粋関数、副作用なし、自由なスケジューリング可能
extern "C" fn sin(x: Double) -> Double

@io     // I/O関連、リソース依存で直列化が必要
extern "C" fn write(fd: i32, buf: *u8, count: usize) -> isize

@block  // 明示的に同期を宣言（デフォルト、省略可能）
extern "C" fn old_api() -> i32
```

- `@pure` としてマークされ、引数にリソース変数が含まれない場合、普通なDAGノードとして扱われ、並列実行できる。
- `@io` としてマークされ、引数にリソースタイプ（例：ラップされた `File`）が含まれる場合、自動的に依存エッジが確立される。

#### 2.3 リソースラップ

標準ライブラリには一般的なFFIリソース（ファイルディスクリプタ、データベース接続など）にラップタイプを提供し、生リソースをYaoXiangリソースタイプにカプセル化し、DAG依存性分析を活用することを推奨する。例：

```yaoxiang
struct File { fd: i32 }  // リソースタイプとしてマーク
impl File {
    @io
    fn write(self: &mut File, data: &[u8]) -> Result { ... }
}
```

同一 `File` 変数に対する複数の `write` 呼び出しは自動的に直列化される。

#### 2.4 外部並行管理

FFI関数が内部でスレッドを開始したり非同期操作を実行したりした場合、完全に外部言語が負責する。YaoXiangは呼び出しの開始と終了のみを関心とし、内部詳細は追跡しない。開発者はこれらの並行操作がYaoXiangのメモリモデルと冲突しないことを保証する必要がある（例：コールバック時に所有権とSend制約に注意）。

### 3. 呼び出しコンテキストベースの最適化

RFC-001の"L1自動フォールバック"静的閾値に取って代わり、**コンパイラは各呼び出しポイントでのDAGにおける実際のコンテキストに基づいてインテリジェントに決定**する。

#### 3.1 最適化決定の根拠

コンパイラは各関数呼び出しノードを分析する：

- **消費者数**：このノードの結果が何個の下流ノードに使用されるか。1の場合、インライン候補の資格がある；1より大きい場合、結果を共有するために独立ノードとして保持する必要がある。
- **副作用**：ノードが `@io` 副作用がある場合、順序を保証するために独立ノードとして保持する必要がある。
- **計算量推定**：命令数などのヒューリスティックを参考にすることはできるが、硬的閾値としては使用せず、インライン収益評価のみに使用。
- **リソース依存**：ノードがリソース変数（例：`File`）涉及し、そのリソース変数が上流下流間で渡される場合、インライン化は依存チェーンを破壊する可能性があり、慎重する必要がある。

#### 3.2 インライン操作

インライン化を決定した場合：

- そのノードの計算ロジックを唯一のダウンストリームノードのコードに直接埋め込む。
- DAGからそのノードを削除し、入力は直接ダウンストリームノードの入力となる。
- 最終コード生成時、インライン化された関数は独立したスケジューリングユニットを生成しない。

#### 3.3 インライン化の制限

- 再帰関数またはループボディ内の呼び出しは通常インライン化しない。無限展開を防ぐため。
- モジュール境界（動的モジュール、FFI）を越える関数はインライン化しない。
- 開発者は `@noinline` アノテーションを通じてインライン化を強制的に禁止したり、`@forceinline` でコンパイラにインライン化を試みるヒントを提供したりできる。

#### 3.4 可観測性

コンパイラは最適化レポートを生成する必要がある（`--emit-optimization-report` で有効化）。以下の情報を含む：

- **各インライン箇所**：インライン化された関数名、呼び出し位置、インライン化理由（例：「唯一の消費者かつ純粋関数」）。
- **独立ノードとして保持された理由**：例：「複数の消費者あり」「副作用を含む」「モジュール間呼び出し」など。
- **決定統計**：インライン総数、保持ノード数。開発者が最適化の効果の評価を支援。

レポート出力フォーマットはテキストまたはJSONのいずれかを使用し、ツール解析を容易にする。

### 4. 制御フローとDAGの合併

#### 4.1 条件分岐（if）の処理

**Phiノード**（SSA形式を参考）を導入して分岐合流点を表す：

- コンパイル時に各 `if` 式に対して以下を構築：
  - 2つの分岐サブDAG（それぞれ `then` と `else` に対応）。
  - 条件変数と2つの分岐の出力inputsとするPhiノード。
- Phiノードのセマンティクス：条件変数が準備できたら、条件値に基づいて対応する分岐の出力を自身の出力として選択する。
- ランタイム時、Phiノードは条件変数に依存する；条件が準備できたら、自身を選択した分岐の下流リストに動的に追加し、その分岐の結果を待つ。

示例DAG：
```
        cond
       /    \
  then DAG  else DAG
       \    /
        Phi
         |
      後続ノード
```

#### 4.2 ループ（loop/while）の処理

ループはフィードバックエッジを持つサブDAGとして扱い、ランタイムで**オンデマンド展開**を行う：

- コンパイル時、ループボディを識別し、**ループテンプレート**を構築。内容包括：
  - 条件ノード。
  - ループボディサブDAG。
  - 繰り返し間で渡される状態変数。
- ランタイム時、ループ結果が必要な場合（例：ループ終了後に累積値を使用）、スケジューラは動的にイテレーションを展開し始める：
  1. 初回、条件ノードをスケジューリング。真であれば最初のイテレーションのサブDAGをインスタンス化し、入力には初期状態と外部変数を含む。
  2. イテレーション完了後に新しい状態を生成し、条件ノードを再度スケジューリング（新状態に依存）して続行するかを決定。
  3. 条件が偽になるまで繰り返し、最後のイテレーションの出力がループ結果となる。

**複雑な例：ループ条件がループボディ内部の更新に依存**  
```yaoxiang
let mut x = 0
while x < 10 {
    x = compute(x)  // xはループボディ内で更新される
}
```
このパターンでは、条件ノード `x < 10` は各イテレーション後に更新される `x` に依存する。DAGの表現は以下のとおり：
- ループテンプレートには状態変数 `x` を含み、初期値は0。
- 各イテレーション：まず条件ノードを実行（現在の `x` に依存）、真であれば `x = compute(x)` を実行して新しい `x` を生成し、条件ノードに再度入る。
- ランタイムは上記のフローに従って動的に展開し、条件が偽になるまで続ける。

イテレーション間の依存関係は状態変数を通じて自然にデータフローを形成し、依存関係のあるイテレーションは自動的に直列化され、依存関係のないイテレーションは並列化可能（例：`map`）。

#### 4.3 無限ループの特別処理

単一の無限ループはメ인DAGとして直接同期実行（スケジューリングオーバーヘッドなし）；複数の無限ループはバックグラウンドDAGとして、スケジューラのタイムスライスで並行実行される。

### 5. ランタイムスケジューラのメモリとパフォーマンス最適化

#### 5.1 ノードのライフサイクル管理

- 各ノードは**参照カウント**（アトミック変数）を維持し、その結果に依存する消費者の数を表す。
- ノードが実行を完了し、結果がすべてのダウンストリームに渡された後、参照カウントがゼロになり、ノードのメモリを解放できる。
- 結果値自体も参照カウントを採用（`Arc<T>`）が最適化可能：結果が単一の消費者のみ使用される場合、所有権を直接移動し、カウントオーバーヘッドを回避。

#### 5.2 エリアメモリ割り当て（Arena）

動的に生成された短命ノード（ループイテレーションなど）に対しては、**エリアアロケータ**を使用：

- 1回のループ展開に対してメモリアリアを割り当て。
- エリア内のノードは連続的に割り当てられ、整体リリースで、断片化と解放オーバーヘッドを削減。
- エリア終了時、すべてのノードメモリを一度に回収。

#### 5.3 ロックフリーデータ構造

- 依存カウンタ：`AtomicUsize` を使用し、`fetch_sub` でアトミックデクリメント。
- レディキュー：Chase-Lev両端キュー（各スレッドローカルキュー + ワークスティーリング）を採用し、ロック競合を削減。
- ダウンストリームリスト：作成後は読み取り専用で、并发変更を回避。

#### 5.4 アダプティブスケジューリング

- システムに無限ループが1つだけ存在する場合、直接同期実行し、スケジューリングオーバーヘッドゼロ。
- タスク粒度とシステム负载に基づいて、並行度を動的に調整（例：キューの長さの監視を通じてワーカースレッド数を調整）。

#### 5.5 低コスト抽象化の原則

すべてのスケジューリングオーバーヘッドはタスク数に比例し、各タスクの追加オーバーヘッド（作成、エanqueue、依存処理）は数十ナノ秒レベルに制御。超細粒度タスクの場合、インライン最適化（第3節参照）を通じてスケジューリングを回避。

## 詳細設計

### 6.1 動的モジュールメタデータフォーマット（草案）

```rust
// メタデータファイル構造（簡略化）
struct Metadata {
    version: u32,
    functions: Vec<FuncMeta>,
}

struct FuncMeta {
    name: String,
    signature: TypeSignature,
    effects: EffectTag,      // Pure | IO | Block
    resource_params: Vec<usize>, // 引数インデックスリスト、リソースタイプの引数を示す
    calls: Vec<String>,       // 呼び出す他のエクスポート関数名（オプション）
    ownership: OwnershipInfo,
    send_sync: SendSync,      // Send/Syncを満たすかどうか
}
```

### 6.2 スケジューラコアデータ構造

```rust
struct TaskNode {
    id: TaskId,
    deps: Vec<TaskId>,                // 上流依存
    remaining_deps: AtomicUsize,
    inputs: Vec<Option<Value>>,
    result: Option<Value>,
    func: Executable,
    downstream: Vec<TaskId>,           // 下流ノード（作成後は読み取り専用）
    effect: EffectTag,
    arena_id: Option<ArenaId>,         // 所属エリア（オプション）
}

struct Scheduler {
    ready_queues: PerThreadQueue<TaskId>, // 各スレッドローカルキュー
    global_work_stealer: WorkStealer,
    arenas: ArenaAllocator,               // エリアアロケータ
}
```

### 6.3 コンテキストベースの最適化分析

コンパイラはMIR層で以下のステップを実行：

1. グローバル呼び出しグラフとデータ依存グラフを構築。
2. 各関数呼び出しノードの出次数（消費者数）を計算。
3. 出次数が1で、関数が副作用なし（`@pure`）で、再帰でない場合、「インライン化可能候補」としてマーク。
4. ヒューリスティック（命令数など）を組み合わせてインライン化収益を評価し、インライン化するかを決定。
5. インライン化時、呼び出しノードのコードをそのダウンストリームに埋め込み、依存関係を更新。

### 6.4 制御フローノード表現

```rust
enum NodeKind {
    Normal(FuncId),
    Phi { cond: TaskId, then_branch: TaskId, else_branch: TaskId },
    LoopTemplate { cond: FuncId, body: FuncId, state_var: VarId },
    // ...
}
```

ランタイム動的展開時、`LoopTemplate` は一連の `Normal` ノードインスタンスを生成する。

## トレードオフ

### メリット

- **動的モジュールの安全な統合**：メタデータ契約により動的ライブラリがメインペログラムとシームレスに並行モデルを共有でき、静的DAGの完全性を維持。
- **FFIの段階的な統合**：開発者は外部関数にアノテーションを徐々に追加でき、安全な降格から効率的な並行性への移行が可能。
- **最適化予測可能性**：コンテキストベースの決定が暗黙的な閾値に取って代わり、行動が透明で、開発者はツールを通じて最適化を理解可能。
- **制御フローの自然な統合**：Phiノードと動的展開により、DAGはすべてのプログラム構造を表現でき、特殊構文が不要。
- **パフォーマンススケーラビリティ**：エリア割り当て、ロックフリーキューなどの設計により、スケジューラが大規模並行性に対応可能。

### デメリット

- **メタデータ契約が追加のコンパイル複雑性を導入**：動的ライブラリ向けのメタデータ生成と解析が必要でツールチェーンのサポートが必要。
- **FFIアノテーションは開発者の正確性に依存**：誤ったラベル付けはデータ競合を引き起こす可能性があり、文書とツールヒントでリスクを軽減する必要がある。
- **コンテキスト最適化分析の時間消費**：グローバル分析はコンパイル時間を増加させる可能性があり、インクリメンタルコンパイルで緩和可能。
- **動的展開が追加のランタイムオーバーヘッドを導入**：ループ展開は動的にノードを生成する必要があるが、エリア割り当てで緩和可能。

## 実装戦略

### フェーズ区分（優先順位推奨付き）

> **実装優先順位の推奨**：初期段階では完全な実装を追求せず、シンプルなソリューションでシステムを動作させ、逐步的に最適化することを推奨。例：
> - 参照カウントは直接 `Arc` を使用可能。
> - ロックフリーキューは成熟したライブラリ（crossbeamのdequeなど）を使用可能。
> - エリア割り当ては最初はシンプルなbump allocatorを使用し、後で最適化。

#### フェーズ 1：基本サポート（v0.7）
- [ ] FFIをデフォルトで `@block` に降格を実装。
- [ ] FFI使用的 `@pure`、`@io` アノテーションを追加。
- [ ] リソースラップタイプ（例：`File`）とその基本メソッドを実装。

#### フェーズ 2：動的モジュールメタデータ（v0.8）
- [ ] メタデータフォーマットを設計し、コンパイラを修正して動的ライブラリ向けの `.yxmeta` を生成。
- [ ] メインペログラムコンパイル時のメタデータ読み取りとプレースホルダノード作成を実装。
- [ ] ランタイムバインディングメカニズムを実装。

#### フェーズ 3：コンテキスト最適化（v0.9）
- [ ] 呼び出しグラフ分析を実装し、ノード出次数を計算。
- [ ] インライン化決定とコード生成サポートを追加。
- [ ] 最適化レポート出力（インライン箇所、理由など）を実装。

#### フェーズ 4：制御フローDAG融合（v0.10）
- [ ] Phiノードと条件分岐のコンパイル時表現を実装。
- [ ] ループテンプレートと動的展開ランタイムを実装。
- [ ] 無限ループのバックグラウンドスケジューリングを完善。

#### フェーズ 5：パフォーマンス最適化（v1.0）
- [ ] エリアアロケータを実装。
- [ ] ロックフリーキューとワークスティーリングを最適化。
- [ ] ベンチマークテストとチューン。
