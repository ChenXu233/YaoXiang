# RFC-020：動的モジュールとFFI統合

## 摘要

本文書はRFC-001、008、018的基础上で、YaoXiangの並発モデルをさらに細分化・拡張し、**動的モジュールのロード**、**外部関数インターフェース（FFI）**、および**よりきめ細やかなスケジューリング最適化**といった現実的なシナリオに対応するものです。コア設計は以下の通りです：

1. **動的モジュールメタデータ契約**：同一言語で書かれた動的ライブラリに対するコンパイル時依存関係記述を提供し、プログラム全体の透明な並髪を保ちながら、静的なDAG構築を可能にする。
2. **FFIスケジューリングセマンティクス**：外部関数はDAG内でデフォルトで`@block`ノードとして動作し、アノテーション 통해パラレルスケジューリングに統合できる（FFIツールチェーンの詳細については[RFC-021](../review/021-library-driven-ffi-extension.md)を参照）。
3. **コールコンテキストベースの最適化**：静的閾値のフォールバックに置き換え、関数がDAG内で果たす実際の役割（消費者数、副作用など）に基づいて、コンパイラがインライン展開または独立ノードとしてのスケジューリングをインテリジェントに決定する。
4. **制御フローとDAGのマージメカニズム**：`if`、`loop`などの動的構造をPhiノードと動的展開を通じて、データフローに自然に統合する。
5. **ランタイムスケジューラのメモリとパフォーマンス最適化**：ノードのライフサイクル管理、エリアアロケーション、ロックフリーキューなどの低コスト抽象化の実装を明確化する。

本文書は言語仕様を完善させ、YaoXiangの並発モデルが静的全プログラム解析に対応しつつ、動的特性と外部連携を柔軟に処理し、高パフォーマンスと開発体験を両立させることを目指すものです。

## 動機

### 既存設計の不足

RFC-001/008/018は优雅な透明な並髪モデルを構築しましたが、現実世界の要件に対しては依然として盲点が存在します：

- **動的モジュール**：プログラムがプラグインや動的リンクライブラリをサポートしている場合、コンパイル時にモジュール内部のコール関係と依存関係を主プログラムが把握できず、グローバルDAG構築に失敗する。
- **FFIコール**：外部関数（Cライブラリなど）は完全なブラックボックスであり、内部に並髪、ブロッキング、副作用が含まれる可能性があり、これを通常のノードとして扱うと並髪の安全性が壊れる。
- **小関数のスケジューリングオーバーヘッド**：RFC-001で提案された"L1自動フォールバック"は静的閾値（命令数<50）を採用しており、この暗黙的なルールは開発者の行動予測を困難にし、複雑なコールコンテキストに適応できない。
- **制御フローとDAGの融合**：`if`、`loop`などの動的構造がDAG内でどのように表現されるかがまだ不明確であり、依存関係解析の正確性に影響を与える可能性がある。
- **ランタイムオーバーヘッド制御**：DAGノード数が増加するにつれ、スケジューラのメモリ管理とパフォーマンス最適化を明確に設計する必要がある。ボトルネックになるのを避けるため。

### 目標

- 透明な並髪のコア哲学を保ちながら、動的モジュールとFFIに対して明確、安全、段階的なサポートを提供する。
- スケジューリング最適化を「暗黙的なグローバルルール」から「コンテキストベースのインテリジェントな決定」に转变させ、予測可能性とパフォーマンスを向上させる。
- 動的制御フローに対するDAG表現を完善させ、すべてのプログラム構造がデータフローモデルに自然に統合できるようにする。
- スケジューラのメモリ管理とパフォーマンス最適化戦略を明確化し、低コスト抽象化を実現する。

## 提案

### 1. 動的モジュールメタデータ契約

#### 1.1 契約内容

YaoXiangでコンパイルされた各動的ライブラリ（`.yxo` / プラットフォーム固有の動的ライブラリ）には、以下の内容を含む**メタデータ記述ファイル**（`.yxmeta`）を添付する必要があります：

- **エクスポート関数リスト**：各関数の完全な型シグネチャ（パラメータ、戻り値、リソースタグ）。
- **副作用タグ**：コンパイラが自動導出する`@pure` / `@io`（開発者が明示的に上書きも可能）。
- **リソース依存**：各パラメータがリソース型かどうか（例：`File`）、戻り値に新リソースが含まれるかどうか。
- **コールグラフサマリ**（オプション）：この関数が呼び出す可能性のある他のエクスポート関数ID。跨モジュール循環依存検出用。
- **所有権情報**：パラメータ的所有権セマンティクス（借用/移動）、戻り値の所有権。
- **並髪が安全**：`Send`/`Sync`の自動導出結果。

メタデータ形式にはバイナリまたは構造化テキスト（例：MessagePack）を採用し、解析効率を確保します。

#### 1.2 コンパイル時処理

主プログラムのコンパイル時に、動的モジュール関数の呼び出しを検出した場合：

1. 対応するモジュールの`.yxmeta`ファイルを読み取る。
2. グローバルDAGに**プレースホルダノード**を作成し、メタデータから取得した入出力依存関係、副作用タグなどを記録する。
3. プレースホルダノードは通常のノード一样に依存関係解析に参加し、スケジューラは事前に実行順序を計画できる。

#### 1.3 ランタイムバインディング

動的モジュールをロードする時：

- ランタイムは実際の関数シグネチャとメタデータの一致を検証する（バージョン不一致防止）。
- プレースホルダノードを実際の関数ポインタにバインドする。

**サブグラフスケジューリングセマンティクスについて**：動的モジュール内に独立したサブDAGがある場合（例：モジュール自体が並髪を-continue含む場合）、そのサブグラフは**独立したスケジューリングユニット**として実行されます。その境界はモジュールのエクスポート関数によって定義されます：エクスポート関数を呼び出す時、サブグラフは全体として実行を開始し、その関数が戻るまで継続します。サブグラフ内のノードスケジューリングは、サブグラフ自身のスケジューラが担当し（モジュール内部では標準スケジューラを継続使用可能）、サブグラフと主DAGの連携は入出力データフローのみに制限されます—主DAGのプレースホルダノードはサブグラフの開始と終了のみ关心し、内部スケジューリングには介入しません。この設計により、モジュールのカプセル化が保証され、同時に主DAGの静的な完全性が維持されます。

#### 1.4 安全保証

- 動的モジュールが契約に違反した場合（例：`@pure`と主張しながらグローバル状態を修正）、結果は開発者の責任となります（FFIのunsafe境界 подобных). しかし同一言語であるため、ランタイムチェック（例：メモリ分離）により安全性を強化できますが、オーバーヘッドが増加します。
- 跨モジュール循環依存：モジュールAがBを呼び、BがAを呼び、且つメタデータに関数呼び出し関係がすでに宣言されている場合、コンパイラは检测してエラーを報告できます；宣言されていない場合、ランタイムでデッドロックが発生する可能性があり、スケジューラが检测してpanicします。

### 2. DAGにおけるFFIのスケジューリングセマンティクス

FFIの完全なツールチェーンサポート（動的ライブラリのロード、バインディング生成、型変換、メモ所有権）は[RFC-021](../review/021-library-driven-ffi-extension.md)で定義されます。本節ではDAGスケジューリングにおけるFFI呼び出しの動作のみを記述します。

#### 2.1 デフォルトスケジューリング動作

外部関数（`native("symbol")`で宣言）はDAG内でデフォルトで**`@block`ノード**として動作します：

- DAGパラレルのスケジューリングには参加せず、現在のスレッドで同期的に直接実行される。
- 実行中はスケジューラがその内部並髪を干预しない。
- 戻り値は使用可能だが、呼び出し自体は依存エッジを生成しない。

#### 2.2 オプションの並髪を示す注釈

開発者はアノテーションによりFFI呼び出しをDAGスケジューリングに統合できます（詳細については[RFC-021 §2.2](../review/021-library-driven-ffi-extension.md)を参照）：

- `@pure`：通常のDAGノードとして扱い、依存関係のない他のノードとパラレルに実行可能。
- `@io`：リソース依存解析に参加し、同じリソースへの複数回の呼び出しは自動的にシリアル化される。

#### 2.3 スケジューラへの影響

FFIノードはスケジューラ内では通常のノードと同じ`TaskNode`構造を使用しますが、違いは`effect`タグが`Block`に設定されていることです。スケジューラは`Block`ノードに遭遇するとパラレルのスケジューリングをスキップし、直接同期的に実行します。

### 3. コールコンテキストベースの最適化

RFC-001の"L1自動フォールバック"静的閾値を置き換え、**コンパイラが各呼び出し地点のDAG内の実際のコンテキストに基づいてインテリジェントに決定**するようにします。

#### 3.1 最適化決定の根拠

コンパイラは各関数呼び出しノードを分析します：

- **消費者数**：このノードの結果が何個のダウンストリームノードに使用されているか。1の場合、インライン展開候補の資格あり；1より大きい場合、結果共有のために独立ノードとして保持する必要がある。
- **副作用**：ノードが`@io`副作用を持つ場合、順序保証のために独立ノードとして保持する必要がある。
- **計算量推定**：命令数などのヒューリスティクスも参考にはできるが、硬性的な閾値としては使用せず、インライン展開の利益評価のみに使用する。
- **リソース依存**：ノードがリソース変数（例：`File`）涉及到、且つリソース変数が上下游間で渡される場合、インライン展開が依存チェーンを破壊する可能性があり、慎重する必要がある。

#### 3.2 インライン展開操作

インライン展開を決定した場合：

- そのノードの計算ロジックを唯一のダウンストリームノードのコードに直接埋め込む。
- DAGからそのノードを削除し、入力をダウンストリームノードの入力として直接使用する。
- 最終的なコード生成時、インライン展開された関数は独立したスケジューリングユニットを生成しない。

#### 3.3 インライン展開の制限

- 再帰関数またはループボディ内の呼び出しは通常インライン展開しない（無限展開防止）。
- 跨モジュール境界（動的モジュール、FFI）の関数はインライン展開しない。
- 開発者は`@noinline`アノテーションによりインライン展開を明示的に禁止したり、`@forceinline`によりコンパイラにインライン展開を試みるように指示したりできます。

#### 3.4 可観測性

コンパイラは最適化レポートを生成する必要があります（`--emit-optimization-report`で有効化可能）。以下の情報を 含みます：

- **各インライン展開箇所**：インライン展開された関数名、呼び出し位置、インライン展開の理由（例：「唯一の消費者かつ純粋関数」）。
- **独立ノードとして保持された理由**：例：「複数の消費者あり」「副作用あり」「跨モジュール呼び出し」など。
- **決定統計**：インライン展開総数、保持ノード数。開発者が最適化の効果を評価するのに役立つ。

レポートの出力形式はテキストまたはJSONとし、ツールでの解析を容易にします。

### 4. 制御フローとDAGのマージ

#### 4.1 条件分岐（if）の処理

**Phiノード**（SSA形式を採用）を導入して分岐のマージポイントを表します：

- コンパイル時に各`if`式に対して以下を構築します：
  - 2つの分岐サブDAG（それぞれ`then`と`else`に対応）。
  - 条件変数と2つの分岐の出力を入力とするPhiノード。
- Phiノードのセマンティクス：条件変数が準備できたら、条件値に基づいて対応する分岐の出力を自分の出力として選択する。
- ランタイムでは、Phiノードは条件変数に依存する；条件が準備できたら、条件に応じて自身を選択した分岐のダウンストリームリストに追加し、その分岐の結果を待つ。

サンプルのDAG：
```
        cond
       /    \
  then DAG  else DAG
       \    /
        Phi
         |
      后续ノード
```

#### 4.2 ループ（loop/while）の処理

ループはフィードバックエッジ付きのサブDAGとして扱い、ランタイムで**必要に応じて展開**されます：

- コンパイル時にループボディを識別し、**ループテンプレート**を構築します。以下の内容を含みます：
  - 条件ノード。
  - ループボディサブDAG。
  - 反復間で渡される状態変数。
- ランタイムで、ループ結果が必要な場合（例：ループ終了後に累積値を使用する場合）、スケジューラは反復を動的に展開し始めます：
  1. 初回は条件ノードをスケジューリングし、真であれば最初の反復のサブDAGをインスタンス化し、入力には初期状態と外部変数を含む。
  2. 反復完了後に新状態が生成され、条件ノードを再度スケジューリング（新状態に依存）して継続決定を行う。
  3. 条件が偽になるまで繰り返し、最後の反復の出力がループ結果となる。

**複雑な例：ループ条件がループボディ内部の更新に依存する場合**：
```yaoxiang
let mut x = 0
while x < 10 {
    x = compute(x)  // xはループボディ内で更新される
}
```
このパターンでは、条件ノード`x < 10`は各反復後に更新される`x`に依存します。DAGの表現は以下の通りです：
- ループテンプレートには状態変数`x`が含まれ、初期値は0。
- 各反復：まず条件ノードを実行（現在の`x`に依存）、真であれば`x = compute(x)`を実行して新しい`x`を生成し、条件ノードに再度入る。
- ランタイムは上述のプロセスで動的に展開し、条件が偽になるまで続ける。

反復間の依存関係は状態変数を介して自然にデータフローを形成し、依存関係のある反復は自動的にシリアル化され、依存関係のない反復はパラレルに実行可能（例：`map`）。

#### 4.3 無限ループの特別な処理

単一の無限ループは主DAGとして直接同期的に実行されます（スケジューリングオーバーヘッドなし）；複数の無限ループはバックグラウンドDAGとして動作し、スケジューラがタイムスライスでパラレート実行します。

### 5. ランタイムスケジューラのメモリとパフォーマンス最適化

#### 5.1 ノードライフサイクル管理

- 各ノードは**参照カウント**（アトミック変数）を保持し、その結果を依存する消費者の数を表します。
- ノードが実行完了し、結果をすべてのダウンストリームに 전달した後、参照カウントが0になり、ノードのメモリを解放できます。
- 結果値自体も参照カウントを採用（`Arc<T>`）しますが、最適化可能：結果が1つの消費者のみ使用される場合、所有権を直接移動し、カウンタオーバーヘッドを避けます。

#### 5.2 エリアメモリアロケーション（Arena）

大量の短ライフスパン 노ード（例：ループ反復）の動的生成に対しては、**エリアアロケータ**を使用します：

- 1回のループ展開に対してメモリエリアを確保します。
- エリア内のノードは連続的に確保され、整体的に解放されるため、断片化と解放オーバーヘッドが減少します。
- エリア終了時、すべてのノードメモリを一括回收します。

#### 5.3 ロックフリーデータ構造

- 依存カウンタ：`AtomicUsize`を使用し、`fetch_sub`でアトミックデクリメント。
- レディキュー：Chase-Lev両端キューを採用（各スレッドのローカルキュー＋ワークスティーリング）。ロック競合を減少させます。
- ダウンストリームリスト：作成後は読み取り専用にし、并发修改を避けます。

#### 5.4 アダプティブスケジューリング

- システムに無限ループが1つだけある場合、直接同期的に実行し、スケジューリングオーバーヘッドをゼロにする。
- タスク粒度とシステム负载に応じて、パラレル度を動的に調整（例：キューの長さ監視によりワーカースレッド数を調整）。

#### 5.5 低コスト抽象化の原則

すべてのスケジューリングオーバーヘッドはタスク数に比例し、各タスクの追加オーバーヘッド（作成、エanqueuing、依存関係処理）は数十ナノ秒级别に制御されます。超細粒度タスクについては、インライン展開最適化（第3節参照）によりスケジューリングを避けます。

## 詳細設計

### 6.1 動的モジュールメタデータ形式（草案）

```rust
// メタデータファイル構造（簡略化）
struct Metadata {
    version: u32,
    functions: Vec<FuncMeta>,
}

struct FuncMeta {
    name: String,
    signature: TypeSignature,
    effects: EffectTag,      // Pure | IO | Block
    resource_params: Vec<usize>, // パラメータインデックスリスト。リソース型のパラメータを示す
    calls: Vec<String>,       // 呼び出す他のエクスポート関数名（オプション）
    ownership: OwnershipInfo,
    send_sync: SendSync,      // Send/Syncを満たすかどうか
}
```

### 6.2 スケジューラコアデータ構造

```rust
struct TaskNode {
    id: TaskId,
    deps: Vec<TaskId>,                // 上流依存
    remaining_deps: AtomicUsize,
    inputs: Vec<Option<Value>>,
    result: Option<Value>,
    func: Executable,
    downstream: Vec<TaskId>,           // ダウンストリームノード（作成後は読み取り専用）
    effect: EffectTag,
    arena_id: Option<ArenaId>,         // 所属エリア（オプション）
}

struct Scheduler {
    ready_queues: PerThreadQueue<TaskId>, // 各スレッドのローカルキュー
    global_work_stealer: WorkStealer,
    arenas: ArenaAllocator,               // エリアアロケータ
}
```

### 6.3 コンテキストベースの最適化分析

コンパイラはMIRレベルで以下のステップを実行します：

1. グローバルコールグラフとデータ依存グラフを構築する。
2. 各関数呼び出しノードに対して、その出次数（消費者数）を計算する。
3. 出次数が1で、関数が副作用なし（`@pure`）、且つ再帰でない場合、「インライン展開可能候補」としてマークする。
4. ヒューリスティクス（例：命令数）を組み合わせてインライン展開の利益を評価し、インライン展開を実行するかを決定する。
5. インライン展開時、呼び出しノードのコードをダウンストリームに埋め込み、依存関係を更新する。

### 6.4 制御フローノード表現

```rust
enum NodeKind {
    Normal(FuncId),
    Phi { cond: TaskId, then_branch: TaskId, else_branch: TaskId },
    LoopTemplate { cond: FuncId, body: FuncId, state_var: VarId },
    // ...
}
```

ランタイムで動的に展開する場合、`LoopTemplate`は一連の`Normal`ノードインスタンスを生成します。

## 权衡

### 利点

- **動的モジュールの安全な統合**：メタデータ契約により、動的ライブラリが主プログラムとシームレスに並髪を-continue共有でき、同時に静的DAGの完全性を維持します。
- **FFIの段階的な統合**：開発者は外部関数に段階的にアノテーションを追加でき、安全なデグレードから高效な並髪へと移行できます。
- **最適化予測可能性**：コンテキストベースの決定は暗黙的な閾値を置き換え、動作が透明になり、開発者はツールを使用して最適化を理解できます。
- **制御フローの自然な統合**：Phiノードと動的展開により、DAGがすべてのプログラム構造を表現でき、特別な構文は不要です。
- **パフォーマンススケーラビリティ**：エリアアロケーション、ロックフリーキューなどの設計により、スケジューラが大規模な並髪を处理できます。

### 欠点

- **メタデータ契約がコンパイルの複雑さを増加させる**：動的ライブラリ用のメタデータ生成と解析が必要で、ツールチェーンのサポートが必要です。
- **FFIアノテーションは開発者の正確性に依存する**：誤ったラベル付けはデータ競合につながる可能性があり、文書とツールによるヒントでリスクを軽減する必要があります。
- **コンテキスト最適化分析に時間を要する**：グローバル分析によりコンパイル時間が增加する可能性がありますが、インクリメンタルコンパイルにより緩和できます。
- **動的展開がランタイムオーバーヘッドを増加させる**：ループ展開はノードの動的生成を必要としますが、エリアアロケーションにより缓解できます。

## 実装戦略

### 段階分け（優先順位の提案付き）

> **実装優先順位の提案**：初期段階では完璧な実装を追求する必要はなく、まずシンプルな解决方案でシステムを動作させ、逐步的に最適化してください。例：
> - 参照カウントは直接`Arc`を使用できます。
> - ロックフリーキューは成熟したライブラリ（例：crossbeamのdeque）を使用できます。
> - エリアアロケーションはまずシンプルなbump allocatorを使用し、後で最適化できます。

#### 段階1：基盤サポート（v0.7）
- [ ] FFIをデフォルトで`@block`にデグレードさせる実装。
- [ ] FFI使用的`@pure`、`@io`アノテーションの追加。
- [ ] リソースラッパ型（例：`File`）とその基本メソッドの実装。

#### 段階2：動的モジュールメタデータ（v0.8）

- [ ] 主プログラムコンパイル時のメタデータ読み取りとプレースホルダノード作成の実装。
- [ ] ランタイムバインディング機構の実装。

#### 段階3：コンテキスト最適化（v0.9）
- [ ] コールグラフ分析とノード出次数の計算の実装。
- [ ] インライン展開決定とコード生成サポートの追加。

#### 段階4：制御フローDAG融合（v0.10）
