## 摘要

本文書は RFC-001、008、018 の基础上，进一步细化和扩展 YaoXiang の並発モデルを、**動的モジュールのロード**、**外部関数インタフェース（FFI）**、以及**より精细なスケジューリング最適化**などの実践的なシナリオに対応するために细分化する。核心設計包括：

1. **動的モジュールメタデータ契約**：同一言語で書かれた動的ライブラリに対してコンパイル時の依存関係記述を提供し、メインプログラムが静的 DAG を構築的同时に透明な並発を維持できるようにする。
2. **FFI レイヤー処理**：外部コードはデフォルトで同期（`@block`）に降格され、開発者はアノテーションを通じてメタデータを提供することで徐々に並発スケジューリングに統合できる。外部の並発は外部言語が自行管理する。
3. **呼び出しコンテキストに基づく最適化**：静的閾値へのフォールバックに取って代わり、コンパイラは関数 DAG における実際の役割（消費者数、副作用など）に基づいて、その関数をインライン化するかす独立ノードとしてスケジューリングするかをインテリジェントに決定する。
4. **制御流と DAG のマージ機構**：Phi ノードと動的展開を通じて、`if`、`loop` などの動的構造をデータフローグラフに自然に統合する。
5. **ランタイムスケジューラのメモリとパフォーマンス最適化**：ノードのライフサイクル管理、領域確保、ロックフリーキューなど低コスト抽象化の実装を明確化する。

本文書は言語仕様を完善させ、YaoXiang の並発モデルが静的全プログラム分析に対応でる同时に、動的性质と外部 상호作用にも柔軟に対応でき、高パフォーマンスと開発体験を维持できるようにすることを目的とする。

## 動機

### 既存設計の不足

RFC-001/008/018 は优雅な透明な並発モデルを構築したが、現実世界の要件に対しては 여전히盲点がある：

- **動的モジュール**：プログラムがプラグインや動的リンクライブラリをサポートしている場合、メインプログラムはコンパイル時にモジュール内部の呼び出し関係や依存関係を把握できず、グローバル DAG の構築が失敗する。
- **FFI 呼び出し**：外部関数（C ライブラリなど）は完全なブラックボックスであり、内部に並発、ブロッキング、副作用が含まれている可能性がある。これを普通のノードとして扱うと、並発安全性が破られる。
- **小関数のスケジューリングオーバーヘッド**：RFC-001 で提案された「L1 自動フォールバック」は静的閾値（命令数 <50）を採用しているが、この暗黙的なルールは開発者の行動を予測しづらく、複雑な呼び出しコンテキストに対応できない。
- **制御流と DAG の融合**：`if`、`loop` などの動的構造の DAG での表現がまだ不明確であり、依存関係分析の正確性に影響する可能性がある。
- **ランタイムオーバーヘッド制御**：DAG ノード数が増加するにつれて、スケジューラのメモリ管理とパフォーマンス最適化を明確に設計する必要がある。ボトルネックになるのを避けるため。

### 目標

- 透明な並発の核心哲学を維持しつつ、動的モジュールと FFI に明確で安全、段階的なサポートを提供する。
- スケジューリング最適化を「暗黙的なグローバルルール」から「コンテキストベースのインテリジェントな決定」に转变させ、予測可能性とパフォーマンスを向上させる。
- 動的制御流に対する DAG の表現を完善させ、すべてのプログラム構造がデータフローモデルに自然に統合できるようにする。
- スケジューラのメモリ管理とパフォーマンス最適化戦略を明確化し、低コスト抽象化を実現する。

## 提案

### 1. 動的モジュールメタデータ契約

#### 1.1 契約内容

YaoXiang でコンパイルされた各動的ライブラリ（`.yxo` / プラットフォーム固有の動的ライブラリ）には、**メタデータ記述ファイル**（`.yxmeta`）を添付する必要がある：

- **エクスポート関数リスト**：各関数の完全な型シグネチャ（パラメータ、戻り値、リソースタグ）。
- **副作用タグ**：コンパイラが自動導出する `@pure` / `@io`（開発者が明示的にオーバーライド也可能）。
- **リソース依存**：各パラメータがリソース型かどうか（例：`File`）、戻り値に新しいリソースが含まれるかどうか。
- **呼び出しグラフサマリー**（オプション）：この関数が呼び出す可能性のある他のエクスポート関数 ID。跨モジュール循環依存検出に使用。
- **所有権情報**：パラメータの所有権セマンティクス（借用/移動）、戻り値の所有権。
- **並発安全**：`Send`/`Sync` の自動導出結果。

メタデータフォーマットはバイナリまたは構造化テキスト（MessagePack など）を使用し、解析効率を確保する。

#### 1.2 コンパイル時処理

メインプログラムのコンパイル時、動的モジュール関数への呼び出しに出会ったら：

1. 対応モジュールの `.yxmeta` ファイルを読み取る。
2. グローバル DAG に**プレースホルダノード**を作成し、メタデータから取得した入出力依存、副作用タグなどを記録する。
3. プレースホルダノードは通常のノード一样に依存関係分析に参加し、スケジューラは事前に実行順序を計画できる。

#### 1.3 ランタイムバインディング

動的モジュールがロードされるとき：

- ランタイムは実際の関数シグネチャとメタデータの一致を検証する（バージョン不一致を防止）。
- プレースホルダノードを実際の関数ポインタにバインディングする。

**サブグラフスケジューリングセマンティクスについて**：動的モジュール内に独立したサブ DAG がある場合（例：モジュール自体が並発ロジックを含む場合）、そのサブグラフは**独立したスケジューリングユニット**として実行される。その境界はモジュールのエクスポート関数によって定義される：エクスポート関数を呼び出すとき、サブグラフは全体として実行を開始し、その関数が戻るまで続く。サブグラフ内のノードスケジューリングはサブグラフ自身のスケジューラが担当する（モジュール内部では標準スケジューラを引き続き使用可能）が、サブグラフとメイン DAG のやり取りは入出力データフローのみに制限される——メイン DAG のプレースホルダノードはサブグラフの開始と終了のみ关心し、内部スケジューリングには介入しない。この設計により、モジュールのカプセル化が保証されると同时に、メイン DAG の静的完全性が維持される。

#### 1.4 安全保証

- 動的モジュールが契約に違反した場合（例：`@pure` と主張しながらグローバル状態を変更）、結果は開発者の責任になる（FFI の unsafe 境界と类似）。しかし同一言語であるため、ランタイムチェック（メモリ隔離など）で安全性を強化することは可能だが、オーバーヘッドが増加する。
- 跨モジュール循環依存：モジュール A が B を呼び、B がまた A を呼び、メタデータに呼び出し関係が宣言されている場合、コンパイラは検出されてエラーを報告できる；宣言されていない場合、ランタイムでデッドロックが発生する可能性があり、スケジューラが検出してから panic する。

### 2. FFI レイヤー処理

#### 2.1 デフォルト降格

`extern` を通じて導入された外部関数は、デフォルトで `@block` 同期呼び出しとして扱われる：

- 呼び出しは DAG スケジューリングに入らず、現在のスレッドで直接実行される。
- 実行中は YaoXiang スケジューラがその内部並発に介入しない。
- 戻り値は使用可能だが、呼び出し自体が依存関係分析に参加しない。

> **⚠️ 開発者への注意**：FFI 呼び出しはデフォルトで DAG スケジューリングに参加しないため、内部に存在する並発（C ライブラリが開始したスレッドなど）は完全に外部言語が管理し、YaoXiang は追踪できない。FFI 呼び出しを並発スケジューリングに統合したい場合は、アノテーションを通じてメタデータを提供する必要がある（2.2 節参照）。「同期コードは自動的に並列化される」と 가정 하지 마라——FFI 呼び出しは自動的に並列化されない。

#### 2.2 オプションのメタデータ注釈

開発者はアノテーションを通じてより正確なメタデータを提供し、FFI 呼び出しを徐々に並発スケジューリングに統合できる：

```yaoxiang
@pure   // 純粋関数、副作用なし、自由なスケジューリング
extern "C" fn sin(x: Double) -> Double

@io     // I/O 関連、リソース依存でシリアライズが必要
extern "C" fn write(fd: i32, buf: *u8, count: usize) -> isize

@block  // 明示的な同期宣言（デフォルト、省略可能）
extern "C" fn old_api() -> i32
```

- `@pure` でマークされ、パラメータにリソース変数が含まれていない場合は、通常の DAG ノードとして扱われ、並列実行できる。
- `@io` でマークされ、パラメータにリソース型が含まれている場合（例：ラップされた `File`）、自動的に依存関係エッジが確立される。

#### 2.3 リソースラップ

標準ライブラリが一般的な FFI リソース（ファイル記述子、データベース接続など）にラップタイプを提供し、生リソースを YaoXiang リソース型にラップして、DAG 依存関係分析を活用することを推奨する。例：

```yaoxiang
struct File { fd: i32 }  // リソース型としてマーク
impl File {
    @io
    fn write(self: &mut File, data: &[u8]) -> Result { ... }
}
```

こうすることで、同じ `File` 変数に対する複数の `write` 呼び出しは自動的にシリアライズされる。

#### 2.4 外部並発管理

FFI 関数が内部でスレッドや非同期操作を開始した場合、完全に外部言語が負責する。YaoXiang は呼び出しの開始と終了のみを关心し、内部の詳細を追踪しない。開発者は、これらの並発操作が YaoXiang のメモリモデルと冲突しないことを確認する必要がある（例如：コールバック時に所有権と Send 制約に注意）。

### 3. 呼び出しコンテキストに基づく最適化

RFC-001 の「L1 自動フォールバック」静的閾値に取って代わり、**コンパイラが各呼び出し点が DAG の実際のコンテキストに基づいてインテリジェントに決定**する。

#### 3.1 最適化決定の根拠

コンパイラは各関数呼び出しノードを分析する：

- **消費者数**：このノードの結果が何個のダウンストリームノードで使用されているか。1 の場合、インライン候補の資格があり；1 より大きい場合、結果を共有するために独立ノードとして保持する必要がある。
- **副作用**：ノードに `@io` 副作用がある場合、順序を保証するために独立ノードとして保持する必要がある。
- **計算量推定**：命令数などのヒューリスティックを引き続き参考として使用できるが、硬的閾値としては使用せず、インライン收益評価のみに使用する。
- **リソース依存**：ノードがリソース変数（例：`File`）を含み、そのリソース変数が上下游間で渡される場合、インライン化すると依存チェーンが破坏される可能性があり、慎重する必要がある。

#### 3.2 インライン操作

インライン化が決定された場合：

- そのノードの計算ロジックを唯一のダウンストリームノードのコードに直接埋め込む。
- DAG からそのノードを移除し、その入力は直接ダウンストリームノードの入力になる。
- 最終コード生成時、インライン化された関数は独立したスケジューリングユニットを生成しない。

#### 3.3 インライン化の制限

- 再帰関数やループ体内的の呼び出しは通常インライン化しない（無限展開を防ぐため）。
- 跨モジュール境界（動的モジュール、FFI）の関数はインライン化しない。
- 開発者は `@noinline` アノテーションでインライン化を强制的に禁止したり、`@forceinline` でコンパイラにインライン化を 시도하도록ヒントを提供したりできる。

#### 3.4 観測可能性

コンパイラは最適化レポートを生成する必要がある（`--emit-optimization-report` で有効化可能）。以下の情報を含む：

- **各インライン点**：インライン化された関数名、呼び出し位置、インライン化理由（「唯一の消費者かつ純粋関数」など）。
- **独立ノードとして保持された理由**：「複数の消費者あり」「副作用を含む」「跨モジュール呼び出し」など。
- **決定統計**：インライン化総数、保持ノード数。開発者が最適化効果を評価するのに役立つ。

レポート出力フォーマットはテキストまたは JSON のいずれかとし、ツール解析を容易にする。

### 4. 制御流と DAG のマージ

#### 4.1 条件分岐（if）の処理

**Phi ノード**（SSA 形式のアイデアを借用）を導入して、分岐合流点を表す：

- コンパイル時に各 `if` 式に対して以下を構築する：
  - 2 つの分岐サブ DAG（それぞれ `then` と `else` に対応）。
  - 条件変数と2 つの分岐の出力入力とする Phi ノード。
- Phi ノードのセマンティクス：条件変数が準備できたら、条件値に応じて対応する分岐の出力を選択して自分の出力とする。
- ランタイムでは、Phi ノードは条件変数に依存する；条件が準備できたら、選択した分岐のダウンストリームリストに动态的に自分を追加し、その分岐の結果を待つ。

示例 DAG：

```
        cond
       /    \
  then DAG  else DAG
       \    /
        Phi
         |
      后续节点
```

#### 4.2 ループ（loop/while）の処理

ループはフィードバックエッジを持つサブ DAG として取り扱い、ランタイムで**必要に応じて展開**する：

- コンパイル時にループ体を識別し、**ループテンプレート**を構築する。含まれるもの：
  - 条件ノード。
  - ループボディサブ DAG。
  - イテレーション間で渡される状態変数。
- ランタイムでは、ループ結果が必要なとき（例如：ループ終了後）にスケジューラがイテレーションを動的に展開し始める：
  1. 初めて条件ノードをスケジューリングし、真であれば最初のイテレーションのサブ DAG をインスタンス化し、入力には初期状態と外部変数を含む。
  2. イテレーション完了後に新しい状態が生成され、条件ノードを再度スケジューリングする（新状態に依存）して継続するかどうかを決定する。
  3. 条件が偽になるまで繰り返し、最後のイテレーションの出力がループ結果となる。

**複雑な例：ループ条件がループボディ内部で更新されることに依存**

```yaoxiang
let mut x = 0
while x < 10 {
    x = compute(x)  // x はループボディ内で更新される
}
```

このパターンでは、条件ノード `x < 10` は各イテレーション後に更新される `x` に依存する。DAG での表現：

- ループテンプレートには状態変数 `x` を含み、初期値は 0。
- 各イテレーション：首先条件ノードを実行（現在の `x` に依存）、真であれば `x = compute(x)` を実行して新しい `x` を生成し、条件ノードに再度入る。
- ランタイムは上記の流程で動的に展開し、条件が偽になるまで続ける。

イテレーション間の依存関係は状態変数を介して自然にデータフローを形成し、依存関係のあるイテレーションは自動的にシリアライズされ、依存関係のないイテレーションは並列化できる（例：`map`）。

#### 4.3 無限ループの特殊処理

単一の無限ループはメイン DAG として直接同期実行する（スケジューリングオーバーヘッドなし）；複数の無限ループはバックグラウンド DAG として、スケジューラのタイムスライスで同時実行される。

### 5. ランタイムスケジューラのメモリとパフォーマンス最適化

#### 5.1 ノードライフサイクル管理

- 各ノードは**参照カウント**（アトミック変数）を維持し、その結果を依赖する消費者数を示す。
- ノードが実行を完了し、結果をすべてのダウンストリームに渡した後、参照カウントがゼロになり、ノードのメモリを解放できる。
- 結果値自体も参照カウント（`Arc<T>`）を使用하지만、最適化 가능：結果が1 つの消費者のみに使用される場合、所有権を直接移動して、カウントオーバーヘッドを回避する。

#### 5.2 領域メモリアロケーション（Arena）

動的に生成された大量の短ライフサイクルのノード（ループイテレーションなど）に対しては、**領域アロケータ**を使用：

- 1 回のループ展開に対してメモリアリーナを確保する。
- 領域内のノードは連続的に確保され、全体で解放されるため、断片化と解放オーバーヘッドが減少する。
- 領域終了時、すべてのノードメモリを一度に回收する。

#### 5.3 ロックフリーデータ構造

- 依存カウンタ：`AtomicUsize` を使用し、`fetch_sub` でアトミックにデクリメントする。
- 準備完了キュー：Chase-Lev 双方端キュー（各スレッドローカルキュー + ワークスティーリング）を採用し、ロック競合を减少する。
- ダウンストリームリスト：作成後は読み取り専用であり、并发変更を避ける。

#### 5.4 アダプティブスケジューリング

- システムに無限ループが1 つだけしかない場合、直接同期実行し、ゼロスケジューリングオーバーヘッドにする。
- タスク粒度とシステム负载に応じて、並行度を動的に調整する（例：キュー長の監視を通じてワークスレッド数を調整）。

#### 5.5 低コスト抽象化の原則

すべてのスケジューリングオーバーヘッドはタスク数に比例し、各タスクの追加オーバーヘッド（作成、エンキュー、依存関係処理）は数十ナノ秒级别に制御する。超細粒度タスクに対しては、インライン最適化（第3 節参照）を通じてスケジューリングを避ける。

## 詳細設計

### 6.1 動的モジュールメタデータフォーマット（草案）

```rust
// メタデータファイル構造（単純化）
struct Metadata {
    version: u32,
    functions: Vec<FuncMeta>,
}

struct FuncMeta {
    name: String,
    signature: TypeSignature,
    effects: EffectTag,      // Pure | IO | Block
    resource_params: Vec<usize>, // パラメータインデックスリスト、リソース型のパラメータを示す
    calls: Vec<String>,       // 呼び出す他のエクスポート関数名（オプション）
    ownership: OwnershipInfo,
    send_sync: SendSync,      // Send/Sync を満たすかどうか
}
```

### 6.2 スケジューラコアデータ構造

```rust
struct TaskNode {
    id: TaskId,
    deps: Vec<TaskId>,                // 上流依存
    remaining_deps: AtomicUsize,
    inputs: Vec<Option<Value>>,
    result: Option<Value>,
    func: Executable,
    downstream: Vec<TaskId>,           // ダウンストリームノード（作成後は読み取り専用）
    effect: EffectTag,
    arena_id: Option<ArenaId>,         // 所属アリーナ（オプション）
}

struct Scheduler {
    ready_queues: PerThreadQueue<TaskId>, // 各スレッドローカルキュー
    global_work_stealer: WorkStealer,
    arenas: ArenaAllocator,               // 領域アロケータ
}
```

### 6.3 コンテキストベースの最適化分析

コンパイラは MIR 層で以下のステップを実行する：

1. グローバル呼び出しグラフとデータ依存グラフを構築する。
2. 各関数呼び出しノードに対して、その出次数（消費者数）を計算する。
3. 出次数が 1 で、関数が副作用なし（`@pure`）、かつ再帰でない場合、「インライン候補可」としてマークする。
4. ヒューリスティック（命令数など）を組み合わせてインライン收益を評価し、インライン化するかどうかを決定する。
5. インライン化時、呼び出しノードのコードをそのダウンストリームに埋め込み、依存関係を更新する。

### 6.4 制御流ノード表現

```rust
enum NodeKind {
    Normal(FuncId),
    Phi { cond: TaskId, then_branch: TaskId, else_branch: TaskId },
    LoopTemplate { cond: FuncId, body: FuncId, state_var: VarId },
    // ...
}
```

ランタイムで動的に展開する際、`LoopTemplate` は一連の `Normal` ノードインスタンスを生成する。

## 权衡

### メリット

- **動的モジュールの 안전한 統合**：メタデータ契約により、動的ライブラリはメインプログラムとシームレスに並発モデルを共有でき、同時に静的 DAG の完全性を維持できる。
- **FFI の段階的統合**：開発者は外部関数に段階的にアノテーションを追加でき、安全な降格から効率的な並発への移行が可能。
- **最適化の予測可能性**：コンテキストベースの決定は暗黙的な閾値に取って代わり、動作が透明であり、開発者はツールを通じて最適化を理解できる。
- **制御流の自然な統合**：Phi ノードと動的展開により、DAG はすべてのプログラム構造を表現でき、特殊構文が不要。
- **パフォーマンスのスケーラビリティ**：領域確保、ロックフリーキューなどの設計により、スケジューラが大規模な並発に対応できる。

### デメリット

- **メタデータ契約の増加によるコンパイル複雑度の増加**：動的ライブラリ用にメタデータを生成・解析する必要があり、ツールチェーンのサポートが必要。
- **FFI アノテーションの開発者の正しさへの依存**：誤ったラベル付けはデータ競合を引き起こす可能性があり、文書とツールヒントでリスクを軽減する必要がある。
- **コンテキスト最適化分析の時間消費**：グローバル分析によりコンパイル時間が增加する可能性があるが、インクリメンタルコンパイルで軽減可能。
- **動的展開によるランタイムオーバーヘッドの増加**：ループ展開には動的にノードを生成する必要が必要だが、領域確保で軽減可能。

## 実装戦略

### フェーズ分け（優先順位提案付き）

> **実装優先順位提案**：初期段階では完璧な実装を追求する必要はなく、システムを動作させるためにシンプルな方案を採用し、その後徐々に最適化することも可能。例：
> - 参照カウントは直接 `Arc` を使用できる。
> - ロックフリーキューは成熟したライブラリ（crossbeam の deque など）を使用できる。
> - 領域確保はまずシンプルな bump allocator を使用し、後で最適化する。

#### フェーズ 1：基本サポート（v0.7）

- [ ] FFI をデフォルトで `@block` に降格する実装。
- [ ] FFI が使用する `@pure`、`@io` アノテーションを追加する。
- [ ] リソースラップタイプ（例：`File`）とその基本メソッドを実装する。

#### フェーズ 2：動的モジュールメタデータ（v0.8）

- [ ] メタデータフォーマットを設計し、コンパイラを修正して動的ライブラリ用に `.yxmeta` を生成する。
- [ ] メインプログラムコンパイル時にメタデータを読み取り、プレースホルダノードを作成するメカニズムを実装する。
- [ ] ランタイムバインディングメカニズムを実装する。

#### フェーズ 3：コンテキスト最適化（v0.9）

- [ ] 呼び出しグラフ分析を実装し、ノード出次数を計算する。
- [ ] インライン決定とコード生成サポートを追加する。
- [ ] 最適化レポート出力（インライン点、理由など）を実装する。

#### フェーズ 4：制御流 DAG 融合（v0.10）

- [ ] Phi ノードと条件分岐のコンパイル時表現を実装する。
- [ ] ループテンプレートと動的展開ランタイムを実装する。
- [ ] 無限ループのバックグラウンドスケジューリングを完善する。

#### フェーズ 5：パフォーマンス最適化（v1.0）

- [ ] 領域アロケータを実装する。
- [ ] ロックフリーキューとワークスティーリングを最適化する。
- [ ] ベンチマークテストと調整。
