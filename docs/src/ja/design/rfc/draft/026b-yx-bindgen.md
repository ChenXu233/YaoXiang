# RFC-026b: yx-bindgen ツールチェーン

> **親 RFC**: [RFC-026: FFI コアメカニズム](../accepted/026-ffi-core-mechanism.md)
>
> **依存関係**: 本 RFC の実装は、RFC-026 のコア FFI メカニズムが先に確立されることに依存する。

## 要約

yx-bindgen は C ヘッダファイルを機械的に `.yx` FFI バインディングファイルに変換し、ライブラリバインディング、型の二分（不透明ハンドル / 透明型）、関数宣言、メソッドバインディングを生成する。

**二つの核心原則**：
1. **機械的出力ドラフト、所有権を推測しない**——所有権は YaoXiang 型が付与し、ユーザーが C ドキュメントに従って確認する
2. **プラットフォーム正しいレイアウト保証**——透明型のフィールドサイズ、アラインメントはターゲットプラットフォームに応じて計算され、C struct とのバイナリ一致を保証する

## 動機

手作業で FFI バインディングを書くのは退屈で間違いやすい——C ライブラリには数十から数百の関数がある。さらに危険なものは**透明型のレイアウト**である：手作業で `Timespec: Type = { tv_sec: Int64, tv_nsec: Int64 }` と書く場合、フィールドサイズ、アラインメント、padding はターゲットプラットフォームの C `struct timespec` と正確に一致しなければならず、そうでなければ C が誤ったレイアウトで読み書きすることで範囲外アクセスが発生する（RFC-026 §2.2 の信頼保証）。yx-bindgen は `.h` からレイアウトを機械的に計算し、この手作業による保証のリスクを排除する。

## 提案

### 1. 使用方法

```bash
yx-bindgen --header /usr/include/sqlite3.h --lib sqlite3 --output sqlite3_bindings.yx
```

`--lib` はリンクするライブラリ名を指定し、`Native.c("libsqlite3")` ヘッダを生成する。

### 2. 生成内容

yx-bindgen は四種類のコンテンツを生成する：
- **ライブラリバインディングヘッダ**：`lib = Native.c("libxxx")`
- **型の二分**：ブラックボックスポインタ → 不透明ハンドル；データ構造 → 透明型（レイアウト付き）
- **関数宣言**：`lib("symbol")` バインディング
- **メソッドバインディング**：`Type.method` または `[N]`

### 3. 型の二分の自動判定

yx-bindgen は C 型の使用法に従って帰属を判定する（RFC-026 §2）：

| C 型 | 判定 | yx-bindgen 出力 |
|--------|------|----------------|
| `typedef struct T T;`（不完全型/ポインタのみ使用） | ブラックボックス → 不透明ハンドル | `T: Type = lib("T")` |
| `struct T { fields };`（フィールド可視、読み書きされる） | データ → 透明型 | `T: Type = { ...レイアウト }` |
| `int/long/float/double` | 値 | `Int32/Int64/Float32/Float64` |
| `char*`（引数/戻り値） | 値（コピー） | `String` |
| `void*` | 型なし | `*Void`（システムレベル） |

不完全型（`typedef struct sqlite3 sqlite3;` 定義体なし）→ 必然的にブラックボックスハンドル。完全定義の struct → 透明型、フィールドレイアウトは機械的に計算。

### 4. レイアウト計算（透明型の鍵）

完全定義の struct に対して、yx-bindgen はターゲットプラットフォームの ABI に従って各フィールドのオフセット、サイズ、アラインメントを計算する：

```c
struct timespec {
    time_t tv_sec;    // プラットフォーム依存：Linux x86_64 = 8 バイト
    long   tv_nsec;   // 8 バイト
};
```

```yaoxiang
// ターゲットプラットフォーム Linux x86_64
Timespec: Type = {
    tv_sec: Int64,    // offset 0, size 8
    tv_nsec: Int64    // offset 8, size 8
}
// 総サイズ 16、アラインメント 8 —— C struct とバイナリ一致
```

**プラットフォーム差異**：`time_t`、`long`、`size_t` のサイズはプラットフォームによって変化する。yx-bindgen は `--target` に応じて正しいマッピングを選択し、生成された透明型がターゲットプラットフォームの C struct とバイト単位で一致することを保証する。これは RFC-026 §2.2 の手作業レイアウト保証リスクを排除する核心的価値である。

### 5. 生成例

入力（`sqlite3.h` 抜粋）：

```c
typedef struct sqlite3 sqlite3;          // 不完全 → ブラックボックスハンドル
typedef struct sqlite3_stmt sqlite3_stmt;

int sqlite3_open(const char *filename, sqlite3 **ppDb);
int sqlite3_close(sqlite3 *db);
int sqlite3_exec(sqlite3 *db, const char *sql, ...);
```

出力（`sqlite3_bindings.yx`）：

```yaoxiang
// sqlite3_bindings.yx —— 自動生成、手動編集不可

// ============================================================================
// ライブラリバインディング
// ============================================================================
sqlite3 = Native.c("libsqlite3")

// ============================================================================
// 型（不完全型 → 不透明ハンドル）
// ============================================================================
SqliteDb: Type = sqlite3("sqlite3")
SqliteStmt: Type = sqlite3("sqlite3_stmt")

// ============================================================================
// 関数 + メソッドバインディング
// ============================================================================
SqliteDb.open: (filename: String) -> ?SqliteDb = sqlite3("sqlite3_open")
SqliteDb.exec: (sql: String) -> Int32 = sqlite3("sqlite3_exec")
SqliteDb.close: () -> Int32 = sqlite3("sqlite3_close")

// ============================================================================
// 析構（オプション、ユーザーが確認）
// ============================================================================
SqliteDb.drop = SqliteDb.close
```

### 6. ユーザー調整

生成されたバインディングはドラフトであり、ユーザーは C ライブラリのドキュメントに従って所有権セマンティクス（yx-bindgen は推測しない）を確認する：

```yaoxiang
// 生成：デフォルト String（コピー）—— ほとんどの場合正しい
SqliteDb.errmsg: () -> String = sqlite3("sqlite3_errmsg")

// getenv は静的領域を返すため、コピーも所有権取得もすべきでない —— ユーザーがベアポインタに変更
getenv: (name: String) -> *const U8 = Native.c("libc")("getenv")

// ライブラリ所有のハンドル、ユーザーが .drop バインディングを確認
SqliteDb.drop = SqliteDb.close   // 生成された提案、ユーザーが確認
```

**鍵**：レイアウトは yx-bindgen によって機械的に保証される（プラットフォーム正しい）が、所有権セマンティクス（`.drop` の有無、`char*` がコピーかベアポインタか）はユーザーが C ドキュメントに従って確認する。ツールは機械的正しさを担当し、ユーザーはセマンティック正しさを担当する。

## トレードオフ

### 利点

1. **レイアウト保証の自動化**：透明型レイアウトはプラットフォームごとに機械的に計算され、手作業の padding/アラインメントエラーを排除
2. **型の二分の自動判定**：不完全型 → ハンドル、完全 struct → 透明型
3. **監査可能**：出力は通常の `.yx` であり、ユーザーは読み取り、修正、バージョン管理にコミット可能

### 欠点

1. **所有権は依然として人手による確認が必要**：yx-bindgen は `.drop` を推測せず、`char*` のセマンティクスも推測しない
2. **C ヘッダ解析の依存**：libclang または tree-sitter-c が必要
3. **プラットフォーム特化**：異なる `--target` は異なるレイアウトを生成し、クロスプラットフォームパッケージには複数コピーまたはランタイム選択が必要
