# RFC-033: `^^` 反射演算子

> **参考**:
>
> - [RFC-010: 統一型構文 - name: type = value モデル](../accepted/010-unified-type-syntax.md)
> - [RFC-011: ジェネリクスシステム設計](../accepted/011-generic-type-system.md)
> - [RFC-027: コンパイル時述語と統一静的検証](../accepted/027-compile-time-evaluation-types.md)
> - [RFC-011a: インターフェース実装と動的ディスパッチ](../draft/011a-interface-implementation.md)

## 概要

本文書では、型と値のメタデータにアクセスするための反射エントリーポイントとして `^^` 演算子を導入することを提案する。`^^T` は型 `T` の静的メタデータオブジェクトを返し、`^^obj` は値 `obj` の動的型メタデータを返す。メタデータオブジェクトは通常のレコード型であり、名前、パラメータ、フィールドなどの情報を含み、コンパイル時と実行時の両方で使用できる。

## 動機

### なぜこの機能が必要なのか？

1. **シリアライズ/デシリアライズ**：型のフィールド情報にアクセスし、シリアライズコードを自動生成する必要がある
2. **コンパイル時メタプログラミング**：コンパイル時に型構造にアクセスし、コード生成や制約検証を行う必要がある
3. **実行時デバッグ/ツール**：実行時に型情報を出力し、デバッグを支援する必要がある
4. **実行時型チェック**：実行時に型関係を判定する必要がある（例：「obj は何の型か？」）

### 現在の問題

現在のYaoXiangには反射機構がなく、コンパイル時または実行時に型のメタデータにアクセスできない。`.name` や `.fields` を直接使用して型メタデータにアクセスすると、ユーザーが定義したフィールドと衝突する：

```yaoxiang
Person: Type = { name: String, age: Int }

# Person.name は型メタデータの名前か、フィールド name か？
# これは解析の困難さと意味の混乱を引き起こす
```

通常のフィールド名前空間を侵食しない構文で型メタデータにアクセスする方法が必要である。

## 提案

### 中核設計

通常コードとメタデータクエリを明確に区別するため、反射エントリーポイントとして `^^` 演算子を導入する。

**2つの使用方法**：

1. **静的反射（型に作用）**：`^^T` は型 `T` の静的メタデータオブジェクトを返す
2. **動的反射（値に作用）**：`^^obj` は値 `obj` の動的型メタデータを返す

**メタデータ構造**：

```yaoxiang
TypeMeta: Type = {
    name: String,
    params: Array(ParamMeta),
    fields: Array(FieldMeta),
    return_type: Type,
    refinement: Option(Expr)  # コンパイル時は Some(Expr)、実行時は None
}

ParamMeta: Type = {
    name: String,
    type: Type
}

FieldMeta: Type = {
    name: String,
    type: Type
}
```

**宇宙階層**：`T: Type_n` の場合、`^^T: Type_{n+1}` となり、型論の標準的な宇宙昇格規則に準拠する。

**優先度**：`^^` は単項前置演算子であり、優先度が最も高い。`^^T.name` は `(^^T).name` と等価である。

### 例

#### 基本的な使用

```yaoxiang
Point: Type = { x: Float, y: Float }

# 静的反射
meta = ^^Point
print(meta.name)           # "Point"
print(meta.fields.len)     # 2
print(meta.fields[0].name) # "x"
print(fields[0].type)      # Float

# 動的反射（実行時反射を有効にする必要がある）
obj = Point(1.0, 2.0)
meta = ^^obj
print(meta.name)           # "Point"
```

#### ジェネリクス型

```yaoxiang
List: (T: Type) -> Type = { data: Array(T), length: Int }

# ジェネリクス型自体を反射
meta = ^^List
print(meta.name)           # "List"
print(meta.params)         # [{ name: "T", type: Type }]

# 具体化されたインスタンス型を反射
meta = ^^List(Int)
print(meta.name)           # "List(Int)"
print(meta.params)         # []
```

#### 関数

```yaoxiang
add: (a: Int, b: Int) -> Int = a + b

meta = ^^add
print(meta.name)           # "add"
print(meta.params)         # [{ name: "a", type: Int }, { name: "b", type: Int }]
print(meta.return_type)    # Int
```

#### 精化型

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

# コンパイル時：refinement は Some(Expr)
meta = ^^Positive
print(meta.name)           # "Positive"
print(meta.refinement)     # Some(AST(x > 0))

# 実行時：refinement は None（消去される）
```

#### コンパイル時述語での使用

```yaoxiang
# 型がフィールドを持つかチェック
HasFields: (T: Type) -> Type = { ^^T.fields.len > 0 }

# フィールドの型をチェック
HasFloatField: (T: Type) -> Type = {
    exists field in ^^T.fields: field.type == Float
}

# 使用
obj: HasFields(Point) = Point(1.0, 2.0)  # ✅ 検証成功
# obj: HasFields(Int) = 42  # ❌ 検証失敗
```

#### シリアライズの例

```yaoxiang
# コンパイル時純粋関数：JSON 文字列を生成
to_json: (T: Type) -> ((obj: T) -> String) = {
    meta = ^^T
    parts: Array(String) = []
    for field in meta.fields {
        # コンパイル時にフィールドアクセスコードを生成
        parts.push("\"${field.name}\": ${obj.${field.name}}")
    }
    return "{" + parts.join(", ") + "}"
}

# 使用
point_to_json = to_json(Point)
print(point_to_json(Point(1.0, 2.0)))  # '{"x": 1.0, "y": 2.0}'
```

### 構文の変化

| 以前 | 以後 |
|------|------|
| 反射機構なし | `^^T` で型メタデータを取得 |
| 反射機構なし | `^^obj` で値の動的型メタデータを取得 |

## 詳細設計

### 型システムへの影響

- **新規型**：`TypeMeta`、`ParamMeta`、`FieldMeta`
- **宇宙階層**：`^^T` の戻り型は `T` より1階層上位になる
- **ジェネリクスとの相互作用**：`^^List` と `^^List(Int)` の両方をサポート
- **関数との相互作用**：`^^add` は関数のメタデータ（パラメータと戻り型を含む）を返す
- **精化型との相互作用**：`^^Positive` は精化型のメタデータ（精化式を含む）を返す

### 実行時の動作

**コンパイル時反射**：
- `^^T` はコンパイル時に完全に評価され、結果は定数としてインライン化される
- 精化式はコンパイル時に利用可能

**実行時反射**：
- デフォルトでは無効、ゼロオーバーヘッド
- `--enable-runtime-reflection` コンパイルオプションで有効化
- 有効化後、`^^obj` は動的型メタデータを返す
- 精化式は実行時に `None` として消去される

**オンデマンド生成 + ツリーシェイク**：
- `^^` が実際に使用される型に対してのみメタデータを生成
- 参照されていない型はメタデータを生成しない（ツリーシェイク）

### コンパイラの改修

1. **字句解析器**：`^^` を単一トークンとして認識
2. **構文解析器**：`^^` 前置式のルールを追加
3. **型システム**：`TypeMeta`、`ParamMeta`、`FieldMeta` 型の定義を追加
4. **型検査器**：各型のメタデータインスタンスを生成
5. **コンパイル時評価器**：`^^T` のコンパイル時評価をサポート
6. **実行時（オプション）**：反射される型に対してRTTIを生成

### 後方互換性

- ✅ 既存構文への影響なし：`^^` は新演算子であり、既存構文と衝突しない
- ✅ 既存型への影響なし：すべての型は自動的に `^^` をサポート
- ✅ 既存関数への影響なし：関数は `^^` を使用できるが、必須ではない
- ✅ コンパイル時述語への影響なし：述語内の `^^T` は通常の内容と同様に動作
- ✅ 実行時への影響なし：実行時反射はデフォルトで無効、ゼロオーバーヘッド

## トレードオフ

### 利点

- **統一性**：関数、ジェネリクス、精化型を統一的に処理
- **ゼロオーバーヘッド**：コンパイル時反射は完全に消去、実行時反射はオプション
- **既存システムとの統合**：コンパイル時述語（RFC-027）とシームレスに統合
- **簡潔さ**：`^^` は純粋な記号であり、ユーザーが定義した識別子と衝突しない
- **オンデマンド生成**：ツリーシェイクによる最適化で、未使用の型はゼロオーバーヘッド

### 欠点

- **学習曲線**：`^^` のセマンティクスとメタデータ構造を理解する必要がある
- **実行時オーバーヘッド**：実行時反射を有効にするとメモリオーバーヘッドが増加（インスタンスごとに1つのポインタ）
- **実装の複雑さ**：コンパイラの複数のコンポーネントを修正する必要がある

## 代替案

| 案 | 選択しない理由 |
|------|--------------|
| `reflect(T)` 関数 | スコープに追加の識別子を導入し、ユーザーがシャドウイングする可能性がある |
| `type_info(T)` 関数 | 同上 |
| 単一の `^` 演算子 | ビット演算と衝突する可能性があり、C++26でも衝突を避けるために `^^` が選ばれた |
| `@@`、`##` などの記号 | 先例がなく、`^^` ほど説明的でない |

## 実装段階

| 段階 | 内容 | 依存関係 |
|------|------|------|
| Phase 1 | コンパイル時の `^^` 演算子解析 | なし |
| Phase 2 | `TypeMeta` データ構造の定義 | Phase 1 |
| Phase 3 | コンパイル時メタデータ生成 | Phase 2 |
| Phase 4 | 実行時反射のサポート（オプション） | Phase 3 |
| Phase 5 | コンパイル時述語との統合 | Phase 3 |

### 依存関係

```
Phase 1 (解析)
    ↓
Phase 2 (データ構造)
    ↓
Phase 3 (コンパイル時メタデータ)
    ↓
    ├────────────┐
    ↓            ↓
Phase 4        Phase 5
(実行時反射)  (コンパイル時述語)
```

### リスク

- **解析の衝突**：`^^` が既存構文と衝突する可能性（解析の結果、衝突なし）
- **パフォーマンスへの影響**：コンパイル時メタデータ生成がコンパイル時間を増加させる可能性（ツリーシェイク最適化で緩和可能）
- **実行時オーバーヘッド**：実行時反射を有効にするとメモリオーバーヘッドが増加（オンデマンド生成で緩和）

## 未解決の問題

- [x] `^^` の作用範囲：型と値のみに作用し、式には作用しない
- [x] チェーンアクセス：サポート、`^^T` が返すメタデータオブジェクトは通常通り属性にアクセス可能
- [x] パターンマッチング：サポート、`TypeMeta` は通常レコード型であり、通常通りパターンマッチング可能
- [x] 比較：サポート、同型のメタデータオブジェクトは等価
- [x] メモリオーバーヘッド：オンデマンド生成 + ツリーシェイク最適化

### 付録B：用語集

| 用語 | 定義 |
|------|------|
| 反射 | 実行時またはコンパイル時に型メタデータにアクセスする能力 |
| メタデータ | 型構造を記述する情報（名前、フィールド、パラメータなど） |
| RTTI | Run-Time Type Information、実行時型情報 |
| ツリーシェイク | コンパイラ最適化、未使用コードの削除 |
| 精化型 | 制約条件を持つ型、例：`Positive: (x: Int) -> Type = { x > 0 }` |
