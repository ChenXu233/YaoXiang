# RFC-030: assert アサーション機構

## 要約

YaoXiang に `assert` アサーション機構を導入し、テストと事前条件チェックに使用する。本 RFC では 2 つの実装案——**native 関数**と**キーワード**——を提案し、それぞれのトレードオフを分析して意思決定に資する。

## 動機

### なぜこの機能が必要か？

現在の YaoXiang の E2E テストは `if` + `io.println` + `return` でアサーションをシミュレートすることしかできない：

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

この記述には 3 つの問題がある：

1. **ボイラープレートが多い**：各アサーションに 4 行必要で、テストファイルが肥大化する
2. **エラーメッセージが弱い**：文字列を手動で連結するため、ソースコード位置情報がない
3. **合成できない**：アサーションの一括登録や、テストフレームワークへの引数渡しができない

### 現状の問題

- 統一されたアサーション機構がない
- テストコードに `if` + 出力 + `return` のパターンが蔓延している
- バイトコード層には既に `Throw` 命令が存在するが、言語層では公開されていない

## 案 A：native 関数

`assert` を native 関数として実装し、新しいキーワードを導入しない。

```yaoxiang
use std.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### 型シグネチャ

```
assert: (Bool, String) -> ()
```

### ランタイム動作

1. 第 1 引数 `condition: Bool` を評価する
2. `true` ならば `Unit` を返す
3. `false` ならばランタイム panic を発動する：
   - `message` の内容を出力
   - （debug モードで）呼び出しスタックを出力
   - 現在の実行を終了

### コンパイラの変更点

**parser、AST、typecheck、IR gen の変更は不要。**

`src/std/` 配下に native 関数の登録を追加するのみ：

1. `src/std/assert.rs` を新規追加
2. `std.assert` native 関数を登録
3. 内部で既存のランタイム panic 機構を呼び出す

### 利点

- **構文変更ゼロ**：parser、AST、IR、codegen の変更が不要
- **実装が単純**：native 関数の登録コードで約 20 行
- **第一級オブジェクト**：引数として渡したり、変数に代入したり、高階関数で使用したりできる
- **漸進的な拡張**：将来的にコードを壊すことなくコンパイル時インライン化を追加可能

### 欠点

- **コンパイル時解析が効かない**：`assert(false)` 以降のコードに対してコンパイラがデッドコード除去を行えない
- **エラーメッセージが限定的**：ソースコード位置はランタイム呼び出しスタックから取得するため、キーワードほど正確ではない
- **シャドーイング可能**：ユーザが同名の関数を定義して `assert` を上書きできる
- **import が必要**：使用するには `use std.assert` の記述が必須

## 案 B：キーワード

`assert` を言語のキーワードとして導入し、コンパイラが IR 層で特殊処理する。

```yaoxiang
main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### 構文

```
assert '(' Expr ',' StringLit ')'
```

`assert` はキーワードであり、`use` でのインポートは不要。

### コンパイラの変更点

全工程にわたる変更が必要：

| 段階 | 変更内容 |
|------|----------|
| Lexer | `KwAssert` トークンを追加 |
| Parser | `assert(expr, "msg")` 構文を追加 |
| AST | `StmtKind::Assert { condition, message }` を追加 |
| IR Gen | Assert を条件ジャンプ + Throw にコンパイル |
| Codegen | `JmpIfNot` + `Throw` 命令列を生成 |

### 利点

- **組み込みで使用可能**：import 不要で任意の場所で直接使用可能
- **ソースコード位置が正確**：コンパイラが AST 層で assert の位置を把握しており、エラーメッセージが行番号・列番号まで正確
- **シャドーイング不可**：キーワードはユーザコードで上書きできない
- **将来的な拡張が可能**：コンパイル時解析（定数畳み込み pass との組み合わせ）が可能
- **意味が明確**：`assert` は構文層で明確なセマンティクスを持つため、誤用されない

### 欠点

- **実装コストが高い**：parser、AST、IR、codegen の全工程の変更が必要
- **第一級オブジェクトではない**：引数として渡したり、変数に代入したりできない
- **キーワードの肥大化**：キーワードを追加するたびに言語の複雑度が増す
- **拡張性が低い**：新たなシグネチャのバリアント（例：`assert_eq`）を追加するには構文変更が必要

## 比較

| 観点 | 案 A：関数 | 案 B：キーワード |
|------|------------|------------------|
| 実装コスト | 低（~20 行） | 高（~200 行、4 モジュール） |
| 構文変更 | なし | あり（新キーワード + 新構文） |
| import の要否 | 必要 | 不要 |
| 第一級オブジェクト | 該当する | 該当しない |
| ソースコード位置の正確性 | ランタイム呼び出しスタック | コンパイル時に行・列まで正確 |
| シャドーイング不可 | いいえ | はい |
| コンパイル時解析 | 不可（定数畳み込みが必要） | 可（定数畳み込みと組み合わせ） |
| 拡張性 | 高（オーバーロード・高階関数） | 低（構文変更が必要） |

### コンパイル時解析の現実的な制約

案 B の核心的利点——コンパイル時解析——を機能させるには、**定数畳み込み pass** が必要である。すなわち、コンパイラがコンパイル時に `assert(false)` 中の `false` を評価して、これがデッドコードであることを認識する必要がある。

YaoXiang には現在、定数畳み込み pass が存在しない。仮に案 B を採用したとしても、`assert(x > 0)` のような一般的な記述はコンパイル時には解析できない。解析可能なのは `assert(true)` / `assert(false)` のようなリテラルの場合のみである。

したがって案 B のコンパイル時における利点は**現時点では理論上のものであって、実質的なものではない**。
