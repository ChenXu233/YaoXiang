## 概要

本文書は YaoXiang 言語の**量子ネイティブサポート**と**マルチバックエンド統合**方案を定義する。核心的な考え方：**YaoXiang の既存設計（デフォルト Move、所有権回流、不透明型、DAG スケジューラ、ジェネリクス定数パラメータ）は本質的に量子プログラミング言語の完全な基盤を構成しており、新しい量子専用構文を導入する必要はない**。少数の組み込み型（`Qubit`、`Complex`、`Topology`）と組み込み関数（量子ゲート、測定、トポロジ制約）を追加し、既存の言語機構を活用することで、量子ネイティブセマンティクス、量子利用率の自動最大化、ハイブリッド古典プログラミング、およびマルチバックエンドサポートを実現する。

## 動機

### なぜ量子ネイティブサポートが必要か？

現在の量子プログラミングエコシステムには深刻な断絶がある：

- **低水準言語（QCIS、OpenQASM）**：物理量子ゲートを直接操作するが、型システムや抽象化機構が欠如しており、複雑なアルゴリズムを書くことが困難。
- **高水準フレームワーク（Qiskit、Cirq、Q#）**：古典言語（Python、C#）を拡張しており、量子セマンティクスはライブラリで実装されるため：
  - 量子状態コピー禁止則はユーザーが手動で遵守する必要がある（または線形型システムの後付けに依存）。
  - 量子ゲート操作と古典コードの構文が分離しており、学習コストが高い。
- **ハイブリッド計算**：量子部分と古典部分は明示的に分離する必要があり、统一的なデータフローモデルが欠如している。

### 現在の問題

YaoXiang の既存設計はこれらの問題を解決するために必要な完全基盤を提供している：

| 量子計算ニーズ | YaoXiang 既存設計 | 説明 |
|-------------|-------------------|------|
| 量子状態コピー禁止 | **デフォルト Move セマンティクス** | 代入時に所有権が移動し、暗黙的なコピーがないため、no-cloning 定理に自然に準拠 |
| 量子ゲートはユニタリ変換 | **所有権回流** | `q = H(q)` は元の qubit を消費し、新しい qubit を返すprecisely ゲートセマンティクスに対応 |
| 量子もつれ状態 | **不透明型** | `BellPair` は全体としてのみ操作可能で、コンパイラがライフサイクルを追跡し、不正な分解を防止 |
| 物理トポロジ制約 | **ジェネリクス定数パラメータ** | `Qubit(Topology, N)` コンパイル時に隣接性を検査 |
| 測定崩壊 | **空状態の再利用** | 測定後 qubit が空になり、再初期化可能で、量子状態崩壊を模倣 |
| 量子回路の自動並列化 | **DAG スケジューラ** | 関数内の文はデータ依存関係に基づいて自動並列化され、依存関係のないゲートは自然に並行実行 |
| ハイブリッド古典-量子制御フロー | **統一構文** | 量子操作と古典操作は同じ `name: type = value` 形式を使用 |

**YaoXiang は「量子サポートを追加する」のではなく、自身の設計がすでに量子ネイティブであることを発見した。**

> **セマンティクス説明**：本文書は量子操作を表現するために YaoXiang の**所有権セマンティクス**を使用している。コンパイラレベルで no-cloning（所有権安全性）を保証する。言語レベルの「消費-生成」は**所有権移転**の構文表現である——消費 = 所有権の取得、戻り値 = 所有権の移行。基盤となる実装は真正な可逆量子ゲート（量子状態をインプレースで変更）であり、実際には「新量子状態を生成」するわけではない。

### 設計目標

1. **新構文ゼロ**：`quantum`、`circuit` などのキーワードを導入せず、すべての量子特性は既存の言語機構で表現する。
2. **型安全性**：コンパイラが量子状態がコピー・不正使用されないことを保証する。
3. **トポロジ制約のコンパイル時検査**：ジェネリクス定数パラメータ `Qubit(T, N)` を通じて、コンパイル時に2量子ビットゲート操作が物理トポロジ是否符合かを検証する。
4. **マルチバックエンド透過性**：同一の量子コードは QIR（汎用エコシステム）または QCIS（国内量子命令セット）にコンパイルでき、コマンドライン引数で切り替え可能。
5. **ハイブリッド古典のシームレス化**：量子計算は古典関数を自由に呼び出すことができ、古典コードも量子データを操作できる（`ref` 共有経由だが、所有権制約に従う）。

## 提案

### 核心設計

#### 1. 量子型システムマッピング

**基本型**：
```yaoxiang
Qubit: Type0 = primitive_qubit
Complex: Type0 = { re: Float, im: Float }
```
- `Qubit` は第一級市民型であり、所有権規則（Move、RAII）に従う。
- `Complex` は振幅を表現するために使用され、コンパイラはインライン最適化できる。

**量子ゲートは関数である**：
```yaoxiang
# 組み込み関数シグネチャ
H: (Qubit) -> Qubit = builtin_hadamard
X: (Qubit) -> Qubit = builtin_pauli_x
Y: (Qubit) -> Qubit = builtin_pauli_y
Z: (Qubit) -> Qubit = builtin_pauli_z
CNOT: (control: Qubit, target: Qubit) -> { Qubit, Qubit } = builtin_cnot
```
- すべてのゲートは入力 qubit を消費し、新しい qubit（または量子もつれ対）を返す。所有権回流構文 `q = H(q)` は数学的セマンティクスに直接対応する。
- 複数 qubit ゲートは構造体を返し、パターン照合またはフィールドアクセスで結果を取得する。

**測定**：
```yaoxiang
measure: (Qubit) -> Int = builtin_measure   # qubit を消費し、古典ビットを返す
measure_all: (List(Qubit)) -> List(Int) = builtin_measure_all
```
- 測定後 qubit は消費され（空になり）、ユーザーは空状態の再利用を通じて再初期化できる。

**初期化**：
```yaoxiang
qubit: (Int) -> Qubit = builtin_qubit   # 0 または 1 で初期化
```

#### 2. 量子もつれと不透明型カプセル化

量子もつれ対を不透明型にカプセル化し、結合操作のみを提供し、分解を禁止する：

```yaoxiang
# 組み込み不透明型
BellPair: Type0 = primitive_bell_pair

# 組み込み関数 - 全体としてのみ操作可能
CNOT: (Qubit, Qubit) -> BellPair
measure_bell: (BellPair) -> { Int, Int }
split_bell: (BellPair) -> { Qubit, Qubit }  # 量子もつれ対の分割（注意して使用）
apply_cnot_to_bell: (BellPair, Qubit) -> BellPair
```

**重要な設計**：
- フィールドアクセッサを提供せず、組み込み関数を介して全体としてのみ操作可能
- `measure_bell(bp)` は量子もつれ対全体を消費し、古典ビットを返す
- コンパイラは量子もつれ対の完全なライフサイクルを追跡可能

**Python/Qiskit との比較**：
```
Python (Qiskit): 実行時に回路を構築し、エラーは送信後に発見される場合がある
YaoXiang:       コンパイル時にほとんどの論理エラーを捕捉
```

**残りの 10%**（物理的デコヒーレンス、ゲート誤差など）はハードウェア問題であり、言語で解決できるものではない。

#### 3. 物理トポロジ制約

量子チップは制約付きトポロジグラフであり、**任意の2つの qubit で2量子ビットゲートを実行できるわけではない**。YaoXiang は**ジェネリクス定数パラメータ**を使用してコンパイル時にトポロジ制約を保証する。

**トポロジ型定義**：
```yaoxiang
# トポロジは型であり、隣接行列を含む
Topology: Type0 = primitive_topology

# 組み込みトポロジ定数
Linear8: Topology = topology(8)          # 線形8ビット: 0-1-2-3-4-5-6-7
Grid3x3: Topology = topology(3, 3)        # 3x3 グリッド
Ring16: Topology = topology(16, ring)    # リング16ビット
```

**Qubit はトポロジと位置をバインド**：
```yaoxiang
# Qubit(T, N) - T はトポロジ型、N は定数位置パラメータ
q0: Qubit(Grid3x3, 0)   # Grid3x3 トポロジ、位置 (0,0)
q1: Qubit(Grid3x3, 1)   # Grid3x3 トポロジ、位置 (0,1)
q2: Qubit(Grid3x3, 2)   # Grid3x3 トポロジ、位置 (0,2)
q3: Qubit(Grid3x3, 3)   # Grid3x3 トポロジ、位置 (1,0)
```

**ゲート操作は自動的に制約される**：
```yaoxiang
# CNOT 型シグネチャはトポロジ制約を含む
CNOT: (T: Topology, I: Int, J: Int) -> (
    (Qubit(T, I), Qubit(T, J)) -> { Qubit(T, I), Qubit(T, J) }
) when adjacent(T, I, J)

# コンパイル時チェック
CNOT(q0, q1)  # ✅ Grid3x3 で (0,0) と (0,1) は隣接
CNOT(q0, q2)  # ❌ コンパイルエラー：(0,0) と (0,2) は隣接しない
```

**`adjacent` コンパイル時制約**：
- `adjacent` はコンパイル時関数であり、トポロジの隣接行列を使用して静的チェックを行う
- 定数インデックスの場合、100% コンパイル時検証
- 動的インデックスの場合、実行時チェックコードを生成

**仮想から物理へのマッピング**：
```yaoxiang
# コンパイル時に具体的な物理位置がわからない場合、型推論を使用
q = qubit(Grid3x3)  # 位置 0 を自動割り当て、以降推論可能
```

#### 4. 所有権と量子状態の線形フロー

すべての量子操作は Move セマンティクスを遵守し、qubit がコピーされないことを確保する：
```yaoxiang
q = qubit(0)
q2 = q          # ❌ コンパイルエラー：q はすでに移動済み、再使用不可
q = H(q)        # ✅ q を消費し、新しい q を返す
measure(q)      # ✅ q を消費した後、q は空になる
q = qubit(0)    # ✅ 空状態の再利用
```

#### 4. 自動並列化と DAG スケジューリング

Standard または Full Runtime では、DAG スケジューラが量子プログラムを自動的に分析する：
```yaoxiang
apply_two_qubit_gates: () -> {Qubit, Qubit} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    # 上記2行はデータ依存関係がなく、DAG は自動並列実行
    CNOT(q1, q2)   # q1 と q2 に依存、自动等待
}
```
- スケジューラは `num_workers` 設定（物理量子プロセッサ数）を活用して真の並列性を実現する。
- ユーザーは手動でゲートの順序を arrang する必要はなく、データフローを記述するだけでよい。

#### 5. ハイブリッド古典計算

古典コードと量子コードは完全に融合する：
```yaoxiang
grover_search: (target: Int) -> Int = () => {
    n = 4
    qubits = List(Qubit)()
    for i in 0..n {
        qubits.append(H(qubit(0)))
    }
    # 古典ループと量子操作が混合
    oracle(qubits, target)   # oracle は量子ゲート列
    qubits = diffusion(qubits)
    results = measure_all(qubits)
    return decode_result(results)   # 古典後処理
}
```
- 同一関数内で量子ゲートと古典制御フローを自由に混合できる。
- 所有権システムは、量子変数が古典分支で誤ってコピーされないことを保証する。

#### 6. マルチバックエンドサポートアーキテクチャ

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   YaoXiang ソース   │     │   型チェック      │     │   DAG 中間表現  │
│   (統一構文)     │────▶│   + 所有権分析    │────▶│   (データフローグラフ) │
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                          │
                                                          ▼
                          ┌─────────────────────────────────────────────┐
                          │           コード生成バックエンド (プラグイン可能)            │
                          ├─────────────────┬───────────────────────────┤
                          │  QIR バックエンド     │  QCIS バックエンド                │
                          │  (汎用エコシステム)    │  (国内量子命令セット)         │
                          ├─────────────────┼───────────────────────────┤
                          │  - .ll ファイル出力   │  - .qcis テキスト出力        │
                          │  - 複数 QPU 対応      │  - 中科院/国盾ハードウェア対応    │
                          └─────────────────┴───────────────────────────┘
```

- **コンパイルフロー**：フロントエンド統一 → DAG 構築 → バックエンド選択 → ターゲットコード生成。
- **QIR バックエンド**：DAG ノードを QIR の量子ゲート intrinsic にマッピングし、LLVM ビットコードを生成、さらに LLVM 最適化を活用可能。
- **QCIS バックエンド**：DAG を QCIS 命令（例：`H q0`）にシリアライズし、量子チップコンソールに直接送信可能。

### 例

#### ベル状態生成と測定
```yaoxiang
bell_measure: () -> {Int, Int} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    bell = CNOT(q1, q2)  # BellPair 不透明型を返す
    result = measure_bell(bell)  # 2ビットを一度に測定
    return result
}
```

#### 量子テレポーテーション（簡略化）
```yaoxiang
teleport: (msg: Qubit, bell: BellPair) -> Qubit = (msg, bell) => {
    # 量子もつれ対を分割して2つの独立した qubit を取得
    (alice_qubit, bob_qubit) = split_bell(bell)

    # Alice の操作
    (msg, alice_qubit) = CNOT(msg, alice_qubit)
    msg = H(msg)
    a1 = measure(msg)
    a2 = measure(alice_qubit)

    # 古典情報伝送 (スケジューラが自動的に依存関係を処理)
    # Bob の操作
    if a2 == 1 { bob_qubit = X(bob_qubit) }
    if a1 == 1 { bob_qubit = Z(bob_qubit) }
    return bob_qubit
}
```

## 詳細設計

### 組み込み型と関数定義

`compiler/builtins` モジュールに追加：
```rust
builtins.insert("Qubit", Ty::Primitive(Primitive::Qubit));
builtins.insert("Complex", Ty::Record(vec![
    ("re", Ty::Primitive(Primitive::Float)),
    ("im", Ty::Primitive(Primitive::Float)),
]));

// 量子ゲート
for (name, sig) in GATES {
    builtins.insert(name, Ty::Function(vec![Ty::Qubit], Ty::Qubit));
}
builtins.insert("CNOT", Ty::Function(
    vec![Ty::Qubit, Ty::Qubit],
    Ty::Record(vec![
        ("q0", Ty::Qubit),
        ("q1", Ty::Qubit)
    ])
));
builtins.insert("measure", Ty::Function(vec![Ty::Qubit], Ty::Primitive(Primitive::Int)));
builtins.insert("qubit", Ty::Function(vec![Ty::Primitive(Primitive::Int)], Ty::Qubit));
```

### 所有権チェッカーの Qubit 特殊処理

- `Qubit` は `!Copy`（デフォルト Move）としてマークされ、暗黙的なコピーが禁止される。
- 測定関数 `measure` のパラメータは `Qubit`（値渡し）であり、所有権を消費する。
- 複数 qubit ゲートが返すレコード型では、フィールドはすべて `Qubit` であり、所有権規則に従う必要がある。

### DAG スケジューラの量子ゲート最適化

- 量子ゲートノードは純関数（副作用なし）として扱われ、スケジューラは依存関係のないゲートを自由に再配置できる。
- スケジューラが「量子命令列」を出力する際、データ依存関係を保持し、並列ゲートをグループ化する（複数量子プロセッサ向け）。
- `--target-num-qubits` と `--target-topology` の設定をサポートし、後続のレイアウトとルーティングに使用（将来拡張）。

### QIR バックエンド詳細マッピング

| YaoXiang 操作 | QIR 命令 |
|---------------|----------|
| `H(q)` | `call void @__quantum__qis__h__body(%Qubit* %q)` |
| `CNOT(q1, q2)` | `call void @__quantum__qis__cnot__body(%Qubit* %q1, %Qubit* %q2)` |
| `measure(q)` | `%result = call i1 @__quantum__qis__mz__body(%Qubit* %q)` |
| `qubit(0)` | `%q = call %Qubit* @__quantum__rt__qubit_allocate()` |

QIR バックエンドは LLVM の `-O2` をさらに活用して最適化し、QIR Alliance 互換ビットコードを出力する。

### QCIS バックエンド詳細マッピング

| YaoXiang 操作 | QCIS 命令 |
|---------------|-----------|
| `H(q)` (q は物理ビット 2 に対応) | `H 2` |
| `CNOT(q1,q2)` (q1→ビット 0, q2→ビット 1) | `CNOT 0 1` |
| `measure(q)` (ビット 0) | `M 0` |
| `qubit(0)` 初期化 | 最初の使用命令に暗黙的に含まれるため、追加命令不要 |

- 仮想 qubit（YaoXiang 変数）から物理ビットへのマッピングテーブルを維持する必要がある。
- トポロジ制約チェックをサポート（将来実装）。

### ハイブリッド古典コード生成

- 古典部分（ループ、条件、整数演算など）は通常通りローカルコード（x86/ARM）を生成し、FFI または埋め込み呼び出しで量子バックエンドとやり取りする。
- QIR バックエンドでは、古典部分は LLVM IR に降格可能であり、QIR と混合コンパイルできる。

### 型システムへの影響

- 新しい `Qubit` と `Complex` プリミティブ型が追加される。
- `Qubit` は自動的に Move セマンティクスを持ち、コピーが禁止される。
- 量子ゲート関数のシグネチャは型システムに登録する必要がある。

### 後方互換性

- ✅ 完全な後方互換性
- 新規組み込み型と関数の追加は、既存コードに影響しない
- 量子特性はオプションであり、有効にしない限り追加オーバーヘッドはない

## トレードオフ

### メリット

- **新構文ゼロ**：開発者は少数の組み込み関数を学ぶだけで量子プログラムを書ける。
- **型安全性**：所有権システムが自動的に qubit のコピーを防止し、一般的な量子プログラミングエラーを回避する。
- **自動並列化**：DAG スケジューラがゲートレベルの並列化を無料で提供し、追加のコンパイラ最適化が不要。
- **エコシステム互換性**：QIR バックエンドにより YaoXiang は複数の量子クラウドプラットフォームで動作可能；QCIS バックエンドにより自主制御を確保。
- **ハイブリッド能力**：古典量子融合は自然であり、複雑な量子アルゴリズム（Shor、Grover の古典制御など）の記述に適している。

### デメリット

- **Qubit 数の静的性**：現在の設計では qubit 数はコンパイル時に知られていると想定しており、動的割り当ては `List(Qubit)` を通じて行う必要があるが、`List` のヒープ割り当ては追加オーバーヘッドをもたらす可能性がある（最適化で緩和可能）。
- **測定後の再利用**：空状態の再利用により qubit の再初期化が可能だが、物理量子ビットには緩和時間が存在する可能性があり、実行時システムが処理する必要がある（現時点ではユーザーが担当）。
- **動的トポロジマッピング**：実行時に物理トポロジが才知道場合、コンパイル時チェックは有効にならず、実行時チェックコードを生成する必要がある（現在のバージョンは静的チェックのみサポート）。

## 代替方案

| 方案 | 採用しない理由 |
|------|--------------|
| `quantum` キーワードと `circuit` 型の導入 | 新構文を追加すると学習コストが高く、YaoXiang の簡潔設計原則に反する |
| 量子サポートをライブラリとしてのみ実装 | コンパイラが量子状態の安全性を保証できず、DAG スケジューラと深く統合できない |
| 量子ハードウェアの成熟を待ってからサポート | 量子プログラミング言語設計の重要なウィンドウ期を逃す |
| 既存の量子フレームワーク（Qiskit など）を再利用 | 量子セマンティクスはライブラリで実装されるため、型システムと所有権システムの安全性を 得られない |
| 量子サブ言語を別途設計 | 言語複雑度が増加しメンテナンスコストが高い |

## 実装戦略

### 段階的区分

| 段階 | 期間 | 内容 |
|------|------|------|
| フェーズ 1 | 1ヶ月 | 基礎量子型と組み込み関数：コンパイラに `Qubit`、`Complex` 型を追加し、組み込み関数の型チェックを実装し、所有権チェッカーを拡張 |
| フェーズ 2 | 1ヶ月 | DAG スケジューラの量子ゲート認識：DAG 構築ロジックを修正し、量子ゲートを純関数としてマークし、並列ゲートグループ出力機能を実装 |
| フェーズ 3 | 2ヶ月 | QIR バックエンドプロトタイプ：DAG から QIR コードジェネレータを実装し、LLVM を統合し、QIR シミュレータ接続で検証 |
| フェーズ 4 | 2ヶ月 | QCIS バックエンドプロトタイプ：DAG から QCIS 命令への翻訳を実装し、仮想-物理ビットマッピングを設計し、国内量子プラットフォーム接続で検証 |
| フェーズ 5 | 2ヶ月 | ハイブリッド古典增强：古典制御フローと量子ゲートの交差が正しくコード生成されることを保証し、`List(Qubit)` をサポートし、サンプルプログラムを追加 |
| フェーズ 6 | 2ヶ月 | 最適化とドキュメント：基本レイアウトとルーティングを実装し、ユーザーガイドと量子プログラミングチュートリアルを執筆し、プレビュー版をリリース |

### リスク

1. **量子ハードウェア可用性**：外部量子シミュレータと実際の QPU の可用性に依存する。
   - **軽減**：オープンソースシミュレータ（QIR runner、Qiskit Aer）への接続を優先し、実際の QPU は長期目標とする。

2. **バックエンド実装の複雑性**：QIR と QCIS 仕様は変更される可能性がある。
   - **軽減**：コード生成インターフェースを抽象化し、バックエンド差異を分離し、後続の適応を容易にする。

3. **パフォーマンス不確実性**：量子プログラムのパフォーマンス特性は古典プログラムと異なる。
   - **軽減**：パフォーマンスプロファイリングツールを提供し、ユーザーがゲートレベル並列化の効果を確認できるようにする。

## 開放問題

- [x] **トポロジ制約**：`Qubit(Topology, N)` ジェネリクス定数パラメータを通じてコンパイル時チェックとしてすでに実装済み。
- [ ] **動的量子レジスタ**：`List(Qubit)` は QCIS バックエンドでどのようにマッピングするか？対応する数の物理ビットを生成できるが、実行時割り当て機構が必要。
- [ ] **エラー緩和**：動的デカップリングなどの組み込みエラー緩和コンストラクトを提供するかどうか？最初はライブラリとして実装可能。
- [ ] **既存の量子 SDK との相互運用性**：QASM または QIR モジュールのインポート是否能？将来 FFI を検討可能。
- [ ] **自動レイアウトとルーティング**：仮想 qubit 数が物理 qubit 数を上回る場合、どのように自動マッピングするか？
