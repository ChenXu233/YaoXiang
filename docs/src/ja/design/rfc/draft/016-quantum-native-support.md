## 概要

本文書は YaoXiang 言語の**量子ネイティブサポート**と**マルチバックエンド統合**方案を定義する。核心思想：**YaoXiang の現行設計（デフォルト Move、所有権回流、不透明型、DAG スケジューラ、ジェネリック定数パラメータ）は量子プログラミング言語の完全な基盤を自然に構成しており、新たな量子専用構文を導入する必要はない**。少数の組み込み型（`Qubit`、`Complex`、`Topology`）と組み込み関数（量子ゲート、測定、トポロジ制約）を追加し、既存の言語機構を活用することで、量子ネイティブセマンティクス、量子利用率の自動最大化する並列化、ハイブリッド古典プログラミング、およびマルチバックエンドサポートを実現する。

## 動機

### なぜ量子ネイティブサポートが必要か？

現在の量子プログラミングエコシステムは深刻な分断状態にある：

- **低水準言語（QCIS、OpenQASM）**：物理量子ゲートを直接操作するが、型システムや抽象化機構がなく、複雑なアルゴリズムを書くのが困難。
- **高水準フレームワーク（Qiskit、Cirq、Q#）**：古典言語（Python、C#）を拡張し、量子セマンティクスをライブラリで実装するため：
  - 量子状態複製不可能定理はユーザーが手動で遵守する必要がある（または線形型システムの後付けに依存）。
  - 量子ゲート操作と古典コードの構文が分離しており、学習コストが高い。
- **ハイブリッド計算**：量子部と古典部の明示的な分離が必要で、统一的なデータフローモデルがない。

### 現状の問題

YaoXiang の既存設計はこれらの問題を解決するための完全な基盤を提供する：

| 量子計算の要件 | YaoXiang の既存設計 | 説明 |
|---------------|-------------------|------|
| 量子状態の複製不可 | **デフォルト Move セマンティクス** | 代入時に所有権が移動し、暗黙的コピーがないため、no-cloning 定理に自然に従う |
| 量子ゲートがユニタリ変換 | **所有権回流** | `q = H(q)` で元の qubit を消費し、新しい qubit を返す。ゲートのセマンティクスを正確に表現 |
| 量子もつれ状態 | **不透明型** | `BellPair` は全体としてのみ操作でき、コンパイラがライフサイクルを追跡し、誤った分解を防ぐ |
| 物理トポロジ制約 | **ジェネリック定数パラメータ** | `Qubit(Topology, N)` でコンパイル時に隣接性をチェック |
| 測定による崩壊 | **空状態の再利用** | 測定後 qubit が空になり、再初期化可能で、量子状態の崩壊を模倣 |
| 量子回路の自動並列化 | **DAG スケジューラ** | 関数内のステートメントがデータ依存に基づいて自動並列化され、依存のないゲートは自然に並行実行 |
| ハイブリッド古典-量子制御フロー | **統一構文** | 量子操作と古典操作は同じ `name: type = value` 形式を使用 |

**YaoXiang は「量子サポートを追加する」のではなく、自分の設計がすでに量子ネイティブであることを発見したものである。**

> **セマンティクスの説明**：本文書は量子操作を表現するために YaoXiang の**所有権セマンティクス**を使用している。コンパイラレベルで no-cloning（所有権安全）を保証する。言語レベルの「消費-作成」は**所有権移動**の構文表現である——消費 = 所有権の取得、返値 = 所有権の委譲。実装レベルでは、実際の可逆量子ゲート（その場で量子状態を変更）であり、真に「新量子状態を作成する」わけではない。

### 設計目標

1. **新構文ゼロ**：`quantum`、`circuit` などのキーワードを導入せず、すべての量子特性を既存の言語機構で表現する。
2. **型安全**：コンパイラが量子状態がコピーまたは不正に使用されないことを保証する。
3. **トポロジ制約のコンパイル時チェック**：ジェネリック定数パラメータ `Qubit(T, N)` を通じて、2量子ビットゲート操作が物理トポロジに従うかをコンパイル時に検証する。
4. **マルチバックエンド透過性**：同一の量子コードを QIR（汎用エコシステム）または QCIS（国産量子命令セット）にコンパイルでき、コマンドラインパラメータで切り替え可能。
5. **ハイブリッド古典のシームレス統合**：量子計算は古典関数を自由に呼び出せ、古典コードも量子データを操作できる（`ref` 共有経由だが、所有権制約に従う）。

## 提案

### コア設計

#### 1. 量子型システムマッピング

**基本型**：
```yaoxiang
Qubit: Type0 = primitive_qubit
Complex: Type0 = { re: Float, im: Float }
```
- `Qubit` は第一級市民型であり、所有権規則（Move、RAII）を遵守する。
- `Complex` は振幅を表現するために使用され、コンパイラはインライン最適化できる。

**量子ゲートを関数として表現**：
```yaoxiang
# 組み込み関数シグネチャ
H: (Qubit) -> Qubit = builtin_hadamard
X: (Qubit) -> Qubit = builtin_pauli_x
Y: (Qubit) -> Qubit = builtin_pauli_y
Z: (Qubit) -> Qubit = builtin_pauli_z
CNOT: (control: Qubit, target: Qubit) -> { Qubit, Qubit } = builtin_cnot
```
- すべてのゲートは入力 qubit を消費し、新しい qubit（または量子もつれ対）を返す。所有権回流構文 `q = H(q)` は数学的セマンティクスを直接対応させる。
- 複数 qubit ゲートは構造体を返し、パターンマッチングまたはフィールドアクセスで結果を取得する。

**測定**：
```yaoxiang
measure: (Qubit) -> Int = builtin_measure   # qubit を消費し、古典ビットを返す
measure_all: (List(Qubit)) -> List(Int) = builtin_measure_all
```
- 測定後 qubit は消費され（空状態になる）、ユーザーは空状態の再利用で再初期化できる。

**初期化**：
```yaoxiang
qubit: (Int) -> Qubit = builtin_qubit   # 0 または 1 で基底状態を初期化
```

#### 2. 量子もつれと不透明型カプセル化

量子もつれ対を不透明型にカプセル化し、合成操作のみを提供し、分解を禁止する：

```yaoxiang
# 組み込み不透明型
BellPair: Type0 = primitive_bell_pair

# 組み込み関数 - 全体としてのみ操作可能
CNOT: (Qubit, Qubit) -> BellPair
measure_bell: (BellPair) -> { Int, Int }
split_bell: (BellPair) -> { Qubit, Qubit }  # 量子もつれ対の分解（注意して使用）
apply_cnot_to_bell: (BellPair, Qubit) -> BellPair
```

**重要な設計**：
- フィールドアクセッサを提供せず、組み込み関数でのみ全体として操作可能
- `measure_bell(bp)` は量子もつれ対全体を一度に消費し、古典ビットを返す
- コンパイラは量子もつれ対の完全なライフサイクルを追跡できる

**Python/Qiskit との比較**：
```
Python (Qiskit): 実行時に回路を構築し、エラーは送信後に才发现する場合がある
YaoXiang:       コンパイル時にほとんどの論理エラーを捕获できる
```

**残りの 10%**（物理的脱コヒーレンス、ゲート誤差はハードウェアの問題であり、言語で解決できるものではない。）

#### 3. 物理トポロジ制約

量子チップは制約付きトポロジグラフであり、**任意の2つの qubit で2量子ビットゲートを実行できるわけではなく**、隣接している必要がある。YaoXiang は**ジェネリック定数パラメータ**を使用してコンパイル時にトポロジ制約を保証する。

**トポロジ型定義**：
```yaoxiang
# トポロジを型として定義し、隣接行列を含む
Topology: Type0 = primitive_topology

# 組み込みトポロジ定数
Linear8: Topology = topology(8)          # 線形8ビット: 0-1-2-3-4-5-6-7
Grid3x3: Topology = topology(3, 3)        # 3x3 グリッド
Ring16: Topology = topology(16, ring)    # リング16ビット
```

**Qubit のトポロジと位置バインディング**：
```yaoxiang
# Qubit(T, N) - T はトポロジ型、N は定数位置パラメータ
q0: Qubit(Grid3x3, 0)   # Grid3x3 トポロジ、位置 (0,0)
q1: Qubit(Grid3x3, 1)   # Grid3x3 トポロジ、位置 (0,1)
q2: Qubit(Grid3x3, 2)   # Grid3x3 トポロジ、位置 (0,2)
q3: Qubit(Grid3x3, 3)   # Grid3x3 トポロジ、位置 (1,0)
```

**ゲート操作の自動制約**：
```yaoxiang
# CNOT 型シグネチャにトポロジ制約を含む
CNOT: (T: Topology, I: Int, J: Int) -> (
    (Qubit(T, I), Qubit(T, J)) -> { Qubit(T, I), Qubit(T, J) }
) when adjacent(T, I, J)

# コンパイル時チェック
CNOT(q0, q1)  # ✅ Grid3x3 で (0,0) と (0,1) は隣接
CNOT(q0, q2)  # ❌ コンパイルエラー：(0,0) と (0,2) は隣接していない
```

**`adjacent` コンパイル時制約**：
- `adjacent` はコンパイル時関数であり、トポロジの隣接行列を使用して静的チェックを行う
- 定数インデックス時は100%コンパイル時検証
- 動的インデックス時は実行時チェックコードを生成

**仮想から物理へのマッピング**：
```yaoxiang
# コンパイル時に具体的な物理位置がわからない場合、型推論を使用
q = qubit(Grid3x3)  # 位置0を自動割り当て、以降推論
```

#### 4. 所有権と量子状態の線形フロー

すべての量子操作は Move セマンティクスを遵守し、qubit がコピーされないことを保証する：
```yaoxiang
q = qubit(0)
q2 = q          # ❌ コンパイルエラー：q はすでに移動済み、使用不可
q = H(q)        # ✅ q を消費し、新しい q を返す
measure(q)      # ✅ q を消費した後、q は空状態
q = qubit(0)    # ✅ 空状態の再利用
```

#### 4. 自動並列化と DAG スケジューリング

Standard または Full Runtime では、DAG スケジューラが量子プログラムを自動解析する：
```yaoxiang
apply_two_qubit_gates: () -> {Qubit, Qubit} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    # 上記2行はデータ依存がなく、DAG は自動的に並列実行
    CNOT(q1, q2)   # q1 と q2 に依存、自動的に待機
}
```
- スケジューラは `num_workers` 設定（物理量子プロセッサ数）を活用して真の並列性を実現する。
- ユーザーはゲートの順序を手動で配置する必要はなく、データフローを記述するだけでよい。

#### 5. ハイブリッド古典計算

古典コードと量子コードは完全に融合する：
```yaoxiang
grover_search: (target: Int) -> Int = () => {
    n = 4
    qubits = List(Qubit)()
    for i in 0..n {
        qubits.append(H(qubit(0)))
    }
    # 古典ループと量子操作の混合
    oracle(qubits, target)   # oracle は量子ゲート列
    qubits = diffusion(qubits)
    results = measure_all(qubits)
    return decode_result(results)   # 古典後処理
}
```
- 同一関数内で量子ゲートと古典制御フローを自由に混合できる。
- 所有権システムは量子変数が古典分岐で誤ってコピーされないことを保証する。

#### 6. マルチバックエンドサポートアーキテクチャ

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   YaoXiang ソース │     │   型チェック    │     │   DAG 中間表現  │
│   (統一構文)      │────▶│   + 所有権解析   │────▶│   (データフロー図)│
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                          │
                                                          ▼
                          ┌─────────────────────────────────────────────┐
                          │           コード生成バックエンド (プラグイン式) │
                          ├─────────────────┬───────────────────────────┤
                          │  QIR バックエンド  │  QCIS バックエンド         │
                          │  (汎用エコシステム) │  (国産量子命令セット)      │
                          ├─────────────────┼───────────────────────────┤
                          │  - .ll ファイル出力 │  - .qcis テキスト出力     │
                          │  - 複数の QPU 対応  │  - 中科院/国盾ハードウェア対応│
                          └─────────────────┴───────────────────────────┘
```

- **コンパイルフロー**：フロントエンド統一 → DAG 構築 → バックエンド選択 → ターゲットコード生成。
- **QIR バックエンド**：DAG ノードを QIR の量子ゲート intrinsic にマッピングし、LLVM bitcode を生成して、LLVM 最適化のメリットを享受できる。
- **QCIS バックエンド**：DAG を QCIS 命令（例：`H q0`）にシリアライズし、量子チップコンソールに直接送信可能。

### 例

#### ベル状態の準備と測定
```yaoxiang
bell_measure: () -> {Int, Int} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    bell = CNOT(q1, q2)  # BellPair 不透明型を返す
    result = measure_bell(bell)  # 2ビットを一度に測定
    return result
}
```

#### 量子テレポーテーション（簡略版）
```yaoxiang
teleport: (msg: Qubit, bell: BellPair) -> Qubit = (msg, bell) => {
    # 量子もつれ対を分解して2つの独立 qubit を取得
    (alice_qubit, bob_qubit) = split_bell(bell)

    # アリス側の操作
    (msg, alice_qubit) = CNOT(msg, alice_qubit)
    msg = H(msg)
    a1 = measure(msg)
    a2 = measure(alice_qubit)

    # 古典情報の伝達 (スケジューラが自動的に依存関係を処理)
    # ボブ側の操作
    if a2 == 1 { bob_qubit = X(bob_qubit) }
    if a1 == 1 { bob_qubit = Z(bob_qubit) }
    return bob_qubit
}
```

## 詳細設計

### 組み込み型と関数定義

`compiler/builtins` モジュールに以下を追加：
```rust
builtins.insert("Qubit", Ty::Primitive(Primitive::Qubit));
builtins.insert("Complex", Ty::Record(vec![
    ("re", Ty::Primitive(Primitive::Float)),
    ("im", Ty::Primitive(Primitive::Float)),
]));

// 量子ゲート
for (name, sig) in GATES {
    builtins.insert(name, Ty::Function(vec![Ty::Qubit], Ty::Qubit));
}
builtins.insert("CNOT", Ty::Function(
    vec![Ty::Qubit, Ty::Qubit],
    Ty::Record(vec![
        ("q0", Ty::Qubit),
        ("q1", Ty::Qubit)
    ])
));
builtins.insert("measure", Ty::Function(vec![Ty::Qubit], Ty::Primitive(Primitive::Int)));
builtins.insert("qubit", Ty::Function(vec![Ty::Primitive(Primitive::Int)], Ty::Qubit));
```

### 所有権チェッカーの Qubit 特殊処理

- `Qubit` は `!Copy`（デフォルト Move）としてマークされ、暗黙的コピーが禁止される。
- 測定関数 `measure` のパラメータは `Qubit`（値渡し）であり、所有権を消費する。
- 複数 qubit ゲートが返すレコード型では、フィールドはすべて `Qubit` であり、所有権規則を遵守する必要がある。

### DAG スケジューラーの量子ゲート最適化

- 量子ゲートノードは純粋関数（副作用なし）として扱われ、スケジューラは依存関係のないゲートを自由に再配置できる。
- スケジューラが「量子命令列」を出力する際にデータ依存を保持し、並列ゲートをグループ化する（複数量子プロセッサ向け）。
- `--target-num-qubits` と `--target-topology` 設定をサポートし、後続のレイアウトとルーティングに使用（将来拡張）。

### QIR バックエンドの詳細マッピング

| YaoXiang 操作 | QIR 命令 |
|---------------|----------|
| `H(q)` | `call void @__quantum__qis__h__body(%Qubit* %q)` |
| `CNOT(q1, q2)` | `call void @__quantum__qis__cnot__body(%Qubit* %q1, %Qubit* %q2)` |
| `measure(q)` | `%result = call i1 @__quantum__qis__mz__body(%Qubit* %q)` |
| `qubit(0)` | `%q = call %Qubit* @__quantum__rt__qubit_allocate()` |

QIR バックエンドは LLVM の `-O2` をさらに活用し、QIR Alliance 互換の bitcode を出力する。

### QCIS バックエンドの詳細マッピング

| YaoXiang 操作 | QCIS 命令 |
|---------------|-----------|
| `H(q)` (q は物理ビット 2 に対応) | `H 2` |
| `CNOT(q1,q2)` (q1→ビット 0, q2→ビット 1) | `CNOT 0 1` |
| `measure(q)` (ビット 0) | `M 0` |
| `qubit(0)` 初期化 | 最初の使用命令に暗黙的に含まれるため、追加命令不要 |

- 仮想 qubit（YaoXiang 変数）から物理ビットへのマッピングテーブルを維持する必要がある。
- トポロジ制約チェックをサポート（将来実装）。

### ハイブリッド古典コード生成

- 古典部（ループ、条件、整数演算など）は通常通りローカルコード（x86/ARM）を生成し、FFI または組み込み呼び出しで量子バックエンドとやり取りする。
- QIR バックエンドでは、古典部は LLVM IR に降格でき、QIR と混合コンパイル可能。

### 型システムへの影響

- 新しい `Qubit` と `Complex` 原始型を追加。
- `Qubit` は自動的に Move セマンティクスを持ち、コピーが禁止される。
- 量子ゲート関数シグネチャは型システムに登録する必要がある。

### 後方互換性

- ✅ 完全な後方互換性
- 新規組み込み型と関数は既存コードに影響しない
- 量子特性はオプションであり、有効にしなければオーバーヘッドなし

## トレードオフ

### メリット

- **新構文ゼロ**：開発者は少数の組み込み関数を学習するだけで量子プログラムを書ける。
- **型安全**：所有権システムが自動的に qubit コピーを防止し、一般的な量子プログラミングエラーを回避。
- **自動並列化**：DAG スケジューラがゲートレベルの並列化を無料で提供し、追加のコンパイラ最適化が不要。
- **エコシステム互換性**：QIR バックエンドにより YaoXiang は複数の量子クラウドプラットフォームで実行可能；QCIS バックエンドは自主制御を保障。
- **ハイブリッド能力**：古典量子融合が自然で、複雑な量子アルゴリズム（Shor、Grover の古典制御など）の記述に適している。

### デメリット

- **Qubit 数の静的な性質**：現在の設計では qubit 数はコンパイル時に既知と想定しており、動的割当ては `List(Qubit)` を通じて可能だが、`List` のヒープ割当てが追加オーバーヘッドをもたらす可能性がある（最適化で緩和可能）。
- **測定後の再利用**：空状態の再利用により qubit の再初期化が可能だが、物理量子ビットには弛緩時間が存在する可能性があり、実行時システムで処理する必要がある（現在はユーザーの責任）。
- **動的トポロジマッピング**：実行時に物理トポロジ才知道する場合、コンパイル時チェックが有効にならず、実行時チェックコードを生成する必要がある（現在のバージョンは静的チェックのみサポート）。

## 代替案

| 方案 | 選択しない理由 |
|------|--------------|
| `quantum` キーワードと `circuit` 型を導入 | 新構文，增加了学習コスト、YaoXiang の簡潔設計原則に反する |
| ライブラリとしてのみ量子サポートを実装 | コンパイラの量子状態安全保証を利用できず、DAG スケジューラとの深い統合ができない |
| 量子ハードウェアの成熟を待ってからサポート | 量子プログラミング言語設計の重要なウィンドウ期を逃す |
| 既存の量子フレームワーク（Qiskitなど）を再利用 | 量子セマンティクスがライブラリで実装されるため、型システムと所有権システムの安全保障を享受できない |
| 量子サブ言語を別途設計 | 言語複雑度增加，维护コストが高い |

## 実装戦略

### 段階的計画

| 段階 | 期間 | 内容 |
|------|------|------|
| フェーズ 1 | 1ヶ月 | 基本量子型と組み込み関数：コンパイラに `Qubit`、`Complex` 型を追加し、組み込み関数の型チェックを実装し、所有権チェッカーを拡張 |
| フェーズ 2 | 1ヶ月 | DAG スケジューラが量子ゲートを認識：DAG 構築ロジックを修正し、量子ゲートを純粋関数としてマークし、並列ゲートグループ出力を実装 |
| フェーズ 3 | 2ヶ月 | QIR バックエンド原型：DAG から QIR コードジェネレータを実装し、LLVM を統合し、QIR シミュレータに接続して検証 |
| フェーズ 4 | 2ヶ月 | QCIS バックエンド原型：DAG から QCIS 命令への変換を実装し、仮想-物理ビットマッピングを設計し、国産量子プラットフォームに接続して検証 |
| フェーズ 5 | 2ヶ月 | ハイブリッド古典の強化：古典制御フローと量子ゲートの交差が正しくコード生成されることを保証し、`List(Qubit)` をサポートし、例文プログラムを追加 |
| フェーズ 6 | 2ヶ月 | 最適化とドキュメント：基本レイアウトとルーティングを実装し、ユーザーガイドと量子プログラミングチュートリアルを编写し、プレビュー版をリリース |

### リスク

1. **量子ハードウェア可用性**：外部量子シミュレータと実際の QPU の可用性に依存する。
   - **緩和**：オープンソースシミュレータ（QIR runner、Qiskit Aer）を優先的に接続し、実際の QPU は長期目標とする。

2. **バックエンド実装の複雑さ**：QIR と QCIS 仕様は変更される可能性がある。
   - **緩和**：コード生成インターフェースを抽象化し、バックエンドの差異を隔离して、後続の適応を容易にする。

3. **性能の不確実性**：量子プログラムの性能特性は古典プログラムと異なる。
   - **緩和**：パフォーマンスプロファイリングツールを提供し、ユーザーがゲートレベル並列効果を確認するできるようにする。

## 開放問題

- [x] **トポロジ制約**：`Qubit(Topology, N)` ジェネリック定数パラメータを通じてコンパイル時チェックが実装済み。
- [ ] **動的量子レジスタ**：`List(Qubit)` は QCIS バックエンドでどのようにマッピングするか？対応する数の物理ビットを生成できるが、実行時割当機構が必要。
- [ ] **エラー緩和**：動的脱結合などの組み込みエラー緩和构造を提供するかどうか？最初はライブラリとして実装可能。
- [ ] **既存の量子 SDK との相互運用**：QASM または QIR モジュールのインポート是否能？将来的に FFI を検討可能。
- [ ] **自動レイアウトとルーティング**：仮想 qubit 数が物理 qubit 数を 초과する場合、自动マッピング方法是？
