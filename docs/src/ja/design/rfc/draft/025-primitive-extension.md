# RFC-025: 拡張可能プリミティブ型メカニズム

## 要約

本文書は YaoXiang コンパイラの**拡張可能プリミティブ型メカニズム**（`Primitive::Extension`）を定義する。外部コードがコンパイラにカスタムプリミティブ型を登録できるようにし、コンパイラはハードコーディングなしでドメイン固有型（量子ビット、GPU バッファ、SIMD ベクトル、ハードウェアレジスタなど）をサポートできるようにする。

## 動機

### なぜこのメカニズムが必要か？

現在、コンパイラはすべてのプリミティブ型をハードコーディングしている：`Int`、`Float`、`String`、`Bool`、`Unit`。新しいプリミティブ型を追加するたびに、コンパイラの複数の箇所（型検査器、コード生成器、所有権分析器、オプティマイザ）を修正する必要がある。

これは開放閉鎖原則に反している：拡張に対しては開放的だが、修正に対しては閉鎖的である。

### 設計境界

```
ハードコーディングされたコア型（言語基盤）：Int, Float, String, Bool, Unit
動的に拡張される型（ドメイン� 플러グイン）：   Primitive::Extension 経由で登録
```

コア型をハードコーディングするのは、コンパイラが算術演算、条件分岐、ハッシュ、比較などのセマンティクスを深く依存しているためである。拡張型はコンパイラにとっては**不透明な値**である——コンパイラはサイズ、アライメント、所有権属性のみ知り、内部セマンティクスは知らない。

これは「すべての型を動的ロードに統一する」ことではない。コア型と拡張型は異なる2つの概念である。

## 提案

### コア設計

#### 1. Extension 型属性

各拡張プリミティブ型を登録する際には、以下の属性を宣言する必要がある：

```rust
pub struct PrimitiveExtension {
    /// 型名（例："Qubit", "Buffer", "Vec128"）
    pub name: String,

    /// バイトサイズ（固定サイズ型）
    /// None はコンパイル時にサイズが不明なことを示す（実行時に決定）
    pub size: Option<usize>,

    /// アライメント要件
    pub align: Option<usize>,

    /// 暗黙的なコピーを許可するか
    /// false = Move セマンティクス（代入で移動，如 Qubit）
    /// true = Copy セマンティクス（代入で複製，如 Vec128）
    pub is_copy: bool,

    /// 空値（zero-sized type）を許可するか
    pub allow_zst: bool,
}
```

#### 2. 登録インターフェース

```rust
// コンパイラ内部 API
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".to_string(),
    size: Some(0),           // 論理サイズは0、物理状態は量子プロセッサ上
    align: Some(1),
    is_copy: false,          // Move セマンティクス、no-cloning に適合
    allow_zst: true,
});
```

登録後、`Qubit` は型システム内で合法的なプリミティブ型となり、変数宣言、関数パラメータ、構造体フィールドに使用できる。

#### 3. 型検査器の動作

拡張プリミティブ型は型検査で以下のルールに従う：

| シナリオ | 動作 |
|------|------|
| 変数宣言 `q: Qubit = ...` | ✅ 合法 |
| 関数パラメータ `fn(q: Qubit)` | ✅ 合法 |
| 構造体フィールド `{ q: Qubit }` | ✅ 合法 |
| `is_copy == false` の代入 | Move セマンティクス、元の変数は無効 |
| `is_copy == true` の代入 | Copy セマンティクス、元の変数は有効 |
| 暗黙的なコピー（関数での複数使用） | `is_copy` に依存 |
| 比較 `==`、`!=` | ❌ コンパイルエラー（組み込み比較なし） |
| 算術 `+`、`-` | ❌ コンパイルエラー（組み込み演算なし） |
| ジェネリクス制約 `T: Copy` | `is_copy == true` の場合のみ満たす |

#### 4. コード生成器の動作

拡張プリミティブ型はコード生成で**不透明な値**として処理される：

- LLVM IR：`{size} x i8` または対応サイズの構造体として生成
- 特殊命令は生成しない——セマンティクスはバックエンドまたはライブラリが担当
- バックエンドが特殊処理を必要とする場合（QIR 量子ゲートなど）は、バックエンド登録メカニズムで実装（本 RFC の範囲外）

### 例

#### Move セマンティクスの型を登録する

```rust
// 量子ビット：複製不可、サイズ0（物理状態は QPU 上）
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".into(),
    size: Some(0),
    align: Some(1),
    is_copy: false,
    allow_zst: true,
});
```

```yaoxiang
# ユーザーコード
q: Qubit = qubit(0)
q2 = q          # ❌ コンパイルエラー：Qubit は Move 型、q は無効
q = H(q)        # ✅ q を消費し、新しい q を返す
```

#### Copy セマンティクスの型を登録する

```rust
// SIMD ベクトル：複製可、固定サイズ
compiler.register_primitive(PrimitiveExtension {
    name: "Vec128".into(),
    size: Some(16),
    align: Some(16),
    is_copy: true,
    allow_zst: false,
});
```

```yaoxiang
# ユーザーコード
a: Vec128 = load_vec128(data)
b = a           # ✅ Copy セマンティクス、a は引き続き有効
c = add_vec128(a, b)  # ✅ a と b の両方が使用可能
```

## 詳細設計

### コンパイラの変更

| コンポーネント | 変更点 |
|------|------|
| 型システム | 新しい `Ty::Extension` バリアントを追加、`PrimitiveExtension` メタデータを格納 |
| 型検査器 | 拡張型は組み込み演算の解析に関与しない、組み込み trait 制約を満たさない（明示的に実装された場合を除く） |
| 所有権分析器 | `is_copy` に基づいて Move または Copy セマンティクスを決定 |
| コード生成器 | `size`/`align` に基づいて不透明な値を生成、特殊命令なし |
| エラーメッセージ | 拡張型のエラーメッセージは登録時の `name` を参照 |

### FFI との関係

`Primitive::Extension` と RFC-021（FFI）は直交している：

| | Primitive::Extension | FFI |
|---|---|---|
| 役割 | 新しい**型**を登録 | 外部**関数**を呼び出す |
| 階層 | 型システム | ランタイム |
| 例 | `Qubit` は型である | `native("sin")` は関数呼び出しである |

1つのドメインが両方を必要とする場合がある：`Qubit` 型は Extension で登録し、量子ゲート関数は FFI で登録する。

### 下位互換性

- ✅ 完全な下位互換性
- 既存の型のセマンティクスを変更しない
- 拡張型は新機能であり、既存のコードに影響しない

## トレードオフ

### 利点

- ✅ コンパイラは新しいドメインごとにソースコードを変更する必要がない
- ✅ ドメインの専門家はコンパイラチームに依存せず自分で型を登録できる
- ✅ コア型はハードコーディングされたまま維持され、コンパイラの基盤型の深い最適化を犠牲にしない
- ✅ インターフェースはシンプルで、1つの構造体ですべての属性を定義

### 欠点

- ⚠️ 拡張型は組み込み演算をサポートしない——セマンティクスを実装するには追加の関数またはバックエンドメカニズムが必要
- ⚠️ デバッグ時に拡張型は不透明な値として表示され、コア型ほど直感的ではない

## 代替案

| 案 | なぜ選択しないか |
|------|--------------|
| すべての型を動的ロード | コア型（Int/Float/Bool）はコンパイラの深い最適化を必要とし、動的ロードはこれらの能力を失う |
| 各ドメインをハードコーディング | ドメインを追加するたびにコンパイラを変更する必要があり、拡張性がない |
| 純粋なライブラリ案（型登録なし） | 型システムレベルでセマンティクス（no-cloning など）を保証できず、ランタイムチェックのみ |

## 実装戦略

### フェーズ 1：コアインターフェース

- [ ] 型システムに `Ty::Extension` バリアントを追加
- [ ] `register_primitive` API を実装
- [ ] 型検査器を拡張して `is_copy` セマンティクスを処理
- [ ] コード生成器を拡張して不透明な値を処理
- [ ] ユニットテスト

### フェーズ 2：登録タイミング

- [ ] コンパイラの初期化時に一括登録をサポート（設定ファイルまたはビルダー API）
- [ ] 標準ライブラリの事前登録をサポート（`std.primitive` モジュールが拡張型定義をエクスポート）

### 依存関係

- ハードな依存関係なし。他の RFC から独立して実装可能。

## 開放問題

- [ ] 拡張型は trait 実装をサポートすべきか（`Qubit` にカスタム `QuantumGate` trait を実装させるなど）？
- [ ] RAII セマンティクスの拡張型をサポートするためにライフサイクルフック（`on_drop` など）が必要か？
- [ ] 設定ファイル形式：TOML、YAML、または YaoXiang 自身の設定構文（RFC-015 を参照）？

## 設計決定記録

| 決定 | 結論 | 理由 | 日付 |
|------|------|------|------|
| コア型を動的ロードしない | ハードコーディングを維持 | コンパイラはコア型のセマンティクスを深く依存しており、動的化の收益はゼロ | 2026-06-05 |
| 拡張型を不透明な値とする | セマンティクスを注入しない | セマンティクスはバックエンド/ライブラリが担当し、コンパイラは型安全性と所有権のみ保証 | 2026-06-05 |
| FFI と直交する | マージしない | 型の登録と関数の呼び出し是不同的抽象レベル | 2026-06-05 |
