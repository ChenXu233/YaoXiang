# RFC-025：拡張可能プリミティブ型機構

> **状態**: 草案
> **作者**: 晨煦
> **作成日**: 2026-06-05
> **最終更新**: 2026-06-05

## 摘要

本ドキュメントは YaoXiang コンパイラの**拡張可能プリミティブ型機構**（`Primitive::Extension`）を定義する。外部コードがコンパイラにカスタムプリミティブ型を登録できるようになり、コンパイラを変更せずに領域固有型（量子ビット、GPUバッファ、SIMDベクトル、ハードウェアレジスタなど）をサポートできる。

## 動機

### なぜこの機構が必要か？

現在、コンパイラはすべてのプリミティブ型をハードコーディングしている：`Int`、`Float`、`String`、`Bool`、`Unit`。新しいプリミティブ型を追加するたびに、型チェッカー、コードジェネレータ、所有権アナライザ、オプティマイザなど、コンパイラの複数箇所を修正する必要がある。

これは開放/閉鎖原則に反する：拡張に対して開いているが、修正に対しては閉じている。

### 設計境界

```
ハードコーディングされたコア型（言語基盤）：Int, Float, String, Bool, Unit
動的に拡張する型（領域プラグイン）：    Primitive::Extension を通じて登録
```

コア型がハードコーディングされるのは、コンパイラが算術演算、条件分岐、ハッシュ、比較などのセマンティクスを深く依存しているためである。拡張型はコンパイラにとって**不透明な値**である——コンパイラはサイズ、アライメント、所有権属性のみを知っており、内部セマンティクスは知らない。

これは「すべての型を動的ロードに統一する」ことではない。コア型と拡張型は性質の異なる2つのものである。

## 提案

### コア設計

#### 1. Extension 型の属性

各拡張プリミティブ型を登録する際、以下の属性を宣言する必要がある：

```rust
pub struct PrimitiveExtension {
    /// 型名（例："Qubit", "Buffer", "Vec128"）
    pub name: String,

    /// バイトサイズ（固定サイズ型）
    /// None はコンパイル時にサイズが不明であることを示す（実行時に決定要）
    pub size: Option<usize>,

    /// アライメント要求
    pub align: Option<usize>,

    /// 暗黙的なコピーを許可するかどうか
    /// false = Move セマンティクス（代入＝移動、例：Qubit）
    /// true = Copy セマンティクス（代入＝複製、例：Vec128）
    pub is_copy: bool,

    /// ゼロサイズ型（空値）を許可するかどうか
    pub allow_zst: bool,
}
```

#### 2. 登録インターフェース

```rust
// コンパイラの内部API
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".to_string(),
    size: Some(0),           // 論理サイズは0、物理状態は量子プロセッサ上にある
    align: Some(1),
    is_copy: false,          // Moveセマンティクス、no-cloningに準拠
    allow_zst: true,
});
```

登録後、`Qubit` は型システム内で正当なプリミティブ型となり、変数宣言、関数パラメータ、構造体フィールドに使用できる。

#### 3. 型チェッカーの動作

拡張プリミティブ型は型チェックで以下の規則に従う：

| シナリオ | 動作 |
|------|------|
| 変数宣言 `q: Qubit = ...` | ✅ 合法 |
| 関数パラメータ `fn(q: Qubit)` | ✅ 合法 |
| 構造体フィールド `{ q: Qubit }` | ✅ 合法 |
| `is_copy == false` の場合の代入 | Moveセマンティクス、元の変数は無効化 |
| `is_copy == true` の場合の代入 | Copyセマンティクス、元の変数は保持 |
| 暗黙的なコピー（関数での複数使用） | `is_copy` により決定 |
| 比較 `==`、`!=` | ❌ コンパイルエラー（組み込み比較なし） |
| 算術 `+`、`-` | ❌ コンパイルエラー（組み込み演算なし） |
| ジェネリクス制約 `T: Copy` | `is_copy == true` の場合のみ満たす |

#### 4. コードジェネレータの動作

拡張プリミティブ型はコード生成において**不透明な値**として扱われる：

- LLVM IR：`{size} x i8` または対応サイズの構造体として生成
- 特殊な命令は生成しない——セマンティクスはバックエンドまたはライブラリが担当
- バックエンドが特殊な処理を必要とする場合（例：QIR量子ゲート）、バックエンド登録機構を通じて実装（本RFCの範囲外）

### 例

#### Moveセマンティクスの型を登録する

```rust
// 量子ビット：複製不可、サイズ0（物理状態はQPU上）
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".into(),
    size: Some(0),
    align: Some(1),
    is_copy: false,
    allow_zst: true,
});
```

```yaoxiang
# ユーザーコード
q: Qubit = qubit(0)
q2 = q          # ❌ コンパイルエラー：Qubit は Move 型、q は無効化済み
q = H(q)        # ✅ q を消費、新しい q を返す
```

#### Copyセマンティクスの型を登録する

```rust
// SIMDベクトル：複製可能、固定サイズ
compiler.register_primitive(PrimitiveExtension {
    name: "Vec128".into(),
    size: Some(16),
    align: Some(16),
    is_copy: true,
    allow_zst: false,
});
```

```yaoxiang
# ユーザーコード
a: Vec128 = load_vec128(data)
b = a           # ✅ Copyセマンティクス、a は仍然有効
c = add_vec128(a, b)  # ✅ a と b の両方を使用可能
```

## 詳細な設計

### コンパイラの修正

| コンポーネント | 修正内容 |
|------|------|
| 型システム | 新しい `Ty::Extension` バリアントを追加、`PrimitiveExtension` メタデータを格納 |
| 型チェッカー | 拡張型は組み込み演算解析に参加しない、組み込みtrait制約を満たさない（明示的に実装された場合を除く） |
| 所有権アナライザ | `is_copy` に応じてMoveまたはCopyセマンティクスを決定 |
| コードジェネレータ | `size`/`align` に応じて不透明な値を生成、特殊命令なし |
| エラーメッセージ | 拡張型のエラーメッセージには登録時の `name` を参照 |

### FFI との関係

`Primitive::Extension` と RFC-021（FFI）は直交している：

| | Primitive::Extension | FFI |
|---|---|---|
| 役割 | 新しい**型**を登録 | 外部**関数**を呼び出す |
| レイヤー | 型システム | 実行時 |
| 例 | `Qubit` は型である | `native("sin")` は関数呼び出しである |

ある領域は両方を同時に必要とする可能性がある：`Qubit` 型はExtensionで登録し、量子ゲート関数はFFIで登録する。

### 下位互換性

- ✅ 完全な下位互換性
- 既存の型のセマンティクスを変更しない
- 拡張型は新機能であり、既存のコードに影響しない

## トレードオフ

### 利点

- ✅ コンパイラは新しい領域ごとにソースコードを修正する必要がない
- ✅ 領域の専門家が自行で型を登録でき、コンパイラチームに依存しない
- ✅ コア型はハードコーディングされたまま保ち、コア型のコンパイラによる深い最適化を犠牲にしない
- ✅ インターフェースがシンプル、1つの構造体で全属性を定義

### 欠点

- ⚠️ 拡張型は組み込み演算をサポートしない——セマンティクスを実装するには追加の関数またはバックエンド機構が必要
- ⚠️ デバッグ時、拡張型は不透明な値として表示されるため、コア型ほど直感的でない

## 代替案

| 方案 | 選択しない理由 |
|------|--------------|
| すべての型を動的にロード | コア型（Int/Float/Bool）はコンパイラの深い最適化を要する、動的ロードはこれらの能力を失う |
| 各領域をハードコーディング | 各領域を追加するたびにコンパイラを修正しなければならず、拡張性がない |
| 純粋なライブラリ方案（型登録なし） | 型システムレベルでセマンティクスを保証できない（例：no-cloning）、実行時チェックのみ |

## 実装策略

### フェーズ1：コアインターフェース

- [ ] 型システムに `Ty::Extension` バリアントを追加
- [ ] `register_primitive` APIを実装
- [ ] 型チェッカーを拡張して `is_copy` セマンティクスを処理
- [ ] コードジェネレータを拡張して不透明値を処理
- [ ] ユニットテスト

### フェーズ2：登録タイミング

- [ ] コンパイラ初期化時のバッチ登録をサポート（設定ファイルまたはbuilder API）
- [ ] 標準ライブラリの事前登録をサポート（`std.primitive` モジュールが拡張型定義をエクスポート）

### 依存関係

- ハードな依存関係なし。他のRFCとは独立して実装可能。

## 設計意思決定記録

| 意思決定 | 決定 | 理由 | 日付 |
|------|------|------|------|
| コア型を動的にロードしない | ハードコーディングを維持 | コンパイラはコア型のセマンティクスを深く依存しており、動的化のメリットはない | 2026-06-05 |
| 拡張型は不透明な値 | セマンティクスを注入しない | セマンティクスはバックエンド/ライブラリが担当し、コンパイラは型安全性と所有権のみ保証 | 2026-06-05 |
| FFIと直交 | マージしない | 型登録と関数呼び出し是不同的抽象レイヤー | 2026-06-05 |
