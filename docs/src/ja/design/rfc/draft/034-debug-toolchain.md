# RFC-034: 統一デバッグツールチェーン

## 要約

YaoXiang に統一デバッグツールチェーンを導入する。中心となる設計は**一つの源、三つの消費**である。コンパイルフロントエンドはソース位置、変数名、型情報を YaoXiang IR に第一級市民として埋め込み、インタプリタ、JIT、LLVM の三つのバックエンドがそれぞれ同一のメタデータを消費する。ユーザーは `yaoxiang run --debug` で DAP（Debug Adapter Protocol）サーバーを起動し、VS Code が stdio 経由で接続する。これにより、ブレークポイント、ステップ実行、変数表示、コールスタック、式評価、並行デバッグの統一された体験が得られる——基盤となる実行エンジンがどれであっても。

## 動機

### なぜこの機能が必要か？

現在、YaoXiang プログラムのエラーを調査する手段は非常に原始的である：

```yaoxiang
io.println("DEBUG: x = " + x.to_string())
io.println("DEBUG: entered branch A")
```

三つの致命的な問題がある：

1. **コンパイラ開発のセルフブートストラップが阻害される**：YaoXiang で YaoXiang コンパイラを書くが、コンパイラを書く人は自分が書いたコードをデバッグできない。セルフブートストラップ段階でインタラクティブなデバッグ手段が欠如しているのは袋小路である。
2. **三つのエンジン、デバッグ機能ゼロ**：インタプリタ、JIT、LLVM がそれぞれ独立して動作し、問題が起きた場合、ユーザーは stdout に `ALL TESTS PASSED` が現れるかどうかしか確認できない。アサーションが失敗した？どの行で起きたのか、変数の値は何かはわからない。
3. **並行処理はブラックボックス**：`spawn` で複数のタスクが作成されるが、どのタスクがハングしたのか？変数は誰に move されたか？すべて運任せの推測になる。

### 設計目標

- **統一された体験**：インタプリタで動作するブレークポイントは、JIT でも、LLVM でも一貫したソースマッピングを持つ。ユーザーは基盤となるエンジンの違いを意識しない。
- **一つの源**：デバッグメタデータは IR と共に流れ、重複定義せず、二重のマッピングを維持しない。
- **ゼロ侵入**：`yaoxiang run --debug` は一つの引数であり、この引数を付けない場合、コンパイルと実行の動作は完全に変わらない。
- **DAP 標準**：VS Code エコシステムに直接接続し、エディタプロトコルを再発明しない。

## 提案

### 中核設計

アーキテクチャ概要：

```
┌──────────────────────────────────────────────────────┐
│                    VS Code / エディタ                  │
│              DAP Client (launch.json)                 │
└────────────────────────┬─────────────────────────────┘
                         │ stdio
┌────────────────────────▼─────────────────────────────┐
│                 DAP サーバー (yx-core)                │
│  ┌─────────┐  ┌──────────┐  ┌───────────────────┐    │
│  │ セッション│  │ブレークポイント│  │式評価エンジン  │    │
│  │ 管理     │  │ 管理     │  │                  │    │
│  └─────────┘  └──────────┘  └───────────────────┘    │
└────────────────────────┬─────────────────────────────┘
                         │ クエリ/制御
┌────────────────────────▼─────────────────────────────┐
│                  ランタイムデバッグインターフェース（trait）│
│  pause / resume / step / get_frames / eval / ...    │
└────┬──────────────────┬──────────────────┬──────────┘
     │                  │                  │
┌────▼────┐    ┌───────▼───────┐    ┌─────▼──────┐
│インタプリタ│    │ JIT (RFC-028)│    │ LLVM AOT   │
│ IR メタ  │    │ 軽量デバッグ │    │ IRメタデータ │
│ データを │    │ テーブル生成  │    │ → DWARF    │
│ 直接消費  │    │              │    │            │
└─────────┘    └───────────────┘    └────────────┘
```

**重要な設計判断**：

1. **DAP サーバーとランタイムは trait で疎結合にする**。サーバーは基盤がインタプリタか JIT かを意識しない——`DebugEngine` trait を通じて命令を出すだけである。各エンジンは同じ trait を独立して実装する。
2. **`yaoxiang run --debug` はインタプリタを強制する**。デバッグには制御可能性が必要であり、性能は不要である。LLVM モードでは DWARF を生成して事後トレース（core dump / クラッシュレポート）にのみ使用し、インタラクティブデバッグは行わない。
3. **`yaoxiang run` のエントリポイント発見ロジックを再利用する**。新しいサブコマンドを導入せず、心のモデルは「プログラムをデバッグモードで実行する」のまま。

### IR デバッグメタデータ

既存の YaoXiang IR にメタデータを付加し、IR 種類は追加しない。すべてのメタデータは**コンパイルフロントエンド**の一箇所で生成され、バックエンドの消費は読み取り専用である：

| メタデータ | 付加点 | 説明 |
|------------|--------|------|
| `SourceLocation` | 各 IR ノード | ソースファイル:行番号:列番号 |
| `VarName` | 変数宣言/バインディングノード | ソースコード中の変数名 |
| `TypeAnnotation` | 変数/式ノード | 推論された型（コンパイル時述語を含む） |
| `ScopeBoundary` | ブロック/関数の入口と出口 | 変数スコープのライフサイクル |
| `SpanInfo` | spawn ノード | spawn ブロック内のタスク境界 |

### 起動フロー

```
yaoxiang run --debug file.yx
    │
    ├── Phase 1: コンパイル（デバッグメタデータ付き）
    │   ├── パース → AST
    │   ├── 型チェック + コンパイル時述語検証
    │   └── IR への降格（デバッグメタデータを付加）
    │
    ├── Phase 2: インタプリタエンジンの使用
    │   └── --debug モードでは --release の有無に関わらずインタプリタを使用
    │
    ├── Phase 3: DAP サーバーの起動
    │   ├── stdio トランスポートチャネルの初期化
    │   ├── VS Code の attach を待機
    │   ├── attach 成功後、プログラム入口で一時停止
    │   └── インタラクティブデバッグループに移行
    │
    └── Phase 4: プログラム終了 / デバッグセッション終了 → 終了
```

### 各モードの差異

| モード | デバッグ方法 |
|--------|-------------|
| `yaoxiang run --debug` | インタプリタを強制、完全な DAP インタラクティブデバッグ |
| `yaoxiang run --release` | DWARF を生成、事後トレース（core dump / クラッシュレポート）に使用、DAP 起動なし |
| `yaoxiang run`（通常） | デバッグメタデータなし、デバッグサポートなし |

## 詳細設計

### 1. ブレークポイント

```
ブレークポイント種別：
├── ソース行ブレークポイント → コンパイルフロントエンドが位置メタデータを生成、バックエンドがマッチを検索
├── 関数入口ブレークポイント → 関数呼び出し時にトリガー（第二フェーズ）
├── 条件付きブレークポイント → 式の評価が true の時にトリガー
└── データブレークポイント → 変数が変更された時にトリガー（第二フェーズ）
```

**ソース行ブレークポイントの中核ロジック**：

```
VS Code 送信: "file.yx:42 にブレークポイントを設定"
    │
DAP サーバー:
    ├── IR 内の SourceLocation == (file.yx, 42) のノードをすべて検索
    ├── ランタイムに転送: "これらの IR アドレスで一時停止"
    └── ランタイムが返す: ブレークポイント ID

プログラムが IR ノードに到達 → ランタイムがチェック: このノードはブレークポイントリストにあるか？
    ├── 通常ブレークポイント → 一時停止、DAP サーバーに通知
    └── 条件付きブレークポイント → 条件式を評価 → true の場合のみ一時停止
```

**三つのエンジンの実装**：

| | インタプリタ | JIT | LLVM |
|---|---|---|---|
| ブレークポイント挿入方式 | 実行ループ内で IR ノード ID をチェック | JIT が機械語に `int3` を挿入 | LLVM DWARF + ハードウェアブレークポイントを利用 |
| 条件付きブレークポイント評価 | 式を直接解釈 | 条件式を一時的に JIT コンパイル | DWARF 式スタック + 評価 |
| 性能オーバーヘッド | IR ノードごとに追加のテーブル参照 | ブレークポイント箇所のみオーバーヘッド | ほぼゼロオーバーヘッド（ハードウェアブレークポイント） |

### 2. ステップ実行

```
Step Over    → 現在行を実行、関数呼び出しの内部をスキップし、次の行で停止
Step Into    → 現在行の関数呼び出し内部に入る
Step Out     → 現在の関数が返るまで実行
Continue     → 次のブレークポイントまたはプログラム終了まで実行を再開
```

**実装ロジック**：ステップ操作は本質的に**一時的なブレークポイント**である。ユーザーが明示的に設定したブレークポイントと同一の機構を共有し、二つのシステムではなく、一つのシステムの二つの使用方法である。

```
Step Over:
    現在のソース行番号 = query_line(frame)
    → 次の行に一時ブレークポイントを設定
    → 現在行が関数呼び出しの場合: 呼び出し点の後に一時ブレークポイントを設定
    → Continue → 一時ブレークポイントにヒット → 削除 → 一時停止

Step Into:
    現在の呼び出し対象の最初の実行可能行
    → 関数本体の最初の IR ノードのソース位置を特定
    → 一時ブレークポイント設定 → Continue → ヒット → 一時停止

Step Out:
    現在のスタックフレームのリターンアドレス
    → caller の次の行を特定
    → 一時ブレークポイント設定 → Continue → ヒット → 一時停止
```

**一時ブレークポイントの四つの境界ケース処理**：

1. **並行性の帰属**：一時ブレークポイントは現在のタスク ID にバインドされ、他のタスクがヒットした場合は直接無視する。
2. **spawn ブロックに対する Step Over**：spawn の外側で Step Over を押すことは、spawn ブロック全体を完了させ、その後にジャンプすることと同義である。spawn 内部に入ってデバッグするには Step Into を使用すべきである。
3. **一時ブレークポイント未ヒット**：ウォッチドッグタイムアウトを設定（30 秒間どのブレークポイントにもヒットしない）→ 強制一時停止 → VS Code に通知。同時にプログラム終了イベントを監視 → 即座にクリーンアップ。
4. **同一行に複数の IR ノード**：Step Over の一時ブレークポイントは `ignore_current_line` フラグを付け、ヒット時にソース行番号が現在の行番号と等しい場合 → 無視して続行。

### 3. 変数検査とスコープ

```
VS Code 要請: "現在のフレームの変数リスト"
    │
DAP サーバー:
    ├── 現在の一時停止点の IR ノードを検索
    ├── 現在の ScopeBoundary 内の変数バインディングを走査
    │   └── 各バインディングを返す: (名前, 型, ランタイム値参照)
    └── VariablesResponse を組み立て → VS Code
```

**スコープ階層**：

```
┌─ Globals ───────────────────────────┐
│  モジュールレベルバインディング:       │
│  定数、型エイリアス、グローバル       │
├─ Locals ────────────────────────────┤
│  現在の関数内で可視のローカル変数       │
│  ├── 引数 (関数パラメータ)            │
│  └── ローカルバインディング            │
│      (let / 代入)                    │
├─ Captured ──────────────────────────┤
│  spawn ブロック / クロージャが捕捉    │
│  した外部変数                         │
│  所有権状態を表示:                     │
│  move 済み / ref 共有                │
└─────────────────────────────────────┘
```

**エンジン差異**：

| エンジン | 変数値取得 |
|----------|-----------|
| インタプリタ | VM スタックフレームとヒープを直接読み取る。各値はメモリ内に明示的な表現を持つ |
| JIT | レジスタとスタック上の値 → JIT コンパイル時に「変数 → レジスタ/スタックスロット」マッピングテーブルを記録する必要がある |
| LLVM | DWARF の `.debug_info` セクション → `DW_AT_location` → LLDB ネイティブサポート |

**特殊型の表示**：コンパイル時述語による精密化された型表示は、デバッグに有用な情報を提供する：

```
x: Positive(x)  →  "Int (x > 0 = True)" と表示
y: Sorted(y)    →  "Array(Int) (ソート保証)" と表示
result: T       →  ランタイムの具体型を表示
```

### 4. コールスタック

```
DAP 要請: StackTrace
    │
戻り値:
┌──────────────────────────────────┐
│ #0  process_item()  file.yx:42  │  ← 現在の一時停止点
│     locals: item = "hello"       │
│     spawn タスク ID: task-3      │
├──────────────────────────────────┤
│ #1  main()          file.yx:67  │  ← caller
│     locals: data = ["hello", ...]│
├──────────────────────────────────┤
│ #2  <entry>         file.yx:1   │  ← ルート
└──────────────────────────────────┘
```

各フレームの記録内容：関数シグネチャ、呼び出し位置（ソースファイル+行番号）、ローカル変数（遅延評価）、spawn コンテキスト（タスク ID）。

インタプリタと JIT はそれぞれ独自にフレームリンクリストを維持する。フレームの取得はゼロコストではない——デバッグモードはゼロオーバーヘッドを追求しない。

### 5. 式評価（Watch / REPL）

ユーザーはブレークポイントで任意の YaoXiang 式を入力する：

```
Watch: x + y         → 計算結果を返す
Watch: items[2].name → 複雑な構造にアクセス
Watch: f(x)          → 関数を呼び出す（副作用リスクあり）
```

**評価戦略**：

```
ユーザーが式を入力
    │
├── コンパイラフロントエンドが式をパース
├── 現在のフレームコンテキストで型チェック
├── 変数値を現在のフレームから取得（読み取り専用参照）
├── 式を独立したマイクロプログラムとして実行
│   └── 外部変数の変更を許可しない
│   └── spawn を許可しない
│   └── IO を許可しない（またはオプションで有効化）
└── 結果値を返す → 元のフレーム状態は完全に不変
```

**インタプリタは自然なサンドボックス**：式評価は新しいサンドボックスを作るわけではない——インタプリタ自体がサンドボックスである。式評価は単に一時的にフレームを push し、使用後に破棄するだけである。通常の実行と同一の VM を共有するが、いかなる副作用もコミットしない。

**関数呼び出し評価**：デフォルトで許可するが、ユーザーに「この式は副作用を持つ可能性がある」と警告し、ユーザーの確認後に実行する。

**エンジン差異**：

| エンジン | 式評価 |
|----------|--------|
| インタプリタ | 既存の eval コードパスを再利用、現在のフレーム環境を注入 |
| JIT | 式を一時的にコンパイル → 現在のフレームにリンク → 実行 → 一時的なコードを破棄 |
| LLVM | 非対応——LLVM モードはインタラクティブデバッグを行わない |

### 6. 並行デバッグ

**タスクモデルの可視性**：

DAP の `threads` 概念は YaoXiang の `spawn` タスクにマッピングされる。各タスクは独自のスタックフレームリンクリストと実行状態を持つ。

```
┌─ Threads ───────────────────────────┐
│  ● task-1  main()     file.yx:10   │ ← 現在フォーカス中
│  ▶ task-2  fetch()    file.yx:34   │ ← 実行中
│  ⏸ task-3  process()  file.yx:56   │ ← ブレークポイントで停止
│  ◼ task-4  write()    終了済み      │
└─────────────────────────────────────┘
```

**並行コンテキストでのブレークポイント**：

| 一時停止モード | 動作 | 適用シナリオ |
|--------------|------|-------------|
| `stop-all`（デフォルト） | 一つのタスクがヒット → すべてのタスクが一時停止 | データ競合、グローバル状態のデバッグ |
| `stop-this-only` | ヒットしたタスクのみ一時停止、他は継続 | 独立したタスクロジックのデバッグ |

**spawn ブロックのステップセマンティクス**：

```
spawn {          // Step Over → spawn ブロック全体を完走
    task_a()     // Step Into → task_a 内部に入る
    task_b()     // 並列実行、個別の step の影響を受けない
}
```

### 7. DAP プロトコルマッピング

#### 第一フェーズ：中核リクエスト

| DAP リクエスト | YaoXiang セマンティクス |
|----------------|----------------------|
| `initialize` | 能力ネゴシエーション：ブレークポイント、ステップ、変数、スタックフレームをサポート |
| `launch` / `attach` | YaoXiang プログラムの起動/アタッチ（`--debug` は attach モード経由） |
| `setBreakpoints` | ソース行ブレークポイントの設定 |
| `configurationDone` | ブレークポイント準備完了、実行開始 |
| `threads` | すべてのアクティブな spawn タスクのリストを返す |
| `stackTrace` | 指定されたタスクのスタックフレームリストを返す |
| `scopes` | 現在のフレームの変数スコープを返す |
| `variables` | 指定されたスコープの変数リストを返す |
| `continue` | 実行を再開 |
| `next` | Step Over |
| `stepIn` | Step Into |
| `stepOut` | Step Out |
| `pause` | すべてのタスクを中断 |
| `evaluate` | 現在のフレームで式を評価 |
| `disconnect` | デバッグセッションを終了 |

#### 第二フェーズ：拡張リクエスト

| DAP リクエスト | YaoXiang セマンティクス |
|----------------|----------------------|
| `setFunctionBreakpoints` | 関数名ブレークポイント |
| `setExceptionBreakpoints` | エラー/panic 発生時に一時停止 |
| `dataBreakpointInfo` | データブレークポイント（変数変更でトリガー） |

## 実装戦略

### フェーズゼロ：インフラ（全フェーズに先行）

**目標**：コンパイルフロントエンドがデバッグメタデータを IR に付加する。

| コンポーネント | 変更内容 |
|----------------|----------|
| IR 定義 | `SourceLocation`、`VarName`、`TypeAnnotation` などのメタデータフィールドを追加 |
| Parser | 各 AST ノードにソース位置を記録 |
| TypeChecker | 型情報を IR ノードに付加 |
| テスト | IR dump に位置と変数情報が含まれることを検証 |

**ランタイムには関与しない。**

### 第一フェーズ：インタプリタ DAP MVP

**目標**：`yaoxiang run --debug file.yx` でブレークポイント設定、ステップ実行、変数表示が可能。

| コンポーネント | 変更内容 |
|----------------|----------|
| DAP サーバー（yx-core 新規モジュール） | stdio トランスポート層、中核リクエスト処理、ブレークポイントマネージャー（ソース行 → IR ノードマッピング） |
| ランタイムデバッグ trait（yx-core） | `DebugEngine` trait 定義（pause, resume, step, get_frames, eval, get_variables） |
| インタプリタ | 実行ループでのブレークポイントチェック、一時停止/再開メカニズム、フレームリンクリスト維持、`InterpreterDebugEngine` 実装 |
| CLI | `yaoxiang run --debug` 引数 |

**受け入れ基準**：`tests/yaoxiang/` 配下の任意の `.yx` ファイルに対して、VS Code でブレークポイント設定、Step Over、変数値表示が可能。

### 第二フェーズ：高度なデバッグ機能

**目標**：式評価、関数ブレークポイント、並行デバッグ、例外ブレークポイント。

| コンポーネント | 変更内容 |
|----------------|----------|
| 式評価エンジン | マイクロプログラムコンパイル（parser + typechecker 再利用）、一時フレームを VM に push、副作用分離 |
| 並行デバッグ | spawn タスクリストマッピング、ブレークポイントをタスク ID にバインド、stop-all / stop-this-only 一時停止戦略 |
| 関数/例外ブレークポイント | `setFunctionBreakpoints`、`setExceptionBreakpoints` マッピング |
| VS Code 拡張 | デフォルトの `launch.json` テンプレート提供 |

### 第三フェーズ：JIT デバッグ & LLVM DWARF

**目標**：JIT エンジンが DAP を再利用、LLVM がクラッシュトレース用の DWARF を出力。

| コンポーネント | 変更内容 |
|----------------|----------|
| JIT | `DebugEngine` trait 実装、コンパイル時に変数→レジスタマッピングテーブル生成、ランタイムフレームリンクリスト、式の一時的コンパイル |
| LLVM | IR デバッグメタデータ → LLVM `DILocation` / `DISubprogram` → DWARF（DAP インタラクションなし） |

### 依存関係

```
フェーズゼロ（IR メタデータ）
    ↓
フェーズ一（インタプリタ DAP MVP）  ← ここから使い始められる
    ↓
フェーズ二（高度な機能）
    ↓
フェーズ三（JIT + LLVM DWARF）
```

### リスク

| リスク | 緩和策 |
|--------|--------|
| インタプリタの一時停止メカニズムの複雑さ | 複雑な状態機械ではなく、シンプルな channel/signal を使用。一時停止とは次の命令を fetch させないこと |
| 式評価の型安全性 | 既存の typechecker を再利用、読み取り専用参照、副作用をコミットしない |
| DAP プロトコルの詳細 | debugpy / delve の実装を参照。プロトコルは成熟している |
| 並行デバッグの stop-all ライブロック | タイムアウト機構 + 強制一時停止 |

## トレードオフ

### 利点

- **一つの源**：デバッグメタデータは一度だけ生成され、三つのエンジンで共有。「インタプリタのデバッグ情報は正しいが LLVM は誤っている」ということはない
- **ゼロ侵入**：`--debug` は一つの引数であり、付けない場合の動作は完全に変わらない
- **DAP 標準**：VS Code エコシステムに直接接続。カスタムエディタプロトコルやデバッガー UI の再開発が不要
- **インタプリタ優先**：デバッグは本質的にインタプリタに適している——柔軟、制御可能、式評価がシンプル。LLVM モードでインタラクティブデバッグを行わないのは最も実用的な選択

### 欠点

- **デバッグモードの性能が低い**：インタプリタは JIT/LLVM よりもはるかに遅い。しかしデバッグに性能は不要——誰も debug モードで本番負荷を動かすことを期待していない
- **LLVM デバッグの制限**：AOT コンパイルではインタラクティブデバッグができず、GDB/LLDB + DWARF のみ。しかしこれはトレードオフ：LLVM モードではそもそもデバッグ動作の差異があってはならない
- **並行一時停止の複雑さ**：インタプリタ上で stop-all セマンティクスを実装するには、すべてのアクティブタスクを走査する必要がある

### 代替案

| 代替案 | 選択しない理由 |
|--------|---------------|
| 三つのエンジンがそれぞれ DAP を実装 | 三倍の作業、三倍のバグ。「良き趣味」に反する |
| DWARF のみ使用、自前の DAP は作らない | インタプリタと JIT に DWARF 概念はなく、LLDB は VM 内部に入れない |
| Python pdb を模したコマンドラインデバッガーを作成 | VS Code の体験はコマンドラインデバッガーを圧倒する |
| DAP を LSP プロセスに詰め込む | ライフサイクルが完全に異なる——LSP はプロジェクトに追従し、DAP はデバッグセッションに追従する。プロセス分離は硬直な要件 |
