# RFC-034: 統一デバッグツールチェーン

## 概要

YaoXiang に統一デバッグツールチェーンを導入する。中核となる設計は**一つの源、三つの消費**である。コンパイルフロントエンドがソース位置、変数名、型情報を第一級市民として YaoXiang IR に埋め込み、インタプリタ、JIT、LLVM という三つのバックエンドがそれぞれ同一のメタデータを消費する。ユーザーは `yaoxiang run --debug` で DAP（Debug Adapter Protocol）サーバを起動し、VS Code が stdio 経由で接続する。これにより、ブレークポイント、ステップ実行、変数表示、コールスタック、式評価、並行デバッグの統一された体験が得られる——基盤となる実行エンジンがどれであっても。

## 動機

### なぜこの機能が必要か？

現在、YaoXiang プログラムのエラーを調査する手段は極めて原始的である：

```yaoxiang
io.println("DEBUG: x = " + x.to_string())
io.println("DEBUG: entered branch A")
```

三つの致命的な問題がある：

1. **コンパイラ開発のブートストラップが阻害される**：YaoXiang で YaoXiang コンパイラを書くが、コンパイラを書く人が自分のコードをデバッグできない。ブートストラップ段階で対話的なデバッグ手段がないことは行き止まりである。
2. **三つのエンジン、デバッグはゼロ**：インタプリタ、JIT、LLVM がそれぞれ独立に動作しており、問題が起きるとユーザーは stdout に `ALL TESTS PASSED` が現れたかを確認するしかない。アサーションが失敗した？どの行で起きたか、変数の値は何かがわからない。
3. **並行処理はブラックボックス**：`spawn` で複数のタスクが作成されるが、どのタスクがハングしたのか？変数は誰に move されたのか？すべて運を天に任せて推測するしかない。

### 設計目標

- **統一された体験**：インタプリタで機能するブレークポイントは、JIT でも、LLVM でも、一貫したソースマッピングで機能する。ユーザーは基盤のエンジンの差異を意識しない。
- **一つの源**：デバッグメタデータは IR と共に流れ、重複定義せず、二系統のマッピングを維持しない。
- **ゼロインパクト**：`yaoxiang run --debug` は一つの引数であり、この引数を付けない場合はコンパイルと実行の動作が完全に変わらない。
- **DAP 標準**：VS Code エコシステムに直接接続し、エディタプロトコルやデバッガ UI を再発明しない。

## 提案

### 中核設計

アーキテクチャ概要：

```
┌──────────────────────────────────────────────────────┐
│                    VS Code / エディタ                  │
│              DAP Client (launch.json)                 │
└────────────────────────┬─────────────────────────────┘
                         │ stdio
┌────────────────────────▼─────────────────────────────┐
│                 DAP サーバ (yx-core)                   │
│  ┌─────────┐  ┌──────────┐  ┌───────────────────┐    │
│  │ セッション│  │ブレークポ│  │ 式評価エンジン    │    │
│  │ 管理     │  │イント管理│  │                    │    │
│  └─────────┘  └──────────┘  └───────────────────┘    │
└────────────────────────┬─────────────────────────────┘
                         │ クエリ / 制御
┌────────────────────────▼─────────────────────────────┐
│              ランタイムデバッグインタフェース (trait)   │
│  pause / resume / step / get_frames / eval / ...    │
└────┬──────────────────┬──────────────────┬──────────┘
     │                  │                  │
┌────▼────┐    ┌───────▼───────┐    ┌─────▼──────┐
│インタプリタ│    │ JIT (RFC-028)│    │  LLVM AOT  │
│ IRメタ    │    │ 軽量デバッグ  │    │ IRメタデータ│
│ データを  │    │ テーブル生成  │    │ →  DWARF  │
│ 直接消費  │    └───────────────┘    └────────────┘
└─────────┘
```

**主要な設計上の決定**：

1. **DAP サーバとランタイムは trait で疎結合する**。サーバは基盤がインタプリタか JIT かを意識しない——`DebugEngine` trait を介して命令を下すだけである。各エンジンは独立に同じ trait を実装する。
2. **`yaoxiang run --debug` はインタプリタを強制する**。デバッグには制御可能性が必要であり、パフォーマンスは不要である。LLVM モードでは DWARF の生成のみを行い、後日のバックトレース（コアダンプ / クラッシュレポート）に用い、対話的デバッグは行わない。
3. **`yaoxiang run` からエントリ発見ロジックを再利用する**。新しいサブコマンドを導入せず、メンタルモデルは「デバッグモードで自分のプログラムを実行する」のままである。

### IR デバッグメタデータ

既存の YaoXiang IR にメタデータを付加し、新たな IR 種別を追加しない。すべてのメタデータは**コンパイルフロントエンド**で一箇所で生成され、バックエンドの消費は読み取り専用である：

| メタデータ | 付加位置 | 説明 |
|----------|----------|------|
| `SourceLocation` | 各 IR ノード | ソースファイル:行番号:列番号 |
| `VarName` | 変数宣言 / 束縛ノード | ソースコード中の変数名 |
| `TypeAnnotation` | 変数 / 式ノード | 推論された型（コンパイル時述語を含む） |
| `ScopeBoundary` | ブロック / 関数の入口・出口 | 変数スコープのライフタイム |
| `SpanInfo` | spawn ノード | spawn ブロック内のタスク境界 |

### 起動フロー

```
yaoxiang run --debug file.yx
    │
    ├── フェーズ 1: コンパイル（デバッグメタデータ付き）
    │   ├── 解析 → AST
    │   ├── 型検査 + コンパイル時述語の検証
    │   └── IR への降格（デバッグメタデータを付加）
    │
    ├── フェーズ 2: インタプリタエンジンの使用
    │   └── --debug モードでは --release に関わらずインタプリタを使用
    │
    ├── フェーズ 3: DAP サーバの起動
    │   ├── stdio トランスポートチャネルの初期化
    │   ├── VS Code のアタッチを待機
    │   ├── アタッチ成功後、プログラムエントリで一時停止
    │   └── 対話的デバッグループへ移行
    │
    └── フェーズ 4: プログラム終了 / デバッグセッション終了 → 終了
```

### 各モードの違い

| モード | デバッグ方式 |
|--------|-------------|
| `yaoxiang run --debug` | インタプリタを強制し、全機能 DAP 対話的デバッグ |
| `yaoxiang run --release` | DWARF を生成し、後日のバックトレース（コアダンプ / クラッシュレポート）に使用。DAP は起動しない |
| `yaoxiang run`（通常） | デバッグメタデータなし、デバッグサポートなし |

## 詳細設計

### 1. ブレークポイント

```
ブレークポイント種別：
├── ソース行ブレークポイント  → コンパイルフロントエンドが位置メタデータを生成、バックエンドがマッチを照会
├── 関数エントリブレークポイント → 関数呼び出し時にトリガ（第二段階）
├── 条件付きブレークポイント  → 式の評価が true のときにトリガ
└── データブレークポイント    → 変数が変更されたときにトリガ（第二段階）
```

**ソース行ブレークポイントの中核ロジック**：

```
VS Code 送信: "file.yx:42 にブレークポイントを設定"
    │
DAP サーバ:
    ├── IR 内のすべての SourceLocation == (file.yx, 42) のノードを照会
    ├── ランタイムに転送: "これらの IR アドレスで一時停止"
    └── ランタイムが返す: ブレークポイント ID

プログラムが IR ノードに到達 → ランタイムがチェック: このノードはブレークポイントリストにあるか？
    ├── 通常ブレークポイント       → 一時停止、DAP サーバに通知
    └── 条件付きブレークポイント → 条件式を評価 → true の場合のみ一時停止
```

**三つのエンジンの実装**：

| | インタプリタ | JIT | LLVM |
|---|---|---|---|
| ブレークポイント挿入方式 | 実行ループで IR ノード ID をチェック | JIT が機械語に `int3` を挿入 | LLVM DWARF + ハードウェアブレークポイントを利用 |
| 条件付きブレークポイントの式評価 | 式を直接解釈 | 条件式を一時的に JIT コンパイル | DWARF 式スタック + 評価 |
| パフォーマンスオーバーヘッド | 各 IR ノードで追加のテーブル参照 | ブレークポイント箇所のみオーバーヘッド | ほぼゼロオーバーヘッド（ハードウェアブレークポイント） |

### 2. ステップ実行

```
Step Over   → 現在の行を実行、関数呼び出しの内部をスキップし、次の行で停止
Step Into   → 現在の行の関数呼び出しの内部に入る
Step Out    → 現在の関数が戻るまで実行
Continue    → 次のブレークポイントまたはプログラム終了まで実行を再開
```

**実装ロジック**：ステップ操作は本質的に**テンポラリブレークポイント**である。ユーザーが明示的に設定したブレークポイントと同一のメカニズムを共有する。二つのシステムではなく、一つのシステムの二つの使用方法である。

```
Step Over:
    現在のソース行番号 = query_line(frame)
    → 次の行にテンポラリブレークポイントを設定
    → 現在の行が関数呼び出しの場合: 呼び出し点より後にテンポラリブレークポイントを設定
    → Continue → テンポラリブレークポイントにヒット → 削除 → 一時停止

Step Into:
    現在の呼び出しターゲットの最初の実行可能位置
    → 関数本体の最初の IR ノードのソース位置を特定
    → テンポラリブレークポイントを設定 → Continue → ヒット → 一時停止

Step Out:
    現在のスタックフレームのリターンアドレス
    → 呼び出し元の次の行を特定
    → テンポラリブレークポイントを設定 → Continue → ヒット → 一時停止
```

**テンポラリブレークポイントの四つの境界ケース処理**：

1. **並行帰属**：テンポラリブレークポイントは現在のタスク ID にバインドされ、他のタスクがヒットした場合は無視する。
2. **Step Over spawn ブロック**：spawn の外で Step Over することは spawn ブロック全体を完走し、その後へジャンプすることに等しい。spawn 内部のデバッグに入るには Step Into を使用すべきである。
3. **テンポラリブレークポイントがヒットしない**：ウォッチドッグタイムアウト（30 秒間ブレークポイントのヒットなし）→ 強制一時停止 → VS Code に通知。同時にプログラム終了イベントを監視 → 直ちにクリーンアップ。
4. **同一行に複数の IR ノード**：Step Over のテンポラリブレークポイントは `ignore_current_line` をマークし、ヒット後にソース行番号が現在の行番号と等しい場合は無視して続行する。

### 3. 変数検査とスコープ

```
VS Code リクエスト: "現在のフレームの変数リスト"
    │
DAP サーバ:
    ├── 現在の一時停止点の IR ノードを照会
    ├── 現在の ScopeBoundary 内の変数束縛を走査
    │   └── 各束縛を返す: (名前, 型, ランタイム値参照)
    └── VariablesResponse を組み立て → VS Code
```

**スコープ階層**：

```
┌─ Globals ───────────────────────────┐
│  モジュールレベル束縛: 定数、型エイリアス、グローバル  │
├─ Locals ────────────────────────────┤
│  現在の関数内で可視のローカル変数                    │
│  ├── 引数 (関数のパラメータ)                       │
│  └── ローカル束縛 (let / 代入)                      │
├─ Captured ──────────────────────────┤
│  spawn ブロック / クロージャがキャプチャした外部変数 │
│  所有権状態を表示: ムーブ済み / ref 共有             │
└─────────────────────────────────────┘
```

**エンジンの違い**：

| エンジン | 変数値の取得 |
|---------|-------------|
| インタプリタ | VM のスタックフレームとヒープを直接読み取る。各値はメモリ内に明確な表現を持つ |
| JIT | レジスタとスタック上の値 → JIT コンパイル時に「変数 → レジスタ / スタックスロット」マッピングテーブルを記録する必要がある |
| LLVM | DWARF の `.debug_info` セクション → `DW_AT_location` → LLDB がネイティブサポート |

**特殊型の表示**：コンパイル時述語は型の表示をデバッグ上有用な情報に精錬する：

```
x: Positive(x)  →  "Int (x > 0 = True)" と表示
y: Sorted(y)    →  "Array(Int) (ソート保証)" と表示
result: T       →  実行時の具体的な型を表示
```

### 4. コールスタック

```
DAP リクエスト: StackTrace
    │
戻り値:
┌──────────────────────────────────┐
│ #0  process_item()  file.yx:42  │  ← 現在の一時停止点
│     locals: item = "hello"       │
│     spawn タスク ID: task-3      │
├──────────────────────────────────┤
│ #1  main()          file.yx:67  │  ← 呼び出し元
│     locals: data = ["hello", ...]│
├──────────────────────────────────┤
│ #2  <entry>         file.yx:1   │  ← ルート
└──────────────────────────────────┘
```

各フレームは以下を記録する：関数シグネチャ、呼び出し位置（ソースファイル + 行番号）、ローカル変数（遅延評価）、spawn コンテキスト（タスク ID）。

インタプリタと JIT はそれぞれ独立にフレームリンクリストを維持する。フレームの取得はゼロコストではない——しかしデバッグモードはゼロオーバーヘッドを追求しない。

### 5. 式評価（Watch / REPL）

ユーザーはブレークポイントで任意の YaoXiang 式を入力できる：

```
Watch: x + y         → 計算結果を返す
Watch: items[2].name → 複雑な構造にアクセス
Watch: f(x)          → 関数を呼び出す（副作用のリスクあり）
```

**評価戦略**：

```
ユーザーが式を入力
    │
├── コンパイラフロントエンドが式を解析
├── 現在のフレームコンテキストで型検査
├── 変数値を現在のフレームから取得（読み取り専用参照）
├── 式を独立したマイクロプログラムとして実行
│   └── 外部変数の変更は許可しない
│   └── spawn は許可しない
│   └── IO は許可しない（またはオプションで有効化）
└── 結果値を返す → オリジナルフレーム状態は完全に不変
```

**インタプリタは天然のサンドボックス**：式評価は新しいサンドボックスを作るわけではない——インタプリタ自体がサンドボックスである。式評価は単に一時的にフレームを push し、使い終わったら破棄する。通常実行と同一の VM を共有するが、いかなる副作用もコミットしない。

**関数呼び出し評価**：デフォルトで許可するが、ユーザーに「この式は副作用がある可能性がある」と警告し、確認後に実行する。

**エンジンの違い**：

| エンジン | 式評価 |
|---------|--------|
| インタプリタ | 既存の eval コードパスを再利用、現在のフレーム環境を注入 |
| JIT | 式を一時的にコンパイル → 現在のフレームにリンク → 実行 → 一時コードを破棄 |
| LLVM | サポートしない——LLVM モードは対話的デバッグを行わない |

### 6. 並行デバッグ

**タスクモデルの可視性**：

DAP の `threads` 概念は YaoXiang の `spawn` タスクにマッピングされる。各タスクは独自のスタックフレームリンクリストと実行状態を持つ。

```
┌─ Threads ───────────────────────────┐
│  ● task-1  main()     file.yx:10   │ ← 現在フォーカス中
│  ▶ task-2  fetch()    file.yx:34   │ ← 実行中
│  ⏸ task-3  process()  file.yx:56   │ ← ブレークポイントで一時停止
│  ◼ task-4  write()    終了済み       │
└─────────────────────────────────────┘
```

**並行コンテキストでのブレークポイント**：

| 一時停止モード | 動作 | 適用シナリオ |
|--------------|------|-------------|
| `stop-all`（デフォルト） | 一つのタスクがヒット → すべてのタスクを一時停止 | データ競合、グローバル状態のデバッグ |
| `stop-this-only` | ヒットしたタスクのみ一時停止、他は続行 | 独立したタスクロジックのデバッグ |

**spawn ブロックのステップセマンティクス**：

```
spawn {          // Step Over → spawn ブロック全体を完走
    task_a()     // Step Into → task_a に入る
    task_b()     // 並列実行、個別ステップの影響を受けない
}
```

### 7. DAP プロトコルマッピング

#### 第一段階：コアリクエスト

| DAP リクエスト | YaoXiang セマンティクス |
|--------------|------------------------|
| `initialize` | 能力ネゴシエーション：ブレークポイント、ステップ、変数、スタックフレームのサポート |
| `launch` / `attach` | YaoXiang プログラムの起動 / アタッチ（`--debug` は attach モード） |
| `setBreakpoints` | ソース行ブレークポイントの設定 |
| `configurationDone` | ブレークポイント準備完了、実行開始 |
| `threads` | すべてのアクティブな spawn タスクのリストを返す |
| `stackTrace` | 指定されたタスクのスタックフレームのリストを返す |
| `scopes` | 現在のフレームの変数スコープを返す |
| `variables` | 指定されたスコープの変数リストを返す |
| `continue` | 実行を再開 |
| `next` | Step Over |
| `stepIn` | Step Into |
| `stepOut` | Step Out |
| `pause` | すべてのタスクを中断 |
| `evaluate` | 現在のフレームで式を評価 |
| `disconnect` | デバッグセッションを終了 |

#### 第二段階：拡張リクエスト

| DAP リクエスト | YaoXiang セマンティクス |
|--------------|------------------------|
| `setFunctionBreakpoints` | 関数名ブレークポイント |
| `setExceptionBreakpoints` | エラー / panic 時に一時停止 |
| `dataBreakpointInfo` | データブレークポイント（変数変更でトリガ） |

## 実装戦略

### フェーズ 0：インフラ（すべてのフェーズに先立つ）

**目標**：コンパイルフロントエンドがデバッグメタデータを IR に付加する。

| コンポーネント | 変更点 |
|--------------|--------|
| IR 定義 | `SourceLocation`、`VarName`、`TypeAnnotation` などのメタデータフィールドを追加 |
| Parser | 各 AST ノードがソース位置を記録 |
| TypeChecker | 型情報を IR ノードに付加 |
| テスト | IR dump に位置と変数情報が含まれることを検証 |

**ランタイムには関与しない。**

### 第一段階：インタプリタ DAP MVP

**目標**：`yaoxiang run --debug file.yx` でブレークポイント設定、ステップ実行、変数表示が可能になる。

| コンポーネント | 変更点 |
|--------------|--------|
| DAP サーバ (yx-core 新モジュール) | stdio トランスポート層、コアリクエスト処理、ブレークポイントマネージャ（ソース行 → IR ノードマッピング） |
| ランタイムデバッグ trait (yx-core) | `DebugEngine` trait 定義（pause, resume, step, get_frames, eval, get_variables） |
| インタプリタ | 実行ループでのブレークポイントチェック、一時停止 / 再開メカニズム、フレームリンクリスト維持、`InterpreterDebugEngine` 実装 |
| CLI | `yaoxiang run --debug` パラメータ |

**受入基準**：`tests/yaoxiang/` 下の任意の `.yx` ファイルに対し、VS Code でブレークポイント設定、Step Over、変数値表示が可能であること。

### 第二段階：高度なデバッグ機能

**目標**：式評価、関数ブレークポイント、並行デバッグ、例外ブレークポイント。

| コンポーネント | 変更点 |
|--------------|--------|
| 式評価エンジン | マイクロプログラムコンパイル（parser + typechecker を再利用）、VM への一時フレーム push、副作用隔離 |
| 並行デバッグ | spawn タスクリストマッピング、ブレークポイントのタスク ID バインド、stop-all / stop-this-only 一時停止戦略 |
| 関数 / 例外ブレークポイント | `setFunctionBreakpoints`、`setExceptionBreakpoints` のマッピング |
| VS Code 拡張 | デフォルトの `launch.json` テンプレートを提供 |

### 第三段階：JIT デバッグ & LLVM DWARF

**目標**：JIT エンジンが DAP を再利用、LLVM はクラッシュバックトレース用の DWARF を生成。

| コンポーネント | 変更点 |
|--------------|--------|
| JIT | `DebugEngine` trait 実装、コンパイル時に変数 → レジスタマッピングテーブル生成、ランタイムフレームリンクリスト、式の一時コンパイル |
| LLVM | IR デバッグメタデータ → LLVM `DILocation` / `DISubprogram` → DWARF（DAP インタラクションなし） |

### 依存関係

```
フェーズ 0（IR メタデータ）
    ↓
フェーズ 1（インタプリタ DAP MVP）← ここから使用可能
    ↓
フェーズ 2（高度な機能）
    ↓
フェーズ 3（JIT + LLVM DWARF）
```

### リスク

| リスク | 緩和策 |
|--------|--------|
| インタプリタの一時停止メカニズムの複雑さ | 複雑なステートマシンではなく、単純な channel / signal を使用。一時停止とは次の命令の fetch を許可しないこと |
| 式評価の型安全性 | 既存の typechecker を再利用、読み取り専用参照、副作用をコミットしない |
| DAP プロトコルの詳細 | debugpy / delve の実装を参考に。プロトコルは成熟している |
| 並行デバッグの stop-all ライブロック | タイムアウト機構 + 強制一時停止 |

## トレードオフ

### 利点

- **一つの源**：デバッグメタデータは一度だけ生成され、三つのエンジンで共有される。「インタプリタのデバッグ情報は正しいが LLVM のものは間違っている」ということは起こらない
- **ゼロインパクト**：`--debug` は一つの引数であり、この引数を付けない場合の動作は完全に変わらない
- **DAP 標準**：VS Code エコシステムに直接接続でき、カスタムエディタプロトコルやデバッガ UI は不要
- **インタプリタ優先**：デバッグは元々インタプリタに適している——柔軟、制御可能、式評価がシンプル。LLVM モードで対話的デバッグを行わないのは最も実用的な選択である

### 欠点

- **デバッグモードのパフォーマンスが低い**：インタプリタは JIT / LLVM より遥かに遅い。しかしデバッグにはパフォーマンスは不要である——誰もデバッグモードで本番負荷を走らせることを期待していない
- **LLVM デバッグが制限される**：AOT コンパイルは対話的デバッグができず、GDB / LLDB + DWARF のみ。しかしこれはトレードオフである：LLVM モードでは元々デバッグ動作の差があるべきではない
- **並行一時停止が複雑**：stop-all セマンティクスをインタプリタ上で実装するには、すべてのアクティブタスクを走査する必要がある

### 代替案

| 案 | 選択しない理由 |
|----|--------------|
| 三つのエンジンでそれぞれ DAP を実装 | 三倍の作業、三セットのバグ。「良いテースト」の原則に反する |
| DWARF のみ使用し、自前の DAP を行わない | インタプリタと JIT には DWARF 概念がなく、LLDB は VM 内部に入れない |
| Python pdb に倣ってコマンドラインデバッガを作成 | VS Code の体験はコマンドラインデバッガを圧倒的に上回る |
| DAP を LSP プロセスに詰め込む | ライフサイクルが全く異なる——LSP はプロジェクトに追従し、DAP はデバッグセッションに追従する。プロセス分離は必須要件 |

## オープン問題

- [ ] 条件付きブレークポイントの式文法は通常の YaoXiang と完全に同一か？（提案：完全に同一で、parser を再利用する）
- [ ] `spawn` ブロック内の Step Into 動作：ユーザーが Step Into を押して spawn ブロックに入ったとき、複数の並列タスクのどれを表示すべきか？（提案：最初に作成されたタスクで一時停止する）
