# RFC-034: 統一デバッグツールチェーン

## 概要

YaoXiang に統一デバッグツールチェーンを導入する。核心デザインは**一つのソース、三つの使い方**：コンパイルフロントエンドがソース位置、変数名、型情報を一流の市民として YaoXiang IR に埋め込み、インタプリタ、JIT、LLVM の三つのバックエンドが同一套メタデータ消費する。ユーザーが `yaoxiang run --debug` で DAP（Debug Adapter Protocol）サーバーを起動し、VS Code が stdio で接続し、ブレークポイント、ステップ実行、変数確認、コールスタック、式評価、並行デバッグの統一体験を得る——基盤の実行エンジンがどれであっても。

## 動機

### なぜこの機能が必要なのか？

現在 YaoXiang プログラムのバグを調べる手段は極めて原始的：

```yaoxiang
io.println("DEBUG: x = " + x.to_string())
io.println("DEBUG: entered branch A")
```

三つの致命的な問題：

1. **コンパイラ開発ブートストラッピングが阻害**：YaoXiang で YaoXiang コンパイラを書くが、コンパイラを書く人は自分のコードがデバッグできない。ブートストラッピング段階で対話式デバッグ手段がないのは袋小路。
2. **三つのエンジン、ゼロデバッグ**：インタプリタ、JIT、LLVM はそれぞれ勝手に動き、問題が起きたらユーザーは stdout に `ALL TESTS PASSED` が出ているか確認するだけ。アサーション失敗？何行目か、変数値が何か分からない。
3. **並行はブラックボックス**：`spawn` が複数のタスクを作成したが、どのタスクが落ちた？変数は誰に move された？全て推測でしかない。

### 設計目標

- **統一体験**：インタプリタで止まるブレークポイントは JIT でも止まり、LLVM も一貫したソースマッピング。ユーザーは基盤エンジンの違いを意識しない。
- **一つのソース**：デバッグメタデータは IR と共に流れ、繰り返し定義せず、二套マッピングを維持しない。
- **零侵入**：`yaoxiang run --debug` 一つのパラメータで、このパラメータ 없을 시 컴파일 및 실행 동작이 완전 불변.
- **DAP 標準**：VS Code エコシステムに直接接続し、エディタプロトコルを再発明しない。

## 提案

### 核心設計

アーキテクチャ概要：

```
┌──────────────────────────────────────────────────────┐
│                    VS Code / エディタ                  │
│              DAP Client (launch.json)                 │
└────────────────────────┬─────────────────────────────┘
                         │ stdio
┌────────────────────────▼─────────────────────────────┐
│                 DAP サーバー (yx-core)                 │
│  ┌─────────┐  ┌──────────┐  ┌───────────────────┐    │
│  │ セッション│  │ ブレーク │  │ 式評価エンジン    │    │
│  │ 管理    │  │ ポイント  │  │                  │    │
│  └─────────┘  │ 管理    │  └───────────────────┘    │
│              └──────────┘                           │
└────────────────────────┬─────────────────────────────┘
                         │ クエリ/制御
┌────────────────────────▼─────────────────────────────┐
│                  ランタイムデバッグインタフェース (trait)    │
│  pause / resume / step / get_frames / eval / ...    │
└────┬──────────────────┬──────────────────┬──────────┘
     │                  │                  │
┌────▼────┐    ┌───────▼───────┐    ┌─────▼──────┐
│ インタプリタ│    │ JIT (RFC-028)│    │ LLVM AOT   │
│ 直接消費   │    │ 軽量デバッグ │    │ IRメタデータ│
│ IRメタデータ│    │ テーブル生成 │    │ → DWARF   │
└─────────┘    └───────────────┘    └────────────┘
```

**重要な設計上の決定**：

1. **DAP サーバーとランタイムは trait で分離**。サーバーは基盤がインタプリタか JIT かを気にしない——`DebugEngine` trait を通じて命令だけを送信。各エンジンが同一 trait を独立実装。
2. **`yaoxiang run --debug` はインタプリタ使用を強制**。デバッグには制御可能性が必要で、性能は不要。LLVM モードでは DWARF 生成のみ（事後トレース用/core dump / crash report）、対話式デバッグ는提供しない.
3. **`yaoxiang run` から入口発見ロジックを再利用**。新しいサブコマンド 도입せず、心智モデルは"デバッグモードでプログラムを実行する"。

### IR デバッグメタデータ

既存の YaoXiang IR にメタデータを付加し、新しい IR 種を追加しない。全てのメタデータは**コンパイルフロントエンド**一箇所で生成され、バックエンド消費は読み取り専用：

| メタデータ | 付着点 | 説明 |
|--------|--------|------|
| `SourceLocation` | 各 IR ノード | ソースファイル:行番号:列番号 |
| `VarName` | 変数宣言/バインディングノード | ソースコード内の変数名 |
| `TypeAnnotation` | 変数/式ノード | 推論された型（コンパイル時述語含む） |
| `ScopeBoundary` | ブロック/関数出入口 | 変数スコープのライフサイクル |
| `SpanInfo` | spawn ノード | spawn ブロック内のタスク境界 |

### 起動フロー

```
yaoxiang run --debug file.yx
    │
    ├── Phase 1: コンパイル（デバッグメタデータ付き）
    │   ├── 解析 → AST
    │   ├── 型チェック + コンパイル時述語検証
    │   └── IR への降級（デバッグメタデータ付加）
    │
    ├── Phase 2: インタプリタエンジン使用
    │   └── --debug モードでは --release 有無に関わらずインタプリタ使用
    │
    ├── Phase 3: DAP サーバー起動
    │   ├── stdio 転送チャネル初期化
    │   ├── VS Code attach 待機
    │   ├── attach 成功後プログラム入口で一時停止
    │   └── 対話デバッグループへ移行
    │
    └── Phase 4: プログラム終了 / デバッグセッション終了 → 終了
```

### 各モードの差異

| モード | デバッグ方式 |
|------|---------|
| `yaoxiang run --debug` | インタプリタ強制、全機能 DAP 対話デバッグ |
| `yaoxiang run --release` | DWARF 生成、事后トレース用（core dump / crash report）、DAP 起動なし |
| `yaoxiang run`（通常） | デバッグメタデータなし、デバッグサポートなし |

## 詳細設計

### 1. ブレークポイント

```
ブレークポイント型：
├── ソース行ブレークポイント    → コンパイルフロントエンドが位置メタデータ生成、バックエンドがクエリ一致
├── 関数入口ブレークポイント  → 関数呼び出し時にトリガー（第二フェーズ）
├── 条件ブレークポイント      → 式評価が true の時にトリガー
└── データブレークポイント      → 変数が変更された時にトリガー（第二フェーズ）
```

**ソース行ブレークポイント核心ロジック**：

```
VS Code 送信: "file.yx:42 にブレークポイント設定"
    │
DAP サーバー:
    ├── IR 中の SourceLocation == (file.yx, 42) の全ノードをクエリ
    ├── ランタイムへ転送: "これらの IR アドレスで一時停止"
    └── ランタイムが返す: ブレークポイント ID

プログラムが IR ノードに到達 → ランタイムがチェック: このノードはブレークポイントリストにあるか？
    ├── 通常ブレークポイント → 一時停止、DAP サーバーに通知
    └── 条件ブレークポイント → 条件式を評価 → true のみ一時停止
```

**三つのエンジン実装**：

| | インタプリタ | JIT | LLVM |
|---|---|---|---|
| ブレークポイント挿入方式 | 実行ループで IR ノード ID チェック | JIT が機械語に `int3` 挿入 | LLVM DWARF + ハードウェアブレークポイント利用 |
| 条件ブレークポイント評価 | 直接式を解釈 | 条件式を一時 JIT コンパイル | DWARF 式スタック + 評価 |
| 性能オーバーヘッド | 各 IR ノードでテーブル参照一回追加 | ブレークポイント箇所のみオーバーヘッド | ほぼゼロオーバーヘッド（ハードウェアブレークポイント） |

### 2. ステップ実行

```
Step Over    → 現在の行を実行、関数呼び出し内部をスキップし次の行で停止
Step Into    → 現在の行の関数呼び出し内部に入る
Step Out     → 現在の関数のリターンまで実行
Continue     → 次のブレークポイントまたはプログラム終了まで実行再開
```

**実装ロジック**：ステップ操作は本質的には**一時ブレークポイント**。ユーザーが明示的に設定したブレークポイントと同一のメカニズムを共有し、二つのシステムではなく、一つのシステムの二種類の使い方。

```
Step Over:
    現在のソース行番号 = query_line(frame)
    → 次の行に一時ブレークポイント設定
    → 現在の行が関数呼び出しの場合: 呼び出し点後に一時ブレークポイント設定
    → Continue → 一時ブレークポイントにヒット → 削除 → 一時停止

Step Into:
    現在の呼び出し目標の最初の実行可能位置
    → 関数本体の最初の IR ノードのソース位置を検索
    → 一時ブレークポイント設定 → Continue → ヒット → 一時停止

Step Out:
    現在のスタックフレームのリターンアドレス
    → caller の次の行を検索
    → 一時ブレークポイント設定 → Continue → ヒット → 一時停止
```

**一時ブレークポイントの四つのエッジケース処理**：

1. **並行帰属**：一時ブレークポイントは現在のタスク ID にバインドされ、他のタスクがヒットした場合は直接無視。
2. **Step Over spawn ブロック**：spawn 外部で Step Over すると spawn ブロック全体を実行してから以降にジャンプ。spawn 内部へのデバッグ 진입には Step Into を使用。
3. **一時ブレークポイント未ヒット**：ウォッチドッグタイムアウト設定（30秒間ブレークポイント未ヒット）→ 強制一時停止 → VS Code に通知。同時にプログラム終了イベントをlisten → 即座にクリーンアップ。
4. **同一行に複数の IR ノード**：Step Over の一時ブレークポイントは `ignore_current_line` をマークし、ヒット後にソース行番号が現在の行番号と等しい場合は無視して続行。

### 3. 変数検査とスコープ

```
VS Code リクエスト: "現在のフレームの変数リスト"
    │
DAP サーバー:
    ├── 現在の一時停止点の IR ノードをクエリ
    ├── 現在の ScopeBoundary 内の変数バインディングを走査
    │   └── 各バインディングが返す: (名前, 型, ランタイム値参照)
    └── VariablesResponse を組立 → VS Code
```

**スコープ階層**：

```
┌─ Globals ───────────────────────────┐
│  モジュールレベルバインディング: 定数、│
│  型エイリアス、グローバル変数          │
├─ Locals ────────────────────────────┤
│  現在の関数内で可視のローカル変数     │
│  ├── 引数 (関数パラメータ)           │
│  └── ローカルバインディング (let / 代入)│
├─ Captured ──────────────────────────┤
│  spawn ブロック / クロージャがキャプチャ │
│  した外部変数                        │
│  所有権状態を表示: move済み / ref共有  │
└─────────────────────────────────────┘
```

**エンジンの差異**：

| エンジン | 変数値取得 |
|------|-----------|
| インタプリタ | VM スタックフレームとヒープを直接読み取り。各値はメモリに明確に表示 |
| JIT | レジスタとスタック上の値 → JIT コンパイル時に"変数 → レジスタ/スタックスロット"マッピングテーブルを記録必要 |
| LLVM | DWARF の `.debug_info` セクション → `DW_AT_location` → LLDB がネイティブサポート |

**特殊型の表示**：コンパイル時述語が型を洗練させた表示はデバッグに価値ある情報を提供：

```
x: Positive(x)  →  表示 "Int (x > 0 = True)"
y: Sorted(y)    →  表示 "Array(Int) (ソート保証)"
result: T       →  表示ランタイムの具体的な型
```

### 4. コールスタック

```
DAP リクエスト: StackTrace
    │
返す:
┌──────────────────────────────────┐
│ #0  process_item()  file.yx:42  │  ← 現在の一時停止点
│     locals: item = "hello"       │
│     spawn タスク ID: task-3     │
├──────────────────────────────────┤
│ #1  main()          file.yx:67  │  ← caller
│     locals: data = ["hello", ...]│
├──────────────────────────────────┤
│ #2  <entry>         file.yx:1   │  ← ルート
└──────────────────────────────────┘
```

各フレームが記録: 関数シグネチャ、呼び出し位置（ソースファイル+行番号）、ローカル変数（遅延評価）、spawn コンテキスト（タスク ID）。

インタプリタと JIT は各自フレームリンクリストを維持。フレームの取得는零コストではない——しかしデバッグモードは零オーバーヘッドを追求しない。

### 5. 式評価（Watch / REPL）

ユーザーがブレークポイントで任意の YaoXiang 式を入力：

```
Watch: x + y         → 計算結果を返す
Watch: items[2].name → 複雑な構造にアクセス
Watch: f(x)          → 関数を呼び出す（副作用リスクあり）
```

**評価策略**：

```
ユーザー入力式
    │
├── コンパイラフロントエンドが式を解析
├── 現在のフレームコンテキストで型チェック
├── 変数値は現在のフレームから取得（読み取り専用参照）
├── 式を独立マイクロプログラムとして実行
│   └── 外部変数の変更不允许
│   └── spawn 不允许
│   └── IO 不允许（またはオプションで有効化）
└── 結果値を返す → 元のフレーム状態は完全不変
```

**インタプリタの天然サンドボックス**：式評価は新しいサンドボックスを作成するのではなく——インタプリタ自体がサンドボックス。式評価只是一時的にフレームを push し、使用后就销毁。正常的実行と同一 VM を共有するが、副作用をコミットしない。

**関数呼び出し評価**：デフォルトで許可だが、ユーザーに"この式には副作用がある可能性があります"と警告し、ユーザーの確認後に実行。

**エンジンの差異**：

| エンジン | 式評価 |
|------|-----------|
| インタプリタ | 既存の eval コードパスを再利用、現在のフレーム環境を注入 |
| JIT | 式を一時コンパイル → 現在のフレームにリンク → 実行 → 臨時代碼を破棄 |
| LLVM | サポート外——LLVM モードでは対話デバッグを提供しない |

### 6. 並行デバッグ

**タスクモデル可視性**：

DAP の `threads` コンセプトは YaoXiang の `spawn` タスクにマッピング。各タスクは独自のスタックフレームリストと実行状態を持つ。

```
┌─ Threads ───────────────────────────┐
│  ● task-1  main()     file.yx:10   │ ← 現在フォーカス中
│  ▶ task-2  fetch()    file.yx:34   │ ← 実行中
│  ⏸ task-3  process()  file.yx:56   │ ← ブレークポイントで一時停止
│  ◼ task-4  write()    終了済み       │
└─────────────────────────────────────┘
```

**並行コンテキストでのブレークポイント**：

| 一時停止モード | 動作 | 適用シナリオ |
|----------|------|---------|
| `stop-all`（デフォルト） | 一つのタスクがヒット → 全タスク一時停止 | データ競合、グローバル状態デバッグ |
| `stop-this-only` | ヒットしたタスクのみ一時停止、他は続行 | 独立タスクロジックデバッグ |

**spawn ブロックのステップセマンティクス**：

```
spawn {          // Step Over → spawn ブロック全体を実行
    task_a()     // Step Into → task_a に入る
    task_b()     // 並行実行、個別の step の影響なし
}
```

### 7. DAP プロトコルマッピング

#### 第一フェーズ：コアリクエスト

| DAP リクエスト | YaoXiang セマンティクス |
|----------|--------------|
| `initialize` | 能力ネゴシエーション: ブレークポイント、ステップ実行、変数、スタックフレームサポート |
| `launch` / `attach` | YaoXiang プログラムを起動/アタッチ（`--debug` は attach モード） |
| `setBreakpoints` | ソース行ブレークポイント設定 |
| `configurationDone` | ブレークポイント準備完了、実行開始 |
| `threads` | 全アクティブな spawn タスクリストを返す |
| `stackTrace` | 指定タスクのスタックフレームリストを返す |
| `scopes` | 現在のフレームの変数スコープを返す |
| `variables` | 指定スコープの変数リストを返す |
| `continue` | 実行再開 |
| `next` | Step Over |
| `stepIn` | Step Into |
| `stepOut` | Step Out |
| `pause` | 全タスクを中断 |
| `evaluate` | 現在のフレームで式を評価 |
| `disconnect` | デバッグセッション終了 |

#### 第二フェーズ：拡張リクエスト

| DAP リクエスト | YaoXiang セマンティクス |
|----------|--------------|
| `setFunctionBreakpoints` | 関数名ブレークポイント |
| `setExceptionBreakpoints` | エラー/panic 時に一時停止 |
| `dataBreakpointInfo` | データブレークポイント（変数変更トリガー） |

## 実装戦略

### フェーズゼロ：インフラ（前置き）

**目標**：コンパイルフロントエンドが IR にデバッグメタデータを付加。

| コンポーネント | 変更点 |
|------|------|
| IR 定義 | 新規 `SourceLocation`、`VarName`、`TypeAnnotation` 等のメタデータフィールド追加 |
| Parser | 各 AST ノードがソース位置を記録 |
| TypeChecker | 型情報を IR ノードに付着 |
| テスト | IR dump が位置と変数情報を含むことを検証 |

**ランタイム无关。**

### 第一フェーズ：インタプリタ DAP MVP

**目標**：`yaoxiang run --debug file.yx` でブレークポイント設定、ステップ実行、変数確認が可能。

| コンポーネント | 変更点 |
|------|------|
| DAP サーバー (yx-core 新モジュール) | stdio 転送層、コアリクエスト処理、ブレークポイント管理（ソース行 → IR ノードマッピング） |
| ランタイムデバッグ trait (yx-core) | `DebugEngine` trait 定義（pause, resume, step, get_frames, eval, get_variables） |
| インタプリタ | 実行ループ内ブレークポイントチェック、一時停止/再開メカニズム、フレームリスト管理、`InterpreterDebugEngine` 実装 |
| CLI | `yaoxiang run --debug` パラメータ |

**受入基準**：`tests/yaoxiang/` 下の任意の `.yx` ファイルに対して、VS Code でブレークポイント設定、Step Over、変数値確認が可能。

### 第二フェーズ：高度なデバッグ機能

**目標**：式評価、関数ブレークポイント、並行デバッグ、例外ブレークポイント。

| コンポーネント | 変更点 |
|------|------|
| 式評価エンジン | マイクロプログラムコンパイル（parser + typechecker 再利用）、VM への一時フレーム push、副作用隔離 |
| 並行デバッグ | spawn タスクリストマッピング、ブレークポイントバインディングタスク ID、stop-all / stop-this-only 一時停止策略 |
| 関数/例外ブレークポイント | `setFunctionBreakpoints`、`setExceptionBreakpoints` マッピング |
| VS Code 拡張 | デフォルト `launch.json` テンプレートを提供 |

### 第三フェーズ：JIT デバッグ & LLVM DWARF

**目標**：JIT エンジンが DAP を再利用、LLVM がクラッシュトレース用の DWARF を生成。

| コンポーネント | 変更点 |
|------|------|
| JIT | `DebugEngine` trait 実装、コンパイル時に変数→レジスタマッピングテーブル生成、ランタイムフレームリスト、式の一時コンパイル |
| LLVM | IR デバッグメタデータ → LLVM `DILocation` / `DISubprogram` → DWARF（DAP 対話なし） |

### 依存関係

```
フェーズゼロ（IR メタデータ）
    ↓
フェーズ一（インタプリタ DAP MVP）  ← ここから使用可能
    ↓
フェーズ二（高度な機能）
    ↓
フェーズ三（JIT + LLVM DWARF）
```

### リスク

| リスク | 緩和策 |
|------|------|
| インタプリタ一時停止メカニズムの複雑さ | 複雑な状態マシンではなくシンプルな channel/signal を使用、一時停止は次の命令 fetch をさせないだけ |
| 式評価の型安全性 | 既存の typechecker を再利用、読み取り専用参照、副作用をコミットしない |
| DAP プロトコルの詳細さ | debugpy / delve の実装を参照、プロトコルは成熟している |
| 並行デバッグの stop-all ライブロック | タイムアウトメカニズム + 強制一時停止 |

## トレードオフ

### メリット

- **一つのソース**：デバッグメタデータは一回だけ生成され、三つのエンジンが共有。"インタプリタのデバッグ情報は正しいが LLVM は間違っている"不会出现
- **零侵入**：`--debug` 一つのパラメータで、このパラメータ 없을 시 동작이 완전 불변
- **DAP 標準**：VS Code エコシステムに直接接続、カスタムエディタプロトコルやデバッガ UI を再発明不要
- **インタプリタ優先**：デバッグは本質的にインタプリタに適している——柔軟、制御可能、式評価シンプル。LLVM モードで対話デバッグを提供しないのは最も現実的な選択

### デメリット

- **デバッグモードの性能は悪い**：インタプリタは JIT/LLVM よりずっと遅い。しかしデバッグには性能は不要——デバッグモードで本番負荷を実行する人はいない
- **LLVM デバッグは制限付き**：AOT コンパイルでは対話デバッグ不可、GDB/LLDB + DWARF のみ使用可能。だがこれはトレードオフ：LLVM モードではそもそもデバッグ動作に差异不该存在
- **並行一時停止は複雑**：stop-all セマンティクスはインタプリタ上で全アクティブなタスクを走査する必要があり実装が難しい

### 代替案

| 案 | なぜ選択しないか |
|------|-------------|
| 三つのエンジンが各自 DAP を実装 | 三倍の仕事、三套の bug。"良い口味"に違反 |
| DWARF のみ使用、自前の DAP なし | インタプリタと JIT には DWARF コンセプトがなく、LLDB は VM 内部に入れない |
| Python pdb 風にコマンドラインデバッガを作成 | VS Code 体験はコマンドラインデバッガを大幅に上回る |
| DAP を LSP プロセスに押し込める | ライフサイクルが完全に異なる——LSP はプロジェクトに従い、DAP はデバッグセッションに従う。プロセス分離はハード要件 |
