## 摘要

本ドキュメントはYaoXiang言語の**量子ネイティブサポート**と**マルチバックエンド統合**方案を定義します。核心思想：**YaoXiangの既存設計（デフォルトMove、所有権回流、不透明型、DAGスケジューラ、ジェネリック定数パラメータ）は、量子プログラミング言語の完全な基盤を自然に構成しており、新しい量子専用構文の導入は不要です**。少数の組み込み型（`Qubit`、`Complex`、`Topology`）と組み込み関数（量子ゲート、測定、トポロジ制約）を追加し、既存の言語メカニズムを活用することで、量子ネイティブセマンティクス、自动並列化による量子利用率最大化、ハイブリッド古典プログラミング、マルチバックエンドサポートを実現します。

## 動機

### なぜ量子ネイティブサポートが必要か？

現在の量子プログラミング生態系には深刻な分断が存在します：

- **低水準言語（QCIS、OpenQASM）**：物理量子ゲートを直接操作しますが、型システムや抽象化メカニズムが欠如しており、複雑なアルゴリズムを書くのが困難です。
- **高水準フレームワーク（Qiskit、Cirq、Q#）**：古典言語（Python、C#）を拡張 기반으로、量子セマンティクスはライブラリで実装しており、以下导致了：
  - 量子状態不可克隆則はユーザーが手動で遵守する必要があります（または線形型システムの补救に依存）。
  - 量子ゲート操作と古典コードの構文が分離しており、学習コストが高い。
- **ハイブリッド計算**：量子部と古典部は明示的に分離する必要があり、统一的なデータフローモデルが欠如しています。

### 現在の問題

YaoXiangの既存設計はこれらの問題を解決するために完全な基盤を提供しています：

| 量子計算ニーズ | YaoXiang既存設計 | 説明 |
|-------------|-------------------|------|
| 量子状態不可克隆 | **デフォルトMoveセマンティクス** | 代入時に所有権が移動し、暗黙的な複製がないため、no-cloning定理に自然に準拠 |
| 量子ゲートがユニタリ変換 | **所有権回流** | `q = H(q)`は元のqubitを消費し、新しいqubitを返し、ゲートのセマンティクスを正確に対応 |
| 絡み合い状態 | **不透明型** | `BellPair`は整体としてのみ操作可能で、コンパイラがライフサイクルを追跡し、誤った分解を防止 |
| 物理トポロジ制約 | **ジェネリック定数パラメータ** | `Qubit(Topology, N)`はコンパイル時に隣接性をチェック |
| 測定崩壊 | **空状態の再利用** | 測定後のqubitが空になり、再初期化して量子状態崩壊をシミュレート |
| 量子回路の自動並列化 | **DAGスケジューラ** | 関数内の文はデータ依存性に基づいて自動並列化され、依存のないゲートは自然に並行実行 |
| ハイブリッド古典-量子制御フロー | **統一構文** | 量子操作と古典操作は同じ`name: type = value`形式を使用 |

**YaoXiangは「量子サポートを追加」するのではなく、自分の設計がすでに量子ネイティブであることを発見しました。**

> **セマンティクス説明**：本ドキュメントではYaoXiangの**所有権セマンティクス**を使用して量子操作を表現します。コンパイラレベルでno-cloning（所有権安全）を保証します。言語レベルの「消費-作成」は**所有権移動**の構文表現です——消費＝所有権の取得、戻り値＝所有権の委譲。実装レベルでは、実際の可逆量子ゲート（量子状態をインプレースで変更）であり、実際に「新しい量子状態を作成」するのではありません。

### 設計目標

1. **ゼロ新構文**：`quantum`、`circuit`などのキーワードを導入せず、すべての量子特性は既存の言語メカニズムで表現されます。
2. **型安全**：コンパイラが量子状態が複製または不法使用されないことを保証します。
3. **トポロジ制約のコンパイル時チェック**：ジェネリック定数パラメータ`Qubit(T, N)`を通じて、コンパイル時に2量子ビットゲート操作が物理トポロジに適合するかを検証します。
4. **マルチバックエンド透過性**：同一の量子コードはQIR（汎用エコシステム）またはQCIS（国産量子命令セット）にコンパイルでき、コマンドライン引数で切り替え可能です。
5. **ハイブリッド古典のシームレス統合**：量子計算は古典関数を自由に呼び出すことができ、古典コードも量子データを操作できます（`ref`共有経由だが、所有権制約に従う）。

## 提案

### コア設計

#### 1. 量子型システムマッピング

**基底型**：
```yaoxiang
Qubit: Type0 = primitive_qubit
Complex: Type0 = { re: Float, im: Float }
```
- `Qubit`は一級市民型であり、所有権ルール（Move、RAII）に従います。
- `Complex`は振幅の表現に使用され、コンパイラはインライン最適化できます。

**量子ゲートを関数として**：
```yaoxiang
# 組み込み関数シグネチャ
H: (Qubit) -> Qubit = builtin_hadamard
X: (Qubit) -> Qubit = builtin_pauli_x
Y: (Qubit) -> Qubit = builtin_pauli_y
Z: (Qubit) -> Qubit = builtin_pauli_z
CNOT: (control: Qubit, target: Qubit) -> { Qubit, Qubit } = builtin_cnot
```
- すべてのゲートは入力qubitを消費し、新しいqubit（または絡み合い対）を返します。所有権回流構文`q = H(q)`は数学的セマンティクスに直接対応します。
- マルチqubitゲートは構造体を返し、パターンマッチングまたはフィールドアクセスで結果を取得します。

**測定**：
```yaoxiang
measure: (Qubit) -> Int = builtin_measure   # qubitを消費し、古典ビットを返す
measure_all: (List(Qubit)) -> List(Int) = builtin_measure_all
```
- 測定後、qubitは消費され（空になり）、ユーザーは空状態の再利用を通じて再初期化できます。

**初期化**：
```yaoxiang
qubit: (Int) -> Qubit = builtin_qubit   # 0または1で基底状態を初期化
```

#### 2. 絡み合いと不透明型カapsule化

絡み合い対を不透明型としてカapsule化し、複合操作のみを提供し、分解を禁止します：

```yaoxiang
# 組み込み不透明型
BellPair: Type0 = primitive_bell_pair

# 組み込み関数 - 整体としてのみ操作可能
CNOT: (Qubit, Qubit) -> BellPair
measure_bell: (BellPair) -> { Int, Int }
split_bell: (BellPair) -> { Qubit, Qubit }  # 絡み合い対の分割（注意して使用）
apply_cnot_to_bell: (BellPair, Qubit) -> BellPair
```

**重要設計**：
- フィールドアクセッサを提供せず、組み込み関数でのみ整体として操作可能
- `measure_bell(bp)`は絡み合い対全体を一度に消費し、古典ビットを返す
- コンパイラは絡み合い対の完全なライフサイクルを追跡可能

**Python/Qiskitとの比較**：
```
Python (Qiskit): 実行時に回路を構築し、エラーは提出後に才发现
YaoXiang:       コンパイル時にほとんどの論理エラーを捕获
```

**残りの10%**（物理的脱コヒーレンス、ゲートエラーなど）はハードウェアの問題であり、言語では解決できません。

#### 3. 物理トポロジ制約

量子チップは制約付きトポロジグラフであり、**任意の2つのqubitが2量子ビットゲートを実行できるわけではなく**、隣接している必要があります。YaoXiangは**ジェネリック定数パラメータ**を使用してコンパイル時にトポロジ制約を保証します。

**トポロジ型定義**：
```yaoxiang
# トポロジを型として定義し、隣接行列を含む
Topology: Type0 = primitive_topology

# 組み込みトポロジ定数
Linear8: Topology = topology(8)          # 線形8ビット: 0-1-2-3-4-5-6-7
Grid3x3: Topology = topology(3, 3)        # 3x3グリッド
Ring16: Topology = topology(16, ring)    # リング16ビット
```

**Qubitのトポロジと位置バインディング**：
```yaoxiang
# Qubit(T, N) - Tはトポロジ型、Nは定数位置パラメータ
q0: Qubit(Grid3x3, 0)   # Grid3x3トポロジ、位置 (0,0)
q1: Qubit(Grid3x3, 1)   # Grid3x3トポロジ、位置 (0,1)
q2: Qubit(Grid3x3, 2)   # Grid3x3トポロジ、位置 (0,2)
q3: Qubit(Grid3x3, 3)   # Grid3x3トポロジ、位置 (1,0)
```

**ゲート操作の自動制約**：
```yaoxiang
# CNOT型シグネチャはトポロジ制約を含む
CNOT: (T: Topology, I: Int, J: Int) -> (
    (Qubit(T, I), Qubit(T, J)) -> { Qubit(T, I), Qubit(T, J) }
) when adjacent(T, I, J)

# コンパイル時チェック
CNOT(q0, q1)  # ✅ Grid3x3では(0,0)と(0,1)は隣接
CNOT(q0, q2)  # ❌ コンパイルエラー：(0,0)と(0,2)は隣接していない
```

**`adjacent`コンパイル時制約**：
- `adjacent`はコンパイル時関数であり、トポロジの隣接行列を使用して静的チェックを実行します
- 定数インデックス時は100%コンパイル時検証
- 動的インデックス時は実行時チェックコードを生成

**仮想から物理へのマッピング**：
```yaoxiang
# コンパイル時に具体的な物理位置が分からない？型推論を使用
q = qubit(Grid3x3)  # 位置0を自動割り当て、以降推論可能
```

#### 4. 所有権と量子状態の線形流動

すべての量子操作はMoveセマンティクスに従い、qubitが複製されないことを確保します：
```yaoxiang
q = qubit(0)
q2 = q          # ❌ コンパイルエラー：qはすでに移動されており、使用不可
q = H(q)        # ✅ qを消費し、新しいqを返す
measure(q)      # ✅ qを消費した後、qは空になる
q = qubit(0)    # ✅ 空状態の再利用
```

#### 4. 自動並列化とDAGスケジューリング

StandardまたはFull Runtimeでは、DAGスケジューラが量子プログラムを自動解析します：
```yaoxiang
apply_two_qubit_gates: () -> {Qubit, Qubit} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    # 上記2行はデータ依存性がなく、DAGは自動並列実行
    CNOT(q1, q2)   # q1とq2に依存、自动待機
}
```
- スケジューラは`num_workers`設定（物理量子プロセッサ数）を活用して本当の並列実行を実現します。
- ユーザーはゲートの順序を手動で配置する必要はなく、データフローだけを記述します。

#### 5. ハイブリッド古典計算

古典コードと量子コードが完全に融合します：
```yaoxiang
grover_search: (target: Int) -> Int = () => {
    n = 4
    qubits = List(Qubit)()
    for i in 0..n {
        qubits.append(H(qubit(0)))
    }
    # 古典ループと量子操作が混合
    oracle(qubits, target)   # oracleは量子ゲートシーケンス
    qubits = diffusion(qubits)
    results = measure_all(qubits)
    return decode_result(results)   # 古典後処理
}
```
- 同一関数内で量子ゲートと古典制御フローを自由に混合できます。
- 所有権システムは量子変数が古典分支で誤って複製されないことを確保します。

#### 6. マルチバックエンドサポートアーキテクチャ

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   YaoXiangソース │     │   型チェック    │     │   DAG中間表現   │
│   (統一構文)     │────▶│   + 所有権分析  │────▶│   (データフロー図)│
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                          │
                                                          ▼
                          ┌─────────────────────────────────────────────┐
                          │           コード生成バックエンド（プラグイン式）│
                          ├─────────────────┬───────────────────────────┤
                          │  QIRバックエンド │  QCISバックエンド          │
                          │  (汎用エコシステム)│  (国産量子命令セット)      │
                          ├─────────────────┼───────────────────────────┤
                          │  - .llファイル出力│  - .qcisテキスト出力      │
                          │  - 複数QPU対応   │  - 中科院/国盾ハードウェア対応│
                          └─────────────────┴───────────────────────────┘
```

- **コンパイルフロー**：フロントエンド統一→DAG構築→バックエンド選択→ターゲットコード生成。
- **QIRバックエンド**：DAGノードをQIRの量子ゲートintrinsicにマッピングし、LLVMビットコードを生成、LLVM оптимизацияをさらに活用可能。
- **QCISバックエンド**：DAGをQCIS命令（例：`H q0`）にシリアル化し、量子チップコンソールに直接submit可能。

### 例

#### ベル状態調製と測定
```yaoxiang
bell_measure: () -> {Int, Int} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    bell = CNOT(q1, q2)  # BellPair不透明型を返す
    result = measure_bell(bell)  # 2ビットを一度に測定
    return result
}
```

#### 量子テレポーテーション（簡略版）
```yaoxiang
teleport: (msg: Qubit, bell: BellPair) -> Qubit = (msg, bell) => {
    # 絡み合い対を分割して2つの独立qubitを取得
    (alice_qubit, bob_qubit) = split_bell(bell)

    # アリス操作
    (msg, alice_qubit) = CNOT(msg, alice_qubit)
    msg = H(msg)
    a1 = measure(msg)
    a2 = measure(alice_qubit)

    # 古典情報伝送（スケジューラが自動処理）
    # ボブ操作
    if a2 == 1 { bob_qubit = X(bob_qubit) }
    if a1 == 1 { bob_qubit = Z(bob_qubit) }
    return bob_qubit
}
```

## 詳細設計

### 組み込み型と関数定義

`compiler/builtins`モジュールに追加：
```rust
builtins.insert("Qubit", Ty::Primitive(Primitive::Qubit));
builtins.insert("Complex", Ty::Record(vec![
    ("re", Ty::Primitive(Primitive::Float)),
    ("im", Ty::Primitive(Primitive::Float)),
]));

// 量子ゲート
for (name, sig) in GATES {
    builtins.insert(name, Ty::Function(vec![Ty::Qubit], Ty::Qubit));
}
builtins.insert("CNOT", Ty::Function(
    vec![Ty::Qubit, Ty::Qubit],
    Ty::Record(vec![
        ("q0", Ty::Qubit),
        ("q1", Ty::Qubit)
    ])
));
builtins.insert("measure", Ty::Function(vec![Ty::Qubit], Ty::Primitive(Primitive::Int)));
builtins.insert("qubit", Ty::Function(vec![Ty::Primitive(Primitive::Int)], Ty::Qubit));
```

### 所有権チェッカーのQubit特殊処理

- `Qubit`は`!Copy`（デフォルトMove）とマークされ、暗黙的な複製を禁止します。
- 測定関数`measure`のパラメータは`Qubit`（値渡し）であり、所有権を消費します。
- マルチqubitゲートが返すレコード型では、フィールドはすべて`Qubit`であり、所有権ルールに従う必要があります。

### DAGスケジューラの量子ゲート最適化

- 量子ゲートノードは純粋関数（副作用なし）と見なされ、スケジューラは依存関係のないゲートを自由に並べ替え可能。
- スケジューラが「量子命令シーケンス」を出力する際、データ依存性を保持し、並列ゲートをグループ化（マルチ量子プロセッサ向け）。
- `--target-num-qubits`と`--target-topology`設定をサポートし、レイアウトとルーティング（将来拡張）に使用。

### QIRバックエンド詳細マッピング

| YaoXiang操作 | QIR命令 |
|---------------|----------|
| `H(q)` | `call void @__quantum__qis__h__body(%Qubit* %q)` |
| `CNOT(q1, q2)` | `call void @__quantum__qis__cnot__body(%Qubit* %q1, %Qubit* %q2)` |
| `measure(q)` | `%result = call i1 @__quantum__qis__mz__body(%Qubit* %q)` |
| `qubit(0)` | `%q = call %Qubit* @__quantum__rt__qubit_allocate()` |

QIRバックエンドはLLVMの`-O2`をさらに活用し、QIR Alliance互換のビットコードを出力します。

### QCISバックエンド詳細マッピング

| YaoXiang操作 | QCIS命令 |
|---------------|-----------|
| `H(q)` (qは物理ビット2に対応) | `H 2` |
| `CNOT(q1,q2)` (q1→ビット0, q2→ビット1) | `CNOT 0 1` |
| `measure(q)` (ビット0) | `M 0` |
| `qubit(0)`初期化 | 最初の使用命令に暗黙含み、追加命令不要 |

- 仮想qubit（YaoXiang変数）から物理ビットへのマッピングテーブルを維持する必要があります。
- トポロジ制約チェックをサポート（将来実装）。

### ハイブリッド古典コード生成

- 古典部（ループ、条件、整数演算など）は通常通りローカルコード（x86/ARM）を生成し、FFIまたは組込み呼び出しで量子バックエンドと相互作用します。
- QIRバックエンドでは、古典部はLLVM IRに降格可能であり、QIRと混合コンパイル可能。

### 型システムへの影響

- `Qubit`と`Complex`という新しい原始型を追加します。
- `Qubit`は自動的にMoveセマンティクスを持ち、複製を禁止します。
- 量子ゲート関数シグネチャは型システムに登録する必要があります。

### 後方互換性

- ✅ 完全な後方互換性
- 新しい組み込み型と関数は既存コードに影響しない
- 量子特性はオプションであり、有効にしない場合はオーバーヘッドなし

## 权衡

### メリット

- **新構文不要**：開発者は少数の組み込み関数だけを学習すれば量子プログラムを書けます。
- **型安全**：所有権システムが自動的にqubit複製を防止し、一般的な量子プログラミングエラーを回避します。
- **自動並列化**：DAGスケジューラがゲートレベルの並列化を無料で提供し、追加のコンパイラ最適化が不要です。
- **エコシステム互換性**：QIRバックエンドによりYaoXiangは複数の量子クラウドプラットフォームで実行可能；QCISバックエンドにより自律制御を確保します。
- **ハイブリッド能力**：古典と量子の融合が自然で、複雑な量子アルゴリズム（例：Shor、Groverの古典制御）の記述に適しています。

### デメリット

- **Qubit数の静的な性質**：現在の設計ではqubit数がコンパイル時に知られていると想定しており、動的割り当ては`List(Qubit)`を通じて可能ですが、`List`のヒープ割り当ては追加オーバーヘッドをもたらす可能性があります（最適化で緩和可能）。
- **測定後の再利用**：空状態の再利用はqubitの再初期化を可能にしますが、物理量子ビットには弛緩時間が存在する可能性があり、ランタイムシステムで処理する必要があります（現時点ではユーザーが負責）。
- **動的トポロジマッピング**：物理トポロジがランタイム才知道の場合、コンパイル時チェックが有効にならず、実行時チェックコードを生成する必要があります（現行バージョンは静的チェックのみサポート）。

## 代替案

| 方案 | 为什么不選択 |
|------|--------------|
| `quantum`キーワードと`circuit`型の導入 | 新構文の追加により学習コストが高く、YaoXiangの簡潔設計原则に反する |
| ライブラリとしてのみ量子サポートを実装 | コンパイラの保証を活用した量子状態安全の確保ができず、DAGスケジューラとの深い統合が実現できない |
| 量子ハードウェアの成熟を待ってからサポート | 量子プログラミング言語設計の重要なウィンドウ期を逃す |
| 既存の量子フレームワーク（Qiskitなど）の再利用 | 量子セマンティクスがライブラリで実装されるため、型システムと所有権システムの安全保障が得られない |
| 量子サブロ言語の別途設計 | 言語複雑度が増加し、メンテナンスコストが高い |

## 実装策略

### フェーズ区分

| フェーズ | 期間 | 内容 |
|------|------|------|
| Phase 1 | 1ヶ月 | 基底量子型と組み込み関数：コンパイラに`Qubit`、`Complex`型を追加し、組み込み関数の型チェックを実装し、所有権チェッカーを拡張 |
| Phase 2 | 1ヶ月 | DAGスケジューラの量子ゲート認識：DAG構築ロジックを修正し、量子ゲートを純粋関数としてマークし、並列ゲートグループ出力实现 |
| Phase 3 | 2ヶ月 | QIRバックエンドプロトタイプ：DAGからQIRコードジェネレータを実装し、LLVMを統合し、QIRシミュレータに接続して検証 |
| Phase 4 | 2ヶ月 | QCISバックエンドプロトタイプ：DAGからQCIS命令への変換を実装し、仮想-物理ビットマッピングを設計し、国産量子プラットフォームに接続して検証 |
| Phase 5 | 2ヶ月 | ハイブリッド古典拡張：古典制御フローと量子ゲートの交差が正しくコード生成されることを確保し、`List(Qubit)`をサポートし、サンプルプログラムを追加 |
| Phase 6 | 2ヶ月 | 最適化とドキュメント：基本レイアウトとルーティングを実装し、ユーザーガイドと量子プログラミングチュートリアルを書き、プレビュー版をリリース |

### リスク

1. **量子ハードウェア可用性**：外部量子シミュレーターと実際のQPUの可用性に依存します。
   - **緩和**：オープンソースシミュレーター（QIR runner、Qiskit Aer）を優先的に接続し、実際のQPUは長期目標とします。

2. **バックエンド実装の複雑性**：QIRとQCIS仕様は変更される可能性があります。
   - **緩和**：コード生成インターフェースを抽象化し、バックエンド差異を隔離し、後の適応を容易にします。

3. **パフォーマンスの不確実性**：量子プログラムのパフォーマンス特性は古典プログラムとは異なります。
   - **緩和**：パフォーマンスプロファイリングツールを提供し、ユーザーがゲートレベル並列効果を理解できるようにします。

## 開放問題

- [x] **トポロジ制約**：`Qubit(Topology, N)`ジェネリック定数パラメータを通じてコンパイル時チェックとして実装済み。
- [ ] **動的量子レジスタ**：`List(Qubit)`はQCISバックエンドでどのようにマッピングしますか？対応する物理ビット数を生成できますが、実行時割り当てメカニズムが必要です。
- [ ] **エラー緩和**：エラー緩和（動的脱結合など）のbuilt-inコンストラクトを提供しますか？最初はライブラリとして実装できます。
- [ ] **既存の量子SDK相互運用性**：QASMまたはQIRモジュールをインポートできますか？将来はFFIを検討できます。
- [ ] **自動レイアウトとルーティング**：仮想qubit数が物理qubit数を超える場合、自动マッピングの方法は？
