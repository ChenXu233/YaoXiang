## 概要

このドキュメントは YaoXiang 言語の**量子ネイティブサポート**と**マルチバックエンド統合**方案を定義します。核心的な考え方：**YaoXiang の既存設計（デフォルト Move、所有権回流、不透明型、DAG スケジューラ、ジェネリクス定数パラメータ）は、量子プログラミング言語の完全な基盤を自然に構成しており、新しい量子専用構文を導入する必要は一切ありません**。少数の組み込み型（`Qubit`、`Complex`、`Topology`）と組み込み関数（量子ゲート、測定、トポロジ制約）を追加し、既存の言語メカニズムを活用することで、量子ネイティブセマンティクス、量子利用率の自動最大化、ハイブリッド古典プログラミング、およびマルチバックエンドサポートを実現します。

## 動機

### なぜ量子ネイティブサポートが必要인가？

現在の量子プログラミングエコシステムには深刻な分断があります：

- **低レベル言語（QCIS、OpenQASM）**：物理量子ゲートを直接操作しますが、型システムや抽象化メカニズムが欠けており、複雑なアルゴリズムを書くことが困難です。
- **高級フレームワーク（Qiskit、Cirq、Q#）**：古典言語（Python、C#）を拡張したもので、量子セマンティクスはライブラリで実装されており、以下の問題が生じます：
  - 量子状態コピー禁止規則はユーザーが手動で遵守する必要があります（または線形型システムの後付けに依存）。
  - 量子ゲート操作と古典コードの構文が分離しており、学習コストが高い。
- **ハイブリッド計算**：量子部分と古典部分は明示的に分離する必要があり、统一されたデータフローモデルが欠けています。

### 現在の問題

YaoXiang の既存の設計は、まさに这些问题を解決するための完全な基盤を提供します：

| 量子計算ニーズ | YaoXiang の既存設計 | 説明 |
|-------------|-------------------|------|
| 量子状態コピー禁止 | **デフォルト Move セマンティクス** | 代入により所有権が移動し、暗黙的なコピーがなく、no-cloning 定理に自然に準拠 |
| 量子ゲートはユニタリ変換 | **所有権回流** | `q = H(q)` で元の qubit を消費し、新しい qubit を返すゲートセマンティクスを正確に再現 |
| エンタングル状態 | **不透明型** | `BellPair` は全体としてのみ操作可能、コンパイラがライフサイクルを追跡し、誤った分解を防止 |
| 物理トポロジ制約 | **ジェネリクス定数パラメータ** | `Qubit(Topology, N)` でコンパイル時に隣接性を検証 |
| 測定崩壊 | **空状態再利用** | 測定後の qubit は空になり、再初期化可能、量子状態崩壊をシミュレート |
| 量子回路の自動並列化 | **DAG スケジューラ** | 関数内のステートメントはデータ依存関係に基づいて自動並列化され、依存のないゲートは自然に同時実行 |
| ハイブリッド古典-量子制御フロー | **統一構文** | 量子操作と古典操作は 동일한 `name: type = value` 形式を使用 |

**YaoXiang は「量子サポートを追加する」ではなく、自分の設計がすでに量子ネイティブであることを発見しました。**

> **セマンティクスの説明**：このドキュメントでは YaoXiang の**所有権セマンティクス**を使用して量子操作を表現します。コンパイラレベルで no-cloning（所有権安全）を保証します。言語レベルの「消費-作成」は**所有権移動の構文表現**です—消費 = 所有権の取得、戻り値 = 所有権の委譲。実際の実装では、真の可逆量子ゲート（量子状態をインプレースで変更）を使用でき、実際に「新量子状態を作成」するものではありません。

### 設計目標

1. **ゼロ新構文**：`quantum`、`circuit` などのキーワードを導入せず、すべての量子特性は既存の言語メカニズムで表現します。
2. **型安全**：コンパイラが量子状態がコピーまたは不正使用されないことを保証します。
3. **トポロジ制約のコンパイル時チェック**：ジェネリクス定数パラメータ `Qubit(T, N)` により、双量子ビットゲート操作が物理トポロジに適合するかをコンパイル時に検証します。
4. **マルチバックエンド透過性**：同一の量子コードは QIR（汎用エコシステム）または QCIS（国内量子命令セット）にコンパイルでき、コマンドライン引数で切り替えます。
5. **ハイブリッド古典的无縫**：量子計算は古典関数を自由に呼び出せ、古典コードも量子データを操作できます（`ref` 共有経由，但受所有権制約）。

## 提案

### コア設計

#### 1. 量子型システムマッピング

**基本型**：
```yaoxiang
Qubit: Type0 = primitive_qubit
Complex: Type0 = { re: Float, im: Float }
```
- `Qubit` はファーストクラスの型であり、所有権規則（Move、RAII）を遵守します。
- `Complex` は振幅を表現するために使用され、コンパイラはインライン最適化を行えます。

**量子ゲートを関数として表現**：
```yaoxiang
# 組み込み関数シグネチャ
H: (Qubit) -> Qubit = builtin_hadamard
X: (Qubit) -> Qubit = builtin_pauli_x
Y: (Qubit) -> Qubit = builtin_pauli_y
Z: (Qubit) -> Qubit = builtin_pauli_z
CNOT: (control: Qubit, target: Qubit) -> { Qubit, Qubit } = builtin_cnot
```
- すべてのゲートは入力 qubit を消費し、新しい qubit（またはエンタングル対）を返します。所有権回流構文 `q = H(q)` は数学的セマンティクスに直接対応します。
- 複数 qubit ゲートは構造体を返し、パターン照合またはフィールドアクセスで結果を取得します。

**測定**：
```yaoxiang
measure: (Qubit) -> Int = builtin_measure   # qubit を消費し、古典ビットを返す
measure_all: (List(Qubit)) -> List(Int) = builtin_measure_all
```
- 測定後、qubit は消費され（空になり）、ユーザーは空状態再利用により再初期化できます。

**初期化**：
```yaoxiang
qubit: (Int) -> Qubit = builtin_qubit   # 0 または 1 で初期状態を設定
```

#### 2. エンタングルメントと不透明型カプセル化

エンタングル対を不透明型にカプセル化し、合成操作のみを提供し、分解を禁止します：

```yaoxiang
# 組み込み不透明型
BellPair: Type0 = primitive_bell_pair

# 組み込み関数 - 全体としてのみ操作可能
CNOT: (Qubit, Qubit) -> BellPair
measure_bell: (BellPair) -> { Int, Int }
split_bell: (BellPair) -> { Qubit, Qubit }  # エンタングル対の分割（注意して使用）
apply_cnot_to_bell: (BellPair, Qubit) -> BellPair
```

**重要な設計**：
- フィールドアクセッサを提供せず、組み込み関数による全体操作のみを許可
- `measure_bell(bp)` はエンタングル対全体を一度に消費し、古典ビットを返す
- コンパイラはエンタングル対の完全なライフサイクルを追跡可能

**Python/Qiskit との比較**：
```
Python (Qiskit): ランタイムで回路を構築し、エラーは送信後に才发现
YaoXiang:       コンパイル時にほとんどのロジックエラーを捕获
```

**残りの 10%**（物理デコヒーレンス、ゲートエラーなど）はハードウェアの問題であり、言語では解决できません。

#### 3. 物理トポロジ制約

量子チップは制約付きトポロジグラフであり、**任意の2つの qubit で双量子ビットゲートを実行できるわけではなく**、隣接している必要があります。YaoXiang は**ジェネリクス定数パラメータ**を使用してコンパイル時にトポロジ制約を保証します。

**トポロジ型定義**：
```yaoxiang
# トポロジを型として定義し、隣接行列を含む
Topology: Type0 = primitive_topology

# 組み込みトポロジ定数
Linear8: Topology = topology(8)          # 線形8ビット: 0-1-2-3-4-5-6-7
Grid3x3: Topology = topology(3, 3)        # 3x3 グリッド
Ring16: Topology = topology(16, ring)    # リング型16ビット
```

**Qubit にトポロジと位置をバインド**：
```yaoxiang
# Qubit(T, N) - T はトポロジ型、N は定数位置パラメータ
q0: Qubit(Grid3x3, 0)   # Grid3x3 トポロジ、位置 (0,0)
q1: Qubit(Grid3x3, 1)   # Grid3x3 トポロジ、位置 (0,1)
q2: Qubit(Grid3x3, 2)   # Grid3x3 トポロジ、位置 (0,2)
q3: Qubit(Grid3x3, 3)   # Grid3x3 トポロジ、位置 (1,0)
```

**ゲート操作の自動制約**：
```yaoxiang
# CNOT タイプシグネチャはトポロジ制約を含む
CNOT: (T: Topology, I: Int, J: Int) -> (
    (Qubit(T, I), Qubit(T, J)) -> { Qubit(T, I), Qubit(T, J) }
) when adjacent(T, I, J)

# コンパイル時チェック
CNOT(q0, q1)  # ✅ Grid3x3 で (0,0) と (0,1) は隣接
CNOT(q0, q2)  # ❌ コンパイルエラー：(0,0) と (0,2) は隣接していない
```

**`adjacent` コンパイル時制約**：
- `adjacent` はコンパイル時関数であり、トポロジの隣接行列を使用して静的チェックを実行します
- 定数インデックス時は 100% コンパイル時検証
- 動的インデックス時はランタイムチェックコードを生成

**仮想から物理へのマッピング**：
```yaoxiang
# コンパイル時に具体的な物理的位置がわからない場合、型推論を使用
q = qubit(Grid3x3)  # 自動的に位置0を割り当て、後続の推論を実行
```

#### 4. 所有権と量子状態の線形フロー

すべての量子操作は Move セマンティクスを遵守し、qubit のコピーを防止します：
```yaoxiang
q = qubit(0)
q2 = q          # ❌ コンパイルエラー：q はすでに移動されており、再使用不可
q = H(q)        # ✅ q を消費し、新しい q を返す
measure(q)      # ✅ q を消費した後、q は空になる
q = qubit(0)    # ✅ 空状態再利用
```

#### 4. 自動並列化と DAG スケジューリング

Standard または Full Runtime 下では、DAG スケジューラが量子プログラムを自動分析します：
```yaoxiang
apply_two_qubit_gates: () -> {Qubit, Qubit} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    # 上記2行はデータ依存関係がなく、DAG が自動的に並列実行
    CNOT(q1, q2)   # q1 と q2 に依存、自動的に待機
}
```
- スケジューラは `num_workers` 設定（物理量子プロセッサ数）を活用して真の並列化を実現します。
- ユーザーはゲートの順序を手動で配置する必要はなく、データフローだけを記述すればよいです。

#### 5. ハイブリッド古典計算

古典コードと量子コードが完全に融合します：
```yaoxiang
grover_search: (target: Int) -> Int = () => {
    n = 4
    qubits = List(Qubit)()
    for i in 0..n {
        qubits.append(H(qubit(0)))
    }
    # 古典ループと量子操作の混合
    oracle(qubits, target)   # oracle は量子ゲート列
    qubits = diffusion(qubits)
    results = measure_all(qubits)
    return decode_result(results)   # 古典後処理
}
```
- 同一関数内で量子ゲートと古典制御フローを自由に混合できます。
- 所有権システムは、量子変数が古典分支で誤ってコピーされないことを保証します。

#### 6. マルチバックエンドサポートアーキテクチャ

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   YaoXiang ソース │     │   型チェック     │     │   DAG 中間表現   │
│   (統一構文)     │────▶│   + 所有権分析    │────▶│   (データフロー図) │
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                          │
                                                          ▼
                          ┌─────────────────────────────────────────────┐
                          │           コード生成バックエンド (プラグイン可能)            │
                          ├─────────────────┬───────────────────────────┤
                          │  QIR バックエンド    │  QCIS バックエンド             │
                          │  (汎用エコシステム)   │  (国内量子命令セット)           │
                          ├─────────────────┼───────────────────────────┤
                          │  - .ll ファイル出力 │  - .qcis テキスト出力          │
                          │  - 複数の QPU に適応 │  - 中科院/国盾ハードウェアに adapting │
                          └─────────────────┴───────────────────────────┘
```

- **コンパイルフロー**：フロントエンド統一 → DAG 構築 → バックエンド選択 → ターゲットコード生成。
- **QIR バックエンド**：DAG ノードを QIR の量子ゲート intrinsic にマッピングし、LLVM ビットコードを生成し、LLVM 最適化をさらに活用できます。
- **QCIS バックエンド**：DAG を QCIS 命令（例：`H q0`）にシリアライズし、量子チップコンソールに直接サブミット可能。

### 例

#### ベル状態の準備と測定
```yaoxiang
bell_measure: () -> {Int, Int} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    bell = CNOT(q1, q2)  # BellPair 不透明型を返す
    result = measure_bell(bell)  # 2つのビットを一度に測定
    return result
}
```

#### 量子テレポーテーション（簡略化）
```yaoxiang
teleport: (msg: Qubit, bell: BellPair) -> Qubit = (msg, bell) => {
    # エンタングル対を分割して2つの独立した qubit を取得
    (alice_qubit, bob_qubit) = split_bell(bell)

    # Alice の操作
    (msg, alice_qubit) = CNOT(msg, alice_qubit)
    msg = H(msg)
    a1 = measure(msg)
    a2 = measure(alice_qubit)

    # 古典情報伝達 (スケジューラが自動的に依存関係を処理)
    # Bob の操作
    if a2 == 1 { bob_qubit = X(bob_qubit) }
    if a1 == 1 { bob_qubit = Z(bob_qubit) }
    return bob_qubit
}
```

## 詳細設計

### 組み込み型と関数定義

`compiler/builtins` モジュールに追加：
```rust
builtins.insert("Qubit", Ty::Primitive(Primitive::Qubit));
builtins.insert("Complex", Ty::Record(vec![
    ("re", Ty::Primitive(Primitive::Float)),
    ("im", Ty::Primitive(Primitive::Float)),
]));

// 量子ゲート
for (name, sig) in GATES {
    builtins.insert(name, Ty::Function(vec![Ty::Qubit], Ty::Qubit));
}
builtins.insert("CNOT", Ty::Function(
    vec![Ty::Qubit, Ty::Qubit],
    Ty::Record(vec![
        ("q0", Ty::Qubit),
        ("q1", Ty::Qubit)
    ])
));
builtins.insert("measure", Ty::Function(vec![Ty::Qubit], Ty::Primitive(Primitive::Int)));
builtins.insert("qubit", Ty::Function(vec![Ty::Primitive(Primitive::Int)], Ty::Qubit));
```

### 所有権チェッカーの Qubit 特殊処理

- `Qubit` は `!Copy`（デフォルト Move）としてマークされ、暗黙的なコピーが禁止されます。
- 測定関数 `measure` のパラメータは `Qubit`（値渡し）であり、所有権を消費します。
- 複数 qubit ゲートが返すレコード型では、フィールドは 모두 `Qubit` であり、所有権規則を遵守する必要があります。

### DAG スケジューラーの量子ゲート最適化

- 量子ゲートノードは純粋関数（副作用なし）として扱われ、スケジューラは依存関係のないゲートを自由に再配置できます。
- スケジューラが「量子命令列」を出力する際、データ依存関係を保持し、並列ゲートをグループ化します（複数量子プロセッサ向け）。
- `--target-num-qubits` と `--target-topology` の設定をサポートし、後続のレイアウトとルーティングに使用します（将来の拡張）。

### QIR バックエンドの詳細マッピング

| YaoXiang 操作 | QIR 命令 |
|---------------|----------|
| `H(q)` | `call void @__quantum__qis__h__body(%Qubit* %q)` |
| `CNOT(q1, q2)` | `call void @__quantum__qis__cnot__body(%Qubit* %q1, %Qubit* %q2)` |
| `measure(q)` | `%result = call i1 @__quantum__qis__mz__body(%Qubit* %q)` |
| `qubit(0)` | `%q = call %Qubit* @__quantum__rt__qubit_allocate()` |

QIR バックエンドは LLVM の `-O2` を活用してさらに最適化し、QIR Alliance と互換性のあるビットコードを出力します。

### QCIS バックエンドの詳細マッピング

| YaoXiang 操作 | QCIS 命令 |
|---------------|-----------|
| `H(q)` (q は物理ビット 2 に対応) | `H 2` |
| `CNOT(q1,q2)` (q1→ビット 0, q2→ビット 1) | `CNOT 0 1` |
| `measure(q)` (ビット 0) | `M 0` |
| `qubit(0)` 初期化 | 最初の使用命令に暗黙的に含まれるため、追加命令は不要 |

- 仮想 qubit（YaoXiang 変数）から物理ビットへのマッピングテーブルを維持する必要があります。
- トポロジ制約チェックをサポートします（将来実装）。

### ハイブリッド古典コード生成

- 古典部分（ループ、条件、整数演算など）は通常通りローカルコード（x86/ARM）を生成し、FFI または組み込み呼び出しで量子バックエンドと対話します。
- QIR バックエンドでは、古典部分を LLVM IR に低下させて QIR と混合コンパイルできます。

### 型システムへの影響

- 新しい `Qubit` と `Complex` 原始型が追加されます。
- `Qubit` は自動的に Move セマンティクスを持ち、コピーが禁止されます。
- 量子ゲート関数シグネチャは型システムに登録する必要があります。

### 後方互換性

- ✅ 完全な後方互換性
- 新しい組み込み型と関数が追加されても既存のコードに影響なし
- 量子特性はオプションであり、有効にしなくてもオーバーヘッドなし

## トレードオフ

### 优点

- **新構文なし**：開発者は少数の組み込み関数を学ぶだけで量子プログラムを記述できます。
- **型安全**：所有権システムが自動的に qubit コピーを防止し、一般的な量子プログラミングエラーを回避します。
- **自動並列化**：DAG スケジューラがゲートレベル並列化を無料で提供し、追加のコンパイラ最適化が不要です。
- **エコシステム互換性**：QIR バックエンドにより YaoXiang は複数の量子クラウドプラットフォームで実行可能；QCIS バックエンドにより自律制御可能です。
- **ハイブリッド能力**：古典量子融合が自然であり、複雑な量子アルゴリズム（Shor、Grover などの古典制御）の記述に適しています。

### 欠点

- **Qubit 数の静的な性質**：現在の設計では qubit 数はコンパイル時に既知と假定されており、動的割り当ては `List(Qubit)` を使用する必要がありますが、`List` のヒープ割り当ても дополнительные オーバーヘッドを引き起こす可能性があります（最適化で軽減可能）。
- **測定後の再利用**：空状態再利用により qubit の再初期化が可能ですが、物理量子ビットには緩和時間が存在するかもしれず、ランタイムシステムで処理する必要があります（現時点ではユーザーが責任を持ちます）。
- **動的トポロジマッピング**：物理トポロジがランタイム才知道の場合、コンパイル時チェックが效かないかもしれず、ランタイムチェックコードの生成が必要です（現在の版本は静的チェックのみサポート）。

## 代替方案

| 方案 | 選択しない理由 |
|------|--------------|
| `quantum` キーワードと `circuit` 型の導入 | 新しい構文の追加により学習コストが高く、YaoXiang の簡潔な設計原則に反する |
| 量子サポートをライブラリとしてのみ実装 | コンパイラの保証を活用して量子状態安全を確保できず、DAG スケジューラと深く統合できない |
| 量子ハードウェアの成熟を待ってからサポート | 量子プログラミング言語設計の重要なウィンドウ期を逃す |
| 既存の量子フレームワーク（Qiskitなど）を再利用 | 量子セマンティクスがライブラリで実装されるため、型システムと所有権システムの安全保障を得られない |
| 量子サブ言語を別途設計 | 言語の複雑性が増し、保守コストが高い |

## 実装戦略

### 段階的区分

| 段階 | 時間 | 内容 |
|------|------|------|
| Phase 1 | 1ヶ月 | 基本量子型と組み込み関数：コンパイラに `Qubit`、`Complex` 型を追加し、組み込み関数の型チェックを実装し、所有権チェッカーを拡張 |
| Phase 2 | 1ヶ月 | DAG スケジューラの量子ゲート認識：DAG 構築ロジックを修正し、量子ゲートを純粋関数としてマークし、並列ゲートグループ出力を実装 |
| Phase 3 | 2ヶ月 | QIR バックエンドプロトタイプ：DAG から QIR コードジェネレータを実装し、LLVM を統合し、QIR シミュレータに接続して検証 |
| Phase 4 | 2ヶ月 | QCIS バックエンドプロトタイプ：DAG から QCIS 命令への翻訳を実装し、仮想-物理ビットマッピングを設計し、国内量子プラットフォームに接続して検証 |
| Phase 5 | 2ヶ月 | ハイブリッド古典の強化：古典制御フローと量子ゲートの交差が正しくコード生成されることを保証し、`List(Qubit)` をサポートし、サンプルプログラムを追加 |
| Phase 6 | 2ヶ月 | 最適化とドキュメント：基本レイアウトとルーティングを実装し、ユーザーガイドと量子プログラミングチュートリアルを 작성し、プレビュー版をリリース |

### リスク

1. **量子ハードウェア可用性**：外部量子シミュレータと実際の QPU の可用性に依存します。
   **リスク軽減**：オープンソースシミュレータ（QIR runner、Qiskit Aer）への接続を優先し、実際の QPU は長期目標とします。

2. **バックエンド実装の複雑性**：QIR と QCIS 仕様は変更される可能性があります。
   **リスク軽減**：コード生成インターフェースを抽象化し、バックエンドの違いを隔离して、後続の適応を容易にします。

3. **性能の不確実性**：量子プログラムの性能特性は古典プログラムとは異なります。
   **リスク軽減**：性能プロファイリングツールを提供し、ユーザーがゲートレベル並列化の效果を理解できるようにします。

## 開放問題

- [x] **トポロジ制約**：`Qubit(Topology, N)` ジェネリクス定数パラメータによりコンパイル時チェックがすでに実装されています。
- [ ] **動的量子レジスタ**：`List(Qubit)` は QCIS バックエンドでどのようにマッピングしますか？対応する数の物理ビットを生成できますが、ランタイム割り当てメカニズムが必要です。
- [ ] **エラー緩和**：組み込みのエラー緩和（動的デカップリングなど）构造を提供するかどうか？最初はライブラリとして実装できます。
- [ ] **既存の量子 SDK との相互運用**：QASM または QIR モジュールをインポートできますか？将来的に FFI を検討できます。
- [ ] **自動レイアウトとルーティング**：仮想 qubit 数が物理 qubit 数を超過する場合、どのように自動マッピングしますか？
