# RFC-025: 拡張可能プリミティブ型機構

## 概要

本文書は YaoXiang コンパイラの**拡張可能プリミティブ型機構**（`Primitive::Extension`）を定義する。外部コードがカスタムプリミティブ型をコンパイラに登録できるようにし、コンパイラがハードコーディングなしでドメイン固有の型（量子ビット、GPU バッファ、SIMD ベクトル、ハードウェアレジスタなど）をサポートできるようにする。

## 動機

### なぜこの機構が必要なのか？

現在のコンパイラはすべてのプリミティブ型をハードコーディングしている：`Int`、`Float`、`String`、`Bool`、`Unit`。新しいプリミティブ型を追加するたびに、コンパイラのソースコードの複数の箇所を修正する必要がある——型検査器、コード生成器、所有権解析器、最適化器。

これは開放閉鎖原則に違反している：拡張には開かれ、修正には閉じられているべきである。

### 設計境界

```
ハードコーディングされたコア型（言語の基盤）：Int, Float, String, Bool, Unit
動的に拡張される型（ドメインプラグイン）：     Primitive::Extension を通じて登録
```

コア型がハードコーディングされているのは、コンパイラがそれらのセマンティクス（算術演算、条件分岐、ハッシュ、比較）に深く依存しているためである。拡張型はコンパイラにとって**不透明値**である——コンパイラはそれらのサイズ、アラインメント、所有権属性を知るだけで、内部セマンティクスは知らない。

これは「すべての型を動的ロードに統一する」ものではない。コア型と拡張型は異なる事柄である。

## 提案

### コア設計

#### 1. Extension 型属性

各拡張プリミティブ型を登録する際、以下の属性を宣言する必要がある：

```rust
pub struct PrimitiveExtension {
    /// 型名（例："Qubit", "Buffer", "Vec128"）
    pub name: String,

    /// バイトサイズ（固定サイズ型）
    /// None はサイズがコンパイル時に不明（実行時に決定される必要あり）を示す
    pub size: Option<usize>,

    /// アラインメント要件
    pub align: Option<usize>,

    /// 暗黙のコピーを許可するかどうか
    /// false = Move セマンティクス（代入は移動、例：Qubit）
    /// true = Copy セマンティクス（代入はコピー、例：Vec128）
    pub is_copy: bool,

    /// 空値（zero-sized type）を許可するかどうか
    pub allow_zst: bool,
}
```

#### 2. 登録インターフェース

```rust
// コンパイラ内部 API
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".to_string(),
    size: Some(0),           // 論理サイズ 0、物理状態は量子プロセッサ上
    align: Some(1),
    is_copy: false,          // Move セマンティクス、no-cloning に準拠
    allow_zst: true,
});
```

登録後、`Qubit` は型システムにおいて合法的なプリミティブ型となり、変数宣言、関数引数、構造体フィールドに使用できる。

#### 3. 型検査器の動作

拡張プリミティブ型は型検査において以下の規則に従う：

| シナリオ | 動作 |
|------|------|
| 変数宣言 `q: Qubit = ...` | ✅ 合法 |
| 関数引数 `fn(q: Qubit)` | ✅ 合法 |
| 構造体フィールド `{ q: Qubit }` | ✅ 合法 |
| `is_copy == false` の場合の代入 | Move セマンティクス、元の変数は無効化 |
| `is_copy == true` の場合の代入 | Copy セマンティクス、元の変数は保持 |
| 暗黙コピー（関数の複数箇所での使用） | `is_copy` に依存 |
| 比較 `==`、`!=` | ❌ コンパイルエラー（組み込み比較なし） |
| 算術 `+`、`-` | ❌ コンパイルエラー（組み込み演算なし） |
| ジェネリクス制約 `T: Copy` | `is_copy == true` の場合のみ満たされる |

#### 4. コード生成器の動作

拡張プリミティブ型はコード生成において**不透明値**として扱われる：

- LLVM IR：対応するサイズの `{size} x i8` または構造体として生成
- 特殊命令を生成しない——セマンティクスはバックエンドまたはライブラリが担当
- バックエンドが特殊処理を必要とする場合（例：QIR 量子ゲート）は、バックエンド登録機構を通じて実装する（本 RFC の範囲外）

### 例

#### Move セマンティクス型の登録

```rust
// 量子ビット：コピー不可、サイズ 0（物理状態は QPU 上）
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".into(),
    size: Some(0),
    align: Some(1),
    is_copy: false,
    allow_zst: true,
});
```

```yaoxiang
# ユーザーコード
q: Qubit = qubit(0)
q2 = q          # ❌ コンパイルエラー：Qubit は Move 型、q は無効化済み
q = H(q)        # ✅ q を消費し、新しい q を返す
```

#### Copy セマンティクス型の登録

```rust
// SIMD ベクトル：コピー可能、固定サイズ
compiler.register_primitive(PrimitiveExtension {
    name: "Vec128".into(),
    size: Some(16),
    align: Some(16),
    is_copy: true,
    allow_zst: false,
});
```

```yaoxiang
# ユーザーコード
a: Vec128 = load_vec128(data)
b = a           # ✅ Copy セマンティクス、a は依然有効
c = add_vec128(a, b)  # ✅ a と b の両方が使用可能
```

## 詳細設計

### コンパイラの改修

| コンポーネント | 改修 |
|------|------|
| 型システム | `Ty::Extension` バリアントを追加し、`PrimitiveExtension` メタデータを保存 |
| 型検査器 | 拡張型は組み込み演算の解決に参加せず、明示的に実装しない限り組み込み trait 制約を満たさない |
| 所有権解析器 | `is_copy` に応じて Move または Copy セマンティクスを決定 |
| コード生成器 | `size`/`align` に基づいて不透明値を生成、特殊命令なし |
| エラーメッセージ | 拡張型のエラーメッセージは登録時の `name` を参照 |

### FFI との関係

`Primitive::Extension` は RFC-021（FFI）と直交する：

| | Primitive::Extension | FFI |
|---|---|---|
| 役割 | 新しい**型**を登録 | 外部**関数**を呼び出す |
| レイヤー | 型システム | 実行時 |
| 例 | `Qubit` は型 | `native("sin")` は関数呼び出し |

ドメインによっては両方を必要とすることがある：`Qubit` 型は Extension を通じて登録し、量子ゲート関数は FFI を通じて登録する。

### 後方互換性

- ✅ 完全な後方互換性
- 既存型のセマンティクスは変更しない
- 拡張型は新機能であり、既存コードに影響しない

## トレードオフ

### 利点

- ✅ コンパイラは新しいドメインごとにソースコードを修正する必要がない
- ✅ ドメイン専門家がコンパイラチームに依存せず自分で型を登録できる
- ✅ コア型はハードコーディングを維持し、基本型に対するコンパイラの深い最適化を犠牲にしない
- ✅ インターフェースがシンプル、1 つの構造体で全属性を定義

### 欠点

- ⚠️ 拡張型は組み込み演算をサポートしない——セマンティクス実装には追加の関数やバックエンド機構が必要
- ⚠️ デバッグ時、拡張型は不透明値として表示され、コア型ほど直感的ではない

## 代替案

| 案 | 選択しない理由 |
|------|--------------|
| すべての型を動的ロード | コア型（Int/Float/Bool）はコンパイラの深い最適化が必要、動的ロードはこれらの能力を喪失させる |
| ドメインごとにハードコーディング | ドメインを追加するたびにコンパイラを修正する必要があり、拡張不可能 |
| 純粋なライブラリ方式（型を登録しない） | 型システムレベルでのセマンティクス保証（例：no-cloning）ができず、実行時チェックのみ可能 |

### フェーズ 2：登録タイミング

- [ ] コンパイラ初期化時の一括登録をサポート（設定ファイルまたは builder API）

### 依存関係

- ハード依存なし。他の RFC とは独立して実装可能。

## 設計決定記録

| 決定 | 結論 | 理由 | 日付 |
|------|------|------|------|
| コア型は動的ロードしない | ハードコーディングを維持 | コンパイラはコア型のセマンティクスに深く依存しており、動的化の利益はゼロ | 2026-06-05 |
| 拡張型は不透明値 | セマンティクスを注入しない | セマンティクスはバックエンド/ライブラリが担当し、コンパイラは型安全性と所有権のみを保証 | 2026-06-05 |
| FFI と直交させる | 統合しない | 型の登録と関数の呼び出しは異なる抽象化レイヤー | 2026-06-05 |
