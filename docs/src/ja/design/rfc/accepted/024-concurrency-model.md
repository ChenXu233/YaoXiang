# RFC-024：spawn ブロックに基づく並行モデル

> **状態**: 承認済み
> **著者**: 晨煦
> **作成日**: 2026-06-05
> **最終更新**: 2026-06-05

> **参照**:
> - [並行モデル仕様](/reference/language-spec/concurrency.md)
> - [RFC-008: Runtime 並行モデルとスケジューラの分離設計](./008-runtime-concurrency-model.md)
> - [RFC-009: 所有権モデル設計](./009-ownership-model.md)
> - [RFC-010: 統一型構文](./010-unified-type-syntax.md)

## 概要

このドキュメントでは、YaoXiang プログラミング言語の新しい並行モデルを定義する：`spawn {}` ブロックを唯一の並列プリミティブとし、依存関係駆動型実行、呼び出し元の同期ブロックを採用する。古い `@block`/`@eager`/`@auto` アノテーション、`Send`/`Sync` trait、全プログラム DAG 分析に基づく並行方案に取って代わる。

**核心設計——1つのプリミティブ、1つのルール**：

```
spawn { ... }        ← 唯一の並列プリミティブ
直接子式がタスクを作成  ← 唯一のルール
同期ブロックで結果を待つ ← 唯一の動作
```

**排除した複雑性**：
- ❌ `@block`/`@eager`/`@auto` アノテーションなし
- ❌ `Send`/`Sync` trait なし
- ❌ `Mutex`/`RwLock`/`Atomic` なし
- ❌ `future`/非同期ハンドラなし
- ❌ 全プログラム DAG 分析なし
- ❌ 関数着色なし（async/await）

> **ユーザーメンタルモデル**：書いた обычный コードは順次実行される。複数のことを同時に行いたい場合は、`spawn { ... }` ブロックに入れる。コールバックなし、`await` なし、おかしなアノテーションなし。

## 動機

### なぜこの設計が必要なのか？

現在の主流言語の並行モデルには明らかな欠陥がある：

| 言語 | 並行モデル | 問題点 |
|------|----------|------|
| Rust | async/await + tokio | 非同期传染、関数着色、学習曲線が険しい |
| Go | goroutine | 型安全性がなく、データ競合の検出が難しい |
| Python | asyncio | GIL の制約、関数着色 |
| JavaScript | Promise/async | コールバック地獄、関数着色 |

### 旧設計（RFC-001）の問題点

RFC-001 が提案した三層並行アーキテクチャ（L1/L2/L3）には以下の問題がある：

| 問題点 | 説明 |
|------|------|
| メンタルモデルが複雑 | L1/L2/L3 三層の抽象が追加の学習負担になる |
| アノテーション冗長 | `@block`/`@eager`/`@auto` アノテーションがコードをノイズだらけにする |
| 分析の複雑さ | 全プログラム DAG 分析のコンパイル時間オーバーヘッドが大きい |
| 型の制約が複雑 | `Send`/`Sync` trait が認知負担を増やす |
| 制御不能 | 自動並行動作は予測とデバッグが難しい |

### 設計目標

1. **シンプル**：並列プリミティブは1つだけ（`spawn`）、ルールは1つだけ（直接子式がタスクを作成）
2. **明示的**：ユーザーはどこが並行でどこが順序的かを明確に理解できる
3. **安全**：所有権ルールが自然に延伸し、追加の型制約が不要
4. **制御可能**：暗黙の並行はなく、予期せぬ並行動作がない
5. **同期**：呼び出し元が同期ブロックし、コールバックも `await` もない

## 提案

### 1. {} ブロックの本質：依存関係駆動の計算単位

YaoXiang では、`{}` は**依存関係駆動の計算単位**である。

| 属性 | 説明 |
|------|------|
| 依存関係駆動 | ブロックは実行時に内部のすべての変数が準備完了かどうかを確認し、準備完了なら即座に実行、そうでなければブロックして待機 |
| 実行タイミング | 依存関係によって決定され、「即座」か「遅延」かは無関係 |
| 返回値 | `return` を使用して明示的に値を返す；`return` がない場合はデフォルトで `Void` を返す |
| 構文統一 | 関数体内、変数初期化、`spawn` の後どれに出現しても意味が一貫している |
| スコープ隔离 | 変数は厳密に `{}` 内部に限定され、外側のスコープには穿透しない |

```yaoxiang
// 依存関係駆動の例
x = compute_x()        // x が準備完了
y = compute_y()        // y が準備完了
result = {
    // x と y に依存，两者準備完了後即座に実行
    return x + y
}
```

### 2. spawn ブロックの意味論

`spawn { ... }` は YaoXiang における**唯一の並列プリミティブ**である。

#### 2.1 核心ルール

- spawn ブロックの**直接子式**が並行タスクを作成する
- ネストされた `{}` 内の式は独立タスクとしてカウントされない
- spawn ブロックは標準返回規則に従う：`return` で明示的に値を返す必要があり、`return` がない場合は `Void` を返す
- 整个 spawn ブロックが同期ブロックし、すべてのタスク完了を待ってから戻る
- コールバック、`await`、アノテーションはない

```yaoxiang
// 2つのタスクが並行実行
(a, b) = spawn {
    t1 = fetch("url1")   // 直接子式 → 並行タスク 1
    t2 = fetch("url2")   // 直接子式 → 並行タスク 2
    return (t1, t2)      // 明示的にタプルを返す
}

// ネスト {} 内のものは直接子式ではない
result = spawn {
    x = {               // このブロック全体が一つの直接子式 → 一つのタスク
        inner_work()    // spawn の直接子式ではなく、独立タスクにはならない
    },
    process(x)          // 直接子式 → 並行タスク
    return process(x)
}
```

#### 2.2 スコープ隔离

spawn ブロックは独立したスコープを作成し、内部変数は外部に影響しない：

```yaoxiang
x = 10
result = spawn {
    x = 20              // これは spawn ブロック内のローカルな x
    compute(x)
}
// x は 여전히 10
```

#### 2.3 所有権ルール

変数が spawn ブロックに入ると、外部ではもう使用できない（Move セマンティクス）：

```yaoxiang
data = load_data()
result = spawn {
    process(data)       // data の所有権が spawn ブロックに移動する
}
// data はここでは使用不可（move 済み）
```

複数のタスク間で共有する必要がある場合は、`ref` を使用する：

```yaoxiang
data = load_data()
shared = ref data       // コンパイラが自動的に Rc または Arc を選択

result = spawn {
    process_a(shared),  // 共有参照
    process_b(shared)   // 共有参照
}
```

#### 2.4 エラー処理

spawn ブロック内のエラー伝播は以下の規則に従う：

1. すべてのタスク完了を待つ（いくつかのタスクが既に失敗しても）
2. 最初に遭遇したエラーを伝播する
3. `?` を使用してエラー伝播点を明示的にマークする

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // 失敗可能性あり
    fetch("url2")?      // 失敗可能性あり
}
// いずれかのタスクが失敗すると、整个 spawn ブロックが最初のエラーを伝播
```

#### 2.5 リソースタイプ

コンパイラはリソースタイプの使用を追跡し、並行安全性を確保する：

| リソースタイプ | 説明 | コンパイラの動作 |
|----------|------|-----------|
| `FilePath` | ファイルシステムパス | 同一パスへの操作は自動的に直列化 |
| `HttpUrl` | HTTP エンドポイント | 同一 URL への操作は自動的に直列化 |
| `DBUrl` | データベース接続 | 同一接続への操作は自動的に直列化 |
| `Console` | 標準出力 | すべての Console 操作は自動的に直列化 |

```yaoxiang
// 同一ファイルへの操作は自動的に直列化
(a, b) = spawn {
    read_file("data.txt"),      // 先に実行
    write_file("data.txt", x)   // 読み取り完了を待つ
}
```

#### 2.6 spawn for：データ並列ループ

```yaoxiang
// リスト内の各要素を並行処理
results = spawn for item in items {
    result = process(item)
}
```

#### 2.7 ネスト spawn

spawn ブロックはネスト可能で、内側の spawn は新しい並行ドメインを作成する：

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

### 3. 旧設計との決別

| 旧設計（RFC-001） | 新設計（RFC-024） |
|------------------|------------------|
| 全プログラム自動 DAG 分析 | spawn ブロック内のみ分析 |
| `@block`/`@eager`/`@auto` アノテーション | アノテーションなし、依存関係駆動 |
| `Send`/`Sync` trait | 不要、所有権 + ref が自動的に処理 |
| `future`/非同期ハンドラ | 同期ブロック、コールバックなし |
| `Mutex`/`RwLock`/`Atomic` | `ref` が自動的に Rc/Arc を選択 |
| L1/L2/L3 三層メンタモデル | 通常コードは順序、spawn ブロックは並行 |
| 関数着色（async/await） | 関数着色なし |

### 4. 返回規則

YaoXiang の返回規則は統一されており明確である：

| 書き方 | 返回値 | 説明 |
|------|--------|------|
| `= expr`（波括弧なし） | `expr` を直接返す | 式が値である |
| `= { ... }`（波括弧あり） | `return` が必要，否则返回 `Void` | ブロックは明示的な返回が必要 |

```yaoxiang
// 波括弧なし：直接返す
add: (a: Int, b: Int) -> Int = a + b

// 波括弧あり：`return` が必要
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// 波括弧ありだが `return` なし：Void を返す
log: (message: String) -> Void = {
    print(message)  // `return` なし、Void を返す
}
```

### 5. ユーザーメンタモデル

> **書いた通常コードは順次実行される。**
>
> **複数のことを同時に行いたい場合は、`spawn { ... }` ブロックに入れる。**
>
> ブロック内の各直接子式は即座に開始し（並行）、`return` で明示的に結果を返す。
> 整个ブロックはすべての処理完了を待ち、最終結果を提供する。
>
> **コールバックなし、`await` なし、おかしなアノテーションなし。**

```yaoxiang
// 通常コード：順次実行
a = compute_a()         // 先に実行
b = compute_b(a)        // a に依存、a 完了後実行
c = compute_c(b)        // b に依存、b 完了後実行

// 並行が必要な場合：spawn を使用
(x, y, z) = spawn {
    fetch("url1"),      // 並行
    fetch("url2"),      // 並行
    fetch("url3")       // 並行
}
// すべて完了後継続
process(x, y, z)
```

## トレードオフ

### メリット

1. **シンプル**：並列プリミティブは1つだけ（`spawn`）、ルールは1つだけ（直接子式がタスクを作成）
2. **明示的**：ユーザーはどこが並行でどこが順序的かを明確に知っている、暗黙の並行はない
3. **安全**：所有権ルールが自然に延伸し、`Send`/`Sync` などの追加型制約が不要
4. **制御可能**：自動並行動作がなく、予期せぬ並行問題を回避
5. **同期**：呼び出し元が同期ブロックし、コードは理解とデバッグが容易
6. **関数着色なし**：async/await の関数着色問題が存在しない
7. **コンパイル効率**：DAG 分析は spawn ブロック内に限られ、コンパイル時間が制御可能

### デメリット

1. **明示的な spawn が必要**：自動並行化はなく、ユーザーは手動で並行点をマークする必要がある
2. **spawn ブロック内の DAG 分析**：コンパイラは spawn ブロック内で依存関係分析を行う必要がある
3. **旧コードとの互換性なし**：古い RFC-001 モードのコードは移行が必要

## 代替案

| 方案 | なぜ選択しないのか |
|------|--------------|
| 全プログラム自動 DAG（RFC-001） | 複雑度が高く、コンパイル時間が長く、動作が制御不能 |
| async/await | 関数着色、学習曲線が険しい、コード可読性が悪い |
| goroutine | 型安全性がなく、データ競合の検出が難しい |
| Actor モデル | メッセージ渡しが複雑で、デバッグが困難 |
| CSP（Go channel） | 型安全性がなく、デッドロックの検出が難しい |

## 実装戦略

### コンパイル時分析

1. **DAG 構築**：spawn ブロック内の依存関係を分析
2. **トポロジカルソート**：spawn ブロック内の実行順序を決定
3. **並行識別**：spawn ブロック内の依存関係のないサブツリーを識別
4. **エスケープ分析**：`ref` → Rc か Arc か
5. **リソース競合検出**：リソースタイプの潜在的な競合を検出

### 実行時実行

[RFC-008](./008-runtime-concurrency-model.md) の Runtime アーキテクチャを参照：

- **Embedded Runtime**：spawn サポートなし、同期実行
- **Standard Runtime**：spawn ブロックをサポート、spawn ブロック内並行
- **Full Runtime**：Standard + WorkStealer 負荷分散

### 依存関係

- RFC-008（Runtime アーキテクチャ）→ 完成済み
- RFC-009（所有権モデル）→ 完成済み
- RFC-010（統一型構文）→ 完成済み
- RFC-011（泛型システム）→ 完成済み
