# RFC-024: spawn ブロックに基づく並行モデル

> **参考**:
> - [並行モデル仕様](/reference/language-spec/concurrency.md)
> - [RFC-008: Runtime 並行モデルとスケジューラの分離設計](./008-runtime-concurrency-model.md)
> - [RFC-009: 所有権モデル設計](./009-ownership-model.md)
> - [RFC-010: 統一型構文](./010-unified-type-syntax.md)

## 要約

本文書では、`spawn {}` ブロックを唯一の並行プリミティブとし、依存関係駆動型の実行と呼び出し元の同期ブロッキングを特徴とする、YaoXiang プログラミング言語の新しい並行モデルを定義します。旧来の `@block`/`@eager`/`@auto` アノテーション、`Send`/`Sync` trait、全プログラム DAG 分析に基づく並行方案に取って代わります。

**コア設計——1つのプリミティブ、1つのルール**：

```
spawn { ... }        ← 唯一の並行プリミティブ
直接子式がタスクを作成  ← 唯一のルール
同期ブロッキングで結果を待機  ← 唯一の動作
```

**排除した複雑さ**：
- ❌ `@block`/`@eager`/`@auto` アノテーションなし
- ❌ `Send`/`Sync` trait なし
- ❌ `Mutex`/`RwLock`/`Atomic` なし
- ❌ `future`/ノンブロッキングハンドラなし
- ❌ 全プログラム DAG 分析なし
- ❌ 関数着色（async/await）なし

> **ユーザーメンタルモデル**：あなたが書く通常のコードは順に実行されます。複数のことを同時に行いたい場合は、それらを `spawn { ... }` ブロックの中に入れます。コールバックはなく、`await`はなく、おかしなアノテーションもありません。

## 動機

### なぜこの設計が必要か？

現在の主流言語の並行モデルには明らかな欠陥があります：

| 言語 | 並行モデル | 問題点 |
|------|----------|------|
| Rust | async/await + tokio | 非同期传染、関数着色、学習曲線が険しい |
| Go | goroutine | 型安全性の欠如、データ競合の検出が困難 |
| Python | asyncio | GIL の制限、関数着色 |
| JavaScript | Promise/async | コールバック地獄、関数着色 |

### 旧設計（RFC-001）の問題点

RFC-001 で提案された3層並行アーキテクチャ（L1/L2/L3）には以下の問題があります：

| 問題点 | 説明 |
|------|------|
| メンタルモデルが複雑 | L1/L2/L3 の3層抽象が追加の学習負荷となる |
| アノテーションが冗長 | `@block`/`@eager`/`@auto` アノテーションがコードを煩雑にする |
| 分析の複雑さ | 全プログラム DAG 分析のコンパイル時間オーバーヘッドが大きい |
| 型制約が複雑 | `Send`/`Sync` trait が認知負荷を追加する |
| 制御不能 | 自動並行動作が予測・デバッグ困難 |

### 設計目標

1. **シンプル**：並行プリミティブは1つだけ（`spawn`）、ルールは1つだけ（直接子式がタスクを作成）
2. **明示的**：ユーザーはどこが並行でどこが順序実行かを明確に把握
3. **安全**：所有権ルールが自然に延伸、追加の型制約が不要
4. **制御可能**：暗黙の並行処理なし、予期せぬ並行動作なし
5. **同期**：呼び出し元が同期ブロッキング、コールバックや `await` なし

## 提案

### 1. {} ブロックの本質：依存関係駆動型の計算ユニット

YaoXiang では、`{}` は**依存関係駆動型の計算ユニット**です。

| 属性 | 説明 |
|------|------|
| 依存関係駆動 | ブロック実行時に内部のすべての変数が準備完了かをチェック、揃っていれば即時実行、否则阻塞待機 |
| 実行タイミング | 依存関係により決定、「即時」か「遅延」かは無関係 |
| 戻り値 | `return` で明示的に値を返す；`return` がない場合はデフォルトで `Void` を返す |
| 構文の統一 | 関数本体、変数初期化、の後ろのどこに出現しても意味は一貫 |
| スコープの分離 | 変数は厳密に `{}` 内部に限定され、外側のスコープには漏れない |

```yaoxiang
// 依存関係駆動の例
x = compute_x()        // x が準備完了
y = compute_y()        // y が準備完了
result = {
    // x と y に依存、两者準備完了後即時実行
    return x + y
}
```

### 2. spawn ブロックのセマンティクス

`spawn { ... }` は YaoXiang における**唯一の並行プリミティブ**です。

#### 2.1 コアルール

- spawn ブロックの**直接子式**が並行タスクを作成
- ネストした `{}` 内の式は独立したタスクとはみなされない
- spawn ブロックは標準の戻り値ルールに従う：`return` で明示的に値を返す必要があり、`return` がない場合は `Void` を返す
- spawn ブロック全体が同期ブロッキングで、すべてのタスク完了後に返回
- コールバック、`await`、アノテーションなし

```yaoxiang
// 2つのタスクが並行実行
(a, b) = spawn {
    t1 = fetch("url1")   // 直接子式 → 並行タスク 1
    t2 = fetch("url2")   // 直接子式 → 並行タスク 2
    return (t1, t2)      // 明示的にタプルを返す
}

// ネストした {} 内のものは直接子式ではない
result = spawn {
    x = {               // このブロック全体が1つの直接子式 → 1つのタスク
        inner_work()    // spawn の直接子式ではないため、独立したタスクにならない
    },
    process(x)          // 直接子式 → 並行タスク
    return process(x)
}

#### 2.2 スコープの分離

spawn ブロックは独立したスコープを作成し、内部変数は外部に影響しない：

```yaoxiang
x = 10
result = spawn {
    x = 20              // これは spawn ブロック内のローカルの x
    compute(x)
}
// x は依然として 10
```

#### 2.3 所有権ルール

変数が spawn ブロックに入ると、外部では使用できなくなる（Move セマンティクス）：

```yaoxiang
data = load_data()
result = spawn {
    process(data)       // data の所有権が spawn ブロックに移動
}
// data は此处では使用不可（move 済み）
```

複数のタスク間で共有する必要がある場合は、`ref` を使用：

```yaoxiang
data = load_data()
shared = ref data       // コンパイラが自動的に Rc または Arc を選択

result = spawn {
    process_a(shared),  // 共有参照
    process_b(shared)   // 共有参照
}
```

#### 2.4 エラー処理

spawn ブロック内のエラー伝播は次のルールに従います：

1. すべてのタスク完了を待機（一部のタスクが失敗しても）
2. 最初に遭遇したエラーを伝播
3. `?` でエラー伝播点を明示的にマーク

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // 失敗の可能性あり
    fetch("url2")?      // 失敗の可能性あり
}
// いずれかのタスクが失敗した場合、spawn ブロック全体が最初のエラーを伝播
```

#### 2.5 リソースタイプ

コンパイラはリソースタイプの使用を追跡し、並行安全性を確保：

| リソースタイプ | 説明 | コンパイラの動作 |
|----------|------|-----------|
| `FilePath` | ファイルシステムパス | 同一パスの操作は自動的に直列化 |
| `HttpUrl` | HTTP エンドポイント | 同一 URL の操作は自動的に直列化 |
| `DBUrl` | データベース接続 | 同一接続の操作は自動的に直列化 |
| `Console` | 標準出力 | すべての Console 操作は自動的に直列化 |

```yaoxiang
// 同一ファイルへの操作は自動的に直列化
(a, b) = spawn {
    read_file("data.txt"),      // 先に実行
    write_file("data.txt", x)   // 読み取り完了を待機
}
```

#### 2.6 spawn for：データ並行ループ

```yaoxiang
// リストの各要素を並行処理
results = spawn for item in items {
    result = process(item)
}
```

#### 2.7 ネストした spawn

spawn ブロックはネスト可能で、内側の spawn は新しい並行ドメインを作成：

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

### 3. 旧設計との決別

| 旧設計（RFC-001） | 新設計（RFC-024） |
|------------------|------------------|
| 全プログラム自動 DAG 分析 | spawn ブロック内のみ分析 |
| `@block`/`@eager`/`@auto` アノテーション | アノテーションなし、依存関係駆動 |
| `Send`/`Sync` trait | 不要、所有権 + ref が自動的に処理 |
| `future`/ノンブロッキングハンドラ | 同期ブロッキング、コールバックなし |
| `Mutex`/`RwLock`/`Atomic` | `ref` が自動的に Rc/Arc を選択 |
| L1/L2/L3 の3層メンタルモデル | 通常コードは順序実行、spawn ブロックは並行 |
| 関数着色（async/await） | 関数着色なし |

### 4. 戻り値ルール

YaoXiang の戻り値ルールは統一されており明確です：

| 書き方 | 戻り値 | 説明 |
|------|--------|------|
| `= expr`（波括弧なし） | `expr` を直接返す | 式がそのまま値 |
| `= { ... }`（波括弧あり） | `return` 必須，否则返回 `Void` | ブロックは明示的な戻り値が必要 |

```yaoxiang
// 波括弧なし：直接返す
add: (a: Int, b: Int) -> Int = a + b

// 波括弧あり：`return` が必要
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// 波括弧ありだが `return` なし：`Void` を返す
log: (message: String) -> Void = {
    print(message)  // `return` なし、`Void` を返す
}
```

### 5. ユーザーメンタルモデル

> **あなたが書く通常のコードは順に実行されます。**
>
> **複数のことを同時に行いたい場合は、それらを `spawn { ... }` ブロックの中に入れてください。**
>
> ブロック内の各直接子式は即座に開始され（並行）、`return` で結果を明示的に返します。
> ブロック全体がすべての作業の完了を待ち、最終結果を返します。
>
> **コールバックはなく、`await`はなく、おかしなアノテーションもありません。**

```yaoxiang
// 通常コード：順序実行
a = compute_a()         // 先に実行
b = compute_b(a)        // a に依存、a 完了後実行
c = compute_c(b)        // b に依存、b 完了後実行

// 並行が必要な場合：spawn を使用
(x, y, z) = spawn {
    fetch("url1"),      // 並行
    fetch("url2"),      // 並行
    fetch("url3")       // 並行
}
// すべて完了後继续
process(x, y, z)
```

## トレードオフ

### 优点

1. **シンプル**：並行プリミティブは1つだけ（`spawn`）、ルールは1つだけ（直接子式がタスクを作成）
2. **明示的**：ユーザーはどこが並行でどこが順序実行かを明確に把握、暗黙の並行処理なし
3. **安全**：所有権ルールが自然に延伸、`Send`/`Sync` などの追加型制約が不要
4. **制御可能**：自動並行動作なし、予期せぬ並行問題の回避
5. **同期**：呼び出し元が同期ブロッキング、コードの理解とデバッグが容易
6. **関数着色なし**：async/await の関数着色問題が存在しない
7. **コンパイル効率**：DAG 分析は spawn ブロック内に限定、コンパイル時間が制御可能

### 缺点

1. **明示的な spawn が必要**：自動並行化不行、ユーザーが手動で並行ポイントをマークする必要がある
2. **spawn ブロック内の DAG 分析が必要**：コンパイラが spawn ブロック内で依存関係分析を行う必要がある
3. **旧コードとの非互換**：旧 RFC-001 モードのコードは移行が必要

---

## 代替方案

| 方案 | 選択しない理由 |
|------|--------------|
| 全プログラム自動 DAG（RFC-001） | 複雑度高、コンパイル時間大、動作が制御不能 |
| async/await | 関数着色、学習曲線が険しい、コード可読性が悪い |
| goroutine | 型安全性の欠如、データ競合の検出が困難 |
| Actor モデル | メッセージ渡しが複雑、デバッグが困難 |
| CSP（Go channel） | 型安全性の欠如、デッドロックの検出が困難 |

## 実装策略

### コンパイル時分析

1. **DAG 構築**：spawn ブロック内の依存関係を分析
2. **トポロジカルソート**：spawn ブロック内の実行順序を決定
3. **並行識別**：spawn ブロック内で依存関係のないサブツリーを識別
4. **エスケープ分析**：`ref` → Rc か Arc か
5. **リソース衝突検出**：リソースタイプの潜在的な衝突を検出

### ランタイム実行

[RFC-008](./008-runtime-concurrency-model.md) の Runtime アーキテクチャを参照：

- **Embedded Runtime**：spawn サポートなし、即時実行
- **Standard Runtime**：spawn ブロックをサポート、spawn ブロック内並行
- **Full Runtime**：Standard + WorkStealer 負荷分散

### 依存関係

- RFC-008（Runtime アーキテクチャ）→ 完成済み
- RFC-009（所有権モデル）→ 完成済み
- RFC-010（統一型構文）→ 完成済み
- RFC-011（ジェネリクスシステム）→ 完成済み
