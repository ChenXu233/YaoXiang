# RFC-024：spawn ブロックに基づく並行モデル

> **ステータス**: 承認済み
> **著者**: 晨煦
> **作成日**: 2026-06-05
> **最終更新**: 2026-06-05

> **参考**:
> - [並行モデル仕様](/reference/language-spec/concurrency.md)
> - [RFC-008: Runtime 並行モデルとスケジューラの分離設計](./008-runtime-concurrency-model.md)
> - [RFC-009: 所有権モデル設計](./009-ownership-model.md)
> - [RFC-010: 統一型構文](./010-unified-type-syntax.md)

## 概要

本文書は YaoXiang プログラミング言語の新しい並行モデルを定義する：`spawn {}` ブロックを唯一の並行プリミティブとし、依存駆動型実行、呼び出し元の同期ブロッキングを採用する。旧来の `@block`/`@eager`/`@auto` アノテーション、`Send`/`Sync` 特質、全プログラム DAG 分析に基づく並行方案に取って代わる。

**核心設計——1つのプリミティブ、1つのルール**：

```
spawn { ... }        ← 唯一の並行プリミティブ
直接子の代入でタスク生成  ← 唯一のルール
同期ブロッキングで結果待機  ← 唯一の動作
```

**排除した複雑性**：
- ❌ `@block`/`@eager`/`@auto` アノテーションなし
- ❌ `Send`/`Sync` 特質なし
- ❌ `Mutex`/`RwLock`/`Atomic` なし
- ❌ `future`/ノンブロッキングハンドラなし
- ❌ 全プログラム DAG 分析なし
- ❌ 関数着色（async/await）なし

> **ユーザーメンタルモデル**：あなたが書く обычный код は順序実行される。複数のことを同時に行いたいときは、`spawn { ... }` ブロックの中に入れる。コールバックなし、`await` なし、おかしなアノテーションなし。

## 動機

### なぜこの設計が必要なのか？

現在の主流言語の並行モデルには明らかな欠陥がある：

| 言語 | 並行モデル | 問題点 |
|------|----------|--------|
| Rust | async/await + tokio | 非同期传染、関数着色、学習曲線が険しい |
| Go | goroutine | 型安全性の欠如、データ競合の検出が困難 |
| Python | asyncio | GIL 制約、関数着色 |
| JavaScript | Promise/async | コールバック地獄、関数着色 |

### 旧設計（RFC-001）の問題点

RFC-001 で提案された3層並行アーキテクチャ（L1/L2/L3）には以下の問題がある：

| 問題点 | 説明 |
|------|------|
| メンタルモデルが複雑 | L1/L2/L3 の3層抽象が学習負担を増す |
| アノテーションが冗長 | `@block`/`@eager`/`@auto` アノテーションがコードを嘈杂させる |
| 分析の複雑度が高い | 全プログラム DAG 分析のコンパイル時間オーバーヘッドが大きい |
| 型制約が複雑 | `Send`/`Sync` 特質が認知負荷を増す |
| 制御不能 | 自動並行動作が予測・デバッグ困難 |

### 設計目標

1. **シンプル**：並行プリミティブは1つだけ（`spawn`）、ルールは1つだけ（直接子の代入でタスク生成）
2. **明示的**：ユーザーがどこが並行で、どこが順序実行か明確に把握できる
3. **安全**：所有権ルールが自然に拡張され、追加の型制約が不要
4. **制御可能**：暗黙の並行処理はなく、予期せぬ並行動作がない
5. **同期**：呼び出し元が同期ブロッキングし、コールバックや `await` がない

## 提案

### 1. {} ブロックの本質：依存駆動型計算ユニット

YaoXiang では、`{}` は**依存駆動型計算ユニット**である。

| 属性 | 説明 |
|------|------|
| 依存駆動 | ブロック実行時に内部の全変数が準備完了かチェックし、準備完了なら即時実行，否则阻塞待機 |
| 実行タイミング | 依存関係で決まる，「即時」か「遅延」かは無関係 |
| 戻り値 | `return` で明示的に値を返す；`return` なしの場合はデフォルトで `Void` を返す |
| 構文の統一 | 関数体内、変数初期化、`spawn` の後、どこに現れてもセマンティクスは一貫している |
| スコープ隔离 | 変数は厳密に `{}` 内部に限定され、外側のスコープには穿透しない |

```yaoxiang
// 依存駆動の例
x = compute_x()        // x が準備完了
y = compute_y()        // y が準備完了
result = {
    // x と y に依存、どちらも準備完了後即時実行
    return x + y
}
```

### 2. spawn ブロックのセマンティクス

`spawn { ... }` は YaoXiang における**唯一の並行プリミティブ**である。

#### 2.1 核心ルール

- spawn ブロックの**直接子の代入**が並行タスクを生成する
- ネストした `{}` 内の代入は独立したタスクとはみなされない
- spawn ブロック全体が同期ブロッキングし、全タスク完了後に結果を返す
- コールバック、`await`、アノテーションはない

```yaoxiang
// 2つのタスクが並行実行
(a, b) = spawn {
    fetch("url1"),      // タスク 1
    fetch("url2")       // タスク 2
}
// 両方の完了を待機してから継続
```

#### 2.2 スコープの隔離

spawn ブロックは独立したスコープを生成し、内部変数は外部に影響しない：

```yaoxiang
x = 10
result = spawn {
    x = 20              // これは spawn ブロック内のローカルの x
    compute(x)
}
// x は仍然として 10
```

#### 2.3 所有権ルール

変数が spawn ブロックに入ると、外部ではもう使用できない（Move セマンティクス）：

```yaoxiang
data = load_data()
result = spawn {
    process(data)       // data の所有権が spawn ブロックに移動する
}
// data はここからは使用不可（move 済み）
```

複数のタスク間で共有する必要がある場合は、`ref` を使用する：

```yaoxiang
data = load_data()
shared = ref data       // コンパイラが自動的に Rc または Arc を選択

result = spawn {
    process_a(shared),  // 共有参照
    process_b(shared)   // 共有参照
}
```

#### 2.4 エラー処理

spawn ブロック内のエラー伝播は以下のルールに従う：

1. 全タスクの完了を待機する（いくつか失敗しても）
2. 最初に遭遇したエラーを伝播する
3. エラー伝播点は `?` で明示的にマークする

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // 失敗の可能性あり
    fetch("url2")?      // 失敗の可能性あり
}
// いずれかのタスクが失敗すると、spawn ブロック全体が最初のエラーを伝播
```

#### 2.5 リソース型

コンパイラはリソース型の使用を追跡し、並行安全性を確保する：

| リソース型 | 説明 | コンパイラの動作 |
|----------|------|-----------|
| `FilePath` | ファイルシステムパス | 同一パスの操作は自動的に直列化 |
| `HttpUrl` | HTTP エンドポイント | 同一 URL の操作は自動的に直列化 |
| `DBUrl` | データベース接続 | 同一接続の操作は自動的に直列化 |
| `Console` | 標準出力 | 全 Console 操作は自動的に直列化 |

```yaoxiang
// 同一ファイルの操作は自動的に直列化される
(a, b) = spawn {
    read_file("data.txt"),      // 先に実行
    write_file("data.txt", x)   // 読み取りの完了を待機
}
```

#### 2.6 spawn for：データ並行ループ

```yaoxiang
// リストの各要素を並行処理
results = spawn for item in items {
    result = process(item)
}
```

#### 2.7 ネストした spawn

spawn ブロックはネスト可能で、内側の spawn は新しい並行ドメインを生成する：

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

### 3. 旧設計との決別

| 旧設計（RFC-001） | 新設計（RFC-024） |
|------------------|------------------|
| 全プログラム自動 DAG 分析 | spawn ブロック内のみ分析 |
| `@block`/`@eager`/`@auto` アノテーション | アノテーションなし、依存駆動 |
| `Send`/`Sync` 特質 | 不要、所有権 + ref で自動処理 |
| `future`/ノンブロッキングハンドラ | 同期ブロッキング、コールバックなし |
| `Mutex`/`RwLock`/`Atomic` | `ref` が自動的に Rc/Arc を選択 |
| L1/L2/L3 の3層メンタルモデル | 通常コードは順序実行、spawn ブロックは並行 |
| 関数着色（async/await） | 関数着色なし |

### 4. 戻り値ルール

YaoXiang の戻り値ルールは統一されており明確である：

| 書き方 | 戻り値 | 説明 |
|------|--------|------|
| `= expr`（波括弧なし） | `expr` を直接返す | 式が値 |
| `= { ... }`（波括弧あり） | `return` が必要，否则は `Void` を返す | ブロックは明示的 возвращение が必要 |

```yaoxiang
// 波括弧なし：直接返す
add: (a: Int, b: Int) -> Int = a + b

// 波括弧あり：return が必要
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// 波括弧ありだが return なし：Void を返す
log: (message: String) -> Void = {
    print(message)  // return なし、Void を返す
}
```

### 5. ユーザーメンタルモデル

> **あなたが書く обычный код は順序実行される。**
>
> **複数のことを同時に行いたいときは、`spawn { ... }` ブロックの中に入れる。**
>
> ブロック内の各直接代入は即座に開始し（並行）、你需要的结果会自动等待。
> ブロック全体が全処理完了を待機し、最終結果を返す。
>
> **コールバックなし、`await` なし、おかしなアノテーションなし。**

```yaoxiang
// 通常コード：順序実行
a = compute_a()         // 先に実行
b = compute_b(a)        // a に依存、a 完了後に実行
c = compute_c(b)        // b に依存、b 完了後に実行

// 並行が必要な場合：spawn を使用
(x, y, z) = spawn {
    fetch("url1"),      // 並行
    fetch("url2"),      // 並行
    fetch("url3")       // 並行
}
// 全完了を待機してから継続
process(x, y, z)
```

## トレードオフ

### 利点

1. **シンプル**：並行プリミティブは1つだけ（`spawn`）、ルールは1つだけ（直接子の代入でタスク生成）
2. **明示的**：どこが並行でどこが順序実行か明確で、暗黙の並行処理がない
3. **安全**：所有権ルールが自然に拡張され、`Send`/`Sync` などの追加型制約が不要
4. **制御可能**：自動並行動作がなく、予期せぬ並行問題の心配がない
5. **同期**：呼び出し元が同期ブロッキングし、コードの理解とデバッグが容易
6. **関数着色なし**：async/await の関数着色問題が存在しない
7. **コンパイル効率**：DAG 分析は spawn ブロック内に限定され、コンパイル時間が制御可能

### 欠点

1. **明示的な spawn が必要**：自動並行化できず、ユーザーが手動で並行点をマークする必要がある
2. **spawn ブロック内の DAG 分析が必要**：コンパイラは spawn ブロック内で依存分析を行う必要がある
3. **旧コードとの非互換性**：旧 RFC-001 モードのコードは移行が必要

---

## 代替案

| 案 | 採用しない理由 |
|------|--------------|
| 全プログラム自動 DAG（RFC-001） | 複雑度が高く、コンパイル時間が長く、動作が制御不能 |
| async/await | 関数着色、学習曲線が険しい、コード的可読性が悪い |
| goroutine | 型安全性がない、データ競合の検出が困難 |
| Actor モデル | メッセージ渡しが複雑、デバッグが困難 |
| CSP（Go channel） | 型安全性がない、デッドロックの検出が困難 |

## 実装戦略

### コンパイル時分析

1. **DAG 構築**：spawn ブロック内の依存関係を分析
2. **トポロジカルソート**：spawn ブロック内の実行順序を決定
3. **並行識別**：spawn ブロック内の依存がないサブツリーを識別
4. **エスケープ分析**：`ref` → Rc か Arc かを選択
5. **リソース競合検出**：リソース型の潜在的な競合を検出

### 実行時実行

[RFC-008](./008-runtime-concurrency-model.md) の Runtime アーキテクチャを参照：

- **Embedded Runtime**：spawn 不支持、即時実行
- **Standard Runtime**：spawn ブロック支持、spawn ブロック内並行
- **Full Runtime**：Standard + WorkStealer 負荷分散

### 依存関係

- RFC-008（Runtime アーキテクチャ）→ 完成済み
- RFC-009（所有権モデル）→ 完成済み
- RFC-010（統一型構文）→ 完成済み
- RFC-011（ジェネリクスシステム）→ 完成済み
