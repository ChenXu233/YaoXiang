# RFC-009a: 令牌ライフタイム解析——ホア証明パイプラインに基づく

> **親 RFC**: [RFC-009: 所有権モデル設計](../accepted/009-ownership-model.md)
>
> **依存**: [RFC-027: コンパイル時述語と統一静的検証](../accepted/027-compile-time-evaluation-types.md)
>
> **前提条件**: RFC-027 は既に承認されている。本 RFC の全機構（証明パイプライン、SMT フォールバック、パス条件収集）は RFC-027 の実装に依存する。
>
> **本 RFC は RFC-009 §"トークン競合検出：フロー敏感生存解析"（第 663-684 行）を修正し、代替する。**

## 摘要

RFC-009 第 684 行は、トークン競合検出は「……NLL を必要としない」と主張している。結論は正しいが、論拠は誤りである。

それは「トークンは値なので、線形追跡で十分だから」ではない。理由は：**トークンの生存はホア論理命題であり、専門的なフロー敏感解析ではないからである。**

`{conflicting_tokens 全部死亡} op {WriteToken 安全取得}` —— 同じ `{P} op {Q}` であり、型検査、述語検証と RFC-027 の証明パイプラインを共有する。新しい解析フレームワークは不要。一本のパイプライン、多種の命題。

---

## 動機

### RFC-009 の混同

RFC-009 は二つの問題を混同している：

1. **線形追跡**（Move 後使用不可）—— `{v 未移動} use(v) {型一致}`。型チェッカーが既に持つ。
2. **トークンライフタイム相互作用**（サブトークン生存 → ペアレントトークン一時停止 → サブトークン死亡 → ペアレントトークン再開）—— `{conflicting_tokens 全部死亡} write(data) {安全}`。**活性解析**が必要であり、線形追跡ではない。

### 現在のコードの实际情况

| コンポーネント | 状態 |
|------|------|
| `BorrowChecker` | IR を線形スキャン、露わな `Borrow`/`Release` 命令に受動的に応答 |
| `ControlFlowAnalyzer::analyze_instruction` | 空実装（`control_flow.rs:145-153`） |
| `liveness_analysis` | 存在するが Drop 挿入のみに使用、トークン競合には未接続 |
| Release 挿入 | Call 命令の後にハードコード——純粋な字句スコープ（`ir_gen.rs:2734-2736`） |

**ユーザーから見える結果**：

```yaoxiang
data = vec![1, 2, 3]
view = &data              # ReadToken を作成
x = view.total_count      # view の最後の使用
data.push(4)              # ❌ Release(view) がまだ実行されていない、ReadToken が「生きている」
```

### なぜ書き直しが必要か

前版（009a v1）は「DAG で NLL を代替」というナラティブを使い、不必要な新概念（保守的分岐ルール、循環の特別処理）を導入した。核心的矛盾が説明不足だった：**借用チェッカーは独立したシステムではない——それはホア命題の一種である。**

## コア設計

### すべてがホア

```
型検査：    { x: Int }        x + 1        { result: Int }
借用検査：  { view 已死亡 }   data.push(4)  { WriteToken 取得成功 }
述語検証：  { y > 0 }         divide(x, y)  { result: Int }
バックエッジ切断：{ i == n }  次ラウンドのループ { cond == false }
```

同じ形式 `{P} op {Q}`。コンパイラは各操作に対して前置命題 P を生成し、証明パイプラインに送って検証する。

**借用検査とユーザー述語は同じパイプラインを共有する。** 違いは命題を誰が生成するか、証明できない時の処理だけ。

### 二種類の述語、一本のパイプライン

| | ユーザー述語 | システム述語（借用） |
|---|---|---|
| 命題生成 | プログラマー（型注釈） | コンパイラ（ブランドツリー + 所有権ルール） |
| 証明提供 | コンパイラ + プログラマー | **コンパイラが全自动** |
| 証明不可時 | 証明関数を書くかリファクタリング | コードのリファクタリング（門は開いているが稀に使用） |
| 視認性 | シグネチャに表示 | 暗黙的、型シグネチャを汚さない |
| 学習コスト | 使いたい時に学ぶ | ゼロ |

**システム述語の証明はプログラマーに証明関数を開かない——コンパイラが全自动で証明する。** 証明できない時はユーザーがコードをリファクタリングする。

**三つの失敗モード、同一検証エンジン。** 型命題が証明できない → コンパイルエラー（回避不可）。借用命題が証明できない → コンパイルエラー、コードのリファクタリング（回避不可）。ユーザー述語が証明できない → コンパイルエラー、証明関数を書ける（回避可能）。失敗戦略は異なるが、検証エンジンは同一——SMT ソルバー + コンパイラカーネルの推論ルール。違いは「証明できない時に誰が証明を補充するか」のみ——コンパイラはプログラマーの代わりに借用証明を書くことはしない（借用命題の証明戦略は構造分析 + SMTであり、プログラマーの介入は不要）が、プログラマーが書いたユーザー述語の証明関数を受け入れる。これはパイプラインの不整合ではない——異なる命題クラスの責任境界が異なる。

Rust の `'a` との違い：`'a` は必修科目、証明関数は選択科目——绝大多数のユーザーは生涯選択科目の門にすら触れない。

### 借用命題：コンパイラが自動生成

ユーザーが `data.push(4)` と書く。コンパイラが自動的に命題を生成する：

```
WriteToken(data, node) が取得可能
  = forall t in conflicting_tokens(data): t は node 処で死亡済み
  = forall t in brand_tree.children(data): forward_reachable(node) ∩ consumers(t) == ∅
```

**三つのルール、ゼロ特殊ケース：**

1. **ブランドツリー**（RFC-009 §2.7）が「誰と誰が競合するか」に答える：前方一致マッチング、O(depth)、depth ≤ 3
2. **消費者リスト**（DAG 構築時に自動収集）が「トークンを最後に誰が消費したか」に答える
3. **前方到達可能性」が「消費者がまだ実行可能か」に答える：構造的切断 + 論理的切断

### 前方到達可能性：消費者から逆走

トークン T の各消費者 C について：

```
C から出発して、逆方向 BFS DAG。
辺が切断される条件：
  1. break である（構造的切断）
  2. パス条件 ⇒ !loop_cond が SMT で証明済み（論理的切断、RFC-027 パイプライン）

すべての未切断の辺を逆方向に伝播（バックエッジを含む、バックエッジは前のイテレーションに生存を伝播）。
到達可能な全ノードをマーク → unsafe。
```

クエリ：ノード W における書き込み操作 → W ∉ unsafe → 安全。

**「保守的分岐ルール」を発明する必要はない。「循環の保守的生存」を必要としない。一つの逆方向 BFS + 二つの切断ルール。**

### 証明戦略：高速パスを優先、SMT がフォールバック

```
トークンが必要な各書き込み操作
  │
  ├→ 高速パス：DAG 構造分析（95%+ のシナリオをカバー）
  │     │
  │     ├→ ブランドツリー前方一致マッチング → 競合トークンを特定（O(depth)）
  │     ├→ 逆方向 BFS、break がバックエッジを切断
  │     └→ バックエッジを横断可能でない → 直接 Proved / Disproved を判定
  │
  └→ 低速パス：SMT 論理的切断（高速パスが横断可能なバックエッジに遭遇した時のみ）
        │
        ├→ バックエッジ起点にパス条件がある → SMT で path_cond ⇒ !loop_cond を判定
        │     ├→ Proved → 論理的切断 → 高速パスに降りて継続
        │     └→ Disproved / Unproven → バックエッジ横断 → unsafe をマーク
        │
        └→ バックエッジ起点にパス条件がない → バックエッジを直接横断
```

**高速パスがカバー**：線形コード、if/else、loop + break、パス条件なしの while。
**低速パスがカバー**：while ループ体内、パス条件がループ終了を暗示する時。
**カバー不可**：実行時条件が静的証明不可 → バックエッジ横断 → unsafe → コンパイルエラー（ユーザーがリファクタリング）。

SMT は主力ではない——安全網である。RFC-027 のユーザー述語との違い：ユーザー述語は SMT を主力とする；借用システム述語は構造分析を主力とし、SMT は構造分析の届かない隅を補足するだけ。

## ユースケース分析

### 線形コード

```yaoxiang
data = vec![1, 2, 3]        # ノード 1
view = &data                # ノード 2：data を消費、ReadToken(#1) を生成
x = view.total_count        # ノード 3：view を消費（= #1 の最後の消費者）
data.push(4)                # ノード 4：WriteToken(data) が必要
```

逆方向 BFS を `view.total_count`（ノード 3）から開始 → ノード 3 は #1 の最後の消費者 → ノード 4 > ノード 3 → ノード 4 は unsafe にない → ✅

### if/else：特殊ルールなし

```yaoxiang
view = &data
if cond {
    use(view)               # then 分岐で view を消費
} else {
    do_something_else()     # view に触れない
}
data.push(4)                # view の最後の消費者が if 内にある → if の後に消費者なし → ✅
```

if/else は DAG の複合ノード一つ。内部の消費はここに帰属させる。分支状態をマージしない。保守的票決もしない。**後に消費者があるかどうか、整数の比較一つ。**

### if/else が戻り値でエスケープ

```yaoxiang
view = &data
result = if cond {
    view                     # view が result にエスケープ
} else {
    something_else
}
use(result)                  # view を間接的に消費
data.push(4)                 # view にはまだ消費者がいる（use(result)）
                             # → push が unsafe 内にある → ❌ 適切にエラー
```

view が戻り値でエスケープ → `use(result)` は view の消費者である → `push` から逆方向にたどると `use(result)` に到達可能 → unsafe。

### ループ：break がバックエッジを切断

```yaoxiang
view = &data
loop {
    use(view)                # consumer
    if is_last {
        data.push(4)         # 書き込み操作
        break                # ← 構造的切断
    }
}
```

逆方向 BFS を `use(view)` から開始 → バックエッジ → `data.push(4)` に向かって前進 → `break` に遭遇 → **切断** → `data.push(4)` は unsafe にない → ✅

break がない場合：

```yaoxiang
view = &data
loop {
    use(view)
    data.push(4)             # break による切断なし → バックエッジを横断可能 → 次のイテレーションの use(view) に到達可能
                             # → push が unsafe 内にある → ❌ 適切にエラー
}
```

### while：SMT 論理的切断

```yaoxiang
view = &data
mut i: UpTo(n) = 0
while i < n {
    use(view)                # consumer
    i += 1
    if i == n {
        data.push(4)         # パス条件：i == n
    }
}
```

逆方向 BFS を `use(view)` から開始 → バックエッジ → `data.push(4)` に向かって前進 → パス条件 `i == n` をチェック → SMT クエリ：`i == n ⇒ !(i < n)`？→ Proved → **論理的切断** → `data.push(4)` は unsafe にない → ✅

## 本質：ブランド ID は `'a` と同じ

「我々は `'a` を必要としない」とは言わない。「`#42` は `'42` である」と言う。

| Rust | YaoXiang | 同値性 |
|------|----------|--------|
| `'a` | `#42` | コンパイル時ライフタイム識別子 |
| `'a: 'b` outlives 制約 | `#42` は `#42.field_x` の前方一致 | 文字列前方一致比較 = 半順序関係 |
| NLL 生存伝播（CFG 不動点） | 逆方向 BFS（DAG） | どちらも到達可能性計算 |
| Polonius の事実 | SMT 論理的切断 | どちらもパス条件推論 |
| 制約系の不動点求解 | ブランドツリー前方一致マッチング + BFS | 異なるエンコーディング、同一の問題 |

**我々は新しい解析を発明していない。ただ `'a` を型シグネチャ層から証明層に降格しただけである。** ブランド ID がすることは `'a` と完全に同じ——借用のアイデンティティをマークし、派生関係を追跡し、競合を判定。唯一の違い：`'a` はユーザーの書く型シグネチャの中にある；`#42` はコンパイラの内部にある。

これは恥ずべきことではない。Curry-Howard は型は命題で、プログラムは証明だという。`'a` は命題の一部ではない——証明戦略の一部である。Rust は証明戦略を命題シグネチャに書き込んだ。我々はそれを本来あるべき場所に置いた。

### 言語設計制約が何を排除したか

| Complexity 源 | 回避できたか | 理由 |
|---|---|---|
| 変数のシャドウイング | ✅ | 言語で禁止——一つの名前はずっと同じものを指す |
| for の跨イテレーション借用 | ✅ | 各イテレーションが新しいバインディング——イテレーション間で自然に分離 |
| `'a` ライフタイム注釈 | ✅ | ブランドパス = `#42.field_x`、コンパイラが導出 |
| 名前付きライフタイム + 制約伝播 | ✅ | ブランドパスの前方一致比較が明示的制約集合を代替 |
| 借用グラフ制約求解（Polonius） | ✅ | ブランドツリー前方一致マッチング + DAG 消費者クエリ |
| ループ体内借用の生存伝播 | ❌ | Rust 同样に処理が必要——逆方向 BFS + 論理的切断を使用 |
| 条件分岐の保守性 | ❌ | Rust 同样——SMT が証明可能なものをカバー、残りはおそらく拒否 |

### なぜ DAG が可行か

YaoXiang の三つの言語設計制約が DAG 分析を可能にしている：

- **変数のシャドウイングなし**——一つの名前はずっと同じものを指す、リバインディングをまたいだ追跡が不要
- **for が各イテレーションで新しいバインディング**——イテレーション間で自然に分離、跨イテレーション借用が存在しない
- **構造化並行性**——タスク境界が明確、跨タスク生存伝播が不要

これらの制約は Rust の CFG 不動点反復の主要複雑性源を排除した。DAG が CFG より「高等」だからではない——より単純な言語設計がより単純な分析を可能にするからである。

## 詳細設計

### システム述語リスト

コンパイラが以下の命題を自動生成し、RFC-027 証明パイプラインに送る：

| システム述語 | トリガータイミング | 命題形式 |
|---|---|---|
| `borrow_conflict` | WriteToken(v) が必要 | `forall t ∈ conflicting(v): dead_at(t, node)` |
| `use_after_move` | 変数 v を使用 | `¬moved(v)` |
| `use_after_drop` | 変数 v を使用 | `¬dropped(v)` |
| `double_drop` | Drop(v) | `¬dropped(v)` |
| `mut_violation` | 不変変数 v に書き込み | `is_mut(v)` |

既存の `BorrowChecker`、`MoveChecker`、`DropChecker`、`MutChecker` は**命題生成器に変化する**——消えるのではなく、役割が変わる。它们は命題を生成し、パイプラインが命題を検証する。

### ブランドツリー

RFC-009 §2.7 のブランド機構をブランドツリーとして形式化する。

**トークンセマンティクス——凍結優先、非複製優先**：

`&T` と `&mut T` の本質的な違いは「複製できるかどうか」ではなく「同時書き込みを許すかどうか」である：

```
ReadToken(T)：読み取り専用権を付与、同時にソースデータ T を冻结——この期間中
              いかなる WriteToken(T) も取得不可。冻结は ReadToken の主セマンティクス。
              Dup（複製可能）は冻结の推論である：データが冻结済み（変異の可能性なし）
              なので、複数の読み取り専用ビューは自然に安全。

WriteToken(T)：排他的読み書き権を付与。書き込みが存在するため、
              他のトークン（読み取りまたは書き込み）は共存不可。
              Dup（線形型で未実装）は排他性の推論である。
```

**因果関係**：
```
ReadToken 存在 → ソースデータ冻结 → 複数の読み取り専用が安全 → Dup
                      ↓
              WriteToken が拒否される（borrow_conflict システム述語が強制）
```

ではなく：
```
ReadToken が Dup を持つ → 複数を保持可能 → 衝突を検査 ← 因果の逆転
```

```
BrandTree:
  nodes: Map<BrandId, BrandNode>

BrandNode:
  id: BrandId               # "#42"、"#42.field_x"
  kind: ReadToken | WriteToken
  source_var: Operand
  parent: Option<BrandId>   # 派生関係の親ノード
  children: Set<BrandId>    # 派生サブトークン
  consumers: Set<NodeId>    # このトークンを消費する DAG ノード
  ref_count: usize          # ReadToken 冻结期間中の安全コピー数
```

**競合判断**——冻结保証の実行メカニズム：

```rust
fn conflicts(a: &BrandId, b: &BrandId) -> bool {
    // 競合条件：同一ソース + 少なくとも一方が書き込み + ブランドパスが重複
    // これは意味する：
    //   1. ReadToken vs ReadToken → 競合なし（どちらも読み取り専用、変異なし）
    //   2. WriteToken vs ReadToken → 競合（書き込みが読み取りの冻结保証を破壊）
    //   3. WriteToken vs WriteToken → 競合（2つの書き込みは共存不可）
    a.source() == b.source()
        && (a.is_write() || b.is_write())
        && (a.is_prefix_of(b) || b.is_prefix_of(a))
}
```

O(depth) の文字列前方一致比較、depth ≤ 3。定数レベル。

### 逆方向 BFS 生存解析

```
アルゴリズム：check_borrow(token, node, dag, brand_tree)

入力：
  token: 検査が必要な WriteToken
  node:  書き込み操作が存在する DAG ノード

出力：Proved | Disproved

アルゴリズム：
  # 高速パス：逆方向 BFS
  unsafe = empty_set
  queue = brand_tree.consumers(token)

  while queue not empty:
    cur = queue.pop()
    unsafe.add(cur)

    for each pred in dag.predecessors(cur):
      # 構造的切断：break は横断不可
      if pred は break 辺:
        continue

      # バックエッジ → SMT フォールバックが必要かチェック
      if pred はバックエッジ:
        path_cond = pred 処のパス条件
        loop_cond = ループ条件
        # まず構造的に切断できるか確認（対応する break が既にパスを切断 → ここには来ない）
        # 次にパス条件を確認
        if path_cond が空でない:
          result = smt_fallback(path_cond, loop_cond)   # ← 低速パス
          if result == Proved:
            continue                    # 論理的切断
        # パス条件なし、または SMT で証明不可 → バックエッジを横断
        # fall through

      if pred ∉ unsafe:
        queue.push(pred)

  # 判定
  if node ∈ unsafe:
    return Disproved
  else:
    return Proved


smt_fallback(path_cond, loop_cond):
  # バックエッジ + パス条件がある時のみ呼び出し
  # RFC-027 証明パイプラインを使用、同一 SMT ソルバー、同一予算を共有
  return smt.prove(path_cond ⇒ !loop_cond)
  # Proved → 論理的切断
  # Disproved / Unproven → 切断なし、バックエッジを横断
```

O(N)、ただし SMT 呼び出し回数 = バックエッジ数 × パス条件付きバックエッジの割合。実際のコードでは SMT 呼び出しは非常に稀——while ループ体内、パス条件付き精緻化型変数の時のみトリガー。

### パス条件収集

RFC-027 §3.2-3.3 の既存機構が提供：

- **if guard**：`if y > 0` → true 分岐に `y > 0` をプッシュ
- **match パターン**：`if let Some(v) = opt` → 分岐olithに `opt == Some(v)` をプッシュ
- **代入**：`i += 1`、コンパイラが変数値域情報を維持
- **while cond**：ループ体内、`cond == true` をプッシュ

各 DAG ノードはパス条件の集合を携带する。逆方向 BFS でバックエッジに遭遇した時、バックエッジ起点の taken 条件を取り、SMT が次のイテレーション入口条件を排除するかを判定する。

### RFC-027 とのインターフェース

借用システム述語とユーザー述語は同じ証明パイプラインを共有する——違いは**主力証明戦略**にある：

| クエリタイプ | 命題ソース | 主力戦略 | フォールバック |
|---|---|---|---|
| 型等式 | 型チェッカー | 構造的等価 | — |
| ユーザー述語 | プログラマーの型注釈 | SMT | プログラマーの証明関数 |
| 借用競合 | コンパイラ自動生成 | **DAG 構造分析（高速パス）** | SMT 論理的切断 |

借用検査における SMT ソルバーの役割：**主力ではなく、安全網である。** while バックエッジが論理的切断を必要とする時のみ呼び出す。绝大多数の借用検査は高速パスで完了——O(N) 逆方向 BFS、SMT コストゼロ。

### 既存コードとの関係

| 既存コンポーネント | 処理 |
|----------|------|
| `BorrowChecker` | `BorrowPredicateEmitter` に変化——借用のホア命題を生成 |
| `MoveChecker` | `MovePredicateEmitter` に変化——`¬moved(v)` 命題を生成 |
| `DropChecker` | 同上——Drop 関連命題を生成 |
| `MutChecker` | 同上——`is_mut(v)` 命題を生成 |
| `ControlFlowAnalyzer` | 不要に——パイプラインが統一処理 |
| `liveness_analysis` | 保持——Drop 挿入にはまだ変数生存情報が必要 |
| `ir_gen.rs` Release ハードコード | 削除——Release 位置は DAG 消費者分析駆動 |

### NLL とイテレーション境界

**トークン死亡時刻 = 最後の使用点（NLL）、字句スコープ末尾ではない。**

これは消費者分析の自然な帰結：消費者の位置がトークンの最後の使用を定義する。`use(v)` は `v` の消費者である → `v` は `use(v)` の直後に死亡する。トークンライフタイムを早期終了させるための余分な `{}` や `drop()` は不要。

**ループイテレーション境界はトークンコピーの死亡線である。** 三つのルール：

```
ルール 1：ループ内で宣言された変数は各イテレーション終了時に自動的に死亡する。
          for の各イテレーションは新しいバインディング（言語設計で保証）、
          loop も同じ。

ルール 2：ブランドツリーの ref_count はループ頭でループ外で作成されたコピーのみをカウント。
          ループ内の Dup で生成された新しいコピーは、
          ref_count がイテレーション境界でゼロにクリアされる。

ルール 3：逆方向 BFS がバックエッジを横断する時、現在のイテレーションの生存情報を携带しない。
          ループ頭処の ref_count のみを携带する
          （すなわち：ループ外のコピー）。
```

インスタンス：

```yaoxiang
view = &data                          # ループ頭：ref_count = 1、consumer = use(view)
loop {
    v2: &Point = view                 # ループ内で Dup → ref_count = 2
    use(v2)                           # consumer：v2 の最後の使用 → v2 死亡 → ref_count = 1
    data.push(4)                      # ✅ 安全！v2 は死亡済み、view のみ（ref_count = 1、書き込み競合なし）
    # イテレーション境界：ルール 3——v2 を次のイテレーションに携带しない。
    # 次のイテレーション開始時、v2 は新しいバインディングで再作成される。
}
```

この設計では余分な「ループの保守的生存」ルールは不要。逆方向 BFS は消費者から開始し、消費者がループ体内にある → 生存は現在のイテレーション内に制限される → バックエッジは横断されない。RFC-009a §ユースケース分析のループ例と完全に一致。

### `?` エラー伝播とスコープ駆動 Release

`?` は早期リターン——スコープの正常終了出口に加えてもう一つの退出パスが追加される。トークンはこのパス上でも释放されなければならず、释放順序が間違えば UB。

**Release 命令はスコープ分析により生成され、Call の後にハードコードされない。**

コンパイラは各スコープの出口点リストを維持：
- `}`（スコープの正常終了）
- `?`（エラー伝播、早期 return）
- 明示的な `return`

各出口点で、そのスコープ内の全アクティブトークンの Release 命令を宣言逆順（LIFO）で挿入する。ブランドツリーの親子関係が派生存コピーのカスケード解放を自動的に処理する：

```yaoxiang
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)    # サブトークン &Float + ペアレントトークン &Point を返す
}

fn use_case(p: Point) -> Result<(), Error> = {
    (x_ref, p_ref) = p.get_x()?   # ? が伝播した場合：
    # ブランドツリーが x_ref が p_ref の派生임을 알고る（#42.field_x は #42 の前方一致）
    # 释放順序：x_ref（子）→ p_ref（親）→ LIFO が自動的に満足
    p.modify()                     # WriteToken——すべての ReadToken は既に解放済み
    Ok(())
}
```

実装場所：`ir_gen.rs` に保持、スコープ駆動に変更——新しい compiler pass を導入しない。

| 操作 | 計算量 | トリガー頻度 |
|------|--------|---------|
| ブランドツリー競合判断 | O(1) | トークンが必要な度 |
| DAG 消費者クエリ | O(1) | トークンが必要な度 |
| 逆方向 BFS（高速パス） | O(N) | トークンが必要な度、N = ブロック内ノード数 |
| SMT 論理的切断（fallback） | ~1ms | **極めて稀**——while + パス条件のみ |

**SMT fallback のトリガー条件は非常に苛刻**：同時に (1) while ループ (2) ループ体内に書き込み操作がある (3) 書き込み操作後にパス条件でループ終了を判定できる (4) コンパイラがその条件に依存してバックエッジを切断する必要がある、を満たす必要がある。実際のコードでは占比は 1% を大幅に下回る。その他の借用検査はすべて高速パスで完了。

RFC-027 ユーザー述語との関係：ユーザー述語は SMT を主力とし、借用システム述語は構造分析を主力とする。两者とも同一の SMT ソルバーと予算上限（RFC-027 §8）を共有するが、借用システム述語は SMT 予算をほぼ消費しない。

線形コード → バックエッジなし → レベル 1 O(N) 即座に結果。ループ + パス条件 → SMT 呼び出し、線形算術はミリ秒レベル（RFC-027 予算 100ms）。一度の BFS 結果は同一トークンの複数クエリで再利用可能。

### エラー情報設計

**基本原則：错误情報にはユーザーが書いたシンボルのみが表示される。**

Rust と借用関連の错误は二種類に分類される：

**変数レベルエラー**：E0597（十分に生存しない）、E0502（可変+不変の同時借用）、E0499（複数の可変借用）。Rust は既にベンチマーク——変数名+行号、`'a` は表示されない。YaoXiang の精度も同じ。情報はすべてブランドツリーにある：トークン作成点、消費者位置、リクエスト点。

**シグネチャレベルエラー**：E0623（lifetime mismatch）、E0106（missing lifetime specifier）、E0477（required lifetime 不満足）。`'a` を中心に展開。YaoXiang **この種のエラーは存在しない**——シグネチャに `'a` がない。「報告できない」ではなく、ユーザーが書いたことのないものは報告しない。

関数内競合の例：

```
エラー：`data` は冻结されており、可変権限を取得できない
 --> src/main.yx:5:9
2 |     view = &data
  |            ----- `data` は冻结中（読み取り専用トークンはここで作成）
4 |         use(view)
  |             ---- `view` はここでまだ使用中、冻结が解除されていない
5 |         data.push(4)
  |         ^^^^ ここで可変権限が必要
```

（Rust E0499 と同じ精度——変数名+行号、ブランド ID は表示されない。）

関数間エスケープの例：

```
エラー：`num`（4 行目）が保持するデータソースの一つは `default_str`（3 行目）であるが、
`default_str` は 6 行目で失效し、`num` は 5 行目でまだ使用中である。

検討：`default_str` の宣言を呼び出し元に移動するか、`ref default_str` を使用して共有所有にする。
```

（Rust E0597 と同じ精度。ブランドサマリーは `num` が二条のソースパスを持つことを知っている——コンパイラに既に存在し、エラーメッセージに使用できる。）

## RFC-009 本文の修正

RFC-009 §"トークン競合検出：フロー敏感生存解析"が更新された：

1. 「必要ないもの：……NLL」の削除——結論が誤りだからではなく、理由が誤りだから（「トークンは値なので、線形追跡で十分」）
2. レベル 1/レベル 2 移行方案は維持、完整方案は本 RFC を参照
3. 明確化：ブランド ID（`#42`）は `'a` である——情報は完全に同じ、エンコーディングが異なる。新しい解析を発明したのではなく、ライフタイムを型層から証明層に降格した

---

## トレードオフ

### メリット

1. **型シグネチャにライフタイムがない**：`#42` は `'42` である——同じ情報、品牌ツリーにエンコードされ、型シグネチャには表示されない。この点は反証不可：Rust で引数に 3 つの参照を持つジェネリック型がいくつの `'a` パラメータを必要とするか、YaoXiang ではいくつか。数えてみれば 3 vs 0 である。

2. **概念の統一**：借用検査とユーザー述語は同じ証明パイプラインを共有——`{P} op {Q}`、パイプラインが P を検証する。Curry-Howard の一貫性。

3. **新解析フレームワークゼロ**：新しい解析フレームワークを導入しない。ユーザーが「借用チェッカー」の存在を認識しない——型チェッカーの実装詳細をユーザーが認識しないのと同じ。

4. **エラー情報にはユーザーが書いたシンボルのみを含む**：丸ごと一つのエラーカテゴリが少ない（E0623、E0106、E0477——すべて `'a` を中心に展開）。変数レベルエラーは Rust と同じ精度。

5. **アルゴリズムは保守的ではない**：逆方向 BFS + break 切断 + SMT 論理的切断。「ループ内の保守的生存」を必要としない。「分支の保守的マージ」を必要としない。

### デメリット

1. **新しい発明ではない**：ブランド ID がすることは `'a` と完全に同じ——コンパイラの内部の制約求解複雑性は消えない，只是エンコーディングが「変数名+制約集合」から「ブランドパス+前方一致マッチング」に変わっただけ。エンドユーザーへの差异はシグネチャに `'a` を書かないことのみ。

2. **完全な新規実装が必要**：ブランドツリーはまだ概念としてのみコードに存在し、ゼロから実装が必要。BorrowChecker、ControlFlowAnalyzer が交換される。

3. **SMT 依存**：論理的切断は Z3 に依存（RFC-027 で既に導入、新しい依存なし）。しかし借用検査はほとんどトリガーしない——while + パス条件の時のみ呼び出す。

4. **極めて稀なパターンはリファクタリングが必要**：コンパイラが自動証明でカバーできない跨分支借用は、ユーザーがコードをリファクタリングする必要がある。Rust の `'a` との違い：Rust には `'a` がペンとしてある（注釈すれば通る）；YaoXiang のセーフティネット（証明関数）は MVP ではない。

## 代替方案

| 方案 | なぜ選択しないか |
|------|-----------|
| 完全な Rust NLL の実装 | YaoXiang の設計制約（シャドーイングなし、for の新しいバインディング）が NLL の主要複雑性源を排除しており、CFG 不動点は不要 |
| 現行維持（ハードコード Release） | 不十分——ユーザーが手動でトークンスコープを管理する必要がある |
| spawn ブロック内でのみ解析 | 不十分——非 spawn コード内のトークン使用が大多数 |
| GC で借用検査を代替 | 言語設計原則に反する——YaoXiang には GC がない |

## 開放問題

- [x] **ループ展開時のブランドツリーの `ref_count` 跨イテレーションセマンティクス**——NLL を採用：トークンは最後の使用後に死亡。ループ内のバインディングのコピーはイテレーション境界で死亡、逆方向 BFS は跨イテレーション生存情報を携带しない。詳細 §NLL とイテレーション境界を参照。
- [x] **`?` エラー伝播パス上のトークン释放順序**——Release はスコープ分析駆動（ir_gen.rs に保持）。各スコープ出口点（`}`、`?`、明示的な return）でアクティブトークンを LIFO で释放。ブランドツリーの親子関係がカスケード解放を自動的に処理。詳細 §`?` エラー伝播とスコープ駆動 Release を参照。
