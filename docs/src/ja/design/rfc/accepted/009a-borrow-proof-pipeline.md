# RFC-009a: トークン生存期間分析——ホーア証明パイプラインに基づく

> **親 RFC**: [RFC-009: 所有権モデル設計](../accepted/009-ownership-model.md)
>
> **依存**: [RFC-027: コンパイル時述語と統一静的検証](../accepted/027-compile-time-evaluation-types.md)
>
> **前提条件**: RFC-027 が承認済みであること。本 RFC のすべての機構（証明パイプライン、SMT フォールバック、経路条件収集）は RFC-027 の実装に依存する。
>
> **本 RFC は RFC-009 §「トークン衝突検出：フロー感度活性分析」（第 663-684 行）を修正し、置き換えるものである。**

## 要約

RFC-009 の 684 行目は、トークン衝突検出が「……NLL を必要としない」と主張している。結論は正しいが、論拠が誤っている。

「トークンは値であり、線形追跡で十分だから」ではない。**トークン活性はホーア論理の命題であり、専用のフロー感度分析ではない。** からだ。

`{競合トークン すべて死亡} op {WriteToken 安全取得}` —— 同じ `{P} op {Q}` が、型検査、述語検証と RFC-027 の証明パイプラインを共有する。新しい分析フレームワークはない。一本のパイプラインが、複数の命題を扱う。

---

## 動機

### RFC-009 の混同

RFC-009 は二つの問題を混同している：

1. **線形追跡**（Move 後は使用不可）—— `{v が move されていない} use(v) {型が一致}`。型検査器がすでに持つ。
2. **トークン生存期間の相互作用**（子トークン生存 → 親トークン停止 → 子トークン死亡 → 親トークン復活）—— `{競合トークン すべて死亡} write(data) {安全}`。**活性分析**が必要であり、線形追跡ではない。

### 現状コードの実情

| コンポーネント | 状態 |
|------|------|
| `BorrowChecker` | IR を線形スキャンし、明示的な `Borrow`/`Release` 命令に受動的に応答 |
| `ControlFlowAnalyzer::analyze_instruction` | 空実装（`control_flow.rs:145-153`） |
| `liveness_analysis` | 存在するが Drop 挿入にのみ使用され、トークン衝突には接続されていない |
| Release 挿入 | Call 命令の後にハードコード——純粋に字句スコープ（`ir_gen.rs:2734-2736`） |

**ユーザーに見える結果**：

```yaoxiang
data = vec![1, 2, 3]
view = &data              # ReadToken を作成
x = view.total_count      # view の最後の使用
data.push(4)              # ❌ Release(view) がまだ実行されていない、ReadToken が「生存中」
```

### なぜ書き直す必要があるか

前版（009a v1）は「DAG が NLL を代替する」物語を用い、不必要な新しい概念（保守的分岐規則、ループ特殊処理）を導入した。核心的矛盾を明確にしなかった：**借用検査は独立したシステムではなく、ホーア命題の一形態である。**

## 中核設計

### すべてはホーア

```
型検査：   { x: Int }        x + 1        { result: Int }
借用検査： { view 死亡済み } data.push(4)  { WriteToken 取得成功 }
述語検証： { y > 0 }         divide(x, y)  { result: Int }
後退辺切断：{ i == n }        次のループ     { cond == false }
```

同じ形式 `{P} op {Q}`。コンパイラは各操作に対して前置命題 P を生成し、証明パイプラインに送って検証する。

**借用検査とユーザー述語は同じパイプラインを共有する。** 違いは命題が誰によって生成されるか、証明できないときにどう処理されるかだけである。

### 二種類の述語、一本のパイプライン

| | ユーザー述語 | システム述語（借用） |
|---|---|---|
| 命題生成 | プログラマー（型アノテーション） | コンパイラ（ブランド木 + 所有権規則） |
| 証明提供 | コンパイラ + プログラマー | **コンパイラが完全自動** |
| 証明失敗時 | 証明関数を書くかリファクタリング | コードをリファクタリング（門は存在するがほぼ不要） |
| 可視性 | シグネチャに見える | 暗黙的、型シグネチャを汚染しない |
| 学習コスト | 使いたければ学ぶ | ゼロ |

**システム述語の証明はプログラマーに証明関数を開かない——コンパイラが完全自動で行う。** 証明できない場合、ユーザーはコードをリファクタリングする。

**三つの失敗モード、同一の検証エンジン。** 型命題が証明できない → コンパイルエラー（回避不可）。借用命題が証明できない → コンパイルエラー、コードをリファクタリング（回避不可）。ユーザー述語が証明できない → コンパイルエラー、証明関数を記述可能（回避可能）。失敗戦略は異なるが、検証エンジンは同一——SMT ソルバー + コンパイラカーネル推論規則。違いは「証明できないときに誰が証明を補うか」のみ——コンパイラはプログラマーに代わって借用証明を書くことを拒否する（借用命題の証明戦略は構造分析 + SMT であり、プログラマーの介入は不要）一方、ユーザー述語の証明関数は受け入れる。これはパイプラインの不整合ではなく、命題カテゴリごとに責任境界が異なるということである。

これは Rust の `'a` とは異なる：`'a` は必修科目、証明関数は選択科目——大多数のユーザーは一生選択科目の門に遭遇しない。

### 借用命題：コンパイラが自動生成

ユーザーが `data.push(4)` と書くと、コンパイラが自動的に命題を生成する：

```
WriteToken(data, node) が取得可能
  = ∀ t ∈ conflicting_tokens(data): t は node で死亡済み
  = ∀ t ∈ brand_tree.children(data): forward_reachable(node) ∩ consumers(t) == ∅
```

**三つの規則、特殊ケースなし：**

1. **ブランド木**（RFC-009 §2.7）が「誰と誰が衝突するか」に答える：プレフィックス一致、O(深さ)、深さ ≤ 3
2. **消費者リスト**（DAG 構築時に自動収集）が「トークンを最後に消費するのは誰か」に答える
3. **前方到達可能性**が「消費者がまだ実行され得るか」に答える：構造的切断 + 論理的切断

### 前方到達可能性：消費者から逆方向に歩く

トークン T の各消費者 C に対して：

```
C から出発し、DAG を逆方向に BFS。
辺は以下の場合に切断される：
  1. それが break である（構造的切断）
  2. 経路条件 ⇒ !loop_cond が SMT で真と証明される（論理的切断、RFC-027 パイプライン）

切断されていないすべての辺を逆方向に伝播する（後退辺を含む、後退辺は活性を前のイテレーションに伝播する）。
到達可能なすべてのノードをマーク → unsafe。
```

クエリ：書込操作がノード W にある → W ∉ unsafe → 安全。

**「保守的分岐規則」を発明する必要はない。「ループ内の保守的生存」を発明する必要もない。一つの逆方向 BFS + 二つの切断規則で十分。**

### 証明戦略：ファーストパス優先、SMT フォールバック

```
トークンを必要とする各書込操作
  │
  ├→ ファーストパス：DAG 構造分析（95% 以上のシナリオをカバー）
  │     │
  │     ├→ ブランド木プレフィックス一致 → 衝突トークンを特定（O(深さ)）
  │     ├→ 逆方向 BFS、break が後退辺を切断
  │     └→ 通過可能な後退辺なし → 直接 Proved / Disproved を判定
  │
  └→ スローパス：SMT 論理的切断（ファーストパスが通過可能な後退辺に遭遇したときのみ）
        │
        ├→ 後退辺の起点に経路条件あり → SMT が path_cond ⇒ !loop_cond を判定
        │     ├→ Proved → 論理的切断 → ファーストパスに戻り続行
        │     └→ Disproved / Unproven → 後退辺を通過 → unsafe マーク
        │
        └→ 後退辺の起点に経路条件なし → 後退辺を直接通過
```

**ファーストパスがカバー**：線形コード、if/else、loop + break、経路条件なしの while。
**スローパスがカバー**：while ループ体内、経路条件がループ終了を示唆する場合。
**カバーしない**：ランタイム条件が静的に証明できない → 後退辺を通過 → unsafe → コンパイルエラー（ユーザーがリファクタリング）。

SMT は主力ではなく、安全網である。RFC-027 のユーザー述語とは異なる：ユーザー述語は SMT を主力とする；借用システム述語は構造分析を主力とし、SMT は構造分析が届かない辺縁を補完するのみ。

## ユースケース分析

### 線形コード

```yaoxiang
data = vec![1, 2, 3]        # ノード 1
view = &data                # ノード 2：data を消費し、ReadToken(#1) を生成
x = view.total_count        # ノード 3：view を消費（= #1 の最後の消費者）
data.push(4)                # ノード 4：WriteToken(data) が必要
```

`view.total_count`（ノード 3）から逆方向 BFS → ノード 3 は #1 の最後の消費者 → ノード 4 > ノード 3 → ノード 4 は unsafe にない → ✅

### if/else：特殊規則なし

```yaoxiang
view = &data
if cond {
    use(view)               # then 分岐が view を消費
} else {
    do_something_else()     # view に触れない
}
data.push(4)                # view の最後の消費者が if 内にある → if 後は消費者なし → ✅
```

if/else は DAG の複合ノードである。内部消費はこのノードに帰属する。分岐状態をマージしない。保守的に判定しない。**後に消費者がいるかどうかを整数比較するだけ。**

### 戻り値のエスケープを伴う if/else

```yaoxiang
view = &data
result = if cond {
    view                     # view が result にエスケープ
} else {
    something_else
}
use(result)                  # 間接的に view を消費
data.push(4)                 # view にまだ消費者がいる（use(result)）
                             # → push は unsafe 内 → ❌ 正しくエラー
```

view 戻り値でエスケープ → `use(result)` が view の消費者 → `push` から逆方向に歩くと `use(result)` に到達可能 → unsafe。

### ループ：break による後退辺切断

```yaoxiang
view = &data
loop {
    use(view)                # consumer
    if is_last {
        data.push(4)         # 書込操作
        break                # ← 構造的切断
    }
}
```

`use(view)` から逆方向 BFS → 後退辺 → 前方に歩いて `data.push(4)` に到達 → `break` に遭遇 → **切断** → `data.push(4)` は unsafe にない → ✅

break がない場合：

```yaoxiang
view = &data
loop {
    use(view)
    data.push(4)             # break 切断なし → 後退辺通過可能 → 次のラウンドの use(view) に到達可能
                             # → push は unsafe 内 → ❌ 正しくエラー
}
```

### while：SMT 論理的切断

```yaoxiang
view = &data
mut i: UpTo(n) = 0
while i < n {
    use(view)                # consumer
    i += 1
    if i == n {
        data.push(4)         # 経路条件：i == n
    }
}
```

`use(view)` から逆方向 BFS → 後退辺 → `data.push(4)` に到達 → 経路条件 `i == n` をチェック → SMT クエリ：`i == n ⇒ !(i < n)`？→ Proved → **論理的切断** → `data.push(4)` は unsafe にない → ✅

## 本質：ブランド ID は `'a` そのもの

「我々は `'a` を必要としない」とは言わない。**「`#42` は `'42` そのものである」** と言う。

| Rust | YaoXiang | 等価性 |
|------|----------|--------|
| `'a` | `#42` | コンパイル時の生存期間識別子 |
| `'a: 'b` outlives 制約 | `#42` は `#42.field_x` のプレフィックス | 文字列プレフィックス比較 = 半順序関係 |
| NLL 活性伝播（CFG 不動点） | 逆方向 BFS（DAG） | どちらも到達可能性計算 |
| Polonius fact | SMT 論理的切断 | どちらも経路条件推論 |
| 制約システム不動点求解 | ブランド木プレフィックス一致 + BFS | 異なるエンコーディング、同じ問題 |

**我々は新しい分析を発明したのではない。`'a` を型シグネチャ層から証明層に降ろしただけである。** ブランド ID がすることは `'a` と完全に同じ——借用の身元をマークし、派生関係を追跡し、衝突を判定する。違いは一つだけ：`'a` はユーザーが書く型シグネチャにある；`#42` はコンパイラ内部にある。

これは恥ずべきことではない。Curry-Howard は型は命題、プログラムは証明と言う。`'a` は命題の一部ではなく、証明戦略の一部である。Rust は証明戦略を命題シグネチャに書き込んだ。我々はそれを本来あるべき場所に戻した。

### 言語設計制約が取り除いたもの

| 複雑さの源泉 | 回避されたか？ | 理由 |
|---|---|---|
| 変数シャドウィング | ✅ | 言語が禁止——一つの名前は同じものを指し続ける |
| for の反復をまたぐ借用 | ✅ | 各イテレーションは新しいバインディング——イテレーション間が自然に分離 |
| `'a` 生存期間アノテーション | ✅ | ブランドパス = `#42.field_x`、コンパイラが推論 |
| 名前付き生存期間 + 制約伝播 | ✅ | ブランドパスのプレフィックス比較が明示的制約集合を代替 |
| 借用グラフ制約求解（Polonius） | ✅ | ブランド木プレフィックス一致 + DAG 消費者クエリ |
| ループ体内の借用活性伝播 | ❌ | Rust と同じく処理が必要——逆方向 BFS + 論理的切断で対応 |
| 条件分岐の保守性 | ❌ | Rust と同じ——SMT が証明可能なものをカバーし、残りは保守的に拒否 |

### なぜ DAG が実行可能か

YaoXiang の三つの言語設計制約が DAG 分析を実用的にする：

- **変数シャドウィングなし**——一つの名前は同じものを指し続けるため、再バインディングをまたぐ追跡が不要
- **for の各イテレーションは新しいバインディング**——イテレーション間が自然に分離され、反復をまたぐ借用が存在しない
- **構造化並列**——タスク境界が明確で、タスクをまたぐ活性伝播が不要

これらの制約は Rust の CFG 不動点反復の主要な複雑さの源泉を除去する。DAG が CFG より「高度」なのではなく、**より単純な言語設計がより単純な分析を可能にする**のである。

## 詳細設計

### システム述語リスト

コンパイラは自動的に以下の命題を生成し、RFC-027 証明パイプラインに送る：

| システム述語 | 発動タイミング | 命題形式 |
|---|---|---|
| `borrow_conflict` | WriteToken(v) が必要 | `∀ t ∈ conflicting(v): dead_at(t, node)` |
| `use_after_move` | 変数 v を使用 | `¬moved(v)` |
| `use_after_drop` | 変数 v を使用 | `¬dropped(v)` |
| `double_drop` | Drop(v) | `¬dropped(v)` |
| `mut_violation` | 不変変数 v に書込 | `is_mut(v)` |

既存の `BorrowChecker`、`MoveChecker`、`DropChecker`、`MutChecker` は**命題生成器**になる——消えるのではなく、役割を変える。彼らは命題を生成し、パイプラインが命題を検証する。

### ブランド木

RFC-009 §2.7 のブランド機構をブランド木として形式化する。

**トークン语义——凍結優先、コピー優先ではない**：

`&T` と `&mut T` の本質的な違いは「コピー可能かどうか」ではなく、「同時書込を許すかどうか」である：

```
ReadToken(T)： 読み取り専用権限を付与し、ソースデータ T を凍結する——この期間中
              あらゆる WriteToken(T) は取得不可。凍結は ReadToken の第一の语义。
              Dup（コピー可能）は凍結の帰結である：データが凍結されているため
              （突然変異の可能性なし）、複数の読み取り専用ビューは本質的に安全。

WriteToken(T)：排他的な読み書き権限を付与する。書込が存在するため、他のトークン
              （読み取りまたは書込）は共存不可。Dup（線形型）を実装しないのは
              排他性の帰結である。
```

**因果関係**：
```
ReadToken 存在 → ソースデータ凍結 → 複数の読み取り専用が安全 → Dup
                      ↓
              WriteToken は拒否される（borrow_conflict システム述語が強制）
```

以下ではなく：
```
ReadToken は Dup を持つ → 複数持てる → ついでに衝突をチェック  ← 因果が逆
```

```
BrandTree:
  nodes: Map<BrandId, BrandNode>

BrandNode:
  id: BrandId               # "#42"、"#42.field_x"
  kind: ReadToken | WriteToken
  source_var: Operand
  parent: Option<BrandId>   # 派生関係の親ノード
  children: Set<BrandId>    # 派生子トークン
  consumers: Set<NodeId>    # このトークンを消費する DAG ノード
  ref_count: usize          # ReadToken 凍結期間中の安全コピー数
```

**衝突判定**——凍結保証の実行メカニズム：

```rust
fn conflicts(a: &BrandId, b: &BrandId) -> bool {
    // 衝突条件：同源 + 少なくとも一方が書込 + ブランドパスがオーバーラップ
    // これは以下を意味する：
    //   1. ReadToken vs ReadToken → 衝突なし（両方とも読み取り専用、突然変異なし）
    //   2. WriteToken vs ReadToken → 衝突（書込が読み取りの凍結保証を破壊する）
    //   3. WriteToken vs WriteToken → 衝突（二つの書込は共存不可）
    a.source() == b.source()
        && (a.is_write() || b.is_write())
        && (a.is_prefix_of(b) || b.is_prefix_of(b))
}
```

O(深さ) 文字列プレフィックス比較、深さ ≤ 3。定数オーダー。

### 逆方向 BFS 活性分析

```
アルゴリズム：check_borrow(token, node, dag, brand_tree)

入力：
  token: チェック対象の WriteToken
  node:  書込操作が位置する DAG ノード

出力：Proved | Disproved

アルゴリズム：
  # ファーストパス：逆方向 BFS
  unsafe = empty_set
  queue = brand_tree.consumers(token)

  while queue が空でない:
    cur = queue.pop()
    unsafe.add(cur)

    for each pred in dag.predecessors(cur):
      # 構造的切断：break は通過しない
      if pred が break 辺:
        continue

      # 後退辺 → SMT フォールバックが必要かチェック
      if pred が後退辺:
        path_cond = pred 位置の経路条件
        loop_cond = ループ条件
        # まず構造的に切断できるか確認（対応する break がすでにパスを切断 → ここに到達しない）
        # 次に経路条件を確認
        if path_cond が空でない:
          result = smt_fallback(path_cond, loop_cond)   # ← スローパス
          if result == Proved:
            continue                    # 論理的切断
        # 経路条件なし または SMT が証明失敗 → 後退辺を通過
        # fall through

      if pred ∉ unsafe:
        queue.push(pred)

  # 判定
  if node ∈ unsafe:
    return Disproved
  else:
    return Proved


smt_fallback(path_cond, loop_cond):
  # 後退辺 + 経路条件ありの場合にのみ呼び出される
  # RFC-027 証明パイプラインを使用、同一 SMT ソルバー、同一予算を共有
  return smt.prove(path_cond ⇒ !loop_cond)
  # Proved → 論理的切断
  # Disproved / Unproven → 切断しない、後退辺を通過
```

O(N)、ここで SMT 呼び出し回数 = 後退辺数 × 経路条件ありの後退辺の割合。実際のコードでは SMT 呼び出しは非常に稀である——`while` ループ体内、精化された型変数の経路条件がある場合にのみ発動する。

### 経路条件収集

RFC-027 §3.2-3.3 の既存機構が提供：

- **if guard**：`if y > 0` → true 分岐に `y > 0` をプッシュ
- **match パターン**：`if let Some(v) = opt` → 分岐内に `opt == Some(v)` をプッシュ
- **代入**：`i += 1`、コンパイラは変数の値域情報を維持
- **while cond**：ループ体内に `cond == true` をプッシュ

各 DAG ノードは経路条件集合を携带する。逆方向 BFS が後退辺に遭遇したとき、後退辺の起点の経路条件を取り、SMT が次のラウンドのループ入口条件を除外するかを判定する。

### RFC-027 とのインタフェース

借用システム述語とユーザー述語は同じ証明パイプラインを共有する——違いは**主力証明戦略**である：

| クエリタイプ | 命題ソース | 主力戦略 | フォールバック |
|---|---|---|---|
| 型等式 | 型検査器 | 構造等価 | — |
| ユーザー述語 | プログラマーの型アノテーション | SMT | プログラマーの証明関数 |
| 借用衝突 | コンパイラ自動生成 | **DAG 構造分析（ファーストパス）** | SMT 論理的切断 |

借用検査における SMT ソルバーの役割：**主力ではなく、安全網である。** while 後退辺の論理的切断が必要な場合にのみ呼び出される。大多数の借用検査はファーストパスで完了する——O(N) 逆方向 BFS、SMT オーバーヘッドゼロ。

### 既存コードとの関係

| 既存コンポーネント | 処理 |
|----------|------|
| `BorrowChecker` | `BorrowPredicateEmitter` になる——借用のホーア命題を生成 |
| `MoveChecker` | `MovePredicateEmitter` になる——`¬moved(v)` 命題を生成 |
| `DropChecker` | 同上——Drop 関連命題を生成 |
| `MutChecker` | 同上——`is_mut(v)` 命題を生成 |
| `ControlFlowAnalyzer` | 不要——パイプラインが統一処理 |
| `liveness_analysis` | 保持——Drop 挿入は依然として変数の活性情報を必要とする |
| `ir_gen.rs` Release ハードコード | 削除——Release 位置は DAG 消費者分析が駆動 |

### NLL とイテレーション境界

**トークン死亡時刻 = 最後の使用点（NLL）、字句スコープ末尾ではない。**

これは消費者分析の自然な帰結である：消費者の位置がトークンの最後の使用を定義する。`use(v)` は `v` の消費者 → `v` は `use(v)` の直後に死亡する。トークン生存期間を早期終了させるための追加の `{}` や `drop()` は不要である。

**ループイテレーション境界はトークンコピーの死亡線である。** 三つの規則：

```
規則 1：ループ内で宣言された変数は各イテレーション終了時に自動的に死亡する。
        for の各イテレーションは新しいバインディング（言語設計保証）であり、loop も同様。

規則 2：ブランド木の ref_count はループ頭でループ外で作成されたコピーのみをカウントする。
        ループ内の Dup が生成する新しいコピーは、ref_count がイテレーション境界でゼロになる。

規則 3：逆方向 BFS が後退辺を通過するとき、現在のイテレーションの活性情報を携带しない。
        ループ頭での ref_count（すなわち、ループ外のコピー）のみを携带する。
```

インスタンス：

```yaoxiang
view = &data                          # ループ頭：ref_count = 1、consumer = use(view)
loop {
    v2: &Point = view                 # ループ内 Dup → ref_count = 2
    use(v2)                           # consumer：v2 の最後の使用 → v2 死亡 → ref_count = 1
    data.push(4)                      # ✅ 安全！v2 は死亡済み、view のみ残る（ref_count = 1、書込衝突なし）
    # イテレーション境界：規則 3——v2 を次のラウンドに携带しない。次のイテレーション開始時に v2 は新しいバインディングとして再作成される。
}
```

この設計は追加の「ループ内保守的生存」規則を必要としない。逆方向 BFS は消費者から出発し、消費者がループ体内にある → 活性が現在のイテレーション内に制限される → 後退辺が通過しない。RFC-009a §ユースケース分析のループ例と完全に一致する。

### `?` エラー伝播とスコープ駆動 Release

`?` は早期リターンである——スコープの通常の出口に加えてもう一つの退出経路がある。トークンはこの経路上で解放されなければならず、解放順序が誤れば UB となる。

**Release 命令は Call の後にハードコードされるのではなく、スコープ分析によって生成される。**

コンパイラは各スコープの出口点リストを維持する：
- `}`（スコープの正常終了）
- `?`（エラー伝播、早期 return）
- 明示的な `return`

各出口点で、宣言の逆順（LIFO）で、そのスコープ内のすべてのアクティブトークンの Release 命令を挿入する。ブランド木の親子関係は派生素牌の段階的解放を自動的に処理する：

```yaoxiang
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)    # 子トークン &Float + 親トークン &Point を返す
}

fn use_case(p: Point) -> Result<(), Error> = {
    (x_ref, p_ref) = p.get_x()?   # ? がエラーを伝播する場合：
    # ブランド木は x_ref が p_ref の派生源であることを知っている（#42.field_x は #42 のプレフィックス）
    # 解放順序：x_ref（子）→ p_ref（親）→ LIFO が自動的に満たされる
    p.modify()                     # WriteToken——すべての ReadToken は解放済み
    Ok(())
}
```

実装場所：`ir_gen.rs` に保持、スコープ駆動に変更——新しい compiler pass を導入しない。

| 操作 | 複雑度 | 発動頻度 |
|------|--------|---------|
| ブランド木衝突判定 | O(1) | トークン必要時毎回 |
| DAG 消費者クエリ | O(1) | トークン必要時毎回 |
| 逆方向 BFS（ファーストパス） | O(N) | トークン必要時毎回、N = ブロック内ノード数 |
| SMT 論理的切断（フォールバック） | ~1ms | **非常に稀**——while + 経路条件の場合のみ |

**SMT フォールバックの発動条件は非常に苛刻である**：同時に (1) while ループ (2) ループ体内に書込操作がある (3) 書込操作の後にループ終了を示唆する経路条件がある (4) コンパイラが後退辺切断のためその条件に依存する必要がある、を満たす必要がある。実際のコードでの割合は 1% よりはるかに低い。残りの借用検査はすべてファーストパスで完了する。

RFC-027 ユーザー述語との関係：ユーザー述語は SMT を主力とし、借用システム述語は構造分析を主力とする。両者は同一 SMT ソルバーと予算上限（RFC-027 §8）を共有するが、借用システム述語は SMT 予算をほとんど消費しない。

線形コード → 後退辺なし → 階層 1 O(N) で即座に結果。ループ + 経路条件 → SMT 呼び出し、線形算術でミリ秒オーダー（RFC-027 予算 100ms）。一度の BFS 結果はキャッシュされ、同一トークンの複数クエリで再利用できる。

### エラーメッセージ設計

**核心原則：エラーメッセージにはユーザーが書いたシンボルだけを表示する。**

Rust の借用関連エラーは二種類に分かれる：

**変数レベルエラー**：E0597（生存期間が短すぎる）、E0502（可変 + 不変の同時借用）、E0499（複数回の可変借用）。Rust はすでに模範——変数名 + 行番号、`'a` は表示されない。YaoXiang は精度で同等。情報はすべてブランド木にある：トークン作成点、消費者位置、要求点。

**シグネチャレベルエラー**：E0623（lifetime mismatch）、E0106（missing lifetime specifier）、E0477（required lifetime を満たさない）。`'a` を中心に展開。YaoXiang には**この種のエラーは存在しない**——シグネチャに `'a` がない。「報告できない」のではなく、ユーザーが書かなかったものを報告する必要がないのである。

関数内衝突の例：

```
エラー：`data` は凍結されており、可変権限を取得できません
 --> src/main.yx:5:9
2 |     view = &data
  |            ----- `data` が凍結されています（読み取り専用トークンがここで作成）
4 |         use(view)
  |             ---- `view` はここでまだ使用中、凍結が解除されていません
5 |         data.push(4)
  |         ^^^^ ここで可変権限が必要です
```

（Rust E0499 と精度同等——変数名 + 行番号、ブランド ID は表示されない。）

関数間エスケープの例：

```
エラー：`num`（4 行目）が保持するデータのソースの一つは `default_str`（3 行目）ですが、
`default_str` は 6 行目で失效、`num` は 5 行目でまだ使用中です。

検討：`default_str` の宣言を呼び出し側に移動するか、`ref default_str` を使用して共有保持してください。
```

（Rust E0597 と精度同等。ブランドサマリは `num` が二つのソースパスを持つことを知っている——コンパイラ内に既存、エラー措辞に使用可能。）

## RFC-009 本文修正

RFC-009 §「トークン衝突検出：フロー感度活性分析」を以下に更新：

1. 「必要ないもの：……NLL」を削除——結論が誤りだからではなく、理由が誤りだから（「トークンは値であり、線形追跡で十分」）
2. 階層 1 / 階層 2 移行案を保持、完全な解決策は本 RFC を指す
3. 明確化：ブランド ID（`#42`）は `'a` そのものである——情報は完全に同じ、エンコーディングが異なる。新しい分析を発明したのではなく、生存期間を型層から証明層に降ろしたのである

---

## トレードオフ

### 利点

1. **型シグネチャに生存期間を含まない**：`#42` は `'42` である——同じ情報、ブランド木にエンコードされ、型シグネチャに露出しない。これは反証可能：Rust で 3 つの参照パラメータを持つ generic 型がいくつの `'a` パラメータを必要とするか、YaoXiang ではいくつか。数えてみよ。答えは 3 vs 0 である。

2. **概念の統一**：借用検査とユーザー述語は同じ証明パイプラインを共有する——`{P} op {Q}`、パイプラインが P を検証する。Curry-Howard 整合的。

3. **新しい分析フレームワークなし**：新しい分析フレームワークを導入しない。ユーザーは「借用検査器」の存在を意識しない——ちょうどユーザーが「型検査器」の実装詳細を意識しないように。

4. **エラーメッセージはユーザーが書いたシンボルだけを含む**：エラー分類が一つの次元全体削減される（E0623、E0106、E0477——すべて `'a` 関連）。変数レベルエラーは Rust と精度同等。

5. **アルゴリズムが非保守的**：逆方向 BFS + break 切断 + SMT 論理的切断。「ループ内保守的生存」は不要。「分岐保守的マージ」は不要。

### 欠点

1. **新しい発明ではない**：ブランド ID がすることは `'a` と完全に同じ——コンパイラ内部の制約求解複雑度は消えておらず、エンコーディングが「変数名 + 制約集合」から「ブランドパス + プレフィックス比較」に変わっただけである。エンドユーザーへの違いはシグネチャに `'a` を書かなくてよいという点のみ。

2. **全新実装**：ブランド木はコード内に概念としてのみ存在し、ゼロから実装する必要がある。BorrowChecker、ControlFlowAnalyzer は置き換えられる。

3. **SMT 依存**：論理的切断は Z3 に依存する（RFC-027 がすでに導入、新規依存は追加なし）。しかし借用検査はめったに発動しない——while + 経路条件の場合のみ呼び出される。

4. **极少数のパターンでリファクタリングが必要**：コンパイラが自動証明できない分岐間借用は、ユーザーがコードをリファクタリングする必要がある。Rust の `'a` フォールバックとは異なる：Rust には `'a` というペンがある（アノテーションすれば通る）；YaoXiang のフォールバック（証明関数）は MVP ではない。

---

## 代替案

| 代替案 | 選択しない理由 |
|------|--------|
| 完全な Rust NLL を実装 | YaoXiang の設計制約（シャドウィングなし、for 新バインディング）がすでに NLL の主要な複雑さの源泉を除去しており、CFG 不動点は不要 |
| 現状維持（Release ハードコード） | 不十分——ユーザーが手動でトークンスコープを管理する必要がある |
| spawn ブロックのみで分析 | 不十分——非 spawn コード内のトークン使用が大多数 |
| GC で借用検査を代替 | 言語設計原則に違反——YaoXiang は GC なし |
