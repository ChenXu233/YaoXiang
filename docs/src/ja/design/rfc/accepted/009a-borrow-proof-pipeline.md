# RFC-009a: トークン生存期間解析——Hoare証明パイプラインに基づく

> **親 RFC**: [RFC-009: 所有権モデル設計](../accepted/009-ownership-model.md)
>
> **依存**: [RFC-027: コンパイル時述語と統一静的検証](../accepted/027-compile-time-evaluation-types.md)
>
> **前提条件**: RFC-027 が承認済みであること。本 RFC のすべての機構（証明パイプライン、SMT フォールバック、パス条件収集）は RFC-027 の実装に依存する。
>
> **本 RFC は RFC-009 §「トークン衝突検出：フロー敏感的活性解析」（第 663-684 行）を修正し、置き換える。**

## 概要

RFC-009 第 684 行は、トークン衝突検出が「……NLL を必要としない」と主張している。結論は正しいが、論拠が誤っている。

「トークンは値であり、線形追跡で十分だから」ではない。**トークン活性は Hoare ロジックの命題であり、専用のフロー敏感解析ではない。**

`{conflicting_tokens すべて死亡} op {WriteToken 安全取得}` —— 同じ `{P} op {Q}` が、型検査、述語検証と RFC-027 の証明パイプラインを共有する。新しい解析フレームワークはない。一本のパイプライン、複数の命題。

---

## 動機

### RFC-009 の混同

RFC-009 は 2 つの問題を混同している：

1. **線形追跡**（Move 後は使用不可）—— `{v が move されていない} use(v) {型が一致}`。型検査器に既に存在。
2. **トークン生存期間相互作用**（子トークン生存 → 親トークン停止 → 子トークン死亡 → 親トークン復活）—— `{conflicting_tokens すべて死亡} write(data) {安全}`。**活性解析**が必要であり、線形追跡ではない。

### 現状のコードの実態

| コンポーネント | 状態 |
|------|------|
| `BorrowChecker` | IR を線形走査し、明示的な `Borrow`/`Release` 命令に受動的に応答 |
| `ControlFlowAnalyzer::analyze_instruction` | 空実装（`control_flow.rs:145-153`） |
| `liveness_analysis` | 存在するが Drop 挿入にのみ使用され、トークン衝突には接続されていない |
| Release 挿入 | Call 命令の後にハードコード——純粋にレキシカルスコープ（`ir_gen.rs:2734-2736`） |

**ユーザーが目にする結果**：

```yaoxiang
data = vec![1, 2, 3]
view = &data              # ReadToken を作成
x = view.total_count      # view の最後の使用
data.push(4)              # ❌ Release(view) がいまだ実行されていないため、ReadToken が「生存中」
```

### なぜ書き直しが必要か

旧版（009a v1）は「DAG が NLL を置き換える」というナラティブを用い、不要な新しい概念（保守的ブランチルール、ループ特殊処理）を導入した。核心的な矛盾が明確でない：**借用検査は独立したシステムではなく、Hoare 命題の一形態である。**

## 中核設計

### すべてが Hoare

```
型検査：        { x: Int }        x + 1        { result: Int }
借用検査：      { view 死亡 }     data.push(4)  { WriteToken 取得成功 }
述語検証：      { y > 0 }         divide(x, y)  { result: Int }
ループ後退切断：{ i == n }        次ループ      { cond == false }
```

同じ形式 `{P} op {Q}`。コンパイラは各操作に対して事前命題 P を生成し、証明パイプラインに投入して検証する。

**借用検査とユーザー述語は同じパイプラインを共有する。** 違いは命題の生成元と、証明できないときの扱いだけである。

### 2 種類の述語、1 本のパイプライン

| | ユーザー述語 | システム述語（借用） |
|---|---|---|
| 命題生成 | プログラマー（型注釈） | コンパイラ（ブランドツリー + 所有権ルール） |
| 証明提供 | コンパイラ + プログラマー | **コンパイラが完全自動** |
| 証明失敗時 | 証明関数を書くか、コードを再構築 | コードを再構築（扉はあるが、ほとんど不要） |
| 可視性 | シグネチャに表れる | 暗黙的、型シグネチャを汚染しない |
| 学習コスト | 使うときに初めて学ぶ | ゼロ |

**システム述語の証明は、プログラマーに証明関数を開放しない——コンパイラが完全自動である。** 証明できない場合、ユーザーがコードを再構築する。

**3 つの失敗モード、同一の検証エンジン。** 型命題が証明できない → コンパイルエラー（回避不可）。借用命題が証明できない → コンパイルエラー、コード再構築（回避不可）。ユーザー述語が証明できない → コンパイルエラー、証明関数を書ける（回避可能）。失敗戦略は異なるが、検証エンジンは同一——SMT ソルバー + コンパイラカーネル推論ルール。違いは「証明できないときに誰が証明を補完するか」だけである——コンパイラは借用証明をプログラマーの代わりに書くことを拒否する（借用命題の証明戦略は構造解析 + SMT であり、プログラマーの介入は不要）。しかし、プログラマーが書いたユーザー述語の証明関数は受け入れる。これはパイプラインの不整合ではなく、命題分類ごとの責任境界が異なるだけである。

これは Rust の `'a` とは異なる：`'a` は必修科目、証明関数は選択科目——大多数のユーザーは一生、選択科目の扉にさえ触れない。

### 借用命題：コンパイラが自動生成

ユーザーが `data.push(4)` と書くと、コンパイラは自動的に次の命題を生成する：

```
WriteToken(data, node) が取得可能
  = ∀ t ∈ conflicting_tokens(data): t は node において死亡
  = ∀ t ∈ brand_tree.children(data): forward_reachable(node) ∩ consumers(t) == ∅
```

**3 つのルール、特殊ケースゼロ：**

1. **ブランドツリー**（RFC-009 §2.7）が「誰と誰が衝突するか」に答える：プレフィックス一致、O(深さ)、深さ ≤ 3
2. **消費者リスト**（DAG 構築時に自動収集）が「トークンを最後に消費したのは誰」に答える
3. **前方到達可能性**が「消費者がまだ実行可能か」に答える：構造的切断 + 論理的切断

### 前方到達可能性：消費者から逆方向に

トークン T の各消費者 C に対して：

```
C から出発し、DAG を逆方向に BFS。
辺は以下の場合に切断される：
  1. それが break である（構造的切断）
  2. パス条件 ⇒ !loop_cond が SMT で真と証明される（論理的切断、RFC-027 パイプライン）

切断されていない辺を逆方向に伝播する（バックエッジを含む。バックエッジは活性を前ループの反復に伝播する）。
到達可能なすべてのノードをマーク → unsafe。
```

クエリ：書込操作がノード W で行われる → W ∉ unsafe → 安全。

**「保守的ブランチルール」を発明する必要はない。「ループ内保守的生存」は不要。逆方向 BFS + 2 つの切断ルールだけ。**

### 証明戦略：ファストパス優先、SMT フォールバック

```
トークンを必要とする各書込操作
  │
  ├→ ファストパス：DAG 構造解析（95% 以上のシナリオをカバー）
  │     │
  │     ├→ ブランドツリー プレフィックス一致 → 衝突トークンを特定（O(深さ)）
  │     ├→ 逆方向 BFS、break がバックエッジを切断
  │     └→ バックエッジを横断できない → 直接 Proved / Disproved を判定
  │
  └→ スローパス：SMT 論理的切断（ファストパスが横断可能なバックエッジに遭遇した場合のみ）
        │
        ├→ バックエッジ始点にパス条件あり → SMT が path_cond ⇒ !loop_cond を判定
        │     ├→ Proved → 論理的切断 → ファストパスに降格して続行
        │     └→ Disproved / Unproven → バックエッジ横断 → unsafe マーク
        │
        └→ バックエッジ始点にパス条件なし → バックエッジを直接横断
```

**ファストパスのカバー範囲**：線形コード、if/else、loop + break、パス条件なしの while。
**スローパスのカバー範囲**：while ループ体内で、パス条件がループ終了を示唆する場合。
**カバー外**：実行時条件を静的に証明できない → バックエッジ横断 → unsafe → コンパイルエラー（ユーザーが再構築）。

SMT は主力ではなく——セーフティネットである。RFC-027 のユーザー述語とは異なる：ユーザー述語は SMT を主力とする；借用システム述語は構造解析を主力とし、SMT は構造解析が届かないエッジケースを補うだけ。

## ユースケース解析

### 線形コード

```yaoxiang
data = vec![1, 2, 3]        # ノード 1
view = &data                # ノード 2：data を消費し、ReadToken(#1) を生成
x = view.total_count        # ノード 3：view を消費（= #1 の最後の消費者）
data.push(4)                # ノード 4：WriteToken(data) を必要とする
```

逆方向 BFS は `view.total_count`（ノード 3）から出発 → ノード 3 は #1 の最後の消費者 → ノード 4 > ノード 3 → ノード 4 は unsafe に含まれない → ✅

### if/else：特殊ルールなし

```yaoxiang
view = &data
if cond {
    use(view)               # then 分岐で view を消費
} else {
    do_something_else()     # view に触れない
}
data.push(4)                # view の最後の消費者は if 内 → if 後は消費者なし → ✅
```

if/else は DAG の複合ノード。内部消費はこのノードに帰属する。分岐状態をマージしない。保守的マージを行わない。**後に消費者がいるかどうかは、整数比較。**

### 戻り値エスケープを伴う if/else

```yaoxiang
view = &data
result = if cond {
    view                     # view が result にエスケープ
} else {
    something_else
}
use(result)                  # view を間接的に消費
data.push(4)                 # view にはまだ消費者がいる（use(result)）
                             # → push は unsafe 内 → ❌ 正しくエラー
```

view は戻り値を通じてエスケープ → `use(result)` は view の消費者 → `push` から逆方向に走査すると `use(result)` に到達可能 → unsafe。

### ループ：break がバックエッジを切断

```yaoxiang
view = &data
loop {
    use(view)                # consumer
    if is_last {
        data.push(4)         # 書込操作
        break                # ← 構造的切断
    }
}
```

逆方向 BFS は `use(view)` から → バックエッジ → 前方の `data.push(4)` に到達 → `break` に遭遇 → **切断** → `data.push(4)` は unsafe に含まれない → ✅

break がない場合：

```yaoxiang
view = &data
loop {
    use(view)
    data.push(4)             # break による切断なし → バックエッジが横断可能 → 次ループの use(view) に到達可能
                             # → push は unsafe 内 → ❌ 正しくエラー
}
```

### while：SMT 論理的切断

```yaoxiang
view = &data
mut i: UpTo(n) = 0
while i < n {
    use(view)                # consumer
    i += 1
    if i == n {
        data.push(4)         # パス条件：i == n
    }
}
```

逆方向 BFS は `use(view)` から → バックエッジ → `data.push(4)` に到達 → パス条件 `i == n` をチェック → SMT クエリ：`i == n ⇒ !(i < n)`? → Proved → **論理的切断** → `data.push(4)` は unsafe に含まれない → ✅

## 本質：ブランド ID は `'a` そのもの

「`'a` は不要」とは言わない。「`#42` は `'42` そのもの」と言う。

| Rust | YaoXiang | 等価性 |
|------|----------|--------|
| `'a` | `#42` | コンパイル時生存期間識別子 |
| `'a: 'b` outlives 制約 | `#42` は `#42.field_x` のプレフィックス | 文字列プレフィックス比較 = 半順序関係 |
| NLL 活性伝播（CFG 不動点） | 逆方向 BFS（DAG） | どちらも到達可能性計算 |
| Polonius facts | SMT 論理的切断 | どちらもパス条件推論 |
| 制約システム不動点求解 | ブランドツリー プレフィックス一致 + BFS | 異なるエンコーディング、同じ問題 |

**新しい解析を発明したのではない。`'a` を型シグネチャ層から証明層に降ろしただけである。** ブランド ID が行うことは `'a` と完全に同じ——借用の識別子をマークし、派生関係を追跡し、衝突を判定する。違いは 1 つだけ：`'a` はユーザーが書いた型シグネチャにある；`#42` はコンパイラ内部にある。

これは恥ずべきことではない。Curry-Howard は型は命題、プログラムは証明と言う。`'a` は命題の一部ではない——証明戦略の一部である。Rust は証明戦略を命題シグネチャに書き込んだ。我々はそれをあるべき場所に戻した。

### 言語設計制約が取り除いたもの

| 複雑性の源 | 回避したか？ | 理由 |
|---|---|---|
| 変数シャドウィング | ✅ | 言語で禁止——同じ名前は同じものを指し続ける |
| for クロス反復借用 | ✅ | 各反復で新しいバインド——反復間は自然に分離 |
| `'a` 生存期間注釈 | ✅ | ブランドパス = `#42.field_x`、コンパイラが推論 |
| 名前付き生存期間 + 制約伝播 | ✅ | ブランドパスプレフィックス比較が明示的制約集合を置換 |
| 借用グラフ制約求解（Polonius） | ✅ | ブランドツリー プレフィックス一致 + DAG 消費者クエリ |
| ループ体内借用活性伝播 | ❌ | Rust と同様に処理が必要——逆方向 BFS + 論理的切断で対応 |
| 条件分岐保守性 | ❌ | Rust と同様——SMT が証明可能なものをカバーし、残りは保守的に拒否 |

### なぜ DAG が機能するか

YaoXiang の 3 つの言語設計制約が DAG 解析を可能にする：

- **変数シャドウィングなし**——同じ名前は同じものを指し続けるため、再バインド間の追跡が不要
- **for の各反復は新しいバインド**——反復間は自然に分離され、クロス反復借用が存在しない
- **構造化並行性**——タスク境界が明確で、タスク間活性伝播が不要

これらの制約は Rust の CFG 不動点反復の主要な複雑性の源を除去する。DAG が CFG より「高度」だからではなく、よりシンプルな言語設計がよりシンプルな解析を可能にする。

## 詳細設計

### システム述語リスト

コンパイラは次の命題を自動生成し、RFC-027 証明パイプラインに投入する：

| システム述語 | トリガタイミング | 命題形式 |
|---|---|---|
| `borrow_conflict` | WriteToken(v) を必要とするとき | `∀ t ∈ conflicting(v): dead_at(t, node)` |
| `use_after_move` | 変数 v を使用するとき | `¬moved(v)` |
| `use_after_drop` | 変数 v を使用するとき | `¬dropped(v)` |
| `double_drop` | Drop(v) | `¬dropped(v)` |
| `mut_violation` | 不変変数 v への書込 | `is_mut(v)` |

既存の `BorrowChecker`、`MoveChecker`、`DropChecker`、`MutChecker` は**命題生成器**になる——消えるのではなく、役割を変える。彼らは命題を生成し、パイプラインがそれを検証する。

### ブランドツリー

RFC-009 §2.7 のブランド機構をブランドツリーとして形式化する。

**トークンセマンティクス——凍結優先、コピー非優先**：

`&T` と `&mut T` の本質的な違いは「コピー可能かどうか」ではなく、「同時書込を許可するか否か」である：

```
ReadToken(T)：  読み取り専用権限を付与し、ソースデータ T を凍結する——この期間中、
              任意の WriteToken(T) は取得不可。凍結は ReadToken の第一のセマンティクス。
              Dup（コピー可能）は凍結の帰結：データがすでに凍結されている（突然変異の
              可能性がない）ため、複数の読み取り専用ビューは自然に安全。

WriteToken(T)： 排他的な読み書き権限を付与。書込が存在するため、他の任意のトークン
              （読み取りまたは書込）は共存不可。Dup（線形型）を実装しないのは
              排他性の帰結。
```

**因果関係**：
```
ReadToken 存在 → ソースデータ凍結 → 複数の読み取り専用が安全 → Dup
                      ↓
              WriteToken は拒否される（borrow_conflict システム述語が強制）
```

次のようではない：
```
ReadToken は Dup を持つ → 複数持てる → ついでに衝突をチェック  ← 因果が逆転
```

```
BrandTree:
  nodes: Map<BrandId, BrandNode>

BrandNode:
  id: BrandId               # "#42"、"#42.field_x"
  kind: ReadToken | WriteToken
  source_var: Operand
  parent: Option<BrandId>   # 派生関係の親ノード
  children: Set<BrandId>    # 派生トークン
  consumers: Set<NodeId>    # トークンを消費する DAG ノード
  ref_count: usize          # ReadToken 凍結期間中の安全コピー数
```

**衝突判定**——凍結保証の実行メカニズム：

```rust
fn conflicts(a: &BrandId, b: &BrandId) -> bool {
    // 衝突条件：同ソース + 少なくとも一方が書込 + ブランドパスがオーバーラップ
    // これは次のことを意味する：
    //   1. ReadToken vs ReadToken → 衝突なし（どちらも読み取り専用、突然変異なし）
    //   2. WriteToken vs ReadToken → 衝突（書込が読み取りの凍結保証を破る）
    //   3. WriteToken vs WriteToken → 衝突（2 つの書込は共存不可）
    a.source() == b.source()
        && (a.is_write() || b.is_write())
        && (a.is_prefix_of(b) || b.is_prefix_of(a))
}
```

O(深さ) 文字列プレフィックス比較、深さ ≤ 3。定数時間。

### 逆方向 BFS 活性解析

```
アルゴリズム：check_borrow(token, node, dag, brand_tree)

入力：
  token: チェック対象の WriteToken
  node:  書込操作が位置する DAG ノード

出力：Proved | Disproved

アルゴリズム：
  # ファストパス：逆方向 BFS
  unsafe = empty_set
  queue = brand_tree.consumers(token)

  while queue not empty:
    cur = queue.pop()
    unsafe.add(cur)

    for each pred in dag.predecessors(cur):
      # 構造的切断：break は横断しない
      if pred が break 辺:
        continue

      # バックエッジ → SMT フォールバックが必要かチェック
      if pred がバックエッジ:
        path_cond = pred におけるパス条件
        loop_cond = ループ条件
        # まず構造的に切断可能か確認（対応する break がパスをすでに切断している → ここに到達しない）
        # 次にパス条件を確認
        if path_cond が空でない:
          result = smt_fallback(path_cond, loop_cond)   # ← スローパス
          if result == Proved:
            continue                    # 論理的切断
        # パス条件なし or SMT が証明できない → バックエッジを横断
        # fall through

      if pred ∉ unsafe:
        queue.push(pred)

  # 判定
  if node ∈ unsafe:
    return Disproved
  else:
    return Proved


smt_fallback(path_cond, loop_cond):
  # バックエッジ + パス条件ありのときのみ呼び出し
  # RFC-027 証明パイプラインを使用し、同じ SMT ソルバー、同じ予算を共有
  return smt.prove(path_cond ⇒ !loop_cond)
  # Proved → 論理的切断
  # Disproved / Unproven → 切断しない、バックエッジを横断
```

O(N)、SMT 呼び出し回数 = バックエッジ数 × パス条件付きバックエッジの割合。実際のコードでは SMT 呼び出しは極めて稀——`while` ループ体内で、精化型変数を持つパス条件があるときのみトリガ。

### パス条件収集

RFC-027 §3.2-3.3 の既存機構が提供する：

- **if guard**：`if y > 0` → true 分岐に `y > 0` をプッシュ
- **match パターン**：`if let Some(v) = opt` → 分岐内に `opt == Some(v)` をプッシュ
- **代入**：`i += 1`、コンパイラが変数の値域情報を保守
- **while cond**：ループ体内に `cond == true` をプッシュ

各 DAG ノードはパス条件集合を保持する。逆方向 BFS のバックエッジで、バックエッジ始点のパス条件を取り、SMT が次ループ入口条件の否定を証明するかを判定する。

### RFC-027 とのインターフェース

借用システム述語とユーザー述語は同じ証明パイプラインを共有する——違いは**主戦略**：

| クエリ種別 | 命題のソース | 主戦略 | フォールバック |
|---|---|---|---|
| 型等式 | 型検査器 | 構造的同値性 | — |
| ユーザー述語 | プログラマーの型注釈 | SMT | プログラマーの証明関数 |
| 借用衝突 | コンパイラ自動生成 | **DAG 構造解析（ファストパス）** | SMT 論理的切断 |

借用検査における SMT ソルバーの役割：**主力ではなく、セーフティネット。** while バックエッジが論理的切断を必要とするときのみ呼び出される。大多数の借用検査はファストパスで完了する——O(N) 逆方向 BFS、SMT オーバーヘッドゼロ。

### 既存コードとの関係

| 既存コンポーネント | 処理 |
|----------|------|
| `BorrowChecker` | `BorrowPredicateEmitter` になる——借用 Hoare 命題を生成 |
| `MoveChecker` | `MovePredicateEmitter` になる——`¬moved(v)` 命題を生成 |
| `DropChecker` | 同上——Drop 関連命題を生成 |
| `MutChecker` | 同上——`is_mut(v)` 命題を生成 |
| `ControlFlowAnalyzer` | 不要になる——パイプラインが統一的に処理 |
| `liveness_analysis` | 維持——Drop 挿入は依然として変数活性情報を必要とする |
| `ir_gen.rs` Release ハードコード | 削除——Release 位置は DAG 消費者解析が駆動 |

### NLL と反復境界

**トークン死亡時刻 = 最後の使用点（NLL）、レキシカルスコープ末尾ではない。**

これは消費者解析の自然な帰結である：消費者の位置がトークンの最後の使用を定義する。`use(v)` は `v` の消費者 → `v` は `use(v)` の直後に死亡する。トークン生存期間を早期終了させるための追加の `{}` や `drop()` は不要。

**ループ反復境界はトークンコピーの死亡線である。** 3 つのルール：

```
ルール 1：ループ内で宣言された変数は、各反復終了時に自動死亡する。
        for の各反復は新しいバインド（言語設計保証）、loop も同様。

ルール 2：ブランドツリー ref_count はループ頭でループ外で作成されたコピーのみカウントする。
        ループ内 Dup により生成された新しいコピーは、ref_count 0 で反復境界で消滅する。

ルール 3：逆方向 BFS がバックエッジを横断するとき、現在の反復の活性情報を伝播しない。
        ループ頭での ref_count（つまり：ループ外のコピー）のみ伝播する。
```

インスタンス：

```yaoxiang
view = &data                          # ループ頭：ref_count = 1、consumer = use(view)
loop {
    v2: &Point = view                 # ループ内 Dup → ref_count = 2
    use(v2)                           # consumer：v2 の最後の使用 → v2 死亡 → ref_count = 1
    data.push(4)                      # ✅ 安全！v2 は死亡済み、view のみ（ref_count = 1、書込衝突なし）
    # 反復境界：ルール 3——v2 を次反復に伝播しない。次反復開始時に v2 は新しいバインドで再作成される。
}
```

この設計は追加の「ループ内保守的生存」ルールを必要としない。逆方向 BFS は消費者から出発、消費者はループ体内 → 活性は現在の反復内に制限される → バックエッジは横断しない。RFC-009a §ユースケース解析のループ例と完全に一致する。

### `?` エラー伝播とスコープ駆動 Release

`?` は早期 return である——スコープの正常終了に加えて、もう 1 つの終了パスがある。トークンもこのパス上で解放される必要があり、解放順序が誤れば UB となる。

**Release 命令はスコープ解析により生成され、Call の後にハードコードされるのではない。**

コンパイラは各スコープに対して出口点リストを保守する：
- `}`（スコープの正常終了）
- `?`（エラー伝播、早期 return）
- 明示的な `return`

各出口点で、宣言の逆順（LIFO）でそのスコープ内のすべての生存中トークンの Release 命令を挿入する。ブランドツリーの親子関係が派生トークンのカスケード解放を自動的に処理する：

```yaoxiang
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)    # 子トークン &Float + 親トークン &Point を返す
}

fn use_case(p: Point) -> Result<(), Error> = {
    (x_ref, p_ref) = p.get_x()?   # ? が伝播する場合：
    # ブランドツリーは x_ref が p_ref の派生（#42.field_x は #42 のプレフィックス）であることを知っている
    # 解放順序：x_ref（子）→ p_ref（親）→ LIFO が自動満足
    p.modify()                     # WriteToken——すべての ReadToken は解放済み
    Ok(())
}
```

実装位置：`ir_gen.rs` に維持、スコープ駆動に変更——新しい compiler pass は導入しない。

| 操作 | 計算量 | トリガ頻度 |
|------|--------|---------|
| ブランドツリー衝突判定 | O(1) | トークン要求ごと |
| DAG 消費者クエリ | O(1) | トークン要求ごと |
| 逆方向 BFS（ファストパス） | O(N) | トークン要求ごと、N = ブロック内ノード数 |
| SMT 論理的切断（フォールバック） | ~1ms | **極めて稀**——while + パス条件時のみ |

**SMT フォールバックのトリガ条件は極めて厳格**：(1) while ループ、(2) ループ体内に書込操作、(3) 書込操作後にループ終了を判定できるパス条件、(4) コンパイラがその条件に依存してバックエッジを切断する必要がある。実際のコードでの比率は 1% よりはるかに低い。残りの借用検査はすべてファストパスで完了する。

RFC-027 ユーザー述語との関係：ユーザー述語は SMT を主力とし、借用システム述語は構造解析を主力とする。両者は同じ SMT ソルバーと予算上限（RFC-027 §8）を共有するが、借用システム述語は SMT 予算をほとんど消費しない。

線形コード → バックエッジなし → 層 1 O(N) で即座に結果。ループ + パス条件 → SMT 呼び出し、線形算術でミリ秒オーダー（RFC-027 予算 100ms）。一度の BFS 結果はキャッシュされ、同じトークンの複数クエリで再利用できる。

### エラーメッセージ設計

**核心原則：エラーメッセージにはユーザーが書いたシンボルだけを表示する。**

Rust の借用関連エラーは 2 種類に分かれる：

**変数レベルエラー**：E0597（生存期間不足）、E0502（可変+不変の同時借用）、E0499（複数可変借用）。Rust はすでに模範——変数名+行番号、`'a` は表示されない。YaoXiang の精度は同等。情報はすべてブランドツリーにある：トークン生成点、消費者位置、要求点。

**シグネチャレベルエラー**：E0623（生存期間不一致）、E0106（生存期間指定子の欠落）、E0477（required lifetime 不満足）。`'a` を中心に展開される。YaoXiang には**この種のエラーは存在しない**——シグネチャに `'a` がないため。「報告できない」のではなく、ユーザーが書かなかったものは報告しない。

関数内衝突の例：

```
エラー：`data` は凍結されており、可変権限を取得できません
 --> src/main.yx:5:9
2 |     view = &data
  |            ----- `data` は凍結されています（読み取り専用トークンがここで作成）
4 |         use(view)
  |             ---- `view` はここでまだ使用中、凍結は解除されていない
5 |         data.push(4)
  |         ^^^^ ここで可変権限が必要
```

（Rust E0499 と同等の精度——変数名+行番号、ブランド ID は表示されない。）

関数間エスケープの例：

```
エラー：`num`（4 行目）が保持するデータソースの 1 つは `default_str`（3 行目）ですが、
`default_str` は 6 行目で失効し、`num` は 5 行目でもまだ使用中です。

検討：`default_str` の宣言を呼び出し側に移動するか、`ref default_str` で共有保持を使用してください。
```

（Rust E0597 と同等の精度。ブランド要約は `num` が 2 つのソースパスを持つことを知っている——コンパイラにすでに存在、エラーメッセージ文言に使用可能。）

## RFC-009 本文の修正

RFC-009 §「トークン衝突検出：フロー敏感的活性解析」を更新：

1. 「不要なもの：……NLL」を削除——結論が誤りだからではなく、理由が誤りだから（「トークンは値であり、線形追跡で十分」）
2. 層 1/層 2 移行ソリューションを保持、完全なソリューションは本 RFC を指す
3. 明確化：ブランド ID（`#42`）は `'a` そのもの——情報は完全に同じ、エンコーディングが異なる。新しい解析を発明したのではなく、生存期間を型層から証明層に降ろしただけ

---

## トレードオフ

### 利点

1. **型シグネチャに生存期間を含まない**：`#42` は `'42` そのもの——同じ情報、ブランドツリーにエンコードされ、型シグネチャに露出しない。この点は反証不可能：Rust で 3 つの参照引数を持つジェネリック型に何個の `'a` パラメータが必要かを数えてみる、YaoXiang で何個必要か。答えは 3 対 0。

2. **概念の統一**：借用検査とユーザー述語は同じ証明パイプラインを共有する——`{P} op {Q}`、パイプラインが P を検証。Curry-Howard と一致。

3. **新しい解析フレームワークゼロ**：新しい解析フレームワークを導入しない。ユーザーは「借用検査器」の存在を感知しない——「型検査器」の実装詳細をユーザーが感知しないのと同じ。

4. **エラーメッセージはユーザーが書いたシンボルのみ**：エラー分類の 1 つの次元全体が消える（E0623、E0106、E0477——すべて `'a` 関連）。変数レベルエラーは Rust と同等の精度。

5. **アルゴリズムは保守的でない**：逆方向 BFS + break 切断 + SMT 論理的切断。「ループ内保守的生存」は不要。「分岐保守的マージ」は不要。

### 欠点

1. **新しい発明ではない**：ブランド ID が行うことは `'a` と完全に同じ——コンパイラ内部の制約求解複雑性は消えていない、「変数名+制約集合」から「ブランドパス+プレフィックス一致」にエンコーディングが変わっただけ。エンドユーザーへの違いはシグネチャに `'a` を書かない点のみ。

2. **まったく新しい実装**：ブランドツリーはコード内にコンセプトのみ存在し、ゼロから実装が必要。BorrowChecker、ControlFlowAnalyzer は置換される。

3. **SMT 依存**：論理的切断は Z3 に依存（RFC-027 で導入済み、新規依存なし）。しかし借用検査はトリガしない——while + パス条件時のみ。

4. **極めて稀なパターンで再構築が必要**：コンパイラの自動証明がカバーできないブランチ間借用では、ユーザーがコードを再構築する必要がある。Rust の `'a` のフォールバックとは異なる：Rust には `'a` という筆がある（注釈すれば通る）；YaoXiang のフォールバック（証明関数）は MVP ではない。

## 代替案

| 案 | なぜ選ばないか |
|------|-----------|
| Rust NLL の完全実装 | YaoXiang の設計制約（シャドウィングなし、for 新バインド）は NLL の主要複雑性源をすでに除去しており、CFG 不動点は不要 |
| 現状維持（ハードコード Release） | 不十分——ユーザーはトークンスコープを手動管理する必要があり |
| spawn ブロックでのみ解析 | 不十分——非 spawn コードでのトークン使用が大多数 |
| 借用検査の代わりに GC | 言語設計原則に違反——YaoXiang は GC なし |
