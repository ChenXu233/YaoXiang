# RFC-008：Runtime 並行モデルとスケジューラ分離設計

> **⚠️ 整合に関する注意**：本ドキュメントは [RFC-024 の新しい並行モデル](/reference/language-spec/concurrency.md) と整合しています。旧来の全プログラム DAG 分析、`@block`/`@eager` アノテーション、L1/L2/L3 レベルモデルは `spawn {}` ブロック並行プリミティブに置き換えられました。DAG 分析は現在 `spawn {}` ブロック内部にのみ作用します。

> **参考**:
> - [RFC-011: ジェネリクスシステム設計](./011-generic-type-system.md)
> - [並行モデル仕様（RFC-024）](/reference/language-spec/concurrency.md)

## 概要

本ドキュメントは Runtime アーキテクチャの重要な設計を定義します：

1. **ランタイム三層アーキテクチャ**：Embedded（即時実行） → Standard（spawn + DAG スケジューリング） → Full（ワークスティーリング）
2. **コンパイルと実行の分離**：コンパイルフェーズはすべて同一で、違いがるのは実行時の実行方式のみ
3. **デュアルバックエンドモデル**：VM（開発・ブレークポイントデバッグ用）と LLVM AOT（本番リリース用）、動作は完全一致
4. **スケジューラ = 静的ライブラリ**：AOT コンパイル時にスケジューラを exe にリンク、約 200-500KB、GC なし
5. **同期はスケジューリングの特例**：num_workers=1 で同期モード

### 重要な澄清：これは Java ではない

```
Java:   .java → .class → JVM（解釈/JIT + GC）        ← 常にVMが必要
YaoXiang 開発: .yx → IR → VM 実行（高速イテレーション、ブレークポイントデバッグ）
YaoXiang 本番: .yx → IR → LLVM → ネイティブ exe（スケジューラが静的ライブラリとしてリンク）

VM は開発ツールであり、ランタイムの本質ではありません。Go の go run vs go build と同じです。
最終的な exe = あなたのネイティブコード + スケジューラ静的ライブラリ + リフレクションメタデータ。
インタプリタなし、JIT なし、GC なし。
```

## 動機

### 中核的矛盾

| 矛盾 | 説明 |
|------|------|
| 透過性 vs 制御性 | spawn ブロックは明示的な並行制御を提供し、通常のコードは順序実行 |
| コア vs オプション | spawn はコアな並行プリミティブ、WorkStealing は num_workers>1 の高度な機能 |
| シングルスレッド vs 並行 | シングルスレッドモードでは並行的動作は非同期として現れ、同期はスケジューリングの特例 |

## 提案

### 1. Runtime 三層アーキテクチャ

```
┌──────────────────────────────────────────────────────────────────┐
│                    コンパイルフェーズ（全モード共通）                │
│                                                                  │
│  Source Code → Lexer → Parser → TypeCheck → Codegen → IR        │
│                                                                  │
│  ⚠️ 同一の構文解析、型チェック、コード生成、IR 出力                  │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即時エグゼキュータ │ │ spawn + DAG   │ │ Full スケジューラ│
│ 同期実行         │ │ spawnブロック内│ │ 並列最適化       │
│ spawn サポートなし│ │ 並行          │ │ ワークスティーリング│
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| 段階 | Embedded | Standard | Full |
|------|----------|----------|------|
| コンパイル | 共通 | 共通 | 共通 |
| 実行モード | 同期 | spawn ブロック内並行 | 並列 |
| メモリ使用量 | 低 | 中 | 高 |
| 並行能力 | なし | spawn ブロック内 | spawn ブロック内 + 並列 |
| spawn サポート | ❌ | ✅ | ✅ |
| DAG 分析 | なし | spawn ブロック内 | spawn ブロック内 |
| WorkStealer | なし | なし | ✅ |

**Embedded Runtime**：目標は WASM/ゲームスクリプト/ルールエンジン。即時エグゼキュータ、spawn サポートなし、高性能低メモリ。

**Standard Runtime**：目標は Web サービス/データパイプライン。`spawn {}` ブロックをサポートし、spawn ブロック内で DAG 分析と自動並行実行を行います。num_workers=1 はシングルスレッド非同期。

**Full Runtime**：目標は科学計算/大規模並列。Standard + WorkStealer によるロードバランシング。

### 2. スケジューラ分離：ジェネリクス + 注入

基本原则：VM は具体的なスケジューラを直接依存せず、ジェネリックパラメータ `[S]` を通じて呼び出します。

```yaoxiang
# スケジューラインターフェース定義
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# シングルスレッドスケジューラ
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# マルチスレッドスケジューラ
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VM はジェネリクスでスケジューラを使用
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**コアポイント**：
- コンパイル時多相、ゼロ런タイムオーバーヘッド
- トレイトオブジェクト不要
- ジェネリック型制約 `[S: Scheduler]` は RFC-011 で既に定義済み

### 3. 同期 = スケジューリングの特例

```
❌ 誤解：スケジューラを無効化
✅ 正しい：単一ワーカーのスケジューラを使用

num_workers = 1 → シングルスレッド非同期スケジューリング
num_workers > 1 → マルチスレッド並列スケジューリング

同一のスケジューラインターフェース，只是設定が異なる。特殊ケースを排除。
```

### 4. DAG の立場

> **重要な変更**：DAG 分析は全プログラムに作用不再し、`spawn {}` ブロック内部のみで行われます。通常のコード（spawn ブロック外部）は順序実行され、DAG 分析は不要です。

| レベル | spawn サポート | DAG 分析範囲 | 説明 |
|------|-----------|-------------|------|
| Core Runtime | ✅ | spawn ブロック内 | 並行の中核 |
| Standard Runtime | ✅ | spawn ブロック内 | spawn + DAG スケジューリング |
| Embedded Runtime | ❌ | なし | 即時実行、並行なし |

### 5. ボトムアップ実行モデル（spawn ブロック内）

> **重要な変更**：ボトムアップ DAG 分析は `spawn {}` ブロック内部にのみ行われ不再、全プログラムの DAG 分析は行いません。

```
ユーザコード（spawn ブロック内並行）：
    (a, b) = spawn {
        fetch(url0),
        fetch(url1)
    }
    print(a)

コンパイル時分析（spawn ブロック内ボトムアップ）：
    fetch(url0) と fetch(url1) は相互依存なし → 並行実行可能
    spawn ブロック外の print(a) → 順序実行、spawn 完了を待機

ランタイムスケジューリング（spawn ブロック内、リーフから開始）：
    fetch(url0) ┐
                ├→ 並列実行
    fetch(url1) ┘
    print(a)                       ← spawn ブロック外、順序実行
```

**コアポイント**：
- ボトムアップ依存分析は `spawn {}` ブロック内部に限定
- spawn ブロック内で依存のないタスクは並列実行
- spawn ブロック外のコードは順序実行、spawn ブロック完了を待機

### 6. コンパイルモデル：デュアルバックエンド + 静的リンクランタイム

#### 6.1 2つのバックエンド、同一の動作

```
                      ┌─────────────────────┐
                      │   コンパイルフロントエンド（共通） │
                      │   Lexer → Parser     │
                      │   → TypeCheck        │
                      │   → spawnブロック内DAG分析│
                      │   → エスケープ分析    │
                      │   → サイクル検出      │
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │   VM バックエンド（開発）│     │  LLVM バックエンド（本番）│
        │                   │     │                   │
        │  IR/バイトコード生成   │     │  ネイティブコード生成    │
        │  VM 解释実行         │     │  ランタイム静的ライブラリ │
        │  ブレークポイントデバッグ│     │  リンク                │
        │  高速イテレーション    │     │  .exe 出力            │
        │                   │     │  ゼロ解释オーバーヘッド │
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           動作完全一致                 動作完全一致
```

**VM バックエンド**：開発時に使用。コード修正 → 即時実行 → ブレークポイントデバッグ → 高速イテレーション。動作は最終 exe と完全一致。

**LLVM バックエンド**：リリース時に使用。AOT コンパイルでネイティブコードに、スケジューラを静的ライブラリとしてリンク。インタプリタなし、JIT なし。

#### 6.2 スケジューラ = 静的ライブラリ、而不是虚拟机

```
最終 exe の内部構造：

┌────────────────────────────────────────────┐
│  あなたのコード（ネイティブ機械語）                       │
│  ├── コンパイル時に確定した DAG 実行計画              │
│  ├── インライン化された Move/ref/clone 操作            │
│  └── RAII 解放コード                          │
├────────────────────────────────────────────┤
│  ランタイム静的ライブラリ（~200-500KB）                   │
│  ├── スレッドプール（固定サイズ = num_workers）         │
│  ├── イベントループ（libuv / io_uring）            │
│  ├── ワークスティーリングキュー（Full Runtime のみ）          │
│  ├── メモリアロケータ（jemalloc / mimalloc）       │
│  └── リフレクションメタデータ（オンデマンドロード、常駐なし）         │
├────────────────────────────────────────────┤
│  なし：                                      │
│  ❌ バイトコードインタプリタ                             │
│  ❌ JIT コンパイラ                               │
│  ❌ GC                                      │
│  ❌ 仮想機械                                    │
└────────────────────────────────────────────┘
```

比較：

| 言語 | Java | Go | YaoXiang |
|------|------|-----|-----------|
| コンパイル产物 | バイトコード | ネイティブコード | ネイティブコード |
| 実行方式 | JVM 解釈/JIT | 直接実行 | 直接実行 |
| ランタイムサイズ | ~200MB（JVM） | ~1-2MB（GC 含む） | **~200-500KB（GC なし）** |
| メモリ管理 | GC | GC | **RAII（决定的）** |
| リフレクション | 常時メモリ常駐 | 常時メモリ常駐 | **exe に保管、オンデマンドロード** |

#### 6.3 なぜスケジューラ性能が一定か

**重要な洞察**：大部分の工作是コンパイル時に完了し、ランタイムでは「実行」のみを行います。

```
コンパイル時（一回限り、ランタイムに入らない）：
    ├── spawn ブロック内 DAG 分析：誰か谁に依存するか
    ├── トポロジカルソート：spawn ブロック内実行順序の確定
    ├── 並行実行可能タスクの識別：spawn ブロック内依存のないサブツリー
    ├── エスケープ分析：ref → Rc か Arc か
    ├── サイクル検出：自動降格 Weak またはエラー
    └── インライン：小関数は直接展開

ランタイム（每次実行、データ構造固定）：
    ├── コンパイル時に確定した spawn ブロック DAG 順序でタスクをスレッドプールに分配
    ├── I/O 遭遇時 → 現在のタスクを停止、イベントループが接管
    ├── タスク準備完了 → 準備完了キューに戻す
    └── 以上。
```

**スケジューラ自体は固定サイズのデータ構造**：スレッドプール、イベントループ、ワークキュー。動的成長なし、アダプティブな再最適化なし、GC スキャンなし。動作は完全に予測可能。

コンパイル時に spawn ブロック内の「何をスケジューリングするか」を計算し終えており、ランタイムでは「実行」のみを行います。これは tokio とは異なり、tokio はランタイムで動的に Future チェーンを構築します。YaoXiang の DAG は静的であり、spawn ブロック内に限定されます。

#### 6.4 リフレクション：保管するが常駐させない

リフレクションメタデータはコンパイル時に生成され、exe の独立セクションに保管されます。プログラム起動時はロードしません。初めてリフレクションリクエスト時に、オンデマンドで mmap によりメモリにロードします。例えるなら：

```
exe レイアウト：
  .text     ← あなたのコード
  .rodata   ← 定数
  .reflect  ← リフレクションメタデータ（型情報、関数シグネチャなど）
              mmap オンデマンドロード、不アクセスならメモリ使用なし
```

**トレードオフ**：exe サイズ増加（リフレクションデータを含む）が、ランタイムでアクセスしなければメモリ使用量ゼロ。初回のアクセスにロード遅延あり（JIT ウォームアップ類似）、以降はゼロオーバーヘッド。

## トレードオフ

### 利点

- **明確な階層化**：Embedded / Standard / Full の三層
- **コンパイル再利用**：フロントエンドコードは完全共有
- **ジェネリクス分離**：コンパイル時多相、ゼロオーバーヘッド
- **一貫性**：同期はスケジューリングの特例
- **組込み向け設計**：高性能 + 低メモリ + 高速起動

### 欠点

- **初期複雑度**：スケジューラインターフェースと複数のランタイムバリアントを定義する必要がある
- **コンパイル時バインディング**：スケジューラ型はコンパイル時に確定
