# RFC-008：Runtime同時実行モデルとスケジューラ分離設計

> **ステータス**: 承認済み
> **著者**: 晨煦
> **作成日**: 2025-01-05
> **最終更新**: 2026-06-05（RFC-024新同時実行モデルに整合：DAG分析範囲をスポーン{}ブロック内に縮小、旧L1/L2/L3参照を削除）

> **⚠️ 整合性に関する注記**：本文書は [RFC-024新同時実行モデル](/reference/language-spec/concurrency.md) に整合しています。旧式の全プログラムDAG分析、`@block`/`@eager`注釈、L1/L2/L3階層モデルは `spawn {}` ブロック並列プリミティブに取って代わられました。DAG分析は現在は `spawn {}` ブロック内部でのみ機能します。

> **参考**:
> - [RFC-011: ジェネリクスシステム設計](./011-generic-type-system.md)
> - [同時実行モデル規範（RFC-024）](/reference/language-spec/concurrency.md)

## 要約

本文書はRuntimeアーキテクチャの重要設計を定義します：

1. **ランタイム三層アーキテクチャ**：Embedded（即時実行） → Standard（spawn + DAGスケジューリング） → Full（ワークスティーリング）
2. **コンパイルと実行の分離**：コンパイル段階は完全に同一、違いは実行時の実行方式のみ
3. **デュアルバックエンドモデル**：VM（開発・デバッグ用）とLLVM AOT（本番リリース用）、動作は完全に一致
4. **スケジューラ = 静的ライブラリ**：AOTコンパイル時にスケジューラをexeにリンク、約200-500KB、GCなし
5. **同期はスケジューリングの特例に過ぎない**：num_workers=1で同期モード

### 重要な澄清：これはJavaではない

```
Java:   .java → .class → JVM（解釈/JIT + GC）        ← 常に仮想マシンが必要
YaoXiang 開発: .yx → IR → VM実行（クイックイテレーション、ステップデバッグ）
YaoXiang 本番: .yx → IR → LLVM → ネイティブexe（スケジューラが静的リンク）

VMは開発ツールであり、ランタイムの本質ではない。Goのgo run vs go buildと同じ。
最終的なexe = あなたのネイティブコード + スケジューラ静的ライブラリ + リフレクションメタデータ。
解釈器なし、JITなし、GCなし。
```

## 動機

### 中核的矛盾

| 矛盾 | 説明 |
|------|------|
| 透明性 vs 制御性 | スポーン{}ブロックは明示的な同時実行制御を提供し、通常コードは順序実行 |
| 中核 vs 任意 | spawnは中核的な並列プリミティブ、WorkStealingはnum_workers>1の高度な機能 |
| 単一スレッド vs 同時実行 | 単一スレッドモードでは同時実行は非同期として振る舞い、同期はスケジューリングの特例 |

## 提案

### 1. Runtime三層アーキテクチャ

```
┌──────────────────────────────────────────────────────────────────┐
│                    コンパイル段階（全モード共通）                   │
│                                                                  │
│  ソースコード → Lexer → Parser → TypeCheck → Codegen → IR       │
│                                                                  │
│  ⚠️ 同一の構文解析、型チェック、コード生成、IR出力                  │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即時実行器        │ │ spawn + DAG   │ │ Fullスケジューラ  │
│ 同期実行         │ │ スポーン{}ブロック内│ │ 並列最適化        │
│ spawnサポートなし │ │ 自動同時実行   │ │ ワークスティーリング│
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| 段階 | Embedded | Standard | Full |
|------|----------|----------|------|
| コンパイル | 共通 | 共通 | 共通 |
| 実行モード | 同期 | スポーン{}ブロック内同時実行 | 並列 |
| メモリ使用量 | 低 | 中 | 高 |
| 同時実行能力 | なし | スポーン{}ブロック内 | スポーン{}ブロック内 + 並列 |
| spawnサポート | ❌ | ✅ | ✅ |
| DAG分析 | なし | スポーン{}ブロック内 | スポーン{}ブロック内 |
| WorkStealer | なし | なし | ✅ |

**Embedded Runtime**：目標はWASM/ゲームスクリプト/ルールエンジン。即時実行器、spawnサポートなし、高性能低占用。

**Standard Runtime**：目標はWebサービス/データパイプライン。`spawn {}`ブロックをサポートし、スポーン{}ブロック内でDAG分析及と自動同時実行を実行。num_workers=1で単一スレッド非同期。

**Full Runtime**：目標は科学計算/大規模並列。Standard + WorkStealerによるロードバランシング。

### 2. スケジューラ分離：ジェネリクス + 注入

中核原則：VMは具体的なスケジューラに直接依存せず、ジェネリックパラメータ `[S]` を通じて呼び出します。

```yaoxiang
# スケジューラインターフェース定義
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# 単一スレッドスケジューラ
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# マルチスレッドスケジューラ
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VMはジェネリクスでスケジューラを使用
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**中核ポイント**：
- コンパイル時多相、ゼロランタイムオーバーヘッド
- トレイトオブジェクト不要
- ジェネリック型制約 `[S: Scheduler]` はRFC-011で既に定義済み

### 3. 同期 = スケジューリングの特例

```
❌ 誤解：スケジューラを無効化
✅ 正しい：単一workerのスケジューラを使用

num_workers = 1 → 単一スレッド非同期スケジューリング
num_workers > 1 → マルチスレッド並列スケジューリング

同じスケジューラインターフェース、設定が異なるだけ。特殊ケースを排除。
```

### 4. DAGの位置づけ

> **重要な変更**：DAG分析は全プログラムに作用しなくなり、`spawn {}`ブロック内部でのみ実行されます。通常コード（スポーン{}ブロック外部）は順序実行され、DAG分析は不要です。

| 階層 | spawnサポート | DAG分析範囲 | 説明 |
|------|-------------|-------------|------|
| Core Runtime | ✅ | スポーン{}ブロック内 | 同時実行の中核 |
| Standard Runtime | ✅ | スポーン{}ブロック内 | spawn + DAGスケジューリング |
| Embedded Runtime | ❌ | なし | 即時実行、同時実行なし |

### 5. ボトムアップ実行モデル（スポーン{}ブロック内）

> **重要な変更**：ボトムアップDAG分析は `spawn {}` ブロック内部でのみ実行され、全プログラムのDAG分析不再実行されます。

```
ユーザコード（スポーン{}ブロック内同時実行）：
    (a, b) = spawn {
        fetch(url0),
        fetch(url1)
    }
    print(a)

コンパイル時分析（スポーン{}ブロック内ボトムアップ）：
    fetch(url0) と fetch(url1) に相互依存なし → 並行実行可能
    スポーン{}ブロック外の print(a) → 順序実行、spawn完了を待機

ランタイムスケジューリング（スポーン{}ブロック内、リーフから開始）：
    fetch(url0) ┐
                ├→ 並行実行
    fetch(url1) ┘
    print(a)                       ← スポーン{}ブロック外、順序実行
```

**中核ポイント**：
- ボトムアップ依存分析は `spawn {}` ブロック内部に限定
- スポーン{}ブロック内で依存関係のないタスクは並列実行
- スポーン{}ブロック外のコードは順序実行、スポーン{}ブロック完了を待機

### 6. コンパイルモデル：デュアルバックエンド + 静的リンクランタイム

#### 6.1  дваバックエンド、一つの動作

```
                      ┌─────────────────────┐
                      │   コンパイル前端（共通） │
                      │   Lexer → Parser     │
                      │   → TypeCheck        │
                      │   → スポーン{}ブロック内DAG分析│
                      │   → エスケープ分析     │
                      │   → サイクル検出       │
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │   VMバックエンド（開発）│     │  LLVMバックエンド（本番）│
        │                   │     │                   │
        │  IR/バイトコード生成   │     │  ネイティブコード生成   │
        │  VM解釈実行          │     │  ランタイム静的ライブラリ│
        │  ステップデバッグ対応  │     │  链接        .exe出力   │
        │  クイックイテレーション│     │  ゼロ解釈オーバーヘッド  │
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           動作完全一致                 動作完全一致
```

**VMバックエンド**：開発時に使用。コード修正 → 即時実行 → ステップデバッグ → クイックイテレーション。動作は最終exeと完全に一致。

**LLVMバックエンド**：リリース時に使用。AOTコンパイルでネイティブコード生成、スケジューラを静的ライブラリとしてリンク。解釈器なし、JITなし。

#### 6.2 スケジューラ = 静的ライブラリ、仮想マシンではない

```
最終的なexeの内部構造：

┌────────────────────────────────────────────┐
│  あなたのコード（ネイティブ機械語）           │
│  ├── コンパイル時に確定したDAG実行計画       │
│  ├── インライン化されたMove/ref/clone操作   │
│  └── RAII解放コード                        │
├────────────────────────────────────────────┤
│  ランタイム静的ライブラリ（~200-500KB）      │
│  ├── スレッドプール（固定サイズ = num_workers）│
│  ├── イベントループ（libuv / io_uring）     │
│  ├── ワークスティーリングキュー（Full Runtimeのみ）│
│  ├── メモリアロケータ（jemalloc / mimalloc）│
│  └── リフレクションメタデータ（オンデマンド読み込み）│
├────────────────────────────────────────────┤
│  なし：                                      │
│  ❌ バイトコード解釈器                        │
│  ❌ JITコンパイラ                            │
│  ❌ GC                                       │
│  ❌ 仮想マシン                                │
└────────────────────────────────────────────┘
```

比較：

| 言語 | Java | Go | YaoXiang |
|------|------|-----|-----------|
| コンパイル成果物 | バイトコード | ネイティブコード | ネイティブコード |
| 実行方式 | JVM解釈/JIT | 直接実行 | 直接実行 |
| ランタイムサイズ | ~200MB（JVM） | ~1-2MB（GC含む） | **~200-500KB（GCなし）** |
| メモリ管理 | GC | GC | **RAII（決定論的）** |
| リフレクション | 常駐メモリ | 常駐メモリ | **exe中に保管、オンデマンド読み込み** |

#### 6.3 なぜスケジューラ性能が一定なのか

**重要な洞察**：大部分の工作是コンパイル時に完了し、ランタイムは「実行」のみを行います。

```
コンパイル時（一度だけ、ランタイムには含まない）：
    ├── スポーン{}ブロック内DAG分析：誰が何に依存するか
    ├── トポロジカルソート：スポーン{}ブロック内の実行順序確定
    ├── 並行実行可能なタスク特定：スポーン{}ブロック内依存のないサブツリー
    ├── エスケープ分析：ref → Rc か Arc か
    ├── サイクル検出：自動降格 Weak またはエラー
    └── インライン化：小関数は直接展開

ランタイム（毎回実行、データ構造は固定）：
    ├── コンパイル時に確定したスポーン{}ブロックDAG順序でタスクをスレッドプールにディスパッチ
    ├── I/O遭遇時 → 現在のタスクをサスペンド、イベントループが接管
    ├── タスク準備完了時 → 準備完了キューに戻す
    └── 以上。
```

**スケジューラ自体は固定サイズのデータ構造です**：スレッドプール、イベントループ、ワークキュー。動的成長なし、アダプティブ再最適化なし、GCスキャンなし。動作は完全に予測可能。

コンパイル時にスポーン{}ブロック内の「何をスケジューリングするか」を計算し終えているので、ランタイムは「実行」のみを行います。これはtokioとは異なります——tokioはランタイムでFutureチェーンを動的に構築します。YaoXiangのDAGは静的であり、スポーン{}ブロック内に限定されます。

#### 6.4 リフレクション：保管するが常駐させない

リフレクションメタデータはコンパイル時に生成され、exeの独立したセクション（section）に保管されます。プログラム起動時に読み込みません。リフレクションが初めてリクエストされたとき、オンデマンドでmmapによりメモリにロードされます。例：

```
exeレイアウト：
  .text     ← あなたのコード
  .rodata   ← 定数
  .reflect  ← リフレクションメタデータ（型情報、関数シグネチャ等）
              mmapでオンデマンド読み込み、アクセスしなければメモリ不使用
```

**トレードオフ**：exeサイズ増加（.reflectデータを含む）が、ランタイムでアクセスしなければメモリコストゼロ。初回のアクセスにロード遅延あり（JITウォームアップと同様）、以降ゼロオーバーヘッド。

## トレードオフ

### メリット

- **明確な階層化**：Embedded / Standard / Fullの三層
- **コンパイル再利用**：前端コードは完全に共有
- **ジェネリクス分離**：コンパイル時多相、ゼロオーバーヘッド
- **一貫性**：同期はスケジューリングの特例
- **組込み向け友好**：高性能 + 低メモリ + クイックスタート

### デメリット

- **初期複雑度**：スケジューラインターフェースと複数のランタイムバリアントを定義する必要がある
- **コンパイル時バインディング**：スケジューラ型はコンパイル時に確定
