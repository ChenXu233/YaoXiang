# RFC-008：Runtime の並行処理モデルとスケジューラ分離設計

> **状態**: 採用済み
> **作者**: 晨煦
> **作成日**: 2025-01-05
> **最終更新**: 2026-05-11（プルーニング + 新しいデュアルバックエンドモデル、コンパイラとランタイムの分離、スケジューラ静的ライブラリ、リフレクションの遅延ロード）

> **参考**:
> - [RFC-001: スパン並行モデルとエラー処理システム](./001-concurrent-model-error-handling.md)
> - [RFC-011: ジェネリクス型システム設計](./011-generic-type-system.md)

## 概要

このドキュメントは Runtime アーキテクチャの重要な設計を定義します：

1. **ランタイム三層アーキテクチャ**：Embedded（即時実行） → Standard（DAG スケジューリング） → Full（ワークスティーリング）
2. **コンパイルと実行の分離**：コンパイル段階は完全に同一で、差異は実行時の実行方式のみ
3. **デュアルバックエンドモデル**：VM（開発・ デバッグ用）と LLVM AOT（本番・リリース用）、動作は完全に一致
4. **スケジューラ = 静的ライブラリ**：AOT コンパイル時にスケジューラを exe にリンク、約 200〜500KB、GC なし
5. **同期はスケジューリングの特例**：num_workers=1 で同期モード

### 重要な澄清：これは Java ではない

```
Java:   .java → .class → JVM（解釈/JIT + GC）        ← 常に仮想マシンが必要
YaoXiang 開発: .yx → IR → VM 実行（素早い反復、ステップデバッグ）
YaoXiang 本番: .yx → IR → LLVM → ネイティブ exe（スケジューラを静的ライブラリとしてリンク）

VM は開発ツールであり、ランタイムの本質ではない。Go の go run と go build と同じ関係。
最終的な exe = あなたのネイティブコード + スケジューラ静的ライブラリ + リフレクションメタデータ。
解释器なし、JIT なし、GC なし。
```

## 動機

### コアの矛盾

| 矛盾 | 説明 |
|------|------|
| 透明性 vs 制御性 | 並行処理はデフォルトの動作であるべきだが、ユーザーが制御できる必要がある |
| コア vs オプション | DAG はコアだが、WorkStealing は num_workers>1 の高度な機能 |
| 単一スレッド vs 並行処理 | 単一スレッドモードでは並行処理は非同期として動作し、同期はスケジューリングの特例 |

## 提案

### 1. Runtime 三層アーキテクチャ

```
┌──────────────────────────────────────────────────────────────────┐
│                    コンパイル段階（すべてのモードで同一）           │
│                                                                  │
│  ソースコード → 字句解析 → 構文解析 → 型検査 → コード生成 → IR    │
│                                                                  │
│  ⚠️ 同一の構文解析、型検査、コード生成、IR 出力                    │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即時実行器        │ │ DAG スケジューラ│ │ Full スケジューラ │
│ 同期実行         │ │ 遅延評価      │ │ 並列最適化       │
│ DAG スケジューリングなし│ │ 自動並行処理   │ │ ワークスティーリング│
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| 段階 | Embedded | Standard | Full |
|------|----------|----------|------|
| コンパイル | 同一 | 同一 | 同一 |
| 実行モード | 同期 | 遅延+並行 | 並列 |
| メモリ使用量 | 低 | 中 | 高 |
| 並行処理能力 | なし | 自動 | 自動+並列 |
| DAG 遅延評価 | なし | ✅ | ✅ |
| WorkStealer | なし | なし | ✅ |

**Embedded Runtime**：ターゲット WASM/ゲームスクリプト/ルールエンジン。即時実行器、DAG なし、高性能低使用量。

**Standard Runtime**：ターゲット Web サービス/データパイプライン。DAG 遅延評価 + 自動並行処理。num_workers=1 で単一スレッド非同期。

**Full Runtime**：ターゲット 科学計算/大規模並列処理。Standard + WorkStealer 負荷分散。

### 2. スケジューラ分離：ジェネリクス + 注入

コア原則：VM は具体的なスケジューラに直接依存せず、ジェネリックパラメータ `[S]` を通じて呼び出します。

```yaoxiang
# スケジューラインターフェース定義
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# 単一スレッドスケジューラ
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# マルチスレッドスケジューラ
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VM はジェネリクスでスケジューラを使用
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**コアポイント**：
- コンパイル時多相、ゼロ実行時オーバーヘッド
- トレイトオブジェクト不要
- ジェネリクス型制約 `[S: Scheduler]` は RFC-011 で既に定義済み

### 3. 同期 = スケジューリングの特例

```
❌ 誤解：スケジューラを無効化する
✅ 正しい：単一ワーカーのスケジューラを使用

num_workers = 1 → 単一スレッド非同期スケジューリング
num_workers > 1 → マルチスレッド並列スケジューリング

同一のスケジューラインターフェース、設定が異なるだけ。特例を排除。
```

### 4. DAG の立場

| 階層 | DAG 含む | 説明 |
|------|----------|------|
| Core Runtime | ✅ | 遅延評価コア |
| Standard Runtime | ✅ | DAG スケジューラ |
| Embedded Runtime | ❌ | 即時実行、DAG なし |

### 5. 下而上実行モデル

```
ユーザーコード（同期構文）：
    a = fetch(url0)
    b = fetch(url1)
    print(a)

コンパイル時分析（下而上）：
    print(a) は a が必要 → fetch(url0) に依存
    fetch(url1) は誰にも必要とされない → 孤立 DAG

実行時スケジューリング（葉から開始）：
    fetch(url0) → print(a)    ← 依存チェーン、順序付き
    fetch(url1)                ← 孤立、獨立で並列
```

**コアポイント**：
- 「結果が必要な箇所」から逆に依存関係を分析
- 葉ノードを優先して並列実行
- 孤立 DAG は独立して並列実行し、メインフローをブロックしない

### 6. コンパイルモデル：デュアルバックエンド + 静的リンクランタイム

#### 6.1 2つのバックエンド、同一の動作

```
                      ┌─────────────────────┐
                      │   コンパイルフロントエンド（統一）│
                      │   字句解析 → 構文解析     │
                      │   → 型検査              │
                      │   → DAG 分析            │
                      │   → エスケープ分析        │
                      │   → 循環検出            │
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │   VM バックエンド（開発）│     │  LLVM バックエンド（本番）│
        │                   │     │                   │
        │  IR/バイトコード生成   │     │  ネイティブコード生成    │
        │  VM 解释執行        │     │  ランタイム静的ライブラリリンク│
        │  ステップデバッグ対応  │     │  .exe 出力            │
        │  快速迭代          │     │  解释オーバーヘッドゼロ  │
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           動作完全一致                 動作完全一致
```

**VM バックエンド**：開発時に使用。コード変更 → 即時実行 → ステップデバッグ → 快速反復。動作は最終 exe と完全に一致。

**LLVM バックエンド**：リリース時に使用。AOT コンパイルでネイティブコード生成、スケジューラを静的ライブラリとしてリンク。解释器なし、JIT なし。

#### 6.2 スケジューラ = 静的ライブラリ、不是虚拟机

```
最終 exe の内部構造：

┌────────────────────────────────────────────┐
│  あなたのコード（ネイティブ機械語）              │
│  ├── コンパイル時に確定した DAG 実行プラン       │
│  ├── インライン化された Move/ref/clone 操作     │
│  └── RAII 解放コード                          │
├────────────────────────────────────────────┤
│  ランタイム静的ライブラリ（~200-500KB）           │
│  ├── スレッドプール（固定サイズ = num_workers）  │
│  ├── イベントループ（libuv / io_uring）         │
│  ├── ワークスティーリングキュー（Full Runtime のみ）│
│  ├── メモリアロケータ（jemalloc / mimalloc）   │
│  └── リフレクションメタデータ（遅延ロード、常駐なし）│
├────────────────────────────────────────────┤
│  なし：                                      │
│  ❌ バイトコード解释器                         │
│  ❌ JIT コンパイラ                            │
│  ❌ GC                                      │
│  ❌ 虚拟机                                    │
└────────────────────────────────────────────┘
```

比較：

| | Java | Go | YaoXiang |
|------|------|-----|-----------|
| コンパイル成果物 | バイトコード | ネイティブコード | ネイティブコード |
| 実行方式 | JVM 解释/JIT | 直接実行 | 直接実行 |
| ランタイムサイズ | ~200MB（JVM） | ~1-2MB（GC 含む） | **~200-500KB（GC なし）** |
| メモリ管理 | GC | GC | **RAII（確定的）** |
| リフレクション | 常駐メモリ | 常駐メモリ | **exe 中储存、遅延ロード** |

#### 6.3 スケジューラ性能が一定な理由

**重要な洞察**：大部分の作業はコンパイル時に完了し、実行時は「実行」のみを行います。

```
コンパイル時（一度だけ、ランタイムに含まれない）：
    ├── グローバル DAG 構築：誰が誰に依存するか
    ├── トポロジカルソート：実行順序の確定
    ├── 孤立識別：並列可能な部分木
    ├── エスケープ分析：ref → Rc か Arc か
    ├── 循環検出：自動降格 Weak またはエラー
    └── インライン化：小規模関数は直接展開

実行時（毎実行、データ構造は固定）：
    ├── コンパイル時に確定した DAG 順序でタスクをスレッドプールに分配
    ├── I/O 遭遇時 → 現在タスクをサスペンド、イベントループが接管
    ├── タスク準備完了時 → 準備キューに戻す
    └── これだけ。
```

**スケジューラ自体は固定サイズのデータ構造**：スレッドプール、イベントループ、ワークキュー。動的成長なし、適応的再最適化なし、GC スキャンなし。動作は完全に予測可能。

コンパイル時に「何をスケジューリングするか」を既に計算終わっているので、実行時は「実行」のみを行います。これは tokio とは異なります——tokio は実行時に Future チェーンを動的に構築します。YaoXiang の DAG は静的です。

#### 6.4 リフレクション：储存不常驻

リフレクションメタデータはコンパイル時に生成され、exe の独立したセグメント（セクション）に储存されます。プログラム起動時にロードされません。リフレクションが初めてリクエストされたときだけ、遅延して mmap でメモリにロードされます。例えるなら：

```
exe レイアウト：
  .text     ← あなたのコード
  .rodata   ← 定数
  .reflect  ← リフレクションメタデータ（型情報、関数シグネチャなど）
              mmap 遅延ロード、不アクセスならメモリ不使用
```

**トレードオフ**：exe 体积增大（リフレクションデータを含む）が、実行時にアクセスしなければメモリコストはゼロ。初回アクセスにロード遅延あり（JIT 预热类似）、以降ゼロオーバーヘッド。

## トレードオフ

### メリット

- **明確な階層化**：Embedded / Standard / Full の三層
- **コンパイル再利用**：フロントエンドコードは完全に共用
- **ジェネリクス分離**：コンパイル時多相、ゼロオーバーヘッド
- **一貫性**：同期はスケジューリングの特例
- **組込みに優しい**：高性能 + 低メモリ + 高速起動

### デメリット

- **初期複雑度**：スケジューラインターフェースと複数のランタイムバリアントを定義する必要がある
- **コンパイル時バインディング**：スケジューラ型はコンパイル時に確定
