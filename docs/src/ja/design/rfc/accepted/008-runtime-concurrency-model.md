# RFC-008：Runtime並行モデルとスケジューラ分離設計

> **状態**: 採用済み
> **著者**: 晨煦
> **作成日**: 2025-01-05
> **最終更新**: 2026-05-11（整形＋新規追加：双バックエンドモデル、コンパイラとRuntimeの分離、スケジューラ静的ライブラリ、リフレクションオンデマンドロード）

> **参考**:
> - [RFC-001: 並作モデルとエラー処理システム](./001-concurrent-model-error-handling.md)
> - [RFC-011: ジェネリクスシステム設計](./011-generic-type-system.md)

## 概要

本ドキュメントはRuntimeアーキテクチャの重要な設計を定義する：

1. **Runtime三層アーキテクチャ**：Embedded（即時実行） → Standard（DAGスケジューリング） → Full（ワークスチール）
2. **コンパイルと実行の分離**：コンパイル段階は完全同一、違いはRuntime実行方式のみ
3. **双バックエンドモデル**：VM（開発デバッグ）とLLVM AOT（本番リリース）、動作は完全一致
4. **スケジューラ = 静的ライブラリ**：AOTコンパイル時にスケジューラをexeにリンク、約200-500KB、GCなし
5. **同期はスケジューリングの特例**：num_workers=1 で同期モード

### 重要な澄清：これはJavaではない

```
Java:   .java → .class → JVM（解釈/JIT + GC）        ← 常時VMが必要
YaoXiang 開発: .yx → IR → VM 実行（高速イテレーション、ステップデバッグ）
YaoXiang 本番: .yx → IR → LLVM → ネイティブexe（スケジューラをリンク）

VM は開発ツールであり、Runtimeの本質ではない。Go の go run vs go build と同じ。
最終exe = あなたのネイティブコード + スケジューラ静的ライブラリ + リフレクションメタデータ。
インタプリタなし、JITなし、GCなし。
```

## 動機

### コアな矛盾

| 矛盾 | 説明 |
|------|------|
| 透過性 vs 制御性 | 並行はデフォルト動作であるべきだが、ユーザーは制御できるべき |
| コア vs オプション | DAGはコアだが、WorkStealingはnum_workers>1の高度な機能 |
| シングルスレッド vs 並行 | シングルスレッドモードでは並髪が非同期として動作、同期はスケジューリングの特例 |

## 提案

### 1. Runtime三層アーキテクチャ

```
┌──────────────────────────────────────────────────────────────────┐
│                    コンパイル段階（全モード共通）                   │
│                                                                  │
│  Source Code → Lexer → Parser → TypeCheck → Codegen → IR        │
│                                                                  │
│  ⚠️ 同一の構文解析、型チェック、コード生成、IR出力                  │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即時エグゼキュータ   │ │ DAGスケジューラ  │ │ Fullスケジューラ  │
│ 同期実行          │ │ 遅延評価        │ │ 並列最適化        │
│ DAGスケジューリングなし│ │ 自動並行        │ │ ワークスチール     │
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| 段階 | Embedded | Standard | Full |
|------|----------|----------|------|
| コンパイル | 共通 | 共通 | 共通 |
| 実行モード | 同期 | 遅延+並行 | 並列 |
| メモリ使用量 | 低 | 中 | 高 |
| 並行能力 | なし | 自動 | 自動+並列 |
| DAG遅延評価 | なし | ✅ | ✅ |
| WorkStealer | なし | なし | ✅ |

**Embedded Runtime**：対象はWASM/ゲームスクリプト/ルールエンジン。即時エグゼキュータ、DAGなし、高性能低占用。

**Standard Runtime**：対象はWebサービス/データパイプライン。DAG遅延評価 + 自動並行。num_workers=1でシングルスレッド非同期。

**Full Runtime**：対象は科学計算/大規模並列。Standard + WorkStealerによるロードバランシング。

### 2. スケジューラ分離：ジェネリクス + 注入

コア原則：VMは具体スケジューラを直接依存せず、ジェネリックパラメータ `[S]` を通じて呼び出す。

```yaoxiang
# スケジューラインターフェース定義
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# シングルスレッドスケジューラ
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# マルチスレッドスケジューラ
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VMはジェネリクスでスケジューラを使用
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**コアポイント**：
- コンパイル時多相、ゼロランタイムオーバーヘッド
- Traitオブジェクト不要
- ジェネリック型制約 `[S: Scheduler]` はRFC-011で既に定義済み

### 3. 同期 = スケジューリングの特例

```
❌ 誤解：スケジューラを無効化する
✅ 正しい：単一workerのスケジューラを使用する

num_workers = 1 → シングルスレッド非同期スケジューリング
num_workers > 1 → マルチスレッド並列スケジューリング

同一のスケジューラインターフェース、設定だけが異なる。特例を排除。
```

### 4. DAGの地位

| レイヤー | DAG包含 | 説明 |
|----------|---------|------|
| Core Runtime | ✅ | 遅延評価コア |
| Standard Runtime | ✅ | DAGスケジューラ |
| Embedded Runtime | ❌ | 即時実行、DAGなし |

### 5. ボトムアップ実行モデル

```
ユーザーコード（同期構文）：
    a = fetch(url0)
    b = fetch(url1)
    print(a)

コンパイル時解析（ボトムアップ）：
    print(a) は a を必要とする → fetch(url0) に依存
    fetch(url1) は誰にも必要とされない → 孤立DAG

Runtimeスケジューリング（叶子から開始）：
    fetch(url0) → print(a)    ← 依存チェーン、順序通り
    fetch(url1)                ← 孤立、獨立して並行
```

**コアポイント**：
- 「結果が必要な場所」から逆向的に依存関係を解析
- 叶子ノードを優先して並列実行
- 孤立DAGは獨立して並列、主フローをブロックしない

### 6. コンパイルモデル：双バックエンド + 静的リンクRuntime

#### 6.1  два後端、一つの動作

```
                      ┌─────────────────────┐
                      │   コンパイルフロントエンド（共通）│
                      │   Lexer → Parser     │
                      │   → TypeCheck         │
                      │   → DAG解析           │
                      │   →  エスケープ解析     │
                      │   → 環検出            │
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │   VMバックエンド（開発）│     │  LLVMバックエンド（本番） │
        │                   │     │                   │
        │  IR/バイトコード生成   │     │  ネイティブコード生成    │
        │  VM解釈実行          │     │  Runtime静的ライブラリリンク│
        │  ステップデバッグ対応   │     │  .exe出力             │
        │  高速イテレーション     │     │  ゼロ解釈オーバーヘッド   │
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           動作完全一致                 動作完全一致
```

**VMバックエンド**：開発時に使用。コード修正 → 即時実行 → ステップデバッグ → 高速イテレーション。動作は最終exeと完全一致。

**LLVMバックエンド**：リリース時に使用。AOTコンパイルでネイティブコード生成、スケジューラを静的ライブラリとしてリンク。インタプリタなし、JITなし。

#### 6.2 スケジューラ = 静的ライブラリ、虚拟机ではない

```
最終exeの内部構造：

┌────────────────────────────────────────────┐
│  あなたのコード（ネイティブ機械語）               │
│  ├── コンパイル期に確定したDAG実行計画           │
│  ├── インライン化されたMove/ref/clone操作      │
│  └── RAII解放コード                          │
├────────────────────────────────────────────┤
│  Runtime静的ライブラリ（~200-500KB）            │
│  ├── スレッドプール（固定サイズ = num_workers）  │
│  ├── イベントループ（libuv / io_uring）        │
│  ├── ワークスチールキュー（Full Runtimeのみ）   │
│  ├── メモリアロケータ（jemalloc / mimalloc）  │
│  └── リフレクションメタデータ（オンデマンドロード、常駐なし）│
├────────────────────────────────────────────┤
│  なし：                                       │
│  ❌ バイトコードインタプリタ                      │
│  ❌ JITコンパイラ                               │
│  ❌ GC                                        │
│  ❌ 虚拟机                                      │
└────────────────────────────────────────────┘
```

比較：

| | Java | Go | YaoXiang |
|------|------|-----|-----------|
| コンパイル产物 | バイトコード | ネイティブコード | ネイティブコード |
| 実行方式 | JVM解釈/JIT | 直接実行 | 直接実行 |
| Runtimeサイズ | ~200MB（JVM） | ~1-2MB（GC含む） | **~200-500KB（GCなし）** |
| メモリ管理 | GC | GC | **RAII（决定的）** |
| リフレクション | 常時メモリ常駐 | 常時メモリ常駐 | **exe中に保存、オンデマンド** |

#### 6.3 なぜスケジューラ性能が一定か

**重要な洞察**：大部分の作業はコンパイル期に完了し、Runtimeは「実行」のみを行う。

```
コンパイル期（一度だけ、Runtimeには含まれない）：
    ├── グローバルDAG構築：誰が何に依存するか
    ├── トポロジカルソート：実行順序確定
    ├── 孤立DAG識別：並列可能な部分木
    ├── エスケープ解析：ref → Rc か Arc か
    ├── 環検出：自動デグラードWeakまたはエラー
    └── インライン：小関数は直接展開

Runtime（毎回実行、データ構造は固定）：
    ├── コンパイル期に確定したDAG順序でスレッドプールにタスクを分配
    ├── I/O遭遇 → 現在のタスクをサスペンド、イベントループが接管
    ├── タスク準備完了 → 準備完了キューに戻す
    └── 以上の繰り返し。
```

**スケジューラ自体は固定サイズのデータ構造**：スレッドプール、イベントループ、ワークキュー。動的成長なし、アダプティブ再最適化なし、GC走査なし。動作は完全に予測可能。

コンパイル期已经把「何をスケジューリングするか」を計算完了し、Runtimeは「実行」のみを行う。これはtokioとは異なる——tokioはRuntimeで動的にFutureチェーンを構築する。YaoXiangのDAGは静的である。

#### 6.4 リフレクション：保存するが常駐させない

リフレクションメタデータはコンパイル期に生成され、exeの獨立セクション（section）に保存される。プログラム起動時にはロードされない。最初のリフレクション要求時に、オンデマンドでmmapによりメモリにロードされる。例：

```
exeレイアウト：
  .text     ← あなたのコード
  .rodata   ← 定数
  .reflect  ← リフレクションメタデータ（型情報、関数シグネチャ等）
              mmapでオンデマンドロード、アクセスしなければメモリ使用なし
```

**トレードオフ**：exeサイズ增大（リフレクションデータ含む）が、Runtimeでアクセスしなければメモリオーバーヘッドゼロ。初回アクセス時にロード遅延あり（JITウォームアップ类似）、以降はゼロオーバーヘッド。

## トレードオフ

### メリット

- **明確なレイヤー化**：Embedded / Standard / Full の三層
- **コンパイル再利用**：フロントエンドコードは完全共有
- **ジェネリック分離**：コンパイル時多相、ゼロオーバーヘッド
- **一貫性**：同期はスケジューリングの特例
- **組込み向け設計**：高性能 + 低メモリ + 高速起動

### デメリット

- **初期複雑度**：スケジューラインターフェースと複数のRuntimeバリアントを定義する必要がある
- **コンパイル時バインディング**：スケジューラ型はコンパイル期に確定
