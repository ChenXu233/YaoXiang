# RFC-008：Runtime 並行モデルとスケジューラの疎結合設計

> **⚠️ 整合に関する説明**：本文書は [RFC-024 新並行モデル](/reference/language-spec/concurrency.md) と整合済みです。旧来の全プログラム DAG 解析、`@block`/`@eager` 注釈、L1/L2/L3 階層モデルは `spawn {}` ブロック並列プリミティブに置き換えられました。DAG 解析は現在は `spawn {}` ブロック内部のみを対象とします。

> **参考**:
> - [RFC-011: ジェネリクスシステム設計](./011-generic-type-system.md)
> - [並行モデル仕様（RFC-024）](/reference/language-spec/concurrency.md)

## 概要

本文書は Runtime アーキテクチャの重要な設計を定義します：

1. **ランタイム三層アーキテクチャ**：Embedded（即時実行） → Standard（spawn + DAG スケジューリング） → Full（ワークスティーリング）
2. **コンパイルと実行の分離**：コンパイル段階は完全に同一で、相違はランタイム実行方式のみ
3. **デュアルバックエンドモデル**：VM（開発デバッグ）と LLVM AOT（本番リリース）、動作は完全に一致
4. **スケジューラ = 静的ライブラリ**：AOT コンパイル時にスケジューラを exe にリンク、約 200-500KB、GC なし
5. **同期はスケジューリングの特例に過ぎない**：num_workers=1 が同期モード

### 重要事項の明確化：これは Java ではない

```
Java:    .java → .class → JVM（解釈/JIT + GC）         ← 常に仮想マシンが必須
YaoXiang 開発: .yx → IR → VM 実行（迅速な反復、ステップデバッグ）
YaoXiang 本番: .yx → IR → LLVM → ネイティブ exe（スケジューラが静的リンクされる）

VM は開発ツールであり、ランタイムの本質ではない。Go の go run vs go build と同じ。
最終 exe = あなたのネイティブコード + スケジューラ静的ライブラリ + リフレクションメタデータ。
インタプリタも JIT も GC もない。
```

## 動機

### 核心的な矛盾

| 矛盾 | 説明 |
|------|------|
| 透明性 vs 制御性 | spawn ブロックは明示的な並行制御を提供し、通常のコードは逐次実行される |
| コア vs オプション | spawn はコア並列プリミティブ、WorkStealing は num_workers>1 の高度な特性 |
| シングルスレッド vs 並行 | シングルスレッドモードでは並行は非同期として現れ、同期はスケジューリングの特例に過ぎない |

## 提案

### 1. Runtime 三層アーキテクチャ

```
┌──────────────────────────────────────────────────────────────────┐
│                  コンパイル段階（全モードで同一）                  │
│                                                                  │
│  Source Code → Lexer → Parser → TypeCheck → Codegen → IR        │
│                                                                  │
│  ⚠️ 同一の構文解析、型検査、コード生成、IR 出力                     │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即時実行器        │ │ spawn + DAG   │ │ Full スケジューラ  │
│ 同期実行         │ │ spawn ブロック内並列│ │ 並列最適化       │
│ spawn 非対応     │ │ 自動並列化      │ │ ワークスティーリング│
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| 段階 | Embedded | Standard | Full |
|------|----------|----------|------|
| コンパイル | 同一 | 同一 | 同一 |
| 実行モード | 同期 | spawn ブロック内並列 | 並列 |
| メモリ使用量 | 低 | 中 | 高 |
| 並行能力 | なし | spawn ブロック内 | spawn ブロック内 + 並列 |
| spawn サポート | ❌ | ✅ | ✅ |
| DAG 解析 | なし | spawn ブロック内 | spawn ブロック内 |
| WorkStealer | なし | なし | ✅ |

**Embedded Runtime**：WASM/ゲームスクリプト/ルールエンジン向け。即時実行器、spawn 非対応、高性能・低フットプリント。

**Standard Runtime**：Web サービス/データパイプライン向け。`spawn {}` ブロックをサポートし、spawn ブロック内で DAG 解析と自動並列化を実施。num_workers=1 がシングルスレッド非同期。

**Full Runtime**：科学計算/大規模並列処理向け。Standard + WorkStealer による負荷分散。

### 2. スケジューラの疎結合：ジェネリクス + 注入

基本原則：VM は具体的なスケジューラに直接依存せず、ジェネリックパラメータ `[S]` を通じて呼び出す。

```yaoxiang
# スケジューラインターフェース定義
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# シングルスレッドスケジューラ
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# マルチスレッドスケジューラ
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VM はジェネリクスを通じてスケジューラを使用
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**核心ポイント**：
- コンパイル時ポリモーフィズム、ランタイムオーバーヘッドゼロ
- Trait オブジェクト不要
- ジェネリック型制約 `[S: Scheduler]` は RFC-011 で既に定義済み

### 3. 同期 = スケジューリングの特例

```
❌ 誤解：スケジューラを無効化する
✅ 正しい：単一 worker のスケジューラを使用する

num_workers = 1 → シングルスレッド非同期スケジューリング
num_workers > 1 → マルチスレッド並列スケジューリング

同一のスケジューラインターフェース、設定のみが異なる。特殊ケースを排除。
```

### 4. DAG の位置づけ

> **重要な変更**：DAG 解析は全プログラムを対象とせず、`spawn {}` ブロック内部でのみ実施される。通常のコード（spawn ブロック外）は逐次実行され、DAG 解析は不要。

| 階層 | spawn サポート | DAG 解析範囲 | 説明 |
|------|---------------|-------------|------|
| Core Runtime | ✅ | spawn ブロック内 | 並行コア |
| Standard Runtime | ✅ | spawn ブロック内 | spawn + DAG スケジューリング |
| Embedded Runtime | ❌ | なし | 即時実行、並行なし |

### 5. ボトムアップ実行モデル（spawn ブロック内）

> **重要な変更**：ボトムアップ DAG 解析は `spawn {}` ブロック内部でのみ実施され、全プログラムに対する DAG 解析は廃止された。

```
ユーザーコード（spawn ブロック内並列）：
    (a, b) = spawn {
        fetch(url0),
        fetch(url1)
    }
    print(a)

コンパイル時解析（spawn ブロック内ボトムアップ）：
    fetch(url0) と fetch(url1) は相互依存なし → 並列実行可能
    spawn ブロック外の print(a) → 逐次実行、spawn の完了を待機

ランタイムスケジューリング（spawn ブロック内は葉から開始）：
    fetch(url0) ┐
                ├→ 並列実行
    fetch(url1) ┘
    print(a)                       ← spawn ブロック外、逐次実行
```

**核心ポイント**：
- ボトムアップ依存解析は `spawn {}` ブロック内部のみ
- spawn ブロック内の依存関係なしタスクは並列実行
- spawn ブロック外のコードは逐次実行され、spawn ブロックの完了を待機

### 6. コンパイルモデル：デュアルバックエンド + ランタイムの静的リンク

#### 6.1 2 つのバックエンド、1 つの動作

```
                      ┌─────────────────────┐
                      │  コンパイルフロントエンド（統一）│
                      │  Lexer → Parser     │
                      │  → TypeCheck        │
                      │  → spawn ブロック内 DAG 解析│
                      │  → エスケープ解析    │
                      │  → サイクル検出      │
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │  VM バックエンド   │     │ LLVM バックエンド  │
        │  （開発）         │     │ （本番）           │
        │                   │     │                   │
        │  IR/バイトコード生成│     │  ネイティブコード生成│
        │  VM 解釈実行       │     │  ランタイム静的ライブラリ│
        │  ステップデバッグ対応│    │  へのリンク       │
        │  迅速な反復       │     │  .exe 出力        │
        │                   │     │  解釈オーバーヘッドゼロ│
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           動作は完全に一致          動作は完全に一致
```

**VM バックエンド**：開発時に使用。コード変更 → 即時実行 → ステップデバッグ → 迅速な反復。動作は最終 exe と完全に一致。

**LLVM バックエンド**：リリース時に使用。ネイティブコードへ AOT コンパイル、スケジューラを静的ライブラリとしてリンク。インタプリタも JIT もない。

#### 6.2 スケジューラ = 静的ライブラリ、仮想マシンではない

```
最終 exe の内部構造：

┌────────────────────────────────────────────┐
│  あなたのコード（ネイティブマシンコード）      │
│  ├── コンパイル時に決定済みの DAG 実行計画   │
│  ├── インライン化された Move/ref/clone 操作 │
│  └── RAII 解放コード                        │
├────────────────────────────────────────────┤
│  ランタイム静的ライブラリ（~200-500KB）       │
│  ├── スレッドプール（固定サイズ = num_workers）│
│  ├── イベントループ（libuv / io_uring）     │
│  ├── ワークスティーリングキュー（Full Runtime のみ）│
│  ├── メモリアロケータ（jemalloc / mimalloc）│
│  └── リフレクションメタデータ（オンデマンド読み込み、│
│      常駐しない）                            │
├────────────────────────────────────────────┤
│  存在しないもの：                            │
│  ❌ バイトコードインタプリタ                 │
│  ❌ JIT コンパイラ                           │
│  ❌ GC                                      │
│  ❌ 仮想マシン                                │
└────────────────────────────────────────────┘
```

比較：

| 言語 | Java | Go | YaoXiang |
|------|------|-----|-----------|
| コンパイル成果物 | バイトコード | ネイティブコード | ネイティブコード |
| 実行方式 | JVM 解釈/JIT | 直接実行 | 直接実行 |
| ランタイムサイズ | ~200MB（JVM） | ~1-2MB（GC 含む） | **~200-500KB（GC なし）** |
| メモリ管理 | GC | GC | **RAII（確定的）** |
| リフレクション | 常駐メモリ | 常駐メモリ | **exe 内に格納、オンデマンド読み込み** |

#### 6.3 なぜスケジューラの性能が一定なのか

**重要な洞察**：作業の大部分はコンパイル時に完了し、ランタイムは「実行」のみを行う。

```
コンパイル時（一度だけ、ランタイムには含まれない）：
    ├── spawn ブロック内 DAG 解析：誰が誰に依存するか
    ├── トポロジカルソート：spawn ブロック内の実行順序決定
    ├── 並列可能タスクの識別：spawn ブロック内依存関係なしサブツリー
    ├── エスケープ解析：ref → Rc か Arc か
    ├── サイクル検出：Weak への自動降格またはエラー
    └── インライン化：小関数の直接展開

ランタイム（毎回実行、データ構造は固定）：
    ├── コンパイル時に決定された spawn ブロック内 DAG 順序で │
    │   タスクをスレッドプールにディスパッチ
    ├── I/O 発生時 → 現在のタスクをサスペンド、イベントループが引き継ぎ
    ├── タスク準備完了 → レディキューに戻す
    └── 以上。
```

**スケジューラ自体は固定サイズのデータ構造**：スレッドプール、イベントループ、ワークキュー。動的拡張なし、自適応再最適化なし、GC スキャンなし。動作は完全に予測可能。

コンパイル時に spawn ブロック内「何をスケジューリングするか」が全て計算済みで、ランタイムは「実行」のみ。これは tokio と異なる——tokio はランタイムで動的に Future チェーンを構築する。YaoXiang の DAG は静的であり、spawn ブロック内部のみに限定される。

#### 6.4 リフレクション：格納するが常駐しない

リフレクションメタデータはコンパイル時に生成され、exe の独立セクションに格納される。プログラム起動時には読み込まれない。リフレクションが最初に要求された時点で、オンデマンドで mmap によりメモリに読み込まれる。以下のイメージ：

```
exe レイアウト：
  .text     ← あなたのコード
  .rodata   ← 定数
  .reflect  ← リフレクションメタデータ（型情報、関数シグネチャ等）
              mmap オンデマンド読み込み、アクセスしなければメモリ消費なし
```

**トレードオフ**：exe サイズが大きくなる（リフレクションデータを含む）が、ランタイムでアクセスしなければメモリオーバーヘッドゼロ。初回アクセス時にロード遅延あり（JIT ウォームアップに類似）、以降はオーバーヘッドゼロ。

## トレードオフ

### 利点

- **明確な階層化**：Embedded / Standard / Full の三層
- **コンパイルの再利用**：フロントエンドコードは完全に共有
- **ジェネリクスによる疎結合**：コンパイル時ポリモーフィズム、オーバーヘッドゼロ
- **一貫性**：同期はスケジューリングの特例に過ぎない
- **組込みに優しい**：高性能 + 低メモリ + 迅速な起動

### 欠点

- **初期複雑度**：スケジューラインターフェースと複数のランタイムバリアントの定義が必要
- **コンパイル時バインディング**：スケジューラ型はコンパイル時に決定

| 三層アーキテクチャ更新 | Embedded は spawn 非対応、Standard は spawn 対応 | 2026-06-05 |
