# RFC-008：Runtime同時実行モデルとスケジューラ分離設計

> **状態**: 採用済み
> **著者**: 晨煦
> **作成日**: 2025-01-05
> **最終更新**: 2026-05-11（刈り込み + 新しいデュアルバックエンドモデル、コンパイラとランタイムの分離、スケジューラ静的ライブラリ、リフレクションオンデマンドロード）

> **参照**:
> - [RFC-001: 同時実行モデルとエラー処理システム](./001-concurrent-model-error-handling.md)
> - [RFC-011: ジェネリクスシステム設計](./011-generic-type-system.md)

## 概要

本文書はRuntimeアーキテクチャの重要な設計を定義します：

1. **ランタイム三層アーキテクチャ**：Embedded（即時実行） → Standard（DAGスケジューリング） → Full（ワークスティール）
2. **コンパイルと実行の分離**：コンパイル段階は完全に同一、違いは実行時の実行方式のみ
3. **デュアルバックエンドモデル**：VM（開発デバッグ） と LLVM AOT（本番リリース）、動作は完全に一致
4. **スケジューラ = 静的ライブラリ**：AOTコンパイル時にスケジューラがexeにリンク、約200-500KB、GCなし
5. **同期はスケジューリングの特例に過ぎない**：num_workers=1 で同期モード

### 重要な澄清：これはJavaではない

```
Java:   .java → .class → JVM（解釈/JIT + GC）        ← 常に仮想マシンが必要
YaoXiang 開発: .yx → IR → VM 実行（素早いイテレーション、ステップデバッグ）
YaoXiang 本番: .yx → IR → LLVM → ネイティブ exe（スケジューラがリンクされる）

VM は開発ツールであり、ランタイムの本質ではない。Go の go run vs go build と同じ。
最終的な exe = あなたのネイティブコード + スケジューラ静的ライブラリ + リフレクションメタデータ。
解释器なし、JITなし、GCなし。
```

## 動機

### 中核的矛盾

| 矛盾 | 説明 |
|------|------|
| 透明性 vs 制御性 | 並行はデフォルトの動作であるべきだが、ユーザーは制御できるべき |
| 中核 vs 任意 | DAGは中核だが、WorkStealingはnum_workers>1の高度な機能 |
| シングルスレッド vs 並行 | シングルスレッドモードでは並行的動作は非同期として、DAGの動的生成は同期の特例 |

## 提案

### 1. Runtime三層アーキテクチャ

```
┌──────────────────────────────────────────────────────────────────┐
│                    コンパイル段階（全モード共通）                   │
│                                                                  │
│  ソースコード → Lexer → Parser → TypeCheck → Codegen → IR       │
│                                                                  │
│  ⚠️ 同一の構文解析、型チェック、コード生成、IR出力                  │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即時エグゼキュータ  │ │ DAGスケジューラ  │ │ Fullスケジューラ  │
│ 同期実行          │ │ 遅延評価        │ │ 並列最適化       │
│ DAGスケジューリングなし│ │ 自動並行的       │ │ ワークスティール │
└──────────────────┘ └───────────────┘ └───────────────┘
```

| 段階 | Embedded | Standard | Full |
|------|----------|----------|------|
| コンパイル | 共通 | 共通 | 共通 |
| 実行モード | 同期 | 遅延+並行的 | 並列 |
| メモリ使用量 | 低 | 中 | 高 |
| 並行能力 | なし | 自動 | 自動+並列 |
| DAG遅延評価 | なし | ✅ | ✅ |
| WorkStealer | なし | なし | ✅ |

**Embedded Runtime**：目標はWASM/ゲームスクリプト/ルールエンジン。即時エグゼキュータ、DAGなし、高性能低メモリ。

**Standard Runtime**：目標はWebサービス/データパイプライン。DAG遅延評価 + 自動並行的。num_workers=1でシングルスレッド非同期。

**Full Runtime**：目標は科学計算/大規模並列。Standard + WorkStealer負荷分散。

### 2. スケジューラ分離：ジェネリクス + 注入

基本原则：VMは具体的なスケジューラに直接依存せず、ジェネリクスパラメータ `[S]` を通じて呼び出す。

```yaoxiang
# スケジューラインターフェース定義
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# シングルスレッドスケジューラ
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# マルチスレッドスケジューラ
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VMはジェネリクスでスケジューラを使用
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**核心ポイント**：
- コンパイル時多相、ゼロ実行時オーバーヘッド
- トレイトオブジェクト不要
- ジェネリクス型制約 `[S: Scheduler]` はRFC-011で既に定義済み

### 3. 同期 = スケジューリングの特例

```
❌ 誤解：スケジューラを無効化する
✅ 正しい：単一ワーカーのスケジューラを使用する

num_workers = 1 → シングルスレッド非同期スケジューリング
num_workers > 1 → マルチスレッド並列スケジューリング

同じスケジューラインターフェース、設定が異なるだけ。特例を排除。
```

### 4. DAGの地位

| 階層 | DAG 포함 | 説明 |
|------|----------|------|
| Core Runtime | ✅ | 遅延評価コア |
| Standard Runtime | ✅ | DAGスケジューラ |
| Embedded Runtime | ❌ | 即時実行、DAGなし |

### 5. ボトムアップ実行モデル

```
ユーザーコード（同期構文）：
    a = fetch(url0)
    b = fetch(url1)
    print(a)

コンパイル時分析（ボトムアップ）：
    print(a) は a が必要 → fetch(url0) に依存
    fetch(url1) は誰も必要としていない → 孤島DAG

実行時スケジューリング（葉から開始）：
    fetch(url0) → print(a)    ← 依存チェーン、順序通り
    fetch(url1)                ← 孤島、独立並列
```

**核心ポイント**：
- 「結果が必要な場所」から逆方向に依存関係を分析
- 葉ノードを優先して並列実行
- 孤島DAGは独立して並列、主フローをブロックしない

### 6. コンパイルモデル：デュアルバックエンド + 静的リンクランタイム

#### 6.1 2つのバックエンド、1つの動作

```
                      ┌─────────────────────┐
                      │   コンパイルフロントエンド（統一） │
                      │   Lexer → Parser     │
                      │   → TypeCheck        │
                      │   → DAG 分析         │
                      │   → エスケープ分析    │
                      │   → サイクル検出      │
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │   VM バックエンド（開発）│     │  LLVM バックエンド（本番）│
        │                   │     │                   │
        │  IR/バイトコード生成  │     │  ネイティブコード生成    │
        │  VM解釈実行          │     │  ランタイム静的ライブラリリンク│
        │  ステップデバッグ対応 │     │  .exe出力              │
        │  素早いイテレーション  │     │  ゼロ解釈オーバーヘッド  │
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           動作完全一致                 動作完全一致
```

**VM バックエンド**：開発時に使用。コード修正 → 即時実行 → ステップデバッグ → 素早いイテレーション。動作は最終exeと完全に一致。

**LLVM バックエンド**：リリース時に使用。AOTコンパイルからネイティブコードへ、スケジューラを静的ライブラリとしてリンク。解释器なし、JITなし。

#### 6.2 スケジューラ = 静的ライブラリ、仮想マシンではない

```
最終exeの内部構造：

┌────────────────────────────────────────────┐
│  あなたのコード（ネイティブ機械語）                       │
│  ├── コンパイル時に確定したDAG実行計画              │
│  ├── インライン化されたMove/ref/clone操作            │
│  └── RAII解放コード                          │
├────────────────────────────────────────────┤
│  ランタイム静的ライブラリ（約200-500KB）                │
│  ├── スレッドプール（固定サイズ = num_workers）       │
│  ├── イベントループ（libuv / io_uring）             │
│  ├── ワークスティールキュー（Full Runtimeのみ）        │
│  ├── メモリアロケータ（jemalloc / mimalloc）        │
│  └── リフレクションメタデータ（オンデマンドロード、常駐なし）│
├────────────────────────────────────────────┤
│  なし：                                      │
│  ❌ バイトコード解释器                             │
│  ❌ JITコンパイラ                               │
│  ❌ GC                                      │
│  ❌ 仮想マシン                                    │
└────────────────────────────────────────────┘
```

比較：

| | Java | Go | YaoXiang |
|------|------|-----|-----------|
| コンパイル产物 | バイトコード | ネイティブコード | ネイティブコード |
| 実行方式 | JVM解釈/JIT | 直接実行 | 直接実行 |
| ランタイムサイズ | 約200MB（JVM） | 約1-2MB（GC含む） | **約200-500KB（GCなし）** |
| メモリ管理 | GC | GC | **RAII（决定的）** |
| リフレクション | 常駐メモリ | 常駐メモリ | **exe中に保管、オンデマンド** |

#### 6.3 なぜスケジューラ性能が一定なのか

**重要な洞察**：大部分の作業はコンパイル時に完了し、実行時は「実行」のみを行う。

```
コンパイル時（一回限り、実行時に含まれない）：
    ├── グローバルDAGの構築：誰かが誰に依存するか
    ├── トポロジカルソート：実行順序の確定
    ├── 孤島の識別：並列可能なサブツリー
    ├── エスケープ分析：ref → Rc か Arc か
    ├── サイクル検出：自動降格 Weak またはエラー
    └── インライン化：小関数は直接展開

実行時（毎実行、データ構造は固定）：
    ├── コンパイル時に確定したDAG順序でスレッドプールにタスクを配布
    ├── I/O遭遇 → 現在タスクを中断、イベントループが接管
    ├── タスク準備完了 → 準備完了キューに戻す
    └── 以上。
```

**スケジューラ自体は固定サイズのデータ構造**：スレッドプール、イベントループ、ワークキュー。動的成長なし、アダプティブな再最適化なし、GCスキャンなし。動作は完全に予測可能。

コンパイル時に「何をスケジュールするか」を計算し終えている。実行時は「実行」のみを行う。これはtokioとは異なる——tokioは実行時に動的にFutureチェーンを構築する。YaoXiangのDAGは静的である。

#### 6.4 リフレクション：保管するが常駐しない

リフレクションメタデータはコンパイル時に生成され、exeの独立したセグメント（セクション）に保管される。プログラム起動時にはロードされない。初めてリフレクションがリクエストされたとき、オンデマンドでmmapによりメモリにロードされる。以下のような方式：

```
exe レイアウト：
  .text     ← あなたのコード
  .rodata   ← 定数
  .reflect  ← リフレクションメタデータ（型情報、関数シグネチャなど）
              mmap でオンデマンドロード、アクセスしなければメモリ不使用
```

**トレードオフ**：exeサイズ增大（リフレクションデータ含む）が、実行時はアクセスしなければメモリコストゼロ。初回のアクセスにはロード遅延（JITウォームアップ類似）があるが、その後ゼロオーバーヘッド。

## トレードオフ

### 利点

- **明確な階層化**：Embedded / Standard / Full の三層
- **コンパイル再利用**：フロントエンドコードは完全に共有
- **ジェネリクス分離**：コンパイル時多相、ゼロオーバーヘッド
- **一貫性**：同期はスケジューリングの特例に過ぎない
- **組込み開発向き**：高性能 + 低メモリ + 高速起動

### 欠点

- **初期複雑性**：スケジューラインターフェースと複数のランタイムバリアントを定義する必要がある
- **コンパイル時バインディング**：スケジューラ型はコンパイル時に確定
