# RFC-009: 所有権モデル設計

> **状態**: 承認済み
> **著者**: 晨煦
> **作成日**: 2025-01-08
> **最終更新**: 2026-05-29（借用トークンシステムで丐版借用に代わり、型システムを統一）

## 概要

このドキュメントは YaoXiang プログラミング言語の**所有権モデル（Ownership Model）**を定義します。

**コア設計——5つの概念、1つのグラデーション**：

```
ちょっと見る/その場で変更     移動           共有保持         コピーする        システムレベル
    │                     │              │              │              │
   &T                  Move           ref          clone()        unsafe
  &mut T               ゼロコピー     コンパイラ自動      明示的ディープコピー   *T
  ゼロサイズトークン       デフォルト     Rc/Arcを選択                  ユーザー負責
  型の属性で権限を
  自然に推論
  
```

- **Move（デフォルト）**：代入/引数渡/返り値 = 所有権移動、ゼロコピー、RAII 自動解放
- **`&T` / `&mut T`（借用トークン）**：ゼロサイズのコンパイル時トークン型。`&T` は複製可能（共有読み取り専用）、`&mut T` は線形（排他的可変）。権限は型の属性から自然に推論され、特殊ルールは不要。戻り値可能、構造体に格納可能、クロージャにキャプチャ可能。
- **`ref` キーワード**：スコープをまたぐ共有。コンパイラが Rc（非タスク跨ぎ）か Arc（タスク跨ぎ）を自動選択
- **`clone()`**：明示的ディープコピー
- **`unsafe` + `*T`**：裸ポインタ、システムレベルのエスケープハッチ

**排除された複雑さ**：
- ❌ ライフタイム `'a` なし
- ❌ 借用チェッカーなし（トークンの型属性——Dup/Linear——が専用の borrow checker を代替）
- ❌ GC なし
- ❌ 「エスケープ禁止」などの特殊ルールなし（トークンは通常の型で、スコープは型システムで統一管理）
- ❌ ユーザーは Rc/Arc の違いを知る必要なし（コンパイラが自動選択）

> **プログラミング負荷**：`&T` は複製可能、`&mut T` は複製不可——2つの型属性、0の特殊ルール、コンパイラ完全自動化。
> **パフォーマンス保証**：Move ゼロオーバーヘッド、トークンゼロオーバーヘッド（ゼロサイズ型、コンパイル後に消滅）、ref はオンデマンド払い、GC ポーズなし。

## 動機

### なぜ所有権モデルが必要か？

| 言語 | メモリ管理 | 問題点 |
|------|----------|------|
| C/C++ | 手動管理 | メモリリーク、野ポインタ、二重解放 |
| Java/Python | GC | 遅延波动、メモリオーバーヘッド、予測不能なポーズ |
| Rust | 所有権 + 借用チェッカー | ライフタイム `'a` の学習カーブが急 |
| **YaoXiang** | **Move + Token + ref** | **シンプル、决定的、GCなし** |

### 設計目標

```yaoxiang
# 1. デフォルト Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p                         # Move、p はこれ以上読めない

# 2. &T / &mut T 借用トークン（ゼロオーバーヘッド、型属性で自然に権限を推論）
print_info(p2)                 # コンパイラが自動的に &Point トークンを作成し、使用後に解放
shift(p2, 1.0, 1.0)           # コンパイラが自動的に &mut Point トークンを作成

# 3. ref = 共有（コンパイラが Rc/Arc を自動選択）
shared = ref p2                # スコープを越えて保持
spawn { use(shared) }          # コンパイラ：タスク跨ぎ → Arc

# 4. clone() = 明示的コピー
backup = p2.clone()            # ディープコピー、排他的

# 5. unsafe + *T = システムレベル
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rust との核心的な違い

| 機能 | Rust | YaoXiang |
|------|------|----------|
| デフォルトセマンティクス | 借用 `&T`（明示的 `.clone()` が必要） | **Move（値渡し、ゼロコピー）** |
| 借用 | `&T`/`&mut T`、戻り値可能、ライフタイム必要 | **`&T`/`&mut T` ゼロサイズトークン、Dup/Linear 型属性で自然に推論** |
| 共有機構 | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが Rc/Arc を自動選択）** |
| コピー | `clone()` | `clone()` |
| 裸ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用チェック | グローバル推論 | **型チェッカーのフロー感知生存解析（トークン状態追跡）** |
| 循環参照 | 手動 Weak | **タスク終了時の統一解放 / タスク跨ぎ lint / 標準ライブラリ Weak** |

## 提案

### 1. Move（デフォルト所有権移動）

```yaoxiang
# ルール：代入 / 引数渡 / 返り値 = Move、ゼロコピー

p: Point = Point(1.0, 2.0)
p2 = p                           # Move、p はこれ以上読めない

# 変数への再代入が可能（Python スタイル、シャドウイングなし）
p = Point(3.0, 4.0)              # p を再バインド、型は一致する必要がある

# 関数引数：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move 返り値
}

# 関数返り値：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move 返り値、ゼロコピー
}
```

**特徴**：
- ゼロコピー（コンパイラがポインタを移動）
- 移動後元のバインディングは読めない（コンパイルエラー）
- RAII：スコープ終了時に自動解放
- 関数シグネチャ `(T) -> T` 自体がドキュメント——T を消費し、T を返す

### 2. &T / &mut T（借用トークン）

**コア原則：`&T` と `&mut T` はゼロサイズのコンパイル時トークン型です。これらは「参照」ではなく「アクセス権限の型レベル証明」です。**

#### 2.1 2つの型属性

```
&T      →  ゼロサイズ、複製可能（Dup）、読み取り専用権限を付与
&mut T  →  ゼロサイズ、線形（非Dup）、排他的読み書き権限を付与
```

**これは記憶すべき「ルール」ではなく——型システムの基本属性です。** Dup 型は自由롭게複製可能（複数の `&T` が共存可能）、線形型は複製不可（`&mut T` は本質的に一意）。「借用チェッカー」はなく——型チェッカーがいつものように型のチェックをしているだけです。

#### 2.2 基本使用

```yaoxiang
# メソッド側：パラメータ型を宣言し、必要な権限を決定
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # &Point トークンが読み取り権限を付与
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # &mut Point トークンが書き込み権限を付与
    self.y = self.y + dy
}

# 呼び出し側：コンパイラが自動的に借用か Move かを選択
p = Point(1.0, 2.0)
p.print()                          # コンパイラが自動的に &Point トークンを作成
p.shift(1.0, 1.0)                  # コンパイラが自動的に &mut Point トークンを作成
p.print()                          # OK、前回のトークンは shift 呼び出し終了時に解放済み

# 自由関数も同理
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 2つの &Point トークンが共存——Dup 型
}
d = distance(p, p2)
```

#### 2.3 なぜ「禁止エスケープ」が不要か

RFC-009 v8 では `&T`/`&mut T` に3つの特殊ルール——パラメータにしか使えない、戻り値不可、構造体に格納不可——を設けていました。これは「借用」概念へのパッチ当てです。

トークンシステムではこれらのルールは不要です。トークンは**通常の型**であり、他の型と同じスコープルールに従います。

**参照を返す——自然にサポート**：

```yaoxiang
# ✅ トークンが返り値と一緒に伝播
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # サブトークンと부모トークンが一緒に返る
}

# 使用例
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # トークンが呼び出し元に返る
print(px_ref)               # OK、トークンはスコープ内
```

**構造体に格納——自然にサポート**：

```yaoxiang
# ✅ 構造体がトークンをフィールドとして携带
Window: Type = {
    target: Point,
    view: &Point,      # トークンフィールド——target への読み取り専用ビューを保持
}

# view のトークンは target から派生、Window が両方の所有権を保持
# Window が存在する限り、view トークンは有効
```

**クロージャキャプチャ——自然にサポート**：

```yaoxiang
# ✅ クロージャがトークンをキャプチャ、他の値と同じ様に
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    # クロージャが threshold の &Float トークンをキャプチャ（Dup 型、自由롭게クロージャに複製）
    items.filter(|p| p.x > threshold)
}

# これは RFC-009 v8 では不可能——v8 では借用クロージャキャプチャを禁止
```

**タスク跨ぎ——トークンはスレッドを跨げない**：

```yaoxiang
# ❌ トークンはタスク境界を跨げない
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ コンパイルエラー：トークンはタスクを跨げない
}

# これは特殊ルールではなく——トークンはコンパイル時の権限証明、タスク跨ぎ共有には ref を使用
# タスク跨ぎ共有が必要な場合は ref を使用
```

**トークンは ref できない**：

```yaoxiang
# ❌ トークンは権限証明であり、所有権ではない
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ コンパイルエラー：&T は所有可能型ではない
}
```

#### 2.4 トークンのライフタイム

トークンのライフタイムは**通常のスコープルール**で決定され、ライフタイムパラメータは不要：

- 関数パラメータのトークン：呼び出し期間存活、呼び出し終了時に解放
- 返されるトークン：所有権が呼び出し元に移動
- 構造体に格納されたトークン：構造体と一緒に存活
- クロージャにキャプチャされたトークン：クロージャと一緒に存活

コンパイラは `'a` 注釈を必要としない、なぜならトークンは**値**であり、値のライフタイムは所有権システム（Move/RAII）で統一管理される。**借用問題を所有権問題に変換（降維）。**

#### 2.5 凍結機構

`&mut T` トークンは一時的に「凍結」して `&T` トークンを生成できる：

```yaoxiang
modify_and_read: (p: &mut Point) -> Void = {
    p.x = 10.0                      # &mut Point を使用して変更
    
    # &mut を凍結して読み取り専用ビューを取得
    view: &Point = freeze(p)         # ここで p が凍結される
    print(view.x)                   # &Point を通じて読み取り
    print(view.y)
    # view がスコープを離れる、凍結解除
    
    p.y = 20.0                      # &mut Point が再び利用可能
}
```

`freeze` のセマンティクス：
- `&mut T` を受け取り、`&T` を返す
- `&T` が存活している間、元の `&mut T` は使用不可
- `&T` がスコープを離れた後、`&mut T` は自動回復
- これは**フロー感知解析**——コンパイラが関数本体でトークン状態を追跡

#### 2.6 トークン競合検出

RFC-009 v8 の「交叉借用チェック」の代替。原理がよりシンプル——トークン値の**フロー感知生存解析**：

```yaoxiang
# ❌ &mut と派生した &T は同時に存活できない
bad_alias: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p が凍結される
    p.x = 10.0                      # ❌ コンパイルエラー：WriteToken が凍結状態
    print(view.x)                   
}

# ✅ 凍結解除後に &mut を続けて使用可能
good_seq: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p が凍結される
    print(view.x)                   # &T を使用
    # view がスコープを離れる
    p.x = 10.0                      # ✅ WriteToken が回復済み
}
```

**検出方式**：これは専用の「借用チェッカー」ではなく——トークン値の**フロー感知生存解析**です。コンパイラが関数本体で各トークンの状態（存活中/凍結中/移動済み）を追跡し、任意の線形型値の追跡方法と同じです。

#### 2.7 コンパイラ内部：ブランド機構

ユーザーはブランドを意識しません。コンパイラは内部で各トークンにコンパイル時一意の識別子を割り当てます：

```
ユーザーが見るもの     コンパイラ内部表現
────────────────────────────────────────
&Point            →  ReadToken(Point, #N)    // #N はコンパイル時一意の整数
&mut Point        →  WriteToken(Point, #M)   // #M はコンパイル時一意の整数
```

ブランドの用途：
- **偽造防止**：トークンは所有者の胶囊または freeze 操作からのみ取得でき、素朴には構築できない
- **関連追跡**：`&Point` から `&Float`（フィールドアクセス）を派生시킬 때、`&Float` は派生ブランド（`#N.field_x`）を携带し、コンパイラは부모トークンに追跡可能
- **競合検出**：同一来源の `WriteToken` と派生の `ReadToken` は同時に存活不可

ブランドは単態化とインライン展開後に完全に消滅し、生成された機械語中に存在しない。**ゼロランタイムオーバーヘッド。**

#### 2.8 自動借用選択ルール

呼び出し側コンパイラは以下の優先順位で自動選択：

```
1. 実引数がその後まだ使用される場合 → トークンを作成優先（&T または &mut T、メソッドシグネチャによる）
2. 実引数がその後不再使用の場合 → Move
3. 優先匹配順序：&T < &mut T < Move
```

```yaoxiang
# 例：自動選択
p = Point(1.0, 2.0)
p.print()        # print は &self を宣言 → コンパイラが &Point トークンを作成
p.shift(1.0, 1.0) # shift は &mut self を宣言 → コンパイラが &mut Point トークンを作成
p2 = p           # Move、p は不再使用
```

#### 2.9 RFC-009 v8 丐版借用との比較

| 機能 | 丐版借用 (v8) | 借用トークン (v9) |
|------|--------------|--------------|
| 参照を返す | ❌ ハードコードで禁止 | ✅ トークンが返り値と一緒に伝播 |
| 構造体に格納 | ❌ ハードコードで禁止 | ✅ トークンが構造体フィールドとして |
| クロージャキャプチャ | ❌ ハードコードで禁止 | ✅ クロージャがトークン値をキャプチャ |
| 特殊ルール | 3条（パラメータ専用/返り値不可/格納不可） | 0条——型属性が自然に推論 |
| 借用チェック | 専用交叉借用チェック | 型チェッカーのフロー感知生存解析 |
| ライフタイム注釈 | 不要 | 不要 |
| ランタイムオーバーヘッド | ゼロ | ゼロ（ゼロサイズ型、コンパイル後に消滅） |
| エラーメッセージ | "借用はエスケープできない" | "WriteToken(#3) は凍結済み"（通常の型エラー） |
| ユーザー Mental Model | 「借用」の特殊性を理解する必要がある | `&T` は複製可能、`&mut T` は複製不可 |

### 3. ref キーワード（コンパイラ自動最適化）

`ref` はスコープをまたぐ共有の唯一の方法です。ベースが Rc か Arc かは、ユーザーが気にする必要はありません。

#### 3.1 基本使用

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共有、コンパイラが自動選択

# タスク跨ぎ共有
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # コンパイラ：タスク跨ぎ → Arc
    spawn { use(data) }           # コンパイラ：タスク跨ぎ → Arc
}

# 単一タスク内共有
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # コンパイラ：タスク跨ぎなし → Rc
}
```

**ユーザー Mental Model**：`ref` = 共有保持。それだけで十分。

#### 3.2 コンパイラエスケープ解析：Rc vs Arc

```
ref のデータフロー解析：

他のタスクにエスケープしない → Rc（非原子参照カウント、低オーバーヘッド）
他のタスクにエスケープする   → Arc（原子参照カウント、スレッドセーフ）
```

#### 3.3 循環検出ストラテジー

```
タスク内循環 → 黙って許可。
  ├── 構造化並行処理がタスク終了時に全リソースを統一解放することを保証。
  ├── ref は常に生存、の意味に変質なし。
  └── ユーザーはタスク内で双方向の強参照を構築する権利を持つ（例：グラフ計算の中間状態）。

タスク跨ぎ循環 → lint（デフォルト warn、構成可能）。
  ├── プログラム動作は正確、真のリークは発生しない（親タスク終了時に子タスクリソースは全て解放）。
  ├── しかしタスク跨ぎの強参照は所有権境界が曖昧であることを意味し、立ち止まって再考する価値あり。
  ├── デフォルト warn レベル、コンパイル成功だがヒントあり。
  └── チームはプロジェクト構成で deny に設定でき、CI 品質ゲートに組み込める。
```

**Lint レベル**（Rust clippy 类似）：

| レベル | 動作 | シナリオ |
|------|------|------|
| `allow` | チェックなし | 個人プロジェクト |
| `warn`（デフォルト） | コンパイル成功、ヒントあり | 開発フェーズ |
| `deny` | コンパイル失敗 | チーム CI 品質ゲート |
| `forbid` | コンパイル失敗、上書き不可 | 組織レベル強制ルール |

```yaoxiang
# タスク内循環：黙って許可、双方向強参照
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # 循環。タスク終了時に統一解放。
}

# タスク跨ぎ循環：lint（デフォルト warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: タスク跨ぎ循環参照
    }
}
```

**プロジェクト構成例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # タスク跨ぎ循環は CI で直接拒否
```

| 循環タイプ | 動作 | 理由 |
|--------|------|------|
| タスク内 ref 循環 | チェックなし | ユーザーの権利、タスク終了時に統一解放 |
| タスク跨ぎ ref 循環 | lint（デフォルト warn） | 再考のヒント、deny に構成可能 |

#### 3.4 Weak：標準ライブラリ提供

```yaoxiang
use std.rc.Weak

# 上級ユーザーが明示的に選択
a.next = ref b
b.prev = Weak.new(a.next)        # ユーザーが明示的に弱い方を制御
```

**`Weak` は言語組み込みではなく、標準ライブラリの型です。** 日常使用は `ref` で十分。メモリを詳細に制御する必要がある上級ユーザーが手動で `Weak` を导入します。

#### 3.5 借用トークン vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 何をするか | ちょっと見る/その場で変更 | 共有保持 |
| 範囲 | トークン値のスコープに従う | スコープをまたぐ |
| コスト | ゼロオーバーヘッド（ゼロサイズ型） | Rc または Arc（コンパイラ選択） |
| エスケープ | 可能（トークンが返り値/構造体/クロージャと共に伝播） | そもそもエスケープ用 |
| タスク跨ぎ | 不可（トークンはコンパイル時の権限証明、タスクを跨げない） | 可能（コンパイラが自動選択 Arc） |
| 循環 | 関係ない | タスク内黙認、タスク跨ぎ lint |

---

### 4. clone() —— 明示的コピー

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # ディープコピー
# p と p2 は独立、相互に影響しない
```

**使用时机**：元の値を保持する必要があり、Move に適さず、共有にも適さない場合。

### 5. unsafe + 裸ポインタ（システムレベルプログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 裸ポインタ
    (*ptr).x = 0.0                # 逆参照（ユーザーが安全を保証）
    ptr2 = ptr + 1                # ポインタ演算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ユーザーはダングリング、使用後の解放を保証
- FFI、メモリ操作などのシステムレベルプログラミングに使用

### 6. 所有権グラデーション総览

```
  借用トークン（ゼロオーバーヘッド）     Move（ゼロオーバーヘッド）      共有（オンデマンド払い）    コピー
   │                          │                  │                │
  &T 複製可能トークン        デフォルト所有権移動     ref Rc/Arc       clone()
  &mut T 線形トークン        チェーン消費回流       コンパイラ自動選択    明示的ディープコピー
   │                          │                  │                │
  トークン値スコープ           スコープ内           スコープ跨ぎ         常時
  返り値/構造体格納可能        T -> T 回流         ref タスク跨ぎ → Arc  独立コピー
  クロージャにキャプチャ可能    T -> Void 消費     ref 非タスク跨ぎ → Rc
  ゼロサイズコンパイル後消滅                         タスク内循環黙認
                                            タスク跨ぎ循環 lint
                                            標準ライブラリ Weak エスケープ
```

## 統合例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：読み取り専用トークン
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：可変トークン
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消費回流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 拿走，改，还给你
    }

    # 参照を返す：トークンが返り値と一緒に伝播
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# クロージャがトークンをキャプチャ（v8 では不可能な能力）
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}

# 統合使用
p = Point(1.0, 2.0)
p.print()                           # &Point トークン
p.shift(1.0, 1.0)                   # &mut Point トークン
p = p.scale(2.0)                    # Move → 回流
shared = ref p                      # ref 共有
spawn { use(shared) }

# clone 独立コピー
backup = p.clone()

# タスク内循環：黙って許可
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # 循環、タスク終了時に統一解放

# unsafe システムレベル
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Dup 型属性

`Dup`（Duplicable、暗黙的コピー可能）はコンパイラが自動管理する重要な型属性です。`Dup` を実装した型は、代入時に所有権移動が発生せず、暗黙的シャローコピーが発生します。`Dup` は `Clone` を蕴含——型が `Dup` であれば常にフィールドごとにクローン可能；但し `Clone` は `Dup` を蕴含しません。

| 型 | Dup | Clone | 説明 |
|------|-----|-------|------|
| Int, Float, Bool, Char, String, Bytes | ✅（コンパイラが自動実装） | ✅ | 基本型 |
| `&T` | ✅（ゼロサイズ、自由</minimax:tool_call> 可能） | ✅ | 読み取り専用トークン |
| `&mut T` | ❌（線形、排他的） | ❌ | 可変トークン |
| struct | ✅（全フィールドが Dup の場合自動派生） | ✅ | 構造体 |
| `ref T` | ✅ | ✅ | 共有保持（コンパイラが Rc/Arc を自動選択） |
| `*T` | ❌ | ❌ | 裸ポインタ |

> **注意**：YaoXiang にはユーザーが見える `Send`/`Sync` trait がありません。`ref` キーワードはコンパイラが Rc（タスク跨ぎなし）か Arc（タスク跨ぎ）を自動選択し、ユーザーは型がタスクを跨いで渡せるかどうかを気にする必要がありません。トークンがタスク境界を跨げないのは trait 制約ではなく、コンパイラの組み込みルールです。

## パフォーマンス分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | ゼロ | ポインタ移動 |
| `&T` / `&mut T` | ゼロ | ゼロサイズ型、コンパイル後に消滅、ゼロランタイムオーバーヘッド |
| `ref`（タスク跨ぎなし）| 低 | Rc にコンパイル、非原子操作 |
| `ref`（タスク跨ぎ）| 中 | Arc にコンパイル、原子操作 |
| `clone()` | 型による | 小さいオブジェクトは高速、大きいオブジェクトは低速 |
| `unsafe + *T` | ゼロ | 直接メモリ操作 |

### 比較

| 言語 | 共有機構 | メモリ管理 | 循環処理 | 複雑さ |
|------|----------|----------|----------|--------|
| Rust | Arc / Mutex + 借用チェッカー | コンパイル時チェック | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 借用トークン** | **RAII** | **タスク境界解放 / タスク跨ぎ lint / 標準ライブラリ Weak** | **低** |

## トレードオフ

### メリット

1. **統一**：`&T`/`&mut T` は通常の型で、特殊言語機能ではない。RFC-010 の `name: type = value` と完全に一致
2. **シンプル**：ライフタイムなし、グローバル借用チェッカーなし。`&T` は複製可能、`&mut T` は複製不可——2つの型属性
3. **強力**：参照を返せる、構造体に格納可能、クロージャにキャプチャ可能——Rust と同等の表現力
4. **コンパイラIntelligent**：ref が Rc/Arc を自動選択、呼び出し側で自動借用選択
5. **確定的**：ref は単に生存保証、弱参照に悄悄变化しない
6. **高性能**：Move ゼロコピー、トークンゼロオーバーヘッド（ゼロサイズ型、コンパイル後に消滅）
7. **柔軟**：`unsafe + *T` がシステムレベルプログラミングをサポート

### デメリット

1. **generics ブランドパラメータ感染**：トークンはブランド識別子を携带し、参照を返す関数のシグネチャに追加のgenerics パラメータが現れる
2. **ref ランタイムオーバーヘッド**：原子操作にはコストがある（しかしこれは共有の必然的コスト）
3. **unsafe リスク**：ユーザーは正確性を保証する必要がある
4. **タスク跨ぎ循環は lint でコンパイルエラーではない**：Rust のようにコンパイルエラーになるのではなく、デフォルト warn、チームで deny を構成しないと品質ゲートになれない

## 代替案

| 案 | なぜ選ばないか |
|------|--------------|
| GC | ランタイムオーバーヘッドがあり、予測不能なポーズがある |
| Rust 借用チェッカー | ライフタイム `'a` が必要で、学習カーブが急 |
| 純 Move | 並行共有を処理できない |
| 裸ポインタなし | システムレベルプログラミングができない |
| Rc/Arc をユーザーに露出 | 実装詳細をユーザーに押し付け、認知負担が増加 |
| 丐版借用（v8） | エスケープ禁止ポリシーによりクロージャキャプチャ、参照戻り値などの重要な表現力を犠牲 |

### トークン競合検出：フロー感知生存解析

**解析範囲**：関数本体のみ。トークン値に対してフロー感知生存解析を行い、各トークンの状態（存活中/凍結中/移動済み）を追跡します。

**レベル1：呼び出しポイントチェック**——同一の実引数に対して同時に `&mut` トークンと他のトークンを作成できない：

```yaoxiang
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)    # ❌ p が同時に &mut と & トークンを派生、コンパイラが拒否
```

**レベル2：関数本体フロー感知**——`&mut` トークンを呼び出しに渡した後、呼び出しが返ればトークンは解放され、その後再度トークンを作成可能：

```yaoxiang
process_twice: (p: &mut Point) -> Void = {
    shift(p, 1.0, 1.0)    # &mut トークンを shift に渡し、shift が返るとトークン解放
    print_info(p)          # &Point トークンを再作成、競合なし
}
```

**レベル3：凍結状態追跡**——`freeze` によって生成された `&T` トークンが存活中の間、元の `&mut` トークンは使用不可：

```yaoxiang
frozen: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)    # p が凍結状態に入る
    print(view.x)
    p.x = 10.0                  # ❌ コンパイルエラー：WriteToken が凍結状態
}
```

**不要なもの**：関数間ライフタイム追跡、グローバルエイリアス解析、借用グラフ制約解決、NLL、`'a` 注釈。トークンは値であり、値の生存解析は型チェッカーが統一的に処理するため——任意の線形型値の追跡方法と同じです。
