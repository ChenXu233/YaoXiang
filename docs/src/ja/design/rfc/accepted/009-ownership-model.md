# RFC-009: 所有権モデル設計

## 摘要

本文書は YaoXiang プログラミング言語の**所有権モデル（Ownership Model）**を定義する。

**コア設計——5つの概念、1つのグラデーション**：

```
一面だけ/その場で改     拿走           共有持有         複製一份        システム級
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         ゼロコピー     コンパイラ自動    明示的ディープコピー  *T
  零大小令牌       デフォルト      Rc/Arc選択                   ユーザ責任
  タイプ属性自然                                
  推导权限                              
```

- **Move（デフォルト）**：代入/引数渡/返値 = 所有権移転、ゼロコピー、RAII 自動解放
- **`&T` / `&mut T`（借用トークン）**：ゼロサイズのコンパイル時トークン型。`&T` は複製可能（共有読取専用）、`&mut T` は線形（排他可変）。権限はタイプ属性から自然に導出され、特殊ルールは不要。返値可能、構造体に格納可能。
- **`ref` キーワード**：スコープを跨ぐ共有。コンパイラが Rc（タスクを跨がない）か Arc（タスクを跨ぐ）かを自動選択
- **`clone()`**：明示的ディープコピー
- **`unsafe` + `*T`**：生ポインタ、システム級エスケープハッチ

**排除した複雑さ**：
- ❌ ライフタイム `'a` なし
- ❌ 独立した借用検査フレームワークなし（借用競合はホア命题に次元削減され、タイプ検査と証明パイプを共有）
- ❌ GC なし
- ❌ 「エスケープ禁止」などの特殊ルールなし（トークンは通常の型であり、スコープは型システムで統一管理）
- ❌ ユーザが Rc/Arc の違いを知る必要はない（コンパイラが自動選択）

> **プログラミング負担**：`&T` は複製可能、`&mut T` は複製不可——2つのタイプ属性、特殊ルールゼロ、コンパイラ完全自動化。
> **パフォーマンス保証**：Move ゼロオーバーヘッド、トークンゼロオーバーヘッド（ゼロサイズ型、コンパイル後に消滅）、ref 必要时才付费、GC ポーズなし。

## 動機

### なぜ所有権モデルが必要か？

| 言語 | メモリ管理 | 問題点 |
|------|----------|------|
| C/C++ | 手動管理 | メモリリーク、野ポインタ、二重解放 |
| Java/Python | GC | 遅延波动、メモリオーバーヘッド、予測不能なポーズ |
| Rust | 所有権 + 借用検査 | ライフタイム `'a` の学習曲線が険しい |
| **YaoXiang** | **Move + Token + ref** | **シンプル、决定的、GCなし** |

### 設計目標

```yaoxiang
# 1. デフォルト Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p                         # Move、p はこれ以上読めない

# 2. &T / &mut T 借用トークン（ゼロオーバーヘッド、タイプ属性から自然に権限を導出）
print_info(p2)                 # コンパイラが自動的に &Point トークンを作成し、使用後即解放
shift(p2, 1.0, 1.0)           # コンパイラが自動的に &mut Point トークンを作成

# 3. ref = 共有（コンパイラが自動的に Rc/Arc を選択）
shared = ref p2                # スコープを跨いで保持
spawn { use(shared) }          # コンパイラ：タスクを跨ぐ → Arc

# 4. clone() = 明示的複製
backup = p2.clone()            # ディープコピー、排他的

# 5. unsafe + *T = システム級
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rust とのコアな違い

| 機能 | Rust | YaoXiang |
|------|------|----------|
| デフォルトセマンティクス | 借用 `&T`（明示的 `.clone()` が必要） | **Move（値渡し、ゼロコピー）** |
| 借用 | `&T`/`&mut T`、返値可能、ライフタイム必要 | **`&T`/`&mut T` ゼロサイズトークン、Dup/Linear タイプ属性から自然に導出** |
| 共有機構 | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが自動的に Rc/Arc を選択）** |
| 複製 | `clone()` | `clone()` |
| 生ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用検査 | 大域的導出 | **型チェッカが自動的に借用命題を生成、証明パイプが統一検証** |
| 循環参照 | 手動 Weak | **タスク終了時に統一解放 / タスク間 lint / 標準ライブラリ Weak** |

## 提案

### 1. Move（デフォルト所有権移転）

```yaoxiang
# ルール：代入 / 引数渡 / 返値 = Move、ゼロコピー

p: Point = Point(1.0, 2.0)
p2 = p                           # Move、p はこれ以上読めない

# 変数は再代入可能（Python スタイル、シェードなし）
p = Point(3.0, 4.0)              # p は再バインド、型は一貫する必要がある

# 関数引数：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move 返値
}

# 関数返値：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move 返値、ゼロコピー
}
```

**特徴**：
- ゼロコピー（コンパイラがポインタを移動）
- 移動後元のバインディングは読めない（コンパイルエラー）
- RAII：スコープ終了時に自動解放
- 関数シグネチャ `(T) -> T` 自体がドキュメント——T を消費し、T を返す

### 2. &T / &mut T（借用トークン）

**コア原則：`&T` と `&mut T` はゼロサイズのコンパイル時トークン型である。それらは「参照」ではなく「アクセス権限の型レベル証明」である。**

#### 2.1 2つのタイプ属性

```
&T      →  ゼロサイズ、ソースデータを凍結（ReadToken 生存中は WriteToken を禁止）、
          凍結保証下で複数読取りビューが安全 → 複製可能（Dup）
&mut T  →  ゼロサイズ、排他的読取り書き取り（WriteToken 生存中は他のトークンを全て禁止）、
          排他アクセスでは複製は無意味 → 線形（非 Dup）
```

**因果関係は逆転しない：冻结が理由で、Dup が結果である。** `&T` が Dup を実装しているから共存できるわけではない——データが冻结されているから（変異の可能性がない）、複数読取りビューが安全であり、Dup が実装できる。もし Dup を通過定義とし、競合検査を「追加のパッチ」と見なせば、デザインが間違っている。

#### 2.2 基本使用

```yaoxiang
# メソッド側：引数型を宣言し、必要な権限を決定
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # &Point トークンが読取り権限を付与
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # &mut Point トークンが書き込み権限を付与
    self.y = self.y + dy
}

# 呼び出し側：コンパイラが自動的に借用または Move を選択
p = Point(1.0, 2.0)
p.print()                          # コンパイラが自動的に &Point トークンを作成
p.shift(1.0, 1.0)                  # コンパイラが自動的に &mut Point トークンを作成
p.print()                          # OK、前のトークンは shift 呼び出し終了時に解放済み

# 自由関数も同じ
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 2つの &Point トークンが共存——Dup 型
}
d = distance(p, p2)
```

#### 2.3 なぜ「エスケープ禁止」が不要か

RFC-009 v8 の丐版借用では `&T`/`&mut T` に3つの特殊ルールを課していた——引数にしか使えない、返値不可、構造体に格納不可。これは「借用」概念にパッチを当てている。

トークンシステムではこれらのルールは不要である。トークンは**通常の型**であり、他の型と同じスコープルールに従う。

**参照を返す——自然にサポート**：

```yaoxiang
# ✅ トークンは返値と一緒に伝播
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # サブトークンと부모トークンが一緒に返値
}

# 使用例
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # トークンが呼び出し元に返される
print(px_ref)               # OK、トークンはまだスコープ内
```

**構造体に格納——自然にサポート**：

```yaoxiang
# ✅ 構造体がトークンをフィールドとして携带
Window: Type = {
    target: Point,
    view: &Point,      # トークンフィールド——target への読取り専用ビューを保持
}

# view のトークンは target から派生、Window は両方の所有権を保持
# Window が存在する限り、view トークンは有効
```

#### 2.3 クロージャと Lambda の明示的引数

Lambda は関数値——返値、格納、現在のスコープ外への传出が可能。因此 Lambda **は外層ローカル変数を暗黙的にキャプチャしない**。外層データが必要な場合は、明示的な引数として渡す：

```yaoxiang
# ✅ Lambda は明示的引数を使用
double: (x: Int) -> Int = (x) => x * 2
filter_by: (items: List(Int), f: (Int) -> Bool) -> List(Int) = { ... }

# ✅ spawn { } はこのルールの影響を受けない——spawn は即座に実行される並列ブロック、親タスクは阻塞で待機
shared = ref data
spawn { use(shared) }

# ❌ Lambda は外層変数を暗黙的にキャプチャできない
x = 42
f = () => { x + 1 }  # コンパイルエラー：x はスコープ内にない

# ✅ 正しい方法：明示的に引数を渡す
f = (x) => { x + 1 }
f(x)
```

**spawn { } は関数値ではない。** spawn がマークする block は if/while body 一样——即座に実行され、親スタックフレーム生存中に完了。spawn body は正常に外層変数にアクセスできる。

**タスク間——トークンは糸を通せない**：

```yaoxiang
# ❌ トークンはタスク境界を跨げない
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ コンパイルエラー：トークンはタスクを跨げない
}

# これは特殊ルールではない——トークンはコンパイル時の権限証明、タスク間共有には ref を使用
# タスク間を跨いで共有する必要がある場合は ref を使用
```

**トークンは ref できない**：

```yaoxiang
# ❌ トークンは権限証明であり、所有権ではない
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ コンパイルエラー：&T は所有可能的型ではない
}
```

#### 2.4 トークンのライフサイクル

トークンのライフサイクルは**通常のスコープルール**によって 결정され、ライフタイムパラメータは不要：

- 関数引数内のトークン：呼び出し期間生存、呼び出し終了時に解放
- 返されるトークン：所有権が呼び出し元に移転
- 構造体に格納されたトークン：構造体と一緒に生存

コンパイラは `'a` アノテーションを必要としない、なぜならトークンは**値**であり、値のライフサイクルは所有権システム（Move/RAII）が統一管理する。**借用問題を所有権問題に成功裏に次元削減した。**

#### 2.5 トークン競合検出

トークン競合検出は**ホア論理命題**であり、独立したフロー敏感分析ではない。

```
{競合する ReadToken が全て死亡} write(data) {WriteToken が安全に取得}
```

型検査、ユーザ述語検証と RFC-027 の証明パイプを共有する。コンパイラは自動的に借用命題を生成し（`borrow_conflict`、`use_after_move`、`use_after_drop`、`mut_violation`）、パイプに送信して検証する。パイプは Proved / Disproved / Unproven を返す。

```yaoxiang
# ❌ &mut トークンは線形であり、複製できない
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move、p はこれ以上読めない
    p.x = 10.0                      # ❌ コンパイルエラー：WriteToken は既に移動済み
}

# ✅ &T トークンは Dup 型であり、自由롭게複製可能
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK、&T は Dup 型
    print(p.x)                      # OK
    print(p2.x)                     # OK、2つの読取り専用トークンが共存
}
```

**借用検査は消えたわけではない——次元削減された。** 既存の `BorrowChecker` は `BorrowPredicateEmitter`（命題生成器）に変わり、生成された借用命題と他の型命題が同じ証明パイプを共有する。これは型検査器の概念と完全に並行している：型検査器が型等式命題を生成し、借用命題生成器が借用命題を生成し、同じパイプが検証する。詳細設計は [RFC-009a](../accepted/009a-borrow-proof-pipeline.md) を参照。

#### 2.7 コンパイラの内部：ブランド機構

ーユザはブランドに触れない。コンパイラは内部で各トークンにコンパイル時一意識別子を割り当て：

```
ューザの表示           コンパイラの内部表現
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N はコンパイル時一意の整数
&mut Point     →  WriteToken(Point, #M)   // #M はコンパイル時一意の整数
```

ブランドの用途：
- **偽造防止**：トークンは所有者胶囊からのみ取得でき、空から構築できない
- **関連追跡**：フィールドアクセス時に `&Point` から `&Float` を派生させると、`&Float` は派生ブランド（`#N.field_x`）を携带し、コンパイラは부모トークンに追跡可能
- **競合検出**：同一来源の `WriteToken` と派生の `ReadToken` は同時にアクティブになれない

ブランドは単態化とインライン展開後に完全に消滅し、生成される機械語中には存在しない。**ゼロランタイムオーバーヘッド。**

#### 2.8 自動借用選択ルール

呼び出し側コンパイラは以下の優先順位で自動選択：

```
1. 实引数が后续でも使用される場合 → トークン作成を優先（&T または &mut T、メソッド署名に従う）
2. 实引数が后续で使用されない場合 → Move
3. 優先一致順：&T < &mut T < Move
```

```yaoxiang
# 例：自動選択
p = Point(1.0, 2.0)
p.print()        # print は &self を宣言 → コンパイラが &Point トークンを作成
p.shift(1.0, 1.0) # shift は &mut self を宣言 → コンパイラが &mut Point トークンを作成
p2 = p           # Move、p は不再使用
```

#### 2.9 RFC-009 v8 丐版借用との比較

| 機能 | 丐版借用 (v8) | 借用トークン (v9) |
|------|--------------|--------------|
| 参照の返値 | ❌ ハードコードで禁止 | ✅ トークンが返値と一緒に伝播 |
| 構造体格納 | ❌ ハードコードで禁止 | ✅ トークンが構造体フィールドとして格納可能 |
| Lambda 明示的引数 | ❌ ハードコードで禁止 | ✅ Lambda は明示的引数を使用 |
| 特殊ルール | 3つ（引数にしか使えない/返値不可/格納不可） | 0つ——タイプ属性から自然に導出 |
| 借用検査 | 専用クロスリ借用検査 | 型チェッカのフロー敏感活性分析 |
| ライフタイム注釈 | 不要 | 不要 |
| ランタイムオーバーヘッド | ゼロ | ゼロ（ゼロサイズ型、コンパイル後に消滅） |
| エラー訊息 | "借用はエスケープできない" | "WriteToken(#3) は既に移動済み"（通常の型エラー） |
| ユーザーメンタルモデル | "借用"の特殊地位を理解する必要がある | `&T` は複製可能、`&mut T` は複製不可 |

### 3. ref キーワード（コンパイラ自動最適化）

`ref` はスコープを跨いで共有する唯一の方法である。基盤が Rc か Arc かは、ューザが気にする必要はない。

#### 3.1 基本使用

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共有、コンパイラが自動的に実装を選択

# タスク間共有
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # コンパイラ：タスクを跨ぐ → Arc
    spawn { use(data) }           # コンパイラ：タスクを跨ぐ → Arc
}

# 単一タスク内共有
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # コンパイラ：タスクを跨がない → Rc
}
```

**ユーザーメンタルモデル**：`ref` = 共有保持。それだけである。

#### 3.2 コンパイラのエスケープ分析：Rc vs Arc

```
ref のデータフロー分析：

他のタスクにエスケープしない → Rc（非原子参照カウント、低オーバーヘッド）
他のタスクにエスケープする   → Arc（原子参照カウント、スレッドセーフ）
```

#### 3.3 環検出戦略

```
タスク内環 → 静かに許可。
  ├── 各タスクには明確なライフサイクル境界がある——タスク終了時に全リソース（ref 環を含む）が統一解放される。
  ├── 長時間実行するサービスはリクエスト/接続ごとにサブタスクを作成すべき——サブタスク終了時に自動回収され、蓄積リークが発生しない。
  ├── ref は常に保持され、セマンティクスに偽りがない。
  └── ユーザにはタスク内で双方向の強参照を構築する権利がある（例：グラフ計算の中間状態）。

タスク間環 → lint（デフォルト warn、構成可能）。
  ├── プログラムの動作は正しい、実際にリークすることはない（親タスク終了時にサブタスクのリソースは全て解放される）。
  ├── しかしタスク間を跨ぐ強参照は所有権境界が曖昧であることを意味し、再考する価値がある。
  ├── デフォルト warn レベル、コンパイルは通るがヒントが表示される。
  └── チームはプロジェクト設定で deny に設定でき、CI 品質ゲートに組み込める。
```

**Lint レベル**（Rust clippy 类似）：

| レベル | 動作 | シナリオ |
|------|------|------|
| `allow` | 検査しない | 個人プロジェクト |
| `warn`（デフォルト） | コンパイル成功、ヒント表示 | 開発フェーズ |
| `deny` | コンパイル失敗 | チーム CI 品質ゲート |
| `forbid` | コンパイル失敗、上書き不可 | 組織レベル強制ルール |

```yaoxiang
# タスク内環：静かに許可、双方向強参照
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # 環。タスク終了時に統一解放。
}

# タスク間環：lint（デフォルト warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: タスク間循環参照
    }
}
```

**プロジェクト設定例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # タスク間環は CI で直接拒否
```

| 環の種類 | 動作 | 理由 |
|--------|------|------|
| タスク内 ref 環 | 検査なし | ューザの権利、タスク終了時に統一解放 |
| タスク間 ref 環 | lint（デフォルト warn） | 再考の促し、deny に構成可能 |

#### 3.4 Weak：標準ライブラリ提供

```yaoxiang
use std.rc.Weak

# 上級ューザが明示的に選択
a.next = ref b
b.prev = Weak.new(a.next)        # ューザがどちらの方向を弱にするかを明示的に制御
```

**`Weak` は言語組み込みではなく、標準ライブラリの型である。** 日常的には `ref` だけで十分である。メモリを精密に制御する必要がある上級ューザが 手動で `Weak` を导入する。

#### 3.5 借用トークン vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 何をするか | 一面だけ/その場で改 | 共有保持 |
| 範囲 | トークン値のスコープに従う | スコープを跨ぐ |
| コスト | ゼロオーバーヘッド（ゼロサイズ型） | Rc または Arc（コンパイラ選択） |
| エスケープ | 可能（トークンは返値/構造体/クロージャ，跟着传播） | 本身就是エスケープ用 |
| タスク間 | 不可（トークンはコンパイル時権限証明、タスクを跨げない） | 可（コンパイラが自動的に Arc を選択） |
| 環 | 関係ない | タスク内なら静かに許可、タスク間なら lint |

### 4. clone() —— 明示的複製

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # ディープコピー
# p と p2 は独立しており、互いに影響しない
```

**使用时机**：元の値を保持する必要があり、Move にも共有にも適さない場合。

### 5. unsafe + 生ポインタ（システム級プログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 生ポインタ
    (*ptr).x = 0.0                # Dereference（ューザが安全を保証）
    ptr2 = ptr + 1                # ポインタ演算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ューザはダングリングこと、解放後の使用を保証する必要がある
- FFI、メモリ操作などのシステム級プログラミングに使用

### 6. 所有権グラデーション総覧

```
  借用トークン（ゼロオーバーヘッド）     Move（ゼロオーバーヘッド）      共有（必要时才付费）    複製
   │                      │                  │                │
  &T 複製可能トークン      デフォルト所有権移転    ref Rc/Arc       clone()
  &mut T 線形トークン      チェーン消費回流      コンパイラ自動選択     明示的ディープコピー
   │                      │                  │                │
  トークン値スコープ        スコープ内            スコープを跨ぐ         いつでも
  返値/構造体格納可能      T -> T 回流         ref タスク間 → Arc   独立コピー
  ゼロサイズコンパイル後消滅  T -> Void 消費    ref 単一タスク内 → Rc
  ゼロサイズコンパイル後消滅                      タスク内環は静かに許可
                                            タスク間環は lint
                                            標準ライブラリ Weak でエスケープ
```

## 統合例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：読取り専用トークン
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：可変トークン
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消費回流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 拿走、改、返す
    }

    # 参照を返す：トークンが返値と一緒に伝播
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# Lambda 明示的引数
double: (x: Int) -> Int = (x) => x * 2

# 統合使用
p = Point(1.0, 2.0)
p.print()                           # &Point トークン
p.shift(1.0, 1.0)                   # &mut Point トークン
p = p.scale(2.0)                    # Move → 回流
shared = ref p                      # ref 共有
spawn { use(shared) }

# clone 独立コピー
backup = p.clone()

# タスク内環：静かに許可
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # 環、タスク終了時に統一解放

# unsafe システム級
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Dup タイプ属性

`Dup`（複製可能）はコンパイラが自動管理するタイプ属性であり、意味は**シャローコピー**：代入/引数渡時に複製されるのはハンドラ/トークンであり、基盤データは共有される。これは Move（所有権移転）や Clone（明示的ディープコピー、独立コピーの作成）と共に3段階グラデーションを形成する。

**Dup と Clone は直交する概念**——Dup はハンドラを複製してデータを共有し、Clone は独立コピーを作成する。型は Dup と Clone の両方をサポートすることも、片方만サポートすることも可能である。

| 型 | Dup | Clone | 説明 |
|------|-----|-------|------|
| `&T` | ✅（トークンを複製、複数のビューが同じデータを指す） | ✅ | 読取り専用トークン |
| `ref T` | ✅（参照カウント+1、ヒープデータを共有） | ✅ | 共有保持（コンパイラが自動的に Rc/Arc を選択） |
| String, Bytes | ✅（内部参照カウント、ハンドラを複製して基盤 buffer を共有） | ✅ | 文字列/バイト |
| `&mut T` | ❌（線形、排他的） | ❌ | 可変トークン |
| `*T` | ❌ | ❌ | 生ポインタ |
| struct | 派生（全フィールドが Dup の時に自動派生） | ✅ | 構造体 |

**原語値型**（Int, Float, Bool, Char）の代入動作はコンパイラの組み込み値複製——2つの値は完全に独立しており、シャローコピーではない。它们は Dup タイプ属性には属さず、コンパイラのネイティブ処理である。

## パフォーマンス分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | ゼロ | ポインタ移動 |
| `&T` / `&mut T` | ゼロ | ゼロサイズ型、コンパイル後に消滅、ゼロランタイムオーバーヘッド |
| `ref`（タスク間なし）| 低 | Rc にコンパイル、非原子操作 |
| `ref`（タスク間）| 中 | Arc にコンパイル、原子操作 |
| `clone()` | 型による | 小さいオブジェクトは速い、大きいオブジェクトは遅い |
| `unsafe + *T` | ゼロ | 直接メモリ操作 |

### 比較

| 言語 | 共有機構 | メモリ管理 | 循環処理 | 複雑さ |
|------|----------|----------|----------|--------|
| Rust | Arc / Mutex + 借用検査 | コンパイル時検査 | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 借用トークン** | **RAII** | **タスク境界解放 / タスク間 lint / 標準ライブラリ Weak** | **低** |

## 权衡

### メリット

1. **統一**：`&T`/`&mut T` は通常の型であり、特殊言語機能ではない。RFC-010 の `name: type = value` と完全に一貫
2. **シンプル**：ライフタイムなし、借用検査は型システム命題に次元削減。`&T` は複製可能、`&mut T` は複製不可——2つのタイプ属性
3. **强大**：参照の返値、構造体格納、クロージャキャプチャが可能——Rust と同等の表現力
4. **コンパイラが高い知的**: ref が自動的に Rc/Arc を選択、呼び出し側が自動的に借用を選択
5. **决定的**：ref は常に保持であり、こっそり弱参照にならない
6. **高性能**：Move ゼロコピー、トークンゼロオーバーヘッド（ゼロサイズ型、コンパイル後に消滅）
7. **柔軟**：`unsafe + *T` がシステム級プログラミングをサポート

### デメリット

1. **総称ブランドパラメータの伝染**：トークンはブランド識別子を携带し、参照を返す関数のシグネチャに追加の総称パラメータが現れる
2. **ref ランタイムオーバーヘッド**：原子操作にはコストがある（しかしこれは共有の必然的な代償）
3. **unsafe リスク**：ューザが正確性を保証する必要がある
4. **タスク間環は lint 而不是コンパイルエラー**：Rust のようにコンパイルエラーにならない、デフォルトは warn、deny をチーム設定しないと品質ゲートにならない

## 代替案

| 案 | なぜ選ばないか |
|------|--------------|
| GC | ランタイムオーバーヘッドがあり、予測不能なポーズがある |
| Rust 借用検査器 | ライフタイム `'a` が必要で、学習曲線が険しい |
| 純粋 Move | タスク間共有を処理できない |
| 生ポインタなし | システム級プログラミングができない |
| Rc/Arc をューザに公開 | 実装詳細をューザに押し付け、認知負担が増加 |
| 丐版借用（v8） | 「エスケープ禁止」策略によりクロージャキャプチャ、参照の返値など重要な表現力が犠牲になる |

### トークン競合検出：ホア論理命題

トークン競合検出の完全な方案は [RFC-009a: トークン生存期分析——ホア証明パイプに基づく](../accepted/009a-borrow-proof-pipeline.md) を参照。コアなポイント：

**トークン活性はホア論理命題である。** `{競合する ReadToken が全て死亡} write(data) {WriteToken が安全に取得}`——型検査、ューザ述語検証と RFC-027 の証明パイプを共有する。コンパイラは自動的に借用命題を生成し（`borrow_conflict`、`use_after_move`、`use_after_drop`、`mut_violation`）、パイプは Proved / Disproved / Unproven を返す。

**借用検査は消えたわけではない——次元削減された。** `BorrowChecker` は `BorrowPredicateEmitter` に変わり、检查ではなく命題を生成する。これは「型検査器」の概念と完全に並行している：型検査器が型等式命題を生成し、借用命題生成器が借用命題を生成し、同じパイプが検証する。

**ブランド ID（`#42`）こそが `'a` である。** 情報は完全に同じで、エンコーディングが異なる。`'a` は型シグネチャ에서 可见であり、`#42` はコンパイラの内部にある。新しい分析を発明したわけではない——ライフタイムを型レイヤーから証明レイヤーへと次元削減した。

**アルゴリズム概要**（詳細 は RFC-009a）：
- ブランドツリーのプレフィックス一致 → 競合トークンの決定（O(depth)、深さ ≤ 3）
- 逆向き BFS → 消費者から出发し、break でバックエッジを切断、構造分析で 95%+ のシナリオをカバー（高速パス）
- SMT 論理切断 → while + パス条件がある時のみ呼び出す（低速パス、極めてまれ）
