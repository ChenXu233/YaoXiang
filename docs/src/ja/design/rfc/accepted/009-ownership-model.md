# RFC-009: 所有権モデルの設計

## 要約

本文書は YaoXiang プログラミング言語の**所有権モデル（Ownership Model）**を定義する。

**中核設計——5 つの概念、1 つのグラデーション**：

```
ちら見/その場変更     持ち去り        共有保持          複製する         システムレベル
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         ゼロコピー      コンパイラ自動   明示的ディープコピー  *T
  ゼロサイズトークン デフォルト    Rc/Arc を選択   ユーザーが責任を持つ
  型属性で自然に                                     
  権限を推論                                          
```

- **Move（デフォルト）**：代入/引数渡し/戻り値 = 所有権移転、ゼロコピー、RAII による自動解放
- **`&T` / `&mut T`（借用トークン）**：ゼロサイズのコンパイル時トークン型。`&T` は複製可能（共有読み取り）、`&mut T` は線形（排他的書き込み）。権限は型属性により自然に推論され、特別なルールは不要。戻り値にでき、構造体に格納でき、closure にキャプチャされることもできる。
- **`ref` キーワード**：スコープをまたいで共有する。コンパイラが自動的に Rc（タスクをまたがない）か Arc（タスクをまたぐ）かを選択する
- **`clone()`**：明示的なディープコピー
- **`unsafe` + `*T`**：生ポインタ、システムレベルのエスケープハッチ

**除去された複雑性**：
- ❌ ライフタイム `'a` なし
- ❌ 独立した借用検査フレームワークなし（借用衝突はホーア命題に次元下げ、型検査と証明パイプラインを共有）
- ❌ GC なし
- ❌ 「エスケープ禁止」のような特別ルールなし（トークンは通常の型であり、スコープは型システムで統一的に処理される）
- ❌ ユーザーは Rc/Arc の違いを知る必要がない（コンパイラが自動選択）

> **プログラミング負担**：`&T` は複製可能、`&mut T` は複製不可——2 つの型属性、0 つの特別ルール、コンパイラは完全自動。
> **性能保証**：Move ゼロコスト、トークンゼロコスト（ゼロサイズ型、コンパイル後に消滅）、ref は必要に応じたコスト、GC 停止なし。

## 動機

### なぜ所有権モデルが必要か？

| 言語 | メモリ管理 | 問題 |
|------|----------|------|
| C/C++ | 手動管理 | メモリリーク、dangling pointer、ダブルフリー |
| Java/Python | GC | レイテンシ変動、メモリオーバーヘッド、予測不能な停止 |
| Rust | 所有権 + 借用検査 | ライフタイム `'a` の学習コストが急峻 |
| **YaoXiang** | **Move + Token + ref** | **単純、決定的、GC なし** |

### 設計目標

```yaoxiang
# 1. デフォルトは Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p                         # Move、p はもう読めない

# 2. &T / &mut T 借用トークン（ゼロコスト、型属性で自然に権限を推論）
print_info(p2)                 # コンパイラが自動的に &Point トークンを生成し、使用後解放
shift(p2, 1.0, 1.0)           # コンパイラが自動的に &mut Point トークンを生成

# 3. ref = 共有（コンパイラが自動的に Rc/Arc を選択）
shared = ref p2                # スコープをまたいで保持
spawn { use(shared) }          # コンパイラ：タスクをまたぐ → Arc

# 4. clone() = 明示的複製
backup = p2.clone()            # ディープコピー、独立

# 5. unsafe + *T = システムレベル
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rust との核心的差異

| 特性 | Rust | YaoXiang |
|------|------|----------|
| デフォルト意味論 | 借用 `&T`（明示的な `.clone()` が必要） | **Move（値渡し、ゼロコピー）** |
| 借用 | `&T`/`&mut T`、戻り値可、ライフタイムが必要 | **`&T`/`&mut T` ゼロサイズトークン、Dup/Linear 型属性で自然に推論** |
| 共有メカニズム | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが自動的に Rc/Arc を選択）** |
| 複製 | `clone()` | `clone()` |
| 生ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用検査 | 大域的推論 | **型検査器が自動的に借用命題を生成し、証明パイプラインで統一検証** |
| 循環参照 | 手動 Weak | **タスク終了時に統一解放 / タスク跨ぎ lint / 標準ライブラリ Weak** |

## 提案

### 1. Move（デフォルトの所有権移転）

```yaoxiang
# ルール：代入 / 引数渡し / 戻り値 = Move、ゼロコピー

p: Point = Point(1.0, 2.0)
p2 = p                           # Move、p はもう読めない

# 変数は再代入可能（Python スタイル、シャドウイングなし）
p = Point(3.0, 4.0)              # p を再バインド、型は一貫している必要がある

# 関数引数：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move で返す
}

# 関数の戻り値：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move で返す、ゼロコピー
}
```

**特徴**：
- ゼロコピー（コンパイラがポインタを移動）
- 移動後、元のバインドは読み取り不可（コンパイルエラー）
- RAII：スコープ終了時に自動解放
- 関数シグネチャ `(T) -> T` がそのままドキュメントになる——T を消費し、T を返す

---

### 2. &T / &mut T（借用トークン）

**核心原則：`&T` と `&mut T` はゼロサイズのコンパイル時トークン型である。これらは「参照」ではなく、「アクセス権限の型レベル証明」である。**

#### 2.1 2 つの型属性

```
&T      →  ゼロサイズ、ソースデータを凍結する（ReadToken の生存中は WriteToken を禁止）、
          凍結保証の下で複数の読み取り専用ビューが安全 → 複製可能（Dup）
&mut T  →  ゼロサイズ、排他的読み書き（WriteToken の生存中は他のあらゆるトークンを禁止）、
          排他的アクセス下では複製は無意味 → 線形（Dup なし）
```

**因果関係を逆転させてはならない：凍結が原因、Dup が結果である。** `&T` が Dup を実装しているから共存できる——のではなく、データが凍結される（突然変異の可能性がない）ため、複数の読み取り専用ビューが安全であり、Dup が実現可能なのである。もし Dup を定義とし、衝突検査を「追加のパッチ」と捉えれば、設計は誤っている。

#### 2.2 基本的な使い方

```yaoxiang
# メソッド側：パラメータの型を宣言し、必要な権限を決定する
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # &Point トークンが読み取り権限を付与
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # &mut Point トークンが書き込み権限を付与
    self.y = self.y + dy
}

# 呼び出し側：コンパイラが自動的に借用または Move を選択
p = Point(1.0, 2.0)
p.print()                          # コンパイラが自動的に &Point トークンを生成
p.shift(1.0, 1.0)                  # コンパイラが自動的に &mut Point トークンを生成
p.print()                          # OK、前のトークンは shift 呼び出し終了時に解放済み

# 自由関数も同様
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 2 つの &Point トークンが共存——Dup 型
}
d = distance(p, p2)
```

#### 2.3 なぜ「エスケープ禁止」が不要なのか

RFC-009 v8 は `&T`/`&mut T` に 3 つの特別ルールを課していた——パラメータとしてのみ使用可、戻り値不可、構造体への格納不可。これは「借用」概念にパッチを当てていた。

トークンシステムにはこれらのルールは不要である。トークンは**通常の型**であり、他のすべての型と同じスコープルールに従う。

**参照の返却——自然にサポート**：

```yaoxiang
# ✅ トークンは戻り値とともに伝播する
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # 子トークンと親トークンがともに返される
}

# 使用
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # トークンは呼び出し元に返される
print(px_ref)               # OK、トークンはまだスコープ内にある
```

**構造体への格納——自然にサポート**：

```yaoxiang
# ✅ 構造体はトークンをフィールドとして保持する
Window: Type = {
    target: Point,
    view: &Point,      # トークンフィールド——target への読み取り専用ビューを保持
}

# view のトークンは target から派生し、Window は両方の所有権を保持する
# Window が存在する限り、view トークンは有効である
```

**closure のキャプチャ——自然にサポート**：

```yaoxiang
# ✅ closure は他のあらゆる値と同様にトークンをキャプチャする
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    # closure は threshold の &Float トークンをキャプチャする（Dup 型、closure へ自由に複製可）
    items.filter(|p| p.x > threshold)
}

# これは RFC-009 v8 では不可能だった——v8 は closure が借用をキャプチャすることを禁止していた
```

**タスクをまたぐ——トークンはスレッドを通過できない**：

```yaoxiang
# ❌ トークンはタスク境界をまたげない
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ コンパイルエラー：トークンはタスクをまたいで渡せない
}

# これは特別ルールではない——トークンはコンパイル時の権限証明であり、タスクをまたぐ共有には ref を使用すること
# タスクをまたぐ共有が必要な場合は、ref を使用すること
```

**トークンは ref できない**：

```yaoxiang
# ❌ トークンは権限証明であり、所有権ではない
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ コンパイルエラー：&T は所有可能な型ではない
}
```

#### 2.4 トークンのライフタイム

トークンのライフタイムは**通常のスコープルール**によって決定され、ライフタイムパラメータは不要である：

- 関数パラメータ内のトークン：呼び出し中に生存し、呼び出し終了後に解放される
- 返されるトークン：所有権が呼び出し元へ移転する
- 構造体に格納されたトークン：構造体とともに生存する
- closure にキャプチャされたトークン：closure とともに生存する

コンパイラは `'a` 注釈を必要としない、なぜならトークンは**値**であり、値のライフタイムは所有権システム（Move/RAII）によって統一的に管理されるからだ。**借用問題を所有権問題に次元を下げる。**

#### 2.5 トークン衝突検出

トークン衝突検出は**ホーア論理命題**であり、独立したフロー依存解析ではない。

```
{衝突する ReadToken がすべて死亡} write(data) {WriteToken は安全に取得可能}
```

型検査やユーザ述語検証と RFC-027 の証明パイプラインを共有する。コンパイラは借用命題（`borrow_conflict`、`use_after_move`、`use_after_drop`、`mut_violation`）を自動生成し、パイプラインに送る。パイプラインは Proved / Disproved / Unproven を返す。

```yaoxiang
# ❌ &mut トークンは線形であり、複製できない
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move、p はもう読めない
    p.x = 10.0                      # ❌ コンパイルエラー：WriteToken は移動済み
}

# ✅ &T トークンは Dup 型であり、自由に複製可能
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK、&T は Dup 型
    print(p.x)                      # OK
    print(p2.x)                     # OK、2 つの読み取り専用トークンが共存
}
```

**借用検査は消えていない——次元が下がったのである。** 既存の `BorrowChecker` は `BorrowPredicateEmitter`（命題生成器）となり、生成された借用命題は他の型命題と同じ証明パイプラインを共有する。これは型検査器の概念と完全に平行している：型検査器は型等式命題を生成し、借用命題生成器は借用命題を生成し、同一のパイプラインで検証される。詳細設計は [RFC-009a](../accepted/009a-borrow-proof-pipeline.md) を参照。

#### 2.7 コンパイラ内部：ブランド機構

ユーザーはブランドに一切触れない。コンパイラは内部で各トークンにコンパイル時一意識別子を割り当てる：

```
ユーザーが見るもの     コンパイラ内部表現
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N はコンパイル時の一意な整数
&mut Point     →  WriteToken(Point, #M)   // #M はコンパイル時の一意な整数
```

ブランドの用途：
- **偽造防止**：トークンは所有者のカプセルからのみ取得でき、凭空に構築できない
- **関連追跡**：`&Point` から `&Float`（フィールドアクセス）を派生するとき、`&Float` は派生ブランド（`#N.field_x`）を携带し、コンパイラは親トークンまで追跡可能
- **衝突検出**：同源の `WriteToken` と派生した `ReadToken` は同時にアクティブにできない

ブランドは単相化とインライン化後に完全に消滅し、生成される機械語には存在しない。**ランタイムオーバーヘッドはゼロ。**

#### 2.8 自動借用選択ルール

呼び出し側コンパイラは以下の優先順位で自動的に選択する：

```
1. 実引数が後続でも使用される → トークン生成を優先（メソッドシグネチャに応じて &T または &mut T）
2. 実引数が後続で使用されない → Move
3. 優先マッチング順序：&T < &mut T < Move
```

```yaoxiang
# 例：自動選択
p = Point(1.0, 2.0)
p.print()        # print は &self を宣言 → コンパイラが &Point トークンを生成
p.shift(1.0, 1.0) # shift は &mut self を宣言 → コンパイラが &mut Point トークンを生成
p2 = p           # Move、p はもう使用されない
```

#### 2.9 RFC-009 v8 簡易借用との比較

| 特性 | 簡易借用 (v8) | 借用トークン (v9) |
|------|--------------|--------------|
| 参照の返却 | ❌ ハードコードで禁止 | ✅ トークンは戻り値とともに伝播 |
| 構造体への格納 | ❌ ハードコードで禁止 | ✅ トークンは構造体フィールドになれる |
| closure のキャプチャ | ❌ ハードコードで禁止 | ✅ closure はトークン値をキャプチャできる |
| 特別ルール | 3 つ（パラメータのみ/返却不可/格納不可） | 0 つ——型属性で自然に推論 |
| 借用検査 | 専用の交差借用検査 | 型検査器のフロー依存活性解析 |
| ライフタイム注釈 | 不要 | 不要 |
| ランタイムオーバーヘッド | ゼロ | ゼロ（ゼロサイズ型、コンパイル後に消滅） |
| エラーメッセージ | 「借用はエスケープできない」 | 「WriteToken(#3) は移動済み」（通常の型エラー） |
| ユーザーのメンタルモデル | 「借用」の特別な地位を理解する | `&T` は複製可能、`&mut T` は複製不可 |

### 3. ref キーワード（コンパイラの自動最適化）

`ref` はスコープをまたいで共有する唯一の方法である。内部実装が Rc か Arc かをユーザーが気にする必要はない。

#### 3.1 基本的な使い方

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共有、コンパイラが自動的に実装を選択

# タスクをまたぐ共有
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # コンパイラ：タスクをまたぐ → Arc
    spawn { use(data) }           # コンパイラ：タスクをまたぐ → Arc
}

# 単一タスク内共有
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # コンパイラ：タスクをまたがない → Rc
}
```

**ユーザーのメンタルモデル**：`ref` = 共有保持。それだけで十分。

#### 3.2 コンパイラのエスケープ分析：Rc vs Arc

```
ref のデータフロー解析：

他のタスクにエスケープしない → Rc（非アトミック参照カウント、低オーバーヘッド）
他のタスクにエスケープする   → Arc（アトミック参照カウント、スレッドセーフ）
```

#### 3.3 循環検出戦略

```
タスク内循環 → 静かに許可。
  ├── 各タスクには明確なライフサイクル境界がある——タスク終了時にすべてのリソース（ref 循環を含む）が統一解放される。
  ├── 長時間稼働するサービスはリクエスト/接続ごとに子タスクを作成すべきである——子タスク終了時に自動回収され、累積リークは発生しない。
  ├── ref は常に生存し続け、意味論に水増しなし。
  └── ユーザーはタスク内で双方向の強参照（例えばグラフ計算の中間状態）を構築する権利を持つ。

タスク跨ぎ循環 → lint（デフォルト warn、設定可能）。
  ├── プログラムの動作は正しく、実際にリークはしない（親タスク終了時に子タスクのリソースは全解放）。
  ├── しかし、タスク跨ぎ強参照は所有権境界が曖昧であることを意味し、立ち止まって再考する価値がある。
  ├── デフォルトは warn レベルで、コンパイルは通るが警告が出る。
  └── チームはプロジェクト設定で deny にでき、CI 品質ゲートに組み込める。
```

**Lint レベル**（Rust clippy と同様）：

| レベル | 動作 | シナリオ |
|------|------|------|
| `allow` | チェックしない | 個人プロジェクト |
| `warn`（デフォルト） | コンパイルは通る、警告あり | 開発段階 |
| `deny` | コンパイル失敗 | チームの CI 品質ゲート |
| `forbid` | コンパイル失敗、上書き不可 | 組織レベルの強制ルール |

```yaoxiang
# タスク内循環：静かに許可、双方向強参照
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # 循環。タスク終了時に統一解放される。
}

# タスク跨ぎ循環：lint（デフォルト warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: タスク跨ぎ循環参照
    }
}
```

**プロジェクト設定例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # タスク跨ぎ循環は CI で直接拒否
```

| 循環タイプ | 動作 | 理由 |
|--------|------|------|
| タスク内 ref 循環 | チェックなし | ユーザーの権利、タスク終了時に統一解放 |
| タスク跨ぎ ref 循環 | lint（デフォルト warn） | 再考を促す、設定で deny 可能 |

#### 3.4 Weak：標準ライブラリ提供

```yaoxiang
use std.rc.Weak

# 上級ユーザーが明示的に選択
a.next = ref b
b.prev = Weak.new(a.next)        # ユーザーが明示的にどちらの方向を弱くするか制御
```

**`Weak` は言語組み込みではなく、標準ライブラリの型である。** 日常的には `ref` を使えば十分。メモリを細かく制御したい上級ユーザーが手動で `Weak` を導入する。

#### 3.5 借用トークン vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 役割 | ちら見/その場変更 | 共有保持 |
| 範囲 | トークン値のスコープに従う | スコープをまたぐ |
| コスト | ゼロコスト（ゼロサイズ型） | Rc または Arc（コンパイラが選択） |
| エスケープ | 可（トークンは戻り値/構造体/closure とともに伝播） | 本来エスケープ目的 |
| タスク跨ぎ | 不可（トークンはコンパイル時の権限証明であり、タスクをまたいで渡せない） | 可（コンパイラが自動的に Arc を選択） |
| 循環 | 関与しない | タスク内は静かに許可、タスク跨ぎは lint |

### 4. clone() —— 明示的複製

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # ディープコピー
# p と p2 は独立しており、相互に影響しない
```

**いつ使うか**：元の値を保持したいが、Move でも共有でもないシナリオ。

### 5. unsafe + 生ポインタ（システムレベルプログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 生ポインタ
    (*ptr).x = 0.0                # 参照外し（ユーザーが安全性を保証）
    ptr2 = ptr + 1                # ポインタ演算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ユーザーは dangling しないこと、解放後の使用をしないことを保証する
- FFI、メモリ操作などのシステムレベルプログラミングに使用する

### 6. 所有権グラデーション概観

```
  借用トークン（ゼロコスト）     Move（ゼロコスト）      共有（必要に応じたコスト）    複製
   │                      │                  │                │
  &T 複製可能トークン    デフォルト所有権移転   ref Rc/Arc       clone()
  &mut T 線形トークン    チェーン消費と還流     コンパイラ自動選択   明示的ディープコピー
   │                      │                  │                │
  トークン値スコープ      スコープ内           スコープをまたぐ   いつでも
  戻り値/構造体格納可     T -> T 還流          ref タスク跨ぎ → Arc  独立コピー
  closure でキャプチャ可   T -> Void 消費      ref タスク内 → Rc
  ゼロサイズでコンパイル後消滅                 タスク内循環は静かに
                                            タスク跨ぎ循環は lint
                                            標準ライブラリ Weak でエスケープ
```

## 総合例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：読み取り専用トークン
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：可変トークン
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消費して還流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 持ち去り、変更、返す
    }

    # 参照の返却：トークンは戻り値とともに伝播
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# closure がトークンをキャプチャ（v8 では不可能だった能力）
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}

# 総合使用
p = Point(1.0, 2.0)
p.print()                           # &Point トークン
p.shift(1.0, 1.0)                   # &mut Point トークン
p = p.scale(2.0)                    # Move → 還流
shared = ref p                      # ref 共有
spawn { use(shared) }

# clone で独立コピー
backup = p.clone()

# タスク内循環：静かに許可
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # 循環、タスク終了時に統一解放

# unsafe システムレベル
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Dup 型属性

`Dup`（Duplicable）はコンパイラが自動管理する型属性であり、意味は**シャローコピー**である：代入/引数渡し時に複製されるのはハンドル/トークンであり、基底データは共有される。これは Move（所有権移転）および Clone（明示的ディープコピー、独立コピー作成）と 3 段階のグラデーションを形成する。

**Dup と Clone は直交した概念である**——Dup はハンドルを複製してデータを共有し、Clone は独立コピー作成する。1 つの型が Dup と Clone の両方をサポートすることも、どちらも片方だけをサポートすることもできる。

| 型 | Dup | Clone | 説明 |
|------|-----|-------|------|
| `&T` | ✅（トークンを複製、複数の視点が同じデータを指す） | ✅ | 読み取り専用トークン |
| `ref T` | ✅（参照カウント+1、ヒープデータを共有） | ✅ | 共有保持（コンパイラが自動的に Rc/Arc を選択） |
| String, Bytes | ✅（内部参照カウント、ハンドルを複製して基底バッファを共有） | ✅ | 文字列/バイト |
| `&mut T` | ❌（線形、排他的） | ❌ | 可変トークン |
| `*T` | ❌ | ❌ | 生ポインタ |
| struct | 派生（全フィールドが Dup のとき自動派生） | ✅ | 構造体 |

**プリミティブ値型**（Int, Float, Bool, Char）の代入動作はコンパイラ組み込みの値コピーであり、2 つの値は完全に独立しており、シャローコピーではない。これらは Dup 型属性には属さず、コンパイラのネイティブ処理である。

## 性能分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | ゼロ | ポインタ移動 |
| `&T` / `&mut T` | ゼロ | ゼロサイズ型、コンパイル後に消滅、ランタイムオーバーヘッドゼロ |
| `ref`（タスク跨ぎなし）| 低 | Rc にコンパイル、非アトミック操作 |
| `ref`（タスク跨ぎ）| 中 | Arc にコンパイル、アトミック操作 |
| `clone()` | 型による | 小オブジェクトは高速、大オブジェクトは低速 |
| `unsafe + *T` | ゼロ | 直接メモリ操作 |

### 比較

| 言語 | 共有メカニズム | メモリ管理 | 循環処理 | 複雑度 |
|------|----------|----------|----------|------|
| Rust | Arc / Mutex + 借用検査 | コンパイル時検査 | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 借用トークン** | **RAII** | **タスク境界解放 / タスク跨ぎ lint / 標準ライブラリ Weak** | **低** |

## トレードオフ

### 利点

1. **統一性**：`&T`/`&mut T` は通常の型であり、特別な言語機能ではない。RFC-010 の `name: type = value` と完全一致する
2. **単純さ**：ライフタイムなし、借用検査は型システム命題に次元下げ。`&T` は複製可能、`&mut T` は複製不可——2 つの型属性
3. **強力**：参照の返却、構造体への格納、closure のキャプチャが可能——表現能力は Rust と同等
4. **コンパイラの賢さ**：ref は自動的に Rc/Arc を選択、呼び出し側は自動的に借用を選択
5. **決定論的**：ref は生存し続け、密かに弱参照にはならない
6. **高性能**：Move ゼロコピー、トークンゼロコスト（ゼロサイズ型、コンパイル後に消滅）
7. **柔軟**：`unsafe + *T` でシステムレベルプログラミングをサポート

### 欠点

1. **ジェネリックブランドパラメータの伝播**：トークンはブランド識別子を持ち、参照を返す関数のシグネチャに追加のジェネリックパラメータが現れる
2. **ref のランタイムオーバーヘッド**：アトミック操作にはコストがある（しかしこれは共有の本質的代償）
3. **unsafe のリスク**：ユーザーが正当性を保証しなければならない
4. **タスク跨ぎ循環は lint でありコンパイルエラーではない**：Rust のようにコンパイルエラーとはならず、デフォルト warn、チームが deny に設定して初めて品質ゲートになる

---

## 代替案

| 代替案 | 不採用の理由 |
|------|--------------|
| GC | ランタイムオーバーヘッドがあり、停止が予測不能 |
| Rust の借用検査器 | ライフタイム `'a` が必要、学習曲線が急峻 |
| 純粋な Move | 並列共有を扱えない |
| 生ポインタなし | システムレベルプログラミングが不可能 |
| Rc/Arc をユーザーに公開 | 実装詳細をユーザーに押し付け、認知負担が増加 |
| 簡易借用（v8） | エスケープ禁止戦略が closure キャプチャや参照返却などの重要な表現能力を犠牲にする |

### トークン衝突検出：ホーア論理命題

トークン衝突検出の完全な方案は [RFC-009a: トークンライフタイム解析——ホーア証明パイプラインに基づく](../accepted/009a-borrow-proof-pipeline.md) を参照。核心要点：

**トークン活性はホーア論理命題である。** `{衝突する ReadToken がすべて死亡} write(data) {WriteToken は安全に取得可能}`——型検査やユーザ述語検証と RFC-027 の証明パイプラインを共有する。コンパイラは借用命題（`borrow_conflict`、`use_after_move`、`use_after_drop`、`mut_violation`）を自動生成し、パイプラインが Proved / Disproved / Unproven を返す。

**借用検査は消えていない——次元が下がったのである。** `BorrowChecker` は `BorrowPredicateEmitter` となり、検査を実行するのではなく命題を生成する。これは「型検査器」の概念と完全に平行している：型検査器は型等式命題を生成し、借用命題生成器は借用命題を生成し、同一のパイプラインで検証される。

**ブランド ID（`#42`）は `'a` そのものである。** 情報は完全に同じで、エンコーディングが異なる。`'a` は型シグネチャに現れるが、`#42` はコンパイラ内部にある。新しい解析を発明したのではなく、ライフタイムを型層から証明層に次元下げしたのである。

**アルゴリズム概要**（詳細は RFC-009a 参照）：
- ブランドツリー前置一致 → 衝突トークンの特定（O(depth)、深さ ≤ 3）
- 逆方向 BFS → 消費者から出発、break で逆辺を切断、構造解析で 95%+ のシナリオをカバー（高速経路）
- SMT 論理切断 → while + パス条件のときのみ呼び出し（低速経路、極めて稀）
