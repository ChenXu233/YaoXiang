# RFC-009: 所有权模型設計

## 概要

本文書はYaoXiangプログラミング言語の**所有権モデル（Ownership Model）**を定義する。

**核心設計——5つの概念、1つのグラデーション**：

```
一眼見る/その場変更   持ち出し          共有保持            コピーする         システムレベル
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         ゼロコピー     コンパイラ自動   明示的ディープコピー  *T
  ゼロサイズトークン デフォルト   Rc/Arcを選択                    ユーザー責任
  型属性から自然に
  権限を推論                              
```

- **Move（デフォルト）**：代入/引数渡し/戻り値 = 所有権移転、ゼロコピー、RAIIによる自動解放
- **`&T` / `&mut T`（借用トークン）**：ゼロサイズのコンパイル時トークン型。`&T` は複製可能（共有読み取り）、`&mut T` は線形（排他的可変）。権限は型属性から自然に推論され、特殊なルールは不要。戻り値や構造体フィールドに格納可能。
- **`ref` キーワード**：スコープをまたいで共有。コンパイラが自動的にRc（タスクをまたがない）かArc（タスクをまたぐ）を選択する
- **`clone()`**：明示的なディープコピー
- **`unsafe` + `*T`**：生ポインタ、システムレベルのエスケープハッチ

**取り除かれた複雑さ**：
- ❌ ライフタイム `'a` なし
- ❌ 独立した借用検査フレームワークなし（借用衝突はホーア命題に次元削減され、型検査と証明パイプラインを共有）
- ❌ GCなし
- ❌「エスケープ禁止」のような特殊ルールなし（トークンは通常の型であり、スコープは型システムで統一的に処理される）
- ❌ ユーザーがRcとArcの違いを知る必要がない（コンパイラが自動選択）

> **プログラミング負担**：`&T` は複製可能、`&mut T` は複製不可——2つの型属性、0の特殊ルール、コンパイラ完全自動化。
> **性能保証**：Moveはゼロコスト、トークンはゼロコスト（ゼロサイズ型、コンパイル後に消滅）、refは必要に応じて課金、GCの一時停止なし。

## 動機

### なぜ所有権モデルが必要か？

| 言語 | メモリ管理 | 問題 |
|------|----------|------|
| C/C++ | 手動管理 | メモリリーク、ダングリングポインタ、二重解放 |
| Java/Python | GC | レイテンシ変動、メモリオーバーヘッド、予測不可能な停止 |
| Rust | 所有権 + 借用検査 | ライフタイム `'a` の学習曲線が急峻 |
| **YaoXiang** | **Move + Token + ref** | **シンプルで決定論的、GCなし** |

### 設計目標

```yaoxiang
# 1. デフォルトMove（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p                         # Move、p は再読込不可

# 2. &T / &mut T 借用トークン（ゼロコスト、型属性から自然に権限を推論）
print_info(p2)                 # コンパイラが自動的に &Point トークンを作成し、使用後解放
shift(p2, 1.0, 1.0)           # コンパイラが自動的に &mut Point トークンを作成

# 3. ref = 共有（コンパイラが自動的にRc/Arcを選択）
shared = ref p2                # スコープをまたいで保持
spawn { use(shared) }          # コンパイラ：タスクをまたぐ → Arc

# 4. clone() = 明示的コピー
backup = p2.clone()            # ディープコピー、独立

# 5. unsafe + *T = システムレベル
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rustとの核心的な違い

| 特性 | Rust | YaoXiang |
|------|------|----------|
| デフォルトセマンティクス | 借用 `&T`（明示的に `.clone()` が必要） | **Move（値渡し、ゼロコピー）** |
| 借用 | `&T`/`&mut T`、戻り値可能、ライフタイムが必要 | **`&T`/`&mut T` ゼロサイズトークン、Dup/Linear 型属性から自然に推論** |
| 共有メカニズム | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが自動的にRc/Arcを選択）** |
| コピー | `clone()` | `clone()` |
| 生ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用検査 | グローバル推論 | **型検査器が自動的に借用命題を生成し、証明パイプラインで統一検証** |
| 循環参照 | 手動 Weak | **タスク終了時に統一解放 / タスク跨ぎはlint / 標準ライブラリ Weak** |

## 提案

### 1. Move（デフォルトの所有権移転）

```yaoxiang
# ルール：代入 / 引数渡し / 戻り値 = Move、ゼロコピー

p: Point = Point(1.0, 2.0)
p2 = p                           # Move、p は再読込不可

# 変数は再代入可能（Pythonスタイル、シャドーイングなし）
p = Point(3.0, 4.0)              # p を再バインド、型は一致必須

# 関数引数：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move で返却
}

# 関数戻り値：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move で返却、ゼロコピー
}
```

**特徴**：
- ゼロコピー（コンパイラがポインタを移動）
- 移動後、元のバインディングは読み取り不可（コンパイルエラー）
- RAII：スコープ終了時に自動解放
- 関数シグネチャ `(T) -> T` 自体がドキュメント——T を消費し、T を返す

---

### 2. &T / &mut T（借用トークン）

**核心原則：`&T` と `&mut T` はゼロサイズのコンパイル時トークン型である。これらは「参照」ではなく、「アクセス権限の型レベル証明」である。**

#### 2.1 2つの型属性

```
&T      →  ゼロサイズ、元データを凍結（ReadToken の生存中は WriteToken を禁止）、
          凍結保証により複数の読み取り専用ビューが安全 → 複製可能（Dup）
&mut T  →  ゼロサイズ、排他的読み書き（WriteToken の生存中は他のすべてのトークンを禁止）、
          排他的アクセスの下では複製は無意味 → 線形（Dup なし）
```

**因果関係を逆転させてはならない：凍結が原因、Dupが結果。** `&T` が Dup を実装しているから共存できるわけではない——データが凍結される（突然変異の可能性がない）ため、複数の読み取り専用ビューが安全であり、Dupが実装可能になる。Dup を定義とし、衝突検査を「追加のパッチ」として扱うと、設計を誤ることになる。

#### 2.2 基本的な使用方法

```yaoxiang
# メソッド側：パラメータ型を宣言し、必要な権限を決定
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # &Point トークンが読み取り権限を付与
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # &mut Point トークンが書き込み権限を付与
    self.y = self.y + dy
}

# 呼び出し側：コンパイラが自動的に借用またはMoveを選択
p = Point(1.0, 2.0)
p.print()                          # コンパイラが自動的に &Point トークンを作成
p.shift(1.0, 1.0)                  # コンパイラが自動的に &mut Point トークンを作成
p.print()                          # OK、前のトークンは shift 呼び出し終了とともに解放済み

# 自由関数も同様
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 2つの &Point トークンが共存——Dup 型
}
d = distance(p, p2)
```

#### 2.3 なぜ「エスケープ禁止」は不要か

RFC-009 v8 は `&T`/`&mut T` に3つの特殊ルールを課していた——パラメータとしてのみ使用可能、戻り値不可、構造体に格納不可。これは「借用」概念にパッチを当てるものであった。

トークンシステムにはこれらのルールは不要。トークンは**通常の型**であり、他のすべての型と同じスコープルールに従う。

**参照を返す——自然にサポート**：

```yaoxiang
# ✅ トークンは戻り値とともに伝播する
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # 子トークンと親トークンを一緒に返す
}

# 使用
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # トークンが呼び出し元に返される
print(px_ref)               # OK、トークンはまだスコープ内
```

**構造体に格納——自然にサポート**：

```yaoxiang
# ✅ 構造体はトークンをフィールドとして保持できる
Window: Type = {
    target: Point,
    view: &Point,      # トークンフィールド——target への読み取り専用ビューを保持
}

# view のトークンは target から派生し、Window は両方の所有権を保持する
# Window が存在する限り、view トークンは有効
```

#### 2.3 クロージャとLambdaの明示的パラメータ

Lambda は関数値であり——返却、格納、現在のスコープ外への送出が可能である。したがって、Lambda は**外側のローカル変数を暗黙的にキャプチャしない**。外側のデータが必要な場合は、明示的パラメータとして渡す：

```yaoxiang
# ✅ Lambda は明示的パラメータを使用
double: (x: Int) -> Int = (x) => x * 2
filter_by: (items: List(Int), f: (Int) -> Bool) -> List(Int) = { ... }

# ✅ spawn { } はこのルールの影響を受けない——spawn は即座に実行される並行ブロックで、親タスクはブロックして待機する
shared = ref data
spawn { use(shared) }

# ❌ Lambda は外側の変数を暗黙的にキャプチャできない
x = 42
f = () => { x + 1 }  # コンパイルエラー：x はスコープ内にない

# ✅ 正しい方法：明示的にパラメータを渡す
f = (x) => { x + 1 }
f(x)
```

**spawn { } は関数値ではない。** spawn でマークされたブロックは if/while の body と同様——即座に実行され、親スタックフレームの生存期間中に完了する。spawn body は外側の変数に通常通りアクセスできる。

**タスク跨ぎ——トークンはスレッドをまたげない**：

```yaoxiang
# ❌ トークンはタスク境界をまたげられない
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ コンパイルエラー：トークンはタスクをまたいで渡せない
}

# これは特殊ルールではない——トークンはコンパイル時の権限証明であり、タスクをまたぐ共有には ref を使用すること
# タスクをまたぐ共有が必要な場合は、ref を使用してください
```

**トークンは ref できない**：

```yaoxiang
# ❌ トークンは権限証明であり、所有権ではない
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ コンパイルエラー：&T は所有可能な型ではない
}
```

#### 2.4 トークンのライフタイム

トークンのライフタイムは**通常のスコープルール**によって決定され、ライフタイムパラメータは不要：

- 関数パラメータ内のトークン：呼び出し中に生存し、呼び出し終了後に解放
- 戻り値のトークン：所有権を呼び出し元に移転
- 構造体に格納されたトークン：構造体とともに生存

コンパイラは `'a` 注釈を必要としない。なぜなら、トークンは**値**であり、値のライフタイムは所有権システム（Move/RAII）によって統一的に管理されるからである。**借用問題を所有権問題に次元削減する。**

#### 2.5 トークン衝突検出

トークン衝突検出は**ホーア論理命題**であり、独立したフロー依存解析ではない。

```
{衝突する ReadToken がすべて死亡} write(data) {WriteToken は安全に取得可能}
```

型検査やユーザー述語検証と RFC-027 の証明パイプラインを共有する。コンパイラが自動的に借用命題（`borrow_conflict`、`use_after_move`、`use_after_drop`、`mut_violation`）を生成し、パイプラインに送る。パイプラインは Proved / Disproved / Unproven を返す。

```yaoxiang
# ❌ &mut トークンは線形で、複製できない
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move、p は再読込不可
    p.x = 10.0                      # ❌ コンパイルエラー：WriteToken はすでに移動済み
}

# ✅ &T トークンは Dup 型で、自由に複製可能
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK、&T は Dup 型
    print(p.x)                      # OK
    print(p2.x)                     # OK、2つの読み取り専用トークンが共存
}
```

**借用検査は消えていない——次元が削減されたのである。** 既存の `BorrowChecker` は `BorrowPredicateEmitter`（命題ジェネレータ）となり、生成された借用命題は他の型命題と同一の証明パイプラインを共有する。これは型検査器の概念と完全に並行する：型検査器は型等価命題を生成し、借用命題ジェネレータは借用命題を生成し、同一のパイプラインが検証する。詳細設計は [RFC-009a](../accepted/009a-borrow-proof-pipeline.md) を参照。

#### 2.7 コンパイラ内部：ブランドメカニズム

ユーザーはブランドに一切触れない。コンパイラは内部で各トークンにコンパイル時一意識別子を割り当てる：

```
ユーザーが見る         コンパイラ内部表現
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N はコンパイル時一意整数
&mut Point     →  WriteToken(Point, #M)   // #M はコンパイル時一意整数
```

ブランドの用途：
- **偽造防止**：トークンは所有者カプセルからのみ取得でき、凭空に構築できない
- **関連追跡**：`&Point` から `&Float`（フィールドアクセス）を派生するとき、`&Float` は派生ブランド（`#N.field_x`）を運び、コンパイラは親トークンを追跡できる
- **衝突検出**：同源の `WriteToken` と派生 `ReadToken` は同時にアクティブになれない

ブランドは単態化とインライン化後に完全に消滅し、生成された機械語には存在しない。**ランタイムオーバーヘッドはゼロ。**

#### 2.8 自動借用選択ルール

呼び出し側コンパイラは以下の優先順位で自動選択する：

```
1. 実引数が後に使用される場合 → トークン作成を優先（&T または &mut T、メソッドシグネチャによる）
2. 実引数が後に使用されない場合 → Move
3. 優先マッチング順序：&T < &mut T < Move
```

```yaoxiang
# 例：自動選択
p = Point(1.0, 2.0)
p.print()        # print は &self を宣言 → コンパイラが &Point トークンを作成
p.shift(1.0, 1.0) # shift は &mut self を宣言 → コンパイラが &mut Point トークンを作成
p2 = p           # Move、p は使用されない
```

#### 2.9 RFC-009 v8 簡素化借用との比較

| 特性 | 簡素化借用 (v8) | 借用トークン (v9) |
|------|--------------|--------------|
| 参照を返す | ❌ ハードコード禁止 | ✅ トークンは戻り値とともに伝播 |
| 構造体に格納 | ❌ ハードコード禁止 | ✅ トークンは構造体フィールドとして格納可能 |
| Lambda の明示的パラメータ | ❌ ハードコード禁止 | ✅ Lambda は明示的パラメータを使用 |
| 特殊ルール | 3つ（パラメータのみ/戻り値不可/格納不可） | 0個——型属性から自然に推論 |
| 借用検査 | 専用のクロス借用検査 | 型検査器のフロー依存活性解析 |
| ライフタイム注釈 | 不要 | 不要 |
| ランタイムオーバーヘッド | ゼロ | ゼロ（ゼロサイズ型、コンパイル後に消滅） |
| エラーメッセージ | 「借用はエスケープできない」 | 「WriteToken(#3)はすでに移動済み」（通常の型エラー） |
| ユーザーのメンタルモデル | 「借用」の特別な地位を理解する | `&T` は複製可能、`&mut T` は複製不可 |

### 3. ref キーワード（コンパイラ自動最適化）

`ref` はスコープをまたぐ共有の唯一の方法である。内部実装がRcかArcかをユーザーが気にする必要はない。

#### 3.1 基本的な使用方法

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共有、コンパイラが自動的に実装を選択

# タスクをまたぐ共有
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # コンパイラ：タスクをまたぐ → Arc
    spawn { use(data) }           # コンパイラ：タスクをまたぐ → Arc
}

# 単一タスク共有
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # コンパイラ：タスクをまたがない → Rc
}
```

**ユーザーのメンタルモデル**：`ref` = 共有保持。以上。

#### 3.2 コンパイラのエスケープ解析：Rc vs Arc

```
ref のデータフロー解析：

他のタスクにエスケープしない → Rc（非アトミック参照カウント、低オーバーヘッド）
他のタスクにエスケープする   → Arc（アトミック参照カウント、スレッドセーフ）
```

#### 3.3 循環検出戦略

```
タスク内循環 → 静的に許可。
  ├── 各タスクには明確なライフサイクル境界がある——タスク終了時にすべてのリソース（ref 循環を含む）が統一解放される。
  ├── 長時間稼働するサービスはリクエスト/接続ごとに子タスクを作成すべき——子タスク終了時に自動回収され、リークは蓄積しない。
  ├── ref は常に生存し、セマンティクスは混じりけがない。
  └── ユーザーはタスク内で双方向強参照を構築する権利を有する（例：グラフ計算の中間状態）。

タスク跨ぎ循環 → lint（デフォルト warn、設定可能）。
  ├── プログラムの挙動は正しく、実際にリークすることはない（親タスク終了時に子タスクのリソースがすべて解放される）。
  ├── ただし、タスク跨ぎの強参照は所有権境界が曖昧であることを意味し、再考する価値がある。
  ├── デフォルトは warn レベルで、コンパイルは成功するがヒントが表示される。
  └── チームはプロジェクト設定で deny にでき、CI 品質ゲートに組み込める。
```

**Lint レベル**（Rust clippy に類似）：

| レベル | 動作 | シナリオ |
|------|------|------|
| `allow` | チェックしない | 個人プロジェクト |
| `warn`（デフォルト） | コンパイル成功、ヒント表示 | 開発段階 |
| `deny` | コンパイル失敗 | チーム CI 品質ゲート |
| `forbid` | コンパイル失敗、オーバーライド不可 | 組織レベルの強制ルール |

```yaoxiang
# タスク内循環：静的に許可、双方向強参照
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # 循環。タスク終了時に統一解放。
}

# タスク跨ぎ循環：lint（デフォルト warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: タスク跨ぎ循環参照
    }
}
```

**プロジェクト設定例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # タスク跨ぎ循環は CI で直接拒否
```

| 循環タイプ | 動作 | 理由 |
|--------|------|------|
| タスク内 ref 循環 | チェックなし | ユーザーの権利、タスク終了時に統一解放 |
| タスク跨ぎ ref 循環 | lint（デフォルト warn） | 再考を促す、deny に設定可能 |

#### 3.4 Weak：標準ライブラリ提供

```yaoxiang
use std.rc.Weak

# 上級ユーザーの明示的選択
a.next = ref b
b.prev = Weak.new(a.next)        # ユーザーが明示的に弱方向を制御
```

**`Weak` は言語組み込みではなく、標準ライブラリ型である。** 日常的には `ref` で十分。メモリを細かく制御したい上級ユーザーは手動で `Weak` を導入する。

#### 3.5 借用トークン vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 役割 | 一眼見る/その場変更 | 共有保持 |
| 範囲 | トークン値のスコープに従う | スコープをまたぐ |
| コスト | ゼロコスト（ゼロサイズ型） | Rc または Arc（コンパイラが選択） |
| エスケープ | 可（トークンは戻り値/構造体/クロージャで伝播） | 元々エスケープ用に設計 |
| タスク跨ぎ | 不可（トークンはコンパイル時権限証明であり、タスクをまたいで渡せない） | 可（コンパイラが自動的に Arc を選択） |
| 循環 | 関与しない | タスク内は静的に許可、タスク跨ぎは lint |

### 4. clone() —— 明示的コピー

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # ディープコピー
# p と p2 は独立しており、互いに影響しない
```

**使用すべき時**：元の値を保持する必要があり、Move や共有が適さないシナリオ。

### 5. unsafe + 生ポインタ（システムレベルプログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 生ポインタ
    (*ptr).x = 0.0                # 逆参照（ユーザーが安全を保証）
    ptr2 = ptr + 1                # ポインタ演算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ユーザーはダングリングや解放後使用がないことを保証する責任がある
- FFI、メモリ操作などのシステムレベルプログラミングに使用

### 6. 所有権グラデーション総覧

```
  借用トークン（ゼロコスト）     Move（ゼロコスト）      共有（必要に応じて課金）    コピー
   │                      │                  │                │

  &mut T 線形トークン      チェイン消費還流     コンパイラ自動選択   明示的ディープコピー

```

## 包括的な例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：読み取り専用トークン
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：可変トークン
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消費還流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 持ち出し、変更、返却
    }

    # 参照を返す：トークンは戻り値とともに伝播
    get_x: (self: &Point) -> (&Float, &Point) = {

    }
}

double: (x: Int) -> Int = (x) => x * 2

spawn { use(shared) }

# タスク内循環：静的に許可
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # 循環、タスク終了時に統一解放

# unsafe システムレベル

    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Dup 型属性

`Dup`（Duplicable）はコンパイラが自動管理する型属性であり、意味は**シャローコピー**である：代入/引数渡し時にコピーされるのはハンドル/トークンであり、基盤データは共有される。これは Move（所有権移転）および Clone（明示的ディープコピーで独立コピー作成）と3段階のグラデーションを形成する。

**Dup と Clone は直交する概念である**——Dup はハンドルをコピーしてデータを共有し、Clone は独立コピーを作成する。1つの型が Dup と Clone の両方をサポートすることも、1つだけをサポートすることも可能。

| 型 | Dup | Clone | 説明 |
|------|-----|-------|------|
| `&T` | ✅（トークン複製、複数の視点が同じデータを指す） | ✅ | 読み取り専用トークン |
| `ref T` | ✅（参照カウント+1、ヒープデータを共有） | ✅ | 共有保持（コンパイラが自動的にRc/Arcを選択） |
| String, Bytes | ✅（内部参照カウント、ハンドルコピー、基盤バッファ共有） | ✅ | 文字列/バイト |
| `&mut T` | ❌（線形、排他的） | ❌ | 可変トークン |
| `*T` | ❌ | ❌ | 生ポインタ |
| struct | 派生（全フィールドが Dup の場合自動派生） | ✅ | 構造体 |

**プリミティブ値型**（Int, Float, Bool, Char）の代入動作はコンパイラ組み込みの値コピーであり、2つの値は完全に独立していて、シャローコピーではない。これらは Dup 型属性ではなく、コンパイラのネイティブ処理に属する。

## パフォーマンス分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | ゼロ | ポインタ移動 |
| `&T` / `&mut T` | ゼロ | ゼロサイズ型、コンパイル後に消滅、ランタイムオーバーヘッドゼロ |
| `ref`（タスク跨ぎなし）| 低 | Rc にコンパイル、非アトミック操作 |
| `ref`（タスク跨ぎ）| 中 | Arc にコンパイル、アトミック操作 |
| `clone()` | 型による | 小オブジェクトは高速、大オブジェクトは低速 |
| `unsafe + *T` | ゼロ | 直接メモリ操作 |

### 比較

| 言語 | 共有メカニズム | メモリ管理 | 循環処理 | 複雑度 |
|------|----------|----------|----------|------|
| Rust | Arc / Mutex + 借用検査 | コンパイル時検査 | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 借用トークン** | **RAII** | **タスク境界解放 / タスク跨ぎ lint / 標準ライブラリ Weak** | **低** |

## トレードオフ

### 利点

1. **統一性**：`&T`/`&mut T` は特殊言語機能ではなく通常の型。RFC-010 の `name: type = value` と完全に一致
2. **シンプルさ**：ライフタイムなし、借用検査は型システム命題に次元削減。`&T` は複製可能、`&mut T` は複製不可——2つの型属性
3. **強力**：参照返し、構造体格納、クロージャキャプチャが可能——表現能力は Rust と同等
4. **コンパイラのスマートさ**：ref が Rc/Arc を自動選択、呼び出し側が借用を自動選択
5. **決定論性**：ref は生存維持、密かに弱参照にはならない
6. **高性能**：Move はゼロコピー、トークンはゼロコスト（ゼロサイズ型、コンパイル後に消滅）
7. **柔軟性**：`unsafe + *T` がシステムレベルプログラミングをサポート

### 欠点

1. **ジェネリクスブランドパラメータの伝染**：トークンがブランド識別子を運ぶため、参照を返す関数のシグネチャに追加のジェネリクスパラメータが現れる
2. **ref のランタイムオーバーヘッド**：アトミック操作にはコストがある（ただしこれは共有の本質的代償）
3. **unsafe のリスク**：ユーザーが正確性を保証する責任がある
4. **タスク跨ぎ循環は lint でありコンパイルエラーではない**：Rust のようにコンパイルエラーとはならず、デフォルトは warn、チームが deny に設定して初めて品質ゲートになる

## 代替案

| 代替案 | 選択しない理由 |
|------|--------------|
| GC | ランタイムオーバーヘッドがあり、停止を予測できない |
| Rust 借用検査器 | ライフタイム `'a` が必要で、学習曲線が急峻 |
| 純粋な Move | 並行共有を処理できない |
| 生ポインタなし | システムレベルプログラミングができない |
| Rc/Arc をユーザーに公開 | 実装詳細をユーザーに押し付け、認知負担が増大 |
| 簡素化借用（v8） | エスケープ禁止の戦略がクロージャキャプチャ、参照返しなどの重要な表現能力を犠牲にする |

### トークン衝突検出：ホーア論理命題

トークン衝突検出の完全な方案は [RFC-009a: トークンライフタイム分析——ホーア証明パイプラインに基づく](../accepted/009a-borrow-proof-pipeline.md) を参照。核心要点：

**トークン活性はホーア論理命題である。** `{衝突する ReadToken がすべて死亡} write(data) {WriteToken は安全に取得可能}`——型検査やユーザー述語検証と RFC-027 の証明パイプラインを共有する。コンパイラが借用命題（`borrow_conflict`、`use_after_move`、`use_after_drop`、`mut_violation`）を自動生成し、パイプラインが Proved / Disproved / Unproven を返す。

**借用検査は消えていない——次元が削減されたのである。** `BorrowChecker` は `BorrowPredicateEmitter` となり、検査を実行するのではなく命題を生成する。これは「型検査器」の概念と完全に並行する：型検査器は型等価命題を生成し、借用命題ジェネレータは借用命題を生成し、同一のパイプラインが検証する。

**ブランド ID（`#42`）は `'a` そのものである。** 情報は完全に同じで、エンコーディングが異なる。`'a` は型シグネチャに可視、`#42` はコンパイラ内部。新規解析を発明したのではなく、ライフタイムを型層から証明層に次元削減したのである。

**アルゴリズム概要**（詳細は RFC-009a）：
- ブランドツリー前置一致 → 衝突トークンの決定（O(depth)、深さ ≤ 3）
- 逆方向 BFS → 消費者から出発、break で逆辺を切断、構造解析で 95%+ のシナリオをカバー（高速パス）
- SMT 論理切断 → while + パス条件時のみ呼び出し（低速パス、極めて稀）
