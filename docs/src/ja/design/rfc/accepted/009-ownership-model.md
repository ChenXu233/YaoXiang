# RFC-009: 所有権モデル設計

## 概要

本ドキュメントは YaoXiang プログラミング言語の**所有権モデル（Ownership Model）**を定義します。

**コア設計——5つの概念、1つのグラデーション**：

```
一目視る/その場で改変     拿走           共有持有         複製一份        システム級
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         ゼロコピー      コンパイラが自動選択    明示的深コピー     *T
  ゼロサイズトークン       デフォルト    Rc/Arcを選択        独立コピー      ユーザ責任
  タイプ属性で自然に                            
  権限を推測                              
```

- **Move（デフォルト）**：代入/引数渡しまたは戻り値 = 所有権移転、ゼロコピー、RAII 自動解放
- **`&T` / `&mut T`（借用トークン）**：ゼロサイズのコンパイル時トークン型。`&T` は複製可能（共有読み取り専用）、`&mut T` は線形（独占的可変）。権限は型属性から自然に推測され、特殊ルールは不要。戻り値可能、構造体に格納可能、クロージャでキャプチャ可能。
- **`ref` キーワード**：スコープをまたぐ共有。コンパイラが Rc（タスクをまたがない）か Arc（タスクをまたぐ）を自動選択
- **`clone()`**：明示的深コピー
- **`unsafe` + `*T`**：生ポインタ、システム級エスケープハッチ

**解消した複雑性**：
- ❌ ライフタイム `'a` なし
- ❌ 借用チェッカーなし（トークンの型属性——Dup/Linear——が専用 borrow checker を代替）
- ❌ GC なし
- ❌ 「エスケープ禁止」などの特殊ルールなし（トークンは通常型で、スコープは型システムで統一処理）
- ❌ ユーザーが Rc/Arc の違いを知る必要なし（コンパイラが自動選択）

> **プログラミング負担**：`&T` は複製可能、`&mut T` は複製不可——2つの型属性、特殊ルールゼロ、コンパイラ完全自動化。
> **パフォーマンス保証**：Move ゼロオーバーヘッド、トークンゼロオーバーヘッド（ゼロサイズ型、コンパイル後に消滅）、ref が必要に応じて支払い、GC ポーズなし。

## 動機

### なぜ所有権モデルが必要なのか？

| 言語 | メモリ管理 | 問題点 |
|------|----------|------|
| C/C++ | 手動管理 | メモリリーク、野ポインタ、二重解放 |
| Java/Python | GC | 遅延変動、メモリオーバーヘッド、予測不能なポーズ |
| Rust | 所有権 + 借用チェッカー | ライフタイム `'a` 学習曲線が険しい |
| **YaoXiang** | **Move + Token + ref** | **シンプル、確定的、GCなし** |

### 設計目標

```yaoxiang
# 1. デフォルト Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p                         # Move、p はこれ以上読めない

# 2. &T / &mut T 借用トークン（ゼロオーバーヘッド、型属性で自然に権限を推測）
print_info(p2)                 # コンパイラが自動的に &Point トークンを作成し、 사용 後即座に解放
shift(p2, 1.0, 1.0)           # コンパイラが自動的に &mut Point トークンを作成

# 3. ref = 共有（コンパイラが自動的に Rc/Arc を選択）
shared = ref p2                # スコープをまたいで保持
spawn { use(shared) }          # コンパイラ：タスクをまたぐ → Arc

# 4. clone() = 明示的複製
backup = p2.clone()            # 深コピー、独立

# 5. unsafe + *T = システム級
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rust とのコアな違い

| 機能 | Rust | YaoXiang |
|------|------|----------|
| デフォルトセマンティクス | 借用 `&T`（明示的 `.clone()` が必要） | **Move（値渡し、ゼロコピー）** |
| 借用 | `&T`/`&mut T`、戻り値可能、ライフタイム必要 | **`&T`/`&mut T` ゼロサイズトークン、Dup/Linear 型属性で自然に推測** |
| 共有機構 | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが Rc/Arc を自動選択）** |
| 複製 | `clone()` | `clone()` |
| 生ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用チェック | 大域的推測 | **型チェッカーのフロー感受性活性分析（トークン状態追跡）** |
| 循環参照 | 手動 Weak | **タスク終了時に統一解放 / タスク間 lint / 標準庫 Weak** |

## 提案

### 1. Move（デフォルト所有権移転）

```yaoxiang
# ルール：代入 / 引数渡し / 戻り値 = Move、ゼロコピー

p: Point = Point(1.0, 2.0)
p2 = p                           # Move、p はこれ以上読めない

# 変数は再代入可能（Python スタイル、シャドウイングなし）
p = Point(3.0, 4.0)              # p を再バインド、型は一致する必要がある

# 関数引数：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move 戻り値
}

# 関数戻り値：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move 戻り値、ゼロコピー
}
```

**特徴**：
- ゼロコピー（コンパイラがポインタを移動）
- 移動後、元バインディングは読めない（コンパイルエラー）
- RAII：スコープ終了時に自動解放
- 関数シグネチャ `(T) -> T` 自体がドキュメント——T を消費し、T を返す

### 2. &T / `&mut T`（借用トークン）

**コア原則：`&T` と `&mut T` はゼロサイズのコンパイル時トークン型です。これらは「参照」ではなく「アクセス権限の型レベル証明」です。**

#### 2.1 2つの型属性

```
&T      →  ゼロサイズ、複製可能（Dup）、読み取り専用権限を付与
&mut T  →  ゼロサイズ、線形（非Dup）、独占的読み書き権限を付与
```

**これは記憶すべき「ルール」ではなく、型システムの基本属性です。** Dup 型は自由に複製可能（複数の `&T` が共存可能）、線形型は複製不可能（`&mut T` は本質的に一意）。「借用チェッカー」はありません——型チェッカーが普段通りに行うことだけがそこにあります。

#### 2.2 基本使用

```yaoxiang
# メソッド側：パラメータ型を宣言し、必要な権限を決定
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # &Point トークンが読み取り権限を付与
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # &mut Point トークンが書き込み権限を付与
    self.y = self.y + dy
}

# 呼び出し側：コンパイラが自動的に借用または Move を選択
p = Point(1.0, 2.0)
p.print()                          # コンパイラが自動的に &Point トークンを作成
p.shift(1.0, 1.0)                  # コンパイラが自動的に &mut Point トークンを作成
p.print()                          # OK、前のトークンは shift 呼び出し終了時に解放済み

# 自由関数も同じ
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 2つの &Point トークンが共存——Dup 型
}
d = distance(p, p2)
```

#### 2.3 なぜ「禁止エスケープ」が不要なのか

RFC-009 v8 では `&T`/`&mut T` に3つの特殊ルール——パラメータにしか使えない、戻り値不可、構造体に格納不可——を施していました。これは「借用」概念へのパッチ当てでした。

トークンシステムではこれらのルールは不要です。トークンは**通常型**であり、他の型と同じスコープルールに従います。

**参照を返す——自然にサポート**：

```yaoxiang
# ✅ トークンは戻り値と共に伝播
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # 子トークンと親トークンが一緒に返る
}

# 使用例
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # トークンが呼び出し元に返る
print(px_ref)               # OK、トークンはまだスコープ内
```

**構造体に格納——自然にサポート**：

```yaoxiang
# ✅ 構造体がトークンをフィールドとして携带
Window: Type = {
    target: Point,
    view: &Point,      # トークンフィールド——target への読み取り専用ビューを保持
}

# view のトークンは target から派生、Window は両方の所有権を保持
# Window が存在する限り、view トークンは有効
```

**クロージャキャプチャ——自然にサポート**：

```yaoxiang
# ✅ クロージャがトークンをキャプチャ、まるで任意の値をキャプチャするように
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    # クロージャが threshold の &Float トークンをキャプチャ（Dup 型、自由롭게クロージャに複製可能）
    items.filter(|p| p.x > threshold)
}

# これは RFC-009 v8 では做不到——v8 はクロージャによる借用キャプチャを禁止
```

**タスクをまたぐ——トークンはスレッドをまたげない**：

```yaoxiang
# ❌ トークンはタスク境界をまたげない
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ コンパイルエラー：トークンはタスクをまたげない
}

# これは特殊ルールではない——トークンはコンパイル時権限証明、タスク間共有には ref を使用
# タスクをまたぐ共有が必要な場合は ref を使用
```

**トークンは ref できない**：

```yaoxiang
# ❌ トークンは権限証明であり、所有権ではない
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ コンパイルエラー：&T は所有可能的型ではない
}
```

#### 2.4 トークンのライフサイクル

トークンのライフサイクルは**通常のスコープルール**で決定され、ライフタイムパラメータは不要です：

- 関数パラメータ内のトークン：呼び出し期間中生착、呼び出し終了時に解放
- 返されるトークン：所有権が呼び出し元に移転
- 構造体に格納されたトークン：構造体と共に生착
- クロージャにキャプチャされたトークン：クロージャと共に生착

コンパイラは `'a` 注釈を必要としません。なぜならトークンは**値**であり、値のライフサイクルは所有権システム（Move/RAII）で統一管理されるからです。**借用問題を所有権問題に引き下げます。**

#### 2.5 トークン競合検出

RFC-009 v8 の「交差借用チェック」を代替します。原理はよりシンプル——トークン値の**フロー感受性活性分析**：

```yaoxiang
# ❌ &mut トークンは線形、複製不可能
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move、p はこれ以上読めない
    p.x = 10.0                      # ❌ コンパイルエラー：WriteToken はすでに移動済み
}

# ✅ &T トークンは Dup 型、自由に複製可能
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK、&T は Dup 型
    print(p.x)                      # OK
    print(p2.x)                     # OK、2つの読み取り専用トークンが共存
}
```

**検出方式**：これは専用「借用チェッカー」ではなく、トークン値の**フロー感受性活性分析**です。コンパイラは関数本体内で各トークンの状態（活性/移動済み）を追跡し、任意の線形型値の追跡方法と同じです。

#### 2.7 コンパイラ内部：ブランド機構

ユーザーはブランドに一切触れません。コンパイラは内部で各トークンにコンパイル時一意の識別子を割り当てます：

```
ユーザーが見るもの       コンパイラ内部表現
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N はコンパイル時一意の整数
&mut Point     →  WriteToken(Point, #M)   // #M はコンパイル時一意の整数
```

ブランドの用途：
- **偽造防止**：トークンは所有者のカプセルからのみ取得でき凭空構築不可
- **関連追跡**：`&Point` から `&Float`（フィールドアクセス）を派生時、`&Float` は派生ブランド（`#N.field_x`）を携带し、コンパイラは親トークンへ追跡可能
- **競合検出**：同源 `WriteToken` と派生の `ReadToken` は同時に活性不可

ブランドは単態化とインライン展開後に完全に消滅し 生成された機械語には存在しません。**ゼロランタイムオーバーヘッド。**

#### 2.8 自動借用選択ルール

呼び出し側コンパイラは以下の優先順位で自動選択：

```
1. 実引数が後続も使用する場合 → トークン作成を優先（&T または &mut T、メソッドシグネチャに応じる）
2. 実引数が後続で使用しない場合 → Move
3. 優先マッチ順序：&T < &mut T < Move
```

```yaoxiang
# 例：自動選択
p = Point(1.0, 2.0)
p.print()        # print が &self を宣言 → コンパイラが &Point トークンを作成
p.shift(1.0, 1.0) # shift が &mut self を宣言 → コンパイラが &mut Point トークンを作成
p2 = p           # Move、p はもう使用しない
```

#### 2.9 RFC-009 v8 簡易借用との比較

| 機能 | 簡易借用 (v8) | 借用トークン (v9) |
|------|--------------|--------------|
| 参照を返す | ❌ ハードコードで禁止 | ✅ トークンが戻り値と共に伝播 |
| 構造体に格納 | ❌ ハードコードで禁止 | ✅ トークンが構造体フィールドに |
| クロージャキャプチャ | ❌ ハードコードで禁止 | ✅ クロージャがトークン値をキャプチャ |
| 特殊ルール | 3つ（パラメータにしか使えない/返せない/格納不可） | 0——型属性で自然に推測 |
| 借用チェック | 専用交差借用チェック | 型チェッカーのフロー感受性活性分析 |
| ライフタイム注釈 | 不要 | 不要 |
| ランタイムオーバーヘッド | ゼロ | ゼロ（ゼロサイズ型、コンパイル後に消滅） |
| エラーメッセージ | 「借用はエスケープできない」 | 「WriteToken(#3) はすでに移動済み」（通常の型エラー） |
| ユーザー mentam model | 「借用」の特殊地位を理解する必要がある | `&T` は複製可能、`&mut T` は複製不可 |

### 3. ref キーワード（コンパイラ自動最適化）

`ref` はスコープをまたぐ共有の唯一の手段です。基盤は Rc か Arc かは、ユーザーが気にする必要はありません。

#### 3.1 基本使用

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共有、コンパイラが自動的に実装を選択

# タスク間をまたぐ共有
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # コンパイラ：タスクをまたぐ → Arc
    spawn { use(data) }           # コンパイラ：タスクをまたぐ → Arc
}

# 単一タスク内の共有
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # コンパイラ：タスクをまたがない → Rc
}
```

**ユーザー mentam model**：`ref` = 共有保持。それだけです。

#### 3.2 コンパイラエスケープ分析：Rc vs Arc

```
ref のデータフロー分析：

他のタスクにエスケープしない → Rc（非原子参照カウント、オーバーヘッド低）
他のタスクにエスケープする   → Arc（原子参照カウント、スレッドセーフ）
```

#### 3.3 環検出策略

```
タスク内環 → サイレント許可。
  ├── 構造化並行性がタスク終了時に全リソースを統一解放することを保証。
  ├── ref は常に安全に保持、意味論は薄められない。
  └── ユーザーがタスク内で双方向の強参照を構築する権利あり（例如グラフ計算の中間状態）。

タスク間環 → lint（デフォルト warn、構成可能）。
  ├── プログラム動作は正確、真のリークは発生しない（親タスク終了時に子タスクリソースは全て解放）。
  ├── しかしタスク間をまたぐ強参照は所有権境界が曖昧であることを意味，重新思考する価値あり。
  ├── デフォルト warn レベル、コンパイル成功하지만 注意あり。
  └── チームはプロジェクト構成で deny に設定でき、CI 品質ゲートに組み込み可能。
```

**Lint レベル**（Rust clippy に類似）：

| レベル | 動作 | シナリオ |
|------|------|------|
| `allow` | チェックなし | 個人プロジェクト |
| `warn`（デフォルト） | コンパイル成功、注意あり | 開発段階 |
| `deny` | コンパイル失敗 | チーム CI 品質ゲート |
| `forbid` | コンパイル失敗、上書き不可 | 組織レベル強制ルール |

```yaoxiang
# タスク内環：サイレント許可、双方向強参照
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # 環。タスク終了時に統一解放。
}

# タスク間環：lint（デフォルト warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: タスク間循環参照
    }
}
```

**プロジェクト構成例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # タスク間環は CI で直接拒否
```

| 環タイプ | 動作 | 理由 |
|--------|------|------|
| タスク内 ref 環 | チェックなし | ユーザーの権利、タスク終了時に統一解放 |
| タスク間 ref 環 | lint（デフォルト warn） | 再考の促し、deny に構成可能 |

#### 3.4 Weak：標準庫提供

```yaoxiang
use std.rc.Weak

# 上級ユーザーが明示的に選択
a.next = ref b
b.prev = Weak.new(a.next)        # ユーザーが明示的にどちらの方向を弱くするか制御
```

**`Weak` は言語組み込みではなく、標準庫の型です。** 日々は `ref` だけで十分。メモリを精密に制御する必要がある上級ユーザーが手動で `Weak` を導入します。

#### 3.5 借用トークン vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 何をするか | 一目視る/その場で改変 | 共有保持 |
| 範囲 | トークン値のスコープに従う | スコープをまたぐ |
| コスト | ゼロオーバーヘッド（ゼロサイズ型） | Rc または Arc（コンパイラ選択） |
| エスケープ | 可能（トークンは戻り値/構造体/クロージャで伝播可能） | そもそもエスケープ用 |
| タスクをまたぐ | 不可（トークンはコンパイル時権限証明、タスクをまたげない） | 可能（コンパイラが自動的に Arc を選択） |
| 環になるか | 関係なし | タスク内ならサイレント許可、タスク間なら lint |

---

### 4. clone() —— 明示的複製

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # 深コピー
# p と p2 は独立、互いに影響しない
```

**使用タイミング**：元の値を保持する必要があり、Move にも共有にも適さない場合。

### 5. unsafe + 生ポインタ（システム級プログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 生ポインタ
    (*ptr).x = 0.0                # 解参照（ユーザーが安全を保証）
    ptr2 = ptr + 1                # ポインタ演算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ユーザーがダングリング/解放後使用しないことを保証
- FFI、メモリ操作などシステム級プログラミングに使用

---

### 6. 所有権グラデーション総覧

```
  借用トークン（ゼロオーバーヘッド）     Move（ゼロオーバーヘッド）      共有（必要に応じて支払い）    複製
   │                      │                  │                │
  &T 複製可能トークン        デフォルト所有権移転     ref Rc/Arc       clone()
  &mut T 線形トークン      チェーン消費回流       コンパイラ自動選択      明示的深コピー
   │                      │                  │                │
  トークン値スコープ           スコープ内           スコープをまたぐ         いつでも
  戻り値/構造体格納可能        T -> T 回流        ref タスク間 → Arc  独立コピー
  クロージャでキャプチャ可能     T -> Void 消費     ref 単一タスク → Rc
  ゼロサイズコンパイル後に消滅                        タスク内環はサイレント
                                                    タスク間環は lint
                                                    標準庫 Weak エスケープ
```

## 統合例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：読み取り専用トークン
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：可変トークン
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消費回流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 拿走，改，还给你
    }

    # 参照を返す：トークンが戻り値と共に伝播
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# クロージャがトークンをキャプチャ（v8 では做不到な能力）
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}

# 統合使用
p = Point(1.0, 2.0)
p.print()                           # &Point トークン
p.shift(1.0, 1.0)                   # &mut Point トークン
p = p.scale(2.0)                    # Move → 回流
shared = ref p                      # ref 共有
spawn { use(shared) }

# clone 独立コピー
backup = p.clone()

# タスク内環：サイレント許可
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # 環、タスク終了時に統一解放

# unsafe システム級
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Dup 型属性

`Dup`（Duplicable）はコンパイラが自動管理する型属性で、意味は**シャローコピー**：代入/引数渡し時に複製されるのはハンドラ/トークンで、基底データは共有されます。これは Move（所有権移転）や Clone（明示的深コピー、独立コピーを作成）とは異なる3段階グラデーションを形成します。

**Dup と Clone は直交する概念**——Dup はハンドラを複製してデータを共有し、Clone は独立コピーを作成します。型は Dup と Clone の両方をサポート，也可能只有其中一个。

| 型 | Dup | Clone | 説明 |
|------|-----|-------|------|
| `&T` | ✅（トークンを複製、複数のビューが同じデータを指す） | ✅ | 読み取り専用トークン |
| `ref T` | ✅（参照カウント+1、ヒープデータを共有） | ✅ | 共有保持（コンパイラ自動選択 Rc/Arc） |
| String, Bytes | ✅（内部参照カウント、ハンドラを複製して基底バッファを共有） | ✅ | 文字列/バイト |
| `&mut T` | ❌（線形、独占） | ❌ | 可変トークン |
| `*T` | ❌ | ❌ | 生ポインタ |
| struct | 派生（全フィールドが Dup の時に自動派生） | ✅ | 構造体 |

**原語値型**（Int, Float, Bool, Char）の代入動作はコンパイラの組み込み値コピー——2つの値は完全に独立、シャローコピーではありません。これらは Dup 型属性に属さず、コンパイラのネイティブ処理です。

## パフォーマンス分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | ゼロ | ポインタ移動 |
| `&T` / `&mut T` | ゼロ | ゼロサイズ型、コンパイル後に消滅、ランタイムオーバーヘッドゼロ |
| `ref`（タスクをまたがない）| 低 | Rc にコンパイル、非原子操作 |
| `ref`（タスクをまたぐ）| 中 | Arc にコンパイル、原子操作 |
| `clone()` | 型による | 小さいオブジェクトは高速、大きいオブジェクトは低速 |
| `unsafe + *T` | ゼロ | 直接メモリ操作 |

### 比較

| 言語 | 共有機構 | メモリ管理 | 循環処理 | 複雑度 |
|------|----------|----------|----------|--------|
| Rust | Arc / Mutex + 借用チェッカー | コンパイル時チェック | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 借用トークン** | **RAII** | **タスク境界解放 / タスク間 lint / 標準庫 Weak** | **低** |

## トレードオフ

### 利点

1. **統一**：`&T`/`&mut T` は通常型であり、特殊言語機能ではない。RFC-010 の `name: type = value` と完全一致
2. **シンプル**：ライフタイムなし、大域的借用チェッカーなし。`&T` は複製可能、`&mut T` は複製不可——2つの型属性
3. **强大**：参照を返せる、構造体に格納可能、クロージャでキャプチャ可能——Rust と同等の表現力
4. **コンパイラがスマート**：ref が Rc/Arc を自動選択、呼び出し側が自動的に借用を選択
5. **確定的**：ref は常に安全に保持、こっそり弱参照にならない
6. **高性能**：Move ゼロコピー、トークンゼロオーバーヘッド（ゼロサイズ型、コンパイル後に消滅）
7. **柔軟**：`unsafe + *T` がシステム級プログラミングをサポート

### 欠点

1. **泛型ブランドパラメータ感染**：トークンがブランド識別子を携带し、参照を返す関数のシグネチャに追加の泛型パラメータが反映される
2. **ref ランタイムオーバーヘッド**：原子操作にはコストがある（しかしこれは共有の必然的代償）
3. **unsafe リスク**：ユーザーが正確性を保証する必要がある
4. **タスク間環は lint でありコンパイルエラーではない**：Rust のようにコンパイルでエラーになるのではなく、デフォルトで warn、チームで deny を構成しないと品質ゲートにならない

---

## 代替案

| 案 | なぜ選ばないか |
|------|--------------|
| GC | ランタイムオーバーヘッドがあり、予測不能なポーズがある |
| Rust 借用チェッカー | ライフタイム `'a` が必要で、学習曲線が險しい |
| 純粋 Move | 並行共有を処理できない |
| 生ポインタなし | システム級プログラミングができない |
| Rc/Arc をユーザーに公開 | 実装詳細をユーザーに押し付け、認知負担が増加 |
| 簡易借用（v8） | 「エスケープ禁止」戦略によりクロージャキャプチャ、参照を返すなど重要な表現力が犠牲になる |

### トークン競合検出：フロー感受性活性分析

**分析範囲**：関数本体内に限る。トークン値に対してフロー感受性活性分析を行い、各トークンの状態（活性/移動済み）を追跡する。

**レベル 1：呼び出し点チェック**——同一の実引数に対して同時に `&mut` トークンと他のトークンを作成不可：

```yaoxiang
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)    # ❌ p が同時に &mut と & トークンを派生、コンパイラが拒否
```

**レベル 2：関数本体フロー感受性**——`&mut` トークンを呼び出しに渡した後、呼び出しが返ればトークンは解放され、その後再度トークンを作成可能：

```yaoxiang
process_twice: (p: &mut Point) -> Void = {
    shift(p, 1.0, 1.0)    # &mut トークンを shift に渡し、shift が返るとトークンは解放
    print_info(p)          # &Point トークンを再作成、競合なし
}
```

**不要なものは**：関数間ライフタイム追跡、大域的エイリアス分析、借用グラフ制約解決、NLL、`'a` 注釈。トークンは値であるため、値の活性分析は型チェッカーが統一して処理——任意の線形型値の追跡方法と同じです。
