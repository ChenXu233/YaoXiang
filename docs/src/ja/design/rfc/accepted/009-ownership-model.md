# RFC-009: 所有権モデル設計

> **ステータス**: 承認済み
> **著者**: 晨煦
> **作成日**: 2025-01-08
> **最終更新**: 2026-05-11（丐版借用の &T/&mut T を追加、所有権グラデーションを完善）

## 摘要

このドキュメントは YaoXiang プログラミング言語の**所有権モデル（Ownership Model）**を定義します。

**コア設計——5つの概念、1つのグラデーション**：

```
看一眼/原地改     拿走           共有持有         複写一份        システム級
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         零コピー       コンパイラ自動      明示的深コピー      *T
  関数パラメータ       デフォルト      選択Rc/Arc                   ユーザ責任
  禁止逃逸                     タスク内ループ黙認
                               タスク間ループ lint
                               標準ライブラリWeak
```

- **Move（デフォルト）**：代入/引数渡し/戻り値 = 所有権移転、ゼロコピー、RAII 自動解放
- **`&T` / `&mut T`（丐版借用）**：関数パラメータのみ、逃逸禁止。ゼロ注釈、ゼロライフタイム。コンパイラが呼び出し側で自動的に借用
- **`ref` キーワード**：スコープをまたぐ共有。コンパイラが Rc（タスク間なし）か Arc（タスク間あり）を自動選択
- **`clone()`**：明示的深コピー
- **`unsafe` + `*T`**：生ポインタ、システム級エスケープハッチ

**排除した複雑さ**：
- ❌ ライフタイム `'a` なし
- ❌ 借用チェッカなし（禁止而非标注、Rust 式のグローバルライフタイム導出が不要）
- ❌ GC なし
- ❌ 소비分析/所有权回流などの「ミニ借用チェッカ」なし
- ❌ ユーザが Rc/Arc の違いを知る必要なし（コンパイラが自動選択）

> **プログラミング負担**：`&T`/`&mut T` の3つのルール、`ref` の1つのキーワード、コンパイラが全自动。
> **パフォーマンス保証**：Move ゼロオーバーヘッド、借用 ゼロオーバーヘッド、ref 必要时才付费、GC ポーズなし。

## 動機

### なぜ所有権モデルが必要か？

| 言語 | メモリ管理 | 問題点 |
|------|----------|------|
| C/C++ | 手動管理 | メモリリーク、野ポインタ、二重解放 |
| Java/Python | GC | 遅延ジッター、メモリオーバーヘッド、予測不能なポーズ |
| Rust | 所有権 + 借用チェッカ | ライフタイム `'a` 学習曲線が急激 |
| **YaoXiang** | **Move + 借用 + ref** | **シンプル、確定的、GC なし** |

### 設計目標

```yaoxiang
# 1. デフォルト Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p                         # Move、p はこれ以降使用不可

# 2. &T / &mut T 借用（ゼロオーバーヘッド、読み取りのみ/インプレース変更、逃逸禁止）
print_info(p2)                 # コンパイラが自動借用 &p2、使用後即座に解放
shift(p2, 1.0, 1.0)           # コンパイラが自動借用 &mut p2

# 3. ref = 共有（コンパイラが自動選択 Rc/Arc）
shared = ref p2                # スコープをまたいで保持
spawn { use(shared) }          # コンパイラ：タスク間をまたぐ → Arc

# 4. clone() = 明示的複写
backup = p2.clone()            # 深コピー、排他的

# 5. unsafe + *T = システム級
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rust とのコアな違い

| 機能 | Rust | YaoXiang |
|------|------|----------|
| デフォルト意味論 | 借用 `&T`（`.clone()` が必要） | **Move（値渡し、ゼロコピー）** |
| 借用 | `&T`/`&mut T`、返り値可能、ライフタイム必要 | **`&T`/`&mut T` パラメータのみ、逃逸禁止** |
| 共有機構 | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが自動選択 Rc/Arc）** |
| 複写 | `clone()` | `clone()` |
| 生ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用チェック | グローバル導出 | **関数ボディ範囲内でのみチェック** |
| 循環参照 | 手動 Weak | **タスク終了時の統一解放 / タスク間 lint / 標準ライブラリ Weak** |

## 提案

### 1. Move（デフォルト所有権移転）

```yaoxiang
# ルール：代入 / 引数渡し / 戻り値 = Move、ゼロコピー

p: Point = Point(1.0, 2.0)
p2 = p                           # Move、p はこれ以降使用不可

# 変数は再代入可能（Python 風、シャドウイングなし）
p = Point(3.0, 4.0)              # p を再束縛、型は一致する必要がある

# 関数パラメータ：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move 戻り値
}

# 関数戻り値：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move 戻り値、ゼロコピー
}
```

**特徴**：
- ゼロコピー（コンパイラがポインタを移動）
- 移動後元の束縛は使用不可（コンパイルエラー）
- RAII：スコープ終了で自動解放
- 関数シグネチャ `(T) -> T` 自体がドキュメント——T を消費し、T を返す

### 2. &T / &mut T（丐版借用）

**コア原則：借用は「ちょっと見る」か「インプレースで変更する」だけで、所有は許さない。**

#### 2.1 3つのルール

```
1. &T / &mut T は関数パラメータとしてのみ出現可能
2. 返り値不可、構造体に格納不可、局所変数に代入不可、閉包に捕獲されて逃げられない
3. 呼び出し側で & を注釈する必要なし、コンパイラがメソッドシグネチャに基づいて借用または Move を自動選択
```

**ゼロ注釈。ゼロライフタイム。** コンパイラが做的就是一件事：借用が現在の関数を離れないことを保証。これは関数間分析不要——跨関数被"禁止"堵死了、導出没有必要。

#### 2.2 基本使用

```yaoxiang
# メソッド側：self 型の宣言で借用方式を決定
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # フィールド読み取り
    print(self.y)
    # 関数終了、借用終了
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # インプレース変更
    self.y = self.y + dy
}

# 呼び出し側：コンパイラが自動選択 借用または Move
p = Point(1.0, 2.0)
p.print()                          # コンパイラ：&p、print 終了後借用自動解放
p.shift(1.0, 1.0)                  # コンパイラ：&mut p、shift 終了後借用自動解放
p.print()                          # OK、p はまだ有効

# 自由関数も同理
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 2つの引数を読み取り
}
d = distance(p, p2)                     # コンパイラ：&p, &p2
```

#### 2.3 禁止される行為

```yaoxiang
# ❌ 禁止：借用が関数を離れる
get_x_ref: (p: &Point) -> &Float = { p.x }      # 借用を返す → コンパイルエラー
store_ref: (p: &Point) -> Wrapper = {             # 構造体に格納 → コンパイルエラー
    Wrapper { ref: p }
}

# ❌ 禁止：借用を局所変数に代入
bad_bind: (p: &Point) -> Void = {
    q = p                         # &Point を局所変数に代入 → コンパイルエラー
}

# ❌ 禁止：タスク境界をまたぐ
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # 借用がタスクに逃げ出す → コンパイルエラー
}

# ❌ 禁止：借用したものは Move 不可
bad_move: (p: &Point) -> Void = {
    p2 = p                        # あなたのものじゃない、移动无权 → コンパイルエラー
}

# ❌ 禁止：借用したものは永続保持不可
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # 借用は所有権ではない、ref 不可 → コンパイルエラー
}
```

#### 2.4 &mut のエイリアス保護

```yaoxiang
# ✅ 許可：複数の &T が同時にアクティブ
read_both: (a: &Point, b: &Point) -> Float = { a.x + b.y }

# ✅ 許可：&mut 終了後 다시 借用可能
shift_and_read: (p: &mut Point) -> Void = {
    shift(p, 1.0, 1.0)           # &mut p の借用はこの呼び出しで終了
    print_info(p)                 # 前の借用終了済み、& を再借用可能
}

# ❌ 禁止：&mut と &T が同時にアクティブ
# コンパイラが関数ボディ範囲でフロー敏感分析し、同時に1つの &mut のみが存在することを保証
```

#### 2.5 自動借用選択ルール

呼び出し側コンパイラは以下の優先順位で自動選択：

```
1. 実引数が以降も使用される場合 → 借用を優先（&T または &mut T、メソッドシグネチャによる）
2. 実引数が以降使用されない場合 → Move
3. 優先匹配順序：&T < &mut T < Move
```

```yaoxiang
# 例：自動選択
p = Point(1.0, 2.0)
p.print()        # print は &self を宣言 → &p、借用終了後 p はまだ使用可能
p.shift(1.0, 1.0) # shift は &mut self を宣言 → &mut p
p2 = p           # Move、p はこれ以降使用不可
```

### 3. ref キーワード（コンパイラ自動最適化）

`ref` はスコープをまたぐ共有の唯一の方法。基盤が Rc か Arc か、ユーザは気にする必要はない。

#### 3.1 基本使用

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共有、コンパイラが自動選択

# タスク間をまたぐ共有
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # コンパイラ：タスク間をまたぐ → Arc
    spawn { use(data) }           # コンパイラ：タスク間をまたぐ → Arc
}

# 単一タスク内の共有
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # コンパイラ：タスク間をまたがない → Rc
}
```

**ユーザ心智モデル**：`ref` = 共有保持。それだけで十分。

#### 3.2 コンパイラの逃逸分析：Rc vs Arc

```
ref のデータフロー分析：

他のタスクに逃げない → Rc（非原子参照カウント、オーバーヘッド小）
他のタスクに逃げる   → Arc（原子参照カウント、スレッドセーフ）
```

#### 3.3 ループ検出戦略

```
タスク内ループ → 黙認許可。
  ├── 構造化並行によりタスク終了時に全リソースが統一解放されることを保証。
  ├── ref は常に生きたまま、意味論に偽りなし。
  └── ユーザには双方向強参照を構築する権利あり（例如グラフ計算の中間状態）。

タスク間ループ → lint（デフォルト warn、構成可能）。
  ├── プログラム動作は正しい、本当にリークすることはない（親タスク終了時に子タスクのリソースは全て解放）。
  ├── しかしタスク間をまたぐ強参照は所有権境界が曖昧であることを意味に立ち止まって再考する価値あり。
  ├── デフォルト warn レベル、コンパイル通過だがヒントあり。
  └── チームはプロジェクト構成で deny に設定でき、CI 品質ゲートに組み込み可能。
```

**Lint レベル**（Rust clippy 类似）：

| レベル | 動作 | シナリオ |
|------|------|------|
| `allow` | チェックなし | 個人プロジェクト |
| `warn`（デフォルト） | コンパイル通過、ヒントあり | 開発フェーズ |
| `deny` | コンパイル失敗 | チーム CI 品質ゲート |
| `forbid` | コンパイル失敗、上書き不可 | 組織レベル強制ルール |

```yaoxiang
# タスク内ループ：黙認許可、双方向強参照
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # ループ。タスク終了時に統一解放。
}

# タスク間ループ：lint（デフォルト warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: タスク間をまたぐ循環参照
    }
}
```

**プロジェクト構成例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # タスク間ループは CI で直接拒否
```

| ループタイプ | 動作 | 理由 |
|--------|------|------|
| タスク内 ref ループ | チェックなし | ユーザの権利、タスク終了時に統一解放 |
| タスク間 ref ループ | lint（デフォルト warn） | 再考を促す通知、deny に構成可能 |

#### 3.4 Weak：標準ライブラリ提供

```yaoxiang
use std.rc.Weak

# 上級ユーザが明示的に選択
a.next = ref b
b.prev = Weak.new(a.next)        # ユーザが明示的にどの方向を弱にするかを制御
```

**`Weak` は言語組み込みではなく、標準ライブラリの型です。** 日常使用は `ref` で十分。上級ユーザがメモリを詳細に制御する必要がある場合に手動で `Weak` を導入。

#### 3.5 借用 vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 何をするか | ちょっと見る/インプレース変更 | 共有保持 |
| 範囲 | 関数パラメータ、使用後即座に解放 | スコープをまたぐ |
| コスト | ゼロオーバーヘッド | Rc または Arc（コンパイラ選択） |
| 逃逸 | 禁止 | そもそも逃逸するためのもの |
| ループ | 関係なし | 黙認許可、タスク間をまたぐ場合は lint |

### 4. clone() —— 明示的複写

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # 深コピー
# p と p2 は独立、相互に影響なし
```

**使用タイミング**：元の値を保持する必要があり、Move にも共有にも適さないシナリオ。

### 5. unsafe + 生ポインタ（システム級プログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 生ポインタ
    (*ptr).x = 0.0                # 逆参照（ユーザが安全を保証）
    ptr2 = ptr + 1                # ポインタ演算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ユーザはダングリングなし、解放後使用なしを保証
- FFI、メモリ操作などのシステム級プログラミングに使用

### 6. 所有権グラデーション総覧

```
  借用（ゼロオーバーヘッド）         Move（ゼロオーバーヘッド）      共有（必要时才付费）    複写
   │                      │                  │                │
  &T ちょっと見る            デフォルト所有権移転     ref Rc/Arc       clone()
  &mut T インプレース変更      チェーン消費回流       コンパイラ自動選択      明示的深コピー
   │                      │                  │                │
  関数パラメータ               スコープ内           スコープをまたぐ         いつでも
  禁止逃逸               T -> T 回流        ref タスク間 → Arc  独立コピー
  自動選択               T -> Void 消費     ref タスク内 → Rc
                                            タスク内ループ黙認
                                            タスク間ループ lint
                                            標準ライブラリWeak エスケープ
```

## 総合例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：読み取りのみ
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：インプレース変更
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消費回流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 拿走，改，还给你
    }
}

# 総合使用
p = Point(1.0, 2.0)
p.print()                           # &p、借用
p.shift(1.0, 1.0)                   # &mut p、借用
p = p.scale(2.0)                    # Move → 回流
shared = ref p                      # ref 共有
spawn { use(shared) }

# clone 独立コピー
backup = p.clone()

# タスク内ループ：黙認許可
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # ループ、タスク終了時に統一解放

# unsafe システム級
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Send / Sync

| 型 | Send | Sync | 説明 |
|------|------|------|------|
| 値型 | ✅ | ✅ | Int、Float、Point... |
| `ref T` | ✅ | ✅ | コンパイラが自動選択 Rc/Arc |
| `&T` / `&mut T` | ❌ | ❌ | 借用はスレッドをまたげない（逃逸禁止） |
| `*T` | ❌ | ❌ | 生ポインタ、単一スレッド |

---

## パフォーマンス分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | ゼロ | ポインタ移動 |
| `&T` / `&mut T` | ゼロ | コンパイル時チェック、ランタイムオーバーヘッドゼロ |
| `ref`（タスク内）| 低 | Rc にコンパイル、非原子操作 |
| `ref`（タスク間）| 中 | Arc にコンパイル、原子操作 |
| `clone()` | 型による | 小オブジェクトは高速、大オブジェクトは低速 |
| `unsafe + *T` | ゼロ | 直接メモリ操作 |

### 比較

| 言語 | 共有機構 | メモリ管理 | 循環処理 | 複雑さ |
|------|----------|----------|----------|--------|
| Rust | Arc / Mutex + 借用チェッカ | コンパイル時チェック | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 丐版借用** | **RAII** | **タスク境界解放 / タスク間 lint / 標準ライブラリ Weak** | **低** |

## トレードオフ

### 优点

1. **シンプル**：ライフタイムなし、グローバル借用チェッカなし。`&T`/`&mut T` の3つのルール
2. **コンパイラがインテリジェント**：ref が Rc/Arc を自動選択、呼び出し側が自動選択借用
3. **確定的**：ref は生きたまま保ち、そっと弱参照にならない
4. **高性能**：Move ゼロコピー、借用 ゼロオーバーヘッド
5. **柔軟**：`unsafe + *T` がシステム級プログラミングをサポート

### 缺点

1. **ref ランタイムオーバーヘッド**：原子操作にはコストがある（しかしこれは共有の必然的な代償）
2. **unsafe リスク**：ユーザが正確性を保証する必要がある
3. **タスク間をまたぐループは lint 而不是コンパイルエラー**：Rust のようにコンパイルエラーにならず、デフォルト warn、deny を設定しないと品質ゲートにならない

## 代替案

| 案 | 为什么不選択 | 理由 |
|------|--------------|------|
| GC | ランタイムオーバーヘッドあり、予測不能なポーズあり | - |
| Rust 借用チェッカ | ライフタイム `'a` が必要、学習曲線が急激 | - |
| 純粋 Move | 並行共有を処理できない | - |
| 生ポインタなし | システム級プログラミングできない | - |
| ユーザに Rc/Arc を公開 | 実装詳細をユーザに押し付け、認知負荷を増加 | - |

### 交差借用チェック：呼び出し点で就地 + フロー敏感

**分析範囲**：関数ボディ内でのみ。`&T`/`&mut T` はパラメータとしてのみ使用可能なので、各呼び出しチェーンが次の関数のレイヤーに進入する時、上一層関数の借用は呼び出し終了時に自動的に解放される。関数間分析不要。

**レベル1：呼び出し点チェック**——各実引数は同時に `&mut` 位置と他の借用位置に出現不可：

```yaoxiang
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)    # ❌ p は同時に &mut と & として使用、コンパイラが拒否
```

**レベル2：関数ボディフロー敏感**——`&mut` を呼び出しに渡した後、呼び出しが返れば借用終了、以後再借用可能：

```yaoxiang
process_twice: (p: &mut Point) -> Void = {
    shift(p, 1.0, 1.0)    # &mut を shift に渡し、shift が返れば借用終了
    print_info(p)          # 再び &p を借用、競合なし
}
```

**不要なもの**：関数間ライフタイム追跡、グローバルエイリアス分析、借用グラフ制約解決、NLL。借用が逃げない、変数に格納されない、返らない——呼び出し点を離れると同時に無効。
