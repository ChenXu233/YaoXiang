# RFC-009: 所有権モデル設計

## 摘要

本文書は YaoXiang プログラミング言語の**所有権モデル（Ownership Model）**を定義する。

**コア設計——5つの概念、1つのグラデーション**：

```
一瞬見る/その場で変更   持ち去る           共有保持           コピー作成          システムレベル
    │                  │                 │                 │                  │
   &T                Move             ref             clone()             unsafe
  &mut T             ゼロコピー       コンパイラ自動     明示的ディープコピー   *T
  ゼロサイズトークン    デフォルト        Rc/Arc選択                        ユーザー責任
  型属性で自然に                                                  
  権限を導出                                                    
```

- **Move（デフォルト）**：代入/引数 전달/戻り値 = 所有権移動、ゼロコピー、RAII 自動解放
- **`&T` / `&mut T`（借用トークン）**：ゼロサイズのコンパイル時トークン型。`&T` は複製可能（共有読み取り専用）、`&mut T` は線形（排他的可変）。権限は型属性で自然に導出され、特殊ルールは不要。戻り値可能、構造体に格納可能、クロージャでキャプチャ可能。
- **`ref` キーワード**：スコープをまたぐ共有。コンパイラが Rc（タスクを越えない）か Arc（タスクを越える）を自動選択
- **`clone()`**：明示的ディープコピー
- **`unsafe` + `*T`**：生ポインタ、システムレベルの脱出ハッチ

**排除した複雑さ**：
- ❌ ライフタイム `'a` なし
- ❌ 借用チェッカーなし（トークンの型属性——Dup/Linear——が専用 borrow checker を代替）
- ❌ GC なし
- ❌ 「エスケープ禁止」などの特殊ルールなし（トークンは обычный 型で、スコープは型システムで統一処理）
- ❌ ユーザーは Rc/Arc の違いを知る必要はない（コンパイラが自動選択）

> **プログラミング負担**：`&T` は複製可能、`&mut T` は複製不可——2つの型属性、0の特殊ルール、完全自動化。
> **パフォーマンス保証**：Move ゼロオーバーヘッド、トークン ゼロオーバーヘッド（ゼロサイズ型、コンパイル後に消失）、ref 必要に応じた支払い、GC ポーズなし。

## 動機

### なぜ所有権モデルが必要か？

| 言語 | メモリ管理 | 問題点 |
|------|----------|----------|
| C/C++ | 手動管理 | メモリリーク、野ポインタ、二重解放 |
| Java/Python | GC | 遅延変動、メモリオーバーヘッド、予測不能なポーズ |
| Rust | 所有権 + 借用チェッカー | ライフタイム `'a` 学習曲線が急峻 |
| **YaoXiang** | **Move + Token + ref** | **シンプル、决定的、GCなし** |

### 設計目標

```yaoxiang
# 1. デフォルト Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p                         # Move、p はこれ以降読めない

# 2. &T / &mut T 借用トークン（ゼロオーバーヘッド、型属性で自然に権限を導出）
print_info(p2)                 # コンパイラが自動的に &Point トークンを作成、使用後に解放
shift(p2, 1.0, 1.0)           # コンパイラが自動的に &mut Point トークンを作成

# 3. ref = 共有（コンパイラが Rc/Arc を自動選択）
shared = ref p2                # スコープをまたいで保持
spawn { use(shared) }          # コンパイラ：タスクを越える → Arc

# 4. clone() = 明示的複製
backup = p2.clone()            # ディープコピー、独立

# 5. unsafe + *T = システムレベル
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rust とのコアな違い

| 機能 | Rust | YaoXiang |
|------|------|----------|
| デフォルトセマンティクス | 借用 `&T`（明示的 `.clone()` が必要） | **Move（値渡し、ゼロコピー）** |
| 借用 | `&T`/`&mut T`、戻り値可能、ライフタイム必要 | **`&T`/`&mut T` ゼロサイズトークン、Dup/Linear 型属性で自然に導出** |
| 共有機構 | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが Rc/Arc を自動選択）** |
| コピー | `clone()` | `clone()` |
| 生ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用チェッカー | 大域的導出 | **型チェッカーのフロースensitive 活性分析（トークン状態追跡）** |
| 循環参照 | 手動 Weak | **タスク終了時の統一解放 / タスク間 lint / 標準ライブラリ Weak** |

## 提案

### 1. Move（デフォルト所有権移動）

```yaoxiang
# ルール：代入 / 引数 / 戻り値 = Move、ゼロコピー

p: Point = Point(1.0, 2.0)
p2 = p                           # Move、p はこれ以降読めない

# 変数は再代入可能（Python スタイル、シャドウイングなし）
p = Point(3.0, 4.0)              # p は再束縛、型は同じでなければならない

# 関数引数：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move 戻り値
}

# 関数戻り値：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move 戻り値、ゼロコピー
}
```

**特徴**：
- ゼロコピー（コンパイラがポインタを移動）
- 移動後、元バインディングは読めない（コンパイルエラー）
- RAII：スコープ終了時に自動解放
- 関数シグネチャ `(T) -> T` 自体がドキュメント——T を消費し、T を返す

### 2. &T / &mut T（借用トークン）

**コア原則：`&T` と `&mut T` はゼロサイズのコンパイル時トークン型である。它们不是"参照"であり、"アクセス権限の型レベル証明"である。**

#### 2.1 2つの型属性

```
&T      →  ゼロサイズ、複製可能（Dup）、読み取り専用権限を付与
&mut T  →  ゼロサイズ、線形（非Dup）、排他的読み書き権限を付与
```

**これは記憶すべき"ルール"ではない——これは型システムの基本属性である。** Dup 型は自由に複製可能（複数の `&T` が共存可能）、線形型は複製不可（`&mut T` は本質的に一意）。"借用チェッカー"はない——型チェッカーが常時行っていることを続けているだけである。

#### 2.2 基本使用

```yaoxiang
# メソッド側：引数型を宣言し、必要な権限を決定
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # &Point トークンが読み取り権限を付与
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # &mut Point トークンが書き込み権限を付与
    self.y = self.y + dy
}

# 呼び出し側：コンパイラが自動的に借用か Move を選択
p = Point(1.0, 2.0)
p.print()                          # コンパイラが自動的に &Point トークンを作成
p.shift(1.0, 1.0)                  # コンパイラが自動的に &mut Point トークンを作成
p.print()                          # OK、前回のトークンは shift 呼び出し終了時に解放済み

# 自由関数も同理
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 2つの &Point トークンが共存——Dup 型
}
d = distance(p, p2)
```

#### 2.3 なぜ「禁止エスケープ」が不要か

RFC-009 v8 は `&T`/`&mut T` に3つの特殊ルールを課していた——引数にしか使えない、戻り値不可、構造体に格納不可。これは"借用"概念へのパッチ当てだった。

トークンシステムではこれらのルールは不要である。トークンは** обычный 型**であり、他の型と同じスコープルールに従う。

**参照を返す——自然にサポート**：

```yaoxiang
# ✅ トークンは戻り値と一緒に伝播
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # サブトークンと親トークンが一緒に戻る
}

# 使用例
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # トークンが呼び出し元に返される
print(px_ref)               # OK、トークンはまだスコープ内
```

**構造体に格納——自然にサポート**：

```yaoxiang
# ✅ 構造体がトークンをフィールドとして携带
Window: Type = {
    target: Point,
    view: &Point,      # トークンフィールド——target への読み取り専用ビューを保持
}

# view のトークンは target から派生、Window は両方の所有権を保持
# Window が存在する限り、view トークンは有効
```

**クロージャキャプチャ——自然にサポート**：

```yaoxiang
# ✅ クロージャがトークンをキャプチャ就跟キャプチャ любого 値一样
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    # クロージャが threshold の &Float トークンをキャプチャ（Dup 型、クロージャに自由に複製）
    items.filter(|p| p.x > threshold)
}

# これは RFC-009 v8 ではできなかった——v8 はクロージャによる借用キャプチャを禁止
```

**タスクを越える——トークンはスレッドをまたげない**：

```yaoxiang
# ❌ トークンはタスク境界を越えられない
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ コンパイルエラー：トークンはタスクをまたげない
}

# これは特殊ルールではない——トークンはコンパイル時の権限証明、タスク間共有には ref を使用
# タスク間共有が必要な場合は ref を使用
```

**トークンは ref できない**：

```yaoxiang
# ❌ トークンは権限証明であり、所有権ではない
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ コンパイルエラー：&T は所有可能型ではない
}
```

#### 2.4 トークンのライフサイクル

トークンのライフサイクルは** обычный スコープルール**で決定され、ライフタイムパラメータは不要：

- 関数引数内のトークン：呼び出し期間中生存在、呼び出し終了時に解放
- 返されるトークン：所有権が呼び出し元に移動
- 構造体に格納されたトークン：構造体と一緒に生存在
- クロージャにキャプチャされたトークン：クロージャと一緒に生存在

コンパイラは `'a` 注釈を必要としない、なぜならトークンは**値**であり、値のライフサイクルは所有権システム（Move/RAII）で統一管理される。**借用問題を所有権問題へ次元削減。**

#### 2.5 冻结メカニズム

`&mut T` トークンは一時的に"冻结"して `&T` トークンを生成できる：

```yaoxiang
modify_and_read: (p: &mut Point) -> Void = {
    p.x = 10.0                      # &mut Point を使用して変更
    
    # &mut を冻结して、読み取り専用ビューを取得
    view: &Point = freeze(p)         # p はここで冻结される
    print(view.x)                   # &Point を通じて読み取り
    print(view.y)
    # view がスコープを離れる、冻结解除
    
    p.y = 20.0                      # &mut Point が利用可能に
}
```

`freeze` のセマンティクス：
- `&mut T` を受け取り、`&T` を返す
- `&T` が生存している間、元 `&mut T` は利用不可
- `&T` がスコープを離れた後、`&mut T` が自動的に回復
- これは**フロースensitive 分析**——コンパイラが関数本体でトークン状態を追跡

#### 2.6 トークン競合検出

RFC-009 v8 の"交叉借用チェック"を代替。原理はよりシンプル——トークン値の**フロースensitive 活性分析**：

```yaoxiang
# ❌ &mut と派生 &T は同時に生存できない
bad_alias: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p が冻结される
    p.x = 10.0                      # ❌ コンパイルエラー：WriteToken が冻结状態にある
    print(view.x)                   
}

# ✅ 冻结解除後は &mut を継続使用可能
good_seq: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p が冻结される
    print(view.x)                   # &T を使用
    # view がスコープを離れる
    p.x = 10.0                      # ✅ WriteToken が回復済み
}
```

**検出方式**：これは専用"借用チェッカー"ではない——これはトークン値の**フロースensitive 活性分析**である。コンパイラが関数本体で各トークンの状態（生存中/冻结中/移動済み）を追跡、これは任何の 線形型 値の追跡方法と同じである。

#### 2.7 コンパイラ内部：ブランドメカニズム

ユーザーはブランドに触れない。コンパイラは内部で各トークンにコンパイル時一意識別子を割り当て：

```
ユーザーが見るもの       コンパイラの内部表現
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N はコンパイル時一意整数
&mut Point     →  WriteToken(Point, #M)   // #M はコンパイル時一意整数
```

ブランドの用途：
- **偽造防止**：トークンは所有者のカプセルまたは freeze 操作からのみ取得でき、空から構築できない
- **関連追跡**：`&Point` から `&Float`（フィールドアクセス）を派生する際、`&Float` は派生ブランド（`#N.field_x`）を携带し、コンパイラは親トークンへ追跡可能
- **競合検出**：同源 `WriteToken` と派生 `ReadToken` は同時に生存不可

ブランドは単態化とインライン展開後に完全に消失し、生成された機械語には存在しない。**ゼロ런タイムオーバーヘッド。**

#### 2.8 自動借用選択ルール

呼び出し側コンパイラは以下の優先順位で自動選択：

```
1. 実引数がその後も使用される場合 → トークン作成を優先（&T または &mut T、メソッドシグネチャによる）
2. 実引数がその後は使用されない場合 → Move
3. 優先一致順序：&T < &mut T < Move
```

```yaoxiang
# 例：自動選択
p = Point(1.0, 2.0)
p.print()        # print は &self を宣言 → コンパイラが &Point トークンを作成
p.shift(1.0, 1.0) # shift は &mut self を宣言 → コンパイラが &mut Point トークンを作成
p2 = p           # Move、p はこれ以降使用しない
```

#### 2.9 RFC-009 v8 乞食版借用との比較

| 機能 | 乞食版借用 (v8) | 借用トークン (v9) |
|------|--------------|--------------|
| 参照を返す | ❌ ハードコードで禁止 | ✅ トークンが戻り値と一緒に伝播 |
| 構造体に格納 | ❌ ハードコードで禁止 | ✅ トークンが構造体フィールドとして |
| クロージャキャプチャ | ❌ ハードコードで禁止 | ✅ クロージャがトークン値をキャプチャ |
| 特殊ルール | 3つ（引数にしか使えない/返せない/格納不可） | 0つ——型属性で自然に導出 |
| 借用チェック | 専用交叉借用チェック | 型チェッカーのフロースensitive 活性分析 |
| ライフタイム注釈 | 不要 | 不要 |
| ランタイムオーバーヘッド | ゼロ | ゼロ（ゼロサイズ型、コンパイル後に消失） |
| エラーメッセージ | "借用はエスケープできない" | "WriteToken(#3) は冻结されています"（通常型エラー） |
| ユーザー心中的モデル | "借用"の特殊地位を理解する | `&T` は複製可能、`&mut T` は複製不可 |

### 3. ref キーワード（コンパイラ自動最適化）

`ref` はスコープをまたぐ共有の唯一の方法である。基盤が Rc か Arc か、ユーザーは気にする必要はない。

#### 3.1 基本使用

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共有、コンパイラが自動選択

# タスク間共有
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # コンパイラ：タスクを越える → Arc
    spawn { use(data) }           # コンパイラ：タスクを越える → Arc
}

# 単一タスク内共有
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # コンパイラ：タスクを越えない → Rc
}
```

**ユーザー心中的モデル**：`ref` = 共有保持。それだけ。

#### 3.2 コンパイラエスケープ分析：Rc vs Arc

```
ref のデータフロー分析：

他のタスクにエスケープしない → Rc（非アトミック参照カウント、低コスト）
他のタスクにエスケープする   → Arc（アトミック参照カウント、スレッドセーフ）
```

#### 3.3 循環検出戦略

```
タスク内循環 → 黙認許可。
  ├── 構造化同時実行がタスク終了時にすべてのリソースを統一解放することを保証。
  ├── ref は常に生存し、セマンティクスは薄められない。
  └── ユーザーはタスク内で双方向の強参照を構築する権利を持つ（例：グラフ計算の中間状態）。

タスク間循環 → lint（デフォルト warn、構成可能）。
  ├── プログラム動作は正確、本当にリークすることはない（親タスク終了時に子タスクのリソースはすべて解放）。
  ├── しかしタスク間の強参照は所有権境界が曖昧であることを意味し、一旦立ち止まって再考する価値がある。
  ├── デフォルト warn レベル、ビルドは通るがヒントがある。
  └── チームはプロジェクト設定で deny に設定でき、CI 品質ゲートに組み込める。
```

**Lint レベル**（Rust clippy 类似）：

| レベル | 動作 | シナリオ |
|------|------|------|
| `allow` | チェックなし | 個人プロジェクト |
| `warn`（デフォルト） | ビルド通る、ヒントあり | 開発段階 |
| `deny` | ビルド失敗 | チーム CI 品質ゲート |
| `forbid` | ビルド失敗、上書き不可 | 組織レベル強制ルール |

```yaoxiang
# タスク内循環：黙認許可、双方向強参照
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # 循環。タスク終了時に統一解放。
}

# タスク間循環：lint（デフォルト warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: タスク間循環参照
    }
}
```

**プロジェクト設定例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # タスク間循環は CI で直接拒否
```

| 循環タイプ | 動作 | 理由 |
|--------|------|------|
| タスク内 ref 循環 | チェックなし | ユーザーの権利、タスク終了時に統一解放 |
| タスク間 ref 循環 | lint（デフォルト warn） | 再考のヒント、deny に構成可能 |

#### 3.4 Weak：標準ライブラリで提供

```yaoxiang
use std.rc.Weak

# 上級ユーザーが明示的に選択
a.next = ref b
b.prev = Weak.new(a.next)        # ユーザーが明示的に、どの方向を弱くするか制御
```

**`Weak` は言語組み込みではなく、標準ライブラリの型である。** 日常使用は `ref` で十分。上級ユーザーがメモリを精密制御する必要がある場合は手動で `Weak` を導入。

#### 3.5 借用トークン vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 何をするか | 一瞬見る/その場で変更 | 共有保持 |
| 範囲 | トークン値のスコープに準ずる | スコープをまたぐ |
| コスト | ゼロオーバーヘッド（ゼロサイズ型） | Rc または Arc（コンパイラ選択） |
| エスケープ | 可能（トークンは戻り値/構造体/クロージャと一緒に伝播） | 元々エスケープ用 |
| タスク間 | 不可（トークンはコンパイル時権限証明、タスクをまたげない） | 可能（コンパイラが Arc を自動選択） |
| 循環 | 関係ない | タスク内黙認許可、タスク間 lint |

---

### 4. clone() —— 明示的複製

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # ディープコピー
# p と p2 は独立、相互に影響しない
```

**使用时机**：元の値を保持する必要があり、Move に適さず、共有にも適さない場合。

### 5. unsafe + 生ポインタ（システムレベルプログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 生ポインタ
    (*ptr).x = 0.0                # 逆参照（ユーザーが安全を保証）
    ptr2 = ptr + 1                # ポインタ演算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ユーザーはダングリング、使用後解放がないことを保証
- FFI、メモリ操作などのシステムレベルプログラミングに使用

### 6. 所有権グラデーション総覧

```
  借用トークン（ゼロオーバーヘッド）   Move（ゼロオーバーヘッド）    共有（必要時払い）      複製
   │                         │                   │                │
  &T 複製可能トークン         デフォルト所有権移動    ref Rc/Arc        clone()
  &mut T 線形トークン         チェーン消費回流      コンパイラ自動選択    明示的ディープコピー
   │                         │                   │                │
  トークン値スコープ            スコープ内            スコープをまたぐ      常時
  戻り値/構造体格納可          T -> T 回流          ref タスク越え → Arc   独立コピー
  クロージャキャプチャ可        T -> Void 消費       ref タスク越えなし → Rc
  ゼロサイズコンパイル後消失                         タスク内循環黙認
                                                    タスク間循環 lint
                                                    標準ライブラリ Weak 脱出
```

## 統合例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：読み取り専用トークン
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：可变トークン
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消費回流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 拿走，改，还给你
    }

    # 参照を返す：トークンが戻り値と一緒に伝播
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# クロージャがトークンをキャプチャ（v8 ではできなかった能力）
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}

# 統合使用
p = Point(1.0, 2.0)
p.print()                           # &Point トークン
p.shift(1.0, 1.0)                   # &mut Point トークン
p = p.scale(2.0)                    # Move → 回流
shared = ref p                      # ref 共有
spawn { use(shared) }

# clone 独立コピー
backup = p.clone()

# タスク内循環：黙認許可
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # 循環、タスク終了時に統一解放

# unsafe システムレベル
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Dup 型属性

`Dup`（複製可能）はコンパイラが自動管理する型属性であり、意味は**シャローコピー**：代入/引数 전달時に複製されるのはハンドラ/トークンであり、基盤データは共有される。これは Move（所有権移動）と Clone（明示的ディープコピー независимых 副本 создание）と共に3段階グラデーションを形成する。

**Dup と Clone は直交する概念**——Dup はハンドルを複製してデータを共有し、Clone は独立した副本を作成する。型は Dup と Clone の両方をサポートすることも、いずれか一方のみをサポートすることもできる。

| 型 | Dup | Clone | 説明 |
|------|-----|-------|------|
| `&T` | ✅（トークンを複製、複数の视角が同じデータを指す） | ✅ | 読み取り専用トークン |
| `ref T` | ✅（参照カウント+1、ヒープデータを共有） | ✅ | 共有保持（コンパイラが Rc/Arc を自動選択） |
| String, Bytes | ✅（内部参照カウント、ハンドルを複製して基盤 buffer を共有） | ✅ | 文字列/バイト |
| `&mut T` | ❌（線形、排他的） | ❌ | 可変トークン |
| `*T` | ❌ | ❌ | 生ポインタ |
| struct | 派生（すべてのフィールドが Dup の場合自動派生） | ✅ | 構造体 |

**原語値型**（Int, Float, Bool, Char）の代入動作はコンパイラが組み込む値コピー——2つの値は完全に独立、シャローコピーではない。これらは Dup 型属性には属さず、コンパイラのネイティブ処理である。

## パフォーマンス分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | ゼロ | ポインタ移動 |
| `&T` / `&mut T` | ゼロ | ゼロサイズ型、コンパイル後に消失、ゼロランタイムオーバーヘッド |
| `ref`（タスク越えなし）| 低 | Rc にコンパイル、非アトミック操作 |
| `ref`（タスク越え）| 中 | Arc にコンパイル、アトミック操作 |
| `clone()` | 型による | 小オブジェクトは高速、大オブジェクトは低速 |
| `unsafe + *T` | ゼロ | 直接メモリ操作 |

### 比較

| 言語 | 共有機構 | メモリ管理 | 循環処理 | 複雑さ |
|------|----------|----------|----------|--------|
| Rust | Arc / Mutex + 借用チェッカー | コンパイル時チェック | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 借用トークン** | **RAII** | **タスク境界解放 / タスク間 lint / 標準ライブラリ Weak** | **低** |

## トレードオフ

### 利点

1. **統一**：`&T`/`&mut T` は обычный 型であり、特殊言語機能ではない。RFC-010 の `name: type = value` と完全に合致
2. **シンプル**：ライフタイムなし、全域借用チェッカーなし。`&T` は複製可能、`&mut T` は複製不可——2つの型属性
3. **強力**：参照を返す、構造体に格納、クロージャキャプチャ——Rust と同等の表現力
4. **コンパイラ知的**：ref が Rc/Arc を自動選択、呼び出し側が自動的に借用を選択
5. **决定的**：ref は常に生存、弱参照に悄悄変化しない
6. **高性能**：Move ゼロコピー、トークン ゼロオーバーヘッド（ゼロサイズ型、コンパイル後に消失）
7. **柔軟**：`unsafe + *T` がシステムレベルプログラミングをサポート

### 欠点

1. **ジェネティックブランドパラメータ伝染**：トークンはブランド識別子を携带し、参照を返す関数のシグネチャに追加のジェネティックパラメータが反映される
2. **ref ランタイムオーバーヘッド**：アトミック操作にはコストがある（しかしこれは共有の必然的代償）
3. **unsafe リスク**：ユーザーは正確性を保証しなければならない
4. **タスク間循環は lint 而非ビルドエラー**：Rust のようにビルドエラーにならず、デフォルトは warn、团队が deny を設定して品質ゲートにしないと

| 案 | なぜ選択しないか |
|------|--------------|
| GC | ランタイムオーバーヘッドがあり、予測不能なポーズがある |
| Rust 借用チェッカー | ライフタイム `'a` が必要で、学習曲線が急峻 |

### トークン競合検出：フロースensitive 活性分析

**分析範囲**：関数本体のみ。トークン値に対してフロースensitive 活性分析を行い、各トークンの状態（生存中/冻结中/移動済み）を追跡。

**レベル1：呼び出し点チェック**——同じ実引数から同時に `&mut` トークンと他のトークンを作成不可：

```yaoxiang
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)    # ❌ p から &mut と & トークンが同時に派生、コンパイラが拒否
```

**レベル2：関数本体フロースensitive**——`&mut` トークンが呼び出しに渡された後、呼び出しが戻ればトークンは解放され、その後またトークンを作成可能：

```yaoxiang
process_twice: (p: &mut Point) -> Void = {
    shift(p, 1.0, 1.0)    # &mut トークンが shift に渡される、shift が戻るとトークン解放
    print_info(p)          # &Point トークンを再作成、競合なし
}
```

**レベル3：冻结状態追跡**——`freeze` が生成した `&T` トークンが生存している間、元 `&mut` トークンは利用不可：

```yaoxiang
frozen: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)    # p が冻结状態に入る
    print(view.x)
    p.x = 10.0                  # ❌ コンパイルエラー：WriteToken が冻结状態にある
}
```

**不要なもの**：関数間ライフタイム追跡、大域的エイリアス分析、借用グラフ制約求解、NLL、`'a` 注釈。トークンは値であり、値の活性分析は型チェッカーが統一処理——任何の 線形型 値の追跡方法と同じ。
