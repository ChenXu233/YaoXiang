# RFC-009: 所有権モデル設計

## 概要

本ドキュメントは YaoXiang プログラミング言語の**所有権モデル（Ownership Model）**を定義する。

**中核設計——5つの概念、1つのグラデーション**：

```
ちら見/その場変更   奪う         共有保持         複製           システムレベル
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         ゼロコピー     コンパイラ自動   明示的ディープコピー  *T
  ゼロサイズトークン デフォルト    選択Rc/Arc                  ユーザ責任
  型属性自然
  権限導出
```

- **Move（デフォルト）**：代入/引数渡し/戻り値 = 所有権移転、ゼロコピー、RAII による自動解放
- **`&T` / `&mut T`（借用トークン）**：ゼロサイズのコンパイル時トークン型。`&T` は複製可能（共有読み取り専用）、`&mut T` は線形（排他的可変）。権限は型属性により自然に導出され、特別なルールは不要。返却可能、構造体への保存可能。
- **`ref` キーワード**：スコープをまたいだ共有。コンパイラが Rc（タスクをまたがない）か Arc（タスクをまたぐ）を自動選択
- **`clone()`**：明示的ディープコピー
- **`unsafe` + `*T`**：生ポインタ、システムレベルのエスケープハッチ

**排除された複雑性**：
- ❌ ライフタイム `'a` なし
- ❌ 独立した借用検査フレームワークなし（借用衝突はホーア命題に次元削減され、型検査と証明パイプラインを共有）
- ❌ GC なし
- ❌ 「エスケープ禁止」などの特殊ルールなし（トークンは通常の型であり、スコープは型システムで統一的に処理）
- ❌ ユーザは Rc/Arc の違いを知る必要がない（コンパイラが自動選択）

> **プログラミング負担**：`&T` は複製可能、`&mut T` は複製不可——2つの型属性、0の特殊ルール、コンパイラ完全自動。
> **性能保証**：Move ゼロコスト、トークンゼロコスト（ゼロサイズ型、コンパイル後に消滅）、ref は必要に応じて課金、GC 停止なし。

## 動機

### なぜ所有権モデルが必要か？

| 言語 | メモリ管理 | 問題 |
|------|----------|------|
| C/C++ | 手動管理 | メモリリーク、ダングリングポインタ、二重解放 |
| Java/Python | GC | レイテンシ変動、メモリオーバーヘッド、予測不能な停止 |
| Rust | 所有権 + 借用検査 | ライフタイム `'a` の学習曲線が急峻 |
| **YaoXiang** | **Move + Token + ref** | **シンプル、決定的、GC なし** |

### 設計目標

```yaoxiang
# 1. 默认 Move（零拷贝）
p = Point(1.0, 2.0)
p2 = p                         # Move，p 不可再读

# 2. &T / &mut T 借用令牌（零开销，类型属性自然推导权限）
print_info(p2)                 # 编译器自动创建 &Point 令牌，用完即释放
shift(p2, 1.0, 1.0)           # 编译器自动创建 &mut Point 令牌

# 3. ref = 共享（编译器自动选 Rc/Arc）
shared = ref p2                # 跨作用域持有
spawn { use(shared) }          # 编译器：跨任务 → Arc

# 4. clone() = 显式复制
backup = p2.clone()            # 深拷贝，独有

# 5. unsafe + *T = 系统级
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rust との主要な違い

| 特性 | Rust | YaoXiang |
|------|------|----------|
| デフォルトセマンティクス | 借用 `&T`（明示的に `.clone()` が必要） | **Move（値渡し、ゼロコピー）** |
| 借用 | `&T`/`&mut T`、返却可能、ライフタイムが必要 | **`&T`/`&mut T` ゼロサイズトークン、Dup/Linear 型属性により自然に導出** |
| 共有機構 | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが Rc/Arc を自動選択）** |
| 複製 | `clone()` | `clone()` |
| 生ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用検査 | 大域導出 | **型検査器が借用命題を自動生成、証明パイプラインで統一検証** |
| 循環参照 | 手動 Weak | **タスク終了時の一括解放 / タスクをまたぐ lint / 標準ライブラリ Weak** |

## 提案

### 1. Move（デフォルトの所有権移転）

```yaoxiang
# 规则：赋值 / 传参 / 返回 = Move，零拷贝

p: Point = Point(1.0, 2.0)
p2 = p                           # Move，p 不可再读

# 变量可以重新赋值（Python 风格，无遮蔽）
p = Point(3.0, 4.0)              # p 重新绑定，类型必须一致

# 函数参数：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move 返回
}

# 函数返回：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move 返回，零拷贝
}
```

**特徴**：
- ゼロコピー（コンパイラがポインタを移動）
- 移動後、元のバインドは読み取り不可（コンパイルエラー）
- RAII：スコープ終了時に自動解放
- 関数シグネチャ `(T) -> T` 自体がドキュメント——T を消費し、T を返す

### 2. &T / &mut T（借用トークン）

**中核原則：`&T` と `&mut T` はゼロサイズのコンパイル時トークン型である。これらは「参照」ではなく、「アクセス権限の型レベル証明」である。**

#### 2.1 2つの型属性

```
&T      →  ゼロサイズ、ソースデータを凍結（ReadToken 生存中は WriteToken 禁止）、
          凍結保証下では複数の読み取り専用ビューが安全 → 複製可能（Dup）
&mut T  →  ゼロサイズ、排他的読み書き（WriteToken 生存中は他のトークンすべて禁止）、
          排他的アクセス下では複製が無意味 → 線形（非 Dup）
```

**因果関係を逆転させてはならない：凍結が原因、Dup は結果。** データが凍結されるから（突然変異の可能性がないから）複数の読み取り専用ビューが安全であり、Dup が実現できる、という関係ではない。Dup を定義とし、衝突検査を「追加のパッチ」とみなす設計は誤りである。

#### 2.2 基本使用

```yaoxiang
# 方法端：声明参数类型，决定需要的权限
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # &Point 令牌授予读权限
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # &mut Point 令牌授予写权限
    self.y = self.y + dy
}

# 调用端：编译器自动选择借用或 Move
p = Point(1.0, 2.0)
p.print()                          # 编译器自动创建 &Point 令牌
p.shift(1.0, 1.0)                  # 编译器自动创建 &mut Point 令牌
p.print()                          # OK，上一个令牌已随 shift 调用结束而释放

# 自由函数同理
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 两个 &Point 令牌共存——Dup 类型
}
d = distance(p, p2)
```

#### 2.3 なぜ「エスケープ禁止」は不要か

RFC-009 v8 では `&T`/`&mut T` に3つの特殊ルール——引数としてのみ使用可能、返却不可、構造体に保存不可——が課されていた。これは「借用」概念へのパッチである。

トークンシステムにはこれらのルールは不要である。トークンは**通常の型**であり、他のすべての型と同じスコープルールに従う。

**参照を返す——自然にサポート**：

```yaoxiang
# ✅ 令牌随返回值一起传播
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # 子令牌和父令牌一起返回
}

# 使用
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # 令牌返回给调用者
print(px_ref)               # OK，令牌仍在作用域
```

**構造体に保存——自然にサポート**：

```yaoxiang
# ✅ 结构体携带令牌作为字段
Window: Type = {
    target: Point,
    view: &Point,      # 令牌字段——持有对 target 的只读视图
}

# view 的令牌从 target 派生，Window 持有两者的所有权
# 只要 Window 存在，view 令牌就有效
```

#### 2.3 クロージャと Lambda の明示的引数

Lambda は関数値——返却、格納、現在のスコープ外への伝播が可能である。したがって Lambda は**外側のローカル変数を暗黙的にキャプチャしない**。外側のデータが必要な場合は、明示的引数として渡す：

```yaoxiang
# ✅ Lambda 使用显式参数
double: (x: Int) -> Int = (x) => x * 2
filter_by: (items: List(Int), f: (Int) -> Bool) -> List(Int) = { ... }

# ✅ spawn { } 不受此规则影响——spawn 是立刻执行的并发块，父任务阻塞等待
shared = ref data
spawn { use(shared) }

# ❌ Lambda 不能隐式捕获外层变量
x = 42
f = () => { x + 1 }  # 编译错误：x 不在作用域内

# ✅ 正确方式：显式传参
f = (x) => { x + 1 }
f(x)
```

**`spawn { }` は関数値ではない。** spawn でマークされた block は if/while body と同様に即座に実行され、親スタックフレーム生存中に完了する。spawn body は外側の変数に通常通りアクセスできる。

**タスクをまたぐ——トークンはスレッドをまたげない**：

```yaoxiang
# ❌ 令牌不能跨任务边界
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ 编译错误：令牌不能跨任务传递
}

# 这不是特殊规则——令牌是编译期权限证明，跨任务共享请使用 ref
# 如果需要跨任务共享，请使用 ref
```

**トークンは ref できない**：

```yaoxiang
# ❌ 令牌是权限证明，不是所有权
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ 编译错误：&T 不是可拥有类型
}
```

#### 2.4 トークンのライフタイム

トークンのライフタイムは**通常のスコープルール**で決定され、ライフタイムパラメータは不要：

- 関数引数中のトークン：呼び出し期間中生存、呼び出し終了後に解放
- 返却されるトークン：所有権を呼び出し側に移転
- 構造体に格納されたトークン：構造体と共に生存

コンパイラは `'a` 注釈を必要としない。トークンは**値**であり、値のライフタイムは所有権システム（Move/RAII）で統一的に管理される。**借用問題を所有権問題に次元削減する。**

#### 2.5 トークン衝突検出

トークン衝突検出は**ホーア論理命題**であり、独立したフローセンシティブ解析ではない。

```
{衝突の ReadToken がすべて死亡} write(data) {WriteToken は安全に取得可能}
```

型検査やユーザ述語検証と RFC-027 の証明パイプラインを共有する。コンパイラが借用命題（`borrow_conflict`、`use_after_move`、`use_after_drop`、`mut_violation`）を自動生成し、パイプラインに投入する。パイプラインは Proved / Disproved / Unproven を返す。

```yaoxiang
# ❌ &mut 令牌是线性的，不能复制
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move，p 不可再读
    p.x = 10.0                      # ❌ 编译错误：WriteToken 已被移动
}

# ✅ &T 令牌是 Dup 类型，可以自由复制
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK，&T 是 Dup 类型
    print(p.x)                      # OK
    print(p2.x)                     # OK，两个只读令牌共存
}
```

**借用検査は消えていない——次元削減されたのである。** 既存の `BorrowChecker` は `BorrowPredicateEmitter`（命題生成器）となり、生成された借用命題は他の型命題と同じ証明パイプラインを共有する。これは型検査器の概念と完全に平行している：型検査器は型等式命題を生成し、借用命題生成器は借用命題を生成し、同一のパイプラインが検証する。詳細設計は [RFC-009a](../accepted/009a-borrow-proof-pipeline.md) を参照。

#### 2.7 コンパイラ内部：ブランド機構

ユーザはブランドに一切触れない。コンパイラが内部で各トークンにコンパイル時一意の識別子を割り当てる：

```
ユーザに見える形        コンパイラ内部表現
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N はコンパイル時一意の整数
&mut Point     →  WriteToken(Point, #M)   // #M はコンパイル時一意の整数
```

ブランドの用途：
- **偽造防止**：トークンは所有カプセルからのみ取得可能で、凭空には構築できない
- **関連追跡**：`&Point` から `&Float`（フィールドアクセス）を派生するとき、`&Float` は派生ブランド（`#N.field_x`）を持ち、コンパイラは親トークンまで追跡可能
- **衝突検出**：同源の `WriteToken` と派生の `ReadToken` は同時にアクティブになれない

ブランドは単態化とインライン化後に完全に消滅し、生成された機械語には存在しない。**ランタイムオーバーヘッドはゼロ。**

#### 2.8 自動借用選択ルール

呼び出し側コンパイラは以下の優先順位で自動選択する：

```
1. 実引数が以降も使用される → トークン作成を優先（&T または &mut T、メソッドシグネチャに応じて）
2. 実引数が以降使用されない → Move
3. マッチング優先順位：&T < &mut T < Move
```

```yaoxiang
# 示例：自动选择
p = Point(1.0, 2.0)
p.print()        # print 声明 &self → 编译器创建 &Point 令牌
p.shift(1.0, 1.0) # shift 声明 &mut self → 编译器创建 &mut Point 令牌
p2 = p           # Move，p 不再使用
```

#### 2.9 RFC-009 v8 ミニ借用との比較

| 特性 | ミニ借用 (v8) | 借用トークン (v9) |
|------|--------------|--------------|
| 参照の返却 | ❌ ハードコード禁止 | ✅ トークンは戻り値と共に伝播 |
| 構造体への保存 | ❌ ハードコード禁止 | ✅ トークンは構造体フィールドとして |
| Lambda 明示的引数 | ❌ ハードコード禁止 | ✅ Lambda は明示的引数を使用 |
| 特殊ルール | 3つ（引数のみ/返却不可/保存不可） | 0——型属性により自然に導出 |
| 借用検査 | 専用の交差借用検査 | 型検査器のフローセンシティブ活性解析 |
| ライフタイム注釈 | 不要 | 不要 |
| ランタイムオーバーヘッド | ゼロ | ゼロ（ゼロサイズ型、コンパイル後に消滅） |
| エラーメッセージ | 「借用はエスケープ不可」 | 「WriteToken(#3) は既に移動済」（通常の型エラー） |
| ユーザメンタルモデル | 「借用」の特殊な位置付けを理解 | `&T` は複製可能、`&mut T` は複製不可 |

### 3. `ref` キーワード（コンパイラの自動最適化）

`ref` はスコープをまたいだ共有の唯一の方法である。内部実装が Rc か Arc かをユーザが意識する必要はない。

#### 3.1 基本使用

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共享，编译器自动选实现

# 跨任务共享
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # 编译器：跨任务 → Arc
    spawn { use(data) }           # 编译器：跨任务 → Arc
}

# 单任务共享
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # 编译器：不跨任务 → Rc
}
```

**ユーザメンタルモデル**：`ref` = 共有保持。それだけ。

#### 3.2 コンパイラのエスケープ分析：Rc vs Arc

```
ref のデータフロー解析：

他のタスクにエスケープしない → Rc（非アトミック参照カウント、低オーバーヘッド）
他のタスクにエスケープする   → Arc（アトミック参照カウント、スレッドセーフ）
```

#### 3.3 循環検出戦略

```
タスク内循環 → 静かに許可。
  ├── 各タスクには明確なライフタイム境界がある——タスク終了時、すべてのリソース（ref 循環を含む）は一括解放される。
  ├── 長時間稼働サービスはリクエスト/接続ごとに子タスクを作成すべき——子タスク終了時に自動回収され、累積リークは発生しない。
  ├── ref は常に生存し続け、セマンティクスは混じりけがない。
  └── ユーザはタスク内で双方向強参照を構築する権利を持つ（例：グラフ計算の中間状態）。

タスクをまたぐ循環 → lint（デフォルト warn、設定可能）。
  ├── プログラムの挙動は正しく、実際にリークすることはない（親タスク終了時、子タスクのリソースはすべて解放される）。
  ├── ただしタスクをまたぐ強参照は所有権境界が曖昧であることを意味し、立ち止まって再考する価値がある。
  ├── デフォルトは warn レベル、コンパイルは通るが警告が出る。
  └── チームはプロジェクト設定で deny にでき、CI 品質ゲートに組み込める。
```

**Lint レベル**（Rust の clippy と同様）：

| レベル | 動作 | シナリオ |
|------|------|------|
| `allow` | チェックしない | 個人プロジェクト |
| `warn`（デフォルト） | コンパイルは通るが警告あり | 開発フェーズ |
| `deny` | コンパイル失敗 | チーム CI 品質ゲート |
| `forbid` | コンパイル失敗、上書き不可 | 組織レベルの強制ルール |

```yaoxiang
# 任务内环：静默允许，双向强引用
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # 环。任务结束时统一释放。
}

# 跨任务环：lint（默认 warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: 跨任务循环引用
    }
}
```

**プロジェクト設定例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # 跨任务环在 CI 上直接拒绝
```

| 循環タイプ | 動作 | 理由 |
|--------|------|------|
| タスク内 ref 循環 | チェックしない | ユーザの権利、タスク終了時一括解放 |
| タスクをまたぐ ref 循環 | lint（デフォルト warn） | 再考を促す、deny に設定可能 |

#### 3.4 Weak：標準ライブラリ提供

```yaoxiang
use std.rc.Weak

# 高级用户显式选择
a.next = ref b
b.prev = Weak.new(a.next)        # 用户显式控制哪个方向是弱的
```

**`Weak` は言語組み込みではなく、標準ライブラリの型である。** 日常的には `ref` で十分。メモリを詳細に制御したい上級ユーザは手動で `Weak` を導入する。

#### 3.5 借用トークン vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 役割 | ちら見 / その場変更 | 共有保持 |
| 範囲 | トークン値のスコープに従う | スコープをまたぐ |
| コスト | ゼロコスト（ゼロサイズ型） | Rc または Arc（コンパイラ選択） |
| エスケープ | 可（トークンは戻り値/構造体/クロージャで伝播） | 本来エスケープ用 |
| タスクをまたぐ | 不可（トークンはコンパイル時権限証明、タスク境界をまたげない） | 可（コンパイラが Arc を自動選択） |
| 循環 | 関係しない | タスク内は静かに許可、タスクをまたぐ場合は lint |

---

### 4. `clone()` —— 明示的複製

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # 深拷贝
# p 和 p2 独立，互不影响
```

**いつ使うか**：元の値を保持する必要があり、Move も共有も適さないシナリオ。

### 5. `unsafe` + 生ポインタ（システムレベルプログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 裸指针
    (*ptr).x = 0.0                # 解引用（用户保证安全）
    ptr2 = ptr + 1                # 指针运算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ユーザはダングリングや解放後使用がないことを保証
- FFI、メモリ操作などのシステムレベルプログラミングに使用

### 6. 所有権グラデーション総覧

```
  借用トークン（ゼロコスト）    Move（ゼロコスト）     共有（必要に応じて課金）   複製
   │                      │                  │                │
  &T 複製可能トークン    デフォルト所有権移転  ref Rc/Arc      clone()
  &mut T 線形トークン    チェイン消費還流      コンパイラ自動選択 明示的ディープコピー
   │                      │                  │                │
  トークン値スコープ        スコープ内          スコープをまたぐ   いつでも
  返却/構造体保存可能       T -> T 還流         ref タスクをまたぐ → Arc 独立コピー
  ゼロサイズコンパイル後消滅  T -> Void 消費     ref タスク内 → Rc
  ゼロサイズコンパイル後消滅                    タスク内循環は静かに
                                                タスクをまたぐ循環は lint
                                                標準ライブラリ Weak でのエスケープ
```

## 総合例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：只读令牌
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：可变令牌
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消费回流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 拿走，改，还给你
    }

    # 返回引用：令牌随返回值传播
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# Lambda 显式参数
double: (x: Int) -> Int = (x) => x * 2

# 综合使用
p = Point(1.0, 2.0)
p.print()                           # &Point 令牌
p.shift(1.0, 1.0)                   # &mut Point 令牌
p = p.scale(2.0)                    # Move → 回流
shared = ref p                      # ref 共享
spawn { use(shared) }

# clone 独立副本
backup = p.clone()

# 任务内环：静默允许
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # 环，任务结束时统一释放

# unsafe 系统级
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Dup 型属性

`Dup`（Duplicable）はコンパイラが自動管理する型属性であり、**シャローコピー**を意味する：代入/引数渡し時にはハンドル/トークンがコピーされ、基盤データは共有される。これは Move（所有権移転）および Clone（明示的ディープコピー、独立コピー作成）との三段階グラデーションを形成する。

**Dup と Clone は直交する概念**——Dup はハンドルをコピーしてデータを共有し、Clone は独立コピーを作成する。一つの型が Dup と Clone の両方をサポートすることも、いずれか一方のみをサポートすることも可能。

| 型 | Dup | Clone | 説明 |
|------|-----|-------|------|
| `&T` | ✅（トークン複製、複数視点が同一データを指す） | ✅ | 読み取り専用トークン |
| `ref T` | ✅（参照カウント+1、ヒープデータ共有） | ✅ | 共有保持（コンパイラが Rc/Arc を自動選択） |
| String, Bytes | ✅（内部参照カウント、ハンドル共有で基盤バッファを共有） | ✅ | 文字列/バイト |
| `&mut T` | ❌（線形、排他） | ❌ | 可変トークン |
| `*T` | ❌ | ❌ | 生ポインタ |
| struct | 派生（全フィールドが Dup の場合自動派生） | ✅ | 構造体 |

**プリミティブ値型**（Int, Float, Bool, Char）の代入動作はコンパイラ組み込みの値コピー——2つの値は完全に独立しており、シャローコピーではない。これらは Dup 型属性には属さず、コンパイラのネイティブ処理である。

## 性能分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | ゼロ | ポインタ移動 |
| `&T` / `&mut T` | ゼロ | ゼロサイズ型、コンパイル後に消滅、ランタイムオーバーヘッドゼロ |
| `ref`（タスクをまたがない）| 低 | Rc にコンパイル、非アトミック操作 |
| `ref`（タスクをまたぐ）| 中 | Arc にコンパイル、アトミック操作 |
| `clone()` | 型による | 小オブジェクトは高速、大オブジェクトは低速 |
| `unsafe + *T` | ゼロ | 直接メモリ操作 |

### 比較

| 言語 | 共有機構 | メモリ管理 | 循環処理 | 複雑度 |
|------|----------|----------|----------|--------|
| Rust | Arc / Mutex + 借用検査 | コンパイル時検査 | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 借用トークン** | **RAII** | **タスク境界での解放 / タスクをまたぐ lint / 標準ライブラリ Weak** | **低** |

## トレードオフ

### 利点

1. **統一**：`&T`/`&mut T` は通常の型であり、特殊言語機能ではない。RFC-010 の `name: type = value` と完全一致
2. **シンプル**：ライフタイムなし、借用検査は型システム命題に次元削減。`&T` は複製可能、`&mut T` は複製不可——2つの型属性
3. **強力**：参照返却、構造体保存、クロージャキャプチャが可能——表現力は Rust と同等
4. **コンパイラの賢さ**：ref は Rc/Arc を自動選択、呼び出し側は借用を自動選択
5. **決定性**：ref は生存を保証し、密かに弱参照にはならない
6. **高性能**：Move ゼロコピー、トークンゼロコスト（ゼロサイズ型、コンパイル後に消滅）
7. **柔軟性**：`unsafe + *T` でシステムレベルプログラミングをサポート

### 欠点

1. **ジェネリクスブランド引数の伝染**：トークンはブランド識別子を持ち、参照を返す関数シグネチャには追加のジェネリクス引数が反映される
2. **ref のランタイムオーバーヘッド**：アトミック操作にはコストがかかる（ただしこれは共有の本質的代償）
3. **`unsafe` のリスク**：ユーザは正しさを保証しなければならない
4. **タスクをまたぐ循環は lint でありコンパイルエラーではない**：Rust のようにコンパイルエラーではなく、デフォルトは warn、チーム設定で deny にして初めて品質ゲートになる

---

## 代替案

| 代替案 | 採用しない理由 |
|------|--------------|
| GC | ランタイムオーバーヘッドあり、停止の予測不可 |
| Rust 借用検査器 | ライフタイム `'a` が必要、学習曲線が急峻 |
| 純粋な Move | 並行共有を扱えない |
| 生ポインタなし | システムレベルプログラミング不可 |
| Rc/Arc をユーザに公開 | 実装詳細をユーザに投げ、認知負担を増大 |
| ミニ借用（v8） | エスケープ禁止戦略はクロージャキャプチャ、参照返却などの重要な表現力を犠牲にする |

### トークン衝突検出：ホーア論理命題

トークン衝突検出の完全な方案は [RFC-009a: トークンライフタイム解析——ホーア証明パイプラインに基づく](../accepted/009a-borrow-proof-pipeline.md) を参照。中核要点：

**トークン活性はホーア論理命題である。** `{衝突の ReadToken がすべて死亡} write(data) {WriteToken は安全に取得可能}`——型検査、ユーザ述語検証と RFC-027 の証明パイプラインを共有する。コンパイラが借用命題（`borrow_conflict`、`use_after_move`、`use_after_drop`、`mut_violation`）を自動生成し、パイプラインが Proved / Disproved / Unproven を返す。

**借用検査は消えていない——次元削減されたのである。** `BorrowChecker` は `BorrowPredicateEmitter` となり、検査ではなく命題を生成する。これは「型検査器」の概念と完全に平行している：型検査器は型等式命題を生成し、借用命題生成器は借用命題を生成し、同一のパイプラインが検証する。

**ブランド ID（`#42`）は `'a` そのものである。** 情報は完全に同じ、エンコーディングが異なる。`'a` は型シグネチャに可視、ブランド `#42` はコンパイラ内部。新規解析を発明したのではない——ライフタイムを型層から証明層に降ろしたのである。

**アルゴリズム概要**（詳細は RFC-009a）：
- ブランドツリーのプレフィックスマッチング → 衝突トークン特定（O(depth)、深さ ≤ 3）
- 逆方向 BFS → 消費者から出発、break で帰還辺を切断、構造解析で 95%+ のシナリオをカバー（高速パス）
- SMT 論理切断 → while + パス条件時のみ呼び出し（低速パス、極めて稀）
