# RFC-009：所有権モデル設計

> **ステータス**: 承認済み
> **著者**: 晨煦
> **作成日**: 2025-01-08
> **最終更新**: 2026-05-29（借用トークンシステムで丐版借用に替代、统一的な型システム）

## 概要

このドキュメントは YaoXiang プログラミング言語の**所有権モデル（Ownership Model）**を定義します。

**核心設計——5つの概念、1つのグラデーション**：

```
ちょっと見る/その場で変更     移動           共有所有権      複製一份        システム級
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         ゼロコピー      コンパイラ自動     明示的深コピー     *T
  ゼロサイズトークン     デフォルト      Rc/Arcを選択      显式深拷贝      ユーザ責任
  型の属性で自然に                          
  権限を導出                              
```

- **Move（デフォルト）**：代入/引数渡/戻り値 = 所有権移動、ゼロコピー、RAII 自動解放
- **`&T` / `&mut T`（借用トークン）**：ゼロサイズのコンパイル時トークン型。`&T` は複製可能（共有読み取り専用）、`&mut T` は線形（排他的可变）。権限は型の属性から自然に導出され、特殊ルールは不要。戻り値可能、構造体に格納可能、クロージャでキャプチャ可能。
- **`ref` キーワード**：スコープを跨いだ共有。コンパイラが Rc（タスクを跨がない）か Arc（タスクを跨ぐ）を自動選択
- **`clone()`**：明示的深コピー
- **`unsafe` + `*T`**：裸ポインタ、システム級エスケープハッチ

**排除した複雑さ**：
- ❌ ライフタイム `'a` なし
- ❌ 借用チェッカなし（トークンの型属性——Dup/Linear——が专用の borrow checker を代替）
- ❌ GC なし
- ❌ 「エスケープ禁止」などの特殊ルールなし（トークンは通常の型であり、スコープは型システムで統一的に処理）
- ❌ ユーザーは Rc/Arc の違いを知る必要はない（コンパイラが自動選択）

> **プログラミング負担**：`&T` は複製可能、`&mut T` は複製不可——型の属性2つ、特殊ルール0つ、コンパイラが全自动。
> **パフォーマンス保証**：Move ゼロオーバーヘッド、トークン ゼロオーバーヘッド（ゼロサイズ型、コンパイル後に消失）、ref が必要に応じた支払い、GC 一時停止なし。

## 動機

### なぜ所有権モデルが必要인가？

| 言語 | メモリ管理 | 問題点 |
|------|----------|------|
| C/C++ | 手動管理 | メモリリーク、野ポインタ、二重解放 |
| Java/Python | GC | 遅延波动、メモリオーバーヘッド、予測不能な一時停止 |
| Rust | 所有権 + 借用チェッカ | ライフタイム `'a` の学習コストが高い |
| **YaoXiang** | **Move + トークン + ref** | **シンプル、確定、GC なし** |

### 設計目標

```yaoxiang
# 1. デフォルト Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p                         # Move、p はこれ以上読み取り不可

# 2. &T / &mut T 借用トークン（ゼロオーバーヘッド、型の属性で自然に権限を導出）
print_info(p2)                 # コンパイラが自動的に &Point トークンを作成、使用後に解放
shift(p2, 1.0, 1.0)           # コンパイラが自動的に &mut Point トークンを作成

# 3. ref = 共有（コンパイラが Rc/Arc を自動選択）
shared = ref p2                # スコープを跨いで保持
spawn { use(shared) }          # コンパイラ：タスクを跨ぐ → Arc

# 4. clone() = 明示的複製
backup = p2.clone()            # 深コピー、独立所有

# 5. unsafe + *T = システム級
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rust との核心的な違い

| 機能 | Rust | YaoXiang |
|------|------|----------|
| デフォルトセマンティクス | 借用 `&T`（明示的 `.clone()` が必要） | **Move（値渡し、ゼロコピー）** |
| 借用 | `&T`/`&mut T`、戻り値可能、ライフタイム必要 | **`&T`/`&mut T` ゼロサイズトークン、Dup/Linear 型の属性で自然に権限を導出** |
| 共有メカニズム | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが Rc/Arc を自動選択）** |
| 複製 | `clone()` | `clone()` |
| 裸ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用チェッカ | 大域的導出 | **型チェッカのフロー感受性活性分析（トークン状態追跡）** |
| 循環参照 | 手動 Weak | **タスク終了時の統一解放 / タスク間 lint / 標準庫 Weak** |

## 提案

### 1. Move（デフォルト所有権移動）

```yaoxiang
# ルール：代入 / 引数渡 / 戻り値 = Move、ゼロコピー

p: Point = Point(1.0, 2.0)
p2 = p                           # Move、p はこれ以上読み取り不可

# 変数は再代入可能（Python スタイル、シェードなし）
p = Point(3.0, 4.0)              # p を再束縛、型は同じでなければならない

# 関数引数：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move 戻り値
}

# 関数戻り値：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move 戻り値、ゼロコピー
}
```

**特徴**：
- ゼロコピー（コンパイラがポインタを移動）
- 移動後、元の束縛は読み取り不可（コンパイルエラー）
- RAII：スコープ終了時に自動解放
- 関数シグネチャ `(T) -> T` それ自体がドキュメント——T を消費し、T を返す

### 2. &T / &mut T（借用トークン）

**核心原則：`&T` と `&mut T` はゼロサイズのコンパイル時トークン型です。それらは「参照」ではなく、「アクセス権限の型レベル証明」です。**

#### 2.1 2つの型属性

```
&T      →  ゼロサイズ、複製可能（Dup）、読み取り専用権限を付与
&mut T  →  ゼロサイズ、線形（非Dup）、排他的読み書き権限を付与
```

**これは記憶すべき「ルール」ではなく——型システムの基本的な属性です。** Dup 型は自由に複製可能（複数の `&T` が共存可能）、線形型は複製不可（`&mut T` は本能的に一意）。「借用チェッカ」はありません——型チェッカがずっと行ってきたことを続けているだけです。

#### 2.2 基本使用

```yaoxiang
# メソッド側：引数型を宣言し、必要な権限を決定
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # &Point トークンが読み取り権限を付与
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # &mut Point トークンが書き込み権限を付与
    self.y = self.y + dy
}

# 呼び出し側：コンパイラが借用または Move を自動選択
p = Point(1.0, 2.0)
p.print()                          # コンパイラが自動的に &Point トークンを作成
p.shift(1.0, 1.0)                  # コンパイラが自動的に &mut Point トークンを作成
p.print()                          # OK、前のトークンは shift 呼び出し終了時に解放済み

# 自由関数も同じ
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 2つの &Point トークンが共存——Dup 型
}
d = distance(p, p2)
```

#### 2.3 なぜ「エスケープ禁止」が不要か

RFC-009 v8 の `&T`/`&mut T` には3つの特殊ルールが課されていました——パラメータとしてのみ使用可能、戻り値不可、構造体に格納不可。これは「借用」概念へのパッチ当てでした。

トークンシステムではこれらのルールは不要です。トークンは**通常の型**であり、他のすべての型と同じスコープルールに従います。

**参照を返す——自然にサポート**：

```yaoxiang
# ✅ トークンが戻り値と共に伝播
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # サブトークンと부모トークンが一緒に返される
}

# 使用例
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # トークンが呼び出し元に返される
print(px_ref)               # OK、トークンはスコープ内
```

**構造体に格納——自然にサポート**：

```yaoxiang
# ✅ 構造体がトークンをフィールドとして携带
Window: Type = {
    target: Point,
    view: &Point,      # トークンフィールド——target への読み取り専用ビューを保持
}

# view のトークンは target から派生、Window が両方の所有権を保持
# Window が存在する限り、view トークンは有効
```

**クロージャキャプチャ——自然にサポート**：

```yaoxiang
# ✅ クロージャがトークンをキャプチャ、他の値と同様に
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    # クロージャが threshold の &Float トークンをキャプチャ（Dup 型、自由にクロージャに複製可能）
    items.filter(|p| p.x > threshold)
}

# これは RFC-009 v8 ではできませんでした——v8 はクロージャでの借用キャプチャを禁止
```

**タスク間——トークンはスレッドを跨げない**：

```yaoxiang
# ❌ トークンはタスク境界を跨げない
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ コンパイルエラー：トークンはタスクを跨げない
}

# これは特殊ルールではない——トークンはコンパイル時の権限証明、タスク間共有には ref を使用
# タスク間共有が必要な場合は ref を使用
```

**トークンは ref できない**：

```yaoxiang
# ❌ トークンは権限証明であり、所有権ではない
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ コンパイルエラー：&T は所有可能的型ではない
}
```

#### 2.4 トークンのライフサイクル

トークンのライフサイクルは**通常のスコープルール**で決定され、ライフタイムパラメータは不要です：

- 関数引数内のトークン：呼び出し期間中に生存、呼び出し終了後に解放
- 返されるトークン：所有権が呼び出し元に移動
- 構造体に格納されたトークン：構造体と共に生存
- クロージャでキャプチャされたトークン：クロージャと共に生存

コンパイラは `'a` 注釈を必要としません、なぜならトークンは**値**であり、値のライフサイクルは所有権システム（Move/RAII）で統一管理されるからです。**借用問題を所有権問題に変換（降維）しました。**

#### 2.5 冻结メカニズム

`&mut T` トークンは一時的に「冻结」して `&T` トークンを生成できます：

```yaoxiang
modify_and_read: (p: &mut Point) -> Void = {
    p.x = 10.0                      # &mut Point を使用して変更
    
    # &mut を冻结して、読み取り専用ビューを取得
    view: &Point = freeze(p)         # p はここで冻结される
    print(view.x)                   # &Point を通じて読み取り
    print(view.y)
    # view がスコープを離れる、冻结解除
    
    p.y = 20.0                      # &mut Point が利用可能に
}
```

`freeze` のセマンティクス：
- `&mut T` を受け取り、`&T` を返す
- `&T` が生存している間、元 の `&mut T` は利用不可
- `&T` がスコープを離れた後、`&mut T` は自動的に回復
- これは**フロー感受性分析**——コンパイラが関数本体でトークン状態を追跡

#### 2.6 トークン衝突検出

RFC-009 v8 の「交差借用チェッカ」を代替。原理はよりシンプル——トークン値の**フロー感受性活性分析**：

```yaoxiang
# ❌ &mut と派生した &T は同時にアクティブになれない
bad_alias: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p が冻结される
    p.x = 10.0                      # ❌ コンパイルエラー：WriteToken が冻结状態
    print(view.x)                   
}

# ✅ 冻结解除後に &mut を続けて使用可能
good_seq: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p が冻结される
    print(view.x)                   # &T を使用
    # view がスコープを離れる
    p.x = 10.0                      # ✅ WriteToken が回復済み
}
```

**検出方式**：これは专用の「借用チェッカ」ではなく——これはトークン値の**フロー感受性活性分析**です。コンパイラが関数本体で各トークンの状態（アクティブ/冻结/移動済み）を追跡し、任意の線形型値の追跡方法と同じです。

#### 2.7 コンパイラ内部：ブランドメカニズム

ユーザーはブランドに触れません。コンパイラは内部で各トークンにコンパイル時一意の識別子を割り当てます：

```
ユーザーが見るもの         コンパイラ内部表現
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N はコンパイル時一意の整数
&mut Point     →  WriteToken(Point, #M)   // #M はコンパイル時一意の整数
```

ブランドの用途：
- **偽造防止**：トークンは所有者の胶囊または freeze 操作からのみ取得でき凭空構築不可
- **関連追跡**：`&Point` から `&Float`（フィールドアクセス）を派生する際、`&Float` は派生ブランド（`#N.field_x`）を携带、コンパイラは부모トークンまで追跡可能
- **衝突検出**：同源 `WriteToken` と派生 `ReadToken` は同時にアクティブになれない

ブランドは単態化とインライン展開後に完全に消失し、生成了機械語には存在しません。**ゼロランタイムオーバーヘッド。**

#### 2.8 自動借用選択ルール

呼び出し側コンパイラは以下の優先順位で自動選択：

```
1. 実引数がその後も使用される場合 → トークン作成を優先（&T または &mut T、メソッドシグネチャによる）
2. 実引数がその後は使用されない場合 → Move
3. 優先順位：&T < &mut T < Move
```

```yaoxiang
# 例：自動選択
p = Point(1.0, 2.0)
p.print()        # print は &self を宣言 → コンパイラが &Point トークンを作成
p.shift(1.0, 1.0) # shift は &mut self を宣言 → コンパイラが &mut Point トークンを作成
p2 = p           # Move、p はこれ以上使用しない
```

#### 2.9 RFC-009 v8 丐版借用との比較

| 機能 | 丐版借用 (v8) | 借用トークン (v9) |
|------|--------------|--------------|
| 参照を返す | ❌ ハードコードで禁止 | ✅ トークンが戻り値と共に伝播 |
| 構造体に格納 | ❌ ハードコードで禁止 | ✅ トークンが構造体フィールドとして |
| クロージャキャプチャ | ❌ ハードコードで禁止 | ✅ クロージャがトークン値をキャプチャ |
| 特殊ルール | 3つ（パラメータのみ/戻り値不可/格納不可） | 0つ——型の属性で自然に導出 |
| 借用チェッカ | 专用交差借用チェッカ | 型チェッカのフロー感受性活性分析 |
| ライフタイム注釈 | 不要 | 不要 |
| ランタイムオーバーヘッド | ゼロ | ゼロ（ゼロサイズ型、コンパイル後に消失） |
| エラーメッセージ | "借用はエスケープできない" | "WriteToken(#3) が冻结済み"（通常の型エラー） |
| ユーザー心智モデル | 「借用」の特殊地位を理解する | `&T` は複製可能、`&mut T` は複製不可 |

### 3. ref キーワード（コンパイラ自動最適化）

`ref` はスコープを跨いだ共有の唯一の方法。基盤は Rc 还是 Arc、ユーザーは気にする必要はない。

#### 3.1 基本使用

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共有、コンパイラが自動的に実装を選択

# タスク間共有
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # コンパイラ：タスクを跨ぐ → Arc
    spawn { use(data) }           # コンパイラ：タスクを跨ぐ → Arc
}

# 単一タスク内共有
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # コンパイラ：タスクを跨がない → Rc
}
```

**ユーザー心智モデル**：`ref` = 共有所有権。それだけ。

#### 3.2 コンパイリエスケープ分析：Rc vs Arc

```
ref のデータフロー分析：

他のタスクにエスケープしない → Rc（非原子参照カウント、低コスト）
他のタスクにエスケープする   → Arc（原子参照カウント、スレッドセーフ）
```

#### 3.3 循環検出戦略

```
タスク内循環 → 静かに許可。
  ├── 構造化並行がタスク終了時にすべてのリソースを統一的に解放することを保証。
  ├── ref は常に生存し、セマンティクスに偽りなし。
  └── ユーザーはタスク内で双方向の強参照を構築する権利を持つ（例：グラフ計算の中間状態）。

タスク間循環 → lint（デフォルト warn、構成可能）。
  ├── プログラム動作は正しい、本当のリークは発生しない（부모タスク終了時に子タスクのリソースは全て解放）。
  ├── しかしタスク間強参照は所有権境界が曖昧であることを意味，重新檢討 заслуговує。
  ├── デフォルト warn レベル、サイズは 通るがヒントあり。
  └── チームはプロジェクト構成で deny に設定し、CI 品質ゲートに含めることができる。
```

**Lint レベル**（Rust clippy 类似）：

| レベル | 動作 | シナリオ |
|------|------|------|
| `allow` | 検査なし | 個人プロジェクト |
| `warn`（デフォルト） | サイズが 通る、ヒントあり | 開発段階 |
| `deny` | サイズが 失敗 | チーム CI 品質ゲート |
| `forbid` | サイズが 失敗、上書き不可 | 組織レベル強制ルール |

```yaoxiang
# タスク内循環：静かに許可、双方向強参照
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # 循環。タスク終了時に統一解放。
}

# タスク間循環：lint（デフォルト warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: タスク間循環参照
    }
}
```

**プロジェクト構成例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # タスク間循環は CI で直接拒否
```

| 循環タイプ | 動作 | 理由 |
|--------|------|------|
| タスク内 ref 循環 | 検査なし | ユーザーの権限、タスク終了時に統一解放 |
| タスク間 ref 循環 | lint（デフォルト warn） | 再考を促すよう促す、deny に構成可能 |

#### 3.4 Weak：標準庫提供

```yaoxiang
use std.rc.Weak

# 上級ユーザーが明示的に選択
a.next = ref b
b.prev = Weak.new(a.next)        # ユーザーがどの方向を弱にするかを明示的に制御
```

**`Weak` は言語組み込みではなく、標準庫の型です。** 日常的には `ref` で十分。メモリを詳細に制御する必要がある上級ユーザーは手動で `Weak` を导入します。

#### 3.5 借用トークン vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 何をするか | ちょっと見る/その場で変更 | 共有所有権 |
| 範囲 | トークン値のスコープに従う | スコープを跨ぐ |
| コスト | ゼロオーバーヘッド（ゼロサイズ型） | Rc または Arc（コンパイラ選択） |
| エスケープ | 可能（トークンが戻り値/構造体/クロージャと共に伝播） | そもそもエスケープするためのもの |
| タスク間 | 不可（トークンはコンパイル時の権限証明、タスクを跨げない） | 可（コンパイラが自動的に Arc を選択） |
| 循環 | 関係ない | タスク内循環は静かに許可、タスク間は lint |

---

### 4. clone() —— 明示的複製

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # 深コピー
# p と p2 は独立、互いに影響しない
```

**使用タイミング**：元の値を保持する必要があり、Move にも共有にも適さない場合。

### 5. unsafe + 裸ポインタ（システム級プログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 裸ポインタ
    (*ptr).x = 0.0                # 逆参照（ユーザーが安全を保証）
    ptr2 = ptr + 1                # ポインタ演算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ユーザーはダングリング、解放後使用しないことを保証
- FFI、メモリ操作などシステム級プログラミングに使用

### 6. 所有権グラデーション総覧

```
  借用トークン（ゼロオーバーヘッド）     Move（ゼロオーバーヘッド）      共有（必要に応じた支払い）    複製
   │                      │                  │                │
  &T 複製可能トークン        デフォルト所有権移動     ref Rc/Arc       clone()
  &mut T 線形トークン      チェーン消費回流       コンパイラ自動選択     明示的深コピー
   │                      │                  │                │
  トークン値スコープ           スコープ内           スコープを跨ぐ         いつでも
  戻り値/構造体格納可能      T -> T 回流        ref タスク間 → Arc  独立コピー
  クロージャでキャプチャ可能   T -> Void 消費     ref 単一タスク内 → Rc
  ゼロサイズコンパイル後消失                          タスク内循環は静かに
                                            タスク間循環は lint
                                            標準庫 Weak エスケープ
```

## 綜合例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：読み取り専用トークン
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：可変トークン
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消費回流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 拿走、変更、戻す
    }

    # 参照を返す：トークンが戻り値と共に伝播
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# クロージャがトークンをキャプチャ（v8 ではできない能力）
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}

# 綜合使用
p = Point(1.0, 2.0)
p.print()                           # &Point トークン
p.shift(1.0, 1.0)                   # &mut Point トークン
p = p.scale(2.0)                    # Move → 回流
shared = ref p                      # ref 共有
spawn { use(shared) }

# clone 独立コピー
backup = p.clone()

# タスク内循環：静かに許可
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # 循環、タスク終了時に統一解放

# unsafe システム級
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Dup 型属性

`Dup`（Duplicable）はコンパイラが自動管理する型属性であり、意味は**浅コピー**：代入/引数渡時に複製されるのはハンドラ/トークンであり、基盤データは共有されます。これは Move（所有権移動）や Clone（明示的深コピー、独立コピーを作成）とは異なる3段階グラデーションを形成します。

**Dup と Clone は直交する概念**——Dup はハンドラを複製してデータを共有し、Clone は独立コピーを作成します。型は Dup と Clone の両方をサポートすることも、いずれか一方のみサポートすることもできます。

| 型 | Dup | Clone | 説明 |
|------|-----|-------|------|
| `&T` | ✅（トークンを複製、複数のビューが同じデータを指す） | ✅ | 読み取り専用トークン |
| `ref T` | ✅（参照カウント+1、ヒープデータを共有） | ✅ | 共有所有権（コンパイラが Rc/Arc を自動選択） |
| String, Bytes | ✅（内部参照カウント、ハンドラを複製して基盤 buffer を共有） | ✅ | 文字列/バイト |
| `&mut T` | ❌（線形、排他） | ❌ | 可変トークン |
| `*T` | ❌ | ❌ | 裸ポインタ |
| struct | 派生（すべてのフィールドが Dup の場合に自動派生） | ✅ | 構造体 |

**原語値型**（Int, Float, Bool, Char）の代入動作はコンパイラの組み込み値コピー——2つの値は完全に独立、浅コピーではない。それらは Dup 型属性に属さず、コンパイラのネイティブ処理です。

## パフォーマンス分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | ゼロ | ポインタ移動 |
| `&T` / `&mut T` | ゼロ | ゼロサイズ型、コンパイル後に消失、ゼロランタイムオーバーヘッド |
| `ref`（タスクを跨がない）| 低 | Rc にコンパイル、非原子操作 |
| `ref`（タスクを跨ぐ）| 中 | Arc にコンパイル、原子操作 |
| `clone()` | 型による | 小さいオブジェクトは速い、大きいオブジェクトは遅い |
| `unsafe + *T` | ゼロ | 直接メモリ操作 |

### 比較

| 言語 | 共有メカニズム | メモリ管理 | 循環処理 | 複雑さ |
|------|----------|----------|----------|--------|
| Rust | Arc / Mutex + 借用チェッカ | コンパイル時検査 | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 借用トークン** | **RAII** | **タスク境界解放 / タスク間 lint / 標準庫 Weak** | **低** |

## トレードオフ

### メリット

1. **統一**：`&T`/`&mut T` は通常の型であり、特殊言語機能ではない。RFC-010 の `name: type = value` と完全に一貫
2. **シンプル**：ライフタイムなし、大域的借用チェッカなし。`&T` は複製可能、`&mut T` は複製不可——型の属性2つ
3. **強力**：参照を返す、構造体に格納、クロージャでキャプチャ——Rust と同等の表現力
4. **コンパイラ知的**：ref が Rc/Arc を自動選択、呼び出し側で借用を自動選択
5. **確定的**：ref は常に生存、弱参照に悄悄變弱しない
6. **高性能**：Move ゼロコピー、トークン ゼロオーバーヘッド（ゼロサイズ型、コンパイル後に消失）
7. **柔軟**：`unsafe + *T` がシステム級プログラミングをサポート

### デメリット

1. **ジェネリックブランドパラメータ伝染**：トークンがブランド識別子を携带、参照を返す関数のシグネチャに追加のジェネリックパラメータが体现
2. **ref ランタイムオーバーヘッド**：原子操作にコストあり（しかしこれは共有の必然的代償）
3. **unsafe リスク**：ユーザーは正確性を保証する必要がある
4. **タスク間循環は lint 而不是コンパイルエラー**：Rust のようにコンパイルエラーにならず、デフォルト warn、チーム構成で deny に設定して品質ゲートにする必要がある

## 代替案

| 案 | なぜ選択しないか |
|------|--------------|
| GC | ランタイムオーバーヘッドがあり、予測不能な一時停止がある |
| Rust 借用チェッカ | ライフタイム `'a` が必要で、学習コストが高い |
| 純粋 Move | 並行共有を処理できない |
| 裸ポインタなし | システム級プログラミングができない |
| Rc/Arc をユーザーに露出 | 実装詳細をユーザーに押し付け、認知負担を増加 |
| 丐版借用（v8） | エスケープ禁止の戦略により、クロージャキャプチャ、参照戻り値などの重要な表現能力を犠牲にした |

### トークン衝突検出：フロー感受性活性分析

**分析範囲**：関数本体のみ。トークン値に対してフロー感受性活性分析を行い、各トークンの状態（アクティブ/冻结/移動済み）を追跡。

**レベル 1：呼び出し点検査**——同一の実引数が同時に `&mut` トークン和其他トークンを作成できない：

```yaoxiang
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)    # ❌ p が同時に &mut と & トークンを派生、コンパイラが拒否
```

**レベル 2：関数本体フロー感受性**——`&mut` トークンが呼び出しに渡された後、呼び出しから戻るとトークンが解放され、その後 다시 トークンを作成可能：

```yaoxiang
process_twice: (p: &mut Point) -> Void = {
    shift(p, 1.0, 1.0)    # &mut トークンを shift に渡し、shift が戻るとトークンが解放
    print_info(p)          # &Point トークンを重新作成、競合なし
}
```

**レベル 3：冻结状態追跡**——`freeze` によって生成された `&T` トークンがアクティブな間、元 の `&mut` トークンは利用不可：

```yaoxiang
frozen: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)    # p が冻结状態に入る
    print(view.x)
    p.x = 10.0                  # ❌ コンパイルエラー：WriteToken が冻结状態にある
}
```

**必要ないもの**：関数間ライフタイム追跡、大域的エイリアス分析、借用グラフ制約解決、NLL、`'a` 注釈。トークンは値であるため、値の活性分析は型チェッカで統一的に処理——任意の線形型値の追跡方法と同じです。
