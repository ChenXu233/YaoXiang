# RFC-009: 所有権モデル設計

> **状態**: 採用済み
> **著者**: 晨煦
> **作成日**: 2025-01-08
> **最終更新**: 2026-05-29（借用トークンシステムで丐版借用を置き換え、型システムを統一）

## 要約

このドキュメントは YaoXiang プログラミング言語の**所有権モデル（Ownership Model）**を定義します。

**コアデザイン——5つの概念、1つのグラデーション**：

```
一目見る/その場で改     拿走           共有持有         一つコピー        システム級
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         零コピー        コンパイル時自動      明示的深コピー      *T
  零サイズトークン       デフォルト          Rc/Arcを選択                   ユーザーが責任
  タイプ属性自然                                
  推論権限                              
```

- **Move（デフォルト）**：代入/引数渡/戻り値 = 所有権移動、零コピー、RAII 自動解放
- **`&T` / `&mut T`（借用トークン）**：零サイズのコンパイル時トークン型。`&T` はコピー可能（共有読み取り専用）、`&mut T` は線形（排他的可变）。権限は型属性から自然に推論され、特別な規則は不要。戻り値可能、構造体に格納可能、クロージャにキャプチャ可能。
- **`ref` キーワード**：スコープをまたぐ共有。コンパイラが Rc（タスクをまたがない）か Arc（タスクをまたぐ）を自動選択
- **`clone()`**：明示的深コピー
- **`unsafe` + `*T`**：裸ポインタ、システム級の脱出ハッチ

**排除された複雑性**：
- ❌ ライフタイム `'a` なし
- ❌ 借用チェッカーなし（トークンの型属性——Copy/Linear——が専用の borrow checker を代替）
- ❌ GC なし
- ❌ 「エスケープ禁止」などの特殊規則なし（トークンは обычный тип、スコープは型システムで統一処理）
- ❌ ユーザーは Rc/Arc の違いを知る必要なし（コンパイラが自動選択）

> **プログラミング負担**：`&T` はコピー可能、`&mut T` はコピー不可——2つの型属性、ゼロの特殊規則、コンパイラ完全自動。
> **パフォーマンス保証**：Move 零オーバーヘッド、トークン零オーバーヘッド（零サイズ型、コンパイル後に消失）、ref 必要時のみコスト、GC ポーズなし。

## 動機

### なぜ所有権モデルが必要か？

| 言語 | メモリ管理 | 問題点 |
|------|----------|------|
| C/C++ | 手動管理 | メモリリーク、野ポインタ、二重解放 |
| Java/Python | GC | 遅延変動、メモリオーバーヘッド、予測不能なポーズ |
| Rust | 所有権 + 借用チェッカー | ライフタイム `'a` の学習曲線が急 |
| **YaoXiang** | **Move + Token + ref** | **シンプル、決定論的、GCなし** |

### 設計目標

```yaoxiang
# 1. デフォルト Move（零コピー）
p = Point(1.0, 2.0)
p2 = p                         # Move、p はこれ以上読めない

# 2. &T / &mut T 借用トークン（零オーバーヘッド、型属性が自然に権限を推論）
print_info(p2)                 # コンパイラが自動的に &Point トークンを作成し、 uży完毕后即释放
shift(p2, 1.0, 1.0)           # コンパイラが自動的に &mut Point トークンを作成し、使用完毕后即释放

# 3. ref = 共有（コンパイラが自動的に Rc/Arc を選択）
shared = ref p2                # スコープを越えて保持
spawn { use(shared) }          # コンパイラ判断：タスクを越える → Arc

# 4. clone() = 明示的コピー
backup = p2.clone()            # 深コピー、独自所有

# 5. unsafe + *T = システム級
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rust とのコアな違い

| 特性 | Rust | YaoXiang |
|------|------|----------|
| デフォルトセマンティクス | 借用 `&T`（明示的 `.clone()` が必要） | **Move（値渡し、零コピー）** |
| 借用 | `&T`/`&mut T`、戻り可能、ライフタイム必要 | **`&T`/`&mut T` 零サイズトークン、Copy/Linear 型属性が自然に権限を推論** |
| 共有機構 | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが自動的に Rc/Arc を選択）** |
| コピー | `clone()` | `clone()` |
| 裸ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用チェック | 大域的推論 | **型チェッカーのフロースensitive 活性分析（トークン状態追跡）** |
| 循環参照 | 手動 Weak | **タスク終了時の統一解放 / タスク間 lint / 標準ライブラリ Weak** |

## 提案

### 1. Move（デフォルト所有権移動）

```yaoxiang
# 規則：代入 / 引数渡 / 戻り値 = Move、零コピー

p: Point = Point(1.0, 2.0)
p2 = p                           # Move、p はこれ以上読めない

# 変数は再代入可能（Python スタイル、シャドウイングなし）
p = Point(3.0, 4.0)              # p の再束縛、型は一致する必要がある

# 関数引数：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move 戻り値
}

# 関数戻り値：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move 戻り値、零コピー
}
```

**特徴**：
- 零コピー（コンパイラがポインタを移動）
- 移動後、元束縛は読めない（コンパイルエラー）
- RAII：スコープ終了時に自動解放
- 関数シグネチャ `(T) -> T` 自体がドキュメント——T を消費し、T を返す

### 2. &T / &mut T（借用トークン）

**コア原則：`&T` と `&mut T` は零サイズのコンパイル時トークン型です。它们不是"参照"而是"アクセス権限の型レベル証明"です。**

#### 2.1 2つの型属性

```
&T      →  零サイズ、コピー可能（Copy）、読み取り専用権限を付与
&mut T  →  零サイズ、線形（NonCopy）、排他的読み書き権限を付与
```

**これは記憶すべき"規則"ではなく——型システムの基本的な属性です。** Copy 型は自由にコピー可能（複数の `&T` が共存可能）、線形型はコピー不可（`&mut T` は本質的に一意）。"借用チェッカー"はなく——型チェッカーがいつものようにそれだけを行っています。

#### 2.2 基本使用

```yaoxiang
# メソッド側：引数型を宣言し、必要な権限を決定
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # &Point トークンが読み取り権限を付与
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # &mut Point トークンが書き込み権限を付与
    self.y = self.y + dy
}

# 呼び出し側：コンパイラが自動的に借用または Move を選択
p = Point(1.0, 2.0)
p.print()                          # コンパイラが自動的に &Point トークンを作成
p.shift(1.0, 1.0)                  # コンパイラが自動的に &mut Point トークンを作成
p.print()                          # OK、前のトークンは shift 呼び出し終了時に解放済み

# 自由関数も同じ
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 2つの &Point トークンが共存——Copy 型
}
d = distance(p, p2)
```

#### 2.3 なぜ「禁止エスケープ」が不要か

RFC-009 v8 の `&T`/`&mut T` には3つの特殊規則——引数のみ可能、戻り値不可、構造体格納不可——が適用されていました。これは「借用」概念へのパッチ当てです。

トークンシステムではこれらの規則は不要です。トークンは** обычный тип**であり、他のすべての型と同じスコープ規則に従います。

**参照を返す——自然にサポート**：

```yaoxiang
# ✅ トークンは戻り値と一緒に伝播
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # サブトークンと親トークンが一緒に返される
}

# 使用例
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # トークンが呼び出し元に返される
print(px_ref)               # OK、トークンはまだスコープ内
```

**構造体に格納——自然にサポート**：

```yaoxiang
# ✅ 構造体がトークンをフィールドとして携带
Window: Type = {
    target: Point,
    view: &Point,      # トークンフィールド——target への読み取り専用ビューを保持
}

# view のトークンは target から派生し、Window は両方の所有権を保持
# Window が存在する限り、view トークンは有効
```

**クロージャキャプチャ——自然にサポート**：

```yaoxiang
# ✅ クロージャがトークンをキャプチャ，就像キャプチャ any 値一样
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    # クロージャが threshold の &Float トークンをキャプチャ（Copy 型、自由にクロージャにコピー可能）
    items.filter(|p| p.x > threshold)
}

# これは RFC-009 v8 では不可能——v8 はクロージャキャプチャを禁止
```

**タスクを越える——トークンはスレッドをまたげない**：

```yaoxiang
# ❌ トークンはタスク境界を越えられない
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ コンパイルエラー：トークン型が Send trait を実装していない
}

# これは特殊規則ではない——&T トークン型が Send trait を実装していない
# タスク間共有が必要な場合は ref を使用
```

**トークンは ref できない**：

```yaoxiang
# ❌ トークンは権限の証明であり、所有権ではない
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ コンパイルエラー：&T は所有可能的型ではない
}
```

#### 2.4 トークンのライフタイム

トークンのライフタイムは**通常のスコープ規則**によって決定され、ライフタイムパラメータは不要：

- 関数引数内のトークン：呼び出し期間中に存活、呼び出し終了時に解放
- 返されるトークン：所有権が呼び出し側に移動
- 構造体に格納されたトークン：構造体と一緒に存活
- クロージャにキャプチャされたトークン：クロージャと一緒に存活

コンパイラは `'a` 注釈を必要としない、なぜならトークンは**値**であり、値のライフタイムは所有権システム（Move/RAII）で統一管理されるから。**借用問題を所有権問題に帰着させる。**

#### 2.5 凍結メカニズム

`&mut T` トークンは一時的に"冻结"して `&T` トークンを生成できます：

```yaoxiang
modify_and_read: (p: &mut Point) -> Void = {
    p.x = 10.0                      # &mut Point を使用して変更
    
    # &mut を凍結し、読み取り専用ビューを取得
    view: &Point = freeze(p)         # p はここで凍結される
    print(view.x)                   # &Point を通じて読み取り
    print(view.y)
    # view がスコープを離れる、凍結解除
    
    p.y = 20.0                      # &mut Point が利用可能に回復
}
```

`freeze` のセマンティクス：
- `&mut T` を受け取り、`&T` を返す
- `&T` が存活している間、元 `&mut T` は利用不可
- `&T` がスコープを離れた後、`&mut T` が自動的に回復
- これは**フロースensitive 分析**——コンパイラが関数ボディ内でトークン状態を追跡

#### 2.6 トークン衝突検出

RFC-009 v8 の「交差借用チェック」の代替。原理はよりシンプル——トークン値に対する**フロースensitive 活性分析**：

```yaoxiang
# ❌ &mut と派生の &T は同時にアクティブになれない
bad_alias: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p が凍結される
    p.x = 10.0                      # ❌ コンパイルエラー：WriteToken が凍結状態にある
    print(view.x)                   
}

# ✅ 凍結解除後に &mut を 계속 使用可能
good_seq: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p が凍結される
    print(view.x)                   # &T を使用
    # view がスコープを離れる
    p.x = 10.0                      # ✅ WriteToken が回復済み
}
```

**検出方式**：これは専用の「借用チェッカー」ではなく——トークン値に対する**フロースensitive 活性分析**です。コンパイラが関数ボディ内で各トークンの状態（アクティブ/凍結/移動済み）を追跡し、任意の線形型値の追跡方法と同じです。

#### 2.7 コンパイラ内部：ブランド機構

ユーザーはブランドに一切触れる必要はありません。コンパイラは内部的に各トークンにコンパイル時一意の識別子を割り当てます：

```
ユーザーが見る         コンパイラの内部表現
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N はコンパイル時一意の整数
&mut Point     →  WriteToken(Point, #M)   // #M はコンパイル時一意の整数
```

ブランドの用途：
- **偽造防止**：トークンは所有者胶囊または freeze 操作からのみ取得でき凭空構築できない
- **関連追跡**：`&Point` から `&Float`（フィールドアクセス）を派生시킬 때、`&Float` は派生ブランド（`#N.field_x`）を携带し、コンパイラは親トークンを追跡可能
- **衝突検出**：同一來源の `WriteToken` と派生の `ReadToken` は同時にアクティブになれない

ブランドは単態化とインライン後に完全に消失し生成される 기계語中には存在しない。**零 런타임オーバーヘッド。**

#### 2.8 自動借用選択規則

呼び出し側コンパイラは以下の優先順位で自動選択：

```
1. 実引数がその後使用される場合 → トークン作成を優先（&T または &mut T、メソッドシグネチャに基づく）
2. 実引数がその後使用されない場合 → Move
3. 優先マッチング順：&T < &mut T < Move
```

```yaoxiang
# 例：自動選択
p = Point(1.0, 2.0)
p.print()        # print が &self を宣言 → コンパイラが &Point トークンを作成
p.shift(1.0, 1.0) # shift が &mut self を宣言 → コンパイラが &mut Point トークンを作成
p2 = p           # Move、p はこれ以上使用されない
```

#### 2.9 RFC-009 v8 丐版借用との比較

| 特性 | 丐版借用 (v8) | 借用トークン (v9) |
|------|--------------|--------------|
| 参照を返す | ❌ ハードコードで禁止 | ✅ トークンが戻り値と一緒に伝播 |
| 構造体に格納 | ❌ ハードコードで禁止 | ✅ トークンが構造体フィールドとして |
| クロージャキャプチャ | ❌ ハードコードで禁止 | ✅ クロージャがトークン値をキャプチャ |
| 特殊規則 | 3条（引数のみ/戻り値不可/格納不可） | 0条——型属性が自然に推論 |
| 借用チェック | 専用の交差借用チェック | 型チェッカーのフロースensitive 活性分析 |
| ライフタイム注釈 | 不要 | 不要 |
|  런타임オーバーヘッド | 零 | 零（零サイズ型、コンパイル後に消失） |
| エラーメッセージ | "借用はエスケープできない" | "WriteToken(#3) は凍結状態です"（通常の型エラー） |
| ユーザー心智モデル | "借用"の特殊地位を理解する必要がある | `&T` はコピー可能、`&mut T` はコピー不可 |

### 3. ref キーワード（コンパイラ自動最適化）

`ref` はスコープを越えて共有する唯一の方法。基盤は Rc か Arc か、ユーザーは気にする必要はない。

#### 3.1 基本使用

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共有、コンパイラが自動的に実装を選択

# タスクを越えた共有
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # コンパイラ判断：タスクを越える → Arc
    spawn { use(data) }           # コンパイラ判断：タスクを越える → Arc
}

# 単一タスク内の共有
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # コンパイラ判断：タスクを越えない → Rc
}
```

**ユーザー心智モデル**：`ref` = 共有持有。それでいい。

#### 3.2 コンパイラエスケープ分析：Rc vs Arc

```
ref のデータフロー分析：

他のタスクにエスケープしない → Rc（非原子参照カウント、低オーバーヘッド）
他のタスクにエスケープする   → Arc（原子参照カウント、スレッドセーフ）
```

#### 3.3 サイクル検出戦略

```
タスク内サイクル → 黙って許可。
  ├── 構造化並行によりタスク終了時にすべてのリソースが統一解放されることが保証される。
  ├── ref は常に生存し、セマンティクスは薄められない。
  └── ユーザーはタスク内で双方向の強参照を構築する権利を有する（例：グラフ計算の中間状態）。

タスク間サイクル → lint（デフォルト warn、設定可能）。
  ├── プログラム動作は正しい、本当にリークはしない（親タスク終了時に子タスクのリソースはすべて解放される）。
  ├── しかしタスク間の強参照は所有権境界が曖昧であることを意味し、再考に立ち止まる価値がある。
  ├── デフォルト warn レベル、コンパイルは成功するがヒントがある。
  └── チームはプロジェクト設定で deny に設定でき、CI 品質ゲートに組み込める。
```

**Lint レベル**（Rust clippy に類似）：

| レベル | 動作 | シナリオ |
|------|------|------|
| `allow` | チェックなし | 個人プロジェクト |
| `warn`（デフォルト） | コンパイル成功、ヒントあり | 開発段階 |
| `deny` | コンパイル失敗 | チーム CI 品質ゲート |
| `forbid` | コンパイル失敗、上書き不可 | 組織レベル強制規則 |

```yaoxiang
# タスク内サイクル：黙って許可、双方向強参照
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # サイクル。タスク終了時に統一解放。
}

# タスク間サイクル：lint（デフォルト warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: タスク間循環参照
    }
}
```

**プロジェクト設定例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # タスク間サイクルは CI 上で直接拒否
```

| サイクル型 | 動作 | 理由 |
|--------|------|------|
| タスク内 ref サイクル | チェックなし | ユーザーの権利、タスク終了時に統一解放 |
| タスク間 ref サイクル | lint（デフォルト warn） | 再考のヒント、deny に設定可能 |

#### 3.4 Weak：標準ライブラリ提供

```yaoxiang
use std.rc.Weak

# 上級ユーザーが明示的に選択
a.next = ref b
b.prev = Weak.new(a.next)        # ユーザーが明示的にどの方向を弱にするかを制御
```

**`Weak` は言語組み込みではなく、标准ライブラリ型です。**日常的には `ref` で十分。メモリを精密に制御したい上級ユーザーは手動で `Weak` を导入。

#### 3.5 借用トークン vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 何をするか | 一目見る/その場で改 | 共有持有 |
| 範囲 | トークン値のスコープに従う | スコープを越える |
| コスト | 零オーバーヘッド（零サイズ型） | Rc または Arc（コンパイラ選択） |
| エスケープ | 可能（トークンが戻り値/構造体/クロージャと一緒に伝播） | 元々エスケープするためのもの |
| タスクを越える | 不可（トークンは Send を実装していない） | 可能（コンパイラが自動的に Arc を選択） |
| サイクル | 無関係 | タスク内默許、タスク間 lint |

---

### 4. clone() —— 明示的コピー

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # 深コピー
# p と p2 は独立、相互に影響しない
```

**いつ使用するか**：元の値を保持する必要があり、Move にも共有にも適さない場合。

### 5. unsafe + 裸ポインタ（システム級プログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 裸ポインタ
    (*ptr).x = 0.0                # 逆参照（ユーザーが安全を保証）
    ptr2 = ptr + 1                # ポインタ演算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ユーザーはダングリング、解放後使用を保証
- FFI、メモリ操作などシステム級プログラミングに使用

---

### 6. 所有権グラデーション総覧

```
  借用トークン（零オーバーヘッド）     Move（零オーバーヘッド）      共有（必要時のみコスト）    コピー
   │                      │                  │                │
  &T コピー可能トークン        デフォルト所有権移動     ref Rc/Arc       clone()
  &mut T 線形トークン      チェーン的消费回流       コンパイラ自動選択      明示的深コピー
   │                      │                  │                │
  トークン値スコープ           スコープ内           スコープを越える         いつでも
  戻り値/構造体格納可能        T -> T 回流        ref タスク越え → Arc  独立コピー
  クロージャにキャプチャ可能     T -> Void 消費     ref タスク越えぬ → Rc
  零サイズコンパイル後に消失                          タスク内サイクル默許
                                            タスク間サイクル lint
                                            標準ライブラリ Weak で脱出
```

## 総合例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：読み取り専用トークン
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：可变トークン
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消費回流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 拿走、改、返す
    }

    # 参照を返す：トークンが戻り値と一緒に伝播
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# クロージャがトークンをキャプチャ（v8 では不可能な能力）
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}

# 総合使用
p = Point(1.0, 2.0)
p.print()                           # &Point トークン
p.shift(1.0, 1.0)                   # &mut Point トークン
p = p.scale(2.0)                    # Move → 回流
shared = ref p                      # ref 共有
spawn { use(shared) }

# clone 独立コピー
backup = p.clone()

# タスク内サイクル：默許
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # サイクル、タスク終了時に統一解放

# unsafe システム級
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Send / Sync

| 型 | Send | Sync | 説明 |
|------|------|------|------|
| 値型 | ✅ | ✅ | Int, Float, Point... |
| `ref T` | ✅ | ✅ | コンパイラが自動的に Rc/Arc を選択 |
| `&T` / `&mut T` | ❌ | ❌ | トークンは Send/Sync を実装していない、タスクを越えられない |
| `*T` | ❌ | ❌ | 裸ポインタ、シングルスレッド |

## パフォーマンス分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | 零 | ポインタ移動 |
| `&T` / `&mut T` | 零 | 零サイズ型、コンパイル後に消失、零 런타임オーバーヘッド |
| `ref`（タスクを越えない）| 低 | Rc にコンパイル、非原子操作 |
| `ref`（タスクを越える）| 中 | Arc にコンパイル、原子操作 |
| `clone()` | 型による | 小オブジェクトは高速、大オブジェクトは低速 |
| `unsafe + *T` | 零 | 直接メモリ操作 |

### 比較

| 言語 | 共有機構 | メモリ管理 | サイクル処理 | 複雑度 |
|------|----------|----------|----------|--------|
| Rust | Arc / Mutex + 借用チェッカー | コンパイル時チェック | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 借用トークン** | **RAII** | **タスク境界解放 / タスク間 lint / 標準ライブラリ Weak** | **低** |

## トレードオフ

### 利点

1. **統一**：`&T`/`&mut T` は обычный 型であり、特殊言語機能ではない。RFC-010 の `name: type = value` と完全に一貫
2. **シンプル**：ライフタイムなし、大域的借用チェッカーなし。`&T` はコピー可能、`&mut T` はコピー不可——2つの型属性
3. **強力**：参照を返す、構造体に格納する、クロージャにキャプチャ——Rust と同等の表現力
4. **コンパイラが知的**：ref が自動的に Rc/Arc を選択、呼び出し側が自動的に借用を選択
5. **決定論的**：ref は常に生存し、静かに弱参照にならない
6. **高性能**：Move 零コピー、トークン零オーバーヘッド（零サイズ型、コンパイル後に消失）
7. **柔軟**：`unsafe + *T` がシステム級プログラミングをサポート

### 欠点

1. **泛型 ブランドパラメータ伝染**：トークンがブランド識別子を携带し、参照を返す関数のシグネチャに追加の泛型パラメータが反映される
2. **ref 런타임オーバーヘッド**：原子操作にコストがある（しかしこれは共有の必然的な代償）
3. **unsafe リスク**：ユーザーは正確性を保証する必要がある
4. **タスク間サイクルは lint 而不是コンパイルエラー**：Rust のようにコンパイルエラーにならず、デフォルトは warn、deny を設定しないと品質ゲートにならない

## 代替案

| 案 | なぜ選択しないか |
|------|--------------|
| GC | 런타임オーバーヘッドがあり、予測不能なポーズがある |
| Rust 借用チェッカー | ライフタイム `'a` が必要で、学習曲線が急 |
| 純 Move | 並行共有を処理できない |
| 裸ポインタなし | システム級プログラミングができない |
| ユーザーに Rc/Arc を露出 | 実装詳細をユーザーに押し付け、認知負担を増加 |
| 丐版借用（v8） | エスケープ禁止の戦略によりクロージャキャプチャ、参照を返すなど重要な表現力が犠牲になる |

### トークン衝突検出：フロースensitive 活性分析

**分析範囲**：関数ボディ内のみ。トークン値に対してフロースensitive 活性分析を行い、各トークンの状態（アクティブ/凍結/移動済み）を追跡。

**レベル 1：呼び出し点チェック**——同一の実引数が同時に `&mut` トークンと他のトークンを生成できない：

```yaoxiang
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)    # ❌ p が同時に &mut と & トークンを派生、コンパイラが拒否
```

**レベル 2：関数ボディ フロースensitive**——`&mut` トークンが呼び出しに渡された後、呼び出しが返ればトークンは解放され、以後再生成可能：

```yaoxiang
process_twice: (p: &mut Point) -> Void = {
    shift(p, 1.0, 1.0)    # &mut トークンを shift に渡し、shift が返ればトークンは解放
    print_info(p)          # &Point トークンを再生成、衝突なし
}
```

**レベル 3：冻结状態追跡**——`freeze` が生成した `&T` トークンがアクティブである間、元 `&mut` トークンは利用不可：

```yaoxiang
frozen: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)    # p が冻结状態に入る
    print(view.x)
    p.x = 10.0                  # ❌ コンパイルエラー：WriteToken が冻结状態にある
}
```

**必要ないもの**：関数間ライフタイム追跡、大域的エイリアス分析、借用グラフ制約解決、NLL、`'a` 注釈。トークンは値であり、値の活性分析は型チェッカーが統一的に処理する——任意の線形型値の追跡方法と同じです。
