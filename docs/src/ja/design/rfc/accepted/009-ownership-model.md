# RFC-009: 所有権モデル設計

## 概要

本文書は YaoXiang プログラミング言語の**所有権モデル（Ownership Model）**を定義する。

**核心設計——五つの概念、一つのグラデーション**：

```
一瞥/その場変更    持ち去り         共有保持         複製する         システムレベル
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         ゼロコピー     コンパイラ自動   明示的ディープコピー  *T
  ゼロサイズ令牌   デフォルト     Rc/Arcを選択    ユーザー責任
  型属性自然
  推論権限
```

- **Move（デフォルト）**：代入/引数渡し/戻り値 = 所有権の移転、ゼロコピー、RAII による自動解放
- **`&T` / `&mut T`（借用令牌）**：ゼロサイズのコンパイル時令牌型。`&T` は複製可能（共有読み取り）、`&mut T` は線形（排他的可変）。権限は型属性から自然に推論され、特別な規則は不要。返却可能、構造体に保存可能。
- **`ref` キーワード**：スコープを越えて共有。コンパイラが自動的に Rc（タスクを跨がない）または Arc（タスクを跨ぐ）を選択
- **`clone()`**：明示的ディープコピー
- **`unsafe` + `*T`**：生ポインタ、システムレベルのエスケープハッチ

**除去された複雑性**：
- ❌ ライフタイム `'a` なし
- ❌ 独立した借用検査フレームワークなし（借用衝突はホーア命题に次元削減され、型検査と証明パイプラインを共有）
- ❌ GC なし
- ❌「エスケープ禁止」などの特別規則なし（令牌は通常の型、スコープは型システムで統一的に処理）
- ❌ ユーザーは Rc/Arc の違いを知る必要がない（コンパイラが自動選択）

> **プログラミング負担**：`&T` は複製可能、`&mut T` は複製不可——二つの型属性、ゼロの特別規則、コンパイラ完全自動。
> **性能保証**：Move ゼロオーバーヘッド、令牌ゼロオーバーヘッド（ゼロサイズ型、コンパイル後消滅）、ref は必要に応じたコスト、GC ポーズなし。

## 動機

### なぜ所有権モデルが必要なのか？

| 言語 | メモリ管理 | 問題 |
|------|----------|------|
| C/C++ | 手動管理 | メモリリーク、ダングリングポインタ、二重解放 |
| Java/Python | GC | レイテンシ変動、メモリオーバーヘッド、予測不能なポーズ |
| Rust | 所有権 + 借用検査 | ライフタイム `'a` の学習曲線が急峻 |
| **YaoXiang** | **Move + Token + ref** | **シンプル、決定的、GC なし** |

### 設計目標

```yaoxiang
# 1. デフォルト Move（ゼロコピー）
p = Point(1.0, 2.0)
p2 = p                         # Move、p は再読込不可

# 2. &T / &mut T 借用令牌（ゼロオーバーヘッド、型属性が自然に権限を推論）
print_info(p2)                 # コンパイラが自動的に &Point 令牌を作成、使用後解放
shift(p2, 1.0, 1.0)           # コンパイラが自動的に &mut Point 令牌を作成

# 3. ref = 共有（コンパイラが自動的に Rc/Arc を選択）
shared = ref p2                # スコープを越えて保持
spawn { use(shared) }          # コンパイラ：タスクを跨ぐ → Arc

# 4. clone() = 明示的コピー
backup = p2.clone()            # ディープコピー、独立

# 5. unsafe + *T = システムレベル
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Rust との核心的な違い

| 特性 | Rust | YaoXiang |
|------|------|----------|
| デフォルトセマンティクス | 借用 `&T`（明示的に `.clone()` が必要） | **Move（値渡し、ゼロコピー）** |
| 借用 | `&T`/`&mut T`、返却可能、ライフタイムが必要 | **`&T`/`&mut T` ゼロサイズ令牌、Dup/Linear 型属性で自然に推論** |
| 共有メカニズム | `Arc::new()` + 手動 Weak | **`ref` キーワード（コンパイラが自動的に Rc/Arc を選択）** |
| 複製 | `clone()` | `clone()` |
| 生ポインタ | `*T` | `*T` |
| ライフタイム | `'a` | ❌ なし |
| 借用検査 | 大域的推論 | **型検査器が自動的に借用命题を生成し、証明パイプラインで統一検証** |
| 循環参照 | 手動 Weak | **タスク終了時統一解放 / タスク跨ぎ lint / 標準ライブラリ Weak** |

## 提案

### 1. Move（デフォルト所有権移転）

```yaoxiang
# 規則：代入 / 引数渡し / 戻り値 = Move、ゼロコピー

p: Point = Point(1.0, 2.0)
p2 = p                           # Move、p は再読込不可

# 変数は再代入可能（Python 風、シャドウイングなし）
p = Point(3.0, 4.0)              # p 再束縛、型は一致必須

# 関数引数：Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move で返す
}

# 関数戻り値：Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move で返す、ゼロコピー
}
```

**特徴**：
- ゼロコピー（コンパイラがポインタを移動）
- 移動後、元の束縛は読み取り不可（コンパイルエラー）
- RAII：スコープ終了時に自動解放
- 関数シグネチャ `(T) -> T` 自体がドキュメント——T を消費し、T を返す

---

### 2. &T / &mut T（借用令牌）

**核心原則：`&T` と `&mut T` はゼロサイズのコンパイル時令牌型である。これらは「参照」ではなく、「アクセス権限の型レベル証明」である。**

#### 2.1 二つの型属性

```
&T      →  ゼロサイズ、ソースデータを凍結（ReadToken 生存中は WriteToken 禁止）、
          凍結保証下で複数の読み取り専用ビューが安全 → 複製可能（Dup）
&mut T  →  ゼロサイズ、排他的読み書き（WriteToken 生存中は他の令牌すべて禁止）、
          排他的アクセス下では複製が無意味 → 線形（非 Dup）
```

**因果関係を逆転させてはならない：凍結が原因、Dup が結果。** `&T` が Dup を実装しているから共存できるのではない——データが凍結されている（突然変異可能性がない）からこそ、複数の読み取り専用ビューが安全であり、Dup が実現できる。もし Dup を定義とし、衝突検査を「追加のパッチ」と位置づければ、設計は誤りである。

#### 2.2 基本使用

```yaoxiang
# メソッド側：引数型を宣言し、必要な権限を決定
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # &Point 令牌が読み取り権限を付与
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # &mut Point 令牌が書き込み権限を付与
    self.y = self.y + dy
}

# 呼び出し側：コンパイラが自動的に借用または Move を選択
p = Point(1.0, 2.0)
p.print()                          # コンパイラが自動的に &Point 令牌を作成
p.shift(1.0, 1.0)                  # コンパイラが自動的に &mut Point 令牌を作成
p.print()                          # OK、前の令牌は shift 呼び出し終了時に解放済み

# 自由関数も同様
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # 二つの &Point 令牌が共存——Dup 型
}
d = distance(p, p2)
```

#### 2.3 なぜ「エスケープ禁止」は不要なのか

RFC-009 v8 は `&T`/`&mut T` に三つの特別規則を課した——引数としてのみ使用可能、返却不可、構造体に保存不可。これは「借用」概念にパッチを当てていた。

令牌システムにはこれらの規則は不要。令牌は**通常の型**であり、他のすべての型と同じスコープ規則に従う。

**参照の返却——自然にサポート**：

```yaoxiang
# ✅ 令牌は戻り値と共に伝播
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # 子令牌と親令牌が共に返される
}

# 使用
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # 令牌が呼び出し元に返却される
print(px_ref)               # OK、令牌はまだスコープ内
```

**構造体への保存——自然にサポート**：

```yaoxiang
# ✅ 構造体が令牌をフィールドとして保持
Window: Type = {
    target: Point,
    view: &Point,      # 令牌フィールド——target への読み取り専用ビューを保持
}

# view の令牌は target から派生、Window が両方の所有権を保持
# Window が存在する限り、view 令牌は有効
```

#### 2.3 クロージャと Lambda の明示的引数

Lambda は関数値——返却、保存、現在のスコープ外への持ち出しが可能。したがって Lambda は**外側のローカル変数を暗黙的にキャプチャしない**。外側のデータが必要な場合は、明示的引数として渡す：

```yaoxiang
# ✅ Lambda は明示的引数を使用
double: (x: Int) -> Int = (x) => x * 2
filter_by: (items: List(Int), f: (Int) -> Bool) -> List(Int) = { ... }

# ✅ spawn { } はこの規則の影響を受けない——spawn は即座に実行される並行ブロックで、親タスクはブロック完了を待つ
shared = ref data
spawn { use(shared) }

# ❌ Lambda は外側の変数を暗黙的にキャプチャできない
x = 42
f = () => { x + 1 }  # コンパイルエラー：x はスコープ外

# ✅ 正しい方法：明示的引数渡し
f = (x) => { x + 1 }
f(x)
```

**spawn { } は関数値ではない。** spawn でマークされたブロックは if/while body と同様——即座に実行され、親スタックフレーム生存中に完了する。spawn body は外側の変数に正常にアクセスできる。

**タスクを跨ぐ——令牌はスレッドを通過できない**：

```yaoxiang
# ❌ 令牌はタスク境界を跨げない
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ コンパイルエラー：令牌はタスクを跨いで渡せない
}

# これは特別規則ではない——令牌はコンパイル時権限証明であり、タスクを跨ぐ共有は ref を使用すること
# タスクを跨ぐ共有が必要な場合は、ref を使用すること
```

**令牌は ref 不可**：

```yaoxiang
# ❌ 令牌は権限証明であり、所有権ではない
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ コンパイルエラー：&T は所有可能な型ではない
}
```

#### 2.4 令牌のライフタイム

令牌のライフタイムは**通常のスコープ規則**によって決定され、ライフタイムパラメータは不要：

- 関数引数内の令牌：呼び出し期間中生存、呼び出し終了後に解放
- 返却された令牌：所有権を呼び出し元に移転
- 構造体に保存された令牌：構造体と共に生存

コンパイラは `'a` 注釈を必要としない、なぜなら令牌は**値**であり、値のライフタイムは所有権システム（Move/RAII）によって統一管理されるから。**借用問題を所有権問題に次元削減する。**

#### 2.5 令牌衝突検出

令牌衝突検出は**ホーア論理命题**であり、独立したフロー依存解析ではない。

```
{衝突する ReadToken 全て死亡} write(data) {WriteToken 安全取得}
```

型検査やユーザー述語検証と RFC-027 の証明パイプラインを共有する。コンパイラが自動的に借用命题（`borrow_conflict`、`use_after_move`、`use_after_drop`、`mut_violation`）を生成し、パイプラインに送って検証する。パイプラインは Proved / Disproved / Unproven を返す。

```yaoxiang
# ❌ &mut 令牌は線形であり、複製できない
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move、p は再読込不可
    p.x = 10.0                      # ❌ コンパイルエラー：WriteToken は既に移動済み
}

# ✅ &T 令牌は Dup 型であり、自由に複製可能
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK、&T は Dup 型
    print(p.x)                      # OK
    print(p2.x)                     # OK、二つの読み取り専用令牌が共存
}
```

**借用検査は消えていない——次元削減されたのだ。** 既存の `BorrowChecker` は `BorrowPredicateEmitter`（命题生成器）となり、生成された借用命题は他の型命题と同一条証明パイプラインを共有する。これは型検査器の概念と完全に並行する：型検査器が型等価命题を生成し、借用命题生成器が借用命题を生成し、同一条のパイプラインで検証する。詳細は [RFC-009a](../accepted/009a-borrow-proof-pipeline.md) を参照。

#### 2.7 コンパイラ内部：ブランド機構

ユーザーはブランドに決して触れない。コンパイラが内部で各令牌にコンパイル時一意識別子を割り当てる：

```
ユーザーが見る         コンパイラ内部表現
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N はコンパイル時一意整数
&mut Point     →  WriteToken(Point, #M)   // #M はコンパイル時一意整数
```

ブランドの用途：
- **偽造防止**：令牌は所有者カプセルからのみ取得でき、凭空には構築できない
- **関連追跡**：`&Point` から `&Float`（フィールドアクセス）を派生するとき、`&Float` は派生ブランド（`#N.field_x`）を携带し、コンパイラは親令牌まで追跡可能
- **衝突検出**：同源 `WriteToken` と派生 `ReadToken` は同時にアクティブ化できない

ブランドは単態化とインライン化後完全に消滅し、生成されたマシンコードには存在しない。**ランタイムオーバーヘッドゼロ。**

#### 2.8 自動借用選択規則

呼び出し側コンパイラは以下の優先順位で自動選択する：

```
1. 実引数が後続も使用される → 令牌を優先作成（メソッドシグネチャに応じて &T または &mut T）
2. 実引数が後続使用されない → Move
3. 優先マッチング順序：&T < &mut T < Move
```

```yaoxiang
# 例：自動選択
p = Point(1.0, 2.0)
p.print()        # print は &self を宣言 → コンパイラが &Point 令牌を作成
p.shift(1.0, 1.0) # shift は &mut self を宣言 → コンパイラが &mut Point 令牌を作成
p2 = p           # Move、p はもう使用されない
```

#### 2.9 RFC-009 v8 簡素化借用との比較

| 特性 | 簡素化借用 (v8) | 借用令牌 (v9) |
|------|--------------|--------------|
| 参照の返却 | ❌ ハードコード禁止 | ✅ 令牌が戻り値と共に伝播 |
| 構造体への保存 | ❌ ハードコード禁止 | ✅ 令牌が構造体フィールド |
| Lambda 明示的引数 | ❌ ハードコード禁止 | ✅ Lambda は明示的引数を使用 |
| 特別規則 | 3 条（引数のみ/返却不可/保存不可） | 0 条——型属性が自然に推論 |
| 借用検査 | 専用交差借用検査 | 型検査器のフロー依存活性解析 |
| ライフタイム注釈 | 不要 | 不要 |
| ランタイムオーバーヘッド | ゼロ | ゼロ（ゼロサイズ型、コンパイル後消滅） |
| エラーメッセージ | 「借用はエスケープ不可」 | 「WriteToken(#3) は既に移動済み」（通常の型エラー） |
| ユーザーメンタルモデル | 「借用」の特別な地位を理解 | `&T` は複製可能、`&mut T` は複製不可 |

### 3. ref キーワード（コンパイラ自動最適化）

`ref` はスコープを越えて共有する唯一の方法。下層が Rc か Arc かをユーザーが知る必要はない。

#### 3.1 基本使用

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # 共有、コンパイラが自動的に実装を選択

# タスクを跨ぐ共有
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # コンパイラ：タスクを跨ぐ → Arc
    spawn { use(data) }           # コンパイラ：タスクを跨ぐ → Arc
}

# 単一タスク共有
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # コンパイラ：タスクを跨がない → Rc
}
```

**ユーザーメンタルモデル**：`ref` = 共有保持。それだけ。

#### 3.2 コンパイラエスケープ解析：Rc vs Arc

```
ref のデータフロー解析：

他のタスクにエスケープしない → Rc（非アトミック参照カウント、低オーバーヘッド）
他のタスクにエスケープする   → Arc（アトミック参照カウント、スレッドセーフ）
```

#### 3.3 環検出戦略

```
タスク内環 → 静的に許可。
  ├── 各タスクには明確なライフサイクル境界がある——タスク終了時すべてのリソース（ref 環を含む）が統一解放される。
  ├── 長時間実行されるサービスはリクエスト/接続ごとに子タスクを作成すべき——子タスク終了時に自動回収され、累積リークは発生しない。
  ├── ref は常に生存し、セマンティクスは混じりけがない。
  └── ユーザーはタスク内で双方向強参照（例如グラフ計算中間状態）を構築する権利を持つ。

タスク跨ぎ環 → lint（デフォルト warn、設定可能）。
  ├── プログラムの挙動は正しく、実際にリークはしない（親タスク終了時に子タスクのリソースは全解放）。
  ├── ただしタスクを跨ぐ強参照は所有権境界が曖昧であることを意味し、立ち止まって再考する価値がある。
  ├── デフォルト warn レベル、コンパイルは成功するがヒントを表示。
  └── チーム設定で deny にでき、CI 品質ゲートに組み込める。
```

**Lint レベル**（Rust clippy に類似）：

| レベル | 挙動 | シナリオ |
|------|------|------|
| `allow` | 検査しない | 個人プロジェクト |
| `warn`（デフォルト） | コンパイル成功、ヒント表示 | 開発段階 |
| `deny` | コンパイル失敗 | チーム CI 品質ゲート |
| `forbid` | コンパイル失敗、上書き不可 | 組織レベルの強制規則 |

```yaoxiang
# タスク内環：静的に許可、双方向強参照
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # 環。タスク終了時に統一解放。
}

# タスク跨ぎ環：lint（デフォルト warn）
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: タスク跨ぎ循環参照
    }
}
```

**プロジェクト設定例**：

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # タスク跨ぎ環は CI で直接拒否
```

| 環タイプ | 挙動 | 理由 |
|--------|------|------|
| タスク内 ref 環 | 検査しない | ユーザーの権限、タスク終了時に統一解放 |
| タスク跨ぎ ref 環 | lint（デフォルト warn） | 再考を促す、設定で deny 可能 |

#### 3.4 Weak：標準ライブラリ提供

```yaoxiang
use std.rc.Weak

# 上級ユーザーの明示的選択
a.next = ref b
b.prev = Weak.new(a.next)        # ユーザーが明示的にどの方向を弱くするかを制御
```

**`Weak` は言語組み込みではなく、標準ライブラリ型である。** 日常的には `ref` で十分。メモリを細かく制御したい上級ユーザーが手動で `Weak` を導入する。

#### 3.5 借用令牌 vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| 機能 | 一瞥 / その場変更 | 共有保持 |
| 範囲 | 令牌値スコープに従う | スコープを跨ぐ |
| コスト | ゼロオーバーヘッド（ゼロサイズ型） | Rc または Arc（コンパイラ選択） |
| エスケープ | 可（令牌は戻り値/構造体/クロージャと共に伝播） | 元々エスケープ用 |
| タスク跨ぎ | 不可（令牌はコンパイル時権限証明、タスクを跨いで渡せない） | 可（コンパイラが自動的に Arc を選択） |
| 環形成 | 関与しない | タスク内は静的許可、タスク跨ぎは lint |

### 4. clone() —— 明示的コピー

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # ディープコピー
# p と p2 は独立し、相互に影響しない
```

**使用時期**：元の値を保持する必要があり、Move や共有が適さないシナリオ。

### 5. unsafe + 生ポインタ（システムレベルプログラミング）

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # 生ポインタ
    (*ptr).x = 0.0                # 逆参照（ユーザーが安全性を保証）
    ptr2 = ptr + 1                # ポインタ演算
}
```

**制限**：
- `unsafe` ブロック内でのみ使用可能
- ユーザーはダングリング、解放後使用がないことを保証する
- FFI、メモリ操作などのシステムレベルプログラミングに使用

### 6. 所有権グラデーション総覧

```
  借用令牌（ゼロオーバーヘッド）  Move（ゼロオーバーヘッド）  共有（必要時コスト）   コピー
   │                      │                  │                │
  &T 複製可能令牌        デフォルト所有権移転  ref Rc/Arc       clone()
  &mut T 線形令牌        チェーン消費回流     コンパイラ自動選択  明示的ディープコピー
   │                      │                  │                │
  令牌値スコープ           スコープ内          スコープを跨ぐ     いつでも
  返却/構造体保存可能       T -> T 回流        ref タスク跨ぎ → Arc  独立コピー
  ゼロサイズ、コンパイル後消滅 T -> Void 消費     ref タスク跨ぎなし → Rc
  ゼロサイズ、コンパイル後消滅                   タスク内環は静的許可
                                            タスク跨ぎ環は lint
                                            標準ライブラリ Weak でエスケープ
```

## 総合例

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T：読み取り専用令牌
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T：可変令牌
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move：消費回流
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # 拿走、改、還你
    }

    # 参照を返す：令牌は戻り値と共に伝播
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# Lambda 明示的引数
double: (x: Int) -> Int = (x) => x * 2

# 総合使用
p = Point(1.0, 2.0)
p.print()                           # &Point 令牌
p.shift(1.0, 1.0)                   # &mut Point 令牌
p = p.scale(2.0)                    # Move → 回流
shared = ref p                      # ref 共有
spawn { use(shared) }

# clone 独立コピー
backup = p.clone()

# タスク内環：静的に許可
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # 環、タスク終了時に統一解放

# unsafe システムレベル
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## 型システム制約

### Dup 型属性

`Dup`（Duplicable）はコンパイラが自動管理する型属性であり、意味は**シャローコピー**である：代入/引数渡し時にコピーされるのはハンドル/令牌であり、下層データは共有される。これは Move（所有権移転）および Clone（明示的ディープコピー、独立コピー作成）とともに三段階グラデーションを形成する。

**Dup と Clone は直交する概念である**——Dup はハンドルをコピーしてデータを共有し、Clone は独立コピー作成する。一つの型が Dup と Clone の両方をサポートすることも、 하나만 지원할 수도 있다。

| 型 | Dup | Clone | 説明 |
|------|-----|-------|------|
| `&T` | ✅（令牌を複製、複数のビューが同一データを指す） | ✅ | 読み取り専用令牌 |
| `ref T` | ✅（参照カウント+1、ヒープデータを共有） | ✅ | 共有保持（コンパイラが自動的に Rc/Arc を選択） |
| String, Bytes | ✅（内部参照カウント、ハンドルをコピーして下層 buffer を共有） | ✅ | 文字列/バイト |
| `&mut T` | ❌（線形、排他的） | ❌ | 可変令牌 |
| `*T` | ❌ | ❌ | 生ポインタ |
| struct | 派生（すべてのフィールドが Dup のとき自動派生） | ✅ | 構造体 |

**原始値型**（Int, Float, Bool, Char）の代入挙動はコンパイラ組み込みの値コピーであり、二つの値は完全に独立しており、シャローコピーではない。これらは Dup 型属性には属さず、コンパイラのネイティブ処理である。

## 性能分析

| 操作 | コスト | 説明 |
|------|------|------|
| Move | ゼロ | ポインタ移動 |
| `&T` / `&mut T` | ゼロ | ゼロサイズ型、コンパイル後消滅、ランタイムオーバーヘッドゼロ |
| `ref`（タスク跨ぎなし）| 低 | Rc にコンパイル、非アトミック操作 |
| `ref`（タスク跨ぎ）| 中 | Arc にコンパイル、アトミック操作 |
| `clone()` | 型による | 小オブジェクトは高速、大オブジェクトは低速 |
| `unsafe + *T` | ゼロ | 直接メモリ操作 |

### 比較

| 言語 | 共有メカニズム | メモリ管理 | 循環処理 | 複雑度 |
|------|----------|----------|----------|------|
| Rust | Arc / Mutex + 借用検査 | コンパイル時検査 | 手動 Weak | 高 |
| Go | chan / pointer | GC | GC | 低 |
| C++ | shared_ptr | RAII | weak_ptr | 中 |
| **YaoXiang** | **ref + 借用令牌** | **RAII** | **タスク境界解放 / タスク跨ぎ lint / 標準ライブラリ Weak** | **低** |

## トレードオフ

### 利点

1. **統一**：`&T`/`&mut T` は通常の型であり、特別な言語特性ではない。RFC-010 の `name: type = value` と完全一致
2. **シンプル**：ライフタイムなし、借用検査は型システム命题に次元削減。`&T` は複製可能、`&mut T` は複製不可——二つの型属性
3. **強力**：参照返却、構造体保存、クロージャキャプチャが可能——表現能力は Rust と同レベル
4. **コンパイラスマート**：ref が自動的に Rc/Arc を選択、呼び出し側が自動的に借用を選択
5. **確定的**：ref は保持するだけで、密かに弱参照にはならない
6. **高性能**：Move ゼロコピー、令牌ゼロオーバーヘッド（ゼロサイズ型、コンパイル後消滅）
7. **柔軟**：`unsafe + *T` がシステムレベルプログラミングをサポート

### 欠点

1. **ジェネリックブランドパラメータの伝染**：令牌がブランド識別子を携带し、参照を返す関数シグネチャに追加のジェネリックパラメータが現れる
2. **ref ランタイムオーバーヘッド**：アトミック操作にコストがある（ただしこれは共有の本質的代償）
3. **unsafe のリスク**：ユーザーが正当性を保証しなければならない
4. **タスク跨ぎ環は lint でありコンパイルエラーではない**：Rust のようにコンパイルエラーとはならず、デフォルトは warn、チームが deny に設定して初めて品質ゲートとなる

## 代替案

| 案 | 選択しない理由 |
|------|--------------|
| GC | ランタイムオーバーヘッドあり、ポーズが予測不能 |
| Rust 借用検査器 | ライフタイム `'a` が必要、学習曲線が急峻 |
| 純粋 Move | 並行共有を処理できない |
| 生ポインタなし | システムレベルプログラミング不可 |
| Rc/Arc をユーザーに公開 | 実装詳細をユーザーに丸投げし、認知負担を増大 |
| 簡素化借用（v8） | エスケープ禁止戦略がクロージャキャプチャ、参照返却などの重要な表現能力を犠牲にする |

### 令牌衝突検出：ホーア論理命题

令牌衝突検出の完全な方案は [RFC-009a: 令牌ライフタイム分析——ホーア証明パイプラインに基づく](../accepted/009a-borrow-proof-pipeline.md) を参照。核心要点：

**令牌活性はホーア論理命题である。** `{衝突する ReadToken 全て死亡} write(data) {WriteToken 安全取得}`——型検査、ユーザー述語検証と RFC-027 の証明パイプラインを共有する。コンパイラが自動的に借用命题（`borrow_conflict`、`use_after_move`、`use_after_drop`、`mut_violation`）を生成し、パイプラインが Proved / Disproved / Unproven を返す。

**借用検査は消えていない——次元削減されたのだ。** `BorrowChecker` は `BorrowPredicateEmitter` となり、検査実行ではなく命题生成となる。これは「型検査器」の概念と完全に並行する：型検査器が型等価命题を生成し、借用命题生成器が借用命题を生成し、同一条のパイプラインで検証する。

**ブランド ID（`#42`）は `'a` そのものである。** 情報は完全一致、エンコーディングが異なる。`'a` は型シグネチャに可視、`#42` はコンパイラ内部。新しい解析を発明したのではない——ライフタイムを型層から証明層に降ろしたのだ。

**アルゴリズム概要**（詳細は RFC-009a）：
- ブランドツリー前置詞一致 → 衝突令牌の確定（O(depth)、深さ ≤ 3）
- 逆方向 BFS → 消費者から出発、break で逆辺を切断、構造解析で 95%+ のシナリオをカバー（高速経路）
- SMT 論理切断 → while + パス条件時のみ呼び出し（低速経路、極めて稀）
