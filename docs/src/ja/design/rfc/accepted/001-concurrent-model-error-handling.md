# RFC-001：並作モデルとエラー処理システム

> **ステータス**: 承認済み
> **著者**: 晨煦

## 概要

YaoXiang の並作モデルを提案する：同期構文でロジックを記述し、実行時に自動的で並列実行する。核心メカニズム：三層並行アーキテクチャ + DAG 依存解析 + Result 型システム。

## クイック選択

| シナリオ | 記述方法 | 説明 |
|------|------|------|
| 自動並列 | アノテーションなし（デフォルト） | 並列性を最大化 |
| 同期待機 | `@eager` | 依存関係の完了を待機 |
| 完全順序 | `@block` | 並発なし、デバッグ用 |
| 局所並発 | `spawn` | @block スコープ内の並発 |

## 動機

現在の主流言語の並発モデルには明显的な欠陥がある：

| 言語 | 並発モデル | 問題点 |
|------|----------|------|
| Rust | async/await + tokio | 非同期伝播、学習曲線が険しい |
| Go | goroutine | 型安全性がなし |
| Python | asyncio | GIL の制限 |
| JavaScript | Promise/async | コールバックが複雑 |

### 核心的矛盾

1. **透明性 vs 制御性**：完全透明だが制御不可 vs 完全制御可能だが不透明
2. **並発 vs デバッグ可能性**：並発プログラムはデバッグが難しい vs デバッグ可能なプログラムは並発が難しい

## 提案

### 1. 並作モデル：三層並行アーキテクチャ

> **説明**：L1/L2/L3 はメンタルモデルであり、ユーザーが異なるシナリオを理解するのに役立つ。実際の実装は一つのメカニズムのみ：DAG 自動分析 + アノテーション制御。

| レイヤー | メンタルモデル | 構文 | 実行方式 | 並列度 |
|------|----------|------|----------|--------|
| **L1** | 並発禁止 | `@block` | 完全順序実行 | ❌ なし |
| **L2** | @block 内の並発 | `spawn` | @block スコープ内の制御可能並発 | ⚠️ 一部 |
| **L3** | 完全並発 | デフォルト（アノテーションなし） | DAG の自動分析 | ✅ 完全 |

#### L1: @block 同期モード

```yaoxiang
main: () -> Void @block = {
    data1 = fetch_sync("api1")
    data2 = fetch_sync("api2")
    process(data1, data2)    # 厳密な順序、並発なし
}
```

#### L2: @block 内の制御可能並発

```yaoxiang
# spawn は @block 関数内部でのみ使用可能
main: () -> Void @block = {
    spawn { data1 = fetch_data("api1") }
    spawn { data2 = fetch_data("api2") }
    # すべての spawn の完了を待機（標準ライブラリ制御）
    process(data1, data2)
}
```

#### L3: 完全透明（デフォルト）

```yaoxiang
# アノテーション不要、コンパイラが自動的に DAG を分析
heavy_calc: (n: Int) -> Int = fibonacci(n)

auto_parallel: (n: Int) -> Int = {
    a = heavy_calc(1)    # 自動並列
    b = heavy_calc(2)    # 自動並列
    c = heavy_calc(3)    # 自動並列
    a + b + c            # 値が必要な時にすべての結果を待機
}
```

### 2. アノテーションの完全な比較

| 次元 | デフォルト（アノテーションなし） | `@eager` | `@block` | `spawn` |
|------|---------------|----------|----------|---------|
| **実行方式** | 自動 DAG 分析 | 依存関係を同期的に待機 | 完全順序 | @block 内の並発 |
| **並列度** | ✅ 完全 | ⚠️ 依存順序に従う | ❌ なし | ⚠️ 一部 |
| **DAG 構築** | ✅ | ✅ | ❌ | ✅ |

**選択ガイド**：
- 最大並列 → アノテーションなし（デフォルト）
- 順序付き副作用が必要 → `@eager`
- デバッグ/初心者/重要なコード → `@block`
- @block 内で並発が必要 → `spawn`

```yaoxiang
# デフォルト：並列性を最大化
calc_all: () -> Int = {
    a = heavy_calc(1)    # 自動並列
    b = heavy_calc(2)    # 自動並列
    a + b
}

# @eager：同期待機
calc_seq: () -> Int @eager = {
    a = heavy_calc(1)    # 同期実行
    b = heavy_calc(2)    # 同期実行
    a + b
}

# @block：完全順序
calc_simple: () -> Int @block = {
    a = heavy_calc(1)    # 強制的に同期
    b = heavy_calc(2)    # 同期
    a + b
}

# spawn：@block 内の並発
calc_mixed: () -> Int @block = {
    spawn { heavy_calc(1) }
    spawn { heavy_calc(2) }
    heavy_calc(3)        # 同期
}
```

### 3. DAG 依存解析

#### 3.1 核心原則：ボトムアップ実行

```
ユーザーコード（同期構文）：
    a = fetch(url0)
    b = fetch(url1)
    print(a)

コンパイル時分析（ボトムアップ）：
    print(a) は a を必要とする → fetch(url0) に依存
    fetch(url1) は誰も必要としていない → 孤立 DAG

実行時スケジューリング（葉から開始）：
    fetch(url0) → print(a)    ← 依存チェーン、順序付き
    fetch(url1)                ← 孤立、独立して並列
```

**重要な洞察**：「トップダウン」に Future を生成するのではなく、「ボトムアップ」に結果から逆算して依存関係を分析する。

#### 3.2 孤立 DAG：独立並列

```
メインフロー：fetch(url0) → process → print
孤立：       fetch(url1)  ← 誰も結果を必要としない、独立して並列

スケジューラー：メインフローは依存チェーンに従って実行、孤立は別コアで並列
```

#### 3.3 リソース型と副作用

**核心思想**：リソース操作は型でマークされ、DAG が自動的に依存関係を構築する。同一リソースは自動的に直列化され、異なるリソースは自動的に並列化される。

**リソース型の境界——明確に定義**：

リソース型はコンパイラの組み込みマーク付き型である。以下 유형はコンパイラによってリソースとして認識される：

| リソース型 | 説明 | コンパイラの動作 |
|----------|------|-----------|
| `FilePath` | ファイルシステムパス | 同一パスの操作は自動的に直列化 |
| `HttpUrl` | HTTP エンドポイント | 同一 URL の操作は自動的に直列化 |
| `DBUrl` | データベース接続 | 同一接続の操作は自動的に直列化 |
| `Console` | 標準出力 | すべての Console 操作は自動的に直列化 |

ユーザー定義のリソース型は明示的にマークする必要がある：
```yaoxiang
Database: Resource              # リソース型として明示的にマーク
query: (Database, String) -> Result(Row, Error)
# パラメータ Database は Resource、操作として自動的に識別
```

Resource でマークされていない型は、コンパイラーによってリソース依存関係が追跡されない。

**使用ルール**：
- 変数でリソースハンドルを渡し、DAG が自動的に順序を管理
- リテラルで直接同一リソースを使用するのはユーザーの設計上の問題であり、言語の責任ではない

```yaoxiang
# ✅ 正しい：変数渡し、DAG が自動的に直列化
filename: String = "data.txt"
File.write(filename, x)
File.write(filename, y)    # DAG が直列化

# ⚠️ ユーザーの責任：リテラル
File.write("data.txt", x)
File.write("data.txt", y)  # 並列実行の可能性あり、ユーザーが自分で責任を持つ
```

#### 3.4 無限ループ処理

```
1 つのループ → 直接的に同期実行、ゼロスケジューリングオーバーヘッド
複数のループ → スケジューラーがスライスを切り替えて、本当の並発
```

### 4. Result 型とエラー処理

```yaoxiang
Result: (T: Type, E: Type) -> Type = { ok: (T) -> Self, err: (E) -> Self }

# ? 演算子は透過的に伝播
process: () -> Result(Data, Error) = {
    data = fetch_data()?
    processed = transform(data)?
    save(processed)?
}
```

### 5. DAG ノード設計

```rust
enum NodeKind {
    Task,      // タスクノード
    Value,     // 値ノード
    Control,   // 制御フローノード
}

struct Node {
    id: NodeId,
    kind: NodeKind,
    inputs: Vec<ValueNodeId>,   // 入力依存
    outputs: Vec<ValueNodeId>,  // 出力値
    span: Span,                 // ソースコードの位置
}
```

| エッジ型 | 記号 | セマンティクス |
|--------|------|------|
| DataEdge | → | データ依存（値の流動） |
| ControlEdge | ● | 制御依存（順序実行） |
| SpawnEdge | ◎ | 並発エントリ（並列起点） |

### 6. 型システム

```
Send → スレッド間安全転送可能
Sync → スレッド間安全共有可能
Arc(T) は Send + Sync を実装（スレッド安全参照カウント）
```

## トレードオフ

### メリット

1. **漸進的な採用**：三層モデルが異なるスキルレベルに対応
2. **自然な構文**：同期コードが並列性能を獲得
3. **コンパイル時安全性**：Send/Sync 制約がデータ競合を排除
4. **デバッグ可能性**：エラーグラフが明確なエラー伝播ビューを提供

### デメリット

1. **学習曲線**：DAG 依存概念の理解が必要
2. **コンパイル時間**：全プログラム DAG 分析は遅い可能性
3. **ツールチェーンの複雑さ**：全新的なデバッグと可視化ツールが必要

## 代替案

| 方案 | なぜ選択しないか |
|------|--------------|
| 明示的な async/await のみサポート | 透過的並発の實現 불가능 |
| 完全透過的並発のみサポート | ユーザーが制御権を失う |
| Go スタイルの goroutine | 型安全性なし、コンパイル時検査不可 |
| L1 モードのみ | 並作モデルの核心価値を放棄 |

## 実装戦略

### 段階的区分

1. **段階 1 (v0.1)**：@block 同期モード、基盤型
2. **段階 2 (v0.2)**：FlowScheduler スケジューラー
3. **段階 3 (v0.3)**：spawn ブロック、明示的並発
4. **段階 4 (v0.5)**：L3 完全透過、DAG 自動分析
5. **段階 5 (v0.6)**：エラーグラフ、グラフデバッガー
6. **段階 6 (v1.0)**：本番使用可能な最適化

### 依存関係

- RFC-001 には外部依存なし（基盤コア）
- RFC-008（Runtime 並発モデル）→ 設計完了
- RFC-011（ジェネリックシステム）→ 設計完了

### リスク

1. **DAG 分析性能**：全プログラム分析は O(n²) の可能性、最適化が必要
2. **ツールチェーン欠如**：デバッガーはゼロから開発が必要
3. **ユーザー受容性**：透過的並発には良好なドキュメントが必要
