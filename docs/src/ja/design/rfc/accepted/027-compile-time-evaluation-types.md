# RFC-027：コンパイル時述語と統一静的検証

> **参考**:
>
> - [RFC-009: 所有権モデル](../accepted/009-ownership-model.md)
> - [RFC-010: 統一型構文 - name: type = value モデル](../accepted/010-unified-type-syntax.md)
> - [RFC-011: ジェネリクスシステム設計](../accepted/011-generic-type-system.md)
> - [RFC-024：spawn ブロックベースの並行モデル](../accepted/024-concurrency-model.md)
>
> **置き換え**：[RFC-022: ホーア論理による静的検証サポート（仕様注釈と仕様型）](../deprecated/022-hoare-logic-static-verification.md) — 廃止済み

## 概要

本文は、YaoXiang に**コンパイル時述語**を一級市民として導入し、すべてのコンパイル時静的検証を一つの**証明パイプライン**に統一することを提案する。コンパイル時述語は外付けの仕様注釈ではない——それは関数そのものある。Type を返す関数は型位置で使用でき、コンパイラがコンパイル時に呼び出して戻り値をチェックする。型は命题、コンパイル時評価は証明である。

**核心的な主張**：型検査がコンパイル時に行う唯一の仕事は、証明項の構築と検証である。型等価、トークン衝突、依存型の簡約、コンパイル時述語の評価、ホーア論理の含意——すべてがコンパイル時証明パイプライン内の異なる型検査であり、同じパイプラインを共有する。SMT ソルバーは型チェッカーのアクセラレーションモジュールであり、独立した信頼境界ではない。コンパイラが Unproven を返すとき、プログラマは証明として YaoXiang 関数を記述する——型チェッカーは他のすべての関数の戻り型を検証するのと同じ方法でこれを検証する。すべてが YaoXiang コード、すべてが型チェッカーによって検証される。

## 動機

### なぜ RFC-022 を廃止するのか？

RFC-022 は仕様を `//!` 注釈形式として設計していた：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← これは型と独立した注釈
    //! ensures: ExistsMax(result, arr[0..n])   ← これは型と独立した注釈
}
```

これは Curry-Howard 同型の根本的な誤りを犯している：**仕様と型を二層に分裂させる**。注釈は型ではない。注釈は型検査に参加しない。注釈は「外部ツール」のメンタルモデルである。

白書には明確に記されている：

> "`//!` 注釈はない。独立した仕様言語もない。すべてが型システムの中にある。"

### 現状の問題

- RFC-022 の `//!` 注釈は型システムと独立した外付け構文である
- 仕様型と通常の型は二体系であり、概念の冗長性を生む
- Debug Build 検証 / Release Build 無視の分裂パターンが統一性を破壊する
- SMT ソルバーは従来の認識では外部ツールとして位置づけられる——YaoXiang はこれを型チェッカーのアクセラレーションモジュールとして内蔵する
- 型検査、借用検証、コンパイル時述語検査、マクロ展開がそれぞれ異なる経路を辿る

### 正しいメンタルモデル

型検査は関数として抽象化できる：

```
verify : Program → Proved | Disproved(Model) | Unproven
```

すべてのコンパイル時検査——単純な型マッチング、借用衝突の検出、コンパイル時述語の検証——がこの関数のサブタスクである。証明項の複雑さと構築戦略が異なるだけで、同じ証明パイプラインを共有する。

コンパイラが Unproven を返すとき、プログラマは証明関数を提供する——その関数の戻り型は証明待ちの命题に等しい。型チェッカーがこれを検証する。これは通常の型検査と同じ操作である。

## 提案

### 1. `{}` は証明空間：型はアサート、検証は型検査

YaoXiang の `{}` はコンパイル時証明空間である。内部のすべてがアサートであり、コンパイラは各項目が True であることを保証する——自動証明されるか、プログラマが証明関数を提供するかのいずれかである。

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  コンパイラは x が Float、y が Float であることを保証

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  コンパイラは data が Array(T) であることを保証
```

**ジェネリクスはコンパイル時述語の特殊ケースである。**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          パラメータはシグネチャに  {} 内にはアサートのみ
#          コンパイラは呼び出し時に x > 0 を検証

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      パラメータはシグネチャに  コンパイラは type_of(T) == Type、type_of(data) == Array(T) を検証
```

同じパターン：`name: (params) -> Type = { アサート }`。コンパイラは「型アサート」と「値アサート」を区別しない——どちらも証明パイプライン内の評価対象である。

**ループ不変条件を別途書く必要はない。変数上の型注釈が Floyd-Hoare 不変条件そのものある。**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # 注釈は i を参照——s の型が i に依存することをコンパイラに伝える
    mut i: UpTo(arr.len) = 0     # 初期化時 i=0、検証：0 == sum(arr[0..0]) → True
    while i < arr.len {
        s += arr[i]  # コンパイラ検証：s_new == sum(arr[0..i+1])
        i += 1       # i 変更 → s 依存の revalidate をトリガー：s は SumUpTo(arr, i_new) を満たす
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

コンパイラはループ本体に対して一度だけ検証条件を生成する——帰納仮説（型注釈）→ 代入操作 → 新しい値が型注釈を満たすかどうか。証明パイプラインが帰納ステップを検証した後、すべての反復が自動的にカバーされる。`: decreases` も、`: Invariant` も、帰納証明も不要——コンパイラは帰納を各代入の局所 VC に分解する。

### 2. 事前/事後条件：パラメータ型と戻り型上のコンパイル時述語

RFC-022 の `//! requires`/`//! ensures` を放棄する。コンパイル時述語はパラメータまたは戻り値の型注釈として機能する。

**パラメータ側は関数呼び出しである。** コンパイル時述語は Type を返す関数であり、パラメータ側での使用方法はそれを呼び出すこと——`factorial(5)` と同じ。戻り値側は新しい概念を導入する：戻り値仮パラメータ。

```yaoxiang
# 事前条件：パラメータ型で明示的にコンパイル時述語を呼び出す
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b
#                       ^^^^^^^^^^  b は現在の仮パラメータ名で、Positive に渡される
#                       コンパイラは呼び出し点で実引数値を抽出し、b に代入し、Positive(実引数) を検証
#                       例：divide(10, 2) → Positive(2) = { 2 > 0 } を検証 → True
#                       例：divide(10, 0) → Positive(0) = { 0 > 0 } を検証 → False → コンパイルエラー

# 事後条件：戻り値仮パラメータ + コンパイル時述語
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

NonEmpty: (arr: Array(T)) -> Type = { arr.len > 0 }

max: (T: Ord) -> ((arr: NonEmpty(arr))) -> (result: IsMax(T, arr, result)) = {
#                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                            result は戻り値仮パラメータ、値は return で提供
#                                            コンパイラは return 点で戻り値を代入し、事後条件を検証
    candidate = arr[0]
    for i in 1..arr.len {
        if arr[i] > candidate { candidate = arr[i] }
    }
    return candidate
}
```

**重要なルール**：

- **パラメータ側**：`b: Positive(b)` —— `b` は現在の仮パラメータ名で、`Positive` の引数として渡される。関数呼び出し構文、暗黙の展開ゼロ。
- **戻り値側**：`-> (result: IsMax(T, arr, result))` —— `result` は戻り値仮パラメータ、値は `return` 文で提供される。`result` は型シグネチャ内にのみ存在し、述語によってのみ参照され、関数本体のスコープに入らず、呼び出し側にも現れない。
- **戻り値仮パラメータはオプション**：事後条件がない場合は記述せず、シグネチャは通常の関数と完全に一致する（`-> Int`）。
- **統一性**：パラメータと戻り値仮パラメータは同じ概念である——`仮パラメータ名: 述語呼び出し(仮パラメータ名)`、違いは値が呼び出し側から提供されるか `return` から提供されるかのみ。

### 3. パス条件の伝播：ランタイム値のコンパイル時検証

コンパイル時述語がバインディング位置で使用されるとき、パラメータはプログラマが明示的に渡す。ランタイム値が精錬型パラメータに入るとき、コンパイラはパス条件の収集と SMT の含意判定によって検証を完了する——プログラマが明示的に証明を渡す必要はない。

#### 3.1 明示的な関数呼び出し

コンパイル時述語がバインディング位置（パラメータ宣言、変数宣言、戻り型）で使用されるとき、パラメータはプログラマが明示的に渡す——関数呼び出しそのもの、暗黙の展開ゼロ。

`Positive: (x: Int) -> Type = { x > 0 }` はコンパイル時述語コンストラクタである。バインディング位置に現れるとき、プログラマは既にバインドされた変数名を明示的に渡す：

```yaoxiang
b: Positive(b)
// b は現在の仮パラメータとして宣言済み、Positive(b) は関数呼び出し
// 正規化後：b: { b > 0 }
```

コンパイラが暗黙的にパラメータを埋める必要はない——`b: Positive(b)` は `f(5)` と同じ、関数呼び出しそのものある。`b` はパラメータ名としてバインドされ、その型注釈 `Positive(b)` は `b` 自身を参照する——これは依存型の標準パターンであり、暗黙の展開ルールではない。

**RFC-010 の `self` との統一**：RFC-010 は `self` がキーワードではなく単なるパラメータ名の慣習（"`p`、`this`、`x` と書いても完全に同じ効果"）であることを確立している。`b: Positive(b)` は同じ機構を共有する——パラメータ名は型注釈内で参照可能。`self` は `self: Point` の位置に現れ、`b` は `b: Positive(b)` の位置に現れる。両者の型注釈はパラメータ自身を参照する。違いは型注釈の複雑さのみ、機構は完全に同じである——名前がバインドされた後、型はその名前に依存できる。

戻り型にも同じ明示的な関数呼び出しが使用される：

```yaoxiang
Sorted: (arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (arr: Array(T)) -> (result: Sorted(result)) = { ... }
//                        ^^^^^^^^^^^^^^^^^^^^^^^
//                        result は戻り値仮パラメータ、Sorted(result) は関数呼び出し
//                        コンパイラは return 点で戻り値を result に代入し、Sorted(戻り値) を検証
```

局所変数宣言にも同様に適用される：

```yaoxiang
let x: Positive(x) = 5
// x は 5 としてバインド、Positive(5) → { 5 > 0 } → True → 通過

// let y: Positive(y) = 0
// y は 0 としてバインド、Positive(0) → { 0 > 0 } → False → コンパイルエラー
```

#### 3.2 パス条件の収集

ランタイム値が条件分岐に現れるとき、コンパイラは自動的にパス条件を収集し、現在のスコープの**仮定集合**を形成する。これらの仮定は、コンパイル時の Bool 評価のための背景知識として検証に参加する。

```yaoxiang
if y > 0 {
    // コンパイラは自動的にこの分岐内で仮定を取得：{ y > 0 }
    let result = divide(x, y)
    // 検証条件：(y > 0) ⇒ (y > 0)
    // 証明パイプラインが含意成立を判定 → Proved
} else {
    // この分岐の仮定：{ !(y > 0) }
    // divide(x, y) を呼び出す場合、検証条件は !(y > 0) ⇒ y > 0
    // 証明パイプラインが不成立を判定 → Disproved
}
```

これはコンパイラがハードコードした特殊パターンではない——これはコンパイル時証明パイプラインの自然な動作である。型検査呼び出し点ごとに、パイプラインに以下を送信する：

```
{背景仮定} ⇒ {検証対象}
```

証明パイプラインが含意性を判定する。Proved → 通過、Disproved → コンパイルエラー + 反例、Unproven → コンパイルエラー + 未解決命題。背景仮定は現在のプログラム点のパス条件から得られる。

#### 3.3 仮定スタック

コンパイラは制御フロー解析において、各基本ブロックに対して仮定集合を維持する：

- **if-guard**：`if y > 0` → true 分岐に `y > 0` をプッシュ、false 分岐に `!(y > 0)` をプッシュ（else が使用されている場合）
- **match パターン**：`if let Some(v) = opt` → 分岐内に `opt == Some(v)` をプッシュ
- **論理結合**：`if x > 0 && y < 10` → 分岐内に `x > 0` と `y < 10` をプッシュ
- **関数事前条件**：`divide(a, b)` を呼び出すとき、`b` が `Positive` の証拠を持つことは、現在の仮定から、または実引数自身の精錬型注釈（`b` が既に `Positive` として注釈されている場合、その型は `b > 0` を運ぶ）から得られる
- **代入**：`let z = y` のとき、`y` 上の既存の精錬条件が `z` に伝播する

すべての仮定がコンパイル時証明パイプラインに入る。SMT アクセラレーション経路に入る際、SMT-LIB 背景アサーションに翻訳される。

#### 3.4 静的証拠なしではコンパイルエラー

プログラマが直接以下のように書いた場合：

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

現在のプログラム点に `y > 0` の仮定がなく、実引数 `y` 自身にも `Positive` 型注釈がない。検証条件は：

```
{} ⇒ { y > 0 }
```

パイプラインが `Disproved`（含意不成立）を返す → コンパイルエラー：

> パラメータ `b` が `divide` 呼び出しで `Positive` を満たすことを証明できない。
> `y` は関数入力から得られ、既に証明された境界がない。
> if 分岐ガードを使用した呼び出しを検討してください：`if y > 0 { divide(x, y) }`。

YaoXiang は、静的証拠を提供せずにランタイム値が精錬型パラメータに直接入ることを受け入れない。これは制約ではない——これはハード安全哲学の中核である。コンパイラが静的に証明できないコードは、コンパイルを通してはならない。

#### 3.5 統一パイプラインとの関係

パス条件の伝播は追加の機構ではない。これは、コンパイル時証明パイプラインの制御フロー解析への直接的な拡張である：

| 段階 | 責務 |
|------|------|
| パス条件の収集 | コンパイラの制御フロー解析段階で、各基本ブロックに仮定集合を注釈 |
| 検証条件の生成 | 検証が必要な型制約に遭遇したとき、パス条件 + 実引数型情報をマージ |
| 証明パイプライン評価 | コンパイラカーネル → SMT アクセラレーション → Proved / Disproved / Unproven を取得 |
| 結果 | `Proved` → 通過；`Disproved` → コンパイルエラー + 反例；`Unproven` → コンパイルエラー + 未解決命題（プログラマが証明関数を提供可能） |

新しい部品はない。特殊なルールもない。パス条件は証明パイプラインの背景知識そのものある——型等価や借用制約と同じパイプライン、同じ予算システムを共有する。

### 4. コンパイル時証明パイプライン

すべてのコンパイル時検査が同じパイプラインを共有する。パイプラインの核心操作は**型検査**である——証明項の型が証明対象の命题と等しいかを検査する。すべてが型検査である。

```
コンパイル時に Bool 式の評価が必要な場合（すなわち：証明項を構築する必要がある）
        │
        ├── 型等価（T1 == T2）
        │   → コンパイラが直接判定（構造等価）
        │
        ├── トークン衝突条件（!conflicting(tokens)）
        │   → フロー敏感生存性分析（Duplicate/Linear 属性追跡）
        │
        ├── 依存型簡約（n + m 簡約）
        │   → コンパイル時項書き換えシステム（βδι-簡約）
        │
        ├── コンパイル時述語（x > 0, forall...）
        │   → コンパイラ自身 + SMT アクセラレーションモジュール
        │
        └── ホーア論理含意（P ⇒ Q）
            → コンパイラ + SMT アクセラレーションモジュール
                    │
                    ▼
             ┌──────────┐
             │ Proved   │  → コンパイル通過
             │ Disproved│  → コンパイルエラー + 反例
             │ Unproven │  → コンパイルエラー + 未解決命題
             └────┬─────┘
                  │
                  ▼
         プログラマが証明関数を記述（YaoXiang コード）
                  │
                  ▼
         型チェッカーが検証 ──→ Proved ──→ コンパイル通過
                  │
                  ▼
            検証失敗 → コンパイルエラー：「証明が成立しない」
```

#### 4.1 証明結果：三値代数

コンパイル時評価は三つの結果を返す——これは停止問題の必然的な結論であり、証明論の自然な分割でもある：

```
eval_compile_time : BoolExpr → Proved | Disproved(Model) | Unproven
```

- **Proved** → 停止、証明項が構築済み、型検査通過。コンパイル継続。
- **Disproved(M)** → 停止、反例 M が存在。コンパイルエラー + 反例 + ソースコード位置。
- **Unproven** → 与えられたリソース上限内で証明が構築されなかった。コンパイルエラー + 未解決命題 + 予算消費レポート。

**Unproven ≠ False。** コンパイラが「証明できない」と言うことは、命题が偽であることと同義ではない——現在の自動証明能力を超えているだけである。これは正直な報告であり、欠陥ではない。

予算のハードリミットは停止問題の工学的解である。ノブは提供しない——提供するということは「あなたのプログラムは停止すると思うか」とユーザーに尋ねることになり、ユーザーは答えを知らず、コンパイラも知らない。

#### 4.2 Unproven の後：プログラマが証明を記述

コンパイラが Unproven を返すとき、プログラマは**証明関数**を記述できる——すなわち、戻り型が証明対象の命题に等しい YaoXiang 関数である。型チェッカーがこの関数を検証する——`add(a, b): Int` を検証するのと同じ機構である。

```
命题  = 型
証明  = プログラム（その型の値）
検証  = 型検査（唯一の信頼のルート）
```

SMT ソルバーは独立した信頼境界ではない——**型チェッカーのアクセラレーションモジュール**である。SMT は証明を見つけるのを手伝うが、証明を検証するのは常に型チェッカーである。SMT が `unsat` を返すとき、コンパイラはその結果を型チェッカーが検証可能な証明項として再構成する。再構成が失敗した場合（SMT の推論ステップがコンパイラカーネルの推論規則を超える場合）、Unproven にフォールバックする——プログラマは手動で証明関数を記述できる。

```yaoxiang
# 命题：コンパイラが自動証明できない精錬属性
FirstIsMin: (T: Ord, arr: Sorted(T)) -> Type = {
    forall i in 0..arr.len: arr[0] <= arr[i]
}

# 証明：プログラマが関数を記述、戻り型が上記の命题
# 型チェッカーがこの関数を検証——add(a,b): Int を検証と完全に一致
first_is_min: (T: Ord, arr: Sorted(T)) -> FirstIsMin(T, arr) = {
    # コンパイラはここで検証：関数本体の型 = FirstIsMin(T, arr)
    ...
}
```

AI も不要、Coq へのエクスポートも不要、新しい概念も不要。**コンパイル時に自動証明できない属性 → プログラマが YaoXiang コードで証明を記述 → 型チェッカーが検証。** プロセス全体は滑らかなグラデーションである——コンパイラが簡単な証明を代行し、難しいものは頭を使う。

#### 4.3 パイプライン内の階層的依存

上記の評価器は同じインターフェースを共有するが、評価順序が存在する。型等価はすべての後続分析の前提条件；所有権/トークン検査は型情報に依存；精錬述語検証は前二層の結果に依存。コンパイラは層ごとに評価し、低層で失敗した式は上層に入らない——型エラーのあるプログラム上で求解予算を浪費することを避ける。

```
評価順序（同じパイプライン、階層的スケジューリング）
├── 第 0 層：型等価（T1 == T2）
│   └── 構造ユニフィケーション → 失敗なら以降は無意味、直接 Disproved を返す
├── 第 1 層：所有権/トークン衝突
│   └── フロー敏感生存性分析 → 失敗ならメモリ安全性が成立しない、直接 Disproved を返す
└── 第 2 層：精錬述語/ホーア含意
    └── コンパイラ自身 → SMT アクセラレーション → Proved / Disproved / Unproven を取得
```

各層は依然として `Proved/Disproved/Unproven` を返し、同じインターフェースと同じ予算システムを共有する。

### 5. 三層関数の統一

| 階層 | 実行タイミング | 入力 | 出力 | 例 |
|------|----------|------|------|------|
| 値レベル関数 | ランタイム | 値 | 値 | `add: (a: Int, b: Int) -> Int = a + b` |
| 型コンストラクタ | コンパイル時 | 型/値 | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| コンパイル時述語 | コンパイル時 | 値 | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

すべて同じ `name: type = value` 構文を使用。コンパイル時述語と型コンストラクタは同じコンパイル時証明パイプラインを辿る——`{}` は証明空間である。

### 6. ループ：Floyd-Hoare 検証条件生成

ループは個別の `: Invariant(...)` や `: decreases(...)` 注釈を必要としない。変数上のコンパイル時述語型注釈が Floyd-Hoare 形式のアサートを定義する——コンパイラは型注釈から検証条件を生成し、証明パイプラインが各代入が型を保持するかをチェックする。

核心機構：各代入操作は Hoare 三つ組 `{P} x := e {Q}` に対応し、検証条件は `P ⇒ Q[e/x]`。コンパイラはループ本体に対して一度だけ検証条件を生成する——証明パイプラインが帰納ステップを検証した後、すべての反復が自動的にカバーされる。

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # 注釈は i を参照；初期化時 i=0、検証：0 == sum(arr[0..0]) → True
    mut i: UpTo(arr.len) = 0     # 検証：0 <= 0 <= arr.len → True
    while i < arr.len {
        # コンパイラはループ本体に対して一度 VC を生成。前提：s は SumUpTo(arr, i) を満たし、i は UpTo(arr.len) を満たす。
        #
        # s += arr[i]：
        #   検証義務：s_new は SumUpTo(arr, i)（現在の i は不変）を満たす
        #   s_new = s_old + arr[i] を代入：
        #     s_old + arr[i] == sum(arr[0..i+1]) が必要
        #     帰納仮説 s_old == sum(arr[0..i]) から、両辺に arr[i] を加える：
        #     sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   コンパイラ + SMT：線形算術、ミリ秒オーダー → Proved
        #
        # i += 1：
        #   i 変更 → 依存図で s の型注釈が i を参照 → revalidate をトリガー
        #   新しい検証対象：s は SumUpTo(arr, i_new) を満たす
        #   すなわち s == sum(arr[0..i_new])、前ステップで保証済み → Proved
        s += arr[i]
        i += 1
    }
    return s  # このとき s: SumUpTo(arr, arr.len)、すなわち s == sum(arr[0..arr.len])
}
```

ループ不変条件は変数上の型注釈そのものある——プログラマが型を記述し、コンパイラが帰納ステップをチェックする。コンパイラは不変条件を「発見」する必要も、帰納を「自動実行」する必要もない——帰納証明を各代入操作の局所検証条件に分解し、証明パイプラインに分割して処理させる。

#### 6.1 依存追跡：可変変数上の依存型

上記の機構の前提は、`s` の型注釈 `SumUpTo(arr, i)` が `i` を参照していることをコンパイラが知っていること——`i` が変更されたとき、`s` の型制約もそれに応じて変化する。これには、コンパイラが**変数間の型依存グラフ**を維持する必要がある。

**データ構造**：

```
TypeDepGraph: Map<VarName, Set<VarName>>
# キーは依存される変数、値は型注釈内でその変数を参照する変数の集合
# 例：{ i: {s}, j: {s, t}, ... }
```

**構築**：型チェッカーが `mut v: Pred(... x ...) = init` を処理するとき、`Pred(...)` 引数内の自由変数参照を解析する。引数が現在のスコープ内の他の可変変数 `x` を参照している場合、依存グラフに `x → v` を記録する。

**トリガー**：依存される変数 `x` が代入されるとき、コンパイラ：
1. 依存グラフで `x` に依存するすべての変数 `{v₁, v₂, ...}` を検索
2. 各 `v` について、検証条件を生成：`v の現在値 は更新された型 Pred(... x_new ...) を満たすか`
3. VC を証明パイプラインに送る

**代入順序に敏感**：依存追跡は正しい代入順序を自然に強制する。`SumUpTo(arr, i)` を例に取る：

```yaoxiang
# 正しい順序
s += arr[i]   # s_new は SumUpTo(arr, i+1) を満たす
i += 1        # i 変更 → s の SumUpTo(arr, i_new) を revalidate → True

# 誤った順序——コンパイラが拒否
i += 1        # i 変更 → s の SumUpTo(arr, i_new) を revalidate
              # s はまだ更新されておらず、s_old == sum(arr[0..i_old]) ≠ sum(arr[0..i_new])
              # → コンパイルエラー：変数 s は型 SumUpTo(arr, i_new) を満たさない
s += arr[i]   # 到達不能
```

**複合依存**：一つの変数が複数の変数に依存できる。型注釈 `{ v: Int; v == x + y }` は `x` と `y` の両方に同時に依存する——いずれの変更も revalidate をトリガーする。

**証明パイプラインとの関係**：依存追跡は VC 生成のトリガーであり、独立した検証機構ではない。「いつ VC を生成する必要があるか」に答え、証明パイプラインが「VC が成立するか」に答える。

### 7. 停止性検査

コンパイル時に完全自動。コンパイラが証明できるループは通過し、証明できないものは直接コンパイルエラーを報告する——プログラマはループの停止性をコンパイラに自動解析させなければならない。半自動注釈のための余地は与えない。

#### 6.1 設計原則

コンパイラは二箇所から停止証明に必要な情報を自動抽出する：

1. **変数型注釈**：精錬型内の境界制約（例：`UpTo(n)` は上界 `n` と下界 `0` を与える）
2. **ループ本体の操作**：各反復で変数に加えられる操作

コンパイラは優先順位に従って四つの度量合成戦略を試し、見つかれば停止する。

#### 6.2 戦略 1：線形ランク関数の自動合成

変数に線形境界注釈がある場合、コンパイラは候補線形度量を列挙し SMT で検証する。

```
入力：
  変数 v₁: UpTo(u₁), v₂: UpTo(u₂), ...（上下界を持つ変数）
  ループ条件 cond
  ループ本体内の代入集合

アルゴリズム：
  1. 型注釈から各変数の境界を抽出：[low_i, high_i]
  2. 候補度量を列挙：v_i, u_i - v_i, v_i - v_j, 等の線形結合
  3. 各候補度量 m について：
     - SMT で m ≥ 0 を検証（型境界から導出）
     - ループ本体の各実行パスについて、SMT で m' < m（厳密減少）を検証
  4. 条件を満たす線形結合を発見 → 停止性が証明された
```

カバー範囲：任意の変数が線形式（`v = a·v + b`）で代入され、有界型注釈を持つループ。`i += const`、`i -= const`、および二分探索的な区間収縮を含む：

```yaoxiang
# 二分探索：low = mid + 1 または high = mid
# 度量 high - low は両パスで厳密に減少
binary_search: (arr: Sorted(Int, arr), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 戦略 2：述語違反カウント——目標型から自動抽出 <span style="color:orange">【実験的戦略】</span>

> ⚠️ **現在の状態：実験的戦略、段階 3 実装時に実際の実現可能性に基づいて含有を決定する。**
> この戦略は隣接交換操作（バブルソート、挿入ソート）に有効だが、非隣接操作（クイックソート partition、ヒープソート sift-down）は自動証明できない。
> カバレッジの境界は下表の通り。段階 3 で実現不可能と判明した場合、この戦略は削除されるか将来の作業に格下げされる。

核心的な洞察：**ユーザーが記述した仕様はコンパイラの推論材料である。** コンパイラに「ソートとは何か」を内蔵する必要はない——`Sorted` の定義を読み、その定義から自動的に度量を抽出する。

```
入力：
  目標型：Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  ループ本体の操作：隣接要素の交換

アルゴリズム：
  1. 述語定義を解析：forall i in range: cond(i, arr)
  2. 自動生成度量：violation_count = |{ i | ¬cond(i, arr) }|
  3. 操作の度量への影響を解析：
     - 隣接交換 arr[j], arr[j+1] = arr[j+1], arr[j]
     - インデックス j-1, j, j+1 の三対のみに影響
     - arr[j] > arr[j+1]（述語違反）なら、交換後この対は述語を満たす
     - violation_count は少なくとも 1 減少
  4. 上界：n·(n-1)/2（最大隣接逆順数）、下界：0
  → 停止性が証明された
```

**現在のカバー範囲**：

| アルゴリズム | 操作モード | 戦略 2 で証明可能？ | 理由 |
|------|----------|:---:|------|
| バブルソート | 隣接交換 | ✅ | violation_count は各交換で厳密減少 |
| 挿入ソート | 隣接移動 | ✅ | 各シフトで違反対を一つ解消 |
| 選択ソート | 非隣接交換 | ❌ | 単一交換で violation_count が増加し得る |
| クイックソート | partition 分割 | ❌ | 非隣接交換、単調減少を保証しない |
| ヒープソート | sift-down | ❌ | ツリー操作、violation_count は非単調 |

**補完戦略**：クイックソートの場合、`low < high` 区間収縮は戦略 1（線形ランク関数）でカバーできる——外層 partition 再帰、各反復で区間が半減する。戦略 1 と戦略 2 は補完的にカバーし、ほとんどの実際のアルゴリズムの停止性はどちらか一方で証明できる。ただし、戦略 2 の汎用化（非隣接操作、ツリー操作）は依然として未解決問題である。

```yaoxiang
sort: (arr: Array(Int)) -> (result: Sorted(result)) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 戦略 3：有界増減モード

`v += const`（正の定数）、変数に上界型注釈あり → 度量 `upper_bound - v` は各反復で `const` ずつ減少し、下界 0。これは戦略 1 の縮退ケースであり、コンパイラは最初に高速処理する。

#### 6.5 戦略 4：乗法スケーリング度量テンプレート

`v *= const`（const > 1）、変数に上下界型注釈あり。コンパイラは対数度量テンプレート `ceil(log_const(upper/v))` を内蔵し、const をかけるたびに度量が 1 減少する。

```yaoxiang
mut i: Positive(i) = 1
while i < n {
    # コンパイラ自動導出：度量 ceil(log₂(n/i))、2 をかけるたびに度量が 1 減少
    i *= 2
}
```

#### 6.6 停止性と正当性の分離

停止性証明と正当性証明は独立している：
- **停止性**：上記四つの戦略がループが有限ステップで終了することを自動証明
- **正当性**：ループ本体が目標型に向かって進行しているかは、コンパイル時証明パイプラインが検証条件でチェック

両方通過 → コンパイル通過。停止性証明成功だが正当性失敗 → コンパイルエラー + 反例。正当性証明成功だが停止性証明失敗 → コンパイルエラー、解析不能な変数または操作を示す。両方失敗 → コンパイルエラーで二項目の失敗理由を別々に報告。

#### 6.7 再帰関数の停止性検査

コンパイル時に評価が必要な再帰関数について、コンパイラは引数の減少をチェックする：

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # コンパイラ解析：n-1 < n → 減少 → 停止
}

# コンパイル時に使用——コンパイラは factorial のコンパイル時停止を保証
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120、コンパイル時に完了
```

| シナリオ | 動作 |
|------|------|
| コンパイラが再帰的減少（例：`n-1`）を解析可能 | コンパイル時評価 |
| 減少しない / 減少を判定不能 | コンパイルエラー |
| ランタイム呼び出し（型位置ではない） | 停止性検査不要 |

#### 6.8 ハード境界

`i = f(i)` で `f` が非可逆、非閉包、いかなる単調性も保持しない——数学的に自動停止証明は不可能。コンパイルエラー：

> このループは停止性を自動証明できない。ループ変数が解析不能な関数 `f` に依存している。コンパイラが解析可能な反復パターンを使用してください。

これはコンパイラの失敗ではない。静的に安全を証明できないコードは、コンパイルを通してはならない。

### 8. SMT ソルバー：型チェッカーのアクセラレーションモジュール

SMT ソルバーは従来の言語では外部ツール（F\* が Z3 を呼び出す、Dafny が Z3 を呼び出す）である。YaoXiang では、**型チェッカーのアクセラレーションモジュール**である——コンパイラカーネル自身が直接判定できない場合にのみ呼び出される。SMT は証明を見つけるのを手伝うが、証明を検証するのは型チェッカーである。

**信頼モデル**：型チェッカーは唯一の信頼のルート。SMT ソルバーはアクセラレーションモジュールである——証明を見つけるのを手伝うが、SMT は独立した信頼境界ではない。コンパイラは Z3 の `unsat` 結果を信頼する（F\*/Dafny の路線と一致——Z3 が誤る確率はコンパイラ自身のバグ率より低く、これは工学的に実務的な選択）。真の信頼性の懸念は SMT 翻訳層で制御される——翻訳にバグがあれば、コンパイラは他のテストで露呈する。

**インターフェース**：コンパイラは内部的に SMT-LIB 2.6 標準形式に翻訳し、特定のソルバー API に縛られない。SMT-LIB は ISO 標準で、Z3、CVC5、MathSAT、Yices がすべてネイティブサポートする。

**デフォルトバックエンド**：Z3（MIT ライセンス、最も広範なドキュメントとコミュニティ検証）。CVC5 は SMT-LIB 互換の代替として——ユーザーはコンパイラフラグで切り替え可能。

「汎用ソルバー抽象化層」は作らない——SMT-LIB が抽象層である。将来 CVC5 が特定の理論でブレークスルーを達成した場合、切り替えはバイナリの置き換えのみで済み、コンパイラコードの変更は不要である。

```
コンパイル時 Bool 式
        │
        ├── コンパイラカーネルが直接判定可能（構造等価、簡単な算術、
        │   定数畳み込み後の自明な公式）
        │   → 直接 Proved / Disproved を返す
        │
        └── コンパイラカーネルが直接判定不能（量化子、記号変数）
            → 依存型事前簡約（factorial(5) → 120）
            → SMT-LIB 形式に翻訳
            → Z3/CVC5 に送信（予算制限付き）
            → 戻り値：unsat → Proved  │  sat + モデル → Disproved  │  unknown → Unproven
```

**求解予算——スタック深度のようなハードリミット**：

| 予算次元 | デフォルト | 説明 |
|----------|--------|------|
| 求解ステップ数 | 10,000 | Z3 は線形算術に対して通常百ステップ以内。10,000 ステップで実用述語の 99% をカバー。 |
| 時間 | 100ms | 単一述語が 100ms を超える = ユーザーがコンパイル時プログラムを書いている。100ms × 50 述語 = 5 秒コンパイル時間上限。 |
| 量化子インスタンス化深度 | 3 | 三層のネスト量化子が実際のパターンをカバー。超えるとおそらく論理演習問題。 |

予算超過で Unproven を返し、コンパイルエラー + 述語位置 + 消費量。降格なし、ランタイムチェックなし、サイレントパスなし。

**なぜこれが実際に機能するか**：工学的に実用述語の 95% は線形算術である——`x > 0`、`arr.len > 0`、`0 <= idx < arr.len`——すべて決定可能フラグメント内にあり、SMT ソルバーはこのような問題に対してミリ秒で返す。予算を超える複雑な述語の稀なケースに遭遇した場合、プログラマは証明関数を記述すればよい。

依存型は SMT 呼び出し前に事前簡約層を経る：`factorial(5)` は直接コンパイル時評価で `120` になり、`append([1,2], [3])` は直接評価で `[1,2,3]` になる。これらの決定論的な値計算は SMT 予算を消費しない。

プログラマは SMT の存在を知る必要はない。メンタルモデルは：**コンパイラが証明できれば通過、できなければエラー報告——コンパイラが不得手な場合は、関数で証明して見せればよい。**

### 9. コンパイル時述語の合成

コンパイル時述語は Type を返す関数であり、合成は関数合成によって自然に実現される：

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(arr)
}
```

### 10. コード例

#### 9.1 除算の安全性

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b

result = divide(10, 2)   # ✅ コンパイラ検証 Positive(2) = { 2 > 0 } → True
# result = divide(10, 0)  # ❌ コンパイラ検証 Positive(0) = { 0 > 0 } → False
```

#### 9.2 配列アクセスの安全性

```yaoxiang
InBounds: (idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (arr: Array(T), idx: InBounds(idx, arr)) -> T = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ コンパイラ検証 InBounds(1, arr) = { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ コンパイラ検証 InBounds(5, arr) = { 0 <= 5 && 5 < 3 } → False
```

#### 9.3 ソートの正当性

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T))) -> (result: Sorted(T, result)) = {
    result = arr.clone()
    # ... ソートアルゴリズムの実装 ...
    return result
}
```

#### 9.4 ループ：コンパイラの VC 生成

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```

## 詳細設計

### 構文変更

| 以前 (RFC-022) | 以降（本 RFC） |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | コンパイル時述語をパラメータ型として使用 `(b: Positive(b))` |
| `//! ensures: ExistsMax(result, arr)` | 戻り型に戻り値仮パラメータを使用 `-> (result: IsMax(T, arr, result))` |
| `/*! invariant: ... !*/` | 変数上のコンパイル時述語型注釈——Floyd-Hoare 不変条件 |
| `//! decreases: n` | コンパイラの完全自動度量関数導出 |
| 仕様は注釈 | 仕様は型システム |

### 構文

**コンパイル時述語に新しいキーワードはない。** `{}` は証明空間であり、既存の型定義構文と完全に一致する。コンパイル時述語は Type を返す関数そのものある——`name: (params) -> Type = { アサート }`。使用時は関数呼び出しそのものある——`Positive(b)`、`IsMax(T, arr, result)`。

```bnf
# コンパイル時述語 = Type を返す関数、{} 内はコンパイラが検証するアサート
# 既存の関数/型構文を使用、新規 BNF ルール不要
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

**新規構文概念：戻り値仮パラメータ**——`-> (name: Type)` 内の `name` は戻り値仮パラメータ。

戻り値仮パラメータは、YaoXiang が既存の関数構文に導入する**唯一の構文概念**である。そのセマンティクス：
- `name` の値は `return` 文で提供される
- `name` は型シグネチャ内にのみ存在し、事後条件述語によって参照される（例：`-> (result: IsMax(T, arr, result))`）
- `name` は関数本体のスコープに入らず、呼び出し側にも現れない
- 戻り値仮パラメータは**オプション**——事後条件がない場合、シグネチャは通常の関数と完全に一致する（`-> Int`）、追加負担なし

導入の理由：事後条件は「関数が返そうとしている値」を参照する必要がある。戻り値仮パラメータがない場合、コンパイラは述語に戻り値を参照させるために特別なルール（暗黙変数 `$result` や `__retval__` など）を使うしかない。戻り値仮パラメータはこの参照を明示化する——それは単なる仮パラメータであり、値が呼び出し側ではなく `return` から提供される点が異なるだけである。

**証明関数**は新しい概念ではない——それは戻り型がアサートされる命题に等しい YaoXiang 関数そのものある。コンパイラが Unproven を返すとき、プログラマが証明関数を提供し、型チェッカーが他のすべての関数の戻り型を検証するのと同じ方法で検証する。新しい構文、新しいキーワード、新しいルールは不要。

### 型システムへの影響

- **型宇宙**：コンパイル時述語は Type₂ 層に位置する——値を受け取り Type を返す関数、型コンストラクタと同じ階層
- **ジェネリクスとの相互作用**：コンパイル時述語はジェネリクスパラメータを取れる、例：`NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **所有権との相互作用**：コンパイル時述語内の式は所有権ルールに従い、読み取りのみ可能、書き込み不可
- **型推論**：コンパイル時述語のパラメータは HM 型推論に参加

### ランタイム表現

コンパイル時述語はランタイムで**完全に消去される**。`Positive: (x: Int) -> Type = { x > 0 }`——パラメータ `b: Positive(b)` のランタイム表現は `Int` そのものである。精錬条件 `{ x > 0 }` はコンパイル時に証明パイプラインで検証され、検証完了後にランタイム痕跡を残さない。

- `Positive(5)` → ランタイム表現は `Int(5)`、精錬条件 `{ 5 > 0 }` は通過済み、消去
- `SumUpTo(arr, 0)` → ランタイム表現は `Int(0)`、等式 `0 == sum(arr[0..0])` は通過済み、消去
- コンパイル時述語を型位置に配置する（例：`f(x: Positive(x))`）ことはラッパー型を生成せず、追加メモリを割り当てず、ランタイムチェックを挿入しない

ジェネリクスの消去と精錬型の消去は同じ原則である：どちらもコンパイル時関数であり、コンパイル時評価後に完全に消失する。Release Build ではコンパイル時述語のランタイムオーバーヘッドはゼロ——これは「述語は関数である」の直接的な帰結である。

**`ref` との相互作用制約**：コンパイル時述語は不変借用または所有権が既に移転された値のみを参照できる。可変借用される値を参照するコンパイル時述語は、コンパイラがコンパイル時に検証結果がランタイムでも成立することを保証できない——このような用法は直接コンパイルエラーを報告する。

### コンパイラ変更

1. **パーサー**：コンパイル時述語は標準関数構文を使用、追加解析ルール不要
2. **コンパイル時証明パイプライン**：統一 Proved/Disproved/Unproven 戻りインターフェース、自動戦略選択
3. **SMT アクセラレーションモジュール**：SMT-LIB 2.6 翻訳層、デフォルトバックエンド Z3、CVC5 代替
4. **型チェッカーカーネル**：推論規則の実装——構造等価、βδι 簡約、全称量化子導入/除去。これが唯一の信頼のルートであり、SMT とプログラマ証明はどちらもこれを通じて検証される
5. **検証条件生成**：WP/SP 計算 + ループ不変条件証明義務
6. **エラー報告**：反例フォーマット + 未解決命題レポート + ソースコード位置関連付け

### 後方互換性

- ✅ コンパイル時述語を使用しないコードは完全に不変
- ✅ コンパイル時述語は Release Build でゼロランタイムオーバーヘッド
- ⚠️ RFC-022 の `//!` 構文はサポートされない——ただし 022 は実装されたことがないため、移行負担なし

## トレードオフ

### 利点

- **Curry-Howard 同型の完全な実現**：型は命题、プログラムは証明、`name: Proposition = Proof`
- **統一性**：コンパイル時述語と通常関数は完全に同じ構文を使用、概念分裂なし
- **SMT 透明性**：プログラマは SMT の存在を知る必要がない、メンタルモデルは型検査と一致
- **段階的採用**：一つのコンパイル時述語から始めて、徐々にカバレッジを拡大可能
- **ゼロランタイムオーバーヘッド**：すべての評価がコンパイル時に完了、Release Build はランタイムアサートを一切挿入しない

### 欠点

- **コンパイル時間**：SMT 求解がコンパイル時間を増加させるが、予算ハードリミットが上限を保証
- **自動証明の境界**：一階線形算術を超える複雑な述語はプログラマの証明関数を必要とする。これは言語の欠陥ではない——停止問題の必然的な結論である。コンパイラは True/False を偽装せず Unproven を正直に報告する
- **学習曲線**：効果的なコンパイル時述語と証明関数の記述には Curry-Howard 同型の基本的な直感の理解が必要
- **実装の複雑さ**：コンパイル時証明パイプラインの統一には慎重な設計が必要

### リスク軽減

- SMT 求解予算のハードリミット（ステップ数 10,000 / 時間 100ms / インスタンス化深度 3）、予算超過で Unproven
- 依存型事前簡約：決定論的な値計算を先に処理し、SMT は非決定的部分のみ担当
- Unproven は行き止まりではない：プログラマは証明関数を記述でき、型チェッカーが検証——他の関数の戻り型検証と一致
- インクリメンタル検証：変更されたモジュールのみ検証
- 明確なエラーメッセージ + 反例表示 + 予算消費レポート + 未解決命題 + 提案（コンパイラが提供可能な場合）

## 代替案

| 案                             | 選択しない理由                           |
| -------------------------------- | -------------------------------------- |
| RFC-022：`//!` 注釈式仕様        | 仕様と型を分裂、Curry-Howard 同型に違反 |
| 独立仕様ファイル（CVL など）           | 仕様とコードが分離、保守コスト増加           |
| ランタイムアサートのみ                     | 静的正当性保証不可                     |
| 外部証明アシスタント（Coq など）           | コンパイラから独立した専用の証明言語と信頼境界が必要。YaoXiang の選択：証明は YaoXiang コードそのもの、型チェッカーが唯一の信頼のルート |
| **本案：コンパイル時述語を一級市民として** | ✅                                     |

## 未解決問題

- [x] **SMT ソルバーの選択**：デフォルト Z3（MIT ライセンス、最も広範に検証済み）。CVC5 を SMT-LIB 互換の代替として、コンパイラフラグで切り替え。コンパイラ内部の翻訳対象は SMT-LIB 2.6 標準形式——SMT-LIB が抽象層であり、カスタム汎用ソルバーインターフェースは作らない。
- [x] **求解予算の具体的な数値**：ステップ数 10,000 / 時間 100ms / 量化子インスタンス化深度 3。コンパイラ内部で固定、ノブは提供しない。実際の使用で本当に不十分であることが証明された場合（ユーザーが間違えたのではなく）、調整する。
- [x] **量化子のサポート範囲**：言語レベルで量化子の階数を制限しない。コンパイル時述語は Type パラメータを受け取る——Type には関数型も含まれる——したがって高階量化子は型システムの自然な帰結であり、特殊な構文は不要。SMT ソルバーは一階量化子（forall/exists、交差ネストサポート、予算深度 3 で制限）を自動判定可能。高階量化子：SMT が Unproven を返し、コンパイラが「この述語は自動証明の範囲外です、証明関数を提供してください」と通知。プログラマは戻り型が该命题に等しい YaoXiang 関数を記述する——型チェッカーがこの関数を検証。外部エクスポート不要、AI 不要、対話的証明モード不要。すべてが YaoXiang コード、すべてが型チェッカーで検証。
- [x] **反例フォーマット**：ソースコード変数名を直接 SMT 変数名として使用（モジュールプレフィックスを追加して衝突回避）。Z3 モデルが返されたとき、変数名で逆引き。出力フォーマット：変数名 = 具体値 + ソースコード位置 + 述語定義位置。複雑なマッピング層は作らない。
- [x] ~~**`ref` スマートポインタとコンパイル時述語の相互作用？**~~ → 決定済み：コンパイル時述語は不変借用または所有権が移転済みの値のみ許可。可変借用される値はコンパイル時述語に現れない。
- [x] **`forall` 述語違反カウント度量の非隣接操作への拡張？** → 拡張しない。現在のカバー範囲（隣接交換、隣接移動）は戦略 1（線形ランク関数）で補完的にカバーされる——クイックソートの外層区間収縮は戦略 1 が補い、ヒープソートは戦略 1（配列インデックスパターン）が補う。どの戦略でも停止性を証明できないループは、コンパイラが直接エラーを報告する——これはハード安全哲学であり、欠陥ではない。将来、実シナリオ（学術的構成ではない）のアルゴリズムが四つの戦略でカバーできないケースが現れた場合、再検討する。
- [x] **線形ランク関数の列挙組み合わせ爆発**：候補列挙上限は 3 つの有界変数。≤3 の場合、すべての線形結合を列挙し SMT で一つずつ検証。>3 の場合、単一変数度量（`v_i`、`u_i - v_i`）のみ試行し、失敗で直接コンパイルエラー——プログラマに「ループに >3 個の有界変数があり、コンパイラが多変量度量を自動合成できません」と通知。これは工学的妥協ではない——プログラマにより単純なループを書くことを強制する。
