# RFC-027：コンパイル時述語と統一静的検証

> **参考**:
>
> - [RFC-009: 所有権モデル](../accepted/009-ownership-model.md)
> - [RFC-010: 統一型構文 - name: type = value モデル](../accepted/010-unified-type-syntax.md)
> - [RFC-011: ジェネリクスシステム設計](../accepted/011-generic-type-system.md)
> - [RFC-024: spawn ブロックに基づく並行モデル](../accepted/024-concurrency-model.md)
>
> **置換**：[RFC-022: ホーア論理静的検証サポート（仕様注釈と仕様型）](../deprecated/022-hoare-logic-static-verification.md) — 廃止済み

## 要約

本文では、YaoXiang に**コンパイル時述語**を第一級市民として導入し、すべてのコンパイル時静的検証を単一の**証明パイプライン**に統一することを提案する。コンパイル時述語は外部仕様注釈ではない——それは関数そのもの。Type を返す関数は型位置で使用でき、コンパイラがコンパイル時にそれを呼び出して戻り値をチェックする。型は命題であり、コンパイル時評価は証明である。

**核心的な主張**：型検査がコンパイル時に行う唯一の仕事は、証明項の構築と検証である。型等式、トークン衝突、依存型簡約、コンパイル時述語評価、ホーア論理含意——これらすべてがコンパイル時証明パイプライン内の異なる型検査であり、単一のパイプラインを共有する。SMT ソルバーは型検査器の加速モジュールであり、独立した信頼境界ではない。コンパイラが Unproven を返したとき、プログラマは YaoXiang 関数を証明として書く——型検査器は、任意の関数の戻り型を検証する方法とまったく同じ方法でそれを検証する。すべてが YaoXiang コードであり、すべてが型検査器によって検証される。

## 動機

### なぜ RFC-022 を廃止するのか？

RFC-022 は仕様を `//!` 注釈形式として設計していた：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← これは型から独立した注釈
    //! ensures: ExistsMax(result, arr[0..n])   ← これは型から独立した注釈
}
```

これは Curry-Howard 同型の根本的な誤りを犯している：**仕様と型を二層に分裂させた**。注釈は型ではない。注釈は型検査に参加しない。注釈は「外部ツール」のメンタルモデルである。

白書には明確に記されている：

> "`//!` 注釈はない。独立した仕様言語はない。すべてが型システム内にある。"

### 現在の問題

- RFC-022 の `//!` 注釈は型システムから独立した外部構文
- 仕様型と通常型は二系統であり、概念の冗長性を生む
- Debug Build 検証 / Release Build 無視の分裂パターンが統一性を破壊
- SMT ソルバーは伝統的な認識では外部ツールとして位置付けられる——YaoXiang はそれを型検査器の加速モジュールとして内蔵
- 型検査、借用検証、コンパイル時述語検査、マクロ展開はそれぞれ異なるパスを通る

### 正しいメンタルモデル

型検査は関数として抽象化できる：

```
verify : Program → Proved | Disproved(Model) | Unproven
```

すべてのコンパイル時検査——単純な型一致、借用衝突検出、コンパイル時述語検証——がこの関数のサブタスクである。それらは単一の証明パイプラインを共有し、証明項の複雑さと構築戦略によってのみ異なる。

コンパイラが Unproven を返したとき、プログラマは証明関数を提供する——その関数の戻り型は証明すべき命題と等しい。型検査器はそれを検証する。これは通常の型検査と同じ操作である。

## 提案

### 1. `{}` は証明空間：型はアサーション、検証は型検査

YaoXiang の `{}` はコンパイル時証明空間である。その中のすべてがアサーションであり、コンパイラは各項目が True であることを保証する——自動証明されるか、プログラマが証明関数を提供するか。

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  コンパイラは x が Float であり、y が Float であることを保証

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  コンパイラは data が Array(T) であることを保証
```

**ジェネリクスはコンパイル時述語の特例である。**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          パラメータはシグネチャに    {} 内にはアサーションのみ
#          コンパイラが呼び出す時に検証

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      パラメータはシグネチャに  コンパイラが type_of(T) == Type、type_of(data) == Array(T) を検証
```

同じパターン：`name: (params) -> Type = { アサーション }`。コンパイラは「型アサーション」と「値アサーション」を区別しない——すべて証明パイプライン内の評価対象である。

**ループ不変条件を別途書く必要はない。変数上の型注釈が Floyd-Hoare 不変条件となる。**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # 注釈は i を参照——コンパイラに s の型が i に依存することを伝える
    mut i: UpTo(arr.len) = 0     # 初期化時 i=0、検証：0 == sum(arr[0..0]) → True
    while i < arr.len {
        s += arr[i]  # コンパイラ検証：s_new == sum(arr[0..i+1])
        i += 1       # i 変更 → s 依存の自動再検証をトリガ：s は SumUpTo(arr, i_new) を満たす
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

コンパイラはループ本体に対して検証条件を一度生成する——帰納仮説（型注釈）→ 代入操作 → 新しい値が型注釈を満たすか。証明パイプラインが帰納ステップを検証した後、すべての反復が自動的にカバーされる。`: decreases` も `: Invariant` も帰納証明も必要ない——コンパイラは帰納を各代入の局所 VC に分解する。

### 2. 事前/事後条件：パラメータ型と戻り型上のコンパイル時述語

RFC-022 の `//! requires`/`//! ensures` を廃止する。コンパイル時述語は引数または戻り値の型注釈として用いる。

**引数側は関数呼び出しである。** コンパイル時述語は Type を返す関数であり、引数側の使用方法はそれを呼び出すこと——`factorial(5)` とまったく同じである。戻り値側は新しい概念を導入する：戻り値仮引数。

```yaoxiang
# 事前条件：パラメータ型内でコンパイル時述語を明示的に呼び出す
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b
#                       ^^^^^^^^^^  b は現在の仮引数名、Positive のパラメータとして渡される
#                       コンパイラは呼び出し点で実引数値を抽出し、b に代入し、Positive(実引数) を検証
#                       例：divide(10, 2) → Positive(2) = { 2 > 0 } を検証 → True
#                       例：divide(10, 0) → Positive(0) = { 0 > 0 } を検証 → False → コンパイルエラー

# 事後条件：戻り値仮引数 + コンパイル時述語
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

NonEmpty: (arr: Array(T)) -> Type = { arr.len > 0 }

max: (T: Ord) -> ((arr: NonEmpty(arr))) -> (result: IsMax(T, arr, result)) = {
#                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                            result は戻り値仮引数、値は return で提供
#                                            コンパイラは return 点で戻り値を代入し、事後条件を検証
    candidate = arr[0]
    for i in 1..arr.len {
        if arr[i] > candidate { candidate = arr[i] }
    }
    return candidate
}
```

**重要なルール**：

- **引数側**：`b: Positive(b)` —— `b` は現在の仮引数名、`Positive` のパラメータとして渡される。関数呼び出し構文、暗黙のルールなし。
- **戻り値側**：`-> (result: IsMax(T, arr, result))` —— `result` は戻り値仮引数、値は `return` 文で提供される。`result` は型シグネチャ内にのみ存在し、述語によってのみ参照され、関数本体スコープには入らず、呼び出し側にも現れない。
- **戻り値仮引数はオプション**：事後条件がない場合は書かず、シグネチャは通常関数と完全に一致する（`-> Int`）。
- **統一性**：引数と戻り値仮引数は同じ概念——`仮引数名: 述語呼び出し(仮引数名)`——違いは値が呼び出し側から提供されるか `return` から提供されるかのみ。

### 3. パス条件伝播：ランタイム値のコンパイル時検証

コンパイル時述語がバインディング位置で使用されるとき、パラメータはプログラマが明示的に渡す。ランタイム値が精緻化型パラメータに入るとき、コンパイラはパス条件の収集と SMT 含意判定を通じて検証を完了する——プログラマが明示的に証明を渡す必要はない。

#### 3.1 明示的関数呼び出し

コンパイル時述語がバインディング位置で使用されるとき、パラメータはプログラマが明示的に渡す——単なる関数呼び出し、暗黙のルールはない。

`Positive: (x: Int) -> Type = { x > 0 }` はコンパイル時述語構築子である。それがバインディング位置（引数宣言、変数宣言、戻り型）に現れるとき、プログラマは既にバインドされた変数名を明示的に渡す：

```yaoxiang
b: Positive(b)
// b は現在の仮引数として宣言済み、Positive(b) は関数呼び出し
// 正格化後：b: { b > 0 }
```

コンパイラが暗黙的にパラメータを埋める必要はない——`b: Positive(b)` は `f(5)` と同じ、関数呼び出しである。`b` はパラメータ名としてバインドされ、その型注釈 `Positive(b)` は `b` 自身を参照する——これは依存型の標準パターンであり、暗黙の展開ルールではない。

**RFC-010 の `self` との統一**：RFC-010 は `self` がキーワードではなく単なるパラメータ名の慣習であることを確立した（"`p`、`this`、`x` と書いても効果は完全に同じ"）。`b: Positive(b)` は同じメカニズムを共有する——パラメータ名は型注釈内で参照可能。`self` は `self: Point` の位置に現れ、`b` は `b: Positive(b)` の位置に現れ、両者の型注釈はパラメータ自身を参照する。違いは型注釈の複雑さのみであり、メカニズムは完全に同じ——名前がバインドされた後、型はこの名前に依存できる。

戻り型も明示的な関数呼び出しを使用する：

```yaoxiang
Sorted: (arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (arr: Array(T)) -> (result: Sorted(result)) = { ... }
//                        ^^^^^^^^^^^^^^^^^^^^^^^
//                        result は戻り値仮引数、Sorted(result) は関数呼び出し
//                        コンパイラは return 点で戻り値を result に代入し、Sorted(戻り値) を検証
```

局所変数宣言にも同様に適用される：

```yaoxiang
let x: Positive(x) = 5
// x は 5 としてバインド、Positive(5) → { 5 > 0 } → True → 通過

// let y: Positive(y) = 0
// y は 0 としてバインド、Positive(0) → { 0 > 0 } → False → コンパイルエラー
```

#### 3.2 パス条件収集

ランタイム値が条件分岐に現れるとき、コンパイラは自動的にパス条件を収集し、現在のスコープの**仮定集合**を形成する。これらの仮定はコンパイル時 Bool 評価の背景知識として検証に参加する。

```yaoxiang
if y > 0 {
    // コンパイラはこの分岐内で自動的に仮定を取得：{ y > 0 }
    let result = divide(x, y)
    // 検証条件：(y > 0) ⇒ (y > 0)
    // 証明パイプラインは含意が成立すると判定 → Proved
} else {
    // この分岐の仮定：{ !(y > 0) }
    // divide(x, y) を呼び出す場合、検証条件は !(y > 0) ⇒ y > 0
    // 証明パイプラインは含意しないと判定 → Disproved
}
```

これはコンパイラがハードコードされた特殊パターンを書くのではなく、コンパイル時証明パイプラインの自然な動作である。型検査呼び出し点が毎回パイプラインに送信する：

```
{背景仮定} ⇒ {検証目標}
```

証明パイプラインは含意性を判定する。Proved → 通過、Disproved → コンパイルエラー + 反例、Unproven → コンパイルエラー + 未解決命題。背景仮定は現在のプログラム点のパス条件から来る。

#### 3.3 仮定スタック

コンパイラは制御フロー解析において、各基本ブロックに対して仮定集合を維持する：

- **if ガード**：`if y > 0` → true 分岐に `y > 0` をプッシュ、false 分岐に `!(y > 0)` をプッシュ（else を使用する場合）
- **match パターン**：`if let Some(v) = opt` → 分岐内に `opt == Some(v)` をプッシュ
- **論理結合**：`if x > 0 && y < 10` → 分岐内に `x > 0` と `y < 10` をプッシュ
- **関数事前条件**：`divide(a, b)` を呼び出すとき、`b` が `Positive` の証拠を持つことは、現在の仮定から来るか、実引数自身の精緻化型注釈から来る（`b` が既に `Positive` として注釈付けされていれば、その型は `b > 0` を運ぶ）
- **代入**：`let z = y` のとき、`y` 上の既存の精緻化条件が `z` に伝播する

すべての仮定がコンパイル時証明パイプラインに入る。SMT 加速パスに入るとき、SMT-LIB 背景アサーションに翻訳される。

#### 3.4 静的証拠なしならコンパイルエラー

プログラマが直接以下のように書いた場合：

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

現在のプログラム点に `y > 0` の仮定がなく、実引数 `y` 自身にも `Positive` 型注釈がない。検証条件は：

```
{} ⇒ { y > 0 }
```

パイプラインが `Disproved`（含意しない）を返す → コンパイルエラー：

> `divide` 呼び出しにおけるパラメータ `b` が `Positive` を満たすことを証明できません。
> `y` は関数入力から来ており、証明済みの境界がありません。
> if 分岐ガードを用いて呼び出しを保護することを検討してください：`if y > 0 { divide(x, y) }`。

YaoXiang は、静的証拠を提供せずにランタイム値が精緻化型パラメータに直接入ることを許可しない。これは制限ではない——これはハード安全哲学の中核である。コンパイラが静的に証明できないコードは、コンパイルを通過することはできない。

#### 3.5 統一パイプラインとの関係

パス条件伝播は追加のメカニズムではない。それはコンパイル時証明パイプラインの制御フロー解析における直接的な拡張である：

| 段階 | 責任 |
|------|------|
| パス条件収集 | コンパイラの制御フロー解析段階、各基本ブロックに仮定集合を注釈付け |
| 検証条件生成 | 検証が必要な型制約に遭遇したとき、パス条件 + 実引数型情報をマージ |
| 証明パイプライン評価 | コンパイラカーネル → SMT 加速 → Proved / Disproved / Unproven を得る |
| 結果 | `Proved` → 通過；`Disproved` → コンパイルエラー + 反例；`Unproven` → コンパイルエラー + 未解決命題（プログラマが証明関数を提供可能） |

新しい部品はない。特殊ルールもない。パス条件は証明パイプラインの背景知識に過ぎない——型等式、借用制約と同じパイプライン、同じ予算システムを共有する。

### 4. コンパイル時証明パイプライン

すべてのコンパイル時検査が単一のパイプラインを共有する。パイプラインの核心操作は**型検査**である——証明項の型が証明すべき命題と等しいかを検査する。すべてが型検査である。

```
コンパイル時が Bool 式の評価を必要とする（すなわち：証明項を構築する必要がある）
        │
        ├── 型等式（T1 == T2）
        │   → コンパイラが直接判定（構造等価）
        │
        ├── トークン衝突条件（!conflicting(tokens)）
        │   → フロー敏感生存解析（Duplicated/Linear 属性追跡）
        │
        ├── 依存型簡約（n + m を簡約）
        │   → コンパイル時項書き換えシステム（βδι-簡約）
        │
        ├── コンパイル時述語（x > 0, forall...）
        │   → コンパイラ自身 + SMT 加速モジュール
        │
        └── ホーア論理含意（P ⇒ Q）
            → コンパイラ + SMT 加速モジュール
                    │
                    ▼
             ┌──────────┐
             │ Proved   │  → コンパイル通過
             │ Disproved│  → コンパイルエラー + 反例
             │ Unproven │  → コンパイルエラー + 未解決命題
             └────┬─────┘
                  │
                  ▼
         プログラマが証明関数を記述（YaoXiang コード）
                  │
                  ▼
         型検査器が検証 ──→ Proved ──→ コンパイル通過
                  │
                  ▼
            検証失敗 → コンパイルエラー：「証明が成立しない」
```

#### 4.1 証明結果：三値代数

コンパイル時評価は三つの結果を返す——これは停止性問題の必然的な結論であり、証明論の自然な区分でもある：

```
eval_compile_time : BoolExpr → Proved | Disproved(Model) | Unproven
```

- **Proved** → 停止、証明項が構築済み、型検査通過。コンパイル続行。
- **Disproved(M)** → 停止、反例 M が存在。コンパイルエラー + 反例 + ソース位置。
- **Unproven** → 与えられたリソース上限内で証明が構築されなかった。コンパイルエラー + 未解決命題 + 予算消費レポート。

**Unproven ≠ False。** コンパイラが「証明できない」と言うのは命題が偽であることを意味しない——現在の自動証明能力を超えているだけである。これは誠実さであり、欠陥ではない。

予算のハードリミットは停止性問題の工学的解決である。knob は与えない——与えると「あなたのプログラムは停止すると思いますか」とユーザーに尋ねるのと同じであり、ユーザーもコンパイラも知らない。

#### 4.2 Unproven の後：プログラマが証明を書く

コンパイラが Unproven を返したとき、プログラマは**証明関数**を書くことができる——YaoXiang 関数であり、その戻り型は証明すべき命題と等しい。型検査器は、この関数を検証する——`add(a, b): Int` を検証するのとまったく同じメカニズムで。

```
命題  = 型
証明  = プログラム（その型の値）
検証  = 型検査（唯一の信頼の根）
```

SMT ソルバーは独立した信頼境界ではない——それは**型検査器の加速モジュール**である。SMT は証明を見つけるのを手伝うが、証明を検証するのは常に型検査器である。SMT が `unsat` を返すとき、コンパイラはその結果を型検査器が検証可能な証明項に再構築する。再構築が失敗した場合（SMT の推論ステップがコンパイラカーネルの推論規則を超える）、Unproven にフォールバックする——プログラマは手動で証明関数を書くことができる。

```yaoxiang
# 命題：コンパイラが自動証明できない精緻化属性
FirstIsMin: (T: Ord, arr: Sorted(T)) -> Type = {
    forall i in 0..arr.len: arr[0] <= arr[i]
}

# 証明：プログラマが関数を書き、戻り型は上記の命題
# 型検査器はこの関数を検証する——add(a,b): Int の検証と完全に一致
first_is_min: (T: Ord, arr: Sorted(T)) -> FirstIsMin(T, arr) = {
    # コンパイラはここで検証：関数本体の型 = FirstIsMin(T, arr)
    ...
}
```

AI も不要、Coq へのエクスポートも不要、新しい概念も不要。**コンパイル時に自動証明できない属性 → プログラマが YaoXiang コードで証明を書く → 型検査器が検証。** プロセス全体はスムーズなグラデーションである——コンパイラが簡単な証明を代行し、頭は難しい方に残す。

#### 4.3 パイプライン内の階層的依存

上記評価器は同じインターフェースを共有するが、評価順序が存在する。型等式は後続解析すべての前提条件；所有権/トークン検査は型情報に依存；精緻化述語検証は前二層の結果に依存。コンパイラは層ごとに評価し、基盤層で失敗した式は上位層に進まない——型エラーのあるプログラムに求解予算を浪費することを避ける。

```
評価順序（同一パイプライン、階層的スケジューリング）
├── 第 0 層：型等式（T1 == T2）
│   └── 構造単一化 → 失敗なら後続は無意味、直接 Disproved を返す
├── 第 1 層：所有権/トークン衝突
│   └── フロー敏感生存解析 → 失敗ならメモリ安全性が成立しない、直接 Disproved を返す
└── 第 2 層：精緻化述語/ホーア含意
    └── コンパイラ自身 → SMT 加速 → Proved / Disproved / Unproven を得る
```

各層は依然として `Proved/Disproved/Unproven` を返し、同じインターフェースと同じ予算システムを共有する。

### 5. 三層関数の統一

| 階層 | 実行タイミング | 入力 | 出力 | 例 |
|------|----------|------|------|------|
| 値レベル関数 | ランタイム | 値 | 値 | `add: (a: Int, b: Int) -> Int = a + b` |
| 型構築子 | コンパイル時 | 型/値 | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| コンパイル時述語 | コンパイル時 | 値 | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

すべて同じ `name: type = value` 構文を使用。コンパイル時述語と型構築子は同じコンパイル時証明パイプラインを通る——`{}` は証明空間である。

### 6. ループ：Floyd-Hoare 検証条件生成

ループは個別の `: Invariant(...)` や `: decreases(...)` 注釈を必要としない。変数上のコンパイル時述語型注釈が Floyd-Hoare 風のアサーションを定義する——コンパイラは型注釈から検証条件を生成し、証明パイプラインが各代入が型を保持するかをチェックする。

核心メカニズム：各代入操作は Hoare 三つ組 `{P} x := e {Q}` に対応し、検証条件は `P ⇒ Q[e/x]` である。コンパイラはループ本体に対して検証条件を一度生成する——証明パイプラインが帰納ステップを検証した後、すべての反復が自動的にカバーされる。

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # 注釈は i を参照；初期化時 i=0、検証：0 == sum(arr[0..0]) → True
    mut i: UpTo(arr.len) = 0     # 検証：0 <= 0 <= arr.len → True
    while i < arr.len {
        # コンパイラはループ本体に対して VC を一度生成。前提：s は SumUpTo(arr, i) を満たし、i は UpTo(arr.len) を満たす。
        #
        # s += arr[i]：
        #   検証義務：s_new は SumUpTo(arr, i) を満たす（現在 i は不変）
        #   s_new = s_old + arr[i] を代入：
        #     s_old + arr[i] == sum(arr[0..i+1]) が必要
        #     帰納仮説 s_old == sum(arr[0..i]) から、両辺に arr[i] を加える：
        #     sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   コンパイラ + SMT：線形算術、ミリ秒単位 → Proved
        #
        # i += 1：
        #   i 変更 → 依存図で s の型注釈が i を参照 → 再検証をトリガ
        #   新しい検証目標：s は SumUpTo(arr, i_new) を満たす
        #   すなわち s == sum(arr[0..i_new])、上一步で保証済み → Proved
        s += arr[i]
        i += 1
    }
    return s  # このとき s: SumUpTo(arr, arr.len)、すなわち s == sum(arr[0..arr.len])
}
```

ループ不変条件は変数上の型注釈である——プログラマが型を書き、コンパイラが帰納ステップを検査する。コンパイラは不変条件を「発見」する必要も、帰納を「自動実行」する必要もない——帰納証明を各代入操作の局所検証条件に分解し、証明パイプラインに分割して征服を委ねる。

#### 6.1 依存追跡：可変変数上の依存型

上記メカニズムの前提は、コンパイラが `s` の型注釈 `SumUpTo(arr, i)` が `i` を参照していることを知っていることである——`i` が変更されるとき、`s` の型制約もそれに応じて変化する。これにはコンパイラが**変数間の型依存グラフ**を維持する必要がある。

**データ構造**：

```
TypeDepGraph: Map<VarName, Set<VarName>>
# キーは依存される変数、値は型注釈内でその変数を参照している変数の集合
# 例：{ i: {s}, j: {s, t}, ... }
```

**構築**：型検査器が `mut v: Pred(... x ...) = init` を処理するとき、`Pred(...)` パラメータ内の自由変数参照を解析する。パラメータが現在のスコープ内の他の可変変数 `x` を参照している場合、依存図に `x → v` を記録する。

**トリガ**：依存される変数 `x` が代入されるとき、コンパイラは：
1. 依存図で `x` に依存するすべての変数 `{v₁, v₂, ...}` を検索
2. 各 `v` について、検証条件を生成：`v` の現在値が更新された型 `Pred(... x_new ...)` を満たすか
3. VC を証明パイプラインに送る

**代入順序感受性**：依存追跡は正しい代入順序を自然に強制する。`SumUpTo(arr, i)` を例に取ると：

```yaoxiang
# 正しい順序
s += arr[i]   # s_new は SumUpTo(arr, i+1) を満たす
i += 1        # i 変更 → s が SumUpTo(arr, i_new) を満たすか再検証 → True

# 誤った順序——コンパイラが拒否
i += 1        # i 変更 → s が SumUpTo(arr, i_new) を満たすか再検証
              # s はまだ更新されておらず、s_old == sum(arr[0..i_old]) ≠ sum(arr[0..i_new])
              # → コンパイルエラー：変数 s が型 SumUpTo(arr, i_new) を満たさない
s += arr[i]   # 到達不能
```

**組み合わせ依存**：一つの変数が複数の変数に依存できる。型注釈 `{ v: Int; v == x + y }` は `x` と `y` の両方に同時に依存する——いずれの変更も再検証をトリガする。

**証明パイプラインとの関係**：依存追跡は VC 生成のトリガであり、独立した検証メカニズムではない。それは「いつ VC を生成する必要があるか」に答え、証明パイプラインが「VC が成立するか」に答える。

### 7. 停止性検査

コンパイル時に完全自動。コンパイラが証明できるループは通過し、証明できないものは直接コンパイルエラーを報告する——プログラマはループの停止性をコンパイラが自動解析できるようにしなければならない。半自動注釈のための余地は与えない。

#### 6.1 設計原則

コンパイラは二箇所から停止性証明に必要な情報を自動抽出する：

1. **変数の型注釈**：精緻化型内の境界制約（例：`UpTo(n)` は上界 `n` と下界 `0` を与える）
2. **ループ本体の操作**：各反復で変数に加えられる操作

コンパイラは四つの度量合成戦略を優先順位に従って試行し、見つかれば停止する。

#### 6.2 戦略 1：線形階数関数の自動合成

変数が線形境界注釈を持つ場合、コンパイラは候補線形度量を列挙し SMT で検証する。

```
入力：
  変数 v₁: UpTo(u₁), v₂: UpTo(u₂), ...（境界を持つ変数）
  ループ条件 cond
  ループ本体内の代入集合

アルゴリズム：
  1. 型注釈から各変数の境界を抽出：[low_i, high_i]
  2. 候補度量を列挙：v_i, u_i - v_i, v_i - v_j 等の線形結合
  3. 各候補度量 m について：
     - SMT で m ≥ 0 を検証（型境界から導出）
     - ループ本体の各実行パスについて、SMT で m' < m を検証（厳密に減少）
  4. 条件を満たす線形結合を発見 → 停止性を証明
```

カバー範囲：任意の変数が線形式（`v = a·v + b`）に代入され、有界型注釈を持つループを含む。`i += const`、`i -= const`、および二分探索のような区間収縮を含む：

```yaoxiang
# 二分探索：low = mid + 1 または high = mid
# 度量 high - low は二つのパスで厳密に減少
binary_search: (arr: Sorted(Int, arr), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 戦略 2：述語違反カウント——目標型から度量を自動抽出 <span style="color:orange">【実験的戦略】</span>

> ⚠️ **現在の状態：実験的戦略、フェーズ 3 実装時に実際の実現可能性次第で含めるかを決定。**
> この戦略は隣接交換操作（バブルソート、挿入ソート）に有効であり、非隣接操作（クイックソートのパーティション、ヒープソートの sift-down）は自動証明できない。
> カバー境界は以下の表を参照。フェーズ 3 で実現不可能と判明した場合、この戦略は削除されるか将来作業に格下げされる。

核心的な洞察：**ユーザーが書いた仕様がコンパイラの推論材料である。** コンパイラは「ソートとは何か」を内蔵する必要がない——`Sorted` の定義を読み、定義から自動的に度量を抽出する。

```
入力：
  目標型：Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  ループ本体の操作：隣接要素の交換

アルゴリズム：
  1. 述語定義を解析：forall i in range: cond(i, arr)
  2. 度量を自動生成：violation_count = |{ i | ¬cond(i, arr) }|
  3. 度量への操作の影響を分析：
     - 隣接交換 arr[j], arr[j+1] = arr[j+1], arr[j]
     - インデックス j-1, j, j+1 の三対のみに影響
     - arr[j] > arr[j+1] の場合（述語違反）、交換後この対は述語を満たす
     - violation_count は少なくとも 1 減少
  4. 上界：n·(n-1)/2（最大隣接反転数）、下界：0
  → 停止性を証明
```

**現在のカバー範囲**：

| アルゴリズム | 操作パターン | 戦略 2 で証明可能？ | 理由 |
|------|----------|:---:|------|
| バブルソート | 隣接交換 | ✅ | violation_count は交換ごとに厳密に減少 |
| 挿入ソート | 隣接移動 | ✅ | 各シフトが違反対を一つ解消 |
| 選択ソート | 非隣接交換 | ❌ | 単一交換で violation_count が増加し得る |
| クイックソート | パーティション分割 | ❌ | 非隣接交換、単調減少を保証しない |
| ヒープソート | sift-down | ❌ | ツリー操作、violation_count は非単調 |

**補完的戦略**：クイックソートの場合、`low < high` 区間収縮は戦略 1（線形階数関数）でカバー可能——外側のパーティション再帰で、毎回区間が半分になる。戦略 1 と戦略 2 は補完的にカバーし、多くの実際のアルゴリズムの停止性は両者のいずれかで証明可能である。しかし戦略 2 の汎用化（非隣接操作、ツリー操作）は依然として未解決問題である。

```yaoxiang
sort: (arr: Array(Int)) -> (result: Sorted(result)) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 戦略 3：有界増減パターン

`v += const`（正の定数）、変数が上界型注釈を持つ → 度量 `upper_bound - v` が毎回 `const` ずつ減少し、下界は 0。これは戦略 1 の縮退ケースであり、コンパイラは最初に高速処理する。

#### 6.5 戦略 4：乗法スケーリング度量テンプレート

`v *= const`（const > 1）、変数が上下界型注釈を持つ。コンパイラは対数目量テンプレート `ceil(log_const(upper/v))` を内蔵し、毎回 const を掛けると度量が 1 減少する。

```yaoxiang
mut i: Positive(i) = 1
while i < n {
    # コンパイラが自動導出：度量 ceil(log₂(n/i))、毎回 2 を掛けると度量が 1 減少
    i *= 2
}
```

#### 6.6 停止性と正当性の分離

停止性証明と正当性証明は独立している：
- **停止性**：上記四つの戦略がループが有限ステップで終了することを自動証明
- **正当性**：ループ本体が目標型に向かって進むかは、コンパイル時証明パイプラインが検証条件を通じてチェック

両方が通過 → コンパイル通過。停止性は証明されたが正当性が失敗 → コンパイルエラー + 反例。正当性は証明されたが停止性が証明できない → コンパイルエラーを報告し、解析不能な変数または操作を示す。両方失敗 → コンパイルエラーを報告し、二つの失敗理由をそれぞれ示す。

#### 6.7 再帰関数の停止性検査

コンパイル時評価が必要な再帰関数について、コンパイラは引数の減少をチェックする：

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # コンパイラ解析：n-1 < n → 減少 → 停止
}

# コンパイル時使用——コンパイラは factorial のコンパイル時停止を保証
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120、コンパイル時に完了
```

| シナリオ | 動作 |
|------|------|
| コンパイラが再帰的減少を解析可能（例：`n-1`） | コンパイル時評価 |
| 減少しない/減少の判定不能 | コンパイルエラー |
| ランタイム呼び出し（型位置でない） | 停止性検査不要 |

#### 6.8 ハード境界

`i = f(i)` かつ `f` が可逆でなく、閉じておらず、単調性を保たない——数学的に自動停止証明は不可能。コンパイルエラー：

> このループの自動停止証明ができません。ループ変数が解析不能な関数 `f` に依存しています。コンパイラが解析可能な反復パターンを使用してください。

これはコンパイラの失敗ではない。静的に安全性を証明できないコードは、コンパイルを通過することはできない。

### 8. SMT ソルバー：型検査器の加速モジュール

SMT ソルバーは従来の言語では外部ツール（F\* は Z3 を呼び、Dafny は Z3 を呼ぶ）である。YaoXiang では、それは**型検査器の一つの加速モジュール**である——コンパイラカーネル自身が直接判定できない場合にのみ呼び出される。SMT は証明を見つけるのを手伝うが、証明を検証するのは型検査器である。

**信頼モデル**：型検査器は唯一の信頼の根である。SMT ソルバーは加速モジュール——証明を見つけるのを手伝うが、SMT は独立した信頼境界ではない。コンパイラは Z3 の `unsat` 結果を信頼する（F\*/Dafny 路線と一致——Z3 が誤る確率はコンパイラ自身のバグ率より低く、これは工学上の実務的選択である）。真の信頼性の制御不能性は SMT 翻訳層にある——翻訳にバグがあれば、コンパイラは他のテストで露呈する。

**インターフェース**：コンパイラ内部で SMT-LIB 2.6 標準形式に翻訳し、特定のソルバー API に縛られない。SMT-LIB は ISO 標準であり、Z3、CVC5、MathSAT、Yices すべてネイティブサポートする。

**デフォルトバックエンド**：Z3（MIT ライセンス、最も広範なドキュメントとコミュニティ検証）。CVC5 は SMT-LIB 互換の代替——ユーザーはコンパイル時にコンパイラフラグで切り替え可能。

「汎用ソルバー抽象層」は作らない——SMT-LIB が抽象層である。将来 CVC5 が特定理論でブレークスルーを達成した場合、切り替えにはバイナリを差し替えるだけでよく、コンパイラコードの変更は不要。

```
コンパイル時 Bool 式
        │
        ├── コンパイラカーネルが直接判定可能（構造等価、簡単な算術、
        │   定数畳み込み後の自明な公式）
        │   → 直接 Proved / Disproved を返す
        │
        └── コンパイラカーネルが直接判定不能（量化子、記号変数）
            → 依存型事前簡約（factorial(5) → 120）
            → SMT-LIB 形式に翻訳
            → Z3/CVC5 に送信（予算制限付き）
            → 戻り値：unsat → Proved  │  sat + モデル → Disproved  │  unknown → Unproven
```

**求解予算——ハードリミット、スタック深度のように**：

| 予算次元 | デフォルト値 | 説明 |
|----------|--------|------|
| 求解ステップ | 10,000 | Z3 は線形算術に対して通常百ステップ以内。10,000 ステップで実際の述語の 99% をカバー。 |
| 時間 | 100ms | 単一述語が 100ms を超える = ユーザーがコンパイル時プログラムを書いている。100ms × 50 述語 = 5 秒コンパイル時間上限。 |
| 量化子インスタンス化深度 | 3 | 三層のネスト量化子が実際のパターンをカバー。三層を超えると論理学の練習問題を書いている可能性が高い。 |

予算超過は Unproven を返し、コンパイルエラー + 述語位置 + 消費量。降格なし、ランタイムチェックなし、silent pass なし。

**なぜこれが実際に機能するか**：工学上、実際の述語の 95% は線形算術である——`x > 0`、`arr.len > 0`、`0 <= idx < arr.len`——すべて決定可能断片内にあり、SMT ソルバーは这类の問題に対してミリ秒単位で返す。予算を超える複雑な述語の稀少なケースに遭遇した場合、プログラマが証明関数を書けばよい。

依存型は SMT 呼び出し前に事前簡約の一層を行う：`factorial(5)` は直接コンパイル時評価で `120` になり、`append([1,2], [3])` は直接評価で `[1,2,3]` になる。これらの確定的な値計算は SMT 予算を消費しない。

プログラマは SMT の存在を知る必要はない。メンタルモデルは：**コンパイラが証明できれば通過、できなければエラー——コンパイラが不得意な場合、関数で証明を見せることができる。**

### 9. コンパイル時述語の合成

コンパイル時述語は Type を返す関数であり、合成は関数合成を通じて自然に実現される：

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(arr)
}
```

### 10. コード例

#### 9.1 除算の安全性

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b

result = divide(10, 2)   # ✅ コンパイラが Positive(2) = { 2 > 0 } を検証 → True
# result = divide(10, 0)  # ❌ コンパイラが Positive(0) = { 0 > 0 } を検証 → False
```

#### 9.2 配列アクセスの安全性

```yaoxiang
InBounds: (idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (arr: Array(T), idx: InBounds(idx, arr)) -> T = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ コンパイラが InBounds(1, arr) = { 0 <= 1 && 1 < 3 } を検証 → True
# y = get(arr, 5)  # ❌ コンパイラが InBounds(5, arr) = { 0 <= 5 && 5 < 3 } を検証 → False
```

#### 9.3 ソートの正当性

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T))) -> (result: Sorted(T, result)) = {
    result = arr.clone()
    # ... ソートアルゴリズム実装 ...
    return result
}
```

#### 9.4 ループ：コンパイラの VC 生成

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```


### 11. dispatch ディスパッチパイプライン：コンパイル時とランタイムの統一ディスパッチ

`assert` と `Assert` は同じ精緻化型プリミティブの二面である。ディスパッチパイプライン `dispatch` は**述語の自由変数がコンパイル時にアクセス可能か**に基づいて、コンパイル時証明とランタイムチェックのいずれを行うかを自動決定する：

| 判定基準 | モード | 動作 |
|------|------|------|
| すべての自由変数がコンパイル時既知（ジェネリクスパラメータ、コンパイル時定数） | **CompileTime** | 証明パイプラインへ：Proved → 消去、Disproved → コンパイルエラー、Unknown → 証明を要求 |
| 自由変数がランタイムから来るものが存在（関数引数、外部入力、mut 変数） | **Runtime** | ランタイム check を挿入し、フロー敏感仮定集合 Γ に精緻化事実を注入 |

**核心**：「判定不能」≠「反証」。CompileTime モードの Unknown は証明を要求し（暗黙に降格せず）、Runtime モードでは命題がコンパイル時に真偽を持たない——prover がどれほど強くても「ユーザーが負数を入力したかもしれない」に恒真証明を書くことはできず、ランタイム check は唯一の sound な選択である。これは prover の能力不足ではなく、理論的必然である。

### 12. フロー敏感仮定集合 Γ ：最強力事後条件伝播

コンパイラはフロー敏感（flow-sensitive）仮定集合 Γ を維持し、各制御フロー点で既知の命題を追跡する。

**SP（最強力事後条件）伝播**：

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
y = x + 1           // Γ = {x > 0, y > 1}  ← SP 伝播
```

**mut 変数の kill セット**：`mut` 変数が再代入された後、その変数に関するすべての仮定を Γ から削除する：

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
mut x = x - 5       // Γ = {}  ← x > 0 が kill される
```

これは soundness のハード要求である——変数の値が変わったため、旧仮定は無効。

**分岐合流**：IF/ELSE または match 分岐が合流するとき、Γ は各分岐の仮定の積集合を取る。すべてのパスで成立する命題のみが分岐外に渡される。

### 13. 消去モデルの明確化：witness 消去 ≠ check 消去

RFC-027 の「精緻化型がランタイムに**完全に消去される**」という主張は、**proof witness（証明証人）**を指す——コンパイル時に検証済みの証明項はランタイムコードを生成しない。しかし Runtime モードで dispatch が挿入する**ランタイム check**は保持される——それは値層で実行される Bool チェックであり、型層の witness ではない。

まとめ：witness は消去、check は保持。二つの事柄は矛盾せず、RFC-027 の元の主張は変わらない。

## 詳細設計

### 構文変化

| 以前 (RFC-022) | 以後 (本 RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | コンパイル時述語を引数型として使用 `(b: Positive(b))` |
| `//! ensures: ExistsMax(result, arr)` | 戻り型に戻り値仮引数を使用 `-> (result: IsMax(T, arr, result))` |
| `/*! invariant: ... !*/` | 変数上のコンパイル時述語型注釈——Floyd-Hoare 不変条件 |
| `//! decreases: n` | コンパイラが度量関数を完全自動導出 |
| 仕様は注釈 | 仕様は型システム |

### 構文

**コンパイル時述語に新しいキーワードはない。** `{}` は証明空間であり、既存の型定義構文と完全に一致する。コンパイル時述語は Type を返す関数に他ならない——`name: (params) -> Type = { アサーション }`。使用時は単なる関数呼び出し——`Positive(b)`、`IsMax(T, arr, result)`。

```bnf
# コンパイル時述語 = Type を返す関数、{} 内はコンパイラが検証するアサーション
# 既存の関数/型構文を使用、新規 BNF ルール不要
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

**新規構文概念：戻り値仮引数**——`-> (name: Type)` において `name` は戻り値仮引数である。

戻り値仮引数は、YaoXiang が既存の関数構文に導入する**唯一の構文概念**である。その意味：
- `name` の値は `return` 文で提供される
- `name` は型シグネチャ内にのみ存在し、事後条件述語によって参照される（例：`-> (result: IsMax(T, arr, result))`）
- `name` は関数本体スコープに入らず、呼び出し側にも現れない
- 戻り値仮引数は**オプション**——事後条件がない場合、シグネチャは通常関数と完全に一致する（`-> Int`）、追加の負担を導入しない

導入理由：事後条件は「関数が返そうとしている値」を参照する必要がある。戻り値仮引数がない場合、コンパイラは述語が戻り値を参照できるようにするための特殊ルール（暗黙の変数 `$result` や `__retval__` など）に頼るしかない。戻り値仮引数はこの参照を明示化する——それは単なる仮引数であり、値が呼び出し側からではなく `return` から提供される点が異なる。

**証明関数**は新しい概念ではない——戻り型がアサートされる命題と等しい YaoXiang 関数に過ぎない。コンパイラが Unproven を返したとき、プログラマは証明関数を提供し、型検査器は任意の関数の戻り型を検証する方法とまったく同じ方法でそれを検証する。新しい構文、新しいキーワード、新しいルールは不要。

### 型システムへの影響

- **型宇宙**：コンパイル時述語は Type₂ 層に位置する——値を受け取り Type を返す関数、型構築子と同じ階層
- **ジェネリクスとの相互作用**：コンパイル時述語はジェネリクスパラメータを取れる、 例：`NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **所有権との相互作用**：コンパイル時述語内の式は所有権ルールに従い、読み取りのみ可能で書き込みは不可
- **型推論**：コンパイル時述語のパラメータは HM 型推論に参加する

### ランタイム表現

コンパイル時述語はランタイムには**dispatch ディスパッチ結果に従って処理される**：
- **CompileTime モード**（すべての自由変数がコンパイル時既知）：証明通過後、witness トークンは完全に消去される。`Positive: (x: Int) -> Type = { x > 0 }`——引数 `b: Positive(5)` のランタイム表現は単なる `Int`。精緻化条件 `{ 5 > 0 }` は既に通過済み、消去される。
- **Runtime モード**（ランタイム自由変数が存在）：ランタイム check を保持——値層で Bool チェックを実行し、フロー敏感仮定集合 Γ に注入。詳細は §11 dispatch ディスパッチパイプラインと §13 消去モデルの明確化を参照。

コンパイル時述語を型位置に置く（例：`f(x: Positive(x))`）ことはラッパー型を生成せず、追加メモリを割り当てない。ただし `x` がランタイム入力から来る場合、**ランタイム** Bool チェックが挿入される。

**`ref` との相互作用制約**：コンパイル時述語は不変借用または所有権が既に移動された値のみを参照できる。可変借用を参照するコンパイル時述語は、コンパイラがコンパイル時に検証結果がランタイムでも成立し続けることを保証できない——这类の使用法は直接コンパイルエラーを報告する。

### コンパイラ変更

1. **パーサー**：コンパイル時述語は標準関数構文を使用し、追加の解析ルール不要
2. **コンパイル時証明パイプライン**：Proved/Disproved/Unproven 戻りインターフェースの統一、自動戦略選択
3. **SMT 加速モジュール**：SMT-LIB 2.6 翻訳層、デフォルトバックエンド Z3、CVC5 代替
4. **型検査器カーネル**：推論規則の実装——構造等価、βδι-簡約、全称量化子の導入/除去。これが唯一の信頼の根であり、SMT とプログラマの証明はどちらもこれを通じて検証される
5. **検証条件生成**：WP/SP 計算 + ループ不変条件証明義務
6. **エラーレポート**：反例フォーマット + 未解決命題レポート + ソース位置関連付け

### 後方互換性

- ✅ コンパイル時述語を使用しないコードは完全に不変
- ✅ コンパイル時述語は CompileTime モードでランタイムオーバーヘッドがゼロ、Runtime モードでは必要な Bool チェックのみ保持
- ⚠️ RFC-022 の `//!` 構文はサポートされなくなる——ただし 022 は一度も実装されておらず、移行負担なし

## トレードオフ

### 利点

- **Curry-Howard 同型の完全実現**：型は命題、プログラムは証明、`name: 命題 = 証明`
- **統一性**：コンパイル時述語と通常関数は完全に同じ構文を使用し、概念分裂なし
- **SMT 透過性**：プログラマは SMT の存在を知る必要がなく、メンタルモデルは型検査と一致
- **漸進的採用**：一つのコンパイル時述語から始めて、カバレッジを徐々に拡大可能
- **最小ランタイムオーバーヘッド**：CompileTime モードはオーバーヘッドゼロ、Runtime モードは必要な Bool チェックのみ保持

### 欠点

- **コンパイル時間**：SMT 求解がコンパイル時間を増加させるが、予算ハードリミットが上限を制御可能に保証
- **自動証明の境界**：一階線形算術を超える複雑な述語はプログラマが証明関数を書く必要がある。これは言語の欠陥ではなく、停止性問題の必然的な結論である。コンパイラは Unproven を偽って True/False と報告せず、誠実に報告する
- **学習曲線**：効果的なコンパイル時述語と証明関数の記述には Curry-Howard 同型の基本直感の理解が必要
- **実装複雑性**：コンパイル時証明パイプラインの統一には慎重な設計が必要

### リスク緩和

- SMT 求解予算ハードリミット（ステップ 10,000 / 時間 100ms / インスタンス化深度 3）、予算超過は Unproven を返す
- 依存型事前簡約：確定的な値計算を先に処理し、SMT は非確定的部分のみに取り組む
- Unproven は行き止まりではない：プログラマは証明関数を記述可能、型検査器が検証——任意の関数の戻り型検証と一致
- 増分検証：変更モジュールのみ検証
- 明確なエラーメッセージ + 反例表示 + 予算消費レポート + 未解決命題 + 提案（コンパイラが提供可能な場合）

## 代替案

| 案                             | なぜ選ばないか                           |
| -------------------------------- | -------------------------------------- |
| RFC-022：`//!` 注釈式仕様        | 仕様と型が分裂、Curry-Howard 同型に違反 |
| 独立仕様ファイル（例：CVL）           | 仕様とコードが分離され、保守コストが増加             |
| ランタイムアサーションのみ                     | 静的に正当性を保証できない                     |
| 外部証明支援系（例：Coq）           | コンパイラから独立し、独立した証明言語と信頼境界が必要。YaoXiang の選択：証明は YaoXiang コード、型検査器が唯一の信頼の根 |
| **本方案：コンパイル時述語を第一級市民に** | ✅                                     |

## 未解決問題

- [x] **SMT ソルバーの選択**：デフォルト Z3（MIT ライセンス、最も広範に検証済み）。CVC5 は SMT-LIB 互換代替、コンパイラフラグで切り替え。コンパイラ内部翻訳ターゲットは SMT-LIB 2.6 標準形式——SMT-LIB が抽象層であり、カスタム汎用ソルバーインターフェースは作らない。
- [x] **求解予算の具体値**：ステップ 10,000 / 時間 100ms / 量化子インスタンス化深度 3。コンパイラ内部で固定、knob なし。実際の使用で「ユーザーが書き間違えた」ではなく本当に不足を証明する実例があれば調整。
- [x] **量化子のサポート範囲**：言語レベルでは量化子の階数を制限しない。コンパイル時述語は Type パラメータを取れる——Type は関数型を含む——したがって高階量化子は型システムの自然な帰結であり、特殊構文は不要。SMT ソルバーは一階量化子（forall/exists、交差ネストをサポート、予算深度 3 により制限）を自動判定可能。高階量化子：SMT は Unproven を返し、コンパイラは「この述語は自動証明の範囲外です、証明関数を提供してください」とプロンプト表示。プログラマは戻り型が当該命題と等しい YaoXiang 関数を記述する——型検査器がその関数を検証。外部エクスポート不要、AI 不要、対話的証明モード不要。すべてが YaoXiang コード、すべてが型検査器によって検証される。
- [x] **反例フォーマット**：ソース変数名を SMT 変数名として直接使用（モジュールプレフィックスを追加して衝突回避）。Z3 モデルが返されるとき、変数名で逆検索。出力形式：変数名 = 具体値 + ソース位置 + 述語定義位置。複雑なマッピング層は作らない。
- [x] ~~**`ref` スマートポインタとのコンパイル時述語の相互作用？**~~ → 決定済み：コンパイル時述語は不変借用または既に所有権が移動された値のみを許可する。可変借用の値はコンパイル時述語に出現できない。
- [x] **`forall` 述語違反カウント度量の非隣接操作への拡張？** → 拡張しない。現在のカバー範囲（隣接交換、隣接移動）は戦略 1（線形階数関数）で補完的にカバーされる——クイックソートの外側区間収縮は戦略 1 がカバー、ヒープソートは戦略 1（配列インデックスパターン）がカバー。どの戦略でも停止性を証明できないループは、コンパイラが直接エラーを報告する——これはハード安全哲学であり、欠陥ではない。将来、（学術的構成ではなく）実際のシナリオで四つの戦略すべてがカバーできないアルゴリズムがあれば、再議論する。
- [x] **線形階数関数の列挙組み合わせ爆発**：候補列挙上限は 3 つの有界変数。≤3 の場合、すべての線形結合を列挙し、SMT で逐一検証。>3 の場合、単一変数度量（`v_i`、`u_i - v_i`）のみ試行し、失敗したら直接コンパイルエラー——プログラマに「ループに 3 つを超える有界変数があり、コンパイラが多変量度量を自動合成できません」とプロンプト表示。これは工学的妥協ではなく、プログラマにより簡単なループを書くことを強制する。
