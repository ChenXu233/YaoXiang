# RFC-027：コンパイル時述語と統一静的検証

> **参考**:
>
> - [RFC-009: 所有権モデル](../accepted/009-ownership-model.md)
> - [RFC-010: 統一型構文 - name: type = value モデル](../accepted/010-unified-type-syntax.md)
> - [RFC-011: 泛型システム設計](../accepted/011-generic-type-system.md)
> - [RFC-024: spawn ブロックベースの並行モデル](../accepted/024-concurrency-model.md)
>
> **廃止**：[RFC-022: ホーア論理静的検証サポート（仕様注釈と仕様型）](../deprecated/022-hoare-logic-static-verification.md) — 廃止済み

## 概要

本文は YaoXiang に**コンパイル時述語**を一級市民として導入し、すべてのコンパイル時静的検証を一つの**証明パイプライン**に統一することを提案する。コンパイル時述語は外付けの仕様注釈ではない——それは関数である。Type を返す関数は型位置で使用可能であり、コンパイラはコンパイル時にそれを呼び出し、戻り値をチェックする。型は命題であり、コンパイル時評価は証明である。

**中核的主張**：型検査がコンパイル時に行う唯一の仕事は、証明項の構築と検証である。型等価、トークン衝突、依存型簡約、コンパイル時述語評価、ホーア論理含意——すべてはコンパイル時証明パイプライン内の異なる型検査であり、同一のパイプラインを共有する。SMT ソルバーは型検査器の加速モジュールであり、独立した信頼境界ではない。コンパイラが Unproven を返すとき、プログラマは YaoXiang 関数を証明として記述する——型検査器は任意の関数の戻り型を検証するのと同じ方法でそれを検証する。すべては YaoXiang コードであり、すべては型検査器によって検証される。

## 動機

### なぜ RFC-022 を廃止するのか？

RFC-022 は仕様を `//!` 注釈形式として設計していた：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← これは型と独立した注釈
    //! ensures: ExistsMax(result, arr[0..n])   ← これは型と独立した注釈
}
```

これは Curry-Howard 同型の根本的な誤りを犯している：**仕様と型を二層に分裂させる**。注釈は型ではない。注釈は型検査に参加しない。注釈は「外部ツール」のメンタルモデルである。

白書には明確に記されている：

> "`//!` 注釈はない。独立した仕様言語はない。すべては型システムの中にある。"

### 現在の問題

- RFC-022 の `//!` 注釈は型システムと独立した外付け構文である
- 仕様型と通常型は二体系であり、概念の冗長性が生じる
- Debug Build 検証 / Release Build 無視の分裂パターンが統一性を破壊する
- SMT ソルバーは従来の認識では外部ツールとして位置づけられる——YaoXiang では型検査器の加速モジュールとして内蔵する
- 型検査、借用検証、コンパイル時述語検査、マクロ展開はそれぞれ異なる経路を辿る

### 正しいメンタルモデル

型検査は次のように抽象化できる：

```
verify : Program → Proved | Disproved(Model) | Unproven
```

すべてのコンパイル時検査——単純な型マッチ、借用衝突検出、コンパイル時述語検証——はこの関数のサブタスクである。これらは同一の証明パイプラインを共有し、証明項の複雑さと構築戦略のみが異なる。

コンパイラが Unproven を返すとき、プログラマは証明関数を提供する——その関数の戻り型は証明すべき命題と等しい。型検査器がそれを検証する。これは通常の型検査と同じ操作である。

## 提案

### 1. `{}` は証明空間：型はすなわち表明、検証はすなわち型検査

YaoXiang の `{}` はコンパイル時証明空間である。その中にあるものすべてが表明であり、コンパイラは各項目が True であることを保証する——自動証明されるか、プログラマが証明関数を提供するかのいずれかである。

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  コンパイラは x が Float、y が Float であることを保証

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  コンパイラは data が Array(T) であることを保証
```

**泛型はコンパイル時述語の特例である。**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          パラメータはシグネチャで  {} 内には表明のみ
#          コンパイラはコンパイル時に呼び出し、x > 0 を検証

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      パラメータはシグネチャで   コンパイラは type_of(T) == Type、type_of(data) == Array(T) を検証
```

同一のパターン：`name: (params) -> Type = { 表明 }`。コンパイラは「型表明」と「値表明」を区別しない——どちらも証明パイプライン内の評価対象である。

**ループ不変条件は別途記述不要。変数上の型注釈が Floyd-Hoare 不変条件である。**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # 注釈は i を参照——コンパイラに s の型が i に依存することを伝える
    mut i: UpTo(arr.len) = 0     # 初期化時 i=0、検証：0 == sum(arr[0..0]) → True
    while i < arr.len {
        s += arr[i]  # コンパイラ検証：s_new == sum(arr[0..i+1])
        i += 1       # i 変更 → s 依存再検証をトリガー：s は SumUpTo(arr, i_new) を満たす
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

コンパイラはループ本体に対して一度検証条件を生成する——帰納仮説（型注釈）→ 代入操作 → 新しい値が型注釈を満たすか。証明パイプラインが帰納ステップの成立を検証した後、すべての反復が自動的にカバーされる。`: decreases` も、`: Invariant` も、帰納証明も不要——コンパイラは帰納を各代入の局所 VC に分解する。

### 2. 事前/事後条件：パラメータ型と戻り型上のコンパイル時述語

RFC-022 の `//! requires`/`//! ensures` を廃止する。コンパイル時述語はパラメータまたは戻りの型注釈として機能する。

**パラメータ側は関数呼び出し。** コンパイル時述語は Type を返す関数であり、パラメータ側の使用方法はそれを呼び出すこと——`factorial(5)` と同じである。戻り値側は新しい概念を導入する：戻り値仮パラメータ。

```yaoxiang
# 事前条件：パラメータ型で明示的にコンパイル時述語を呼び出す
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b
#                       ^^^^^^^^^^  b は現在の仮パラメータ名、Positive のパラメータとして渡される
#                       コンパイラは呼び出し点で実パラメータ値を抽出し、b に代入し、Positive(実パラメータ) を検証
#                       例：divide(10, 2) → Positive(2) = { 2 > 0 } の検証 → True
#                       例：divide(10, 0) → Positive(0) = { 0 > 0 } の検証 → False → コンパイルエラー

# 事後条件：戻り値仮パラメータ + コンパイル時述語
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

NonEmpty: (arr: Array(T)) -> Type = { arr.len > 0 }

max: (T: Ord) -> ((arr: NonEmpty(arr))) -> (result: IsMax(T, arr, result)) = {
#                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                            result は戻り値仮パラメータ、値は return によって提供される
#                                            コンパイラは return 点で戻り値を代入し、事後条件を検証
    candidate = arr[0]
    for i in 1..arr.len {
        if arr[i] > candidate { candidate = arr[i] }
    }
    return candidate
}
```

**主要規則**：

- **パラメータ側**：`b: Positive(b)` —— `b` は現在の仮パラメータ名、`Positive` のパラメータとして渡される。関数呼び出し構文、暗黙の要素ゼロ。
- **戻り値側**：`-> (result: IsMax(T, arr, result))` —— `result` は戻り値仮パラメータ、値は `return` 文によって提供される。`result` は型シグネチャ内にのみ存在し、述語によってのみ参照され、関数本体スコープには入らず、呼び出し側にも現れない。
- **戻り値仮パラメータはオプション**：事後条件がない場合は記述せず、シグネチャは通常関数と完全に一致する（`-> Int`）。
- **統一性**：パラメータと戻り値仮パラメータは同じ概念である——`仮パラメータ名: 述語呼び出し(仮パラメータ名)`、違いは値が呼び出し側から提供されるか `return` から提供されるかのみ。

### 3. パス条件伝播：ランタイム値のコンパイル時検証

コンパイル時述語が束縛位置で使用されるとき、パラメータはプログラマによって明示的に渡される。ランタイム値が精化型パラメータに入るとき、コンパイラはパス条件収集と SMT 含意判定を通じて検証を完了する——プログラマが明示的に証明を渡す必要はない。

#### 3.1 明示的関数呼び出し

コンパイル時述語が束縛位置で使用されるとき、パラメータはプログラマによって明示的に渡される——それは関数呼び出しであり、暗黙の要素はゼロ。

`Positive: (x: Int) -> Type = { x > 0 }` はコンパイル時述語コンストラクタである。束縛位置（パラメータ宣言、変数宣言、戻り型）に現れるとき、プログラマは束縛済みの変数名を明示的に渡す：

```yaoxiang
b: Positive(b)
// b は現在の仮パラメータとして宣言済み、Positive(b) は関数呼び出し
// 正格化後：b: { b > 0 }
```

コンパイラが暗黙的にパラメータを埋める必要はない——`b: Positive(b)` は `f(5)` と同じ、関数呼び出しである。`b` はパラメータ名として束縛され、その型注釈 `Positive(b)` は `b` 自身を参照する——これは依存型の標準パターンであり、暗黙の展開規則ではない。

**RFC-010 の `self` との統一**：RFC-010 は `self` がキーワードではなく、単なるパラメータ名の慣習であることを確立した（「`p`、`this`、`x` と書いても効果は完全に同じ」）。`b: Positive(b)` は同じメカニズムを共有する——パラメータ名は型注釈内で参照可能。`self` は `self: Point` の位置に現れ、`b` は `b: Positive(b)` の位置に現れる、両者の型注釈はパラメータ自身を参照する。違いは型注釈の複雑さのみであり、メカニズムは完全に同じ——名前が束縛された後、型はその名前に依存可能。

戻り型も同様に明示的関数呼び出しを使用する：

```yaoxiang
Sorted: (arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (arr: Array(T)) -> (result: Sorted(result)) = { ... }
//                        ^^^^^^^^^^^^^^^^^^^^^^^
//                        result は戻り値仮パラメータ、Sorted(result) は関数呼び出し
//                        コンパイラは return 点で戻り値を result に代入し、Sorted(戻り値) を検証
```

局所変数宣言にも同様に適用される：

```yaoxiang
let x: Positive(x) = 5
// x は 5 に束縛、Positive(5) → { 5 > 0 } → True → 通過

// let y: Positive(y) = 0
// y は 0 に束縛、Positive(0) → { 0 > 0 } → False → コンパイルエラー
```

#### 3.2 パス条件収集

ランタイム値が条件分岐に現れるとき、コンパイラは自動的にパス条件を収集し、現在のスコープの**仮定集合**を形成する。これらの仮定は、コンパイル時 Bool 評価の背景知識として検証に参加する。

```yaoxiang
if y > 0 {
    // コンパイラはこの分岐内で自動的に仮定を取得：{ y > 0 }
    let result = divide(x, y)
    // 検証条件：(y > 0) ⇒ (y > 0)
    // 証明パイプラインは含意成立を判定 → Proved
} else {
    // この分岐の仮定：{ !(y > 0) }
    // divide(x, y) を呼び出す場合、検証条件は !(y > 0) ⇒ y > 0
    // 証明パイプラインは含意不成立を判定 → Disproved
}
```

これはコンパイラが特別パターンをハードコードしているわけではない——これはコンパイル時証明パイプラインの自然な動作である。型検査呼び出し点は毎回パイプラインに以下を送信する：

```
{背景仮定} ⇒ {検証目標}
```

証明パイプラインは含意性を判定する。Proved → 通過、Disproved → コンパイルエラー + 反例、Unproven → コンパイルエラー + 未解決命題。背景仮定は現在のプログラム点のパス条件から来る。

#### 3.3 仮定スタック

コンパイラは制御流を解析する際、各基本ブロックに対して仮定集合を維持する：

- **if-ガード**：`if y > 0` → true 分岐は `y > 0` をプッシュ、false 分岐（else が使用される場合）は `!(y > 0)` をプッシュ
- **match パターン**：`if let Some(v) = opt` → 分岐内に `opt == Some(v)` をプッシュ
- **論理結合**：`if x > 0 && y < 10` → 分岐内に `x > 0` と `y < 10` をプッシュ
- **関数事前条件**：`divide(a, b)` を呼び出す際、`b` は現在の仮定からの `Positive` の証拠を満たさなければならない、または実引数自身の精化型注釈を満たす（`b` が `Positive` として注釈されている場合、その型は `b > 0` を運ぶ）
- **代入**：`let z = y` のとき、`y` 上の既存の精化条件が `z` に伝播する

すべての仮定はコンパイル時証明パイプラインに入る。SMT 加速パスに入るとき、SMT-LIB 背景表明に翻訳される。

#### 3.4 静的証拠なしはコンパイルエラー

プログラマが直接以下のように書いた場合：

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

現在のプログラム点には `y > 0` の仮定がなく、実引数 `y` 自体にも `Positive` 型注釈がない。検証条件は：

```
{} ⇒ { y > 0 }
```

パイプラインは `Disproved`（含意不成立）を返す → コンパイルエラー：

> `divide` 呼び出しでパラメータ `b` が `Positive` を満たすことを証明できない。
> `y` は関数入力から来ており、証明済みの境界がない。
> if 分岐ガードで呼び出すことを検討：`if y > 0 { divide(x, y) }`。

YaoXiang は静的証拠を提供せずにランタイム値が精化型パラメータに直接入ることを許容しない。これは制限ではない——これはハード安全哲学の中核である。コンパイラが静的に証明できないコードは、コンパイルを通してはならない。

#### 3.5 統一パイプラインとの関係

パス条件伝播は追加メカニズムではない。これは制御流解析におけるコンパイル時証明パイプラインの直接の拡張である：

| 段階 | 責務 |
|------|------|
| パス条件収集 | コンパイラ制御流解析段階、各基本ブロックに仮定セットを注釈 |
| 検証条件生成 | 検証が必要な型制約に遭遇したとき、パス条件 + 実引数型情報をマージ |
| 証明パイプライン評価 | コンパイラカーネル → SMT 加速 → Proved / Disproved / Unproven を導出 |
| 結果 | `Proved` → 通過；`Disproved` → コンパイルエラー + 反例；`Unproven` → コンパイルエラー + 未解決命題（プログラマは証明関数を提供可能） |

新しい部品はない。特別な規則もない。パス条件は証明パイプラインの背景知識である——型等価、借用制約と同一のパイプライン、同一の予算システムを共有する。

### 4. コンパイル時証明パイプライン

すべてのコンパイル時検査は同一のパイプラインを共有する。パイプラインの中核操作は**型検査**である——証明項の型が証明すべき命題と等しいかをチェックする。すべては型検査である。

```
コンパイル時に Bool 式の評価が必要（即ち：証明項を構築する必要がある）
        │
        ├── 型等価（T1 == T2）
        │   → コンパイラ直接判定（構造等価）
        │
        ├── トークン衝突条件（!conflicting(tokens)）
        │   → フロー依存活性分析（Dup/Linear 属性追跡）
        │
        ├── 依存型簡約（n + m 簡約）
        │   → コンパイル時項書き換えシステム（βδι-簡約）
        │
        ├── コンパイル時述語（x > 0, forall...）
        │   → コンパイラ自身 + SMT 加速モジュール
        │
        └── ホーア論理含意式（P ⇒ Q）
            → コンパイラ + SMT 加速モジュール
                    │
                    ▼
             ┌──────────┐
             │ Proved   │  → コンパイル通過
             │ Disproved│  → コンパイルエラー + 反例
             │ Unproven │  → コンパイルエラー + 未解決命題
             └────┬─────┘
                  │
                  ▼
         プログラマが証明関数を記述（YaoXiang コード）
                  │
                  ▼
         型検査器が検証 ──→ Proved ──→ コンパイル通過
                  │
                  ▼
            検証失敗 → コンパイルエラー：「証明が成立しない」
```

#### 4.1 証明結果：三値代数

コンパイル時評価は三つの結果を返す——これは停止問題の必然的な帰結であり、証明論の自然な分割である：

```
eval_compile_time : BoolExpr → Proved | Disproved(Model) | Unproven
```

- **Proved** → 停止、証明項が構築済み、型検査通過。コンパイル続行。
- **Disproved(M)** → 停止、反例 M が存在。コンパイルエラー + 反例 + ソース位置。
- **Unproven** → 与えられたリソース上限内で証明が構築されなかった。コンパイルエラー + 未解決命題 + 予算消費レポート。

**Unproven ≠ False。** コンパイラが「証明できない」と言うことは命題が偽であることと同義ではない——現在の自動証明能力を超えているだけである。これは正直さであり、欠陥ではない。

予算ハード制限は停止問題の工学的解決である。ノブを与えない——与えると「あなたのプログラムは停止すると思うか」とユーザーに尋ねることになり、ユーザーは知らないし、コンパイラも知らない。

#### 4.2 Unproven の後：プログラマが証明を記述

コンパイラが Unproven を返すとき、プログラマは**証明関数**を記述できる——すなわち、戻り型が証明すべき命題と等しい YaoXiang 関数である。型検査器はこの関数を検証する——`add(a, b): Int` を検証するのと同じメカニズムで。

```
命題  = 型
証明  = プログラム（その型の値）
検証  = 型検査（唯一の信頼ルート）
```

SMT ソルバーは独立した信頼境界ではない——それは**型検査器の加速モジュール**である。SMT は証明発見を助けるが、証明を検証するのは常に型検査器である。SMT が `unsat` を返すとき、コンパイラはその結果を型検査器が検証可能な証明項に再構成する。再構成が失敗した場合（SMT の推論ステップがコンパイラカーネルの推論規則を超える）、Unproven にフォールバックする——プログラマは手動で証明関数を記述可能。

```yaoxiang
# 命題：コンパイラが自動証明できない精化属性
FirstIsMin: (T: Ord, arr: Sorted(T)) -> Type = {
    forall i in 0..arr.len: arr[0] <= arr[i]
}

# 証明：プログラマが関数を記述、戻り型が上の命題
# 型検査器がこの関数を検証——add(a,b): Int の検証と完全に一致
first_is_min: (T: Ord, arr: Sorted(T)) -> FirstIsMin(T, arr) = {
    # コンパイラはここで検証：関数本体の型 = FirstIsMin(T, arr)
    ...
}
```

AI 不要、Coq へのエクスポート不要、新しい概念不要。**コンパイル時に自動証明できない属性 → プログラマが YaoXiang コードで証明を記述 → 型検査器が検証。** 全体プロセスはスムーズなグラデーションである——コンパイラが簡単な証明を行い、難しいものは頭を使う。

#### 4.3 パイプライン内の階層的依存

上記の評価器は同一インターフェースを共有するが、評価順序が存在する。型等価はすべての後続解析の前提条件である；所有権/トークン検査は型情報に依存する；精化述語検証は前 2 層の結果に依存する。コンパイラは層ごとに評価し、低層で失敗した式は上層に進まない——型エラーのプログラム上で求解予算を浪費することを避ける。

```
評価順序（同一パイプライン、階層的スケジューリング）
├── 第 0 層：型等価（T1 == T2）
│   └── 構造ユニフィケーション → 失敗なら後続無意味、直接 Disproved を返す
├── 第 1 層：所有権/トークン衝突
│   └── フロー依存活性分析 → 失敗ならメモリ安全性不成立、直接 Disproved を返す
└── 第 2 層：精化述語/ホーア含意
    └── コンパイラ自身 → SMT 加速 → Proved / Disproved / Unproven を導出
```

各層は依然として `Proved/Disproved/Unproven` を返し、同一インターフェースと同一予算システムを共有する。

### 5. 三層関数の統一

| 階層 | 実行タイミング | 入力 | 出力 | 例 |
|------|----------|------|------|------|
| 値レベル関数 | ランタイム | 値 | 値 | `add: (a: Int, b: Int) -> Int = a + b` |
| 型コンストラクタ | コンパイル時 | 型/値 | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| コンパイル時述語 | コンパイル時 | 値 | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

すべて同じ `name: type = value` 構文を使用。コンパイル時述語と型コンストラクタは同一のコンパイル時証明パイプラインを通る——`{}` は証明空間。

### 6. ループ：Floyd-Hoare 検証条件生成

ループは別途の `: Invariant(...)` や `: decreases(...)` 注釈を必要としない。変数上のコンパイル時述語型注釈が Floyd-Hoare スタイルのアサーションを定義する——コンパイラは型注釈から検証条件を生成し、証明パイプラインは各代入が型を保持するかをチェックする。

中核メカニズム：各代入操作は Hoare 三つ組 `{P} x := e {Q}` に対応し、検証条件は `P ⇒ Q[e/x]` である。コンパイラはループ本体に対して一度検証条件を生成する——証明パイプラインが帰納ステップの成立を検証した後、すべての反復が自動的にカバーされる。

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # 注釈は i を参照；初期化時 i=0、検証：0 == sum(arr[0..0]) → True
    mut i: UpTo(arr.len) = 0     # 検証：0 <= 0 <= arr.len → True
    while i < arr.len {
        # コンパイラはループ本体に対して一度 VC を生成。前提：s は SumUpTo(arr, i) を満たし、i は UpTo(arr.len) を満たす。
        #
        # s += arr[i]：
        #   検証義務：s_new は SumUpTo(arr, i) を満たす（現在の i は不変）
        #   s_new = s_old + arr[i] を代入：
        #     s_old + arr[i] == sum(arr[0..i+1]) が必要
        #     帰納仮定 s_old == sum(arr[0..i]) から、両辺に arr[i] を加える：
        #     sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   コンパイラ + SMT：線形算術、ミリ秒級 → Proved
        #
        # i += 1：
        #   i 変更 → 依存図で s の型注釈が i を参照 → 再検証をトリガー
        #   新しい検証目標：s は SumUpTo(arr, i_new) を満たす
        #   即ち s == sum(arr[0..i_new])、上一步で保証 → Proved
        s += arr[i]
        i += 1
    }
    return s  # この時 s: SumUpTo(arr, arr.len)、即ち s == sum(arr[0..arr.len])
}
```

ループ不変条件は変数上の型注釈である——プログラマが型を記述し、コンパイラが帰納ステップをチェックする。コンパイラは不変条件を「発見」する必要も、「自動的に帰納を行う」必要もない——帰納証明を各代入操作の局所検証条件に分解し、証明パイプラインに分業させる。

#### 6.1 依存追跡：可変変数上の依存型

上記のメカニズムの前提は：コンパイラが `s` の型注釈 `SumUpTo(arr, i)` が `i` を参照していることを知っている——`i` が変更されるとき、`s` の型制約もそれに従って変化する。これにはコンパイラが**変数間の型依存グラフ**を維持する必要がある。

**データ構造**：

```
TypeDepGraph: Map<VarName, Set<VarName>>
# キーは依存される変数、値は型注釈内でその変数を参照する変数の集合
# 例：{ i: {s}, j: {s, t}, ... }
```

**構築**：型検査器が `mut v: Pred(... x ...) = init` を処理するとき、`Pred(...)` パラメータ内の自由変数参照を解析する。パラメータが現在のスコープ内の他の可変変数 `x` を参照する場合、依存図に `x → v` を記録する。

**トリガー**：依存される変数 `x` が代入されるとき、コンパイラは：
1. 依存図で `x` に依存するすべての変数 `{v₁, v₂, ...}` を検索
2. 各 `v` について、検証条件を生成：`v の現在値が更新された型 Pred(... x_new ...) を満たすか`
3. VC を証明パイプラインに送る

**代入順序依存**：依存追跡は正しい代入順序を自然に強制する。`SumUpTo(arr, i)` を例に：

```yaoxiang
# 正しい順序
s += arr[i]   # s_new は SumUpTo(arr, i+1) を満たす
i += 1        # i 変更 → s が SumUpTo(arr, i_new) を満たすか再検証 → True

# 間違った順序——コンパイラが拒否
i += 1        # i 変更 → s が SumUpTo(arr, i_new) を満たすか再検証
              # s はまだ更新されていない、s_old == sum(arr[0..i_old]) ≠ sum(arr[0..i_new])
              # → コンパイルエラー：変数 s は型 SumUpTo(arr, i_new) を満たさない
s += arr[i]   # 到達不能
```

**組合せ依存**：一つの変数は複数の変数に依存可能。型注釈 `{ v: Int; v == x + y }` は同時に `x` と `y` に依存する——どちらの変更も再検証をトリガーする。

**証明パイプラインとの関係**：依存追跡は VC 生成のトリガーであり、独立した検証メカニズムではない。これは「いつ VC を生成する必要があるか」に答え、証明パイプラインは「VC が成立するか」に答える。

### 7. 停止性検査

コンパイル時全自動。コンパイラが証明できるループは通過し、証明できないものは直接コンパイルエラーを報告する——プログラマはコンパイラにループの停止性を自動解析させなければならない。半自動注釈の余地は与えない。

#### 6.1 設計原則

コンパイラは二箇所から停止性証明に必要な情報を自動抽出する：

1. **変数型注釈**：精化型内の境界制約（例：`UpTo(n)` は上界 `n` と下界 `0` を提供）
2. **ループ本体操作**：各反復が変数に加える操作

コンパイラは優先順位に従って四つの度量合成戦略を試し、見つかれば停止する。

#### 6.2 戦略 1：線形ランク関数の自動合成

変数に線形境界注釈がある場合、コンパイラは候補線形度量を列挙し SMT で検証する。

```
入力：
  変数 v₁: UpTo(u₁), v₂: UpTo(u₂), ...（境界を持つ変数）
  ループ条件 cond
  ループ本体内の代入集合

アルゴリズム：
  1. 型注釈から各変数の境界を抽出：[low_i, high_i]
  2. 候補度量を列挙：v_i, u_i - v_i, v_i - v_j, 等の線形結合
  3. 各候補度量 m について：
     - SMT で m ≥ 0 を検証（型境界から導出）
     - ループ本体の各実行パスについて、SMT で m' < m を検証（厳密減少）
  4. 条件を満たす線形結合を発見 → 停止性が証明される
```

カバー範囲：変数が線形式（`v = a·v + b`）に代入され、境界型注釈を持つ任意のループ。`i += const`、`i -= const`、および二分探索のような区間収縮を含む：

```yaoxiang
# 二分探索：low = mid + 1 または high = mid
# 度量 high - low は両方のパスで厳密減少
binary_search: (arr: Sorted(Int, arr), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 戦略 2：述語違反カウント——目標型から自動抽出度量 <span style="color:orange">【実験的戦略】</span>

> ⚠️ **現在の状態：実験的戦略、段階 3 実装時に実際の実現可能性に応じて含めるか決定。**
> この戦略は隣接交換操作（バブルソート、挿入ソート）に有効だが、非隣接操作（クイックソート partition、ヒープソート sift-down）は自動証明できない。
> カバレッジ境界は下表の通り。段階 3 で実現不可能と判明した場合、この戦略は削除されるか将来の作業に降格される。

中核洞察：**ユーザーが記述した仕様がコンパイラ推論の素材である。** コンパイラは「ソートとは何か」を内蔵する必要がない——`Sorted` の定義を読み、定義から自動的に度量を抽出する。

```
入力：
  目標型：Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  ループ本体操作：隣接要素の交換

アルゴリズム：
  1. 述語定義を解析：forall i in range: cond(i, arr)
  2. 度量を自動生成：violation_count = |{ i | ¬cond(i, arr) }|
  3. 操作が度量に与える影響を分析：
     - 隣接交換 arr[j], arr[j+1] = arr[j+1], arr[j]
     - インデックス j-1, j, j+1 の三対にのみ影響
     - arr[j] > arr[j+1]（述語違反）の場合、交換後この対は述語を満たす
     - violation_count は少なくとも 1 減少
  4. 上界：n·(n-1)/2（最大隣接逆序数）、下界：0
  → 停止性が証明される
```

**現在のカバレッジ範囲**：

| アルゴリズム | 操作モード | 戦略 2 で証明可能？ | 理由 |
|------|----------|:---:|------|
| バブルソート | 隣接交換 | ✅ | violation_count は交換ごとに厳密減少 |
| 挿入ソート | 隣接移動 | ✅ | 各シフトが一つの違反対を解消 |
| 選択ソート | 非隣接交換 | ❌ | 単一交換が violation_count を増加させる可能性 |
| クイックソート | partition 分割 | ❌ | 非隣接交換、単調減少を保証しない |
| ヒープソート | sift-down | ❌ | ツリー操作、violation_count は非単調 |

**補完的戦略**：クイックソートの場合、`low < high` 区間収縮は戦略 1（線形ランク関数）でカバー可能——外層 partition 再帰、各反復で区間が半減。戦略 1 と戦略 2 は補完的にカバーし、ほとんどの実際のアルゴリズムの停止性は両者のいずれかで証明可能。しかし戦略 2 の汎用化（非隣接操作、ツリー操作）は依然として未解決問題。

```yaoxiang
sort: (arr: Array(Int)) -> (result: Sorted(result)) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 戦略 3：有界増加/減少パターン

`v += const`（正の定数）、変数に上界型注釈あり → 度量 `upper_bound - v` は毎回 `const` 減少、下界 0。これは戦略 1 の縮退ケースであり、コンパイラは最初に高速処理する。

#### 6.5 戦略 4：乗法スケール度量テンプレート

`v *= const`（const > 1）、変数に上下界型注釈あり。コンパイラは対数度量テンプレート `ceil(log_const(upper/v))` を内蔵し、毎回 const を掛けると度量が 1 減少。

```yaoxiang
mut i: Positive(i) = 1
while i < n {
    # コンパイラ自動推導：度量 ceil(log₂(n/i))、毎回 2 倍で度量が 1 減少
    i *= 2
}
```

#### 6.6 停止性と正当性の分離

停止性証明と正当性証明は独立している：
- **停止性**：上記四つの戦略がループが有限ステップで終了することを自動証明
- **正当性**：ループ本体が目標型に向かって進むか、コンパイル時証明パイプラインが検証条件でチェック

両方が通過 → コンパイル通過。停止性は証明されたが正当性が失敗 → コンパイルエラー + 反例。正当性は証明されたが停止性を証明できない → コンパイルエラー + 分析できない変数または操作を指摘。両方失敗 → コンパイルエラーが二つの失敗理由を別々に報告。

#### 6.7 再帰関数の停止性検査

コンパイル時に評価が必要な再帰関数について、コンパイラはパラメータの減少をチェックする：

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # コンパイラ分析：n-1 < n → 減少 → 停止
}

# コンパイル時使用——コンパイラは factorial がコンパイル時に停止することを保証
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120、コンパイル時完了
```

| シナリオ | 動作 |
|------|------|
| コンパイラが再帰減少を分析可能（例：`n-1`） | コンパイル時評価 |
| 減少しない/減少判定不能 | コンパイルエラー |
| ランタイム呼び出し（型位置でない） | 停止性検査不要 |

#### 6.8 ハード境界

`i = f(i)` であり、`f` が非可逆、非閉、単調性を保たない——数学的に自動停止証明は不可能。コンパイルエラー：

> このループは自動停止証明ができない。ループ変数が分析不能な関数 `f` に依存する。コンパイラが分析可能な反復パターンを使用してください。

これはコンパイラの失敗ではない。静的に安全性を証明できないコードは、コンパイルを通してはならない。

### 8. SMT ソルバー：型検査器の加速モジュール

SMT ソルバーは従来の言語では外部ツールである（F\* が Z3 を呼び出す、Dafny が Z3 を呼び出す等）。YaoXiang では、**型検査器の一つの加速モジュール**である——コンパイラカーネル自身が直接判定できない場合のみ呼び出される。SMT は証明発見を助けるが、証明を検証するのは型検査器である。

**信頼モデル**：型検査器が唯一の信頼ルートである。SMT ソルバーは加速モジュールである——証明発見を助けるが、SMT は独立した信頼境界ではない。コンパイラは Z3 の `unsat` 結果を信頼する（F\*/Dafny 路線と一致——Z3 が誤る確率はコンパイラ自身のバグ率より低く、これは工学的実用的選択である）。真の信頼性の制御は SMT 翻訳層にある——翻訳にバグがあれば、コンパイラは他のテストで露呈する。

**インターフェース**：コンパイラ内部は SMT-LIB 2.6 標準形式に翻訳し、特定のソルバー API に束縛しない。SMT-LIB は ISO 標準であり、Z3、CVC5、MathSAT、Yices すべてネイティブサポート。

**デフォルトバックエンド**：Z3（MIT ライセンス、最も広範な文書化とコミュニティ検証）。CVC5 は SMT-LIB 互換の代替として——ユーザーはコンパイル時にコンパイラフラグで切り替え可能。

「汎用ソルバー抽象層」は作らない——SMT-LIB が抽象層である。将来 CVC5 が特定の理論で突破を達成した場合、切り替えはバイナリを交換するだけで、コンパイラコードの変更は不要。

```
コンパイル時 Bool 式
        │
        ├── コンパイラカーネルが直接判定可能（構造等価、単純算術、
        │   定数畳み込み後の自明な公式）
        │   → 直接 Proved / Disproved を返す
        │
        └── コンパイラカーネルが直接判定不能（量詞、記号変数）
            → 依存型事前簡約（factorial(5) → 120）
            → SMT-LIB 形式に翻訳
            → Z3/CVC5 に送信（予算制限付き）
            → 戻り値：unsat → Proved  │  sat + モデル → Disproved  │  unknown → Unproven
```

**求解予算——スタック深度のようなハード制限**：

| 予算次元 | デフォルト値 | 説明 |
|----------|--------|------|
| 求解ステップ数 | 10,000 | Z3 は線形算術に対して通常百ステップ以内。10,000 ステップで実際の述語の 99% をカバー。 |
| 時間 | 100ms | 単一述語が 100ms を超える = ユーザーが型注釈ではなくコンパイル時プログラムを記述している。100ms × 50 述語 = 5 秒のコンパイル時間上限。 |
| 量詞インスタンス化深度 | 3 | 三層のネスト量詞が実際のパターンをカバー。三層を超えると論理学の練習問題を記述している可能性が高い。 |

予算超過は Unproven を返し、コンパイルエラー + 述語位置 + 消費量。降格なし、ランタイムチェックなし、サイレントパスなし。

**なぜこれが実際に機能するか**：工学的に実際の述語の 95% は線形算術である——`x > 0`、`arr.len > 0`、`0 <= idx < arr.len`——すべて決定可能フラグメント内であり、SMT ソルバーは这类の問題に対してミリ秒単位で返す。予算を超える少数の複雑な述語に遭遇した場合、プログラマは証明関数を記述すればよい。

依存型は SMT 呼び出し前に一層の事前簡約を行う：`factorial(5)` は直接コンパイル時評価で `120`、`append([1,2], [3])` は直接評価で `[1,2,3]`。これらの確定的な値計算は SMT 予算を消費しない。

プログラマは SMT の存在を知る必要はない。メンタルモデルは：**コンパイラが証明できれば通過、できなければエラー報告——もしコンパイラができなければ、証明関数を記述して見せることができる。**

### 9. コンパイル時述語の組合せ

コンパイル時述語は Type を返す関数であり、組合せは関数合成で自然に実現される：

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(arr)
}
```

### 10. コード例

#### 9.1 除算安全性

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b

result = divide(10, 2)   # ✅ コンパイラ検証 Positive(2) = { 2 > 0 } → True
# result = divide(10, 0)  # ❌ コンパイラ検証 Positive(0) = { 0 > 0 } → False
```

#### 9.2 配列アクセス安全性

```yaoxiang
InBounds: (idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (arr: Array(T), idx: InBounds(idx, arr)) -> T = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ コンパイラ検証 InBounds(1, arr) = { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ コンパイラ検証 InBounds(5, arr) = { 0 <= 5 && 5 < 3 } → False
```

#### 9.3 ソート正当性

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T))) -> (result: Sorted(T, result)) = {
    result = arr.clone()
    # ... ソートアルゴリズム実装 ...
    return result
}
```

#### 9.4 ループ：コンパイラ VC 生成

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```


### 11. dispatch 分派パイプライン：コンパイル時とランタイムの統一分派

`assert` と `Assert` は同一の精化型プリミティブの二面である。分派パイプライン `dispatch` は**述語の自由変数がコンパイル時に到達可能か**によって自動的にコンパイル時証明とランタイムチェックのいずれかに分派する：

| 判準 | モード | 動作 |
|------|------|------|
| すべての自由変数がコンパイル時既知（泛型パラメータ、コンパイル時定数） | **CompileTime** | 証明パイプラインへ：Proved → 消去、Disproved → コンパイルエラー、Unknown → 証明を要求 |
| 自由変数がランタイムから来るものが存在（関数パラメータ、外部入力、mut 変数） | **Runtime** | ランタイム check を挿入し、フロー依存仮定セット Γ に精化事実を注入 |

**核心**：「判定不能」≠「反証」。CompileTime モードで Unknown は証明を要求する（静かに降格しない）、Runtime モードでは命題はコンパイル時にはそもそも真偽を持たない——prover がどれほど強くても「ユーザーが負数を入力したかもしれない」に対する恒真証明は書けない、ランタイム check が唯一の sound な選択である。これは prover の性能不足ではなく、理論的必然である。

### 12. フロー依存仮定セット Γ：最強後置条件伝播

コンパイラはフロー依存（flow-sensitive）仮定セット Γ を維持し、各制御流点で既知の成立命題を追跡する。

**SP（最強後置条件）伝播**：

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
y = x + 1           // Γ = {x > 0, y > 1}  ← SP 伝播
```

**mut 変数の kill set**：`mut` 変数が再代入された後、その変数に関するすべての仮定は Γ から削除される：

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
mut x = x - 5       // Γ = {}  ← x > 0 が kill される
```

これは soundness のハード要求である——変数の値が変わったので、旧仮定は無効。

**分岐合流**：IF/ELSE または match 分岐が合流するとき、Γ は各分岐仮定の交差を取る。すべてのパスで成立する命題のみが分岐外に引き継がれる。

### 13. 消去モデル明確化：witness 消去 ≠ check 消去

RFC-027 の「精化型はランタイムで**完全消去**」という主張は **proof witness（証明トークン）**を指す——コンパイル時に検証済みの証明項はランタイムコードを生成しない。しかし Runtime モードで dispatch が挿入する**ランタイム check**は保持される——それは値レベルで実行される Bool チェックであり、型レベルの witness ではない。

まとめ：witness は消去、check は保持。二つの事柄は衝突せず、RFC-027 の元の主張は変わらない。

## 詳細設計

### 構文変更

| 以前 (RFC-022) | 以後 (本 RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | コンパイル時述語がパラメータ型として `(b: Positive(b))` |
| `//! ensures: ExistsMax(result, arr)` | 戻り型が戻り値仮パラメータを使用 `-> (result: IsMax(T, arr, result))` |
| `/*! invariant: ... !*/` | 変数上のコンパイル時述語型注釈——Floyd-Hoare 不変条件 |
| `//! decreases: n` | コンパイラが度量関数を全自动導出 |
| 仕様は注釈 | 仕様は型システム |

### 構文

**コンパイル時述語に新しいキーワードはない。** `{}` は証明空間であり、既存の型定義構文と完全に一致する。コンパイル時述語は Type を返す関数である——`name: (params) -> Type = { 表明 }`。使用時は関数呼び出し——`Positive(b)`、`IsMax(T, arr, result)`。

```bnf
# コンパイル時述語 = Type を返す関数、{} 内はコンパイラ検証対象の表明
# 既存関数/型構文を使用、新規 BNF 規則不要
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

**新規構文概念：戻り値仮パラメータ**——`-> (name: Type)` の `name` は戻り値仮パラメータ。

戻り値仮パラメータは YaoXiang が既存関数構文上で導入する**唯一の構文概念**である。そのセマンティクス：
- `name` の値は `return` 文で提供される
- `name` は型シグネチャ内にのみ存在し、事後条件述語に参照される（例：`-> (result: IsMax(T, arr, result))`）
- `name` は関数本体スコープに入らず、呼び出し側にも現れない
- 戻り値仮パラメータは**オプション**——事後条件がない場合、シグネチャは通常関数と完全に一致する（`-> Int`）、追加負担なし

導入理由：事後条件は「関数が返そうとしている値」を参照する必要がある。戻り値仮パラメータがない場合、コンパイラは述語に戻り値を参照させるための特殊規則（暗黙変数 `$result` や `__retval__` 等）を使用せざるを得ない。戻り値仮パラメータはこの参照を明示化する——それは仮パラメータであり、値は呼び出し側ではなく `return` から提供される点が異なる。

**証明関数**は新しい概念ではない——それは戻り型が表明対象命題と等しい YaoXiang 関数である。コンパイラが Unproven を返すとき、プログラマは証明関数を提供し、型検査器は任意の関数の戻り型を検証するのと同じ方法でそれを検証する。新しい構文、新しいキーワード、新しい規則不要。

### 型システムへの影響

- **型宇宙**：コンパイル時述語は Type₂ 層に位置する——値を受け取り Type を返す関数、型コンストラクタと同層
- **泛型との対話**：コンパイル時述語は泛型パラメータを持てる、例：`NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **所有権との対話**：コンパイル時述語内の式は所有権規則に従い、読み取りのみ可能、書き込み不可
- **型推導**：コンパイル時述語のパラメータは HM 型推導に参加

### ランタイム表現

コンパイル時述語のランタイム表現は **dispatch 分派結果に従って処理される**：
- **CompileTime モード**（すべての自由変数がコンパイル時既知）：証明通過後、witness トークンは完全消去。`Positive: (x: Int) -> Type = { x > 0 }`——パラメータ `b: Positive(5)` のランタイム表現は `Int` である。精化条件 `{ 5 > 0 }` は通過済み、削除される。
- **Runtime モード**（ランタイム自由変数が存在）：ランタイム check を保持——値レベルで Bool チェックを実行し、フロー依存仮定セット Γ に注入。詳細は §11 dispatch 分派パイプラインと §13 消去モデル明確化を参照。

コンパイル時述語を型位置に置く（例：`f(x: Positive(x))`）ことはラッパ型を生成せず、追加メモリを割り当てない。しかし `x` がランタイム入力から来る場合、**ランタイム Bool チェックが挿入される**。

**`ref` との対話制約**：コンパイル時述語は不変借用または所有権が既に移動済みの値のみ参照可能。可変借用を参照するコンパイル時述語は、コンパイラがコンパイル時の検証結果がランタイムでも成立することを保証できない——这类の用法は直接コンパイルエラーを報告する。

### コンパイラ変更

1. **パーサー**：コンパイル時述語は標準関数構文を使用、追加解析規則不要
2. **コンパイル時証明パイプライン**：統一 Proved/Disproved/Unproven 戻りインターフェース、自動戦略選択
3. **SMT 加速モジュール**：SMT-LIB 2.6 翻訳層、デフォルトバックエンド Z3、CVC5 代替
4. **型検査器カーネル**：推論規則実装——構造等価、βδι-簡約、全称量詞導入/消去。これが唯一の信頼ルートであり、SMT とプログラマ証明はどちらもこれを通じて検証される
5. **検証条件生成**：WP/SP 計算 + ループ不変条件証明義務
6. **エラー報告**：反例フォーマット + 未解決命題レポート + ソース位置関連付け

### 後方互換性

- ✅ コンパイル時述語を使用しないコードは完全に不変
- ✅ コンパイル時述語は CompileTime モードでランタイムオーバーヘッドゼロ、Runtime モードで必要な Bool チェックのみ保持
- ⚠️ RFC-022 の `//!` 構文はサポートされない——しかし 022 は実装されたことがなく、移行負担なし

## トレードオフ

### 利点

- **Curry-Howard 同型の完全実現**：型は命題、プログラムは証明、`name: Proposition = Proof`
- **統一性**：コンパイル時述語と通常関数は完全に同じ構文を使用、概念分裂なし
- **SMT 透明性**：プログラマは SMT の存在を知る必要がない、メンタルモデルは型検査と一致
- **漸進的採用**：一つのコンパイル時述語から開始し、カバレッジを段階的に拡大可能
- **最小ランタイムオーバーヘッド**：CompileTime モードはオーバーヘッドゼロ、Runtime モードは必要な Bool チェックのみ保持

### 欠点

- **コンパイル時間**：SMT 求解がコンパイル時間を増加させるが、予算ハード制限が上限を制御可能に保証
- **自動証明境界**：一階線形算術を超える複雑な述語はプログラマの証明関数を必要とする可能性がある。これは言語の欠陥ではない——これは停止問題の必然的な帰結である。コンパイラは False と偽って報告するより Unproven と正直に報告する
- **学習曲線**：効果的なコンパイル時述語と証明関数の記述には Curry-Howard 同型の基本直感の理解が必要
- **実装複雑性**：コンパイル時証明パイプラインの統一には精巧な設計が必要

### リスク緩和

- SMT 求解予算ハード制限（ステップ 10,000 / 時間 100ms / インスタンス化深度 3）、予算超過は Unproven を返す
- 依存型事前簡約：確定的な値計算を先に処理し、SMT は非決定的部分のみを処理
- Unproven は行き止まりではない：プログラマは証明関数を記述可能、型検査器が検証——任意の関数の戻り型検証と一致
- 増分検証：変更モジュールのみ検証
- 明確なエラーメッセージ + 反例表示 + 予算消費レポート + 未解決命題 + 提案（コンパイラが提供可能な場合）

## 代替案

| 案                             | なぜ選択しないか                           |
| -------------------------------- | -------------------------------------- |
| RFC-022：`//!` 注釈式仕様        | 仕様と型が分裂、Curry-Howard 同型に違反 |
| 独立仕様ファイル（例：CVL）           | 仕様とコードが分離、保守コスト増加           |
| ランタイム表明のみ                     | 静的に正当性を保証できない                     |
| 外部証明アシスタント（例：Coq）           | コンパイラと分断、独立した証明言語と信頼境界が必要。YaoXiang の選択：証明は YaoXiang コード、型検査器が唯一の信頼ルート |
| **本案：コンパイル時述語を一級市民として** | ✅                                     |

## 未解決問題

- [x] **SMT ソルバー選択**：デフォルト Z3（MIT ライセンス、最も広範に検証）。CVC5 は SMT-LIB 互換代替として、コンパイラフラグで切り替え。コンパイラ内部は SMT-LIB 2.6 標準形式に翻訳——SMT-LIB が抽象層、カスタム汎用ソルバーインターフェースは作らない。
- [x] **求解予算の具体値**：ステップ 10,000 / 時間 100ms / 量詞インスタンス化深度 3。コンパイラ内部で固定、ノブは与えない。実際の使用で不十分を証明する真实的ユースケースがあれば（「ユーザーが書き間違えた」のではなく）、再調整。
- [x] **量詞サポート範囲**：言語レベルでは量詞階数を制限しない。コンパイル時述語は Type パラメータを受け取る——Type は関数型を含む——したがって高階量詞は型システムの自然な帰結であり、特殊構文は不要。SMT ソルバーは自動的，一階量詞（forall/exists、交差ネストサポート、予算深度 3 で制限）を判定可能。高階量詞：SMT は Unproven を返し、コンパイラは「この述語は自動証明範囲を超えています、証明関数を提供してください」と提示。プログラマは戻り型が該命題と等しい YaoXiang 関数を記述——型検査器がその関数を検証。外部エクスポート不要、AI 不要、対話的証明モード不要。すべては YaoXiang コード、すべては型検査器によって検証される。
- [x] **反例フォーマット**：ソース変数名を直接 SMT 変数名として使用（モジュール接頭辞で衝突回避）。Z3 モデル返戻時、変数名で逆引き。出力形式：変数名 = 具体値 + ソース位置 + 述語定義位置。複雑なマッピング層は作らない。
- [x] ~~**`ref` スマートポインタとのコンパイル時述語対話？**~~ → 決定済み：コンパイル時述語は不変借用または所有権が既に移動済みの値のみ許可。可変借用される値はコンパイル時述語に出現不可。
- [x] **`forall` 述語違反カウント度量の非隣接操作への拡張？** → 拡張しない。カバレッジ範囲（隣接交換、隣接移動）は戦略 1（線形ランク関数）で補完的にカバーされる——クイックソート外層区間収縮は戦略 1 が、クイックソート外層区間収縮は戦略 1 がバックアップ、ヒープソートは戦略 1（配列インデックスパターン）がバックアップ。どの戦略でも停止性を証明できないループは、コンパイラが直接エラー報告——これはハード安全哲学であり、欠陥ではない。もし将来真实的シナリオ（非学術的構成）のアルゴリズムが四つの戦略でもカバーできない場合、再度議論する。
- [x] **線形ランク関数列挙の組合せ爆発**：候補列挙上限は 3 個の有界変数。≤3 のときすべての線形結合を列挙し SMT で逐一検証。>3 のとき単変数量度（`v_i`、`u_i - v_i`）のみ試み、失敗時は直接コンパイルエラー——「ループに >3 個の有界変数があり、コンパイラが多変数量度を自動合成できない」とプログラマに提示。これは工学的妥協ではない——プログラマによりシンプルなループを記述させるためである。
