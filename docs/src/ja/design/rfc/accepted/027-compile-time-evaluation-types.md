# RFC-027：コンパイル時述語と統一静的検証

> **参考**:
>
> - [RFC-009: 所有権モデル](../accepted/009-ownership-model.md)
> - [RFC-010: 統一型構文 - name: type = value モデル](../accepted/010-unified-type-syntax.md)
> - [RFC-011: ジェネリクスシステム設計](../accepted/011-generic-type-system.md)
> - [RFC-024: spawn ブロックベースの並行モデル](../accepted/024-concurrency-model.md)
>
> **置換**：[RFC-022: ホーア論理による静的検証サポート（仕様注釈と仕様型）](../deprecated/022-hoare-logic-static-verification.md) — 廃止済み

## 概要

本文では YaoXiang に**コンパイル時述語**を一級市民として導入し、すべてのコンパイル時静的検証を単一の**証明パイプライン**に統一することを提案する。コンパイル時述語は外部仕様の注釈ではない——それ自体が関数である。Type を返す関数は型位置で使用でき、コンパイラがコンパイル時に呼び出して戻り値を検査する。型は命題であり、コンパイル時評価は証明である。

**中心的主張**：型検査がコンパイル時に行う唯一の仕事は、証明項の構築と検証である。型等価、トークン衝突、依存型簡約、コイル時述語評価、ホーア論理含意——すべてはコンパイル時証明パイプライン内の異なる型検査であり、同じパイプラインを共有する。SMT ソルバーは型検査器の加速モジュールであり、独立した信頼境界ではない。コンパイラが Unproven を返したとき、プログラマは YaoXiang の関数を証明として書く——型検査器は任意の関数の戻り型を検証するのとまったく同じ方法でそれを検証する。すべては YaoXiang のコードであり、すべては型検査器によって検証される。

## 動機

### なぜ RFC-022 を廃止するのか？

RFC-022 は仕様を `//!` 注釈の形式として設計していた：

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← これは型から独立した注釈
    //! ensures: ExistsMax(result, arr[0..n])   ← これは型から独立した注釈
}
```

これは Curry-Howard 同型の根本的な誤りを犯している：**仕様と型を 2 つの層に分割している**。注釈は型ではない。注釈は型検査に参加しない。注釈は「外部ツール」のメンタルモデルである。

ホワイトペーパーは明確に述べている：

> "`//!` 注釈はない。独立した仕様言語はない。すべては型システムの中にある。"

### 現在の問題

- RFC-022 の `//!` 注釈は型システムから独立した外部構文である
- 仕様型と通常型は 2 つの体系であり、概念の冗長性を生む
- Debug Build 検証 / Release Build 無視という分割パターンが統一性を破壊している
- SMT ソルバーは従来の認識では外部ツールとして位置づけられている——YaoXiang では型検査器の加速モジュールとして内蔵する
- 型検査、借用検証、コンパイル時述語検査、マクロ展開がそれぞれ異なるパスを通っている

### 正しいメンタルモデル

型検査は関数として抽象化できる：

```
verify : Program → Proved | Disproved(Model) | Unproven
```

すべてのコンパイル時検査——単純な型一致、借用衝突検出、コンパイル時述語検証——はこの関数のサブタスクである。それらは同じ証明パイプラインを共有し、違いは証明項の複雑さと構築戦略にすぎない。

コンパイラが Unproven を返したとき、プログラマは証明関数を提供する——その関数の戻り型は証明すべき命題に等しい。型検査器がそれを検証する。これは通常の型検査と同じ操作である。

## 提案

### 1. `{}` は証明空間：型はアサート、検証は型検査

YaoXiang の `{}` はコンパイル時の証明空間である。その中のすべてはアサートであり、コンパイラは各項目が True であることを保証する——自動証明されるか、プログラマが証明関数を提供するかのいずれかである。

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  コンパイラは x が Float、y が Float であることを保証

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  コンパイラは data が Array(T) であることを保証
```

**ジェネリクスはコンパイル時述語の特例である。**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          パラメータはシグネチャに  {} 内にはアサートのみ
#          コンパイラはコンパイル時に呼び出して x > 0 を検証

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      パラメータはシグネチャに   コンパイラは type_of(T) == Type、type_of(data) == Array(T) を検証
```

同じパターン：`name: (params) -> Type = { アサート }`。コンパイラは「型アサート」と「値アサート」を区別しない——両方とも証明パイプライン内の評価ターゲットである。

**ループ不変条件を個別に書く必要はない。変数上の型注釈がそのまま Floyd-Hoare 不変条件となる。**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # 注釈は i を参照する——コンパイラに s の型が i に依存することを伝える
    mut i: UpTo(arr.len) = 0     # 初期化時 i=0、検証：0 == sum(arr[0..0]) → True
    while i < arr.len {
        s += arr[i]  # コンパイラ検証：s_new == sum(arr[0..i+1])
        i += 1       # i 変更→s の依存再検証をトリガー：s は SumUpTo(arr, i_new) を満たす
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

コンパイラはループ本体に対して検証条件を 1 回生成する——帰納仮説（型注釈）→ 代入操作 → 新しい値が型注釈を満たすか。証明パイプラインが帰納ステップの成立を検証した後、すべての反復が自動的にカバーされる。`: decreases` も、`: Invariant` も、帰納的証明も必要ない——コンパイラは帰納を各代入の局所 VC に分解する。

### 2. 事前/事後条件：パラメータ型と戻り型上のコンパイル時述語

RFC-022 の `//! requires` / `//! ensures` を廃止する。コンパイル時述語はパラメータまたは戻りの型注釈として表現する。

**パラメータ側は関数呼び出し。** コンパイル時述語は Type を返す関数であり、パラメータ側での使用方法は単にそれを呼び出すこと——`factorial(5)` と同じである。戻り値側では新しい概念を導入する：戻り値仮引数。

```yaoxiang
# 事前条件：パラメータ型で明示的にコンパイル時述語を呼び出す
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b
#                       ^^^^^^^^^^  b は現在の仮引数名、Positive に引数として渡される
#                       コンパイラは呼び出し点で実引数値を抽出し、b に代入し、Positive(実引数) を検証
#                       例：divide(10, 2) → Positive(2) = { 2 > 0 } を検証 → True
#                       例：divide(10, 0) → Positive(0) = { 0 > 0 } を検証 → False → コンパイルエラー

# 事後条件：戻り値仮引数 + コンパイル時述語
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

NonEmpty: (arr: Array(T)) -> Type = { arr.len > 0 }

max: (T: Ord) -> ((arr: NonEmpty(arr))) -> (result: IsMax(T, arr, result)) = {
#                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                            result は戻り値仮引数、値は return によって提供される
#                                            コンパイラは return 点で戻り値を result に代入し、事後条件を検証
    candidate = arr[0]
    for i in 1..arr.len {
        if arr[i] > candidate { candidate = arr[i] }
    }
    return candidate
}
```

**重要なルール**：

- **パラメータ側**：`b: Positive(b)` —— `b` は現在の仮引数名であり、`Positive` の引数として渡される。関数呼び出し構文で、ゼロの暗黙性。
- **戻り側**：`-> (result: IsMax(T, arr, result))` —— `result` は戻り値仮引数であり、値は `return` 文によって提供される。`result` は型シグネチャ内にのみ存在し、述語によってのみ参照され、関数本体のスコープには入らず、呼び出し側にも現れない。
- **戻り値仮引数はオプション**：事後条件がない場合は書かず、シグネチャは通常の関数と完全に一致する（`-> Int`）。
- **統一性**：パラメータと戻り値仮引数は同じ概念——`仮引数名: 述語呼び出し(仮引数名)`——であり、違いは値が呼び出し側から提供されるか `return` から提供されるかだけ。

### 3. パス条件の伝播：ランタイム値のコンパイル時検証

コンパイル時述語がバインディング位置で使用されるとき、パラメータはプログラマが明示的に渡す。ランタイム値が精緻化型のパラメータに入るとき、コンパイラはパス条件の収集と SMT 含意判定を通じて検証を完了する——プログラマが明示的に証明を渡す必要はない。

#### 3.1 明示的な関数呼び出し

コンパイル時述語がバインディング位置で使用されるとき、パラメータはプログラマが明示的に渡す——単なる関数呼び出しであり、暗黙性はない。

`Positive: (x: Int) -> Type = { x > 0 }` はコンパイル時述語コンストラクタである。それがバインディング位置（パラメータ宣言、変数宣言、戻り型）に現れるとき、プログラマは既にバインドされている変数名を明示的に渡す：

```yaoxiang
b: Positive(b)
// b は現在の仮引数として既に宣言されている、Positive(b) は関数呼び出し
// 正規化後：b: { b > 0 }
```

コンパイラが暗黙にパラメータを埋める必要はない——`b: Positive(b)` は `f(5)` と同じ、単なる関数呼び出しである。`b` がパラメータ名としてバインドされ、その型注釈 `Positive(b)` が `b` 自身を参照している——これは依存型の標準パターンであり、暗黙の展開ルールではない。

**RFC-010 の `self` との統一**：RFC-010 では `self` はキーワードではなく、単にパラメータ名の慣例（「`p`、`this`、`x` と書いても完全に同じ効果」）として確立されている。`b: Positive(b)` も同じ機構を共有する——パラメータ名は型注釈内で参照可能。`self` は `self: Point` の位置に現れ、`b` は `b: Positive(b)` の位置に現れる。両者の型注釈はパラメータ自身を参照しており、違いは型注釈の複雑さだけで、機構はまったく同じである——名前がバインドされた後、型はその名前に依存できる。

戻り型も同様に明示的な関数呼び出しを使用する：

```yaoxiang
Sorted: (arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (arr: Array(T)) -> (result: Sorted(result)) = { ... }
//                        ^^^^^^^^^^^^^^^^^^^^^^^
//                        result は戻り値仮引数、Sorted(result) は関数呼び出し
//                        コンパイラは return 点で戻り値を result に代入し、Sorted(戻り値) を検証
```

ローカル変数宣言にも同様に適用される：

```yaoxiang
let x: Positive(x) = 5
// x は 5 にバインドされる、Positive(5) → { 5 > 0 } → True → 通過

// let y: Positive(y) = 0
// y は 0 にバインドされる、Positive(0) → { 0 > 0 } → False → コンパイルエラー
```

#### 3.2 パス条件の収集

ランタイム値が条件分岐に現れるとき、コンパイラは自動的にパス条件を収集し、現在のスコープの**仮定集合**を形成する。これらの仮定はコンパイル時 Bool 評価の背景知識として検証に参加する。

```yaoxiang
if y > 0 {
    // コンパイラはこの分岐内で自動的に仮定を取得：{ y > 0 }
    let result = divide(x, y)
    // 検証条件：(y > 0) ⇒ (y > 0)
    // 証明パイプラインは含意が成立すると判定 → Proved
} else {
    // この分岐の仮定：{ !(y > 0) }
    // divide(x, y) を呼び出す場合、検証条件は !(y > 0) ⇒ y > 0
    // 証明パイプラインは含意しないと判定 → Disproved
}
```

これはコンパイラが特別のパターンをハードコードしているわけではない——これはコンパイル時証明パイプラインの自然な振る舞いである。型検査の各呼び出し点がパイプラインに送信する：

```
{背景仮定} ⇒ {検証ターゲット}
```

証明パイプラインは含意性を判定する。Proved → 通過、Disproved → コンパイルエラー + 反例、Unproven → コンパイルエラー + 未解決命題。背景仮定は現在のプログラム点のパス条件から来る。

#### 3.3 仮定スタック

コンパイラは制御フロー解析において、各基本ブロックに対して仮定集合を維持する：

- **if ガード**：`if y > 0` → 真分岐に `y > 0` をプッシュ、偽分岐に（else を使用する場合）`!(y > 0)` をプッシュ
- **match パターン**：`if let Some(v) = opt` → 分岐内に `opt == Some(v)` をプッシュ
- **論理結合**：`if x > 0 && y < 10` → 分岐内に `x > 0` と `y < 10` をプッシュ
- **関数事前条件**：`divide(a, b)` を呼び出すとき、`b` は現在の仮定または実引数自身の精緻化型注釈（`b` が `Positive` として注釈されている場合、その型は `b > 0` を運ぶ）から `Positive` の証拠を導出できなければならない
- **代入**：`let z = y` のとき、`y` 上の既存の精緻化条件が `z` に伝播する

すべての仮定がコンパイル時証明パイプラインに入る。SMT 加速パスに入るとき、SMT-LIB 背景アサーションに変換される。

#### 3.4 静的証拠なしではコンパイルエラー

プログラマが直接こう書いた場合：

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

現在のプログラム点には `y > 0` の仮定がなく、実引数 `y` 自身にも `Positive` 型注釈がない。検証条件は：

```
{} ⇒ { y > 0 }
```

パイプラインは `Disproved`（含意しない）を返す → コンパイルエラー：

> `divide` の呼び出しにおいて、引数 `b` が `Positive` を満たすことを証明できない。
> `y` は関数入力に由来し、証明済みの境界がない。
> if 分岐で呼び出しをガードすることを検討してください：`if y > 0 { divide(x, y) }`。

YaoXiang は、静的証拠を提供せずにランタイム値が精緻化型のパラメータに直接入ることを許可しない。これは制限ではなく——ハード安全哲学の中核である。コンパイラが静的に証明できないコードは、コンパイルを通してはならない。

#### 3.5 統一パイプラインとの関係

パス条件の伝播は追加の機構ではない。これはコンパイル時証明パイプラインが制御フロー解析に延長したものである：

| フェーズ | 責務 |
|------|------|
| パス条件の収集 | コンパイラの制御フロー解析フェーズ、各基本ブロックに仮定集合を注釈 |
| 検証条件の生成 | 検証が必要な型制約に遭遇したとき、パス条件 + 実引数型情報をマージ |
| 証明パイプライン評価 | コンパイラカーネル → SMT 加速 → Proved / Disproved / Unproven を導出 |
| 結果 | `Proved` → 通過；`Disproved` → コンパイルエラー + 反例；`Unproven` → コンパイルエラー + 未解決命題（プログラマが証明関数を提供可能） |

新しいコンポーネントはない。特別なルールもない。パス条件は証明パイプラインの背景知識である——型等価、借用制約と同じパイプラインと同じ予算システムを共有する。

### 4. コンパイル時証明パイプライン

すべてのコンパイル時検査は同じパイプラインを共有する。パイプラインの中心的操作は**型検査**である——証明項の型が証明すべき命題に等しいかを検査する。すべてが型検査である。

```
コンパイル時に Bool 式の評価が必要（即ち証明項の構築が必要）
        │
        ├── 型等価（T1 == T2）
        │   → コンパイラが直接判定（構造的等価性）
        │
        ├── トークン衝突条件（!conflicting(tokens)）
        │   → フロー敏感的生存解析（Duplicate/Linear 属性追跡）
        │
        ├── 依存型簡約（n + m 簡約）
        │   → コンパイル時項書き換えシステム（βδι-簡約）
        │
        ├── コンパイル時述語（x > 0, forall...）
        │   → コンパイラ自身 + SMT 加速モジュール
        │
        └── ホーア論理含意（P ⇒ Q）
            → コンパイラ + SMT 加速モジュール
                    │
                    ▼
             ┌──────────┐
             │ Proved   │  → コンパイル通過
             │ Disproved│  → コンパイルエラー + 反例
             │ Unproven │  → コンパイルエラー + 未解決命題
             └────┬─────┘
                  │
                  ▼
         プログラマが証明関数を書く（YaoXiang コード）
                  │
                  ▼
         型検査器が検証 ──→ Proved ──→ コンパイル通過
                  │
                  ▼
            検証失敗 → コンパイルエラー："証明が成立しない"
```

#### 4.1 証明結果：3 値代数

コンパイル時評価は 3 つの結果を返す——これは停止問題の必然的な結論であり、証明論の自然な区分でもある：

```
eval_compile_time : BoolExpr → Proved | Disproved(Model) | Unproven
```

- **Proved** → 停止、証明項が構築済み、型検査通過。コンパイル続行。
- **Disproved(M)** → 停止、反例 M が存在。コンパイルエラー + 反例 + ソース位置。
- **Unproven** → 与えられたリソース上限内で証明が構築されなかった。コンパイルエラー + 未解決命題 + 予算消費レポート。

**Unproven ≠ False。** コンパイラが「証明できない」と言うことは命題が偽であることを意味しない——単に現在の自動証明能力を超えているだけである。これは正直さであり、欠陥ではない。

予算のハード制限は停止問題の工学的解決である。ノブは提供しない——提供してしまうと「あなたのプログラムは停止すると思いますか」とユーザーに尋ねることになり、ユーザーは答えを知らず、コンパイラも知らない。

#### 4.2 Unproven の後：プログラマが証明を書く

コンパイラが Unproven を返したとき、プログラマは**証明関数**を書くことができる——戻り型が証明すべき命題に等しい YaoXiang 関数である。型検査器はこの関数を検証する——`add(a, b): Int` を検証するのとまったく同じ機構で。

```
命題  = 型
証明  = プログラム（その型の値）
検証  = 型検査（唯一の信頼の根）
```

SMT ソルバーは独立した信頼境界ではない——**型検査器の加速モジュール**である。SMT は証明の発見を手伝うが、証明を検証するのは常に型検査器である。SMT が `unsat` を返すとき、コンパイラはその結果を型検査器が検証可能な証明項として再構成する。再構成が失敗した場合（SMT の推論ステップがコンパイラカーネルの推論規則を超える場合）、Unproven にフォールバックする——プログラマは手動で証明関数を書くことができる。

```yaoxiang
# 命題：コンパイラが自動証明できない精緻化属性
FirstIsMin: (T: Ord, arr: Sorted(T)) -> Type = {
    forall i in 0..arr.len: arr[0] <= arr[i]
}

# 証明：プログラマが関数を書き、戻り型は上記の命題
# 型検査器がこの関数を検証する——add(a,b): Int の検証と完全に一致
first_is_min: (T: Ord, arr: Sorted(T)) -> FirstIsMin(T, arr) = {
    # コンパイラはここで検証する：関数本体の型 = FirstIsMin(T, arr)
    ...
}
```

AI も不要、Coq へのエクスポートも不要、新しい概念も不要。**コンパイル時に自動証明できない属性 → プログラマが YaoXiang コードで証明を書く → 型検査器が検証。** 全プロセスはスムーズなグラデーションである——コンパイラが簡単な証明を引き受け、難しいものはプログラマの頭に残す。

#### 4.3 パイプライン内の階層的依存

これらの評価器は同じインターフェースを共有するが、評価順序が存在する。型等価はすべての後続解析の前提条件であり；所有権/トークン検査は型情報に依存し；精緻化述語検証は前 2 層の結果に依存する。コンパイラは層ごとに評価し、低い層で失敗した式は上の層に進まない——型エラーのあるプログラムで求解予算を浪費することを避ける。

```
評価順序（同じパイプライン、階層的スケジューリング）
├── 第 0 層：型等価（T1 == T2）
│   └── 構造的単一化 → 失敗すれば後続は無意味、直接 Disproved を返す
├── 第 1 層：所有権/トークン衝突
│   └── フロー敏感的生存解析 → 失敗すればメモリ安全性が成立しない、直接 Disproved を返す
└── 第 2 層：精緻化述語 / ホーア含意
    └── コンパイラ自身 → SMT 加速 → Proved / Disproved / Unproven を導出
```

各層はまだ `Proved/Disproved/Unproven` を返し、同じインターフェースと同じ予算システムを共有する。

### 5. 3 層関数の統一

| 階層 | 実行タイミング | 入力 | 出力 | 例 |
|------|----------|------|------|------|
| 値レベル関数 | ランタイム | 値 | 値 | `add: (a: Int, b: Int) -> Int = a + b` |
| 型コンストラクタ | コンパイル時 | 型/値 | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| コンパイル時述語 | コンパイル時 | 値 | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

すべてが同じ `name: type = value` 構文を使用する。コンパイル時述語と型コンストラクタは同じコンパイル時証明パイプラインを通る——`{}` は証明空間である。

### 6. ループ：Floyd-Hoare 検証条件生成

ループに個別の `: Invariant(...)` や `: decreases(...)` 注釈は不要である。変数上のコンパイル時述語型注釈が Floyd-Hoare 風のアサートを定義する——コンパイラは型注釈から検証条件を生成し、証明パイプラインが各代入が型を保持するかをチェックする。

核心機構：各代入操作は Hoare 三つ組 `{P} x := e {Q}` に対応し、検証条件は `P ⇒ Q[e/x]` である。コンパイラはループ本体に対して検証条件を 1 回生成する——証明パイプラインが帰納ステップの成立を検証した後、すべての反復が自動的にカバーされる。

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # 注釈は i を参照する；初期化時 i=0、検証：0 == sum(arr[0..0]) → True
    mut i: UpTo(arr.len) = 0     # 検証：0 <= 0 <= arr.len → True
    while i < arr.len {
        # コンパイラはループ本体に対して VC を 1 回生成。前提：s は SumUpTo(arr, i) を満たし、i は UpTo(arr.len) を満たす。
        #
        # s += arr[i]：
        #   検証義務：s_new は SumUpTo(arr, i) を満たす（現在の i は不変）
        #   s_new = s_old + arr[i] を代入：
        #     s_old + arr[i] == sum(arr[0..i+1]) が必要
        #     帰納仮説 s_old == sum(arr[0..i]) から、両辺に arr[i] を加える：
        #     sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   コンパイラ + SMT：線形算術、ミリ秒で → Proved
        #
        # i += 1：
        #   i 変更 → 依存グラフで s の型注釈が i を参照している → 再検証をトリガー
        #   新しい検証ターゲット：s は SumUpTo(arr, i_new) を満たす
        #   即ち s == sum(arr[0..i_new])、前のステップで保証される → Proved
        s += arr[i]
        i += 1
    }
    return s  # このとき s: SumUpTo(arr, arr.len)、即ち s == sum(arr[0..arr.len])
}
```

ループ不変条件は変数上の型注釈そのものである——プログラマが型を書き、コンパイラが帰納ステップをチェックする。コンパイラは不変条件を「発見」する必要も、「自動的に帰納を行う」必要もない——帰納的証明を各代入操作の局所検証条件に分解し、証明パイプラインに分配して征服させる。

#### 6.1 依存追跡：可変変数上の依存型

上記の機構の前提は、コンパイラが `s` の型注釈 `SumUpTo(arr, i)` が `i` を参照していることを知っていることである——`i` が変更されたとき、`s` の型制約もそれに応じて変化する。これにはコンパイラが**変数間の型依存グラフ**を維持する必要がある。

**データ構造**：

```
TypeDepGraph: Map<VarName, Set<VarName>>
# キーは依存される変数、値は型注釈内でその変数を参照している変数の集合
# 例：{ i: {s}, j: {s, t}, ... }
```

**構築**：型検査器が `mut v: Pred(... x ...) = init` を処理するとき、`Pred(...)` の引数内の自由変数参照を解析する。引数が現在のスコープ内の他の可変変数 `x` を参照している場合、依存グラフに `x → v` を記録する。

**トリガー**：依存される変数 `x` が代入されたとき、コンパイラは：
1. 依存グラフで `x` に依存するすべての変数 `{v₁, v₂, ...}` を検索
2. 各 `v` について、検証条件を生成：`v` の現在の値が更新された型 `Pred(... x_new ...)` を満たすか
3. VC を証明パイプラインに送る

**代入順序に敏感**：依存追跡は正しい代入順序を自然に強制する。`SumUpTo(arr, i)` を例に取る：

```yaoxiang
# 正しい順序
s += arr[i]   # s_new は SumUpTo(arr, i+1) を満たす
i += 1        # i 変更 → s が SumUpTo(arr, i_new) を満たすかを再検証 → True

# 間違った順序——コンパイラが拒否
i += 1        # i 変更 → s が SumUpTo(arr, i_new) を満たすかを再検証
              # s はまだ更新されていない、s_old == sum(arr[0..i_old]) ≠ sum(arr[0..i_new])
              # → コンパイルエラー：変数 s は型 SumUpTo(arr, i_new) を満たさない
s += arr[i]   # 到達不能
```

**組み合わせ依存**：1 つの変数が複数の変数に依存できる。型注釈 `{ v: Int; v == x + y }` は `x` と `y` の両方に同時に依存する——いずれの変更も再検証をトリガーする。

**証明パイプラインとの関係**：依存追跡は VC 生成のトリガーであり、独立した検証機構ではない。「いつ VC を生成する必要があるか」に答え、証明パイプラインが「VC が成立するか」に答える。

### 7. 停止性検査

コンパイル時に完全自動。コンパイラが証明できるループは通過し、証明できないループは直接コンパイルエラーを報告する——プログラマはループの停止性をコンパイラが自動解析できるようにしなければならない。半自動注釈のための隙間は作らない。

#### 7.1 設計原則

コンパイラは 2 箇所から停止性証明に必要な情報を自動的に抽出する：

1. **変数の型注釈**：精緻化型の境界制約（例：`UpTo(n)` は上界 `n` と下界 `0` を提供）
2. **ループ本体の操作**：各反復で変数に適用される操作

コンパイラは 4 つの度量合成戦略を優先度順に試行し、見つかれば停止する。

#### 7.2 戦略 1：線形ランク関数の自動合成

変数に線形境界注釈がある場合、コンパイラは線形度量の候補を列挙し SMT で検証する。

```
入力：
  変数 v₁: UpTo(u₁), v₂: UpTo(u₂), ...（境界を持つ変数）
  ループ条件 cond
  ループ本体内の代入集合

アルゴリズム：
  1. 型注釈から各変数の境界を抽出：[low_i, high_i]
  2. 候補度量を列挙：v_i, u_i - v_i, v_i - v_j 等の線形結合
  3. 各候補度量 m について：
     - SMT が m ≥ 0 を検証（型境界から導出）
     - ループ本体の各実行パスについて、SMT が m' < m を検証（厳密に減少）
  4. 条件を満たす線形結合を発見 → 停止性が証明される
```

カバー範囲：任意の変数が線形式（`v = a·v + b`）で代入され、有界型注釈を持つループ。`i += const`、`i -= const`、および二分探索式の間隔収縮を含む：

```yaoxiang
# 二分探索：low = mid + 1 または high = mid
# 度量 high - low は両パスで厳密に減少
binary_search: (arr: Sorted(Int, arr), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 7.3 戦略 2：述語違反カウント——目標型から自動的に度量を抽出 <span style="color:orange">【実験的戦略】</span>

> ⚠️ **現在の状態：実験的戦略、フェーズ 3 実装時に実際の実現可能性に応じて含めるかどうかを判断。**
> この戦略は隣接交換操作（バブルソート、挿入ソート）には有効だが、非隣接操作（クイックソートのパーティション、ヒープソートの sift-down）は自動証明できない。
> カバー範囲の境界は下表の通り。フェーズ 3 で実現不可能と検証された場合、この戦略は削除されるか将来作業として降格される。

核心的な洞察：**ユーザー書いた仕様はコンパイラの推論材料である。** コンパイラは「ソートとは何か」を内蔵する必要はない——`Sorted` の定義を読み、その定義から自動的に度量を抽出する。

```
入力：
  目標型：Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  ループ本体の操作：隣接要素の交換

アルゴリズム：
  1. 述語定義を解析：forall i in range: cond(i, arr)
  2. 度量を自動生成：violation_count = |{ i | ¬cond(i, arr) }|
  3. 操作の度量への影響を分析：
     - 隣接交換 arr[j], arr[j+1] = arr[j+1], arr[j]
     - インデックス j-1, j, j+1 の 3 対のみに影響
     - arr[j] > arr[j+1]（述語違反）の場合、交換後にこの対は述語を満たす
     - violation_count は少なくとも 1 減少
  4. 上界：n·(n-1)/2（最大隣接転倒数）、下界：0
  → 停止性が証明される
```

**現在のカバー範囲**：

| アルゴリズム | 操作パターン | 戦略 2 で証明可能？ | 理由 |
|------|----------|:---:|------|
| バブルソート | 隣接交換 | ✅ | violation_count は各交換で厳密に減少 |
| 挿入ソート | 隣接移動 | ✅ | 各シフトで 1 つの違反対が消える |
| 選択ソート | 非隣接交換 | ❌ | 単一交換で violation_count が増加する可能性がある |
| クイックソート | パーティション分割 | ❌ | 非隣接交換、単調減少を保証しない |
| ヒープソート | sift-down | ❌ | 木構造操作、violation_count は非単調 |

**補完戦略**：クイックソートの場合、`low < high` 間隔収縮は戦略 1（線形ランク関数）でカバーできる——外側のパーティション再帰、各反復で間隔が半分になる。戦略 1 と戦略 2 は補完的にカバーし、ほとんどの実際のアルゴリズムの停止性は両者のいずれかで証明できる。ただし、戦略 2 の汎用化（非隣接操作、木構造操作）は依然として未解決問題である。

```yaoxiang
sort: (arr: Array(Int)) -> (result: Sorted(result)) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 7.4 戦略 3：有界増減パターン

`v += const`（正の定数）、変数に上界型注釈がある → 度量 `upper_bound - v` は反復ごとに `const` ずつ減少し、下界は 0。これは戦略 1 の縮退ケースであり、コンパイラが最初に高速処理する。

#### 7.5 戦略 4：乗法的スケーリング度量テンプレート

`v *= const`（const > 1）、変数に上下界型注釈がある。コンパイラは対数量度テンプレート `ceil(log_const(upper/v))` を内蔵し、const 倍するごとに度量が 1 減少する。

```yaoxiang
mut i: Positive(i) = 1
while i < n {
    # コンパイラが自動推論：度量 ceil(log₂(n/i))、2 倍するごとに度量が 1 減少
    i *= 2
}
```

#### 7.6 停止性と正しさの分離

停止性証明と正当性証明は独立している：
- **停止性**：上記 4 つの戦略がループが有限ステップで終了することを自動証明する
- **正しさ**：ループ本体が目標型に向かって進むかどうかは、コンパイル時証明パイプラインが検証条件でチェックする

両方が通過 → コンパイル通過。停止性は証明されたが正しさが失敗 → コンパイルエラー + 反例。正しさは証明されたが停止性が証明できない → コンパイルエラーで分析できない変数または操作を指摘。両方失敗 → コンパイルエラーで 2 つの失敗理由を別々に報告。

#### 7.7 再帰関数の停止性検査

コンパイル時に評価が必要な再帰関数について、コンパイラは引数の減少をチェックする：

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # コンパイラ分析：n-1 < n → 減少 → 停止
}

# コンパイル時使用——コンパイラは factorial がコンパイル時に停止することを保証
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120、コンパイル時に完了
```

| シナリオ | 振る舞い |
|------|------|
| コンパイラが再帰的減少（例：`n-1`）を分析できる | コンパイル時評価 |
| 減少しない / 減少を判定できない | コンパイルエラー |
| ランタイム呼び出し（型位置ではない） | 停止性検査は不要 |

#### 7.8 ハード境界

`i = f(i)` で `f` が可逆ではなく、閉じておらず、単調性を保持しない場合——数学的に自動停止証明は不可能。コンパイルエラー：

> このループは自動的に停止性を証明できない。ループ変数が解析不可能な関数 `f` に依存している。コンパイラが解析可能な反復パターンを使用してください。

これはコンパイラの失敗ではない。静的に安全性を証明できないコードは、コンパイルを通してはならない。

### 8. SMT ソルバー：型検査器の加速モジュール

SMT ソルバーは従来の言語では外部ツール（F\* の Z3 呼び出し、Dafny の Z3 呼び出しなど）である。YaoXiang では、**型検査器の加速モジュール**である——コンパイラカーネル自身が直接判定できない場合にのみ呼び出される。SMT は証明の発見を手伝うが、証明を検証するのは型検査器である。

**信頼モデル**：型検査器が唯一の信頼の根である。SMT ソルバーは加速モジュールである——証明の発見を手伝うが、SMT は独立した信頼境界ではない。コンパイラは Z3 の `unsat` 結果を信頼する（F\* / Dafny 路線と一致——Z3 が誤る確率はコンパイラ自身のバグ率より低く、これは工学的な実用的な選択である）。真の信頼できない部分は SMT 翻訳層にある——翻訳にバグがあれば、コンパイラは他のテストで露呈する。

**インターフェース**：コンパイラ内部は SMT-LIB 2.6 標準形式に変換し、特定のソルバー API に縛られない。SMT-LIB は ISO 標準であり、Z3、CVC5、MathSAT、Yices すべてがネイティブサポートする。

**デフォルトバックエンド**：Z3（MIT ライセンス、最も広範なドキュメントとコミュニティ検証）。CVC5 は SMT-LIB 互換の代替として——ユーザーはコンパイル時にコンパイラフラグで切り替え可能。

「汎用ソルバー抽象層」は作らない——SMT-LIB が抽象層である。将来 CVC5 が特定の理論でブレークスルーを達成した場合、切り替えはバイナリを置き換えるだけで、コンパイラコードの変更は不要。

```
コンパイル時 Bool 式
        │
        ├── コンパイラカーネルが直接判定可能（構造的等価性、簡単な算術、
        │   定数折りたたみ後の自明な公式）
        │   → 直接 Proved / Disproved を返す
        │
        └── コンパイラカーネルが直接判定不能（量詞、記号変数）
            → 依存型事前簡約（factorial(5) → 120）
            → SMT-LIB 形式に変換
            → Z3/CVC5 に送信（予算制限付き）
            → 戻り値：unsat → Proved  │  sat + モデル → Disproved  │  unknown → Unproven
```

**求解予算——ハード制限、スタック深度のように**：

| 予算次元 | デフォルト | 説明 |
|----------|--------|------|
| 求解ステップ数 | 10,000 | Z3 は線形算術に対して通常 100 ステップ以内。10,000 ステップで実用述語の 99% をカバー。 |
| 時間 | 100ms | 単一述語が 100ms を超える = ユーザーがコンパイル時プログラムを書いている。100ms × 50 述語 = 5 秒のコンパイル時間上限。 |
| 量詞インスタンス化深度 | 3 | 3 層のネストされた量詞で実際のパターンをカバー。これを超えると論理パズルを書いている可能性が高い。 |

予算超過は Unproven を返し、コンパイルエラー + 述語位置 + 消費量。降格なし、ランタイムチェックなし、サイレントパスなし。

**なぜこれが実際に可能か**：工学的に実用述語の 95% は線形算術である——`x > 0`、`arr.len > 0`、`0 <= idx < arr.len`——すべて決定可能な断片内にあり、SMT ソルバーはこの種の問題に対してミリ秒で返す。予算を超える複雑な述語に遭遇する稀なケースでは、プログラマが証明関数を書けばよい。

依存型は SMT 呼び出しの前に事前簡約の層を通る：`factorial(5)` は直接コンパイル時評価で `120` になり、`append([1,2], [3])` は直接評価で `[1,2,3]` になる。これらの決定論的な値計算は SMT 予算を消費しない。

プログラマは SMT の存在を知る必要はない。メンタルモデルは：**コンパイラが証明できれば通過、できなければエラー——コンパイラがわからない場合は、関数で証明して見せる。**

### 9. コンパイル時述語の組み合わせ

コンパイル時述語は Type を返す関数であり、組み合わせは関数合成で自然に実現される：

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(arr)
}
```

### 10. コード例

#### 10.1 除算の安全性

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b

result = divide(10, 2)   # ✅ コンパイラが Positive(2) = { 2 > 0 } を検証 → True
# result = divide(10, 0)  # ❌ コンパイラが Positive(0) = { 0 > 0 } を検証 → False
```

#### 10.2 配列アクセスの安全性

```yaoxiang
InBounds: (idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (arr: Array(T), idx: InBounds(idx, arr)) -> T = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ コンパイラが InBounds(1, arr) = { 0 <= 1 && 1 < 3 } を検証 → True
# y = get(arr, 5)  # ❌ コンパイラが InBounds(5, arr) = { 0 <= 5 && 5 < 3 } を検証 → False
```

#### 10.3 ソートの正しさ

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T))) -> (result: Sorted(T, result)) = {
    result = arr.clone()
    # ... ソートアルゴリズムの実装 ...
    return result
}
```

#### 10.4 ループ：コンパイラ VC 生成

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```

## 詳細設計

### 構文変更

| 以前 (RFC-022) | 以後 (本 RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | コンパイル時述語をパラメータ型として使用 `(b: Positive(b))` |
| `//! ensures: ExistsMax(result, arr)` | 戻り型で戻り値仮引数を使用 `-> (result: IsMax(T, arr, result))` |
| `/*! invariant: ... !*/` | 変数上のコンパイル時述語型注釈——Floyd-Hoare 不変条件 |
| `//! decreases: n` | コンパイラが度量関数を完全に自動推論 |
| 仕様は注釈 | 仕様は型システム |

### 構文

**コンパイル時述語に新しいキーワードはない。** `{}` は証明空間であり、既存の型定義構文と完全に一致する。コンパイル時述語は Type を返す関数に他ならない——`name: (params) -> Type = { アサート }`。使用時は単なる関数呼び出し——`Positive(b)`、`IsMax(T, arr, result)`。

```bnf
# コンパイル時述語 = Type を返す関数、{} 内はコンパイラが検証するアサート
# 既存の関数/型構文を使用、新たな BNF ルール不要
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

**新たに導入する構文概念：戻り値仮引数**——`-> (name: Type)` の `name` は戻り値仮引数。

戻り値仮引数は YaoXiang が既存の関数構文に導入する**唯一の構文概念**である。その意味：
- `name` の値は `return` 文によって提供される
- `name` は型シグネチャ内にのみ存在し、事後条件述語によって参照される（例：`-> (result: IsMax(T, arr, result))`）
- `name` は関数本体のスコープに入らず、呼び出し側にも現れない
- 戻り値仮引数は**オプション**である——事後条件がない場合、シグネチャは通常の関数と完全に一致する（`-> Int`）、追加の負担はない

導入理由：事後条件は「関数が返そうとしている値」を参照する必要がある。戻り値仮引数がない場合、コンパイラは述語が戻り値を参照できるように特別なルール（暗黙の変数 `$result` や `__retval__` など）に頼るしかない。戻り値仮引数でこの参照を明示化する——単なる仮引数であり、値が `return` ではなく呼び出し側から提供される点が異なるだけ。

**証明関数**は新しい概念ではない——戻り型がアサートされる命題である YaoXiang 関数に過ぎない。コンパイラが Unproven を返したとき、プログラマが証明関数を提供し、型検査器が任意の関数の戻り型を検証するのと完全に同じ方法でそれを検証する。新しい構文、新しいキーワード、新しいルールは不要。

### 型システムへの影響

- **型宇宙**：コンパイル時述語は Type₂ 層に位置する——値を受け取り Type を返す関数、型コンストラクタと同じ階層
- **ジェネリクスとの相互作用**：コンパイル時述語はジェネリクスパラメータを持てる、例：`NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **所有権との相互作用**：コンパイル時述語内の式は所有権ルールに従い、読み取りのみ可能、書き込み不可
- **型推論**：コンパイル時述語のパラメータは HM 型推論に参加する

### ランタイム表現

コンパイル時述語はランタイムで**完全に消去される**。`Positive: (x: Int) -> Type = { x > 0 }`——パラメータ `b: Positive(b)` のランタイム表現は単に `Int` である。精緻化条件 `{ x > 0 }` はコンパイル時に証明パイプラインで検証され、検証完了後はランタイム痕跡を残さない。

- `Positive(5)` → ランタイム表現は `Int(5)`、精緻化条件 `{ 5 > 0 }` は通過済み、消去
- `SumUpTo(arr, 0)` → ランタイム表現は `Int(0)`、等式 `0 == sum(arr[0..0])` は通過済み、消去
- コンパイル時述語を型位置に配置する（例：`f(x: Positive(x))`）ことはいかなるラッパー型も生成せず、追加メモリも確保せず、ランタイムチェックも挿入しない

ジェネリクス消去と精緻化型消去は同じ原則である：両者ともコンパイル時関数であり、コンパイル時に評価された後完全に消える。Release Build でコンパイル時述語のランタイムオーバーヘッドはゼロ——これは「述語は関数である」の直接の帰結。

**`ref` との相互作用制約**：コンパイル時述語は不変借用または所有権が既に移譲された値のみを参照できる。可変借用を参照するコンパイル時述語は、コンパイラがコンパイル時に検証結果がランタイムでも成立することを保証できない——このような用法は直接コンパイルエラーを報告する。

### コンパイラ変更

1. **パーサー**：コンパイル時述語は標準的な関数構文を使用し、追加の解析ルールは不要
2. **コンパイル時証明パイプライン**：Proved/Disproved/Unproven 戻り値の統一インターフェース、自動戦略選択
3. **SMT 加速モジュール**：SMT-LIB 2.6 翻訳層、デフォルトバックエンドは Z3、CVC5 が代替
4. **型検査器カーネル**：推論ルールの実装——構造的等価性、βδι-簡約、全称量詞の導入/除去。これが唯一の信頼の根であり、SMT とプログラマの証明はどちらもこれを通じて検証される
5. **検証条件生成**：WP/SP 計算 + ループ不変条件証明義務
6. **エラー報告**：反例フォーマット + 未解決命題レポート + ソース位置関連付け

### 後方互換性

- ✅ コンパイル時述語を使用しないコードは完全に不変
- ✅ コンパイル時述語は Release Build でランタイムオーバーヘッドゼロ
- ⚠️ RFC-022 の `//!` 構文はもはやサポートされない——ただし 022 は一度も実装されたことがなく、移行負担はない

## トレードオフ

### 利点

- **Curry-Howard 同型の完全な実現**：型は命題、プログラムは証明、`name: Proposition = Proof`
- **統一性**：コンパイル時述語と通常の関数は完全に同じ構文を使用し、概念の分裂がない
- **SMT の透明性**：プログラマは SMT の存在を知る必要がなく、メンタルモデルは型検査と一致
- **漸進的採用**：1 つのコンパイル時述語から始めて、徐々にカバレッジを拡大可能
- **ランタイムオーバーヘッドゼロ**：すべての評価がコンパイル時に完了、Release Build はいかなるランタイムアサートも挿入しない

### 欠点

- **コンパイル時間**：SMT 求解がコンパイル時間を増加させるが、予算のハード制限により上限は制御可能
- **自動証明の境界**：一階線形算術を超える複雑な述語はプログラマが証明関数を書く必要がある。これは言語の欠陥ではなく、停止問題の必然的な結論である。コンパイラは False と偽装する代わりに Unproven を正直に報告する
- **学習曲線**：効果的なコンパイル時述語と証明関数の記述には Curry-Howard 同型の基本的な直感の理解が必要
- **実装の複雑さ**：コンパイル時証明パイプラインの統一には慎重な設計が必要

### リスク軽減

- SMT 求解予算のハード制限（ステップ数 10,000 / 時間 100ms / インスタンス化深度 3）、予算超過は Unproven を返す
- 依存型の事前簡約：決定論的な値計算を先に行い、SMT は非決定的な部分のみを担当
- Unproven は行き止まりではない：プログラマは証明関数を記述でき、型検査器が検証する——任意の関数の戻り型検証と一致
- 増分検証：変更されたモジュールのみを検証
- 明確なエラーメッセージ + 反例表示 + 予算消費レポート + 未解決命題 + 提案（コンパイラが提供可能な場合）

## 代替案

| 案                                | 選択しない理由                          |
| -------------------------------- | -------------------------------------- |
| RFC-022：`//!` 注釈式仕様        | 仕様と型が分裂、Curry-Howard 同型に違反 |
| 独立仕様ファイル（CVL など）      | 仕様とコードが分離、保守コストが増加    |
| ランタイムアサートのみ            | 静的に正しさを保証できない              |
| 外部証明支援ツール（Coq など）    | コンパイラと切断、独立した証明言語と信頼境界が必要。YaoXiang の選択：証明は YaoXiang コード、型検査器が唯一の信頼の根 |
| **本提案：コンパイル時述語を一級市民として** | ✅                                     |

## 未解決問題

- [x] **SMT ソルバーの選択**：デフォルトは Z3（MIT ライセンス、最も広く検証済み）。CVC5 は SMT-LIB 互換の代替として、コンパイラフラグで切り替え。コンパイラ内部は SMT-LIB 2.6 標準形式への翻訳を目標とする——SMT-LIB が抽象層であり、カスタム汎用ソルバーインターフェースは作らない。
- [x] **求解予算の具体値**：ステップ数 10,000 / 時間 100ms / 量詞インスタンス化深度 3。コンパイラ内部で固定、ノブは提供しない。実際の使用で十分なケース（「ユーザーが間違えた」のではない）が証明された場合、再調整する。
- [x] **量詞のサポート範囲**：言語レベルでは量詞の階数を制限しない。コンパイル時述語は Type パラメータを受け取る——Type は関数型を含む——したがって高階量詞は型システムの自然な帰結であり、特別な構文は不要。SMT ソルバーは一階量詞（forall/exists、交差ネストをサポート、予算深度 3 で制限）を自動判定できる。高階量詞：SMT は Unproven を返し、コンパイラは「この述語は自動証明範囲を超えています、証明関数を提供してください」と促す。プログラマは戻り型が命題に等しい YaoXiang 関数を書く——型検査器がその関数を検証する。外部エクスポート不要、AI 不要、対話的証明モード不要。すべてが YaoXiang コード、すべてが型検査器によって検証される。
- [x] **反例のフォーマット**：ソース変数名を直接 SMT 変数名として使用（モジュールプレフィックス付きで衝突回避）。Z3 モデルが返されたとき、変数名で逆引き。出力形式：変数名 = 具体値 + ソース位置 + 述語定義位置。複雑なマッピング層は作らない。
- [x] ~~**`ref` スマートポインタとコンパイル時述語の相互作用？**~~ → 決定済み：コンパイル時述語は不変借用または所有権移譲済みの値のみを許可。可変借用された値はコンパイル時述語に現れない。
- [x] **`forall` 述語違反カウント度量の非隣接操作への拡張？** → 拡張しない。現在のカバー範囲（隣接交換、隣接移動）は戦略 1（線形ランク関数）で補完的にカバーされる——クイックソートの外側間隔収縮は戦略 1 がカバー、ヒープソートは戦略 1（配列インデックスパターン）がカバー。どの戦略でも停止性を証明できないループは、コンパイラが直接エラーを報告する——これはハード安全哲学であり、欠陥ではない。将来、4 つの戦略すべてでカバーできないアルゴリズム（非学術的構成）の実ケースがあれば、再議論する。
- [x] **線形ランク関数の列挙組み合わせ爆発**：候補列挙の上限は 3 つの有界変数。≤3 の場合、すべての線形結合を列挙し SMT で 1 つずつ検証。>3 の場合、単一変数度量（`v_i`、`u_i - v_i`）のみを試行し、失敗すれば直接コンパイルエラーを報告する——「ループに >3 の有界変数があり、コンパイラが多変量度量を自動合成できません」とプログラマに促す。これは工学的妥協ではなく、プログラマによりシンプルなループを書くことを強制するものである。
