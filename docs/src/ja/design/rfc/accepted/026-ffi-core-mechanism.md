## 要約

本文書はYaoXiangのFFI（外部関数インターフェース）コア機構を定義する。コア思想：**外部ライブラリはコンパイル時リンクされる第一級の値であり、跨境界データのメモリレイアウトは型定義時に固定し、YaoXiangのヒープオブジェクトは外部コードと構造的に分離する。**

1. **外部ライブラリは値**：`Native.c("libsqlite3")`はコンパイル時にライブラリをリンクし、カリー化でライブラリ情報を保持するリゾルバを返す
2. **外部シンボルも値**：リゾルバにシンボル名を適用して外部参照を取得し、`name: type = value`（RFC-007/010）で型または関数にバインドする
3. **型の二分化**：不透明ハンドラ（レイアウトは外部）/ 透明型（レイアウトはYaoXiang）、第三の型はない
4. **マーシャリング分離**：跨境界データはデフォルトで呼び出し一時領域にコピーされ、YaoXiangヒープオブジェクトは外部コードと分離
5. **所有権的安全性**：ハンドラの唯一的所有権（Move）+ RAIIにより、二重解放とuse-after-freeを構造的に防止
6. **エスケープハッチ**：`*T`生ポインタ + `unsafe {}`、ユーザがゼロコピーの直接メモリアクセスリスクを明示的に受け入れる

**コア境界——五条の違反不可な契約**：

```
1. ライブラリはコンパイル時にリンク、シンボルはコンパイル時に存在性を検証
2. 型レイアウトの帰属は定義時に確定：不透明ハンドラは外部、透明型はYaoXiang
3. デフォルトマーシャリングは一時領域経由コピー、YaoXiangヒープオブジェクトは外部コードと分離
4. ハンドラの唯一的所有権 + Move、二重解放/ダングリングを構造的に防止
5. 外部コードは常に「レイアウト明確、所有権明確」なメモリを読み書き、曖昧な領域は存在しない
```

## 動機

### 現状と目標

現在のコードベースの`native("symbol")`はYaoXiangバイトコードがRust std関数を呼び出す分发 механизм（`FfiRegistry` = `HashMap<String, RustFnPtr>`）に過ぎず、**真の跨ABI境界はない**——dlopenなし、C ABIマーシャリングなし、メモリの所有権跨境界なし。

真のFFIは四つの問題を解決する必要がある：

| 問題 | 本RFCの回答 |
|------|-------------|
| **シンボル解決** | ライブラリはコンパイル時リンクの第一級の値（`Native.c("lib")`）、シンボルはコンパイル時に検証 |
| **値マーシャリング** | シグネチャ駆動、各引数位置の変換ルールをコンパイル時に決定 |
| **メモリの所有権** | 型の二分化で帰属を決定；デフォルトでコピー分離 |
| **ライフタイム安全性** | Move + RAII + 借用で単一呼び出しを限定 |

RFC-020とRFC-021はそれぞれFFIの異なる側面を定義しており、互いに重複がある。本文書は統一仕様として統合する。

### 設計目標

1. **生ポインタがユーザコードにリークしない**：通常のFFI使用では、`.yx`ソースコードに生ポインタ不会出现
2. **レイアウト帰属の明示化**：ユーザが型を定義する際にメモリ帰属を決定、実行時推論に依存しない
3. **構造的安全性**：リークなし、二重解放なし、use-after-freeなしは型システムが保証、約束に依存しない
4. **正直な信頼境界**：Cはコンパイル時検証可能な型契約を提供できず、信頼はバインディング宣言箇所に局所化
5. **ブートストラップ互換性**：ホスト言語独自の過度な抽象化は行わない

### 範囲外

- **マルチABI機構プラグイン体系**（Wasm/Python/カスタムABI）：RFC-026a参照
- **yx-bindgenツールチェーン**：RFC-026b参照
- **YaoXiangからCへの関数エクスポート（逆FFI）**：後続RFC、本文書は原則のみ宣言
- **インラインアセンブリ、SIMD intrinsics**：本RFCの範囲外

## 提案

### 1. 外部ライブラリとシンボル：カリー化された第一級の値

FFIの情報ギャップ——「どのライブラリ哪个シンボル」——は、ライブラリを第一級の値にすることで埋め合わせ、新しいキーワードを導入しない。

#### 1.1 ライブラリは値

```yaoxiang
// Native.cにライブラリ名を適用 → コンパイル時にそのライブラリをリンクし、シンボルリゾルバを返す
sqlite3 = Native.c("libsqlite3")
```

`Native.c("libsqlite3")`は**コンパイル時アクション + 実行時値**である：

- **コンパイル時**：リンカ`-lsqlite3`、ライブラリがシンボルテーブルに追加、シンボル存在性が検証可能
- **値**：`sqlite3`はリゾルバであり、シンボル名を適用するとそのライブラリの外部参照を得る

`.c`はABI機構ラベル（C ABI）。コアは`.c`のみをビルトイン；他の機構（`.wasm`等）はRFC-026a参照。

#### 1.2 シンボルは値、バインディングは`name: type = value`

リゾルバにシンボル名を適用して得られる外部参照は、RFC-007/010の統一構文でバインドする。**LHSの型注釈でこの参照が型か関数かを決定**：

```yaoxiang
sqlite3 = Native.c("libsqlite3")

// LHSがType → 不透明型としてバインド
SqliteDb: Type = sqlite3("sqlite3")

// LHSが関数シグネチャ → 関数としてバインド
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")
SqliteDb.exec: (sql: String) -> Int32 = sqlite3("sqlite3_exec")
SqliteDb.close: () -> Int32 = sqlite3("sqlite3_close")

// .dropは通常のメソッドバインディング（RFC-009 RAII約束）
SqliteDb.drop = SqliteDb.close
```

コンパイル時検証：`sqlite3("sqlite3_open")`において`sqlite3_open`が`libsqlite3`のシンボルテーブルに存在しない場合、コンパイルエラー。

#### 1.3 メソッドバインディングとself位置

`Type.method: (...) -> ...`の記法では、`self`が暗黙的に第一位——`db.exec("SELECT")`呼び出し時、`db`はC関数`sqlite3_exec`の0番目の引数として渡される。

既に宣言されている独立関数をメソッドとしてバインドする必要がある場合、`[N]`構文でself位置を指定（RFC-004カリー化多位置バインディング）：

```yaoxiang
// 独立関数
sqlite3_close_v2: (db: SqliteDb) -> Int32 = sqlite3("sqlite3_close_v2")

// メソッドとしてバインド、[0]はdbがself임을 나타냄
SqliteDb.soft_close = sqlite3_close_v2[0]
```

`Native.c(...)`直接メソッドバインディングと`[N]`手動バインディングはどちらも`name: type = value`であり、関数値を`=`の右辺に置くだけであり、二つの仕組みはない。

#### 1.4 ユーザ使用：ゼロunsafe、ゼロ生ポインタ

```yaoxiang
import sqlite3_bindings

db = SqliteDb.open("test.db")
db.exec("SELECT * FROM users")
// ← スコープ終了、RAIIによりSqliteDb.dropが自動的に呼び出される → sqlite3_close(db)
```

### 2. 型の二分化：レイアウト帰属は定義時に固定

外部データがYaoXiangに入る際，唯一の問題は：**このメモリのレイアウト，谁が说了算？**

```
├─ レイアウトは外部のブラックボックス（sqlite3、FILE*、socket fd）
│   → 不透明ハンドラ  =  lib("symbol")
│   → YaoXiangはポインタのみを保持，逆参照は一切行わず，ライブラリ関数間でのみ 전달
│   → 外部コードは自身のメモリを読み書き，YaoXiangは関与しない
│
└─ レイアウトはYaoXiangが定義（timespec、point、フィールドを読み込むstruct）
    → 透明型  =  { field: Type, ... }
    → YaoXiangはメモリを所有，レイアウトを定義，フィールドを読み書き
    → 外部コードはYaoXiangがレイアウトを定義済みのメモリに書き込み/読み込み
```

**第三の型はない。** 以前の設計における「三層メモリモード（コピー/接管/システムレベル）」はパッチ的な思考——真実はレイアウト帰属の二分法である。

#### 2.1 不透明ハンドラ：レイアウトは外部

```yaoxiang
SqliteDb: Type = sqlite3("sqlite3")
```

- YaoXiang内部はポインタサイズのハンドラを一つ保持
- ユーザは構築不可（`SqliteDb {}` → コンパイルエラー）、フィールドアクセス不可（アクセス可能なフィールドがない）
- 唯一の発生源：`SqliteDb`を返す外部関数
- メソッド呼び出し時にハンドラをライブラリに戻し渡与、ライブラリは**自身の**メモリを読み取る（`sqlite3`構造体はライブラリのヒープ上にある）

外部コードが「内部を読み取る”是指すのは自身が割り当てた構造体であり、YaoXiangはハンドラを搬运するだけ。メモリの競合はない。

#### 2.2 透明型：レイアウトはYaoXiang

```yaoxiang
// フィールドに意味があり、読み書きする必要がある → 透明型、レイアウトはYaoXiangが宣言
Timespec: Type = {
    tv_sec: Int64,
    tv_nsec: Int64
}
clock_gettime: (clk: Int32, ts: *Timespec) -> Int32 = Native.c("librt")("clock_gettime")

ts = clock_gettime(CLOCK_REALTIME)   // §3参照、マーシャリングは一時領域経由
print(ts.tv_sec)                      // YaoXiangは自身のフィールド定義に従って読み取り
```

外部コードは**YaoXiangがレイアウトを定義，所有する**メモリを読み書きする。レイアウトはYaoXiangの契約であり、外部のものではない。

#### 2.3 判断規則

ユーザは一つのことだけを判断する必要がある：**この型のフィールドを読みたいか？**

| 判断 | 型 | レイアウト帰属 |
|------|------|---------|
| フィールドを読みません，霍在庫関数間でハンドラのみ передаём | 不透明ハンドラ `= lib("sym")` | 外部 |
| フィールドを読み書きします | 透明型 `{ ... }` | YaoXiang |

### 3. マーシャリング：シグネチャ駆動，一時領域分離

跨境界データ変換は**シグネチャ駆動**であり、コンパイル時に各引数位置の変換ルールを決定する。**コア安全保証：外部コードが読み書きするのはマーシャリング一時領域であり、YaoXiangのヒープオブジェクトではない。**

#### 3.1 デフォルトは一時領域コピー経由

```
YaoXiang → C（入力引数）：
    データを一時領域にコピー → 一時領域へのポインタをCに渡す
    → Cが境界外アクセス/書き込み损坏しても一時領域のみ被害、YaoXiangヒープオブジェクトは分離

C → YaoXiang（戻り値/出力引数）：
    Cが一時領域に書き込み → YaoXiangがmemcpyで自身のオブジェクトに戻す
    → CはYaoXiangの最終オブジェクトに触れない
```

**外部コードは常にマーシャリング一時領域を読み書きし、YaoXiangのヒープオブジェクトと完全に分離する。** レイアウト宣言ミス，Cがポインタを保存してdangling，Cが境界外アクセス——这些都是一時領域のみ被害，YaoXiangのオブジェクトは無傷。代价是一次memcpy。

#### 3.2 マーシャリングルール表

**入力引数方向（YaoXiang → C）**：

| YaoXiang型 | C表現 | マーシャリング動作 | 所有権 |
|--------------|--------|---------|--------|
| `Int32/Int64/Float` | `int/long/double` | 直接レジスタ配置、ゼロ変換 | 値セマンティクス |
| `String` | `const char*` | 読み取り専用ビューを借用（一時、呼び出し期間有効） | YaoXiangが保持、Cは読み取りのみ |
| 透明型 | `struct T*` | 一時領域にコピー、一時領域ポインタを渡す | YaoXiangがオブジェクトを所有、Cはコピーを読み取り |
| 不透明ハンドラ | `void*` | 内部ハンドラポインタを取り出す | YaoXiangが所有、Cに借用 |
| `*T` | `T*` | 生ポインタをそのまま渡す（unsafe） | ユーザ責任 |

**戻り値方向（C → YaoXiang）**：

| C戻り値 | YaoXiang型 | マーシャリング動作 | 所有権 |
|--------|--------------|---------|--------|
| `int/double` | `Int32/Float` | 直接レジスタ読み取り | 値セマンティクス |
| `char*` | `String` | strlen + memcpyしてYaoXiang Stringに | YaoXiangがコピーを所有、元メモリに触れない |
| `struct T*`（新規ハンドラ） | 不透明ハンドラ | ハンドラをYaoXiangオブジェクトに保存 | YaoXiangが接管 |
| `struct T`（値/出力引数） | 透明型 | Cが一時領域に書き込み → memcpyしてYaoXiangに戻す | YaoXiangが所有 |
| `char*`（静的領域） | `*const U8` | 生ポインタを保存、コピーなし（unsafe読み取り） | 接管なし、ユーザ責任 |

#### 3.3 借用ライフタイム：単一呼び出しを厳密に限定

YaoXiangが外部コードに借用するポインタ（String読み取り専用ビュー、透明型一時領域、ハンドラ）は、**ライフタイムが単一呼び出し内に厳密に限定される**：

- 呼び出し期間：ポインタ有効、外部コードは読み書き可能
- 呼び出し戻り後：借用は直ちに失效

外部コードがポインタを保存して呼び出し後に使用する場合、それは外部コードがFFI標準契約に違反したことになる（ライブラリbug等同然）、YaoXiangはその責任负わない。これはすべての言語のC FFI契約と一致（Rustの`&T`をCに渡す，同样に制約あり）。

#### 3.4 Stringは永続ポインタを渡さない

`String`は「CはYaoXiangメモリに手を出すな」を実現する鍵：

- Cへ：呼び出し期間有効な**一時読み取り専用ビュー**を借用として渡す
- Cから：strlen + memcpyしてYaoXiangが所有する**コピー**に

CはYaoXiang Stringの永続ポインタを絶対に取得できず、YaoXiangはCの`char*`の長期参照を保持しない。構造的に分離。

### 4. 所有権とライフタイム：Move + RAII

不透明ハンドラはRFC-009的所有権モデルに従い、新しい概念はない。

#### 4.1 コア原則

- **Moveセマンティクス**：不透明ハンドラはデフォルトでMove、代入/引数渡し/戻り = 所有権移転、複製不可
- **ハンドラの唯一的所有権**：任意の時点でハンドラは一つの所有者のみ → 構造的に二重解放杜绝
- **RAII解放**：スコープ終了時に`.drop`がバインディングされていれば自動呼び出し
- **소비追跡**：明示的デストラクトまたはMove後、変数は消費済みとなり再利用不可 → use-after-free杜绝

#### 4.2 `.drop`はオプションの外部副作用

```yaoxiang
SqliteDb.drop = SqliteDb.close     // スコープ終了時にsqlite3_closeを呼び出す
```

**`.drop`はYaoXiang侧泄漏防止機構ではない**——YaoXiang側のハンドラストレージ（ポインタサイズの値）は`.drop`に関係なく自動回收される。`.drop`は**スコープ終了時に外部関数を呼び出すオプションの副作用**：

- `.drop`をバインディング → スコープ終了時に呼び出す（外部リソースのクリーンアップ）
- `.drop`をバインディングしない → 何もしない、**エラーなし、警告なし**

外部リソースをクリーンアップする必要があるかどうかは外部ライブラリの仕様問題（`getenv`が静的領域を返す場合は解放不可、グローバルシングルトンは解放不可）であり、YaoXiangは越権して强制しない。泄漏防止はMove + 唯一的所有権（無条件、構造的）に依赖し、`.drop`には依赖しない。

#### 4.3 自動デストラクトと順序

```yaoxiang
{
    db = SqliteDb.open("test.db")
    stmt = db.prepare("SELECT * FROM users")
    // ← スコープ終了、逆順で自動デストラクト（.dropがある場合にのみ呼び出し）：
    //   stmt.drop()  → sqlite3_finalize(stmt)
    //   db.drop()    → sqlite3_close(db)
}
```

デストラクト順序：定義順序の逆、RAIIに準拠。

#### 4.4 Moveと消費

```yaoxiang
db = SqliteDb.open("test.db")
db2 = db                // Move：所有権移転
db.exec("...")          // ❌ コンパイルエラー：dbは既にMove済み、消費後は読み取り不可

process_db: (db: SqliteDb) -> Void = {
    db.exec("...")
    // ← 関数終了、db在此デストラクト
}
process_db(some_db)     // Moveして関数に入る
// some_dbは此处無効
```

#### 4.5 Null処理

```yaoxiang
// nullを返す可能性がある → ?T、ユーザは処理必须
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")

db = SqliteDb.open("test.db")
match db {
    Some(db) => db.exec("SELECT 1"),
    None => print("開くのに失敗した")
}

// 契約上nullを返さない → マークなし、null時はpanicで露出
```

Cがnullを返す場合、ユーザが処理（`?T`）またはpanicで露出する。！「默認忽略」という第三の選択肢はない。

#### 4.6 デストラクト失敗処理

`.drop`バインディングされた関数の戻り値で動作を決定：

| `.drop`の戻り型 | 動作 |
|----------------|------|
| `Void` | 失敗なし |
| `Int32`（エラーコード） | 非0時はpanic——デストラクト失敗は状態異常を意味する、露出优于默認 |
| `?Error` | 非None時はpanic——同上 |

デストラクト失敗は黙認できない。特定のエラーを無視する必要がある場合、`.drop`バインディングされたラッパー関数で明示的に処理する。

### 5. spawnブロック内のFFI動作

リソース型判定は`.drop`バインディングで决定（RFC-024）、追加マーク不要：

| 判定 | 動作 |
|------|------|
| 不透明ハンドラに`.drop`がバインディングされている | リソース型——spawnブロック内で同一インスタンス操作は自動的に直列化 |
| 不透明ハンドラに`.drop`がない | 非リソース型——並列可能（解放副作用のない純粋データハンドラ） |
| 透明型 / 値型 | 非リソース型——並列可能 |

```yaoxiang
SqliteDb.drop = SqliteDb.close   // → リソース型

(a, b) = spawn {
    r1 = db.exec("SELECT ..."),   // 同一インスタンス、自動的に直列化
    r2 = db.exec("INSERT ...")    // r1を待つ
}

(x, y) = spawn {
    db1 = SqliteDb.open("a.db"),   // 異なるインスタンス、並列可能
    db2 = SqliteDb.open("b.db")
}
```

`.drop`がある型はspawn内で自動的に同一インスタンス操作を直列化し、デストラクト時の并发競合を保証しない。

---

### 6. エスケープハッチ：生ポインタ + unsafe

デフォルトマーシャリングは一時領域コピー経由、安全だがmemcpyオーバーヘッドがある。性能が重要なシナリオ（大構造体、高頻度呼び出し）でゼロコピーが必要な場合、ユーザは明示的に生ポインタエスケープハッチを使用：

```yaoxiang
// CはYaoXiangメモリを直接読み取り、ゼロコピー——ユーザは明示的にリスクを受け入れる
ptr: *const U8 = Native.c("libc")("getenv")("HOME")
unsafe {
    value = read_c_string(ptr)   // ポインタ有効をユーザが保証
}
```

**`unsafe`は生ポインタ操作のみに使用し、不透明ハンドラ、透明型と完全に直交する。** 通常のFFI（ハンドラ + 透明型）ではunsafeは不要。`unsafe {}`を書く = ユーザが直接メモリアクセスのリスクを明示的に承認する。

**信頼境界**：Cはコンパイル時検証可能な型契約を提供できない（`.h`はABI契約ではない、シンボルテーブルには名前だけでシグネチャがない）。したがってCシグネチャの正しさは自動検証不可能——バインディング作成者が`Native.c(...)` + シグネチャを書く時に保証する。**信頼はバインディング宣言箇所に局所化**：バインディング作成者が保証し、パッケージユーザは安全なAPIを得る。Rustの`extern "C"`と一致（externを書くことは信頼行為、安全なラッパーで包んだ後の呼び出しは安全）。

## トレードオフ

### 优点

1. **情報が完全**：ライブラリはコンパイル時にリンク、シンボルはコンパイル時に検証、実行時に「ライブラリが見つからない」という曖昧さがない
2. **レイアウト帰属が明示**：型の二分化、定義時に固定、実行時推論なし
3. **構造的安全性**：一時領域分離 + Move + RAII、外部コードはYaoXiangヒープオブジェクトに触れない
4. **新しいキーワードゼロ**：`Native.c`カリー化 + `name: type = value`、すべて既存の構文を再利用
5. **正直な境界**：Cシグネチャを検証できると偽らず、信頼を宣言箇所に局所化

### 缺点

1. **memcpyオーバーヘッド**：デフォルトマーシャリングはコピー、大構造体高頻度呼び出しでは明示的にエスケープハッチを使用必要
2. **レイアウト保証は手作業**：透明型のレイアウトとC structの一致はバインディング作成者/yx-bindgenが保証
3. **Cシグネチャはコンパイル時検証不可**：FFIの根本的な制限、YaoXiangはCから排除できない

### フェーズ1：外部ライブラリとシンボル (v0.8)

- [ ] 借用ライフタイムの単一呼び出し限定を実装

## 設計決定記録

| 決定 | 結論 | 理由 | 日付 |
|------|------|------|------|
| **ライブラリは値** | `Native.c("lib")`カリー化でリゾルバを返す | ライブラリ情報がコンパイル時可见の第一級の値となり、「どのライブラリにリンクするか」のギャップを埋め、新しいキーワードゼロ | 2026-07-03 |
| **コンパイル時リンク** | `Native.c("lib")`が`-llib`をトリガー | シンボルテーブルはコンパイル時に読み取り可能、シンボル存在性は検証可能、型は実在する | 2026-07-03 |
| **型の二分化** | 不透明ハンドラ / 透明型 | レイアウト帰属の二分法はすべてをカバー；「三層メモリモード」パッチを削除 | 2026-07-03 |
| **マーシャリング一時領域分離** | デフォルトコピー、ヒープオブジェクトは外部と分離 | 外部の境界外/danglingは一時領域のみ被害、YaoXiangオブジェクトは無傷；ゼロコピーは明示的なエスケープハッチ必要 | 2026-07-03 |
| **`.drop`オプション** | 欠落は何もしない、エラーなし | YaoXiangのハンドラストレージは自動回收；外部リソースクリーンアップは外部仕様であり、越権して强制しない | 2026-07-03 |
| **泄漏防止機構** | Move + ハンドラの唯一的所有権（無条件） | 構造的保証、`.drop`无关 | 2026-07-03 |
| **信頼境界** | `Native.c(...)`宣言箇所 | Cシグネチャはコンパイル時検証不可、信頼を局所化、unsafeは生ポインタのみに使用 | 2026-07-03 |
| **Null処理** | `?T`またはpanic | Cの問題は隠さない、「默認忽略」オプションなし | 2026-07-03 |
| **デストラクト失敗** | `.drop`の戻り型で決定、统一panic | デストラクト失敗は黙認できない | 2026-07-03 |
