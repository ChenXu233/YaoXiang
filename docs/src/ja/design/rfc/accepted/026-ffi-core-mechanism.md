## 概要

本文書は YaoXiang の FFI（外部関数インターフェース）コアメカニズムを定義する。核心思想は**外部ライブラリはコンパイル時にリンクされる一次値であり、境界を跨ぐデータのメモリレイアウトの帰属は型定義時に固定され、YaoXiang のヒープオブジェクトは外部コードと構造的に隔離される**ことである。

1. **外部ライブラリ＝値**：`Native.c("libsqlite3")` でコンパイル時にライブラリを link し、パーサを返す。ライブラリ情報をカリー化で保持する
2. **外部シンボル＝値**：パーサにシンボル名を適用して外部参照を得る。`name: type = value`（RFC-007/010）で型または関数としてバインドする
3. **型の二分**：不透明ハンドル（レイアウトは外部に帰属）/ 透明型（レイアウトは YaoXiang に帰属）、第三の型は存在しない
4. **マーシャリング隔離**：境界を跨ぐデータはデフォルトで呼び出し一時領域にコピーされ、YaoXiang のヒープオブジェクトは外部コードから隔離される
5. **所有権の安全性**：ハンドルの一意所有権（Move）+ RAII、二重解放と use-after-free を構造的に根絶
6. **エスケープハッチ**：`*T` 生ポインタ + `unsafe {}`、ユーザーが明示的にゼロコピーの直接メモリアクセスリスクを受け入れる

**核心境界——5 つの不可侵な契約**：

```
1. ライブラリはコンパイル時に link され、シンボルはコンパイル時に存在が検証される
2. 型レイアウトの帰属は定義時に確定する：不透明ハンドルは外部、透明型は YaoXiang
3. デフォルトのマーシャリングは一時領域のコピーを経由し、YaoXiang ヒープオブジェクトと外部コードは隔離される
4. ハンドルの一意所有権 + Move、二重解放/ダングリングを構造的に防止
5. 外部コードは常に「レイアウト明確、所有権明確」なメモリを読み書きする、曖昧な地帯は存在しない
```

## 動機

### 現状と目標

現在のコードベースの `native("symbol")` は YaoXiang バイトコードが Rust 標準関数を呼び出すディスパッチ機構（`FfiRegistry` = `HashMap<String, RustFnPtr>`）に過ぎず、**真の ABI 境界を跨ぐ処理は一切ない**——dlopen もなく、C ABI マーシャリングもなく、メモリの所有権の境界横断もない。

真の FFI は 4 つの問題を解決しなければならない：

| 問題 | 本 RFC の回答 |
|------|--------------|
| **シンボル解決** | ライブラリはコンパイル時に link される一次値（`Native.c("lib")`）、シンボルはコンパイル時に検証 |
| **値マーシャリング** | シグネチャ駆動、コンパイル時に各引数位置の変換規則を確定 |
| **メモリ所有権** | 型の二分で帰属を決定；デフォルトはコピーで隔離 |
| **ライフタイム安全性** | Move + RAII + 借用は単一呼び出しに限定 |

RFC-020 と RFC-021 はそれぞれ FFI の異なる側面を定義しており、重複がある。本ドキュメントはそれらを統合し統一規範とする。

### 設計目標

1. **生ポインタの漏れをユーザーコードにゼロにする**：通常の FFI 使用において、`.yx` ソースに生ポインタを出現させない
2. **レイアウト帰属の明示化**：ユーザーが型を定義する時点でこのメモリの帰属を決定し、ランタイム推論に依存しない
3. **構造的な安全性**：メモリリーク、二重解放、use-after-free は型システムによって保証され、規約に依存しない
4. **誠実な信頼境界**：C はコンパイル時に検証可能な型契約を提供できないため、信頼はバインディング宣言箇所に局所化する
5. **セルフホスティング互換**：ホスト言語に固有の過剰な抽象化を行わない

### 範囲外

- **複数 ABI メカニズムプラグイン体系**（Wasm/Python/カスタム ABI）：RFC-026a を参照
- **yx-bindgen ツールチェーン**：RFC-026b を参照
- **YaoXiang が関数を C にエクスポートする（逆方向 FFI）**：後続 RFC、本文書では原則のみ宣言
- **インラインアセンブラ、SIMD intrinsics**：本 RFC の範囲外

## 提案

### 1. 外部ライブラリとシンボル：カリー化された一次値

FFI の情報ギャップ——「どのライブラリ、どのシンボルをリンクするか」——は、ライブラリを一次値にすることで埋める。新しいキーワードは一切導入しない。

#### 1.1 ライブラリ＝値

```yaoxiang
// Native.c にライブラリ名を適用 → コンパイル時にライブラリを link、パーサを返す
sqlite3 = Native.c("libsqlite3")
```

`Native.c("libsqlite3")` は**コンパイル時アクション + ランタイム値**である：

- **コンパイル時**：リンカが `-lsqlite3` を実行し、ライブラリがシンボルテーブルに入り、存在性を検証可能
- **値**：`sqlite3` はパーサであり、シンボル名を適用することでそのライブラリの外部参照を得る

`.c` は ABI メカニズムタグ（C ABI）である。コアには `.c` のみ組み込まれ、他のメカニズム（`.wasm` 等）は RFC-026a を参照。

#### 1.2 シンボル＝値、バインディング＝`name: type = value`

パーサにシンボル名を適用し、得られた外部参照を RFC-007/010 の統一構文でバインドする。**LHS の型注釈がこの参照を型とするか関数とするかを決定する**：

```yaoxiang
sqlite3 = Native.c("libsqlite3")

// LHS が Type → 不透明型としてバインド
SqliteDb: Type = sqlite3("sqlite3")

// LHS が関数シグネチャ → 関数としてバインド
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")
SqliteDb.exec: (sql: String) -> Int32 = sqlite3("sqlite3_exec")
SqliteDb.close: () -> Int32 = sqlite3("sqlite3_close")

// .drop は通常メソッドバインディング（RFC-009 RAII 規約）
SqliteDb.drop = SqliteDb.close
```

コンパイル時検証：`sqlite3("sqlite3_open")` の `sqlite3_open` は `libsqlite3` のシンボルテーブルに存在しなければならない、でなければコンパイルエラー。

#### 1.3 メソッドバインディングと self 位置

`Type.method: (...) -> ...` の記法では、`self` は暗黙的に最初の位置にある——`db.exec("SELECT")` を呼び出す際、`db` は C 関数 `sqlite3_exec` の第 0 引数として渡される。

既に宣言された独立関数をメソッドとしてバインドする必要がある場合、`[N]` 構文で self 位置を指定する（RFC-004 カリー化複数位置バインディング）：

```yaoxiang
// 独立関数
sqlite3_close_v2: (db: SqliteDb) -> Int32 = sqlite3("sqlite3_close_v2")

// メソッドとしてバインド、[0] は db が self であることを示す
SqliteDb.soft_close = sqlite3_close_v2[0]
```

`Native.c(...)` による直接メソッドバインディングと `[N]` による手動バインディングはどちらも `name: type = value` であり、どちらも関数値を `=` の右辺に置く。2 つの機構は存在しない。

#### 1.4 ユーザー使用：unsafe ゼロ、生ポインタゼロ

```yaoxiang
import sqlite3_bindings

db = SqliteDb.open("test.db")
db.exec("SELECT * FROM users")
// ← スコープ終了、RAII が自動的に SqliteDb.drop → sqlite3_close(db) を呼び出す
```

### 2. 型の二分：レイアウト帰属は定義時に固定

外部データが YaoXiang に入る際、問うべきことは一つだけ：**このメモリのレイアウトは誰が決めるのか？**

```
├─ レイアウトが外部のブラックボックス（sqlite3、FILE*、socket fd）
│   → 不透明ハンドル  =  lib("symbol")
│   → YaoXiang はポインタを保持するだけで、参照を解除せず、ライブラリ関数間で受け渡しのみ
│   → 外部コードは自分自身のメモリを読み、YaoXiang は触らない
│
└─ レイアウトが YaoXiang によって定義される（timespec、point、フィールドを読む必要がある struct）
    → 透明型  =  { field: Type, ... }
    → YaoXiang がメモリを所有し、レイアウトを定義し、フィールドを読み書きする
    → 外部コードは YaoXiang が定義したレイアウトのメモリに書き込む/読み取る
```

**第三の型は存在しない。** 以前の設計における「3 層メモリモデル（コピー/引継ぎ/システムレベル）」はパッチ的思考だった——真実はレイアウト帰属の二分である。

#### 2.1 不透明ハンドル：レイアウトは外部に帰属

```yaoxiang
SqliteDb: Type = sqlite3("sqlite3")
```

- YaoXiang の内部にはポインタサイズのハンドルしか保持しない
- ユーザーはコンストラクトできず（`SqliteDb {}` → コンパイルエラー）、フィールドにもアクセスできない（アクセス可能なフィールドがない）
- 唯一のソース：`SqliteDb` を返す外部関数
- メソッド呼び出し時、ハンドルをライブラリに貸し出し、ライブラリが**自身の**メモリ（ライブラリのヒープ上の `sqlite3` 構造体）を読む

外部コードが「内部を読む」のは自身が確保した構造であり、YaoXiang はハンドルを運ぶだけ。メモリの衝突はない。

#### 2.2 透明型：レイアウトは YaoXiang に帰属

```yaoxiang
// フィールドに意味があり、読み書きが必要 → 透明型、レイアウトは YaoXiang が宣言
Timespec: Type = {
    tv_sec: Int64,
    tv_nsec: Int64
}
clock_gettime: (clk: Int32, ts: *Timespec) -> Int32 = Native.c("librt")("clock_gettime")

ts = clock_gettime(CLOCK_REALTIME)   // §3 参照、マーシャリングは一時領域経由
print(ts.tv_sec)                      // YaoXiang は自身のフィールド定義に従って読む
```

外部コードは**YaoXiang がレイアウトを定義し、YaoXiang が所有する**メモリに対して読み書きする。レイアウトは YaoXiang の契約であり、外部のものではない。

#### 2.3 判定規則

ユーザーが判断すべきことは一つだけ：**この型のフィールドを読み取る必要があるか？**

| 判定 | 型 | レイアウト帰属 |
|------|------|---------|
| フィールドを読まず、ライブラリ関数間でハンドルを渡すだけ | 不透明ハンドル `= lib("sym")` | 外部 |
| フィールドを読み書きする必要がある | 透明型 `{ ... }` | YaoXiang |

### 3. マーシャリング：シグネチャ駆動、一時領域による隔離

境界を跨ぐデータ変換は**シグネチャ駆動**であり、コンパイル時に各引数位置の変換規則を確定する。**核心的な安全保証：外部コードが読み書きするのはマーシャリング一時領域であり、YaoXiang のヒープオブジェクトではない。**

#### 3.1 デフォルトは一時領域へのコピー

```
YaoXiang → C（入力引数）：
    データを呼び出し一時領域にコピー → C に一時領域のポインタを渡す
    → C の範囲外アクセス/破壊は一時領域のみを傷つけ、YaoXiang のヒープオブジェクトは隔離

C → YaoXiang（戻り値/出力引数）：
    C が一時領域に書き込む → YaoXiang が memcpy で自身のオブジェクトに戻す
    → C は YaoXiang の最終オブジェクトに触れられない
```

**外部コードは常にマーシャリング一時領域を読み書きし、YaoXiang のヒープオブジェクトと完全に隔離される。** レイアウト宣言の誤り、C がポインタをダングリング化、C の範囲外アクセス——すべて一時領域のみを傷つけ、YaoXiang のオブジェクトは無事である。代償は 1 回の memcpy。

#### 3.2 マーシャリング規則表

**入力引数方向（YaoXiang → C）**：

| YaoXiang 型 | C 表現 | マーシャリング動作 | 所有権 |
|--------------|--------|---------|--------|
| `Int32/Int64/Float` | `int/long/double` | レジスタに直接配置、変換ゼロ | 値セマンティクス |
| `String` | `const char*` | 読み取り専用ビューを貸し出す（一時的、呼び出し中のみ有効） | YaoXiang が保持、C は読み取り専用 |
| 透明型 | `struct T*` | 一時領域にコピー、一時領域のポインタを渡す | YaoXiang がオブジェクトを所有、C はコピーを読む |
| 不透明ハンドル | `void*` | 内部ハンドルのポインタを取り出す | YaoXiang が保持、C に貸し出す |
| `*T` | `T*` | 生ポインタを直接渡す（unsafe） | ユーザーが責任を持つ |

**戻り値方向（C → YaoXiang）**：

| C 戻り値 | YaoXiang 型 | マーシャリング動作 | 所有権 |
|--------|--------------|---------|--------|
| `int/double` | `Int32/Float` | レジスタから直接読む | 値セマンティクス |
| `char*` | `String` | strlen + memcpy で YaoXiang の String にコピー | YaoXiang がコピーを所有、元メモリは触らない |
| `struct T*`（新規ハンドル） | 不透明ハンドル | ハンドルを YaoXiang オブジェクトに格納 | YaoXiang が引継ぐ |
| `struct T`（値/出力引数） | 透明型 | C が一時領域に書き込み → memcpy で YaoXiang に | YaoXiang が所有 |
| `char*`（静的領域） | `*const U8` | 生ポインタを格納、コピーしない（unsafe 読み取り） | 引継がない、ユーザーが責任を持つ |

#### 3.3 借用ライフタイム：単一呼び出しに厳格に限定

YaoXiang が外部コードに貸し出すポインタ（String の読み取り専用ビュー、透明型の一時領域、ハンドル）の**ライフタイムは単一呼び出し内に厳格に限定される**：

- 呼び出し中：ポインタ有効、外部コードは読み書き可能
- 呼び出し終了後：借用は即座に失効

外部コードがポインタを保存して呼び出し後に使用した場合、それは外部コードが FFI 標準契約に違反した（ライブラリのバグと同等）ものであり、YaoXiang はそれに対して責任を負わない。これはすべての言語の C FFI 契約と一致する（Rust の `&T` を C に渡す場合も同様の制約）。

#### 3.4 String は永続ポインタを渡さない

`String` は「C は YaoXiang のメモリに手を出さない」の鍵である：

- C に入る：呼び出し中のみ有効な**一時的読み取り専用ビュー**を貸し出す
- C から出る：strlen + memcpy で YaoXiang が所有する**コピー**に入れる

C は決して YaoXiang の String の永続ポインタを得られず、YaoXiang は C の `char*` の長期参照を保持しない。構造的に隔離される。

### 4. 所有権とライフタイム：Move + RAII

不透明ハンドルは RFC-009 の所有権モデルに従い、新しい概念はゼロ。

#### 4.1 核心原則

- **Move セマンティクス**：不透明ハンドルはデフォルトで Move、代入/引数渡し/戻り値 = 所有権の移動、コピー不可
- **ハンドルの一意所有権**：任意の時点で 1 つのハンドルは 1 つの所有者のみ → 二重解放を構造的に根絶
- **RAII 解放**：スコープ終了時、`.drop` がバインドされていれば自動的に呼び出す
- **消費追跡**：明示的析出または Move 後、変数は消費済み、再利用不可 → use-after-free を根絶

#### 4.2 `.drop` はオプションの外部副作用

```yaoxiang
SqliteDb.drop = SqliteDb.close     // スコープ終了時に sqlite3_close を呼び出す
```

**`.drop` は YaoXiang のリーク防止機構ではない**——YaoXiang 側のハンドル格納（ポインタサイズの値）は自動回収され、`.drop` とは無関係。`.drop` は**スコープ終了時に外部関数を副次的に呼び出す**オプション機能である：

- `.drop` をバインドした → スコープ終了時にそれを呼び出す（外部リソースをクリーンアップ）
- `.drop` をバインドしない → 何もしない、**エラーも警告も出さない**

外部リソースのクリーンアップが必要かどうかは外部ライブラリの仕様問題（`getenv` は静的領域を返すので解放すべきでない、グローバルシングルトンは解放すべきでない）であり、YaoXiang は越権的に強制しない。リーク防止は Move + 一意所有権（無条件、構造的）によって達成され、`.drop` に依存しない。

#### 4.3 自動析出と順序

```yaoxiang
{
    db = SqliteDb.open("test.db")
    stmt = db.prepare("SELECT * FROM users")
    // ← スコープ終了、逆順で自動析出（.drop がバインドされている場合のみ呼び出し）：
    //   stmt.drop()  → sqlite3_finalize(stmt)
    //   db.drop()    → sqlite3_close(db)
}
```

析出順序：定義順序の逆順、RAII と一致する。

#### 4.4 Move と消費

```yaoxiang
db = SqliteDb.open("test.db")
db2 = db                // Move：所有権の移動
db.exec("...")          // ❌ コンパイルエラー：db は Move 済み、消費後は読み取り不可

process_db: (db: SqliteDb) -> Void = {
    db.exec("...")
    // ← 関数終了、db はここで析出
}
process_db(some_db)     // Move して関数に入る
// some_db はここで無効
```

#### 4.5 Null 処理

```yaoxiang
// null を返す可能性あり → ?T、ユーザーは必ず処理
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")

db = SqliteDb.open("test.db")
match db {
    Some(db) => db.exec("SELECT 1"),
    None => print("オープン失敗")
}

// null を返さない規約 → マークしない、null なら panic で露呈
```

C の null 戻り値は、ユーザーが処理する（`?T`）か、panic で露呈するかのどちらか。「暗黙に無視する」という第三の選択肢はない。

#### 4.6 析出失敗の処理

`.drop` にバインドされた関数の戻り値が振る舞いを決定する：

| `.drop` の戻り型 | 振る舞い |
|----------------|------|
| `Void` | 失敗なし |
| `Int32`（エラーコード） | 0 以外なら panic——析出失敗は状態が異常であることを意味し、露呈する方が沈黙より良い |
| `?Error` | None 以外なら panic——同上 |

析出失敗を静かに処理することはできない。特定のエラーを無視する必要がある場合、`.drop` にバインドするラッパ関数で明示的に処理する。

---

### 5. spawn ブロックでの FFI の振る舞い

リソース型の判定は `.drop` バインディングによって決定され（RFC-024）、追加のマーキングは不要：

| 判定 | 振る舞い |
|------|------|
| 不透明ハンドルに `.drop` がバインド済み | リソース型——spawn ブロック内で同じインスタンスへの操作が自動的にシリアライズ |
| 不透明ハンドルに `.drop` が未バインド | 非リソース型——並列可能（純粋なデータハンドル、解放副作用なし） |
| 透明型 / 値型 | 非リソース型——並列可能 |

```yaoxiang
SqliteDb.drop = SqliteDb.close   // → リソース型

(a, b) = spawn {
    r1 = db.exec("SELECT ..."),   // 同じインスタンス、自動的にシリアライズ
    r2 = db.exec("INSERT ...")    // r1 を待つ
}

(x, y) = spawn {
    db1 = SqliteDb.open("a.db"),   // 異なるインスタンス、並列可能
    db2 = SqliteDb.open("b.db")
}
```

`.drop` がバインドされた型は spawn 内で同じインスタンスへの操作が自動的にシリアライズされ、析出時の並行競合がないことを保証する。

---

### 6. エスケープハッチ：生ポインタ + unsafe

デフォルトのマーシャリングは一時領域へのコピーで安全だが、memcpy のオーバーヘッドがある。パフォーマンスが重要なシナリオ（大きな構造、高頻度の呼び出し）でゼロコピーが必要な場合、ユーザーは明示的に生ポインタのエスケープハッチを使用する：

```yaoxiang
// C が YaoXiang のメモリを直接読む、ゼロコピー——ユーザーが明示的にリスクを受け入れる
ptr: *const U8 = Native.c("libc")("getenv")("HOME")
unsafe {
    value = read_c_string(ptr)   // ユーザーが ptr の有効性を保証
}
```

**`unsafe` は生ポインタ操作にのみ使用され、不透明ハンドルや透明型とは完全に直交する。** 通常の FFI（ハンドル + 透明型）には unsafe は不要。`unsafe {}` を書く = ユーザーが直接メモリアクセスのリスクを明確に承認したことになる。

**信頼境界**：C はコンパイル時に検証可能な型契約を提供できない（`.h` は ABI 契約ではなく、シンボルテーブルには名前しかなくシグネチャがない）。したがって C シグネチャの正しさは自動検証不可能——`Native.c(...)` + シグネチャを書く際にバインディング作者が保証する。**信頼はバインディング宣言箇所に局所化される**：バインディング作者が保証し、パッケージユーザーは安全な API を取得する。これは Rust の `extern "C"` と一致する（extern を書くことは信頼行為であり、安全な wrapper で包んだ後は呼び出しが安全）。

---

## トレードオフ

### 利点

1. **情報の完全性**：ライブラリはコンパイル時に link され、シンボルはコンパイル時に検証され、ランタイムの「ライブラリが見つからない」曖昧さがない
2. **レイアウト帰属の明示化**：型の二分により定義時に固定、ランタイム推論なし
3. **構造的な安全性**：一時領域隔離 + Move + RAII、外部コードは YaoXiang のヒープオブジェクトに触れられない
4. **新キーワードゼロ**：`Native.c` カリー化 + `name: type = value`、すべて既存構文を再利用
5. **誠実な境界**：C シグネチャを検証できるかのように装わず、信頼を宣言箇所に局所化

### 欠点

1. **memcpy のオーバーヘッド**：デフォルトのマーシャリングはコピー、大きな構造の高頻度呼び出しには明示的なエスケープハッチが必要
2. **レイアウト保証が手作業**：透明型のレイアウトと C struct の一致はバインディング作者/yx-bindgen が保証
3. **C シグネチャはコンパイル時に検証不可能**：FFI の根本的制限、YaoXiang は C からこれを除去できない

## 実装戦略

### フェーズ 1：外部ライブラリとシンボル (v0.8)

- [ ] `Native.c("lib")` のコンパイル時 link + パーサ値返却を実装
- [ ] シンボルパーサの適用（`lib("symbol")`）+ コンパイル時シンボルテーブル検証を実装
- [ ] 型の二分（不透明ハンドル / 透明型）を実装
- [ ] メソッドバインディング（直接バインディング + `[N]` 位置バインディング）を実装

### フェーズ 2：マーシャリングと安全性 (v0.8)

- [ ] シグネチャ駆動のマーシャリングコード生成を実装
- [ ] 一時領域コピー隔離（入力引数コピー、戻り値 memcpy）を実装
- [ ] String の一時読み取り専用ビュー + 戻り値コピーを実装
- [ ] 借用ライフタイムを単一呼び出しに限定することを実装

### フェーズ 3：所有権とライフタイム (v0.9)

- [ ] 不透明ハンドルの Move + 一意所有権を実装
- [ ] `.drop` RAII 自動析出を実装（オプション、欠落時にエラーなし）
- [ ] 消費追跡（Move 後無効化）を実装
- [ ] `?T` と null 戻り値統合を実装
- [ ] spawn リソース型のシリアライズを実装

### 後続作業

- **拡張可能 FFI メカニズム**（RFC-026a）：`FfiMechanism` 抽象、`.wasm`/`.python` 等のプラグイン、動的ロード

## 設計意思決定記録

| 決定 | 決定内容 | 理由 | 日付 |
|------|------|------|------|
| **ライブラリ＝値** | `Native.c("lib")` カリー化でパーサを返す | ライブラリ情報がコンパイル時に可視な一次値となり、「どのライブラリをリンクするか」のギャップを埋め、新キーワードゼロ | 2026-07-03 |
| **コンパイル時 link** | `Native.c("lib")` が `-llib` をトリガ | シンボルテーブルがコンパイル時に読み取り可能、シンボル存在性を検証可能、型が実在する | 2026-07-03 |
| **型の二分** | 不透明ハンドル / 透明型 | レイアウト帰属の二分がすべてをカバー；「3 層メモリモデル」パッチを削除 | 2026-07-03 |
| **マーシャリング一時領域隔離** | デフォルトはコピー、ヒープオブジェクトと外部を隔離 | 外部の範囲外アクセス/ダングリングは一時領域のみを傷つけ、YaoXiang オブジェクトは無事；ゼロコピーには明示的エスケープハッチが必要 | 2026-07-03 |
| **`.drop` オプション** | 欠落時は何もしない、エラーなし | YaoXiang のハンドル格納は自動回収；外部リソースのクリーンアップは外部仕様であり、越権的に強制しない | 2026-07-03 |
| **リーク防止機構** | Move + ハンドル一意所有権（無条件） | 構造的保証、`.drop` と無関係 | 2026-07-03 |
| **信頼境界** | `Native.c(...)` 宣言箇所 | C シグネチャはコンパイル時に検証不可能、信頼を局所化、unsafe は生ポインタにのみ | 2026-07-03 |
| **Null 処理** | `?T` または panic | C の問題を隠さない、「暗黙に無視」オプションなし | 2026-07-03 |
| **析出失敗** | `.drop` の戻り型で決定、統一して panic | 析出失敗を静かに処理することはできない | 2026-07-03 |
