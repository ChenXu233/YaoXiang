## 要約

このドキュメントはYaoXiangのFFI（外部関数インターフェース）コア機構を定義する。核心的な考え方：**外部ライブラリはコンパイル時リンクのファーストクラス値であり、境界を跨ぐデータのメモリレイアウト帰属は型定義時に固定され、YaoXiangのヒープオブジェクトと外部コードは構造的に隔離される。**

1. **外部ライブラリは値である**：`Native.c("libsqlite3")` でコンパイル時にライブラリをリンクし、解析器を返す。カリー化でライブラリ情報を保持する
2. **外部シンボルは値である**：解析器にシンボル名を適用して外部参照を取得し、`name: type = value`（RFC-007/010）で型または関数としてバインドする
3. **型の二分類**：不透明ハンドル（レイアウトは外部に帰属）/ 透明型（レイアウトはYaoXiangに帰属）、第三の選択肢は存在しない
4. **マーシャリング隔離**：境界を跨ぐデータはデフォルトで呼び出し用テンポラリ領域にコピーされ、YaoXiangヒープオブジェクトと外部コードは隔離される
5. **所有権の安全性**：ハンドルの唯一の所有権（Move）+ RAIIにより、二重解放とuse-after-freeを構造的に杜绝
6. **エスケープハッチ**：`*T` 生のポインタ + `unsafe {}` で、ユーザが明示的にゼロコピー直接メモリアクセスのリスクを引き受ける

**核心的境界——5つの不可侵契約**：

```
1. ライブラリはコンパイル時リンク、シンボルはコンパイル時に存在を検証
2. 型レイアウト帰属は定義時に決定：不透明ハンドルは外部、透明型はYaoXiang
3. デフォルトのマーシャリングはテンポラリ領域経由のコピーで、YaoXiangヒープオブジェクトと外部コードは隔離
4. ハンドルの唯一の所有権 + Move、二重解放/ダングリングを構造的に防止
5. 外部コードは常に「レイアウト明確、所有権明確」なメモリを読み書きし、曖昧な領域は存在しない
```

## 動機

### 現状と目標

現在のコードベースの `native("symbol")` はYaoXiangバイトコードがRust std関数を呼び出すディスパッチ機構（`FfiRegistry` = `HashMap<String, RustFnPtr>`）に過ぎず、**真のABI境界を跨ぐ仕組みは存在しない**——dlopen もなければ、C ABIマーシャリングも、メモリ所有権の境界横断もない。

真のFFIは4つの問題を解決しなければならない：

| 問題 | 本RFCの回答 |
|------|--------------|
| **シンボル解決** | ライブラリはコンパイル時リンクのファーストクラス値（`Native.c("lib")`）、シンボルはコンパイル時検証 |
| **値マーシャリング** | シグネチャ駆動で、各引数位置ごとにコンパイル時に変換規則を決定 |
| **メモリ所有権** | 型の二分類で帰属を決定；デフォルトはコピーで隔離 |
| **ライフサイクル安全性** | Move + RAII + 借用は単一呼び出しに限定 |

RFC-020とRFC-021はそれぞれFFIの異なる側面を定義しており、重複があるため、本ドキュメントで統一規範に統合する。

### 設計目標

1. **生のポインタをユーザコードに漏らさない**：通常のFFI利用では `.yx` ソースに生ポインタが現れない
2. **レイアウト帰属を明示的に**：ユーザが型を定義する時点でこのメモリが誰に帰属するかを決定し、ランタイム推論に依存しない
3. **構造的安全性**：リーク、二重解放、use-after-freeを型システムで保証し、規約に依存しない
4. **正直な信頼境界**：Cはコンパイル時検証可能な型契約を提供できないため、信頼をバインディング宣言箇所に局所化
5. **セルフホスティング互換**：ホスト言語特有の過剰な抽象化を行わない

### 範囲外

- **複数ABI機構プラグイン体系**（Wasm/Python/カスタムABI）：RFC-026a参照
- **yx-bindgen ツールチェーン**：RFC-026b参照
- **YaoXiangからCへの関数エクスポート（逆方向FFI）**：後続RFC、本ドキュメントは原則のみ宣言
- **インラインアセンブリ、SIMD組み込み関数**：本RFCの範囲外

## 提案

### 1. 外部ライブラリとシンボル：カリー化されたファーストクラス値

FFIの情報欠落——「どのライブラリ、どのシンボルをリンクするか」——を、新しいキーワードを導入せず、ライブラリをファーストクラス値にすることで埋める。

#### 1.1 ライブラリは値

```yaoxiang
// Native.c にライブラリ名を適用 → コンパイル時にライブラリをリンクし、シンボル解析器を返す
sqlite3 = Native.c("libsqlite3")
```

`Native.c("libsqlite3")` は**コンパイル時アクション + ランタイム値**：

- **コンパイル時**：リンカが `-lsqlite3` を発行し、ライブラリがシンボルテーブルに入り、シンボルの存在を検証可能
- **値**：`sqlite3` は解析器であり、シンボル名を適用するとそのライブラリの外部参照を取得する

`.c` はABI機構タグ（C ABI）である。コアには `.c` のみ組み込み；他の機構（`.wasm` など）についてはRFC-026a参照。

#### 1.2 シンボルは値、バインディングは `name: type = value`

解析器にシンボル名を適用して得られる外部参照は、RFC-007/010の統一構文でバインドされる。**LHSの型注釈がこの参照を型にするか関数にするかを決定する**：

```yaoxiang
sqlite3 = Native.c("libsqlite3")

// LHSがType → 不透明型としてバインド
SqliteDb: Type = sqlite3("sqlite3")

// LHSが関数シグネチャ → 関数としてバインド
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")
SqliteDb.exec: (sql: String) -> Int32 = sqlite3("sqlite3_exec")
SqliteDb.close: () -> Int32 = sqlite3("sqlite3_close")

// .dropは通常メソッドバインディング（RFC-009 RAII規約）
SqliteDb.drop = SqliteDb.close
```

コンパイル時検証：`sqlite3("sqlite3_open")` の `sqlite3_open` は `libsqlite3` のシンボルテーブルに存在しなければならず、そうでなければコンパイルエラー。

#### 1.3 メソッドバインディングとself位置

`Type.method: (...) -> ...` の記法では、`self` は暗黙的に最初の位置にある——`db.exec("SELECT")` を呼び出す際、`db` はC関数 `sqlite3_exec` の第0引数として渡される。

既に宣言された独立関数をメソッドとしてバインドしたい場合は、`[N]` 構文でself位置を指定する（RFC-004 カリー化多位置バインディング）：

```yaoxiang
// 独立関数
sqlite3_close_v2: (db: SqliteDb) -> Int32 = sqlite3("sqlite3_close_v2")

// メソッドとしてバインド、[0] は db が self であることを示す
SqliteDb.soft_close = sqlite3_close_v2[0]
```

`Native.c(...)` の直接メソッドバインディングと `[N]` 手動バインディングはどちらも `name: type = value` であり、関数値を `=` の右辺に置くだけなので、機構は1つしかない。

#### 1.4 ユーザの使用法：ゼロunsafe、ゼロ生ポインタ

```yaoxiang
import sqlite3_bindings

db = SqliteDb.open("test.db")
db.exec("SELECT * FROM users")
// ← スコープ終了、RAIIにより自動的に SqliteDb.drop → sqlite3_close(db) を呼び出し
```

### 2. 型の二分類：レイアウト帰属は定義時に固定

外部データがYaoXiangに入る際に問うのは一つだけ：**このメモリのレイアウトは誰が決めるのか？**

```
├─ レイアウトは外部のブラックボックス（sqlite3、FILE*、socket fd）
│   → 不透明ハンドル  =  lib("symbol")
│   → YaoXiangはポインタを保持するだけで、参照外しせず、ライブラリ関数間でのみ受け渡し
│   → 外部コードが自身のメモリを読み書き、YaoXiangは触らない
│
└─ レイアウトはYaoXiangが定義する（timespec、point、フィールドを読むstruct）
    → 透明型  =  { field: Type, ... }
    → YaoXiangがメモリを所有し、レイアウトを定義し、フィールドを読み書き
    → 外部コードはYaoXiangがレイアウトを定義済みのメモリに書き込み/読み出しする
```

**第三の選択肢はない。** 以前の設計にあった「3層メモリモード（コピー/引き継ぎ/システムレベル）」はパッチ的思考であり、真実はレイアウト帰属の二分類である。

#### 2.1 不透明ハンドル：レイアウトは外部に帰属

```yaoxiang
SqliteDb: Type = sqlite3("sqlite3")
```

- YaoXiang内部にはポインタサイズのハンドルのみを保持
- ユーザは構築不可（`SqliteDb {}` → コンパイルエラー）、フィールドアクセス不可（アクセス可能なフィールドがない）
- 唯一のソース：`SqliteDb` を返す外部関数
- メソッド呼び出し時はハンドルをライブラリに貸し戻し、ライブラリが**自身の**メモリ（ライブラリのヒープ上の `sqlite3` 構造体）を読み取る

外部コードが「内部を読む」のは自身が確保した構造体であり、YaoXiangはハンドルを運ぶだけ。メモリの衝突は発生しない。

#### 2.2 透明型：レイアウトはYaoXiangに帰属

```yaoxiang
// フィールドに意味があり、読み書きが必要 → 透明型、レイアウトはYaoXiangが宣言
Timespec: Type = {
    tv_sec: Int64,
    tv_nsec: Int64
}
clock_gettime: (clk: Int32, ts: *Timespec) -> Int32 = Native.c("librt")("clock_gettime")

ts = clock_gettime(CLOCK_REALTIME)   // §3参照、マーシャリングはテンポラリ領域経由
print(ts.tv_sec)                      // YaoXiangが自身のフィールド定義に従って読む
```

外部コードは**YaoXiangがレイアウトを定義し、YaoXiangが所有する**メモリに読み書きする。レイアウトはYaoXiangの契約であり、外部のものではない。

#### 2.3 判定ルール

ユーザが判断すべきことは一つだけ：**この型のフィールドを読みたいか？**

| 判定 | 型 | レイアウト帰属 |
|------|------|---------|
| フィールドを読まず、ライブラリ関数間でハンドルを渡すだけ | 不透明ハンドル `= lib("sym")` | 外部 |
| フィールドを読み書きしたい | 透明型 `{ ... }` | YaoXiang |

### 3. マーシャリング：シグネチャ駆動、テンポラリ領域隔離

境界を跨ぐデータ変換は**シグネチャ駆動**で、コンパイル時に各引数位置ごとに変換規則を決定する。**核心的な安全性保証：外部コードが読み書きするのはマーシャリング用テンポラリ領域であり、YaoXiangのヒープオブジェクトではない。**

#### 3.1 デフォルトはテンポラリ領域経由のコピー

```
YaoXiang → C（入力引数）：
    データを呼び出し用テンポラリ領域にコピー → テンポラリ領域のポインタをCに渡す
    → Cの境界外アクセス/破壊書き込みはテンポラリ領域のみ傷つけ、YaoXiangヒープオブジェクトは隔離

C → YaoXiang（戻り値/出力引数）：
    Cがテンポラリ領域に書き込み → YaoXiangが memcpy で自身のオブジェクトにコピー
    → CはYaoXiangの最終オブジェクトに触れられない
```

**外部コードは常にマーシャリング用テンポラリ領域を読み書きし、YaoXiangヒープオブジェクトと完全に隔離される。** レイアウト宣言の誤り、Cがポインタをダングリング化、Cの境界外アクセス——すべてテンポラリ領域のみを傷つけ、YaoXiangのオブジェクトは無事である。代償は1回のmemcpy。

#### 3.2 マーシャリング規則表

**入力引数方向（YaoXiang → C）**：

| YaoXiang型 | C表現 | マーシャリング動作 | 所有権 |
|--------------|--------|---------|--------|
| `Int32/Int64/Float` | `int/long/double` | 直接レジスタに格納、変換ゼロ | 値セマンティクス |
| `String` | `const char*` | 読み取り専用ビューを貸し出す（一時的、呼び出し期間中有効） | YaoXiangが保持、Cは読み取りのみ |
| 透明型 | `struct T*` | テンポラリ領域にコピーし、テンポラリ領域のポインタを渡す | YaoXiangがオブジェクトを所有、Cはコピーを読む |
| 不透明ハンドル | `void*` | 内部ハンドルのポインタを取り出す | YaoXiangが保持、Cに貸し出す |
| `*T` | `T*` | 生ポインタを直接渡す（unsafe） | ユーザ責任 |

**戻り値方向（C → YaoXiang）**：

| C戻り値 | YaoXiang型 | マーシャリング動作 | 所有権 |
|--------|--------------|---------|--------|
| `int/double` | `Int32/Float` | レジスタから直接読む | 値セマンティクス |
| `char*` | `String` | strlen + memcpy で YaoXiang String にコピー | YaoXiangがコピーを所有、元メモリには触れない |
| `struct T*`（新規ハンドル） | 不透明ハンドル | ハンドルをYaoXiangオブジェクトに格納 | YaoXiangが引き継ぎ |
| `struct T`（値/出力引数） | 透明型 | Cがテンポラリ領域に書き込み → memcpy でYaoXiangに戻す | YaoXiangが所有 |
| `char*`（静的領域） | `*const U8` | 生ポインタを格納、コピーしない（unsafe読み取り） | 引き継ぎなし、ユーザ責任 |

#### 3.3 借用ライフサイクル：単一呼び出しに厳格に限定

YaoXiangが外部コードに貸し出すポインタ（String読み取り専用ビュー、透明型テンポラリ領域、ハンドル）の**ライフサイクルは単一呼び出し内に厳格に限定**：

- 呼び出し期間中：ポインタ有効、外部コードは読み書き可
- 呼び出し終了後：借用は即座に失効

外部コードがポインタを保存して呼び出し後に使用する場合、それは外部コードがFFI標準契約に違反した（ライブラリのバグと同等）ものであり、YaoXiangは責任を負わない。これは全言語のC FFI契約と一致する（Rustの `&T` をCに渡すのも同じ制約）。

#### 3.4 Stringは永続ポインタを渡さない

`String` は「CがYaoXiangメモリに介入しない」ための鍵：

- Cに入る：**一時的読み取り専用ビュー**を貸し出し、呼び出し期間中有効
- Cから出る：strlen + memcpy で YaoXiangが所有する**コピー**に格納

CはYaoXiang Stringの永続ポインタを取得できず、YaoXiangはCの `char*` の長期参照を保持しない。構造的に隔離される。

### 4. 所有権とライフサイクル：Move + RAII

不透明ハンドルはRFC-009の所有権モデルに従い、新たな概念はゼロ。

#### 4.1 核心原則

- **Moveセマンティクス**：不透明ハンドルはデフォルトでMove、代入/引数渡し/戻り値 = 所有権の移動、コピー不可
- **ハンドルの唯一の所有権**：任意の時点で1つのハンドルは1人の所有者のみ → 二重解放を構造的に杜绝
- **RAII解放**：スコープ終了時、`.drop` がバインドされていれば自動呼び出し
- **消費追跡**：明示的析出またはMove後、変数は消費済みで再利用不可 → use-after-freeを杜绝

#### 4.2 `.drop` は任意の外部副作用

```yaoxiang
SqliteDb.drop = SqliteDb.close     // スコープ終了時に sqlite3_close を呼び出し
```

**`.drop` はYaoXiangのリーク防止機構ではない**——YaoXiang側のハンドル格納（ポインタサイズの値）は自動回収され、`.drop` とは無関係。`.drop` は**スコープ終了時に外部関数をついでに呼び出す**任意の副作用である：

- `.drop` をバインドした → スコープ終了時に呼び出し（外部リソースをクリーンアップ）
- `.drop` をバインドしない → 何もしない、**エラーも警告も出さない**

外部リソースのクリーンアップが必要かどうかは外部ライブラリの仕様次第（`getenv` は静的領域を返すため解放不要、グローバルシングルトンは解放不要）であり、YaoXiangは越権して強制しない。リーク防止はMove + 唯一の所有権（無条件、構造的）に依存し、`.drop` には依存しない。

#### 4.3 自動析出と順序

```yaoxiang
{
    db = SqliteDb.open("test.db")
    stmt = db.prepare("SELECT * FROM users")
    // ← スコープ終了、定義の逆順で自動析出（.drop がバインドされている場合のみ呼び出し）：
    //   stmt.drop()  → sqlite3_finalize(stmt)
    //   db.drop()    → sqlite3_close(db)
}
```

析出順序：定義順序の逆順、RAIIと一致。

#### 4.4 Moveと消費

```yaoxiang
db = SqliteDb.open("test.db")
db2 = db                // Move：所有権の移動
db.exec("...")          // ❌ コンパイルエラー：db はMove済み、消費後は読み取り不可

process_db: (db: SqliteDb) -> Void = {
    db.exec("...")
    // ← 関数終了、db はここで析出
}
process_db(some_db)     // 関数にMove
// some_db はここでは無効
```

#### 4.5 Null処理

```yaoxiang
// nullを返す可能性あり → ?T、ユーザは必ず処理
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")

db = SqliteDb.open("test.db")
match db {
    Some(db) => db.exec("SELECT 1"),
    None => print("オープン失敗")
}

// nullを返さないと規約 → マークなし、null時はpanicで露呈
```

Cがnullを返す場合、ユーザが処理する（`?T`）か、panicで露呈する。「黙って無視する」という第3の選択肢はない。

#### 4.6 析出失敗処理

`.drop` にバインドされた関数の戻り値が動作を決定：

| `.drop` 戻り型 | 動作 |
|----------------|------|
| `Void` | 失敗なし |
| `Int32`（エラーコード） | 非0ならpanic——析出失敗は状態が異常であることを意味し、沈黙より露呈が望ましい |
| `?Error` | None以外ならpanic——同上 |

析出失敗を沈黙させることは不可。特定のエラーを無視したい場合は、`.drop` にバインドするラッパ関数で明示的に処理する。

### 5. spawnブロック内のFFI動作

リソース型判定は `.drop` バインディングで決定し（RFC-024）、追加のマークは不要：

| 判定 | 動作 |
|------|------|
| `.drop` をバインドした不透明ハンドル | リソース型——spawnブロック内で同一インスタンスの操作は自動的にシリアライズ |
| `.drop` をバインドしていない不透明ハンドル | 非リソース型——並列実行可（純粋なデータハンドル、解放副作用なし） |
| 透明型 / 値型 | 非リソース型——並列実行可 |

```yaoxiang
SqliteDb.drop = SqliteDb.close   // → リソース型

(a, b) = spawn {
    r1 = db.exec("SELECT ..."),   // 同一インスタンス、自動的にシリアライズ
    r2 = db.exec("INSERT ...")    // r1 を待機
}

(x, y) = spawn {
    db1 = SqliteDb.open("a.db"),   // 異なるインスタンス、並列実行可
    db2 = SqliteDb.open("b.db")
}
```

`.drop` 付きの型はspawn内で同一インスタンスの操作を自動的にシリアライズし、析出時の並行競合を保証する。

---

### 6. エスケープハッチ：生ポインタ + unsafe

デフォルトのマーシャリングはテンポラリ領域経由のコピーで安全だが、memcpyのオーバーヘッドがある。パフォーマンスが重要なシナリオ（大きな構造、高頻度呼び出し）でゼロコピーが必要な場合、ユーザは明示的に生ポインタのエスケープハッチを使用する：

```yaoxiang
// CがYaoXiangメモリを直接読む、ゼロコピー——ユーザが明示的にリスクを引き受ける
ptr: *const U8 = Native.c("libc")("getenv")("HOME")
unsafe {
    value = read_c_string(ptr)   // ユーザが ptr の有効性を保証
}
```

**`unsafe` は生ポインタ操作にのみ使用され、不透明ハンドルや透明型とは完全に直交する。** 通常のFFI（ハンドル + 透明型）には unsafe は不要。`unsafe {}` を書く = ユーザが明確に署名して直接メモリアクセスのリスクを引き受ける。

**信頼境界**：Cはコンパイル時検証可能な型契約を提供できない（`.h` はABI契約ではなく、シンボルテーブルは名前のみでシグネチャを含まない）。したがってCシグネチャの正当性は自動検証できず、結合作者が `Native.c(...)` + シグネチャを書く際に保証する。**信頼はバインディング宣言箇所に局所化**：バインディング作者が保証し、パッケージユーザは安全なAPIを受け取る。これはRustの `extern "C"` と一致する（externを書くのは信頼行為、安全ラッパで包んだ後の呼び出しは安全）。

## トレードオフ

### 利点

1. **情報の完全性**：ライブラリはコンパイル時リンク、シンボルはコンパイル時検証、ランタイムの「ライブラリが見つからない」という曖昧さがない
2. **レイアウト帰属の明示性**：型の二分類、定義時に固定、ランタイム推論なし
3. **構造的安全性**：テンポラリ領域隔離 + Move + RAII、外部コードはYaoXiangヒープオブジェクトに触れられない
4. **新キーワードゼロ**：`Native.c` カリー化 + `name: type = value`、すべて既存構文を再利用
5. **正直な境界**：Cシグネチャを検証可能と偽らず、信頼を宣言箇所に局所化

### 欠点

1. **memcpyオーバーヘッド**：デフォルトのマーシャリングはコピー、大きな構造の高頻度呼び出しには明示的にエスケープハッチが必要
2. **レイアウト保証は手作業**：透明型レイアウトとC structの整合性はバインディング作者/yx-bindgenが保証
3. **Cシグネチャはコンパイル時検証不可**：FFIの根本的制限、YaoXiangはCからこれを除去できない

## 設計決定記録

| 決定 | 決定内容 | 理由 | 日付 |
|------|------|------|------|
| **ライブラリは値** | `Native.c("lib")` のカリー化で解析器を返す | ライブラリ情報がコンパイル時に見えるファーストクラス値になり、「どのライブラリをリンクするか」の欠落を埋め、新キーワードゼロ | 2026-07-03 |
| **コンパイル時リンク** | `Native.c("lib")` が `-llib` をトリガー | シンボルテーブルがコンパイル時に読め、シンボル存在を検証可能、型は実在 | 2026-07-03 |
| **型の二分類** | 不透明ハンドル / 透明型 | レイアウト帰属の二分類で全ケースを網羅；「3層メモリモード」のパッチを削除 | 2026-07-03 |
| **マーシャリングテンポラリ領域隔離** | デフォルトはコピー、ヒープオブジェクトは外部と隔離 | 外部の境界外アクセス/ダングリングはテンポラリ領域のみ傷つけ、YaoXiangオブジェクトは無事；ゼロコピーには明示的なエスケープハッチが必要 | 2026-07-03 |
| **`.drop` は任意** | 欠落時は何もしず、エラーもなし | YaoXiangハンドル格納は自動回収；外部リソースのクリーンアップは外部仕様、越権して強制しない | 2026-07-03 |
| **リーク防止機構** | Move + ハンドルの唯一の所有権（無条件） | 構造的保証、`.drop` とは無関係 | 2026-07-03 |
| **信頼境界** | `Native.c(...)` 宣言箇所 | Cシグネチャはコンパイル時検証不可、信頼を局所化、unsafe は生ポインタのみ | 2026-07-03 |
| **Null処理** | `?T` またはpanic | Cの問題を隠さず、「黙って無視」オプションなし | 2026-07-03 |
| **析出失敗** | `.drop` 戻り型で決定、統一してpanic | 析出失敗を沈黙させることは不可 | 2026-07-03 |
