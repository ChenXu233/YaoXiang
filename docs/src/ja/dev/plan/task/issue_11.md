# タスク：LSP（Language Server Protocol）サポートの実装

## 概要

YaoXiang の開発者体験の向上のため、Language Server Protocol に準拠した言語サーバーを実装する。当該サーバーはコンパイラ前端に組み込まれ、リアルタイムのコード分析、补完、ジャンプ、診断などの機能を提供し、主要なエディタ（VS Code、Vim、Emacs など）との統合を実現する。

## 目標

- YaoXiang 言語サーバー、バイナリ `yaoxiang-lsp` を実装する。
- 以下の LSP リクエストに対応する：
  - `textDocument/didOpen` / `didChange` / `didSave` / `didClose`：ファイル内容を同期する。
  - `textDocument/documentSymbol`：ドキュメントシンボル（関数、型、変数など）を一覧表示する。
  - `textDocument/completion`：スコープ内シンボルに基づくコード补完候補を提供する。
  - `textDocument/definition`：定義位置への跳转を提供する。
  - `textDocument/hover`：ホバー時に型情報とドキュメントコメントを表示する。
  - `textDocument/publishDiagnostics`：リアルタイムで構文エラー、型エラーなどの診断情報をプッシュする。
- コンパイラの增量コンパイル/チェック機能を実装し、毎回完全な解析を避ける。

## 具体的な手順

### 1. サーバ架構の設計（1週間）

- 既存のコンパイラ前端ライブラリ（字句解析、解析、型チェック）を基盤として構築する。
- 非同期 I/O（Tokio など）を採用して LSP リクエストを処理する。
- ドキュメントキャッシュと增量更新のデータ構造を設計する。

### 2. 基本 LSP プロトコル通信の実装（1週間）

- JSON-RPC メッセージの送受信を実装する。
- 初期化リクエスト、シャットダウンリクエストなどのライフサイクル管理を処理する。

### 3. コンパイラ前端の統合（2週間）

- 字句解析、解析、型チェックを再入可能なライブラリ関数として封装する。
- 各ドキュメントに対して「コンパイルセッション」を維持し、增量更新（変更部分のみを再解析）をサポートする。
- AST ノードからソースコード位置へのマッピングを実装し、ジャンプやホバーに利用する。

### 4. 診断機能の実装（1週間）

- 型チェック的过程中でのエラーと警告を捕获し、LSP 診断メッセージに変換する。
- ファイル修改後に自動的に診断をプッシュする。

### 5. 补完とシンボルナビゲーションの実装（2週間）

- スコープシンボルテーブルを構築し、补完候補リストを提供する（キーワード、組み込み型、現在のスコープ変数などを含む）。
- 定義への跳转を実装する：カーソル位置に基づいて、シンボル定義のファイルと位置を見つける。
- ホバー情報を実装する：型シグネチャと関連するドキュメントコメント（もしサポートされていれば）を表示する。

### 6. テストと統合（1週間）

- テスト用に VS Code クライアントプラグイン（単純なラッパー）を作成する。
- すべての機能が一般的なシナリオで使用可能であることを確認し、性能上の明显的な問題がないことを保証する。

## 検収基準

- サーバーが正しく起動し、初期化リクエストに応答できること。
- YaoXiang ファイルを開いた後、診断情報（構文エラー、型エラー）がエディタにリアルタイム表示されること。
- 変数または関数名にホバーすると、型情報が表示されること。
- 識別子で定義にジャンプすると、対応するファイルが開かれ、定義行にカーソルが移動すること。
- ドットを入力した後、レコード型のフィールドを补完できること（型情報が必要）。
- サーバーが大型ファイル（数千行）で許容可能な応答時間（<500ms）であること。

## 依存関係

- コンパイラ前端ライブラリは安定しており、AST、シンボルテーブル、型情報を取得するための API を提供する必要がある。
- 增量コンパイル機能（少なくともドキュメントレベルのキャッシュ）の実装が必要である。
