# Task: LSP（Language Server Protocol）サポートの実装

## 概要

YaoXiang の開発者体験向上のため、Language Server Protocol に準拠した言語サーバーを実装する。当該サーバーはコンパイラ前端に組み込まれ、リアルタイムのコード分析、补全（自動補完）、ジャンプ、診断などの機能を提供し、主流エディタ（VS Code、Vim、Emacs など）との統合を実現する。

## 目標

- YaoXiang 言語サーバー바이너리 `yaoxiang-lsp` を実装する。
- 以下の LSP リクエストをサポートする：
  - `textDocument/didOpen` / `didChange` / `didSave` / `didClose`：ファイル内容を同期する。
  - `textDocument/documentSymbol`：ドキュメントシンボル（関数、型、変数など）を一覧表示する。
  - `textDocument/completion`：スコープ内シンボルに基づくコード补全候補を提供する。
  - `textDocument/definition`：定義位置へのジャンプを提供する。
  - `textDocument/hover`：ホバー時に型信息和ドキュメントコメントを表示する。
  - `textDocument/publishDiagnostics`：リアルタイムに構文エラー、型エラーなどの診断情報をプッシュする。
- コンパイラの增量コンパイル/チェック機能を実装し、都度全量解析を回避する。

## 具体的な手順

1. **サーバーアーキテクチャの設計**（1 週間）
   - 既存のコンパイラ前端ライブラリ（字句解析、パーシング、型チェック）をベースとして構築する。
   - 非同期 I/O（Tokio など）を使用して LSP リクエストを処理する。
   - ドキュメントキャッシュと增量更新のデータ構造を設計する。

2. **基本 LSP プロトコル通信の実装**（1 週間）
   - JSON-RPC メッセージの送受信を実装する。
   - 初期化リクエスト、关闭リクエストなどのライフサイクル管理を処理する。

3. **コンパイラ前端との統合**（2 週間）
   - 字句解析、パーシング、型チェックを再入可能なライブラリ関数として封装する。
   - 各ドキュメントに対して「コンパイルセッション」を维持し、增量更新（修正された部分のみ再解析）をサポートする。
   - ジャンプやホバーのため、AST ノードからソースコード位置へのマッピングを実装する。

4. **診断機能の実装**（1 週間）
   - 型チェック 工程で発生したエラーと警告をキャプチャし、LSP 診断メッセージに変換する。
   - ファイル修正後に自動的に診断をプッシュする。

5. **补全とシンボルナビゲーションの実装**（2 週間）
   - スコープシンボルテーブルを構築し、补全に候補リストを提供する（キーワード、組み込み型、現在のスコープ変数などを含む）。
   - 定義へのジャンプを実装する：カーソル位置に基づいてシンボル定義のファイルと位置を見つける。
   - ホバー情報を実装する：型シグネチャと関連するドキュメントコメント（サポートされている場合）を表示する。

6. **テストと統合**（1 週間）
   - テスト用の VS Code クライアント插件（シンプルなラッパー）を書く。
   - 一般的なシナリオですべての機能が利用可能であること、パフォーマンス上の明らかな問題がないことを確認する。

## 验收基準

- サーバーが正常に起動し、初期化リクエストに応答できること。
- YaoXiang ファイルを開いた後、診断情報（構文エラー、型エラー）がエディタにリアルタイムで表示されること。
- 変数または関数名にホバーすると、型情報が表示されること。
- 識別子から定義にジャンプすると、対応するファイルが開かれ、定義行にカーソルが移動すること。
- ドット（.）を入力すると、レコード型のフィールドが补完できること（型情報が必要）。
- サーバーが大型ファイル（数千行）でも許容可能な応答時間（<500ms）であること。

## 依存関係

- コンパイラ前端ライブラリは安定しており、AST、シンボルテーブル、型情報を取得するための API を提供していること。
- 增量コンパイル機能（少なくともドキュメントレベルのキャッシュ）の実装が必要である。
