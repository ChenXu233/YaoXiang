## 概要

`yaoxiang check` は、YaoXiang ソースコードに対して静的検査（型検査、所有権検査、lint）を実行するコマンドラインツールであり、コードは一切生成しません。高速なフィードバックを提供し、CI ワークフローへの統合や開発過程での検証ステップとして適しています。

## 目標

- 1 つまたは複数のファイル/ディレクトリを入力として受け付ける `yaoxiang check` コマンドを実装する。
- コンパイラの型検査モジュールを再利用し、エラーと警告を標準エラー出力に出力する。
- `--json` オプションをサポートし、JSON 形式で診断情報を出力することで、他のツールとの統合を容易にする。
- `--watch` モードをサポート（任意）、ファイルの変更を監視して再検査を行う。
- エラーメッセージには、ファイル名、行番号、列番号、エラー等級、説明を含める。

## 具体的な手順

### 1. コマンドラインインターフェースの設計（3 日）

- `clap` または同様のライブラリを使用して引数を解析する: `yaoxiang check [OPTIONS] <PATH>...`。
- `--json`、`--watch`、`--color` などのオプションをサポート。

### 2. 型検査器との統合（1 週間）

- 型検査モジュールを独立して呼び出し可能な関数としてラップし、ソースコード文字列を受け取り、診断リストを返す。
- 複数ファイル検査をサポート: すべてのファイルを解析し、グローバルな symbol table を構築する必要がある。
- 標準ライブラリの paths と組み込み定義を処理する。

### 3. 出力フォーマットの実装（3 日）

- Rustc と同様のカラーエラー出力を実装し、エラーの位置、エラー等級、説明を含める。
- JSON フォーマット出力を実装し、一般的な仕様（`rustc --error-format=json` のような）に準拠する。

### 4. 監視モードのサポート（任意）（1 週間）

- ファイルシステム監視ライブラリ（`notify` など）を使用してファイルの変更を監視し、自動的に再検査を行う。
- 繰り返し全量検査を避け、可能な限り増分検査を行う。

### 5. テストと統合（1 週間）

- テストケースを作成: エラーのあるファイルに対して検査を実行し、出力が期待通りか検証する。
- CI ワークフローに統合し、毎コミットで検査が通ることを保証する。

## 受け入れ基準

- エラーのない単一ファイルに対して `yaoxiang check` を実行すると、出力がない（または成功メッセージ）、終了コードは 0 である。
- 型エラーが含まれるファイルに対して検査を実行すると、明確なエラー情報（ファイル名、行番号、列番号、説明）を出力し、終了コードは 0 以外である。
- `--json` オプションは有効な JSON 配列を出力し、各要素にはエラーの位置とメッセージが含まれる。
- 検査速度はコンパイラの型検査速度に近く、頻繁な使用に適している。
- プロジェクト内の複数のファイルを正しく処理し、ファイル間参照を識別できる。

## 依存関係

- 型検査モジュールはファイルシステムからファイルを読み取り、グローバル環境を構築できる必要がある。
- 標準ライブラリの定義にアクセス可能である必要がある（コンパイラに組み込まれているか、パスで指定される）。
