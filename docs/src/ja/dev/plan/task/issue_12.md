## 概要

`yaoxiang check` は、YaoXiang のソースコードに対して静的検査（型チェック、所有権チェック、lint）を実行するコマンドラインツールであり、コードの生成は行いません。迅速なフィードバックを提供し、CI パイプラインへの統合や開発プロセス中の検証ステップとして適しています。

## 目標

- `yaoxiang check` コマンドを実装し、1 つ以上のファイルまたはディレクトリを入力として受け入れる。
- コンパイラの型チェックモジュールを再利用し、エラーと警告を標準エラーに出力する。
- `--json` オプションをサポートし、他のツールへの統合に便利な JSON 形式で診断情報を出力する。
- `--watch` モードをサポート（任意）。ファイルの変更を監視し、再検査を自動実行する。
- エラーメッセージには、ファイル名、行番号、列番号、エラー深刻度、説明を含める。

## 具体的な手順

1. **コマンドラインインタフェースの設計**（3 日）
   - `clap` などのライブラリを使用して引数を解析：`yaoxiang check [OPTIONS] <PATH>...`。
   - `--json`、`--watch`、`--color` などのオプションをサポート。

2. **型チェッカーの統合**（1 週間）
   - 型チェックモジュールを独立して呼び出し可能な関数としてラップし、ソースコード文字列を受け取り、診断リストを返す。
   - 複数ファイルチェックをサポート：すべてのファイルを解析し、グローバルシンボルテーブルを構築する必要がある。
   - 標準ライブラリの paths と組み込み定義を処理する。

3. **出力フォーマットの実装**（3 日）
   - Rustc のようなカラーエラー出力を実装し、エラーの位置、深刻度、説明を含める。
   - JSON フォーマット出力を実装し、共通仕様（例：`rustc --error-format=json` と同等）に準拠する。

4. **監視モードのサポート（任意）**（1 週間）
   - ファイルシステム監視ライブラリ（例：`notify`）を使用してファイルの変更を監視し、自動的に再検査を実行する。
   - 繰り返しのフルチェックを避け、可能な限りインクリメンタルチェックを行う。

5. **テストと統合**（1 週間）
   - テストケースの作成：错误を含むファイルに対してチェックを実行し、出力が期待通りか検証する。
   - CI パイプラインへの統合，各コミットでチェックがパスすることを確認。

## 受け入れ基準

- エラーのない単一ファイルに対して `yaoxiang check` を実行すると、出力なし（または成功メッセージ）、終了コード 0 となる。
- 型エラーが含まれるファイルに対してチェックを実行すると、明確なエラー情報（ファイル名、行番号、列番号、説明）を出力し、終了コードは 0 以外となる。
- `--json` オプションは有効な JSON 配列を出力し、各要素にエラーの位置とメッセージを含める。
- チェック速度はコンパイラの型チェック速度に近く、頻繁な使用に適している。
- プロジェクト内の複数ファイルを適切に処理し、ファイル間参照を認識できる。

## 依存関係

- 型チェックモジュールは、ファイルシステムからファイルを読み取り、グローバル環境を構築できる必要がある。
- 標準ライブラリの定義にはアクセス可能である必要がある（コンパイラに組み込まれているか、パスで指定可能）。
