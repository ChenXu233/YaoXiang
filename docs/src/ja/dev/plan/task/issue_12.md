## 概要

`yaoxiang check` は、YaoXiang ソースコードに対して静的検査（型チェック、所有権チェック、lint）を実行するコマンドラインツールです。コードの生成は一切行いません。フィードバックの即時性を重視し、CI フローへの統合や開発过程中的な検証ステップとして最適です。

## 目標

- `yaoxiang check` コマンドを実装し、1 つ以上のファイルまたはディレクトリを入力として受け入れること。
- コンパイラの型チェックモジュールを再利用し、エラーと警告情報を標準エラー出力に出力すること。
- `--json` オプションをサポートし、他のツールとの連携に適した JSON 形式での診断情報出力を行うこと。
- `--watch` モードをサポートすること（任意）。
- エラー情報にファイル名、行番号、列番号、エラー等級、説明を含めること。

## 具体的な手順

### 1. コマンドラインインターフェース設計（3 日）

- `clap` または類似のライブラリを使用して引数を解析：`yaoxiang check [OPTIONS] <PATH>...`。
- `--json`、`--watch`、`--color` などのオプションをサポート。

### 2. 型チェック器の統合（1 週間）

- 型チェックモジュールを独立して呼び出し可能な関数としてラップし、ソースコード文字列を受け取り、診断リストを返す。
- 複数ファイルチェックをサポート：すべてのファイルを解析しグローバルシンボルテーブルを構築する必要がある。
- 標準ライブラリの paths および組み込み定義を処理。

### 3. 出力フォーマットの実装（3 日）

- Rustc と同様のカラーエラー出力を実装し、エラーの位置、エラー等級、説明を含む。
- JSON フォーマット出力を実装し、一般的な仕様に準拠（例：`rustc --error-format=json` と同等）。

### 4. 監視モードのサポート（任意）（1 週間）

- ファイルシステム監視ライブラリ（例：`notify`）を使用してファイル変更を監視し、自動的に再チェックを実行。
- 可能なかぎり増分チェックを実施し、全量チェックの繰り返しを回避。

### 5. テストと統合（1 週間）

- テストケースの作成：错误を含むファイルに対してチェックを実行し、出力が期待通りであるか検証。
- CI フローへの統合：各コミットでチェックが通過することを確認。

## 受け入れ条件

- エラーのない単一ファイルに対して `yaoxiang check` を実行すると、出力がない（または成功メッセージ出力）、終了コードは 0。
- 型エラーが含まれるファイルに対してチェックを実行すると、明確なエラー情報（ファイル名、行番号、列番号、説明）が出力され、終了コードは 0 以外。
- `--json` オプションは有効な JSON 配列を出力し、各要素にエラーの位置とメッセージを含む。
- チェック速度はコンパイラの型チェック速度に近く、頻繁な使用に適している。
- プロジェクト内の複数ファイルを正しく処理し、ファイル間の参照を識別できる。

## 依存関係

- 型チェックモジュールはファイルシステムからファイルを読み取り、グローバル環境を構築できる必要がある。
- 標準ライブラリの定義にアクセス可能であること（コンパイラに組み込まれているか、パスで指定される）。
