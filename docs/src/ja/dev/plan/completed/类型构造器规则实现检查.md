# タイプコンストラクタ規則の実装チェック

> チェック対象：コンパイラが「`: Type` が 있으면タイプコンストラクタ、それ以外は関数/変数」という規則を正しく実装しているかの検証
>
> 作成日：2026-03-21

---

## 機能説明

### コア規則

| 書き方 | コンパイラ動作 | 説明 |
|------|------------|------|
| `x: Type = {...}` | **タイプコンストラクタ**（`: Type` で返り値的类型が Type） | |
| `f = {...}` | **関数/変数**（`{...}` はコードブロックとして解析） | |

**重要原則**：`Type` を省略した場合は絶対に型として解析されず、`{...}` は常にコードブロックとして解析される。

### 曖昧さが生じるシナリオ

| 形式 | `: Type` あり | `: Type` なし |
|------|----------------|----------------|
| `{ x: Float }` | ✅ 型リテラル | ✅ コードブロック（式、無意味） |
| `{ a = 1 + 1 }` | ✅ 型、フィールドデフォルト値は 2 | ✅ コードブロック（代入式、無意味） |
| `{}` | ✅ 空レコード型 | ✅ 空コードブロック `() -> Void` |

**重要なポイント**：

- `: Type` あり → `{ ... }` は型リテラルとして解析
- `: Type` なし → `{ ... }` はコードブロックとして解析（型の位置にあってもコードブロック！）

## チェックポイント

### チェック 1：タイプコンストラクタは最終的に `Type` を返さなければならない

**重要規則**：「`Type` があれば型」ではなく、**最終的に返り値の类型が `Type`** である場合のみタイプコンストラクタとなる。

> **コンパイル時の確定性**：Type に関連する内容はすべて**コンパイル時に確定**する。ジェネリック関数也不例外で、T は実行時に渡されるのではなく、**コンパイル時に単態化**される。

| シグネチャ | 返り値类型 | 意味 | 判定 |
|------|----------|------|------|
| `(T: Type) -> Type` | `Type` | ✅ **タイプコンストラクタ**（コンパイル時） | |
| `(T: Type) -> (() -> Void)` | `() -> Void` | ❌ **ジェネリック関数**（コンパイル時に単態化） | |
| `(T: Type) -> (T -> Void)` | `T -> Void` | ❌ **ジェネリック関数**（コンパイル時に単態化） | |

**テストコード**：

```yaoxiang
// ✅ タイプコンストラクタ：最終的に Type を返す
Point: Type = { x: Float, y: Float }
List: (T: Type) -> Type = { data: Array[T] }
Option: (T: Type) -> Type = { some: (T) -> Self, none: () -> Self }

// ❌ ジェネリック関数：最終的に関数型を返す、Type ではない
maybe: (T: Type) -> (() -> Void) = (T) => { println("test") }
factory: (T: Type) -> (T -> Void) = (T) => { (x) => println(x) }
```

**チェック項目**：

- [ ] `Point: Type = { ... }` はコンパイル可能か？
- [ ] `List: (T: Type) -> Type = { ... }` はコンパイル可能か？
- [ ] `Option: (T: Type) -> Type = { ... }` はコンパイル可能か？

**期待結果**：すべてコンパイル成功

---

### チェック 2：`: Type` を省略すると関数として推断される

**重要規則**：入力パラメータがある場合、**型注釈**または **lambda ヘッダ**の**少なくとも一方**に型を标注する必要がある。両方を省略することはできない。

```yaoxiang
// ✅ 正しい：型注釈の位置に标注
add: (a: Int, b: Int) -> Int = (a, b) => a + b

// ✅ 正しい：lambda ヘッダの位置に标注
add = (a: Int, b: Int) => a + b

// ✅ 正しい：両方に标注
add: (a: Int, b: Int) -> Int = (a: Int, b: Int) => a + b

// ✅ 正しい：lambda ヘッダを省略（シグネチャが完整の場合）
add: (a: Int, b: Int) -> Int = a + b

// ❌ 錯誤：両方に型を标注していない
add = (a, b) => a + b

// ✅ 正しい：空ブロックはパラメータ型が不要
main = { println("Hello") }
```

**チェック項目**：

- [ ] `main = { println("Hello") }` → 型は `() -> Void` か？
- [ ] `add = (a: Int, b: Int) => a + b` → 型は `(a: Int, b: Int) -> Int` か？
- [ ] `add: (a: Int, b: Int) -> Int = a + b` → lambda ヘッダを省略、型は `(a: Int, b: Int) -> Int` か？
- [ ] `add = (a, b) => a + b` → ❌ コンパイル失敗？両方に型を标注していない

### チェック 3：空ブロック `{}` の型

**テストコード**：

```yaoxiang

void_func = { println("test") }

// ❌ 錯誤：Int を期望しているが Void が得られる

```

**チェック項目**：

- [ ] `empty_func = {}` はコンパイル可能か？型は `() -> Void` か？
- [ ] `void_func = { println("test") }` はコンパイル可能か？型は `() -> Void` か？
- [ ] `bad: Int = {}` はエラーとなるか？エラーメッセージは明確か？

**期待結果**：

- 前2つはコンパイル成功
- 3番目は型が合わないエラー（Void vs Int）

### チェック 4：错误ケース：`: Type` を省略した型定義

**テストコード**：


**チェック項目**：

- [ ] `Point = { x: Float, y: Float }` は何としてコンパイルされるか？関数か型か？

**期待結果**：

- `{ x: Float, y: Float }` がコードブロックとして解析された場合、`Point` は `() -> ???` 型の関数となる（ブロック内有効な式ではないためエラーになる可能性あり）
- **核心**：`: Type` を省略すると絶対に型定義にはならない

## まとめ

**コア規則**：

| 書き方 | 返り値の型 | 意味 |
|------|----------|------|
| `x: Type = {...}` | `Type` | ✅ タイプコンストラクタ |
| `f = {...}` | `Void` | ✅ 関数/変数（`{...}` はコードブロック） |
| `f: (T: Type) -> Type = ...` | `Type` | ✅ タイプコンストラクタ |
| `f: (T: Type) -> (() -> Void) = ...` | `() -> Void` | ❌ ジェネリック関数（型ではない） |

**重要な区別**：

- 「`Type` があればタイプコンストラクタ」ではない
- 「**最終的に `Type` を返す**」場合のみタイプコンストラクタ
- `(T: Type) -> (() -> Void)` は関数型を返すため、ジェネリック関数である

**コンパイル時の確定性**：

> Type に関連する内容はすべて**コンパイル時に確定**する。ジェネリック関数也不例外で、T は**実行時に渡されるのではなく**、**コンパイル時に単態化**される。

**実装のポイント**：

1. **パーサー**：`: Type` あり → タイプコンストラクタとしてマーク；`: Type` なし → 関数/変数としてマーク
2. **タイプチェッカー**：
   - `: Type` あり → `{ ... }` は型リテラルとして解析
   - `: Type` なし → `{ ... }` はコードブロックとして解析
3. **HM 推論**：省略形の関数型を正しく推論
4. **エラー処理**：型を期望しているが実際には関数の場合のエラーメッセージを明確に
