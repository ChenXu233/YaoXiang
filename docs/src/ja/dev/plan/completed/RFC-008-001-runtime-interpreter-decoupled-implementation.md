# インタープリタ優先：デカップリング型 Runtime（DAG+スケジューリング）実装計画

> **タスク**：独立 Runtime（DAG/スケジューリング/エラー伝播/リソース直列化）を先に実装し、インタープリタを接続して実行可能闭环を形成する。后续编译器が `@block/@eager/@auto`、`spawn`、`Result/?` などのメタ情報/制約を Runtime に「渡す」。  
> **ベース RFC**：RFC-008（Runtime 并发モデルとスケジューラのデカップリング）、RFC-001（並作モデルと錯誤処理システム）、RFC-009（所有権モデル：タスク境界はリーク境界）  
> **日付**：2026-03-11（起草）、2026-03-12（更新）  
> **ステータス**：進行中（計画）

---

## 摘要

- 「インタープリタ/コンパイラとデカップリングされた」Runtime を優先実装：タスクライフサイクル、依存グラフ（DAG）管理、スケジューリング戦略、錯誤伝播とキャンセル、リソース競合の直列化のみを责務とする。バイト码/AST/型システムの詳細は理解しない。
- インタープリタ/コンパイラは「実行可能ユニット（タスク）+ メタ情報（依存/リソース/戦略/Span）」を Runtime にsubmitする責務のみを持つ。
- 段階 A はインタープリタ闭环为主（既存のセマンティクスはデフォルトで変更しない）；段階 B でコンパイラが言語セマンティクスを Runtime に贯通させる。

---

## 対外インターフェース/振る舞い変更（当計画が 生成する変更）

- 新規追加（または安定化）一套の **Runtime 公共インターフェース**：タスクsubmit/依存/待機/キャンセル/統計、リソース直列化キー、錯誤伝播トレース/オプショナルな錯誤グラフ开关。
- インタープリタに **ランタイム設定エントリ新增**：Embedded/Standard/Full の選択、workers 数、work-stealing の有効化与否（CLI への延期可能であり、プログラム的なエントリのみでも可）。

---

## コード実装マッピング（現在）

- Runtime コア（DAG/依存/キャンセル/リソース直列化）：`src/backends/runtime/engine.rs`
- Runtime 三層ファサード（Embedded/Standard/Full）：`src/backends/runtime/facade.rs`
- インタープリタランタイム設定（RuntimeMode + EvalStrategy）：`src/backends/interpreter/runtime.rs`
- インタープリタ：タスクスケジューリング/駆動/force（AsyncValue + drive_dag_until）：`src/backends/interpreter/executor/executor.rs`
- インタープリタ：`Call*` 命令における戦略注入（@block/@auto/@eager）と保守的なリソースキー（ffi）：`src/backends/interpreter/executor/execute.rs`

> 說明：インタープリタタスクは `&mut Interpreter` を必要とするため、現在のインタープリタでは **Host-driven**（单一スレッド駆動）実行可能タスク方式を採用している；Full runtime の並列実行は主として将来のコンパイル後段とランタイム検証用途に使用する（RFC-008 の階層化と RFC-009 のタスク境界制約に準拠）。

## 段階 A：Runtime + インタープリタ（先に闭环、後增强）

### A1. Runtime 境界の定義と固定（インタープリタ/コンパイラとのデカップリング）

**目標**
- Runtime の責務范围：タスクライフサイクル、DAG、スケジューリング戦略、錯誤伝播とキャンセル、リソース直列化のみ。
- Runtime はインタープリタ/コンパイラの内部構造に依存しない；インタープリタ/コンパイラはタスクとメタ情報のみをsubmitする。

**受入基準**
- [x] Runtime モジュールがインタープリタとコンパイラの内部モジュールを参照しない（依存境界チェックを基準とする）。
- [x] Runtime インターフェースが「インタープリタタスク」と「将来のコンパイラ/AOT タスク」に共用可能である。

**テスト項目**
- [x] 单元テスト：最小「MOCK タスクエグゼキュータ」を使用した Runtime 独立動作検証（インタープリタ导入なし）。

---

### A2. インタープリタ Embedded パスの接続（セマンティクス変更なし）

**目標**
- 並发/遅延評価を導入せず、まずインタープリタが Runtime を 통해最も基础的な「 즉시 실행」パスを実行底座として確立する。

**受入基準**
- [x] 既存のインタープリタ端到端テストケースの結果とエラーメッセージが変化しない。
- [x] 既存のテストセットが通过する。

**テスト項目**
- [x] 回帰テスト：インタープリタ/統合テスト全量通过。
- [x] スモークテスト：最小プログラム（並发セマンティクスなし）の出力の一致。

---

### A3. Runtime：Standard Runtime（单一スレッド DAG スケジューリング + 依存待機）

**目標**
- 单一スレッドスケジューリングの実装：DAG の構築/維持、依存関係による驱动、孤島 DAG がメインチェーンをブロックしない；循環依存検出と錯誤 возвращение の поддержка。

**受入基準**
- [x] 線形依存/菱形依存/孤島 DAG の実行順序が RFC-001 の記述と一致する。
- [x] 循環依存でデッドロックせず、明確な錯誤を返す（依存チェーンのサマリーを含み、可能なら Span 付き）。

**テスト項目**
- [x] 单元テスト：線形依存、菱形依存、孤島 DAG、循環依存（2ノード循環、自己循環）。
- [x] 单元テスト：キャンセルとクリーンアップ（キャンセル後の進行停止、リソース解放の検証）。

---

### A4. Runtime：Full Runtime（マルチスレッドスケジューリング + オプショナル work-stealing）

**目標**
- workers>1 の並列スケジューリングを追加；work-stealing はオプショナルな增强として实现し、无効化しても正常に動作（RFC-008 の核心要件）。

**受入基準**
- [x] workers=1 は Standard の振る舞いと一致する（結果と錯誤が一致）。
- [x] workers>1 で並列実行が观测可能（性能では验收せず、观测可能な並列度で验收）。
- [x] work-stealing のオン/オフが正确性に影響しない。

**テスト項目**
- [x] 单元/統合テスト：同一タスクグラフが workers=1 と workers>1 で結果が一致する。
- [x] 单元テスト：work-stealing オン/オフマトリクスでデッドロックなし、タスクロストなし。

---

### A5. インタープリタ：遅延値と「需要に応じた force」（@auto/@eager/@block の支撑）

**目標**
- 「先に遅延評価可能なプレースホルダ値を生成し、値使用時に評価をトリガーする」のをサポート。
- ランタイムに三種類の戦略を設定可能：
  - デフォルト：現在の同期振る舞いを維持（グローバル @block に相当）。
  - オプショナル：グローバル @auto（最大遅延と並发）/ グローバル @eager（より急切な評価）。

**受入基準**
- [x] @auto：`Async` プレースホルダ値を返す；値が消費される際に Runtime が驱动して目標達成と値取得を行う。
- [x] @block：デフォルトの振る舞いが現状と一致（DAG を通らない；`Async` プレースホルダ値を生成しない）。
- [x] @eager：呼び出し時点で即座に驱动して値を取得（`Async` プレースホルダ値を残さない）。

**テスト項目**
- [x] 单元テスト：@auto 下で lazy call が消費されるときに実行をトリガーして値を取得する。
- [x] 单元テスト：@eager 下では `Async` プレースホルダ値を残さない。
- [x] 回帰テスト：新戦略が無効な場合に既存のテストが全量通过する。

---

### A6. Runtime + インタープリタ：タスク失敗の伝播、マージ、キャンセル連動

**目標**
- タスク失敗が依存エッジに沿って伝播：依存者の失敗またはキャンセル；複数タスク同時失敗時に主錯誤と付随錯誤サマリーを出力。

**受入基準**
- [x] 依存失敗時、上流が永久に待機しない；錯誤がトップレベルに到達可能（キャンセル理由に依存錯誤サマリー 포함）。
- [x] 上流が特定のターゲット値のみを待機している場合、Runtime は依存クロージャを優先的に進行させ、孤島タスクがターゲットの完了を遅延させない（進行停止；后续オプションの明示的キャンセル戦略）。

**テスト項目**
- [x] 单元テスト：リーフ失敗が上游の失敗/キャンセルを引き起こす；エラーメッセージに根因のサマリーを含む。
- [x] 单元テスト：複数依存同時失敗時に主錯誤と他のサマリーをマージして出力可能（キャンセル理由の集約）。
- [x] 单元テスト：キャンセル後はキャンセルされたタスクの進行を継続しない（依存キャンセル伝播を含む）。
- [x] 单元テスト：`drive_until(Some(target))` は無関係な孤島タスクを実行しない（ターゲットチェーン優先）。

---

### A7. リソース競合の最小闭环（先に保守的、後精细）

**目標**
- インタープリタはまず「保守的なリソースキー」で副作用操作にラベル付け（例如統一为同一类リソース）し、Runtime が直列化を強制して正確性を確保；后续コンパイラがより精细なリソースキーを提供して並列度を向上（RFC-001 のリソースタイプ体系）。

**受入基準**
- [x] workers>1 下では同一リソースキーの操作が厳密に直列（順序が稳定して观测可能）。
- [x] 不同リソースキーの操作は並列を許可（少なくとも观测可能な並列進行が可能）。

**テスト項目**
- [x] 单元テスト：Full runtime 下では同一リソースキーのタスクが并发起動しない。
- [x] 单元テスト：workers>1 下では無リソース/異なるリソースキーのタスクが并发起動可能。

---

### A8. 無限ループと公平性（スライシング/譲歩）

**目標**
- RFC-001 の実装：単一ループでスケジューリングオーバーヘッドなし、複数ループで自動スライシング；_starvation_ の防止；単一タスク時に观测可能なオーバーヘッドを導入しない。

**受入基準**
- [x] 複数タスクシナリオでは某个長時間ループが他のタスクの永遠の停止を引き起こすことはない（協調的スライシング：タスクは `yield` で譲歩可能）。
- [x] 単一タスクシナリオではスライシングなしで直接完了可能（競合がない場合にエグゼキュータが `yield` しないため、观测可能なオーバーヘッドなし）。

**テスト項目**
- [x] 单元テスト：2つの長時間ループタスクが有限ステップ内で進行可能（カウンター/ラウンドロビンスケジュール驗收）。
- [x] 单元テスト：単一タスクが単一の poll で完了可能（スライシングスケジューリングオーバーヘッドなし）。

---

## 段階 B：コンパイラ接続（后续で言語セマンティクスを Runtime に「渡す」）

### B1. 構文と制約：`@block/@eager/@auto` と `spawn`（@block 内でのみ許可）

**目標**
- コンパイラが関数/ブロックレベルの戦略メタ情報を提供；`spawn` の使用を `@block` スコープ内に制限（RFC-001/008）。

**受入基準**
- [x] 不正な `spawn` 位置がコンパイル時にエラーとなり、位置特定が明確である。
- [x] アノテーション戦略がランタイム設定/メタ情報に安定的に伝達される。

**テスト項目**
- [x] 解析/型チェックテスト：合法/不正 `spawn` シナリオマトリクス。
- [x] 端到端テスト：アノテーションがランタイム戦略選択に影響（观测可能な差異 + 結果の一致）。

---

### B2. `Result[T,E]` と `?` の言語層闭环（Runtime 錯誤伝播との整合）

**目標**
- `Result` は真に `T/E` を携带し、`?` はコンパイル時に規則の正しさを保証；ランタイムは RFC-001 に従って透過的に伝播；並发依存の錯誤伝播と一貫性を保つ。

**受入基準**
- [x] 非 `Result` での `?` 使用または錯誤タイプ不一致時にコンパイルエラー。
- [x] 並発シナリオで `?` の早期返回とリソース直列化ルールが衝突しない（説明可能でテスト可能）。

**テスト項目**
- [x] 型チェックテスト：`Result/?` 正誤用例。
- [x] 端到端テスト：並発タスク失敗が `?` 伝播でトップレベルに到達。
- [x] 单元テスト：リソース直列化は control edge として実装（排序のみ、錯誤伝播なし）；コントロール deps のキャンセル待受でリソース順序を維持（`src/backends/runtime/engine.rs` の対応テスト参照）。

---

### B3. 錯誤グラフ（オプショナル）とメモリ上限（関数スコープ内）

**目標**
- 錯誤伝播パスの可視化を実装；錯誤グラフは単一関数/ブロックスコープ内に限定し、上限と开关付き。

**受入基準**
- [ ] 有効時に安定かつ比較可能な錯誤グラフサマリーを出力；無効時に追加オーバーヘッドとノイズなし。
- [ ] グラフ規模がプログラム規模に応じて無界に成長しない。

**テスト項目**
- [ ] スナップショットテスト：固定ソースコードから固定錯誤グラフサマリーを生成（テキスト/JSON）。
- [ ] 压力テスト：大量ノード時も上限内で出力し安定終了可能。

---

## デフォルトの前提/選択（調整可能）

- 段階 A はデフォルトで既存のユーザーセマンティクスを変更しない：インタープリタはデフォルトでグローバル `@block`（Embedded パス）に相当し、Standard/Full と `@auto/@eager` は明示的な設定でのみ有効化。
- リソースシステムは段階 A ではまず正確性を保証するために保守的に対応し、段階 B でコンパイラが精细なリソースキーを提供して並列度を向上させる。
