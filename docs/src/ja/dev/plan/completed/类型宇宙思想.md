## 序論：型ユニバース思想——心のモデル

YaoXiang 言語の設計哲学は **「型で全てを解決する」** です。開発者が型システム内の各種エンティティの役割とその相互関係を理解するために、私たちは **「型ユニバース思想」**——階層的な心のモデルを提唱します。このモデルは言語内の概念をセマンティックな役割に基づいて異なる階層に划分し、関数、型コンストラクタ、Trait、値依存型などの概念を統一し、開発者が言語全体を一貫した視点で捉えることを可能にします。

**重要な説明**：このモデルは **外部に提示する心のモデル** であり、構文の統一、Semantic Highlighting のガイド、学習曲線の簡素化することを目的としています。**実際のコンパイラの内部実装** は、従来の累積宇宙階層（`Type₀, Type₁, Type₂, …`）に基づいており、論理的厳密性を保証し、逆説を回避します。両者はconflictせず、外部モデルはコンパイラの実装を変更せず、理解と記述のためだけに使用されます。

## 外部心のモデル：型ユニバース階層

| 階層   | 役割               | 例                                                                 | 日本語表記 |
|--------|--------------------|----------------------------------------------------------------------|------------|
| Type-1 | 値                 | `42`, `"hello"`, `add` 関数そのもの（関数も値である）                         | 値         |
| Type0  | メタ型キーワード       | `Type`（「これは型である」と标注するため）                                      | メタ型キーワード |
| Type1  | 具体型           | `Int`, `String`, `Point`（変数の宣言に直接使用可能）                         | 具体型     |
| Type2  | 関数/型コンストラクタ/値依存型 | `add: (Int,Int)->Int`, `List: (T: Type) -> Type`, `Vec: (n: Int) -> Type` | 関数/型コンストラクタ/値依存型 |
| Type3  | Trait/制約          | `Clone: Type = { clone: (Self) -> Self }`                            | Trait/制約 |
| Type4+ | メタプログラミング構成要素（予約） | マクロ、コードジェネレータなど                                                     | メタプログラミング（予約） |

- **Type-1** 階層にはすべての実行時値が含まれ、関数自体也不例外です。
- **Type0** 階層は特殊な `Type` キーワードであり、型の型を示すために使用されます。
- **Type1** 階層は具体型、すなわち通常の意味での「データ型」です。
- **Type2** 階層はすべての呼び出し可能なエンティティ：通常関数（戻り値が値）、型コンストラクタ（戻り値が型）、値依存型（戻り値が型）です。
- **Type3** 階層は Trait/制約であり、一連の基本仕様を記述します。
- **Type4+** は将来のメタプログラミング機能のために予約されています。

この階層分けは論理的な階数ではなく、完全に**セマンティックな役割**に基づいています。開発者が各シンボルの用途を直感的に理解するのに役立ちます。

## 内部実装：従来の累積宇宙階層

コンパイラの内部では、標準的な累積宇宙階層（Coq/Agda 如く）を使用しています：

- `Type₀`：具体型（例：`Int`）とその型の値を含みます。
- `Type₁`： `Type₀` の型（つまり型の型）を含みます。例えば `Type₀` itself、型コンストラクタ `List : Type₀ → Type₀` の型など。
- `Type₂`： `Type₁` の型を含み、以下同様に続きます。
- 関係：`Typeₙ : Typeₙ₊₁`、無限に上昇する階層を形成し、`Type: Type` 逆説を回避します。

各型式は内部で宇宙レベルを付与され、宇宙レベルは型チェック時に自動的に管理され、開発者からは完全に透過的です。外部心のモデルの階層と内部宇宙レベルの間には自然な対応関係があります：

- 具体型（Type1）は通常、内部の `Type₀` に対応します。
- 型コンストラクタ（Type2）は通常、内部の `Type₁` またはそれ以上に対応します（引数宇宙に依存）。
- Trait（Type3）は通常、内部の `Type₁` またはそれ以上に対応します。

しかし、この対応はコンパイラの内部詳細であり、開発者が気にする必要はありません。

## 統一構文

外部心のモデルがどのように階層化されていても、YaoXiang の構文は一貫して統一されています：

```
identifier : type = expression
```

ここで `type` は型式、`expression` は値式です。`fn`、`struct`、`trait` などの専用キーワードはありません。

### 型コンストラクタ vs 関数/変数：コアルール

**これは構文の歧路消除ルールを決定する重要な設計選択です：**

| 書き方 | 意味 | ルール |
|--------|------|--------|
| **`x: Type = ...`** | 型コンストラクタ | `: Type` 显式宣言 → 型として強制 |
| **`f = ...`** | 関数または変数 | `: Type` なし → HM が関数/変数として推論 |

**なぜこの設計なのか？**

`{ ... }` 構文自体は歧路があります：

- `{ x: Float, y: Float }` は**型リテラル**（記録型）になり得る
- `{ a = 1 + 1 }` は**コードブロック**（文の実行、Void を返す）になり得る

**歧路消除ルール**：

- **あり** `: Type` → 型コンストラクタとして強制解決、`{ ... }` は型リテラル
- **なし** `: Type` → HM が `{ ... }` をコードブロックとして解決し、関数型として推論

```yaoxiang
# ✅ 型コンストラクタ：: Type あり
Point: Type = { x: Float, y: Float }

# ✅ 関数：: Type なし、HM が () -> Void として推論
main = { println("Hello") }

# ❌ エラー：: Type なし、コンパイラが { ... } を型として解決できない
Point = { x: Float, y: Float }  // HM が関数として推論！型ではない！
```

### 構文例

| 概念 | コード写法 | 説明 |
|------|----------|------|
| 変数 | `x: Int = 42` | 型标注あり |
| 関数 | `add: (a: Int, b: Int) -> Int = a + b` | 完全な型 |
| 関数（HM推論） | `add = (a, b) => a + b` | `: Type` なし、HM が関数として推論 |
| 記録型 | `Point: Type = { x: Float, y: Float }` | **必ず `: Type` が必要** |
| Trait | `Clone: Type = { clone: (Self) -> Self }` | **必ず `: Type` が必要** |
| 型コンストラクタ | `List: (T: Type) -> Type = { data: Array[T] }` | **必ず `: Type` が必要** |
| ジェネリクス関数 | `map: (T: Type, R: Type) -> ((list: List[T], f: (T) -> R) -> List[R]) = ...` | **必ず `: Type` が必要** |
| 値依存型 | `Vec: (n: Int) -> Type = (n) => { data: Array[n] }` | **必ず `: Type` が必要** |
| 制約付き関数 | `clone: (T: Clone) -> ((value: T) -> T) = (value) => value.clone()` | `: Type` なし、HM が関数として推論 |

- 関数型には引数名を含める必要があります（例：`(a: Int, b: Int) -> Int`）。
- 型コンストラクタも関数であり、その型は `(T: Type) -> Type` であり、完全に一致しています。
- Trait も型であり、定義形式は記録型と一貫しています。
- **AST は Binding ノードに統一**され、`fn`/`struct`/`trait` キーワードの区別がなくなりました。
- **セマンティック判定ルール**：
  - `: Type` あり → 型コンストラクタ
  - `: Type` なし → HM が関数/変数として推論

## ジェネリクスシステム

ジェネリクスシステムは「型も値である」という原則に基づいて構築されており、すべてのジェネリクス概念は型レベル関数または制約付きパラメータ化関数として統一されています。

### 型コンストラクタ

```
List: (T: Type) -> Type = { data: Array[T] }
```

- 型パラメータ `T` を受け取り、記録型を返します。
- 適用時は角括弧 `List[Int]` を使用します（糖衣構文、`List(Int)` と等価）。

### ジェネリクス関数

```
map: (T: Type, R: Type) -> (
    (list: List[T], f: (x: T) -> R) -> List[R]
    ) = ...
```

- 型はカリー化された型レベル関数であり、まず型パラメータを受け取り、値レベル関数の型を返します。
- 呼び出し時に型推論が可能：`map(ints, double)` は `T`、`R` を自動的に埋めます。

### 型制約：ジェネリクス制約 vs Trait オブジェクト

YaoXiang は型制約を実装するための2つの方式をサポートしています：

#### 方式一：ジェネリクス制約（コンパイル時多相）

```
Clone: Type = { clone: (Self) -> Self }
clone: (T: Clone) -> ((value: T) -> T) = (value) => value.clone()
```

- 制約 `T: Clone` は `T` が `Clone` Trait を実装する必要があることを意味します。
- コンパイル時に単相化され、各具体型に対して独立したコードが生成されます。
- 実行時オーバーヘッド零（静的ディスパッチ）。
- 複合制約は交差型を使用可能：`(T: Clone & Debug) -> ...`

#### 方式二：Trait オブジェクト（実行時多相）

```
clone_obj: (value: Clone) -> Clone = (value) => value.clone()
```

- パラメータ型に直接 Trait 名 `Clone` を記述します。
- 実行時に動的ディスパッチし、vtable 経由でメソッドを呼び出します。
- 異種混合コレクション、実行時多相シナリオに適しています。

| 特性 | ジェネリクス制約 `(T: Clone)` | Trait オブジェクト `value: Clone` |
|------|----------------------|------------------------|
| パラメータ型 | 具体型 `T` | Trait 型 `Clone` |
| 戻り値型 | 具体型 `T` | Trait 型 `Clone` |
| コード生成 | コンパイル時単相化 | 実行時動的ディスパッチ |
| 性能 | 零オーバーヘッド | 間接呼び出しオーバーヘッド |
| 適用シーン | 高性能、具体型保持 | 異種混合コレクション、実行時多相 |

両方式とも合法であり、開発者は必要に応じて選択できます。

### 値依存型とコンパイル時定数

YaoXiang の統一設計の下で、**型は関数呼び出しの結果を含め、任意の値に依存できます**。これにより、関数呼び出しがコンパイル時に評価可能であり、なおかつ評価プロセスが必ず終了することが自然に要求されます。さもなければ、型システムは無限ループに陥ります。

#### コンパイル時評価ルール

- **関数には一つの定義形式만存在**し、`constexpr` などの标记は不要です：
  ```
  factorial: (n: Int) -> Int = {
      if n <= 1 { return 1 }
      return n * factorial(n - 1)
  }
  ```
- **型位置でのコンパイル時評価强制**：型式中出现する関数呼び出し（配列サイズ、値依存型パラメータなど）はすべて、コンパイル時に計算可能かつ必ず終了することが要求されます：
  ```
  matrix: Array[Int, factorial(5)] = ...  # factorial(5) はコンパイル時に評価
  vec: Vec[factorial(3)] = ...             # 依存型パラメータもコンパイル時に評価
  ```

#### 終了チェックメカニズム（ホア論理ベース）

コンパイル時評価の安全性を确保するため、YaoXiang は** 감소 측정（decreases）規則**を導入し、RFC-022 とシームレスに統合します：

**再帰関数の終了規則**：
```
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n              # 每次再帰で n が厳密に減少
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}
```

**ループの終了規則**：
```
sum: (arr: Array[Int, n]) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

- `decreases` 式は整礎である必要があります（例：自然数）。
- コンパイラは終了条件を生成し、SMT ソルバを呼び出して証明します。
- タプル測定（辞書式減少）とユーザー定義整礎型のサポートがあります。

#### コンパイル時評価と終了チェックの協調

1. 型チェック段階で型位置上に関数呼び出しが出现すると、コンパイラはその関数が `decreases` 規則で标记されているか確認します。
2. 存在する場合は、組み込み評価器を呼び出して関数値を計算し、同時に減少条件が評価パス上で成立することを検証します。
3. `decreases` 标记がないが関数が明显に終了可能（例：非再帰）の場合は直接評価します。さもなければ終了規則の追加を促すエラーメッセージを出力します。

#### 優位性

- **安全性**：依存型のコンパイル時評価が必ず終了することを确保し、型システムが無限ループに陥ることを回避します。
- **統一性**：終了チェックと部分的正しさは同じ規則メカニズムを共有します。
- **漸進的強化**：実行時チェックから完全静的証明へと段階的に移行できます。

## Type4：メタプログラミング階層

「型ユニバース」の完全な絵において、Type4 階層は **コンパイル時に YaoXiang コード生成制約を実行する** メタプログラミング階層として定義されています。

### コア思想：コンパイル時実行、結果インライン化

Type4 は YaoXiang 自身の interpreter を利用し、**コンパイル時に任意の YaoXiang コードを実行します**：

- 既存の interpreter を再利用し、別途の「型レベル言語」が不要
- コンパイル時評価結果を生成されたバイナリに直接インライン化
- 実行時はデフォルトで零オーバーヘッド——リフレクション構造はなく、通常のコードのみ

```
┌─────────────────────────────────────────┐
│  コンパイル時 derive                      │
│  ├── Interpreter が derive 関数を実行    │
│  ├── 型のメタデータを読み取り            │
│  ├── 生成された Trait 実装を TraitTable に書き込み│
│  └── コード生成 → バイナリ               │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  実行時                                  │
│  └── 通常バイナリ、リフレクションなし、オーバーヘッドなし│
└─────────────────────────────────────────┘
```

### 実装メカニズム

1. **型リフレクション API**：型にはメタデータが携带され（フィールドリスト、メソッドリストなど）、標準ライブラリ関数を介して公開されます
   ```yaoxiang
   // 標準ライブラリが提供
   type Type = { fields: () -> Array[Field], ... }
   ```

2. **コンパイル時評価**：interpreter がコンパイル時に関数呼び出しを実行し、結果をインライン化します
   ```yaoxiang
   let fields = Point.fields()  // コンパイル時評価 → 定数配列としてインライン化
   ```

3. **制約書き込み**：評価結果（生成された Trait 実装）を TraitTable に書き込みます

### 構文と例

#### 制約の導出：構造体に対して `Debug` 実装を生成

```yaoxiang
Debug: Type = { fmt: (Self, Formatter) -> Result }

// 型リフレクション API（標準ライブラリ組み込み）
type Field = { name: String, type: Type }
type Type = {
    fields: () -> Array[Field],
    implements: (trait: Type) -> Bool,
}

// derive 実装
derive_debug: (T: Type) -> Type = (T) => {
    fields = T.fields()
    for f in fields {
        if !f.type.implements(Debug) {
            return err("Field " + f.name + " does not implement Debug")
        }
    }
    // メソッド実装を生成（TraitTable に書き込み）
    T.fmt(self, f: Formatter) -> Result = { ... }
}

// 使用
Point: Type = { x: Float, y: Float }
derive_debug(Point)   # コンパイル時実行、Point の Debug 実装を生成
```

#### 条件制約

```yaoxiang
conditional: (T: Type) -> Type = (T) => {
    if T == Int { return Add[T] }
    else { return Default[T] }
}
```

#### 組合制約

```yaoxiang
both: (c1: Type, c2: Type) -> Type = (c1, c2) => c1 & c2
```

### 従来のマクロとの比較

| 次元 | 従来のマクロ（Rust） | YaoXiang Type4 |
|------|----------------|-----------------|
| 実行タイミング | マクロ展開フェーズ | 型チェックフェーズ |
| 言語能力 | テキスト置換 | 任意の YaoXiang コード |
| 評価方式 | テキスト生成 | コンパイル時 interpreter 評価 |
| エラー捕捉 | 展開後の型チェック | 評価時に即时捕捉 |
| 実行時オーバーヘッド | なし | なし（結果インライン化） |
| 合成可能性 | 悪い | 良い（関数合成） |

| 次元 | 従来のマクロ | YaoXiang Type4 |
|------|--------|-----------------|
| 操作対象 | テキスト/構文木 | 型/制約 |
| 実行フェーズ | マクロ展開フェーズ | 型チェックフェーズ |
| hygiene | 複雑なメカニズムが必要 | 自然に hygienic |
| 可読性 | マクロ構文が特殊でデバッグ困難 | 通常関数と一貫 |
| コンパイル検証 | 展開後の型チェック | 型チェックに直接参加 |

### なぜ型システムが今はマクロ取代できるのですか？

1. **型即ち第一級市民**：型は値のように計算され、传递され、内省されます。
2. **制約即ちデータ**：Type3 階層の制約は構築、組合、導出可能です。
3. **内省メカニズム**：型システムは型の構造を自省できます。
4. **型レベル関数**：型チェックフェーズで純粋関数を実行し、新しい制約を生成します。

これは型システム 내부에 **チューリング完全な、純粋関数型の、強型付けのメタプログラミング言語** を構築することに相当します。

## セマンティック Highlighting と LSP 実装

外部心のモデルの役割分類に基づき、LSP はコードに対して正確なセマンティック Highlighting を提供できます：

| セマンティック役割 | 外部階層 | トークン型 | 例 |
|----------|----------|----------|------|
| 変数/定数 | Type-1 | `variable` | `x`, `PI` |
| 関数引数 | Type-1 | `parameter` | `(a: Int)` の `a` |
| 通常関数 | Type2（戻り値が `Type` でない） | `function` | `add`, `factorial` |
| 型コンストラクタ | Type2（戻り値が `Type`） | `type` | `List`, `Vec` |
| 具体型 | Type1 | `type` | `Int`, `Point` |
| Trait | Type3 | `interface` | `Clone` |
| メタ型キーワード | Type0 | `keyword` | `Type` |

判定ルールは型チェック後の情報に基づきます：シンボル型が関数型 `(…) -> …` の場合、戻り値型が `Type`（またはその特殊化）かどうかを確認し、そうであれば型コンストラクタ → `type`、そうでなければ通常関数 → `function`。

## まとめ

- **外部心のモデル**：「型ユニバース思想」は開発者に言語内の各種概念（値、型、関数、Trait など）を理解し、統一的かつ直感的な視点を提供します。構文設計とセマンティック Highlighting をガイドします。
- **内部実装**：コンパイラは従来の累積宇宙階層を採用하여型システムの論理的厳密性を確保し、逆説を回避します。外部モデルに対しては完全に透過的です。
- **両者の関係**：外部モデルは記述的であり、コンパイラの実装に影響しません。内部実装は技術的であり、外部に階層詳細を露出しません。両者は相まって、YaoXiang の強力で使い易い型システムを共に構築します。

この分離により、YaoXiang は以下を実現します：

- **究極の統一**された構文
- **強力な表現力**を持つジェネリクスと依存型
- **開発者に優しい**セマンティックヒント
- **理論的に厳密**な基盤

これが YaoXiang 型ユニバース思想の完全な提示です——理解のための心のモデルの裏側に、堅実な従来の型理論の支えがあります。
