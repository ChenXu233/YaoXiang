## 概要

本計画は issue_13 要件文書に基づき、`yaoxiang fmt` フォーマッタを複数の実装段階に分解したものである。各段階には詳細な目標、受入基準、テスト項目を含む。

### アーキテクチャ選択

**AST + ソースマッピング** 方案を採用：

1. **Lexer** がコメント/空白の位置を記録 → Token ストリーム + ソースマッピングを生成
2. **Parser** が AST を生成（Span 情報含む）
3. **Formatter** が設定に基づいて AST + ソースマッピングを走査 → フォーマット済みコードを出力

**主な優位性**：

- 完全なコメント情報を保持
- 複雑なコード構造のリファクタリングをサポート
- LSP との深い統合が可能

### 0.1 ソースマッピング（SourceMap）データ構造

**実装目標**：

- ソースコードの位置マッピングを記録する `SourceMap` 構造を設計・実装
- 記録内容：コメント位置、空白領域、元の Token 位置
- 文字オフセットから行/列への変換をサポート

**データ構造設計**：

```
SourceMap {
    source: String,                    // 元のソースコード
    comments: Vec<Comment>,            // コメントリスト
    tokens: Vec<TokenWithSpan>,        // Token + 位置
    line_offsets: Vec<usize>,          // 各行の開始オフセット
    blank_lines: Vec<usize>,           // 空白行番号リスト
}

Comment {
    content: String,
    span: Span,
    style: CommentStyle,  // SingleLine, MultiLine, Doc
}
```

### 0.2 Lexer の強化 ✅ 完了

**実装目標**：

- Lexer を修正し、解析時にコメントをスキップするのではなく記録
- `tokenize()` 返戻時に `SourceMap` も返戻
- 後方互換性を維持し、既存の Token ストリームに影響しない

**実装説明**：独立した SourceMap スキャン方案を採用し、Lexer 自体は変更しない。
`SourceMap::build()` がソースコードを独立してスキャンしてコメント情報を抽出し、既存の Lexer/Parser フローに完全影響を与えない。

**受入基準**：

- [x] 既存の Token ストリーム出力が変わらない
