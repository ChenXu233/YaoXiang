# spawn for 構文の直交性

## 問題

現在の `spawn for x in items { body }` は独立したキーワード組み合わせ（`spawn` + `for`）であり、他のコンストラクトの `spawn { }` パターンと一貫していない：

```yx
spawn { }              ← spawn の後ろに {} ブロックをバインド
spawn for { }          ← spawn の後ろに for をバインド？これは spawn for キーワード組み合わせを形成する
for ... spawn { }      ← spawn の後ろに {} をバインド、for の body = spawn
```

核心的な問題：**`spawn for` と `spawn { for }` の意味的等価性、および `for spawn { }` と `spawn { for }` の意味的差異。**

## 3 つの形式の意味

| 形式 | 意味 | DAG 分解 |
|------|------|----------|
| `spawn for x in items { body }` | 各イテレーションが独立した並列タスク（データ並列） | コンパイラが N 個のイテレーションを N 個のクロージャに展開する |
| `spawn { for x in items { body } }` | for ループが並列ブロック内で 1 つのまとまりとして逐次実行される | for のイテレーションを分解しない |
| `for x in items spawn { body }` | for が逐次イテレーションし、各イテレーションで 1 つのタスクを spawn する | 各イテレーションで 1 つの spawn を作成する |

3 者は意味が異なり、自由に置き換えることはできない。

## DAG 分解の境界

現在の spawn の DAG 解析（`analyze_spawn_body`）が認識するのは spawn ブロック内の**静的な直接の子式**——コンパイラに見え、数も固定されたトップレベル式——である。for ループのイテレーション数は**動的**（実行時に決定）であり、静的な直接の子式リストにコンパイル時に展開することはできない。

したがって `spawn { for }` は自動的に `spawn for` と等価にはならない——DAG 解析が動的なイテレーション展開を扱えるようになるか、言語がイテレーション間の独立性をマークする別の仕組みを持つ場合を除く。

## 直交性の理想

```
spawn が {} ブロックを修飾 → spawn { ... }
{} には任意の文を含められる → spawn { for ... { ... } }
```

理想的には、`spawn { for }` はイテレーション間に依存のないループ本体を自動的に認識し、データ並列に展開すべきである。しかしこれは（自動ベクトル化のような）コンパイラ最適化の課題であり、構文設計の問題ではない。

## サスペンド項目

- [ ] `spawn for` を独立した構文として残すか？それとも `for ... spawn { }` または `spawn { for }` に縮退させるか？
- [ ] データ並列の表現には専用構文が必要か、それとも標準ライブラリ（例：`par_iter`）を使うか？
- [ ] 独立した構文を残す場合、`spawn for` の結合優先順位ルールは何か？
- [ ] `spawn for` は `else spawn` などのより複雑な組み合わせをサポートするか？
