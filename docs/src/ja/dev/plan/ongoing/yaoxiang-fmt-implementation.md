## 概要

本計画は issue_13  요구ドキュメントに基づき、`yaoxiang fmt` フォーマッターを複数の実装段階に分解したもの。各段階には詳細な目標、検証基準、テスト項目を含む。

### アーキテクチャ選択

**AST + ソースマップ** 方式を採用：

1. **Lexer** がコメント/空白の位置を記録 → Token ストリーム + ソースマップを生成
2. **Parser** が AST を生成（Span 情報を含む）
3. **Formatter** が設定に基づき AST + ソースマップを走査 → フォーマット済みコードを出力

**主な優位性**：

- 完全なコメント情報を保持
- 複雑なコード構造のリファクタリングをサポート
- LSP との深い統合が可能

- 文字オフセットから行/列への変換をサポート

**データ構造設計**：

```
SourceMap {
    source: String,                    // 元のソースコード
    comments: Vec<Comment>,            // コメントリスト
    tokens: Vec<TokenWithSpan>,        // Token + 位置
    line_offsets: Vec<usize>,          // 各行の開始オフセット
    blank_lines: Vec<usize>,           // 空白行番号リスト
}

Comment {
    content: String,
    span: Span,
    style: CommentStyle,  // SingleLine, MultiLine, Doc
}
```

### 0.2 Lexer の強化 ✅ 完了

**実装目標**：

- Lexer を修正し、解析時にコメントをスキップせずに記録
- `tokenize()` の戻り値で `SourceMap` を同時に返す
- 後方互換性を維持し、既存の Token ストリームに影響しない

**実装説明**：独立した SourceMap スキャン方式を採用し、Lexer 自体は変更しない。
`SourceMap::build()` がソースコードを独立してスキャンしてコメント情報を抽出するため、既存の Lexer/Parser プロセスに完全に影響しない。

**検証基準**：

```
終了コード:
```
