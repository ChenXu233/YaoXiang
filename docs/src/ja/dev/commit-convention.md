## コミット形式

**非常に重要です！！！！絶対に忘れないでください！！！**
すべてのコミットメッセージは以下の形式に従います：

```
:絵文字コード: type(scope): テーマ（日本語）

[任意のボディ内容]

[任意のフッター]
```

> ⚠️ **重要**: 直接絵文字を入力するのではなく、**絵文字コード**（例：`:sparkles:`）を使用する必要があります。
>
> **日本語でコミットメッセージを作成することを推奨**し、チーム内のコミュニケーションの一貫性を保ちます。

### 構成要素

| 要素 | 説明 | 必須 |
|------|------|------|
| 絵文字コード | コミットタイプを示す絵文字 | ✅ |
| type | コミットタイプ | ✅ |
| scope | 影響範囲 | ✅ |
| subject | 短い説明（日本語、50文字以内） | ✅ |
| body | 詳細な説明（任意） | ❌ |
| footer | 破壊的変更またはissueの закрытие（任意） | ❌ |

## メッセージ規範

### バージョン管理

**すべてのコミット前にバージョンを必ず上げてくだい**：

| バージョンタイプ | 更新場所 | 説明 |
| ---------- | ---------- | ---- |
| **major** | `pubspec.yaml` (version) + `release_v*.md` | 重大な更新、非互換のAPI変更 |
| **minor** | `pubspec.yaml` (version) | 新機能、後方互換 |
| **patch** | `pubspec.yaml` (version) | バグ修正、後方互換 |

### バージョン番号形式

セマンティックバージョニング `MAJOR.MINOR.PATCH` を採用：

```
# メジャーバージョン (breaking changes)
1.0.0 -> 2.0.0

# マイナーバージョン (new features)
1.0.0 -> 1.1.0

# パッチバージョン (bug fixes)
1.0.0 -> 1.0.1
```

### コミットフロー

```bash
# 1. コード変更後、まずバージョンを上げる
# semantic_releaseツールを使用してバージョンとChangelogを自動管理
npx semantic-release

# または手動でバージョンを更新
# pubspec.yamlのversionフィールドを編集

# 2. コードをコミット（バージョン変更は次のrelease時に自動生成）
git add .
git commit -m ":tada: Release v1.0.0"
git push
```

> 💡 バージョンの上げとChangelog生成はCIによって自動完了し、コミット時にはコード変更にバージョン更新が含まれていることを確認するだけで構いません。

## メッセージ規範

### 言語規範

**日本語でコミットメッセージを作成することを推奨**し、チーム内のコミュニケーションの一貫性を保ちます。

- Subjectは日本語で、簡潔で明確に
- Bodyは日本語で詳細に説明可能
- 特別な技術用語がある場合は英語のままでも可

### Subject（件名）

- 日本語で、簡潔で明確に
- 50文字を超えない
- 末尾に句点を付けない

### Body（本文）

- 変更理由と方法を詳細に説明
- 各行は72文字以内
- 要点を列挙する場合は - または * を使用

### Footer（フッター）

- **破壊的変更**: `BREAKING CHANGE:` で始まる
- **Issueの закрытие**: `閉じる #123` または `修正 #456` を使用

## 例

### ✨ feat - 新機能

```
:sparkles: feat(db): 批量削除功能的追加

一括でTodoを削除する機能を実装：
- TodoRepositoryにbatchDeleteメソッドを追加
- 削除確認ダイアログを追加
- 複数選択操作対応のUI更新

閉じる #42
```

### 🐛 fix - バグ修正

```
:bug: fix(provider): タイマー再開功能の修正

バックグラウンドから恢复した際、ポモドーロタイマーが продолжатьсяできない问题。
initStateに状态回复ロジックを追加。

修正 #128
```

### 📝 docs - ドキュメント更新

```
:memo: docs: README新機能説明の更新

以下の章节を追加：
-  التركيزモード
- データ統計
- BGM
```

### ♻️ refactor - リファクタリング

```
:recycle: refactor(ui): 共通のガラス風コンテナコンポーネントの抽出

再利用可能はGlassContainerコンポーネントを作成、
複数の画面間でコードを削減。
```

### ⚡️ perf - パフォーマンス改善

```
:zap: perf(db): 完了したTodo查询の性能最適化

completed_atフィールドにインデックスを追加、
WHERE句で完了したTodoをフィルタリングし、メモリ内フィルタリングではなく使用。

最適化前：45ms
最適化後：12ms
```

### ✅ test - テスト

```
:white_check_mark: test(provider): TodoProviderユニットテストの追加

対象シーン：
- Todoの追加
- 完了状态の切り替え
- Todoの削除
```

### 🔧 chore - 雑多

```
:wrench: chore: Flutterバージョンを3.19.0に更新

最小Flutterバージョンの要件を引き上げ、互換性のある依存関係を更新。
```

### 🚀 ci - CI設定

```
:rocket: ci: GitHub Actionsテストワークフローの追加

以下のステップを含むワークフローを作成：
- ユニットテスト
- 統合テスト
- コードカバレッジ
```

### 💄 style - フォーマット調整

```
:lipstick: style(todo_item): dart fixでコードをフォーマット

自動フォーマット修正を適用、
コードスタイルの統一性を維持。
```

## 🔖 リリースコミット

本次のコミットが**リリース（Release）**の場合、以下の規範に従う必要があります：

### リリースコミット形式

```
:bookmark: V<バージョン番号>: <リリースタイトル>

## 📦 バージョン情報

**发布日期:** YYYY-MM-DD

**バージョン番号:** <旧バージョン> → <新バージョン>

---

## ✨ 新機能

### <機能モジュール>
- :sparkles: feat(<scope>): <機能説明>

---

## ♻️ リファクタリング最適化

- :recycle: refactor(<scope>): <リファクタリング説明>

---

## 🐛 バグ修正

- :bug: fix(<scope>): <修正説明>

---

## 🔧 その他の変更

- :wrench: chore: <変更説明>

---

## 📦 新規追加ファイル

- `<ファイルパス>` - <ファイル説明>

---

## 📝 コミット記録

| コミット | 説明 |
|:---:|------|
| `<hash>` | :bookmark: V<バージョン番号> |
| `<hash>` | <コミットメッセージ> |
```

### リリース要件

1. **メッセージヘッダー**: `:bookmark:` + `V<バージョン番号>` 形式を必ず使用
2. **バージョン番号**: セマンティックバージョニング規範に従う
3. **内容の完全性**: 前回リリース以降の変更点を**すべて** 含める必要がある
4. **タイプ別分類**: `feat`、`fix`、`refactor`、`chore` などのタイプ別に整理
5. **コミット記録**: 関連するすべてのコミットのhashと説明を含める

### リリース例

```
:bookmark: V0.8.0: タスク統計とデータ分析機能の追加
```

### 1. Commit Template の設定

```bash
# プロジェクトルートディレクトリで実行
git config commit.template .gitmessage.txt
```

### 2. Templateファイル

プロジェクトルートディレクトリの `.gitmessage.txt` ファイル形式は以下の通りです：

```
# 絵文字コード type(scope): テーマ（日本語）
#
# 本文内容（任意）
#
# フッター（任意）
#
# Types: ✨feat, 🐛fix, 📝docs, 💄style, ♻️refactor, ⚡️perf, ✅test, 🔧chore, 🚀ci, 🔖release
# Scopes: core, db, ui, screen, widget, provider, repo, i18n, router, dep
#
# 例:
# ✨ feat(db): 一括削除機能の追加
# 🐛 fix(provider): タイマー再開問題の修正
#
# リリース形式: 🔖 V1.0.0: リリースタイトル
```

## よくある質問

### Q: コミットタイプはどのように選択すればよいですか？

- **feat**: ユーザーが目にする機能変更
- **fix**: ユーザーが報告した問題の修正
- **docs**: README、コメントなどのドキュメント
- **chore**: 依存関係の更新、設定ファイル
- **refactor**: 動作を変えないコード最適化

### Q: いつコミットを分割する必要がありますか？

- 各コミットは**1つのことだけ**を行う
- 関連する機能は一緒にコミットし、関係ないものは分ける
- Atomic Commitsの原則に従う
