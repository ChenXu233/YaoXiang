## コミットフォーマット

**非常重要！！！！忘れてはいけません！！！**
すべてのコミットメッセージは以下のフォーマットに従います：

```
:emojiコード: type(scope): テーマ（日本語）

[任意のボディ内容]

[任意のフッター]
```

> ⚠️ **重要**: **emoji コード**（例：`:sparkles:`）を使用しなければならず、emoji 文字を直接入力することはできません。
>
> **日本語でコミットメッセージを書くことを推奨します**。チーム間のコミュニケーションの一貫性を保つためです。

### 構成要素

| 部分 | 説明 | 必須 |
|------|------|------|
| emojiコード | コミットタイプを識別する絵文字 | ✅ |
| type | コミットタイプ | ✅ |
| scope | 影響範囲 | ✅ |
| subject | 簡潔な説明（日本語、50文字以内） | ✅ |
| body | 詳細な説明（任意） | ❌ |
| footer | 破壊的変更または Issue 終了（任意） | ❌ |

## メッセージ規範

### バージョニング

**コミット前に必ずバージョンを bump してください**：

| バージョンタイプ | 更新箇所 | 説明 |
|----------|----------|------|
| **major** | `pubspec.yaml` (version) + `release_v*.md` | 重大な更新、非互換の API 変更 |
| **minor** | `pubspec.yaml` (version) | 新機能、後方互換 |
| **patch** | `pubspec.yaml` (version) | バグ修正、後方互換 |

### バージョン番号フォーマット

セマンティックバージョニング `MAJOR.MINOR.PATCH` を採用：

```
# メジャーバージョン (breaking changes)
1.0.0 -> 2.0.0

# マイナーバージョン (new features)
1.0.0 -> 1.1.0

# パッチバージョン (bug fixes)
1.0.0 -> 1.0.1
```

### コミットフロー

```bash
# 1. コード変更後、まずバージョンを bump する
# semantic_release ツールを使用してバージョンと Changelog を自動管理
npx semantic-release

# または手動でバージョンを更新
# pubspec.yaml の version フィールドを編集

# 2. コードをコミット（バージョン変更は次の release時に自動生成される）
git add .
git commit -m ":tada: Release v1.0.0"
git push
```

> 💡 バージョンの bump と Changelog 生成は CI によって自動的に行われます。コミット時にはコード変更にバージョンの更新が含まれていることを確認してください。

## メッセージ規範

### 言語規範

**コミットメッセージには日本語を使用することを推奨します**。チーム間のコミュニケーションの一貫性を保つためです。

- Subject は日本語で、簡潔明瞭に
- Body は日本語で詳細に説明可能
- 特殊な技術用語がある場合は英語のままでも可

### Subject（テーマ）

- 日本語で、簡潔明瞭に
- 50文字以内に収める
- 末尾に句点を付けない

### Body（ボディ）

- 変更理由と方法を詳細に説明
- 各行は72文字以内に収める
- 要点は - または * で列挙

### Footer（フッター）

- **破壊的変更**: `BREAKING CHANGE:` で始める
- **Issue の終了**: `閉じる #123` または `修正 #456` を使用

## 示例

### ✨ feat - 新機能

```
:sparkles: feat(db): 批量削除功能的追加

批量削除待办事項機能を実装：
- TodoRepository に batchDelete メソッドを追加
- 削除確認ダイアログを追加
- 複数選択操作対応の UI を更新

#42 を閉じる
```

### 🐛 fix - バグ修正

```
:bug: fix(provider): ポモドーロタイマーがバックグラウンドで再開できない問題を修正

アプリがバックグラウンドから恢复”时、ポモドーロタイマーが続きません。
initState に狀態回復ロジックを追加しました。

#128 を修正
```

### 📝 docs - ドキュメント更新

```
:memo: docs: README の新機能説明を更新

以下のセクションを追加：
- フォーカスモード
- データ統計
- BGM
```

### ♻️ refactor - リファクタリング

```
:recycle: refactor(ui): 共通のガラス態コンテナコンポーネントを抽出

再利用可能な GlassContainer コンポーネントを作成し、
複数画面間のコード重複を削減。
```

### ⚡️ perf - パフォーマンス最適化

```
:zap: perf(db): 完了した待办の查询パフォーマンスを最適化

completed_at フィールドにインデックスを追加し、
メモリ内でフィルタリングするのではなく WHERE 句を使用して完了した待办をフィルタリング。

最適化前：45ms
最適化後：12ms
```

### ✅ test - テスト

```
:white_check_mark: test(provider): TodoProvider のユニットテストを追加

カバーするシナリオ：
- 待办の追加
- 完了狀態の切り替え
- 待办の削除
```

### 🔧 chore - 雑多

```
:wrench: chore: Flutter バージョンを 3.19.0 に更新

最小 Flutter バージョンの要件を上げ、互換性のある依存関係を更新。
```

### 🚀 ci - CI 設定

```
:rocket: ci: GitHub Actions テストワークフローを追加

以下のステップを含むワークフローを作成：
- ユニットテスト
- 統合テスト
- コードカバレッジ
```

### 💄 style - フォーマット調整

```
:lipstick: style(todo_item): dart fix でコードをフォーマット

自動フォーマット修正を適用し、
コードスタイルの一貫性を保つ。
```

## 🔖 リリースコミット

今回のコミットが**リリース（Release）**である場合、以下の規範に従う必要があります：

### リリースコミットフォーマット

```
:bookmark: V<バージョン番号>: <リリースタイトル>

## 📦 バージョン情報

**发布日期:** YYYY-MM-DD

**バージョン番号:** <旧バージョン> → <新バージョン>

---

## ✨ 新機能

### <機能モジュール>
- :sparkles: feat(<scope>): <機能説明>

---

## ♻️ リファクタリング最適化

- :recycle: refactor(<scope>): <リファクタリング説明>

---

## 🐛 バグ修正

- :bug: fix(<scope>): <修正説明>

---

## 🔧 その他の変更

- :wrench: chore: <変更説明>

---

## 📦 新規追加ファイル

- `<ファイルパス>` - <ファイル説明>

---

## 📝 コミット記録

| コミット | 説明 |
|:---:|------|
| `<hash>` | :bookmark: V<バージョン番号> |
| `<hash>` | <コミットメッセージ> |
```

### リリース要件

1. **メッセージヘッダー**: `:bookmark:` + `V<バージョン番号>` フォーマットを必ず使用
2. **バージョン番号**: セマンティックバージョニング仕様に従う
3. **コンテンツの完全性**: 前回のリリースからの**すべての commit** 内容紹介を含める必要がある
4. **タイプ別分類**: `feat`, `fix`, `refactor`, `chore` などのタイプ別に整理
5. **コミット記録**: すべての関連コミットの hash と説明を一括列出

### リリース例

```
:bookmark: V0.8.0: タスク統計とデータ分析機能の追加

## 📦 バージョン情報

**发布日期:** 2025-01-15

**バージョン番号:** 0.7.0 → 0.8.0

---

- :sparkles: feat(statistics): 日次/週次のデータ可視化チャートを追加

### 待办の強化
- :sparkles: feat(todo): タスク優先度フィルタリング機能を追加

---

## 🐛 バグ修正

- :bug: fix(todo): タスクリストのスワイプト滞滞問題を修正

---

## 🔧 その他の変更

- :wrench: chore: 依存関係バージョンを最新安定版に更新
- :memo: docs: README インストール説明を更新

## 📝 コミット記録

| コミット | 説明 |
```

```bash

git log --oneline -20
```

### 1. Commit Template の設定

```bash

```

### 2. Template ファイル

プロジェクトルートディレクトリの `.gitmessage.txt` ファイルフォーマットは以下の通りです：

```
# emojiコード type(scope): テーマ（日本語）
#

#
# フッター（任意）
#
# Types: ✨feat, 🐛fix, 📝docs, 💄style, ♻️refactor, ⚡️perf, ✅test, 🔧chore, 🚀ci, 🔖release

#
# リリースフォーマット: 🔖 V1.0.0: リリースタイトル
```

## よくある質問

### Q: コミットタイプはどのように選択すればよいですか？

- **feat**: ユーザーが目にする機能の変更
- **fix**: ユーザーから報告された問題の修正
- **docs**: README、コメントなどのドキュメント
- **chore**: 依存関係の更新、設定ファイル
- **refactor**: 動作を変えないコード最適化

### Q: いつコミットを分割する必要がありますか？

- 各コミットは**1つのことだけ**を行う
- 関連する機能は一緒にコミットし、関連しないものは別々にコミット
- Atomic Commits の原則に従う
