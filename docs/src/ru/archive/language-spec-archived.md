## Глава 2: Лексическая структура

### 2.1 Исходные файлы

Исходные файлы YaoXiang должны использовать кодировку UTF-8. Исходные файлы обычно имеют расширение `.yx`.

### 2.2 Классификация лексем

| Категория | Описание | Пример |
|-----------|----------|--------|
| Идентификаторы | Начинаются с буквы или подчёркивания | `x`, `_private`, `my_var` |
| Ключевые слова | Зарезервированные слова языка | `Type`, `pub`, `use` |
| 字面量 | Фиксированные значения | `42`, `"hello"`, `true` |
| Операторы | Знаки операций | `+`, `-`, `*`, `/` |
| Разделители | Синтаксические разделители | `(`, `)`, `{`, `}`, `,` |

### 2.3 Ключевые слова

YaoXiang определяет минимальное количество ключевых слов:

```
pub    use    spawn
ref    mut    if     elif
else   match  while  for    return
break  continue as     in     unsafe
```

Эти ключевые слова имеют особое значение в любом контексте и не могут использоваться в качестве идентификаторов.

### 2.4 Зарезервированные слова

| Зарезервированное слово | Тип | Описание |
|------------------------|------|----------|
| `Type` | Type | Метатип |
| `true` | Bool | Логическая истина |
| `false` | Bool | Логическая ложь |
| `void` | Void | 空值 |
| `some(T)` | Option | 值变体 Option |
| `ok(T)` | Result | Вариант успеха Result |
| `err(E)` | Result | Вариант ошибки Result |

### 2.5 Идентификаторы

Идентификаторы начинаются с буквы или подчёркивания, последующие символы могут быть буквами, цифрами или подчёркиваниями. Идентификаторы чувствительны к регистру.

Специальные идентификаторы:
- `_` используется как заполнитель, означающий игнорирование значения
- Идентификаторы, начинающиеся с подчёркивания, обозначают приватные члены

## Глава 3: Типы

### 3.1 Классификация типов

```
TypeExpr    ::= PrimitiveType
              | StructType
              | EnumType
              | InterfaceType
              | TupleType
              | FnType
              | GenericType
              | TypeRef
              | TypeUnion
              | TypeIntersection
              | ConstrainedType
              | AssociatedType
```

### 3.2 原类型

| Тип | Описание | Размер по умолчанию |
|------|----------|---------------------|
| `Type` | Метатип | 0 байт |
| `Void` | 空值 | 0 байт |
| `Bool` | Логическое значение | 1 байт |
| `Int` | Знаковое целое | 8 байт |
| `Uint` | Беззнаковое целое | 8 байт |
| `Float` | 浮点数 | 8 байт |
| `String` | Строка UTF-8 | Переменный |
| `Char` | Символ Unicode | 4 байта |
| `Bytes` | Сырые байты | Переменный |

Целые с шириной бит: `Int8`, `Int16`, `Int32`, `Int64`, `Int128`
Плавающие с шириной бит: `Float32`, `Float64`

### 3.3 记录类型

**Унифицированный синтаксис**：`Name: Type = { field1: Type1, field2: Type2, ... }`

```
RecordType  ::= '{' FieldList? '}'
FieldList   ::= Field (',' Field)* ','?
Field       ::= Identifier ':' TypeExpr
            |  Identifier                 # 接口约束
```

```yaoxiang
// Простой记录类型
Point: Type = { x: Float, y: Float }

// Пустой记录类型
Empty: Type = {}

// записывающий тип с дженериками
Pair: Type[T] = { first: T, second: T }

// записывающий тип с реализацией интерфейса
Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable
}
```

**Правила**：
- 记录类型 определяются фигурными скобками `{}`
- После имени поля следует двоеточие и тип
- Имена интерфейсов внутри тела типа означают реализацию этого интерфейса

#### 3.3.1 Значения полей по умолчанию

Поля типа могут иметь значения по умолчанию, которые можно указать при конструировании:

```yaoxiang
// Поля со значениями по умолчанию - необязательны при конструировании
Point: Type = {
    x: Float = 0,
    y: Float = 0
}

// Использование
Point()           # → Point(x=0, y=0)
Point(x=1)       # → Point(x=1, y=0)
Point(x=1, y=2) # → Point(x=1, y=2)

// Поля без значений по умолчанию - обязательны при конструировании
Point2: Type = {
    x: Float,
    y: Float
}

// Использование
Point2(x=1, y=2) # ✓
Point2()          # ✗ Ошибка
```

**Правила**：
- `field: Type = expression` → есть значение по умолчанию, необязательно при конструировании
- `field: Type` → нет значения по умолчанию, обязательно при конструировании

#### 3.3.2 内置绑定

Внутри определения типа можно напрямую привязать методы:

```yaoxiang
// Способ 1: Привязка внешней функции
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]    # Привязка к позиции 0
}
// Вызов：p1.distance(p2) → distance(p1, p2)

// Способ 2: Анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис：((params) => body)[position]
// Вызов：p1.distance(p2) → distance(p1, p2)
```

### 3.4 枚举类型 (вариантные типы)

```
EnumType    ::= '{' Variant ('|' Variant)* '}'
Variant     ::= Identifier (':' TypeExpr)?
```

**Синтаксис**：`Name: Type = { Variant1 | Variant2(params) | ... }`

```yaoxiang
// Варианты без параметров
Color: Type = { red | green | blue }

// Варианты с параметрами
Option: Type[T] = { some(T) | none }

// Смешанный
Result: Type[T, E] = { ok(T) | err(E) }

// Варианты без параметров эквивалентны конструкторам без параметров
Bool: Type = { true | false }
```

### 3.5 接口类型

```
InterfaceType ::= '{' FnField (',' FnField)* ','?
FnField       ::= Identifier ':' FnType
FnType        ::= '(' ParamTypes? ')' '->' TypeExpr
```

**Синтаксис**：Интерфейс — это запись, все поля которой являются типами функций

```yaoxiang
// Определение интерфейса
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой интерфейс
EmptyInterface: Type = {}
```

**Реализация интерфейсов**：Тип реализует интерфейс, указывая имя интерфейса в конце определения

```yaoxiang
// Тип, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,        # Реализует интерфейс Drawable
    Serializable     # Реализует интерфейс Serializable
}
```

**Прямое присваивание интерфейсу**：Конкретный тип можно напрямую присвоить переменной типа интерфейса (структурное подтипирование)

```yaoxiang
// Прямое присваивание (конкретный тип известен на этапе компиляции → вызов без накладных расходов)
d: Drawable = Circle(1)
d.draw(screen)        # После компиляции: прямой вызов circle_draw, без vtable

// Возврат функции (конкретный тип неизвестен на этапе компиляции → вызов через vtable)
d: Drawable = get_shape()
d.draw(screen)        # Поиск метода через vtable

// Интерфейс как параметр функции
process: (d: Drawable) -> Void = d.draw(screen)
```

**Стратегия оптимизации на этапе компиляции**：

| Сценарий | Результат вывода | Способ вызова |
|----------|------------------|---------------|
| Прямое присваивание конкретного типа | Конкретный тип определён | Прямой вызов (без накладных расходов) |
| Возврат из функции | Неизвестен | vtable |
| Гетерогенная коллекция | Несколько типов | vtable |

### 3.9 类型约束

#### 3.9.1 单个约束

```

```

```yaoxiang
// Определение интерфейсного типа (как ограничения)
Clone: Type = {
    clone: (Self) -> Self
}

// Использование ограничения
clone: [T: Clone](value: T) -> T = value.clone()
```

#### 3.9.2 多重约束

```yaoxiang
// Синтаксис множественных ограничений
combine: [T: Clone + Add](a: T, b: T) -> T = {
    return a.clone() + b
}

// Сортировка обобщённого контейнера
sort: [T: Clone + PartialOrd](list: List[T]) -> List[T] = {
    result = list.clone()
    quicksort(&mut result)
    return result
}
```

#### 3.9.3 Ограничения типов функций

```yaoxiang
Iterator: Type[T] = {
    Item: T,                    # Ассоциированный тип
    next: (Self) -> Option[T],
    has_next: (Self) -> Bool
}

// Использование ассоциированного типа

        if let Some(item) = iter.next() {
            result.push(item)
        }
    }
    return result
}
```

### 3.11 编译期泛型

#### 3.11.1 Ограничения литерных типов

```
LiteralType   ::= Identifier ':' Int          # Константа этапа компиляции
CompileTimeFn ::= '[' Identifier ':' Int ']' '(' Identifier ')' '->' TypeExpr
```

**Основная идея**：Использование обобщённого параметра `[n: Int]` + параметра значения `(n: n)` для различения констант этапа компиляции и значений этапа выполнения.

```yaoxiang
// Факториал этапа компиляции: параметр должен быть литералом, известным на этапе компиляции
factorial: [n: Int](n: n) -> Int = {
    return match n {
        0 => 1,
        _ => n * factorial(n - 1)
    }
}

// Массив констант этапа компиляции
StaticArray: Type[T, N: Int] = {
    data: T[N],      # Массив с размером, известным на этапе компиляции
    length: N
}

// Использование
arr: StaticArray[Int, factorial(5)]  # Компилятор вычисляет factorial(5) = 120 на этапе компиляции
```

#### 3.11.2 Массивы констант этапа компиляции

```yaoxiang
identity_matrix: [T: Add + Zero + One, N: Int](size: N) -> Matrix[T, N, N] = {
    // ...
}
```

## Глава 3 (продолжение)：Пояснения к дизайну синтаксиса

### 3.17 Связь между именованными функциями и Lambda

**Основное понимание**：Именованные функции и лямбда-выражения — это одно и то же! Единственное различие: именованная функция просто даёт имя лямбда-выражению.

```yaoxiang
// По сути это одно и то же
add: (a: Int, b: Int) -> Int = a + b           # Именованная функция (рекомендуется)
add: (a: Int, b: Int) -> Int = (a, b) => a + b  # Лямбда-форма (полностью эквивалентно)
```

**Модель синтаксического сахара**：

```
// Именованная функция = Lambda + имя
name: (Params) -> ReturnType = body

// По сути это
name: (Params) -> ReturnType = (params) => body
```

**Ключевой момент**：Когда сигнатура полностью объявляет типы параметров, имена параметров в заголовке лямбды становятся избыточными и могут быть опущены.

### 3.18 Правила области видимости параметров

**Параметры перекрывают внешние переменные**：Область видимости параметров в сигнатуре перекрывает тело функции, причём внутренняя область видимости имеет более высокий приоритет.

```yaoxiang
x = 10  # Внешняя переменная
double: (x: Int) -> Int = x * 2  # ✅ Параметр x перекрывает внешний x, результат 20
```

### 3.19 Расположение аннотаций типов

Аннотации типов могут находиться в любом из следующих мест, **достаточно указать хотя бы в одном**:

| Расположение | Форма | Описание |
|--------------|-------|----------|
| Только в сигнатуре | `double: (x: Int) -> Int = x * 2` | ✅ Рекомендуется |
| Только в заголовке лямбды | `double = (x: Int) => x * 2` | ✅ Допустимо |
| С обеих сторон | `double: (x: Int) -> Int = (x: Int) => x * 2` | ✅ Избыточно, но допустимо |

### 4.1 Классификация выражений

#### 5.9.1 Семантика: каждая итерация связывает новое значение

Семантика цикла for в YaoXiang отличается от традиционных языков: **каждая итерация связывает новое значение, а не изменяет одну и ту же переменную**.

```yaoxiang
// Пример：for i in 1..5
for i in 1..5 {
    print(i)
}
```

**Процесс выполнения**：

| Итерация | Поведение переменной цикла |
|----------|----------------------------|
| Первая | Создаётся новая привязка `i = 1`, выполняется тело, печатается 1 |
| Вторая | Создаётся новая привязка `i = 2` (предыдущая привязка уничтожена), выполняется тело, печатается 2 |
| Третья | Создаётся новая привязка `i = 3`, выполняется тело, печатается 3 |
| Четвёртая | Создаётся новая привязка `i = 4`, выполняется тело, печатается 4 |
| Конец | Тело цикла завершено, привязка уничтожена |

**Ключевой момент**：В конце каждой итерации привязка, созданная в этой итерации, уничтожается. Следующая итерация — это совершенно новая привязка, не связанная с предыдущей.

#### 5.9.2 Разница между for и for mut

| Синтаксис | Изменяемость переменной цикла | Описание |
|-----------|-------------------------------|----------|
| `for i in 1..5` | Неизменяемая | В теле цикла нельзя изменить привязку |
| `for mut i in 1..5` | Изменяемая | В теле цикла можно изменить привязку |

```yaoxiang
// ✅ Допустимо: каждая итерация связывает новое значение, изменение не требуется
for i in 1..5 {
    print(i)  # Чтение значения i
}

// ❌ Ошибка: неизменяемая привязка, нельзя изменить
for i in 1..5 {
    i = i + 1  # Ошибка: нельзя изменять неизменяемую привязку
}

// ✅ Допустимо: использование for mut позволяет изменять привязку
for mut i in 1..5 {
    i = i + 1  # Допускается изменение
}
```

#### 5.9.3 Проверка затенения

Переменная цикла for не может затенять переменную, уже существующую во внешней области видимости:

```yaoxiang
// ❌ Ошибка: i уже объявлена снаружи
i = 10
for i in 1..5 {
    print(i)
}

// ✅ Правильно: использовать другое имя переменной
i = 10
for j in 1..5 {
    print(j)
}
```

Код ошибки: `E2013 - Cannot shadow existing variable`

#### 5.9.4 Сравнение с другими языками

| Язык | Семантика переменной цикла for |
|------|--------------------------------|
| YaoXiang | Каждая итерация связывает новое значение |
| Rust | Изменяется одна и та же переменная (нужен mut) |
| Python | Изменяется одна и та же переменная (mut не нужен) |
| C/C++ | Изменяется одна и та же переменная (нужен указатель или ссылка) |

**Обоснование дизайна**：YaoXiang использует семантику привязки потому, что:
1. Переменные в теле цикла уничтожаются в конце каждой итерации
2. Следующая итерация — это совершенно новая привязка
3. Это безопаснее, не нужно учитывать состояние между итерациями

### 6.1 Унифицированная модель функций

**Основной синтаксис**：`name: type = value`

YaoXiang использует **унифицированную модель объявлений**: переменные, функции и методы используют одинаковую форму `name: type = value`.

```
Declaration   ::= Identifier ':' Type '=' Expression
FunctionDef   ::= Identifier GenericParams? '(' Parameters? ')' '->' Type '=' (Expression | Block)
GenericParams ::= '[' Identifier (',' Identifier)* ']'
Parameters    ::= Parameter (',' Parameter)*
Parameter     ::= Identifier ':' TypeExpr
```

### 6.2 Объявление переменных

```yaoxiang
// Базовый синтаксис
```

### 6.3 Определение функций

#### 6.3.1 Полный синтаксис

```yaoxiang
// Имена параметров объявляются в сигнатуре
add: (a: Int, b: Int) -> Int = {
    return a + b
}

// Один параметр

// Функция без параметров

    print("Hello")
}

// Тело из нескольких строк

        _ => 0.0
    }
}
```

#### 6.3.2 Правила возврата

```yaoxiang
// Не-void тип возврата - необходимо использовать return
add: (a: Int, b: Int) -> Int = {
    return a + b
}

// Void тип возврата - return необязателен
print: (msg: String) -> Void = {
    // return не нужен
}
```

### 6.4 Обобщённые функции

```yaoxiang
// Определение обобщённой функции
map: [T, R](list: List[T], f: Fn(T) -> R) -> List[R] = {

// Использование ограничений типа
```

### 6.5 Определение методов

#### 6.5.1 Типовые методы

**Синтаксис**：`Type.method: (self: Type, ...) -> Return = ...`

```yaoxiang
// Типовой метод: привязан к конкретному типу
Point.draw: (self: Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}

Point.serialize: (self: Point) -> String = {

}

// Использование синтаксического сахара методов
p: Point = Point(1.0, 2.0)
p.draw(screen)           # Синтаксический сахар → Point.draw(p, screen)
```

#### 6.5.2 Обычные методы

**Синтаксис**：`name: (Type, ...) -> Return = ...` (не привязаны к типу)

```yaoxiang
// Обычный метод: не привязан к типу, как независимая функция
distance: (p1: Point, p2: Point) -> Float = {
    dx = p1.x - p2.x
    dy = p1.y - p2.y
    return (dx * dx + dy * dy).sqrt()
}
```

### 6.6 Привязка методов

#### 6.6.1 Ручная привязка

**Синтаксис**：`Type.method = function[positions]`

```yaoxiang
// Привязка к позиции 0 (по умолчанию)
Point.distance = distance[0]

// Привязка к позиции 1
Point.transform = transform[1]

// Привязка к нескольким позициям
Point.scale = scale[0, 1]

// Использование заполнителей
Point.calc = func[0, _, 2]
```

#### 6.6.2 pub автопривязка

Функции, объявленные с `pub`, автоматически привязываются компилятором к типам, определённым в том же файле:

```yaoxiang
// Объявление с pub, компилятор автоматически привязывает
pub distance: (p1: Point, p2: Point) -> Float = {
    dx = p1.x - p2.x
    dy = p1.y - p2.y
    return (dx * dx + dy * dy).sqrt()
}

// Компилятор автоматически выводит:
// 1. Point определён в текущем файле
// 2. Параметр функции содержит Point
// 3. Выполняется Point.distance = distance[0]

// Вызов
d = distance(p1, p2)           # Функциональный стиль
d2 = p1.distance(p2)           # ООП синтаксический сахар
```

### 6.7 Правила привязки методов

| Правило | Описание |
|---------|----------|
| Позиции начинаются с 0 | `func[0]` привязывает первый параметр (индекс 0) |
| Максимальная позиция | Должна быть < количества параметров функции |
| Отрицательные индексы | `[-1]` означает последний параметр |
| Заполнители | `_` пропускает эту позицию, предоставляется пользователем |

### 6.8 Поддержка карринга

Привязка естественно поддерживает карринг. Когда при вызове предоставлено меньше параметров, чем осталось, возвращается функция, принимающая оставшиеся параметры:

```yaoxiang
// Исходная функция: 5 параметров
calculate: (scale: Float, a: Point, b: Point, x: Float, y: Float) -> Float = ...

// Привязка: Point.calc = calculate[1, 2]
// После привязки остались параметры: scale, x, y

// Сценарии вызова
p1.calc(2.0, 10.0, 20.0)       # Предоставлено 3 параметра → прямой вызов
p1.calc(2.0)                    # Предоставлен 1 параметр → возврат (Float, Float) -> Float
p1.calc()                       # Предоставлено 0 параметров → возврат (Float, Float, Float) -> Float
```

### 6.9 并作函数与注解

#### 6.9.1 spawn функции (并作函数)

```
SpawnFn     ::= Identifier ':' FnType 'spawn' '=' Lambda
FnType      ::= '(' ParamTypes? ')' '->' TypeExpr ('@' Annotation)?
Annotation  ::= 'block' | 'eager'
```

**Аннотации функций**：

| Аннотация | Расположение | Поведение |
|-----------|--------------|-----------|
| `@block` | После типа возврата | Отключает оптимизацию параллелизма, полностью последовательное выполнение |
| `@eager` | После типа возврата | Принудительное энергичное вычисление |

**Примеры синтаксиса**：

```
// 并作 функция: может выполняться параллельно
fetch_data: (url: String) -> JSON spawn = { ... }

// @block синхронная функция: полностью последовательное выполнение
main: () -> Void @block = { ... }

// @eager энергичная функция: немедленное выполнение
compute: (n: Int) -> Int @eager = { ... }
```

#### 6.9.2 并作块

Явно объявленная область параллелизма, задачи внутри блока будут выполнены 并作:

```
SpawnBlock  ::= '(' Pattern (',' Pattern)* ')' '=' 'spawn' '{' Expr (',' Expr)* '}'
```

**Пример**：

```
// 并作 блок: явный параллелизм
(result_a, result_b) = spawn {
    parse(fetch("url1")),
    parse(fetch("url2"))
}
```

#### 6.9.3 并作循环

Цикл с данными параллелизмом, тело цикла выполняется 并作 для всех элементов данных:

```
SpawnFor    ::= Identifier '=' 'spawn' 'for' Identifier 'in' Expr '{' Expr '}'
```

**Пример**：

```
// 并作 цикл: параллелизм данных
results = spawn for item in items {
    process(item)
}
```

#### 6.9.4 Оператор распространения ошибок

```
ErrorPropagate ::= Expr '?'
```

**Пример**：

```
process: (p: Point) -> Result[Data, Error] = {
    data = fetch_data()?      # Автоматическое распространение ошибки
    return transform(data)?
}
```

## Глава 8: Управление памятью

### 8.1 Модель владения

YaoXiang использует **модель владения** для управления памятью, каждый значение имеет единственного владельца:

| Семантика | Описание | Синтаксис |
|-----------|----------|-----------|
| **Move** | Семантика по умолчанию, передача владения | `p2 = p` |
| **ref** | Совместное использование (Arc с подсчётом ссылок) | `shared = ref p` |
| **clone()** | Явное копирование | `p2 = p.clone()` |

### 8.2 Семантика Move (по умолчанию)

```yaoxiang
// Присваивание = Move (нуль-копирование)
p: Point = Point(1.0, 2.0)
p2 = p              # Move, p становится недействительным

// Передача параметров функции = Move
process: (p: Point) -> Void = {
    // Владение p передаётся внутрь
}

// Возврат значения = Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    return p        # Move, владение передаётся
}
```

### 8.3 Ключевое слово ref (Arc)

Ключевое слово `ref` создаёт **указатель с подсчётом ссылок** (Arc) для безопасного совместного использования:

```yaoxiang
// Создание Arc
p: Point = Point(1.0, 2.0)
shared = ref p      # Arc, потокобезопасный

// Совместный доступ
spawn(() => print(shared.x))   # ✅ Безопасно

// Arc автоматически управляет жизненным циклом
// Когда shared покидает область видимости, счётчик становится нулём и память освобождается
```

**Характеристики**：
- Потокобезопасный подсчёт ссылок
- Автоматическое управление жизненным циклом
- Безопасность через границы spawn

### 8.4 clone() явное копирование

```yaoxiang
// Явное копирование значения
p: Point = Point(1.0, 2.0)
p2 = p.clone()      # p и p2 независимы

// Оба могут быть изменены, не влияя друг на друга
p.x = 0.0           # ✅
p2.x = 0.0          # ✅
```

### 8.5 unsafe блоки

Блоки `unsafe` позволяют использовать сырые указатели для системного программирования:

```yaoxiang
// Тип сырого указателя
PtrType ::= '*' TypeExpr

// unsafe блок
UnsafeBlock ::= 'unsafe' '{' Stmt* '}'
```

**Пример**：

```yaoxiang
p: Point = Point(1.0, 2.0)

// Сырые указатели можно использовать только в unsafe блоках
unsafe {
    ptr: *Point = &p     # Получение сырого указателя
    (*ptr).x = 0.0       # Разыменование
}
```

**Ограничения**：
- Сырые указатели можно использовать только в блоках `unsafe`
- Пользователь гарантирует отсутствие висячих указателей и использования после освобождения
- Не участвуют в проверках Send/Sync

### 8.7 BNF синтаксиса владения

```bnf
// === Выражения владения ===

// Move (по умолчанию)
MoveExpr     ::= Expr

// ref Arc
RefExpr      ::= 'ref' Expr

// clone
CloneExpr    ::= Expr '.clone' '(' ')'

// === Сырые указатели (только unsafe) ===

PtrType       ::= '*' TypeExpr
UnsafeBlock   ::= 'unsafe' '{' Stmt* '}'
```

### 8.8 Ограничения Send / Sync

| Ограничение | Семантика | Описание |
|-------------|-----------|----------|
| **Send** | Можно безопасно передавать между потоками | Значение можно переместить в другой поток |
| **Sync** | Можно безопасно совместно использовать между потоками | Неизменяемые ссылки можно передавать в другой поток |

**Автоматический вывод**：

```
// Правила вывода Send
Struct[T1, T2]: Send ⇐ T1: Send 且 T2: Send

// Правила вывода Sync
Struct[T1, T2]: Sync ⇐ T1: Sync 且 T2: Sync
```

**Ограничения типа**：

| Тип | Send | Sync | Описание |
|-----|------|------|----------|
| `T` (значение) | ✅ | ✅ | Неизменяемые данные |
| `ref T` | ✅ | ✅ | Arc потокобезопасный |
| `*T` | ❌ | ❌ | Сырой указатель небезопасен |

### 8.7 Ограничения Send/Sync

YaoXiang использует ограничения типов в стиле Rust для обеспечения безопасности параллелизма:

| Ограничение | Семантика | Описание |
|-------------|-----------|----------|
| **Send** | Можно безопасно передавать между потоками | Значение можно переместить в другой поток |
| **Sync** | Можно безопасно совместно использовать между потоками | Неизменяемые ссылки можно передавать в другой поток |

**Иерархия ограничений**：

```
Send ──► Можно безопасно передавать между потоками
  │
  └──► Sync ──► Можно безопасно совместно использовать между потоками
       │
       └──► Типы, удовлетворяющие Send + Sync, могут автоматически работать параллельно

Arc[T] реализует Send + Sync (потокобезопасный подсчёт ссылок)
Mutex[T] обеспечивает внутреннюю изменяемость
```

### 8.8 Потокобезопасные типы

| Тип | Семантика | Параллельная безопасность | Описание |
|-----|-----------|---------------------------|----------|
| `T` | Неизменяемые данные | ✅ Безопасно | Тип по умолчанию, чтение несколькими задачами без гонок |
| `Ref[T]` | Изменяемая ссылка | ⚠️ Требует синхронизации | Отмечен как изменяемый для параллельного изменения, компилятор проверяет использование блокировок |
| `Atomic[T]` | Атомарный тип | ✅ Безопасно | Низкоуровневые атомарные операции, безблокировочный параллелизм |
| `Mutex[T]` | Обёртка мьютекса | ✅ Безопасно | Автоматическая блокировка/разблокировка, гарантируется компилятором |
| `RwLock[T]` | Обёртка читатель-писатель | ✅ Безопасно | Оптимизация для сценариев с частыми чтениями и редкими записями |

**Синтаксис**：

```
Mutex[T]    # Мьютекс, обёртывающий изменяемые данные
Atomic[T]   # Атомарный тип (только для Int, Float и т.д.)
RwLock[T]   # Читатель-писатель, обёртывающий данные
```

**Синтаксический сахар with**：

```
with mutex.lock() {
    // Критическая секция: защищена Mutex
    ...
}
```

## Глава 9: Обработка ошибок

### 9.1 Тип Result

```
Result: Type[T, E] = ok(T) | err(E)
```

**Конструкторы вариантов**：

| Вариант | Синтаксис | Описание |
|---------|-----------|----------|
| `ok(T)` | `ok(value)` | Значение успеха |
| `err(E)` | `err(error)` | Значение ошибки |

### 9.2 Тип Option

```
Option: Type[T] = some(T) | none
```

**Конструкторы вариантов**：

| Вариант | Синтаксис | Описание |
|---------|-----------|----------|
| `some(T)` | `some(value)` | Есть значение |
| `none` | `none` | Нет значения |

### 9.3  распространение ошибок

```
ErrorPropagate ::= Expr '?'
```

Оператор `?` автоматически распространяет ошибки типа Result:

```
// При успехе возвращает значение, при неудаче возвращает err наверх
data = fetch_data()?

// Эквивалентно
data = match fetch_data() {
    ok(v) => v
    err(e) => return err(e)
}
```

## Глава 10: Привязка методов

### 10.1 Обзор привязки

YaoXiang использует **чисто функциональный дизайн**, все операции реализуются через функции. Механизм привязки связывает функции с типами, позволяя вызывающему вызывать функцию как метод.

```
Привязка ::= Тип '.' Идентификатор '=' ИмяФункции '[' СписокПозиций ']'
СписокПозиций ::= Позиция (',' Позиция)* ','?
Позиция     ::= Целое (начиная с 0) | Отрицательное целое | Заполнитель
```

**Основные правила**：
- Индексы позиций начинаются с **0**
- По умолчанию привязка к позиции **0** (первый параметр)
- Поддерживаются отрицательные индексы `[-1]` для последнего параметра
- Объединённая привязка нескольких позиций `[0, 1, 2]`
- Заполнитель `_` означает пропуск этой позиции

### 10.2 Синтаксис привязки

**Синтаксис**：
```
Type.method = func[position]
Type.method = func[0, 1, 2]    # Привязка к нескольким позициям
Type.method = func[0, _, 2]   # Использование заполнителей
Type.method = func[-1]        # Отрицательный индекс (последний параметр)
```

**Семантика**：
- `Type.method = func[0]` означает, что при вызове `obj.method(arg)` объект `obj` привязывается к параметру `func` с индексом 0
- Оставшиеся параметры заполняются в исходном порядке

### 10.3 Примеры привязки

```yaoxiang
// === Базовая привязка ===

// Исходная функция
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

// Привязка к типу Point (позиция 0)
Point.distance = distance[0]

// Использование
p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)
d = p1.distance(p2)  # → distance(p1, p2)

// === Привязка к нескольким позициям ===

// Исходная функция
calculate: (scale: Float, a: Point, b: Point, x: Float, y: Float) -> Float = ...

// Привязка нескольких позиций
Point.calc_scale = calculate[0]      # Привязка только scale
Point.calc_both = calculate[1, 2]    # Привязка двух параметров Point

// Использование
f = p1.calc_scale(2.0)  # → calculate(2.0, p1, _, _, _)
result = f(p2, 10.0, 20.0)  # → calculate(2.0, p1, p2, 10.0, 20.0)

// === Карринг (при недостатке параметров автоматически возвращается функция) ===

// Привязка одного параметра
Point.distance_to = distance[0]

// Использование - без предоставления второго параметра возвращается каррированная функция
f = p1.distance_to(p2)  # → distance(p1, p2) прямой вызов
f2 = p1.distance_to()   # → distance(p1, _) возвращает функцию (Point) -> Float
result = f2(p2)         # → distance(p1, p2)
```

### 10.4 Правила привязки

**Правила позиций**：
| Правило | Описание |
|---------|----------|
| Позиции начинаются с 0 | `func[0]` привязывает первый параметр (индекс 0) |
| Максимальная позиция | Должна быть < количества параметров функции |
| Отрицательные индексы | `[-1]` означает последний параметр |
| Заполнители | `_` пропускает эту позицию |

**Проверка типов**：
```yaoxiang
// ✅ Допустимая привязка
Point.distance = distance[0]          # distance(Point, Point)
Point.calc = calculate[1, 2]          # calculate(Float, Point, Point, ...)

// ❌ Недопустимая привязка (ошибка компиляции)
Point.wrong = distance[5]             # 5 >= 2 (количество параметров)
Point.wrong = distance[0, 0]          # Повторяющаяся позиция (если запрещено)
Point.wrong = distance[-2]            # -2 вне диапазона
```

### 10.5 Автопривязка

Для функций, определённых в модуле и имеющих первым параметром тип модуля, автоматически поддерживается синтаксис вызова методов:

```yaoxiang
// === Point.yx ===
Point: Type = { x: Float, y: Float }

// Первый параметр Point, автоматическая поддержка вызова методов
distance: (a: Point, b: Point) -> Float = { ... }
add: (a: Point, b: Point) -> Point = { ... }

// === main.yx ===
use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

// ✅ Автопривязка: p1.distance(p2) → distance(p1, p2)
d = p1.distance(p2)
// ✅ p1.add(p2) → add(p1, p2)
p3 = p1.add(p2)
```

**Правила автопривязки**：
- Функция определена в файле модуля
- Тип параметра функции с индексом 0 совпадает с именем модуля
- Функция должна быть `pub` для автопривязки за пределами модуля

### 10.6 Связь привязки и карринга

Привязка естественно поддерживает карринг. Когда при вызове предоставлено меньше параметров, чем осталось, возвращается функция, принимающая оставшиеся параметры:

```yaoxiang
// Исходная функция: 5 параметров
calculate: (scale: Float, a: Point, b: Point, x: Float, y: Float) -> Float = ...

// Привязка: Point.calc = calculate[1, 2]
// После привязки остались параметры: scale, x, y

// Сценарии вызова
p1.calc(2.0, 10.0, 20.0)              # Предоставлено 3 параметра → прямой вызов
p1.calc(2.0)                          # Предоставлен 1 параметр → возврат (Float, Float) -> Float
p1.calc()                             # Предоставлено 0 параметров → возврат (Float, Float, Float) -> Float
```
