## 🏛️ 一、Основные определения: модель YaoXiang

**Модель YaoXiang**, заимствованная из «И-цзин. Гуа Фу» — «Все существа действуют совместно, я созерцаю возвращение». Это парадигма параллельного программирования, позволяющая разработчикам описывать логику синхронным, последовательным образом, в то время как среда выполнения языка автоматически и эффективно выполняет вычислительные единицы параллельно, подобно тому как все существа действуют совместно, и в конечном итоге объединяет их.

### Основные принципы проектирования: ленивость по умолчанию + типовая маркировка spawn

| Принцип проектирования | Описание |
|------------------------|----------|
| **Ленивые вычисления по умолчанию** | Все функции по умолчанию ленивы (как в Haskell), возвращают Lazy[T] |
| **Конфигурация количества ядер** | Объявление в заголовке скрипта `// @cores: N` автоматически включает параллелизацию |
| **Типовая маркировка spawn** | `-> T spawn` помечает функцию как строго асинхронную и параллельную, остальные потенциально параллельны |
| **Гибридный режим вычислений** | `@eager` (декоратор, принудительные вычисления), `@auto` (декоратор, сохранение параллелизма) |
| **Void автоматически строгий** | Функции, возвращающие `Void`, автоматически вычисляются строго (побочные эффекты должны выполняться) |

### Три основных принципа

| Основной принцип | Пояснение |
|------------------|-----------|
| **Синхронный синтаксис** | Код соответствует написанному — последовательные инструкции — это то, что выполняется |
| **Параллельная сущность** | Среда выполнения автоматически извлекает параллелизм, выявляя возможности параллелизма в зависимости данных |
| **Единая координация** | Результаты автоматически объединяются при необходимости, обеспечивая логическую корректность |

**Это достигается двумя фундаментальными преобразованиями:**

1. **Преобразование «управления потоком» в «поток данных»**: программа рассматривается как чисто функциональный граф ленивых вычислений
2. **Преобразование «асинхронного заражения» в «разрешение зависимостей»**: асинхронность больше не является эффектом сигнатуры функции, а становится операцией автоматического ожидания в точках зависимости данных во время выполнения

## 📚 二、Терминологическая система: единая карта понятий

Вокруг «YaoXiang» мы построили чёткую, самосогласованную терминологическую систему, связывающую все проектные решения:

| Официальный термин | Соответствующий синтаксис/концепция | Пояснение |
|--------------------|-----------------------------------|-----------|
| **Функция YaoXiang** | `-> T spawn` | Маркировка возвращаемого типа, указывающая, что это вычислительная единица, строго пригодная для параллельного выполнения |
| **Блок YaoXiang** | `spawn { a(), b() }` | Явно объявленная область параллелизма, задачи внутри блока выполняются «совместно» |
| **Цикл YaoXiang** | `spawn for x in xs { ... }` | Парадигма данных-параллелизма, тело цикла выполняется «совместно» для всех элементов данных |
| **Значение YaoXiang** | Тип-посредник `Async[T]` | «Будущее значение», находящееся в процессе выполнения, автоматически ожидает завершения при использовании |
| **Граф YaoXiang** | Ленивый вычислительный граф (DAG) | Сцена, на которой происходит «YaoXiang», описывает зависимости и параллельные отношения между всеми вычислительными единицами |
| **Планировщик YaoXiang** | Планировщик задач среды выполнения | Интеллектуальный центр, ответственный за координацию «всего сущего», чтобы они «действовали совместно» в нужный момент |
| **Граф ошибок** | Error Graph | Визуализация путей распространения ошибок в параллельной среде, аналогична стеку вызовов, но показывает поток ошибок в DAG |
| **Конфликт ресурсов** | Resource Conflict | Конфликт при одновременном доступе нескольких задач к одному записываемому ресурсу, обнаруживается во время компиляции и автоматически сериализуется |

> **Пример технического обсуждения**: «Здесь мы используем блок YaoXiang для параллельного вызова двух функций YaoXiang, что автоматически даст нам их значения YaoXiang.»

## 三、Трёхуровневая архитектура параллелизма: постепенная прозрачность

### 3.1 Обзор архитектуры

Модель YaoXiang обеспечивает **три уровня прогрессивной абстракции параллелизма**, позволяя разработчикам с разным уровнем навыков найти подходящий режим использования:

| Уровень | Режим | Синтаксическая метка | Способ выполнения | Управляемость | Сценарий использования |
|---------|-------|----------------------|-------------------|---------------|------------------------|
| **L1** | Синхронный `@blocking` | `@blocking` | Полностью последовательное выполнение | Наивысшая | Отладка, обучение новичков, критические участки кода |
| **L2** | Явный spawn | `spawn` | Параллелизм под контролем разработчика | Средняя | Продвинутые пользователи, требующие тонкого контроля параллелизма |
| **L3** | Полностью прозрачный | Нет (по умолчанию) | Автоматическая оптимальная параллелизация | Наименьшая | Эксперты, автоматическая оптимизация параллелизма |

### 3.2 L1: Синхронный режим `@blocking`

**Основные характеристики**: отключение всех оптимизаций параллелизма, полностью последовательное выполнение для удобства отладки и понимания.

```yaoxiang
# L1: Синхронный режим @blocking (аннотация после возвращаемого типа)
fetch_sync: (String) -> JSON @blocking = (url) => {
    HTTP.get(url).json()
}

main: () -> Void @blocking = () => {
    # Строго последовательное выполнение, без какого-либо параллелизма
    data1 = fetch_sync("https://api.example.com/data1")
    data2 = fetch_sync("https://api.example.com/data2")
    process(data1, data2)
}
```

### 3.3 L2: Явный параллелизм spawn

**Основные характеристики**: разработчик явно маркирует параллельные единицы, сохраняя управляемость при получении преимуществ параллелизма.

```yaoxiang
# L2: Явный параллелизм spawn
fetch_data: (String) -> JSON spawn = (url) => {
    HTTP.get(url).json()
}

process_users_and_posts: () -> Void spawn = () => {
    users = fetch_data("https://api.example.com/users")
    posts = fetch_data("https://api.example.com/posts")
    # users и posts автоматически выполняются параллельно
    print(users.length.to_string())
    print(posts.length.to_string())
}

# Явный параллельный блок
compute_all: () -> (Int, Int, Int) spawn = () => {
    (a, b, c) = spawn {
        heavy_calc(1),
        heavy_calc(2),
        heavy_calc(3)
    }
    (a, b, c)
}
```

### 3.4 L3: Полностью прозрачный (по умолчанию)

**Основные характеристики**: без каких-либо меток, компилятор автоматически анализирует зависимости и генерирует оптимальный план параллельного выполнения.

```yaoxiang
# L3: Полностью прозрачный (режим по умолчанию)
heavy_calc: (Int) -> Int = (n) => {
    fibonacci(n)
}

auto_parallel: (Int) -> Int = (n) => {
    # Система автоматически анализирует: a, b, c не зависят друг от друга, могут выполняться полностью параллельно
    a = heavy_calc(1)
    b = heavy_calc(2)
    c = heavy_calc(3)
    a + b + c
}
```

### 3.5 Ручные управляющие аннотации

| Аннотация | Поведение | Сценарий использования |
|-----------|-----------|------------------------|
| `@eager` | Принудительные вычисления | Когда необходимо немедленно получить результат |

## 二、Основные концепции

### 2.1 Граф YaoXiang: сцена действия всего сущего

Все программы во время компиляции преобразуются в **направленный ациклический вычислительный граф (DAG)**, который мы называем **графом YaoXiang**.

| Элемент | Описание |
|---------|----------|
| **Узел** | Представляет вычислительную единицу выражения |
| **Ребро** | Представляет отношение зависимости данных (A → B означает, что B зависит от результата A) |
| **Ленивость** | Узел вычисляется только тогда, когда его вывод **действительно необходим** |

### 2.2 Ленивые вычисления по умолчанию

Все функции по умолчанию используют стратегию **ленивых вычислений**:

```yaoxiang
# Заголовок скрипта: настройка количества параллельных ядер
# @cores: 4

# Все функции по умолчанию ленивы (потенциально параллельны)
heavy_computation: (Int) -> Int = (x) => {
    # Эта функция не будет выполнена немедленно
    # Выполняется только при использовании результата
    fibonacci(x)
}

main: () -> Void = () => {
    # heavy_computation возвращает Int, тип — Lazy[Int]
    result = heavy_computation(100)

    # Здесь result используется в сложении, что запускает вычисление
    # Система автоматически находит оптимальный момент для параллельного выполнения
    total = result + heavy_computation(200)
}
```

### 2.3 Гибридные аннотации вычислений (стиль декораторов)

Аннотации в YaoXiang аналогичны декораторам (decorator) Python, используются для изменения поведения функций или выражений:

| Аннотация (декоратор) | Поведение |
|-----------------------|-----------|
| `@eager` | **Декоратор**: принудительные вычисления, немедленное выполнение |
| `@auto` | **Декоратор**: сохранение параллелизма (по умолчанию, можно опустить) |

**Правило автоматической строгости для Void:** Функции, возвращающие `Void`, автоматически вычисляются строго (без каких-либо аннотаций), потому что побочные эффекты должны выполняться.

```yaoxiang
# Декоратор @eager: принудительные вычисления
heavy_computation: (Int) -> Int = (x) => {
    fibonacci(x)
}

# Функции, возвращающие Void, автоматически строгие (функции с побочными эффектами)
log: (String) -> Void = (message) => {
    print(message)
}

main: () -> Void = () => {
    # log автоматически строго выполняется, потому что возвращает Void
    log("Processing started")

    # Использование @eager для принудительных вычислений
    @eager heavy_computation(100)
}
```

### 2.4 Значения YaoXiang: тип-посредник Async[T]

Любая функция, возвращаемый тип которой помечен как `-> T spawn`, немедленно возвращает значение типа `Async[T]`, которое мы называем **значением YaoXiang**.

```yaoxiang
# Функция YaoXiang: возвращаемый тип помечен как -> JSON spawn
# Указывает, что это строго вычислительная единица, пригодная для YaoXiang
fetch: (String) -> JSON spawn = (url) => {
    HTTP.get(url).json()
}

main: () -> Void = () => {
    # fetch возвращает значение YaoXiang — Async[JSON]
    # Но при использовании не требуется никакого дополнительного синтаксиса
    data = fetch("https://api.example.com")  # Async[JSON]

    # Здесь data автоматически ожидает и распаковывается в JSON
    print(data.name)  # Так же естественно, как синхронный код
}
```

#### Основные характеристики значений YaoXiang

| Характеристика | Описание |
|----------------|----------|
| **Прозрачность синтаксиса** | `Async[T]` в системе типов является подтипом `T`, может использоваться в любом контексте, ожидающем `T` |
| **Ожидание по требованию** | Когда необходимо использовать конкретное значение типа `T` (например, доступ к полю, арифметические операции), среда выполнения автоматически приостанавливает выполнение и ожидает |
| **Распространение ошибок** | Внутри фактически представляет `Result<T, E>`, ошибки естественно распространяются вдоль потока данных |

### 2.7 Конструкции YaoXiang: от «модификатора» к «типовая маркировка»

Ключевое слово `spawn` — единственный мост между синхронным мышлением и асинхронной реализацией, имеющий три значения:

| Синтаксическая форма | Официальный термин | Семантика | Поведение среды выполнения |
|:---------------------|:-------------------|:----------|:----------------------------|
| **`-> T spawn`** | Функция YaoXiang | Маркировка возвращаемого типа, указывающая на строго параллельную вычислительную единицу | Её вызов возвращает `Async[T]`, означающий создание узла графа YaoXiang |
| **`spawn { ... }`** | Блок YaoXiang | Явно объявленная область параллелизма | Среда выполнения **агрессивно** выполняет выражения внутри блока как независимые задачи параллельно, неявно ожидая все результаты в конце блока |
| **`spawn for`** | Цикл YaoXiang | Цикл с data-параллелизмом | Преобразует тело цикла в несколько параллельных задач, автоматически выполняет разбиение данных, планирование и сбор результатов |

## 三、Принцип работы: от кода к выполнению

### 3.1 Во время компиляции: построение графа YaoXiang

```yaoxiang
# Определение функции YaoXiang: возвращаемый тип помечен как spawn
fetch: (String) -> String spawn = (url) => { ... }
parse: (String) -> Model spawn = (data) => { ... }

process: () -> Report = () => {
    # Компилятор создаёт узлы графа YaoXiang здесь
    data_a = fetch("url1")  # Узел A: Async[String]
    data_b = fetch("url2")  # Узел B: Async[String]

    # Блок YaoXiang: явная область параллелизма
    (model_a, model_b) = spawn {
        parse(data_a),  # Узел C: зависит от A
        parse(data_b)   # Узел D: зависит от B
    }

    # Финальный узел объединения
    generate_report(model_a, model_b)  # Узел E
}
```

**Операции компилятора:**
1. Парсинг исходного кода, построение глобального графа YaoXiang
2. Создание вычислительного узла для каждого выражения
3. Анализ зависимостей данных, установление рёбер
4. Подграфы внутри блоков `spawn { }` и `spawn for` помечаются флагом **«параллельное вычисление»**

### 4.2 Во время выполнения: планировщик YaoXiang

Интеллектуальный **планировщик YaoXiang**, поддерживающий кражу работы, отвечает за выполнение графа YaoXiang:

```rust
// Основная логика планировщика YaoXiang
impl FlowScheduler {
    fn execute_node(&self, node_id: NodeId) {
        let node = self.get_node(node_id);
        
        match &node.kind {
            NodeKind::AsyncCompute => {
                // Функция YaoXiang: отправка в пул корутин
                self.submit_async(node_id);
            }
            NodeKind::ParallelBlock => {
                // Блок YaoXiang: агрессивное параллельное выполнение всех непосредственных дочерних узлов
                self.submit_parallel(node_id);
            }
            NodeKind::DataParallel { iterator, body } => {
                // Цикл YaoXiang: автоматическое разбиение данных
                self.submit_data_parallel(node_id, iterator, body);
            }
            _ => { /* Синхронное выполнение */ }
        }
    }
}
```

#### Поток выполнения

```
1. Для вычисления [E] нужны [C] и [D]
2. [C] зависит от [A], [D] зависит от [B]
3. Планировщик YaoXiang обнаруживает, что [A] и [B] не зависят друг от друга → немедленное параллельное выполнение
4. После завершения [A] и [B], благодаря метке блока YaoXiang → немедленное параллельное выполнение [C] и [D]
5. После завершения [C] и [D] выполняется [E]
```

**Ключевые механизмы:**

| Механизм | Описание |
|----------|----------|
| **Ленивый запуск** | Выполнение начинается с запроса конечного результата, обратное отслеживание зависимостей |
| **Автоматическое ожидание** | При встрече с `Async[T]` автоматическая приостановка, выполнение других готовых задач |
| **Кража работы** | Потоки крадут задачи из очередей других потоков для повышения использования CPU |

## 四、Подробное описание ключевых механизмов

### 4.1 Побочные эффекты и гарантии вычислений

Чисто ленивые вычисления могут привести к тому, что побочные эффекты (например, логирование, запись) никогда не будут выполнены. Модель YaoXiang использует **автоматический вывод на основе возвращаемого типа**:

| Правило | Условие | Поведение |
|---------|--------|-----------|
| **Правило первое** | Функция возвращает `Void` | **Автоматические строгие вычисления** (побочные эффекты должны выполняться) |
| **Правило второе** | Выражение с декоратором `@eager` | Независимо от возвращаемого типа, **принудительные строгие вычисления** |
| **Правило третье** | Возвращает не-Void тип | **Ленивые вычисления** (по умолчанию) |

```yaoxiang
# Функции, возвращающие Void, автоматически строгие (побочные эффекты)
log: (String) -> Void = (message) => {
    print(message)
}

# Декоратор @eager: принудительные вычисления
cache_compute: (Int) -> Int = (x) => {
    # Даже если возвращает Int, принудительно выполняется немедленно
    expensive_calculation(x)
}

main: () -> Void = () => {
    # log автоматически строго выполняется (возвращает Void)
    log("Processing started")

    # @eager принудительно строго выполняется
    @eager
    cache_compute(100)

    # Обычная функция лениво выполняется (возвращает Int)
    result = heavy_computation(200)  # Пока не выполняется
    print(result)  # Выполняется здесь
}
```

### 4.2 Обработка ошибок

#### Определение типа Result

```yaoxiang
# Стандартный тип Result (единый синтаксис конструктора)
type Result[T, E] = ok(T) | err(E)

# Пользовательский тип ошибки
type ParseError = invalid_format | unexpected_eof | position(Int)

parse_config: (String) -> Result[Config, ParseError] = (content) => {
    if content.is_empty() {
        err(invalid_format)
    } else {
        ok(parse(content))
    }
}
```

#### Синтаксис распространения ошибок

Используется оператор `?` в стиле Rust для прозрачного распространения ошибок:

```yaoxiang
# Оператор ? в стиле Rust
process() -> Result[Data, Error] = {
    data = fetch_data()?      # Автоматическое ожидание и проверка ошибок
    processed = transform(data)?
    save(processed)?          # Ошибки автоматически распространяются вверх
}

# Обработка ошибок через pattern matching
handle_result: (Result[Int, Error]) -> String = (result) => {
    match result {
        ok(value) => "Success: " + value.to_string()
        err(e) => match e {
            network_error => "Network failed"
            parse_error => "Parse failed"
            _ => "Unknown error"
        }
    }
}
```

### 4.3 Чистые функции и синхронные гарантии `@blocking`

**Основная идея: чистые функции не блокируют!**

Потому что:
- Чистые функции не содержат I/O, только CPU-вычисления
- Даже длительные вычисления не блокируют планировщик, только занимают CPU-время

**Стратегия выполнения:**

| Тип функции | Стратегия выполнения | Блокировка? |
|-------------|---------------------|-------------|
| Чистая функция (без I/O) | Синхронное выполнение | Нет (только использование CPU) |
| Асинхронная функция (возвращает `Async[T]`) | Асинхронное выполнение | Нет |
| Функция с аннотацией `@blocking` | Синхронное выполнение, внутреннее планирование | Нет |

**Аннотация `@blocking`: гарантия синхронного выполнения**

Аннотация `@blocking` гарантирует, что функция выполняется синхронно:
- При возврате из функции результат уже готов
- Если внутри есть асинхронные вызовы, они завершаются внутри
- Подходит для сценариев, требующих синхронной семантики, но с внутренними асинхронными операциями

```yaoxiang
# @blocking: синхронное выполнение, внутреннее асинхронное планирование завершается перед возвратом
heavy_compute: (List[Int]) -> Int = (data) => {
    # Внутри могут быть асинхронные операции, но завершаются перед возвратом
    processed = data.map(x => async_transform(x))
    processed.sum()
}

# Обычная асинхронная функция: возвращает Async[T]
fetch_user: (Int) -> Async[User] = (id) => {
    database.query("SELECT * FROM users WHERE id = ?", id)
}

# Чистая функция: автоматически синхронная (без I/O)
factorial: (Int) -> Int = (n) => {
    if n <= 1 then 1 else n * factorial(n - 1)
}

main: () -> Void = () => {
    # Функция @blocking: синхронное выполнение
    result = heavy_compute([1, 2, 3, 4, 5])  # Немедленно возвращает результат
    print(result)  # 15

    # Асинхронная функция: возвращает Async[User]
    user = fetch_user(123)  # Async[User]
    print(user.name)  # Автоматически ожидает и распаковывает
}
```

**Стратегия среды выполнения:**

```rust
fn execute_function(node: &DAGNode) {
    match node.execution_mode {
        ExecutionMode::Pure => {
            // Чистая функция: синхронное выполнение
            node.execute();
        }
        ExecutionMode::Async => {
            // Асинхронная функция: отправка в async-планировщик
            async_runtime.submit(node);
        }
        ExecutionMode::Blocking => {
            // Функция @blocking: синхронное выполнение, внутреннее планирование асинхронных операций
            execute_blocking(node);
        }
    }
}

fn execute_blocking(node: &DAGNode) {
    // Выполнение тела функции
    let result = node.execute_body();
    
    // Сбор всех внутренних асинхронных операций
    let internal_async_ops = collect_async_ops(node);
    
    // Ожидание всех внутренних асинхронных операций
    if !internal_async_ops.is_empty() {
        async_runtime.wait_all(internal_async_ops);
    }
    
    // Возврат результата
    result
}
```

**Преимущества дизайна:**
- **Простота**: не требуется сложная система эффектов
- **Гибкость**: `@blocking` опционален, используется при необходимости синхронной семантики
- **Эффективность**: чистые функции автоматически синхронны
- **Безопасность**: главный планировщик никогда не блокируется

### 4.5 Контроль гонок в параллелизме: гарантии атомарности через систему типов

**Основная идея: использование системы типов для маркировки данных с параллельным доступом, компилятор проверяет корректность синхронизации.**

**Система типовых маркировок:**

| Тип | Семантика | Параллельная безопасность | Описание |
|-----|-----------|--------------------------|----------|
| `T` | Неизменяемые данные | ✅ Безопасно | Тип по умолчанию, чтение несколькими задачами без гонок |
| `Ref[T]` | Изменяемая ссылка | ⚠️ Требуется синхронизация | Маркировка для параллельного изменения, компилятор проверяет использование блокировок |
| `Atomic[T]` | Атомарный тип | ✅ Безопасно | Примитивные атомарные операции, lock-free параллелизм |
| `Mutex[T]` | Обёртка мьютекса | ✅ Безопасно | Автоматическая блокировка и разблокировка, компилятор гарантирует |
| `RwLock[T]` | Обёртка读写锁 | ✅ Безопасно | Оптимизация для сценариев с большим количеством чтения и малым количеством записи |

**Гарантии типобезопасности:**

```yaoxiang
# По умолчанию неизменяемо - естественно без гонок
data: List[Int] = [1, 2, 3, 4, 5]
spawn for x in data { process(x) }  # ✅ Безопасно, только чтение без гонок

# Изменяемая ссылка - требуется синхронизация
counter: Ref[Int] = Ref.new(0)

# Пример ошибки: доступ к Ref без блокировки (ошибка компиляции)
spawn for i in 1..10 {
    # ❌ Ошибка компиляции: доступ к Ref должен быть в защищённой области
    counter.value = counter.value + i
}

# Правильный пример: использование синтаксического сахара with для автоматической блокировки
spawn for i in 1..10 {
    # ✅ Блок with автоматически получает и освобождает блокировку
    with counter.lock() {
        counter.value = counter.value + i
    }
}

# Атомарный тип - lock-free параллелизм
atomic_counter: Atomic[Int] = Atomic.new(0)
spawn for i in 1..10 {
    # ✅ Атомарные операции, без блокировок безопасно
    atomic_counter.fetch_add(i)
}
```

**Тип Mutex[T] - гарантии блокировки во время компиляции:**

```yaoxiang
# Создание данных, обёрнутых в мьютекс
shared_state: Mutex[Map[String, Int]] = Mutex.new(Map.empty())

# Использование синтаксического сахара with (аналогично defer в Go)
main: () -> Void = () => {
    spawn for i in 1..100 {
        # with автоматически получает блокировку, автоматически освобождает после блока
        with shared_state.lock() {
            # Критическая секция: защищена Mutex
            current = shared_state.get("count").or(0)
            shared_state.set("count", current + 1)
        }
    }

    # Ожидание завершения всех задач
    print(shared_state.get("count"))  # 100
}
```

**Вывод типов и проверка блокировок:**

```rust
// Компилятор проверяет во время компиляции
fn compile_check_locks(func: &Function) {
    for node in func.nodes {
        match node {
            NodeKind::ReadRef(ref_var) => {
                // Проверка, находится ли внутри защищённой области блокировки
                if !is_inside_lock_guard(ref_var) {
                    compile_error!("Доступ к Ref должен быть внутри защищённой области lock()");
                }
            }
            NodeKind::WriteRef(ref_var, _) => {
                // Двойная проверка: блокировка + единственный писатель
                if !is_inside_lock_guard(ref_var) {
                    compile_error!("Модификация Ref должна быть внутри защищённой области lock()");
                }
                if has_multiple_writers(func, ref_var) {
                    compile_error!("Mutex[T] может иметь только одного писателя, используйте RwLock[T]");
                }
            }
            _ => {}
        }
    }
}
```

**Преимущества дизайна:**

| Преимущество | Описание |
|--------------|----------|
| **Проверка во время компиляции** | Пропущенные блокировки обнаруживаются во время компиляции, а не приводят к deadlock во время выполнения |
| **Нулевые накладные расходы во время выполнения** | Обёртка Mutex не имеет накладных расходов при отсутствии конкуренции |
| **Простой синтаксис** | Синтаксический сахар `with lock() { ... }` автоматически управляет жизненным циклом |
| **Типобезопасность** | Ошибка использования Ref вместо Atomic приводит к ошибке на уровне типов |
