# **《Спавн: модель бесшовного асинхронного параллелизма на основе ленивых вычислений» — техническая документация**

## 🏛️ I. Основные определения: модель спавна

**Модель спавна**, заимствованная из «И-цзин. Гуа Фу» — «Все существа взаимно действуют, и я созерцаю возвращение». Это парадигма параллельного программирования, позволяющая разработчикам описывать логику синхронным, последовательным образом, в то время как среда выполнения языка автоматически и эффективно выполняет вычислительные единицы во взаимном действии, как все существа во вселенной, и объединяет результаты в итоге.

### Основные принципы проектирования: ленивость по умолчанию + маркер типа spawn

| Принцип проектирования | Описание |
|------------------------|----------|
| **Ленивые вычисления по умолчанию** | Все функции по умолчанию ленивы (как в Haskell), возвращают Lazy[T] |
| **Конфигурация количества ядер** | Объявление `# @cores: N` в заголовке скрипта автоматически включает параллелизацию |
| **Маркер типа spawn** | `-> T spawn` помечает функцию как строго асинхронную и параллельную, остальные потенциально параллельны |
| **Гибридный режим вычислений** | `@eager` (декоратор, принудительные вычисления), `@auto` (декоратор, сохранение параллелизма) |
| **Void автоматически энергичен** | Функции, возвращающие `Void`, автоматически вычисляются энергично (побочные эффекты должны выполняться) |

### Три основных принципа

| Основной принцип | Пояснение |
|-------------------|-----------|
| **Синхронный синтаксис** | Видишь то, что получаешь — последовательный код, написанный так, как он выполняется |
| **Параллельная сущность** | Среда выполнения автоматически извлекает параллелизм, выявляя возможности параллелизма в зависимости данных |
| **Единая координация** | Результаты автоматически объединяются при необходимости, обеспечивая логическую корректность |

**Это достигается двумя фундаментальными преобразованиями:**

1. **Преобразование «потока управления» в «поток данных»**: программа рассматривается как чисто функциональный граф ленивых вычислений
2. **Преобразование «асинхронного заражения» в «разрешение зависимостей»**: асинхронность больше не является эффектом сигнатуры функции, а становится автоматической операцией ожидания, которую среда выполнения выполняет в точках зависимости данных

## 📚 II. Система терминов: единая карта понятий

Вокруг «спавна» мы построили чёткую, самосогласованную систему терминов, объединяющую все проектные решения:

| Официальный термин | Соответствующий синтаксис/концепция | Пояснение |
|---------------------|--------------------------------------|-----------|
| **Спавн-функция** | `-> T spawn` | Маркер типа возврата, указывающий, что это вычислительная единица, строго способная участвовать в параллельном выполнении «спавна» |
| **Спавн-блок** | `spawn { a(), b() }` | Явно объявленная область параллелизма; задачи внутри блока выполняются «во спавне» |
| **Спавн-цикл** | `spawn for x in xs { ... }` | Парадигма данных-параллелизма; тело цикла выполняется «во спавне» для всех элементов данных |
| **Спавн-значение** | Тип-посредник `Async[T]` | «Будущее значение», находящееся в процессе спавна; при использовании автоматически ожидает его «свершения» |
| **Граф спавна** | Ленивый вычислительный граф (DAG) | Сцена, на которой происходит «спавн»; описывает зависимости и параллельные отношения между всеми вычислительными единицами |
| **Планировщик спавна** | Планировщик задач среды выполнения | Интеллектуальный центр, ответственный за координацию «всего сущего», чтобы они «во спавне» в нужный момент |
| **Граф ошибок** | Error Graph | Визуализация путей распространения ошибок в параллельной среде; аналогична стеку вызовов, но показывает поток ошибок в DAG |
| **Конфликт ресурсов** | Resource Conflict | Конфликт при одновременном доступе нескольких задач к одному записываемому ресурсу; обнаруживается на этапе компиляции и автоматически сериализуется |

> **Пример технического обсуждения**: «Здесь мы используем спавн-блок для параллельного вызова двух спавн-функций и автоматически получим их спавн-значения.»

## III. Трёхуровневая архитектура параллелизма: прогрессивная прозрачность

### 3.1 Обзор архитектуры

Модель спавна обеспечивает **три уровня прогрессивных абстракций параллелизма**, позволяя разработчикам любого уровня квалификации найти подходящий режим использования:

| Уровень | Режим | Синтаксический маркер | Способ выполнения | Контролируемость | Сценарий применения |
|---------|-------|----------------------|-------------------|------------------|---------------------|
| **L1** | Синхронный `@blocking` | `@blocking` | Полностью последовательное выполнение | Максимальная | Отладка, обучение новичков, критические участки кода |
| **L2** | Явный spawn | `spawn` | Управляемый разработчиком параллелизм | Средняя | Продвинутые пользователи, тонкий контроль параллелизма |
| **L3** | Полная прозрачность | Нет (по умолчанию) | Автоматическая оптимальная параллелизация | Минимальная | Эксперты, автоматическая оптимизация параллелизма |

### 3.2 L1: синхронный режим `@blocking`

**Основные характеристики**: отключение всех оптимизаций параллелизма, полностью последовательное выполнение для удобства отладки и понимания.

```yaoxiang
# L1: синхронный режим @blocking (аннотация после типа возврата)
fetch_sync: (String) -> JSON @blocking = (url) => {
    HTTP.get(url).json()
}

main: () -> Void @blocking = () => {
    # Строго последовательное выполнение, без параллелизма
    data1 = fetch_sync("https://api.example.com/data1")
    data2 = fetch_sync("https://api.example.com/data2")
    process(data1, data2)
}
```

### 3.3 L2: явный параллелизм spawn

**Основные характеристики**: разработчик явно помечает параллельные единицы, сохраняя контролируемость и одновременно получая выгоду от параллелизма.

```yaoxiang
# L2: явный параллелизм spawn
fetch_data: (String) -> JSON spawn = (url) => {
    HTTP.get(url).json()
}

process_users_and_posts: () -> Void spawn = () => {
    users = fetch_data("https://api.example.com/users")
    posts = fetch_data("https://api.example.com/posts")
    # users и posts автоматически выполняются параллельно
    print(users.length.to_string())
    print(posts.length.to_string())
}

# Явный параллельный блок
compute_all: () -> (Int, Int, Int) spawn = () => {
    (a, b, c) = spawn {
        heavy_calc(1),
        heavy_calc(2),
        heavy_calc(3)
    }
    (a, b, c)
}
```

### 3.4 L3: полная прозрачность (по умолчанию)

**Основные характеристики**: без каких-либо маркеров, компилятор автоматически анализирует зависимости и генерирует оптимальный план параллельного выполнения.

```yaoxiang
# L3: полная прозрачность (режим по умолчанию)
heavy_calc: (Int) -> Int = (n) => {
    fibonacci(n)
}

auto_parallel: (Int) -> Int = (n) => {
    # Система автоматически анализирует: a, b, c не зависят друг от друга, возможен полный параллелизм
    a = heavy_calc(1)
    b = heavy_calc(2)
    c = heavy_calc(3)
    a + b + c
}
```

### 3.5 Ручные управляющие аннотации

| Аннотация | Поведение | Сценарий использования |
|-----------|-----------|------------------------|
| `@eager` | Принудительные энергичные вычисления | Когда нужно немедленно получить результат |

## IV. Основные концепции

### 4.1 Граф спавна: сцена взаимного действия всего сущего

Все программы при компиляции преобразуются в **направленный ациклический вычислительный граф (DAG)**, который мы называем **графом спавна**.

| Элемент | Описание |
|---------|----------|
| **Узел** | Представляет вычислительную единицу выражения |
| **Ребро** | Представляет отношение зависимости данных (A → B означает, что B зависит от результата A) |
| **Ленивость** | Узел вычисляется только тогда, когда его выходные данные **реально необходимы** |

### 4.2 Ленивые вычисления по умолчанию

Все функции по умолчанию используют стратегию **ленивых вычислений**:

```yaoxiang
# Заголовок скрипта: настройка количества параллельных ядер
# @cores: 4

# Все функции по умолчанию с ленивыми вычислениями (потенциально параллельны)
heavy_computation: (Int) -> Int = (x) => {
    # Эта функция не выполняется немедленно
    # Выполняется только при использовании результата
    fibonacci(x)
}

main: () -> Void = () => {
    # heavy_computation возвращает Int, тип — Lazy[Int]
    result = heavy_computation(100)

    # Здесь result используется в сложении, что запускает вычисления
    # Система автоматически находит оптимальный момент для параллельного выполнения
    total = result + heavy_computation(200)
}
```

### 4.3 Гибридные вычислительные аннотации (стиль декораторов)

Аннотации в YaoXiang аналогичны декораторам Python (decorator), используемым для изменения поведения функций или выражений:

| Аннотация (декоратор) | Поведение |
|-----------------------|-----------|
| `@eager` | **Декоратор**: принудительные энергичные вычисления, немедленное выполнение |
| `@auto` | **Декоратор**: сохранение параллелизма (по умолчанию, можно опустить) |

**Правило автоматической энергичности для Void:** Функции, возвращающие `Void`, автоматически вычисляются энергично (без каких-либо аннотаций), потому что побочные эффекты должны выполняться.

```yaoxiang
# Декоратор @eager: принудительные энергичные вычисления
heavy_computation: (Int) -> Int = (x) => {
    fibonacci(x)
}

# Функция, возвращающая Void, автоматически вычисляется энергично (побочные эффекты)
log: (String) -> Void = (message) => {
    print(message)
}

main: () -> Void = () => {
    # log автоматически выполняется энергично, потому что возвращает Void
    log("Processing started")

    # Использование @eager для принудительных вычислений
    @eager heavy_computation(100)
}
```

### 4.4 Спавн-значение: ленивый тип-посредник Async[T]

Любая функция с типом возврата, помеченным как `-> T spawn`, немедленно возвращает значение типа `Async[T]`, которое мы называем **спавн-значением**.

```yaoxiang
# Спавн-функция: тип возврата помечен как spawn
# Указывает, что это вычислительная единица, строго способная к параллельному выполнению
fetch: (String) -> JSON spawn = (url) => {
    HTTP.get(url).json()
}

main: () -> Void = () => {
    # fetch возвращает спавн-значение Async[JSON]
    # Но при использовании не требуется никакого дополнительного синтаксиса
    data = fetch("https://api.example.com")  # Async[JSON]

    # Здесь data автоматически ожидает и распаковывается в JSON
    print(data.name)  # Так же естественно, как синхронный код
}
```

#### Основные характеристики спавн-значений

| Характеристика | Описание |
|----------------|---------|
| **Синтаксическая прозрачность** | `Async[T]` является подтипом `T` в системе типов и может использоваться в любом контексте, где ожидается `T` |
| **Ожидание по требованию** | Когда необходимо использовать конкретное значение типа `T` (например, доступ к полю, арифметические операции), среда выполнения автоматически приостанавливает выполнение и ожидает |
| **Распространение ошибок** | Внутри представляет собой `Result<T, E>`, ошибки естественно распространяются вдоль потока данных |

### 4.5 Спавн-конструкции: от «модификатора» к «маркеру типа»

Ключевое слово `spawn` — единственный мост, соединяющий синхронное мышление с асинхронной реализацией, обладающий тройной семантикой:

| Синтаксическая форма | Официальный термин | Семантика | Поведение среды выполнения |
|:---------------------|:-------------------|:----------|:---------------------------|
| **`-> T spawn`** | Спавн-функция | Маркер типа возврата, указывающий, что это вычислительная единица, строго способная участвовать в спавне | Её вызов возвращает `Async[T]`, сигнализируя о создании узла графа спавна |
| **`spawn { ... }`** | Спавн-блок | Явно объявленная область параллелизма | Среда выполнения **агрессивно** выполняет выражения внутри блока как независимые задачи и **неявно ожидает** все результаты в конце блока |
| **`spawn for`** | Спавн-цикл | Цикл сdata-параллелизмом | Преобразует тело цикла в несколько параллельных задач с автоматическим разбиением данных, планированием и сбором результатов |

## V. Принцип работы: от кода к выполнению

### 5.1 Время компиляции: построение графа спавна

```yaoxiang
# Определение спавн-функции: тип возврата помечен как spawn
fetch: (String) -> String spawn = (url) => { ... }
parse: (String) -> Model spawn = (data) => { ... }

process: () -> Report = () => {
    # Компилятор здесь создаёт узлы графа спавна
    data_a = fetch("url1")  # Узел A: Async[String]
    data_b = fetch("url2")  # Узел B: Async[String]

    # Спавн-блок: явная область параллелизма
    (model_a, model_b) = spawn {
        parse(data_a),  # Узел C: зависит от A
        parse(data_b)   # Узел D: зависит от B
    }

    # Финальный узел слияния
    generate_report(model_a, model_b)  # Узел E
}
```

**Операции компилятора:**
1. Парсинг исходного кода, построение глобального графа спавна
2. Создание вычислительных узлов для каждого выражения
3. Анализ зависимостей данных, установление рёбер
4. Подграфы внутри блоков `spawn { }` и `spawn for` помечаются флагом **«параллельное вычисление»**

### 5.2 Время выполнения: планировщик спавна

Интеллектуальный **планировщик спавна**, поддерживающий кражу работы (work-stealing), отвечает за выполнение графа спавна:

```rust
// Основная логика планировщика спавна
impl FlowScheduler {
    fn execute_node(&self, node_id: NodeId) {
        let node = self.get_node(node_id);
        
        match &node.kind {
            NodeKind::AsyncCompute => {
                // Спавн-функция: отправка в пул корутин
                self.submit_async(node_id);
            }
            NodeKind::ParallelBlock => {
                // Спавн-блок: агрессивное параллельное выполнение всех прямых дочерних узлов
                self.submit_parallel(node_id);
            }
            NodeKind::DataParallel { iterator, body } => {
                // Спавн-цикл: автоматическое разбиение данных
                self.submit_data_parallel(node_id, iterator, body);
            }
            _ => { /* Синхронное выполнение */ }
        }
    }
}
```

#### Поток выполнения

```
1. Для вычисления [E] нужны [C] и [D]
2. [C] зависит от [A], [D] зависит от [B]
3. Планировщик спавна обнаруживает, что [A] и [B] не зависят друг от друга → немедленное параллельное выполнение
4. После завершения [A], [B], из-за пометки спавн-блока → немедленное параллельное выполнение [C] и [D]
5. После завершения [C], [D] выполняется [E]
```

**Ключевые механизмы:**

| Механизм | Описание |
|----------|----------|
| **Ленивый запуск** | Выполнение начинается с запроса финального результата и идёт в обратном направлении по зависимостям |
| **Автоматическое ожидание** | При встрече с `Async[T]` автоматическая приостановка и выполнение других готовых задач |
| **Кража работы** | Потоки крадут задачи из очередей других потоков для повышения загрузки CPU |

## VI. Подробное описание ключевых механизмов

### 6.1 Побочные эффекты и гарантии вычислений

Чистые ленивые вычисления могут привести к тому, что побочные эффекты (такие как логирование, запись) никогда не будут выполнены. Модель спавна использует **автоматический вывод на основе типа возврата**:

| Правило | Условие | Поведение |
|---------|---------|-----------|
| **Правило 1** | Функция возвращает `Void` | **Автоматические энергичные вычисления** (побочные эффекты должны выполняться) |
| **Правило 2** | Выражение с декоратором `@eager` | Независимо от типа возврата — **принудительные энергичные вычисления** |
| **Правило 3** | Возвращает тип, отличный от Void | **Ленивые вычисления** (по умолчанию) |

```yaoxiang
# Функция, возвращающая Void, автоматически выполняется энергично (побочные эффекты)
log: (String) -> Void = (message) => {
    print(message)
}

# Декоратор @eager: принудительные энергичные вычисления
cache_compute: (Int) -> Int = (x) => {
    # Даже если возвращает Int, принудительно немедленное выполнение
    expensive_calculation(x)
}

main: () -> Void = () => {
    # log автоматически выполняется энергично (возвращает Void)
    log("Processing started")

    # @eager принудительные энергичные вычисления
    @eager
    cache_compute(100)

    # Обычная функция с ленивыми вычислениями (возвращает Int)
    result = heavy_computation(200)  # Пока не выполняется
    print(result)  # Выполняется только здесь
}
```

### 6.2 Обработка ошибок

#### Определение типа Result

```yaoxiang
# Стандартный тип Result (унифицированный конструктор)
type Result[T, E] = ok(T) | err(E)

# Пользовательский тип ошибок
type ParseError = invalid_format | unexpected_eof | position(Int)

parse_config: (String) -> Result[Config, ParseError] = (content) => {
    if content.is_empty() {
        err(invalid_format)
    } else {
        ok(parse(content))
    }
}
```

#### Синтаксис распространения ошибок

Используется оператор `?` в стиле Rust для прозрачного распространения ошибок:

```yaoxiang
# Оператор ? в стиле Rust
```

### 6.3 Чистые функции и гарантии синхронности `@blocking`

**Основная идея: чистые функции не блокируют!**

Потому что:
- Чистые функции не имеют I/O, только CPU-вычисления
- Даже длительные вычисления не блокируют планировщик, а только занимают CPU-время

**Стратегия выполнения:**

| Тип функции | Стратегия выполнения | Блокировка? |
|--------------|---------------------|-------------|
| Чистая функция (без I/O) | Синхронное выполнение | Нет (только использование CPU) |
| Асинхронная функция (возвращает `Async[T]`) | Асинхронное выполнение | Нет |
| Функция с аннотацией `@blocking` | Синхронное выполнение, внутреннее планирование | Нет |

**Аннотация `@blocking`: гарантия синхронного выполнения**

Аннотация `@blocking` гарантирует, что функция выполняется в синхронном режиме:
- При возврате функции результат уже готов
- Если есть асинхронные вызовы внутри, они завершаются перед возвратом
- Подходит для сценариев, требующих синхронной семантики, но с возможными асинхронными операциями внутри

```yaoxiang
# @blocking: синхронное выполнение, асинхронные операции внутри завершаются перед возвратом
heavy_compute: (List[Int]) -> Int = (data) => {
    # Внутри могут быть асинхронные операции, но завершаются перед возвратом
    processed = data.map(x => async_transform(x))
    processed.sum()
}

# Обычная асинхронная функция: возвращает Async[T]
fetch_user: (Int) -> Async[User] = (id) => {
    database.query("SELECT * FROM users WHERE id = ?", id)
}

# Чистая функция: автоматически синхронная (без I/O)
factorial: (Int) -> Int = (n) => {
    if n <= 1 then 1 else n * factorial(n - 1)
}

main: () -> Void = () => {
    # Функция @blocking: синхронное выполнение
    result = heavy_compute([1, 2, 3, 4, 5])  # Немедленное получение результата
    print(result)  # 15

    # Асинхронная функция: возвращает Async[User]
    user = fetch_user(123)  # Async[User]
    print(user.name)  # Автоматическое ожидание и распаковка
}
```

**Стратегия времени выполнения:**

```rust
fn execute_function(node: &DAGNode) {
    match node.execution_mode {
        ExecutionMode::Pure => {
            // Чистая функция: синхронное выполнение
            node.execute();
        }
        ExecutionMode::Async => {
            // Асинхронная функция: отправка в асинхронный планировщик
            async_runtime.submit(node);
        }
        ExecutionMode::Blocking => {
            // Функция @blocking: синхронное выполнение с внутренним планированием асинхронных операций
            execute_blocking(node);
        }
    }
}

fn execute_blocking(node: &DAGNode) {
    // Выполнение тела функции
    let result = node.execute_body();
    
    // Сбор всех внутренних асинхронных операций
    let internal_async_ops = collect_async_ops(node);
    
    // Ожидание завершения всех внутренних асинхронных операций
    if !internal_async_ops.is_empty() {
        async_runtime.wait_all(internal_async_ops);
    }
    
    // Возврат результата
    result
}
```

**Преимущества дизайна:**
- **Простота**: не требуется сложная система эффектов
- **Гибкость**: `@blocking` опциональна, используется при необходимости синхронной семантики
- **Эффективность**: чистые функции автоматически синхронны
- **Безопасность**: главный планировщик никогда не блокируется

### 6.5 Управление параллельной конкуренцией: гарантия атомарности через систему типов

**Основная идея: использование системы типов для маркировки данных с параллельным доступом, компилятор проверяет корректность синхронизации.**

**Система маркеров типов:**

| Тип | Семантика | Параллельная безопасность | Описание |
|-----|-----------|---------------------------|----------|
| `T` | Неизменяемые данные | ✅ Безопасно | Тип по умолчанию, несколько задач читают без конкуренции |
| `Ref[T]` | Изменяемая ссылка | ⚠️ Требуется синхронизация | Маркировано для параллельной модификации, компилятор проверяет использование блокировок |
| `Atomic[T]` | Атомарный тип | ✅ Безопасно | Нижнеуровневые атомарные операции, безблокировочный параллелизм |
| `Mutex[T]` | Обёртка мьютекса | ✅ Безопасно | Автоматическая блокировка и разблокировка, компилятор гарантирует |
| `RwLock[T]` | Обёртка блокировки чтения-записи | ✅ Безопасно | Оптимизация для сценариев с частым чтением и редкой записью |

**Гарантии безопасности типов:**

```yaoxiang
# По умолчанию неизменяемо - естественно без конкуренции
data: List[Int] = [1, 2, 3, 4, 5]
spawn for x in data { process(x) }  # ✅ Безопасно, только чтение без конкуренции

# Изменяемая ссылка - требуется синхронизация
counter: Ref[Int] = Ref.new(0)

# Ошибочный пример: доступ к Ref без блокировки (ошибка компиляции)
spawn for i in 1..10 {
    # ❌ Ошибка компиляции: доступ к Ref должен быть защищён синхронизацией
    counter.value = counter.value + i
}

# Правильный пример: использование синтаксического сахара with для автоматической блокировки
spawn for i in 1..10 {
    # ✅ блок with автоматически получает и освобождает блокировку
    with counter.lock() {
        counter.value = counter.value + i
    }
}

# Атомарный тип - безблокировочный параллелизм
atomic_counter: Atomic[Int] = Atomic.new(0)
spawn for i in 1..10 {
    # ✅ Атомарные операции, безблокировочно безопасно
    atomic_counter.fetch_add(i)
}
```

**Тип Mutex[T] - гарантия блокировки на этапе компиляции:**

```yaoxiang
# Создание данных, обёрнутых в мьютекс
shared_state: Mutex[Map[String, Int]] = Mutex.new(Map.empty())

# Использование синтаксического сахара with (аналогично defer в Go)
main: () -> Void = () => {
    spawn for i in 1..100 {
        # with автоматически получает блокировку, автоматически освобождает после блока
        with shared_state.lock() {
            # Критическая секция: защищена Mutex
            current = shared_state.get("count").or(0)
            shared_state.set("count", current + 1)
        }
    }

    # Ожидание завершения всех задач
    print(shared_state.get("count"))  # 100
}
```

**Вывод типов и проверка блокировок:**

```rust
// Компилятор проверяет на этапе компиляции
fn compile_check_locks(func: &Function) {
    for node in func.nodes {
        match node {
            NodeKind::ReadRef(ref_var) => {
                // Проверка: находится ли внутри защищённой блокировкой области
                if !is_inside_lock_guard(ref_var) {
                    compile_error!("Доступ к Ref должен быть в области, защищённой lock()");
                }
            }
            NodeKind::WriteRef(ref_var, _) => {
                // Двойная проверка: блокировка + единственный писатель
                if !is_inside_lock_guard(ref_var) {
                    compile_error!("Модификация Ref должна быть в области, защищённой lock()");
                }
                if has_multiple_writers(func, ref_var) {
                    compile_error!("Mutex[T] может иметь только одного писателя; используйте RwLock[T]");
                }
            }
            _ => {}
        }
    }
}
```

**Преимущества дизайна:**

| Преимущество | Описание |
|--------------|----------|
| **Проверка на этапе компиляции** | Пропущенные блокировки обнаруживаются на этапе компиляции, а не приводят к взаимоблокировке во время выполнения |
| **Нулевые накладные расходы во время выполнения** | Обёртка Mutex не имеет накладных расходов при отсутствии конкуренции |
| **Простой синтаксис** | Синтаксический сахар `with lock() { ... }`, автоматическое управление жизненным циклом |
| **Типовая безопасность** | Неправильное использование Ref вместо Atomic приводит к ошибке на уровне типов |
