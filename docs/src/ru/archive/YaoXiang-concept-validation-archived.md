## 1. Обзор языка

### 1.1 Цели дизайна

YaoXiang（爻象）— это экспериментальный универсальный язык программирования, объединяющий следующие характеристики:

- **Типы — это всё**：Значения, функции, модули, обобщения — всё является типами, типы — граждане первого класса
- **Математическая абстракция**：Единая абстрактная среда на основе теории типов
- **Абстракции с нулевой стоимостью**：Высокая производительность, без сборщика мусора, модель владения обеспечивает безопасность памяти
- **Естественный синтаксис**：Читаемость как у Python, приближенная к естественному языку
- **Незаметный асинхрон**：Нет необходимости в явном await, компилятор обрабатывает автоматически
- **Удобство для ИИ**：Строгая структурированность, чёткий AST, лёгкость анализа и модификации

### 1.2 Основная философия дизайна

```
┌─────────────────────────────────────────────────────────────┐
│               Философия дизайна YaoXiang                    │
├─────────────────────────────────────────────────────────────┤
│  Всё — типы → Единая абстракция → Типы как данные → Доступны в runtime│
│                                                              │
│  Модель владения → Абстракции с нулевой стоимостью → Без GC → Высокая производительность│
│                                                              │
│  Синтаксис Python → Ощущение естественного языка → Читаемость → Дружелюбность для новичков│
│                                                              │
│  Автоматический вывод → Минимум ключевых слов → Краткость → Удобство для ИИ│
└─────────────────────────────────────────────────────────────┘
```

### 1.3 Позиционирование языка

| Измерение | Позиционирование |
|-----------|------------------|
| Парадигма | Мультипарадигмальный（функциональный + императивный + объектно-ориентированный）|
| Система типов | Зависимые типы + параметрический полиморфизм |
| Управление памятью | Владение + RAII（без GC）|
| Модель компиляции | AOT-компиляция（опционально JIT）|
| Целевые сценарии | Системное программирование, разработка приложений, ИИ-ассистируемое программирование |

## 2. Проверка основных концепций

### 2.1 Проверка осуществимости «Всё — типы»

#### Теоретическое обоснование

В теории типов типы можно рассматривать как высказывания, а значения — как доказательства. Этот изоморфизм Карри — Ховарда раскрывает глубокую связь между типами и значениями. YaoXiang распространяет эту идею до предела:

```
Значения — это экземпляры типов
Типы — это экземпляры типов (метатипы)
Функции — это отображения между типами
Модули — это композиции типов
Обобщения — это фабрики типов
```

#### Пример проверки

```yaoxiang
# Значения — это экземпляры типов
x: Int = 42
# x — экземпляр типа Int

# Типы — это экземпляры типов
MyList: type = List(Int)
# MyList — экземпляр type (метатипа)

# Функции — это отображения между типами
add(Int, Int) -> Int = (a, b) => a + b
# add — экземпляр типа (Int, Int) -> Int

# Модули — это композиции типов (использование файлов как модулей)
# Math.yx
pi: Float = 3.14159
sqrt(Float) -> Float = (x) => { ... }
# Модуль Math — это тип пространства имён
```

#### Вывод проверки

✅ **Осуществимо** — «Всё — типы» имеет прочную математическую основу (теория типов, теория категорий) и может быть реализовано на практике через единое представление типов.

### 2.2 Гарантия высокой производительности зависимых типов

#### Вызов

Языки с зависимыми типами (как Agda, Idris) обычно имеют низкую производительность из-за:

1. Сложных проверок типов
2. Представления типов во время выполнения
3. Полноты проверки сопоставления с образцом

#### Решение YaoXiang

```yaoxiang
# Стирание типов во время компиляции (опционально)
# Информация о типах во время выполнения загружается по требованию

# Гарантия абстракций с нулевой стоимостью
identity<T>(T) -> T = (x) => x
# Компилируется в прямой возврат, без дополнительных накладных расходов

# Оптимизация на уровне типов
type Nat = { n: Int }
# Компилируется в обычный целочисленный тип, без дополнительной обёртки
```

#### Механизмы гарантии производительности

| Механизм | Описание |
|----------|----------|
| Мономорфизация | Обобщённые функции раскрываются во время компиляции в конкретные версии |
| Встраивание | Простые функции автоматически встраиваются |
| Распределение в стеке | Малые объекты по умолчанию распределяются в стеке |
| Анализ побега | Большие объекты распределяются в куче |
| Условное стирание типов | Опциональная информация о типах во время выполнения |

#### Вывод проверки

✅ **Осуществимо** — Благодаря тщательно спроектированной стратегии компиляции можно сохранить возможности зависимых типов и одновременно достичь высокой производительности.

### 2.3 Проверка осуществимости незаметного асинхрона

#### Основная идея

```yaoxiang
# Модель автоматического await
# При вызове функций компилятор автоматически определяет асинхронные зависимости
# и вставляет соответствующие барьеры синхронизации

fetch_user: (Int) -> User spawn = (id) => {
    return database.query("SELECT * FROM users WHERE id = ?", id)
}

display_user: (Int) -> String = (id) => {
    user = fetch_user(id)  # Автоматическое ожидание результата
    return "User: " + user.name   # Гарантия готовности user
}
```

#### Процесс автоматической обработки компилятором

```
Исходный код
   ↓
Проверка типов + Анализ асинхронных зависимостей
   ↓
Идентификация spawn-вызовов
   ↓
Генерация конечного автомата
   ↓
Автоматическая вставка await-точек
   ↓
Оптимизация барьеров синхронизации
   ↓
Целевой код
```

#### Вывод проверки

✅ **Осуществимо** — Подобно корутинам Kotlin и async/await Rust, но с автоматическим управлением через анализ времени компиляции, что снижает нагрузку на программиста.

## 3. Дизайн системы типов

### 3.1 Иерархия типов

```
┌─────────────────────────────────────────────────────────────┐
│               Иерархия типов YaoXiang                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  type (метатип)                                              │
│    │                                                        │
│    ├── Примитивные типы (Primitive Types)                   │
│    │   ├── Void                                             │
│    │   ├── Bool                                             │
│    │   ├── Int (8/16/32/64/128)                            │
│    │   ├── Uint (8/16/32/64/128)                           │
│    │   ├── Float (32/64)                                   │
│    │   ├── Char, String                                    │
│    │   └── Bytes                                           │
│    │                                                        │
│    ├── Составные типы (Composite Types)                     │
│    │   ├── struct { fields }                               │
│    │   ├── union { variants }                              │
│    │   ├── enum { variants }                               │
│    │   ├── tuple (T1, T2, ...)                             │
│    │   ├── list [T], dict [K->V]                           │
│    │   └── option [T]                                      │
│    │                                                        │
│    ├── Функциональные типы (Function Types)                 │
│    │   fn (T1, T2, ...) -> R                               │
│    │                                                        │
│    ├── Обобщённые типы (Generic Types)                      │
│    │   List[T], Map[K, V], etc.                            │
│    │                                                        │
│    ├── Зависимые типы (Dependent Types)                     │
│    │   type { n: Nat } -> type                             │
│    │   Vec[n: Nat, T]                                      │
│    │                                                        │
│    └── Типы модулей (Module Types)                          │
│        mod { exports }                                      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 Информация о типах во время выполнения

```yaoxiang
# Рефлексия типов
fn describe(t: type) -> String {
    match t {
        struct { fields } -> "Struct with " + fields.length + " fields"
        union { variants } -> "Union with " + variants.length + " variants"
        enum { variants } -> "Enum with " + variants.length + " cases"
        list { element } -> "List of " + element.name
        fn { params, ret } -> "Function: (" + params + ") -> " + ret
        primitive { name } -> "Primitive: " + name
    }
}

# Проверка типа
fn is_number(t: type) -> Bool {
    t == Int or t == Float or t == Number
}

# Проверка экземпляра типа
value: type = ...
if value has_type Int {
    print("It's an integer")
}

# Преобразование типов
fn safe_cast[T, U](value: T, target: type) -> option[U] {
    if value has_type target {
        some(value as U)
    } else {
        none
    }
}
```

## 4. Владение и модель памяти

### 4.1 Принципы владения

```yaoxiang
# Ссылка по умолчанию неизменяемая
process(ref Data) -> Void = (data) => {
    # data доступен только для чтения
    # Нельзя изменять поля data
    # Нельзя передавать владение data
}

# Изменяемая ссылка
modify(mut Data) -> Void = (data) => {
    # Можно изменять поля data
    # Не должно быть других активных ссылок
}

# Передача владения
consume(Data) -> Void = (data) => {
    # Владение data передаётся внутрь
    # После завершения функции data уничтожается
}

# Возврат заимствования
borrow_field(ref Data) -> ref Field = (data) => ref data.field
```

### 4.2 Время жизни

```yaoxiang
# Явные аннотации времени жизни (сложные случаи)
longest<'a>(&'a str, &'a str) -> &'a str = (s1, s2) => {
    if s1.length > s2.length { s1 } else { s2 }
}

# Автоматический вывод времени жизни
first<T>(ref List[T]) -> ref T = (list) => ref list[0]
```

### 4.3 Умные указатели

```yaoxiang
# Box — распределение в куче
heap_data: Box[List[Int]] = Box.new([1, 2, 3])

# Rc — подсчёт ссылок
shared: Rc[Data] = Rc.new(data)

# Arc — атомарный подсчёт ссылок (потокобезопасный)
thread_safe: Arc[Data] = Arc.new(data)

# RefCell — внутренняя изменяемость
internal_mut: RefCell[Data] = RefCell.new(data)
```

### 4.4 Гарантии безопасности памяти

```yaoxiang
# Проверки во время компиляции
unsafe_example() -> Void = () => {
    data: Data = ...
    ref1 = ref data
```

## 6. Дизайн синтаксиса

### 6.1 Ключевые слова (17)

YaoXiang определяет 17 ключевых слов. Эти ключевые слова зарезервированы и не могут использоваться как идентификаторы.

| # | Ключевое слово | Назначение | Пример |
|---|----------------|------------|--------|
| 1 | `type` | Определение типа | `type Point = { x: Int, y: Int }` |
| 2 | `pub` | Публичный экспорт | `pub add(Int, Int) -> Int = ...` |
| 3 | `use` | Импорт модуля | `use std.io` |
| 4 | `spawn` | Пометка асинхронности | `fetch(String) -> T spawn = ...` |
| 5 | `ref` | Неизменяемая ссылка | `process(ref Data) -> Void = ...` |
| 6 | `mut` | Изменяемая ссылка | `modify(mut Data) -> Void = ...` |
| 7 | `if` | Условное ветвление | `if x > 0 { ... }` |
| 8 | `elif` | Множественные условия | `elif x == 0 { ... }` |
| 9 | `else` | Ветвь по умолчанию | `else { ... }` |
| 10 | `match` | Сопоставление с образцом | `match x { 0 -> "zero" }` |
| 11 | `while` | Цикл с условием | `while i < 10 { ... }` |
| 12 | `for` | Итерационный цикл | `for item in items { ... }` |
| 13 | `return` | Возврат значения | `return result` |
| 14 | `break` | Выход из цикла | `break` |
| 15 | `continue` | Продолжение цикла | `continue` |
| 16 | `as` | Преобразование типа | `x as Float` |
| 17 | `in` | Проверка принадлежности/списковое включение | `x in [1, 2, 3]`, `[x * 2 for x in list]` |

**Альтернатива бесконечному циклу:**

```yaoxiang
# Использование while True вместо ключевого слова loop
while True {
    input = read_line()
    if input == "quit" {
        break
    }
    process(input)
}
```

### 6.2 Зарезервированные слова

Зарезервированные слова — это предопределённые специальные значения языка, которые нельзя использовать как идентификаторы, но они не являются ключевыми словами (не используются в синтаксических конструкциях).

| Зарезервированное слово | Тип | Описание |
|------------------------|-----|----------|
| `true` | Bool | Булево истина |
| `false` | Bool | Булево ложь |
| `null` | Void | Пустое значение |
| `none` | Option | Вариант без значения типа Option |
| `some(T)` | Option | Вариант со значением типа Option (функция) |
| `ok(T)` | Result | Вариант успеха типа Result (функция) |
| `err(E)` | Result | Вариант ошибки типа Result (функция) |

```yaoxiang
# Булевы значения
flag = true
flag = false

# Использование типа Option
maybe_value: option[String] = none
maybe_value = some("hello")

# Использование типа Result
result: result[Int, String] = ok(42)
result = err("error message")
```

### 6.3 Переменные и присваивание

```yaoxiang
# Автоматический вывод типов
x = 42                    # Int
name = "YaoXiang"         # String
pi = 3.14159              # Float
is_valid = true           # Bool

# Явные аннотации типов (опционально)
```

## 10. Риски и вызовы

### 10.1 Технические риски

| Риск | Вероятность | Влияние | Меры снижения |
|------|-------------|---------|---------------|
| Слишком долгая компиляция из-за зависимых типов | Средняя | Высокое | Инкрементная компиляция, кэширование |
| Семантика автоматического await слишком сложная | Средняя | Среднее | Постепенная реализация |
| Кривая обучения модели владения | Низкая | Среднее | Дружественные подсказки компилятора |
| Слишком сложная система типов | Средняя | Высокое | Приоритет упрощённого подмножества |

### 10.2 Вызовы реализации

```yaoxiang
# Вызов 1: Полнота вывода типов
# Требуется реализация расширенной системы вывода типов Хиндли-Милнера

# Вызов 2: Проверка зависимых типов
# Требуется реализация решающих алгоритмов из теории типов

# Вызов 3: Корректность автоматического await
# Требуется гарантия правильной идентификации всех зависимостей

# Вызов 4: Проверка владения
# Требуется реализация borrow checker, аналогичного Rust
```

### 10.3 Риски дизайна языка

- **Риск**: Слишком мощная система типов может привести к слишком долгой компиляции
- **Смягчение**: Предоставить выбор режима проверки типов
- **Риск**: Ограничения синтаксиса могут повлиять на гибкость
- **Смягчение**: Сохранять ядро простым, расширения опциональными
