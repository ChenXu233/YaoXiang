## 1. Обзор языка

### 1.1 Цели проектирования

YaoXiang（爻象）— экспериментальный универсальный язык программирования, объединяющий следующие характеристики:

- **Типы — это всё**: значения, функции, модули, generics — всё является типами, типы — граждане первого класса
- **Математическая абстракция**: единая абстрактная среда на основе теории типов
- **Абстракции нулевой стоимости**: высокая производительность, без сборщика мусора, модель владения обеспечивает безопасность памяти
- **Естественный синтаксис**: читаемость на уровне Python, близость к естественному языку
- **Незаметный асинхронность**: без явного await, компилятор обрабатывает автоматически
- **Удобство для ИИ**: строгая структурированность, чёткий AST, лёгкость парсинга и модификации

### 1.2 Основная философия дизайна

```
┌─────────────────────────────────────────────────────────────┐
│                 Философия дизайна YaoXiang                 │
├─────────────────────────────────────────────────────────────┤
│  Всё является типом → Единая абстракция → Тип как данные → Доступно в runtime    │
│                                                              │
│  Модель владения → Абстракции нулевой стоимости → Без GC → Высокая производительность              │
│                                                              │
│  Синтаксис Python → Ощущение естественного языка → Читаемость → Удобство для новичков            │
│                                                              │
│  Автоматический вывод → Минимум ключевых слов → Краткость → Удобство для ИИ            │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 Позиционирование языка

| Измерение | Позиционирование |
|------|------|
| Парадигма | Мультипарадигмальный (функциональный + императивный + объектно-ориентированный) |
| Типовая система | Зависимые типы + параметрический полиморфизм |
| Управление памятью | Владение + RAII (без GC) |

## 2. Проверка основных концепций

### 2.1 Проверка осуществимости принципа «всё является типом»

#### Теоретическое обоснование

В теории типов типы можно рассматривать как предложения, а значения — как доказательства. Этот изоморфизм Карри — Ховарда раскрывает глубокую связь между типами и значениями. YaoXiang распространяет эту идею до предела:

```
Значения являются экземплярами типов
Типы являются экземплярами типов (метатипы)
Функции являются отображениями между типами
Модули являются комбинациями типов
Generics являются фабриками типов
```

#### Примеры проверки

```yaoxiang
# Значения являются экземплярами типов
x: Int = 42
# x является экземпляром типа Int

# Типы являются экземплярами типов
MyList: type = List(Int)
# MyList является экземпляром типа type (метатип)

# Функции являются отображениями между типами
add(Int, Int) -> Int = (a, b) => a + b
# add является экземпляром типа (Int, Int) -> Int

# Модули являются комбинациями типов (модуль = файл)
# Math.yx
pi: Float = 3.14159
sqrt(Float) -> Float = (x) => { ... }
# Модуль Math — это тип пространства имён
```

#### Вывод проверки

✅ **Осуществимо** — принцип «всё является типом» имеет прочную математическую основу (теория типов, теория категорий) и может быть реализован на практике через единое представление типов.

### 2.2 Гарантия высокой производительности при зависимых типах

#### Вызовы

Языки с зависимыми типами (такие как Agda, Idris) обычно имеют низкую производительность из-за:

1. Сложных проверок типов
2. Представления типов в runtime
3. Полноты проверки сопоставления с образцом

#### Решение YaoXiang

```yaoxiang
# Стирание типов при компиляции (опционально)
# Информация о типах в runtime загружается по требованию

# Гарантия абстракций нулевой стоимости
identity<T>(T) -> T = (x) => x
# Компилируется в прямой возврат, без дополнительных накладных расходов

# Оптимизация на уровне типов
type Nat = { n: Int }
# Компилируется в обычный целочисленный тип, без дополнительной упаковки
```

#### Механизмы гарантии производительности

| Механизм | Описание |
|------|------|
| Мономорфизация | Generic-функции при компиляции разворачиваются в конкретные версии |
| Оптимизация встраивания | Простые функции автоматически встраиваются |
| Распределение в стеке | Малые объекты по умолчанию размещаются в стеке |
| Анализ побега | Большие объекты только в куче |
| Условное стирание типов | Опциональная информация о типах в runtime |

#### Вывод проверки

✅ **Осуществимо** — через тщательно спроектированную стратегию компиляции можно сохранить возможности зависимых типов и при этом обеспечить высокую производительность.

### 2.3 Проверка осуществимости незаметной асинхронности

#### Основная идея

```yaoxiang
# Модель автоматического await
# При вызове функции компилятор автоматически определяет асинхронные зависимости
# и вставляет соответствующие барьеры синхронизации

fetch_user(Int) -> User spawn = (id) => {
    database.query("SELECT * FROM users WHERE id = ?", id)
}

display_user(Int) -> String = (id) => {
    user = fetch_user(id)  # Автоматическое ожидание результата
    "User: " + user.name   # Гарантия готовности user
}
```

#### Процесс автоматической обработки компилятором

```
Исходный код
   ↓
Проверка типов + Анализ асинхронных зависимостей
   ↓
Определение вызовов spawn
   ↓
Генерация конечного автомата
   ↓
Автоматическая вставка await-точек
   ↓
Оптимизация барьеров синхронизации
   ↓
Целевой код
```

#### Вывод проверки

✅ **Осуществимо** — подобно корутинам Kotlin и async/await в Rust, но с автоматическим управлением на этапе компиляции, что снижает нагрузку на программиста.

## 4. Владение и модель памяти

### 4.1 Принципы владения

```yaoxiang
# По умолчанию — неизменяемая ссылка
process(ref Data) -> Void = (data) => {
    # data доступен только для чтения
    # Нельзя изменять поля data
    # Нельзя передавать владение data
}

# Изменяемая ссылка
modify(mut Data) -> Void = (data) => {
    # Можно изменять поля data
    # Не может быть других активных ссылок
}

# Передача владения
consume(Data) -> Void = (data) => {
    # Владение data передаётся в функцию
    # После окончания функции data уничтожается
}

# Возврат заимствования
borrow_field(ref Data) -> ref Field = (data) => ref data.field
```

## 6. Дизайн синтаксиса

### 6.1 Ключевые слова (17)

В YaoXiang определено 17 ключевых слов, которые зарезервированы и не могут использоваться как идентификаторы.

| # | Ключевое слово | Назначение | Пример |
|---|--------|------|------|
| 1 | `type` | Определение типа | `type Point = { x: Int, y: Int }` |
| 2 | `pub` | Публичный экспорт | `pub add(Int, Int) -> Int = ...` |
| 3 | `use` | Импорт модуля | `use std.io` |
| 4 | `spawn` | Метка асинхронности | `fetch(String) -> T spawn = ...` |
| 5 | `ref` | Неизменяемая ссылка | `process(ref Data) -> Void = ...` |
| 6 | `mut` | Изменяемая ссылка | `modify(mut Data) -> Void = ...` |
| 7 | `if` | Условное ветвление | `if x > 0 { ... }` |
| 8 | `elif` | Множественные условия | `elif x == 0 { ... }` |
| 9 | `else` | Блок по умолчанию | `else { ... }` |
| 10 | `match` | Сопоставление с образцом | `match x { 0 -> "zero" }` |
| 11 | `while` | Цикл с условием | `while i < 10 { ... }` |
| 12 | `for` | Итерационный цикл | `for item in items { ... }` |
| 13 | `return` | Возврат значения | `return result` |
| 14 | `break` | Выход из цикла | `break` |
| 15 | `continue` | Продолжение цикла | `continue` |
| 16 | `as` | Преобразование типа | `x as Float` |
| 17 | `in` | Проверка вхождения/генератор списка | `x in [1, 2, 3]`, `[x * 2 for x in list]` |

**Альтернатива бесконечному циклу:**

```yaoxiang

    input = read_line()

        break

### 6.2 Зарезервированные слова

Зарезервированные слова — это предопределённые языком специальные значения, которые нельзя использовать как идентификаторы, но они не являются ключевыми словами (не используются в синтаксических конструкциях).

| Зарезервированное слово | Тип | Описание |
|--------|------|------|
| `true` | Bool | Булево истина |
| `false` | Bool | Булево ложь |
| `null` | Void | Пустое значение |
| `none` | Option | Вариант без значения типа Option |
| `some(T)` | Option | Вариант со значением типа Option (функция) |
| `ok(T)` | Result | Вариант успеха типа Result (функция) |
| `err(E)` | Result | Вариант ошибки типа Result (функция) |

```yaoxiang
# Булевы значения
flag = true
flag = false

# Использование типа Option
maybe_value: option[String] = none
maybe_value = some("hello")

# Использование типа Result
result: result[Int, String] = ok(42)
result = err("error message")
```

## 10. Риски и вызовы

### 10.1 Технические риски

| Риск | Вероятность | Влияние | Меры снижения |
|------|--------|------|----------|
| Слишком долгая компиляция из-за зависимых типов | Средняя | Высокое | Инкрементальная компиляция, кэширование |
| Сложная семантика автоматического await | Средняя | Среднее | Поэтапная реализация |
| Кривая обучения модели владения | Низкая | Среднее | Дружелюбные подсказки компилятора |
| Слишком сложная система типов | Средняя | Высокое | Приоритет упрощённого подмножества |

### 10.2 Вызовы реализации

### 10.3 Риски дизайна языка

- **Риск**: Слишком мощная система типов может привести к увеличению времени компиляции
- **Смягчение**: Предоставить выбор режима проверки типов
- **Риск**: Ограничения синтаксиса могут повлиять на гибкость
- **Смягчение**: Сохранить ядро простым, расширения опциональны
