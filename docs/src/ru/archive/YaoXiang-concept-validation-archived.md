## 1. Обзор языка

### 1.1 Цели дизайна

YaoXiang（爻象）— экспериментальный универсальный язык программирования, объединяющий следующие характеристики:

- **Типы — это всё**: значения, функции, модули, generics — всё является типами, типы — граждане первого класса
- **Математическая абстракция**: единая абстрактная структура на основе теории типов
- **Абстракции с нулевой стоимостью**: высокая производительность, без GC, модель ownership обеспечивает безопасность памяти
- **Естественный синтаксис**: читаемость на уровне Python, близость к естественному языку
- **Незаметный асинхрон**: без явного await, компилятор обрабатывает автоматически
- **Удобство для ИИ**: строгая структурированность, чистый AST, лёгкость парсинга и модификации

### 1.2 Ключевая философия дизайна

```
┌─────────────────────────────────────────────────────────────┐
│                 Философия дизайна YaoXiang                 │
├─────────────────────────────────────────────────────────────┤
│  Всё — типы → Единая абстракция → Типы как данные →       │
│  Доступны в runtime                                        │
│                                                              │
│  Модель ownership → Абстракции с нулевой стоимостью →     │
│  Без GC → Высокая производительность                        │
│                                                              │
│  Синтаксис Python → Ощущение естественного языка →         │
│  Читаемость → Дружелюбие для новичков                      │
│                                                              │
│  Автоматический вывод → Минимум ключевых слов →            │
│  Краткость выражения → Удобство для ИИ                     │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 Позиционирование языка

| Измерение | Позиционирование |
|------|------|
| Парадигма | Мультипарадигма（функциональная + императивная + объектно-ориентированная）|

## 2. Проверка ключевых концепций

### 2.1 Проверка осуществимости «Всё — типы»

#### Теоретическое обоснование

В теории типов типы можно рассматривать как высказывания, а значения — как доказательства. Этот изоморфизм Curry-Howard раскрывает глубокую связь между типами и значениями. YaoXiang распространяет эту идею до предела:

```
Значения — это экземпляры типов
Типы — это экземпляры типов（метатипы）
Функции — это отображения входных типов в выходные
Модули — это комбинации типов
Generics — это фабрики типов
```

#### Примеры проверки

```yaoxiang
# Значения — экземпляры типов
x: Int = 42
# x — экземпляр типа Int

# Типы — экземпляры типов
MyList: type = List(Int)
# MyList — экземпляр type（метатипа）

# Функции — отображения между типами
add(Int, Int) -> Int = (a, b) => a + b
# add — экземпляр типа (Int, Int) -> Int

# Модули — комбинации типов（использование файлов как модулей）
# Math.yx
pi: Float = 3.14159
sqrt(Float) -> Float = (x) => { ... }
# Модуль Math — это тип пространства имён
```

#### Вывод проверки

✅ **Осуществимо** — «Всё — типы» имеет прочную математическую базу（теория типов, теория категорий）, на практике может быть реализовано через унифицированное представление типов.

### 2.2 Гарантии высокой производительности зависимых типов

#### Вызовы

Языки с зависимыми типами（如Agda、Idris）обычно имеют низкую производительность, потому что:

1. Сложная проверка типов
2. Представление типов в runtime
3. Полная проверка исчерпываемости pattern matching

#### Решение YaoXiang

```yaoxiang
# Опциональное стирание типов на этапе компиляции
# Информация о типах в runtime загружается по требованию

# Гарантия абстракций с нулевой стоимостью
identity<T>(T) -> T = (x) => x
# Компилируется в прямой возврат, без дополнительных накладных расходов

# Оптимизация на уровне типов
type Nat = { n: Int }
# Компилируется в обычный целочисленный тип, без дополнительной обёртки
```

#### Механизмы гарантии производительности

| Механизм | Описание |
|------|------|
| Специализация | Generic-функции раскрываются в конкретные версии на этапе компиляции |
| Встраивание | Простые функции автоматически встраиваются |
| Распределение в стеке | Малые объекты по умолчанию размещаются в стеке |
| Анализ побега | Большие объекты только в куче |
| Условное стирание типов | Опциональная информация о типах в runtime |

#### Вывод проверки

✅ **Осуществимо** — через тщательно спроектированную стратегию компиляции можно сохранить возможности зависимых типов и при этом обеспечить высокую производительность.

### 2.3 Проверка осуществимости незаметного асинхрона

#### Ключевая идея

```yaoxiang
# Модель автоматического await
# При вызове функций компилятор автоматически обнаруживает асинхронные зависимости
# и вставляет соответствующие барьеры синхронизации

fetch_user(Int) -> User spawn = (id) => {
    database.query("SELECT * FROM users WHERE id = ?", id)
}

display_user(Int) -> String = (id) => {
    user = fetch_user(id)  # Автоматическое ожидание результата
    "User: " + user.name   # Гарантия готовности user
}
```

#### Процесс автоматической обработки компилятором

```
Исходный код
   ↓
Проверка типов + Анализ асинхронных зависимостей
   ↓
Идентификация spawn-вызовов
   ↓
Генерация конечного автомата
   ↓
Автоматическая вставка await-точек
   ↓
Оптимизация барьеров синхронизации
   ↓
Целевой код
```

#### Вывод проверки

✅ **Осуществимо** — похоже на корутины Kotlin и async/await в Rust, но через анализ на этапе компиляции автоматическое управление, уменьшающее нагрузку на программиста.

## 4. Ownership и модель памяти

### 4.1 Принципы ownership

```yaoxiang
# По умолчанию неизменяемая ссылка
process(ref Data) -> Void = (data) => {
    # data только для чтения
    # Нельзя изменять поля data
    # Нельзя передавать ownership data
}

# Изменяемая ссылка
modify(mut Data) -> Void = (data) => {
    # Можно изменять поля data
    # Не должно быть других активных ссылок
}

# Передача ownership
consume(Data) -> Void = (data) => {
    # Ownership data передаётся внутрь
    # После завершения функции data уничтожается
}

# Возврат по ссылке
borrow_field(ref Data) -> ref Field = (data) => ref data.field
```

### 4.2 Время жизни

```yaoxiang
# Явные аннотации времени жизни（сложные случаи）
longest<'a>(&'a str, &'a str) -> &'a str = (s1, s2) => {
    if s1.length > s2.length { s1 } else { s2 }
}

# Автоматический вывод времени жизни
first<T>(ref List[T]) -> ref T = (list) => ref list[0]
```

### 4.3 Умные указатели

```yaoxiang
# Box — распределение в куче
heap_data: Box[List[Int]] = Box.new([1, 2, 3])

# Rc — подсчёт ссылок
shared: Rc[Data] = Rc.new(data)

# Arc — атомарный подсчёт ссылок（потокобезопасный）
thread_safe: Arc[Data] = Arc.new(data)

# RefCell — внутренняя изменяемость
internal_mut: RefCell[Data] = RefCell.new(data)
```

### 4.4 Гарантии безопасности памяти

```yaoxiang
# Проверки на этапе компиляции
unsafe_example() -> Void = () => {
    data: Data = ...
    ref1 = ref data
    ref2 = ref data  # Ошибка компиляции! Несколько активных ссылок

    mut_data = mut data
    ref_mut = ref mut_data
    mut_data2 = mut mut_data  # Ошибка компиляции! Изменяемая и неизменяемая ссылка одновременно
}
```

## 6. Дизайн синтаксиса

### 6.1 Ключевые слова（17）

YaoXiang определяет 17 ключевых слов, которые являются зарезервированными и не могут использоваться как идентификаторы.

| # | Ключевое слово | Назначение | Пример |
|---|--------|------|------|
| 1 | `type` | Определение типа | `type Point = { x: Int, y: Int }` |
| 2 | `pub` | Публичный экспорт | `pub add(Int, Int) -> Int = ...` |
| 3 | `use` | Импорт модуля | `use std.io` |
| 4 | `spawn` | Маркер асинхрона | `fetch(String) -> T spawn = ...` |
| 5 | `ref` | Неизменяемая ссылка | `process(ref Data) -> Void = ...` |
| 6 | `mut` | Изменяемая ссылка | `modify(mut Data) -> Void = ...` |
| 7 | `if` | Условное ветвление | `if x > 0 { ... }` |
| 8 | `elif` | Множественные условия | `elif x == 0 { ... }` |
| 9 | `else` | Ветвь по умолчанию | `else { ... }` |
| 10 | `match` | Pattern matching | `match x { 0 -> "zero" }` |
| 11 | `while` | Цикл с условием | `while i < 10 { ... }` |
| 12 | `for` | Цикл итерации | `for item in items { ... }` |
| 13 | `return` | Возврат значения | `return result` |
| 14 | `break` | Выход из цикла | `break` |
| 15 | `continue` | Продолжение цикла | `continue` |
| 16 | `as` | Преобразование типа | `x as Float` |
| 17 | `in` | Проверка вхождения/списковое включение | `x in [1, 2, 3]`, `[x * 2 for x in list]` |

**Альтернатива бесконечному циклу:**

```yaoxiang
# Использование while True вместо ключевого слова loop
while True {
    input = read_line()
    if input == "quit" {
        break
    }
    process(input)
}
```

### 6.2 Зарезервированные слова

Зарезервированные слова — это предопределённые языком специальные значения, которые нельзя использовать как идентификаторы, но они не являются ключевыми словами（не используются в синтаксических конструкциях）。

| Зарезервированное | Тип | Описание |
|--------|------|------|
| `true` | Bool | Логическое значение истина |
| `false` | Bool | Логическое значение ложь |
| `null` | Void | Пустое значение |
| `none` | Option | Вариант без значения типа Option |
| `some(T)` | Option | Вариант со значением типа Option（функция）|
| `ok(T)` | Result | Вариант успеха типа Result（функция）|
| `err(E)` | Result | Вариант ошибки типа Result（функция）|

```yaoxiang
# Логические значения
flag = true
flag = false

# Использование типа Option
maybe_value: option[String] = none
maybe_value = some("hello")

# Использование типа Result
result: result[Int, String] = ok(42)
result = err("error message")
```

### 6.3 Переменные и присваивание


## 10. Риски и вызовы

### 10.1 Технические риски

| Риск | Вероятность | Влияние | Меры снижения |
|------|--------|------|----------|
| Слишком долгая компиляция из-за зависимых типов | Средняя | Высокое | Инкрементная компиляция, кэширование |
| Сложная семантика автоматического await | Средняя | Среднее | Поэтапная реализация |
| Крутая кривая обучения модели ownership | Низкая | Среднее | Дружелюбные подсказки компилятора |
| Слишком сложная система типов | Средняя | Высокое | Приоритет упрощённого подмножества |

### 10.2 Вызовы реализации

```yaoxiang
# Вызов 1: Полнота вывода типов
# Требуется реализация расширенной системы типов Hindley-Milner

# Вызов 2: Проверка зависимых типов
# Требуется реализация решающих алгоритмов из теории типов

# Вызов 3: Корректность автоматического await
# Требуется гарантия правильной идентификации всех зависимостей

# Вызов 4: Проверка ownership
# Требуется реализация borrow checker похожего на Rust
```

### 10.3 Риски дизайна языка

- **Риск**: Слишком мощная система типов может привести к слишком долгой компиляции
- **Снижение**: Предоставить выбор режима проверки типов
- **Риск**: Ограничения синтаксиса могут повлиять на гибкость
- **Снижение**: Сохранить ядро простым, расширения опциональны
