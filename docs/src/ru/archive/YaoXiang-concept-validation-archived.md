## 1. Обзор языка

### 1.1 Цели проектирования

YaoXiang (爻象) — экспериментальный универсальный язык программирования, целью которого является объединение следующих возможностей:

- **Тип — это всё**: значения, функции, модули, дженерики — всё это типы; типы являются гражданами первого класса
- **Математические абстракции**: единая абстрактная основа, основанная на теории типов
- **Абстракции с нулевой стоимостью**: высокая производительность, отсутствие GC, безопасность памяти гарантируется моделью владения
- **Естественный синтаксис**: читаемость на уровне Python, близость к естественному языку
- **Прозрачная асинхронность**: явный `await` не требуется, компилятор обрабатывает всё автоматически
- **ИИ-дружественность**: строгая структурированность, ясное AST, лёгкость разбора и модификации

### 1.2 Основная философия проектирования

```
┌─────────────────────────────────────────────────────────────┐
│                    Философия проектирования YaoXiang         │
├─────────────────────────────────────────────────────────────┤
│  Всё есть тип → Единая абстракция → Тип — это данные → Доступно в runtime
│                                                              │
│  Модель владения → Абстракции с нулевой стоимостью → Без GC → Высокая производительность
│                                                              │
│  Синтаксис Python → Ощущение естественного языка → Читаемость → Дружелюбность к новичкам
│                                                              │
│  Автоматический вывод → Минимум ключевых слов → Лаконичное выражение → ИИ-дружественность
└─────────────────────────────────────────────────────────────┘
```

### 1.3 Позиционирование языка

| Измерение | Позиционирование |
|------|------|
| Парадигма | Мультипарадигменный (функциональный + императивный + объектно-ориентированный) |
| Система типов | Зависимые типы + параметрический полиморфизм |
| Управление памятью | Владение + RAII (без GC) |
| Модель компиляции | AOT-компиляция (опционально JIT) |
| Целевые сценарии | Системное программирование, разработка приложений, ИИ-ассистированное программирование |

## 2. Проверка основных концепций

### 2.1 Осуществимость принципа «Всё есть тип»

#### Теоретическое обоснование

В теории типов типы можно рассматривать как утверждения, а значения — как доказательства. Это изоморфизм Карри-Ховарда, раскрывающий глубокую связь между типами и значениями. YaoXiang доводит эту идею до предела:

```
Значение — это экземпляр типа
Тип — это экземпляр типа (метатип)
Функция — это отображение между входным и выходным типами
Модуль — это композиция типов
Дженерики — это фабрика типов
```

#### Проверочный пример

```yaoxiang
# Значение — это экземпляр типа
x: Int = 42
# x — экземпляр типа Int

# Тип — это экземпляр типа
MyList: type = List(Int)
# MyList — экземпляр type (метатипа)

# Функция — это отображение между типами
add(Int, Int) -> Int = (a, b) => a + b
# add — экземпляр типа (Int, Int) -> Int

# Модуль — это композиция типов (файл как модуль)
# Math.yx
pi: Float = 3.14159
sqrt(Float) -> Float = (x) => { ... }
# Модуль Math — это тип пространства имён
```

#### Вывод проверки

✅ **Осуществимо** — принцип «всё есть тип» имеет прочную математическую базу (теория типов, теория категорий); на практике он может быть реализован через унифицированное представление типов.

### 2.2 Высокопроизводительные гарантии зависимых типов

#### Вызов

Языки с зависимыми типами (такие как Agda, Idris) обычно обладают низкой производительностью, поскольку:

1. Сложная проверка типов
2. Представление типов в runtime
3. Проверка полноты сопоставления с образцом

#### Решение YaoXiang

```yaoxiang
# Стирание типов во время компиляции (опционально)
# Информация о типах в runtime загружается по необходимости

# Гарантия абстракций с нулевой стоимостью
identity<T>(T) -> T = (x) => x
# Компилируется в прямой возврат, без дополнительных накладных расходов

# Оптимизации на уровне типов
type Nat = { n: Int }
# Компилируется в обычный целочисленный тип, без дополнительной обёртки
```

#### Механизмы гарантии производительности

| Механизм | Описание |
|------|------|
| Мономорфизация | Обобщённые функции во время компиляции раскрываются в конкретные версии |
| Встраивание | Простые функции автоматически встраиваются |
| Стековое размещение | Маленькие объекты по умолчанию размещаются в стеке |
| Escape-анализ | Большие объекты размещаются в куче |
| Условное стирание типов | Опциональная информация о типах в runtime |

#### Вывод проверки

✅ **Осуществимо** — посредством тщательно спроектированной стратегии компиляции возможно достичь высокой производительности при сохранении возможностей зависимых типов.

### 2.3 Осуществимость прозрачной асинхронности

#### Основная идея

```yaoxiang
# Модель автоматического await
# При вызове функции компилятор автоматически обнаруживает асинхронные зависимости
# и вставляет соответствующие барьеры синхронизации

fetch_user: (Int) -> User spawn = (id) => {
    return database.query("SELECT * FROM users WHERE id = ?", id)
}

display_user: (Int) -> String = (id) => {
    user = fetch_user(id)  # Автоматическое ожидание результата
    return "User: " + user.name   # Гарантируется, что user уже готов
}
```

#### Автоматическая обработка компилятором

```
Исходный код
   ↓
Проверка типов + анализ асинхронных зависимостей
   ↓
Идентификация вызовов spawn
   ↓
Генерация конечного автомата
   ↓
Автоматическая вставка точек await
   ↓
Оптимизация барьеров синхронизации
   ↓
Целевой код
```

#### Вывод проверки

✅ **Осуществимо** — аналогично корутинам Kotlin и async/await в Rust, но управление осуществляется посредством анализа на этапе компиляции, что снижает нагрузку на программиста.

### 3.4 Информация о типах в runtime

```yaoxiang
# Рефлексия типов
fn describe(t: type) -> String {
    match t {
        struct { fields } -> "Struct with " + fields.length + " fields"
        union { variants } -> "Union with " + variants.length + " variants"
        enum { variants } -> "Enum with " + variants.length + " cases"
        list { element } -> "List of " + element.name
        fn { params, ret } -> "Function: (" + params + ") -> " + ret
        primitive { name } -> "Primitive: " + name
    }
}

# Проверка типа
fn is_number(t: type) -> Bool {
    t == Int or t == Float or t == Number
}

# Проверка принадлежности экземпляра к типу
value: type = ...
if value has_type Int {
    print("It's an integer")
}

# Преобразование типа
fn safe_cast[T, U](value: T, target: type) -> option[U] {
    if value has_type target {
        some(value as U)
    } else {
        none
    }
}
```

## 4. Модель владения и памяти

### 4.1 Принципы владения

```yaoxiang
# Неизменяемая ссылка по умолчанию
process(ref Data) -> Void = (data) => {
    # data — только для чтения
    # нельзя изменять поля data
    # нельзя передавать владение data
}

# Изменяемая ссылка
modify(mut Data) -> Void = (data) => {
    # можно изменять поля data
    # не должно быть других активных ссылок
}

# Передача владения
consume(Data) -> Void = (data) => {
    # владение data передаётся внутрь
    # по завершении функции data уничтожается
}

# Возврат заимствования
borrow_field(ref Data) -> ref Field = (data) => ref data.field
```

### 4.2 Время жизни

```yaoxiang
# Явные аннотации времени жизни (в сложных случаях)
longest<'a>(&'a str, &'a str) -> &'a str = (s1, s2) => {
    if s1.length > s2.length { s1 } else { s2 }
}

# Автоматический вывод времени жизни
first<T>(ref List[T]) -> ref T = (list) => ref list[0]
```

### 4.3 Умные указатели

```yaoxiang
# Box — размещение в куче
heap_data: Box[List[Int]] = Box.new([1, 2, 3])

# Rc — подсчёт ссылок
shared: Rc[Data] = Rc.new(data)

# Arc — атомарный подсчёт ссылок (потокобезопасный)
thread_safe: Arc[Data] = Arc.new(data)

# RefCell — внутренняя изменяемость
internal_mut: RefCell[Data] = RefCell.new(data)
```

### 4.4 Гарантии безопасности памяти

```yaoxiang
# Проверки на этапе компиляции
unsafe_example() -> Void = () => {
    data: Data = ...
    ref1 = ref data
    ref2 = ref data  # Ошибка компиляции! Несколько активных ссылок

    mut_data = mut data
    ref_mut = ref mut_data
    mut_data2 = mut mut_data  # Ошибка компиляции! Сосуществование изменяемой и неизменяемой ссылок
}
```

## 5. Прозрачный механизм асинхронности

### 5.1 Пометка функций как spawn

```yaoxiang
# Пометка асинхронной функции с помощью spawn
fetch_api: (String) -> JSON spawn = (url) => {
    response = HTTP.get(url)
    return JSON.parse(response.body)
}

calculate_heavy: (Int) -> Int spawn = (n) => {
    # Долгое вычисление
    mut result = 0
    for i in 0..n {
        result += i
    }
    return result
}
```

### 5.2 Автоматическое ожидание

```yaoxiang
# Код, вызывающий spawn-функции, ожидает их автоматически
main() -> Void = () => {
    # fetch_api — асинхронная, но при вызове ожидается автоматически
    data = fetch_api("https://api.example.com/data")
    # к этому моменту data уже готова

    # Можно продолжить использовать data
    print(data.value)

    # Несколько асинхронных вызовов могут выполняться параллельно
    users = fetch_api("https://api.example.com/users")

    # Автоматическое ожидание при присваивании
    # users и posts могут выполняться параллельно
    print(users.length + posts.length)
}
```

## 6. Проектирование синтаксиса

### 6.1 Ключевые слова (17 штук)

YaoXiang определяет всего 17 ключевых слов. Эти ключевые слова зарезервированы и не могут использоваться в качестве идентификаторов.

| # | Ключевое слово | Назначение | Пример |
|---|--------|------|------|
| 1 | `type` | Определение типа | `type Point = { x: Int, y: Int }` |
| 2 | `pub` | Публичный экспорт | `pub add(Int, Int) -> Int = ...` |
| 3 | `use` | Импорт модуля | `use std.io` |
| 4 | `spawn` | Пометка асинхронности | `fetch(String) -> T spawn = ...` |
| 5 | `ref` | Неизменяемая ссылка | `process(ref Data) -> Void = ...` |
| 6 | `mut` | Изменяемая ссылка | `modify(mut Data) -> Void = ...` |
| 7 | `if` | Ветвление условия | `if x > 0 { ... }` |
| 8 | `elif` | Множественное условие | `elif x == 0 { ... }` |
| 9 | `else` | Ветвь по умолчанию | `else { ... }` |
| 10 | `match` | Сопоставление с образцом | `match x { 0 -> "zero" }` |
| 11 | `while` | Цикл по условию | `while i < 10 { ... }` |
| 12 | `for` | Итерационный цикл | `for item in items { ... }` |
| 13 | `return` | Возврат значения | `return result` |
| 14 | `break` | Выход из цикла | `break` |
| 15 | `continue` | Продолжение цикла | `continue` |
| 16 | `as` | Преобразование типа | `x as Float` |
| 17 | `in` | Проверка принадлежности / list comprehension | `x in [1, 2, 3]`, `[x * 2 for x in list]` |

**Альтернатива для бесконечного цикла:**

```yaoxiang
# Использование while True вместо ключевого слова loop
while True {
    input = read_line()
    if input == "quit" {
        break
    }
    process(input)
}
```

### 6.2 Зарезервированные слова

Зарезервированные слова — это предопределённые специальные значения языка, которые не могут использоваться как идентификаторы, но они не являются ключевыми словами (не используются в синтаксических конструкциях).

| Зарезервированное слово | Тип | Описание |
|--------|------|------|
| `true` | Bool | Логическое значение «истина» |
| `false` | Bool | Логическое значение «ложь» |
| `null` | void | Пустое значение |
| `none` | Option | Вариант отсутствия значения типа Option |
| `some(T)` | Option | Вариант-значение типа Option (функция) |
| `ok(T)` | Result | Успешный вариант типа Result (функция) |
| `err(E)` | Result | Вариант ошибки типа Result (функция) |

```yaoxiang
# Логические значения
flag = true
flag = false

# Использование типа Option
maybe_value: option[String] = none
maybe_value = some("hello")

# Использование типа Result
result: result[Int, String] = ok(42)
result = err("error message")
```

### 6.3 Переменные и присваивание

```yaoxiang
# Автоматический вывод типов
x = 42                    # Int
name = "YaoXiang"         # String
pi = 3.14159              # Float
is_valid = true           # Bool

# Явная аннотация типа (опционально)
```

## 7. Проектирование с учётом ИИ-дружественности

### 7.1 Принципы проектирования

```yaoxiang
# Цели ИИ-дружественного проектирования:
# 1. Строгая структурированность, однозначный синтаксис
# 2. Ясное AST, лёгкость позиционирования
# 3. Явная семантика, отсутствие скрытого поведения
# 4. Чёткие границы блоков кода
# 5. Полная информация о типах
```

### 7.2 Строгие правила отступов

```yaoxiang
# Должны использоваться 4 пробела
# Табуляция запрещена

# Корректный пример
example() -> Void = () => {
    if condition {
        do_something()
    } else {
        do_other()
    }
}

# Ошибочный пример (запрещено)
example() -> Void = () => {
if condition {
do_something()  # Недостаточный отступ
  }               # Несогласованный отступ
}
```

### 7.3 Явные границы блоков кода

### 7.4 Однозначный синтаксис


## 10. Риски и вызовы

### 10.1 Технические риски

| Риск | Вероятность | Влияние | Меры по снижению |
|------|--------|------|----------|
| Слишком долгая компиляция с зависимыми типами | Средняя | Высокое | Инкрементальная компиляция, кэширование |
| Сложная семантика автоматического await | Средняя | Среднее | Поэтапная реализация |
| Кривая обучения модели владения | Низкая | Среднее | Дружелюбные подсказки компилятора |
| Чрезмерная сложность системы типов | Средняя | Высокое | Приоритет упрощённого подмножества |

### 10.2 Проблемы реализации

```yaoxiang
# Проблема 1: Полнота вывода типов
# Требуется реализация расширения системы типов Хиндли-Милнера

# Требуется реализация алгоритма разрешения для теории типов

# Проблема 3: Корректность автоматического await
# Необходимо обеспечить правильное распознавание всех зависимостей

# Проблема 4: Проверка владения
# Требуется реализация проверки заимствований, аналогичной Rust
```

### 10.3 Риски проектирования языка

- **Риск**: Чрезмерно мощная система типов может привести к слишком долгой компиляции
- **Снижение**: Предоставить выбор режима проверки типов
- **Риск**: Ограничения синтаксиса могут повлиять на гибкость
- **Снижение**: Сохранять простоту ядра, опциональные расширения
