## 一、Обзор языка

### 1.1 Что такое YaoXiang?

YaoXiang (YaoXiang) — экспериментальный универсальный язык программирования, чья концепция дизайна берёт начало в ключевых понятиях «Yao» (爻) и «Xiang» (象) из «И цзин» («Книги перемен»). «Yao» — это базовые символы, составляющие триграммы и гексаграммы, символизирующие изменения инь и ян; «Xiang» — внешнее проявление сущности вещей, представляющее все явления мира.

YaoXiang интегрирует эту философскую мысль в систему типов языка программирования, выдвигая основополагающую концепцию **«Всё есть тип»**. В мировоззрении YaoXiang:

- **Значение** — это экземпляр типа
- **Тип** сам по себе является экземпляром типа (meta type)
- **Функция** — это отображение из входного типа в выходной
- **Модуль** — это комбинация пространств имён типов

### 1.2 Цели дизайна

Цели дизайна YaoXiang можно обобщить следующим образом:

| Цель | Описание |
|------|----------|
| **Единообразная абстракция типов** | Тип — это абстракция высшего уровня, упрощающая семантику языка |
| **Естественный опыт программирования** | Синтаксис в стиле Python, акцент на читаемости |
| **Безопасное управление памятью** | Модель ownership в стиле Rust, без GC |
| **Незаметное асинхронное программирование** | Автоматическое управление асинхронностью, без явного await |
| **Полная рефлексия типов** | Полная доступность информации о типах во время runtime |
| **AI-дружественный синтаксис** | Строго структурированный, легко обрабатываемый AI |

### 1.3 Позиционирование языка

| Измерение | Позиционирование |
|-----------|------------------|
| Парадигма | Мультипарадигменный (функциональный + императивный + ООП) |
| Type system | Зависимые типы + параметрический полиморфизм |
| Управление памятью | Ownership + RAII (без GC) |
| Модель компиляции | AOT-компиляция (опционально JIT) |
| Целевые сценарии | Системное программирование, разработка приложений, AI-ассистированное программирование |

## 二、Основные особенности

### 2.1 Всё есть тип

Основная философия дизайна YaoXiang — **Всё есть тип**. Это означает, что в YaoXiang:

1. **Значение — это экземпляр типа**: `42` — экземпляр типа `Int`
2. **Тип — это экземпляр типа**: `Int` — экземпляр meta type `type`
3. **Функция — это отображение между типами**: `add: (Int, Int) -> Int` — это function type
4. **Модуль — это комбинация типов**: модуль — это пространство имён, содержащее функции и типы

```yaoxiang
# Значение — экземпляр типа
x: Int = 42

# Тип — экземпляр типа
MyList: type = List(Int)

# Функция — отображение между типами
add: (a: Int, b: Int) -> Int = a + b

# Модуль — комбинация типов (файл как модуль)
# Math.yx
pi: Float = 3.14159
sqrt: (x: Float) -> Float = { ... }
```

### 2.2 Математическая абстракция

Type system YaoXiang основан на теории типов и теории категорий и предоставляет:

- **Зависимые типы**: типы могут зависеть от значений
- **Дженерик-программирование**: параметризация типов
- **Композиция типов**: объединение типов, пересечение типов

```yaoxiang
# Зависимый тип: вектор фиксированной длины
Vector: (T: Type, n: Int) -> Type = vector(T, n)

# Объединение типов
type Number = Int | Float

# Пересечение типов
type Printable = printable(fn() -> String)
type Serializable = serializable(fn() -> String)
type Versatile = Printable & Serializable
```

### 2.3 Абстракции с нулевой стоимостью

YaoXiang гарантирует абстракции с нулевой стоимостью, то есть высокоуровневые абстракции не влекут за собой накладных расходов во время выполнения:

- **Мономорфизация**: дженерик-функции разворачиваются в конкретные версии во время компиляции
- **Инлайн-оптимизация**: простые функции автоматически инлайнятся
- **Аллокация на стеке**: мелкие объекты по умолчанию размещаются на стеке

```yaoxiang
# Разворачивание дженериков (мономорфизация)
identity: (T: Type) -> ((x: T) -> T) = x

# Использование
int_val = identity(42)      # Разворачивается в identity(Int) -> Int
str_val = identity("hello") # Разворачивается в identity(String) -> String

# Никаких дополнительных накладных расходов после компиляции
```

### 2.4 Естественный синтаксис

YaoXiang использует синтаксис в стиле Python, стремясь к читаемости и ощущению естественного языка:

```yaoxiang
# Автоматический type inference
x = 42
name = "YaoXiang"

# Лаконичное определение функций
greet: (name: String) -> String = "Hello, " + name

# Сопоставление с образцом
classify: (n: Int) -> String = {
    match n {
        0 -> "zero"
        1 -> "one"
        _ if n < 0 -> "negative"
        _ -> "many"
    }
}
```

### 2.5 Полная спецификация синтаксиса

YaoXiang применяет единый синтаксис объявления: **идентификатор: тип = выражение**. Одновременно предоставляется обратно совместимый старый синтаксис.

#### 2.5.1 Стратегия двойного синтаксиса и конвенция централизации типов

Для баланса инноваций и совместимости YaoXiang поддерживает две формы синтаксиса, но применяет единую **конвенцию централизованного аннотирования типов**.

**Сравнение форм синтаксиса:**

| Тип синтаксиса | Формат | Статус | Описание |
|----------------|--------|--------|----------|
| **Новый синтаксис (стандартный)** | `name: Type = Lambda` | ✅ Рекомендуется | Официальный стандарт, должен использоваться во всём новом коде |
| **Старый синтаксис (совместимый)** | `name(Types) -> Ret = Lambda` | ⚠️ Только для совместимости | Зарезервирован для исторического кода, не рекомендуется в новых проектах |

**Основная конвенция: централизованное аннотирование типов**

YaoXiang применяет дизайн-конвенцию **«объявление прежде всего, типы централизованы»**:

```yaoxiang
# ✅ Правильно: информация о типах единообразно размещена в строке объявления
add: (a: Int, b: Int) -> Int = a + b
#   └─────────────────┘   └─────────────┘
#    Полная сигнатура типа   Логика реализации

# ❌ Избегать: информация о типах разбросана по реализации
add: (a: Int, b: Int) -> Int = a + b
#     └───────────────┘
#     Типы перемешаны с телом реализации
```

**Преимущества конвенции:**

1. **Согласованность синтаксиса**: все объявления следуют форме `идентификатор: тип = выражение`
2. **Разделение объявления и реализации**: информация о типах видна с первого взгляда, тело реализации сосредоточено на логике
3. **AI-дружественность**: AI достаточно прочитать строку объявления, чтобы понять полную сигнатуру функции
4. **Безопасность модификации**: изменение типов требует правки только объявления, не затрагивая тело реализации
5. **Каррирование-дружественность**: поддерживает ясные каррированные сигнатуры типов

**Рекомендации по выбору**:
- **Новые проекты**: необходимо использовать новый синтаксис + конвенцию централизации типов
- **Мигрируемые проекты**: постепенно переходите на новый синтаксис и конвенцию централизации типов
- **Поддержка старого кода**: можно продолжать использовать старый синтаксис, но рекомендуется принять конвенцию централизации типов

#### 2.5.2 Базовый синтаксис объявлений

```yaoxiang
# === Новый синтаксис (рекомендуется)===
# Все объявления следуют форме: идентификатор: тип = выражение

# Объявление переменных
x: Int = 42
name: String = "YaoXiang"
mut counter: Int = 0

# Объявление функций
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1
getAnswer: () -> Int = 42
log: (msg: String) -> Void = print(msg)

# === Старый синтаксис (для совместимости)===
# Только для функций, формат: name(Types) -> Ret = Lambda
add(Int, Int) -> Int = (a, b) => a + b
square(Int) -> Int = (x) => x * x
empty() -> Void = () => {}
getRandom() -> Int = () => 42
```

#### 2.5.3 Синтаксис function type

```
function type ::= '(' список_типов_параметров ')' '->' тип_возврата
                | тип_параметра '->' тип_возврата         # Сокращение для одного параметра

список_типов_параметров ::= [тип (',' тип)*]
тип_возврата ::= тип | function type | 'Void'

# Function type — это гражданин первого класса, может быть вложенным
# function type высшего порядка ::= '(' function type ')' '->' тип_возврата
```

| Пример | Значение |
|--------|----------|
| `Int -> Int` | Function type с одним параметром |
| `(Int, Int) -> Int` | Function type с двумя параметрами |
| `() -> Void` | Function type без параметров |
| `(Int -> Int) -> Int` | Функция высшего порядка: принимает функцию, возвращает Int |
| `Int -> Int -> Int` | Каррированная функция (правоассоциативно) |

#### 2.5.4 Синтаксис дженериков (только для параметров типов)

```yaoxiang
# Дженерик-функция: <параметр_типа> префикс
identity: (T: Type) -> ((x: T) -> T) = x
map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) = case xs of
  [] => []
  (x :: rest) => f(x) :: map(f, rest)

# Дженерик-тип
List: (T: Type) -> Type
```

#### 2.5.5 Синтаксис Lambda-выражений

```
Lambda ::= '(' список_параметров ')' '=>' выражение
         | параметр '=>' выражение              # Сокращение для одного параметра

список_параметров ::= [параметр (',' параметр)*]
параметр ::= идентификатор [':' тип]               # Необязательная аннотация типа
```

| Пример | Значение | Пояснение |
|--------|----------|-----------|
| `(a, b) => a + b` | Lambda с несколькими параметрами | Используется с объявлением:<br>`add: (Int, Int) = (a, b) => a + b` |
| `x => x + 1` | Сокращение для одного параметра | Используется с объявлением:<br>`inc: Int = x => x + 1` |
| `(x: Int) => x + 1` | С аннотацией типа | Только для временной необходимости внутри Lambda |
| `() => 42` | Lambda без параметров | Используется с объявлением:<br>`get: () = () => 42` |

**Примечание**: аннотация типов в Lambda-выражении `(x: Int) => ...` является **временной, локальной**, используется главным образом для:
- Случаев, когда внутри Lambda требуется информация о типе
- Использования вместе с синтаксисом объявления (типы уже даны в объявлении)
- Не должна служить основным способом объявления типов

```yaoxiang

# === Функции высшего порядка и присваивание function type ===

# Function type как гражданин первого класса
IntToInt: Type = (Int) -> Int
IntBinaryOp: Type = (Int, Int) -> Int

# Объявление функции высшего порядка
applyTwice: (f: IntToInt, x: Int) -> Int = f(f(x))

# Каррированная функция
addCurried: (a: Int) -> (b: Int) -> Int = a + b

# Композиция функций
compose: (f: Int -> Int, g: Int -> Int) -> (x: Int) -> Int =
  f(g(x))

# Функция, возвращающая функцию
makeAdder: (x: Int) -> (y: Int) -> Int =
  x + y

# === Дженерик-функции ===

# Дженерик-функция
identity: (T: Type) -> ((x: T) -> T) = x

# Дженерик-функция высшего порядка
map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) =
  case xs of
    [] => []
    (x :: rest) => f(x) :: map(f, rest)

# Дженерик-function type
Transformer: Type = (A: Type, B: Type) -> ((A) -> B)

# Использование дженерик-типа
applyTransformer: (A: Type, B: Type) -> ((f: Transformer(A, B), x: A) -> B) =
  f(x)

# === Примеры сложных типов ===

# Вложенный function type
higherOrder: (A: Type) -> ((f: (A) -> Int) -> (A) -> Int) =
  f => x => f(x) + 1

# Многопараметрическая функция высшего порядка
zipWith: (A: Type, B: Type, C: Type) -> ((f: (A, B) -> C, xs: List(A), ys: List(B)) -> List(C)) =
  case (xs, ys) of
    ([], _) => []
    (_, []) => []
    (x::xs', y::ys') => f(x, y) :: zipWith(f, xs', ys')

# Псевдонимы function type
Predicate: (T: Type) -> Type = { apply: (T) -> Bool }
Mapper: Type = (A: Type, B: Type) -> ((A) -> B)
Reducer: Type = (A: Type, B: Type) -> ((B, A) -> B)

# === Примеры старого синтаксиса (только для обратной совместимости) ===
# Не рекомендуется в новом коде

mul(Int, Int) -> Int = (a, b) => a * b    # Несколько параметров
square(Int) -> Int = (x) => x * x          # Один параметр
empty() -> Void = () => {}                  # Без параметров
get_random() -> Int = () => 42              # С возвращаемым значением

# Эквивалентный новый синтаксис (рекомендуется)
mul: (a: Int, b: Int) -> Int = a * b
square: (x: Int) -> Int = x * x
empty: () -> Void = {}
get_random: () -> Int = 42
```

#### 2.5.7 Правила разбора синтаксиса

**Приоритет разбора типов:**

| Приоритет | Тип | Описание |
|-----------|-----|----------|
| 1 (наивысший) | Применение дженерика `List(T)` | Левоассоциативно |
| 2 | Скобки `(T)` | Изменяют ассоциативность |
| 3 | Function type `->` | Правоассоциативно |
| 4 (низший) | Базовые типы `Int, String` | Атомарные типы |

**Примеры разбора типов:**

```yaoxiang
# (A -> B) -> C -> D
# Разбирается как: ((A -> B) -> (C -> D))

# A -> B -> C
# Разбирается как: (A -> (B -> C))  # Правоассоциативно

# (Int -> Int) -> Int
# Разбирается как: принимает функцию, возвращает Int -> Int

# List<Int -> Int>
# Разбирается как: тип элемента List — это Int -> Int
```

**Примеры разбора Lambda:**

```yaoxiang
# a => b => a + b
# Разбирается как: a => (b => (a + b))  # Правоассоциативно, каррирование

# (a, b) => a + b
# Разбирается как: принимает два параметра, возвращает a + b
```

#### 2.5.8 Правила type inference

YaoXiang применяет стратегию **двухуровневой обработки**: парсер пропускает снисходительно, а слой проверки типов строго выводит.

**Правила уровня парсера:**
- Парсер проверяет только синтаксическую структуру, не выполняет type inference
- Для объявлений без аннотации типа поле аннотации типа устанавливается в `None`
- Все объявления, удовлетворяющие базовой синтаксической структуре, проходят парсер
- **Ключевой момент**: `add: (a: Int, b: Int) -> Int = a + b` на уровне парсера является **легальным**

**Правила уровня проверки типов:**
- Проверка семантической корректности, включая полноту типов
- **Параметры должны иметь аннотацию типа**: это обязательное требование
- Тип возврата может быть выведен, но типы параметров должны быть объявлены явно

**Полные правила type inference:**

| Сценарий | Вывод параметров | Вывод возврата | Результат парсинга | Результат проверки типов | Рекомендация |
|----------|------------------|----------------|--------------------|--------------------------|--------------|
| **Стандартная функция** | Использует аннотированные типы | Использует аннотированный тип | ✅ | ✅ | ⭐⭐⭐⭐⭐ |
| `add: (a: Int, b: Int) -> Int = a + b` | | | | | |
| **Частичный вывод** | Использует аннотированные типы | Выводится из выражения | ✅ | ✅ | ⭐⭐⭐⭐ |
| `add: (Int, Int) = (a, b) => a + b` | | | | | |
| `inc: (x: Int) -> Int = x + 1` | | | | | |
| `get: () = () => 42` | | | | | |
| **Частичный вывод в старом синтаксисе** | Использует аннотированные типы | Выводится из выражения | ✅ | ✅ | ⭐⭐⭐ (совместимость) |
| `add(Int, Int) = (a, b) => a + b` | | | | | |
| `square(Int) = (x) => x * x` | | | | | |
| **Параметры без аннотации** | **Невозможно вывести** | - | ✅ | ❌ Ошибка | ❌ Запрещено |
| `add: (a: Int, b: Int) -> Int = a + b` | | | | | |
| `identity: (T: Type) -> ((x: T) -> T) = x` | | | | | |
| **Блок без аннотации возврата** | - | Выводится из содержимого блока | ✅ | ✅ | ⭐⭐⭐⭐ |
| `main = () => {}` | | | | | |
| `get = () => { return 42; }` | | | | | |
| **Блок кода без return** | - | По умолчанию `Void` | ✅ | ✅ Корректно | ✅ Корректно |
| `add: (a: Int, b: Int) -> Int = { return a + b }` | | | | | |

**Подробные правила вывода:**

```
Уровень парсера: проверяет только синтаксическую структуру
├── Структура корректна → Пройдено
└── Структура некорректна → Ошибка

Уровень проверки типов: проверяет семантику
├── Вывод типов параметров
│   ├── Параметр имеет аннотацию типа → Используется аннотированный тип ✅
│   ├── Параметр не имеет аннотации типа → Отказ ❌
│   └── Параметры Lambda должны быть аннотированы → Обязательно
│
├── Вывод типа возврата
│   ├── Есть return expr → Выводится из expr ✅
│   ├── Нет return, есть выражение → Выводится из выражения ✅
│   ├── Блок кода без return → По умолчанию возвращает Void ✅
│   └── Невозможно вывести → Отказ ❌
│
└── Полностью невозможно вывести → Отказ ❌
```

**Примечание**: при отсутствии `return` внутри блока кода по умолчанию возвращается `Void`. Например:
- `() => { 42 }` выводится как `() -> Void` (блок без return, по умолчанию возвращает Void)
- `() => { return 42 }` выводится как `() -> Int` (есть return, выводится из return)
- `() => 42` выводится как `() -> Int` (форма выражения, возвращает значение напрямую)

**Примеры вывода:**

```yaoxiang
# === Успешный вывод ===

# Стандартная форма
main: () -> Void = () => {}                    # Полная аннотация
num: () -> Int = () => 42                      # Полная аннотация
inc: (x: Int) -> Int = x + 1                   # Сокращение для одного параметра

# Частичный вывод (новый синтаксис)
add: (Int, Int) = (a, b) => a + b              # Параметры аннотированы, возврат выводится
square: (x: Int) -> Int = x * x                # Параметры аннотированы, возврат выводится
get_answer: () = () => 42                      # Параметры аннотированы (пусто), возврат выводится

# Частичный вывод (старый синтаксис, совместимость)
add2(Int, Int) = (a, b) => a + b               # Параметры аннотированы, возврат выводится
square2(Int) = (x) => x * x                    # Параметры аннотированы, возврат выводится

# Вывод из return
fact: Int -> Int = (n) => {
    if n <= 1 { return 1 }
    return n * fact(n - 1)
}

# === Сбой вывода ===

# Невозможно вывести параметры (парсинг пройден, проверка типов не пройдена)
add: (a: Int, b: Int) -> Int = a + b                          # ✗ Параметры без типа
identity: (T: Type) -> ((x: T) -> T) = x                              # ✗ Параметры без типа

# Блок кода без return
no_return = (x: Int) => { x }                  # ✓ Выводится как Void (блок без return, по умолчанию возвращает Void)

# Полностью невозможно вывести
bad_fn: (T: Type) -> ((x: T) -> T) = x                                # ✗ Параметры и возврат не могут быть выведены
```

#### 2.5.9 Старый синтаксис (обратная совместимость)

YaoXiang предоставляет поддержку старого синтаксиса для совместимости с историческим кодом, **не рекомендуется в новом коде**.

```
старый_синтаксис ::= идентификатор '(' [список_типов_параметров] ')' '->' тип_возврата '=' Lambda
```

| Свойство | Стандартный синтаксис | Старый синтаксис |
|----------|----------------------|------------------|
| Формат объявления | `name: Type = ...` | `name(Types) -> Type = ...` |
| Позиция типов параметров | В аннотации типа | В скобках после имени функции |
| Пустые параметры | Обязательно писать `()` | Можно опустить `()` |
| **Рекомендация** | ✅ **Официально рекомендуется** | ⚠️ **Только для обратной совместимости** |
| **Сценарий использования** | Весь новый код | Поддержка исторического кода |

**Причины нерекомендации:**
1. **Стоимость обучения**: несовместимо со стандартным синтаксисом, увеличивает сложность языка
2. **Согласованность**: позиции типов параметров неединообразны (один в аннотации типа, другой после имени функции)
3. **Стоимость поддержки**: парсеру необходимо дополнительно обрабатывать две формы
4. **AI-дружественность**: усложняет понимание и генерацию кода AI

**Рекомендации по миграции:**
```yaoxiang
# Старый код (не рекомендуется)
mul(Int, Int) -> Int = (a, b) => a * b
square(Int) -> Int = (x) => x * x
empty() -> Void = () => {}

# Новый код (рекомендуется)
mul: (Int, Int) -> Int = (a, b) => a * b
square: (Int) -> Int = (x) => x * x
empty: () -> Void = () => {}
```

```yaoxiang
# Единый синтаксис типов: только конструкторы, без ключевых слов enum/struct/union

}
```

## 四、Управление памятью

### 4.1 Модель ownership

YaoXiang использует **модель ownership** для управления памятью: каждое значение имеет единственного владельца:

```yaoxiang
# === Move по умолчанию (без копирования) ===
p: Point = Point(1.0, 2.0)
p2 = p              # Move, право владения передаётся, p становится недействительным

# === Ключевое слово ref = Arc (безопасное разделение) ===
shared = ref p      # Arc, потокобезопасно

spawn(() => print(shared.x))   # ✅ Безопасно

# === Явное копирование clone() ===
p3 = p.clone()      # p и p3 независимы
```

### 4.2 Семантика Move (по умолчанию)

```yaoxiang
# Присваивание = Move (без копирования)
p: Point = Point(1.0, 2.0)
p2 = p              # Move, p становится недействительным

# Передача параметра = Move
process: (p: Point) -> Void = {
    # Право владения на p передаётся внутрь
}

# Возвращаемое значение = Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    return p        # Move, право владения передаётся
}
```

### 4.3 Ключевое слово ref (Arc)

```yaoxiang
# Ключевое слово ref создаёт Arc (подсчёт ссылок)
p: Point = Point(1.0, 2.0)
shared = ref p      # Arc, потокобезопасно

spawn(() => print(shared.x))   # ✅ Arc потокобезопасен

# Arc автоматически управляет временем жизни
# Когда shared выходит из области видимости, счётчик обнуляется и память освобождается
```

### 4.4 Явное копирование clone()

```yaoxiang
# Когда нужно сохранить исходное значение, используйте clone()
p: Point = Point(1.0, 2.0)
p2 = p.clone()   # p и p2 независимы

p.x = 0.0        # ✅
p2.x = 0.0       # ✅ Не влияют друг на друга
```

### 4.5 Блоки unsafe (системный уровень)

```yaoxiang
# Сырые указатели можно использовать только в блоках unsafe
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p     # Сырой указатель
    (*ptr).x = 0.0       # Пользователь гарантирует безопасность
}
```

### 4.6 RAII

```yaoxiang
# RAII — автоматическое освобождение
with_file: (path: String) -> String = {
    file = File.open(path)  # Автоматически открывается
    content = file.read_all()
    # По завершении функции file автоматически закроется
    content
}
```

### 4.7 Ограничения Send / Sync

| Ограничение | Семантика | Описание |
|-------------|-----------|----------|
| **Send** | Безопасно передавать между потоками | Значение можно переместить в другой поток |
| **Sync** | Безопасно разделять между потоками | Неизменяемую ссылку можно разделять между потоками |

```yaoxiang
# ref T автоматически удовлетворяет Send + Sync (Arc потокобезопасен)
p: Point = Point(1.0, 2.0)
shared = ref p

spawn(() => print(shared.x))   # ✅ Arc потокобезопасен

# Сырой указатель *T не удовлетворяет Send/Sync
unsafe {
    ptr: *Point = &p         # Можно использовать только в одном потоке
}
```

## 五、Асинхронное программирование и параллелизм

> «Все вещи рождаются вместе; я наблюдаю их возвращение.» — «И цзин», гексаграмма Фу (Возврат)
>
> YaoXiang применяет **spawn-модель** — парадигму незаметного асинхронного параллелизма, основанную на **ленивом вычислении**. Основная идея дизайна: **позволить разработчику описывать логику в синхронном, последовательном мышлении, а runtime языка организует автоматическое и эффективное параллельное выполнение вычислительных единиц, словно все вещи рождаются вместе, с унифицированной координацией в конце**.

> Подробности см. в [«Белой книге spawn-модели»](YaoXiang-async-whitepaper.md) и [реализации асинхронности](YaoXiang-async-implementation.md).

### 5.1 Основные концепции spawn-модели

#### 5.1.1 Граф spawn: сцена, где рождаются все вещи

Все программы на этапе компиляции преобразуются в **ориентированный ациклический вычислительный граф (DAG)**, называемый **графом spawn**. Узлы представляют вычисления выражений, рёбра — зависимости данных. Этот граф ленив: узлы вычисляются только тогда, когда их выход **действительно нужен**.

```yaoxiang
# Компилятор автоматически строит граф spawn
fetch_user: spawn () -> User = (id) => { ... }
fetch_posts: spawn (User) -> Posts = (user) => { ... }

main:() -> Void = () => {
    user = fetch_user(1)     # Узел A (Async(User))
    posts = fetch_posts(user) # Узел B (Async(Posts)), зависит от A

    # Узлу C нужны результаты A и B
    print(posts.title)       # Автоматическое ожидание: сначала дождаться завершения A и B
}
```

#### 5.1.2 Значение spawn: Async(T)

Любой вызов функции, помеченной `spawn fn`, немедленно возвращает значение типа `Async(T)`, называемое **значением spawn**. Это легковесный прокси, который не является фактическим результатом, а представляет **будущее значение в процессе spawn**.

**Ключевые свойства**:
- **Прозрачность типов**: `Async(T)` в системе типов является подтипом `T` и может использоваться в любом контексте, где ожидается `T`
- **Автоматическое ожидание**: когда выполнение программы доходит до операции, требующей конкретного значения типа `T`, runtime автоматически приостанавливает текущую задачу и ожидает завершения вычисления
- **Нулевое заражение**: асинхронный код и синхронный код синтаксически и по сигнатурам типов не отличаются

```yaoxiang
# Пример использования значения spawn
fetch_data: spawn (String) -> JSON = (url) => { ... }

main: () -> Void = () => {
    data = fetch_data("url")  # Async(JSON)

    # Async(JSON) можно использовать непосредственно как JSON
    # Автоматическое ожидание срабатывает при доступе к полю
    print(data.name)          # Эквивалентно data.await().name
}
```

### 5.2 Система синтаксиса spawn

Ключевое слово `spawn` имеет тройную семантику и является единственным мостом между синхронным мышлением и асинхронной реализацией:

| Официальный термин | Синтаксическая форма | Семантика | Поведение runtime |
|--------------------|----------------------|-----------|-------------------|
| **spawn function** | `spawn fn` | Определяет вычислительную единицу, участвующую в spawn-выполнении | Её вызов возвращает `Async(T)` |
| **spawn block** | `spawn { a(), b() }` | Явно объявленная параллельная область | Задачи внутри блока принудительно выполняются параллельно |
| **spawn loop** | `spawn for x in xs { ... }` | Парадигма параллелизма по данным | Тело цикла выполняется spawn на всех элементах |

#### 5.2.1 spawn function

```yaoxiang
# Используйте spawn для пометки spawn-функции
# Синтаксис полностью идентичен обычной функции, без дополнительной нагрузки

fetch_api: spawn (String) -> JSON = (url) => {
    response = HTTP.get(url)
    JSON.parse(response.body)
}

# Вложенные spawn-вызовы
process_user: (Int) -> Report = (user_id) => {
    user = fetch_user(user_id)     # Async(User)
    profile = fetch_profile(user)  # Async(Profile), зависит от user
    generate_report(user, profile) # Зависит от profile
}
```

#### 5.2.2 spawn block

```yaoxiang
# spawn { } - явная параллельная конструкция
# Все выражения внутри блока выполняются как независимые задачи параллельно

compute_all: (Int, Int) -> (Int, Int, Int) spawn = (a, b) => {
    # Три независимых вычисления выполняются параллельно
    (x, y, z) = spawn {
        heavy_calc(a),        # Задача 1
        heavy_calc(b),        # Задача 2
        another_calc(a, b)    # Задача 3
    }
    (x, y, z)
}
```

#### 5.2.3 spawn loop

```yaoxiang
# spawn for - цикл с параллелизмом по данным
# Каждая итерация выполняется как независимая задача параллельно

parallel_sum: (Int) -> Int spawn = (n) => {
    total = spawn for i in 0..n {
```

### 5.5 Разделение между задачами: ключевое слово ref

YaoXiang не требует ручных примитивов синхронизации вроде `Send`/`Sync` trait, `Mutex`, `RwLock`. Ownership + `ref` автоматически обрабатывают безопасность параллелизма.

#### 5.5.1 ref: разделение между областями видимости

```yaoxiang
# ref — единственный способ разделения между областями видимости
# Компилятор автоматически выбирает Rc (одна задача) или Arc (между задачами)

data = load_data()
shared = ref data       # Компилятор автоматически выбирает реализацию

result = spawn {
    process_a(shared),  # Общая ссылка, между задачами → Arc
    process_b(shared)   # Общая ссылка
}
```

#### 5.5.2 Автоматическая оптимизация компилятора

```
Анализ потока данных ref:

Не ускользает в другие задачи → Rc (неатомарный подсчёт ссылок, низкие накладные расходы)
Ускользает в другие задачи    → Arc (атомарный подсчёт ссылок, потокобезопасно)
```

Пользователю не нужно заботиться о том, что в основе — Rc или Arc: `ref` означает разделяемое владение, и этого достаточно.

#### 5.5.3 Автоматическая сериализация типов ресурсов

```yaoxiang

    write_file("data.txt", x)   # Ожидает завершения чтения
}
```

## 七、Привязка методов и каррирование

YaoXiang использует **чисто функциональный дизайн** и через передовые механизмы привязки обеспечивает бесшовный вызов методов и каррирование, без необходимости вводить ключевые слова `struct`, `class` и т.п.

### 7.1 Определения базовых функций

Все операции реализуются через обычные функции, причём первый параметр по соглашению является субъектом операции:

```yaoxiang
# === Point.yx (модуль) ===

# Единый синтаксис: конструктор и есть тип
type Point = { x: Float, y: Float }

# Базовая функция: первый параметр — субъект операции
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

add: (a: Point, b: Point) -> Point = {
    Point(a.x + b.x, a.y + b.y)
}

scale: (p: Point, s: Float) -> Point = {
    Point(p.x * s, p.y * s)
}

# Более сложная функция
distance_with_scale: (s: Float, p1: Point, p2: Point) -> Float = {
    dx = (p1.x - p2.x) * s
    dy = (p1.y - p2.y) * s
    (dx * dx + dy * dy).sqrt()
}
```

### 7.2 Базовая привязка методов

#### 7.2.1 Автоматическая привязка (в стиле MoonBit)

YaoXiang поддерживает автоматическую привязку на основе пространств имён, **без каких-либо дополнительных объявлений**:

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Базовая функция
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# === main.yx ===

use Point

main: () -> Void = {
    p1 = Point(3.0, 4.0)
    p2 = Point(1.0, 2.0)

    # ✅ Автоматическая привязка: прямой вызов метода
    result = p1.distance(p2)  # Разбирается как distance(p1, p2)
}
```

**Правила автоматической привязки**:
- Функции, определённые внутри модуля
- Если тип первого параметра совпадает с именем модуля
- Тогда автоматически поддерживается синтаксис вызова метода

#### 7.2.2 Отсутствие привязки (поведение по умолчанию)

```yaoxiang
# === Vector.yx ===

type Vector = Vector(x: Float, y: Float, z: Float)

# Внутренняя вспомогательная функция, без намерения автоматической привязки
dot_product_internal: (a: Vector, b: Vector) -> Float = {

    # v1.dot_product_internal(v2)  # Ошибка компиляции!

    # ✅ Необходимо вызывать напрямую (не виден за пределами модуля)
}
```

### 7.3 Привязка на основе позиций

YaoXiang предоставляет **наиболее элегантный синтаксис привязки**, использующий позиционные метки `[n]` для точного контроля позиции привязки:

#### 7.3.1 Базовый синтаксис

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Базовая функция
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}
add: (a: Point, b: Point) -> Point = {
    Point(a.x + b.x, a.y + b.y)
}
scale: (p: Point, s: Float) -> Point = {
    Point(p.x * s, p.y * s)
}

# Синтаксис привязки: Type.method = func[позиция]
# Означает: при вызове метода вызывающий привязывается к параметру func с позицией [позиция]

Point.distance = distance[0]      # Привязка к 1-му параметру
Point.add = add[0]                 # Привязка к 1-му параметру
Point.scale = scale[0]             # Привязка к 1-му параметру
```

**Семантический разбор**:
- `Point.distance = distance[0]`
  - Функция `distance` имеет два параметра: `distance(Point, Point)`
  - `[0]` означает, что вызывающий привязывается к 1-му параметру
  - Использование: `p1.distance(p2)` → `distance(p1, p2)`

#### 7.3.2 Многопозиционная совместная привязка

```yaoxiang
# === Math.yx ===

# Функция: scale, point1, point2, extra1, extra2
calculate: (scale: Float, a: Point, b: Point, x: Float, y: Float) -> Float = { ... }

# === Point.yx ===

type Point = { x: Float, y: Float }

# Привязка нескольких позиций
Point.calc1 = calculate[1, 2]      # Привязка scale и point1
Point.calc2 = calculate[1, 3]      # Привязка scale и point2
Point.calc3 = calculate[2, 3]      # Привязка point1 и point2

# === main.yx ===

use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# 1. Привязка [1,2] - остаётся 3,4,5
f1 = p1.calc1(2.0)  # Привязка scale=2.0, point1=p1
# f1 теперь требует p2, x, y
result1 = f1(p2, 10.0, 20.0)  # calculate(2.0, p1, p2, 10.0, 20.0)

# 2. Привязка [1,3] - остаётся 2,4,5
f2 = p2.calc2(2.0)  # Привязка scale=2.0, point2=p2
# f2 теперь требует point1, x, y
result2 = f2(p1, 10.0, 20.0)  # calculate(2.0, p1, p2, 10.0, 20.0)

# 3. Привязка [2,3] - остаётся 1,4,5
f3 = p1.calc3(p2)  # Привязка point1=p1, point2=p2
# f3 теперь требует scale, x, y
result3 = f3(2.0, 10.0, 20.0)  # calculate(2.0, p1, p2, 10.0, 20.0)
```

#### 7.3.3 Порядок заполнения остальных параметров

**Ключевое правило**: после привязки остальные параметры заполняются в **исходном порядке функции**, пропуская уже связанные позиции.

```yaoxiang
# Предположим, функция: func(p1, p2, p3, p4, p5)

# Привязка 1-го и 3-го параметров
Type.method = func[1, 3]

# При вызове:
method(p2_value, p4_value, p5_value)

# Отображается в:
func(p1_bound, p2_value, p3_bound, p4_value, p5_value)
# Остальные параметры: 2,4,5 заполняются в исходном порядке
```

#### 7.3.4 Преимущества проверки типов

```yaoxiang
# ✅ Легальная привязка
Point.distance = distance[0]          # distance(Point, Point)
Point.calc = calculate[1, 2]          # calculate(scale, Point, Point, ...)

# ❌ Нелегальная привязка (ошибка компилятора)
Point.wrong = distance[5]             # 5-го параметра не существует
Point.wrong = distance[0]             # Параметры начинаются с 1
Point.wrong = distance[1, 2, 3, 4]    # Превышение числа параметров
```

### 7.4 Тонкий контроль каррированной привязки

```yaoxiang
# === Math.yx ===

distance_with_scale: (scale: Float, a: Point, b: Point) -> Float = { ... }

# === Point.yx ===

type Point = { x: Float, y: Float }

# Стратегия привязки: гибкое управление каждой позицией

Point.distance_scaled = distance_with_scale[2]  # Привязка ко 2-му параметру

# === main.yx ===

p2 = Point(1.0, 2.0)

# 1. Базовая автоматическая привязка

# 2. Привязка к разным позициям
f = p1.distance_scaled(2.0)  # Привязка 2-го параметра, остаются 1-й и 3-й
result = f(p2)               # distance_with_scale(2.0, p1, p2)

# 3. Цепочка привязок
d2 = p1.distance(p2).distance_scaled(2.0)  # Цепочечный вызов
```

### 7.6 Область видимости и правила привязки

#### 7.6.1 Роль pub

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Не-pub функция
internal_distance: (a: Point, b: Point) -> Float = {

# pub функция
pub distance: (a: Point, b: Point) -> Float = {

    (dx * dx + dy * dy).sqrt()
}

# === main.yx ===

use Point

# Автоматическая привязка действует только для pub функций
p1.distance(p2)      # ✅ distance — pub, может быть автоматически привязана
# p1.internal_distance(p2)  # ❌ Не pub, не может быть привязана
```

#### 7.6.2 Механизм автоматической привязки pub

Для функций, объявленных с `pub`, компилятор автоматически привязывает их к типам, определённым в том же файле:

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Объявлено с pub, компилятор автоматически привязывает
pub distance: (p1: Point, p2: Point) -> Float = {
    dx = p1.x - p2.x
    dy = p1.y - p2.y
    (dx * dx + dy * dy).sqrt()
}

pub translate: (self: Point, dx: Float, dy: Float) -> Point = {
    Point(self.x + dx, self.y + dy)
}

# Компилятор автоматически выводит и выполняет привязку:
# Point.distance = distance[0]
# Point.translate = translate[0]

# === main.yx ===

use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# ✅ Функциональный вызов
d = distance(p1, p2)

# ✅ ООП-синтаксический сахар (автоматическая привязка)
d2 = p1.distance(p2)
p3 = p1.translate(1.0, 1.0)
```

**Правила автоматической привязки**:
1. Функция определена в файле модуля (в том же файле, что и тип)
2. Параметры функции включают этот тип
3. Экспортирована с `pub`
4. Компилятор автоматически выполняет `Type.method = function[0]`

**Преимущества**:
- Не нужно вручную писать объявления привязки
- Код становится лаконичнее
- Избегаются забытые или ошибочные привязки

#### 7.6.3 Привязка внутри модуля

```yaoxiang

# Но автоматическая привязка действует снаружи только для pub функций
```

### 7.7 Сводка преимуществ дизайна

| Особенность | Описание |
|-------------|----------|
| **Нулевая синтаксическая нагрузка** | Автоматическая привязка не требует объявлений |
| **Точный позиционный контроль** | `[n]` точно указывает позицию привязки |
| **Многопозиционная совместность** | Поддержка `[1, 2, 3]` для нескольких параметров |
| **Типобезопасность** | Компилятор проверяет валидность позиций привязки |
| **Без ключевых слов** | Не нужен `bind` или другие ключевые слова |
| **Гибкое каррирование** | Поддержка привязки параметров в произвольных позициях |
| **Контроль через pub** | Только pub функции могут быть привязаны снаружи |

### 7.8 Отличие от традиционной привязки методов

| Традиционные языки | YaoXiang |
|--------------------|----------|
| `obj.method(arg)` | `obj.method(arg)` |
| Требуется определение класса/метода | Требуется только функция + объявление привязки |
| Синтаксис `class { method() {} }` | Синтаксис `Type.method = func[n]` |
| Наследование, полиморфизм | Чисто функциональный, без наследования |
| Поиск в таблице методов | Привязка на этапе компиляции, без накладных расходов в runtime |

**Ключевое преимущество**: привязка в YaoXiang — это **механизм времени компиляции** с нулевыми затратами в runtime, сохраняющий чистоту и гибкость функционального программирования.

```yaoxiang
add2: (a: Int, b: Int) -> Int = a + b  # Типы разбросаны по реализации

# ⚠️ Совместимость: старый синтаксис только для поддержки
# AI должен в первую очередь генерировать новый синтаксис + конвенцию централизации типов
mul(Int, Int) -> Int = (a, b) => a * b  # Не рекомендуется в новом коде
```

**AI-преимущества конвенции централизации типов:**

1. **Сигнатура видна с первого взгляда**: AI достаточно прочитать строку объявления, чтобы понять полную сигнатуру функции
2. **Безопасность модификации**: изменение типов требует правки только объявления, не затрагивая тело реализации
3. **Простота генерации**: AI может сначала сгенерировать объявление, затем заполнить реализацию
4. **Каррирование-дружественность**: ясные каррированные сигнатуры типов облегчают обработку AI

```yaoxiang
# Пример обработки AI
# Вход: тело реализации (a, b) => a + b
# AI видит объявление: add: (Int, Int) -> Int
# Вывод: типы параметров — Int, Int, тип возврата — Int

# Сравнение: если типы разбросаны
# Вход: тело реализации (a: Int, b: Int) => a + b
# AI необходимо: анализировать тело реализации для извлечения информации о типах
# Результат: более сложная логика обработки, легче допустить ошибку
```

#### 8.2.2 Стратегия двойного синтаксиса и AI

| Тип синтаксиса | Стратегия AI-генерации | Сценарий использования |
|----------------|------------------------|------------------------|
| **Новый синтаксис** | ✅ Приоритетная генерация, полная информация о типах | Вся генерация нового кода |
| **Старый синтаксис** | ⚠️ Только при поддержке старого кода | Модификация исторического кода |
| **Без аннотации** | ❌ Избегать генерации | Ни в коем случае не должен генерироваться |

#### 8.2.3 Ясные границы синтаксиса

```yaoxiang
# AI-дружественные границы блоков кода

# ✅ Чёткие метки начала и конца
function_name: (Type1, Type2) -> ReturnType = (param1, param2) => {
    # Тело функции
    if condition {
        do_something()
    } else {
        do_other()
    }
}

# ✅ Условные операторы должны иметь фигурные скобки
if condition {
    # Тело условия
}

# ✅ Определения типов ясны
type MyType = Type1 | Type2

# ❌ Избегать неоднозначных формулировок
if condition    # Отсутствуют фигурные скобки
    do_something()
```

#### 8.2.4 Ограничения однозначного синтаксиса

```yaoxiang
# Ограничения, которые AI должен соблюдать при генерации

# 1. Запрещено опускать скобки
# ✅ Правильно
foo: (T: Type) -> ((x: T) -> T) = x
my_list = [1, 2, 3]

# ❌ Неправильно (запрещено)
foo T { T }             # Параметры должны быть в скобках
my_list = [1 2 3]       # В списках должны быть запятые

# 2. Должен быть явный возвращаемый тип или выводимая форма
# ✅ Правильно
get_num: () -> Int = 42
get_num2: () = 42          # Тип возврата выводится
get_void = () => { 42 }    # ✓ Выводится как Void (блок без return, по умолчанию возвращает Void)

# 3. Параметры должны иметь аннотацию типа (новый синтаксис)
# ✅ Правильно
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1

# ❌ Неправильно
add: (a: Int, b: Int) -> Int = a + b            # Параметры без типа
identity: (T: Type) -> ((x: T) -> T) = x                # Параметры без типа
```

#### 8.2.5 Рекомендуемые шаблоны AI-генерации

```yaoxiang

# Шаблон 1: полная аннотация типа
function_name: (param1: ParamType1, param2: ParamType2, ...) -> ReturnType = {
    # Тело функции
    return expression
}

# Шаблон 2: вывод возвращаемого типа
function_name: (param1: ParamType1, param2: ParamType2) = {
    # Тело функции
    return expression
}

# Шаблон 3: сокращение для одного параметра
function_name: (param: ParamType) -> ReturnType = expression

# Шаблон 4: функция без параметров
function_name: () -> ReturnType = expression

# Шаблон 5: пустая функция
function_name: () -> Void = {}
```

## 九、Конвенция централизации типов (основная дизайн-философия)

### 9.1 Обзор конвенции

Основная дизайн-конвенция YaoXiang — **«объявление прежде всего, типы централизованы»**. Эта конвенция является краеугольным камнем AI-дружественности языка и эффективности разработки.

```yaoxiang
# ✅ Основная конвенция: информация о типах единообразно размещена в строке объявления
add: (a: Int, b: Int) -> Int = a + b

# ❌ Избегать: информация о типах разбросана по реализации
add: (a: Int, b: Int) -> Int = a + b
```

### 9.2 Пять ключевых преимуществ конвенции

#### 1. Согласованность синтаксиса
```yaoxiang
# Все объявления следуют одному формату
x: Int = 42                           # Переменная
name: String = "YaoXiang"             # Переменная
add: (a: Int, b: Int) -> Int = a + b  # Функция
inc: (x: Int) -> Int = x + 1          # Функция
type Point = { x: Float, y: Float } # Тип
```

#### 2. Разделение объявления и реализации
```yaoxiang
# Строка объявления предоставляет полную информацию о типах
add: (a: Int, b: Int) -> Int = a + b
# └────────────────────┘
#   Полная сигнатура функции

# Тело реализации сосредоточено на бизнес-логике
# (a, b) => a + b  не нужно заботиться о типах, только реализовать функциональность
```

#### 3. AI-дружественность
```yaoxiang
# Процесс обработки AI:
# 1. Чтение строки объявления → полное понимание сигнатуры функции
# 2. Генерация реализации → без необходимости анализа вывода типов
# 3. Изменение типов → правка только строки объявления, не затрагивая реализацию

# Сравнение: способ с разбросанными типами
add: (a: Int, b: Int) -> Int = a + b
# AI необходимо: анализ тела реализации для извлечения информации о типах → сложнее, легче ошибиться
```

#### 4. Безопасность модификации
```yaoxiang
# Изменение типа параметра
# Было: add: (a: Int, b: Int) -> Int = a + b
# Стало: add: (Float, Float) -> Float = (a, b) => a + b
# Тело реализации: (a, b) => a + b  без изменений!

# Если типы разбросаны:
# Было: add: (a: Int, b: Int) -> Int = a + b
# Стало: add: (a: Float, b: Float) -> Float = a + b  # нужно изменить в двух местах
```

#### 5. Каррирование-дружественность
```yaoxiang
# Каррированная сигнатура видна с первого взгляда
add_curried: (a: Int) -> (b: Int) -> Int = a + b
#              └─────────────┘
#              Каррированная сигнатура

# Композиция функций как гражданин первого класса
compose: (Int -> Int, Int -> Int) -> Int -> Int = (f, g) => x => f(g(x))
```

### 9.3 Правила реализации конвенции

#### Правило 1: параметры должны указывать тип в объявлении
```yaoxiang
# ✅ Правильно
add: (a: Int, b: Int) -> Int = a + b

# ❌ Неправильно
add: (a: Int, b: Int) -> Int = a + b            # Типы параметров отсутствуют
identity: (T: Type) -> ((x: T) -> T) = x                # Типы параметров отсутствуют
```

#### Правило 2: тип возврата может выводиться, но рекомендуется аннотация
```yaoxiang
# ✅ Рекомендуется: полная аннотация
get_num: () -> Int = () => 42

# ✅ Допустимо: вывод типа возврата
get_num: () = () => 42

# ✅ Пустая функция выводится как Void
empty: () = () => {}
```

#### Правило 3: аннотации типов внутри Lambda являются временными
```yaoxiang
# ✅ Правильно: полагаться на типы в объявлении
add: (a: Int, b: Int) -> Int = a + b

# ⚠️ Возможно, но не рекомендуется: дублирование аннотаций в Lambda
add: (Int, Int) -> Int = (a: Int, b: Int) => a + b

# ❌ Неправильно: отсутствует аннотация в объявлении
add: (a: Int, b: Int) -> Int = a + b
```

#### Правило 4: старый синтаксис следует той же философии
```yaoxiang
# Старый синтаксис также должен по возможности предоставлять информацию о типах в позиции объявления
# Хотя формат отличается, идея та же:
# - Строка объявления содержит основную информацию о типах
# - Тело реализации относительно лаконично
add(Int, Int) -> Int = (a, b) => a + b
```

### 9.4 Связь конвенции с type inference

```yaoxiang
# Конвенция не препятствует type inference, а направляет его

# 1. Полная аннотация (без вывода)
add: (a: Int, b: Int) -> Int = a + b

# 2. Частичный вывод (объявление предоставляет типы параметров)
add: (Int, Int) = (a, b) => a + b  # Тип возврата выводится

# 3. Вывод для пустой функции
empty: () = () => {}  # Выводится как () -> Void
```

### 9.5 Преимущества конвенции для AI-реализации

**Процесс AI-генерации кода:**

1. **Чтение требования** → генерация объявления
   ```
   Требование: функция сложения
   Генерация: add: (Int, Int) -> Int = (a, b) => ???
   ```

2. **Заполнение реализации** → без необходимости анализа типов
   ```
   Реализация: add: (a: Int, b: Int) -> Int = a + b
   ```

3. **Изменение типов** → правка только объявления
   ```
   Изменение: add: (Float, Float) -> Float = (a, b) => a + b
   Реализация: (a, b) => a + b  остаётся без изменений
   ```

**Сравнение с обработкой AI без конвенции:**
```
Требование: функция сложения
AI необходимо:
  1. Вывести типы параметров
  2. Вывести тип возврата
  3. Сгенерировать тело реализации
  4. Проверить согласованность
  5. Обрабатывать сложные обновления при изменении типов

Результат: сложнее, легче допустить ошибку
```

### 9.6 Философский смысл конвенции

Эта конвенция отражает основные идеи YaoXiang:

- **Объявление — это документация**: строка объявления и есть полная документация функции
- **Тип — это контракт**: информация о типах — контракт между вызывающим и реализующим
- **Логика — это реализация**: тело реализации заботится только о «что делать», а не о «какого типа»
- **Инструмент — это помощник**: система типов, AI-инструменты могут работать на основе ясных объявлений

### 9.7 Сравнение практического применения

#### Полный пример: модуль калькулятора

```yaoxiang
# === Рекомендуемый подход: конвенция централизации типов ===

# Объявления модуля
pub add: (a: Int, b: Int) -> Int = a + b
pub multiply: (a: Int, b: Int) -> Int = a * b

# Функция высшего порядка
pub apply_twice: (f: Int -> Int, x: Int) -> Int = f(f(x))

# Каррированная функция
pub make_adder: (x: Int) -> (Int) -> Int = y => x + y

# Дженерик-функция
pub map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) = case xs of
  [] => []
  (x :: rest) => f(x) :: map(f, rest)

# Определение типа
type Point = { x: Float, y: Float }
pub distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# === Нерекомендуемый подход: разбросанные типы ===

# Типы параметров в Lambda
add: (a: Int, b: Int) -> Int = a + b
multiply = (a: Int, b: Int) => a * b

# Типы функции высшего порядка разбросаны
apply_twice = (f: (Int) -> Int, x: Int) => f(f(x))

# Каррированные типы разбросаны
make_adder = (x: Int) => (y: Int) => x + y

# Дженерик-типы разбросаны
map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) = case xs of
  [] => []
  (x :: rest) => f(x) :: map(f, rest)
```

#### Сравнение поддержки кода

```yaoxiang
# Требование: изменить add с Int на Float

# === Рекомендуемый подход: изменить нужно только строку объявления ===
# Было
add: (a: Int, b: Int) -> Int = a + b

# Стало
add: (a: Float, b: Float) -> Float = a + b
#              ↑↑↑↑↑↑↑↑↑          ↑↑↑↑↑↑↑
#              Изменено объявление          Тело реализации без изменений

# === Нерекомендуемый подход: нужно изменить в нескольких местах ===
# Было
add: (a: Int, b: Int) -> Int = a + b

# Стало
add: (a: Float, b: Float) -> Float = a + b
#     ↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑
#     Все типы параметров нужно изменить
```

#### Сравнение AI-ассистированного программирования

```yaoxiang
# Требование AI: реализовать функцию, вычисляющую манхэттенское расстояние между двумя точками

# === AI видит рекомендуемое написание ===
type Point = { x: Float, y: Float }
pub manhattan: (a: Point, b: Point) -> Float = ???  # AI сразу знает полную сигнатуру

# AI генерирует:
pub manhattan: (a: Point, b: Point) -> Float = {
    (a.x - b.x).abs() + (a.y - b.y).abs()
}

# === AI видит нерекомендуемое написание ===
type Point = { x: Float, y: Float }
pub manhattan = ???  # AI нужно выводить: типы параметров? тип возврата?

# AI может сгенерировать:
pub manhattan = (a: Point, b: Point) => Float => {
    (a.x - b.x).abs() + (a.y - b.y).abs()
}
# Или может допустить ошибку, потому что информация о типах неполная
```

### 9.8 Контрольный список реализации конвенции

При написании кода на YaoXiang можно использовать следующий контрольный список:

- [ ] Все объявления функций имеют полные аннотации типов в строке объявления
- [ ] Типы параметров указаны в объявлении, а не в Lambda
- [ ] Тип возврата по возможности аннотирован в объявлении
- [ ] Объявления переменных используют формат `name: Type = value`
- [ ] Тело Lambda остаётся лаконичным, без дублирования информации о типах
- [ ] Используется новый синтаксис, а не старый
- [ ] Сложные типы определяются через type, объявления остаются ясными
