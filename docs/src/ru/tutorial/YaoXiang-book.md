## 一、Обзор языка

### 1.1 Что такое YaoXiang?

YaoXiang（爻象） — экспериментальный универсальный язык программирования, чья философия дизайна уходит корнями в ключевые концепции «爻» (Яо) и «象» (Сян) из «Книги Перемен» (И-цзин). «爻» — это базовые символы, составляющие триграммы и гексаграммы, олицетворяющие изменения Инь и Ян; «象» — внешнее проявление сущности вещей, представляющее все явления и сущности.

YaoXiang интегрирует эту философскую мысль в систему типов языка программирования, выдвигая основополагающую концепцию **«всё есть тип»**. В мировоззрении YaoXiang:

- **Значения** — это экземпляры типов
- **Типы** сами являются экземплярами типов (мета-типы)
- **Функции** — это отображения входных типов в выходные
- **Модули** — это композиции пространств имён типов

### 1.2 Цели дизайна

Цели дизайна YaoXiang можно обобщить следующим образом:

| Цель | Описание |
|------|----------|
| **Единая абстракция типов** | Тип является абстракцией высшего уровня, что упрощает семантику языка |
| **Естественный опыт программирования** | Синтаксис в стиле Python, акцент на читаемости |
| **Безопасное управление памятью** | Модель владения в стиле Rust, без GC |
| **Незаметное асинхронное программирование** | Автоматическое управление асинхронностью без явного `await` |
| **Полная рефлексия типов** | Полностью доступная информация о типах во время выполнения |
| **AI-дружественный синтаксис** | Строго структурированный, легко обрабатываемый ИИ |

### 1.3 Позиционирование языка

| Измерение | Позиционирование |
|-----------|------------------|
| Парадигма | Мультипарадигменный (функциональный + императивный + объектно-ориентированный) |
| Система типов | Зависимые типы + параметрический полиморфизм |
| Управление памятью | Ownership (владение) + RAII (без GC) |
| Модель компиляции | AOT-компиляция (опционально JIT) |
| Целевые сценарии | Системное программирование, разработка приложений, AI-ассистированное программирование |

### 1.4 Примеры кода

```yaoxiang
// Автоматический вывод типов
x: Int = 42 // Явный тип
y = 42 // Выводится как Int
name = "YaoXiang" // Выводится как String

// По умолчанию неизменяемые
x: Int = 10
x = 20 // ❌ Ошибка компиляции! Неизменяемое

// Унифицированный синтаксис объявлений: идентификатор: Тип = выражение
add: (a: Int, b: Int) -> Int = a + b // Объявление функции
inc: (x: Int) -> Int = x + 1 // Функция с одним параметром

// Унифицированный синтаксис типов: конструктор — это тип
Point: Type = { x: Float, y: Float }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

// Незаметная асинхронность (spawn function)
fetch_data: (url: String) -> JSON spawn = {
 HTTP.get(url).json()
}

main: () -> Void = {
 // Конструирование значений: полностью совпадает с вызовом функции
 p = Point(3.0, 4.0)
 r = ok("success")

 data = fetch_data("https://api.example.com")
 // Автоматическое ожидание, без await
 print(data.name)
}

// Обобщённая функция
identity: (T: Type) -> ((x: T) -> T) = x

// Функция высшего порядка
apply: (f: (Int) -> Int, x: Int) -> Int = f(x)

// Каррирование
add_curried: (a: Int) -> (b: Int) -> Int = a + b
```

## 二、Основные возможности

### 2.1 Всё есть тип

Основная философия дизайна YaoXiang — **всё есть тип**. Это означает, что в YaoXiang:

1. **Значения — это экземпляры типов**: `42` — экземпляр типа `Int`
2. **Типы — это экземпляры типов**: `Int` — экземпляр метатипа `type`
3. **Функции — это отображения типов**: `add: (Int, Int) -> Int` — это функциональный тип
4. **Модули — это композиции типов**: модули — это пространства имён, содержащие функции и типы

```yaoxiang
// Значения — экземпляры типов
x: Int = 42

// Типы — экземпляры типов
MyList: type = List(Int)

// Функции — отображения между типами
add: (a: Int, b: Int) -> Int = a + b

// Модули — композиции типов (файлы как модули)
// Math.yx
pi: Float = 3.14159
sqrt: (x: Float) -> Float = { ... }
```

### 2.2 Математические абстракции

Система типов YaoXiang основана на теории типов и теории категорий и предоставляет:

- **Зависимые типы**: типы могут зависеть от значений
- **Обобщённое программирование**: параметризация типов
- **Композиция типов**: объединение, пересечение

```yaoxiang
// Зависимый тип: вектор фиксированной длины
Vector: (T: Type, n: Int) -> Type = vector(T, n)

// Объединение типов
Number: Type = { Int | Float }

// Пересечение типов
Printable: Type = printable(() -> String)
Serializable: Type = serializable(() -> String)
Versatile: Type = Printable & Serializable
```

### 2.3 Абстракции с нулевой стоимостью

YaoXiang гарантирует абстракции с нулевой стоимостью, то есть высокоуровневые абстракции не приводят к накладным расходам во время выполнения:

- **Мономорфизация**: обобщённые функции во время компиляции разворачиваются в конкретные версии
- **Инлайнинг**: простые функции автоматически инлайнятся
- **Стековое размещение**: мелкие объекты по умолчанию размещаются в стеке

```yaoxiang
// Разворачивание обобщений (мономорфизация)
identity: (T: Type) -> ((x: T) -> T) = x

// Использование
int_val = identity(42) // Разворачивается в identity(Int) -> Int
str_val = identity("hello") // Разворачивается в identity(String) -> String

// Никаких дополнительных расходов после компиляции
```

### 2.4 Естественный синтаксис

YaoXiang использует синтаксис в стиле Python, стремясь к читаемости и ощущению естественного языка:

```yaoxiang
// Автоматический вывод типов
x = 42
name = "YaoXiang"

// Лаконичное определение функций
greet: (name: String) -> String = "Hello, " + name

// Сопоставление с образцом
classify: (n: Int) -> String = {
 match n {
 0 -> "zero"
 1 -> "one"
 _ if n < 0 -> "negative"
 _ -> "many"
 }
}
```

### 2.5 Полная спецификация синтаксиса

YaoXiang использует унифицированный синтаксис объявлений: **идентификатор: Тип = выражение**. Одновременно предоставляется обратно совместимый старый синтаксис.

#### 2.5.1 Стратегия двух синтаксисов и конвенция централизации типов

Для баланса между инновациями и совместимостью YaoXiang поддерживает две формы синтаксиса, но применяет унифицированную **конвенцию централизованного аннотирования типов**.

**Сравнение форм синтаксиса:**

| Тип синтаксиса | Формат | Статус | Описание |
|----------------|--------|--------|----------|
| **Новый синтаксис (стандартный)** | `name: Type = Lambda` | ✅ Рекомендуется | Официальный стандарт, должен использоваться во всём новом коде |
| **Старый синтаксис (совместимость)** | `name(Types) -> Ret = Lambda` | ⚠️ Только совместимость | Сохранён для исторического кода, не рекомендуется для новых проектов |

**Основная конвенция: централизованное аннотирование типов**

YaoXiang применяет конвенцию дизайна **«приоритет объявления, централизация типов»**:

```yaoxiang
// ✅ Правильно: информация о типах унифицирована в строке объявления
add: (a: Int, b: Int) -> Int = a + b
// └─────────────────┘ └─────────────┘
// Полная сигнатура типа  Логика реализации

// ❌ Избегать: информация о типах разбросана в реализации
add: (a: Int, b: Int) -> Int = a + b
// └───────────────┘
// Типы смешаны с телом реализации
```

**Преимущества конвенции:**

1. **Синтаксическая согласованность**: все объявления следуют форме `идентификатор: Тип = выражение`
2. **Разделение объявления и реализации**: информация о типах очевидна, тело реализации сосредоточено на логике
3. **AI-дружественность**: ИИ достаточно прочитать строку объявления, чтобы понять полную сигнатуру функции
4. **Безопасность изменений**: изменение типа требует изменения только объявления, не затрагивая тело реализации
5. **Каррирование-дружественность**: поддерживает ясные каррированные сигнатуры типов

**Рекомендации по выбору**:
- **Новые проекты**: должны использовать новый синтаксис + конвенцию централизации типов
- **Мигрируемые проекты**: постепенно переходите на новый синтаксис и конвенцию централизации типов
- **Поддержка старого кода**: можно продолжать использовать старый синтаксис, но рекомендуется применять конвенцию централизации типов

#### 2.5.2 Базовый синтаксис объявлений

```yaoxiang
// === Новый синтаксис (рекомендуется) ===
// Все объявления следуют форме: идентификатор: Тип = выражение

// Объявление переменных
x: Int = 42
name: String = "YaoXiang"
mut counter: Int = 0

// Объявление функций
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1
getAnswer: () -> Int = 42
log: (msg: String) -> Void = print(msg)

// === Старый синтаксис (совместимость) ===
// Только для функций, формат: name(Types) -> Ret = Lambda
add(Int, Int) -> Int = (a, b) => a + b
square(Int) -> Int = (x) => x * x
empty() -> Void = () => {}
getRandom() -> Int = () => 42
```

#### 2.5.3 Синтаксис функциональных типов

```
Функциональный тип ::= '(' список типов параметров ')' '->' тип возврата
                     | тип параметра '->' тип возврата         # Сокращение для одного параметра

список типов параметров ::= [тип (',' тип)*]
тип возврата ::= тип | функциональный тип | 'Void'

# Функциональный тип — это гражданин первого класса, может быть вложенным
# Функциональный тип высшего порядка ::= '(' функциональный тип ')' '->' тип возврата
```

| Пример | Значение |
|--------|----------|
| `Int -> Int` | Функциональный тип с одним параметром |
| `(Int, Int) -> Int` | Функциональный тип с двумя параметрами |
| `() -> Void` | Функциональный тип без параметров |
| `(Int -> Int) -> Int` | Функция высшего порядка: принимает функцию, возвращает Int |
| `Int -> Int -> Int` | Каррированная функция (правоассоциативна) |

#### 2.5.4 Синтаксис обобщений (только для параметров типов)

```yaoxiang
// Обобщённая функция: <параметр типа> префикс
identity: (T: Type) -> ((x: T) -> T) = x
map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) = case xs of
 [] => []
 (x :: rest) => f(x) :: map(f, rest)

// Обобщённый тип
List: (T: Type) -> Type
```

#### 2.5.5 Синтаксис лямбда-выражений

```
Lambda ::= '(' список параметров ')' '=>' выражение
         | параметр '=>' выражение                # Сокращение для одного параметра

список параметров ::= [параметр (',' параметр)*]
параметр ::= идентификатор [':' тип]               # Опциональная аннотация типа
```

| Пример | Значение | Описание |
|--------|----------|----------|
| `(a, b) => a + b` | Лямбда с несколькими параметрами | Используется с объявлением:<br>`add: (Int, Int) = (a, b) => a + b` |
| `x => x + 1` | Сокращение для одного параметра | Используется с объявлением:<br>`inc: Int = x => x + 1` |
| `(x: Int) => x + 1` | С аннотацией типа | Только для временного использования внутри Lambda |
| `() => 42` | Лямбда без параметров | Используется с объявлением:<br>`get: () = () => 42` |

**Примечание**: аннотация типа в лямбда-выражении `(x: Int) => ...` является **временной, локальной** и используется в основном для:
- Когда внутри лямбды нужна информация о типе
- При использовании совместно с синтаксисом объявления (тип уже задан в объявлении)
- Не должна быть основным способом объявления типов

#### 2.5.6 Полные примеры

```yaoxiang
// === Базовые объявления функций ===

// Базовая функция (новый синтаксис)
add: (a: Int, b: Int) -> Int = a + b

// Функция с одним параметром (две формы)
inc: (x: Int) -> Int = x + 1
inc2: (x: Int) -> Int = x + 1

// Функция без параметров
getAnswer: () -> Int = 42

// Функция без возвращаемого значения
log: (msg: String) -> Void = print(msg)

// === Рекурсивные функции ===
// Рекурсия естественно поддерживается в лямбдах
fact: (n: Int) -> Int =
 if n <= 1 then 1 else n * fact(n - 1)

// === Функции высшего порядка и присваивание функциональных типов ===

// Функциональный тип как гражданин первого класса
IntToInt: Type = (Int) -> Int
IntBinaryOp: Type = (Int, Int) -> Int

// Объявление функции высшего порядка
applyTwice: (f: IntToInt, x: Int) -> Int = f(f(x))

// Каррированная функция
addCurried: (a: Int) -> (b: Int) -> Int = a + b

// Композиция функций
compose: (f: Int -> Int, g: Int -> Int) -> (x: Int) -> Int =
 f(g(x))

// Функция, возвращающая функцию
makeAdder: (x: Int) -> (y: Int) -> Int =
 x + y

// === Обобщённые функции ===

// Обобщённая функция
identity: (T: Type) -> ((x: T) -> T) = x

// Обобщённая функция высшего порядка
map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) =
 case xs of
 [] => []
 (x :: rest) => f(x) :: map(f, rest)

// Обобщённый функциональный тип
Transformer: Type = (A: Type, B: Type) -> ((A) -> B)

// Использование обобщённого типа
applyTransformer: (A: Type, B: Type) -> ((f: Transformer(A, B), x: A) -> B) =
 f(x)

// === Примеры сложных типов ===

// Вложенный функциональный тип
higherOrder: (A: Type) -> ((f: (A) -> Int) -> (A) -> Int) =
 f => x => f(x) + 1

// Многопараметрическая функция высшего порядка
zipWith: (A: Type, B: Type, C: Type) -> ((f: (A, B) -> C, xs: List(A), ys: List(B)) -> List(C)) =
 case (xs, ys) of
 ([], _) => []
 (_, []) => []
 (x::xs', y::ys') => f(x, y) :: zipWith(f, xs', ys')

// Псевдонимы функциональных типов
Predicate: (T: Type) -> Type = { apply: (T) -> Bool }
Mapper: Type = (A: Type, B: Type) -> ((A) -> B)
Reducer: Type = (A: Type, B: Type) -> ((B, A) -> B)

// === Примеры старого синтаксиса (только обратная совместимость) ===
// Не рекомендуется для нового кода

mul(Int, Int) -> Int = (a, b) => a * b // Несколько параметров
square(Int) -> Int = (x) => x * x // Один параметр
empty() -> Void = () => {} // Без параметров
get_random() -> Int = () => 42 // С возвращаемым значением

// Эквивалентный новый синтаксис (рекомендуется)
mul: (a: Int, b: Int) -> Int = a * b
square: (x: Int) -> Int = x * x
empty: () -> Void = {}
get_random: () -> Int = 42
```

#### 2.5.7 Правила синтаксического разбора

**Приоритеты разбора типов:**

| Приоритет | Тип | Описание |
|-----------|-----|----------|
| 1 (высший) | Применение обобщения `List(T)` | Левоассоциативно |
| 2 | Скобки `(T)` | Изменяют ассоциативность |
| 3 | Функциональный тип `->` | Правоассоциативно |
| 4 (низший) | Базовые типы `Int, String` | Атомарные типы |

**Примеры разбора типов:**

```yaoxiang
// (A -> B) -> C -> D
// Разбирается как: ((A -> B) -> (C -> D))

// A -> B -> C
// Разбирается как: (A -> (B -> C)) // Правоассоциативно

// (Int -> Int) -> Int
// Разбирается как: принимает функцию, возвращает Int -> Int

// List<Int -> Int>
// Разбирается как: элемент типа List — это Int -> Int
```

**Примеры разбора лямбд:**

```yaoxiang
// a => b => a + b
// Разбирается как: a => (b => (a + b)) // Правоассоциативно, каррирование

// (a, b) => a + b
// Разбирается как: принимает два параметра, возвращает a + b
```

#### 2.5.8 Правила вывода типов

YaoXiang применяет стратегию **двухуровневой обработки**: на уровне разбора допускается снисходительность, на уровне проверки типов — строгий вывод.

**Правила уровня разбора:**
- Парсер только проверяет синтаксическую структуру, не выполняет вывод типов
- В объявлениях без аннотаций типов поле аннотации типа равно `None`
- Все объявления, соответствующие базовой синтаксической структуре, проходят разбор
- **Ключевой момент**: `add: (a: Int, b: Int) -> Int = a + b` на уровне разбора является **допустимым**

**Правила уровня проверки типов:**
- Проверяется семантическая корректность, включая полноту типов
- **Параметры должны иметь аннотации типов**: это обязательное требование
- Тип возврата может быть выведен, но типы параметров должны быть объявлены явно

**Полные правила вывода типов:**

| Сценарий | Вывод параметров | Вывод возврата | Результат разбора | Результат проверки типов | Рекомендация |
|----------|------------------|----------------|-------------------|--------------------------|--------------|
| **Стандартная функция** | Используются аннотированные типы | Используется аннотированный тип | ✅ | ✅ | ⭐⭐⭐⭐⭐ |
| `add: (a: Int, b: Int) -> Int = a + b` | | | | | |
| **Частичный вывод** | Используются аннотированные типы | Выводится из выражения | ✅ | ✅ | ⭐⭐⭐⭐ |
| `add: (Int, Int) = (a, b) => a + b` | | | | | |
| `inc: (x: Int) -> Int = x + 1` | | | | | |
| `get: () = () => 42` | | | | | |
| **Частичный вывод в старом синтаксисе** | Используются аннотированные типы | Выводится из выражения | ✅ | ✅ | ⭐⭐⭐ (совместимость) |
| `add(Int, Int) = (a, b) => a + b` | | | | | |
| `square(Int) = (x) => x * x` | | | | | |
| **Параметры без аннотаций** | **Невозможно вывести** | - | ✅ | ❌ Ошибка | ❌ Запрещено |
| `add: (a: Int, b: Int) -> Int = a + b` | | | | | |
| `identity: (T: Type) -> ((x: T) -> T) = x` | | | | | |
| **Блок без аннотации возврата** | - | Выводится из содержимого блока | ✅ | ✅ | ⭐⭐⭐⭐ |
| `main = () => {}` | | | | | |
| `get = () => { return 42; }` | | | | | |
| **Блок кода без return** | - | По умолчанию `Void` | ✅ | ✅ Корректно | ✅ Корректно |
| `add: (a: Int, b: Int) -> Int = { return a + b }` | | | | | |

**Подробные правила вывода:**

```
Уровень разбора: только синтаксическая структура
├── Структура корректна → пройдено
└── Структура некорректна → ошибка

Уровень проверки типов: проверка семантики
├── Вывод типов параметров
│   ├── Параметр имеет аннотацию типа → использовать аннотированный тип ✅
│   ├── Параметр без аннотации типа → отклонить ❌
│   └── Параметры лямбды должны иметь аннотации → обязательное требование
│
├── Вывод типа возврата
│   ├── Есть return expr → вывести из expr ✅
│   ├── Нет return, есть выражение → вывести из выражения ✅
│   ├── Блок кода без return → по умолчанию Void ✅
│   └── Невозможно вывести → отклонить ❌
│
└── Полностью невозможно вывести → отклонить ❌
```

**Примечание**: когда в блоке кода нет `return`, по умолчанию возвращается `Void`. Например:
- `() => { 42 }` выводится как `() -> Void` (блок без return, по умолчанию возвращает Void)
- `() => { return 42 }` выводится как `() -> Int` (есть return, выводится из return)
- `() => 42` выводится как `() -> Int` (форма выражения, возвращает значение напрямую)

**Примеры вывода:**

```yaoxiang
// === Успешный вывод ===

// Стандартная форма
main: () -> Void = () => {} // Полная аннотация
num: () -> Int = () => 42 // Полная аннотация
inc: (x: Int) -> Int = x + 1 // Сокращение для одного параметра

// Частичный вывод (новый синтаксис)
add: (Int, Int) = (a, b) => a + b // Параметры аннотированы, возвращаемый тип выводится
square: (x: Int) -> Int = x * x // Параметры аннотированы, возвращаемый тип выводится
get_answer: () = () => 42 // Параметры аннотированы (пустые), возвращаемый тип выводится

// Частичный вывод (старый синтаксис, совместимость)
add2(Int, Int) = (a, b) => a + b // Параметры аннотированы, возвращаемый тип выводится
square2(Int) = (x) => x * x // Параметры аннотированы, возвращаемый тип выводится

// Вывод из return
fact: Int -> Int = (n) => {
 if n <= 1 { return 1 }
 return n * fact(n - 1)
}

// === Сбой вывода ===

// Параметры не могут быть выведены (разбор пройден, проверка типов не пройдена)
add: (a: Int, b: Int) -> Int = a + b // ✗ Параметры без типа
identity: (T: Type) -> ((x: T) -> T) = x // ✗ Параметры без типа

// Блок кода без return
no_return = (x: Int) => { x } // ✓ Выводится как Void (блок без return по умолчанию возвращает Void)

// Полностью невозможно вывести
bad_fn: (T: Type) -> ((x: T) -> T) = x // ✗ И параметры, и возвращаемое значение не могут быть выведены
```

#### 2.5.9 Старый синтаксис (обратная совместимость)

YaoXiang обеспечивает поддержку старого синтаксиса для совместимости с историческим кодом, **не рекомендуется для нового кода**.

```
Старый синтаксис ::= идентификатор '(' [список типов параметров] ')' '->' тип возврата '=' Lambda
```

| Характеристика | Стандартный синтаксис | Старый синтаксис |
|----------------|----------------------|------------------|
| Формат объявления | `name: Type = ...` | `name(Types) -> Type = ...` |
| Позиция типов параметров | В аннотации типа | В скобках после имени функции |
| Пустые параметры | Должны быть записаны как `()` | Можно опустить `()` |
| **Рекомендация** | ✅ **Официально рекомендуется** | ⚠️ **Только обратная совместимость** |
| **Сценарий использования** | Весь новый код | Поддержка исторического кода |

**Причины нерекомендации:**
1. **Стоимость обучения**: несовместим со стандартным синтаксисом, увеличивает сложность языка
2. **Согласованность**: позиции типов параметров не унифицированы (один в аннотации типа, другой после имени функции)
3. **Стоимость поддержки**: парсеру необходимо обрабатывать две формы
4. **AI-дружественность**: увеличивает сложность понимания и генерации кода ИИ

**Рекомендации по миграции:**
```yaoxiang
// Старый код (не рекомендуется)
mul(Int, Int) -> Int = (a, b) => a * b
square(Int) -> Int = (x) => x * x
empty() -> Void = () => {}

// Новый код (рекомендуется)
mul: (Int, Int) -> Int = (a, b) => a * b
square: (Int) -> Int = (x) => x * x
empty: () -> Void = () => {}
```

## 四、Управление памятью

### 4.1 Модель владения (Ownership)

YaoXiang использует **модель владения** для управления памятью, где каждое значение имеет единственного владельца:

```yaoxiang
// === По умолчанию Move (без копирования) ===
p: Point = Point(1.0, 2.0)
p2 = p // Move, передача владения, p становится недействительным

// === Ключевое слово ref = Arc (безопасное разделение) ===
shared = ref p // Arc, потокобезопасный

spawn(() => print(shared.x)) // ✅ Безопасно

// === clone() для явного копирования ===
p3 = p.clone() // p и p3 независимы
```

### 4.2 Семантика Move (по умолчанию)

```yaoxiang
// Присваивание = Move (без копирования)
p: Point = Point(1.0, 2.0)
p2 = p // Move, p становится недействительным

// Передача в функцию = Move
process: (p: Point) -> Void = {
 // Владение p передаётся внутрь
}

// Возвращаемое значение = Move
create: () -> Point = {
 p = Point(1.0, 2.0)
 return p // Move, передача владения
}
```

### 4.3 Ключевое слово ref (Arc)

```yaoxiang
// Ключевое слово ref создаёт Arc (подсчёт ссылок)
p: Point = Point(1.0, 2.0)
shared = ref p // Arc, потокобезопасный

spawn(() => print(shared.x)) // ✅ Безопасно

// Arc автоматически управляет временем жизни
// Когда shared выходит из области видимости, счётчик обнуляется и память освобождается
```

### 4.4 clone() — явное копирование

```yaoxiang
// Используйте clone(), когда нужно сохранить исходное значение
p: Point = Point(1.0, 2.0)
p2 = p.clone() // p и p2 независимы

p.x = 0.0 // ✅
p2.x = 0.0 // ✅ Не влияют друг на друга
```

### 4.5 Блоки unsafe (системный уровень)

```yaoxiang
// Сырые указатели могут использоваться только в блоках unsafe
p: Point = Point(1.0, 2.0)

unsafe {
 ptr: *Point = &p // Сырой указатель
 (*ptr).x = 0.0 // Безопасность гарантируется пользователем
}
```

### 4.6 RAII

```yaoxiang
// RAII — автоматическое освобождение
with_file: (path: String) -> String = {
 file = File.open(path) // Автоматическое открытие
 content = file.read_all()
 // В конце функции file автоматически закрывается
 content
}
```

### 4.7 Ограничения Send / Sync

| Ограничение | Семантика | Описание |
|-------------|-----------|----------|
| **Send** | Можно безопасно передавать между потоками | Значение может быть перемещено в другой поток |
| **Sync** | Можно безопасно разделять между потоками | Неизменяемые ссылки могут быть разделены между потоками |

```yaoxiang
// ref T автоматически удовлетворяет Send + Sync (Arc потокобезопасен)
p: Point = Point(1.0, 2.0)
shared = ref p

spawn(() => print(shared.x)) // ✅ Arc потокобезопасен

// Сырой указатель *T не удовлетворяет Send/Sync
unsafe {
 ptr: *Point = &p // Может использоваться только в одном потоке
}
```

| Проверка заимствований | ref = Arc как замена |
| Синтаксис заимствования `&T` | Используется семантика Move |

## 五、Асинхронное программирование и параллелизм

> «Все вещи свершаются совместно, я наблюдаю их возвращение.» — «И-цзин», гексаграмма Фу (Возврат)
>
> YaoXiang использует **модель spawn** — парадигму незаметной асинхронной параллельности, основанную на **ленивом вычислении**. Основная идея дизайна: **позволить разработчикам описывать логику в синхронном, последовательном стиле мышления, а среда выполнения языка автоматически и эффективно параллельно выполняет вычислительные единицы, как «все вещи, свершающиеся совместно», унифицируя координацию в конце**.

> Подробности см. в [«Белой книге модели spawn»](YaoXiang-async-whitepaper.md) и [документе реализации асинхронности](YaoXiang-async-implementation.md).

### 5.1 Основные концепции модели spawn

#### 5.1.1 Граф spawn: сцена совместного свершения

Все программы во время компиляции преобразуются в **ориентированный ациклический вычислительный граф (DAG)**, называемый **графом spawn**. Узлы представляют вычисления выражений, рёбра — зависимости данных. Этот граф ленив — узлы вычисляются только тогда, когда их выход **действительно необходим**.

```yaoxiang
// Компилятор автоматически строит граф spawn
fetch_user: spawn () -> User = (id) => { ... }
fetch_posts: spawn (User) -> Posts = (user) => { ... }

main:() -> Void = () => {
 user = fetch_user(1) // Узел A (Async(User))
 posts = fetch_posts(user) // Узел B (Async(Posts)), зависит от A

 // Узел C нуждается в результатах A и B
 print(posts.title) // Автоматическое ожидание: сначала завершение A и B
}
```

#### 5.1.2 Значения spawn: Async(T)

Любой вызов функции, помеченной как `spawn fn`, немедленно возвращает значение типа `Async(T)`, называемое **значением spawn**. Это лёгкий прокси, который не является фактическим результатом, а представляет **будущее значение в процессе spawn**.

**Основные характеристики**:
- **Прозрачность типов**: `Async(T)` в системе типов является подтипом `T` и может использоваться в любом контексте, ожидающем `T`
- **Автоматическое ожидание**: когда программа выполняет операцию, требующую конкретного значения типа `T`, среда выполнения автоматически приостанавливает текущую задачу и ожидает завершения вычисления
- **Нулевое заражение**: асинхронный и синхронный код не отличаются по синтаксису и сигнатурам типов

```yaoxiang
// Пример использования значений spawn
fetch_data: spawn (String) -> JSON = (url) => { ... }

main: () -> Void = () => {
 data = fetch_data("url") // Async(JSON)

 // Async(JSON) может использоваться напрямую как JSON
 // Автоматическое ожидание происходит при доступе к полю
 print(data.name) // Эквивалентно data.await().name
}
```

### 5.2 Система синтаксиса spawn

Ключевое слово `spawn` имеет тройную семантику и является единственным мостом между синхронным мышлением и асинхронной реализацией:

| Официальный термин | Синтаксическая форма | Семантика | Поведение во время выполнения |
|--------------------|----------------------|-----------|--------------------------------|
| **spawn function** | `spawn fn` | Определяет вычислительную единицу, участвующую в spawn-исполнении | Её вызов возвращает `Async(T)` |
| **spawn block** | `spawn { a(), b() }` | Явно объявленная параллельная область | Задачи внутри блока принудительно выполняются параллельно |
| **spawn loop** | `spawn for x in xs { ... }` | Парадигма параллелизма данных | Тело цикла выполняется параллельно для всех элементов |

#### 5.2.1 spawn function (spawn функция)

```yaoxiang
// Использование spawn для пометки spawn-функции
// Синтаксис полностью совпадает с обычной функцией, без дополнительной нагрузки

fetch_api: spawn (String) -> JSON = (url) => {
 response = HTTP.get(url)
 JSON.parse(response.body)
}

// Вложенные spawn-вызовы
process_user: (Int) -> Report = (user_id) => {
 user = fetch_user(user_id) // Async(User)
 profile = fetch_profile(user) // Async(Profile), зависит от user
 generate_report(user, profile) // Зависит от profile
}
```

#### 5.2.2 spawn block (spawn блок)

```yaoxiang
// spawn { } - явная параллельная конструкция
// Все выражения внутри блока выполняются как независимые задачи параллельно
```

### 5.5 Разделение между задачами: ключевое слово ref

YaoXiang не требует ручных примитивов синхронизации, таких как трейты `Send`/`Sync`, `Mutex`, `RwLock` и т.д. Владение + `ref` автоматически обрабатывают безопасность параллелизма.

#### 5.5.1 ref: разделение между областями видимости

```yaoxiang
// ref — единственный способ разделения между областями видимости
// Компилятор автоматически выбирает Rc (одна задача) или Arc (между задачами)

data = load_data()
shared = ref data // Компилятор автоматически выбирает реализацию

result = spawn {
 process_a(shared), // Общая ссылка, между задачами → Arc
 process_b(shared) // Общая ссылка
}
```

#### 5.5.2 Автоматическая оптимизация компилятора

```
Анализ потока данных для ref:

Не уходит в другие задачи → Rc (неатомарный подсчёт ссылок, низкие расходы)
Уходит в другие задачи     → Arc (атомарный подсчёт ссылок, потокобезопасный)
```

Пользователю не нужно заботиться о том, что находится под капотом — Rc или Arc. `ref` — это общее владение, и этого достаточно.

## 七、Привязка методов и каррирование

YaoXiang использует **чисто функциональный дизайн**, реализуя бесшовные вызовы методов и каррирование через продвинутый механизм привязки, без необходимости вводить ключевые слова `struct`, `class` и т.д.

### 7.1 Определение базовых функций

Все операции реализуются через обычные функции, при этом первый параметр по соглашению является субъектом операции:

```yaoxiang
// === Point.yx (модуль) ===

// Унифицированный синтаксис: конструктор — это тип
Point: Type = { x: Float, y: Float }

// Базовая функция: первый параметр — субъект операции
distance: (a: Point, b: Point) -> Float = {
 dx = a.x - b.x
 dy = a.y - b.y
 (dx * dx + dy * dy).sqrt()
}

add: (a: Point, b: Point) -> Point = {
 Point(a.x + b.x, a.y + b.y)
}

scale: (p: Point, s: Float) -> Point = {
 Point(p.x * s, p.y * s)
}

// Более сложные функции
distance_with_scale: (s: Float, p1: Point, p2: Point) -> Float = {
 dx = (p1.x - p2.x) * s
 dy = (p1.y - p2.y) * s
 (dx * dx + dy * dy).sqrt()
}
```

### 7.2 Базовая привязка методов

#### 7.2.1 Автоматическая привязка (в стиле MoonBit)

YaoXiang поддерживает автоматическую привязку на основе пространств имён, **без каких-либо дополнительных объявлений**:

```yaoxiang
// === Point.yx ===

Point: Type = { x: Float, y: Float }

// Базовая функция
distance: (a: Point, b: Point) -> Float = {
 dx = a.x - b.x
 dy = a.y - b.y
 (dx * dx + dy * dy).sqrt()
}

// === main.yx ===

use Point

main: () -> Void = {
 p1 = Point(3.0, 4.0)
 p2 = Point(1.0, 2.0)

 // ✅ Автоматическая привязка: прямой вызов метода
 result = p1.distance(p2) // Разбирается как distance(p1, p2)
}
```

**Правила автоматической привязки**:
- Функции, определённые внутри модуля
- Если тип первого параметра совпадает с именем модуля
- Тогда автоматически поддерживается синтаксис вызова метода

#### 7.2.2 Опция без привязки (поведение по умолчанию)

```yaoxiang
// === Vector.yx ===

Vector: Type = Vector(x: Float, y: Float, z: Float)

// Внутренняя вспомогательная функция, без автоматической привязки
dot_product_internal: (a: Vector, b: Vector) -> Float = {
 a.x * b.x + a.y * b.y + a.z * b.z
}

// === main.yx ===

use Vector

main: () -> Void = {
 v1 = Vector(1.0, 0.0, 0.0)
 v2 = Vector(0.0, 1.0, 0.0)

 // ❌ Невозможно привязать: не-pub функция не привязывается автоматически
 // v1.dot_product_internal(v2) // Ошибка компиляции!

 // ✅ Необходимо вызывать напрямую (невидимо за пределами модуля)
}
```

### 7.3 Позиционный синтаксис привязки

YaoXiang предоставляет **самый элегантный синтаксис привязки**, использующий позиционные маркеры `[n]` для точного управления позицией привязки:

#### 7.3.1 Базовый синтаксис

```yaoxiang
// === Point.yx ===

Point: Type = { x: Float, y: Float }

// Базовая функция
distance: (a: Point, b: Point) -> Float = {
 dx = a.x - b.x
 dy = a.y - b.y
 (dx * dx + dy * dy).sqrt()
}
add: (a: Point, b: Point) -> Point = {
 Point(a.x + b.x, a.y + b.y)
}
scale: (p: Point, s: Float) -> Point = {
 Point(p.x * s, p.y * s)
}

// Синтаксис привязки: Type.method = func[position]
// Означает: при вызове метода вызывающий объект привязывается к параметру [position] функции func

Point.distance = distance[0] // Привязка к 1-му параметру
Point.add = add[0] // Привязка к 1-му параметру
Point.scale = scale[0] // Привязка к 1-му параметру
```

**Семантический разбор**:
- `Point.distance = distance[0]`
  - Функция `distance` имеет два параметра: `distance(Point, Point)`
  - `[0]` означает, что вызывающий объект привязывается к 1-му параметру
  - Использование: `p1.distance(p2)` → `distance(p1, p2)`

#### 7.3.2 Многопозиционная совместная привязка

```yaoxiang
// === Math.yx ===

// Функция: scale, point1, point2, extra1, extra2
calculate: (scale: Float, a: Point, b: Point, x: Float, y: Float) -> Float = { ... }

// === Point.yx ===

Point: Type = { x: Float, y: Float }

// Привязка нескольких позиций
Point.calc1 = calculate[1, 2] // Привязка scale и point1
Point.calc2 = calculate[1, 3] // Привязка scale и point2
Point.calc3 = calculate[2, 3] // Привязка point1 и point2

// === main.yx ===

use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

// 1. Привязка [1,2] - остаются 3,4,5
f1 = p1.calc1(2.0) // Привязка scale=2.0, point1=p1
// f1 теперь требует p2, x, y
result1 = f1(p2, 10.0, 20.0) // calculate(2.0, p1, p2, 10.0, 20.0)

// 2. Привязка [1,3] - остаются 2,4,5
f2 = p2.calc2(2.0) // Привязка scale=2.0, point2=p2
// f2 теперь требует point1, x, y
result2 = f2(p1, 10.0, 20.0) // calculate(2.0, p1, p2, 10.0, 20.0)

// 3. Привязка [2,3] - остаются 1,4,5
f3 = p1.calc3(p2) // Привязка point1=p1, point2=p2
// f3 теперь требует scale, x, y
result3 = f3(2.0, 10.0, 20.0) // calculate(2.0, p1, p2, 10.0, 20.0)
```

#### 7.3.3 Порядок заполнения оставшихся параметров

**Ключевое правило**: после привязки оставшиеся параметры заполняются в **порядке исходной функции**, пропуская привязанные позиции.

```yaoxiang
// Допустим, функция: func(p1, p2, p3, p4, p5)

// Привязка 1-го и 3-го параметров
Type.method = func[1, 3]

// При вызове:
method(p2_value, p4_value, p5_value)

// Отображается в:
func(p1_bound, p2_value, p3_bound, p4_value, p5_value)
// Оставшиеся параметры: 2,4,5 заполняются в исходном порядке
```

#### 7.3.4 Преимущества проверки типов

```yaoxiang
// ✅ Допустимая привязка
Point.distance = distance[0] // distance(Point, Point)
Point.calc = calculate[1, 2] // calculate(scale, Point, Point, ...)

// ❌ Недопустимая привязка (компилятор сообщает об ошибке)
Point.wrong = distance[5] // 5-й параметр не существует
Point.wrong = distance[0] // Параметры нумеруются с 1
Point.wrong = distance[1, 2, 3, 4] // Превышает количество параметров функции
```

### 7.4 Тонкий контроль привязки каррирования

```yaoxiang
// === Math.yx ===

distance_with_scale: (scale: Float, a: Point, b: Point) -> Float = { ... }

// === Point.yx ===

Point: Type = { x: Float, y: Float }

// Стратегия привязки: гибкое управление каждой позицией

Point.distance_scaled = distance_with_scale[2] // Привязка ко 2-му параметру

// === main.yx ===

p2 = Point(1.0, 2.0)

// 1. Базовая автоматическая привязка

// 2. Привязка к другой позиции
f = p1.distance_scaled(2.0) // Привязка 2-го параметра, остаются 1-й и 3-й
result = f(p2) // distance_with_scale(2.0, p1, p2)

// 3. Цепочечная привязка
d2 = p1.distance(p2).distance_scaled(2.0) // Цепочечный вызов
```

### 7.6 Область видимости и правила привязки

#### 7.6.1 Роль pub

```yaoxiang
// === Point.yx ===

Point: Type = { x: Float, y: Float }

// Не-pub функция
internal_distance: (a: Point, b: Point) -> Float = {
 dx = a.x - b.x

}

// pub функция
pub distance: (a: Point, b: Point) -> Float = {
 dx = a.x - b.x
 dy = a.y - b.y
 (dx * dx + dy * dy).sqrt()
}

// === main.yx ===

use Point

// Автоматическая привязка действует только для pub функций
p1.distance(p2) // ✅ distance — pub, может быть привязана автоматически
// p1.internal_distance(p2) // ❌ Не pub, не может быть привязана
```

#### 7.6.2 Механизм автоматической привязки pub

Функции, объявленные с `pub`, автоматически привязываются компилятором к типам, определённым в том же файле:

```yaoxiang
// === Point.yx ===

Point: Type = { x: Float, y: Float }

// Использование объявления pub, компилятор автоматически привязывает
pub distance: (p1: Point, p2: Point) -> Float = {
 dx = p1.x - p2.x
 dy = p1.y - p2.y
 (dx * dx + dy * dy).sqrt()
}

pub translate: (self: Point, dx: Float, dy: Float) -> Point = {
 Point(self.x + dx, self.y + dy)
}

// Компилятор автоматически выводит и выполняет привязку:
// Point.distance = distance[0]
// Point.translate = translate[0]

// === main.yx ===

use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

// ✅ Функциональный вызов
d = distance(p1, p2)

// ✅ Синтаксический сахар ООП (автоматическая привязка)
d2 = p1.distance(p2)
p3 = p1.translate(1.0, 1.0)
```

**Правила автоматической привязки**:
1. Функция определена в файле модуля (в том же файле, что и тип)
2. Параметры функции содержат этот тип
3. Экспортируется с помощью `pub`
4. Компилятор автоматически выполняет `Type.method = function[0]`

**Преимущества**:
- Нет необходимости вручную писать объявления привязки
- Код более лаконичный
- Избегаются забытые или ошибочные привязки

#### 7.6.3 Привязка внутри модуля

```yaoxiang
// === Point.yx ===

Point: Type = { x: Float, y: Float }

 dy = a.y - b.y

}

// Внутри модуля все функции видны
// Но автоматическая привязка действует только для функций, экспортированных с pub, за пределами модуля

pub distance // Экспорт, может использоваться автоматическая привязка за пределами
```

### 7.7 Сводка преимуществ дизайна

| Характеристика | Описание |
|----------------|----------|
| **Нулевая синтаксическая нагрузка** | Автоматическая привязка не требует объявлений |
| **Точный позиционный контроль** | `[n]` точно указывает позицию привязки |
| **Многопозиционная совместная привязка** | Поддержка `[1, 2, 3]` для нескольких параметров |
| **Типобезопасность** | Компилятор проверяет допустимость позиции привязки |
| **Без ключевых слов** | Нет необходимости в `bind` или других ключевых словах |
| **Гибкое каррирование** | Поддержка привязки аргументов в любой позиции |
| **Контроль через pub** | Только pub функции могут быть привязаны извне |

### 7.8 Отличия от традиционной привязки методов

| Традиционный язык | YaoXiang |
|-------------------|----------|
| `obj.method(arg)` | `obj.method(arg)` |
| Требуется определение класса/метода | Нужна только функция + объявление привязки |
| Синтаксис `class { method() {} }` | Синтаксис `Type.method = func[n]` |
| Наследование, полиморфизм | Чистый функциональный стиль, без наследования |
| Поиск в таблице методов | Привязка во время компиляции, без накладных расходов в runtime |

**Ключевое преимущество**: привязка в YaoXiang — это **механизм времени компиляции** с нулевой стоимостью в runtime, сохраняющий чистоту и гибкость функционального программирования.

### 8.2 Строго структурированный синтаксис

#### 8.2.1 AI-дружественные стратегии синтаксиса объявлений

```yaoxiang
// === Лучшие практики AI-генерации кода ===

// ✅ Рекомендуется: использование полного нового синтаксиса + конвенции централизации типов

inc: (x: Int) -> Int = x + 1
empty: () -> Void = {}

// ❌ Избегать: пропуск аннотаций типов или разбросанные типы
// ИИ не может определить типы параметров, может сгенерировать ошибочный код
add: (a: Int, b: Int) -> Int = a + b // Параметры без типа
identity: (T: Type) -> ((x: T) -> T) = x // Параметры без типа
add2: (a: Int, b: Int) -> Int = a + b // Типы разбросаны в реализации

// ⚠️ Совместимость: старый синтаксис только для поддержки
// ИИ должен отдавать приоритет генерации нового синтаксиса + конвенции централизации типов
mul(Int, Int) -> Int = (a, b) => a * b // Не рекомендуется в новом коде
```

**AI-преимущества конвенции централизации типов:**

1. **Сигнатура очевидна с первого взгляда**: ИИ достаточно прочитать строку объявления, чтобы понять полную сигнатуру функции
2. **Безопасность изменений**: изменение типа требует изменения только объявления, не затрагивая тело реализации
3. **Упрощение генерации**: ИИ может сначала сгенерировать объявление, затем заполнить реализацию
4. **Каррирование-дружественность**: чёткие каррированные сигнатуры типов упрощают обработку для ИИ

```yaoxiang
// Пример обработки ИИ
// Вход: тело реализации (a, b) => a + b
// ИИ видит объявление: add: (Int, Int) -> Int
// Вывод: типы параметров Int, Int, возвращаемый тип Int

// Сравнение: если типы разбросаны
// Вход: тело реализации (a: Int, b: Int) => a + b
// ИИ требуется: анализ тела реализации для извлечения информации о типах
// Результат: более сложная логика обработки, легче допустить ошибку
```

#### 8.2.2 Стратегия двух синтаксисов и ИИ

| Тип синтаксиса | Стратегия генерации ИИ | Сценарий использования |
|----------------|------------------------|------------------------|
| **Новый синтаксис** | ✅ Приоритетная генерация, полная информация о типах | Вся генерация нового кода |
| **Старый синтаксис** | ⚠️ Только при поддержке старого кода | Изменение исторического кода |
| **Без аннотаций** | ❌ Избегать генерации | Ни в каком случае не должен генерироваться |

#### 8.2.3 Чёткие границы синтаксиса

```yaoxiang

// ✅ Чёткие маркеры начала и конца
function_name: (Type1, Type2) -> ReturnType = (param1, param2) => {
 // Тело функции
 if condition {
 do_something()
 } else {
 do_other()
 }
}

// ✅ Условные операторы должны иметь фигурные скобки
if condition {
 // Тело условия
}

// ✅ Чёткие определения типов
MyType: Type = { Type1 | Type2 }

// ❌ Избегать неоднозначных записей
if condition // Отсутствуют фигурные скобки
 do_something()
```

#### 8.2.4 Ограничения недвусмысленного синтаксиса

```yaoxiang
// Ограничения, которые ИИ должен соблюдать при генерации

// 1. Запрещено опускать скобки
// ✅ Правильно
foo: (T: Type) -> ((x: T) -> T) = x

// ❌ Ошибка (запрещено)
foo T { T } // Параметры должны быть в скобках

// 2. Должен быть явный возвращаемый тип или выводимая форма
// ✅ Правильно
get_num: () -> Int = 42
get_num2: () = 42 // Возвращаемый тип выводится
get_void = () => { 42 } // ✓ Выводится как Void (блок без return по умолчанию возвращает Void)

// 3. Параметры должны иметь аннотации типов (новый синтаксис)
// ✅ Правильно
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1

// ❌ Ошибка
add: (a: Int, b: Int) -> Int = a + b // Параметры без типа
identity: (T: Type) -> ((x: T) -> T) = x // Параметры без типа
```

## 九、Конвенция централизации типов (основная философия дизайна)

### 9.1 Обзор конвенции

Основная конвенция дизайна YaoXiang — **«приоритет объявления, централизация типов»**. Эта конвенция является краеугольным камнем AI-дружественности языка и эффективности разработки.

```yaoxiang
// ✅ Основная конвенция: информация о типах унифицирована в строке объявления
add: (a: Int, b: Int) -> Int = a + b

// ❌ Избегать: информация о типах разбросана в реализации
add: (a: Int, b: Int) -> Int = a + b
```

### 9.2 Пять основных преимуществ конвенции

#### 1. Синтаксическая согласованность
```yaoxiang
// Все объявления следуют одному формату
x: Int = 42 // Переменная
name: String = "YaoXiang" // Переменная
add: (a: Int, b: Int) -> Int = a + b // Функция
inc: (x: Int) -> Int = x + 1 // Функция
Point: Type = { x: Float, y: Float } // Тип
```

#### 2. Разделение объявления и реализации
```yaoxiang
// Строка объявления предоставляет полную информацию о типах
add: (a: Int, b: Int) -> Int = a + b
// └────────────────────┘
// Полная сигнатура функции

// Тело реализации сосредоточено на бизнес-логике
// (a, b) => a + b не нужно заботиться о типах, только реализовать функциональность
```

#### 3. AI-дружественность
```yaoxiang
// Процесс обработки ИИ:
// 1. Чтение строки объявления → полное понимание сигнатуры функции
// 2. Генерация реализации → без необходимости анализа вывода типов
// 3. Изменение типа → только изменение строки объявления, не затрагивая реализацию

// Сравнение: способ с разбросанными типами
add: (a: Int, b: Int) -> Int = a + b
// ИИ требуется: анализ тела реализации для извлечения информации о типах → сложнее, легче ошибиться
```

#### 4. Безопасность изменений
```yaoxiang
// Изменение типа параметра
// Было: add: (a: Int, b: Int) -> Int = a + b
// Стало: add: (Float, Float) -> Float = (a, b) => a + b
// Тело реализации: (a, b) => a + b не требует изменений!

// Если типы разбросаны:
// Было: add: (a: Int, b: Int) -> Int = a + b
// Стало: add: (a: Float, b: Float) -> Float = a + b // Нужно менять два места
```

#### 5. Каррирование-дружественность
```yaoxiang
// Каррированный тип очевиден с первого взгляда
add_curried: (a: Int) -> (b: Int) -> Int = a + b
// └─────────────┘
// Каррированная сигнатура

// Композиция функций как гражданин первого класса
compose: (Int -> Int, Int -> Int) -> Int -> Int = (f, g) => x => f(g(x))
```

### 9.3 Правила реализации конвенции

#### Правило 1: параметры должны иметь типы, указанные в объявлении
```yaoxiang
// ✅ Правильно
add: (a: Int, b: Int) -> Int = a + b

// ❌ Ошибка
add: (a: Int, b: Int) -> Int = a + b // Типы параметров отсутствуют
identity: (T: Type) -> ((x: T) -> T) = x // Типы параметров отсутствуют
```

#### Правило 2: возвращаемый тип может быть выведен, но рекомендуется аннотировать
```yaoxiang
// ✅ Рекомендуется: полная аннотация
get_num: () -> Int = () => 42

// ✅ Допустимо: вывод возвращаемого типа
get_num: () = () => 42

// ✅ Пустая функция выводится как Void
empty: () = () => {}
```

#### Правило 3: аннотации типов внутри лямбды являются временными
```yaoxiang
// ✅ Правильно: полагаемся на типы в объявлении
add: (a: Int, b: Int) -> Int = a + b

// ⚠️ Можно, но не рекомендуется: повторная аннотация внутри лямбды
add: (Int, Int) -> Int = (a: Int, b: Int) => a + b

// ❌ Ошибка: отсутствует аннотация в объявлении
add: (a: Int, b: Int) -> Int = a + b
```

#### Правило 4: старый синтаксис следует той же философии
```yaoxiang
// Старый синтаксис также должен по возможности предоставлять информацию о типах в позиции объявления
// Хотя формат отличается, философия та же:
// - Строка объявления содержит основную информацию о типах
// - Тело реализации относительно лаконично
add(Int, Int) -> Int = (a, b) => a + b
```

### 9.4 Связь между конвенцией и выводом типов

```yaoxiang
// Конвенция не препятствует выводу типов, а направляет его

// 1. Полная аннотация (без вывода)
add: (a: Int, b: Int) -> Int = a + b

// 2. Частичный вывод (объявление предоставляет типы параметров)
add: (Int, Int) = (a, b) => a + b // Вывод возвращаемого типа

// 3. Вывод пустой функции
empty: () = () => {} // Выводится как () -> Void
```

### 9.5 Преимущества конвенции для AI-реализации

**Процесс AI-генерации кода:**

1. **Чтение требования** → генерация объявления
   ```
   Требование: функция сложения
   Генерация: add: (Int, Int) -> Int = (a, b) => ???
   ```

2. **Заполнение реализации** → без анализа типов
   ```
   Реализация: add: (a: Int, b: Int) -> Int = a + b
   ```

3. **Изменение типа** → только изменение объявления
   ```
   Изменение: add: (Float, Float) -> Float = (a, b) => a + b
   Реализация: (a, b) => a + b  остаётся без изменений
   ```

**Сравнение с обработкой без конвенции:**
```
Требование: функция сложения
ИИ требуется:
  1. Вывести типы параметров
  2. Вывести возвращаемый тип
  3. Сгенерировать тело реализации
  4. Проверить согласованность
  5. Обработать сложные обновления при изменении типов

Результат: сложнее, легче допустить ошибку
```

### 9.6 Философское значение конвенции

Эта конвенция отражает основные идеи YaoXiang:

- **Объявление — это документация**: строка объявления — это полная документация функции
- **Тип — это контракт**: информация о типах — это контракт между вызывающим и реализующим
- **Логика — это реализация**: тело реализации заботится только о том, «что делать», а не «какого типа»
- **Инструмент — это помощник**: система типов, инструменты ИИ могут работать на основе чётких объявлений

### 9.7 Сравнение практического применения

#### Полный пример: модуль калькулятора

```yaoxiang
// === Рекомендуемый подход: конвенция централизации типов ===

// Объявления модуля
pub add: (a: Int, b: Int) -> Int = a + b
pub multiply: (a: Int, b: Int) -> Int = a * b

// Функция высшего порядка
pub apply_twice: (f: Int -> Int, x: Int) -> Int = f(f(x))

// Каррированная функция
pub make_adder: (x: Int) -> (Int) -> Int = y => x + y

// Обобщённая функция
pub map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) = case xs of
 [] => []
 (x :: rest) => f(x) :: map(f, rest)

// Определение типа
Point: Type = { x: Float, y: Float }
pub distance: (a: Point, b: Point) -> Float = {
 dx = a.x - b.x
 dy = a.y - b.y
 (dx * dx + dy * dy).sqrt()
}

// === Нерекомендуемый подход: разбросанные типы ===

// Типы параметров в лямбде
add: (a: Int, b: Int) -> Int = a + b
multiply = (a: Int, b: Int) => a * b

// Типы функции высшего порядка разбросаны
apply_twice = (f: (Int) -> Int, x: Int) => f(f(x))

// Каррированные типы разбросаны
make_adder = (x: Int) => (y: Int) => x + y

// Обобщённые типы разбросаны
map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) = case xs of
 [] => []
 (x :: rest) => f(x) :: map(f, rest)
```

#### Сравнение поддержки кода

```yaoxiang
// Требование: изменить add с Int на Float

// === Рекомендуемый подход: нужно изменить только строку объявления ===
// Было
add: (a: Int, b: Int) -> Int = a + b

// Стало
add: (a: Float, b: Float) -> Float = a + b
// ↑↑↑↑↑↑↑↑↑ ↑↑↑↑↑↑↑
// Изменение строки объявления  Реализация остаётся без изменений

// === Нерекомендуемый подход: нужно менять несколько мест ===
// Было
add: (a: Int, b: Int) -> Int = a + b

// Стало
add: (a: Float, b: Float) -> Float = a + b
// ↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑
// Все типы параметров нужно менять
```

#### Сравнение AI-ассистированного программирования

```yaoxiang
// Требование ИИ: реализовать функцию для вычисления манхэттенского расстояния между двумя точками

// === Когда ИИ видит рекомендуемый стиль ===
Point: Type = { x: Float, y: Float }
pub manhattan: (a: Point, b: Point) -> Float = ??? // ИИ сразу знает полную сигнатуру

// ИИ генерирует:

 (a.x - b.x).abs() + (a.y - b.y).abs()
}

// === Когда ИИ видит нерекомендуемый стиль ===
Point: Type = { x: Float, y: Float }
pub manhattan = ??? // ИИ нужно выводить: типы параметров? возвращаемый тип?

// ИИ может сгенерировать:
pub manhattan = (a: Point, b: Point) => Float => {
 (a.x - b.x).abs() + (a.y - b.y).abs()
}
// Или может ошибиться, потому что информация о типах неполная
```

### 9.8 Контрольный список реализации конвенции

При написании кода на YaoXiang можно использовать следующий контрольный список:

- [ ] Все объявления функций имеют полные аннотации типов в строке объявления
- [ ] Типы параметров указаны в объявлении, а не в лямбде
- [ ] Возвращаемые типы по возможности аннотированы в объявлении
- [ ] Объявления переменных используют формат `name: Type = value`
- [ ] Тело лямбды остаётся лаконичным, без дублирования информации о типах
- [ ] Используется новый синтаксис вместо старого
- [ ] Сложные типы определяются через `type`, объявления остаются чёткими

- **Python**: стиль синтаксиса, читаемость
- **Idris/Agda**: зависимые типы, разработка через типы
- **TypeScript**: аннотации типов, типы во время выполнения
