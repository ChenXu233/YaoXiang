## 1. Обзор языка

### 1.1 Что такое YaoXiang?

YaoXiang（爻象）— это экспериментальный универсальный язык программирования, чья философия дизайна основана на ключевых концепциях «Яо» и «Сян» из «И-Цзин». «Яо» — это основной символ, составляющий гексаграмму, олицетворяющий изменения инь и ян; «Сян» — это внешнее проявление сущности вещей, представляющее все существующие явления.

YaoXiang интегрирует эту философию в систему типов языка программирования, выдвигая **«Всё есть тип»** как основную идею. В мировоззрении YaoXiang:

- **Значения** являются экземплярами типов
- **Типы** сами являются экземплярами типов (метатипы)
- **Функции** представляют собой отображения из входных типов в выходные типы
- **Модули** являются комбинациями пространств имён типов

### 1.2 Цели дизайна

Цели дизайна YaoXiang можно резюмировать следующим образом:

| Цель | Описание |
|------|----------|
| **Единая абстракция типов** | Тип как высший уровень абстракции, упрощение семантики языка |
| **Естественный опыт программирования** | Синтаксис в стиле Python, акцент на читаемости |
| **Безопасное управление памятью** | Модель владения в стиле Rust, без сборщика мусора |
| **Незаметное асинхронное программирование** | Автоматическое управление асинхронностью без явного await |
| **Полная рефлексия типов** | Полная доступность информации о типах во время выполнения |
| **AI-дружелюбный синтаксис** | Строгая структурированность, лёгкость обработки AI |

### 1.3 Позиционирование языка

| Измерение | Позиционирование |
|-----------|------------------|
| Парадигма | Мультипарадигма (функциональная + императивная + объектно-ориентированная) |
| Система типов | Зависимые типы + параметрический полиморфизм |
| Управление памятью | Владение + RAII (без GC) |
| Модель компиляции | AOT-компиляция (опционально JIT) |
| Целевые сценарии | Системное программирование, разработка приложений, AI-辅助ное программирование |

### 1.4 Примеры кода

```yaoxiang
# Автоматический вывод типов
x: Int = 42                           # Явный тип
y = 42                                # Выводится как Int
name = "YaoXiang"                     # Выводится как String

# По умолчанию неизменяемо
x: Int = 10
x = 20                                # ❌ Ошибка компиляции! Неизменяемо

# Унифицированный синтаксис объявлений: идентификатор: тип = выражение
add: (a: Int, b: Int) -> Int = a + b  # Объявление функции
inc: (x: Int) -> Int = x + 1               # Однопараметрическая функция

# Унифицированный синтаксис типов: конструктор является типом
type Point = { x: Float, y: Float }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# Незаметная асинхронность (并作函数)
fetch_data: (url: String) -> JSON spawn = {
    HTTP.get(url).json()
}

main: () -> Void = {
    # Конструкция значений: полностью идентична вызову функции
    p = Point(3.0, 4.0)
    r = ok("success")

    data = fetch_data("https://api.example.com")
    # Автоматическое ожидание, без await
    print(data.name)
}

# Обобщённая функция
identity: (T: Type) -> ((x: T) -> T) = x

# Функции высшего порядка
apply: (f: (Int) -> Int, x: Int) -> Int = f(x)

# Каррирование
add_curried: (a: Int) -> (b: Int) -> Int = a + b
```

## 2. Основные возможности

### 2.1 Всё есть тип

Основная философия дизайна YaoXiang — **всё есть тип**. Это означает, что в YaoXiang:

1. **Значения являются экземплярами типов**: `42` — экземпляр типа `Int`
2. **Типы являются экземплярами типов**: `Int` — экземпляр меатипа `type`
3. **Функции являются отображениями типов**: `add: (Int, Int) -> Int` — тип функции
4. **Модули являются комбинациями типов**: модуль — это пространство имён с функциями и типами

```yaoxiang
# Значения являются экземплярами типов
x: Int = 42

# Типы являются экземплярами типов
MyList: type = List(Int)

# Функции являются отображениями между типами
add: (a: Int, b: Int) -> Int = a + b

# Модули являются комбинациями типов (файл как модуль)
# Math.yx
pi: Float = 3.14159
sqrt: (x: Float) -> Float = { ... }
```

### 2.2 Математические абстракции

Система типов YaoXiang основана на теории типов и теории категорий, предоставляя:

- **Зависимые типы**: типы могут зависеть от значений
- **Обобщённое программирование**: параметризация типов
- **Комбинация типов**: объединённые типы, типы-пересечения

```yaoxiang
# Зависимые типы: вектор фиксированной длины
Vector: (T: Type, n: Int) -> Type = vector(T, n)

# Объединение типов
type Number = Int | Float

# Пересечение типов
type Printable = printable(fn() -> String)
type Serializable = serializable(fn() -> String)
type Versatile = Printable & Serializable
```

### 2.3 Абстракция с нулевой стоимостью

YaoXiang гарантирует абстракцию с нулевой стоимостью: высокоуровневые абстракции не влекут накладных расходов во время выполнения:

- **Мономорфизация**: обобщённые функции разворачиваются во время компиляции в конкретные версии
- **Встраивание (inlining)**: простые функции автоматически встраиваются
- **Распределение в стеке**: небольшие объекты по умолчанию распределяются в стеке

```yaoxiang
# Разворачивание обобщений (мономорфизация)
identity: (T: Type) -> ((x: T) -> T) = x

# Использование
int_val = identity(42)      # Разворачивается в identity(Int) -> Int
str_val = identity("hello") # Разворачивается в identity(String) -> String

# После компиляции без дополнительных накладных расходов
```

### 2.4 Естественный синтаксис

YaoXiang использует синтаксис в стиле Python, стремясь к читаемости и естественности языка:

```yaoxiang
# Автоматический вывод типов
x = 42
name = "YaoXiang"

# Краткие определения функций
greet: (name: String) -> String = "Hello, " + name

# Паттерн матчинг
classify: (n: Int) -> String = {
    match n {
        0 -> "zero"
        1 -> "one"
        _ if n < 0 -> "negative"
        _ -> "many"
    }
}
```

### 2.5 Полная спецификация синтаксиса

YaoXiang использует унифицированный синтаксис объявлений: **идентификатор: тип = выражение**. Также предоставляется обратно совместимый старый синтаксис.

#### 2.5.1 Стратегия двойного синтаксиса и соглашение о концентрации типов

Для баланса между инновациями и совместимостью YaoXiang поддерживает две формы синтаксиса, но использует унифицированное **соглашение о концентрации типов**.

**Сравнение форм синтаксиса:**

| Тип синтаксиса | Формат | Статус | Описание |
|----------------|--------|--------|----------|
| **Новый синтаксис (стандартный)** | `name: Type = Lambda` | ✅ Рекомендуется | Официальный стандарт, вся новая код должен использовать эту форму |
| **Старый синтаксис (совместимость)** | `name(Types) -> Ret = Lambda` | ⚠️ Только для совместимости | Сохраняется для исторического кода, не рекомендуется для новых проектов |

**Основное соглашение: концентрация типов**

YaoXiang использует принцип проектирования **«Объявление прежде всего, типы концентрируются»**:

```yaoxiang
# ✅ Правильно: информация о типах унифицирована в строке объявления
add: (a: Int, b: Int) -> Int = a + b
#   └─────────────────┘   └─────────────┘
#       Полная сигнатура типа    Реализация логики

# ❌ Избегайте: информация о типах разбросана в реализации
add: (a: Int, b: Int) -> Int = a + b
#     └───────────────┘
#     Типы смешаны с телом реализации
```

**Преимущества соглашения:**

1. **Синтаксическая согласованность**: все объявления следуют формату `идентификатор: тип = выражение`
2. **Разделение объявления и реализации**: информация о типах видна сразу, тело реализации сосредоточено на логике
3. **AI-дружелюбность**: AI достаточно прочитать строку объявления, чтобы понять полную сигнатуру функции
4. **Более безопасные изменения**: изменение типа требует только изменения объявления, без влияния на тело реализации
5. **Дружелюбность к каррированию**: поддержка чётких каррированных сигнатур типов

**Рекомендации по выбору**:
- **Новые проекты**: должны использовать новый синтаксис + соглашение о концентрации типов
- **Проекты миграции**: постепенно переходить на новый синтаксис и соглашение о концентрации типов
- **Поддержка старого кода**: можно продолжать использовать старый синтаксис, но рекомендуется следовать соглашению о концентрации типов

#### 2.5.2 Базовый синтаксис объявлений

```yaoxiang
# === Новый синтаксис (рекомендуется) ===
# Все объявления следуют формату: идентификатор: тип = выражение

# Объявление переменной
x: Int = 42
name: String = "YaoXiang"
mut counter: Int = 0

# Объявление функции
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1
getAnswer: () -> Int = 42
log: (msg: String) -> Void = print(msg)

# === Старый синтаксис (совместимость) ===
# Только для функций, формат: name(Types) -> Ret = Lambda
add(Int, Int) -> Int = (a, b) => a + b
square(Int) -> Int = (x) => x * x
empty() -> Void = () => {}
getRandom() -> Int = () => 42
```

#### 2.5.3 Синтаксис типов функций

```
Тип функции ::= '(' список типов параметров ')' '->' тип возврата
           | тип параметра '->' тип возврата              # Сокращение для одного параметра

Список типов параметров ::= [тип (',' тип)*]
Тип возврата ::= тип | тип функции | 'Void'

# Типы функций являются объектами первого класса, могут быть вложены
# Тип функции высшего порядка ::= '(' тип функции ')' '->' тип возврата
```

| Пример | Значение |
|--------|----------|
| `Int -> Int` | Тип функции с одним параметром |
| `(Int, Int) -> Int` | Тип функции с двумя параметрами |
| `() -> Void` | Тип функции без параметров |
| `(Int -> Int) -> Int` | Функция высшего порядка: принимает функцию, возвращает Int |
| `Int -> Int -> Int` | Каррированная функция (правоассоциативная) |

#### 2.5.4 Обобщённый синтаксис (только для параметров типов)

```yaoxiang
# Обобщённая функция: префикс <параметры типов>
identity: (T: Type) -> ((x: T) -> T) = x
map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) = case xs of
  [] => []
  (x :: rest) => f(x) :: map(f, rest)

# Обобщённый тип
List: (T: Type) -> Type
```

#### 2.5.5 Синтаксис Lambda-выражений

```
Lambda ::= '(' список параметров ')' '=>' выражение
         | параметр '=>' выражение              # Сокращение для одного параметра

Список параметров ::= [параметр (',' параметр)*]
Параметр ::= идентификатор [':' тип]               # Необязательная аннотация типа
```

| Пример | Значение | Описание |
|--------|----------|----------|
| `(a, b) => a + b` | Lambda с несколькими параметрами | Используется с объявлением:<br>`add: (Int, Int) = (a, b) => a + b` |
| `x => x + 1` | Сокращение для одного параметра | Используется с объявлением:<br>`inc: Int = x => x + 1` |
| `(x: Int) => x + 1` | С аннотацией типа | Только для случаев, когда внутри Lambda нужна информация о типе |
| `() => 42` | Lambda без параметров | Используется с объявлением:<br>`get: () = () => 42` |

**Примечание**: аннотации типов в Lambda-выражениях `(x: Int) => ...` являются **временными, локальными**, в основном используются для:
- Когда внутри Lambda нужна информация о типе
- При использовании с синтаксисом объявлений (тип указан в объявлении)
- Не должны использоваться как основной способ объявления типов

#### 2.5.6 Полные примеры

```yaoxiang
# === Базовые объявления функций ===

# Базовая функция (новый синтаксис)
add: (a: Int, b: Int) -> Int = a + b

# Функция с одним параметром (две формы)
inc: (x: Int) -> Int = x + 1
inc2: (x: Int) -> Int = x + 1

# Функция без параметров
getAnswer: () -> Int = 42

# Функция без возвращаемого значения
log: (msg: String) -> Void = print(msg)

# === Рекурсивные функции ===
# Рекурсия естественно поддерживается в lambda
fact: (n: Int) -> Int =
  if n <= 1 then 1 else n * fact(n - 1)

# === Функции высшего порядка и присваивание типов функций ===

# Типы функций как объекты первого класса
IntToInt: Type = (Int) -> Int
IntBinaryOp: Type = (Int, Int) -> Int

# Объявление функции высшего порядка
applyTwice: (f: IntToInt, x: Int) -> Int = f(f(x))

# Каррированная функция
addCurried: (a: Int) -> (b: Int) -> Int = a + b

# Композиция функций
compose: (f: Int -> Int, g: Int -> Int) -> (x: Int) -> Int =
  f(g(x))

# Функция, возвращающая функцию
makeAdder: (x: Int) -> (y: Int) -> Int =
  x + y

# === Обобщённые функции ===

# Обобщённая функция
identity: (T: Type) -> ((x: T) -> T) = x

# Обобщённая функция высшего порядка
map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) =
  case xs of
    [] => []
    (x :: rest) => f(x) :: map(f, rest)

# Обобщённый тип функции
Transformer: Type = (A: Type, B: Type) -> ((A) -> B)

# Использование обобщённого типа
applyTransformer: (A: Type, B: Type) -> ((f: Transformer(A, B), x: A) -> B) =
  f(x)

# === Примеры сложных типов ===

# Вложенные типы функций
higherOrder: (A: Type) -> ((f: (A) -> Int) -> (A) -> Int) =
  f => x => f(x) + 1

# Многопараметрическая функция высшего порядка
zipWith: (A: Type, B: Type, C: Type) -> ((f: (A, B) -> C, xs: List(A), ys: List(B)) -> List(C)) =
  case (xs, ys) of
    ([], _) => []
    (_, []) => []
    (x::xs', y::ys') => f(x, y) :: zipWith(f, xs', ys')

# Псевдонимы типов функций
Predicate: (T: Type) -> Type = { apply: (T) -> Bool }
Mapper: Type = (A: Type, B: Type) -> ((A) -> B)
Reducer: Type = (A: Type, B: Type) -> ((B, A) -> B)

# === Примеры старого синтаксиса (только для обратной совместимости) ===
# Не рекомендуется для нового кода

mul(Int, Int) -> Int = (a, b) => a * b    # Несколько параметров
square(Int) -> Int = (x) => x * x          # Один параметр
empty() -> Void = () => {}                  # Без параметров
get_random() -> Int = () => 42              # С возвращаемым значением

# Эквивалентный новый синтаксис (рекомендуется)
mul: (a: Int, b: Int) -> Int = a * b
square: (x: Int) -> Int = x * x
empty: () -> Void = {}
get_random: () -> Int = 42
```

#### 2.5.7 Правила разбора синтаксиса

**Приоритеты разбора типов:**

| Приоритет | Тип | Описание |
|-----------|-----|----------|
| 1 (высший) | Обобщённое применение `List(T)` | Левоассоциативность |
| 2 | Скобки `(T)` | Изменение ассоциативности |
| 3 | Тип функции `->` | Правоассоциативность |
| 4 (низший) | Базовые типы `Int, String` | Атомарные типы |

**Примеры разбора типов:**

```yaoxiang
# (A -> B) -> C -> D
# Разбирается как: ((A -> B) -> (C -> D))

# A -> B -> C
# Разбирается как: (A -> (B -> C))  # Правоассоциативность

# (Int -> Int) -> Int
# Разбирается: принимает функцию, возвращает Int -> Int

# List<Int -> Int>
# Разбирается: тип элемента List — Int -> Int
```

**Примеры разбора Lambda:**

```yaoxiang
# a => b => a + b
# Разбирается: a => (b => (a + b))  # Правоассоциативность, каррирование

# (a, b) => a + b
# Разбирается: принимает два параметра, возвращает a + b
```

#### 2.5.8 Правила вывода типов

YaoXiang использует стратегию **двухуровневой обработки**: разборщик宽松но пропускает, уровень проверки типов строго выводит.

**Правила уровня разбора:**
- Разборщик только проверяет синтаксическую структуру, не выполняет вывод типов
- Для объявлений без аннотаций типов поле аннотации типа равно `None`
- Все объявления, соответствующие базовой синтаксической структуре, проходят разбор
- **Ключевой момент**: `add: (a: Int, b: Int) -> Int = a + b` является **допустимым** на уровне разбора

**Правила уровня проверки типов:**
- Проверяется семантическая корректность, включая полноту типов
- **Параметры должны иметь аннотации типов**: это обязательное требование
- Тип возврата может быть выведен, но типы параметров должны быть объявлены явно

**Полные правила вывода типов:**

| Сценарий | Вывод параметров | Вывод возврата | Результат разбора | Результат проверки типов | Рекомендация |
|----------|------------------|----------------|-------------------|-------------------------|--------------|
| **Стандартная функция** | Используется аннотированный тип | Используется аннотированный тип | ✅ | ✅ | ⭐⭐⭐⭐⭐ |
| `add: (a: Int, b: Int) -> Int = a + b` | | | | | |
| **Частичный вывод** | Используется аннотированный тип | Выводится из выражения | ✅ | ✅ | ⭐⭐⭐⭐ |
| `add: (Int, Int) = (a, b) => a + b` | | | | | |
| `inc: (x: Int) -> Int = x + 1` | | | | | |
| `get: () = () => 42` | | | | | |
| **Частичный вывод старого синтаксиса** | Используется аннотированный тип | Выводится из выражения | ✅ | ✅ | ⭐⭐⭐ (совместимость) |
| `add(Int, Int) = (a, b) => a + b` | | | | | |
| `square(Int) = (x) => x * x` | | | | | |
| **Параметры без аннотаций** | **Невозможно вывести** | - | ✅ | ❌ Ошибка | ❌ Запрещено |
| `add: (a: Int, b: Int) -> Int = a + b` | | | | | |
| `identity: (T: Type) -> ((x: T) -> T) = x` | | | | | |
| **Блок без аннотации возврата** | - | Выводится из содержимого блока | ✅ | ✅ | ⭐⭐⭐⭐ |
| `main = () => {}` | | | | | |
| `get = () => { return 42; }` | | | | | |
| **Блок без аннотации возврата (нет явного возврата)** | - | Выводится как `Void` | ✅ | ✅ Не рекомендуется | ⚠️ Избегайте |
| `bad = (x: Int) => { x }` | | | | | |

**Детальные правила вывода:**

```
Уровень разбора: смотрит только на синтаксическую структуру
├── Структура корректна → Проходит
└── Структура некорректна → Ошибка

Уровень проверки типов: проверяет семантику
├── Вывод типов параметров
│   ├── Параметр имеет аннотацию типа → Используется аннотированный тип ✅
│   ├── Параметр без аннотации типа → Отклоняется ❌
│   └── Параметры Lambda должны быть аннотированы → Обязательное требование
│
├── Вывод типа возврата
│   ├── Есть return expr → Выводится из expr ✅
│   ├── Нет return, есть выражение → Выводится из выражения ✅
│   ├── Нет return, есть блок `{ ... }`
│   │   ├── Блок пустой `{}` → Void ✅
│   │   ├── Блок имеет return → Выводится из return ✅
│   │   └── Блок без return и без явного возврата → Выводится как Void ✅ (но не рекомендуется)
│   └── Невозможно вывести → Отклоняется ❌
│
└── Полностью невозможно вывести → Отклоняется ❌
```

**Примечание**: форма `bad = (x: Int) => { x }` позволяет вывести тип возврата как `Void`, но крайне не рекомендуется, потому что:
- Намерение кода неясно
- Легко вызывает недопонимание
- Не соответствует стилю функционального программирования на основе выражений

**Примеры вывода:**

```yaoxiang
# === Успешный вывод ===

# Стандартная форма
main: () -> Void = () => {}                    # Полная аннотация
num: () -> Int = () => 42                      # Полная аннотация
inc: (x: Int) -> Int = x + 1                   # Сокращение для одного параметра

# Частичный вывод (новый синтаксис)
add: (Int, Int) = (a, b) => a + b              # Параметры аннотированы, возврат выводится
square: (x: Int) -> Int = x * x                # Параметры аннотированы, возврат выводится
get_answer: () = () => 42                      # Параметры аннотированы (пустые), возврат выводится

# Частичный вывод (старый синтаксис, совместимость)
add2(Int, Int) = (a, b) => a + b               # Параметры аннотированы, возврат выводится
square2(Int) = (x) => x * x                    # Параметры аннотированы, возврат выводится

# Вывод из return
fact: Int -> Int = (n) => {
    if n <= 1 { return 1 }
    return n * fact(n - 1)
}

# === Ошибка вывода ===

# Параметры невозможно вывести (разбор проходит, проверка типов неудачна)
add: (a: Int, b: Int) -> Int = a + b                          # ✗ Параметры без типа
identity: (T: Type) -> ((x: T) -> T) = x                              # ✗ Параметры без типа

# Блок без явного возврата
no_return = (x: Int) => { x }                  # ✗ Блок без return, невозможно вывести неявный возврат

# Полностью невозможно вывести
bad_fn: (T: Type) -> ((x: T) -> T) = x                                # ✗ Невозможно вывести ни параметры, ни возврат
```

#### 2.5.9 Старый синтаксис (обратная совместимость)

YaoXiang предоставляет поддержку старого синтаксиса для совместимости с историческим кодом, **не рекомендуется для нового кода**.

```
Старый синтаксис ::= идентификатор '(' [список типов параметров] ')' '->' тип возврата '=' Lambda
```

| Возможность | Стандартный синтаксис | Старый синтаксис |
|-------------|----------------------|-------------------|
| Формат объявления | `name: Type = ...` | `name(Types) -> Type = ...` |
| Позиция типов параметров | В аннотации типа | После имени функции в скобках |
| Пустые параметры | Обязательно писать `()` | Можно опустить `()` |
| **Рекомендация** | ✅ **Официальная рекомендация** | ⚠️ **Только для обратной совместимости** |
| **Сценарий использования** | Весь новый код | Поддержка исторического кода |

**Причины нерекомендации:**
1. **Затраты на обучение**: несоответствие со стандартным синтаксисом, увеличивает сложность языка
2. **Согласованность**: неединообразная позиция типов параметров (один в аннотации типа, другой после имени функции)
3. **Затраты на поддержку**: разборщику нужно дополнительно обрабатывать две формы
4. **AI-дружелюбность**: усложняет понимание и генерацию кода AI

**Рекомендации по миграции:**
```yaoxiang
# Старый код (не рекомендуется)
mul(Int, Int) -> Int = (a, b) => a * b
square(Int) -> Int = (x) => x * x
empty() -> Void = () => {}

# Новый код (рекомендуется)
mul: (Int, Int) -> Int = (a, b) => a * b
square: (Int) -> Int = (x) => x * x
empty: () -> Void = () => {}
```

## 4. Управление памятью

### 4.1 Модель владения

YaoXiang использует **модель владения** для управления памятью, каждое значение имеет единственного владельца:

```yaoxiang
# === По умолчанию Move (нулевое копирование) ===
p: Point = Point(1.0, 2.0)
p2 = p              # Move, передача владения, p становится недействительным

# === Ключевое слово ref = Arc (безопасное совместное использование) ===
shared = ref p      # Arc, потокобезопасность

spawn(() => print(shared.x))   # ✅ Безопасно

# === clone() явное копирование ===
p3 = p.clone()      # p и p3 независимы
```

### 4.2 Move-семантика (по умолчанию)

```yaoxiang
# Присваивание = Move (нулевое копирование)
p: Point = Point(1.0, 2.0)
p2 = p              # Move, p становится недействительным

# Передача параметров функции = Move
process: (p: Point) -> Void = {
    # Владение p передаётся внутрь
}

# Возвращаемое значение = Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    return p        # Move, передача владения
}
```

### 4.3 Ключевое слово ref (Arc)

```yaoxiang
# Ключевое слово ref создаёт Arc (подсчёт ссылок)
p: Point = Point(1.0, 2.0)
shared = ref p      # Arc, потокобезопасность

spawn(() => print(shared.x))   # ✅ Безопасно

# Arc автоматически управляет жизненным циклом
# Когда shared выходит из области видимости, счётчик становится равным нулю и автоматически освобождается
```

### 4.4 clone() явное копирование

```yaoxiang
# Используйте clone(), когда нужно сохранить оригинальное значение
p: Point = Point(1.0, 2.0)
p2 = p.clone()   # p и p2 независимы

p.x = 0.0        # ✅
p2.x = 0.0       # ✅ Взаимное влияние отсутствует
```

### 4.5 Блоки unsafe (системный уровень)

```yaoxiang
# Голые указатели можно использовать только в блоках unsafe
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p     # Голый указатель
    (*ptr).x = 0.0       # Пользователь гарантирует безопасность
}
```

### 4.6 RAII

```yaoxiang
# RAII автоматическое освобождение
with_file: (path: String) -> String = {
    file = File.open(path)  # Автоматическое открытие
    content = file.read_all()
    # По окончании функции file автоматически закрывается
    content
}
```

### 4.7 Ограничения Send / Sync

| Ограничение | Семантика | Описание |
|-------------|-----------|----------|
| **Send** | Можно безопасно передавать между потоками | Значение можно переместить в другой поток |
| **Sync** | Можно безопасно совместно использовать между потоками | Неизменяемые ссылки можно передавать в другой поток |

```yaoxiang
# ref T автоматически удовлетворяет Send + Sync (Arc потокобезопасен)
p: Point = Point(1.0, 2.0)
shared = ref p

spawn(() => print(shared.x))   # ✅ Arc потокобезопасен

# Голый указатель *T не удовлетворяет Send/Sync
unsafe {
    ptr: *Point = &p         # Можно использовать только в одном потоке
}
```

| Возможность | Причина |
|-------------|---------|
| Время жизни `'a` | Нет концепции ссылок, время жизни не нужно |
| Borrow checker | ref = Arc как замена |
| Синтаксис заимствования `&T` | Используется Move-семантика |

## 5. Асинхронное программирование и параллелизм

> «Все существа действуют одновременно, и я наблюдаю их возвращение.» — «И-Цзин. Гексаграмма 24 (Фу/Возвращение)»
>
> YaoXiang использует **модель 并作**, вид незаметного асинхронного параллелизма на основе **ленивых вычислений**. Её основная идея: **позволить разработчику описывать логику синхронным, последовательным мышлением, а среда выполнения языка автоматически и эффективно выполняет вычислительные единицы подобно «все существа действуют одновременно», и в итоге координирует их**.

> Подробности см. в [Бумаге о модели 并作](YaoXiang-async-whitepaper.md) и [Схеме реализации асинхронности](YaoXiang-async-implementation.md).

### 5.1 Основные концепции модели 并作

#### 5.1.1 并作图: сцена для «все существа действуют одновременно»

Вся программа во время компиляции преобразуется в **ориентированный ациклический вычислительный граф (DAG)**, называемый **并作图**. Узлы представляют вычисления выражений, рёбра — зависимости по данным. Этот граф ленивый, то есть узел вычисляется только тогда, когда его вывод **действительно нужен**.

```yaoxiang
# Компилятор автоматически строит 并作图
fetch_user: spawn () -> User = (id) => { ... }
fetch_posts: spawn (User) -> Posts = (user) => { ... }

main:() -> Void = () => {
    user = fetch_user(1)     # Узел A (Async(User))
    posts = fetch_posts(user) # Узел B (Async(Posts)), зависит от A

    # Узел C нуждается в результатах A и B
    print(posts.title)       # Автоматическое ожидание: сначала убедиться, что A и B завершены
}
```

#### 5.1.2 并作值: Async(T)

Любой вызов функции, помеченной `spawn fn`, немедленно возвращает значение типа `Async(T)`, называемое **并作值**. Это лёгкий прокси-объект, он не является реальным результатом, а представляет **будущее значение, находящееся в процессе 并作**.

**Основные характеристики**:
- **Прозрачность типа**: `Async(T)` является подтипом `T` в системе типов и может использоваться в любом контексте, ожидающем `T`
- **Автоматическое ожидание**: когда программа доходит до операции, требующей конкретное значение типа `T`, среда выполнения автоматически приостанавливает текущую задачу и ожидает завершения вычисления
- **Нулевая «заразность»**: асинхронный код не отличается от синхронного синтаксически и в сигнатурах типов

```yaoxiang
# Пример использования 并作值
fetch_data: spawn (String) -> JSON = (url) => { ... }

main: () -> Void = () => {
    data = fetch_data("url")  # Async(JSON)

    # Async(JSON) можно напрямую использовать как JSON
    # Автоматическое ожидание происходит при обращении к полю
    print(data.name)          # Эквивалентно data.await().name
}
```

### 5.2 Система синтаксиса 并作

Ключевое слово `spawn` имеет тройную семантику и является единственным мостом между синхронным мышлением и асинхронной реализацией:

| Официальный термин | Синтаксическая форма | Семантика | Поведение во время выполнения |
|--------------------|----------------------|-----------|--------------------------------|
| **并作函数** | `spawn fn` | Определение вычислительной единицы, участвующей в 并作 | Её вызов возвращает `Async(T)` |
| **并作块** | `spawn { a(), b() }` | Явное объявление области параллелизма | Задачи внутри блока выполняются принудительно параллельно |
| **并作循环** | `spawn for x in xs { ... }` | Парадигма данных-параллелизма | Тело цикла выполняется параллельно для всех элементов |

#### 5.2.1 并作函数

```yaoxiang
# Используйте spawn для标记 并作函数
# Синтаксис полностью идентичен обычной функции, без дополнительной нагрузки

fetch_api: spawn (String) -> JSON = (url) => {
    response = HTTP.get(url)
```

### 5.5 Потокобезопасность: ограничения Send/Sync

YaoXiang использует подобную Rust **систему ограничений типа Send/Sync** для обеспечения потокобезопасности, устраняя гонки данных на этапе компиляции.

#### 5.5.1 Ограничение Send

**Send**: тип можно безопасно передавать между потоками **с передачей владения**.

```yaoxiang
# Базовые типы автоматически удовлетворяют Send
# Int, Float, Bool, String все Send

# Структуры автоматически выводят Send
type Point = { x: Int, y: Float }
# Point является Send, потому что Int и Float являются Send

# Типы, содержащие не-Send поля, не являются Send
type NonSend = NonSend(data: Rc(Int))
# Rc не является Send (подсчёт ссылок неатомарный), поэтому NonSend не является Send
```

#### 5.5.2 Ограничение Sync

**Sync**: тип можно безопасно совместно использовать между потоками **разделяя ссылки**.

```yaoxiang
# Базовые типы все Sync
type Point = { x: Int, y: Float }
# &Point является Sync, потому что &Int и &Float являются Sync

# Типы с внутренней изменяемостью
type Counter = Counter(value: Int, mutex: Mutex(Int))
# &Counter является Sync, потому что Mutex обеспечивает внутреннюю изменяемость
```

#### 5.5.3 spawn и потокобезопасность

```yaoxiang
# spawn требует, чтобы параметры и возвращаемое значение удовлетворяли Send

# Допустимо: Data является Send
type Data = Data(value: Int)
task = spawn(|| => Data(42))

# Недопустимо: Rc не является Send
type SharedData = SharedData(rc: Rc(Int))
# task = spawn(|| => SharedData(Rc.new(42))  # Ошибка компиляции!

# Решение: использовать Arc (атомарный подсчёт ссылок)
type SafeData = SafeData(value: Arc(Int))
task = spawn(|| => SafeData(Arc.new(42)))  # Arc является Send + Sync
```

#### 5.5.4 Правила вывода потокобезопасности типов

```yaoxiang
# Типы структур
type Struct(T1, T2) = Struct(f1: T1, f2: T2)

# Вывод Send
Struct(T1, T2): Send ⇐ T1: Send и T2: Send

# Вывод Sync
Struct(T1, T2): Sync ⇐ T1: Sync и T2: Sync

# Объединённые типы
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# Вывод Send
Result(T, E): Send ⇐ T: Send и E: Send
```

#### 5.5.5 Реализация потокобезопасности в стандартной библиотеке

| Тип | Send | Sync | Описание |
|------|:----:|:----:|----------|
| `Int`, `Float`, `Bool` | ✅ | ✅ | Примитивные типы |
| `Arc(T)` | ✅ | ✅ | T: Send + Sync |
| `Mutex(T)` | ✅ | ✅ | T: Send |
| `RwLock(T)` | ✅ | ✅ | T: Send |
| `Channel(T)` | ✅ | ❌ | Только отправляющий конец Send |
| `Rc(T)` | ❌ | ❌ | Неатомарный подсчёт ссылок |
| `RefCell(T)` | ❌ | ❌ | Проверка заимствования во время выполнения |


```yaoxiang
# Пример потокобезопасного счётчика
type SafeCounter = SafeCounter(mutex: Mutex(Int))

main: () -> Void = () => {
    counter: Arc(SafeCounter) = Arc.new(SafeCounter(Mutex.new(0)))

    # Параллельные обновления
    spawn(|| => {
        guard = counter.mutex.lock()  # Mutex обеспечивает потокобезопасность
        guard.value = guard.value + 1
    })

    spawn(|| => {
        guard = counter.mutex.lock()
        guard.value = guard.value + 1
    })
}
```

## 7. Привязка методов и каррирование

YaoXiang использует **чистый функциональный дизайн**, реализуя беспрепятственный вызов методов и каррирование через развитый механизм привязки, без введения таких ключевых слов, как `struct`, `class` и других.

### 7.1 Определения основных функций

Все операции реализуются через обычные функции, первый параметр по соглашению является субъектом операции:

```yaoxiang
# === Point.yx (модуль) ===

# Унифицированный синтаксис: конструктор является типом
type Point = { x: Float, y: Float }

# Основная функция: первый параметр — субъект операции
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

add: (a: Point, b: Point) -> Point = {
    Point(a.x + b.x, a.y + b.y)
}

scale: (p: Point, s: Float) -> Point = {
    Point(p.x * s, p.y * s)
}

# Более сложная функция
distance_with_scale: (s: Float, p1: Point, p2: Point) -> Float = {
    dx = (p1.x - p2.x) * s
    dy = (p1.y - p2.y) * s
    (dx * dx + dy * dy).sqrt()
}
```

### 7.2 Базовая привязка методов

#### 7.2.1 Автоматическая привязка (стиль MoonBit)

YaoXiang поддерживает автоматическую привязку на основе пространства имён, **без каких-либо дополнительных объявлений**:

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# === main.yx ===

use Point

    p2 = Point(1.0, 2.0)

```

**Правила автоматической привязки**:
- Функция определена внутри модуля
- Если первый параметр типа соответствует имени модуля
- То автоматически поддерживается синтаксис вызова метода

### 7.3 Синтаксис привязки на основе позиции

YaoXiang предоставляет **наиболее элегантный синтаксис привязки**, использующий позиционные метки `[n]` для точного контроля позиции привязки:

#### 7.3.1 Базовый синтаксис

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Основная функция
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}
add: (a: Point, b: Point) -> Point = {
    Point(a.x + b.x, a.y + b.y)
}
scale: (p: Point, s: Float) -> Point = {
    Point(p.x * s, p.y * s)
}

# Синтаксис привязки: Type.method = func[position]
# Значение: при вызове метода вызывающий объект привязывается к параметру [position] функции

Point.distance = distance[0]      # Привязка к 1-му параметру
Point.add = add[0]                 # Привязка к 1-му параметру
Point.scale = scale[0]             # Привязка к 1-му параметру
```

**Семантический разбор**:
- `Point.distance = distance[0]`
  - Функция `distance` имеет два параметра: `distance(Point, Point)`
  - `[0]` означает, что вызывающий объект привязывается к 1-му параметру
  - Использование: `p1.distance(p2)` → `distance(p1, p2)`

#### 7.3.2 Многопозиционная совместная привязка

```yaoxiang
# === Math.yx ===

# Функция: scale, point1, point2, extra1, extra2
calculate: (scale: Float, a: Point, b: Point, x: Float, y: Float) -> Float = { ... }

# === Point.yx ===

type Point = { x: Float, y: Float }

# Привязка нескольких позиций
Point.calc1 = calculate[1, 2]      # Привязка scale и point1
Point.calc2 = calculate[1, 3]      # Привязка scale и point2
Point.calc3 = calculate[2, 3]      # Привязка point1 и point2

# === main.yx ===

use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# 1. Привязка[1,2] - остаются 3,4,5
f1 = p1.calc1(2.0)  # Привязка scale=2.0, point1=p1
# f1 теперь требует p2, x, y
result1 = f1(p2, 10.0, 20.0)  # calculate(2.0, p1, p2, 10.0, 20.0)

# 2. Привязка[1,3] - остаются 2,4,5
f2 = p2.calc2(2.0)  # Привязка scale=2.0, point2=p2
# f2 теперь требует point1, x, y
result2 = f2(p1, 10.0, 20.0)  # calculate(2.0, p1, p2, 10.0, 20.0)

# 3. Привязка[2,3] - остаются 1,4,5
f3 = p1.calc3(p2)  # Привязка point1=p1, point2=p2
# f3 теперь требует scale, x, y
result3 = f3(2.0, 10.0, 20.0)  # calculate(2.0, p1, p2, 10.0, 20.0)
```

#### 7.3.3 Порядок заполнения оставшихся параметров

**Основное правило**: после привязки оставшиеся параметры заполняются в **исходном порядке функции**, с пропуском уже привязанных позиций.

```yaoxiang
# Предположим функция: func(p1, p2, p3, p4, p5)

# Привязка 1-го и 3-го параметров
Type.method = func[1, 3]

# При вызове:
method(p2_value, p4_value, p5_value)

# Соответствие:
func(p1_bound, p2_value, p3_bound, p4_value, p5_value)
# Оставшиеся параметры: 2,4,5 заполняются в исходном порядке
```

#### 7.3.4 Преимущества проверки типов

```yaoxiang
# ✅ Допустимая привязка
Point.distance = distance[0]          # distance(Point, Point)
Point.calc = calculate[1, 2]          # calculate(scale, Point, Point, ...)

# ❌ Недопустимая привязка (ошибка компилятора)
```


### 7.6 Область видимости и правила привязки

#### 7.6.1 Роль pub

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Не-pub функция
internal_distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x

# pub функция
pub distance: (a: Point, b: Point) -> Float = {

}

# === main.yx ===

use Point

# Автоматическая привязка работает только для pub функций
p1.distance(p2)      # ✅ distance является pub, возможна автоматическая привязка
# p1.internal_distance(p2)  # ❌ Не pub, привязка невозможна
```

#### 7.6.2 Механизм автоматической привязки pub

Функции, объявленные с `pub`, компилятор автоматически привязывает к типум, определённым в том же файле:

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Использование pub, компилятор автоматически выполняет привязку
pub distance: (p1: Point, p2: Point) -> Float = {
    dx = p1.x - p2.x
    dy = p1.y - p2.y
    (dx * dx + dy * dy).sqrt()
}

pub translate: (self: Point, dx: Float, dy: Float) -> Point = {
    Point(self.x + dx, self.y + dy)
}

# Компилятор автоматически выводит и выполняет привязку:
# Point.distance = distance[0]
# Point.translate = translate[0]

# === main.yx ===

use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# ✅ Функциональный вызов
d = distance(p1, p2)

# ✅ Синтаксический сахар ООП (автоматическая привязка)
d2 = p1.distance(p2)
p3 = p1.translate(1.0, 1.0)
```

**Правила автоматической привязки**:
1. Функция определена в файле модуля (в том же файле, что и тип)
2. Параметр функции содержит этот тип
3. Экспортировано с помощью `pub`
4. Компилятор автоматически выполняет `Type.method = function[0]`

**Преимущества**:
- Не нужно вручную писать объявления привязки
- Код более лаконичен
- Исключается забывание или ошибки привязки

#### 7.6.3 Привязка внутри модуля

```yaoxiang

}

# Внутри модуля все функции видны
# Но автоматическая привязка работает для pub-экспортированных функций только извне

pub distance  # Экспорт, доступна автоматическая привязка извне
```

### 7.7 Сводка преимуществ дизайна

| Возможность | Описание |
|-------------|----------|
| **Нулевая синтаксическая нагрузка** | Автоматическая привязка без дополнительных объявлений |
| **Точный позиционный контроль** | `[n]` точно указывает позицию привязки |
| **Многопозиционная совместная привязка** | Поддержка привязки нескольких параметров `[1, 2, 3]` |
| **Безопасность типов** | Компилятор проверяет допустимость позиций привязки |
| **Без ключевых слов** | Не требуются `bind` или другие ключевые слова |
| **Гибкое каррирование** | Поддержка привязки параметров в любой позиции |
| **Контроль через pub** | Только pub функции привязываются извне |

### 7.8 Отличия от традиционной привязки методов

| Традиционные языки | YaoXiang |
|-------------------|----------|
| `obj.method(arg)` | `obj.method(arg)` |
| Нужны определения класс/метод | Только функции + объявления привязки |
| Синтаксис `class { method() {} }` | Синтаксис `Type.method = func[n]` |
| Наследование, полиморфизм | Чистый функциональный стиль, без наследования |
| Поиск в таблице методов | Привязка на этапе компиляции, нулевые накладные расходы выполнения |

**Основное преимущество**: привязки YaoXiang являются **механизмом времени компиляции**, с нулевыми накладными расходами выполнения, при этом сохраняя чистоту и гибкость функционального программирования.

## 9. Соглашение о концентрации типов (основная философия дизайна)

### 9.1 Обзор соглашения

Основное проектное соглашение YaoXiang — **«Объявление прежде всего, типы концентрируются»**. Это соглашение является основой AI-дружелюбности языка и эффективности разработки.

```yaoxiang
# ✅ Основное соглашение: информация о типах унифицирована в строке объявления
add: (a: Int, b: Int) -> Int = a + b

# ❌ Избегайте: информация о типах разбросана в реализации
add: (a: Int, b: Int) -> Int = a + b
```

### 9.2 Пять ключевых преимуществ соглашения

#### 1. Синтаксическая согласованность
```yaoxiang
# Все объявления следуют одинаковому формату
x: Int = 42                           # Переменная
name: String = "YaoXiang"             # Переменная
add: (a: Int, b: Int) -> Int = a + b  # Функция
inc: (x: Int) -> Int = x + 1          # Функция
type Point = { x: Float, y: Float } # Тип
```

#### 2. Разделение объявления и реализации
```yaoxiang
# Строка объявления предоставляет полную информацию о типах
add: (a: Int, b: Int) -> Int = a + b
# └────────────────────┘
#   Полная сигнатура функции

# Тело реализации сосредоточено на бизнес-логике
# (a, b) => a + b  Не нужно заботиться о типах, только реализовать функционал
```

#### 3. AI-дружелюбность
```yaoxiang
# Процесс обработки AI:
# 1. Читать строку объявления → Полностью понять сигнатуру функции
# 2. Генерировать реализацию → Не нужно анализировать вывод типов
# 3. Изменять типы → Только изменить строку объявления, не затронуть реализацию

# Сравнение: при разбросанных типах
add: (a: Int, b: Int) -> Int = a + b
# AI требуется: анализировать тело реализации для извлечения информации о типах → Сложнее, легче ошибиться
```

#### 4. Более безопасные изменения
```yaoxiang
# Изменение типа параметра
# Было: add: (a: Int, b: Int) -> Int = a + b
# Изменено: add: (Float, Float) -> Float = (a, b) => a + b
# Тело реализации: (a, b) => a + b  Не требует изменений!

# При разбросанных типах:
# Было: add: (a: Int, b: Int) -> Int = a + b
# Изменено: add: (a: Float, b: Float) -> Float = a + b  # Нужно менять в двух местах
```

#### 5. Дружелюбность к каррированию
```yaoxiang
# Каррированные типы видны сразу
add_curried: (a: Int) -> (b: Int) -> Int = a + b
#              └─────────────┘
#              Каррированная сигнатура

# Композиция функций как объект первого класса
compose: (Int -> Int, Int -> Int) -> Int -> Int = (f, g) => x => f(g(x))
```

### 9.3 Правила применения соглашения

#### Правило 1: Параметры должны иметь типы, указанные в объявлении
```yaoxiang
# ✅ Правильно
add: (a: Int, b: Int) -> Int = a + b

# ❌ Ошибка
add: (a: Int, b: Int) -> Int = a + b            # Типы параметров отсутствуют
identity: (T: Type) -> ((x: T) -> T) = x                # Типы параметров отсутствуют
```

#### Правило 2: Тип возврата может быть выведен, но рекомендуется аннотировать
```yaoxiang
# ✅ Рекомендуется: полная аннотация
get_num: () -> Int = () => 42

# ✅ Допустимо: вывод типа возврата
get_num: () = () => 42

# ✅ Пустая функция выводится как Void
empty: () = () => {}
```

#### Правило 3: Временные аннотации типов внутри Lambda
```yaoxiang
# ✅ Правильно: опираться на типы из объявления
add: (a: Int, b: Int) -> Int = a + b

# ⚠️ Можно, но не рекомендуется: повторная аннотация внутри Lambda
add: (Int, Int) -> Int = (a: Int, b: Int) => a + b

# ❌ Ошибка: отсутствует аннотация в объявлении
add: (a: Int, b: Int) -> Int = a + b
```

#### Правило 4: Старый синтаксис следует той же идеологии
```yaoxiang
# В старом синтаксисе тоже нужно стараться предоставлять информацию о типах в позиции объявления
# Хотя формат другой, идеология一致:
# - Строка объявления содержит основную информацию о типах
# - Тело реализации относительно лаконично
add(Int, Int) -> Int = (a, b) => a + b
```

### 9.4 Соотношение соглашения и вывода типов

```yaoxiang
# Соглашение не препятствует выводу типов, а направляет направление вывода

# 1. Полная аннотация (без вывода)
add: (a: Int, b: Int) -> Int = a + b

# 2. Частичный вывод (объявление предоставляет типы параметров)
add: (Int, Int) = (a, b) => a + b  # Тип возврата выводится

# 3. Вывод пустой функции
empty: () = () => {}  # Выводится как () -> Void
```

### 9.5 AI-преимущества реализации соглашения

**Процесс генерации кода AI:**

1. **Читать требования** → Генерировать объявление
   ```
   Требование: функция сложения
   Генерировать: add: (Int, Int) -> Int = (a, b) => ???
   ```

2. **Заполнять реализацию** → Не требуется анализ типов
   ```
   Реализация: add: (a: Int, b: Int) -> Int = a + b
   ```

3. **Изменение типов** → Только изменить объявление
   ```
   Изменение: add: (Float, Float) -> Float = (a, b) => a + b
   Реализация: (a, b) => a + b  Остаётся без изменений
   ```

**Сравнение обработки AI без соглашения:**
```
Требование: функция сложения
AI требуется:
  1. Вывести типы параметров
  2. Вывести тип возврата
  3. Сгенерировать тело реализации
  4. Проверить согласованность
  5. Обработать сложные обновления при изменении типов

Результат: Сложнее, легче допустить ошибку
```

### 9.6 Философский смысл соглашения

Это соглашение отражает ключевую идею YaoXiang:

- **Объявление — это документация**: строка объявления является полной документацией функции
- **Тип — это контракт**: информация о типах — это контракт между вызывающим и реализатором
- **Логика — это реализация**: тело реализации фокусируется только на «что делать», а не на «какой тип»
- **Инструменты — это помощники**: система типов и AI-инструменты могут работать на основе чётких объявлений

### 9.7 Сравнение практического применения

#### Полный пример: модуль калькулятора

```yaoxiang

# Функции высшего порядка

# Каррированные функции
pub make_adder: (x: Int) -> (Int) -> Int = y => x + y

# Обобщённые функции
pub map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) = case xs of
  [] => []
  (x :: rest) => f(x) :: map(f, rest)

```

```yaoxiang
# Было
```

### 9.8 Контрольный список применения соглашения

При написании кода YaoXiang можно использовать следующий контрольный список:

- [ ] Все объявления функций имеют полные аннотации типов в строке объявления
- [ ] Типы параметров указаны в объявлении, а не в Lambda
- [ ] Типы возврата по возможности аннотированы в объявлении
- [ ] Объявления переменных используют формат `name: Type = value`
- [ ] Тела Lambda остаются лаконичными, без повторения информации о типах
- [ ] Используется новый синтаксис, а не старый
- [ ] Сложные типы определены через type, объявления остаются чёткими

### 10.3 Пример привязки методов

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Основная функция
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# Автоматическая привязка
Point.distance = distance[0]

# === main.yx ===

use Point

main: () -> Void = {
    p1 = Point(3.0, 4.0)
    p2 = Point(1.0, 2.0)

    # Использование привязки
    d = p1.distance(p2)  # distance(p1, p2)
    print(d)
}
```

### 10.4 Пример каррированной привязки

```yaoxiang
# === Math.yx ===

distance_with_scale: (scale: Float, a: Point, b: Point) -> Float = {
    dx = (p1.x - p2.x) * scale
    dy = (p1.y - p2.y) * scale
    (dx * dx + dy * dy).sqrt()
}

# === Point.yx ===

type Point = { x: Float, y: Float }

Point.distance_scaled = distance_with_scale[2]  # Привязка ко 2-му параметру

# === main.yx ===

use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# Использование привязки
f = p1.distance_scaled(2.0)  # Привязка scale и p1
result = f(p2)               # Финальный вызов

# Или прямой вызов
result2 = p1.distance_scaled(2.0, p2)
```
