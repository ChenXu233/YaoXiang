## 一、Обзор языка

### 1.1 Что такое YaoXiang？

YaoXiang（爻象）— это экспериментальный универсальный язык программирования, чья философия дизайна основана на ключевых концепциях「爻」и「象」из «Ицзин». «爻» — это основной символ, составляющий гексаграмму, олицетворяющий инь и ян; «象» — это внешнее проявление сущности вещей, представляющее все явления мира.

YaoXiang воплощает эту философию в типовое система языка, выдвигая **«всё является типом»** в качестве основной идеи. В мировоззрении YaoXiang:

- **Значения** — это экземпляры типов
- **Типы** сами являются экземплярами типов (метатипы)
- **Функции** — это отображения из входных типов в выходные типы
- **Модули** — это комбинации пространств имён типов

### 1.2 Цели дизайна

Цели дизайна YaoXiang можно резюмировать следующим образом:

| Цель | Описание |
|------|----------|
| **Единая абстракция типов** | Тип — это высший уровень абстракции, упрощающий семантику языка |
| **Естественный опыт программирования** | Синтаксис в стиле Python, акцент на читаемости |
| **Безопасное управление памятью** | Модель владения в стиле Rust, без сборщика мусора |
| **Незаметное асинхронное программирование** | Автоматическое управление асинхронностью без явного await |
| **Полная рефлексия типов** | Полная доступность информации о типах во время выполнения |
| **AI-дружелюбный синтаксис** | Строгая структуризация, лёгкость обработки AI |

### 1.3 Позиционирование языка

| Измерение | Позиционирование |
|-----------|------------------|
| Парадигма | Мультипарадигма (функциональная + императивная + объектно-ориентированная) |
| Типовая система | Зависимые типы + параметрический полиморфизм |
| Управление памятью | Владение + RAII (без GC) |
| Модель компиляции | AOT-компиляция (опциональный JIT) |
| Целевые сценарии | Системное программирование, разработка приложений, AI-辅助编程 |

### 1.4 Примеры кода

```yaoxiang
# Автоматический вывод типов
x: Int = 42                           # Явный тип
y = 42                                # Выводится как Int
name = "YaoXiang"                     # Выводится как String

# По умолчанию неизменяемо
x: Int = 10
x = 20                                # ❌ Ошибка компиляции! Неизменяемо

# Унифицированный синтаксис объявления: идентификатор: тип = выражение
add: (a: Int, b: Int) -> Int = a + b  # Объявление функции
inc: (x: Int) -> Int = x + 1          # Функция с одним параметром

# Унифицированный синтаксис типов: конструктор является типом
type Point = { x: Float, y: Float }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# Незаметная асинхронность (spawn-функция)
fetch_data: (url: String) -> JSON spawn = {
    HTTP.get(url).json()
}

main: () -> Void = {
    # Конструкция значений: полностью идентична вызову функции
    p = Point(3.0, 4.0)
    r = ok("success")

    data = fetch_data("https://api.example.com")
    # Автоматическое ожидание, без await
    print(data.name)
}

# Обобщённая функция
identity: (T: Type) -> ((x: T) -> T) = x

# Функции высшего порядка
apply: (f: (Int) -> Int, x: Int) -> Int = f(x)

# Каррирование
add_curried: (a: Int) -> (b: Int) -> Int = a + b
```

## 二、Основные возможности

### 2.1 Всё является типом

Основная философия дизайна YaoXiang — **всё является типом**. Это означает:

1. **Значения являются экземплярами типов**: `42` — экземпляр типа `Int`
2. **Типы являются экземплярами типов**: `Int` — экземпляр метатипа `type`
3. **Функции являются отображениями типов**: `add: (Int, Int) -> Int` — тип функции
4. **Модули являются комбинациями типов**: модуль — это пространство имён с функциями и типами

```yaoxiang
# Значения являются экземплярами типов
x: Int = 42

# Типы являются экземплярами типов
MyList: type = List(Int)

# Функции являются отображениями между типами
add: (a: Int, b: Int) -> Int = a + b

# Модули являются комбинациями типов (используем файл как модуль)
# Math.yx
pi: Float = 3.14159
sqrt: (x: Float) -> Float = { ... }
```

### 2.2 Математическая абстракция

Типовая система YaoXiang основана на теории типов и теории категорий, предоставляя:

- **Зависимые типы**: типы могут зависеть от значений
- **Обобщённое программирование**: параметризация типов
- **Комбинирование типов**: объединённые типы, типы-пересечения

```yaoxiang
# Зависимые типы: вектор фиксированной длины
Vector: (T: Type, n: Int) -> Type = vector(T, n)

# Объединение типов
type Number = Int | Float

# Пересечение типов
type Printable = printable(fn() -> String)
type Serializable = serializable(fn() -> String)
type Versatile = Printable & Serializable
```

### 2.3 Абстракция с нулевой стоимостью

YaoXiang гарантирует абстракцию с нулевой стоимостью: высокоуровневые абстракции не влекут накладных расходов во время выполнения:

- **Мономорфизация**: обобщённые функции раскрываются в конкретные версии во время компиляции
- **Встраивание**: простые функции автоматически встраиваются
- **Распределение в стеке**: небольшие объекты по умолчанию распределяются в стеке

```yaoxiang
# Раскрытие обобщений (мономорфизация)
identity: (T: Type) -> ((x: T) -> T) = x

# Использование
int_val = identity(42)      # Раскрывается в identity(Int) -> Int
str_val = identity("hello") # Раскрывается в identity(String) -> String

# После компиляции без дополнительных накладных расходов
```

### 2.4 Естественный синтаксис

YaoXiang использует синтаксис в стиле Python, стремясь к читаемости и естественности языка:

```yaoxiang
# Автоматический вывод типов
x = 42
name = "YaoXiang"

# Краткое определение функций
greet: (name: String) -> String = "Hello, " + name

# Паттерн-матчинг
classify: (n: Int) -> String = {
    match n {
        0 -> "zero"
        1 -> "one"
        _ if n < 0 -> "negative"
        _ -> "many"
    }
}
```

### 2.5 Полная спецификация синтаксиса

YaoXiang использует унифицированный синтаксис объявления: **идентификатор: тип = выражение**. Также предоставляется обратно совместимый старый синтаксис.

#### 2.5.1 Стратегия двойного синтаксиса и соглашение о централизации типов

Для баланса инноваций и совместимости YaoXiang поддерживает две формы синтаксиса, но использует унифицированное **соглашение о централизации типов**.

**Сравнение форм синтаксиса:**

| Тип синтаксиса | Формат | Статус | Описание |
|----------------|--------|--------|----------|
| **Новый синтаксис (стандартный)** | `name: Type = Lambda` | ✅ Рекомендуется | Официальный стандарт, вся новый код должен использовать эту форму |
| **Старый синтаксис (совместимость)** | `name(Types) -> Ret = Lambda` | ⚠️ Только совместимость | Сохранён для исторического кода, не рекомендуется для новых проектов |

**Основное соглашение: централизация типов**

YaoXiang использует **«объявление в первую очередь, типы централизованы»** как проектное соглашение:

```yaoxiang
# ✅ Правильно: информация о типах унифицирована в строке объявления
add: (a: Int, b: Int) -> Int = a + b
#   └─────────────────┘   └─────────────┘
#       Полная сигнатура типа    Реализация логики

# ❌ Избегайте: информация о типах разбросана в реализации
add: (a: Int, b: Int) -> Int = a + b
#     └───────────────┘
#     Типы смешаны с телом реализации
```

**Преимущества соглашения:**

1. **Синтаксическая согласованность**: все объявления следуют формату `идентификатор: тип = выражение`
2. **Разделение объявления и реализации**: информация о типах понятна с первого взгляда, тело реализации сосредоточено на логике
3. **AI-дружелюбность**: AI достаточно прочитать строку объявления, чтобы понять полную сигнатуру функции
4. **Более безопасные модификации**: изменение типа требует изменения только объявления, без влияния на тело реализации
5. **Каррирование дружелюбно**: поддержка чётких каррированных сигнатур типов

**Рекомендации по выбору:**
- **Новые проекты**: обязательно использовать новый синтаксис + соглашение о централизации типов
- **Проекты миграции**: постепенно мигрировать на новый синтаксис и соглашение о централизации типов
- **Поддержка старого кода**: можно продолжать использовать старый синтаксис, но рекомендуется применять соглашение о централизации типов

#### 2.5.2 Базовый синтаксис объявления

```yaoxiang
# === Новый синтаксис (рекомендуется) ===
# Все объявления следуют формату: идентификатор: тип = выражение

# Объявление переменной
x: Int = 42
name: String = "YaoXiang"
mut counter: Int = 0

# Объявление функции
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1
getAnswer: () -> Int = 42
log: (msg: String) -> Void = print(msg)

# === Старый синтаксис (совместимость) ===
# Только для функций, формат: name(Types) -> Ret = Lambda
add(Int, Int) -> Int = (a, b) => a + b
square(Int) -> Int = (x) => x * x
empty() -> Void = () => {}
getRandom() -> Int = () => 42
```

#### 2.5.3 Синтаксис типов функций

```
Тип функции ::= '(' список типов параметров ')' '->' тип возврата
              | тип параметра '->' тип возврата                    # Одно параметрное сокращение

Список типов параметров ::= [тип (',' тип)*]
Тип возврата ::= тип | тип функции | 'Void'

# Тип функции — объект первого класса, может быть вложенным
# Тип функции высшего порядка ::= '(' тип функции ')' '->' тип возврата
```

| Пример | Значение |
|--------|----------|
| `Int -> Int` | Тип функции с одним параметром |
| `(Int, Int) -> Int` | Тип функции с двумя параметрами |
| `() -> Void` | Тип функции без параметров |
| `(Int -> Int) -> Int` | Функция высшего порядка: принимает функцию, возвращает Int |
| `Int -> Int -> Int` | Каррированная функция (правоассоциативная) |

#### 2.5.4 Синтаксис обобщений (только для параметров типов)

```yaoxiang
# Обобщённая функция: префикс <параметры типа>
identity: (T: Type) -> ((x: T) -> T) = x
map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) = case xs of
  [] => []
  (x :: rest) => f(x) :: map(f, rest)

# Обобщённый тип
List: (T: Type) -> Type
```

#### 2.5.5 Синтаксис Lambda-выражений

```
Lambda ::= '(' список параметров ')' '=>' выражение
         | параметр '=>' выражение                        # Одно параметрное сокращение

Список параметров ::= [параметр (',' параметр)*]
Параметр ::= идентификатор [':' тип]                       # Необязательная аннотация типа
```

| Пример | Значение | Описание |
|--------|----------|----------|
| `(a, b) => a + b` | Lambda с несколькими параметрами | Используется с объявлением:<br>`add: (Int, Int) = (a, b) => a + b` |
| `x => x + 1` | Одно параметрное сокращение | Используется с объявлением:<br>`inc: Int = x => x + 1` |
| `(x: Int) => x + 1` | С аннотацией типа | Только для случаев, когда внутри Lambda нужна информация о типе |
| `() => 42` | Lambda без параметров | Используется с объявлением:<br>`get: () = () => 42` |

**Внимание**: аннотации типов в Lambda-выражениях `(x: Int) => ...` являются **временными и локальными**, в основном используются для:
- Когда внутри Lambda нужна информация о типах
- При использовании с синтаксисом объявления (тип указан в объявлении)
- Не должны использоваться как основной способ объявления типов

#### 2.5.6 Полные примеры

```yaoxiang
# === Базовые объявления функций ===

# Базовая функция (новый синтаксис)
add: (a: Int, b: Int) -> Int = a + b

# Функция с одним параметром (две формы)
inc: (x: Int) -> Int = x + 1
inc2: (x: Int) -> Int = x + 1

# Функция без параметров
getAnswer: () -> Int = 42

# Функция без возвращаемого значения
log: (msg: String) -> Void = print(msg)

# === Рекурсивные функции ===
# Рекурсия естественно поддерживается в lambda
fact: (n: Int) -> Int =
  if n <= 1 then 1 else n * fact(n - 1)

# === Функции высшего порядка и присваивание типов функций ===

# Тип функции как объект первого класса
IntToInt: Type = (Int) -> Int
IntBinaryOp: Type = (Int, Int) -> Int

# Объявление функции высшего порядка
applyTwice: (f: IntToInt, x: Int) -> Int = f(f(x))

# Каррированная функция
addCurried: (a: Int) -> (b: Int) -> Int = a + b

# Композиция функций
compose: (f: Int -> Int, g: Int -> Int) -> (x: Int) -> Int =
  f(g(x))

# Функция, возвращающая функцию
makeAdder: (x: Int) -> (y: Int) -> Int =
  x + y

# === Обобщённые функции ===

# Обобщённая функция
identity: (T: Type) -> ((x: T) -> T) = x

# Обобщённая функция высшего порядка
map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) =
  case xs of
    [] => []
    (x :: rest) => f(x) :: map(f, rest)

# Обобщённый тип функции
Transformer: Type = (A: Type, B: Type) -> ((A) -> B)

# Использование обобщённого типа
applyTransformer: (A: Type, B: Type) -> ((f: Transformer(A, B), x: A) -> B) =
  f(x)

# === Примеры сложных типов ===

# Вложенные типы функций
higherOrder: (A: Type) -> ((f: (A) -> Int) -> (A) -> Int) =
  f => x => f(x) + 1

# Функция высшего порядка с несколькими параметрами
zipWith: (A: Type, B: Type, C: Type) -> ((f: (A, B) -> C, xs: List(A), ys: List(B)) -> List(C)) =
  case (xs, ys) of
    ([], _) => []
    (_, []) => []
    (x::xs', y::ys') => f(x, y) :: zipWith(f, xs', ys')

# Псевдонимы типов функций
Predicate: (T: Type) -> Type = { apply: (T) -> Bool }
Mapper: Type = (A: Type, B: Type) -> ((A) -> B)
Reducer: Type = (A: Type, B: Type) -> ((B, A) -> B)

# === Примеры старого синтаксиса (только для обратной совместимости) ===
# Не рекомендуется использовать в новом коде

mul(Int, Int) -> Int = (a, b) => a * b    # Несколько параметров
square(Int) -> Int = (x) => x * x          # Один параметр
empty() -> Void = () => {}                  # Без параметров
get_random() -> Int = () => 42              # С возвращаемым значением

# Эквивалентный новый синтаксис (рекомендуется)
mul: (a: Int, b: Int) -> Int = a * b
square: (x: Int) -> Int = x * x
empty: () -> Void = {}
get_random: () -> Int = 42
```

#### 2.5.7 Правила синтаксического анализа

**Приоритеты解析 типов:**

| Приоритет | Тип | Описание |
|-----------|-----|----------|
| 1 (высший) | Обобщённое применение `List(T)` | Левоассоциативное |
| 2 | Скобки `(T)` | Изменение ассоциативности |
| 3 | Тип функции `->` | Правоассоциативное |
| 4 (низший) | Базовые типы `Int, String` | Атомарные типы |

**Примеры解析 типов:**

```yaoxiang
# (A -> B) -> C -> D
#解析为: ((A -> B) -> (C -> D))

# A -> B -> C
#解析为: (A -> (B -> C))  # Правоассоциативность

# (Int -> Int) -> Int
#解析为: Принимает функцию, возвращает Int -> Int

# List<Int -> Int>
#解析为: Тип элемента List — Int -> Int
```

**Примеры解析 Lambda:**

```yaoxiang
# a => b => a + b
#解析为: a => (b => (a + b))  # Правоассоциативность, каррирование

# (a, b) => a + b
#解析为: Принимает два параметра, возвращает a + b
```

#### 2.5.8 Правила вывода типов

YaoXiang использует стратегию **двухуровневой обработки**: уровень синтаксического анализа宽松но пропускает, уровень проверки типов строго выводит.

**Правила уровня синтаксического анализа:**
- Анализатор только проверяет синтаксическую структуру, не выполняет вывод типов
- Для объявлений без аннотаций типов поле аннотации типа равно `None`
- Все объявления, соответствующие базовой синтаксической структуре, проходят синтаксический анализ
- **Ключевой момент**: `add: (a: Int, b: Int) -> Int = a + b` на уровне синтаксического анализа **законно**

**Правила уровня проверки типов:**
- Проверка семантической корректности, включая полноту типов
- **Параметры должны иметь аннотации типов**: это обязательное требование
- Тип возврата может быть выведен, но типы параметров должны быть объявлены явно

**Полные правила вывода типов:**

| Сценарий | Вывод параметров | Вывод возврата | Результат解析 | Результат проверки | Рекомендация |
|----------|-------------------|-----------------|---------------|---------------------|--------------|
| **Стандартная функция** | Использовать аннотированный тип | Использовать аннотированный тип | ✅ | ✅ | ⭐⭐⭐⭐⭐ |
| `add: (a: Int, b: Int) -> Int = a + b` | | | | | |
| **Частичный вывод** | Использовать аннотированный тип | Вывести из выражения | ✅ | ✅ | ⭐⭐⭐⭐ |
| `add: (Int, Int) = (a, b) => a + b` | | | | | |
| `inc: (x: Int) -> Int = x + 1` | | | | | |
| `get: () = () => 42` | | | | | |
| **Старый синтаксис с частичным выводом** | Использовать аннотированный тип | Вывести из выражения | ✅ | ✅ | ⭐⭐⭐ (совместимость) |
| `add(Int, Int) = (a, b) => a + b` | | | | | |
| `square(Int) = (x) => x * x` | | | | | |
| **Параметры без аннотаций** | **Невозможно вывести** | - | ✅ | ❌ Ошибка | ❌ Запрещено |
| `add: (a: Int, b: Int) -> Int = a + b` | | | | | |
| `identity: (T: Type) -> ((x: T) -> T) = x` | | | | | |
| **Блок без возвращаемой аннотации** | - | Вывести из содержимого блока | ✅ | ✅ | ⭐⭐⭐⭐ |
| `main = () => {}` | | | | | |
| `get = () => { return 42; }` | | | | | |
| **Блок кода без return** | - | По умолчанию `Void` | ✅ | ✅ Правильно | ✅ Правильно |
| `add: (a: Int, b: Int) -> Int = { return a + b }` | | | | | |

**Подробные правила вывода:**

```
Уровень解析: смотрим только на синтаксическую структуру
├── Структура верна → Проходит
└── Структура неверна → Ошибка

Уровень проверки типов: проверка семантики
├── Вывод типов параметров
│   ├── Параметр с аннотацией типа → Использовать аннотированный тип ✅
│   ├── Параметр без аннотации типа → Отклонить ❌
│   └── Параметры Lambda должны иметь аннотации → Обязательное требование
│
├── Вывод типа возврата
│   ├── Есть return expr → Вывести из expr ✅
│   ├── Нет return, есть выражение → Вывести из выражения ✅
│   ├── Блок кода без return → По умолчанию Void ✅
│   └── Невозможно вывести → Отклонить ❌
│
└── Полностью невозможно вывести → Отклонить ❌
```

**Внимание**: когда внутри блока кода нет `return`, по умолчанию возвращается `Void`. Например:
- `() => { 42 }` выводится как `() -> Void` (блок без return, по умолчанию возвращает Void)
- `() => { return 42 }` выводится как `() -> Int` (есть return, выводится из return)
- `() => 42` выводится как `() -> Int` (форма выражения, напрямую возвращает значение)

**Примеры вывода:**

```yaoxiang
# === Успешный вывод ===

# Стандартная форма
main: () -> Void = () => {}                    # Полная аннотация
num: () -> Int = () => 42                      # Полная аннотация
inc: (x: Int) -> Int = x + 1                   # Одно параметрное сокращение

# Частичный вывод (новый синтаксис)
add: (Int, Int) = (a, b) => a + b              # Параметры аннотированы, возвращаемый тип выводится
square: (x: Int) -> Int = x * x                # Параметры аннотированы, возвращаемый тип выводится
get_answer: () = () => 42                      # Параметры аннотированы (пустые), возвращаемый тип выводится

# Частичный вывод (старый синтаксис, совместимость)
add2(Int, Int) = (a, b) => a + b               # Параметры аннотированы, возвращаемый тип выводится
square2(Int) = (x) => x * x                    # Параметры аннотированы, возвращаемый тип выводится

# Вывод из return
fact: Int -> Int = (n) => {
    if n <= 1 { return 1 }
    return n * fact(n - 1)
}

# === Ошибка вывода ===

# Параметры невозможно вывести (проходит解析, ошибка проверки типов)
add: (a: Int, b: Int) -> Int = a + b                          # ✗ Параметры без типа
identity: (T: Type) -> ((x: T) -> T) = x                              # ✗ Параметры без типа

# Блок кода без return
no_return = (x: Int) => { x }                  # ✓ Выводится как Void (блок без return по умолчанию Void)

# Полностью невозможно вывести
bad_fn: (T: Type) -> ((x: T) -> T) = x                                # ✗ Параметры и возвращаемый тип невозможно вывести
```

#### 2.5.9 Старый синтаксис (обратная совместимость)

YaoXiang предоставляет поддержку старого синтаксиса для совместимости с историческим кодом, **не рекомендуется использовать в новом коде**.

```
Старый синтаксис ::= идентификатор '(' [список типов параметров] ')' '->' тип возврата '=' Lambda
```

| Возможность | Стандартный синтаксис | Старый синтаксис |
|-------------|----------------------|-------------------|
| Формат объявления | `name: Type = ...` | `name(Types) -> Type = ...` |
| Расположение типов параметров | В аннотации типа | После имени функции в скобках |
| Пустые параметры | Обязательно `()` | Можно опустить `()` |
| **Рекомендуемая степень** | ✅ **Официальная рекомендация** | ⚠️ **Только обратная совместимость** |
| **Сценарий использования** | Весь новый код | Поддержка исторического кода |

**Причины нерекомендации:**
1. **Затраты на обучение**: несоответствие со стандартным синтаксисом, увеличивает сложность языка
2. **Согласованность**: неuniformное расположение типов параметров (одно в аннотации типа, другое после имени функции)
3. **Затраты на поддержку**: анализатор требует дополнительной обработки двух форм
4. **AI-дружелюбность**: усложняет понимание и генерацию кода AI

**Рекомендации по миграции:**
```yaoxiang
# Старый код (не рекомендуется)
mul(Int, Int) -> Int = (a, b) => a * b
square(Int) -> Int = (x) => x * x
empty() -> Void = () => {}

# Новый код (рекомендуется)
mul: (Int, Int) -> Int = (a, b) => a * b
square: (Int) -> Int = (x) => x * x
empty: () -> Void = () => {}
```

## 三、Типовая система

### 3.1 Иерархия типов

Типовая система YaoXiang является иерархической:

```
┌─────────────────────────────────────────────────────────────┐
│                    Иерархия типов YaoXiang                 │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  type (метатип)                                             │
│    │                                                        │
│    ├── Примитивные типы (Primitive Types)                  │
│    │   ├── Void, Bool                                       │
│    │   ├── Int, Uint, Float                                 │
│    │   ├── Char, String, Bytes                              │
│    │                                                        │
│    ├── Типы-конструкторы (Constructor Types)               │
│    │   ├── Name(args)              # Один конструктор (структура) │
│    │   ├── A(T) | B(U)             # Несколько конструкторов (объединение/enum) │
│    │   ├── A | B | C               # Конструкторы без параметров (enum) │
│    │   ├── tuple (T1, T2, ...)                            │
│    │   ├── list(T), dict(K, V)                           │
│    │                                                        │
│    ├── Типы функций (Function Types)                       │
│    │   fn (T1, T2, ...) -> R                               │
│    │                                                        │
│    ├── Обобщённые типы (Generic Types)                     │
│    │   List(T), Map(K, V), etc.                            │
│    │                                                        │
│    ├── Зависимые типы (Dependent Types)                    │
│    │   (n: Int) -> Type                               │
│    │                                                        │
│    └── Типы модулей (Module Types)                          │
│        Файл как модуль                                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Определение типов

```yaoxiang
# Унифицированный синтаксис типов: только конструкторы, нет ключевых слов enum/struct/union
# Правило: всё, разделённое |, является конструктором, конструктор(параметры) — это тип

# === Конструкторы без параметров (стиль enum) ===
type Color = { red | green | blue }              # Эквивалентно red() | green() | blue()

# === Конструкторы с несколькими параметрами (стиль структуры) ===

# === Обобщённый конструктор ===

# === Смешанные конструкторы ===
type Shape = circle(Float) | rect(Float, Float)

# === Конструкция значений (полностью идентично вызову функции) ===

s: Shape = circle(5.0)

# === Определение интерфейса (запись, все поля которой являются функциями) ===
type Drawable = {
    draw: (Surface) -> Void,

}

type Serializable = {
    serialize: () -> String
}

# === Реализация интерфейса (перечисление имён интерфейсов в конце типа) ===
type Point = {
    x: Float,
    y: Float,
    Drawable,        # Реализует интерфейс Drawable

}
```

## 四、Управление памятью

### 4.1 Модель владения

YaoXiang использует **модель владения** для управления памятью, у каждого значения есть единственный владелец:

```yaoxiang
# === По умолчанию Move (нулевое копирование) ===
p: Point = Point(1.0, 2.0)
p2 = p              # Move, владение передаётся, p становится недействительным

# === Ключевое слово ref = Arc (безопасное разделение) ===
shared = ref p      # Arc, потокобезопасно

spawn(() => print(shared.x))   # ✅ Безопасно

# === clone() явное копирование ===
p3 = p.clone()      # p и p3 независимы
```

### 4.2 Move-семантика (по умолчанию)

```yaoxiang
# Присваивание = Move (нулевое копирование)
p: Point = Point(1.0, 2.0)
p2 = p              # Move, p становится недействительным

# Передача параметров функции = Move
process: (p: Point) -> Void = {
    # Владение p передаётся внутрь
}

# Возвращаемое значение = Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    return p        # Move, владение передаётся
}
```

### 4.3 Ключевое слово ref (Arc)

```yaoxiang
# Ключевое слово ref создаёт Arc (подсчёт ссылок)
p: Point = Point(1.0, 2.0)
shared = ref p      # Arc, потокобезопасно

spawn(() => print(shared.x))   # ✅ Безопасно

# Arc автоматически управляет жизненным циклом
# Когда shared покидает область видимости, счётчик становится нулём и происходит освобождение
```

### 4.4 clone() явное копирование

```yaoxiang
# Используйте clone(), когда нужно сохранить оригинальное значение
p: Point = Point(1.0, 2.0)
p2 = p.clone()   # p и p2 независимы

p.x = 0.0        # ✅
p2.x = 0.0       # ✅ Взаимно не влияют
```

### 4.5 Блоки unsafe (системный уровень)

```yaoxiang
# Голые указатели можно использовать только в блоках unsafe
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p     # Голый указатель
    (*ptr).x = 0.0       # Пользователь гарантирует безопасность
}
```

### 4.6 RAII

```yaoxiang
# RAII автоматическое освобождение
with_file: (path: String) -> String = {
    file = File.open(path)  # Автоматическое открытие
    content = file.read_all()
    # Функция завершается, file автоматически закрывается
    content
}
```

### 4.7 Ограничения Send / Sync

| Ограничение | Семантика | Описание |
|-------------|-----------|----------|
| **Send** | Можно безопасно передавать между потоками | Значение можно переместить в другой поток |
| **Sync** | Можно безопасно разделять между потоками | Неизменяемая ссылка может быть разделена для другого потока |

```yaoxiang
# ref T автоматически удовлетворяет Send + Sync (Arc потокобезопасно)
p: Point = Point(1.0, 2.0)
shared = ref p

spawn(() => print(shared.x))   # ✅ Arc потокобезопасно

# Голый указатель *T не удовлетворяет Send/Sync
unsafe {
    ptr: *Point = &p         # Можно использовать только в одном потоке
}
```

### 4.9 Не реализовано

| Возможность | Причина |
|-------------|---------|
| Время жизни `'a` | Нет концепции ссылок, не требуется время жизни |
| Borrow checker | ref = Arc заменяет |
| Синтаксис заимствования `&T` | Используется Move-семантика |

## 五、Асинхронное программирование и параллелизм

> «Все существа действуют одновременно, я наблюдаю их возвращение.»——《И·Цзин, гуа Фу (Возвращение)》
>
> YaoXiang использует **spawn-модель**, вид незаметного асинхронного параллелизма на основе **ленивых вычислений**. Её основная идея: **позволить разработчикам описывать логику синхронным, последовательным мышлением, а среда выполнения языка автоматически и эффективно выполняет вычислительные единицы параллельно, как все существа действуют одновременно, и в итоге координирует их**.

> Подробнее: [Белая книга spawn-модели](YaoXiang-async-whitepaper.md) и [План реализации асинхронности](YaoXiang-async-implementation.md).

### 5.1 Основные концепции spawn-модели

#### 5.1.1 Spawn-граф: сцена для параллельного действия

Все программы компилятором преобразуются в **направленный ациклический вычислительный граф (DAG)**, называемый **spawn-графом**. Узлы представляют вычисления выражений, рёбра — зависимости по данным. Этот граф ленивый, то есть узел вычисляется только тогда, когда его выходные данные **реально нужны**.

```yaoxiang
# Компилятор автоматически строит spawn-граф
fetch_user: spawn () -> User = (id) => { ... }
fetch_posts: spawn (User) -> Posts = (user) => { ... }

main:() -> Void = () => {
    user = fetch_user(1)     # Узел A (Async(User))
    posts = fetch_posts(user) # Узел B (Async(Posts)), зависит от A

    # Узел C требует результаты A и B
    print(posts.title)       # Автоматическое ожидание: сначала гарантируем завершение A и B
}
```

#### 5.1.2 Spawn-значение: Async(T)

Любой вызов функции, помеченной `spawn fn`, немедленно возвращает значение типа `Async(T)`, называемое **spawn-значением**. Это лёгкий proxy, не являющийся реальным результатом, а представляющий **значение-будущее, находящееся в процессе выполнения**.

**Основные характеристики:**
- **Типовая прозрачность**: `Async(T)` является подтипом `T` в типовой системе и может использоваться в любом контексте, ожидающем `T`
- **Автоматическое ожидание**: когда программа доходит до операции, требующей конкретное значение типа `T`, среда выполнения автоматически приостанавливает текущую задачу и ожидает завершения вычисления
- **Нулевая заразность**: асинхронный код синтаксически и сигнатурно не отличается от синхронного

```yaoxiang
# Пример использования spawn-значения
fetch_data: spawn (String) -> JSON = (url) => { ... }

main: () -> Void = () => {
    data = fetch_data("url")  # Async(JSON)

    # Async(JSON) можно напрямую использовать как JSON
    # Автоматическое ожидание происходит при доступе к полю
    print(data.name)          # Эквивалентно data.await().name
}
```

### 5.2 Синтаксическая система spawn

Ключевое слово `spawn` имеет тройную семантику и является единственным мостом между синхронным мышлением и асинхронной реализацией:

| Официальный термин | Синтаксическая форма | Семантика | Поведение среды выполнения |
|-------------------|---------------------|----------|---------------------------|
| **Spawn-функция** | `spawn fn` | Определение вычислительной единицы, участвующей в параллельном выполнении | Её вызов возвращает `Async(T)` |
| **Spawn-блок** | `spawn { a(), b() }` | Явно объявленная область параллелизма | Задачи внутри блока принудительно выполняются параллельно |
| **Spawn-цикл** | `spawn for x in xs { ... }` | Парадигма данных-параллелизма | Тело цикла параллельно выполняется для всех элементов |

#### 5.2.1 Spawn-функция

```yaoxiang
# Используйте spawn для пометки spawn-функции
# Синтаксис полностью идентичен обычной функции, без дополнительной нагрузки

fetch_api: spawn (String) -> JSON = (url) => {
    response = HTTP.get(url)
    JSON.parse(response.body)
}

# Вложенные spawn-вызовы
process_user: (Int) -> Report = (user_id) => {
    user = fetch_user(user_id)     # Async(User)
    profile = fetch_profile(user)  # Async(Profile), зависит от user
    generate_report(user, profile) # Зависит от profile
}
```

### 5.5 Потокобезопасность: ограничения Send/Sync

YaoXiang использует похожую на Rust систему **ограничений типов Send/Sync** для гарантии потокобезопасности, устраняя гонки данных на этапе компиляции.

#### 5.5.1 Ограничение Send

**Send**: тип может безопасно **передавать владение** между потоками.

```yaoxiang
# Базовые типы автоматически удовлетворяют Send
# Int, Float, Bool, String все являются Send

# Структурные типы автоматически наследуют Send
type Point = { x: Int, y: Float }
# Point является Send, потому что Int и Float являются Send

# Типы, содержащие не-Send поля, не являются Send
type NonSend = NonSend(data: Rc(Int))
# Rc не является Send (подсчёт ссылок неатомарный), поэтому NonSend не является Send
```

#### 5.5.2 Ограничение Sync

**Sync**: тип можно безопасно **разделять ссылку** между потоками.

```yaoxiang
# Базовые типы все являются Sync
type Point = { x: Int, y: Float }
# &Point является Sync, потому что &Int и &Float являются Sync

# Типы с внутренней изменяемостью
type Counter = Counter(value: Int, mutex: Mutex(Int))
# &Counter является Sync, потому что Mutex обеспечивает внутреннюю изменяемость
```

#### 5.5.3 Spawn и потокобезопасность

```yaoxiang
# Spawn требует, чтобы параметры и возвращаемое значение удовлетворяли Send

# Допустимо: Data является Send
type Data = Data(value: Int)
task = spawn(|| => Data(42))

# Недопустимо: Rc не является Send
type SharedData = SharedData(rc: Rc(Int))
# task = spawn(|| => SharedData(Rc.new(42)))  # Ошибка компиляции!

# Решение: используйте Arc (атомарный подсчёт ссылок)
type SafeData = SafeData(value: Arc(Int))
task = spawn(|| => SafeData(Arc.new(42)))  # Arc является Send + Sync
```

#### 5.5.4 Правила вывода потокобезопасности для типов

```yaoxiang
# Структурные типы
type Struct(T1, T2) = Struct(f1: T1, f2: T2)

# Вывод Send
Struct(T1, T2): Send ⇐ T1: Send и T2: Send

# Вывод Sync
Struct(T1, T2): Sync ⇐ T1: Sync и T2: Sync

# Объединённые типы
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# Вывод Send
Result(T, E): Send ⇐ T: Send и E: Send
```

#### 5.5.5 Реализация потокобезопасности в стандартной библиотеке

| Тип | Send | Sync | Описание |
|-----|:----:|:----:|----------|
| `Int`, `Float`, `Bool` | ✅ | ✅ | Примитивные типы |
| `Arc(T)` | ✅ | ✅ | T: Send + Sync |
| `Mutex(T)` | ✅ | ✅ | T: Send |
| `RwLock(T)` | ✅ | ✅ | T: Send |
| `Channel(T)` | ✅ | ❌ | Только отправляющий конец Send |
| `Rc(T)` | ❌ | ❌ | Неатомарный подсчёт ссылок |
| `RefCell(T)` | ❌ | ❌ | Проверка заимствования во время выполнения |


```yaoxiang
# Пример потокобезопасного счётчика
type SafeCounter = SafeCounter(mutex: Mutex(Int))

main: () -> Void = () => {
    counter: Arc(SafeCounter) = Arc.new(SafeCounter(Mutex.new(0)))

    # Параллельное обновление
    spawn(|| => {
        guard = counter.mutex.lock()  # Mutex обеспечивает потокобезопасность
        guard.value = guard.value + 1
    })

    spawn(|| => {
        guard = counter.mutex.lock()
        guard.value = guard.value + 1
    })
}
```

## 七、Привязка методов и каррирование

YaoXiang использует **чистый функциональный дизайн**, реализуя бесшовные вызовы методов и каррирование через продвинутый механизм привязки, без введения ключевых слов вроде `struct`, `class` и т.д.

### 7.1 Определение основных функций

Все операции реализуются через обычные функции, первый параметр по соглашению является субъектом операции:

```yaoxiang
# === Point.yx (модуль) ===

# Унифицированный синтаксис: конструктор и есть тип
type Point = { x: Float, y: Float }

# Основная функция: первый параметр — субъект операции
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

add: (a: Point, b: Point) -> Point = {
    Point(a.x + b.x, a.y + b.y)
}

scale: (p: Point, s: Float) -> Point = {
    Point(p.x * s, p.y * s)
}

# Более сложная функция
distance_with_scale: (s: Float, p1: Point, p2: Point) -> Float = {
    dx = (p1.x - p2.x) * s
    dy = (p1.y - p2.y) * s
    (dx * dx + dy * dy).sqrt()
}
```

### 7.2 Базовая привязка методов

#### 7.2.1 Автоматическая привязка (в стиле MoonBit)

YaoXiang поддерживает автоматическую привязку на основе пространства имён, **без каких-либо дополнительных объявлений**:

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Основная функция
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# === main.yx ===

use Point

main: () -> Void = {
    p1 = Point(3.0, 4.0)
    p2 = Point(1.0, 2.0)

    # ✅ Автоматическая привязка: прямой вызов метода
    result = p1.distance(p2)  #解析为 distance(p1, p2)
}
```

**Правила автоматической привязки:**
- Функция определена внутри модуля
- Если первый параметр типа совпадает с именем модуля
- То автоматически поддерживается синтаксис вызова метода

#### 7.2.2 Опция без привязки (поведение по умолчанию)

```yaoxiang
# === Vector.yx ===

type Vector = Vector(x: Float, y: Float, z: Float)

# Внутренняя вспомогательная функция, не требует автоматической привязки

    a.x * b.x + a.y * b.y + a.z * b.z
}

# === main.yx ===

use Vector

    v1 = Vector(1.0, 0.0, 0.0)
    v2 = Vector(0.0, 1.0, 0.0)

    # ❌ Привязка невозможна: не-pub функции не привязываются автоматически
    # v1.dot_product_internal(v2)  # Ошибка компиляции!

    # ✅ Требуется прямой вызов (невидим извне модуля)
}
```

### 7.3 Синтаксис привязки на основе позиции

YaoXiang предоставляет **наиболее элегантный синтаксис привязки**, использующий позиционные метки `[n]` для точного контроля позиции привязки:

#### 7.3.1 Базовый синтаксис

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Основная функция
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}
add: (a: Point, b: Point) -> Point = {
    Point(a.x + b.x, a.y + b.y)
}
scale: (p: Point, s: Float) -> Point = {
    Point(p.x * s, p.y * s)
}

# Синтаксис привязки: Type.method = func[position]
# Обозначает: при вызове метода вызывающий объект привязывается к параметру [position] функции

Point.distance = distance[0]      # Привязка к первому параметру
Point.add = add[0]                 # Привязка к первому параметру
Point.scale = scale[0]             # Привязка к первому параметру
```

**Семантический анализ:**
- `Point.distance = distance[0]`
  - У функции `distance` два параметра: `distance(Point, Point)`
  - `[0]` означает, что вызывающий объект привязывается к первому параметру
  - Использование: `p1.distance(p2)` → `distance(p1, p2)`

#### 7.3.2 Мульти-позиционная объединённая привязка

```yaoxiang
# === Math.yx ===

# Функция: scale, point1, point2, extra1, extra2
calculate: (scale: Float, a: Point, b: Point, x: Float, y: Float) -> Float = { ... }

# === Point.yx ===

type Point = { x: Float, y: Float }

# Привязка нескольких позиций
Point.calc1 = calculate[1, 2]      # Привязка scale и point1
Point.calc2 = calculate[1, 3]      # Привязка scale и point2
Point.calc3 = calculate[2, 3]      # Привязка point1 и point2

# === main.yx ===

use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# 1. Привязка[1,2] - остаются 3,4,5
f1 = p1.calc1(2.0)  # Привязка scale=2.0, point1=p1
# f1 теперь требует p2, x, y
result1 = f1(p2, 10.0, 20.0)  # calculate(2.0, p1, p2, 10.0, 20.0)

# 2. Привязка[1,3] - остаются 2,4,5
f2 = p2.calc2(2.0)  # Привязка scale=2.0, point2=p2
# f2 теперь требует point1, x, y
result2 = f2(p1, 10.0, 20.0)  # calculate(2.0, p1, p2, 10.0, 20.0)

# 3. Привязка[2,3] - остаются 1,4,5
f3 = p1.calc3(p2)  # Привязка point1=p1, point2=p2
# f3 теперь требует scale, x, y
result3 = f3(2.0, 10.0, 20.0)  # calculate(2.0, p1, p2, 10.0, 20.0)
```

#### 7.3.3 Порядок заполнения оставшихся параметров

**Основное правило**: после привязки оставшиеся параметры заполняются в **исходном порядке функции**, пропуская уже привязанные позиции.

```yaoxiang
# Предположим функция: func(p1, p2, p3, p4, p5)

# Привязка первого и третьего параметров
Type.method = func[1, 3]

# При вызове:
method(p2_value, p4_value, p5_value)

# Соответствие:
func(p1_bound, p2_value, p3_bound, p4_value, p5_value)
# Оставшиеся параметры: 2,4,5 заполняются по порядку
```

#### 7.3.4 Преимущества проверки типов

```yaoxiang
# ✅ Допустимая привязка
Point.distance = distance[0]          # distance(Point, Point)
Point.calc = calculate[1, 2]          # calculate(scale, Point, Point, ...)

# ❌ Недопустимая привязка (ошибка компилятора)
Point.wrong = distance[5]             # Пятый параметр не существует
Point.wrong = distance[0]             # Параметры начинаются с 1
Point.wrong = distance[1, 2, 3, 4]    # Превышено количество параметров функции
```

### 7.4 Тонкий контроль каррированной привязки

```yaoxiang
# === Math.yx ===
```

### 7.6 Область действия и правила привязки

#### 7.6.1 Действие pub

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Не-pub функция
internal_distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# pub функция
pub distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# === main.yx ===

use Point

# Автоматическая привязка работает только для pub функций
p1.distance(p2)      # ✅ distance является pub, можно автоматически привязать
# p1.internal_distance(p2)  # ❌ Не pub, невозможно привязать
```

#### 7.6.2 Механизм автоматической привязки pub

Функции, объявленные с `pub`, компилятор автоматически привязывает к определённым в том же файле типам:

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Объявление с pub, компилятор автоматически привязывает
pub distance: (p1: Point, p2: Point) -> Float = {
    dx = p1.x - p2.x
    dy = p1.y - p2.y
    (dx * dx + dy * dy).sqrt()
}

pub translate: (self: Point, dx: Float, dy: Float) -> Point = {
    Point(self.x + dx, self.y + dy)
}

# Компилятор автоматически выводит и выполняет привязку:
# Point.distance = distance[0]
# Point.translate = translate[0]

# === main.yx ===

use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# ✅ Функциональный вызов
d = distance(p1, p2)

# ✅ Синтаксический сахар ООП (автоматическая привязка)
d2 = p1.distance(p2)
p3 = p1.translate(1.0, 1.0)
```

**Правила автоматической привязки:**
1. Функция определена в файле модуля (в том же файле, что и тип)
2. Функция содержит этот тип среди параметров
3. Экспортирована с `pub`
4. Компилятор автоматически выполняет `Type.method = function[0]`

**Преимущества:**
- Не требуется вручную писать объявления привязки
- Код более лаконичен
- Исключается забывание или ошибки привязки

#### 7.6.3 Привязка внутри модуля


|-------------|----------|
| **Нулевая синтаксическая нагрузка** | Автоматическая привязка без каких-либо объявлений |
| **Точный контроль позиции** | `[n]` точно указывает позицию привязки |
| **Мульти-позиционная** | Поддержка `[1, 2, 3]` для привязки нескольких параметров |
| **Типовая безопасность** | Компилятор проверяет допустимость позиций привязки |
| **Без ключевых слов** | Не требуются `bind` или другие ключевые слова |
| **Гибкое каррирование** | Поддержка привязки параметров в любой позиции |
| **Контроль через pub** | Только pub функции могут быть привязаны извне |

### 7.8 Отличия от традиционных методов привязки

| Традиционные языки | YaoXiang |
|-------------------|----------|
| `obj.method(arg)` | `obj.method(arg)` |
| Требуется определение класса/метода | Нужны только функция + объявление привязки |
| Синтаксис `class { method() {} }` | Синтаксис `Type.method = func[n]` |
| Наследование, полиморфизм | Чистый функциональный стиль, без наследования |
| Поиск в таблице методов | Компиляция времени привязки, без накладных расходов выполнения |

**Ключевое преимущество**: Привязка YaoXiang — это **механизм времени компиляции** с нулевыми накладными расходами выполнения, сохраняя при этом чистоту и гибкость функционального программирования.

## 九、Соглашение о централизации типов (ключевая философия дизайна)

### 9.1 Обзор соглашения

Основное проектное соглашение YaoXiang — **«объявление в первую очередь, типы централизованы»**. Это соглашение является основой AI-дружелюбности языка и эффективности разработки.

```yaoxiang
# ✅ Основное соглашение: информация о типах унифицирована в строке объявления
add: (a: Int, b: Int) -> Int = a + b

# ❌ Избегайте: информация о типах разбросана в реализации
add: (a: Int, b: Int) -> Int = a + b
```

### 9.2 Пять ключевых преимуществ соглашения

#### 1. Синтаксическая согласованность
```yaoxiang
# Все объявления следуют одинаковому формату
x: Int = 42                           # Переменная
name: String = "YaoXiang"             # Переменная
add: (a: Int, b: Int) -> Int = a + b  # Функция
inc: (x: Int) -> Int = x + 1          # Функция
type Point = { x: Float, y: Float } # Тип
```

#### 2. Разделение объявления и реализации
```yaoxiang
# Строка объявления предоставляет полную информацию о типах
add: (a: Int, b: Int) -> Int = a + b
# └────────────────────┘
#   Полная сигнатура функции

# Тело реализации сосредоточено на бизнес-логике
# (a, b) => a + b  Не нужно заботиться о типах, только реализовать функциональность
```

#### 3. AI-дружелюбность
```yaoxiang
# Процесс обработки AI:
# 1. Читать строку объявления → Полностью понять сигнатуру функции
# 2. Сгенерировать реализацию → Не требуется анализ вывода типов
# 3. Изменить тип → Только изменить строку объявления, не влияет на реализацию

# Сравнение: при разбросанных типах
add: (a: Int, b: Int) -> Int = a + b
# AI требуется: анализ тела реализации для извлечения информации о типах → Более сложно, легче ошибиться
```

#### 4. Более безопасные модификации
```yaoxiang
# Изменение типа параметра
# Было: add: (a: Int, b: Int) -> Int = a + b
# Изменено: add: (Float, Float) -> Float = (a, b) => a + b
# Тело реализации: (a, b) => a + b  Не требует изменений!

# При разбросанных типах:
# Было: add: (a: Int, b: Int) -> Int = a + b
# Изменено: add: (a: Float, b: Float) -> Float = a + b  # Требуется изменить в двух местах
```

#### 5. Каррирование дружелюбно
```yaoxiang
# Каррированные типы понятны с первого взгляда
add_curried: (a: Int) -> (b: Int) -> Int = a + b
#              └─────────────┘
#              Каррированная сигнатура

# Композиция функций как объект первого класса
compose: (Int -> Int, Int -> Int) -> Int -> Int = (f, g) => x => f(g(x))
```

### 9.3 Правила применения соглашения

#### Правило 1: Параметры должны иметь типы в объявлении
```yaoxiang
# ✅ Правильно
add: (a: Int, b: Int) -> Int = a + b

# ❌ Ошибка
add: (a: Int, b: Int) -> Int = a + b            # Типы параметров отсутствуют
identity: (T: Type) -> ((x: T) -> T) = x                # Типы параметров отсутствуют
```

#### Правило 2: Возвращаемый тип может быть выведен, но рекомендуется аннотировать
```yaoxiang
# ✅ Рекомендуется: полная аннотация
get_num: () -> Int = () => 42

# ✅ Допустимо: возвращаемый тип выводится
get_num: () = () => 42

# ✅ Пустая функция выводится как Void
empty: () = () => {}
```

#### Правило 3: Временные аннотации типов внутри Lambda
```yaoxiang
# ✅ Правильно: опираемся на типы в объявлении
add: (a: Int, b: Int) -> Int = a + b

# ⚠️ Можно, но не рекомендуется: повторная аннотация внутри Lambda
add: (Int, Int) -> Int = (a: Int, b: Int) => a + b

# ❌ Ошибка: отсутствует аннотация в объявлении
add: (a: Int, b: Int) -> Int = a + b
```

#### Правило 4: Старый синтаксис следует той же идеологии
```yaoxiang
# В старом синтаксисе тоже старайтесь предоставлять информацию о типах в позиции объявления
# Хотя формат другой, идеология一致:
# - Строка объявления содержит основную информацию о типах
# - Тело реализации относительно лаконично
add(Int, Int) -> Int = (a, b) => a + b
```

### 9.4 Соотношение соглашения и вывода типов

```yaoxiang
# Соглашение не препятствует выводу типов, а направляет направление вывода

# 1. Полная аннотация (без вывода)
add: (a: Int, b: Int) -> Int = a + b

# 2. Частичный вывод (объявление предоставляет типы параметров)
add: (Int, Int) = (a, b) => a + b  # Возвращаемый тип выводится

# 3. Вывод пустой функции
empty: () = () => {}  # Выводится как () -> Void
```

### 9.5 Преимущества соглашения для AI

**Процесс генерации кода AI:**

1. **Чтение требований** → Генерация объявления
   ```
   Требование: функция сложения
   Генерация: add: (Int, Int) -> Int = (a, b) => ???
   ```

2. **Заполнение реализации** → Без анализа типов
   ```
   Реализация: add: (a: Int, b: Int) -> Int = a + b
   ```

3. **Изменение типов** → Только объявление
   ```
   Изменение: add: (Float, Float) -> Float = (a, b) => a + b
   Реализация: (a, b) => a + b  Остаётся без изменений
   ```

**Сравнение обработки AI без соглашения:**
```
Требование: функция сложения
AI требуется:
  1. Вывести типы параметров
  2. Вывести возвращаемый тип
  3. Сгенерировать тело реализации
  4. Проверить согласованность
  5. Обработать сложные обновления при изменении типов

Результат: Более сложно, легче ошибиться
```

### 9.6 Философское значение соглашения

Это соглашение воплощает ключевую идею YaoXiang:

- **Объявление — это документация**: строка объявления — это полная документация функции
- **Тип — это контракт**: информация о типах — это контракт между вызывающим и реализатором
- **Логика — это реализация**: тело реализации сосредоточено только на «что делать», не на «какие типы»
- **Инструменты — это помощники**: типовая система, AI-инструменты могут работать на основе чётких объявлений

### 9.7 Практическое сравнение

#### Полный пример: модуль калькулятора

```yaoxiang
# === Рекомендуемый подход: соглашение о централизации типов ===

# Объявления модуля
pub add: (a: Int, b: Int) -> Int = a + b
pub multiply: (a: Int, b: Int) -> Int = a * b

# Функции высшего порядка
pub apply_twice: (f: Int -> Int, x: Int) -> Int = f(f(x))

# Каррированные функции
pub make_adder: (x: Int) -> (Int) -> Int = y => x + y

# Обобщённые функции
pub map: (A: Type, B: Type) -> ((f: (A) -> B, xs: List(A)) -> List(B)) = case xs of
  [] => []
  (x :: rest) => f(x) :: map(f, rest)

# Определения типов
type Point = { x: Float, y: Float }
pub distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# === Не рекомендуемый подход: разбросанные типы ===

# Типы параметров в Lambda
add: (a: Int, b: Int) -> Int = a + b
multiply = (a: Int, b: Int) => a * b

# Типы функций высшего порядка разбросаны
apply_twice = (f: (Int) -> Int, x: Int) => f(f(x))

# Типы каррирования разбросаны
make_adder = (x: Int) => (y: Int) => x + y

# Типы обобщений разбросаны

  [] => []
  (x :: rest) => f(x) :: map(f, rest)
```

#### Сравнение поддержки кода

```yaoxiang

add: (a: Float, b: Float) -> Float = a + b

#              Изменение в строке объявления   Тело реализации остаётся без изменений

# Было

# После изменения
add: (a: Float, b: Float) -> Float = a + b
#     ↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑
#     Все типы параметров требуют изменения
```

#### Сравнение AI-辅助编程

```yaoxiang

pub manhattan = (a: Point, b: Point) => Float => {
```

### 9.8 Чек-лист применения соглашения

При написании кода YaoXiang можно использовать следующий чек-лист:

- [ ] Все объявления функций имеют полные аннотации типов в строке объявления
- [ ] Типы параметров указаны в объявлении, не в Lambda
- [ ] Возвращаемые типы по возможности аннотированы в объявлении
- [ ] Объявления переменных используют формат `name: Type = value`
- [ ] Тело Lambda остаётся лаконичным, без повторения информации о типах
- [ ] Используется новый синтаксис вместо старого
- [ ] Сложные типы используют type-определения, сохраняя чёткость объявлений

```yaoxiang
# Условия
if x > 0 {
    "positive"
} elif x == 0 {
    "zero"
} else {
    "negative"
}

# Циклы
for i in 0..10 {
    print(i)
}
```

### 10.3 Пример привязки методов

```yaoxiang
# === Point.yx ===

type Point = { x: Float, y: Float }

# Основная функция
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# Автоматическая привязка
Point.distance = distance[0]

# === main.yx ===

use Point

main: () -> Void = {
    p1 = Point(3.0, 4.0)
    p2 = Point(1.0, 2.0)

    # Использование привязки
    d = p1.distance(p2)  # distance(p1, p2)
    print(d)
}
```

### 10.4 Пример каррированной привязки

```yaoxiang
# === Math.yx ===

distance_with_scale: (scale: Float, a: Point, b: Point) -> Float = {
    dx = (p1.x - p2.x) * scale
    dy = (p1.y - p2.y) * scale
    (dx * dx + dy * dy).sqrt()
}

# === Point.yx ===

type Point = { x: Float, y: Float }

Point.distance_scaled = distance_with_scale[2]  # Привязка ко второму параметру

# === main.yx ===

use Point

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# Использование привязки
f = p1.distance_scaled(2.0)  # Привязка scale и p1
result = f(p2)               # Финальный вызов

# Или использовать напрямую
result2 = p1.distance_scaled(2.0, p2)
```
