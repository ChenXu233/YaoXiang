# Спецификация модели конкурентности

> **Статус**: В данном документе описывается новая модель конкурентности языка YaoXiang. Она заменяет старую схему на основе аннотаций `@block`/`@eager`/`@auto`, трейтов `Send`/`Sync` и `Mutex`/`RwLock`. Часть содержимого ещё не реализована — ориентируйтесь на фактическое поведение компилятора.

В данном файле определяется спецификация модели конкурентности языка программирования YaoXiang, включая семантику блоков `{}`, примитив конкурентности `spawn`, обработку ошибок и типы ресурсов.

## Глава 1: Обзор

### 1.1 Сущность блока {}

В YaoXiang блок `{}` представляет собой **вычислительную единицу с зависимостным управлением**.

| Атрибут | Описание |
|---------|----------|
| Зависимостное управление | При выполнении блок проверяет готовность всех внутренних переменных; если все готовы — выполняется немедленно, иначе блокируется в ожидании |
| Момент выполнения | Определяется зависимостями, не связан с «немедленным» или «отложенным» |
| Возвращаемое значение | Явно возвращается через `return`; без `return` по умолчанию возвращается `Void` |
| Единообразие синтаксиса | Семантика одинакова независимо от того, находится ли блок в теле функции, в инициализации переменной или после `spawn` |
| Изоляция области видимости | Переменные строго ограничены внутри `{}`, не проникают во внешнюю область видимости |

```yaoxiang
// Пример зависимостного управления
x = compute_x()        // x готов
y = compute_y()        // y готов
result = {
    // Зависит от x и y; выполняется сразу после готовности обоих
    return x + y
}
```

### 1.2 Правила возврата

Правила возврата в YaoXiang унифицированы и однозначны:

| Запись | Возвращаемое значение | Описание |
|--------|----------------------|----------|
| `= expr` (без фигурных скобок) | Непосредственно `expr` | Выражение является значением |
| `= { ... }` (с фигурными скобками) | Обязателен `return`, иначе `Void` | Блок требует явного возврата |

```yaoxiang
// Без фигурных скобок: непосредственный возврат
add: (a: Int, b: Int) -> Int = a + b

// С фигурными скобками: обязателен return
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// С фигурными скобками, но без return: возврат Void
log: (message: String) -> Void = {
    print(message)  // Без return, возвращает Void
}
```

### 1.3 Семантика блока spawn

`spawn { ... }` является единственным примитивом параллелизма в YaoXiang.

**Основные правила**:
- **Непосредственные присваивания** внутри блока spawn создают параллельные задачи
- Присваивания внутри вложенных `{}` не являются независимыми задачами
- Весь блок spawn синхронно блокируется, ожидая завершения всех задач, и возвращает результат
- Никаких коллбэков, `await` или аннотаций

```yaoxiang
// Две задачи выполняются параллельно
(a, b) = spawn {
    fetch("url1"),      // Задача 1
    fetch("url2")       // Задача 2
}
// Продолжаем после завершения обеих задач
```

### 1.4 Ментальная модель пользователя

> Написанный вами обычный код выполняется последовательно.
> Когда вы хотите, чтобы несколько вещей происходили одновременно, поместите их в блок `spawn { ... }`.
> Каждое непосредственное присваивание внутри блока начинается немедленно (параллельно), а нужный вам результат автоматически ожидается.
> Весь блок ждёт завершения всех задач и возвращает конечный результат.
> Никаких коллбэков, никакого `await`, никаких странных аннотаций.

## Глава 2: Синтаксис и семантика

### 2.1 Обычный код

Обычный код (вне блока spawn) выполняется **последовательно**.

```yaoxiang
a = compute_a()     // Выполняется первым
b = compute_b(a)    // Зависит от a; выполняется после завершения a
c = compute_c(b)    // Зависит от b; выполняется после завершения b
```

### 2.2 Блок spawn

```
SpawnBlock  ::= '(' Pattern (',' Pattern)* ')' '=' 'spawn' '{' SpawnBody '}'
SpawnBody   ::= Assignment (',' Assignment)*
```

**Семантика**:
1. Непосредственные присваивания внутри блока spawn выполняются как независимые параллельные задачи
2. Результат каждой задачи связывается с соответствующей переменной шаблона
3. Весь блок блокируется до завершения всех задач
4. Возвращает кортеж всех результатов

```yaoxiang
// Одиночная задача
result = spawn {
    fetch("url")
}

// Несколько задач
(a, b, c) = spawn {
    fetch("url1"),
    fetch("url2"),
    fetch("url3")
}
```

### 2.3 spawn в теле функции

Тело функции само по себе является блоком `{}`, в котором можно использовать `spawn`.

```yaoxiang
fetch_and_parse: (urls: List(String)) -> List(Data) = {
    results = spawn for url in urls {
        parsed = parse(fetch(url))
    }
    return results
}
```

### 2.4 spawn в цикле

```
SpawnFor    ::= Identifier '=' 'spawn' 'for' Identifier 'in' Expr '{' Assignment '}'
```

**Семантика**: Цикл с распределением данных; каждая итерация является независимой задачей.

```yaoxiang
// Параллельная обработка каждого элемента списка
results = spawn for item in items {
    result = process(item)
}
```

> **Примечание**: Тело цикла `spawn for` является независимой задачей и не поддерживает совместное изменяемое состояние между итерациями. Для агрегации результатов следует использовать `spawn for` для сбора результатов с последующей обработкой извне (см. пример ниже).

```yaoxiang
// Правильно: параллельная обработка с последующей агрегацией
transformed = spawn for item in items {
    result = transform(item)
}
total = sum(transformed)   // Последовательная агрегация
```

### 2.5 Вложенный spawn

Блоки spawn могут быть вложенными; вложенный spawn создаёт новую конкурентную область.

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

**Примечание**: Непосредственные присваивания во вложенном spawn являются задачами; внешний spawn не проникает внутрь.

## Глава 3: Взаимодействие с моделью владения

### 3.1 Семантика Move

Move является семантикой по умолчанию в YaoXiang (нулевое копирование).

```yaoxiang
data = load_data()
result = spawn {
    process(data)   // Владение data перемещается в блок spawn
}
// data здесь недоступна (перемещена)
```

**Правила**:
- После входа переменной в блок spawn она больше не может использоваться извне
- Если требуется совместное использование между несколькими задачами, используйте `ref`

### 3.2 Заёмные токены

`&T` и `&mut T` являются компиляторными доказательствами прав нулевого размера и **не могут пересекать границы задач**.

```yaoxiang
data = load_data()
ref_data = &data

// Ошибка компиляции: заёмный токен не может пересекать границу задачи
result = spawn {
    process(ref_data)   // Ошибка!
}
```

### 3.3 Совместное использование через ref

`ref` является единственным способом совместного использования данных между областями видимости.

```yaoxiang
data = load_data()
shared = ref data       // Компилятор автоматически выбирает Rc или Arc

result = spawn {
    process_a(shared),  // Совместная ссылка
    process_b(shared)   // Совместная ссылка
}
```

**Выбор компилятором (консервативная стратегия)**:

| Условие | Выбор |
|---------|-------|
| По умолчанию | `Arc` (приоритет безопасности) |
| Компилятор может доказать, что данные используются только внутри одной задачи | `Rc` (без накладных расходов атомарных операций) |

### 3.4 Захват замыканием

Захват замыканием = Move; замыкание может быть передано только одной задаче.

```yaoxiang
data = load_data()
fn = (x: Int) -> Int = data.value + x   // Замыкание захватывает data через move

// Ошибка компиляции: замыкание может использоваться только одной задачей
result = spawn {
    fn(1),      // Использование замыкания
    fn(2)       // Ошибка! Замыкание уже перемещено
}
```

**Правильный подход**: создайте независимые замыкания для каждой задачи или используйте `ref`.

```yaoxiang
data = load_data()
shared = ref data

result = spawn {
    ((x: Int) -> Int = shared.value + x)(1),
    ((x: Int) -> Int = shared.value + x)(2)
}
```

## Глава 4: Обработка ошибок

### 4.1 Оператор ?

Оператор `?` используется для явного распространения ошибок; семантика аналогична Rust.

```yaoxiang
read_file: (path: FilePath) -> Result(String, IoError) = {
    content = open(path)?      // При ошибке — немедленное распространение
    return content.read_all()
}
```

### 4.2 Распространение ошибок внутри блока spawn

**Правила**:
1. Ожидание завершения всех задач (даже если некоторые уже завершились с ошибкой)
2. Распространение первой встреченной ошибки
3. Явная пометка точек распространения ошибок через `?`

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // Может завершиться с ошибкой
    fetch("url2")?      // Может завершиться с ошибкой
}
// Если любая задача завершилась с ошибкой, весь блок spawn распространяет первую ошибку
```

### 4.3 Типы ошибок

**Автоматическая генерация**: компилятор автоматически генерирует объединённый тип ошибки (аналогично объединённым типам TypeScript).

```yaoxiang
// Компилятор выводит тип ошибки как HttpError | IoError
(a, b) = spawn {
    fetch("url"),           // Может выбросить HttpError
    read_file("data.txt")  // Может выбросить IoError
}
```

**Ручное переопределение**: пользователь может вручную определить унифицированный тип ошибки.

```yaoxiang
AppError: Type = {
    Http: (HttpError) -> Self,
    Io: (IoError) -> Self,
    Parse: (ParseError) -> Self
}

process: (url: String, path: FilePath) -> Result(Data, AppError) = {
    (a, b) = spawn {
        fetch(url).map_err(AppError.Http)?,
        read_file(path).map_err(AppError.Io)?
    }
    return parse(a + b).map_err(AppError.Parse)?
}
```

## Глава 5: Типы ресурсов и побочные эффекты

### 5.1 Встроенные типы ресурсов

| Тип ресурса | Описание | Поведение компилятора |
|--------------|----------|----------------------|
| `FilePath` | Путь в файловой системе | Операции с одним путём автоматически сериализуются |
| `HttpUrl` | HTTP-эндпоинт | Операции с одним URL автоматически сериализуются |
| `DBUrl` | Подключение к базе данных | Операции с одним подключением автоматически сериализуются |
| `Console` | Стандартный вывод | Все операции с Console автоматически сериализуются |

```yaoxiang
// Операции с одним файлом автоматически сериализуются
(a, b) = spawn {
    read_file("data.txt"),      // Выполняется первым
    write_file("data.txt", x)   // Ожидает завершения чтения
}
```

### 5.2 Пользовательские типы ресурсов

Пользовательские типы ресурсов требуют явной пометки.

```yaoxiang
Database: Type = {
    connection_string: String,
    query: (db: Database, sql: String) -> Result(Rows, DbError)
}
```

### 5.3 Отслеживание побочных эффектов

Компилятор отслеживает использование типов ресурсов, обеспечивая конкурентную безопасность.

```yaoxiang
// Предупреждение компилятора: операции с Console могут перемежаться
spawn {
    print("Hello"),     // Может перемежаться со следующей строкой
    print("World")
}

// Правильно: явная сериализация
spawn {
    print("Hello\nWorld")
}
```

## Глава 6: Поведение компилятора

### 6.1 DAG-анализ

Компилятор выполняет анализ зависимостей блока spawn во время компиляции (DAG-анализ), определяя:
1. Какие выражения могут выполняться параллельно
2. Какие должны выполняться последовательно
3. Как распределять задачи

```yaoxiang
(a, b, c) = spawn {
    x = fetch("url1"),      // Задача 1
    y = fetch("url2"),      // Задача 2 (параллельно с задачей 1)
    z = process(x, y)       // Задача 3 (зависит от x и y, должна ожидать)
}
```

### 6.2 Выбор Rc/Arc (консервативная стратегия)

Компилятор использует **консервативную стратегию**, по умолчанию выбирая `Arc` для обеспечения потокобезопасности:

| Условие | Выбор | Причина |
|---------|-------|---------|
| По умолчанию (невозможно доказать безопасность) | `Arc` | Приоритет безопасности, избежание гонок данных |
| Компилятор может **доказать**, что данные используются только внутри одной задачи | `Rc` | Без накладных расходов атомарных операций |

**Описание стратегии**:
- **По умолчанию `Arc`**: когда компилятор не может определить, используется ли `ref` только внутри одной задачи, выбирается `Arc` (консервативный выбор)
- **Понижение до `Rc`**: только когда компилятор может **доказать** через DAG-анализ, что данные никогда не будут передаваться между задачами
- **Лучше медленнее, чем с ошибкой**: дополнительные накладные расходы `Arc` несопоставимы с риском гонок данных

```yaoxiang
data = load_data()

// По умолчанию: компилятор выбирает Arc (консервативная стратегия)
result = spawn {
    shared = ref data
    process(shared)
}

// Только когда компилятор может доказать использование только в одной задаче: понижение до Rc
// (требуется способность DAG-анализа компилятора однозначно исключить передачу между задачами)
```

### 6.4 Предупреждение об отсутствии параллелизма

Если задачи внутри блока spawn не имеют реальной возможности для параллельного выполнения, компилятор выдаёт предупреждение.

```yaoxiang
// Предупреждение компилятора: нет возможности для параллелизма
result = spawn {
    a = fetch("url")    // Единственная задача
}
// Рекомендация: используйте обычный код
result = fetch("url")
```

### 6.4 Обнаружение конфликтов ресурсов

Компилятор обнаруживает потенциальные конфликты типов ресурсов.

```yaoxiang
// Ошибка компиляции: параллельная запись в один файл
spawn {
    write_file("data.txt", "a"),
    write_file("data.txt", "b")  // Ошибка!
}
```

## Глава 7: Сравнение со старым дизайном

### 7.1 Устаревшие возможности

| Старая возможность | Статус | Замена |
|---------------------|--------|--------|
| Аннотации `@block`, `@eager`, `@auto` | Устарело | Нет; зависимостное управление обрабатывается автоматически |
| Полнопрограммный автоматический DAG-анализ | Устарело | Анализ только внутри блоков spawn |
| Трейты `Send`, `Sync` | Устарело | Владение + ref обрабатываются автоматически |
| future/неблокирующие дескрипторы | Устарело | Блок spawn синхронно возвращает результат |
| `Mutex[T]`, `Atomic[T]`, `RwLock[T]` | Устарело | ref автоматически выбирает Rc/Arc |

### 7.2 Изменение философии дизайна

**Старая модель**:
- Явные аннотации для управления конкурентным поведением
- Сложные trait-ограничения
- Асинхронная модель программирования

**Новая модель**:
- Зависимостное управление, неявный параллелизм
- Владение + ref упрощают совместное использование
- Синхронная модель программирования; блок spawn блокируется и возвращает результат

### 7.3 Руководство по миграции

```yaoxiang
// Старый код (псевдокод для демонстрации стиля старой модели; @block/let/await не являются ключевыми словами YaoXiang)
@block async fetch_data(): Future<Data> = {
    let data = await fetch("url")
    return data
}

// Новый код
fetch_data: () -> Data = {
    data = fetch("url")     // Синхронный вызов
    return data
}

// Конкурентная версия
fetch_multiple: (urls: List(String)) -> List(Data) = {
    results = spawn for url in urls {
        result = fetch(url)
    }
    return results
}
```

## Приложение: Шпаргалка по синтаксису

### A.1 Операторы spawn

```
SpawnBlock  ::= '(' Pattern (',' Pattern)* ')' '=' 'spawn' '{' SpawnBody '}'
SpawnFor    ::= Identifier '=' 'spawn' 'for' Identifier 'in' Expr '{' Assignment '}'
SpawnStmt   ::= SpawnBlock | SpawnFor
SpawnBody   ::= Assignment (',' Assignment)*
```

### A.2 Обработка ошибок

```
Expr '?'              // Распространение ошибки (тип Result)
```

### A.3 Выражение ref

```
RefExpr     ::= 'ref' Expr
```

### A.4 Объявление типа ресурса

```
ResourceDecl ::= Identifier ':' 'Type' '=' RecordType
```
