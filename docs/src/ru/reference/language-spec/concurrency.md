# Модель параллелизма — Спецификация

> **Статус**: Официальная спецификация. Основана на RFC-024 (модель параллелизма), RFC-009 (модель владения), RFC-008 (архитектура runtime).

Настоящий документ определяет спецификацию модели параллелизма языка программирования YaoXiang, включая семантику блоков `{}`, примитив параллелизма `spawn`, взаимодействие с владением, обработку ошибок и типы ресурсов.

**Ключевой принцип проектирования — одна примитива, одно правило**:

```
spawn { ... }               ← единственный примитив параллелизма
прямое присваивание создаёт задачи   ← единственное правило
синхронное блокирующее ожидание      ← единственное поведение
```

## Глава 1: Обзор

### 1.1 Природа блока `{}`

В YaoXiang `{}` — это **управляемая зависимостями вычислительная единица**.

| Свойство | Описание |
|----------|----------|
| Управление зависимостями | При выполнении блок проверяет готовность всех внутренних переменных; если всё готово — выполняется немедленно, иначе блокируется в ожидании |
| Момент выполнения | Определяется зависимостями, не зависит от «немедленного» или «отложенного» выполнения |
| Возвращаемое значение | Используйте `return` для явного возврата; без `return` по умолчанию возвращается `Void` |
| Единый синтаксис | Семантика одинакова независимо от того, где блок находится: в теле функции, инициализации переменной или после `spawn` |
| Изоляция области видимости | Переменные строго ограничены внутренней областью `{}`, не «просачиваются» во внешнюю область |

```yaoxiang
// Пример управления зависимостями
x = compute_x()        // x готова
y = compute_y()        // y готова
result = {
    // Зависит от x и y; выполняется сразу, как обе готовы
    return x + y
}
```

### 1.2 Правила возврата

| Запись | Возвращаемое значение | Описание |
|--------|----------------------|----------|
| `= expr` (без фигурных скобок) | Прямой возврат `expr` | Выражение и есть значение |
| `= { ... }` (с фигурными скобками) | Требуется `return`, иначе возвращается `Void` | Блок требует явного возврата |

```yaoxiang
// Без фигурных скобок: прямой возврат
add: (a: Int, b: Int) -> Int = a + b

// С фигурными скобками: требуется return
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// С фигурными скобками, но без return: возврат Void
log: (message: String) -> Void = {
    print(message)  // нет return, возвращается Void
}
```

### 1.3 Семантика блока spawn

`spawn { ... }` — это **единственный примитив параллелизма** в YaoXiang.

**Ключевые правила**:
- **Прямые присваивания** внутри блока spawn создают параллельные задачи
- Присваивания во вложенных `{}` не считаются отдельными задачами
- Весь блок spawn синхронно блокируется и ожидает завершения всех задач, после чего возвращает результат
- Никаких обратных вызовов (callbacks), `await` или аннотаций

```yaoxiang
// Две задачи выполняются параллельно
(a, b) = spawn {
    fetch("url1"),      // задача 1
    fetch("url2")       // задача 2
}
// Ожидание завершения обеих задач
```

### 1.4 Ментальная модель пользователя

> Ваш обычный код выполняется последовательно.
> Если вы хотите, чтобы несколько дел выполнялись одновременно, поместите их в блок `spawn { ... }`.
> Каждое прямое присваивание в блоке стартует немедленно (параллельно); результаты, которые вам нужны, будут автоматически ожидаться.
> Весь блок дождётся завершения всех задач и вернёт итоговый результат.
> Никаких обратных вызовов, никаких `await`, никаких странных аннотаций.

## Глава 2: Синтаксис и семантика

### 2.1 Обычный код

Обычный код (вне блоков spawn) выполняется **последовательно**.

```yaoxiang
a = compute_a()     // выполняется первой
b = compute_b(a)    // зависит от a; выполняется после a
c = compute_c(b)    // зависит от b; выполняется после b
```

### 2.2 Блок spawn

```
SpawnBlock  ::= '(' Pattern (',' Pattern)* ')' '=' 'spawn' '{' SpawnBody '}'
SpawnBody   ::= Assignment (',' Assignment)*
```

**Семантика**:
1. Прямые присваивания внутри блока spawn выполняются как независимые задачи параллельно
2. Результат каждой задачи связывается с соответствующей переменной паттерна
3. Весь блок блокируется до завершения всех задач
4. Возвращается кортеж всех результатов

```yaoxiang
// Одна задача
result = spawn {
    fetch("url")
}

// Несколько задач
(a, b, c) = spawn {
    fetch("url1"),
    fetch("url2"),
    fetch("url3")
}
```

### 2.3 spawn в теле функции

Тело функции само по себе является блоком `{}`, в котором можно использовать `spawn`.

```yaoxiang
fetch_and_parse: (urls: List(String)) -> List(Data) = {
    results = spawn for url in urls {
        parsed = parse(fetch(url))
    }
    return results
}
```

### 2.4 spawn в цикле

```
SpawnFor    ::= Identifier '=' 'spawn' 'for' Identifier 'in' Expr '{' Assignment '}'
```

**Семантика**: параллельный по данным цикл, каждая итерация — независимая задача.

```yaoxiang
// Параллельная обработка каждого элемента списка
results = spawn for item in items {
    result = process(item)
}
```

> **Внимание**: тело цикла `spawn for` представляет собой независимые задачи и не поддерживает разделение изменяемого состояния между итерациями. Если требуется агрегирование результатов, используйте `spawn for` для сбора результатов и обрабатывайте их снаружи.

```yaoxiang
// Правильно: параллельная обработка, агрегация снаружи
transformed = spawn for item in items {
    result = transform(item)
}
total = sum(transformed)   // последовательная агрегация
```

### 2.5 Вложенные spawn

Блоки spawn могут быть вложенными; внутренний spawn создаёт новую область параллелизма.

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

Только прямые присваивания внутреннего spawn являются задачами; внешний spawn не «просвечивает» сквозь него.

## Глава 3: Взаимодействие с моделью владения

### 3.1 Семантика Move

Move — это семантика YaoXiang по умолчанию (zero-copy). После того как переменная перемещается в блок spawn, её нельзя использовать снаружи.

```yaoxiang
data = load_data()
result = spawn {
    process(data)   // владение data перемещено в блок spawn
}
// data здесь недоступна (перемещена)
```

### 3.2 Токены заимствования

`&T` и `&mut T` — это доказательства разрешений нулевого размера на этапе компиляции; они **не могут пересекать границу задач**. Это не особое правило — токены являются доказательствами разрешений на этапе компиляции; для разделения между задачами используйте `ref`.

```yaoxiang
data = load_data()

// Ошибка компиляции: токен заимствования не может пересекать границу задачи
result = spawn {
    process(&data)   // Ошибка! &T нельзя передавать между задачами
}
```

**Свойства типов токенов**:

| Токен | Основная семантика | Вторичные свойства |
|-------|-------------------|-------------------|
| `&T` | **Замораживает исходные данные** — пока существует ReadToken, ни один WriteToken(T) не может быть получен | Нулевой размер, копируемый (Dup) — при гарантии замораживания множественные представления только для чтения безопасны по построению |
| `&mut T` | **Эксклюзивный доступ на чтение и запись** — пока существует WriteToken, никакой другой токен (ни на чтение, ни на запись) не может сосуществовать | Нулевой размер, линейный (не Dup) — копирование бессмысленно при эксклюзивном доступе |

> **Причинно-следственный порядок**: дублирование ReadToken является следствием гарантии замораживания, а не наоборот. Данные заморожены (мутация невозможна) → множественные представления только для чтения безопасны → можно реализовать Dup. Если принять Dup за определение, а проверку конфликтов — за «заплатку», причинность перевёрнута.

### 3.3 Разделение через ref

`ref` — это единственный способ разделения между областями видимости. Компилятор автоматически выбирает `Rc` (для одной задачи) или `Arc` (между задачами); пользователю не нужно об этом заботиться.

```yaoxiang
data = load_data()
shared = ref data       // компилятор автоматически выбирает Rc или Arc

result = spawn {
    process_a(shared),  // общая ссылка
    process_b(shared)   // общая ссылка
}
```

**Стратегия выбора компилятора**:

| Условие | Выбор | Причина |
|---------|-------|---------|
| По умолчанию (безопасность не доказана) | `Arc` | Приоритет безопасности; предотвращение гонок данных |
| Компилятор может доказать, что данные используются только в одной задаче | `Rc` | Отсутствие накладных расходов на атомарные операции |

**ref vs токены заимствования**:

| | `&T` / `&mut T` | `ref` |
|---|-----------------|-------|
| Назначение | Взглянуть / изменить на месте | Разделяемое владение |
| Стоимость | Нулевые накладные расходы (тип нулевого размера) | Rc или Arc (выбирает компилятор) |
| Между задачами | Нельзя | Можно (компилятор автоматически выбирает Arc) |

### 3.4 Захват замыканиями

Захват замыканиями = Move; одно замыкание может быть использовано только в одной задаче.

```yaoxiang
data = load_data()
fn = (x: Int) -> Int = data.value + x   // замыкание move-захватывает data

// Ошибка компиляции: замыкание можно использовать только в одной задаче
result = spawn {
    fn(1),      // использование замыкания
    fn(2)       // Ошибка! замыкание уже перемещено
}
```

**Правильный подход**: создавать отдельное замыкание для каждой задачи или использовать `ref`.

```yaoxiang
data = load_data()
shared = ref data

result = spawn {
    ((x: Int) -> Int = shared.value + x)(1),
    ((x: Int) -> Int = shared.value + x)(2)
}
```

## Глава 4: Обработка ошибок

### 4.1 Оператор `?`

Оператор `?` используется для явного распространения ошибок; его семантика совпадает с Rust.

```yaoxiang
read_file: (path: FilePath) -> Result(String, IoError) = {
    content = open(path)?      // при ошибке — немедленное распространение
    return content.read_all()
}
```

### 4.2 Распространение ошибок внутри блока spawn

**Правила**:
1. Ожидать завершения всех задач (даже если некоторые уже завершились с ошибкой)
2. Распространить первую встретившуюся ошибку
3. Использовать `?` для явной пометки точек распространения ошибки

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // может завершиться ошибкой
    fetch("url2")?      // может завершиться ошибкой
}
// При сбое любой задачи весь блок spawn распространяет первую ошибку
```

### 4.3 Типы ошибок

**Автогенерация**: компилятор автоматически генерирует объединённый тип ошибок.

```yaoxiang
// Компилятор выводит тип ошибки как HttpError | IoError
(a, b) = spawn {
    fetch("url"),           // может выбросить HttpError
    read_file("data.txt")  // может выбросить IoError
}
```

**Ручное переопределение**: пользователь может вручную определить унифицированный тип ошибки.

```yaoxiang
AppError: Type = {
    Http: (http_error: HttpError) -> AppError,
    Io: (io_error: IoError) -> AppError,
    Parse: (parse_error: ParseError) -> AppError
}

process: (url: String, path: FilePath) -> Result(Data, AppError) = {
    (a, b) = spawn {
        fetch(url).map_err(AppError.Http)?,
        read_file(path).map_err(AppError.Io)?
    }
    return parse(a + b).map_err(AppError.Parse)?
}
```

## Глава 5: Типы ресурсов и побочные эффекты

### 5.1 Встроенные типы ресурсов

| Тип ресурса | Описание | Поведение компилятора |
|-------------|----------|----------------------|
| `FilePath` | Путь файловой системы | Операции над одним путём автоматически сериализуются |
| `HttpUrl` | HTTP-эндпоинт | Операции над одним URL автоматически сериализуются |
| `DBUrl` | Соединение с базой данных | Операции над одним соединением автоматически сериализуются |
| `Console` | Стандартный вывод | Все операции Console автоматически сериализуются |

```yaoxiang
// Операции над одним файлом автоматически сериализуются
(a, b) = spawn {
    read_file("data.txt"),      // выполняется первой
    write_file("data.txt", x)   // ожидает завершения чтения
}
```

### 5.2 Пользовательские типы ресурсов

Пользовательские типы ресурсов требуют явной разметки.

```yaoxiang
Database: Type = {
    connection_string: String,
    query: (db: Database, sql: String) -> Result(Rows, DbError)
}
```

### 5.3 Отслеживание побочных эффектов

Компилятор отслеживает использование типов ресурсов, обеспечивая безопасность параллелизма.

```yaoxiang
// Предупреждение компилятора: операции Console могут чередоваться
spawn {
    print("Hello"),     // может чередоваться со следующей строкой
    print("World")
}

// Правильно: явная сериализация
spawn {
    print("Hello\nWorld")
}
```

## Глава 6: Поведение компилятора

### 6.1 DAG-анализ

Компилятор на этапе компиляции анализирует граф зависимостей (DAG) внутри блоков spawn, определяя:
1. Какие выражения могут выполняться параллельно
2. Какие должны выполняться последовательно
3. Как распределять задачи

```yaoxiang
(a, b, c) = spawn {
    x = fetch("url1"),      // задача 1
    y = fetch("url2"),      // задача 2 (параллельна задаче 1)
    z = process(x, y)       // задача 3 (зависит от x и y; должна ожидать)
}
```

### 6.2 Выбор Rc/Arc (консервативная стратегия)

Компилятор применяет **консервативную стратегию**, по умолчанию используя `Arc` для гарантии потокобезопасности:

- **`Arc` по умолчанию**: если компилятор не может определить, что `ref` используется только в одной задаче, консервативно выбирается `Arc`
- **Понижение до `Rc`**: только если компилятор может **доказать** посредством DAG-анализа, что данные гарантированно не будут разделены между задачами
- **Лучше медленнее, чем неправильно**: дополнительные накладные расходы от выбора `Arc` значительно меньше рисков гонок данных

### 6.3 Предупреждение об отсутствии параллелизма

Если задачи внутри блока spawn не имеют реальной возможности для параллельного выполнения, компилятор выдаёт предупреждение.

```yaoxiang
// Предупреждение компилятора: нет возможности параллельного выполнения
result = spawn {
    a = fetch("url")    // единственная задача
}
// Рекомендация: использовать обычный код
result = fetch("url")
```

### 6.4 Обнаружение конфликтов ресурсов

Компилятор обнаруживает потенциальные конфликты типов ресурсов.

```yaoxiang
// Ошибка компиляции: параллельная запись в один файл
spawn {
    write_file("data.txt", "a"),
    write_file("data.txt", "b")  // Ошибка!
}
```

## Глава 7: Уровни Runtime

Фаза компиляции полностью идентична; различие заключается только в способе выполнения во время исполнения (RFC-008).

| Уровень | Поддержка spawn | DAG-анализ | Сценарии применения |
|---------|----------------|------------|---------------------|
| Embedded Runtime | ❌ | Нет | WASM, игровые скрипты, движки правил |
| Standard Runtime | ✅ | Внутри блоков spawn | Веб-сервисы, конвейеры данных |
| Full Runtime | ✅ | Внутри блоков spawn + work stealing | Научные вычисления, массовый параллелизм |

**Embedded Runtime**: исполнитель «на лету», без поддержки spawn, высокая производительность и низкие накладные расходы.

**Standard Runtime**: поддерживает блоки `spawn {}`; DAG-анализ и автоматический параллелизм выполняются внутри блоков spawn. `num_workers=1` — однопоточный режим.

**Full Runtime**: Standard + балансировка нагрузки WorkStealer.

### A.1 Инструкция spawn

```
SpawnBlock  ::= '(' Pattern (',' Pattern)* ')' '=' 'spawn' '{' SpawnBody '}'
SpawnFor    ::= Identifier '=' 'spawn' 'for' Identifier 'in' Expr '{' Assignment '}'
SpawnStmt   ::= SpawnBlock | SpawnFor
SpawnBody   ::= Assignment (',' Assignment)*
```

```
Expr '?'              // распространение ошибки (тип Result)
```

### A.3 Выражение ref

```
RefExpr     ::= 'ref' Expr
```

### A.4 Разметка типов ресурсов

```
ResourceDecl ::= Identifier ':' 'Type' '=' RecordType
```
