# Спецификация модели конкурентности

> **Статус**: Данный документ описывает проект новой модели конкурентности языка YaoXiang. Она заменяет старую схему конкурентности, основанную на аннотациях `@block`/`@eager`/`@auto`, трейтах `Send`/`Sync` и `Mutex`/`RwLock`. Часть содержимого ещё не реализована, ориентируйтесь на фактическое поведение компилятора.

Настоящий файл определяет спецификацию модели конкурентности языка программирования YaoXiang, включая семантику блоков `{}`, примитив конкурентности `spawn`, обработку ошибок и типы ресурсов.

## Глава первая: Обзор

### 1.1 Сущность блока {}

В YaoXiang `{}` — это **вычислительная единица, управляемая зависимостями**.

| Атрибут | Описание |
|---------|----------|
| Управление зависимостями | При выполнении блок проверяет готовность всех внутренних переменных; если все готовы — выполняется немедленно, иначе блокируется в ожидании |
| Момент выполнения | Определяется зависимостями, не связан с «немедленным» или «отложенным» выполнением |
| Возвращаемое значение | Используйте `return` для явного возврата значения; без `return` по умолчанию возвращается `Void` |
| Единый синтаксис | Семантика одинакова независимо от того, где блок появляется: в теле функции, при инициализации переменной или после `spawn` |
| Изоляция области видимости | Переменные строго ограничены пределами `{}`, не проникают во внешнюю область видимости |

```yaoxiang
// Пример управления зависимостями
x = compute_x()        // x готова
y = compute_y()        // y готова
result = {
    // Зависит от x и y; выполняется, как только обе готовы
    return x + y
}
```

### 1.2 Правила возврата

Правила возврата в YaoXiang единообразны и однозначны:

| Запись | Возвращаемое значение | Описание |
|--------|----------------------|----------|
| `= expr` (без фигурных скобок) | Возвращает `expr` напрямую | Выражение есть значение |
| `= { ... }` (с фигурными скобками) | Требуется `return`, иначе возвращается `Void` | Блок требует явного возврата |

```yaoxiang
// Без фигурных скобок: прямой возврат
add: (a: Int, b: Int) -> Int = a + b

// С фигурными скобками: требуется return
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// С фигурными скобками, но без return: возвращается Void
log: (message: String) -> Void = {
    print(message)  // Без return возвращается Void
}
```

### 1.3 Семантика блока spawn

`spawn { ... }` — единственный примитив параллелизма в YaoXiang.

**Основные правила**:
- **Прямые дочерние присваивания** блока spawn создают параллельные задачи
- Присваивания внутри вложенных `{}` не считаются отдельными задачами
- Весь блок spawn блокируется синхронно, ожидая завершения всех задач, и возвращает результат
- Никаких обратных вызовов, `await` или аннотаций

```yaoxiang
// Две задачи выполняются параллельно
(a, b) = spawn {
    fetch("url1"),      // Задача 1
    fetch("url2")       // Задача 2
}
// Ожидание завершения обеих, затем продолжение
```

### 1.4 Ментальная модель пользователя

> Ваш обычный код выполняется последовательно.
> Когда вы хотите, чтобы несколько действий выполнялись одновременно, поместите их в блок `spawn { ... }`.
> Каждое прямое присваивание внутри блока запускается немедленно (параллельно), а нужные вам результаты будут автоматически ожидаться.
> Весь блок ждёт завершения всех действий, а затем возвращает вам итоговый результат.
> Никаких обратных вызовов, никаких `await`, никаких странных аннотаций.

## Глава вторая: Синтаксис и семантика

### 2.1 Обычный код

Обычный код (вне блока spawn) выполняется **последовательно**.

```yaoxiang
a = compute_a()     // Выполняется первой
b = compute_b(a)    // Зависит от a, выполняется после a
c = compute_c(b)    // Зависит от b, выполняется после b
```

### 2.2 Блок spawn

```
SpawnBlock  ::= '(' Pattern (',' Pattern)* ')' '=' 'spawn' '{' SpawnBody '}'
SpawnBody   ::= Assignment (',' Assignment)*
```

**Семантика**:
1. Прямые дочерние присваивания внутри блока spawn выполняются как отдельные задачи параллельно
2. Результат каждой задачи привязывается к соответствующей переменной образца
3. Весь блок блокируется до завершения всех задач
4. Возвращает кортеж всех результатов

```yaoxiang
// Одна задача
result = spawn {
    fetch("url")
}

// Несколько задач
(a, b, c) = spawn {
    fetch("url1"),
    fetch("url2"),
    fetch("url3")
}
```

### 2.3 spawn в теле функции

Тело функции само по себе является блоком `{}`, в котором можно использовать `spawn`.

```yaoxiang
fetch_and_parse: (urls: List(String)) -> List(Data) = {
    results = spawn for url in urls {
        parsed = parse(fetch(url))
    }
    return results
}
```

### 2.4 spawn в цикле

```
SpawnFor    ::= Identifier '=' 'spawn' 'for' Identifier 'in' Expr '{' Assignment '}'
```

**Семантика**: Цикл с параллелизмом по данным, каждая итерация — отдельная задача.

```yaoxiang
// Параллельная обработка каждого элемента списка
results = spawn for item in items {
    result = process(item)
}
```

> **Примечание**: Тело цикла `spawn for` представляет собой отдельные задачи и не поддерживает совместное изменяемое состояние между итерациями. Для агрегации результатов следует использовать `spawn for` для сбора результатов, а затем обрабатывать их снаружи (см. пример ниже).

```yaoxiang
// Правильно: параллельная обработка, затем агрегация снаружи
transformed = spawn for item in items {
    result = transform(item)
}
total = sum(transformed)   // Последовательная агрегация
```

### 2.5 Вложенный spawn

Блоки spawn могут быть вложенными, при этом внутренний spawn создаёт новую область конкурентности.

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

**Примечание**: Только прямые дочерние присваивания внутреннего spawn являются задачами; внешний spawn не «проникает» внутрь.

## Глава третья: Взаимодействие с моделью владения

### 3.1 Семантика перемещения (Move)

Move — семантика по умолчанию в YaoXiang (нулевое копирование).

```yaoxiang
data = load_data()
result = spawn {
    process(data)   // Владение data перемещается в блок spawn
}
// data здесь недоступна (перемещена)
```

**Правила**:
- После того как переменная входит в блок spawn, её нельзя использовать снаружи
- Если необходимо совместно использовать данные в нескольких задачах, используйте `ref`

### 3.2 Токены заимствования

`&T` и `&mut T` — это доказательства прав доступа нулевого размера во время компиляции, **они не могут пересекать границу задач**.

```yaoxiang
data = load_data()
ref_data = &data

// Ошибка компиляции: токен заимствования не может пересекать границу задач
result = spawn {
    process(ref_data)   // Ошибка!
}
```

### 3.3 Совместное использование через ref

`ref` — единственный способ совместного использования через границы областей видимости.

```yaoxiang
data = load_data()
shared = ref data       // Компилятор автоматически выбирает Rc или Arc

result = spawn {
    process_a(shared),  // Общая ссылка
    process_b(shared)   // Общая ссылка
}
```

**Выбор компилятора (консервативная стратегия)**:
| Условие | Выбор |
|---------|-------|
| По умолчанию | `Arc` (приоритет безопасности) |
| Компилятор может доказать, что данные используются только в одной задаче | `Rc` (без накладных расходов на атомарные операции) |

### 3.4 Захват замыкания

Захват замыкания = Move; одно замыкание может быть использовано только в одной задаче.

```yaoxiang
data = load_data()
fn = (x: Int) -> Int = data.value + x   // Замыкание перемещает data

// Ошибка компиляции: замыкание может быть использовано только в одной задаче
result = spawn {
    fn(1),      // Использование замыкания
    fn(2)       // Ошибка! Замыкание уже перемещено
}
```

**Правильный подход**: Создавайте отдельное замыкание для каждой задачи или используйте `ref`.

```yaoxiang
data = load_data()
shared = ref data

result = spawn {
    ((x: Int) -> Int = shared.value + x)(1),
    ((x: Int) -> Int = shared.value + x)(2)
}
```

## Глава четвёртая: Обработка ошибок

### 4.1 Оператор `?`

Оператор `?` используется для явного распространения ошибок; его семантика совпадает с Rust.

```yaoxiang
read_file: (path: FilePath) -> Result(String, IoError) = {
    content = open(path)?      // В случае ошибки — немедленное распространение
    return content.read_all()
}
```

### 4.2 Распространение ошибок внутри блока spawn

**Правила**:
1. Дождаться завершения всех задач (даже если некоторые уже завершились ошибкой)
2. Распространить первую встретившуюся ошибку
3. Использовать `?` для явной отметки точки распространения ошибки

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // Может завершиться ошибкой
    fetch("url2")?      // Может завершиться ошибкой
}
// Если какая-либо задача завершилась ошибкой, весь блок spawn распространяет первую ошибку
```

### 4.3 Типы ошибок

**Автоматическая генерация**: Компилятор автоматически генерирует объединённый тип ошибок (аналогично объединению типов в TypeScript).

```yaoxiang
// Компилятор выводит, что тип ошибки — HttpError | IoError
(a, b) = spawn {
    fetch("url"),           // Может выбросить HttpError
    read_file("data.txt")  // Может выбросить IoError
}
```

**Ручное переопределение**: Пользователь может вручную определить единый тип ошибок.

```yaoxiang
AppError: Type = {
    Http: (http_error: HttpError) -> AppError,
    Io: (io_error: IoError) -> AppError,
    Parse: (parse_error: ParseError) -> AppError
}

process: (url: String, path: FilePath) -> Result(Data, AppError) = {
    (a, b) = spawn {
        fetch(url).map_err(AppError.Http)?,
        read_file(path).map_err(AppError.Io)?
    }
    return parse(a + b).map_err(AppError.Parse)?
}
```

## Глава пятая: Типы ресурсов и побочные эффекты

### 5.1 Встроенные типы ресурсов

| Тип ресурса | Описание | Поведение компилятора |
|-------------|----------|------------------------|
| `FilePath` | Путь в файловой системе | Операции с одним путём автоматически упорядочиваются |
| `HttpUrl` | HTTP-эндпоинт | Операции с одним URL автоматически упорядочиваются |
| `DBUrl` | Соединение с базой данных | Операции с одним соединением автоматически упорядочиваются |
| `Console` | Стандартный вывод | Все операции с Console автоматически упорядочиваются |

```yaoxiang
// Операции с одним файлом автоматически упорядочиваются
(a, b) = spawn {
    read_file("data.txt"),      // Выполняется первой
    write_file("data.txt", x)   // Ожидает завершения чтения
}
```

### 5.2 Пользовательские типы ресурсов

Пользовательские типы ресурсов требуют явной маркировки.

```yaoxiang
Database: Type = {
    connection_string: String,
    query: (db: Database, sql: String) -> Result(Rows, DbError)
}
```

### 5.3 Отслеживание побочных эффектов

Компилятор отслеживает использование типов ресурсов, чтобы обеспечить безопасность конкурентности.

```yaoxiang
// Предупреждение компилятора: операции с Console могут чередоваться
spawn {
    print("Hello"),     // Может чередоваться со следующей строкой
    print("World")
}

// Правильно: явная последовательность
spawn {
    print("Hello\nWorld")
}
```

## Глава шестая: Поведение компилятора

### 6.1 DAG-анализ

Компилятор во время компиляции анализирует граф зависимостей (DAG) внутри блока spawn, чтобы определить:
1. Какие выражения могут выполняться параллельно
2. Какие должны быть последовательными
3. Как распределить задачи

```yaoxiang
(a, b, c) = spawn {
    x = fetch("url1"),      // Задача 1
    y = fetch("url2"),      // Задача 2 (параллельно с задачей 1)
    z = process(x, y)       // Задача 3 (зависит от x и y, должна ждать)
}
```

### 6.2 Выбор Rc/Arc (консервативная стратегия)

Компилятор применяет **консервативную стратегию**, по умолчанию используя `Arc` для обеспечения потокобезопасности:

| Условие | Выбор | Причина |
|---------|-------|---------|
| По умолчанию (не удаётся доказать безопасность) | `Arc` | Приоритет безопасности, избежание гонок данных |
| Компилятор может **доказать**, что данные используются только в одной задаче | `Rc` | Без накладных расходов на атомарные операции |

**Пояснение стратегии**:
- **`Arc` по умолчанию**: Когда компилятор не может определить, используется ли `ref` только в пределах одной задачи, он консервативно выбирает `Arc`
- **Понижение до `Rc`**: Только если компилятор может с помощью DAG-анализа **доказать**, что данные гарантированно не будут совместно использоваться между задачами, происходит понижение до `Rc`
- **Лучше медленнее, чем неправильно**: Дополнительные накладные расходы на выбор `Arc` значительно меньше рисков, связанных с гонками данных

```yaoxiang
data = load_data()

// По умолчанию: компилятор выбирает Arc (консервативная стратегия)
result = spawn {
    shared = ref data
    process(shared)
}

// Только если компилятор может доказать использование в одной задаче: понижение до Rc
// (Требуется, чтобы DAG-анализ компилятора явно исключал возможность совместного использования между задачами)
```

### 6.3 Предупреждение об отсутствии параллелизма

Если задачи внутри блока spawn фактически не имеют возможности параллельного выполнения, компилятор выдаёт предупреждение.

```yaoxiang
// Предупреждение компилятора: нет возможности параллелизма
result = spawn {
    a = fetch("url")    // Единственная задача
}
// Рекомендация: используйте обычный код напрямую
result = fetch("url")
```

### 6.4 Обнаружение конфликтов ресурсов

Компилятор обнаруживает потенциальные конфликты типов ресурсов.

```yaoxiang
// Ошибка компиляции: параллельная запись в один файл
spawn {
    write_file("data.txt", "a"),
    write_file("data.txt", "b")  // Ошибка!
}
```

## Глава седьмая: Сравнение со старой моделью

### 7.1 Устаревшие возможности

| Старая возможность | Статус | Замена |
|--------------------|--------|--------|
| Аннотации `@block`, `@eager`, `@auto` | Устарели | Нет, обрабатывается автоматически на основе зависимостей |
| Автоматический DAG-анализ всей программы | Устарел | Только анализ внутри блока spawn |
| Трейты `Send`, `Sync` | Устарели | Автоматическая обработка через владение + `ref` |
| future/неблокирующие дескрипторы | Устарели | Синхронный возврат блока spawn |
| `Mutex[T]`, `Atomic[T]`, `RwLock[T]` | Устарели | Автоматический выбор Rc/Arc через `ref` |

### 7.2 Изменение концепции проектирования

**Старая модель**:
- Явные аннотации для управления поведением конкурентности
- Сложные ограничения трейтов
- Модель асинхронного программирования

**Новая модель**:
- Управление зависимостями, неявный параллелизм
- Владение + `ref` упрощают совместное использование
- Модель синхронного программирования, блок spawn блокируется до возврата

### 7.3 Руководство по миграции

> **Примечание об устаревании**: Примеры старого кода ниже приведены только для иллюстрации направления миграции. `@block`, `@eager`, `@auto`, `let`, `await`, `Future` не являются ключевыми словами YaoXiang и были удалены в новой модели.

```yaoxiang
// Старый код (псевдокод, демонстрирующий стиль старой модели)
@block async fetch_data(): Future<Data> = {
    let data = await fetch("url")
    return data
}

// Новый код
fetch_data: () -> Data = {
    data = fetch("url")     // Синхронный вызов
    return data
}

// Параллельная версия
fetch_multiple: (urls: List(String)) -> List(Data) = {
    results = spawn for url in urls {
        result = fetch(url)
    }
    return results
}
```

## Приложение: Краткий справочник по синтаксису

### A.1 Инструкция spawn

```
SpawnBlock  ::= '(' Pattern (',' Pattern)* ')' '=' 'spawn' '{' SpawnBody '}'
SpawnFor    ::= Identifier '=' 'spawn' 'for' Identifier 'in' Expr '{' Assignment '}'
SpawnStmt   ::= SpawnBlock | SpawnFor
SpawnBody   ::= Assignment (',' Assignment)*
```

### A.2 Обработка ошибок

```
Expr '?'              // Распространение ошибки (тип Result)
```

### A.3 Выражение ref

```
RefExpr     ::= 'ref' Expr
```

### A.4 Маркировка типа ресурса

```
ResourceDecl ::= Identifier ':' 'Type' '=' RecordType
```
