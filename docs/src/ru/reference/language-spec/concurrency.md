# Спецификация модели параллелизма

> **Статус**: Официальная спецификация. Основана на RFC-024 (Модель параллелизма), RFC-009 (Модель владения), RFC-008 (Архитектура времени выполнения).

Настоящий документ определяет спецификацию модели параллелизма языка программирования YaoXiang, включая семантику блоков `{}`, примитивы параллелизма `spawn`, взаимодействие с владением, обработку ошибок и типы ресурсов.

**Ключевой замысел — одна примитива, одно правило**:

```
spawn { ... }        ← единственная примитива параллелизма
присваивание в прямом потомке создаёт задачу    ← единственное правило
синхронное блокирующее ожидание результата      ← единственное поведение
```

## Глава первая: Обзор

### 1.1 Сущность блока {}

В YaoXiang `{}` — это **управляемая зависимостями вычислительная единица**.

| Свойство | Описание |
|------|------|
| Управляемость зависимостями | При выполнении блок проверяет, готовы ли все внутренние переменные; если все готовы — выполняется немедленно, иначе блокируется в ожидании |
| Момент выполнения | Определяется зависимостями, не зависит от «немедленного» или «отложенного» выполнения |
| Возвращаемое значение | Явное возвращение значения через `return`; при отсутствии `return` по умолчанию возвращается `Void` |
| Единообразие синтаксиса | Семантика одинакова независимо от места появления: в теле функции, при инициализации переменной или после `spawn` |
| Изоляция области видимости | Переменные строго ограничены внутренней областью `{}`, не проникают во внешнюю область видимости |

```yaoxiang
// Пример управляемой зависимостями семантики
x = compute_x()        // x готов
y = compute_y()        // y готов
result = {
    // Зависит от x и y; выполняется сразу после готовности обоих
    return x + y
}
```

### 1.2 Правила возврата

| Запись | Возвращаемое значение | Описание |
|------|--------|------|
| `= expr` (без фигурных скобок) | Прямой возврат `expr` | Выражение есть значение |
| `= { ... }` (с фигурными скобками) | Требуется `return`, иначе возвращается `Void` | Блок требует явного возврата |

```yaoxiang
// Без фигурных скобок: прямой возврат
add: (a: Int, b: Int) -> Int = a + b

// С фигурными скобками: обязателен return
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// С фигурными скобками, но без return: возвращается Void
log: (message: String) -> Void = {
    print(message)  // Без return возвращается Void
}
```

### 1.3 Семантика блока spawn

`spawn { ... }` — это **единственная примитива параллелизма** в YaoXiang.

**Ключевые правила**:
- **Присваивания в прямых потомках** блока spawn создают параллельные задачи
- Присваивания внутри вложенных `{}` не считаются самостоятельными задачами
- Весь блок spawn синхронно блокируется, ожидая завершения всех задач, после чего возвращает результат
- Нет ни колбэков, ни `await`, ни аннотаций

```yaoxiang
// Две задачи выполняются параллельно
(a, b) = spawn {
    fetch("url1"),      // Задача 1
    fetch("url2")       // Задача 2
}
// Ожидание завершения обоих перед продолжением
```

### 1.4 Пользовательская ментальная модель

> Обычный код, который вы пишете, выполняется последовательно.
> Когда вы хотите, чтобы несколько дел выполнялись одновременно, поместите их в блок `spawn { ... }`.
> Каждое присваивание в прямом потомке блока запускается немедленно (параллельно), и нужные вам результаты будут автоматически ожидаться.
> Весь блок дождётся завершения всех дел, а затем вернёт вам итоговый результат.
> Никаких колбэков, никакого `await`, никаких странных аннотаций.

## Глава вторая: Синтаксис и семантика

### 2.1 Обычный код

Обычный код (за пределами блока spawn) выполняется **последовательно**.

```yaoxiang
a = compute_a()     // Выполняется первым
b = compute_b(a)    // Зависит от a, выполняется после a
c = compute_c(b)    // Зависит от b, выполняется после b
```

### 2.2 Блок spawn

```
SpawnBlock  ::= '(' Pattern (',' Pattern)* ')' '=' 'spawn' '{' SpawnBody '}'
SpawnBody   ::= Assignment (',' Assignment)*
```

**Семантика**:
1. Присваивания в прямых потомках блока spawn выполняются как самостоятельные задачи параллельно
2. Результат каждой задачи связывается с соответствующей переменной-шаблоном
3. Весь блок блокируется до завершения всех задач
4. Возвращается кортеж всех результатов

```yaoxiang
// Одна задача
result = spawn {
    fetch("url")
}

// Несколько задач
(a, b, c) = spawn {
    fetch("url1"),
    fetch("url2"),
    fetch("url3")
}
```

### 2.3 spawn в теле функции

Тело функции само по себе является блоком `{}`, в котором можно использовать `spawn`.

```yaoxiang
fetch_and_parse: (urls: List(String)) -> List(Data) = {
    results = spawn for url in urls {
        parsed = parse(fetch(url))
    }
    return results
}
```

### 2.4 spawn в цикле

```
SpawnFor    ::= Identifier '=' 'spawn' 'for' Identifier 'in' Expr '{' Assignment '}'
```

**Семантика**: параллельный по данным цикл, каждая итерация — самостоятельная задача.

```yaoxiang
// Параллельная обработка каждого элемента списка
results = spawn for item in items {
    result = process(item)
}
```

> **Примечание**: Тело цикла `spawn for` представляет собой самостоятельные задачи и не поддерживает совместное изменяемое состояние между итерациями. Для агрегации результатов следует собирать их через `spawn for` и обрабатывать снаружи.

```yaoxiang
// Правильно: параллельная обработка с внешней агрегацией
transformed = spawn for item in items {
    result = transform(item)
}
total = sum(transformed)   // Последовательная агрегация
```

### 2.5 Вложенный spawn

Блоки spawn могут быть вложенными; внутренний spawn создаёт новую область параллелизма.

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

Только присваивания в прямых потомках внутреннего spawn являются задачами; внешний spawn не проникает внутрь.

## Глава третья: Взаимодействие с моделью владения

### 3.1 Семантика перемещения

Перемещение (Move) — это семантика по умолчанию в YaoXiang (без копирования). После того как переменная входит в блок spawn, её нельзя использовать снаружи.

```yaoxiang
data = load_data()
result = spawn {
    process(data)   // Право владения data перемещается в блок spawn
}
// Здесь data недоступна (была перемещена)
```

### 3.2 Маркеры заимствования

`&T` и `&mut T` — это доказательства прав доступа нулевого размера в compile-time, **которые не могут пересекать границу задач**. Это не специальное правило — маркер является доказательством прав доступа в compile-time, для совместного использования между задачами используйте `ref`.

```yaoxiang
data = load_data()

// Ошибка компиляции: маркер заимствования не может пересекать задачи
result = spawn {
    process(&data)   // Ошибка! &T не может передаваться между задачами
}
```

**Свойства типов маркеров**:
- `&T`: нулевого размера, копируемый (Dup), предоставляет право чтения
- `&mut T`: нулевого размера, линейный (не Dup), предоставляет исключительное право чтения-записи

### 3.3 Совместное использование через ref

`ref` — это единственный способ совместного использования между областями видимости. Компилятор автоматически выбирает `Rc` (одна задача) или `Arc` (между задачами); пользователю не нужно об этом заботиться.

```yaoxiang
data = load_data()
shared = ref data       // Компилятор автоматически выбирает Rc или Arc

result = spawn {
    process_a(shared),  // Разделяемая ссылка
    process_b(shared)   // Разделяемая ссылка
}
```

**Стратегия выбора компилятора**:

| Условие | Выбор | Причина |
|------|------|------|
| По умолчанию (безопасность не доказуема) | `Arc` | Приоритет безопасности, избежание гонок данных |
| Компилятор может доказать, что данные используются только в одной задаче | `Rc` | Отсутствие накладных расходов на атомарные операции |

**ref vs маркеры заимствования**:

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Назначение | Взглянуть / изменить на месте | Совместное владение |
| Стоимость | Нулевые накладные расходы (тип нулевого размера) | Rc или Arc (выбирает компилятор) |
| Между задачами | Нельзя | Можно (компилятор автоматически выбирает Arc) |

### 3.4 Захват замыканиями

Захват замыканием = Move, одно замыкание может быть использовано только в одной задаче.

```yaoxiang
data = load_data()
fn = (x: Int) -> Int = data.value + x   // Замыкание захватывает data перемещением

// Ошибка компиляции: замыкание может быть использовано только в одной задаче
result = spawn {
    fn(1),      // Использование замыкания
    fn(2)       // Ошибка! Замыкание уже перемещено
}
```

**Правильный подход**: создавать отдельное замыкание для каждой задачи или использовать `ref`.

```yaoxiang
data = load_data()
shared = ref data

result = spawn {
    ((x: Int) -> Int = shared.value + x)(1),
    ((x: Int) -> Int = shared.value + x)(2)
}
```

## Глава четвёртая: Обработка ошибок

### 4.1 Оператор `?`

Оператор `?` используется для явного распространения ошибок, семантика совпадает с Rust.

```yaoxiang
read_file: (path: FilePath) -> Result(String, IoError) = {
    content = open(path)?      // При ошибке — немедленное распространение
    return content.read_all()
}
```

### 4.2 Распространение ошибок внутри блока spawn

**Правила**:
1. Дождаться завершения всех задач (даже если некоторые из них завершились ошибкой)
2. Распространить первую встреченную ошибку
3. Использовать `?` для явной отметки точки распространения ошибки

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // Может завершиться ошибкой
    fetch("url2")?      // Может завершиться ошибкой
}
// Если какая-либо задача завершилась ошибкой, весь блок spawn распространяет первую ошибку
```

### 4.3 Типы ошибок

**Автогенерация**: компилятор автоматически генерирует объединённый тип ошибок.

```yaoxiang
// Компилятор выводит тип ошибки как HttpError | IoError
(a, b) = spawn {
    fetch("url"),           // Может выбросить HttpError
    read_file("data.txt")  // Может выбросить IoError
}
```

**Ручное переопределение**: пользователь может вручную определить унифицированный тип ошибок.

```yaoxiang
AppError: Type = {
    Http: (http_error: HttpError) -> AppError,
    Io: (io_error: IoError) -> AppError,
    Parse: (parse_error: ParseError) -> AppError
}

process: (url: String, path: FilePath) -> Result(Data, AppError) = {
    (a, b) = spawn {
        fetch(url).map_err(AppError.Http)?,
        read_file(path).map_err(AppError.Io)?
    }
    return parse(a + b).map_err(AppError.Parse)?
}
```

## Глава пятая: Типы ресурсов и побочные эффекты

### 5.1 Встроенные типы ресурсов

| Тип ресурса | Описание | Поведение компилятора |
|----------|------|-----------|
| `FilePath` | Путь в файловой системе | Операции над одним путём автоматически сериализуются |
| `HttpUrl` | HTTP-эндпоинт | Операции над одним URL автоматически сериализуются |
| `DBUrl` | Соединение с базой данных | Операции над одним соединением автоматически сериализуются |
| `Console` | Стандартный вывод | Все операции Console автоматически сериализуются |

```yaoxiang
// Операции над одним файлом автоматически сериализуются
(a, b) = spawn {
    read_file("data.txt"),      // Выполняется первой
    write_file("data.txt", x)   // Ожидает завершения чтения
}
```

### 5.2 Пользовательские типы ресурсов

Пользовательские типы ресурсов требуют явной разметки.

```yaoxiang
Database: Type = {
    connection_string: String,
    query: (db: Database, sql: String) -> Result(Rows, DbError)
}
```

### 5.3 Отслеживание побочных эффектов

Компилятор отслеживает использование типов ресурсов, обеспечивая безопасность при параллелизме.

```yaoxiang
// Предупреждение компилятора: операции Console могут чередоваться
spawn {
    print("Hello"),     // Может чередоваться со следующей строкой
    print("World")
}

// Правильно: явная сериализация
spawn {
    print("Hello\nWorld")
}
```

## Глава шестая: Поведение компилятора

### 6.1 DAG-анализ

Компилятор в compile-time анализирует зависимости внутри блока spawn (DAG) и определяет:
1. Какие выражения могут выполняться параллельно
2. Какие должны выполняться последовательно
3. Как распределять задачи

```yaoxiang
(a, b, c) = spawn {
    x = fetch("url1"),      // Задача 1
    y = fetch("url2"),      // Задача 2 (параллельна задаче 1)
    z = process(x, y)       // Задача 3 (зависит от x и y, должна ожидать)
}
```

### 6.2 Выбор Rc/Arc (консервативная стратегия)

Компилятор применяет **консервативную стратегию**, по умолчанию используя `Arc` для обеспечения потокобезопасности:

- **`Arc` по умолчанию**: когда компилятор не может определить, используется ли `ref` только в пределах одной задачи, он консервативно выбирает `Arc`
- **Понижение до `Rc`**: только если компилятор может посредством DAG-анализа **доказать**, что данные гарантированно не будут разделяться между задачами
- **Лучше медленнее, чем неправильно**: дополнительные накладные расходы `Arc` значительно меньше рисков гонки данных

### 6.3 Предупреждение об отсутствии параллелизма

Если задачи внутри блока spawn не имеют реальной возможности параллельного выполнения, компилятор выдаёт предупреждение.

```yaoxiang
// Предупреждение компилятора: нет возможности параллелизма
result = spawn {
    a = fetch("url")    // Единственная задача
}
// Рекомендация: использовать обычный код
result = fetch("url")
```

### 6.4 Обнаружение конфликтов ресурсов

Компилятор обнаруживает потенциальные конфликты типов ресурсов.

```yaoxiang
// Ошибка компиляции: параллельная запись в один файл
spawn {
    write_file("data.txt", "a"),
    write_file("data.txt", "b")  // Ошибка!
}
```

## Глава седьмая: Слои времени выполнения

Этап компиляции полностью идентичен; отличие только в способе выполнения в runtime (RFC-008).

| Слой | Поддержка spawn | DAG-анализ | Сценарии использования |
|------|-----------|----------|----------|
| Embedded Runtime | ❌ | Нет | WASM, игровые скрипты, движки правил |
| Standard Runtime | ✅ | Внутри блока spawn | Веб-сервисы, конвейеры данных |
| Full Runtime | ✅ | Внутри блока spawn + кража работы | Научные вычисления, крупномасштабный параллелизм |

**Embedded Runtime**: исполнитель немедленного выполнения, без поддержки spawn, высокая производительность при низком потреблении.

**Standard Runtime**: поддерживает блоки `spawn {}`, выполняет DAG-анализ и автоматический параллелизм внутри блока spawn. `num_workers=1` — однопоточный режим.

**Full Runtime**: Standard + балансировка нагрузки WorkStealer.

## Приложение: Краткий справочник по синтаксису

### A.1 Инструкция spawn

```
SpawnBlock  ::= '(' Pattern (',' Pattern)* ')' '=' 'spawn' '{' SpawnBody '}'
SpawnFor    ::= Identifier '=' 'spawn' 'for' Identifier 'in' Expr '{' Assignment '}'
SpawnStmt   ::= SpawnBlock | SpawnFor
SpawnBody   ::= Assignment (',' Assignment)*
```

### A.2 Обработка ошибок

```
Expr '?'              // Распространение ошибки (тип Result)
```

### A.3 Выражение ref

```
RefExpr     ::= 'ref' Expr
```

### A.4 Разметка типа ресурса

```
ResourceDecl ::= Identifier ':' 'Type' '=' RecordType
```
