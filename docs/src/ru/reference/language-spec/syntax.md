# Синтаксическая спецификация

Этот документ определяет синтаксическую спецификацию языка программирования YaoXiang, включая лексическую структуру, синтаксические правила и приоритет операторов.

---

## Глава 1: Лексическая структура

### 1.1 Исходный файл

Исходные файлы YaoXiang должны использовать кодировку UTF-8. Исходные файлы обычно имеют расширение `.yx`.

### 1.2 Классификация лексических единиц

| Категория | Описание | Примеры |
|------|------|------|
| Идентификаторы | Начинаются с буквы или подчёркивания | `x`, `_private`, `my_var` |
| Ключевые слова | Предопределённые зарезервированные слова языка | `Type`, `pub`, `use` |
| Литералы | Фиксированные значения | `42`, `"hello"`, `true` |
| Операторы | Символы операций | `+`, `-`, `*`, `/` |
| Разделители | Синтаксические разделители | `(`, `)`, `{`, `}`, `,` |

### 1.3 Ключевые слова

YaoXiang определяет крайне малое количество ключевых слов:

```
pub    use    spawn
ref    mut    if     elif
else   match  while  for    return
break  continue as     in     unsafe
```

Эти ключевые слова имеют специальное значение в любом контексте и не могут использоваться в качестве идентификаторов.

### 1.4 Зарезервированные слова

«Зарезервированные слова» YaoXiang разделены на три слоя, которые распознаются парсером (parser) и проверщиком типов (type checker) на разных этапах:

#### 1.4.1 Литеральные зарезервированные слова

Идентификаторы литералов, имеющие собственные токены в парсере, которые не могут использоваться в качестве обычных идентификаторов:

| Идентификатор | Принадлежит типу | Описание |
|--------|---------|------|
| `Type` | — | Ключевое слово метатипа |
| `true` | Bool | Логическое значение истина |
| `false` | Bool | Логическое значение ложь |
| `void` | Void | Литерал Void (значение Unit). Строчное `void` является литералом значения; заглавное `Void` является именем типа (см. §1.4.3). |

#### 1.4.2 Выражения-конструкторы

Следующие конструкторы распознаются парсером в контекстах сопоставления с образцом и выражений:

| Конструктор | Принадлежит типу | Описание |
|--------|---------|------|
| `some(T)` | Option | Конструкция варианта значения Option |
| `ok(T)` | Result | Успешный вариант Result |
| `err(E)` | Result | Вариант ошибки Result |

#### 1.4.3 Встроенные имена типов

Следующие имена типов предварительно зарегистрированы проверщиком типов и могут использоваться в позициях типов без импорта. Парсер рассматривает их как обычные идентификаторы — **не являются зарезервированными словами и могут быть затенены (shadow) локальными привязками (не рекомендуется)**.

| Имя типа | Логическое соответствие | Описание |
|--------|---------|------|
| `Void` | ⊤ (истина/Unit) | Тип-произведение с нулём полей, имеющий ровно одного обитателя (литерал `void`, см. §1.4.1) |
| `Never` | ⊥ (ложь/пустой тип) | Тип с нулём вариантов и нулём обитателей. Никакое выражение не может породить значение `Never`. `Never <: T` выполняется для всех `T` (принцип взрыва). |
| `Int` | — | Целое число со знаком |
| `Float` | — | Число с плавающей точкой |
| `Bool` | — | Логическое значение: `true` / `false` |
| `Char` | — | Символ Unicode |
| `String` | — | Строка |

### 1.5 Идентификаторы

Идентификаторы начинаются с буквы или подчёркивания, последующие символы могут быть буквами, цифрами или подчёркиваниями. Идентификаторы чувствительны к регистру.

Специальные идентификаторы:
- `_` используется как заполнитель, означающий игнорирование значения
- Идентификаторы, начинающиеся с подчёркивания, обозначают приватные члены

### 1.6 Литералы

#### 1.6.1 Целые числа

```
Decimal     ::= [0-9][0-9_]*
Octal       ::= 0o[0-7][0-7_]*
Hex         ::= 0x[0-9a-fA-F][0-9a-fA-F_]*
Binary      ::= 0b[01][01_]*
```

#### 1.6.2 Числа с плавающей точкой

```
Float       ::= [0-9][0-9_]* '.' [0-9][0-9_]* ([eE][+-]?[0-9][0-9_]*)?
```

#### 1.6.3 Строки

```
String      ::= '"' ([^"\\] | EscapeSequence)* '"'
Escape      ::= '\\' ([nrt'"\\] | UnicodeEscape)
Unicode     ::= 'u' '{' HexDigit+ '}'
```

#### 1.6.4 Коллекции

```
List        ::= '[' Expr (',' Expr)* ']'
Dict        ::= '{' String ':' Expr (',' String ':' Expr)* '}'
Set         ::= '{' Expr (',' Expr)* '}'
```

#### 1.6.5 List comprehension

```
ListComp    ::= '[' Expr 'for' Identifier 'in' Expr (',' Expr)* ('if' Expr)? ']'
```

#### 1.6.6 Проверка принадлежности

```
Membership  ::= Expr 'in' Expr
```

### 1.7 Комментарии

```
// однострочный комментарий

/* многострочный комментарий
   может охватывать несколько строк */
```

### 1.8 Правила отступов

Код должен использовать отступ в 4 пробела, использование символов табуляции запрещено. Это обязательное синтаксическое правило.

## Глава 2: Синтаксические правила

### 2.1 Классификация выражений

```
Expr        ::= Literal
              | Identifier
              | FnCall
              | MemberAccess
              | IndexAccess
              | UnaryOp
              | BinaryOp
              | TypeCast
              | RangeExpr
              | ErrorPropagate
              | RefExpr
              | IfExpr
              | MatchExpr
              | Block
              | Lambda
```

### 2.2 Приоритет операторов

| Приоритет | Операторы | Ассоциативность |
|--------|--------|--------|
| 1 | `()` `[]` `.` `?` | слева направо |
| 2 | `as` | слева направо |
| 3 | `*` `/` `%` | слева направо |
| 4 | `+` `-` | слева направо |
| 5 | `..` | слева направо |
| 6 | `<<` `>>` | слева направо |
| 7 | `&` `\|` `^` | слева направо |
| 8 | `==` `!=` `<` `>` `<=` `>=` | слева направо |
| 9 | `not` | справа налево |
| 10 | `and` `or` | слева направо |
| 11 | `if...else` | справа налево |
| 12 | `=` `+=` `-=` `*=` `/=` | справа налево |

### 2.3 Вызовы функций

```
FnCall      ::= Expr '(' ArgList? ')'
ArgList     ::= Expr (',' Expr)* (',' NamedArg)* | NamedArg (',' NamedArg)*
NamedArg    ::= Identifier ':' Expr
```

### 2.4 Доступ к членам

```
MemberAccess::= Expr '.' Identifier
```

### 2.5 Доступ по индексу

```
IndexAccess ::= Expr '[' Expr ']'
```

### 2.6 Приведение типов

```
TypeCast    ::= Expr 'as' TypeExpr
```

### 2.7 Условное выражение

```
IfExpr      ::= 'if' Expr Block ('elif' Expr Block)* ('else' Block)?
```

### 2.8 Сопоставление с образцом

```
MatchExpr   ::= 'match' Expr '{' MatchArm+ '}'
MatchArm    ::= Pattern ('|' Pattern)* ('if' Expr)? '=>' Expr ','
Pattern     ::= Literal
              | Identifier
              | Wildcard
              | StructPattern
              | TuplePattern
              | EnumPattern
              | OrPattern
```

### 2.9 Блочное выражение

```
Block       ::= '{' Stmt* Expr? '}'
```

**Единая семантика**: семантика `return` для всех блоков `{}` одинакова:

| Тип блока | Семантика return | Возврат по умолчанию |
|--------|------------|----------|
| Обычный `{}` | Возвращает значение | Void |
| `unsafe {}` | Возвращает определение типа | Void |
| `spawn {}` | Возвращает результат | Void |

**Основные принципы**:
- `return` в `{}` всегда возвращает содержимое во внешнюю область видимости
- По умолчанию без `return` возвращается `Void`
- Форма выражения `= expr` возвращает значение напрямую

```yaoxiang
// Обычный блок {}: return возвращает значение
result = {
    x = compute()
    return x  // возвращает значение во внешнюю область видимости
}

// Блок unsafe {}: return возвращает определение типа
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void
    }
    return SqliteDb  // возвращает определение типа во внешнюю область видимости
}

// Блок spawn {}: return возвращает результат
(a, b) = spawn {
    result1 = fetch("url1"),
    result2 = fetch("url2")
    return (result1, result2)  // возвращает результат во внешнюю область видимости
}
```

### 2.10 Лямбда-выражение

```
Lambda      ::= '(' ParamList? ')' '=>' Expr
            |  '(' ParamList? ')' '=>' Block
```

### 2.11 Оператор распространения ошибок

```
ErrorPropagate ::= Expr '?'
```

Оператор `?` является постфиксным оператором с тем же приоритетом, что и `.`. Для типа `Result(T, E)`:
- При `Ok(v)` извлекает значение `v` и продолжает выполнение
- При `Err(e)` распространяет ошибку вверх (`return Err(e)`)

```yaoxiang
process: (data: Data) -> Result(Data, Error) = {
    validated = validate(data)?     // при успехе извлекает значение, при неудаче распространяет вверх
    transform(validated)
}
```

### 2.12 Выражение диапазона

```
RangeExpr   ::= Expr '..' Expr
```

`..` создаёт тип диапазона, используемый для циклов `for` и срезов.

```yaoxiang
for i in 0..10 { print(i) }
slice = array[0..5]
```

### 2.13 Выражение ref

```
RefExpr     ::= 'ref' Expr
```

`ref` создаёт разделяемое владение. Компилятор автоматически выбирает Rc (внутри одной задачи) или Arc (между задачами), пользователю не нужно беспокоиться о деталях реализации.

```yaoxiang
data = ref heavy_data
spawn { use(data) }   // между задачами: компилятор автоматически выберет Arc
```

### 2.14 Выражение unsafe

```
UnsafeExpr  ::= 'unsafe' Block
```

Блок `unsafe` используется для определения непрозрачных типов и работы с сырыми указателями. Используйте `return` для возврата определения типа во внешнюю область видимости.

**Семантика**:
- В `unsafe {}` можно определять типы и работать с сырыми указателями
- Возвращаемый тип доступен за пределами `unsafe {}`
- Доступ к полям типа требует привилегий unsafe

```yaoxiang
// Определение непрозрачного типа в блоке unsafe
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void  // сырой указатель
    }
    return SqliteDb
}

// SqliteDb доступен за пределами блока unsafe
db = sqlite3_open("test.db")
```

### 2.15 Область видимости

**Основные правила**:
- Каждый блок `{}` создаёт область видимости
- Внутренняя область видимости может обращаться к переменным внешней области
- Внешняя область видимости не может обращаться к переменным внутренней области
- Объявление переменных следует принципу «присваивание в приоритете»

```yaoxiang
// Блочная область видимости
{
    x = 10
    // x видна в этой области видимости
}
// x не видна за пределами этой области видимости

// Область видимости функции
add: (a: Int, b: Int) -> Int = {
    result = a + b
    return result
}
// result не виден за пределами функции
```

**Объявление и затенение переменных**:
- `x = value`: поиск `x` по цепочке областей видимости наружу — если найден, присваивает; если не найден, объявляет новую
- `mut x = value`: явное новое изменяемое объявление, запрещено совпадение имён с внешней областью
- В одной области видимости любое имя может быть объявлено только один раз

> **Подробное определение**: полные правила области видимости, объявления переменных и механизма затенения см. в [Спецификации системы модулей](./modules.md#第四章作用域).

### 3.9 Оператор for

```
ForStmt     ::= 'for' 'mut'? Identifier 'in' Expr Block
```

#### 3.9.1 Семантика: каждая итерация — новая привязка значения

Семантика цикла for в YaoXiang отличается от традиционных языков: **каждая итерация — это новая привязка значения, а не модификация одной и той же переменной**.

```yaoxiang
// Пример: for i in 1..5
for i in 1..5 {
    print(i)
}
```

**Процесс выполнения**:

| Итерация | Поведение переменной цикла |
|------|----------------|
| 1-я | Создаётся новая привязка `i = 1`, выполняется тело цикла, печатается 1 |
| 2-я | Создаётся новая привязка `i = 2` (предыдущая привязка уничтожена), выполняется тело цикла, печатается 2 |
| 3-я | Создаётся новая привязка `i = 3`, выполняется тело цикла, печатается 3 |
| 4-я | Создаётся новая привязка `i = 4`, выполняется тело цикла, печатается 4 |
| Завершение | Тело цикла завершается, привязка уничтожается |

**Ключевой момент**: после каждой итерации привязка, созданная в этой итерации, уничтожается. Следующая итерация — это совершенно новая привязка, не имеющая никакого отношения к привязке предыдущей итерации.

#### 3.9.2 Различие между for и for mut

| Синтаксис | Изменяемость переменной цикла | Описание |
|------|----------------|------|
| `for i in 1..5` | Неизменяемая | В теле цикла нельзя модифицировать привязку |
| `for mut i in 1..5` | Изменяемая | В теле цикла можно модифицировать привязку |

```yaoxiang
// Допустимо: каждая итерация привязывает новое значение, модификация не нужна
for i in 1..5 {
    print(i)  // чтение значения i
}

// Ошибка: неизменяемая привязка, модификация невозможна
for i in 1..5 {
    i = i + 1  // ошибка: нельзя модифицировать неизменяемую привязку
}

// Допустимо: использование for mut разрешает модификацию привязки
for mut i in 1..5 {
    i = i + 1  // модификация разрешена
}
```

#### 3.9.3 Проверка затенения

YaoXiang запрещает затенение переменных. Переменная цикла for не может иметь то же имя, что и переменная во внешней области видимости:

```yaoxiang
// Ошибка: i уже объявлена во внешней области
i = 10
for i in 1..5 {
    print(i)
}

// Правильно: используйте разные имена переменных
i = 10
for j in 1..5 {
    print(j)
}
```

Это правило применяется ко всем блокам кода, подробности см. в [4.3 Правила затенения](./modules.md#43-遮蔽规则).

#### 3.9.4 Сравнение с другими языками

| Язык | Семантика переменной цикла for |
|------|------------------|
| YaoXiang | Каждая итерация — новая привязка значения |
| Rust | Модификация одной и той же переменной (требуется mut) |
| Python | Модификация одной и той же переменной (mut не требуется) |
| C/C++ | Модификация одной и той же переменной (требуется указатель или ссылка) |

**Обоснование дизайна**: YaoXiang использует семантику привязки по следующим причинам:

1. **Более соответствует естественной семантике**
   В естественном языке «для каждого элемента x в коллекции» означает, что каждый x — это отдельная сущность. `for i in 1..5` в YaoXiang читается как «для каждого i от 1 до 5», и i в каждой итерации — это совершенно новая привязка, что согласуется с человеческой интуицией.

2. **Предотвращение случайных модификаций**
   Семантика неизменяемой привязки по умолчанию означает, что в теле цикла невозможно случайно модифицировать переменную цикла. Не нужно беспокоиться о том, что где-то в сложном теле цикла случайно напишется `i = ...`, что приведёт к трудноотслеживаемым ошибкам.

3. **Высокопроизводительные решения всегда под рукой**
   Когда действительно необходимо повторно использовать переменную между итерациями (например, аккумулятор, кэш), достаточно использовать объявление `for mut` для переключения в режим изменяемой привязки. Это яснее, чем неявное разделение состояния — намерение выражается явно через синтаксис, а не скрывается в поведении времени выполнения.

### 3.10 Оператор spawn

```
SpawnBlock  ::= '(' Pattern (',' Pattern)* ')' '=' 'spawn' '{' Expr (',' Expr)* '}'
SpawnFor    ::= Identifier '=' 'spawn' 'for' 'mut'? Identifier 'in' Expr '{' Expr '}'
SpawnStmt   ::= SpawnBlock | SpawnFor
```

**spawn блок**: явное объявление параллельной области, выражения внутри блока выполняются параллельно.

```yaoxiang
(result_a, result_b) = spawn {
    parse(fetch("url1")),
    parse(fetch("url2"))
}
```

**spawn цикл**: цикл с параллелизмом по данным.

```yaoxiang
results = spawn for item in items {
    process(item)
}
```

## Приложение: Краткий справочник по синтаксису

### A.1 Управление потоком выполнения

Expr '?'              // распространение ошибки (тип Result)

### A.3 Синтаксис match

```
match value {
    pattern1 => expr1,
    pattern2 if guard => expr2,
    _ => default_expr,
}
```
