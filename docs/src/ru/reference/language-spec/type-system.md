# Спецификация системы типов

Настоящий документ определяет спецификацию системы типов языка программирования YaoXiang, включая базовые типы, составные типы, обобщения и trait.

---

## Глава 0: Теоретические основы

### 0.1 Соответствие Curry-Howard

Соответствие Curry-Howard (Curry-Howard correspondence) является теоретической основой системы типов YaoXiang. Оно раскрывает глубокую связь между системой типов языка программирования и математической логикой:

| Логика | Язык программирования |
|--------|-----------------------|
| Высказывание \(P\) | Тип `Type` |
| Доказательство \(p: P\) | Программа `x: T = ...` |
| Импликация \(P \rightarrow Q\) | Тип функции `(P) -> Q` |
| Конъюнкция \(P \wedge Q\) | Тип произведения `{ a: P, b: Q }` |
| Дизъюнкция \(P \vee Q\) | Тип суммы `{ a(P) \| b(Q) }` |
| Универсальная квантификация \(\forall x:T. P(x)\) | Обобщения `(T: Type) -> ...` |
| Истина \(\top\) | Пустой тип `{}` |
| Ложь \(\bot\) | `Void` / `Never` |
| Тип-вселенная \(Type_n : Type_{n+1}\) | Иерархия вселенных (защита от парадокса Рассела) |
| Математическая индукция | Типовый `match` |

### 0.2 Типы как высказывания, программы как доказательства

В YaoXiang данное соответствие является основным принципом проектирования:

- **Тип — это логическое высказывание**. `Int` — высказывание «целое число существует», `fn(a: Int, b: Int) -> Int` — высказывание «для двух целых чисел существует целое число».
- **Проверка типов — это верификация доказательства**. Когда программа проходит проверку типов, это эквивалентно конструктивному доказательству логического высказывания.
- **Завершающиеся типовые вычисления соответствуют корректной индуктивной дедукции**. Типовые семейства YaoXiang (такие как сопоставление с образцом `Add` на `Nat`) по сути являются кодированием математической индукции на уровне типов.

### 0.3 Влияние на проектирование языка

Соответствие Curry-Howard в YaoXiang проявляется конкретно следующим образом:

1. **Иерархия вселенных** (RFC-010): `Type₀ : Type₁ : Type₂ …` позволяет избежать логических парадоксов (парадокс Жирара), вызываемых `Type: Type`
2. **Типовые семейства** (RFC-011): типовый pattern matching натуральных чисел `Nat(Zero/Succ)` соответствует индуктивному доказательству в аксиоматике Пеано
3. **Условные типы** (RFC-011): `If: (C: Bool, T: Type, E: Type) -> Type` соответствует case-дизъюнкции в логике
4. **Зависимые от значений типы** (RFC-011): `Vec: (n: Int) -> Type` соответствует конечной квантификации «для каждого целого n существует тип»

## Глава 3: Составные типы

### 3.1 Типы записи

**Унифицированный синтаксис**: `Name: Type = { field1: Type1, field2: Type2, ... }`

```
RecordType  ::= '{' FieldList? '}'
FieldList   ::= Field (',' Field)* ','?
Field       ::= Identifier ':' TypeExpr
            |  Identifier                 // Ограничение интерфейса
```

```yaoxiang
// Простой тип записи
Point: Type = { x: Float, y: Float }

// Пустой тип записи
Empty: Type = {}

// Обобщённый тип записи
Pair: (T: Type) -> Type = { first: T, second: T }

// Тип записи, реализующий интерфейс
Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable
}
```

**Правила**:
- Типы записи определяются с помощью фигурных скобок `{}`
- После имени поля непосредственно следует двоеточие и тип
- Имена интерфейсов внутри тела типа означают реализацию этого интерфейса

#### 3.1.1 Значения полей по умолчанию

Поля типа могут иметь значения по умолчанию, которые можно не указывать при конструировании:

```yaoxiang
// Поле со значением по умолчанию — необязательно при конструировании
Point: Type = {
    x: Float = 0,
    y: Float = 0
}

// Использование
Point()           // -> Point(x=0, y=0)
Point(x=1)       // -> Point(x=1, y=0)
Point(x=1, y=2) // -> Point(x=1, y=2)

// Поле без значения по умолчанию — обязательно при конструировании
Point2: Type = {
    x: Float,
    y: Float
}

// Использование
Point2(x=1, y=2) // Правильно
Point2()          // Ошибка
```

**Правила**:
- `field: Type = expression` -> со значением по умолчанию, необязательно при конструировании
- `field: Type` -> без значения по умолчанию, обязательно при конструировании

#### 3.1.2 Встроенные привязки

Внутри определения типа можно напрямую привязывать методы:

```yaoxiang
// Способ 1: Привязка внешней функции
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]    // Привязка к позиции 0
}
// Вызов: p1.distance(p2) -> distance(p1, p2)

// Способ 2: Анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) -> distance(p1, p2)
```

### 3.2 Перечислимые типы (типы вариантов)

```
EnumType    ::= '{' Variant ('|' Variant)* '}'
Variant     ::= Identifier (':' TypeExpr)?
```

**Синтаксис**: `Name: Type = { Variant1 | Variant2(params) | ... }`

```yaoxiang
// Вариант без параметров
Color: Type = { red | green | blue }

// Вариант с параметрами
Option: (T: Type) -> Type = { some(T) | none }

// Смешанный
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

// Вариант без параметров эквивалентен конструктору без параметров
Bool: Type = { true | false }
```

### 3.3 Типы интерфейсов

```
InterfaceType ::= '{' FnField (',' FnField)* ','?
FnField       ::= Identifier ':' FnType
FnType        ::= '(' ParamTypes? ')' '->' TypeExpr
```

**Синтаксис**: интерфейс — это тип записи, все поля которого являются типами функций

```yaoxiang
// Определение интерфейса
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой интерфейс
EmptyInterface: Type = {}
```

**Реализация интерфейса**: тип реализует интерфейс путём перечисления имён интерфейсов в конце определения

```yaoxiang
// Тип, реализующий интерфейс
Point: Type = {
    x: Float,
    y: Float,
    Drawable,        // Реализует интерфейс Drawable
    Serializable     // Реализует интерфейс Serializable
}
```

**Прямое присваивание интерфейсу**: конкретный тип можно напрямую присвоить переменной типа интерфейса (структурное подтипование)

```yaoxiang
// Прямое присваивание (конкретный тип известен во время компиляции -> вызов без накладных расходов)
d: Drawable = Circle(1)
d.draw(screen)        // После компиляции: прямой вызов circle_draw, без vtable

// Возврат из функции (конкретный тип неизвестен во время компиляции -> вызов через vtable)
d: Drawable = get_shape()
d.draw(screen)        // Поиск метода через vtable

// Интерфейс как параметр функции
process: (d: Drawable) -> Void = d.draw(screen)
```

**Стратегия оптимизации компиляции**:

| Сценарий | Результат вывода | Способ вызова |
|----------|------------------|---------------|
| Прямое присваивание конкретного типа | Конкретный тип определён | Прямой вызов (без накладных расходов) |
| Возврат из функции | Неизвестен | vtable |
| Гетерогенная коллекция | Несколько типов | vtable |

### 3.4 Кортежные типы


### 3.5 Типы функций

## Глава 4: Обобщения

### 4.1 Синтаксис обобщённых параметров

```
GenericType     ::= Identifier '[' TypeArgList ']'
TypeArgList     ::= TypeExpr (',' TypeExpr)* ','?
GenericParams   ::= '[' Identifier (',' Identifier)* ']'
                 |  '[' Identifier ':' TypeBound (',' Identifier ':' TypeBound)* ']'

                 |  Identifier '+' Identifier ('+' Identifier)*
```

### 4.2 Определение обобщённых типов

```yaoxiang
// Базовый обобщённый тип

    ok: (T) -> Self,
    err: (E) -> Self

}
```

## Глава 5: Ограничения типов

### 5.1 Одиночное ограничение


```yaoxiang
// Определение типа интерфейса (как ограничения)
Clone: Type = {

}

// Использование ограничения
```

### 5.2 Множественные ограничения

```yaoxiang
// Синтаксис множественных ограничений
combine: (T: Clone + Add)(a: T, b: T) -> T = {
    a.clone() + b
}

// Сортировка обобщённого контейнера
sort: (T: Clone + PartialOrd)(list: List(T)) -> List(T) = {
    result = list.clone()
    quicksort(&mut result)
    return result
}
```

### 5.3 Ограничения типов функций

```yaoxiang
// Ограничения высшего порядка

compose: (A: Type, B: Type, C: Type, F: Fn(A) -> B, G: Fn(B) -> C)(a: A, f: F, g: G) -> C = g(f(a))
```

```yaoxiang
        if let Some(item) = iter.next() {

        }
    }
```

**Основной дизайн**: использование обобщённого параметра `(n: Int)` + значение параметра `(n: n)` для различения константы времени компиляции и значения времени выполнения.

## Глава 11: Атрибуты типов

YaoXiang использует атрибуты типов (type properties) для обозначения семантики копирования и параллелизма. Атрибуты типов автоматически выводятся компилятором и не указываются пользователем напрямую.

### 11.1 Dup (неявное поверхностное копирование)

**Dup** (Duplicable) — метка неявного поверхностного копирования. Для типов, реализующих Dup, при присваивании и передаче параметров автоматически выполняется поверхностное копирование (побитовое копирование), при этом исходное и новое значение полностью независимы.

**Типы Dup**:
| Тип | Описание |
|-----|----------|
| `Int`, `Int8`..`Int128` | Все целые числа |
| `Float`, `Float32`, `Float64` | Все числа с плавающей точкой |
| `Bool` | Булевы значения |
| `Char` | Unicode-символы |
| `String` | UTF-8 строки (поверхностное копирование) |
| `Bytes` | Сырые байты (поверхностное копирование) |
| `&T` | Токен чтения (см. главу 12) |

**Типы НЕ Dup** (по умолчанию Move):
| Тип | Описание |
|-----|----------|
| `&mut T` | Токен записи, линейный тип |
| Большинство struct | По умолчанию Move, если только все поля не Dup |
| enum (варианты с параметрами) | Если данные не являются Dup, то весь вариант не Dup |

```yaoxiang
// Тип Dup: свободное копирование
a: Int = 42
b = a           // Dup: неявное поверхностное копирование, a всё ещё доступна
c = a           // Можно копировать многократно

// Тип НЕ Dup (по умолчанию Move)
p: Point = Point(1.0, 2.0)
q = p           // Move, p больше нельзя читать
// r = p        // ❌ Ошибка компиляции: p уже перемещена
```

**Автоматический вывод Dup**:
- Базовые типы (Int, Float, Bool, Char, String, Bytes) автоматически реализуют Dup
- Структуры: реализуют Dup тогда и только тогда, когда все поля типов Dup
- Варианты перечислений: реализуют Dup тогда и только тогда, когда все типы данных всех вариантов Dup

### 11.2 Clone (явное глубокое копирование)

**Clone** — это явный интерфейс глубокого копирования. Любой тип может реализовать Clone, предоставляя метод `.clone()`.

```yaoxiang
// Определение интерфейса Clone (стандартная библиотека)
Clone: Type = {
    clone: (Self) -> Self
}

// Использование
p: Point = Point(1.0, 2.0)
backup = p.clone()    // Глубокое копирование, p всё ещё доступна
p2 = p.clone()        // Можно клонировать многократно
```

### 11.3 Связь между Dup и Clone

**Dup влечёт Clone, но Clone не влечёт Dup**:

```
Dup ⇒ Clone (побитовое копирование полей реализует .clone())
Clone ⇏ Dup (явное глубокое копирование не мешает семантике Move по умолчанию)
```

```yaoxiang
// Dup ⇒ Clone: Int является и Dup, и Clone
x: Int = 42
y = x              // Dup: неявное поверхностное копирование
z = x.clone()      // Clone: явное глубокое копирование (эффект тот же)

// Clone ⇏ Dup: Point может быть Clone, но по умолчанию Move
p: Point = Point(1.0, 2.0)
q = p.clone()      // Clone: явное глубокое копирование, p всё ещё доступна
r = p              // Move: передача владения, потому что Point не Dup
```

**Замысел проектирования**:
- Dup — это обещание «этот тип копируется дёшево/естественно»
- Clone — это способность «я могу дать вам независимую копию»
- Большинство struct не реализуют Dup автоматически, сохраняя Move по умолчанию — нулевое копирование для высокой производительности

### 11.4 Send / Sync (невидимы для пользователя)

**Send** и **Sync** — это не пользовательские атрибуты типов; они автоматически обрабатываются компилятором и ключевым словом `ref`.

| Атрибут | Значение | Как пользователь запускает |
|---------|----------|---------------------------|
| **Send** | Можно безопасно передавать между задачами | `ref` при межзадачном использовании — компилятор автоматически выбирает Arc |
| **Sync** | Можно безопасно разделять между задачами | `ref` при межзадачном использовании — компилятор автоматически выбирает Arc |

**Правила автоматического вывода (внутренние для компилятора)**:

| Тип | Send | Sync | Описание |
|-----|------|------|----------|
| Значимые типы (Int, Float, Point...) | Да | Да | Передача по значению естественно безопасна |
| `ref T` | Да | Да | Компилятор автоматически выбирает Rc (одна задача) / Arc (между задачами) |
| `&T` / `&mut T` | Нет | Нет | Токены не могут пересекать границы задач |
| `*T` | Нет | Нет | Голые указатели — однопоточные |

```yaoxiang
// Межзадачное разделение: ref автоматически обрабатывает Send/Sync
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }    // Компилятор: между задачами → Arc (Send + Sync)
    spawn { use(data) }    // Компилятор: между задачами → Arc (Send + Sync)
}

// Токены не могут пересекать задачи (не Send)
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }   // ❌ Ошибка компиляции: &T не реализует Send
}
```

**Пользователю не нужно заботиться о Send/Sync**: ключевое слово `ref` инкапсулирует всю логику параллельной безопасности.

## Глава 12: Типы заимствования токенов

### 12.1 Основные концепции

`&T` и `&mut T` — это **токены компиляции нулевого размера**. Они не являются «ссылками», а представляют собой **типовые доказательства прав доступа**.

```
&T      →  Нулевой размер, Dup (копируемый), предоставляет права только для чтения
&mut T  →  Нулевой размер, Linear (не копируемый), предоставляет права эксклюзивного чтения/записи
```

**Ключевые характеристики**:
- Токены — это **обычные типы**, подчиняющиеся тем же правилам области видимости, что и все остальные типы
- Не требуют аннотаций времени жизни `'a`
- Не требуют специального проверяющего заимствования — атрибуты типа (Dup/Linear) естественным образом выводят права
- Полностью исчезают после компиляции, нулевые накладные расходы во время выполнения

### 12.2 Базовое использование

```yaoxiang
// Сторона метода: объявление типа параметра, определяющего необходимые права
Point.print: (self: &Point) -> Void = {
    print(self.x)               // Токен &Point предоставляет права чтения
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx        // Токен &mut Point предоставляет права записи
    self.y = self.y + dy
}

// Сторона вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                       // Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)               // Компилятор автоматически создаёт токен &mut Point
p.print()                       // OK, предыдущий токен освобождён после завершения вызова shift

// Сосуществование нескольких токенов &T — тип Dup позволяет свободно копировать
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)
}
d = distance(p, p2)
```

### 12.3 Область видимости и распространение токенов

Токены — обычные типы, поэтому поддерживают все операции обычных типов:

**Возврат токенов** — токены распространяются вместе с возвращаемым значением:

```yaoxiang
// ✅ Дочерний токен и родительский токен возвращаются вместе
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()        // Токен возвращается вызывающему
print(px_ref)                    // OK, токен всё ещё в области видимости
```

**Хранение в структуре** — структуры могут содержать поля-токены:

```yaoxiang
// ✅ Структура содержит токен как поле
Window: Type = {
    target: Point,
    view: &Point,              // Поле-токен — держит представление только для чтения на target
}
```

**Захват в замыканиях** — замыкания захватывают токены как любые значения:

```yaoxiang
// ✅ Замыкание захватывает токен &Float (тип Dup, свободно копируется в замыкание)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}
```

### 12.4 Автоматический выбор заимствования

Компилятор на стороне вызова автоматически выбирает по следующему приоритету:

```
1. Если фактический параметр ещё используется в дальнейшем → создать токен (&T или &mut T)
2. Если фактический параметр больше не используется → Move
3. Приоритет сопоставления: &T < &mut T < Move
```

```yaoxiang
p = Point(1.0, 2.0)
p.print()          // print объявляет &self → компилятор создаёт токен &Point
p.shift(1.0, 1.0)  // shift объявляет &mut self → компилятор создаёт токен &mut Point
p2 = p             // Больше не используется → Move
```

### 12.5 Механизм заморозки

Токен `&mut T` можно временно «заморозить» для получения токена `&T`:

```yaoxiang
modify_and_read: (p: &mut Point) -> Void = {
    p.x = 10.0                   // Использование &mut Point для модификации
    
    // Заморозить &mut, получить представление только для чтения
    view: &Point = freeze(p)     // p заморожен здесь
    print(view.x)                // Чтение через &Point
    print(view.y)
    // view покидает область видимости, заморозка снята
    
    p.y = 20.0                   // &mut Point снова доступен
}
```

Семантика `freeze`:
- Принимает `&mut T`, возвращает `&T`
- Пока живёт `&T`, исходный `&mut T` недоступен
- После выхода `&T` из области видимости `&mut T` автоматически восстанавливается
- Это **потокочувствительный анализ живучести** — компилятор отслеживает состояние токена внутри тела функции

### 12.6 Обнаружение конфликтов токенов

Компилятор выполняет **потокочувствительный анализ живучести** значений токенов, отслеживая состояние каждого токена (активен/заморожен/перемещён):

```yaoxiang
// ❌ &mut и производный &T не могут быть активны одновременно
bad_alias: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)     // p заморожен
    p.x = 10.0                   // ❌ Ошибка компиляции: WriteToken в замороженном состоянии
    print(view.x)
}

// ✅ После снятия заморозки можно продолжить использовать &mut
good_seq: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)     // p заморожен
    print(view.x)                // Использование &T
    // view покидает область видимости, заморозка снята
    p.x = 10.0                   // ✅ WriteToken восстановлен
}

// ❌ Нельзя одновременно создавать &mut токен и другие токены из одного аргумента
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)                  // ❌ p одновременно получает &mut и & токены
```

### 12.7 Внутреннее устройство компилятора: механизм брендов

Пользователь никогда не сталкивается с брендами. Компилятор внутренне назначает каждому токену компиляционно-уникальный идентификатор:

```
Пользователь видит           Внутреннее представление компилятора
────────────────────────────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — компиляционно-уникальное целое
&mut Point     →  WriteToken(Point, #M)   // #M — компиляционно-уникальное целое
```

Назначение брендов:
- **Защита от подделки**: токены можно получить только от владельца-капсулы или операции freeze, их нельзя создать из воздуха
- **Отслеживание происхождения**: производные `&Float` от доступа к полям несут производный бренд (`#N.field_x`), компилятор может отследить связь до родительского токена
- **Обнаружение конфликтов**: WriteToken и производный ReadToken из одного источника не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнинга; в сгенерированном машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

### 12.8 Типы суммы токенов заимствования

```
&BorrowToken ::= &T          // ReadToken (Dup, копируемый)
               | &mut T      // WriteToken (Linear, эксклюзивный)
```

### 12.9 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Назначение | Взглянуть / изменить на месте | Совместное владение |
| Область действия | По области видимости токена | За пределами области видимости |
| Стоимость | Нулевые накладные расходы (нулевой размер, исчезает после компиляции) | Rc или Arc (выбирает компилятор) |
| Побег из области | Возможен (токен распространяется через возвращаемое значение/структуру/замыкание) | Предназначен для побега |
| Между задачами | Невозможен (токен не реализует Send) | Возможен (компилятор автоматически выбирает Arc) |
| Обнаружение циклов | Не относится | Тихая проверка внутри задачи, lint между задачами |
