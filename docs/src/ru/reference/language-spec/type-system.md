# Спецификация системы типов

Данный документ определяет спецификацию системы типов языка программирования YaoXiang, включая базовые типы, составные типы, дженерики и trait.

---

## Глава 0: Теоретические основы

### 0.1 Соответствие Карри — Ховарда

Соответствие Карри — Ховарда (Curry-Howard correspondence) является теоретической основой системы типов YaoXiang. Оно раскрывает глубокую связь между системой типов языка программирования и математической логикой:

| Логика | Язык программирования |
|--------|----------------------|
| Пропозиция \(P\) | Тип `Type` |
| Доказательство \(p: P\) | Программа `x: T = ...` |
| Импликация \(P \rightarrow Q\) | Тип-функция `(P) -> Q` |
| Конъюнкция \(P \wedge Q\) | Тип-произведение `{ a: P, b: Q }` |
| Дизъюнкция \(P \vee Q\) | Тип-сумма `{ a(P) \| b(Q) }` |
| Квантор всеобщности \(\forall x:T. P(x)\) | Дженерики `(T: Type) -> ...` |
| Истина \(\top\) | Пустой тип `{}` |
| Ложь \(\bot\) | `Void` / `Never` |
| Вселенная типов \(Type_n : Type_{n+1}\) | Стратификация вселенных (защита от парадокса Рассела) |
| Математическая индукция | Сопоставление с образцом на уровне типов |

### 0.2 Тип как пропозиция, программа как доказательство

В YaoXiang это соответствие является первостепенным принципом дизайна:

- **Тип — это логическая пропозиция**. `Int` — это пропозиция «существования целого числа», `fn(a: Int, b: Int) -> Int` — это пропозиция «для двух заданных целых чисел существует целое число».
- **Проверка типов — это верификация доказательства**. Когда программа проходит проверку типов, это эквивалентно конструктивному доказательству логической пропозиции.
- **Завершающиеся вычисления на уровне типов соответствуют корректным индуктивным рассуждениям**. Семейства типов YaoXiang (например, сопоставление с образцом для `Add` на `Nat`) по сути являются типизированным кодированием математической индукции.

### 0.3 Влияние на дизайн языка

Конкретные проявления соответствия Карри — Ховарда в YaoXiang:

1. **Стратификация вселенных** (RFC-010): `Type₀ : Type₁ : Type₂ …` позволяет избежать логических парадоксов (парадокс Жирара), вызванных `Type: Type`
2. **Семейства типов** (RFC-011): сопоставление с образцом на уровне типов для натуральных чисел `Nat(Zero/Succ)` соответствует индуктивному доказательству по аксиомам Пеано
3. **Условные типы** (RFC-011): `If: (C: Bool, T: Type, E: Type) -> Type` соответствует логической дизъюнкции case
4. **Зависимые от значений типы** (RFC-011): `Vec: (n: Int) -> Type` соответствует конечной квантификации «для каждого целого n существует тип»

## Глава 3: Составные типы

### 3.1 Record type

**Унифицированный синтаксис**: `Name: Type = { field1: Type1, field2: Type2, ... }`

```
RecordType  ::= '{' FieldList? '}'
FieldList   ::= Field (',' Field)* ','?
Field       ::= Identifier ':' TypeExpr
            |  Identifier                 // ограничение интерфейса
```

```yaoxiang
// Простой record type
Point: Type = { x: Float, y: Float }

// Пустой record type
Empty: Type = {}

// Record type с дженериками
Pair: (T: Type) -> Type = { first: T, second: T }

// Record type с реализацией интерфейса
Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable
}
```

**Правила**:
- Record type определяется с помощью фигурных скобок `{}`
- После имени поля указывается двоеточие и тип
- Имя интерфейса в теле типа означает реализацию этого интерфейса

> **Принадлежность пространству имён**: префикс `Type.name` (например, `Point.draw`) означает, что функция принадлежит пространству имён `Point`.
> Он не вызывает неявных привязок. Чтобы синтаксис вызова через `.` (например, `p.draw()`) работал, необходима явная привязка:
> `Point.draw = draw[0]`.
> Подробности см. в RFC-004 и RFC-010.

#### 3.1.1 Значения полей по умолчанию

Поля типов могут иметь значения по умолчанию, которые при конструировании указывать необязательно:

```yaoxiang
// Поле со значением по умолчанию — при конструировании необязательно
Point: Type = {
    x: Float = 0,
    y: Float = 0
}

// Использование
Point()           // -> Point(x=0, y=0)
Point(x=1)       // -> Point(x=1, y=0)
Point(x=1, y=2) // -> Point(x=1, y=2)

// Поле без значения по умолчанию — при конструировании обязательно
Point2: Type = {
    x: Float,
    y: Float
}

// Использование
Point2(x=1, y=2) // Корректно
Point2()          // Ошибка
```

**Правила**:
- `field: Type = expression` -> значение по умолчанию, при конструировании необязательно
- `field: Type` -> без значения по умолчанию, при конструировании обязательно

#### 3.1.2 Встроенные привязки

В теле определения типа можно напрямую привязывать методы:

```yaoxiang
// Способ 1: привязка ссылки на внешнюю функцию
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]    // привязка к позиции 0
}
// Вызов: p1.distance(p2) -> distance(p1, p2)

// Способ 2: анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) -> distance(p1, p2)
```

### 3.2 Interface type

```
InterfaceType ::= '{' FnField (',' FnField)* ','?
FnField       ::= Identifier ':' FnType
FnType        ::= '(' ParamTypes? ')' '->' TypeExpr
```

**Синтаксис**: интерфейс — это record type, все поля которого имеют тип-функцию

```yaoxiang
// Определение интерфейса
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой интерфейс
EmptyInterface: Type = {}
```

**Реализация интерфейса**: тип реализует интерфейс путём перечисления его имени в конце определения

```yaoxiang
// Тип, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,        // реализует интерфейс Drawable
    Serializable     // реализует интерфейс Serializable
}
```

**Прямое присваивание интерфейсу**: конкретный тип можно напрямую присвоить переменной интерфейсного типа (структурная подтипизация)

```yaoxiang
// Прямое присваивание (конкретный тип известен на этапе компиляции -> вызов без накладных расходов)
d: Drawable = Circle(1)
d.draw(screen)        // после компиляции: прямой вызов circle_draw, без vtable

// Возвращаемое значение функции (конкретный тип неизвестен на этапе компиляции -> вызов через vtable)
d: Drawable = get_shape()
d.draw(screen)        // поиск метода через vtable

// Интерфейс в качестве параметра функции
process: (d: Drawable) -> Void = d.draw(screen)
```

**Стратегии оптимизации на этапе компиляции**:

| Сценарий | Результат вывода | Способ вызова |
|----------|------------------|---------------|
| Прямое присваивание конкретного типа | Конкретный тип определим | Прямой вызов (без накладных расходов) |
| Возвращаемое значение функции | Неизвестно | vtable |
| Гетерогенная коллекция | Несколько типов | vtable |

### 3.4 Tuple type

```
TupleType   ::= '(' TypeList? ')'
TypeList    ::= TypeExpr (',' TypeExpr)* ','?
```

### 3.5 Function type

```
FnType      ::= '(' ParamList? ')' '->' TypeExpr
ParamList   ::= TypeExpr (',' TypeExpr)*
```

## Глава 4: Дженерики

### 4.1 Синтаксис параметров дженериков

Параметры дженериков являются частью типа-функции и используют унифицированный синтаксис `()` наравне с обычными параметрами:

```
GenericType     ::= Identifier '(' TypeArgList ')'
TypeArgList     ::= TypeExpr (',' TypeExpr)* ','?
TypeBound       ::= Identifier
                 |  Identifier '+' Identifier ('+' Identifier)*
```

В определении обобщённого типа `(T: Type)` — это сигнатура параметров конструктора типов, а `-> Type` обозначает возвращаемый тип:

```yaoxiang
List: (T: Type) -> Type = { ... }
Map: (K: Type, V: Type) -> Type = { ... }
```

В обобщённой функции параметры типов также объявляются в сигнатуре, компилятор автоматически выводит их из фактических аргументов:

```yaoxiang
map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R)) = ...
```

### 4.2 Определение обобщённых типов

```yaoxiang
// Базовый обобщённый тип
Option: (T: Type) -> Type = {
    some: (T) -> Option(T),
    none: () -> Option(T)
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Result(T, E),
    err: (E) -> Result(T, E)
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — просто соглашение об именовании, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T)
}
```

### 4.3 Вывод типов

```yaoxiang
// Компилятор автоматически выводит параметры дженериков
numbers: List(Int) = List(1, 2, 3)  // компилятор выводит List(Int)
```

## Глава 5: Type constraint

### 5.1 Single constraint

```
ConstrainedType ::= '(' Identifier ':' TypeBound ')' TypeExpr
```

```yaoxiang
// Определение интерфейса (в качестве ограничения)
Clone: Type = {
    clone: () -> Clone
}

// Использование ограничения
clone: (T: Clone)(value: T) -> T = value.clone()
```

### 5.2 Multiple constraint

```yaoxiang
// Синтаксис множественного ограничения
combine: (T: Clone + Add)(a: T, b: T) -> T = {
    a.clone() + b
}

// Сортировка обобщённой коллекции
sort: (T: Clone + PartialOrd)(list: List(T)) -> List(T) = {
    result = list.clone()
    quicksort(&mut result)
    return result
}
```

### 5.3 Ограничения типа-функции

```yaoxiang
// Ограничение для функций высшего порядка
call_twice: (T: Type, F: () -> T)(f: F) -> (T, T) = (f(), f())

compose: (A: Type, B: Type, C: Type, F: (A) -> B, G: (B) -> C)(a: A, f: F, g: G) -> C = g(f(a))
```

## Глава 6: Associated type

### 6.1 Определение ассоциированного типа

```
AssociatedType ::= Identifier ':' TypeExpr
```

```yaoxiang
// Trait Iterator (использует синтаксис record type)
Iterator: (T: Type) -> Type = {
    Item: T,                    // ассоциированный тип
    next: () -> Option(T),
    has_next: () -> Bool
}

// Использование ассоциированного типа
collect: (T: Type, I: Iterator(T))(iter: I) -> List(T) = {
    result = List(T)()
    while iter.has_next() {
        if let Some(item) = iter.next() {
            result.push(item)
        }
    }
    return result
}
```

## Глава 7: Compile-time дженерики

### 7.1 Параметры-константы времени компиляции

```
LiteralType   ::= Identifier ':' Int          // константа времени компиляции
CompileTimeFn ::= '(' Identifier ':' Int ')' '(' Identifier ')' '->' TypeExpr
```

**Ключевая идея дизайна**: использование обобщённого параметра `(n: Int)` + параметра-значения `(n: n)` для различения констант времени компиляции и значений времени выполнения.

```yaoxiang
// Факториал времени компиляции: аргумент должен быть литералом, известным на этапе компиляции
factorial: (n: Int)(n: n) -> Int = {
    match n {
        0 => 1,
        _ => n * factorial(n - 1)
    }
}

// Массив времени компиляции
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),      // массив с размером, известным на этапе компиляции
    length: N
}

// Использование
arr: StaticArray(Int, factorial(5))  // компилятор вычисляет factorial(5) = 120 на этапе компиляции
```

// Базовая специализация: использование перегрузки функций (автоматический выбор компилятором)

## Глава 11: Свойства типов

YaoXiang различает только одно свойство типов: линейные vs копируемые. Это выводится компилятором автоматически.

### 11.1 Move (передача ownership по умолчанию)

Все типы по умолчанию следуют семантике Move. Присваивание, передача в качестве аргумента, возврат = передача ownership.

```yaoxiang
p: Point = Point(1.0, 2.0)
q = p           // Move, p больше нельзя читать
```

### 11.2 Dup (поверхностное копирование: копирование дескриптора, разделение данных)

**Свойство Dup используется для типов-ссылок/токенов**. Присваивание Dup-типа = поверхностное копирование — копируется дескриптор/токен, нижележащие данные разделяются. Несколько владельцев указывают на один и тот же блок данных.

| Тип | Свойство | Описание |
|------|------|------|
| `&T` | Dup | Токен чтения нулевого размера, копирование токена = несколько представлений одних данных |
| `ref T` | Dup | Копирование Rc/Arc = счётчик ссылок +1, разделение данных в куче |
| `&mut T` | Linear | Токен записи нулевого размера, эксклюзивный, не может быть скопирован |
| Все остальные типы | Move | Передача ownership по умолчанию |

**Типы примитивных значений** (Int, Float, Bool, Char) обрабатываются компилятором особым образом: при присваивании автоматически копируется значение, два значения полностью независимы. Это встроенное поведение компилятора, не относящееся к свойству Dup.

```yaoxiang
// &T: Dup, свободное алиасирование
view: &Point = &p
view2 = view     // Dup: копирование токена, оба валидны
print(view.x)    // OK
print(view2.x)   // OK

// &mut T: Linear, не может быть скопирован
mut_ref: &mut Point = &mut p
// r2 = mut_ref  // ❌ &mut T не является Dup, нельзя копировать
```

### 11.3 Clone (явное глубокое копирование) и его связь с Dup

**Clone** — это интерфейс явного глубокого копирования. Любой тип может реализовать Clone, предоставляя метод `.clone()`.

```yaoxiang
// Определение интерфейса Clone (стандартная библиотека)
Clone: Type = {
    clone: () -> Clone
}

// Использование
p: Point = Point(1.0, 2.0)
backup = p.clone()    // глубокое копирование, p всё ещё доступен
p2 = p.clone()        // можно клонировать многократно
```

**Различие между Dup и Clone**:

| | Dup | Clone |
|---|---|---|
| **Семантика** | Поверхностное копирование: копирование дескриптора/токена, нижележащие данные разделяются | Глубокое копирование: создание полной независимой копии |
| **Способ вызова** | Неявно (автоматически при присваивании/передаче аргумента) | Явно (`.clone()`) |
| **Влияние изменений** | Взаимное (разделяемые нижележащие данные) | Без взаимного влияния (независимые копии) |
| **Применимые типы** | Токены `&T`, `ref T` | Любой тип, реализующий интерфейс Clone |
| **Стоимость** | Нулевая (токен — тип нулевого размера) | Зависит от типа |

**Dup не влечёт Clone, Clone не влечёт Dup** — это два ортогональных понятия:

```yaoxiang
// Dup-тип: копирование токена, нижележащие данные разделяются
view: &Point = &p
view2 = view        // Dup: копирование токена, оба указывают на один и тот же p
print(view.x)       // OK
print(view2.x)      // OK, видят одни и те же данные

// Тип примитивного значения: автоматическое копирование значения компилятором (не Dup)
x: Int = 42
y = x               // копирование значения, x и y полностью независимы
print(x)            // OK

// Clone: явное глубокое копирование, создание независимой копии
p: Point = Point(1.0, 2.0)
q = p.clone()       // Clone: глубокое копирование, p всё ещё доступен
r = p               // Move: передача ownership, так как Point не является Dup и не типом примитивного значения
```

**Дизайнерская интенция**:
- Dup используется для токенов/ссылочных типов, решая проблему «несколько представлений одних и тех же данных»
- Clone используется в сценариях, где требуется независимая копия, явный вызов делает стоимость видимой
- Копирование типов примитивных значений (Int/Float/Bool/Char) — встроенное поведение компилятора, не относится к Dup
- Большинство пользовательских типов по умолчанию Move, нулевые накладные расходы на копирование, высокая производительность

## Глава 12: Типы токенов заимствования

### 12.1 Основные понятия

`&T` и `&mut T` — это **токены нулевого размера на этапе компиляции**. Это не «ссылки», а «доказательства на уровне типов прав доступа».

```
&T      →  нулевого размера, Dup (может копироваться), предоставляет право чтения
&mut T  →  нулевого размера, Linear (не Dup), предоставляет эксклюзивное право чтения и записи
```

**Ключевые характеристики**:
- Токен — это **обычный тип**, подчиняющийся тем же правилам области видимости, что и все остальные типы
- Аннотации времени жизни `'a` не требуются
- Специальный анализатор заимствований не требуется — свойства типа (Dup/Linear) естественным образом выводят права
- После компиляции полностью исчезают, нулевые накладные расходы во время выполнения

### 12.2 Базовое использование

```yaoxiang
// На стороне метода: объявление типа параметра определяет необходимые права
Point.print: (self: &Point) -> Void = {
    print(self.x)               // токен &Point предоставляет право чтения
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx        // токен &mut Point предоставляет право записи
    self.y = self.y + dy
}

// На стороне вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                       // компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)               // компилятор автоматически создаёт токен &mut Point
p.print()                       // OK, предыдущий токен был освобождён после завершения вызова shift

// Сосуществование нескольких токенов &T — Dup-тип допускает свободное копирование
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)
}
d = distance(p, p2)
```

### 12.3 Область видимости и распространение токенов

Токен — это обычный тип, поэтому он поддерживает все операции, доступные обычным типам:

**Возврат токена** — токен распространяется вместе с возвращаемым значением:

```yaoxiang
// ✅ Дочерний токен возвращается вместе с родительским токеном
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()        // токен возвращается вызывающему коду
print(px_ref)                    // OK, токен всё ещё в области видимости
```

**Сохранение в структуре** — структура может содержать поля-токены:

```yaoxiang
// ✅ Структура несёт токен в качестве поля
Window: Type = {
    target: Point,
    view: &Point,              // поле-токен — содержит представление target только для чтения
}
```

**Захват замыканием** — замыкание захватывает токен так же, как любое другое значение:

```yaoxiang
// ✅ Замыкание захватывает токен &Float (Dup-тип, свободное копирование в замыкание)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}
```

### 12.4 Автоматический выбор заимствования

Компилятор на стороне вызова автоматически выбирает в следующем порядке приоритета:

```
1. Если фактический аргумент впоследствии используется → предпочтительно создать токен (&T или &mut T, согласно сигнатуре метода)
2. Если фактический аргумент впоследствии не используется → Move
3. Порядок приоритета сопоставления: &T < &mut T < Move
```

```yaoxiang
p = Point(1.0, 2.0)
p.print()          // тип параметра print — &Point → компилятор создаёт токен &Point
p.shift(1.0, 1.0)  // тип параметра shift — &mut Point → компилятор создаёт токен &mut Point
p2 = p             // впоследствии не используется → Move
```

### 12.5 Обнаружение конфликтов токенов

Компилятор выполняет **потокочувствительный анализ живучести** значений токенов, отслеживая состояние каждого токена (активен/перемещён):

```yaoxiang
// ❌ &mut и производный &T не могут быть одновременно активны
bad_alias: (p: &mut Point) -> Void = {
    p.x = 10.0                   // ✅ штатное использование WriteToken
    print(p.y)
}

// ✅ Автоматическое освобождение после завершения области видимости токена
good_seq: (p: &mut Point) -> Void = {
    {
        // внутренняя область видимости
        print(p.x)               // использование &mut Point
    }
    // конец внутренней области видимости
    p.x = 10.0                   // ✅ WriteToken всё ещё доступен
}

// ❌ Для одного фактического аргумента нельзя одновременно создать токен &mut и другие токены
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)                  // ❌ p одновременно порождает токены &mut и &
```

### 12.6 Внутренний механизм компилятора: брендирование

Пользователь никогда не сталкивается с брендированием. Компилятор внутри присваивает каждому токену уникальный идентификатор на этапе компиляции:

```
Видимое пользователю           Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — уникальное целое число на этапе компиляции
&mut Point     →  WriteToken(Point, #M)   // #M — уникальное целое число на этапе компиляции
```

Назначение брендирования:
- **Защита от подделки**: токен можно получить только из капсулы владельца, нельзя сконструировать из ничего
- **Отслеживание связей**: доступ к полю порождает `&Float`, несущий производный бренд (`#N.field_x`), что позволяет компилятору отследить родительский токен
- **Обнаружение конфликтов**: WriteToken и производный ReadToken из одного источника не могут быть одновременно активны

Брендирование полностью исчезает после мономорфизации и инлайнинга, в генерируемом машинном коде его нет. **Нулевые накладные расходы во время выполнения.**

### 12.7 Тип-сумма токенов

```
&BorrowToken ::= &T          // ReadToken (Dup, может копироваться)
               | &mut T      // WriteToken (Linear, эксклюзивный)
```

### 12.8 Токен заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Назначение | Взглянуть/изменить на месте | Разделяемое владение |
| Область действия | С областью видимости значения токена | Между областями видимости |
| Стоимость | Нулевые накладные расходы (тип нулевого размера, исчезает после компиляции) | Rc или Arc (выбирает компилятор) |
| Побег | Допустим (токен распространяется через возвращаемое значение/структуру/замыкание) | Изначально предназначен для побега |
| Между задачами | Недопустим (токен не реализует передачу между задачами) | Допустим (компилятор автоматически выбирает Arc) |
| Обнаружение циклов | Не задействовано | Беззвучно в пределах задачи, lint между задачами |
