# Спецификация системы типов

Этот документ определяет спецификацию системы типов языка программирования YaoXiang, включая базовые типы, составные типы, generics и trait.

---

## Глава нулевая: Теоретические основы

### 0.1 Curry-Howard соответствие

Curry-Howard соответствие (Curry-Howard correspondence) — теоретическая основа системы типов YaoXiang. Оно раскрывает глубокое соответствие между системой типов языка программирования и математической логикой:

| Логика | Язык программирования |
|--------|----------|
| Пропозиция \(P\) | type `Type` |
| Доказательство \(p: P\) | программа `x: T = ...` |
| Импликация \(P \rightarrow Q\) | function type `(P) -> Q` |
| Конъюнкция \(P \wedge Q\) | тип произведения `{ a: P, b: Q }` |
| Дизъюнкция \(P \vee Q\) | тип суммы `{ a(P) \| b(Q) }` |
| Универсальная квантификация \(\forall x:T. P(x)\) | generics `(T: Type) -> ...` |
| Истина \(\top\) | `Void` (Unit, имеет значение по умолчанию) |
| Ложь \(\bot\) | `Never` (нуль конструкторов, ни одно значение не может обитать) |
| Вселенная типов \(Type_n : Type_{n+1}\) | иерархия вселенных (для предотвращения парадокса Рассела) |
| case-анализ | type-level `match` |

> **Примечание**: type-level `match` — это классифицирующий разбор (case analysis), а не математическая индукция. Для индукции требуются рекурсивные функции на уровне типов + проверка завершаемости компилятором.

### 0.2 Тип как пропозиция, программа как доказательство

В YaoXiang это соответствие является первичным принципом дизайна:

- **Завершающиеся вычисления на уровне типов соответствуют корректным конструктивным доказательствам**. Семейства типов YaoXiang (такие как `Add` для case-анализа + рекурсивного вызова на `Nat`) по сути являются типизированным кодированием математической индукции — при условии, что компилятор может выполнить проверку завершаемости.
- **Проверка типов — это верификация доказательства**. Когда программа проходит проверку типов, это эквивалентно конструктивному доказательству логической пропозиции.

### 0.3 Влияние на дизайн языка

Конкретное воплощение Curry-Howard соответствия в YaoXiang:

1. **Иерархия вселенных** (RFC-010): `Type₀ : Type₁ : Type₂ …` позволяет избежать логического парадокса, вызванного `Type: Type` (парадокс Жирара)
2. **Семейства типов** (RFC-011): case-анализ на уровне типов + рекурсивные вызовы для натуральных чисел `Nat(Zero/Succ)` соответствуют аксиомам Пеано — при условии, что компилятор выполняет проверку завершаемости
3. **Conditional type** (RFC-011): `If: (C: Bool, T: Type, E: Type) -> Type` соответствует case-дизъюнкции в логике
4. **Зависимые от значения типы** (RFC-011): `Vec: (n: Int) -> Type` соответствует ограниченной квантификации «для каждого целого n существует тип»

## Глава первая: Классификация типов

### 1.1 Выражения типов

```
TypeExpr    ::= PrimitiveType
              | RecordType
              | InterfaceType
              | TupleType
              | FnType
              | GenericType
              | TypeRef
              | TypeUnion
              | TypeIntersection
```

> **Замечание о дизайне**: Хотя RFC-010 предлагает унифицированную модель «всё есть присваивание» (`name: type = value`), на уровне синтаксиса типы и значения всё же необходимо различать. В реализации компилятора `Type` и `Expr` — это два независимых перечисления AST (`ast.rs:406` и `ast.rs:25`), `TypeExpr` служит BNF-заполнителем, соответствующим перечислению `Type` в реализации, и обозначает «в этой позиции ожидается тип».

## Глава вторая: Базовые типы

### 2.1 Primitive type

| Тип | Логическое соответствие | Описание | Размер по умолчанию |
|------|---------|------|----------|
| `Type` | — | meta type | 0 байт |
| `Never` | ⊥ (ложь/пустой тип) | Ноль конструкторов, ни одно значение не существует. Тип возврата для расходимости/panic. `Never <: T` для любого T. | 0 байт |
| `Void` | ⊤ (истина/Unit) | Имеет void-значение по умолчанию, тип произведения с нулём полей. `x: Void = <по умолчанию>` допустимо. | 0 байт |
| `Bool` | — | Булево значение: `true` / `false` | 1 байт |
| `Int` | — | Знаковое целое | 8 байт |
| `Uint` | — | Беззнаковое целое | 8 байт |
| `Float` | — | Число с плавающей точкой | 8 байт |
| `String` | — | UTF-8 строка | переменный |
| `Char` | — | Unicode-символ | 4 байта |
| `Bytes` | — | Сырые байты | переменный |

Целые с разрядностью: `Int8`, `Int16`, `Int32`, `Int64`, `Int128`
Числа с плавающей точкой с разрядностью: `Float32`, `Float64`

### 2.2 Never и Void: ⊥ и ⊤

`Never` и `Void` — логические примитивы системы типов — соответствуют лжи (⊥) и истине (⊤) соответственно.

**Never (⊥, ложь/пустой тип)** — три непреложных свойства:

1. **Ноль конструкторов**: Ни один literal или выражение не может породить значение типа `Never`. Для `x: Never = ...` нет правой части для записи.
2. **Принцип взрыва**: `Never <: T` для любого типа `T`. `assert(false)` возвращает `Never`, после чего код может пройти проверку типов (хотя никогда не выполнится).
3. **Метка расходимости**: `f: (...) -> Never` означает, что `f` гарантированно не возвращается. Компилятор использует это для анализа мёртвого кода и слияния ветвей `match`.

`Never` — это имя встроенного типа (с тем же путём регистрации, что и `Int`/`Bool`), а не ключевое слово.

**Void (⊤, истина/Unit)** — имеет ровно одного обитателя (void-значение по умолчанию). `Void` — единичный элемент типа произведения с нулём полей. `x: Void = <по умолчанию>` допустимо; функция без явного `return` возвращает `Void`.

## Глава третья: Составные типы

### 3.1 Record type

**Унифицированный синтаксис**: `Name: Type = { field1: Type1, field2: Type2, ... }`

```
RecordType  ::= '{' FieldList? '}'
FieldList   ::= Field (',' Field)* ','?
Field       ::= Identifier ':' TypeExpr
            |  Identifier                 // ограничение интерфейса
```

```yaoxiang
// Простой record type
Point: Type = { x: Float, y: Float }

// Пустой record type
Empty: Type = {}

// Record type с generics
Pair: (T: Type) -> Type = { first: T, second: T }

// Record type, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable
}
```

**Правила**:
- Record type определяется с помощью фигурных скобок `{}`
- После имени поля сразу идут двоеточие и тип
- Имя интерфейса в теле типа указывает на его реализацию

> **Принадлежность к пространству имён**: Префикс `Type.name` (например, `Point.draw`) указывает, что функция принадлежит пространству имён `Point`.
> Он не вызывает никаких неявных привязок. Чтобы синтаксис вызова через `.` (например, `p.draw()`) работал, необходима явная привязка:
> `Point.draw = draw[0]`.
> Подробности см. в RFC-004 и RFC-010.

#### 3.1.1 Значения полей по умолчанию

Поля типа могут иметь значения по умолчанию и быть необязательными при конструировании:

```yaoxiang
// Поля со значениями по умолчанию — необязательны при конструировании
Point: Type = {
    x: Float = 0,
    y: Float = 0
}

// Использование
Point()           // -> Point(x=0, y=0)
Point(x=1)       // -> Point(x=1, y=0)
Point(x=1, y=2) // -> Point(x=1, y=2)

// Поля без значений по умолчанию — обязательны при конструировании
Point2: Type = {
    x: Float,
    y: Float
}

// Использование
Point2(x=1, y=2) // Верно
Point2()          // Ошибка
```

**Правила**:
- `field: Type = expression` -> есть значение по умолчанию, необязательно при конструировании
- `field: Type` -> нет значения по умолчанию, обязательно при конструировании

#### 3.1.2 Builtin binding

В теле определения типа можно напрямую привязывать методы:

```yaoxiang
// Способ 1: Привязка через ссылку на внешнюю функцию
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]    // Привязка к позиции 0
}
// Вызов: p1.distance(p2) -> distance(p1, p2)

// Способ 2: Анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = b.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) -> distance(p1, p2)
```

### 3.2 Interface type

```
InterfaceType ::= '{' FnField (',' FnField)* ','?
FnField       ::= Identifier ':' FnType
FnType        ::= '(' ParamTypes? ')' '->' TypeExpr
```

**Синтаксис**: Интерфейс — это record type, все поля которого имеют function type

```yaoxiang
// Определение интерфейса
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой интерфейс
EmptyInterface: Type = {}
```

**Реализация интерфейса**: Тип реализует интерфейсы, перечисляя их имена в конце определения

```yaoxiang
// Тип, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,        // Реализует интерфейс Drawable
    Serializable     // Реализует интерфейс Serializable
}
```

**Прямое присваивание интерфейсу**: Конкретный тип можно напрямую присвоить переменной интерфейсного типа (структурная подтипизация)

```yaoxiang
// Прямое присваивание (конкретный тип известен на этапе компиляции -> вызов с нулевыми накладными расходами)
d: Drawable = Circle(1)
d.draw(screen)        // После компиляции: прямой вызов circle_draw, без vtable

// Возвращаемое значение функции (неизвестно на этапе компиляции -> вызов через vtable)
d: Drawable = get_shape()
d.draw(screen)        // Поиск метода через vtable

// Интерфейс как параметр функции
process: (d: Drawable) -> Void = d.draw(screen)
```

**Стратегии оптимизации на этапе компиляции**:

| Сценарий | Результат вывода | Способ вызова |
|------|----------|----------|
| Прямое присваивание конкретного типа | Конкретный тип определим | Прямой вызов (нулевые накладные расходы) |
| Возвращаемое значение функции | Неизвестно | vtable |
| Гетерогенная коллекция | Несколько типов | vtable |

### 3.4 Tuple type

```
TupleType   ::= '(' TypeList? ')'
TypeList    ::= TypeExpr (',' TypeExpr)* ','?
```

### 3.5 Function type

```
FnType      ::= '(' ParamList? ')' '->' TypeExpr
ParamList   ::= TypeExpr (',' TypeExpr)*
```

## Глава четвёртая: Generics

### 4.1 Синтаксис параметров generics

Параметры generics являются частью function type и используют унифицированный синтаксис `()` с обычными параметрами:

```
GenericType     ::= Identifier '(' TypeArgList ')'
TypeArgList     ::= TypeExpr (',' TypeExpr)* ','?
TypeBound       ::= Identifier
                 |  Identifier '+' Identifier ('+' Identifier)*
```

В определении обобщённого типа `(T: Type)` — это сигнатура параметра конструктора типа, `-> Type` указывает возвращаемый тип:

```yaoxiang
List: (T: Type) -> Type = { ... }
Map: (K: Type, V: Type) -> Type = { ... }
```

В обобщённых функциях параметры типа также объявляются в сигнатуре, компилятор автоматически выводит их из фактических аргументов:

```yaoxiang
map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R)) = ...
```

### 4.2 Определение обобщённого типа

```yaoxiang
// Базовый обобщённый тип
Option: (T: Type) -> Type = {
    some: (T) -> Option(T),
    none: () -> Option(T)
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Result(T, E),
    err: (E) -> Result(T, E)
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   // self — просто имя по соглашению, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T)
}
```

### 4.3 Type inference

```yaoxiang
// Компилятор автоматически выводит параметры generics
numbers: List(Int) = List(1, 2, 3)  // Компилятор выводит List(Int)
```

## Глава пятая: Type constraint

### 5.1 Single constraint

```
ConstrainedType ::= '(' Identifier ':' TypeBound ')' TypeExpr
```

```yaoxiang
// Определение interface type (как ограничение)
Clone: Type = {
    clone: () -> Clone
}

// Использование ограничения
clone: (T: Clone)(value: T) -> T = value.clone()
```

### 5.2 Multiple constraint

```yaoxiang
// Синтаксис множественных ограничений
combine: (T: Clone + Add)(a: T, b: T) -> T = {
    a.clone() + b
}

// Сортировка обобщённого контейнера
sort: (T: Clone + PartialOrd)(list: List(T)) -> List(T) = {
    result = list.clone()
    quicksort(&mut result)
    return result
}
```

### 5.3 Ограничения на function type

```yaoxiang
// Ограничения на функции высшего порядка
call_twice: (T: Type, F: () -> T)(f: F) -> (T, T) = (f(), f())

compose: (A: Type, B: Type, C: Type, F: (A) -> B, G: (B) -> C)(a: A, f: F, g: G) -> C = g(f(a))
```

## Глава шестая: Associated type

### 6.1 Определение associated type

```
AssociatedType ::= Identifier ':' TypeExpr
```

```yaoxiang
// Iterator trait (с использованием синтаксиса record type)
Iterator: (T: Type) -> Type = {
    Item: T,                    // associated type
    next: () -> Option(T),
    has_next: () -> Bool
}

// Использование associated type
collect: (T: Type, I: Iterator(T))(iter: I) -> List(T) = {
    result = List(T)()
    while iter.has_next() {
        if let Some(item) = iter.next() {
            result.push(item)
        }
    }
    return result
}
```

### 6.2 Обобщённые associated type (GAT)

```yaoxiang
// Более сложный associated type
Container: (T: Type) -> Type = {
    Item: T,
    IteratorType: Iterator(T),  // associated type тоже обобщённый
    iter: () -> IteratorType
}
```

## Глава седьмая: Дженерики compile-time

### 7.1 Параметры-константы compile-time

```
LiteralType   ::= Identifier ':' Int          // константа compile-time
CompileTimeFn ::= '(' Identifier ':' Int ')' '(' Identifier ')' '->' TypeExpr
```

**Основная идея**: Параметр-дженерик `(n: Int)` + параметр-значение `(n: n)` различают константы compile-time и значения runtime.

```yaoxiang
// Факториал compile-time: аргумент должен быть литералом, известным на этапе компиляции
factorial: (n: Int)(n: n) -> Int = {
    match n {
        0 => 1,
        _ => n * factorial(n - 1)
    }
}

// Массив констант compile-time
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),      // Размер известен на этапе компиляции
    length: N
}

// Использование
arr: StaticArray(Int, factorial(5))  // Компилятор вычисляет factorial(5) = 120 на этапе компиляции
```

### 7.2 Массивы констант compile-time

```yaoxiang
// Использование для матричного типа
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows)
}

// Проверка размеров на этапе компиляции
identity_matrix: (T: Add + Zero + One, N: Int)(size: N) -> Matrix(T, N, N) = {
    // ...
}
```

## Глава восьмая: Conditional type

### 8.1 Условный тип If

```
IfType        ::= 'If' '(' BoolExpr ',' TypeExpr ',' TypeExpr ')'
```

```yaoxiang
// If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E
}

// Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)
// IsTrue мост и уточняющий тип Assert (см. §8.3)
IsTrue: (b: Bool) -> Type = match b {
    true => Void,      // ⊤, программа продолжается
    false => Never,    // ⊥, расходимость/ошибка компиляции
}
Assert: (cond: Bool) -> Type = IsTrue(cond)
```
### 8.2 Type family

```yaoxiang
// Преобразование типов на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String
}
```


### 8.3 Уточняющий тип Assert и assert-утверждение

`assert` и `Assert` — две стороны одного уточняющего примитива, автоматически выбираемые dispatch-конвейером по критерию «достижимости свободных переменных предиката на этапе компиляции».

**Основная сигнатура**: `assert: (cond: Bool, ?msg: String | Error) -> Assert(IsTrue(cond))`

**Правила диспетчеризации**:

| Критерий | Режим | Поведение |
|------|------|------|
| Все свободные переменные известны на этапе компиляции (параметры generics, константы compile-time) | CompileTime | Переход в конвейер доказательств: true → стирается в Void, false → ошибка компиляции (Never необитаем) |
| Есть свободные переменные runtime (параметры функций, внешний ввод) | Runtime | Вставка проверки Bool во время выполнения, инъекция уточняющих фактов в потокочувствительный набор предположений Γ |

**Потокочувствительный набор предположений Γ**:

Компилятор поддерживает набор известных пропозиций для каждой точки потока управления:

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
y = x + 1           // Γ = {x > 0, y > 1}  ← SP-распространение
mut x = x - 5       // Γ = {}  ← kill set для mut: старые предположения недействительны
```

После присваивания `mut`-переменной все предположения, связанные с этой переменной, удаляются (kill set). При слиянии ветвей Γ берётся как пересечение предположений из ветвей.

### 9.2 Type intersection

```
TypeIntersection ::= TypeExpr '&' TypeExpr
```

**Синтаксис**: Type intersection `A & B` обозначает тип, одновременно удовлетворяющий и A, и B

```yaoxiang
DrawableSerializable: Type = Drawable & Serializable
```

## Глава десятая: Перегрузка функций и специализация

### 10.1 Перегрузка функций


## Глава одиннадцатая: Свойства типов

В YaoXiang есть только одно различие свойств типов: линейный vs копируемый. Выводится компилятором автоматически.

### 11.1 Move (перенос ownership по умолчанию)

Все типы по умолчанию следуют семантике Move. Присваивание, передача параметров, возврат = перенос ownership.

```yaoxiang
p: Point = Point(1.0, 2.0)
q = p           // Move, p больше нельзя читать
```

### 11.2 Dup (поверхностное копирование: копирование дескриптора, разделение данных)

**Свойство Dup используется для типов-ссылок/токенов**. Присваивание типа Dup = поверхностное копирование — копируется дескриптор/токен, нижележащие данные разделяются. Несколько держателей указывают на одни и те же данные.

| Тип | Свойство | Описание |
|------|------|------|
| `&T` | Dup | Токен чтения нулевого размера, копирование токена = несколько взглядов на одни данные |
| `ref T` | Dup | Копирование Rc/Arc = счётчик ссылок +1, разделяемые данные в куче |
| `&mut T` | Linear | Токен записи нулевого размера, исключительный, не копируется |
| Все остальные типы | Move | Перенос ownership по умолчанию |

**Примитивные типы значений** (Int, Float, Bool, Char) — особая встроенная обработка компилятора: при присваивании автоматически копируется значение, два значения полностью независимы. Это встроенное поведение компилятора, не относится к свойству типа Dup.

```yaoxiang
// &T: Dup, свободное алиасирование
view: &Point = &p
view2 = view     // Dup: копирование токена, оба действительны
print(view.x)    // Допустимо
print(view2.x)   // Допустимо

// &mut T: Linear, нельзя копировать
mut_ref: &mut Point = &mut p
// r2 = mut_ref  // ❌ &mut T не Dup, нельзя копировать
```

### 11.3 Clone (явное глубокое копирование) и его связь с Dup

**Clone** — это интерфейс явного глубокого копирования. Любой тип может реализовать Clone, предоставляя метод `.clone()`.

```yaoxiang
// Определение интерфейса Clone (стандартная библиотека)
Clone: Type = {
    clone: () -> Clone
}

// Использование
p: Point = Point(1.0, 2.0)
backup = p.clone()    // Глубокое копирование, p всё ещё доступен
p2 = p.clone()        // Можно клонировать многократно
```

**Различие между Dup и Clone**:

| | Dup | Clone |
|---|---|---|
| **Семантика** | Поверхностное копирование: копирование дескриптора/токена, нижележащие данные разделяются | Глубокое копирование: создание полной независимой копии |
| **Способ вызова** | Неявное (автоматически при присваивании/передаче) | Явное (`.clone()`) |
| **Влияние модификации** | Взаимное (разделяемые данные) | Отсутствует (независимые копии) |
| **Применимые типы** | Токены `&T`, `ref T` | Любой тип, реализующий интерфейс Clone |
| **Стоимость** | Нулевая (токен — тип нулевого размера) | Зависит от типа |

**Dup не влечёт Clone, Clone не влечёт Dup** — это две ортогональные концепции:

```yaoxiang
// Dup-тип: копирование токена, нижележащие данные разделяются
view: &Point = &p
view2 = view        // Dup: копирование токена, оба указывают на одно и то же p
print(view.x)       // Допустимо
print(view2.x)      // Допустимо, видят одни и те же данные

// Примитивный тип значения: автоматическое копирование значения компилятором (не Dup)
x: Int = 42
y = x               // Копирование значения, x и y полностью независимы
print(x)            // Допустимо

// Clone: явное глубокое копирование, создание независимой копии
p: Point = Point(1.0, 2.0)
q = p.clone()       // Clone: глубокое копирование, p всё ещё доступен
r = p               // Move: перенос ownership, потому что Point не Dup и не примитивный тип значения
```

**Дизайнерское намерение**:
- Dup используется для токенов/ссылочных типов, решает проблему «несколько взглядов на одни и те же данные»
- Clone используется в случаях, когда нужна независимая копия, явный вызов делает стоимость видимой
- Копирование примитивных типов значений (Int/Float/Bool/Char) — встроенное поведение компилятора, не относится к Dup
- Большинство пользовательских типов по умолчанию Move, высокопроизводительно без копирования

## Глава двенадцатая: Токены заимствования

### 12.1 Основные концепции

`&T` и `&mut T` — это **токены нулевого размера compile-time**. Это не «ссылки», а «доказательства прав доступа на уровне типов».

```
&T      →  нулевого размера, замораживает исходные данные (запрещает получение WriteToken в этот период),
          при заморозке множественное чтение безопасно → Dup (копируемый)
&mut T  →  нулевого размера, исключительное чтение-запись (запрещает любые другие токены),
          при исключительном доступе копирование бессмысленно → Linear (не Dup)
```

**Ключевые свойства**:
- Токен — это **обычный тип**, подчиняющийся тем же правилам области видимости, что и все остальные типы
- Не требуются аннотации времени жизни `'a`
- Не требуется специальный проверщик заимствований — свойства типа (Dup/Linear) естественно выводят права доступа
- Полностью исчезают после компиляции, нулевые накладные расходы во время выполнения

### 12.2 Базовое использование

```yaoxiang
// Сторона метода: объявление типа параметра определяет требуемые права
Point.print: (self: &Point) -> Void = {
    print(self.x)               // Токен &Point даёт право чтения
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx        // Токен &mut Point даёт право записи
    self.y = self.y + dy
}

// Сторона вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                       // Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)               // Компилятор автоматически создаёт токен &mut Point
p.print()                       // OK, предыдущий токен уже освобождён после завершения вызова shift

// Несколько токенов &T сосуществуют — Dup-тип разрешает свободное копирование
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)
}
d = distance(p, p2)
```

### 12.3 Область видимости и распространение токенов

Токены — обычные типы, поэтому они поддерживают все операции с обычными типами:

**Возврат токенов** — токены распространяются вместе с возвращаемым значением:

```yaoxiang
// ✅ Дочерний и родительский токены возвращаются вместе
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()        // Токен возвращается вызывающему
print(px_ref)                    // OK, токен всё ещё в области видимости
```

**Хранение в структурах** — структуры могут содержать поля-токены:

```yaoxiang
// ✅ Структура содержит токен как поле
Window: Type = {
    target: Point,
    view: &Point,              // Поле-токен — содержит представление target только для чтения
}
```

**Захват замыканиями** — замыкания захватывают токены, как и любые другие значения:

```yaoxiang
// ✅ Замыкание захватывает токен &Float (Dup-тип, свободное копирование в замыкание)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}
```

### 12.4 Автоматический выбор заимствования

На стороне вызова компилятор автоматически выбирает по следующему приоритету:

```
1. Если фактический аргумент используется далее → предпочтительно создать токен (&T или &mut T, в зависимости от сигнатуры метода)
2. Если фактический аргумент далее не используется → Move
3. Порядок предпочтения: &T < &mut T < Move
```

```yaoxiang
p = Point(1.0, 2.0)
p.print()          // Тип параметра print — &Point → компилятор создаёт токен &Point
p.shift(1.0, 1.0)  // Тип параметра shift — &mut Point → компилятор создаёт токен &mut Point
p2 = p             // Далее не используется → Move
```

### 12.5 Обнаружение конфликтов токенов

Компилятор выполняет **потокочувствительный анализ живучести** значений-токенов, отслеживая состояние каждого токена (активен/перемещён):

```yaoxiang
// ❌ &mut и производный &T не могут быть активны одновременно
bad_alias: (p: &mut Point) -> Void = {
    p.x = 10.0                   // ✅ Нормальное использование WriteToken
    print(p.y)
}

// ✅ Автоматическое освобождение после завершения области видимости токена
good_seq: (p: &mut Point) -> Void = {
    {
        // Внутренняя область видимости
        print(p.x)               // Используется &mut Point
    }
    // Внутренняя область видимости завершена
    p.x = 10.0                   // ✅ WriteToken всё ещё доступен
}

// ❌ Для одного фактического аргумента нельзя одновременно создать токен &mut и другие токены
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)                  // ❌ p одновременно порождает токены &mut и &
```

### 12.6 Внутреннее устройство компилятора: механизм брендирования

Пользователь никогда не взаимодействует с брендами. Компилятор внутри назначает каждому токену уникальный идентификатор compile-time:

```
Видимое пользователю           Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — уникальное целое compile-time
&mut Point     →  WriteToken(Point, #M)   // #M — уникальное целое compile-time
```

Применение брендов:
- **Защита от подделки**: Токен можно получить только из капсулы владельца, нельзя сконструировать из воздуха
- **Отслеживание связей**: Производный от доступа к полю `&Float` несёт производный бренд (`#N.field_x`), компилятор может отследить его до родительского токена
- **Обнаружение конфликтов**: Однопроисходный WriteToken и производный ReadToken не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнинга, в генерируемом машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

### 12.7 Sum type токенов

```
&BorrowToken ::= &T          // ReadToken (замораживает исходные данные → безопасно для Dup)
               | &mut T      // WriteToken (исключительное чтение-запись → Linear)
```

### 12.8 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Что делает | Взглянуть/изменить на месте | Разделяемое ownership |
| Область | С областью видимости значения-токена | Между областями видимости |
| Стоимость | Нулевая (тип нулевого размера, исчезает после компиляции) | Rc или Arc (выбирает компилятор) |
| Побег | Возможен (токен распространяется через возвращаемое значение/структуру/замыкание) | Для этого и предназначен |
| Между задачами | Невозможен (токены не реализуют передачу между задачами) | Возможен (компилятор автоматически выбирает Arc) |
| Обнаружение циклов | Не задействовано | Безмолвно в пределах задачи, lint между задачами |
