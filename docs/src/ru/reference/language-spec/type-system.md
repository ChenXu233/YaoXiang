# Спецификация системы типов

Данный документ определяет спецификацию системы типов языка программирования YaoXiang, включая базовые типы, составные типы, дженерики и Trait.

---

## Глава 0: Теоретические основы

### 0.1 Соответствие Карри-Ховарда

Соответствие Карри-Ховарда (Curry-Howard correspondence) является теоретической основой системы типов YaoXiang. Оно раскрывает глубокую связь между системой типов языка программирования и математической логикой:

| Логика | Язык программирования |
|--------|----------|
| Пропозиция \(P\) | Тип `Type` |
| Доказательство \(p: P\) | Программа `x: T = ...` |
| Импликация \(P \rightarrow Q\) | Тип-функция `(P) -> Q` |
| Конъюнкция \(P \wedge Q\) | Тип-произведение `{ a: P, b: Q }` |
| Дизъюнкция \(P \vee Q\) | Тип-сумма `{ a(P) \| b(Q) }` |
| Универсальная квантификация \(\forall x:T. P(x)\) | Дженерики `(T: Type) -> ...` |
| Истина \(\top\) | `Void` (Unit, имеет значение по умолчанию) |
| Ложь \(\bot\) | `Never` (нулевой конструктор, не имеет значений) |
| Вселенная типов \(Type_n : Type_{n+1}\) | Стратификация вселенных (защита от парадокса Рассела) |
| case-анализ | `match` на уровне типов |

> **Примечание**: `match` на уровне типов — это разбор случаев (case analysis), а не математическая индукция. Индукция требует рекурсивных функций на уровне типов + проверки завершимости компилятором.

### 0.2 Тип как пропозиция, программа как доказательство

В YaoXiang это соответствие является основополагающим принципом дизайна:

- **Завершающиеся вычисления на уровне типов соответствуют корректным конструктивным доказательствам**. Семейства типов YaoXiang (например, `Add` с case-анализом и рекурсивными вызовами на `Nat`) по сути являются закодированной на уровне типов математической индукцией — при условии, что компилятор может выполнить проверку завершимости.
- **Проверка типов — это верификация доказательства**. Когда программа проходит проверку типов, это эквивалентно конструктивному доказательству логической пропозиции.

### 0.3 Влияние на дизайн языка

Конкретные проявления соответствия Карри-Ховарда в YaoXiang:

1. **Стратификация вселенных** (RFC-010): `Type₀ : Type₁ : Type₂ …` позволяет избежать логических парадоксов (парадокс Жирара), вызванных `Type: Type`
2. **Семейства типов** (RFC-011): case-анализ и рекурсивные вызовы на уровне типов для натуральных чисел `Nat(Zero/Succ)` соответствуют аксиомам Пеано — при условии, что компилятор выполняет проверку завершимости
3. **Условные типы** (RFC-011): `If: (C: Bool, T: Type, E: Type) -> Type` соответствует case-дизъюнкции в логике
4. **Зависимые от значений типы** (RFC-011): `Vec: (n: Int) -> Type` соответствует финитной квантификации «для каждого целого n существует тип»

## Глава 1: Классификация типов

### 1.1 Выражения типов

```
TypeExpr    ::= PrimitiveType
              | RecordType
              | InterfaceType
              | TupleType
              | FnType
              | GenericType
              | TypeRef
              | TypeUnion
              | TypeIntersection
```

> **Пояснение к дизайну**: Хотя RFC-010 предлагает унифицированную модель «всё есть присваивание» (`name: type = value`), на синтаксическом уровне типы и значения всё же необходимо различать. В реализации компилятора `Type` и `Expr` — это два независимых enum-варианта AST (`ast.rs:406` и `ast.rs:25`), `TypeExpr` служит placeholder-ом в BNF, соответствующим enum `Type` в реализации и обозначающим «в этой позиции ожидается тип».

## Глава 2: Базовые типы

### 2.1 Примитивные типы
| Тип | Логическое соответствие | Описание | Размер по умолчанию |
|------|---------|------|----------|
| `Type` | — | Метатип | 0 байт |
| `Never` | ⊥ (ложь/пустой тип) | Нулевой конструктор, не имеет значений. Тип возврата для расходимости/panic. `Never <: T` для любого T. | 0 байт |
| `Void` | ⊤ (истина/Unit) | Имеет значение void по умолчанию, тип-произведение с нулём полей. `x: Void = <по умолчанию>` допустимо. | 0 байт |
| `Bool` | — | Логическое значение: `true` / `false` | 1 байт |
| `Int` | — | Знаковое целое | 8 байт |
| `Uint` | — | Беззнаковое целое | 8 байт |
| `Float` | — | Число с плавающей точкой | 8 байт |
| `String` | — | UTF-8 строка | Переменный |
| `Char` | — | Unicode-символ | 4 байта |
| `Bytes` | — | Сырые байты | Переменный |

Целые с указанием разрядности: `Int8`, `Int16`, `Int32`, `Int64`, `Int128`
Дробные с указанием разрядности: `Float32`, `Float64`

### 2.2 Never и Void: ⊥ и ⊤

`Never` и `Void` — это логические примитивы системы типов, соответствующие лжи (⊥) и истине (⊤).

**Never (⊥, ложь/пустой тип)** — три непреложных свойства:

1. **Нулевой конструктор**: Никакой литерал или выражение не может породить значение типа `Never`. Для `x: Never = ...` нет правой части, которую можно было бы написать.
2. **Принцип взрыва**: `Never <: T` для любого типа `T`. `assert(false)` возвращает `Never`, после чего последующий код проходит проверку типов (хотя никогда не будет выполнен).
3. **Маркер расходимости**: `f: (...) -> Never` означает, что `f` гарантированно не возвращается. Компилятор использует это для анализа мёртвого кода и слияния ветвей в `match`.

`Never` — это встроенное имя типа (с тем же путём регистрации, что и `Int`/`Bool`), а не ключевое слово.

**Void (⊤, истина/Unit)** — ровно один обитатель (значение void по умолчанию). `Void` — единичный элемент типа-произведения с нулём полей. `x: Void = <по умолчанию>` допустимо, функции без явного `return` возвращают `Void`.

## Глава 3: Составные типы

### 3.1 Записной тип

**Унифицированный синтаксис**: `Name: Type = { field1: Type1, field2: Type2, ... }`

```
RecordType  ::= '{' FieldList? '}'
FieldList   ::= Field (',' Field)* ','?
Field       ::= Identifier ':' TypeExpr
            |  Identifier                 // ограничение интерфейса
```

```yaoxiang
// Простой записной тип
Point: Type = { x: Float, y: Float }

// Пустой записной тип
Empty: Type = {}

// Записной тип с дженериками
Pair: (T: Type) -> Type = { first: T, second: T }

// Записной тип, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable
}
```

**Правила**:
- Записные типы определяются с помощью фигурных скобок `{}`
- После имени поля ставится двоеточие и тип
- Имя интерфейса, записанное в теле типа, означает реализацию этого интерфейса

> **Принадлежность к пространству имён**: Префикс `Type.name` (например, `Point.draw`) означает, что функция принадлежит пространству имён `Point`.
> Он не вызывает никаких неявных привязок. Чтобы синтаксис вызова через `.` (как `p.draw()`) работал, необходима явная привязка:
> `Point.draw = draw[0]`.
> Подробности см. в RFC-004 и RFC-010.

#### 3.1.1 Значения полей по умолчанию

Поля типа могут иметь значения по умолчанию; при конструировании их можно не указывать:

```yaoxiang
// Поля со значениями по умолчанию — необязательны при конструировании
Point: Type = {
    x: Float = 0,
    y: Float = 0
}

// Использование
Point()           // -> Point(x=0, y=0)
Point(x=1)       // -> Point(x=1, y=0)
Point(x=1, y=2) // -> Point(x=1, y=2)

// Поля без значений по умолчанию — обязательны при конструировании
Point2: Type = {
    x: Float,
    y: Float
}

// Использование
Point2(x=1, y=2) // корректно
Point2()          // ошибка
```

**Правила**:
- `field: Type = expression` -> есть значение по умолчанию, необязательно при конструировании
- `field: Type` -> нет значения по умолчанию, обязательно при конструировании

#### 3.1.2 Встроенные привязки

В теле определения типа можно напрямую привязывать методы:

```yaoxiang
// Способ 1: привязка ссылки на внешнюю функцию
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]    // привязка к позиции 0
}
// Вызов: p1.distance(p2) -> distance(p1, p2)

// Способ 2: анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) -> distance(p1, p2)
```

### 3.2 Интерфейсный тип

```
InterfaceType ::= '{' FnField (',' FnField)* ','?
FnField       ::= Identifier ':' FnType
FnType        ::= '(' ParamTypes? ')' '->' TypeExpr
```

**Синтаксис**: Интерфейс — это записной тип, все поля которого имеют функциональный тип

```yaoxiang
// Определение интерфейса
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой интерфейс
EmptyInterface: Type = {}
```

**Реализация интерфейса**: Тип реализует интерфейс, перечисляя его имя в конце определения

```yaoxiang
// Тип, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,        // реализация интерфейса Drawable
    Serializable     // реализация интерфейса Serializable
}
```

**Прямое присваивание интерфейсу**: Конкретный тип может быть напрямую присвоен переменной интерфейсного типа (структурная подтипизация)

```yaoxiang
// Прямое присваивание (конкретный тип известен на этапе компиляции -> вызов без накладных расходов)
d: Drawable = Circle(1)
d.draw(screen)        // после компиляции: прямой вызов circle_draw, без vtable

// Возвращаемое значение функции (неизвестно на этапе компиляции -> вызов через vtable)
d: Drawable = get_shape()
d.draw(screen)        // поиск метода через vtable

// Интерфейс как параметр функции
process: (d: Drawable) -> Void = d.draw(screen)
```

**Стратегии оптимизации на этапе компиляции**:

| Сценарий | Результат вывода | Способ вызова |
|------|----------|----------|
| Прямое присваивание конкретного типа | Конкретный тип определим | Прямой вызов (без накладных расходов) |
| Возвращаемое значение функции | Неизвестно | vtable |
| Гетерогенная коллекция | Несколько типов | vtable |

### 3.4 Кортежный тип

```
TupleType   ::= '(' TypeList? ')'
TypeList    ::= TypeExpr (',' TypeExpr)* ','?
```

### 3.5 Функциональный тип

```
FnType      ::= '(' ParamList? ')' '->' TypeExpr
ParamList   ::= TypeExpr (',' TypeExpr)*
```

## Глава 4: Дженерики

### 4.1 Синтаксис параметров дженериков

Параметры дженериков являются частью функционального типа и используют тот же синтаксис `()`, что и обычные параметры:

```
GenericType     ::= Identifier '(' TypeArgList ')'
TypeArgList     ::= TypeExpr (',' TypeExpr)* ','?
TypeBound       ::= Identifier
                 |  Identifier '+' Identifier ('+' Identifier)*
```

В определении обобщённого типа `(T: Type)` — это сигнатура параметров конструктора типов, `-> Type` обозначает возвращаемый тип:

```yaoxiang
List: (T: Type) -> Type = { ... }
Map: (K: Type, V: Type) -> Type = { ... }
```

В обобщённых функциях параметры типов также объявляются в сигнатуре, компилятор автоматически выводит их из фактических аргументов:

```yaoxiang
map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R)) = ...
```

### 4.2 Определения обобщённых типов

```yaoxiang
// Базовый обобщённый тип
Option: (T: Type) -> Type = {
    some: (T) -> Option(T),
    none: () -> Option(T)
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Result(T, E),
    err: (E) -> Result(T, E)
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — просто имя по соглашению, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T)
}
```

### 4.3 Вывод типов

```yaoxiang
// Компилятор автоматически выводит параметры дженериков
numbers: List(Int) = List(1, 2, 3)  // компилятор выводит List(Int)
```

## Глава 5: Ограничения типов

### 5.1 Единичное ограничение

```
ConstrainedType ::= '(' Identifier ':' TypeBound ')' TypeExpr
```

```yaoxiang
// Определение интерфейса (в качестве ограничения)
Clone: Type = {
    clone: () -> Clone
}

// Использование ограничения
clone: (T: Clone)(value: T) -> T = value.clone()
```

### 5.2 Множественные ограничения

```yaoxiang
// Синтаксис множественных ограничений
combine: (T: Clone + Add)(a: T, b: T) -> T = {
    a.clone() + b
}

// Сортировка обобщённого контейнера
sort: (T: Clone + PartialOrd)(list: List(T)) -> List(T) = {
    result = list.clone()
    quicksort(&mut result)
    return result
}
```

### 5.3 Ограничения функционального типа

```yaoxiang
// Ограничения для функций высшего порядка
call_twice: (T: Type, F: () -> T)(f: F) -> (T, T) = (f(), f())

compose: (A: Type, B: Type, C: Type, F: (A) -> B, G: (B) -> C)(a: A, f: F, g: G) -> C = g(f(a))
```

### 6.1 Определение ассоциированного типа

```

// Trait Iterator (с использованием синтаксиса записного типа)
Iterator: (T: Type) -> Type = {
    Item: T,                    // ассоциированный тип
    next: () -> Option(T),
    has_next: () -> Bool
}

// Использование ассоциированного типа
collect: (T: Type, I: Iterator(T))(iter: I) -> List(T) = {
    result = List(T)()
    while iter.has_next() {
        if let Some(item) = iter.next() {
            result.push(item)
        }
    }
    return result
}
```

## Глава 7: Обобщения на этапе компиляции

### 7.1 Параметры-константы времени компиляции

```
LiteralType   ::= Identifier ':' Int          // константа времени компиляции
CompileTimeFn ::= '(' Identifier ':' Int ')' '(' Identifier ')' '->' TypeExpr
```

**Ключевая идея дизайна**: Сочетание параметра-дженерика `(n: Int)` и параметра-значения `(n: n)` различает константы времени компиляции и значения времени выполнения.

```yaoxiang
// Факториал на этапе компиляции: аргумент должен быть литералом, известным на этапе компиляции
factorial: (n: Int)(n: n) -> Int = {
    match n {
        0 => 1,
        _ => n * factorial(n - 1)
    }
}

// Массив с константой времени компиляции
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),      // массив с размером, известным на этапе компиляции
    length: N
}

// Пример использования
arr: StaticArray(Int, factorial(5))  // компилятор вычисляет factorial(5) = 120 на этапе компиляции
```

## Глава 8: Условные типы

### 8.1 Условный тип If

```
IfType        ::= 'If' '(' BoolExpr ',' TypeExpr ',' TypeExpr ')'
```

```yaoxiang
// If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E
}

// Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

IsTrue: (b: Bool) -> Type = match b {

    false => Never,    # ⊥, расходимость/ошибка компиляции
}
Assert: (cond: Bool) -> Type = IsTrue(cond)
```
### 8.2 Семейства типов

```yaoxiang
// Преобразование типов на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String
}
```


### 8.3 Уточняющий тип Assert и утверждение assert

`assert` и `Assert` — это две стороны одного и того же уточняющего примитива, автоматически выбираемые конвейером диспетчеризации в зависимости от того, доступны ли свободные переменные предиката на этапе компиляции.

**Базовая сигнатура**: `assert: (cond: Bool, ?msg: String | Error) -> Assert(IsTrue(cond))`

**Правила диспетчеризации**:

| Критерий | Режим | Поведение |
|------|------|------|
| Все свободные переменные известны на этапе компиляции (параметры дженериков, константы времени компиляции) | CompileTime | Идёт в конвейер доказательств: true → стирается до Void, false → ошибка компиляции (Never необитаем) |
| Существуют свободные переменные времени выполнения (параметры функций, внешний ввод) | Runtime | Вставляется проверка Bool во время выполнения, в потокочувствительный набор гипотез Γ добавляется уточняющий факт |

**Потокочувствительный набор гипотез Γ**:

Компилятор поддерживает набор известных пропозиций для каждой точки потока управления:

```yaoxiang
assert(x > 0)       # Γ = {x > 0}
y = x + 1           # Γ = {x > 0, y > 1}  ← SP-распространение
mut x = x - 5       # Γ = {}  ← mut kill set: старые гипотезы теряют силу
```

После присваивания переменной `mut` все гипотезы, связанные с этой переменной, удаляются (kill set). При слиянии ветвей Γ берётся как пересечение гипотез всех ветвей.

## Глава 11: Свойства типов

В YaoXiang есть лишь одно свойство типов, которое нужно различать: линейность vs возможность копирования. Оно автоматически выводится компилятором.

### 11.1 Move (перенос владения по умолчанию)

Все типы по умолчанию следуют семантике Move. Присваивание, передача параметра, возврат = перенос владения.

```yaoxiang
p: Point = Point(1.0, 2.0)
q = p           // Move, p больше нельзя читать
```

### 11.2 Dup (поверхностное копирование: копия дескриптора, общие данные)

**Свойство Dup используется для ссылочных/токенных типов**. Присваивание типа Dup = поверхностное копирование — копируется дескриптор/токен, нижележащие данные являются общими. Несколько владельцев указывают на одни и те же данные.

| Тип | Свойство | Пояснение |
|------|------|------|
| `&T` | Dup | Токен чтения нулевого размера; копирование токена = несколько точек зрения на одни данные |
| `ref T` | Dup | Копия Rc/Arc = счётчик ссылок +1, общие данные в куче |
| `&mut T` | Linear | Токен записи нулевого размера, эксклюзивный, не копируется |
| Все остальные типы | Move | Перенос владения по умолчанию |

**Примитивные типы значений** (Int, Float, Bool, Char) — это специальная встроенная обработка компилятора: при присваивании значение автоматически копируется, оба значения полностью независимы. Это нативное поведение компилятора, не относящееся к свойству Dup.

```yaoxiang
// &T: Dup, можно свободно создавать псевдонимы
view: &Point = &p
view2 = view     // Dup: копия токена, оба действительны
print(view.x)    // допустимо
print(view2.x)   // допустимо

// &mut T: Linear, нельзя копировать
mut_ref: &mut Point = &mut p
// r2 = mut_ref  // ❌ &mut T не является Dup, нельзя копировать
```

### 11.3 Clone (явное глубокое копирование) и его связь с Dup

**Clone** — это интерфейс явного глубокого копирования. Любой тип может реализовать Clone, предоставляя метод `.clone()`.

```yaoxiang
// Определение интерфейса Clone (в стандартной библиотеке)
Clone: Type = {
    clone: () -> Clone
}

// Использование
p: Point = Point(1.0, 2.0)
backup = p.clone()    // глубокое копирование, p по-прежнему доступен
p2 = p.clone()        // можно клонировать многократно
```

**Различие между Dup и Clone**:

| | Dup | Clone |
|---|---|---|
| **Семантика** | Поверхностное копирование: копия дескриптора/токена, общие нижележащие данные | Глубокое копирование: создание полной независимой копии |
| **Способ вызова** | Неявно (автоматически при присваивании/передаче) | Явно (`.clone()`) |
| **Влияние модификации** | Взаимное (общие нижележащие данные) | Отсутствует (независимые копии) |
| **Применимые типы** | `&T` токены, `ref T` | Любой тип, реализующий интерфейс Clone |
| **Стоимость** | Без накладных расходов (токен — тип нулевого размера) | Зависит от типа |

**Dup не влечёт Clone, Clone не влечёт Dup** — это два ортогональных понятия:

```yaoxiang
// Тип Dup: копия токена, общие нижележащие данные
view: &Point = &p
view2 = view        // Dup: копия токена, оба указывают на один и тот же p
print(view.x)       // допустимо
print(view2.x)      // допустимо, видят одни и те же данные

// Примитивный тип значения: автоматическое копирование значения компилятором (не Dup)
x: Int = 42
y = x               // копирование значения, x и y полностью независимы
print(x)            // допустимо

// Clone: явное глубокое копирование, создание независимой копии
p: Point = Point(1.0, 2.0)
q = p.clone()       // Clone: глубокое копирование, p по-прежнему доступен
r = p               // Move: перенос владения, потому что Point не является ни Dup, ни примитивным типом значения
```

**Дизайнерский замысел**:
- Dup используется для токенов/ссылок и решает задачу «несколько точек зрения на одни и те же данные»
- Clone используется в сценариях, где нужна независимая копия; явный вызов делает стоимость видимой
- Копирование примитивных типов значений (Int/Float/Bool/Char) — встроенное поведение компилятора, не относящееся к Dup
- Большинство пользовательских типов по умолчанию Move, что обеспечивает высокую производительность без копирования

## Глава 12: Токены заимствования

### 12.1 Основные понятия

`&T` и `&mut T` — это **токенные типы нулевого размера на этапе компиляции**. Это не «ссылки», а «доказательства на уровне типов права доступа».

```
&T      →  нулевого размера, замораживает исходные данные (запрещает получение WriteToken в этот период),
          при гарантии заморозки множественное чтение безопасно → Dup (можно копировать)
&mut T  →  нулевого размера, эксклюзивное чтение-запись (запрещены любые другие токены),
          при эксклюзивном доступе копирование бессмысленно → Linear (не Dup)
```

**Ключевые особенности**:
- Токен — это **обычный тип**, подчиняющийся тем же правилам области видимости, что и все остальные типы
- Не нужны аннотации времени жизни `'a`
- Не нужен специальный анализатор заимствований — свойство типа (Dup/Linear) естественным образом определяет права доступа
- Полностью исчезают после компиляции, нулевые накладные расходы во время выполнения

### 12.2 Базовое использование

```yaoxiang
// На стороне метода: объявление типа параметра определяет требуемые права
Point.print: (self: &Point) -> Void = {
    print(self.x)               // токен &Point даёт право чтения
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx        // токен &mut Point даёт право записи
    self.y = self.y + dy
}

// На стороне вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                       // компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)               // компилятор автоматически создаёт токен &mut Point
p.print()                       // OK, предыдущий токен был освобождён по завершении вызова shift

// Сосуществование нескольких токенов &T — Dup допускает свободное копирование
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)
}
d = distance(p, p2)
```

### 12.3 Область видимости и распространение токена

Токен — это обычный тип, поэтому он поддерживает все операции, доступные обычным типам:

**Возврат токена** — токен распространяется вместе с возвращаемым значением:

```yaoxiang
// ✅ Дочерний токен и родительский токен возвращаются вместе
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()        // токен возвращается вызывающему
print(px_ref)                    // OK, токен всё ещё в области видимости
```

**Хранение в структуре** — структуры могут содержать токены в качестве полей:

```yaoxiang
// ✅ Структура содержит токен в качестве поля
Window: Type = {
    target: Point,
    view: &Point,              // поле-токен — хранит представление target только для чтения
}
```

**Захват замыканием** — замыкание захватывает токен так же, как и любое другое значение:

```yaoxiang
// ✅ Замыкание захватывает токен &Float (тип Dup, свободно копируется в замыкание)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}
```

### 12.4 Автоматический выбор заимствования

На стороне вызова компилятор автоматически выбирает в соответствии со следующим приоритетом:

```
1. Если аргумент используется далее → сначала создать токен (&T или &mut T, согласно сигнатуре метода)
2. Если аргумент далее не используется → Move
3. Порядок приоритета: &T < &mut T < Move
```

```yaoxiang
p = Point(1.0, 2.0)
p.print()          // тип параметра print — &Point → компилятор создаёт токен &Point
p.shift(1.0, 1.0)  // тип параметра shift — &mut Point → компилятор создаёт токен &mut Point
p2 = p             // далее не используется → Move
```

### 12.5 Обнаружение конфликтов токенов

Компилятор выполняет **потокочувствительный анализ живучести** для токенов, отслеживая состояние каждого токена (активен/перемещён):

```yaoxiang
// ❌ &mut и производный от него &T не могут быть одновременно активны
bad_alias: (p: &mut Point) -> Void = {
    p.x = 10.0                   // ✅ штатное использование WriteToken
    print(p.y)
}

// ✅ Токен автоматически освобождается по завершении области видимости
good_seq: (p: &mut Point) -> Void = {
    {
        // внутренняя область видимости
        print(p.x)               // использование &mut Point
    }
    // конец внутренней области видимости
    p.x = 10.0                   // ✅ WriteToken по-прежнему доступен
}

// ❌ Один и тот же аргумент не может породить одновременно токен &mut и другие токены
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)                  // ❌ p одновременно порождает токены &mut и &
```

### 12.6 Внутри компилятора: механизм брендов

Пользователь никогда не взаимодействует с брендами. Компилятор внутри назначает каждому токену уникальный идентификатор времени компиляции:

```
Видимое пользователю       Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — уникальное целое на этапе компиляции
&mut Point     →  WriteToken(Point, #M)   // #M — уникальное целое на этапе компиляции
```

Назначение брендов:
- **Защита от подделки**: Токен может быть получен только из капсулы владельца, его нельзя сконструировать из ничего
- **Отслеживание связей**: Производный от доступа к полю `&Float` несёт бренд производного (`#N.field_x`), компилятор может отследить его до родительского токена
- **Обнаружение конфликтов**: WriteToken и производный от него ReadToken из одного источника не могут быть одновременно активны

Бренды полностью исчезают после мономорфизации и инлайнинга, в генерируемом машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

### 12.7 Тип-сумма токенов

```
&BorrowToken ::= &T          // ReadToken (замораживает исходные данные → Dup безопасен)
               | &mut T      // WriteToken (эксклюзивное чтение-запись → Linear)
```

### 12.8 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Что делает | Взглянуть/изменить на месте | Разделяемое владение |
| Область действия | Следует за областью видимости значения-токена | Пересекает области видимости |
| Стоимость | Без накладных расходов (тип нулевого размера, исчезает после компиляции) | Rc или Arc (выбирает компилятор) |
| Побег | Допустим (токен распространяется через возвращаемое значение/структуру/замыкание) | Изначально предназначен для побега |
| Между задачами | Невозможно (передача токенов между задачами не реализована) | Возможно (компилятор автоматически выбирает Arc) |
| Обнаружение циклов | Не применимо | Внутри задачи — молча, между задачами — lint |
