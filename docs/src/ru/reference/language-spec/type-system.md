# Спецификация системы типов

Настоящий документ определяет спецификацию системы типов языка программирования YaoXiang, включая базовые типы, составные типы, generics и trait.

---

## Глава 0: Теоретические основы

### 0.1 Соответствие Карри — Ховарда

Соответствие Карри — Ховарда (Curry-Howard correspondence) является теоретической основой системы типов YaoXiang. Оно раскрывает глубокую связь между системами типов языков программирования и математической логикой:

| Логика | Язык программирования |
|--------|------------------------|
| Суждение \(P\) | Тип `Type` |
| Доказательство \(p: P\) | Программа `x: T = ...` |
| Импликация \(P \rightarrow Q\) | Функциональный тип `(P) -> Q` |
| Конъюнкция \(P \wedge Q\) | Тип-произведение `{ a: P, b: Q }` |
| Дизъюнкция \(P \vee Q\) | Тип-сумма `{ a(P) \| b(Q) }` |
| Универсальная квантификация \(\forall x:T. P(x)\) | Generics `(T: Type) -> ...` |
| Истина \(\top\) | Пустой тип `{}` |
| Ложь \(\bot\) | `Void` / `Never` |
| Вселенная типов \(Type_n : Type_{n+1}\) | Иерархия вселенных (защита от парадокса Рассела) |
| Математическая индукция | Типовый `match` |

### 0.2 Типы как суждения, программы как доказательства

В YaoXiang данное соответствие является основным принципом проектирования:

- **Тип — это логическое суждение**. `Int` — это суждение о существовании целого числа, `fn(a: Int, b: Int) -> Int` — это суждение о том, что по двум целым числам существует целое число.
- **Проверка типов — это верификация доказательства**. Когда программа проходит проверку типов, это эквивалентно конструктивному доказательству логического суждения.
- **Завершающиеся вычисления на уровне типов соответствуют корректной индуктивной дедукции**. Типовые семейства YaoXiang (такие как pattern matching `Add` на `Nat`) по сути являются кодированием математической индукции на уровне типов.

### 0.3 Влияние на дизайн языка

Соответствие Карри — Ховарда в YaoXiang проявляется следующим образом:

1. **Иерархия вселенных** (RFC-010): `Type₀ : Type₁ : Type₂ …` позволяет избежать логических парадоксов, вызванных `Type: Type` (парадокс Жирара)
2. **Семейства типов** (RFC-011): Типовый pattern matching натуральных чисел `Nat(Zero/Succ)` соответствует индуктивному доказательству в аксиомах Пеано
3. **Условные типы** (RFC-011): `If: (C: Bool, T: Type, E: Type) -> Type` соответствует case-дизъюнкции в логике
4. **Зависимые от значения типы** (RFC-011): `Vec: (n: Int) -> Type` соответствует конечной квантификации «для каждого целого n существует тип»

## Глава 3: Составные типы

### 3.1 Типы записей

**Единый синтаксис**: `Name: Type = { field1: Type1, field2: Type2, ... }`

```
RecordType  ::= '{' FieldList? '}'
FieldList   ::= Field (',' Field)* ','?
Field       ::= Identifier ':' TypeExpr
            |  Identifier                 // Ограничение интерфейса
```

```yaoxiang
// Простой тип записи
Point: Type = { x: Float, y: Float }

// Пустой тип записи
Empty: Type = {}

// Тип записи с generics
Pair: (T: Type) -> Type = { first: T, second: T }

// Тип записи, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable
}
```

**Правила**:
- Типы записей определяются фигурными скобками `{}`
- Имя поля непосредственно followed by двоеточием и типом
- Имя интерфейса в теле типа означает его реализацию

> **Принадлежность к пространству имён**: Префикс `Type.name` (например, `Point.draw`) означает, что функция принадлежит пространству имён `Point`.
> Это не вызывает никаких неявных связываний. Чтобы синтаксис вызова `.` (например, `p.draw()`) работал, необходимо явное связывание:
> `Point.draw = draw[0]`.
> Подробности см. в RFC-004 и RFC-010.

#### 3.1.1 Значения полей по умолчанию

Поля типа могут иметь значения по умолчанию, которые можно не указывать при конструировании:

```yaoxiang
// Поле со значением по умолчанию — необязательно при конструировании
Point: Type = {
    x: Float = 0,
    y: Float = 0
}

// Использование
Point()           // -> Point(x=0, y=0)
Point(x=1)       // -> Point(x=1, y=0)
Point(x=1, y=2) // -> Point(x=1, y=2)

// Поле без значения по умолчанию — обязательно при конструировании
Point2: Type = {
    x: Float,
    y: Float
}

// Использование
Point2(x=1, y=2) // Правильно
Point2()          // Ошибка
```

**Правила**:
- `field: Type = expression` -> есть значение по умолчанию, необязательно при конструировании
- `field: Type` -> нет значения по умолчанию, обязательно при конструировании

#### 3.1.2 Встроенные связывания

Внутри определения типа можно напрямую связывать методы:

```yaoxiang
// Способ 1: Ссылка на внешнюю функцию
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]    // Связать с позицией 0
}
// Вызов: p1.distance(p2) -> distance(p1, p2)

// Способ 2: Анонимная функция + позиционное связывание
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) -> distance(p1, p2)
```

### 3.2 Типы интерфейсов

```
InterfaceType ::= '{' FnField (',' FnField)* ','?
FnField       ::= Identifier ':' FnType
FnType        ::= '(' ParamTypes? ')' '->' TypeExpr
```

**Синтаксис**: Интерфейс — это тип записи, все поля которого являются функциональными типами

```yaoxiang
// Определение интерфейса
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой интерфейс
EmptyInterface: Type = {}
```

**Реализация интерфейса**: Тип реализует интерфейс, перечисляя имена интерфейсов в конце определения

```yaoxiang
// Тип, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,        // Реализует интерфейс Drawable
    Serializable     // Реализует интерфейс Serializable
}
```

**Прямое присваивание интерфейсу**: Конкретный тип можно напрямую присвоить переменной типа интерфейса (структурное подтипование)

```yaoxiang
// Прямое присваивание (конкретный тип определяется на compile-time -> вызов без накладных расходов)
d: Drawable = Circle(1)
d.draw(screen)        // После компиляции: прямой вызов circle_draw, без vtable

// Возврат функции (конкретный тип неизвестен на compile-time -> вызов через vtable)
d: Drawable = get_shape()
d.draw(screen)        // Поиск метода через vtable

// Интерфейс как параметр функции
process: (d: Drawable) -> Void = d.draw(screen)
```

**Стратегия оптимизации на compile-time**:

| Сценарий | Результат вывода | Способ вызова |
|----------|------------------|---------------|
| Прямое присваивание конкретного типа | Конкретный тип определён | Прямой вызов (без накладных расходов) |
| Возврат из функции | Неизвестен | vtable |
| Гетерогенная коллекция | Несколько типов | vtable |

### 3.4 Кортежные типы

```
TupleType   ::= '(' TypeList? ')'
TypeList    ::= TypeExpr (',' TypeExpr)* ','?
```

### 3.5 Функциональные типы

```
FnType      ::= '(' ParamList? ')' '->' TypeExpr
ParamList   ::= TypeExpr (',' TypeExpr)*
```

## Глава 4: Generics

### 4.1 Синтаксис параметров generics

Параметры generics являются частью функционального типа и统一使用 синтаксис `()` вместе с обычными параметрами:

```
GenericType     ::= Identifier '(' TypeArgList ')'
TypeArgList     ::= TypeExpr (',' TypeExpr)* ','?
TypeBound       ::= Identifier
                 |  Identifier '+' Identifier ('+' Identifier)*
```

В определении generic типа `(T: Type)` — это сигнатура параметров конструктора типа, а `-> Type` — возвращаемый тип:

```yaoxiang
List: (T: Type) -> Type = { ... }
Map: (K: Type, V: Type) -> Type = { ... }
```

В generic функциях параметры типа также объявляются в сигнатуре, и компилятор автоматически выводит их из фактических аргументов:

```yaoxiang
map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R)) = ...
```

### 4.2 Определение generic типов

```yaoxiang
// Базовый generic тип
Option: (T: Type) -> Type = {
    some: (T) -> Option(T),
    none: () -> Option(T)
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Result(T, E),
    err: (E) -> Result(T, E)
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — это соглашение об именовании, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T)
}
```

### 4.3 Выведение типов

```yaoxiang
// Компилятор автоматически выводит параметры generics
numbers: List(Int) = List(1, 2, 3)  // Компилятор выводит List(Int)
```

## Глава 5: Ограничения типов

### 5.1 Одиночное ограничение

```
ConstrainedType ::= '(' Identifier ':' TypeBound ')' TypeExpr
```

```yaoxiang
// Определение интерфейсного типа (как ограничение)
Clone: Type = {
    clone: () -> Clone
}

// Использование ограничения
clone: (T: Clone)(value: T) -> T = value.clone()
```

### 5.2 Множественные ограничения

```yaoxiang
// Синтаксис множественных ограничений
combine: (T: Clone + Add)(a: T, b: T) -> T = {
    a.clone() + b
}

// Сортировка generic контейнера
sort: (T: Clone + PartialOrd)(list: List(T)) -> List(T) = {
    result = list.clone()
    quicksort(&mut result)
    return result
}
```

### 5.3 Ограничения функционального типа

```yaoxiang
// Ограничения высшего порядка
call_twice: (T: Type, F: () -> T)(f: F) -> (T, T) = (f(), f())

compose: (A: Type, B: Type, C: Type, F: (A) -> B, G: (B) -> C)(a: A, f: F, g: G) -> C = g(f(a))
```

Iterator: (T: Type) -> Type = {

collect: (T: Type, I: Iterator(T))(iter: I) -> List(T) = {

## Глава 7: Compile-time generics

### 7.1 Параметры compile-time констант

```
LiteralType   ::= Identifier ':' Int          // Compile-time константа
CompileTimeFn ::= '(' Identifier ':' Int ')' '(' Identifier ')' '->' TypeExpr
```

**Основной дизайн**: Использование `(n: Int)` generic параметра + `(n: n)` значения параметра позволяет различать compile-time константы и значения времени исполнения.

```yaoxiang
// Факториал на compile-time: параметр должен быть известным литералом
factorial: (n: Int)(n: n) -> Int = {
    match n {
        0 => 1,
        _ => n * factorial(n - 1)
    }
}

// Массив compile-time констант
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),      // Массив с известным на compile-time размером
    length: N
}

// Использование
arr: StaticArray(Int, factorial(5))  // Компилятор вычисляет factorial(5) = 120 на compile-time
```

### 7.2 Массивы compile-time констант

```yaoxiang
// Использование для типов матриц
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows)
}

// Проверка размерностей на compile-time
identity_matrix: (T: Add + Zero + One, N: Int)(size: N) -> Matrix(T, N, N) = {
    // ...
}
```

// Базовая специализация: использование перегрузки функций (компилятор выбирает автоматически)
sum: (arr: Array(Int)) -> Int = {

    return result
}

## Глава 11: Атрибуты типов

YaoXiang различает только один атрибут типов: линейный vs копируемый. Выводится компилятором автоматически.

### 11.1 Move (передача ownership по умолчанию)

Все типы по умолчанию следуют семантике Move. Присваивание, передача параметров, возврат = передача ownership.

```yaoxiang
p: Point = Point(1.0, 2.0)
q = p           // Move, p больше нельзя читать
```

### 11.2 Dup (неглубокое копирование: копирование дескриптора, общие данные)

**Атрибут Dup используется для типов ссылок/токенов**. Присваивание типа Dup = неглубокое копирование — копирование дескриптора/токена, общие базовые данные. Несколько владельцев указывают на одни и те же данные.

| Тип | Атрибут | Описание |
|------|---------|----------|
| `&T` | Dup | Токен чтения нулевого размера, копирование токена = несколько представлений одних данных |
| `ref T` | Dup | Rc/Arc копирование = +1 к счётчику ссылок, общие кучевые данные |
| `&mut T` | Linear | Токен записи нулевого размера, эксклюзивный, не копируемый |
| Все остальные типы | Move | Передача ownership по умолчанию |

**Примитивные типы значений** (Int, Float, Bool, Char) имеют специальную встроенную обработку компилятором: присваивание автоматически выполняет копирование значения, два значения полностью независимы. Это встроенное поведение компилятора, не относящееся к атрибуту типа Dup.

```yaoxiang
// &T: Dup, можно свободно создавать псевдонимы
view: &Point = &p
view2 = view     // Dup: копирование токена, оба действительны
print(view.x)    // Можно использовать
print(view2.x)   // Можно использовать

// &mut T: Linear, не копируемый
mut_ref: &mut Point = &mut p
// r2 = mut_ref  // ❌ &mut T не является Dup, нельзя копировать
```

### 11.3 Clone (явное глубокое копирование) и связь с Dup

**Clone** — это явное глубокое копирование интерфейса. Любой тип может реализовать Clone, предоставляя метод `.clone()`.

```yaoxiang
// Определение интерфейса Clone (стандартная библиотека)
Clone: Type = {
    clone: () -> Clone
}

// Использование
p: Point = Point(1.0, 2.0)
backup = p.clone()    // Глубокое копирование, p всё ещё можно использовать
p2 = p.clone()        // Можно клонировать несколько раз
```

**Разница между Dup и Clone**:

| | Dup | Clone |
|---|---|---|
| **Семантика** | Неглубокое копирование: копирование дескриптора/токена, общие базовые данные | Глубокое копирование: создание полностью независимой копии |
| **Способ вызова** | Неявный (автоматический при присваивании/передаче параметров) | Явный (`.clone()`) |
| **Влияние изменений** | Взаимное влияние (общие базовые данные) | Независимость (независимые копии) |
| **Применимые типы** | Токены `&T`, `ref T` | Любой тип, реализующий интерфейс Clone |
| **Стоимость** | Нулевые накладные расходы (токен — тип нулевого размера) | Зависит от типа |

**Dup не влечёт Clone, Clone не влечёт Dup** — это два ортогональных понятия:

```yaoxiang
// Тип Dup: копирование токена, общие базовые данные
view: &Point = &p
view2 = view        // Dup: копирование токена, оба указывают на один p
print(view.x)       // Можно использовать
print(view2.x)      // Можно использовать, видим те же данные

// Примитивные типы значений: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x               // Копирование значения, x и y полностью независимы
print(x)            // Можно использовать

// Clone: явное глубокое копирование, создание независимой копии
p: Point = Point(1.0, 2.0)
q = p.clone()       // Clone: глубокое копирование, p всё ещё можно использовать
r = p               // Move: передача ownership, так как Point не является Dup и не является примитивным типом
```

**Замысел дизайна**:
- Dup используется для типов токенов/ссылок, решая проблему «нескольких представлений одних данных»
- Clone используется для сценариев, требующих независимых копий, явный вызов делает стоимость видимой
- Примитивные типы значений (Int/Float/Bool/Char) имеют встроенное в компилятор поведение копирования, не относящееся к Dup
- Большинство пользовательских типов по умолчанию имеют Move, обеспечивая высокую производительность без копирования

## Глава 12: Типы заимствования токенов

### 12.1 Основные концепции

`&T` и `&mut T` — это **токены компиляционного времени нулевого размера**. Это не «ссылки», а «типовые доказательства прав доступа».

```
&T      →  Нулевой размер, Dup (копируемый), предоставляет права только на чтение
&mut T  →  Нулевой размер, Linear (не Dup), предоставляет эксклюзивные права на чтение/запись
```

**Ключевые характеристики**:
- Токены — это **обычные типы**, следующие тем же правилам области видимости, что и все остальные типы
- Не требуют аннотаций времени жизни `'a`
- Не требуют специального borrow checker — атрибуты типа (Dup/Linear) естественно выводят права
- Полностью исчезают после компиляции, нулевые накладные расходы во время исполнения

### 12.2 Базовое использование

```yaoxiang
// Сторона метода: объявление типа параметра, определяющее необходимые права
Point.print: (self: &Point) -> Void = {
    print(self.x)               // Токен &Point предоставляет права на чтение
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx        // Токен &mut Point предоставляет права на запись
    self.y = self.y + dy
}

// Сторона вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                       // Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)               // Компилятор автоматически создаёт токен &mut Point
p.print()                       // OK, предыдущий токен был освобождён после вызова shift

// Сосуществование нескольких токенов &T — тип Dup позволяет свободно копировать
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)
}
d = distance(p, p2)
```

### 12.3 Область видимости и распространение токенов

Токены — обычные типы, поэтому поддерживают все операции обычных типов:

**Возврат токенов** — токены распространяются вместе с возвращаемым значением:

```yaoxiang
// ✅ Дочерний и родительский токены возвращаются вместе
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()        // Токен возвращается вызывающему
print(px_ref)                    // OK, токен всё ещё в области видимости
```

**Хранение в структуре** — структуры могут содержать поля-токены:

```yaoxiang
// ✅ Структура содержит токен как поле
Window: Type = {
    target: Point,
    view: &Point,              // Поле-токен — держит только читаемое представление target
}
```

**Захват в замыканиях** — замыкания захватывают токены как любые значения:

```yaoxiang
// ✅ Замыкание захватывает токен &Float (тип Dup, свободно копируется в замыкание)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}
```

### 12.4 Автоматический выбор заимствования

Компилятор на стороне вызова автоматически выбирает по следующему приоритету:

```
1. Если фактический аргумент ещё будет использоваться → создаётся токен (&T или &mut T, в зависимости от сигнатуры метода)
2. Если фактический аргумент больше не используется → Move
3. Приоритет匹配: &T < &mut T < Move
```

```yaoxiang
p = Point(1.0, 2.0)
p.print()          // Параметр print имеет тип &Point → компилятор создаёт токен &Point
p.shift(1.0, 1.0)  // Параметр shift имеет тип &mut Point → компилятор создаёт токен &mut Point
p2 = p             // Больше не используется → Move
```

### 12.5 Обнаружение конфликтов токенов

Компилятор выполняет **потокочувствительный анализ активности** значений токенов, отслеживая состояние каждого токена (активен/перемещён):

```yaoxiang
// ❌ &mut и производные &T не могут быть активны одновременно
bad_alias: (p: &mut Point) -> Void = {
    p.x = 10.0                   // ✅ Нормальное использование WriteToken
    print(p.y)
}

// ✅ После окончания области видимости токена он автоматически освобождается
good_seq: (p: &mut Point) -> Void = {
    {
        // Внутренняя область видимости
        print(p.x)               // Используется &mut Point
    }
    // Конец внутренней области видимости
    p.x = 10.0                   // ✅ WriteToken всё ещё доступен
}

// ❌ Для одного фактического аргумента нельзя одновременно создавать токен &mut и другие токены
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)                  // ❌ p одновременно получает &mut и & токены
```

### 12.6 Внутреннее устройство компилятора: механизм брендов

Пользователь никогда не сталкивается с брендами. Компилятор внутренне назначает каждому токену компиляционно-уникальный идентификатор:

```
Видимое пользователю       Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — компиляционно-уникальное целое
&mut Point     →  WriteToken(Point, #M)   // #M — компиляционно-уникальное целое
```

Назначение брендов:
- **Защита от подделки**: Токен можно получить только от владеющего capsule, создать из ничего нельзя
- **Отслеживание связей**: Производные от доступа к полям `&Float` несут производный бренд (`#N.field_x`), компилятор может отследить до родительского токена
- **Обнаружение конфликтов**: Токены WriteToken и производные ReadToken из одного источника не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнинга, в сгенерированном машинном коде их нет. **Нулевые накладные расходы во время исполнения.**

### 12.7 Типы Sum для токенов

```
&BorrowToken ::= &T          // ReadToken (Dup, копируемый)
               | &mut T      // WriteToken (Linear, эксклюзивный)
```

### 12.8 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Что делает | Взглянуть/изменить на месте | Совместное владение |
| Область действия | Вместе с областью видимости токена | За пределами области видимости |
| Стоимость | Нулевые накладные расходы (тип нулевого размера, исчезает после компиляции) | Rc или Arc (выбирает компилятор) |
| Эскапирование | Возможно (токен распространяется с возвратом/структурой/замыканием) | Предназначен для эскапирования |
| Межзадачность | Невозможно (токен не реализует Send) | Возможно (компилятор автоматически выбирает Arc) |
| Обнаружение циклов | Не применяется | Внутри задачи — тихо, между задачами — lint |
