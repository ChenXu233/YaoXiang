# Спецификация системы типов

Этот документ определяет спецификацию системы типов языка программирования YaoXiang, включая базовые типы, составные типы, обобщённые типы (дженерики) и трейты (trait).

---

## Глава 0: Теоретические основы

### 0.1 Соответствие Карри-Ховарда

Соответствие Карри-Ховарда (Curry-Howard correspondence) является теоретическим фундаментом системы типов YaoXiang. Оно раскрывает глубокую связь между системой типов языка программирования и математической логикой:

| Логика | Язык программирования |
|--------|----------|
| Пропозиция \(P\) | Тип `Type` |
| Доказательство \(p: P\) | Программа `x: T = ...` |
| Импликация \(P \rightarrow Q\) | Тип функции `(P) -> Q` |
| Конъюнкция \(P \wedge Q\) | Тип произведения `{ a: P, b: Q }` |
| Дизъюнкция \(P \vee Q\) | Тип суммы `{ a(P) \| b(Q) }` |
| Квантор всеобщности \(\forall x:T. P(x)\) | Дженерики `(T: Type) -> ...` |
| Истина \(\top\) | Пустой тип `{}` |
| Ложь \(\bot\) | `Void` / `Never` |
| Вселенная типов \(Type_n : Type_{n+1}\) | Иерархия вселенных (защита от парадокса Рассела) |
| Математическая индукция | Сопоставление с образцом на уровне типов |

### 0.2 Тип есть пропозиция, программа есть доказательство

В YaoXiang это соответствие является первостепенным принципом проектирования:

- **Тип — это логическая пропозиция**. `Int` — это пропозиция «существование целого числа», `fn(a: Int, b: Int) -> Int` — это пропозиция «даны два целых числа, существует целое число».
- **Проверка типов есть проверка доказательства**. Когда программа проходит проверку типов, это эквивалентно конструктивному доказательству логической пропозиции.
- **Терминирующее вычисление на уровне типов соответствует корректному индуктивному рассуждению**. Семейства типов (type family) YaoXiang (например, `Add` на `Nat` с сопоставлением с образцом) по сути являются кодированием математической индукции на уровне типов.

### 0.3 Влияние на проектирование языка

Конкретные проявления соответствия Карри-Ховарда в YaoXiang:

1. **Иерархия вселенных** (RFC-010): `Type₀ : Type₁ : Type₂ …` — во избежание логических парадоксов (парадокс Жирара), вызванных `Type: Type`
2. **Семейства типов** (RFC-011): сопоставление с образцом на уровне типов для натуральных чисел `Nat(Zero/Succ)` соответствует индуктивному доказательству по аксиомам Пеано
3. **Условные типы** (RFC-011): `If: (C: Bool, T: Type, E: Type) -> Type` соответствует логической дизъюнкции case
4. **Зависимые от значений типы** (RFC-011): `Vec: (n: Int) -> Type` соответствует ограниченной квантификации «для каждого целого n существует тип»

## Глава 1: Классификация типов

### 1.1 Выражения типов

```
TypeExpr    ::= PrimitiveType
              | RecordType
              | InterfaceType
              | TupleType
              | FnType
              | GenericType
              | TypeRef
              | TypeUnion
              | TypeIntersection
```

> **Пояснение к проектированию**: хотя в RFC-010 предложена унифицированная модель «всё есть присваивание» (`name: type = value`), на синтаксическом уровне типы и значения всё же необходимо различать. В реализации компилятора `Type` и `Expr` — это два независимых перечисления AST (`ast.rs:406` и `ast.rs:25`); `TypeExpr` в качестве BNF-заполнителя соответствует перечислению `Type` в реализации и обозначает «в этой позиции ожидается тип».

## Глава 3: Составные типы

### 3.1 Записи (record type)

**Унифицированный синтаксис**: `Name: Type = { field1: Type1, field2: Type2, ... }`

```
RecordType  ::= '{' FieldList? '}'
FieldList   ::= Field (',' Field)* ','?
Field       ::= Identifier ':' TypeExpr
            |  Identifier                 // ограничение интерфейса
```

```yaoxiang
// Простой тип записи
Point: Type = { x: Float, y: Float }

// Пустой тип записи
Empty: Type = {}

// Тип записи с дженериками
Pair: (T: Type) -> Type = { first: T, second: T }

// Тип записи, реализующий интерфейс
Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable
}
```

**Правила**:
- Типы записей определяются с помощью фигурных скобок `{}`
- После имени поля сразу указывается двоеточие и тип
- Имя интерфейса в теле типа означает реализацию этого интерфейса

> **Принадлежность пространству имён**: префикс `Type.name` (например, `Point.draw`) указывает, что функция принадлежит пространству имён `Point`.
> Он не вызывает никаких неявных привязок. Чтобы синтаксис вызова через `.` (например, `p.draw()`) работал, необходима явная привязка:
> `Point.draw = draw[0]`.
> Подробности см. в RFC-004 и RFC-010.

#### 3.1.1 Значения полей по умолчанию

Поля типов могут иметь значения по умолчанию; при конструировании их можно не указывать:

```yaoxiang
// Поле со значением по умолчанию - при конструировании можно опустить
Point: Type = {
    x: Float = 0,
    y: Float = 0
}

// Использование
Point()           // -> Point(x=0, y=0)
Point(x=1)       // -> Point(x=1, y=0)
Point(x=1, y=2) // -> Point(x=1, y=2)

// Поле без значения по умолчанию - при конструировании обязательно
Point2: Type = {
    x: Float,
    y: Float
}

// Использование
Point2(x=1, y=2) // корректно
Point2()          // ошибка
```

**Правила**:
- `field: Type = expression` -> есть значение по умолчанию, при конструировании можно опустить
- `field: Type` -> нет значения по умолчанию, при конструировании обязательно

#### 3.1.2 Встроенные привязки

В теле определения типа можно напрямую привязывать методы:

```yaoxiang
// Способ 1: привязка через ссылку на внешнюю функцию
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]    // привязка к позиции 0
}
// Вызов: p1.distance(p2) -> distance(p1, p2)

// Способ 2: анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) -> distance(p1, p2)
```

### 3.2 Интерфейсы (interface type)

```
InterfaceType ::= '{' FnField (',' FnField)* ','?
FnField       ::= Identifier ':' FnType
FnType        ::= '(' ParamTypes? ')' '->' TypeExpr
```

**Синтаксис**: интерфейс — это тип записи, все поля которой имеют функциональный тип

```yaoxiang
// Определение интерфейса
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой интерфейс
EmptyInterface: Type = {}
```

**Реализация интерфейса**: тип реализует интерфейс, перечисляя его имя в конце определения

```yaoxiang
// Тип, реализующий интерфейс
Point: Type = {
    x: Float,
    y: Float,
    Drawable,        // реализация интерфейса Drawable
    Serializable     // реализация интерфейса Serializable
}
```

**Прямое присваивание интерфейсу**: конкретный тип может быть напрямую присвоен переменной интерфейсного типа (структурная подтипизация)

```yaoxiang
// Прямое присваивание (конкретный тип известен в compile-time -> вызов без накладных расходов)
d: Drawable = Circle(1)
d.draw(screen)        // после компиляции: прямой вызов circle_draw, без vtable

// Возвращаемое значение функции (неизвестно в compile-time -> вызов через vtable)
d: Drawable = get_shape()
d.draw(screen)        // поиск метода через vtable

// Интерфейс как параметр функции
process: (d: Drawable) -> Void = d.draw(screen)
```

**Стратегии оптимизации в compile-time**:

| Сценарий | Результат вывода | Способ вызова |
|------|----------|----------|
| Прямое присваивание конкретного типа | Конкретный тип определим | Прямой вызов (нулевые накладные расходы) |
| Возвращаемое значение функции | Неизвестно | vtable |
| Гетерогенная коллекция | Несколько типов | vtable |

### 3.4 Кортежи (tuple type)

```
TupleType   ::= '(' TypeList? ')'
TypeList    ::= TypeExpr (',' TypeExpr)* ','?
```

### 3.5 Функциональные типы (function type)

```
FnType      ::= '(' ParamList? ')' '->' TypeExpr
ParamList   ::= TypeExpr (',' TypeExpr)*
```

## Глава 4: Дженерики (generics)

### 4.1 Синтаксис параметров дженериков

Параметры дженериков являются частью функционального типа и используют единый синтаксис `()`:

```
GenericType     ::= Identifier '(' TypeArgList ')'
TypeArgList     ::= TypeExpr (',' TypeExpr)* ','?
TypeBound       ::= Identifier
                 |  Identifier '+' Identifier ('+' Identifier)*
```

В определении обобщённого типа `(T: Type)` — это сигнатура параметра конструктора типов, а `-> Type` обозначает возвращаемый тип:

```yaoxiang
List: (T: Type) -> Type = { ... }
Map: (K: Type, V: Type) -> Type = { ... }
```

В обобщённой функции параметры типов также объявляются в сигнатуре; компилятор автоматически выводит их из фактических аргументов:

```yaoxiang
map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R)) = ...
```

### 4.2 Определение обобщённого типа

```yaoxiang
// Базовый обобщённый тип
Option: (T: Type) -> Type = {
    some: (T) -> Option(T),
    none: () -> Option(T)
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Result(T, E),
    err: (E) -> Result(T, E)
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self - просто имя по соглашению, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T)
}
```

### 4.3 Вывод типов (type inference)

```yaoxiang
// Компилятор автоматически выводит параметры дженериков
numbers: List(Int) = List(1, 2, 3)  // компилятор выводит List(Int)
```

## Глава 5: Ограничения типов (type constraint)

### 5.1 Одиночное ограничение

```
ConstrainedType ::= '(' Identifier ':' TypeBound ')' TypeExpr
```

```yaoxiang
// Определение интерфейса (в качестве ограничения)
Clone: Type = {
    clone: () -> Clone
}

// Использование ограничения
clone: (T: Clone)(value: T) -> T = value.clone()
```

### 5.2 Множественные ограничения (multiple constraint)

```yaoxiang
// Синтаксис множественных ограничений
combine: (T: Clone + Add)(a: T, b: T) -> T = {
    a.clone() + b
}

// Сортировка обобщённой коллекции
sort: (T: Clone + PartialOrd)(list: List(T)) -> List(T) = {
    result = list.clone()
    quicksort(&mut result)
    return result
}
```

### 5.3 Ограничения на функциональные типы

```yaoxiang
// Ограничения для функций высшего порядка
call_twice: (T: Type, F: () -> T)(f: F) -> (T, T) = (f(), f())

compose: (A: Type, B: Type, C: Type, F: (A) -> B, G: (B) -> C)(a: A, f: F, g: G) -> C = g(f(a))
```

## Глава 6: Ассоциированные типы (associated type)

### 6.1 Определение ассоциированного типа

```
AssociatedType ::= Identifier ':' TypeExpr
```

```yaoxiang
// Trait Iterator (с использованием синтаксиса типа записи)
Iterator: (T: Type) -> Type = {
    Item: T,                    // ассоциированный тип
    next: () -> Option(T),
    has_next: () -> Bool
}

// Использование ассоциированного типа
collect: (T: Type, I: Iterator(T))(iter: I) -> List(T) = {
    result = List(T)()
    while iter.has_next() {
        if let Some(item) = iter.next() {
            result.push(item)
        }
    }
    return result
}
```

```yaoxiang
    IteratorType: Iterator(T),  // ассоциированный тип тоже обобщённый
    iter: () -> IteratorType
}
```

## Глава 7: Дженерики на этапе компиляции (compile-time)

### 7.1 Параметры-константы этапа компиляции

```
LiteralType   ::= Identifier ':' Int          // константа этапа компиляции
CompileTimeFn ::= '(' Identifier ':' Int ')' '(' Identifier ')' '->' TypeExpr
```

**Ключевая идея проектирования**: параметр дженерика `(n: Int)` + параметр значения `(n: n)` различают константу этапа компиляции и значение этапа выполнения.

```yaoxiang
// Факториал на этапе компиляции: аргумент должен быть литералом, известным в compile-time
factorial: (n: Int)(n: n) -> Int = {
    match n {
        0 => 1,
        _ => n * factorial(n - 1)
    }
}

// Массив с константой этапа компиляции
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),      // массив с размером, известным в compile-time
    length: N
}

// Пример использования
arr: StaticArray(Int, factorial(5))  // компилятор вычисляет factorial(5) = 120 на этапе компиляции
```

### 7.2 Массивы с константами этапа компиляции

```yaoxiang
// Использование типа матрицы
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows)
}

// Проверка измерений на этапе компиляции
identity_matrix: (T: Add + Zero + One, N: Int)(size: N) -> Matrix(T, N, N) = {
    // ...
}
```

## Глава 10: Перегрузка и специализация функций

### 10.1 Перегрузка функций

```yaoxiang
// Базовая специализация: использование перегрузки функций (компилятор выбирает автоматически)
sum: (arr: Array(Int)) -> Int = {
    return native_sum_int(arr.data, arr.length)
}

sum: (arr: Array(Float)) -> Float = {
    return simd_sum_float(arr.data, arr.length)
}

// Универсальная реализация
sum: (T: Add)(arr: Array(T)) -> T = {
    result = Zero::zero()
    for item in arr {
        result = result + item
    }
    return result
}
```

```yaoxiang
// Перечисление типов платформ (определение стандартной библиотеки)

sum: (P: X86_64)(arr: Array(Float)) -> Float = {
    return avx2_sum(arr.data, arr.length)
```

## Глава 11: Атрибуты типов

В YaoXiang есть лишь один атрибут типа, который необходимо различать: линейный vs копируемый. Он выводится компилятором автоматически.

### 11.1 Move (перенос владения по умолчанию)

Все типы по умолчанию следуют семантике Move. Присваивание, передача параметров, возврат = перенос владения.

```yaoxiang
p: Point = Point(1.0, 2.0)
q = p           // Move, p больше нельзя читать
```

### 11.2 Dup (поверхностное копирование: копирование дескриптора, общие данные)

**Атрибут Dup предназначен для типов ссылок/токенов**. Присваивание Dup-типа = поверхностное копирование — копируется дескриптор/токен, а нижележащие данные являются общими. Несколько владельцев указывают на один и тот же блок данных.

| Тип | Атрибут | Пояснение |
|------|------|------|
| `&T` | Dup | Токен чтения нулевого размера; копирование токена = несколько точек зрения на одни данные |
| `ref T` | Dup | Копирование Rc/Arc = счётчик ссылок +1, общие данные в куче |
| `&mut T` | Linear | Токен записи нулевого размера, эксклюзивный, не копируется |
| Все остальные типы | Move | Перенос владения по умолчанию |

**Примитивные типы значений** (Int, Float, Bool, Char) обрабатываются компилятором особым образом: при присваивании автоматически выполняется копирование значения, и оба значения полностью независимы. Это встроенное поведение компилятора, а не атрибут Dup-типа.

```yaoxiang
// &T: Dup, свободное алиасинг
view: &Point = &p
view2 = view     // Dup: копирование токена, оба валидны
print(view.x)    // OK
print(view2.x)   // OK

// &mut T: Linear, не копируется
mut_ref: &mut Point = &mut p
// r2 = mut_ref  // ❌ &mut T не Dup, копировать нельзя
```

### 11.3 Clone (явное глубокое копирование) и его отношение к Dup

**Clone** — это интерфейс явного глубокого копирования. Любой тип может реализовать Clone, предоставив метод `.clone()`.

```yaoxiang
// Определение интерфейса Clone (стандартная библиотека)
Clone: Type = {
    clone: () -> Clone
}

// Использование
p: Point = Point(1.0, 2.0)
backup = p.clone()    // глубокое копирование, p остаётся доступным
p2 = p.clone()        // можно клонировать многократно
```

**Различие между Dup и Clone**:

| | Dup | Clone |
|---|---|---|
| **Семантика** | Поверхностное копирование: копируется дескриптор/токен, данные общие | Глубокое копирование: создаётся полная независимая копия |
| **Способ вызова** | Неявный (автоматически при присваивании/передаче) | Явный (`.clone()`) |
| **Влияние изменений** | Влияют друг на друга (общие данные) | Не влияют друг на друга (независимые копии) |
| **Применимые типы** | Токены `&T`, `ref T` | Любой тип, реализующий интерфейс Clone |
| **Стоимость** | Нулевые накладные расходы (токен — тип нулевого размера) | Зависит от типа |

**Dup не влечёт Clone, Clone не влечёт Dup** — это два ортогональных понятия:

```yaoxiang
// Dup-тип: копируется токен, нижележащие данные общие
view: &Point = &p
view2 = view        // Dup: копирование токена, оба указывают на один p
print(view.x)       // OK
print(view2.x)      // OK, видят те же данные

// Примитивный тип значения: автоматическое копирование значения компилятором (не Dup)
x: Int = 42
y = x               // копирование значения, x и y полностью независимы
print(x)            // OK

// Clone: явное глубокое копирование, создание независимой копии
p: Point = Point(1.0, 2.0)
q = p.clone()       // Clone: глубокое копирование, p остаётся доступным
r = p               // Move: перенос владения, так как Point не Dup и не примитивный тип значения
```

**Замысел проектирования**:
- Dup предназначен для токенов/ссылочных типов и решает задачу «несколько точек зрения на одни данные»
- Clone предназначен для сценариев, где требуется независимая копия; явный вызов делает стоимость видимой
- Копирование примитивных типов значений (Int/Float/Bool/Char) — встроенное поведение компилятора, а не Dup
- Большинство пользовательских типов по умолчанию Move, обеспечивая высокую производительность без копирования

## Глава 12: Токены заимствования

### 12.1 Ключевые концепции

`&T` и `&mut T` — это **токены этапа компиляции нулевого размера**. Это не «ссылки», а «доказательства на уровне типов права доступа».

```
&T      →  нулевого размера, Dup (копируемый), предоставляет право чтения
&mut T  →  нулевого размера, Linear (не Dup), предоставляет эксклюзивное право чтения и записи
```

**Ключевые свойства**:
- Токен — это **обычный тип**, подчиняющийся тем же правилам области видимости, что и все остальные типы
- Аннотации времени жизни `'a` не нужны
- Специализированный анализатор заимствований не нужен — атрибуты типов (Dup/Linear) естественно выводят права
- Полностью исчезают после компиляции, нулевые накладные расходы во время выполнения

### 12.2 Базовое использование

```yaoxiang
// На стороне метода: объявление типа параметра определяет требуемые права
Point.print: (self: &Point) -> Void = {
    print(self.x)               // токен &Point даёт право чтения
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx        // токен &mut Point даёт право записи
    self.y = self.y + dy
}

// На стороне вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                       // компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)               // компилятор автоматически создаёт токен &mut Point
p.print()                       // OK, предыдущий токен был освобождён по завершении shift

// Несколько токенов &T сосуществуют — Dup-тип допускает свободное копирование
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)
}
d = distance(p, p2)
```

### 12.3 Область видимости и распространение токенов

Токен — это обычный тип, поэтому он поддерживает все обычные операции над типами:

**Возврат токена** — токен распространяется вместе с возвращаемым значением:

```yaoxiang
// ✅ Дочерний токен возвращается вместе с родительским
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()        // токен возвращается вызывающему
print(px_ref)                    // OK, токен всё ещё в области видимости
```

**Хранение в структуре** — структура может содержать поле-токен:

```yaoxiang
// ✅ Структура несёт токен в качестве поля
Window: Type = {
    target: Point,
    view: &Point,              // поле-токен — владеет представлением target только для чтения
}
```

**Захват замыканием** — замыкание захватывает токен точно так же, как любое значение:

```yaoxiang
// ✅ Замыкание захватывает токен &Float (Dup-тип, свободно копируется в замыкание)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}
```

### 12.4 Автоматический выбор заимствования

На стороне вызова компилятор по следующему приоритету автоматически выбирает:

```
1. Если аргумент будет использован далее → приоритетно создаётся токен (&T или &mut T, согласно сигнатуре метода)
2. Если аргумент далее не используется → Move
3. Порядок приоритета при сопоставлении: &T < &mut T < Move
```

```yaoxiang
p = Point(1.0, 2.0)
p.print()          // тип параметра print — &Point → компилятор создаёт токен &Point
p.shift(1.0, 1.0)  // тип параметра shift — &mut Point → компилятор создаёт токен &mut Point
p2 = p             // далее не используется → Move
```

### 12.5 Обнаружение конфликтов токенов

Компилятор выполняет **чувствительный к потоку управления анализ живучести** (liveness) для токенов, отслеживая состояние каждого токена (активен/перемещён):

```yaoxiang
// ❌ &mut и производный &T не могут быть активны одновременно
bad_alias: (p: &mut Point) -> Void = {
    p.x = 10.0                   // ✅ штатное использование WriteToken
    print(p.y)
}

// ✅ После завершения области видимости токен автоматически освобождается
good_seq: (p: &mut Point) -> Void = {
    {
        // внутренняя область видимости
        print(p.x)               // использование &mut Point
    }
    // конец внутренней области видимости
    p.x = 10.0                   // ✅ WriteToken всё ещё доступен
}

// ❌ Из одного аргумента нельзя одновременно создать токен &mut и другие токены
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)                  // ❌ из p одновременно порождаются токены &mut и &
```

### 12.6 Внутреннее устройство компилятора: механизм брендов

Пользователь никогда не сталкивается с брендами. Компилятор внутри назначает каждому токену уникальный compile-time идентификатор:

```
Видимое пользователю       Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — уникальное целое этапа компиляции
&mut Point     →  WriteToken(Point, #M)   // #M — уникальное целое этапа компиляции
```

Назначение брендов:
- **Защита от подделки**: токен может быть получен только из капсулы владельца, не может быть сконструирован «из воздуха»
- **Отслеживание связей**: производный от доступа к полю `&Float` несёт производный бренд (`#N.field_x`), что позволяет компилятору отследить его до родительского токена
- **Обнаружение конфликтов**: WriteToken и производный от него ReadToken одного происхождения не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнинга; в генерируемом машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

### 12.7 Тип-сумма токенов заимствования

```
&BorrowToken ::= &T          // ReadToken (Dup, копируемый)
               | &mut T      // WriteToken (Linear, эксклюзивный)
```

### 12.8 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Назначение | Взглянуть/изменить на месте | Совместное владение |
| Область видимости | С областью видимости значения токена | Между областями видимости |
| Стоимость | Нулевые накладные расходы (тип нулевого размера, исчезает после компиляции) | Rc или Arc (выбирается компилятором) |
| Побег (escape) | Допустим (токен распространяется через возвращаемое значение/структуру/замыкание) | Изначально предназначен для побега |
| Между задачами | Невозможно (токен не поддерживает передачу между задачами) | Возможно (компилятор автоматически выбирает Arc) |
| Обнаружение циклов | Не применимо | Молча внутри задачи, lint между задачами |
