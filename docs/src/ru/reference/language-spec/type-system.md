# Спецификация системы типов

Настоящий документ определяет спецификацию системы типов языка программирования YaoXiang, включая базовые типы, составные типы, дженерики и trait.

---

## Нулевая глава: Теоретические основы

### 0.1 Соответствие Карри-Ховарда

Соответствие Карри-Ховарда (Curry-Howard correspondence) является теоретической основой системы типов YaoXiang. Оно раскрывает глубокую связь между системой типов языка программирования и математической логикой:

| Логика | Язык программирования |
|--------|----------|
| Высказывание \(P\) | тип `Type` |
| Доказательство \(p: P\) | программа `x: T = ...` |
| Импликация \(P \rightarrow Q\) | функциональный тип `(P) -> Q` |
| Конъюнкция \(P \wedge Q\) | тип-произведение `{ a: P, b: Q }` |
| Дизъюнкция \(P \vee Q\) | тип-сумма `{ a(P) \| b(Q) }` |
| Квантор всеобщности \(\forall x:T. P(x)\) | дженерики `(T: Type) -> ...` |
| Истина \(\top\) | пустой тип `{}` |
| Ложь \(\bot\) | `Void` / `Never` |
| Вселенная типов \(Type_n : Type_{n+1}\) | иерархия вселенных (для защиты от парадокса Рассела) |
| Математическая индукция | `match` на уровне типов |

### 0.2 Тип как высказывание, программа как доказательство

В YaoXiang это соответствие является основополагающим принципом проектирования:

- **Тип — это логическое высказывание**. `Int` — это высказывание «существует целое число», `fn(a: Int, b: Int) -> Int` — высказывание «даны два целых числа, существует целое число».
- **Проверка типов — это верификация доказательства**. Когда программа проходит проверку типов, это эквивалентно конструктивному доказательству логического высказывания.
- **Завершающиеся вычисления на уровне типов соответствуют корректному индуктивному рассуждению**. Семейства типов YaoXiang (например, сопоставление с образцом `Add` на `Nat`) по сути представляют собой кодирование математической индукции на уровне типов.

### 0.3 Влияние на проектирование языка

Конкретное воплощение соответствия Карри-Ховарда в YaoXiang:

1. **Иерархия вселенных** (RFC-010): `Type₀ : Type₁ : Type₂ …` позволяет избежать логического парадокса, вызванного `Type: Type` (парадокс Жирара)
2. **Семейства типов** (RFC-011): сопоставление с образцом на уровне типов для натуральных чисел `Nat(Zero/Succ)` соответствует индуктивному доказательству в аксиоматике Пеано
3. **Условные типы** (RFC-011): `If: (C: Bool, T: Type, E: Type) -> Type` соответствует case-анализу в логике
4. **Зависимые от значений типы** (RFC-011): `Vec: (n: Int) -> Type` соответствует ограниченной квантификации «для каждого целого числа n существует тип»

## Глава первая: Классификация типов

### 1.1 Выражения типов

```
TypeExpr    ::= PrimitiveType
              | RecordType
              | InterfaceType
              | TupleType
              | FnType
              | GenericType
              | TypeRef
              | TypeUnion
              | TypeIntersection
```

> **Пояснение по проектированию**: хотя RFC-010 предлагает единую модель «всё есть присваивание» (`name: type = value`), на синтаксическом уровне типы и значения всё же необходимо различать. В реализации компилятора `Type` и `Expr` — это два независимых перечисления AST (`ast.rs:406` и `ast.rs:25`), где `TypeExpr` в роли заполнителя BNF соответствует перечислению `Type` в реализации и обозначает «в этой позиции ожидается тип».

## Глава третья: Составные типы

### 3.1 Типы записей

**Единый синтаксис**: `Name: Type = { field1: Type1, field2: Type2, ... }`

```
RecordType  ::= '{' FieldList? '}'
FieldList   ::= Field (',' Field)* ','?
Field       ::= Identifier ':' TypeExpr
            |  Identifier                 // ограничение интерфейса
```

```yaoxiang
// Простой тип записи
Point: Type = { x: Float, y: Float }

// Пустой тип записи
Empty: Type = {}

// Обобщённый тип записи
Pair: (T: Type) -> Type = { first: T, second: T }

// Тип записи с реализацией интерфейсов
Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable
}
```

**Правила**:
- Типы записей определяются с помощью фигурных скобок `{}`
- После имени поля сразу следует двоеточие и тип
- Имя интерфейса, записанное в теле типа, означает реализацию этого интерфейса

> **Принадлежность к пространству имён**: префикс `Type.name` (например, `Point.draw`) указывает, что функция принадлежит пространству имён `Point`.
> Это не вызывает никаких неявных привязок. Чтобы синтаксис вызова через `.` (например, `p.draw()`) работал, требуется явная привязка:
> `Point.draw = draw[0]`.
> Подробности см. в RFC-004 и RFC-010.

#### 3.1.1 Значения полей по умолчанию

Поля типов могут иметь значения по умолчанию, и их указание при конструировании необязательно:

```yaoxiang
// Поле со значением по умолчанию — необязательно при конструировании
Point: Type = {
    x: Float = 0,
    y: Float = 0
}

// Использование
Point()           // -> Point(x=0, y=0)
Point(x=1)       // -> Point(x=1, y=0)
Point(x=1, y=2) // -> Point(x=1, y=2)

// Поле без значения по умолчанию — обязательно при конструировании
Point2: Type = {
    x: Float,
    y: Float
}

// Использование
Point2(x=1, y=2) // корректно
Point2()          // ошибка
```

**Правила**:
- `field: Type = expression` -> имеет значение по умолчанию, при конструировании необязательно
- `field: Type` -> без значения по умолчанию, при конструировании обязательно

#### 3.1.2 Встроенные привязки

В теле определения типа можно напрямую привязывать методы:

```yaoxiang
// Способ 1: привязка со ссылкой на внешнюю функцию
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]    // привязка к позиции 0
}
// Вызов: p1.distance(p2) -> distance(p1, p2)

// Способ 2: анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) -> distance(p1, p2)
```

### 3.2 Типы интерфейсов

```
InterfaceType ::= '{' FnField (',' FnField)* ','?
FnField       ::= Identifier ':' FnType
FnType        ::= '(' ParamTypes? ')' '->' TypeExpr
```

**Синтаксис**: интерфейс — это тип записи, все поля которого имеют функциональный тип

```yaoxiang
// Определение интерфейса
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой интерфейс
EmptyInterface: Type = {}
```

**Реализация интерфейса**: тип реализует интерфейс путём перечисления имени интерфейса в конце определения

```yaoxiang
// Тип, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,        // реализация интерфейса Drawable
    Serializable     // реализация интерфейса Serializable
}
```

**Прямое присваивание интерфейсу**: конкретный тип может быть напрямую присвоен переменной интерфейсного типа (структурная подтипизация)

```yaoxiang
// Прямое присваивание (конкретный тип известен во время компиляции -> вызов без накладных расходов)
d: Drawable = Circle(1)
d.draw(screen)        // после компиляции: прямой вызов circle_draw, без vtable

// Возвращаемое значение функции (конкретный тип неизвестен во время компиляции -> вызов через vtable)
d: Drawable = get_shape()
d.draw(screen)        // поиск метода через vtable

// Интерфейс в качестве параметра функции
process: (d: Drawable) -> Void = d.draw(screen)
```

**Стратегии оптимизации на этапе компиляции**:

| Сценарий | Результат вывода | Способ вызова |
|------|----------|----------|
| Прямое присваивание конкретного типа | конкретный тип может быть определён | прямой вызов (нулевые накладные расходы) |
| Возвращаемое значение функции | неизвестно | vtable |
| Гетерогенная коллекция | несколько типов | vtable |

### 3.4 Кортежные типы

```
TupleType   ::= '(' TypeList? ')'
TypeList    ::= TypeExpr (',' TypeExpr)* ','?
```

### 3.5 Функциональные типы

```
FnType      ::= '(' ParamList? ')' '->' TypeExpr
ParamList   ::= TypeExpr (',' TypeExpr)*
```

## Глава четвёртая: Дженерики

### 4.1 Синтаксис параметров дженериков

Параметры дженериков являются частью функционального типа и используют тот же синтаксис `()`, что и обычные параметры:

```
GenericType     ::= Identifier '(' TypeArgList ')'
TypeArgList     ::= TypeExpr (',' TypeExpr)* ','?
TypeBound       ::= Identifier
                 |  Identifier '+' Identifier ('+' Identifier)*
```

В определении обобщённого типа `(T: Type)` является сигнатурой параметра конструктора типа, а `-> Type` обозначает возвращаемый тип:

```yaoxiang
List: (T: Type) -> Type = { ... }
Map: (K: Type, V: Type) -> Type = { ... }
```

В обобщённых функциях параметры типов также объявляются в сигнатуре, а компилятор автоматически выводит их из фактических аргументов:

```yaoxiang
map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R)) = ...
```

### 4.2 Определение обобщённых типов

```yaoxiang
// Базовый обобщённый тип
Option: (T: Type) -> Type = {
    some: (T) -> Option(T),
    none: () -> Option(T)
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Result(T, E),
    err: (E) -> Result(T, E)
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — лишь имя по соглашению, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T)
}
```

### 4.3 Вывод типов

```yaoxiang
// Компилятор автоматически выводит параметры дженериков
numbers: List(Int) = List(1, 2, 3)  // компилятор выводит List(Int)
```

## Глава пятая: Ограничения типов

### 5.1 Единичное ограничение

```
ConstrainedType ::= '(' Identifier ':' TypeBound ')' TypeExpr
```

```yaoxiang
// Определение интерфейса (как ограничение)
Clone: Type = {
    clone: () -> Clone
}

// Использование ограничения
clone: (T: Clone)(value: T) -> T = value.clone()
```

### 5.2 Множественные ограничения

```yaoxiang
// Синтаксис множественных ограничений
combine: (T: Clone + Add)(a: T, b: T) -> T = {
    a.clone() + b
}

// Сортировка в обобщённом контейнере
sort: (T: Clone + PartialOrd)(list: List(T)) -> List(T) = {
    result = list.clone()
    quicksort(&mut result)
    return result
}
```

### 5.3 Ограничения на функциональные типы

```yaoxiang
// Ограничения для функций высшего порядка
call_twice: (T: Type, F: () -> T)(f: F) -> (T, T) = (f(), f())

compose: (A: Type, B: Type, C: Type, F: (A) -> B, G: (B) -> C)(a: A, f: F, g: G) -> C = g(f(a))
```

### 6.1 Определение ассоциированных типов

```

```

```yaoxiang
// Iterator trait (используется синтаксис типа записи)
Iterator: (T: Type) -> Type = {
    Item: T,                    // ассоциированный тип
    next: () -> Option(T),
    has_next: () -> Bool
}

// Использование ассоциированного типа
collect: (T: Type, I: Iterator(T))(iter: I) -> List(T) = {
    result = List(T)()
    while iter.has_next() {
        if let Some(item) = iter.next() {
            result.push(item)
        }
    }
    return result
}
```

## Глава седьмая: Компиляционные дженерики

### 7.1 Параметры-константы времени компиляции

```
LiteralType   ::= Identifier ':' Int          // константа времени компиляции
CompileTimeFn ::= '(' Identifier ':' Int ')' '(' Identifier ')' '->' TypeExpr
```

**Ключевая идея проектирования**: использование параметра дженерика `(n: Int)` в сочетании с параметром-значением `(n: n)` для различения констант времени компиляции и значений времени выполнения.

```yaoxiang
// Факториал во время компиляции: аргумент должен быть литералом, известным на этапе компиляции
factorial: (n: Int)(n: n) -> Int = {
    match n {
        0 => 1,
        _ => n * factorial(n - 1)
    }
}

// Массив с константой времени компиляции
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),      // массив с размером, известным на этапе компиляции
    length: N
}

// Пример использования
arr: StaticArray(Int, factorial(5))  // компилятор вычисляет factorial(5) = 120 во время компиляции
```

## Глава десятая: Перегрузка и специализация функций

### 10.1 Перегрузка функций

```yaoxiang
// Базовая специализация: используется перегрузка функций (компилятор выбирает автоматически)
sum: (arr: Array(Int)) -> Int = {
    return native_sum_int(arr.data, arr.length)
}

sum: (T: Add)(arr: Array(T)) -> T = {
    result = Zero::zero()
    for item in arr {

    }
    return result
}
```

## Глава одиннадцатая: Атрибуты типов

В YaoXiang необходимо различать только один атрибут типа: линейный и копируемый. Он выводится компилятором автоматически.

### 11.1 Move (передача владения по умолчанию)

Все типы по умолчанию следуют семантике Move. Присваивание, передача параметров, возврат = передача владения.

```yaoxiang
p: Point = Point(1.0, 2.0)
q = p           // Move, p больше нельзя читать
```

### 11.2 Dup (поверхностное копирование: копирование дескриптора, общие данные)

**Атрибут Dup используется для типов ссылок/токенов**. Присваивание типа Dup = поверхностное копирование — копируется дескриптор/токен, а базовые данные являются общими. Несколько владельцев указывают на один и тот же блок данных.

| Тип | Атрибут | Описание |
|------|------|------|
| `&T` | Dup | токен чтения нулевого размера, копирование токена = несколько представлений указывают на одни данные |
| `ref T` | Dup | копирование Rc/Arc = счётчик ссылок +1, общие данные в куче |
| `&mut T` | Linear | токен записи нулевого размера, эксклюзивный, не копируемый |
| Все остальные типы | Move | передача владения по умолчанию |

**Примитивные типы значений** (Int, Float, Bool, Char) обрабатываются компилятором особым встроенным образом: при присваивании автоматически копируется значение, и оба значения полностью независимы. Это встроенное поведение компилятора и не относится к атрибуту Dup.

```yaoxiang
// &T: Dup, свободный алиасинг
view: &Point = &p
view2 = view     // Dup: копирование токена, оба действительны
print(view.x)    // допустимо
print(view2.x)   // допустимо

// &mut T: Linear, не копируется
mut_ref: &mut Point = &mut p
// r2 = mut_ref  // ❌ &mut T не является Dup, копировать нельзя
```

### 11.3 Clone (явное глубокое копирование) и его связь с Dup

**Clone** — это интерфейс явного глубокого копирования. Любой тип может реализовать Clone и предоставить метод `.clone()`.

```yaoxiang
// Определение интерфейса Clone (в стандартной библиотеке)
Clone: Type = {
    clone: () -> Clone
}

// Использование
p: Point = Point(1.0, 2.0)
backup = p.clone()    // глубокое копирование, p по-прежнему доступен
p2 = p.clone()        // можно клонировать несколько раз
```

**Различия между Dup и Clone**:

| | Dup | Clone |
|---|---|---|
| **Семантика** | поверхностное копирование: копирование дескриптора/токена, общие базовые данные | глубокое копирование: создание полной независимой копии |
| **Способ вызова** | неявный (автоматически при присваивании/передаче параметров) | явный (`.clone()`) |
| **Влияние изменений** | взаимное (общие базовые данные) | без взаимного влияния (независимые копии) |
| **Применимые типы** | токены `&T`, `ref T` | любой тип, реализующий интерфейс Clone |
| **Стоимость** | нулевые накладные расходы (токен — тип нулевого размера) | зависит от типа |

**Dup не влечёт Clone, а Clone не влечёт Dup** — это два ортогональных понятия:

```yaoxiang
// Тип Dup: копирование токена, общие базовые данные
view: &Point = &p
view2 = view        // Dup: копирование токена, оба указывают на один и тот же p
print(view.x)       // допустимо
print(view2.x)      // допустимо, видят одни и те же данные

// Примитивный тип значения: автоматическое копирование значения компилятором (не Dup)
x: Int = 42
y = x               // копирование значения, x и y полностью независимы
print(x)            // допустимо

// Clone: явное глубокое копирование, создание независимой копии
p: Point = Point(1.0, 2.0)
q = p.clone()       // Clone: глубокое копирование, p по-прежнему доступен
r = p               // Move: передача владения, поскольку Point не является Dup и не относится к примитивным типам
```

**Замысел проектирования**:
- Dup используется для токенов/ссылочных типов и решает задачу «несколько представлений видят одни и те же данные»
- Clone используется в сценариях, требующих независимых копий; явный вызов делает стоимость видимой
- Копирование примитивных типов значений (Int/Float/Bool/Char) — встроенное поведение компилятора, не относящееся к Dup
- Большинство пользовательских типов по умолчанию используют Move, что обеспечивает высокую производительность без копирования

## Глава двенадцатая: Типы токенов заимствования

### 12.1 Ключевые понятия

`&T` и `&mut T` — это **токены времени компиляции нулевого размера**. Это не «ссылки», а «доказательство на уровне типов прав доступа».

```
&T      →  нулевой размер, замораживает исходные данные (запрещает получение
          WriteToken в течение этого периода), при заморозке безопасны
          множественные read-only токены → Dup (копируемый)
&mut T  →  нулевой размер, эксклюзивные чтение и запись (запрещает любые другие
          токены), при эксклюзивном доступе копирование бессмысленно →
          Linear (не Dup)
```

**Ключевые свойства**:
- Токен — это **обычный тип**, подчиняющийся тем же правилам области видимости, что и все остальные типы
- Не требуются аннотации времени жизни `'a`
- Не требуется специальный анализатор заимствований — атрибуты типа (Dup/Linear) естественным образом определяют права
- Полностью исчезают после компиляции, нулевые накладные расходы во время выполнения

### 12.2 Базовое использование

```yaoxiang
// На стороне метода: объявление типа параметра определяет требуемые права
Point.print: (self: &Point) -> Void = {
    print(self.x)               // токен &Point даёт право на чтение
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx        // токен &mut Point даёт право на запись
    self.y = self.y + dy
}

// На стороне вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                       // компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)               // компилятор автоматически создаёт токен &mut Point
p.print()                       // OK, предыдущий токен уже освобождён после вызова shift

// Сосуществование нескольких токенов &T — Dup позволяет свободное копирование
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)
}
d = distance(p, p2)
```

### 12.3 Область видимости и распространение токенов

Токены — это обычные типы, поэтому они поддерживают все операции, доступные обычным типам:

**Возврат токенов** — токены распространяются вместе с возвращаемым значением:

```yaoxiang
// ✅ Дочерний токен возвращается вместе с родительским
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()        // токен возвращается вызывающему
print(px_ref)                    // OK, токен всё ещё в области видимости
```

**Хранение в структурах** — структуры могут содержать поля-токены:

```yaoxiang
// ✅ Структура содержит токен в качестве поля
Window: Type = {
    target: Point,
    view: &Point,              // поле-токен — содержит read-only представление target
}
```

**Захват замыканиями** — замыкания захватывают токены так же, как и любые другие значения:

```yaoxiang
// ✅ Замыкание захватывает токен &Float (Dup, свободно копируется в замыкание)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}
```

### 12.4 Автоматический выбор заимствования

На стороне вызова компилятор автоматически выбирает в соответствии со следующими приоритетами:

```
1. Если фактический аргумент будет использоваться далее → приоритетно создаётся токен (&T или &mut T, согласно сигнатуре метода)
2. Если фактический аргумент далее не используется → Move
3. Приоритет сопоставления: &T < &mut T < Move
```

```yaoxiang
p = Point(1.0, 2.0)
p.print()          // тип параметра print — &Point → компилятор создаёт токен &Point
p.shift(1.0, 1.0)  // тип параметра shift — &mut Point → компилятор создаёт токен &mut Point
p2 = p             // далее не используется → Move
```

### 12.5 Обнаружение конфликтов токенов

Компилятор выполняет **потокочувствительный анализ живучести** для значений токенов, отслеживая состояние каждого токена (активен/перемещён):

```yaoxiang
// ❌ &mut и производный от него &T не могут быть одновременно активны
bad_alias: (p: &mut Point) -> Void = {
    p.x = 10.0                   // ✅ штатное использование WriteToken
    print(p.y)
}

// ✅ После выхода из области видимости токен автоматически освобождается
good_seq: (p: &mut Point) -> Void = {
    {
        // внутренняя область видимости
        print(p.x)               // используется &mut Point
    }
    // конец внутренней области видимости
    p.x = 10.0                   // ✅ WriteToken по-прежнему доступен
}

// ❌ Для одного фактического аргумента нельзя одновременно создать &mut и другие токены
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)                  // ❌ p одновременно порождает токены &mut и &
```

### 12.6 Внутреннее устройство компилятора: механизм брендов

Пользователь никогда не взаимодействует с брендами. Компилятор внутри назначает каждому токену уникальный идентификатор времени компиляции:

```
Видимое пользователю       Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — уникальное целое число времени компиляции
&mut Point     →  WriteToken(Point, #M)   // #M — уникальное целое число времени компиляции
```

Назначение брендов:
- **Защита от подделки**: токен может быть получен только из капсулы владельца, его нельзя сконструировать из ничего
- **Отслеживание связей**: производный `&Float` от доступа к полю несёт производный бренд (`#N.field_x`), и компилятор может отследить его до родительского токена
- **Обнаружение конфликтов**: WriteToken и производный от него ReadToken не могут быть одновременно активны

Бренды полностью исчезают после мономорфизации и инлайнинга, в генерируемом машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

### 12.7 Тип-сумма токенов

```
&BorrowToken ::= &T          // ReadToken (замораживает исходные данные → Dup безопасен)
               | &mut T      // WriteToken (эксклюзивные чтение и запись → Linear)
```

### 12.8 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Назначение | взглянуть/изменить на месте | общее владение |
| Область | вместе с областью видимости значения токена | между областями видимости |
| Стоимость | нулевые накладные расходы (тип нулевого размера, исчезает после компиляции) | Rc или Arc (выбирается компилятором) |
| Побег | возможен (токен распространяется через возвращаемое значение/структуру/замыкание) | именно для этого и предназначен |
| Между задачами | невозможно (передача токенов между задачами не реализована) | возможно (компилятор автоматически выбирает Arc) |
| Обнаружение циклов | не задействовано | в задаче без уведомления, между задачами — lint |
