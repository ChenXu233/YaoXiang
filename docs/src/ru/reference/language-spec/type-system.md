# Спецификация системы типов

В данном документе определяется спецификация системы типов языка программирования YaoXiang, включая базовые типы, составные типы, generics и trait.

---

## Глава 0: Теоретические основы

### 0.1 Соответствие Карри — Ховарда

Соответствие Карри — Ховарда (Curry-Howard correspondence) является теоретической основой системы типов YaoXiang. Оно раскрывает глубокую связь между системой типов языков программирования и математической логикой:

| Логика | Язык программирования |
|--------|-----------------------|
| Суждение \(P\) | Тип `Type` |
| Доказательство \(p: P\) | Программа `x: T = ...` |
| Импликация \(P \rightarrow Q\) | Функциональный тип `(P) -> Q` |
| Конъюнкция \(P \wedge Q\) | Тип-произведение `{ a: P, b: Q }` |
| Дизъюнкция \(P \vee Q\) | Тип-сумма `{ a(P) \| b(Q) }` |
| Универсальная квантификация \(\forall x:T. P(x)\) | generics `(T: Type) -> ...` |
| Истина \(\top\) | Пустой тип `{}` |
| Ложь \(\bot\) | `Void` / `Never` |
| Тип вселенной \(Type_n : Type_{n+1}\) | Иерархия вселенных (защита от парадокса Рассела) |
| Математическая индукция | Типовый `match` |

### 0.2 Типы как суждения, программы как доказательства

В YaoXiang данное соответствие является основным принципом проектирования:

- **Тип — это логическое суждение**. `Int` — суждение о существовании целого числа, `fn(a: Int, b: Int) -> Int` — суждение о существовании целого числа при заданных двух целых.
- **Проверка типов — это верификация доказательства**. Когда программа проходит проверку типов, это равносильно конструктивному доказательству логического суждения.
- **Завершающиеся вычисления на уровне типов соответствуют корректным индуктивным рассуждениям**. Типовые семейства YaoXiang (например, `Add` с pattern matching на `Nat`) по сути являются кодированием математической индукции на уровне типов.

### 0.3 Влияние на дизайн языка

Соответствие Карри — Ховарда в YaoXiang проявляется следующим образом:

1. **Иерархия вселенных** (RFC-010): `Type₀ : Type₁ : Type₂ …` позволяет избежать логических парадоксов (парадокса Жирара), вызванных `Type: Type`
2. **Типовые семейства** (RFC-011): Паттерн-матчинг на уровне типов для натуральных чисел `Nat(Zero/Succ)` соответствует индуктивному доказательству в аксиоматике Пеано
3. **Условные типы** (RFC-011): `If: (C: Bool, T: Type, E: Type) -> Type` соответствует case-дизъюнкции в логике
4. **Зависимые от значений типы** (RFC-011): `Vec: (n: Int) -> Type` соответствует конечной квантификации «для каждого целого n существует тип»

## Глава 3: Составные типы

### 3.1 Записываемые типы

**Унифицированный синтаксис**: `Name: Type = { field1: Type1, field2: Type2, ... }`

```
RecordType  ::= '{' FieldList? '}'
FieldList   ::= Field (',' Field)* ','?
Field       ::= Identifier ':' TypeExpr
            |  Identifier                 // 接口约束
```

```yaoxiang
// Простой записиваемый тип
Point: Type = { x: Float, y: Float }

// Пустой записиваемый тип
Empty: Type = {}

// Записиваемый тип с generics
Pair: (T: Type) -> Type = { first: T, second: T }

// Записиваемый тип с реализацией интерфейса
Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable
}
```

**Правила**:
- Записываемые типы определяются фигурными скобками `{}`
- Имя поля непосредственно сопровождается двоеточием и типом
- Имя интерфейса внутри тела типа означает реализацию этого интерфейса

#### 3.1.1 Значения полей по умолчанию

Поля типов могут иметь значения по умолчанию, которые можно не указывать при конструировании:

```yaoxiang
// Поле со значением по умолчанию — при конструировании необязательно
Point: Type = {
    x: Float = 0,
    y: Float = 0
}

// Использование
Point()           // -> Point(x=0, y=0)
Point(x=1)       // -> Point(x=1, y=0)
Point(x=1, y=2) // -> Point(x=1, y=2)

// Поле без значения по умолчанию — при конструировании обязательно
Point2: Type = {
    x: Float,
    y: Float
}

// Использование
Point2(x=1, y=2) // Правильно
Point2()          // Ошибка
```

**Правила**:
- `field: Type = expression` -> есть значение по умолчанию, при конструировании необязательно
- `field: Type` -> нет значения по умолчанию, при конструировании обязательно

#### 3.1.2 Встроенные привязки

Внутри определения типа можно напрямую привязывать методы:

```yaoxiang
// Способ 1: ссылка на внешнюю функцию
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]    // Привязка к позиции 0
}
// Вызов: p1.distance(p2) -> distance(p1, p2)

// Способ 2: анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) -> distance(p1, p2)
```

### 3.2 Интерфейсные типы

```
InterfaceType ::= '{' FnField (',' FnField)* ','?
FnField       ::= Identifier ':' FnType
FnType        ::= '(' ParamTypes? ')' '->' TypeExpr
```

**Синтаксис**: Интерфейс — это записиваемый тип, все поля которого являются функциональными типами

```yaoxiang
// Определение интерфейса
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой интерфейс
EmptyInterface: Type = {}
```

**Реализация интерфейса**: Тип реализует интерфейс путём перечисления имени интерфейса в конце определения

```yaoxiang
// Тип, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,        // Реализует интерфейс Drawable
    Serializable     // Реализует интерфейс Serializable
}
```

**Прямое присваивание интерфейсу**: Конкретный тип может напрямую присваиваться переменной интерфейсного типа (структурное подтипирование)

```yaoxiang
// Прямое присваивание (конкретный тип определяется на этапе компиляции -> вызов без накладных расходов)
d: Drawable = Circle(1)
d.draw(screen)        // После компиляции: прямой вызов circle_draw, без vtable

// Возврат функции (конкретный тип неизвестен на этапе компиляции -> вызов через vtable)
d: Drawable = get_shape()
d.draw(screen)        // Поиск метода через vtable

// Интерфейс как параметр функции
process: (d: Drawable) -> Void = d.draw(screen)
```

**Стратегия оптимизации компиляции**:

| Сценарий | Результат вывода | Способ вызова |
|----------|-------------------|---------------|
| Прямое присваивание конкретного типа | Конкретный тип определён | Прямой вызов (ноль накладных расходов) |
| Возврат из функции | Неизвестно | vtable |
| Гетерогенная коллекция | Несколько типов | vtable |

### 3.4 Кортежные типы

```
TupleType   ::= '(' TypeList? ')'
TypeList    ::= TypeExpr (',' TypeExpr)* ','?
```

### 3.5 Функциональные типы

```
FnType      ::= '(' ParamList? ')' '->' TypeExpr
ParamList   ::= TypeExpr (',' TypeExpr)*
```

## Глава 4: Generics

### 4.1 Синтаксис параметров generics

Параметры generics являются частью функционального типа и унифицированно используют синтаксис `()` наряду с обычными параметрами:

```
GenericType     ::= Identifier '(' TypeArgList ')'
TypeArgList     ::= TypeExpr (',' TypeExpr)* ','?
TypeBound       ::= Identifier
                 |  Identifier '+' Identifier ('+' Identifier)*
```

В определении generic-типа `(T: Type)` — это сигнатура параметра конструктора типа, а `-> Type` — возвращаемый тип:

```yaoxiang
List: (T: Type) -> Type = { ... }
Map: (K: Type, V: Type) -> Type = { ... }
```

В generic-функциях параметры типа также объявляются в сигнатуре, компилятор автоматически выводит их из фактических аргументов:

```yaoxiang
map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R)) = ...
```

### 4.2 Определение generic-типов

```yaoxiang
// Базовый generic-тип
Option: (T: Type) -> Type = {
    some: (T) -> Option(T),
    none: () -> Option(T)
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Result(T, E),
    err: (E) -> Result(T, E)
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,
    get: (self: List(T), index: Int) -> Option(T)
}
```

### 4.3 Вывод типов

```yaoxiang
// Компилятор автоматически выводит параметры generics
numbers: List(Int) = List(1, 2, 3)  // Компилятор выводит List(Int)
```

## Глава 5: Ограничения типов

### 5.1 Единичное ограничение

```
ConstrainedType ::= '(' Identifier ':' TypeBound ')' TypeExpr
```

```yaoxiang
// Определение интерфейсного типа (как ограничение)
Clone: Type = {
    clone: () -> Clone
}

// Использование ограничения
clone: (T: Clone)(value: T) -> T = value.clone()
```

### 5.2 Множественные ограничения

```yaoxiang
// Синтаксис множественных ограничений
combine: (T: Clone + Add)(a: T, b: T) -> T = {
    a.clone() + b
}

// Сортировка generic-контейнера
sort: (T: Clone + PartialOrd)(list: List(T)) -> List(T) = {
    result = list.clone()
    quicksort(&mut result)
    return result
}
```

### 5.3 Ограничения функциональных типов

```yaoxiang
// Ограничения для функций высшего порядка
call_twice: (T: Type, F: () -> T)(f: F) -> (T, T) = (f(), f())

compose: (A: Type, B: Type, C: Type, F: (A) -> B, G: (B) -> C)(a: A, f: F, g: G) -> C = g(f(a))
```

```yaoxiang
// Iterator trait (использование синтаксиса записиваемого типа)
Iterator: (T: Type) -> Type = {
    Item: T,                    // Ассоциированный тип
    next: () -> Option(T),
    has_next: () -> Bool
}

// Использование ассоциированного типа
collect: (T: Type, I: Iterator(T))(iter: I) -> List(T) = {

        if let Some(item) = iter.next() {
            result.push(item)
        }
    }
    return result
}
```

## Глава 7: Компиляционные generics

### 7.1 Параметры — литералы компиляционного времени

```
LiteralType   ::= Identifier ':' Int          // Литерал компиляционного времени
CompileTimeFn ::= '(' Identifier ':' Int ')' '(' Identifier ')' '->' TypeExpr
```

**Основной дизайн**: использование `(n: Int)` в качестве generic-параметра и `(n: n)` в качестве параметра значения для различения compile-time-литералов от runtime-значений.

```yaoxiang
// Факториал на этапе компиляции: параметр должен быть литералом, известным компилятору
factorial: (n: Int)(n: n) -> Int = {
    match n {
        0 => 1,
        _ => n * factorial(n - 1)
    }
}

// Массив compile-time-констант
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),      // Массив с известным на этапе компиляции размером
    length: N
}

// Использование
arr: StaticArray(Int, factorial(5))  // Компилятор вычисляет factorial(5) = 120 на этапе компиляции
```

### 7.2 Массивы compile-time-констант

```yaoxiang
// Использование для типов матриц
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows)
}

// Верификация размерностей на этапе компиляции
identity_matrix: (T: Add + Zero + One, N: Int)(size: N) -> Matrix(T, N, N) = {
    // ...
}
```

```yaoxiang
// Базовая специализация: использование перегрузки функций (компилятор выбирает автоматически)

}
```

## Глава 11: Атрибуты типов

В YaoXiang существует только один атрибут типа, который необходимо различать: линейный vs копируемый. Выводится компилятором автоматически.

### 11.1 Move (передача владения по умолчанию)

Все типы по умолчанию следуют семантике Move. Присваивание, передача параметров, возврат = передача владения.

```yaoxiang
p: Point = Point(1.0, 2.0)
q = p           // Move, p больше нельзя читать
```

### 11.2 Dup (поверхностное копирование: копирование дескриптора, общие данные)

**Атрибут Dup используется для типов-ссылок/токенов**. Присваивание типа Dup = поверхностное копирование — копируется дескриптор/токен, базовые данные общие. Несколько владельцев указывают на одни и те же данные.

| Тип | Атрибут | Описание |
|-----|---------|----------|
| `&T` | Dup | Токен чтения нулевого размера, копирование токена = несколько представлений на одни данные |
| `ref T` | Dup | Rc/Arc копирование = +1 к счётчику ссылок, общие кучевов данные |
| `&mut T` | Linear | Токен записи нулевого размера, эксклюзивный, не копируемый |
| Все остальные типы | Move | Передача владения по умолчанию |

**Примитивные типы значений** (Int, Float, Bool, Char) имеют специальную встроенную обработку компилятором: при присваивании автоматически происходит копирование значения, два значения полностью независимы. Это встроенное поведение компилятора, не являющееся атрибутом типа Dup.

```yaoxiang
// &T: Dup, можно свободно создавать псевдонимы
view: &Point = &p
view2 = view     // Dup: копирование токена, оба действительны
print(view.x)    // Доступно
print(view2.x)   // Доступно

// &mut T: Linear, не копируется
mut_ref: &mut Point = &mut p
// r2 = mut_ref  // ❌ &mut T не является Dup, нельзя копировать
```

### 11.3 Clone (явное глубокое копирование) и связь с Dup

**Clone** — это интерфейс явного глубокого копирования. Все типы могут реализовать Clone, предоставляя метод `.clone()`.

```yaoxiang
// Определение интерфейса Clone (стандартная библиотека)
Clone: Type = {
    clone: () -> Clone
}

// Использование
p: Point = Point(1.0, 2.0)
backup = p.clone()    // Глубокое копирование, p всё ещё доступен
p2 = p.clone()        // Можно клонировать многократно
```

**Различие между Dup и Clone**:

| | Dup | Clone |
|---|---|---|
| **Семантика** | Поверхностное копирование: копирование дескриптора/токена, общие базовые данные | Глубокое копирование: создание полностью независимой копии |
| **Способ вызова** | Неявный (автоматически при присваивании/передаче параметров) | Явный (`.clone()`) |
| **Влияние изменений** | Влияют друг на друга (общие базовые данные) | Не влияют друг на друга (независимые копии) |
| **Применимые типы** | Токены `&T`, `ref T` | Любой тип, реализующий интерфейс Clone |
| **Стоимость** | Нулевые накладные расходы (токены — типы нулевого размера) | Зависит от типа |

**Dup не влечёт Clone, Clone не влечёт Dup** — это два ортогональных понятия:

```yaoxiang
// Тип Dup: копирование токена, общие базовые данные
view: &Point = &p
view2 = view        // Dup: копирование токена, оба указывают на один p
print(view.x)       // Доступно
print(view2.x)      // Доступно, видны одни и те же данные

// Примитивные типы значений: автоматическое копирование компилятором (не Dup)
x: Int = 42
y = x               // Копирование значения, x и y полностью независимы
print(x)            // Доступно

// Clone: явное глубокое копирование, создание независимой копии
p: Point = Point(1.0, 2.0)
q = p.clone()       // Clone: глубокое копирование, p всё ещё доступен
r = p               // Move: передача владения, так как Point не является Dup и не является примитивом
```

**Замысел дизайна**:
- Dup используется для типов токенов/ссылок, решая проблему «несколько представлений одних данных»
- Clone используется для сценариев, требующих независимых копий, явный вызов делает стоимость видимой
- Примитивные типы значений (Int/Float/Bool/Char) копируются встроенным поведением компилятора, не относятся к Dup
- Большинство пользовательских типов по умолчанию Move, нулевое копирование для высокой производительности

## Глава 12: Типы заимствования токенов

### 12.1 Основные концепции

`&T` и `&mut T` — это **токены-дескрипторы компиляционного времени нулевого размера**. Они не являются «ссылками», а представляют собой «типовые доказательства прав доступа».

```
&T      →  Нулевой размер, Dup (копируемый), предоставляет права только на чтение
&mut T  →  Нулевой размер, Linear (не Dup), предоставляет права на эксклюзивное чтение-запись
```

**Ключевые характеристики**:
- Токены — это **обычные типы**, следующие тем же правилам области видимости, что и все остальные типы
- Не требуют аннотаций времени жизни `'a`
- Не требуют специального проверяющего заимствования — атрибуты типа (Dup/Linear) естественно выводят права
- Полностью исчезают после компиляции, нулевые накладные расходы во время выполнения

### 12.2 Базовое использование

```yaoxiang
// Сторона метода: объявление типа параметра, определяющего требуемые права
Point.print: (self: &Point) -> Void = {
    print(self.x)               // Токен &Point предоставляет права на чтение
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx        // Токен &mut Point предоставляет права на запись
    self.y = self.y + dy
}

// Сторона вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                       // Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)               // Компилятор автоматически создаёт токен &mut Point
p.print()                       // OK, предыдущий токен освобождён после завершения вызова shift

// Сосуществование нескольких токенов &T — тип Dup позволяет свободно копировать
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)
}
d = distance(p, p2)
```

### 12.3 Область видимости и распространение токенов

Токены — обычные типы, поэтому поддерживают все операции обычных типов:

**Возврат токенов** — токены распространяются вместе с возвращаемым значением:

```yaoxiang
// ✅ Дочерний токен и родительский токен возвращаются вместе
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()        // Токен возвращается вызывающему
print(px_ref)                    // OK, токен всё ещё в области видимости
```

**Хранение в структурах** — структуры могут содержать поля-токены:

```yaoxiang
// ✅ Структура содержит токен как поле
Window: Type = {
    target: Point,
    view: &Point,              // Поле-токен — хранит只读 представление target
}
```

**Захват замыканиями** — замыкания захватывают токены как любые другие значения:

```yaoxiang
// ✅ Замыкание захватывает токен &Float (тип Dup, свободно копируется в замыкание)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}
```

### 12.4 Автоматический выбор заимствования

Компилятор на стороне вызова автоматически выбирает по следующему приоритету:

```
1. Если фактический аргумент ещё используется далее → предпочтительно создание токена (&T или &mut T)
2. Если фактический аргумент далее не используется → Move
3. Приоритет сопоставления: &T < &mut T < Move
```

```yaoxiang
p = Point(1.0, 2.0)
p.print()          // print объявляет &self → компилятор создаёт токен &Point
p.shift(1.0, 1.0)  // shift объявляет &mut self → компилятор создаёт токен &mut Point
p2 = p             // Далее не используется → Move
```

### 12.5 Механизм заморозки

Токен `&mut T` можно временно «заморозить» для получения токена `&T`:

```yaoxiang
modify_and_read: (p: &mut Point) -> Void = {
    p.x = 10.0                   // Использование &mut Point для изменения
    
    // Заморозить &mut, получить只读 представление
    view: &Point = freeze(p)     // p заморожен здесь
    print(view.x)                // Чтение через &Point
    print(view.y)
    // view покидает область видимости, заморозка снята
    
    p.y = 20.0                   // &mut Point снова доступен
}
```

Семантика `freeze`:
- Принимает `&mut T`, возвращает `&T`
- Пока `&T` жив, исходный `&mut T` недоступен
- После выхода `&T` из области видимости `&mut T` автоматически восстанавливается
- Это **потоково-чувствительный анализ активности** — компилятор отслеживает состояние токена внутри тела функции

### 12.6 Обнаружение конфликтов токенов

Компилятор выполняет **потоково-чувствительный анализ активности** для значений токенов, отслеживая состояние каждого токена (активен/заморожен/перемещён):

```yaoxiang
// ❌ &mut и производный &T не могут быть активны одновременно
bad_alias: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)     // p заморожен
    p.x = 10.0                   // ❌ Ошибка компиляции: WriteToken в замороженном состоянии
    print(view.x)
}

// ✅ После снятия заморозки можно продолжить использовать &mut
good_seq: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)     // p заморожен
    print(view.x)                // Использование &T
    // view покидает область видимости, заморозка снята
    p.x = 10.0                   // ✅ WriteToken восстановлен
}

// ❌ Один фактический аргумент не может одновременно порождать &mut токен и другие токены
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)                  // ❌ p одновременно порождает &mut и & токены
```

### 12.7 Внутреннее устройство компилятора: механизм брендов

Пользователь никогда не сталкивается с брендами. Компилятор внутренне назначает каждому токену компиляционно-уникальный идентификатор:

```
Видимое пользователю         Внутреннее представление компилятора
────────────────────────────────────────────────────────────────
&Point              →  ReadToken(Point, #N)    // #N — компиляционно-уникальное целое
&mut Point          →  WriteToken(Point, #M)   // #M — компиляционно-уникальное целое
```

Назначение брендов:
- **Защита от подделки**: токены можно получить только от владельца-капсулы или операции freeze, нельзя сконструировать из воздуха
- **Отслеживание происхождения**: `&Float`, производный от доступа к полю, несёт производный бренд (`#N.field_x`), компилятор может отследить до родительского токена
- **Обнаружение конфликтов**: WriteToken и производный ReadToken из одного источника не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнинга, в сгенерированном машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

### 12.8 Суммарные типы токенов заимствования

```
&BorrowToken ::= &T          // ReadToken (Dup, копируемый)
               | &mut T      // WriteToken (Linear, эксклюзивный)
```

### 12.9 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Что делает | Взглянуть / изменить на месте | Совместное владение |
| Область действия | В области видимости значения токена | Через область видимости |
| Стоимость | Нулевые накладные расходы (тип нулевого размера, исчезает после компиляции) | Rc или Arc (выбирается компилятором) |
| Побег | Возможен (токен распространяется через返回值/структуры/замыкания) | Предназначен для побега |
| Межзадачность | Невозможно (токен не реализует Send) | Возможно (компилятор автоматически выбирает Arc) |
| Обнаружение циклов | Не применимо | Внутри задачи — без уведомления, между задачами — lint |
