# Спецификация системы типов

Настоящий документ определяет спецификацию системы типов языка программирования YaoXiang, включая базовые типы, составные типы, generics и trait.

---

## Глава 0: Теоретические основы

### 0.1 Соответствие Curry-Howard

Соответствие Curry-Howard (Curry-Howard correspondence) является теоретической основой системы типов YaoXiang. Оно раскрывает глубокую связь между системой типов языка программирования и математической логикой:

| Логика | Язык программирования |
|--------|------------------------|
| Высказывание \(P\) | Тип `Type` |
| Доказательство \(p: P\) | Программа `x: T = ...` |
| Импликация \(P \rightarrow Q\) | Функциональный тип `(P) -> Q` |
| Конъюнкция \(P \wedge Q\) | Тип-произведение `{ a: P, b: Q }` |
| Дизъюнкция \(P \vee Q\) | Тип-сумма `{ a(P) \| b(Q) }` |
| Универсальная квантификация \(\forall x:T. P(x)\) | Generics `(T: Type) -> ...` |
| Истина \(\top\) | Пустой тип `{}` |
| Ложь \(\bot\) | `Void` / `Never` |
| Типовой вселенная \(Type_n : Type_{n+1}\) | Иерархия вселенных (защита от парадокса Рассела) |
| Математическая индукция | Типовой `match` |

### 0.2 Типы как высказывания, программы как доказательства

В YaoXiang это соответствие является основным принципом проектирования:

- **Тип — это логическое высказывание**. `Int` — это высказывание «целое число существует», `fn(a: Int, b: Int) -> Int` — это высказывание «для двух целых чисел существует целое число».
- **Проверка типов — это верификация доказательства**. Когда программа проходит проверку типов, это эквивалентно конструктивному доказательству логического высказывания.
- **Завершающиеся типовые вычисления соответствуют корректным индуктивным рассуждениям**. Типовые семейства YaoXiang (такие как паттерн-матчинг `Add` на `Nat`) по своей сути являются кодированием математической индукции на уровне типов.

### 0.3 Влияние на дизайн языка

Соответствие Curry-Howard в YaoXiang проявляется следующим образом:

1. **Иерархия вселенных** (RFC-010): `Type₀ : Type₁ : Type₂ …` позволяет избежать логических парадоксов (парадокса Жирара), вызванных `Type: Type`
2. **Типовые семейства** (RFC-011): паттерн-матчинг на уровне типов для натуральных чисел `Nat(Zero/Succ)` соответствует индуктивному доказательству в аксиоматике Пеано
3. **Условные типы** (RFC-011): `If: (C: Bool, T: Type, E: Type) -> Type` соответствует case-дизъюнкции в логике
4. **Зависимые от значений типы** (RFC-011): `Vec: (n: Int) -> Type` соответствует конечной квантификации «для каждого целого n существует тип»

## Глава 3: Составные типы

### 3.1 Записи (记录类型)

**Унифицированный синтаксис**: `Name: Type = { field1: Type1, field2: Type2, ... }`

```
RecordType  ::= '{' FieldList? '}'
FieldList   ::= Field (',' Field)* ','?
Field       ::= Identifier ':' TypeExpr
            |  Identifier                 // 接口约束
```

```yaoxiang
// Простая запись
Point: Type = { x: Float, y: Float }

// Пустая запись
Empty: Type = {}

// Запись с generics
Pair: (T: Type) -> Type = { first: T, second: T }

// Запись, реализующая интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable
}
```

**Правила**:
- Записи определяются фигурными скобками `{}`
- После имени поля ставится двоеточие и тип
- Имена интерфейсов в теле типа означают их реализацию

#### 3.1.1 Значения полей по умолчанию

Поля типа могут иметь значения по умолчанию, которые можно не указывать при конструировании:

```yaoxiang
// Поля со значениями по умолчанию - необязательны при конструировании
Point: Type = {
    x: Float = 0,
    y: Float = 0
}

// Использование
Point()           // -> Point(x=0, y=0)
Point(x=1)       // -> Point(x=1, y=0)
Point(x=1, y=2) // -> Point(x=1, y=2)

// Поля без значений по умолчанию - обязательны при конструировании
Point2: Type = {
    x: Float,
    y: Float
}

// Использование
Point2(x=1, y=2) // Правильно
Point2()          // Ошибка
```

**Правила**:
- `field: Type = expression` -> есть значение по умолчанию, необязательно при конструировании
- `field: Type` -> нет значения по умолчанию, обязательно при конструировании

#### 3.1.2 Встроенные привязки

Внутри определения типа можно напрямую привязывать методы:

```yaoxiang
// Способ 1: ссылка на внешнюю функцию
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]    // Привязка к позиции 0
}
// Вызов: p1.distance(p2) -> distance(p1, p2)

// Способ 2: анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) -> distance(p1, p2)
```

### 3.2 Интерфейсные типы

```
InterfaceType ::= '{' FnField (',' FnField)* ','?
FnField       ::= Identifier ':' FnType
FnType        ::= '(' ParamTypes? ')' '->' TypeExpr
```

**Синтаксис**: интерфейс — это запись, все поля которой являются функциональными типами

```yaoxiang
// Определение интерфейса
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой интерфейс
EmptyInterface: Type = {}
```

**Реализация интерфейсов**: тип реализует интерфейс, перечисляя его имя в конце определения

```yaoxiang
// Тип, реализующий интерфейсы
Point: Type = {
    x: Float,
    y: Float,
    Drawable,        // Реализует интерфейс Drawable
    Serializable     // Реализует интерфейс Serializable
}
```

**Прямое присваивание интерфейсу**: конкретный тип можно напрямую присвоить переменной интерфейсного типа (структурное подтипирование)

```yaoxiang
// Прямое присваивание (конкретный тип определяется на этапе компиляции -> вызов без накладных расходов)
d: Drawable = Circle(1)
d.draw(screen)        // После компиляции: прямой вызов circle_draw, без vtable

// Возврат функции (конкретный тип неизвестен на этапе компиляции -> вызов через vtable)
d: Drawable = get_shape()
d.draw(screen)        // Поиск метода через vtable

// Интерфейс как параметр функции
process: (d: Drawable) -> Void = d.draw(screen)
```

**Стратегия оптимизации компиляции**:

| Сценарий | Результат вывода | Способ вызова |
|----------|------------------|---------------|
| Прямое присваивание конкретного типа | Конкретный тип определён | Прямой вызов (ноль накладных расходов) |
| Возврат из функции | Неизвестен | vtable |
| Гетерогенная коллекция | Несколько типов | vtable |

### 3.4 Кортежные типы

```
TupleType   ::= '(' TypeList? ')'
TypeList    ::= TypeExpr (',' TypeExpr)* ','?
```

### 3.5 Функциональные типы

```
FnType      ::= '(' ParamList? ')' '->' TypeExpr
ParamList   ::= TypeExpr (',' TypeExpr)*
```

## Глава 4: Generics

### 4.1 Синтаксис параметров generics

Параметры generics являются частью сигнатуры функционального типа и统一使用 синтаксис `()` наравне с обычными параметрами:

```
GenericType     ::= Identifier '(' TypeArgList ')'
TypeArgList     ::= TypeExpr (',' TypeExpr)* ','?
TypeBound       ::= Identifier
                 |  Identifier '+' Identifier ('+' Identifier)*
```

В определении generic-типа `(T: Type)` — это сигнатура параметров конструктора типа, а `-> Type` — тип возвращаемого значения:

```yaoxiang
List: (T: Type) -> Type = { ... }
Map: (K: Type, V: Type) -> Type = { ... }
```

В generic-функциях параметры типа同样声明ются в сигнатуре, компилятор автоматически выводит их из фактических аргументов:

```yaoxiang
map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R)) = ...
```

### 4.2 Определение generic-типов

```yaoxiang
// Базовый generic-тип
Option: (T: Type) -> Type = {
    some: (T) -> Option(T),
    none: () -> Option(T)
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Result(T, E),
    err: (E) -> Result(T, E)
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,
    get: (self: List(T), index: Int) -> Option(T)
}
```

### 4.3 Вывод типов

```yaoxiang
// Компилятор автоматически выводит параметры generics
numbers: List(Int) = List(1, 2, 3)  // Компилятор выводит List(Int)
```

## Глава 5: Ограничения типов

### 5.1 Единичное ограничение

```
ConstrainedType ::= '(' Identifier ':' TypeBound ')' TypeExpr
```

```yaoxiang
// Определение интерфейсного типа (как ограничение)
Clone: Type = {
    clone: () -> Clone
}

// Использование ограничения
clone: (T: Clone)(value: T) -> T = value.clone()
```

### 5.2 Множественные ограничения

```yaoxiang
// Синтаксис множественных ограничений
combine: (T: Clone + Add)(a: T, b: T) -> T = {
    a.clone() + b
}

// Сортировка generic-контейнера
sort: (T: Clone + PartialOrd)(list: List(T)) -> List(T) = {
    result = list.clone()
    quicksort(&mut result)
    return result
}
```

### 5.3 Ограничения функциональных типов

```yaoxiang
// Ограничения для функций высшего порядка
call_twice: (T: Type, F: () -> T)(f: F) -> (T, T) = (f(), f())

compose: (A: Type, B: Type, C: Type, F: (A) -> B, G: (B) -> C)(a: A, f: F, g: G) -> C = g(f(a))
```

collect: (T: Type, I: Iterator(T))(iter: I) -> List(T) = {

## Глава 7: Компиляционные generics

### 7.1 Параметры — компиляционные константы

```
LiteralType   ::= Identifier ':' Int          // Компиляционная константа
CompileTimeFn ::= '(' Identifier ':' Int ')' '(' Identifier ')' '->' TypeExpr
```

**Основной дизайн**: использование `(n: Int)` generic-параметра + `(n: n)` значения параметра позволяет различать компиляционные константы и значения времени выполнения.

```yaoxiang
// Компиляционный факториал: параметр должен быть известен на этапе компиляции литерал
factorial: (n: Int)(n: n) -> Int = {
    match n {
        0 => 1,
        _ => n * factorial(n - 1)
    }
}

// Массив компиляционных констант
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),      // Массив с размером, известным на этапе компиляции
    length: N
}

// Использование
arr: StaticArray(Int, factorial(5))  // Компилятор вычисляет factorial(5) = 120 на этапе компиляции
```

### 7.2 Массивы компиляционных констант

## Глава 10: Перегрузка и специализация функций

### 10.1 Перегрузка функций

```yaoxiang
// Базовая специализация: использование перегрузки функций (компилятор выбирает автоматически)
sum: (arr: Array(Int)) -> Int = {
    return native_sum_int(arr.data, arr.length)
}

sum: (arr: Array(Float)) -> Float = {

// Универсальная реализация
sum: (T: Add)(arr: Array(T)) -> T = {
    result = Zero::zero()
    for item in arr {
        result = result + item
    }
    return result
}
```

## Глава 11: Свойства типов

В YaoXiang существует только одно свойство типа, которое необходимо различать: линейный vs копируемый. Выводится компилятором автоматически.

### 11.1 Move (передача владения по умолчанию)

Все типы по умолчанию следуют семантике Move. Присваивание, передача параметров, возврат = передача владения.

```yaoxiang
p: Point = Point(1.0, 2.0)
q = p           // Move, p больше нельзя использовать
```

### 11.2 Dup (поверхностное копирование: копирование дескриптора, совместное использование данных)

**Атрибут Dup используется для типов ссылок/токенов**. Присваивание значения типа Dup = поверхностное копирование — копируется дескриптор/токен, базовые данные совместно используются. Несколько владельцев указывают на одни и те же данные.

| Тип | Атрибут | Описание |
|------|---------|----------|
| `&T` | Dup | Токен чтения нулевого размера, копирование токена = несколько представлений на одни данные |
| `ref T` | Dup | Копирование Rc/Arc = +1 к счётчику ссылок, совместное использование данных в куче |
| `&mut T` | Linear | Токен записи нулевого размера, эксклюзивный, не копируемый |
| Все остальные типы | Move | Передача владения по умолчанию |

**Примитивные типы-значения** (Int, Float, Bool, Char) — это специальный случай, встроенный в компилятор: при присваивании автоматически выполняется копирование значения, два значения полностью независимы. Это нативное поведение компилятора, не относящееся к атрибуту Dup.

```yaoxiang
// &T: Dup, можно свободно создавать алиасы
view: &Point = &p
view2 = view     // Dup: копирование токена, оба валидны
print(view.x)    // Можно использовать
print(view2.x)   // Можно использовать

// &mut T: Linear, не копируемый
mut_ref: &mut Point = &mut p
// r2 = mut_ref  // ❌ &mut T не является Dup, нельзя копировать
```

### 11.3 Clone (явное глубокое копирование) и связь с Dup

**Clone** — это интерфейс явного глубокого копирования. Любой тип может реализовать Clone, предоставляя метод `.clone()`.

```yaoxiang
// Определение интерфейса Clone (стандартная библиотека)
Clone: Type = {
    clone: () -> Clone
}

// Использование
p: Point = Point(1.0, 2.0)
backup = p.clone()    // Глубокое копирование, p всё ещё можно использовать
p2 = p.clone()        // Можно клонировать многократно
```

**Разница между Dup и Clone**:

| | Dup | Clone |
|---|---|---|
| **Семантика** | Поверхностное копирование: копирование дескриптора/токена, базовые данные совместны | Глубокое копирование: создаётся полностью независимая копия |
| **Способ вызова** | Неявный (автоматически при присваивании/передаче параметров) | Явный (`.clone()`) |
| **Влияние изменений** | Взаимное влияние (совместные базовые данные) | Независимость (независимые копии) |
| **Применимые типы** | Токены `&T`, `ref T` | Любой тип, реализующий интерфейс Clone |
| **Стоимость** | Нулевые накладные расходы (токен — тип нулевого размера) | Зависит от типа |

**Dup не влечёт Clone, Clone не влечёт Dup** — это два ортогональных понятия:

```yaoxiang
// Тип Dup: копирование токена, базовые данные совместны
view: &Point = &p
view2 = view        // Dup: копирование токена, оба указывают на один p
print(view.x)       // Можно использовать
print(view2.x)      // Можно использовать, видите одни и те же данные

// Примитивные типы-значения: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x               // Копирование значения, x и y полностью независимы
print(x)            // Можно использовать

// Clone: явное глубокое копирование, создание независимой копии
p: Point = Point(1.0, 2.0)
q = p.clone()       // Clone: глубокое копирование, p всё ещё можно использовать
r = p               // Move: передача владения, потому что Point не Dup и не примитив
```

**Замысел дизайна**:
- Dup используется для токенов/ссылочных типов, решая проблему «нескольких представлений одних данных»
- Clone используется в сценариях, требующих независимых копий, явный вызов делает стоимость видимой
- Примитивные типы-значения (Int/Float/Bool/Char) имеют встроенное в компилятор поведение копирования, не относящееся к Dup
- Большинство пользовательских типов по умолчанию Move — нулевое копирование, высокая производительность

## Глава 12: Типы заимствования токенов

### 12.1 Основные концепции

`&T` и `&mut T` — это **токены компиляционного времени нулевого размера**. Это не «ссылки», а «типовые доказательства прав доступа».

```
&T      →  Нулевой размер, Dup (копируемый), предоставляет права только на чтение
&mut T  →  Нулевой размер, Linear (не Dup), предоставляет эксклюзивные права на чтение/запись
```

**Ключевые характеристики**:
- Токены — это **обычные типы**, следующие тем же правилам области видимости, что и все остальные типы
- Не нужны аннотации времени жизни `'a`
- Не нужен специальный проверяющий заимствования — свойства типа (Dup/Linear) естественно выводят права доступа
- Полностью исчезают после компиляции, нулевые накладные расходы во время выполнения

### 12.2 Базовое использование

```yaoxiang
// Сторона метода: объявление типа параметра определяет необходимые права
Point.print: (self: &Point) -> Void = {
    print(self.x)               // Токен &Point предоставляет права на чтение
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx        // Токен &mut Point предоставляет права на запись
    self.y = self.y + dy
}

// Сторона вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                       // Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)               // Компилятор автоматически создаёт токен &mut Point
p.print()                       // OK, предыдущий токен освобождён после вызова shift

// Сосуществование нескольких токенов &T — тип Dup позволяет свободно копировать
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)
}
d = distance(p, p2)
```

### 12.3 Область действия токенов и их распространение

Токены — обычные типы, поэтому поддерживают все операции обычных типов:

**Возврат токенов** — токен распространяется вместе с возвращаемым значением:

```yaoxiang
// ✅ Субтокен и родительский токен возвращаются вместе
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()        // Токен возвращается вызывающему
print(px_ref)                    // OK, токен всё ещё в области видимости
```

**Хранение в структурах** — структуры могут содержать поля-токены:

```yaoxiang
// ✅ Структура содержит токен как поле
Window: Type = {
    target: Point,
    view: &Point,              // Поле-токен — держит только представление для чтения на target
}
```

**Захват замыканиями** — замыкания захватывают токены как любые значения:

```yaoxiang
// ✅ Замыкание захватывает токен &Float (тип Dup, свободно копируется в замыкание)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}
```

### 12.4 Автоматический выбор заимствования

Компилятор на стороне вызова автоматически выбирает по следующему приоритету:

```
1. Если фактический аргумент ещё используется в дальнейшем → создаётся токен (&T или &mut T, согласно сигнатуре метода)
2. Если фактический аргумент больше не используется → Move
3. Приоритет匹配: &T < &mut T < Move
```

```yaoxiang
p = Point(1.0, 2.0)
p.print()          // print объявляет &self → компилятор создаёт токен &Point
p.shift(1.0, 1.0)  // shift объявляет &mut self → компилятор создаёт токен &mut Point
p2 = p             // больше не используется → Move
```

### 12.5 Обнаружение конфликтов токенов

Компилятор выполняет **потоково-чувствительный анализ активности** значений токенов, отслеживая состояние каждого токена (активен/перемещён):

```yaoxiang
// ❌ &mut и производный от него &T не могут быть активны одновременно
bad_alias: (p: &mut Point) -> Void = {
    p.x = 10.0                   // ✅ Нормальное использование WriteToken
    print(p.y)
}

// ✅ После окончания области действия токена он автоматически освобождается
good_seq: (p: &mut Point) -> Void = {
    {
        // Внутренняя область видимости
        print(p.x)               // Используется &mut Point
    }
    // Конец внутренней области видимости
    p.x = 10.0                   // ✅ WriteToken всё ещё доступен
}

// ❌ Нельзя одновременно создавать &mut токен и другие токены из одного аргумента
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)                  // ❌ p одновременно получает &mut и & токены
```

### 12.6 Внутреннее устройство компилятора: механизм брендов

Пользователь никогда не сталкивается с брендами. Компилятор внутренне назначает каждому токену компиляционно-уникальный идентификатор:

```
Видимое пользователю           Внутреннее представление компилятора
────────────────────────────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — компиляционно-уникальное целое
&mut Point     →  WriteToken(Point, #M)   // #M — компиляционно-уникальное целое
```

Назначение брендов:
- **Защита от подделки**: токены можно получить только от владеющего объекта, нельзя создать из воздуха
- **Отслеживание происхождения**: производные от доступа к полям `&Float` содержат производный бренд (`#N.field_x`), компилятор может отследить до родительского токена
- **Обнаружение конфликтов**: WriteToken и производный ReadToken из одного источника не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнинга, в сгенерированном машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

### 12.7 Суммарные типы токенов заимствования

```
&BorrowToken ::= &T          // ReadToken (Dup, копируемый)
               | &mut T      // WriteToken (Linear, эксклюзивный)
```

### 12.8 Заимствование токенов vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Что делает | Взглянуть/изменить на месте | Совместное владение |
| Область | В области действия значения токена | Пересекает области видимости |
| Стоимость | Нулевые накладные расходы (тип нулевого размера, исчезает после компиляции) | Rc или Arc (выбирается компилятором) |
| Побег | Возможен (токен распространяется с возвращаемым значением/структурой/замыканием) | Используется специально для побега |
| Межзадачность | Невозможно (токен не реализует Send) | Возможно (компилятор автоматически выбирает Arc) |
| Обнаружение циклов | Не применяется | Внутри задачи — тихо, между задачами — lint |
