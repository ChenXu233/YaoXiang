## Общие положения

### Область применения

Настоящий стандарт применяется ко всему коду тестов на Rust в проекте YaoXiang:

| Тип теста | Расположение | Фреймворк |
|-----------|--------------|-----------|
| Модульный тест | `src/<module>/tests/` | `#[test]` + `#[cfg(test)]` |
| Интеграционный тест | `tests/` | `#[test]` |
| Бенчмарк-тест | `benches/` | Criterion.rs |
| Doctest | Комментарии документации API | `cargo test --doc` |
| Property-based тест | В любом месте тестов | proptest / quickcheck |

### Основные принципы

**Принцип 0: Единственным авторитетным источником для тестов является спецификация, а не код.** Это самый важный принцип данного документа. Тесты проверяют, соответствует ли код спецификации, а не то, «проходит ли код текущую реализацию». Когда тест обнаруживает несоответствие поведения кода спецификации, **исправляется код, тест не изменяется**.

Файлы спецификаций расположены:
- `docs/src/design/language-spec.md` —— Ядро языковой спецификации
- `docs/src/design/rfc/accepted/` —— Принятые проектные документы RFC

В начале каждого файла тестов должна быть указана соответствующая глава спецификации (см. правило 2.1). Любой разработчик должен иметь возможность взять документ спецификации, сверить его с тестами и проверить корректность реализации. И наоборот — если фрагмент кода не имеет соответствующего описания в спецификации, его не должно существовать, и тем более его не следует тестировать.

```rust
// 🟢 Хорошо——тест напрямую ссылается на спецификацию, проверяя соответствие кода
//! Тесты литералов — на основе спецификации языка §2.6
//!
//! §2.6.1: Целочисленные Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: F-String интерполяция

#[test]
fn test_decimal_literal_parsing() {
    // Спецификация §2.6.1: Decimal ::= [0-9][0-9_]*
    let result = parse_literal("42").unwrap();
    assert_eq!(result, Literal::Int(42));
}

// 🔴 Мусор——тест подстраивается под поведение текущего кода, а не проверяет спецификацию
#[test]
fn test_literal_1() {
    // Неизвестно, какой раздел спецификации соответствует этому коду
    // Если parse_literal вернёт неверное значение, этот тест «пройдёт»
    // Потому что он только проверяет, что функция не паникует
    let result = parse_literal("42");
    assert!(result.is_ok());
}
```

**Ситуация**: вы написали тест и обнаружили, что поведение кода не соответствует спецификации. У вас есть два варианта:
| Неправильный подход | Правильный подход |
|---------------------|-------------------|
| Изменить тест, чтобы он «проходил» | Исправить код так, чтобы поведение соответствовало спецификации |
| Добавить в тест `#[ignore]` | Немедленно исправить реализацию кода |
| Добавить в тест условные ветки, подстраивающиеся под код | Удалить ветки, позволив тесту напрямую выявить проблему |

Запомните: **Красный свет = ошибка в коде, а не в тесте.** (Если только сам тест не содержит ошибку — это совсем другое дело.)

**Принцип 1: Тесты являются документацией.** Любой разработчик должен понять поведение тестируемого кода, читая только тесты — без дополнительных комментариев или внешней документации.

```rust
// 🟢 Хорошо——название теста сообщает, что тестируется и чего ожидать
#[test]
fn test_tokenize_empty_input_returns_eof() {
    let tokens = tokenize("").unwrap();
    assert_eq!(tokens.len(), 1);
    assert!(matches!(tokens[0].kind, TokenKind::Eof));
}

// 🔴 Мусор——непонятно, что именно тестируется
#[test]
fn test_tokenize_1() {
    let tokens = tokenize("").unwrap();
    assert!(tokens.len() > 0);
}
```

**Принцип 2: Нулевая терпимость к случайным сбоям.** Тесты должны быть воспроизводимы в любой среде. Тесты, зависящие от случайных чисел, системного времени или порядка планирования потоков, должны использовать фиксированные сиды или заменяться на mock-объекты.

**Принцип 3: Один тест проверяет одно поведение.** Если для названия теста нужно использовать «и» для соединения нескольких действий, разделите их на несколько тестов.

```rust
// 🟢 Хорошо——каждый тест проверяет только один сценарий
#[test]
fn test_parse_int_positive() { /* ... */ }
#[test]
fn test_parse_int_zero() { /* ... */ }

// 🔴 Мусор——один тест перегружен множеством несвязанных проверок
#[test]
fn test_parser() {
    // Тестируем tokenize, parse, typecheck, codegen...
}
```

**Принцип 4: Тестируем поведение, а не реализацию.** Рефакторинг внутренней реализации не должен приводить к падению тестов. Если при изменении одной строки реализации у вас падает 10 тестов — тесты написаны неправильно.

Однако здесь важно различать: **определение «поведения» исходит из спецификации, а не из текущего поведения кода.** Если код изменил поведение (т.е. появилось новое поведение, не соответствующее спецификации), тесты должны падать. Если вы не можете этого достичь — ваши тесты «подстраиваются под код», позволяя багам проникать беспрепятственно.

```
Спецификация (language-spec.md / RFC)  ──определяет──►  Ожидаемое поведение  ──управляет──►  Тесты
                                           │
Текущий код  ──реализует──►  Фактическое поведение  ──сравнивается──►  Результаты тестов

Если фактическое поведение ≠ ожидаемому:
  Тесты должны падать (красный свет)  ──►  Исправление кода  ──►  Тесты проходят (зелёный свет)
  
Если фактическое поведение = ожидаемому (но реализация плохая):
  Тесты проходят  ──►  Рефакторинг реализации  ──►  Тесты всё ещё проходят  ← В этом смысл принципа 4
```

**Принцип 5: Не писать fallback-код / код совместимости / код, зависящий от特定ного режима.** Тестовая среда — это среда, которую вы полностью контролируете. Если вам нужно использовать `#[cfg(not(ci))]` для пропуска теста — это указывает на фундаментальную проблему в дизайне теста.

### Определения терминов

| Термин | Определение |
|--------|-------------|
| Модульный тест | Тестирует поведение одной функции или модуля без зависимости от внешних систем |
| Интеграционный тест | Тестирует взаимодействие нескольких модулей через публичный API или интерфейс командной строки |
| Бенчмарк-тест | Измеряет производительность кода, обнаруживает регрессию производительности |
| Doctest | Исполняемые примеры кода, встроенные в комментарии документации |
| Property-based тест | Тесты, проверяющие инварианты (properties) на основе случайных входных данных |

### Связь со стандартами коммитов

Все коммиты, связанные с тестами, должны использовать тип `:white_check_mark: test:`, согласно [стандартам коммитов](./commit-convention.md).

```
:white_check_mark: test(parser): добавить тесты инфиксных выражений для Pratt-парсера
:white_check_mark: test(codegen): дополнить тесты генерации IR для оператора switch
```

## Стандарты модульных тестов

### Организация файлов

**Правило 1.1**: Каталог `tests/` для модульных тестов должен располагаться **на том же уровне**, что и `mod.rs` тестируемого модуля. `tests/` не агрегируется вверх и не суммируется на несколько уровней.

```
src/frontend/core/parser/
├── mod.rs              # #[cfg(test)] mod tests; ——объявляет tests/ на том же уровне
├── ast.rs
├── pratt/
│   ├── mod.rs          # #[cfg(test)] mod tests; ——тесты внутри pratt
│   └── tests/
│       ├── mod.rs
│       ├── led.rs
│       ├── nud.rs
│       └── precedence.rs
└── tests/              # Тесты уровня модуля parser (не включают подмодули pratt)
    ├── mod.rs
    ├── ast.rs
    ├── expressions.rs
    ├── error_recovery.rs
    └── parser_state.rs
```

Ключевой критерий: **`tests/` размещается в каком каталоге — в `mod.rs` этого каталога должен быть объявлен `#[cfg(test)] mod tests;`.**

#### Правила размещения тестов для однофайловых модулей и модулей-каталогов

**Основное различие**: Организация модуля определяет расположение тестов.

| Тип модуля | Критерий | Расположение тестов | Пример |
|------------|----------|---------------------|--------|
| **Модуль-каталог** | Имеет отдельный каталог и `mod.rs` | `tests/` внутри этого каталога | `inference/tests/` |
| **Однофайловый модуль** | Только `.rs` файл, без отдельного каталога | `tests/` родительского модуля | `overload.rs` → `typecheck/tests/overload.rs` |

**Подробное объяснение**:

```
src/frontend/core/typecheck/
├── mod.rs                          # mod.rs модуля typecheck
├── checker.rs                      # Однофайловый модуль
├── environment.rs                  # Однофайловый модуль
├── overload.rs                    # Однофайловый модуль
├── type_eval.rs                    # Однофайловый модуль
├── dead_code.rs                    # Однофайловый модуль
├── spawn_placement.rs              # Однофайловый модуль
├── signature.rs                    # Однофайловый модуль
├── types.rs                        # Однофайловый модуль
│
├── tests/                          # ✅ Каталог тестов typecheck
│   ├── mod.rs                      # Объявление тестов однофайловых модулей
│   ├── checker.rs                  # Тесты checker.rs
│   ├── environment.rs              # Тесты environment.rs
│   ├── overload.rs                 # Тесты overload.rs (тесты однофайлового модуля здесь)
│   ├── type_eval.rs                # Тесты type_eval.rs
│   ├── dead_code.rs               # Тесты dead_code.rs
│   ├── spawn_placement.rs         # Тесты spawn_placement.rs
│   ├── signature.rs                # Тесты signature.rs
│   └── types.rs                    # Тесты types.rs
│
├── inference/                      # Модуль-каталог (есть mod.rs)
│   ├── mod.rs                      # #[cfg(test)] mod tests; ——объявляет tests/ на том же уровне
│   ├── expressions.rs
│   ├── statements.rs
│   ├── patterns.rs
│   ├── bounds.rs
│   ├── subtyping.rs
│   ├── generics.rs
│   ├── compatibility.rs
│   ├── scope.rs
│   ├── assignment.rs
│   └── tests/                      # ✅ Каталог тестов inference
│       ├── mod.rs
│       ├── expressions.rs          # Тесты expressions.rs
│       ├── statements.rs           # Тесты statements.rs
│       └── ...
│
└── traits/                         # Модуль-каталог (есть mod.rs)
    ├── mod.rs                      # #[cfg(test)] mod tests; ——объявляет tests/ на том же уровне
    ├── solver.rs
    ├── impl_check.rs
    ├── inheritance.rs
    ├── coherence.rs
    ├── auto_derive.rs

    ├── gat/

    └── tests/                      # ✅ Каталог тестов traits
        ├── mod.rs

        ├── impl_check.rs           # Тесты impl_check.rs
        └── ...
```

**Почему тесты однофайловых модулей располагаются в родительском `tests/`?**

Потому что однофайловый модуль (например, `overload.rs`) не имеет собственного `mod.rs`, и поэтому не может объявить `#[cfg(test)] mod tests;`. Согласно системы модулей Rust, тестовые файлы должны быть объявлены в каком-либо `mod.rs` для компиляции. Следовательно, тесты однофайлового модуля могут быть объявлены только родительским `mod.rs` и располагаться в родительском каталоге `tests/`.

**Алгоритм принятия решения**:

```
Встретили модуль — куда поместить тесты?
│
├── Это каталог-модуль (есть mod.rs)?
│   └── Да → Создать tests/ в этом каталоге, объявить в mod.rs этого каталога
│
├── Это однофайловый модуль (только .rs)?
│   └── Да → Тесты в tests/ родительского каталога, объявить в родительском mod.rs
│
└── Не уверены?
    └── Проверить наличие отдельного каталога и mod.rs
```

**Типичные ошибки**:

```
# ❌ Ошибка 1: Создание отдельного каталога tests/ для однофайлового модуля
src/frontend/core/typecheck/
├── overload.rs
└── overload/                       # ❌ Не нужно создавать каталог для однофайлового модуля
    └── tests/
        └── overload.rs

# ❌ Ошибка 2: Объявление #[cfg(test)] mod tests; внутри однофайлового модуля
# overload.rs
#[cfg(test)]                        # ❌ Однофайловый модуль не может так объявлять
mod tests;                          # Потому что нет каталога overload/tests/

# ✅ Правильно: тесты в родительском tests/
src/frontend/core/typecheck/
├── overload.rs                     # Исходный файл
└── tests/
    └── overload.rs                 # Тестовый файл, объявлен в typecheck/mod.rs
```

⚠️ **Антипаттерн — не делайте так:**

```
# ❌ Ошибка: Тесты подмодулей собраны в родительском каталоге
src/frontend/core/types/
├── mod.rs              # Должен объявлять только base и computation
├── base/
│   ├── mod.rs
│   └── var.rs
└── tests/              # ❌ Родительский tests/ содержит тесты подмодулей
    ├── mod.rs          # ❌ Вынужден объявлять mod base; mod computation;
    ├── base/           # ❌ Это должно быть в base/tests/
    │   └── var.rs
    └── computation/    # ❌ Это должно быть в computation/tests/
        └── ...
```

```
# ✅ Правильно: Тесты каждого модуля независимы
src/frontend/core/types/
├── mod.rs              # Объявляет только pub mod base; pub mod computation;
├── base/
│   ├── mod.rs          # #[cfg(test)] mod tests; ——объявляет tests/ на том же уровне
│   ├── var.rs
│   └── tests/
│       ├── mod.rs
│       └── var.rs
└── computation/
    ├── mod.rs          # #[cfg(test)] mod tests; ——объявляет tests/ на том же уровне
    ├── operations.rs
    └── tests/
        ├── mod.rs
        └── operations.rs
```

**Почему нельзя агрегировать вверх?** Потому что система модулей Rust требует, чтобы `#[cfg(test)] mod tests;` решал вопрос компиляции тестовых файлов в точке объявления. Если `types/mod.rs` объявляет `mod tests;`, то содержимое `types/tests/` становится приватным содержимым модуля `types` — оно не должно заходить на территорию `base` или `computation`. Тесты каждого модуля являются деталями реализации этого модуля, а не родительского. Это правило также применимо при рефакторинге модулей: когда вы разделяете `types` на `base` и `computation`, тесты также должны следовать за разделёнными модулями, а не оставаться на месте. **Каталог тестов не зеркалирует структуру исходного кода, а следует за границами модулей.**

**Правило 1.2**: `tests/mod.rs` отвечает только за объявление модулей и re-export, тестовые функции не содержатся.

```rust
//! Тесты ядра парсера — зеркало src/frontend/core/parser/
//!
//! Тесты для ast.rs, parser_state.rs и парсинга выражений/интеграции.

mod ast;
mod error_recovery;
mod expressions;
mod integration;
mod parser_state;
```

**Правило 1.3**: Каждый тестовый файл соответствует только одному исходному файлу. Недопустимо смешивать тесты нескольких исходных модулей в одном файле.

**Правило 1.4**: `#[cfg(test)]` должен появляться только в двух местах — объявление `mod tests` в `lib.rs` или inline-объявление `#[cfg(test)] mod tests;` внутри тестируемого исходного файла. В других местах использование запрещено.

```rust
// src/frontend/core/parser/mod.rs или lib.rs
#[cfg(test)]
mod tests;
```

### Стандарты объявления модулей

**Правило 2.1**: В начале всех тестовых файлов должны быть модульные doc-комментарии `//!`, указывающие источник спецификации (номер раздела языковой спецификации + номер RFC). Если тест не ссылается ни на один раздел спецификации — этот код не имеет обоснования в спецификации и не должен существовать.

```rust
//! Тесты литералов — на основе спецификации языка §2.6
//!
//! §2.6.1: Целочисленные Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: F-String интерполяция
```

**Почему обязательна ссылка на спецификацию?** Потому что ожидаемые значения тестов происходят из спецификации, а не из «текущего вывода кода». Если有一天 код изменит вывод, а тест随之更新 — такой тест ничего не защищает. Только тесты, привязанные к спецификации, могут отличить «намеренное breaking change» от «непреднамеренной регрессии».

**Правило 2.2**: Импорты `use` в тестовых модулях должны быть точными — до конкретного типа/функции, запрещены glob-импорты `use super::*`.

```rust
// 🟢 Хорошо——точный импорт
use crate::frontend::core::lexer::{tokenize, TokenKind};
use crate::frontend::core::parser::{ParserState, ParseError};

// 🔴 Мусор——непонятно, что именно тестируется
use super::*;
```

### Стандарты именования

**Правило 3.1**: Формат имени тестовой функции: `test_<что>_<сценарий>`, все строчные буквы, разделение подчёркиванием.

```rust
#[test]
fn test_tokenize_empty_string() { /* ... */ }
#[test]
fn test_parse_int_overflow() { /* ... */ }
#[test]
fn test_typecheck_fn_return_mismatch() { /* ... */ }
```

**Правило 3.2**: Имя тестовой функции должно быть самодокументирующимся. Прочитав имя функции, должно быть понятно, что тестируется и чего ожидать. Запрещено именование числовыми порядковыми номерами.

```rust
// 🟢 Хорошо
fn test_skip_semicolon_success() { /* ... */ }
fn test_skip_semicolon_failure_when_identifier() { /* ... */ }

// 🔴 Мусор——совершенно непонятно, что тестируется
fn test_skip_1() { /* ... */ }
fn test_skip_2() { /* ... */ }
```

**Правило 3.3**: Вспомогательным функциям не нужен префикс `test_`, следует использовать глаголы или существительные, описывающие их назначение.

```rust
fn parse_expr(source: &str) -> Expr { /* ... */ }
fn tokenize_single(source: &str) -> Token { /* ... */ }
fn setup_parser_with_tokens(tokens: &[Token]) -> ParserState { /* ... */ }
```

### Стандарты структуры тестов (Arrange-Act-Assert)

**Правило 4.1**: Каждая тестовая функция должна следовать трёхчастной структуре: Подготовка (Arrange) → Исполнение (Act) → Утверждение (Assert), между частями — пустая строка.

```rust
#[test]
fn test_parse_binary_addition() {
    // Arrange
    let source = "1 + 2";

    // Act
    let expr = parse_expr(source);

    // Assert
    assert!(matches!(expr, Expr::Binary { op: BinOp::Add, .. }));
}
```

**Правило 4.2**: Простые тесты (одиночный вызов + одиночное утверждение) могут не иметь текстовых комментариев к частям, но не должны превышать 5 строк логического кода. Тесты, превышающие 5 строк, должны явно размечать три части.

### Стандарты вспомогательных функций

**Правило 5.1**: Повторяющийся 3 и более раз код setup-а должен быть извлечён во вспомогательную функцию.

```rust
// 🟢 Хорошо——извлечён общий setup
fn with_state<F>(source: &str, mut f: F)
where
    F: FnMut(&mut ParserState<'_>),
{
    let tokens = tokenize(source).unwrap();
    let mut state = ParserState::new(&tokens);
    f(&mut state);
}

#[test]
fn test_current_returns_first_token() {
    with_state("42", |state| {
        let tok = state.current();
        assert_eq!(&tok.unwrap().kind, &TokenKind::IntLiteral(42));
    });
}
```

**Правило 5.2**: `unwrap()` / `expect()` во вспомогательных функциях должны при панике выводить достаточный контекст. В теле тестовой функции (`#[test] fn ...`) можно использовать `unwrap()` напрямую — при падении Rust автоматически выводит номер строки; но при падении внутри вспомогательной функции номер строки указывает на определение вспомогательной функции, без контекста вызова.

```rust
// 🟢 Хорошо——при падении выводится содержимое исходного кода
fn run_ok(source: &str) {
    run(source).unwrap_or_else(|e| panic!("Execution failed:\nSource:\n{}\nError:\n{:?}", source, e));
}

// 🔴 Мусор——при падении непонятно, какой исходный файл вызвал проблему
fn run_ok(source: &str) {
    run(source).unwrap();
}
```

**Правило 5.3**: Вспомогательные функции следует размещать в верхней части тестового файла, сразу после импортов `use`. Если они используются несколькими тестовыми модулями — поместите их в `tests/mod.rs` и экспортируйте через `pub(crate)`.

### Стиль утверждений

**Правило 6.1**: Для сопоставления вариантов enum-а предпочтительно использовать `assert!(matches!(...))`, не допускается использование `if let` + `panic!`.

```rust
// 🟢 Хорошо
assert!(matches!(tokens[0].kind, TokenKind::IntLiteral(42)));

// 🔴 Мусор
if let TokenKind::IntLiteral(v) = tokens[0].kind {
    assert_eq!(v, 42);
} else {
    panic!("Expected IntLiteral");
}
```

**Правило 6.2**: Для точного сравнения значений используйте `assert_eq!`, для булевых утверждений — `assert!`. Запрещено использовать `assert!(a == b)` вместо `assert_eq!(a, b)`.

**Правило 6.3**: Все утверждения должны содержать пользовательское сообщение об ошибке, если само утверждение не описывает причину падения полностью.

```rust
// 🟢 Хорошо——при падении утверждения можно быстро локализовать
assert!(
    state.infix_info().is_some(),
    "infix_info should handle '{op}'"
);

// 🟢 Хорошо——при падении assert_eq! автоматически выводит разницу значений, доп. сообщение не нужно
assert_eq!(error_count, 0);

// 🔴 Мусор——при падении只知道 "assertion failed"
assert!(state.infix_info().is_some());
```

**Правило 6.4**: Порядок в утверждениях должен быть `assert_eq!(actual, expected)` — фактическое значение первым, ожидаемое вторым.

### Чек-лист антипаттернов

Ниже приведены запрещённые写法及其替代方案:

| Антипаттерн | Проблема | Альтернатива |
|-------------|----------|--------------|
| Тест подстраивается под ошибочное поведение кода | Маскирует отклонение от спецификации, легитимизирует баг | Сверить со спецификацией и исправить код, тест не менять |
| Обратный вывод ожидаемых значений из вывода кода | Тест становится «диктофоном текущей реализации» | Выводить ожидаемые значения из спецификации |
| Постоянная метка `#[ignore]` | Скрывает устаревшие тесты | Исправить или удалить |
| Отладочный вывод `println!` | Загрязняет вывод тестов | Использовать `assert!` для явных утверждений |
| `thread::sleep` в тестах | Случайные сбои + медленно | Использовать синхронизацию или mock |
| Работа с реальной файловой системой в тестах | Медленно и невоспроизводимо | Использовать `tempfile` |
| Зависимость от порядка выполнения тестов | Случайные сбои | Независимый setup для каждого теста |
| Одна тестовая функция более 30 строк логики | Невозможно понять | Разделить тесты или использовать вспомогательные функции |
| `unwrap()` во вспомогательных функциях без контекста | Сложно локализовать | Использовать `expect("why")` или кастомный panic (см. правило 5.2) |
| copy-paste одного и того же setup 3+ раза | Высокая стоимость изменений | Извлечь во вспомогательную функцию |

## Стандарты интеграционных тестов

### Организация тестов

**Правило 7.1**: Интеграционные тесты располагаются в каталоге `tests/` корня проекта. Входной файл `tests/integration.rs` использует атрибут `#[path]` для подключения подмодулей.

```rust
// tests/integration.rs
#[path = "integration/backends.rs"]
mod backends;
#[path = "integration/codegen.rs"]
mod codegen;
#[path = "integration/execution.rs"]
mod execution;
```

**Правило 7.2**: Каждый файл `tests/integration/*.rs` соответствует одной теме тестирования (бэкенды компилятора, кодогенерация, исполнитель и т.д.), смешивание запрещено.

**Правило 7.3**: Интеграционные тесты должны проходить через публичный API проекта. Запрещено в интеграционных тестах напрямую ссылаться на внутренние модули `crate::`. Используйте публичный путь `yaoxiang::`.

```rust
// 🟢 Хорошо——через публичный API
use yaoxiang::run;

// 🔴 Мусор——обход границы публичного API
use yaoxiang::middle::codegen::bytecode::BytecodeFile;
```

### Управление тестовыми данными

**Правило 8.1**: В интеграционных тестах предпочтительно использовать inline-строки с исходным кодом. Внешние fixture-файлы (в `tests/fixtures/`) используются только когда исходный код превышает 30 строк.

```rust
#[test]
fn test_fibonacci() {
    run_ok(
        r#"
        main = {
            mut a = 0
            mut b = 1
            while a < 100 {
                mut next = a + b
                a = b
                b = next
            }
        }
        "#,
    );
}
```

**Правило 8.2**: Fixture-файлы должны иметь расширение `.yx`, имя файла описывает намерение теста.

### Принципы E2E-покрытия

**Правило 9.1**: Интеграционные тесты для каждой языковой features должны покрывать три路径:

| Путь | Описание |
|------|----------|
| Happy path | Корректный вход produces ожидаемый выход |
| Error path | Некорректный вход produces чёткое сообщение об ошибке (не panic) |
| Boundary | Граничные значения (пустой ввод, максимальные значения, предел глубины вложенности) |

**Правило 9.2**: Интеграционные тесты не должны зависеть от сети, переменных окружения или внешних сервисов.

## Стандарты бенчмарк-тестов

### Стандарты использования Criterion.rs

**Правило 10.1**: Бенчмарк-тесты统一 размещаются в каталоге `benches/`, входной файл `benches/lib.rs`. Разделение по темам тестирования.

```
benches/
├── lib.rs              # Входной файл, определяет criterion_group/criterion_main
├── lang_compare/
│   └── fibonacci.rs    # Межъязыковое сравнение бенчмарков
├── parser.rs           # Бенчмарки парсера
└── codegen.rs          # Бенчмарки кодогенерации
```

**Правило 10.2**: Каждая бенчмарк-функция должна содержать модульный doc-комментарий `//!` с описанием цели теста и измеряемых показателей.

```rust
//! Бенчмарки производительности интерпретатора YaoXiang
//!
//! Измеряемый показатель: время одной итерации (wall time)
//! Базовый уровень: нативная реализация на Rust
```

### Защита от оптимизации компилятора

**Правило 11.1**: Все выходные данные бенчмарк-тестов должны проходить через `criterion::black_box` для предотвращения elimination оптимизатором компилятора.

```rust
use criterion::{black_box, Criterion};

fn bench_parse(c: &mut Criterion) {
    c.bench_function("parse_fib", |b| {
        b.iter(|| {
            let result = parse(black_box(FIB_SOURCE));
            black_box(result)
        })
    });
}
```

**Правило 11.2**: Входные данные бенчмарк-тестов должны быть `const` или `lazy_static`, не должны генерироваться внутри замыкания `iter` — иначе измеряется суммарное время генерации данных и исполнения тестируемой логики.

### Группировка и именование бенчмарков

**Правило 12.1**: Формат имени бенчмарк-функции: `<тестируемый_модуль>_<сценарий>`, все строчные буквы, разделение подчёркиванием. Согласуется с правилами именования модульных тестов.

**Правило 12.2**: Необходимо использовать `criterion_group!` для логической группировки связанных бенчмарков. Запрещено把所有 бенчмарки挤在一个分组中.

```rust
criterion_group!(parser, bench_parse_expr, bench_parse_stmt);
criterion_group!(codegen, bench_codegen_module, bench_codegen_switch);
criterion_main!(parser, codegen);
```

## Стандарты doctest-ов

### Сценарии использования

**Правило 13.1**: Все `pub` функции, типы, методы должны содержать в doc-комментариях хотя бы один исполняемый пример кода. Этот пример выполняется через `cargo test --doc`.

```rust
/// Разбивает исходную строку на последовательность токенов.
///
/// ```
/// use yaoxiang::frontend::core::lexer::tokenize;
///
/// let tokens = tokenize("42").unwrap();
/// assert_eq!(tokens.len(), 2); // IntLiteral + Eof
/// ```
pub fn tokenize(source: &str) -> Result<Vec<Token>, LexError> {
    // ...
}
```

**Правило 13.2**: Примеры кода в doctest должны компилироваться и успешно проходить утверждения. Не допускаются примеры с меткой `ignore`, за исключением случаев демонстрации compile-time ошибок.

```rust
/// ```ignore
/// // Демонстрация compile-time ошибки——можно ignore
/// let x: int = "string";
/// ```
```

### Требования к покрытию

**Правило 14.1**: Doctest покрывает только happy path API. Граничные случаи и пути ошибок покрываются модульными тестами.

**Правило 14.2**: Примеры кода в doctest должны быть лаконичными — не более 10 строк. Если пример требует более длинного контекста, это указывает на проблему в дизайне API.

## Стандарты property-based тестов

### Сценарии использования

**Правило 15.1**: Следующие сценарии должны использовать property-based тесты (proptest или quickcheck) вместо ручного написания нескольких граничных значений:

| Сценарий | Пример |
|----------|--------|
| Parser round-trip | `parse(pretty_print(ast)) == ast` |
| Сериализация/десериализация | `deserialize(serialize(data)) == data` |
| Математические тождества | `a + b == b + a` |
| Компиляторная оптимизация без изменения семантики | `eval(code) == eval(optimize(code))` |

**Правило 15.2**: Для property-based тестов используется `proptest`作为主要框架（已 объявлен в `dev-dependencies` в `Cargo.toml`）。

```rust
use proptest::prelude::*;

proptest! {
    #[test]
    fn test_roundtrip_serialize_deserialize(value: i64) {
        let serialized = serialize(&value);
        let deserialized: i64 = deserialize(&serialized).unwrap();
        prop_assert_eq!(deserialized, value);
    }
}
```

### Принципы определения свойств

**Правило 16.1**: Каждый property-based тест должен иметь явное объявление свойства — в комментарии указывается проверяемый инвариант.

```rust
// Свойство: любой целочисленный литерал после tokenize → tokens_to_string produces то же значение
proptest! {
    #[test]
    fn test_int_literal_roundtrip(n in any::<i64>()) {
        let source = n.to_string();
        let tokens = tokenize(&source).unwrap();
        // ...
    }
}
```

**Правило 16.2**: Если property-based тест обнаруживает падение, необходимо использовать механизм регрессии `proptest` — добавить упавшие входные данные в каталог `proptest-regressions/`, не заменять обычным тестом.

## Требования к покрытию

### Цели покрытия для нового кода

**Правило 17.1**: Требования к покрытию тестами для新增代码:

| Тип кода | Покрытие строк | Покрытие ветвлений |
|----------|----------------|---------------------|
| Основные модули компилятора (frontend/middle/backends) | ≥ 85% | ≥ 80% |
| Утилитарные модули (util) | ≥ 75% | ≥ 70% |
| Модули runtime (vm/runtime) | ≥ 80% | ≥ 75% |
| Стандартная библиотека (std) | ≥ 75% | ≥ 70% |
| Обработка ошибок и диагностика | ≥ 90% | ≥ 85% |

**Правило 17.2**: Пути обработки ошибок (все ветки `Err`) должны быть покрыты на 100%. Сообщения об ошибках, видимые пользователю, должны быть проверены тестами.

### Чек-лист проверки PR

**Правило 18.1**: Перед提交 PR автор должен самопроверить следующие пункты:

- [ ] `cargo test`全部通过
- [ ] `cargo test --doc`全部通过
- [ ] `cargo bench`无性能回归（如涉及热路径变更）
- [ ] 新增代码符合覆盖率目标
- [ ] 测试命名符合命名规范
- [ ] 每个测试文件声明了对应的规范章节（规则 2.1）
- [ ] 测试期望值来自规范定义，而非"当前代码的输出"
- [ ] 无 `#[ignore]` 标记的测试（除非有明确 issue 号注释）
- [ ] 无不必要的 `unwrap()` （应使用 `expect` 或自定义 panic 消息）
- [ ] 提交信息使用 `:white_check_mark: test:` 类型
- [ ] **没有因为"代码行为与规范不符"而修改测试期望值——改的是代码，不是测试**

**Правило 18.2**: Reviewer 必须拒绝包含以下问题的 PR:

- 只有 happy path 测试，缺少错误路径
- 测试中有 `thread::sleep` 或依赖执行顺序
- 复制粘贴的测试代码超过 3 次而未提取辅助函数
- 测试名不符合命名规范
- 存在永久 `#[ignore]` 的测试
- **测试迁就代码的错误行为**（代码与规范不符时修改测试而非修改代码）
- **测试没有声明对应的规范章节**（参见规则 2.1）
- **测试期望值来自代码输出而非规范定义**（反推出来的测试等于没测）
- 测试只验证"不 panic"而不断言具体行为
- 删除了暴露代码 bug 的失败测试（而不是修复代码后再看到它变绿）

## Приложения

### A. Шпаргалка по командам тестирования

```bash
# 运行所有测试
cargo test

# 只运行单元测试
cargo test --lib

# 只运行集成测试
cargo test --test integration

# 只运行文档测试
cargo test --doc

# 运行特定测试（按名称过滤）
cargo test test_parse_expr

# 运行基准测试
cargo bench

# 显示测试输出（默认隐藏 stdout）
cargo test -- --nocapture

# 单线程运行（排查并发问题）
cargo test -- --test-threads=1

# 生成覆盖率报告（需要 cargo-llvm-cov）
cargo llvm-cov --html
```

### B. Шаблон сообщения коммита

Тестовые коммиты должны следовать следующему шаблону:

```
:white_check_mark: test(<scope>): <краткое описание>

<Опционально: список покрываемых сценариев>
```

Пример:

```
:white_check_mark: test(parser): добавить тесты инфиксных операторов для Pratt-парсера

Покрываемые сценарии:
- Приоритет арифметических операторов (+, -, *, /, %)
- Связывание операторов сравнения (1 < x < 10)
- Short-circuit логических операторов
- Правоассоциативность операторов присваивания
```

### C. Чек-лист новых тестовых файлов

При создании нового тестового модуля убедитесь, что包含以下文件:

```
# 新增测试模块时包含的文件
src/<module>/tests/
├── mod.rs          # 模块声明 + 公共辅助函数
└── <subject>.rs    # 测试文件，对应被测源文件命名

# 新增集成测试时包含的文件
tests/
├── integration.rs   # 更新：添加 #[path] 声明
└── integration/
    └── <topic>.rs   # 新测试文件
```

> 💡 **Запомните**: Тесты не проверяют, работает ли ваш код — они проверяют, соответствует ли ваш код спецификации. Спецификация меняется, тесты следуют за спецификацией. Код написан неправильно — исправьте код, не меняйте тесты. **Код служит спецификации, тесты защищают спецификацию. Момент, когда тесты начинают подстраиваться под код, вы теряете всю защиту.**
