## Общие положения

### Область применения

Настоящий стандарт применяется ко всему коду тестов на Rust в проекте YaoXiang, включая:

| Тип теста | Расположение | Фреймворк |
|-----------|--------------|-----------|
| Модульные тесты | `src/<module>/tests/` | `#[test]` + `#[cfg(test)]` |
| Интеграционные тесты | `tests/` | `#[test]` |
| Бенчмарки | `benches/` | Criterion.rs |
| doctest | Комментарии API-документации | `cargo test --doc` |
| Property-тесты | В любом месте тестов | proptest / quickcheck |

### Ключевые принципы

**Принцип 0: Единственным авторитетным источником для тестов является спецификация, а не код.** Это самый важный принцип данного документа. Тесты проверяют, соответствует ли код спецификации, а не то, «работает ли код так, как реализовано сейчас». Когда тест обнаруживает несоответствие поведения кода спецификации, **исправляется код, тест никогда не корректируется**.

Файлы спецификаций расположены:
- `docs/src/design/language-spec.md` —— ядро спецификации языка
- `docs/src/design/rfc/accepted/` —— принятые RFC-документы проектирования

В начале каждого файла тестов должна быть указана соответствующая глава спецификации (см. правило 2.1). Любой разработчик должен иметь возможность сверить тесты со спецификацией и убедиться в корректности реализации. Обратное же — если код не имеет соответствующего описания в спецификации, его не должно существовать, и тем более его не следует тестировать.

```rust
// 🟢 Хорошо — тест напрямую ссылается на спецификацию, проверяя соответствие кода
//! Тесты литералов — на основе спецификации языка §2.6
//!
//! §2.6.1: Целочисленные Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: Интерполяция F-String

#[test]
fn test_decimal_literal_parsing() {
    // Спецификация §2.6.1: Decimal ::= [0-9][0-9_]*
    let result = parse_literal("42").unwrap();
    assert_eq!(result, Literal::Int(42));
}

// 🔴 Мусор — тест подстраивается под текущее поведение кода, а не проверяет спецификацию
#[test]
fn test_literal_1() {
    // Неизвестно, к какой части спецификации относится этот код
    // Если parse_literal вернёт неверное значение, тест «пройдёт»
    // Потому что он лишь проверяет, что функция не паникует
    let result = parse_literal("42");
    assert!(result.is_ok());
}
```

**Ситуация**: вы написали тест и обнаружили, что поведение кода не соответствует спецификации. У вас есть два варианта:
| Неправильный подход | Правильный подход |
|---------------------|-------------------|
| Изменить тест, чтобы он «проходил» | Исправить код, чтобы поведение соответствовало спецификации |
| Добавить в тест `#[ignore]` | Немедленно исправить код |
| Добавить в тест условные ветки под существующий код | Удалить ветки, чтобы тест напрямую暴露ил проблему |

Запомните: **Красный тест = ошибка в коде, а не в тесте.** (Если только сам тест не содержит ошибку — это совсем другое дело.)

**Принцип 1: Тесты являются документацией.** Любой разработчик должен понять поведение тестируемого кода, читая тесты — без дополнительных комментариев или внешней документации.

```rust
// 🟢 Хорошо — имя теста описывает, что тестируется и что ожидается
#[test]
fn test_tokenize_empty_input_returns_eof() {
    let tokens = tokenize("").unwrap();
    assert_eq!(tokens.len(), 1);
    assert!(matches!(tokens[0].kind, TokenKind::Eof));
}

// 🔴 Мусор — непонятно, что именно тестируется
#[test]
fn test_tokenize_1() {
    let tokens = tokenize("").unwrap();
    assert!(tokens.len() > 0);
}
```

**Принцип 2: Нулевая терпимость к случайным сбоям.** Тесты должны быть воспроизводимыми в любом окружении. Тесты, зависящие от случайных чисел, системного времени или порядка планирования потоков, должны использовать фиксированные seed или mock-объекты.

**Принцип 3: Один тест проверяет одно поведение.** Если для описания имени теста нужно использовать «и», разделите его на несколько тестов.

```rust
// 🟢 Хорошо — каждый тест проверяет один сценарий
#[test]
fn test_parse_int_positive() { /* ... */ }
#[test]
fn test_parse_int_zero() { /* ... */ }

// 🔴 Мусор — в одном тесте слишком много разнородного содержимого
#[test]
fn test_parser() {
    // Тестируем tokenize, parse, typecheck, codegen...
}
```

**Принцип 4: Тестируем поведение, а не реализацию.** Рефакторинг внутренней реализации не должен приводить к падению тестов. Если изменение одной строки реализации привело к падению 10 тестов — тесты написаны неправильно.

Но здесь важно различие: **определение «поведения» исходит из спецификации, а не из текущего поведения кода.** Если код изменил поведение (то есть появилось новое поведение, не соответствующее спецификации), тесты должны падать. Если тест не может этого обеспечить — это «тест, подстраивающийся под код», и он пропускает баги.

```
Спецификация (language-spec.md / RFC)  ──определяет──►  Ожидаемое поведение  ──управляет──►  Тесты
                                                   │
Текущий код  ──реализует──►  Фактическое поведение  ──сравнивается──►  Результат теста

Если фактическое ≠ ожидаемое:
  Тест должен упасть (красный)  ──►  Исправляем код  ──►  Тест проходит (зелёный)
  
Если фактическое = ожидаемое (но реализация плохая):
  Тест проходит  ──►  Рефакторим реализацию  ──►  Тест всё ещё проходит  ← Вот смысл принципа 4
```

**Принцип 5: Не пишите fallback- / compat- / специфичные для режима тестов.** Тестовое окружение — это среда, которую вы полностью контролируете. Если вам нужен `#[cfg(not(ci))]` для пропуска теста — это говорит о фундаментальной проблеме в дизайне теста.

### Определения терминов

| Термин | Определение |
|--------|-------------|
| Модульный тест | Тестирование поведения отдельной функции или модуля без зависимостей от внешних систем |
| Интеграционный тест | Тестирование взаимодействия нескольких модулей через публичные API или CLI |
| Бенчмарк | Измерение производительности кода, обнаружение регрессий |
| doctest | Исполняемые примеры кода, встроенные в документацию |
| Property-тест | Тестирование инвариантов (property) на основе случайных входных данных |

### Связь со стандартами коммитов

Все коммиты, связанные с тестами, должны использовать тип `:white_check_mark: test:`, согласно [стандартам коммитов](./commit-convention.md).

```
:white_check_mark: test(parser): добавить тесты инфиксных выражений в парсере Pratt
:white_check_mark: test(codegen): дополнить тесты генерации IR для switch
```

## Стандарты модульных тестов

### Организация файлов

**Правило 1.1**: Каталог `tests/` для модульных тестов должен располагаться **на том же уровне**, что и `mod.rs` тестируемого модуля. `tests/` не агрегируется вверх и не объединяется на разных уровнях.

```
src/frontend/core/parser/
├── mod.rs              # #[cfg(test)] mod tests; —— объявляет tests/ того же уровня
├── ast.rs
├── pratt/
│   ├── mod.rs          # #[cfg(test)] mod tests; —— собственные тесты pratt
│   └── tests/
│       ├── mod.rs
│       ├── led.rs
│       ├── nud.rs
│       └── precedence.rs
└── tests/              # Тесты уровня модуля parser (без содержимого подмодуля pratt)
    ├── mod.rs
    ├── ast.rs
    ├── expressions.rs
    ├── error_recovery.rs
    └── parser_state.rs
```

Ключевой критерий: **куда помещён `tests/`, там же в `mod.rs` должен быть объявлен `#[cfg(test)] mod tests;`.**

**Правило 1.1 (дополнение): Запрет на агрегацию вверх.** Тесты подмодулей должны располагаться в собственном `tests/` этого подмодуля, а не агрегироваться в родительский `tests/`.

| Тип модуля | Расположение тестов | Пример |
|------------|---------------------|--------|
| Каталожный модуль (с `mod.rs`) | `tests/` в этом каталоге | `emitter/tests/`, `codes/tests/` |
| Однофайловый модуль (только `.rs`) | `tests/` родительского каталога | `session.rs` → `diagnostic/tests/session.rs` |

```text
# ✅ Правильно: тесты каждого каталожного модуля независимы
src/util/diagnostic/
├── codes/
│   ├── mod.rs              # #[cfg(test)] mod tests;
│   └── tests/              # ✅ собственные тесты codes
│       ├── mod.rs
│       └── codes.rs
├── emitter/
│   ├── mod.rs              # #[cfg(test)] mod tests;
│   └── tests/              # ✅ собственные тесты emitter
│       ├── mod.rs
│       ├── text.rs
│       └── ansi.rs
└── tests/                  # ✅ уровень diagnostic (однофайловые модули)
    ├── mod.rs
    ├── session.rs
    ├── suggest.rs
    └── collect.rs

# ❌ Неправильно: тесты emitter и codes агрегированы в diagnostic/tests/
src/util/diagnostic/
└── tests/
    ├── mod.rs              # ❌ вынуждены объявлять mod emitter; mod codes;
    ├── emitter/            # ❌ должны быть в emitter/tests/
    └── codes/              # ❌ должны быть в codes/tests/
```

#### Правила размещения тестов для однофайловых vs каталожных модулей

**Ключевое различие**: форма организации модуля определяет место тестов.

| Тип модуля | Критерий | Расположение тестов | Пример |
|------------|----------|---------------------|--------|
| **Каталожный модуль** | Есть отдельный каталог и `mod.rs` | `tests/` в этом каталоге | `inference/tests/` |
| **Однофайловый модуль** | Только файл `.rs`, без отдельного каталога | `tests/` родительского модуля | `overload.rs` → `typecheck/tests/overload.rs` |

**Подробное описание**:

```
src/frontend/core/typecheck/
├── mod.rs                          # mod.rs модуля typecheck
├── checker.rs                      # Однофайловый модуль
├── environment.rs                  # Однофайловый модуль
├── overload.rs                     # Однофайловый модуль
├── type_eval.rs                    # Однофайловый модуль
├── dead_code.rs                    # Однофайловый модуль
├── spawn_placement.rs              # Однофайловый модуль
├── signature.rs                    # Однофайловый модуль
├── types.rs                        # Однофайловый модуль
│
├── tests/                          # ✅ каталог тестов typecheck
│   ├── mod.rs                      # Объявляет тесты однофайловых модулей
│   ├── checker.rs                  # Тесты checker.rs
│   ├── environment.rs              # Тесты environment.rs
│   ├── overload.rs                 # Тесты overload.rs (тест однофайлового модуля здесь)
│   ├── type_eval.rs                # Тесты type_eval.rs
│   ├── dead_code.rs                # Тесты dead_code.rs
│   ├── spawn_placement.rs          # Тесты spawn_placement.rs
│   ├── signature.rs                # Тесты signature.rs
│   └── types.rs                    # Тесты types.rs
│
├── inference/                      # Каталожный модуль (есть mod.rs)
│   ├── mod.rs                      # #[cfg(test)] mod tests; —— объявляет tests/ того же уровня
│   ├── expressions.rs
│   ├── statements.rs
│   ├── patterns.rs
│   ├── bounds.rs
│   ├── subtyping.rs
│   ├── generics.rs
│   ├── compatibility.rs
│   ├── scope.rs
│   ├── assignment.rs
│   └── tests/                      # ✅ каталог тестов inference
│       ├── mod.rs
│       ├── expressions.rs          # Тесты expressions.rs
│       ├── statements.rs           # Тесты statements.rs
│       └── ...
│
└── traits/                         # Каталожный модуль (есть mod.rs)
    ├── mod.rs                      # #[cfg(test)] mod tests; —— объявляет tests/ того же уровня
    ├── solver.rs
    ├── impl_check.rs
    ├── inheritance.rs
    ├── coherence.rs
    ├── auto_derive.rs
    ├── object_safety.rs
    ├── resolution.rs
    ├── std_traits.rs
    ├── gat/
    ├── specialization/
    └── tests/                      # ✅ каталог тестов traits
        ├── mod.rs
        ├── solver.rs               # Тесты solver.rs
        ├── impl_check.rs           # Тесты impl_check.rs
        └── ...
```

**Почему тесты однофайловых модулей располагаются в родительском `tests/`?**

Потому что однофайловый модуль (например, `overload.rs`) не имеет собственного `mod.rs` и не может объявить `#[cfg(test)] mod tests;`. Согласно системе модулей Rust, тестовые файлы должны быть объявлены в каком-либо `mod.rs` для компиляции. Поэтому тесты однофайловых модулей могут быть объявлены только родительским `mod.rs` и располагаются в `tests/` родительского каталога.

**Алгоритм принятия решения**:

```
Встретили модуль — куда поместить тесты?
│
├── Это каталожный модуль (есть mod.rs)?
│   └── Да → Создать tests/ в этом каталоге, объявить в его mod.rs
│
├── Это однофайловый модуль (только .rs)?
│   └── Да → Тесты в tests/ родительского каталога, объявить в родительском mod.rs
│
└── Неясно?
    └── Проверить наличие отдельного каталога и mod.rs
```

**Типичные ошибки**:

```
# ❌ Ошибка 1: создание отдельного tests/ для однофайлового модуля
src/frontend/core/typecheck/
├── overload.rs
└── overload/                       # ❌ Не нужно для однофайлового модуля
    └── tests/
        └── overload.rs

# ❌ Ошибка 2: объявление #[cfg(test)] mod tests; в однофайловом модуле
# overload.rs
#[cfg(test)]                        # ❌ Однофайловый модуль не может так объявлять
mod tests;                          # потому что нет каталога overload/tests/

# ✅ Правильно: тесты в родительском tests/
src/frontend/core/typecheck/
├── overload.rs                     # Исходный файл
└── tests/
    └── overload.rs                 # Тестовый файл, объявлен в typecheck/mod.rs
```

⚠️ **Антипаттерн — так не делать:**

```
# ❌ Неправильно: тесты подмодулей集中在 родительском
src/frontend/core/types/
├── mod.rs              # Должен объявлять только base и computation
├── base/
│   ├── mod.rs
│   └── var.rs
└── tests/              # ❌ Родительский tests/ содержит тесты подмодулей
    ├── mod.rs          # ❌ Вынужден объявлять mod base; mod computation;
    ├── base/           # ❌ Эта часть должна быть в base/tests/
    │   └── var.rs
    └── computation/    # ❌ Эта часть должна быть в computation/tests/
        └── ...
```

```
# ✅ Правильно: тесты каждого модуля независимы
src/frontend/core/types/
├── mod.rs              # Объявляет только pub mod base; pub mod computation;
├── base/
│   ├── mod.rs          # #[cfg(test)] mod tests; —— объявляет tests/ того же уровня
│   ├── var.rs
│   └── tests/
│       ├── mod.rs
│       └── var.rs
└── computation/
    ├── mod.rs          # #[cfg(test)] mod tests; —— объявляет tests/ того же уровня
    ├── operations.rs
    └── tests/
        ├── mod.rs
        └── operations.rs
```

**Почему нельзя агрегировать вверх?** Потому что система модулей Rust требует, чтобы `#[cfg(test)] mod tests;` решал вопрос компиляции тестов в точке объявления. Если `types/mod.rs` объявляет `mod tests;`, то содержимое `types/tests/` становится приватным содержимым модуля `types` — оно не должно заходить на территорию `base` или `computation`. Тесты каждого модуля должны быть внутренними деталями реализации этого модуля, а не родительского. Это правило также применимо при рефакторинге модулей: при разделении `types` на `base` и `computation` тесты должны следовать за разделёнными модулями, а не оставаться на месте. **Каталог тестов не зеркалирует структуру исходного кода, а следует за границами модулей.**

**Правило 1.2**: `tests/mod.rs` отвечает только за объявление модулей и re-export, тестовые функции не содержатся.

```rust
//! Тесты ядра парсера — зеркало src/frontend/core/parser/
//!
//! Тесты для ast.rs, parser_state.rs и парсинга выражений/интеграции.

mod ast;
mod error_recovery;
mod expressions;
mod integration;
mod parser_state;
```

**Правило 1.3**: Каждый тестовый файл соответствует только одному исходному файлу. Недопустимо смешивать тесты нескольких исходных модулей в одном файле.

**Правило 1.4**: `#[cfg(test)]` может появляться только в двух местах — объявление `mod tests` в `lib.rs` или inline-объявление `#[cfg(test)] mod tests;` в тестируемом исходном файле. В других местах использование запрещено.

```rust
// src/frontend/core/parser/mod.rs или lib.rs
#[cfg(test)]
mod tests;
```

### Стандарты объявления модулей

**Правило 2.1**: В начале каждого тестового файла должен быть модульный doc-комментарий `//!`, указывающий источник спецификации (номера глав спецификации языка + номера RFC). Если тест не ссылается ни на какую главу спецификации — этот код не имеет обоснования в спецификации, его не должно существовать.

```rust
//! Тесты литералов — на основе спецификации языка §2.6
//!
//! §2.6.1: Целочисленные Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: Интерполяция F-String
```

**Почему обязательна ссылка на спецификацию?** Потому что ожидаемые значения в тестах происходят из спецификации, а не из «текущего вывода кода». Если有一天 код изменит вывод, а тест вслед за этим обновится — такой тест ничего не защищает. Только тесты, привязанные к спецификации, могут отличить «намеренное breaking change» от «случайной регрессии».

**Правило 2.2**: Импорты в тестовых модулях должны быть точными — до конкретного типа/функции, запрещены glob-импорты `use super::*`.

```rust
// 🟢 Хорошо — точный импорт
use crate::frontend::core::lexer::{tokenize, TokenKind};
use crate::frontend::core::parser::{ParserState, ParseError};

// 🔴 Мусор — непонятно, что тестируется
use super::*;
```

### Стандарты именования

**Правило 3.1**: Формат имени тестовой функции: `test_<что>_<сценарий>`, все буквы строчные, слова разделены подчёркиваниями.

```rust
#[test]
fn test_tokenize_empty_string() { /* ... */ }
#[test]
fn test_parse_int_overflow() { /* ... */ }
#[test]
fn test_typecheck_fn_return_mismatch() { /* ... */ }
```

**Правило 3.2**: Имя тестовой функции должно быть самодокументирующимся. Прочитав имя, должно быть понятно, что тестируется и что ожидается. Запрещено именование порядковыми номерами.

```rust
// 🟢 Хорошо
fn test_skip_semicolon_success() { /* ... */ }
fn test_skip_semicolon_failure_when_identifier() { /* ... */ }

// 🔴 Мусор — совершенно непонятно, что тестируется
fn test_skip_1() { /* ... */ }
fn test_skip_2() { /* ... */ }
```

**Правило 3.3**: Вспомогательные функции не требуют префикса `test_`, должны описываться глаголом или существительным.

```rust
fn parse_expr(source: &str) -> Expr { /* ... */ }
fn tokenize_single(source: &str) -> Token { /* ... */ }
fn setup_parser_with_tokens(tokens: &[Token]) -> ParserState { /* ... */ }
```

### Стандарты структуры тестов (Arrange-Act-Assert)

**Правило 4.1**: Каждая тестовая функция должна следовать трёхчастной структуре: Подготовка (Arrange) → Выполнение (Act) → Проверка (Assert), части разделяются пустой строкой.

```rust
#[test]
fn test_parse_binary_addition() {
    // Arrange
    let source = "1 + 2";

    // Act
    let expr = parse_expr(source);

    // Assert
    assert!(matches!(expr, Expr::Binary { op: BinOp::Add, .. }));
}
```

**Правило 4.2**: Простые тесты (один вызов + одно утверждение) могут не иметь комментариев разделов, но не более 5 строк логики. Тесты свыше 5 строк должны явно указывать три части.

### Стандарты вспомогательных функций

**Правило 5.1**: Повторяющийся 3 и более раз setup должен быть извлечён во вспомогательную функцию.

```rust
// 🟢 Хорошо — извлечён общий setup
fn with_state<F>(source: &str, mut f: F)
where
    F: FnMut(&mut ParserState<'_>),
{
    let tokens = tokenize(source).unwrap();
    let mut state = ParserState::new(&tokens);
    f(&mut state);
}

#[test]
fn test_current_returns_first_token() {
    with_state("42", |state| {
        let tok = state.current();
        assert_eq!(&tok.unwrap().kind, &TokenKind::IntLiteral(42));
    });
}
```

**Правило 5.2**: `unwrap()` / `expect()` во вспомогательных функциях при панике должны выводить достаточный контекст. В теле тестовой функции (`#[test] fn ...`) допустим прямой `unwrap()` — при падении Rust автоматически укажет номер строки; но при падении внутри вспомогательной функции номер строки укажет на определение функции, а не на место вызова — контекст потеряется.

```rust
// 🟢 Хорошо — при падении выводится содержимое исходного кода
fn run_ok(source: &str) {
    run(source).unwrap_or_else(|e| panic!("Execution failed:\nSource:\n{}\nError:\n{:?}", source, e));
}

// 🔴 Мусор — при падении непонятно, какой исходный файл вызвал проблему
fn run_ok(source: &str) {
    run(source).unwrap();
}
```

**Правило 5.3**: Вспомогательные функции располагаются в верхней части тестового файла, сразу после импортов. Если используются несколькими тестовыми модулями — помещаются в `tests/mod.rs` и экспортируются через `pub(crate)`.

### Стиль утверждений

**Правило 6.1**: При сопоставлении вариантов перечисления предпочтителен `assert!(matches!(...))`, запрещено использование `if let` + `panic!`.

```rust
// 🟢 Хорошо
assert!(matches!(tokens[0].kind, TokenKind::IntLiteral(42)));

// 🔴 Мусор
if let TokenKind::IntLiteral(v) = tokens[0].kind {
    assert_eq!(v, 42);
} else {
    panic!("Expected IntLiteral");
}
```

**Правило 6.2**: Точные сравнения значений используют `assert_eq!`, булевы утверждения используют `assert!`. Запрещено использовать `assert!(a == b)` вместо `assert_eq!(a, b)`.

**Правило 6.3**: Все утверждения должны содержать пользовательское сообщение об ошибке, за исключением случаев, когда само утверждение уже полностью описывает причину падения.

```rust
// 🟢 Хорошо — при падении можно быстро найти проблему
assert!(
    state.infix_info().is_some(),
    "infix_info should handle '{op}'"
);

// 🟢 Хорошо — assert_eq! при падении автоматически показывает разницу значений
assert_eq!(error_count, 0);

// 🔴 Мусор — при падении只知道 "assertion failed"
assert!(state.infix_info().is_some());
```

**Правило 6.4**: Порядок в утверждениях: `assert_eq!(actual, expected)` — фактическое значение первым, ожидаемое вторым.

### Чёрный список антипаттернов

Ниже приведены запрещённые写法及其替代方案:

| Антипаттерн | Проблема | Альтернатива |
|-------------|----------|--------------|
| Тест подстраивается под ошибочное поведение кода | Маскирует отклонение от спецификации, легитимизирует баг | Сверяться со спецификацией, исправлять код, тест не менять |
| Обратный вывод ожидаемых значений теста из вывода кода | Тест становится «диктофоном текущей реализации» | Выводить ожидаемые значения из спецификации |
| Постоянная метка `#[ignore]` | Скрывает устаревшие тесты | Исправить или удалить |
| `println!` для отладки | Загрязняет вывод тестов | Использовать `assert!` для явных проверок |
| `thread::sleep` | Случайные сбои + медленно | Использовать синхронизацию или mock |
| Работа с реальной файловой системой в тестах | Медленно и невоспроизводимо | Использовать `tempfile` |
| Зависимость от порядка выполнения тестов | Случайные сбои | Независимый setup для каждого теста |
| Тестовая функция свыше 30 строк логики | Невозможно понять | Разделить тест или использовать вспомогательные функции |
| `unwrap()` во вспомогательной функции без контекста | Трудно локализовать | Использовать `expect("why")` или custom panic (см. правило 5.2) |
| Copy-paste одинакового setup 3+ раза | Высокая стоимость изменений | Извлечь во вспомогательную функцию |

## Стандарты интеграционных тестов

### Организация тестов

**Правило 7.1**: Интеграционные тесты располагаются в каталоге `tests/` корня проекта. Входной файл `tests/integration.rs` использует атрибут `#[path]` для включения подмодулей.

```rust
// tests/integration.rs
#[path = "integration/backends.rs"]
mod backends;
#[path = "integration/codegen.rs"]
mod codegen;
#[path = "integration/execution.rs"]
mod execution;
```

**Правило 7.2**: Каждый файл `tests/integration/*.rs` соответствует одной теме тестирования (бэкенды компилятора, кодогенерация, исполнитель и т.д.), смешивание запрещено.

**Правило 7.3**: Интеграционные тесты должны использовать публичные API проекта. Запрещено обращаться к внутренним модулям `crate::` в интеграционных тестах. Использовать публичный путь `yaoxiang::`.

```rust
// 🟢 Хорошо — через публичный API
use yaoxiang::run;

// 🔴 Мусор —绕过 публичный API
use yaoxiang::middle::codegen::bytecode::BytecodeFile;
```

### Управление тестовыми данными

**Правило 8.1**: В интеграционных тестах предпочтительны inline-строки исходного кода. Внешние fixture-файлы (в `tests/fixtures/`) используются только когда исходный код превышает 30 строк.

```rust
#[test]
fn test_fibonacci() {
    run_ok(
        r#"
        main = {
            mut a = 0
            mut b = 1
            while a < 100 {
                mut next = a + b
                a = b
                b = next
            }
        }
        "#,
    );
}
```

**Правило 8.2**: Fixture-файлы должны иметь расширение `.yx`, имя файла описывает намерение теста.

### Принципы E2E-покрытия

**Правило 9.1**: Интеграционные тесты каждой языковой конструкции должны покрывать три пути:

| Путь | Описание |
|------|----------|
| Happy path | Корректный вход даёт ожидаемый выход |
| Error path | Некорректный вход выдаёт понятное сообщение об ошибке (не panic) |
| Boundary | Граничные значения (пустой ввод, максимальные значения, предел глубины вложенности) |

**Правило 9.2**: Интеграционные тесты не должны зависеть от сети, переменных окружения или внешних сервисов.

## Стандарты бенчмарков

### Стандарты использования Criterion.rs

**Правило 10.1**: Бенчмарки统一放在 `benches/` каталоге, входной файл — `benches/lib.rs`. Разделение по темам тестирования.

```
benches/
├── lib.rs              # Входной файл, определяет criterion_group/criterion_main
├── lang_compare/
│   └── fibonacci.rs    # Кроссязыковое сравнение
├── parser.rs           # Бенчмарки парсера
└── codegen.rs          # Бенчмарки кодогенерации
```

**Правило 10.2**: Каждая бенчмарк-функция должна содержать модульный doc-комментарий `//!` с описанием цели теста и метрик измерения.

```rust
//! Бенчмарки производительности интерпретатора YaoXiang
//!
//! Метрики: время одной итерации (wall time)
//! Baseline: нативная реализация на Rust
```

### Предотвращение оптимизаций компилятора

**Правило 11.1**: Все выходы измеряемого кода в бенчмарках должны быть обёрнуты в `criterion::black_box` для предотвращения elimination оптимизатором.

```rust
use criterion::{black_box, Criterion};

fn bench_parse(c: &mut Criterion) {
    c.bench_function("parse_fib", |b| {
        b.iter(|| {
            let result = parse(black_box(FIB_SOURCE));
            black_box(result)
        })
    });
}
```

**Правило 11.2**: Входные данные бенчмарков должны быть `const` или `lazy_static`, не должны генерироваться динамически внутри замыкания `iter` — иначе измеряется время генерации данных + измеряемой логики.

### Группировка и именование бенчмарков

**Правило 12.1**: Формат имени бенчмарк-функции: `<тестируемый_модуль>_<сценарий>`, все буквы строчные, слова разделены подчёркиваниями. Соответствует правилам именования модульных тестов.

**Правило 12.2**: Необходимо использовать `criterion_group!` для логической группировки связанных бенчмарков. Запрещено все бенчмарки помещать в одну группу.

```rust
criterion_group!(parser, bench_parse_expr, bench_parse_stmt);
criterion_group!(codegen, bench_codegen_module, bench_codegen_switch);
criterion_main!(parser, codegen);
```

## Стандарты doctest

### Сценарии использования

**Правило 13.1**: Все `pub` функции, типы и методы должны содержать в doc-комментариях минимум один исполняемый пример кода. Этот пример выполняется через `cargo test --doc`.

```rust
/// Токенизирует исходную строку в последовательность Token.
///
/// ```
/// use yaoxiang::frontend::core::lexer::tokenize;
///
/// let tokens = tokenize("42").unwrap();
/// assert_eq!(tokens.len(), 2); // IntLiteral + Eof
/// ```
pub fn tokenize(source: &str) -> Result<Vec<Token>, LexError> {
    // ...
}
```

**Правило 13.2**: Примеры кода в doctest должны компилироваться и содержать успешные утверждения. Запрещены примеры с меткой `ignore`, за исключением демонстрации ошибок компиляции.

```rust
/// ```ignore
/// // Демонстрация ошибки компиляции — можно ignore
/// let x: int = "string";
/// ```
```

### Требования к покрытию

**Правило 14.1**: doctest покрывают только happy path API. Граничные случаи и пути ошибок покрываются модульными тестами.

**Правило 14.2**: Примеры кода в doctest должны быть лаконичными — не более 10 строк. Если пример требует более длинного контекста — это признак проблемы в дизайне API.

## Стандарты property-тестов

### Сценарии использования

**Правило 15.1**: В следующих场景х обязательно использование property-тестов (proptest или quickcheck) вместо ручного перечисления граничных значений:

| Сценарий | Пример |
|----------|--------|
| Round-trip парсера | `parse(pretty_print(ast)) == ast` |
| Сериализация/десериализация | `deserialize(serialize(data)) == data` |
| Тождества математических операций | `a + b == b + a` |
| Компиляторные оптимизации без изменения семантики | `eval(code) == eval(optimize(code))` |

**Правило 15.2**: В качестве основного фреймворка для property-тестов используется `proptest` (уже объявлен в `dev-dependencies` `Cargo.toml`).

```rust
use proptest::prelude::*;

proptest! {
    #[test]
    fn test_roundtrip_serialize_deserialize(value: i64) {
        let serialized = serialize(&value);
        let deserialized: i64 = deserialize(&serialized).unwrap();
        prop_assert_eq!(deserialized, value);
    }
}
```

### Принципы определения property

**Правило 16.1**: Каждый property-тест должен иметь явное объявление property — в комментарии указывается проверяемый инвариант.

```rust
// Property: любой целочисленный литерал после tokenize → tokens_to_string даёт то же значение
proptest! {
    #[test]
    fn test_int_literal_roundtrip(n in any::<i64>()) {
        let source = n.to_string();
        let tokens = tokenize(&source).unwrap();
        // ...
    }
}
```

**Правило 16.2**: При обнаружении падения property-теста необходимо использовать механизм регрессии `proptest` — добавить упавший вход в каталог `proptest-regressions/`, а не заменять обычным тестом.

## Требования к покрытию

### Цели покрытия для нового кода

**Правило 17.1**: Требования к тестовому покрытию нового кода:

| Тип кода | Покрытие строк | Покрытие веток |
|----------|----------------|----------------|
| Основные модули компилятора (frontend/middle/backends) | ≥ 85% | ≥ 80% |
| Утилиты/вспомогательные модули (util) | ≥ 75% | ≥ 70% |
| Модули runtime (vm/runtime) | ≥ 80% | ≥ 75% |
| Стандартная библиотека (std) | ≥ 75% | ≥ 70% |
| Обработка ошибок и диагностика | ≥ 90% | ≥ 85% |

**Правило 17.2**: Пути обработки ошибок (все ветки `Err`) должны быть покрыты на 100%. Сообщения об ошибках, видимые пользователю, должны быть проверены тестами.

### Чеклист для проверки PR

**Правило 18.1**: Перед отправкой PR автор должен проверить:

- [ ] `cargo test` все проходят
- [ ] `cargo test --doc` все проходят
- [ ] `cargo bench` без регрессий производительности (если затронуты hot paths)
- [ ] Новый код соответствует целям покрытия
- [ ] Имена тестов соответствуют стандартам именования
- [ ] Каждый тестовый файл объявляет соответствующую главу спецификации (правило 2.1)
- [ ] Ожидаемые значения тестов происходят из определений спецификации, а не из «вывода текущего кода»
- [ ] Нет тестов с меткой `#[ignore]` (без комментария с явным номером issue)
- [ ] Нет лишних `unwrap()` (должны использоваться `expect` или custom panic-сообщения)
- [ ] Сообщение коммита использует тип `:white_check_mark: test:`
- [ ] **Нет изменения ожидаемых значений тестов из-за «код не соответствует спецификации» — исправляется код, а не тест**

**Правило 18.2**: Reviewer обязан отклонить PR, содержащий:

- Только happy path тесты без путей ошибок
- `thread::sleep` или зависимость от порядка выполнения в тестах
- Copy-paste тестового кода более 3 раз без извлечения вспомогательных функций
- Имена тестов не соответствуют стандартам именования
- Постоянные `#[ignore]` тесты
- **Тесты, подстраивающиеся под ошибочное поведение кода** (когда код не соответствует спецификации — исправлять код, а не тест)
- **Тесты без объявления соответствующей главы спецификации** (см. правило 2.1)
- **Ожидаемые значения тестов происходят из вывода кода, а не из спецификации** (обратно выведенные тесты эквивалентны отсутствию тестов)
- Тесты проверяют только «не паникует», но не утверждают конкретное поведение
- Удалены упавшие тесты,暴露вшие баг в коде (вместо исправления кода с последующим «зелёным» результатом)

## Приложения

### A. Шпаргалка команд тестирования

```bash
# Запустить все тесты
cargo test

# Только модульные тесты
cargo test --lib

# Только интеграционные тесты
cargo test --test integration

# Только doctest
cargo test --doc

# Запустить конкретный тест (фильтр по имени)
cargo test test_parse_expr

# Запустить бенчмарки
cargo bench

# Показать вывод тестов (по умолчанию скрыт stdout)
cargo test -- --nocapture

# Однопоточный запуск (для排查 проблем параллелизма)
cargo test -- --test-threads=1

# Сгенерировать отчёт о покрытии (требуется cargo-llvm-cov)
cargo llvm-cov --html
```

### B. Шаблон сообщения коммита

Коммиты, связанные с тестами, должны следовать шаблону:

```
:white_check_mark: test(<scope>): <краткое описание>

<опционально: список покрытых场景>
```

Пример:

```
:white_check_mark: test(parser): добавить тесты инфиксных операторов в парсере Pratt

Покрытые场景:
- Приоритет арифметических операторов (+, -, *, /, %)
- Связывание операторов сравнения (1 < x < 10)
- Short-circuit логических операторов
- Правоассоциативность операторов присваивания
```

### C. Чеклист создания нового тестового файла

При создании нового тестового модуля убедитесь в наличии:

```
# Новые тесты в каталоге src/<module>/
src/<module>/tests/
├── mod.rs          # Объявление модуля + общие вспомогательные функции
└── <subject>.rs    # Тестовый файл, соответствует имени тестируемого исходного файла

# Новые интеграционные тесты в каталоге tests/
tests/
├── integration.rs   # Обновить: добавить объявление #[path]
└── integration/
    └── <topic>.rs   # Новый тестовый файл
```

> 💡 **Запомните**: Тесты не проверяют, работает ли ваш код — они проверяют, соответствует ли ваш код спецификации. Спецификация меняется — тесты следуют за спецификацией. Код написан неправильно — исправляйте код, а не тест. **Код служит спецификации, тесты охраняют спецификацию. Момент, когда тест подстраивается под код — вы теряете всю защиту.**
