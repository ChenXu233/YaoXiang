## Общие положения

### Область применения

Настоящие правила распространяются на весь тестовый код на Rust в проекте YaoXiang, включая:

| Тип теста | Расположение | Фреймворк |
|-----------|--------------|-----------|
| Модульные тесты | `src/<module>/tests/` | `#[test]` + `#[cfg(test)]` |
| Интеграционные тесты | `tests/` | `#[test]` |
| Нагрузочные тесты | `benches/` | Criterion.rs |
| Документационные тесты | Комментарии документации API | `cargo test --doc` |
| Property-тесты | Любое место тестов | proptest / quickcheck |

### Основные принципы

**Принцип 0: Авторитетным источником для тестов является спецификация, а не код.** Это важнейший принцип настоящего документа. Тесты проверяют, соответствует ли код спецификации, а не «работает ли код в текущей реализации». Когда тест обнаруживает несоответствие поведения кода спецификации, **исправляется код, и никогда — тест**.

Файлы спецификации расположены по адресам:
- `docs/src/design/language-spec.md` — спецификация ядра языка
- `docs/src/design/rfc/accepted/` — принятые проектные документы RFC

В верхней части каждого тестового файла должна быть ссылка на соответствующий раздел спецификации (см. правило 2.1). Любой разработчик должен иметь возможность сверить тесты с документом спецификации и убедиться в корректности реализации. Обратное утверждение: если для некоторого кода нет соответствующего описания в спецификации — он не должен существовать, и тем более не должен тестироваться.

```rust
// 🟢 Хорошо — тест напрямую ссылается на спецификацию и проверяет соответствие кода
//! Тесты литералов — на основе спецификации языка §2.6
//!
//! §2.6.1: Целые числа Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: F-String интерполяция

#[test]
fn test_decimal_literal_parsing() {
    // Спецификация §2.6.1: Decimal ::= [0-9][0-9_]*
    let result = parse_literal("42").unwrap();
    assert_eq!(result, Literal::Int(42));
}

// 🔴 Мусор — тест подстраивается под текущее поведение кода, а не проверяет спецификацию
#[test]
fn test_literal_1() {
    // Непонятно, какому разделу спецификации соответствует этот код
    // Если parse_literal вернёт неверное значение, этот тест «зелёный»,
    // потому что он проверяет лишь отсутствие паники
    let result = parse_literal("42");
    assert!(result.is_ok());
}
```

**Сценарий**: вы написали тест и обнаружили, что поведение кода не соответствует спецификации. У вас два варианта:
| Неправильный подход | Правильный подход |
|---------------------|-------------------|
| Изменить тест, чтобы он «прошёл» | Изменить код, чтобы поведение соответствовало спецификации |
| Добавить в тест `#[ignore]` | Немедленно исправить реализацию кода |
| Добавить в тест особые условные ветки, приспособленные к коду | Удалить ветки, чтобы тест прямо выявлял проблему |

Помните: **красный сигнал = ошибка в коде, а не в тесте.** (Если только в самом тесте нет бага — это другой случай.)

**Принцип 1: Тест — это документация.** Любой разработчик должен иметь возможность понять поведение тестируемого кода, читая сам тест, без дополнительных комментариев или внешней документации.

```rust
// 🟢 Хорошо — из имени теста понятно, что и как проверяется
#[test]
fn test_tokenize_empty_input_returns_eof() {
    let tokens = tokenize("").unwrap();
    assert_eq!(tokens.len(), 1);
    assert!(matches!(tokens[0].kind, TokenKind::Eof));
}

// 🔴 Мусор — непонятно, что именно проверяется
#[test]
fn test_tokenize_1() {
    let tokens = tokenize("").unwrap();
    assert!(tokens.len() > 0);
}
```

**Принцип 2: Нулевая терпимость к случайным сбоям.** Тесты должны воспроизводимо выполняться в любом окружении. Тесты, зависящие от случайных чисел, системного времени, порядка планирования потоков, должны использовать фиксированный seed или подменяться mock-объектами.

**Принцип 3: Один тест — одна проверка.** Если имя теста приходится соединять союзом «и» для описания нескольких поведений — разделите на несколько тестов.

```rust
// 🟢 Хорошо — каждый тест проверяет один сценарий
#[test]
fn test_parse_int_positive() { /* ... */ }
#[test]
fn test_parse_int_zero() { /* ... */ }

// 🔴 Мусор — в одном тесте слишком много несвязанного
#[test]
fn test_parser() {
    // Тестирует и tokenize, и parse, и typecheck, и codegen...
}
```

**Принцип 4: Тестируйте поведение, а не реализацию.** Рефакторинг внутренней реализации не должен приводить к падению тестов. Если из-за изменения одной строки реализации упало 10 тестов — ваши тесты написаны неправильно.

Но здесь важное разграничение: **определение «поведения» берётся из спецификации, а не из текущего поведения кода.** Если код изменил поведение (то есть выдал новое поведение, не соответствующее спецификации), тест обязан упасть. Если этого не происходит — ваши тесты «подстраиваются под код», и они открывают дорогу багам.

```
Спецификация (language-spec.md / RFC)  ──определяет──►  Ожидаемое поведение  ──движет──►  Тест
                                              │
Текущий код  ──реализует──►  Фактическое поведение  ──сравнивается──►  Результат теста

Если фактическое поведение ≠ ожидаемого:
  Тест обязан упасть (красный)  ──►  Исправить код  ──►  Тест проходит (зелёный)
  
Если фактическое поведение = ожидаемому (но реализация плохая):
  Тест проходит  ──►  Рефакторинг реализации  ──►  Тест по-прежнему проходит  ← в этом смысл принципа 4
```

**Принцип 5: Не писать откаты/совместимость/код, зависящий от активации определённых режимов.** Тестовая среда — это среда, которую вы полностью контролируете. Если вам нужен `#[cfg(not(ci))]`, чтобы пропустить какой-то тест, это указывает на фундаментальную проблему в его дизайне.

### Определения терминов

| Термин | Определение |
|--------|-------------|
| Модульный тест | Тестирует поведение одной функции или модуля, не зависит от внешних систем |
| Интеграционный тест | Тестирует взаимодействие нескольких модулей через публичный API или CLI-вход |
| Нагрузочный тест | Измеряет производительность кода, выявляет регрессии производительности |
| Документационный тест | Исполняемый пример кода, встроенный в комментарий документации |
| Property-тест | Тест, основанный на проверке инвариантов (свойств) на случайных входных данных |

### Связь с правилами коммитов

Все коммиты, относящиеся к тестам, должны использовать тип `:white_check_mark: test:`, согласно [правилам коммитов](./commit-convention.md).

```
:white_check_mark: test(parser): добавить тесты инфиксных выражений парсера Pratt
:white_check_mark: test(codegen): дополнить тесты генерации IR для switch
```

## Правила модульных тестов

### Организация файлов

**Правило 1.1**: каталог `tests/` модульных тестов должен располагаться **на одном уровне** с `mod.rs` тестируемого модуля. `tests/` не агрегируется вверх и не сводится на более высокий уровень.

```
src/frontend/core/parser/
├── mod.rs              # #[cfg(test)] mod tests; — объявляет tests/ на том же уровне
├── ast.rs
├── pratt/
│   ├── mod.rs          # #[cfg(test)] mod tests; — собственные тесты pratt
│   └── tests/
│       ├── mod.rs
│       ├── led.rs
│       ├── nud.rs
│       └── precedence.rs
└── tests/              # Тесты уровня модуля parser (без тестов подмодуля pratt)
    ├── mod.rs
    ├── ast.rs
    ├── expressions.rs
    ├── error_recovery.rs
    └── parser_state.rs
```

Ключевой критерий: в каком каталоге лежит `tests/`, в `mod.rs` этого каталога должно быть объявление `#[cfg(test)] mod tests;`.

**Правило 1.1 (дополнение): запрещена агрегация вверх.** Тесты подмодуля должны находиться в собственном `tests/` этого подмодуля и не должны агрегироваться в родительский `tests/`.

| Тип модуля | Расположение тестов | Пример |
|------------|--------------------|--------|
| Каталог-модуль (есть `mod.rs`) | `tests/` внутри этого каталога | `emitter/tests/`, `codes/tests/` |
| Однофайловый модуль (только `.rs`) | `tests/` родительского уровня | `session.rs` → `diagnostic/tests/session.rs` |

```text
# ✅ Правильно: тесты каждого модуля-каталога независимы
src/util/diagnostic/
├── codes/
│   ├── mod.rs              # #[cfg(test)] mod tests;
│   └── tests/              # ✅ Собственные тесты codes
│       ├── mod.rs
│       └── codes.rs
├── emitter/
│   ├── mod.rs              # #[cfg(test)] mod tests;
│   └── tests/              # ✅ Собственные тесты emitter
│       ├── mod.rs
│       ├── text.rs
│       └── ansi.rs
└── tests/                  # ✅ Уровень diagnostic (однофайловые модули)
    ├── mod.rs
    ├── session.rs
    ├── suggest.rs
    └── collect.rs

# ❌ Неправильно: тесты emitter и codes агрегированы в diagnostic/tests/
src/util/diagnostic/
└── tests/
    ├── mod.rs              # ❌ Вынуждены объявлять mod emitter; mod codes;
    ├── emitter/            # ❌ Должно быть в emitter/tests/
    └── codes/              # ❌ Должно быть в codes/tests/
```

#### Правила размещения тестов: однофайловый модуль vs модуль-каталог

**Ключевое различие**: форма организации модуля определяет место расположения тестов.

| Тип модуля | Критерий определения | Расположение тестов | Пример |
|------------|----------------------|---------------------|--------|
| **Модуль-каталог** | Имеет собственный каталог и `mod.rs` | `tests/` внутри каталога | `inference/tests/` |
| **Однофайловый модуль** | Только файл `.rs`, без отдельного каталога | `tests/` родительского модуля | `overload.rs` → `typecheck/tests/overload.rs` |

**Подробное пояснение**:

```
src/frontend/core/typecheck/
├── mod.rs                          # mod.rs модуля typecheck
├── checker.rs                      # Однофайловый модуль
├── environment.rs                  # Однофайловый модуль
├── overload.rs                     # Однофайловый модуль
├── type_eval.rs                    # Однофайловый модуль
├── dead_code.rs                    # Однофайловый модуль
├── spawn_placement.rs              # Однофайловый модуль
├── signature.rs                    # Однофайловый модуль
├── types.rs                        # Однофайловый модуль
│
├── tests/                          # ✅ Каталог тестов typecheck
│   ├── mod.rs                      # Объявляет тесты однофайловых модулей
│   ├── checker.rs                  # Тесты для checker.rs
│   ├── environment.rs              # Тесты для environment.rs
│   ├── overload.rs                 # Тесты для overload.rs (однофайловый модуль)
│   ├── type_eval.rs                # Тесты для type_eval.rs
│   ├── dead_code.rs                # Тесты для dead_code.rs
│   ├── spawn_placement.rs          # Тесты для spawn_placement.rs
│   ├── signature.rs                # Тесты для signature.rs
│   └── types.rs                    # Тесты для types.rs
│
├── inference/                      # Модуль-каталог (есть mod.rs)
│   ├── mod.rs                      # #[cfg(test)] mod tests; — объявляет tests/ на одном уровне
│   ├── expressions.rs
│   ├── statements.rs
│   ├── patterns.rs
│   ├── bounds.rs
│   ├── subtyping.rs
│   ├── generics.rs
│   ├── compatibility.rs
│   ├── scope.rs
│   ├── assignment.rs
│   └── tests/                      # ✅ Каталог тестов inference
│       ├── mod.rs
│       ├── expressions.rs          # Тесты для expressions.rs
│       ├── statements.rs           # Тесты для statements.rs
│       └── ...
│
└── traits/                         # Модуль-каталог (есть mod.rs)
    ├── mod.rs                      # #[cfg(test)] mod tests; — объявляет tests/ на одном уровне
    ├── solver.rs
    ├── impl_check.rs
    ├── inheritance.rs
    ├── coherence.rs
    ├── auto_derive.rs
    ├── object_safety.rs
    ├── resolution.rs
    ├── std_traits.rs
    ├── gat/
    ├── specialization/
    └── tests/                      # ✅ Каталог тестов traits
        ├── mod.rs
        ├── solver.rs               # Тесты для solver.rs
        ├── impl_check.rs           # Тесты для impl_check.rs
        └── ...
```

**Почему тесты однофайлового модуля лежат в родительском `tests/`?**

Потому что у однофайлового модуля (например, `overload.rs`) нет собственного `mod.rs`, и он не может объявить `#[cfg(test)] mod tests;`. Согласно системе модулей Rust, тестовые файлы должны быть объявлены в каком-то `mod.rs`, чтобы компилироваться. Поэтому тесты однофайловых модулей могут быть объявлены только в `mod.rs` родительского модуля и находиться в родительском `tests/`.

**Алгоритм определения**:

```
Встретился модуль — куда положить тесты?
│
├── Модуль — каталог (есть mod.rs)?
│   └── Да → создать tests/ внутри каталога, объявить в его mod.rs
│
├── Модуль — однофайловый (только .rs)?
│   └── Да → тесты кладутся в tests/ родительского уровня, объявляются в его mod.rs
│
└── Не уверены?
    └── Проверить наличие собственного каталога и mod.rs
```

**Распространённые ошибки**:

```
# ❌ Ошибка 1: создание отдельного каталога tests/ для однофайлового модуля
src/frontend/core/typecheck/
├── overload.rs
└── overload/                       # ❌ Нельзя создавать каталог для однофайлового модуля
    └── tests/
        └── overload.rs

# ❌ Ошибка 2: объявление #[cfg(test)] mod tests; внутри однофайлового модуля
# overload.rs
#[cfg(test)]                        # ❌ Однофайловый модуль не может так объявлять
mod tests;                          # потому что нет каталога overload/tests/

# ✅ Правильный подход: тесты лежат в родительском tests/
src/frontend/core/typecheck/
├── overload.rs                     # Исходный файл
└── tests/
    └── overload.rs                 # Тестовый файл, объявленный в typecheck/mod.rs
```

⚠️ **Антипаттерн — так не делать:**

```
# ❌ Неправильно: тесты подмодулей собраны в родительский
src/frontend/core/types/
├── mod.rs              # Должен объявлять только base и computation
├── base/
│   ├── mod.rs
│   └── var.rs
└── tests/              # ❌ Родительский tests/ содержит тесты подмодулей
    ├── mod.rs          # ❌ Вынуждены объявлять mod base; mod computation;
    ├── base/           # ❌ Должно быть в base/tests/
    │   └── var.rs
    └── computation/    # ❌ Должно быть в computation/tests/
        └── ...
```

```
# ✅ Правильный подход: тесты каждого модуля независимы
src/frontend/core/types/
├── mod.rs              # Объявляет только pub mod base; pub mod computation;
├── base/
│   ├── mod.rs          # #[cfg(test)] mod tests; — объявляет tests/ на одном уровне
│   ├── var.rs
│   └── tests/
│       ├── mod.rs
│       └── var.rs
└── computation/
    ├── mod.rs          # #[cfg(test)] mod tests; — объявляет tests/ на одном уровне
    ├── operations.rs
    └── tests/
        ├── mod.rs
        └── operations.rs
```

**Почему нельзя агрегировать вверх?** Потому что система модулей Rust требует, чтобы `#[cfg(test)] mod tests;` в точке объявления определял компиляцию тестовых файлов. Если `types/mod.rs` объявляет `mod tests;`, то содержимое `types/tests/` — это приватное содержимое модуля `types` — оно не должно вторгаться в `base` или `computation`. Тесты каждого модуля должны быть деталями его внутренней реализации, а не родительского модуля. Это правило применимо и при рефакторинге модулей: когда вы разбиваете `types` на `base` и `computation`, тесты тоже следуют за границами модулей, а не остаются на прежнем месте. **Каталог тестов не зеркалит структуру исходников, а следует за границами модулей.**

**Правило 1.2**: `tests/mod.rs` отвечает только за объявление модулей и re-export, тестовые функции в нём не размещаются.

```rust
//! Parser core tests — mirrors src/frontend/core/parser/
//!
//! Тесты для ast.rs, parser_state.rs и парсинга выражений/интеграции.

mod ast;
mod error_recovery;
mod expressions;
mod integration;
mod parser_state;
```

**Правило 1.3**: каждый тестовый файл соответствует ровно одному исходному файлу. Смешивание тестов нескольких модулей в одном файле запрещено.

**Правило 1.4**: объявление тестов должно использовать файловую форму `mod tests;` (с точкой с запятой), указывающую на одноуровневый каталог `tests/`. **Запрещена inline-форма `mod tests { ... }`, при которой тестовый код пишется прямо в исходном файле.**

```rust
// ✅ Правильно — объявление в файловой форме, тестовый код в отдельных файлах
// src/frontend/core/parser/mod.rs
#[cfg(test)]
mod tests;

// 🔴 Запрещено — inline-форма, тестовый код паразитирует в исходном файле
// src/frontend/core/parser/mod.rs
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_something() {
        // Тестовый код не должен находиться в исходном файле
    }
}
```

**Почему запрещён inline?**
1. Единственная ответственность исходного файла: в нём только реализация, в тестовом — только тесты. Смешивание вынуждает прокручивать файл к концу, чтобы править тесты, и пропускать тесты, чтобы править реализацию.
2. Чёткие границы модулей: каталог `tests/` — это физическая граница, по которой сразу видно, какие модули покрыты тестами, а какие нет.
3. Безопасность рефакторинга: при разделении модуля каталог `tests/` перемещается вместе с ним; inline-тесты нужно вручную вырезать из исходного файла.
4. Code Review: в diff-е PR изменения исходников и тестов находятся в разных файлах и не смешиваются.

### Правила объявления модулей

**Правило 2.1**: в верхней части каждого тестового файла должен быть документационный комментарий уровня модуля `//!`, в котором указан источник покрываемой спецификации (номер раздела спецификации языка + номер RFC). Если какой-то тест не ссылается ни на один раздел спецификации — у этого кода нет спецификационного обоснования, и он не должен существовать.

```rust
//! Тесты литералов — на основе спецификации языка §2.6
//!
//! §2.6.1: Целые числа Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: F-String интерполяция
```

**Почему обязательна ссылка на спецификацию?** Потому что ожидаемые значения теста берутся из спецификации, а не из «текущего вывода кода». Если когда-нибудь код изменит вывод, а тест обновится вслед за ним — тест ничего не защищает. Только привязанные к спецификации тесты способны отличить «намеренное breaking change» от «непреднамеренной регрессии».

**Правило 2.2**: импорты `use` в тестовом модуле должны быть точными — до конкретных типов/функций. Glob-импорт `use super::*` запрещён.

```rust
// 🟢 Хорошо — точечный импорт
use crate::frontend::core::lexer::{tokenize, TokenKind};
use crate::frontend::core::parser::{ParserState, ParseError};

// 🔴 Мусор — непонятно, что именно тестируется
use super::*;
```

### Правила именования

**Правило 3.1**: формат имени тестовой функции — `test_<что>_<сценарий>`, полностью в нижнем регистре с разделением подчёркиваниями.

```rust
#[test]
fn test_tokenize_empty_string() { /* ... */ }
#[test]
fn test_parse_int_overflow() { /* ... */ }
#[test]
fn test_typecheck_fn_return_mismatch() { /* ... */ }
```

**Правило 3.2**: имя тестовой функции должно быть самодостаточным. Прочитав имя, нужно понимать, что проверяется и каково ожидание. Цифровые суффиксы в именах запрещены.

```rust
// 🟢 Хорошо
fn test_skip_semicolon_success() { /* ... */ }
fn test_skip_semicolon_failure_when_identifier() { /* ... */ }

// 🔴 Мусор — совершенно непонятно, что проверяется
fn test_skip_1() { /* ... */ }
fn test_skip_2() { /* ... */ }
```

**Правило 3.3**: вспомогательные функции не требуют префикса `test_`; их имена должны описывать назначение глаголом или существительным.

```rust
fn parse_expr(source: &str) -> Expr { /* ... */ }
fn tokenize_single(source: &str) -> Token { /* ... */ }
fn setup_parser_with_tokens(tokens: &[Token]) -> ParserState { /* ... */ }
```

### Структура теста (Arrange-Act-Assert)

**Правило 4.1**: каждая тестовая функция должна следовать трёхчастной структуре: подготовка (Arrange) → действие (Act) → проверка (Assert); части разделяются пустой строкой.

```rust
#[test]
fn test_parse_binary_addition() {
    // Arrange
    let source = "1 + 2";

    // Act
    let expr = parse_expr(source);

    // Assert
    assert!(matches!(expr, Expr::Binary { op: BinOp::Add, .. }));
}
```

**Правило 4.2**: в простых тестах (один вызов + одна проверка) можно не указывать разделяющие комментарии, но логический код не должен превышать 5 строк. В тестах длиннее 5 строк разделение на три части обязательно.

### Правила для вспомогательных функций

**Правило 5.1**: setup-логика, повторяющаяся 3 и более раз, должна быть вынесена во вспомогательную функцию.

```rust
// 🟢 Хорошо — выделен общий setup
fn with_state<F>(source: &str, mut f: F)
where
    F: FnMut(&mut ParserState<'_>),
{
    let tokens = tokenize(source).unwrap();
    let mut state = ParserState::new(&tokens);
    f(&mut state);
}

#[test]
fn test_current_returns_first_token() {
    with_state("42", |state| {
        let tok = state.current();
        assert_eq!(&tok.unwrap().kind, &TokenKind::IntLiteral(42));
    });
}
```

**Правило 5.2**: `unwrap()` / `expect()` во вспомогательных функциях обязаны при panic-е выводить достаточно контекста. Внутри тела тестовой функции (`#[test] fn ...`) можно вызывать `unwrap()` напрямую — при сбое Rust автоматически печатает номер строки; но при сбое во вспомогательной функции номер строки указывает на её определение, и контекст вызова теряется.

```rust
// 🟢 Хорошо — при сбое вспомогательной функции выводится содержимое исходника
fn run_ok(source: &str) {
    run(source).unwrap_or_else(|e| panic!("Execution failed:\nSource:\n{}\nError:\n{:?}", source, e));
}

// 🔴 Мусор — при сбое не видно, какой источник вызвал проблему
fn run_ok(source: &str) {
    run(source).unwrap();
}
```

**Правило 5.3**: вспомогательные функции размещаются в начале тестового файла, сразу после `use`-импортов. Если они используются несколькими тестовыми модулями, их размещают в `tests/mod.rs` и экспортируют с `pub(crate)`.

### Стиль утверждений

**Правило 6.1**: для сопоставления с вариантом enum предпочтительно `assert!(matches!(...))`. Использование `if let` + `panic!` запрещено.

```rust
// 🟢 Хорошо
assert!(matches!(tokens[0].kind, TokenKind::IntLiteral(42)));

// 🔴 Мусор
if let TokenKind::IntLiteral(v) = tokens[0].kind {
    assert_eq!(v, 42);
} else {
    panic!("Expected IntLiteral");
}
```

**Правило 6.2**: для сравнения точных значений используется `assert_eq!`, для булевых утверждений — `assert!`. Запрещено заменять `assert_eq!(a, b)` на `assert!(a == b)`.

**Правило 6.3**: все утверждения должны сопровождаться пользовательским сообщением об ошибке, кроме случаев, когда само утверждение полностью описывает причину сбоя.

```rust
// 🟢 Хорошо — при сбое легко локализовать проблему
assert!(
    state.infix_info().is_some(),
    "infix_info should handle '{op}'"
);

// 🟢 Хорошо — assert_eq! при сбое сам выводит разницу значений, дополнительное сообщение не нужно
assert_eq!(error_count, 0);

// 🔴 Мусор — при сбое видно только "assertion failed"
assert!(state.infix_info().is_some());
```

**Правило 6.4**: порядок аргументов утверждения — `assert_eq!(actual, expected)`: фактическое значение первым, ожидаемое — вторым.

### Список антипаттернов

Ниже приведены запрещённые приёмы и их замены:

| Антипаттерн | Проблема | Замена |
|-------------|----------|--------|
| Inline-тесты `#[cfg(test)] mod tests { ... }` | Раздутый исходник, размытые границы модуля, сложный рефакторинг | Тестовый код в отдельном каталоге `tests/`, объявление через `mod tests;` (см. правило 1.4) |
| Тест подстраивается под ошибочное поведение кода | Скрывает отклонения от спецификации, легитимизирует баг | Сверяться со спецификацией и править код, оставляя тест неизменным |
| Ожидаемые значения в тесте выводятся из вывода кода | Тест становится «диктофоном текущей реализации» | Выводить ожидаемые значения из спецификации |
| Постоянный `#[ignore]` | Прячет сгнившие тесты | Исправить или удалить |
| Отладочный вывод через `println!` | Загрязняет вывод тестов | Использовать явные `assert!` |
| `thread::sleep` | Случайные сбои + медленно | Использовать механизмы синхронизации или mock-объекты |
| Работа с реальной файловой системой в тестах | Медленно и не воспроизводимо | Использовать `tempfile` |
| Зависимость от порядка выполнения тестов | Случайные сбои | Независимый setup в каждом тесте |
| Тестовая функция длиннее 30 строк логики | Никто не сможет разобраться | Разбить на тесты или вынести во вспомогательные функции |
| `unwrap()` во вспомогательной функции без вывода контекста | Сложно локализовать | Использовать `expect("why")` или собственный panic (см. правило 5.2) |
| Copy-paste setup-логики более 3 раз | Высокая стоимость изменений | Вынести во вспомогательную функцию |

## Правила интеграционных тестов

### Организация тестов

**Правило 7.1**: интеграционные тесты размещаются в каталоге `tests/` в корне проекта. Входной файл `tests/integration.rs` подключает подмодули с помощью атрибута `#[path]`.

```rust
// tests/integration.rs
#[path = "integration/backends.rs"]
mod backends;
#[path = "integration/codegen.rs"]
mod codegen;
#[path = "integration/execution.rs"]
mod execution;
```

**Правило 7.2**: каждый файл `tests/integration/*.rs` соответствует одной теме (бэкенды компилятора, кодогенерация, исполнитель и т. д.); смешивание запрещено.

**Правило 7.3**: интеграционные тесты должны проводиться через публичный API проекта. В интеграционных тестах запрещено напрямую обращаться к внутренним модулям `crate::`. Следует использовать публичный путь `yaoxiang::`.

```rust
// 🟢 Хорошо — через публичный API
use yaoxiang::run;

// 🔴 Мусор — обход границ публичного API
use yaoxiang::middle::codegen::bytecode::BytecodeFile;
```

### Управление тестовыми данными

**Правило 8.1**: в интеграционных тестах предпочтительно использовать inline-строки с исходным кодом. Внешние fixture-файлы (в `tests/fixtures/`) применяются только при длине исходника свыше 30 строк.

```rust
#[test]
fn test_fibonacci() {
    run_ok(
        r#"
        main = {
            mut a = 0
            mut b = 1
            while a < 100 {
                mut next = a + b
                a = b
                b = next
            }
        }
        "#,
    );
}
```

**Правило 8.2**: fixture-файлы должны иметь расширение `.yx`; имя файла описывает назначение теста.

### Принципы покрытия E2E

**Правило 9.1**: интеграционные тесты для каждой возможности языка должны покрывать три пути:

| Путь | Описание |
|------|----------|
| Happy path | Корректный ввод даёт ожидаемый результат |
| Error path | Некорректный ввод даёт внятное сообщение об ошибке (а не панику) |
| Boundary | Граничные значения (пустой ввод, максимальное значение, предел глубины вложенности) |

**Правило 9.2**: интеграционные тесты не должны зависеть от сети, переменных окружения или внешних сервисов.

## Правила нагрузочных тестов

### Правила использования Criterion.rs

**Правило 10.1**: нагрузочные тесты единообразно размещаются в каталоге `benches/`, входной файл — `benches/lib.rs`. Файлы разделяются по темам.

```
benches/
├── lib.rs              # Входной файл, определяет criterion_group/criterion_main
├── lang_compare/
│   └── fibonacci.rs    # Кросс-языковые нагрузочные тесты
├── parser.rs           # Нагрузочные тесты парсера
└── codegen.rs          # Нагрузочные тесты кодогенерации
```

**Правило 10.2**: каждая нагрузочная функция должна содержать документационный комментарий `//!` с указанием цели теста и измеряемой метрики.

```rust
//! Нагрузочные тесты интерпретатора YaoXiang
//!
//! Метрика: время одной итерации (wall time)
//! Базовая линия: нативная реализация на Rust
```

### Предотвращение оптимизаций компилятора

**Правило 11.1**: весь измеряемый вывод в нагрузочных тестах должен проходить через `criterion::black_box`, чтобы компилятор не мог удалить его как мёртвый код.

```rust
use criterion::{black_box, Criterion};

fn bench_parse(c: &mut Criterion) {
    c.bench_function("parse_fib", |b| {
        b.iter(|| {
            let result = parse(black_box(FIB_SOURCE));
            black_box(result)
        })
    });
}
```

**Правило 11.2**: входные данные нагрузочного теста должны быть объявлены как `const` или `lazy_static`; динамическая генерация в замыкании `iter` запрещена — иначе измеряется суммарное время генерации данных и тестируемой логики.

### Группировка и именование нагрузочных тестов

**Правило 12.1**: формат имени нагрузочного теста — `<тестируемый модуль>_<сценарий>`, полностью в нижнем регистре с подчёркиваниями. Соответствует правилам именования модульных тестов.

**Правило 12.2**: для логической группировки связанных нагрузочных тестов необходимо использовать `criterion_group!`. Все тесты в одной группе запрещены.

```rust
criterion_group!(parser, bench_parse_expr, bench_parse_stmt);
criterion_group!(codegen, bench_codegen_module, bench_codegen_switch);
criterion_main!(parser, codegen);
```

## Правила документационных тестов

### Сценарии использования

**Правило 13.1**: для всех `pub`-функций, типов и методов в документационном комментарии должен быть как минимум один запускаемый пример кода. Этот пример выполняется через `cargo test --doc`.

```rust
/// Разбивает строку исходного кода на последовательность токенов.
///
/// ```
/// use yaoxiang::frontend::core::lexer::tokenize;
///
/// let tokens = tokenize("42").unwrap();
/// assert_eq!(tokens.len(), 2); // IntLiteral + Eof
/// ```
pub fn tokenize(source: &str) -> Result<Vec<Token>, LexError> {
    // ...
}
```

**Правило 13.2**: примеры кода в документационных тестах должны компилироваться и успешно проходить проверки. Использование `ignore` запрещено, кроме случаев, когда пример демонстрирует ошибку компиляции.

```rust
/// ```ignore
/// // Демонстрация ошибки компиляции — допустимо ignore
/// let x: int = "string";
/// ```
```

### Требования к покрытию

**Правило 14.1**: документационные тесты покрывают только happy path API. Граничные случаи и пути ошибок покрываются модульными тестами.

**Правило 14.2**: примеры кода в документационных тестах должны быть лаконичными — не более 10 строк. Если для примера нужен более длинный контекст, это указывает на проблему в дизайне API.

## Правила property-тестов

### Сценарии использования

**Правило 15.1**: в перечисленных сценариях необходимо использовать property-тесты (proptest или quickcheck), а не ручной набор граничных случаев:

| Сценарий | Пример |
|----------|--------|
| Round-trip парсера | `parse(pretty_print(ast)) == ast` |
| Сериализация/десериализация | `deserialize(serialize(data)) == data` |
| Тождества математических операций | `a + b == b + a` |
| Оптимизации компилятора не меняют семантику | `eval(code) == eval(optimize(code))` |

**Правило 15.2**: основным фреймворком property-тестов является `proptest` (уже объявлен в `dev-dependencies` файла `Cargo.toml`).

```rust
use proptest::prelude::*;

proptest! {
    #[test]
    fn test_roundtrip_serialize_deserialize(value: i64) {
        let serialized = serialize(&value);
        let deserialized: i64 = deserialize(&serialized).unwrap();
        prop_assert_eq!(deserialized, value);
    }
}
```

### Принципы формулировки свойств

**Правило 16.1**: каждый property-тест должен иметь явное объявление свойства — в комментарии указывается проверяемый инвариант.

```rust
// Свойство: для любого целочисленного литерала tokenize → tokens_to_string даёт то же значение
proptest! {
    #[test]
    fn test_int_literal_roundtrip(n in any::<i64>()) {
        let source = n.to_string();
        let tokens = tokenize(&source).unwrap();
        // ...
    }
}
```

**Правило 16.2**: если property-тест обнаруживает сбой, необходимо использовать механизм регрессий `proptest` — добавить не прошедший вход в каталог `proptest-regressions/`, а не подменять его обычным тестом, написанным вручную.

## Требования к покрытию

### Целевые показатели покрытия нового кода

**Правило 17.1**: требования к покрытию тестами нового кода:

| Тип кода | Покрытие строк | Покрытие ветвей |
|----------|----------------|-----------------|
| Модули ядра компилятора (frontend/middle/backends) | ≥ 85% | ≥ 80% |
| Утилитарные/вспомогательные модули (util) | ≥ 75% | ≥ 70% |
| Модули рантайма (vm/runtime) | ≥ 80% | ≥ 75% |
| Стандартная библиотека (std) | ≥ 75% | ≥ 70% |
| Обработка ошибок и диагностика | ≥ 90% | ≥ 85% |

**Правило 17.2**: пути обработки ошибок (все ветки `Err`) должны быть покрыты на 100%. Сообщения об ошибках, которые видит пользователь, должны быть проверены тестами.

### Чек-лист проверки PR

**Правило 18.1**: перед отправкой PR автор обязан самостоятельно проверить следующие пункты:

- [ ] `cargo test` проходит полностью
- [ ] `cargo test --doc` проходит полностью
- [ ] `cargo bench` не показывает регрессий производительности (если затронуты горячие пути)
- [ ] Новый код соответствует целевым показателям покрытия
- [ ] Именование тестов соответствует правилам именования
- [ ] Каждый тестовый файл объявляет соответствующий раздел спецификации (правило 2.1)
- [ ] Ожидаемые значения в тестах взяты из определения спецификации, а не из «вывода текущего кода»
- [ ] Отсутствуют тесты с `#[ignore]` (кроме случаев с явной ссылкой на issue)
- [ ] Отсутствуют лишние `unwrap()` (следует использовать `expect` или собственный panic-сообщение)
- [ ] Сообщение коммита использует тип `:white_check_mark: test:`
- [ ] **Ожидаемые значения в тестах не менялись из-за «несоответствия поведения кода спецификации» — правится код, а не тест**
- [ ] **Отсутствуют inline-тесты** (`#[cfg(test)] mod tests { ... }` должны быть заменены на `mod tests;` + отдельный файл, см. правило 1.4)

**Правило 18.2**: Reviewer обязан отклонить PR, содержащий следующие проблемы:

- Тесты только для happy path, отсутствуют пути ошибок
- В тестах используется `thread::sleep` или зависимость от порядка выполнения
- Скопированный тестовый код встречается более 3 раз без вынесения во вспомогательную функцию
- Имена тестов не соответствуют правилам именования
- Присутствуют тесты с постоянным `#[ignore]`
- **Тест подстраивается под ошибочное поведение кода** (при несоответствии кода спецификации правится тест, а не код)
- **Тест не объявляет соответствующий раздел спецификации** (см. правило 2.1)
- **Ожидаемые значения в тесте взяты из вывода кода, а не из определения спецификации** (тест, построенный обратным выводом, равен отсутствию теста)
- **Присутствуют inline-тесты** (`#[cfg(test)] mod tests { ... }` вместо `mod tests;` + отдельный файл, см. правило 1.4)
- Тест проверяет только «отсутствие паники», а не конкретное поведение
- Удалён упавший тест, выявляющий баг в коде (вместо того чтобы исправить код и увидеть зелёный сигнал)

## Приложения

### A. Шпаргалка по командам тестирования

```bash
# Запустить все тесты
cargo test

# Только модульные тесты
cargo test --lib

# Только интеграционные тесты
cargo test --test integration

# Только документационные тесты
cargo test --doc

# Запустить конкретный тест (фильтр по имени)
cargo test test_parse_expr

# Запустить нагрузочные тесты
cargo bench

# Показывать вывод тестов (по умолчанию скрыт)
cargo test -- --nocapture

# Однопоточное выполнение (для отладки проблем параллелизма)
cargo test -- --test-threads=1

# Генерация отчёта о покрытии (требуется cargo-llvm-cov)
cargo llvm-cov --html
```

### B. Шаблон сообщения коммита

Коммиты, относящиеся к тестам, должны следовать шаблону:

```
:white_check_mark: test(<scope>): <краткое описание>

<необязательно: список покрытых сценариев>
```

Пример:

```
:white_check_mark: test(parser): добавить тесты инфиксных операторов парсера Pratt

Покрытые сценарии:
- Приоритеты арифметических операторов (+, -, *, /, %)
- Цепочки операторов сравнения (1 < x < 10)
- Короткое замыкание логических операторов
- Правоассоциативность оператора присваивания
```

### C. Чек-лист при создании нового тестового файла

При создании нового тестового модуля убедитесь, что присутствуют следующие файлы:

```
# Добавление тестов в src/<module>/
src/<module>/tests/
├── mod.rs          # Объявление модулей + общие вспомогательные функции
└── <subject>.rs    # Тестовый файл, именованный по исходному файлу

# Добавление интеграционных тестов в tests/
tests/
├── integration.rs   # Обновить: добавить объявление #[path]
└── integration/
    └── <topic>.rs   # Новый тестовый файл
```
