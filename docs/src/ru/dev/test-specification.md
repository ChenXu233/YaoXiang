## Общие положения

### Область применения

Настоящий стандарт распространяется на весь код тестов на Rust в проекте YaoXiang:

| Тип теста | Расположение | Фреймворк |
|-----------|--------------|-----------|
| Модульные тесты | `src/<module>/tests/` | `#[test]` + `#[cfg(test)]` |
| Интеграционные тесты | `tests/` | `#[test]` |
| Бенчмарки | `benches/` | Criterion.rs |
| Документационные тесты | Комментарии в API-документации | `cargo test --doc` |
| Тесты на свойства | В любом месте тестов | proptest / quickcheck |

### Основные принципы

**Принцип 0: Единственный авторитетный источник для тестов — спецификация, а не код.** Это важнейший принцип данного документа. Тесты проверяют, соответствует ли код спецификации, а не то, «заработал ли код в текущей реализации». Когда тест обнаруживает несоответствие поведения кода спецификации, **исправляется код, тест никогда не исправляется**.

Файлы спецификаций находятся в:
- `docs/src/design/language-spec.md` — ядро языковой спецификации
- `docs/src/design/rfc/accepted/` — принятые RFC-дизайны

Каждый файл тестов должен объявлять соответствующую секцию спецификации в заголовке (см. правило 2.1). Любой разработчик должен иметь возможность взять документ спецификации и, сверяясь с тестами, проверить корректность реализации. И наоборот — если фрагмент кода не имеет соответствующего описания в спецификации, его не должно существовать, и тестировать его не нужно.

```rust
// ✅ Хорошо — тест напрямую ссылается на спецификацию, проверяя соответствие кода
//! Тесты литералов — на основе спецификации языка §2.6
//!
//! §2.6.1: Целочисленные Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: F-String интерполяция

#[test]
fn test_decimal_literal_parsing() {
    // Спецификация §2.6.1: Decimal ::= [0-9][0-9_]*
    let result = parse_literal("42").unwrap();
    assert_eq!(result, Literal::Int(42));
}

// 🔴 Плохо — тест подстраивается под текущее поведение кода, а не проверяет спецификацию
#[test]
fn test_literal_1() {
    // Непонятно, к какой секции спецификации относится этот тест
    // Если parse_literal вернёт неправильное значение, тест «пройдёт»
    // Потому что он лишь проверяет, что функция не паникует
    let result = parse_literal("42");
    assert!(result.is_ok());
}
```

**Ситуация**: Вы написали тест и обнаружили, что код не соответствует спецификации. У вас есть два варианта:
| Неправильный подход | Правильный подход |
|---------------------|-------------------|
| Изменить тест, чтобы он «проходил» | Исправить код, чтобы он соответствовал спецификации |
| Добавить `#[ignore]` в тест | Немедленно исправить реализацию |
| Добавить в тест условные ветки, подстраивающиеся под код | Удалить ветки, позволив тесту выявить проблему |

Помните: **Красный тест = ошибка в коде, а не в тесте.** (Если только сам тест не содержит ошибку — это другое дело.)

**Принцип 1: Тесты — это документация.** Любой разработчик должен понять поведение тестируемого кода, читая тесты, без дополнительных комментариев или внешней документации.

```rust
// ✅ Хорошо — имя теста говорит, что тестируется и что ожидается
#[test]
fn test_tokenize_empty_input_returns_eof() {
    let tokens = tokenize("").unwrap();
    assert_eq!(tokens.len(), 1);
    assert!(matches!(tokens[0].kind, TokenKind::Eof));
}

// 🔴 Плохо — непонятно, что именно тестируется
#[test]
fn test_tokenize_1() {
    let tokens = tokenize("").unwrap();
    assert!(tokens.len() > 0);
}
```

**Принцип 2: Нулевая терпимость к случайным сбоям.** Тесты должны воспроизводиться в любом окружении. Тесты, зависящие от случайных чисел, системного времени или порядка планирования потоков, должны использовать фиксированные сиды или mock-объекты.

**Принцип 3: Один тест проверяет одно поведение.** Если для названия теста нужно использовать «и» для соединения нескольких действий, разделите его на несколько тестов.

```rust
// ✅ Хорошо — каждый тест проверяет один сценарий
#[test]
fn test_parse_int_positive() { /* ... */ }
#[test]
fn test_parse_int_zero() { /* ... */ }

// 🔴 Плохо — в одном тесте слишком много разнородного
#[test]
fn test_parser() {
    // Тестируем tokenize, parse, typecheck, codegen...
}
```

**Принцип 4: Тестируем поведение, а не реализацию.** Рефакторинг внутренней реализации не должен приводить к падению тестов. Если изменение одной строки реализации привело к падению 10 тестов — тесты написаны неправильно.

Но здесь важно различие: **определение «поведения» исходит из спецификации, а не из текущего поведения кода.** Если код изменил поведение (т.е. появилось новое поведение, не соответствующее спецификации), тесты должны падать. Если этого не происходит — ваши тесты «подстраиваются под код» — они пропускают баги.

```
Спецификация (language-spec.md / RFC)  ──определяет──►  Ожидаемое поведение  ──управляет──►  Тесты
                                           │
Текущий код  ──реализует──►  Фактическое поведение  ──сравнивается──►  Результат теста

Если фактическое ≠ ожидаемое:
  Тест должен упасть (красный)  ──►  Исправление кода  ──►  Тест проходит (зелёный)
  
Если фактическое = ожидаемое (но реализация плохая):
  Тест проходит  ──►  Рефакторинг реализации  ──►  Тест всё ещё проходит  ← Вот смысл принципа 4
```

**Принцип 5: Не пишите тестовый код с обходными/совместимыми/специфичными паттернами.** Тестовое окружение полностью под вашим контролем. Если вам нужен `#[cfg(not(ci))]` для пропуска теста — у теста фундаментальная проблема дизайна.

### Определения терминов

| Термин | Определение |
|--------|-------------|
| Модульный тест | Тестирование отдельной функции или модуля без внешних зависимостей |
| Интеграционный тест | Тестирование взаимодействия нескольких модулей через публичный API или CLI |
| Бенчмарк | Измерение производительности кода, обнаружение регрессий |
| Документационный тест | Исполняемые примеры кода, встроенные в документацию |
| Тест на свойства | Тестирование инвариантов на случайных входных данных |

### Связь со стандартом коммитов

Все коммиты, связанные с тестами, должны использовать тип `:white_check_mark: test:`, согласно [стандарту коммитов](./commit-convention.md).

```
:white_check_mark: test(parser): добавить тесты инфиксных выражений для Pratt-парсера
:white_check_mark: test(codegen): дополнить тесты генерации IR для switch
```

## Стандарты модульных тестов

### Организация файлов

**Правило 1.1**: Каталог `tests/` для модульных тестов должен располагаться **на одном уровне** с тестируемым модулем `mod.rs`. `tests/` не агрегируется вверх и не суммируется кросс-уровнево.

```
src/frontend/core/parser/
├── mod.rs              # #[cfg(test)] mod tests; —— объявляет tests/ на том же уровне
├── ast.rs
├── pratt/
│   ├── mod.rs          # #[cfg(test)] mod tests; —— свои тесты pratt
│   └── tests/
│       ├── mod.rs
│       ├── led.rs
│       ├── nud.rs
│       └── precedence.rs
└── tests/              # тесты уровня модуля parser (без тестов подмодуля pratt)
    ├── mod.rs
    ├── ast.rs
    ├── expressions.rs
    ├── error_recovery.rs
    └── parser_state.rs
```

Ключевой критерий: **`tests/` располагается в том каталоге, в `mod.rs` которого должен быть объявлен `#[cfg(test)] mod tests;`.**

#### Правила размещения тестов: модуль-файл против модуля-каталога

**Основное различие**: форма организации модуля определяет место тестов.

| Тип модуля | Критерий | Расположение тестов | Пример |
|------------|----------|---------------------|--------|
| **Модуль-каталог** | Есть отдельный каталог и `mod.rs` | `tests/` в этом каталоге | `inference/tests/` |
| **Модуль-файл** | Только `.rs` файл, без отдельного каталога | `tests/` родительского модуля | `overload.rs` → `typecheck/tests/overload.rs` |

**Подробное объяснение**:

```
src/frontend/core/typecheck/
├── mod.rs                          # mod.rs модуля typecheck
├── checker.rs                      # модуль-файл
├── environment.rs                  # модуль-файл
├── overload.rs                     # модуль-файл
├── type_eval.rs                    # модуль-файл
├── dead_code.rs                    # модуль-файл
├── spawn_placement.rs              # модуль-файл
├── signature.rs                    # модуль-файл
├── types.rs                        # модуль-файл
│
├── tests/                          # ✅ тесты typecheck
│   ├── mod.rs                      # объявляет тесты модулей-файлов
│   ├── checker.rs                  # тесты checker.rs
│   ├── environment.rs              # тесты environment.rs
│   ├── overload.rs                 # тесты overload.rs (модуль-файл тестируется здесь)
│   ├── type_eval.rs                # тесты type_eval.rs
│   ├── dead_code.rs                # тесты dead_code.rs
│   ├── spawn_placement.rs          # тесты spawn_placement.rs
│   ├── signature.rs                # тесты signature.rs
│   └── types.rs                    # тесты types.rs
│
├── inference/                      # модуль-каталог (есть mod.rs)
│   ├── mod.rs                      # #[cfg(test)] mod tests; —— объявляет tests/ на том же уровне
│   ├── expressions.rs
│   ├── statements.rs
│   ├── patterns.rs
│   ├── bounds.rs
│   ├── subtyping.rs
│   ├── generics.rs
│   ├── compatibility.rs
│   ├── scope.rs
│   ├── assignment.rs
│   └── tests/                      # ✅ тесты inference
│       ├── mod.rs
│       ├── expressions.rs          # тесты expressions.rs
│       ├── statements.rs           # тесты statements.rs
│       └── ...
│
└── traits/                         # модуль-каталог (есть mod.rs)
    ├── mod.rs                      # #[cfg(test)] mod tests; —— объявляет tests/ на том же уровне
    ├── solver.rs
    ├── impl_check.rs
    ├── inheritance.rs
    ├── coherence.rs
    ├── auto_derive.rs
    ├── object_safety.rs
    ├── resolution.rs
    ├── std_traits.rs
    ├── gat/
    ├── specialization/
    └── tests/                      # ✅ тесты traits
        ├── mod.rs
        ├── solver.rs               # тесты solver.rs
        ├── impl_check.rs           # тесты impl_check.rs
        └── ...
```

**Почему тесты модуля-файла помещаются в родительский `tests/`?**

Потому что модуль-файл (например, `overload.rs`) не имеет собственного `mod.rs`, и не может объявить `#[cfg(test)] mod tests;`. Согласно модульной системе Rust, тестовые файлы должны быть объявлены в каком-либо `mod.rs` для компиляции. Поэтому тесты модуля-файла могут быть объявлены только родительским `mod.rs` и располагаться в родительском `tests/`.

**Алгоритм принятия решения**:

```
Встретили модуль — куда поместить тесты?
│
├── Это каталог (есть mod.rs)?
│   └── Да → Создать tests/ в этом каталоге, объявить в этом mod.rs
│
├── Это файл (только .rs)?
│   └── Да → Тесты в tests/ родительского каталога, объявить в родительском mod.rs
│
└── Не уверены?
    └── Проверить наличие отдельного каталога и mod.rs
```

**Типичные ошибки**:

```
# ❌ Ошибка 1: создание отдельного tests/ для модуля-файла
src/frontend/core/typecheck/
├── overload.rs
└── overload/                       # ❌ Не нужно создавать каталог для модуля-файла
    └── tests/
        └── overload.rs

# ❌ Ошибка 2: объявление #[cfg(test)] mod tests; в модуле-файле
# overload.rs
#[cfg(test)]                        # ❌ Модуль-файл не может так объявлять
mod tests;                          # потому что нет каталога overload/tests/

# ✅ Правильно: тесты в родительском tests/
src/frontend/core/typecheck/
├── overload.rs                     # исходный файл
└── tests/
    └── overload.rs                 # тестовый файл, объявлен в typecheck/mod.rs
```

⚠️ **Антипаттерн — не делайте так:**

```
# ❌ Ошибка: тесты подмодулей собраны в родительском
src/frontend/core/types/
├── mod.rs              # должен объявлять только base и computation
├── base/
│   ├── mod.rs
│   └── var.rs
└── tests/              # ❌ Родительский tests/ содержит тесты подмодулей
    ├── mod.rs          # ❌ Вынужден объявлять mod base; mod computation;
    ├── base/           # ❌ Эта часть должна быть в base/tests/
    │   └── var.rs
    └── computation/    # ❌ Эта часть должна быть в computation/tests/
        └── ...
```

```
# ✅ Правильно: тесты каждого модуля независимы
src/frontend/core/types/
├── mod.rs              # объявляет только pub mod base; pub mod computation;
├── base/
│   ├── mod.rs          # #[cfg(test)] mod tests; —— объявляет tests/ на том же уровне
│   ├── var.rs
│   └── tests/
│       ├── mod.rs
│       └── var.rs
└── computation/
    ├── mod.rs          # #[cfg(test)] mod tests; —— объявляет tests/ на том же уровне
    ├── operations.rs
    └── tests/
        ├── mod.rs
        └── operations.rs
```

**Почему нельзя агрегировать вверх?** Потому что модульная система Rust требует, чтобы `#[cfg(test)] mod tests;` решал вопрос компиляции тестов в точке объявления. Если `types/mod.rs` объявляет `mod tests;`, то содержимое `types/tests/` является приватным содержимым модуля `types` — оно не должно проникать в области `base` или `computation`. Тесты каждого модуля должны быть деталями реализации этого модуля, а не родительского. Это правило также применимо при рефакторинге модулей: при разделении `types` на `base` и `computation` тесты также должны следовать за разделёнными модулями, а не оставаться на месте. **Структура каталога тестов не зеркалирует структуру исходников, а следует за границами модулей.**

**Правило 1.2**: `tests/mod.rs` отвечает только за объявление модулей и re-export, без тестовых функций.

```rust
//! Тесты ядра парсера — зеркало src/frontend/core/parser/
//!
//! Тесты для ast.rs, parser_state.rs и разбора выражений/интеграции.

mod ast;
mod error_recovery;
mod expressions;
mod integration;
mod parser_state;
```

**Правило 1.3**: Каждый тестовый файл соответствует одному исходному файлу. Недопустимо смешивать тесты нескольких исходных модулей в одном файле.

**Правило 1.4**: `#[cfg(test)]` может появляться только в двух местах — в `lib.rs` для объявления `mod tests`, или inline в тестируемом исходном файле `#[cfg(test)] mod tests;`. В других местах использование запрещено.

```rust
// src/frontend/core/parser/mod.rs или lib.rs
#[cfg(test)]
mod tests;
```

### Стандарты объявления модулей

**Правило 2.1**: Все тестовые файлы должны содержать модульную документацию `//!` в заголовке, указывающую источник тестируемых спецификаций (номера секций спецификации языка + номера RFC). Если тест не ссылается ни на одну секцию спецификации — этот код не имеет обоснования в спецификации и не должен существовать.

```rust
//! Тесты литералов — на основе спецификации языка §2.6
//!
//! §2.6.1: Целочисленные Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: F-String интерполяция
```

**Почему обязательна ссылка на спецификацию?** Потому что ожидаемые значения тестов происходят из спецификации, а не из «текущего вывода кода». Если有一天 код изменит вывод, а тест будет обновлён вместе с ним — такой тест ничего не защищает. Только тесты, привязанные к спецификации, могут отличить «намеренное breaking change» от «непреднамеренной регрессии».

**Правило 2.2**: Импорты `use` в тестовых модулях должны быть точными до конкретного типа/функции, запрещены glob-импорты `use super::*`.

```rust
// ✅ Хорошо — точный импорт
use crate::frontend::core::lexer::{tokenize, TokenKind};
use crate::frontend::core::parser::{ParserState, ParseError};

// 🔴 Плохо — непонятно, что именно тестируется
use super::*;
```

### Стандарты именования

**Правило 3.1**: Имена тестовых функций имеют формат `test_<что>_<сценарий>`, все буквы строчные, слова разделены подчёркиванием.

```rust
#[test]
fn test_tokenize_empty_string() { /* ... */ }
#[test]
fn test_parse_int_overflow() { /* ... */ }
#[test]
fn test_typecheck_fn_return_mismatch() { /* ... */ }
```

**Правило 3.2**: Имя тестовой функции должно быть самодокументирующимся. Прочитав имя, должно быть понятно, что тестируется и что ожидается. Запрещено именование числовыми индексами.

```rust
// ✅ Хорошо
fn test_skip_semicolon_success() { /* ... */ }
fn test_skip_semicolon_failure_when_identifier() { /* ... */ }

// 🔴 Плохо — совершенно непонятно, что тестируется
fn test_skip_1() { /* ... */ }
fn test_skip_2() { /* ... */ }
```

**Правило 3.3**: Вспомогательным функциям не нужен префикс `test_`, следует использовать глаголы или существительные, описывающие их назначение.

```rust
fn parse_expr(source: &str) -> Expr { /* ... */ }
fn tokenize_single(source: &str) -> Token { /* ... */ }
fn setup_parser_with_tokens(tokens: &[Token]) -> ParserState { /* ... */ }
```

### Стандарты структуры тестов (Arrange-Act-Assert)

**Правило 4.1**: Каждая тестовая функция должна следовать трёхсекционной структуре: Подготовка (Arrange) → Действие (Act) → Проверка (Assert), секции разделяются пустой строкой.

```rust
#[test]
fn test_parse_binary_addition() {
    // Arrange
    let source = "1 + 2";

    // Act
    let expr = parse_expr(source);

    // Assert
    assert!(matches!(expr, Expr::Binary { op: BinOp::Add, .. }));
}
```

**Правило 4.2**: Простые тесты (одиночный вызов + одиночная проверка) могут не содержать комментариев секций, но не должны превышать 5 строк логического кода. Тесты свыше 5 строк должны явно размечать три секции.

### Стандарты вспомогательных функций

**Правило 5.1**: Повторяющийся 3 и более раз код подготовки должен быть извлечён во вспомогательную функцию.

```rust
// ✅ Хорошо — вынесена общая подготовка
fn with_state<F>(source: &str, mut f: F)
where
    F: FnMut(&mut ParserState<'_>),
{
    let tokens = tokenize(source).unwrap();
    let mut state = ParserState::new(&tokens);
    f(&mut state);
}

#[test]
fn test_current_returns_first_token() {
    with_state("42", |state| {
        let tok = state.current();
        assert_eq!(&tok.unwrap().kind, &TokenKind::IntLiteral(42));
    });
}
```

**Правило 5.2**: `unwrap()` / `expect()` во вспомогательных функциях должны при панике выводить достаточно контекста. В теле тестовой функции (`#[test] fn ...`) можно использовать `unwrap()` напрямую — при неудаче Rust автоматически выводит номер строки; но при неудаче во вспомогательной функции номер строки указывает на определение вспомогательной функции, без контекста вызова.

```rust
// ✅ Хорошо — при неудаче выводится содержимое исходного кода
fn run_ok(source: &str) {
    run(source).unwrap_or_else(|e| panic!("Execution failed:\nSource:\n{}\nError:\n{:?}", source, e));
}

// 🔴 Плохо — при неудаче непонятно, какой исходный файл вызвал проблему
fn run_ok(source: &str) {
    run(source).unwrap();
}
```

**Правило 5.3**: Вспомогательные функции располагаются в верхней части тестового файла, сразу после импортов `use`. Если используются несколькими тестовыми модулями — помещаются в `tests/mod.rs` и экспортируются как `pub(crate)`.

### Стиль проверок (assertions)

**Правило 6.1**: Для сопоставления вариантов перечисления предпочтителен `assert!(matches!(...))`, запрещено использование `if let` + `panic!`.

```rust
// ✅ Хорошо
assert!(matches!(tokens[0].kind, TokenKind::IntLiteral(42)));

// 🔴 Плохо
if let TokenKind::IntLiteral(v) = tokens[0].kind {
    assert_eq!(v, 42);
} else {
    panic!("Expected IntLiteral");
}
```

**Правило 6.2**: Точные сравнения значений используют `assert_eq!`, булевы проверки — `assert!`. Запрещено использовать `assert!(a == b)` вместо `assert_eq!(a, b)`.

**Правило 6.3**: Все проверки должны содержать собственное сообщение об ошибке, если только сама проверка уже полностью описывает причину неудачи.

```rust
// ✅ Хорошо — при неудаче легко локализовать
assert!(
    state.infix_info().is_some(),
    "infix_info should handle '{op}'"
);

// ✅ Хорошо — assert_eq! при неудаче автоматически показывает разницу значений
assert_eq!(error_count, 0);

// 🔴 Плохо — при неудаче只知道 "assertion failed"
assert!(state.infix_info().is_some());
```

**Правило 6.4**: Порядок в проверках: `assert_eq!(actual, expected)` — фактическое значение первым, ожидаемое вторым.

### Чёрный список антипаттернов

Запрещённые写法及其替代方案:

| Антипаттерн | Проблема | Решение |
|-------------|----------|---------|
| Тест подстраивается под ошибочное поведение кода | Скрывает отклонение от спецификации, легализует баги | Сверить с спецификацией и исправить код, тест не менять |
| Обратный вывод ожидаемых значений из вывода кода | Тест становится «диктофоном текущей реализации» | Выводить ожидаемые значения из спецификации |
| Постоянная метка `#[ignore]` | Скрывает устаревшие тесты | Исправить или удалить |
| Отладочный вывод `println!` | Загрязняет вывод тестов | Использовать `assert!` с явными проверками |
| `thread::sleep` | Случайные сбои + медленно | Использовать синхронизацию или mock |
| Работа с реальной файловой системой в тестах | Медленно и невоспроизводимо | Использовать `tempfile` |
| Зависимость от порядка выполнения тестов | Случайные сбои | Каждый тест имеет независимую подготовку |
| Одна тестовая функция более 30 строк | Невозможно понять | Разделить или использовать вспомогательные функции |
| copy-paste одинаковой подготовки 3+ раза | Высокая стоимость изменений | Вынести во вспомогательную функцию |

## Стандарты интеграционных тестов

### Организация тестов

**Правило 7.1**: Интеграционные тесты располагаются в каталоге `tests/` корня проекта. Входной файл `tests/integration.rs` использует атрибут `#[path]` для подключения подмодулей.

```rust
// tests/integration.rs
#[path = "integration/backends.rs"]
mod backends;
#[path = "integration/codegen.rs"]
mod codegen;
#[path = "integration/execution.rs"]
mod execution;
```

**Правило 7.2**: Каждый файл `tests/integration/*.rs` соответствует одной теме тестирования (бэкенды компилятора, генерация кода, исполнитель и т.д.), смешивание запрещено.

**Правило 7.3**: Интеграционные тесты должны использовать публичный API проекта. Запрещено обращение к внутренним модулям `crate::` в интеграционных тестах. Использовать публичный путь `yaoxiang::`.

```rust
// ✅ Хорошо — через публичный API
use yaoxiang::run;

// 🔴 Плохо —绕过 публичный API
use yaoxiang::middle::codegen::bytecode::BytecodeFile;
```

### Управление тестовыми данными

**Правило 8.1**: Интеграционные тесты предпочитают использовать инлайновые строки исходного кода. Внешние fixture-файлы (в `tests/fixtures/`) используются только когда исходный код превышает 30 строк.

```rust
#[test]
fn test_fibonacci() {
    run_ok(
        r#"
        main = {
            mut a = 0
            mut b = 1
            while a < 100 {
                mut next = a + b
                a = b
                b = next
            }
        }
        "#,
    );
}
```

**Правило 8.2**: Fixture-файлы должны иметь расширение `.yx`, имя файла описывает назначение теста.

### Принципы E2E покрытия

**Правило 9.1**: Интеграционные тесты каждой языковой возможности должны покрывать три пути:

| Путь | Описание |
|------|----------|
| Happy path | Корректный вход даёт ожидаемый выход |
| Error path | Некорректный вход даёт чёткое сообщение об ошибке (не panic) |
| Boundary | Граничные значения (пустой ввод, максимальные значения, предел вложенности) |

**Правило 9.2**: Интеграционные тесты не должны зависеть от сети, переменных окружения или внешних сервисов.

## Стандарты бенчмарков

### Стандарты использования Criterion.rs

**Правило 10.1**: Бенчмарки统一放在 `benches/` 目录，入口文件为 `benches/lib.rs`。按测试主题分文件。

```
benches/
├── lib.rs              # 入口，定义 criterion_group/criterion_main
├── lang_compare/
│   └── fibonacci.rs    # 跨语言对比基准
├── parser.rs           # 解析器基准
└── codegen.rs          # 代码生成基准
```

**Правило 10.2**: 每个基准函数必须包含模块文档注释 `//!` 说明测试目的和测量指标。

```rust
//! YaoXiang 解释器性能基准测试
//!
//! 测量指标：单次迭代耗时（wall time）
//! 基准线：Rust 原生实现
```

### Предотвращение оптимизаций компилятора

**Правило 11.1**: Все выходы тестируемого кода в бенчмарках必须通过 `criterion::black_box` 阻止编译器优化消除。

```rust
use criterion::{black_box, Criterion};

fn bench_parse(c: &mut Criterion) {
    c.bench_function("parse_fib", |b| {
        b.iter(|| {
            let result = parse(black_box(FIB_SOURCE));
            black_box(result)
        })
    });
}
```

**Правило 11.2**: Бенчмарки的输入数据必须是 `const` 或 `lazy_static`，不得在 `iter` 闭包内动态生成——否则测量的是数据生成 + 被测逻辑的总时间。

### 基准分组与命名

**Правило 12.1**: 基准测试命名格式为 `<被测模块>_<场景>`，全小写下划线分隔。与单元测试命名规则一致。

**Правило 12.2**: 必须使用 `criterion_group!` 对相关基准进行逻辑分组。禁止所有基准挤在一个分组中。

```rust
criterion_group!(parser, bench_parse_expr, bench_parse_stmt);
criterion_group!(codegen, bench_codegen_module, bench_codegen_switch);
criterion_main!(parser, codegen);
```

## Стандарты文档测试

### 使用场景

**Правило 13.1**: 所有 `pub` 函数、类型、方法必须在文档注释中包含至少一个可运行的代码示例。该示例通过 `cargo test --doc` 执行。

```rust
/// 将源码字符串分词为 Token 序列。
///
/// ```
/// use yaoxiang::frontend::core::lexer::tokenize;
///
/// let tokens = tokenize("42").unwrap();
/// assert_eq!(tokens.len(), 2); // IntLiteral + Eof
/// ```
pub fn tokenize(source: &str) -> Result<Vec<Token>, LexError> {
    // ...
}
```

**Правило 13.2**: 文档测试的代码示例必须编译通过且断言成功。不得包含 `ignore` 标记的示例，除非该示例展示的是编译期错误。

```rust
/// ```ignore
/// // 展示编译期错误——可以 ignore
/// let x: int = "string";
/// ```
```

### 覆盖要求

**Правило 14.1**: 文档测试覆盖 API 的 happy path 即可。边界情况和错误路径由单元测试覆盖。

**Правило 14.2**: 文档测试中的示例代码必须简洁——不超过 10 行。如果示例需要更长的上下文，说明 API 设计有问题。

## 属性测试规范

### 使用场景

**Правило 15.1**: 以下场景必须使用属性测试（proptest 或 quickcheck）而不是手写多个边界值用例：

| 场景 | 示例 |
|------|------|
| 解析器 round-trip | `parse(pretty_print(ast)) == ast` |
| 序列化/反序列化 | `deserialize(serialize(data)) == data` |
| 数学运算恒等式 | `a + b == b + a` |
| 编译器优化不改变语义 | `eval(code) == eval(optimize(code))` |

**Правило 15.2**: 属性测试使用 `proptest` 作为主要的属性测试框架（已在 `Cargo.toml` 的 `dev-dependencies` 中声明）。

```rust
use proptest::prelude::*;

proptest! {
    #[test]
    fn test_roundtrip_serialize_deserialize(value: i64) {
        let serialized = serialize(&value);
        let deserialized: i64 = deserialize(&serialized).unwrap();
        prop_assert_eq!(deserialized, value);
    }
}
```

### 属性定义原则

**Правило 16.1**: 每个属性测试必须有明确的属性声明——注释中写明验证的不变量。

```rust
// 属性：任意整数字面量在 tokenize → tokens_to_string 后产生相同值
proptest! {
    #[test]
    fn test_int_literal_roundtrip(n in any::<i64>()) {
        let source = n.to_string();
        let tokens = tokenize(&source).unwrap();
        // ...
    }
}
```

**Правило 16.2**: 如果属性测试发现失败，必须使用 `proptest` 的回归机制——将失败的输入添加到 `proptest-regressions/` 目录，不要手写一个普通测试代替。

## 覆盖率要求

### 新增代码覆盖率目标

**Правило 17.1**: 新增代码的测试覆盖率要求：

| 代码类型 | 行覆盖率 | 分支覆盖率 |
|----------|----------|------------|
| 核心编译器模块（frontend/middle/backends） | ≥ 85% | ≥ 80% |
| 工具/辅助模块（util） | ≥ 75% | ≥ 70% |
| 运行时模块（vm/runtime） | ≥ 80% | ≥ 75% |
| 标准库（std） | ≥ 75% | ≥ 70% |
| 错误处理和诊断 | ≥ 90% | ≥ 85% |

**Правило 17.2**: 错误处理路径（所有 `Err` 分支）必须 100% 覆盖。用户能看到的错误信息必须被测试验证过。

### PR 审查检查清单

**Правило 18.1**: 提交 PR 前，作者必须自查以下项目：

- [ ] `cargo test` 全部通过
- [ ] `cargo test --doc` 全部通过
- [ ] `cargo bench` 无性能回归（如涉及热路径变更）
- [ ] 新增代码符合覆盖率目标
- [ ] 测试命名符合命名规范
- [ ] 每个测试文件声明了对应的规范章节（规则 2.1）
- [ ] 测试期望值来自规范定义，而非"当前代码的输出"
- [ ] 无 `#[ignore]` 标记的测试（除非有明确 issue 号注释）
- [ ] 无不必要的 `unwrap()` （应使用 `expect` 或自定义 panic 消息）
- [ ] 提交信息使用 `:white_check_mark: test:` 类型
- [ ] **没有因为"代码行为与规范不符"而修改测试期望值——改的是代码，不是测试**

**Правило 18.2**: Reviewer 必须拒绝包含以下问题的 PR：

- 只有 happy path 测试，缺少错误路径
- 测试中有 `thread::sleep` 或依赖执行顺序
- 复制粘贴的测试代码超过 3 次而未提取辅助函数
- 测试名不符合命名规范
- 存在永久 `#[ignore]` 的测试
- **测试迁就代码的错误行为**（代码与规范不符时修改测试而非修改代码）
- **测试没有声明对应的规范章节**（参见规则 2.1）
- **测试期望值来自代码输出而非规范定义**（反推出来的测试等于没测）
- 测试只验证"不 panic"而不断言具体行为
- 删除了暴露代码 bug 的失败测试（而不是修复代码后再看到它变绿）
