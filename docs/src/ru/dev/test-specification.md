## Общие положения

### Область применения

Настоящие нормы распространяются на весь тестовый код Rust в проекте YaoXiang, включая:

| Тип теста | Расположение | Фреймворк |
|----------|------|------|
| Модульные тесты | `src/<модуль>/tests/` | `#[test]` + `#[cfg(test)]` |
| Интеграционные тесты | `tests/` | `#[test]` |
| Нагрузочные тесты | `benches/` | Criterion.rs |
| Документационные тесты | Комментарии в документации API | `cargo test --doc` |
| Property-тесты | Любое место в тестах | proptest / quickcheck |

### Основные принципы

**Принцип 0: Авторитетным источником для тестов является спецификация, а не код.** Это самый важный принцип настоящего документа. Тесты проверяют, соответствует ли код спецификации, а не «работает ли код при текущей реализации». Когда тест обнаруживает расхождение между поведением кода и спецификацией, **исправляется код, никогда — тест**.

Файлы спецификации расположены по адресам:
- `docs/src/design/language-spec.md` — базовая спецификация языка
- `docs/src/design/rfc/accepted/` — принятые дизайн-документы RFC

В начале каждого тестового файла должна быть ссылка на соответствующий раздел спецификации (см. правило 2.1). Любой разработчик должен иметь возможность взять документ спецификации и сверить с ним тесты, чтобы убедиться в корректности реализации. И наоборот: если для фрагмента кода нет соответствующего описания в спецификации, его не должно существовать, и тем более его не нужно тестировать.

```rust
// 🟢 Хорошо — тест прямо ссылается на спецификацию и проверяет её соблюдение
//! Тесты литералов — на основе §2.6 спецификации языка
//!
//! §2.6.1: Целые числа Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: F-строковая интерполяция

#[test]
fn test_decimal_literal_parsing() {
    // Спецификация §2.6.1: Decimal ::= [0-9][0-9_]*
    let result = parse_literal("42").unwrap();
    assert_eq!(result, Literal::Int(42));
}

// 🔴 Мусор — тест подстраивается под текущее поведение кода, а не под спецификацию
#[test]
fn test_literal_1() {
    // Непонятно, какому разделу спецификации соответствует этот код
    // Если parse_literal вернёт неверное значение, тест «пройдёт на зелёном»,
    // потому что проверяет лишь отсутствие паники
    let result = parse_literal("42");
    assert!(result.is_ok());
}
```

**Сценарий**: вы написали тест и обнаружили, что поведение кода не соответствует спецификации. У вас два варианта:
| Неправильно | Правильно |
|----------|----------|
| Изменить тест, чтобы «прошёл» | Изменить код, чтобы поведение соответствовало спецификации |
| Добавить в тест `#[ignore]` | Немедленно исправить реализацию |
| Добавить особую ветку условия в тест, подгоняющуюся под код | Удалить ветку и дать тесту явно выявить проблему |

Запомните: **красный = код неправильный, а не тест.** (Если только в самом тесте нет бага — это уже другой разговор.)

**Принцип 1: Тест — это документация.** Любой разработчик должен мочь, прочитав тест, понять поведение тестируемого кода без дополнительных комментариев или внешней документации.

```rust
// 🟢 Хорошо — из имени теста понятно, что и с каким ожиданием тестируется
#[test]
fn test_tokenize_empty_input_returns_eof() {
    let tokens = tokenize("").unwrap();
    assert_eq!(tokens.len(), 1);
    assert!(matches!(tokens[0].kind, TokenKind::Eof));
}

// 🔴 Мусор — непонятно, что вообще тестируется
#[test]
fn test_tokenize_1() {
    let tokens = tokenize("").unwrap();
    assert!(tokens.len() > 0);
}
```

**Принцип 2: Нулевая терпимость к случайным сбоям.** Тесты должны воспроизводимо выполняться в любой среде. Тесты, зависящие от случайных чисел, системного времени, порядка планирования потоков, обязаны использовать фиксированные сиды или заменяться моками.

**Принцип 3: Один тест проверяет одно.** Если в имени теста нужно соединить «и» несколько поведений — разбейте на несколько тестов.

```rust
// 🟢 Хорошо — каждый тест проверяет один сценарий
#[test]
fn test_parse_int_positive() { /* ... */ }
#[test]
fn test_parse_int_zero() { /* ... */ }

// 🔴 Мусор — один тест вмещает слишком много несвязанного
#[test]
fn test_parser() {
    // Тестируется tokenize, parse, typecheck, codegen...
}
```

**Принцип 4: Тестируйте поведение, а не реализацию.** Рефакторинг внутренней реализации не должен приводить к падению тестов. Если от изменения одной строки реализации упало 10 тестов — ваши тесты написаны неправильно.

Но здесь есть ключевое разграничение: **определение «поведения» приходит из спецификации, а не из наблюдаемого поведения текущего кода.** Если код изменил поведение (то есть новое поведение не соответствует спецификации), тест обязан упасть. Если это не так, ваши тесты — «подстраивающиеся под код», и они пропускают баги насквозь.

```
Спецификация (language-spec.md / RFC)  ──определяет──►  Ожидаемое поведение  ──движет──►  Тест
                                              │
Текущий код  ──реализует──►  Фактическое поведение  ──сравнивается──►  Результат теста

Если фактическое поведение ≠ ожидаемого:
  Тест обязан упасть (красный)  ──►  Исправить код  ──►  Тест проходит (зелёный)
  
Если фактическое поведение = ожидаемому (но реализация ужасна):
  Тест проходит  ──►  Рефакторинг реализации  ──►  Тест всё ещё проходит  ← Вот смысл принципа 4
```

**Принцип 5: Не писать тестовый код с откатами, совместимостью и условной активацией.** Тестовая среда — это среда, которой вы полностью управляете. Если вам нужен `#[cfg(not(ci))]`, чтобы пропустить какой-то тест, значит, в дизайне теста фундаментальная проблема.

### Терминология

| Термин | Определение |
|------|------|
| Модульный тест | Тестирует поведение одной функции или модуля, не зависит от внешних систем |
| Интеграционный тест | Тестирует совместную работу нескольких модулей через публичный API или CLI-вход |
| Нагрузочный тест | Измеряет производительность кода, выявляет регрессии производительности |
| Документационный тест | Исполняемые примеры кода, встроенные в комментарии документации |
| Property-тест | Тест, основанный на проверке инвариантов (свойств) на случайных входных данных |

### Связь с конвенцией коммитов

Все коммиты, связанные с тестами, должны использовать тип `:white_check_mark: test:`, согласно [конвенции коммитов](./commit-convention.md).

```
:white_check_mark: test(parser): добавить тесты инфиксных выражений парсера Pratt
:white_check_mark: test(codegen): дополнить тесты генерации IR для switch
```

## Нормы модульных тестов

### Организация файлов

**Правило 1.1**: Каталог `tests/` модульных тестов должен быть **на одном уровне** с `mod.rs` тестируемого модуля. `tests/` не агрегируется вверх и не сводится через уровни.

```
src/frontend/core/parser/
├── mod.rs              # #[cfg(test)] mod tests; — объявляет tests/ того же уровня
├── ast.rs
├── pratt/
│   ├── mod.rs          # #[cfg(test)] mod tests; — собственные тесты pratt
│   └── tests/
│       ├── mod.rs
│       ├── led.rs
│       ├── nud.rs
│       └── precedence.rs
└── tests/              # Тесты уровня модуля parser (не включают содержимое подмодуля pratt)
    ├── mod.rs
    ├── ast.rs
    ├── expressions.rs
    ├── error_recovery.rs
    └── parser_state.rs
```

Ключевой критерий: **в каком каталоге лежит `tests/`, в `mod.rs` этого каталога и должен быть `#[cfg(test)] mod tests;`.**

**Правило 1.1, дополнение: восходящая агрегация запрещена.** Тесты подмодуля в подкаталоге должны находиться в его собственном `tests/`, а не сводиться в родительский `tests/`.

| Тип модуля | Расположение тестов | Пример |
|----------|----------|------|
| Каталоговый модуль (с `mod.rs`) | `tests/` внутри этого каталога | `emitter/tests/`, `codes/tests/` |
| Однопроходный модуль (только `.rs`) | `tests/` родителя | `session.rs` → `diagnostic/tests/session.rs` |

```text
# ✅ Правильно: тесты каждого каталогового модуля независимы
src/util/diagnostic/
├── codes/
│   ├── mod.rs              # #[cfg(test)] mod tests;
│   └── tests/              # ✅ собственные тесты codes
│       ├── mod.rs
│       └── codes.rs
├── emitter/
│   ├── mod.rs              # #[cfg(test)] mod tests;
│   └── tests/              # ✅ собственные тесты emitter
│       ├── mod.rs
│       ├── text.rs
│       └── ansi.rs
└── tests/                  # ✅ уровень diagnostic (однопроходные модули)
    ├── mod.rs
    ├── session.rs
    ├── suggest.rs
    └── collect.rs

# ❌ Неправильно: сводить тесты emitter и codes в diagnostic/tests/
src/util/diagnostic/
└── tests/
    ├── mod.rs              # ❌ вынуждены объявлять mod emitter; mod codes;
    ├── emitter/            # ❌ должны быть в emitter/tests/
    └── codes/              # ❌ должны быть в codes/tests/
```

#### Правила размещения тестов: однопроходный модуль vs каталоговый модуль

**Ключевое отличие**: форма организации модуля определяет место расположения тестов.

| Тип модуля | Критерий определения | Расположение тестов | Пример |
|----------|----------|----------|------|
| **Каталоговый модуль** | Имеет собственный каталог и `mod.rs` | `tests/` внутри этого каталога | `inference/tests/` |
| **Однопроходный модуль** | Только файл `.rs`, без собственного каталога | `tests/` родительского модуля | `overload.rs` → `typecheck/tests/overload.rs` |

**Подробное описание**:

```
src/frontend/core/typecheck/
├── mod.rs                          # mod.rs модуля typecheck
├── checker.rs                      # однопроходный модуль
├── environment.rs                  # однопроходный модуль
├── overload.rs                     # однопроходный модуль
├── type_eval.rs                    # однопроходный модуль
├── dead_code.rs                    # однопроходный модуль
├── spawn_placement.rs              # однопроходный модуль
├── signature.rs                    # однопроходный модуль
├── types.rs                        # однопроходный модуль
│
├── tests/                          # ✅ каталог тестов typecheck
│   ├── mod.rs                      # объявляет тесты однопроходных модулей
│   ├── checker.rs                  # тесты checker.rs
│   ├── environment.rs              # тесты environment.rs
│   ├── overload.rs                 # тесты overload.rs (тесты однопроходного модуля)
│   ├── type_eval.rs                # тесты type_eval.rs
│   ├── dead_code.rs                # тесты dead_code.rs
│   ├── spawn_placement.rs          # тесты spawn_placement.rs
│   ├── signature.rs                # тесты signature.rs
│   └── types.rs                    # тесты types.rs
│
├── inference/                      # каталоговый модуль (с mod.rs)
│   ├── mod.rs                      # #[cfg(test)] mod tests; — объявляет tests/ того же уровня
│   ├── expressions.rs
│   ├── statements.rs
│   ├── patterns.rs
│   ├── bounds.rs
│   ├── subtyping.rs
│   ├── generics.rs
│   ├── compatibility.rs
│   ├── scope.rs
│   ├── assignment.rs
│   └── tests/                      # ✅ каталог тестов inference
│       ├── mod.rs
│       ├── expressions.rs          # тесты expressions.rs
│       ├── statements.rs           # тесты statements.rs
│       └── ...
│
└── traits/                         # удалён (логика слита в types/trait_data.rs)
```

**Почему тесты однопроходного модуля лежат в родительском `tests/`?**

Потому что однопроходный модуль (например, `overload.rs`) не имеет собственного `mod.rs` и не может объявить `#[cfg(test)] mod tests;`. Согласно модульной системе Rust, тестовые файлы должны быть объявлены каким-либо `mod.rs`, чтобы быть скомпилированными. Поэтому тесты однопроходного модуля объявляются `mod.rs` родительского модуля и помещаются в родительский `tests/`.

**Алгоритм определения**:

```
Встретился модуль. Куда положить тесты?
│
├── Модуль — каталоговый (есть mod.rs)?
│   └── Да → создать tests/ внутри каталога, объявить в mod.rs этого каталога
│
├── Модуль — однопроходный (только .rs)?
│   └── Да → тесты в tests/ родителя, объявить в mod.rs родителя
│
└── Неясно?
    └── Проверить наличие собственного каталога и mod.rs
```

**Частые ошибки**:

```
# ❌ Ошибка 1: создавать отдельный tests/ для однопроходного модуля
src/frontend/core/typecheck/
├── overload.rs
└── overload/                       # ❌ нельзя создавать каталог для однопроходного модуля
    └── tests/
        └── overload.rs

# ❌ Ошибка 2: объявлять #[cfg(test)] mod tests; внутри однопроходного модуля
# overload.rs
#[cfg(test)]                        # ❌ однопроходный модуль не может так объявлять
mod tests;                          # потому что нет каталога overload/tests/

# ✅ Правильно: тесты лежат в родительском tests/
src/frontend/core/typecheck/
├── overload.rs                     # исходный файл
└── tests/
    └── overload.rs                 # тестовый файл, объявлен typecheck/mod.rs
```

⚠️ **Антипаттерн — так не делайте**:

```
# ❌ Неправильно: тесты подмодулей собраны в родительском tests/
src/frontend/core/types/
├── mod.rs              # должен бы объявлять только base и computation
├── base/
│   ├── mod.rs
│   └── var.rs
└── tests/              # ❌ родительский tests/ содержит тесты подмодулей
    ├── mod.rs          # ❌ вынуждены объявлять mod base; mod computation;
    ├── base/           # ❌ должно быть в base/tests/
    │   └── var.rs
    └── computation/    # ❌ должно быть в computation/tests/
        └── ...
```

```
# ✅ Правильно: тесты каждого модуля независимы
src/frontend/core/types/
├── mod.rs              # объявляет только pub mod base; pub mod computation;
├── base/
│   ├── mod.rs          # #[cfg(test)] mod tests; — объявляет tests/ того же уровня
│   ├── var.rs
│   └── tests/
│       ├── mod.rs
│       └── var.rs
└── computation/
    ├── mod.rs          # #[cfg(test)] mod tests; — объявляет tests/ того же уровня
    ├── operations.rs
    └── tests/
        ├── mod.rs
        └── operations.rs
```

**Почему нельзя агрегировать вверх?** Потому что модульная система Rust требует, чтобы `#[cfg(test)] mod tests;` в точке объявления определял компиляцию тестовых файлов. Если `types/mod.rs` объявляет `mod tests;`, то содержимое `types/tests/` — это приватное содержимое модуля `types`, и оно не должно вторгаться на территорию `base` или `computation`. Тесты модуля — это деталь внутренней реализации этого модуля, а не родительского. Это правило действует и при рефакторинге модулей: когда `types` разделяется на `base` и `computation`, тесты тоже следуют за границами разделённых модулей, а не остаются на месте. **Каталог тестов не зеркалит структуру исходного кода, а следует за границами модулей.**

**Правило 1.2**: `tests/mod.rs` отвечает только за объявление модулей и re-export, без тестовых функций.

```rust
//! Тесты ядра парсера — зеркалит src/frontend/core/parser/
//!
//! Тесты для ast.rs, parser_state.rs и парсинга выражений/интеграции.

mod ast;
mod error_recovery;
mod expressions;
mod integration;
mod parser_state;
```

**Правило 1.3**: Каждый тестовый файл соответствует ровно одному исходному файлу. Запрещено смешивать в одном файле тесты нескольких исходных модулей.

**Правило 1.4**: Объявление тестов должно быть файловой формы `mod tests;` (с точкой с запятой), указывающей на `tests/` того же уровня. **Запрещена inline-форма `mod tests { ... }`, когда тестовый код пишется прямо в исходном файле.**

```rust
// ✅ Правильно — файловая форма объявления, тестовый код в отдельных файлах
// src/frontend/core/parser/mod.rs
#[cfg(test)]
mod tests;

// 🔴 Запрещено — inline-форма, тестовый код паразитирует в исходном файле
// src/frontend/core/parser/mod.rs
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_something() {
        // Тестовый код не должен появляться в исходном файле
    }
}
```

**Почему inline-форма запрещена?**
1. Единственная ответственность исходного файла: в исходном файле — только реализация, в тестовом — только тесты. Смешивание заставляет прокручивать к концу файла при правке тестов и пропускать тесты при правке реализации.
2. Чёткие границы модулей: каталог `tests/` — это физическая граница; сразу видно, у каких модулей есть тесты, а у каких нет.
3. Безопасность рефакторинга: при разделении модулей каталог `tests/` перемещается вместе с ними; inline-тесты приходится вручную вырезать из исходных файлов.
4. Code Review: в PR diff изменения исходного кода и тестов находятся в разных файлах и не смешиваются.

### Нормы объявления модулей

**Правило 2.1**: В начале каждого тестового файла должен быть модульный документационный комментарий `//!` с указанием источника в спецификации (номер раздела спецификации языка + номер RFC). Если какой-то тест не ссылается ни на один раздел спецификации — этот код не имеет нормативного обоснования, и его не должно существовать.

```rust
//! Тесты литералов — на основе §2.6 спецификации языка
//!
//! §2.6.1: Целые числа Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: F-строковая интерполяция
```

**Почему обязательна ссылка на спецификацию?** Потому что ожидаемые значения в тестах берутся из спецификации, а не из «выхода текущего кода». Если когда-нибудь код поменяет вывод и тест подстроится, он ничего не защищает. Только привязанный к спецификации тест может отличить «преднамеренное breaking change» от «непреднамеренной регрессии».

**Правило 2.2**: Импорты `use` в тестовом модуле должны быть точными — до конкретных типов/функций. Запрещён glob-импорт `use super::*`.

```rust
// 🟢 Хорошо — точные импорты
use crate::frontend::core::lexer::{tokenize, TokenKind};
use crate::frontend::core::parser::{ParserState, ParseError};

// 🔴 Мусор — непонятно, что вы вообще тестируете
use super::*;
```

### Правила именования

**Правило 3.1**: Имя тестовой функции имеет формат `test_<что>_<сценарий>`, всё в нижнем регистре, разделение подчёркиваниями.

```rust
#[test]
fn test_tokenize_empty_string() { /* ... */ }
#[test]
fn test_parse_int_overflow() { /* ... */ }
#[test]
fn test_typecheck_fn_return_mismatch() { /* ... */ }
```

**Правило 3.2**: Имя тестовой функции должно быть самообъясняющим. Прочитав имя, должно быть понятно, что тестируется и что ожидается. Запрещено числовое именование.

```rust
// 🟢 Хорошо
fn test_skip_semicolon_success() { /* ... */ }
fn test_skip_semicolon_failure_when_identifier() { /* ... */ }

// 🔴 Мусор — совершенно непонятно, что тестируется
fn test_skip_1() { /* ... */ }
fn test_skip_2() { /* ... */ }
```

**Правило 3.3**: Вспомогательные функции не требуют префикса `test_` и должны описывать своё назначение глаголом или существительным.

```rust
fn parse_expr(source: &str) -> Expr { /* ... */ }
fn tokenize_single(source: &str) -> Token { /* ... */ }
fn setup_parser_with_tokens(tokens: &[Token]) -> ParserState { /* ... */ }
```

### Структура теста (Arrange-Act-Assert)

**Правило 4.1**: Каждая тестовая функция должна следовать трёхчастной структуре: подготовка (Arrange) → выполнение (Act) → проверка (Assert), части разделяются пустой строкой.

```rust
#[test]
fn test_parse_binary_addition() {
    // Arrange
    let source = "1 + 2";

    // Act
    let expr = parse_expr(source);

    // Assert
    assert!(matches!(expr, Expr::Binary { op: BinOp::Add, .. }));
}
```

**Правило 4.2**: Для простых тестов (один вызов + одна проверка) можно не писать комментарии-разделители, но логический код не должен превышать 5 строк. Если код длиннее 5 строк, разделение на три части обязательно.

### Правила вспомогательных функций

**Правило 5.1**: Повторяющийся 3 и более раз setup должен выделяться во вспомогательную функцию.

```rust
// 🟢 Хорошо — выделен общий setup
fn with_state<F>(source: &str, mut f: F)
where
    F: FnMut(&mut ParserState<'_>),
{
    let tokens = tokenize(source).unwrap();
    let mut state = ParserState::new(&tokens);
    f(&mut state);
}

#[test]
fn test_current_returns_first_token() {
    with_state("42", |state| {
        let tok = state.current();
        assert_eq!(&tok.unwrap().kind, &TokenKind::IntLiteral(42));
    });
}
```

**Правило 5.2**: `unwrap()` / `expect()` во вспомогательных функциях при панике должны печатать достаточно контекста. Внутри тела тестовой функции (`#[test] fn ...`) можно напрямую `unwrap()` — при сбое Rust автоматически напечатает номер строки; но при сбое во вспомогательной функции номер строки указывает на её определение, без контекста вызова.

```rust
// 🟢 Хорошо — при сбое вспомогательной функции печатается содержимое исходника
fn run_ok(source: &str) {
    run(source).unwrap_or_else(|e| panic!("Выполнение не удалось:\nИсходник:\n{}\nОшибка:\n{:?}", source, e));
}

// 🔴 Мусор — при сбое не видно, какой исходник вызвал проблему
fn run_ok(source: &str) {
    run(source).unwrap();
}
```

**Правило 5.3**: Вспомогательные функции размещаются в начале тестового файла сразу после импортов `use`. Если они используются несколькими тестовыми модулями — кладутся в `tests/mod.rs` с экспортом `pub(crate)`.

### Стиль утверждений

**Правило 6.1**: Для сопоставления с вариантом перечисления предпочтительно `assert!(matches!(...))`. Запрещено использовать `if let` + `panic!`.

```rust
// 🟢 Хорошо
assert!(matches!(tokens[0].kind, TokenKind::IntLiteral(42)));

// 🔴 Мусор
if let TokenKind::IntLiteral(v) = tokens[0].kind {
    assert_eq!(v, 42);
} else {
    panic!("Ожидался IntLiteral");
}
```

**Правило 6.2**: Для сравнения точных значений используется `assert_eq!`, для булевых утверждений — `assert!`. Запрещено использовать `assert!(a == b)` вместо `assert_eq!(a, b)`.

**Правило 6.3**: Все утверждения должны иметь пользовательское сообщение об ошибке, если само утверждение не описывает полностью причину сбоя.

```rust
// 🟢 Хорошо — при сбое легко локализовать проблему
assert!(
    state.infix_info().is_some(),
    "infix_info должен обрабатывать '{op}'"
);

// 🟢 Хорошо — assert_eq! при сбое сам печатает разницу значений, дополнительное сообщение не нужно
assert_eq!(error_count, 0);

// 🔴 Мусор — при сбое видно только "утверждение не выполнено"
assert!(state.infix_info().is_some());
```

**Правило 6.4**: Порядок аргументов утверждений: `assert_eq!(фактическое, ожидаемое)` — фактическое значение первым, ожидаемое вторым.

### Список антипаттернов

Ниже приведены запрещённые приёмы и их замены:

| Антипаттерн | Проблема | Замена |
|--------|------|------|
| `#[cfg(test)] mod tests { ... }` — inline-тесты | Раздувание исходного файла, размытие границ модулей, сложность рефакторинга | Тестовый код в отдельном каталоге `tests/`, объявлен через `mod tests;` (см. правило 1.4) |
| Тест подстраивается под неверное поведение кода | Скрывает отклонения от спецификации, легализует баги | Сверять со спецификацией и править код, сохраняя тест неизменным |
| Ожидаемое значение выводится из вывода кода | Тест превращается в «диктофон текущей реализации» | Выводить ожидаемые значения из спецификации |
| Постоянная пометка `#[ignore]` | Прячет гнилые тесты | Исправить или удалить |
| Отладочный вывод `println!` | Засоряет вывод тестов | Использовать явные утверждения `assert!` |
| `thread::sleep` | Случайные сбои + медленно | Использовать механизмы синхронизации или моки |
| Операции с реальной файловой системой в тестах | Медленно и невоспроизводимо | Использовать `tempfile` |
| Зависимость от порядка выполнения тестов | Случайные сбои | Каждый тест с независимым setup |
| Тестовая функция длиннее 30 строк логики | Никто не может разобраться | Разбить на тесты или вынести во вспомогательные функции |
| `unwrap()` во вспомогательных функциях без контекста | Сложно локализовать | Использовать `expect("причина")` или собственное сообщение паники (см. правило 5.2) |
| Copy-paste одного setup более 3 раз | Дорого менять | Выделить во вспомогательную функцию |

## Нормы интеграционных тестов

### Организация тестов

**Правило 7.1**: Интеграционные тесты размещаются в каталоге `tests/` в корне проекта. Входной файл `tests/integration.rs` подключает подмодули через атрибут `#[path]`.

```rust
// tests/integration.rs
#[path = "integration/backends.rs"]
mod backends;
#[path = "integration/codegen.rs"]
mod codegen;
#[path = "integration/execution.rs"]
mod execution;
```

**Правило 7.2**: Каждый файл `tests/integration/*.rs` соответствует одной теме тестирования (бэкенды компилятора, кодогенерация, исполнитель и т. д.); смешивание запрещено.

**Правило 7.3**: Интеграционные тесты должны проходить через публичный API проекта. В интеграционных тестах запрещено напрямую ссылаться на внутренние модули `crate::`. Следует использовать публичные пути `yaoxiang::`.

```rust
// 🟢 Хорошо — через публичный API
use yaoxiang::run;

// 🔴 Мусор — обход границы публичного API
use yaoxiang::middle::codegen::bytecode::BytecodeFile;
```

### Управление тестовыми данными

**Правило 8.1**: В интеграционных тестах предпочтительны встроенные строки исходного кода. Внешние фикстуры (в `tests/fixtures/`) применяются только если исходник превышает 30 строк.

```rust
#[test]
fn test_fibonacci() {
    run_ok(
        r#"
        main = {
            mut a = 0
            mut b = 1
            while a < 100 {
                mut next = a + b
                a = b
                b = next
            }
        }
        "#,
    );
}
```

**Правило 8.2**: Файлы фикстур должны иметь расширение `.yx`, имя файла описывает назначение теста.

### Принципы покрытия E2E

**Правило 9.1**: Интеграционные тесты для каждой возможности языка должны покрывать три пути:

| Путь | Описание |
|------|------|
| Happy path | Корректный ввод даёт ожидаемый вывод |
| Error path | Некорректный ввод даёт ясное сообщение об ошибке (не панику) |
| Boundary | Граничные значения (пустой ввод, максимальное значение, верхний предел глубины вложенности) |

**Правило 9.2**: Интеграционные тесты не должны зависеть от сети, переменных среды системы или внешних сервисов.

## Нормы нагрузочных тестов

### Правила использования Criterion.rs

**Правило 10.1**: Нагрузочные тесты единообразно размещаются в каталоге `benches/`, входной файл — `benches/lib.rs`. Файлы делятся по темам тестирования.

```
benches/
├── lib.rs              # входной файл, определяет criterion_group/criterion_main
├── lang_compare/
│   └── fibonacci.rs    # межъязыковое сравнение
├── parser.rs           # бенчмарки парсера
└── codegen.rs          # бенчмарки кодогенерации
```

**Правило 10.2**: Каждая функция бенчмарка должна содержать модульный документационный комментарий `//!` с описанием цели теста и измеряемой метрики.

```rust
//! Нагрузочные тесты интерпретатора YaoXiang
//!
//! Измеряемая метрика: время одной итерации (wall time)
//! Базовый уровень: нативная реализация на Rust
```

### Предотвращение оптимизаций компилятора

**Правило 11.1**: Все результаты измерения в бенчмарках должны прогоняться через `criterion::black_box`, чтобы компилятор не мог удалить их как мёртвый код.

```rust
use criterion::{black_box, Criterion};

fn bench_parse(c: &mut Criterion) {
    c.bench_function("parse_fib", |b| {
        b.iter(|| {
            let result = parse(black_box(FIB_SOURCE));
            black_box(result)
        })
    });
}
```

**Правило 11.2**: Входные данные бенчмарка должны быть `const` или `lazy_static`. Генерировать данные внутри замыкания `iter` запрещено — иначе измеряется суммарное время генерации данных и тестируемой логики.

### Группировка и именование бенчмарков

**Правило 12.1**: Имя бенчмарка имеет формат `<тестируемый_модуль>_<сценарий>`, всё в нижнем регистре, разделение подчёркиваниями. Соответствует правилам именования модульных тестов.

**Правило 12.2**: Связанные бенчмарки обязательно группируются через `criterion_group!`. Запрещено сваливать все бенчмарки в одну группу.

```rust
criterion_group!(parser, bench_parse_expr, bench_parse_stmt);
criterion_group!(codegen, bench_codegen_module, bench_codegen_switch);
criterion_main!(parser, codegen);
```

## Нормы документационных тестов

### Сценарии использования

**Правило 13.1**: Все `pub`-функции, типы и методы должны иметь в комментариях документации как минимум один запускаемый пример кода. Этот пример выполняется через `cargo test --doc`.

```rust
/// Разбивает строку исходного кода на последовательность токенов.
///
/// ```
/// use yaoxiang::frontend::core::lexer::tokenize;
///
/// let tokens = tokenize("42").unwrap();
/// assert_eq!(tokens.len(), 2); // IntLiteral + Eof
/// ```
pub fn tokenize(source: &str) -> Result<Vec<Token>, LexError> {
    // ...
}
```

**Правило 13.2**: Примеры кода в документационных тестах должны компилироваться и проходить проверки. Запрещены примеры с пометкой `ignore`, если только они не демонстрируют ошибку этапа компиляции.

```rust
/// ```ignore
/// // Демонстрация ошибки компиляции — допустимо ignore
/// let x: int = "string";
/// ```
```

### Требования к покрытию

**Правило 14.1**: Документационные тесты покрывают happy path API. Граничные случаи и пути ошибок покрываются модульными тестами.

**Правило 14.2**: Примеры кода в документационных тестах должны быть краткими — не длиннее 10 строк. Если для примера нужен более длинный контекст, это признак проблемы в дизайне API.

## Нормы property-тестов

### Сценарии использования

**Правило 15.1**: В перечисленных ниже сценариях обязательно используются property-тесты (proptest или quickcheck), а не ручной набор граничных значений:

| Сценарий | Пример |
|------|------|
| Round-trip парсера | `parse(pretty_print(ast)) == ast` |
| Сериализация/десериализация | `deserialize(serialize(data)) == data` |
| Тождества математических операций | `a + b == b + a` |
| Оптимизации компилятора не меняют семантику | `eval(code) == eval(optimize(code))` |

**Правило 15.2**: Property-тесты используют `proptest` как основной фреймворк (уже объявлен в `dev-dependencies` файла `Cargo.toml`).

```rust
use proptest::prelude::*;

proptest! {
    #[test]
    fn test_roundtrip_serialize_deserialize(value: i64) {
        let serialized = serialize(&value);
        let deserialized: i64 = deserialize(&serialized).unwrap();
        prop_assert_eq!(deserialized, value);
    }
}
```

### Принципы определения свойств

**Правило 16.1**: Каждый property-тест должен иметь явное объявление свойства — в комментарии указать проверяемый инвариант.

```rust
// Свойство: любой целочисленный литерал после tokenize → tokens_to_string даёт то же значение
proptest! {
    #[test]
    fn test_int_literal_roundtrip(n in any::<i64>()) {
        let source = n.to_string();
        let tokens = tokenize(&source).unwrap();
        // ...
    }
}
```

**Правило 16.2**: Если property-тест обнаруживает сбой, обязательно использовать механизм регрессий `proptest` — добавить упавший вход в каталог `proptest-regressions/`, а не подменять его обычным тестом вручную.

## Требования к покрытию

### Целевые показатели покрытия нового кода

**Правило 17.1**: Требования к покрытию тестами для нового кода:

| Тип кода | Покрытие строк | Покрытие ветвей |
|----------|----------|------------|
| Ядро компилятора (frontend/middle/backends) | ≥ 85 % | ≥ 80 % |
| Утилитарные/вспомогательные модули (util) | ≥ 75 % | ≥ 70 % |
| Модули рантайма (vm/runtime) | ≥ 80 % | ≥ 75 % |
| Стандартная библиотека (std) | ≥ 75 % | ≥ 70 % |
| Обработка ошибок и диагностика | ≥ 90 % | ≥ 85 % |

**Правило 17.2**: Пути обработки ошибок (все ветки `Err`) должны быть покрыты на 100 %. Видимые пользователю сообщения об ошибках должны быть проверены тестами.

### Чек-лист проверки PR

**Правило 18.1**: Перед отправкой PR автор обязан самопроверить следующие пункты:

- [ ] `cargo test` — все тесты проходят
- [ ] `cargo test --doc` — все тесты проходят
- [ ] `cargo bench` — нет регрессий производительности (если затронуты горячие пути)
- [ ] Новый код соответствует целевым показателям покрытия
- [ ] Имена тестов соответствуют правилам именования
- [ ] В каждом тестовом файле объявлен соответствующий раздел спецификации (правило 2.1)
- [ ] Ожидаемые значения в тестах взяты из определения спецификации, а не из «выхода текущего кода»
- [ ] Нет тестов с пометкой `#[ignore]` (если только нет комментария с явным номером issue)
- [ ] Нет ненужных `unwrap()` (следует использовать `expect` или собственное сообщение паники)
- [ ] Сообщение коммита использует тип `:white_check_mark: test:`
- [ ] **Никаких изменений ожидаемых значений теста по причине «поведение кода не соответствует спецификации» — правится код, а не тест**
- [ ] **Нет inline-тестов** (`#[cfg(test)] mod tests { ... }` необходимо заменить на `mod tests;` + отдельный файл, см. правило 1.4)

**Правило 18.2**: Reviewer обязан отклонить PR со следующими проблемами:

- Есть только тесты happy path, отсутствуют пути ошибок
- В тестах присутствует `thread::sleep` или зависимость от порядка выполнения
- Copy-paste тестового кода более 3 раз без выделения вспомогательной функции
- Имена тестов не соответствуют правилам именования
- Существуют постоянно помеченные `#[ignore]` тесты
- **Тест подстраивается под неверное поведение кода** (когда код не соответствует спецификации, правится тест, а не код)
- **Тест не объявляет соответствующий раздел спецификации** (см. правило 2.1)
- **Ожидаемые значения в тесте взяты из вывода кода, а не из определения спецификации** (выведенный «обратно» тест равен ненаписанному)
- **Присутствуют inline-тесты** (`#[cfg(test)] mod tests { ... }` вместо `mod tests;` + отдельный файл, см. правило 1.4)
- Тест проверяет только «отсутствие паники» без утверждений о конкретном поведении
- Удалён упавший тест, выявляющий баг кода (вместо того чтобы исправить код и увидеть его зелёным)

## Приложения

### A. Краткий справочник команд тестирования

```bash
# Запустить все тесты
cargo test

# Только модульные тесты
cargo test --lib

# Только интеграционные тесты
cargo test --test integration

# Только документационные тесты
cargo test --doc

# Запустить конкретный тест (фильтр по имени)
cargo test test_parse_expr

# Запустить нагрузочные тесты
cargo bench

# Показать вывод тестов (по умолчанию stdout скрыт)
cargo test -- --nocapture

# Однопоточный запуск (для диагностики проблем с конкурентностью)
cargo test -- --test-threads=1

# Генерация отчёта о покрытии (нужен cargo-llvm-cov)
cargo llvm-cov --html
```

### B. Шаблон сообщения коммита

Коммиты, связанные с тестами, должны следовать шаблону:

```
:white_check_mark: test(<область>): <краткое описание>

<опционально: список покрытых сценариев>
```

Пример:

```
:white_check_mark: test(parser): добавить тесты инфиксных операторов парсера Pratt

Покрытые сценарии:
- Приоритеты арифметических операторов (+, -, *, /, %)
- Цепочки операторов сравнения (1 < x < 10)
- Короткое замыкание логических операторов
- Правоассоциативность оператора присваивания
```

### C. Чек-лист для нового тестового файла

При создании нового тестового модуля убедитесь, что присутствуют следующие файлы:

```
# Добавление тестов в каталоге src/<модуль>/
src/<модуль>/tests/
├── mod.rs          # объявления модулей + общие вспомогательные функции
└── <тема>.rs       # тестовый файл, названный по исходному файлу

# Добавление интеграционных тестов в каталоге tests/
tests/
├── integration.rs   # обновить: добавить объявление #[path]
└── integration/
    └── <тема>.rs    # новый тестовый файл
```

> 💡 **Запомните**: тест проверяет не «запускается ли ваш код» — он проверяет, соответствует ли ваш код спецификации. Спецификация меняется — тесты следуют за ней. Код написан неправильно — правится код, а не тест. **Код служит спецификации, тесты охраняют спецификацию. В тот момент, когда тест начинает подстраиваться под код, вы теряете всю защиту.**
