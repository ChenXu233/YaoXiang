## Общие положения

### Область применения

Настоящий стандарт применяется ко всему коду тестов на Rust в проекте YaoXiang:

| Тип теста | Расположение | Фреймворк |
|-----------|--------------|-----------|
| Модульные тесты | `src/<module>/tests/` | `#[test]` + `#[cfg(test)]` |
| Интеграционные тесты | `tests/` | `#[test]` |
| Бенчмарк-тесты | `benches/` | Criterion.rs |
| Doctest | API-документация | `cargo test --doc` |
| Property-based тесты | В любом месте тестов | proptest / quickcheck |

### Основные принципы

**Принцип 0: Единственным авторитетным источником для тестов является спецификация, а не код.** Это важнейший принцип данного документа. Тесты проверяют, соответствует ли код спецификации, а не то, «работает ли код в текущей реализации». Когда тесты обнаруживают несоответствие поведения кода спецификации, **исправляется код, тесты не изменяются**.

Файлы спецификаций находятся здесь:
- `docs/src/design/language-spec.md` —— Основная спецификация языка
- `docs/src/design/rfc/accepted/` —— Принятые RFC-дизайны

В начале каждого файла тестов должна быть указана соответствующая глава спецификации (см. правило 2.1). Любой разработчик должен иметь возможность взять спецификацию и, сверяясь с тестами, проверить корректность реализации. И наоборот — если фрагмент кода не имеет соответствующего описания в спецификации, его не должно существовать, и он не должен тестироваться.

```rust
// 🟢 Хорошо——тест напрямую ссылается на спецификацию, проверяя соответствие кода спецификации
//! Тесты литералов — на основе спецификации языка §2.6
//!
//! §2.6.1: Целые Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: Интерполяция F-String

#[test]
fn test_decimal_literal_parsing() {
    // Спецификация §2.6.1: Decimal ::= [0-9][0-9_]*
    let result = parse_literal("42").unwrap();
    assert_eq!(result, Literal::Int(42));
}

// 🔴 Мусор——тест подстраивается под текущее поведение кода, а не проверяет спецификацию
#[test]
fn test_literal_1() {
    // Непонятно, к какой главе спецификации относится этот код
    // Если parse_literal вернёт неправильное значение, тест «пройдёт»
    // Потому что он только проверяет, что функция не паникует
    let result = parse_literal("42");
    assert!(result.is_ok());
}
```

**Ситуация**: Вы написали тест и обнаружили, что поведение кода не соответствует спецификации. У вас есть два варианта:
| Неправильный подход | Правильный подход |
|---------------------|--------------------|
| Изменить тест, чтобы он «проходил» | Изменить код, чтобы поведение соответствовало спецификации |
| Добавить в тест `#[ignore]` | Немедленно исправить реализацию |
| Добавить в тест условные ветки под код | Удалить ветки, чтобы тест напрямую暴露ил проблему |

Запомните: **Красный тест = ошибка в коде, а не в тесте.** (Если только в самом тесте нет бага — это другое дело.)

**Принцип 1: Тесты — это документация.** Любой разработчик должен понять поведение тестируемого кода, читая только тесты, без дополнительных комментариев или внешней документации.

```rust
// 🟢 Хорошо——имя теста говорит, что тестируется и чего ожидаем
#[test]
fn test_tokenize_empty_input_returns_eof() {
    let tokens = tokenize("").unwrap();
    assert_eq!(tokens.len(), 1);
    assert!(matches!(tokens[0].kind, TokenKind::Eof));
}

// 🔴 Мусор——непонятно, что это тестирует
#[test]
fn test_tokenize_1() {
    let tokens = tokenize("").unwrap();
    assert!(tokens.len() > 0);
}
```

**Принцип 2: Нулевая терпимость к случайным сбоям.** Тесты должны быть воспроизводимы в любой среде. Тесты, зависящие от случайных чисел, системного времени или порядка планирования потоков, должны использовать фиксированные seed или заменяться mock-объектами.

**Принцип 3: Один тест проверяет одну вещь.** Если для описания имени теста нужно использовать «и», разделите его на несколько тестов.

```rust
// 🟢 Хорошо——каждый тест проверяет только один сценарий
#[test]
fn test_parse_int_positive() { /* ... */ }
#[test]
fn test_parse_int_zero() { /* ... */ }

// 🔴 Мусор——один тест содержит слишком много несвязанного содержимого
#[test]
fn test_parser() {
    // Тестируем tokenize, parse, typecheck, codegen...
}
```

**Принцип 4: Тестируем поведение, а не реализацию.** Рефакторинг внутренней реализации не должен приводить к падению тестов. Если изменение одной строки реализации привело к падению 10 тестов — тесты написаны неправильно.

Но здесь важное различие: **определение «поведения» исходит из спецификации, а не из текущего поведения кода.** Если код изменил поведение (то есть появилось новое поведение, не соответствующее спецификации), тесты должны падать. Если вы не можете этого достичь — ваши тесты «подстраиваются под код» — они впускают баги.

```
Спецификация (language-spec.md / RFC)  ──определяет──►  Ожидаемое поведение  ──управляет──►  Тесты
                                                             │
Текущий код  ──реализует──►  Фактическое поведение  ──сравнивается──►  Результат теста

Если фактическое поведение ≠ ожидаемому:
  Тест должен упасть (красный)  ──►  Исправляем код  ──►  Тест проходит (зелёный)
  
Если фактическое поведение = ожидаемому (но реализация плохая):
  Тест проходит  ──►  Рефакторим реализацию  ──►  Тест всё ещё проходит  ← Вот смысл принципа 4
```

**Принцип 5: Не пишите fallback/совместимый/зависящий от паттернов тестовый код.** Тестовая среда — это среда, которую вы полностью контролируете. Если вам нужен `#[cfg(not(ci))]`, чтобы пропустить тест, значит, дизайн этого теста имеет фундаментальную проблему.

### Определения терминов

| Термин | Определение |
|--------|-------------|
| Модульный тест | Тестирование поведения одной функции или модуля без зависимостей от внешних систем |
| Интеграционный тест | Тестирование взаимодействия нескольких модулей через публичный API или командную строку |
| Бенчмарк-тест | Измерение производительности кода, обнаружение регрессий производительности |
| Doctest | Исполняемые примеры кода, встроенные в документацию |
| Property-based тест | Тестирование инвариантов (property) на основе случайных входных данных |

### Связь со стандартами коммитов

Все коммиты, связанные с тестами, должны использовать тип `:white_check_mark: test:`, согласно [стандартам коммитов](./commit-convention.md).

```
:white_check_mark: test(parser): добавить тесты инфиксных выражений в Pratt-парсере
:white_check_mark: test(codegen): дополнить тесты генерации IR для switch
```

## Стандарты модульных тестов

### Организация файлов

**Правило 1.1**: Директория `tests/` для модульных тестов должна находиться **на одном уровне** с тестируемым `mod.rs`. Директория `tests/` не агрегируется вверх и не суммируется между уровнями.

```
src/frontend/core/parser/
├── mod.rs              # #[cfg(test)] mod tests; ——объявляет tests/ того же уровня
├── ast.rs
├── pratt/
│   ├── mod.rs          # #[cfg(test)] mod tests; ——собственные тесты pratt
│   └── tests/
│       ├── mod.rs
│       ├── led.rs
│       ├── nud.rs
│       └── precedence.rs
└── tests/              # Тесты уровня модуля parser (не включают подмодули pratt)
    ├── mod.rs
    ├── ast.rs
    ├── expressions.rs
    ├── error_recovery.rs
    └── parser_state.rs
```

Ключевой критерий: **в какой директории находится `tests/`, в той же директории `mod.rs` должен объявить `#[cfg(test)] mod tests;`.**

#### Правила размещения тестов для однофайловых модулей vs модулей-директорий

**Основное различие**: форма организации модуля определяет место размещения тестов.

| Тип модуля | Критерий | Расположение тестов | Пример |
|------------|----------|---------------------|--------|
| **Модуль-директория** | Имеет отдельную директорию и `mod.rs` | `tests/` в этой директории | `inference/tests/` |
| **Однофайловый модуль** | Только файл `.rs`, без отдельной директории | `tests/` родительского модуля | `overload.rs` → `typecheck/tests/overload.rs` |

**Подробное объяснение**:

```
src/frontend/core/typecheck/
├── mod.rs                          # mod.rs модуля typecheck
├── checker.rs                      # Однофайловый модуль
├── environment.rs                  # Однофайловый модуль
├── overload.rs                     # Однофайловый модуль
├── type_eval.rs                    # Однофайловый модуль
├── dead_code.rs                    # Однофайловый модуль
├── spawn_placement.rs              # Однофайловый модуль
├── signature.rs                    # Однофайловый модуль
├── types.rs                        # Однофайловый модуль
│
├── tests/                          # ✅ Директория тестов typecheck
│   ├── mod.rs                      # Объявление тестов однофайловых модулей
│   ├── checker.rs                  # Тесты для checker.rs
│   ├── environment.rs              # Тесты для environment.rs
│   ├── overload.rs                 # Тесты для overload.rs (тесты однофайлового модуля здесь)
│   ├── type_eval.rs                # Тесты для type_eval.rs
│   ├── dead_code.rs                # Тесты для dead_code.rs
│   ├── spawn_placement.rs          # Тесты для spawn_placement.rs
│   ├── signature.rs                # Тесты для signature.rs
│   └── types.rs                    # Тесты для types.rs
│
├── inference/                      # Модуль-директория (с mod.rs)
│   ├── mod.rs                      # #[cfg(test)] mod tests; ——объявляет tests/ того же уровня
│   ├── expressions.rs
│   ├── statements.rs
│   ├── patterns.rs
│   ├── bounds.rs
│   ├── subtyping.rs
│   ├── generics.rs
│   ├── compatibility.rs
│   ├── scope.rs
│   ├── assignment.rs
│   └── tests/                      # ✅ Директория тестов inference
│       ├── mod.rs
│       ├── expressions.rs          # Тесты для expressions.rs
│       ├── statements.rs           # Тесты для statements.rs
│       └── ...
│
└── traits/                         # Модуль-директория (с mod.rs)
    ├── mod.rs                      # #[cfg(test)] mod tests; ——объявляет tests/ того же уровня
    ├── solver.rs
    ├── impl_check.rs
    ├── inheritance.rs
    ├── coherence.rs
    ├── auto_derive.rs
    ├── object_safety.rs
    ├── resolution.rs
    ├── std_traits.rs
    ├── gat/
    ├── specialization/
    └── tests/                      # ✅ Директория тестов traits
        ├── mod.rs
        ├── solver.rs               # Тесты для solver.rs
        ├── impl_check.rs           # Тесты для impl_check.rs
        └── ...
```

**Почему тесты однофайловых модулей размещаются в родительском `tests/`?**

Потому что однофайловые модули (такие как `overload.rs`) не имеют собственного `mod.rs`, поэтому не могут объявить `#[cfg(test)] mod tests;`. Согласно системе модулей Rust, тестовые файлы должны быть объявлены в каком-либо `mod.rs` для компиляции. Поэтому тесты однофайловых модулей могут быть объявлены только родительским `mod.rs` и размещаться в родительской директории `tests/`.

**Алгоритм принятия решения**:

```
Встретили модуль, куда поместить тесты?
│
├── Этот модуль является директорией (с mod.rs)?
│   └── Да → Создать tests/ в этой директории, объявить в mod.rs этой директории
│
├── Этот модуль однофайловый (только .rs)?
│   └── Да → Тесты в директории tests/ родительского модуля, объявить в родительском mod.rs
│
└── Не уверены?
    └── Проверить наличие отдельной директории и mod.rs
```

**Типичные ошибки**:

```
# ❌ Ошибка 1: Создание отдельной директории tests/ для однофайлового модуля
src/frontend/core/typecheck/
├── overload.rs
└── overload/                       # ❌ Не должно быть директории для однофайлового модуля
    └── tests/
        └── overload.rs

# ❌ Ошибка 2: Объявление #[cfg(test)] mod tests; внутри однофайлового модуля
# overload.rs
#[cfg(test)]                        # ❌ Однофайловый модуль не может так объявлять
mod tests;                          # потому что нет директории overload/tests/

# ✅ Правильно: Тесты в родительском tests/
src/frontend/core/typecheck/
├── overload.rs                     # Исходный файл
└── tests/
    └── overload.rs                 # Тестовый файл, объявлен в typecheck/mod.rs
```

⚠️ **Антипаттерн——не делайте так:**

```
# ❌ Ошибка: Тесты подмодулей собраны в родительском
src/frontend/core/types/
├── mod.rs              # Должен объявлять только base и computation
├── base/
│   ├── mod.rs
│   └── var.rs
└── tests/              # ❌ Родительский tests/ содержит тесты подмодулей
    ├── mod.rs          # ❌ Вынужден объявлять mod base; mod computation;
    ├── base/           # ❌ Эта часть должна быть в base/tests/
    │   └── var.rs
    └── computation/    # ❌ Эта часть должна быть в computation/tests/
        └── ...
```

```
# ✅ Правильно: Тесты каждого модуля независимы
src/frontend/core/types/
├── mod.rs              # Только pub mod base; pub mod computation;
├── base/
│   ├── mod.rs          # #[cfg(test)] mod tests; ——объявляет tests/ того же уровня
│   ├── var.rs
│   └── tests/
│       ├── mod.rs
│       └── var.rs
└── computation/
    ├── mod.rs          # #[cfg(test)] mod tests; ——объявляет tests/ того же уровня
    ├── operations.rs
    └── tests/
        ├── mod.rs
        └── operations.rs
```

**Почему нельзя агрегировать вверх?** Потому что система модулей Rust требует, чтобы `#[cfg(test)] mod tests;` решал вопрос компиляции тестовых файлов в месте объявления. Если `types/mod.rs` объявляет `mod tests;`, то содержимое `types/tests/` становится приватным содержимым модуля `types` — оно не должно проникать в территорию `base` или `computation`. Тесты каждого модуля должны быть внутренними деталями реализации этого модуля, а не родительского. Это правило также применяется при рефакторинге модулей: когда вы разделяете `types` на `base` и `computation`, тесты также должны следовать за разделёнными модулями, а не оставаться на месте. **Директория тестов не зеркалирует структуру исходного кода, а следует за границами модулей.**

**Правило 1.2**: `tests/mod.rs` отвечает только за объявление модулей и re-export, тестовые функции не размещаются.

```rust
//! Тесты ядра парсера — зеркало src/frontend/core/parser/
//!
//! Тесты для ast.rs, parser_state.rs и парсинга выражений/интеграции.

mod ast;
mod error_recovery;
mod expressions;
mod integration;
mod parser_state;
```

**Правило 1.3**: Каждый тестовый файл соответствует только одному исходному файлу. Недопустимо смешивать тесты нескольких исходных модулей в одном файле.

**Правило 1.4**: `#[cfg(test)]` должен появляться только в двух местах——в `lib.rs` для объявления `mod tests`, или внутри тестируемого исходного файла для inline-объявления `#[cfg(test)] mod tests;`. В других местах использование запрещено.

```rust
// src/frontend/core/parser/mod.rs или lib.rs
#[cfg(test)]
mod tests;
```

### Стандарты объявления модулей

**Правило 2.1**: Все тестовые файлы должны иметь модульный doc-комментарий `//!` в начале, указывающий источник покрываемой спецификации (номер главы спецификации языка + номер RFC). Если какой-либо тест не ссылается ни на одну главу спецификации, это означает, что данный код не имеет обоснования в спецификации——он не должен существовать.

```rust
//! Тесты литералов — на основе спецификации языка §2.6
//!
//! §2.6.1: Целые Decimal, Octal(0o), Hex(0x), Binary(0b)
//! §2.6.2: Числа с плавающей точкой (с десятичной точкой и экспонентой)
//! §2.6.3: Строки (escape-последовательности \\nrt'"\\, \\x, \\u{})
//! RFC-012: Интерполяция F-String
```

**Почему обязательна ссылка на спецификацию?** Потому что ожидаемые значения тестов происходят из спецификации, а не из «текущего вывода кода». Если有一天 код изменит вывод, а тест随之更新, значит тест ничего не защищает. Только тесты, привязанные к спецификации, могут отличить «намеренное breaking change» от «непреднамеренной регрессии».

**Правило 2.2**: Импорты `use` в тестовых модулях должны быть точными до конкретного типа/функции, запрещены glob-импорты `use super::*`.

```rust
// 🟢 Хорошо——точный импорт
use crate::frontend::core::lexer::{tokenize, TokenKind};
use crate::frontend::core::parser::{ParserState, ParseError};

// 🔴 Мусор——непонятно, что именно тестируется
use super::*;
```

### Стандарты именования

**Правило 3.1**: Формат имени тестовых функций: `test_<что>_<сценарий>`, все буквы строчные, слова разделены подчёркиваниями.

```rust
#[test]
fn test_tokenize_empty_string() { /* ... */ }
#[test]
fn test_parse_int_overflow() { /* ... */ }
#[test]
fn test_typecheck_fn_return_mismatch() { /* ... */ }
```

**Правило 3.2**: Имена тестовых функций должны быть самоописательными. Прочитав имя функции, должно быть понятно, что тестируется и чего ожидаем. Запрещено именование числовыми порядковыми номерами.

```rust
// 🟢 Хорошо
fn test_skip_semicolon_success() { /* ... */ }
fn test_skip_semicolon_failure_when_identifier() { /* ... */ }

// 🔴 Мусор——совершенно непонятно, что тестируется
fn test_skip_1() { /* ... */ }
fn test_skip_2() { /* ... */ }
```

**Правило 3.3**: Вспомогательным функциям префикс `test_` не нужен, следует использовать глаголы или существительные, описывающие их назначение.

```rust
fn parse_expr(source: &str) -> Expr { /* ... */ }
fn tokenize_single(source: &str) -> Token { /* ... */ }
fn setup_parser_with_tokens(tokens: &[Token]) -> ParserState { /* ... */ }
```

### Стандарты структуры тестов (Arrange-Act-Assert)

**Правило 4.1**: Каждая тестовая функция должна следовать трёхчастной структуре: Подготовка (Arrange) → Выполнение (Act) → Проверка (Assert), части разделяются пустой строкой.

```rust
#[test]
fn test_parse_binary_addition() {
    // Arrange
    let source = "1 + 2";

    // Act
    let expr = parse_expr(source);

    // Assert
    assert!(matches!(expr, Expr::Binary { op: BinOp::Add, .. }));
}
```

**Правило 4.2**: Простые тесты (один вызов + одно утверждение) могут не иметь комментариев о частях, но не более 5 строк логического кода. Тесты более 5 строк должны явно указывать три части.

### Стандарты вспомогательных функций

**Правило 5.1**: Повторяющийся 3 и более раз setup-код должен быть извлечён во вспомогательные функции.

```rust
// 🟢 Хорошо——извлечён общий setup
fn with_state<F>(source: &str, mut f: F)
where
    F: FnMut(&mut ParserState<'_>),
{
    let tokens = tokenize(source).unwrap();
    let mut state = ParserState::new(&tokens);
    f(&mut state);
}

#[test]
fn test_current_returns_first_token() {
    with_state("42", |state| {
        let tok = state.current();
        assert_eq!(&tok.unwrap().kind, &TokenKind::IntLiteral(42));
    });
}
```

**Правило 5.2**: `unwrap()` / `expect()` во вспомогательных функциях при панике должны выводить достаточный контекст. В теле тестовой функции (`#[test] fn ...`) можно напрямую использовать `unwrap()`——при неудаче Rust автоматически выводит номер строки; но при неудаче во вспомогательной функции номер строки указывает на определение вспомогательной функции, контекст вызова теряется.

```rust
// 🟢 Хорошо——при неудаче выводится содержимое исходного кода
fn run_ok(source: &str) {
    run(source).unwrap_or_else(|e| panic!("Execution failed:\nSource:\n{}\nError:\n{:?}", source, e));
}

// 🔴 Мусор——при неудаче непонятно, какой исходный файл вызвал проблему
fn run_ok(source: &str) {
    run(source).unwrap();
}
```

**Правило 5.3**: Вспомогательные функции следует размещать в верхней части тестового файла, сразу после импортов `use`. Если они используются несколькими тестовыми модулями, размещайте их в `tests/mod.rs` и экспортируйте через `pub(crate)`.

### Стиль утверждений

**Правило 6.1**: Для сопоставления вариантов перечисления предпочтительно использовать `assert!(matches!(...))`, запрещено использовать `if let` + `panic!`.

```rust
// 🟢 Хорошо
assert!(matches!(tokens[0].kind, TokenKind::IntLiteral(42)));

// 🔴 Мусор
if let TokenKind::IntLiteral(v) = tokens[0].kind {
    assert_eq!(v, 42);
} else {
    panic!("Expected IntLiteral");
}
```

**Правило 6.2**: Для точного сравнения значений используйте `assert_eq!`, для булевых утверждений используйте `assert!`. Запрещено использовать `assert!(a == b)` вместо `assert_eq!(a, b)`.

**Правило 6.3**: Все утверждения должны содержать пользовательское сообщение об ошибке, если само утверждение не описывает причину неудачи полностью.

```rust
// 🟢 Хорошо——при неудаче можно быстро локализовать
assert!(
    state.infix_info().is_some(),
    "infix_info should handle '{op}'"
);

// 🟢 Хорошо——assert_eq! при неудаче автоматически выводит разницу значений
assert_eq!(error_count, 0);

// 🔴 Мусор——при неудаче只知道 "assertion failed"
assert!(state.infix_info().is_some());
```

**Правило 6.4**: Порядок в утверждениях должен быть `assert_eq!(actual, expected)` — фактическое значение впереди, ожидаемое сзади.

### Чёрный список антипаттернов

Ниже приведены запрещённые写法及其替代方案:

| Антипаттерн | Проблема | Альтернатива |
|-------------|----------|-------------|
| Тест подстраивается под ошибочное поведение кода | Маскирует отклонение от спецификации, легитимизирует баги | Сверяясь со спецификацией, исправить код, тесты не менять |
| Обратный вывод ожидаемых значений теста по выводу кода | Тест становится «диктофоном текущей реализации» | Выводить ожидаемые значения из спецификации |
| Постоянная метка `#[ignore]` | Скрывает устаревшие тесты | Исправить или удалить |
| Отладочный вывод `println!` | Загрязняет вывод тестов | Использовать `assert!` для явных утверждений |
| `thread::sleep` | Случайные сбои + медленно | Использовать синхронизацию или mock |
| Операции с реальной файловой системой в тестах | Медленно и невоспроизводимо | Использовать `tempfile` |
| Зависимость от порядка выполнения тестов | Случайные сбои | Независимый setup для каждого теста |
| Одна тестовая функция более 30 строк логики | Никто не поймёт | Разделить тесты или использовать вспомогательные функции |
| `unwrap()` во вспомогательных функциях без контекста | Сложно локализовать | Использовать `expect("причина")` или пользовательскую панику (см. правило 5.2) |
| copy-paste одного и того же setup 3+ раза | Высокая стоимость изменений | Извлечь вспомогательную функцию |

## Стандарты интеграционных тестов

### Организация тестов

**Правило 7.1**: Интеграционные тесты располагаются в директории `tests/` в корне проекта. Входной файл `tests/integration.rs` использует атрибут `#[path]` для подключения подмодулей.

```rust
// tests/integration.rs
#[path = "integration/backends.rs"]
mod backends;
#[path = "integration/codegen.rs"]
mod codegen;
#[path = "integration/execution.rs"]
mod execution;
```

**Правило 7.2**: Каждый файл `tests/integration/*.rs` соответствует одной теме тестирования (бэкенды компилятора, генерация кода, исполнитель и т.д.), смешивание запрещено.

**Правило 7.3**: Интеграционные тесты должны тестироваться через публичный API проекта. Запрещено в интеграционных тестах напрямую ссылаться на внутренние модули `crate::`. Используйте публичный путь `yaoxiang::`.

```rust
// 🟢 Хорошо——через публичный API
use yaoxiang::run;

// 🔴 Мусор——обход границы публичного API
use yaoxiang::middle::codegen::bytecode::BytecodeFile;
```

### Управление тестовыми данными

**Правило 8.1**: В интеграционных тестах предпочтительно использовать inline-строки исходного кода. Внешние fixture-файлы используются только когда исходный код превышает 30 строк (размещаются в `tests/fixtures/`).

```rust
#[test]
fn test_fibonacci() {
    run_ok(
        r#"
        main = {
            mut a = 0
            mut b = 1
            while a < 100 {
                mut next = a + b
                a = b
                b = next
            }
        }
        "#,
    );
}
```

**Правило 8.2**: Fixture-файлы должны иметь расширение `.yx`, имя файла описывает намерение теста.

### Принципы E2E-покрытия

**Правило 9.1**: Интеграционные тесты каждой языковой функции должны покрывать три пути:

| Путь | Описание |
|------|----------|
| Happy path | Корректный вход produces ожидаемый выход |
| Error path | Некорректный вход produces чёткое сообщение об ошибке (не panic) |
| Boundary | Граничные значения (пустой ввод, максимальные значения, предел глубины вложенности) |

**Правило 9.2**: Интеграционные тесты не должны зависеть от сети, системных переменных окружения или внешних сервисов.

## Стандарты бенчмарк-тестов

### Стандарты использования Criterion.rs

**Правило 10.1**: Бенчмарк-тесты统一放在 `benches/` 目录, входной файл — `benches/lib.rs`. Разделение по темам тестирования.

```
benches/
├── lib.rs              # Вход, определяет criterion_group/criterion_main
├── lang_compare/
│   └── fibonacci.rs    # Кроссъязыковое сравнение бенчмарков
├── parser.rs           # Бенчмарки парсера
└── codegen.rs          # Бенчмарки кодогенерации
```

**Правило 10.2**: Каждая бенчмарк-функция должна содержать модульный doc-комментарий `//!` с описанием цели теста и метрик измерения.

```rust
//! Бенчмарки производительности интерпретатора YaoXiang
//!
//! Метрика: время одной итерации (wall time)
//! Базовый уровень: нативная реализация Rust
```

### Предотвращение оптимизации компилятором

**Правило 11.1**: Все выходы протестированного кода в бенчмарках должны通过 `criterion::black_box`阻止编译器优化消除。

```rust
use criterion::{black_box, Criterion};

fn bench_parse(c: &mut Criterion) {
    c.bench_function("parse_fib", |b| {
        b.iter(|| {
            let result = parse(black_box(FIB_SOURCE));
            black_box(result)
        })
    });
}
```

**Правило 11.2**: Входные данные бенчмарков должны быть `const` или `lazy_static`, не должны генерироваться динамически внутри замыкания `iter`——иначе измеряется общее время генерации данных + тестируемой логики.

### Группировка и именование бенчмарков

**Правило 12.1**: Формат именования бенчмарк-тестов: `<тестируемый_модуль>_<сценарий>`, все буквы строчные, слова разделены подчёркиваниями. Согласуется со стандартами именования модульных тестов.

**Правило 12.2**: Необходимо использовать `criterion_group!` для логической группировки связанных бенчмарков. Запрещено все бенчмарки挤在одной группе.

```rust
criterion_group!(parser, bench_parse_expr, bench_parse_stmt);
criterion_group!(codegen, bench_codegen_module, bench_codegen_switch);
criterion_main!(parser, codegen);
```

## Стандарты doctest

### Сценарии использования

**Правило 13.1**: Все `pub` функции, типы и методы должны содержать в документационных комментариях минимум один исполняемый пример кода. Этот пример выполняется через `cargo test --doc`.

```rust
/// Разбивает исходную строку на последовательность Token.
///
/// ```
/// use yaoxiang::frontend::core::lexer::tokenize;
///
/// let tokens = tokenize("42").unwrap();
/// assert_eq!(tokens.len(), 2); // IntLiteral + Eof
/// ```
pub fn tokenize(source: &str) -> Result<Vec<Token>, LexError> {
    // ...
}
```

**Правило 13.2**: Примеры кода в doctest должны компилироваться, а утверждения должны успешно выполняться. Запрещены примеры с меткой `ignore`, за исключением случаев, когда пример демонстрирует ошибку компиляции.

```rust
/// ```ignore
/// // Демонстрация ошибки компиляции——можно ignore
/// let x: int = "string";
/// ```
```

### Требования к покрытию

**Правило 14.1**: Doctest покрывает только happy path API. Граничные случаи и ошибочные пути покрываются модульными тестами.

**Правило 14.2**: Примеры кода в doctest должны быть лаконичными——не более 10 строк. Если пример требует более длинного контекста, значит дизайн API проблематичен.

## Стандарты property-based тестов

### Сценарии использования

**Правило 15.1**: В следующих сценариях必须使用 property-based тесты (proptest или quickcheck) вместо ручного написания нескольких граничных значений:

| Сценарий | Пример |
|----------|--------|
| Round-trip парсера | `parse(pretty_print(ast)) == ast` |
| Сериализация/десериализация | `deserialize(serialize(data)) == data` |
| Тождества математических операций | `a + b == b + a` |
| Компиляторная оптимизация не меняет семантику | `eval(code) == eval(optimize(code))` |

**Правило 15.2**: Для property-based тестов используется `proptest`作为主要的 property testing框架（已在 `Cargo.toml` 的 `dev-dependencies` 中声明）。

```rust
use proptest::prelude::*;

proptest! {
    #[test]
    fn test_roundtrip_serialize_deserialize(value: i64) {
        let serialized = serialize(&value);
        let deserialized: i64 = deserialize(&serialized).unwrap();
        prop_assert_eq!(deserialized, value);
    }
}
```

### Принципы определения свойств

**Правило 16.1**: Каждый property-based тест должен иметь явное объявление свойства——в комментарии указать проверяемый инвариант.

```rust
// Свойство: любой целочисленный литерал после tokenize → tokens_to_string产生相同值
proptest! {
    #[test]
    fn test_int_literal_roundtrip(n in any::<i64>()) {
        let source = n.to_string();
        let tokens = tokenize(&source).unwrap();
        // ...
    }
}
```

**Правило 16.2**: Если property-based тест обнаруживает сбой, необходимо использовать регрессионный механизм `proptest`——добавить входные данные провалившихся тестов в директорию `proptest-regressions/`, не заменять обычным тестом.

## Требования к покрытию

### Цели покрытия新增代码

**Правило 17.1**: Требования к покрытию тестами新增代码:

| Тип кода | Покрытие строк | Покрытие ветвлений |
|----------|----------------|-------------------|
| Основные модули компилятора (frontend/middle/backends) | ≥ 85% | ≥ 80% |
| Утилиты/вспомогательные модули (util) | ≥ 75% | ≥ 70% |
| Модули runtime (vm/runtime) | ≥ 80% | ≥ 75% |
| Стандартная библиотека (std) | ≥ 75% | ≥ 70% |
| Обработка ошибок и диагностика | ≥ 90% | ≥ 85% |

**Правило 17.2**: Пути обработки ошибок (все ветки `Err`) должны быть покрыты на 100%. Сообщения об ошибках, видимые пользователю, должны быть проверены тестами.

### Чеклист для PR Review

**Правило 18.1**: Перед提交 PR, автор должен самопроверить следующие пункты:

- [ ] `cargo test` все проходят
- [ ] `cargo test --doc` все проходят
- [ ] `cargo bench` без регрессий производительности (если затрагиваются hot paths)
- [ ] 新增代码 соответствует целям покрытия
- [ ] Имена тестов соответствуют стандартам именования
- [ ] Каждый тестовый файл объявляет соответствующую главу спецификации (правило 2.1)
- [ ] Ожидаемые значения тестов происходят из определений спецификации, а не из «вывода текущего кода»
- [ ] Нет тестов с меткой `#[ignore]` (если только нет явного номера issue в комментарии)
- [ ] Нет необязательных `unwrap()` (должны использоваться `expect` или пользовательские panic-сообщения)
- [ ] Сообщение коммита использует тип `:white_check_mark: test:`
- [ ] **Не изменялись ожидаемые значения тестов из-за «несоответствия поведения кода спецификации»——исправлялся код, а не тесты**

**Правило 18.2**: Reviewer должен отклонять PR, содержащие следующие проблемы:

- Только happy path тесты, отсутствуют ошибочные пути
- В тестах есть `thread::sleep` или зависимость от порядка выполнения
- Скопированный тестовый код более 3 раз без извлечения вспомогательных функций
- Имена тестов не соответствуют стандартам именования
- Есть постоянные тесты с `#[ignore]`
- **Тесты подстраиваются под ошибочное поведение кода** (когда код не соответствует спецификации, меняют тесты вместо кода)
- **Тесты не объявляют соответствующие главы спецификации** (см. правило 2.1)
- **Ожидаемые значения тестов происходят из вывода кода, а не из определений спецификации** (выведенные тесты равносильны отсутствию тестов)
- Тесты проверяют только «не паникует», но не утверждают конкретное поведение
- Удалены провалившиеся тесты,暴露ляющие баги в коде (а не исправлен код, чтобы они стали зелёными)

## Приложения

### A. Шпаргалка по командам тестирования

```bash
# Запустить все тесты
cargo test

# Только модульные тесты
cargo test --lib

# Только интеграционные тесты
cargo test --test integration

# Только doctest
cargo test --doc

# Запустить конкретный тест (фильтр по имени)
cargo test test_parse_expr

# Запустить бенчмарки
cargo bench

# Показать вывод тестов (по умолчанию stdout скрыт)
cargo test -- --nocapture

# Запустить в одном потоке (для排查并发问题)
cargo test -- --test-threads=1

# Сгенерировать отчёт о покрытии (нужен cargo-llvm-cov)
cargo llvm-cov --html
```

### B. Шаблон сообщения коммита

Коммиты, связанные с тестами, должны следовать следующему шаблону:

```
:white_check_mark: test(<scope>): <краткое описание>

<опционально: список покрытых сценариев>
```

Пример:

```
:white_check_mark: test(parser): добавить тесты инфиксных операторов в Pratt-парсере

Покрытые сценарии:
- Приоритет арифметических операторов (+, -, *, /, %)
- Связывание операторов сравнения (1 < x < 10)
- Ленивое вычисление логических операторов
- Правоассоциативность операторов присваивания
```

### C. Чеклист новых тестовых файлов

При создании нового тестового модуля убедитесь, что включены следующие файлы:

```
# Новые тесты в директории src/<module>/
src/<module>/tests/
├── mod.rs          # Объявление модуля + общие вспомогательные функции
└── <subject>.rs    # Тестовый файл, соответствует имени тестируемого исходного файла

# Новые интеграционные тесты в директории tests/
tests/
├── integration.rs   # Обновить: добавить объявление #[path]
└── integration/
    └── <topic>.rs   # Новый тестовый файл
```
