# Runtime (Среда выполнения)

> **Состояние модуля**: Есть пробелы (4 пункта требуют улучшения, этап B не начат)
> **Расположение**: `src/backends/runtime/`
> **Последнее обновление**: 2026-06-05

---

## Обзор модуля

Модуль runtime отвечает за планирование задач и параллельное выполнение. Реализует трёхуровневую архитектуру runtime, определённую в RFC-008: Embedded / Standard / Full.

**Объём кода**: ~95 КБ (4 исходных файла)

---

## Список функций

### engine.rs — Ядро планирования DAG (реализовано)

- ✅ **Управление жизненным циклом задач**: spawn / mark_running / complete / cancel / yield_now
- ✅ **DAG зависимостей**: жёсткие зависимости (hard_deps, распространение ошибок/отмены) + контрольные зависимости (control_deps, только упорядочение без распространения ошибок)
- ✅ **Сериализация ресурсов**: механизм ResourceKey, задачи с одинаковым ключом строго сериализуются, при отмене ожидают контрольные зависимости для сохранения порядка ресурсов
- ✅ **Обнаружение циклических зависимостей**: проверка связности в add_dependency, возврат ошибки CycleDetected
- ✅ **Распространение ошибок/отмены**: ошибки распространяются BFS по рёбрам жёстких зависимостей, при одновременных ошибках нескольких зависимостей объединяются причины отмены (primary + others)
- ✅ **Кооперативное временное срезание**: drive_until_polled поддерживает TaskPoll::Pending для передачи управления, честное переключение между задачами
- ✅ **Планирование с приоритетом цели**: next_ready_for(target) приоритизирует продвижение цепочки зависимостей цели, изолированные задачи не блокируют завершение цели
- ✅ **Статистика**: RuntimeStats (pending/running/completed/failed/cancelled/total_spawned/avg_execution_time)

### facade.rs — Фасад трёхуровневого runtime (реализовано)

- ✅ **Embedded Runtime**: немедленное выполнение, замыкание запускается сразу при spawn, без DAG, не поддерживает deps/resources
- ✅ **Standard Runtime**: однопоточное DAG-планирование, поддержка обычных TaskFn и кооперативных CoopTaskFn
- ✅ **Full Runtime**: многопоточное выполнение, связь через crossbeam channel, пул рабочих потоков
- ✅ **Унифицированный фасад Runtime**: конфигурация через RuntimeConfig(mode, workers, work_stealing)

### task.rs — Абстракция задачи (реализовано)

- ✅ **TaskId**: уникальный идентификатор, поддержка Display
- ✅ **TaskPriority**: четыре уровня — Low/Normal/High/Critical
- ✅ **TaskState**: Pending/Running/Completed/Failed/Cancelled
- ✅ **TaskConfig**: конфигурация через builder (priority/name/stack_size/parent_id)
- ✅ **Task**: сущность задачи, содержит id/config/state/result
- ✅ **TaskContext**: контекст выполнения задачи (registers/stack/locals/entry_ip)
- ✅ **Scheduler trait**: абстрактный интерфейс планировщика
- ✅ **TaskSpawner**: обёртка универсального планировщика задач

## Сравнение с RFC (RFC-008 / RFC-024)

| Требование RFC | Состояние реализации | Пояснение |
|-------------|---------|------|
| Трёхуровневая архитектура Embedded/Standard/Full | ✅ Реализовано | Три варианта RuntimeInner в facade.rs |
| Отвязка планировщика (generics + внедрение) | ⚠️ Частично реализовано | Scheduler trait есть в task.rs, но facade.rs использует enum напрямую |
| Синхронный = особый случай планирования (num_workers=1) | ✅ Реализовано | Тест Full workers=1 подтверждает согласованность со Standard |
| Ленивое вычисление DAG | ✅ Реализовано | LocalRuntime в engine.rs |
| Модель выполнения снизу вверх | ✅ Реализовано | drive_until / next_ready_for приоритизируют цепочку зависимостей цели |
| Изолированные DAG параллельно не блокируют друг друга | ✅ Реализовано | Есть специальный тест |
| WorkStealer | ⚠️ Заявлена поддержка, но не реализована независимо | FullRuntime использует crossbeam channel, нет настоящей work-stealing очереди |
| Компиляционный анализ DAG | ❌ Не реализовано (этап B по RFC-024) | Текущий DAG строится в runtime, RFC-024 планирует перенос в компиляцию |
| Параллельное выполнение прямых подвыражений spawn-блока | ❌ Не реализовано (RFC-024) | Текущий spawn целиком оборачивается в одиночное замыкание |
| Статическая библиотека планировщика (200-500 КБ) | ❌ Не реализовано (этап B) | Относится к области LLVM AOT компилятора |
|反射чные метаданные с загрузкой по требованию | ❌ Не реализовано (этап B) | Относится к дальнейшему планированию |

## Ключевые находки

1. **Scheduler trait в task.rs и реальное планирование в facade.rs разделены**: task.rs определяет Scheduler trait, но facade.rs не использует этот трейт, а напрямую диспетчеризирует через enum
2. **Дублирование определений типов в task.rs**: SyncValue, TaskResult, RuntimeError, SchedulerStats определены отдельно в engine.rs и task.rs
3. **WorkStealing реально не реализован**: поле work_stealing есть в RuntimeConfig, но FullRuntime фактически использует простую модель пула потоков + channel
4. **RFC-024 изменит модель выполнения spawn**: текущий spawn целиком оборачивается в одиночное замыкание и планируется runtime DAG; RFC-024 планирует анализ зависимостей прямых подвыражений внутри spawn-блока на этапе компиляции, генерацию плана выполнения, а runtime выполняет группы параллельно по плану
