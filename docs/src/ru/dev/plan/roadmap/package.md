# Управление пакетами (Package)

> **Статус модуля**：Завершено (Phase 1 + Phase 2)
> **Расположение**：`src/package/`
> **Последнее обновление**：2026-06-01

---

## Обзор модуля

Модуль управления пакетами отвечает за управление зависимостями проекта, разбор конфигурации пакетов, загрузку зависимостей и т.д. Реализует Phase 1 (разбор toml, локальные зависимости, генерация lock) и Phase 2 (поддержка GitHub, управление .yaoxiang/vendor, инструменты загрузки), определённые в RFC-014.

**Объём кода**：около 5000 строк (23 исходных файла)

---

## Список функций

### Реализованные функции (12 пунктов)

1. ✅ **Файл манифеста yaoxiang.toml** — метаданные пакета (name, version, description, authors, license), объявление зависимостей (dependencies / dev-dependencies), сериализация/десериализация TOML
2. ✅ **Lock-файл yaoxiang.lock** — блокировка записей зависимостей (version, source, checksum), синхронизация из манифеста, принудительное обновление, очистка устаревших зависимостей
3. ✅ **Разбор спецификаций зависимостей (DependencySpec)** — разбор из TOML значений (строковая форма `"1.0.0"` и табличная форма `{version, git, path}`)
4. ✅ **Разбор семантических версий (SemVer / VersionReq)** — разбор формата `major.minor.patch[-pre]`, поддержка операторов `^`, `~`, `>=`, `>`, `<=`, `<`, точное совпадение, `*`
5. ✅ **Абстракция источников зависимостей (Source trait)** — `LocalSource` (локальный путь), `GitSource` (клон Git-репозитория, поддержка tag/branch/rev), `RegistrySource` (заглушка, Phase 3)
6. ✅ **Поддержка Git-зависимостей** — разбор URL (`?tag=`, `?branch=`, `?rev=` параметры), получение списка тегов через `git ls-remote`, сопоставление semver-тегов, неглубокое клонирование `git clone --depth 1`
7. ✅ **Обнаружение конфликтов версий** — автоматическое обнаружение несовместимых требований к версиям одного пакета
8. ✅ **Резолвер модулей (ModuleResolver)** — поиск по приоритету: vendor -> src -> YXPATH -> std
9. ✅ **Управление Vendor-каталогом (VendorManager)** — управление каталогом `.yaoxiang/vendor/<name>-<version>/`, установка/удаление/список/очистка
10. ✅ **Контрольная сумма SHA-256** — собственная встроенная реализация SHA-256 (без внешних зависимостей), проверка на уровне файлов и каталогов
11. ✅ **Пакетный загрузчик (fetcher)** — унифицированный интерфейс загрузки зависимостей, интеграция с source/vendor/lock
12. ✅ **CLI-команды (6 штук)** — `init`, `add`, `rm`, `install`, `list`, `update`

### Упомянутые в RFC, но не реализованные функции (3 пункта)

- ❌ Команда `outdated` — проверка устаревших зависимостей
- ❌ Команда `clean` — очистка результатов сборки (есть только метод clean на уровне vendor)
- ❌ Команда `task <name>` — выполнение пользовательских задач

## Сравнение с RFC (RFC-014)

### Полностью соответствующие части RFC

- ✅ Формат yaoxiang.toml ([package], [dependencies], [dev-dependencies])
- ✅ Структура проекта (src/, .yaoxiang/vendor/, yaoxiang.toml, yaoxiang.lock)
- ✅ Порядок разрешения модулей (vendor -> src -> YXPATH -> std)
- ✅ Расширяемая архитектура Source trait (Local, Git, Registry три типа источников)
- ✅ CLI-команды (init, add, rm, install, update, list)
- ✅ Семантическое версионирование (^, ~, точное, операторы диапазона)

### Различия с RFC

1. **Небольшая корректировка формата Lock-файла** — RFC использует форму массива `[[package]]`, реализация использует форму map `[package.name]`, функционально эквивалентно
2. **Расширения сверх RFC** — автоматическое обнаружение конфликтов версий, встроенная реализация SHA-256, команда init дополнительно генерирует стандартные файлы интерфейсов библиотеки `.yaoxiang/std/`

### Будущие расширения (Phase 3, отмеченные в RFC как "reserved")

- ❌ Registry-источник — только заглушка
- ❌ Поддержка workspace
- ❌ Механизм override зависимостей
