## Обзор модуля

Проверщик владения отвечает за анализ владения в YaoXiang — семантику Move, конфликты токенов заимствования, корректность Drop, нарушения изменяемости, точное освобождение NLL, захват замыканий, запросы сигнатур функций, анализ побега ref.

**Текущая архитектура** (v9 конвейер утверждений Хоара):
- Дерево брендов (BrandTree) отслеживает отношения вывода токенов и оценку конфликтов
- Анализ потребителей управляет освобождением NLL (ReleasePlan)
- Обратный BFS-анализ живости (быстрый канал, покрывает 95%+ сценариев)
- Резервный вариант логического отсечения SMT (крайне редкие сценарии while + условия пути)
- Drop, управляемый областью видимости (автоматическая пометка VarState::Dropped при выходе из области видимости)
- Анализ захвата замыканий (save/restore/diff → CapturesStore)
- Запросы сигнатур функций (TypeEnvironment → T/&T/&mut T → Move/ReadBorrow/WriteBorrow)
- Анализ побега ref (использование внутри spawn → Arc, иначе → Rc)
- ir_gen читает ReleasePlan и вставляет инструкции Drop + выбирает RcNew/ArcNew по escaped_refs

## Состояние соответствия RFC

### RFC-009: пять основных концепций

| Концепция | Видимое пользователю поведение | Базовая реализация |
|------|-------------|---------|
| **Move** | ✅ Завершено | OwnershipChecker, обнаружение UseAfterMove |
| **&T / &mut T** | ✅ Завершено | Обнаружение конфликтов токенов через дерево брендов (быстрый канал + резервный вариант SMT) |
| **ref** | ✅ Завершено | Анализ побега автоматически выбирает Rc/Arc |
| **clone()** | ✅ Завершено | CloneChecker, 0 тестов |
| **unsafe + *T** | ✅ Завершено | UnsafeChecker |

### RFC-009a: шесть этапов (новая версия — реализация на фронтенде)

| Этап | Содержание | Статус | Описание |
|------|------|------|------|
| 1 | Структура данных дерева брендов | ✅ Завершено | `BrandTree` + `BrandNode` + `BrandId` |
| 2 | Анализ потребителей | ✅ Завершено | `BrandNode.consumers`, автоматический сбор при обходе AST |
| 3 | Обратный BFS-анализ живости | ✅ Завершено | `fast_path_check()`, break отсекает обратные рёбра |

## Основные компоненты новой архитектуры

### `src/frontend/core/typecheck/layers/ownership.rs` (~1600 строк)

| Компонент | Функция |
|------|---------|
| `BrandId` / `BrandTree` | Идентификатор токена + дерево вывода + оценка конфликтов + отслеживание потребителей |
| `ControlFlowGraph` | Узлы/рёбра/условия пути CFG, Break/BackEdge |
| `fast_path_check()` | Обратный BFS-анализ живости (быстрый канал) |
| `smt_cut()` | Логическое отсечение SMT (медленный канал, Z3) |
| 5 системных предикатов | borrow_conflict / use_after_move / use_after_drop / double_drop / mut_violation |
| `OwnershipChecker` | Обход AST + дерево брендов + CFG + проверка предикатов |
| `ReleasePlan` | План точного освобождения NLL (потребители + Drop по области видимости, слияние двух источников) |
| `VarState` | Три состояния Alive / Moved / Dropped |
| `Captures` / `CapturesStore` | Множество захваченных переменных замыкания + хранилище |
| `StateSnapshot` | save_state / restore_state / diff_captures |
| `ParamOwnership` | Move / ReadBorrow / WriteBorrow |
| `ref_vars` / `escaped_refs` / `inside_spawn` | Анализ побега ref (с распространением атрибутов) |

### `src/middle/core/ir_gen.rs`

- Чтение `TypeCheckResult.release_plan` → вставка инструкций `Drop` (точки точного освобождения NLL)
- Чтение `TypeCheckResult.escaped_refs` → `Expr::Ref` выбирает `RcNew` или `ArcNew`

### `src/middle/core/ir.rs` / `bytecode.rs` / `opcode.rs`

- Добавлена инструкция `RcNew` + `Opcode::RcNew(0x89)`
