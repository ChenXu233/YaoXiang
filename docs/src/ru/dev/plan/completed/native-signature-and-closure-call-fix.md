## Обзор

### Контекст проблемы

В текущей версии языка YaoXiang при использовании функций высшего порядка (таких как `list.map`, `list.filter`, `list.reduce`) существуют две связанные проблемы:

1. **Ошибка разбора сигнатуры**: в `src/std/list.rs` строки сигнатур функций `map`/`filter`/`reduce` используют недопустимый тип `Fn`
2. **Вводящее в заблуждение сообщение об ошибке**: когда разбор сигнатуры завершается неудачно, сообщение об ошибке показывает «Invalid signature 'Float': missing '->'», вместо более корректного сообщения

### Проблемный код

`src/std/list.rs` строки 72-87:

```rust
NativeExport::new(
    "map",
    "std.list.map",
    "(list: List, fn: Fn) -> List",  // ❌ Fn — недопустимый тип (должно быть (T) -> T)
    native_map as NativeHandler,
),
NativeExport::new(
    "filter",
    "std.list.filter",
    "(list: List, fn: Fn) -> List",   // ❌ Fn — недопустимый тип
    native_filter as NativeHandler,
),
NativeExport::new(
    "reduce",
    "std.list.reduce",
    "(list: List, fn: Fn, init: Any) -> Any",  // ❌ Fn — недопустимый тип
    native_reduce as NativeHandler,
),
```

**Текущий формат сигнатуры** (правильный):
```
(list: List, fn: Fn) -> List
  ↑       ↑     ↑
  │       │     └── тип параметра (недопустимый)
  │       └── имя параметра
  └── тип параметра
```

**Проблема**: `fn` — это имя параметра, `Fn` — это имя типа. `Fn` не является допустимым именем базового типа.

### Ошибка времени выполнения

При выполнении тестового кода:

```yaoxiang
main = {
    doubled = list.map([1, 2, 3], x => x * 2);
    io.println(doubled);
}
```

Вывод (текущее поведение — ошибка):

```
[Warning] Invalid signature 'Float': missing '->'
[Warning] Invalid signature 'Float': missing '->'
[Warning] Invalid signature 'Float': missing '->'
Error: Runtime error: Type error: Expected function value
```

Вывод после исправления (ожидаемое поведение):

```
[Error] Invalid signature: unknown type 'Fn'
Error: (compilation failed)
```

## Цели реализации

### Цель 1: Исправление определений сигнатур

Согласно RFC-010 унифицированного синтаксиса типов, правильный формат для generics-функций:

```
имя_функции: [список_обобщенных_параметров](список_параметров) -> тип_возврата
```

Где обобщенные параметры `[T]` объявляются на уровне функции и действуют на всю сигнатуру.

Изменение сигнатур `map`/`filter`/`reduce` на (обобщенный параметр `[T]` перед именем функции):

```rust
// map: обобщенный параметр [T] действует на всю функцию
"[T](list: List<T>, fn: (item: T) -> T) -> List<T>"

// filter: обобщенный параметр [T] действует на всю функцию
"[T](list: List<T>, fn: (item: T) -> Bool) -> List<T>"

// reduce: обобщенный параметр [T] действует на всю функцию
"[T](list: List<T>, fn: (acc: Any, item: T) -> Any, init: Any) -> Any"
```

**Описание структуры сигнатуры**:

```
[T](list: List<T>, fn: (item: T) -> T) -> List<T>
│  │         │      │    │        │
│  │         │      │    │        └── тип возврата (использует T)
│  │         │      │    └── тип параметра (использует T)
│  │         │      └── имя параметра
│  │         └── тип параметра (функциональный тип)
│  └── тип параметра (List обобщенный, использует T)
└── объявление обобщенного параметра (действует на всю функцию)
```

### Цель 2: Правила области видимости обобщенных параметров

**Запрет затенения обобщенных параметров** (No Shadowing):

В сигнатуре существует несколько уровней области видимости:

```
[T](list: List[T], fn: [T](item: T) -> T) -> List[T]
│                      │
│                      └── внутренняя область видимости типа функции (параметры типа fn)
└── внешняя область видимости функции (обобщенные параметры функции)
```

**Правила**:

1. **Запрет затенения на одном уровне**: обобщенные параметры в одной области видимости не могут иметь одинаковые имена
2. **Запрет затенения внутренним внешнего**: обобщенные параметры внутренней области видимости не могут иметь те же имена, что и во внешней
3. **Запрет затенения параметрами функции**: имена параметров функции не могут совпадать с любыми обобщенными параметрами

**Допустимые примеры**:

```yaoxiang
// ✅ Допустимо: обобщенный параметр T действует на всю функцию
map: [T](list: List[T], fn: (item: T) -> T) -> List[T]

// ✅ Допустимо: внутренний функциональный тип без обобщенных параметров
map: [T](list: List[T], fn: (item: T) -> T) -> List[T]

// ✅ Допустимо: несколько обобщенных параметров
zip: [T, U](a: List[T], b: List[U]) -> List<(T, U)>

// ✅ Допустимо: имена параметров функции отличаются от обобщенных параметров
foo: [T](x: Int, y: T) -> T
```

**Недопустимые примеры**:

```yaoxiang
// ❌ Недопустимо: обобщенный параметр внутренней функции затеняет внешний (запрет затенения)
"[T](list: List[T], fn: [T](item: T) -> T) -> List[T]"
# Ошибка: Generic parameter 'T' in function type shadows outer generic parameter 'T'

// ❌ Недопустимо: обобщенные параметры с одинаковыми именами (запрет затенения на одном уровне)
"[T, T](x: T, y: T) -> T"
# Ошибка: Duplicate generic parameter 'T'

// ❌ Недопустимо: имя параметра функции совпадает с обобщенным параметром (запрет затенения)
"[T](T: Int) -> T"
# Ошибка: Parameter 'T' shadows generic parameter 'T'
```

### Цель 3: Проверка имен параметров в сигнатуре

При разборе сигнатуры должна проверяться допустимость имен параметров:
1. **Имена параметров не могут повторяться** (E2093)
2. **Запрет затенения обобщенных параметров** (E2095, E2096)

> **Примечание**: случаи, когда имя параметра является ключевым словом, будут автоматически вызывать синтаксические ошибки при разборе сигнатуры парсером, поэтому отдельная проверка не требуется.

Примеры:
```
// Допустимая сигнатура (соответствует RFC-010)
"[T](list: List<T>, fn: (item: T) -> T) -> List<T>"

// Недопустимая сигнатура — имя параметра функции совпадает с обобщенным параметром (запрет затенения)
"[T](list: List<T>, fn: (T: T) -> T) -> List<T>"
# Должна выдаваться ошибка: Parameter 'T' shadows generic parameter 'T'

// Недопустимая сигнатура — повторяющиеся имена параметров
"[T](x: Int, x: Int) -> Int"
# Должна выдаваться ошибка: Invalid signature: duplicate parameter name 'x'

// Примечание: случаи с ключевыми словами автоматически вызывают синтаксические ошибки при разборе
```

### Цель 4: Исправление сообщений об ошибках

При ошибке разбора сигнатуры следует использовать систему кодов ошибок (E2xxx — этап семантического анализа):

**Требуемые новые коды ошибок**:

| Код ошибки | Шаблон сообщения | Описание |
|------------|------------------|----------|
| E2090 | Invalid signature: {reason} | Ошибка разбора сигнатуры (общий случай) |
| E2091 | Invalid signature: unknown type '{type_name}' | Неизвестный тип |
| E2092 | Invalid signature: missing '->' | Отсутствует стрелка |
| E2093 | Invalid signature: duplicate parameter '{name}' | Повторяющееся имя параметра |
| E2094 | Invalid signature: generic '{name}' shadows outer generic | Обобщенный параметр затеняет внешний |
| E2095 | Invalid signature: parameter '{name}' shadows generic | Имя параметра затеняет обобщенный |

> **Примечание**: случаи, когда имя параметра является ключевым словом, не требуют отдельной проверки, так как парсер при разборе сигнатуры при встрече ключевого слова автоматически выдаст синтаксическую ошибку.

**Формат сообщений об ошибках** (соответствует RFC-013):

```
[Error] E2091: Invalid signature: unknown type 'Fn'
 --> std/list.yx:1:1
  |
1 | "(list: List, fn: Fn) -> List"
  | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
help: Use a valid type like '(T) -> T' for function parameters
```

## Схема приемки

### Условие приемки 1: Компиляция проходит успешно

После изменения сигнатур тестовый код должен компилироваться без предупреждений «Invalid signature»:

```bash
$ cargo run -- run tests/closure_test2.yx
# Должен выводить:
# [Test map:]
# [2, 4, 6]
# [Test filter:]
# [3, 4, 5]
# [Test reduce:]
# 10
# [All tests passed!]
```

### Условие приемки 2: Корректные сообщения об ошибках

При использовании недопустимой сигнатуры должен отображаться **Error** (а не Warning) с кодом ошибки (E2xxx):

```bash
# Тест недопустимой сигнатуры
# Ожидаемый вывод:
[Error] E2091: Invalid signature: unknown type 'Fn'
[Error] E2092: Invalid signature: missing '->'
[Error] E2093: Invalid signature: duplicate parameter 'x'
```

### Условие приемки 3: Совпадение имен параметров Lambda

Имена параметров переданной lambda должны совпадать с именами параметров функции, определенными в сигнатуре:

```yaoxiang
// Определение сигнатуры: fn: (item: T) -> T
// Переданная lambda: x => x * 2

// ❌ Ошибка — имена параметров не совпадают
list.map([1, 2, 3], x => x * 2)
# Ожидаемая ошибка: Parameter name mismatch: expected 'item', got 'x'

// ✅ Корректно — имена параметров совпадают
list.map([1, 2, 3], item => item * 2)

// Для reduce сигнатура: fn: (acc: Any, item: T) -> Any
list.reduce([1, 2, 3], (accumulator, item) => accumulator + item, 0)
# ✅ Корректно — имена параметров совпадают
```

### Условие приемки 4: Проверка имен параметров (разбор сигнатуры)

При разборе сигнатуры должны обнаруживаться недопустимые имена параметров:
- Повторяющиеся имена параметров должны вызывать ошибку (E2093)
- Закрытие обобщенных параметров должно вызывать ошибку (E2095, E2096)

> **Примечание**: случаи, когда имя параметра является ключевым словом, автоматически вызывают синтаксические ошибки при разборе, поэтому отдельная проверка не требуется.

```bash
# Ожидаемые ошибки:
[Error] E2093: Invalid signature: duplicate parameter 'x'
[Error] E2095: Invalid signature: generic 'T' shadows outer generic
```

### Условие приемки 5: Функции высшего порядка работают корректно

- `list.map` применяет функцию к каждому элементу списка, возвращает новый список
- `list.filter` оставляет элементы, удовлетворяющие условию, возвращает новый список
- `list.reduce` выполняет накопительное вычисление над элементами

### Процесс разбора сигнатуры

1. `TypeCheckResult::new()` вызывает `register_std_native_signatures()`
2. `register_std_native_signatures()` перебирает экспорты std модуля
3. Для каждого `Export` вызывается `parse_signature(&export.signature, env)`
4. `parse_signature` разбирает строку сигнатуры в `MonoType::Fn`
5. При ошибке разбора используется система кодов ошибок (E2090-E2095)

## Риски и примечания

1. **Поддержка generics**: ✅ Система типов поддерживает разбор generics `List<T>`, обобщенные параметры解析为 TypeRef
2. **Захват переменных в замыканиях**: Текущая реализация не обрабатывает захват внешних переменных в замыканиях,用例 map/filter/reduce не требуют данной функциональности
3. **Дополнительные обнаруженные и исправленные проблемы**:
   - Отсутствует декодер байткода MakeClosure (приводил к превращению замыканий в Nop)
   - Ошибка обработки `FunctionRef::Index` в исполнителе MakeClosure (конструировалось имя «fn_N» вместо использования индекса)
   - io.println не может форматировать отображение содержимого списков (только адрес handle)
