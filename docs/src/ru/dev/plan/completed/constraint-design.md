# RFC-010 / RFC-011 Дизайн ограничений (Constraint)

> **Статус**: Принято
> **Дата создания**: 2026-02-02
> **Последнее обновление**: 2026-02-02
> **Связанные RFC**: [RFC-010 Унифицированный синтаксис типов](../design/accepted/010-unified-type-syntax.md), [RFC-011 Обобщённая система типов](../design/accepted/011-generic-type-system.md)

---

## 一、Основные принципы проектирования

### 1.1 Ограничение = Требование к возможностям

Ограничения определяют возможности (поля и методы), которые должен предоставлять тип:

```yaoxiang
# Определение ограничения (требование к возможностям)
type Drawable = {
    draw: (Self, Surface) -> Void,
    bounding_box: (Self) -> Rect
}

type Serializable = {
    serialize: (Self) -> String
}
```

### 1.2 Ограничения можно использовать только в обобщённом контексте

**❌ Запрещено**: Прямое присваивание с утиной типизацией

```yaoxiang
let d: Drawable = Circle(1)  # Запрещено!
```

**✅ Допустимо**: Обобщённые ограничения

```yaoxiang
draw: [T: Drawable](item: T) -> Void = (item) => {
    item.draw(screen)
}

# При вызове автоматическая проверка
draw(Circle(1))  # ✅ У Circle есть draw, проверка пройдена
draw(Rect(2))    # ❌ У Rect нет draw, ошибка компиляции
```

**Причина**:
- Прямое присваивание — это «после знания», возможно случайное совпадение
- Обобщённые ограничения — это «заблаговременная проверка», намерение ясно

## 二、Сценарии использования

### 2.1 Параметры обобщённых функций

```yaoxiang
# Обработка любых отрисовываемых объектов
process: [T: Drawable](items: List[T]) -> Void = (items) => {
    for item in items {
        item.draw(screen)
    }
}
```

### 2.2 Обобщённые возвращаемые типы

```yaoxiang
# Возврат сериализуемых объектов
serialize_all: [T: Serializable](items: List[T]) -> List[String] = {
    items.map((item) => item.serialize())
}
```

### 2.3 Обобщённые контейнеры данных

```yaoxiang
# Элементы контейнера должны быть отрисовываемыми - неправильный подход
let shapes: List[Drawable] = []  # Ошибка! Ограничения нельзя использовать в необобщённом контексте

# Правильно: использовать обобщённый параметр
let shapes: List[Circle] = []
```

## 三、Правила структурного подтипирования

### 3.1 Правила сопоставления

```yaoxiang
# Определение ограничения
type Config = {
    load: () -> String,
    save: (String) -> Void,
    name: String
}

# Определение типа
type File = {
    filename: String,
    load: () -> String,
    save: (String) -> Void,
    size: Int,          # Дополнительные поля игнорируются
}

# Проверка соответствия File ограничению Config:
#   - load: () -> String ✅ Совпадает
#   - save: (String) -> Void ✅ Совпадает
#   - name: String ❌ Не совпадает (имя поля filename, а не name)
#
# Результат: ❌ File не удовлетворяет Config
```

### 3.2 Алгоритм сопоставления

| Требование | Предоставлено типом | Результат |
|------|---------|------|
| `x: Int` | `x: Int` | ✅ Совпадает |
| `x: Int` | `y: Int` | ❌ Не совпадает |
| `x: Int` | `x: String` | ❌ Не совпадает |
| `fn: (A) -> B` | `fn: (Self, A) -> B` | ✅ Совпадает (Self удаляется) |
| Требуемое поле/метод | Дополнительные поля/методы | ✅ Игнорируются |

### 3.3 Процесс проверки компилятором

```rust
fn check_type_satisfies_constraint(
    typ: &MonoType,
    constraint: &MonoType,
) -> Result<(), ConstraintCheckError> {
    // 1. Проверка, что constraint — допустимое ограничение (все поля — функции)
    if !constraint.is_valid_constraint() {
        return Err(ConstraintCheckError::NotAConstraint);
    }

    // 2. Обход всех требований ограничения
    for (name, required_type) in constraint.required_fields() {
        match lookup_type_field(typ, name) {
            Some(found_type) => {
                // 3. Проверка совместимости типов
                if !types_compatible(found_type, required_type, typ) {
                    return Err(ConstraintCheckError::TypeMismatch {
                        field: name,
                        expected: required_type,
                        found: found_type,
                    });
                }
            }
            None => {
                // 4. Отсутствует обязательное поле
                return Err(ConstraintCheckError::MissingField {
                    field: name,
                    constraint: constraint.name(),
                });
            }
        }
    }

    Ok(())
}
```

## 四、Объявление ограничений при определении типа (опционально)

При определении типа можно объявить, какие ограничения он реализует, для удобства чтения кода и подсказок IDE:

```yaoxiang
# Объявление реализации ограничений при определении типа
type Point = {
    x: Int,
    y: Int,
    draw: (Point, Surface) -> Void,
    bounding_box: (Point) -> Rect,
    serialize: (Point) -> String,
    Drawable,      # Объявление реализации Drawable
    Serializable   # Объявление реализации Serializable
}
```

**Эффект**:
- ✅ Самодокументируемый код
- ✅ IDE может подсказать «Point реализует Drawable»
- ✅ Компилятор проверит корректность объявления

## 五、Обработка ошибок

### 5.1 Типы ошибок

```rust
pub enum ConstraintCheckError {
    #[error("'{0}' не является допустимым ограничением (должны быть только поля функций)")]
    NotAConstraint(String),

    #[error("Тип '{type}' не удовлетворяет ограничению '{constraint}': отсутствует поле '{field}'")]
    MissingField {
        type_name: String,
        constraint: String,
        field: String,
        span: Span,
    },

    #[error("Тип '{type}' не удовлетворяет ограничению '{constraint}': несовпадение типа поля '{field}'")]
    TypeMismatch {
        type_name: String,
        constraint: String,
        field: String,
        expected: String,
        found: String,
        span: Span,
    },

    #[error("Ограничение '{0}' можно использовать только в обобщённом контексте")]
    NotInGenericContext {
        constraint_name: String,
        span: Span,
    },
}
```

### 5.2 Примеры ошибок

```
Error: Тип 'Rect' не удовлетворяет ограничению 'Drawable'
  Требуемый метод 'draw: (Rect, Surface) -> Void' не найден
  Примечание: Добавьте метод 'draw' в Rect для удовлетворения Drawable

Error: Ограничение 'Serializable' можно использовать только в обобщённом контексте
  Возможно, вы имели в виду: 'serialize_all: [T: Serializable](List[T]) -> List[String]'
```

## 六、Почему именно такой дизайн?

### 6.1 Сравнение с альтернативными подходами

| Подход | Проблема |
|------|------|
| `let d: Drawable = Circle(1)` | Случайное совпадение, утиная типизация слишком свободная |
| `impl Drawable for Circle` | Требует нового ключевого слова, нарушает принципы RFC-010 |
| `as Config` преобразование | Увеличивает синтаксическую сложность |
| **Текущий подход: обобщённые ограничения** | Чёткое намерение, проверка на этапе компиляции, без случайных совпадений |

### 6.2 Принципы проектирования

1. **Ограничение = Требование к возможностям**: определяет только «что требуется»
2. **Обобщения = Заблаговременная проверка**: проверка перед вызовом, без случайностей
3. **Ноль новых ключевых слов**: переиспользование существующего синтаксиса
4. **Безопасность на этапе компиляции**: все проверки выполняются во время компиляции

## 九、Вопросы и ответы

### В: Почему запрещено `let d: Drawable = Circle(1)`?

О: Это «проверка задним числом» — Circle случайно имеет метод draw и принимается, хотя это может быть не тем, что задумано. Обобщённые ограничения — это «заблаговременная проверка», код ясно говорит: «мне нужна возможность Drawable».

### В: Как хранить коллекцию объектов Drawable?

О: Используйте обобщённый контейнер или паттерн интерфейса:

```yaoxiang
# Метод 1: Унифицированный конкретный тип
let shapes: List[Circle] = []
shapes.push(Circle(1))

# Метод 2: Использование trait/паттерна интерфейса (требует runtime dispatch)
# Если действительно нужна гетерогенная коллекция, в будущем можно добавить поддержку trait object
```

### В: Чем это отличается от Trait в Rust?

О: Суть похожа, но:
- Нет ключевого слова `impl`
- Нет явного требования объявления (опционально)
- Используется только в обобщённом контексте

### В: Могут ли ограничения содержать поля данных?

О: Да. Ограничения в YaoXiang не ограничены методами:

```yaoxiang
type HasPosition = {
    x: Int,
    y: Int
}

move: [T: HasPosition](item: T, dx: Int, dy: Int) -> T = (item, dx, dy) => {
    # item.x и item.y должны существовать
    item
}
```
