# Детальное отслеживание ошибок во время выполнения (Этап 2): Покрытие Span, Multi-file SourceMap, Bytecode DebugSection, CLI-переключатели

> Статус: Запланировано (для управления реализацией и приёмкой)  
> Связано: `detailed_runtime_errors.md` (Этап 1 — реализованная основная цепочка)

## Цели (4 ограничения, которые необходимо решить на этом этапе)

1. **Покрытие Span**: Добавить `Span` для большего числа IR-инструкций и включить их в DebugMap на этапе Codegen.
2. **Многофайловая структура/модули**: Записывать исходные файлы в DebugMap; ввести `SourceMap` (`file_id -> path/content`) для поддержки позиционирования и рендеринга между модулями.
3. **DebugSection в файле байткода**: Расширить формат `.42` (BytecodeFile), добавив необязательную DebugSection; реализовать чтение/запись для сохранения информации о позиционировании при офлайн-выполнении/отладке.
4. **CLI-переключатель**: Добавить `--debug-info` для команд `run` / `build` (build-bytecode) для управления генерацией DebugMap и (реализуемой на этом этапе) сериализацией.

## Проектные ограничения (должны соблюдаться)

- **Нулевая стоимость в главном цикле интерпретатора**: Логика выполнения во время выполнения полагается только на `ip` и `StackFrame` для проброса ошибок, не обращаясь к отладочной информации на горячих путях.
- **Рендеринг по требованию**: Информация об исходном коде загружается/ищется/рендерится только при всплытии ошибки на уровень CLI.
- **Необязательный переключатель**: При отключении `--debug-info` DebugMap не создаётся, DebugSection не записывается, что исключает лишние выделения памяти и разбухание файлов.

## Решение для структур данных (избегаем破坏性 изменений)

Чтобы не встраивать `file_id` в существующий `Span` (от которого зависят многие компоненты), на этом этапе используется «композиционное позиционирование»:
- `Span` по-прежнему выражает только строку/столбец/offset (сохраняем текущее состояние).
- Добавляется новый `FileId` (целочисленный ID) и `DebugSpan { file_id, span }`.
- Добавляется новый `SourceMap`: индексирует `SourceFile { name, content, ... }` по `file_id`.
- Значение `BytecodeFunction.debug_map` повышается с `Span` до `DebugSpan`.

> Преимущество: не требуется массовый перенос логики Span в Parser/LSP/TypeCheck; обновляется только отладочная цепочка.

## DebugSection в BytecodeFile (расширение формата файла)

### Условие активации

- В `BytecodeFile.header.flags` устанавливается бит `DEBUG_INFO` (согласуется с Codegen/CLI).

### Содержимое хранилища (минимальный замкнутый цикл)

- **SourceMap**: `file_count`, для каждого файла записываются `path` и `content`.
- **DebugMap**: По порядку функций хранятся маппинги `ip -> DebugSpan`:
  - `entry_count`
  - Каждая запись: `ip`, `file_id`, `Span.start(line/col/offset)`, `Span.end(line/col/offset)`

### Стратегия совместимости

- Если `DEBUG_INFO` не включён: DebugSection не записывается, старый формат остаётся без изменений.
- Чтение: если флаги не установлены — DebugSection пропускается; если установлены — парсится и заполняется `FunctionCode.debug_map`.

## Поведение CLI-переключателя (критерии приёмки)

### `yaoxiang run <file> --debug-info`

- Включает сбор DebugMap в Codegen.
- При ошибках во время выполнения вывод включает подсветку фрагментов исходного кода и (при кросс-файловых ошибках) корректные имена файлов/путей.

### `yaoxiang build <file> -o out.42 --debug-info`

- Записывает DebugSection (включая SourceMap + DebugMap).
- При отключённом переключателе `.42` не содержит DebugSection.

## Расширение покрытия Span (рекомендуемый приоритет)

> В первую очередь добавляем для IR-инструкций, которые могут вызывать ошибки во время выполнения.

### P0 (напрямую связаны с существующими типами ошибок)

- `Call/CallVirt/CallDyn`: `FunctionNotFound`
- `Div/Mod`: `DivisionByZero`

### P1 (резервирование для будущих типов ошибок/проверок)

- `LoadIndex/StoreIndex`: `IndexOutOfBounds` (в будущем в сочетании с `BoundsCheck`)
- `LoadField/StoreField`: `FieldNotFound`
- `PtrDeref/PtrLoad/PtrStore`: `InvalidHandle` / ошибки unsafe

## Тестирование и приёмка

- `BytecodeFile`: после записи DebugSection возможно полное восстановление `SourceMap` и `DebugMap` через `read_from` (round-trip).
- `render_runtime_error`: способна выбирать корректный файл для рендеринга по `DebugSpan.file_id` из `SourceMap` (по одному тесту для однофайлового и многофайлового случая).
- CLI: переключатель `--debug-info` значительно меняет вывод (включён: подсветка исходного кода; выключен: только ip/имя функции).
