## 一、问题总结

### 1.1 问题 1：块内函数定义找不到变量

**Симптомы**:
```yaoxiang
main = {
  add = (a, b) => a + b;      // Функция без аннотации типа
  result = add(1, 2);         // ❌ Unknown variable: 'add'
}
```

**Коренная причина**: строки 548-563 в `src/frontend/typecheck/checking/mod.rs`

```rust
// Функция добавляется в область видимости только при наличии аннотации типа!
if let Some(crate::frontend::core::parser::ast::Type::Fn { ... }) = type_annotation {
    // ... Построение типа функции
    self.add_var(name.to_string(), PolyType::mono(fn_type));  // ❌ Не выполняется
}
```

При использовании минимальной формы `add = (a, b) => ...` переменная `type_annotation = None`, поэтому имя функции вообще не добавляется в область видимости.

### 1.2 问题 2：模块级函数能正常工作

Модульные функции (такие как `main = { ... }`) работают корректно, потому что используют другой путь выполнения кода:

```
check_module
  → collect_function_signature (строки 379-382)
    → Добавление переменной типа даже для функций без аннотации
```

Это указывает на то, что инфраструктура для вывода типов существует, просто `check_fn_stmt` использует её некорректно.

### 1.3 问题 3：use std.{io} 字段访问错误

**Симптомы**:
```yaoxiang
use std.{io}
add: (a: Int, b: Int) -> Int = (a, b) => a + b;
main = {
  result = add(1, 2);
  io.println(result);  // ❌ Cannot access field on non-struct type 'fn(t113) -> void'
}
```

**Связанная, но другая проблема**: `io` распознаётся как тип функции, а не модуля.

### 1.4 函数定义的四种形式测试结果

| Форма | Код | На уровне модуля | Внутри блока |
|-------|-----|------------------|--------------|
| Полная форма | `add: (a: Int, b: Int) -> Int = (a, b) => a + b` | ✅ | ✅ |
| Сокращённая (без заголовка Lambda) | `add: (a: Int, b: Int) -> Int = { return a + b }` | ✅ | ✅ |
| Сокращённая (без типов параметров) | `add: (a, b) -> Int = (a, b) => { return a + b }` | ✅ | ❌ |
| Минимальная форма | `add = (a, b) => { return a + b }` | ✅ | ❌ |

## 二、修复方案

### 2.1 问题 1 修复：块内函数定义

**状态**: ✅ Исправлено

Исправление состоит из двух частей:

#### 2.1.1 Исправление проверки типов

Изменена функция `check_fn_stmt` в `src/frontend/typecheck/checking/mod.rs` (строки 546-583):
- Функция добавляется в область видимости независимо от наличия аннотации типа
- При наличии аннотации типа используется аннотированный тип
- В противном случае типовая переменная создаётся из параметров

#### 2.1.2 Исправление генерации IR

Изменён `src/middle/core/ir_gen.rs`:
1. Добавлено поле `nested_functions` для хранения вложенных функций (строка 152)
2. Изменена `generate_local_stmt_ir` для генерации IR вложенных функций (строки 1013-1032)
3. Изменена `generate_module_ir` для добавления вложенных функций в список модульных функций (строки 416-417)

**Результат проверки**:
- ✅ Этап компиляции пройден (ошибка `Unknown variable` больше не возникает)
- ✅ Выполнение в runtime работает корректно

### 2.3 问题 3 修复：use std.{io}

Это требует отдельного исследования, возможно:
1. Оператор `use` после парсинга не корректно устанавливает тип модуля
2. При проверке доступа к полям модуль обрабатывается некорректно

## 四、待处理问题

### 4.1 use std.{io} 字段访问错误

**状态**: ✅ Исправлено

**Содержание исправления**:
Изменена функция `collect_use_statement` в `src/frontend/typecheck/mod.rs` (строки 645-678):
- Для подмодулей создаётся `StructType`, содержащая экспортируемые функции, вместо ошибочного типа `Fn`

**Результат проверки**:
```yaoxiang

main = {
  add = (a, b) => a + b;

  io.println(result)  // ✅ Работает корректно
}
```
