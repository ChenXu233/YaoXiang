## Введение: Мысль о типовой вселенной — ментальная модель

Философия дизайна языка YaoXiang — **«типы решают всё»**. Чтобы помочь разработчикам понять роли различных сущностей в системе типов и их взаимосвязи, мы предлагаем **«мысль о типовой вселенной»** — многоуровневую ментальную модель. Эта модель разделяет концепции языка по семантическим ролям на разные уровни, унифицирует понятия функций, конструкторов типов, интерфейсов, типов, зависящих от значений, и позволяет разработчикам рассматривать весь язык с единой точки зрения.

**Важное замечание**: данная модель представляет собой **внешнюю ментальную модель для пользователей**, цель которой — унификация синтаксиса, руководство семантической подсветкой и упрощение кривой обучения. **Реальная внутренняя реализация компилятора** основана на традиционной кумулятивной стратификации вселенных (`Type₀, Type₁, Type₂, …`), обеспечивающей логическую строгость и предотвращающей парадоксы. Обе модели не противоречат друг другу: внешняя модель не изменяет реализацию компилятора, а лишь служит для понимания и описания.

## Внешняя ментальная модель: Уровни типовой вселенной

| Уровень  | Роль                          | Примеры                                                               |
|----------|-------------------------------|-----------------------------------------------------------------------|
| Type-1   | Значения                      | `42`, `"hello"`, сама функция `add` (функции тоже являются значениями) |
| Type0    | Ключевое слово меатипа        | `Type` (используется для пометки «это тип»)                            |
| Type1    | Конкретные типы               | `Int`, `String`, `Point` (можно напрямую использовать для объявления переменных) |
| Type2    | Функции/конструкторы типов/типы, зависящие от значений | `add: (Int,Int)->Int`, `List: (T: Type) -> Type`, `Vec: (n: Int) -> Type` |
| Type3    | Интерфейсы/ограничения        | `Clone: Type = { clone: (Self) -> Self }`                            |
| Type4+   | Метапрограммирование (резерв) | Макросы, генераторы кода и т.д.                                       |

- Уровень **Type-1** содержит все значения времени выполнения, включая сами функции.
- Уровень **Type0** — это специальное ключевое слово `Type`, указывающее на тип типа.
- Уровень **Type1** — конкретные типы, то есть то, что обычно понимается под «типами данных».
- Уровень **Type2** — все вызываемые сущности: обычные функции (возвращают значения), конструкторы типов (возвращают типы), типы, зависящие от значений (возвращают типы).
- Уровень **Type3** — интерфейсы/ограничения, описывающие набор сигнатур методов.
- **Type4+** зарезервирован для будущих возможностей метапрограммирования.

Эта стратификация полностью основана на **семантических ролях**, а не на логических порядках. Она помогает разработчикам интуитивно понять назначение каждого символа.

## Внутренняя реализация: Традиционная кумулятивная стратификация вселенных

Внутри компилятор использует стандартную кумулятивную стратификацию вселенных (как в Coq/Agda):

- `Type₀`: содержит конкретные типы (например, `Int`) и значения этих типов.
- `Type₁`: содержит типы `Type₀` (то есть типы типов), такие как сам `Type₀`, тип конструктора типа `List : Type₀ → Type₀` и т.д.
- `Type₂`: содержит типы `Type₁` и так далее.
- Соотношение: `Typeₙ : Typeₙ₊₁`, образующее бесконечно восходящую иерархию, что позволяет избежать парадокса `Type: Type`.

Каждое типовое выражение внутри получает уровень вселенной, который автоматически управляется при проверке типов и полностью прозрачен для разработчика. Между уровнями внешней ментальной модели и внутренними уровнями вселенных существует естественное соответствие:

- Конкретные типы (Type1) обычно соответствуют внутреннему `Type₀`.
- Конструкторы типов (Type2) обычно соответствуют внутреннему `Type₁` или выше (в зависимости от вселенной параметров).
- Интерфейсы (Type3) обычно соответствуют внутреннему `Type₁` или выше.

Однако это соответствие является внутренней деталью реализации компилятора, не требующей внимания разработчиков.

## Унифицированный синтаксис

Независимо от стратификации во внешней ментальной модели, синтаксис YaoXiang всегда остаётся унифицированным:

```
identifier : type = expression
```

Здесь `type` — типовое выражение, а `expression` — выражение значения. Никаких специальных ключевых слов вроде `fn`, `struct`, `trait`.

### Конструкторы типов vs Функции/переменные: Основное правило

**Это ключевой выбор дизайна, определяющий правила устранения синтаксической неоднозначности:**

| Запись      | Значение              | Правило                            |
|-------------|-----------------------|-----------------------------------|
| **`x: Type = ...`** | Конструктор типа | `: Type` явно объявляет → принудительно тип |
| **`f = ...`**       | Функция или переменная | Нет `: Type` → HM активно выводит как функцию/переменную |

**Почему так сделано?**

Синтаксис `{ ... }` сам по себе неоднозначен:
- `{ x: Float, y: Float }` может быть **литералом типа** (тип записи)
- `{ a = 1 + 1 }` может быть **блоком кода** (выполняет операторы, возвращает Void)

**Правила устранения неоднозначности:**
- **Есть** `: Type` → принудительная интерпретация как конструктор типа, `{ ... }` — литерал типа
- **Нет** `: Type` → HM активно интерпретирует `{ ... }` как блок кода, выводит функциональный тип

```yaoxiang
# ✅ Конструктор типа: есть : Type
Point: Type = { x: Float, y: Float }

# ✅ Функция: нет : Type, HM выводит как () -> Void
main = { println("Hello") }

# ❌ Ошибка: нет : Type, компилятор не может интерпретировать { ... } как тип
Point = { x: Float, y: Float }  // HM выводит как функцию, не как тип!
```

### Примеры синтаксиса

| Концепция             | Запись в коде                                                     | Пояснение                                      |
|-----------------------|-------------------------------------------------------------------|-----------------------------------------------|
| Переменная            | `x: Int = 42`                                                     | С аннотацией типа                             |
| Функция               | `add: (a: Int, b: Int) -> Int = a + b`                            | С полным типом                                |
| Функция (вывод HM)    | `add = (a, b) => a + b`                                           | Нет `: Type`, HM выводит как функцию          |
| Тип записи            | `Point: Type = { x: Float, y: Float }`                           | **Обязательно** `: Type`                      |
| Интерфейс             | `Clone: Type = { clone: (Self) -> Self }`                        | **Обязательно** `: Type`                      |
| Конструктор типа      | `List: (T: Type) -> Type = { data: Array[T] }`                   | **Обязательно** `: Type`                      |
| Дженерик-функция      | `map: (T: Type, R: Type) -> ((list: List[T], f: (T) -> R) -> List[R]) = ...` | **Обязательно** `: Type`           |
| Тип, зависящий от значения | `Vec: (n: Int) -> Type = (n) => { data: Array[n] }`        | **Обязательно** `: Type`                      |
| Функция с ограничением | `clone: (T: Clone) -> ((value: T) -> T) = (value) => value.clone()` | Нет `: Type`, HM выводит как функцию    |

- Функциональные типы должны содержать имена параметров, например `(a: Int, b: Int) -> Int`.
- Конструкторы типов тоже являются функциями, их тип — `(T: Type) -> Type`, полное соответствие.
- Интерфейсы тоже являются типами, формат определения такой же, как у типов записей.
- **AST унифицирован в узлы Binding**, без разделения по ключевым словам fn/struct/trait.
- **Правила семантической классификации**:
  - Есть `: Type` → конструктор типа
  - Нет `: Type` → HM выводит как функцию/переменную

## Система дженериков

Система дженериков построена на принципе «типы тоже являются значениями»; все концепции дженериков унифицированы в тип-level функции или параметризованные функции с ограничениями.

### Конструкторы типов

```
List: (T: Type) -> Type = { data: Array[T] }
```

- Принимает типовый параметр `T`, возвращает тип записи.
- Применение осуществляется с помощью квадратных скобок `List[Int]` (синтаксический сахар, эквивалент `List(Int)`).

### Дженерик-функции

```
map: (T: Type, R: Type) -> (
    (list: List[T], f: (x: T) -> R) -> List[R]
    ) = ...
```

- Тип — каррированная тип-level функция: сначала принимает типовые параметры, возвращает функциональный тип уровня значений.
- При вызове возможен автоматический вывод типов: `map(ints, double)` автоматически заполняет `T`, `R`.

### Типовые ограничения: Ограничения дженериков vs Объекты интерфейсов

YaoXiang поддерживает два способа реализации типовых ограничений:

#### Способ первый: Ограничения дженериков (полиморфизм времени компиляции)

```
Clone: Type = { clone: (Self) -> Self }
clone: (T: Clone) -> ((value: T) -> T) = (value) => value.clone()
```

- Ограничение `T: Clone` означает, что `T` должен реализовывать интерфейс `Clone`.
- Мономорфизация во время компиляции, для каждого конкретного типа генерируется отдельный код.
- Нулевые накладные расходы во время выполнения (статическая диспетчеризация).
- Множественные ограничения可以用交集типа: `(T: Clone & Debug) -> ...`

#### Способ второй: Объекты интерфейсов (полиморфизм времени выполнения)

```
clone_obj: (value: Clone) -> Clone = (value) => value.clone()
```

- Тип параметра записывается напрямую как имя интерфейса `Clone`.
- Динамическая диспетчеризация во время выполнения через виртуальную таблицу.
- Подходит для разнородных коллекций, сценариев полиморфизма во время выполнения.

| Характеристика    | Ограничение дженерика `(T: Clone)` | Объект интерфейса `value: Clone` |
|-------------------|------------------------------------|----------------------------------|
| Тип параметра     | Конкретный тип `T`                | Интерфейсный тип `Clone`         |
| Тип возвращаемого значения | Конкретный тип `T`        | Интерфейсный тип `Clone`          |
| Генерация кода    | Мономорфизация во время компиляции | Динамическая диспетчеризация во время выполнения |
| Производительность| Нулевые накладные расходы          | Накладные расходы косвенного вызова |
| Сценарии использования | Высокая производительность, сохранение конкретного типа | Разнородные коллекции, полиморфизм во время выполнения |

Оба способа легальны; разработчики могут выбирать по потребностям.

### Типы, зависящие от значений, и константы времени компиляции

В унифицированном дизайне YaoXiang **типы могут зависеть от любых значений, включая результаты вызовов функций**. Это естественным образом требует, чтобы вызовы функций могли вычисляться во время компиляции, а процесс вычисления должен гарантированно завершаться — иначе система типов погрузится в бесконечный цикл.

#### Правила вычисления во время компиляции

- **Функции имеют только одну форму определения**, без специальных маркеров вроде `constexpr`:
  ```
  factorial: (n: Int) -> Int = {
      if n <= 1 { return 1 }
      return n * factorial(n - 1)
  }
  ```
- **Типовые позиции принуждают к вычислению во время компиляции**: любой вызов функции в типовом выражении (например, размер массива, параметр типа, зависящего от значения) требует, чтобы этот вызов был вычислим во время компиляции и гарантированно завершался:
  ```
  matrix: Array[Int, factorial(5)] = ...  # factorial(5) вычисляется во время компиляции
  vec: Vec[factorial(3)] = ...             # параметр зависимого типа тоже вычисляется во время компиляции
  ```

#### Механизм проверки завершимости (на основе логики Хоара)

Для обеспечения безопасности вычислений во время компиляции YaoXiang вводит **редукцию с убывающей метрикой (decreases)**, бесшовно интегрированную с RFC-022:

**Терминационная редукция рекурсивных функций:**

```
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n              # при каждой рекурсии n строго убывает
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}
```

**Терминационная редукция циклов:**

```
sum: (arr: Array[Int, n]) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

- Выражение `decreases` должно быть well-founded (например, натуральные числа).
- Компилятор генерирует условия завершения и вызывает SMT-решатель для доказательства.
- Поддерживаются кортежные метрики (лексикографическое убывание) и пользовательские well-founded типы.

#### Синергия вычислений во время компиляции и проверки завершимости

1. Когда этап проверки типов встречает вызов функции в типовой позиции, компилятор проверяет, помечена ли функция редукцией `decreases`.
2. Если да, то вызывается встроенный интерпретатор для вычисления значения функции с одновременной проверкой выполнения условия убывания на всех путях вычисления.
3. Если `decreases` не указан, но функция очевидно завершаема (например, нерекурсивная), вычисление выполняется напрямую; иначе выдаётся ошибка с предложением добавить терминационную редукцию.

#### Преимущества

- **Безопасность**: гарантируется завершимость вычислений зависимых типов во время компиляции, предотвращается зацикливание системы типов.
- **Унифицированность**: проверка завершимости и верификация частичной корректности используют единый механизм редукции.
- **Постепенное усиление**: возможен переход от проверок во время выполнения к полностью статическому доказательству.

## Type4: Уровень метапрограммирования

В полной картине «типовой вселенной» Type4 определён как уровень метапрограммирования, **выполняющий ограничения генерации кода YaoXiang во время компиляции**.

### Основная идея: Выполнение во время компиляции, результаты инлайнятся

Type4 использует собственный интерпретатор YaoXiang для **выполнения произвольного кода YaoXiang во время компиляции**:
- Повторное использование существующего интерпретатора, не требуется отдельный «тип-level язык»
- Вычисление во время компиляции, результаты напрямую инлайнятся в генерируемый бинарный код
- Нулевые накладные расходы во время выполнения по умолчанию — никакой рефлексии, только обычный код

```
┌─────────────────────────────────────────┐
│  Компиляция derive                      │
│  ├── Интерпретатор выполняет derive-функцию  │
│  ├── Чтение метаданных типа             │
│  ├── Генерация реализации trait, запись в TraitTable │
│  └── Генерация кода → бинарник          │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  Время выполнения                       │
│  └── Обычный бинарник, без рефлексии, без доп. накладных расходов │
└─────────────────────────────────────────┘
```

### Механизм реализации

1. **API рефлексии типов**: типы несут метаданные (список полей, список методов), доступные через функции стандартной библиотеки
   ```yaoxiang
   // Предоставляется стандартной библиотекой
   type Type = { fields: () -> Array[Field], ... }
   ```

2. **Вычисление во время компиляции**: интерпретатор выполняет вызовы функций во время компиляции, результаты инлайнятся
   ```yaoxiang
   let fields = Point.fields()  // Вычисляется во время компиляции → инлайнится как константный массив
   ```

3. **Запись ограничений**: результаты вычисления (сгенерированные реализации trait) записываются в TraitTable

### Синтаксис и примеры

#### Derived-ограничения: Генерация реализации `Debug` для структуры

```yaoxiang
Debug: Type = { fmt: (Self, Formatter) -> Result }

# API рефлексии типов (встроен в стандартную библиотеку)
type Field = { name: String, type: Type }
type Type = {
    fields: () -> Array[Field],
    implements: (trait: Type) -> Bool,
}

# Реализация derive
derive_debug: (T: Type) -> Type = (T) => {
    fields = T.fields()
    for f in fields {
        if !f.type.implements(Debug) {
            return err("Field " + f.name + " does not implement Debug")
        }
    }
    # Генерация реализации метода (запись в TraitTable)
    T.fmt(self, f: Formatter) -> Result = { ... }
}

# Использование
Point: Type = { x: Float, y: Float }
derive_debug(Point)   # Выполняется во время компиляции, генерирует реализацию Debug для Point
```

#### Условные ограничения

```yaoxiang
conditional: (T: Type) -> Type = (T) => {
    if T == Int { return Add[T] }
    else { return Default[T] }
}
```

#### Композиционные ограничения

```yaoxiang
both: (c1: Type, c2: Type) -> Type = (c1, c2) => c1 & c2
```

| Способ вычисления | Генерация текста         | Вычисление интерпретатором во время компиляции |

|---------------|------------------------|---------------------------|
| Объект операций | Текст/синтаксическое дерево | Типы/ограничения   |
| Фаза выполнения | Фаза раскрытия макросов | Фаза проверки типов   |
| Гигиена      | Требует сложных механизмов | Естественная гигиена      |
| Читаемость    | Специальный синтаксис макросов, сложная отладка | Полностью соответствует обычным функциям |
| Проверка компиляции | После раскрытия     | Непосредственно участвует в проверке типов |

### Почему система типов теперь может заменить макросы?

1. **Типы как объекты первого класса**: типы могут вычисляться, передаваться, подвергаться интроспекции как значения.
2. **Ограничения как данные**: ограничения уровня Type3 могут конструироваться, комбинироваться, деривироваться.
3. **Механизм интроспекции**: система типов способна интроспекцировать структуру типов.
4. **Типовые функции**: выполнение чистых функций на этапе проверки типов, генерация новых ограничений.

Это эквивалентно построению внутри системы типов **полноценного по Тьюрингу, чисто функционального, строго типизированного языка метапрограммирования**.

## Семантическая подсветка и реализация LSP

На основе классификации ролей во внешней ментальной модели LSP может предоставлять точную семантическую подсветку кода:

| Семантическая роль      | Внешний уровень | Тип токена      | Пример           |
|-------------------------|-----------------|-----------------|------------------|
| Переменная/константа    | Type-1          | `variable`      | `x`, `PI`        |
| Параметр функции        | Type-1          | `parameter`     | `a` в `(a: Int)` |
| Обычная функция         | Type2 (возвращает не `Type`) | `function` | `add`, `factorial` |
| Конструктор типа        | Type2 (возвращает `Type`) | `type` | `List`, `Vec` |
| Конкретный тип          | Type1           | `type`          | `Int`, `Point`   |
| Интерфейс               | Type3           | `interface`     | `Clone`          |
| Ключевое слово меатипа  | Type0           | `keyword`       | `Type`           |

Правило определения основано на информации после проверки типов: если символьный тип — функциональный тип `(…) -> …`, проверяется, является ли возвращаемый тип `Type` (или его специализацией); если да — это конструктор типа → `type`, иначе — обычная функция → `function`.

## Резюме

- **Внешняя ментальная модель**: «мысль о типовой вселенной» предоставляет разработчикам унифицированную, интуитивную точку зрения для понимания различных концепций языка (значений, типов, функций, интерфейсов и т.д.), направляет дизайн синтаксиса и семантическую подсветку.
- **Внутренняя реализация**: компилятор использует традиционную кумулятивную стратификацию вселенных, обеспечивающую логическую строгость системы типов и предотвращающую парадоксы, полностью прозрачную для внешней модели.
- **Взаимосвязь**: внешняя модель описательна, не влияет на реализацию компилятора; внутренняя реализация технична, не раскрывает детали уровней наружу. Обе дополняют друг друга, совместно формируя мощную и удобную систему типов YaoXiang.

Благодаря такому разделению YaoXiang достигает:

- **Предельной унификации** синтаксиса
- **Мощной выразительности** дженериков и зависимых типов
- **Дружелюбности к разработчикам** в семантических подсказках
- **Теоретической строгости** базовой основы

Таково полное изложение мысли о типовой вселенной YaoXiang — ментальная модель для понимания, подкреплённая прочной традиционной теоретической базой типов.
