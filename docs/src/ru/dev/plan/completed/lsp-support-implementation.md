# План реализации поддержки LSP

> **Задача**: Реализация поддержки протокола языкового сервера (LSP) для YaoXiang
> **На основе RFC**: RFC-017 Дизайн поддержки протокола языкового сервера (LSP)
> **Дата**: 2026-02-23
> **Статус**: В процессе
> **Целевая версия**: v0.7 - v0.9

---

## Обзор

Данный план, основанный на документе RFC-017, разбивает реализацию LSP на 6 этапов, включающих 20 подзадач. Каждая задача содержит подробные цели реализации, критерии приёмки и пункты тестирования.

### Обзор зависимостей

### 0.1 Режим сбора ошибок

**Цели реализации**:
- Модифицировать модуль `src/frontend/typecheck/inference/`, возвращать `Result<Type, Vec<Error>>` вместо немедленного возврата при ошибках
- Реализовать перечисление `ErrorKind`, включающее `Error` (критическая ошибка), `Warning` (предупреждение), `Note` (дополнительная информация)
- Коллектор ошибок продолжает накапливать ошибки, после проверки возвращает все ошибки единовременно

**Критерии приёмки**:
- [x] Типовая проверка возвращает все ошибки для одного файла (без раннего выхода)
- [x] Ошибки содержат информацию об уровне Severity
- [x] При наличии Error публикуются диагностики с ошибками
- [x] При наличии только Warning компиляция продолжается с отображением предупреждений

**Пояснения к реализации**:
- В `StatementChecker` добавлен режим `collect_all_errors`, ошибки не вызывают ранний возврат, а накапливаются в `collected_errors: Vec<Diagnostic>`
- `TypeChecker::check_module_collect_all()` предоставляет точку входа для полного сбора ошибок для LSP
- Используется существующее перечисление `Severity` (Error/Warning/Info/Hint)
- Изменённые файлы: `src/frontend/typecheck/inference/statements.rs`, `src/frontend/typecheck/mod.rs`

**Пункты тестирования**:

- [x] Тест различения уровней Error/Warning/Note
- [x] Тест единовременного возврата накопленных ошибок

**Критерии приёмки**:

### 0.4 Система кэширования документов (DocumentCache)

**Цели реализации**:
- Реализовать структуру `DocumentCache`, содержащую:
  - `version: u32` - номер версии документа LSP
  - `content: String` - текущее содержимое
  - `content_hash: u64` - хэш содержимого (для быстрого сравнения)
  - `ast: Option<Ast>` - кэшированный AST
- Реализовать детектирование инкрементальных изменений (сравнение content_hash)
- Кэширование на уровне файлов: при изменении переразбирать весь файл

**Критерии приёмки**:
- [x] DocumentCache корректно управляет номером версии
- [x] Хэш-детектирование быстро идентифицирует неизменённые документы
- [x] При изменении корректно выполняется повторный парсинг
- [x] Разумное использование памяти (с механизмом очистки)

**Пояснения к реализации**:
- Структура `DocumentCache`: version, content, content_hash, ast (`Option<Module>`), file_path, dirty
- `DocumentStore` управляет всеми открытыми документами, `HashMap<String, DocumentCache>`, с ограничением ёмкости и автоочисткой
- Хэш содержимого использует `DefaultHasher`, `update()` обновляет содержимое и инвалидирует AST-кэш только при изменении хэша
- Стратегия очистки: при превышении `max_documents` (по умолчанию 128) удаляются документы с наименьшим номером версии
- Полный набор тестов (7 юнит-тестов)
- Изменённый файл: `src/util/cache.rs`

**Пункты тестирования**:
- [x] Тест инкремента номера версии
- [x] Тест точности хэш-детектирования
- [x] Тест применения инкрементальных изменений

## Стратегия тестирования

### Юнит-тесты
- Независимые юнит-тесты для каждого модуля
- Использование mock для изоляции зависимостей

### Интеграционные тесты
- Тесты совместимости протокола LSP
- Интеграционные тесты с реальными IDE (VS Code, Neovim)

### Нагрузочные тесты
- Производительность парсинга больших файлов
- Время ответа на дополнение
- Время ответа на переход

| Риск | Меры снижения |
