# План реализации значений по умолчанию для типов и встроенных привязок

> **Статус: Ядро реализовано** — Фазы 1-4 основной фреймворк + улучшения legacy завершены, все 1499 тестов пройдены.

## Обзор

Данный план реализует две ключевые возможности языка YaoXiang, соответствующие RFC-004 (каррирование с много-position привязками) и RFC-010 (унифицированный синтаксис типов):

1. **Значения по умолчанию для полей типов** (RFC-010): поддержка значений по умолчанию для полей типов, опциональное предоставление значений при конструировании
2. **Встроенные привязки** (RFC-004): привязка методов непосредственно в определении типа (ссылки на внешние функции или анонимные функции), точное управление параметрами привязки через position индексы

### Базовый синтаксис (унифицированная модель RFC-010)

```yaoxiang
Point: Type = {
    x: Float = 0,                                    # Поле + значение по умолчанию
    y: Float = 0,                                    # Поле + значение по умолчанию
    distance = distance[0],                          # Привязка внешней функции (position синтаксис RFC-004)
    norm: ((p: Point) -> Float)[0] = ((p) => ...)    # Привязка анонимной функции
}
```

### Изменения основных структур данных

```
Поля тела определения типа:
├── Объявление поля: field: Type
├── Значение по умолчанию поля: field: Type = expression          (RFC-010)
├── Внешняя привязка функции: field = function[position]          (RFC-004)
└── Привязка анонимной функции: field: ((params) -> Return)[position] = ((params) => body)  (RFC-004)
```

### Затрагиваемые модули

| Модуль | Файл | Ответственность |
|--------|------|-----------------|
| AST | `ast.rs` | `StructField`, `TypeBodyBinding`, `BindingKind` |
| Парсер | `declarations.rs` | `parse_struct_type()` - парсинг четырёх типов полей |
| Система типов | `mono.rs` | `StructType.field_has_default` |
| Семантический анализ | `checking/mod.rs` | `check_type_def`, `check_field_default`, `check_field_binding` |
| Генерация IR | `ir_gen.rs` | `CreateStruct`, заполнение значений по умолчанию, пересылка вызовов привязанных методов |
| Байткод | `bytecode.rs` | Кодирование/декодирование `CreateStruct` байткода |
| Runtime | `executor.rs` | Выполнение `CreateStruct` |

## Шаги реализации

### Фаза 1: Расширение парсера ✅

#### 1.1 Расширение AST полей типов

**Цель**: Добавить новые типы полей для различения обычных полей, полей со значениями по умолчанию и привязанных полей

**Критерии приёмки**:
- [x] Парсинг `Point: Type = { x: Float = 0 }` генерирует корректный AST
- [x] Парсинг `Point: Type = { distance = distance[0] }` генерирует узел Binding
- [x] Парсинг `Point: Type = { distance: ((a, b) => Float)[0] = ((a, b) => a + b) }` генерирует узел AnonBinding

**Описание реализации**:
- `StructField` получает новое поле `default: Option<Box<Expr>>`
- Новые структуры: `TypeBodyBinding` и перечисление `BindingKind` (`External` / `Anonymous`)
- `Type::Struct` изменён с tuple варианта на структурный вариант `Type::Struct { fields, bindings }`
- `parse_struct_type()` полностью переписан, поддерживает парсинг четырёх типов полей
- Новые вспомогательные функции: `parse_optional_binding_positions()`, `parse_binding_positions()`, `extract_fn_type_info()`

### Фаза 2: Расширение семантического анализа ✅

#### 2.1 Проверка типов полей со значениями по умолчанию

**Цель**: Проверить, что тип выражения значения по умолчанию соответствует типу поля

**Критерии приёмки**:
- [x] `x: Float = 0` проходит проверку типов (поддержка неявного числового повышения Int → Float)
- [x] `x: Float = "str"` выдаёт ошибку несоответствия типов (String ≠ Float)
- [x] `x: Int = 1.0` выдаёт ошибку (Float не может быть присвоен Int, обратное повышение не поддерживается)

**Описание реализации**:
- `BodyChecker::check_type_def()` → итерация по полям структуры и привязкам для проверки
- `check_field_default()` → вывод типа выражения значения по умолчанию и унификация с типом поля
  - Поддержка неявного числового повышения Int → Float (соответствует использованию `x: Float = 0` в RFC-010)
  - Другие несоответствия типов (String → Float, Float → Int) вызывают ошибку `type_mismatch`
- `StructType` получает новое поле `field_has_default: Vec<bool>`, распространяется во всех местах преобразования типов

#### 2.2 Семантическая проверка привязанных полей (RFC-004 обеспечение безопасности типов)

**Цель**: Проверить, что функция, на которую ссылается привязка, существует, position индекс валиден, типы совпадают

**Критерии приёмки**:
- [x] `distance = distance[0]` для `distance: (a: Point, b: Point) -> Float` проходит проверку
- [x] `distance = distance[5]` для 2-параметрической функции выдаёт ошибку выхода за границы индекса
- [x] `distance = distance[0]` для `distance: (a: String, b: String)` выдаёт ошибку несоответствия типов
- [x] Проверка непустоты списка position индексов

**Описание реализации**:
- `check_field_binding(type_name, binding, span)` принимает имя типа для проверки
- Процесс проверки внешней привязки (RFC-004 § Правила проверки типов):
  1. Список position индексов непустой
  2. Найти полиморфный тип функции, инстанцировать в мономорфный тип функции
  3. Проверить каждый position индекс < количества параметров функции
  4. Проверить совместимость типа параметра в позиции привязки с текущим типом (через `unify`)
- Если функция не найдена в текущей области видимости, пропускается углублённая проверка (функция может быть во внешней области или определена позже)

#### 2.3 Семантическая проверка привязок анонимных функций

**Цель**: Проверить параметры position и типы для привязок анонимных функций

**Критерии приёмки**:
- [x] Проверка непустоты position индексов
- [x] Проверка position индексов в диапазоне параметров анонимной функции
- [x] Соответствие типов параметров в позиции привязки текущему типу

**Описание реализации**:
- Проверка анонимной привязки: непустой position + position < количества параметров + унификация типа параметра в позиции привязки с именем типа

### Фаза 3: Расширение генерации кода ✅

#### 3.1 Генерация кода для инициализации значений по умолчанию

**Цель**: Генерировать код конструктора по умолчанию и конструктора с переопределением значений по умолчанию

**Критерии приёмки**:
- [x] `Point()` генерирует код с вызовом значений по умолчанию
- [x] `Point(1.0)` переопределяет только x, y использует значение по умолчанию
- [x] `Point(1.0, 2.0)` переопределяет все поля

**Описание реализации**:
- Новый IR-оператор `CreateStruct` (`ir.rs`) и байткод-оператор (`bytecode.rs`)
- Новый `Opcode::CreateStruct = 0x79`
- `generate_struct_constructor_ir()` переписан: загрузка всех параметров → `CreateStruct` → `Ret`
- Заполнение значений по умолчанию на стороне вызова: в ветке `Expr::Call` функции `generate_expr_ir`, детекция вызовов конструктора структуры, генерация IR для выражений значений по умолчанию для отсутствующих параметров
- Реализация транслятора `translate_create_struct()` + декодера байткода
- Интерпретатор в `CreateStruct` выделяет `HeapValue::Tuple` и создаёт `RuntimeValue::Struct`

#### 3.2 Генерация кода привязанных методов (RFC-004 перестановка параметров)

**Цель**: Генерировать корректную пересылку вызовов функций для привязанных методов

**Критерии приёмки**:
- [x] `p1.distance(p2)` + `distance = distance[0]` → генерирует вызов `distance(p1, p2)`
- [x] `p1.distance(p2)` + `distance = distance[1]` → генерирует вызов `distance(p2, p1)`
- [x] Много-position привязка `transform = transform[0, 1]` корректно пересылает

**Описание реализации**:
- Новая структура `BindingInfo` (запись исходного имени функции + список позиций привязки)
- Новая `type_bindings: HashMap<String, HashMap<String, BindingInfo>>` (имя типа → имя метода → информация о привязке)
- `register_type_bindings()` на этапе генерации IR конструктора извлекает информацию о привязках из `Type::Struct { bindings }`
- Расширение генерации IR для вызовов методов (`Expr::Call` + ветка `FieldAccess`):
  1. Вывести имя типа объекта (через `get_expr_struct_type_name()`)
  2. Найти информацию о привязках для данного типа
  3. При наличии привязки: переставить параметры согласно правилам position из RFC-004
     - Создать слоты параметров размером `total_params = positions.len() + method_args.len()`
     - Поместить obj в позицию привязки
     - Заполнить оставшиеся позиции параметрами метода по порядку
  4. Вызвать исходное имя функции (не имя метода)
- Привязки анонимных функций используют соглашение об именовании `类型名.__anon_方法名`

#### 3.3 Динамическое разрешение индексов полей

**Цель**: `resolve_field_index()` динамически ищет индексы полей из информации о типах

**Критерии приёмки**:
- [x] Убрана жёсткая кодировка (было x→0, y→1, z→2)
- [x] Точный поиск по имени поля из `struct_definitions`

**Описание реализации**:
- `resolve_field_index(expr, field_name)` переписан:
  1. Через `get_expr_struct_type_name(expr)` вывести имя структурного типа выражения
  2. Найти список полей этого типа из `struct_definitions`, по имени поля вернуть индекс
  3. Запасной вариант: итерация по всем определениям структур для поиска имени поля (когда вывод типов недоступен)
- Новая вспомогательная функция `get_expr_struct_type_name(expr)`:
  - Переменная: поиск в `type_result.local_var_types`, `bindings`, `local_var_types`
  - Вызов конструктора: `Point(...)` напрямую возвращает `"Point"`
- Новая вспомогательная функция `mono_type_to_struct_name(mono_type)`:
  - `MonoType::TypeRef(name)` → `Some(name)`
  - `MonoType::Struct(st)` → `Some(st.name)`

## Риски и зависимости

### Зависимости

- **RFC-004** (каррирование с много-position привязками): синтаксис привязок `[position]`, правила перестановки параметров
- **RFC-010** (унифицированный синтаксис типов): унифицированная модель `name: Type = { ... }`, синтаксис значений по умолчанию полей

### Риски

| Риск | Влияние | Смягчение |
|------|---------|-----------|
| Неоднозначность парсинга | `field = value` может быть присваиванием или привязкой | Различие по синтаксису правой части `=` (ссылка на функцию + position vs Lambda) |
| Неполный вывод типов | Запасной遍历 в `resolve_field_index` может быть неточным | Приоритетное использование результатов проверки типов |
