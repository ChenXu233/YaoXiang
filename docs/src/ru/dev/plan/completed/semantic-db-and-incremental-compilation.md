# План реализации Центра семантической информации и инкрементной компиляции

> **Задача**: Реализовать центр семантической информации для LSP семантической подсветки, инкрементной компиляции и предупреждений о мёртвом коде
> **На основе RFC**: Данный план предназначен для проектирования новой функциональности
> **Связанный RFC**: [RFC-008: Модель параллелизма во время выполнения](../design/rfc/accepted/008-runtime-concurrency-model.md) — DAG параллелизм относится к runtime, не входит в данный план
> **Дата**: 2026-02-23
> **Статус**: Этап 1 + Этап 2 + Этап 3 завершён
> **Целевая версия**: v0.10 - v0.11

---

## Обзор

Данный план разбивает реализацию центра семантической информации на 3 основных этапа. Ключевая идея — **один проход, множественное использование**:

1. **Сбор семантической информации выполняется на этапе проверки типов** (а не отдельный обход AST на уровне LSP)
2. Собранная семантическая информация одновременно используется для LSP семантической подсветки, инкрементной компиляции и анализа мёртвого кода

> **Важное уточнение**:
> - **DAG параллелизм** относится к особенностям runtime (RFC-008), не входит в данный план
> - **Параллельная компиляция модульного графа зависимостей** — это особенность системы сборки, отдельная концепция от runtime DAG
> - Сбор семантической информации должен выполняться на этапе проверки типов, LSP просто повторно использует данные, а не реализует отдельный обход

```rust
pub struct SemanticDB {
    // Путь к файлу -> семантическая информация этого файла
    by_file: HashMap<String, FileSemanticInfo>,
    // Имя символа -> все позиции определения
    symbol_defs: HashMap<String, Vec<SymbolLocation>>,
    // Имя символа -> все позиции ссылок
    symbol_refs: HashMap<String, Vec<SymbolLocation>>,
}

// Семантическая информация одного файла
pub struct FileSemanticInfo {
    pub file_path: String,
    pub tokens: Vec<SemanticToken>,
    pub scopes: Vec<ScopeInfo>,
}

// Семантический токен (используется структура + enum типов, вместо вариантов перечисления)
pub struct SemanticToken {
    pub name: String,
    pub token_type: SemanticTokenType,
    pub modifiers: Vec<SemanticTokenModifier>,
    pub span: Span,
}

pub enum SemanticTokenType {
    Function, Type, Variable, Property, Method,
    Namespace, Parameter, LocalVariable, TypeParameter,
    Keyword, String, Number,
}

pub enum SemanticTokenModifier {
    Declaration, Readonly, Mutable, Public, Generic,
}

// Информация об области видимости
pub struct ScopeInfo {
    pub span: Span,
    pub parent: Option<usize>,  // Индекс родительской области видимости
    pub symbols: Vec<String>,    // Символы внутри области видимости
    pub kind: ScopeKind,         // Global, Function, Block, Lambda
}
```

### 1.2 Интеграция сборщика семантики в TypeCheck

**Проектное решение**: Сбор семантики **не должен** реализовываться отдельно на уровне LSP, а должен выполняться на этапе проверки типов.

**Причины**:
- typecheck уже обходит AST и знает все позиции определений и ссылок символов
- Отдельная реализация SemanticCollector в LSP = повторный обход + поддержка двух наборов логики
- **Хороший стиль**: один проход, множественное использование

**Цели реализации**:
- Расширить функциональность сбора семантики в модуле `src/frontend/typecheck/`
- При проверке типов одновременно создавать данные `SemanticDB`
- Уровень LSP напрямую запрашивает и повторно использует, без повторного обхода AST

**Правила сбора от typecheck** (результаты этапа):
```
StmtKind::Fn        → SemanticTokenType::Function (определение)
StmtKind::TypeDef   → SemanticTokenType::Type (определение)
StmtKind::Var       → SemanticTokenType::Variable (определение)
StmtKind::MethodBind→ SemanticTokenType::Method (определение)
StmtKind::Use       → SemanticTokenType::Namespace (ссылка)
Param               → SemanticTokenType::Parameter (определение)
Expr::Var           → SemanticTokenType::Variable (ссылка)
Expr::Call          → SemanticTokenType::Function (ссылка)
Expr::FieldAccess   → SemanticTokenType::Property (ссылка)
Expr::Cast          → SemanticTokenType::Type (ссылка)
```

**Критерии приёмки**:
- [x] Этап typecheck создаёт SemanticDB
- [x] LSP может запрашивать семантическую информацию от typecheck
- [x] Устранён повторный обход AST на уровне LSP

## Этап 4: Предупреждения о мёртвом коде (интеграция в процесс компиляции)

> **Описание**: Предупреждения о мёртвом коде зависят от информации о ссылках на символы этапа typecheck, являются функцией анализа времени компиляции, не являются особенностью runtime.

> **Корректировка архитектуры**: Анализ мёртвого кода интегрирован в этап typecheck, поскольку оба требуют обхода AST/SemanticDB

### 4.1 Анализатор мёртвого кода

**Цели реализации**:
- Реализовать структуру `DeadCodeAnalyzer`
- Анализировать неиспользуемые экспортированные символы
- Анализировать неиспользуемые импорты
- Генерировать предупреждающие сообщения

**Проектное решение**: Анализ мёртвого кода должен выполняться на этапе **typecheck**, потому что:
- typecheck уже знает все определения и ссылки символов
- Не требуется дополнительный обход AST
- Семантическая информация уже предоставляется через SemanticDB

**Правила анализа**:
```
1. Собрать все точки входа (main, pub функции)
2. Начиная от точек входа, пометить все достижимые символы
3. Недостижимые экспортированные символы -> предупреждение
4. Неиспользуемые импорты -> предупреждение
```

**Структуры данных**:
```rust
pub struct DeadCodeAnalyzer {
    // Точки входа
    entry_points: HashSet<SymbolId>,
    // Все определения символов
    all_defs: HashMap<SymbolId, SymbolDef>,
    // Ссылки на символы (получены из SemanticDB)
    references: HashMap<SymbolId, Vec<Location>>,
    // Список импортов
    imports: Vec<ImportInfo>,
}

pub struct SymbolDef {
    pub name: String,
    pub kind: SymbolKind,
    pub location: Location,
    pub is_exported: bool,
}
```

**Критерии приёмки**:
- [x] Обнаружение неиспользуемых экспортированных функций
- [x] Обнаружение неиспользуемых экспортированных типов
- [x] Обнаружение неиспользуемых импортов
- [x] Формат предупреждающих сообщений корректен

**Тестовые проекты**:

**Цели реализации**:
- Интегрировать обнаружение мёртвого кода в процессе компиляции
- Публиковать предупреждения через событие `CompilationWarning`
- Поддержка различных форматов вывода (терминал, JSON)

**Формат предупреждения**:
```
warning: unused function `dead_function`
  --> src/utils.yx:10:1
   |
10 | fn dead_function() { }
   | ^^^^^^^^^^^^^^^^^^^^^^
   |
   = note: function is never used
```

**Критерии приёмки**:
- [x] Предупреждения о мёртвом коде корректно триггерятся
- [x] Информация о позиции в предупреждениях точная
- [x] Предупреждения конфигурируемые (включение/выключение)
- [x] Формат вывода в терминал эстетичный

**Тестовые проекты**:
- [x] Тест триггера предупреждения
- [x] Тест позиции предупреждения
- [x] Тест конфигурации
- [x] Тест формата вывода

## Пояснение о DAG параллелизме

**DAG параллельная компиляция НЕ включена в данный план**, по следующим причинам:

| Концепция | Принадлежность | Описание |
|------|------|------|
| **Runtime DAG** | RFC-008 Runtime | Граф зависимостей ленивых вычислений, управляет планированием задач runtime |
| **Граф зависимостей модулей** | Этап 3 данного плана | Модульные зависимости на уровне компилятора, используются для инкрементной компиляции |
| **Параллельная компиляция на уровне модулей** | Система сборки | Реализуется на основе графа зависимостей этапа 3, не относится к LSP |

**Правильное расположение**:
- Runtime DAG параллелизм → см. [RFC-008: Модель параллелизма во время выполнения](../design/rfc/accepted/008-runtime-concurrency-model.md)
- Граф зависимостей модулей → Этап 3 данного плана (завершён/в процессе)
- Параллельная компиляция на уровне модулей → должна реализовываться как функция системы сборки, может быть основана на графе зависимостей этапа 3

### Принципы проектирования

1. **Один проход**: Этап typecheck одновременно создаёт семантическую информацию, без повторного обхода AST
2. **Множественное использование**: LSP семантическая подсветка, инкрементная компиляция, анализ мёртвого кода используют одни и те же данные
3. **Хороший стиль**: Не добавлять ненужные абстракции ради "разделения"
