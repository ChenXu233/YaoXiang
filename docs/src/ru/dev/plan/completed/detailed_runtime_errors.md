# Детальное отслеживание ошибок во время выполнения: разделение хранения и отложенный рендеринг (Detached Storage & Deferred Loading)

> Статус: основной путь реализован (2026-03-15); DebugSection / CLI-переключатель реализован (2026-03); более полное покрытие Span остаётся в планах на будущее.

## 1. Предыстория и цели
Когда текущий интерпретатор YaoXiang выбрасывает ошибки времени выполнения (например, `Runtime error: Function not found: int.to_string`), он может вывести только имя функции и указатель инструкций (IP), например `at main (ip: 0)`. Это затрудняет для разработчиков интуитивную привязку ошибки к конкретным строкам и столбцам исходного кода.

Для обеспечения красивых и подробных сообщений об ошибках с подсветкой кода, аналогичных компилятору Rust, и при этом **абсолютно недопустимо добавлять временные затраты и потребление памяти в основном цикле интерпретации**, данный план предлагает архитектуру «разделения хранения и отложенной загрузки (Detached Storage & Deferred Loading)».

## 2. Основные принципы проектирования
1. **Нулевые затраты во время выполнения** (Zero-Cost Runtime): Логика выполнения виртуальной машины (Interpreter/Runtime) никогда не обрабатывает сложные объекты связывания с исходными файлами (такие как Span, идентификаторы файлов), сохраняя только указатель `ip` (Instruction Pointer) при возникновении исключения.
2. **Разделённое хранение в побочной таблице** (Side-Table Storage): генерируемая информация отслеживания `Span` хранится в специальной структуре `DebugSection / DebugMap`, не участвующей в горячем кэшировании кода во избежание замедления производительности чтения инструкций.
3. **Отложенный рендеринг по требованию** (Deferred Rendering): выравнивание фрагментов исходного кода и рендеринг выполняются только в тот самый момент, когда ошибка фактически происходит и всплывает до точки срабатывания (CLI/TUI). В период нормального выполнения не происходит сборки строк «на всякий случай» для возможного предотвращения ошибок.

## 3. Обзор текущей реализации (已完成)
- **DebugMap сквозной**: на этапе Codegen опционально генерируется отображение `ip -> Span`, которое передаётся в `BytecodeFunction.debug_map` во время выполнения.
- **Нулевые затраты интерпретатора**: главный цикл выполнения интерпретатора не обращается к `Span/debug_map`; ошибки времени выполнения по-прежнему полагаются только на `StackFrame { function_name, ip }` для всплытия.
- **Отложенный рендеринг на верхнем уровне**: CLI верхнего уровня перехватывает `ExecutorError`, использует `debug_map` для обратного отображения `ip` в `Span`, вызывает существующий `TextEmitter` для вывода диагностики с подсветкой фрагментов кода и добавляет stack trace.

## 4. Конкретные точки реализации (соответствующие оригинальному плану этапов)

### Этап 1: Расширение структуры данных байткода (Middle Core / Bytecode) ✅
`BytecodeFunction` уже содержит поле `debug_map: HashMap<usize, Span>` и в данной реализации обеспечивает сквозную передачу данных от Codegen до Runtime:
```rust
pub struct BytecodeFunction {
    // ... существующая логика
    /// Debug info: mapping from IP to source Span
    pub debug_map: std::collections::HashMap<usize, crate::util::span::Span>,
}
```
Дополнение: `FunctionCode` на стороне Codegen также получила поле с тем же именем, и при преобразовании `BytecodeModule::from(BytecodeFile)` оно передаётся в `BytecodeFunction`.

*Впоследствии*: запись/чтение DebugSection при сериализации/десериализации файла байткода (`src/middle/passes/codegen/bytecode.rs`) ещё не реализованы.

### Этап 2: Сбор отображений во время генерации промежуточного кода (Codegen Translation) ✅
В рабочем процессе `src/middle/passes/codegen/translator.rs`:
- Перед записью `FunctionCode.instructions.push(...)` используется `instructions.len()` в качестве генерируемого `ip`.
- `Span` извлекается из текущей IR-инструкции и записывается в `debug_map[ip] = span`.
- Управление генерацией DebugMap осуществляется через `CodegenContext::set_generate_debug_info(bool)` (при отключении каждая функция сохраняет только пустой `HashMap`, избегая дополнительных выделений памяти).

Текущие IR-инструкции с точной локализацией (имеющие `Span`):
- `Call / CallVirt / CallDyn` (运行时错误相关函数调用错误，如 `FunctionNotFound`)
- `Div / Mod` (ошибки деления, например, деление на ноль)
- `Store / StoreField / StoreIndex` (зарезервировано для большего количества типов ошибок времени выполнения в будущем)

Для поддержки вышеуказанного отображения IR-слой в `src/middle/core/ir.rs` дополняет поля `span: Span` для `Call/CallVirt/CallDyn/Div/Mod`, и заполнение происходит в `src/middle/core/ir_gen.rs` (не влияет на производительность времени выполнения интерпретатора).

### Этап 3: Упрощённое всплытие стека во время выполнения (Interpreter Execution) ✅
* Структура данных `StackFrame` в `src/backends/mod.rs` остаётся неизменной, содержа только:
  ```rust
  pub struct StackFrame {
      pub function_name: String,
      pub ip: usize,
  }
  ```
* При возникновении исключения (например: деление на ноль, `FunctionNotFound`) выбрасывается `ExecutorError`, который вместе с упрощённым `StackFrame` возвращается через стек вызовов (Err mapping).

### Этап 4: Перехват на самом верхнем уровне и отложенный рендеринг с подсветкой (CLI / Diagnostics) ✅
В `src/util/diagnostic/mod.rs` добавлена функция `render_runtime_error(...)`, которая вызывается после перехвата ошибки времени выполнения в `run_file_with_diagnostics(...)`:
1. Перехватывается окончательно выброшенный `ExecutorError`, распаковывается `stack_trace()`.
2. Получается информация о кадре, вызвавшем сбой (первый кадр), из которого извлекаются `function_name` и `ip`.
3. В текущем `BytecodeModule` находится соответствующий `BytecodeFunction`, и `debug_map.get(&ip)` используется для получения `Span`.
4. **Отложенный рендеринг**: рендеринг фрагмента кода с подсветкой через `TextEmitter` выполняется только при возникновении ошибки, после чего добавляется текст stack trace.

Примечание: текущий `Span` не содержит `file_id`; при однопроходном вводе CLI верхнего уровня считывает исходный код и создаёт `SourceFile` (необходимый для компиляции). Рендеринг ошибок времени выполнения повторно использует этот `SourceFile`, не внося обработку ввода-вывода/строк в главный цикл интерпретатора.

## 6. Известные ограничения и дальнейшая работа
1. **Покрытие Span**: на основе этапа 1 расширено до `LoadField/LoadIndex` (и включено в DebugMap); можно продолжить дополнение полей `Span` для большего количества IR-инструкций для покрытия большего числа типов ошибок времени выполнения.
2. **Многофайловые/модульные системы**: `Span` по-прежнему содержит только информацию о строках и столбцах, однако DebugMap обновлён до `ip -> (file_id + span)`, и введён `SourceMap` (`file_id -> path/content`) для кросс-файлового рендеринга; в дальнейшем можно ввести в конвейер компиляции реальную стратегию распределения `file_id` для нескольких файлов.
3. **DebugSection файла байткода**: формат `.42` расширен опциональной DebugSection (sources + ip mapping), чтение/запись реализованы; может использоваться для сохранения информации о локализации при автономном выполнении/отладке.
4. **CLI-переключатель**: добавлен `--debug-info` для `run` / `build`, используемый для управления генерацией DebugMap и встраиванием DebugSection в `.42`.
