# Приоритет интерпретатора: план реализации де耦пленного Runtime (DAG+планирование)

> **Задача**: Сначала реализовать независимый Runtime (DAG/планирование/распространение ошибок/сериализация ресурсов), чтобы интерпретатор подключился и образовал работоспособный замкнутый цикл; в дальнейшем компилятор будет «feedить» Runtime метаинформацией/ограничениями, такими как `@block/@eager/@auto`, `spawn`, `Result/?`.
> **На основе RFC**: RFC-008 (Модель параллельного выполнения Runtime и разъединение планировщика), RFC-001 (Модель 并作 и система обработки ошибок), RFC-009 (Модель собственности: граница задачи является границей утечки)
> **Дата**: 2026-03-11 (черновик), 2026-03-12 (обновление)
> **Статус**: В процессе (план)

---

## Аннотация

- Приоритетная реализация Runtime, «decoupled» от интерпретатора/компилятора: отвечает только за жизненный цикл задач, управление графом зависимостей (DAG), стратегию планирования, распространение ошибок и отмену, сериализацию конфликтов ресурсов; не понимает деталей байткода/AST/системы типов.
- Интерпретатор/компилятор только отправляет в Runtime «исполняемые единицы (задачи) + метаинформацию (зависимости/ресурсы/стратегии/Span)».
- Этап A ориентирован на замкнутый цикл интерпретатора (по умолчанию без изменения существующей семантики); Этап B позволит компилятору связать языковую семантику с Runtime.

---

## Внешние интерфейсы/изменения поведения (которые产生 данный план)

- Добавить (или зафиксировать) набор **публичных интерфейсов Runtime**: отправка задач/зависимости/ожидание/отмена/статистика, ключи сериализации ресурсов, траектория распространения ошибок/опциональный граф ошибок.
- Интерпретатор добавляет **входную точку конфигурации runtime**: выбор Embedded/Standard/Full, количество workers, включение work-stealing (можно отложить до CLI, достаточно оставить программный вход).

---

## Маппинг кода (текущий)

- Runtime core (DAG/зависимости/отмена/сериализация ресурсов): `src/backends/runtime/engine.rs`
- Runtime три уровня фасадов (Embedded/Standard/Full): `src/backends/runtime/facade.rs`
- Конфигурация runtime интерпретатора (RuntimeMode + EvalStrategy): `src/backends/interpreter/runtime.rs`
- Интерпретатор: планирование задач/драйвер/force (AsyncValue + drive_dag_until): `src/backends/interpreter/executor/executor.rs`
- Интерпретатор: инъекция стратегии в инструкции Call* (@block/@auto/@eager) и консервативные ключи ресурсов (ffi): `src/backends/interpreter/executor/execute.rs`

> Примечание: задачи интерпретатора需要 `&mut Interpreter`, поэтому текущий интерпретатор использует **Host-driven** (однопоточный драйвер) исполнение готовых задач; параллельное выполнение Full runtime в основном用于 будущего компиляторного backend и верификации runtime (соответствует многоуровневой архитектуре RFC-008 и ограничениям границ задач RFC-009).

## Этап A: Runtime + интерпретатор (сначала замкнутый цикл, потом улучшения)

### A1. Определить и зафиксировать границы Runtime (decoupled от интерпретатора/компилятора)

**Цель**
- Runtime отвечает только за: жизненный цикл задач, DAG, стратегию планирования, распространение ошибок и отмену, сериализацию ресурсов.
- Runtime не зависит от внутренних структур интерпретатора/компилятора; интерпретатор/компилятор только отправляет задачи и метаинформацию.

**Критерии приёмки**
- [x] Модуль Runtime не ссылается на внутренние модули интерпретатора и компилятора (проверяется по границам зависимостей).
- [x] Интерфейс Runtime может использоваться совместно «задачами интерпретатора» и «будущими задачами компилятора/AOT».

**Тестовые пункты**
- [x] Модульные тесты: с минимальным «поддельным исполнительным устройством» проверить, что Runtime работает независимо (без подключения интерпретатора).

---

### A2. Подключение интерпретатора через Embedded path (нулевые семантические изменения)

**Цель**
- Без введения параллелизма/ленивых вычислений, сначала让интерпретатор通过 Runtime выполняет наиболее базовый путь «немедленного выполнения», как безопасное основание.

**Критерии приёмки**
- [x] Результаты и сообщения об ошибках существующих end-to-end кейсов интерпретатора не изменяются.
- [x] Существующий набор тестов проходит.

**Тестовые пункты**
- [x] Регрессионные тесты: все интеграционные тесты интерпретатора проходят.
- [x] Дымовые тесты: вывод минимальной программы (без семантики параллелизма) согласован.

---

### A3. Runtime: Standard Runtime (однопоточное DAG-планирование + ожидание зависимостей)

**Цель**
- Реализовать однопоточное планирование: построение/поддержка DAG, продвижение по зависимостям; изолированные DAG не блокируют основную цепочку; поддержка обнаружения циклических зависимостей и возврат ошибок.

**Критерии приёмки**
- [x] Порядок выполнения линейных зависимостей/ромбовидных зависимостей/изолированных DAG соответствует описанию в RFC-001.
- [x] Циклические зависимости не вызывают deadlock, возвращают чёткую ошибку (включая сводку цепочки зависимостей, если возможно — с Span).

**Тестовые пункты**
- [x] Модульные тесты: линейные зависимости, ромбовидные зависимости, изолированные DAG, циклические зависимости (2-узловой цикл, самозамыкание).
- [x] Модульные тесты: отмена и очистка (после отмены продвижение прекращается, можно проверить освобождение ресурсов).

---

### A4. Runtime: Full Runtime (многопоточное планирование + опциональное work-stealing)

**Цель**
- Добавить параллельное планирование с workers>1; work-stealing как опциональное улучшение, при отключении также работает нормально (основное требование RFC-008).

**Критерии приёмки**
- [x] workers=1 соответствует поведению Standard (идентичные результаты и ошибки).
- [x] workers>1 позволяет наблюдать параллельное выполнение (приёмка не по производительности, а по наблюдаемому параллелизму).
- [x] work-stealing вкл/выкл не влияет на корректность.

**Тестовые пункты**
- [x] Модульные/интеграционные тесты: один и тот же граф задач при workers=1 и workers>1 даёт одинаковые результаты.
- [x] Модульные тесты: матрица вкл/выкл work-stealing без deadlock и без потери задач.

---

### A5. Интерпретатор: ленивые значения и «force по требованию» (поддержка @auto/@eager/@block)

**Цель**
- Поддержка «сначала создать отложенно вычисляемый placeholder, при необходимости использовать значение — вызвать вычисление».
- Три стратегии конфигурации в runtime:
  - По умолчанию: сохранить текущее синхронное поведение (эквивалент глобального @block).
  - Опционально: глобальный @auto (максимальная задержка и параллелизм) / глобальный @eager (более агрессивное вычисление).

**Критерии приёмки**
- [x] @auto: вызов возвращает placeholder `Async`; когда значение потребляется, Runtime доводит до завершения и получает значение.
- [x] @block: поведение по умолчанию соответствует текущему (не идёт через DAG; не создаёт placeholder `Async`).
- [x] @eager: в точке вызова немедленно доводит до завершения и получает значение (не оставляет placeholder `Async`).

**Тестовые пункты**
- [x] Модульные тесты: при @auto lazy call при потреблении значения вызывает выполнение и получает значение.
- [x] Модульные тесты: при @eager не остаётся placeholder `Async`.
- [x] Регрессионные тесты: без включения новых стратегий все существующие тесты проходят.

---

### A6. Runtime + интерпретатор: распространение, слияние и отмена при сбоях задач

**Цель**
- Сбой задачи распространяется по рёбрам зависимостей: зависимые задачи также терпят сбой или отменяются; при одновременных сбоях нескольких задач выводится основная ошибка и сводка побочных ошибок.

**Критерии приёмки**
- [x] При сбое зависимости upstream не ждёт вечно; ошибка может достичь верхнего уровня (причина отмены включает сводку зависимых ошибок).
- [x] Когда upstream только ожидает определённое значение, Runtime в первую очередь продвигает closure зависимости, изолированные задачи не задерживают завершение цели (прекращают продвижение; в дальнейшем можно добавить опциональную стратегию явной отмены).

**Тестовые пункты**
- [x] Модульные тесты: сбой листа приводит к сбою/отмене upstream; сообщение об ошибке включает сводку корневой причины.
- [x] Модульные тесты: при одновременном сбое нескольких зависимостей можно合并输出 основной ошибки和其他 сводки (причина отмены агрегируется).
- [x] Модульные тесты: после отмены отменённые задачи больше не продвигаются (включая распространение отмены по зависимостям).
- [x] Модульные тесты: `drive_until(Some(target))` не выполняет несвязанные изолированные задачи (приоритет цепочки цели).

---

### A7. Минимальный замкнутый цикл для конфликтов ресурсов (сначала консервативно, потом детально)

**Цель**
- Интерпретатор сначала использует «консервативные ключи ресурсов» для标记副作用 операций (например, объединяются в одну категорию ресурсов), Runtime принудительно сериализует для обеспечения корректности; в дальнейшем компилятор предоставит более детальные ключи ресурсов для повышения параллелизма (система типов ресурсов RFC-001).

**Критерии приёмки**
- [x] При workers>1 операции с одинаковым ключом ресурса строго сериализованы (стабильная наблюдаемая очерёдность).
- [x] Операции с разными ключами ресурсов允许параллельное выполнение (минимум наблюдаемое параллельное продвижение).

**Тестовые пункты**
- [x] Модульные тесты: в Full runtime задачи с одинаковым ключом ресурса не запускаются параллельно.
- [x] Модульные тесты: при workers>1 задачи без ресурсов/с разными ключами могут запускаться параллельно.

---

### A8. Бесконечные циклы и справедливость (срезы/yield)

**Цель**
- Реализация RFC-001: одиночные циклы без накладных расходов планирования, многократные циклы автоматически разрезаются; избежать starvation; при одной задаче нет наблюдаемых накладных расходов.

**Критерии приёмки**
- [x] В сценарии с несколькими задачами одна длинная задача не приводит к тому, что другие задачи никогда не выполнятся (cooperative slicing: задачи могут `yield` для передачи управления).
- [x] В сценарии с одной задачей возможно завершение без slicing (исполнитель при отсутствии конкуренции не делает `yield`, чтобы избежать наблюдаемых накладных расходов).

**Тестовые пункты**
- [x] Модульные тесты: две задачи с длинными циклами обе продвигаются за ограниченное количество шагов (счётчик/ротационная последовательность для приёмки).
- [x] Модульные тесты: одна задача может завершиться за один poll (без накладных расходов slice-планирования).

## Этап B: Подключение компилятора (в дальнейшем «feedить» язык семантики в Runtime)

### B1. Синтаксис и ограничения: `@block/@eager/@auto` и `spawn` (только внутри @block)

**Цель**
- Компилятор предоставляет метаинформацию о стратегии на уровне функций/блоков; принудительно `spawn`只能用在 `@block` области видимости (RFC-001/008).

**Критерии приёмки**
- [x] Недопустимое положение `spawn`报错在编译期且定位明确。
- [x] Аннотации стратегии могут стабильно передаваться в конфигурацию runtime/метаинформацию.

**Тестовые пункты**
- [x] Тесты парсинга/проверки типов: матрица допустимых/недопустимых сценариев `spawn`.
- [x] End-to-end тесты: аннотации влияют на выбор стратегии runtime (наблюдаемые различия + согласованные результаты).

---

### B2. `Result[T,E]` и `?` языковой замкнутый цикл (согласован с распространением ошибок Runtime)

**Цель**
- `Result`真正содержит `T/E`, `?` 在编译期保证规则正确; runtime 按 RFC-001 прозрачно распространяет; согласовано с распространением ошибок параллельных зависимостей.

**Критерии приёмки**
- [x] 非 `Result` 使用 `?` 或错误类型不匹配时报错在编译期。
- [x] 在并发场景下 `?` 的短路规则与资源串行化规则不冲突（可解释且可测试）。

**Тестовые пункты**
- [x] Типовые тесты: правильные/неправильные用例 `Result/?`.
- [x] End-to-end тесты: параллельные задачи触发 `?` 传播到顶层。
- [x] Модульные тесты: ресурс串行化为 control edge (仅排序不传播错误); 取消等待 control deps 以保持资源顺序 (见 `src/backends/runtime/engine.rs` соответствующие тесты).

## 默认假设/选择 (可调整)

- Этап A 默认不改变现有щую семантику пользователя: интерпретатор по умолчанию эквивалентен глобальному `@block` (Embedded path), Standard/Full и `@auto/@eager` включаются через явную конфигурацию.
- Система ресурсов на Этапе A сначала консервативно обеспечивает корректность, на Этапе B компилятор предоставит детальные ключи ресурсов для повышения параллелизма.
