# Проверка реализации правил конструктора типов

> Цель проверки: убедиться, что компилятор корректно реализует правило «есть `: Type` — конструктор типа, иначе — функция/переменная»
>
> Дата создания: 2026-03-21

---

## Описание функциональности

### Базовое правило

| Запись | Поведение компилятора |
|--------|-----------------------|
| `x: Type = {...}` | **Конструктор типа** (`: Type` указывает, что возвращается Type) |
| `f = {...}` | **Функция/переменная** (`{...}` интерпретируется как блок кода) |

**Ключевой принцип**: пропуск `: Type` никогда не интерпретируется как тип, `{...}` всегда интерпретируется как блок кода.

### Сценарии неоднозначности

| Форма | С `: Type` | Без `: Type` |
|-------|------------|--------------|
| `{ x: Float }` | ✅ Литерал типа | ✅ Блок кода (выражение, бессмысленное) |
| `{ a = 1 + 1 }` | ✅ Тип, значение поля по умолчанию 2 | ✅ Блок кода (выражение присваивания, бессмысленное) |
| `{}` | ✅ Пустой тип записи | ✅ Пустой блок кода `() -> Void` |

**Ключевые моменты**:
- Есть `: Type` → `{ ... }` интерпретируется как литерал типа
- Нет `: Type` → `{ ... }` интерпретируется как блок кода (даже в позиции типа — это блок кода!)

## Пункты проверки

### Проверка 1: Конструктор типа должен в итоге возвращать `Type`

**Ключевое правило**: не «если есть `Type`, то это тип», а **итоговый возвращаемый тип должен быть `Type`**, чтобы считаться конструктором типа.

> **Компиляционная определённость**: Всё, связанное с Type, **определяется на этапе компиляции**. Даже в обобщённых функциях T передаётся не в runtime, а **мономорфизируется на этапе компиляции**.

| Сигнатура | Возвращаемый тип | Значение |
|-----------|------------------|----------|
| `(T: Type) -> Type` | `Type` | ✅ **Конструктор типа** (compile-time) |
| `(T: Type) -> (() -> Void)` | `() -> Void` | ❌ **Обобщённая функция** (compile-time monomorphism) |
| `(T: Type) -> (T -> Void)` | `T -> Void` | ❌ **Обобщённая функция** (compile-time monomorphism) |

**Тестовый код**:
```yaoxiang
// ✅ Конструктор типа: в итоге возвращает Type
Point: Type = { x: Float, y: Float }
List: (T: Type) -> Type = { data: Array[T] }
Option: (T: Type) -> Type = { some: (T) -> Self, none: () -> Self }

// ❌ Обобщённая функция: в итоге возвращает функциональный тип, не Type
maybe: (T: Type) -> (() -> Void) = (T) => { println("test") }
factory: (T: Type) -> (T -> Void) = (T) => { (x) => println(x) }
```

**Пункты проверки**:
- [ ] `Point: Type = { ... }` компилируется?
- [ ] `List: (T: Type) -> Type = { ... }` компилируется?
- [ ] `Option: (T: Type) -> Type = { ... }` компилируется?

**Ожидаемый результат**: полная компиляция

---

### Проверка 2: Пропуск `: Type` приводит к выведению типа функции

**Ключевое правило**: при наличии входных параметров тип должен быть указан либо в **аннотации типа**, либо в **заголовке lambda**, или в обоих местах.

```yaoxiang
// ✅ Правильно: тип указан в позиции аннотации
add: (a: Int, b: Int) -> Int = (a, b) => a + b

// ✅ Правильно: тип указан в позиции заголовка lambda
add = (a: Int, b: Int) => a + b

// ✅ Правильно: тип указан в обоих местах
add: (a: Int, b: Int) -> Int = (a: Int, b: Int) => a + b

// ✅ Правильно: пропуск заголовка lambda (сигнатура полная)
add: (a: Int, b: Int) -> Int = a + b

// ❌ Ошибка: тип не указан нигде
add = (a, b) => a + b

// ✅ Правильно: пустой блок без типов параметров
main = { println("Hello") }
```

**Пункты проверки**:
- [ ] `main = { println("Hello") }` → тип `() -> Void`?
- [ ] `add = (a: Int, b: Int) => a + b` → тип `(a: Int, b: Int) -> Int`?
- [ ] `add: (a: Int, b: Int) -> Int = a + b` → пропущен заголовок lambda, тип `(a: Int, b: Int) -> Int`?
- [ ] `add = (a, b) => a + b` → ❌ Ошибка компиляции? Тип не указан нигде

---

### Проверка 3: Тип пустого блока `{}`

**Тестовый код**:
```yaoxiang
// ✅ Правильно: пустой блок {} выводится как () -> Void
empty_func = {}
void_func = { println("test") }

// ❌ Ошибка: ожидается Int, но получен Void
bad: Int = {}
```

**Пункты проверки**:
- [ ] `empty_func = {}` компилируется? Тип `() -> Void`?
- [ ] `void_func = { println("test") }` компилируется? Тип `() -> Void`?
- [ ] `bad: Int = {}` выдаёт ошибку? Сообщение об ошибке понятное?

**Ожидаемый результат**:
- Первые два компилируются
- Третий выдаёт ошибку: несоответствие типов (Void vs Int)

---

### Проверка 4: Ошибочные примеры: пропуск `: Type` в определении типа

**Тестовый код**:
```yaoxiang
// ❌ Никогда не тип: нет : Type, {...} интерпретируется как блок кода
Point = { x: Float, y: Float }  // Это функция/переменная, не тип!
```

**Пункты проверки**:
- [ ] `Point = { x: Float, y: Float }` во что компилируется? Функция или тип?
- [ ] Считает ли компилятор это функциональным типом `() -> Type` (а не типом записи)?

**Ожидаемый результат**:
- Если `{ x: Float, y: Float }` интерпретируется как блок кода, то `Point` — функция с типом `() -> ???` (возможно ошибка, т.к. блок не содержит валидного выражения)
- **Суть**: пропуск `: Type` никогда не является определением типа

## Тесты граничных условий

### Граница 1: Конструктор типа и функция могут использовать `-> Type`

```yaoxiang
// Это конструктор типа
Id: (T: Type) -> Type = { x: T }

// Это функция, возвращающая значение типа Type
id_func: (T: Type) -> Type = (T) => { x: T }
```

**Пункты проверки**:
- [ ] `Id: (T: Type) -> Type = { x: T }` — конструктор типа
- [ ] `id_func: (T: Type) -> Type = (T) => { x: T }` — функция

---

### Граница 2: Обобщённые без `: Type` — это функции, не типы

```yaoxiang
// ✅ Конструктор типа: в итоге возвращает Type
List: (T: Type) -> Type = { data: Array[T] }

// ✅ Обобщённая функция: в итоге возвращает функциональный тип, не Type
List = (T: Type) => { data: Array[T] }  // Это функция, не тип!
```

**Пункты проверки**:
- [ ] `List = (T: Type) => { data: Array[T] }` компилируется как функция?
- [ ] Какой тип у этой функции?

---

### Граница 3: Определение интерфейсов

```yaoxiang
// ✅ Правильно
Clone: Type = { clone: (Self) -> Self }
Drawable: Type = { draw: (Surface) -> Void }
```

**Пункты проверки**:
- [ ] Должно ли определение интерфейса использовать `: Type`?
- [ ] Ошибка ли отсутствие `: Type`?

## Внутренняя проверка компилятора

### Уровень парсера

Проверка поля `btype` узла AST `Binding`:

```rust
enum BindingType {
    TypeConstructor,    // Возвращает Type
    Function,          // Возвращает другой тип (функция/переменная)
}
```

**Внимание**: различие делается не по «есть ли `: Type`», а по **итоговому возвращаемому типу**.

**Пункты проверки**:
- [ ] Парсер устанавливает `btype` по возвращаемому типу?
- [ ] Binding, возвращающий `Type`, помечен как `TypeConstructor`?
- [ ] Binding, возвращающий другой тип, помечен как `Function`?

### Уровень проверки типов

**Пункты проверки**:
- [ ] Проверка типов различает обработку по `btype`?
- [ ] Для `TypeConstructor` `{ ... }` интерпретируется как литерал типа?
- [ ] Для `Function` `{ ... }` интерпретируется как блок кода?

## Проверка сообщений об ошибках

### Ошибочные примеры

**Тест**:
```yaoxiang
Point = { x: Float, y: Float }
```

**Фактическое поведение**:
- Компилятор **не** выдаёт ошибку «отсутствует `: Type`»
- Компилятор пытается интерпретировать `{ x: Float, y: Float }` как блок кода
- Если парсинг успешен, `Point` будет функцией с типом `() -> Void // или тип возвращаемого значения`

**Пункты проверки**:
- [ ] `Point = { x: Float, y: Float }` во что компилируется? Какой тип?
- [ ] Компилятор корректно интерпретирует `{ ... }` как блок кода, а не как литерал типа?

## Резюме

**Базовые правила**:

| Запись | Возвращаемый тип | Значение |
|--------|------------------|----------|
| `x: Type = {...}` | `Type` | ✅ Конструктор типа |
| `f = {...}` | `Void` | ✅ Функция/переменная (`{...}` — блок кода) |
| `f: (T: Type) -> Type = ...` | `Type` | ✅ Конструктор типа |
| `f: (T: Type) -> (() -> Void) = ...` | `() -> Void` | ❌ Обобщённая функция (не тип) |

**Ключевые различия**:
- Не «если есть `Type`, то конструктор типа»
- А «**в итоге возвращает `Type`** — конструктор типа»
- `(T: Type) -> (() -> Void)` возвращает функциональный тип — это обобщённая функция

**Компиляционная определённость**:
> Всё, связанное с Type, **определяется на этапе компиляции**. Даже в обобщённых функциях T **не передаётся в runtime**, а **мономорфизируется на этапе компиляции**.

**Пункты реализации**:

1. **Парсер**: есть `: Type` → помечать как конструктор типа; нет `: Type` → помечать как функцию/переменную
2. **Проверка типов**:
   - Есть `: Type` → `{ ... }` интерпретировать как литерал типа
   - Нет `: Type` → `{ ... }` интерпретировать как блок кода
3. **HM-выведение**: корректно выводить типы сокращённых форм функций
4. **Обработка ошибок**: когда ожидается тип, но получена функция — сообщение об ошибке понятное
