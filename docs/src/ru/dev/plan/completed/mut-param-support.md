# План реализации поддержки синтаксиса `mut` для параметров функций

> **Статус**: Реализовано
> **Дата**: 2026-02-19

---

## Обзор

### Контекст проблемы

В текущей версии языка YaoXiang параметры функций не поддерживают ключевое слово `mut`, параметры по умолчанию являются неизменяемыми. Это приводит к следующим проблемам:

1. **Неизменяемые параметры замыканий**: Например, в `list.map([1,2,3], x => x * 2)` параметр замыкания `x` является неизменяемым, и его нельзя изменить в теле замыкания
2. **Неизменяемые параметры функций**: Обычные параметры функций также нельзя изменять, что делает невозможным реализацию паттерна «модификация параметра на месте»

### Цель

Реализовать поддержку синтаксиса `mut` для параметров функций, чтобы параметры функций можно было объявлять изменяемыми.

### Дизайн синтаксиса

```yaoxiang
// Обычные параметры функций
fn foo(mut x: Int) -> Int {
    x = x + 1  // Допустимо, можно изменять
    x
}

// Параметры Lambda
f = (mut x) => x + 1

// Вызовы высшего порядка
list.map([1, 2, 3], (mut x) => x * 2)  // Допустимо
```

## Дизайн кодов ошибок

### Различие E2010 vs E2011

| Код ошибки | Описание | Ситуация |
|------------|----------|----------|
| E2010 | cannot assign to immutable variable | `x = 1; x = 2` (явное изменение переменной пользователем) |
| **E2011** | **closure parameter requires mut** | `list.map([..., x => ...])` (параметр замыкания требует mut) |

### Дизайн E2011

**Сообщение об ошибке**:
```
error[E2011]: closure parameter '{param_name}' requires 'mut' to be modified
 --> example.yx:1:20
  |
1 | list.map([1,2,3], x => x * 2);
  |                    ^ consider adding 'mut' to parameter: (mut x) => ...
```

**Условия срабатывания**:
1. Замыкание передаётся как параметр в функцию высшего порядка
2. Функция высшего порядка внутри пытается изменить параметр замыкания
3. Параметр замыкания не объявлен как `mut`

**Подсказка по исправлению**:
- Рекомендуется изменить `x => ...` на `(mut x) => ...`

### Phase 2: Модификация парсера

#### 2.1 Модификация логики解析 параметров

**Файл**: `src/frontend/core/parser/statements/declarations.rs`

**Функция**: `parse_fn_params`

Обнаружение ключевого слова `mut` перед解析 имени параметра:

```rust
// Обнаружение ключевого слова mut
let is_mut = state.skip(&TokenKind::KwMut);

let name = match state.current().map(|t| &t.kind) {
    Some(TokenKind::Identifier(n)) => n.clone(),
    _ => break,
};
state.bump();

// Разбор аннотации типа
let ty = if state.skip(&TokenKind::Colon) {
    parse_type_annotation(state)
} else {
    None
};

params.push(Param {
    name,
    ty,
    is_mut,  // Новый
    span: param_span,
});
```

**Приёмка**:
- [x] `(mut x: Int)` корректно解析
- [x] `(mut x)` корректно解析 (без аннотации типа)
- [x] `(x: Int)`解析 как неизменяемый (is_mut = false)

### Phase 4: Генерация IR

#### 4.1 Модификация generate_function_ir

**Файл**: `src/middle/core/ir_gen.rs`

Модификация логики регистрации параметров на основе `is_mut`:

```rust
for (i, param) in params.iter().enumerate() {
    // Регистрация параметра
    self.register_local(&param.name, i);
    // Только параметры mut регистрируются как изменяемые
    if param.is_mut {
        self.current_mut_locals.insert(i);
    }
}
```

**Файл**: `src/middle/core/ir_gen.rs`

Аналогичная модификация обработки параметров замыканий:

```rust
for (i, param) in params.iter().enumerate() {

```

**Приёмка**:
- [x] Изменяемые параметры корректно помечены как изменяемые в IR

## Тестовые случаи

### Сценарии прохождения

```yaoxiang
// 1. Изменяемые параметры обычной функции
fn increment(mut x: Int) -> Int {
    x = x + 1
}
main = {
    result = increment(5);
}

// 2. Изменяемые параметры Lambda
main = {
    f = (mut x) => {
        x = x + 1
        x
    };
    result = f(5);
}

// 3. Использование замыкания с изменяемыми параметрами в функции высшего порядка
use std.{io, list}
main = {
    result = list.map([1, 2, 3], (mut x) => {
        x = x * 2
        x
    });
}
```

### Сценарии ошибок

#### E2010 - Изменение обычной неизменяемой переменной

```yaoxiang
// Изменение неизменяемого параметра - E2010
fn foo(x: Int) -> Int {
    x = x + 1  // E2010: cannot assign to immutable variable
}
```

#### E2011 - Параметр замыкания требует mut (новый)

```yaoxiang
// Параметр замыкания не объявлен как mut - E2011
list.map([1, 2, 3], x => x * 2)
// E2011: closure parameter 'x' requires 'mut' to be modified
// help: consider adding 'mut' to parameter: (mut x) => ...

list.filter([1, 2, 3], x => x > 2)
// E2011: closure parameter 'x' requires 'mut' to be modified

list.reduce([1, 2, 3], (acc, x) => acc + x, 0)
// E2011: closure parameter 'acc' requires 'mut' to be modified
// E2011: closure parameter 'x' requires 'mut' to be modified
```

## Риски и примечания

1. **Обратная совместимость**: Существующий код без параметра `mut` сохраняет неизменяемое поведение
2. **Вывод типов**: При опущенной аннотации типа `(mut x)` должен автоматически выводиться тип
3. **Сценарии замыканий**: Обеспечить корректную передачу изменяемых параметров в тело функции замыкания

## Дополнительные проблемы, обнаруженные и исправленные в процессе реализации

### 1. Изоляция состояния при генерации IR для Lambda

**Проблема**: `generate_lambda_body_ir` при генерации тела функции замыкания очищает `current_mut_locals` и `current_local_names`, что приводит к потере информации о изменяемости из родительской функции.

**Исправление**: Перед входом в генерацию IR тела lambda сохранить состояние родительской функции (`current_mut_locals`, `current_local_names`, `next_temp`), после выхода восстановить.

### 2. Конфликт регистров возвращаемого значения Lambda

**Проблема**: `generate_lambda_body_ir` использует фиксированный регистр 0 как регистр возвращаемого значения, что конфликтует с регистром параметра 0, вызывая ложное срабатывание MutChecker.

**Исправление**: Изменено на использование `self.next_temp_reg()` для выделения отдельного регистра возвращаемого значения.

### 3. Изменяемость StoreIndex для списковых литералов

**Проблема**: Списковый литерал `[1, 2, 3]` записывает элементы через множественные `StoreIndex`, и вторая и последующие записи воспринимаются MutChecker как изменение неизменяемой переменной.

**Исправление**: После `AllocArray` временный регистр списка регистрируется как изменяемый (`current_mut_locals.insert(result_reg)`).

### 4. Распознавание параметров `mut` в аннотациях типов функций

**Проблема**: При разборе аннотаций типов функций RFC-010 для `(mut x: Int) -> Int`, функция `looks_like_named_params` не может распознать список параметров, начинающийся с `KwMut`, что приводит к неверной идентификации как старый синтаксис.

**Исправление**: Добавлена проверка `state.at(&TokenKind::KwMut)` в двух местах функции `looks_like_named_params`.
