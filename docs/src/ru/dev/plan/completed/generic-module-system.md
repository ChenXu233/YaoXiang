# YaoXiang План рефакторинга универсальной модульной системы

## 1. Обзор

Текущая модульная система YaoXiang представляет собой специальную реализацию для встроенных модулей `std`, содержащую многочисленные жёсткие коды и не поддерживающую пользовательские модули. В данном документе описано, как реализовать универсальную модульную систему.

## 2. Критерии приёмки

### 2.1 Обязательные функции

| Функция | Пример синтаксиса | Описание |
|---------|-------------------|----------|
| Импорт модуля | `use my_module` | Загрузка модуля из файловой системы |
| Выборочный импорт | `use my_module.{func1, func2}` | Импорт только указанных функций |
| Псевдоним модуля | `use my_module as m` | Доступ через псевдоним |
| Вызов модуля | `my_module.func()` | Вызов функции через модуль |
| Подмодули | `my_module.sub.func()` | Доступ к подмодулю |
| Атрибуты модуля | `my_module.CONST` | Доступ к константам модуля |

### 2.2 Совместимость с модулем std

После рефакторинга должно быть сохранено прежнее поведение:

| Синтаксис | Пример | Статус |
|-----------|--------|--------|
| `std.io.print()` | `use std.io` + `std.io.println("x")` | ✅ Должно быть сохранено |
| `print()` (выборочный импорт) | `use std.io.{print}` + `print("x")` | ✅ Должно быть сохранено |
| `io.print()` (импорт модуля) | `use std.{io}` + `io.println("x")` | ✅ Должно быть сохранено |
| `std.math.PI` | Доступ к константе | ✅ Должно быть сохранено |

### 2.3 Правила загрузки модулей

```
// Пути поиска модулей (относительно текущего файла)
1. ./my_module.yx           // Текущий каталог
2. ./my_module/mod.yx       // Подкаталог
3. ./my_module/index.yx     // index-файл

// Специальная обработка модуля std
// std модуль расположен в src/std/ и всегда доступен
```

## 3. Требования к тестированию

### 3.1 Функциональное тестирование

```yaoxiang
// test_user_module.yx
// Предполагается наличие файла my_module.yx

// 1. Базовый импорт модуля
use my_module
main = {
    my_module.greet()  // Вызов функции модуля
}

// 2. Выборочный импорт
use my_module.{add, sub}
main = {
    add(1, 2)
}

// 3. Псевдоним модуля
use my_module as m
main = {
    m.greet()
}

// 4. Подмодули
use my_module.utils
main = {
    my_module.utils.help()
}

// 5. Вложенные импорты
use my_module.{sub1, sub2.sub3}
```

### 3.2 Тестирование совместимости std

Все существующие тестовые случаи должны продолжать работать:

```yaoxiang
// Тестирование существующего синтаксиса
use std.io
main = { std.io.println("test") }

use std.io.{print}
main = { print("test") }

use std.{io}
main = { io.println("test") }

use std.math
main = { std.math.PI }
```

### 3.3 Тестирование граничных случаев

- Пустой импорт модуля
- Циклические зависимости модулей (должна выдаваться ошибка)
- Несуществующий модуль (должна выдаваться ошибка)
- Повторный импорт
- Конфликт импортов

### 4.1 Расположение code smells

| Файл | Номер строки | Проблема |
|------|--------------|----------|
| `src/std/mod.rs` | 29-100 | Жёсткий код `get_module_exports`, обрабатывающий только модуль std |
| `src/frontend/typecheck/mod.rs` | 593-622 | Жёсткий код обработки модуля std |
| `src/frontend/typecheck/inference/expressions.rs` | 515-598 | Жёсткий код списка `io\|math\|net\|concurrent` |
| `src/middle/core/ir_gen.rs` | 65-131 | Жёсткий код обработки пространства имён |

### 4.2 Анализ ключевых структур

#### 4.2.1 ModuleExport (src/std/mod.rs)

```rust
pub struct ModuleExport {
    pub short_name: &'static str,      // Короткое имя
    pub qualified_name: &'static str, // Полный путь
    pub signature: &'static str,      // Сигнатура функции
}
```

**Проблема**: Используется только для std модуля, требуется обобщение для универсального экспорта модуля

#### 4.2.2 NativeDeclaration (src/std/io.rs и др.)

```rust
pub struct NativeDeclaration {
    pub name: &'static str,
    pub native_name: &'static str,
    pub signature: &'static str,
    pub doc: &'static str,
    pub implemented: bool,
}
```

**Пояснение**: `NativeDeclaration` разработан для FFI (Foreign Function Interface) и предназначен специально для вызова YaoXiang функций Rust. Пользователи реализуют взаимодействие с Rust через функцию `native` в std.ffi.

**Не требует изменений**: Пользовательские модули пишутся на YaoXiang, для них не нужна структура NativeDeclaration. Модульная система просто загружает исходные файлы YaoXiang.

**Разделение дизайна**:
| Тип модуля | Способ реализации | Способ загрузки |
|------------|-------------------|-----------------|
| FFI модуль | Rust + NativeDeclaration | Встроенная регистрация |
| Пользовательский модуль | Исходные файлы YaoXiang | Загрузка из файловой системы |

#### 4.2.3 StmtKind::Use (AST)

```rust
StmtKind::Use {
    path: String,           // Путь модуля
    items: Option<Vec<String>>, // Импортируемые элементы
    alias: Option<String>,     // Псевдоним
}
```

**Текущее состояние**: Парсер уже поддерживает полный синтаксис, проверка типов реализована не полностью

**Возможные изменения в файлах**:
- Новый `src/frontend/module.rs` - Основной интерфейс модульной системы

**Содержание дизайна**:
```rust
// Элемент экспорта модуля
pub struct ModuleExport {
    pub name: String,           // Имя экспорта
    pub full_path: String,     // Полный путь
    pub kind: ExportKind,      // Функция/константа/подмодуль
```

## 7. Обработка циклических зависимостей

### 7.1 Дизайнерское решение: активная генерация ошибки

Модульная система использует стратегию **активной генерации ошибки** — при обнаружении циклической зависимости компиляция немедленно прекращается с выводом понятного сообщения об ошибке.

**Обоснование**:
- Ясность: пользователь сразу знает о проблеме
- Полезность: сообщение об ошибке показывает путь цикла
- Соответствие интуиции: аналогично подходу в Rust/Go и других языках

### 7.2 Алгоритм обнаружения

```rust
// 1. Построение графа зависимостей
//    Каждый модуль -> список зависимых модулей

// 2. Топологическая сортировка (алгоритм Кана)
//    - Узлы с нулевой степенью входа приоритетны
//    - Невозможность сортировки = наличие цикла

// 3. При неудаче сортировки — сообщение о циклической зависимости
```

### 7.3 Типы циклов

#### Прямой цикл
```yaoxiang
// a.yx
use b

// b.yx
use a  // Error: циклическая зависимость a <-> b
```

#### Косвенный цикл
```yaoxiang
// a.yx
use b

// b.yx
use c

// c.yx
use a  // Error: косвенный цикл a -> b -> c -> a
```

#### Самореференция
```yaoxiang
// a.yx
use a  // Error: самореференция
```

### 7.4 Примеры сообщений об ошибках

```yaoxiang
error[E1001]: detected cyclic dependency
  --> a.yx:1:1
   |
1  | use b
   | ^^^^^^
   |
note: dependency path: a -> b -> c -> a
help: break the cycle by removing one of these imports
```

### 7.5 Место реализации

- **Загрузчик модулей** (`src/frontend/module/loader.rs`)
- Обнаружение сразу после построения графа модулей
- Раннее обнаружение во время компиляции, во избежание напрасной работы

## 9. Стратегия кэширования модулей

### 9.1 Цели дизайна

Со ссылкой на структуру проекта RFC-014, поддержка hot reload при разработке:

```
my-project/
├── yaoxiang.toml
├── yaoxiang.lock
├── src/
│   └── main.yx
└── .yaoxiang/
    └── vendor/
        ├── foo-1.2.3/
        │   └── src/
        │       └── foo.yx
        └── bar-0.5.0/
```

### 9.2 Типы кэша

| Тип кэша | Время | Стратегия |
|----------|-------|-----------|
| **Компиляционный кэш** | Во время компиляции | Кэш в памяти, повторное использование в рамках единицы компиляции |
| **Разработочный кэш** | Время выполнения `yaoxiang run` | Файловый кэш, автоматическая перезагрузка при изменении файлов |
| **Публикационный кэш** | Сборка для релиза | Зафиксированные версии, без отслеживания изменений файлов |

### 9.3 Механизм Hot Reload

```rust
// Режим разработки: наблюдение за файлами
struct HotReloader {
    watcher: notify::Watcher,  // Наблюдение за файловой системой
}

// Условия срабатывания:
// 1. Изменение зависимого .yx файла
// 2. Изменение yaoxiang.toml или yaoxiang.lock

// Стратегия перезагрузки:
// - Инкрементная перекомпиляция изменившихся модулей
// - Перестроение графа зависимостей
// - Обнаружение циклических зависимостей
```

### 9.4 Место реализации

- **Кэш модулей**: `src/frontend/module/cache.rs`
- **Hot reload**: `src/frontend/module/hot_reload.rs`
- **Точки интеграции**: `src/frontend/compiler.rs`

## 11. Правила разрешения путей модулей

### 11.1 Основа дизайна

Со ссылкой на порядок разрешения модулей в RFC-014, строки 125-135.

### 11.2 Синтаксис путей

| Синтаксис | Пример | Описание |
|-----------|--------|----------|
| Стандартный путь | `use foo.bar` | Поиск в vendor/ и src/ |
| Подмодуль | `use foo.bar.baz` | Вложенный модуль |
| Текущий модуль | `use .` | Текущий модуль (self) |
| Родительский модуль | `use ..` | Родительский модуль (parent) |

**Не поддерживается**:
- Относительные пути (`use ./utils`)
- Абсолютные пути (`use /usr/local/lib`)
- Строковые имена пакетов (`use "@org/package"`)

### 11.3 Порядок поиска

```
use foo.bar.baz;

Порядок поиска:
1. ./.yaoxiang/vendor/*/src/foo/bar/baz.yx  (каталог vendor/)
2. ./src/foo/bar/baz.yx                      (локальные модули)
3. $YXPATH/foo/bar/baz.yx                    (глобальный путь, зарезервировано)
4. $YXLIB/std/foo/bar/baz.yx                 (стандартная библиотека)
```

### 11.4 Специальная обработка модуля std

```
use std.io          ->  отображение на $YXLIB/std/io/
use std.math        ->  отображение на $YXLIB/std/math/
```

### 11.5 Место реализации

- **Резолвер модулей**: `src/frontend/module/resolver.rs`
- **Поиск путей**: `src/frontend/module/loader.rs`

## 12. Дизайн унифицированного интерфейса

### 12.1 Цели дизайна

Модули std и пользовательские модули используют унифицированный модульный интерфейс для удобства расширения.

### 12.2 Базовые интерфейсы

```rust
// Модуль
pub trait Module {
    fn path(&self) -> &str;
    fn exports(&self) -> &HashMap<String, Export>;
}

// Реестр модулей
pub trait ModuleRegistry {
    fn get(&self, path: &str) -> Option<Box<dyn Module>>;
    fn register(&mut self, path: String, module: Box<dyn Module>);
}

// Загрузчик модулей
pub trait ModuleLoader {
    fn load(&self, path: &str) -> Result<Box<dyn Module>, ModuleError>;
}
```

### 12.3 Конкретные реализации

| Реализация | Назначение |
|------------|------------|
| `StdModule` | Стандартная библиотека std (встроенная) |
| `UserModule` | Пользовательские модули (загрузка файлов) |
| `VendorModule` | Зависимости из .yaoxiang/vendor/ |
| `CompositeRegistry` | Объединение нескольких реестров с порядком поиска |

### 12.4 Композитный реестр

```rust
struct CompositeRegistry {
    // Порядок поиска: первый имеет приоритет
    std: StdModule,
    vendor: VendorRegistry,
    user: FileModuleRegistry,
}

impl ModuleRegistry for CompositeRegistry {
    fn get(&self, path: &str) -> Option<Box<dyn Module>> {
        // Попытка каждого реестра по порядку
        self.std.get(path)
            .or_else(|| self.vendor.get(path))
            .or_else(|| self.user.get(path))
    }
}
```

### 12.5 Место реализации

- **Определение интерфейсов**: `src/frontend/module/mod.rs`
- **Реализация std**: `src/frontend/module/std.rs`
- **Загрузка файлов**: `src/frontend/module/file.rs`
- **Компоновщик**: `src/frontend/module/registry.rs`
