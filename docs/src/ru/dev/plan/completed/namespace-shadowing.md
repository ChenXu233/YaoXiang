## Требования (напоминание)

Согласно проекту, правила затенения переменных языка следующие:

1. **Запрет затенения** — любая попытка повторного объявления имени переменной, уже существующей во внешней области видимости, во внутренней области вызывает ошибку
2. **Без ключевого слова let** — переменные создаются через `mut` или неявное объявление
3. **Затенение — ошибка** — как обычные объявления, так и объявления `mut` вызывают ошибку при затенении
4. **Локальные переменные уничтожаются при выходе из области видимости** — переменные уничтожаются при завершении локальной области видимости
5. **Семантика цикла For** — `i` в `for i in iter` является повторным связыванием, при каждой итерации создаётся новая локальная привязка

## Исправленные проблемы

### Исправление 1: Стек областей видимости BodyChecker (исходная проблема 2)

**Изменённый файл**: `src/frontend/typecheck/checking/mod.rs`

**Изменения**: Замена `vars: HashMap<String, PolyType>` на `scopes: Vec<HashMap<String, PolyType>>`

Новые методы:
- `enter_scope()` — войти в новую область видимости
- `exit_scope()` — выйти из текущей области видимости
- `var_exists_in_any_scope(name)` — проверить, существует ли переменная в любой области видимости
- `var_exists_in_current_scope(name)` — проверить, существует ли переменная в текущей области видимости
- `update_var(name, poly)` — обновить переменную в существующей области видимости

`add_var` теперь добавляет в текущую область видимости (`scopes.last_mut()`), `get_var` теперь ищет от внутренней области к внешней.

---

### Исправление 2: Проверка затенения объявления mut (исходная проблема 1)

**Изменённый файл**: `src/frontend/typecheck/checking/mod.rs` — `check_var_stmt`

**До**:
```rust
// Если переменная уже существует, унифицировать типы  ← Ошибка! Разрешено затенение
if let Some(existing_poly) = self.vars.get(name) {
    let _ = self.solver.unify(&existing_poly.body, &ty);
}
self.vars.insert(name.to_string(), PolyType::mono(ty));
```

**После**:
```rust
// Проверка затенения: если переменная существует в любой области видимости, ошибка
if self.var_exists_in_any_scope(name) {
    return Err(Box::new(
        ErrorCodeDefinition::variable_shadowing(name).build(),
    ));
}
self.add_var(name.to_string(), PolyType::mono(ty));
```

### Исправление 3: Область видимости цикла For (исходная проблема 3)

**Изменённый файл**: `src/frontend/typecheck/checking/mod.rs` — `check_for_stmt`

**Изменения**: Добавлены `enter_scope()` / `exit_scope()` перед и после проверки переменной цикла и тела цикла, чтобы гарантировать уничтожение переменной цикла после его завершения.

---

### Исправление 4: Проверка затенения выражения присваивания (новое)

**Изменённый файл**: `src/frontend/typecheck/checking/mod.rs` — `check_expr_stmt`

**Изменения**: Обработка `BinOp::Assign` теперь различает три случая:
1. **Существует в текущей области видимости** → операция присваивания (унификация типов)
2. **Существует только во внешней области видимости** → ошибка затенения
3. **Не существует** → создание новой переменной

---

### Исправление 5: Область видимости определения функции (новое)

**Изменённый файл**: `src/frontend/typecheck/checking/mod.rs` — `check_fn_def`

**Изменения**: Проверка определения функции теперь создаёт независимую область видимости функции; параметры функции и локальные переменные уничтожаются после завершения функции.

При этом из `check_fn_stmt` удалена дублирующая логика добавления параметров; параметры теперь управляются `check_fn_def` в области видимости функции.

---

### Исправление 6: Область видимости If/блока кода (новое)

**Изменённый файл**: `src/frontend/typecheck/checking/mod.rs` — `check_block`

**Изменения**: `check_block` теперь автоматически создаёт и выходит из области видимости, используется для ветвей then/elif/else оператора If.

---

### Исправление 7: Область видимости If/While в ExprInferrer (новое)

**Изменённый файл**: `src/frontend/typecheck/inference/expressions.rs`

**Изменения**:
- Выражение If: каждая ветвь (then/elif/else) использует независимую область видимости
- Выражение While: тело цикла использует независимую область видимости

## Примечания оставленных задач

### Семантика «повторного связывания» цикла For

Семантика `i` в цикле For (создание новой привязки при каждой итерации) обрабатывается на уровне генерации IR и интерпретатора; на этапе проверки типов достаточно убедиться, что переменная находится в области видимости. Текущая реализация приемлема.
