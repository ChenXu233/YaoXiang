# Движок вычислений на этапе компиляции (CTE Engine) и статическая верификация по Хору — план реализации

> **Задача**: Реализовать движок вычислений на этапе компиляции + канал статической верификации по Хору для поддержки зависимых от значений типов и верификации размерностей на этапе компиляции
> **На основе RFC**: RFC-010 (унифицированный синтаксис типов), RFC-011 (система generics/зависимые от значений типы), RFC-022 (статическая верификация по Хору)
> **Дата**: 2026-05-10
> **Статус**: В разработке
> **Целевые этапы**:
> - M1: Скелет для свёртывания констант + анализа чистоты
> - M2: Вычисление чистых функций на этапе компиляции + проверка завершения
> - M3: Вычисление на уровне типов (`If`/`Assert`/`match` семейства типов)
> - M4: Верификация по Хору (`//!` парсинг спецификаций → генерация VC → интеграция с SMT)

---

## Аннотация

Зависимые от значений типы YaoXiang (RFC-011) требуют, чтобы типы могли зависеть от значений, известных на этапе компиляции (например, `Vec(factorial(5))` → `Vec(120)`), а статическая верификация по Хору (RFC-022) требует проверки спецификаций чистых функций на этапе компиляции. Обе задачи разделяют одну и ту же ключевую потребность: **безопасно выполнять/анализировать чистые функции на этапе компиляции**.

Данный план предлагает **унифицированный движок вычислений на этапе компиляции (CTE Engine)**, абстрагирующий анализ чистоты, проверку завершения и вычисление выражений в общую инфраструктуру, обслуживающую двух потребителей: вычисления на уровне типов и верификацию по Хору.

---

## Ключевые принципы проектирования

1. **Повторное использование системы владения для анализа чистоты**: `&mut` в YaoXiang — это маркер побочных эффектов — проверка заимствований уже говорит нам, что будет изменено
2. **Чистая функция = вычислимая на этапе компиляции**: никакого ключевого слова `const fn`, компилятор автоматически выводит чистоту
3. **Доказательство завершения = фундамент безопасности типов**: вычисления в позиции типа должны доказать завершение (`decreases` спецификация), иначе система типов неразрешима
4. **Частичное вычисление лучше полного**: известно столько параметров — вычисляем столько, что нельзя вычислить — оставляем для runtime
5. **Общий интерпретатор для вычислений на этапе компиляции и логики Хора**: одно и то же ядро вычисления выражений обслуживает оба типа потребителей
6. **Двойной режим вычисления**: конкретное вычисление (известные параметры → результат `CTValue`) и символьное вычисление (неизвестные параметры → результат SMT) используют общую структуру интерпретатора, отличаясь только средой вычисления

## 一、Значения этапа компиляции (CTValue)

Ядро数据类型编译期计算。设计为编译器内部 IR 值，不与运行时值混淆。

```rust
/// Результат вычисления на этапе компиляции
enum CTValue {
    /// Целое число (охватывает все применения Bool на этапе компиляции)
    Int(i64),

    /// Число с плавающей запятой
    Float(f64),

    /// Строка (сообщения об ошибках, имена типов и т.д.)
    String(SmolStr),

    /// Ссылка на тип — ядро вычислений на уровне типов
    /// Типы YaoXiang сами являются "значениями" на уровне Type1
    Type(TypeId),

    /// Разнородный кортеж
    Tuple(Vec<CTValue>),

    /// Однородный массив
    Array(Vec<CTValue>),

    /// Структурированное значение
    Struct {
        type_id: TypeId,
        fields: HashMap<SmolStr, CTValue>,
    },

    /// Ссылка на невычисленную функцию (при частичном вычислении)
    /// Когда все параметры известны — инлайн-вычисление, иначе — сохранение для runtime
    Thunk {
        func: FunctionId,
        known_args: Vec<CTValue>,
        unknown_params: Vec<ParamId>,
    },
}
```

**Ключевой дизайн**: `CTValue::Type(TypeId)` делает тип первоклассным значением этапа компиляции. `If(C, T, E)` вычисляет C в `CTValue::Bool`, а T/E в `CTValue::Type`.

## 二、Подсистема 1：Анализатор чистоты

### 2.1 Концепция дизайна

Повторное использование системы владения YaoXiang (RFC-009), побочные эффекты естественно выражаются сигнатурами типов:

| Паттерн параметра | Значение | Вычислимо на этапе компиляции? |
|----------|------|---------------|
| `x: T` (owned) | Получение владения, свободная модификация | ✅ |
| `x: &T` (shared ref) | Только чтение | ✅ |
| `x: &mut T` (exclusive ref) | Может изменяться | ⚠️ зависит от источника T |
| I/O вызовы | Внешние побочные эффекты | ❌ |
| Вызов нечистой функции | Транзитивность | ❌ |

### 2.2 Алгоритм

```
analyze_purity(func: FunctionId, ctx: &mut PurityContext) -> PurityResult:
    // 1. Быстрый путь: уже аннотирован
    if ctx.has_purity_annotation(func):
        return ctx.get_annotation(func)

    // 2. Проверка прямых побочных эффектов
    for op in func.body.operations():
        match op:
            Call(callee, _) if is_io_operation(callee):
                return Impure("IO operation")
            Call(callee, args) where has_mut_arg(args):
                if arg_escapes_function(args):
                    return Impure("mutation of external state via &mut")
            Call(callee, _):
                // Транзитивность: вызываемая функция тоже должна быть чистой
                if analyze_purity(callee, ctx).is_impure():
                    return Impure("calls impure function: {callee}")

    // 3. По умолчанию выводим как чистую функцию
    return Pure
```

### 2.3 Отсутствие явных аннотаций чистоты

**Проектное решение: не предоставлять явных аннотаций вроде `//! pure`.**

Система владения (RFC-009) уже выражает информацию о побочных эффектах через сигнатуры типов — `&mut T` означает модификацию, I/O операции — внешние побочные эффекты. Компилятор способен автоматически вывести чистоту.

Если компилятор ошибочно считает чистую функцию нечистой — это баг компилятора, и следует исправить компилятор, а не позволять пользователю ставить заплатки. Предоставление аннотации "поверьте мне, эта функция чистая" только скроет реальную проблему.

> *"Не пишите совместимый, запасной, временный, альтернативный код, действующий только в определённых паттернах. Позвольте проблемам проявиться напрямую."*

### 2.4 Отношение к RFC-022

Анализатор чистоты одновременно обслуживает:
- **CTE**: нечистые функции нельзя использовать в позиции типа
- **Логику Хора**: выражения спецификаций (`requires`/`ensures` справа) должны быть вызовами чистых функций

## 三、Подсистема 2：Проверка завершения

### 3.1 Концепция дизайна

Вычисления на этапе компиляции в позиции типа должны гарантировать завершение, иначе система типов становится неразрешимой. YaoXiang использует `//! decreases` спецификацию для доказательства завершения.

```
//! decreases: <expr>
```

Где `<expr>` — это значение с нижней границей в well-founded множестве (обычно натуральное число типа `Int`).

### 3.2 Алгоритм

```
check_termination(func: FunctionId, ctx: &mut TermContext) -> TermResult:
    // 1. Поиск decreases спецификации
    let decreases_expr = find_decreases_spec(func)
        .or_else(|| infer_decreases(func))

    match decreases_expr:
        None if has_recursive_call(func):
            return TermError::NoDecreasesAnnotation
        None:
            return TermOk  // Нет рекурсии, доказательство не требуется

        Some(decreases):
            // 2. Верификация каждой точки рекурсивного вызова
            for call in func.recursive_calls():
                let dec_at_call = eval_decreases_at(call, decreases)
                let dec_at_entry = eval_decreases_at(func.entry, decreases)

                if !strictly_less_than(dec_at_call, dec_at_entry):
                    return TermError::NotDecreasing {
                        at: call.location,
                        expected_less_than: dec_at_entry,
                        actual: dec_at_call,
                    }

            // 3. Верификация нижней границы
            if !has_lower_bound(decreases):
                return TermError::NoLowerBound

            return TermOk
```

### 3.3 Автоматический вывод

Некоторые очевидные случаи завершения не требуют аннотации:

```yaoxiang
// decreases не требуется — компилятор видит цикл с известной верхней границей n
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n { s += arr[i]; i += 1 }
    return s
}
```

Случаи, требующие аннотации:
```yaoxiang
// decreases обязательна — рекурсивный вызов n-1
factorial: (n: Int) -> Int = {
    //! decreases: n
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}
```

### 3.4 Отношение к RFC-022

Проверка завершения одновременно обслуживает:
- **CTE**: decreases — это пропуск на этап вычислений на этапе компиляции
- **Логику Хора**: decreases вариант для инвариантов циклов (`/*! decreases: n - i !*/`) также верифицируется проверкой завершения

## 四、Подсистема 3：AST интерпретатор

### 4.1 Концепция дизайна

Интерпретатор основан на обходе AST, поддерживая среду вычисления (отображение имён переменных → CTValue). Ключевая способность — **частичное вычисление**: известные параметры вычисляем, неизвестные сохраняем.

```
eval(expr: &Expr, env: &mut EvalEnv) -> EvalResult<CTValue>:
    match expr:
        // Литерал → прямое преобразование
        Literal(lit) => lit.into_ctvalue()

        // Переменная → поиск в среде
        Variable(name) => env.get(name).ok_or(NotInScope)

        // Бинарная операция
        BinaryOp(l, op, r) =>
            let lv = eval(l, env)?; let rv = eval(r, env)?
            apply_op(op, lv, rv)

        // Условная ветвь
        If(cond, then, else) =>
            match eval(cond, env)? {
                Bool(true) => eval(then, env),
                Bool(false) => eval(else, env),
                _ => Err(ExpectedBool),
            }

        // Вызов функции — ключевая логика
        Call(func, args) =>
            let known_args = args.filter_map(|a| eval(a, env).ok())
            if known_args.len() == args.len():
                // Все параметры известны → инлайн-вычисление
                inline_and_eval(func, known_args, env)
            else if known_args.len() > 0:
                // Частично известно → частичное вычисление (генерация унитарного кода)
                partial_eval(func, known_args, env)
            else:
                // Всё неизвестно → Thunk
                CTValue::Thunk { func, known_args: vec![], unknown_params: args }

        // Pattern matching
        Match(scrutinee, arms) =>
            let val = eval(scrutinee, env)?
            for arm in arms:
                if arm.pattern.matches(val):
                    return eval(arm.body, env.with_bindings(arm.bindings))
            Err(NoMatch)

        // Блок кода
        Block(stmts) =>
            for stmt in stmts[..len-1]:
                eval(stmt, env)?
            eval(stmts.last(), env)

        // Цикл
        While(cond, body) =>
            let mut result = CTValue::Void
            while eval(cond, env)? == Bool(true):
                check_step_limit()?  // Предотвращение бесконечного цикла
                result = eval(body, env)?
            result
```

### 4.2 Инлайн-вычисление

Когда все параметры известны, интерпретатор инлайнит тело функции в среду вычисления:

```
inline_and_eval(func, args, env):
    // 1. Проверка чистоты
    purity.check(func)?

    // 2. Проверка кэша
    if let Some(cached) = cache.get(func, args):
        return cached

    // 3. Создание инлайн-среды
    let mut inline_env = env.child()
    for (param, arg) in func.params.zip(args):
        inline_env.bind(param.name, arg)

    // 4. Вычисление тела функции
    let result = eval(&func.body, &mut inline_env)?

    // 5. Кэширование результата
    cache.insert(func, args, result.clone())
    result
```

### 4.3 Лимит шагов

Вычисления на этапе компиляции должны иметь жёсткий лимит, предотвращающий непредвиденные таймауты даже при наличии `decreases`:

```
const MAX_EVAL_STEPS: u64 = 1_000_000;  // Жёсткий лимит в миллион шагов

struct EvalEnv {
    variables: HashMap<SmolStr, CTValue>,
    step_count: u64,
    step_limit: u64,
}
```

### 4.4 Двойной режим вычисления: конкретное vs символьное

Ядро интерпретатора (обход AST, инлайн-разворот, pattern matching) унифицировано, но **среда вычисления** определяет два режима:

#### 4.4.1 Конкретное вычисление (Concrete Evaluation)

**Потребитель**: CTE движок → вычисления на уровне типов, мономорфизация

**Характеристики**:
- Все переменные в среде имеют конкретное значение `CTValue`
- Параметры вызовов функций полностью известны → инлайн-вычисление
- Результат: `CTValue` (конкретное значение или ссылка на тип)
- Ошибка = ошибка компиляции

```
// Сценарий: Vec(factorial(5))
// env = { factorial → Function(...) }
eval(Call("factorial", [Literal(5)]), env):
    → inline_and_eval(factorial, [CTValue::Int(5)], env)
    → CTValue::Int(120)
// Замена типа: Vec(120)
```

#### 4.4.2 Символьное вычисление (Symbolic Evaluation)

**Потребитель**: Верификатор логики Хора → SMT решатель

**Характеристики**:
- В среде существуют **символьные переменные** (параметры функций `n`, `arr`, неизвестные на этапе компиляции)
- Известные подвыражения вычисляются в конкретные значения, неизвестные части сохраняются как SMT символы
- Вызовы функций не инлайнятся — вместо этого разворачиваются в логические формулы
- Результат: `SMTExpr` (выражение первопорядковой логики), передаётся в Z3
- Ошибка = верификация не пройдена (не ошибка компиляции)

```
// Сценарий: верификация ensures для max:
//   //! ensures: GreaterOrEqual(result, arr[0..n]) = result >= forall arr[i]
// env = { result → Symbol("result"), arr → Symbol("arr"), n → Symbol("n") }
eval(BinaryOp(Variable("result"), GtEq, Call("arr_max", [Symbol("arr"), Symbol("n")]))):
    // result — символ → сохраняется
    // arr_max(arr, n) — чистая функция, но параметры неизвестны → разворачивается в логическое определение
    → SMTExpr::Forall(i in 0..n, Symbol("result") >= Symbol("arr")[i])
// Передаётся в Z3: ∀arr, n, result. (n > 0 ∧ ...) → result >= arr[0] ∧ ... ∧ result >= arr[n-1]
```

#### 4.4.3 Ключевые различия двух режимов

| Измерение | Конкретное вычисление | Символьное вычисление |
|------|----------|----------|
| Среда | `HashMap<Name, CTValue>` | `HashMap<Name, SMTTerm>` |
| Неизвестная переменная | Ошибка | Сохраняется как символ |
| Вызов функции | Инлайн + выполнение тела | Разворачивается в логическое определение |
| Цикл | Реальная итерация (с лимитом шагов) | Преобразуется в VC инварианта цикла |
| Тип результата | `Result<CTValue, CTError>` | `Result<SMTExpr, SMError>` |
| Семантика ошибки | Ошибка компиляции | Ошибка верификации (может понижаться до Runtime Check) |
| Характеристика производительности | Быстро (прямое вычисление) | Медленно (SMT решение) |

#### 4.4.4 Общая структура интерпретатора

Оба режима используют общий каркас обхода AST:

```rust
/// trait Интерпретатор: конкретное и символьное вычисления реализуют各自
trait Interpreter {
    type Value;       // CTValue или SMTExpr
    type Error;       // CTError или SMError

    fn eval_literal(&mut self, lit: &Literal) -> Result<Self::Value, Self::Error>;
    fn eval_variable(&mut self, name: &str) -> Result<Self::Value, Self::Error>;
    fn eval_binary_op(&mut self, op: BinOp, l: Self::Value, r: Self::Value) -> Result<Self::Value, Self::Error>;
    fn eval_call(&mut self, func: FunctionId, args: &[Expr]) -> Result<Self::Value, Self::Error>;
    fn eval_if(&mut self, cond: &Expr, then: &Expr, else_: &Expr) -> Result<Self::Value, Self::Error>;
    fn eval_match(&mut self, scrutinee: &Expr, arms: &[MatchArm]) -> Result<Self::Value, Self::Error>;
    fn eval_while(&mut self, cond: &Expr, body: &Expr) -> Result<Self::Value, Self::Error>;
}

/// Унифицированный обходчик AST, делегирующий в конкретную реализацию
fn eval_ast<I: Interpreter>(interp: &mut I, expr: &Expr) -> Result<I::Value, I::Error> {
    match expr {
        Expr::Literal(lit) => interp.eval_literal(lit),
        Expr::Variable(name) => interp.eval_variable(name),
        Expr::BinaryOp { op, left, right } => {
            let l = eval_ast(interp, left)?;
            let r = eval_ast(interp, right)?;
            interp.eval_binary_op(*op, l, r)
        }
        Expr::Call { func, args } => interp.eval_call(*func, args),
        Expr::If { cond, then, else_ } => interp.eval_if(cond, then, else_),
        // ... остальные узлы AST аналогично
    }
}
```

**Ключевой инсайт**: Логика обхода AST для конкретного и символьного вычисления полностью идентична, различия только в:
- **Что используется для представления значений** (`CTValue` vs `SMTExpr`)
- **Как обрабатываются вызовы функций** (инлайн-выполнение vs логическое разворачивание)
- **Как обрабатываются неизвестные переменные** (ошибка vs сохранение символа)

## 五、Взаимодействие с типами и мономорфизацией

### 5.1 Точки использования CTE

```
1. Позиция аннотации типа
   Vec(factorial(5))        → CTE::eval(factorial(5)) → CTValue::Int(120)
   Замена типа на Vec(120)

2. Значения параметров generics
   Array(Int, factorial(3)) → CTE::eval(factorial(3)) → CTValue::Int(6)
   Инстанцирование Array(Int, 6)

3. Assert тип
   Assert(N > 0)            → CTE::eval(N > 0) → CTValue::Bool(true/false)
   True → Void, False → compile_error("N must be > 0")

4. If условный тип
   If(C, T, E)              → CTE::eval(C) → CTValue::Bool(b)
   True → T, False → E

5. Match семейство типов
   AsString(Int)            → match Int { Int => String, ... } → String
```

### 5.2 Взаимодействие с мономорфизацией

```
Мономорфизация использует результаты CTE в следующих точках:

1. Известные значения параметров generics → генерация конкретных инстансов
   Метод push для List(Int) → генерация push_List_Int

2. Известные зависимые от значений типы → разворачиваются в конкретные типы
   Matrix(Float, 3, 3).data → Array(Array(Float, 3), 3)

3. Частичное вычисление → генерация унитарного кода
   map(Int, String) → генерация map_Int_String, где T=Int, R=String уже зафиксированы
```

### 5.3 Взаимодействие с верификатором логики Хора

```
Верификатор использует CTE в следующих точках:

1. Частичное вычисление выражений спецификаций
   //! requires: n > 0 && factorial(n) < MAX
   CTE::eval(factorial(n)) → если n известно на этапе компиляции → константа
                           → если n неизвестно → сохраняется как символ, передаётся в SMT

2. Упрощение условий спецификаций
   //! ensures: result >= 0 && result < n
   CTE пытается упростить известные подвыражения, уменьшая нагрузку на SMT

3. Инстанцирование типов спецификаций
   NonEmpty(n) = n > 0
   CTE разворачивает типы спецификаций в булевы выражения
```

## 六、Статическая верификация по Хору (дизайн реализации RFC-022)

### 6.1 Парсинг спецификаций

`//!` и `/*! ... !*/` распознаются парсером как специальные узлы комментариев, присоединяемые к AST:

```rust
struct SpecAnnotation {
    kind: SpecKind,        // Requires | Ensures | Invariant | Decreases
    name: Option<SmolStr>, // Имя спецификации (необязательное пользовательское имя)
    spec_type: TypeExpr,   // Типовое выражение спецификации
    expr: Expr,            // Булево выражение
    span: Span,
}

enum SpecKind {
    Requires,
    Ensures,
    Invariant,
    Decreases,
}
```

### 6.2 Генерация условий верификации (VCGen)

Используется исчисление weakest precondition (наиболее слабого предусловия):

```
generate_vc(func: FunctionId) -> Vec<VerificationCondition>:
    let requires = collect_requires(func)
    let ensures = collect_ensures(func)
    let invariants = collect_invariants(func)
    let decreases = collect_decreases(func)

    let mut vcs = Vec::new()

    // VC1: Согласованность предусловий
    vcs.push(VC::PreconditionConsistency(requires))

    // VC2: Верификация постусловий (для каждого пути выполнения)
    for path in func.paths():
        let wp = compute_wp(path.body, ensures)
        vcs.push(VC::Postcondition {
            path: path.id,
            formula: implies(requires, wp),
        })

    // VC3: Инварианты циклов
    for (loop_, invariant) in invariants:
        // Выполняется перед входом в цикл
        vcs.push(VC::InvariantEntry { loop_, invariant })
        // Сохраняется при каждой итерации
        vcs.push(VC::InvariantPreservation { loop_, invariant })
        // После выхода влечёт постусловие
        vcs.push(VC::InvariantExit { loop_, invariant, post: ensures })

    vcs
```

### 6.3 Интеграция SMT решателя

```
┌─────────────┐     SMT-LIB формат      ┌───────────┐
│  Генератор VC │ ──────────────────→ │  Z3 решатель│
└─────────────┘                       └─────┬─────┘
                                            │
                          ┌─────────────────┴──────────────┐
                          ↓                                ↓
                      unsat                            sat
                          ↓                                ↓
                    ┌──────────┐                   ┌──────────────┐
                    │ Верификация│                   │ Извлечение   │
                    │  пройдена   │                   │ контрпримера  │
                    │  кэширование│                   │ преобразование│
                    └──────────┘                   │ в читаемый    │
                                                  │ формат        │
                                                  └──────┬───────┘
                                                         ↓
                                                  ┌──────────────┐
                                                  │ Отчёт об     │
                                                  │ ошибке       │

```

### 6.4 Режимы компиляции

| Режим | Поведение | CLI |
|------|------|-----|
| **Debug Build** | Парсит спецификации, генерирует VC, вызывает Z3 для доказательства; верификация должна пройти перед Release Build | `yaoxiangc --debug` |
| **Release Build** | Игнорирует все `//!` комментарии, нулевые накладные расходы, очищает кэш верификации | `yaoxiangc --release` |
| **Runtime Checks** | Спецификации понижаются до `assert` операторов, нарушение → panic | `yaoxiangc --runtime-checks` |

## 七、Этапы реализации

### Phase 1: Скелет для свёртывания констант + анализа чистоты

**Цель**: Создать инфраструктуру CTE, поддерживающую базовое вычисление на этапе компиляции

**Содержание**:
- [ ] Определить enum `CTValue` и структуру `EvalEnv`
- [ ] Реализовать базовые пути `eval()`: литералы, переменные, бинарные операции, условия, блоки
- [ ] Реализовать первую версию анализатора чистоты: распознавание I/O вызовов как нечистых, остальное по умолчанию чистое
- [ ] Вставить точки вызова CTE в проверщик типов (позиции аннотаций типов)
- [ ] Свёртывание констант: `1 + 2 * 3` вычисляется в `7` на этапе компиляции
- [ ] Устранение мёртвых ветвей: `if true { ... } else { ... }` → непосредственный выбор then ветви
- [ ] Модульные тесты: вычисление литералов, простые выражения, свёртывание констант

**Результат**: модуль `src/middle/cte/`, включающий `value.rs`, `eval.rs`, `purity.rs`

### Phase 2: Вычисление чистых функций на этапе компиляции + проверка завершения

**Цель**: Поддержка полного вычисления чистых функций на этапе компиляции

**Содержание**:
- [ ] Реализовать инлайн-вычисление функций: все параметры известны → разворачивание тела функции
- [ ] Реализовать парсинг и верификацию `//! decreases`
- [ ] Реализовать вычисление рекурсивных функций на этапе компиляции (с лимитом шагов)
- [ ] Реализовать кэширование результатов вычисления (Memoization)
- [ ] Усовершенствовать анализатор чистоты: использовать информацию о владении для распознавания побочных эффектов `&mut`
- [ ] Частичное вычисление: оптимизация генерации кода при частично известных параметрах
- [ ] Интеграционные тесты: `factorial(5)` вычисляется в `120` в позиции типа

**Результат**: `src/middle/cte/interpreter.rs`, `src/middle/cte/termination.rs`

### Phase 3: Вычисления на уровне типов

**Цель**: Поддержка семейств типов `If`/`Assert`/`match`

**Содержание**:
- [ ] Реализовать операции на уровне типов для `CTValue::Type(TypeId)`
- [ ] Реализовать условный тип `If: (C: Bool, T: Type, E: Type) -> Type`
- [ ] Реализовать `Assert(C)` → `True → Void, False → compile_error`
- [ ] Реализовать type-level `match`: `AsString: (T: Type) -> Type = match T { ... }`
- [ ] Полное инстанцирование зависимых от значений типов: `Matrix(Float, 3, 3)` → конкретный тип
- [ ] Верификация размерностей на этапе компиляции: несовпадение размерностей при умножении матриц → ошибка компиляции
- [ ] Интеграция с мономорфизацией (mono pass)

**Результат**: `src/middle/cte/type_level.rs`, обновление `src/middle/passes/mono/`

### Phase 4: Статическая верификация по Хору

**Цель**: Полный канал парсинга спецификаций, генерации VC, SMT верификации

**Содержание**:
- [ ] Расширение парсера: распознавание `//!` и `/*! ... !*/` как узлов спецификаций
- [ ] Определение типов встроенных спецификаций (`NonEmpty`, `Sorted`, `GreaterOrEqual` и т.д.)
- [ ] Поддержка пользовательских типов спецификаций
- [ ] Генератор VC: исчисление наиболее слабого предусловия
- [ ] Интеграция SMT решателя Z3 (через crate `z3`)
- [ ] Трансляция в формат SMT-LIB
- [ ] Извлечение и форматирование контрпримера
- [ ] Переключение между тремя режимами компиляции: Debug/Release/RuntimeChecks
- [ ] Интеграционные тесты: верификация спецификаций для `max`, `binary_search` и других функций

**Результат**: модуль `src/middle/verification/`

## 九、Журнал ключевых проектных решений

| Решение | Варианты | Выбор | Обоснование |
|------|------|------|------|
| Метод определения чистоты | Явные аннотации vs автоматический вывод vs комбинация | **Автоматический вывод** | Система владения уже предоставляет достаточно информации; не предоставляем "лазейку" с явными аннотациями |
| Вычислитель на этапе компиляции | Ограниченное подмножество vs полный язык | **Полный язык (с лимитом шагов)** | Согласуется с "всё есть `name: type = value`" в унифицированном синтаксисе типов |
| Доказательство завершения | Обязательные аннотации vs автоматический вывод | **В позиции типа обязательно, иначе автоматически** | Невозможность разрешения в позиции типа = ошибка компиляции, в других местах можно放宽 |
| Генерация VC | WP исчисление vs SP исчисление | **WP исчисление** | WP проще и прямолинейнее, ошибки локализуются чётче |
| SMT решатель | Z3 vs CVC5 vs собственный | **Z3** | Наиболее зрелый, лучшая поддержка Rust binding, largest community |
| Стратегия кэширования | Без кэша vs кэш между модулями | **LRU кэш + инкрементное аннулирование** | Результаты вычислений на этапе компиляции детерминированы для чистых функций, идеально для кэширования |
| Режим компиляции | Единый режим vs раздельный Debug/Release | **Debug верификация → Release нулевые накладные расходы** | Верификация дорога, Release не должен нести эти затраты |

## 十、Риски и их смягчение

| Риск | Влияние | Смягчение |
|------|------|------|
| Сложность интеграции Z3 | Задержка Phase 4 | Использование зрелого crate `z3`; начать с простой арифметики, расширять постепенно |
| Таймаут вычислений на этапе компиляции | Плохой UX | Лимит шагов + понятные сообщения о таймауте + предложения по упрощению выражений |
| Ошибки определения чистоты | Несоответствие результатов CTE и runtime | Система владения обеспечивает сильные гарантии; если ошибка — это баг компилятора, следует исправлять компилятор |
| Сложность отладки ошибок SMT верификации | Пользователь не понимает, почему спецификация не выполняется | Извлечение контрпримера + отображение конкретных входных значений + подсветка пути выполнения |
| Значительное увеличение времени компиляции | Замедление CI | Инкрементная верификация + кэш на уровне модулей + файлы результатов верификации (подобно `.o` файлам) |
