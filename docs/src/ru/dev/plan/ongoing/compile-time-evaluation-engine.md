# Движок вычислений на этапе компиляции (CTE Engine) и статическая верификация по Хоровой логике — план реализации

> **Задача**: реализовать движок вычислений на этапе компиляции + канал статической верификации по Хоровой логике для поддержки зависящих от значений типов и проверки размерностей на этапе компиляции
> **На основе RFC**: RFC-010 (унифицированный синтаксис типов), RFC-011 (обобщённая система/типы, зависящие от значений), RFC-022 (статическая верификация по Хоровой логике)
> **Дата**: 2026-05-10
> **Статус**: в разработке
> **Целевые вехи**:
> - M1: скелет свёртывания констант + анализа чистоты
> - M2: вычисление чистых функций на этапе компиляции + проверка завершения
> - M3: вычисления на уровне типов (`If`/`Assert`/`match` семейства типов)
> - M4: верификация по Хоровой логике (`//!` парсинг спецификаций → генерация VC → интеграция с SMT)

---

## Аннотация

Зависящие от значений типы YaoXiang (RFC-011) требуют, чтобы типы могли зависеть от значений, известных на этапе компиляции (например, `Vec(factorial(5))` → `Vec(120)`), а статическая верификация по Хоровой логике (RFC-022) требует проверки спецификаций чистых функций на этапе компиляции. Обе задачи разделяют одну и ту же ключевую потребность: **безопасно выполнять/анализировать чистые функции на этапе компиляции**.

Данный план предлагает **единый движок вычислений на этапе компиляции (CTE Engine)**, абстрагирующий анализ чистоты, проверку завершения и вычисление выражений в общую инфраструктуру, обслуживающую два потребителя: вычисления на уровне типов и верификацию по Хоровой логике.

---

## Ключевые принципы проектирования

1. **Повторное использование системы владения для анализа чистоты**: `&mut` в YaoXiang — это маркер побочных эффектов — проверка заимствований уже говорит нам, что будет изменено
2. **Чистая функция = вычислимая на этапе компиляции**: без ключевого слова `const fn`, компилятор автоматически выводит чистоту
3. **Доказательство завершения = основа безопасности типов**: вычисления в позиции типа должны доказывать завершение (`decreases`-спецификация), иначе система типов становится неразрешимой
4. **Частичное вычисление лучше полного**: известно сколько параметров — столько и вычисляем, что не удалось вычислить — остаётся для времени выполнения
5. **Разделение интерпретатора для вычислений и Хоровой логики**: единое ядро вычисления выражений поддерживает оба типа потребителей
6. **Двойной режим вычислений**: конкретное вычисление (известные параметры → результат `CTValue`) и символьное вычисление (неизвестные параметры → результат SMT-выражение) используют одну и ту же структуру интерпретатора, отличаясь лишь средой вычислений

## 一、编译期值（CTValue）

编译期计算的核心数据类型。设计为编译器内部 IR 值，不与运行时值混淆。

编译期 значения (CTValue) представляют собой основной тип данных для вычислений на этапе компиляции. Они разработаны как внутренние IR-значения компилятора и не смешиваются со значениями времени выполнения.

```rust
/// Результат вычисления на этапе компиляции
enum CTValue {
    /// Целое число (охватывает все применения Bool на этапе компиляции)
    Int(i64),

    /// Число с плавающей запятой
    Float(f64),

    /// Строка (сообщения об ошибках, имена типов и т.д.)
    String(SmolStr),

    /// Ссылка на тип — ядро вычислений на уровне типов
    /// Типы YaoXiang сами по себе являются "значениями" на уровне Type1
    Type(TypeId),

    /// Разнородный кортеж
    Tuple(Vec<CTValue>),

    /// Однородный массив
    Array(Vec<CTValue>),

    /// Структурированное значение
    Struct {
        type_id: TypeId,
        fields: HashMap<SmolStr, CTValue>,
    },

    /// Неоценённая ссылка на функцию (сохраняется при частичном вычислении)
    /// Когда все параметры известны — инлайнинг и вычисление, иначе — сохранение как вызов времени выполнения
    Thunk {
        func: FunctionId,
        known_args: Vec<CTValue>,
        unknown_params: Vec<ParamId>,
    },
}
```

**Ключевой момент дизайна**: `CTValue::Type(TypeId)` делает тип первоклассным значением на этапе компиляции. Для `If(C, T, E)`: C вычисляется в `CTValue::Bool`, T/E вычисляются в `CTValue::Type`.

## 二、Подсистема 1: Анализатор чистоты

### 2.1 Принцип проектирования

Повторное использование системы владения YaoXiang (RFC-009), побочные эффекты естественно выражаются через сигнатуры типов:

| Паттерн параметра | Значение | Вычислимо на этапе компиляции? |
|----------|------|---------------|
| `x: T` (владение) | Получение владения, свободная модификация | ✅ |
| `x: &T` (общее ссылка) | Только чтение | ✅ |
| `x: &mut T` (монопольная ссылка) | Может быть изменено | ⚠️ зависит от источника T |
| I/O вызовы | Внешние побочные эффекты | ❌ |
| Вызов нечистой функции | Транзитивность | ❌ |

### 2.2 Алгоритм

```
analyze_purity(func: FunctionId, ctx: &mut PurityContext) -> PurityResult:
    // 1. Быстрый путь: уже аннотирован
    if ctx.has_purity_annotation(func):
        return ctx.get_annotation(func)

    // 2. Проверка прямых побочных эффектов
    for op in func.body.operations():
        match op:
            Call(callee, _) if is_io_operation(callee):
                return Impure("IO operation")
            Call(callee, args) where has_mut_arg(args):
                if arg_escapes_function(args):
                    return Impure("mutation of external state via &mut")
            Call(callee, _):
                // Транзитивность: вызываемая функция тоже должна быть чистой
                if analyze_purity(callee, ctx).is_impure():
                    return Impure("calls impure function: {callee}")

    // 3. По умолчанию — чистая функция
    return Pure
```

### 2.3 Отсутствие явных аннотаций чистоты

**Проектное решение: не предоставлять явные аннотации вроде `//! pure`.**

Система владения (RFC-009) уже выражает информацию о побочных эффектах через сигнатуры типов — `&mut T` означает модификацию, I/O операции — внешние побочные эффекты. Компилятор способен автоматически вывести чистоту.

Если компилятор ошибочно считает чистую функцию нечистой — это баг компилятора, который следует исправить, а не позволять пользователю ставить заплатки. Предоставление аннотации "поверьте, эта функция чистая" только скроет реальную проблему.

> *"Не пишите совместимый, запасной, временный, резервный код, действующий только в определённых режимах. Позвольте проблемам проявляться напрямую."*

### 2.4 Связь с RFC-022

Анализатор чистоты одновременно обслуживает:
- **CTE**: нечистые функции нельзя использовать в позиции типа
- **Хорову логику**: выражения спецификаций (`requires`/`ensures` справа) должны быть вызовами чистых функций

## 三、Подсистема 2: Проверка завершения

### 3.1 Принцип проектирования

Вычисления на этапе компиляции в позиции типа должны гарантировать завершение, иначе система типов становится неразрешимой. YaoXiang использует `//! decreases`-спецификацию для доказательства завершения.

```
//! decreases: <expr>
```

где `<expr>` — это значение с нижней границей, принадлежащее хорошо обоснованному множеству (обычно натуральное число типа `Int`).

### 3.2 Алгоритм

```
check_termination(func: FunctionId, ctx: &mut TermContext) -> TermResult:
    // 1. Поиск decreases-спецификации
    let decreases_expr = find_decreases_spec(func)
        .or_else(|| infer_decreases(func))

    match decreases_expr:
        None if has_recursive_call(func):
            return TermError::NoDecreasesAnnotation
        None:
            return TermOk  // Без рекурсии, доказательство не требуется

        Some(decreases):
            // 2. Проверка каждой точки рекурсивного вызова
            for call in func.recursive_calls():
                let dec_at_call = eval_decreases_at(call, decreases)
                let dec_at_entry = eval_decreases_at(func.entry, decreases)

                if !strictly_less_than(dec_at_call, dec_at_entry):
                    return TermError::NotDecreasing {
                        at: call.location,
                        expected_less_than: dec_at_entry,
                        actual: dec_at_call,
                    }

            // 3. Проверка нижней границы
            if !has_lower_bound(decreases):
                return TermError::NoLowerBound

            return TermOk
```

### 3.3 Автоматический вывод

Некоторые очевидные случаи завершения не требуют аннотации:

```yaoxiang
// decreases не требуется — компилятор видит цикл с известной верхней границей n
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n { s += arr[i]; i += 1 }
    return s
}
```

Случаи, требующие аннотации:
```yaoxiang
// decreases обязательна — рекурсивный вызов n-1
factorial: (n: Int) -> Int = {
    //! decreases: n
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}
```

### 3.4 Связь с RFC-022

Проверка завершения одновременно обслуживает:
- **CTE**: decreases — это пропуск на этап вычисления на этапе компиляции
- **Хорову логику**: вариант decreases для инвариантов циклов (`/*! decreases: n - i !*/`) также проверяется проверкой завершения

## 四、Подсистема 3: AST Интерпретатор

### 4.1 Принцип проектирования

Интерпретатор основан на обходе AST, поддерживая среду вычислений (отображение имён переменных → CTValue). Ключевая способность — **частичное вычисление**: известные параметры вычисляются, неизвестные сохраняются.

```
eval(expr: &Expr, env: &mut EvalEnv) -> EvalResult<CTValue>:
    match expr:
        // Литерал → прямое преобразование
        Literal(lit) => lit.into_ctvalue()

        // Переменная → поиск в среде
        Variable(name) => env.get(name).ok_or(NotInScope)

        // Бинарная операция
        BinaryOp(l, op, r) =>
            let lv = eval(l, env)?; let rv = eval(r, env)?
            apply_op(op, lv, rv)

        // Условная ветвь
        If(cond, then, else) =>
            match eval(cond, env)? {
                Bool(true) => eval(then, env),
                Bool(false) => eval(else, env),
                _ => Err(ExpectedBool),
            }

        // Вызов функции — ключевая логика
        Call(func, args) =>
            let known_args = args.filter_map(|a| eval(a, env).ok())
            if known_args.len() == args.len():
                // Все параметры известны → инлайн-вычисление
                inline_and_eval(func, known_args, env)
            else if known_args.len() > 0:
                // Частично известно → частичное вычисление (генерация мономорфного кода)
                partial_eval(func, known_args, env)
            else:
                // Всё неизвестно → Thunk
                CTValue::Thunk { func, known_args: vec![], unknown_params: args }

        // Паттерн матчинг
        Match(scrutinee, arms) =>
            let val = eval(scrutinee, env)?
            for arm in arms:
                if arm.pattern.matches(val):
                    return eval(arm.body, env.with_bindings(arm.bindings))
            Err(NoMatch)

        // Блок кода
        Block(stmts) =>
            for stmt in stmts[..len-1]:
                eval(stmt, env)?
            eval(stmts.last(), env)

        // Цикл
        While(cond, body) =>
            let mut result = CTValue::Void
            while eval(cond, env)? == Bool(true):
                check_step_limit()?  // Предотвращение бесконечного цикла
                result = eval(body, env)?
            result
```

### 4.2 Инлайн-вычисление

Когда все параметры известны, интерпретатор инлайнит тело функции в контекст вычисления:

```
inline_and_eval(func, args, env):
    // 1. Проверка чистоты
    purity.check(func)?

    // 2. Проверка кэша
    if let Some(cached) = cache.get(func, args):
        return cached

    // 3. Создание инлайн-среды
    let mut inline_env = env.child()
    for (param, arg) in func.params.zip(args):
        inline_env.bind(param.name, arg)

    // 4. Вычисление тела функции
    let result = eval(&func.body, &mut inline_env)?

    // 5. Кэширование результата
    cache.insert(func, args, result.clone())
    result
```

### 4.3 Ограничение числа шагов

Вычисление на этапе компиляции должно иметь жёсткий лимит для предотвращения неожиданных таймаутов даже при наличии `decreases`:

```
const MAX_EVAL_STEPS: u64 = 1_000_000;  // Жёсткий лимит в миллион шагов

struct EvalEnv {
    variables: HashMap<SmolStr, CTValue>,
    step_count: u64,
    step_limit: u64,
}
```

### 4.4 Двойной режим вычислений: конкретное vs символьное

Основа интерпретатора (обход AST, инлайн-раскрытие, паттерн матчинг) едина, но **среда вычислений** определяет два режима:

#### 4.4.1 Конкретное вычисление (Concrete Evaluation)

**Потребитель**: CTE движок → вычисления на уровне типов, мономорфизация

**Характеристики**:
- Все переменные в среде имеют конкретные `CTValue`
- Все параметры вызовов функций известны → инлайн-вычисление
- Результат: `CTValue` (конкретное значение или ссылка на тип)
- Ошибка = ошибка компиляции

```
// Сценарий: Vec(factorial(5))
// env = { factorial → Function(...) }
eval(Call("factorial", [Literal(5)]), env):
    → inline_and_eval(factorial, [CTValue::Int(5)], env)
    → CTValue::Int(120)
// Подстановка типа: Vec(120)
```

#### 4.4.2 Символьное вычисление (Symbolic Evaluation)

**Потребитель**: Верификатор по Хоровой логике → SMT-решатель

**Характеристики**:
- В среде присутствуют **символьные переменные** (параметры функций `n`, `arr`, неизвестные на этапе компиляции)
- Известные подвыражения вычисляются в конкретные значения, неизвестные части сохраняются как SMT-символы
- Вызовы функций не инлайнятся — вместо этого раскрываются в логические формулы
- Результат: `SMTExpr` (выражение первопорядковой логики), передаётся в Z3
- Ошибка = верификация не пройдена (не ошибка компиляции)

```
// Сценарий: верификация ensures для max:
//   //! ensures: GreaterOrEqual(result, arr[0..n]) = result >= forall arr[i]
// env = { result → Symbol("result"), arr → Symbol("arr"), n → Symbol("n") }
eval(BinaryOp(Variable("result"), GtEq, Call("arr_max", [Symbol("arr"), Symbol("n")]))):
    // result — символ → сохраняется
    // arr_max(arr, n) — чистая функция, но параметры неизвестны → раскрытие в логическое определение
    → SMTExpr::Forall(i in 0..n, Symbol("result") >= Symbol("arr")[i])
// Передаётся в Z3: ∀arr, n, result. (n > 0 ∧ ...) → result >= arr[0] ∧ ... ∧ result >= arr[n-1]
```

#### 4.4.3 Ключевые различия двух режимов

| Измерение | Конкретное вычисление | Символьное вычисление |
|------|----------|----------|
| Среда | `HashMap<Name, CTValue>` | `HashMap<Name, SMTTerm>` |
| Неизвестная переменная | Ошибка | Сохраняется как символ |
| Вызов функции | Инлайн + вычисление тела | Раскрытие в логическое определение (без выполнения) |
| Цикл | Реальные итерации (с лимитом шагов) | Преобразование в VC через инвариант цикла |
| Тип результата | `Result<CTValue, CTError>` | `Result<SMTExpr, SMError>` |
| Семантика ошибки | Ошибка компиляции | Ошибка верификации (можно понизить до Runtime Check) |
| Характеристика производительности | Быстро (прямое вычисление) | Медленно (SMT-решение) |

#### 4.4.4 Общий каркас интерпретатора

Оба режима используют единый каркас обхода AST:

```rust
/// Интерпретатор trait: конкретное и символьное вычисления реализуют各自实现
trait Interpreter {
    type Value;       // CTValue или SMTExpr
    type Error;       // CTError или SMError

    fn eval_literal(&mut self, lit: &Literal) -> Result<Self::Value, Self::Error>;
    fn eval_variable(&mut self, name: &str) -> Result<Self::Value, Self::Error>;
    fn eval_binary_op(&mut self, op: BinOp, l: Self::Value, r: Self::Value) -> Result<Self::Value, Self::Error>;
    fn eval_call(&mut self, func: FunctionId, args: &[Expr]) -> Result<Self::Value, Self::Error>;
    fn eval_if(&mut self, cond: &Expr, then: &Expr, else_: &Expr) -> Result<Self::Value, Self::Error>;
    fn eval_match(&mut self, scrutinee: &Expr, arms: &[MatchArm]) -> Result<Self::Value, Self::Error>;
    fn eval_while(&mut self, cond: &Expr, body: &Expr) -> Result<Self::Value, Self::Error>;
}

/// Единый обходчик AST, делегирующий конкретной реализации
fn eval_ast<I: Interpreter>(interp: &mut I, expr: &Expr) -> Result<I::Value, I::Error> {
    match expr {
        Expr::Literal(lit) => interp.eval_literal(lit),
        Expr::Variable(name) => interp.eval_variable(name),
        Expr::BinaryOp { op, left, right } => {
            let l = eval_ast(interp, left)?;
            let r = eval_ast(interp, right)?;
            interp.eval_binary_op(*op, l, r)
        }
        Expr::Call { func, args } => interp.eval_call(*func, args),
        Expr::If { cond, then, else_ } => interp.eval_if(cond, then, else_),
        // ... остальные узлы AST аналогично
    }
}
```

**Ключевой инсайт**: Логика обхода AST для конкретного и символьного вычислений полностью идентична, различия только в:
- **Как представлять значения** (`CTValue` vs `SMTExpr`)
- **Как обрабатывать вызовы функций** (инлайн-выполнение vs логическое раскрытие)
- **Как обрабатывать неизвестные переменные** (ошибка vs сохранение символа)

## 五、Взаимодействие с другими компонентами

### 5.1 Точки использования CTE

CTE результаты используются в следующих позициях:

1. **Аннотации типа**
   `Vec(factorial(5))` → `CTE::eval(factorial(5))` → `CTValue::Int(120)`
   Подстановка типа: `Vec(120)`

2. **Обобщённые параметры-значения**
   `Array(Int, factorial(3))` → `CTE::eval(factorial(3))` → `CTValue::Int(6)`
   Инстанциирование: `Array(Int, 6)`

3. **Тип Assert**
   `Assert(N > 0)` → `CTE::eval(N > 0)` → `CTValue::Bool(true/false)`
   True → `Void`, False → `compile_error("N must be > 0")`

4. **Условные типы If**
   `If(C, T, E)` → `CTE::eval(C)` → `CTValue::Bool(b)`
   True → T, False → E

5. **Семейство типов Match**
   `AsString(Int)` → `match Int { Int => String, ... }` → `String`

### 5.2 Взаимодействие с мономорфизацией

```
Мономорфизация использует результаты CTE в следующих позициях:

1. Известные обобщённые параметры-значения → генерация конкретных инстансов
   Метод push для List(Int) → генерация push_List_Int

2. Известные типы, зависящие от значений → раскрытие в конкретный тип
   Matrix(Float, 3, 3).data → Array(Array(Float, 3), 3)

3. Частичное вычисление → генерация мономорфного кода
   map(Int, String) → генерация map_Int_String, где T=Int, R=String уже зафиксированы
```

### 5.3 Взаимодействие с верификатором по Хоровой логике

```
Верификатор использует CTE в следующих позициях:

1. Частичное вычисление выражений спецификаций
   //! requires: n > 0 && factorial(n) < MAX
   CTE::eval(factorial(n)) → если n известно на этапе компиляции → константа
                            → если n неизвестно → сохранение как символ, передача в SMT

2. Упрощение условий спецификаций
   //! ensures: result >= 0 && result < n
   CTE пытается упростить известные подвыражения, уменьшая нагрузку на SMT-решатель

3. Инстанциирование типов спецификаций
   NonEmpty(n) = n > 0
   CTE раскрывает типы спецификаций в булевы выражения
```

## 六、Статическая верификация по Хоровой логике (реализация RFC-022)

### 6.1 Парсинг спецификаций

`//!` и `/*! ... !*/` распознаются парсером как специальные узлы комментариев, присоединяемые к AST:

```rust
struct SpecAnnotation {
    kind: SpecKind,        // Requires | Ensures | Invariant | Decreases
    name: Option<SmolStr>, // Имя спецификации (необязательное пользовательское имя)
    spec_type: TypeExpr,   // Выражение типа спецификации
    expr: Expr,            // Булево выражение
    span: Span,
}

enum SpecKind {
    Requires,
    Ensures,
    Invariant,
    Decreases,
}
```

### 6.2 Генерация условий верификации (VCGen)

Используется исчисление наиweakest precondition (слабейшее предусловие):

```
generate_vc(func: FunctionId) -> Vec<VerificationCondition>:
    let requires = collect_requires(func)
    let ensures = collect_ensures(func)
    let invariants = collect_invariants(func)
    let decreases = collect_decreases(func)

    let mut vcs = Vec::new()

    // VC1: Согласованность предусловий
    vcs.push(VC::PreconditionConsistency(requires))

    // VC2: Верификация постусловий (для каждого пути выполнения)
    for path in func.paths():
        let wp = compute_wp(path.body, ensures)
        vcs.push(VC::Postcondition {
            path: path.id,
            formula: implies(requires, wp),
        })

    // VC3: Инварианты циклов
    for (loop_, invariant) in invariants:
        // Выполняется перед входом в цикл
        vcs.push(VC::InvariantEntry { loop_, invariant })
        // Сохраняется при каждой итерации
        vcs.push(VC::InvariantPreservation { loop_, invariant })
        // После выхода влечёт постусловие
        vcs.push(VC::InvariantExit { loop_, invariant, post: ensures })

    vcs
```

### 6.3 Интеграция с SMT-решателем

```
┌─────────────┐     SMT-LIB формат      ┌───────────┐
│ VC Генератор │ ──────────────────→   │  Z3 Решатель│
└─────────────┘                        └─────┬─────┘
                                            │

                      unsat                            sat

                    ┌──────────┐                   ┌──────────────┐
```

### 6.4 Режимы компиляции

| Режим | Поведение | CLI |
|------|------|-----|
| **Debug Build** | Парсит спецификации, генерирует VC, вызывает Z3 для доказательства; верификация должна пройти перед Release Build | `yaoxiangc --debug` |
| **Release Build** | Игнорирует все `//!` комментарии, нулевые накладные расходы, очищает кэш верификации | `yaoxiangc --release` |

## 七、Этапы реализации

### Phase 1: Скелет свёртывания констант + анализа чистоты

**Цель**: создать инфраструктуру CTE для поддержки базовых вычислений на этапе компиляции

**Содержание**:
- [ ] Определить enum `CTValue` и структуру `EvalEnv`
- [ ] Реализовать базовые пути `eval()`: литералы, переменные, бинарные операции, условия, блоки
- [ ] Реализовать первую версию анализатора чистоты: распознавание I/O как нечистых, остальное по умолчанию чистое
- [ ] Вставить вызовы CTE в проверщик типов (позиции аннотаций типа)
- [ ] Свёртывание констант: `1 + 2 * 3` вычисляется в `7` на этапе компиляции
- [ ] Устранение мёртвых ветвей: `if true { ... } else { ... }` → непосредственный выбор then-ветви
- [ ] Модульные тесты: вычисление литералов, простые выражения, свёртывание констант

**Результат**: модуль `src/middle/cte/`, включающий `value.rs`, `eval.rs`, `purity.rs`

### Phase 2: Вычисление чистых функций на этапе компиляции + проверка завершения

**Цель**: поддержка полного вычисления чистых функций на этапе компиляции

**Содержание**:
- [ ] Реализовать инлайн-вычисление функций: все параметры известны → раскрытие тела функции
- [ ] Реализовать парсинг `//! decreases` и проверку завершения
- [ ] Реализовать вычисление рекурсивных функций на этапе компиляции (с лимитом шагов)
- [ ] Реализовать кэширование результатов вычислений (Memoization)
- [ ] Усовершенствовать анализатор чистоты: использовать информацию о владении для распознавания побочных эффектов `&mut`
- [ ] Частичное вычисление: оптимизация генерации кода при известных частичных параметрах
- [ ] Интеграционные тесты: `factorial(5)` в позиции типа вычисляется в `120`

**Результат**: `src/middle/cte/interpreter.rs`, `src/middle/cte/termination.rs`

### Phase 3: Вычисления на уровне типов

**Цель**: поддержка семейств типов `If`/`Assert`/`match`

**Содержание**:
- [ ] Реализовать операции на уровне типов для `CTValue::Type(TypeId)`
- [ ] Реализовать `If: (C: Bool, T: Type, E: Type) -> Type` условный тип
- [ ] Реализовать `Assert(C)` → `True → Void, False → compile_error`
- [ ] Реализовать `match` на уровне типов: `AsString: (T: Type) -> Type = match T { ... }`
- [ ] Полная инстанциация типов, зависящих от значений: `Matrix(Float, 3, 3)` → конкретный тип
- [ ] Проверка размерностей на этапе компиляции: несовпадение размерностей при умножении матриц → ошибка компиляции
- [ ] Интеграция с мономорфизацией (mono pass)

**Результат**: `src/middle/cte/type_level.rs`, обновление `src/middle/passes/mono/`

### Phase 4: Статическая верификация по Хоровой логике

**Цель**: полный канал парсинга спецификаций, генерации VC, SMT-верификации

**Содержание**:
- [ ] Расширение парсера: распознавание `//!` и `/*! ... !*/` как узлов спецификаций
- [ ] Определение типов встроенных спецификаций (`NonEmpty`, `Sorted`, `GreaterOrEqual` и др. в стандартной библиотеке)
- [ ] Поддержка пользовательских типов спецификаций
- [ ] Генератор VC: исчисление слабейшего предусловия (WP)
- [ ] Интеграция SMT-решателя Z3 (через crate `z3`)
- [ ] Перевод в формат SMT-LIB
- [ ] Извлечение и форматирование контрпримеров
- [ ] Переключение трёх режимов компиляции: Debug/Release/RuntimeChecks
- [ ] Интеграционные тесты: верификация спецификаций для `max`, `binary_search` и других функций

**Результат**: модуль `src/middle/verification/`

## 九、Журнал ключевых проектных решений

| Решение | Варианты | Выбор | Обоснование |
|------|------|------|------|
| Способ определения чистоты | Явная аннотация vs автоматический вывод vs комбинация | **Автоматический вывод** | Система владения уже предоставляет достаточно информации; не предоставлять аварийный люк с явной аннотацией |
| Вычислитель на этапе компиляции | Ограниченное подмножество vs полный язык | **Полный язык (с лимитом шагов)** | Соответствует унифицированному синтаксису типов "всё — `name: type = value`" |
| Доказательство завершения | Обязательная аннотация vs автоматический вывод | **В позиции типа обязательно, остальное автоматически** | Невозможность разрешения в позиции типа = ошибка компиляции, в остальных местах можно ослабить |
| Генерация VC | WP-исчисление vs SP-исчисление | **WP-исчисление** | WP проще и прямее, локализация ошибок чётче |
| SMT-решатель | Z3 vs CVC5 vs собственная разработка | **Z3** | Наиболее зрелый, лучшие Rust-binding'и, крупнейшее сообщество |
| Стратегия кэширования | Без кэша vs кэш между модулями | **LRU-кэш + инкрементное аннулирование** | Результаты вычислений на этапе компиляции — детерминированные чистые функции, естественно подходят для кэширования |
| Режим компиляции | Единый режим vs разделение Debug/Release | **Debug верификация → Release с нулевыми накладными расходами** | Верификация дорогостояща, Release не должен её нести |

## 十、Риски и меры снижения

| Риск | Влияние | Снижение |
|------|------|------|
| Сложность интеграции Z3 | Задержка Phase 4 | Использование зрелого crate `z3`; поддержка сначала простой арифметики, постепенное расширение |
| Таймаут вычислений на этапе компиляции | Плохой UX | Лимит шагов + понятные сообщения о таймауте + предложения по упрощению выражений |
| Ошибочное определение чистоты | Несоответствие результатов вычисления на этапе компиляции и времени выполнения | Система владения обеспечивает сильные гарантии; если ошиблись — это баг компилятора, следует чинить компилятор |
| Сложность отладки при сбое SMT-верификации | Пользователи не понимают, почему спецификация не выполняется | Извлечение контрпримера + отображение конкретных входных значений + подсветка пути выполнения |
| Значительное увеличение времени компиляции | Замедление CI | Инкрементная верификация + кэш на уровне модулей + файлы результатов верификации (аналогично `.o`-файлам) |
