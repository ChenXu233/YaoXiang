> Расположение тестов: `src/frontend/core/typecheck/layers/tests/ownership.rs`

## Дефекты корректности

- [x] ### 1. ref-алиасинг в spawn не помечается как escape (P0) — Исправлено (2026-06-15)

**Сценарий**:
```yaoxiang
shared = ref x
alias = shared       // shared Move → alias
spawn { use(alias) } // alias ∉ ref_vars → пропущена пометка escape → выбор Rc (не атомарный, небезопасен для потоков)
```

**Корневая причина**: `OwnershipChecker` отслеживает только имена переменных, непосредственно присвоенных из `Expr::Ref` (множество `ref_vars`). Когда ref-переменная передаётся через Move промежуточной переменной, промежуточная переменная не приводит к обновлению `ref_vars`.

**Влияние**: ref, используемый между spawn, может быть ошибочно скомпилирован в `RcNew`; неатомарный подсчёт ссылок при использовании между потоками может привести к гонке данных.

**Исправление**: В обработчиках `StmtKind::Var` и `BinOp::Assign`, когда правая часть представляет собой `Expr::Var(name)` и `name ∈ ref_vars`, цель слева добавляется в `ref_vars` (commit `9029d5b`).

- [x] ### 2. После Move захваченной spawn-переменной внешняя область остаётся доступной (P1) — Исправлено (2026-06-16)

**Сценарий**:
```yaoxiang
shared = ref data
spawn { a = shared }  // walk тела spawn (save/restore) → shared в теле Moved → внешняя область восстановлена
use(shared)            // во внешней области shared всё ещё Alive — корректно, но в теле spawn shared уже перемещён
```

**Корневая причина**: `Expr::Spawn` использует save/restore — изменения владения внутри тела spawn не влияют на внешнюю область видимости. Это правильный дизайн, но Move `a = shared` внутри тела spawn обнаруживается только в рамках «временного walk» spawn. Если тело spawn выполняет Move `shared`, save/restore восстанавливает внешнее состояние, **однако ничто не мешает внешнему коду продолжить использование `shared` после spawn**.

**Влияние**: Если spawn при фактическом выполнении перемещает `shared` (например, `a = shared`), внешний код после spawn всё ещё может обращаться к `shared` — это может быть корректным в модели конкурентности YaoXiang (spawn получает независимую копию), но семантика не определена однозначно.

**Направление исправления**: Необходимо уточнить спецификацию языка: влияет ли Move-семантика захваченных spawn-переменных на внешнюю область видимости. Если принята модель «spawn получает независимую копию», текущее поведение корректно. Если принята модель «spawn потребляет владение», требуется убрать save/restore либо ввести Captures по аналогии с замыканиями.

## Компромиссы точности

- [ ] ### 3. Консервативное сообщение о конфликте для взаимоисключающих ветвей (P1)

**Сценарий**:
```yaoxiang
if cond {
    a = &mut x   // ветвь A
} else {
    b = &mut x   // ветвь B
}
// Теоретически: A и B взаимоисключающие, конфликта быть не должно
// На практике: два WriteToken создаются последовательно → сообщается BorrowConflict
```

**Корневая причина**: Архитектурное ограничение `NLL without fixpoint` — однопроходный walk по AST не моделирует условия пути и не различает взаимоисключающие ветви от последовательного выполнения.

**Направление исправления**: Требуется подключение медленного канала SMT на уровне CFG (в настоящее время `smt_cut` уже реализован, но срабатывает только в сценариях `while + path_condition`). Расширение на ветви if/else требует распространения `path_condition` в обработчик Borrow.

- [ ] ### 4. Тип ref не распознаётся как Dup (P1)

**Сценарий**:
```yaoxiang
shared = ref x
a = shared    // Move — но ref теоретически является Dup-типом и должен копироваться
b = shared    // use after move — в действительности должно быть разрешено
```

**Корневая причина**: Проверщик владения не учитывает, что `ref T` — это Dup-тип (копируемая ручка со подсчётом ссылок). Логика Move в `StmtKind::Var` обрабатывает все типы единообразно.

**Влияние**: Семантика значений типа ref оказывается строже, чем предполагалось, — отсутствует возможность «свободного копирования», как это задумано в RFC-009.

**Направление исправления**: Необходимо запрашивать тип переменной у `TypeEnvironment` и пропускать логику Move для Dup-типов. Это согласуется с общим дизайном, требующим явного вызова `clone()` — текущее консервативное поведение не шире корректной семантики.

## Инфраструктура

- [ ] ### 5. Формат кодов ошибок не унифицирован (P2)

**Описание**: Проверщик владения во фронтенде использует `DisproofModel.into_diagnostic()` → коды ошибок E2014-E2020. Унаследованный `lifetime/error.rs` в middle-слое использует собственный `ValueState` + `Checker` trait. В настоящее время сосуществуют две системы.

**Направление исправления**: Удалить `ValueState` и `Checker` trait из `error.rs` в middle-слое (осталось всего 2 ссылки: `lifecycle.rs` и тест `cycle_check`) и унифицировать с системой кодов ошибок фронтенда.

- [ ] ### 6. Вложенные функции с параметрами не анализируются (P2)

**Описание**: `StmtKind::Binding` выполняет анализ захвата только для замыканий вида `params.is_empty() && !body.is_empty()`. Для вложенных функций с параметрами возвращается `vec![]` (тело проверяется отдельно через `check_module`, но семантика захвата не анализируется).

**Влияние**: Ошибки владения в теле вложенной функции с параметрами не обнаруживаются (в настоящее время просто пропускаются), информация о захвате также не формируется. Если вложенная функция с параметрами использует переменные из внешней области, семантика владения остаётся непроанализированной.

**Направление исправления**: Унифицировать обработку Binding как с параметрами, так и без параметров, выполняя для их тел одновременно `check_function` и анализ захвата.
