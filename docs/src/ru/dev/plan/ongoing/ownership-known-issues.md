> Последнее обновление: 2026-06-16
> Расположение реализации: `src/frontend/core/typecheck/layers/ownership.rs`
> Расположение тестов: `src/frontend/core/typecheck/layers/tests/ownership.rs`
> 61 тестов, 0 сбоев

## Дефекты корректности

- [x] ### 1. ref-псевдоним попадает в spawn — пропуск пометки об утечке (P0) — Исправлено (2026-06-15)

**Сценарий**:
```yaoxiang
shared = ref x
alias = shared       // shared Move → alias
spawn { use(alias) } // alias ∉ ref_vars → 漏标逃逸 → 选 Rc（非原子，跨线程不安全）
```

**Корневая причина**: `OwnershipChecker` отслеживает только имена переменных, напрямую присвоенных через `Expr::Ref` (`ref_vars`). Когда ref-переменная перемещается в промежуточную переменную, промежуточная переменная не изменяет `ref_vars`.

**Влияние**: ref, используемый через spawn, может ошибочно компилироваться в `RcNew`; неатомарный подсчёт ссылок при использовании между потоками может приводить к гонкам данных.

**Исправление**: в обработчиках `StmtKind::Var` и `BinOp::Assign`, когда правая часть представляет собой `Expr::Var(name)` и `name ∈ ref_vars`, левая цель добавляется в `ref_vars` (commit `9029d5b`).

- [ ] ### 2. После Move переменной, захваченной spawn, она остаётся доступной во внешней области (P1 — семантика не определена)

**Сценарий**:
```yaoxiang
shared = ref data
spawn { a = shared }  // spawn 体 walk（save/restore）→ shared 在体内 Moved → 外层恢复
use(shared)            // 外层 shared 仍 Alive——正确，但 spawn 体内 shared 已 Move
```

**Корневая причина**: `Expr::Spawn` использует save/restore, изменения ownership внутри тела spawn не влияют на внешнюю область. Это правильное решение, но Move для `a = shared` внутри тела spawn обнаруживается только в ходе «временного обхода» spawn. Если тело spawn выполняет Move для `shared`, save/restore восстанавливает состояние внешней области, **но ничто не мешает внешней области продолжать использовать `shared` после spawn**.

**Влияние**: если spawn во время выполнения перемещает `shared` (например, `a = shared`), внешний код после spawn по-прежнему может обращаться к `shared` — в модели параллелизма YaoXiang это может быть корректным (spawn получает независимую копию), однако семантика явно не определена.

**Направление исправления**: необходимо уточнить спецификацию языка: влияет ли семантика Move при захвате в spawn на внешнюю область видимости. Если принят вариант «spawn получает независимую копию», текущее поведение корректно. Если принят вариант «spawn потребляет ownership», требуется убрать save/restore либо ввести Captures по аналогии с замыканиями.

## Компромиссы точности

- [ ] ### 3. Взаимоисключающие ветви консервативно сообщают о конфликте (P1)

**Сценарий**:
```yaoxiang
if cond {
    a = &mut x   // 分支 A
} else {
    b = &mut x   // 分支 B
}
// 理论上：A 和 B 互斥，不应冲突
// 实际：两个 WriteToken 先后创建 → 报 BorrowConflict
```

**Корневая причина**: архитектурное ограничение `NLL without fixpoint` — однопроходный обход AST не моделирует условия путей, что делает невозможным различение взаимоисключающих ветвей и последовательного выполнения.

**Направление исправления**: требуется подключение медленного SMT-пути CFG (в настоящее время `smt_cut` уже реализован, но срабатывает только в сценарии `while + path_condition`). Распространение на ветви if/else требует передачи `path_condition` в обработчик Borrow.

- [ ] ### 4. Тип ref не распознаётся как Dup (P1)

**Сценарий**:
```yaoxiang
shared = ref x
a = shared    // Move——但 ref 理论上是 Dup 类型，应可复制
b = shared    // use after move——实际上应允许
```

**Корневая причина**: проверщик ownership не знает, что `ref T` является типом Dup (копируемый дескриптор со счётчиком ссылок). Логика Move в `StmtKind::Var` обрабатывает все типы единообразно.

**Влияние**: семантика значений типа ref получается строже ожидаемой — «свободное копирование», предусмотренное дизайном RFC-009, не работает.

**Направление исправления**: требуется запрашивать тип переменной из `TypeEnvironment` и пропускать логику Move для типов Dup. Это согласуется с общим дизайном, требующим явного вызова `clone()` — текущее консервативное поведение не более мягкое, чем корректная семантика.

## Инфраструктура

- [ ] ### 5. Формат кодов ошибок не унифицирован (P2)

**Описание**: проверщик ownership на фронтенде использует `DisproofModel.into_diagnostic()` → коды ошибок E2014–E2020. Унаследованный middle-уровень `lifetime/error.rs` использует собственные `ValueState` + `Checker` trait. В настоящее время сосуществуют две системы.

**Направление исправления**: удалить `ValueState` и `Checker` trait из `error.rs` middle-уровня (осталось всего 2 ссылки: `lifecycle.rs` и тест `cycle_check`), унифицировав с системой кодов ошибок фронтенда.

- [ ] ### 6. Вложенные функции с параметрами не анализируются (P2)

**Описание**: `StmtKind::Binding` выполняет анализ захвата только для замыканий, у которых `params.is_empty() && !body.is_empty()`. Вложенные функции с параметрами возвращают `vec![]` (их body проверяется отдельно через `check_module`, но семантика захвата не анализируется).

**Влияние**: ошибки ownership в теле вложенных функций с параметрами не обнаруживаются (в текущей реализации они попросту пропускаются), информация о захвате также не генерируется. Если вложенная функция с параметрами использует переменные из внешней области, семантика ownership для неё не анализируется.

**Направление исправления**: унифицировать обработку Binding с параметрами и без, одновременно выполняя `check_function` + анализ захвата для их тел.
