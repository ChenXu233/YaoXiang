> Расположение реализации: `src/frontend/core/typecheck/layers/ownership.rs`
> Расположение тестов: `src/frontend/core/typecheck/layers/tests/ownership.rs`

## Дефекты корректности

- [x] ### 1. Псевдоним ref попадает в spawn без пометки об утечке (P0) — Исправлено (2026-06-15)

**Сценарий**:
```yaoxiang
shared = ref x
alias = shared       // shared Move → alias
spawn { use(alias) } // alias ∉ ref_vars → 漏标逃逸 → 选 Rc（非原子，跨线程不安全）
```

**Корневая причина**: `OwnershipChecker` отслеживает только имена переменных, напрямую присвоенных через `Expr::Ref` (набор `ref_vars`). Когда ref-переменная перемещается (Move) в промежуточную переменную, эта промежуточная переменная не изменяет `ref_vars`.

**Влияние**: ref, используемые через границу spawn, могут быть ошибочно скомпилированы в `RcNew`; неатомарный подсчёт ссылок при пересечении потоков может привести к гонке данных.

**Исправление**: в обработчиках `StmtKind::Var` и `BinOp::Assign`, когда правая часть представляет собой `Expr::Var(name)` и `name ∈ ref_vars`, левая цель добавляется в `ref_vars` (коммит `9029d5b`).

- [x] ### 2. Переменная, захваченная в spawn, остаётся доступной снаружи после Move (P1) — Исправлено (2026-06-16)

**Сценарий**:
```yaoxiang
shared = ref data
spawn { a = shared }  // spawn 体 walk（save/restore）→ shared 在体内 Moved → 外层恢复
use(shared)            // 外层 shared 仍 Alive——正确，但 spawn 体内 shared 已 Move
```

**Корневая причина**: `Expr::Spawn` использует save/restore, изменения владения внутри тела spawn не влияют на внешнюю область видимости. Это корректное архитектурное решение, но Move `a = shared` внутри тела spawn обнаруживается только во «временном walk» spawn. Если тело spawn выполняет Move `shared`, save/restore восстанавливает внешнее состояние, **но ничто не мешает внешнему коду продолжить использовать `shared` после spawn**.

**Влияние**: если spawn во время выполнения перемещает `shared` (например, `a = shared`), внешний код после spawn всё ещё может обращаться к `shared` — это может быть корректным в модели параллелизма YaoXiang (spawn получает независимую копию), однако семантика чётко не определена.

**Направление исправления**: требуется уточнить спецификацию языка: влияет ли семантика Move захваченных переменных в spawn на внешнюю область видимости. Если действует правило «spawn получает независимую копию» — текущее поведение корректно. Если действует правило «spawn потребляет владение» — необходимо убрать save/restore или ввести механизм Captures по аналогии с замыканиями.

## Компромиссы по точности

- [x] ### 3. Консервативное сообщение о конфликте для взаимоисключающих ветвей (P1) — Исправлено (2026-06-16)

**Сценарий**:
```yaoxiang
if cond {
    a = &mut x   // 分支 A
} else {
    b = &mut x   // 分支 B
}
// 理论上：A 和 B 互斥，不应冲突
// 实际：两个 WriteToken 先后创建 → 报 BorrowConflict
```

**Корневая причина**: ограничение архитектуры `NLL without fixpoint` — однопроходный обход AST не моделирует условия пути и не может отличить взаимоисключающие ветви от последовательного выполнения.

**Направление исправления**: требуется подключение медленного канала SMT на основе CFG (в настоящее время `smt_cut` реализован, но срабатывает только в сценариях `while + path_condition`). Распространение на ветви if/else требует проброса `path_condition` в обработчик Borrow.

- [ ] ### 4. Тип ref не распознаётся как Dup (P1)

**Сценарий**:
```yaoxiang
shared = ref x
a = shared    // Move——但 ref 理论上是 Dup 类型，应可复制
b = shared    // use after move——实际上应允许
```

**Корневая причина**: проверщик владения не знает, что `ref T` — это тип Dup (копируемый дескриптор со счётчиком ссылок). Логика Move в `StmtKind::Var` обрабатывает все типы единообразно.

**Влияние**: семантика значений типа ref получается строже, чем предполагалось, — нельзя «свободно копировать», как задумано в дизайне RFC-009.

**Направление исправления**: необходимо запрашивать тип переменной из `TypeEnvironment` и пропускать логику Move для типов Dup. Это согласуется с общим дизайном, требующим явного вызова `clone()`, — текущее консервативное поведение не более либерально, чем корректная семантика.

## Инфраструктура

- [ ] ### 5. Формат кодов ошибок не унифицирован (P2)

**Описание**: проверщик владения во фронтенде использует `DisproofModel.into_diagnostic()` → коды ошибок E2014–E2020. Унаследованный код в middle-слое `lifetime/error.rs` использует собственные `ValueState` и trait `Checker`. В настоящее время сосуществуют две системы.

**Направление исправления**: удалить `ValueState` и trait `Checker` из `error.rs` middle-слоя (осталось лишь 2 ссылки: `lifecycle.rs` и тест `cycle_check`), унифицировать с системой кодов ошибок фронтенда.

- [ ] ### 6. Вложенные функции с параметрами не анализируются (P2)

**Описание**: `StmtKind::Binding` выполняет анализ захвата только для замыканий, удовлетворяющих условию `params.is_empty() && !body.is_empty()`. Вложенные функции с параметрами возвращают `vec![]` (`check_module` отдельно проверяет тело, но не анализирует семантику захвата).

**Влияние**: ошибки владения внутри тела вложенной функции с параметрами не обнаруживаются (в настоящее время попросту пропускаются), информация о захвате не порождается. Если такая вложенная функция обращается к переменным внешней области, семантика владения не анализируется.

**Направление исправления**: унифицировать обработку Binding с параметрами и без, одновременно выполняя `check_function` и анализ захвата для их тел.
