# Ортогональность синтаксиса spawn for

## Проблема

Текущая конструкция `spawn for x in items { body }` представляет собой независимую комбинацию ключевых слов (`spawn` + `for`), несовместимую с шаблоном `spawn { }`, используемым в других конструкциях:

```yx
spawn { }              ← spawn с последующим блоком {}
spawn for { }          ← spawn с последующим for? Это образует составное ключевое слово spawn for
for ... spawn { }      ← spawn с последующим {}, тело for = spawn
```

Основной вопрос: **семантическая эквивалентность `spawn for` и `spawn { for }`, а также семантическое различие между `for spawn { }` и `spawn { for }`.**

## Семантика трёх форм

| Форма | Семантика | Декомпозиция DAG |
|------|----------|----------|
| `spawn for x in items { body }` | Каждая итерация — независимая параллельная задача (параллелизм по данным) | Компилятор разворачивает N итераций в N замыканий |
| `spawn { for x in items { body } }` | Цикл for как единое целое выполняется последовательно в параллельном блоке | Декомпозиция итераций for не выполняется |
| `for x in items spawn { body }` | Последовательная итерация for, на каждой итерации порождается задача spawn | На каждой итерации создаётся spawn |

Семантика всех трёх различна, их нельзя произвольно заменять друг другом.

## Границы декомпозиции DAG

Текущий DAG-анализ spawn (`analyze_spawn_body`) распознаёт **статические прямые подвыражения** внутри блока spawn — фиксированное число видимых компилятору выражений верхнего уровня. Число итераций цикла for является **динамическим** (определяется в runtime) и не может быть развёрнуто во время компиляции в список статических прямых подвыражений.

Поэтому `spawn { for }` не может быть автоматически эквивалентно `spawn for` — если только DAG-анализ не сможет обрабатывать динамическое разворачивание итераций, или язык не предоставит специальный механизм для маркировки независимости итераций.

## Идеал ортогональности

```
spawn модифицирует блок {}   → spawn { ... }
{} может содержать любые операторы → spawn { for ... { ... } }
```

В идеале `spawn { for }` должно автоматически распознавать циклы с независимыми между итерациями телами и разворачивать их в параллелизм по данным. Однако это вопрос компиляторных оптимизаций (аналог автовекторизации), а не дизайна синтаксиса.

## Отложенные вопросы

- [ ] Сохранять ли `spawn for` как отдельный синтаксис? Или свести его к `for ... spawn { }` или `spawn { for }`?
- [ ] Нужен ли специальный синтаксис для выражения параллелизма по данным, или достаточно средств стандартной библиотеки (например, `par_iter`)?
- [ ] Если сохранять отдельный синтаксис, каковы правила приоритета связывания `spawn for`?
- [ ] Поддерживает ли `spawn for` более сложные комбинации, такие как `else spawn`?
