# План реализации поддержки LSP

> **Задача**: Реализация поддержки протокола Language Server Protocol (LSP) для языка YaoXiang
> **На основе RFC**: RFC-017 Дизайн поддержки протокола языкового сервера (LSP)
> **Дата**: 2026-02-23
> **Статус**: В процессе
> **Целевая версия**: v0.7 - v0.9

---

## Обзор

Данный план, основанный на документе RFC-017, разбивает реализацию LSP на 6 этапов, включающих 20 подзадач. Каждая задача содержит подробные цели реализации, критерии приёмки и тестовые сценарии.

### Обзор зависимостей

```

                                │          │          │          │
```

### 0.4 Система кэширования документов (DocumentCache)

**Цели реализации**:
- Реализовать структуру `DocumentCache`, содержащую:
  - `version: u32` — номер версии документа LSP
  - `content: String` — текущее содержимое
  - `content_hash: u64` — хеш содержимого (быстрое сравнение)
  - `ast: Option<Ast>>` — кэшированный AST
- Реализовать детектирование инкрементных изменений (сравнение content_hash)
- Файловое кэширование: при изменении переразбирать файл целиком

**Критерии приёмки**:
- [x] DocumentCache корректно управляет номером версии
- [x] Хеш-детектирование быстро идентифицирует неизменённые документы
- [x] При изменении корректно выполняется переразборка
- [x] Разумное использование памяти (механизм очистки)

**Заметки по реализации**:
- Структура `DocumentCache`: version, content, content_hash, ast (`Option<Module>>`), file_path, dirty
- `DocumentStore` управляет всеми открытыми документами, `HashMap<String, DocumentCache>`, поддерживает ограничение ёмкости и автоочистку
- Хеш содержимого использует `DefaultHasher`, `update()` обновляет содержимое и инвалидирует AST-кэш только при изменении хеша
- Стратегия очистки: при превышении `max_documents` (по умолчанию 128) удаляются документы с наименьшим номером версии
- Полный набор тестов (7 юнит-тестов)
- Изменённый файл: `src/util/cache.rs`

**Тестовые сценарии**:

## Продвинутые возможности (последующие версии)

### Внутренние подсказки (Inlay Hints) ✅ Выполнено

**Приоритет**: P0

| Возможность | Цель реализации |
|-------------|-----------------|
| Подсказки значений констант | Отображение вычисленных на этапе компиляции констант (например, рядом с `const MAX = 100 + 200` отображается `300`)|
| Подсказки изменяемости | Отображение изменяемости переменной (например, `mut x` имеет явную метку)|
| Подсказки потребления владения | Отображение того, потребляется ли параметр функции |
| Подсказки вывода типов | Отображение выведенного конкретного типа (например, рядом с `x = vec()` отображается `Vec<i32>`)|

**Критерии приёмки**:
- [x] Все типы Inlay Hint отображаются корректно
- [x] Влияние на производительность < 50 мс

## Стратегия тестирования

### Юнит-тесты
- Независимые юнит-тесты для каждого модуля
- Использование mock для изоляции зависимостей

### Интеграционные тесты
- Тесты совместимости протокола LSP
- Интеграционные тесты с реальными IDE (VS Code, Neovim)

### Нагрузочные тесты
- Производительность парсинга больших файлов
- Время ответа автодополнения
- Время ответа переходов

## Риски и меры снижения

| Риск | Меры снижения |
|------|---------------|
| Проблемы производительности | Инкрементный парсинг, обработка в фоновых потоках |
| Использование памяти | LRU-кэш, отложенная загрузка |
| Совместимость протокола | Объявление поддерживаемой версии протокола |
