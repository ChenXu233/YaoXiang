# План реализации поддержки LSP

> **Задача**: Реализация поддержки протокола Language Server Protocol (LSP) для языка YaoXiang
> **На основе RFC**: RFC-017 Проектирование поддержки Language Server Protocol (LSP)
> **Дата**: 2026-02-23
> **Статус**: В процессе
> **Целевая версия**: v0.7 - v0.9

---

## Обзор

Данный план, основанный на документе RFC-017, разделяет реализацию LSP на 6 этапов, включающих 20 подзадач. Каждая задача содержит детальные цели реализации, критерии приёмки и тестовые сценарии.

### Обзор зависимостей

```

    │               │
    ▼               │
Этап 1 ──────────────┼──► Этап 2 ──► Этап 3 ──► Этап 4 ──► Этап 5
                    │         │         │         │
                    └─────────┴─────────┴─────────┘
                              (параллельная разработка)
```

### 0.1 Режим сбора ошибок

**Цели реализации**:
- Модифицировать модуль `src/frontend/typecheck/inference/`, возвращая `Result<Type, Vec<Error>>` вместо немедленного возврата при обнаружении ошибки
- Реализовать перечисление `ErrorKind`, содержащее `Error` (критическая ошибка), `Warning` (предупреждение), `Note` (дополнительная информация)
- Коллектор ошибок продолжает накапливать ошибки, возвращая все ошибки после завершения проверки

**Критерии приёмки**:
- [x] Проверщик типов возвращает все ошибки для отдельного файла (без короткого замыкания)
- [x] Ошибки содержат информацию об уровне Severity
- [x] При наличии Error отображаются ошибки в publishDiagnostics
- [x] При наличии только Warning компиляция продолжается с отображением предупреждений

**Описание реализации**:

- `TypeChecker::check_module_collect_all()` предоставляет точку входа для полного сбора ошибок в LSP
- Повторно используется существующее перечисление `Severity` (Error/Warning/Info/Hint)

### 0.4 Система кэширования документов (DocumentCache)

**Цели реализации**:
- Реализовать структуру `DocumentCache`, содержащую:
  - `version: u32` - версия документа LSP
  - `content: String` - текущее содержимое
  - `content_hash: u64` - хэш содержимого (для быстрого сравнения)
  - `ast: Option<Ast>>` - кэшированный AST
- Реализовать детектирование инкрементальных изменений (сравнение content_hash)
- Кэширование на уровне файлов: при изменении переразбирается весь файл

**Критерии приёмки**:
- [x] DocumentCache корректно управляет номером версии
- [x] Хэш-детектирование быстро идентифицирует неизменённые документы
- [x] При изменении корректно выполняется переразборка
- [x] Разумное использование памяти (есть механизм очистки)

**Описание реализации**:
- Структура `DocumentCache`: version, content, content_hash, ast (`Option<Module>>`), file_path, dirty
- `DocumentStore` управляет всеми открытыми документами, `HashMap<String, DocumentCache>`, с ограничением ёмкости и автоочисткой
- Хэш содержимого использует `DefaultHasher`, `update()` обновляет содержимое и инвалидирует AST-кэш только при изменении хэша
- Стратегия очистки: при превышении `max_documents` (по умолчанию 128) удаляются документы с наименьшим номером версии
- Полный тестовый набор (7 unit-тестов)
- Изменённый файл: `src/util/cache.rs`

**Тестовые сценарии**:

- [x] Тест применения инкрементальных изменений

### Подсказки в коде (Inlay Hints) ✅ Завершено

**Приоритет**: P0

| Возможность | Цель реализации |

| Подсказки значений констант | Отображение вычисленных при компиляции констант (например, рядом с `const MAX = 100 + 200` отображается `300`)|

- Независимые модульные тесты для каждого модуля

- Время ответа переходов

| Риск | Меры снижения |
|------|---------------|
