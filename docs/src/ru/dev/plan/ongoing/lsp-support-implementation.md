# План реализации поддержки LSP

> **Задача**: Реализация поддержки протокола языкового сервера (LSP) для YaoXiang
> **На основе RFC**: RFC-017 Дизайн поддержки протокола языкового сервера (LSP)
> **Дата**: 2026-02-23
> **Статус**: В процессе
> **Целевая версия**: v0.7 - v0.9

---

## Обзор

Данный план, основанный на документе RFC-017, разбивает реализацию LSP на 6 этапов, включающих 20 подзадач. Каждый этап содержит детальные цели реализации, критерии приёмки и тестовые задания.

### Обзор зависимостей


### 0.1 Режим сбора ошибок

**Цели реализации**:
- Модифицировать модуль `src/frontend/typecheck/inference/` для возврата `Result<Type, Vec<Error>>` вместо немедленного возврата при ошибке
- Реализовать перечисление `ErrorKind`, содержащее `Error` (критическая ошибка), `Warning` (предупреждение), `Note` (дополнительная информация)
- Коллектор ошибок продолжает накапливать ошибки, после проверки возвращает все ошибки единовременно

**Критерии приёмки**:
- [x] Проверщик типов возвращает все ошибки для одного файла (без раннего возврата)
- [x] Ошибки содержат информацию об уровне Severity
- [x] При наличии Error отображаются ошибки в publishDiagnostics
- [x] При наличии только Warning компиляция продолжается с отображением предупреждений

**Реализация**:
- В `StatementChecker` добавлен режим `collect_all_errors`, ошибки не возвращаются немедленно, а накапливаются в `collected_errors: Vec<Diagnostic>`
- `TypeChecker::check_module_collect_all()` предоставляет точку входа для полного сбора ошибок в LSP
- Повторно используется существующее перечисление `Severity` (Error/Warning/Info/Hint)
- Изменённые файлы: `src/frontend/typecheck/inference/statements.rs`, `src/frontend/typecheck/mod.rs`

**Тестовые задания**:

- [x] Тест накопления и единовременного возврата ошибок

- Пример: `x = ;` → `x = MissingExpression`

**Критерии приёмки**:
- [x] При синтаксической ошибке парсер генерирует placeholder-узлы вместо panic
- [x] Placeholder-узлы имеют корректную информацию о Span
- [x] Проверщик типов обрабатывает placeholder-узлы (выдаёт ошибку, но не panic)

**Реализация**:

- `ExpressionInferrer` и `StatementChecker` могут обрабатывать вариант Error (выдают ошибку `invalid_syntax`, но не panic)

### 0.4 Система кэширования документов (DocumentCache)

**Цели реализации**:
- Реализовать структуру `DocumentCache`, содержащую:
  - `version: u32` — номер версии документа LSP
  - `content: String` — текущее содержимое
  - `content_hash: u64` — хеш содержимого (для быстрого сравнения)
  - `ast: Option<Ast>` — кэшированный AST
- Реализовать детектирование инкрементных изменений (сравнение content_hash)
- Кэширование на уровне файлов: при изменении переразбирается весь файл

**Критерии приёмки**:
- [x] DocumentCache корректно управляет номером версии
- [x] Хеш-детектирование быстро идентифицирует неизменённые документы
- [x] При изменении корректно выполняется переразборка
- [x] Разумное использование памяти (есть механизм очистки)

**Реализация**:
- Структура `DocumentCache`: version, content, content_hash, ast (`Option<Module>`), file_path, dirty
- `DocumentStore` управляет всеми открытыми документами, `HashMap<String, DocumentCache>`, поддерживает ограничение ёмкости и автоочистку
- Хеш содержимого использует `DefaultHasher`, `update()` обновляет содержимое и инвалидирует кэш AST только при изменении хеша
- Стратегия очистки: при превышении `max_documents` (по умолчанию 128) удаляются документы с наименьшим номером версии
- Полный тестовый набор (7 юнит-тестов)
- Изменённый файл: `src/util/cache.rs`

**Тестовые задания**:
- [x] Тест инкремента номера версии
- [x] Тест точности хеш-детектирования
- [x] Тест применения инкрементных изменений

## Стратегия тестирования

### Юнит-тесты
- Независимые юнит-тесты для каждого модуля
- Использование mock для изоляции зависимостей

### Интеграционные тесты
- Тесты совместимости протокола LSP
- Интеграционные тесты с реальными IDE (VS Code, Neovim)

### Нагрузочные тесты
- Производительность парсинга больших файлов
- Время ответа дополнения
- Время ответа перехода

## Риски и меры снижения

| Риск | Меры снижения |
|------|----------|
| Проблемы с производительностью | Инкрементный парсинг, обработка в фоновых потоках |
| Использование памяти | LRU-кэш, отложенная загрузка |
| Совместимость протокола | Объявление поддерживаемой версии протокола |
