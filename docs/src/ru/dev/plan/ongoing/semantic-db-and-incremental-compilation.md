# План реализации семантического информационного хаба и инкрементной компиляции

> **Задача**: Реализовать семантический информационный хаб, обеспечивающий возможности семантической подсветки LSP, инкрементной компиляции и предупреждений о мёртвом коде
> **На основе RFC**: Данный план предназначен для разработки новой функциональности
> **Связанный RFC**: [RFC-008: Модель параллелизма во время выполнения](../design/rfc/accepted/008-runtime-concurrency-model.md) — DAG-параллелизм относится к рантайму и не входит в данный план
> **Дата**: 2026-02-23
> **Статус**: Этап 1 + Этап 2 + Этап 3 завершены
> **Целевая версия**: v0.10 - v0.11

---

## Обзор

Данный план разбивает реализацию семантического информационного хаба на 3 основных этапа. Ключевая идея — **однократный обход с многократным использованием**:

1. **Сбор семантической информации выполняется на этапе проверки типов** (а не отдельный обход AST на уровне LSP)
2. Собранная семантическая информация одновременно используется для LSP-подсветки, инкрементной компиляции и анализа мёртвого кода

> **Важное уточнение**:
> - **DAG-параллелизм** относится к характеристикам времени выполнения (RFC-008) и не входит в данный план
> - **Параллельная компиляция модульных зависимостей** является характеристикой системы сборки и представляет собой две разные концепции с рантайм-подобным DAG
> - Сбор семантической информации должен выполняться на этапе проверки типов, LSP просто повторно использует результаты, а не реализует отдельный обход

```rust
pub struct SemanticDB {
    // Путь к файлу -> семантическая информация этого файла
    by_file: HashMap<String, FileSemanticInfo>,
    // Имя символа -> все позиции определения
    symbol_defs: HashMap<String, Vec<SymbolLocation>>,
    // Имя символа -> все позиции ссылок
    symbol_refs: HashMap<String, Vec<SymbolLocation>>,

// Семантическая информация одного файла
pub struct FileSemanticInfo {
    pub file_path: String,
    pub tokens: Vec<SemanticToken>,
    pub scopes: Vec<ScopeInfo>,
}

// Семантический токен (используется структура + перечисление типов, а не варианты перечисления из плана)
pub struct SemanticToken {
    pub name: String,
    pub token_type: SemanticTokenType,
    pub modifiers: Vec<SemanticTokenModifier>,
    pub span: Span,
}

pub enum SemanticTokenType {
    Function, Type, Variable, Property, Method,
    Namespace, Parameter, LocalVariable, TypeParameter,
    Keyword, String, Number,
}

pub enum SemanticTokenModifier {
    Declaration, Readonly, Mutable, Public, Generic,
}

// Информация о области видимости
pub struct ScopeInfo {
    pub span: Span,
    pub parent: Option<usize>,  // Индекс родительской области видимости
    pub symbols: Vec<String>,   // Символы внутри области видимости
    pub kind: ScopeKind,        // Global, Function, Block, Lambda
}
```

### 1.2 Интеграция коллектора семантики в TypeCheck

**Проектное решение**: Сбор семантики **не должен** реализовываться отдельно на уровне LSP, а должен выполняться на этапе проверки типов.

**Причины**:
- Проверка типов уже выполняет обход AST и знает все позиции определений и ссылок символов
- Отдельная реализация SemanticCollector в LSP = повторный обход + поддержка двух логик
- **Хороший стиль**: однократный обход, многократное использование

**Цели реализации**:
- Расширить функциональность сбора семантики в модуле `src/frontend/typecheck/`
- При проверке типов одновременно генерировать данные `SemanticDB`
- Уровень LSP напрямую запрашивает и повторно использует, без повторного обхода AST

**Правила сбора при проверке типов** (результаты этапа):
```
StmtKind::Fn        → SemanticTokenType::Function (определение)
StmtKind::TypeDef   → SemanticTokenType::Type (определение)
StmtKind::Var       → SemanticTokenType::Variable (определение)
StmtKind::MethodBind→ SemanticTokenType::Method (определение)
StmtKind::Use       → SemanticTokenType::Namespace (ссылка)
Param               → SemanticTokenType::Parameter (определение)
Expr::Var           → SemanticTokenType::Variable (ссылка)
Expr::Call          → SemanticTokenType::Function (ссылка)
Expr::FieldAccess   → SemanticTokenType::Property (ссылка)
Expr::Cast          → SemanticTokenType::Type (ссылка)
```

**Критерии приёмки**:
- [x] Этап проверки типов генерирует SemanticDB
- [x] LSP может запрашивать семантическую информацию от проверки типов
- [x] Устранён повторный обход AST на уровне LSP

## Этап 4: Предупреждения о мёртвом коде (интеграция в процесс компиляции)

> **Целевая версия**: v0.11
> **Зависимости**: Завершение этапа 1 (семантическая информация этапа проверки типов)

> **Примечание**: Предупреждения о мёртвом коде зависят от информации о ссылках на символы на этапе проверки типов и являются функцией анализа времени компиляции, а не времени выполнения.

> **Корректировка архитектуры**: Анализ мёртвого кода интегрирован в этап проверки типов, поскольку оба требуют обхода AST/SemanticDB

### 4.1 Анализатор мёртвого кода

**Цели реализации**:
- Реализовать структуру `DeadCodeAnalyzer`
- Анализировать неиспользуемые экспортированные символы
- Анализировать неиспользуемые импорты
- Генерировать предупреждающие сообщения

**Проектное решение**: Анализ мёртвого кода должен выполняться на этапе **проверки типов**, потому что:
- Проверка типов уже знает все определения и ссылки символов
- Не требуется дополнительный обход AST
- Семантическая информация уже предоставлена через SemanticDB

**Правила анализа**:
```
1. Собрать все точки входа (main, pub-функции)
2. Начиная с точек входа, пометить все достижимые символы
3. Недостижимые экспортированные символы -> предупреждение
4. Неиспользуемые импорты -> предупреждение
```

**Структуры данных**:
```rust
pub struct DeadCodeAnalyzer {
    // Точки входа
    entry_points: HashSet<SymbolId>,
    // Все определения символов
    all_defs: HashMap<SymbolId, SymbolDef>,
    // Ссылки на символы (получены из SemanticDB)
    references: HashMap<SymbolId, Vec<Location>>,
    // Список импортов
    imports: Vec<ImportInfo>,
}

pub struct SymbolDef {
    pub name: String,
    pub kind: SymbolKind,
    pub location: Location,
    pub is_exported: bool,
}
```

**Критерии приёмки**:
- [x] Обнаружение неиспользуемых экспортированных функций
- [x] Обнаружение неиспользуемых экспортированных типов
- [x] Обнаружение неиспользуемых импортов
- [x] Формат предупреждающих сообщений корректен

## Пояснение о DAG-параллелизме

**DAG-параллелизм не входит в данный план**, и вот почему:

| Концепция | Принадлежность | Описание |
|-----------|----------------|----------|
| **DAG времени выполнения** | RFC-008 Runtime | Граф зависимостей ленивых вычислений, управляет调度кой задач времени выполнения |
| **Граф модульных зависимостей** | Этап 3 данного плана | Модульные зависимости на уровне компилятора, используются для инкрементной компиляции |
| **Параллельная компиляция на уровне модулей** | Система сборки | Реализуется на основе графа зависимостей этапа 3, не относится к LSP |

**Правильное расположение**:
- Параллелизм DAG времени выполнения → см. [RFC-008: Модель параллелизма во время выполнения](../design/rfc/accepted/008-runtime-concurrency-model.md)
- Граф модульных зависимостей → Этап 3 данного плана (завершён/в процессе)
- Параллельная компиляция на уровне модулей → должна реализовываться как функция системы сборки, может быть основана на графе зависимостей этапа 3

### Принципы проектирования

1. **Однократный обход**: Этап проверки типов одновременно генерирует семантическую информацию без повторного обхода AST
2. **Многократное использование**: LSP-подсветка, инкрементная компиляция и анализ мёртвого кода совместно используют одни и те же данные
3. **Хороший стиль**: Не добавлять ненужные уровни абстракции ради "разделения"
