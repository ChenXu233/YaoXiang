# План реализации семантической информационной платформы и инкрементной компиляции

> **Задача**: Реализовать семантическую информационную платформу, обеспечивающую возможности семантической подсветки LSP, инкрементной компиляции и предупреждений о мёртвом коде
> **На основе RFC**: Данный план предназначен для проектирования новой функциональности
> **Связанный RFC**: [RFC-008: Модель параллелизма во время выполнения](../design/rfc/accepted/008-runtime-concurrency-model.md) — DAG-параллелизм относится к рантайму, не входит в данный план
> **Дата**: 2026-02-23
> **Статус**: Этап 1 + Этап 2 + Этап 3 завершены
> **Целевая версия**: v0.10 - v0.11

---

## Обзор

Данный план декомпозирует реализацию семантической информационной платформы на 3 основных этапа. Ключевая идея — **один обход, множественное использование**:

1. **Сбор семантики выполняется на этапе typecheck** (а не отдельный обход AST на уровне LSP)
2. Собранная семантическая информация одновременно используется для LSP семантической подсветки, инкрементной компиляции и анализа мёртвого кода

> **Важное уточнение**:
> - **DAG-параллелизм** относится к характеристикам рантайма (RFC-008), не входит в данный план
> - **Параллельная компиляция на основе графа зависимостей модулей** — это характеристика системы сборки, отдельная от DAG рантайма
> - Сбор семантики должен выполняться на этапе typecheck, LSP просто переиспользует данные, а не реализует собственный независимый обход

```rust
    by_file: HashMap<String, FileSemanticInfo>,
    // Имя символа -> все позиции определения

    // Имя символа -> все позиции ссылок
    symbol_refs: HashMap<String, Vec<SymbolLocation>>,
}

// Семантическая информация одного файла
pub struct FileSemanticInfo {
    pub file_path: String,
    pub tokens: Vec<SemanticToken>,
    pub scopes: Vec<ScopeInfo>,
}

// Семантический Token (используется структура + enum типов вместо вариантов перечисления из плана)
pub struct SemanticToken {
    pub name: String,
    pub token_type: SemanticTokenType,
    pub modifiers: Vec<SemanticTokenModifier>,
    pub span: Span,
}

pub enum SemanticTokenType {
    Function, Type, Variable, Property, Method,
    Namespace, Parameter, LocalVariable, TypeParameter,
    Keyword, String, Number,
}

pub enum SemanticTokenModifier {
    Declaration, Readonly, Mutable, Public, Generic,
}

// Информация о области видимости
pub struct ScopeInfo {
    pub span: Span,
    pub parent: Option<usize>,  // Индекс родительской области видимости
    pub symbols: Vec<String>,   // Символы внутри области видимости
    pub kind: ScopeKind,        // Global, Function, Block, Lambda
}
```

### 1.2 Интеграция сборщика семантики в TypeCheck

**Проектное решение**: Сбор семантики **не должен** быть реализован отдельно на уровне LSP, а должен выполняться на этапе typecheck.

**Причины**:
- typecheck уже обходит AST и знает все позиции определений и ссылок символов
- Реализация SemanticCollector отдельно в LSP = повторный обход + поддержка двух наборов логики
- **Хороший стиль**: один обход, множественное использование

**Цели реализации**:
- Расширить функциональность сбора семантики в модуле `src/frontend/typecheck/`
- При проверке типов одновременно производить данные `SemanticDB`
- Уровень LSP напрямую запрашивает и переиспользует данные, без повторного обхода AST

**Правила сбора от typecheck** (результаты этапа):
```
StmtKind::Fn        → SemanticTokenType::Function (определение)
StmtKind::TypeDef   → SemanticTokenType::Type (определение)
StmtKind::Var       → SemanticTokenType::Variable (определение)
StmtKind::MethodBind→ SemanticTokenType::Method (определение)
StmtKind::Use       → SemanticTokenType::Namespace (ссылка)
Param               → SemanticTokenType::Parameter (определение)
Expr::Var           → SemanticTokenType::Variable (ссылка)
Expr::Call          → SemanticTokenType::Function (ссылка)
Expr::FieldAccess   → SemanticTokenType::Property (ссылка)
Expr::Cast          → SemanticTokenType::Type (ссылка)
```

**Критерии приёмки**:
- [x] Этап typecheck производит SemanticDB
- [x] LSP может запрашивать семантическую информацию от typecheck
- [x] Устранён повторный обход AST на уровне LSP

## Этап 4: Предупреждения о мёртвом коде (интеграция в процесс компиляции)

> **Примечание**: Предупреждения о мёртвом коде зависят от информации о ссылках на символы на этапе typecheck, это функция анализа времени компиляции, не рантайма.

> **Корректировка архитектуры**: Анализ мёртвого кода интегрирован в этап typecheck, так как оба требуют обхода AST/SemanticDB

### 4.1 Анализатор мёртвого кода

**Цели реализации**:
- Реализовать структуру `DeadCodeAnalyzer`
- Анализ неиспользуемых экспортируемых символов
- Анализ неиспользуемых импортов
- Генерация предупреждающих сообщений

**Проектное решение**: Анализ мёртвого кода должен выполняться на этапе **typecheck**, потому что:
- typecheck уже знает все определения и ссылки символов
- Не требуется дополнительный обход AST
- Семантическая информация уже предоставляется через SemanticDB

**Правила анализа**:
```
1. Собрать все точки входа (main, pub функции)
2. Начиная с точек входа, пометить все достижимые символы
3. Недостижимые экспортируемые символы -> предупреждение
4. Неиспользуемые импорты -> предупреждение
```

**Структуры данных**:
```rust
pub struct DeadCodeAnalyzer {
    // Точки входа
    entry_points: HashSet<SymbolId>,
    // Все определения символов
    all_defs: HashMap<SymbolId, SymbolDef>,
    // Ссылки на символы (получены из SemanticDB)
    references: HashMap<SymbolId, Vec<Location>>,
    // Список импортов
    imports: Vec<ImportInfo>,
}

pub struct SymbolDef {
    pub name: String,
    pub kind: SymbolKind,
    pub location: Location,
    pub is_exported: bool,
}
```

**Критерии приёмки**:
- [x] Обнаружение неиспользуемых экспортируемых функций
- [x] Обнаружение неиспользуемых экспортируемых типов

- [x] Формат предупреждающих сообщений корректен

### 4.2 Интеграция системы предупреждений

**Цели реализации**:
- Интегрировать обнаружение мёртвого кода в процесс компиляции
- Публикация предупреждений через событие `CompilationWarning`
- Поддержка различных форматов вывода (терминал, JSON)

**Формат предупреждения**:
```
warning: unused function `dead_function`
  --> src/utils.yx:10:1
   |
10 | fn dead_function() { }
   | ^^^^^^^^^^^^^^^^^^^^^^
   |
   = note: function is never used
```

**Критерии приёмки**:
- [x] Предупреждения о мёртвом коде корректно триггерятся
- [x] Информация о позиции в предупреждениях точная
- [x] Предупреждения конфигурируемы (включение/выключение)
- [x] Формат вывода в терминале эстетичен

**Тестовые проекты**:
- [x] Тест триггера предупреждения
- [x] Тест позиции предупреждения
- [x] Тест конфигурации
- [x] Тест формата вывода

## Пояснение о DAG-параллелизме

**DAG-параллельная компиляция НЕ включена в данный план**, по следующим причинам:

| Концепция | Принадлежность | Описание |
|------|------|------|
| **DAG рантайма** | RFC-008 Runtime | Граф зависимостей ленивых вычислений, управляет调度кой задач во время выполнения |
| **Граф зависимостей модулей** | Этап 3 данного плана | Зависимости модулей на уровне компилятора, для инкрементной компиляции |
| **Параллельная компиляция на уровне модулей** | Система сборки | Реализуется на основе графа зависимостей этапа 3, не относится к LSP |

**Корректное расположение**:
- DAG-параллелизм во время выполнения → см. [RFC-008: Модель параллелизма во время выполнения](../design/rfc/accepted/008-runtime-concurrency-model.md)
- Граф зависимостей модулей → Этап 3 данного плана (завершён/в процессе)
- Параллельная компиляция на уровне модулей → должна быть реализована как функция системы сборки, может быть основана на графе зависимостей этапа 3

### Принципы проектирования

1. **Один обход**: Этап typecheck одновременно производит семантическую информацию, без повторного обхода AST
2. **Множественное использование**: Семантическая подсветка LSP, инкрементная компиляция и анализ мёртвого кода совместно используют одни и те же данные
3. **Хороший стиль**: Не добавлять ненужные абстракции ради "разделения"
