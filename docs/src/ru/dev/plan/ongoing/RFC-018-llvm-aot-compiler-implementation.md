# RFC-018 План реализации LLVM AOT компилятора и L3 прозрачной конкурентности (ленивое планирование DAG)

> **⚠️ Примечание о выравнивании**: Данный документ основан на старой модели конкурентности (аннотации `@block`/`@eager`/`@auto`, `Send`/`Sync` trait, уровни L1/L2/L3) и был заменён [RFC-024 Новая модель конкурентности](/design/rfc/accepted/024-concurrency-model.md). Документ может продвигаться дальше только после выравнивания с RFC-024. Текущая модель конкурентности использует блок `spawn {}` в качестве единственной параллельной примитивы, без аннотаций, без Send/Sync.

> **Задача**: Реализация LLVM AOT бэкенда + рантайм DAG планировщика (~~реализация трёх стратегий планирования `@auto/@eager/@block`~~ устарело, требуется выравнивание с RFC-024)  
> **На основе RFC**: RFC-018 (черновик)  
> **Зависимые RFC**: RFC-001 (модель spawn и обработка ошибок), RFC-008 (трёхуровневый рантайм), RFC-009 (ownership/Arc)  
> **Дата**: 2026-03-10  
> **Статус**: В процессе  
> **Целевые этапы**:  
> - M1: LLVM AOT (компилируемый исполняемый файл, последовательный)  
> - M2: Метаданные DAG + однопоточное планирование (Standard Runtime, num_workers=1)  
> - M3: Многопоточное параллельное планирование + контроль гранулярности (Full Runtime, num_workers>1)  
> - M4: Ленивое планирование (Lazy Task Creation) + **абстракция побочных эффектов через типы ресурсов (Resource)** + **распространение ошибок/граф ошибок** + сквозная поддержка аннотаций

---

## Резюме (замкнутый цикл реализации)

- Добавление LLVM бэкенда в `yaoxiang` (feature gate), компилирующего `BytecodeModule` в машинный код (COFF/ELF/Mach-O) с возможностью загрузки и выполнения в рантайме.
- Введение **стабильного ABI**: код, генерируемый AOT, взаимодействует с рантаймом через `extern "C"` интерфейс `RtValue/RtContext`, что позволяет избежать проблем с нестабильным ABI Rust enum.
- Реализация ядра RFC-018: **DAG в пределах тела функции** + **ленивое планирование (Lazy Scheduling)**. Параллельность/последовательность определяются совместно **рёбрами DAG (Data/Control/Spawn)** и **правилами типов ресурсов (Resource)**; ошибки распространяются вдоль рёбер зависимостей согласно RFC-001 и могут формировать граф ошибок.

---

## Изменения публичного интерфейса/поведения (наблюдаемые извне)

1. **Cargo features**
   - Добавление фичи `llvm-aot`: включает зависимости LLVM/inkwell и AOT бэкенд; по умолчанию выключена (гарантирует сборку без окружения LLVM).
2. **CLI**
   - В `yaoxiang run` добавлен `--backend {interpreter,llvm}` (по умолчанию interpreter).
   - Опционально: добавление `--runtime {embedded,standard,full}` и `--workers <N>` для управления уровнем рантайма и степенью параллелизма (RFC-008):
     - `--runtime embedded`: немедленное выполнение (без DAG, без возможностей планировщика, подходит для встраиваемых/минималистичных сценариев)
     - `--runtime standard`: DAG + Scheduler (num_workers=1 для асинхронности; >1 для параллелизма)
     - `--runtime full`: standard + WorkStealer (продвинутые возможности, опционально)
3. **ABI рантайма (внутреннее, но межмодульное)**
   - Добавлены `RtValue` (`#[repr(C)]`) и `RtContext` (содержит только указатели/базовые типы) в качестве границы взаимодействия между AOT и рантаймом.

## Ключевые проектные ограничения (выровнены с RFC-001 / RFC-008 / RFC-018)

### A. Семантика конкурентности (L1/L2/L3 — лишь ментальная модель)

- **L3 (по умолчанию / @auto)**: прозрачная конкурентность; строится DAG; при встрече вызова сначала возвращается "отложенно вычисляемое" значение, **вычисление запускается только в момент, когда значение требуется**.
- **L1 (@block)**: предоставляется стандартной библиотекой (RFC-008), семантика "принудительное немедленное вычисление", не попадает в ленивую очередь DAG; используется преимущественно для отладки и критически упорядоченных участков.
- **L2 (spawn)**: **допускается только в области видимости @block** (RFC-001/008), применяется для вставки конкурентности в синхронный код; относится к возможностям Full Runtime.

### B. Три уровня рантайма (RFC-008)

- **Embedded Runtime**: немедленное выполнение; возможен полный отказ от построения DAG (экономия памяти/времени запуска); используется в ограниченных средах.
- **Standard Runtime**: ядром являются DAG + Scheduler (ленивое вычисление естественно поддерживает асинхронность/параллелизм).
- **Full Runtime**: поверх standard добавляется WorkStealer, а также возможности уровня стандартной библиотеки: `@block` / `spawn` и др.

### C. Область построения DAG и память (RFC-001/018)

- DAG строится только **в пределах одного тела функции/блока**; не раскрывается рекурсивно в тела вызываемых функций (во избежание взрывного роста графа ошибок и узлов DAG).
- Метаданные DAG должны содержать **стабильные ID узлов/рёбер** и **Span** (для распространения ошибок и позиционирования в графе ошибок).

### D. Абстракция побочных эффектов (RFC-001: типы ресурсов)

- Не вводится дополнительная "явная система аннотаций побочных эффектов"; побочные эффекты унифицируются как **операции с ресурсами**:
  - Вызовы функций, у которых типы параметров содержат `Resource` (или производный тип ресурса), считаются операциями с ресурсами;
  - Операции над одним и тем же ресурсом автоматически образуют **ControlEdge** (последовательное выполнение); операции над разными ресурсами могут выполняться параллельно;
  - Когда невозможно статически определить, тот же это ресурс или нет, по умолчанию используется консервативная последовательность (впоследствии возможно введение явной подсказки unsafe-параллелизма как расширения).

### E. Распространение ошибок (RFC-001)

- Ошибки распространяются вверх вдоль рёбер зависимостей DAG (независимо от фактического порядка параллельного выполнения), и путь распространения записывается для графа ошибок.

### 0.3 Фиксация ограничений реализации модели spawn (выравнивание с RFC-001/008)

**Цель**
- Зафиксировать и закрепить следующие ограничения реализации (в комментариях к коде/документации разработчика и тестовых сценариях):
  - `spawn` допускается только в области видимости `@block` (защита на этапах разбора/проверки типов/IR).
  - Семантика `@block` — "немедленное вычисление", обеспечивается возможностями стандартной библиотеки (для MVP можно реализовать встроенно в компилятор, но сохранить интерфейс для будущей передачи в стандартную библиотеку).
  - DAG строится только в пределах тела функции; должен нести стабильные `node_id` и `span` (основа для распространения ошибок/графа ошибок).
  - Типы ресурсов (Resource) управляют генерацией ControlEdge, без введения дополнительной видимой пользователю системы аннотаций эффектов.
  - **Ограничения параллельной безопасности (RFC-001/009)**: параллельное выполнение между потоками разрешено только если захваченные/возвращаемые значения узла удовлетворяют `Send + Sync` (или эквивалентному ограничению на уровне языка); в противном случае необходимо понижение до последовательного режима (либо фиксация на одном worker).

**Критерии приёмки**
- [ ] Компилятор в недопустимых сценариях `spawn` выдаёт явную ошибку (включая Span).
- [ ] Семантические различия `@block/@eager/@auto` наблюдаемы и тестируемы на минимальных примерах.
- [ ] Документация (данный план) согласована с ключевыми решениями RFC-001/008/018, без противоречивых пунктов.

**Тестовые пункты**
- [ ] Тест ограничения позиции `spawn`: использование `spawn` вне `@block` должно приводить к ошибке.
- [ ] Тест области DAG: подтвердить, что DAG не раскрывается между телами функций (количество узлов не зависит от глубины вызовов).
- [ ] Тесты ограничений Send/Sync:
  - Захват `spawn` значения, не являющегося `Send`, должен приводить к ошибке (с указанием Span).
  - В `@auto` узлы со значениями, не являющимися `Send + Sync`, не должны планироваться между потоками (можно проверить статистикой `std.concurrent.thread_id`).

## Этап 2: Стабильный ABI (RtValue/RtContext) и Runtime API (3–5 дней)

> Этот этап является ключом к "исполнимости кода, генерируемого LLVM": сначала необходимо стабилизировать представление значений на границе.

### 2.1 Определение `RtValue` (стабильный C ABI)

**Цель**
- Определить `#[repr(C)] struct RtValue { tag: u8, _pad: [u8; 7], payload: u64 }` (или 16-байтовая структура, для простоты выравнивания).
- Определить `#[repr(u8)] enum RtTag { Unit, Bool, Int, Float, Handle, Async, Error }` (минимальный набор; возможно расширение).
- Соглашения:
  - Int: `payload` = биты `i64`
  - Float: `payload` = биты `f64`
  - Bool: 0/1
  - Handle: `payload` = `usize` (расширяется до u64)
  - Async: `payload` = `TaskId` (u64)
  - Error: `payload` = код ошибки или указатель (для MVP можно ограничиться кодом ошибки)

**Критерии приёмки**
- [ ] `RtValue` можно безопасно конструировать/читать внутри Rust (без UB), имеется вывод `Debug` и базовые функции-ассерты.
- [ ] Согласовано с LLVM IR: в inkwell можно создать struct type той же раскладки (порядок/размер полей совпадают).

**Тестовые пункты**
- [ ] Roundtrip `RtValue`: юнит-тесты кодирования/декодирования для int/float/bool/unit.
- [ ] Тест размера ABI: `size_of::<RtValue>()` и `align_of::<RtValue>()` фиксированы (жёсткие assert против случайных изменений).

---

### 2.2 Определение `RtContext` (контекст рантайма)

**Цель**
- Определить `#[repr(C)] struct RtContext`, содержащий только указатели/целые:
  - `heap: *mut Heap`
  - `ffi: *const FfiRegistry`
  - `scheduler: *mut DynScheduler` (или указатель на конкретную реализацию)
  - `max_parallelism: usize`
  - `error_graph: *mut ErrorGraph` (опционально: для записи распространения ошибок RFC-001; для MVP может быть null)
  - Зарезервированные поля (версия/флаги), но с сохранением минимализма (KISS).

**Критерии приёмки**
- [ ] `RtContext` может быть сконструирован интерпретатором/LLVM executor и передан сгенерированному коду.
- [ ] Не содержит полей Rust с нестабильной раскладкой (запрещено непосредственное встраивание значений `Heap`/`FfiRegistry`).

**Тестовые пункты**
- [ ] Тест безопасности памяти при конструировании/уничтожении `RtContext` (без реального LLVM).

---

### 2.3 Runtime C API: минимальный набор функций, вызываемых из сгенерированного кода

**Цель**
- Предоставить функции `#[no_mangle] extern "C"` (единый префикс имён `yx_rt_*`), для MVP минимум:
  - `yx_rt_force(ctx: *mut RtContext, v: RtValue) -> RtValue`
  - `yx_rt_lazy_call(ctx, callee: *const u8 /*указатель на функцию*/, args: *const RtValue, argc: usize, node_id: u32, span_id: u32) -> RtValue`
  - `yx_rt_native_call(ctx, name_ptr: *const u8, name_len: usize, args: *const RtValue, argc: usize, node_id: u32, span_id: u32) -> RtValue`
  - `yx_rt_panic(code: u32)` или `yx_rt_trap(msg_ptr, len)` (отладочное)
- Ограничение: код, генерируемый AOT, **взаимодействует только через указанный API**, не обращаясь напрямую к структурам Rust.

**Критерии приёмки**
- [ ] Runtime API компилируется и без LLVM (управляется feature `llvm-aot`: API может быть постоянным или доступным только под feature, но должен быть тестируемым).
- [ ] `yx_rt_native_call` способен вызвать handler из `FfiRegistry` (для MVP поддерживаются только параметры и возвращаемые значения Int/Float/Bool/Unit; иначе возвращается Error RtValue), и при неудаче записывает `node_id/span_id` в граф ошибок (если включён).

**Тестовые пункты**
- [ ] Чистый Rust юнит-тест: прямой вызов `yx_rt_native_call`, проверка пути `std.io.println` (или собственной зарегистрированной функции).
- [ ] Тест ошибочного пути: передача несуществующего native-имени, возврат RtValue `Error`, конвертируемой в `ExecutorError::FunctionNotFound`.

## Этап 3: Инфраструктура LLVM Codegen (2–3 дня)

### 3.1 Инициализация контекста/модуля/TargetMachine LLVM

**Цель**
- `context.rs`: инкапсулировать жизненный цикл `Context/Module/Builder` из inkwell.

- Поддержка вывода:
  - LLVM IR (`.ll`) — для отладки

**Критерии приёмки**
- [ ] Для произвольного пустого `BytecodeModule` можно сгенерировать LLVM Module, содержащий `main` (хотя бы с телом, возвращающим Unit).
- [ ] IR проходит верификацию (вызов LLVM verify; при ошибке — читаемое сообщение).

**Тестовые пункты**
- [ ] Юнит: генерация минимального модуля с прохождением verify.
- [ ] Snapshot-тест (опционально): строковые assert на ключевые фрагменты `.ll` (избегая хрупких полных снимков).

### 3.2 `TypeMap`: YaoXiang Type → LLVM Type (MVP)

**Цель**
- `types.rs`: реализовать `fn llvm_type(yao_type: &Type) -> BasicTypeEnum`, первоначально охватывая:
  - `Int` → i64
  - `Float` → f64
  - `Bool` → i1
  - `()`/Void → void (либо унифицированный возврат `RtValue(Unit)`)
- Выбор стратегии (для сокращения площади ABI): **все функции единообразно возвращают `RtValue`** (а не по своему типу), чтобы codegen и планировщик/FFI обрабатывали унифицированно; информация о типах используется для статических проверок и генерации логики конструирования/деконструирования `RtValue`.

**Критерии приёмки**
- [ ] `TypeMap` стабильно отображает указанные типы, и сигнатура функции в LLVM IR согласована: `fn(*mut RtContext, *const RtValue, usize) -> RtValue`.

**Тестовые пункты**
- [ ] Юнит-тест `TypeMap`: по заданным `Type::Int/Float/Bool/Void` LLVM-типы генерируются успешно.
- [ ] Сигнатура сгенерированной функции обнаруживается в LLVM module и assert-ится соответствие параметров/возвращаемого типа.

## Этап 4: MVP трансляции инструкций (5–8 дней)

### 4.1 Отображение регистров в значения LLVM (минимальная SSA-изация)

## Этап 6: Генерация метаданных DAG (4–7 дней)

### 6.1 Определение `DAGMetadata` (версионированное)

**Цель**
- `dag.rs` определяет:
  - `dag_version: u32`
  - `nodes: Vec<DAGNode>` (несёт `node_id` и `span_id` для распространения ошибок)
  - `edges: Vec<DAGEdge>` (с типом ребра: Data/Control/Spawn)
- `DAGEdge` минимум содержит:
  - `from: u32`
  - `to: u32`
  - `kind: EdgeKind { Data, Control, Spawn }`
- Правила конфликтов/планирования (RFC-001):
  - DataEdge + DataEdge: могут выполняться параллельно (при отсутствии других зависимостей)
  - Любая комбинация, содержащая ControlEdge: должна быть сериализована (сохранение порядка)
- `DAGNode` минимум содержит:
  - `node_id: u32` (уникален внутри функции)
  - `ip: u32` (позиция call-инструкции)
  - `strategy: ScheduleStrategy { Serial, Eager, Lazy }` (получено из аннотации или стратегии по умолчанию)
  - `span_id: u32` (позиционирование и граф ошибок)
  - `thread_safety: ThreadSafety { SendSync, LocalOnly }` (выводится системой типов; узлы `LocalOnly` запрещено планировать между потоками)
- Соглашение: узел описывает только "планируемую точку вызова"; аргументы захватываются в рантайме при `yx_rt_lazy_call` (во избежание сложности статического вычисления выражений).

**Критерии приёмки**
- [ ] `DAGMetadata` сериализуется/десериализуется (используя имеющиеся `ron` или `serde_json`).
- [ ] При несовпадении `dag_version` загрузка возвращает ошибку.

**Тестовые пункты**
- [ ] Юнит-тест roundtrip сериализации.
- [ ] Юнит-тест несовпадения версии (вручную сконструировать старую версию).
- [ ] Тест вывода `thread_safety`: охватить как минимум один сценарий `LocalOnly` и проверить при `num_workers>1`, что узел не выполняется в другом потоке.

---

### 6.2 Типы ресурсов и генерация ControlEdge (минимально работающая абстракция побочных эффектов)

**Цель**
> **Обновление**: согласно RFC-001 побочные эффекты не выражаются через дополнительную систему эффектов, а абстрагируются через **типы ресурсов (Resource)** как операции с ресурсами, порождающие ControlEdge.

- Распознавание операций с ресурсами (MVP):
  - Если какой-либо из типов параметров вызова — `Resource` или его производный тип ресурса (например, `Console/FilePath/HttpUrl/DBUrl`), то точка вызова является операцией с ресурсом;
  - Функции стандартной библиотеки, работающие с ресурсами, должны иметь распознаваемые type constraint (рекомендуется: явно нести типы ресурсов в экспортируемых сигнатурах std; либо маркировать "операция с ресурсом" в метаданных экспорта FFI registry с привязкой к позициям параметров-ресурсов).
- Генерация ControlEdge (MVP):
  - Для **одного и того же значения/хэндла ресурса (одно SSA-значение/один и тот же ключ резидентной константы)** множественные операции с ресурсом добавляют ControlEdge по программному порядку (автоматическая сериализация).
  - Если невозможно определить, тот же это ресурс или нет (алиасинг/сложное происхождение), по умолчанию — консервативная сериализация (в будущем возможно введение явной unsafe-подсказки параллелизма как расширения).
  - **Идентификация ресурса распространяется по потоку данных (RFC-001)**: обнаружение конфликтов ресурсов выполняется по "равенству значений/одному и тому же источнику", а не по "совпадению типа ресурса" (два разных значения `FilePath` могут выполняться параллельно; одно и то же значение `FilePath` — обязательно последовательно).

**Критерии приёмки**
- [ ] На примере: `log → save → log` формирует ControlEdge из-за ресурсов Console/FilePath, стабильно сериализуется; операции с разными ресурсами могут идти параллельно.
- [ ] Распознавание операций с ресурсами стабильно (один и тот же входной module даёт одинаковый результат при повторных прогонах).

**Тестовые пункты**
- [ ] Юнит: тест распознавания параметра-ресурса (наличие параметра Resource — обязательная генерация ControlEdge).
- [ ] Юнит: две операции с одним и тем же значением ресурса (одна переменная/одна константа) — обязательная генерация ControlEdge; операции с разными значениями ресурсов (разные переменные/константы) — могут не порождать.
- [ ] Интеграция: запуск примера с несколькими `std.io.println/std.io.write_file`, assert порядка вывода/записи соответствует интерпретатору.

### 6.3 Автоматический откат L1 (понижение малых функций до @block, чтобы избежать накладных расходов планирования)

> **Источник**: RFC-001 5.2 (автоматический откат L1).  
> **Цель**: без изменения семантики сократить накладные расходы DAG/планировщика для малых функций (в первую очередь для согласованного поведения интерпретатора и AOT бэкенда).

**Цель**
- На этапе компиляции выполнять лёгкую пороговую оценку функций; при выполнении любого из условий стратегия по умолчанию для этой функции (или отдельных её блоков) понижается до `Serial`:
  - Количество инструкций `< 50`
  - Количество узлов DAG `< 10`
- Предоставить переключатели через CLI/конфигурацию (для MVP: достаточно и внутренней конфигурации):
  - `--l1-threshold=<N>` — изменить порог
  - `--no-l1-fallback` — отключить автоматический откат

**Критерии приёмки**
- [ ] Малые функции при `@auto` фактически не попадают в DAG/очередь планировщика (проверяется по статистическим полям или трассировке), результат совпадает с вариантом без отката.
- [ ] Большие функции не затрагиваются; принудительные аннотации `@eager/@block` имеют приоритет над автоматическим откатом.

**Тестовые пункты**
- [ ] Юнит: построить граничные значения (49/50 инструкций, 9/10 узлов) и проверить срабатывание отката.
- [ ] Регрессия: для одного и того же исходного кода при включённом/выключенном откате вывод и возвращаемые значения совпадают.

## Этап 7: Рантайм DAG планировщик (ядро ленивого планирования) (6–10 дней)

### 7.1 Реализация модели задач (стыковка с `RtValue::Async`)

**Цель**
- `scheduler.rs` (или миграция в `src/backends/runtime/` для "совместного использования интерпретатором/LLVM") реализует:
  - Выделение `TaskId`
  - `TaskState { Pending, Running, Completed(RtValue), Failed(RtValue) }`
  - `spawn(task_fn_ptr, args, deps, node_id, span_id)`: создаёт задачу с возможностью отложенного запуска (для распространения ошибок/графа ошибок)
  - `force(task_id)`: запускает выполнение по топологии зависимостей и ожидает результат

**Критерии приёмки**
- [ ] `yx_rt_lazy_call` возвращает `Async(TaskId)`, и задача регистрируется (не выполняется немедленно).
- [ ] `yx_rt_force` способен запустить выполнение задачи и вернуть результат (включая цепочку зависимостей).

**Тестовые пункты**
- [ ] Чистый Rust: с mock-"скомпилированной" функцией (фактически `extern "C"` функция Rust) построить DAG из 3 узлов, проверить порядок зависимостей и корректность результатов.
- [ ] Распространение ошибок: force нижестоящего узла получает Error, и не возникает deadlock.

---

### 7.2 Реализация стратегий планирования (Serial/Eager/Lazy)

**Цель**
- `Serial` (соответствует `@block`): не создаёт Async; вызов выполняется немедленно; интерфейс планировщика может обходиться.
- `Eager`: создаёт задачу, но сразу `force` (гарантирует порядок), для отладки/семантического выравнивания.
- `Lazy` (по умолчанию `@auto`): `force` только в момент, когда значение требуется; планировщик может в фоновом окне предварительно запускать "готовые" задачи (ограничено числом конкурентных задач).
- Снизу вверх (RFC-001/008): поведение рантайма должно отражать характеристику "обратного запуска вычисления от места, где требуется результат"; **ветви/островные под-DAG, которые не потребляются и не задействуют операций с ресурсами (без ControlEdge), не должны выполняться**, что снижает накладные расходы; операции с ресурсами должны упорядочиваться и завершаться в соответствии с ControlEdge (как в интерпретаторе).
- Фоновый DAG (RFC-018): при наличии в одной области видимости нескольких долгоработающих/бесконечных задач планировщик должен обеспечивать **кооперативное нарезание** (например, на основе budget или явного `yield_now`), во избежание голодания основного DAG и "зависания в цикле".

**Критерии приёмки**
- [ ] Одна и та же программа при стратегиях Serial/Eager/Lazy даёт одинаковый результат.
- [ ] При Lazy, если результат какого-либо вызова ни разу не force-ится/не используется, задача не выполняется (Lazy Task Creation).

**Тестовые пункты**
- [ ] Сравнительный тест: вывод трёх стратегий совпадает.
- [ ] Тест ленивого пропуска: написать ветвь/переменную, "вычисляемую, но не используемую", assert-ить счётчик выполнения соответствующей задачи = 0 (по статистическим полям планировщика).
- [ ] Тест фонового нарезания: построить 2 долгоработающие задачи + 1 основную, assert, что в пределах временного окна у всех трёх есть прогресс (можно через счётчики или `thread_id` + лог-статистику).

---

### 7.3 Управление параллелизмом и гранулярностью

**Цель**
- Верхний предел параллелизма: `max_parallelism = num_workers * 2` (рекомендация RFC-018).
- Ограничения по ресурсам: операции с одним и тем же ресурсом должны выполняться последовательно в соответствии с ControlEdge (правила типов ресурсов RFC-001), планировщик не должен нарушать порядок ControlEdge.
- Ограничения потокобезопасности (RFC-001/009): планировщик обязан учитывать `DAGNode.thread_safety`:
  - `SendSync`: может выполняться на разных worker (с учётом верхнего предела и ограничений зависимостей)
  - `LocalOnly`: запрещено планирование между потоками / запрещено быть украденным work-stealing-ом; при необходимости понижается до последовательного (либо фиксируется на worker, где создан)
- Адаптивная гранулярность (MVP): когда готовых задач значительно больше верхнего предела параллелизма, объединять "множество готовых задач, **не имеющих ограничений ControlEdge**" в пакетную отправку (реализуется как последовательное выполнение пакета одним worker, что сокращает накладные расходы планирования).

**Критерии приёмки**
- [ ] Большое количество независимых задач без ограничений по ресурсам (1e4) не приводит к взрыву памяти (пиковое число структур задач — O(числа конкурентных) или контролируемая верхняя граница).
- [ ] Узлы `LocalOnly` при `num_workers>1` не выполняются в другом потоке (проверяется через `std.concurrent.thread_id`).
- [ ] Порядок вывода/побочных эффектов операций с ресурсами (например, `std.io.*`) строго соответствует порядку в интерпретаторе.

**Тестовые пункты**
- [ ] Нагрузочный юнит-тест: построить 10000 mock-задач, пик памяти/числа задач контролируем (можно assert-ить по статистике, без точного измерения памяти).
- [ ] Интеграционный тест LocalOnly: пример с узлом `LocalOnly`, при `num_workers>1` assert-ить неизменность ID потока выполнения.
- [ ] Интеграционный тест порядка ресурсов: множество операций с ресурсами (println/write_file) должны выводить/записывать в порядке исходного кода.

### 7.4 Распространение ошибок и запись графа ошибок (минимальный замкнутый цикл RFC-001)

**Цель**
- Определить минимальную структуру данных `ErrorGraph` (на старте — только для отладки/трассировки):
  - Узел: `node_id + span_id + message/error_code`
  - Ребро: `from_node_id -> to_node_id` (означает "ошибка распространяется от зависимого узла к потребителю")
- Стратегия записи (решения RFC-001):
  - Ошибки распространяются вверх вдоль рёбер зависимостей, **независимо от фактического порядка выполнения**;
  - DAG строится только внутри функции, поэтому и граф ошибок ограничен уровнем функции (во избежание взрывного роста памяти).
- Согласование с ABI:
  - `yx_rt_lazy_call/yx_rt_native_call` обязаны нести `node_id/span_id` (зафиксировано на этапе 2.3)
  - При сбое задачи и возврате ошибки из `force` — запись в `ErrorGraph` (если `ctx.error_graph != null`)

**Критерии приёмки**
- [ ] При сбое нижнего узла цепочки зависимостей верхняя точка потребления получает ошибку (с возможностью локализации до span сбойного узла).
- [ ] При параллельном выполнении путь распространения ошибки стабильно воспроизводим (не зависит от порядка планирования).

**Тестовые пункты**

## Этап 8: Сквозная поддержка синтаксических аннотаций (@block/@eager/@auto) (5–8 дней)

### 8.1 Поддержка аннотаций во фронтенде и их прохождение вниз до байткода/метаданных

**Цель**
- Слой разбора: распознавать аннотации функций/блоков `@block`, `@eager`; по умолчанию — `@auto`.
- Разбор/проверка типов: обеспечить, что `spawn` допускается только в области видимости `@block` (RFC-001/008).
- IR/Bytecode: в `BytecodeFunction` или дополнительной side-table нести стратегию по умолчанию; в точке вызова должна быть возможность решить — идти ли путём lazy/eager/direct.

**Критерии приёмки**
- [ ] Без аннотации: по умолчанию Lazy (@auto).
- [ ] `@block`: в данной области видимости не создаётся Async, поведение совпадает с последовательным интерпретатором.
- [ ] `@eager`: после создания задачи немедленный `force` (результат тот же, удобнее для отладки).

**Тестовые пункты**
- [ ] Фронтенд: тесты разбора/генерации IR с аннотациями (assert-ы по AST/IR).
- [ ] Бэкенд: один и тот же исходный код с разными аннотациями выполняется в соответствии со стратегией.

### 8.2 Стандартная библиотека: интерфейсы рантайма `@block` и `spawn` (Full Runtime)

> **Источник**: RFC-008 (`@block` предоставляется стандартной библиотекой), RFC-001 (семантика ожидания `spawn` контролируется стандартной библиотекой).

**Цель**
- Добавить модуль рантайма стандартной библиотеки (рекомендуемый путь: `std.runtime` или `std.full`), предоставляющий:
  - `block`: принудительное немедленное вычисление (эквивалентно установке стратегии области видимости в `Serial`/непопаданию в очередь DAG)
  - `spawn`/`join_all` (или неявный join): в области видимости `@block` создаёт конкурентные задачи и ожидает их завершения
- Компилятор для MVP может реализовать это встроенно, но обязан абстрагировать интерфейс, который можно будет вынести в стандартную библиотеку (во избежание затрат на будующий рефакторинг).

**Критерии приёмки**
- [ ] Блоки `spawn { ... }` в теле `@block` способны выполняться конкурентно и завершаются до выхода из области видимости (без "тихой утечки фоновых задач").
- [ ] Поведение `@block` явно отличимо от L3-поведения по умолчанию (например: попадает ли в очередь DAG, порождается ли Async-значение).

**Тестовые пункты**
- [ ] Интеграция: пример с двумя `spawn { std.concurrent.sleep(50) }` при нескольких worker занимает время, близкое к одному sleep (грубая проверка конкурентности).
- [ ] Негативный: использование `spawn` вне `@block` приводит к ошибке (согласованно с 0.3/8.1).

## Допущения и значения по умолчанию (при отсутствии бизнес-требований)

- Основная версия LLVM по умолчанию — **17**; при отличии инструментальной цепочки команды достаточно унифицированно изменить feature `inkwell` и документацию.
- Путь выполнения AOT строится в два шага: сначала выполнение в памяти (проверка на этапе разработки), затем загрузка с диска с линковкой (собственно AOT).
- На начальном этапе `llvm-aot` обязуется покрыть только MVP-поднабор инструкций; продвинутые возможности (замыкания/динамическая диспетчеризация/исключения и т. д.) расширяются по необходимости (при встрече возвращается явная ошибка "не реализовано").
- Рёбра зависимостей DAG **могут выводиться рантаймом динамически из TaskId Async-аргументов**; поле edges на этапе компиляции сначала выступает как опциональная оптимизация и отладочная валидация, не блокируя поставку M2.
  - **Дополнение (RFC-001)**: основным источником ControlEdge являются правила типов ресурсов; при отсутствии информации о ресурсах по умолчанию используется консервативная последовательность для обеспечения корректности.
