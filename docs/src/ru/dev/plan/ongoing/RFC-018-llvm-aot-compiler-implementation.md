# RFC-018 План реализации LLVM AOT компилятора и прозрачной параллели L3 (планирование DAG с задержкой)

> **Задача**: Реализовать LLVM AOT бэкенд + DAG планировщик времени выполнения, обеспечить три стратегии планирования (@auto/@eager/@block) (L3/L2/L1)  
> **На основе RFC**: RFC-018 (черновик)  
> **Зависимости RFC**: RFC-001 (并作 модель и обработка ошибок), RFC-008 (трёхуровневая среда выполнения), RFC-009 (ownership/Arc)  
> **Дата**: 2026-03-10  
> **Статус**: В процессе  
> **Целевые вехи**:  
> - M1: LLVM AOT (компилируемо и исполняемо, последовательно)  
> - M2: DAG метаданные + однопоточное планирование (Standard Runtime, num_workers=1)  
> - M3: многопоточное параллельное планирование + контроль гранулярности (Full Runtime, num_workers>1)  
> - M4: отложенное планирование (Lazy 并作 Creation) + **типы ресурсов (Resource) как абстракция побочных эффектов** + ** распространение ошибок/граф ошибок** + сквозные аннотации

---

## Аннотация (замыкание реализации)

- Добавить в `yaoxiang` LLVM бэкенд (feature gate), компилирующий `BytecodeModule` в машинный код (COFF/ELF/Mach-O) с возможностью загрузки и выполнения во время выполнения.
- Ввести **стабильный ABI**: код, сгенерированный AOT, взаимодействует со временем выполнения через `extern "C"` функции `RtValue/RtContext`, избегая проблем нестабильного ABI Rust enum.
- Реализовать ядро RFC-018: **DAG внутри функционального блока** + **отложенное планирование (Lazy Scheduling)**. Параллелизм/последовательность определяются **рёбрами DAG (Data/Control/Spawn)** и **правилами типов ресурсов (Resource)**; ошибки распространяются по рёбрам зависимости согласно RFC-001 и могут формировать граф ошибок.

---

## Публичные интерфейсы/изменения поведения (видимые извне)

1. **Cargo features**
   - Новая feature `llvm-aot`: включает зависимости LLVM/inkwell и AOT бэкенд; отключена по умолчанию (обеспечивает сборку без LLVM окружения).
2. **CLI**
   - `yaoxiang run` получает `--backend {interpreter,llvm}` (по умолчанию interpreter).
   - Опционально: добавление `--runtime {embedded,standard,full}` и `--workers <N>` для контроля уровня среды выполнения и степени параллелизма (RFC-008):
     - `--runtime embedded`: немедленное выполнение (без DAG, без функций планировщика, для встраиваемых/минималистичных сценариев)
     - `--runtime standard`: DAG + Scheduler (num_workers=1 для асинхронного; >1 для параллельного)
     - `--runtime full`: standard + WorkStealer (продвинутые функции, опционально)
3. **ABI среды выполнения (внутренний, но межмодульный)**
   - Новые `RtValue` (`#[repr(C)]`) и `RtContext` (содержит только указатели/базовые типы) как граница взаимодействия AOT и runtime.

## Ключевые проектные ограничения (согласование с RFC-001 / RFC-008 / RFC-018)

### A. Семантика 并发 (L1/L2/L3 — лишь ментальная модель)

- **L3 (по умолчанию / @auto)**: прозрачный параллелизм; построение DAG; при встрече с вызовом возвращается "отложенно вычислимое" значение, **вычисление запускается только при запросе значения**.
- **L1 (@block)**: предоставляется стандартной библиотекой (RFC-008), семантика "принудительное急切ное вычисление", не попадает в очередь отложенного DAG; в основном для отладки и критических последовательных участков.
- **L2 (spawn)**: **может использоваться только внутри области @block** (RFC-001/008), для вставки параллелизма в синхронный код; принадлежит возможностям Full Runtime.

### B. Три уровня среды выполнения (RFC-008)

- **Embedded Runtime**: немедленное выполнение; можно полностью отказаться от построения DAG (экономия памяти/времени старта); для ограниченных окружений.
- **Standard Runtime**: DAG + Scheduler — ядро (lazy вычисление естественно поддерживает асинхронность/параллелизм).
- **Full Runtime**: поверх standard добавляет WorkStealer, а также возможности стандартной библиотеки `@block` / `spawn` и др.

### C. Область построения DAG и память (RFC-001/018)

- DAG строится только **внутри одного тела функции/блока**; не рекурсивно раскрывает тела вызываемых функций (во избежание взрыва графа ошибок и DAG узлов).
- DAG метаданные должны нести **стабильные ID узлов/рёбер** и **Span** (для распространения ошибок и локализации в графе ошибок).

### D. Абстракция побочных эффектов (RFC-001: типы ресурсов)

- Не вводится дополнительная "система явных аннотаций побочных эффектов"; побочные эффекты统一为**операции с ресурсами**:
  - Вызов функции, параметры которой содержат `Resource` (или производные типы ресурсов), считается операцией с ресурсом;
  - Операции над одним ресурсом автоматически формируют **ControlEdge** (последовательно); над разными ресурсами возможен параллелизм;
  - При невозможности статически определить, один ли это ресурс, по умолчанию保守но последовательно (в дальнейшем возможно расширение с явными unsafe подсказками параллелизма).

### E. Распространение ошибок (RFC-001)

- Ошибки распространяются вверх по рёбрам зависимости DAG (независимо от фактического порядка параллельного выполнения), с записью пути распространения для графа ошибок.

### 0.3 Фиксация ограничений реализации 并作 модели (согласование с RFC-001/008)

**Цель**
- Чётко зафиксировать и задокументировать следующие ограничения реализации (в коде комментариями/документации разработки и в тест-кейсах):
  - `spawn` разрешён только внутри области видимости `@block` (защита на этапах парсинга/проверки типов/IR).
  - `@block` семантика "принудительное急切ное вычисление", предоставляется возможностями стандартной библиотеки (можно сначала реализовать как встроенную в компилятор для MVP, но сохранить интерфейс для будущего переноса в стандартную библиотеку).
  - DAG строится только внутри функционального блока; должен содержать стабильные `node_id` и `span` (поддержка распространения ошибок/графа ошибок).
  - Типы ресурсов (Resource) управляют генерацией ControlEdge, без引入额外 пользовательской системы effect аннотаций.
  - **Ограничения параллельной безопасности (RFC-001/009)**: параллелизм между потоками разрешён только когда захваченные/возвращаемые значения узла удовлетворяют `Send + Sync` (или эквивалентному ограничению на уровне языка); иначе должен降级为 последовательное выполнение (или выполнение на одном worker).

**Критерии приёмки**
- [ ] Компилятор выдаёт明确ную ошибку (с Span) при недопустимом использовании `spawn`.
- [ ] Семантические различия `@block/@eager/@auto` наблюдаемы и тестируемы на минимальных примерах.
- [ ] Документация (этот план) и ключевые решения RFC-001/008/018 согласованы, нет противоречивых пунктов.

**Тестовые проекты**
- [ ] Тест ограничения位置 `spawn`: spawn вне @block должен报错.
- [ ] Тест области видимости DAG: подтвердить, что DAG не выходит за пределы тела функции (число узлов не зависит от уровня вызовов).
- [ ] Тесты ограничений Send/Sync:
  - `spawn` с захватом не-`Send` значения должен报错 (с Span).
  - При `@auto` узлы с не-`Send + Sync` значениями не должны планироваться межпотоково (можно проверить через `std.concurrent.thread_id`).

## Этап 2: Стабильный ABI (RtValue/RtContext) и Runtime API (3-5 дней)

> Этот этап критичен для "исполняемости сгенерированного LLVM кода": сначала необходимо стабилизировать представление значений на границе.

### 2.1 Определение `RtValue` (стабильный C ABI)

**Цель**
- Определить `#[repr(C)] struct RtValue { tag: u8, _pad: [u8; 7], payload: u64 }` (или 16-байтовая структура для простоты выравнивания).
- Определить `#[repr(u8)] enum RtTag { Unit, Bool, Int, Float, Handle, Async, Error }` (минимальный набор; расширяемо).
- Соглашения:
  - Int: `payload` = биты `i64`
  - Float: `payload` = биты `f64`
  - Bool: 0/1
  - Handle: `payload` = `usize` (расширяется до u64)
  - Async: `payload` = `TaskId` (u64)
  - Error: `payload` = код ошибки или указатель (в MVP можно сначала код ошибки)

**Критерии приёмки**
- [ ] `RtValue` безопасно конструируется/читается внутри Rust (без UB), с `Debug` выводом и базовыми утилитами для утверждений.
- [ ] Выравнивание с LLVM IR: возможность создавать в inkwell struct type той же структуры (одинаковый порядок полей/размер).

**Тестовые проекты**
- [ ] Roundtrip `RtValue`: модульные тесты encode/decode для int/float/bool/unit.
- [ ] Тест размера ABI: `size_of::<RtValue>()` и `align_of::<RtValue>()` фиксированы (защитные assertions против случайных изменений).

---

### 2.2 Определение `RtContext` (контекст времени выполнения)

**Цель**
- Определить `#[repr(C)] struct RtContext`, содержащий только указатели/целые:
  - `heap: *mut Heap`
  - `ffi: *const FfiRegistry`
  - `scheduler: *mut DynScheduler` (или указатель на конкретную реализацию)
  - `max_parallelism: usize`
  - `error_graph: *mut ErrorGraph` (опционально: для записи распространения ошибок RFC-001; в MVP может быть null)
  - Резервные поля (номер версии/flags), но минимизировать (KISS).

**Критерии приёмки**
- [ ] `RtContext` может быть сконструирован интерпретатором/LLVM executor и передан сгенерированному коду.
- [ ] Не содержит полей с нестабильным layout в Rust (запрещено прямое встраивание значений `Heap`/`FfiRegistry`).

**Тестовые проекты**
- [ ] Тесты безопасности памяти при конструировании/уничтожении `RtContext` (без реального LLVM).

---

### 2.3 Runtime C API: минимальный набор функций для сгенерированного кода

**Цель**
- Предоставить функции `#[no_mangle] extern "C"` (единый префикс `yx_rt_*`), MVP минимум:
  - `yx_rt_force(ctx: *mut RtContext, v: RtValue) -> RtValue`
  - `yx_rt_lazy_call(ctx, callee: *const u8 /*указатель функции*/, args: *const RtValue, argc: usize, node_id: u32, span_id: u32) -> RtValue`
  - `yx_rt_native_call(ctx, name_ptr: *const u8, name_len: usize, args: *const RtValue, argc: usize, node_id: u32, span_id: u32) -> RtValue`
  - `yx_rt_panic(code: u32)` или `yx_rt_trap(msg_ptr, len)` (для отладки)
- Ограничение: AOT сгенерированный код **взаимодействует только через вышеуказанный API**, не напрямую с Rust структурами.

**Критерии приёмки**
- [ ] Runtime API компилируется и без LLVM (управляется feature `llvm-aot`: API может присутствовать всегда или только под feature, но должно быть тестируемым).
- [ ] `yx_rt_native_call` может вызывать handler `FfiRegistry` (MVP поддерживает только параметры и返回值 Int/Float/Bool/Unit; при неподдержке возвращает Error RtValue), при неудаче записывает `node_id/span_id` в граф ошибок (если включён).

**Тестовые проекты**
- [ ] Чистые Rust модульные тесты: прямой вызов `yx_rt_native_call`, проверка работоспособности пути `std.io.println` (или самозарегистрированной функции).
- [ ] Тест ошибочного пути: передача несуществующего имени native возвращает `Error` RtValue с возможностью преобразования в `ExecutorError::FunctionNotFound`.

**Цель**

**Тестовые проекты**

- [ ] Сгенерированные сигнатуры функций в LLVM module могут быть найдены и проверены на соответствие типов параметров/возврата.

### 4.2 Перевод основного подмножества инструкций (покрытие "запускаемости")

**Цель**
- `codegen.rs`: реализовать минимум следующие `BytecodeInstr`:

  - `CallNative` (через `yx_rt_native_call`)

- Принудительное правило: все операнды, участвующие в арифметике/сравнении/ветвлении, должны сначала пройти `yx_rt_force` ("при запросе значения запускается" прозрачного параллелизма).

**Критерии приёмки**
- [ ] AOT бэкенд исполняет простые программы:
  - чистая арифметика
  - if/else
  - вызов `std.io.println` для вывода
- [ ] Неподдерживаемые инструкции产生 читаемую ошибку (не panic).

**Тестовые проекты**
- [ ] Интеграция: новый `tests/integration/llvm_aot_smoke.rs` (feature gate), запускает 5 программных фрагментов и утверждает результаты/вывод (вывод可以通过 перенаправление stdout).
- [ ] Негативный: при встрече `MakeClosure/CallVirt/...` возвращает явное "не реализовано".

## Этап 6: Генерация DAG метаданных (4-7 дней)

### 6.1 Определение `DAGMetadata` (версионированное)

**Цель**
- `dag.rs` определяет:
  - `dag_version: u32`
  - `nodes: Vec<DAGNode>` (содержит `node_id` и `span_id` для распространения ошибок)
  - `edges: Vec<DAGEdge>` (с типами рёбер: Data/Control/Spawn)
- `DAGEdge` минимум содержит:
  - `from: u32`
  - `to: u32`
  - `kind: EdgeKind { Data, Control, Spawn }`
- Правила конфликтов/планирования (RFC-001):
  - DataEdge + DataEdge: возможен параллелизм (если нет других зависимостей)
  - Любая комбинация с ControlEdge: обязательная сериализация (сохранение порядка)
- `DAGNode` минимум содержит:
  - `node_id: u32` (уникален внутри функции)
  - `ip: u32` (позиция инструкции call)
  - `strategy: ScheduleStrategy { Serial, Eager, Lazy }` (происходит от аннотации или стратегии по умолчанию)
  - `span_id: u32` (локализация и граф ошибок)
  - `thread_safety: ThreadSafety { SendSync, LocalOnly }` (выводится системой типов; `LocalOnly` запрещает межпотоковое планирование)
- Соглашение: узел описывает только "точку вызова, которую можно планировать", параметры захватываются при выполнении `yx_rt_lazy_call` (во избежание сложности статического вычисления выражений).

**Критерии приёмки**
- [ ] `DAGMetadata` сериализуем/десериализуем (используя существующий `ron` или `serde_json`).
- [ ] При несовпадении `dag_version` загрузка报错.

**Тестовые проекты**
- [ ] Модульный тест roundtrip сериализации.
- [ ] Модульный тест несовпадения версий (вручную сконструировать старую версию).
- [ ] Тест вывода `thread_safety`: минимум 1 сценарий `LocalOnly`, и при `num_workers>1` проверяется отсутствие межпотокового выполнения.

---

### 6.2 Типы ресурсов и генерация ControlEdge (минимально рабочая абстракция побочных эффектов)

**Цель**
> **Обновление**: согласно RFC-001, побочные эффекты выражаются не через дополнительную систему эффектов, а абстрагируются как **операции с ресурсами (Resource)**, с генерацией ControlEdge.

- Распознавание операций с ресурсами (MVP):
  - Если любой параметр вызываемой функции имеет тип `Resource` или производный тип ресурса (например `Console/FilePath/HttpUrl/DBUrl`), то эта точка вызова является операцией с ресурсом;
  - Функции операций с ресурсами стандартной библиотеки должны иметь распознаваемые ограничения типа (рекомендуемый подход: в сигнатурах экспорта std явно указывать тип ресурса; или в метаданных экспорта FFI registry помечать "операция с ресурсом" и связывать с позицией параметра ресурса).
- Генерация ControlEdge (MVP):
  - Для нескольких операций с ресурсами над **одним и тем же значением ресурса/дескриптором (одинаковое SSA значение/одинаковый константный ключ)** добавлять ControlEdge по программному порядку (автоматическая сериализация).
  - При невозможности определить, один ли это ресурс (алиасинг/сложное происхождение), по умолчанию保守но последовательно (в будущем можно ввести явную unsafe подсказку параллелизма как расширение).
  - **Идентификатор ресурса распространяется по потоку данных (RFC-001)**: обнаружение конфликтов ресурсов основано на "равенстве значений/одинаковом источнике", а не на "одинаковом типе ресурса" (два разных `FilePath` значения могут выполняться параллельно; одно и то же `FilePath` значение должно выполняться последовательно).

**Критерии приёмки**
- [ ] Для примера: `log → save → log` из-за ресурсов Console/FilePath формирует ControlEdge, стабильно последовательно; операции с разными ресурсами могут быть параллельными.
- [ ] Распознавание операций с ресурсами стабильно (多次 результат для одного входного module одинаков).

**Тестовые проекты**
- [ ] Модульный: тест распознавания типа параметра ресурса (при наличии параметра Resource должен генерироваться ControlEdge).
- [ ] Модульный: две операции с ресурсами над одним значением ресурса (одна переменная/одна константа) должны генерировать ControlEdge; над разными значениями ресурса (разные переменные/константы) могут не генерировать.
- [ ] Интеграция: запустить пример с несколькими `std.io.println/std.io.write_file`, утверждать порядок вывода/записи совпадает с интерпретатором.

### 6.3 L1 автоматический откат (малые функции降级为 @block, избежание накладных расходов планирования)

> **Источник**: RFC-001 5.2 (L1 автоматический откат).  
> **Цель**: без изменения семантики сократить накладные расходы DAG/планировщика для малых функций (особенно统一ное поведение для интерпретатора и AOT бэкенда).

**Цель**
- Во время компиляции выполнить лёгкую пороговую оценку функций; при удовлетворении любого условия стратегия по умолчанию для функции (или некоторых блоков внутри функции)降级为 `Serial`:
  - Число инструкций `< 50`
  - Число DAG узлов `< 10`
- Через CLI/конфигурацию предоставить переключатели (MVP: только внутренняя конфигурация тоже допустима):
  - `--l1-threshold=<N>` настроить порог
  - `--no-l1-fallback` отключить автоматический откат

**Критерии приёмки**
- [ ] Малые функции при `@auto` фактически не попадают в DAG/очередь планировщика (можно проверить через статистическое поле или trace), результат совпадает с отсутствием отката.
- [ ] Большие функции не затрагиваются; принудительные аннотации `@eager/@block` имеют приоритет над автоматическим откатом.

**Тестовые проекты**
- [ ] Модульный: сконструировать граничные значения (49/50 инструкций, 9/10 узлов) проверить срабатывание отката.
- [ ] Регрессия: для одного исходного кода с/без отката вывод и返回值 совпадают.

## Этап 7: DAG планировщик среды выполнения (ядро Lazy Scheduling) (6-10 дней)

### 7.1 Реализация модели задач (сопряжение с `RtValue::Async`)

**Цель**
- `scheduler.rs` (или перенести в `src/backends/runtime/` для реализации "интерпретатор/LLVM совместно") реализует:
  - `TaskId` распределение
  - `TaskState { Pending, Running, Completed(RtValue), Failed(RtValue) }`
  - `spawn(task_fn_ptr, args, deps, node_id, span_id)`: создать задачу но można отложить запуск (用于错误传播/错误图)
  - `force(task_id)`: по топологии зависимостей запустить выполнение и ждать результата

**Критерии приёмки**
- [ ] `yx_rt_lazy_call` возвращает `Async(TaskId)` и задача регистрируется (не выполняется немедленно).
- [ ] `yx_rt_force` может запустить выполнение задачи и вернуть результат (включая цепочку зависимостей).

**Тестовые проекты**
- [ ] Чистый Rust: через mock "compiled fn" указатель (фактически Rust `extern "C"` функция) сконструировать 3-узловой DAG, проверить правильность порядка зависимостей и результата.
- [ ] Распространение ошибок: при force нижестоящего получаем Error, и нет deadlock.

---

### 7.2 Реализация стратегий планирования (Serial/Eager/Lazy)

**Цель**
- `Serial` (соответствует `@block`): не создаёт Async; call выполняется немедленно; интерфейс планировщика можно обойти.
- `Eager`: создаёт задачу но сразу `force` (гарантирует порядок), для отладки/согласования семантики.
- `Lazy` (по умолчанию `@auto`): `force` только при запросе значения; планировщик может заранее запускать "готовые" задачи в фоновом окне (с ограничением степени параллелизма).
- Снизу вверх (RFC-001/008): поведение среды выполнения должно отражать特性 "обратный запрос вычисления от места, где нужен результат"; **ветви/изолированные острова DAG, которые не потребляются и не связаны с операциями с ресурсами (без ControlEdge), не должны выполняться**, для снижения накладных расходов; операции с ресурсами должны гарантировать порядок по ControlEdge и завершение (согласованно с интерпретатором).
- Фоновый DAG (RFC-018): когда в одной области видимости существуют несколько долго выполняющихся/бесконечных задач, планировщик должен предоставлять **кооперативные срезы** (например на основе budget или явный `yield_now`), во избежание starvation основного DAG и "застревания в цикле".

**Критерии приёмки**
- [ ] Одна и та же программа при Serial/Eager/Lazy даёт одинаковый результат.
- [ ] При Lazy, если результат вызова никогда не был force/использован, задача не выполняется (Lazy 并作 Creation).

**Тестовые проекты**
- [ ] Сравнительный тест: три стратегии дают一致的 вывод.
- [ ] Тест пропуска Lazy: написать ветвь/переменную "вычислить но не использовать", утверждать счётчик выполнения соответствующей задачи равен 0 (статистическое поле планировщика).
- [ ] Тест фоновых срезов: сконструировать 2 долго выполняющихся задачи + 1 главная задача, утверждать что за времявое окно все три продвинулись (можно использовать счётчики или `thread_id` + логи).

---

### 7.3 Контроль параллелизма и гранулярности

**Цель**
- Верхний предел параллелизма: `max_parallelism = num_workers * 2` (RFC-018 рекомендация).
- Ограничения ресурсов: операции над одним ресурсом должны выполняться последовательно по ControlEdge (правила типа ресурса RFC-001), планировщик не должен нарушать порядок ControlEdge.
- Ограничения потокобезопасности (RFC-001/009): планировщик должен уважать `DAGNode.thread_safety`:
  - `SendSync`: можно выполнять меж worker (с учётом ограничений параллелизма и зависимостей)
  - `LocalOnly`: запрещено межпотоковое планирование/запрещено быть украденным work-stealing; при необходимости降级为 последовательное выполнение (или закрепление на создавшем его worker)
- Адаптивная гранулярность (MVP): когда число задач в очереди намного превышает верхний предел параллелизма, "несколько готовых задач **без ограничений ControlEdge**"合并批量提交 (реализовать как последовательное выполнение пакета одним worker, сокращение накладных расходов планирования).

**Критерии приёмки**
- [ ] Большое количество независимых задач без ограничений ресурсов (1e4) не вызывают взрыв памяти (структура задач O(степень параллелизма) или контролируемая верхняя граница).
- [ ] `LocalOnly` узел при `num_workers>1` не выполняется межпотоково (можно проверить через `std.concurrent.thread_id`).
- [ ] Порядок вывода/побочных эффектов операций с ресурсами (например `std.io.*`) строго сохраняет порядок интерпретатора.

**Тестовые проекты**
- [ ] Стресс-тест: сконструировать 10000 mock задач, пиковое потребление памяти/количество задач контролируемо (статистические assertions, точное измерение памяти не обязательно).
- [ ] Интеграционный тест LocalOnly: сконструировать пример с узлами `LocalOnly`, при `num_workers>1` утверждать что ID потока выполнения не меняется.
- [ ] Интеграционный тест порядка ресурсов: несколько операций с ресурсами (println/write_file) должны выводиться/записываться в порядке исходного кода.

---

### 7.4 Распространение ошибок и запись графа ошибок (минимальное замыкание RFC-001)

**Цель**
- Определить минимальную структуру данных `ErrorGraph` (можно сначала только для отладки/trace):
  - Узлы: `node_id + span_id + message/error_code`
  - Рёбра: `from_node_id -> to_node_id` (обозначает "ошибка распространяется от зависимого узла к потребляющему")
- Стратегия записи (решение RFC-001):
  - Ошибки распространяются вверх по рёбрам зависимости, **независимо от фактического порядка выполнения**;
  - DAG строится только внутри функции, поэтому граф ошибок также ограничен уровнем функции, во избежание взрыва памяти.
- Согласование с ABI:
  - `yx_rt_lazy_call/yx_rt_native_call` должны нести `node_id/span_id` (зафиксировано в этапе 2.3)
  - При сбое задачи и возврате ошибки из `force`, записать в `ErrorGraph` (если `ctx.error_graph != null`)

**Критерии приёмки**
- [ ] При сбое узла в нижней части цепочки зависимостей, верхняя точка потребления получает ошибку (и может локализовать span отказавшего узла).
- [ ] При параллельном выполнении путь распространения ошибок стабильно воспроизводим (не зависит от порядка планирования).

**Тестовые проекты**
- [ ] Модульный: сконструировать 3-узловую цепочку зависимостей, симулировать сбой среднего узла, утверждать что рёбра ErrorGraph: `leaf->mid->root`.
- [ ] Регрессия параллелизма:多次 при num_workers>1 запускать, структура ErrorGraph согласованна.

### 8.2 Стандартная библиотека: интерфейсы времени выполнения `@block` и `spawn` (Full Runtime)

> **Источник**: RFC-008 (@block предоставляется стандартной библиотекой), RFC-001 (семантика ожидания spawn контролируется стандартной библиотекой).

**Цель**
- Добавить модуль runtime стандартной библиотеки (рекомендуемый путь: `std.runtime` или `std.full`), предоставляющий:
  - `block`: принудительное急切ное вычисление (эквивалент установки стратегии области видимости в `Serial`/не в очередь DAG)
  - `spawn`/`join_all` (или неявный join): создание параллельных задач и ожидание завершения внутри области `@block`
- Компилятор может сначала реализовать встроенно для MVP, но должен абстрагировать интерфейс для возможного переноса в стандартную библиотеку (во избежание затрат на рефакторинг в будущем).

**Критерии приёмки**
- [ ] Блок `spawn { ... }` внутри функции `@block` выполняется параллельно и завершается до конца области (не происходит "тихой утечки задач в фон").
- [ ] Поведение `@block` и поведение по умолчанию L3 параллелизма чётко различимы (например: попадает ли в очередь DAG, создаётся ли Async значение).

**Тестовые проекты**
- [ ] Интеграция: пример с двумя `spawn { std.concurrent.sleep(50) }` при нескольких worker затрачивает время, близкое к однократному sleep (грубая проверка параллелизма).
- [ ] Негативный: использование spawn вне @block报错 (согласуется с 0.3/8.1).

## Предположения и значения по умолчанию (выбор при отсутствии бизнес-требований)

- Основная версия LLVM по умолчанию выбрана **17**; при отличии инструментария команды достаточно изменить inkwell feature и документацию.
- AOT путь выполнения采取 "два шага": сначала выполнение в памяти (разработка и верификация), затем запись на диск链接/загрузка (настоящий AOT).
- На начальном этапе `llvm-aot` гарантирует только покрытие одного MVP подмножества инструкций; замыкания/динамическая диспетчеризация/исключения и другие продвинутые функции расширяются по необходимости (при встрече возвращается явное "не реализовано").
- Рёбра зависимости DAG **могут быть динамически выведены runtime из Async TaskId аргументов**; поле edges на этапе компиляции сначала作为可选тельная оптимизация и отладочная проверка, не блокирует доставку M2.
  - **Дополнение (RFC-001)**: Основной источник ControlEdge — правила типов ресурсов; при отсутствии информации о ресурсах по умолчанию保守но последовательно для гарантии корректности.
