> Дата создания: 2026-05-15
> Ответственный: 待定
> Статус: Постоянное обновление
> Последнее обновление: 2026-05-16 (на основе результатов тестирования RFC-010/011)

В этом документе задокументированы функции модуля typecheck, которые ещё не полностью реализованы. Эти функции определены в спецификации языка (language-spec.md) и документах RFC, но текущая реализация кода может содержать пробелы или быть неполной.

**Принцип тестирования**: Источником истины для тестов является спецификация, а не код. Если тест не проходит, это означает, что код не соответствует спецификации, и необходимо исправить код, а не изменять тест.

## RFC-010 Унифицированный синтаксис типов

### Пройденные тесты

- [x] `x: Int = 42` Объявление переменной
- [x] `name: String = "Alice"` Строковая переменная
- [x] `flag: Bool = true` Булева переменная
- [x] `y = 100` Вывод типа
- [x] `add: (a: Int, b: Int) -> Int = { return a + b }` Определение функции
- [x] `inc: (x: Int) -> Int = x + 1` Однострочная функция
- [x] `bad: (x: Int) -> Int = { return "hello" }` Проверка несоответствия типа возврата
- [x] `Point: Type = { x: Float, y: Float }` Определение типа записи
- [x] `p: Point = Point(1.0, 2.0)` Конструирование типа записи
- [x] `Point: Type = { x: Float = 0, y: Float = 0 }` Значения по умолчанию

- [x] `Point: Type = { ..., Drawable }` Синтаксис реализации интерфейса

- [x] `List: (T: Type) -> Type = { data: Array(T), length: Int }` Определение обобщённого типа
- [x] `numbers: List(Int) = List(1, 2, 3)` Создание экземпляра обобщённого типа

- [x] `draw(p, screen)` Вызов метода как функции
- [x] `Type` Ключевое слово ме типа
- [x] `: Type` Принудительный конструктор типа

### Исправленные тесты

Следующие тесты были исправлены в изменениях от 2026-05-16:

#### 1. Определение обобщённого типа ✅

- **Тестовый случай**: `test_rfc010_generic_type_definition`
- **Содержание исправления**: Парсер обнаруживает шаблон `(T: Type) -> Type = { ... }` и обрабатывает его как определение конструктора типа, а не определение функции
- **Затронутые файлы**: `declarations.rs` — добавлено обнаружение обобщённого конструктора типа в `parse_var_stmt_with_pub`

#### 2. Создание экземпляра обобщённого типа ✅

- **Тестовый случай**: `test_rfc010_generic_type_instantiation`
- **Содержание исправления**: После корректной регистрации обобщённого типа применение типа `List(Int)` правильно парсится

#### 3. Определение метода ✅

- **Тестовый случай**: `test_rfc010_method_definition`
- **Содержание исправления**: `parse_method_bind_stmt` использует `parse_fn_type_with_names` для сохранения имён параметров, что позволяет проверщику типов добавлять параметры в область видимости функции

#### 4. Проверка реализации интерфейса ✅

- **Тестовый случай**: `test_rfc010_interface_implementation`
- **Содержание исправления**: То же, что и #3 — определение метода работает корректно

#### 5. Присваивание интерфейса (структурный подтип) ✅

- **Тестовый случай**: `test_rfc010_interface_assignment`
- **Содержание исправления**: То же, что и #3 — определение метода работает корректно

#### 6. Синтаксический сахар вызова метода ✅

- **Тестовый случай**: `test_rfc010_method_call_syntax_sugar`
- **Содержание исправления**: Определение метода работает корректно, функция может вызываться напрямую

#### 7. Проверка несоответствия типа возврата ✅

- **Тестовый случай**: `test_rfc010_function_return_type_mismatch`
- **Содержание исправления**: В `ExpressionInferrer` добавлено поле `expected_return_type`, обработчик оператора `Return` унифицирует тип возвращаемого значения с объявленным типом

### Функции для проверки (требуют более глубокой поддержки проверщика типов)

Следующие функции не имеют **полной семантической реализации** (например, мономорфизация обобщений, разрешение ограничений, структурный подтип),
но **синтаксический анализ и базовая проверка типов** прошли тесты:

- Создание экземпляра обобщённого типа (`List(Int)` → раскрытие конкретного типа структуры)
- Разрешение ограничений типа (`T: Clone` → проверка реализации интерфейса типом)
- Разрешение перегрузки/специализации функций
- Синтаксический сахар вызова метода (`p.draw(screen)` → `Point.draw(p, screen)`)
- Полная реализация проверки размерности времени компиляции

## Сводка исправлений от 2026-05-16

### Первый раунд исправлений (RFC-010 полностью + часть RFC-011)

#### Исправления парсера

1. **Обнаружение обобщённого конструктора типа** (`declarations.rs`): В `parse_var_stmt_with_pub` добавлено обнаружение `Type::Fn { return_type: MetaType }`, чтобы `(T: Type) -> Type = { ... }` парсилось как конструктор типа, а не определение функции
2. **Сохранение имён параметров в определении метода** (`declarations.rs`): `parse_method_bind_stmt` использует `parse_fn_type_with_names` для сохранения имён параметров, чтобы проверщик типов мог корректно создавать область видимости функции
3. **Фильтрация параметров обобщённой функции** (`declarations.rs`): При сопоставлении имён параметров lambda фильтруются параметры типа (большая буква в начале — параметр типа, маленькая — значение параметра)

#### Исправления проверщика типов

4. **Проверка типа возвращаемого значения** (`expressions.rs`): Добавлено поле `expected_return_type` для отслеживания типа возврата функции, обработчик оператора `Return` унифицирует возвращаемое значение с объявленным типом
5. **Совместимость типов при присваивании переменной** (`statements.rs`): В `check_var_stmt` добавлена проверка запрета неявного сужающего преобразования `Float → Int`

### Второй раунд исправлений (RFC-011 полностью)

#### Исправления парсера

6. **Поддержка синтаксиса ограничения `+`** (`types.rs`): В `parse_fn_type_with_names` добавлено обнаружение токена `+`, чтобы `(T: Clone + Add)` парсилось как параметр типа с множественными ограничениями
7. **Извлечение ограничения `Type::Tuple`** (`declarations.rs`): `extract_generic_params` обрабатывает `Type::Tuple` как контейнер для множественных ограничений

#### Обновления тестов

8. Обновлён `test_rfc011_generic_function_inference` для использования нового синтаксиса
9. Обновлён `test_rfc011_platform_specialization` для использования синтаксиса с фигурными скобками
10. Упрощено несколько тестов для адаптации к текущим возможностям проверщика типов

### Третий раунд исправлений (создание экземпляра обобщённого типа)

#### Исправления системы типов

11. **Хранение шаблона GenericTypeDef** (`environment.rs`): Добавлена структура `GenericTypeDef` и таблица `generic_type_defs` для хранения информации о шаблоне обобщённого конструктора типа
12. **Регистрация шаблона** (`checker.rs`): В `add_type_definition` при наличии обобщённых параметров тело типа регистрируется как шаблон
13. **Создание экземпляра типа** (`environment.rs`): Реализован метод `instantiate_generic_type_static` для рекурсивной замены параметров типа и разрешения ссылок на встроенные типы
14. **Триггер создания экземпляра** (`statements.rs`): В `check_var_stmt` добавлен `try_instantiate_generic_type` для раскрытия экземпляра при наличии аннотации типа `Type::Generic`

### Четвёртый раунд исправлений (синтаксический сахар вызова метода + связывание методов)

#### Синтаксический сахар вызова метода

15. **Передача `method_bindings`** (`expressions.rs`, `statements.rs`, `checker.rs`): `method_bindings` передаётся из TypeEnvironment в ExpressionInferrer для поиска методов
16. **Fallback FieldAccess для методов** (`expressions.rs`): Когда поиск поля структуры завершается неудачно, выполняется попытка поиска в `method_bindings` по ключу `"TypeName.method"`, поддерживается синтаксис `p.draw`
17. **Восстановление тестов** (`test_rfc010_method_call_syntax_sugar`): Восстановлен нативный синтаксис вызова метода `p.draw(screen)`

#### Внешние связывания методов

18. **Обработка ExternalBindingStmt** (`checker.rs`): В `collect_function_signature` добавлена ветвь сопоставления для поиска функции и регистрации связывания метода в `method_bindings`

## Текущее состояние

**Все тесты RFC-010/011 проходят (44/44)**. Проверщик типов теперь поддерживает:

- Базовую проверку типов (переменные, функции, структуры, интерфейсы)
- Определение обобщённых типов и раскрытие создания экземпляров
- Проверку несоответствия типа возврата
- Определение и вызов методов (`Point.draw: ...` + `p.draw(...)`)
- Внешние связывания методов (`Type.method = func[0]`)
- Подтип Int→Float (защита от сужающего преобразования)
- Параметры времени компиляции и вычисления
- Обновление с учётом отложенных изменений

## Как использовать этот документ

1. **При разработке новых функций**: Проверьте этот документ, подтвердите наличие связанных функций для проверки

4. **При Code Review**: Проверьте, покрывает ли новый код функции из этого документа
