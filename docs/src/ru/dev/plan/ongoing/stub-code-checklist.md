### 1. Методы пошагового выполнения отладчика (4 `todo!()`)

**Файл**: `src/backends/interpreter/executor/debug.rs`

| Номер строки | Сигнатура функции | Ожидаемая функциональность | Состояние |
|------|----------|----------|------|
| 32-34 | `fn step(&mut self) -> ExecutorResult<()>` | Пошаговое выполнение одной инструкции | `todo!()` |
| 36-38 | `fn step_over(&mut self) -> ExecutorResult<()>` | Шаг с пропуском (без захода внутрь функции) | `todo!()` |
| 40-42 | `fn step_out(&mut self) -> ExecutorResult<()>` | Шаг с выходом (выполнение до конца текущей функции) | `todo!()` |
| 44-46 | `fn run(&mut self) -> ExecutorResult<()>` | Продолжение выполнения до следующей точки останова | `todo!()` |

**Контекст**: Другие методы trait `DebuggableExecutor` (`set_breakpoint`, `has_breakpoint`, `current_ip`, `current_function`, `breakpoints`) уже реализованы; не реализовано только управление пошаговым выполнением.

**Предложения по реализации**:
- `step()`: выполнить инструкцию по текущему IP, затем IP++.
- `step_over()`: если текущая инструкция — вызов функции, установить временную точку останова на следующей инструкции, затем `run`.
- `step_out()`: запомнить текущую глубину стека вызовов, выполнять `run`, пока глубина стека не уменьшится.
- `run()`: циклически выполнять до точки останова или завершения программы.

### 2. Ядро анализа потока управления (2 пустых тела функций)

**Файл**: `src/middle/passes/lifetime/control_flow.rs`

| Номер строки | Сигнатура функции | Ожидаемая функциональность | Состояние |
|------|----------|----------|------|
| 145-154 | `fn analyze_instruction(&self, _instr: &Instruction, _state: &mut HashMap<Operand, ValueState>, _pos: (usize, usize))` | Анализ изменения состояния жизненного цикла одной инструкции | Пустая реализация |
| 155-163 | `fn merge_block_state(&mut self, _block_state: &HashMap<Operand, ValueState>, _block_idx: usize)` | Слияние ValueState из разных базовых блоков | Пустая реализация |

**Контекст**: Комментарии поясняют: «в настоящее время пустая реализация, может быть расширена при необходимости», «анализ потока управления уже имеет базовую реализацию в MoveChecker».

**Предложения по реализации**:
- `analyze_instruction`:
  - Сопоставление типа инструкции (Move, Copy, Call, Branch и т.д.).
  - Обновление ValueState (Moved/Empty/Partial) соответствующего Operand в `_state`.
  - Обработка Move для аргументов вызова функции.
- `merge_block_state`:
  - Для случая слияния из нескольких блоков-предшественников взять наименьшую верхнюю грань (LUB) ValueState.
  - Если любой предшественник находится в состоянии Moved, после слияния состояние — Moved.
  - При конфликте состояний предшественников сообщить об ошибке или пометить как MaybeMoved.

**Влияние**: В настоящее время lifetime pass вырождается в no-op и не может обнаружить ошибки use-after-move.
