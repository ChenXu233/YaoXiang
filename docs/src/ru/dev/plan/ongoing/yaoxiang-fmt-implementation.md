> **Задача**: Реализация инструмента форматирования `yaoxiang fmt`
> **На основе RFC**: RFC-010 Унифицированный синтаксис, RFC-017 Проект поддержки LSP
> **Дата**: 2026-03-01
> **Статус**: Всё завершено ✅
> **Версия**: v0.6.7
> **Дата завершения**: 2026-03-01

## Обзор

Данный план, основанный на документе требований issue_13, разбивает инструмент форматирования `yaoxiang fmt` на несколько этапов реализации. Каждый этап содержит детальные цели, критерии приёмки и тестовые задания.

### Выбор архитектуры

Применяется схема **AST + исходное отображение**:
1. **Lexer** записывает позиции комментариев/пробелов → генерирует поток Token + исходное отображение
2. **Parser** генерирует AST (с информацией о Span)
3. **Formatter** обходит AST + исходное отображение согласно конфигурации → выводит отформатированный код

**Ключевые преимущества**:
- Сохранение полной информации о комментариях
- Поддержка сложной реструктуризации кода
- Глубокая интеграция с LSP

### 0.1 Структура данных SourceMap (Исходное отображение)

**Цели реализации**:
- Спроектировать и реализовать структуру `SourceMap` для записи отображения позиций исходного кода
- Записывать содержимое: позиции комментариев, области пробелов, позиции оригинальных Token
- Поддерживать преобразование из смещения символов в строку/столбец

**Проектирование структуры данных**:
```
SourceMap {
    source: String,                    // Исходный код
    comments: Vec<Comment>,            // Список комментариев
    tokens: Vec<TokenWithSpan>,        // Token + позиция
    line_offsets: Vec<usize>,          // Начальное смещение каждой строки
    blank_lines: Vec<usize>,           // Список номеров пустых строк
}

Comment {
    content: String,
    span: Span,
    style: CommentStyle,  // SingleLine, MultiLine, Doc
}
```

**Цели реализации**:

- При возврате из `tokenize()` также возвращать `SourceMap`
- Сохранить обратную совместимость, не влияя на существующий поток Token
