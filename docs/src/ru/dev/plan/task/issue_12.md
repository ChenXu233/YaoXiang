## Обзор

`yaoxiang check` — это инструмент командной строки для выполнения статической проверки (проверка типов, проверка владения, lint) исходного кода YaoXiang без генерации кода. Он должен обеспечивать быструю обратную связь и подходить для интеграции в CI-конвейеры или использования в качестве шага проверки в процессе разработки.

## Цели

- Реализовать команду `yaoxiang check`, принимающую один или несколько файлов/каталогов в качестве входных данных.
- Повторно использовать модуль проверки типов компилятора, выводя ошибки и предупреждения в стандартный поток ошибок.
- Поддерживать опцию `--json` для вывода диагностической информации в формате JSON для удобной интеграции с другими инструментами.
- Поддерживать режим `--watch` (опционально) для наблюдения за изменениями файлов и автоматической повторной проверки.
- Сообщения об ошибках должны содержать имя файла, номер строки, номер столбца, уровень серьёзности и описание.

## Конкретные шаги

1. **Проектирование интерфейса командной строки** (3 дня)
   - Использовать `clap` или аналогичную библиотеку для разбора аргументов: `yaoxiang check [OPTIONS] <PATH>...`.
   - Поддержать опции `--json`, `--watch`, `--color` и другие.

2. **Интеграция проверки типов** (1 неделя)
   - Инкапсулировать модуль проверки типов в виде независимо вызываемой функции, принимающей исходный код в виде строки и возвращающей список диагностик.
   - Поддержать проверку нескольких файлов: необходимо проанализировать все файлы и построить глобальную таблицу символов.
   - Обработать пути к стандартной библиотеке и встроенные определения.

3. **Форматирование вывода** (3 дня)
   - Реализовать цветной вывод ошибок в стиле Rustc, включающий позицию ошибки, уровень серьёзности и описание.
   - Реализовать вывод в формате JSON в соответствии с распространёнными соглашениями (аналогично `rustc --error-format=json`).

4. **Поддержка режима наблюдения (опционально)** (1 неделя)
   - Использовать библиотеку мониторинга файловой системы (например, `notify`) для наблюдения за изменениями файлов и автоматической повторной проверки.
   - Избегать повторного полного анализа, по возможности выполнять инкрементную проверку.

5. **Тестирование и интеграция** (1 неделя)
   - Написать тестовые сценарии: выполнить проверку файлов с ошибками и убедиться, что вывод соответствует ожиданиям.
   - Интегрировать в CI-конвейер, чтобы проверка выполнялась при каждом коммите.

## Критерии приёмки

- При выполнении `yaoxiang check` для одного файла без ошибок не должно быть вывода (или должно выводиться сообщение об успехе), код завершения — 0.
- При выполнении проверки файла с ошибками типов должен выводиться чёткий текст ошибки (имя файла, номер строки, номер столбца, описание), код завершения — ненулевой.
- Опция `--json` должна выводить корректный JSON-массив, каждый элемент которого содержит позицию ошибки и сообщение.
- Скорость проверки должна быть близка к скорости проверки типов компилятора, подходящей для частого использования.
- Корректная обработка нескольких файлов в проекте с распознаванием межфайловых ссылок.

## Зависимости

- Модуль проверки типов должен поддерживать чтение файлов из файловой системы и построение глобального окружения.
- Определения стандартной библиотеки должны быть доступны (встроены в компилятор или указаны через путь).
