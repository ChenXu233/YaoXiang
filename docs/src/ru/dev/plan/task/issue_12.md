## Обзор

`yaoxiang check` — это инструмент командной строки для выполнения статической проверки исходного кода YaoXiang (проверка типов, проверка владения, lint), не генерирующий никакого кода. Он должен обеспечивать быструю обратную связь, подходящую для интеграции в CI-контур или использования в качестве шага верификации в процессе разработки.

## Цели

- Реализовать команду `yaoxiang check`, принимающую один или несколько файлов/каталогов в качестве входных данных.
- Повторно использовать модуль проверки типов компилятора, выводить ошибки и предупреждения в стандартный поток ошибок.
- Поддерживать опцию `--json` для вывода диагностической информации в формате JSON для удобной интеграции с другими инструментами.
- Поддерживать режим `--watch` (опционально) для наблюдения за изменениями файлов и повторной проверки.
- Сообщения об ошибках должны содержать имя файла, номер строки, номер столбца, уровень серьёзности и описание.

## Конкретные шаги

1. **Проектирование интерфейса командной строки** (3 дня)
   - Использовать `clap` или аналогичную библиотеку для разбора аргументов: `yaoxiang check [OPTIONS] <PATH>...`.
   - Поддержать опции `--json`, `--watch`, `--color` и другие.

2. **Интеграция проверщика типов** (1 неделя)
   - Обернуть модуль проверки типов в функцию, которую можно вызывать независимо, принимающую исходный код в виде строки и возвращающую список диагностик.
   - Поддержать проверку нескольких файлов: необходимо разобрать все файлы и построить глобальную таблицу символов.
   - Обрабатывать пути к стандартной библиотеке и встроенные определения.

3. **Форматирование вывода** (3 дня)
   - Реализовать цветной вывод ошибок в стиле Rustc, содержащий позицию ошибки, уровень серьёзности и описание.
   - Реализовать вывод в формате JSON, соответствующий распространённым соглашениям (например, аналогично `rustc --error-format=json`).

4. **Поддержка режима наблюдения (опционально)** (1 неделя)
   - Использовать библиотеку мониторинга файловой системы (например, `notify`) для отслеживания изменений файлов и автоматической повторной проверки.
   - Избегать повторных полных проверок, по возможности выполнять инкрементную проверку.

5. **Тестирование и интеграция** (1 неделя)
   - Написать тестовые сценарии: выполнить проверку файлов с ошибками и убедиться, что вывод соответствует ожиданиям.
   - Интегрировать в CI-контур, чтобы каждая фиксация проходила проверку.

## Критерии приёмки

- При выполнении `yaoxiang check` над единственным файлом без ошибок вывод должен отсутствовать (или содержать сообщение об успехе), а код завершения должен быть равен 0.
- При проверке файла, содержащего ошибки типов, должен выводиться чёткий текст ошибки (имя файла, номер строки, номер столбца, описание), а код завершения должен быть отличен от 0.
- Опция `--json` должна выводить корректный JSON-массив, каждый элемент которого содержит позицию ошибки и сообщение.
- Скорость проверки должна быть близка к скорости проверки типов компилятором, подходящей для частого использования.
- Должна обеспечиваться корректная обработка нескольких файлов в проекте с распознаванием межфайловых ссылок.

## Зависимости

- Модуль проверки типов должен поддерживать чтение файлов из файловой системы и построение глобального окружения.
- Определения стандартной библиотеки должны быть доступны (возможно, встроены в компилятор или указаны через путь).
