## Обзор

`yaoxiang check` — это инструмент командной строки для статической проверки исходного кода YaoXiang (проверка типов, проверка владения, lint), без генерации кода. Он должен обеспечивать быструю обратную связь и подходить для интеграции в CI/CD или в качестве этапа проверки в процессе разработки.

## Цели

- Реализовать команду `yaoxiang check`, принимающую один или несколько файлов/каталогов в качестве входных данных.
- Повторно использовать модуль проверки типов компилятора, выводить ошибки и предупреждения в стандартный поток ошибок.
- Поддержка опции `--json` для вывода диагностической информации в формате JSON, что упрощает интеграцию с другими инструментами.
- Поддержка режима `--watch` (опционально) для отслеживания изменений файлов и автоматической повторной проверки.
- Сообщения об ошибках должны содержать имя файла, номер строки, номер столбца, уровень серьёзности и описание.

## Конкретные шаги

1. **Разработка интерфейса командной строки** (3 дня)

   - Использование `clap` или аналогичной библиотеки для разбора аргументов: `yaoxiang check [OPTIONS] <PATH>...`.
   - Поддержка опций `--json`, `--watch`, `--color` и других.

2. **Интеграция проверки типов** (1 неделя)

   - Инкапсуляция модуля проверки типов в виде независимо вызываемой функции, принимающей исходный код в виде строки и возвращающей список диагностик.
   - Поддержка проверки нескольких файлов: необходимо разобрать все файлы и построить глобальную таблицу символов.
   - Обработка путей к стандартной библиотеке и встроенных определений.

3. **Форматирование вывода** (3 дня)

   - Реализация цветного вывода ошибок, аналогичного Rustc, с указанием местоположения, уровня серьёзности и описания.
   - Реализация вывода в формате JSON, соответствующего распространённым стандартам (например, аналогично `rustc --error-format=json`).

4. **Поддержка режима отслеживания** (опционально) (1 неделя)

   - Использование библиотеки мониторинга файловой системы (например, `notify`) для отслеживания изменений файлов и автоматической повторной проверки.
   - Избегание повторных полных проверок, по возможности использовать инкрементную проверку.

5. **Тестирование и интеграция** (1 неделя)

   - Написание тестовых случаев: проверка файлов с ошибками и верификация соответствия вывода ожиданиям.
   - Интеграция в CI/CD конвейер для обеспечения проверки каждого коммита.

## Критерии приёмки

- При запуске `yaoxiang check` на корректном файле не должно быть вывода (или выводится сообщение об успехе), код завершения — 0.
- При проверке файла с типа ошибками должен выводиться чёткий message об ошибке (имя файла, номер строки, номер столбца, описание), код завершения отличен от 0.
- Опция `--json` выводит валидный JSON-массив, каждый элемент которого содержит местоположение ошибки и сообщение.
- Скорость проверки должна быть близка к скорости проверки типов компилятором, подходящей для частого использования.
- Корректная обработка нескольких файлов в проекте и распознавание межфайловых ссылок.

## Зависимости

- Модуль проверки типов должен поддерживать чтение файлов из файловой системы и построение глобального окружения.
- Определения стандартной библиотеки должны быть доступны (могут быть встроены в компилятор или указаны через путь).
