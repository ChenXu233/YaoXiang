# Реализация поддержки LSP (Language Server Protocol)

## Обзор

Для улучшения опыта разработчиков YaoXiang необходимо реализовать язык сервер, соответствующий Language Server Protocol. Этот сервер будет встроен в интерфейсный компонент компилятора, обеспечивая возможности анализа кода в реальном времени, дополнения, перехода к определениям, диагностики и интеграции с популярными редакторами (VS Code, Vim, Emacs и др.).

## Цели

- Реализовать двоичный файл языкового сервера YaoXiang `yaoxiang-lsp`.
- Поддержать следующие LSP-запросы:
  - `textDocument/didOpen` / `didChange` / `didSave` / `didClose`: синхронизация содержимого файла.
  - `textDocument/documentSymbol`: перечисление символов документа (функции, типы, переменные и т.д.).
  - `textDocument/completion`: предоставление предложений по дополнению кода (на основе символов в области видимости).
  - `textDocument/definition`: переход к месту определения.
  - `textDocument/hover`: отображение информации о типе и документационных комментариев при наведении.
  - `textDocument/publishDiagnostics`: push-уведомление в реальном времени о диагностике синтаксических ошибок, ошибок типов и др.
- Реализовать возможность инкрементной компиляции/проверки компилятором во избежание полного повторного разбора при каждом запросе.

## Конкретные шаги

1. **Проектирование архитектуры сервера** (1 неделя)
   - Построение на основе существующих библиотек интерфейсной части компилятора (лексический анализ, парсинг, проверка типов).
   - Использование асинхронного ввода-вывода (например, Tokio) для обработки LSP-запросов.
   - Проектирование структур данных для кэширования документов и инкрементного обновления.

2. **Реализация базового протокола LSP-коммуникации** (1 неделя)
   - Реализация отправки и получения JSON-RPC сообщений.
   - Обработка запросов инициализации, запросов закрытия и другого управления жизненным циклом.

3. **Интеграция интерфейсной части компилятора** (2 недели)
   - Инкапсуляция лексического анализа, парсинга и проверки типов в виде повторно используемых библиотечных функций.
   - Поддержка «компиляционной сессии» для каждого документа с возможностью инкрементного обновления (повторный разбор только изменённых частей).
   - Реализация маппинга узлов AST на позиции в исходном коде для перехода и отображения информации при наведении.

4. **Реализация диагностической функциональности** (1 неделя)
   - Перехват ошибок и предупреждений в процессе проверки типов с преобразованием в диагностические сообщения LSP.
   - Автоматическая отправка диагностики после изменения файла.

5. **Реализация дополнения и навигации по символам** (2 недели)
   - Построение таблицы символов области видимости для предоставления списка кандидатов дополнения (включая ключевые слова, встроенные типы, переменные текущей области видимости).
   - Реализация перехода к определению: поиск файла и позиции определения символа по позиции курсора.
   - Реализация отображения информации при наведении: показ сигнатуры типа и связанных документационных комментариев (при наличии поддержки).

6. **Тестирование и интеграция** (1 неделя)
   - Написание плагина-клиента VS Code (простая обёртка) для тестирования.
   - Обеспечение работоспособности всех функций в типичных сценариях без существенных проблем с производительностью.

## Критерии приёмки

- Сервер корректно запускается и отвечает на запрос инициализации.
- После открытия файла YaoXiang диагностическая информация (синтаксические ошибки, ошибки типов) отображается в редакторе в реальном времени.
- При наведении на имя переменной или функции отображается информация о её типе.
- При переходе к определению по идентификатору открывается соответствующий файл с переходом на строку определения.
- После ввода точки доступно дополнение полей типа записи (необходима информация о типе).
- Время отклика сервера на файлах большого размера (тысячи строк) находится в приемлемых пределах (<500 мс).

## Зависимости

- Интерфейсная библиотека компилятора должна быть стабильной и предоставлять API для получения AST, таблицы символов и информации о типах.
- Необходима реализация возможности инкрементной компиляции (как минимум кэширование на уровне документа).
