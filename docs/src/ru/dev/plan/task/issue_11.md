# Задача: Реализация поддержки LSP (Language Server Protocol)

## Обзор

Для улучшения опыта разработчиков YaoXiang необходимо реализовать сервер языка, соответствующий Language Server Protocol. Этот сервер будет встроен в интерфейс компилятора и обеспечит возможности анализа кода в реальном времени, дополнения, перехода к определениям, диагностики и других функций, которые можно интегрировать с популярными редакторами (VS Code, Vim, Emacs и др.).

## Цели

- Реализовать бинарный файл сервера YaoXiang — `yaoxiang-lsp`.
- Поддержать следующие LSP-запросы:
  - `textDocument/didOpen` / `didChange` / `didSave` / `didClose`: синхронизация содержимого файлов.
  - `textDocument/documentSymbol`: перечисление символов документа (функции, типы, переменные и др.).
  - `textDocument/completion`: предоставление предложений для дополнения кода (на основе символов в текущей области видимости).
  - `textDocument/definition`: переход к месту определения.
  - `textDocument/hover`: отображение информации о типе и документационных комментариев при наведении.
  - `textDocument/publishDiagnostics`: отправка в реальном времени диагностических сообщений об ошибках синтаксиса, типов и др.
- Реализовать возможность инкрементной компиляции/проверки совместно с компилятором, чтобы избежать полного переразбора при каждом запросе.

## Конкретные шаги

### 1. Проектирование архитектуры сервера (1 неделя)

- Построить на основе существующих библиотек интерфейса компилятора (лексический анализ, парсинг, проверка типов).
- Использовать асинхронный ввод-вывод (например, Tokio) для обработки LSP-запросов.
- Спроектировать структуры данных для кэширования документов и инкрементного обновления.

### 2. Реализация базового протокола LSP-коммуникации (1 неделя)

- Реализовать приём и передачу JSON-RPC сообщений.
- Обработать запросы жизненного цикла: инициализация, закрытие и др.

### 3. Интеграция интерфейса компилятора (2 недели)

- Объединить лексический анализ, парсинг и проверку типов в виде повторно используемых библиотечных функций.
- Для каждого документа поддерживать «сессию компиляции» с возможностью инкрементного обновления (повторный разбор только изменённых частей).
- Реализовать отображение узлов AST на позиции в исходном коде для функций перехода и отображения при наведении.

### 4. Реализация диагностической функции (1 неделя)

- Перехватывать ошибки и предупреждения в процессе проверки типов и преобразовывать их в LSP-диагностические сообщения.
- Автоматически отправлять диагностику после изменений файла.

### 5. Реализация дополнения и навигации по символам (2 недели)

- Построить таблицу символов области видимости для предоставления списка кандидатов при дополнении (включая ключевые слова, встроенные типы, переменные текущей области видимости и др.).
- Реализовать переход к определению: по позиции курсора находить файл и позицию определения символа.
- Реализовать отображение информации при наведении: показывать сигнатуру типа и связанные документационные комментарии (если поддерживаются).

### 6. Тестирование и интеграция (1 неделя)

- Написать простой клиентский плагин для VS Code для тестирования.
- Убедиться, что все функции работают в типичных сценариях без существенных проблем с производительностью.

## Критерии приёмки

- Сервер корректно запускается и отвечает на запросы инициализации.
- При открытии файла YaoXiang диагностическая информация (синтаксические ошибки, ошибки типов) отображается в редакторе в реальном времени.
- При наведении на имя переменной или функции отображается информация о её типе.
- При переходе к определению идентификатора открывается соответствующий файл с переходом на строку определения.
- После ввода точки предлагается дополнение полей record type (требуется информация о типе).
- Время отклика сервера на больших файлах (тысячи строк) находится в приемлемых пределах (<500 мс).

## Зависимости

- Библиотека интерфейса компилятора должна быть стабильной и предоставлять API для получения AST, таблицы символов и информации о типах.
- Необходимо реализовать возможность инкрементной компиляции (как минимум кэширование на уровне документа).
