# Задача: Реализация инструмента форматирования кода yaoxiang fmt

## Обзор
`yaoxiang fmt` — это инструмент автоматического форматирования исходного кода YaoXiang, предназначенный для унификации стиля кода в сообществе и устранения споров о форматировании. Он должен переформатировать код в соответствии со спецификацией языка (RFC-010 и др.) и пользовательскими настройками.

## Цели
- Реализовать команду `yaoxiang fmt`, принимающую файлы или каталоги в качестве входных данных, форматирующую код и выводящую результат (или выполняющую модификацию на месте).
- Поддержать режим `--check`: проверить, отформатирован ли код, не изменяя файлы, вернуть ненулевой код завершения.
- Поддержать конфигурационные файлы (например, `yaoxiang.toml`) для настройки правил форматирования (ширина отступа, стиль переноса строк и т.д.).
- Сохранять комментарии и замысел исходных пустых строк (интеллектуальная обработка).
- Сгенерированный код должен сохранять семантическую неизменность и соответствовать синтаксису языка.

## Конкретные шаги

1. **Проектирование правил форматирования** (1 неделя)
   - На основе унифицированного синтаксиса RFC-010 определить соглашения об отступах, пробелах и переносах строк.
   - С учётом дизайна Rustfmt определить настраиваемые параметры: размер отступа, максимальная ширина строки, выравнивание полей и т.д.
   - Написать проектную документацию для обсуждения сообществом.

2. **Реализация повторного вывода AST в код** (2 недели)
   - Использовать существующий парсер для получения полного AST (включая позиционную информацию комментариев; если текущий парсер отбрасывает комментарии, его необходимо улучшить).
   - Спроектировать «печатник» (printer), который обходит AST и выводит отформатированный код согласно правилам.
   - Обрабатывать особые случаи: многострочные строки, встроенные комментарии, пустые строки и т.д.

3. **Поддержка конфигурации** (1 неделя)
   - Читать `yaoxiang.toml` из корневого каталога проекта и объединять с конфигурацией по умолчанию.
   - Передавать конфигурацию в печатник для влияния на отступы, переносы строк и другое поведение.

4. **Реализация режима `--check`** (3 дня)
   - После форматирования файла сравнить исходное содержимое с отформатированным; при наличии различий выдать ошибку и перечислить файлы с различиями.
   - Код завершения, отличный от 0, означает, что хотя бы один файл требует форматирования.

5. **Тестирование и интеграция** (1 неделя)
   - Написать обширный набор тестовых случаев, охватывающий все синтаксические особенности и обеспечивающий семантическую идентичность до и после форматирования.
   - Тестировать корректность режима `--check` в различных ситуациях.
   - Обеспечить достаточную производительность — обработку не менее нескольких тысяч строк кода в секунду.

## Критерии приёмки
- Способность форматировать полные YaoXiang-файлы, содержащие переменные, функции, определения типов, методы и управляющие конструкции.
- Отформатированный код должен проходить проверку синтаксическим анализатором и проверкой типов.
- Комментарии и пустые строки должны сохраняться разумно, без случайного удаления или перемещения.
- Режим `--check` должен корректно обнаруживать неформатированные файлы и сообщать о них.
- Поддержка настраиваемой ширины отступа (2 пробела / 4 пробела / tab).
- Обработка типичных граничных случаев: многострочные вызовы функций, длинные выражения с переносами, выравнивание списков полей и т.д.

## Зависимости
- Парсер должен сохранять информацию о комментариях и пробелах (или предоставлять исходное отображение, позволяющее восстановить код из AST). Если текущий парсер отбрасывает эту информацию, его необходимо расширить.
- Необходимо понимание полной структуры синтаксического дерева языка.
