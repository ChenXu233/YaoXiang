# Задача: Реализация инструмента форматирования кода yaoxiang fmt

## Обзор

`yaoxiang fmt` — это инструмент автоматического форматирования исходного кода YaoXiang, предназначенный для унификации стиля кода в сообществе и устранения разногласий по форматированию. Он должен переформатировать код в соответствии со спецификацией языка (RFC-010 и др.) и пользовательскими настройками.

## Цели

- Реализовать команду `yaoxiang fmt`, принимающую файлы или каталоги в качестве входных данных, форматирующую код и выводящую результат (или модифицирующую файлы на месте).
- Поддерживать режим `--check`: проверить, отформатирован ли код, не изменяя файлы, с возвратом ненулевого кода завершения.
- Поддерживать конфигурационные файлы (например, `yaoxiang.toml`) для настройки правил форматирования (ширина отступа, стиль переноса строк и т. д.).
- Сохранять намерение комментариев и исходных пустых строк (интеллектуальная обработка).
- Сгенерированный код должен сохранять семантическую неизменность и соответствовать синтаксису языка.

## Конкретные шаги

1. **Проектирование правил форматирования** (1 неделя)
   - На основе унифицированного синтаксиса RFC-010 определить соглашения об отступах, пробелах и переносах строк.
   - Обратиться к дизайну Rustfmt для определения настраиваемых параметров: размер отступа, максимальная ширина строки, выравнивание полей и т. д.
   - Написать проектную документацию для обсуждения сообществом.

2. **Реализация перепечатывания AST в код** (2 недели)
   - Использовать существующий парсер для получения полного AST (включая позиционную информацию комментариев; если текущий парсер отбрасывает комментарии, его необходимо расширить).
   - Разработать «принтер», обходящий AST и выводящий отформатированный код в соответствии с правилами.
   - Обработать особые случаи: многострочные строки, встроенные комментарии, пустые строки и т. д.

3. **Поддержка конфигурации** (1 неделя)
   - Читать `yaoxiang.toml` из корневого каталога проекта и объединять с конфигурацией по умолчанию.
   - Передавать конфигурацию принтеру для влияния на отступы, переносы строк и другое поведение.

4. **Реализация режима `--check`** (3 дня)
   - После форматирования файла сравнить исходное содержимое с отформатированным; при наличии различий выдать ошибку и перечислить файлы с различиями.
   - Код завершения, отличный от 0, означает, что хотя бы один файл требует форматирования.

5. **Тестирование и интеграция** (1 неделя)
   - Написать обширный набор тестов, охватывающий все синтаксические конструкции и обеспечивающий семантическую неизменность до и после форматирования.
   - Протестировать корректность режима `--check` в различных ситуациях.
   - Обеспечить достаточную производительность — обработку не менее нескольких тысяч строк кода в секунду.

## Критерии приёмки

- Возможность форматировать полные файлы YaoXiang, содержащие переменные, функции, определения типов, методы, управляющие конструкции.
- Отформатированный код должен проходить проверку синтаксическим анализатором и проверкой типов.
- Комментарии и пустые строки должны сохраняться разумно, без случайного удаления или перемещения.
- Режим `--check` должен правильно обнаруживать неотформатированные файлы и сообщать о них.
- Поддержка настраиваемой ширины отступа (2 пробела / 4 пробела / tab).
- Обработка типичных граничных случаев: многострочные вызовы функций, переносы длинных выражений, выравнивание списков полей и т. д.

## Зависимости

- Парсер должен сохранять информацию о комментариях и пробелах (или предоставлять исходное отображение, позволяющее восстановить код из AST). Если текущий парсер отбрасывает эту информацию, его необходимо сначала расширить.
- Необходимо понимание полной структуры дерева синтаксиса языка.
