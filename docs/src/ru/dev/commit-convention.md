## Формат коммита

**Очень важно!!!!!!!! Не забудьте!!!!!**
Все сообщения коммитов должны соответствовать следующему формату:

```
:emoji-код: type(scope): тема (на китайском)

[Необязательное тело]

[Необязательный footer]
```

> ⚠️ **Важно**: Необходимо использовать **emoji-код** (например, `:sparkles:`), а не вводить emoji-символ напрямую.
>
> **Рекомендуется использовать китайский язык для сообщений коммитов** для единообразия командной коммуникации.

### Составные части

| Часть | Описание | Обязательно |
|------|------|------|
| emoji-код | Идентификатор типа коммита | ✅ |
| type | Тип коммита | ✅ |
| scope | Область действия | ✅ |
| subject | Краткое описание (на китайском, не более 50 символов) | ✅ |
| body | Подробное описание (необязательно) | ❌ |
| footer | Критические изменения или закрытие задач (необязательно) | ❌ |

## Правила оформления сообщений

### Управление версиями

**Перед каждым коммитом необходимо сначала увеличить номер версии**:

| Тип версии | Место обновления | Описание |
|----------|----------|------|
| **major** | `pubspec.yaml` (version) + `release_v*.md` | Крупные обновления, несовместимые изменения API |
| **minor** | `pubspec.yaml` (version) | Новая функциональность, обратная совместимость |
| **patch** | `pubspec.yaml` (version) | Исправление багов, обратная совместимость |

### Формат номера версии

Используется семантическое версионирование `MAJOR.MINOR.PATCH`:

```
# Крупная версия (breaking changes)
1.0.0 -> 2.0.0

# Минорная версия (new features)
1.0.0 -> 1.1.0

# Патчевая версия (bug fixes)
1.0.0 -> 1.0.1
```

### Процесс коммита

```bash
# 1. После изменения кода сначала увеличьте версию
# Используйте инструмент semantic_release для автоматического управления версиями и Changelog
npx semantic-release

# Или обновите версию вручную
# Отредактируйте поле version в pubspec.yaml

# 2. Коммитите код (изменения версии будут автоматически включены в следующий релиз)
git add .
git commit -m ":tada: Release v1.0.0"
git push
```

> 💡 Увеличение версии и генерация Changelog выполняются автоматически CI. При коммите достаточно убедиться, что изменения кода включают обновление версии.

## Правила оформления сообщений

### Языковые правила

**Рекомендуется использовать китайский язык для сообщений коммитов** для единообразия командной коммуникации.

- Subject на китайском, кратко и понятно
- Body на китайском, с подробным описанием
- При наличии специальных технических терминов можно оставить английский

### Subject (тема)

- На китайском, кратко и понятно
- Длина не более 50 символов
- В конце не ставится точка

### Body (тело)

- Подробное описание причины и способа изменения
- Каждая строка не более 72 символов
- Используйте - или * для перечисления пунктов

### Footer (нижний колонтитул)

- **Критические изменения**: начинаются с `BREAKING CHANGE:`
- **Закрытие Issue**: используйте `关闭 #123` или `修复 #456`

## Примеры

### ✨ feat — Новая функциональность

```
:sparkles: feat(db): 添加批量删除待办功能

实现批量删除待办事项功能：
- 在 TodoRepository 中添加 batchDelete 方法
- 添加删除确认对话框
- 更新 UI 支持多选操作

关闭 #42
```

### 🐛 fix — Исправление бага

```
:bug: fix(provider): 修复番茄钟计时器在后台无法恢复的问题

当应用从后台恢复时，番茄钟计时器无法继续计时。
在 initState 中添加了状态恢复逻辑。

修复 #128
```

### 📝 docs — Обновление документации

```
:memo: docs: 更新 README 新功能说明

新增以下章节：
- 专注模式
- 数据统计
- 背景音乐
```

### ♻️ refactor — Рефакторинг

```
:recycle: refactor(ui): 提取公共玻璃态容器组件

创建可复用的 GlassContainer 组件，
减少多个屏幕间的代码重复。
```

### ⚡️ perf — Оптимизация производительности

```
:zap: perf(db): 优化已完成待办查询性能

在 completed_at 字段添加索引，
使用 WHERE 子句过滤已完成待办而非在内存中过滤。

优化前：45ms
优化后：12ms
```

### ✅ test — Тестирование

```
:white_check_mark: test(provider): 添加 TodoProvider 单元测试

覆盖场景：
- 添加待办
- 切换完成状态
- 删除待办
```

### 🔧 chore — Разное

```
:wrench: chore: 更新 Flutter 版本至 3.19.0

提高最小 Flutter 版本要求并更新兼容依赖。
```

### 🚀 ci — Конфигурация CI

```
:rocket: ci: 添加 GitHub Actions 测试工作流

创建包含以下步骤的工作流：
- 单元测试
- 集成测试
- 代码覆盖率
```

### 💄 style — Корректировка форматирования

```
:lipstick: style(todo_item): 使用 dart fix 格式化代码

应用自动格式化修复，
保持代码风格一致性。
```

### Требования к релизу

1. **Заголовок сообщения**: необходимо использовать формат `:bookmark:` + `V<номер_версии>`
2. **Номер версии**: соответствует правилам семантического версионирования
3. **Полнота содержимого**: должно включать **все коммиты** с момента предыдущего релиза
4. **Классификация по типам**: упорядочить по типам `feat`, `fix`, `refactor`, `chore` и т.д.
5. **История коммитов**: перечислить хэши и описания всех связанных коммитов

## Часто задаваемые вопросы

### В: Как выбрать тип коммита?

- **feat**: Изменения функциональности, видимые пользователю
- **fix**: Исправление проблем, о которых сообщили пользователи
- **docs**: README, комментарии и т.д.
- **chore**: Обновление зависимостей, конфигурационные файлы
- **refactor**: Оптимизация кода без изменения поведения

### В: Когда следует разделять коммиты?

- Каждый коммит должен делать **только одно дело**
- Связанные функции коммитятся вместе, несвязанные — отдельно
- Следуйте принципу Atomic Commits
