## Формат коммита

**ОЧЕНЬ ВАЖНО！！！！！！ НЕ ЗАБЫВАЙТЕ ОБ ЭТОМ！！！**
Все сообщения коммитов должны следовать следующему формату:

```
:emoji-код: type(scope): тема (на русском)

[Необязательное тело сообщения]

[Необязательный футер]
```

> ⚠️ **Важно**: Необходимо использовать **emoji-коды** (например, `:sparkles:`), а не вводить emoji-символы напрямую.
>
> **Рекомендуется использовать сообщения коммитов на русском языке** для единообразия коммуникации в команде.

### Составляющие части

| Часть | Описание | Обязательно |
|-------|----------|-------------|
| emoji-код | Идентификатор типа коммита в виде эмодзи | ✅ |
| type | Тип коммита | ✅ |
| scope | Область действия | ✅ |
| subject | Краткое описание (на русском, не более 50 символов) | ✅ |
| body | Подробное описание (необязательно) | ❌ |
| footer | Критические изменения или закрытие задач (необязательно) | ❌ |

Область действия основана на структуре каталога `src/` проекта. **Необходимо использовать только определённые ниже scope**:

### Топ-модули

| scope | Соответствующий каталог | Описание |
|-------|-------------------------|----------|
| `frontend` | `src/frontend/` | Фронтенд: лексический анализ, синтаксический анализ, проверка типов |
| `middle` | `src/middle/` | Средний слой: IR, оптимизация, мономорфизация |
| `backends` | `src/backends/` | Бэкенды: интерпретатор, runtime, REPL |
| `std` | `src/std/` | Стандартная библиотека |
| `formatter` | `src/formatter/` | Форматтер кода |
| `lsp` | `src/lsp/` | LSP - протокол сервера языка |
| `package` | `src/package/` | Менеджер пакетов |
| `util` | `src/util/` | Утилиты: диагностика, кэширование, i18n |

## Управление версиями

Версия проекта определяется в поле `version` файла `Cargo.toml` в корневом каталоге:

```toml
[package]
version = "0.7.2"
```

Используется семантическое версионирование `MAJOR.MINOR.PATCH`:

| Тип версии | Описание | Пример |
|------------|----------|--------|
| **major** | Крупные изменения, несовместимые изменения API | 0.7.2 → 1.0.0 |
| **minor** | Новая функциональность, обратная совместимость | 0.7.2 → 0.8.0 |
| **patch** | Исправление багов, обратная совместимость | 0.7.2 → 0.7.3 |

> ⚠️ При релизе **обновите номер версии в `Cargo.toml` в ветке dev**, после слияния PR в main CI автоматически создаст тег и Release. **Не создавайте теги вручную**, иначе CI пропустит процесс релиза.

---

## CI/CD процесс релиза

Релиз автоматически выполняется через GitHub Actions (`release.yml`) по следующему алгоритму:

```
1. Обновление поля version в Cargo.toml в ветке dev
2. cargo build для обновления Cargo.lock
3. Коммит в формате релиза (см. ниже 🔖 Коммит релиза)
   - Сообщение коммита должно содержать все изменения с последнего релиза
     (т.е. полное содержимое PR)
4. Создание PR из dev в main
5. Слияние PR в main
6. CI автоматически проверяет:
   - Чтение версии из Cargo.toml → "v{version}"
   - Проверка существования тега
   - Не существует → запуск полного процесса релиза
   - Существует → пропуск (дублирование не выполняется)
7. CI автоматически выполняет:
   - Параллельно: кроссплатформенная сборка (Linux/Windows/macOS) +
     проверка безопасности + тесты
   - При успехе всех этапов: создание тега, упаковка артефактов,
     публикация GitHub Release
```

### Ключевые правила

| Правило | Описание |
|---------|----------|
| **Не создавать теги вручную** | CI принимает решение о публикации на основе наличия тега; ручное создание тега приведёт к пропуску CI |
| **Bump версии в dev** | Коммит релиза создаётся в dev, затем через PR сливается в main |
| **Коммит релиза содержит полный changelog** | Сообщение коммита должно содержать все изменения данного релиза, так как оно служит описанием PR |
| **Не сливать main обратно в dev** | После слияния PR dev синхронизируется автоматически; обратное слияние не требуется |

## Правила сообщений

### Языковые правила

**Рекомендуется использовать сообщения коммитов на русском языке** для единообразия коммуникации в команде.

- Subject на русском языке, краткий и понятный
- Body на русском языке с подробным описанием
- Специальные технические термины можно оставить на английском

### Subject (тема)

- На русском языке, кратко и ясно
- Длина не более 50 символов
- Точка в конце не ставится

### Body (тело)

- Подробное описание причин и способа изменений
- Каждая строка не более 72 символов
- Использовать - или * для списков

### Footer (футер)

- **Критические изменения**: начинаются с `BREAKING CHANGE:`
- **Закрытие Issue**: использовать `Закрыть #123` или `Исправить #456`

## Примеры

### ✨ feat — Новая функциональность

```
:sparkles: feat(parser): Добавлена поддержка синтаксического анализа замыканий

Реализован синтаксический анализ выражений замыканий:
- Поддержка краткого синтаксиса |args| body
- Поддержка захвата move-семантики
- Добавлен вывод типов замыканий

Закрыть #42
```

### 🐛 fix — Исправление бага

```
:bug: fix(repl): Исправлена неработа автодополнения при многострочном вводе

SessionREPL неправильно регистрировал дополнятор в многострочном режиме,
из-за чего Tab-дополнение не срабатывало.

Исправить #128
```

### 📝 docs — Обновление документации

```
:memo: docs(design): Обновлена спецификация модели владения и системы типов

Синхронизация последних изменений дизайна из RFC-009 и RFC-011.
```

### ♻️ refactor — Рефакторинг

```
:recycle: refactor(typecheck): Разделение примитивных типов значений и семантики Dup мелкого копирования

Декорреляция типов значений и семантики копирования в MonoType,
устранение особых случаев в ветках match.
```

### ⚡️ perf — Оптимизация производительности

```
:zap: perf(types): Оптимизация вычисления const generic

Добавлен лимит глубины для рекурсивного вычисления (по умолчанию 128),
предотвращающий переполнение стека от вредоносных выражений типов.
```

### ✅ test — Тестирование

```
:white_check_mark: test(typecheck): Добавлены тесты изменчивости VarInfo для scope

Покрытые сценарии:
- Только чтение для неизменяемых привязок
- Отслеживание изменчивости для mut-привязок
- Распространение изменчивости через области видимости
```

### 🔧 chore — Разное

```
:wrench: chore(build): обновление rand, hashbrown, tempfile, ron, clap

Обновление 6 production-зависимостей до последних стабильных версий.
```

### 🚀 ci — Конфигурация CI

```
:rocket: ci: Исправление слишком низкой версии Rust для nightly-сборки

Обновление RUST_TOOLCHAIN с 1.91.0 до 1.96.0,
в соответствии с rust-version в Cargo.toml.
```

### 💄 style — Форматирование

```
:lipstick: style(frontend): Применено форматирование cargo fmt

Унификация стиля переноса строк в сигнатурах функций.
```

## Часто задаваемые вопросы

### В: Как выбрать тип коммита?

- **feat**: Изменения функциональности, видимые пользователю
- **fix**: Исправление проблем, о которых сообщили пользователи
- **docs**: README, комментарии и другая документация
- **chore**: Обновление зависимостей, конфигурационных файлов
- **refactor**: Оптимизация кода без изменения поведения

### В: Когда следует разделять коммиты?

- Каждый коммит должен делать **одно дело**
- Связанные изменения объединяются, несвязанные — разделяются
- Следуйте принципу атомарных коммитов (Atomic Commits)
