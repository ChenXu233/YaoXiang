## Формат коммита

**Очень важно!!!!!!!!! Нельзя забывать!!!!!**
Все сообщения коммитов должны соответствовать следующему формату:

```
:emoji-код: type(scope): тема (на русском)

[Необязательное тело сообщения]

[Необязательный футер]
```

> ⚠️ **Важно**: Необходимо использовать **emoji-коды** (например, `:sparkles:`) вместо прямого ввода emoji-символов.
>
> **Рекомендуется использовать сообщения коммитов на русском языке** для единообразия коммуникации в команде.

### Составляющие части

| Часть | Описание | Обязательно |
|------|----------|--------------|
| emoji-код | Emoji-символ, идентифицирующий тип коммита | ✅ |
| type | Тип коммита | ✅ |
| scope | Область влияния | ✅ |
| subject | Краткое описание (на русском, не более 50 символов) | ✅ |
| body | Подробное описание (необязательно) | ❌ |
| footer | Критические изменения или закрытие задач (необязательно) | ❌ |

## Правила сообщений

### Управление версиями

**Перед каждым коммитом необходимо увеличивать номер версии**:

| Тип версии | Место обновления | Описание |
|----------|----------|------|
| **major** | `pubspec.yaml` (version) + `release_v*.md` | Крупные обновления, несовместимые изменения API |
| **minor** | `pubspec.yaml` (version) | Новая функциональность, обратная совместимость |
| **patch** | `pubspec.yaml` (version) | Исправление багов, обратная совместимость |

### Формат номера версии

Используется семантическое версионирование `MAJOR.MINOR.PATCH`:

```
# Крупная версия (breaking changes)
1.0.0 -> 2.0.0

# Минорная версия (new features)
1.0.0 -> 1.1.0

# Патч-версия (bug fixes)
1.0.0 -> 1.0.1
```

### Процесс коммита

```bash
# 1. После изменения кода сначала увеличьте версию
# Используйте инструмент semantic_release для автоматического управления версиями и Changelog
npx semantic-release

# Или обновите версию вручную
# Отредактируйте поле version в pubspec.yaml

# 2. Сделайте коммит кода (изменения версии будут автоматически добавлены в следующий релиз)
git add .
git commit -m ":tada: Release v1.0.0"
git push
```

> 💡 Увеличение версии и генерация Changelog выполняются автоматически CI, при коммите достаточно убедиться, что изменения кода включают обновление версии.

## Правила сообщений

### Языковые соглашения

**Рекомендуется использовать сообщения коммитов на русском языке** для единообразия коммуникации в команде.

- Subject используется на русском языке, кратко и понятно
- Body может содержать подробное описание на русском языке
- При наличии специальных технических терминов можно сохранить английский язык

### Subject (тема)

- Используется на русском языке, кратко и понятно
- Длина не превышает 50 символов
- В конце не ставится точка

### Body (тело)

- Подробно описывает причину и способ изменения
- Каждая строка не превышает 72 символов
- Пункты перечисляются с помощью - или *

### Footer (футер)

- **Критические изменения**: начинаются с `BREAKING CHANGE:`
- **Закрытие Issue**: используется `Закрыть #123` или `Исправить #456`

## Примеры

### ✨ feat — Новая функциональность

```
:sparkles: feat(db): Добавить функцию массового удаления задач

Реализация функции массового удаления задач:
- Добавлен метод batchDelete в TodoRepository
- Добавлен диалог подтверждения удаления
- Обновлён UI для поддержки множественного выбора

Закрыть #42
```

### 🐛 fix — Исправление бага

```
:bug: fix(provider): Исправить проблему с восстановлением таймера Pomodoro из фона

При восстановлении приложения из фона таймер Pomodoro не продолжал отсчёт.
Добавлена логика восстановления состояния в initState.

Исправить #128
```

### 📝 docs — Обновление документации

```
:memo: docs: Обновить описание новых функций в README

Добавлены следующие разделы:
- Режим концентрации
- Статистика данных
- Фоновая музыка
```

### ♻️ refactor — Рефакторинг

```
:recycle: refactor(ui): Извлечь общий компонент стеклянного контейнера

Создан многократно используемый компонент GlassContainer,
уменьшено дублирование кода между несколькими экранами.
```

### ⚡️ perf — Оптимизация производительности

```
:zap: perf(db): Оптимизировать производительность запросов выполненных задач

Добавлен индекс по полю completed_at,
используется фильтрация выполненных задач через WHERE вместо фильтрации в памяти.

До оптимизации: 45ms
После оптимизации: 12ms
```

### ✅ test — Тестирование

```
:white_check_mark: test(provider): Добавить модульные тесты для TodoProvider

Покрытые сценарии:
- Добавление задачи
- Переключение статуса выполнения
- Удаление задачи
```

### 🔧 chore — Разное

```
:wrench: chore: Обновить Flutter до версии 3.19.0

Повышены минимальные требования к версии Flutter и обновлены совместимые зависимости.
```

### 🚀 ci — Конфигурация CI

```
:rocket: ci: Добавить рабочий процесс тестирования GitHub Actions

Создан рабочий процесс со следующими шагами:
- Модульные тесты
- Интеграционные тесты
- Покрытие кода тестами
```

### 💄 style — Форматирование

```
:lipstick: style(todo_item): Применить форматирование с помощью dart fix

Применены автоматические исправления форматирования,
обеспечена согласованность стиля кода.
```

### Требования к релизу

1. **Заголовок сообщения**: Необходимо использовать формат `:bookmark:` + `V<номер_версии>`
2. **Номер версии**: Соблюдать правила семантического версионирования
3. **Полнота содержимого**: Необходимо включить описание **всех коммитов** с момента предыдущего релиза
4. **Классификация по типам**: Классифицировать по типам `feat`, `fix`, `refactor`, `chore` и т.д.
5. **История коммитов**: Перечислить хэши и описания всех связанных коммитов

## Часто задаваемые вопросы

### В: Как выбрать тип коммита?

- **feat**: Функциональные изменения, видимые пользователю
- **fix**: Исправление проблем, о которых сообщили пользователи
- **docs**: README, комментарии и другая документация
- **chore**: Обновление зависимостей, конфигурационные файлы
- **refactor**: Оптимизация кода без изменения поведения

### В: Когда следует разделять коммиты?

- Каждый коммит делает **только одно дело**
- Связанные функции коммитятся вместе, несвязанные — отдельно
- Следуйте принципу атомарных коммитов (Atomic Commits)
