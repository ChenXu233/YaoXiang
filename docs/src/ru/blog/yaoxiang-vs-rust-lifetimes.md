# Мы спрятали `'a` в компилятор

*— Честная заметка о модели владения YaoXiang*

---

Сколько времени у вас ушло, чтобы по-настоящему понять этот код на Rust, когда вы увидели его впервые?

```rust
struct Parser<'a> {
    input: &'a str,
    pos: usize,
}

impl<'a> Parser<'a> {
    fn advance(&mut self) -> &'a str {
        let start = self.pos;
        self.pos += 1;
        &self.input[start..self.pos]
    }
}
```

Три `'a`. Один на структуре, два в блоке impl. Они говорят об одном и том же: `Parser` не может жить дольше, чем заимствованный им `input`. Это правильно. Безопасность Rust зиждется на этом механизме.

Но когда мы пишем такой код, часто возникает мысль: **неужели компилятор сам не может разобраться?** Ответ Rust — «нет», по крайней мере, не меняя модель заимствования.

**Эта статья не о том, что «YaoXiang решил эту проблему». Она о том, что «YaoXiang пробует другой путь — спрятать `'a` в компилятор, чтобы компилятор писал его за вас. Как далеко мы продвинулись, и какие кости остаются самыми твёрдыми».**

---

## Почему Rust нуждается в `'a`

`&T` и `&mut T` в Rust — это указатели, указывающие на данные. Займствовать значение — значит создать ссылку на него. У ссылки есть собственный lifetime (время жизни). Когда ссылка распространяется через границы функций (как возвращаемое значение, при сохранении в структуру), компилятор не может в пределах одной функции вывести, как долго проживёт ссылка — программисту нужно указать через `'a` информацию о том, что «это возвращаемое значение и этот параметр разделяют время жизни».

Сообщество Rust не стоит на месте. Правила элизии (lifetime elision) избавляют от аннотаций в большинстве простых функций. NLL вошёл в версию 2018 года, освободив заимствования от пут лексической области видимости. Но когда ссылку нужно сохранить в структуру, вернуть из функции, захватить в замыкание — **сама модель диктует, что в этих сценариях программист обязан аннотировать отношения между ссылками.**

---

## Другой угол зрения: заимствование — это не указатель, а токен

Основной проект YaoXiang описан в [RFC-009 (Модель владения)](/design/rfc/accepted/009-ownership-model). Он не меняет семантику по умолчанию (это всегда Move), но меняет **онтологию заимствования**.

В YaoXiang `&T` и `&mut T` — **не указатели**. Это **нуль-размерные токены времени компиляции** — доказательства доступа на уровне типов. Займствовать значение — значит не создать указатель на него, а создать доказательство «мне разрешено к нему обращаться»:

```
&T     →  Гарантирует неизменяемость данных. Реализует Dup (копируемо), несколько read-токенов сосуществуют безопасно
&mut T →  Гарантирует исключительную изменяемость. Не реализует Dup (линеен), от одного источника может быть только один
```

В Rust вы пишете `&` в точке вызова (`distance(&p1, &p2)`). В YaoXiang компилятор видит, что сигнатура функции требует `&Point`, и автоматически создаёт токен в точке вызова — точка вызова становится `distance(p1, p2)`. Цена — в том, что в сигнатуре определения должен быть `&`, иначе Move по умолчанию съест владение:

```yaoxiang
# В сигнатуре нужен & — компилятор видит &Point, токен создаётся автоматически
check_dimensions: (v: &Vec3) -> Bool = { ... }
check_bounds: (v: &Vec3) -> Bool = { ... }

v = Vec3(1.0, 2.0, 3.0)
if check_dimensions(v) && check_bounds(v) {  # Каждый раз автоматически создаётся токен &Vec3
    # v по-прежнему доступен
}
```

В Rust `&` стоит в точке вызова; в YaoXiang `&` стоит в точке определения. Аннотация не исчезла — она переместилась.

## Механизм брендов: спрятанный в компиляторе `'a`

Пользователь никогда с этим не соприкасается — но без понимания этого не понять, что именно сделал YaoXiang.

Компилятор внутри себя присваивает каждому токену заимствования уникальный номер времени компиляции:

```
Видит пользователь       Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)
&mut Point     →  WriteToken(Point, #M)
```

Когда вы обращаетесь через `&Point` к полю и получаете `&Float`, последний несёт производный бренд: `#N.field_x`. Когда вы передаёте `&mut` токен через Move другой переменной, компилятор знает, что исходная переменная больше его не держит — это базовая возможность семантики Move.

**`#N` — это и есть `'a`.** Отношение префикса — `#N` является префиксом для `#N.field_x` — это и есть ограничение outlives. Та же самая информация. Программист Rust пишет `'a`, компилятор YaoXiang пишет `#N`.

Различие — только в проценте успешного вывода. Правила элизии и NLL в Rust покрывают около 80% сценариев. Ставка YaoXiang: **если дизайн языка даёт компилятору более чистый вход, сможет ли он покрыть больше?**

Эту ставку подпирают несколько языковых ограничений:

- **Запрет затенения переменных** — у `x` в этой области видимости одна-единственная идентичность, компилятору не нужно различать «какой x ты имеешь в виду».
- **Явный `return`** — что выходит за границы блока, записано явно, компилятору не нужно угадывать, «является ли последняя строка возвращаемым значением».
- **`for` создаёт новую привязку на каждой итерации** — переменные между итерациями не интерферируют, компилятору не нужно отслеживать, «что изменилось в прошлой итерации».

Эти правила — не «нормативные предписания». В отличие от бесполезного ритуала getter/setter в Java — каждое из них **превращает информацию, которую компилятору нужно выводить, в информацию, уже записанную в программе**. Компилятору не нужно гадать «какую переменную ты имеешь в виду», «убежало ли это значение», «как меняется переменная цикла между итерациями» — ответы лежат в самом коде.

## Что механизм брендов умеет — то же, что и Rust

Поскольку токены — обычные типы, они подчиняются всем обычным правилам типов. Нет специальных запретов вроде «ссылку нельзя возвращать», «ссылку нельзя хранить в структуре», «замыкание не может захватить ссылку». **Но и Rust всё это умеет.** Различие — не в возможностях, а в том, кто пишет аннотации.

**Возврат ссылки — программист Rust пишет `'a`, компилятор YaoXiang пишет `#N`:**

```yaoxiang
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)
}

p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()   # Токен распространяется к вызывающему, компилятор отслеживает цепочку порождения брендов
```

Rust делает то же самое, только программисту нужно написать `'a`, чтобы связать вход и выход. В YaoXiang компилятор автоматически выводит порождение через путь бренда — `px_ref` (`#N.field_x`) порождается из `p` (`#N`). То же ограничение, иной способ записи.

**Структура хранит ссылку — без параметра времени жизни:**

```yaoxiang
Window: Type = {
    target: Point,
    view: &Point,   # Поле-токен, ничем не отличается от других полей
}
```

В Rust, когда у структуры есть поле-ссылка, `'a` должна появиться в определении структуры и во всех блоках impl — программист явно аннотирует ограничение времени жизни `Window<'a>`. В YaoXiang `view: &Point` пишется без `'a`, но номер бренда по-прежнему играет ту же роль внутри компилятора — когда экземпляр `Window` уничтожается, содержащийся в нём токен умирает вместе с ним. Та же гарантия, другая видимость.

**Захват замыканием — нуль-стоимостное копирование Dup-токена:**

```yaoxiang
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)   # Токен threshold копируется в замыкание с нулевыми накладными расходами
}
```

`&Float` реализует Dup (копируемо), замыкание захватывает его так же, как захватило бы нуль-размерное целое. Тот же набор правил, что и для автоматического заимствования в вызовах функций. Ноль аннотаций от пользователя.

## Цена: аннотация исчезла из сигнатуры

У `'a` в Rust есть часто упоминаемая ценность: это ещё и документация. `fn split_at_mut<'a>(slice: &'a mut [T], mid: usize) -> (&'a mut [T], &'a mut [T])` — `'a` говорит читателю, что оба возвращаемых среза ссылаются на одни и те же исходные данные.

В реальности сила этого аргумента ограничена — большинство новичков в Rust не читают `'a` как документацию, а списывают как заклинание, которого требует компилятор. Но, если быть честным: в сложных сценариях заимствования `'a` в Rust хотя бы даёт отправную точку для отслеживания потоков данных. В YaoXiang вам нужно понимать цепочки порождения брендов — бренды невидимы для пользователя, это зависит от инструментария, а инструментарий ещё не созрел.

## Обнаружение конфликтов токенов: тот же конвейер доказательств

В Rust есть отдельный «проверщик заимствований». **Проектное направление** YaoXiang — объединить конфликты заимствований в конвейер доказательств, общий с проверкой типов ([RFC-027 (Предикаты времени компиляции и унифицированная статическая верификация)](/design/rfc/accepted/027-compile-time-evaluation-types)).

Конфликт токенов — это предложение Хоара:

```
{ конфликтующий ReadToken мёртв } data.push(4) { WriteToken безопасно получен }
```

```yaoxiang
# &mut T — линейный тип — после Move исходная переменная больше не держит
bad: (p: &mut Point) -> Void = {
    p2: &mut Point = p    # WriteToken перемещается из p в p2
    p.x = 10.0            # { p владеет WriteToken } p.x = 10.0 { безопасно }
}                          # → WriteToken p был перемещён → Disproved

# &T — Dup — копируемый
good: (p: &Point) -> Void = {
    p2: &Point = p        # Копируем read-токен
    print(p.x)            # OK, два read-токена сосуществуют
}
```

Тот же путь вывода ошибок, что и для ошибок типов, провалов верификации предикатов. Не нужно изучать две диагностические системы. **Но цена:** сложный конфликт токенов в Rust порождает тщательно сформулированную ошибку проверки заимствований; в YaoXiang это может выглядеть как «WriteToken(#7.field_x) конфликтует с WriteToken(#7)» — технически точно, но номера брендов бессмысленны для человеческого читателя. Объяснимость сообщений об ошибках — область, которая ещё не проверена.

## Ключевое слово `ref`: автоматический выбор Rc/Arc

Токены не пересекают границы задач (потоков) — это доказательства времени компиляции, а не значения времени выполнения. Для разделения между областями видимости используется `ref`:

```yaoxiang
shared_data = ref Point(1.0, 2.0)   # Компилятор через escape-анализ автоматически выбирает Rc или Arc

spawn {
    print(shared_data.x)   # Пересекает границу задачи → компилятор выбирает Arc
}
```

- Не убегает в spawn-блок → `Rc` (неатомарный подсчёт ссылок)
- Убегает в spawn-блок → `Arc` (атомарный подсчёт ссылок)

Цена: читая код, невозможно по его фрагменту узнать, `ref` — это Rc или Arc. Один рефакторинг (обернуть код в spawn) может молча изменить реализацию подсчёта ссылок — компилятор вас не предупредит. Изменение производительности неявно.

## Главная кость на данный момент: RAII слишком груб

Мы говорили, что токены — это значения, а их время жизни управляется через RAII. Но правила RAII для обычных значений: **значение живёт до конца области видимости.** Это как раз та проблема Rust до NLL — заимствование длится до конца блока, даже если вы давно им не пользуетесь.

```yaoxiang
process: (data: &mut Data) -> Void = {
    header_view: &Header = data.header()    # Порождаем &Header из &mut Data
    header_info = parse_header(header_view) # ← последнее использование header_view
    # header_view не обязан жить до конца функции —
    # но RAII держит его до }

    data.modify(header_info)   # ❌ ReadToken ещё «жив», WriteToken заблокирован
}
```

NLL в Rust анализирует последнее использование, а не лексическую область видимости. YaoXiang нуждается в той же способности. Решение, которое сейчас прорабатывается, — подключить анализ активности токенов к тому же конвейеру доказательств — три слоя:

1. **Быстрый путь** — повторное использование существующего BorrowChecker (линейное сканирование, на уровне IR-инструкций). Сценарии, когда токен использован и освобождён внутри одного базового блока, проходят сразу.
2. **Структурный анализ** — сопоставление префиксов дерева брендов (кто с кем конфликтует) + DAG-запрос потребителей (находится ли последний потребитель токена после текущего узла).
3. **SMT-решатель** — активируется, только когда нужна логическая дедукция, например для условий циклов.

Инфраструктура конвейера доказательств (трёхзначный возврат `Proved/Disproved/Unproven`, бэкенд Z3 SMT, стек гипотез) уже частично реализована в `src/frontend/core/typecheck/proof/`. Слой владения (`layers/ownership.rs`) — пока скелет, который напрямую возвращает Proved без реальных проверок. Заполняем.

Текущее обходное решение — вручную вкладывать блоки, чтобы сократить область жизни токена:

```yaoxiang
process: (data: &mut Data) -> Void = {
    header_info = {
        header_view: &Header = data.header()
        parse_header(header_view)
    }   # header_view освобождается с концом блока
    data.modify(header_info)   # ✅
}
```

Это реальное трение. С ним столкнётся каждый, кто приходит из Rust. Можно ли его устранить — зависит от того, как подключение слоя владения к конвейеру сработает.

## Самый трудный вопрос: запасной вариант

`'a` в Rust — не только бремя, но и запасной вариант. Компилятор не может вывести — программист аннотирует отношение времён жизни, компилятор верифицирует. **У вас есть ручка.**

Теоретический запасной вариант YaoXiang — **доказывающие функции** времени компиляции (RFC-027 §4.2): автоматический вывод компилятора провалился → программист пишет функцию, чей возвращаемый тип и есть предложение «токены не конфликтуют» → компилятор верифицирует тип этой функции. Но —

Как должна выглядеть доказывающая функция «токены не конфликтуют»? Как пользователь сконструирует значение типа `WriteTokenAvailable`? Должен ли он понимать отношения префиксов между номерами брендов `#N` и `#N.field_x`?

**Если доказывающая функция требует от пользователя понимания номеров брендов — мы просто переписали `'a` под видом `#1`. Ничего не сэкономили.**

Ответа пока нет. Это самое вероятное место, где весь эксперимент может застрять.

## Девять итераций RFC-009

Это не дизайн, придуманный в башне из слоновой кости. RFC-009 прошёл через девять крупных версий:

| Версия | Ключевое изменение | Причина отвержения |
|------|----------|------------|
| v1–v7 | Основано на модели владения Rust, постепенное добавление анализа потребления, обратных функций, полевой изменяемости | Избыточная сложность, вышедшая из-под контроля |
| **v8** | «Урезанное заимствование» — `&T`/`&mut T` только как параметры, нельзя возвращать, хранить в структуре, захватывать замыканием | Три жёстко прописанных запрета. Крайне ограниченная выразительность |
| **v9** | Система токенов заимствования — `&T`/`&mut T` это обычные типы, подчиняются обычным правилам | Устраняет специальные правила, но опускает отслеживание брендов внутрь компилятора |

Переход от v8 к v9 — настоящий прорыв: от трёх запретов к нулю специальных правил. Но v9 устранил видимые пользователю правила, а не внутреннюю сложность системы — механизм брендов, отслеживание порождения, обнаружение конфликтов между источниками — всё это по-прежнему существует, просто переехало в компилятор. Объединение проверки заимствований с конвейером доказательств — одно из направлений, но сможет ли оно работать на реальном коде и как устроить запасной вариант — всё ещё верифицируется.

## В заключение

Мы не уничтожили `'a`. `#1` — это и есть `'a` — та же информация, в другом месте.

Ставка эксперимента: языковые проектные ограничения (запрет затенения, явный return, новые привязки в `for`, DAG-семантика `{}`) дают компилятору более чистый вход, и, возможно, вывод брендов сможет автоматически срабатывать в большинстве сценариев, где правила элизии Rust отказывают. Если сможет — пользователю больше не придётся писать `'a`, различать аннотацию и элизию, изучать проверщик заимствований. Если не сможет — или если запасной механизм (доказывающие функции) потребует от пользователя понимания номеров брендов — это будет просто изобретение заново.

Работаем. Напишем, когда появятся результаты.
