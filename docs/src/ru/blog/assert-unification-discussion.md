# Ночной разговор о «единстве»

С позднего вечера 11 июля до рассвета 12 июля 2026 года мы провели шесть часов, обсуждая один вопрос — чем же должен быть `assert` в YaoXiang. Результат оказался неожиданным: нам казалось, что мы обсуждаем «как реализовать assert», а на самом деле обсуждали «какие дыры есть в фундаменте системы типов».

Эта статья — не про итоговый вывод (он в спецификации), а про то, как мы постепенно обнаруживали свои ошибки и как нас каждый раз возвращали назад.

---

## Почему существуют два assert

YaoXiang одновременно разрабатывает две вещи:

**Уточняющий тип времени компиляции** `Assert(N > 0)`, записывается в определении типа:
```
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    _assert: Assert(N > 0),   // N должно быть больше 0, иначе компиляция не пройдёт
}
```
Верификация обобщённых параметров во время компиляции. N равно 5 — проходит, N равно 0 — ошибка компиляции. Нулевые накладные расходы во время выполнения.

**Утверждение времени выполнения** `assert(x > 0)`, записывается в теле функции:
```
x = read_int()
assert(x > 0)   // проверка пользовательского ввода
```
Значение приходит извне, на этапе компиляции неизвестно. При неудаче — panic.

Assert с большой буквы, assert с маленькой. Один — времени компиляции, другой — времени выполнения. Одно понятие, два облика.

Самая естественная реакция — самый ленивый путь — рассматривать их как две совершенно независимые сущности: Assert — уточняющий тип, идущий по конвейеру доказательств, assert — нативная функция строк на двадцать, возвращающая пустое значение. Каждый сам по себе, не мешают другому.

Но в этом решении всегда было что-то неуловимо неприятное. Очевидно, что они связаны — это две стороны одной семантики, «я утверждаю, что это условие должно выполняться», — и заставлять пользователя запоминать «здесь пиши с большой буквы, там с маленькой, здесь проверится во время компиляции, там только в runtime» — это когнитивный долг. В дизайне языка каждый новый элемент «выглядит похоже, но надо запоминать отдельно» — это долг.

## Мы хотели объединить — и врезались в стену

Идея объединения на самом деле довольно проста:

```
assert: (cond: Bool) -> Assert(cond)
```

`assert` возвращает уточняющий тип `Assert(cond)`. Если компилятор может вычислить значение cond, оно сворачивается в проверку времени компиляции (с удалением); если компилятор не может вычислить, проверка остаётся на runtime. Одна функция, две судьбы. Сама сигнатура — точка соединения.

Но тут же кто-то указал на «фундаментальное противоречие»:

> `Assert(C)` требует, чтобы C было вычислимо во время компиляции. В `assert(x > 0)` x — это значение времени выполнения — компилятор не может вычислить `Assert(x > 0)`. По правилам уточняющих типов это должно быть **ошибкой компиляции**. Но `assert(x > 0)` — совершенно нормальное утверждение времени выполнения. Значит, сигнатура `Assert(cond)` для объединения не подходит. Нужно разделять.

Этот аргумент звучит неопровержимо. Если уточняющий тип при встрече со значением времени выполнения выдаёт ошибку, ты никогда не сможешь впихнуть «утверждение времени компиляции» и «утверждение времени выполнения» в одну сигнатуру. Две вещи, две судьбы — разделяй.

**Но это ложь.** Ложь — в последней фразе «должно быть ошибкой компиляции». Результат уточняющего типа — не двузначный — не только «доказано» и «ошибка». Он трёхзначный: Proved (доказано), Disproved (опровергнуто, с контрпримером), **Unproven (недоказуемо)**.

«Недоказуемо» и «опровергнуто» — совершенно разные состояния. Опровергнуто — это ошибка, навсегда. А недоказуемо — это когда значение неизвестно до runtime, поэтому и нельзя доказать, — как раз и оставляется на проверку в runtime. То, что компилятор не может доказать, — это не провал компилятора, а **естественное свойство этого утверждения**.

«Утверждение времени выполнения» — это не «другой механизм», а утверждение, которое компилятор не может доказать, и его проверка откладывается до runtime. **Один уточняющий тип, две судьбы — судьба автоматически определяется тем, может ли компилятор его разрешить.**

Ложный конфликт разбит — путь к объединению открыт. Но сначала нужно залатать ещё одну дыру в фундаменте.

## Не хватало одного типа: Never (ложь / расходимость / невозможность)

Схема объединения требует, чтобы `assert(false)` сводилось к типу «значение невозможно» — потому что утверждение «ложь истинна» не имеет доказательства, и там, где оно используется в качестве типа, программа должна расходиться.

И тут мы обнаружили: в YaoXiang этого типа просто нет.

Есть `Void`. Но `Void` может возвращать значение по умолчанию — вызвал функцию, возвращающую `Void`, и выполнение кода продолжается. У `Void` есть обитатель. В соответствии Карри–Говарда тип с одним значением — это «истинное ⊤» (одно доказательство), а не «ложное ⊥» (ноль доказательств). **`Void` всегда был «истинным ⊤», просто никто так его не называл.**

Нам не хватало «ложного ⊥»: типа, который не может быть обитаем никогда — ни одно значение не может стоять в правой части `x: Never = ...`. Типом возврата `assert(false)`, `panic`, бесконечного цикла должен быть именно он. Без него `assert(false)` вернёт законное значение void по умолчанию, и код продолжит выполняться — провал утверждения превратится в нормальный поток. Вся система уточняющих типов в этот момент логически рушится.

`Never` в итоге был определён как **аксиома**: он не выводится ни из чего, он — первоначальное понятие системы типов. Три свойства встроены в ядро — ноль конструкторов, Never является подтипом всех типов («из лжи следует всё что угодно»), а в качестве типа возврата функции он отмечает, что функция расходится и никогда не возвращается. Если эти три свойства непонятны — не страшно, достаточно знать вывод: язык должен иметь встроенный тип с именем `Never`, у которого нет ни одного значения; он сообщает системе, что код после `assert(false)` в принципе не может быть достигнут.

## Мы наплодили кучу ложных классификаций — и сами в них запутались

После того как `Never` был добавлен, нужно было ответить на вопрос «что делать, когда недоказуемо». На этом этапе мы потеряли целых два раунда обсуждений, потому что совершили ошибку из учебника.

**Шаг первый: разбить Unproven на части.**

«Недоказуемо» выглядит как несколько разных случаев. Зависит от переменной времени выполнения — `x` приходит извне, компилятор, конечно, не знает. Превышает бюджет поиска — теоретически можно доказать, но слишком дорого вычислять, и мы отказались. Не хватает контекстных предпосылок — дело не в самом утверждении, а в том, что вызывающий не предоставил достаточно статической информации. Гёделевски неразрешимо — дай сколько угодно времени и бюджета, всё равно не разрешишь, теоретически невозможно.

И вот мы разбили это на четыре категории, напридумывали им кучу названий и начали спорить, что делать с каждой. Зависит от runtime — понизить до проверки в runtime? Превышает бюджет — увеличить бюджет или попросить у пользователя доказательство? Не хватает предпосылок — добавить предпосылки или выдать ошибку? Неразрешимо — пользователь должен написать доказательство?

**После разбивки выяснилось, что классификация ошибочна.**

Ось для классификации выбрана неверно. Мы классифицировали по «что компилятор должен делать после того, как не смог доказать». Но «что делать» — это результат рассуждения, а не основание для классификации. Правильный вопрос таков: **может ли компилятор получить истинное значение предиката этого утверждения?**

Может получить — параметр в обобщённой позиции, значение определено во время компиляции, например, N в `StaticArray: (T: Type, N: Int) -> Type` — отправляется в конвейер доказательств. Конвейер выдаёт три ответа: Proved (истина, удаляется), Disproved (ложь, ошибка компиляции, навсегда), Unknown (истинно·недоказуемо, но в этой позиции компилятор способен доказать, поэтому попросит доказательство).

Не может получить — параметр является аргументом функции, значение передаётся вызывающим в runtime, например, n в `process: (n: Int) -> ...` — вообще не должен попадать в конвейер доказательств. Сразу идёт на проверку в runtime. На этом пути Unproven не существует, потому что нет самой задачи «доказывать» — невозможно написать доказательство для утверждения вроде «пользователь мог ввести отрицательное число», потому что оно не является тавтологией.

**Всего два пути. Из четырёх категорий стало две, лишние классификации — это мы сами себя запутывали.**

Когда этот вывод только прозвучал, мы ему не очень поверили. Чтобы проверить, я привёл пример — `process: (N: Int) -> ... { assert(N > 0) }`, — и сказал, что здесь N известен компилятору, но prover не справляется; это «недоказуемо + известно компилятору» — новая категория, для которой нужно принудительно писать доказательство.

**Это тут же было опровергнуто.** `process` — обычная функция, `N` — обычный параметр функции: его значение приходит от вызывающего, компилятор его в принципе не знает. Я воспринял `N: Int` в сигнатуре функции как «значение N известно компилятору» — но в YaoXiang «известно компилятору» точно соответствует **позиции обобщённого параметра**, а не позиции обычного параметра функции. Одно и то же написание `N: Int` в конструкторе типа (`-> Type`) — это константа времени компиляции, а в обычной функции — значение времени выполнения. Два N — две вселенные.

После исправления истинное лицо «недоказуемо + известно компилятору» обнажилось: это не «обычный человек написал утверждение, которое prover случайно не смог доказать», а **кто-то неправильно использовал маленький `assert` в обобщённом контексте** — в той позиции компилятор знает значение параметра, и runtime-проверка тут неуместна. Это не новая категория, это и есть исходное правило уточняющих типов: находясь в области, где требуется доказательство, при недоказуемости — предоставь доказательство.

**Аварийный люк runtime не нужно проектировать — он существует естественным образом.** Это и есть путь «значение неизвестно компилятору, проверка в runtime». *Sound* (проверка реально происходит в runtime), удобный (не нужно писать prove), с явными накладными расходами (значение и так приходит извне). Это не «нам не хватает prover, поэтому нужен runtime», а **для утверждений этого класса во время компиляции нет и не может быть истинного значения** — prover сколько угодно мощный, пользователь действительно мог ввести значение, не удовлетворяющее условию, само утверждение ложно, доказательство истинного утверждения не напишешь. Runtime — единственный *sound* выбор, теоретически необходимый, а не компромисс.

## Каждый раз, когда мы сбивались, корень был один и тот же

Оглядываясь назад, мы опровергли девять ошибочных выводов:

1. `assert` возвращает пустое значение `()` — тогда результат runtime-проверки не передаётся обратно на этап компиляции, и последующий код не знает, что условие выполнено
2. `assert` может «снять» Unproven — нет, он не участвует в доказательстве, он выполняет реальную проверку в runtime
3. Утверждения runtime и утверждения с нехваткой контекста — разные подклассы Unproven — нет, они разделяются прямо на входе в конвейер, и Unproven у них не возникает
4. Нужно переделать RFC-027 в «специализированное стирание» — не нужно, стирается токен доказательства нулевого размера, а в runtime остаётся та самая проверка Bool
5. Неизменяемая переменная сразу идёт в compile-time — нет, неизменяемость не означает «значение известно компилятору» (`x = read_int()` неизменяемо, но значение компилятору неизвестно)
6. Параметр функции `N: Int` известен компилятору — только обобщённый параметр известен, обычный параметр получает значение только в runtime
7. Если есть prove, runtime не нужен — для runtime-ввода нет тавтологий, которые можно доказать, runtime теоретически необходим
8. «Недоказуемо» должно по умолчанию понижаться до runtime-проверки — недоказуемо + известно компилятору требует prove, понижается только «значение неизвестно компилятору»
9. Сопоставление на уровне типов — это индукция — match — это просто разбор случаев, индуктивной гипотезы в нём нет; настоящая индукция требует рекурсии + проверки завершимости

**Из девяти ошибок у пяти один и тот же корень: путаница между двумя концепциями, которые похожи, но на деле ортогональны.** Неизменяемость против известности компилятору. Токен доказательства против вычисления, порождающего токен. Обобщённый параметр против параметра функции. Разбор случаев против индукции. Недоказуемость против опровергнутости. Каждую пару мы задевали — не по небрежности, а потому что сами концепции похожи и легко проскальзывают друг в друга.

## Какие проблемы стоит выносить в issue для коллективного обсуждения

Во время обсуждения мы создали issue#156 с заголовком «Трудность объединения». Один ошибочный пример и одна несуществующая категория подпитывали вопрос «нужна помощь сообщества». Центральное противоречие было ложным конфликтом, категория «недоказуемо + известно компилятору» вообще не существует — она была замаскирована ошибочным примером. Ответ на самом деле сам всплыл во второй половине обсуждения — **схема «объединения» — это не компромисс, а схлопывание. Везде, где казалось, что «может быть, есть и третий случай», в итоге всё сводилось к уже существующим путям.**

Это не значит, что нельзя обращаться за помощью в issue. Это значит, что сначала нужно проверить: ты действительно видел реальную форму этой проблемы? Твой пример действительно принадлежит той категории, которую ты для него выделил? Твоё противоречие — настоящее противоречие, или «тебе казалось, что некоторая предпосылка выполняется, а на самом деле — нет»?

Часто «нужно мнение сообщества» — это не потому что ответ слишком далёк; это **ты сам запутался в созданных тобой же классификациях**.

## Что мы действительно вынесли

Это обсуждение длилось шесть часов, и мы девять раз опровергли сами себя. Конечный продукт — не новый API для assert, а прибитый гвоздями фундамент: `Never` должен существовать (раньше никто не осознавал его отсутствия), результат уточняющего типа трёхзначный, а не двузначный («недоказуемо» ≠ «опровергнуто»), известность компилятору ≠ неизменяемость, runtime-проверка — не слабая замена prove, а единственный *sound* выбор, `assert(x > 0)` — это не «что-то другое», а другая судьба `Assert` во вселенной значений. assert и Assert — две стороны одной сущности — разница между ними не в том, кто они, а в том, может ли компилятор в этот момент вычислить значение.

**Лучший дизайн — это схлопывание, а не добавление.** Мы не добавляли «механизм runtime assert», а обнаружили, что это всегда было одной из сторон `Assert`. Мы не добавляли «обработку недоказуемости», а обнаружили, что у неё уже было место в конвейере. Каждый раз, когда тебе кажется, что нужно «добавить особый случай», сначала спроси себя: это действительно что-то новое, или это какая-то уже существующая концепция, которую ты не узнал?

**Классифицируй по сути, а не по тому, что ты хочешь делать.** Мы сначала разбили недоказуемое на четыре подкласса по «что компилятор должен делать» — после разбивки выяснилось, что классификация неверна, потому что «что делать» — это результат рассуждения, а не основание для классификации. Правильный путь — классифицировать по «есть ли у утверждения истинное значение во время компиляции» — есть или нет. Два класса, чисто, без пересечений.
