# Дизайн-документ YaoXiang

> Дао рождает единое, единое рождает двойственное, двойственное рождает тройственное, тройственное рождает все сущее.

Данный каталог содержит проектные решения, предложения и обсуждения по языку программирования YaoXiang.

## Основные проектные принципы

| Принцип | Описание |
|------|------|
| **Всё есть тип** | Значения, функции, модули — всё это типы; типы являются гражданами первого класса |
| **Естественный синтаксис** | Читаемость как у Python, близость к естественному языку |
| **Модель владения** | Абстракции с нулевой стоимостью, без GC, высокая производительность |
| **Модель spawn (порождения)** | Синхронный синтаксис, асинхронная сущность, автоматический параллелизм |
| **AI-дружественность** | Строгая структурированность, чёткое AST |

## Участие в обсуждении дизайна

### Жизненный цикл RFC

Предложения RFC имеют 5 состояний:

| Состояние | Значение |
|------|------|
| Черновик | Работа в процессе |
| На рассмотрении | Открытое обсуждение |
| Принят | Дизайн одобрен |
| Устарел | Был принят, заменён новым дизайном |
| Отклонён | Не одобрен |

Полный жизненный цикл:
```
Черновик → На рассмотрении → Принят → Устарел (заменён)
                              ↓
                         Отклонён (не одобрен)
```

### Процесс подачи предложений

```
1. Составить предложение (используя шаблон RFC)
   → Поместить в rfc/draft/

2. Подать на рассмотрение
   → Переместить в rfc/review/, открыть общественное обсуждение

3. Оценка основной командой
   → Принять → Переместить в rfc/accepted/
   → Отклонить → Переместить в rfc/rejected/

4. Последующее сопровождение
   → Заменён → Переместить в rfc/deprecated/
```

### Принципы дизайна

- **Чёткие границы**: каждое проектное решение должно иметь ясную область применения
- **Приоритет практичности**: решать реальные проблемы, а не воображаемые угрозы
- **Поведение, наблюдаемое пользователем, не изменяется**: Never break userspace

## Ключевые проектные решения

### 1. Система типов

- **Унифицированный синтаксис типов**: отказ от `enum`, `struct`, `union` в пользу единого `Name: Type = {...}`
- **Конструктор как тип**: устранение пропасти между «типом» и «значением»
- **Поддержка обобщений (дженериков)**: мономорфизация на этапе компиляции, нулевые накладные расходы во время выполнения

### 2. Модель spawn (порождения)

```yaoxiang
// Модель spawn: по умолчанию последовательное выполнение, spawn вводит параллелизм по потоку данных

// По умолчанию последовательное выполнение
compute: (Int) -> Int = (n) => {
    a = heavy_calc(1)
    b = heavy_calc(2)  // Последовательное выполнение, ожидание завершения a
    c = heavy_calc(3)  // Последовательное выполнение, ожидание завершения b
    a + b + c
}

// spawn-блок вводит параллелизм по потоку данных
process: () -> Void = () => {
    spawn {
        users = fetch_users()   // Параллельно
        posts = fetch_posts()   // Параллельно
    }
    // Вызывающая сторона синхронно блокируется в ожидании результатов
    render(users, posts)
}
```

### 3. Обработка ошибок

```yaoxiang
Result: Type(T, E) = { ok(T) | err(E) }

process: () -> Result(Data, Error) = {
    data = fetch_data()?      // Оператор ? прозрачно распространяет ошибку
    transformed = transform(data)?
    save(transformed)?
}
```
