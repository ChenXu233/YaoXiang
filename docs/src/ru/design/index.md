# Документ проектирования YaoXiang

> Дао рождает одно, одно рождает два, два рождают три, три рождают все сущее.

Данный каталог содержит проектные решения, предложения и обсуждения языка программирования YaoXiang.

## Основные проектные принципы

| Принцип | Описание |
|---------|----------|
| **Всё есть тип** | Значения, функции и модули являются типами; типы — граждане первого класса |
| **Естественный синтаксис** | Читаемость как у Python, близость к естественному языку |
| **Модель владения** | Абстракции с нулевой стоимостью, без GC, высокая производительность |
| **Модель 并作** | Синхронный синтаксис, асинхронная сущность, автоматический параллелизм |
| **Дружественность к AI** | Строгая структурированность, чёткий AST |

## Участие в проектных обсуждениях

### Жизненный цикл RFC

Предложения RFC имеют 5 статусов:

| Статус | Значение |
|--------|----------|
| Черновик | Работа в процессе |
| На рассмотрении | Открытое обсуждение |
| Принято | Проект одобрен |
| Устарело | Было принято, заменено новым проектом |
| Отклонено | Не одобрено |

Полный жизненный цикл:
```
Черновик → На рассмотрении → Принято → Устарело (заменено)
                        ↓
                   Отклонено (не одобрено)
```

### Процесс подачи предложений

```
1. Составление черновика (использование шаблона RFC)
   → помещается в rfc/draft/

2. Подача на рассмотрение
   → перемещается в rfc/review/, открыто общественное обсуждение

3. Рассмотрение основной командой
   → принятие → перемещается в rfc/accepted/
   → отклонение → перемещается в rfc/rejected/

4. Последующее сопровождение
   → замена → перемещается в rfc/deprecated/
```

### Принципы проектирования

- **Чёткие границы**: каждое проектное решение должно иметь чёткую область применения
- **Практичность прежде всего**: решать реальные проблемы, а не воображаемые угрозы
- **Неизменность видимого поведения для пользователей**: Never break userspace

## Ключевые проектные решения

### 1. Система типов

- **Унифицированный синтаксис типов**: отказ от `enum`, `struct`, `union`, унификация через `Name: Type = {...}`
- **Конструкторы как типы**: устранение разрыва между «типами» и «значениями»
- **Поддержка generics**: компиляция времени мономорфизация, нулевые накладные расходы runtime

### 2. Модель 并作

```yaoxiang
// Модель 并作: последовательное выполнение по умолчанию, spawn вводит поток данных параллелизм

// Последовательное выполнение по умолчанию
compute: (Int) -> Int = (n) => {
    a = heavy_calc(1)
    b = heavy_calc(2)  // Последовательное выполнение, ожидание a
    c = heavy_calc(3)  // Последовательное выполнение, ожидание b
    a + b + c
}

// Блок spawn вводит поток данных параллелизм
process: () -> Void = () => {
    spawn {
        users = fetch_users()   // параллельно
        posts = fetch_posts()   // параллельно
    }
    // Вызывающий код синхронно блокируется в ожидании результатов
    render(users, posts)
}
```

### 3. Обработка ошибок

```yaoxiang
Result: Type(T, E) = { ok(T) | err(E) }

process: () -> Result(Data, Error) = {
    data = fetch_data()?      // Оператор ? прозрачно распространяет
    transformed = transform(data)?
    save(transformed)?
}
```
