## Аннотация

Добавление более мощных возможностей сопоставления с образцом для YaoXiang, включая вложенные образцы, охранные выражения и привязку переменных через `let`.

## Мотивация

### Зачем нужна эта функция?

Текущие выражения `match` имеют ограниченные возможности и не могут обрабатывать следующие распространённые сценарии:

```yaoxiang
# Невозможность деструктурировать вложенные структуры
Person: Type = { name: String, address: Address }
Address: Type = { city: String, zip: Int }
match person {
    Person(name: "Alice", address: Address(city: "Beijing", _)) => "Alice from Beijing"  # ❌ Не поддерживается
}

# Невозможность привязать переменные в образце
match result {
    ok(value) => print(value)          # ❌ Требуется явная деструктуризация
}
```

### Текущие проблемы

1. Отсутствует поддержка деструктуризации вложенных образцов
2. Невозможно использовать охранные выражения в образцах
3. Оператор `let` не поддерживает сопоставление с образцом

## Предложение

### Основной дизайн

Расширение синтаксиса выражений `match` для поддержки:

1. **Деструктуризация вложенных образцов**: произвольная глубина структурной деструктуризации
2. **Охранные выражения**: добавление условия `if` после образца
3. **Привязка переменных образца**: прямая привязка переменных из образца

### Примеры

```yaoxiang
# Вложенная деструктуризация
Person: Type = { name: String, address: Address }
Address: Type = { city: String, zip: Int }

match person {
    Person(name: "Alice", address: Address(city: "Beijing", _)) => "Alice from Beijing"
    Person(name: n, address: Address(city: c, _)) => n + " from " + c
}

# Охранные выражения
match n {
    n if n > 0 && n < 10 => "1-9"
    n if n >= 10 => "10+"
    _ => "unknown"
}

# Привязка образца
match result {
    ok(value) => print(value)          # value уже привязана
    err(e) => log_error(e)
}

# Вложенность + привязка
match data {
    User(name: first, profile: Profile(age: a)) if a >= 18 => first + " is adult"
}
```

### Сопоставление с образцом в операторе `let`

```yaoxiang
# Новый синтаксис
let Point(x: 0, y: _) = point  # Привязка только если x == 0
let Ok(value) = result         # Деструктурировать Result

# Множественная привязка
let (a, b, c) = tuple          # Деструктурировать кортеж
```

## Детальный дизайн

### Синтаксические изменения

```
MatchExpr   ::= 'match' Expr '{' MatchArm+ '}'
MatchArm    ::= Pattern ('|' Pattern)* ('if' Expr)? '=>' Expr ','
Pattern     ::= LiteralPattern
              | IdentifierPattern
              | StructPattern
              | TuplePattern
              | OrPattern
              | RestPattern

LiteralPattern ::= '_' | Literal
IdentifierPattern ::= Identifier (':' Pattern)?
StructPattern ::= Identifier '(' FieldPattern (',' FieldPattern)* ','? ')'
FieldPattern  ::= Identifier ':' Pattern | Identifier
TuplePattern  ::= '(' Pattern (',' Pattern)* ','? ')'
OrPattern     ::= Pattern '|' Pattern
RestPattern   ::= '...'
```

### Влияние на типовую систему

- Проверка типов при сопоставлении с образцом требует расширения
- Типы переменных образца должны корректно определяться при успешном сопоставлении

### Изменения в компиляторе

| Компонент | Изменения |
|-----------|-----------|
| lexer | Новые токены, связанные с образцами |
| parser | Новая логика разбора образцов |
| typecheck | Вывод типов образцов и привязка |
| codegen | Генерация кода для сопоставления с образцом |

### Обратная совместимость

- ✅ Полная обратная совместимость
- Добавляется только новый синтаксис, старый синтаксис `match` не меняется

## Компромиссы

### Преимущества

- Более выразительный синтаксис, более лаконичный код
- Согласованность с主流ными языками сопоставления с образцом (Rust, Scala, Elixir)
- Снижение количества ошибок времени выполнения, раннее обнаружение несовпадений

### Недостатки

- Увеличение сложности реализации компилятора
- Незначительное увеличение кривой обучения

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Только деструктуризация верхнего уровня | Невозможность обработки распространённых вложенных сценариев |
| Использование функционального стиля | Естественное смешивание с императивным кодом |
| Отложить до v2.0 | У пользователей уже есть острая потребность |

## Стратегия реализации

### Этапы

1. **Этап 1 (v0.6)**: Вложенная деструктуризация и охранные выражения
2. **Этап 2 (v0.7)**: Привязка переменных образца
3. **Этап 3 (v0.8)**: Сопоставление с образцом в `let`

### Зависимости

- Нет внешних зависимостей
- Требуется предварительное завершение базовой типовой системы

### Риски

- Сложность компиляции образцов может привести к проблемам производительности
- Слишком глубокая вложенность может вызвать переполнение стека
