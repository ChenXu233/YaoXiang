# RFC-пример: Расширенный синтаксис сопоставления с образцом

## Краткое содержание

Добавление более мощных возможностей сопоставления с образцом в YaoXiang, включая вложенные образцы, guard-выражения и привязку переменных через `let`.

## Мотивация

### Зачем нужна эта функция?

Текущие выражения `match` имеют ограниченный функционал и не могут обрабатывать следующие распространённые сценарии:

```yaoxiang
# Невозможна деструктуризация вложенных структур
Person: Type = { name: String, address: Address }
Address: Type = { city: String, zip: Int }
match person {
    Person(name: "Alice", address: Address(city: "Beijing", _)) => "Alice from Beijing"  # ❌ Не поддерживается
}

# Невозможна привязка переменных в образце
match result {
    ok(value) => print(value)          # ❌ Требуется явная деструктуризация
}
```

### Текущие проблемы

1. Отсутствует поддержка вложенной деструктуризации образцов
2. Невозможно использовать guard-выражения в образцах
3. Оператор `let` не поддерживает сопоставление с образцом

## Предложение

### Основной дизайн

Расширение синтаксиса выражений `match` для поддержки:

1. **Вложенная деструктуризация образцов**: деструктуризация структур произвольной глубины
2. **Guard-выражения**: добавление условий `if` после образца
3. **Привязка переменных через образец**: прямая привязка переменных из образца

### Примеры

```yaoxiang
# Вложенная деструктуризация
Person: Type = { name: String, address: Address }
Address: Type = { city: String, zip: Int }

match person {
    Person(name: "Alice", address: Address(city: "Beijing", _)) => "Alice from Beijing"
    Person(name: n, address: Address(city: c, _)) => n + " from " + c
}

# Guard-выражения
match n {
    n if n > 0 && n < 10 => "1-9"
    n if n >= 10 => "10+"
    _ => "unknown"
}

# Привязка через образец
match result {
    ok(value) => print(value)          # value уже привязана
    err(e) => log_error(e)
}

# Вложенность + привязка
match data {
    User(name: first, profile: Profile(age: a)) if a >= 18 => first + " is adult"
}
```

### Сопоставление с образцом в операторе `let`

```yaoxiang
# Новый синтаксис
let Point(x: 0, y: _) = point  # Привязка только если x == 0
let Ok(value) = result         # Деструктуризация Result

# Множественная привязка
let (a, b, c) = tuple          # Деструктуризация кортежа
```

## Детальный дизайн

### Синтаксические изменения

```
MatchExpr   ::= 'match' Expr '{' MatchArm+ '}'
MatchArm    ::= Pattern ('|' Pattern)* ('if' Expr)? '=>' Expr ','
Pattern     ::= LiteralPattern
              | IdentifierPattern
              | StructPattern
              | TuplePattern
              | OrPattern
              | RestPattern

LiteralPattern ::= '_' | Literal
IdentifierPattern ::= Identifier (':' Pattern)?
StructPattern ::= Identifier '(' FieldPattern (',' FieldPattern)* ','? ')'
FieldPattern  ::= Identifier ':' Pattern | Identifier
TuplePattern  ::= '(' Pattern (',' Pattern)* ','? ')'
OrPattern     ::= Pattern '|' Pattern
RestPattern   ::= '...'
```

### Влияние на систему типов

- Проверка типов при сопоставлении с образцом требует расширения
- Переменные образца получают корректный тип при успешном сопоставлении

### Изменения в компиляторе

| Компонент | Изменение |
|-----------|-----------|
| lexer | Новые token для образцов |
| parser | Новая логика разбора образцов |
| typecheck | Вывод типов образцов и привязка |
| codegen | Генерация кода для сопоставления с образцом |

### Обратная совместимость

- ✅ Полная обратная совместимость
- Добавляется только новый синтаксис, старый синтаксис `match` не меняется

## Компромиссы

### Преимущества

- Более выразительный синтаксис, более лаконичный код
- Согласованность с сопоставлением с образцом в основных языках (Rust, Scala, Elixir)
- Сокращение ошибок времени выполнения, ранний перехват несопоставленных случаев

### Недостатки

- Увеличение сложности реализации компилятора
- Незначительное повышение кривой обучения

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Только деструктуризация верхнего уровня | Не позволяет обрабатывать распространённые вложенные сценарии |
| Использование функционального стиля | Неестественное смешение с императивным кодом |
| Отложить до v2.0 | У пользователей уже есть острая потребность |

## Стратегия реализации

### Фазы

1. **Фаза 1 (v0.6)**: Вложенная деструктуризация и guard-выражения
2. **Фаза 2 (v0.7)**: Привязка переменных через образец
3. **Фаза 3 (v0.8)**: Сопоставление с образцом в `let`

### Зависимости

- Нет внешних зависимостей
- Требуется предварительное завершение базовой системы типов

### Риски

- Сложность компиляции образцов может вызвать проблемы с производительностью
- Слишком глубокая вложенность может привести к переполнению стека
