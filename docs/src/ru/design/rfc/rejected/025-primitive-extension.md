# RFC-025: Механизм расширяемых примитивных типов

## Резюме

Данный документ определяет **механизм расширяемых примитивных типов** (`Primitive::Extension`) компилятора YaoXiang. Он позволяет внешнему коду регистрировать в компиляторе пользовательские примитивные типы, чтобы компилятор мог поддерживать специфичные для предметной области типы (квантовые биты, буферы GPU, SIMD-векторы, аппаратные регистры и т.д.) без хардкода.

## Мотивация

### Зачем нужен этот механизм?

В настоящее время компилятор жёстко прописывает все примитивные типы: `Int`, `Float`, `String`, `Bool`, `Unit`. Добавление нового примитивного типа требует изменения исходного кода компилятора во многих местах — в проверщике типов, генераторе кода, анализаторе владения, оптимизаторе.

Это нарушает принцип открытости/закрытости: открыт для расширения, закрыт для изменения.

### Границы проектирования

```
Хардкод базовых типов (фундамент языка): Int, Float, String, Bool, Unit
Динамически расширяемые типы (плагины предметной области):  регистрация через Primitive::Extension
```

Базовые типы жёстко прописаны, поскольку компилятор глубоко зависит от их семантики (арифметические операции, условные переходы, хеширование, сравнение). Расширяемые типы являются для компилятора **непрозрачными значениями** — компилятор знает только их размер, выравнивание и свойства владения, но не знает их внутренней семантики.

Это не «унификация всех типов через динамическую загрузку». Базовые и расширяемые типы — это две разные вещи.

## Предложение

### Основной дизайн

#### 1. Атрибуты типа Extension

При регистрации каждый расширяемый примитивный тип должен объявить следующие атрибуты:

```rust
pub struct PrimitiveExtension {
    /// Имя типа, например "Qubit", "Buffer", "Vec128"
    pub name: String,

    /// Размер в байтах (тип фиксированного размера)
    /// None означает, что размер неизвестен на этапе компиляции (определяется во время выполнения)
    pub size: Option<usize>,

    /// Требования к выравниванию
    pub align: Option<usize>,

    /// Допускается ли неявное копирование
    /// false = семантика Move (присваивание перемещает, как Qubit)
    /// true = семантика Copy (присваивание копирует, как Vec128)
    pub is_copy: bool,

    /// Допускается ли пустое значение (zero-sized type)
    pub allow_zst: bool,
}
```

#### 2. Интерфейс регистрации

```rust
// Внутренний API компилятора
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".to_string(),
    size: Some(0),           // логический размер 0, физическое состояние в квантовом процессоре
    align: Some(1),
    is_copy: false,          // семантика Move, соответствует no-cloning
    allow_zst: true,
});
```

После регистрации `Qubit` становится допустимым примитивным типом в системе типов и может использоваться в объявлениях переменных, параметрах функций, полях структур.

#### 3. Поведение проверщика типов

В проверке типов расширяемые примитивные типы подчиняются следующим правилам:

| Сценарий | Поведение |
|------|------|
| Объявление переменной `q: Qubit = ...` | ✅ Допустимо |
| Параметр функции `fn(q: Qubit)` | ✅ Допустимо |
| Поле структуры `{ q: Qubit }` | ✅ Допустимо |
| Присваивание при `is_copy == false` | Семантика Move, исходная переменная становится недействительной |
| Присваивание при `is_copy == true` | Семантика Copy, исходная переменная сохраняется |
| Неявное копирование (использование в нескольких местах функции) | Зависит от `is_copy` |
| Сравнение `==`, `!=` | ❌ Ошибка компиляции (нет встроенного сравнения) |
| Арифметика `+`, `-` | ❌ Ошибка компиляции (нет встроенных операций) |
| Ограничение дженериков `T: Copy` | Удовлетворяется только при `is_copy == true` |

#### 4. Поведение генератора кода

При генерации кода расширяемые примитивные типы обрабатываются как **непрозрачные значения**:

- LLVM IR: генерируется как `{size} x i8` или структура соответствующего размера
- Никакие специальные инструкции не генерируются — семантику обеспечивает бэкенд или библиотека
- Если бэкенду требуется специальная обработка (например, квантовые вентили QIR), она реализуется через механизм регистрации бэкенда (выходит за рамки данного RFC)

### Примеры

#### Регистрация типа с семантикой Move

```rust
// Квантовый бит: не копируется, размер 0 (физическое состояние в QPU)
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".into(),
    size: Some(0),
    align: Some(1),
    is_copy: false,
    allow_zst: true,
});
```

```yaoxiang
# Пользовательский код
q: Qubit = qubit(0)
q2 = q          # ❌ Ошибка компиляции: Qubit — тип Move, q стал недействительным
q = H(q)        # ✅ Потребление q, возвращается новый q
```

#### Регистрация типа с семантикой Copy

```rust
// SIMD-вектор: копируется, фиксированный размер
compiler.register_primitive(PrimitiveExtension {
    name: "Vec128".into(),
    size: Some(16),
    align: Some(16),
    is_copy: true,
    allow_zst: false,
});
```

```yaoxiang
# Пользовательский код
a: Vec128 = load_vec128(data)
b = a           # ✅ Семантика Copy, a по-прежнему действителен
c = add_vec128(a, b)  # ✅ a и b оба доступны
```

## Подробный дизайн

### Изменения в компиляторе

| Компонент | Изменения |
|------|------|
| Система типов | Новый вариант `Ty::Extension`, хранящий метаданные `PrimitiveExtension` |
| Проверщик типов | Расширяемые типы не участвуют в разрешении встроенных операций, не удовлетворяют ограничениям встроенных trait (если явно не реализованы) |
| Анализатор владения | Семантика Move или Copy определяется в зависимости от `is_copy` |
| Генератор кода | Генерирует непрозрачные значения по `size`/`align`, без специальных инструкций |
| Сообщения об ошибках | Сообщения об ошибках для расширяемых типов ссылаются на `name`, указанное при регистрации |

### Связь с FFI

`Primitive::Extension` и RFC-021 (FFI) ортогональны:

| | Primitive::Extension | FFI |
|---|---|---|
| Назначение | Регистрация нового **типа** | Вызов внешней **функции** |
| Уровень | Система типов | Среда выполнения |
| Пример | `Qubit` — это тип | `native("sin")` — это вызов функции |

Одна предметная область может нуждаться в обоих: тип `Qubit` регистрируется через Extension, функции квантовых вентилей — через FFI.

### Обратная совместимость

- ✅ Полностью обратно совместимо
- Семантика существующих типов не изменяется
- Расширяемые типы — это новая возможность, не затрагивающая существующий код

## Компромиссы

### Преимущества

- ✅ Не нужно менять исходный код компилятора для каждой новой предметной области
- ✅ Эксперты в предметной области могут регистрировать типы самостоятельно, не полагаясь на команду компилятора
- ✅ Базовые типы остаются жёстко прописанными, без ущерба для глубокой оптимизации компилятором
- ✅ Простой интерфейс — одна структура определяет все атрибуты

### Недостатки

- ⚠️ Расширяемые типы не поддерживают встроенные операции — для реализации семантики нужны дополнительные функции или механизмы бэкенда
- ⚠️ При отладке расширяемые типы отображаются как непрозрачные значения, менее наглядно по сравнению с базовыми типами

## Альтернативы

| Вариант | Почему не выбран |
|------|--------------|
| Динамическая загрузка всех типов | Базовые типы (Int/Float/Bool) требуют глубокой оптимизации компилятором; динамическая загрузка лишит этих возможностей |
| Хардкод для каждой предметной области | Изменения в компиляторе при добавлении каждой предметной области — не масштабируется |
| Чисто библиотечный вариант (без регистрации типов) | Невозможно гарантировать семантику на уровне системы типов (например, no-cloning), только проверки во время выполнения |

## Стратегия реализации

### Этап 1: Базовый интерфейс

- [ ] Добавить вариант `Ty::Extension` в систему типов
- [ ] Реализовать API `register_primitive`
- [ ] Расширить проверщик типов для обработки семантики `is_copy`
- [ ] Расширить генератор кода для обработки непрозрачных значений
- [ ] Модульные тесты

### Этап 2: Время регистрации

- [ ] Поддержка пакетной регистрации при инициализации компилятора (конфигурационный файл или builder API)
- [ ] Поддержка предварительной регистрации в стандартной библиотеке (модуль `std.primitive` экспортирует определения расширяемых типов)

### Зависимости

- Нет жёстких зависимостей. Может быть реализован независимо от других RFC.

## Открытые вопросы

- [ ] Должны ли расширяемые типы поддерживать реализацию trait (например, позволить `Qubit` реализовать пользовательский trait `QuantumGate`)?
- [ ] Нужны ли хуки жизненного цикла (например, `on_drop`) для поддержки семантики RAII в расширяемых типах?
- [ ] Формат конфигурационного файла: TOML, YAML или собственный синтаксис конфигурации YaoXiang (см. RFC-015)?
