# RFC-002：Кроссплатформенный I/O и интеграция с libuv

> **Статус**: Отклонено
> **Автор**: Чэнь Сюй
> **Дата создания**: 2025-01-05
> **Последнее обновление**: 2026-02-15

## Причина отклонения

Данный RFC отклонён по следующим причинам:

### 1. libuv — это C-библиотека, которая не может использоваться после бутстрэппинга YaoXiang

YaoXiang в конечном счёте должен поддерживать бутстрэппинг (реализация интерпретатора на самом YaoXiang), и в этом случае невозможно полагаться на C-библиотеки.
libuv как C-библиотека требует FFI-вызовов, что будет препятствовать процессу бутстрэппинга.

### 2. tokio является более подходящим выбором

В экосистеме Rust tokio является доминирующей асинхронной средой выполнения (доля >90%), это чистая реализация на Rust,
после бутстрэппинга её можно продолжать использовать через биндинги, что больше соответствует долгосрочной архитектуре, чем libuv.

### 3. Прагматические соображения

На текущем этапе приоритетом должно быть успешное функционирование языка YaoXiang, I/O можно быстро реализовать с помощью Rust std,
настоящую асинхронную среду выполнения можно разработать или использовать биндинги tokio после бутстрэппинга.

---

## Аннотация

Предлагается кроссплатформенная асинхронная I/O схема для YaoXiang с интеграцией libuv для унифицированного асинхронного абстрагирования. Основная цель проектирования — автоматическое и прозрачное асинхронизирование блокирующих I/O операций, чтобы разработчикам не требовалось заботиться о низкоуровневых деталях.

## Мотивация

### Зачем нужен libuv?

Модель параллелизма YaoXiang требует эффективной поддержки асинхронного I/O:

| Потребность     | Проблема традиционных решений                                          |
| ----------- | --------------------------------------------------------------------- |
| Кроссплатформенный I/O | Различные API на разных платформах (Windows IOCP, Linux epoll, macOS kqueue) |
| Асинхронный цикл событий | Реализация с нуля сложна и подвержена ошибкам                              |
| Управление пулом потоков  | Блокирующие операции требуют выделенного пула потоков                          |
| Требования к производительности   | Требуется асинхронная абстракция с нулевыми накладными расходами                       |

### Преимущества libuv

```
libuv ✓ Зрелость и стабильность - Нижний уровень времени выполнения Node.js, проверенный в крупномасштабных сценариях
libuv ✓ Кроссплатформенность - Унифицированные I/O API для Windows, Linux, macOS
libuv ✓ Высокая производительность - Управляемый событиями, неблокирующий I/O
libuv ✓ Пул потоков - Встроенное управление пулом потоков для блокирующих операций
```

## Предложение

### 1. Решение по техническому выбору

| Компонент | Выбор | Обоснование |
|-----------|-------|-------------|
| Среда выполнения I/O | libuv | Кроссплатформенная зрелость, верификация Node.js |
| Цикл событий | libuv loop | Лёгкий вес, высокая эффективность |
| Пул потоков | libuv + собственная реализация | Для блокирующих операций |
| Алгоритм планирования | Work stealing + оптимизация DAG | Высокая производительность, балансировка нагрузки |
| Управление памятью | Ownership + размещение в стеке | Без GC, абстракция с нулевой стоимостью |

### 2. Проектирование планировщика

```rust
// Планировщик с похищением работы
pub struct WorkStealingScheduler {
    workers: Vec<Worker>,
    global_queue: ConcurrentDeque<Task>,
    victim_queue: AtomicUsize,
}

impl WorkStealingScheduler {
    pub fn schedule(&self, task: Task) {
        // Приоритетное добавление в локальную очередь
        if let Ok(worker) = self.current_worker() {
            worker.local_queue.push_back(task);
        } else {
            // Без воркера добавляем в глобальную очередь
            self.global_queue.push_back(task);
        }
    }

    pub fn steal(&self, victim: &Worker) -> Option<Task> {
        // Похищение задачи из очереди другого воркера
        victim.local_queue.pop_back()
    }
}
```

```rust
#[derive(Debug)]
pub enum IoError {
    FileNotFound(String),
    PermissionDenied(String),
    IoErrno(i32, String),
    Cancelled,
}

impl From<uv::UvError> for IoError {
    fn from(err: uv::UvError) -> Self {
        match err.code() {
            uv::ENOENT => IoError::FileNotFound(err.path()),
            uv::EACCES => IoError::PermissionDenied(err.path()),
            _ => IoError::IoErrno(err.code(), err.message()),
        }
    }
```

## Компромиссы

### Преимущества

1. **Кроссплатформенная согласованность**: Единый API охватывает все основные платформы
2. **Высокая производительность**: Управляемый событиями + похищение работы, производительность близка к рукописному асинхронному коду
3. **Прозрачная асинхронизация**: Разработчикам не нужно вручную обрабатывать асинхронные детали
4. **Безопасность блокировки**: Блокирующие операции автоматически попадают в пул потоков, не блокируя цикл событий
5. **Зрелость и стабильность**: libuv проверен в крупномасштабных сценариях Node.js

### Недостатки

1. **Введение зависимости**: Требуется биндинг к C-библиотеке libuv
2. **Совместимость с Windows**: Некоторые API имеют незначительные различия в поведении на Windows
3. **Поддержка WASM**: Требуется дополнительная работа по адаптации
4. **Сложность отладки**: Асинхронные трассировки стека могут быть неполными

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Реализация цикла событий с нуля | Сложно и подвержено ошибкам, невозможно сравнить со зрелостью libuv |
| Использование mio | Только примитивные асинхронные операции, отсутствует пул потоков |
| Использование async-std/tokio | Экосистема Rust, но YaoXiang нужна собственная среда выполнения |
| Прямое использование libc epoll | Невозможно кроссплатформенно |

## Стратегия реализации

### Фазы разделения

1. **Фаза 1 (v0.1)**: Базовые биндинги libuv, простой файловый I/O
2. **Фаза 2 (v0.3)**: Сетевой I/O, интеграция пула потоков
3. **Фаза 3 (v0.5)**: Продвинутые возможности, потоковый API
4. **Фаза 4 (v1.0)**: Адаптация WASM, оптимизация производительности

### Зависимости

- Нет внешних зависимостей от RFC
- **RFC-001 Модель параллелизма**: Определяет DAG планировщик, RFC-002 предоставляет I/O абстракцию

## Интеграция с RFC-001 Моделью параллелизма

RFC-001 определяет **DAG планировщик** (уровень планирования), RFC-002 определяет **libuv + пул потоков** (уровень I/O). Оба работают совместно для реализации «синхронный синтаксис, автоматический параллелизм».

### Многоуровневая архитектура

```
┌─────────────────────────────────────────────────────────────┐
│                    YaoXiang Runtime                         │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────────────┐    ┌─────────────────────┐        │
│  │   RFC-001: DAG      │    │  RFC-002: libuv     │        │
```

### Процесс сотрудничества

```markdown
1. **Во время компиляции**: Операции с типами ресурсов распознаются как узлы I/O
   - File.read, HTTP.get и др. помечаются как "требуют асинхронного выполнения"
   - Создаётся DAG узел, помеченный как тип I/O

2. **Во время выполнения**: Планировщик DAG встречает узел I/O
   - Распознаётся как невычислительный узел, отправляется в libuv
   - Планировщик продолжает выполнение других исполняемых узлов

3. **Завершение I/O**: Срабатывает callback libuv
   - Пул потоков libuv завершает блокирующую операцию
   - Callback уведомляет DAG планировщик
   - Нижестоящие узлы становятся исполняемыми
```

### Протокол интерфейса

```rust
// Интерфейс узла I/O, определённый в RFC-001
trait IoScheduler {
    // Отправка I/O задачи, возврат future/handle
    fn submit_io(&self, task: IoTask) -> IoHandle;

    // При завершении I/O вызывается libuv, активирует DAG узел
    fn on_io_complete(&self, handle: IoHandle);
}

// Реализация интеграции libuv из RFC-002
impl IoScheduler for LibUvRuntime {
    fn submit_io(&self, task: IoTask) -> IoHandle {
        // 1. Отправка задачи в пул потоков libuv
        let handle = self.thread_pool.submit(|| {
            // Блокирующее выполнение реального I/O
            let result = perform_blocking_io(&task);
            // 2. Завершение I/O, вызов callback
            self.on_io_complete(handle);
        });
        handle
    }

    fn on_io_complete(&self, handle: IoHandle) {
        // Уведомление DAG планировщика об активации нижестоящих узлов
        self.dag_scheduler.wake_dependents(handle.node_id);
    }
}
```

### Механизм прозрачной асинхронизации

#### Обработка во время компиляции

```yaoxiang
# Код пользователя (синхронный синтаксис)
read_config: String -> Config = (path) => {
    content = File.read(path)  # Операция с ресурсом
    parse_yaml(content)
}

# Автоматическое преобразование во время компиляции
# 1. Идентификация File.read как операции с типом ресурса
# 2. Создание DAG узла, помеченного как тип I/O
# 3. Добавление неявной точки await
```

#### Обработка во время выполнения

```markdown
| Шаг | Операция | Описание |
|------|------|------|
| 1 | Разбор DAG | Обнаружение узла I/O |
| 2 | Отправка I/O | Добавление задачи в пул потоков libuv |
| 3 | Продолжение планирования | Выполнение других исполняемых узлов |
| 4 | Завершение I/O | Срабатывает callback libuv |
| 5 | Активация нижестоящих | DAG планировщик возобновляет ожидающие узлы |
```

### Отображение типов ресурсов и I/O операций

```yaoxiang
# RFC-001 Определение: Типы ресурсов
FilePath: Resource
HttpUrl: Resource

# RFC-002 Реализация: I/O семантика операций с ресурсами
File.read: (FilePath) -> String = path => {
    # Помечается как I/O операция, автоматически входит в пул потоков libuv
}

HTTP.get: (HttpUrl) -> Response = url => {
    # Помечается как I/O операция, использует асинхронный сетевой API libuv
}
```

**Правила обработки**:
- Операции с параметрами типа ресурса → Помечаются как узлы I/O
- Узлы I/O отправляются в пул потоков libuv
- Callback завершения активирует нижестоящие узлы DAG

### Риски

1. **Полнота биндингов libuv**: Полный биндинг требует значительной работы
2. **Совместимость с Windows**: Некоторые API требуют специальной обработки
3. **Накладные расходы производительности**: FFI-вызовы имеют определённые накладные расходы
4. **Сложность интеграции**: Координация пула потоков libuv и DAG планировщика требует тщательного проектирования
