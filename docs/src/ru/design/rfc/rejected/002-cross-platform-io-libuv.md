# RFC-002：Кроссплатформенный ввод-вывод и интеграция с libuv

## Причина отклонения

Данный RFC был отклонён по следующим причинам:

### 1. libuv — это C-библиотека, её нельзя использовать после бутстрэппинга YaoXiang

YaoXiang в конечном итоге должен пройти бутстрэппинг (реализовать интерпретатор на самом YaoXiang), и в этот момент нельзя полагаться на C-библиотеки.
libuv как C-библиотека требует FFI-вызовов, что будет препятствовать процессу бутстрэппинга.

### 2. tokio — более подходящий выбор

В экосистеме Rust tokio является доминирующей асинхронной runtime-средой (доля >90%), она реализована на чистом Rust,
после бутстрэппинга её можно продолжать использовать через биндинги, что лучше соответствует долгосрочной архитектуре.

### 3. Прагматические соображения

На данном этапе приоритетом должно быть создание работающего языка YaoXiang; реализация ввода-вывода может быть быстро выполнена с помощью Rust std,
а реальная асинхронная runtime-среда может быть разработана после бутстрэппинга с использованием биндингов tokio или собственной разработки.

---

## Аннотация

Предлагается кроссплатформенное асинхронное решение для ввода-вывода YaoXiang с интеграцией libuv для унифицированных асинхронных абстракций. Основная цель проектирования — автоматическая прозрачная асинхронизация блокирующих операций ввода-вывода, чтобы разработчикам не нужно было заботиться о низкоуровневых деталях.

## Мотивация

### Почему нужен libuv?

Модель spawn в YaoXiang требует эффективной поддержки асинхронного ввода-вывода:

| Потребность | Проблема традиционных решений |
| ----------- | ----------------------------- |
| Кроссплатформенный ввод-вывод | API разных платформ не унифицированы (Windows IOCP, Linux epoll, macOS kqueue) |
| Асинхронный цикл событий | Реализация с нуля сложна и подвержена ошибкам |
| Управление пулом потоков | Блокирующие операции требуют выделенного пула потоков |
| Требования к производительности | Необходимы асинхронные абстракции с нулевыми накладными расходами |

### Преимущества libuv

```
libuv ✓ Зрелость и стабильность — низкоуровневая runtime-среда Node.js, проверенная в масштабе
libuv ✓ Кроссплатформенность — унифицированные API ввода-вывода для Windows, Linux, macOS
libuv ✓ Высокая производительность — событийно-управляемая, неблокирующая модель ввода-вывода
libuv ✓ Пул потоков — встроенное управление пулом потоков для блокирующих операций
```

## Предложение

### 1. Решение по выбору технологии

| Компонент | Выбор | Обоснование |
|-----------|-------|-------------|
| I/O runtime | libuv | Кроссплатформенная зрелость, проверена в Node.js |
| Цикл событий | libuv loop | Лёгкий, эффективный |
| Пул потоков | libuv + собственный | Для блокирующих операций |
| Алгоритм планирования | Work-stealing + оптимизация DAG | Высокая производительность, балансировка нагрузки |
| Управление памятью | Ownership + распределение в стеке | Без GC, абстракции с нулевой стоимостью |

### 2. Проектирование планировщика

```rust
// Планировщик с похищением работы
pub struct WorkStealingScheduler {
    workers: Vec<Worker>,
    global_queue: ConcurrentDeque<Task>,
    victim_queue: AtomicUsize,
}

impl WorkStealingScheduler {
    pub fn schedule(&self, task: Task) {
        // Приоритетное добавление в локальную очередь
        if let Ok(worker) = self.current_worker() {
            worker.local_queue.push_back(task);
        } else {
            // Без воркера — добавление в глобальную очередь
            self.global_queue.push_back(task);
        }
    }

    pub fn steal(&self, victim: &Worker) -> Option<Task> {
        // Похищение задачи из очереди другого воркера
        victim.local_queue.pop_back()
    }
}
```

```rust
#[derive(Debug)]
pub enum IoError {
    FileNotFound(String),
    PermissionDenied(String),
    IoErrno(i32, String),
    Cancelled,
}

impl From<uv::UvError> for IoError {
    fn from(err: uv::UvError) -> Self {
        match err.code() {
            uv::ENOENT => IoError::FileNotFound(err.path()),

            _ => IoError::IoErrno(err.code(), err.message()),

    }
```

## Компромиссы

### Преимущества

1. **Кроссплатформенная согласованность**: единый API для всех основных платформ
2. **Высокая производительность**: событийно-управляемая модель с похищением работы, близка к производительности ручной асинхронной реализации
3. **Прозрачная асинхронизация**: разработчикам не нужно вручную обрабатывать асинхронные детали
4. **Безопасность блокировки**: блокирующие операции автоматически переходят в пул потоков, не блокируя цикл событий
5. **Зрелость и стабильность**: libuv проверен в масштабе Node.js

### Недостатки

1. **Введение зависимости**: требуются биндинги к C-библиотеке libuv
2. **Совместимость с Windows**: некоторые API имеют незначительные различия в поведении на Windows
3. **Поддержка WASM**: требуется дополнительная работа по адаптации
4. **Сложность отладки**: асинхронные трассировки стека могут быть неполными

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Реализация цикла событий с нуля | Сложно и подвержено ошибкам, невозможно сравниться со зрелостью libuv |
| Использование mio | Предоставляет только низкоуровневые асинхронные примитивы, отсутствует пул потоков |
| Использование async-std/tokio | Экосистема Rust, но YaoXiang нужна собственная runtime-среда |
| Прямое использование libc epoll | Невозможно обеспечить кроссплатформенность |

## Стратегия реализации

### Фазы

1. **Фаза 1 (v0.1)**: Базовые биндинги libuv, простой файловый ввод-вывод
2. **Фаза 2 (v0.3)**: Сетевой ввод-вывод, интеграция пула потоков
3. **Фаза 3 (v0.5)**: Продвинутые возможности, потоковые API
4. **Фаза 4 (v1.0)**: Адаптация WASM, оптимизация производительности

### Зависимости

- Нет внешних зависимостей от RFC
- **RFC-001 Модель конкурентности**: определяет DAG-планировщик, RFC-002 предоставляет абстракцию ввода-вывода

## Интеграция с моделью конкурентности RFC-001

RFC-001 определяет **DAG-планировщик** (уровень планирования), RFC-002 определяет **libuv + пул потоков** (уровень ввода-вывода). Оба работают совместно для реализации «синхронный синтаксис, автоматическая конкурентность».

### Процесс сотрудничества

```markdown
1. **Компиляция**: операции с типами ресурсов распознаются как узлы ввода-вывода
   - File.read, HTTP.get и т.д. помечаются как «требуют асинхронного выполнения»
   - Создаются узлы DAG, помеченные как тип I/O

2. **Выполнение**: DAG-планировщик встречает узел ввода-вывода
   - Распознаётся как невычислительный узел, отправляется в libuv
   - Планировщик продолжает выполнять другие исполняемые узлы

3. **Завершение I/O**: срабатывает коллбэк libuv
   - Пул потоков libuv завершает блокирующую операцию
   - Коллбэк completion уведомляет DAG-планировщик
   - Нижестоящие узлы становятся исполняемыми
```

```rust
// Интерфейс узла ввода-вывода, определённый в RFC-001
trait IoScheduler {
    // Отправка задачи ввода-вывода, возврат future/handle
    fn submit_io(&self, task: IoTask) -> IoHandle;

    // Вызывается libuv при завершении ввода-вывода, пробуждает узел DAG
    fn on_io_complete(&self, handle: IoHandle);
}

// Реализация интеграции libuv в RFC-002
impl IoScheduler for LibUvRuntime {
    fn submit_io(&self, task: IoTask) -> IoHandle {
        // 1. Отправка задачи в пул потоков libuv
        let handle = self.thread_pool.submit(|| {
            // Блокирующее выполнение реального ввода-вывода
            let result = perform_blocking_io(&task);
            // 2. Завершение ввода-вывода, вызов коллбэка
            self.on_io_complete(handle);
        });
        handle
    }

    fn on_io_complete(&self, handle: IoHandle) {
        // Уведомление DAG-планировщика о пробуждении нижестоящих узлов
        self.dag_scheduler.wake_dependents(handle.node_id);
    }
}
```

### Механизм прозрачной асинхронизации

#### Обработка на этапе компиляции

```yaoxiang
# Код пользователя (синхронный синтаксис)
read_config: String -> Config = (path) => {
    content = File.read(path)  # Операция с ресурсом
    parse_yaml(content)
}

# Автоматическое преобразование на этапе компиляции
# 1. Идентификация File.read как операции с типом ресурса
# 2. Создание узла DAG, помеченного как тип I/O
# 3. Добавление неявной точки await
```

#### Обработка во время выполнения

```markdown
| Шаг | Операция | Описание |
|-----|----------|----------|
| 1 | Разбор DAG | Обнаружение узла ввода-вывода |
| 2 | Отправка I/O | Добавление задачи в пул потоков libuv |
| 3 | Продолжение планирования | Выполнение других исполняемых узлов |
| 4 | Завершение I/O | Срабатывание коллбэка libuv |
| 5 | Пробуждение нижестоящих | DAG-планировщик возобновляет ожидающие узлы |
```

### Отображение типов ресурсов и операций ввода-вывода

```yaoxiang
# RFC-001 определение: типы ресурсов
FilePath: Resource
HttpUrl: Resource

# RFC-002 реализация: семантика I/O операций с ресурсами
File.read: (FilePath) -> String = path => {
    # Помечается как операция I/O, автоматически входит в пул потоков libuv
}

HTTP.get: (HttpUrl) -> Response = url => {
    # Помечается как операция I/O, использует асинхронные сетевые API libuv
}
```

**Правила обработки**:
- Операции с параметрами типа ресурса → помечаются как узлы ввода-вывода
- Узлы ввода-вывода отправляются в пул потоков libuv для выполнения
- Коллбэк completion пробуждает нижестоящие узлы DAG

### Риски

1. **Полнота биндингов libuv**: полный биндинг требует большого объёма работы
2. **Совместимость с Windows**: некоторые API требуют специальной обработки
3. **Накладные расходы производительности**: FFI-вызовы имеют определённые накладные расходы
4. **Сложность интеграции**: координация между пулом потоков libuv и DAG-планировщиком требует тщательного проектирования
