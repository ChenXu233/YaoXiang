# RFC-002：Кроссплатформенный ввод-вывод и интеграция с libuv

> **Статус**: Отклонено
> **Автор**: Чэнь Сюй (晨煦)
> **Дата создания**: 2025-01-05
> **Последнее обновление**: 2026-02-15

## Причина отклонения

Данный RFC отклонён по следующим причинам:

### 1. libuv — это библиотека на C, которая не может использоваться после бутстрэппинга YaoXiang

YaoXiang в конечном итоге должен выполнить бутстрэппинг (реализовать интерпретатор на самом YaoXiang), и на этом этапе нельзя будет зависеть от C-библиотек.
libuv как C-библиотека требует вызовов FFI, что будет препятствовать процессу бутстрэппинга.

### 2. tokio является более подходящим выбором

В экосистеме Rust tokio является доминирующей асинхронной средой выполнения (доля рынка >90%), это реализация на чистом Rust,
после бутстрэппинга можно продолжить использовать через привязки, что лучше соответствует долгосрочной архитектуре.

### 3. Прагматические соображения

На текущем этапе приоритетом должно быть достижение работоспособности языка YaoXiang, реализация ввода-вывода может быть быстро выполнена с помощью Rust std,
настоящая асинхронная среда выполнения может быть разработана после бутстрэппинга с использованием привязок tokio или собственной реализации.

---

## Аннотация

Предлагается方案 кроссплатформенного асинхронного ввода-вывода для YaoXiang с интеграцией libuv для реализации единой асинхронной абстракции. Основная цель проектирования — автоматическое и прозрачное асинхронное преобразование блокирующих операций ввода-вывода, чтобы разработчикам не нужно было заботиться о низкоуровневых деталях.

## Мотивация

### Почему нужен libuv?

Модель 并作 YaoXiang требует эффективной поддержки асинхронного ввода-вывода:

| Потребность     | Проблема традиционного решения                                          |
| ------ | ------------------------------------------------ |
| Кроссплатформенный ввод-вывод | Различные API платформ (Windows IOCP, Linux epoll, macOS kqueue) |
| Асинхронный цикл событий | Реализация с нуля сложна и подвержена ошибкам                                       |
| Управление пулом потоков  | Блокирующие операции требуют выделенного пула потоков                                      |
| Требования к производительности   | Требуется асинхронная абстракция с нулевыми накладными расходами                                       |

### Преимущества libuv

```
libuv ✓ Зрелость и стабильность - Нижний уровень среды выполнения Node.js, проверенный в крупномасштабных приложениях
libuv ✓ Кроссплатформенность - Единые API ввода-вывода для Windows, Linux, macOS
libuv ✓ Высокая производительность - Событийно-управляемый, неблокирующий ввод-вывод
libuv ✓ Пул потоков - Встроенное управление пулом потоков для блокирующих операций
```

### 1. Решение по выбору технологии

| Компонент | Выбор | Обоснование |
|------|------|------|
| Среда выполнения ввода-вывода | libuv | Кроссплатформенная зрелость, проверено в Node.js |
| Цикл событий | libuv loop | Лёгкий вес, высокая эффективность |

### 2. Проектирование планировщика

```rust
// Планировщик с кражей работы
pub struct WorkStealingScheduler {
    workers: Vec<Worker>,
    global_queue: ConcurrentDeque<Task>,
    victim_queue: AtomicUsize,
}

impl WorkStealingScheduler {
    pub fn schedule(&self, task: Task) {
        // Приоритетное добавление в локальную очередь
        if let Ok(worker) = self.current_worker() {
            worker.local_queue.push_back(task);
        } else {
            // Без рабочего потока - добавление в глобальную очередь
            self.global_queue.push_back(task);
        }
    }

    pub fn steal(&self, victim: &Worker) -> Option<Task> {
        // Кража задачи из очереди другого рабочего потока
        victim.local_queue.pop_back()
    }
}
```

#[derive(Debug)]
pub enum IoError {
    FileNotFound(String),
    PermissionDenied(String),
    IoErrno(i32, String),
    Cancelled,
}

impl From<uv::UvError> for IoError {
    fn from(err: uv::UvError) -> Self {
        match err.code() {
            uv::ENOENT => IoError::FileNotFound(err.path()),

            _ => IoError::IoErrno(err.code(), err.message()),

## Компромиссы

### Преимущества

1. **Кроссплатформенная согласованность**: Единый API для всех основных платформ
2. **Высокая производительность**: Событийное управление + кража работы, близко к производительности рукописного асинхронного кода
3. **Прозрачная асинхронизация**: Разработчикам не нужно вручную обрабатывать асинхронные детали
4. **Безопасность блокировки**: Блокирующие операции автоматически попадают в пул потоков, не блокируя цикл событий
5. **Зрелость и стабильность**: libuv проверен в крупномасштабных приложениях Node.js

### Недостатки

1. **Введение зависимости**: Требуется привязка к библиотеке C libuv
2. **Совместимость с Windows**: Некоторые API имеют незначительные различия в поведении на Windows
3. **Поддержка WASM**: Требуется дополнительная работа по адаптации
4. **Сложность отладки**: Трассировка асинхронного стека может быть неполной

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| Реализация цикла событий с нуля | Сложно и подвержено ошибкам, невозможно сравниться со зрелостью libuv |
| Использование mio | Предоставляет только низкоуровневые асинхронные примитивы, отсутствует пул потоков |
| Использование async-std/tokio | Экосистема Rust, но YaoXiang нужна собственная среда выполнения |
| Прямое использование libc epoll | Невозможно обеспечить кроссплатформенность |

## Стратегия реализации

### Разделение на этапы

1. **Этап 1 (v0.1)**: Базовые привязки libuv, простой файловый ввод-вывод
2. **Этап 2 (v0.3)**: Сетевой ввод-вывод, интеграция пула потоков
3. **Этап 3 (v0.5)**: Продвинутые возможности, потоковый API
4. **Этап 4 (v1.0)**: Адаптация WASM, оптимизация производительности

### Зависимости

- Нет зависимостей от внешних RFC
- **RFC-001 Модель 并发**: Определяет DAG планировщик, RFC-002 предоставляет абстракцию ввода-вывода

## Интеграция с моделью 并发 RFC-001

RFC-001 определяет **DAG планировщик** (уровень планирования), RFC-002 определяет **libuv + пул потоков** (уровень ввода-вывода). Оба работают вместе для реализации «синхронный синтаксис, автоматическая 并发».

### Процесс сотрудничества

```markdown
1. **Компиляция**: Операции с типами ресурсов распознаются как узлы ввода-вывода
   - File.read, HTTP.get и т.д. помечаются как "требуют асинхронного выполнения"
   - Создаются узлы DAG, помеченные как тип ввода-вывода

2. **Выполнение**: Планировщик DAG встречает узел ввода-вывода
   - Распознаётся как невычислительный узел, отправляется в libuv
   - Планировщик продолжает выполнение других исполняемых узлов

3. **Завершение ввода-вывода**: Срабатывает обратный вызов libuv
   - Пул потоков libuv завершает блокирующую операцию
   - Обратный вызов completion уведомляет планировщик DAG
   - Нижестоящие узлы становятся исполняемыми
```

### Протокол интерфейса

```rust
// Интерфейс узла ввода-вывода, определённый в RFC-001
trait IoScheduler {
    // Отправка задачи ввода-вывода, возврат future/handle
    fn submit_io(&self, task: IoTask) -> IoHandle;

    // Вызывается libuv при завершении ввода-вывода, активирует узел DAG
    fn on_io_complete(&self, handle: IoHandle);
}

// Реализация интеграции libuv из RFC-002
impl IoScheduler for LibUvRuntime {
    fn submit_io(&self, task: IoTask) -> IoHandle {
        // 1. Отправка задачи в пул потоков libuv
        let handle = self.thread_pool.submit(|| {
            // Блокирующее выполнение фактического ввода-вывода
            let result = perform_blocking_io(&task);
            // 2. Завершение ввода-вывода, вызов обратного вызова
            self.on_io_complete(handle);
        });
        handle
    }

    fn on_io_complete(&self, handle: IoHandle) {
        // Уведомление планировщика DAG о активации нижестоящих узлов
        self.dag_scheduler.wake_dependents(handle.node_id);
    }
}
```

### Сопоставление типов ресурсов и операций ввода-вывода

```yaoxiang
# RFC-001 Определение: Типы ресурсов
FilePath: Resource
HttpUrl: Resource

# RFC-002 Реализация: Семантика ввода-вывода операций с ресурсами
File.read: (FilePath) -> String = path => {
    # Помечается как операция ввода-вывода, автоматически попадает в пул потоков libuv
}

HTTP.get: (HttpUrl) -> Response = url => {
    # Помечается как операция ввода-вывода, использует асинхронный сетевой API libuv
}
```

**Правила обработки**：
- Операции с параметрами типов ресурсов → Помечаются как узлы ввода-вывода
- Узлы ввода-вывода отправляются в пул потоков libuv для выполнения
- Обратные вызовы completion активируют нижестоящие узлы DAG

### Риски

1. **Полнота привязок libuv**: Полная привязка требует значительной работы
2. **Совместимость с Windows**: Некоторые API требуют специальной обработки
3. **Накладные расходы производительности**: Вызовы FFI имеют определённые накладные расходы
4. **Сложность интеграции**: Координация между пулом потоков libuv и планировщиком DAG требует тщательного проектирования
