# RFC-003：Планирование версий

> **Дата отклонения**: 2026-06-01

## ⚠️ Причина отклонения

**Данный RFC не соответствует стандартам RFC и был отклонён.**

### Анализ проблем

| Проблема | Пояснение |
|------|------|
| **Неправильный тип документа** | RFC — это документ решений о том, «почему так спроектировано», а данный документ — это документ управления проектом о том, «где мы сейчас и куда идём» |
| **Отсутствие проектных решений** | Нет анализа компромиссов при выборе технологий, нет сравнения альтернативных вариантов, нет обоснований дизайна |
| **Чрезмерное планирование** | Запланировано 9 версий (v0.1-v1.0), но завершён только 1.5, всё после v0.3 — предположения |
| **Высокая стоимость поддержки** | Таблица успешных показателей содержит 50 состояний, что затрудняет поддержку |
| **Путаница в номерах версий** | Пропущен v0.8 без объяснений |

### Правильный тип документа

Данный документ следует вести как **дорожную карту проекта (Roadmap)**, а не RFC. Дорожная карта фиксирует факты и кракосрочные планы, RFC фиксирует проектные решения.

## Аннотация

План выпуска версий YaoXiang — дорожная карта от v0.1 до v1.0.

**Ключевые цели**:
1. **Компиляция в байт-код**: поддержка REPL и инкрементальной компиляции
2. **Самопривязка (Bootstrap)**: написание компилятора YaoXiang на YaoXiang
3. **AOT-компиляция**: компиляция байт-кода в машинный код

## 一、Мотивация

### Зачем нужно планирование версий?

1. **Управление проектом**: разбиение целей на выполнимые этапы
2. **Ожидания пользователей**: информирование о стадиях развития языка
3. **Распределение ресурсов**: определение фокуса на каждом этапе
4. **Контроль рисков**: своевременное выявление проблем и корректировка курса

### Ключевые проектные решения

- **Приоритет байт-кода**: сначала реализуем интерпретацию, затем AOT
- **Поэтапная поставка**: каждая версия содержит работающий функционал
- **Обратная совместимость**: API может измениться до v1.0, но с предварительным уведомлением
- **Верификация самопривязкой**: доказательство выразительности языка через самопривязку
- **Слоёная оптимизация производительности**: сначала запустить, потом оптимизировать

## 四、Трёхуровневая стратегия компиляции

| Уровень | Версия | Входные данные | Выходные данные | Описание |
|------|------|------|------|------|
| L1: Байт-код | v0.1+ | Исходный код (.yx) | Байт-код (.yxb) | Интерпретация VM |
| L2: Самопривязка | v0.9+ | Исходный код YaoXiang | Байт-код | Компиляция самого себя |
| L3: AOT | v1.0+ | Исходный код/байт-код | Машинный код | Производительность на уровне натива |

**Обоснование приоритета байт-кода**:
1. **Поддержка REPL**: немедленная компиляция введённого кода, интерактивная разработка
2. **Инкрементальная компиляция**: изменение одной функции требует перекомпиляции только этой части
3. **Кроссплатформенность**: файлы .yxb работают на разных платформах,只需要 соответствующая VM

## 五、Стратегия зависимостей

**Краткосрочная перспектива**: использование библиотек Rust через повторное применение crates.io (Cargo-паразитизм)

**Текущие зависимости**:
- Параллелизм: parking_lot, crossbeam, rayon
- Структуры данных: indexmap, hashbrown, smallvec
- Сеть: tokio
- Сериализация: serde, ron

**Долгосрочная перспектива**: создание собственной стандартной библиотеки и менеджера пакетов
