## Аннотация

В данном документе определяется схема **квантовой нативной поддержки** и **интеграции с множественными backends** для языка YaoXiang. Ключевая идея: **Существующий дизайн YaoXiang (по умолчанию Move, возврат владения, непрозрачные типы, DAG планировщик, параметры 泛型 констант) естественным образом образует полную основу для квантового языка программирования без введения какого-либо нового квантового специализированного синтаксиса**. Мы добавляем небольшое количество встроенных типов (`Qubit`, `Complex`, `Topology`) и встроенных функций (квантовые вентили, измерения, топологические ограничения), используем существующие механизмы языка для реализации квантовой нативной семантики, автоматического параллелизма для максимизации использования квантовых ресурсов, гибридного классического программирования и поддержки множественных backends.

## Мотивация

### Зачем нужна квантовая нативная поддержка?

Текущая экосистема квантового программирования имеет серьёзные разрывы:

- **Низкоуровневые языки (QCIS, OpenQASM)**: напрямую управляют физическими квантовыми вентилями, но lack типа системы, механизмов абстракции, что затрудняет написание сложных алгоритмов.
- **Высокоуровневые фреймворки (Qiskit, Cirq, Q#)**: построены на расширении классических языков (Python, C#), квантовая семантика реализуется через библиотеки, что приводит к:
  - Правило невозможности клонирования квантового состояния должно соблюдаться пользователем вручную (или полагаться на линейную систему типов как запоздалую добавку).
  - Операции квантовых вентилей синтаксически отделены от классического кода, высокая стоимость обучения.
- **Гибридные вычисления**: квантовая и классическая части требуют явного разделения, отсутствует унифицированная модель потока данных.

### Текущие проблемы

Существующий дизайн YaoXiang恰好提供了完备ную основу для решения этих проблем:

| Потребность квантовых вычислений | Имеющийся дизайн YaoXiang | Пояснение |
|----------------|-------------------|------|
| Невозможность клонирования квантового состояния | **Семантика Move по умолчанию** | Присваивание перемещает владение, нет неявного копирования, естественным образом соответствует теореме о неклонировании |
| Квантовые вентили как унитарные преобразования | **Возврат владения** | `q = H(q)` потребляет исходный qubit, возвращает новый qubit, точно соответствует семантике вентиля |
| Запутанные состояния | **Непрозрачные типы** | `BellPair` можно оперировать только целиком, компилятор отслеживает время жизни, предотвращает ошибочное разложение |
| Ограничения физической топологии | **Параметры 泛型 констант** | `Qubit(Topology, N)` проверка смежности на этапе компиляции |
| Коллапс при измерении | **Повторное использование пустого состояния** | После измерения qubit становится пустым, может быть повторно инициализирован, моделируя коллапс квантового состояния |
| Автоматический параллелизм квантовых цепей | **DAG планировщик** | Операторы внутри функции автоматически параллелизуются на основе зависимостей по данным, независимые вентили естественным образом параллельны |
| Гибридный классическо-квантовый поток управления | **Унифицированный синтаксис** | Квантовые и классические операции используют одну и ту же форму `name: type = value` |

**YaoXiang 不是"добавляет квантовую поддержку", а обнаруживает, что его дизайн уже является квантовым нативным.**

> **Пояснение семантики**: В данном документе используется **семантика владения** YaoXiang для выражения квантовых операций. На уровне компилятора гарантируется неклонирование (безопасность владения). На уровне языка "потребление-создание" является **синтаксическим выражением передачи владения** — потребление = получение владения, возврат = передача владения. Базовая реализация может представлять собой реальные обратимые квантовые вентили (модификация квантового состояния на месте), а не真的要"создавать новое квантовое состояние".

### Цели проектирования

1. **Ноль нового синтаксиса**: не вводить ключевые слова `quantum`, `circuit` и т.д., все квантовые возможности выражаются через существующие механизмы языка.
2. **Безопасность типов**: компилятор гарантирует, что квантовые состояния не копируются и не используются неправомерно.
3. **Проверка топологических ограничений на этапе компиляции**: через параметры 泛型 констант `Qubit(T, N)` проверка соответствия физической топологии для операций двухкубитных вентилей на этапе компиляции.
4. **Прозрачность для множественных backends**: один и тот же квантовый код может компилироваться в QIR (универсальная экосистема) или QCIS (отечественный набор квантовых инструкций), переключение через параметры командной строки.
5. **Бесшовное смешение классики**: квантовые вычисления могут свободно вызывать классические функции, классический код также может оперировать квантовыми данными (через `ref` для совместного использования, но с ограничениями владения).

## Предложение

### Основной дизайн

#### 1. Маппинг квантовой системы типов

**Базовые типы**:
```yaoxiang
Qubit: Type0 = primitive_qubit
Complex: Type0 = { re: Float, im: Float }
```
- `Qubit` — это тип первого класса, следующий правилам владения (Move, RAII).
- `Complex` используется для представления амплитуды, компилятор может выполнять встраивание и оптимизацию.

**Квантовые вентили как функции**:
```yaoxiang
# Встроенные сигнатуры функций
H: (Qubit) -> Qubit = builtin_hadamard
X: (Qubit) -> Qubit = builtin_pauli_x
Y: (Qubit) -> Qubit = builtin_pauli_y
Z: (Qubit) -> Qubit = builtin_pauli_z
CNOT: (control: Qubit, target: Qubit) -> { Qubit, Qubit } = builtin_cnot
```
- Все вентили потребляют входной qubit и возвращают новый qubit (или пару запутанных). Синтаксис возврата владения `q = H(q)` напрямую соответствует математической семантике.
- Многокубитные вентили возвращают структуры, результат получается через сопоставление с образцом или доступ к полям.

**Измерение**:
```yaoxiang
measure: (Qubit) -> Int = builtin_measure   # Потребляет qubit, возвращает классический бит
measure_all: (List(Qubit)) -> List(Int) = builtin_measure_all
```
- После измерения qubit потребляется (становится пустым), пользователь может повторно инициализировать через повторное использование пустого состояния.

**Инициализация**:
```yaoxiang
qubit: (Int) -> Qubit = builtin_qubit   # Инициализация базисного состояния 0 или 1
```

#### 2. Запутанность и инкапсуляция непрозрачных типов

Запутанные пары инкапсулируются как непрозрачные типы, предоставляются только комбинированные операции, разложение запрещено:

```yaoxiang
# Встроенные непрозрачные типы
BellPair: Type0 = primitive_bell_pair

# Встроенные функции - только целостные операции
CNOT: (Qubit, Qubit) -> BellPair
measure_bell: (BellPair) -> { Int, Int }
split_bell: (BellPair) -> { Qubit, Qubit }  # Разложение запутанной пары (следует использовать осторожно)
apply_cnot_to_bell: (BellPair, Qubit) -> BellPair
```

**Ключевой дизайн**:
- Не предоставляются accessorы для полей, разрешены только встроенные функции для целостных операций
- `measure_bell(bp)` потребляет всю запутанную пару за один раз, возвращает классические биты
- Компилятор может отслеживать полный жизненный цикл запутанной пары

**Сравнение с Python/Qiskit**:
```
Python (Qiskit): Построение цепи во время выполнения, ошибки могут быть обнаружены только после отправки
YaoXiang:       Большинство логических ошибок перехватываются на этапе компиляции
```

**Оставшиеся 10%** (такие как физическая декогеренция, погрешности вентилей) являются проблемами hardware, которые язык не может решить.

#### 3. Ограничения физической топологии

Квантовый чип представляет собой топологический граф с ограничениями, **не любые два qubit могут выполнять двухкубитный вентиль**, они должны быть соседними. YaoXiang использует **параметры 泛型 констант** для гарантии топологических ограничений на этапе компиляции.

**Определение типа топологии**:
```yaoxiang
# Топология как тип, содержит матрицу смежности
Topology: Type0 = primitive_topology

# Встроенные константы топологии
Linear8: Topology = topology(8)          # Линейный 8-битный: 0-1-2-3-4-5-6-7
Grid3x3: Topology = topology(3, 3)        # Сетка 3x3
Ring16: Topology = topology(16, ring)    # Кольцевой 16-битный
```

**Qubit связывает топологию и позицию**:
```yaoxiang
# Qubit(T, N) - T это тип топологии, N - параметр константы позиции
q0: Qubit(Grid3x3, 0)   # Grid3x3 топология, позиция (0,0)
q1: Qubit(Grid3x3, 1)   # Grid3x3 топология, позиция (0,1)
q2: Qubit(Grid3x3, 2)   # Grid3x3 топология, позиция (0,2)
q3: Qubit(Grid3x3, 3)   # Grid3x3 топология, позиция (1,0)
```

**Автоматические ограничения для операций вентилей**:
```yaoxiang
# CNOT сигнатура типа с топологическим ограничением
CNOT: (T: Topology, I: Int, J: Int) -> (
    (Qubit(T, I), Qubit(T, J)) -> { Qubit(T, I), Qubit(T, J) }
) when adjacent(T, I, J)

# Проверка на этапе компиляции
CNOT(q0, q1)  # ✅ В Grid3x3 (0,0) и (0,1) соседние
CNOT(q0, q2)  # ❌ Ошибка компиляции: (0,0) и (0,2) не соседние
```

**`adjacent` ограничение на этапе компиляции**:
- `adjacent` — это функция этапа компиляции, использующая матрицу смежности топологии для статической проверки
- При константных индексах — 100% проверка на этапе компиляции
- При динамических индексах — генерация кода проверки во время выполнения

**Маппинг виртуальных на физические кубиты**:
```yaoxiang
# Физическая позиция неизвестна на этапе компиляции? Использовать вывод типов
q = qubit(Grid3x3)  # Автоматическое распределение позиции 0, последующий вывод
```

#### 4. Владение и линейный поток квантовых состояний

Все квантовые операции следуют семантике Move, гарантируя, что qubit не копируется:
```yaoxiang
q = qubit(0)
q2 = q          # ❌ Ошибка компиляции: q уже перемещён, нельзя использовать повторно
q = H(q)        # ✅ Потребляет q, возвращает новый q
measure(q)      # ✅ Потребляет q, после чего q становится пустым
q = qubit(0)    # ✅ Повторное использование пустого состояния
```

#### 4. Автоматический параллелизм и DAG планирование

В Standard или Full Runtime DAG планировщик автоматически анализирует квантовые программы:
```yaoxiang
apply_two_qubit_gates: () -> {Qubit, Qubit} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    # Вышеуказанные две строки не имеют зависимости по данным, DAG автоматически выполняет параллельно
    CNOT(q1, q2)   # Зависит от q1 и q2, автоматическое ожидание
}
```
- Планировщик использует конфигурацию `num_workers` (количество физических квантовых процессоров) для реального параллельного выполнения.
- Пользователю не нужно вручную упорядочивать вентили, достаточно описать поток данных.

#### 5. Гибридные классические вычисления

Классический и квантовый код полностью слиты:
```yaoxiang
grover_search: (target: Int) -> Int = () => {
    n = 4
    qubits = List(Qubit)()
    for i in 0..n {
        qubits.append(H(qubit(0)))
    }
    # Классический цикл смешан с квантовыми операциями
    oracle(qubits, target)   # oracle — это последовательность квантовых вентилей
    qubits = diffusion(qubits)
    results = measure_all(qubits)
    return decode_result(results)   # Классическая постобработка
}
```
- В одной функции можно произвольно смешивать квантовые вентили и классический поток управления.
- Система владения гарантирует, что квантовые переменные не будут ошибочно скопированы в классических ветвлениях.

#### 6. Архитектура поддержки множественных backends

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   YaoXiang 源码   │     │   Проверка типов │     │   Промежууточный │
│   (унифицированный │     │   + Анализ владения│     │   представление  │
│   синтаксис)       │────▶│                    │────▶│   DAG (граф       │
└─────────────────┘     └─────────────────┘     │   потока данных)  │
                                                └────────┬────────┘
                                                          │
                                                          ▼
                          ┌─────────────────────────────────────────────┐
                          │           Backend'ы кодогенерации (вставляемые)│
                          ├─────────────────┬───────────────────────────┤
                          │  QIR Backend    │  QCIS Backend             │
                          │  (универсальная │  (отечественный набор     │
                          │   экосистема)    │   квантовых инструкций)   │
                          ├─────────────────┼───────────────────────────┤
                          │  - Вывод .ll    │  - Вывод .qcis текста     │
                          │    файлов       │  - Адаптация под           │
                          │  - Адаптация     │    оборудование           │
                          │    под разные   │    Университета /         │
                          │    QPU          │    Гуодунь                 │
                          └─────────────────┴───────────────────────────┘
```

- **Процесс компиляции**: унифицированный frontend → построение DAG → выбор backend'а → генерация целевого кода.
- **QIR Backend**: маппит узлы DAG на квантовые gate intrinsic QIR, генерирует LLVM bitcode, может дополнительно использовать оптимизации LLVM.
- **QCIS Backend**: сериализует DAG в QCIS инструкции (например, `H q0`), поддерживает прямую отправку в консоль управления квантовым чипом.

### Примеры

#### Приготовление и измерение состояния Белла
```yaoxiang
bell_measure: () -> {Int, Int} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    bell = CNOT(q1, q2)  # Возвращает непрозрачный тип BellPair
    result = measure_bell(bell)  # Измеряет два бита за один раз
    return result
}
```

#### Квантовая телепортация (упрощённая)
```yaoxiang
teleport: (msg: Qubit, bell: BellPair) -> Qubit = (msg, bell) => {
    # Разложение запутанной пары для получения двух независимых кубитов
    (alice_qubit, bob_qubit) = split_bell(bell)

    # Операции Алисы
    (msg, alice_qubit) = CNOT(msg, alice_qubit)
    msg = H(msg)
    a1 = measure(msg)
    a2 = measure(alice_qubit)

    # Передача классической информации (автоматически обрабатывается планировщиком)
    # Операции Боба
    if a2 == 1 { bob_qubit = X(bob_qubit) }
    if a1 == 1 { bob_qubit = Z(bob_qubit) }
    return bob_qubit
}
```

## Детальной дизайн

### Встроенные типы и определения функций

В модуле `compiler/builtins` добавить:
```rust
builtins.insert("Qubit", Ty::Primitive(Primitive::Qubit));
builtins.insert("Complex", Ty::Record(vec![
    ("re", Ty::Primitive(Primitive::Float)),
    ("im", Ty::Primitive(Primitive::Float)),
]));

// Квантовые вентили
for (name, sig) in GATES {
    builtins.insert(name, Ty::Function(vec![Ty::Qubit], Ty::Qubit));
}
builtins.insert("CNOT", Ty::Function(
    vec![Ty::Qubit, Ty::Qubit],
    Ty::Record(vec![
        ("q0", Ty::Qubit),
        ("q1", Ty::Qubit)
    ])
));
builtins.insert("measure", Ty::Function(vec![Ty::Qubit], Ty::Primitive(Primitive::Int)));
builtins.insert("qubit", Ty::Function(vec![Ty::Primitive(Primitive::Int)], Ty::Qubit));
```

### Специальная обработка Qubit в проверятеле владения

- `Qubit` помечен как `!Copy` (по умолчанию Move), неявное копирование запрещено.
- Параметр функции измерения `measure` имеет тип `Qubit` (передача по значению), потребляет владение.
- В записях типов, возвращаемых многокубитными вентилями, все поля являются `Qubit`, должны соблюдать правила владения.

### Оптимизация DAG планировщиком квантовых вентилей

- Узлы квантовых вентилей рассматриваются как чистые функции (без побочных эффектов), планировщик может произвольно переупорядочивать независимые вентили.
- При выводе "последовательности квантовых инструкций" планировщик сохраняет зависимости по данным и группирует параллельные вентили (применимо для мультипроцессорных квантовых процессоров).
- Поддержка конфигурации `--target-num-qubits` и `--target-topology` для последующего размещения и маршрутизации (будущее расширение).

### Детальной маппинг QIR Backend

| Операция YaoXiang | Инструкция QIR |
|-------------------|----------------|
| `H(q)` | `call void @__quantum__qis__h__body(%Qubit* %q)` |
| `CNOT(q1, q2)` | `call void @__quantum__qis__cnot__body(%Qubit* %q1, %Qubit* %q2)` |
| `measure(q)` | `%result = call i1 @__quantum__qis__mz__body(%Qubit* %q)` |
| `qubit(0)` | `%q = call %Qubit* @__quantum__rt__qubit_allocate()` |

QIR Backend использует LLVM `-O2` для дальнейшей оптимизации и выводит bitcode, совместимый со стандартом QIR Alliance.

### Детальной маппинг QCIS Backend

| Операция YaoXiang | Инструкция QCIS |
|-------------------|-----------------|
| `H(q)` (q соответствует физическому биту 2) | `H 2` |
| `CNOT(q1,q2)` (q1→бит 0, q2→бит 1) | `CNOT 0 1` |
| `measure(q)` (бит 0) | `M 0` |
| `qubit(0)` инициализация | Подразумевается в первой инструкции использования, не требует дополнительных инструкций |

- Необходимо вести таблицу маппинга от виртуальных кубитов (переменных YaoXiang) к физическим битам.
- Поддержка проверки топологических ограничений (будущая реализация).

### Кодогенерация для гибридного классического кода

- Классическая часть (циклы, условия, целочисленные операции) генерирует нативный код (x86/ARM) как обычно, через FFI или встроенные вызовы взаимодействует с квантовым backend'ом.
- В QIR Backend классическая часть может быть понижена до LLVM IR и смешанно компилироваться с QIR.

### Влияние на систему типов

- Новые примитивные типы `Qubit` и `Complex`.
- `Qubit` автоматически имеет семантику Move, копирование запрещено.
- Сигнатуры функций квантовых вентилей должны быть зарегистрированы в системе типов.

### Обратная совместимость

- ✅ Полная обратная совместимость
- Новые встроенные типы и функции не влияют на существующий код
- Квантовые возможности являются опциональными, при отключении нет дополнительных издержек

## Компромиссы

### Преимущества

- **Нет нового синтаксиса**: разработчикам нужно изучить только небольшое количество встроенных функций для написания квантовых программ.
- **Безопасность типов**: система владения автоматически предотвращает копирование кубитов, избегая распространённых ошибок квантового программирования.
- **Автоматический параллелизм**: DAG планировщик бесплатно предоставляет параллелизм на уровне вентилей без дополнительной оптимизации компилятора.
- **Совместимость с экосистемой**: QIR Backend позволяет YaoXiang работать на множестве облачных квантовых платформ; QCIS Backend обеспечивает технологическую независимость.
- **Гибридные возможности**: естественное слияние классических и квантовых вычислений подходит для написания сложных квантовых алгоритмов (классическое управление в Shor, Grover).

### Недостатки

- **Статическое количество кубитов**: текущий дизайн предполагает, что количество кубитов известно на этапе компиляции, динамическое распределение возможно через `List(Qubit)`, но кучевое выделение памяти для `List` может вызвать дополнительные издержки (можно смягчить оптимизацией).
- **Повторное использование после измерения**: повторное использование пустого состояния позволяет повторно инициализировать кубит, но физические кубиты могут иметь время релаксации, необходимо управление системой выполнения (в настоящее время ответственность пользователя).
- **Динамическое маппинг топологии**: когда физическая топология становится известна только во время выполнения, проверка на этапе компиляции не может работать, требуется генерация кода проверки во время выполнения (текущая версия поддерживает только статическую проверку).

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Введение ключевого слова `quantum` и типа `circuit` | Увеличение нового синтаксиса, высокая стоимость обучения, противоречит принципу простого дизайна YaoXiang |
| Реализация квантовой поддержки только как библиотеки | Невозможно использовать гарантии компилятора для безопасности квантовых состояний, невозможна глубокая интеграция с DAG планировщиком |
| Подождать созревания квантового hardware перед поддержкой | Пропустить критическое окно для проектирования квантовых языков программирования |
| Повторное использование существующих квантовых фреймворков (например, Qiskit) | Квантовая семантика реализуется через библиотеки, невозможно получить гарантии безопасности системы типов и владения |
| Отдельное проектирование квантового подъязыка | Увеличение сложности языка, высокие затраты на поддержку |

## Стратегия реализации

### Фазовое разделение

| Фаза | Время | Содержание |
|------|-------|------------|
| Phase 1 | 1 месяц | Базовые квантовые типы и встроенные функции: добавление типов `Qubit`, `Complex` в компилятор, реализация проверки типов для встроенных функций, расширение проверятеля владения |
| Phase 2 | 1 месяц | DAG планировщик распознаёт квантовые вентили: модификация логики построения DAG, пометка квантовых вентилей как чистых функций, реализация параллельной группировки вывода вентилей |
| Phase 3 | 2 месяца | QIR Backend прототип: реализация генератора кода от DAG к QIR, интеграция LLVM, подключение QIR симулятора для верификации |
| Phase 4 | 2 месяца | QCIS Backend прототип: реализация перевода DAG в QCIS инструкции, проектирование таблицы маппинга виртуальных-физических битов, подключение отечественной квантовой платформы для верификации |
| Phase 5 | 2 месяца | Усиление гибридной классики: обеспечение корректной кодогенерации при пересечении классического потока управления и квантовых вентилей, поддержка `List(Qubit)`, добавление примеров программ |
| Phase 6 | 2 месяца | Оптимизация и документация: реализация базового размещения и маршрутизации, написание руководства пользователя и учебника по квантовому программированию, выпуск preview версии |

### Риски

1. **Доступность квантового hardware**: зависимость от доступности внешних квантовых симуляторов и реальных QPU.
   - **Смягчение**: приоритет — подключение к открытым симуляторам (QIR runner, Qiskit Aer), реальные QPU как долгосрочная цель.

2. **Сложность реализации backend'ов**: спецификации QIR и QCIS могут изменяться.
   - **Смягчение**: абстракция интерфейса кодогенерации, изоляция различий backend'ов, удобство последующей адаптации.

3. **Неопределённость производительности**: характеристики производительности квантовых программ отличаются от классических.
   - **Смягчение**: предоставление инструментов профилирования, чтобы пользователи понимали эффект параллелизма на уровне вентилей.

## Открытые вопросы

- [x] **Топологические ограничения**: реализовано через параметры 泛型 констант `Qubit(Topology, N)`, проверка на этапе компиляции.
- [ ] **Динамические квантовые регистры**: как `List(Qubit)` маппится в QCIS Backend? Можно генерировать соответствующее количество физических битов, но требуется механизм распределения во время выполнения.
- [ ] **Смягчение ошибок**: предоставлять ли встроенные конструкции смягчения ошибок (например, динамическая декoupling)? Можно сначала реализовать как библиотеку.
- [ ] **Взаимодействие с существующими квантовыми SDK**: можно ли импортировать модули QASM или QIR? В будущем можно рассмотреть FFI.
- [ ] **Автоматическое размещение и маршрутизация**: когда количество виртуальных кубитов превышает количество физических кубитов, как автоматически маппить?
