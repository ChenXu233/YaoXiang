## Краткое описание

В данном документе определяется **квантовая нативная поддержка** и схема **интеграции с несколькими бэкендами** для языка YaoXiang. Ключевая идея: **Существующий дизайн YaoXiang (по умолчанию Move, возврат владения, непрозрачные типы, DAG-планировщик, обобщённые константные параметры) естественным образом формирует полную основу для квантового языка программирования без введения какого-либо нового квантового специализированного синтаксиса**. Мы добавляем несколько встроенных типов (`Qubit`, `Complex`, `Topology`) и встроенных функций (квантовые вентили, измерения, топологические ограничения), используя существующие механизмы языка для реализации квантовой нативной семантики, максимального автоматического параллелизма для повышения использования квантовых ресурсов, гибридного классического программирования и поддержки нескольких бэкендов.

## Мотивация

### Зачем нужна квантовая нативная поддержка?

Текущая экосистема квантового программирования имеет серьёзные разрывы:

- **Низкоуровневые языки (QCIS, OpenQASM)**: напрямую управляют физическими квантовыми вентилями, но не имеют системы типов, механизмов абстракции, что затрудняет написание сложных алгоритмов.
- **Высокоуровневые фреймворки (Qiskit, Cirq, Q#)**: построены на расширениях классических языков (Python, C#), квантовая семантика реализуется через библиотеки, что приводит к:
  - Правило о невозможности клонирования квантового состояния требует ручного соблюдения пользователем (или зависит от линейной системы типов, добавляемой позже).
  - Операции квантовых вентилей синтаксически отделены от классического кода, высокая стоимость обучения.
- **Гибридные вычисления**: квантовые и классические части требуют явного разделения, отсутствует единая модель потока данных.

### Текущие проблемы

Существующий дизайн YaoXiang恰好提供ёт完备基础 для решения этих проблем:

| Потребность квантовых вычислений | Существующий дизайн YaoXiang | Описание |
|----------------|-------------------|------|
| Невозможность клонирования квантового состояния | **Семантика Move по умолчанию** | Присваивание перемещает владение, нет неявного копирования, естественное соблюдение теоремы о неклонировании |
| Квантовые вентили как унитарные преобразования | **Возврат владения** | `q = H(q)` потребляет исходный qubit, возвращает новый qubit, точно соответствует семантике вентиля |
| Состояния запутывания | **Непрозрачные типы** | `BellPair` можно оперировать только как целое, компилятор отслеживает время жизни, предотвращает неправильное разделение |
| Ограничения физической топологии | **Обобщённые константные параметры** | `Qubit(Topology, N)` проверка смежности на этапе компиляции |
| Коллапс при измерении | **Повторное использование пустого состояния** | После измерения qubit становится пустым, можно переинициализировать, моделируя коллапс квантового состояния |
| Автоматический параллелизм квантовых схем | **DAG-планировщик** | Операторы внутри функции автоматически распараллеливаются по зависимостям данных, независимые вентили естественно параллельны |
| Гибридный классическо-квантовый поток управления | **Единый синтаксис** | Квантовые и классические операции используют одинаковую форму `name: type = value` |

**YaoXiang 不是"添加量子支持"，而是发现自己的设计已经量子原生。**

> **Пояснение семантики**: В данном документе квантовые операции выражаются через **семантику владения** YaoXiang. На уровне компилятора обеспечивается неклонирование (безопасность владения). На уровне языка "потребление-создание" является **синтаксическим выражением передачи владения** — потребление = получение владения, возврат = передача владения. Базовая реализация может быть истинно обратимыми квантовыми вентилями (изменение квантового состояния на месте), а не действительно "созданием нового квантового состояния".

### Цели проектирования

1. **Ноль нового синтаксиса**: не вводить ключевые слова `quantum`, `circuit` и др., все квантовые возможности выражаются через существующие механизмы языка.
2. **Типовая безопасность**: компилятор гарантирует, что квантовые состояния не копируются и не используются неправомерно.
3. **Проверка топологических ограничений на этапе компиляции**: через обобщённые константные параметры `Qubit(T, N)` проверка допустимости двухкубитных вентилей относительно физической топологии на этапе компиляции.
4. **Прозрачная поддержка нескольких бэкендов**: один код для квантовых вычислений может компилироваться в QIR (универсальная экосистема) или QCIS (отечественный набор квантовых инструкций), переключение через параметры командной строки.
5. **Бесшовное гибридное классическое**: квантовые вычисления могут свободно вызывать классические функции, классический код также может оперировать квантовыми данными (через `ref` разделяемое, но с ограничениями владения).

## Предложение

### Основной дизайн

#### 1. Маппинг квантовой системы типов

**Базовые типы**:
```yaoxiang
Qubit: Type0 = primitive_qubit
Complex: Type0 = { re: Float, im: Float }
```
- `Qubit` является типом первого класса, следует правилам владения (Move, RAII).
- `Complex` используется для представления амплитуд, компилятор может выполнить встраивание и оптимизацию.

**Квантовые вентили как функции**:
```yaoxiang
# Встроенные сигнатуры функций
H: (Qubit) -> Qubit = builtin_hadamard
X: (Qubit) -> Qubit = builtin_pauli_x
Y: (Qubit) -> Qubit = builtin_pauli_y
Z: (Qubit) -> Qubit = builtin_pauli_z
CNOT: (control: Qubit, target: Qubit) -> { Qubit, Qubit } = builtin_cnot
```
- Все вентили потребляют входной qubit, возвращают новый qubit (или пару запутанных). Синтаксис возврата владения `q = H(q)` напрямую соответствует математической семантике.
- Многокубитные вентили возвращают структуры, результаты получаются через сопоставление с образцом или доступ к полям.

**Измерение**:
```yaoxiang
measure: (Qubit) -> Int = builtin_measure   # Потребляет qubit, возвращает классический бит
measure_all: (List(Qubit)) -> List(Int) = builtin_measure_all
```
- После измерения qubit потребляется (становится пустым), пользователь может переинициализировать через повторное использование пустого состояния.

**Инициализация**:
```yaoxiang
qubit: (Int) -> Qubit = builtin_qubit   # Инициализация базового состояния 0 или 1
```

#### 2. Запутывание и инкапсуляция непрозрачных типов

Пары запутанных кубитов инкапсулируются в непрозрачный тип, предоставляются только комбинированные операции, разделение запрещено:

```yaoxiang
# Встроенный непрозрачный тип
BellPair: Type0 = primitive_bell_pair

# Встроенные функции - только целостные операции
CNOT: (Qubit, Qubit) -> BellPair
measure_bell: (BellPair) -> { Int, Int }
split_bell: (BellPair) -> { Qubit, Qubit }  # Разделение пары запутанных кубитов (требует осторожного использования)
apply_cnot_to_bell: (BellPair, Qubit) -> BellPair
```

**Ключевой дизайн**:
- Нет доступа к полям, только целостные операции через встроенные функции
- `measure_bell(bp)` одновременно потребляет всю пару запутанных кубитов, возвращает классические биты
- Компилятор может отслеживать полный жизненный цикл пары запутанных кубитов

**Сравнение с Python/Qiskit**:
```
Python (Qiskit): Построение схемы во время выполнения, ошибки могут быть обнаружены только после отправки
YaoXiang:       Большинство логических ошибок перехватываются на этапе компиляции
```

**Оставшиеся 10%** (такие как физическая декогеренция, погрешности вентилей) являются проблемами оборудования, которые язык не может решить.

#### 3. Ограничения физической топологии

Квантовый чип является ограниченным топологическим графом, **не произвольные два кубита могут выполнять двухкубитные вентили**, они должны быть соседними. YaoXiang использует **обобщённые константные параметры** для обеспечения топологических ограничений на этапе компиляции.

**Определение типа топологии**:
```yaoxiang
# Топология как тип, содержит матрицу смежности
Topology: Type0 = primitive_topology

# Встроенные топологические константы
Linear8: Topology = topology(8)          # Линейные 8 бит: 0-1-2-3-4-5-6-7
Grid3x3: Topology = topology(3, 3)        # Сетка 3x3
Ring16: Topology = topology(16, ring)    # Кольцо из 16 бит
```

**Кубиты связывают топологию и позицию**:
```yaoxiang
# Qubit(T, N) - T это тип топологии, N это константный позиционный параметр
q0: Qubit(Grid3x3, 0)   # Grid3x3 топология, позиция (0,0)
q1: Qubit(Grid3x3, 1)   # Grid3x3 топология, позиция (0,1)
q2: Qubit(Grid3x3, 2)   # Grid3x3 топология, позиция (0,2)
q3: Qubit(Grid3x3, 3)   # Grid3x3 топология, позиция (1,0)
```

**Автоматические ограничения операций вентилей**:
```yaoxiang
# CNOT сигнатура типа с топологическим ограничением
CNOT: (T: Topology, I: Int, J: Int) -> (
    (Qubit(T, I), Qubit(T, J)) -> { Qubit(T, I), Qubit(T, J) }
) when adjacent(T, I, J)

# Проверка на этапе компиляции
CNOT(q0, q1)  # ✅ В Grid3x3 (0,0) и (0,1) соседние
CNOT(q0, q2)  # ❌ Ошибка компиляции: (0,0) и (0,2) не соседние
```

**`adjacent` ограничение на этапе компиляции**:
- `adjacent` — функция этапа компиляции, использующая матрицу смежности топологии для статической проверки
- При константных индексах — 100% проверка на этапе компиляции
- При динамических индексах — генерация кода проверки во время выполнения

**Виртуальное-to-физическое отображение**:
```yaoxiang
# Конкретная физическая позиция неизвестна на этапе компиляции? Использовать вывод типов
q = qubit(Grid3x3)  # Автоматическое распределение позиции 0, последующий вывод
```

#### 4. Владение и линейный поток квантовых состояний

Все квантовые операции следуют семантике Move, гарантируя, что qubit не копируется:
```yaoxiang
q = qubit(0)
q2 = q          # ❌ Ошибка компиляции: q уже перемещён, нельзя использовать повторно
q = H(q)        # ✅ Потребляет q, возвращает новый q
measure(q)      # ✅ Потребляет q, после чего q становится пустым
q = qubit(0)    # ✅ Повторное использование пустого состояния
```

#### 4. Автоматический параллелизм и DAG-планирование

В Standard или Full Runtime DAG-планировщик автоматически анализирует квантовые программы:
```yaoxiang
apply_two_qubit_gates: () -> {Qubit, Qubit} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    # Выше два оператора без зависимости по данным, DAG автоматически параллелизует
    CNOT(q1, q2)   # Зависит от q1 и q2, автоматическое ожидание
}
```
- Планировщик использует конфигурацию `num_workers` (количество физических квантовых процессоров) для реального параллелизма.
- Пользователю не нужно вручную упорядочивать вентили, достаточно описать поток данных.

#### 5. Гибридные классические вычисления

Классический и квантовый код полностью слиты:
```yaoxiang
grover_search: (target: Int) -> Int = () => {
    n = 4
    qubits = List(Qubit)()
    for i in 0..n {
        qubits.append(H(qubit(0)))
    }
    # Классический цикл смешан с квантовыми операциями
    oracle(qubits, target)   # oracle — последовательность квантовых вентилей
    qubits = diffusion(qubits)
    results = measure_all(qubits)
    return decode_result(results)   # Классическая постобработка
}
```
- В одной функции можно произвольно смешивать квантовые вентили и классический поток управления.
- Система владения гарантирует, что квантовые переменные не будут ошибочно скопированы в классических ветвлениях.

#### 6. Архитектура поддержки нескольких бэкендов

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   YaoXiang 源   │     │   类型检查      │     │   DAG 中间表示  │
│   (统一语法)     │────▶│   + 所有权分析   │────▶│   (数据流图)    │
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                          │
                                                          ▼
                          ┌─────────────────────────────────────────────┐
                          │           代码生成后端 (可插拔)              │
                          ├─────────────────┬───────────────────────────┤
                          │  QIR 后端       │  QCIS 后端                │
                          │  (通用生态)      │  (国产量子指令集)         │
                          ├─────────────────┼───────────────────────────┤
                          │  - 输出 .ll 文件 │  - 输出 .qcis 文本        │
                          │  - 适配多种 QPU  │  - 适配中科院/国盾硬件    │
                          └─────────────────┴───────────────────────────┘
```

- **Процесс компиляции**: унифицированный фронтенд → построение DAG → выбор бэкенда → генерация целевого кода.
- **QIR бэкенд**: маппит узлы DAG на квантовые intrinsic QIR, генерирует LLVM bitcode, может дополнительно использовать оптимизацию LLVM.
- **QCIS бэкенд**: сериализует DAG в инструкции QCIS (например, `H q0`), поддерживает прямую отправку в консоль управления квантовым чипом.

### Примеры

#### Подготовка и измерение состояния Белла
```yaoxiang
bell_measure: () -> {Int, Int} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    bell = CNOT(q1, q2)  # Возвращает непрозрачный тип BellPair
    result = measure_bell(bell)  # Одновременное измерение двух бит
    return result
}
```

#### Квантовая телепортация (упрощённая)
```yaoxiang
teleport: (msg: Qubit, bell: BellPair) -> Qubit = (msg, bell) => {
    # Разделение пары запутанных кубитов для получения двух независимых кубитов
    (alice_qubit, bob_qubit) = split_bell(bell)

    # Операции Алисы
    (msg, alice_qubit) = CNOT(msg, alice_qubit)
    msg = H(msg)
    a1 = measure(msg)
    a2 = measure(alice_qubit)

    # Передача классической информации (автоматически обрабатывается планировщиком)
    # Операции Боба
    if a2 == 1 { bob_qubit = X(bob_qubit) }
    if a1 == 1 { bob_qubit = Z(bob_qubit) }
    return bob_qubit
}
```

## Детальной дизайн

### Встроенные типы и определения функций

В модуле `compiler/builtins` добавить:
```rust
builtins.insert("Qubit", Ty::Primitive(Primitive::Qubit));
builtins.insert("Complex", Ty::Record(vec![
    ("re", Ty::Primitive(Primitive::Float)),
    ("im", Ty::Primitive(Primitive::Float)),
]));

// Квантовые вентили
for (name, sig) in GATES {
    builtins.insert(name, Ty::Function(vec![Ty::Qubit], Ty::Qubit));
}
builtins.insert("CNOT", Ty::Function(
    vec![Ty::Qubit, Ty::Qubit],
    Ty::Record(vec![
        ("q0", Ty::Qubit),
        ("q1", Ty::Qubit)
    ])
));
builtins.insert("measure", Ty::Function(vec![Ty::Qubit], Ty::Primitive(Primitive::Int)));
builtins.insert("qubit", Ty::Function(vec![Ty::Primitive(Primitive::Int)], Ty::Qubit));
```

### Специальная обработка Qubit в проверщике владения

- `Qubit` помечен как `!Copy` (по умолчанию Move), неявное копирование запрещено.
- Параметр функции измерения `measure` имеет тип `Qubit` (передача по значению), потребляет владение.
- В записях, возвращаемых многокубитными вентилями, поля являются `Qubit`, по-прежнему должны соблюдать правила владения.

### Оптимизация квантовых вентилей DAG-планировщиком

- Узлы квантовых вентилей рассматриваются как чистые функции (без побочных эффектов), планировщик может произвольно переупорядочивать независимые вентили.
- При выводе "последовательности квантовых инструкций" планировщик сохраняет зависимости данных и группирует параллельные вентили (применимо для многопроцессорных квантовых систем).
- Поддержка настройки `--target-num-qubits` и `--target-topology` для последующей раскладки и маршрутизации (будущее расширение).

### Детальной маппинг QIR бэкенда

| Операция YaoXiang | Инструкция QIR |
|-------------------|----------------|
| `H(q)` | `call void @__quantum__qis__h__body(%Qubit* %q)` |
| `CNOT(q1, q2)` | `call void @__quantum__qis__cnot__body(%Qubit* %q1, %Qubit* %q2)` |
| `measure(q)` | `%result = call i1 @__quantum__qis__mz__body(%Qubit* %q)` |
| `qubit(0)` | `%q = call %Qubit* @__quantum__rt__qubit_allocate()` |

QIR бэкенд использует `-O2` LLVM для дальнейшей оптимизации и выводит bitcode, совместимый со спецификацией QIR Alliance.

### Детальной маппинг QCIS бэкенда

| Операция YaoXiang | Инструкция QCIS |
|-------------------|-----------------|
| `H(q)` (q соответствует физическому биту 2) | `H 2` |
| `CNOT(q1,q2)` (q1→бит 0, q2→бит 1) | `CNOT 0 1` |
| `measure(q)` (бит 0) | `M 0` |
| `qubit(0)` инициализация | Подразумевается в первой используемой инструкции, дополнительная инструкция не требуется |

- Необходимо вести таблицу отображения виртуальных кубитов (переменных YaoXiang) на физические биты.
- Поддержка проверки топологических ограничений (будущая реализация).

### Генерация гибридного классического кода

- Классическая часть (циклы, условия, целочисленные операции) генерирует нативный код (x86/ARM) как обычно, взаимодействует с квантовым бэкендом через FFI или встроенные вызовы.
- В QIR бэкенде классическая часть может быть понижена до LLVM IR и смешанно скомпилирована с QIR.

### Влияние на систему типов

- Добавлены примитивные типы `Qubit` и `Complex`.
- `Qubit` автоматически имеет семантику Move, копирование запрещено.
- Сигнатуры функций квантовых вентилей должны быть зарегистрированы в системе типов.

### Обратная совместимость

- ✅ Полная обратная совместимость
- Добавлены встроенные типы и функции, не влияют на существующий код
- Квантовые возможности опциональны, без дополнительных накладных расходов при отключении

## Компромиссы

### Преимущества

- **Нет нового синтаксиса**: разработчикам нужно изучить только несколько встроенных функций для написания квантовых программ.
- **Типовая безопасность**: система владения автоматически предотвращает копирование кубитов, избегая распространённых ошибок квантового программирования.
- **Автоматический параллелизм**: DAG-планировщик бесплатно предоставляет параллелизм на уровне вентилей без дополнительной оптимизации компилятора.
- **Совместимость с экосистемой**: QIR бэкенд позволяет YaoXiang работать на нескольких облачных квантовых платформах; QCIS бэкенд обеспечивает независимость и контроль.
- **Гибридные возможности**: естественное слияние классических и квантовых вычислений подходит для написания сложных квантовых алгоритмов (классическое управление в Shor, Grover).

### Недостатки

- **Статическое количество кубитов**: текущий дизайн предполагает, что количество кубитов известно на этапе компиляции, динамическое распределение через `List(Qubit)`, но кучевое выделение `List` может добавить дополнительные накладные расходы (можно смягчить оптимизацией).
- **Повторное использование после измерения**: повторное использование пустого состояния позволяет переинициализировать кубит, но физические кубиты могут иметь время релаксации, требуется обработка runtime (в настоящее время ответственность пользователя).
- **Динамическое отображение топологии**: когда физическая топология становится известна только во время выполнения, проверка на этапе компиляции не работает, требуется генерация кода проверки во время выполнения (текущая версия поддерживает только статическую проверку).

## Альтернативные варианты

| Вариант | Почему не выбран |
|---------|------------------|
| Введение ключевого слова `quantum` и типа `circuit` | Добавление нового синтаксиса, высокая стоимость обучения, нарушение принципа простоты дизайна YaoXiang |
| Реализация квантовой поддержки только как библиотеки | Невозможно использовать гарантии компилятора для безопасности квантовых состояний, невозможна глубокая интеграция с DAG-планировщиком |
| Ожидание созревания квантового оборудования перед поддержкой | Упущение ключевого окна для проектирования языка квантового программирования |
| Повторное использование существующих квантовых фреймворков (например, Qiskit) | Квантовая семантика реализуется через библиотеку, невозможно получить гарантии безопасности системы типов и владения |
| Отдельное проектирование квантового подъязыка | Увеличение сложности языка, высокие затраты на поддержку |

## Стратегия реализации

### Разделение на этапы

| Этап | Время | Содержание |
|------|-------|------------|
| Phase 1 | 1 месяц | Базовые квантовые типы и встроенные функции: добавление типов `Qubit`, `Complex` в компилятор, реализация проверки типов встроенных функций, расширение проверщика владения |
| Phase 2 | 1 месяц | DAG-планировщик распознаёт квантовые вентили: модификация логики построения DAG, пометка квантовых вентилей как чистых функций, реализация параллельной группировки вывода вентилей |
| Phase 3 | 2 месяца | Прототип QIR бэкенда: реализация генератора кода из DAG в QIR, интеграция LLVM, подключение симулятора QIR для верификации |
| Phase 4 | 2 месяца | Прототип QCIS бэкенда: реализация транслятора из DAG в инструкции QCIS, проектирование таблицы отображения виртуальных-физических бит, подключение отечественной квантовой платформы для верификации |
| Phase 5 | 2 месяца | Усиление гибридного классического: обеспечение корректной генерации кода при пересечении классического потока управления и квантовых вентилей, поддержка `List(Qubit)`, добавление примеров программ |
| Phase 6 | 2 месяца | Оптимизация и документация: реализация базовой раскладки и маршрутизации, написание руководства пользователя и учебника по квантовому программированию, выпуск превью-версии |

### Риски

1. **Доступность квантового оборудования**: зависимость от доступности внешних квантовых симуляторов и реальных QPU.
   - **Смягчение**: приоритет подключения к открытым симуляторам (QIR runner, Qiskit Aer), реальные QPU как долгосрочная цель.

2. **Сложность реализации бэкендов**: спецификации QIR и QCIS могут изменяться.
   - **Смягчение**: абстракция интерфейса генерации кода, изоляция различий бэкендов для удобства последующей адаптации.

3. **Неопределённость производительности**: характеристики производительности квантовых программ отличаются от классических.
   - **Смягчение**: предоставление инструментов профилирования для понимания пользователем эффекта параллелизма на уровне вентилей.

## Открытые вопросы

- [x] **Топологические ограничения**: реализованы через обобщённые константные параметры `Qubit(Topology, N)`, проверка на этапе компиляции.
- [ ] **Динамический квантовый регистр**: как `List(Qubit)` отображается в QCIS бэкенде? Можно генерировать соответствующее количество физических бит, но требуется механизм распределения во время выполнения.
- [ ] **Смягчение ошибок**: предоставлять ли встроенные конструкции смягчения ошибок (например, динамическая развязка)? Можно сначала реализовать как библиотеку.
- [ ] **Взаимодействие с существующими квантовыми SDK**: можно ли импортировать модули QASM или QIR? В будущем можно рассмотреть FFI.
- [ ] **Автоматическая раскладка и маршрутизация**: когда количество виртуальных кубитов превышает количество физических кубитов, как автоматически отображать?
