# RFC-024：Модель параллелизма на основе 并作-блоков

> **Статус**: Принято
> **Автор**: Чэнь Сюй (晨煦)

> **Последнее обновление**: 2026-06-05

> **Ссылки**:

> - [RFC-008: Декoupling Runtime модели параллелизма и планировщика](./008-runtime-concurrency-model.md)

## Аннотация

В данном документе определяется новая модель параллелизма языка программирования YaoXiang: использование блока `spawn {}` в качестве единственного примитива параллелизма, управление зависимостями, синхронная блокировка вызывающей стороны. Заменяет старую схему параллелизма на основе аннотаций `@block`/`@eager`/`@auto`,特质 Send/Sync и анализа полной программы DAG.

**Основной дизайн — один примитив, одно правило**:

```
spawn { ... }        ← единственный примитив параллелизма
прямые подвыражения создают задачи  ← единственное правило
синхронная блокировка ожидания результата      ← единственное поведение
```

**Устранённая сложность**:
- ❌ Нет аннотаций `@block`/`@eager`/`@auto`
- ❌ Нет特质 Send/Sync
- ❌ Нет `Mutex`/`RwLock`/`Atomic`
- ❌ Нет `future`/неблокирующих дескрипторов
- ❌ Нет анализа полной программы DAG
- ❌ Нет окрашивания функций (async/await)

> **Ментальная модель пользователя**: Ваш обычный код выполняется последовательно. Когда вы хотите, чтобы несколько вещей происходили одновременно, поместите их в блок `spawn { ... }`. Никаких коллбэков, никакого `await`, никаких странных аннотаций.

## Мотивация

### Почему нужен этот дизайн?

Модели параллелизма текущих основных языков имеют очевидные недостатки:

| Язык | Модель параллелизма | Проблема |
|------|----------|------|
| Rust | async/await + tokio | Асинхронное заражение, окрашивание функций, крутая кривая обучения |
| Go | goroutine | Нет безопасности типов, трудно обнаружить гонки данных |
| Python | asyncio | Ограничения GIL, окрашивание функций |
| JavaScript | Promise/async | Ад коллбэков, окрашивание функций |

### Проблемы старого дизайна (RFC-001)

Трёхуровневая архитектура параллелизма (L1/L2/L3), предложенная в RFC-001, имеет следующие проблемы:

| Проблема | Описание |
|------|------|
| Сложная ментальная модель | Три уровня абстракции L1/L2/L3 увеличивают нагрузку на обучение |
| Избыточные аннотации | Аннотации `@block`/`@eager`/`@auto` загрязняют код |
| Высокая сложность анализа | Накладные расходы на время компиляции при анализе полной программы DAG |
| Сложные ограничения типов | 特质 Send/Sync увеличивают когнитивную нагрузку |
| Неконтролируемость | Автоматическое параллельное поведение трудно предсказать и отлаживать |

### Цели дизайна

1. **Простота**: только один примитив параллелизма (`spawn`), одно правило (прямые подвыражения создают задачи)
2. **Явность**: пользователь точно знает, где параллельно, где последовательно
3. **Безопасность**: правила владения естественно расширяются, без дополнительных ограничений типов
4. **Контролируемость**: нет неявного параллелизма, нет неожиданного параллельного поведения
5. **Синхронность**: вызывающая сторона синхронно блокируется, нет коллбэков и `await`

## Предложение

### 1. Сущность блока {}: вычислительная единица с управлением по зависимостям

В YaoXiang блок `{}` представляет собой **вычислительную единицу с управлением по зависимостям**.

| Атрибут | Описание |
|------|------|
| Управление по зависимостям | При выполнении блок проверяет готовность всех внутренних переменных; если все готовы — выполняется немедленно, иначе блокируется в ожидании |
| Время выполнения | Определяется зависимостями, не связано с "немедленным" или "отложенным" |
| Возвращаемое значение | Явно возвращается через `return`; без `return` по умолчанию возвращается `Void` |
| Единообразие синтаксиса | Семантика едина независимо от места появления: в теле функции, при инициализации переменной или после `spawn` |
| Изоляция области видимости | Переменные строго ограничены внутри `{}`, не проникают во внешнюю область видимости |

```yaoxiang
// Пример управления по зависимостям
x = compute_x()        // x готов
y = compute_y()        // y готов
result = {
    // Зависит от x и y; выполняется немедленно после готовности обоих
    return x + y
}
```

### 2. Семантика 并作-блока

`spawn { ... }` является **единственным примитивом параллелизма** в YaoXiang.

#### 2.1 Основные правила

- **Прямые подвыражения** блока spawn создают параллельные задачи
- Выражения внутри вложенных блоков `{}` не считаются независимыми задачами
- 并作-блок следует стандартным правилам возврата: необходимо явно вернуть значение через `return`; без `return` возвращается `Void`
- Весь 并作-блок синхронно блокируется до завершения всех задач и затем возвращает результат
- Никаких коллбэков, `await` или аннотаций

```yaoxiang
// Две задачи выполняются параллельно
(a, b) = spawn {
    t1 = fetch("url1")   // прямое подвыражение → параллельная задача 1
    t2 = fetch("url2")   // прямое подвыражение → параллельная задача 2
    return (t1, t2)      // явный возврат кортежа
}

// Вложенные {} не являются прямыми подвыражениями
result = spawn {
    x = {               // этот блок — одно прямое подвыражение → одна задача
        inner_work()    // не является прямым подвыражением spawn, не становится независимой задачей
    },
    process(x)          // прямое подвыражение → параллельная задача
    return process(x)
}
```

#### 2.2 Изоляция области видимости

并作-блок создаёт независимую область видимости; внутренние переменные не влияют на внешние:

```yaoxiang
x = 10
result = spawn {
    x = 20              // это локальный x внутри блока spawn
    compute(x)
}
// x всё ещё равен 10
```

#### 2.3 Правила владения

После того как переменная перемещается в 并作-блок, внешний код больше не может её использовать (семантика Move):

```yaoxiang
data = load_data()
result = spawn {
    process(data)       // владение data перемещается в блок spawn
}
// data здесь недоступна (перемещена)
```

Если необходимо разделить данные между несколькими задачами, используйте `ref`:

```yaoxiang
data = load_data()
shared = ref data       // компилятор автоматически выбирает Rc или Arc

result = spawn {
    process_a(shared),  // общедоступная ссылка
    process_b(shared)   // общедоступная ссылка
}
```

#### 2.4 Обработка ошибок

Ошибки внутри 并作-блока распространяются по следующим правилам:

1. Ожидание завершения всех задач (даже если некоторые уже завершились с ошибкой)
2. Распространение первой встреченной ошибки
3. Использование `?` для явного указания точки распространения ошибки

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // может завершиться с ошибкой
    fetch("url2")?      // может завершиться с ошибкой
}
// Если любая задача завершилась с ошибкой, весь блок spawn распространяет первую ошибку
```

#### 2.5 Типы ресурсов

Компилятор отслеживает использование типов ресурсов, обеспечивая безопасность параллелизма:

| Тип ресурса | Описание | Поведение компилятора |
|----------|------|-----------|
| `FilePath` | Путь к файловой системе | Операции с одним путём автоматически сериализуются |
| `HttpUrl` | HTTP endpoint | Операции с одним URL автоматически сериализуются |
| `DBUrl` | Подключение к БД | Операции с одним подключением автоматически сериализуются |
| `Console` | Стандартный вывод | Все операции Console автоматически сериализуются |

```yaoxiang
// Операции с одним файлом автоматически сериализуются
(a, b) = spawn {
    read_file("data.txt"),      // выполняется первым
    write_file("data.txt", x)   // ожидает завершения чтения
}
```

#### 2.6 并作 for: параллельный цикл по данным

```yaoxiang
// Параллельная обработка каждого элемента списка
results = spawn for item in items {
    result = process(item)
}
```

#### 2.7 Вложенный spawn

并作-блоки могут быть вложенными; внутренний spawn создаёт новую область параллелизма:

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

### 3. Разрыв со старым дизайном

| Старый дизайн (RFC-001) | Новый дизайн (RFC-024) |
|------------------|------------------|
| Автоматический анализ DAG всей программы | Анализ только внутри блока spawn |
| Аннотации `@block`/`@eager`/`@auto` | Без аннотаций, управление по зависимостям |
| 特质 Send/Sync | Не нужны; владение + ref обрабатываются автоматически |
| `future`/неблокирующие дескрипторы | Синхронная блокировка, без коллбэков |
| `Mutex`/`RwLock`/`Atomic` | `ref` автоматически выбирает Rc/Arc |
| Трёхуровневая ментальная модель L1/L2/L3 | Обычный код последователен, блок spawn параллелен |
| Окрашивание функций (async/await) | Без окрашивания функций |

### 4. Правила возврата

Правила возврата в YaoXiang унифицированы и однозначны:

| Запись | Возвращаемое значение | Описание |
|------|--------|------|
| `= expr` (без фигурных скобок) | Непосредственно возвращает `expr` | Выражение является значением |
| `= { ... }` (с фигурными скобками) | Обязателен `return`, иначе возвращается `Void` | Блок требует явного возврата |

```yaoxiang
// Без фигурных скобок: непосредственный возврат
add: (a: Int, b: Int) -> Int = a + b

// С фигурными скобками: обязателен return
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// С фигурными скобками, но без return: возвращает Void
log: (message: String) -> Void = {
    print(message)  // без return, возвращает Void
}
```

### 5. Ментальная модель пользователя

> **Ваш обычный код выполняется последовательно.**
>
> **Когда вы хотите, чтобы несколько вещей происходили одновременно, поместите их в блок `spawn { ... }`.**
>
> Каждое прямое подвыражение внутри блока начинает выполняться немедленно (параллельно), используйте `return` для явного возврата результата.
> Весь блок ожидает завершения всех задач и возвращает итоговый результат.
>
> **Никаких коллбэков, никакого `await`, никаких странных аннотаций.**

```yaoxiang
// Обычный код: последовательное выполнение
a = compute_a()         // выполняется первым
b = compute_b(a)        // зависит от a, выполняется после завершения a
c = compute_c(b)        // зависит от b, выполняется после завершения b

// Когда нужен параллелизм: используем spawn
(x, y, z) = spawn {
    fetch("url1"),      // параллельно
    fetch("url2"),      // параллельно
    fetch("url3")       // параллельно
}
// После ожидания всех задач продолжаем
process(x, y, z)
```

## Компромиссы

### Преимущества

1. **Простота**: только один примитив параллелизма (`spawn`), одно правило (прямые подвыражения создают задачи)
2. **Явность**: пользователь точно знает, где параллельно, где последовательно, нет неявного параллелизма
3. **Безопасность**: правила владения естественно расширяются, не нужны дополнительные ограничения типов вроде `Send`/`Sync`
4. **Контролируемость**: нет автоматического параллельного поведения, избегаются неожиданные проблемы параллелизма
5. **Синхронность**: вызывающая сторона синхронно блокируется, код легко понимать и отлаживать
6. **Без окрашивания функций**:不存在 async/await проблемы окрашивания функций
7. **Эффективная компиляция**: Анализ DAG только внутри блока spawn, время компиляции контролируемо

### Недостатки

1. **Требуется явный spawn**: нет автоматического параллелизма, пользователь должен вручную помечать точки параллелизма
2. **Анализ DAG внутри блока spawn**: компилятору необходимо выполнять анализ зависимостей внутри блока spawn
3. **Несовместимость со старым кодом**: код, использующий старый режим RFC-001, требует миграции

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| Автоматический DAG всей программы (RFC-001) | Высокая сложность, большое время компиляции, неконтролируемое поведение |
| async/await | Окрашивание функций, крутая кривая обучения, плохая читаемость кода |
| goroutine | Нет безопасности типов, трудно обнаружить гонки данных |
| Модель Actor | Сложность передачи сообщений, трудности при отладке |
| CSP (Go channel) | Нет безопасности типов, трудно обнаружить взаимоблокировки |

## Стратегия реализации

### Анализ на этапе компиляции

1. **Построение DAG**: анализ зависимостей внутри блока spawn
2. **Топологическая сортировка**: определение порядка выполнения внутри блока spawn
3. **Распознавание параллелизма**: идентификация поддеревьев без зависимостей внутри блока spawn
4. **Анализ побега**: `ref` → Rc или Arc
5. **Обнаружение конфликтов ресурсов**: обнаружение потенциальных конфликтов типов ресурсов

### Выполнение на этапе прогона

См. архитектуру Runtime в [RFC-008](./008-runtime-concurrency-model.md):

- **Embedded Runtime**: без поддержки spawn, немедленное выполнение
- **Standard Runtime**: поддержка блоков spawn, параллелизм внутри блока spawn
- **Full Runtime**: Standard + балансировка нагрузки WorkStealer

### Зависимости
