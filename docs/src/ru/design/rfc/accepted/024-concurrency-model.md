# RFC-024: Модель параллелизма на основе spawn-блоков

> **Ссылки**:
> - [Спецификация модели параллелизма](/reference/language-spec/concurrency.md)
> - [RFC-008: Отвязанный дизайн модели параллелизма Runtime и планировщика](./008-runtime-concurrency-model.md)
> - [RFC-009: Дизайн модели владения](./009-ownership-model.md)
> - [RFC-010: Унифицированный синтаксис типов](./010-unified-type-syntax.md)

## Резюме

Этот документ определяет новую модель параллелизма языка программирования YaoXiang: блок `spawn {}` как единственный примитив параллелизма, исполнение, управляемое зависимостями, синхронная блокировка вызывающей стороны. Заменяет старую схему, основанную на аннотациях `@block`/`@eager`/`@auto`, trait `Send`/`Sync` и анализе DAG всей программы.

**Ключевой дизайн — один примитив, одно правило**:

```
spawn { ... }        ← единственный примитив параллелизма
прямые подвыражения создают задачи  ← единственное правило
синхронная блокировка в ожидании результата      ← единственное поведение
```

**Устранённая сложность**:
- ❌ Без аннотаций `@block`/`@eager`/`@auto`
- ❌ Без trait `Send`/`Sync`
- ❌ Без `Mutex`/`RwLock`/`Atomic`
- ❌ Без `future`/неблокирующих дескрипторов
- ❌ Без анализа DAG всей программы
- ❌ Без окрашивания функций (async/await)

> **Ментальная модель пользователя**: обычный код, который вы пишете, выполняется последовательно. Когда вы хотите сделать несколько дел одновременно, поместите их в блок `spawn { ... }`. Никаких колбэков, никакого `await`, никаких странных аннотаций.

## Мотивация

### Зачем нужен этот дизайн?

Современные модели параллелизма в основных языках имеют явные недостатки:

| Язык | Модель параллелизма | Проблемы |
|------|----------|------|
| Rust | async/await + tokio | асинхронное заражение, окрашивание функций, крутая кривая обучения |
| Go | goroutine | нет типобезопасности, гонки данных сложно обнаружить |
| Python | asyncio | ограничение GIL, окрашивание функций |
| JavaScript | Promise/async | ад колбэков, окрашивание функций |

### Проблемы старого дизайна (RFC-001)

Трёхуровневая архитектура параллелизма (L1/L2/L3), предложенная в RFC-001, имеет следующие проблемы:

| Проблема | Описание |
|------|------|
| Сложная ментальная модель | трёхуровневая абстракция L1/L2/L3 увеличивает нагрузку на обучение |
| Избыточные аннотации | аннотации `@block`/`@eager`/`@auto` делают код шумным |
| Высокая сложность анализа | большие накладные расходы на время компиляции при анализе DAG всей программы |
| Сложные ограничения типов | trait `Send`/`Sync` увеличивают когнитивную нагрузку |
| Неконтролируемость | автоматическое параллельное поведение сложно предсказать и отлаживать |

### Цели дизайна

1. **Простота**: только один примитив параллелизма (`spawn`) и одно правило (прямые подвыражения создают задачи)
2. **Явность**: пользователь точно знает, где параллелизм, а где последовательность
3. **Безопасность**: правила владения естественно расширяются, без дополнительных ограничений типов
4. **Контролируемость**: нет неявного параллелизма, нет неожиданного параллельного поведения
5. **Синхронность**: вызывающая сторона синхронно блокируется, без колбэков и `await`

## Предложение

### 1. Сущность блока {}: вычислительная единица, управляемая зависимостями

В YaoXiang `{}` — это **вычислительная единица, управляемая зависимостями**.

| Свойство | Описание |
|------|------|
| Управляемый зависимостями | блок проверяет готовность всех внутренних переменных при выполнении; если все готовы — немедленно выполняется, иначе — блокируется в ожидании |
| Время выполнения | определяется зависимостями, не зависит от «немедленного» или «отложенного» |
| Возвращаемое значение | явное возвращение значения с помощью `return`; без `return` по умолчанию возвращается `Void` |
| Единый синтаксис | семантика согласована, вне зависимости от того, находится ли блок в теле функции, инициализации переменной или после `spawn` |
| Изоляция области видимости | переменные строго ограничены внутренней частью `{}`, не проникают во внешнюю область видимости |

```yaoxiang
// Пример с управлением по зависимостям
x = compute_x()        // x готов
y = compute_y()        // y готов
result = {
    // зависит от x и y, выполняется сразу после готовности обоих
    return x + y
}
```

### 2. Семантика spawn-блока

`spawn { ... }` — **единственный примитив параллелизма** в YaoXiang.

#### 2.1 Ключевые правила

- Прямые подвыражения spawn-блока создают параллельные задачи
- Выражения внутри вложенных `{}` не считаются отдельными задачами
- Spawn-блок следует стандартным правилам возврата: должен явно возвращать значение с помощью `return`, без `return` возвращается `Void`
- Весь spawn-блок синхронно блокируется, ожидает завершения всех задач, затем возвращает
- Нет колбэков, `await` или аннотаций

```yaoxiang
// Две задачи выполняются параллельно
(a, b) = spawn {
    t1 = fetch("url1")   // прямое подвыражение → параллельная задача 1
    t2 = fetch("url2")   // прямое подвыражение → параллельная задача 2
    return (t1, t2)      // явный возврат кортежа
}

// Вложенный {} не является прямым подвыражением
result = spawn {
    x = {               // весь этот блок — одно прямое подвыражение → одна задача
        inner_work()    // не является прямым подвыражением spawn, не станет отдельной задачей
    },
    process(x)          // прямое подвыражение → параллельная задача
    return process(x)
}

#### 2.2 Изоляция области видимости

Spawn-блок создаёт независимую область видимости, внутренние переменные не влияют на внешние:

```yaoxiang
x = 10
result = spawn {
    x = 20              // это локальный x внутри spawn-блока
    compute(x)
}
// x по-прежнему 10
```

#### 2.3 Правила владения

После того как переменная входит в spawn-блок, внешний код больше не может её использовать (семантика Move):

```yaoxiang
data = load_data()
result = spawn {
    process(data)       // владение data перемещается в spawn-блок
}
// data здесь недоступна (перемещена)
```

Если нужно разделить данные между несколькими задачами, используйте `ref`:

```yaoxiang
data = load_data()
shared = ref data       // компилятор автоматически выбирает Rc или Arc

result = spawn {
    process_a(shared),  // разделяемая ссылка
    process_b(shared)   // разделяемая ссылка
}
```

#### 2.4 Обработка ошибок

Распространение ошибок внутри spawn-блока следует следующим правилам:

1. Ожидание завершения всех задач (даже если некоторые завершились неудачно)
2. Распространение первой обнаруженной ошибки
3. Явная маркировка точек распространения ошибок с помощью `?`

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // может завершиться неудачно
    fetch("url2")?      // может завершиться неудачно
}
// если любая задача завершится неудачно, весь spawn-блок распространяет первую ошибку
```

#### 2.5 Типы ресурсов

Компилятор отслеживает использование типов ресурсов, обеспечивая безопасность параллелизма:

| Тип ресурса | Описание | Поведение компилятора |
|----------|------|-----------|
| `FilePath` | путь в файловой системе | операции с тем же путём автоматически сериализуются |
| `HttpUrl` | HTTP-эндпоинт | операции с тем же URL автоматически сериализуются |
| `DBUrl` | соединение с базой данных | операции с тем же соединением автоматически сериализуются |
| `Console` | стандартный вывод | все операции Console автоматически сериализуются |

```yaoxiang
// Операции с одним и тем же файлом автоматически сериализуются
(a, b) = spawn {
    read_file("data.txt"),      // выполняется первой
    write_file("data.txt", x)   // ожидание завершения чтения
}
```

#### 2.6 spawn for: параллельный цикл по данным

```yaoxiang
// Параллельная обработка каждого элемента списка
results = spawn for item in items {
    result = process(item)
}
```

#### 2.7 Вложенный spawn

Spawn-блоки могут быть вложенными; внутренний spawn создаёт новую область параллелизма:

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

### 3. Разрыв со старым дизайном

| Старый дизайн (RFC-001) | Новый дизайн (RFC-024) |
|------------------|------------------|
| Автоматический анализ DAG всей программы | Анализ только внутри spawn-блока |
| Аннотации `@block`/`@eager`/`@auto` | Без аннотаций, управляемый зависимостями |
| Trait `Send`/`Sync` | Не требуется, владение + ref обрабатываются автоматически |
| `future`/неблокирующие дескрипторы | Синхронная блокировка, без колбэков |
| `Mutex`/`RwLock`/`Atomic` | `ref` автоматически выбирает Rc/Arc |
| Трёхуровневая ментальная модель L1/L2/L3 | Обычный код последовательный, spawn-блок параллельный |
| Окрашивание функций (async/await) | Без окрашивания функций |

### 4. Правила возврата

Правила возврата в YaoXiang унифицированы и ясны:

| Запись | Возвращаемое значение | Описание |
|------|--------|------|
| `= expr` (без фигурных скобок) | прямой возврат `expr` | выражение есть значение |
| `= { ... }` (с фигурными скобками) | необходим `return`, иначе возвращается `Void` | блок требует явного возврата |

```yaoxiang
// Без фигурных скобок: прямой возврат
add: (a: Int, b: Int) -> Int = a + b

// С фигурными скобками: необходим return
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// С фигурными скобками, но без return: возвращается Void
log: (message: String) -> Void = {
    print(message)  // без return, возвращается Void
}
```

### 5. Ментальная модель пользователя

> **Обычный код, который вы пишете, выполняется последовательно.**
>
> **Когда вы хотите сделать несколько дел одновременно, поместите их в блок `spawn { ... }`.**
>
> Каждое прямое подвыражение в блоке запускается немедленно (параллельно); для явного возврата результата используется `return`.
> Весь блок ждёт завершения всех дел, затем выдаёт окончательный результат.
>
> **Без колбэков, без `await`, без странных аннотаций.**

```yaoxiang
// Обычный код: последовательное выполнение
a = compute_a()         // выполняется первой
b = compute_b(a)        // зависит от a, выполняется после завершения a
c = compute_c(b)        // зависит от b, выполняется после завершения b

// При необходимости параллелизма: используйте spawn
(x, y, z) = spawn {
    fetch("url1"),      // параллельно
    fetch("url2"),      // параллельно
    fetch("url3")       // параллельно
}
// продолжение после завершения всех
process(x, y, z)
```

## Компромиссы

### Преимущества

1. **Простота**: только один примитив параллелизма (`spawn`) и одно правило (прямые подвыражения создают задачи)
2. **Явность**: пользователь точно знает, где параллелизм, а где последовательность; нет неявного параллелизма
3. **Безопасность**: правила владения естественно расширяются, без дополнительных ограничений типов вроде `Send`/`Sync`
4. **Контролируемость**: нет автоматического параллелизма, что исключает неожиданные проблемы с конкурентностью
5. **Синхронность**: вызывающая сторона синхронно блокируется, код проще понимать и отлаживать
6. **Без окрашивания функций**: нет проблемы окрашивания функций, свойственной async/await
7. **Эффективная компиляция**: анализ DAG ограничен рамками spawn-блока, время компиляции контролируемо

### Недостатки

1. **Необходимость явного spawn**: нет автоматического параллелизма, пользователь должен вручную отмечать точки параллелизма
2. **Анализ DAG внутри spawn-блока**: компилятор должен выполнять анализ зависимостей внутри spawn-блока
3. **Несовместимость со старым кодом**: код, использующий старую модель RFC-001, нуждается в миграции

## Альтернативы

| Вариант | Почему не выбран |
|------|--------------|
| Автоматический анализ DAG всей программы (RFC-001) | Высокая сложность, долгая компиляция, неконтролируемое поведение |
| async/await | Окрашивание функций, крутая кривая обучения, плохая читаемость кода |
| goroutine | Нет типобезопасности, гонки данных сложно обнаружить |
| Модель акторов | Сложная передача сообщений, трудная отладка |
| CSP (каналы Go) | Нет типобезопасности, взаимные блокировки сложно обнаружить |

## Стратегия реализации

### Анализ на этапе компиляции

1. **Построение DAG**: анализ зависимостей внутри spawn-блока
2. **Топологическая сортировка**: определение порядка выполнения внутри spawn-блока
3. **Идентификация параллелизма**: распознавание поддеревьев без зависимостей внутри spawn-блока
4. **Анализ побега**: `ref` → Rc или Arc
5. **Обнаружение конфликтов ресурсов**: обнаружение потенциальных конфликтов типов ресурсов

### Организация модуля

Код, связанный со spawn, единообразно размещается в `frontend/core/spawn/`:

```
frontend/core/spawn/
├── mod.rs           # вход в модуль spawn
├── placement.rs     # проверка допустимости размещения spawn
└── analysis.rs      # идентификация задач, анализ зависимостей, обнаружение конфликтов ресурсов (требуется для этапа 4 RFC-018)
```

> **Пояснение по миграции** (2026-06-11): существующий `frontend/core/typecheck/passes/spawn_placement.rs` будет перенесён в `frontend/core/spawn/placement.rs`. Объявление модуля `spawn_placement` в каталоге `typecheck/passes/` должно быть синхронно удалено. Эта миграция обусловлена RFC-018 (LLVM AOT-компилятор) — LLVM-бэкенду нужны результаты анализа spawn, и анализ spawn как отдельный общий модуль фронтенда более разумен, чем встраивание в typecheck.

### Выполнение в runtime

Ссылаясь на архитектуру Runtime из [RFC-008](./008-runtime-concurrency-model.md):

- **Embedded Runtime**: без поддержки spawn, немедленное выполнение
- **Standard Runtime**: поддержка spawn-блоков, параллелизм внутри spawn-блока
- **Full Runtime**: Standard + WorkStealer для балансировки нагрузки

### Зависимости

- RFC-008 (Архитектура Runtime) → завершён
- RFC-009 (Модель владения) → завершён
- RFC-010 (Унифицированный синтаксис типов) → завершён

| Создание задач | Прямые подвыражения | Явность, отсутствие неоднозначности | 2026-06-05 |
