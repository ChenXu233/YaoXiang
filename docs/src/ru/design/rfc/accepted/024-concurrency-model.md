# RFC-024: Семантика конкурентного runtime на основе spawn

> **Настоящий документ определяет семантику поведения runtime конструкции `spawn`.**
> Синтаксическая ортогональность, рефакторинг AST/IR и расширения системы типов см. в [RFC-032](./032-spawn-unified-expression.md).
>
> Два RFC совместно определяют `spawn` — 024 отвечает на вопрос «что делать», 032 — «как представить».

> **Ссылки**:
> - [Спецификация модели конкурентности](/reference/language-spec/concurrency.md)
> - [RFC-008: Разделение модели конкурентности runtime и планировщика](./008-runtime-concurrency-model.md)

> - [RFC-032: Унифицированный модификатор выражений spawn — рефакторинг AST/IR](./032-spawn-unified-expression.md)

## Резюме

Настоящий документ определяет **семантику поведения runtime** конструкции `spawn` в языке программирования YaoXiang: `spawn <expr>` — единственная примитива параллелизма, которая может модифицировать произвольное выражение, а вызывающая сторона синхронно блокируется. Форма выражения определяет гранулярность декомпозиции задач; runtime планирует по модели GMP — задачи без зависимостей помещаются в рабочую очередь, worker'ы «перехватывают» их.

**Ключевой дизайн — одна примитива, один набор правил**:

```
spawn <expr>               ← единственная примитива параллелизма
декомпозиция задач определяется формой выражения  ← единственное правило
синхронная блокировка в ожидании результата        ← единственное поведение
```

**Устранённая сложность**:
- ❌ Нет аннотаций `@block`/`@eager`/`@auto`
- ❌ Нет trait'ов `Send`/`Sync`
- ❌ Нет `Mutex`/`RwLock`/`Atomic`
- ❌ Нет `future`/неблокирующих дескрипторов
- ❌ Нет полнопрограммного DAG-анализа
- ❌ Нет окрашивания функций (async/await)

> **Ментальная модель пользователя**: ваш обычный код выполняется последовательно. Когда вы хотите, чтобы несколько вещей выполнялись одновременно, поместите их в `spawn <expr>`. Никаких обратных вызовов, никакого `await`, никаких странных аннотаций.

## Мотивация

### Зачем нужен этот дизайн?

Существующие модели конкурентности в主流 языках имеют явные недостатки:

| Язык | Модель конкурентности | Проблема |
|------|----------|------|
| Rust | async/await + tokio | Заражение асинхронностью, окрашивание функций, крутая кривая обучения |
| Go | goroutine | Нет типобезопасности, гонки данных трудно обнаружить |
| Python | asyncio | Ограничение GIL, окрашивание функций |
| JavaScript | Promise/async | Ад обратных вызовов, окрашивание функций |

### Проблемы старого дизайна (RFC-001)

Трёхуровневая архитектура конкурентности (L1/L2/L3), предложенная в RFC-001, имеет следующие проблемы:

| Проблема | Описание |
|------|------|
| Сложная ментальная модель | Три уровня абстракции L1/L2/L3 увеличивают нагрузку на обучение |
| Избыточные аннотации | Аннотации `@block`/`@eager`/`@auto` делают код шумным |
| Высокая сложность анализа | Полнопрограммный DAG-анализ даёт большие накладные расходы на компиляцию |
| Сложные ограничения типов | Trait'ы `Send`/`Sync` увеличивают когнитивную нагрузку |
| Непредсказуемость | Автоматическое конкурентное поведение трудно предсказать и отлаживать |

### Цели дизайна

1. **Простота**: только одна примитива параллелизма (`spawn`), может модифицировать произвольное выражение
2. **Явность**: пользователь точно знает, где параллельно, а где последовательно
3. **Безопасность**: правила владения естественно расширяются, без дополнительных ограничений типов
4. **Управляемость**: нет неявной конкурентности, нет непреднамеренного параллельного поведения
5. **Синхронность**: вызывающая сторона синхронно блокируется, нет обратных вызовов и `await`

## Предложение

### 1. Природа блока `{}`: вычислительная единица, управляемая зависимостями

В YaoXiang `{}` — это **вычислительная единица, управляемая зависимостями**.

| Свойство | Описание |
|------|------|
| Управляемость зависимостями | При выполнении блок проверяет, все ли внутренние переменные готовы; если готовы — выполняется немедленно, иначе блокируется в ожидании |
| Момент выполнения | Определяется зависимостями, не зависит от «немедленного» или «отложенного» |
| Возвращаемое значение | Используйте `return` для явного возврата значения; без `return` по умолчанию возвращается `Void` |
| Единый синтаксис | Семантика одинакова независимо от того, встречается ли в теле функции, инициализации переменной или после `spawn` |
| Изоляция области видимости | Переменные строго ограничены внутренней частью `{}`, не проникают во внешнюю область видимости |

```yaoxiang
// Пример, управляемый зависимостями
x = compute_x()        // x готово
y = compute_y()        // y готово
result = {
    // Зависит от x и y, выполняется немедленно после их готовности
    return x + y
}
```

### 2. Семантика выражения spawn

`spawn <expr>` — **единственная примитива параллелизма** в YaoXiang. Она может модифицировать произвольное выражение; форма выражения определяет гранулярность декомпозиции задач.

#### 2.1 Правила создания задач

| Форма выражения | Декомпозиция задач | Семантика синхронизации |
|-----------|---------|---------|
| `spawn { a, b, c }` | Непосредственные подвыражения → N независимых задач | Ожидание завершения всех задач |
| `spawn for x in items { body }` | Каждая итерация → 1 задача | Ожидание завершения всех итераций |
| `spawn while cond { body }` | Каждая итерация цикла → 1 задача (между итерациями — по условию) | Ожидание, пока условие не станет false |
| `spawn if c { a } else { b }` | Условие c вычисляется последовательно, выбранная ветвь целиком → 1 задача | Ожидание завершения выбранной ветви |
| `spawn call(x)` | Сам вызов → 1 задача | Ожидание завершения вызова |
| `spawn expr` (произвольное выражение) | Само выражение → 1 задача | Ожидание завершения выражения |

> **Мотивация дизайна**: почему spawn может модифицировать произвольное выражение? См. [RFC-032 §Ключевой дизайн](./032-spawn-unified-expression.md).
>
> **Ортогональность потока управления**: семантические различия между `spawn <expr>` (spawn спереди) и `<expr> spawn { body }` (spawn сзади) см. в [RFC-032 §Ортогональность потока управления](./032-spawn-unified-expression.md) (ключевое определение). Поведение во время выполнения всех обратных комбинаций (`for ... spawn { }` / `while ... spawn { }` / `if ... spawn { }`) — распространение ошибок, типы ресурсов, правила вложенности — наследуют правила из §2.4 / §2.5 / §2.6 настоящего документа.

```yaoxiang
// spawn-блок: непосредственные подвыражения параллельны
(a, b) = spawn {
    t1 = fetch("url1")   // непосредственное подвыражение → параллельная задача 1
    t2 = fetch("url2")   // непосредственное подвыражение → параллельная задача 2
    return (t1, t2)      // явный возврат кортежа
}

// spawn for: каждая итерация параллельна
results = spawn for item in items {
    process(item)        // каждая итерация → независимая задача
}

// spawn while: каждая итерация цикла параллельна
spawn while has_next() {
    step()               // каждая итерация цикла → независимая задача
}

// spawn if: выбранная ветвь целиком как задача
result = spawn if cond {
    branch_a()
} else {
    branch_b()
}
```

#### 2.2 Изоляция области видимости

Выражение spawn создаёт независимую область видимости; внутренние переменные не влияют на внешнюю:

```yaoxiang
x = 10
result = spawn {
    x = 20              // это локальный x внутри выражения spawn
    compute(x)
}
// x по-прежнему равен 10

result = spawn for item in items {
    item = item + 1     // итерационно-локальный item, независимая копия в каждой итерации
    process(item)
}
// Внешний item не затрагивается
```

**Переменные итерации** (`x` в for) — независимая копия в каждом раунде, автоматически уничтожаются по завершении итерации.

#### 2.3 Правила владения

После входа переменной в выражение spawn её нельзя использовать снаружи (семантика Move):

```yaoxiang
data = load_data()
result = spawn {
    process(data)       // владение data перемещено в выражение spawn
}
// data здесь недоступна (уже перемещена)
```

Если нужно разделить между несколькими задачами, используйте `ref`:

```yaoxiang
data = load_data()
shared = ref data       // компилятор автоматически выбирает Rc или Arc

result = spawn {
    process_a(shared),  // разделяемая ссылка
    process_b(shared)   // разделяемая ссылка
}
```

**Совместное использование между итерациями**: захват с помощью `ref` во внешнюю область, итерации совместно используют одну и ту же ссылку.

#### 2.4 Правила распространения ошибок

##### `spawn { a, b, c }` (блок)

1. Ожидание завершения всех задач (даже если некоторые уже завершились ошибкой)
2. Распространение первой встреченной ошибки
3. Используйте `?` для явной отметки точки распространения ошибки

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // может завершиться ошибкой
    fetch("url2")?      // может завершиться ошибкой
}
// Если какая-либо задача завершилась ошибкой, всё выражение spawn распространяет первую ошибку
```

##### `spawn for x in items { body? }`

- Ожидание завершения всех итераций перед возвратом первой ошибки
- Оставшиеся итерации **продолжают выполняться** после неудачной итерации (не отменяются)
- Используйте `?` для явной отметки точки распространения ошибки

```yaoxiang
results = spawn for item in items {
    process(item)?      // ошибка любой итерации → ожидание завершения всех → распространение первой ошибки
}
```

##### `spawn while cond { body? }`

Наследует собственную семантику ошибок while:

- step использует `?` для распространения ошибки → весь spawn while завершается ошибкой, следующий раунд не входит
- step не распространяет ошибку (ошибка проглатывается) → переход к следующему раунду итерации

```yaoxiang
spawn while has_next() {
    item = next()       // без распространения ошибки: даже при сбое переходит к следующему раунду
    process(item)
}
```

##### `spawn if c { a } else { b }`

- Условие c **вычисляется последовательно**
- Ошибка при вычислении c → общая ошибка
- Ошибка внутри выбранной ветви → общая ошибка

```yaoxiang
result = spawn if cond()? {  // cond вычисляется последовательно, ошибка → общая ошибка
    fetch_a()?
} else {
    fetch_b()?
}
```

#### 2.5 Правила для типов ресурсов

Компилятор отслеживает использование типов ресурсов для обеспечения безопасности конкурентности:

| Тип ресурса | Описание | Поведение компилятора |
|----------|------|-----------|
| `FilePath` | Путь файловой системы | Операции с одним путём автоматически сериализуются |
| `HttpUrl` | Конечная точка HTTP | Операции с одним URL автоматически сериализуются |
| `DBUrl` | Соединение с базой данных | Операции с одним соединением автоматически сериализуются |
| `Console` | Стандартный вывод | Все операции с Console автоматически сериализуются |

##### Внутри блока `spawn { ... }`

```yaoxiang
// Операции с одним файлом автоматически сериализуются
(a, b) = spawn {
    read_file("data.txt"),      // выполняется первой
    write_file("data.txt", x)   // ожидает завершения чтения
}
```

##### `spawn for ... { ... }` один ресурс между итерациями

Когда все итерации оперируют одним и тем же типом ресурса, компилятор **автоматически понижает до последовательного** (spawn вырождается в последовательный for, без ошибки):

```yaoxiang
// Все итерации пишут в один путь к файлу → автоматическое понижение до последовательного
results = spawn for item in items {
    write_file("data.txt", item)
}
// Компилятор автоматически сериализует все итерации
```

> **Обоснование дизайна**: ключевое слово spawn по-прежнему выражает намерение параллелизма; при конфликте ресурсов компилятор автоматически понижает, что лучше соответствует принципу наименьшего удивления, чем прямой отказ.

##### Захват `&mut` в `spawn while ... { ... }`

**Ошибка компиляции**: `spawn while` не позволяет захватывать внешние переменные типа `&mut`:

```yaoxiang
iter = make_iter()
spawn while iter.has_next() {       // ошибка компиляции
    item = iter.next()              // iter имеет тип &mut, разделяемая изменяемость между итерациями = гонка данных
}
```

> **Без повторного введения trait'а `Sync`**: согласуется с обязательством RFC-024 «без Send/Sync». Требует от пользователя перейти на `ref` или не-spawn форму записи.

##### Один ресурс в обеих ветвях `spawn if c { ... } else { ... }`

**Допустимо без предупреждения**: условия if взаимоисключающие, выполняется максимум одна ветвь, конфликта параллелизма нет:

```yaoxiang
result = spawn if use_cache {
    load_from_cache(key)            // ветвь 1: чтение кэша
} else {
    fetch(key)                      // ветвь 2: чтение URL
}
```

#### 2.6 Вложенный spawn

Выражения spawn могут быть вложенными, при этом внутренний уровень создаёт **независимый домен конкурентности**:

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

**Семантика вложенности**:
- Внутренний spawn — независимый домен конкурентности (независимая очередь задач, независимое распространение ошибок)
- Внутренние ошибки независимо распространяются наружу (внешняя задача получает ошибку при ожидании завершения внутренней)
- Правила типов ресурсов внутреннего уровня отслеживаются независимо (не объединяются с внешним уровнем)

```yaoxiang
// spawn for вложен в spawn while
results = spawn for x in items {
    inner = spawn while has_more(x) {
        step(x)
    }
    process(inner)
}
```

### 3. Разрыв со старым дизайном

| Старый дизайн (RFC-001) | Новый дизайн (RFC-024 + RFC-032) |
|------------------|---------------------------|
| Полнопрограммный автоматический DAG-анализ | Анализ только внутри выражений spawn |
| Аннотации `@block`/`@eager`/`@auto` | Без аннотаций, управляемость зависимостями |
| Trait'ы `Send`/`Sync` | Не нужны, владение + ref справляются автоматически |
| `future`/неблокирующие дескрипторы | Синхронная блокировка, без обратных вызовов |
| `Mutex`/`RwLock`/`Atomic` | `ref` автоматически выбирает Rc/Arc |
| Трёхуровневая ментальная модель L1/L2/L3 | Обычный код последовательно, выражения spawn параллельно |
| Окрашивание функций (async/await) | Без окрашивания функций |
| `spawn` модифицирует только блок `{}` | `spawn` модифицирует произвольное выражение (см. RFC-032) |

### 4. Правила возврата

Правила возврата в YaoXiang едины и ясны:

| Запись | Возвращаемое значение | Описание |
|------|--------|------|
| `= expr` (без фигурных скобок) | Прямой возврат `expr` | Выражение есть значение |
| `= { ... }` (с фигурными скобками) | Требуется `return`, иначе возвращается `Void` | Блок требует явного возврата |

```yaoxiang
// Без фигурных скобок: прямой возврат
add: (a: Int, b: Int) -> Int = a + b

// С фигурными скобками: требуется return
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// С фигурными скобками, но без return: возвращается Void
log: (message: String) -> Void = {
    print(message)  // без return, возвращается Void
}
```

### 5. Ментальная модель пользователя

> **Ваш обычный код выполняется последовательно.**
>
> **Когда вы хотите, чтобы несколько вещей выполнялись одновременно, поместите их в `spawn <expr>`.**
>
> Форма выражения определяет, как декомпозируются задачи: каждое непосредственное подвыражение в блоке параллельно; каждая итерация for параллельна; выбранная ветвь if — одна задача.
>
> **Всё выражение spawn синхронно блокируется, ожидая завершения всех задач.**
>
> **Никаких обратных вызовов, никакого `await`, никаких странных аннотаций.**

```yaoxiang
// Обычный код: последовательное выполнение
a = compute_a()         // сначала выполняется
b = compute_b(a)        // зависит от a, выполняется после завершения a
c = compute_c(b)        // зависит от b, выполняется после завершения b

// Когда нужен параллелизм: используйте spawn
(x, y, z) = spawn {
    fetch("url1"),      // параллельно
    fetch("url2"),      // параллельно
    fetch("url3")       // параллельно
}
// Ожидание завершения всех, затем продолжение
process(x, y, z)

// Параллелизм данных: spawn for
results = spawn for item in items {
    process(item)
}
```

## Компромиссы

### Преимущества

1. **Простота**: только одна примитива параллелизма (`spawn`), может модифицировать произвольное выражение
2. **Явность**: пользователь точно знает, где параллельно, а где последовательно, нет неявной конкурентности
3. **Безопасность**: правила владения естественно расширяются, без дополнительных ограничений типов вроде `Send`/`Sync`
4. **Управляемость**: нет автоматического параллельного поведения, что позволяет избежать непреднамеренных проблем конкурентности
5. **Синхронность**: вызывающая сторона синхронно блокируется, код легко понимать и отлаживать
6. **Без окрашивания функций**: нет проблемы окрашивания функций из async/await
7. **Эффективная компиляция**: DAG-анализ только внутри выражений spawn, время компиляции контролируемо
8. **Ортогональность**: spawn естественно сочетается с любыми конструкциями потока управления (подробнее в RFC-032)

### Недостатки

1. **Требуется явный spawn**: нельзя сделать параллельно автоматически, пользователь должен вручную отмечать точки параллелизма
2. **DAG-анализ внутри выражения spawn**: компилятор должен выполнять анализ зависимостей внутри выражений spawn
3. **Несовместимость со старым кодом**: код, использующий старый шаблон RFC-001, требует миграции

## Альтернативы

| Альтернатива | Почему не выбрана |
|------|--------------|
| Полнопрограммный автоматический DAG (RFC-001) | Высокая сложность, длительная компиляция, непредсказуемое поведение |
| async/await | Окрашивание функций, крутая кривая обучения, плохая читаемость кода |
| goroutine | Нет типобезопасности, гонки данных трудно обнаружить |
| Модель акторов | Сложный обмен сообщениями, трудная отладка |
| CSP (каналы Go) | Нет типобезопасности, deadlock'и трудно обнаружить |
| `spawn` модифицирует только блок `{}` | Нарушает ортогональность, делает `spawn for` особым случаем (см. RFC-032) |

## Стратегия реализации

### Анализ на этапе компиляции

1. **Распознавание формы выражения**: по форме выражения после spawn определяется декомпозиция задач (подробнее в RFC-032 §DAG-анализ)
2. **Построение DAG**: анализ отношений зависимостей внутри выражения spawn
3. **Топологическая сортировка**: определение порядка выполнения внутри выражения spawn
4. **Идентификация параллелизма**: обнаружение поддеревьев без зависимостей внутри выражения spawn
5. **Escape-анализ**: `ref` → Rc или Arc
6. **Обнаружение конфликтов ресурсов**: обнаружение потенциальных конфликтов типов ресурсов

### Организация модулей

Код, связанный со spawn, единообразно размещается в `frontend/core/spawn/`:

```
frontend/core/spawn/
├── mod.rs           # точка входа модуля spawn
├── placement.rs     # проверка допустимости места появления spawn
└── analysis.rs      # идентификация задач, анализ зависимостей, обнаружение конфликтов ресурсов
```

> **Заметка о миграции** (2026-06-11): существующий `frontend/core/typecheck/passes/spawn_placement.rs` будет перенесён в `frontend/core/spawn/placement.rs`. Объявление модуля `spawn_placement` в каталоге `typecheck/passes/` необходимо синхронно удалить.

### Выполнение в runtime

Со ссылкой на архитектуру Runtime из [RFC-008](./008-runtime-concurrency-model.md):

- **Embedded Runtime**: без поддержки spawn, немедленное выполнение
- **Standard Runtime**: поддержка выражений spawn
- **Full Runtime**: Standard + балансировка нагрузки WorkStealer

### Зависимости

- RFC-008 (архитектура Runtime) → завершён
- RFC-009 (модель владения) → завершён
- RFC-010 (унифицированный синтаксис типов) → завершён
- RFC-011 (система дженериков) → завершён
- RFC-032 (рефакторинг AST/IR) → совместно с настоящим документом определяет spawn

| future/дескрипторы | Удалены | Синхронная блокировка проще | 2026-06-05 |

| Вложенный spawn | Внутренний уровень — независимый домен конкурентности | Независимые очередь задач, ошибки, ресурсы | 2026-07-04 |
