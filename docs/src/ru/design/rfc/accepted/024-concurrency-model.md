# RFC-024：Модель параллелизма на основе блоков spawn

> **Статус**: Принято
> **Автор**: Чэнь Сюй
> **Дата создания**: 2026-06-05
> **Последнее обновление**: 2026-06-05

> **Ссылки**:
> - [Спецификация модели параллелизма](/reference/language-spec/concurrency.md)
> - [RFC-008: Отделение модели параллелизма Runtime от планировщика](./008-runtime-concurrency-model.md)
> - [RFC-009: Модель владения](./009-ownership-model.md)
> - [RFC-010: Унифицированный синтаксис типов](./010-unified-type-syntax.md)

## Аннотация

В данном документе определяется новая модель параллелизма языка программирования YaoXiang: использование блока `spawn {}` в качестве единственного примитива параллелизма, зависимое выполнение, синхронная блокировка вызывающей стороны. Заменяет старую схему параллелизма на основе аннотаций `@block`/`@eager`/`@auto`, трейтов `Send`/`Sync` и анализа DAG всей программы.

**Основной дизайн — один примитив, одно правило**:

```
spawn { ... }        ← единственный примитив параллелизма
Прямое присваивание дочерним элементам создаёт задачи    ← единственное правило
Синхронная блокировка ожидания результата      ← единственное поведение
```

**Устранённая сложность**:
- ❌ Нет аннотаций `@block`/`@eager`/`@auto`
- ❌ Нет трейтов `Send`/`Sync`
- ❌ Нет `Mutex`/`RwLock`/`Atomic`
- ❌ Нет `future`/неблокирующих дескрипторов
- ❌ Нет анализа DAG всей программы
- ❌ Нет окрашивания функций (async/await)

> **Ментальная модель пользователя**: Ваш обычный код выполняется последовательно. Когда вы хотите, чтобы несколько вещей происходили одновременно, поместите их в блок `spawn { ... }`. Никаких колбэков, никакого `await`, никаких странных аннотаций.

## Мотивация

### Зачем нужен этот дизайн?

Текущие модели параллелизма основных языков имеют очевидные недостатки:

| Язык | Модель параллелизма | Проблема |
|------|---------------------|----------|
| Rust | async/await + tokio | Асинхронное заражение, окрашивание функций, крутая кривая обучения |
| Go | goroutine | Отсутствие типобезопасности, трудность обнаружения гонок данных |
| Python | asyncio | Ограничение GIL, окрашивание функций |
| JavaScript | Promise/async | Ада колбэков, окрашивание функций |

### Проблемы старого дизайна (RFC-001)

Трёхуровневая архитектура параллелизма (L1/L2/L3), предложенная в RFC-001, имеет следующие проблемы:

| Проблема | Описание |
|----------|----------|
| Сложная ментальная модель | Три уровня абстракции L1/L2/L3 увеличивают нагрузку на обучение |
| Избыточность аннотаций | Аннотации `@block`/`@eager`/`@auto` загрязняют код |
| Высокая сложность анализа | Накладные расходы на время компиляции из-за анализа DAG всей программы |
| Сложные ограничения типов | Трейты `Send`/`Sync` увеличивают когнитивную нагрузку |
| Неконтролируемость | Автоматическое параллельное поведение трудно предсказать и отлаживать |

### Цели дизайна

1. **Простота**: только один примитив параллелизма (`spawn`), одно правило (прямое присваивание дочерним элементам создаёт задачи)
2. **Явность**: пользователь точно знает, где параллелизм, а где последовательность
3. **Безопасность**: правила владения естественно расширяются, без дополнительных ограничений типов
4. **Контролируемость**: нет неявного параллелизма, нет неожиданного параллельного поведения
5. **Синхронность**: вызывающая сторона синхронно блокируется, без колбэков и `await`

## Предложение

### 1. Сущность блока {}: вычислительная единица с управлением по зависимостям

В YaoXiang `{}` — это **вычислительная единица с управлением по зависимостям**.

| Атрибут | Описание |
|---------|----------|
| Управление по зависимостям | При выполнении блок проверяет готовность всех внутренних переменных; если все готовы — выполняется немедленно, иначе блокируется в ожидании |
| Время выполнения | Определяется зависимостями, не связано с "немедленным" или "отложенным" |
| Возвращаемое значение | Явно возвращается через `return`; без `return` по умолчанию возвращается `Void` |
| Единый синтаксис | Семантика одинакова независимо от того,出现在函数体、变量初始化还是 `spawn` 后 |
| Изоляция области видимости | Переменные строго ограничены внутри `{}`, не проникают во внешнюю область |

```yaoxiang
// Пример управления по зависимостям
x = compute_x()        // x готов
y = compute_y()        // y готов
result = {
    // Зависит от x и y; выполняется сразу после готовности обоих
    return x + y
}
```

### 2. Семантика блока spawn

`spawn { ... }` — это **единственный примитив параллелизма** в YaoXiang.

#### 2.1 Основные правила

- **Прямые дочерние присваивания** блока spawn создают параллельные задачи
- Присваивания внутри вложенных `{}` не считаются отдельными задачами
- Весь блок spawn синхронно блокируется, ожидая завершения всех задач перед возвратом результата
- Никаких колбэков, `await` или аннотаций

```yaoxiang
// Две задачи выполняются параллельно
(a, b) = spawn {
    fetch("url1"),      // Задача 1
    fetch("url2")       // Задача 2
}
// Продолжаем после завершения обоих
```

#### 2.2 Изоляция области видимости

Блок spawn создаёт независимую область видимости; внутренние переменные не влияют на внешние:

```yaoxiang
x = 10
result = spawn {
    x = 20              // Это локальный x внутри блока spawn
    compute(x)
}
// x по-прежнему равен 10
```

#### 2.3 Правила владения

После того как переменная входит в блок spawn, она больше не может использоваться извне (семантика Move):

```yaoxiang
data = load_data()
result = spawn {
    process(data)       // Владение data перемещается в блок spawn
}
// data здесь недоступна (перемещена)
```

Если необходимо разделить данные между несколькими задачами, используйте `ref`:

```yaoxiang
data = load_data()
shared = ref data       // Компилятор автоматически выбирает Rc или Arc

result = spawn {
    process_a(shared),  // Общая ссылка
    process_b(shared)   // Общая ссылка
}
```

#### 2.4 Обработка ошибок

Распространение ошибок внутри блока spawn следует следующим правилам:

1. Ожидать завершения всех задач (даже если некоторые уже завершились с ошибкой)
2. Распространять первую встреченную ошибку
3. Использовать `?` для явного обозначения точки распространения ошибки

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // Может завершиться ошибкой
    fetch("url2")?      // Может завершиться ошибкой
}
// Если любая задача завершилась ошибкой, весь блок spawn распространяет первую ошибку
```

#### 2.5 Типы ресурсов

Компилятор отслеживает использование типов ресурсов, обеспечивая безопасность параллелизма:

| Тип ресурса | Описание | Поведение компилятора |
|--------------|----------|----------------------|
| `FilePath` | Путь к файловой системе | Операции с одинаковым путём автоматически сериализуются |
| `HttpUrl` | HTTP endpoint | Операции с одинаковым URL автоматически сериализуются |
| `DBUrl` | Подключение к БД | Операции с одинаковым подключением автоматически сериализуются |
| `Console` | Стандартный вывод | Все операции с Console автоматически сериализуются |

```yaoxiang
// Операции с одним файлом автоматически сериализуются
(a, b) = spawn {
    read_file("data.txt"),      // Выполняется первым
    write_file("data.txt", x)   // Ожидает завершения чтения
}
```

#### 2.6 spawn for: параллельный цикл по данным

```yaoxiang
// Параллельная обработка каждого элемента списка
results = spawn for item in items {
    result = process(item)
}
```

#### 2.7 Вложенные spawn

Блоки spawn могут быть вложенными; внутренний spawn создаёт новую область параллелизма:

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

### 3. Разрыв со старым дизайном

| Старый дизайн (RFC-001) | Новый дизайн (RFC-024) |
|-------------------------|------------------------|
| Автоматический анализ DAG всей программы | Анализ только внутри блока spawn |
| Аннотации `@block`/`@eager`/`@auto` | Без аннотаций, управление по зависимостям |
| Трейты `Send`/`Sync` | Не нужны; владение + ref обрабатываются автоматически |
| `future`/неблокирующие дескрипторы | Синхронная блокировка, без колбэков |
| `Mutex`/`RwLock`/`Atomic` | `ref` автоматически выбирает Rc/Arc |
| Трёхуровневая ментальная модель L1/L2/L3 | Обычный код последователен, блок spawn параллелен |
| Окрашивание функций (async/await) | Без окрашивания функций |

### 4. Правила возврата

Правила возврата в YaoXiang унифицированы и однозначны:

| Запись | Возвращаемое значение | Описание |
|--------|---------------------|----------|
| `= expr` (без фигурных скобок) | Непосредственно возвращает `expr` | Выражение является значением |
| `= { ... }` (с фигурными скобками) | Обязательно использовать `return`, иначе возвращается `Void` | Блок требует явного возврата |

```yaoxiang
// Без фигурных скобок: непосредственный возврат
add: (a: Int, b: Int) -> Int = a + b

// С фигурными скобками: обязателен return
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// С фигурными скобками, но без return: возвращает Void
log: (message: String) -> Void = {
    print(message)  // Без return, возвращает Void
}
```

### 5. Ментальная модель пользователя

> **Ваш обычный код выполняется последовательно.**
>
> **Когда вы хотите, чтобы несколько вещей происходили одновременно, поместите их в блок `spawn { ... }`.**
>
> Каждое прямое присваивание внутри блока начинается немедленно (параллельно), а нужный вам результат автоматически ожидается.
> Весь блок ждёт завершения всех задач, а затем возвращает вам окончательный результат.
>
> **Никаких колбэков, никакого `await`, никаких странных аннотаций.**

```yaoxiang
// Обычный код: последовательное выполнение
a = compute_a()         // Выполняется первым
b = compute_b(a)        // Зависит от a; выполняется после готовности a
c = compute_c(b)        // Зависит от b; выполняется после готовности b

// Когда нужен параллелизм: используем spawn
(x, y, z) = spawn {
    fetch("url1"),      // Параллельно
    fetch("url2"),      // Параллельно
    fetch("url3")       // Параллельно
}
// Ждём завершения всех перед продолжением
process(x, y, z)
```

## Компромиссы

### Преимущества

1. **Простота**: только один примитив параллелизма (`spawn`), одно правило (прямое присваивание дочерним элементам создаёт задачи)
2. **Явность**: пользователь точно знает, где параллелизм, а где последовательность; нет неявного параллелизма
3. **Безопасность**: правила владения естественно расширяются; не нужны дополнительные ограничения типов вроде `Send`/`Sync`
4. **Контролируемость**: нет автоматического параллельного поведения; избегаются неожиданные проблемы параллелизма
5. **Синхронность**: вызывающая сторона синхронно блокируется; код легко понять и отладить
6. **Без окрашивания функций**: отсутствуют проблемы окрашивания функций async/await
7. **Эффективная компиляция**: анализ DAG только внутри блоков spawn; время компиляции контролируемо

### Недостатки

1. **Требуется явный spawn**: нет автоматического параллелизма; пользователь должен вручную отмечать точки параллелизма
2. **Анализ DAG внутри блока spawn**: компилятору необходимо выполнять анализ зависимостей внутри блока spawn
3. **Несовместимость со старым кодом**: код, использующий старый паттерн RFC-001, требует миграции

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Автоматический DAG всей программы (RFC-001) | Высокая сложность, долгое время компиляции, неконтролируемое поведение |
| async/await | Окрашивание функций, крутая кривая обучения, плохая читаемость кода |
| goroutine | Отсутствие типобезопасности, трудность обнаружения гонок данных |
| Модель Actor | Сложность передачи сообщений, трудность отладки |
| CSP (каналы Go) | Отсутствие типобезопасности, трудность обнаружения deadlock |

## Стратегия реализации

### Анализ на этапе компиляции

1. **Построение DAG**: анализ зависимостей внутри блока spawn
2. **Топологическая сортировка**: определение порядка выполнения внутри блока spawn
3. **Распознавание параллелизма**: идентификация поддеревьев без зависимостей внутри блока spawn
4. **Анализ побега**: `ref` → Rc или Arc
5. **Обнаружение конфликтов ресурсов**: обнаружение потенциальных конфликтов типов ресурсов

### Выполнение на этапе runtime

См. архитектуру Runtime из [RFC-008](./008-runtime-concurrency-model.md):

- **Embedded Runtime**: без поддержки spawn, синхронное выполнение
- **Standard Runtime**: поддержка блоков spawn, параллелизм внутри блоков spawn
- **Full Runtime**: Standard + балансировка нагрузки WorkStealer

### Зависимости

- RFC-008 (Архитектура Runtime) → Завершено

- RFC-010 (Унифицированный синтаксис типов) → Завершено
