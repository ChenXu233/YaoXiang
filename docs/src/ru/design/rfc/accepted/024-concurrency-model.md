# RFC-024: Семантика среды выполнения параллелизма на основе spawn

> **Данный документ определяет семантику поведения среды выполнения `spawn`.**
> Синтаксическая ортогональность, рефакторинг AST/IR, расширения системы типов см. в [RFC-032](./032-spawn-unified-expression.md).
>
> Два RFC совместно определяют `spawn` — 024 отвечает на вопрос «что делать», 032 — «как представить».

> **Ссылки**:
> - [Спецификация модели параллелизма](/reference/language-spec/concurrency.md)
> - [RFC-008: Отделение модели параллелизма Runtime от планировщика](./008-runtime-concurrency-model.md)
> - [RFC-009: Проект модели владения](./009-ownership-model.md)
> - [RFC-010: Унифицированный синтаксис типов](./010-unified-type-syntax.md)
> - [RFC-032: Унифицированная модификация выражений spawn — рефакторинг AST/IR](./032-spawn-unified-expression.md)

## Резюме

Данный документ определяет **семантику поведения среды выполнения** `spawn` в языке программирования YaoXiang: `spawn <expr>` — единственный примитив параллелизма, способный модифицировать любое выражение, при этом вызывающая сторона блокируется синхронно. Форма выражения определяет гранулярность декомпозиции задач, а среда выполнения планирует их по модели GMP — задачи без зависимостей помещаются в рабочую очередь, и воркеры конкурируют за их выполнение.

**Основной проект — один примитив, один набор правил**:

```
spawn <expr>               ← единственный примитив параллелизма
декомпозиция задач определяется формой выражения  ← единственное правило
синхронная блокировка в ожидании результата         ← единственное поведение
```

**Устранённая сложность**:
- ❌ Нет аннотаций `@block`/`@eager`/`@auto`
- ❌ Нет трейтов `Send`/`Sync`
- ❌ Нет `Mutex`/`RwLock`/`Atomic`
- ❌ Нет `future`/неблокирующих дескрипторов
- ❌ Нет анализа DAG по всей программе
- ❌ Нет окрашивания функций (async/await)

> **Ментальная модель пользователя**: ваш обычный код выполняется последовательно. Когда вы хотите, чтобы несколько вещей выполнялись одновременно, поместите их в `spawn <expr>`. Нет колбэков, нет `await`, нет странных аннотаций.

## Мотивация

### Почему необходим этот проект?

Существующие модели параллелизма в主流 языках имеют явные недостатки:

| Язык | Модель параллелизма | Проблема |
|------|---------------------|----------|
| Rust | async/await + tokio | Асинхронное заражение, окрашивание функций, крутая кривая обучения |
| Go | горутины | Нет типобезопасности, сложно обнаружить гонки данных |
| Python | asyncio | Ограничения GIL, окрашивание функций |
| JavaScript | Promise/async | Ад колбэков, окрашивание функций |

### Проблемы старого проекта (RFC-001)

Трёхуровневая архитектура параллелизма (L1/L2/L3), предложенная в RFC-001, имеет следующие проблемы:

| Проблема | Описание |
|----------|----------|
| Сложная ментальная модель | Три уровня абстракции L1/L2/L3 увеличивают нагрузку на обучение |
| Избыточность аннотаций | Аннотации `@block`/`@eager`/`@auto` делают код шумным |
| Высокая сложность анализа | Большие затраты времени компиляции на анализ DAG всей программы |
| Сложные ограничения типов | Трейты `Send`/`Sync` увеличивают когнитивную нагрузку |
| Неуправляемость | Автоматическое параллельное поведение сложно предсказать и отлаживать |

### Цели проектирования

1. **Простота**: только один примитив параллелизма (`spawn`), модифицирующий любое выражение
2. **Явность**: пользователь точно знает, где параллельно, а где последовательно
3. **Безопасность**: правила владения естественно расширяются, без дополнительных ограничений типов
4. **Управляемость**: нет неявного параллелизма, нет непреднамеренного параллельного поведения
5. **Синхронность**: вызывающая сторона блокируется синхронно, без колбэков и `await`

## Предложение

### 1. Сущность блока `{}`: вычислительная единица, управляемая зависимостями

В YaoXiang `{}` — это **вычислительная единица, управляемая зависимостями**.

| Атрибут | Описание |
|---------|----------|
| Управление зависимостями | При выполнении блок проверяет готовность всех внутренних переменных; если все готовы — выполняется немедленно, иначе блокируется в ожидании |
| Момент выполнения | Определяется зависимостями, не связан с «немедленным» или «отложенным» |
| Возвращаемое значение | Используйте `return` для явного возврата значения; без `return` по умолчанию возвращается `Void` |
| Синтаксическое единство | Семантика одинакова независимо от того, где находится: в теле функции, при инициализации переменной или после `spawn` |
| Изоляция области видимости | Переменные строго ограничены внутренней частью `{}`, не проникают во внешнюю область видимости |

```yaoxiang
// Пример управления через зависимости
x = compute_x()        // x готово
y = compute_y()        // y готово
result = {
    // зависит от x и y; после готовности обоих — немедленное выполнение
    return x + y
}
```

### 2. Семантика выражения spawn

`spawn <expr>` — **единственный примитив параллелизма** в YaoXiang. Может модифицировать любое выражение; форма выражения определяет гранулярность декомпозиции задач.

#### 2.1 Правила создания задач

| Форма выражения | Декомпозиция задач | Семантика синхронизации |
|-----------------|--------------------|-------------------------|
| `spawn { a, b, c }` | Прямые подвыражения → N независимых задач | Ожидание завершения всех задач |
| `spawn for x in items { body }` | Каждая итерация → 1 задача | Ожидание завершения всех итераций |
| `spawn while cond { body }` | Каждый цикл итерации → 1 задача (управление условием между циклами) | Ожидание, пока условие не станет false |
| `spawn if c { a } else { b }` | Условие `c` вычисляется последовательно, выбранная ветвь целиком → 1 задача | Ожидание завершения выбранной ветви |
| `spawn call(x)` | Сам вызов → 1 задача | Ожидание завершения вызова |
| `spawn expr` (любое выражение) | Само выражение → 1 задача | Ожидание завершения выражения |

> **Мотивация проектирования**: почему `spawn` может модифицировать любое выражение? Подробности см. в [RFC-032 §Основной проект](./032-spawn-unified-expression.md).
>
> **Ортогональность потока управления**: семантические различия между `spawn <expr>` (spawn спереди) и `<expr> spawn { body }` (spawn сзади) подробно описаны в [RFC-032 §Ортогональность потока управления](./032-spawn-unified-expression.md) (основное определение). Поведение во время выполнения всех обратных комбинаций (`for ... spawn { }` / `while ... spawn { }` / `if ... spawn { }`) — распространение ошибок, типы ресурсов, правила вложенности — наследует правила §2.4 / §2.5 / §2.6 данного документа.

```yaoxiang
// блок spawn: прямые подвыражения выполняются параллельно
(a, b) = spawn {
    t1 = fetch("url1")   // прямое подвыражение → параллельная задача 1
    t2 = fetch("url2")   // прямое подвыражение → параллельная задача 2
    return (t1, t2)      // явный возврат кортежа
}

// spawn for: каждая итерация параллельна
results = spawn for item in items {
    process(item)        // каждая итерация → независимая задача
}

// spawn while: каждый цикл итерации параллелен
spawn while has_next() {
    step()               // каждый цикл итерации → независимая задача
}

// spawn if: выбранная ветвь целиком как задача
result = spawn if cond {
    branch_a()
} else {
    branch_b()
}
```

#### 2.2 Изоляция области видимости

Выражение spawn создаёт независимую область видимости; внутренние переменные не влияют на внешнюю:

```yaoxiang
x = 10
result = spawn {
    x = 20              // это локальная x внутри выражения spawn
    compute(x)
}
// x по-прежнему равна 10

result = spawn for item in items {
    item = item + 1     // итерационно-локальный item, независимая копия для каждой итерации
    process(item)
}
// внешний item не затрагивается
```

**Итерационные переменные** (`x` в `for`) — независимая копия для каждой итерации, автоматически уничтожаются по завершении итерации.

#### 2.3 Правила владения

После того как переменная входит в выражение spawn, её нельзя использовать снаружи (семантика перемещения):

```yaoxiang
data = load_data()
result = spawn {
    process(data)       // владение data перемещено в выражение spawn
}
// data здесь недоступна (перемещена)
```

Если необходимо разделить между несколькими задачами, используйте `ref`:

```yaoxiang
data = load_data()
shared = ref data       // компилятор автоматически выбирает Rc или Arc

result = spawn {
    process_a(shared),  // общая ссылка
    process_b(shared)   // общая ссылка
}
```

**Совместное использование между итерациями**: используйте `ref` для захвата во внешнюю область; итерации совместно используют одну и ту же ссылку.

#### 2.4 Правила распространения ошибок

##### `spawn { a, b, c }` (блок)

1. Ожидание завершения всех задач (даже если некоторые завершились с ошибкой)
2. Распространение первой встреченной ошибки
3. Используйте `?` для явной отметки точки распространения ошибки

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // может завершиться с ошибкой
    fetch("url2")?      // может завершиться с ошибкой
}
// при сбое любой задачи всё выражение spawn распространяет первую ошибку
```

##### `spawn for x in items { body? }`

- Ожидание завершения всех итераций, затем возврат первой ошибки
- После неудачной итерации оставшиеся итерации **продолжают выполняться** (не отменяются)
- Используйте `?` для явной отметки точки распространения ошибки

```yaoxiang
results = spawn for item in items {
    process(item)?      // сбой любой итерации → ожидание завершения всех → распространение первой ошибки
}
```

##### `spawn while cond { body? }`

Наследует семантику ошибок самого while:

- если `step` использует `?` для распространения ошибки → весь `spawn while` завершается с ошибкой, следующий цикл не начинается
- если `step` не распространяет ошибку (ошибка поглощается) → переход к следующей итерации

```yaoxiang
spawn while has_next() {
    item = next()       // без распространения ошибки — сбой также переходит к следующему циклу
    process(item)
}
```

##### `spawn if c { a } else { b }`

- Условие `c` вычисляется **последовательно**
- Ошибка вычисления `c` → общая ошибка
- Ошибка в выбранной ветви → общая ошибка

```yaoxiang
result = spawn if cond()? {  // cond вычисляется последовательно, сбой → общая ошибка
    fetch_a()?
} else {
    fetch_b()?
}
```

#### 2.5 Правила типов ресурсов

Компилятор отслеживает использование типов ресурсов, обеспечивая безопасность параллелизма:

| Тип ресурса | Описание | Поведение компилятора |
|-------------|----------|------------------------|
| `FilePath` | Путь файловой системы | Операции с одним путём автоматически сериализуются |
| `HttpUrl` | Конечная точка HTTP | Операции с одним URL автоматически сериализуются |
| `DBUrl` | Соединение с БД | Операции с одним соединением автоматически сериализуются |
| `Console` | Стандартный вывод | Все операции Console автоматически сериализуются |

##### Внутри блока `spawn { ... }`

```yaoxiang
// операции с одним и тем же файлом автоматически сериализуются
(a, b) = spawn {
    read_file("data.txt"),      // выполняется сначала
    write_file("data.txt", x)   // ожидает завершения чтения
}
```

##### `spawn for ... { ... }` — общий ресурс между итерациями

Когда все итерации работают с одним и тем же типом ресурса, компилятор **автоматически понижает до последовательного** (spawn деградирует до последовательного `for`, без ошибки):

```yaoxiang
// все итерации пишут в один и тот же файл → автоматически сериализуются
results = spawn for item in items {
    write_file("data.txt", item)
}
// компилятор автоматически сериализует все итерации
```

> **Обоснование проектирования**: ключевое слово `spawn` по-прежнему выражает намерение параллелизма; при конфликте ресурсов компилятор автоматически понижает уровень, что лучше соответствует принципу наименьшего удивления, чем прямой отказ.

##### `spawn while ... { ... }` — захват `&mut`

**Ошибка компиляции**: `spawn while` не разрешает захват внешних переменных типа `&mut`:

```yaoxiang
iter = make_iter()
spawn while iter.has_next() {       // ошибка компиляции
    item = iter.next()              // iter имеет тип &mut, общий изменяемый доступ между итерациями = гонка данных
}
```

> **Не возвращаем трейт `Sync`**: согласуется с обещанием RFC-024 «нет Send/Sync». Требуем от пользователя использовать `ref` или запись без `spawn`.

##### `spawn if c { ... } else { ... }` — общий ресурс в обеих ветвях

**Допустимо без предупреждений**: условия `if` взаимоисключающие, выполняется максимум одна ветвь, конфликта параллелизма нет:

```yaoxiang
result = spawn if use_cache {
    load_from_cache(key)            // ветвь 1: чтение из кеша
} else {
    fetch(key)                      // ветвь 2: чтение по URL
}
```

#### 2.6 Вложенный spawn

Выражения `spawn` могут быть вложенными; внутренний уровень создаёт **независимую область параллелизма**:

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

**Семантика вложенности**:
- Внутренний `spawn` — независимая область параллелизма (независимая очередь задач, независимое распространение ошибок)
- Внутренние ошибки независимо распространяются наружу (внешняя задача получает ошибку при ожидании завершения внутренней)
- Правила типов ресурсов внутреннего уровня отслеживаются независимо (не объединяются с внешним уровнем для проверки)

```yaoxiang
// spawn for вложен в spawn while
results = spawn for x in items {
    inner = spawn while has_more(x) {
        step(x)
    }
    process(inner)
}
```

### 3. Разрыв со старым проектом

| Старый проект (RFC-001) | Новый проект (RFC-024 + RFC-032) |
|--------------------------|----------------------------------|
| Автоматический анализ DAG по всей программе | Анализ только внутри выражений `spawn` |
| Аннотации `@block`/`@eager`/`@auto` | Нет аннотаций, управление через зависимости |
| Трейты `Send`/`Sync` | Не нужны, владение + `ref` обрабатываются автоматически |
| `future`/неблокирующие дескрипторы | Синхронная блокировка, без колбэков |
| `Mutex`/`RwLock`/`Atomic` | `ref` автоматически выбирает Rc/Arc |
| Трёхуровневая ментальная модель L1/L2/L3 | Обычный код последовательный, выражения `spawn` параллельны |
| Окрашивание функций (async/await) | Нет окрашивания функций |
| `spawn` модифицирует только блок `{}` | `spawn` модифицирует любое выражение (см. RFC-032) |

### 4. Правила возврата

Правила возврата в YaoXiang единообразны и ясны:

| Запись | Возвращаемое значение | Описание |
|--------|------------------------|----------|
| `= expr` (без фигурных скобок) | Прямой возврат `expr` | Выражение есть значение |
| `= { ... }` (с фигурными скобками) | Необходим `return`, иначе возвращается `Void` | Блок требует явного возврата |

```yaoxiang
// без фигурных скобок: прямой возврат
add: (a: Int, b: Int) -> Int = a + b

// с фигурными скобками: необходим return
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// с фигурными скобками, но без return: возвращается Void
log: (message: String) -> Void = {
    print(message)  // без return, возвращается Void
}
```

### 5. Ментальная модель пользователя

> **Обычный код, который вы пишете, выполняется последовательно.**
>
> **Когда вы хотите, чтобы несколько вещей выполнялось одновременно, поместите их в `spawn <expr>`.**
>
> Форма выражения определяет, как декомпозировать задачи: каждое прямое подвыражение в блоке параллельно; каждая итерация в `for` параллельна; выбранная ветвь в `if` — одна задача.
>
> **Всё выражение `spawn` блокируется синхронно в ожидании завершения всех задач.**
>
> **Нет колбэков, нет `await`, нет странных аннотаций.**

```yaoxiang
// обычный код: последовательное выполнение
a = compute_a()         // выполняется сначала
b = compute_b(a)        // зависит от a, выполняется после завершения a
c = compute_c(b)        // зависит от b, выполняется после завершения b

// при необходимости параллелизма: используйте spawn
(x, y, z) = spawn {
    fetch("url1"),      // параллельно
    fetch("url2"),      // параллельно
    fetch("url3")       // параллельно
}
// ожидание завершения всех перед продолжением
process(x, y, z)

// параллелизм данных: spawn for
results = spawn for item in items {
    process(item)
}
```

## Компромиссы

### Преимущества

1. **Простота**: только один примитив параллелизма (`spawn`), модифицирующий любое выражение
2. **Явность**: пользователь точно знает, где параллельно, а где последовательно, без неявного параллелизма
3. **Безопасность**: правила владения естественно расширяются, без дополнительных ограничений типов вроде `Send`/`Sync`
4. **Управляемость**: нет автоматического параллельного поведения, что позволяет избежать непреднамеренных проблем параллелизма
5. **Синхронность**: вызывающая сторона блокируется синхронно, код легко понимать и отлаживать
6. **Нет окрашивания функций**: нет проблемы окрашивания функций, свойственной async/await
7. **Эффективная компиляция**: анализ DAG ограничен выражениями `spawn`, время компиляции контролируемо
8. **Ортогональность**: `spawn` естественно сочетается с любыми управляющими конструкциями (подробности в RFC-032)

### Недостатки

1. **Требуется явный `spawn`**: нельзя распараллелить автоматически, пользователь должен вручную отмечать точки параллелизма
2. **Анализ DAG внутри выражений `spawn`**: компилятор должен выполнять анализ зависимостей внутри выражений `spawn`
3. **Несовместимость со старым кодом**: код, использующий старые шаблоны RFC-001, необходимо мигрировать

## Альтернативы

| Альтернатива | Почему не выбрана |
|--------------|-------------------|
| Автоматический DAG по всей программе (RFC-001) | Высокая сложность, длительная компиляция, неуправляемое поведение |
| async/await | Окрашивание функций, крутая кривая обучения, плохая читаемость кода |
| Горутины | Нет типобезопасности, сложно обнаружить гонки данных |
| Модель акторов | Сложная передача сообщений, трудная отладка |
| CSP (каналы Go) | Нет типобезопасности, сложно обнаружить взаимоблокировки |
| `spawn` модифицирует только блок `{}` | Нарушает ортогональность, `spawn for` становится особым случаем (см. RFC-032) |

## Стратегия реализации

### Анализ на этапе компиляции

1. **Распознавание формы выражения**: форма выражения после `spawn` определяет декомпозицию задач (подробности в RFC-032 §Анализ DAG)
2. **Построение DAG**: анализ отношений зависимостей внутри выражений `spawn`
3. **Топологическая сортировка**: определение порядка выполнения внутри выражений `spawn`
4. **Идентификация параллелизма**: распознавание поддеревьев без зависимостей внутри выражений `spawn`
5. **Анализ побега**: `ref` → Rc или Arc
6. **Обнаружение конфликтов ресурсов**: обнаружение потенциальных конфликтов типов ресурсов

### Организация модулей

Связанный с `spawn` код единообразно размещается в `frontend/core/spawn/`:

```
frontend/core/spawn/
├── mod.rs           # вход в модуль spawn
├── placement.rs     # проверка допустимости позиции появления spawn
└── analysis.rs      # распознавание задач, анализ зависимостей, обнаружение конфликтов ресурсов
```

> **Заметка о миграции** (2026-06-11): существующий `frontend/core/typecheck/passes/spawn_placement.rs` будет перемещён в `frontend/core/spawn/placement.rs`. Объявление модуля `spawn_placement` в каталоге `typecheck/passes/` необходимо одновременно удалить.

### Выполнение в среде выполнения

Ссылка на архитектуру Runtime из [RFC-008](./008-runtime-concurrency-model.md):

- **Embedded Runtime**: без поддержки `spawn`, немедленное выполнение
- **Standard Runtime**: поддержка выражений `spawn`
- **Full Runtime**: Standard + балансировка нагрузки WorkStealer

### Зависимости

- RFC-008 (Архитектура Runtime) → завершён
- RFC-009 (Модель владения) → завершён
- RFC-010 (Унифицированный синтаксис типов) → завершён
- RFC-011 (Система обобщений) → завершён
- RFC-032 (Рефакторинг AST/IR) → совместно с данным документом определяет `spawn`
