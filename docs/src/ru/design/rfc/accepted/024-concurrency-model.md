# RFC-024：Модель параллелизма на основе спавн-блоков

> **Справочные материалы**:
> - [Спецификация модели параллелизма](/reference/language-spec/concurrency.md)
> - [RFC-008: Отделение модели параллелизма Runtime от планировщика](./008-runtime-concurrency-model.md)
> - [RFC-009: Проектирование модели владения](./009-ownership-model.md)
> - [RFC-010: Унифицированный синтаксис типов](./010-unified-type-syntax.md)

## Аннотация

В данном документе определяется новая модель параллелизма языка программирования YaoXiang: использование блока `spawn {}` в качестве единственного примитива параллелизма, зависимое выполнение, синхронная блокировка вызывающей стороны. Заменяет старую схему параллелизма на основе аннотаций `@block`/`@eager`/`@auto`, трэйтов `Send`/`Sync` и анализа DAG всей программы.

**Основной дизайн — один примитив, одно правило**：

```
spawn { ... }        ← единственный примитив параллелизма
Прямые подвыражения создают задачи  ← единственное правило
Синхронная блокировка ожидания результата      ← единственное поведение
```

**Устранённая сложность**：
- ❌ Нет аннотаций `@block`/`@eager`/`@auto`
- ❌ Нет трэйтов `Send`/`Sync`
- ❌ Нет `Mutex`/`RwLock`/`Atomic`
- ❌ Нет `future`/неблокирующих дескрипторов
- ❌ Нет анализа DAG всей программы
- ❌ Нет окрашивания функций (async/await)

> **Ментальная модель пользователя**：Ваш обычный код выполняется последовательно. Когда вы хотите, чтобы несколько вещей выполнялись одновременно, поместите их в блок `spawn { ... }`. Никаких колбэков, никакого `await`, никаких странных аннотаций.

## Мотивация

### Зачем нужен этот дизайн?

Текущие модели параллелизма в основных языках имеют очевидные недостатки:

| Язык | Модель параллелизма | Проблема |
|------|----------|------|
| Rust | async/await + tokio | Асинхронное заражение, окрашивание функций, крутая кривая обучения |
| Go | goroutine | Нет безопасности типов, гонки данных трудно обнаружить |
| Python | asyncio | Ограничение GIL, окрашивание функций |
| JavaScript | Promise/async | Callback hell, окрашивание функций |

### Проблемы старого дизайна (RFC-001)

Трёхуровневая архитектура параллелизма (L1/L2/L3), предложенная в RFC-001, имеет следующие проблемы:

| Проблема | Описание |
|------|------|
| Сложная ментальная модель | Три уровня абстракции L1/L2/L3 увеличивают нагрузку на обучение |
| Избыточность аннотаций | Аннотации `@block`/`@eager`/`@auto` загрязняют код |
| Высокая сложность анализа | Накладные расходы на время компиляции при анализе DAG всей программы |
| Сложные ограничения типов | Трэйты `Send`/`Sync` увеличивают когнитивную нагрузку |
| Неконтролируемость | Автоматическое параллельное поведение трудно предсказать и отлаживать |

### Цели дизайна

1. **Простота**：Только один примитив параллелизма（`spawn`）, одно правило（прямые подвыражения создают задачи）
2. **Явность**：Пользователь точно знает, где параллельно, а где последовательно
3. **Безопасность**：Правила владения естественно расширяются, без дополнительных ограничений типов
4. **Контролируемость**：Нет неявного параллелизма, нет неожиданного параллельного поведения
5. **Синхронность**：Вызывающая сторона синхронно блокируется, нет колбэков и `await`

## Предложение

### 1. Сущность блока {}：Единица вычисления с управлением по зависимостям

В YaoXiang `{}` — это **единица вычисления с управлением по зависимостям**.

| Атрибут | Описание |
|------|------|
| Управление по зависимостям | При выполнении блок проверяет готовность всех внутренних переменных; если все готовы — выполняется немедленно, иначе блокируется в ожидании |
| Время выполнения | Определяется зависимостями, не связано с «немедленным» или «отложенным» |
| Возвращаемое значение | Явно возвращается через `return`; без `return` по умолчанию возвращается `Void` |
| Единый синтаксис | Семантика одинакова независимо от того,出现在函数体、变量初始化还是 `spawn` 后 |
| Изоляция области видимости | Переменные строго ограничены внутри `{}`, не проникают во внешнюю область |

```yaoxiang
// Пример управления по зависимостям
x = compute_x()        // x готов
y = compute_y()        // y готов
result = {
    // Зависит от x и y, выполняется немедленно после готовности обоих
    return x + y
}
```

### 2. Семантика спавн-блока

`spawn { ... }` — это **единственный примитив параллелизма** в YaoXiang.

#### 2.1 Основные правила

- **Прямые подвыражения** спавн-блока создают параллельные задачи
- Выражения внутри вложенных `{}` не считаются независимыми задачами
- Спавн-блок следует стандартным правилам возврата: необходимо явно возвращать значение через `return`, без `return` возвращается `Void`
- Весь спавн-блок синхронно блокируется и ожидает завершения всех задач перед возвратом
- Никаких колбэков, `await` или аннотаций

```yaoxiang
// Две задачи выполняются параллельно
(a, b) = spawn {
    t1 = fetch("url1")   // Прямое подвыражение → параллельная задача 1
    t2 = fetch("url2")   // Прямое подвыражение → параллельная задача 2
    return (t1, t2)      // Явный возврат кортежа
}

// Внутри вложенных {} — не прямые подвыражения
result = spawn {
    x = {               // Этот блок — одно прямое подвыражение → одна задача
        inner_work()    // Не является прямым подвыражением спавна, не станет независимой задачей
    },
    process(x)          // Прямое подвыражение → параллельная задача
    return process(x)
}

#### 2.2 Изоляция области видимости

Спавн-блок создаёт независимую область видимости; внутренние переменные не влияют на внешние:

```yaoxiang
x = 10
result = spawn {
    x = 20              // Это локальный x внутри спавн-блока
    compute(x)
}
// x по-прежнему 10
```

#### 2.3 Правила владения

После того как переменная входит в спавн-блок, она больше не может использоваться извне (семантика Move):

```yaoxiang
data = load_data()
result = spawn {
    process(data)       // Владение data перемещается в спавн-блок
}
// data здесь недоступна (перемещена)
```

Если необходимо совместное использование между задачами, используйте `ref`:

```yaoxiang
data = load_data()
shared = ref data       // Компилятор автоматически выбирает Rc или Arc

result = spawn {
    process_a(shared),  // Общая ссылка
    process_b(shared)   // Общая ссылка
}
```

#### 2.4 Обработка ошибок

Распространение ошибок внутри спавн-блока следует следующим правилам:

1. Ожидание завершения всех задач (даже если некоторые уже завершились с ошибкой)
2. Распространение первой встреченной ошибки
3. Использование `?` для явного обозначения точки распространения ошибки

```yaoxiang
(a, b) = spawn {
    fetch("url1")?,     // Может завершиться ошибкой
    fetch("url2")?      // Может завершиться ошибкой
}
// Если любая задача завершилась ошибкой, весь спавн-блок распространяет первую ошибку
```

#### 2.5 Типы ресурсов

Компилятор отслеживает использование типов ресурсов, обеспечивая безопасность параллелизма:

| Тип ресурса | Описание | Поведение компилятора |
|----------|------|-----------|
| `FilePath` | Путь к файловой системе | Операции с одним путём автоматически сериализуются |
| `HttpUrl` | HTTP-endpoint | Операции с одним URL автоматически сериализуются |
| `DBUrl` | Подключение к базе данных | Операции с одним подключением автоматически сериализуются |
| `Console` | Стандартный вывод | Все операции Console автоматически сериализуются |

```yaoxiang
// Операции с одним файлом автоматически сериализуются
(a, b) = spawn {
    read_file("data.txt"),      // Выполняется первым
    write_file("data.txt", x)   // Ожидает завершения чтения
}
```

#### 2.6 spawn for：Параллельный цикл по данным

```yaoxiang
// Параллельная обработка каждого элемента списка
results = spawn for item in items {
    result = process(item)
}
```

#### 2.7 Вложенный спавн

Спавн-блоки могут быть вложенными; внутренний спавн создаёт новую область параллелизма:

```yaoxiang
(a, b) = spawn {
    x = spawn {
        fetch("url1"),
        fetch("url2")
    },
    y = compute(x)
}
```

### 3. Разрыв со старым дизайном

| Старый дизайн（RFC-001） | Новый дизайн（RFC-024） |
|------------------|------------------|
| Автоматический анализ DAG всей программы | Только анализ внутри спавн-блока |
| Аннотации `@block`/`@eager`/`@auto` | Без аннотаций, управление по зависимостям |
| Трэйты `Send`/`Sync` | Не нужны, владение + ref обрабатываются автоматически |
| `future`/неблокирующие дескрипторы | Синхронная блокировка, без колбэков |
| `Mutex`/`RwLock`/`Atomic` | `ref` автоматически выбирает Rc/Arc |
| Трёхуровневая ментальная модель L1/L2/L3 | Обычный код последователен, спавн-блоки параллельны |
| Окрашивание функций（async/await） | Без окрашивания функций |

### 4. Правила возврата

Правила возврата в YaoXiang унифицированы и однозначны:

| Запись | Возвращаемое значение | Описание |
|------|--------|------|
| `= expr`（без фигурных скобок） | Непосредственный возврат `expr` | Выражение является значением |
| `= { ... }`（с фигурными скобками） | Обязательно использовать `return`, иначе возврат `Void` | Блок требует явного возврата |

```yaoxiang
// Без фигурных скобок: непосредственный возврат
add: (a: Int, b: Int) -> Int = a + b

// С фигурными скобками: обязательно return
process: (data: Data) -> Result = {
    validated = validate(data)?
    return ok(transform(validated))
}

// С фигурными скобками, но без return: возврат Void
log: (message: String) -> Void = {
    print(message)  // Без return, возврат Void
}
```

### 5. Ментальная модель пользователя

> **Ваш обычный код выполняется последовательно.**
>
> **Когда вы хотите, чтобы несколько вещей выполнялись одновременно, поместите их в блок `spawn { ... }`.**
>
> Каждое прямое подвыражение внутри блока начнёт выполняться немедленно (параллельно), используйте `return` для явного возврата результата.
> Весь блок дождётся завершения всех задач и вернёт вам окончательный результат.
>
> **Никаких колбэков, никакого `await`, никаких странных аннотаций.**

```yaoxiang
// Обычный код: последовательное выполнение
a = compute_a()         // Выполняется первым
b = compute_b(a)        // Зависит от a, выполняется после завершения a
c = compute_c(b)        // Зависит от b, выполняется после завершения b

// Когда нужен параллелизм: используем spawn
(x, y, z) = spawn {
    fetch("url1"),      // Параллельно
    fetch("url2"),      // Параллельно
    fetch("url3")       // Параллельно
}
// После ожидания завершения всех продолжаем
process(x, y, z)
```

## Компромиссы

### Преимущества

1. **Простота**：Только один примитив параллелизма（`spawn`）, одно правило（прямые подвыражения создают задачи）
2. **Явность**：Пользователь точно знает, где параллельно, а где последовательно; нет неявного параллелизма
3. **Безопасность**：Правила владения естественно расширяются; не нужны дополнительные ограничения типов вроде `Send`/`Sync`
4. **Контролируемость**：Нет автоматического параллельного поведения; исключены неожиданные проблемы параллелизма
5. **Синхронность**：Вызывающая сторона синхронно блокируется; код легко понимать и отлаживать
6. **Без окрашивания функций**：Отсутствует проблема окрашивания функций async/await
7. **Эффективная компиляция**：Анализ DAG только внутри спавн-блоков; время компиляции контролируемо

### Недостатки

1. **Необходим явный спавн**：Нет автоматического параллелизма; пользователь должен вручную отмечать точки параллелизма
2. **Анализ DAG внутри спавн-блока**：Компилятор должен выполнять анализ зависимостей внутри спавн-блока
3. **Несовместимость со старым кодом**：Код, использующий старый паттерн RFC-001, требует миграции

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| Автоматический DAG всей программы（RFC-001） | Высокая сложность, большое время компиляции, неконтролируемое поведение |
| async/await | Окрашивание функций, крутая кривая обучения, плохая читаемость кода |
| goroutine | Нет безопасности типов, гонки данных трудно обнаружить |
| Actor model | Сложная передача сообщений, трудная отладка |
| CSP（Go channel） | Нет безопасности типов, тупики трудно обнаружить |

## Стратегия реализации

### Компиляционный анализ

1. **Построение DAG**：Анализ зависимостей внутри спавн-блока
2. **Топологическая сортировка**：Определение порядка выполнения внутри спавн-блока
3. **Распознавание параллелизма**：Распознавание поддеревьев без зависимостей внутри спавн-блока
4. **Анализ побега**：`ref` → Rc или Arc
5. **Обнаружение конфликтов ресурсов**：Обнаружение потенциальных конфликтов типов ресурсов

### Выполнение во время выполнения

См. архитектуру Runtime в [RFC-008](./008-runtime-concurrency-model.md):

- **Embedded Runtime**：Без поддержки спавна, синхронное выполнение
- **Standard Runtime**：Поддержка спавн-блоков, параллелизм внутри спавн-блоков
- **Full Runtime**：Standard + балансировка нагрузки WorkStealer

### Зависимости

- RFC-008（Архитектура Runtime）→ Завершено
- RFC-009（Модель владения）→ Завершено
- RFC-010（Унифицированный синтаксис типов）→ Завершено
- RFC-011（Система дженериков）→ Завершено
