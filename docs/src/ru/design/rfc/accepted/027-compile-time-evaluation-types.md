# RFC-027: Предикаты времени компиляции и унифицированная статическая верификация

>

## Резюме

В данном документе предлагается ввести в YaoXiang **предикаты времени компиляции** как полноправных граждан первого класса, объединив всю статическую верификацию на стадии компиляции в единый **конвейер доказательств**. Предикат времени компиляции — это не внешняя спецификационная аннотация: это функция. Функция, возвращающая `Type`, может использоваться в позиции типа, и компилятор вызывает её во время компиляции и проверяет возвращаемое значение. Тип — это утверждение, вычисление во время компиляции — это доказательство.

**Основной тезис**: единственная задача проверки типов во время компиляции — построить и верифицировать доказательственное свидетельство. Равенство типов, конфликты токенов, редукция зависимых типов, вычисление предикатов времени компиляции, импликации логики Хоара — всё это различные проверки типов в едином конвейере доказательств. SMT-решатель — это ускоряющий модуль верификатора типов, а не отдельный корень доверия. Когда компилятор возвращает `Unproven`, программист пишет функцию на YaoXiang в качестве доказательства — верификатор типов проверяет её ровно тем же способом, что и проверку возвращаемого типа любой функции. Всё — код на YaoXiang, всё верифицируется верификатором типов.

## Мотивация

### Почему RFC-022 устарел?

В RFC-022 спецификации проектировались как комментарии `//!`:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← это аннотация вне типа
    //! ensures: ExistsMax(result, arr[0..n])   ← это аннотация вне типа
}
```

Это фундаментальная ошибка изоморфизма Карри-Ховарда: **разделение спецификации и типа на два слоя**. Аннотация — это не тип. Аннотация не участвует в проверке типов. Аннотация — это ментальная модель «внешнего инструмента».

В белой книге сказано прямо:

> «Нет комментариев `//!`. Нет отдельного языка спецификаций. Всё внутри системы типов».

### Текущие проблемы

- Аннотации `//!` в RFC-022 — это внешний синтаксис, не встроенный в систему типов
- Спецификационные типы и обычные типы — это две разные системы, что приводит к концептуальной избыточности
- Разделение на верификацию в Debug Build / игнорирование в Release Build разрушает единство
- SMT-решатель в традиционном понимании позиционируется как внешний инструмент — YaoXiang встраивает его как ускоряющий модуль верификатора типов
- Проверка типов, верификация заимствований, проверка предикатов времени компиляции, раскрытие макросов идут по разным путям

### Корректная ментальная модель

Проверку типов можно абстрагировать как функцию:

```
verify : Program → Proved | Disproved(Model) | Unproven
```

Все проверки времени компиляции — простое сопоставление типов, обнаружение конфликтов заимствований, верификация предикатов времени компиляции — являются подзадачами этой функции. Они используют один конвейер доказательств, различаясь лишь сложностью доказательственного свидетельства и стратегией его построения.

Когда компилятор возвращает `Unproven`, программист предоставляет функцию-доказательство — возвращаемый тип этой функции равен доказываемому утверждению. Верификатор типов проверяет её. Это та же самая операция, что и обычная проверка типов.

## Предложение

### 1. `{}` — это пространство доказательств: тип есть утверждение, верификация есть проверка типов

`{}` в YaoXiang — это пространство доказательств времени компиляции. Всё внутри — утверждения, и компилятор гарантирует, что каждое из них истинно — либо автоматически доказывается, либо предоставляется функция-доказательство от программиста.

```yaoxiang
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  компилятор гарантирует, что x — Float, y — Float

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  компилятор гарантирует, что data — Array(T)
```

**Дженерики — частный случай предикатов времени компиляции.**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          параметр в сигнатуре  внутри {} только утверждения
#          компилятор верифицирует x > 0 при вызове во время компиляции

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      параметр в сигнатуре  компилятор верифицирует type_of(T) == Type, type_of(data) == Array(T)
```

Тот же шаблон: `name: (params) -> Type = { утверждения }`. Компилятор не различает «утверждения о типах» и «утверждения о значениях» — всё это цели вычисления в конвейере доказательств.

**Инварианты цикла не нужно записывать отдельно. Аннотация типа переменной — это инвариант Флойда-Хоара.**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # аннотация ссылается на i — указывает компилятору, что тип s зависит от i
    mut i: UpTo(arr.len) = 0     # при инициализации i=0, верификация: 0 == sum(arr[0..0]) → True
    while i < arr.len {
        s += arr[i]  # компилятор верифицирует: s_new == sum(arr[0..i+1])
        i += 1       # i изменён → запускается реверификация зависимости s: s удовлетворяет SumUpTo(arr, i_new)
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

Компилятор генерирует одно условие верификации для тела цикла — индуктивная гипотеза (аннотация типа) → операция присваивания → удовлетворяет ли новое значение аннотации типа. Конвейер доказательств верифицирует индуктивный шаг, после чего все итерации покрываются автоматически. Не нужны `: decreases`, не нужны `: Invariant`, не нужны индуктивные доказательства — компилятор разбивает индукцию на локальные VC для каждого присваивания.

### 2. Предусловия/постусловия: предикаты времени компиляции в позиции параметра и возвращаемого типа

Отказываемся от `//! requires`/`//! ensures` из RFC-022. Предикаты времени компиляции становятся аннотациями типа параметра или возвращаемого значения.

**На стороне параметра — вызов функции.** Предикат времени компиляции — это функция, возвращающая `Type`, и использование в позиции параметра — это её вызов — как `factorial(5)`. На стороне возвращаемого значения вводится новое понятие: формальный параметр возвращаемого значения.

```yaoxiang
# Предусловие: явный вызов предиката времени компиляции в типе параметра
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b
#                       ^^^^^^^^^^  b — имя текущего формального параметра, передаётся в Positive
#                       компилятор извлекает значение аргумента в точке вызова, подставляет в b,
#                       верифицирует Positive(аргумент)
#                       Пример: divide(10, 2) → верификация Positive(2) = { 2 > 0 } → True
#                       Пример: divide(10, 0) → верификация Positive(0) = { 0 > 0 } → False → ошибка компиляции

# Постусловие: формальный параметр возвращаемого значения + предикат времени компиляции
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

NonEmpty: (arr: Array(T)) -> Type = { arr.len > 0 }

max: (T: Ord) -> ((arr: NonEmpty(arr))) -> (result: IsMax(T, arr, result)) = {
#                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                            result — формальный параметр возвращаемого значения,
#                                            значение предоставляется оператором return
#                                            компилятор подставляет возвращаемое значение в точке return,
#                                            верифицирует постусловие
    candidate = arr[0]
    for i in 1..arr.len {
        if arr[i] > candidate { candidate = arr[i] }
    }
    return candidate
}
```

**Ключевые правила**:

- **На стороне параметра**: `b: Positive(b)` — `b` — имя текущего формального параметра, передаётся в `Positive` в качестве аргумента. Синтаксис вызова функции, никакой неявности.
- **На стороне возвращаемого значения**: `-> (result: IsMax(T, arr, result))` — `result` — формальный параметр возвращаемого значения, значение предоставляется оператором `return`. `result` существует только в сигнатуре типа, на него ссылается только предикат, он не попадает в область видимости тела функции и не появляется у вызывающего.
- **Формальный параметр возвращаемого значения опционален**: при отсутствии постусловия он не пишется, сигнатура полностью совпадает с обычной функцией (`-> Int`).
- **Единство**: параметры и формальные параметры возвращаемого значения — одно и то же понятие — `имя_параметра: вызов_предиката(имя_параметра)`, различие лишь в том, предоставляется ли значение вызывающим или оператором `return`.

### 3. Распространение путевых условий: верификация значений времени выполнения на этапе компиляции

Когда предикат времени компиляции используется в позиции связывания, аргументы передаются программистом явно. Когда значения времени выполнения попадают в параметры уточняющего типа, компилятор выполняет верификацию посредством сбора путевых условий и SMT-импликации — без явной передачи доказательства от программиста.

#### 3.1 Явный вызов функции

Когда предикат времени компиляции используется в позиции связывания, аргументы передаются программистом явно — это просто вызов функции, без какой-либо неявности.

`Positive: (x: Int) -> Type = { x > 0 }` — это конструктор предиката времени компиляции. Когда он появляется в позиции связывания (объявление параметра, объявление переменной, возвращаемый тип), программист явно передаёт имя уже связанной переменной:

```yaoxiang
b: Positive(b)
// b уже объявлен как текущий формальный параметр, Positive(b) — вызов функции
// После нормализации: b: { b > 0 }
```

Компилятору не нужно неявно заполнять аргументы — `b: Positive(b)` — это то же самое, что `f(5)`, просто вызов функции. `b` связывается как имя параметра, и его аннотация типа `Positive(b)` ссылается на сам `b` — это стандартный паттерн зависимых типов, а не правило неявного раскрытия.

**Единство с `self` из RFC-010**: RFC-010 установил, что `self` — не ключевое слово, а лишь общепринятое имя параметра («написать `p`, `this` или `x` — совершенно одинаково»). `b: Positive(b)` использует тот же механизм — имя параметра может использоваться в аннотации типа. `self` появляется в позиции `self: Point`, `b` появляется в позиции `b: Positive(b)`, обе аннотации типа ссылаются на сам параметр. Различие лишь в сложности аннотации типа, механизм полностью одинаков — после связывания имени тип может зависеть от этого имени.

Возвращаемый тип также использует явный вызов функции:

```yaoxiang
Sorted: (arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (arr: Array(T)) -> (result: Sorted(result)) = { ... }
//                        ^^^^^^^^^^^^^^^^^^^^^^^
//                        result — формальный параметр возвращаемого значения, Sorted(result) — вызов функции
//                        компилятор подставляет возвращаемое значение в result в точке return, верифицирует Sorted(возвращаемое_значение)
```

То же применимо к объявлениям локальных переменных:

```yaoxiang
let x: Positive(x) = 5
// x связывается с 5, Positive(5) → { 5 > 0 } → True → проходит

// let y: Positive(y) = 0
// y связывается с 0, Positive(0) → { 0 > 0 } → False → ошибка компиляции
```

#### 3.2 Сбор путевых условий

Когда значения времени выполнения появляются в условных ветвях, компилятор автоматически собирает путевые условия, формируя **набор гипотез** для текущей области видимости. Эти гипотезы участвуют в верификации как фоновые знания для вычисления `Bool` на этапе компиляции.

```yaoxiang
if y > 0 {
    // компилятор автоматически получает гипотезу в этой ветке: { y > 0 }
    let result = divide(x, y)
    // Условие верификации: (y > 0) ⇒ (y > 0)
    // Конвейер доказательств устанавливает импликацию → Proved
} else {
    // Гипотеза в этой ветке: { !(y > 0) }
    // Если вызвать divide(x, y), условие верификации: !(y > 0) ⇒ y > 0
    // Конвейер доказательств не устанавливает импликацию → Disproved
}
```

Это не зашитый в компилятор специальный паттерн — это естественное поведение конвейера доказательств времени компиляции. Каждая точка проверки типа отправляет в конвейер:

```
{фоновые_гипотезы} ⇒ {цель_верификации}
```

Конвейер доказательств проверяет импликацию. `Proved` → проходит, `Disproved` → ошибка компиляции + контрпример, `Unproven` → ошибка компиляции + неразрешённое утверждение. Фоновые гипотезы берутся из путевых условий текущей программной точки.

#### 3.3 Стек гипотез

Компилятор при анализе потока управления поддерживает набор гипотез для каждого базового блока:

- **if-guard**: `if y > 0` → в true-ветку помещается `y > 0`, в false-ветку помещается `!(y > 0)` (если есть else)
- **match-паттерн**: `if let Some(v) = opt` → в ветку помещается `opt == Some(v)`
- **Логические связки**: `if x > 0 && y < 10` → в ветку помещаются `x > 0` и `y < 10`
- **Предусловие функции**: при вызове `divide(a, b)` для `b` должно выполняться `Positive` — либо из текущих гипотез, либо из аннотации уточняющего типа самого аргумента (если `b` помечен как `Positive`, его тип несёт `b > 0`)
- **Присваивание**: при `let z = y` уже имеющиеся уточняющие условия `y` передаются `z`

Все гипотезы поступают в конвейер доказательств времени компиляции. При переходе на путь SMT-ускорения они транслируются в фоновые утверждения SMT-LIB.

#### 3.4 Отсутствие статического свидетельства — ошибка компиляции

Если программист пишет напрямую:

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

В текущей программной точке нет гипотезы `y > 0`, и сам аргумент `y` не имеет аннотации типа `Positive`. Условие верификации:

```
{} ⇒ { y > 0 }
```

Конвейер возвращает `Disproved` (импликация не выполняется) → ошибка компиляции:

> Не удаётся доказать, что параметр `b` удовлетворяет `Positive` в вызове `divide`.
> `y` приходит из ввода функции, без доказанных границ.
> Рассмотрите охрану вызова ветвью if: `if y > 0 { divide(x, y) }`.

YaoXiang не допускает попадания значений времени выполнения в параметры уточняющего типа без предоставления статического свидетельства. Это не ограничение — это ядро философии жёсткой безопасности. Любой код, который компилятор не может статически доказать, не проходит компиляцию.

#### 3.5 Связь с унифицированным конвейером

Распространение путевых условий — не дополнительный механизм. Это прямое расширение конвейера доказательств времени компиляции на анализ потока управления:

| Этап | Обязанность |
|------|-------------|
| Сбор путевых условий | Фаза анализа потока управления компилятора, аннотирование набором гипотез каждый базовый блок |
| Генерация условий верификации | При обнаружении ограничения типа, требующего верификации, объединить путевые условия + информацию о типе аргумента |
| Вычисление в конвейере доказательств | Ядро компилятора → SMT-ускорение → получение `Proved` / `Disproved` / `Unproven` |
| Результат | `Proved` → проходит; `Disproved` → ошибка компиляции + контрпример; `Unproven` → ошибка компиляции + неразрешённое утверждение (программист может предоставить функцию-доказательство) |

Никаких новых компонентов. Никаких специальных правил. Путевые условия — это фоновые знания конвейера доказательств — разделяющие один конвейер и одну систему бюджетов с равенством типов и ограничениями заимствования.

### 4. Конвейер доказательств времени компиляции

Все проверки времени компиляции используют один конвейер. Основная операция конвейера — **проверка типа** — проверка, равен ли тип доказательственного свидетельства доказываемому утверждению. Всё есть проверка типов.

```
При вычислении выражения Bool во время компиляции
(то есть: требуется построить доказательственное свидетельство)
        │
        ├── Равенство типов (T1 == T2)
        │   → прямое решение компилятора (структурная эквивалентность)
        │
        ├── Условие конфликта токенов (!conflicting(tokens))
        │   → потоко-чувствительный анализ живучести (отслеживание Dup/Linear)
        │
        ├── Редукция зависимых типов (n + m свёртка)
        │   → система переписывания термов времени компиляции (βδι-редукция)
        │
        ├── Предикат времени компиляции (x > 0, forall...)
        │   → ядро компилятора + SMT-ускоряющий модуль
        │
        └── Импликация логики Хоара (P ⇒ Q)
            → ядро компилятора + SMT-ускоряющий модуль
                    │
                    ▼
             ┌──────────┐
             │ Proved   │  → компиляция успешна
             │ Disproved│  → ошибка компиляции + контрпример
             │ Unproven │  → ошибка компиляции + неразрешённое утверждение
             └────┬─────┘
                  │
                  ▼
         Программист пишет функцию-доказательство (код на YaoXiang)
                  │
                  ▼
         Верификатор типов проверяет ──→ Proved ──→ компиляция успешна
                  │
                  ▼
            Верификация не прошла → ошибка компиляции: «доказательство неверно»
```

#### 4.1 Результат доказательства: трёхзначная алгебра

Вычисление во время компиляции возвращает три результата — неизбежное следствие проблемы остановки и естественное деление в теории доказательств:

```
eval_compile_time : BoolExpr → Proved | Disproved(Model) | Unproven
```

- **Proved** → завершено, доказательственное свидетельство построено, проверка типа пройдена. Компиляция продолжается.
- **Disproved(M)** → завершено, существует контрпример M. Ошибка компиляции + контрпример + позиция в исходном коде.
- **Unproven** → в пределах заданного ресурсного лимита доказательство не построено. Ошибка компиляции + неразрешённое утверждение + отчёт о потреблении бюджета.

**Unproven ≠ False.** Когда компилятор говорит «я не могу это доказать», это не значит, что утверждение ложно — просто это выходит за пределы возможностей текущего автоматического доказательства. Это честность, а не дефект.

Жёсткое ограничение бюджета — инженерное решение проблемы остановки. Никаких ручек — они бы означали вопрос пользователю «как вы думаете, ваша программа остановится?», на который не знает ответа ни пользователь, ни компилятор.

#### 4.2 После Unproven: программист пишет доказательство

Когда компилятор возвращает `Unproven`, программист может написать **функцию-доказательство** — это функция на YaoXiang, чей возвращаемый тип равен доказываемому утверждению. Верификатор типов проверяет эту функцию — тем же механизмом, что и проверку `add(a, b): Int`.

```
Утверждение = тип
Доказательство = программа (значение этого типа)
Верификация = проверка типа (единственный корень доверия)
```

SMT-решатель — не отдельный корень доверия — это **ускоряющий модуль верификатора типов**. SMT помогает находить доказательства, но верифицирует доказательства всегда верификатор типов. Когда SMT возвращает `unsat`, компилятор перестраивает его результат в доказательственное свидетельство, которое верификатор типов может проверить. Если перестроение не удаётся (шаги вывода SMT выходят за пределы правил вывода ядра компилятора), происходит откат к `Unproven` — программист может вручную написать функцию-доказательство.

```yaoxiang
# Утверждение: уточняющее свойство, которое компилятор не может доказать автоматически
FirstIsMin: (T: Ord, arr: Sorted(T)) -> Type = {
    forall i in 0..arr.len: arr[0] <= arr[i]
}

# Доказательство: программист пишет функцию, чей возвращаемый тип — это утверждение выше
# Верификатор типов проверяет эту функцию — совершенно так же, как проверяет add(a,b): Int
first_is_min: (T: Ord, arr: Sorted(T)) -> FirstIsMin(T, arr) = {
    # компилятор верифицирует здесь: тип тела функции = FirstIsMin(T, arr)
    ...
}
```

Никакого ИИ, никакого экспорта в Coq, никаких новых концепций. **Свойства, которые нельзя доказать автоматически во время компиляции → программист пишет доказательство в виде кода на YaoXiang → верификатор типов проверяет.** Весь процесс — плавный градиент — компилятор берёт на себя простые доказательства, оставляя сложные программисту.

#### 4.3 Слоистые зависимости внутри конвейера

Вышеописанные вычислители используют общий интерфейс, но имеют порядок вычисления. Равенство типов — предпосылка для всего последующего анализа; проверка владения/токенов зависит от информации о типах; верификация уточняющих предикатов зависит от результатов первых двух слоёв. Компилятор вычисляет по слоям — неудачные выражения нижнего слоя не попадают в верхний — что позволяет избежать траты бюджета решения на типово-ошибочных программах.

```
Порядок вычисления (один конвейер, послойное планирование)
├── Слой 0: равенство типов (T1 == T2)
│   └── структурная унификация → при неудаче последующее бессмысленно, сразу Disproved
├── Слой 1: конфликты владения/токенов
│   └── потоко-чувствительный анализ живучести → при неудаче безопасность памяти не выполняется, сразу Disproved
└── Слой 2: уточняющие предикаты / импликации Хоара
    └── ядро компилятора → SMT-ускорение → получение Proved / Disproved / Unproven
```

Каждый слой по-прежнему возвращает `Proved/Disproved/Unproven`, разделяя общий интерфейс и систему бюджетов.

### 5. Единство трёх слоёв функций

| Слой | Момент выполнения | Вход | Выход | Пример |
|------|-------------------|------|-------|--------|
| Функция-значение | Время выполнения | значения | значение | `add: (a: Int, b: Int) -> Int = a + b` |
| Конструктор типа | Время компиляции | типы/значения | `Type` | `List: (T: Type) -> Type = { data: Array(T) }` |
| Предикат времени компиляции | Время компиляции | значения | `Type` | `Positive: (x: Int) -> Type = { x > 0 }` |

Все используют одинаковый синтаксис `name: type = value`. Предикаты времени компиляции и конструкторы типов идут по одному конвейеру доказательств времени компиляции — `{}` — это пространство доказательств.

### 6. Циклы: генерация условий верификации Флойда-Хоара

Циклам не нужны отдельные аннотации `: Invariant(...)` или `: decreases(...)`. Уточняющие предикатные аннотации типа на переменных определяют утверждения в стиле Флойда-Хоара — компилятор генерирует условия верификации из аннотаций типа, а конвейер доказательств проверяет, сохраняет ли каждое присваивание тип.

Основной механизм: каждой операции присваивания соответствует тройка Хоара `{P} x := e {Q}`, условие верификации — `P ⇒ Q[e/x]`. Компилятор генерирует одно условие верификации для тела цикла — после того, как конвейер доказательств верифицирует индуктивный шаг, все итерации покрываются автоматически.

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # аннотация ссылается на i; при инициализации i=0, верификация: 0 == sum(arr[0..0]) → True
    mut i: UpTo(arr.len) = 0     # верификация: 0 <= 0 <= arr.len → True
    while i < arr.len {
        # Компилятор генерирует одну VC для тела цикла. Предпосылка: s удовлетворяет SumUpTo(arr, i),
        # i удовлетворяет UpTo(arr.len).
        #
        # s += arr[i]:
        #   Обязательство верификации: s_new удовлетворяет SumUpTo(arr, i) (текущий i не изменился)
        #   Подставляя s_new = s_old + arr[i]:
        #     Требуется s_old + arr[i] == sum(arr[0..i+1])
        #     Из индуктивной гипотезы s_old == sum(arr[0..i]), прибавив arr[i] к обеим сторонам:
        #     sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   Компилятор + SMT: линейная арифметика, миллисекунды → Proved
        #
        # i += 1:
        #   i изменился → в графе зависимостей аннотация типа s ссылается на i → запуск реверификации
        #   Новая цель верификации: s удовлетворяет SumUpTo(arr, i_new)
        #   То есть s == sum(arr[0..i_new]), гарантировано предыдущим шагом → Proved
        s += arr[i]
        i += 1
    }
    return s  # Здесь s: SumUpTo(arr, arr.len), то есть s == sum(arr[0..arr.len])
}
```

Инвариант цикла — это аннотация типа на переменной — программист пишет тип, компилятор проверяет индуктивный шаг. Компилятору не нужно «обнаруживать» инвариант и не нужно «автоматически делать индукцию» — он разбивает индуктивное доказательство на локальные условия верификации для каждой операции присваивания, передавая их в конвейер доказательств для раздельной обработки.

#### 6.1 Отслеживание зависимостей: зависимые типы на мутабельных переменных

Предпосылка вышеописанного механизма: компилятор знает, что аннотация типа `s` — `SumUpTo(arr, i)` — ссылается на `i` — при изменении `i` ограничение типа `s` также изменяется. Для этого компилятор должен поддерживать **граф зависимостей типов между переменными**.

**Структура данных**:

```
TypeDepGraph: Map<VarName, Set<VarName>>
# Ключ — зависимая переменная, значение — множество переменных, на которые ссылается аннотация типа
# Пример: { i: {s}, j: {s, t}, ... }
```

**Построение**: при обработке `mut v: Pred(... x ...) = init` верификатор типов анализирует ссылки на свободные переменные в аргументах `Pred(...)`. Если в аргументах есть ссылки на другие мутабельные переменные `x` в текущей области видимости, в графе зависимостей фиксируется `x → v`.

**Срабатывание**: когда зависимой переменной `x` присваивается значение, компилятор:
1. Ищет в графе зависимостей все переменные, зависящие от `x`: `{v₁, v₂, ...}`
2. Для каждого `v` генерирует условие верификации: удовлетворяет ли текущее значение `v` обновлённому типу `Pred(... x_new ...)`
3. Отправляет VC в конвейер доказательств

**Чувствительность к порядку присваивания**: отслеживание зависимостей естественным образом форсирует правильный порядок присваивания. Для `SumUpTo(arr, i)`:

```yaoxiang
# Правильный порядок
s += arr[i]   # s_new удовлетворяет SumUpTo(arr, i+1)
i += 1        # i изменён → реверификация: s удовлетворяет SumUpTo(arr, i_new) → True

# Неправильный порядок — компилятор отвергает
i += 1        # i изменён → реверификация: s удовлетворяет SumUpTo(arr, i_new)
              # s ещё не обновлён, s_old == sum(arr[0..i_old]) ≠ sum(arr[0..i_new])
              # → ошибка компиляции: переменная s не удовлетворяет типу SumUpTo(arr, i_new)
s += arr[i]   # недостижимо
```

**Составные зависимости**: переменная может зависеть от нескольких переменных. Аннотация типа `{ v: Int; v == x + y }` одновременно зависит от `x` и `y` — изменение любой из них запускает реверификацию.

**Связь с конвейером доказательств**: отслеживание зависимостей — это триггер генерации VC, а не отдельный механизм верификации. Он отвечает на вопрос «когда нужно генерировать VC» — конвейер доказательств отвечает на вопрос «выполняется ли VC».

### 7. Проверка завершения

Полностью автоматически на этапе компиляции. Циклы, которые компилятор может доказать, проходят; циклы, которые компилятор не может доказать, сразу вызывают ошибку компиляции — программист обязан сделать цикл анализируемым компилятором автоматически. Никаких лазеек для полуавтоматических аннотаций.

#### 6.1 Принципы проектирования

Компилятор автоматически извлекает информацию, необходимую для доказательства завершения, из двух источников:

1. **Аннотации типа переменной**: ограничения границ в уточняющих типах (например, `UpTo(n)` даёт верхнюю границу `n` и нижнюю `0`)
2. **Операции тела цикла**: операции, применяемые к переменным на каждой итерации

Компилятор по приоритету пробует четыре стратегии синтеза меры, останавливаясь на первой успешной.

#### 6.2 Стратегия 1: автоматический синтез линейной ранговой функции

Когда переменная имеет аннотацию линейной границы, компилятор перебирает кандидатов на линейную меру и верифицирует через SMT.

```
Вход:
  Переменные v₁: UpTo(u₁), v₂: UpTo(u₂), ... (переменные с границами)
  Условие цикла cond
  Множество присваиваний в теле цикла

Алгоритм:
  1. Извлечь границы каждой переменной из аннотации типа: [low_i, high_i]
  2. Перебрать кандидатов меры: v_i, u_i - v_i, v_i - v_j, и т.д.
  3. Для каждого кандидата m:
     - SMT верифицирует m ≥ 0 (из типовых границ)
     - Для каждого пути выполнения тела цикла SMT верифицирует m' < m (строгое убывание)
  4. Найти подходящую линейную комбинацию → завершение доказано
```

Охват: циклы, в которых переменной присваивается линейное выражение (`v = a·v + b`) и имеется аннотация ограниченного типа. Включая `i += const`, `i -= const`, а также сокращение интервалов в стиле двоичного поиска:

```yaoxiang
# Двоичный поиск: low = mid + 1 или high = mid
# мера high - low строго убывает на обоих путях
binary_search: (arr: Sorted(Int, arr), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 Стратегия 2: счётчик нарушений предиката — автоматическое извлечение меры из целевого типа <span style="color:orange">【Экспериментальная стратегия】</span>

> ⚠️ **Текущий статус: экспериментальная стратегия, во время реализации Фазы 3 решение о включении принимается на основе фактической осуществимости.**
> Эта стратегия работает для обменов соседних элементов (сортировка пузырьком, сортировка вставками), но не может автоматически доказать завершение для не-смежных операций (partition быстрой сортировки, sift-down пирамидальной сортировки).
> Границы охвата см. в таблице ниже. Если на Фазе 3 верификация покажет неосуществимость, эта стратегия будет удалена или переведена в будущую работу.

Ключевое наблюдение: **спецификации, написанные пользователем, — это материал для рассуждений компилятора.** Компилятору не нужно встраивать «что такое сортировка» — он читает определение `Sorted` и автоматически извлекает из него меру.

```
Вход:
  Целевой тип: Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  Операции тела цикла: обмен соседних элементов

Алгоритм:
  1. Разобрать определение предиката: forall i in range: cond(i, arr)
  2. Автоматически сгенерировать меру: violation_count = |{ i | ¬cond(i, arr) }|
  3. Проанализировать влияние операции на меру:
     - Обмен соседних arr[j], arr[j+1] = arr[j+1], arr[j]
     - Затрагивает только пары индексов j-1, j, j+1
     - Если arr[j] > arr[j+1] (нарушение предиката), после обмена эта пара удовлетворяет предикату
     - violation_count уменьшается как минимум на 1
  4. Верхняя граница: n·(n-1)/2 (максимальное число соседних инверсий), нижняя: 0
  → завершение доказано
```

**Текущий охват**:

| Алгоритм | Паттерн операции | Стратегия 2 доказуема? | Причина |
|----------|------------------|:---:|---------|
| Сортировка пузырьком | Обмен соседних | ✅ | violation_count строго убывает при каждом обмене |
| Сортировка вставками | Смещение соседних | ✅ | Каждое смещение устраняет одну нарушающую пару |
| Сортировка выбором | Не-смежный обмен | ❌ | Один обмен может увеличить violation_count |
| Быстрая сортировка | partition | ❌ | Не-смежный обмен, не гарантирует монотонного убывания |
| Пирамидальная сортировка | sift-down | ❌ | Древовидная операция, violation_count немонотонен |

**Дополняющие стратегии**: для быстрой сортировки сокращение интервала `low < high` покрывается стратегией 1 (линейная ранговая функция) — внешняя partition-рекурсия, каждый интервал делится пополам. Стратегии 1 и 2 дополняют друг друга; завершение большинства практических алгоритмов доказуемо одной из них. Однако обобщение стратегии 2 (не-смежные операции, древовидные операции) остаётся открытой проблемой.

```yaoxiang
sort: (arr: Array(Int)) -> (result: Sorted(result)) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 Стратегия 3: шаблон ограниченного инкремента/декремента

`v += const` (неотрицательная константа), переменная с аннотацией верхней границы → мера `upper_bound - v` убывает на `const` каждую итерацию, нижняя граница 0. Это вырожденный случай стратегии 1, компилятор обрабатывает его в первую очередь.

#### 6.5 Стратегия 4: шаблон меры с мультипликативным масштабированием

`v *= const` (const > 1), переменная с аннотациями верхней и нижней границ. Компилятор имеет встроенный шаблон логарифмической меры `ceil(log_const(upper/v))`, при каждом умножении на const мера уменьшается на 1.

```yaoxiang
mut i: Positive(i) = 1
while i < n {
    # компилятор автоматически выводит: мера ceil(log₂(n/i)), при каждом умножении на 2 мера уменьшается на 1
    i *= 2
}
```

#### 6.6 Разделение завершения и корректности

Доказательство завершения и доказательство корректности независимы:
- **Завершение**: четыре стратегии выше автоматически доказывают, что цикл завершается за конечное число шагов
- **Корректность**: продвигается ли тело цикла к целевому типу, проверяется конвейером доказательств через условия верификации

Оба проходят → компиляция успешна. Завершение доказано, но корректность не прошла → ошибка компиляции + контрпример. Корректность доказана, но завершение не удалось → ошибка компиляции с указанием переменных или операций, которые не удаётся проанализировать. Оба не прошли → ошибка компиляции с раздельным указанием причин обоих сбоев.

#### 6.7 Проверка завершения рекурсивных функций

Для рекурсивных функций, вычисляемых во время компиляции, компилятор проверяет убывание аргумента:

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # компилятор анализирует: n-1 < n → убывает → завершается
}

# Использование во время компиляции — компилятор гарантирует завершение factorial во время компиляции
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120, вычисление во время компиляции
```

| Сценарий | Поведение |
|----------|-----------|
| Компилятор может проанализировать убывание рекурсии (например, `n-1`) | Вычисление во время компиляции |
| Не убывает / невозможно определить убывание | Ошибка компиляции |
| Вызов во время выполнения (не в позиции типа) | Проверка завершения не требуется |

#### 6.8 Жёсткая граница

`i = f(i)`, где `f` необратима, не замкнута и не сохраняет никакой монотонности — математически невозможно автоматически доказать завершение. Ошибка компиляции:

> Завершение этого цикла не может быть доказано автоматически. Переменная цикла зависит от неанализируемой функции `f`. Используйте шаблон итерации, анализируемый компилятором.

Это не провал компилятора. Любой код, который нельзя статически доказать как безопасный, не проходит компиляцию.

### 8. SMT-решатель: ускоряющий модуль верификатора типов

В традиционных языках SMT-решатель — это внешний инструмент (F\* вызывает Z3, Dafny вызывает Z3). В YaoXiang это **ускоряющий модуль верификатора типов** — он вызывается только тогда, когда ядро компилятора само не может принять решение. SMT помогает находить доказательства, но верифицирует доказательства верификатор типов.

**Модель доверия**: верификатор типов — единственный корень доверия. SMT-решатель — ускоряющий модуль — он помогает находить доказательства, но SMT — не отдельный корень доверия. Компилятор доверяет результату `unsat` от Z3 (согласуется с подходами F\*/Dafny — вероятность ошибки Z3 ниже, чем частота багов самого компилятора, это прагматичный инженерный выбор). Реальная ненадёжность контролируется на уровне трансляции в SMT — если в трансляции есть баг, компилятор обнажит его в других тестах.

**Интерфейс**: компилятор внутри транслирует в стандартный формат SMT-LIB 2.6, а не привязывается к API конкретного решателя. SMT-LIB — стандарт ISO, Z3, CVC5, MathSAT, Yices — все нативно поддерживают.

**Бэкенд по умолчанию**: Z3 (лицензия MIT, наиболее обширная документация и валидация сообщества). CVC5 как SMT-LIB-совместимая альтернатива — пользователь может переключаться флагом компилятора.

Никакого «универсального слоя абстракции решателя» — SMT-LIB и есть слой абстракции. В будущем, если CVC5 добьётся прорыва в определённой теории, переключение — это просто смена бинарника, без изменения кода компилятора.

```
Выражение Bool во время компиляции
        │
        ├── Ядро компилятора может решить напрямую (структурная эквивалентность,
        │   простая арифметика, тривиальные формулы после свёртки констант)
        │   → сразу возвращает Proved / Disproved
        │
        └── Ядро компилятора не может решить напрямую (кванторы, символьные переменные)
            → предварительная редукция зависимых типов (factorial(5) → 120)
            → трансляция в формат SMT-LIB
            → отправка Z3/CVC5 (с ограничением бюджета)
            → возврат: unsat → Proved  │  sat + модель → Disproved  │  unknown → Unproven
```

**Бюджет решения — жёсткое ограничение, как глубина стека**:

| Измерение бюджета | Значение по умолчанию | Пояснение |
|-------------------|----------------------|-----------|
| Число шагов решения | 10 000 | Z3 обычно укладывается в сотни шагов для линейной арифметики. 10 000 шагов покрывает 99% практических предикатов. |
| Время | 100 мс | Один предикат дольше 100 мс = пользователь пишет программу времени компиляции, а не аннотацию типа. 100 мс × 50 предикатов = 5 секунд верхней границы компиляции. |
| Глубина инстанцирования кванторов | 3 | Три уровня вложенности кванторов покрывают практические паттерны. Больше трёх — скорее всего, логическая задача. |

Превышение бюджета возвращает `Unproven`, ошибка компиляции + позиция предиката + объём потребления. Никаких деградаций, никаких проверок во время выполнения, никакого silent pass.

**Почему это практически осуществимо**: 95% практических предикатов — это линейная арифметика — `x > 0`, `arr.len > 0`, `0 <= idx < arr.len` — всё в пределах разрешимого фрагмента, SMT-решатель возвращается за миллисекунды. Для редких сложных предикатов, превышающих бюджет, программист может написать функцию-доказательство.

Зависимые типы перед вызовом SMT проходят предварительную редукцию: `factorial(5)` вычисляется во время компиляции в `120`, `append([1,2], [3])` вычисляется в `[1,2,3]`. Эти детерминированные вычисления значений не потребляют бюджет SMT.

Программисту не нужно знать о существовании SMT. Ментальная модель: **компилятор может доказать — проходит, не может — сообщает об ошибке — если компилятор не умеет, можно написать функцию-доказательство**.

### 9. Композиция предикатов времени компиляции

Предикат времени компиляции — это функция, возвращающая `Type`, и композиция естественным образом реализуется через композицию функций:

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(arr)
}
```

### 10. Примеры кода

#### 9.1 Безопасность деления

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b

result = divide(10, 2)   # ✅ компилятор верифицирует Positive(2) = { 2 > 0 } → True
# result = divide(10, 0)  # ❌ компилятор верифицирует Positive(0) = { 0 > 0 } → False
```

#### 9.2 Безопасность доступа к массиву

```yaoxiang
InBounds: (idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (arr: Array(T), idx: InBounds(idx, arr)) -> T = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ компилятор верифицирует InBounds(1, arr) = { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ компилятор верифицирует InBounds(5, arr) = { 0 <= 5 && 5 < 3 } → False
```

#### 9.3 Корректность сортировки

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T))) -> (result: Sorted(T, result)) = {
    result = arr.clone()
    # ... реализация алгоритма сортировки ...
    return result
}
```

#### 9.4 Цикл: генерация VC компилятором

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```

## Детальное проектирование

### Синтаксические изменения

| Ранее (RFC-022) | Далее (этот RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | Предикат времени компиляции как тип параметра `(b: Positive(b))` |
| `//! ensures: ExistsMax(result, arr)` | Возвращаемый тип использует формальный параметр возвращаемого значения `-> (result: IsMax(T, arr, result))` |
| `/*! invariant: ... !*/` | Уточняющие предикатные аннотации типа на переменных — инварианты Флойда-Хоара |
| `//! decreases: n` | Полностью автоматический вывод функции меры компилятором |
| Спецификация — это комментарий | Спецификация — это система типов |

### Синтаксис

**У предикатов времени компиляции нет новых ключевых слов.** `{}` — это пространство доказательств, полностью согласующееся с существующим синтаксисом определения типов. Предикат времени компиляции — это функция, возвращающая `Type` — `name: (params) -> Type = { утверждения }`. Использование — вызов функции — `Positive(b)`, `IsMax(T, arr, result)`.

```bnf
# Предикат времени компиляции = функция, возвращающая Type; внутри {} — утверждения, верифицируемые компилятором
# Используется существующий синтаксис функций/типов, новые правила BNF не нужны
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

**Новое синтаксическое понятие: формальный параметр возвращаемого значения** — в `-> (name: Type)` `name` — это формальный параметр возвращаемого значения.

Формальный параметр возвращаемого значения — это **единственное новое синтаксическое понятие**, вводимое в YaoXiang поверх существующего синтаксиса функций. Его семантика:
- Значение `name` предоставляется оператором `return`
- `name` существует только в сигнатуре типа, на него ссылается предикат постусловия (например, `-> (result: IsMax(T, arr, result))`)
- `name` не попадает в область видимости тела функции и не появляется у вызывающего
- Формальный параметр возвращаемого значения **опционален** — при отсутствии постусловия сигнатура полностью совпадает с обычной функцией (`-> Int`), без какой-либо дополнительной нагрузки

Причина его введения: постусловию нужно ссылаться на «значение, которое функция собирается вернуть». Без формального параметра возвращаемого значения компилятор мог бы позволить предикату ссылаться на возвращаемое значение только через специальные правила (например, неявная переменная `$result` или `__retval__`). Формальный параметр возвращаемого значения делает эту ссылку явной — это просто формальный параметр, значение которого предоставляется `return`, а не вызывающим.

**Функция-доказательство** — не новое понятие — это функция на YaoXiang, чей возвращаемый тип — доказываемое утверждение. Когда компилятор возвращает `Unproven`, программист предоставляет функцию-доказательство, и верификатор типов проверяет её совершенно так же, как проверяет возвращаемый тип любой функции. Никакого нового синтаксиса, ключевых слов, правил не нужно.

### Влияние на систему типов

- **Вселенная типов**: предикаты времени компиляции находятся на уровне `Type₂` — функции, принимающие значения и возвращающие `Type`, на том же уровне, что и конструкторы типов
- **Взаимодействие с дженериками**: предикаты времени компиляции могут иметь параметры-дженерики, например `NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **Взаимодействие с владением**: выражения в предикатах времени компиляции подчиняются правилам владения, только чтение, без записи
- **Вывод типов**: аргументы предикатов времени компиляции участвуют в HM-выводе типов

### Представление во время выполнения

Предикаты времени компиляции **полностью стираются** во время выполнения. `Positive: (x: Int) -> Type = { x > 0 }` — параметр `b: Positive(b)` во время выполнения представляет собой просто `Int`. Условие уточнения `{ x > 0 }` верифицируется конвейером доказательств во время компиляции, и после верификации не остаётся никаких следов во время выполнения.

- `Positive(5)` → во время выполнения представлен как `Int(5)`, условие уточнения `{ 5 > 0 }` верифицировано, стёрто
- `SumUpTo(arr, 0)` → во время выполнения представлен как `Int(0)`, равенство `0 == sum(arr[0..0])` верифицировано, стёрто
- Размещение предиката времени компиляции в позиции типа (например, `f(x: Positive(x))`) не создаёт оборачивающего типа, не выделяет дополнительной памяти, не вставляет проверок времени выполнения

Стирание дженериков и стирание уточняющих типов — один и тот же принцип: оба являются функциями времени компиляции, полностью исчезающими после вычисления во время компиляции. В Release Build предикаты времени компиляции имеют нулевые накладные расходы во время выполнения — это прямое следствие того, что «предикат — это функция».

**Ограничение взаимодействия с `ref`**: предикаты времени компиляции могут ссылаться только на значения с неизменяемым заимствованием или с уже переданным владением. Предикаты времени компиляции, ссылающиеся на значения с мутабельным заимствованием, не могут гарантировать, что результат верификации останется в силе во время выполнения — такое использование сразу вызывает ошибку компиляции.

### Изменения в компиляторе

1. **Парсер**: предикаты времени компиляции используют стандартный синтаксис функций, дополнительные правила разбора не нужны
2. **Конвейер доказательств времени компиляции**: унифицированный интерфейс возврата `Proved/Disproved/Unproven`, автоматический выбор стратегии
3. **SMT-ускоряющий модуль**: слой трансляции в SMT-LIB 2.6, бэкенд по умолчанию Z3, CVC5 как альтернатива
4. **Ядро верификатора типов**: реализация правил вывода — структурная эквивалентность, βδι-редукция, введение/устранение кванторов всеобщности. Это единственный корень доверия, через который проходят и SMT, и доказательства программиста
5. **Генерация условий верификации**: исчисление WP/SP + обязательства доказательства инвариантов цикла
6. **Отчёты об ошибках**: форматирование контрпримеров + отчёт о неразрешённых утверждениях + привязка к позиции в исходном коде

### Обратная совместимость

- ✅ Код, не использующий предикаты времени компиляции, остаётся полностью неизменным
- ✅ Предикаты времени компиляции имеют нулевые накладные расходы во время выполнения в Release Build
- ⚠️ Синтаксис `//!` из RFC-022 больше не поддерживается — однако 022 никогда не был реализован, миграционной нагрузки нет

## Компромиссы

### Преимущества

- **Полная реализация изоморфизма Карри-Ховарда**: тип — утверждение, программа — доказательство, `name: Proposition = Proof`
- **Единство**: предикаты времени компиляции и обычные функции используют полностью одинаковый синтаксис, без концептуального разделения
- **Прозрачность SMT**: программисту не нужно знать о существовании SMT, ментальная модель совпадает с проверкой типов
- **Постепенное внедрение**: можно начать с одного предиката времени компиляции и постепенно расширять покрытие
- **Нулевые накладные расходы во время выполнения**: все вычисления завершаются на этапе компиляции, в Release Build не вставляется никаких утверждений времени выполнения

### Недостатки

- **Время компиляции**: SMT-решение увеличивает время компиляции, но жёсткие ограничения бюджета гарантируют контролируемый верхний предел
- **Границы автоматического доказательства**: сложные предикаты за пределами линейной арифметики первого порядка могут потребовать от программиста написания функций-доказательств. Это не дефект языка — это неизбежное следствие проблемы остановки. Компилятор честно сообщает `Unproven`, а не ложно `True/False`
- **Кривая обучения**: написание эффективных предикатов времени компиляции и функций-доказательств требует понимания базовой интуиции изоморфизма Карри-Ховарда
- **Сложность реализации**: унификация конвейера доказательств времени компиляции требует тщательного проектирования

### Снижение рисков

- Жёсткие ограничения бюджета решения SMT (шаги 10 000 / время 100 мс / глубина инстанцирования 3), превышение бюджета возвращает `Unproven`
- Предварительная редукция зависимых типов: детерминированные вычисления значений поглощаются первыми, SMT кусает только недетерминированную часть
- `Unproven` — не тупик: программист может написать функцию-доказательство, верификатор типов проверит — ровно так же, как проверка возвращаемого типа любой функции
- Инкрементальная верификация: верифицируются только изменённые модули
- Чёткие сообщения об ошибках + показ контрпримеров + отчёт о потреблении бюджета + неразрешённые утверждения + предложения (если компилятор может их дать)

## Альтернативы

| Альтернатива | Почему не выбрана |
|--------------|-------------------|
| RFC-022: спецификации в комментариях `//!` | Спецификация и тип разделены, нарушает изоморфизм Карри-Ховарда |
| Отдельный файл спецификаций (как CVL) | Спецификация и код разделены, увеличивает стоимость поддержки |
| Только утверждения времени выполнения | Не может статически гарантировать корректность |
| Внешний инструмент доказательства (как Coq) | Отрезан от компилятора, требует отдельного языка доказательств и корня доверия. Выбор YaoXiang: доказательство — это код на YaoXiang, верификатор типов — единственный корень доверия |
| **Данное предложение: предикаты времени компиляции как полноправные граждане первого класса** | ✅ |

## Стратегия реализации

### Разделение на фазы

| Фаза | Содержание |
|------|------------|
| **Фаза 1** | Ядро компилятора: структурная эквивалентность + βδι-редукция + введение/устранение кванторов всеобщности. Поддержка простых арифметических предикатов (`x > 0`, `arr.len > 0`) |
| **Фаза 2** | Слой трансляции в SMT-LIB + интеграция Z3/CVC5. Конвейер возвращает `Proved/Disproved/Unproven`. При `Unproven` поддержка функций-доказательств от программиста |
| **Фаза 3** | Генерация VC для инвариантов цикла + проверка завершения (линейная ранговая функция + счётчик нарушений предиката + ограниченный паттерн + контроль комбинаторного взрыва) |
| **Фаза 4** | Инкрементальная верификация + кэширование + поддержка IDE |

### Зависимости

- RFC-010: Унифицированный синтаксис типов — предикаты времени компиляции основаны на `name: type = value`
- RFC-011: Система дженериков — предикаты времени компиляции могут иметь параметры-дженерики
- RFC-009: Модель владения — выражения предикатов времени компиляции подчиняются правилам владения

## Открытые вопросы

- [x] **Выбор SMT-решателя**: по умолчанию Z3 (лицензия MIT, наиболее валидированный). CVC5 как SMT-LIB-совместимая альтернатива, переключение флагом компилятора. Внутренняя цель трансляции компилятора — стандартный формат SMT-LIB 2.6 — SMT-LIB и есть слой абстракции, никакого собственного универсального интерфейса решателя.
- [x] **Конкретные значения бюджета решения**: шаги 10 000 / время 100 мс / глубина инстанцирования кванторов 3. Зафиксированы внутри компилятора, без ручек. Если в реальном использовании появятся реальные случаи, доказывающие недостаточность (не «пользователь ошибся»), значения будут скорректированы.
- [x] **Область поддержки кванторов**: язык не ограничивает арность кванторов. Предикаты времени компиляции принимают параметры типа — `Type` включает типы функций — поэтому кванторы высокого порядка — естественное следствие системы типов и не требуют специального синтаксиса. SMT-решатель может автоматически разрешать кванторы первого порядка (forall/exists, поддерживается чередующееся вложение, ограничено бюджетом глубины 3). Кванторы высокого порядка: SMT возвращает `Unproven`, компилятор выдаёт подсказку «этот предикат выходит за пределы автоматического доказательства, предоставьте функцию-доказательство». Программист пишет функцию на YaoXiang, чей возвращаемый тип равен этому утверждению — верификатор типов проверяет эту функцию. Никакого внешнего экспорта, никакого ИИ, никакого интерактивного режима доказательства. Всё — код на YaoXiang, всё верифицируется верификатором типов.
- [x] **Форматирование контрпримеров**: имена переменных исходного кода используются непосредственно как имена SMT-переменных (с префиксом модуля для предотвращения конфликтов). При возврате модели Z3 выполняется обратный поиск по имени переменной. Формат вывода: имя_переменной = конкретное_значение + позиция_в_исходном_коде + позиция_определения_предиката. Никакого сложного слоя отображения.
- [x] ~~**Взаимодействие предикатов времени компиляции с умными указателями `ref`?**~~ → Решено: предикаты времени компиляции допускают только неизменяемые заимствования или значения с переданным владением. Значения с мутабельным заимствованием не могут появляться в предикатах времени компиляции.
- [x] **Расширение меры счётчика нарушений предиката `forall` на не-смежные операции?** → Не расширяется. Текущий охват (обмен соседних, смещение соседних) дополняется стратегией 1 (линейная ранговая функция) — сокращение внешнего интервала в быстрой сортировке покрывается стратегией 1, пирамидальная сортировка покрывается стратегией 1 (паттерн индексации массива). Циклы, завершение которых не может быть доказано ни одной стратегией, компилятор сразу отвергает — это философия жёсткой безопасности, а не дефект. Если в будущем появится реальный сценарий (не академическая конструкция) с алгоритмом, который не покрывается четырьмя стратегиями, вопрос будет пересмотрен.
- [x] **Комбинаторный взрыв перебора линейной ранговой функции**: верхний предел перебора кандидатов — 3 переменные с границами. При ≤3 перебираются все линейные комбинации с индивидуальной SMT-верификацией. При >3 пробуются только однопеременные меры (`v_i`, `u_i - v_i`); при неудаче сразу ошибка компиляции — с подсказкой программисту «в цикле более 3 переменных с границами, компилятор не может автоматически синтезировать многопеременную меру». Это не инженерный компромисс — это принуждение программиста к написанию более простых циклов.
