## Резюме

Настоящий документ предлагает ввести в YaoXiang **предикаты времени компиляции** как полноправных граждан языка и унифицировать всю статическую верификацию времени компиляции в единый **конвейер доказательства**. Предикат времени компиляции — это не внешняя аннотация спецификации — это функция. Функция, возвращающая Type, может использоваться в позиции типа; компилятор вызывает её во время компиляции и проверяет возвращаемое значение. Тип — это утверждение, вычисление во время компиляции — это доказательство.

**Ключевой тезис**: единственная задача проверки типов во время компиляции — конструировать и верифицировать термы доказательств. Равенство типов, конфликты токенов, редукция зависимых типов, вычисление предикатов времени компиляции, импликации логики Хоара — всё это разные проверки типов в едином конвейере доказательства, разделяющие один и тот же конвейер. SMT-решатель — это ускоряющий модуль проверщика типов, а не отдельный доверенный рубеж. Когда компилятор возвращает Unproven, программист пишет функцию на YaoXiang в качестве доказательства — проверщик типов верифицирует её ровно тем же способом, что и проверку возвращаемого типа любой функции. Всё — код на YaoXiang, всё верифицируется проверщиком типов.

## Мотивация

### Почему RFC-022 устарел?

RFC-022 проектировал спецификации в форме комментариев `//!`:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← Это аннотация, отдельная от типа
    //! ensures: ExistsMax(result, arr[0..n])   ← Это аннотация, отдельная от типа
}
```

Это совершает фундаментальную ошибку изоморфизма Карри—Ховарда: **разделяет спецификации и типы на два слоя**. Аннотации — это не типы. Аннотации не участвуют в проверке типов. Аннотации — это ментальная модель «внешнего инструмента».

Белая книга говорит прямо:

> «Нет комментариев `//!`. Нет отдельного языка спецификаций. Всё внутри системы типов.»

### Текущие проблемы

- Аннотации `//!` из RFC-022 — это внешний синтаксис, отдельный от системы типов
- Типы спецификаций и обычные типы — две разные системы, что порождает концептуальную избыточность
- Разделение на верификацию в Debug Build и игнорирование в Release Build подрывает единство
- SMT-решатель в традиционном понимании позиционируется как внешний инструмент — YaoXiang встраивает его как ускоряющий модуль проверщика типов
- Проверка типов, верификация заимствований, проверка предикатов времени компиляции, раскрытие макросов идут по разным путям

### Правильная ментальная модель

Проверку типов можно абстрагировать как функцию:

```
verify : Program → Proved | Disproved(Model) | Unproven
```

Все проверки времени компиляции — проверка простого соответствия типов, обнаружение конфликтов заимствований, верификация предикатов времени компиляции — являются подзадачами этой функции. Они разделяют один и тот же конвейер доказательства; различие только в сложности терма доказательства и стратегии конструирования.

Когда компилятор возвращает Unproven, программист предоставляет функцию-доказательство — возвращаемый тип этой функции равен доказываемому утверждению. Проверщик типов верифицирует её. Это та же операция, что и обычная проверка типов.

## Предложение

### 1. `{}` — пространство доказательств: тип есть утверждение, верификация есть проверка типов

`{}` в YaoXiang — это пространство доказательств времени компиляции. Всё внутри — утверждения; компилятор гарантирует, что каждый элемент истинен — либо доказывается автоматически, либо доказательство предоставляет программист.

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  Компилятор гарантирует, что x — Float, y — Float

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  Компилятор гарантирует, что data — Array(T)
```

**Дженерики — частный случай предикатов времени компиляции.**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          Параметр в           Внутри {} только утверждения
#          сигнатуре            Компилятор проверяет x > 0 при вызове времени компиляции

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      Параметр в сигнатуре  Компилятор проверяет type_of(T) == Type, type_of(data) == Array(T)
```

Один и тот же шаблон: `name: (params) -> Type = { утверждения }`. Компилятор не различает «утверждения о типе» и «утверждения о значении» — оба являются целями вычисления в конвейере доказательства.

**Инварианты цикла не нужно записывать отдельно. Аннотации типов на переменных — это инварианты Флойда—Хоара.**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # Аннотация ссылается на i — сообщает компилятору, что тип s зависит от i
    mut i: UpTo(arr.len) = 0     # При инициализации i=0, проверка: 0 == sum(arr[0..0]) → True
    while i < arr.len {
        s += arr[i]  # Компилятор проверяет: s_new == sum(arr[0..i+1])
        i += 1       # i изменяется → срабатывает реверификация зависимости s: s удовлетворяет SumUpTo(arr, i_new)
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

Компилятор генерирует одно условие верификации для тела цикла — индуктивное предположение (аннотация типа) → операция присваивания → удовлетворяет ли новое значение аннотации типа. После того как конвейер доказательства установит индуктивный шаг, все итерации покрываются автоматически. Не нужен `: decreases`, не нужен `: Invariant`, не нужны доказательства индукцией — компилятор разбивает индукцию на локальные VC для каждого присваивания.

### 2. Предусловия/постусловия: предикаты времени компиляции на типах параметров и возвращаемого значения

Отказываемся от `//! requires`/`//! ensures` из RFC-022. Предикаты времени компиляции становятся аннотациями типов на параметрах или возвращаемых значениях.

**Сторона параметров — это вызов функции.** Предикат времени компиляции — это функция, возвращающая Type; её использование на стороне параметров — это её вызов — точно так же, как `factorial(5)`. На стороне возвращаемого значения вводится новое понятие: формальный параметр возвращаемого значения.

```yaoxiang
# Предусловие: явный вызов предиката времени компиляции в типе параметра
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b
#                       ^^^^^^^^^^  b — имя текущего формального параметра, передаётся в Positive
#                       Компилятор в точке вызова извлекает значение аргумента, подставляет в b, проверяет Positive(аргумент)
#                       Пример: divide(10, 2) → проверка Positive(2) = { 2 > 0 } → True
#                       Пример: divide(10, 0) → проверка Positive(0) = { 0 > 0 } → False → ошибка компиляции

# Постусловие: формальный параметр возвращаемого значения + предикат времени компиляции
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

NonEmpty: (arr: Array(T)) -> Type = { arr.len > 0 }

max: (T: Ord) -> ((arr: NonEmpty(arr))) -> (result: IsMax(T, arr, result)) = {
#                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                            result — формальный параметр возвращаемого значения, значение задаётся return
#                                            Компилятор в точке return подставляет возвращаемое значение, проверяет постусловие
    candidate = arr[0]
    for i in 1..arr.len {
        if arr[i] > candidate { candidate = arr[i] }
    }
    return candidate
}
```

**Ключевые правила**:

- **Сторона параметров**: `b: Positive(b)` — `b` — имя текущего формального параметра, передаётся в `Positive` как аргумент. Синтаксис вызова функции, ноль неявного.
- **Сторона возврата**: `-> (result: IsMax(T, arr, result))` — `result` — формальный параметр возвращаемого значения, значение задаётся оператором `return`. `result` существует только в сигнатуре типа, на него ссылается только предикат, он не входит в область видимости тела функции и не появляется у вызывающего.
- **Формальный параметр возвращаемого значения необязателен**: при отсутствии постусловия не пишется, сигнатура полностью совпадает с обычной функцией (`-> Int`).
- **Единство**: параметры и формальные параметры возвращаемого значения — одно и то же понятие — `имя_параметра: вызов_предиката(имя_параметра)`, различие лишь в том, задаётся ли значение вызывающим или оператором `return`.

### 3. Распространение путевых условий: верификация значений времени выполнения во время компиляции

Когда предикат времени компиляции используется в позиции связывания, аргументы передаются программистом явно. Когда значение времени выполнения попадает в параметр уточнённого типа, компилятор выполняет верификацию через сбор путевых условий и SMT-импликацию — без явной передачи доказательства от программиста.

#### 3.1 Явный вызов функции

Когда предикат времени компиляции используется в позиции связывания, аргументы передаются программистом явно — это вызов функции, ноль неявного.

`Positive: (x: Int) -> Type = { x > 0 }` — это конструктор предиката времени компиляции. Когда он появляется в позиции связывания (объявление параметра, объявление переменной, тип возврата), программист явно передаёт имя уже связанной переменной:

```yaoxiang
b: Positive(b)
// b уже объявлен как текущий формальный параметр, Positive(b) — это вызов функции
// После нормализации: b: { b > 0 }
```

Компилятору не нужно неявно заполнять аргументы — `b: Positive(b)` и `f(5)` — это одно и то же: вызов функции. `b` связывается как имя параметра, и его аннотация типа `Positive(b)` ссылается на сам `b` — это стандартный шаблон зависимых типов, а не неявное правило раскрытия.

**Единство с `self` из RFC-010**: RFC-010 установил, что `self` — не ключевое слово, а просто конвенциональное имя параметра («написать `p`, `this`, `x` — совершенно одинаковый эффект»). `b: Positive(b)` использует тот же механизм — имя параметра может быть использовано в аннотации типа. `self` появляется в позиции `self: Point`, `b` появляется в позиции `b: Positive(b)`, обе аннотации типов ссылаются на сам параметр. Различие — только в сложности аннотации типа, механизм абсолютно одинаков — после связывания имени тип может зависеть от этого имени.

Тип возврата также использует явный вызов функции:

```yaoxiang
Sorted: (arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (arr: Array(T)) -> (result: Sorted(result)) = { ... }
//                        ^^^^^^^^^^^^^^^^^^^^^^^
//                        result — формальный параметр возвращаемого значения, Sorted(result) — вызов функции
//                        Компилятор в точке return подставляет возвращаемое значение в result, проверяет Sorted(возврат)
```

То же применимо к объявлениям локальных переменных:

```yaoxiang
let x: Positive(x) = 5
// x связывается с 5, Positive(5) → { 5 > 0 } → True → пройдено

// let y: Positive(y) = 0
// y связывается с 0, Positive(0) → { 0 > 0 } → False → ошибка компиляции
```

#### 3.2 Сбор путевых условий

Когда значение времени выполнения появляется в условной ветви, компилятор автоматически собирает путевые условия, формируя **множество гипотез** текущей области видимости. Эти гипотезы участвуют в верификации как фоновые знания для вычисления Bool во время компиляции.

```yaoxiang
if y > 0 {
    // Компилятор в этой ветви автоматически получает гипотезу: { y > 0 }
    let result = divide(x, y)
    // Условие верификации: (y > 0) ⇒ (y > 0)
    // Конвейер доказательства устанавливает импликацию → Proved
} else {
    // В этой ветви гипотеза: { !(y > 0) }
    // При вызове divide(x, y) условие верификации: !(y > 0) ⇒ y > 0
    // Конвейер доказательства не устанавливает импликацию → Disproved
}
```

Это не зашитый в компилятор специальный шаблон — это естественное поведение конвейера доказательства времени компиляции. В каждой точке проверки типов конвейеру отправляется:

```
{фоновые гипотезы} ⇒ {цель верификации}
```

Конвейер доказательства определяет импликацию. Proved → пройдено, Disproved → ошибка компиляции + контрпример, Unproven → ошибка компиляции + недоказанная пропозиция. Фоновые гипотезы поступают из путевых условий текущей точки программы.

#### 3.3 Стек гипотез

Компилятор при анализе потока управления поддерживает множество гипотез для каждого базового блока:

- **if-guard**: `if y > 0` → в истинную ветвь помещается `y > 0`, в ложную — `!(y > 0)` (если используется else)
- **match-паттерн**: `if let Some(v) = opt` → в ветвь помещается `opt == Some(v)`
- **Логические связки**: `if x > 0 && y < 10` → в ветвь помещаются `x > 0` и `y < 10`
- **Предусловие функции**: при вызове `divide(a, b)` требуется, чтобы `b` удовлетворял свидетельству `Positive` — либо из текущих гипотез, либо из аннотации уточнённого типа самого аргумента (если `b` аннотирован как `Positive`, его тип несёт `b > 0`)
- **Присваивание**: при `let z = y` уже имеющиеся у `y` уточняющие условия передаются в `z`

Все гипотезы поступают в конвейер доказательства времени компиляции. При входе в путь SMT-ускорения они транслируются в фоновые утверждения SMT-LIB.

#### 3.4 Отсутствие статического свидетельства — ошибка компиляции

Если программист пишет напрямую:

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

В текущей точке программы нет гипотезы `y > 0`, и сам аргумент `y` не имеет аннотации типа `Positive`. Условие верификации:

```
{} ⇒ { y > 0 }
```

Конвейер возвращает `Disproved` (импликация не выполняется) → ошибка компиляции:

> Невозможно доказать, что параметр `b` в вызове `divide` удовлетворяет `Positive`.
> `y` поступает из входных данных функции, доказанных границ не имеет.
> Рассмотрите защиту вызова ветвью if: `if y > 0 { divide(x, y) }`.

YaoXiang не допускает прямого попадания значений времени выполнения в параметры уточнённого типа без предоставления статического свидетельства. Это не ограничение — это ядро философии жёсткой безопасности. Всё, что компилятор не может статически доказать, не должно проходить компиляцию.

#### 3.5 Связь с унифицированным конвейером

Распространение путевых условий — не дополнительный механизм. Это прямое расширение конвейера доказательства времени компиляции на анализ потока управления:

| Этап | Обязанности |
|------|------|
| Сбор путевых условий | Этап анализа потока управления компилятора, аннотирование каждого базового блока множеством гипотез |
| Генерация условий верификации | При встрече с ограничением типа, требующим проверки, объединение путевых условий + информация о типе аргумента |
| Вычисление в конвейере доказательства | Ядро компилятора → SMT-ускорение → получение Proved / Disproved / Unproven |
| Результат | `Proved` → пройдено; `Disproved` → ошибка компиляции + контрпример; `Unproven` → ошибка компиляции + недоказанная пропозиция (программист может предоставить функцию-доказательство) |

Новых компонентов нет. Особых правил нет. Путевые условия — это фоновые знания конвейера доказательства, разделяющие один конвейер и одну систему бюджетов с равенством типов и ограничениями заимствования.

### 4. Конвейер доказательства времени компиляции

Все проверки времени компиляции разделяют один конвейер. Центральная операция конвейера — **проверка типов** — проверка того, равен ли тип терма доказательства доказываемой пропозиции. Всё — проверка типов.

```
Во время компиляции встретилось выражение Bool, требующее вычисления (т.е. требуется сконструировать терм доказательства)
        │
        ├── Равенство типов (T1 == T2)
        │   → Прямое решение компилятора (структурная эквивалентность)
        │
        ├── Условие конфликта токенов (!conflicting(tokens))
        │   → Потоко-чувствительный анализ живучести (отслеживание атрибутов Dup/Linear)
        │
        ├── Редукция зависимых типов (n + m упрощается)
        │   → Система переписывания термов времени компиляции (βδι-редукция)
        │
        ├── Предикат времени компиляции (x > 0, forall...)
        │   → Собственно компилятор + модуль SMT-ускорения
        │
        └── Импликация логики Хоара (P ⇒ Q)
            → Компилятор + модуль SMT-ускорения
                    │
                    ▼
             ┌──────────┐
             │ Proved   │  → Компиляция пройдена
             │ Disproved│  → Ошибка компиляции + контрпример
             │ Unproven │  → Ошибка компиляции + недоказанная пропозиция
             └────┬─────┘
                  │
                  ▼
         Программист пишет функцию-доказательство (код на YaoXiang)
                  │
                  ▼
         Проверщик типов верифицирует ──→ Proved ──→ Компиляция пройдена
                  │
                  ▼
            Верификация не пройдена → Ошибка компиляции: «Доказательство не устанавливается»
```

#### 4.1 Результат доказательства: трёхзначная алгебра

Вычисление времени компиляции возвращает три результата — это неизбежное следствие проблемы остановки и естественное разбиение теории доказательств:

```
eval_compile_time : BoolExpr → Proved | Disproved(Model) | Unproven
```

- **Proved** → остановка, терм доказательства сконструирован, проверка типов пройдена. Компиляция продолжается.
- **Disproved(M)** → остановка, существует контрпример M. Ошибка компиляции + контрпример + позиция в исходном коде.
- **Unproven** → в пределах заданного ресурсного лимита терм доказательства не сконструирован. Ошибка компиляции + недоказанная пропозиция + отчёт о потреблении бюджета.

**Unproven ≠ False.** Когда компилятор говорит «я не могу это доказать», это не эквивалентно ложности пропозиции — просто это за пределами текущих возможностей автоматического доказательства. Это честность, а не дефект.

Жёсткий лимит бюджета — инженерное решение проблемы остановки. Без ручек — давать ручку означает спрашивать пользователя «как вы думаете, ваша программа остановится?», пользователь не знает, и компилятор тоже.

#### 4.2 После Unproven: программист пишет доказательство

Когда компилятор возвращает Unproven, программист может написать **функцию-доказательство** — это функция на YaoXiang, чей возвращаемый тип равен доказываемой пропозиции. Проверщик типов верифицирует эту функцию — тем же механизмом, что и верификация `add(a, b): Int`.

```
Пропозиция  = Тип
Доказательство  = Программа (значение этого типа)
Верификация  = Проверка типов (единственный корень доверия)
```

SMT-решатель — не отдельный доверенный рубеж — это **ускоряющий модуль проверщика типов**. SMT помогает найти доказательство, но верифицирует доказательство всегда проверщик типов. Когда SMT возвращает `unsat`, компилятор реконструирует его результат в терм доказательства, верифицируемый проверщиком типов. Если реконструкция не удалась (шаги рассуждения SMT выходят за пределы правил вывода ядра компилятора), происходит откат к Unproven — программист может вручную написать функцию-доказательство.

```yaoxiang
# Пропозиция: уточнённое свойство, которое компилятор не может доказать автоматически
FirstIsMin: (T: Ord, arr: Sorted(T)) -> Type = {
    forall i in 0..arr.len: arr[0] <= arr[i]
}

# Доказательство: программист пишет функцию, чей возвращаемый тип — вышеприведённая пропозиция
# Проверщик типов верифицирует эту функцию — полностью аналогично верификации add(a,b): Int
first_is_min: (T: Ord, arr: Sorted(T)) -> FirstIsMin(T, arr) = {
    # Компилятор здесь проверяет: тип тела функции = FirstIsMin(T, arr)
    ...
}
```

ИИ не нужен, экспорт в Coq не нужен, новые концепции не нужны. **Свойства, которые компилятор не может доказать автоматически → программист пишет доказательство в коде на YaoXiang → проверщик типов верифицирует.** Весь процесс — плавный градиент — компилятор берёт на себя простые доказательства, оставляя мозг для сложных.

#### 4.3 Слоистые зависимости внутри конвейера

Вышеописанные вычислители разделяют один интерфейс, но имеют порядок вычисления. Равенство типов — необходимое условие для всего последующего анализа; проверка владения/токенов зависит от информации о типах; верификация уточняющих предикатов зависит от результатов двух предыдущих слоёв. Компилятор вычисляет по слоям; выражения, не прошедшие нижний слой, не поступают на верхний — это избегает траты бюджета решения на программу с ошибками типов.

```
Порядок вычисления (один конвейер, слоистая диспетчеризация)
├── Слой 0: Равенство типов (T1 == T2)
│   └── Структурная унификация → при неудаче дальнейший анализ бессмыслен, прямой возврат Disproved
├── Слой 1: Конфликты владения/токенов
│   └── Потоко-чувствительный анализ живучести → при неудаче безопасность памяти не устанавливается, прямой возврат Disproved
└── Слой 2: Уточняющие предикаты / импликации Хоара
    └── Ядро компилятора → SMT-ускорение → получение Proved / Disproved / Unproven
```

Каждый слой по-прежнему возвращает `Proved/Disproved/Unproven`, разделяя один интерфейс и одну систему бюджета.

### 5. Унификация функций трёх слоёв

| Слой | Время выполнения | Вход | Выход | Пример |
|------|----------|------|------|------|
| Функции-значения | Время выполнения | Значения | Значения | `add: (a: Int, b: Int) -> Int = a + b` |
| Конструкторы типов | Время компиляции | Типы/значения | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| Предикаты времени компиляции | Время компиляции | Значения | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

Все используют одинаковый синтаксис `name: type = value`. Предикаты времени компиляции и конструкторы типов идут по одному конвейеру доказательства — `{}` — это пространство доказательств.

### 6. Циклы: генерация условий верификации Флойда—Хоара

Циклам не нужны отдельные аннотации `: Invariant(...)` или `: decreases(...)`. Уточняющие предикатные аннотации типов на переменных определяют утверждения в стиле Флойда—Хоара — компилятор генерирует условия верификации из аннотаций типов, а конвейер доказательства проверяет, сохраняет ли каждое присваивание тип.

Центральный механизм: каждой операции присваивания соответствует тройка Хоара `{P} x := e {Q}`, условие верификации — `P ⇒ Q[e/x]`. Компилятор генерирует одно условие верификации для тела цикла — после того как конвейер доказательства устанавливает индуктивный шаг, все итерации покрываются автоматически.

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # Аннотация ссылается на i; при инициализации i=0, проверка: 0 == sum(arr[0..0]) → True
    mut i: UpTo(arr.len) = 0     # Проверка: 0 <= 0 <= arr.len → True
    while i < arr.len {
        # Компилятор генерирует одно VC для тела цикла. Предпосылка: s удовлетворяет SumUpTo(arr, i), i удовлетворяет UpTo(arr.len).
        #
        # s += arr[i]:
        #   Обязательство верификации: s_new удовлетворяет SumUpTo(arr, i) (текущий i не изменён)
        #   Подставляя s_new = s_old + arr[i]:
        #     Требуется s_old + arr[i] == sum(arr[0..i+1])
        #     Из индуктивного предположения s_old == sum(arr[0..i]), прибавив arr[i] к обеим частям:
        #     sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   Компилятор + SMT: линейная арифметика, миллисекунды → Proved
        #
        # i += 1:
        #   i изменяется → в графе зависимостей аннотация типа s ссылается на i → срабатывает реверификация
        #   Новая цель верификации: s удовлетворяет SumUpTo(arr, i_new)
        #   Т.е. s == sum(arr[0..i_new]), гарантируется предыдущим шагом → Proved
        s += arr[i]
        i += 1
    }
    return s  # В этот момент s: SumUpTo(arr, arr.len), т.е. s == sum(arr[0..arr.len])
}
```

Инвариант цикла — это аннотация типа на переменной — программист пишет тип, компилятор проверяет индуктивный шаг. Компилятору не нужно «обнаруживать» инвариант и не нужно «автоматически проводить индукцию» — он разбивает индуктивное доказательство на локальные условия верификации для каждой операции присваивания, передавая их в конвейер доказательства для divide and conquer.

#### 6.1 Отслеживание зависимостей: зависимые типы на изменяемых переменных

Предпосылка вышеописанного механизма: компилятор знает, что аннотация типа `s`, а именно `SumUpTo(arr, i)`, ссылается на `i` — когда `i` изменяется, ограничение типа `s` также изменяется. Это требует от компилятора поддерживать **граф типовых зависимостей между переменными**.

**Структура данных**:

```
TypeDepGraph: Map<VarName, Set<VarName>>
# Ключ — зависимая переменная, значение — множество переменных, на которые ссылается аннотация типа
# Пример: { i: {s}, j: {s, t}, ... }
```

**Построение**: при обработке `mut v: Pred(... x ...) = init` проверщик типов разрешает свободные переменные в аргументах `Pred(...)`. Если в аргументах есть ссылки на другие изменяемые переменные `x` в текущей области видимости, в графе зависимостей записывается `x → v`.

**Срабатывание**: когда зависимая переменная `x` получает присваивание, компилятор:
1. Находит в графе зависимостей все переменные, зависящие от `x`: `{v₁, v₂, ...}`
2. Для каждого `v` генерирует условие верификации: удовлетворяет ли текущее значение `v` обновлённому типу `Pred(... x_new ...)`
3. Отправляет VC в конвейер доказательства

**Чувствительность к порядку присваиваний**: отслеживание зависимостей естественным образом форсирует правильный порядок присваиваний. Для `SumUpTo(arr, i)`:

```yaoxiang
# Правильный порядок
s += arr[i]   # s_new удовлетворяет SumUpTo(arr, i+1)
i += 1        # i изменяется → реверифицируется s удовлетворяет SumUpTo(arr, i_new) → True

# Неправильный порядок — компилятор отвергает
i += 1        # i изменяется → реверифицируется s удовлетворяет SumUpTo(arr, i_new)
              # s ещё не обновлён, s_old == sum(arr[0..i_old]) ≠ sum(arr[0..i_new])
              # → Ошибка компиляции: переменная s не удовлетворяет типу SumUpTo(arr, i_new)
s += arr[i]   # Недостижимо
```

**Комбинированные зависимости**: переменная может зависеть от нескольких переменных. Аннотация типа `{ v: Int; v == x + y }` одновременно зависит от `x` и `y` — изменение любой из них запускает реверификацию.

**Связь с конвейером доказательства**: отслеживание зависимостей — триггер генерации VC, а не самостоятельный механизм верификации. Оно отвечает на вопрос «когда нужно генерировать VC» — конвейер доказательства отвечает на вопрос «выполняется ли VC».

### 7. Проверка завершаемости

Полностью автоматическая во время компиляции. Циклы, которые компилятор может доказать, проходят; циклы, которые компилятор не может доказать, сразу вызывают ошибку компиляции — программист обязан сделать цикл анализируемым для автоматической проверки завершаемости. Полуавтоматическим аннотациям не оставлено лазеек.

#### 6.1 Принципы проектирования

Компилятор автоматически извлекает информацию, необходимую для доказательства завершаемости, из двух источников:

1. **Аннотации типов переменных**: ограничения границ в уточнённых типах (например, `UpTo(n)` даёт верхнюю границу `n` и нижнюю `0`)
2. **Операции в теле цикла**: операции, применяемые к переменным на каждой итерации

Компилятор по приоритету пробует четыре стратегии синтеза метрик, останавливаясь на первой успешной.

#### 6.2 Стратегия 1: автоматический синтез линейной ранговой функции

Когда переменная имеет аннотацию линейной границы, компилятор перечисляет кандидаты линейных метрик и верифицирует их с помощью SMT.

```
Вход:
  Переменные v₁: UpTo(u₁), v₂: UpTo(u₂), ... (переменные с границами)
  Условие цикла cond
  Множество присваиваний в теле цикла

Алгоритм:
  1. Извлечь границы каждой переменной из аннотаций типов: [low_i, high_i]
  2. Перечислить кандидаты метрик: v_i, u_i - v_i, v_i - v_j, и т.д. — линейные комбинации
  3. Для каждого кандидата m:
     - SMT-верификация m ≥ 0 (из границ типа)
     - Для каждого пути выполнения в теле цикла SMT-верификация m' < m (строгое убывание)
  4. Найти подходящую линейную комбинацию → завершаемость доказана
```

Покрытие: циклы, в которых любая переменная получает линейное выражение (`v = a·v + b`) и имеет аннотацию ограниченного типа. Включает `i += const`, `i -= const`, а также сокращение интервалов в стиле бинарного поиска:

```yaoxiang
# Бинарный поиск: low = mid + 1 или high = mid
# Метрика high - low строго убывает на обоих путях
binary_search: (arr: Sorted(Int, arr), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 Стратегия 2: счётчик нарушений предиката — автоматическое извлечение метрики из целевого типа <span style="color:orange">【Экспериментальная стратегия】</span>

> ⚠️ **Текущий статус: экспериментальная стратегия, на этапе 3 будет решено о включении по результатам проверки реализуемости.**
> Эта стратегия работает для операций обмена соседних элементов (сортировка пузырьком, сортировка вставками) и не может автоматически доказать завершаемость для не соседних операций (разбиение быстрой сортировки, просеивание вниз в пирамидальной сортировке).
> Границы покрытия см. в таблице ниже. Если на этапе 3 будет установлена нереализуемость, эта стратегия будет удалена или переведена в будущую работу.

Ключевое наблюдение: **спецификации, написанные пользователем, — это материал для рассуждений компилятора.** Компилятору не нужно встраивать «что такое сортировка» — он читает определение `Sorted` и автоматически извлекает из него метрику.

```
Вход:
  Целевой тип: Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  Операции в теле цикла: обмен соседних элементов

Алгоритм:
  1. Разобрать определение предиката: forall i in range: cond(i, arr)
  2. Автоматически сгенерировать метрику: violation_count = |{ i | ¬cond(i, arr) }|
  3. Проанализировать влияние операции на метрику:
     - Обмен соседних arr[j], arr[j+1] = arr[j+1], arr[j]
     - Затрагивает только пары с индексами j-1, j, j+1
     - Если arr[j] > arr[j+1] (нарушение предиката), после обмена пара удовлетворяет предикату
     - violation_count уменьшается минимум на 1
  4. Верхняя граница: n·(n-1)/2 (максимальное число соседних инверсий), нижняя: 0
  → Завершаемость доказана
```

**Текущее покрытие**:

| Алгоритм | Шаблон операции | Стратегия 2 доказуема? | Причина |
|------|----------|:---:|------|
| Сортировка пузырьком | Обмен соседних | ✅ | violation_count строго убывает с каждым обменом |
| Сортировка вставками | Смещение соседних | ✅ | Каждое смещение устраняет одну нарушающую пару |
| Сортировка выбором | Обмен не соседних | ❌ | Одиночный обмен может увеличить violation_count |
| Быстрая сортировка | Разбиение partition | ❌ | Обмен не соседних, не гарантируется монотонное убывание |
| Пирамидальная сортировка | sift-down | ❌ | Древовидная операция, violation_count не монотонна |

**Дополняющие стратегии**: для быстрой сортировки сокращение интервала `low < high` покрывается стратегией 1 (линейная ранговая функция) — внешняя рекурсия partition, каждый интервал уменьшается вдвое. Стратегии 1 и 2 дополняют друг друга, завершаемость большинства реальных алгоритмов доказуема одной из них. Однако обобщение стратегии 2 (не соседние операции, древовидные операции) остаётся открытой проблемой.

```yaoxiang
sort: (arr: Array(Int)) -> (result: Sorted(result)) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 Стратегия 3: шаблоны ограниченного возрастания/убывания

`v += const` (const > 0), переменная имеет аннотацию верхней границы → метрика `upper_bound - v` убывает на `const` каждую итерацию, нижняя граница 0. Это вырожденный случай стратегии 1, компилятор обрабатывает его в первую очередь.

#### 6.5 Стратегия 4: шаблон метрики мультипликативного масштабирования

`v *= const` (const > 1), переменная имеет аннотации верхней и нижней границ. Компилятор имеет встроенный шаблон логарифмической метрики `ceil(log_const(upper/v))`, при каждом умножении на const метрика убывает на 1.

```yaoxiang
mut i: Positive(i) = 1
while i < n {
    # Компилятор автоматически выводит: метрика ceil(log₂(n/i)), при каждом умножении на 2 метрика убывает на 1
    i *= 2
}
```

#### 6.6 Разделение завершаемости и корректности

Доказательство завершаемости и доказательство корректности независимы:
- **Завершаемость**: четыре вышеописанные стратегии автоматически доказывают, что цикл завершается за конечное число шагов
- **Корректность**: продвигается ли тело цикла к целевому типу, проверяется конвейером доказательства времени компиляции через условия верификации

Оба проходят → компиляция пройдена. Завершаемость доказана, но корректность не прошла → ошибка компиляции + контрпример. Корректность доказана, но завершаемость не доказуема → ошибка компиляции с указанием переменной или операции, которые не удаётся проанализировать. Оба не прошли → ошибка компиляции с раздельным отчётом по обеим причинам.

#### 6.7 Проверка завершаемости рекурсивных функций

Для рекурсивных функций, которые требуется вычислить во время компиляции, компилятор проверяет убывание аргументов:

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # Компилятор анализирует: n-1 < n → убывает → завершается
}

# Использование во время компиляции — компилятор гарантирует завершение factorial во время компиляции
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120, вычисляется во время компиляции
```

| Сценарий | Поведение |
|------|------|
| Компилятор может проанализировать рекурсивное убывание (например, `n-1`) | Вычисление во время компиляции |
| Не убывает / невозможно установить убывание | Ошибка компиляции |
| Вызов во время выполнения (не в позиции типа) | Проверка завершаемости не требуется |

#### 6.8 Жёсткая граница

`i = f(i)`, где `f` необратима, не замкнута, не сохраняет никакой монотонности — математически невозможно автоматически доказать завершаемость. Ошибка компиляции:

> Этот цикл не может быть автоматически доказан как завершающийся. Переменная цикла зависит от неанализируемой функции `f`. Пожалуйста, используйте шаблон итерации, анализируемый компилятором.

Это не провал компилятора. Всё, что не может быть статически доказано как безопасное, не должно проходить компиляцию.

### 8. SMT-решатель: ускоряющий модуль проверщика типов

SMT-решатель в традиционных языках — внешний инструмент (например, F\* вызывает Z3, Dafny вызывает Z3). В YaoXiang это **ускоряющий модуль проверщика типов** — вызывается только тогда, когда ядро компилятора не может принять решение напрямую. SMT помогает найти доказательство, но верифицирует доказательство проверщик типов.

**Модель доверия**: проверщик типов — единственный корень доверия. SMT-решатель — ускоряющий модуль — он помогает найти доказательство, но SMT — не отдельный доверенный рубеж. Компилятор доверяет результату `unsat` от Z3 (как и в подходе F\*/Dafny — вероятность ошибки Z3 ниже частоты багов самого компилятора, это инженерно-прагматичный выбор). Реальная ненадёжность контролируется на уровне трансляции в SMT — если в трансляции есть баг, компилятор обнажит его в других тестах.

**Интерфейс**: внутренняя трансляция компилятора — в стандартный формат SMT-LIB 2.6, а не привязка к API конкретного решателя. SMT-LIB — стандарт ISO, Z3, CVC5, MathSAT, Yices — все поддерживают его нативно.

**Бэкенд по умолчанию**: Z3 (лицензия MIT, наиболее широкая документация и валидация сообществом). CVC5 в качестве SMT-LIB-совместимой альтернативы — пользователь может переключаться флагом компилятора.

Не делаем «абстрактный слой универсального решателя» — SMT-LIB и есть абстрактный слой. В будущем, если CVC5 добьётся прорыва в определённой теории, переключение требует только замены бинарника, без изменения кода компилятора.

```
Выражение Bool времени компиляции
        │
        ├── Ядро компилятора может решить напрямую (структурная эквивалентность,
        │   простая арифметика, тривиальные формулы после свёртки констант)
        │   → Прямой возврат Proved / Disproved
        │
        └── Ядро компилятора не может решить напрямую (кванторы, символьные переменные)
            → Предварительная редукция зависимых типов (factorial(5) → 120)
            → Трансляция в формат SMT-LIB
            → Отправка в Z3/CVC5 (с ограничением бюджета)
            → Возврат: unsat → Proved  │  sat + модель → Disproved  │  unknown → Unproven
```

**Бюджет решения — жёсткое ограничение, как глубина стека**:

| Измерение бюджета | Значение по умолчанию | Пояснение |
|----------|--------|------|
| Число шагов решения | 10 000 | Z3 для линейной арифметики обычно укладывается в сотни шагов. 10 000 шагов покрывают 99% реальных предикатов. |
| Время | 100 мс | Один предикат более 100 мс = пользователь пишет программу времени компиляции, а не аннотацию типа. 100 мс × 50 предикатов = 5 секунд верхней границы времени компиляции. |
| Глубина инстанциации кванторов | 3 | Три уровня вложенности кванторов покрывают реальные шаблоны. Больше трёх — скорее всего, пишется логическая задача. |

Превышение бюджета → Unproven, ошибка компиляции + позиция предиката + объём потребления. Без деградации, без проверок времени выполнения, без тихого прохода.

**Почему это практически реализуемо**: 95% реальных предикатов в инженерии — линейная арифметика — `x > 0`, `arr.len > 0`, `0 <= idx < arr.len` — всё в пределах разрешимого фрагмента, SMT-решатель возвращается за миллисекунды. Для редких сложных предикатов, превышающих бюджет, программист пишет функцию-доказательство.

Зависимые типы перед вызовом SMT проходят слой предварительной редукции: `factorial(5)` напрямую вычисляется во время компиляции в `120`, `append([1,2], [3])` — в `[1,2,3]`. Эти детерминированные вычисления значений не потребляют бюджет SMT.

Программисту не нужно знать о существовании SMT. Ментальная модель: **компилятор может доказать — проходит, не может — ошибка — если компилятор не умеет, можно написать функцию-доказательство**.

### 9. Композиция предикатов времени компиляции

Предикат времени компиляции — это функция, возвращающая Type; композиция естественно реализуется через композицию функций:

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(arr)
}
```

### 10. Примеры кода

#### 9.1 Безопасность деления

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b

result = divide(10, 2)   # ✅ Компилятор проверяет Positive(2) = { 2 > 0 } → True
# result = divide(10, 0)  # ❌ Компилятор проверяет Positive(0) = { 0 > 0 } → False
```

#### 9.2 Безопасность доступа к массиву

```yaoxiang
InBounds: (idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (arr: Array(T), idx: InBounds(idx, arr)) -> T = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ Компилятор проверяет InBounds(1, arr) = { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ Компилятор проверяет InBounds(5, arr) = { 0 <= 5 && 5 < 3 } → False
```

#### 9.3 Корректность сортировки

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T))) -> (result: Sorted(T, result)) = {
    result = arr.clone()
    # ... реализация алгоритма сортировки ...
    return result
}
```

#### 9.4 Цикл: генерация VC компилятором

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```

## Детальное проектирование

### Изменения синтаксиса

| Ранее (RFC-022) | Далее (этот RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | Предикат времени компиляции как тип параметра `(b: Positive(b))` |
| `//! ensures: ExistsMax(result, arr)` | Тип возврата использует формальный параметр возвращаемого значения `-> (result: IsMax(T, arr, result))` |
| `/*! invariant: ... !*/` | Уточняющая предикатная аннотация типа на переменной — инвариант Флойда—Хоара |
| `//! decreases: n` | Полностью автоматический вывод функций-мер компилятором |
| Спецификации — это аннотации | Спецификации — это система типов |

### Синтаксис

**У предикатов времени компиляции нет новых ключевых слов.** `{}` — пространство доказательств, полностью согласующееся с существующим синтаксисом определения типов. Предикат времени компиляции — это функция, возвращающая Type — `name: (params) -> Type = { утверждения }`. Использование — вызов функции — `Positive(b)`, `IsMax(T, arr, result)`.

```bnf
# Предикат времени компиляции = функция, возвращающая Type, внутри {} — утверждения, верифицируемые компилятором
# Используется существующий синтаксис функций/типов, новые правила BNF не требуются
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

**Новое синтаксическое понятие: формальный параметр возвращаемого значения** — в `-> (name: Type)` `name` — формальный параметр возвращаемого значения.

Формальный параметр возвращаемого значения — это **единственное синтаксическое понятие**, вводимое YaoXiang в существующий синтаксис функций. Его семантика:
- Значение `name` задаётся оператором `return`
- `name` существует только в сигнатуре типа, на него ссылаются предикаты-постусловия (например, `-> (result: IsMax(T, arr, result))`)
- `name` не входит в область видимости тела функции и не появляется у вызывающего
- Формальный параметр возвращаемого значения **необязателен** — при отсутствии постусловия сигнатура полностью совпадает с обычной функцией (`-> Int`), не внося никакой дополнительной нагрузки

Причина его введения: постусловиям нужно ссылаться на «значение, которое функция собирается вернуть». Без формального параметра возвращаемого значения компилятор может позволить предикату ссылаться на возвращаемое значение только через специальные правила (например, неявная переменная `$result` или `__retval__`). Формальный параметр возвращаемого значения делает эту ссылку явной — это просто формальный параметр, только значение задаётся `return`, а не вызывающим.

**Функция-доказательство** — не новое понятие — это функция на YaoXiang, чей возвращаемый тип — доказываемая пропозиция. Когда компилятор возвращает Unproven, программист предоставляет функцию-доказательство; проверщик типов верифицирует её ровно тем же способом, что и верификацию возвращаемого типа любой функции. Не нужны новый синтаксис, новые ключевые слова, новые правила.

### Влияние на систему типов

- **Вселенная типов**: предикаты времени компиляции находятся на уровне Type₂ — функции, принимающие значения и возвращающие Type, на том же уровне, что и конструкторы типов
- **Взаимодействие с дженериками**: предикаты времени компиляции могут иметь параметры-дженерики, например `NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **Взаимодействие с владением**: выражения в предикатах времени компиляции подчиняются правилам владения, могут только читать, не писать
- **Вывод типов**: аргументы предикатов времени компиляции участвуют в выводе типов по Хиндли—Милнеру

### Представление во время выполнения

Предикаты времени компиляции полностью **стираются** во время выполнения. `Positive: (x: Int) -> Type = { x > 0 }` — параметр `b: Positive(b)` во время выполнения представляется как `Int`. Уточняющее условие `{ x > 0 }` верифицируется конвейером доказательства во время компиляции; после верификации не остаётся никаких следов во время выполнения.

- `Positive(5)` → во время выполнения представлено как `Int(5)`, уточняющее условие `{ 5 > 0 }` пройдено, стёрто
- `SumUpTo(arr, 0)` → во время выполнения представлено как `Int(0)`, равенство `0 == sum(arr[0..0])` пройдено, стёрто
- Помещение предиката времени компиляции в позицию типа (например, `f(x: Positive(x))`) не порождает никакого оборачивающего типа, не выделяет дополнительную память, не вставляет проверок времени выполнения

Стирание дженериков и стирание уточнённых типов — один и тот же принцип: оба являются функциями времени компиляции, полностью исчезающими после вычисления во время компиляции. Нулевые накладные расходы времени выполнения у предикатов времени компиляции в Release Build — прямое следствие «предикаты — это функции».

**Ограничения взаимодействия с `ref`**: предикаты времени компиляции могут ссылаться только на неизменяемые заимствования или значения с переданным владением. Предикат времени компиляции, ссылающийся на значение по изменяемому заимствованию, не может гарантировать компилятору сохранение результата верификации во время выполнения — такое использование сразу вызывает ошибку компиляции.

### Изменения в компиляторе

1. **Парсер**: предикаты времени компиляции используют стандартный синтаксис функций, дополнительные правила разбора не нужны
2. **Конвейер доказательства времени компиляции**: унифицированный интерфейс возврата Proved/Disproved/Unproven, автоматический выбор стратегии
3. **Модуль SMT-ускорения**: слой трансляции в SMT-LIB 2.6, бэкенд по умолчанию Z3, альтернатива CVC5
4. **Ядро проверщика типов**: реализация правил вывода — структурная эквивалентность, βδι-редукция, введение/устранение кванторов всеобщности. Это единственный корень доверия; SMT и программистские доказательства проходят верификацию здесь
5. **Генерация условий верификации**: исчисление WP/SP + обязательства доказательства инвариантов цикла
6. **Отчёты об ошибках**: форматирование контрпримеров + отчёт о недоказанных пропозициях + привязка к позициям в исходном коде

### Обратная совместимость

- ✅ Код, не использующий предикаты времени компиляции, остаётся полностью без изменений
- ✅ Нулевые накладные расходы времени выполнения у предикатов времени компиляции в Release Build
- ⚠️ Синтаксис `//!` из RFC-022 больше не поддерживается — но 022 никогда не был реализован, бремя миграции отсутствует

## Компромиссы

### Преимущества

- **Полная реализация изоморфизма Карри—Ховарда**: тип — пропозиция, программа — доказательство, `name: Proposition = Proof`
- **Единство**: предикаты времени компиляции и обычные функции используют полностью идентичный синтаксис, без концептуального разделения
- **Прозрачность SMT**: программисту не нужно знать о существовании SMT, ментальная модель совпадает с проверкой типов
- **Постепенное внедрение**: можно начать с одного предиката времени компиляции, постепенно расширяя покрытие
- **Нулевые накладные расходы времени выполнения**: все вычисления завершаются во время компиляции, Release Build не вставляет никаких утверждений времени выполнения

### Недостатки

- **Время компиляции**: SMT-решение увеличивает время компиляции, но жёсткое ограничение бюджета гарантирует контролируемый верхний предел
- **Границы автоматического доказательства**: сложные предикаты, выходящие за пределы линейной арифметики первого порядка, могут потребовать от программиста написания функций-доказательств. Это не дефект языка — это неизбежное следствие проблемы остановки. Компилятор честно сообщает Unproven, а не ложно объявляет True/False
- **Кривая обучения**: написание эффективных предикатов времени компиляции и функций-доказательств требует понимания базовой интуиции изоморфизма Карри—Ховарда
- **Сложность реализации**: унификация конвейера доказательства времени компиляции требует тщательного проектирования

### Снижение рисков

- Жёсткое ограничение бюджета SMT (шаги 10 000 / время 100 мс / глубина инстанциации 3), превышение → Unproven
- Предварительная редукция зависимых типов: детерминированные вычисления значений проглатываются первыми, SMT кусает только недетерминированную часть
- Unproven — не тупик: программист может написать функцию-доказательство, проверщик типов верифицирует — согласовано с верификацией любого возвращаемого типа функции
- Инкрементальная верификация: верифицируются только изменённые модули
- Понятные сообщения об ошибках + отображение контрпримеров + отчёт о потреблении бюджета + недоказанные пропозиции + предложения (если компилятор может их дать)

| Подход                             | Почему не выбран                           |
| -------------------------------- | -------------------------------------- |
| RFC-022: спецификации как комментарии `//!` | Разделение спецификаций и типов, нарушение изоморфизма Карри—Ховарда |
| Отдельные файлы спецификаций (например, CVL) | Отделение спецификаций от кода, увеличение стоимости сопровождения           |
| Только утверждения времени выполнения                     | Невозможно статически гарантировать корректность                     |
| Внешний инструмент доказательства (например, Coq)           | Отрезан от компилятора, требует отдельного языка доказательств и доверенного рубежа. Выбор YaoXiang: доказательство — это код на YaoXiang, проверщик типов — единственный корень доверия |
| **Этот подход: предикаты времени компиляции как полноправные граждане** | ✅                                     |

## Стратегия реализации

### Разбиение на этапы

| Этап       | Содержание                                                                 |
| ---------- | -------------------------------------------------------------------- |
| **Этап 1** | Ядро компилятора: структурная эквивалентность + βδι-редукция + введение/устранение кванторов всеобщности. Поддержка простых арифметических предикатов (`x > 0`, `arr.len > 0`) |
| **Этап 2** | Слой трансляции в SMT-LIB + интеграция Z3/CVC5. Конвейер возвращает Proved/Disproved/Unproven. При Unproven поддержка функций-доказательств от программиста |
| **Этап 3** | Генерация VC для инвариантов цикла + проверка завершаемости (линейная ранговая функция + счётчик нарушений предиката + ограниченные шаблоны + контроль комбинаторного взрыва) |
| **Этап 4** | Инкрементальная верификация + кэширование + поддержка IDE |

## Открытые вопросы

- [x] **Выбор SMT-решателя**: по умолчанию Z3 (лицензия MIT, наиболее широкая валидация). CVC5 в качестве SMT-LIB-совместимой альтернативы, переключение флагом компилятора. Внутренняя трансляция компилятора нацелена на стандартный формат SMT-LIB 2.6 — SMT-LIB и есть абстрактный слой, собственный универсальный интерфейс решателя не делается.
- [x] **Конкретные значения бюджета решения**: шаги 10 000 / время 100 мс / глубина инстанциации кванторов 3. Зафиксировано внутри компилятора, без ручек. Если реальные сценарии использования (не «пользователь ошибся») докажут, что недостаточно, тогда корректируем.
- [x] **Диапазон поддержки кванторов**: язык не ограничивает порядок кванторов. Предикаты времени компиляции принимают параметры Type — Type включает типы функций — поэтому кванторы высшего порядка — естественное следствие системы типов, специальный синтаксис не нужен. SMT-решатель может автоматически разрешать кванторы первого порядка (forall/exists, поддержка чередующейся вложенности, ограничена бюджетной глубиной 3). Кванторы высшего порядка: SMT возвращает Unproven, компилятор подсказывает «этот предикат за пределами автоматического доказательства, предоставьте функцию-доказательство». Программист пишет функцию на YaoXiang, чей возвращаемый тип равен пропозиции — проверщик типов верифицирует эту функцию. Внешний экспорт не нужен, ИИ не нужен, интерактивный режим доказательства не нужен. Всё — код на YaoXiang, всё верифицируется проверщиком типов.
- [x] **Форматирование контрпримеров**: имена переменных исходного кода напрямую используются как имена переменных SMT (с префиксом модуля для избежания конфликтов). При возврате модели Z3 — обратный поиск по именам переменных. Формат вывода: имя переменной = конкретное значение + позиция в исходном коде + позиция определения предиката. Сложный слой отображения не делается.
- [x] ~~**Взаимодействие предикатов времени компиляции с умными указателями `ref`?**~~ → Решено: предикаты времени компиляции допускают только неизменяемые заимствования или значения с переданным владением. Значения по изменяемому заимствованию не могут появляться в предикатах времени компиляции.
- [x] **Расширение метрики счётчика нарушений `forall` на не соседние операции?** → Не расширяется. Текущее покрытие (обмен соседних, смещение соседних) дополняется стратегией 1 (линейная ранговая функция) — сокращение внешнего интервала быстрой сортировки покрывается стратегией 1, пирамидальная сортировка — стратегией 1 (шаблон индексации массива). Циклы, завершаемость которых не доказуема ни одной стратегией, компилятор сразу отвергает — это философия жёсткой безопасности, а не дефект. Если в будущем появится реальный сценарий (не академическая конструкция), в котором ни одна из четырёх стратегий не работает, обсудим заново.
- [x] **Комбинаторный взрыв перечисления линейных ранговых функций**: верхний предел перечисления кандидатов — 3 переменные с границами. При ≤3 перечисляются все линейные комбинации, каждая верифицируется SMT. При >3 пробуются только однопеременные метрики (`v_i`, `u_i - v_i`), при неудаче сразу ошибка компиляции — с подсказкой программисту «цикл имеет >3 переменных с границами, компилятор не может автоматически синтезировать многопеременную метрику». Это не инженерный компромисс — это принуждение программиста к более простым циклам.
