## Резюме

Настоящий документ предлагает ввести в YaoXiang **предикаты времени компиляции** как полноправных граждан языка, объединив всю статическую верификацию времени компиляции в единый **конвейер доказательств**. Предикат времени компиляции — это не внешняя аннотация спецификации, а сама функция. Функция, возвращающая Type, может использоваться в позиции типа; компилятор вызывает её во время компиляции и проверяет возвращаемое значение. Тип — это утверждение, вычисление во время компиляции — это доказательство.

**Ключевой тезис**: единственная задача проверки типов во время компиляции — построение и верификация доказательств. Равенство типов, конфликты токенов, редукция зависимых типов, вычисление предикатов времени компиляции, импликации логики Хоара — всё это различные проверки типов в конвейере доказательств времени компиляции, разделяющие один и тот же конвейер. SMT-решатель — это ускоряющий модуль проверщика типов, а не отдельный доверенный рубеж. Когда компилятор возвращает Unproven, программист пишет функцию на YaoXiang в качестве доказательства — проверщик типов верифицирует её точно так же, как проверяет возвращаемый тип любой функции. Всё — это код на YaoXiang, всё верифицируется проверщиком типов.

## Мотивация

### Почему RFC-022 устарел?

RFC-022 проектировал спецификации в форме комментариев `//!`:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← это аннотация, отдельная от типа
    //! ensures: ExistsMax(result, arr[0..n])   ← это аннотация, отдельная от типа
}
```

Это допускало фундаментальную ошибку изоморфизма Карри-Ховарда: **разделяло спецификации и типы на два слоя**. Аннотация — это не тип. Аннотация не участвует в проверке типов. Аннотация — это ментальная модель «внешнего инструмента».

В техническом описании сказано прямо:

> «Нет комментариев `//!`. Нет отдельного языка спецификаций. Всё внутри системы типов.»

### Текущие проблемы

- Аннотации `//!` из RFC-022 — это внешний синтаксис, отдельный от системы типов
- Типы спецификаций и обычные типы — это две разные системы, что приводит к концептуальной избыточности
- Разделение на Debug Build с верификацией и Release Build с игнорированием нарушает единство
- SMT-решатель в традиционном понимании позиционируется как внешний инструмент — YaoXiang встраивает его как ускоряющий модуль проверщика типов
- Проверка типов, верификация заимствований, проверка предикатов времени компиляции, раскрытие макросов идут разными путями

### Корректная ментальная модель

Проверку типов можно абстрагировать как функцию:

```
verify : Program → Proved | Disproved(Model) | Unproven
```

Все проверки времени компиляции — простое сопоставление типов, обнаружение конфликтов заимствований, верификация предикатов времени компиляции — являются подзадачами этой функции. Они разделяют один и тот же конвейер доказательств, отличаясь лишь сложностью термов доказательств и стратегиями их построения.

Когда компилятор возвращает Unproven, программист предоставляет функцию-доказательство — возвращаемый тип которой равен доказываемому утверждению. Проверщик типов верифицирует её. Это та же самая операция, что и обычная проверка типов.

## Предложение

### 1. `{}` — пространство доказательств: тип есть утверждение, верификация есть проверка типов

`{}` в YaoXiang — это пространство доказательств времени компиляции. Всё внутри — утверждения, и компилятор гарантирует, что каждый элемент равен True — либо доказывается автоматически, либо программист предоставляет функцию-доказательство.

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  компилятор гарантирует, что x — это Float, y — это Float

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  компилятор гарантирует, что data — это Array(T)
```

**Обобщённые типы — частный случай предикатов времени компиляции.**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          параметр в сигнатуре  в {} только утверждения
#          компилятор верифицирует x > 0 при вызове во время компиляции

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      параметр в сигнатуре  компилятор верифицирует type_of(T) == Type, type_of(data) == Array(T)
```

Один и тот же паттерн: `name: (params) -> Type = { утверждение }`. Компилятор не различает «утверждения о типах» и «утверждения о значениях» — это всё цели вычисления в конвейере доказательств.

**Инварианты цикла не нужно писать отдельно. Аннотация типа на переменной — это инвариант Флойда-Хоара.**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # аннотация ссылается на i — сообщает компилятору, что тип s зависит от i
    mut i: UpTo(arr.len) = 0     # при инициализации i=0, верификация: 0 == sum(arr[0..0]) → True
    while i < arr.len {
        s += arr[i]  # компилятор верифицирует: s_new == sum(arr[0..i+1])
        i += 1       # i изменён → срабатывает реверификация зависимости s: s удовлетворяет SumUpTo(arr, i_new)
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

Компилятор генерирует условие верификации для тела цикла один раз — индуктивное предположение (аннотация типа) → операция присваивания → удовлетворяет ли новое значение аннотации типа. После того как конвейер доказательств верифицирует индуктивный шаг, автоматически покрываются все итерации. Не нужен `: decreases`, не нужен `: Invariant`, не нужны доказательства по индукции — компилятор разбивает индукцию на локальные VC для каждого присваивания.

### 2. Предусловия/постусловия: предикаты времени компиляции на типах параметров и возвращаемого значения

Отказываемся от `//! requires`/`//! ensures` из RFC-022. Предикаты времени компиляции становятся аннотациями типов на параметрах или возвращаемом значении.

**Сторона параметра — это вызов функции.** Предикат времени компиляции — это функция, возвращающая Type; использование в позиции параметра — это её вызов — точно так же, как `factorial(5)`. Сторона возвращаемого значения вводит новое понятие: формальный параметр возвращаемого значения.

```yaoxiang
# Предусловие: явный вызов предиката времени компиляции в типе параметра
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b
#                       ^^^^^^^^^^  b — имя текущего формального параметра, передаётся в Positive
#                       компилятор извлекает значение фактического аргумента в точке вызова,
#                       подставляет в b, верифицирует Positive(фактический аргумент)
#                       Пример: divide(10, 2) → верифицирует Positive(2) = { 2 > 0 } → True
#                       Пример: divide(10, 0) → верифицирует Positive(0) = { 0 > 0 } → False → ошибка компиляции

# Постусловие: формальный параметр возвращаемого значения + предикат времени компиляции
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

NonEmpty: (arr: Array(T)) -> Type = { arr.len > 0 }

max: (T: Ord) -> ((arr: NonEmpty(arr))) -> (result: IsMax(T, arr, result)) = {
#                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                            result — формальный параметр возвращаемого значения,
#                                            значение предоставляется оператором return
#                                            компилятор подставляет возвращаемое значение в точке return
#                                            и верифицирует постусловие
    candidate = arr[0]
    for i in 1..arr.len {
        if arr[i] > candidate { candidate = arr[i] }
    }
    return candidate
}
```

**Ключевые правила**:

- **Сторона параметра**: `b: Positive(b)` — `b` — имя текущего формального параметра, передаётся в `Positive` как аргумент. Синтаксис вызова функции, ноль неявного.
- **Сторона возвращаемого значения**: `-> (result: IsMax(T, arr, result))` — `result` — формальный параметр возвращаемого значения, значение предоставляется оператором `return`. `result` существует только в сигнатуре типа, на него ссылается только предикат, он не входит в область видимости тела функции и не появляется у вызывающей стороны.
- **Формальный параметр возвращаемого значения необязателен**: при отсутствии постусловия не пишется, сигнатура полностью совпадает с обычной функцией (`-> Int`).
- **Единство**: параметры и формальные параметры возвращаемого значения — одно и то же понятие — `имя_параметра: вызов_предиката(имя_параметра)`, различие лишь в том, предоставляется значение вызывающей стороной или оператором `return`.

### 3. Распространение путевых условий: верификация значений времени выполнения во время компиляции

Когда предикат времени компиляции используется в позиции связывания, аргументы передаются программистом явно. Когда значения времени выполнения попадают в параметры уточнённого типа, компилятор выполняет верификацию через сбор путевых условий и импликации SMT — без необходимости явной передачи доказательства программистом.

#### 3.1 Явный вызов функции

Когда предикат времени компиляции используется в позиции связывания, аргументы передаются программистом явно — это вызов функции, ноль неявного.

`Positive: (x: Int) -> Type = { x > 0 }` — это конструктор предиката времени компиляции. Когда он появляется в позиции связывания (объявление параметра, объявление переменной, тип возвращаемого значения), программист явно передаёт имя уже связанной переменной:

```yaoxiang
b: Positive(b)
// b уже объявлен как текущий формальный параметр, Positive(b) — вызов функции
// После нормализации: b: { b > 0 }
```

Не требуется, чтобы компилятор неявно подставлял аргументы — `b: Positive(b)` — это точно такой же вызов функции, как `f(5)`. `b` связывается как имя параметра, и его аннотация типа `Positive(b)` ссылается на сам `b` — это стандартный паттерн зависимых типов, а не неявное правило раскрытия.

**Единство с `self` из RFC-010**: RFC-010 установил, что `self` — не ключевое слово, а лишь условное имя параметра («написать `p`, `this`, `x` — эффект полностью одинаков»). `b: Positive(b)` использует тот же механизм — имя параметра может быть упомянуто в аннотации типа. `self` появляется в позиции `self: Point`, `b` появляется в позиции `b: Positive(b)` — обе аннотации типа ссылаются на сам параметр. Различие лишь в сложности аннотации, механизм полностью одинаков — после связывания имени тип может зависеть от этого имени.

Тип возвращаемого значения также использует явный вызов функции:

```yaoxiang
Sorted: (arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (arr: Array(T)) -> (result: Sorted(result)) = { ... }
//                        ^^^^^^^^^^^^^^^^^^^^^^^
//                        result — формальный параметр возвращаемого значения,
//                        Sorted(result) — вызов функции
//                        компилятор подставляет возвращаемое значение в result в точке return,
//                        верифицирует Sorted(возвращаемое_значение)
```

Аналогично для объявлений локальных переменных:

```yaoxiang
let x: Positive(x) = 5
// x связывается с 5, Positive(5) → { 5 > 0 } → True → проходит

// let y: Positive(y) = 0
// y связывается с 0, Positive(0) → { 0 > 0 } → False → ошибка компиляции
```

#### 3.2 Сбор путевых условий

Когда значение времени выполнения появляется в условной ветви, компилятор автоматически собирает путевые условия, формируя **множество гипотез** для текущей области видимости. Эти гипотезы участвуют в верификации как фоновые знания для вычисления Bool во время компиляции.

```yaoxiang
if y > 0 {
    // компилятор автоматически получает гипотезу в этой ветви: { y > 0 }
    let result = divide(x, y)
    // Условие верификации: (y > 0) ⇒ (y > 0)
    // Конвейер доказательств устанавливает импликацию → Proved
} else {
    // Гипотеза в этой ветви: { !(y > 0) }
    // Если вызвать divide(x, y), условие верификации: !(y > 0) ⇒ y > 0
    // Конвейер доказательств устанавливает, что импликация неверна → Disproved
}
```

Это не зашитый в компилятор специальный паттерн — это естественное поведение конвейера доказательств времени компиляции. При каждой проверке типа в точке вызова в конвейер отправляется:

```
{фоновые_гипотезы} ⇒ {цель_верификации}
```

Конвейер доказательств устанавливает импликацию. Proved → проходит, Disproved → ошибка компиляции + контрпример, Unproven → ошибка компиляции + недоказанная proposition. Фоновые гипотезы берутся из путевых условий текущей точки программы.

#### 3.3 Стек гипотез

При анализе потока управления компилятор поддерживает множество гипотез для каждого базового блока:

- **if-guard**: `if y > 0` → истинная ветвь добавляет `y > 0`, ложная ветвь добавляет `!(y > 0)` (при наличии else)
- **match-паттерн**: `if let Some(v) = opt` → в ветви добавляется `opt == Some(v)`
- **Логическая конъюнкция**: `if x > 0 && y < 10` → в ветви добавляются `x > 0` и `y < 10`
- **Предусловие функции**: при вызове `divide(a, b)` свидетельство того, что `b` удовлетворяет `Positive`, должно поступать либо из текущих гипотез, либо из аннотации уточнённого типа самого фактического аргумента (если `b` аннотирован как `Positive`, его тип несёт `b > 0`)
- **Присваивание**: при `let z = y` существующие уточнённые условия `y` передаются в `z`

Все гипотезы поступают в конвейер доказательств времени компиляции. При переходе на путь SMT-ускорения они транслируются в фоновые утверждения SMT-LIB.

#### 3.4 Отсутствие статического свидетельства — ошибка компиляции

Если программист пишет напрямую:

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

В текущей точке программы нет гипотезы `y > 0`, и сам фактический аргумент `y` не имеет аннотации типа `Positive`. Условие верификации:

```
{} ⇒ { y > 0 }
```

Конвейер возвращает `Disproved` (импликация неверна) → ошибка компиляции:

> Невозможно доказать, что параметр `b` удовлетворяет `Positive` в вызове `divide`.
> `y` поступает из ввода функции, доказанных границ нет.
> Рекомендуется охранять вызов ветвью if: `if y > 0 { divide(x, y) }`.

YaoXiang не допускает попадания значений времени выполнения в параметры уточнённого типа без предоставления статического свидетельства. Это не ограничение — это ядро философии жёсткой безопасности. Любой код, который компилятор не может статически доказать, не проходит компиляцию.

#### 3.5 Связь с унифицированным конвейером

Распространение путевых условий — не дополнительный механизм. Это прямое расширение конвейера доказательств времени компиляции на анализ потока управления:

| Этап | Обязанности |
|------|------------|
| Сбор путевых условий | Этап анализа потока управления компилятора, аннотирующий каждый базовый блок множеством гипотез |
| Генерация условий верификации | При обнаружении ограничения типа, требующего верификации, объединение путевых условий + информация о типах фактических аргументов |
| Вычисление в конвейере доказательств | Ядро компилятора → SMT-ускорение → результат: Proved / Disproved / Unproven |
| Результат | `Proved` → проходит; `Disproved` → ошибка компиляции + контрпример; `Unproven` → ошибка компиляции + недоказанная proposition (программист может предоставить функцию-доказательство) |

Никаких новых компонентов. Никаких специальных правил. Путевые условия — это фоновые знания конвейера доказательств — разделяющие один конвейер и один бюджет с равенством типов и ограничениями заимствования.

### 4. Конвейер доказательств времени компиляции

Все проверки времени компиляции разделяют один конвейер. Основная операция конвейера — **проверка типов** — проверка того, равен ли тип терма доказательства доказываемой proposition. Всё — это проверка типов.

```
Компиляция встречает выражение Bool, требующее вычисления (т.е. нужно построить терм доказательства)
        │
        ├── Равенство типов (T1 == T2)
        │   → компилятор проверяет напрямую (структурная эквивалентность)
        │
        ├── Условие конфликта токенов (!conflicting(tokens))
        │   → потоко-чувствительный анализ liveness (отслеживание Dup/Linear-атрибутов)
        │
        ├── Редукция зависимых типов (n + m свёртка)
        │   → система переписывания термов времени компиляции (βδι-редукция)
        │
        ├── Предикат времени компиляции (x > 0, forall...)
        │   → ядро компилятора + SMT-ускоряющий модуль
        │
        └── Импликация логики Хоара (P ⇒ Q)
            → ядро компилятора + SMT-ускоряющий модуль
                    │
                    ▼
             ┌──────────┐
             │ Proved   │  → компиляция проходит
             │ Disproved│  → ошибка компиляции + контрпример
             │ Unproven │  → ошибка компиляции + недоказанная proposition
             └────┬─────┘
                  │
                  ▼
         Программист пишет функцию-доказательство (код на YaoXiang)
                  │
                  ▼
         Проверщик типов верифицирует ──→ Proved ──→ компиляция проходит
                  │
                  ▼
            Верификация не прошла → ошибка компиляции: «доказательство неверно»
```

#### 4.1 Результат доказательства: трёхзначная алгебра

Вычисление времени компиляции возвращает три результата — это неизбежное следствие проблемы остановки и естественное разделение теории доказательств:

```
eval_compile_time : BoolExpr → Proved | Disproved(Model) | Unproven
```

- **Proved** → остановка, терм доказательства построен, проверка типа прошла. Компиляция продолжается.
- **Disproved(M)** → остановка, существует контрпример M. Ошибка компиляции + контрпример + позиция в исходном коде.
- **Unproven** → в пределах данного лимита ресурсов терм доказательства не построен. Ошибка компиляции + недоказанная proposition + отчёт о потреблении бюджета.

**Unproven ≠ False.** Когда компилятор говорит «я не могу это доказать», это не эквивалентно ложности proposition — это лишь выход за пределы возможностей текущего автоматического доказательства. Это честность, а не дефект.

Жёсткое ограничение бюджета — инженерное решение проблемы остановки. Никаких ручек — дать ручку означало бы спросить пользователя «как вы думаете, остановится ли ваша программа», пользователь не знает, компилятор тоже не знает.

#### 4.2 После Unproven: программист пишет доказательство

Когда компилятор возвращает Unproven, программист может написать **функцию-доказательство** — то есть функцию на YaoXiang, чей возвращаемый тип равен доказываемой proposition. Проверщик типов верифицирует эту функцию — тем же механизмом, которым он верифицирует `add(a, b): Int`.

```
proposition = тип
доказательство = программа (значение этого типа)
верификация = проверка типов (единственный корень доверия)
```

SMT-решатель — не отдельный доверенный рубеж, а **ускоряющий модуль проверщика типов**. SMT помогает найти доказательство, но верифицирует доказательство всегда проверщик типов. Когда SMT возвращает `unsat`, компилятор реконструирует этот результат в терм доказательства, верифицируемый проверщиком типов. Если реконструкция не удалась (шаги рассуждения SMT выходят за пределы правил вывода ядра компилятора), происходит откат к Unproven — программист может вручную написать функцию-доказательство.

```yaoxiang
# Proposition: уточнённое свойство, которое компилятор не может доказать автоматически
FirstIsMin: (T: Ord, arr: Sorted(T)) -> Type = {
    forall i in 0..arr.len: arr[0] <= arr[i]
}

# Доказательство: программист пишет функцию, чей возвращаемый тип — proposition выше
# Проверщик типов верифицирует эту функцию — полностью аналогично верификации add(a,b): Int
first_is_min: (T: Ord, arr: Sorted(T)) -> FirstIsMin(T, arr) = {
    # Компилятор верифицирует здесь: тип тела функции = FirstIsMin(T, arr)
    ...
}
```

Никакого ИИ, никакого экспорта в Coq, никаких новых концепций. **Свойства, которые компилятор не может доказать автоматически → программист пишет доказательство в виде кода на YaoXiang → проверщик типов верифицирует.** Весь процесс — плавный градиент — компилятор берёт на себя простые доказательства, оставляя мозг для сложных.

#### 4.3 Слоистые зависимости внутри конвейера

Вышеописанные вычислители разделяют один интерфейс, но имеют порядок вычисления. Равенство типов — предпосылка для всего последующего анализа; проверка владения/токенов зависит от информации о типах; верификация уточняющих предикатов зависит от результатов первых двух слоёв. Компилятор вычисляет по слоям — выражения, не прошедшие нижний слой, не поступают в верхний — что позволяет не тратить бюджет решателя на типонеправильных программах.

```
Порядок вычисления (один конвейер, слоистое планирование)
├── Слой 0: равенство типов (T1 == T2)
│   └── структурная унификация → при неудаче дальнейшее бессмысленно, сразу Disproved
├── Слой 1: конфликты владения/токенов
│   └── потоко-чувствительный анализ liveness → при неудаче безопасность памяти не выполняется, сразу Disproved
└── Слой 2: уточняющие предикаты / импликации Хоара
    └── ядро компилятора → SMT-ускорение → Proved / Disproved / Unproven
```

Каждый слой по-прежнему возвращает `Proved/Disproved/Unproven`, разделяя один интерфейс и одну систему бюджета.

### 5. Единство трёх слоёв функций

| Слой | Момент выполнения | Вход | Выход | Пример |
|------|-------------------|------|-------|--------|
| Функция значений | Время выполнения | значения | значение | `add: (a: Int, b: Int) -> Int = a + b` |
| Конструктор типов | Время компиляции | типы/значения | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| Предикат времени компиляции | Время компиляции | значения | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

Все используют одинаковый синтаксис `name: type = value`. Предикаты времени компиляции и конструкторы типов идут по одному конвейеру доказательств — `{}` — это пространство доказательств.

### 6. Циклы: генерация условий верификации по Флойду-Хоару

Циклам не нужны отдельные аннотации `: Invariant(...)` или `: decreases(...)`. Аннотация типа предиката времени компиляции на переменной определяет утверждение в стиле Флойда-Хоара — компилятор генерирует условия верификации из аннотации типа, а конвейер доказательств проверяет, сохраняет ли каждое присваивание тип.

Базовый механизм: каждая операция присваивания соответствует тройке Хоара `{P} x := e {Q}`, условие верификации — `P ⇒ Q[e/x]`. Компилятор генерирует условие верификации для тела цикла один раз — после того как конвейер доказательств верифицирует индуктивный шаг, автоматически покрываются все итерации.

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # аннотация ссылается на i; при инициализации i=0,
                                 # верификация: 0 == sum(arr[0..0]) → True
    mut i: UpTo(arr.len) = 0     # верификация: 0 <= 0 <= arr.len → True
    while i < arr.len {
        # Компилятор генерирует VC для тела цикла один раз.
        # Предпосылка: s удовлетворяет SumUpTo(arr, i), i удовлетворяет UpTo(arr.len).
        #
        # s += arr[i]:
        #   Обязательство верификации: s_new удовлетворяет SumUpTo(arr, i) (текущий i не меняется)
        #   Подставляем s_new = s_old + arr[i]:
        #     требуется s_old + arr[i] == sum(arr[0..i+1])
        #     Из индуктивной гипотезы s_old == sum(arr[0..i]), прибавим arr[i] к обеим сторонам:
        #     sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   Компилятор + SMT: линейная арифметика, миллисекунды → Proved
        #
        # i += 1:
        #   i изменён → в графе зависимостей аннотация типа s ссылается на i → срабатывает реверификация
        #   Новая цель верификации: s удовлетворяет SumUpTo(arr, i_new)
        #   Т.е. s == sum(arr[0..i_new]), гарантируется предыдущим шагом → Proved
        s += arr[i]
        i += 1
    }
    return s  # в этой точке s: SumUpTo(arr, arr.len), т.е. s == sum(arr[0..arr.len])
}
```

Инвариант цикла — это аннотация типа на переменной — программист пишет тип, компилятор проверяет индуктивный шаг. Компилятору не нужно «обнаруживать» инвариант или «автоматически проводить индукцию» — он разбивает индуктивное доказательство на локальные условия верификации для каждой операции присваивания, передавая их в конвейер доказательств для разделения и властвования.

#### 6.1 Отслеживание зависимостей: зависимые типы на изменяемых переменных

Предпосылка вышеописанного механизма: компилятор знает, что аннотация типа `s` — `SumUpTo(arr, i)` — ссылается на `i` — при изменении `i` ограничение типа `s` также изменяется. Это требует от компилятора поддерживать **граф зависимостей типов между переменными**.

**Структура данных**:

```
TypeDepGraph: Map<VarName, Set<VarName>>
# Ключ — зависимая переменная, значение — множество переменных, на которые ссылается аннотация типа
# Пример: { i: {s}, j: {s, t}, ... }
```

**Построение**: при обработке `mut v: Pred(... x ...) = init` проверщик типов анализирует свободные переменные в аргументах `Pred(...)`. Если в аргументах есть ссылка на другую изменяемую переменную `x` в текущей области видимости, в графе зависимостей записывается `x → v`.

**Срабатывание**: при присваивании зависимой переменной `x` компилятор:
1. Ищет в графе зависимостей все переменные, зависящие от `x`: `{v₁, v₂, ...}`
2. Для каждого `v` генерирует условие верификации: удовлетворяет ли текущее значение `v` обновлённому типу `Pred(... x_new ...)`
3. Отправляет VC в конвейер доказательств

**Чувствительность к порядку присваивания**: отслеживание зависимостей естественным образом форсирует правильный порядок присваивания. На примере `SumUpTo(arr, i)`:

```yaoxiang
# Правильный порядок
s += arr[i]   # s_new удовлетворяет SumUpTo(arr, i+1)
i += 1        # i изменён → реверификация s удовлетворяет SumUpTo(arr, i_new) → True

# Неправильный порядок — компилятор отвергает
i += 1        # i изменён → реверификация s удовлетворяет SumUpTo(arr, i_new)
              # s ещё не обновлён, s_old == sum(arr[0..i_old]) ≠ sum(arr[0..i_new])
              # → ошибка компиляции: переменная s не удовлетворяет типу SumUpTo(arr, i_new)
s += arr[i]   # недостижимо
```

**Составные зависимости**: переменная может зависеть от нескольких переменных. Аннотация типа `{ v: Int; v == x + y }` одновременно зависит от `x` и `y` — изменение любой из них вызывает реверификацию.

**Связь с конвейером доказательств**: отслеживание зависимостей — триггер генерации VC, а не независимый механизм верификации. Он отвечает на вопрос «когда нужно генерировать VC» — конвейер доказательств отвечает на вопрос «выполняется ли VC».

### 7. Проверка завершимости

Полностью автоматическая во время компиляции. Циклы, которые компилятор может доказать, проходят; которые не может — сразу сообщают об ошибке компиляции — программист должен позволить компилятору автоматически анализировать завершимость цикла. Никаких лазеек для полуавтоматических аннотаций.

#### 7.1 Принципы проектирования

Компилятор автоматически извлекает информацию, необходимую для доказательства завершимости, из двух источников:

1. **Аннотации типов переменных**: ограничения границ в уточнённых типах (например, `UpTo(n)` даёт верхнюю границу `n` и нижнюю `0`)
2. **Операции в теле цикла**: операции, применяемые к переменным на каждой итерации

Компилятор по приоритету пробует четыре стратегии синтеза метрик, останавливаясь на первой подходящей.

#### 7.2 Стратегия 1: автоматический синтез линейной ранговой функции

Когда переменная имеет аннотацию линейной границы, компилятор перебирает кандидатов линейных метрик и верифицирует через SMT.

```
Вход:
  Переменные v₁: UpTo(u₁), v₂: UpTo(u₂), ... (переменные с верхней/нижней границами)
  Условие цикла cond
  Множество присваиваний в теле цикла

Алгоритм:
  1. Извлечь границы каждой переменной из аннотаций типа: [low_i, high_i]
  2. Перебрать кандидатов метрик: v_i, u_i - v_i, v_i - v_j, и т.д. — линейные комбинации
  3. Для каждого кандидата m:
     - SMT верифицирует m ≥ 0 (из границ типа)
     - Для каждого пути выполнения в теле цикла SMT верифицирует m' < m (строгое убывание)
  4. Найти подходящую линейную комбинацию → завершимость доказана
```

Охват: циклы, в которых переменной присваивается линейное выражение (`v = a·v + b`) и имеется аннотация ограниченного типа. Включает `i += const`, `i -= const`, а также сокращение интервалов в стиле двоичного поиска:

```yaoxiang
# Бинарный поиск: low = mid + 1 или high = mid
# Метрика high - low строго убывает на обоих путях
binary_search: (arr: Sorted(Int, arr), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 7.3 Стратегия 2: подсчёт нарушений предиката — автоматическое извлечение метрики из целевого типа <span style="color:orange">【Экспериментальная стратегия】</span>

> ⚠️ **Текущий статус: экспериментальная стратегия; решение о включении принимается на этапе 3 на основе реальной осуществимости.**
> Эта стратегия работает для операций соседнего обмена (сортировка пузырьком, сортировка вставками), но не может автоматически доказать не соседние операции (быстрая сортировка partition, пирамидальная сортировка sift-down).
> Границы охвата см. в таблице ниже. Если на этапе 3 будет установлена неосуществимость, эта стратегия будет удалена или переведена в разряд будущей работы.

Ключевое наблюдение: **спецификации, написанные пользователем, — это сырьё для рассуждений компилятора.** Компилятору не нужно встраивать «что такое сортировка» — он читает определение `Sorted` и автоматически извлекает из него метрику.

```
Вход:
  Целевой тип: Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  Операции в теле цикла: обмен соседних элементов

Алгоритм:
  1. Разобрать определение предиката: forall i in range: cond(i, arr)
  2. Автоматически сгенерировать метрику: violation_count = |{ i | ¬cond(i, arr) }|
  3. Проанализировать влияние операций на метрику:
     - Обмен соседних элементов arr[j], arr[j+1] = arr[j+1], arr[j]
     - Влияет только на пары с индексами j-1, j, j+1
     - Если arr[j] > arr[j+1] (нарушение предиката), после обмена пара удовлетворяет предикату
     - violation_count уменьшается как минимум на 1
  4. Верхняя граница: n·(n-1)/2 (максимальное число соседних инверсий), нижняя: 0
  → Завершимость доказана
```

**Текущий охват**:

| Алгоритм | Паттерн операций | Стратегия 2 доказуема? | Причина |
|----------|------------------|:---:|---------|
| Сортировка пузырьком | Соседний обмен | ✅ | violation_count строго убывает при каждом обмене |
| Сортировка вставками | Соседнее перемещение | ✅ | Каждое перемещение устраняет одну нарушающую пару |
| Сортировка выбором | Несоседний обмен | ❌ | Одиночный обмен может увеличить violation_count |
| Быстрая сортировка | partition | ❌ | Несоседний обмен, не гарантирует монотонного убывания |
| Пирамидальная сортировка | sift-down | ❌ | Древовидные операции, violation_count немонотонен |

**Дополняющие стратегии**: для быстрой сортировки сокращение интервала `low < high` покрывается стратегией 1 (линейная ранговая функция) — внешняя рекурсия partition, на каждом шаге интервал сокращается вдвое. Стратегии 1 и 2 дополняют друг друга, завершимость большинства практических алгоритмов может быть доказана одной из них. Однако обобщение стратегии 2 (несоседние операции, древовидные операции) остаётся открытой проблемой.

```yaoxiang
sort: (arr: Array(Int)) -> (result: Sorted(result)) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 7.4 Стратегия 3: паттерн ограниченного инкремента/декремента

`v += const` (const > 0), переменная имеет аннотацию верхней границы → метрика `upper_bound - v` убывает на `const` каждую итерацию, нижняя граница 0. Это вырожденный случай стратегии 1, компилятор обрабатывает его первым.

#### 7.5 Стратегия 4: шаблон метрики мультипликативного масштабирования

`v *= const` (const > 1), переменная имеет аннотацию верхней/нижней границы. Компилятор имеет встроенный шаблон логарифмической метрики `ceil(log_const(upper/v))`, при умножении на const метрика убывает на 1.

```yaoxiang
mut i: Positive(i) = 1
while i < n {
    # Компилятор автоматически выводит: метрика ceil(log₂(n/i)),
    # при умножении на 2 метрика убывает на 1
    i *= 2
}
```

#### 7.6 Разделение завершимости и корректности

Доказательство завершимости и доказательство корректности независимы:
- **Завершимость**: четыре вышеописанные стратегии автоматически доказывают, что цикл завершается за конечное число шагов
- **Корректность**: продвигается ли тело цикла к целевому типу, проверяется конвейером доказательств времени компиляции через условия верификации

Оба проходят → компиляция проходит. Завершимость доказана, но корректность не прошла → ошибка компиляции + контрпример. Корректность доказана, но завершимость не доказуема → ошибка компиляции с указанием переменной или операции, которые не удаётся проанализировать. Оба не прошли → ошибка компиляции с раздельным отчётом о двух причинах.

#### 7.7 Проверка завершимости рекурсивных функций

Для рекурсивных функций, которые нужно вычислить во время компиляции, компилятор проверяет убывание аргументов:

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # Анализ компилятора: n-1 < n → убывает → завершается
}

# Использование во время компиляции — компилятор гарантирует завершение factorial во время компиляции
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120, вычисляется во время компиляции
```

| Сценарий | Поведение |
|----------|-----------|
| Компилятор может проанализировать убывание рекурсии (например, `n-1`) | Вычисление во время компиляции |
| Не убывает / невозможно определить убывание | Ошибка компиляции |
| Вызов во время выполнения (не в позиции типа) | Проверка завершимости не требуется |

#### 7.8 Жёсткие границы

`i = f(i)`, где `f` необратима, не замкнута, не сохраняет никакой монотонности — математически невозможно автоматически доказать завершимость. Ошибка компиляции:

> Этот цикл не может быть автоматически доказан как завершающийся. Переменная цикла зависит от неанализируемой функции `f`. Используйте итеративный паттерн, который может быть проанализирован компилятором.

Это не провал компилятора. Любой код, который невозможно статически доказать как безопасный, не проходит компиляцию.

### 8. SMT-решатель: ускоряющий модуль проверщика типов

SMT-решатель в традиционных языках — внешний инструмент (F\* вызывает Z3, Dafny вызывает Z3). В YaoXiang это **ускоряющий модуль проверщика типов** — вызывается только тогда, когда ядро компилятора само не может вынести решение. SMT помогает найти доказательство, но верифицирует доказательство проверщик типов.

**Модель доверия**:
- Проверщик типов — единственный корень доверия. Когда SMT возвращает `unsat`, компилятор реконструирует его след рассуждений в терм доказательства, верифицируемый ядром проверщика типов. Если реконструкция не удалась (шаги рассуждения SMT выходят за пределы правил вывода ядра компилятора), происходит откат к Unproven — программист может вручную написать функцию-доказательство.
- На этапах 1/2 в качестве прагматичного старта результат `unsat` от SMT принимается компилятором на доверие (это путь F\*/Dafny — вероятность ошибки Z3 ниже, чем частота багов в самом компиляторе). На этапах 3/4 добавляется полная реконструкция доказательств как более сильная гарантия.

**Интерфейс**: компилятор внутри транслирует в стандартный формат SMT-LIB 2.6, а не привязывается к API конкретного решателя. SMT-LIB — стандарт ISO, Z3, CVC5, MathSAT, Yices — все нативно его поддерживают.

**Бэкенд по умолчанию**: Z3 (лицензия MIT, самая обширная документация и общественная валидация). CVC5 как SMT-LIB-совместимая альтернатива — пользователь может переключаться флагом компилятора.

Не делаем «универсального слоя абстракции решателя» — SMT-LIB и есть слой абстракции. В будущем, если CVC5 добьётся прорыва в определённой теории, переключение требует лишь замены бинарника, без изменения кода компилятора.

```
Выражение Bool времени компиляции
        │
        ├── Ядро компилятора может решить напрямую (структурная эквивалентность,
        │   простая арифметика, тривиальные формулы после свёртки констант)
        │   → сразу возвращает Proved / Disproved
        │
        └── Ядро компилятора не может решить напрямую (кванторы, символьные переменные)
            → Предварительная редукция зависимых типов (factorial(5) → 120)
            → Трансляция в формат SMT-LIB
            → Отправка в Z3/CVC5 (с ограничением бюджета)
            → Возврат: unsat → Proved  │  sat + модель → Disproved  │  unknown → Unproven
```

**Бюджет решателя — жёсткое ограничение, как глубина стека**:

| Измерение бюджета | Значение по умолчанию | Пояснение |
|-------------------|----------------------|-----------|
| Число шагов решателя | 10 000 | Z3 для линейной арифметики обычно укладывается в сотни шагов. 10 000 шагов покрывают 99% практических предикатов. |
| Время | 100 мс | Один предикат свыше 100 мс = пользователь пишет программу времени компиляции, а не аннотацию типа. 100 мс × 50 предикатов = 5 секунд верхней границы времени компиляции. |
| Глубина инстанцирования кванторов | 3 | Три уровня вложенных кванторов покрывают практические паттерны. Свыше трёх уровней — скорее всего, пишется логическая задачка. |

Превышение бюджета возвращает Unproven, ошибка компиляции + позиция предиката + потребление. Никаких деградаций, никаких проверок во время выполнения, никаких тихих проходов.

**Почему это практически осуществимо**: в инженерной практике 95% предикатов — линейная арифметика — `x > 0`, `arr.len > 0`, `0 <= idx < arr.len` — всё в пределах разрешимого фрагмента, SMT-решатель для таких задач возвращает ответ за миллисекунды. Для редких сложных предикатов, превышающих бюджет, программист пишет функцию-доказательство.

Зависимые типы делают один слой предварительной редукции до вызова SMT: `factorial(5)` напрямую вычисляется во время компиляции в `120`, `append([1,2], [3])` напрямую вычисляется в `[1,2,3]`. Эти детерминированные вычисления значений не потребляют бюджет SMT.

Программисту не нужно знать о существовании SMT. Ментальная модель: **если компилятор может доказать — проходит, не может — сообщает об ошибке — а если компилятор не умеет, можно написать функцию-доказательство, чтобы показать ему**.

### 9. Композиция предикатов времени компиляции

Предикат времени компиляции — это функция, возвращающая Type; композиция естественно реализуется через композицию функций:

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(arr)
}
```

### 10. Примеры кода

#### 10.1 Безопасность деления

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b

result = divide(10, 2)   # ✅ Компилятор верифицирует Positive(2) = { 2 > 0 } → True
# result = divide(10, 0)  # ❌ Компилятор верифицирует Positive(0) = { 0 > 0 } → False
```

#### 10.2 Безопасность доступа к массиву

```yaoxiang
InBounds: (idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (arr: Array(T), idx: InBounds(idx, arr)) -> T = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ Компилятор верифицирует InBounds(1, arr) = { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ Компилятор верифицирует InBounds(5, arr) = { 0 <= 5 && 5 < 3 } → False
```

#### 10.3 Корректность сортировки

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T))) -> (result: Sorted(T, result)) = {
    result = arr.clone()
    # ... реализация алгоритма сортировки ...
    return result
}
```

#### 10.4 Цикл: генерация VC компилятором

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```

## Детальное проектирование

### Синтаксические изменения

| Ранее (RFC-022) | Теперь (данный RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | Предикат времени компиляции как тип параметра `(b: Positive(b))` |
| `//! ensures: ExistsMax(result, arr)` | Тип возвращаемого значения использует формальный параметр возвращаемого значения `-> (result: IsMax(T, arr, result))` |
| `/*! invariant: ... !*/` | Аннотация типа предиката времени компиляции на переменной — инвариант Флойда-Хоара |
| `//! decreases: n` | Полностью автоматический вывод функции-меры компилятором |
| Спецификация — это комментарий | Спецификация — это система типов |

### Синтаксис

**У предикатов времени компиляции нет новых ключевых слов.** `{}` — пространство доказательств, полностью совместимо с существующим синтаксисом определения типов. Предикат времени компиляции — это функция, возвращающая Type — `name: (params) -> Type = { утверждение }`. Использование — это вызов функции — `Positive(b)`, `IsMax(T, arr, result)`.

```bnf
# Предикат времени компиляции = функция, возвращающая Type, внутри {} — утверждения,
# верифицируемые компилятором
# Используется существующий синтаксис функций/типов, новые правила BNF не требуются
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

**Новое синтаксическое понятие: формальный параметр возвращаемого значения** — в `-> (name: Type)` `name` — формальный параметр возвращаемого значения.

Формальный параметр возвращаемого значения — это **единственное синтаксическое понятие**, вводимое YaoXiang в существующий синтаксис функций. Его семантика:
- Значение `name` предоставляется оператором `return`
- `name` существует только в сигнатуре типа, на него ссылается предикат постусловия (например, `-> (result: IsMax(T, arr, result))`)
- `name` не входит в область видимости тела функции и не появляется у вызывающей стороны
- **Формальный параметр возвращаемого значения необязателен** — при отсутствии постусловия сигнатура полностью совпадает с обычной функцией (`-> Int`), не создавая дополнительной нагрузки

Причина его введения: постусловию нужно ссылаться на «значение, которое функция собирается вернуть». Без формального параметра возвращаемого значения компилятор может позволить предикату ссылаться на возвращаемое значение только через специальные правила (например, неявная переменная `$result` или `__retval__`). Формальный параметр возвращаемого значения делает эту ссылку явной — это просто формальный параметр, только значение предоставляется `return`, а не вызывающей стороной.

**Функция-доказательство** — не новое понятие, это функция на YaoXiang, чей возвращаемый тип — это доказываемая proposition. Когда компилятор возвращает Unproven, программист предоставляет функцию-доказательство, и проверщик типов верифицирует её точно так же, как верифицирует возвращаемый тип любой функции. Никакого нового синтаксиса, никаких новых ключевых слов, никаких новых правил.

### Влияние на систему типов

- **Вселенная типов**: предикаты времени компиляции находятся на уровне Type₂ — функции, принимающие значения и возвращающие Type, на том же уровне, что и конструкторы типов
- **Взаимодействие с обобщёнными типами**: предикаты времени компиляции могут иметь параметры обобщённых типов, например `NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **Взаимодействие с владением**: выражения в предикатах времени компиляции подчиняются правилам владения, только чтение, без записи
- **Вывод типов**: аргументы предикатов времени компиляции участвуют в выводе типов HM

### Представление во время выполнения

Предикаты времени компиляции **полностью стираются** во время выполнения. `Positive: (x: Int) -> Type = { x > 0 }` — параметр `b: Positive(b)` во время выполнения представлен просто как `Int`. Условие уточнения `{ x > 0 }` верифицируется конвейером доказательств во время компиляции, после верификации не оставляет никаких следов во время выполнения.

- `Positive(5)` → во время выполнения представлено как `Int(5)`, условие уточнения `{ 5 > 0 }` пройдено, стёрто
- `SumUpTo(arr, 0)` → во время выполнения представлено как `Int(0)`, равенство `0 == sum(arr[0..0])` пройдено, стёрто
- Размещение предиката времени компиляции в позиции типа (например, `f(x: Positive(x))`) не порождает никакого обёрточного типа, не выделяет дополнительной памяти, не вставляет проверок времени выполнения

Стирание обобщённых типов и стирание уточнённых типов — один и тот же принцип: оба являются функциями времени компиляции, полностью исчезающими после вычисления во время компиляции. В Release Build предикаты времени компиляции имеют нулевые накладные расходы во время выполнения — прямое следствие «предикат — это функция».

**Ограничения взаимодействия с `ref`**: предикаты времени компиляции могут ссылаться только на неизменяемые заимствования или значения с переданным владением. Предикат времени компиляции, ссылающийся на значение с изменяемым заимствованием, не может быть гарантирован компилятором как остающийся верным во время выполнения — такое использование сразу сообщает об ошибке компиляции.

### Изменения в компиляторе

1. **Парсер**: предикаты времени компиляции используют стандартный синтаксис функций, дополнительных правил разбора не требуется
2. **Конвейер доказательств времени компиляции**: унифицированный интерфейс возврата Proved/Disproved/Unproven, автоматический выбор стратегии
3. **SMT-ускоряющий модуль**: слой трансляции в SMT-LIB 2.6, бэкенд по умолчанию Z3, альтернатива CVC5
4. **Ядро проверщика типов**: реализация правил вывода — структурная эквивалентность, βδι-редукция, введение/устранение кванторов всеобщности. Это единственный корень доверия, через который верифицируются и результаты SMT, и доказательства программиста
5. **Генерация условий верификации**: исчисление WP/SP + обязательства доказательства инвариантов цикла
6. **Отчёты об ошибках**: форматирование контрпримеров + отчёт о недоказанных proposition + привязка к позициям в исходном коде

### Обратная совместимость

- ✅ Код, не использующий предикаты времени компиляции, полностью не меняется
- ✅ Предикаты времени компиляции имеют нулевые накладные расходы во время выполнения в Release Build
- ⚠️ Синтаксис `//!` из RFC-022 больше не поддерживается — однако 022 никогда не был реализован, бремя миграции отсутствует

## Компромиссы

### Преимущества

- **Полная реализация изоморфизма Карри-Ховарда**: тип есть proposition, программа есть доказательство, `name: Proposition = Proof`
- **Единство**: предикаты времени компиляции и обычные функции используют полностью одинаковый синтаксис, без концептуального разделения
- **Прозрачность SMT**: программисту не нужно знать о существовании SMT, ментальная модель совпадает с проверкой типов
- **Постепенное внедрение**: можно начать с одного предиката времени компиляции, постепенно расширяя покрытие
- **Нулевые накладные расходы во время выполнения**: все вычисления завершаются во время компиляции, Release Build не вставляет никаких утверждений времени выполнения

### Недостатки

- **Время компиляции**: вызов SMT увеличивает время компиляции, но жёсткое ограничение бюджета гарантирует контролируемый верхний предел
- **Границы автоматического доказательства**: сложные предикаты за пределами линейной арифметики первого порядка могут потребовать от программиста написания функций-доказательств. Это не дефект языка — это неизбежное следствие проблемы остановки. Компилятор честно сообщает Unproven, а не ложно сообщает True/False
- **Кривая обучения**: написание эффективных предикатов времени компиляции и функций-доказательств требует понимания базовой интуиции изоморфизма Карри-Ховарда
- **Сложность реализации**: единство конвейера доказательств времени компиляции требует тщательного проектирования

### Снижение рисков

- Жёсткое ограничение бюджета SMT-решателя (10 000 шагов / 100 мс / 3 уровня инстанцирования), превышение возвращает Unproven
- Предварительная редукция зависимых типов: детерминированные вычисления значений поглощаются первыми, SMT кусает только недетерминированную часть
- Unproven — не тупик: программист может написать функцию-доказательство, проверщик типов верифицирует — аналогично верификации любого возвращаемого типа функции
- Инкрементальная верификация: верифицируются только изменённые модули
- Чёткие сообщения об ошибках + отображение контрпримеров + отчёт о потреблении бюджета + недоказанная proposition + предложения (если компилятор может их дать)

## Альтернативы

| Альтернатива | Почему не выбрана |
|--------------|-------------------|
| RFC-022: спецификации в форме комментариев `//!` | Разделяет спецификации и типы, нарушает изоморфизм Карри-Ховарда |
| Отдельные файлы спецификаций (например, CVL) | Отделение спецификаций от кода увеличивает стоимость сопровождения |
| Только утверждения времени выполнения | Невозможно статически гарантировать корректность |
| Внешний помощник доказательств (например, Coq) | Отрыв от компилятора, требует отдельного языка доказательств и доверенного рубежа. Выбор YaoXiang: доказательство — это код на YaoXiang, проверщик типов — единственный корень доверия |
| **Данный подход: предикаты времени компиляции как полноправные граждане** | ✅ |

## Стратегия реализации

### Этапы

| Этап | Содержание |
|------|------------|
| **Этап 1** | Ядро компилятора: структурная эквивалентность + βδι-редукция + введение/устранение кванторов всеобщности. Поддержка простых арифметических предикатов (`x > 0`, `arr.len > 0`) |
| **Этап 2** | Слой трансляции в SMT-LIB + интеграция Z3/CVC5. Конвейер возвращает Proved/Disproved/Unproven. При Unproven поддержка функций-доказательств, написанных программистом |
| **Этап 3** | Генерация VC для инвариантов цикла + проверка завершимости (линейная ранговая функция + подсчёт нарушений предиката + паттерн ограниченности + контроль комбинаторного взрыва) |
| **Этап 4** | Реконструкция доказательств (SMT unsat → терм доказательства, верифицируемый ядром) + инкрементальная верификация + кэширование + форматирование контрпримеров + поддержка IDE |

### Зависимости

- RFC-010: унифицированный синтаксис типов — предикаты времени компиляции основаны на `name: type = value`
- RFC-011: система обобщённых типов — предикаты времени компиляции могут иметь параметры обобщённых типов
- RFC-009: модель владения — выражения предикатов времени компиляции подчиняются правилам владения

## Открытые вопросы

- [x] **Выбор SMT-решателя**: по умолчанию Z3 (лицензия MIT, самая широкая валидация). CVC5 как SMT-LIB-совместимая альтернатива, переключение флагом компилятора. Внутренняя трансляция компилятора направлена на стандартный формат SMT-LIB 2.6 — SMT-LIB и есть слой абстракции, специальный универсальный интерфейс решателя не создаётся.
- [x] **Конкретные значения бюджета решателя**: 10 000 шагов / 100 мс / 3 уровня инстанцирования кванторов. Жёстко зашито в компилятор, ручки не предоставляются. Если в практическом использовании обнаружится реальный случай, когда этого недостаточно (не «пользователь ошибся»), значения будут скорректированы.
- [x] **Охват поддержки кванторов**: на уровне языка порядок кванторов не ограничивается. Предикаты времени компиляции принимают параметры Type — Type включает типы функций — поэтому кванторы высокого порядка — естественное следствие системы типов, специальный синтаксис не требуется. SMT-решатель может автоматически разрешать кванторы первого порядка (forall/exists, поддерживается чередующееся вложение, ограничено бюджетом глубины 3). Кванторы высокого порядка: SMT возвращает Unproven, компилятор подсказывает «этот предикат выходит за пределы автоматического доказательства, предоставьте функцию-доказательство». Программист пишет функцию на YaoXiang, чей возвращаемый тип равен proposition — проверщик типов верифицирует эту функцию. Никакого внешнего экспорта, никакого ИИ, никакого интерактивного режима доказательства. Всё — код на YaoXiang, всё верифицируется проверщиком типов.
- [x] **Форматирование контрпримеров**: имена переменных исходного кода используются напрямую как имена SMT-переменных (с префиксом модуля для предотвращения конфликтов). При возврате модели Z3 выполняется обратный поиск по именам переменных. Формат вывода: имя_переменной = конкретное_значение + позиция в исходном коде + позиция определения предиката. Никакого сложного слоя отображения.
- [x] ~~**Взаимодействие предикатов времени компиляции с умным указателем `ref`?**~~ → Решено: предикаты времени компиляции допускают только неизменяемые заимствования или значения с переданным владением. Значения с изменяемым заимствованием не могут появляться в предикатах времени компиляции.
- [x] **Расширение метрики подсчёта нарушений `forall`-предиката на несмежные операции?** → Не расширяется. Текущий охват (соседний обмен, соседнее перемещение) дополняется стратегией 1 (линейная ранговая функция) — внешнее сокращение интервала быстрой сортировки покрывается стратегией 1, пирамидальная сортировка — стратегией 1 (паттерн индексов массива). Циклы, завершимость которых не может быть доказана ни одной стратегией, компилятор сразу отвергает — это философия жёсткой безопасности, а не дефект. Если в будущем появится реальный сценарий (не академическая конструкция), для которого все четыре стратегии не работают, вопрос будет пересмотрен.
- [x] **Комбинаторный взрыв перебора линейной ранговой функции**: верхний предел перебора кандидатов — 3 переменные с границами. При ≤3 перебираются все линейные комбинации и SMT верифицирует каждую. При >3 пробуются только однопеременные метрики (`v_i`, `u_i - v_i`); при неудаче — сразу ошибка компиляции — подсказка программисту «цикл имеет >3 переменных с границами, компилятор не может автоматически синтезировать многопеременную метрику». Это не инженерный компромисс — это принуждение программиста к написанию более простых циклов.
