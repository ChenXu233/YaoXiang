# RFC-027：Компиляторные предикаты и унифицированная статическая верификация

## Краткое содержание

В данном документе предлагается ввести в YaoXiang **компиляторные предикаты** как объекты первого класса и унифицировать всю компиляторную статическую верификацию в единый **конвейер доказательств**. Компиляторные предикаты — это не внешние спецификационные комментарии — они являются функциями. Функция, возвращающая Type, которую можно использовать в позиции типа; компилятор вызывает её во время компиляции и проверяет возвращаемое значение. Тип — это утверждение, а вычисление во время компиляции — это доказательство.

**Ключевой тезис**: единственная работа проверки типов во время компиляции — это построение и верификация доказательственных элементов. Типовые равенства, конфликты токенов, редукция зависимых типов, вычисление компиляторных предикатов, импликации по логике Хора — всё это различные проверки типов в едином компиляторном конвейере доказательств, разделяющие одну и ту же трубу. SMT-решатель — это ускорительный модуль проверщика типов, а не отдельная граница доверия. Когда компилятор возвращает Unproven, программист пишет функцию на YaoXiang в качестве доказательства — проверщик типов верифицирует её точно так же, как верифицирует тип возврата любой другой функции. Всё — это код YaoXiang, всё верифицируется проверщиком типов.

## Мотивация

### Почему упраздняется RFC-022?

RFC-022 проектировала спецификации в форме комментариев `//!`:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← это независимый от типа комментарий
    //! ensures: ExistsMax(result, arr[0..n])   ← это независимый от типа комментарий
}
```

Это фундаментальная ошибка соответствия Curry-Howard: **разделение спецификаций и типов на два слоя**. Комментарии — это не типы. Комментарии не участвуют в проверке типов. Комментарии — это ментальная модель «внешнего инструмента».

Белая книга чётко говорит:

> «Нет комментариев `//!`. Нет отдельного языка спецификаций. Всё внутри системы типов.»

### Текущие проблемы

- Комментарии `//!` в RFC-022 — это внешний синтаксис, независимый от системы типов
- Спецификационные типы и обычные типы — это две разные системы, создающие концептуальную избыточность
- Разделение на Debug Build верификация / Release Build игнорирование нарушает унификацию
- SMT-решатель в традиционном понимании позиционируется как внешний инструмент — YaoXiang встраивает его как ускорительный модуль проверщика типов
- Проверка типов, верификация заимствований, проверка компиляторных предикатов, раскрытие макросов идут разными путями

### Правильная ментальная модель

Проверку типов можно абстрагировать как функцию:

```
verify : Program → Proved | Disproved(Model) | Unproven
```

Все проверки во время компиляции — простое сопоставление типов, обнаружение конфликтов заимствований, верификация компиляторных предикатов — это подзадачи этой функции. Они разделяют единый конвейер доказательств, отличаясь лишь сложностью доказательственных элементов и стратегией построения.

Когда компилятор возвращает Unproven, программист предоставляет доказательную функцию — тип возврата этой функции равен доказываемому утверждению. Проверщик типов её верифицирует. Это та же самая операция, что и обычная проверка типов.

## Предложение

### 1. `{}` — пространство доказательств: тип — это утверждение, верификация — это проверка типов

`{}` в YaoXiang — это пространство доказательств времени компиляции. Всё внутри — это утверждения, которые компилятор гарантирует истинными — либо автоматически доказанные, либо предоставленные программистом через доказательственную функцию.

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  компилятор гарантирует, что x — это Float, y — это Float

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  компилятор гарантирует, что data — это Array(T)
```

**Обобщённые типы — это частный случай компиляторных предикатов.**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          параметр в сигнатуре   внутри {} только утверждение
#          компилятор проверяет x > 0 при вызове во время компиляции

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      параметр в сигнатуре   компилятор проверяет type_of(T) == Type, type_of(data) == Array(T)
```

Тот же паттерн: `name: (params) -> Type = { утверждения }`. Компилятор не различает «типовые утверждения» и «значимые утверждения» — всё это цели вычисления в конвейере доказательств.

**Инварианты циклов не нужно записывать отдельно. Типовые аннотации на переменных — это инварианты Флойда-Хора.**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # аннотация ссылается на i — говорит компилятору, что тип s зависит от i
    mut i: UpTo(arr.len) = 0     # при инициализации i=0, проверка: 0 == sum(arr[0..0]) → True
    while i < arr.len {
        s += arr[i]  # компилятор проверяет: s_new == sum(arr[0..i+1])
        i += 1       # i изменяется → триггер повторной проверки s: s удовлетворяет SumUpTo(arr, i_new)
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

Компилятор генерирует условия верификации для тела цикла один раз — индуктивная гипотеза (типовая аннотация) → операция присваивания → удовлетворяет ли новое значение типовая аннотация. После того как конвейер доказательств подтвердит индуктивный шаг, все итерации автоматически покрываются. Не нужно `: decreases`, не нужно `: Invariant`, не нужно индуктивное доказательство — компилятор разлагает индукцию на локальные VC для каждого присваивания.

### 2. Предусловия/постусловия: компиляторные предикаты в типах параметров и возвращаемого типа

Отказ от `//! requires`/`//! ensures` из RFC-022. Компиляторные предикаты как типы параметров или возвращаемого типа.

**Сторона параметров — это вызов функции.** Компиляторный предикат — это функция, возвращающая Type, использование на стороне параметров — это просто её вызов — как `factorial(5)`. Сторона возвращаемого значения вводит новую концепцию: параметр возвращаемого значения.

```yaoxiang
# Предусловие: явный вызов компиляторного предиката в типе параметра
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b
#                       ^^^^^^^^^^  b — это имя текущего формального параметра, передаётся в Positive как аргумент
#                       компилятор в точке вызова извлекает фактический аргумент, подставляет в b, проверяет Positive(фактический)
#                       пример: divide(10, 2) → проверка Positive(2) = { 2 > 0 } → True
#                       пример: divide(10, 0) → проверка Positive(0) = { 0 > 0 } → False → ошибка компиляции

# Постусловие: параметр возвращаемого значения + компиляторный предикат
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

NonEmpty: (arr: Array(T)) -> Type = { arr.len > 0 }

max: (T: Ord) -> ((arr: NonEmpty(arr))) -> (result: IsMax(T, arr, result)) = {
#                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                            result — это параметр возвращаемого значения, значение предоставляется return
#                                            компилятор в точке return подставляет возвращаемое значение, проверяет постусловие
    candidate = arr[0]
    for i in 1..arr.len {
        if arr[i] > candidate { candidate = arr[i] }
    }
    return candidate
}
```

**Ключевые правила**:

- **Сторона параметров**: `b: Positive(b)` — `b` — это имя текущего формального параметра, передаётся в `Positive`. Синтаксис вызова функции, ноль неявности.
- **Сторона возврата**: `-> (result: IsMax(T, arr, result))` — `result` — это параметр возвращаемого значения, значение предоставляется оператором `return`. `result` существует только в типовой сигнатуре, только предикат его упоминает, не входит в область видимости тела функции, не появляется у вызывающей стороны.
- **Параметр возвращаемого значения опционален**: без постусловия сигнатура полностью идентична обычной функции (`-> Int`).
- **Унификация**: параметры и параметры возвращаемого значения — это одна концепция — `имя_параметра: вызов_предиката(имя_параметра)`, разница лишь в том, предоставляется ли значение вызывающей стороной или `return`.

### 3. Распространение путевых условий: верификация значений времени выполнения во время компиляции

Когда компиляторный предикат используется в позиции связывания, параметры передаются программистом явно. Когда значения времени выполнения попадают в параметры уточнённого типа, компилятор завершает верификацию через сбор путевых условий и проверку SMT-импликации — без явной передачи доказательства программистом.

#### 3.1 Явный вызов функции

Когда компиляторный предикат используется в позиции связывания, параметры передаются программистом явно — это просто вызов функции, ноль неявности.

`Positive: (x: Int) -> Type = { x > 0 }` — это конструктор компиляторного предиката. Когда он появляется в позиции связывания (объявление параметра, объявление переменной, тип возврата), программист явно передаёт имя уже связанной переменной:

```yaoxiang
b: Positive(b)
// b уже объявлен как текущий формальный параметр, Positive(b) — это вызов функции
// после нормализации: b: { b > 0 }
```

Не нужно, чтобы компилятор неявно подставлял параметры — `b: Positive(b)` и `f(5)` — это одно и то же, просто вызов функции. `b` связывается как имя параметра, его типовая аннотация `Positive(b)` ссылается на `b` саму — это стандартный паттерн зависимых типов, не правило неявного раскрытия.

**Унификация с `self` из RFC-010**: RFC-010 установила, что `self` — это не ключевое слово, а договорённость об именовании параметра («написать `p`, `this`, `x` — эффект полностью одинаков»). `b: Positive(b)` разделяет тот же механизм — имя параметра может быть использовано в его типовой аннотации. `self` появляется в позиции `self: Point`, `b` появляется в позиции `b: Positive(b)`, обе типовые аннотации ссылаются на параметр. Разница только в сложности типовой аннотации, механизм абсолютно одинаков — после связывания имени тип может зависеть от этого имени.

Тип возврата также использует явный вызов функции:

```yaoxiang
Sorted: (arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (arr: Array(T)) -> (result: Sorted(result)) = { ... }
//                        ^^^^^^^^^^^^^^^^^^^^^^^
//                        result — это параметр возвращаемого значения, Sorted(result) — вызов функции
//                        компилятор в точке return подставляет возвращаемое значение, проверяет Sorted(возвращаемое)
```

То же применимо к объявлениям локальных переменных:

```yaoxiang
let x: Positive(x) = 5
// x связывается как 5, Positive(5) → { 5 > 0 } → True → проходит

// let y: Positive(y) = 0
// y связывается как 0, Positive(0) → { 0 > 0 } → False → ошибка компиляции
```

#### 3.2 Сбор путевых условий

Когда значение времени выполнения появляется в условной ветке, компилятор автоматически собирает путевые условия, формируя **множество предположений** для текущей области видимости. Эти предположения служат фоновым знанием при верификации вычислений типа Bool во время компиляции.

```yaoxiang
if y > 0 {
    // компилятор автоматически получает в этой ветке предположение: { y > 0 }
    let result = divide(x, y)
    // условие верификации: (y > 0) ⇒ (y > 0)
    // конвейер доказательств определяет импликацию → Proved
} else {
    // в этой ветке предположение: { !(y > 0) }
    // если вызвать divide(x, y), условие верификации: !(y > 0) ⇒ y > 0
    // конвейер доказательств определяет, что импликации нет → Disproved
}
```

Это не встроенный в компилятор специальный паттерн — это естественное поведение компиляторного конвейера доказательств. Каждая проверка типа в точке вызова отправляет в конвейер:

```
{фоновые предположения} ⇒ {цель верификации}
```

Конвейер доказательств проверяет импликативность. Proved → проходит, Disproved → ошибка компиляции + контрпример, Unproven → ошибка компиляции + нерешённое утверждение. Фоновые предположения берутся из путевых условий текущей точки программы.

#### 3.3 Стек предположений

При анализе потока управления компилятор поддерживает для каждого базового блока множество предположений:

- **if-guard**: `if y > 0` → в true-ветку добавляется `y > 0`, в false-ветку добавляется `!(y > 0)` (если есть else)
- **match паттерн**: `if let Some(v) = opt` → внутри ветки добавляется `opt == Some(v)`
- **логическое И**: `if x > 0 && y < 10` → внутри ветки добавляются `x > 0` и `y < 10`
- **предусловия функции**: при вызове `divide(a, b)`, доказательство того, что `b` удовлетворяет `Positive`, должно либо следовать из текущих предположений, либо из собственного уточнённого типа аннотации `b` (если `b` уже аннотирован как `Positive`, его тип несёт `b > 0`)
- **присваивание**: при `let z = y`, уточнённые условия с `y` передаются на `z`

Все предположения поступают в компиляторный конвейер доказательств. При входе на ускоренный путь через SMT транслируются как фоновые аксиомы SMT-LIB.

#### 3.4 Без статического доказательства — ошибка компиляции

Если программист напишет напрямую:

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

В текущей точке программы нет предположения `y > 0`, у фактического аргумента `y` самого по себе нет типовой аннотации `Positive`. Условие верификации:

```
{} ⇒ { y > 0 }
```

Конвейер возвращает `Disproved` (импликация не выполняется) → ошибка компиляции:

> Невозможно доказать, что параметр `b` в вызове `divide` удовлетворяет `Positive`.
> `y` взят из ввода функции, не имеет доказанных границ.
> Рассмотрите вызов с охранением в if-ветке: `if y > 0 { divide(x, y) }`.

YaoXiang не принимает значения времени выполнения, напрямую попадающие в параметры уточнённого типа без предоставления статического доказательства. Это не ограничение — это ядро философии жёсткой безопасности. Код, который компилятор не может статически доказать, не компилируется.

#### 3.5 Связь с унифицированным конвейером

Распространение путевых условий — это не дополнительный механизм. Это прямое расширение компиляторного конвейера доказательств на анализ потока управления:

| Фаза | Обязанности |
|------|-------------|
| Сбор путевых условий | Фаза анализа потока управления компилятора, аннотирует каждый базовый блок множеством предположений |
| Генерация условий верификации | При встрече с типом ограничения, требующим верификации, объединяет путевые условия + информацию о типах фактических аргументов |
| Вычисление конвейера доказательств | Ядро компилятора → ускорение через SMT → получение Proved / Disproved / Unproven |
| Результат | `Proved` → проходит; `Disproved` → ошибка компиляции + контрпример; `Unproven` → ошибка компиляции + нерешённое утверждение (программист может предоставить доказательственную функцию) |

Нет новых компонентов. Нет специальных правил. Путевые условия — это фоновые знания конвейера доказательств — разделяют ту же трубу и ту же бюджетную систему, что и типовые равенства, ограничения заимствования.

### 4. Компиляторный конвейер доказательств

Все проверки во время компиляции разделяют единый конвейер. Центральная операция конвейера — это **проверка типов** — проверка, равен ли тип доказательственного элемента доказываемому утверждению. Всё — это проверка типов.

```
Компилятор встречает Bool-выражение, требующее вычисления
(т.е.: нужно построить доказательский элемент)
        │
        ├── Типовое равенство (T1 == T2)
        │   → компилятор определяет напрямую (структурная эквивалентность)
        │
        ├── Условия конфликта токенов (!conflicting(tokens))
        │   → анализ потоково-чувствительной активности (отслеживание атрибутов Dup/Linear)
        │
        ├── Редукция зависимых типов (n + m упрощается)
        │   → система переписывания термов времени компиляции (βδι-редукция)
        │
        ├── Компиляторные предикаты (x > 0, forall...)
        │   → само ядро компилятора + ускорительный модуль SMT
        │
        └── Импликации по логике Хора (P ⇒ Q)
            → ядро компилятора + ускорительный модуль SMT
                    │
                    ▼
             ┌──────────┐
             │ Proved   │  → компиляция успешна
             │ Disproved│  → ошибка компиляции + контрпример
             │ Unproven │  → ошибка компиляции + нерешённое утверждение
             └────┬─────┘
                  │
                  ▼
         Программист пишет доказательственную функцию (код YaoXiang)
                  │
                  ▼
         Проверщик типов верифицирует ──→ Proved ──→ компиляция успешна
                  │
                  ▼
            Верификация не удалась → ошибка компиляции: «доказательство не выполняется»
```

#### 4.1 Результаты доказательства: трёхзначная алгебра

Вычисление во время компиляции возвращает три результата — это неизбежный вывод из проблемы остановки, а также естественное разделение в теории доказательств:

```
eval_compile_time : BoolExpr → Proved | Disproved(Model) | Unproven
```

- **Proved** → остановка, доказательский элемент построен, проверка типов пройдена. Компиляция продолжается.
- **Disproved(M)** → остановка, существует контрпример M. Ошибка компиляции + контрпример + позиция в исходном коде.
- **Unproven** → в пределах заданного лимита ресурсов доказательство не построено. Ошибка компиляции + нерешённое утверждение + отчёт о потреблении бюджета.

**Unproven ≠ False.** Когда компилятор говорит «я не могу доказать» — это не эквивалентно тому, что утверждение ложно — просто оно за пределами текущих возможностей автоматического доказательства. Это честность, а не недостаток.

Жёсткий лимит бюджета — это инженерное решение проблемы остановки. Никаких ручек — дать их значит спросить пользователя «как вы думаете, ваша программа остановится?», пользователь не знает, компилятор не знает.

#### 4.2 После Unproven: программист пишет доказательство

Когда компилятор возвращает Unproven, программист может написать **доказательственную функцию** — просто функцию YaoXiang, тип возврата которой равен доказываемому утверждению. Проверщик типов верифицирует эту функцию — и верифицирует точно так же, как верифицирует `add(a, b): Int`.

```
Утверждение  = Тип
Доказательство = Программа (значение этого типа)
Верификация  = Проверка типов (единственный корень доверия)
```

SMT-решатель — это не отдельная граница доверия — это **ускорительный модуль проверщика типов**. SMT помогает найти доказательство, но верифицирует доказательство всегда проверщик типов. Когда SMT возвращает `unsat`, компилятор реконструирует этот результат в проверяемый проверщиком типов доказательский элемент. Если реконструкция не удалась (шаги вывода SMT выходят за рамки правил вывода ядра компилятора), происходит откат к Unproven — программист может вручную написать доказательственную функцию.

```yaoxiang
# Утверждение: уточнённое свойство, которое компилятор не может автоматически доказать
FirstIsMin: (T: Ord, arr: Sorted(T)) -> Type = {
    forall i in 0..arr.len: arr[0] <= arr[i]
}

# Доказательство: программист пишет функцию, тип возврата которой — это утверждение выше
# Проверщик типов верифицирует эту функцию — точно так же, как верифицирует add(a,b): Int
first_is_min: (T: Ord, arr: Sorted(T)) -> FirstIsMin(T, arr) = {
    # компилятор здесь проверяет: тип тела функции = FirstIsMin(T, arr)
    ...
}
```

Не нужно AI, не нужно экспорта в Coq, не нужны новые концепции. **Свойства, которые нельзя автоматически доказать во время компиляции → программист пишет доказательство кодом YaoXiang → проверщик типов верифицирует.** Всё это плавный градиент — компилятор делает простые доказательства за вас, сложные оставляет вам.

#### 4.3 Многоуровневые зависимости внутри конвейера

Вышеупомянутые вычислители разделяют общий интерфейс, но имеют порядок вычисления. Типовое равенство — это предпосылка для любого последующего анализа; проверка владения/токенов зависит от информации о типах; верификация уточняющих предикатов зависит от результатов предыдущих двух уровней. Компилятор вычисляет послойно, выражения с неудавшимся нижним уровнем не попадают на верхний — чтобы не тратить бюджет решения на программах с ошибками типов.

```
Порядок вычисления (единый конвейер, послойное планирование)
├── Уровень 0: Типовое равенство (T1 == T2)
│   └── структурная унификация → при неудаче дальнейшее бессмысленно, сразу возврат Disproved
├── Уровень 1: Конфликты владения/токенов
│   └── анализ потоково-чувствительной активности → при неудаче безопасность памяти не выполняется, сразу возврат Disproved
└── Уровень 2: Уточняющие предикаты/импликации Хора
    └── ядро компилятора → ускорение через SMT → получение Proved / Disproved / Unproven
```

Каждый уровень по-прежнему возвращает `Proved/Disproved/Unproven`, разделяя тот же интерфейс и ту же бюджетную систему.

### 5. Триярусное объединение функций

| Уровень | Время выполнения | Вход | Выход | Пример |
|---------|-----------------|------|-------|--------|
| Функция значения | время выполнения | значение | значение | `add: (a: Int, b: Int) -> Int = a + b` |
| Конструктор типа | время компиляции | тип/значение | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| Компиляторный предикат | время компиляции | значение | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

Все используют идентичный синтаксис `name: type = value`. Компиляторные предикаты и конструкторы типов идут по единому компиляторному конвейеру доказательств — `{}` — это пространство доказательств.

### 6. Циклы: генерация условий верификации по Флойду-Хору

Циклам не нужны отдельные аннотации `: Invariant(...)` или `: decreases(...)`. Типовые аннотации с компиляторными предикатами на переменных определяют утверждения в стиле Флойда-Хора — компилятор генерирует из типовых аннотаций условия верификации, конвейер доказательств проверяет, сохраняет ли каждое присваивание тип.

Центральный механизм: каждая операция присваивания соответствует тройке Хора `{P} x := e {Q}`, условие верификации — `P ⇒ Q[e/x]`. Компилятор генерирует условия верификации для тела цикла один раз — после того как конвейер доказательств подтвердит индуктивный шаг, все итерации автоматически покрываются.

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # аннотация ссылается на i; при инициализации i=0, проверка: 0 == sum(arr[0..0]) → True
    mut i: UpTo(arr.len) = 0     # проверка: 0 <= 0 <= arr.len → True
    while i < arr.len {
        # компилятор генерирует VC для тела цикла один раз. Предпосылка: s удовлетворяет SumUpTo(arr, i), i удовлетворяет UpTo(arr.len).
        #
        # s += arr[i]:
        #   обязательство верификации: s_new удовлетворяет SumUpTo(arr, i) (текущее i неизменно)
        #   подстановка s_new = s_old + arr[i]:
        #     требуется s_old + arr[i] == sum(arr[0..i+1])
        #     из индуктивной гипотезы s_old == sum(arr[0..i]), прибавляем arr[i] к обеим сторонам:
        #     sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   компилятор + SMT: линейная арифметика, миллисекунды → Proved
        #
        # i += 1:
        #   i изменяется → в графе зависимостей типовая аннотация s ссылается на i → триггер повторной верификации
        #   новая цель верификации: s удовлетворяет SumUpTo(arr, i_new)
        #   т.е. s == sum(arr[0..i_new]), гарантировано предыдущим шагом → Proved
        s += arr[i]
        i += 1
    }
    return s  # теперь s: SumUpTo(arr, arr.len), т.е. s == sum(arr[0..arr.len])
}
```

Инвариант цикла — это просто типовая аннотация на переменной — программист пишет тип, компилятор проверяет индуктивный шаг. Компилятору не нужно «открывать» инвариант и не нужно «автоматически делать индукцию» — он разлагает индуктивное доказательство на локальные условия верификации для каждой операции присваивания и передаёт их в конвейер доказательств для покомпонентного решения.

#### 6.1 Отслеживание зависимостей: зависимые типы на изменяемых переменных

Предпосылка вышеизложенного механизма: компилятор знает, что типовая аннотация `SumUpTo(arr, i)` на `s` ссылается на `i` — когда `i` изменяется, типовое ограничение `s` тоже должно измениться. Для этого компилятор поддерживает **граф зависимостей типов между переменными**.

**Структура данных**:

```
TypeDepGraph: Map<VarName, Set<VarName>>
# ключ — зависимая переменная, значение — множество переменных, типовая аннотация которых ссылается на эту переменную
# пример: { i: {s}, j: {s, t}, ... }
```

**Построение**: при обработке `mut v: Pred(... x ...) = init` проверщик типов解析 `Pred(...)` на предмет свободных переменных в параметрах. Если в параметрах есть ссылка на другую изменяемую переменную `x` из текущей области видимости, в графе зависимостей записывается `x → v`.

**Триггер**: когда зависимой переменной `x` присваивается значение, компилятор:
1. Находит в графе зависимостей все переменные, зависящие от `x` — `{v₁, v₂, ...}`
2. Для каждой `v`, генерирует условие верификации: `удовлетворяет ли текущее значение v обновлённому типу Pred(... x_new ...)`
3. Отправляет VC в конвейер доказательств

**Чувствительность к порядку присваиваний**: отслеживание зависимостей естественным образом принуждает к правильному порядку присваиваний. На примере `SumUpTo(arr, i)`:

```yaoxiang
# правильный порядок
s += arr[i]   # s_new удовлетворяет SumUpTo(arr, i+1)
i += 1        # i изменяется → повторная верификация s удовлетворяет SumUpTo(arr, i_new) → True

# неправильный порядок — компилятор отвергает
i += 1        # i изменяется → повторная верификация s удовлетворяет SumUpTo(arr, i_new)
              # s ещё не обновлена, s_old == sum(arr[0..i_old]) ≠ sum(arr[0..i_new])
              # → ошибка компиляции: переменная s не удовлетворяет типу SumUpTo(arr, i_new)
s += arr[i]   # недостижимо
```

**Комбинированные зависимости**: переменная может зависеть от нескольких переменных. Типовая аннотация `{ v: Int; v == x + y }` зависит одновременно от `x` и `y` — изменение любой из них триггерит повторную верификацию.

**Связь с конвейером доказательств**: отслеживание зависимостей — это триггер генерации VC, а не независимый механизм верификации. Оно отвечает на вопрос «когда нужно генерировать VC» — конвейер доказательств отвечает на вопрос «выполняется ли VC».

### 7. Проверка终止性

Полностью автоматическая во время компиляции. Циклы, которые компилятор может доказать, проходят; те, что не может — выдают ошибку компиляции — программист должен сделать так, чтобы компилятор мог автоматически анализировать终止性 цикла. Никаких полуавтоматических аннотаций.

#### 6.1 Принципы проектирования

Компилятор автоматически извлекает информацию, необходимую для доказательства终止性, из двух источников:

1. **Типовые аннотации переменных**: граничные ограничения в уточнённых типах (например, `UpTo(n)` даёт верхнюю границу `n` и нижнюю границу `0`)
2. **Операции в теле цикла**: операции, применяемые к переменным при каждой итерации

Компилятор по приоритету пытается применить четыре стратегии синтеза меры, находит одну — останавливается.

#### 6.2 Стратегия 1: Автоматический синтез линейной ранк-функции

Когда у переменных есть линейные границы, компилятор перечисляет候选линейные меры и верифицирует их через SMT.

```
Вход:
  переменные v₁: UpTo(u₁), v₂: UpTo(u₂), ... (переменные с верхними и нижними границами)
  условие цикла cond
  множество присваиваний в теле цикла

Алгоритм:
  1. Из типовых аннотаций извлечь границы каждой переменной: [low_i, high_i]
  2. Перечислить候选меры: v_i, u_i - v_i, v_i - v_j, и другие линейные комбинации
  3. Для каждой候选меры m:
     - SMT верифицирует m ≥ 0 (из типовых границ)
     - Для каждого пути выполнения в теле цикла, SMT верифицирует m' < m (строгое убывание)
  4. Найдена линейная комбинация, удовлетворяющая условиям →终止性 доказана
```

Покрытие: циклы, где произвольная переменная получает линейное выражение (`v = a·v + b`), и есть ограниченная типовая аннотация. Включая `i += const`, `i -= const`, а также сжатие интервала в стиле бинарного поиска:

```yaoxiang
# Бинарный поиск: low = mid + 1 или high = mid
# Мера high - low строго убывает на обоих путях
binary_search: (arr: Sorted(Int, arr), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 Стратегия 2: Подсчёт нарушений предиката — автоматическое извлечение меры из целевого типа <span style="color:orange">【Экспериментальная стратегия】</span>

> ⚠️ **Текущее состояние: экспериментальная стратегия, при реализации фазы 3 будет решён вопрос о её включении в зависимости от практической осуществимости.**
> Эта стратегия эффективна для соседних перестановок (пузырьковая сортировка, сортировка вставкой), но не может автоматически доказать для не-соседних операций (partition быстрой сортировки, sift-down пирамидальной сортировки).
> Границы покрытия см. в таблице ниже. Если верификация на фазе 3 окажется нецелесообразной, эта стратегия будет удалена или понижена до будущей работы.

Ключевая интуиция: **пользовательские спецификации — это материал для рассуждений компилятора.** Компилятору не нужно встраивать понятие «что такое сортировка» — он читает определение `Sorted`, автоматически извлекает меру из определения.

```
Вход:
  Целевой тип: Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  Операции в теле цикла: соседние элементы меняются местами

Алгоритм:
  1.解析предикат определения: forall i in range: cond(i, arr)
  2. Автоматически генерировать меру: violation_count = |{ i | ¬cond(i, arr) }|
  3. Анализировать влияние операций на меру:
     - соседняя перестановка arr[j], arr[j+1] = arr[j+1], arr[j]
     - затрагивает только пары индексов j-1, j, j+1
     - если arr[j] > arr[j+1] (нарушение предиката), после перестановки эта пара удовлетворяет предикату
     - violation_count уменьшается как минимум на 1
  4. Верхняя граница: n·(n-1)/2 (максимальное число соседних инверсий), нижняя граница: 0
  →终止性 доказана
```

**Текущее покрытие**:

| Алгоритм | Режим операции | Стратегия 2 доказывает? | Причина |
|----------|---------------|:---:|---------|
| Пузырьковая сортировка | соседняя перестановка | ✅ | violation_count при каждой перестановке строго убывает |
| Сортировка вставкой | соседнее перемещение | ✅ | каждое перемещение устраняет одно нарушение |
| Сортировка выбором | не-соседняя перестановка | ❌ | одна перестановка может увеличить violation_count |
| Быстрая сортировка | partition | ❌ | не-соседняя перестановка, не гарантирует монотонное убывание |
| Пирамидальная сортировка | sift-down | ❌ | древовидная операция, violation_count не монотонен |

**Дополняющие стратегии**: для быстрой сортировки сжатие интервала `low < high` покрывается стратегией 1 (линейная ранк-функция) — внешний partition рекурсивно уменьшает интервал вдвое. Стратегии 1 и 2 взаимно дополняют друг друга,终止性 большинства практических алгоритмов может быть доказана одной из них. Но обобщение стратегии 2 (не-соседние операции, древовидные операции) остаётся открытой проблемой.

```yaoxiang
sort: (arr: Array(Int)) -> (result: Sorted(result)) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 Стратегия 3: Паттерн ограниченного возрастания/убывания

`v += const` (положительная константа), у переменной есть верхняя граница типовая аннотация → мера `upper_bound - v` убывает на `const` каждую итерацию, нижняя граница 0. Это вырожденный случай стратегии 1, компилятор обрабатывает его первым для быстрого решения.

#### 6.5 Стратегия 4: Мера с мультипликативным масштабированием

`v *= const` (const > 1), у переменной есть верхняя и нижняя границы типовая аннотация. Компилятор имеет встроенный шаблон логарифмической меры `ceil(log_const(upper/v))`, при каждом умножении на const мера уменьшается на 1.

```yaoxiang
mut i: Positive(i) = 1
while i < n {
    # компилятор автоматически выводит: мера ceil(log₂(n/i)), при каждом умножении на 2 мера уменьшается на 1
    i *= 2
}
```

#### 6.6 Разделение终止性 и корректности

Доказательство终止性 и доказательство корректности независимы:
- **终止性**: четыре вышеописанные стратегии автоматически доказывают, что цикл завершается за конечное число шагов
- **Корректность**: продвигает ли тело цикла состояние к целевому типу, проверяется компиляторным конвейером доказательств через условия верификации

Оба прошли → компиляция успешна.终止性 доказана, но корректность не прошла → ошибка компиляции + контрпример. Корректность доказана, но终止性 не может быть доказана → ошибка компиляции с указанием неанализируемой переменной или операции. Оба не прошли → ошибка компиляции с раздельным сообщением о причинах неудачи каждого.

#### 6.7 Проверка终止性 рекурсивных функций

Для рекурсивных функций, которые требуют вычисления во время компиляции, компилятор проверяет убывание параметров:

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # компилятор анализирует: n-1 < n → убывает →终止на
}

# использование во время компиляции — компилятор гарантирует终止性 factorial во время компиляции
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120, вычислено во время компиляции
```

| Сценарий | Поведение |
|----------|-----------|
| Компилятор может проанализировать рекурсивное убывание (например, `n-1`) | Вычисление во время компиляции |
| Не убывает / невозможно определить убывание | Ошибка компиляции |
| Вызов во время выполнения (не в типовой позиции) | Проверка终止性 не требуется |

#### 6.8 Жёсткие границы

`i = f(i)` и `f` необратима, не замкнута, не сохраняет монотонность — математически невозможно автоматически доказать终止性. Ошибка компиляции:

> Этот цикл невозможно автоматически доказать как завершающийся. Переменная цикла зависит от неанализируемой функции `f`. Пожалуйста, используйте итерационный паттерн, поддающийся анализу компилятора.

Это не неудача компилятора. Весь код, безопасность которого невозможно статически доказать, не компилируется.

### 8. SMT-решатель: ускорительный модуль проверщика типов

В традиционных языках SMT-решатель — это внешний инструмент (как F* вызывает Z3, Dafny вызывает Z3). В YaoXiang это **ускорительный модуль проверщика типов** — вызывается только когда само ядро компилятора не может напрямую определить результат. SMT помогает найти доказательство, но верифицирует доказательство проверщик типов.

**Модель доверия**: проверщик типов — единственный корень доверия. SMT-решатель — ускорительный модуль — он помогает найти доказательство, но SMT не является отдельной границей доверия. Компилятор доверяет результату `unsat` от Z3 (в соответствии с подходом F*/Dafny — вероятность ошибки Z3 ниже, чем вероятность бага самого компилятора, это прагматичный инженерный выбор). Настоящая ненадёжность контролируется на уровне трансляции в SMT — если в трансляции есть баг, компилятор выявит его на других тестах.

**Интерфейс**: компилятор внутренне транслирует в стандартный формат SMT-LIB 2.6, а не привязывается к конкретному API решателя. SMT-LIB — это стандарт ISO, Z3, CVC5, MathSAT, Yices все нативно его поддерживают.

**Бэкенд по умолчанию**: Z3 (лицензия MIT, наиболее широкая документация и верификация сообществом). CVC5 как совместимая с SMT-LIB альтернатива — пользователь может переключить через флаг компилятора.

Никакого «универсального уровня абстракции решателя» — SMT-LIB и есть уровень абстракции. Если в будущем CVC5 сделает прорыв в конкретной теории, переключение — это просто замена бинарника, без изменения кода компилятора.

```
Bool-выражение времени компиляции
        │
        ├── Ядро компилятора может определить напрямую (структурная эквивалентность, простая арифметика,
        │   тривиальные формулы после константного сворачивания)
        │   → напрямую возврат Proved / Disproved
        │
        └── Ядро компилятора не может определить напрямую (кванторы, символьные переменные)
            → предварительная редукция зависимых типов (factorial(5) → 120)
            → трансляция в формат SMT-LIB
            → отправка в Z3/CVC5 (с лимитом бюджета)
            → возвращаемое значение: unsat → Proved │ sat + модель → Disproved │ unknown → Unproven
```

**Бюджет решения — жёсткий лимит, как глубина стека**:

| Измерение бюджета | Значение по умолчанию | Описание |
|-------------------|----------------------|---------|
| Число шагов решения | 10,000 | Z3 для линейной арифметики обычно за несколько сотен шагов. 10,000 шагов покрывают 99% практических предикатов. |
| Время | 100ms | Один предикат более 100ms = пользователь пишет программу времени компиляции, а не типовую аннотацию. 100ms × 50 предикатов = 5 секунд верхнего предела времени компиляции. |
| Глубина инстанцирования кванторов | 3 | Три уровня вложенных кванторов покрывают практические паттерны. Более трёх уровней — скорее всего логические упражнения. |

При превышении бюджета возврат Unproven, ошибка компиляции + позиция предиката + потреблённый объём. Никакой деградации, никаких проверок во время выполнения, никакого silent pass.

**Почему это практически осуществимо**: в инженерной практике 95% практических предикатов — это линейная арифметика — `x > 0`, `arr.len > 0`, `0 <= idx < arr.len` — всё в разрешимом фрагменте, SMT-решатель возвращает за миллисекунды. Когда встречаются极少数 сложных предикатов, превышающих бюджет, программист просто пишет доказательственную функцию.

Перед вызовом SMT выполняется предварительная редукция зависимых типов: `factorial(5)` напрямую вычисляется во время компиляции в `120`, `append([1,2], [3])` напрямую вычисляется в `[1,2,3]`. Эти детерминированные вычисления значений не потребляют бюджет SMT.

Программисту не нужно знать о существовании SMT. Ментальная модель: **компилятор может доказать — проходит, не может — ошибка — если компилятор не может, вы можете написать функцию и доказать ему.**

### 9. Композиция компиляторных предикатов

Компиляторный предикат — это функция, возвращающая Type, композиция естественно осуществляется через функциональную композицию:

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(arr)
}
```

### 10. Примеры кода

#### 9.1 Безопасное деление

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b

result = divide(10, 2)   # ✅ компилятор верифицирует Positive(2) = { 2 > 0 } → True
# result = divide(10, 0)  # ❌ компилятор верифицирует Positive(0) = { 0 > 0 } → False
```

#### 9.2 Безопасность доступа к массиву

```yaoxiang
InBounds: (idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (arr: Array(T), idx: InBounds(idx, arr)) -> T = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ компилятор верифицирует InBounds(1, arr) = { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ компилятор верифицирует InBounds(5, arr) = { 0 <= 5 && 5 < 3 } → False
```

#### 9.3 Корректность сортировки

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T))) -> (result: Sorted(T, result)) = {
    result = arr.clone()
    # ... реализация алгоритма сортировки ...
    return result
}
```

#### 9.4 Циклы: генерация VC компилятором

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```


### 11. Диспетчерский конвейер dispatch: унифицированная диспетчеризация времени компиляции и выполнения

`assert` и `Assert` — это две стороны одной уточняющей примитивы типа. Конвейер диспетчеризации `dispatch` автоматически решает по **свободным переменным предиката, доступны ли они во время компиляции**, идти ли по пути доказательства во время компиляции или проверки во время выполнения:

| Критерий | Режим | Поведение |
|----------|-------|-----------|
| Все свободные переменные известны во время компиляции (параметры обобщённого типа, константы времени компиляции) | **CompileTime** | Вход в конвейер доказательств: Proved → стирание, Disproved → ошибка компиляции, Unknown → требуется доказательство |
| Существуют свободные переменные из времени выполнения (параметры функции, внешний ввод, mut переменные) | **Runtime** | Вставка проверки во время выполнения и инъекция уточнённого факта в потоково-чувствительное множество предположений Γ |

**Ключевой момент**: «не могу определить» ≠ «опровергнуто». В режиме CompileTime Unknown требует доказательства (без тихого понижения), в режиме Runtime утверждение во время компиляции вообще не имеет истинностного значения — никакой prover не сможет написать тавтологию для «пользователь может ввести отрицательное число», проверка во время выполнения — единственный sound выбор. Это не слабость prover'а, это теоретическая неизбежность.

### 12. Потоково-чувствительное множество предположений Γ: распространение наиболее сильного постусловия

Компилятор поддерживает потоково-чувствительное (flow-sensitive) множество предположений Γ, отслеживающее известные истинные утверждения в каждой точке потока управления.

**Распространение SP (наиболее сильное постусловие)**:

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
y = x + 1           // Γ = {x > 0, y > 1}  ← распространение SP
```

**Kill set для mut переменных**: после переприсваивания `mut` переменной все предположения, затрагивающие эту переменную, удаляются из Γ:

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
mut x = x - 5       // Γ = {}  ← x > 0 убито
```

Это жёсткое требование soundness — значение переменной изменилось, старые предположения недействительны.

**Слияние веток**: при слиянии IF/ELSE или веток match, Γ берёт пересечение предположений веток. Только утверждения, выполняющиеся на всех путях, выходят из ветки.

### 13. Уточнение модели стирания: стирание witness ≠ стирание check

Утверждение RFC-027 о том, что «уточнённые типы **полностью стираются** во время выполнения», относится к **proof witness (доказательский токен)** — проверенные во время компиляции доказательские элементы не генерируют кода времени выполнения. Но **проверка времени выполнения**, вставляемая dispatch в режиме Runtime, сохраняется — это Bool-проверка на уровне значений, а не witness на уровне типов.

Резюме: witness стирается, check сохраняется. Две вещи не конфликтуют, исходное утверждение RFC-027 остаётся в силе.

## Детальное проектирование

### Синтаксические изменения

| Раньше (RFC-022) | Теперь (этот RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | Компиляторный предикат как тип параметра `(b: Positive(b))` |
| `//! ensures: ExistsMax(result, arr)` | Возврат через параметр возвращаемого значения `-> (result: IsMax(T, arr, result))` |
| `/*! invariant: ... !*/` | Типовые аннотации с компиляторными предикатами на переменных — инварианты Флойда-Хора |
| `//! decreases: n` | Полностью автоматический вывод функции меры компилятором |
| Спецификации — комментарии | Спецификации — типовая система |

### Синтаксис

**Для компиляторных предикатов новых ключевых слов нет.** `{}` — это пространство доказательств, полностью согласовано с существующим синтаксисом определения типов. Компиляторный предикат — это функция, возвращающая Type — `name: (params) -> Type = { утверждения }`. При использовании — просто вызов функции — `Positive(b)`, `IsMax(T, arr, result)`.

```bnf
# Компиляторный предикат = функция, возвращающая Type, {} — это утверждения, верифицируемые компилятором
# Используется существующий синтаксис функций/типов, новые BNF-правила не нужны
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

**Новая синтаксическая концепция: параметр возвращаемого значения** — в `-> (name: Type)` `name` — это параметр возвращаемого значения.

Параметр возвращаемого значения — это **единственная синтаксическая концепция**, которую YaoXiang вводит поверх существующего синтаксиса функций. Её семантика:
- Значение `name` предоставляется оператором `return`
- `name` существует только в типовой сигнатуре, упоминается только постусловным предикатом (например, `-> (result: IsMax(T, arr, result))`)
- `name` не входит в область видимости тела функции, не появляется у вызывающей стороны
- Параметр возвращаемого значения **опционален** — без постусловия сигнатура полностью идентична обычной функции (`-> Int`), не создаёт никакой дополнительной нагрузки

Причина его введения: постусловию нужно ссылаться на «значение, которое функция вернёт». Без параметра возвращаемого значения компилятор мог бы использовать только специальные правила (неявные переменные вроде `$result` или `__retval__`), чтобы предикат ссылался на возвращаемое значение. Параметр возвращаемого значения делает эту ссылку явной — это просто параметр, только значение предоставляется `return`, а не вызывающей стороной.

**Доказательственная функция** — это не новая концепция — это просто функция YaoXiang, тип возврата которой равен доказываемому утверждению. Когда компилятор возвращает Unproven, программист предоставляет доказательственную функцию, проверщик типов верифицирует её точно так же, как верифицирует тип возврата любой другой функции. Никакого нового синтаксиса, новых ключевых слов, новых правил.

### Влияние на систему типов

- **Типовые вселенные**: компиляторные предикаты расположены на уровне Type₂ — функции, принимающие значение и возвращающие Type, на том же уровне, что и конструкторы типов
- **Взаимодействие с обобщениями**: компиляторные предикаты могут иметь обобщённые параметры, например `NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **Взаимодействие с владением**: выражения в компиляторных предикатах подчиняются правилам владения, могут только читать, не могут писать
- **Вывод типов**: параметры компиляторных предикатов участвуют в HM-выводе типов

### Представление во время выполнения

Компиляторные предикаты во время выполнения обрабатываются **в соответствии с результатом диспетчеризации dispatch**:
- **Режим CompileTime** (все свободные переменные известны во время компиляции): после подтверждения доказательства witness полностью стирается. `Positive: (x: Int) -> Type = { x > 0 }` — параметр `b: Positive(5)` во время выполнения представляется просто как `Int`. Уточнённое условие `{ 5 > 0 }` проверено, стёрто.
- **Режим Runtime** (существуют свободные переменные времени выполнения): сохраняется проверка времени выполнения — выполняется Bool-проверка на уровне значений, инъекция в потоково-чувствительное множество предположений Γ. Подробности см. в §11 конвейер диспетчеризации dispatch и §13 уточнение модели стирания.

Размещение компиляторного предиката в типовой позиции (например, `f(x: Positive(x))`) не создаёт обёрточного типа, не выделяет дополнительной памяти. Но когда `x` происходит из ввода времени выполнения, **вставляется** проверка Bool во время выполнения.

**Ограничение взаимодействия с `ref`**: компиляторный предикат может ссылаться только на неизменяемое заимствование или на значение, владение которым передано. Значение, на которое есть изменяемое заимствование, не может появляться в компиляторном предикате — компилятор не может гарантировать во время компиляции, что результат верификации останется верным во время выполнения — такое использование вызывает直接ную ошибку компиляции.

### Изменения в компиляторе

1. **Парсер**: компиляторные предикаты используют стандартный синтаксис функций, дополнительные правила разбора не нужны
2. **Компиляторный конвейер доказательств**: унифицированный интерфейс возврата Proved/Disproved/Unproven, автоматический выбор стратегии
3. **Ускорительный модуль SMT**: уровень трансляции в SMT-LIB 2.6, бэкенд по умолчанию Z3, альтернатива CVC5
4. **Ядро проверщика типов**: реализация правил вывода — структурная эквивалентность, βδι-редукция, введение/удаление универсальных кванторов. Это единственный корень доверия, через него верифицируются и SMT, и программистские доказательства
5. **Генерация условий верификации**: исчисление WP/SP + обязательства по доказательству инвариантов циклов
6. **Сообщения об ошибках**: форматирование контрпримеров + отчёты о нерешённых утверждениях + связывание с позициями исходного кода

### Обратная совместимость

- ✅ Код, не использующий компиляторные предикаты, полностью неизменен
- ✅ Компиляторные предикаты в режиме CompileTime имеют нулевые накладные расходы во время выполнения, в режиме Runtime сохраняют только необходимые Bool-проверки
- ⚠️ Синтаксис `//!` из RFC-022 более не поддерживается — но 022 никогда не был реализован, миграционная нагрузка отсутствует

## Компромиссы

### Преимущества

- **Полная реализация соответствия Curry-Howard**: тип — это утверждение, программа — это доказательство, `name: Proposition = Proof`
- **Унификация**: компиляторные предикаты и обычные функции используют полностью идентичный синтаксис, нет концептуального разделения
- **Прозрачность SMT**: программисту не нужно знать о существовании SMT, ментальная модель согласована с проверкой типов
- **Постепенное принятие**: можно начать с одного компиляторного предиката и постепенно расширять покрытие
- **Минимальные накладные расходы во время выполнения**: режим CompileTime имеет нулевые накладные расходы, режим Runtime сохраняет только необходимые Bool-проверки

### Недостатки

- **Время компиляции**: SMT-решение увеличивает время компиляции, но жёсткий лимит бюджета гарантирует контролируемый верхний предел
- **Границы автоматического доказательства**: сложные предикаты за пределами логики первого порядка и линейной арифметики могут требовать от программиста написания доказательственной функции. Это не недостаток языка — это неизбежный вывод из проблемы остановки. Компилятор честно сообщает Unproven, а не выдаёт ложное True/False
- **Кривая обучения**: написание эффективных компиляторных предикатов и доказательственных функций требует понимания базовой интуиции соответствия Curry-Howard
- **Сложность реализации**: унификация компиляторного конвейера доказательств требует тщательного проектирования

### Смягчение рисков

- Жёсткий лимит бюджета SMT-решения (шаги 10,000 / время 100ms / глубина инстанцирования 3), при превышении бюджета возврат Unproven
- Предварительная редукция зависимых типов: детерминированные вычисления значений выполняются первыми, SMT получает только недетерминированную часть
- Unproven — не тупик: программист может написать доказательственную функцию, проверщик типов верифицирует — и верифицирует точно так же, как верифицирует тип возврата любой функции
- Инкрементальная верификация: верифицируются только изменённые модули
- Чёткие сообщения об ошибках +展示контрпримеров + отчёты о потреблении бюджета + нерешённые утверждения + предложения (если компилятор может их дать)

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| RFC-022: спецификации в стиле `//!` комментариев | Спецификации разделены с типами, нарушение соответствия Curry-Howard |
| Отдельные файлы спецификаций (как CVL) | Спецификации отделены от кода, увеличение стоимости сопровождения |
| Только проверки времени выполнения | Невозможна статическая гарантия корректности |
| Внешние помощники доказательства (как Coq) | Разрыв с компилятором, требуется отдельный язык доказательств и граница доверия. Выбор YaoXiang: доказательство — это код YaoXiang, проверщик типов — единственный корень доверия |
| **Наше решение: компиляторные предикаты как объекты первого класса** | ✅ |

## Открытые вопросы

- [x] **Выбор SMT-решателя**: по умолчанию Z3 (лицензия MIT, наиболее широкая верификация). CVC5 как SMT-LIB-совместимая альтернатива, переключение через флаг компилятора. Внутренний транслятор компилятора нацелен на стандартный формат SMT-LIB 2.6 — SMT-LIB и есть уровень абстракции, не делается кастомный универсальный интерфейс решателя.
- [x] **Конкретные значения бюджета решения**: шаги 10,000 / время 100ms / глубина инстанцирования кванторов 3. Фиксированы внутри компилятора, ручек нет. Если на практике окажется недостаточно (реальные用例而不是«пользователь ошибся»), скорректировать.
- [x] **Область поддержки кванторов**: на уровне языка порядок кванторов не ограничивается. Компиляторные предикаты принимают параметры типа Type — Type включает функциональные типы — поэтому кванторы высших порядков являются естественным следствием системы типов, не требуют специального синтаксиса. SMT-решатель может автоматически разрешать кванторы первого порядка (forall/exists, поддержка чередующихся вложений, ограничено бюджетной глубиной 3). Кванторы высших порядков: SMT возвращает Unproven, компилятор предлагает «этот предикат за пределами автоматического доказательства, пожалуйста, предоставьте доказательственную функцию». Программист пишет функцию, тип возврата которой равен этому утверждению — проверщик типов верифицирует эту функцию. Никакого внешнего экспорта, никакого AI, никакого интерактивного режима доказательства. Всё — код YaoXiang, всё верифицируется проверщиком типов.
- [x] **Форматирование контрпримеров**: имена переменных исходного кода напрямую используются как SMT-имена переменных (плюс префикс модуля во избежание конфликтов). При возврате модели Z3 — обратный поиск по имени переменной. Формат вывода: имя переменной = конкретное значение + позиция в исходном коде + позиция определения предиката. Никаких сложных слоёв映射.
- [x] ~~**Взаимодействие компиляторных предикатов с `ref` умными указателями?**~~ → Решено: компиляторные предикаты допускают только неизменяемое заимствование или переданное владение. Значения с изменяемым заимствованием не могут появляться в компиляторных предикатах.
- [x] **Расширение меры подсчёта нарушений forall-предикатов на не-соседние операции?** → Не расширять. Текущее покрытие (соседние перестановки, соседние перемещения) взаимно покрывается стратегией 1 (линейная ранк-функция) — внешнее сжатие интервала быстрой сортировки兜底 через стратегию 1, пирамидальная сортировка兜底 через стратегию 1 (паттерн индексов массива). Циклы,终止性 которых не может быть доказана ни одной стратегией, компилятор напрямую выдаёт ошибку — это философия жёсткой безопасности, а не недостаток. Если в будущем появятся реальные（非学术构造） алгоритмы, которые не покрываются четырьмя стратегиями, обсудим повторно.
- [x] **Комбинаторный взрыв перечисления линейных ранк-функций**: верхний предел перечисления候选— 3 ограниченные переменные. ≤3 — перечисление всех линейных комбинаций и SMT-верификация по одной. >3 — только попытка одно-переменных мер (`v_i`, `u_i - v_i`), при неудаче直接报 ошибка компиляции — подсказка программисту «в цикле >3 ограниченных переменных, компилятор не может автоматически синтезировать многопеременную меру». Это не инженерный компромисс — это вынуждение программиста писать более простые циклы.
