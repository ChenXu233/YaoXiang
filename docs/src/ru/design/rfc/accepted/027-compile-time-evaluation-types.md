# RFC-027: Предикаты времени компиляции и единая статическая верификация

>
> - [RFC-009: Модель владения](../accepted/009-ownership-model.md)

## Резюме

В данном документе предлагается ввести в YaoXiang **предикаты времени компиляции** как полноправные сущности первого класса, объединив всю статическую верификацию времени компиляции в единый **конвейер доказательств**. Предикаты времени компиляции — это не внешние аннотации спецификаций — это функции. Функция, возвращающая Type, может использоваться в позиции типа; компилятор вызывает её во время компиляции и проверяет возвращаемое значение. Тип — это утверждение, вычисление времени компиляции — это доказательство.

**Ключевой тезис**: единственная работа проверки типов во время компиляции — конструировать и верифицировать термы доказательств. Равенство типов, конфликты токенов, редукция зависимых типов, вычисление предикатов времени компиляции, импликации логики Хоара — всё это различные проверки типов в конвейере доказательств времени компиляции, использующие один и тот же конвейер. SMT-решатель — это ускоряющий модуль средства проверки типов, а не отдельная граница доверия. Когда компилятор возвращает Unproven, программист пишет функцию на YaoXiang в качестве доказательства — средство проверки типов верифицирует её точно так же, как проверяет любой возвращаемый тип функции. Всё — это код на YaoXiang, всё верифицируется средством проверки типов.

## Мотивация

### Почему RFC-022 устарел?

RFC-022 проектировал спецификации в форме комментариев `//!`:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← Это аннотация, отдельная от типа
    //! ensures: ExistsMax(result, arr[0..n])   ← Это аннотация, отдельная от типа
}
```

Это фундаментальная ошибка изоморфизма Карри-Ховарда: **разделение спецификации и типа на два слоя**. Аннотации — это не типы. Аннотации не участвуют в проверке типов. Аннотации — это ментальная модель «внешнего инструмента».

В белой книге сказано чётко:

> «Нет комментариев `//!`. Нет отдельного языка спецификаций. Всё внутри системы типов.»

### Текущие проблемы

- Аннотации `//!` из RFC-022 — это внешний синтаксис, отдельный от системы типов
- Типы спецификаций и обычные типы — это две разные системы, что создаёт концептуальную избыточность
- Модель с разделением Debug Build верифицирует / Release Build игнорирует нарушает единство
- SMT-решатель в традиционном понимании позиционируется как внешний инструмент — YaoXiang встраивает его как ускоряющий модуль средства проверки типов
- Проверка типов, верификация заимствований, проверка предикатов времени компиляции, раскрытие макросов идут по разным путям

### Правильная ментальная модель

Проверку типов можно абстрагировать как функцию:

```
verify : Program → Proved | Disproved(Model) | Unproven
```

Все проверки времени компиляции — простое сопоставление типов, обнаружение конфликтов заимствований, верификация предикатов времени компиляции — являются подзадачами этой функции. Они разделяют один конвейер доказательств и отличаются лишь сложностью терма доказательства и стратегией конструирования.

Когда компилятор возвращает Unproven, программист предоставляет функцию-доказательство — возвращаемый тип которой равен доказываемому утверждению. Средство проверки типов верифицирует её. Это та же операция, что и обычная проверка типов.

## Предложение

### 1. `{}` — пространство доказательств: тип — это утверждение, верификация — это проверка типов

`{}` в YaoXiang — это пространство доказательств времени компиляции. Всё внутри — это утверждения; компилятор гарантирует, что каждое из них истинно — либо автоматическим доказательством, либо предоставлением функции-доказательства программистом.

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  Компилятор гарантирует, что x — Float, y — Float

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  Компилятор гарантирует, что data — Array(T)
```

**Дженерики — это частный случай предикатов времени компиляции.**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          Параметр в сигнатуре   Внутри {} только утверждения
#          Компилятор верифицирует x > 0 во время компиляции

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      Параметр в сигнатуре   Компилятор верифицирует type_of(T) == Type, type_of(data) == Array(T)
```

Тот же шаблон: `name: (params) -> Type = { утверждения }`. Компилятор не различает «утверждения о типах» и «утверждения о значениях» — все они являются целями вычисления в конвейере доказательств.

**Инварианты цикла не нужно записывать отдельно. Аннотации типов на переменных — это инварианты Флойда-Хоара.**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # Аннотация ссылается на i — сообщает компилятору, что тип s зависит от i
    mut i: UpTo(arr.len) = 0     # При инициализации i=0, верификация: 0 == sum(arr[0..0]) → True
    while i < arr.len {
        s += arr[i]  # Компилятор верифицирует: s_new == sum(arr[0..i+1])
        i += 1       # Изменение i → запуск реверификации зависимости s: s удовлетворяет SumUpTo(arr, i_new)
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

Компилятор генерирует условие верификации для тела цикла один раз — индуктивная гипотеза (аннотация типа) → операция присваивания → удовлетворяет ли новое значение аннотации типа. После того как конвейер доказательств верифицирует индуктивный шаг, все итерации покрываются автоматически. Не нужно `: decreases`, не нужно `: Invariant`, не нужно доказательство индукции — компилятор разбивает индукцию на локальные VC для каждого присваивания.

### 2. Предусловия/постусловия: предикаты времени компиляции в типах параметров и возврата

Отказываемся от `//! requires`/`//! ensures` из RFC-022. Предикаты времени компиляции выступают в роли аннотаций типов параметров или возвращаемого значения.

**Сторона параметра — это вызов функции.** Предикат времени компиляции — это функция, возвращающая Type; использование на стороне параметра — это её вызов, точно так же как `factorial(5)`. Сторона возвращаемого значения вводит новую концепцию: параметр возвращаемого значения.

```yaoxiang
# Предусловие: явный вызов предиката времени компиляции в типе параметра
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b
#                       ^^^^^^^^^^  b — имя текущего формального параметра, передаётся в Positive как аргумент
#                       Компилятор в точке вызова извлекает значение фактического параметра, подставляет в b, верифицирует Positive(факт. парам.)
#                       Пример: divide(10, 2) → верификация Positive(2) = { 2 > 0 } → True
#                       Пример: divide(10, 0) → верификация Positive(0) = { 0 > 0 } → False → ошибка компиляции

# Постусловие: параметр возвращаемого значения + предикат времени компиляции
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

NonEmpty: (arr: Array(T)) -> Type = { arr.len > 0 }

max: (T: Ord) -> ((arr: NonEmpty(arr))) -> (result: IsMax(T, arr, result)) = {
#                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                            result — параметр возвращаемого значения, значение предоставляется оператором return
#                                            Компилятор в точке return подставляет возвращаемое значение, верифицирует постусловие
    candidate = arr[0]
    for i in 1..arr.len {
        if arr[i] > candidate { candidate = arr[i] }
    }
    return candidate
}
```

**Ключевые правила**:

- **Сторона параметра**: `b: Positive(b)` — `b` — это имя текущего формального параметра, передаваемое в `Positive` как аргумент. Синтаксис вызова функции, ноль неявного.
- **Сторона возврата**: `-> (result: IsMax(T, arr, result))` — `result` — это параметр возвращаемого значения, значение которого предоставляется оператором `return`. `result` существует только в сигнатуре типа, на него ссылается только предикат, не попадает в область видимости тела функции, не появляется у вызывающей стороны.
- **Параметр возвращаемого значения опционален**: при отсутствии постусловия не пишется, сигнатура полностью совпадает с обычной функцией (`-> Int`).
- **Единство**: параметры и параметры возвращаемого значения — это одна и та же концепция — `имя_параметра: вызов_предиката(имя_параметра)`, различие лишь в том, предоставляется ли значение вызывающей стороной или оператором `return`.

### 3. Распространение путевых условий: верификация времени компиляции для значений времени выполнения

Предикаты времени компиляции при использовании в позиции связывания получают параметры, явно переданные программистом. Когда значение времени выполнения попадает в параметр уточнённого типа, компилятор выполняет верификацию путём сбора путевых условий и SMT-импликации — без явной передачи доказательства программистом.

#### 3.1 Явный вызов функции

Предикаты времени компиляции при использовании в позиции связывания получают параметры, явно переданные программистом — это просто вызов функции, ноль неявного.

`Positive: (x: Int) -> Type = { x > 0 }` — это конструктор предикатов времени компиляции. Когда он появляется в позиции связывания (объявление параметра, объявление переменной, возвращаемый тип), программист явно передаёт имя уже связанной переменной:

```yaoxiang
b: Positive(b)
// b уже объявлен как текущий формальный параметр, Positive(b) — это вызов функции
// После нормализации: b: { b > 0 }
```

Компилятору не нужно неявно заполнять параметр — `b: Positive(b)` — это то же самое, что `f(5)`, просто вызов функции. `b` связывается как имя параметра, аннотация его типа `Positive(b)` ссылается на сам `b` — это стандартный шаблон зависимых типов, а не правило неявного раскрытия.

**Единство с `self` из RFC-010**: RFC-010 устанавливает, что `self` — не ключевое слово, а просто конвенция для имени параметра («написать `p`, `this`, `x` — эффект полностью тот же»). `b: Positive(b)` использует тот же механизм — имя параметра может быть использовано в аннотации типа. `self` появляется в позиции `self: Point`, `b` появляется в позиции `b: Positive(b)`, обе аннотации типа ссылаются на сам параметр. Различие лишь в сложности аннотации типа, механизм полностью тот же — после связывания имени тип может зависеть от этого имени.

Возвращаемый тип также использует явный вызов функции:

```yaoxiang
Sorted: (arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (arr: Array(T)) -> (result: Sorted(result)) = { ... }
//                        ^^^^^^^^^^^^^^^^^^^^^^^
//                        result — параметр возвращаемого значения, Sorted(result) — вызов функции
//                        Компилятор в точке return подставляет возвращаемое значение в result, верифицирует Sorted(возвращ. значение)
```

То же применимо к объявлениям локальных переменных:

```yaoxiang
let x: Positive(x) = 5
// x связывается с 5, Positive(5) → { 5 > 0 } → True → проходит

// let y: Positive(y) = 0
// y связывается с 0, Positive(0) → { 0 > 0 } → False → ошибка компиляции
```

#### 3.2 Сбор путевых условий

Когда значение времени выполнения появляется в условной ветви, компилятор автоматически собирает путевые условия, формируя **множество предположений** текущей области видимости. Эти предположения участвуют в верификации как фоновые знания для вычисления Bool времени компиляции.

```yaoxiang
if y > 0 {
    // Компилятор автоматически получает предположение в этой ветви: { y > 0 }
    let result = divide(x, y)
    // Условие верификации: (y > 0) ⇒ (y > 0)
    // Конвейер доказательств проверяет импликацию → Proved
} else {
    // Предположение в этой ветви: { !(y > 0) }
    // При вызове divide(x, y) условие верификации: !(y > 0) ⇒ y > 0
    // Конвейер доказательств: не выполняется → Disproved
}
```

Это не специальный шаблон, зашитый в компиляторе — это естественное поведение конвейера доказательств времени компиляции. Каждая проверка типов в точке вызова отправляет в конвейер:

```
{фоновые_предположения} ⇒ {цель_верификации}
```

Конвейер доказательств проверяет импликацию. Proved → проходит, Disproved → ошибка компиляции + контрпример, Unproven → ошибка компиляции + неразрешённое утверждение. Фоновые предположения поступают из путевых условий текущей точки программы.

#### 3.3 Стек предположений

Компилятор при анализе потока управления поддерживает множество предположений для каждого базового блока:

- **if-guard**: `if y > 0` → true-ветвь добавляет `y > 0`, false-ветвь добавляет `!(y > 0)` (если используется else)
- **match-шаблоны**: `if let Some(v) = opt` → в ветвь добавляется `opt == Some(v)`
- **Логические связки**: `if x > 0 && y < 10` → в ветвь добавляются `x > 0` и `y < 10`
- **Предусловия функций**: при вызове `divide(a, b)` `b` должен удовлетворять `Positive` либо из текущих предположений, либо из аннотации уточнённого типа самого аргумента (если `b` аннотирован как `Positive`, его тип несёт `b > 0`)
- **Присваивание**: при `let z = y` уже имеющиеся уточняющие условия `y` передаются `z`

Все предположения поступают в конвейер доказательств времени компиляции. При переходе на путь SMT-ускорения они транслируются в фоновые утверждения SMT-LIB.

#### 3.4 Отсутствие статического свидетельства — ошибка компиляции

Если программист пишет напрямую:

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

В текущей точке программы нет предположения `y > 0`, сам аргумент `y` не имеет аннотации типа `Positive`. Условие верификации:

```
{} ⇒ { y > 0 }
```

Конвейер возвращает `Disproved` (импликация не выполняется) → ошибка компиляции:

> Невозможно доказать, что параметр `b` в вызове `divide` удовлетворяет `Positive`.
> `y` поступает из ввода функции, доказанных границ не имеется.
> Рассмотрите использование if-ветви для охраны вызова: `if y > 0 { divide(x, y) }`.

YaoXiang не принимает значения времени выполнения, попадающие непосредственно в параметры уточнённого типа, без предоставления статического свидетельства. Это не ограничение — это ядро философии жёсткой безопасности. Любой код, который компилятор не может статически доказать, не проходит компиляцию.

#### 3.5 Отношение с единым конвейером

Распространение путевых условий — не дополнительный механизм. Это прямое расширение конвейера доказательств времени компиляции в анализе потока управления:

| Фаза | Обязанности |
|------|------------|
| Сбор путевых условий | Фаза анализа потока управления компилятора, аннотирование множеств предположений для каждого базового блока |
| Генерация условий верификации | При встрече ограничения типа, требующего верификации, объединение путевых условий + информации о типе аргумента |
| Вычисление в конвейере доказательств | Ядро компилятора → SMT-ускорение → получение Proved / Disproved / Unproven |
| Результат | `Proved` → проходит; `Disproved` → ошибка компиляции + контрпример; `Unproven` → ошибка компиляции + неразрешённое утверждение (программист может предоставить функцию-доказательство) |

Никаких новых компонентов. Никаких специальных правил. Путевые условия — это фоновые знания конвейера доказательств — разделяющие один конвейер и одну систему бюджетов с равенством типов и ограничениями заимствований.

### 4. Конвейер доказательств времени компиляции

Все проверки времени компиляции разделяют один конвейер. Основная операция конвейера — **проверка типов** — проверка того, равен ли тип терма доказательства доказываемому утверждению. Всё есть проверка типов.

```
[Bool-выражение, требующее вычисления времени компиляции (= требуется сконструировать терм доказательства)]
        │
        ├── Равенство типов (T1 == T2)
        │   → Прямое решение компилятора (структурная эквивалентность)
        │
        ├── Условия конфликта токенов (!conflicting(tokens))
        │   → Потокочувствительный анализ живости (отслеживание Dup/Linear атрибутов)
        │
        ├── Редукция зависимых типов (упрощение n + m)
        │   → Система переписывания термов времени компиляции (βδι-редукция)
        │
        ├── Предикаты времени компиляции (x > 0, forall...)
        │   → Ядро компилятора + модуль SMT-ускорения
        │
        └── Импликации логики Хоара (P ⇒ Q)
            → Ядро компилятора + модуль SMT-ускорения
                    │
                    ▼
             ┌──────────┐
             │ Proved   │  → Компиляция проходит
             │ Disproved│  → Ошибка компиляции + контрпример
             │ Unproven │  → Ошибка компиляции + неразрешённое утверждение
             └────┬─────┘
                  │
                  ▼
         Программист пишет функцию-доказательство (код на YaoXiang)
                  │
                  ▼
         Средство проверки типов верифицирует ──→ Proved ──→ Компиляция проходит
                  │
                  ▼
            Верификация не прошла → Ошибка компиляции: «Доказательство не установлено»
```

#### 4.1 Результат доказательства: трёхзначная алгебра

Вычисление времени компиляции возвращает три результата — это неизбежное следствие проблемы остановки и естественное разбиение из теории доказательств:

```
eval_compile_time : BoolExpr → Proved | Disproved(Model) | Unproven
```

- **Proved** → Остановка, терм доказательства сконструирован, проверка типов пройдена. Компиляция продолжается.
- **Disproved(M)** → Остановка, существует контрпример M. Ошибка компиляции + контрпример + позиция в исходном коде.
- **Unproven** → В пределах заданного ресурсного лимита терм доказательства не сконструирован. Ошибка компиляции + неразрешённое утверждение + отчёт о потреблении бюджета.

**Unproven ≠ False.** Утверждение компилятора «я не могу это доказать» не эквивалентно ложности утверждения — просто это превышает возможности текущего автоматического доказательства. Это честность, а не дефект.

Жёсткое ограничение бюджета — инженерное решение проблемы остановки. Без настраиваемых параметров — их наличие означало бы вопрос пользователю «как вы думаете, ваша программа остановится?», пользователь не знает, и компилятор тоже не знает.

#### 4.2 После Unproven: программист пишет доказательство

Когда компилятор возвращает Unproven, программист может написать **функцию-доказательство** — функцию на YaoXiang, тип возвращаемого значения которой равен доказываемому утверждению. Средство проверки типов верифицирует эту функцию — тем же способом, каким оно верифицирует, что `add(a, b): Int`.

```
Утверждение = Тип
Доказательство = Программа (значение этого типа)
Верификация = Проверка типов (единственный корень доверия)
```

SMT-решатель — не отдельная граница доверия — это **ускоряющий модуль средства проверки типов**. SMT помогает найти доказательство, но доказательство всегда верифицирует средство проверки типов. Когда SMT возвращает `unsat`, компилятор реконструирует его результат в терм доказательства, верифицируемый средством проверки типов. Если реконструкция не удалась (шаги вывода SMT выходят за пределы правил вывода ядра компилятора), происходит откат к Unproven — программист может вручную написать функцию-доказательство.

```yaoxiang
# Утверждение: уточняющее свойство, которое компилятор не может доказать автоматически
FirstIsMin: (T: Ord, arr: Sorted(T)) -> Type = {
    forall i in 0..arr.len: arr[0] <= arr[i]
}

# Доказательство: программист пишет функцию, тип возврата которой — вышеуказанное утверждение
# Средство проверки типов верифицирует эту функцию — точно так же, как верифицирует add(a,b): Int
first_is_min: (T: Ord, arr: Sorted(T)) -> FirstIsMin(T, arr) = {
    # Компилятор здесь верифицирует: тип тела функции = FirstIsMin(T, arr)
    ...
}
```

Не нужен ИИ, не нужен экспорт в Coq, не нужны новые концепции. **Свойства, которые компилятор не может автоматически доказать → программист пишет доказательство кодом на YaoXiang → средство проверки типов верифицирует.** Весь процесс — плавный градиент — компилятор делает за вас простые доказательства, оставляя мозг для сложных.

#### 4.3 Стратифицированные зависимости внутри конвейера

Вышеуказанные вычислители разделяют один интерфейс, но имеют порядок вычисления. Равенство типов — предпосылка для всего последующего анализа; проверка владения/токенов зависит от информации о типах; верификация уточняющих предикатов зависит от результатов первых двух слоёв. Компилятор вычисляет по слоям; выражения с неудачей нижнего слоя не поступают в верхний — что позволяет избежать траты бюджета решения на программах с ошибками типов.

```
Порядок вычисления (один конвейер, стратифицированная диспетчеризация)
├── Слой 0: Равенство типов (T1 == T2)
│   └── Структурная унификация → при неудаче последующее бессмысленно, прямой возврат Disproved
├── Слой 1: Конфликты владения/токенов
│   └── Потокочувствительный анализ живости → при неудаче безопасность памяти не устанавливается, прямой возврат Disproved
└── Слой 2: Уточняющие предикаты / импликации Хоара
    └── Ядро компилятора → SMT-ускорение → получение Proved / Disproved / Unproven
```

Каждый слой по-прежнему возвращает `Proved/Disproved/Unproven`, разделяя один интерфейс и одну систему бюджетов.

### 5. Единство трёх слоёв функций

| Слой | Время выполнения | Вход | Выход | Пример |
|------|------------------|------|-------|--------|
| Функции уровня значений | Время выполнения | Значения | Значения | `add: (a: Int, b: Int) -> Int = a + b` |
| Конструкторы типов | Время компиляции | Типы/Значения | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| Предикаты времени компиляции | Время компиляции | Значения | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

Все используют одинаковый синтаксис `name: type = value`. Предикаты времени компиляции и конструкторы типов проходят через один конвейер доказательств времени компиляции — `{}` — это пространство доказательств.

### 6. Циклы: генерация условий верификации по Флойду-Хоару

Циклам не нужны отдельные аннотации `: Invariant(...)` или `: decreases(...)`. Аннотации типов компиляции времени на переменных определяют утверждения в стиле Флойда-Хоара — компилятор генерирует условия верификации из аннотаций типов, а конвейер доказательств проверяет, сохраняет ли каждое присваивание тип.

Основной механизм: каждой операции присваивания соответствует тройка Хоара `{P} x := e {Q}`, условие верификации — `P ⇒ Q[e/x]`. Компилятор генерирует условие верификации для тела цикла один раз — после того как конвейер доказательств верифицирует индуктивный шаг, все итерации покрываются автоматически.

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # Аннотация ссылается на i; при инициализации i=0 верификация: 0 == sum(arr[0..0]) → True
    mut i: UpTo(arr.len) = 0     # Верификация: 0 <= 0 <= arr.len → True
    while i < arr.len {
        # Компилятор генерирует VC для тела цикла один раз. Посылка: s удовлетворяет SumUpTo(arr, i), i удовлетворяет UpTo(arr.len).
        #
        # s += arr[i]:
        #   Обязательство верификации: s_new удовлетворяет SumUpTo(arr, i) (текущий i не изменён)
        #   Подставляя s_new = s_old + arr[i]:
        #     Требуется s_old + arr[i] == sum(arr[0..i+1])
        #     Из индуктивной гипотезы s_old == sum(arr[0..i]), прибавляя arr[i] к обеим сторонам:
        #     sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   Компилятор + SMT: линейная арифметика, миллисекунды → Proved
        #
        # i += 1:
        #   Изменение i → в графе зависимостей аннотация типа s ссылается на i → запуск реверификации
        #   Новая цель верификации: s удовлетворяет SumUpTo(arr, i_new)
        #   Т.е. s == sum(arr[0..i_new]), гарантируется предыдущим шагом → Proved
        s += arr[i]
        i += 1
    }
    return s  # В этот момент s: SumUpTo(arr, arr.len), т.е. s == sum(arr[0..arr.len])
}
```

Инварианты цикла — это аннотации типов на переменных — программист пишет тип, компилятор проверяет индуктивный шаг. Компилятору не нужно «обнаруживать» инвариант и не нужно «автоматически проводить индукцию» — он разбивает индуктивное доказательство на локальные условия верификации для каждой операции присваивания и передаёт их в конвейер доказательств.

#### 6.1 Отслеживание зависимостей: зависимые типы на изменяемых переменных

Предпосылка вышеуказанного механизма: компилятор знает, что аннотация типа `s` — `SumUpTo(arr, i)` — ссылается на `i` — когда `i` изменяется, ограничение типа `s` также изменяется. Это требует от компилятора поддерживать **граф зависимостей типов между переменными**.

**Структура данных**:

```
TypeDepGraph: Map<VarName, Set<VarName>>
# Ключ — зависимая переменная, значение — множество переменных, чьи аннотации типа ссылаются на эту переменную
# Пример: { i: {s}, j: {s, t}, ... }
```

**Построение**: средство проверки типов при обработке `mut v: Pred(... x ...) = init` анализирует свободные переменные в аргументах `Pred(...)`. Если аргумент ссылается на другую изменяемую переменную `x` в текущей области видимости, в графе зависимостей регистрируется `x → v`.

**Триггер**: когда зависимая переменная `x` получает значение, компилятор:
1. Находит в графе зависимостей все переменные `{v₁, v₂, ...}`, зависящие от `x`
2. Для каждого `v` генерирует условие верификации: удовлетворяет ли текущее значение `v` обновлённому типу `Pred(... x_new ...)`
3. Отправляет VC в конвейер доказательств

**Чувствительность к порядку присваивания**: отслеживание зависимостей естественным образом обеспечивает правильный порядок присваивания. На примере `SumUpTo(arr, i)`:

```yaoxiang
# Правильный порядок
s += arr[i]   # s_new удовлетворяет SumUpTo(arr, i+1)
i += 1        # Изменение i → реверификация s удовлетворяет SumUpTo(arr, i_new) → True

# Неправильный порядок — компилятор отвергает
i += 1        # Изменение i → реверификация s удовлетворяет SumUpTo(arr, i_new)
              # s ещё не обновлён, s_old == sum(arr[0..i_old]) ≠ sum(arr[0..i_new])
              # → Ошибка компиляции: переменная s не удовлетворяет типу SumUpTo(arr, i_new)
s += arr[i]   # Недостижимо
```

**Комбинированные зависимости**: переменная может зависеть от нескольких переменных. Аннотация типа `{ v: Int; v == x + y }` одновременно зависит от `x` и `y` — изменение любой из них запускает реверификацию.

**Отношение с конвейером доказательств**: отслеживание зависимостей — триггер генерации VC, а не самостоятельный механизм верификации. Он отвечает на вопрос «когда нужно генерировать VC» — конвейер доказательств отвечает «выполняется ли VC».

### 7. Проверка завершения

Полностью автоматическая во время компиляции. Циклы, которые компилятор может доказать, проходят; циклы, которые не может — сразу выдают ошибку компиляции — программист должен позволить компилятору автоматически анализировать завершение цикла. Без лазеек для полуавтоматических аннотаций.

#### 6.1 Принципы проектирования

Компилятор автоматически извлекает информацию, необходимую для доказательства завершения, из двух источников:

1. **Аннотации типов переменных**: граничные ограничения в уточнённых типах (например, `UpTo(n)` даёт верхнюю границу `n` и нижнюю `0`)
2. **Операции в теле цикла**: операции, применяемые к переменным на каждой итерации

Компилятор пытается применить четыре стратегии синтеза метрики по приоритету, прекращая при нахождении.

#### 6.2 Стратегия 1: автоматический синтез линейной ранговой функции

Когда переменная имеет аннотацию линейной границы, компилятор перечисляет кандидатов линейных метрик и верифицирует их через SMT.

```
Вход:
  Переменные v₁: UpTo(u₁), v₂: UpTo(u₂), ... (переменные с границами)
  Условие цикла cond
  Множество присваиваний в теле цикла

Алгоритм:
  1. Извлечь границы каждой переменной из аннотаций типов: [low_i, high_i]
  2. Перечислить кандидатов метрик: v_i, u_i - v_i, v_i - v_j и т.д. (линейные комбинации)
  3. Для каждого кандидата m:
     - SMT верифицирует m ≥ 0 (из границ типа)
     - Для каждого пути выполнения тела цикла SMT верифицирует m' < m (строгое убывание)
  4. Найти подходящую линейную комбинацию → завершение доказано
```

Покрытие: циклы, где произвольная переменная получает значение линейного выражения (`v = a·v + b`) с аннотацией ограниченного типа. Включает `i += const`, `i -= const`, а также сжатие интервалов в стиле двоичного поиска:

```yaoxiang
# Двоичный поиск: low = mid + 1 или high = mid
# Метрика high - low строго убывает на обоих путях
binary_search: (arr: Sorted(Int, arr), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 Стратегия 2: счётчик нарушений предиката — автоматическое извлечение метрики из целевого типа <span style="color:orange">【Экспериментальная стратегия】</span>

> ⚠️ **Текущее состояние: экспериментальная стратегия, решение о включении принимается на основе фактической осуществимости на этапе 3 реализации.**
> Эта стратегия эффективна для операций обмена соседних элементов (сортировка пузырьком, сортировка вставками), но не может автоматически доказать операции с несмежными элементами (быстрая сортировка partition, пирамидальная сортировка sift-down).
> Границы покрытия см. в таблице ниже. Если на этапе 3 верификация покажет неосуществимость, эта стратегия будет удалена или переведена в будущую работу.

Ключевое наблюдение: **спецификации, написанные пользователем, — это материал для рассуждений компилятора.** Компилятору не нужно встраивать «что такое отсортированность» — он читает определение `Sorted` и автоматически извлекает метрику из него.

```
Вход:
  Целевой тип: Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  Операции в теле цикла: обмен соседних элементов

Алгоритм:
  1. Разобрать определение предиката: forall i in range: cond(i, arr)
  2. Автоматически сгенерировать метрику: violation_count = |{ i | ¬cond(i, arr) }|
  3. Проанализировать влияние операции на метрику:
     - Обмен соседних arr[j], arr[j+1] = arr[j+1], arr[j]
     - Влияет только на пары с индексами j-1, j, j+1
     - Если arr[j] > arr[j+1] (нарушение предиката), после обмена пара удовлетворяет предикату
     - violation_count уменьшается как минимум на 1
  4. Верхняя граница: n·(n-1)/2 (максимальное число соседних инверсий), нижняя: 0
  → Завершение доказано
```

**Текущее покрытие**:

| Алгоритм | Шаблон операции | Стратегия 2 доказуема? | Причина |
|----------|-----------------|:---:|---------|
| Сортировка пузырьком | Обмен соседних | ✅ | violation_count строго убывает при каждом обмене |
| Сортировка вставками | Смещение соседних | ✅ | Каждое смещение устраняет одну нарушающую пару |
| Сортировка выбором | Обмен несмежных | ❌ | Один обмен может увеличить violation_count |
| Быстрая сортировка | partition | ❌ | Обмен несмежных, не гарантирует монотонного убывания |
| Пирамидальная сортировка | sift-down | ❌ | Древовидные операции, violation_count немонотонна |

**Дополняющие стратегии**: для быстрой сортировки сжатие интервала `low < high` может быть покрыто стратегией 1 (линейная ранговая функция) — рекурсия partition внешнего уровня, каждый интервал уменьшается вдвое. Стратегии 1 и 2 дополняют друг друга, завершение большинства практических алгоритмов может быть доказано одной из них. Но обобщение стратегии 2 (операции с несмежными, древовидные операции) остаётся открытой проблемой.

```yaoxiang
sort: (arr: Array(Int)) -> (result: Sorted(result)) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 Стратегия 3: шаблон ограниченного возрастания/убывания

`v += const` (положительная константа), переменная имеет аннотацию верхней границы → метрика `upper_bound - v` убывает на `const` каждую итерацию, нижняя граница 0. Это вырожденный случай стратегии 1, компилятор обрабатывает его первым.

#### 6.5 Стратегия 4: шаблон мультипликативной метрики

`v *= const` (const > 1), переменная имеет аннотации нижней и верхней границ. Компилятор имеет встроенный шаблон логарифмической метрики `ceil(log_const(upper/v))`, убывающей на 1 при каждом умножении на const.

```yaoxiang
mut i: Positive(i) = 1
while i < n {
    # Компилятор автоматически выводит: метрика ceil(log₂(n/i)), убывает на 1 при каждом умножении на 2
    i *= 2
}
```

#### 6.6 Разделение завершения и корректности

Доказательство завершения и доказательство корректности независимы:
- **Завершение**: четыре вышеуказанные стратегии автоматически доказывают, что цикл завершается за конечное число шагов
- **Корректность**: продвигается ли тело цикла к целевому типу, проверяется конвейером доказательств времени компиляции через условия верификации

Оба проходят → компиляция проходит. Завершение доказано, но корректность не прошла → ошибка компиляции + контрпример. Корректность доказана, но завершение не доказуемо → ошибка компиляции с указанием переменных или операций, которые не удалось проанализировать. Оба не прошли → ошибка компиляции с раздельным указанием причин обоих сбоев.

#### 6.7 Проверка завершения рекурсивных функций

Для рекурсивных функций, требующих вычисления во время компиляции, компилятор проверяет убывание аргументов:

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # Компилятор анализирует: n-1 < n → убывание → завершение
}

# Использование во время компиляции — компилятор гарантирует завершение factorial во время компиляции
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120, выполняется во время компиляции
```

| Сценарий | Поведение |
|----------|-----------|
| Компилятор может проанализировать убывание рекурсии (например, `n-1`) | Вычисление во время компиляции |
| Не убывает / не удаётся определить убывание | Ошибка компиляции |
| Вызов во время выполнения (не в позиции типа) | Проверка завершения не требуется |

#### 6.8 Жёсткая граница

`i = f(i)`, где `f` необратима, незамкнута, не сохраняет никакой монотонности — математически невозможно автоматически доказать завершение. Ошибка компиляции:

> Этот цикл не может быть автоматически доказан как завершающийся. Переменная цикла зависит от неанализируемой функции `f`. Пожалуйста, используйте шаблон итерации, анализируемый компилятором.

Это не провал компилятора. Любой код, который не может быть статически доказан как безопасный, не проходит компиляцию.

### 8. SMT-решатель: ускоряющий модуль средства проверки типов

SMT-решатель в традиционных языках — внешний инструмент (например, F\* вызывает Z3, Dafny вызывает Z3). В YaoXiang это **ускоряющий модуль средства проверки типов** — вызывается только тогда, когда ядро компилятора не может принять решение напрямую. SMT помогает найти доказательство, но доказательство верифицирует средство проверки типов.

**Модель доверия**: средство проверки типов — единственный корень доверия. SMT-решатель — ускоряющий модуль — он помогает найти доказательство, но SMT не является отдельной границей доверия. Компилятор доверяет результату `unsat` от Z3 (в соответствии с подходом F\*/Dafny — вероятность ошибки Z3 ниже, чем частота багов в самом компиляторе, это прагматичный инженерный выбор). Истинный контроль ненадёжности находится на уровне трансляции в SMT — если в трансляции есть баг, он будет обнаружен в других тестах компилятора.

**Интерфейс**: компилятор внутри транслирует в стандартный формат SMT-LIB 2.6, а не привязывается к API конкретного решателя. SMT-LIB — стандарт ISO, Z3, CVC5, MathSAT, Yices — все поддерживают его изначально.

**Бэкенд по умолчанию**: Z3 (лицензия MIT, наиболее обширная документация и проверенность сообществом). CVC5 как совместимая с SMT-LIB альтернатива — пользователь может переключаться флагом компилятора.

Без «универсального слоя абстракции решателя» — SMT-LIB и есть слой абстракции. В будущем, если CVC5 добьётся прорыва в определённой теории, переключение потребует лишь смены бинарника, без изменения кода компилятора.

```
Bool-выражение времени компиляции
        │
        ├── Ядро компилятора может решить напрямую (структурная эквивалентность,
        │   простая арифметика, тривиальные формулы после свёртки констант)
        │   → Прямой возврат Proved / Disproved
        │
        └── Ядро компилятора не может решить напрямую (кванторы, символьные переменные)
            → Прередукция зависимых типов (factorial(5) → 120)
            → Трансляция в формат SMT-LIB
            → Отправка в Z3/CVC5 (с ограничением бюджета)
            → Возврат: unsat → Proved  │  sat + модель → Disproved  │  unknown → Unproven
```

**Бюджет решения — жёсткое ограничение, как глубина стека**:

| Измерение бюджета | Значение по умолчанию | Пояснение |
|-------------------|----------------------|-----------|
| Число шагов решения | 10 000 | Z3 для линейной арифметики обычно укладывается в сотни шагов. 10 000 шагов покрывает 99% практических предикатов. |
| Время | 100 мс | Один предикат более 100 мс = пользователь пишет программу времени компиляции, а не аннотацию типа. 100 мс × 50 предикатов = 5 секунд верхней границы компиляции. |
| Глубина инстанцирования кванторов | 3 | Три уровня вложенных кванторов покрывают практические шаблоны. Более трёх уровней — скорее всего, пишется логическая задача. |

Превышение бюджета возвращает Unproven, ошибка компиляции + позиция предиката + объём потребления. Без деградации, без проверок времени выполнения, без silent pass.

**Почему это практически осуществимо**: 95% практических предикатов в инженерии — это линейная арифметика — `x > 0`, `arr.len > 0`, `0 <= idx < arr.len` — всё в пределах разрешимого фрагмента, SMT-решатель возвращает ответ за миллисекунды. Для редких сложных предикатов, превышающих бюджет, программист пишет функцию-доказательство.

Зависимые типы перед вызовом SMT проходят слой прередукции: `factorial(5)` вычисляется во время компиляции непосредственно в `120`, `append([1,2], [3])` вычисляется непосредственно в `[1,2,3]`. Эти детерминированные вычисления значений не потребляют бюджет SMT.

Программисту не нужно знать о существовании SMT. Ментальная модель: **компилятор может доказать — проходит, не может — ошибка — если компилятор не умеет, можно написать функцию и доказать ему**.

### 9. Композиция предикатов времени компиляции

Предикаты времени компиляции — это функции, возвращающие Type, композиция естественным образом реализуется через композицию функций:

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(arr)
}
```

### 10. Примеры кода

#### 9.1 Безопасность деления

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b

result = divide(10, 2)   # ✅ Компилятор верифицирует Positive(2) = { 2 > 0 } → True
# result = divide(10, 0)  # ❌ Компилятор верифицирует Positive(0) = { 0 > 0 } → False
```

#### 9.2 Безопасность доступа к массиву

```yaoxiang
InBounds: (idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (arr: Array(T), idx: InBounds(idx, arr)) -> T = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ Компилятор верифицирует InBounds(1, arr) = { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ Компилятор верифицирует InBounds(5, arr) = { 0 <= 5 && 5 < 3 } → False
```

#### 9.3 Корректность сортировки

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T))) -> (result: Sorted(T, result)) = {
    result = arr.clone()
    # ... реализация алгоритма сортировки ...
    return result
}
```

#### 9.4 Цикл: генерация VC компилятором

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```

### 11. Конвейер диспетчеризации dispatch: единая диспетчеризация времени компиляции и времени выполнения

`assert` и `Assert` — это две стороны одного уточняющего примитива типа. Конвейер диспетчеризации `dispatch` автоматически решает, идти ли по пути доказательства времени компиляции или проверки времени выполнения, на основе того, **достижимы ли свободные переменные предиката во время компиляции**:

| Критерий | Режим | Поведение |
|----------|-------|-----------|
| Все свободные переменные известны во время компиляции (параметры-дженерики, константы времени компиляции) | **CompileTime** | Переход в конвейер доказательств: Proved → стирание, Disproved → ошибка компиляции, Unknown → требование доказательства |
| Существуют свободные переменные из времени выполнения (параметры функций, внешний ввод, mut-переменные) | **Runtime** | Вставка проверки времени выполнения и инъекция уточняющего факта в потокочувствительное множество предположений Γ |

**Ключевое**: «Не удаётся определить» ≠ «Ложь». В режиме CompileTime Unknown требует доказательства (без тихой деградации), в режиме Runtime утверждение во время компиляции вообще не имеет истинностного значения — никакой prover не способен записать тавтологически истинное доказательство для «пользователь мог ввести отрицательное число», проверка времени выполнения — единственный sound-выбор. Это не слабость prover, а теоретическая неизбежность.

### 12. Потокочувствительное множество предположений Γ: распространение сильнейшего постусловия

Компилятор поддерживает потокочувствительное (flow-sensitive) множество предположений Γ, отслеживающее утверждения, известные как истинные в каждой точке потока управления.

**SP-распространение (сильнейшее постусловие)**:

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
y = x + 1           // Γ = {x > 0, y > 1}  ← SP-распространение
```

**Kill set для mut-переменных**: после повторного присваивания `mut`-переменной все предположения, затрагивающие эту переменную, удаляются из Γ:

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
mut x = x - 5       // Γ = {}  ← x > 0 убито
```

Это жёсткое требование soundness — значение переменной изменилось, старые предположения недействительны.

**Слияние ветвей**: при слиянии ветвей IF/ELSE или match Γ берёт пересечение предположений ветвей. Выходят из ветви только те утверждения, которые истинны на всех путях.

### 13. Уточнение модели стирания: стирание witness ≠ стирание check

Утверждение RFC-027 «уточнённые типы **полностью стираются** во время выполнения» относится к **proof witness (токену доказательства)** — термы доказательства, верифицированные во время компиляции, не порождают runtime-кода. Но **проверки времени выполнения**, вставленные dispatch в режиме Runtime, сохраняются — это Bool-проверки на уровне значений, а не witness на уровне типов.

Резюме: witness стирается, check сохраняется. Эти две вещи не противоречат, исходное утверждение RFC-027 остаётся в силе.

## Детальное проектирование

### Изменения синтаксиса

| Ранее (RFC-022) | Теперь (данный RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | Предикат времени компиляции как тип параметра `(b: Positive(b))` |
| `//! ensures: ExistsMax(result, arr)` | Возвращаемый тип использует параметр возвращаемого значения `-> (result: IsMax(T, arr, result))` |
| `/*! invariant: ... !*/` | Аннотация уточняющего типа времени компиляции на переменной — инвариант Floyd-Hoare |
| `//! decreases: n` | Компилятор полностью автоматически выводит функцию-меру |
| Спецификации — комментарии | Спецификации — система типов |

### Синтаксис

**Для предикатов времени компиляции нет новых ключевых слов.** `{}` — пространство доказательств, полностью совместимое с существующим синтаксисом определения типов. Предикат времени компиляции — это функция, возвращающая Type — `name: (params) -> Type = { утверждения }`. Использование — это вызов функции — `Positive(b)`, `IsMax(T, arr, result)`.

```bnf
# Предикат времени компиляции = функция, возвращающая Type, внутри {} — утверждения, верифицируемые компилятором
# Используется существующий синтаксис функций/типов, новые правила BNF не требуются
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

**Новая синтаксическая концепция: параметр возвращаемого значения** — в `-> (name: Type)` `name` — это параметр возвращаемого значения.

Параметр возвращаемого значения — это **единственная новая синтаксическая концепция**, вводимая YaoXiang в существующий синтаксис функций. Его семантика:
- Значение `name` предоставляется оператором `return`
- `name` существует только в сигнатуре типа, на него ссылаются постусловия-предикаты (например, `-> (result: IsMax(T, arr, result))`)
- `name` не попадает в область видимости тела функции, не появляется у вызывающей стороны
- Параметр возвращаемого значения **опционален** — при отсутствии постусловия сигнатура полностью совпадает с обычной функцией (`-> Int`), без какой-либо дополнительной нагрузки

Причина его введения: постусловию необходимо ссылаться на «значение, которое функция собирается вернуть». Без параметра возвращаемого значения компилятор может разрешить предикату ссылаться на возвращаемое значение только через специальные правила (например, неявная переменная `$result` или `__retval__`). Параметр возвращаемого значения делает эту ссылку явной — это просто параметр, только значение предоставляется через `return`, а не вызывающей стороной.

**Функция-доказательство** — не новая концепция — это просто функция на YaoXiang, тип возвращаемого значения которой равен утверждаемому предложению. Когда компилятор возвращает Unproven, программист предоставляет функцию-доказательство, и средство проверки типов верифицирует её тем же способом, каким верифицирует любой возвращаемый тип функции. Никакого нового синтаксиса, ключевых слов или правил не требуется.

### Влияние на систему типов

- **Вселенная типов**: предикаты времени компиляции находятся на слое Type₂ — функции, принимающие значения и возвращающие Type, на том же уровне, что и конструкторы типов
- **Взаимодействие с дженериками**: предикаты времени компиляции могут иметь параметры-дженерики, например `NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **Взаимодействие с владением**: выражения в предикатах времени компиляции подчиняются правилам владения, только чтение, без записи
- **Вывод типов**: параметры предикатов времени компиляции участвуют в выводе типов по Хиндли-Милнеру

### Представление во время выполнения

Предикаты времени компиляции во время выполнения **обрабатываются по результату диспетчеризации dispatch**:
- **Режим CompileTime** (все свободные переменные известны во время компиляции): токен свидетельства (witness) полностью стирается после прохождения доказательства. `Positive: (x: Int) -> Type = { x > 0 }` — представление параметра `b: Positive(5)` во время выполнения — это просто `Int`. Уточняющее условие `{ 5 > 0 }` пройдено, стирается.
- **Режим Runtime** (существуют свободные переменные времени выполнения): сохраняется проверка времени выполнения — Bool-проверка на уровне значений с инъекцией в потокочувствительное множество предположений Γ. См. §11 конвейер диспетчеризации dispatch и §13 уточнение модели стирания.

Помещение предиката времени компиляции в позицию типа (например, `f(x: Positive(x))`) не порождает обёрточного типа и не выделяет дополнительную память. Но когда `x` поступает из ввода времени выполнения, **выполняется** вставка Bool-проверки времени выполнения.

**Ограничения взаимодействия с `ref`**: предикаты времени компиляции могут ссылаться только на неизменяемые заимствования или значения с переданным владением. Предикат времени компиляции, ссылающийся на изменяемое заимствование, не может гарантировать компилятору, что результат верификации остаётся действительным во время выполнения — такое использование сразу выдаёт ошибку компиляции.

### Изменения в компиляторе

1. **Парсер**: предикаты времени компиляции используют стандартный синтаксис функций, дополнительные правила разбора не требуются
2. **Конвейер доказательств времени компиляции**: единый интерфейс возврата Proved/Disproved/Unproven, автоматический выбор стратегии
3. **Модуль SMT-ускорения**: слой трансляции в SMT-LIB 2.6, бэкенд по умолчанию Z3, CVC5 как альтернатива
4. **Ядро средства проверки типов**: реализация правил вывода — структурная эквивалентность, βδι-редукция, введение/устранение кванторов всеобщности. Это единственный корень доверия, SMT и доказательства программиста верифицируются через него
5. **Генерация условий верификации**: WP/SP-исчисление + обязательства доказательства инвариантов цикла
6. **Отчёты об ошибках**: форматирование контрпримеров + отчёт о неразрешённых утверждениях + привязка к позициям в исходном коде

### Обратная совместимость

- ✅ Код, не использующий предикаты времени компиляции, остаётся полностью неизменным
- ✅ Предикаты времени компиляции в режиме CompileTime имеют нулевые накладные расходы во время выполнения, в режиме Runtime сохраняются только необходимые Bool-проверки
- ⚠️ Синтаксис `//!` из RFC-022 больше не поддерживается — но 022 никогда не был реализован, бремя миграции отсутствует

## Компромиссы

### Преимущества

- **Полная реализация изоморфизма Карри-Ховарда**: тип — это утверждение, программа — это доказательство, `name: Утверждение = Доказательство`
- **Единство**: предикаты времени компиляции и обычные функции используют полностью идентичный синтаксис, без концептуального разделения
- **Прозрачность SMT**: программисту не нужно знать о существовании SMT, ментальная модель совпадает с проверкой типов
- **Постепенное внедрение**: можно начать с одного предиката времени компиляции, постепенно расширяя покрытие
- **Минимальные накладные расходы во время выполнения**: режим CompileTime — нулевые накладные расходы, режим Runtime — только необходимые Bool-проверки

### Недостатки

- **Время компиляции**: решение SMT увеличивает время компиляции, но жёсткое ограничение бюджета гарантирует контролируемую верхнюю границу
- **Границы автоматического доказательства**: сложные предикаты, выходящие за пределы линейной арифметики первого порядка, могут потребовать от программиста написания функции-доказательства. Это не дефект языка — это неизбежное следствие проблемы остановки. Компилятор честно сообщает Unproven вместо ложного True/False
- **Кривая обучения**: написание эффективных предикатов времени компиляции и функций-доказательств требует понимания базовой интуиции изоморфизма Карри-Ховарда
- **Сложность реализации**: единство конвейера доказательств времени компиляции требует тщательного проектирования

### Снижение рисков

- Жёсткое ограничение бюджета SMT (шаги 10 000 / время 100 мс / глубина инстанцирования 3), превышение возвращает Unproven
- Прередукция зависимых типов: детерминированные вычисления значений выполняются первыми, SMT берёт на себя только недетерминированную часть
- Unproven — не тупик: программист может написать функцию-доказательство, средство проверки типов верифицирует — как и любой возвращаемый тип функции
- Инкрементальная верификация: верифицируются только изменённые модули
- Чёткие сообщения об ошибках + отображение контрпримеров + отчёт о потреблении бюджета + неразрешённые утверждения + предложения (если компилятор может их дать)

## Альтернативы

| Подход | Почему не выбран |
|--------|------------------|
| RFC-022: спецификации в виде комментариев `//!` | Разделение спецификации и типа, нарушение изоморфизма Карри-Ховарда |
| Отдельные файлы спецификаций (например, CVL) | Разделение спецификации и кода, увеличение стоимости поддержки |
| Только утверждения времени выполнения | Не позволяет статически гарантировать корректность |
| Внешний ассистент доказательства (например, Coq) | Отрыв от компилятора, требует отдельного языка доказательств и границы доверия. Выбор YaoXiang: доказательство — это код на YaoXiang, средство проверки типов — единственный корень доверия |
| **Данный подход: предикаты времени компиляции как полноправные сущности первого класса** | ✅ |

## Стратегия реализации

### Разделение на фазы

| Фаза | Содержание |
|------|------------|
| **Фаза 1** | Ядро компилятора: структурная эквивалентность + βδι-редукция + введение/устранение кванторов всеобщности. Поддержка простых арифметических предикатов (`x > 0`, `arr.len > 0`) |
| **Фаза 2** | Слой трансляции в SMT-LIB + интеграция с Z3/CVC5. Конвейер возвращает Proved/Disproved/Unproven. При Unproven поддержка функции-доказательства от программиста |
| **Фаза 3** | Генерация VC для инвариантов цикла + проверка завершения (линейная ранговая функция + счётчик нарушений предиката + шаблон ограниченной метрики + контроль комбинаторного взрыва) |
| **Фаза 4** | Инкрементальная верификация + кэширование + поддержка IDE |

## Открытые вопросы

- [x] **Выбор SMT-решателя**: по умолчанию Z3 (лицензия MIT, наиболее обширная проверка). CVC5 как совместимая с SMT-LIB альтернатива, переключение флагом компилятора. Внутри компилятора цель трансляции — стандартный формат SMT-LIB 2.6 — SMT-LIB и есть слой абстракции, без специального универсального интерфейса решателя.
- [x] **Конкретные значения бюджета решения**: шаги 10 000 / время 100 мс / глубина инстанцирования кванторов 3. Жёстко зафиксированы в компиляторе, без настраиваемых параметров. Если в реальном использовании появятся подлинные случаи, доказывающие недостаточность (не «пользователь ошибся»), значения будут скорректированы.
- [x] **Объём поддержки кванторов**: язык не ограничивает порядок кванторов. Предикаты времени компиляции принимают параметры Type — Type включает функциональные типы — поэтому кванторы высшего порядка — естественное следствие системы типов, специальный синтаксис не нужен. SMT-решатель может автоматически разрешать кванторы первого порядка (forall/exists, с поддержкой чередующегося вложения, ограничено глубиной бюджета 3). Кванторы высшего порядка: SMT возвращает Unproven, компилятор подсказывает «этот предикат выходит за пределы автоматического доказательства, предоставьте функцию-доказательство». Программист пишет функцию на YaoXiang, тип возврата которой равен этому утверждению — средство проверки типов верифицирует эту функцию. Никакого внешнего экспорта, никакого ИИ, никакого интерактивного режима доказательства. Всё — код на YaoXiang, всё верифицируется средством проверки типов.
- [x] **Форматирование контрпримеров**: имена переменных исходного кода используются напрямую как имена SMT-переменных (с префиксом модуля для предотвращения конфликтов). При возврате модели Z3 — обратный поиск по именам переменных. Формат вывода: имя_переменной = конкретное_значение + позиция в исходном коде + позиция определения предиката. Без сложного слоя отображения.
- [x] ~~**Взаимодействие предикатов времени компиляции с умными указателями `ref`?**~~ → Решено: предикаты времени компиляции разрешают только неизменяемые заимствования или значения с переданным владением. Значения по изменяемому заимствованию не могут появляться в предикатах времени компиляции.
- [x] **Расширение метрики счётчика нарушений предиката `forall` на несмежные операции?** → Не расширяется. Текущее покрытие (обмен соседних, смещение соседних) дополняется стратегией 1 (линейная ранговая функция) — сжатие внешнего интервала быстрой сортировки покрывается стратегией 1, пирамидальная сортировка покрывается стратегией 1 (шаблон индексации массива). Циклы, завершение которых не может быть доказано ни одной стратегией, приводят к прямой ошибке компилятора — это философия жёсткой безопасности, а не дефект. Если в будущем появятся реальные сценарии (не академические конструкции), в которых ни одна из четырёх стратегий не сработает, вопрос будет пересмотрен.
- [x] **Комбинаторный взрыв перечисления линейных ранговых функций**: верхний предел перечисления кандидатов — 3 переменные с границами. При ≤3 перечисляются все линейные комбинации с индивидуальной SMT-верификацией. При >3 пробуются только однопеременные метрики (`v_i`, `u_i - v_i`); при неудаче — прямая ошибка компиляции с подсказкой «цикл имеет >3 переменных с границами, компилятор не может автоматически синтезировать многопеременную метрику». Это не инженерный компромисс — это принуждение программиста к написанию более простых циклов.
