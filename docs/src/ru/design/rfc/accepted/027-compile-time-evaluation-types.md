# RFC-027: Предикаты времени компиляции и унифицированная статическая верификация

## Резюме

Данный документ предлагает ввести в YaoXiang **предикаты времени компиляции** как полноправных граждан языка и объединить всю статическую верификацию времени компиляции в единый **конвейер доказательства**. Предикаты времени компиляции — это не внешние спецификационные комментарии — это функции. Функция, возвращающая Type, может использоваться в позиции типа; компилятор вызывает её во время компиляции и проверяет возвращаемое значение. Тип есть утверждение, вычисление во время компиляции есть доказательство.

**Центральный тезис**: единственная задача проверки типов во время компиляции — построить и верифицировать термы доказательства. Равенство типов, конфликты токенов, редукция зависимых типов, вычисление предикатов времени компиляции, импликации логики Хоара — всё это разные проверки в одном конвейере доказательства времени компиляции. SMT-решатель — это ускоряющий модуль проверщика типов, а не отдельный доверенный компонент. Когда компилятор возвращает Unproven, программист пишет функцию YaoXiang в качестве доказательства — проверщик типов верифицирует её точно так же, как он верифицирует возвращаемый тип любой функции. Всё — код YaoXiang, всё верифицируется проверщиком типов.

## Мотивация

### Почему RFC-022 устарел?

RFC-022 проектировал спецификации в форме комментариев `//!`:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← это комментарий, отдельный от типов
    //! ensures: ExistsMax(result, arr[0..n])   ← это комментарий, отдельный от типов
}
```

Это совершает фундаментальную ошибку изоморфизма Карри-Ховарда: **разделяет спецификации и типы на два слоя**. Комментарии — не типы. Комментарии не участвуют в проверке типов. Это ментальная модель «внешнего инструмента».

В белой книге сказано прямо:

> «Никаких комментариев `//!`. Никакого отдельного языка спецификаций. Всё — внутри системы типов.»

### Текущие проблемы

- Комментарии `//!` из RFC-022 — это внешний синтаксис, отдельный от системы типов
- Спецификационные типы и обычные типы — две разные системы, что порождает концептуальную избыточность
- Разделение на Debug Build (с верификацией) / Release Build (без) разрушает единство
- SMT-решатель в традиционном понимании — внешний инструмент; YaoXiang встраивает его как ускоряющий модуль проверщика типов
- Проверка типов, верификация заимствований, проверка предикатов времени компиляции, раскрытие макросов — каждое идёт своим путём

### Корректная ментальная модель

Проверку типов можно абстрагировать как функцию:

```
verify : Program → Proved | Disproved(Model) | Unproven
```

Все проверки времени компиляции — сопоставление простых типов, обнаружение конфликтов заимствований, верификация предикатов времени компиляции — являются подзадачами этой функции. Они разделяют один конвейер доказательства; различие только в сложности терма доказательства и стратегии построения.

Когда компилятор возвращает Unproven, программист предоставляет функцию-доказательство — возвращаемый тип этой функции равен доказываемому утверждению. Проверщик типов верифицирует её. Это та же операция, что и обычная проверка типов.

## Предложение

### 1. `{}` — пространство доказательства: тип есть утверждение, верификация есть проверка типов

`{}` в YaoXiang — это пространство доказательства времени компиляции. Всё внутри — утверждение; компилятор гарантирует истинность каждого — либо автоматическим доказательством, либо предоставленной программистом функцией-доказательством.

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  компилятор гарантирует, что x — Float, y — Float

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  компилятор гарантирует, что data — Array(T)
```

**Обобщения — частный случай предикатов времени компиляции.**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          параметр в сигнатуре      внутри {} только утверждения
#          компилятор проверяет x > 0 при вызове времени компиляции

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      параметр в сигнатуре       компилятор проверяет type_of(T) == Type, type_of(data) == Array(T)
```

Один и тот же паттерн: `name: (params) -> Type = { утверждения }`. Компилятор не различает «утверждения о типе» и «утверждения о значении» — оба являются целями вычисления в конвейере доказательства.

**Инварианты цикла не нужно писать отдельно. Аннотация типа на переменной и есть инвариант Флойда-Хоара.**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # аннотация ссылается на i — сообщает компилятору, что тип s зависит от i
    mut i: UpTo(arr.len) = 0     # при инициализации i=0, проверка: 0 == sum(arr[0..0]) → True
    while i < arr.len {
        s += arr[i]  # компилятор проверяет: s_new == sum(arr[0..i+1])
        i += 1       # i изменяется → запускает реверификацию зависимости s: s удовлетворяет SumUpTo(arr, i_new)
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

Компилятор генерирует одно условие верификации для тела цикла — индуктивная гипотеза (аннотация типа) → операция присваивания → удовлетворяет ли новое значение аннотации типа. Когда конвейер доказательства верифицирует индуктивный шаг, все итерации автоматически покрываются. Никаких `: decreases`, никаких `: Invariant`, никакого индуктивного доказательства — компилятор разбивает индукцию на локальные УВ для каждого присваивания.

### 2. Предусловия/постусловия: предикаты времени компиляции в позиции типа параметра и возврата

Отказываемся от `//! requires`/`//! ensures` из RFC-022. Предикаты времени компиляции становятся аннотациями типа параметра или возврата.

**На стороне параметра — это вызов функции.** Предикат времени компиляции — это функция, возвращающая Type; использование на стороне параметра — это её вызов, в точности как `factorial(5)`. На стороне возврата вводится новое понятие: формальный параметр возвращаемого значения.

```yaoxiang
# Предусловие: явный вызов предиката времени компиляции в типе параметра
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b
#                       ^^^^^^^^^^  b — имя текущего формального параметра, передаётся в Positive
#                       компилятор в точке вызова извлекает значение аргумента, подставляет в b, проверяет Positive(аргумент)
#                       пример: divide(10, 2) → проверка Positive(2) = { 2 > 0 } → True
#                       пример: divide(10, 0) → проверка Positive(0) = { 0 > 0 } → False → ошибка компиляции

# Постусловие: формальный параметр возвращаемого значения + предикат времени компиляции
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

NonEmpty: (arr: Array(T)) -> Type = { arr.len > 0 }

max: (T: Ord) -> ((arr: NonEmpty(arr))) -> (result: IsMax(T, arr, result)) = {
#                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                            result — формальный параметр возврата, значение задаётся return
#                                            компилятор в точке return подставляет возвращаемое значение, проверяет постусловие
    candidate = arr[0]
    for i in 1..arr.len {
        if arr[i] > candidate { candidate = arr[i] }
    }
    return candidate
}
```

**Ключевые правила**:

- **На стороне параметра**: `b: Positive(b)` — `b` — имя текущего формального параметра, передаётся в `Positive`. Синтаксис вызова функции, никакой неявности.
- **На стороне возврата**: `-> (result: IsMax(T, arr, result))` — `result` — формальный параметр возвращаемого значения; его значение задаётся оператором `return`. `result` существует только в сигнатуре типа, на него ссылается только предикат, он не входит в область видимости тела функции и не виден вызывающему.
- **Формальный параметр возврата необязателен**: при отсутствии постусловия сигнатура идентична обычной функции (`-> Int`).
- **Единство**: параметры и формальные параметры возврата — одно и то же понятие — `имя_параметра: вызов_предиката(имя_параметра)`; разница лишь в том, задаёт ли значение вызывающий или оператор `return`.

### 3. Распространение путевых условий: верификация значений времени выполнения во время компиляции

Когда предикат времени компиляции используется в позиции связывания, аргументы передаются программистом явно. Когда значение времени выполнения попадает в аргумент уточнённого типа, компилятор выполняет верификацию посредством сбора путевых условий и SMT-импликации — без необходимости явно передавать доказательство.

#### 3.1 Явный вызов функции

Когда предикат времени компиляции используется в позиции связывания, аргументы передаются программистом явно — это вызов функции, никакой неявности.

`Positive: (x: Int) -> Type = { x > 0 }` — это конструктор предиката времени компиляции. Когда он появляется в позиции связывания (объявление параметра, объявление переменной, тип возврата), программист явно передаёт имя уже связанной переменной:

```yaoxiang
b: Positive(b)
// b уже объявлен как текущий формальный параметр, Positive(b) — вызов функции
// после нормализации: b: { b > 0 }
```

Никакого неявного заполнения аргументов компилятором — `b: Positive(b)`, как и `f(5)`, это вызов функции. `b` связывается как имя параметра, и аннотация типа `Positive(b)` ссылается на `b` сам по себе — это стандартный паттерн зависимых типов, а не правило неявного раскрытия.

**Единство с `self` из RFC-010**: RFC-010 устанавливает, что `self` — не ключевое слово, а лишь конвенциональное имя параметра («написать `p`, `this`, `x` — совершенно то же самое»). `b: Positive(b)` использует тот же механизм — имя параметра может быть использовано в аннотации типа. `self` появляется в позиции `self: Point`, `b` появляется в позиции `b: Positive(b)`; в обоих случаях аннотация типа ссылается на сам параметр. Различие только в сложности аннотации, механизм полностью идентичен — после связывания имени тип может зависеть от этого имени.

Тип возврата использует тот же явный вызов функции:

```yaoxiang
Sorted: (arr: Array(T)) -> Type = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }

sort: (arr: Array(T)) -> (result: Sorted(result)) = { ... }
//                        ^^^^^^^^^^^^^^^^^^^^^^^
//                        result — формальный параметр возврата, Sorted(result) — вызов функции
//                        компилятор в точке return подставляет возвращаемое значение в result, проверяет Sorted(возврат)
```

То же применимо к локальным объявлениям переменных:

```yaoxiang
let x: Positive(x) = 5
// x связывается с 5, Positive(5) → { 5 > 0 } → True → пройдено

// let y: Positive(y) = 0
// y связывается с 0, Positive(0) → { 0 > 0 } → False → ошибка компиляции
```

#### 3.2 Сбор путевых условий

Когда значение времени выполнения появляется в условной ветви, компилятор автоматически собирает путевые условия, формируя **множество гипотез** текущей области видимости. Эти гипотезы участвуют в верификации как фоновые знания для вычисления Bool времени компиляции.

```yaoxiang
if y > 0 {
    // компилятор автоматически получает гипотезу в этой ветви: { y > 0 }
    let result = divide(x, y)
    // условие верификации: (y > 0) ⇒ (y > 0)
    // конвейер доказательства признаёт импликацию истинной → Proved
} else {
    // гипотеза в этой ветви: { !(y > 0) }
    // при вызове divide(x, y) условие верификации: !(y > 0) ⇒ y > 0
    // конвейер доказательства признаёт недоказуемым → Disproved
}
```

Это не выученный компилятором специальный паттерн — это естественное поведение конвейера доказательства времени компиляции. Каждая проверка типа в точке вызова отправляет в конвейер:

```
{фоновые_гипотезы} ⇒ {цель_верификации}
```

Конвейер доказательства оценивает импликацию. Proved → пройдено, Disproved → ошибка компиляции + контрпример, Unproven → ошибка компиляции + неразрешённое утверждение. Фоновые гипотезы берутся из путевых условий текущей точки программы.

#### 3.3 Стек гипотез

При анализе потока управления компилятор поддерживает множество гипотез для каждого базового блока:

- **if-guard**: `if y > 0` → истинная ветвь кладёт `y > 0`, ложная ветвь (если есть else) кладёт `!(y > 0)`
- **паттерн match**: `if let Some(v) = opt` → в ветви кладётся `opt == Some(v)`
- **логическая конъюнкция**: `if x > 0 && y < 10` → в ветвь кладутся `x > 0` и `y < 10`
- **предусловие функции**: при вызове `divide(a, b)` свидетельство того, что `b` удовлетворяет `Positive`, берётся либо из текущих гипотез, либо из уточнённой аннотации типа самого аргумента (если `b` помечен как `Positive`, его тип несёт `b > 0`)
- **присваивание**: при `let z = y` уже имеющееся на `y` уточнённое условие переносится на `z`

Все гипотезы поступают в конвейер доказательства времени компиляции. При переходе на ускоренный путь SMT они переводятся в фоновые утверждения SMT-LIB.

#### 3.4 Отсутствие статического свидетельства → ошибка компиляции

Если программист пишет напрямую:

```yaoxiang
divide_user_input: (x: Int, y: Int) -> Int = divide(x, y)
```

В текущей точке программы нет гипотезы `y > 0`, и сам аргумент `y` не имеет аннотации `Positive`. Условие верификации:

```
{} ⇒ { y > 0 }
```

Конвейер возвращает `Disproved` (не доказано) → ошибка компиляции:

> Не удаётся доказать, что параметр `b` в вызове `divide` удовлетворяет `Positive`.
> `y` получен из входных данных функции, его границы не доказаны.
> Рассмотрите охрану вызова ветвью `if`: `if y > 0 { divide(x, y) }`.

YaoXiang не допускает передачу значений времени выполнения непосредственно в аргументы уточнённых типов без предоставления статического свидетельства. Это не ограничение — это ядро философии жёсткой безопасности. Всё, что компилятор не может статически доказать, не проходит компиляцию.

#### 3.5 Связь с единым конвейером

Распространение путевых условий — не дополнительный механизм. Это прямое расширение конвейера доказательства времени компиляции на анализ потока управления:

| Этап | Обязанность |
|------|------------|
| Сбор путевых условий | Этап анализа потока управления компилятора, аннотирующий каждый базовый блок множеством гипотез |
| Генерация условий верификации | При обнаружении ограничения типа объединяет путевые условия и информацию о типе аргумента |
| Вычисление в конвейере доказательства | Ядро компилятора → ускорение SMT → получение Proved / Disproved / Unproven |
| Результат | `Proved` → пройдено; `Disproved` → ошибка компиляции + контрпример; `Unproven` → ошибка компиляции + неразрешённое утверждение (программист может предоставить функцию-доказательство) |

Никаких новых компонентов. Никаких специальных правил. Путевые условия — это фоновые знания конвейера доказательства — разделяют один конвейер и одну систему бюджетов с равенством типов и ограничениями заимствования.

### 4. Конвейер доказательства времени компиляции

Все проверки времени компиляции разделяют один конвейер. Ядро конвейера — **проверка типа** — проверка, равен ли тип терма доказательства доказываемому утверждению. Всё сводится к проверке типа.

```
Во время компиляции встречается выражение Bool, требующее вычисления
(т.е. нужно построить терм доказательства)
        │
        ├── Равенство типов (T1 == T2)
        │   → компилятор решает напрямую (структурная эквивалентность)
        │
        ├── Условие конфликта токенов (!conflicting(tokens))
        │   → потокочувствительный анализ живучести (отслеживание Dup/Linear)
        │
        ├── Редукция зависимых типов (n + m упрощается)
        │   → система переписывания термов времени компиляции (βδι-редукция)
        │
        ├── Предикат времени компиляции (x > 0, forall...)
        │   → ядро компилятора + ускоряющий SMT-модуль
        │
        └── Импликация логики Хоара (P ⇒ Q)
            → ядро компилятора + ускоряющий SMT-модуль
                    │
                    ▼
             ┌──────────┐
             │ Proved   │  → компиляция проходит
             │ Disproved│  → ошибка компиляции + контрпример
             │ Unproven │  → ошибка компиляции + неразрешённое утверждение
             └────┬─────┘
                  │
                  ▼
         Программист пишет функцию-доказательство (код YaoXiang)
                  │
                  ▼
         Проверщик типов верифицирует ──→ Proved ──→ компиляция проходит
                  │
                  ▼
         Верификация не прошла → ошибка компиляции: "доказательство несостоятельно"
```

#### 4.1 Результат доказательства: трёхзначная алгебра

Вычисление времени компиляции возвращает три результата — неизбежное следствие проблемы останова и естественное деление в теории доказательств:

```
eval_compile_time : BoolExpr → Proved | Disproved(Model) | Unproven
```

- **Proved** → останов, терм доказательства построен, проверка типа пройдена. Компиляция продолжается.
- **Disproved(M)** → останов, существует контрпример M. Ошибка компиляции + контрпример + позиция в исходнике.
- **Unproven** → в пределах данного лимита ресурсов терм доказательства не построен. Ошибка компиляции + неразрешённое утверждение + отчёт о потреблении бюджета.

**Unproven ≠ False.** Когда компилятор говорит «я не могу это доказать», это не означает, что утверждение ложно — просто это выходит за пределы текущих возможностей автоматического доказательства. Это честность, а не дефект.

Жёсткий лимит бюджета — инженерное решение проблемы останова. Без настраиваемых параметров — если бы они были, мы бы спрашивали пользователя «как вы думаете, ваша программа остановится?», а пользователь не знает, и компилятор тоже.

#### 4.2 После Unproven: программист пишет доказательство

Когда компилятор возвращает Unproven, программист может написать **функцию-доказательство** — это просто функция YaoXiang, чей возвращаемый тип равен доказываемому утверждению. Проверщик типов верифицирует эту функцию — тем же механизмом, что и `add(a, b): Int`.

```
Утверждение  = тип
Доказательство  = программа (значение этого типа)
Верификация  = проверка типа (единственный корень доверия)
```

SMT-решатель — не отдельный доверенный компонент — это **ускоряющий модуль проверщика типов**. SMT помогает найти доказательство, но верифицирует доказательство проверщик типов. Когда SMT возвращает `unsat`, компилятор перестраивает его результат в терм доказательства, верифицируемый проверщиком типов. Если перестроить не удалось (шаги рассуждения SMT выходят за правила вывода ядра компилятора), происходит откат к Unproven — программист может написать функцию-доказательство вручную.

```yaoxiang
# Утверждение: уточнённое свойство, которое компилятор не может доказать автоматически
FirstIsMin: (T: Ord, arr: Sorted(T)) -> Type = {
    forall i in 0..arr.len: arr[0] <= arr[i]
}

# Доказательство: программист пишет функцию, чей возвращаемый тип равен утверждению выше
# Проверщик типов верифицирует эту функцию — точно так же, как add(a,b): Int
first_is_min: (T: Ord, arr: Sorted(T)) -> FirstIsMin(T, arr) = {
    # компилятор верифицирует здесь: тип тела функции = FirstIsMin(T, arr)
    ...
}
```

Никакого ИИ, никакого экспорта в Coq, никаких новых концепций. **Свойства, которые компилятор не может доказать автоматически → программист пишет доказательство на YaoXiang → проверщик типов верифицирует.** Весь процесс — плавный градиент — компилятор делает за вас простые доказательства, оставляя мозг для сложных.

#### 4.3 Слои зависимостей внутри конвейера

Вышеописанные вычислители разделяют один интерфейс, но имеют порядок вычисления. Равенство типов — необходимое условие для всего последующего анализа; проверка владения/токенов зависит от информации о типах; верификация уточняющих предикатов зависит от результатов предыдущих двух слоёв. Компилятор вычисляет по слоям; выражения, провалившиеся на нижнем слое, не передаются выше — чтобы не тратить бюджет решателя на типонеправильных программах.

```
Порядок вычисления (один конвейер, послойное планирование)
├── Слой 0: равенство типов (T1 == T2)
│   └── структурная унификация → при провале дальнейшее бессмысленно, сразу Disproved
├── Слой 1: конфликты владения/токенов
│   └── потокочувствительный анализ живучести → при провале безопасность памяти не установлена, сразу Disproved
└── Слой 2: уточняющие предикаты / импликации Хоара
    └── ядро компилятора → ускорение SMT → Proved / Disproved / Unproven
```

Каждый слой по-прежнему возвращает `Proved/Disproved/Unproven`, разделяя один интерфейс и одну систему бюджетов.

### 5. Единство трёх слоёв функций

| Слой | Момент исполнения | Вход | Выход | Пример |
|------|-------------------|------|-------|--------|
| Функция-значение | Время выполнения | значение | значение | `add: (a: Int, b: Int) -> Int = a + b` |
| Конструктор типа | Время компиляции | тип/значение | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| Предикат времени компиляции | Время компиляции | значение | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

Все используют одинаковый синтаксис `name: type = value`. Предикаты времени компиляции и конструкторы типов идут по одному конвейеру доказательства времени компиляции — `{}` есть пространство доказательства.

### 6. Циклы: генерация условий верификации Флойда-Хоара

Циклам не нужны отдельные аннотации `: Invariant(...)` или `: decreases(...)`. Уточняющие аннотации типов на переменных задают утверждения в стиле Флойда-Хоара — компилятор генерирует условия верификации из аннотаций типов, а конвейер доказательства проверяет, сохраняет ли каждое присваивание тип.

Базовый механизм: каждое присваивание соответствует тройке Хоара `{P} x := e {Q}`, условие верификации — `P ⇒ Q[e/x]`. Компилятор генерирует одно условие верификации для тела цикла — когда конвейер доказательства верифицирует индуктивный шаг, все итерации автоматически покрываются.

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0   # аннотация ссылается на i; при инициализации i=0, проверка: 0 == sum(arr[0..0]) → True
    mut i: UpTo(arr.len) = 0     # проверка: 0 <= 0 <= arr.len → True
    while i < arr.len {
        # компилятор генерирует одно УВ для тела цикла. Посылка: s удовлетворяет SumUpTo(arr, i), i удовлетворяет UpTo(arr.len).
        #
        # s += arr[i]:
        #   обязанство доказать: s_new удовлетворяет SumUpTo(arr, i) (текущий i не изменился)
        #   подставляем s_new = s_old + arr[i]:
        #     требуется s_old + arr[i] == sum(arr[0..i+1])
        #     из индуктивной гипотезы s_old == sum(arr[0..i]), прибавим arr[i] к обеим сторонам:
        #     sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   компилятор + SMT: линейная арифметика, миллисекунды → Proved
        #
        # i += 1:
        #   i изменяется → в графе зависимостей аннотация типа s ссылается на i → запускает реверификацию
        #   новая цель верификации: s удовлетворяет SumUpTo(arr, i_new)
        #   т.е. s == sum(arr[0..i_new]), гарантировано предыдущим шагом → Proved
        s += arr[i]
        i += 1
    }
    return s  # в этот момент s: SumUpTo(arr, arr.len), т.е. s == sum(arr[0..arr.len])
}
```

Инвариант цикла — это аннотация типа на переменной — программист пишет тип, компилятор проверяет индуктивный шаг. Компилятору не нужно «находить» инвариант и не нужно «автоматически делать индукцию» — он разбивает индуктивное доказательство на локальные условия верификации для каждого присваивания, разделяй и властвуй через конвейер доказательства.

#### 6.1 Отслеживание зависимостей: зависимые типы на мутабельных переменных

Предпосылка вышеописанного механизма: компилятор знает, что аннотация типа `s`, а именно `SumUpTo(arr, i)`, ссылается на `i` — когда `i` изменяется, ограничение типа `s` также изменяется. Это требует от компилятора вести **граф зависимостей типов между переменными**.

**Структура данных**:

```
TypeDepGraph: Map<VarName, Set<VarName>>
# ключ — зависимая переменная, значение — множество переменных, чьи аннотации типа на неё ссылаются
# пример: { i: {s}, j: {s, t}, ... }
```

**Построение**: проверщик типов при обработке `mut v: Pred(... x ...) = init` разбирает ссылки на свободные переменные в аргументах `Pred(...)`. Если в аргументах встречается ссылка на другую мутабельную переменную `x` в текущей области видимости, в графе зависимостей записывается `x → v`.

**Срабатывание**: когда зависимая переменная `x` получает присваивание, компилятор:
1. Ищет в графе зависимостей все переменные, зависящие от `x`: `{v₁, v₂, ...}`
2. Для каждого `v` генерирует условие верификации: удовлетворяет ли текущее значение `v` обновлённому типу `Pred(... x_new ...)`
3. Передаёт УВ в конвейер доказательства

**Чувствительность к порядку присваиваний**: отслеживание зависимостей естественным образом форсирует правильный порядок присваиваний. На примере `SumUpTo(arr, i)`:

```yaoxiang
# Правильный порядок
s += arr[i]   # s_new удовлетворяет SumUpTo(arr, i+1)
i += 1        # i изменяется → реверификация: s удовлетворяет SumUpTo(arr, i_new) → True

# Неправильный порядок — компилятор отвергает
i += 1        # i изменяется → реверификация: s удовлетворяет SumUpTo(arr, i_new)
              # s ещё не обновлён, s_old == sum(arr[0..i_old]) ≠ sum(arr[0..i_new])
              # → ошибка компиляции: переменная s не удовлетворяет типу SumUpTo(arr, i_new)
s += arr[i]   # недостижимо
```

**Комбинированные зависимости**: переменная может зависеть от нескольких переменных. Аннотация `{ v: Int; v == x + y }` зависит одновременно от `x` и `y` — изменение любой из них запускает реверификацию.

**Связь с конвейером доказательства**: отслеживание зависимостей — триггер генерации УВ, а не отдельный механизм верификации. Оно отвечает на вопрос «когда нужно генерировать УВ» — конвейер доказательства отвечает на вопрос «выполнено ли УВ».

### 7. Проверка завершения

Полностью автоматическая во время компиляции. Компилятор доказывает завершение циклов, которые может; циклы, которые не может, — сразу ошибка компиляции — программист обязан сделать цикл анализируемым. Без полуавтоматических аннотаций.

#### 7.1 Принципы проектирования

Компилятор автоматически извлекает информацию для доказательства завершения из двух источников:

1. **Аннотации типов переменных**: граничные ограничения в уточнённых типах (например, `UpTo(n)` даёт верхнюю границу `n` и нижнюю `0`)
2. **Операции в теле цикла**: операции, применяемые к переменным на каждой итерации

Компилятор пробует четыре стратегии синтеза меры в порядке приоритета; найдя подходящую — останавливается.

#### 7.2 Стратегия 1: автоматический синтез линейной ранговой функции

Когда у переменной есть аннотация линейной границы, компилятор перебирает кандидаты линейных мер и проверяет их через SMT.

```
Вход:
  переменные v₁: UpTo(u₁), v₂: UpTo(u₂), ... (переменные с границами)
  условие цикла cond
  множество присваиваний в теле цикла

Алгоритм:
  1. Извлечь из аннотаций границы каждой переменной: [low_i, high_i]
  2. Перебрать кандидаты мер: v_i, u_i - v_i, v_i - v_j, и т.д. — линейные комбинации
  3. Для каждого кандидата m:
     - SMT проверяет m ≥ 0 (из типовых границ)
     - Для каждого пути исполнения тела SMT проверяет m' < m (строгое убывание)
  4. Найти подходящую линейную комбинацию → завершение доказано
```

Охват: циклы, в которых переменной присваивается линейное выражение (`v = a·v + b`) и есть аннотация ограниченного типа. Включает `i += const`, `i -= const`, а также сжатие интервала в бинарном поиске:

```yaoxiang
# Бинарный поиск: low = mid + 1 или high = mid
# мера high - low строго убывает на обоих путях
binary_search: (arr: Sorted(Int, arr), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 7.3 Стратегия 2: счётчик нарушений предиката — автоматическое извлечение меры из целевого типа <span style="color:orange">【Экспериментальная стратегия】</span>

> ⚠️ **Текущий статус: экспериментальная стратегия; на этапе 3 реализации решение о включении принимается по результатам проверки практической реализуемости.**
> Эта стратегия работает для операций обмена соседних элементов (сортировка пузырьком, сортировка вставками) и не может автоматически доказать завершение для не соседних операций (разбиение быстрой сортировки, просеивание вниз в пирамидальной сортировке).
> Границы охвата см. в таблице ниже. Если на этапе 3 верификация покажет нереализуемость, стратегия будет удалена или перенесена в будущие работы.

Ключевое наблюдение: **спецификация, написанная пользователем, — это материал для рассуждений компилятора.** Компилятору не нужно встраивать «что такое сортировка» — он читает определение `Sorted` и автоматически извлекает из него меру.

```
Вход:
  целевой тип: Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  операции тела цикла: обмен соседних элементов

Алгоритм:
  1. Разобрать определение предиката: forall i in range: cond(i, arr)
  2. Автоматически сгенерировать меру: violation_count = |{ i | ¬cond(i, arr) }|
  3. Проанализировать влияние операции на меру:
     - обмен соседних arr[j], arr[j+1] = arr[j+1], arr[j]
     - затрагивает только пары с индексами j-1, j, j+1
     - если arr[j] > arr[j+1] (нарушение предиката), после обмена пара удовлетворяет предикату
     - violation_count уменьшается как минимум на 1
  4. Верхняя граница: n·(n-1)/2 (максимум соседних инверсий), нижняя: 0
  → завершение доказано
```

**Текущий охват**:

| Алгоритм | Паттерн операции | Стратегия 2 доказуема? | Причина |
|----------|------------------|:---:|--------|
| Сортировка пузырьком | Обмен соседних | ✅ | violation_count строго убывает при каждом обмене |
| Сортировка вставками | Соседний сдвиг | ✅ | Каждый сдвиг устраняет одну нарушающую пару |
| Сортировка выбором | Не соседний обмен | ❌ | Один обмен может увеличить violation_count |
| Быстрая сортировка | Разбиение partition | ❌ | Не соседний обмен, не гарантирует монотонного убывания |
| Пирамидальная сортировка | sift-down | ❌ | Древовидная операция, violation_count не монотонна |

**Взаимодополняющие стратегии**: для быстрой сортировки сжатие интервала `low < high` покрывается стратегией 1 (линейная ранговая функция) — внешняя рекурсия partition, на каждом шаге интервал сжимается вдвое. Стратегии 1 и 2 дополняют друг друга; завершение большинства практических алгоритмов доказуемо одной из них. Однако обобщение стратегии 2 (не соседние операции, древовидные операции) остаётся открытой проблемой.

```yaoxiang
sort: (arr: Array(Int)) -> (result: Sorted(result)) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 7.4 Стратегия 3: паттерн ограниченного инкрементирования/декрементирования

`v += const` (с положительной константой), переменная имеет аннотацию верхней границы → мера `upper_bound - v` убывает на `const` за шаг, нижняя граница 0. Это вырожденный случай стратегии 1; компилятор обрабатывает его первым делом.

#### 7.5 Стратегия 4: шаблон меры с мультипликативным масштабированием

`v *= const` (const > 1), переменная имеет аннотации верхней и нижней границ. Компилятор использует встроенный шаблон логарифмической меры `ceil(log_const(upper/v))`; на каждом умножении на const мера убывает на 1.

```yaoxiang
mut i: Positive(i) = 1
while i < n {
    # компилятор автоматически выводит: мера ceil(log₂(n/i)), на каждом умножении на 2 мера убывает на 1
    i *= 2
}
```

#### 7.6 Разделение завершения и корректности

Доказательство завершения и доказательство корректности независимы:
- **Завершение**: четыре стратегии выше автоматически доказывают, что цикл завершается за конечное число шагов
- **Корректность**: приближается ли тело цикла к целевому типу — проверяется конвейером доказательства времени компиляции через условия верификации

Оба пройдены → компиляция успешна. Завершение доказано, но корректность не доказана → ошибка компиляции + контрпример. Корректность доказана, но завершение не доказано → ошибка компиляции с указанием переменной или операции, не поддающейся анализу. Оба не доказаны → ошибка компиляции с раздельным указанием обеих причин.

#### 7.7 Проверка завершения рекурсивных функций

Для рекурсивных функций, вычисляемых во время компиляции, компилятор проверяет убывание аргумента:

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # компилятор анализирует: n-1 < n → убывает → завершается
}

# Использование во время компиляции — компилятор гарантирует, что factorial завершится во время компиляции
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120, вычислено во время компиляции
```

| Сценарий | Поведение |
|----------|-----------|
| Компилятор может установить убывание рекурсии (например, `n-1`) | Вычисление во время компиляции |
| Не убывает / убывание неразрешимо | Ошибка компиляции |
| Вызов во время выполнения (не в позиции типа) | Проверка завершения не требуется |

#### 7.8 Жёсткая граница

`i = f(i)`, где `f` необратима, незамкнута, не сохраняет никакой монотонности — математически невозможно автоматически доказать завершение. Ошибка компиляции:

> Завершение этого цикла не может быть доказано автоматически. Переменная цикла зависит от неанализируемой функции `f`. Используйте паттерн итерации, доступный для анализа компилятором.

Это не провал компилятора. Всё, что не может быть статически доказано как безопасное, не проходит компиляцию.

### 8. SMT-решатель: ускоряющий модуль проверщика типов

SMT-решатель в традиционных языках — внешний инструмент (например, F\* вызывает Z3, Dafny вызывает Z3). В YaoXiang это **ускоряющий модуль проверщика типов** — вызывается только тогда, когда ядро компилятора само не может разрешить задачу. SMT помогает найти доказательство, но верифицирует доказательство проверщик типов.

**Модель доверия**: проверщик типов — единственный корень доверия. SMT-решатель — ускоряющий модуль — он помогает найти доказательство, но SMT — не отдельный доверенный компонент. Компилятор доверяет результату Z3 `unsat` (как и в подходе F\*/Dafny — вероятность ошибки в Z3 ниже, чем вероятность ошибки в самом компиляторе; это инженерно-прагматичный выбор). Реальная ненадёжность контролируется на уровне трансляции в SMT — при наличии бага трансляции он проявится в других тестах компилятора.

**Интерфейс**: внутренняя трансляция компилятора — в стандартный формат SMT-LIB 2.6, а не в API-интерфейс конкретного решателя. SMT-LIB — стандарт ISO; Z3, CVC5, MathSAT, Yices поддерживают его изначально.

**Бэкенд по умолчанию**: Z3 (лицензия MIT, самая широкая документация и валидация сообществом). CVC5 как SMT-LIB-совместимая альтернатива — пользователь может переключать флагом компилятора.

Никакого «универсального слоя абстракции решателя» — SMT-LIB и есть этот слой. Если в будущем CVC5 добьётся прорыва в определённой теории, переключение — это замена бинарника, без изменения кода компилятора.

```
Выражение Bool времени компиляции
        │
        ├── Ядро компилятора решает напрямую (структурная эквивалентность,
        │   простая арифметика, тривиальные формулы после свёртки констант)
        │   → сразу Proved / Disproved
        │
        └── Ядро компилятора не может решить (кванторы, символьные переменные)
            → предварительная редукция зависимых типов (factorial(5) → 120)
            → трансляция в формат SMT-LIB
            → отправка Z3/CVC5 (с лимитом бюджета)
            → возврат: unsat → Proved  │  sat + модель → Disproved  │  unknown → Unproven
```

**Бюджет решателя — жёсткое ограничение, как глубина стека**:

| Измерение бюджета | Значение по умолчанию | Пояснение |
|-------------------|----------------------|-----------|
| Число шагов решателя | 10 000 | Z3 обычно укладывается в сотни шагов для линейной арифметики. 10 000 шагов покрывают 99% практических предикатов. |
| Время | 100 мс | Один предикат свыше 100 мс = пользователь пишет программу времени компиляции, а не аннотацию типа. 100 мс × 50 предикатов = верхний предел 5 секунд на компиляцию. |
| Глубина инстанцирования кванторов | 3 | Три уровня вложенности кванторов покрывают практические паттерны. Больше — скорее всего пользователь пишет логическую задачу. |

Превышение бюджета → Unproven, ошибка компиляции + позиция предиката + объём потребления. Никаких послаблений, никаких проверок во время выполнения, никакого silent pass.

**Почему это практически реализуемо**: 95% практических предикатов — линейная арифметика — `x > 0`, `arr.len > 0`, `0 <= idx < arr.len` — всё в разрешимом фрагменте, SMT-решатель возвращает ответ за миллисекунды. Для редких предикатов, превышающих бюджет, программист пишет функцию-доказательство.

Перед обращением к SMT зависимые типы проходят предварительную редукцию: `factorial(5)` прямо вычисляется во время компиляции в `120`, `append([1,2], [3])` прямо вычисляется в `[1,2,3]`. Эти детерминированные вычисления не потребляют бюджет SMT.

Программисту не обязательно знать о существовании SMT. Ментальная модель: **компилятор может доказать — проходит, не может — сообщает об ошибке; если компилятор не умеет — можно написать функцию и показать ему доказательство.**

### 9. Композиция предикатов времени компиляции

Предикаты времени компиляции — это функции, возвращающие Type; композиция естественным образом выражается через композицию функций:

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(arr)
}
```

### 10. Примеры кода

#### 10.1 Безопасность деления

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive(b)) -> Int = a / b

result = divide(10, 2)   # ✅ компилятор проверяет Positive(2) = { 2 > 0 } → True
# result = divide(10, 0)  # ❌ компилятор проверяет Positive(0) = { 0 > 0 } → False
```

#### 10.2 Безопасность доступа к массиву

```yaoxiang
InBounds: (idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (arr: Array(T), idx: InBounds(idx, arr)) -> T = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ компилятор проверяет InBounds(1, arr) = { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ компилятор проверяет InBounds(5, arr) = { 0 <= 5 && 5 < 3 } → False
```

#### 10.3 Корректность сортировки

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T))) -> (result: Sorted(T, result)) = {
    result = arr.clone()
    # ... реализация алгоритма сортировки ...
    return result
}
```

#### 10.4 Цикл: генерация УВ компилятором

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, i) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```


### 11. Конвейер диспетчеризации `dispatch`: единая диспетчеризация для времени компиляции и времени выполнения

`assert` и `Assert` — две стороны одного примитива уточнённого типа. Конвейер диспетчеризации `dispatch` автоматически решает, использовать ли доказательство во время компиляции или проверку во время выполнения, в зависимости от того, **доступны ли свободные переменные предиката во время компиляции**:

| Критерий | Режим | Поведение |
|----------|-------|-----------|
| Все свободные переменные известны во время компиляции (параметры обобщений, константы времени компиляции) | **CompileTime** | Вход в конвейер доказательства: Proved → стирается, Disproved → ошибка компиляции, Unknown → требуется доказательство |
| Есть свободные переменные из времени выполнения (параметры функции, внешний ввод, переменные `mut`) | **Runtime** | Вставка проверки во время выполнения и инъекция уточняющего факта в потокочувствительное множество гипотез Γ |

**Ключевое**: «Не удаётся определить» ≠ «Опровергнуто». В режиме CompileTime Unknown требует доказательства (без тихой деградации); в режиме Runtime утверждение во время компиляции вообще не имеет истинности — никакой prover, сколь бы сильным он ни был, не может дать тавтологическое доказательство для «пользователь мог ввести отрицательное число»; проверка во время выполнения — единственный sound-выбор. Это не слабость prover'а, это теоретическая неизбежность.

### 12. Потокочувствительное множество гипотез Γ: распространение сильнейшего постусловия

Компилятор поддерживает потокочувствительное (flow-sensitive) множество гипотез Γ, отслеживающее утверждения, истинные в каждой точке потока управления.

**SP-распространение (сильнейшее постусловие)**:

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
y = x + 1           // Γ = {x > 0, y > 1}  ← SP-распространение
```

**Kill set для `mut`-переменных**: после переприсваивания `mut`-переменной все гипотезы, затрагивающие эту переменную, удаляются из Γ:

```yaoxiang
assert(x > 0)       // Γ = {x > 0}
mut x = x - 5       // Γ = {}  ← x > 0 убито
```

Это жёсткое требование корректности (soundness) — значение переменной изменилось, старые гипотезы недействительны.

**Слияние ветвей**: при слиянии ветвей IF/ELSE или match Γ берётся как пересечение гипотез ветвей. Из ветви выносится только то утверждение, которое истинно на всех путях.

### 13. Уточнение модели стирания: стирание witness ≠ стирание check

Утверждение RFC-027 о том, что «уточнённые типы полностью стираются во время выполнения», относится к **proof witness (свидетельству доказательства)** — терм доказательства, верифицированный во время компиляции, не порождает кода времени выполнения. Но проверка **во время выполнения**, вставленная диспетчером в режиме Runtime, сохраняется — это проверка Bool на уровне значений, а не witness на уровне типов.

Итог: witness стирается, check сохраняется. Эти две вещи не противоречат друг другу, исходное утверждение RFC-027 остаётся в силе.

## Детальное проектирование

### Синтаксические изменения

| Ранее (RFC-022) | Теперь (данный RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | Предикат времени компиляции как тип параметра `(b: Positive(b))` |
| `//! ensures: ExistsMax(result, arr)` | Тип возврата использует формальный параметр возврата `-> (result: IsMax(T, arr, result))` |
| `/*! invariant: ... !*/` | Уточняющая аннотация типа на переменной — инвариант Флойда-Хоара |
| `//! decreases: n` | Компилятор полностью автоматически выводит функцию меры |
| Спецификация — комментарий | Спецификация — система типов |

### Синтаксис

**У предикатов времени компиляции нет новых ключевых слов.** `{}` — пространство доказательства, полностью совпадает с существующим синтаксисом определения типов. Предикат времени компиляции — это функция, возвращающая Type — `name: (params) -> Type = { утверждения }`. Использование — это вызов функции — `Positive(b)`, `IsMax(T, arr, result)`.

```bnf
# Предикат времени компиляции = функция, возвращающая Type, внутри {} — верифицируемые компилятором утверждения
# Используется существующий синтаксис функций/типов, новые BNF-правила не нужны
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

**Новое синтаксическое понятие: формальный параметр возвращаемого значения** — в `-> (name: Type)` `name` является формальным параметром возврата.

Формальный параметр возврата — это **единственное** новое синтаксическое понятие, вводимое YaoXiang в дополнение к существующему синтаксису функций. Его семантика:
- Значение `name` задаётся оператором `return`
- `name` существует только в сигнатуре типа; на него ссылается предикат постусловия (например, `-> (result: IsMax(T, arr, result))`)
- `name` не входит в область видимости тела функции и не виден вызывающему
- Формальный параметр возврата **необязателен** — при отсутствии постусловия сигнатура идентична обычной функции (`-> Int`), никакой дополнительной нагрузки

Причина введения: постусловию нужно ссылаться на «значение, которое функция собирается вернуть». Без формального параметра возврата компилятор мог бы ссылаться на возвращаемое значение в предикате только через специальные правила (например, неявная переменная `$result` или `__retval__`). Формальный параметр возврата делает эту ссылку явной — это просто формальный параметр, чьё значение задаётся `return`, а не вызывающим.

**Функция-доказательство** — не новое понятие, это просто функция YaoXiang, чей возвращаемый тип равен утверждаемому предложению. Когда компилятор возвращает Unproven, программист предоставляет функцию-доказательство, и проверщик типов верифицирует её тем же способом, что и возвращаемый тип любой функции. Никакого нового синтаксиса, ключевых слов или правил.

### Влияние на систему типов

- **Вселенная типов**: предикаты времени компиляции находятся на уровне Type₂ — функции, принимающие значения и возвращающие Type, на том же уровне, что и конструкторы типов
- **Взаимодействие с обобщениями**: предикаты времени компиляции могут иметь параметры обобщений, например `NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **Взаимодействие с владением**: выражения в предикатах времени компиляции подчиняются правилам владения, только чтение, без записи
- **Вывод типов**: аргументы предикатов времени компиляции участвуют в HM-выводе типов

### Представление во время выполнения

Предикаты времени компиляции обрабатываются во время выполнения **в соответствии с результатом диспетчеризации `dispatch`**:
- **Режим CompileTime** (все свободные переменные известны во время компиляции): свидетельство (witness) полностью стирается после прохождения доказательства. `Positive: (x: Int) -> Type = { x > 0 }` — параметр `b: Positive(5)` во время выполнения представляется как `Int`. Уточняющее условие `{ 5 > 0 }` пройдено и стёрто.
- **Режим Runtime** (есть свободные переменные времени выполнения): сохраняется проверка во время выполнения — выполняется проверка Bool на уровне значений с инъекцией в потокочувствительное множество гипотез Γ. Подробности см. в §11 «Конвейер диспетчеризации dispatch» и §13 «Уточнение модели стирания».

Использование предиката времени компиляции в позиции типа (например, `f(x: Positive(x))`) не порождает обёрточного типа и не выделяет дополнительной памяти. Но если `x` поступает из ввода времени выполнения, **будет** вставлена проверка Bool во время выполнения.

**Ограничения взаимодействия с `ref`**: предикат времени компиляции может ссылаться только на значения с неизменяемым заимствованием или с переданным правом владения. Предикат времени компиляции, ссылающийся на значение с изменяемым заимствованием, не может гарантировать компилятору, что результат верификации остаётся действительным во время выполнения — такое использование сразу приводит к ошибке компиляции.

### Изменения в компиляторе

1. **Парсер**: предикаты времени компиляции используют стандартный синтаксис функций, дополнительные правила разбора не нужны
2. **Конвейер доказательства времени компиляции**: единый интерфейс возврата Proved/Disproved/Unproven, автоматический выбор стратегии
3. **Ускоряющий SMT-модуль**: слой трансляции в SMT-LIB 2.6, бэкенд по умолчанию Z3, CVC5 как альтернатива
4. **Ядро проверщика типов**: реализация правил вывода — структурная эквивалентность, βδι-редукция, введение/устранение кванторов всеобщности. Это единственный корень доверия; SMT и доказательства программиста верифицируются через него
5. **Генерация условий верификации**: WP/SP-исчисление + обязанности доказательства инвариантов цикла
6. **Отчёты об ошибках**: форматирование контрпримеров + отчёт о неразрешённых утверждениях + привязка к позициям в исходнике

### Обратная совместимость

- ✅ Код, не использующий предикаты времени компиляции, остаётся без изменений
- ✅ В режиме CompileTime предикаты времени компиляции не дают накладных расходов во время выполнения; в режиме Runtime сохраняются только необходимые проверки Bool
- ⚠️ Синтаксис `//!` из RFC-022 больше не поддерживается — но 022 никогда не был реализован, бремя миграции отсутствует

- **Полное воплощение изоморфизма Карри-Ховарда**: тип есть утверждение, программа есть доказательство, `name: Proposition = Proof`
- **Единство**: предикаты времени компиляции и обычные функции используют полностью идентичный синтаксис, без концептуального разделения
- **Прозрачность SMT**: программисту не обязательно знать о существовании SMT; ментальная модель идентична проверке типов
- **Постепенное внедрение**: можно начать с одного предиката времени компиляции, постепенно расширяя охват
- **Минимальные накладные расходы во время выполнения**: режим CompileTime — нулевые накладные расходы, режим Runtime — только необходимые проверки Bool

### Недостатки

- **Время компиляции**: решение SMT увеличивает время компиляции, но жёсткий лимит бюджета гарантирует контролируемый верхний предел
- **Границы автоматического доказательства**: сложные предикаты за пределами линейной арифметики первого порядка могут потребовать от программиста функции-доказательства. Это не дефект языка — это неизбежное следствие проблемы останова. Компилятор честно сообщает Unproven вместо ложного True/False
- **Кривая обучения**: написание эффективных предикатов времени компиляции и функций-доказательств требует понимания базовой интуиции изоморфизма Карри-Ховарда
- **Сложность реализации**: единство конвейера доказательства времени компиляции требует тщательного проектирования

### Снижение рисков

- Жёсткий лимит бюджета SMT (шаги 10 000 / время 100 мс / глубина инстанцирования 3), превышение → Unproven
- Предварительная редукция зависимых типов: сначала детерминированные вычисления значений, SMT берёт на себя только недетерминированную часть
- Unproven — не тупик: программист может написать функцию-доказательство, проверщик типов верифицирует — как и любую другую функцию с возвращаемым типом
- Инкрементальная верификация: верифицировать только изменённые модули
- Чёткие сообщения об ошибках + показ контрпримеров + отчёт о потреблении бюджета + неразрешённые утверждения + предложения (если компилятор может их дать)

## Альтернативы

| Подход | Почему не выбран |
| ------ | ---------------- |
| RFC-022: спецификации как комментарии `//!` | Спецификации и типы разделены, нарушает изоморфизм Карри-Ховарда |
| Отдельный файл спецификаций (например, CVL) | Спецификации отделены от кода, увеличивает стоимость сопровождения |
| Только утверждения во время выполнения | Не может статически гарантировать корректность |
| Внешний помощник доказательства (например, Coq) | Отрезан от компилятора, требует отдельного языка доказательств и границы доверия. Выбор YaoXiang: доказательство — это код YaoXiang, проверщик типов — единственный корень доверия |
| **Данный подход: предикаты времени компиляции как полноправные граждане** | ✅ |

## Стратегия реализации

### Разбиение на этапы

| Этап | Содержание |
| ---- | ---------- |
| **Этап 1** | Ядро компилятора: структурная эквивалентность + βδι-редукция + введение/устранение кванторов всеобщности. Поддержка простых арифметических предикатов (`x > 0`, `arr.len > 0`) |
| **Этап 2** | Слой трансляции в SMT-LIB + интеграция Z3/CVC5. Конвейер возвращает Proved/Disproved/Unproven. При Unproven поддерживается написание функций-доказательств программистом |
| **Этап 3** | Генерация УВ для инвариантов цикла + проверка завершения (линейная ранговая функция + счётчик нарушений предиката + ограниченный паттерн + контроль комбинаторного взрыва) |
| **Этап 4** | Инкрементальная верификация + кеширование + поддержка IDE |

### Зависимости

- RFC-009: Модель владения — выражения предикатов времени компиляции подчиняются правилам владения

## Открытые вопросы

- [x] **Выбор SMT-решателя**: по умолчанию Z3 (лицензия MIT, самая широкая валидация). CVC5 как SMT-LIB-совместимая альтернатива, переключение флагом компилятора. Внутренняя трансляция компилятора целится в стандартный формат SMT-LIB 2.6 — SMT-LIB и есть слой абстракции, без собственного универсального интерфейса решателя.
- [x] **Конкретные значения бюджета решателя**: шаги 10 000 / время 100 мс / глубина инстанцирования кванторов 3. Жёстко зашито в компиляторе, без настраиваемых параметров. Если реальные случаи использования (не «пользователь ошибся») докажут, что этого мало, значения будут скорректированы.
- [x] **Диапазон поддержки кванторов**: язык не ограничивает арность кванторов. Предикат времени компиляции принимает параметры типа Type — Type включает функциональные типы — поэтому кванторы высшего порядка — естественное следствие системы типов, специальный синтаксис не нужен. SMT-решатель может автоматически разрешать кванторы первого порядка (forall/exists, поддерживается чередующееся вложение, ограничено бюджетом глубины 3). Кванторы высшего порядка: SMT возвращает Unproven; компилятор предлагает «этот предикат выходит за пределы автоматического доказательства, предоставьте функцию-доказательство». Программист пишет функцию YaoXiang, чей возвращаемый тип равен утверждению — проверщик типов верифицирует её. Никакого внешнего экспорта, никакого ИИ, никакого интерактивного режима доказательства. Всё — код YaoXiang, всё верифицируется проверщиком типов.
- [x] **Форматирование контрпримеров**: имена переменных исходника используются непосредственно как имена SMT-переменных (с префиксом модуля во избежание конфликтов). При возврате модель Z3 обратно сопоставляется с именами. Формат вывода: имя переменной = конкретное значение + позиция в исходнике + позиция определения предиката. Без сложного слоя отображения.
- [x] ~~**Взаимодействие предикатов времени компиляции с умным указателем `ref`?**~~ → Решено: предикаты времени компиляции допускают только неизменяемые заимствования или значения с переданным правом владения. Значения с изменяемым заимствованием не могут появляться в предикатах времени компиляции.
- [x] **Расширение меры «счётчик нарушений предиката `forall`» на не соседние операции?** → Не расширяется. Текущий охват (соседний обмен, соседний сдвиг) покрывается стратегией 1 (линейная ранговая функция) — внешнее сжатие интервала быстрой сортировки покрывается стратегией 1, пирамидальная сортировка покрывается стратегией 1 (паттерн индексации массива). Циклы, для которых ни одна стратегия не может доказать завершение, — сразу ошибка компиляции — это философия жёсткой безопасности, а не дефект. Если в будущем появится реальный сценарий (не академическая конструкция) с алгоритмом, который не покрывается ни одной из четырёх стратегий, вопрос будет пересмотрен.
- [x] **Комбинаторный взрыв перебора линейных ранговых функций**: верхний предел числа переменных с границами в переборе — 3. При ≤3 перебираются все линейные комбинации и проверяются через SMT по очереди. При >3 пробуются только однопеременные меры (`v_i`, `u_i - v_i`); при неудаче — сразу ошибка компиляции — с сообщением «в цикле >3 переменных с границами, компилятор не может автоматически синтезировать многопеременную меру». Это не инженерный компромисс — это принуждение программиста к более простым циклам.
