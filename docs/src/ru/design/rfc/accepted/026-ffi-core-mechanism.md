## Резюме

Настоящий документ определяет основной механизм FFI (внешнего интерфейса функций) YaoXiang. Основная идея: **внешняя библиотека — это значение первого класса, линкуемое на этапе компиляции; принадлежность размещения в памяти межграничных данных фиксируется при определении типа; объекты кучи YaoXiang структурно изолированы от внешнего кода.**

1. **Внешняя библиотека как значение**: `Native.c("libsqlite3")` линкует библиотеку на этапе компиляции и возвращает резолвер; информация о библиотеке переносится через каррирование
2. **Внешний символ как значение**: применение резолвером имени символа даёт внешнюю ссылку, привязываемую через `name: type = value` (RFC-007/010) как тип или функцию
3. **Двухчастная типология**: непрозрачный дескриптор (размещение — внешнее) / прозрачный тип (размещение — YaoXiang); третьего варианта нет
4. **Изоляция маршалинга**: межграничные данные по умолчанию копируются во временную область вызова; объекты кучи YaoXiang изолированы от внешнего кода
5. **Безопасность владения**: единоличное владение дескриптором (перемещение) + RAII; структурная защита от двойного освобождения и use-after-free
6. **Аварийный люк**: сырые указатели `*T` + `unsafe {}` — пользователь явно принимает на себя риск прямого доступа к памяти без копирования

**Ключевая граница — пять нерушимых контрактов**:

```
1. Библиотека линкуется на этапе компиляции, существование символа проверяется на этапе компиляции
2. Принадлежность размещения типа определяется при объявлении: непрозрачный дескриптор — внешняя, прозрачный тип — YaoXiang
3. Маршалинг по умолчанию идёт через копирование во временную область; объекты кучи YaoXiang изолированы от внешнего кода
4. Единоличное владение дескриптором + перемещение; структурная защита от двойного освобождения и висячих указателей
5. Внешний код всегда читает и пишет в память с явным размещением и явным владением — никаких размытых зон
```

## Мотивация

### Текущее положение и цели

Текущий `native("symbol")` в кодовой базе — это всего лишь механизм диспетчеризации, через который байт-код YaoXiang вызывает функции Rust std (`FfiRegistry` = `HashMap<String, RustFnPtr>`). **Здесь нет никакой реальной межABI-границы**: ни `dlopen`, ни маршалинга C ABI, ни передачи владения памятью через границу.

Настоящий FFI должен решать четыре задачи:

| Задача | Ответ настоящего RFC |
|--------|---------------------|
| **Разрешение символов** | Библиотека — значение первого класса, линкуемое на этапе компиляции (`Native.c("lib")`); существование символа проверяется на этапе компиляции |
| **Маршалинг значений** | Управляется сигнатурой; правила преобразования для каждой позиции аргумента фиксируются на этапе компиляции |
| **Владение памятью** | Двухчастная типология определяет принадлежность; по умолчанию — изоляция через копирование |
| **Безопасность времени жизни** | Перемещение + RAII + заимствование, ограниченное одним вызовом |

RFC-020 и RFC-021 определяли разные стороны FFI и перекрывались; настоящий документ объединяет их в единую спецификацию.

### Цели проектирования

1. **Никаких утечек сырых указателей в пользовательский код**: в обычном использовании FFI в исходниках `.yx` сырые указатели не появляются
2. **Явная принадлежность размещения**: при определении типа пользователь решает, кому принадлежит эта память, без выводов во время выполнения
3. **Структурная безопасность**: защита от утечек, двойного освобождения и use-after-free обеспечивается системой типов, а не соглашениями
4. **Честная граница доверия**: C не может предоставить проверяемый на этапе компиляции контракт типов; доверие локализуется в точке объявления привязки
5. **Совместимость с самоподдержкой (self-hosting)**: никакой чрезмерной абстракции, уникальной для языка-хоста

### За рамками документа

- **Система плагинов для нескольких ABI-механизмов** (Wasm/Python/пользовательские ABI): см. RFC-026a
- **Инструментарий yx-bindgen**: см. RFC-026b
- **Экспорт функций YaoXiang для вызова из C (обратный FFI)**: последующие RFC; здесь изложены только принципы
- **Встроенный ассемблер, SIMD-интринсики**: за рамками настоящего RFC

## Предложение

### 1. Внешние библиотеки и символы: каррированные значения первого класса

Информационный пробел FFI — «какую библиотеку и какой символ линковать» — заполняется тем, что библиотека становится значением первого класса, без введения новых ключевых слов.

#### 1.1 Библиотека как значение

```yaoxiang
// Native.c применяет имя библиотеки → линкует её на этапе компиляции, возвращает резолвер символов
sqlite3 = Native.c("libsqlite3")
```

`Native.c("libsqlite3")` — это **действие на этапе компиляции + значение этапа выполнения**:

- **На этапе компиляции**: компоновщик получает `-lsqlite3`; библиотека попадает в таблицу символов; существование символа можно проверить
- **Значение**: `sqlite3` — это резолвер; применение имени символа даёт внешнюю ссылку этой библиотеки

`.c` — это тег ABI-механизма (C ABI). В ядро встроен только `.c`; прочие механизмы (`.wasm` и т. д.) см. в RFC-026a.

#### 1.2 Символ как значение; привязка — это `name: type = value`

Резолвер применяется к имени символа, и полученная внешняя ссылка привязывается через единый синтаксис RFC-007/010. **Аннотация типа в LHS определяет, является ли эта ссылка типом или функцией**:

```yaoxiang
sqlite3 = Native.c("libsqlite3")

// LHS — это Type → привязка как непрозрачного типа
SqliteDb: Type = sqlite3("sqlite3")

// LHS — это сигнатура функции → привязка как функции
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")
SqliteDb.exec: (sql: String) -> Int32 = sqlite3("sqlite3_exec")
SqliteDb.close: () -> Int32 = sqlite3("sqlite3_close")

// .drop — обычная привязка метода (соглашение RAII из RFC-009)
SqliteDb.drop = SqliteDb.close
```

Проверка на этапе компиляции: в `sqlite3("sqlite3_open")` символ `sqlite3_open` должен присутствовать в таблице символов `libsqlite3`, иначе — ошибка компиляции.

#### 1.3 Привязка методов и позиция self

В записи `Type.method: (...) -> ...` `self` неявно стоит на первом месте: при вызове `db.exec("SELECT")` значение `db` передаётся C-функции `sqlite3_exec` как нулевой аргумент.

Если нужно привязать уже объявленную отдельную функцию как метод, используется синтаксис `[N]` для указания позиции self (RFC-004 «каррирование и привязка к нескольким позициям»):

```yaoxiang
// Отдельная функция
sqlite3_close_v2: (db: SqliteDb) -> Int32 = sqlite3("sqlite3_close_v2")

// Привязка как метода, [0] означает, что self — db
SqliteDb.soft_close = sqlite3_close_v2[0]
```

И прямая привязка метода из `Native.c(...)`, и ручная привязка через `[N]` — это всё `name: type = value`; функциональное значение кладётся справа от `=`, и никаких двух механизмов нет.

#### 1.4 Использование: ноль unsafe, ноль сырых указателей

```yaoxiang
import sqlite3_bindings

db = SqliteDb.open("test.db")
db.exec("SELECT * FROM users")
// ← По окончании области видимости RAII автоматически вызывает SqliteDb.drop → sqlite3_close(db)
```

### 2. Двухчастная типология: принадлежность размещения фиксируется при определении

Когда внешние данные попадают в YaoXiang, задаётся лишь один вопрос: **кто определяет размещение этой памяти?**

```
├─ Размещение — внешний чёрный ящик (sqlite3, FILE*, сокет fd)
│   → Непрозрачный дескриптор  =  lib("symbol")
│   → YaoXiang хранит только указатель, никогда не разыменует его, лишь передаёт между функциями библиотеки
│   → Внешний код читает свою собственную память, YaoXiang её не трогает
│
└─ Размещение определяет YaoXiang (timespec, point, struct, у которого нужно читать поля)
    → Прозрачный тип  =  { field: Type, ... }
    → YaoXiang владеет памятью, определяет размещение, читает/пишет поля
    → Внешний код пишет и читает в память, размещение которой определил YaoXiang
```

**Третьего варианта нет.** Прежнее «трёхслойное» деление режимов памяти (копирование / поглощение / системный) было заплаточным мышлением — истина в бинарной принадлежности размещения.

#### 2.1 Непрозрачный дескриптор: размещение — внешнее

```yaoxiang
SqliteDb: Type = sqlite3("sqlite3")
```

- Внутри YaoXiang хранится только дескриптор размером в указатель
- Пользователь не может его сконструировать (`SqliteDb {}` → ошибка компиляции), не может обращаться к полям (полей для обращения нет)
- Единственный источник: внешняя функция, возвращающая `SqliteDb`
- При вызове метода дескриптор передаётся обратно в библиотеку, и та читает **свою** память (структура `sqlite3` лежит в куче библиотеки)

«Чтение внутренностей» внешним кодом — это чтение структур, выделенных им самим; YaoXiang лишь переносит дескриптор. Конфликтов в памяти нет.

#### 2.2 Прозрачный тип: размещение — YaoXiang

```yaoxiang
// Поля осмысленны и должны читаться/писаться → прозрачный тип, размещение объявляет YaoXiang
Timespec: Type = {
    tv_sec: Int64,
    tv_nsec: Int64
}
clock_gettime: (clk: Int32, ts: *Timespec) -> Int32 = Native.c("librt")("clock_gettime")

ts = clock_gettime(CLOCK_REALTIME)   // см. §3, маршалинг идёт через временную область
print(ts.tv_sec)                      // YaoXiang читает по своему объявлению полей
```

Внешний код читает и пишет в память, **размещение и владение которой определяет YaoXiang**. Размещение — это контракт YaoXiang, а не внешней стороны.

#### 2.3 Правило принятия решения

Пользователю нужно решить лишь одно: **нужно ли мне читать поля этого типа?**

| Решение | Тип | Принадлежность размещения |
|---------|-----|---------------------------|
| Поля не читаются, дескриптор лишь передаётся между функциями библиотеки | Непрозрачный дескриптор `= lib("sym")` | Внешняя |
| Нужно читать и писать поля | Прозрачный тип `{ ... }` | YaoXiang |

### 3. Маршалинг: на основе сигнатуры, изоляция через временную область

Преобразование межграничных данных управляется **сигнатурой**; для каждой позиции аргумента правило преобразования фиксируется на этапе компиляции. **Ключевая гарантия безопасности: внешний код читает и пишет во временную область маршалинга, а не в объекты кучи YaoXiang.**

#### 3.1 По умолчанию — копирование во временную область

```
YaoXiang → C (входные аргументы):
    копировать данные во временную область вызова → передать указатель на неё в C
    → выход за границы или порча памяти в C задевает только временную область, объект кучи YaoXiang изолирован

C → YaoXiang (возврат / выходные аргументы):
    C пишет во временную область → YaoXiang делает memcpy в свой собственный объект
    → C не прикасается к итоговому объекту YaoXiang
```

**Внешний код всегда читает и пишет во временную область маршалинга; объекты кучи YaoXiang полностью изолированы.** Ошибка в объявлении размещения, висячий указатель, сохранённый C-кодом, выход за границы — всё это вредит только временной области; объекты YaoXiang остаются целы. Цена — одно `memcpy`.

#### 3.2 Таблица правил маршалинга

**Направление входа (YaoXiang → C)**:

| Тип YaoXiang | Представление в C | Действие маршалинга | Владение |
|--------------|-------------------|---------------------|----------|
| `Int32/Int64/Float` | `int/long/double` | Кладётся прямо в регистр, ноль преобразований | Семантика значений |
| `String` | `const char*` | Выдаётся временное read-only представление (живо только во время вызова) | YaoXiang сохраняет владение, C только читает |
| Прозрачный тип | `struct T*` | Копируется во временную область, передаётся указатель на неё | YaoXiang владеет объектом, C читает копию |
| Непрозрачный дескриптор | `void*` | Извлекается указатель дескриптора | YaoXiang владеет, заимствуется C |
| `*T` | `T*` | Передаётся сырой указатель напрямую (unsafe) | Ответственность пользователя |

**Направление возврата (C → YaoXiang)**:

| Возврат C | Тип YaoXiang | Действие маршалинга | Владение |
|-----------|--------------|---------------------|----------|
| `int/double` | `Int32/Float` | Прямое чтение из регистра | Семантика значений |
| `char*` | `String` | `strlen` + `memcpy` в String YaoXiang | YaoXiang владеет копией, оригинальная память не затрагивается |
| `struct T*` (новый дескриптор) | Непрозрачный дескриптор | Указатель сохраняется в объекте YaoXiang | YaoXiang берёт на себя |
| `struct T` (значение / выходной аргумент) | Прозрачный тип | C пишет во временную область → `memcpy` обратно в YaoXiang | YaoXiang владеет |
| `char*` (статическая область) | `*const U8` | Сохраняется сырой указатель, без копирования (unsafe чтение) | Не поглощается, ответственность на пользователе |

#### 3.3 Время жизни заимствования: строго в пределах одного вызова

Указатели, которые YaoXiang заимствует у внешнего кода (read-only представление String, временная область прозрачного типа, дескриптор), **имеют время жизни, строго ограниченное одним вызовом**:

- Во время вызова: указатель валиден, внешний код может читать и писать
- После возврата: заимствование немедленно прекращается

Если внешний код сохраняет указатель для использования после вызова, он нарушает стандартный контракт FFI (равносильно багу в библиотеке); YaoXiang за это не отвечает. Это совпадает с контрактами C FFI во всех языках (аналогичное ограничение действует для `&T` из Rust, передаваемого в C).

#### 3.4 String никогда не отдаёт долгоживущий указатель

`String` — ключ к тому, чтобы «C не вмешивался в память YaoXiang»:

- Вход в C: выдаётся **временное read-only представление**, валидное только на время вызова
- Возврат из C: `strlen` + `memcpy` в **копию**, принадлежащую YaoXiang

C никогда не получает долгоживущий указатель на String YaoXiang, а YaoXiang никогда не хранит долгосрочную ссылку на C `char*`. Структурная изоляция.

### 4. Владение и время жизни: перемещение + RAII

Непрозрачные дескрипторы подчиняются модели владения из RFC-009, без каких-либо новых концепций.

#### 4.1 Ключевые принципы

- **Семантика перемещения**: непрозрачные дескрипторы по умолчанию перемещаются; присваивание / передача аргумента / возврат = передача владения; копирование запрещено
- **Единоличное владение дескриптором**: в любой момент у дескриптора ровно один владелец → структурная защита от двойного освобождения
- **Освобождение через RAII**: по окончании области видимости, если привязан `.drop`, он вызывается автоматически
- **Отслеживание поглощения**: после явного уничтожения или перемещения переменная считается поглощённой и недоступна → защита от use-after-free

#### 4.2 `.drop` — необязательный внешний побочный эффект

```yaoxiang
SqliteDb.drop = SqliteDb.close     // при выходе из области видимости вызывается sqlite3_close
```

**`.drop` — это не механизм защиты от утечек YaoXiang**: хранение дескриптора на стороне YaoXiang (значение размером в указатель) освобождается автоматически и не зависит от `.drop`. `.drop` — это **необязательный побочный эффект «по случаю выхода из области видимости» — вызов внешней функции**:

- Привязан `.drop` → по окончании области видимости он вызывается (освобождает внешний ресурс)
- Не привязан `.drop` → ничего не происходит, **ни ошибки, ни предупреждения**

Нужно ли очищать внешний ресурс, определяется спецификацией внешней библиотеки (`getenv` возвращает статическую область, которую нельзя освобождать; глобальные одиночки не освобождаются); YaoXiang не выходит за рамки своих полномочий. Защита от утечек обеспечивается перемещением и единоличным владением (безусловно и структурно), а не `.drop`.

#### 4.3 Автоматический вызов деструкторов и их порядок

```yaoxiang
{
    db = SqliteDb.open("test.db")
    stmt = db.prepare("SELECT * FROM users")
    // ← По окончании области видимости деструкторы вызываются в обратном порядке (если привязан .drop):
    //   stmt.drop()  → sqlite3_finalize(stmt)
    //   db.drop()    → sqlite3_close(db)
}
```

Порядок вызова деструкторов: обратный порядку объявления, в соответствии с RAII.

#### 4.4 Перемещение и поглощение

```yaoxiang
db = SqliteDb.open("test.db")
db2 = db                // Перемещение: владение передаётся
db.exec("...")          // ❌ Ошибка компиляции: db уже перемещён, после поглощения чтение невозможно

process_db: (db: SqliteDb) -> Void = {
    db.exec("...")
    // ← По окончании функции db уничтожается здесь
}
process_db(some_db)     // Перемещение в функцию
// some_db здесь уже невалиден
```

#### 4.5 Обработка null

```yaoxiang
// Может вернуть null → ?T, пользователь обязан обработать
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")

db = SqliteDb.open("test.db")
match db {
    Some(db) => db.exec("SELECT 1"),
    None => print("не удалось открыть")
}

// Соглашение «не возвращает null» → не помечается, при null — panic, чтобы вскрыть проблему
```

Возврат null из C либо обрабатывается пользователем (`?T`), либо приводит к panic. Третьего «тихого игнорирования» нет.

#### 4.6 Обработка сбоя деструктора

Поведение определяется типом возврата функции, привязанной к `.drop`:

| Возврат `.drop` | Поведение |
|-----------------|-----------|
| `Void` | Сбоя нет |
| `Int32` (код ошибки) | При ненулевом значении — panic: сбой деструктора означает аномальное состояние, лучше вскрыть, чем молчать |
| `?Error` | При `Some` — panic: то же |

Сбой деструктора нельзя игнорировать молча. Если нужно проглотить конкретную ошибку, это делается явно в обёрточной функции, привязанной к `.drop`.

### 5. Поведение FFI в spawn-блоках

Решение о принадлежности к ресурсному типу определяется привязкой `.drop` (RFC-024), без каких-либо дополнительных пометок:

| Решение | Поведение |
|---------|-----------|
| У непрозрачного дескриптора привязан `.drop` | Ресурсный тип — операции над одним и тем же экземпляром в spawn-блоке автоматически сериализуются |
| У непрозрачного дескриптора не привязан `.drop` | Не ресурсный тип — допускается параллелизм (чистый дескриптор-данные, без побочного эффекта освобождения) |
| Прозрачный тип / тип-значение | Не ресурсный тип — допускается параллелизм |

```yaoxiang
SqliteDb.drop = SqliteDb.close   // → ресурсный тип

(a, b) = spawn {
    r1 = db.exec("SELECT ..."),   // один и тот же экземпляр, автоматическая сериализация
    r2 = db.exec("INSERT ...")    // ожидает r1
}

(x, y) = spawn {
    db1 = SqliteDb.open("a.db"),   // разные экземпляры, можно параллельно
    db2 = SqliteDb.open("b.db")
}
```

Типы с привязанным `.drop` автоматически сериализуют операции над одним экземпляром внутри spawn, гарантируя отсутствие гонок при вызове деструктора.

### 6. Аварийный люк: сырые указатели + unsafe

По умолчанию маршалинг идёт через копирование во временную область — это безопасно, но несёт накладные расходы `memcpy`. В чувствительных к производительности сценариях (большие структуры, частые вызовы), когда нужно обойтись без копирования, пользователь явно использует аварийный люк с сырыми указателями:

```yaoxiang
// C напрямую читает память YaoXiang, ноль копирований — пользователь явно принимает риск
ptr: *const U8 = Native.c("libc")("getenv")("HOME")
unsafe {
    value = read_c_string(ptr)   // пользователь гарантирует валидность ptr
}
```

**`unsafe` используется только для операций с сырыми указателями и полностью ортогонален непрозрачным дескрипторам и прозрачным типам.** В обычном FFI (дескрипторы + прозрачные типы) `unsafe` не нужен. Писать `unsafe {}` означает, что пользователь явно подписывается под риск прямого доступа к памяти.

**Граница доверия**: C не может предоставить проверяемый на этапе компиляции контракт типов (`.h` — это не ABI-контракт; в таблице символов есть только имена, но не сигнатуры). Поэтому корректность C-сигнатуры нельзя верифицировать автоматически — её гарантирует автор привязки, когда пишет `Native.c(...)` вместе с сигнатурой. **Доверие локализуется в точке объявления**: автор привязки даёт гарантию, а пользователь пакета получает безопасный API. Это согласуется с `extern "C"` в Rust (написание extern — акт доверия; безопасная обёртка делает вызовы безопасными).

## Компромиссы

### Преимущества

1. **Информационная полнота**: библиотека линкуется на этапе компиляции, символ проверяется на этапе компиляции; никакой неопределённости вроде «библиотека не найдена» во время выполнения
2. **Явная принадлежность размещения**: двухчастная типология фиксируется при объявлении, без выводов во время выполнения
3. **Структурная безопасность**: изоляция через временную область + перемещение + RAII; внешний код не прикасается к объектам кучи YaoXiang
4. **Никаких новых ключевых слов**: каррирование `Native.c` + `name: type = value`, повторное использование существующего синтаксиса
5. **Честная граница**: не притворяемся, что можем верифицировать C-сигнатуру; доверие локализуется в точке объявления

### Недостатки

1. **Накладные расходы `memcpy`**: по умолчанию маршалинг копирует; для больших структур с частыми вызовами приходится явно идти через аварийный люк
2. **Совпадение размещения проверяется вручную**: соответствие размещения прозрачного типа C-структуре обеспечивается автором привязки / yx-bindgen
3. **C-сигнатуру нельзя верифицировать на этапе компиляции**: фундаментальное ограничение FFI; YaoXiang не может устранить его за C

## Стратегия реализации

### Этап 1: внешние библиотеки и символы (v0.8)

- [ ] Реализовать линковку на этапе компиляции в `Native.c("lib")` + возврат значения-резолвера
- [ ] Реализовать применение резолвера символов (`lib("symbol")`) + проверку таблицы символов на этапе компиляции
- [ ] Реализовать двухчастную типологию (непрозрачный дескриптор / прозрачный тип)
- [ ] Реализовать привязку методов (прямую + позиционную через `[N]`)

### Этап 2: маршалинг и безопасность (v0.8)

- [ ] Реализовать генерацию маршалингового кода на основе сигнатуры
- [ ] Реализовать изоляцию через временную область (копирование входных аргументов, `memcpy` возврата)
- [ ] Реализовать временное read-only представление для String + копирование при возврате
- [ ] Реализовать время жизни заимствования в пределах одного вызова

### Этап 3: владение и время жизни (v0.9)

- [ ] Реализовать перемещение непрозрачных дескрипторов + единоличное владение
- [ ] Реализовать автоматический вызов RAII-деструктора через `.drop` (необязательно, отсутствие — не ошибка)
- [ ] Реализовать отслеживание поглощения (запрет использования после перемещения)
- [ ] Реализовать интеграцию `?T` с возвратом null
- [ ] Реализовать сериализацию ресурсных типов в spawn

### Дальнейшая работа

- **Расширяемые механизмы FFI** (RFC-026a): абстракция `FfiMechanism`, плагины `.wasm`/`.python` и т. д., динамическая загрузка
- **yx-bindgen** (RFC-026b): C-заголовки → привязки `.yx` + платформо-корректная генерация размещения

## Журнал проектных решений

| Решение | Выбор | Причина | Дата |
|---------|-------|---------|------|
| **Библиотека как значение** | `Native.c("lib")` через каррирование возвращает резолвер | Информация о библиотеке становится значением первого класса, видимым на этапе компиляции; заполняется пробел «какую библиотеку линковать», без новых ключевых слов | 2026-07-03 |
| **Линковка на этапе компиляции** | `Native.c("lib")` запускает `-llib` | Таблица символов доступна на этапе компиляции; существование символа проверяемо; типы — настоящие | 2026-07-03 |
| **Двухчастная типология** | Непрозрачный дескриптор / прозрачный тип | Бинарная принадлежность размещения покрывает все случаи; убрана «заплаточная» трёхслойная модель памяти | 2026-07-03 |
| **Изоляция маршалинга через временную область** | По умолчанию — копирование, объект кучи изолирован | Выход за границы, висячие указатели во внешнем коде задевают только временную область; объекты YaoXiang целы; для нулевого копирования — явный аварийный люк | 2026-07-03 |
| **`.drop` необязателен** | Если отсутствует — ничего не делать, без ошибок | Хранение дескриптора на стороне YaoXiang освобождается автоматически; очистка внешнего ресурса — дело спецификации внешней библиотеки, YaoXiang не выходит за рамки | 2026-07-03 |
| **Защита от утечек** | Перемещение + единоличное владение (безусловно) | Структурная гарантия, не зависит от `.drop` | 2026-07-03 |
| **Граница доверия** | В точке объявления `Native.c(...)` | C-сигнатуру нельзя верифицировать на этапе компиляции; доверие локализуется; `unsafe` нужен только для сырых указателей | 2026-07-03 |
| **Обработка null** | `?T` или panic | Проблемы C не скрываются; опции «тихого игнорирования» нет | 2026-07-03 |
| **Сбой деструктора** | Определяется типом возврата `.drop`, единообразно — panic | Сбой деструктора нельзя игнорировать молча | 2026-07-03 |
