## Аннотация

Данный документ определяет основные механизмы FFI (Foreign Function Interface, интерфейс внешних функций) для YaoXiang. Ключевая идея: **внешние библиотеки являются значениями первого класса, линкуемыми во время compile-time; принадлежность расположения данных при пересечении границ закрепляется во время определения типа; кучные объекты YaoXiang структурно изолированы от внешнего кода.**

1. **Внешняя библиотека как значение**: `Native.c("libsqlite3")` линкует библиотеку во время compile-time и возвращает резолвер, несущий информацию о библиотеке посредством каррирования.
2. **Внешний символ как значение**: применение резолвером имени символа возвращает внешнюю ссылку, которая посредством `name: type = value` (RFC-007/010) привязывается как тип или функция.
3. **Два типа**: непрозрачный дескриптор (расположение принадлежит внешнему коду) / прозрачный тип (расположение принадлежит YaoXiang); третьего варианта не существует.
4. **Изоляция при маршалинге**: данные, пересекающие границу, по умолчанию копируются во временную область вызова; кучные объекты YaoXiang изолированы от внешнего кода.
5. **Безопасность ownership**: уникальное владение дескриптором (Move) + RAII, структурное предотвращение двойного освобождения и use-after-free.
6. **Аварийный люк**: сырые указатели `*T` + `unsafe {}`, пользователь явно принимает риск прямого доступа к памяти без копирования.

**Ключевая граница — пять нерушимых контрактов**:

```
1. Библиотека линкуется во время compile-time, существование символов проверяется во время compile-time.
2. Принадлежность расположения типа закрепляется во время определения: непрозрачный дескриптор принадлежит внешнему коду, прозрачный тип — YaoXiang.
3. Маршалинг по умолчанию проходит через копирование во временную область; кучные объекты YaoXiang изолированы от внешнего кода.
4. Уникальное владение дескриптором + Move, структурное предотвращение двойного освобождения / Dangling-указателей.
5. Внешний код всегда читает/пишет память с «чётко определённым расположением и ясным ownership»; размытых зон не существует.
```

## Мотивация

### Текущее состояние и цели

Текущий `native("symbol")` в кодовой базе — это лишь механизм диспетчеризации байт-кода YaoXiang к функциям Rust std (`FfiRegistry` = `HashMap<String, RustFnPtr>`); **настоящего пересечения границ ABI здесь нет** — нет ни dlopen, ни маршалинга C ABI, ни передачи ownership памяти через границу.

Настоящий FFI должен решать четыре задачи:

| Задача | Ответ данного RFC |
|--------|-------------------|
| **Разрешение символов** | Библиотека — значение первого класса, линкуемое во время compile-time (`Native.c("lib")`); символы проверяются во время compile-time. |
| **Маршалинг значений** | Управляется сигнатурой: compile-time определяет правила преобразования для каждой позиции аргумента. |
| **Ownership памяти** | Два типа определяют принадлежность; по умолчанию изоляция через копирование. |
| **Безопасность жизненного цикла** | Move + RAII + заимствование ограничено одним вызовом. |

RFC-020 и RFC-021 определяют разные аспекты FFI, при этом перекрывающиеся; данный документ объединяет их в единую спецификацию.

### Цели проектирования

1. **Никаких сырых указателей в пользовательском коде**: в исходниках `.yx` в штатном использовании FFI не должно быть сырых указателей.
2. **Явная принадлежность расположения**: пользователь при определении типа решает, кому принадлежит эта память, без выводов во время выполнения.
3. **Структурная безопасность**: утечки, двойное освобождение, use-after-free гарантируются системой типов, а не соглашениями.
4. **Честная граница доверия**: C не может предоставить верифицируемые во время compile-time контракты типов; доверие локализуется в точке объявления привязки.
5. **Совместимость с bootstrap**: никакой избыточной абстракции, специфичной для языка-хоста.

### За пределами области применения

- **Система плагинов множественных ABI-механизмов** (Wasm/Python/пользовательский ABI): см. RFC-026a.
- **Цепочка инструментов yx-bindgen**: см. RFC-026b.
- **Экспорт функций YaoXiang для вызова из C (обратный FFI)**: последующий RFC; данный документ лишь провозглашает принципы.
- **Встроенный ассемблер, SIMD-intrinsics**: за пределами данного RFC.

## Предложение

### 1. Внешние библиотеки и символы: каррированные значения первого класса

Информационный пробел FFI — «к какой библиотеке, к какому символу подключаться» — заполняется тем, что библиотека становится значением первого класса, без введения каких-либо новых ключевых слов.

#### 1.1 Библиотека как значение

```yaoxiang
// Native.c применяет имя библиотеки → линкует её во время compile-time, возвращает резолвер символов
sqlite3 = Native.c("libsqlite3")
```

`Native.c("libsqlite3")` — это **compile-time действие + runtime значение**:

- **Compile-time**: линковщик получает `-lsqlite3`, библиотека попадает в таблицу символов, существование символов верифицируемо.
- **Значение**: `sqlite3` — это резолвер; применение имени символа возвращает внешнюю ссылку на эту библиотеку.

`.c` — метка ABI-механизма (C ABI). В ядре встроен только `.c`; прочие механизмы (`.wasm` и т. д.) см. в RFC-026a.

#### 1.2 Символ как значение, привязка как `name: type = value`

Применение резолвером имени символа возвращает внешнюю ссылку, которая привязывается посредством унифицированного синтаксиса из RFC-007/010. **Аннотация типа в LHS определяет, является ли эта ссылка типом или функцией**:

```yaoxiang
sqlite3 = Native.c("libsqlite3")

// LHS — Type → привязка как непрозрачный тип
SqliteDb: Type = sqlite3("sqlite3")

// LHS — сигнатура функции → привязка как функция
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")
SqliteDb.exec: (sql: String) -> Int32 = sqlite3("sqlite3_exec")
SqliteDb.close: () -> Int32 = sqlite3("sqlite3_close")

// .drop — обычная привязка метода (соглашение RAII из RFC-009)
SqliteDb.drop = SqliteDb.close
```

Проверка во время compile-time: `sqlite3_open` в `sqlite3("sqlite3_open")` должен присутствовать в таблице символов `libsqlite3`, иначе ошибка компиляции.

#### 1.3 Привязка методов и позиция self

В записи `Type.method: (...) -> ...` `self` неявно стоит на первой позиции — при вызове `db.exec("SELECT")` `db` передаётся как нулевой аргумент C-функции `sqlite3_exec`.

Если нужно привязать уже объявленную отдельную функцию как метод, используется синтаксис `[N]` для указания позиции self (многопозиционное каррирование из RFC-004):

```yaoxiang
// Отдельная функция
sqlite3_close_v2: (db: SqliteDb) -> Int32 = sqlite3("sqlite3_close_v2")

// Привязка как метод, [0] означает, что db — это self
SqliteDb.soft_close = sqlite3_close_v2[0]
```

Прямая привязка метода через `Native.c(...)` и ручная привязка через `[N]` — это всё `name: type = value`, всё это помещение функционального значения справа от `=`; никаких двух механизмов.

#### 1.4 Использование пользователем: ноль unsafe, ноль сырых указателей

```yaoxiang
import sqlite3_bindings

db = SqliteDb.open("test.db")
db.exec("SELECT * FROM users")
// ← Конец области видимости, RAII автоматически вызывает SqliteDb.drop → sqlite3_close(db)
```

### 2. Два типа: принадлежность расположения закрепляется во время определения

При попадании внешних данных в YaoXiang задаётся лишь один вопрос: **кто определяет расположение этой памяти?**

```
├─ Расположение — чёрный ящик снаружи (sqlite3, FILE*, socket fd)
│   → Непрозрачный дескриптор  =  lib("symbol")
│   → YaoXiang хранит лишь указатель, никогда не разыменует, лишь передаёт между функциями библиотеки
│   → Внешний код читает собственную память, YaoXiang её не трогает
│
└─ Расположение определяется YaoXiang (timespec, point, struct с читаемыми полями)
    → Прозрачный тип  =  { field: Type, ... }
    → YaoXiang владеет памятью, определяет расположение, читает/пишет поля
    → Внешний код читает/пишет в память, расположение которой задано YaoXiang
```

**Третьего варианта нет.** «Трёхуровневая модель памяти (копирование / принятие / системный уровень)» из предыдущих редакций — мышление через заплатки; правда же — это деление по принадлежности расположения на две части.

#### 2.1 Непрозрачный дескриптор: расположение принадлежит внешнему коду

```yaoxiang
SqliteDb: Type = sqlite3("sqlite3")
```

- Внутри YaoXiang хранится лишь дескриптор размером с указатель.
- Пользователь не может конструировать (`SqliteDb {}` → ошибка компиляции), не может обращаться к полям (полей для обращения нет).
- Единственный источник: внешняя функция, возвращающая `SqliteDb`.
- При вызове метода дескриптор передаётся обратно в библиотеку, и она читает **свою собственную** память (структура `sqlite3` в куче библиотеки).

Когда внешний код «читает внутренности», он читает свою собственную выделенную структуру; YaoXiang лишь переносит дескриптор. Конфликтов памяти нет.

#### 2.2 Прозрачный тип: расположение принадлежит YaoXiang

```yaoxiang
// Поля осмысленны, нужно читать/писать → прозрачный тип, расположение объявляет YaoXiang
Timespec: Type = {
    tv_sec: Int64,
    tv_nsec: Int64
}
clock_gettime: (clk: Int32, ts: *Timespec) -> Int32 = Native.c("librt")("clock_gettime")

ts = clock_gettime(CLOCK_REALTIME)   // см. §3, маршалинг через временную область
print(ts.tv_sec)                      // YaoXiang читает по своему определению полей
```

Внешний код читает/пишет в память, **расположение которой задано YaoXiang и которая принадлежит YaoXiang**. Расположение — это контракт YaoXiang, а не внешнего кода.

#### 2.3 Правило определения

Пользователь отвечает лишь на один вопрос: **нужно ли мне читать поля этого типа?**

| Решение | Тип | Принадлежность расположения |
|---------|-----|----------------------------|
| Поля не читаются, дескриптор лишь передаётся между функциями библиотеки | Непрозрачный дескриптор `= lib("sym")` | Внешний код |
| Поля читаются/пишутся | Прозрачный тип `{ ... }` | YaoXiang |

### 3. Маршалинг: управляется сигнатурой, изоляция через временную область

Преобразование данных при пересечении границы **управляется сигнатурой**: compile-time определяет правила преобразования для каждой позиции аргумента. **Ключевая гарантия безопасности: внешний код читает/пишет временную область маршалинга, а не кучные объекты YaoXiang.**

#### 3.1 По умолчанию через копирование во временную область

```
YaoXiang → C (входные аргументы):
    Копировать данные во временную область вызова → передать указатель на временную область в C
    → Выход C за границы / порча повреждает только временную область,
       кучные объекты YaoXiang изолированы

C → YaoXiang (возврат / выходные аргументы):
    C пишет во временную область → YaoXiang memcpy в свой объект
    → C не может коснуться конечного объекта YaoXiang
```

**Внешний код всегда читает/пишет временную область маршалинга, полностью изолированную от кучных объектов YaoXiang.** Ошибка в объявлении расположения, сохранённый C Dangling-указатель, выход C за границы — всё это повреждает лишь временную область, объект YaoXiang остаётся цел. Цена — один memcpy.

#### 3.2 Таблица правил маршалинга

**Направление входных аргументов (YaoXiang → C)**:

| Тип YaoXiang | Представление C | Действие маршалинга | Ownership |
|--------------|-----------------|---------------------|-----------|
| `Int32/Int64/Float` | `int/long/double` | Непосредственно в регистр, без преобразований | value-семантика |
| `String` | `const char*` | Выдать readonly-представление (временное, действительно на время вызова) | YaoXiang сохраняет, C только читает |
| Прозрачный тип | `struct T*` | Скопировать во временную область, передать указатель на временную область | YaoXiang владеет объектом, C читает копию |
| Непрозрачный дескриптор | `void*` | Извлечь внутренний указатель дескриптора | YaoXiang держит, передаёт C в заимствование |
| `*T` | `T*` | Непосредственно передать сырой указатель (unsafe) | Ответственность пользователя |

**Направление возврата (C → YaoXiang)**:

| Возврат C | Тип YaoXiang | Действие маршалинга | Ownership |
|-----------|--------------|---------------------|-----------|
| `int/double` | `Int32/Float` | Непосредственно из регистра | value-семантика |
| `char*` | `String` | strlen + memcpy в String YaoXiang | YaoXiang владеет копией, исходную память не трогает |
| `struct T*` (новый дескриптор) | Непрозрачный дескриптор | Дескриптор сохраняется в объект YaoXiang | YaoXiang принимает |
| `struct T` (значение / выходной аргумент) | Прозрачный тип | C пишет во временную область → memcpy в YaoXiang | YaoXiang владеет |
| `char*` (статическая область) | `*const U8` | Сохранить сырой указатель, не копировать (unsafe-чтение) | Не принимается, ответственность пользователя |

#### 3.3 Время жизни заимствования: строго в пределах одного вызова

Указатели, заимствованные YaoXiang у внешнего кода (readonly-представление String, временная область прозрачного типа, дескриптор), **имеют время жизни, строго ограниченное одним вызовом**:

- Во время вызова: указатель действителен, внешний код может читать/писать.
- После возврата из вызова: заимствование немедленно прекращается.

Если внешний код сохраняет указатель для использования после вызова, это нарушение внешним кодом стандартного контракта FFI (эквивалентно багу библиотеки); YaoXiang за это не отвечает. Это согласуется с FFI-контрактом C во всех языках (аналогично `&T` в Rust, передаваемый в C).

#### 3.4 String никогда не отдаёт постоянный указатель

`String` — ключевой момент в принципе «C не вмешивается в память YaoXiang»:

- В C: выдаётся **временное readonly-представление**, действительное на время вызова.
- Из C: strlen + memcpy в **копию**, принадлежащую YaoXiang.

C никогда не получает постоянного указателя на String YaoXiang, YaoXiang никогда не хранит долгосрочную ссылку на C `char*`. Структурная изоляция.

### 4. Ownership и время жизни: Move + RAII

Непрозрачные дескрипторы следуют модели ownership из RFC-009, без новых концепций.

#### 4.1 Основные принципы

- **Семантика Move**: непрозрачный дескриптор по умолчанию Move; присваивание / передача аргумента / возврат = передача ownership, копирование запрещено.
- **Уникальное владение дескриптором**: в любой момент у дескриптора лишь один владелец → структурное предотвращение двойного освобождения.
- **RAII-освобождение**: при завершении области видимости, если привязан `.drop`, он вызывается автоматически.
- **Отслеживание потребления**: после явного уничтожения или Move переменная считается потреблённой и не может быть использована → предотвращение use-after-free.

#### 4.2 `.drop` — опциональный внешний побочный эффект

```yaoxiang
SqliteDb.drop = SqliteDb.close     // при завершении области видимости вызывает sqlite3_close
```

**`.drop` — это не механизм защиты от утечек YaoXiang** — хранение дескриптора со стороны YaoXiang (значение размером с указатель) освобождается автоматически, и `.drop` тут ни при чём. `.drop` — это **опциональный побочный эффект «попутно вызвать внешнюю функцию» при завершении области видимости**:

- Привязан `.drop` → при завершении области видимости вызывается (очистка внешних ресурсов).
- Не привязан `.drop` → ничего не происходит, **ни ошибки, ни предупреждения**.

Нужно ли очищать внешний ресурс — вопрос спецификации внешней библиотеки (результат `getenv` в статической области освобождать нельзя, глобальный singleton освобождать нельзя); YaoXiang не выходит за свои полномочия и не принуждает. Защита от утечек обеспечивается Move + уникальным владением (безусловно, структурно), а не `.drop`.

#### 4.3 Автоматический вызов деструкторов и порядок

```yaoxiang
{
    db = SqliteDb.open("test.db")
    stmt = db.prepare("SELECT * FROM users")
    // ← Конец области видимости, в обратном порядке автоматически
    //   вызываются деструкторы (только при наличии .drop):
    //   stmt.drop()  → sqlite3_finalize(stmt)
    //   db.drop()    → sqlite3_close(db)
}
```

Порядок вызова деструкторов: обратный порядку определения, согласно RAII.

#### 4.4 Move и потребление

```yaoxiang
db = SqliteDb.open("test.db")
db2 = db                // Move: передача ownership
db.exec("...")          // ❌ Ошибка компиляции: db уже перемещён, чтение после потребления невозможно

process_db: (db: SqliteDb) -> Void = {
    db.exec("...")
    // ← конец функции, db уничтожается здесь
}
process_db(some_db)     // Move в функцию
// some_db здесь уже недействителен
```

#### 4.5 Обработка null

```yaoxiang
// Возможен возврат null → ?T, пользователь обязан обработать
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")

db = SqliteDb.open("test.db")
match db {
    Some(db) => db.exec("SELECT 1"),
    None => print("Не удалось открыть")
}

// По контракту null не возвращается → не помечается, при null — panic для обнаружения
```

При возврате C значения null пользователь либо обрабатывает его (`?T`), либо получает panic. Третьего варианта «молчаливого игнорирования» нет.

#### 4.6 Обработка ошибок деструкторов

Тип возвращаемого значения функции, привязанной к `.drop`, определяет поведение:

| Возврат `.drop` | Поведение |
|-----------------|-----------|
| `Void` | Без сбоев |
| `Int32` (код ошибки) | При ненулевом значении — panic: сбой деструктора означает аномальное состояние, обнаружение лучше замалчивания |
| `?Error` | При non-None — panic: то же |

Сбой деструктора нельзя замалчивать. Если нужно проигнорировать конкретную ошибку, это делается явно в функции-обёртке, привязанной к `.drop`.

### 5. Поведение FFI в spawn-блоках

Определение ресурсного типа зависит от привязки `.drop` (RFC-024), без дополнительных пометок:

| Определение | Поведение |
|-------------|-----------|
| Непрозрачный дескриптор с привязанным `.drop` | Ресурсный тип — операции над одним и тем же экземпляром в spawn-блоке автоматически сериализуются |
| Непрозрачный дескриптор без привязанного `.drop` | Нересурсный тип — допускает параллелизм (чисто данные-дескриптор, без побочного эффекта освобождения) |
| Прозрачный тип / value-тип | Нересурсный тип — допускает параллелизм |

```yaoxiang
SqliteDb.drop = SqliteDb.close   // → ресурсный тип

(a, b) = spawn {
    r1 = db.exec("SELECT ..."),   // один экземпляр, автоматически сериализуется
    r2 = db.exec("INSERT ...")    // ожидает r1
}

(x, y) = spawn {
    db1 = SqliteDb.open("a.db"),   // разные экземпляры, параллельно
    db2 = SqliteDb.open("b.db")
}
```

Типы с привязанным `.drop` в spawn автоматически сериализуют операции над одним экземпляром, гарантируя отсутствие гонок при уничтожении.

### 6. Аварийный люк: сырые указатели + unsafe

По умолчанию маршалинг проходит через копирование во временную область — это безопасно, но несёт расходы на memcpy. В сценариях, чувствительных к производительности (большие структуры, высокочастотные вызовы), где нужен доступ без копирования, пользователь явно использует аварийный люк с сырыми указателями:

```yaoxiang
// C читает память YaoXiang напрямую, без копирования — пользователь явно принимает риск
ptr: *const U8 = Native.c("libc")("getenv")("HOME")
unsafe {
    value = read_c_string(ptr)   // пользователь гарантирует валидность ptr
}
```

**`unsafe` предназначен только для операций с сырыми указателями и полностью ортогонален непрозрачным дескрипторам и прозрачным типам.** В штатном FFI (дескрипторы + прозрачные типы) unsafe не нужен. Написание `unsafe {}` = пользователь явно подписывается под принятие риска прямого доступа к памяти.

**Граница доверия**: C не может предоставить верифицируемые во время compile-time контракты типов (`.h` — это не контракт ABI, в таблице символов есть только имена, но не сигнатуры). Поэтому корректность C-сигнатуры нельзя верифицировать автоматически — её гарантирует автор привязки при написании `Native.c(...)` + сигнатуры. **Доверие локализуется в точке объявления привязки**: автор привязки гарантирует, пользователь пакета получает безопасный API. Это согласуется с `extern "C"` в Rust (написание extern — акт доверия, после оборачивания в безопасный wrapper вызов безопасен).

## Компромиссы

### Преимущества

1. **Полнота информации**: библиотека линкуется во время compile-time, символы проверяются во время compile-time; нет неоднозначности «библиотека не найдена» во время выполнения.
2. **Явная принадлежность расположения**: деление на два типа, закреплённое во время определения, без выводов во время выполнения.
3. **Структурная безопасность**: изоляция через временную область + Move + RAII, внешний код не может коснуться кучных объектов YaoXiang.
4. **Никаких новых ключевых слов**: каррирование `Native.c` + `name: type = value`, полностью на существующем синтаксисе.
5. **Честная граница**: не притворяется, что можно верифицировать C-сигнатуру; доверие локализуется в точке объявления.

### Недостатки

1. **Расходы на memcpy**: маршалинг по умолчанию через копирование; для больших структур с высокочастотными вызовами нужен явный аварийный люк.
2. **Совпадение расположения задаётся вручную**: соответствие расположения прозрачного типа C-структуре гарантируется автором привязки / yx-bindgen.
3. **C-сигнатуру нельзя верифицировать во время compile-time**: фундаментальное ограничение FFI, YaoXiang не может устранить его со стороны C.

## Стратегия реализации

### Этап 1: Внешние библиотеки и символы (v0.8)

- [ ] Реализовать compile-time линковку `Native.c("lib")` + возврат значения-резолвера.
- [ ] Реализовать применение резолвера символов (`lib("symbol")`) + проверку таблицы символов во время compile-time.
- [ ] Реализовать деление типов на два (непрозрачный дескриптор / прозрачный тип).
- [ ] Реализовать привязку методов (прямая привязка + позиционная привязка через `[N]`).

### Этап 2: Маршалинг и безопасность (v0.8)

- [ ] Реализовать генерацию кода маршалинга, управляемого сигнатурой.
- [ ] Реализовать изоляцию через копирование во временную область (копирование входных аргументов, memcpy возврата).
- [ ] Реализовать временное readonly-представление String + копирование возврата.
- [ ] Реализовать ограничение времени жизни заимствования одним вызовом.

### Этап 3: Ownership и время жизни (v0.9)

- [ ] Реализовать Move + уникальное владение для непрозрачных дескрипторов.
- [ ] Реализовать RAII автоматический вызов деструктора через `.drop` (опционально, отсутствие — не ошибка).
- [ ] Реализовать отслеживание потребления (запрет использования после Move).
- [ ] Реализовать интеграцию `?T` с возвратом null.
- [ ] Реализовать сериализацию ресурсных типов в spawn.

### Дальнейшая работа

- **Расширяемые механизмы FFI** (RFC-026a): абстракция `FfiMechanism`, плагины `.wasm`/`.python` и др., динамическая загрузка.
- **yx-bindgen** (RFC-026b): заголовочные файлы C → привязки `.yx` + генерация корректного для платформы расположения.

## Журнал проектных решений

| Решение | Выбор | Причина | Дата |
|---------|-------|---------|------|
| **Библиотека как значение** | `Native.c("lib")` возвращает резолвер через каррирование | Информация о библиотеке становится compile-time видимым значением первого класса, закрывает пробел «к какой библиотеке подключаться», без новых ключевых слов | 2026-07-03 |
| **Compile-time линковка** | `Native.c("lib")` запускает `-llib` | Таблица символов доступна во время compile-time, существование символов верифицируемо, типы — реальны | 2026-07-03 |
| **Два типа** | Непрозрачный дескриптор / прозрачный тип | Деление по принадлежности расположения покрывает всё; «трёхуровневая модель памяти» как заплатка удалена | 2026-07-03 |
| **Изоляция маршалинга через временную область** | По умолчанию копирование, изоляция кучных объектов от внешнего кода | Выход за границы / Dangling-указатели затрагивают лишь временную область, объект YaoXiang цел; для доступа без копирования нужен явный аварийный люк | 2026-07-03 |
| **`.drop` опционален** | При отсутствии — ничего, без ошибки | Хранение дескриптора со стороны YaoXiang освобождается автоматически; очистка внешних ресурсов — вопрос внешней спецификации, без принуждения | 2026-07-03 |
| **Механизм защиты от утечек** | Move + уникальное владение дескриптором (безусловно) | Структурная гарантия, не зависит от `.drop` | 2026-07-03 |
| **Граница доверия** | В точке объявления `Native.c(...)` | C-сигнатуру нельзя верифицировать во время compile-time; доверие локализуется, unsafe — только для сырых указателей | 2026-07-03 |
| **Обработка null** | `?T` или panic | Проблему C не скрываем, варианта «молчаливого игнорирования» нет | 2026-07-03 |
| **Сбой деструктора** | Определяется типом возврата `.drop`, единообразно — panic | Сбой деструктора нельзя замалчивать | 2026-07-03 |
