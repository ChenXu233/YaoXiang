## Аннотация

Данный документ определяет основные механизмы FFI (внешнего интерфейса функций) в YaoXiang. Ключевая идея: **внешние библиотеки являются значениями первого класса с компиляционной компоновкой, распределение памяти для данных через границы определяется при определении типа, а объекты кучи YaoXiang структурно изолированы от внешнего кода.**

1. **Внешняя библиотека как значение**: `Native.c("libsqlite3")` компонует библиотеку на этапе компиляции и возвращает резолвер, использующий каррирование для хранения информации о библиотеке
2. **Внешний символ как значение**: применение резолвера к имени символа даёт внешнюю ссылку, которая связывается через `name: type = value` (RFC-007/010) как тип или функция
3. **Двухчастная типизация**: непрозрачные дескрипторы (распределение у внешней стороны) / прозрачные типы (распределение у YaoXiang), третьего не дано
4. **Изоляция маршалинга**: данные через границу по умолчанию копируются во временную зону вызова, объекты кучи YaoXiang изолированы от внешнего кода
5. **Безопасность владения**: единоличное владение дескриптором (Move) + RAII, структурно исключает двойное освобождение и use-after-free
6. **Аварийный выход**: `*T` сырой указатель + `unsafe {}`, пользователь явно принимает риск прямого доступа к памяти без копирования

**Основные границы — пять неизменяемых контрактов**:

```
1. Библиотека компонуется на этапе компиляции, символ верифицируется на этапе компиляции
2. Принадлежность распределения типа определяется при определении: непрозрачные дескрипторы у внешней стороны, прозрачные типы у YaoXiang
3. Маршалинг по умолчанию через копирование во временную зону, объекты кучи YaoXiang изолированы от внешнего кода
4. Единоличное владение дескриптором + Move, структурная защита от двойного освобождения/висячих ссылок
5. Внешний код всегда читает/записывает память с "чётким распределением, чётким владением", никаких неопределённостей
```

## Мотивация

### Текущее состояние и цели

Текущая реализация `native("symbol")` в кодовой базе — это лишь механизм диспетчеризации вызовов функций Rust std из байткода YaoXiang (`FfiRegistry` = `HashMap<String, RustFnPtr>`), **без реального пересечения границ ABI** — нет dlopen, нет C ABI маршалинга, нет владения памятью между границами.

Настоящий FFI должен решать четыре проблемы:

| Проблема | Ответ данного RFC |
|----------|-------------------|
| **Разрешение символов** | Библиотека как значение первого класса с компиляционной компоновкой (`Native.c("lib")`), символы верифицируются на этапе компиляции |
| **Маршалинг значений** | Управляется сигнатурой, компилятор определяет правила преобразования для каждой позиции параметра |
| **Владение памятью** | Двухчастная типизация определяет принадлежность; по умолчанию копирование и изоляция |
| **Безопасность времени жизни** | Move + RAII + заимствование с ограничением на один вызов |

RFC-020 и RFC-021 соответственно определяли различные аспекты FFI, они перекрывались, данный документ объединяет их в единую спецификацию.

### Цели проектирования

1. **Ноль сырых указателей в пользовательском коде**: при обычном использовании FFI, в исходном коде `.yx` не появляются сырые указатели
2. **Явная принадлежность распределения**: пользователь определяет, кому принадлежит эта память, при определении типа, без выводов во время выполнения
3. **Структурная безопасность**: отсутствие утечек, двойного освобождения, use-after-free гарантируется системой типов, а не договорённостями
4. **Честные границы доверия**: C не может предоставить компиляционно верифицируемые контракты типов, доверие локализовано в объявлениях привязок
5. **Совместимость с самозагрузкой**: без чрезмерных абстракций, уникальных для языка-хозяина

### За пределами рассмотрения

- **Расширяемая система плагинов для множественных ABI механизмов** (Wasm/Python/настраиваемый ABI): см. RFC-026a
- **Инструментарий yx-bindgen**: см. RFC-026b
- **Экспорт функций YaoXiang для вызова из C (обратный FFI)**: последующий RFC, данный документ только объявляет принципы
- **Встроенный ассемблер, SIMD intrinsics**: не входит в данный RFC

## Предложение

### 1. Внешние библиотеки и символы: каррированные значения первого класса

Информационный пробел FFI — "какую библиотеку скомпоновать, какой символ" — заполняется тем, что библиотека становится значением первого класса, без введения новых ключевых слов.

#### 1.1 Библиотека как значение

```yaoxiang
// Native.c применяет имя библиотеки → компиляционная компоновка, возвращает резолвер символов
sqlite3 = Native.c("libsqlite3")
```

`Native.c("libsqlite3")` — это **компиляционное действие + значение времени выполнения**:

- **Компиляция**: компоновщик `-lsqlite3`, библиотека входит в таблицу символов, существование символа верифицируемо
- **Значение**: `sqlite3` — это резолвер, применение имени символа даёт внешнюю ссылку на эту библиотеку

`.c` — это метка ABI механизма (C ABI). Ядро содержит только встроенный `.c`; другие механизмы (`.wasm` и др.) — см. RFC-026a.

#### 1.2 Символ как значение, связывание как `name: type = value`

Применение резолвера к имени символа даёт внешнюю ссылку, которая связывается через унифицированный синтаксис RFC-007/010. **Типовая аннотация LHS определяет, является ли эта ссылка типом или функцией**:

```yaoxiang
sqlite3 = Native.c("libsqlite3")

// LHS — Type → связывается как непрозрачный тип
SqliteDb: Type = sqlite3("sqlite3")

// LHS — сигнатура функции → связывается как функция
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")
SqliteDb.exec: (sql: String) -> Int32 = sqlite3("sqlite3_exec")
SqliteDb.close: () -> Int32 = sqlite3("sqlite3_close")

// .drop — обычное связывание метода (соглашение RAII из RFC-009)
SqliteDb.drop = SqliteDb.close
```

Компиляционная верификация: `sqlite3("sqlite3_open")` должен содержать `sqlite3_open` в таблице символов `libsqlite3`, иначе ошибка компиляции.

#### 1.3 Связывание методов и позиция self

В записи `Type.method: (...) -> ...`, `self` неявно на первой позиции — при вызове `db.exec("SELECT")`, `db` передаётся как 0-й параметр C функции `sqlite3_exec`.

Если нужно привязать уже объявленную свободную функцию как метод, используется синтаксис `[N]` для указания позиции self (RFC-004 — каррирование с несколькими позициями):

```yaoxiang
// Свободная функция
sqlite3_close_v2: (db: SqliteDb) -> Int32 = sqlite3("sqlite3_close_v2")

// Привязка как метод, [0] означает, что db — это self
SqliteDb.soft_close = sqlite3_close_v2[0]
```

Прямая привязка метода через `Native.c(...)` и ручная привязка через `[N]` — оба варианта используют `name: type = value`, оба помещают значение функции справа от `=`, никаких двух механизмов.

#### 1.4 Использование пользователем: ноль unsafe, ноль сырых указателей

```yaoxiang
import sqlite3_bindings

db = SqliteDb.open("test.db")
db.exec("SELECT * FROM users")
// ← конец области видимости, RAII автоматически вызывает SqliteDb.drop → sqlite3_close(db)
```

### 2. Двухчастная типизация: принадлежность распределения определяется при определении

Когда внешние данные попадают в YaoXiang, задаётся один вопрос: **кто определяет распределение этой памяти?**

```
├─ Распределение — внешний чёрный ящик (sqlite3, FILE*, socket fd)
│   → Непрозрачный дескриптор  =  lib("symbol")
│   → YaoXiang только хранит указатель, никогда не разыменовывает, передаёт только между функциями библиотеки
│   → Внешний код читает свою память, YaoXiang не трогает
│
└─ Распределение определено YaoXiang (timespec, point, структура с читаемыми полями)
    → Прозрачный тип  =  { field: Type, ... }
    → YaoXiang владеет памятью, определяет распределение, читает/записывает поля
    → Внешний код записывает/читает в/из память с распределением, определённым YaoXiang
```

**Третьего не дано.** Трёхуровневая модель памяти ("копирование/接管/системный уровень") из предыдущих проектов — это паттерн заплаток. Истина — дихотомия принадлежности распределения.

#### 2.1 Непрозрачные дескрипторы: распределение у внешней стороны

```yaoxiang
SqliteDb: Type = sqlite3("sqlite3")
```

- YaoXiang внутри хранит только дескриптор размером с указатель
- Пользователь не может конструировать (`SqliteDb {}` → ошибка компиляции), не может обращаться к полям (полей нет для обращения)
- Единственный источник: внешняя функция, возвращающая `SqliteDb`
- При вызове методов дескриптор передаётся обратно библиотеке, библиотека читает **свою** память (структура `sqlite3` в куче библиотеки)

Внешний код "читает своё" — читает структуру, которую выделил сам. YaoXiang только перемещает дескриптор. Нет конфликтов памяти.

#### 2.2 Прозрачные типы: распределение у YaoXiang

```yaoxiang
// Поля имеют значение, нужны чтение/запись → прозрачный тип, распределение объявляет YaoXiang
Timespec: Type = {
    tv_sec: Int64,
    tv_nsec: Int64
}
clock_gettime: (clk: Int32, ts: *Timespec) -> Int32 = Native.c("librt")("clock_gettime")

ts = clock_gettime(CLOCK_REALTIME)   // см. §3, маршалинг через временную зону
print(ts.tv_sec)                      // YaoXiang читает по своему определению полей
```

Внешний код читает/записывает в память **с распределением, определённым YaoXiang, которой YaoXiang владеет**. Распределение — это контракт YaoXiang, не внешней стороны.

#### 2.3 Правило принятия решения

Пользователю нужно решить только одно: **нужно ли мне читать поля этого типа?**

| Решение | Тип | Принадлежность распределения |
|---------|-----|----------------------------|
| Не читать поля, только передавать дескриптор между функциями библиотеки | Непрозрачный дескриптор `= lib("sym")` | Внешняя сторона |
| Нужно читать/записывать поля | Прозрачный тип `{ ... }` | YaoXiang |

### 3. Маршалинг: управляется сигнатурой, временная зона изоляции

Преобразование данных через границу управляется **сигнатурой**, компилятор определяет правила преобразования для каждой позиции параметра. **Основная гарантия безопасности: внешний код читает/записывает временную зону маршалинга, не объекты кучи YaoXiang.**

#### 3.1 По умолчанию через копирование во временную зону

```
YaoXiang → C (входные параметры):
    Копирование данных во временную зону вызова → передача указателя на временную зону в C
    → C вышел за границы/испортил данные — только временная зона пострадала, объекты кучи YaoXiang изолированы

C → YaoXiang (возвращаемые/выходные параметры):
    C записывает во временную зону → YaoXiang memcpy обратно в свой объект
    → C не касается финального объекта YaoXiang
```

**Внешний код всегда читает/записывает временную зону маршалинга, полностью изолированную от объектов кучи YaoXiang.** Неправильное объявление распределения, C сохранил указатель который станет висячим, C вышел за границы — всё это затрагивает только временную зону, объекты YaoXiang целы. Цена — одно memcpy.

#### 3.2 Таблица правил маршалинга

**Направление входных параметров (YaoXiang → C)**:

| Тип YaoXiang | C представление | Действие маршалинга | Владение |
|--------------|-----------------|-------------------|----------|
| `Int32/Int64/Float` | `int/long/double` | Непосредственно в регистр, нулевое преобразование | Семантика значения |
| `String` | `const char*` | Предоставление временного read-only представления (временное, действительно во время вызова) | YaoXiang сохраняет, C только для чтения |
| Прозрачный тип | `struct T*` | Копирование во временную зону, передача указателя на временную зону | YaoXiang владеет объектом, C читает копию |
| Непрозрачный дескриптор | `void*` | Извлечение внутреннего указателя-дескриптора | YaoXiang владеет, предоставляет C |
| `*T` | `T*` | Непосредственная передача сырого указателя (unsafe) | Пользователь несёт ответственность |

**Направление возврата (C → YaoXiang)**:

| Возврат C | Тип YaoXiang | Действие маршалинга | Владение |
|-----------|--------------|--------------------|----------|
| `int/double` | `Int32/Float` | Непосредственное чтение из регистра | Семантика значения |
| `char*` | `String` | strlen + memcpy в String YaoXiang | YaoXiang владеет копией, исходная память не трогается |
| `struct T*` (новый дескриптор) | Непрозрачный дескриптор | Дескриптор сохраняется в объекте YaoXiang | YaoXiang принимает владение |
| `struct T` (значение/выходной параметр) | Прозрачный тип | C записывает во временную зону → memcpy в YaoXiang | YaoXiang владеет |
| `char*` (статическая область) | `*const U8` | Сохранение сырого указателя, без копирования (unsafe чтение) | Не принимается, пользователь несёт ответственность |

#### 3.3 Время жизни заимствования: строго ограничено одним вызовом

Указатели, которые YaoXiang предоставляет внешнему коду (временное read-only представление String, временная зона прозрачного типа, дескриптор), **время жизни строго ограничено одним вызовом**:

- Во время вызова: указатель действителен, внешний код может читать/записывать
- После возврата из вызова: заимствование немедленно прекращается

Если внешний код сохранил указатель для использования после вызова — это внешний код нарушает стандартный FFI контракт (эквивалент бага библиотеки), YaoXiang за это не отвечает. Это согласуется с FFI контрактами C во всех языках (`&T` Rust, переданный в C, имеет те же ограничения).

#### 3.4 String никогда не отдаёт персистентный указатель

`String` — ключ к "C не вмешивается в память YaoXiang":

- В C: предоставляется **временное read-only представление**, действительное во время вызова
- Из C: strlen + memcpy в **копию**, которой владеет YaoXiang

C никогда не получает персистентный указатель на String YaoXiang, YaoXiang никогда не хранит долгосрочную ссылку на C `char*`. Структурная изоляция.

### 4. Владение и время жизни: Move + RAII

Непрозрачные дескрипторы следуют модели владения из RFC-009, без новых концепций.

#### 4.1 Основные принципы

- **Семантика Move**: непрозрачные дескрипторы по умолчанию Move, присваивание/передача параметров/возврат = передача владения, не копируются
- **Единоличное владение дескриптором**: в любой момент у дескриптора только один владелец → структурное исключение двойного освобождения
- **RAII освобождение**: при выходе из области видимости, если привязан `.drop`, автоматически вызывается
- **Отслеживание потребления**: после явного уничтожения или Move переменная потреблена, не может использоваться снова → исключение use-after-free

#### 4.2 `.drop` — это опциональный побочный эффект для внешней стороны

```yaoxiang
SqliteDb.drop = SqliteDb.close     // при выходе из области видимости вызывается sqlite3_close
```

**`.drop` — это не механизм предотвращения утечки в YaoXiang** — хранение дескриптора на стороне YaoXiang (значение размером с указатель) освобождается автоматически, не связано с `.drop`. `.drop` — это **опциональный побочный эффект: при выходе из области видимости дополнительно вызывается внешняя функция**:

- Привязан `.drop` → при выходе из области видимости вызывается (очистка внешних ресурсов)
- Не привязан `.drop` → ничего не делается, **без ошибок, без предупреждений**

Нужно ли очищать внешние ресурсы — это вопрос спецификации внешней библиотеки (`getenv` возвращает статическую область, нельзя освобождать; глобальный синглтон, нельзя освобождать), YaoXiang не берёт на себя полномочия强制ить это. Защита от утечки через Move + единоличное владение (безусловно, структурно), а не через `.drop`.

#### 4.3 Автоматическое уничтожение и порядок

```yaoxiang
{
    db = SqliteDb.open("test.db")
    stmt = db.prepare("SELECT * FROM users")
    // ← конец области видимости, автоматическое уничтожение в обратном порядке (вызывается только если есть .drop):
    //   stmt.drop()  → sqlite3_finalize(stmt)
    //   db.drop()    → sqlite3_close(db)
}
```

Порядок уничтожения: обратный порядку определения, согласуется с RAII.

#### 4.4 Move и потребление

```yaoxiang
db = SqliteDb.open("test.db")
db2 = db                // Move: передача владения
db.exec("...")          // ❌ Ошибка компиляции: db уже перемещён, после потребления нечитаемо

process_db: (db: SqliteDb) -> Void = {
    db.exec("...")
    // ← конец функции, db уничтожается здесь
}
process_db(some_db)     // Move в функцию
// some_db здесь уже недействителен
```

#### 4.5 Обработка Null

```yaoxiang
// Может вернуть null → ?T, пользователь должен обработать
SqliteDb.open: (file: String) -> ?SqliteDb = sqlite3("sqlite3_open")

db = SqliteDb.open("test.db")
match db {
    Some(db) => db.exec("SELECT 1"),
    None => print("Ошибка открытия")
}

// Соглашение не возвращать null → без маркировки, при null происходит panic
```

C возвращает null либо пользователь обрабатывает (`?T`), либо panic раскрывает проблему. Третьего "тихо игнорировать" нет.

#### 4.6 Обработка ошибок уничтожения

Возвращаемое значение привязанной к `.drop` функции определяет поведение:

| Тип возврата `.drop` | Поведение |
|---------------------|-----------|
| `Void` | Без ошибок |
| `Int32` (код ошибки) | Ненулевое → panic — ошибка уничтожения означает аномальное состояние, раскрытие лучше молчания |
| `?Error` | Не None → panic — аналогично |

Ошибка уничтожения не может быть проигнорирована. Если нужно игнорировать конкретные ошибки, явно обработайте их в обёрточной функции, привязанной к `.drop`.

### 5. Поведение FFI в spawn блоках

Определение типа ресурса определяется привязкой `.drop` (RFC-024), без дополнительных маркировок:

| Определение | Поведение |
|------------|-----------|
| Непрозрачный дескриптор с привязкой `.drop` | Тип ресурса — операции с одним экземпляром в spawn блоке автоматически сериализуются |
| Непрозрачный дескриптор без привязки `.drop` | Не тип ресурса — возможна параллелизация (дескриптор данных, без побочного эффекта освобождения) |
| Прозрачный тип / тип значения | Не тип ресурса — возможна параллелизация |

```yaoxiang
SqliteDb.drop = SqliteDb.close   // → тип ресурса

(a, b) = spawn {
    r1 = db.exec("SELECT ..."),   // тот же экземпляр, автоматическая сериализация
    r2 = db.exec("INSERT ...")    // ожидание r1
}

(x, y) = spawn {
    db1 = SqliteDb.open("a.db"),   // разные экземпляры, возможна параллелизация
    db2 = SqliteDb.open("b.db")
}
```

Типы с `.drop` в spawn автоматически сериализуют операции с одним экземпляром, гарантируя уничтожение без конкуренции.

### 6. Аварийный выход: сырые указатели + unsafe

Маршалинг по умолчанию через копирование во временную зону безопасен, но имеет накладные расходы memcpy. Для чувствительных к производительности сценариев (большие структуры, высокочастотные вызовы), когда нужно копирование ноль, пользователь явно использует аварийный выход с сырыми указателями:

```yaoxiang
// C напрямую читает память YaoXiang, ноль копирований — пользователь явно принимает риск
ptr: *const U8 = Native.c("libc")("getenv")("HOME")
unsafe {
    value = read_c_string(ptr)   // пользователь гарантирует действительность ptr
}
```

**`unsafe` используется только для операций с сырыми указателями, полностью ортогонален непрозрачным дескрипторам и прозрачным типам.** Обычный FFI (дескрипторы + прозрачные типы) не требует unsafe. Написание `unsafe {}` = пользователь явно подписывается под риском прямого доступа к памяти.

**Граница доверия**: C не может предоставить компиляционно верифицируемые контракты типов (`.h` — не ABI контракт, таблица символов содержит только имена без сигнатур). Поэтому корректность C сигнатур не может быть автоматически верифицирована — это гарантируется автором привязки при написании `Native.c(...)` + сигнатуры. **Доверие локализовано в объявлении привязки**: автор привязки гарантирует, пользователь пакета получает безопасный API. Это согласуется с Rust `extern "C"` (написание extern — это акт доверия, безопасная обёртка скрывает это от вызывающего).

## Компромиссы

### Преимущества

1. **Полная информация**: библиотека компонуется на этапе компиляции, символы верифицируются на этапе компиляции, никаких "библиотека не найдена" во время выполнения
2. **Явная принадлежность распределения**: двухчастная типизация, определяется при определении, без выводов во время выполнения
3. **Структурная безопасность**: изоляция временной зоны + Move + RAII, внешний код не касается объектов кучи YaoXiang
4. **Ноль новых ключевых слов**: каррирование `Native.c` + `name: type = value`, всё переиспользует существующий синтаксис
5. **Честные границы**: не притворяемся, что можем верифицировать C сигнатуры, доверие локализовано в объявлениях

### Недостатки

1. **Накладные расходы memcpy**: маршалинг по умолчанию с копированием, для больших структур высокочастотных вызовов нужно явно использовать аварийный выход
2. **Ручные гарантии распределения**: соответствие распределения прозрачного типа и C struct обеспечивается автором привязки/yx-bindgen
3. **C сигнатуры не верифицируются на этапе компиляции**: фундаментальное ограничение FFI, YaoXiang не может устранить его из C

- [ ] Реализация изоляции через копирование во временную зону (копирование входных параметров, возврат через memcpy)

- [ ] Реализация ограничения времени жизни заимствования одним вызовом

## Журнал решений по проектированию

| Решение | Определение | Причина | Дата |
|---------|-------------|---------|------|
| **Библиотека как значение** | `Native.c("lib")` каррирование возвращает резолвер | Информация о библиотеке становится значением первого класса, видимым на этапе компиляции, заполняет пробел "какую библиотеку скомпоновать", ноль новых ключевых слов | 2026-07-03 |
| **Компиляционная компоновка** | `Native.c("lib")` триггерит `-llib` | Таблица символов читаема на этапе компиляции, существование символов верифицируемо, типы реальны | 2026-07-03 |
| **Двухчастная типизация** | Непрозрачные дескрипторы / прозрачные типы | Принадлежность распределения дихотомична, покрывает все случаи; удалён патч "трёхуровневой модели памяти" | 2026-07-03 |
| **Изоляция временной зоны маршалинга** | По умолчанию копирование, объекты кучи изолированы от внешней стороны | Выход за границы/висячие указатели внешней стороны затрагивают только временную зону, объекты YaoXiang целы; ноль копирования требует явно использовать аварийный выход | 2026-07-03 |
| **`.drop` опционально** | При отсутствии ничего не делается, без ошибок | Хранение дескриптора YaoXiang освобождается автоматически; очистка внешних ресурсов — вопрос спецификации внешней библиотеки, не берём полномочий强制ять | 2026-07-03 |
| **Механизм защиты от утечки** | Move + единоличное владение дескриптором (безусловно) | Структурная гарантия, не связана с `.drop` | 2026-07-03 |
| **Граница доверия** | В объявлении `Native.c(...)` | C сигнатуры не верифицируются на этапе компиляции, доверие локализовано, unsafe только для сырых указателей | 2026-07-03 |
| **Обработка Null** | `?T` или panic | Проблемы C не скрываются, нет варианта "тихо игнорировать" | 2026-07-03 |
| **Ошибка уничтожения** | Тип возврата `.drop` определяет,统一ный panic | Ошибка уничтожения не может быть проигнорирована | 2026-07-03 |
