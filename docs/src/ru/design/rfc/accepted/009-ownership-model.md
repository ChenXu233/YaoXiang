# RFC-009: Дизайн модели владения

## Резюме

Данный документ определяет **модель владения (Ownership Model)** языка программирования YaoXiang.

**Основная идея — пять концепций, один градиент**:

```
Взглянуть/Изменить     Забрать         Разделить         Скопировать     Системный
  на месте                              владение          копию           уровень
    │                │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         Бескопир.      Авто-        Явное          *T
  Токен          по умолч.      выбор        глубокое       Пользователь
  нулевого                       Rc/Arc       копирование    несёт
  размера,                                                    ответств.
  свойства
  типа естеств.
  выводят
  права
```

- **Move (по умолчанию)**: присваивание / передача параметра / возврат = передача права владения, бескопировочно, автоматическое освобождение через RAII
- **`&T` / `&mut T` (токены заимствования)**: токены нулевого размера на этапе компиляции. `&T` копируем (разделяемое неизменяемое), `&mut T` линеен (исключительное изменяемое). Права естественно выводятся из свойств типа, никаких специальных правил. Можно возвращать, хранить в структурах, захватывать замыканиями.
- **Ключевое слово `ref`**: разделение владения через границы областей видимости. Компилятор автоматически выбирает Rc (без пересечения задач) или Arc (с пересечением задач)
- **`clone()`**: явное глубокое копирование
- **`unsafe` + `*T`**: сырой указатель, системный аварийный люк

**Устранённая сложность**:
- ❌ Отсутствуют времена жизни `'a`
- ❌ Отсутствует отдельный фреймворк проверки заимствований (конфликты заимствования сводятся к утверждениям Хоара и используют общий конвейер доказательства с проверкой типов)
- ❌ Отсутствует GC
- ❌ Отсутствуют специальные правила вроде «запрещено к экранированию» (токены — обычные типы, область видимости обрабатывается системой типов единообразно)
- ❌ Пользователю не нужно знать разницу между Rc/Arc (компилятор выбирает автоматически)

> **Бремя программирования**: `&T` копируем, `&mut T` не копируем — два свойства типа, ноль специальных правил, всё автоматизировано компилятором.
> **Гарантии производительности**: Move — нулевые накладные расходы, токены — нулевые накладные расходы (типы нулевого размера, исчезают после компиляции), ref оплачивается по необходимости, паузы GC отсутствуют.

## Мотивация

### Зачем нужна модель владения?

| Язык | Управление памятью | Проблема |
|------|--------------------|----------|
| C/C++ | Ручное управление | Утечки памяти, висячие указатели, двойное освобождение |
| Java/Python | GC | Колебания задержек, расход памяти, непредсказуемые паузы |
| Rust | Владение + проверка заимствований | Крутая кривая обучения для времён жизни `'a` |
| **YaoXiang** | **Move + Token + ref** | **Простота, детерминированность, отсутствие GC** |

### Цели дизайна

```yaoxiang
# 1. По умолчанию Move (бескопировочно)
p = Point(1.0, 2.0)
p2 = p                         # Move, p больше нельзя читать

# 2. &T / &mut T — токены заимствования (нулевые накладные расходы, свойства типа естественно выводят права)
print_info(p2)                 # Компилятор автоматически создаёт токен &Point, освобождается после использования
shift(p2, 1.0, 1.0)           # Компилятор автоматически создаёт токен &mut Point

# 3. ref = разделение (компилятор автоматически выбирает Rc/Arc)
shared = ref p2                # Удержание через границы областей видимости
spawn { use(shared) }          # Компилятор: пересечение задач → Arc

# 4. clone() = явное копирование
backup = p2.clone()            # Глубокая копия, независимая

# 5. unsafe + *T = системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Ключевые отличия от Rust

| Возможность | Rust | YaoXiang |
|-------------|------|----------|
| Семантика по умолчанию | Заимствование `&T` (требуется явный `.clone()`) | **Move (передача по значению, бескопировочно)** |
| Заимствование | `&T`/`&mut T`, можно возвращать, требует времён жизни | **`&T`/`&mut T` — токены нулевого размера, свойства типа Dup/Linear естественно выводят права** |
| Механизм разделения | `Arc::new()` + ручной Weak | **`ref` (компилятор автоматически выбирает Rc/Arc)** |
| Копирование | `clone()` | `clone()` |
| Сырые указатели | `*T` | `*T` |
| Времена жизни | `'a` | ❌ Отсутствуют |
| Проверка заимствований | Глобальный вывод | **Генератор проверки типов автоматически создаёт утверждения заимствования, общий конвейер верификации** |
| Циклические ссылки | Ручной Weak | **Единое освобождение при завершении задачи / lint для межзадачных циклов / Weak в стандартной библиотеке** |

## Предложение

### 1. Move (передача владения по умолчанию)

```yaoxiang
# Правило: присваивание / передача параметра / возврат = Move, бескопировочно

p: Point = Point(1.0, 2.0)
p2 = p                           # Move, p больше нельзя читать

# Переменная может быть переприсвоена (стиль Python, без затенения)
p = Point(3.0, 4.0)              # p перепривязывается, тип должен совпадать

# Параметр функции: Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Возврат через Move
}

# Возврат из функции: Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Возврат через Move, бескопировочно
}
```

**Особенности**:
- Бескопировочно (компилятор перемещает указатель)
- После перемещения исходная привязка недоступна для чтения (ошибка компиляции)
- RAII: автоматическое освобождение при выходе из области видимости
- Сигнатура функции `(T) -> T` сама по себе является документацией — потребляет T, возвращает T

### 2. &T / &mut T (токены заимствования)

**Ключевой принцип: `&T` и `&mut T` — это токены нулевого размера на этапе компиляции. Это не «ссылки», а «доказательство права доступа на уровне типов».**

#### 2.1 Два свойства типа

```
&T      →  Нулевой размер, замораживает исходные данные (пока живёт ReadToken, WriteToken запрещён),
          многократные неизменяемые представления безопасны при заморозке → копируем (Dup)
&mut T  →  Нулевой размер, исключительное чтение-запись (пока живёт WriteToken, любые другие токены запрещены),
          копирование бессмысленно при исключительном доступе → линеен (не Dup)
```

**Причинно-следственная связь не обратима: заморозка — причина, Dup — следствие.** Не потому что `&T` реализует Dup и потому может сосуществовать — а потому что данные заморожены (мутация невозможна), множественные неизменяемые представления безопасны, и Dup становится возможным. Если принять Dup за определение, а проверку конфликтов за «дополнительную заплатку», дизайн ошибочен.

#### 2.2 Базовое использование

```yaoxiang
# Сторона метода: объявляет тип параметра, определяя необходимые права
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # Токен &Point даёт право на чтение
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # Токен &mut Point даёт право на запись
    self.y = self.y + dy
}

# Сторона вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                          # Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)                  # Компилятор автоматически создаёт токен &mut Point
p.print()                          # OK, предыдущий токен уже освобождён после завершения shift

# Свободные функции — аналогично
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # Два токена &Point сосуществуют — тип Dup
}
d = distance(p, p2)
```

#### 2.3 Почему не нужно правило «запрещено к экранированию»

RFC-009 v8 накладывал на `&T`/`&mut T` три специальных правила — можно использовать только как параметр, нельзя возвращать, нельзя хранить в структурах. Это были заплатки к концепции «заимствования».

Системе токенов эти правила не нужны. Токены — это **обычные типы**, подчиняющиеся тем же правилам областей видимости, что и все остальные типы.

**Возврат ссылки — естественно поддерживается**:

```yaoxiang
# ✅ Токен распространяется вместе с возвращаемым значением
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # Дочерний и родительский токен возвращаются вместе
}

# Использование
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # Токен возвращён вызывающему коду
print(px_ref)               # OK, токен всё ещё в области видимости
```

**Хранение в структурах — естественно поддерживается**:

```yaoxiang
# ✅ Структура несёт токен как поле
Window: Type = {
    target: Point,
    view: &Point,      # Поле-токен — содержит неизменяемое представление target
}

# Токен view производится от target, Window владеет обоими
# Пока существует Window, токен view валиден
```

**Захват замыканием — естественно поддерживается**:

```yaoxiang
# ✅ Замыкание захватывает токен, как и любое другое значение
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    # Замыкание захватывает токен &Float от threshold (тип Dup, свободно копируется в замыкание)
    items.filter(|p| p.x > threshold)
}

# Это было невозможно в RFC-009 v8 — v8 запрещал замыканиям захватывать заимствования
```

**Между задачами — токен не пересекает границу потока**:

```yaoxiang
# ❌ Токен не может пересекать границу задачи
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ Ошибка компиляции: токен не может передаваться между задачами
}

# Это не специальное правило — токен является доказательством права на этапе компиляции;
# для разделения между задачами используйте ref
```

**Токен не может быть ref**:

```yaoxiang
# ❌ Токен — доказательство права, а не владение
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ Ошибка компиляции: &T не является владеющим типом
}
```

#### 2.4 Время жизни токена

Время жизни токена определяется **обычными правилами областей видимости**, без необходимости параметров времени жизни:

- Токен в параметре функции: живёт в течение вызова, освобождается после его завершения
- Возвращаемый токен: право владения передаётся вызывающему коду
- Токен, хранимый в структуре: живёт вместе со структурой
- Токен, захваченный замыканием: живёт вместе с замыканием

Компилятору не нужны аннотации `'a`, потому что токены — это **значения**, чьё время жизни управляется системой владения (Move/RAII) единообразно. **Сведение проблемы заимствования к проблеме владения.**

#### 2.5 Обнаружение конфликтов токенов

Обнаружение конфликтов токенов — это **утверждение логики Хоара**, а не отдельный потокочувствительный анализ.

```
{Все конфликтующие ReadToken мертвы} write(data) {WriteToken безопасно получен}
```

Совместно с проверкой типов и верификацией пользовательских предикатов использует конвейер доказательств из RFC-027. Компилятор автоматически генерирует утверждения заимствования (`borrow_conflict`, `use_after_move`, `use_after_drop`, `mut_violation`) и отправляет их в конвейер для верификации. Конвейер возвращает Proved / Disproved / Unproven.

```yaoxiang
# ❌ &mut — токен линеен, не может быть скопирован
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move, p больше нельзя читать
    p.x = 10.0                      # ❌ Ошибка компиляции: WriteToken уже перемещён
}

# ✅ &T — токен типа Dup, может свободно копироваться
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK, &T — тип Dup
    print(p.x)                      # OK
    print(p2.x)                     # OK, два неизменяемых токена сосуществуют
}
```

**Проверка заимствований не исчезла — она понижена в размерности.** Существующий `BorrowChecker` становится `BorrowPredicateEmitter` (генератором утверждений), генерируемые утверждения заимствования используют общий конвейер с другими утверждениями о типах. Это полностью параллельно концепции проверки типов: проверка типов генерирует утверждения о равенстве типов, генератор утверждений заимствования генерирует утверждения заимствования, верификация выполняется единым конвейером. Подробности см. в [RFC-009a](../accepted/009a-borrow-proof-pipeline.md).

#### 2.7 Внутри компилятора: механизм брендов

Пользователь никогда не взаимодействует с брендами. Компилятор внутри присваивает каждому токену уникальный идентификатор на этапе компиляции:

```
Видимое пользователю     Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — уникальное целое на этапе компиляции
&mut Point     →  WriteToken(Point, #M)   // #M — уникальное целое на этапе компиляции
```

Назначение брендов:
- **Защита от подделки**: токен может быть получен только из капсулы владельца, не может быть сконструирован произвольно
- **Отслеживание связей**: при получении `&Float` (доступ к полю) из `&Point` `&Float` несёт производный бренд (`#N.field_x`), и компилятор может отследить его до родительского токена
- **Обнаружение конфликтов**: одноимённые `WriteToken` и производный `ReadToken` не могут быть одновременно активны

Бренды полностью исчезают после мономорфизации и инлайнинга, отсутствуют в генерируемом машинном коде. **Нулевые накладные расходы во время выполнения.**

#### 2.8 Правила автоматического выбора заимствования

На стороне вызова компилятор автоматически выбирает по следующему приоритету:

```
1. Если аргумент далее используется → приоритетно создаётся токен (&T или &mut T, согласно сигнатуре метода)
2. Если аргумент далее не используется → Move
3. Порядок предпочтения: &T < &mut T < Move
```

```yaoxiang
# Пример: автоматический выбор
p = Point(1.0, 2.0)
p.print()        # print объявляет &self → компилятор создаёт токен &Point
p.shift(1.0, 1.0) # shift объявляет &mut self → компилятор создаёт токен &mut Point
p2 = p           # Move, p больше не используется
```

#### 2.9 Сравнение с упрощённым заимствованием RFC-009 v8

| Возможность | Упрощённое заимствование (v8) | Токены заимствования (v9) |
|-------------|-------------------------------|---------------------------|
| Возврат ссылки | ❌ Жёстко запрещено | ✅ Токен распространяется с возвращаемым значением |
| Хранение в структуре | ❌ Жёстко запрещено | ✅ Токен как поле структуры |
| Захват замыканием | ❌ Жёстко запрещено | ✅ Замыкание захватывает значение токена |
| Специальные правила | 3 (только параметр / нельзя возвращать / нельзя хранить) | 0 — свойства типа естественно выводят права |
| Проверка заимствований | Специализированная перекрёстная проверка | Потокочувствительный анализ живучести в проверщике типов |
| Аннотации времён жизни | Не нужны | Не нужны |
| Накладные расходы во время выполнения | Ноль | Ноль (типы нулевого размера, исчезают после компиляции) |
| Сообщения об ошибках | «Заимствование не может экранироваться» | «WriteToken(#3) уже перемещён» (обычная ошибка типа) |
| Ментальная модель пользователя | Понять особый статус «заимствования» | `&T` копируем, `&mut T` не копируем |

### 3. Ключевое слово ref (автоматическая оптимизация компилятора)

`ref` — единственный способ разделения владения через границы областей видимости. В основе лежит Rc или Arc, о выборе пользователю заботиться не нужно.

#### 3.1 Базовое использование

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # Разделение, компилятор автоматически выбирает реализацию

# Разделение между задачами
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # Компилятор: пересечение задач → Arc
    spawn { use(data) }           # Компилятор: пересечение задач → Arc
}

# Разделение в пределах задачи
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # Компилятор: без пересечения задач → Rc
}
```

**Ментальная модель пользователя**: `ref` = разделяемое удержание. Достаточно.

#### 3.2 Анализ экранирования компилятора: Rc vs Arc

```
Анализ потоков данных для ref:

Не экранируется в другие задачи → Rc (неатомарный подсчёт ссылок, низкие накладные расходы)
Экранируется в другие задачи   → Arc (атомарный подсчёт ссылок, потокобезопасно)
```

#### 3.3 Стратегия обнаружения циклов

```
Цикл в пределах задачи → допускается без уведомлений.
  ├── Каждая задача имеет явную границу времени жизни — при завершении задачи все ресурсы (включая циклы ref) освобождаются единообразно.
  ├── Долгоживущие сервисы должны создавать дочерние задачи по запросам/соединениям — дочерние задачи освобождаются автоматически, накопления утечек не происходит.
  ├── ref всегда остаётся живым, семантика не размывается.
  └── Пользователь имеет право создавать двусторонние сильные ссылки внутри задачи (например, промежуточные состояния графовых вычислений).

Межзадачный цикл → lint (по умолчанию warn, настраивается).
  ├── Поведение программы корректно, реальной утечки нет (при завершении родительской задачи все ресурсы дочерних задач освобождаются).
  ├── Но межзадачные сильные ссылки означают размытие границ владения — повод остановиться и пересмотреть.
  ├── Уровень warn по умолчанию, компиляция проходит с предупреждением.
  └── Команда может установить deny в конфигурации проекта как часть CI-гейта качества.
```

**Уровни lint** (аналог Rust clippy):

| Уровень | Поведение | Сценарий |
|---------|-----------|----------|
| `allow` | Без проверки | Личный проект |
| `warn` (по умолчанию) | Компиляция проходит, есть предупреждения | Этап разработки |
| `deny` | Ошибка компиляции | CI-гейт качества команды |
| `forbid` | Ошибка компиляции, нельзя переопределить | Обязательные правила уровня организации |

```yaoxiang
# Цикл в пределах задачи: допускается без уведомлений, двусторонние сильные ссылки
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # Цикл. Единообразное освобождение при завершении задачи.
}

# Межзадачный цикл: lint (по умолчанию warn)
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: межзадачный цикл ссылок
    }
}
```

**Пример конфигурации проекта**:

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # Межзадачные циклы сразу отклоняются в CI
```

| Тип цикла | Поведение | Причина |
|-----------|-----------|---------|
| Цикл ref в пределах задачи | Без проверки | Право пользователя, единое освобождение при завершении задачи |
| Межзадачный цикл ref | lint (по умолчанию warn) | Повод пересмотреть, можно настроить deny |

#### 3.4 Weak: предоставляется стандартной библиотекой

```yaoxiang
use std.rc.Weak

# Продвинутый пользователь делает явный выбор
a.next = ref b
b.prev = Weak.new(a.next)        # Пользователь явно контролирует, какое направление слабое
```

**`Weak` не встроен в язык, это тип стандартной библиотеки.** Для повседневных задач достаточно `ref`. Продвинутые пользователи, которым нужен тонкий контроль памяти, подключают `Weak` вручную.

#### 3.5 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|---|----------------|-------|
| Назначение | Взглянуть / изменить на месте | Разделяемое удержание |
| Область действия | С областью видимости значения токена | Через границы областей видимости |
| Стоимость | Нулевые накладные расходы (тип нулевого размера) | Rc или Arc (выбирает компилятор) |
| Экранирование | Да (токен распространяется с возвращаемым значением / структурой / замыканием) | Для этого и предназначен |
| Между задачами | Нет (токен — доказательство права на этапе компиляции, не может передаваться между задачами) | Да (компилятор автоматически выбирает Arc) |
| Образование циклов | Неприменимо | В пределах задачи допускается, межзадачный — lint |

### 4. clone() — явное копирование

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # Глубокое копирование
# p и p2 независимы, не влияют друг на друга
```

**Когда использовать**: когда нужно сохранить исходное значение, и при этом Move или разделение не подходят.

### 5. unsafe + сырые указатели (системное программирование)

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # Сырой указатель
    (*ptr).x = 0.0                # Разыменование (безопасность гарантирует пользователь)
    ptr2 = ptr + 1                # Арифметика указателей
}
```

**Ограничения**:
- Используется только в блоках `unsafe`
- Пользователь гарантирует отсутствие висячих указателей и использования после освобождения
- Применяется для FFI, операций с памятью и другого системного программирования

## Сводный пример

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T: токен только для чтения
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T: токен для изменения
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move: потребление и возврат
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # Забрать, изменить, вернуть
    }

    # Возврат ссылки: токен распространяется с возвращаемым значением
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# Замыкание захватывает токен (недостижимо в v8)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}

# Комплексное использование
p = Point(1.0, 2.0)
p.print()                           # Токен &Point
p.shift(1.0, 1.0)                   # Токен &mut Point
p = p.scale(2.0)                    # Move → возврат
shared = ref p                      # ref — разделение
spawn { use(shared) }

# clone — независимая копия
backup = p.clone()

# Цикл в пределах задачи: допускается
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # Цикл, единое освобождение при завершении задачи

# unsafe — системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## Ограничения системы типов

### Свойство типа Dup

`Dup` (Duplicable) — автоматически управляемое компилятором свойство типа, означающее **поверхностное копирование**: при присваивании / передаче параметра копируется дескриптор / токен, нижележащие данные разделяются. Это образует трёхуровневый градиент с Move (передача владения) и Clone (явное глубокое копирование, создание независимого дубликата).

**Dup и Clone — ортогональные концепции** — Dup копирует дескриптор с разделением данных, Clone создаёт независимый дубликат. Тип может одновременно поддерживать Dup и Clone, либо только одно из них.

| Тип | Dup | Clone | Пояснение |
|-----|-----|-------|-----------|
| `&T` | ✅ (копирование токена, несколько представлений указывают на одни данные) | ✅ | Токен только для чтения |
| `ref T` | ✅ (счётчик ссылок +1, разделяет данные в куче) | ✅ | Разделяемое удержание (компилятор выбирает Rc/Arc) |
| String, Bytes | ✅ (внутренний подсчёт ссылок, копируется дескриптор, разделяется нижележащий буфер) | ✅ | Строка / байты |
| `&mut T` | ❌ (линеен, исключительный) | ❌ | Токен для изменения |
| `*T` | ❌ | ❌ | Сырой указатель |
| struct | Производное (автоматически, если все поля Dup) | ✅ | Структура |

**Примитивные типы значений** (Int, Float, Bool, Char) при присваивании ведут себя как встроенное в компилятор копирование значений — два значения полностью независимы, это не поверхностное копирование. Они не относятся к свойству Dup, а обрабатываются компилятором изначально.

## Анализ производительности

| Операция | Стоимость | Пояснение |
|----------|-----------|-----------|
| Move | Ноль | Перемещение указателя |
| `&T` / `&mut T` | Ноль | Тип нулевого размера, исчезает после компиляции, нулевые накладные расходы |
| `ref` (без пересечения задач) | Низкая | Компилируется в Rc, неатомарная операция |
| `ref` (с пересечением задач) | Средняя | Компилируется в Arc, атомарная операция |
| `clone()` | Зависит от типа | Маленькие объекты быстро, большие — медленно |
| `unsafe + *T` | Ноль | Прямые операции с памятью |

### Сравнение

| Язык | Механизм разделения | Управление памятью | Обработка циклов | Сложность |
|------|---------------------|--------------------|------------------|-----------|
| Rust | Arc / Mutex + проверка заимствований | Проверка на этапе компиляции | Ручной Weak | Высокая |
| Go | chan / pointer | GC | GC | Низкая |
| C++ | shared_ptr | RAII | weak_ptr | Средняя |
| **YaoXiang** | **ref + токены заимствования** | **RAII** | **Освобождение на границе задачи / lint для межзадачных / Weak в stdlib** | **Низкая** |

## Компромиссы

### Преимущества

1. **Единообразие**: `&T`/`&mut T` — обычные типы, а не специальная возможность языка. Полностью согласуется с `name: type = value` из RFC-010
2. **Простота**: нет времён жизни, проверка заимствований сводится к утверждениям системы типов. `&T` копируем, `&mut T` не копируем — два свойства типа
3. **Мощность**: можно возвращать ссылки, хранить в структурах, захватывать замыканиями — выразительность на уровне Rust
4. **Умный компилятор**: ref автоматически выбирает Rc/Arc, сторона вызова автоматически выбирает заимствование
5. **Детерминированность**: ref — это и есть удержание, не превращается молча в слабую ссылку
6. **Высокая производительность**: Move — бескопировочно, токены — нулевые накладные расходы (типы нулевого размера, исчезают после компиляции)
7. **Гибкость**: `unsafe + *T` поддерживает системное программирование

### Недостатки

1. **Заражение параметрами брендов в дженериках**: токены несут идентификаторы брендов, в сигнатурах функций, возвращающих ссылки, проявляются дополнительные параметры дженериков
2. **Накладные расходы ref во время выполнения**: атомарные операции имеют стоимость (но это неизбежная плата за разделение)
3. **Риск unsafe**: пользователь должен гарантировать корректность
4. **Межзадачный цикл — lint, а не ошибка компиляции**: в отличие от Rust, по умолчанию warn, для использования как гейта качества требуется настройка deny в команде

## Альтернативы

| Альтернатива | Почему не выбрана |
|--------------|-------------------|
| GC | Накладные расходы во время выполнения, непредсказуемые паузы |
| Проверщик заимствований Rust | Требует времён жизни `'a`, крутая кривая обучения |
| Чистый Move | Не справляется с конкурентным разделением |
| Отсутствие сырых указателей | Невозможно системное программирование |
| Предоставление Rc/Arc пользователю | Перекладывание деталей реализации на пользователя, рост когнитивной нагрузки |
| Упрощённое заимствование (v8) | Стратегия запрета экранирования жертвует ключевыми возможностями: захват замыканиями, возврат ссылок и т.д. |

## Записи о дизайнерских решениях

| Решение | Выбор | Причина | Дата |
|---------|-------|---------|------|
| **Значение по умолчанию** | Move (бескопировочно) | Высокая производительность, нулевые накладные расходы | 2025-01-15 |
| **Механизм разделения** | Ключевое слово `ref`, автоматическая оптимизация компилятора | Простота для пользователя, ответственность на компиляторе | 2025-01-15 |
| **Заимствование** | `&T`/`&mut T` как токены нулевого размера | Свойства типа (Dup/Linear) естественно выводят права, единая система типов | 2025-01-15 |
| **Токены заимствования** | Замена упрощённого заимствования, `&T` Dup, `&mut T` Linear | Устранение специальных правил вроде «запрет экранирования», поддержка захвата замыканиями / возврата ссылок / хранения в структурах | 2026-05-29 |
| **Копирование** | `clone()` | Явная семантика | 2025-01-15 |
| **Системный уровень** | `*T` + `unsafe` | Поддержка системного программирования | 2025-01-15 |
| **Времена жизни** | Не реализуются | Токены — значения, время жизни управляется Move/RAII единообразно, проблема заимствования сводится к проблеме владения | 2025-01-15 |
| **Rc/Arc** | Автоматический выбор компилятором, невидим для пользователя | Снижение когнитивной нагрузки | 2025-01-15 |
| **Циклические ссылки** | В пределах задачи без проверки, межзадачные — lint (по умолчанию warn) | Структурная конкурентность гарантирует естественно, lint настраивается как deny | 2025-01-16 |
| **Weak** | Предоставляется стандартной библиотекой | Явный выбор для продвинутых пользователей | 2025-01-16 |
| **Анализ потребления** | Удалён | Мини-проверщик заимствований не нужен | 2026-05-11 |
| **Возврат владения** | Удалён | Сигнатура `(T) -> T` сама по себе документация | 2026-05-11 |
| **Переиспользование пустого состояния** | Удалено (как возможность) | Переприсваивание после Move — естественное поведение | 2026-05-11 |
| **Обратные функции / частичное потребление / трёхуровневая мутация полей** | Удалены | Избыточная сложность | 2026-05-11 |

### Обнаружение конфликтов токенов: утверждения логики Хоара

Полное решение для обнаружения конфликтов токенов см. в [RFC-009a: Анализ времени жизни токенов — на основе конвейера доказательств Хоара](../accepted/009a-borrow-proof-pipeline.md). Ключевые моменты:

**Живучесть токенов — утверждение логики Хоара.** `{Все конфликтующие ReadToken мертвы} write(data) {WriteToken безопасно получен}` — совместно с проверкой типов и верификацией пользовательских предикат использует конвейер доказательств из RFC-027. Компилятор автоматически генерирует утверждения заимствования (`borrow_conflict`, `use_after_move`, `use_after_drop`, `mut_violation`), конвейер возвращает Proved / Disproved / Unproven.

**Проверка заимствований не исчезла — она понижена в размерности.** `BorrowChecker` становится `BorrowPredicateEmitter`, генерирующим утверждения, а не выполняющим проверку. Это полностью параллельно концепции «проверщика типов»: проверщик типов генерирует утверждения о равенстве типов, генератор утверждений заимствования генерирует утверждения заимствования, верификация — единый конвейер.

**Идентификатор бренда (`#42`) — это и есть `'a`.** Информация полностью идентична, кодирование различается. `'a` виден в сигнатурах типов, `#42` — внутри компилятора. Никакого нового анализа не изобретено — время жизни понижено с уровня типов до уровня доказательств.

**Краткое описание алгоритма** (подробности в RFC-009a):
- Префиксное сопоставление по дереву брендов → определение конфликтующих токенов (O(глубина), глубина ≤ 3)
- Обратный BFS → от потребителя, break обрезает обратные рёбра, структурный анализ покрывает 95%+ сценариев (быстрый путь)
- Логический отсекатель SMT → вызывается только при наличии while + условий пути (медленный путь, крайне редко)
