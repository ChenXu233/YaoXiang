# RFC-009: Дизайн модели владения

## Аннотация

Данный документ определяет **модель владения (Ownership Model)** языка программирования YaoXiang.

**Ключевой дизайн — пять концепций, один градиент**:

```
Посмотреть/Изменить    Забрать              Совместное хранение    Клонировать       Системный уровень
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         нулевое        компилятор    явное глубокое  *T
  токен нулевого копирование     выбирает      копирование    ответственность
  размера       по умолчанию   Rc/Arc                       на пользователе
  атрибуты типов
  естественный
  вывод прав
```

- **Move (по умолчанию)**: присваивание / передача параметра / возврат = передача владения, нулевое копирование, RAII автоматически освобождает ресурсы
- **`&T` / `&mut T` (токены заимствования)**: токены нулевого размера на этапе компиляции. `&T` является Dup (разделяемое неизменяемое владение), `&mut T` — линейным (эксклюзивное изменяемое). Права естественно выводятся из атрибутов типа, без специальных правил. Можно возвращать и хранить в структурах.
- **`ref` ключевое слово**: разделяемое владение между областями видимости. Компилятор автоматически выбирает Rc (без пересечения задач) или Arc (между задачами)
- **`clone()`**: явное глубокое копирование
- **`unsafe` + `*T`**: сырые указатели, системный аварийный люк

**Устранённая сложность**:
- ❌ Нет времени жизни `'a`
- ❌ Нет отдельного фреймворка проверки заимствований (конфликты заимствования сводятся к утверждениям Хоара, разделяя конвейер доказательства с проверкой типов)
- ❌ Нет GC
- ❌ Нет специальных правил вида «запрет на выход за пределы» (токены — обычные типы, область видимости единообразно обрабатывается системой типов)
- ❌ Пользователю не нужно различать Rc/Arc (выбор делает компилятор)

> **Когнитивная нагрузка**: `&T` копируем, `&mut T` — нет — два атрибута типа, ноль специальных правил, полная автоматизация компилятора.
> **Гарантии производительности**: Move — нулевые накладные расходы, токены — нулевые накладные расходы (тип нулевого размера, исчезает после компиляции), `ref` оплачивается по необходимости, без пауз GC.

## Мотивация

### Зачем нужна модель владения?

| Язык | Управление памятью | Проблемы |
|------|--------------------|----------|
| C/C++ | Ручное | Утечки памяти, висячие указатели, двойное освобождение |
| Java/Python | GC | Колебания задержки, накладные расходы памяти, непредсказуемые паузы |
| Rust | Владение + проверка заимствований | Крутая кривая обучения из-за времени жизни `'a` |
| **YaoXiang** | **Move + Token + ref** | **Просто, детерминированно, без GC** |

### Цели дизайна

```yaoxiang
# 1. Move по умолчанию (нулевое копирование)
p = Point(1.0, 2.0)
p2 = p                         # Move, p больше нельзя читать

# 2. &T / &mut T токены заимствования (нулевые накладные, права выводятся из атрибутов типов)
print_info(p2)                 # компилятор автоматически создаёт токен &Point, освобождается после использования
shift(p2, 1.0, 1.0)           # компилятор автоматически создаёт токен &mut Point

# 3. ref = совместное владение (компилятор автоматически выбирает Rc/Arc)
shared = ref p2                # хранение между областями видимости
spawn { use(shared) }          # компилятор: между задачами → Arc

# 4. clone() = явное копирование
backup = p2.clone()            # глубокое копирование, независимый экземпляр

# 5. unsafe + *T = системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Ключевые отличия от Rust

| Свойство | Rust | YaoXiang |
|------|------|----------|
| Семантика по умолчанию | Заимствование `&T` (требуется явный `.clone()`) | **Move (передача значения, нулевое копирование)** |
| Заимствование | `&T`/`&mut T`, можно возвращать, нужны времена жизни | **Токены `&T`/`&mut T` нулевого размера, права естественно выводятся из атрибутов Dup/Linear** |
| Разделяемое владение | `Arc::new()` + ручной `Weak` | **Ключевое слово `ref` (компилятор автоматически выбирает Rc/Arc)** |
| Копирование | `clone()` | `clone()` |
| Сырой указатель | `*T` | `*T` |
| Время жизни | `'a` | ❌ Нет |
| Проверка заимствований | Глобальный вывод | **Генератор утверждений о заимствованиях в проверщике типов, единый конвейер доказательства** |
| Циклические ссылки | Ручной `Weak` | **Единое освобождение при завершении задачи / lint для межзадачных / `Weak` в стандартной библиотеке** |

## Предложение

### 1. Move (передача владения по умолчанию)

```yaoxiang
# Правило: присваивание / передача параметра / возврат = Move, нулевое копирование

p: Point = Point(1.0, 2.0)
p2 = p                           # Move, p больше нельзя читать

# Переменную можно переприсвоить (стиль Python, без затенения)
p = Point(3.0, 4.0)              # p перепривязывается, тип должен совпадать

# Параметры функции: Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Возврат через Move
}

# Возврат из функции: Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Возврат через Move, нулевое копирование
}
```

**Особенности**:
- Нулевое копирование (компилятор перемещает указатель)
- Исходная привязка после перемещения недоступна для чтения (ошибка компиляции)
- RAII: автоматическое освобождение при выходе из области видимости
- Сигнатура функции `(T) -> T` сама по себе является документацией — потребляет T, возвращает T

### 2. `&T` / `&mut T` (токены заимствования)

**Ключевой принцип: `&T` и `&mut T` — это токены нулевого размера на этапе компиляции. Это не «ссылки», а «типизированные доказательства права доступа».**

#### 2.1 Два атрибута типа

```
&T      →  нулевой размер, замораживает исходные данные (пока живёт ReadToken, WriteToken запрещён),
          при заморозке несколько неизменяемых представлений безопасны → копируем (Dup)
&mut T  →  нулевой размер, эксклюзивный доступ на чтение и запись (пока живёт WriteToken, любой другой токен запрещён),
          копирование бессмысленно при эксклюзивном доступе → линейный (не Dup)
```

**Причинно-следственная связь не может быть обратной: заморозка — причина, Dup — следствие.** Не потому что `&T` реализует Dup и потому может сосуществовать — а потому что данные заморожены (мутация невозможна), несколько неизменяемых представлений безопасны, и Dup становится возможным. Если принять Dup за определение, а проверку конфликтов — за «дополнительную заплатку», дизайн ошибочен.

#### 2.2 Базовое использование

```yaoxiang
# Сторона метода: объявляет типы параметров, определяя необходимые права
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # токен &Point даёт право чтения
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # токен &mut Point даёт право записи
    self.y = self.y + dy
}

# Сторона вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                          # компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)                  # компилятор автоматически создаёт токен &mut Point
p.print()                          # OK, предыдущий токен уже освобождён после вызова shift

# Свободные функции — то же самое
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # два токена &Point сосуществуют — Dup
}
d = distance(p, p2)
```

#### 2.3 Почему не нужно «правило запрета выхода»

RFC-009 v8 накладывал на `&T`/`&mut T` три специальных правила — только параметры, нельзя возвращать, нельзя хранить в структурах. Это были заплатки на концепцию «заимствования».

Системе токенов эти правила не нужны. Токены — это **обычные типы**, подчиняющиеся тем же правилам области видимости, что и любые другие типы.

**Возврат ссылок — естественная поддержка**:

```yaoxiang
# ✅ Токен передаётся вместе с возвращаемым значением
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # дочерний и родительский токены возвращаются вместе
}

# Использование
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # токен возвращён вызывающему
print(px_ref)               # OK, токен всё ещё в области видимости
```

**Хранение в структурах — естественная поддержка**:

```yaoxiang
# ✅ Структура несёт токен в качестве поля
Window: Type = {
    target: Point,
    view: &Point,      # поле-токен — хранит неизменяемое представление target
}

# Токен view порождён из target, Window владеет обоими
# Пока Window существует, токен view действителен
```

#### 2.3 Замыкания и Lambda с явными параметрами

Lambda — это значение функции — оно может быть возвращено, сохранено, передано за пределы текущей области видимости. Поэтому Lambda **не захватывает неявно переменные внешней области видимости**. При необходимости данные извне передаются явно как параметры:

```yaoxiang
# ✅ Lambda использует явные параметры
double: (x: Int) -> Int = (x) => x * 2
filter_by: (items: List(Int), f: (Int) -> Bool) -> List(Int) = { ... }

# ✅ spawn { } не подчиняется этому правилу — spawn выполняет параллельный блок немедленно, родительская задача блокируется в ожидании
shared = ref data
spawn { use(shared) }

# ❌ Lambda не может неявно захватывать внешние переменные
x = 42
f = () => { x + 1 }  # ошибка компиляции: x не в области видимости

# ✅ Правильный способ: явная передача параметра
f = (x) => { x + 1 }
f(x)
```

**spawn { } — это не значение функции.** Блок, помеченный `spawn`, как и тело `if`/`while`, выполняется немедленно и завершается в течение жизни родительского фрейма стека. Тело `spawn` может свободно обращаться к внешним переменным.

**Между задачами — токены не могут пересекать границу потока**:

```yaoxiang
# ❌ Токены не могут пересекать границу задачи
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ ошибка компиляции: токен нельзя передавать между задачами
}

# Это не специальное правило — токен является доказательством прав на этапе компиляции, для разделяемого владения между задачами используйте ref
# Если нужно разделить владение между задачами, используйте ref
```

**Токен не может быть ref**:

```yaoxiang
# ❌ Токен — это доказательство прав, а не владение
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ ошибка компиляции: &T не является владеющим типом
}
```

#### 2.4 Время жизни токенов

Время жизни токенов определяется **обычными правилами области видимости**, без параметров времени жизни:

- Токен в параметре функции: живёт в течение вызова, освобождается после
- Возвращённый токен: владение передаётся вызывающему
- Токен, хранящийся в структуре: живёт вместе со структурой

Компилятору не нужны аннотации `'a`, потому что токены — это **значения**, а время жизни значений единообразно управляется системой владения (Move/RAII). **Сведение проблемы заимствования к проблеме владения.**

#### 2.5 Обнаружение конфликтов токенов

Обнаружение конфликтов токенов — это **утверждение логики Хоара**, а не отдельный потоко-чувствительный анализ.

```
{все конфликтующие ReadToken мертвы} write(data) {WriteToken безопасно получен}
```

Разделяет конвейер доказательства RFC-027 с проверкой типов и валидацией пользовательских предикатов. Компилятор автоматически генерирует утверждения о заимствованиях (`borrow_conflict`, `use_after_move`, `use_after_drop`, `mut_violation`) и отправляет их в конвейер. Конвейер возвращает Proved / Disproved / Unproven.

```yaoxiang
# ❌ Токен &mut является линейным, его нельзя копировать
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move, p больше нельзя читать
    p.x = 10.0                      # ❌ ошибка компиляции: WriteToken уже перемещён
}

# ✅ Токен &T — Dup, может свободно копироваться
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK, &T — Dup
    print(p.x)                      # OK
    print(p2.x)                     # OK, два неизменяемых токена сосуществуют
}
```

**Проверка заимствований не исчезла — она была понижена в размерности.** Существующий `BorrowChecker` становится `BorrowPredicateEmitter` (генератор утверждений), генерирующим утверждения о заимствованиях, которые разделяют конвейер доказательства с другими утверждениями о типах. Это полностью параллельно концепции проверщика типов: проверщик типов генерирует утверждения о равенстве типов, генератор утверждений о заимствованиях генерирует утверждения о заимствованиях — один конвейер их валидирует. Подробности см. в [RFC-009a](../accepted/009a-borrow-proof-pipeline.md).

#### 2.7 Внутреннее устройство компилятора: механизм брендов

Пользователь никогда не сталкивается с брендами. Компилятор внутри назначает каждому токену уникальный идентификатор на этапе компиляции:

```
Видимое пользователю       Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — уникальное целое на этапе компиляции
&mut Point     →  WriteToken(Point, #M)   // #M — уникальное целое на этапе компиляции
```

Назначение брендов:
- **Защита от подделки**: токен может быть получен только из капсулы владельца, не может быть сконструирован из ниоткуда
- **Отслеживание связей**: при порождении `&Float` из `&Point` (доступ к полю) `&Float` несёт порождённый бренд (`#N.field_x`), компилятор может отследить его до родительского токена
- **Обнаружение конфликтов**: одноимённый `WriteToken` и порождённый `ReadToken` не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнинга, в генерируемом машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

#### 2.8 Правила автоматического выбора заимствования

На стороне вызова компилятор руководствуется следующим приоритетом:

```
1. Если аргумент используется далее → предпочтительно создать токен (&T или &mut T, согласно сигнатуре метода)
2. Если аргумент далее не используется → Move
3. Приоритет соответствия: &T < &mut T < Move
```

```yaoxiang
# Пример: автоматический выбор
p = Point(1.0, 2.0)
p.print()        # print объявляет &self → компилятор создаёт токен &Point
p.shift(1.0, 1.0) # shift объявляет &mut self → компилятор создаёт токен &mut Point
p2 = p           # Move, p больше не используется
```

#### 2.9 Сравнение с упрощённым заимствованием RFC-009 v8

| Свойство | Упрощённое заимствование (v8) | Токены заимствования (v9) |
|------|--------------|--------------|
| Возврат ссылок | ❌ Жёстко запрещено | ✅ Токен передаётся с возвращаемым значением |
| Хранение в структурах | ❌ Жёстко запрещено | ✅ Токен как поле структуры |
| Явные параметры Lambda | ❌ Жёстко запрещено | ✅ Lambda использует явные параметры |
| Специальные правила | 3 (только параметры / нельзя возвращать / нельзя хранить) | 0 — атрибуты типа естественно выводят права |
| Проверка заимствований | Специализированная перекрёстная проверка | Потоко-чувствительный анализ активности в проверщике типов |
| Аннотации времени жизни | Не нужны | Не нужны |
| Накладные расходы во время выполнения | Ноль | Ноль (тип нулевого размера, исчезает после компиляции) |
| Сообщения об ошибках | «Заимствование не может выйти за пределы» | «WriteToken(#3) уже перемещён» (обычная ошибка типа) |
| Ментальная модель пользователя | Понимание особого статуса «заимствования» | `&T` копируем, `&mut T` — нет |

### 3. Ключевое слово `ref` (автоматическая оптимизация компилятора)

`ref` — единственный способ разделяемого владения между областями видимости. Что под капотом — Rc или Arc — пользователя не должно волновать.

#### 3.1 Базовое использование

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # совместное владение, компилятор сам выберет реализацию

# Разделяемое владение между задачами
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # компилятор: между задачами → Arc
    spawn { use(data) }           # компилятор: между задачами → Arc
}

# Разделяемое владение в пределах задачи
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # компилятор: без пересечения задач → Rc
}
```

**Ментальная модель пользователя**: `ref` = совместное хранение. Этого достаточно.

#### 3.2 Анализ побега компилятора: Rc vs Arc

```
Анализ потока данных ref:

Не покидает пределы задачи → Rc (неатомарный подсчёт ссылок, низкие накладные расходы)
Покидает в другие задачи    → Arc (атомарный подсчёт ссылок, потокобезопасный)
```

#### 3.3 Стратегия обнаружения циклов

```
Циклы внутри задачи → молча допускаются.
  ├── Каждая задача имеет чёткую границу жизненного цикла — при завершении задачи все ресурсы (включая циклы ref) освобождаются единообразно.
  ├── Долгоживущие сервисы должны создавать дочерние задачи на каждый запрос/соединение — дочерние задачи автоматически освобождают ресурсы, не накапливая утечек.
  ├── `ref` всегда сохраняет живучесть, семантика не размывается.
  └── Пользователь имеет право строить двунаправленные сильные ссылки внутри задачи (например, промежуточные состояния графовых вычислений).

Межзадачные циклы → lint (по умолчанию warn, настраивается).
  ├── Поведение программы корректно, реальной утечки нет (при завершении родительской задачи все ресурсы дочерних освобождаются).
  ├── Но межзадачные сильные ссылки означают размытые границы владения, что стоит пересмотреть.
  ├── По умолчанию уровень warn, компиляция проходит с предупреждением.
  └── Команда может установить deny в конфигурации проекта, включив его в качественный шлюз CI.
```

**Уровни Lint** (аналогично Rust clippy):

| Уровень | Поведение | Сценарий |
|------|------|------|
| `allow` | Не проверять | Личный проект |
| `warn` (по умолчанию) | Компиляция проходит, есть предупреждения | Этап разработки |
| `deny` | Ошибка компиляции | Командный качественный шлюз CI |
| `forbid` | Ошибка компиляции, нельзя переопределить | Обязательные правила уровня организации |

```yaoxiang
# Цикл внутри задачи: молча допускается, двунаправленные сильные ссылки
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # Цикл. Единообразно освобождается при завершении задачи.
}

# Межзадачный цикл: lint (по умолчанию warn)
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: межзадачная циклическая ссылка
    }
}
```

**Пример конфигурации проекта**:

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # межзадачные циклы немедленно отклоняются в CI
```

| Тип цикла | Поведение | Причина |
|--------|------|------|
| Цикл ref внутри задачи | Проверка не выполняется | Право пользователя, единообразное освобождение при завершении задачи |
| Межзадачный цикл ref | lint (по умолчанию warn) | Напоминание о необходимости пересмотра, можно настроить на deny |

#### 3.4 `Weak`: предоставляется стандартной библиотекой

```yaoxiang
use std.rc.Weak

# Продвинутые пользователи делают явный выбор
a.next = ref b
b.prev = Weak.new(a.next)        # Пользователь явно управляет тем, какое направление слабое
```

**`Weak` не встроен в язык — это тип стандартной библиотеки.** Для повседневных задач достаточно `ref`. Продвинутые пользователи, которым нужен тонкий контроль памяти, подключают `Weak` вручную.

#### 3.5 Токены заимствования vs `ref`

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Назначение | Посмотреть / изменить на месте | Совместное хранение |
| Область | Область видимости значения токена | Между областями видимости |
| Стоимость | Нулевые накладные расходы (тип нулевого размера) | Rc или Arc (выбор компилятора) |
| Побег | Возможен (токен передаётся с возвратом / структурой / замыканием) | Для этого и предназначен |
| Между задачами | Невозможно (токен — доказательство прав, не передаётся между задачами) | Возможно (компилятор автоматически выбирает Arc) |
| Циклы | Не задействовано | Внутри задачи молча допускаются, между задачами — lint |

### 4. `clone()` — явное копирование

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # глубокое копирование
# p и p2 независимы, не влияют друг на друга
```

**Когда использовать**: нужно сохранить исходное значение, и ни Move, ни совместное владение не подходят.

### 5. `unsafe` + сырые указатели (системное программирование)

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # сырой указатель
    (*ptr).x = 0.0                # разыменование (безопасность гарантирует пользователь)
    ptr2 = ptr + 1                # арифметика указателей
}
```

**Ограничения**:
- Доступно только в блоке `unsafe`
- Пользователь гарантирует отсутствие висячих указателей и использования после освобождения
- Предназначено для FFI, операций с памятью и другого системного программирования

### 6. Общий градиент владения

```
  Токен заимствования     Move           Совместное владение   Копирование

   │                      │                  │                │

  &mut T линейный токен  по умолчанию      Выбор компилятора  Явное глубокое
                                              копирование

  значения токена                          видимости

## Комплексный пример

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T: токен неизменяемого доступа
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T: токен изменяемого доступа
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move: потребление с возвратом
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # забрал, изменил, вернул
    }

    # Возврат ссылки: токен передаётся с возвращаемым значением
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# Lambda с явными параметрами
double: (x: Int) -> Int = (x) => x * 2

# Комплексное использование
p = Point(1.0, 2.0)
p.print()                           # токен &Point
p.shift(1.0, 1.0)                   # токен &mut Point
p = p.scale(2.0)                    # Move → возврат
shared = ref p                      # ref совместное владение
spawn { use(shared) }

# clone — независимая копия
backup = p.clone()

# Цикл внутри задачи: молча допускается
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # цикл, единообразно освобождается при завершении задачи

# unsafe — системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## Ограничения системы типов

### Атрибут типа `Dup`

`Dup` (Duplicable) — автоматически управляемый компилятором атрибут типа, означающий **поверхностное копирование**: при присваивании / передаче параметра копируется дескриптор/токен, а нижележащие данные разделяются. Это формирует трёхуровневый градиент вместе с Move (передача владения) и Clone (явное глубокое копирование, создающее независимую копию).

**Dup и Clone — ортогональные концепции** — Dup копирует дескриптор с разделением данных, Clone создаёт независимую копию. Тип может одновременно поддерживать и Dup, и Clone, либо только один из них.

| Тип | Dup | Clone | Пояснение |
|------|-----|-------|------|
| `&T` | ✅ (копируется токен, несколько представлений указывают на одни данные) | ✅ | Токен неизменяемого доступа |
| `ref T` | ✅ (счётчик ссылок +1, разделяемые данные в куче) | ✅ | Совместное хранение (компилятор сам выбирает Rc/Arc) |
| String, Bytes | ✅ (внутренний подсчёт ссылок, копируется дескриптор, разделяется буфер) | ✅ | Строка / байты |
| `&mut T` | ❌ (линейный, эксклюзивный) | ❌ | Токен изменяемого доступа |
| `*T` | ❌ | ❌ | Сырой указатель |
| struct | Производный (автоматически, если все поля Dup) | ✅ | Структура |

**Примитивные типы значений** (Int, Float, Bool, Char) при присваивании подвергаются встроенному копированию значений компилятором — два значения полностью независимы, это не поверхностное копирование. Они не относятся к атрибуту Dup, а обрабатываются компилятором изначально.

## Анализ производительности

| Операция | Стоимость | Пояснение |
|------|------|------|
| Move | Ноль | Перемещение указателя |
| `&T` / `&mut T` | Ноль | Тип нулевого размера, исчезает после компиляции, нулевые накладные расходы во время выполнения |
| `ref` (без пересечения задач) | Низкая | Компилируется в Rc, неатомарная операция |
| `ref` (между задачами) | Средняя | Компилируется в Arc, атомарная операция |
| `clone()` | Зависит от типа | Малые объекты быстро, большие — медленно |
| `unsafe + *T` | Ноль | Прямые операции с памятью |

### Сравнение

| Язык | Механизм разделяемого владения | Управление памятью | Обработка циклов | Сложность |
|------|----------|----------|----------|----------|
| Rust | Arc / Mutex + проверка заимствований | Проверка на этапе компиляции | Ручной `Weak` | Высокая |
| Go | chan / pointer | GC | GC | Низкая |
| C++ | shared_ptr | RAII | weak_ptr | Средняя |
| **YaoXiang** | **ref + токены заимствования** | **RAII** | **Освобождение по границе задачи / lint для межзадачных / Weak в стандартной библиотеке** | **Низкая** |

## Компромиссы

### Преимущества

1. **Единообразие**: `&T`/`&mut T` — обычные типы, а не специальные возможности языка. Полностью согласуется с `name: type = value` из RFC-010
2. **Простота**: нет времён жизни, проверка заимствований сведена к утверждениям системы типов. `&T` копируем, `&mut T` — нет — два атрибута типа
3. **Мощность**: можно возвращать ссылки, хранить в структурах, захватывать в замыканиях — выразительность на уровне Rust
4. **Умный компилятор**: `ref` автоматически выбирает Rc/Arc, сторона вызова автоматически выбирает заимствование
5. **Детерминированность**: `ref` сохраняет живучесть, не превращается молча в слабую ссылку
6. **Высокая производительность**: Move — нулевое копирование, токены — нулевые накладные расходы (тип нулевого размера, исчезает после компиляции)
7. **Гибкость**: `unsafe + *T` поддерживает системное программирование

### Недостатки

1. **Распространение параметров брендов в дженериках**: токены несут идентификаторы брендов, в сигнатурах функций, возвращающих ссылки, проявляются дополнительные параметры дженериков
2. **Накладные расходы `ref` во время выполнения**: атомарные операции имеют стоимость (но это неизбежная цена разделяемого владения)
3. **Риски `unsafe`**: пользователь должен гарантировать корректность
4. **Межзадачные циклы — lint, а не ошибка компиляции**: в отличие от Rust не приводит к ошибке компиляции, по умолчанию warn; для превращения в качественный шлюз требуется настройка `deny` командой

| **Копирование** | `clone()` | Явная семантика | 2025-01-15 |
| **Системный уровень** | `*T` + `unsafe` | Поддержка системного программирования | 2025-01-15 |
| **Время жизни** | Не реализуется | Токены — значения, их время жизни управляется единообразно через Move/RAII, что сводит проблему заимствования к проблеме владения | 2025-01-15 |
| **Rc/Arc** | Автоматический выбор компилятором, невидим для пользователя | Снижение когнитивной нагрузки | 2025-01-15 |
| **Циклические ссылки** | Внутри задачи не проверяются, между задачами — lint (по умолчанию warn) | Структурная конкурентность даёт естественную гарантию, lint может быть настроен на deny | 2025-01-16 |

### Обнаружение конфликтов токенов: утверждения логики Хоара

Полное решение по обнаружению конфликтов токенов см. в [RFC-009a: Анализ жизненного цикла токенов — на основе конвейера доказательств Хоара](../accepted/009a-borrow-proof-pipeline.md). Ключевые моменты:

**Активность токенов — это утверждение логики Хоара.** `{все конфликтующие ReadToken мертвы} write(data) {WriteToken безопасно получен}` — разделяет конвейер доказательства RFC-027 с проверкой типов и валидацией пользовательских предикатов. Компилятор автоматически генерирует утверждения о заимствованиях (`borrow_conflict`, `use_after_move`, `use_after_drop`, `mut_violation`), конвейер возвращает Proved / Disproved / Unproven.

**Проверка заимствований не исчезла — она была понижена в размерности.** `BorrowChecker` становится `BorrowPredicateEmitter`, генерирующим утверждения вместо выполнения проверки. Это полностью параллельно концепции «проверщика типов»: проверщик типов генерирует утверждения о равенстве типов, генератор утверждений о заимствованиях генерирует утверждения о заимствованиях — один конвейер их валидирует.

**Идентификатор бренда (`#42`) — это и есть `'a`.** Информация полностью идентична, различается только кодирование. `'a` виден в сигнатурах типов, `#42` — внутренний для компилятора. Никакого нового анализа не изобретено — время жизни перенесено с уровня типов на уровень доказательств.

**Краткое описание алгоритма** (подробности в RFC-009a):
- Сопоставление префиксов дерева брендов → определение конфликтующих токенов (O(depth), глубина ≤ 3)
- Обратный BFS → от потребителя, break обрезает обратные рёбра, структурный анализ покрывает 95%+ сценариев (быстрый канал)
- SMT-логика обрезки → вызывается только при `while` + условии пути (медленный канал, крайне редко)
