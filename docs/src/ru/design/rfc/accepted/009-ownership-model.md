# RFC-009: Проектирование модели владения

> **Статус**: Принято
> **Автор**: Чэнь Сюй (晨煦)
> **Дата создания**: 2025-01-08
> **Последнее обновление**: 2026-05-29 (система заимствования токенов заменила упрощённое заимствование, унифицированная система типов)

## Аннотация

В данном документе определяется **модель владения (Ownership Model)** языка программирования YaoXiang.

**Основной дизайн — пять концепций, один градиент**:

```
Посмотреть/Изменить      Забрать            Разделение         Скопировать         Системный
на месте                 себе               владения           экземпляр          уровень
    │                      │                   │                  │                  │
   &T                    Move                ref               clone()            unsafe
  &mut T                 Нулевое             Компилятор         Явное              *T
  Токен                  копирование         автоматически      глубокое           Пользователь
  нулевого               По умолчанию        выбирает           копирование        ответственен
  размера                RAII                Rc/Arc
  Тип атрибута                               
  естественно                               
  выводит права                              
  
```

- **Move (по умолчанию)**: присваивание/передача параметров/возврат = передача владения, нулевое копирование, автоматическое освобождение через RAII
- **`&T` / `&mut T` (токены заимствования)**: типы токенов нулевого размера на этапе компиляции. `&T` копируемый (разделение только для чтения), `&mut T` линейный (исключительная мутация). Права естественно выводятся из атрибутов типа, без специальных правил. Можно возвращать, сохранять в структуры, захватывать в замыкания.
- **`ref` ключевое слово**: разделение владения между областями видимости. Компилятор автоматически выбирает Rc (не пересекает задачи) или Arc (пересекает задачи)
- **`clone()`**: явное глубокое копирование
- **`unsafe` + `*T`**: голые указатели, системный аварийный выход

**Устранённая сложность**:
- ❌ Нет времени жизни `'a`
- ❌ Нет проверки заимствований (атрибуты типа токена — Dup/Linear — заменяют специализированный проверяльщик заимствований)
- ❌ Нет сборщика мусора
- ❌ Нет специальных правил типа «запрет на побег» (токены — обычные типы, области видимости обрабатываются единой системой типов)
- ❌ Пользователю не нужно знать разницу между Rc/Arc (компилятор выбирает автоматически)

> **Бремя программирования**: `&T` копируемый, `&mut T` не копируемый — два атрибута типа, ноль специальных правил, полная автоматизация компилятора.
> **Гарантия производительности**: Move с нулевыми накладными расходами, токены с нулевыми накладными расходами (типы нулевого размера, исчезают после компиляции), ref с оплатой по факту использования, нет пауз сборщика мусора.

## Мотивация

### Зачем нужна модель владения?

| Язык | Управление памятью | Проблемы |
|------|---------------------|----------|
| C/C++ | Ручное управление | Утечки памяти, висячие указатели, двойное освобождение |
| Java/Python | Сборщик мусора | Задержки, накладные расходы памяти, непредсказуемые паузы |
| Rust | Владение + проверка заимствований | Крутая кривая обучения времени жизни `'a` |
| **YaoXiang** | **Move + Token + ref** | **Простота, детерминированность, без GC** |

### Цели проектирования

```yaoxiang
# 1. Move по умолчанию (нулевое копирование)
p = Point(1.0, 2.0)
p2 = p                         # Move, p больше нельзя читать

# 2. &T / &mut T токены заимствования (нулевые накладные расходы, атрибуты типа естественно выводят права)
print_info(p2)                 # Компилятор автоматически создаёт токен &Point, освобождает после использования
shift(p2, 1.0, 1.0)           # Компилятор автоматически создаёт токен &mut Point

# 3. ref = разделение (компилятор автоматически выбирает Rc/Arc)
shared = ref p2                # Разделение владения между областями видимости
spawn { use(shared) }          # Компилятор: пересекает задачи → Arc

# 4. clone() = явное копирование
backup = p2.clone()            # Глубокое копирование, независимая копия

# 5. unsafe + *T = системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Ключевые отличия от Rust

| Характеристика | Rust | YaoXiang |
|----------------|------|----------|
| Семантика по умолчанию | Заимствование `&T` (нужно явное `.clone()`) | **Move (передача значений, нулевое копирование)** |
| Заимствование | `&T`/`&mut T`, можно возвращать, требуется время жизни | **`&T`/`&mut T` токены нулевого размера, Dup/Linear атрибуты типа естественно выводят** |
| Механизм разделения | `Arc::new()` + ручной Weak | **`ref` ключевое слово (компилятор автоматически выбирает Rc/Arc)** |
| Копирование | `clone()` | `clone()` |
| Голые указатели | `*T` | `*T` |
| Время жизни | `'a` | ❌ Нет |
| Проверка заимствований | Глобальный вывод | **Потокочувствительный анализ активности проверки типов (отслеживание состояния токена)** |
| Циклические ссылки | Ручной Weak | **Единое освобождение при завершении задачи / lint для межзадачных / стандартная библиотека Weak** |

## Предложение

### 1. Move (передача владения по умолчанию)

```yaoxiang
# Правило: присваивание / передача параметров / возврат = Move, нулевое копирование

p: Point = Point(1.0, 2.0)
p2 = p                           # Move, p больше нельзя читать

# Переменной можно переприсвоить (стиль Python, без затенения)
p = Point(3.0, 4.0)              # p перепривязывается, тип должен совпадать

# Параметры функции: Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move возврат
}

# Возврат из функции: Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move возврат, нулевое копирование
}
```

**Характеристики**:
- Нулевое копирование (компилятор перемещает указатель)
- После перемещения исходная привязка недоступна для чтения (ошибка компиляции)
- RAII: автоматическое освобождение при выходе из области видимости
- Сигнатура функции `(T) -> T` сама по себе документация — потребляет T, возвращает T

### 2. &T / &mut T (токены заимствования)

**Основной принцип: `&T` и `&mut T` — это типы токенов нулевого размера на этапе компиляции. Это не «ссылки», а «типизированные доказательства прав доступа».**

#### 2.1 Два атрибута типа

```
&T      →  Нулевой размер, копируемый (Dup), предоставляет права только на чтение
&mut T  →  Нулевой размер, линейный (не Dup), предоставляет права на эксклюзивное чтение-запись
```

**Это не «правило», которое нужно запоминать — это базовые атрибуты системы типов.** Типы Dup могут свободно копироваться (несколько `&T` сосуществуют), линейные типы не копируются (`&mut T` по природе уникален). Нет «проверяльщика заимствований» — только проверка типов делает то, что она всегда делала.

#### 2.2 Базовое использование

```yaoxiang
# Методы: объявление типов параметров, определяющее необходимые права
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # Токен &Point предоставляет право чтения
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # Токен &mut Point предоставляет право записи
    self.y = self.y + dy
}

# Сторона вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                          # Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)                  # Компилятор автоматически создаёт токен &mut Point
p.print()                          # OK, предыдущий токен освобождён после вызова shift

# Свободные функции аналогично
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # Два токена &Point сосуществуют — тип Dup
}
d = distance(p, p2)
```

#### 2.3 Почему не нужно «запрет на побег»

В RFC-009 v8 для `&T`/`&mut T` применялись три специальных правила — только параметры, нельзя возвращать, нельзя сохранять в структурах. Это патчи для концепции «заимствования».

Система токенов не нуждается в этих правилах. Токены — **обычные типы**, следующие тем же правилам области видимости, что и все остальные типы.

**Возврат ссылок — естественно поддерживается**:

```yaoxiang
# ✅ Токен распространяется вместе с возвращаемым значением
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # Дочерний токен и родительский токен возвращаются вместе
}

# Использование
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # Токен возвращается вызывающему
print(px_ref)               # OK, токен всё ещё в области видимости
```

**Сохранение в структурах — естественно поддерживается**:

```yaoxiang
# ✅ Структура содержит токен как поле
Window: Type = {
    target: Point,
    view: &Point,      # Поле токена — держит представление target только для чтения
}

# Токен view производен от target, Window владеет обоими
# Пока существует Window, токен view валиден
```

**Захват в замыканиях — естественно поддерживается**:

```yaoxiang
# ✅ Замыкание захватывает токен, как любую величину
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    # Замыкание захватывает токен &Float порога (тип Dup, свободно копируется в замыкание)
    items.filter(|p| p.x > threshold)
}

# Это то, что RFC-009 v8 не мог — v8 запрещал захват заимствований в замыкания
```

**Межзадачность — токены не проходят через потоки**:

```yaoxiang
# ❌ Токены не могут пересекать границы задач
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ Ошибка компиляции: токен нельзя передавать между задачами
}

# Это не специальное правило — токен это доказательство прав на этапе компиляции,
# для межзадачного разделения используйте ref
```

**Токен нельзя обернуть в ref**:

```yaoxiang
# ❌ Токен это доказательство прав, а не владение
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ Ошибка компиляции: &T не является владеющим типом
}
```

#### 2.4 Жизненный цикл токена

Жизненный цикл токена определяется **обычными правилами области видимости**, без параметров времени жизни:

- Токены в параметрах функции: живут во время вызова, освобождаются после завершения вызова
- Возвращаемые токены: владение передаётся вызывающему
- Токены, хранящиеся в структурах: живут вместе со структурой
- Захваченные в замыканиях токены: живут вместе с замыканием

Компилятору не нужны аннотации `'a`, потому что токены — **значения**, жизненный цикл значений управляется системой владения (Move/RAII). **Заимствование сводится к проблеме владения.**

#### 2.5 Механизм заморозки

Токен `&mut T` можно временно «заморозить» для получения токена `&T`:

```yaoxiang
modify_and_read: (p: &mut Point) -> Void = {
    p.x = 10.0                      # Используем &mut Point для модификации
    
    # Замораживаем &mut, получаем представление только для чтения
    view: &Point = freeze(p)         # p заморожен в этой точке
    print(view.x)                   # Читаем через &Point
    print(view.y)
    # view выходит из области видимости, заморозка снята
    
    p.y = 20.0                      # &mut Point снова доступен
}
```

Семантика `freeze`:
- Принимает `&mut T`, возвращает `&T`
- Пока жив `&T`, оригинальный `&mut T` недоступен
- После выхода `&T` из области видимости `&mut T` автоматически восстанавливается
- Это **потокочувствительный анализ** — компилятор отслеживает состояние токена внутри тела функции

#### 2.6 Обнаружение конфликтов токенов

Замена «перекрёстной проверки заимствований» из RFC-009 v8. Принцип проще — **потокочувствительный анализ активности** значений токенов:

```yaoxiang
# ❌ &mut и производный &T не могут быть активны одновременно
bad_alias: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p заморожен
    p.x = 10.0                      # ❌ Ошибка компиляции: WriteToken в замороженном состоянии
    print(view.x)                   
}

# ✅ После снятия заморозки можно продолжить использовать &mut
good_seq: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p заморожен
    print(view.x)                   # Используем &T
    # view выходит из области видимости
    p.x = 10.0                      # ✅ WriteToken восстановлен
}
```

**Способ обнаружения**: Это не специализированный «проверяльщик заимствований» — это **потокочувствительный анализ активности** значений токенов. Компилятор отслеживает состояние каждого токена внутри тела функции (активен/заморожен/перемещён), точно так же, как отслеживает любые значения линейного типа.

#### 2.7 Внутренности компилятора: механизм брендов

Пользователь никогда не видит бренды. Компилятор внутренне назначает каждому токену уникальный идентификатор времени компиляции:

```
Видимое пользователю    Внутреннее представление компилятора
────────────────────────────────────────────────────────────────
&Point                →  ReadToken(Point, #N)    // #N — уникальное целое число времени компиляции
&mut Point            →  WriteToken(Point, #M)   // #M — уникальное целое число времени компиляции
```

Назначение брендов:
- **Защита от подделки**: токен можно получить только от владеющей капсулы или операции freeze, создать из воздуха нельзя
- **Отслеживание производности**: при получении `&Float` из `&Point` (доступ к полю), `&Float` несёт производный бренд (`#N.field_x`), компилятор может отследить до родительского токена
- **Обнаружение конфликтов**: однотипные `WriteToken` и производный `ReadToken` не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнга, в сгенерированном машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

#### 2.8 Правила автоматического выбора заимствования

Компилятор на стороне вызова автоматически выбирает по следующему приоритету:

```
1. Если фактический аргумент ещё используется далее → предпочтительно создать токен (&T или &mut T, согласно сигнатуре метода)
2. Если фактический аргумент далее не используется → Move
3. Приоритет сопоставления: &T < &mut T < Move
```

```yaoxiang
# Пример: автоматический выбор
p = Point(1.0, 2.0)
p.print()        # print объявляет &self → компилятор создаёт токен &Point
p.shift(1.0, 1.0) # shift объявляет &mut self → компилятор создаёт токен &mut Point
p2 = p           # Move, p больше не используется
```

#### 2.9 Сравнение с упрощённым заимствованием из RFC-009 v8

| Характеристика | Упрощённое заимствование (v8) | Токены заимствования (v9) |
|----------------|-------------------------------|---------------------------|
| Возврат ссылок | ❌ Жёстко запрещено | ✅ Токен распространяется с возвращаемым значением |
| Сохранение в структурах | ❌ Жёстко запрещено | ✅ Токен как поле структуры |
| Захват в замыканиях | ❌ Жёстко запрещено | ✅ Замыкание захватывает значение токена |
| Специальные правила | 3 (только параметры/нельзя возвращать/нельзя сохранять) | 0 — атрибуты типа естественно выводят |
| Проверка заимствований | Специализированная перекрёстная проверка | Потокочувствительный анализ активности проверки типов |
| Аннотации времени жизни | Не нужны | Не нужны |
| Накладные расходы во время выполнения | Ноль | Ноль (типы нулевого размера, исчезают после компиляции) |
| Сообщения об ошибках | "Заимствование не может выйти за пределы" | "WriteToken(#3) заморожен" (обычная ошибка типа) |
| Ментальная модель пользователя | Понимание «особого статуса» заимствования | `&T` копируемый, `&mut T` не копируемый |

### 3. ref ключевое слово (автоматическая оптимизация компилятора)

`ref` — единственный способ разделения владения между областями видимости. Что в основе — Rc или Arc, пользователю знать не нужно.

#### 3.1 Базовое использование

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # Разделение, компилятор автоматически выбирает реализацию

# Разделение между задачами
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # Компилятор: межзадачность → Arc
    spawn { use(data) }           # Компилятор: межзадачность → Arc
}

# Разделение в одной задаче
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # Компилятор: не межзадачность → Rc
}
```

**Ментальная модель пользователя**: `ref` = разделяемое владение. Этого достаточно.

#### 3.2 Анализ побега компилятора: Rc vs Arc

```
Анализ потока данных ref:

Не сбегает в другие задачи → Rc (неатомарный подсчёт ссылок, низкие накладные расходы)
Сбегает в другие задачи   → Arc (атомарный подсчёт ссылок, потокобезопасность)
```

#### 3.3 Стратегия обнаружения циклов

```
Циклы внутри задачи → бездействие.
  ├── Структурированный параллелизм гарантирует единое освобождение всех ресурсов при завершении задачи.
  ├── ref всегда живой, семантика без поблажек.
  └── Пользователь имеет право строить двунаправленные сильные ссылки внутри задачи (например, промежуточные состояния графовых вычислений).

Циклы между задачами → lint (по умолчанию warn, настраивается).
  ├── Программа работает корректно, настоящих утечек не будет (при завершении родительской задачи все ресурсы дочерних задач освобождаются).
  ├── Но межзадачные сильные ссылки означают размытые границы владения, стоит переосмыслить.
  ├── По умолчанию warn, компиляция проходит с предупреждением.
  └── Команда может в проекте установить deny, включив в CI как контроль качества.
```

**Уровни lint** (аналогично Rust clippy):

| Уровень | Поведение | Сценарий |
|---------|-----------|----------|
| `allow` | Не проверять | Личные проекты |
| `warn` (по умолчанию) | Компиляция проходит, есть предупреждение | Этап разработки |
| `deny` | Ошибка компиляции | Командный CI контроль качества |
| `forbid` | Ошибка компиляции, нельзя переопределить | Организационные обязательные правила |

```yaoxiang
# Цикл внутри задачи: бездействие, двунаправленные сильные ссылки
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # Цикл. При завершении задачи всё освобождается.
}

# Цикл между задачами: lint (по умолчанию warn)
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: циклическая ссылка между задачами
    }
}
```

**Пример конфигурации проекта**:

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # Межзадачные циклы в CI отклоняются напрямую
```

| Тип цикла | Поведение | Причина |
|-----------|-----------|---------|
| Цикл ref внутри задачи | Не проверяется | Право пользователя, при завершении задачи всё освобождается |
| Цикл ref между задачами | lint (по умолчанию warn) | Напоминание переосмыслить, можно настроить deny |

#### 3.4 Weak: предоставляется стандартной библиотекой

```yaoxiang
use std.rc.Weak

# Продвинутые пользователи явно выбирают
a.next = ref b
b.prev = Weak.new(a.next)        # Пользователь явно контролирует, какое направление слабое
```

**`Weak` не встроен в язык, это тип стандартной библиотеки.** Для повседневного использования хватает `ref`. Продвинутые пользователи, которым нужна тонкая настройка памяти, вручную используют `Weak`.

#### 3.5 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Что делает | Посмотреть/изменить на месте | Разделяемое владение |
| Область действия | По области видимости значения токена | Между областями видимости |
| Стоимость | Нулевые накладные расходы (типы нулевого размера) | Rc или Arc (выбирает компилятор) |
| Побег | Можно (токен распространяется с возвращаемым значением/структурой/замыканием) | По природе предназначен для побега |
| Межзадачность | Нельзя (токен — доказательство прав времени компиляции, нельзя передавать между задачами) | Можно (компилятор автоматически выбирает Arc) |
| Циклы | Не применимо | Внутри задачи бездействие, между задачами lint |

### 4. clone() —— Явное копирование

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # Глубокое копирование
# p и p2 независимы, не влияют друг на друга
```

**Когда использовать**: когда нужно сохранить оригинальное значение и не подходят Move или разделение.

### 5. unsafe + голые указатели (системное программирование)

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # Голый указатель
    (*ptr).x = 0.0                # Разыменование (пользователь гарантирует безопасность)
    ptr2 = ptr + 1                # Арифметика указателей
}
```

**Ограничения**:
- Можно использовать только в блоках `unsafe`
- Пользователь гарантирует отсутствие висячих указателей и использования после освобождения
- Для FFI, операций с памятью и прочего системного программирования

### 6. Общий обзор градиента владения

```
  Токены заимствования (нулевые накладные расходы)    Move (нулевые накладные расходы)    Разделение (оплата по факту)    Копирование
   │                                                       │                              │                         │
  &T Копируемый токен                                  Захват по умолчанию            ref Rc/Arc                clone()
  &mut T Линейный токен                               Связанное потребление          Компилятор автоматически    Явное глубокое
   │                                                       │                              выбирает                   копирование
   │                                                       │                              │                         │
  Область видимости                                    Внутри области видимости        Между областями видимости    В любой момент
  значения токена                                                                          │
  Можно возвращать/сохранять в структурах             T -> T возврат              ref между задачами → Arc
  Можно захватывать в замыканиях                      T -> Void потребление       ref внутри задачи → Rc
  Нулевой размер, исчезает после компиляции                                        Циклы внутри задачи бездействие
                                                                                      Циклы между задачами lint
                                                                                      Weak стандартной библиотеки как аварийный выход
```

## Комплексный пример

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T: токен только для чтения
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T: изменяемый токен
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move: потребление с возвратом
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # Забрал, изменил, вернул
    }

    # Возврат ссылки: токен распространяется с возвращаемым значением
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# Захват токена в замыкании (возможность, которую v8 не имел)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}

# Комплексное использование
p = Point(1.0, 2.0)
p.print()                           # Токен &Point
p.shift(1.0, 1.0)                   # Токен &mut Point
p = p.scale(2.0)                    # Move → возврат
shared = ref p                      # ref разделяемое владение
spawn { use(shared) }

# clone независимая копия
backup = p.clone()

# Цикл внутри задачи: бездействие
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # Цикл, при завершении задачи всё освобождается

# unsafe системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## Ограничения системы типов

### Атрибут типа Dup

`Dup` (Duplicable, можно неявно копировать) — ключевой атрибут типа, автоматически управляемый компилятором. Типы, реализующие `Dup`, при присваивании не передают владение, а неявно делают мелкое копирование. `Dup` влечёт `Clone` — если тип является `Dup`, его всегда можно клонировать поле за поле; но `Clone` не влечёт `Dup`.

| Тип | Dup | Clone | Описание |
|------|-----|-------|----------|
| Int, Float, Bool, Char, String, Bytes | ✅ (автоматически реализуется компилятором) | ✅ | Базовые типы |
| `&T` | ✅ (нулевой размер, свободное создание псевдонимов) | ✅ | Токен только для чтения |
| `&mut T` | ❌ (линейный, эксклюзивный) | ❌ | Изменяемый токен |
| struct | ✅ (автоматически выводится, когда все поля Dup) | ✅ | Структура |
| `ref T` | ✅ | ✅ | Разделяемое владение (компилятор автоматически выбирает Rc/Arc) |
| `*T` | ❌ | ❌ | Голый указатель |

> **Примечание**: В YaoXiang нет видимых пользователю трэйтов `Send`/`Sync`. Ключевое слово `ref` заставляет компилятор автоматически выбирать Rc (не пересекает задачи) или Arc (пересекает задачи), пользователю не нужно заботиться о том, можно ли тип передавать между задачами. Невозможность передачи токенов через границы задач — встроенное правило компилятора, а не ограничение трэйта.

## Анализ производительности

| Операция | Стоимость | Описание |
|----------|-----------|----------|
| Move | Ноль | Перемещение указателя |
| `&T` / `&mut T` | Ноль | Типы нулевого размера, исчезают после компиляции, нулевые накладные расходы во время выполнения |
| `ref` (не межзадачность) | Низкая | Компилируется в Rc, неатомарные операции |
| `ref` (межзадачность) | Средняя | Компилируется в Arc, атомарные операции |
| `clone()` | Зависит от типа | Маленькие объекты быстро, большие медленно |
| `unsafe + *T` | Ноль | Прямые операции с памятью |

### Сравнение

| Язык | Механизм разделения | Управление памятью | Обработка циклов | Сложность |
|------|----------------------|--------------------|-------------------|-----------|
| Rust | Arc / Mutex + проверка заимствований | Проверка времени компиляции | Ручной Weak | Высокая |
| Go | chan / pointer | Сборщик мусора | Сборщик мусора | Низкая |
| C++ | shared_ptr | RAII | weak_ptr | Средняя |
| **YaoXiang** | **ref + токены заимствования** | **RAII** | **Освобождение на границе задач / lint между задачами / Weak стандартной библиотеки** | **Низкая** |

## Компромиссы

### Преимущества

1. **Единство**: `&T`/`&mut T` — обычные типы, не специальная языковая конструкция. Полностью согласовано с `name: type = value` из RFC-010
2. **Простота**: без времени жизни, без глобального проверяльщика заимствований. `&T` копируемый, `&mut T` не копируемый — два атрибута типа
3. **Выразительность**: можно возвращать ссылки, сохранять в структуры, захватывать в замыкания — выразительность на уровне Rust
4. **Умный компилятор**: ref автоматически выбирает Rc/Arc, на стороне вызова автоматический выбор заимствования
5. **Детерминированность**: ref просто живёт, не становится тихо слабой ссылкой
6. **Высокая производительность**: Move с нулевым копированием, токены с нулевыми накладными расходами (типы нулевого размера, исчезают после компиляции)
7. **Гибкость**: `unsafe + *T` поддерживает системное программирование

### Недостатки

1. **Заражение обобщёнными параметрами брендов**: токены несут идентификаторы брендов, сигнатуры функций с возвратом ссылок будут содержать дополнительные обобщённые параметры
2. **Накладные расходы ref во время выполнения**: атомарные операции имеют стоимость (но это неизбежная цена разделения)
3. **Риски unsafe**: пользователь должен гарантировать корректность
4. **Межзадачные циклы — lint, а не ошибка компиляции**:，不像 Rust那样编译报错，默认 warn，需要团队配置 deny才能作为质量门

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Сборщик мусора | Накладные расходы времени выполнения, непредсказуемые паузы |
| Проверяльщик заимствований Rust | Требуется время жизни `'a`, крутая кривая обучения |
| Только Move | Не справляется с параллельным разделением |
| Без голых указателей | Невозможно системное программирование |
| Предоставление Rc/Arc пользователю | Детали реализации перекладываются на пользователя, увеличивается когнитивная нагрузка |
| Упрощённое заимствование (v8) | Стратегия запрета побега лишает ключевых возможностей: захват в замыканиях, возврат ссылок и т.д. |

### Обнаружение конфликтов токенов: потокочувствительный анализ активности

**Область анализа**: только тело функции. Потокочувствительный анализ активности значений токенов, отслеживание состояния каждого токена (активен/заморожен/перемещён).

**Уровень 1: Проверка в точке вызова** — один фактический аргумент не может одновременно создавать токен `&mut` и другие токены:

```yaoxiang
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)    # ❌ p одновременно порождает токены &mut и &, компилятор отклоняет
```

**Уровень 2: Потокочувствительный внутри тела функции** — после передачи токена `&mut` вызову, при возврате вызова токен освобождается, далее можно снова создавать токены:

```yaoxiang
process_twice: (p: &mut Point) -> Void = {
    shift(p, 1.0, 1.0)    # Токен &mut передан shift, после возврата shift токен освобождён
    print_info(p)          # Снова создаётся токен &Point, конфликта нет
}
```

**Уровень 3: Отслеживание состояния заморозки** — пока активен токен `&T`, полученный через `freeze`, оригинальный токен `&mut` недоступен:

```yaoxiang
frozen: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)    # p переходит в замороженное состояние
    print(view.x)
    p.x = 10.0                  # ❌ Ошибка компиляции: WriteToken в замороженном состоянии
}
```

**Что не нужно**: межфункциональное отслеживание времени жизни, глобальный анализ псевдонимов, решение ограничений borrow graph, NLL, аннотации `'a`. Потому что токены — значения, анализ активности значений единообразно обрабатывается проверкой типов — точно так же, как отслеживание любых значений линейного типа.
