# RFC-009: Проектирование модели владения

## Аннотация

В данном документе определяется **модель владения (Ownership Model)** языка программирования YaoXiang.

**Основной дизайн — пять концепций, один градиент**:

```
Посмотреть/Изменить    Забрать         Общее владение       Скопировать        Системный
    │                    │                 │                    │                 │
   &T                  Move             ref                clone()           unsafe
  &mut T            Нуль-копирование    Компилятор          Явное             *T
  Токен              По умолчанию       автоматически       глубокое
  нулевого            RAII              выбирает            копирование
  размера             автоматическое     Rc/Arc
  Тип атрибута                            Пользователь
  естественно                            отвечает
  выводит
  разрешения
```

- **Move (по умолчанию)**: присваивание/передача параметров/возврат = передача владения, нуль-копирование, автоматическое освобождение через RAII
- **`&T` / `&mut T` (токены заимствования)**: токены типов времени компиляции нулевого размера. `&T` копируемый (общее чтение), `&mut T` линейный (монопольная запись). Разрешения естественно выводятся из атрибутов типа, без специальных правил. Можно возвращать, сохранять в структуры, захватывать замыканиями.
- **`ref` ключевое слово**: обмен данными между областями видимости. Компилятор автоматически выбирает Rc (не межзадачный) или Arc (межзадачный)
- **`clone()`**: явное глубокое копирование
- **`unsafe` + `*T`**: голые указатели, системный аварийный выход

**Устранённая сложность**:
- ❌ Нет времени жизни `'a`
- ❌ Нет проверки заимствований (атрибуты типа токенов — Dup/Linear — заменяют специализированный проверяльщик заимствований)
- ❌ Нет GC
- ❌ Нет специальных правил вроде «запрета побега» (токены — обычные типы, области видимости обрабатываются единой системой типов)
- ❌ Пользователю не нужно знать разницу между Rc/Arc (компилятор выбирает автоматически)

> **Нагрузка на программиста**: `&T` копируемый, `&mut T` некопируемый — два атрибута типа, ноль специальных правил, полная автоматизация компилятора.
> **Гарантия производительности**: Move с нулевыми накладными расходами, токены с нулевыми накладными расходами (типы нулевого размера, исчезают после компиляции), ref с оплатой по потреблению, без пауз GC.

## Мотивация

### Зачем нужна модель владения?

| Язык | Управление памятью | Проблемы |
|------|-------------------|----------|
| C/C++ | Ручное | Утечки памяти, висячие указатели, двойное освобождение |
| Java/Python | GC | Задержки, накладные расходы памяти, непредсказуемые паузы |
| Rust | Владение + проверка заимствований | Крутая кривая изучения времени жизни `'a` |
| **YaoXiang** | **Move + Token + ref** | **Просто, определённо, без GC** |

### Цели проектирования

```yaoxiang
# 1. Move по умолчанию (нуль-копирование)
p = Point(1.0, 2.0)
p2 = p                         # Move, p больше не читается

# 2. &T / &mut T токены заимствования (нулевые накладные, атрибуты типа естественно выводят разрешения)
print_info(p2)                 # Компилятор автоматически создаёт токен &Point, освобождает после использования
shift(p2, 1.0, 1.0)           # Компилятор автоматически создаёт токен &mut Point

# 3. ref = обмен данными (компилятор автоматически выбирает Rc/Arc)
shared = ref p2                # Владение между областями видимости
spawn { use(shared) }          # Компилятор: межзадачный → Arc

# 4. clone() = явное копирование
backup = p2.clone()            # Глубокое копирование, единоличное

# 5. unsafe + *T = системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Ключевые отличия от Rust

| Характеристика | Rust | YaoXiang |
|------|------|----------|
| Семантика по умолчанию | Заимствование `&T` (нужно явное `.clone()`) | **Move (передача значений, нуль-копирование)** |
| Заимствование | `&T`/`&mut T`, может возвращаться, нужны времена жизни | **`&T`/`&mut T` токены нулевого размера, Dup/Linear атрибуты типа естественно выводят** |
| Механизм обмена | `Arc::new()` + ручной Weak | **`ref` ключевое слово (компилятор автоматически выбирает Rc/Arc)** |
| Копирование | `clone()` | `clone()` |
| Голые указатели | `*T` | `*T` |
| Время жизни | `'a` | ❌ Нет |
| Проверка заимствований | Глобальный вывод | **Потоково-чувствительный анализ активности (отслеживание состояния токенов)** |
| Циклические ссылки | Ручной Weak | **Освобождение по окончании задачи / lint межзадачных / Weak в стандартной библиотеке** |

## Предложение

### 1. Move (передача владения по умолчанию)

```yaoxiang
# Правило: присваивание / передача параметров / возврат = Move, нуль-копирование

p: Point = Point(1.0, 2.0)
p2 = p                           # Move, p больше не читается

# Переменные можно переприсваивать (стиль Python, без затенения)
p = Point(3.0, 4.0)              # p перепривязывается, тип должен совпадать

# Параметры функции: Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move возврат
}

# Возврат из функции: Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move возврат, нуль-копирование
}
```

**Характеристики**:
- Нуль-копирование (компилятор перемещает указатель)
- После перемещения оригинальная привязка недоступна для чтения (ошибка компиляции)
- RAII: автоматическое освобождение при выходе из области видимости
- Сигнатура функции `(T) -> T` сама по себе документация — потребляет T, возвращает T

### 2. &T / &mut T (токены заимствования)

**Основной принцип: `&T` и `&mut T` — это типы токенов времени компиляции нулевого размера. Это не «ссылки», а «типизированные доказательства разрешений доступа».**

#### 2.1 Два атрибута типа

```
&T      →  Нулевой размер, копируемый (Dup), даёт разрешение только на чтение
&mut T  →  Нулевой размер, линейный (не-Dup), даёт монопольное разрешение на чтение-запись
```

**Это не «правило», которое нужно запоминать — это базовые атрибуты системы типов.** Типы Dup могут свободно копироваться (несколько `&T` сосуществуют), линейные типы не копируются (`&mut T` по природе уникален). Нет «проверяльщика заимствований» — только проверяльщик типов делает то, что он всегда делал.

#### 2.2 Базовое использование

```yaoxiang
# Метод-сторона: объявление типа параметра, определение нужных разрешений
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # Токен &Point даёт разрешение на чтение
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # Токен &mut Point даёт разрешение на запись
    self.y = self.y + dy
}

# Сторона вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                          # Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)                  # Компилятор автоматически создаёт токен &mut Point
p.print()                          # OK, предыдущий токен освобождён после завершения вызова shift

# Свободные функции аналогично
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # Два токена &Point сосуществуют — тип Dup
}
d = distance(p, p2)
```

#### 2.3 Почему не нужно «запрещать побег»

RFC-009 v8 накладывал на `&T`/`&mut T` три специальных правила — только параметры, нельзя возвращать, нельзя сохранять в структуры. Это патчи для концепции «заимствования».

Система токенов не нуждается в этих правилах. Токены — **обычные типы**, подчиняющиеся тем же правилам области видимости, что и все остальные типы.

**Возврат ссылок — естественно поддерживается**:

```yaoxiang
# ✅ Токен распространяется вместе с возвращаемым значением
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # Дочерний и родительский токены возвращаются вместе
}

# Использование
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # Токен возвращается вызывающему
print(px_ref)               # OK, токен всё ещё в области видимости
```

**Сохранение в структуры — естественно поддерживается**:

```yaoxiang
# ✅ Структура переносит токен как поле
Window: Type = {
    target: Point,
    view: &Point,      # Поле токена — держит представление только для чтения target
}

# Токен view производён от target, Window владеет обоими
# Пока Window существует, токен view валиден
```

**Захват замыканиями — естественно поддерживается**:

```yaoxiang
# ✅ Замыкание захватывает токен, как любую величину
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    # Замыкание захватывает токен &Float threshold (тип Dup, свободно копируется в замыкание)
    items.filter(|p| p.x > threshold)
}

# Это то, что RFC-009 v8 не мог — v8 запрещал захват заимствований замыканиями
```

**Межзадачность — токены не проходят через потоки**:

```yaoxiang
# ❌ Токены не могут пересекать границы задач
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ Ошибка компиляции: токены нельзя передавать между задачами
}

# Это не специальное правило — токены это доказательства разрешений времени компиляции,
# для межзадачного обмена используйте ref
```

**Токены нельзя оборачивать в ref**:

```yaoxiang
# ❌ Токены это доказательства разрешений, не владение
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ Ошибка компиляции: &T не тип для владения
}
```

#### 2.4 Жизненный цикл токенов

Жизненный цикл токенов определяется **обычными правилами области видимости**, без параметров времени жизни:

- Токены в параметрах функции: живут во время вызова, освобождаются после завершения вызова
- Возвращаемые токены: владение передаётся вызывающему
- Токены, хранящиеся в структурах: живут вместе со структурой
- Захваченные замыканиями токены: живут вместе с замыканием

Компилятору не нужны аннотации `'a`, потому что токены — **значения**, жизненный цикл значений управляется системой владения (Move/RAII). **Снижение размерности проблемы заимствования до проблемы владения.**

#### 2.5 Механизм заморозки

Токен `&mut T` можно временно «заморозить» для получения токена `&T`:

```yaoxiang
modify_and_read: (p: &mut Point) -> Void = {
    p.x = 10.0                      # Используем &mut Point для модификации
    
    # Заморозить &mut, получить представление только для чтения
    view: &Point = freeze(p)         # p заморожен здесь
    print(view.x)                   # Читаем через &Point
    print(view.y)
    # view выходит из области видимости, заморозка снята
    
    p.y = 20.0                      # &mut Point снова доступен
}
```

Семантика `freeze`:
- Принимает `&mut T`, возвращает `&T`
- Пока `&T` жив, оригинальный `&mut T` недоступен
- После выхода `&T` из области видимости, `&mut T` автоматически восстанавливается
- Это **потоково-чувствительный анализ** — компилятор отслеживает состояние токена внутри тела функции

#### 2.6 Обнаружение конфликтов токенов

Замена «проверки перекрёстного заимствования» из RFC-009 v8. Принцип проще — **потоково-чувствительный анализ активности** значений токенов:

```yaoxiang
# ❌ &mut и производный &T не могут быть активны одновременно
bad_alias: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p заморожен
    p.x = 10.0                      # ❌ Ошибка компиляции: WriteToken в замороженном состоянии
    print(view.x)                   
}

# ✅ После размораживания можно продолжать использовать &mut
good_seq: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)        # p заморожен
    print(view.x)                   # Используем &T
    # view выходит из области видимости
    p.x = 10.0                      # ✅ WriteToken восстановлен
}
```

**Метод обнаружения**: Это не специализированный «проверяльщик заимствований» — это **потоково-чувствительный анализ активности** значений токенов. Компилятор отслеживает состояние каждого токена внутри тела функции (активен/заморожен/перемещён), точно так же, как отслеживает любые значения линейного типа.

#### 2.7 Внутренности компилятора: механизм брендов

Пользователь никогда не видит бренды. Компилятор внутренне назначает каждому токену уникальный идентификатор времени компиляции:

```
Что видит пользователь    Внутреннее представление компилятора
─────────────────────────────────────────────────────────────
&Point                  →  ReadToken(Point, #N)    // #N — уникальное целое время компиляции
&mut Point              →  WriteToken(Point, #M)   // #M — уникальное целое время компиляции
```

Назначение брендов:
- **Защита от подделки**: токены можно получить только от владеющего типа или операции freeze, нельзя создать из ничего
- **Отслеживание происхождения**: при получении `&Float` из `&Point` (доступ к полю), `&Float` несёт производный бренд (`#N.field_x`), компилятор может отследить до родительского токена
- **Обнаружение конфликтов**: одноимённые `WriteToken` и производный `ReadToken` не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнга, в сгенерированном машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

#### 2.8 Правила автоматического выбора заимствования

Компилятор на стороне вызова автоматически выбирает по следующему приоритету:

```
1. Если фактический аргумент ещё используется далее → приоритет создания токена (&T или &mut T, по сигнатуре метода)
2. Если фактический аргумент далее не используется → Move
3. Приоритет匹配顺序：&T < &mut T < Move
```

```yaoxiang
# Пример: автоматический выбор
p = Point(1.0, 2.0)
p.print()        # print объявляет &self → компилятор создаёт токен &Point
p.shift(1.0, 1.0) # shift объявляет &mut self → компилятор создаёт токен &mut Point
p2 = p           # Move, p больше не используется
```

#### 2.9 Сравнение с упрощённым заимствованием RFC-009 v8

| Характеристика | Упрощённое заимствование (v8) | Токены заимствования (v9) |
|------|--------------|--------------|
| Возврат ссылок | ❌ Жёстко запрещён | ✅ Токен распространяется с возвращаемым значением |
| Сохранение в структуры | ❌ Жёстко запрещено | ✅ Токен как поле структуры |
| Захват замыканиями | ❌ Жёстко запрещён | ✅ Замыкание захватывает значение токена |
| Специальные правила | 3 (только параметры/нельзя возвращать/нельзя сохранять) | 0 — атрибуты типа естественно выводят |
| Проверка заимствований | Специализированная проверка перекрёстного заимствования | Потоково-чувствительный анализ активности проверяльщика типов |
| Аннотации времени жизни | Не нужны | Не нужны |
| Накладные расходы выполнения | Ноль | Ноль (типы нулевого размера, исчезают после компиляции) |
| Сообщения об ошибках | «Заимствование не может покидать область» | «WriteToken(#3) заморожен» (обычная ошибка типа) |
| Ментальная модель пользователя | Понимание особого статуса «заимствования» | `&T` копируемый, `&mut T` некопируемый |

### 3. ref ключевое слово (автооптимизация компилятора)

`ref` — единственный способ межобластного обмена данными. Что внизу — Rc или Arc, пользователю знать не нужно.

#### 3.1 Базовое использование

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # Общий доступ, компилятор автоматически выбирает реализацию

# Межзадачный обмен
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # Компилятор: межзадачный → Arc
    spawn { use(data) }           # Компилятор: межзадачный → Arc
}

# Однозадачный обмен
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # Компилятор: не межзадачный → Rc
}
```

**Ментальная модель пользователя**: `ref` = общее владение. Вот и всё.

#### 3.2 Анализ побега компилятора: Rc vs Arc

```
Анализ потока данных ref:

Не сбегает в другие задачи → Rc (неатомарный подсчёт ссылок, низкие накладные)
Сбегает в другие задачи   → Arc (атомарный подсчёт ссылок, потокобезопасность)
```

#### 3.3 Стратегия обнаружения циклов

```
Циклы внутри задачи → Тихо разрешены.
  ├── Структурированный параллелизм гарантирует的统一ное освобождение всех ресурсов по окончании задачи.
  ├── ref всегда живёт, семантика без разбавления.
  └── Пользователь имеет право строить двунаправленные сильные ссылки внутри задачи (например, промежуточные состояния графовых вычислений).

Циклы между задачами → lint (по умолчанию warn, настраиваемо).
  ├── Программа работает корректно, реальных утечек не будет (при завершении родительской задачи все ресурсы дочерних освобождаются).
  ├── Но межзадачные сильные ссылки означают размытые границы владения, стоит остановиться и подумать заново.
  ├── По умолчанию warn, компиляция проходит с подсказкой.
  └── Команда может установить deny в конфигурации проекта, включить в CI quality gate.
```

**Уровни lint** (аналогично Rust clippy):

| Уровень | Поведение | Сценарий |
|------|------|------|
| `allow` | Не проверять | Личные проекты |
| `warn` (по умолчанию) | Компиляция проходит, есть подсказка | Этап разработки |
| `deny` | Компиляция не удаётся | Командный CI quality gate |
| `forbid` | Компиляция не удаётся, нельзя переопределить | Организационные обязательные правила |

```yaoxiang
# Циклы внутри задачи: тихо разрешены, двунаправленные сильные ссылки
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # Цикл. Освобождается统一но при завершении задачи.
}

# Циклы между задачами: lint (по умолчанию warn)
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: межзадачный циклический ссылка
    }
}
```

**Пример конфигурации проекта**:

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # Межзадачные циклы отклоняются на CI
```

| Тип цикла | Поведение | Причина |
|--------|------|------|
| Цикл ref внутри задачи | Не проверяется | Право пользователя, освобождение по окончании задачи |
| Цикл ref между задачами | lint (по умолчанию warn) | Напоминание переосмыслить, можно настроить deny |

#### 3.4 Weak: предоставляется стандартной библиотекой

```yaoxiang
use std.rc.Weak

# Продвинутый пользователь явно выбирает
a.next = ref b
b.prev = Weak.new(a.next)        # Пользователь явно контролирует, какая сторона слабая
```

**`Weak` не встроен в язык, это тип стандартной библиотеки.** Повседневно хватает `ref`. Продвинутые пользователи, которым нужна тонкая настройка памяти, вручную вводят `Weak`.

#### 3.5 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Что делает | Посмотреть/Изменить на месте | Общее владение |
| Область | Область видимости значения токена | Межобластная |
| Стоимость | Нулевые накладные (типы нулевого размера) | Rc или Arc (выбирает компилятор) |
| Побег | Можно (токен распространяется с возвратом/структурой/замыканием) | По природе для побега |
| Межзадачность | Нельзя (токен — доказательство разрешения времени компиляции, нельзя передавать между задачами) | Можно (компилятор автоматически выбирает Arc) |
| Циклы | Не применимо | Внутри задачи тихо, между задачами lint |

### 4. clone() —— Явное копирование

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # Глубокое копирование
# p и p2 независимы, не влияют друг на друга
```

**Когда использовать**: когда нужно сохранить оригинальное значение и сценарий не подходит для Move или общего доступа.

### 5. unsafe + голые указатели (системное программирование)

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # Голый указатель
    (*ptr).x = 0.0                # Разыменование (пользователь гарантирует безопасность)
    ptr2 = ptr + 1                # Арифметика указателей
}
```

**Ограничения**:
- Можно использовать только в блоках `unsafe`
- Пользователь гарантирует отсутствие висячих указателей и использования после освобождения
- Для FFI, операций с памятью и другого системного программирования

## Комплексный пример

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T: токен только для чтения
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T: изменяемый токен
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move: потребление с возвратом
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # Забрать, изменить, вернуть
    }

    # Возврат ссылки: токен распространяется с возвращаемым значением
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# Захват токена замыканием (способность, которой v8 не обладал)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}

# Комплексное использование
p = Point(1.0, 2.0)
p.print()                           # Токен &Point
p.shift(1.0, 1.0)                   # Токен &mut Point
p = p.scale(2.0)                    # Move → возврат
shared = ref p                      # ref общий доступ
spawn { use(shared) }

# clone независимая копия
backup = p.clone()

# Цикл внутри задачи: тихо разрешён
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # Цикл, освобождается统一но при завершении задачи

# unsafe системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## Ограничения системы типов

### Атрибут типа Dup

`Dup` (Дублируемый) — это атрибут типа, автоматически управляемый компилятором, означающий **поверхностное копирование**: при присваивании/передаче параметров копируется дескриптор/токен, базовые данные общие. Это отличается от Move (передача владения) и Clone (явное глубокое копирование, создание независимой копии) — три уровня градиента.

**Dup и Clone — ортогональные концепции** — Dup копирует дескриптор с общим доступом к данным, Clone создаёт независимую копию. Тип может одновременно поддерживать и Dup, и Clone, или только один из них.

| Тип | Dup | Clone | Пояснение |
|------|-----|-------|--------|
| `&T` | ✅ (копирование токена, несколько представлений указывают на одни данные) | ✅ | Токен только для чтения |
| `ref T` | ✅ (счётчик ссылок+1, общие кучевые данные) | ✅ | Общее владение (компилятор автоматически выбирает Rc/Arc) |
| String, Bytes | ✅ (внутренний счётчик ссылок, копирование дескриптора с общим базовым буфером) | ✅ | Строки/байты |
| `&mut T` | ❌ (линейный, монопольный) | ❌ | Изменяемый токен |
| `*T` | ❌ | ❌ | Голый указатель |
| struct | Производный (автоматически производный, когда все поля Dup) | ✅ | Структура |

**Примитивные типы значений** (Int, Float, Bool, Char) при присваивании ведут себя как встроенное копирование значений компилятором — два полностью независимых значения, не поверхностное копирование. Они не принадлежат атрибуту типа Dup, а являются родной обработкой компилятора.

## Анализ производительности

| Операция | Стоимость | Пояснение |
|------|------|------|
| Move | Ноль | Перемещение указателя |
| `&T` / `&mut T` | Ноль | Типы нулевого размера, исчезают после компиляции, нулевые накладные выполнения |
| `ref` (не межзадачный)| Низкая | Компилируется в Rc, неатомарные операции |
| `ref` (межзадачный)| Средняя | Компилируется в Arc, атомарные операции |
| `clone()` | Зависит от типа | Маленькие объекты быстро, большие медленно |
| `unsafe + *T` | Ноль | Прямые операции с памятью |

### Сравнение

| Язык | Механизм общего доступа | Управление памятью | Обработка циклов | Сложность |
|------|----------|----------|----------|--------|
| Rust | Arc / Mutex + проверка заимствований | Время компиляции | Ручной Weak | Высокая |
| Go | chan / pointer | GC | GC | Низкая |
| C++ | shared_ptr | RAII | weak_ptr | Средняя |
| **YaoXiang** | **ref + токены заимствования** | **RAII** | **Освобождение по границе задач / lint межзадачных / Weak стандартной библиотеки** | **Низкая** |

## Компромиссы

### Преимущества

1. **Единство**: `&T`/`&mut T` — обычные типы, не специальная языковая конструкция. Полностью согласовано с `name: type = value` из RFC-010
2. **Простота**: Без времени жизни, без глобального проверяльщика заимствований. `&T` копируемый, `&mut T` некопируемый — два атрибута типа
3. **Мощность**: Можно возвращать ссылки, сохранять в структуры, захватывать замыканиями — выразительность на уровне Rust
4. **Интеллект компилятора**: ref автоматически выбирает Rc/Arc, сторона вызова автоматически выбирает заимствование
5. **Определённость**: ref просто живёт, не становится тихо Weak-ссылкой
6. **Высокая производительность**: Move нуль-копирование, токены нулевые накладные (типы нулевого размера, исчезают после компиляции)
7. **Гибкость**: `unsafe + *T` поддерживает системное программирование

### Недостатки

1. **Заражение обобщённых параметров брендов**: токены несут идентификаторы брендов, в сигнатурах функций с возвратом ссылок появляются дополнительные обобщённые параметры
2. **Накладные выполнения ref**: у атомарных операций есть цена (но это неизбежная плата за общий доступ)
3. **Риски unsafe**: пользователь должен гарантировать корректность
4. **Межзадачные циклы — lint, не ошибка компиляции**:，不像 Rust那样编译报错，默认 warn, 需要团队配置 deny才能作为质量门

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| GC | Накладные выполнения, непредсказуемые паузы |
| Проверяльщик заимствований Rust | Нужны времена жизни `'a`, крутая кривая изучения |
| Чистый Move | Не справляется с параллельным общим доступом |
| Без голых указателей | Невозможно системное программирование |
| Открыть Rc/Arc пользователю | Детали реализации сваливаются на пользователя,增加的认知负担 |
| Упрощённое заимствование (v8) | Стратегия запрета побега sacrifice ключевые возможности вроде захвата замыканиями, возврата ссылок |

### Обнаружение конфликтов токенов: потоково-чувствительный анализ активности

**Область анализа**: только тело функции. Потоково-чувствительный анализ активности значений токенов, отслеживание состояния каждого токена (активен/заморожен/перемещён).

**Уровень 1: Проверка в точке вызова** — один фактический аргумент не может одновременно создавать токены `&mut` и другие токены:

```yaoxiang
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)    # ❌ p одновременно создаёт токены &mut и &, компилятор отклоняет
```

**Уровень 2: Потоковая чувствительность тела функции** — после передачи токена `&mut` вызову, после возврата вызова токен освобождается, можно снова создавать токены:

```yaoxiang
process_twice: (p: &mut Point) -> Void = {
    shift(p, 1.0, 1.0)    # Токен &mut передан shift, после возврата shift токен освобождён
    print_info(p)          # Заново создаём токен &Point, конфликта нет
}
```

**Уровень 3: Отслеживание состояния заморозки** — пока активен токен `&T`, произведённый `freeze`, оригинальный токен `&mut` недоступен:

```yaoxiang
frozen: (p: &mut Point) -> Void = {
    view: &Point = freeze(p)    # p входит в замороженное состояние
    print(view.x)
    p.x = 10.0                  # ❌ Ошибка компиляции: WriteToken в замороженном состоянии
}
```

**Что не нужно**: межфункциональное отслеживание времени жизни, глобальный анализ алиасов, ограничения solvingborrow graph, NLL, аннотации `'a`. Потому что токены — значения, анализ активности значений единообразно обрабатывается проверяльщиком типов — точно так же, как отслеживание любых значений линейного типа.
