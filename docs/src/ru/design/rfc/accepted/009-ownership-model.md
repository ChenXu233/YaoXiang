# RFC-009: Проектирование модели владения

## Аннотация

В данном документе определяется **модель владения (Ownership Model)** языка программирования YaoXiang.

**Основной дизайн — пять концепций, один градиент**:

```
Взглянуть/Изменить    Забрать         Общее владение        Скопировать          Системный уровень
     │                   │                 │                    │                    │
    &T                 Move             ref              clone()              unsafe
   &mut T            Нулевое          Компилятор          Явное             *T
   Токен              копирование       автоматически      глубокое          Пользователь
   нулевого            по умолчанию      выбирает Rc/Arc    копирование        отвечает
   размера                                                    
   Тип атрибута                                      
   естественно                                          
   выводит                                              
   права                                              
```

- **Move (по умолчанию)**: присваивание/передача параметров/возврат = передача владения, нулевое копирование, автоматическое освобождение через RAII
- **`&T` / `&mut T` (токены заимствования)**: типы токенов времени компиляции нулевого размера. `&T` копируемый (общее чтение), `&mut T` линейный (исключительная запись). Права естественно выводятся из атрибутов типа, без специальных правил. Можно возвращать, сохранять в структурах, захватывать в замыканиях.
- **`ref` ключевое слово**: общее владение между областями видимости. Компилятор автоматически выбирает Rc (не пересекает задачи) или Arc (пересекает задачи)
- **`clone()`**: явное глубокое копирование
- **`unsafe` + `*T`**: голые указатели, системный уровень «аварийный выход»

**Устранённая сложность**:
- ❌ Нет времени жизни `'a`
- ❌ Нет проверки заимствований (атрибуты типа токена — Dup/Linear — заменяют специализированный borrow checker)
- ❌ Нет GC
- ❌ Нет специальных правил типа «запрет на побег» (токены — обычные типы, области видимости обрабатываются единой системой типов)
- ❌ Пользователю не нужно знать разницу между Rc и Arc (компилятор выбирает автоматически)

> **Нагрузка на программирование**: `&T` копируемый, `&mut T` не копируемый — два атрибута типа, ноль специальных правил, полная автоматизация компилятором.
> **Гарантия производительности**: Move с нулевыми накладными расходами, токены с нулевыми накладными расходами (типы нулевого размера, исчезают после компиляции), ref платит по мере необходимости, нет пауз GC.

## Мотивация

### Зачем нужна модель владения?

| Язык | Управление памятью | Проблемы |
|------|--------------------|----------|
| C/C++ | Ручное управление | Утечки памяти, висячие указатели, двойное освобождение |
| Java/Python | GC | Задержки, накладные расходы памяти, непредсказуемые паузы |
| Rust | Владение + проверка заимствований | Крутая кривая обучения времени жизни `'a` |
| **YaoXiang** | **Move + Token + ref** | **Просто, детерминировано, без GC** |

### Цели проектирования

```yaoxiang
# 1. Move по умолчанию (нулевое копирование)
p = Point(1.0, 2.0)
p2 = p                         # Move, p больше нельзя читать

# 2. &T / &mut T токены заимствования (нулевые накладные расходы, атрибуты типа естественно выводят права)
print_info(p2)                 # Компилятор автоматически создаёт токен &Point, освобождает после использования
shift(p2, 1.0, 1.0)           # Компилятор автоматически создаёт токен &mut Point

# 3. ref = общее владение (компилятор автоматически выбирает Rc/Arc)
shared = ref p2                # Владение через область видимости
spawn { use(shared) }          # Компилятор: пересекает задачи → Arc

# 4. clone() = явное копирование
backup = p2.clone()            # Глубокое копирование, собственное

# 5. unsafe + *T = системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Ключевые отличия от Rust

| Характеристика | Rust | YaoXiang |
|----------------|------|----------|
| Семантика по умолчанию | Заимствование `&T` (нужно явное `.clone()`) | **Move (передача значений, нулевое копирование)** |
| Заимствование | `&T`/`&mut T`, можно возвращать, нужно время жизни | **`&T`/`&mut T` токены нулевого размера, атрибуты типа Dup/Linear естественно выводят** |
| Механизм совместного использования | `Arc::new()` + ручной Weak | **`ref` ключевое слово (компилятор автоматически выбирает Rc/Arc)** |
| Копирование | `clone()` | `clone()` |
| Голые указатели | `*T` | `*T` |
| Время жизни | `'a` | ❌ Нет |
| Проверка заимствований | Глобальный вывод | **Потоково-чувствительный анализ активности (отслеживание состояния токенов)** |
| Циклические ссылки | Ручной Weak | **Освобождение по границе задач / lint для межзадачных / стандартная библиотека Weak** |

## Предложение

### 1. Move (передача владения по умолчанию)

```yaoxiang
# Правило: присваивание / передача параметров / возврат = Move, нулевое копирование

p: Point = Point(1.0, 2.0)
p2 = p                           # Move, p больше нельзя читать

# Переменным можно переприсваивать (стиль Python, без затенения)
p = Point(3.0, 4.0)              # Перепривязка p, тип должен совпадать

# Параметры функции: Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move возврат
}

# Возврат из функции: Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move возврат, нулевое копирование
}
```

**Характеристики**:
- Нулевое копирование (компилятор перемещает указатель)
- После перемещения исходная привязка недоступна для чтения (ошибка компиляции)
- RAII: автоматическое освобождение при выходе из области видимости
- Сигнатура функции `(T) -> T` сама по себе документация — потребляет T, возвращает T

### 2. &T / &mut T (токены заимствования)

**Основной принцип: `&T` и `&mut T` — это типы токенов времени компиляции нулевого размера. Это не «ссылки», а «типизированное доказательство прав доступа».**

#### 2.1 Два атрибута типа

```
&T      →  Нулевой размер, копируемый (Dup), предоставляет права только на чтение
&mut T  →  Нулевой размер, линейный (не-Dup), предоставляет права на эксклюзивное чтение-запись
```

**Это не «правило», которое нужно запоминать — это базовые атрибуты системы типов.** Типы Dup могут свободно копироваться (несколько `&T` сосуществуют), линейные типы не могут копироваться (`&mut T` естественно уникален). Нет «проверки заимствований» — только проверка типов делает то, что она всегда делала.

#### 2.2 Базовое использование

```yaoxiang
# Методы: объявление типов параметров, определяющих необходимые права
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # Токен &Point предоставляет права на чтение
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # Токен &mut Point предоставляет права на запись
    self.y = self.y + dy
}

# Сторона вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                          # Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)                  # Компилятор автоматически создаёт токен &mut Point
p.print()                          # OK, предыдущий токен освобождён после завершения вызова shift

# Свободные функции аналогично
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # Два токена &Point сосуществуют — тип Dup
}
d = distance(p, p2)
```

#### 2.3 Почему не нужно «запрещение побега»

RFC-009 v8 накладывал на `&T`/`&mut T` три специальных правила — только параметры, нельзя возвращать, нельзя сохранять в структурах. Это заплатки для концепции «заимствования».

Система токенов не нуждается в этих правилах. Токены — **обычные типы**, подчиняющиеся тем же правилам области видимости, что и все остальные типы.

**Возврат ссылок — естественная поддержка**:

```yaoxiang
# ✅ Токен распространяется вместе с возвращаемым значением
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # Дочерний токен и родительский токен возвращаются вместе
}

# Использование
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # Токен возвращается вызывающему
print(px_ref)               # OK, токен всё ещё в области видимости
```

**Сохранение в структурах — естественная поддержка**:

```yaoxiang
# ✅ Структура содержит токен как поле
Window: Type = {
    target: Point,
    view: &Point,      # Поле токена — держит представление только для чтения на target
}

# Токен view производен от target, Window владеет обоими
# Пока существует Window, токен view валиден
```

**Захват в замыканиях — естественная поддержка**:

```yaoxiang
# ✅ Замыкание захватывает токен, как любую переменную
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    # Замыкание захватывает токен &Float переменной threshold (тип Dup, свободно копируется в замыкание)
    items.filter(|p| p.x > threshold)
}

# Это то, что RFC-009 v8 не мог — v8 запрещал захват заимствований в замыканиях
```

**Между задачами — токены нельзя передавать через потоки**:

```yaoxiang
# ❌ Токены нельзя передавать через границы задач
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ Ошибка компиляции: токены нельзя передавать между задачами
}

# Это не специальное правило — токены это доказательство прав времени компиляции,
# для совместного использования между задачами используйте ref
```

**Токены нельзя оборачивать в ref**:

```yaoxiang
# ❌ Токены это доказательство прав, не владения
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ Ошибка компиляции: &T не является владеющим типом
}
```

#### 2.4 Жизненный цикл токенов

Жизненный цикл токенов определяется **обычными правилами области видимости**, без параметров времени жизни:

- Токены в параметрах функции: живут во время вызова, освобождаются после завершения вызова
- Возвращаемые токены: владение передаётся вызывающему
- Токены, хранящиеся в структурах: живут вместе со структурой
- Захваченные в замыканиях токены: живут вместе с замыканием

Компилятору не нужны аннотации `'a`, потому что токены — **значения**, жизненный цикл значений управляется системой владения (Move/RAII). **Сведение проблемы заимствования к проблеме владения.**

#### 2.5 Обнаружение конфликтов токенов

Замена «перекрёстной проверки заимствований» из RFC-009 v8. Принцип проще — **потоково-чувствительный анализ активности** значений токенов:

```yaoxiang
# ❌ Токен &mut линейный, нельзя копировать
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move, p больше нельзя читать
    p.x = 10.0                      # ❌ Ошибка компиляции: WriteToken уже перемещён
}

# ✅ Токен &T это тип Dup, можно свободно копировать
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK, &T это тип Dup
    print(p.x)                      # OK
    print(p2.x)                     # OK, два токена только для чтения сосуществуют
}
```

**Способ обнаружения**: Это не специальный «проверщик заимствований» — это **потоково-чувствительный анализ активности** значений токенов. Компилятор отслеживает состояние каждого токена внутри тела функции (активен/перемещён), точно так же как отслеживает любые значения линейного типа.

#### 2.7 Внутреннее устройство компилятора: механизм брендов

Пользователь никогда не сталкивается с брендами. Компилятор внутренне назначает каждому токену уникальный идентификатор времени компиляции:

```
Что видит пользователь    Внутреннее представление компилятора
────────────────────────────────────────────────────────────────
&Point                 →  ReadToken(Point, #N)    // #N — уникальное целое число времени компиляции
&mut Point             →  WriteToken(Point, #M)   // #M — уникальное целое число времени компиляции
```

Назначение брендов:
- **Защита от подделки**: токены можно получить только от владеющего владельца (capsule), нельзя создать из ничего
- **Отслеживание производности**: когда из `&Point` получаем `&Float` (доступ к полю), `&Float` несёт производный бренд (`#N.field_x`), компилятор может отследить до родительского токена
- **Обнаружение конфликтов**: однотипные `WriteToken` и производные `ReadToken` не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнга, в сгенерированном машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

#### 2.8 Правила автоматического выбора заимствования

Компилятор на стороне вызова автоматически выбирает по следующему приоритету:

```
1. Если фактический аргумент ещё используется в дальнейшем → приоритетно создать токен (&T или &mut T, по сигнатуре метода)
2. Если фактический аргумент больше не используется → Move
3. Приоритет соответствия: &T < &mut T < Move
```

```yaoxiang
# Пример: автоматический выбор
p = Point(1.0, 2.0)
p.print()        # print объявляет &self → компилятор создаёт токен &Point
p.shift(1.0, 1.0) # shift объявляет &mut self → компилятор создаёт токен &mut Point
p2 = p           # Move, p больше не используется
```

#### 2.9 Сравнение с упрощённым заимствованием RFC-009 v8

| Характеристика | Упрощённое заимствование (v8) | Токены заимствования (v9) |
|----------------|-------------------------------|---------------------------|
| Возврат ссылок | ❌ Жёстко запрещено | ✅ Токен распространяется с возвращаемым значением |
| Сохранение в структурах | ❌ Жёстко запрещено | ✅ Токен как поле структуры |
| Захват в замыканиях | ❌ Жёстко запрещено | ✅ Замыкание захватывает значение токена |
| Специальные правила | 3 (только параметры / нельзя возвращать / нельзя сохранять) | 0 — атрибуты типа естественно выводят |
| Проверка заимствований | Специализированная перекрёстная проверка | Потоково-чувствительный анализ активности проверки типов |
| Аннотации времени жизни | Не нужны | Не нужны |
| Накладные расходы выполнения | Ноль | Ноль (типы нулевого размера, исчезают после компиляции) |
| Сообщения об ошибках | «Заимствование не может покидать область» | «WriteToken(#3) уже перемещён» (обычная ошибка типа) |
| Ментальная модель пользователя | Понимание особого статуса «заимствования» | `&T` копируемый, `&mut T` не копируемый |

### 3. ref ключевое слово (автоматическая оптимизация компилятора)

`ref` — единственный способ совместного владения между областями видимости. Внутри Rc или Arc — пользователю не нужно об этом думать.

#### 3.1 Базовое использование

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # Совместное использование, компилятор автоматически выбирает реализацию

# Совместное использование между задачами
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # Компилятор: между задачами → Arc
    spawn { use(data) }           # Компилятор: между задачами → Arc
}

# Совместное использование в одной задаче
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # Компилятор: не между задачами → Rc
}
```

**Ментальная модель пользователя**: `ref` = совместное владение. Вот и всё.

#### 3.2 Escape-анализ компилятора: Rc vs Arc

```
Анализ потока данных ref:

Не сбегает в другие задачи → Rc (неатомический подсчёт ссылок, низкие накладные расходы)
Сбегает в другие задачи    → Arc (атомический подсчёт ссылок, потокобезопасность)
```

#### 3.3 Стратегия обнаружения циклов

```
Внутри задачи циклы → тихо допускаются.
  ├── Структурированный параллелизм гарантирует унифицированное освобождение всех ресурсов по завершении задачи.
  ├── ref всегда жив, семантика не разбавляется.
  └── Пользователь имеет право строить двунаправленные сильные ссылки внутри задачи (например, промежуточные состояния графовых вычислений).

Между задачами циклы → lint (по умолчанию warn, настраивается).
  ├── Программа работает корректно, настоящих утечек не будет (при завершении родительской задачи все ресурсы дочерних задач освобождаются).
  ├── Но межзадачные сильные ссылки означают размытые границы владения, стоит остановиться и переосмыслить.
  ├── По умолчанию warn, компиляция проходит, но есть предупреждение.
  └── Команда может установить deny в конфигурации проекта, включить в CI quality gate.
```

**Уровни lint** (по аналогии с Rust clippy):

| Уровень | Поведение | Сценарий |
|---------|----------|----------|
| `allow` | Не проверять | Личные проекты |
| `warn` (по умолчанию) | Компиляция проходит, предупреждение | Этап разработки |
| `deny` | Компиляция не проходит | Командный CI quality gate |
| `forbid` | Компиляция не проходит, нельзя переопределить | Организационные обязательные правила |

```yaoxiang
# Циклы внутри задачи: тихо допускаются, двунаправленные сильные ссылки
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # Цикл. Освобождается унифицированно при завершении задачи.
}

# Циклы между задачами: lint (по умолчанию warn)
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: циклическая ссылка между задачами
    }
}
```

**Пример конфигурации проекта**:

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # Межзадачные циклы отклоняются на CI
```

| Тип цикла | Поведение | Причина |
|-----------|-----------|---------|
| Цикл ref внутри задачи | Не проверяется | Право пользователя, освобождение по завершении задачи |
| Цикл ref между задачами | lint (по умолчанию warn) | Напоминание переосмыслить, можно настроить deny |

#### 3.4 Weak: предоставляется стандартной библиотекой

```yaoxiang
use std.rc.Weak

# Продвинутые пользователи явно выбирают
a.next = ref b
b.prev = Weak.new(a.next)        # Пользователь явно контролирует, какое направление слабое
```

**`Weak` не является встроенным в язык — это тип стандартной библиотеки.** Для повседневного использования достаточно `ref`. Продвинутые пользователи, которым нужна тонкая настройка памяти, вручную используют `Weak`.

#### 3.5 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Назначение | Взглянуть/изменить на месте | Совместное владение |
| Область действия | Область видимости токена | Между областями видимости |
| Стоимость | Ноль (типы нулевого размера) | Rc или Arc (выбирает компилятор) |
| Побег | Можно (токен распространяется с возвращаемым значением/структурой/замыканием) | По своей природе для побега и создан |
| Между задачами | Нельзя (токен — доказательство прав времени компиляции, не передаётся между задачами) | Можно (компилятор автоматически выбирает Arc) |
| Циклы | Не применимо | Внутри задач тихо, между задачами lint |

### 4. clone() —— Явное копирование

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # Глубокое копирование
# p и p2 независимы, не влияют друг на друга
```

**Когда использовать**: когда нужно сохранить оригинальное значение и сценарий не подходит для Move или совместного использования.

### 5. unsafe + голые указатели (системное программирование)

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # Голый указатель
    (*ptr).x = 0.0                # Разыменование (пользователь гарантирует безопасность)
    ptr2 = ptr + 1                # Арифметика указателей
}
```

**Ограничения**:
- Можно использовать только в блоках `unsafe`
- Пользователь гарантирует отсутствие висячих указателей и использования после освобождения
- Используется для FFI, операций с памятью и другого системного программирования

## Комплексный пример

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T: токен только для чтения
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T: изменяемый токен
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move: потребление с возвратом
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # Забрать, изменить, вернуть
    }

    # Возврат ссылок: токен распространяется с возвращаемым значением
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# Захват токенов в замыканиях (возможность, которой v8 не обладал)
filter_by_threshold: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)
}

# Комплексное использование
p = Point(1.0, 2.0)
p.print()                           # Токен &Point
p.shift(1.0, 1.0)                   # Токен &mut Point
p = p.scale(2.0)                    # Move → возврат
shared = ref p                      # ref совместное использование
spawn { use(shared) }

# clone независимая копия
backup = p.clone()

# Циклы внутри задачи: тихо допускаются
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # Цикл, освобождается унифицированно при завершении задачи

# unsafe системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## Ограничения системы типов

### Атрибут типа Dup

`Dup` (Дублируемый) — это атрибут типа, автоматически управляемый компилятором, означающий **поверхностное копирование**: при присваивании/передаче параметров копируются дескрипторы/токены, а базовые данные общие. Это образует трёхуровневый градиент с Move (передача владения) и Clone (явное глубокое копирование, создание независимых копий).

**Dup и Clone — ортогональные концепции** — Dup копирует дескрипторы совместно используя данные, Clone создаёт независимые копии. Тип может одновременно поддерживать Dup и Clone, либо только один из них.

| Тип | Dup | Clone | Описание |
|-----|-----|-------|----------|
| `&T` | ✅ (копирование токена, несколько представлений указывают на одни данные) | ✅ | Токен только для чтения |
| `ref T` | ✅ (счётчик ссылок +1, общие данные в куче) | ✅ | Совместное владение (компилятор автоматически выбирает Rc/Arc) |
| String, Bytes | ✅ (внутренний счётчик ссылок, копирование дескриптора использует общий буфер) | ✅ | Строки/байты |
| `&mut T` | ❌ (линейный, эксклюзивный) | ❌ | Изменяемый токен |
| `*T` | ❌ | ❌ | Голый указатель |
| struct | Производный (автоматически производный, когда все поля Dup) | ✅ | Структуры |

**Примитивные типы значений** (Int, Float, Bool, Char) при присваивании ведут себя как встроенное компилятором копирование значений — два значения полностью независимы, не поверхностное копирование. Они не относятся к атрибуту типа Dup, а являются нативной обработкой компилятора.

## Анализ производительности

| Операция | Стоимость | Описание |
|----------|-----------|----------|
| Move | Ноль | Перемещение указателя |
| `&T` / `&mut T` | Ноль | Типы нулевого размера, исчезают после компиляции, нулевые накладные расходы выполнения |
| `ref` (не между задачами) | Низкая | Компилируется в Rc, неатомические операции |
| `ref` (между задачами) | Средняя | Компилируется в Arc, атомические операции |
| `clone()` | Зависит от типа | Маленькие объекты быстро, большие медленно |
| `unsafe + *T` | Ноль | Прямые операции с памятью |

### Сравнение

| Язык | Механизм совместного использования | Управление памятью | Обработка циклов | Сложность |
|------|------------------------------------|--------------------|------------------|-----------|
| Rust | Arc / Mutex + проверка заимствований | Время компиляции | Ручной Weak | Высокая |
| Go | chan / pointer | GC | GC | Низкая |
| C++ | shared_ptr | RAII | weak_ptr | Средняя |
| **YaoXiang** | **ref + токены заимствования** | **RAII** | **Освобождение по границе задач / lint между задачами / стандартная библиотека Weak** | **Низкая** |

## Компромиссы

### Преимущества

1. **Единообразие**: `&T`/`&mut T` — обычные типы, не специальные языковые конструкции. Полностью соответствует `name: type = value` из RFC-010
2. **Простота**: Нет времени жизни, нет глобального проверщика заимствований. `&T` копируемый, `&mut T` не копируемый — два атрибута типа
3. **Выразительность**: Можно возвращать ссылки, сохранять в структурах, захватывать в замыканиях — выразительность на уровне Rust
4. **Интеллект компилятора**: ref автоматически выбирает Rc/Arc, сторона вызова автоматически выбирает заимствование
5. **Детерминированность**: ref просто живёт, не становится тихо слабой ссылкой
6. **Высокая производительность**: Move с нулевым копированием, токены с нулевыми накладными (типы нулевого размера, исчезают после компиляции)
7. **Гибкость**: `unsafe + *T` поддерживает системное программирование

### Недостатки

1. **Заражение обобщённых параметров брендов**: токены несут идентификаторы брендов, сигнатуры функций с возвратом ссылок содержат дополнительные обобщённые параметры
2. **Накладные расходы выполнения ref**: атомические операции имеют стоимость (но это неизбежная цена совместного использования)
3. **Риски unsafe**: пользователь должен гарантировать корректность
4. **Межзадачные циклы — это lint, не ошибка компиляции**:，不像 Rust 那样编译报错，默认 warn，需要团队配置 deny 才能作为质量门

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| GC | Накладные расходы выполнения, непредсказуемые паузы |
| Проверщик заимствований Rust | Нужно время жизни `'a`, крутая кривая обучения |
| Только Move | Не справляется с параллельным совместным использованием |
| Без голых указателей | Невозможно системное программирование |
| Предоставить Rc/Arc пользователю | Детали реализации перекладываются на пользователя, увеличивается когнитивная нагрузка |
| Упрощённое заимствование (v8) | Стратегия запрета побега жертвует ключевыми возможностями: захват в замыканиях, возврат ссылок и т.д. |

### Обнаружение конфликтов токенов: потоково-чувствительный анализ активности

**Область анализа**: только тело функции. Потоково-чувствительный анализ активности значений токенов, отслеживание состояния каждого токена (активен/перемещён).

**Уровень 1: Проверка в точке вызова** — один фактический аргумент не может одновременно создавать токен `&mut` и другие токены:

```yaoxiang
alias_bad: (a: &mut Point, b: &Point) -> Void = { ... }
p = Point(1.0, 2.0)
alias_bad(p, p)    # ❌ p одновременно производит токены &mut и &, компилятор отклоняет
```

**Уровень 2: Потоковая чувствительность тела функции** — после передачи токена `&mut` вызову, при возврате вызова токен освобождается, в дальнейшем можно создать новый токен:

```yaoxiang
process_twice: (p: &mut Point) -> Void = {
    shift(p, 1.0, 1.0)    # Токен &mut передан в shift, после возврата shift токен освобождён
    print_info(p)          # Создаётся новый токен &Point, конфликта нет
}
```

**Что не нужно**: Межфункциональное отслеживание времени жизни, глобальный анализ псевдонимов, решение ограничений графа заимствований, NLL, аннотации `'a`. Потому что токены — это значения, анализ активности значений единообразно обрабатывается проверщиком типов — точно так же как отслеживание любых значений линейного типа.
