# RFC-009: Модель владения

## Резюме

Этот документ определяет **модель владения (Ownership Model)** языка программирования YaoXiang.

**Основная идея — пять концепций, один градиент**:

```
Взглянуть/изменить    Забрать            Общее владение       Клонировать    Системный
    │                    │                    │                    │              │
   &T                  Move              ref                 clone()        unsafe
  &mut T              Нулевая копия     Автовыбор комп.      Явное глубокое  *T
  Токен нулевого       По умолчанию     Rc/Arc               копирование    Ответств.
  размера                                                               пользователя
  Свойство типа                                                          
  Естественный                                                              
  вывод прав                                                                
```

- **Move (по умолчанию)**: присваивание/передача параметров/возврат = передача владения, нулевая копия, RAII автоматически освобождает
- **`&T` / `&mut T` (токены заимствования)**: токены компиляции нулевого размера. `&T` копируемый (разделяемое чтение), `&mut T` линейный (исключительная мутация). Права естественно выводятся из свойств типа, никаких специальных правил не требуется. Можно возвращать, хранить в структурах.
- **Ключевое слово `ref`**: разделяемое владение между областями видимости. Компилятор автоматически выбирает Rc (без пересечения задач) или Arc (с пересечением задач)
- **`clone()`**: явное глубокое копирование
- **`unsafe` + `*T`**: сырые указатели, системный аварийный выход

**Устранённая сложность**:
- ❌ Нет времён жизни `'a`
- ❌ Нет отдельного фреймворка проверки заимствований (конфликты заимствования сводятся к утверждениям Хоара, разделяют конвейер доказательства с проверкой типов)
- ❌ Нет GC
- ❌ Нет специальных правил вроде «запрет на побег» (токены — обычные типы, область видимости обрабатывается системой типов единообразно)
- ❌ Пользователю не нужно знать разницу между Rc/Arc (компилятор выбирает автоматически)

> **Бремя программирования**: `&T` копируем, `&mut T` не копируем — два свойства типа, ноль специальных правил, всё автоматически.
> **Гарантия производительности**: Move — нулевые накладные расходы, токены — нулевые накладные расходы (тип нулевого размера, исчезает после компиляции), `ref` — оплата по необходимости, нет пауз GC.

## Мотивация

### Зачем нужна модель владения?

| Язык | Управление памятью | Проблема |
|------|-------------------|----------|
| C/C++ | Ручное управление | Утечки памяти, висячие указатели, двойное освобождение |
| Java/Python | GC | Колебания задержек, расход памяти, непредсказуемые паузы |
| Rust | Владение + проверка заимствований | Крутая кривая обучения для времён жизни `'a` |
| **YaoXiang** | **Move + Token + ref** | **Простота, детерминированность, без GC** |

### Цели проектирования

```yaoxiang
# 1. По умолчанию Move (нулевая копия)
p = Point(1.0, 2.0)
p2 = p                         # Move, p больше недоступен для чтения

# 2. &T / &mut T — токены заимствования (нулевые накладные расходы, права выводятся из свойств типа)
print_info(p2)                 # Компилятор автоматически создаёт токен &Point, освобождается после использования
shift(p2, 1.0, 1.0)           # Компилятор автоматически создаёт токен &mut Point

# 3. ref = разделяемое владение (компилятор автоматически выбирает Rc/Arc)
shared = ref p2                # Удержание между областями видимости
spawn { use(shared) }          # Компилятор: пересечение задач → Arc

# 4. clone() = явное копирование
backup = p2.clone()            # Глубокая копия, независимая

# 5. unsafe + *T = системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Ключевые отличия от Rust

| Свойство | Rust | YaoXiang |
|----------|------|----------|
| Семантика по умолчанию | Заимствование `&T` (требуется явный `.clone()`) | **Move (передача значения, нулевая копия)** |
| Заимствование | `&T`/`&mut T`, можно возвращать, требует времён жизни | **`&T`/`&mut T` — токены нулевого размера, права естественно выводятся из свойств типа Dup/Linear** |
| Механизм разделяемого владения | `Arc::new()` + ручной Weak | **Ключевое слово `ref` (компилятор автоматически выбирает Rc/Arc)** |
| Копирование | `clone()` | `clone()` |
| Сырые указатели | `*T` | `*T` |
| Времена жизни | `'a` | ❌ Нет |
| Проверка заимствований | Глобальный вывод | **Генератор типов автоматически создаёт утверждения заимствования, общий конвейер проверки** |
| Циклические ссылки | Ручной Weak | **Единое освобождение при завершении задачи / lint для пересечения задач / стандартная библиотека Weak** |

## Предложение

### 1. Move (передача владения по умолчанию)

```yaoxiang
# Правило: присваивание / передача параметров / возврат = Move, нулевая копия

p: Point = Point(1.0, 2.0)
p2 = p                           # Move, p больше недоступен для чтения

# Переменную можно переприсвоить (стиль Python, без затенения)
p = Point(3.0, 4.0)              # p перепривязывается, тип должен совпадать

# Параметры функции: Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Возврат Move
}

# Возврат из функции: Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Возврат Move, нулевая копия
}
```

**Особенности**:
- Нулевая копия (компилятор перемещает указатель)
- После перемещения исходная привязка недоступна для чтения (ошибка компиляции)
- RAII: автоматическое освобождение при выходе из области видимости
- Сигнатура функции `(T) -> T` сама по себе является документацией — потребляет T, возвращает T

### 2. &T / &mut T (токены заимствования)

**Ключевой принцип: `&T` и `&mut T` — это типы токенов компиляции нулевого размера. Это не «ссылки», а «типизированные доказательства права доступа».**

#### 2.1 Два свойства типа

```
&T      →  Нулевой размер, замораживает исходные данные (пока жив ReadToken, запрещён WriteToken),
          При заморозке множественные представления только для чтения безопасны → Копируем (Dup)
&mut T  →  Нулевой размер, исключительное чтение-запись (пока жив WriteToken, любой другой токен запрещён),
          При исключительном доступе копирование бессмысленно → Линейный (не Dup)
```

**Причинно-следственная связь не может быть обратной: заморозка — причина, Dup — следствие.** Не потому что `&T` реализует Dup, поэтому может сосуществовать — а потому что данные заморожены (мутация невозможна), множественные представления только для чтения безопасны, Dup становится возможным. Если считать Dup определением, а проверку конфликтов «дополнительной заплаткой», проектирование ошибочно.

#### 2.2 Базовое использование

```yaoxiang
# На стороне метода: объявляем тип параметра, определяем требуемые права
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # Токен &Point даёт право чтения
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # Токен &mut Point даёт право записи
    self.y = self.y + dy
}

# На стороне вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                          # Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)                  # Компилятор автоматически создаёт токен &mut Point
p.print()                          # OK, предыдущий токен освобождён после вызова shift

# Свободные функции аналогично
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # Два токена &Point сосуществуют — тип Dup
}
d = distance(p, p2)
```

#### 2.3 Почему не нужно «запрещать побег»

RFC-009 v8 накладывал на `&T`/`&mut T` три специальных правила — можно использовать только как параметр, нельзя возвращать, нельзя хранить в структуре. Это заплатки к концепции «заимствования».

Система токенов не нуждается в этих правилах. Токены — **обычные типы**, подчиняющиеся тем же правилам области видимости, что и все остальные типы.

**Возврат ссылки — естественно поддерживается**:

```yaoxiang
# ✅ Токен распространяется вместе с возвращаемым значением
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # Дочерний токен и родительский токен возвращаются вместе
}

# Использование
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # Токен возвращён вызывающему
print(px_ref)               # OK, токен всё ещё в области видимости
```

**Хранение в структуре — естественно поддерживается**:

```yaoxiang
# ✅ Структура несёт токен как поле
Window: Type = {
    target: Point,
    view: &Point,      # Поле-токен — содержит представление target только для чтения
}

# Токен view порождён от target, Window владеет обоими
# Пока Window существует, токен view действителен
```

#### 2.3 Замыкания и Lambda с явными параметрами

Lambda — это значение функции, которое может быть возвращено, сохранено, передано за пределы текущей области видимости. Поэтому Lambda **не захватывает неявно переменные внешней области**. Когда нужны внешние данные, их передают через явные параметры:

```yaoxiang
# ✅ Lambda использует явные параметры
double: (x: Int) -> Int = (x) => x * 2
filter_by: (items: List(Int), f: (Int) -> Bool) -> List(Int) = { ... }

# ✅ spawn { } не подчиняется этому правилу — spawn это немедленно выполняемый параллельный блок, родительская задача блокируется в ожидании
shared = ref data
spawn { use(shared) }

# ❌ Lambda не может неявно захватывать внешние переменные
x = 42
f = () => { x + 1 }  # Ошибка компиляции: x не в области видимости

# ✅ Правильный способ: явная передача параметра
f = (x) => { x + 1 }
f(x)
```

**`spawn { }` — это не значение функции.** Блок, помеченный spawn, подобен телу if/while — выполняется немедленно, завершается за время жизни родительского кадра стека. Тело spawn может нормально обращаться к внешним переменным.

**Пересечение задач — токены не могут пересекать границы потоков**:

```yaoxiang
# ❌ Токены не могут пересекать границу задач
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ Ошибка компиляции: токен не может пересекать границу задачи
}

# Это не специальное правило — токен это доказательство права времени компиляции, для разделяемого между задачами используйте ref
# Если нужно разделять между задачами, используйте ref
```

**Токен нельзя обернуть в `ref`**:

```yaoxiang
# ❌ Токен это доказательство права, а не владение
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ Ошибка компиляции: &T не является владеющим типом
}
```

#### 2.4 Жизненный цикл токенов

Жизненный цикл токенов определяется **обычными правилами области видимости**, без необходимости в параметрах времени жизни:

- Токены в параметрах функции: живут во время вызова, освобождаются после вызова
- Возвращённые токены: владение передаётся вызывающему
- Токены, хранящиеся в структурах: живут вместе со структурой

Компилятору не нужна аннотация `'a`, потому что токены — это **значения**, чей жизненный цикл управляется системой владения (Move/RAII) единообразно. **Сведение задачи заимствования к задаче владения.**

#### 2.5 Обнаружение конфликтов токенов

Обнаружение конфликтов токенов — это **утверждение логики Хоара**, а не отдельный потоково-чувствительный анализ.

```
{Все конфликтующие ReadToken мертвы} write(data) {WriteToken безопасно получен}
```

Разделяет конвейер доказательства RFC-027 с проверкой типов и проверкой пользовательских предикатов. Компилятор автоматически генерирует утверждения заимствования (`borrow_conflict`, `use_after_move`, `use_after_drop`, `mut_violation`) и отправляет в конвейер. Конвейер возвращает Proved / Disproved / Unproven.

```yaoxiang
# ❌ &mut токен линеен, не может быть скопирован
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move, p больше недоступен для чтения
    p.x = 10.0                      # ❌ Ошибка компиляции: WriteToken уже перемещён
}

# ✅ &T токен имеет тип Dup, может свободно копироваться
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK, &T имеет тип Dup
    print(p.x)                      # OK
    print(p2.x)                     # OK, два токена только для чтения сосуществуют
}
```

**Проверка заимствований не исчезла — она понижена в размерности.** Существующий `BorrowChecker` становится `BorrowPredicateEmitter` (генератор утверждений), генерируемые утверждения заимствования разделяют общий конвейер доказательства с другими утверждениями типов. Это полностью параллельно концепции проверки типов: проверщик типов генерирует утверждения равенства типов, генератор утверждений заимствования генерирует утверждения заимствования, тот же конвейер верифицирует. Подробности см. в [RFC-009a](../accepted/009a-borrow-proof-pipeline.md).

#### 2.7 Внутри компилятора: механизм брендов

Пользователь никогда не сталкивается с брендами. Компилятор внутренне присваивает каждому токену уникальный идентификатор времени компиляции:

```
Видимое пользователю       Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — уникальное целое число времени компиляции
&mut Point     →  WriteToken(Point, #M)   // #M — уникальное целое число времени компиляции
```

Назначение брендов:
- **Защита от подделки**: токен может быть получен только из капсулы владельца, не может быть сконструирован из воздуха
- **Отслеживание связей**: при порождении `&Float` из `&Point` (доступ к полю) `&Float` несёт порождённый бренд (`#N.field_x`), компилятор может отследить родительский токен
- **Обнаружение конфликтов**: `WriteToken` того же источника и порождённый `ReadToken` не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнинга, отсутствуют в генерируемом машинном коде. **Нулевые накладные расходы в рантайме.**

#### 2.8 Правила автоматического выбора заимствования

На стороне вызова компилятор выбирает по следующему приоритету:

```
1. Если аргумент используется далее → предпочтительно создать токен (&T или &mut T, в зависимости от сигнатуры метода)
2. Если аргумент далее не используется → Move
3. Приоритет сопоставления: &T < &mut T < Move
```

```yaoxiang
# Пример: автоматический выбор
p = Point(1.0, 2.0)
p.print()        # print объявляет &self → компилятор создаёт токен &Point
p.shift(1.0, 1.0) # shift объявляет &mut self → компилятор создаёт токен &mut Point
p2 = p           # Move, p больше не используется
```

#### 2.9 Сравнение с убогим заимствованием RFC-009 v8

| Свойство | Убогое заимствование (v8) | Токены заимствования (v9) |
|----------|---------------------------|---------------------------|
| Возврат ссылки | ❌ Жёстко запрещено | ✅ Токен распространяется с возвращаемым значением |
| Хранение в структуре | ❌ Жёстко запрещено | ✅ Токен как поле структуры |
| Lambda с явными параметрами | ❌ Жёстко запрещено | ✅ Lambda с явными параметрами |
| Специальные правила | 3 правила (только параметр / нельзя возвращать / нельзя хранить) | 0 правил — свойства типа естественно выводят права |
| Проверка заимствований | Выделенная перекрёстная проверка заимствований | Потоково-чувствительный анализ активности проверщика типов |
| Аннотации времён жизни | Не нужны | Не нужны |
| Накладные расходы в рантайме | Ноль | Ноль (тип нулевого размера, исчезает после компиляции) |
| Сообщения об ошибках | «Заимствование не может убежать» | «WriteToken(#3) уже перемещён» (обычная ошибка типа) |
| Ментальная модель пользователя | Понимание особого статуса «заимствования» | `&T` копируем, `&mut T` не копируем |

### 3. Ключевое слово `ref` (автоматическая оптимизация компилятора)

`ref` — единственный способ разделяемого владения между областями видимости. В основе лежит Rc или Arc, о выборе пользователю думать не нужно.

#### 3.1 Базовое использование

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # Разделяемое владение, компилятор автоматически выбирает реализацию

# Разделяемое между задачами
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # Компилятор: пересечение задач → Arc
    spawn { use(data) }           # Компилятор: пересечение задач → Arc
}

# Разделяемое в одной задаче
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # Компилятор: без пересечения задач → Rc
}
```

**Ментальная модель пользователя**: `ref` = разделяемое владение. Достаточно.

#### 3.2 Анализ побегов компилятора: Rc vs Arc

```
Анализ потоков данных ref:

Не убегает в другие задачи → Rc (неатомарный подсчёт ссылок, низкие накладные расходы)
Убегает в другие задачи    → Arc (атомарный подсчёт ссылок, потокобезопасный)
```

#### 3.3 Стратегия обнаружения циклов

```
Цикл внутри задачи → допускается безмолвно.
  ├── Каждая задача имеет чёткую границу жизненного цикла — при завершении задачи все ресурсы (включая циклы ref) освобождаются единообразно.
  ├── Долгоживущие сервисы должны создавать дочерние задачи для каждого запроса/соединения — дочерние задачи освобождают ресурсы автоматически, утечки не накапливаются.
  ├── ref всегда поддерживает жизнь, семантика не разбавляется.
  └── Пользователь имеет право строить двусторонние сильные ссылки внутри задачи (например, промежуточные состояния графовых вычислений).

Цикл между задачами → lint (по умолчанию warn, настраивается).
  ├── Поведение программы корректно, реальной утечки нет (при завершении родительской задачи все ресурсы дочерних задач освобождаются).
  ├── Но сильные ссылки между задачами означают размытие границ владения, стоит остановиться и переосмыслить.
  ├── По умолчанию уровень warn, компиляция проходит с предупреждением.
  └── Команда может установить deny в конфигурации проекта для включения в CI как quality gate.
```

**Уровни Lint** (аналог Rust clippy):

| Уровень | Поведение | Сценарий |
|---------|-----------|----------|
| `allow` | Не проверять | Личный проект |
| `warn` (по умолчанию) | Компиляция проходит, есть предупреждение | Этап разработки |
| `deny` | Ошибка компиляции | Командный CI quality gate |
| `forbid` | Ошибка компиляции, нельзя переопределить | Организационное принудительное правило |

```yaoxiang
# Цикл внутри задачи: допускается безмолвно, двусторонние сильные ссылки
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # Цикл. При завершении задачи освобождается единообразно.
}

# Цикл между задачами: lint (по умолчанию warn)
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: циклическая ссылка между задачами
    }
}
```

**Пример конфигурации проекта**:

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # Циклы между задачами отклоняются в CI напрямую
```

| Тип цикла | Поведение | Причина |
|-----------|-----------|---------|
| Цикл ref внутри задачи | Без проверки | Право пользователя, единое освобождение при завершении задачи |
| Цикл ref между задачами | lint (по умолчанию warn) | Напоминание переосмыслить, можно настроить deny |

#### 3.4 Weak: предоставляется стандартной библиотекой

```yaoxiang
use std.rc.Weak

# Продвинутый пользователь выбирает явно
a.next = ref b
b.prev = Weak.new(a.next)        # Пользователь явно контролирует, какое направление слабое
```

**`Weak` не встроен в язык, это тип стандартной библиотеки.** Для повседневного использования достаточно `ref`. Продвинутые пользователи, которым нужен тонкий контроль памяти, импортируют `Weak` вручную.

#### 3.5 Токены заимствования vs `ref`

| | `&T` / `&mut T` | `ref` |
|---|-----------------|-------|
| Что делает | Взглянуть / изменить на месте | Разделяемое владение |
| Область | Область видимости значения токена | Между областями видимости |
| Стоимость | Нулевые накладные расходы (тип нулевого размера) | Rc или Arc (выбирает компилятор) |
| Побег | Допустим (токен распространяется с возвращаемым значением / структурой / замыканием) | Для этого и предназначен |
| Пересечение задач | Невозможно (токен это доказательство права времени компиляции, не может пересекать границу задачи) | Возможно (компилятор автоматически выбирает Arc) |
| Образование циклов | Не применимо | Внутри задачи безмолвно, между задачами lint |

### 4. `clone()` — явное копирование

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # Глубокая копия
# p и p2 независимы, не влияют друг на друга
```

**Когда использовать**: когда нужно сохранить исходное значение, но Move и разделяемое владение не подходят.

### 5. `unsafe` + сырые указатели (системное программирование)

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # Сырой указатель
    (*ptr).x = 0.0                # Разыменование (пользователь гарантирует безопасность)
    ptr2 = ptr + 1                # Арифметика указателей
}
```

**Ограничения**:
- Можно использовать только в блоке `unsafe`
- Пользователь гарантирует отсутствие висячих указателей и использования после освобождения
- Применяется для FFI, операций с памятью и другого системного программирования

## Комплексный пример

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T: токен только для чтения
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T: токен мутации
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move: потребление с возвратом
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # Забрать, изменить, вернуть
    }

    # Возврат ссылки: токен распространяется с возвращаемым значением
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# Lambda с явными параметрами
double: (x: Int) -> Int = (x) => x * 2

# Комплексное использование
p = Point(1.0, 2.0)
p.print()                           # Токен &Point
p.shift(1.0, 1.0)                   # Токен &mut Point
p = p.scale(2.0)                    # Move → возврат
shared = ref p                      # Разделяемое владение ref
spawn { use(shared) }

# Независимая копия clone
backup = p.clone()

# Цикл внутри задачи: безмолвно допускается
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # Цикл, при завершении задачи освобождается единообразно

# Системный unsafe
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## Ограничения системы типов

### Свойство типа Dup

`Dup` (Duplicable) — это автоматически управляемое компилятором свойство типа, означающее **поверхностное копирование**: при присваивании/передаче параметров копируется дескриптор/токен, базовые данные разделяются. Это образует трёхуровневый градиент с Move (передача владения) и Clone (явное глубокое копирование, создающее независимую копию).

**Dup и Clone — ортогональные концепции** — Dup копирует дескриптор с разделением данных, Clone создаёт независимую копию. Тип может одновременно поддерживать Dup и Clone, либо только одно из них.

| Тип | Dup | Clone | Пояснение |
|-----|-----|-------|-----------|
| `&T` | ✅ (копирование токена, множество представлений указывают на те же данные) | ✅ | Токен только для чтения |
| `ref T` | ✅ (счётчик ссылок +1, разделяет кучу) | ✅ | Разделяемое владение (компилятор автоматически выбирает Rc/Arc) |
| String, Bytes | ✅ (внутренний подсчёт ссылок, дескриптор разделяет буфер) | ✅ | Строка/байты |
| `&mut T` | ❌ (линейный, исключительный) | ❌ | Токен мутации |
| `*T` | ❌ | ❌ | Сырой указатель |
| struct | Производное (автоматически, если все поля Dup) | ✅ | Структура |

**Примитивные типы значений** (Int, Float, Bool, Char) при присваивании подвергаются встроенному в компилятор копированию значений — два значения полностью независимы, это не поверхностное копирование. Они не относятся к свойству Dup, это встроенная обработка компилятора.

## Анализ производительности

| Операция | Стоимость | Пояснение |
|----------|-----------|-----------|
| Move | Ноль | Перемещение указателя |
| `&T` / `&mut T` | Ноль | Тип нулевого размера, исчезает после компиляции, нулевые накладные расходы в рантайме |
| `ref` (без пересечения задач) | Низкая | Компилируется в Rc, неатомарная операция |
| `ref` (с пересечением задач) | Средняя | Компилируется в Arc, атомарная операция |
| `clone()` | Зависит от типа | Малые объекты быстро, большие медленно |
| `unsafe + *T` | Ноль | Прямые операции с памятью |

### Сравнение

| Язык | Механизм разделяемого владения | Управление памятью | Обработка циклов | Сложность |
|------|--------------------------------|--------------------|--------------------|-----------|
| Rust | Arc / Mutex + проверка заимствований | Проверка времени компиляции | Ручной Weak | Высокая |
| Go | chan / pointer | GC | GC | Низкая |
| C++ | shared_ptr | RAII | weak_ptr | Средняя |
| **YaoXiang** | **ref + токены заимствования** | **RAII** | **Освобождение по границе задачи / lint между задачами / стандартная библиотека Weak** | **Низкая** |

## Компромиссы

### Преимущества

1. **Единообразие**: `&T`/`&mut T` — обычные типы, не специальные языковые конструкции. Полностью соответствует `name: type = value` из RFC-010
2. **Простота**: нет времён жизни, проверка заимствований сводится к утверждениям системы типов. `&T` копируем, `&mut T` не копируем — два свойства типа
3. **Мощность**: можно возвращать ссылки, хранить в структурах, захватывать замыканиями — выразительность на уровне Rust
4. **Умный компилятор**: `ref` автоматически выбирает Rc/Arc, на стороне вызова автоматически выбирается заимствование
5. **Детерминированность**: `ref` — это поддержка жизни, без тихого превращения в слабую ссылку
6. **Высокая производительность**: Move — нулевая копия, токены — нулевые накладные расходы (тип нулевого размера, исчезает после компиляции)
7. **Гибкость**: `unsafe + *T` поддерживает системное программирование

### Недостатки

1. **Заражение параметрами обобщённого бренда**: токены несут идентификатор бренда, в сигнатурах функций, возвращающих ссылки, отражаются дополнительные параметры обобщений
2. **Накладные расходы `ref` в рантайме**: атомарные операции имеют стоимость (но это неизбежная цена разделяемого владения)
3. **Риск `unsafe`**: пользователь должен гарантировать корректность
4. **Циклы между задачами — lint, а не ошибка компиляции**: в отличие от Rust, по умолчанию warn, нужна настройка deny командой для использования как quality gate

## Альтернативы

| Альтернатива | Почему не выбрана |
|--------------|-------------------|
| GC | Накладные расходы в рантайме, непредсказуемые паузы |
| Проверщик заимствований Rust | Требует времён жизни `'a`, крутая кривая обучения |
| Чистый Move | Не справляется с конкурентным разделением |
| Без сырых указателей | Невозможно системное программирование |
| Выставить Rc/Arc пользователю | Сбросить детали реализации на пользователя, увеличить когнитивную нагрузку |
| Убогое заимствование (v8) | Стратегия запрета побега жертвует ключевой выразительностью: захват замыканиями, возврат ссылок и т.д. |

## Журнал проектных решений

| Решение | Выбор | Причина | Дата |
|---------|-------|---------|------|
| **Значение по умолчанию** | Move (нулевая копия) | Высокая производительность, нулевые накладные расходы | 2025-01-15 |
| **Механизм разделяемого владения** | Ключевое слово `ref`, автоматическая оптимизация компилятора | Простота для пользователя, ответственность на компиляторе | 2025-01-15 |
| **Заимствование** | `&T`/`&mut T` как токены нулевого размера | Свойства типа (Dup/Linear) естественно выводят права, единая система типов | 2025-01-15 |
| **Токены заимствования** | Замена убогого заимствования, `&T` Dup, `&mut T` Linear | Устранение специальных правил вроде «запрета на побег», поддержка захвата замыканиями / возврата ссылок / хранения в структурах | 2026-05-29 |
| **Копирование** | `clone()` | Явная семантика | 2025-01-15 |
| **Системный уровень** | `*T` + `unsafe` | Поддержка системного программирования | 2025-01-15 |
| **Времена жизни** | Не реализуются | Токены — значения, жизненный цикл управляется Move/RAII единообразно, задача заимствования сводится к задаче владения | 2025-01-15 |
| **Rc/Arc** | Автоматический выбор компилятором, невидим для пользователя | Снижение когнитивной нагрузки | 2025-01-15 |
| **Циклические ссылки** | Внутри задачи без проверки, между задачами lint (по умолчанию warn) | Структурная конкурентность даёт естественную гарантию, lint можно настроить в deny | 2025-01-16 |
| **Weak** | Предоставляется стандартной библиотекой | Явный выбор для продвинутых пользователей | 2025-01-16 |
| **Анализ потребления** | Удалён | Мини-проверщик заимствований, не нужен | 2026-05-11 |
| **Возврат владения** | Удалён | Сигнатура `(T) -> T` сама по себе документация | 2026-05-11 |
| **Повторное использование пустого состояния** | Удалено (как функция) | Переприсвоение после Move — естественное поведение | 2026-05-11 |
| **Обратные функции / частичное потребление / трёхуровневая мутация полей** | Удалено | Избыточное проектирование | 2026-05-11 |
| **Lambda без неявного захвата** | Lambda использует только явные параметры, не захватывает внешние переменные неявно | Философия явности, упрощение компилятора | 2026-06-16 |

### Обнаружение конфликтов токенов: утверждение логики Хоара

Полное решение по обнаружению конфликтов токенов см. в [RFC-009a: Анализ жизненного цикла токенов — на основе конвейера доказательства Хоара](../accepted/009a-borrow-proof-pipeline.md). Ключевые моменты:

**Активность токенов — утверждение логики Хоара.** `{Все конфликтующие ReadToken мертвы} write(data) {WriteToken безопасно получен}` — разделяет конвейер доказательства RFC-027 с проверкой типов и проверкой пользовательских предикатов. Компилятор автоматически генерирует утверждения заимствования (`borrow_conflict`, `use_after_move`, `use_after_drop`, `mut_violation`), конвейер возвращает Proved / Disproved / Unproven.

**Проверка заимствований не исчезла — она понижена в размерности.** `BorrowChecker` становится `BorrowPredicateEmitter`, генерирует утверждения, а не выполняет проверку. Это полностью параллельно концепции «проверщика типов»: проверщик типов генерирует утверждения равенства типов, генератор утверждений заимствования генерирует утверждения заимствования, тот же конвейер верифицирует.

**Идентификатор бренда (`#42`) — это и есть `'a`.** Информация полностью идентична, различается кодирование. `'a` виден в сигнатуре типа, `#42` — внутри компилятора. Никакого нового анализа не изобретено — время жизни перенесено с уровня типов на уровень доказательств.

**Схема алгоритма** (подробности в RFC-009a):
- Префиксное сопоставление дерева брендов → определение конфликтующих токенов (O(глубина), глубина ≤ 3)
- Обратный BFS → от потребителя, break обрезает обратные рёбра, структурный анализ покрывает 95%+ сценариев (быстрый канал)
- Логическое отсечение SMT → вызывается только при while + условии пути (медленный канал, крайне редко)
