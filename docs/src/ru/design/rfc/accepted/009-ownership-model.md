# RFC-009: Дизайн модели владения

## Аннотация

Данный документ определяет **модель владения (Ownership Model)** языка программирования YaoXiang.

**Ключевой дизайн — пять концепций, один градиент**:

```
Взглянуть/Изменить на месте     Забрать           Разделяемое владение       Скопировать        Системный уровень
    │              │              │              │              │
   &T            Move           ref          clone()        unsafe
  &mut T         Нулевое копирование   Компилятор автоматически     Явное глубокое копирование     *T
  Токен нулевого размера       По умолчанию       Выбирает Rc/Arc                  Пользователь отвечает
  Свойства типа                                   естественно определяют
  естественно выводят права                      права доступа
```

- **Move (по умолчанию)**: присваивание/передача параметров/возврат = передача владения, нулевое копирование, автоматическое освобождение через RAII
- **`&T` / `&mut T` (токены заимствования)**: токены нулевого размера на этапе компиляции. `&T` копируемый (разделяемое только-чтение), `&mut T` линейный (исключительное изменение). Права естественно выводятся из свойств типа без специальных правил. Можно возвращать и хранить в структурах.
- **Ключевое слово `ref`**: разделяемое владение между областями видимости. Компилятор автоматически выбирает Rc (без пересечения задач) или Arc (с пересечением задач)
- **`clone()`**: явное глубокое копирование
- **`unsafe` + `*T`**: сырые указатели, системный аварийный люк

**Устранённая сложность**:
- ❌ Нет времён жизни `'a`
- ❌ Нет отдельного фреймворка проверки заимствований (конфликты заимствования сводятся к утверждениям Хоара, разделяют конвейер доказательства с проверкой типов)
- ❌ Нет GC
- ❌ Нет специальных правил вроде «запрет на выход за пределы» (токены — обычные типы, область видимости обрабатывается системой типов единообразно)
- ❌ Пользователю не нужно знать разницу между Rc/Arc (компилятор выбирает автоматически)

> **Бремя программиста**: `&T` копируемый, `&mut T` некопируемый — два свойства типа, ноль специальных правил, всё полностью автоматизировано компилятором.
> **Гарантии производительности**: Move — нулевые накладные расходы, токены — нулевые накладные расходы (типы нулевого размера, исчезают после компиляции), ref — оплата по необходимости, без пауз GC.

## Мотивация

### Зачем нужна модель владения?

| Язык | Управление памятью | Проблемы |
|------|----------|------|
| C/C++ | Ручное управление | Утечки памяти, висячие указатели, двойное освобождение |
| Java/Python | GC | Колебания задержек, накладные расходы памяти, непредсказуемые паузы |
| Rust | Владение + проверка заимствований | Крутая кривая обучения из-за времён жизни `'a` |
| **YaoXiang** | **Move + Token + ref** | **Просто, детерминировано, без GC** |

### Цели дизайна

```yaoxiang
# 1. Move по умолчанию (нулевое копирование)
p = Point(1.0, 2.0)
p2 = p                         # Move, p больше нельзя читать

# 2. &T / &mut T — токены заимствования (нулевые накладные расходы, свойства типа естественно выводят права)
print_info(p2)                 # Компилятор автоматически создаёт токен &Point, освобождается после использования
shift(p2, 1.0, 1.0)           # Компилятор автоматически создаёт токен &mut Point

# 3. ref = разделяемое владение (компилятор автоматически выбирает Rc/Arc)
shared = ref p2                # Владение между областями видимости
spawn { use(shared) }          # Компилятор: пересекает задачи → Arc

# 4. clone() = явное копирование
backup = p2.clone()            # Глубокое копирование, независимая копия

# 5. unsafe + *T = системный уровень
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

### Ключевые отличия от Rust

| Характеристика | Rust | YaoXiang |
|------|------|----------|
| Семантика по умолчанию | Заимствование `&T` (требуется явный `.clone()`) | **Move (передача значения, нулевое копирование)** |
| Заимствование | `&T`/`&mut T`, можно возвращать, требует времён жизни | **`&T`/`&mut T` — токены нулевого размера, свойства типа Dup/Linear естественно выводят права** |
| Механизм разделяемого владения | `Arc::new()` + ручной Weak | **Ключевое слово `ref` (компилятор автоматически выбирает Rc/Arc)** |
| Копирование | `clone()` | `clone()` |
| Сырые указатели | `*T` | `*T` |
| Времена жизни | `'a` | ❌ Отсутствуют |
| Проверка заимствований | Глобальный вывод | **Генератор проверки типов автоматически создаёт утверждения заимствования, единый конвейер доказательства** |
| Циклические ссылки | Ручной Weak | **Единое освобождение по завершении задачи / lint для пересечения задач / Weak в стандартной библиотеке** |

## Предложение

### 1. Move (передача владения по умолчанию)

```yaoxiang
# Правило: присваивание / передача параметров / возврат = Move, нулевое копирование

p: Point = Point(1.0, 2.0)
p2 = p                           # Move, p больше нельзя читать

# Переменную можно переприсвоить (стиль Python, без затенения)
p = Point(3.0, 4.0)              # p перепривязывается, тип должен совпадать

# Параметры функции: Move
process: (p: Point) -> Point = {
    p.transform()
    p                            # Move возврат
}

# Возврат из функции: Move
create: () -> Point = {
    p = Point(1.0, 2.0)
    p                            # Move возврат, нулевое копирование
}
```

**Особенности**:
- Нулевое копирование (компилятор перемещает указатель)
- После перемещения исходная привязка недоступна для чтения (ошибка компиляции)
- RAII: автоматическое освобождение при выходе из области видимости
- Сигнатура функции `(T) -> T` сама по себе является документацией — потребляет T, возвращает T

### 2. &T / &mut T (токены заимствования)

**Ключевой принцип: `&T` и `&mut T` — это токены нулевого размера на этапе компиляции. Это не «ссылки», а «типизированные доказательства прав доступа».**

#### 2.1 Два свойства типа

```
&T      →  Нулевой размер, замораживает исходные данные (пока живёт ReadToken, WriteToken запрещён),
          при гарантии заморозки множественные представления только-для-чтения безопасны → копируемый (Dup)
&mut T  →  Нулевой размер, исключительное чтение-запись (пока живёт WriteToken, любой другой токен запрещён),
          при исключительном доступе копирование бессмысленно → линейный (не Dup)
```

**Причинно-следственная связь не может быть обратной: заморозка — причина, Dup — следствие.** Не потому, что `&T` реализует Dup, поэтому может сосуществовать — а потому что данные заморожены (мутация невозможна), множественные представления только-для-чтения безопасны, и Dup становится возможным. Если принять Dup за определение, а проверку конфликтов за «дополнительную заплатку», дизайн ошибочен.

#### 2.2 Базовое использование

```yaoxiang
# На стороне метода: объявляем тип параметра, определяя необходимые права
Point.print: (self: &Point) -> Void = {
    print(self.x)                  # Токен &Point даёт право на чтение
    print(self.y)
}

Point.shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
    self.x = self.x + dx           # Токен &mut Point даёт право на запись
    self.y = self.y + dy
}

# На стороне вызова: компилятор автоматически выбирает заимствование или Move
p = Point(1.0, 2.0)
p.print()                          # Компилятор автоматически создаёт токен &Point
p.shift(1.0, 1.0)                  # Компилятор автоматически создаёт токен &mut Point
p.print()                          # OK, предыдущий токен уже освобождён после вызова shift

# То же для свободных функций
distance: (a: &Point, b: &Point) -> Float = {
    sqrt((a.x - b.x)**2 + (a.y - b.y)**2)  # Два токена &Point сосуществуют — тип Dup
}
d = distance(p, p2)
```

#### 2.3 Почему не нужно «правило запрета выхода за пределы»

RFC-009 v8 накладывал на `&T`/`&mut T` три специальных правила — только как параметр, нельзя возвращать, нельзя хранить в структурах. Это была заплатка к концепции «заимствования».

Системе токенов эти правила не нужны. Токены — **обычные типы**, подчиняющиеся тем же правилам областей видимости, что и все остальные типы.

**Возврат ссылки — естественная поддержка**:

```yaoxiang
# ✅ Токен распространяется вместе с возвращаемым значением
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)  # Дочерний токен и родительский токен возвращаются вместе
}

# Использование
p = Point(1.0, 2.0)
(px_ref, p) = p.get_x()    # Токен возвращается вызывающему
print(px_ref)               # OK, токен всё ещё в области видимости
```

**Хранение в структурах — естественная поддержка**:

```yaoxiang
# ✅ Структура несёт токен как поле
Window: Type = {
    target: Point,
    view: &Point,      # Поле-токен — хранит представление target только-для-чтения
}

# view наследуется от target, Window владеет обоими
# Пока Window существует, токен view действителен
```

#### 2.3 Замыкания и Lambda с явными параметрами

Lambda — это значение функции — его можно возвращать, хранить, передавать за пределы текущей области видимости. Поэтому Lambda **не захватывает неявно переменные внешней области видимости**. Для доступа к внешним данным используйте явные параметры:

```yaoxiang
# ✅ Lambda использует явные параметры
double: (x: Int) -> Int = (x) => x * 2
filter_by: (items: List(Int), f: (Int) -> Bool) -> List(Int) = { ... }

# ✅ spawn { } не подчиняется этому правилу — spawn выполняется немедленно как параллельный блок, родительская задача блокируется в ожидании
shared = ref data
spawn { use(shared) }

# ❌ Lambda не может неявно захватывать внешние переменные
x = 42
f = () => { x + 1 }  # Ошибка компиляции: x не в области видимости

# ✅ Правильный способ: явная передача параметра
f = (x) => { x + 1 }
f(x)
```

**spawn { } — это не значение функции.** Блок, помеченный spawn, как и тело if/while — выполняется немедленно, завершается пока жив родительский стековый кадр. Тело spawn может нормально обращаться к внешним переменным.

**Пересечение задач — токены не пересекают границы потоков**:

```yaoxiang
# ❌ Токены не могут пересекать границу задач
bad_task: (p: &Point) -> Void = {
    spawn { print(p.x) }          # ❌ Ошибка компиляции: токен не может передаваться между задачами
}

# Это не специальное правило — токен является доказательством прав на этапе компиляции, для разделяемого владения между задачами используйте ref
# Если нужно разделяемое владение между задачами, используйте ref
```

**Токены не могут быть ref**:

```yaoxiang
# ❌ Токен — это доказательство прав, а не владение
bad_ref: (p: &Point) -> Void = {
    shared = ref p                # ❌ Ошибка компиляции: &T не является владеющим типом
}
```

#### 2.4 Жизненный цикл токенов

Жизненный цикл токенов определяется **обычными правилами областей видимости** и не требует параметров времён жизни:

- Токены в параметрах функции: живут во время вызова, освобождаются после вызова
- Возвращённые токены: владение передаётся вызывающему
- Токены, хранящиеся в структурах: живут вместе со структурой

Компилятору не нужны аннотации `'a`, потому что токены — это **значения**, чей жизненный цикл управляется системой владения (Move/RAII) единообразно. **Сведение проблемы заимствования к проблеме владения.**

#### 2.5 Обнаружение конфликтов токенов

Обнаружение конфликтов токенов — это **утверждение логики Хоара**, а не отдельный потоко-чувствительный анализ.

```
{Все конфликтующие ReadToken мертвы} write(data) {WriteToken безопасно получен}
```

Разделяет конвейер доказательства RFC-027 с проверкой типов и проверкой пользовательских предикатов. Компилятор автоматически генерирует утверждения заимствования (`borrow_conflict`, `use_after_move`, `use_after_drop`, `mut_violation`) и отправляет их в конвейер. Конвейер возвращает Proved / Disproved / Unproven.

```yaoxiang
# ❌ &mut токен линеен, его нельзя копировать
bad_dup: (p: &mut Point) -> Void = {
    p2: &mut Point = p              # Move, p больше нельзя читать
    p.x = 10.0                      # ❌ Ошибка компиляции: WriteToken уже перемещён
}

# ✅ &T токен — тип Dup, его можно свободно копировать
good_dup: (p: &Point) -> Void = {
    p2: &Point = p                  # OK, &T — тип Dup
    print(p.x)                      # OK
    print(p2.x)                     # OK, два токена только-для-чтения сосуществуют
}
```

**Проверка заимствований не исчезла — она понижена в размерности.** Существующий `BorrowChecker` превращается в `BorrowPredicateEmitter` (генератор утверждений), генерируемые утверждения заимствования разделяют тот же конвейер доказательства, что и другие утверждения типов. Это полностью параллельно концепции проверки типов: проверщик типов генерирует утверждения равенства типов, генератор утверждений заимствования генерирует утверждения заимствования, и тот же конвейер их проверяет. Подробности см. в [RFC-009a](../accepted/009a-borrow-proof-pipeline.md).

#### 2.7 Внутри компилятора: механизм брендов

Пользователь никогда не сталкивается с брендами. Компилятор внутренне присваивает каждому токену уникальный идентификатор на этапе компиляции:

```
Видимое пользователю         Внутреннее представление компилятора
────────────────────────────────────────
&Point         →  ReadToken(Point, #N)    // #N — уникальное целое на этапе компиляции
&mut Point     →  WriteToken(Point, #M)   // #M — уникальное целое на этапе компиляции
```

Назначение брендов:
- **Защита от подделки**: токен может быть получен только из капсулы владельца, не может быть сконструирован произвольно
- **Отслеживание связей**: при получении `&Float` из `&Point` (доступ к полю), `&Float` несёт производный бренд (`#N.field_x`), компилятор может отследить его до родительского токена
- **Обнаружение конфликтов**: WriteToken и производный от него ReadToken с тем же источником не могут быть активны одновременно

Бренды полностью исчезают после мономорфизации и инлайнинга, в сгенерированном машинном коде их нет. **Нулевые накладные расходы во время выполнения.**

#### 2.8 Правила автоматического выбора заимствования

На стороне вызова компилятор автоматически выбирает в следующем порядке приоритета:

```
1. Если аргумент используется далее → предпочтительно создать токен (&T или &mut T, согласно сигнатуре метода)
2. Если аргумент далее не используется → Move
3. Порядок предпочтения: &T < &mut T < Move
```

```yaoxiang
# Пример: автоматический выбор
p = Point(1.0, 2.0)
p.print()        # print объявляет &self → компилятор создаёт токен &Point
p.shift(1.0, 1.0) # shift объявляет &mut self → компилятор создаёт токен &mut Point
p2 = p           # Move, p больше не используется
```

#### 2.9 Сравнение с упрощённым заимствованием RFC-009 v8

| Характеристика | Упрощённое заимствование (v8) | Токены заимствования (v9) |
|------|--------------|--------------|
| Возврат ссылки | ❌ Жёстко запрещено | ✅ Токен распространяется с возвращаемым значением |
| Хранение в структурах | ❌ Жёстко запрещено | ✅ Токен как поле структуры |
| Lambda с явными параметрами | ❌ Жёстко запрещено | ✅ Lambda использует явные параметры |
| Специальные правила | 3 (только параметр / нельзя возвращать / нельзя хранить) | 0 — свойства типа естественно выводят права |
| Проверка заимствований | Специализированная перекрёстная проверка заимствований | Потоко-чувствительный анализ активности в проверщике типов |
| Аннотации времён жизни | Не нужны | Не нужны |
| Накладные расходы во время выполнения | Ноль | Ноль (типы нулевого размера, исчезают после компиляции) |
| Сообщения об ошибках | «Заимствование не может выйти за пределы» | «WriteToken(#3) уже перемещён» (обычная ошибка типа) |
| Ментальная модель пользователя | Понимание особого статуса «заимствования» | `&T` копируемый, `&mut T` некопируемый |

### 3. Ключевое слово ref (автоматическая оптимизация компилятора)

`ref` — единственный способ разделяемого владения между областями видимости. На нижнем уровне это Rc или Arc, но пользователю не нужно об этом заботиться.

#### 3.1 Базовое использование

```yaoxiang
p: Point = Point(1.0, 2.0)
shared = ref p                   # Разделяемое владение, компилятор автоматически выбирает реализацию

# Разделяемое владение между задачами
@block
main: () -> Void = {
    data = ref heavy_data
    spawn { use(data) }           # Компилятор: пересекает задачи → Arc
    spawn { use(data) }           # Компилятор: пересекает задачи → Arc
}

# Разделяемое владение в одной задаче
@block
main: () -> Void = {
    data = ref heavy_data
    use(data)                     # Компилятор: не пересекает задачи → Rc
}
```

**Ментальная модель пользователя**: `ref` = разделяемое владение. Этого достаточно.

#### 3.2 Анализ выхода компилятора: Rc vs Arc

```
Анализ потоков данных ref:

Не выходит за пределы других задач → Rc (неатомарный подсчёт ссылок, низкие накладные расходы)
Выходит за пределы других задач   → Arc (атомарный подсчёт ссылок, потокобезопасный)
```

#### 3.3 Стратегия обнаружения циклов

```
Циклы внутри задачи → допускаются без предупреждений.
  ├── Каждая задача имеет чёткие границы жизненного цикла — при завершении задачи все ресурсы (включая циклы ref) освобождаются единообразно.
  ├── Долгоживущие сервисы должны создавать дочерние задачи для каждого запроса/соединения — дочерние задачи освобождаются автоматически по завершении, накопления утечек не происходит.
  ├── ref всегда сохраняет активность, семантика без подвоха.
  └── Пользователь имеет право строить двусторонние сильные ссылки внутри задачи (например, промежуточные состояния графовых вычислений).

Циклы между задачами → lint (по умолчанию warn, настраивается).
  ├── Поведение программы корректно, реальной утечки нет (при завершении родительской задачи все ресурсы дочерних задач освобождаются).
  ├── Однако сильные ссылки между задачами означают размытие границ владения, что стоит пересмотреть.
  ├── По умолчанию уровень warn, компиляция проходит, но с предупреждением.
  └── Команда может установить deny в конфигурации проекта, включив в шлюз качества CI.
```

**Уровни Lint** (аналогично Rust clippy):

| Уровень | Поведение | Сценарий |
|------|------|------|
| `allow` | Без проверки | Личный проект |
| `warn` (по умолчанию) | Компиляция проходит, с предупреждением | Этап разработки |
| `deny` | Ошибка компиляции | Шлюз качества командного CI |
| `forbid` | Ошибка компиляции, нельзя переопределить | Организационные принудительные правила |

```yaoxiang
# Цикл внутри задачи: допускается без предупреждений, двусторонние сильные ссылки
build_graph: () -> Void = {
    a = Node("a")
    b = Node("b")
    a.next = ref b
    b.prev = ref a                # Цикл. Единообразно освобождается по завершении задачи.
}

# Цикл между задачами: lint (по умолчанию warn)
@block
parent_task: () -> Void = {
    shared_a = ref a
    shared_b = ref b
    spawn {
        shared_a.child = ref shared_b   # ⚠️ warn: циклическая ссылка между задачами
    }
}
```

**Пример конфигурации проекта**:

```toml
# yaoxiang.toml
[lints]
cross-task-cycle = "deny"    # Циклы между задачами отклоняются напрямую на CI
```

| Тип цикла | Поведение | Причина |
|--------|------|------|
| Цикл ref внутри задачи | Без проверки | Право пользователя, единообразно освобождается по завершении задачи |
| Цикл ref между задачами | lint (по умолчанию warn) | Предупреждение о необходимости пересмотра, настраивается на deny |

#### 3.4 Weak: предоставляется стандартной библиотекой

```yaoxiang
use std.rc.Weak

# Продвинутый пользователь делает явный выбор
a.next = ref b
b.prev = Weak.new(a.next)        # Пользователь явно контролирует, какое направление слабое
```

**`Weak` не встроен в язык, это тип стандартной библиотеки.** Для повседневного использования достаточно `ref`. Продвинутые пользователи, которым нужен тонкий контроль памяти, вручную вводят `Weak`.

#### 3.5 Токены заимствования vs ref

| | `&T` / `&mut T` | `ref` |
|------|------|------|
| Что делает | Взглянуть/изменить на месте | Разделяемое владение |
| Область | В пределах области видимости значения токена | Между областями видимости |
| Стоимость | Нулевые накладные расходы (типы нулевого размера) | Rc или Arc (выбирает компилятор) |
| Выход за пределы | Да (токен распространяется с возвращаемым значением/структурой/замыканием) | Для этого и предназначен |
| Пересечение задач | Невозможно (токен — доказательство прав на этапе компиляции, не может передаваться между задачами) | Возможно (компилятор автоматически выбирает Arc) |
| Образование циклов | Не применимо | Внутри задачи — без проверки, между задачами — lint |

### 4. clone() — явное копирование

```yaoxiang
p: Point = Point(1.0, 2.0)
p2 = p.clone()                   # Глубокое копирование
# p и p2 независимы, не влияют друг на друга
```

**Когда использовать**: когда нужно сохранить исходное значение, и при этом Move или разделяемое владение не подходят.

### 5. unsafe + сырые указатели (системное программирование)

```yaoxiang
p: Point = Point(1.0, 2.0)

unsafe {
    ptr: *Point = &p              # Сырой указатель
    (*ptr).x = 0.0                # Разыменование (пользователь гарантирует безопасность)
    ptr2 = ptr + 1                # Арифметика указателей
}
```

**Ограничения**:
- Можно использовать только в блоке `unsafe`
- Пользователь гарантирует отсутствие висячих указателей и использования после освобождения
- Применяется для FFI, операций с памятью и другого системного программирования

## Комплексный пример

```yaoxiang
Point: Type = {
    x: Float,
    y: Float,

    # &T: токен только-для-чтения
    print: (self: &Point) -> Void = {
        print(self.x)
        print(self.y)
    }

    # &mut T: токен для изменения
    shift: (self: &mut Point, dx: Float, dy: Float) -> Void = {
        self.x = self.x + dx
        self.y = self.y + dy
    }

    # Move → Move: потребление с возвратом
    scale: (self: Point, f: Float) -> Point = {
        self.x = self.x * f
        self.y = self.y * f
        self                            # Забрать, изменить, вернуть
    }

    # Возврат ссылки: токен распространяется с возвращаемым значением
    get_x: (self: &Point) -> (&Float, &Point) = {
        return (&self.x, self)
    }
}

# Lambda с явными параметрами
double: (x: Int) -> Int = (x) => x * 2

# Комплексное использование
p = Point(1.0, 2.0)
p.print()                           # Токен &Point
p.shift(1.0, 1.0)                   # Токен &mut Point
p = p.scale(2.0)                    # Move → возврат
shared = ref p                      # Разделяемое владение ref
spawn { use(shared) }

# clone — независимая копия
backup = p.clone()

# Цикл внутри задачи: допускается без проверки
a = Node("a")
b = Node("b")
a.next = ref b
b.prev = ref a                      # Цикл, единообразно освобождается по завершении задачи

# unsafe на системном уровне
unsafe {
    ptr: *Point = &p
    (*ptr).x = 0.0
}
```

## Ограничения системы типов

### Свойство типа Dup

`Dup` (Duplicable, дублируемый) — это автоматически управляемое компилятором свойство типа, означающее **поверхностное копирование**: при присваивании/передаче параметров копируется дескриптор/токен, базовые данные используются совместно. Это формирует трёхуровневый градиент с Move (передача владения) и Clone (явное глубокое копирование, создание независимой копии).

**Dup и Clone — ортогональные концепции** — Dup копирует дескриптор, разделяя данные, Clone создаёт независимую копию. Тип может одновременно поддерживать и Dup, и Clone, либо только одно из них.

| Тип | Dup | Clone | Пояснение |
|------|-----|-------|------|
| `&T` | ✅ (копирование токена, множественные представления указывают на те же данные) | ✅ | Токен только-для-чтения |
| `ref T` | ✅ (счётчик ссылок +1, разделяемые данные в куче) | ✅ | Разделяемое владение (компилятор автоматически выбирает Rc/Arc) |
| String, Bytes | ✅ (внутренний подсчёт ссылок, копирование дескриптора разделяет базовый буфер) | ✅ | Строка/байты |
| `&mut T` | ❌ (линейный, исключительный) | ❌ | Токен для изменения |
| `*T` | ❌ | ❌ | Сырой указатель |
| struct | Производное (автоматически, если все поля Dup) | ✅ | Структура |

**Примитивные типы значений** (Int, Float, Bool, Char) — их присваивание является встроенным в компилятор копированием значений — два значения полностью независимы, это не поверхностное копирование. Они не относятся к свойству типа Dup, а обрабатываются компилятором нативно.

## Анализ производительности

| Операция | Стоимость | Пояснение |
|------|------|------|
| Move | Ноль | Перемещение указателя |
| `&T` / `&mut T` | Ноль | Тип нулевого размера, исчезает после компиляции, нулевые накладные расходы во время выполнения |
| `ref` (без пересечения задач) | Низкая | Компилируется в Rc, неатомарные операции |
| `ref` (с пересечением задач) | Средняя | Компилируется в Arc, атомарные операции |
| `clone()` | Зависит от типа | Маленькие объекты быстро, большие — медленно |
| `unsafe + *T` | Ноль | Прямые операции с памятью |

### Сравнение

| Язык | Механизм разделяемого владения | Управление памятью | Обработка циклов | Сложность |
|------|----------|----------|----------|----------|
| Rust | Arc / Mutex + проверка заимствований | Проверка на этапе компиляции | Ручной Weak | Высокая |
| Go | chan / pointer | GC | GC | Низкая |
| C++ | shared_ptr | RAII | weak_ptr | Средняя |
| **YaoXiang** | **ref + токены заимствования** | **RAII** | **Освобождение по границе задачи / lint для пересечения задач / Weak в стандартной библиотеке** | **Низкая** |

## Компромиссы

### Преимущества

1. **Единообразие**: `&T`/`&mut T` — обычные типы, не специальные возможности языка. Полностью согласуется с `name: type = value` из RFC-010
2. **Простота**: нет времён жизни, проверка заимствований сводится к утверждениям системы типов. `&T` копируемый, `&mut T` некопируемый — два свойства типа
3. **Мощность**: можно возвращать ссылки, хранить в структурах, захватывать замыканиями — выразительность на уровне Rust
4. **Умный компилятор**: ref автоматически выбирает Rc/Arc, на стороне вызова автоматически выбирается заимствование
5. **Детерминированность**: ref — это гарантия активности, не превращается молча в слабую ссылку
6. **Высокая производительность**: Move — нулевое копирование, токены — нулевые накладные расходы (типы нулевого размера, исчезают после компиляции)
7. **Гибкость**: `unsafe + *T` поддерживает системное программирование

### Недостатки

1. **Заражение параметрами брендов в дженериках**: токены несут идентификаторы брендов, в сигнатурах функций, возвращающих ссылки, появляются дополнительные параметры дженериков
2. **Накладные расходы ref во время выполнения**: атомарные операции имеют стоимость (но это неизбежная плата за разделяемое владение)
3. **Риск unsafe**: пользователь должен гарантировать корректность
4. **Циклы между задачами — lint, а не ошибка компиляции**: в отличие от Rust, по умолчанию warn, для использования в качестве шлюза качества требуется настройка командой на deny

| Альтернатива | Почему не выбрана |
|------|--------------|
| GC | Накладные расходы во время выполнения, непредсказуемые паузы |

| Упрощённое заимствование (v8) | Стратегия запрета выхода жертвует ключевыми возможностями выразительности: захват замыканий, возврат ссылок |

### Обнаружение конфликтов токенов: утверждения логики Хоара

Полное решение по обнаружению конфликтов токенов см. в [RFC-009a: Анализ жизненного цикла токенов — на основе конвейера доказательств Хоара](../accepted/009a-borrow-proof-pipeline.md). Основные моменты:

**Активность токенов — это утверждение логики Хоара.** `{Все конфликтующие ReadToken мертвы} write(data) {WriteToken безопасно получен}` — разделяет конвейер доказательства RFC-027 с проверкой типов и проверкой пользовательских предикатов. Компилятор автоматически генерирует утверждения заимствования (`borrow_conflict`, `use_after_move`, `use_after_drop`, `mut_violation`), конвейер возвращает Proved / Disproved / Unproven.

**Проверка заимствований не исчезла — она понижена в размерности.** `BorrowChecker` превращается в `BorrowPredicateEmitter`, генерирующий утверждения, а не выполняющий проверку. Это полностью параллельно концепции «проверщика типов»: проверщик типов генерирует утверждения равенства типов, генератор утверждений заимствования генерирует утверждения заимствования, и тот же конвейер их проверяет.

**Идентификатор бренда (`#42`) — это и есть `'a`.** Информация полностью идентична, кодирование различается. `'a` виден в сигнатурах типов, `#42` находится внутри компилятора. Никакого нового анализа не изобретено — время жизни понижено с уровня типов до уровня доказательств.

**Сводка алгоритма** (подробности в RFC-009a):
- Сопоставление префиксов дерева брендов → определение конфликтующих токенов (O(глубина), глубина ≤ 3)
- Обратный BFS → от потребителя, break обрезает обратные рёбра, структурный анализ покрывает 95%+ сценариев (быстрый канал)
- Логическое отсечение SMT → вызывается только при while + условии пути (медленный канал, крайне редкий)
