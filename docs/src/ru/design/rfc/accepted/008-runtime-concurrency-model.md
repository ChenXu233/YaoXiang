# RFC-008: Модель параллелизма Runtime и дизайн декаплинга планировщика

> **⚠️ Примечание о согласовании**: Данный документ согласован с [RFC-024 Новая модель параллелизма](/reference/language-spec/concurrency.md). Устаревший анализ DAG всей программы, аннотации `@block`/`@eager`, иерархическая модель L1/L2/L3 были заменены параллельной примитивной `spawn {}`-блоков. Анализ DAG теперь применяется только внутри `spawn {}`-блоков.

> **Ссылки**:
> - [RFC-011: Дизайн системы дженериков](./011-generic-type-system.md)
> - [Спецификация модели параллелизма (RFC-024)](/reference/language-spec/concurrency.md)

## Резюме

Данный документ определяет ключевые аспекты дизайна архитектуры Runtime:

1. **Трёхуровневая архитектура Runtime**: Embedded (немедленное выполнение) → Standard (spawn + планирование DAG) → Full (work stealing)
2. **Разделение компиляции и выполнения**: фаза компиляции полностью идентична, различия только в способе выполнения в Runtime
3. **Модель с двумя бэкендами**: VM (разработка и отладка) и LLVM AOT (продакшн-релиз), поведение полностью согласовано
4. **Планировщик = статическая библиотека**: при AOT-компиляции планировщик линкуется в exe, ~200-500КБ, без GC
5. **Синхронность — это частный случай планирования**: num_workers=1 — это синхронный режим

### Ключевое уточнение: это не Java

```
Java:   .java → .class → JVM (интерпретация/JIT + GC)        ← всегда нужна виртуальная машина
YaoXiang dev:  .yx → IR → выполнение VM (быстрая итерация, пошаговая отладка)
YaoXiang prod: .yx → IR → LLVM → нативный exe (планировщик слинкован внутрь)

VM — это инструмент разработки, а не сущность Runtime. Точно так же, как go run vs go build в Go.
Итоговый exe = ваш нативный код + статическая библиотека планировщика + метаданные рефлексии. Никакого интерпретатора, никакого JIT, никакого GC.
```

## Мотивация

### Основные противоречия

| Противоречие | Описание |
|------|------|
| Прозрачность vs Контролируемость | spawn-блок обеспечивает явное управление параллелизмом, обычный код выполняется последовательно |
| Ядро vs Опционально | spawn — основная параллельная примитивная, WorkStealing — продвинутая возможность при num_workers>1 |
| Однопоточность vs Параллелизм | В однопоточном режиме параллелизм проявляется как асинхронность, синхронность — это частный случай планирования |

## Предложение

### 1. Трёхуровневая архитектура Runtime

```
┌──────────────────────────────────────────────────────────────────┐
│                  Фаза компиляции (идентична для всех режимов)      │
│                                                                  │
│  Source Code → Lexer → Parser → TypeCheck → Codegen → IR        │
│                                                                  │
│  ⚠️ Единый синтаксический разбор, проверка типов, генерация кода, │
│     вывод IR                                                     │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ Немедленный      │ │ spawn + DAG   │ │ Полный           │
│ исполнитель      │ │               │ │ планировщик      │
│ Синхронное       │ │ Параллелизм   │ │ Параллельная     │
│ выполнение       │ │ внутри spawn- │ │ оптимизация      │
│ Без поддержки    │ │ блоков        │ │ Work stealing    │
│ spawn            │ │ Автоматический│ │                  │
│                  │ │ параллелизм   │ │                  │
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| Этап | Embedded | Standard | Full |
|------|----------|----------|------|
| Компиляция | одинаковая | одинаковая | одинаковая |
| Режим выполнения | синхронный | параллелизм внутри spawn-блоков | параллельный |
| Потребление памяти | низкое | среднее | высокое |
| Возможности параллелизма | нет | внутри spawn-блоков | внутри spawn-блоков + параллелизм |
| Поддержка spawn | ❌ | ✅ | ✅ |
| Анализ DAG | нет | внутри spawn-блоков | внутри spawn-блоков |
| WorkStealer | нет | нет | ✅ |

**Embedded Runtime**: целевая платформа WASM/игровые скрипты/правиловые движки. Немедленный исполнитель, без поддержки spawn, высокая производительность при низком потреблении.

**Standard Runtime**: целевая платформа — веб-сервисы/конвейеры данных. Поддержка `spawn {}`-блоков, анализ DAG и автоматический параллелизм внутри spawn-блоков. num_workers=1 — однопоточная асинхронность.

**Full Runtime**: целевая платформа — научные вычисления/крупномасштабный параллелизм. Standard + WorkStealer для балансировки нагрузки.

### 2. Декаплинг планировщика: дженерики + инъекция

Ключевой принцип: VM не зависит напрямую от конкретного планировщика, а вызывается через обобщённый параметр `[S]`.

```yaoxiang
# Определение интерфейса планировщика
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# Однопоточный планировщик
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# Многопоточный планировщик
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VM использует планировщик через дженерики
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**Ключевые моменты**:
- Полиморфизм на этапе компиляции, нулевые накладные расходы в Runtime
- Нет необходимости в trait-объектах
- Ограничение типов дженериков `[S: Scheduler]` уже определено в RFC-011

### 3. Синхронность = частный случай планирования

```
❌ Заблуждение: отключить планировщик
✅ Правильно: использовать планировщик с одним worker'ом

num_workers = 1 → однопоточное асинхронное планирование
num_workers > 1 → многопоточное параллельное планирование

Один и тот же интерфейс планировщика, разная конфигурация. Устраняем особые случаи.
```

### 4. Место DAG

> **Важное изменение**: анализ DAG больше не применяется ко всей программе, а только внутри `spawn {}`-блоков. Обычный код (вне spawn-блоков) выполняется последовательно и не требует анализа DAG.

| Уровень | Поддержка spawn | Область анализа DAG | Описание |
|------|-----------|-------------|------|
| Core Runtime | ✅ | внутри spawn-блоков | ядро параллелизма |
| Standard Runtime | ✅ | внутри spawn-блоков | spawn + планирование DAG |
| Embedded Runtime | ❌ | нет | немедленное выполнение, без параллелизма |

### 5. Модель восходящего выполнения (внутри spawn-блоков)

> **Важное изменение**: восходящий анализ DAG выполняется только внутри `spawn {}`-блоков и больше не применяется ко всей программе.

```
Код пользователя (параллелизм внутри spawn-блока):
    (a, b) = spawn {
        fetch(url0),
        fetch(url1)
    }
    print(a)

Анализ на этапе компиляции (восходящий внутри spawn-блока):
    fetch(url0) и fetch(url1) не имеют взаимных зависимостей → могут выполняться параллельно
    print(a) вне spawn-блока → последовательное выполнение, ожидание завершения spawn

Планирование в Runtime (от листьев внутри spawn-блока):
    fetch(url0) ┐
                ├→ параллельное выполнение
    fetch(url1) ┘
    print(a)                       ← вне spawn-блока, последовательное выполнение
```

**Ключевые моменты**:
- Восходящий анализ зависимостей только внутри `spawn {}`-блоков
- Независимые задачи внутри spawn-блока выполняются параллельно
- Код вне spawn-блоков выполняется последовательно, ожидая завершения spawn-блока

### 6. Модель компиляции: два бэкенда + статическая линковка Runtime

#### 6.1 Два бэкенда, одно поведение

```
                      ┌─────────────────────┐
                      │  Компиляционный       │
                      │  фронтенд (единый)    │
                      │   Lexer → Parser     │
                      │   → TypeCheck        │
                      │   → анализ DAG       │
                      │     внутри spawn-блоков│
                      │   → escape-анализ    │
                      │   → детекция циклов  │
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │ Бэкенд VM          │     │ Бэкенд LLVM        │
        │ (разработка)       │     │ (продакшн)         │
        │                   │     │                   │
        │ Генерация IR/      │     │ Генерация         │
        │ байткода           │     │ нативного кода     │
        │ Интерпретация VM   │     │ Линковка           │
        │ Поддержка          │     │ статической        │
        │ пошаговой отладки  │     │ библиотеки Runtime │
        │ Быстрая итерация   │     │ Вывод .exe         │
        │                   │     │ Нулевые накладные   │
        │                   │     │ расходы            │
        │                   │     │ интерпретации      │
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           Поведение полностью        Поведение полностью
           согласовано                согласовано
```

**Бэкенд VM**: используется при разработке. Изменение кода → немедленный запуск → пошаговая отладка → быстрая итерация. Поведение полностью согласовано с итоговым exe.

**Бэкенд LLVM**: используется при выпуске. AOT-компиляция в нативный код, планировщик линкуется как статическая библиотека. Нет интерпретатора, нет JIT.

#### 6.2 Планировщик = статическая библиотека, а не виртуальная машина

```
Внутренняя структура итогового exe:

┌────────────────────────────────────────────┐
│  Ваш код (нативный машинный код)             │
│  ├── Определённый на этапе компиляции план   │
│  │   выполнения DAG                         │
│  ├── Инлайненные операции Move/ref/clone     │
│  └── Код освобождения RAII                  │
├────────────────────────────────────────────┤
│  Статическая библиотека Runtime (~200-500КБ)  │
│  ├── Пул потоков (фиксированный размер =     │
│  │   num_workers)                           │
│  ├── Цикл событий (libuv / io_uring)        │
│  ├── Очереди work stealing (только Full      │
│  │   Runtime)                               │
│  ├── Аллокатор памяти (jemalloc / mimalloc) │
│  └── Метаданные рефлексии (загружаются по     │
│      необходимости, не постоянно в памяти)   │
├────────────────────────────────────────────┤
│  Отсутствует:                                │
│  ❌ Интерпретатор байткода                   │
│  ❌ JIT-компилятор                            │
│  ❌ GC                                      │
│  ❌ Виртуальная машина                        │
└────────────────────────────────────────────┘
```

Сравнение:

| Язык | Java | Go | YaoXiang |
|------|------|-----|-----------|
| Артефакт компиляции | байткод | нативный код | нативный код |
| Способ выполнения | JVM-интерпретация/JIT | прямое выполнение | прямое выполнение |
| Размер Runtime | ~200МБ (JVM) | ~1-2МБ (с GC) | **~200-500КБ (без GC)** |
| Управление памятью | GC | GC | **RAII (детерминированное)** |
| Рефлексия | постоянно в памяти | постоянно в памяти | **хранится в exe, загружается по необходимости** |

#### 6.3 Почему производительность планировщика постоянна

**Ключевая идея**: большая часть работы выполняется на этапе компиляции, Runtime только «исполняет».

```
Этап компиляции (однократно, не попадает в Runtime):
    ├── Анализ DAG внутри spawn-блоков: кто от кого зависит
    ├── Топологическая сортировка: определение порядка выполнения
    │   внутри spawn-блоков
    ├── Идентификация параллелизуемых задач: поддеревья без
    │   зависимостей внутри spawn-блоков
    ├── Escape-анализ: ref → Rc или Arc
    ├── Детекция циклов: автоматическое понижение до Weak
    │   или сообщение об ошибке
    └── Инлайнинг: маленькие функции раскрываются напрямую

Runtime (при каждом выполнении, структуры данных фиксированы):
    ├── Распределение задач по пулу потоков в порядке DAG,
    │   определённом на этапе компиляции, для spawn-блоков
    ├── При I/O → приостановка текущей задачи, цикл событий
    │   берёт на себя управление
    ├── Задача готова → возврат в очередь готовых
    └── Вот и всё.
```

**Сам планировщик — это структура данных фиксированного размера**: пул потоков, цикл событий, очереди работ. Никакого динамического роста, никакой адаптивной перекомпиляции, никакого GC-сканирования. Поведение полностью предсказуемо.

На этапе компиляции уже вычислено «что планировать» внутри spawn-блоков, Runtime только «исполняет». Это отличается от tokio — tokio строит цепочку Future динамически в Runtime. DAG в YaoXiang статический, причём только внутри spawn-блоков.

#### 6.4 Рефлексия: хранение без постоянного присутствия в памяти

Метаданные рефлексии генерируются на этапе компиляции и хранятся в отдельной секции exe. При запуске программы не загружаются. При первом запросе рефлексии — mmap по необходимости. Аналогично:

```
Раскладка exe:
  .text     ← ваш код
  .rodata   ← константы
  .reflect  ← метаданные рефлексии (информация о типах,
              сигнатуры функций и т.п.)
              mmap по необходимости, не используется —
              не занимает памяти
```

**Компромисс**: увеличение размера exe (включая данные рефлексии), но нулевые накладные расходы памяти в Runtime, если не используется. Задержка при первом обращении (аналог прогрева JIT), далее — нулевые накладные расходы.

## Компромиссы

### Преимущества

- **Чёткое разделение слоёв**: Embedded / Standard / Full
- **Переиспользование компиляции**: код фронтенда полностью общий
- **Декаплинг через дженерики**: полиморфизм на этапе компиляции, нулевые накладные расходы
- **Согласованность**: синхронность — это частный случай планирования
- **Дружелюбность к встраиваемым системам**: высокая производительность + низкое потребление памяти + быстрый запуск

### Недостатки

- **Начальная сложность**: необходимо определить интерфейс планировщика и несколько вариантов Runtime
- **Привязка на этапе компиляции**: тип планировщика определяется во время компиляции
