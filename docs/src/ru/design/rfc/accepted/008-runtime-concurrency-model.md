# RFC-008: Архитектура параллельной модели Runtime с отделённым планировщиком

> **Статус**: Принято
> **Автор**: Чэнь Сюй (晨煦)
> **Дата создания**: 2025-01-05
> **Последнее обновление**: 2026-05-11 (обрезано + новая двух backend модель, разделение компилятора и runtime, статическая библиотека планировщика, отложенная загрузка рефлексии)

> **См. также**:
> - [RFC-001: Модель 并作 и система обработки ошибок](./001-concurrent-model-error-handling.md)
> - [RFC-011: Проектирование системы обобщённых типов](./011-generic-type-system.md)

## Аннотация

В данном документе определяются ключевые аспекты архитектуры Runtime:

1. **Трёхуровневая архитектура Runtime**: Embedded (немедленное выполнение) → Standard (планирование DAG) → Full (work-stealing)
2. **Разделение компиляции и выполнения**: Этапы компиляции полностью идентичны, различается только способ выполнения в runtime
3. **Двух backend модель**: VM (разработка и отладка) и LLVM AOT (производство и выпуск), поведение полностью идентично
4. **Планировщик = статическая библиотека**: При AOT компиляции планировщик линкуется в exe, примерно 200-500 КБ, без GC
5. **Синхронность — частный случай планирования**: num_workers=1 означает синхронный режим

### Ключевое уточнение: это не Java

```
Java:   .java → .class → JVM (интерпретация/JIT + GC)       ← всегда требуется виртуальная машина
YaoXiang разработка: .yx → IR → выполнение в VM (быстрая итерация, пошаговая отладка)
YaoXiang production: .yx → IR → LLVM → нативный exe (планировщик линкуется внутрь)

VM — инструмент разработки, а не сущность runtime. По аналогии с Go: go run vs go build.
Итоговый exe = ваш нативный код + статическая библиотека планировщика + метаданные рефлексии.
Без интерпретатора, без JIT, без GC.
```

## Мотивация

### Основное противоречие

| Противоречие | Описание |
|--------------|----------|
| Прозрачность vs управляемость | 并作 должен быть поведением по умолчанию, но пользователь должен иметь возможность управления |
| Ядро vs опциональное | DAG является ядром, но WorkStealing — это расширенная возможность при num_workers>1 |
| Однопоточный vs параллельный | В однопоточном режиме 并作 проявляется как асинхронность, синхронность — лишь частный случай планирования |

## Предложение

### 1. Трёхуровневая архитектура Runtime

```
┌──────────────────────────────────────────────────────────────────┐
│                    Этап компиляции (одинаковый для всех режимов)  │
│                                                                  │
│  Исходный код → Лексер → Парсер → Проверка типов → Генерация    │
│                        кода → IR                                 │
│                                                                  │
│  ⚠️ Единый синтаксический парсер, проверка типов, генерация      │
│     кода и вывод IR                                              │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即时执行器        │ │ DAG 调度器     │ │ Full 调度器      │
│ Синхронное       │ │ Ленивое       │ │ Параллельное     │
│ выполнение       │ │ вычисление    │ │ оптимизация      │
│ Без DAG          │ │ Авто 并作     │ │ Work-stealing    │
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| Уровень | Embedded | Standard | Full |
|---------|----------|----------|------|
| Компиляция | одинаковая | одинаковая | одинаковая |
| Режим выполнения | синхронный | ленивый + 并作 | параллельный |
| Потребление памяти | низкое | среднее | высокое |
| Возможности 并作 | нет | авто | авто + параллелизм |
| Ленивое вычисление DAG | нет | ✅ | ✅ |
| WorkStealer | нет | нет | ✅ |

**Embedded Runtime**: ориентирован на WASM/скрипты игр/движки правил. 即时执行器, без DAG, высокая производительность при низком потреблении.

**Standard Runtime**: ориентирован на веб-сервисы/конвейеры данных. Ленивое вычисление DAG + автоматический 并作. При num_workers=1 — однопоточная асинхронность.

**Full Runtime**: ориентирован на научные вычисления/масштабный параллелизм. Standard + WorkStealer для балансировки нагрузки.

### 2. Отделение планировщика: generics + внедрение

Основной принцип: VM не зависит напрямую от конкретного планировщика, использует его через generics параметр `[S]`.

```yaoxiang
# Определение интерфейса планировщика
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# Однопоточный планировщик
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# Многопоточный планировщик
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VM использует планировщик через generics
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**Ключевые моменты**:
- Компиляционная полиморфия, нулевые накладные расходы в runtime
- Не требуются Trait объекты
- Generic ограничение типа `[S: Scheduler]` уже определено в RFC-011

### 3. Синхронность = частный случай планирования

```
❌ Заблуждение: отключение планировщика
✅ Правильно: использование планировщика с одним worker

num_workers = 1 → однопоточный асинхронный планировщик
num_workers > 1 → многопоточный параллельный планировщик

Один и тот же интерфейс планировщика, разница только в конфигурации. Устраняем особые случаи.
```

### 4. Статус DAG

| Уровень | Содержит DAG | Описание |
|---------|-------------|----------|
| Core Runtime | ✅ | Ядро ленивого вычисления |
| Standard Runtime | ✅ | Планировщик DAG |
| Embedded Runtime | ❌ | 即时 выполнение, без DAG |

### 5. Модель выполнения снизу вверх

```
Код пользователя (синхронный синтаксис):
    a = fetch(url0)
    b = fetch(url1)
    print(a)

Компиляционный анализ (снизу вверх):
    print(a) требует a → зависит от fetch(url0)
    fetch(url1) никому не нужен → изолированный DAG

Runtime планирование (от листьев):
    fetch(url0) → print(a)    ← цепочка зависимостей, по порядку
    fetch(url1)                ← изолированный, независимый параллелизм
```

**Ключевые моменты**:
- Обратный анализ зависимостей от "мест, где требуется результат"
- Листовые узлы выполняются параллельно в первую очередь
- Изолированные DAG параллельны независимо, не блокируют основной поток

### 6. Модель компиляции: два backend + статически линкуемый runtime

#### 6.1 Два backend, одно поведение

```
                      ┌─────────────────────┐
                      │   Компиляционный    │
                      │   фронтенд (единый)  │
                      │   Лексер → Парсер    │
                      │   → Проверка типов   │
                      │   → Анализ DAG       │
                      │   → Анализ побега    │
                      │   → Обнаружение циклов│
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │  VM backend       │     │  LLVM backend     │
        │  (разработка)      │     │  (производство)   │
        │                   │     │                   │
        │  Генерация IR/    │     │  Генерация        │
        │  байткода         │     │  нативного кода   │
        │  VM интерпретация │     │  Линковка runtime │
        │  Поддержка        │     │  статической      │
        │  пошаговой        │     │  библиотеки       │
        │  отладки          │     │  Вывод .exe       │
        │  Быстрая итерация │     │  Нулевые накладые │
        └───────────────────┘     │  интерпретации    │
                 │                 └───────────────────┘
                 │
                 ▼
           Поведение полностью идентично
```

**VM backend**: используется при разработке. Изменение кода → мгновенный запуск → пошаговая отладка → быстрая итерация. Поведение полностью идентично финальному exe.

**LLVM backend**: используется при выпуске. AOT компиляция в нативный код, планировщик как статическая библиотека линкуется внутрь. Без интерпретатора, без JIT.

#### 6.2 Планировщик = статическая библиотека, не виртуальная машина

```
Внутренняя структура финального exe:

┌────────────────────────────────────────────┐
│  Ваш код (нативный машинный код)            │
│  ├── DAG план выполнения, определённый       │
│  │   при компиляции                         │
│  ├── Инлайненные операции Move/ref/clone    │
│  └── Код освобождения RAII                  │
├────────────────────────────────────────────┤
│  Статическая библиотека runtime (~200-500КБ)│
│  ├── Пул потоков (фиксированный размер =     │
│  │   num_workers)                           │
│  ├── Цикл событий (libuv / io_uring)        │
│  ├── Очереди work-stealing (только Full      │
│  │   Runtime)                               │
│  ├── Аллокатор памяти (jemalloc / mimalloc) │
│  └── Метаданные рефлексии (отложенная        │
│      загрузка, не постоянно в памяти)       │
├────────────────────────────────────────────┤
│  Отсутствует:                               │
│  ❌ Байткод интерпретатор                    │
│  ❌ JIT компилятор                           │
│  ❌ GC                                      │
│  ❌ Виртуальная машина                       │
└────────────────────────────────────────────┘
```

Сравнение:

| | Java | Go | YaoXiang |
|------|------|-----|-----------|
| Результат компиляции | Байткод | Нативный код | Нативный код |
| Способ выполнения | JVM интерпретация/JIT | Прямое выполнение | Прямое выполнение |
| Размер runtime | ~200 МБ (JVM) | ~1-2 МБ (с GC) | **~200-500 КБ (без GC)** |
| Управление памятью | GC | GC | **RAII (детерминированное)** |
| Рефлексия | Постоянно в памяти | Постоянно в памяти | **Хранится в exe, загружается по требованию** |

#### 6.3 Почему производительность планировщика постоянна

**Ключевое наблюдение**: основная часть работы выполняется при компиляции, runtime только "исполняет".

```
Компиляция (однократно, не в runtime):
    ├── Построение глобального DAG: кто от кого зависит
    ├── Топологическая сортировка: определение порядка выполнения
    ├── Обнаружение изолированных подграфов: параллелизуемые поддеревья
    ├── Анализ побега: ref → Rc или Arc
    ├── Обнаружение циклов: авто downgrade до Weak или ошибка
    └── Инлайнинг: мелкие функции напрямую разворачиваются

Runtime (при каждом выполнении, структуры данных фиксированы):
    ├── Диспетчеризация задач в пул потоков согласно DAG,
    │   определённому при компиляции
    ├── При I/O → приостановка текущей задачи, цикл событий берёт управление
    ├── Задача готова → возврат в очередь готовности
    └── Вот и всё.

**Сам планировщик — структура данных фиксированного размера**: пул потоков, цикл событий, рабочие очереди. Нет динамического роста, нет адаптивной реоптимизации, нет GC сканирования. Поведение полностью предсказуемо.

При компиляции уже вычислено "что планировать", runtime только "исполняет". Это отличается от tokio — tokio динамически строит цепочки Future в runtime. DAG YaoXiang статичен.

#### 6.4 Рефлексия: хранится, не постоянно загружена

Метаданные рефлексии генерируются при компиляции, хранятся в отдельном сегменте (section) exe. При запуске программы не загружаются. При первом запросе рефлексии загружаются в память через mmap по требованию. По аналогии:

```
Структура exe:
  .text     ← ваш код
  .rodata   ← константы
  .reflect  ← метаданные рефлексии (информация о типах,
              сигнатуры функций и т.д.)
              mmap загрузка по требованию, не используется —
              не занимает память
```

**Компромисс**: увеличение размера exe (с данными рефлексии), но нулевые накладые расходы памяти в runtime при отсутствии обращений. При первом доступе есть задержка загрузки (аналогично JIT warm-up), в дальнейшем нулевые накладые расходы.

## Компромиссы

### Преимущества

- **Чёткое разделение**: три уровня — Embedded / Standard / Full
- **Переиспользование компиляции**: код фронтенда полностью общий
- **Generic decoupling**: компиляционный полиморфизм, нулевые накладые расходы
- **Согласованность**: синхронность — лишь частный случай планирования
- **Дружелюбность к встраиваемым системам**: высокая производительность + низкое потребление памяти + быстрый запуск

### Недостатки

- **Начальная сложность**: требуется определение интерфейса планировщика и множества вариантов runtime
- **Компиляционная привязка**: тип планировщика определяется при компиляции
