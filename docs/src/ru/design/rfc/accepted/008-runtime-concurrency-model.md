# RFC-008：Модель параллелизма Runtime и проектирование с разделением планировщика

> **⚠️ Пояснение о согласовании**：Данный документ согласован с [RFC-024 Новая модель параллелизма](/reference/language-spec/concurrency.md). Старый анализ DAG всей программы, аннотации `@block`/`@eager`, модель уровней L1/L2/L3 заменены на параллельный примитив блока `spawn {}`. Анализ DAG теперь применяется только внутри блока `spawn {}`.

> **Справочные материалы**:
> - [RFC-011: Проектирование системы обобщённых типов](./011-generic-type-system.md)
> - [Спецификация модели параллелизма（RFC-024）](/reference/language-spec/concurrency.md)

## Аннотация

Данный документ определяет ключевой дизайн архитектуры Runtime:

1. **Трёхуровневая архитектура Runtime**: Embedded (немедленное выполнение) → Standard (spawn + DAG-планирование) → Full (work-stealing)
2. **Разделение компиляции и выполнения**: Этапы компиляции полностью идентичны, различие только в способе выполнения во время исполнения
3. **Двухбэкендная модель**: VM (разработка и отладка) и LLVM AOT (производство и выпуск), поведение полностью идентично
4. **Планировщик = статическая библиотека**: При AOT-компиляции планировщик линкуется в exe, ~200-500KB, без GC
5. **Синхронность — частный случай планирования**: num_workers=1 означает синхронный режим

### Ключевое уточнение: это не Java

```
Java:   .java → .class → JVM（интерпретация/JIT + GC）        ← всегда требуется виртуальная машина
YaoXiang 开发: .yx → IR → VM 执行（快速迭代、步进调试）
YaoXiang 生产: .yx → IR → LLVM → 原生 exe（调度器链接进去）

VM 是开发工具，不是运行时本质。跟 Go 的 go run vs go build 一样。
最终 exe = 你的原生代码 + 调度器静态库 + 反射元数据。没有解释器，没有 JIT，没有 GC。

Java:   .java → .class → JVM（интерпретация/JIT + GC）        ← всегда требуется виртуальная машина
YaoXiang разработка: .yx → IR → выполнение в VM（быстрая итерация, пошаговая отладка）
YaoXiang production: .yx → IR → LLVM → нативный exe（планировщик линкуется внутрь）

VM — это инструмент разработки, а не суть runtime. Как go run vs go build в Go.
Итоговый exe = ваш нативный код + статическая библиотека планировщика + метаданные рефлексии. Никакого интерпретатора, никакого JIT, никакого GC.
```

## Мотивация

### Основное противоречие

| Противоречие | Описание |
|------|------|
| Прозрачность vs Управляемость | Блок spawn обеспечивает явный параллельный контроль, обычный код выполняется последовательно |
| Ядро vs Опциональность | spawn — это основной параллельный примитив, WorkStealing — это расширенная функция при num_workers>1 |
| Однопоточность vs Параллелизм | В однопоточном режиме параллелизм проявляется как асинхронность, синхронность — это частный случай планирования |

## Предложение

### 1. Трёхуровневая архитектура Runtime

```
┌──────────────────────────────────────────────────────────────────┐
│                    Этап компиляции（все режимы идентичны）         │
│                                                                  │
│  Исходный код → Lexer → Parser → TypeCheck → Codegen → IR       │
│                                                                  │
│  ⚠️ Единый синтаксический парсинг, проверка типов,               │
│     генерация кода, вывод IR                                    │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即时执行器        │ │ spawn + DAG   │ │ Full 调度器      │
│ 同步执行         │ │ spawn 块内并发 │ │ 并行优化         │
│ 无 spawn 支持    │ │ 自动并发      │ │ 工作窃取         │
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| Уровень | Embedded | Standard | Full |
|------|----------|----------|------|
| Компиляция | Идентична | Идентична | Идентична |
| Режим выполнения | Синхронный | Параллелизм внутри блока spawn | Параллельный |
| Использование памяти | Низкое | Среднее | Высокое |
| Возможности параллелизма | Нет | Внутри блока spawn | Внутри блока spawn + параллелизм |
| Поддержка spawn | ❌ | ✅ | ✅ |
| Анализ DAG | Нет | Внутри блока spawn | Внутри блока spawn |
| WorkStealer | Нет | Нет | ✅ |

**Embedded Runtime**: Цель — WASM/скрипты игр/движок правил. Немедленный исполнитель, без поддержки spawn, высокопроизводительный с малым потреблением.

**Standard Runtime**: Цель — веб-сервисы/конвейеры данных. Поддержка блоков `spawn {}`, автоматический параллелизм и анализ DAG внутри блока spawn. num_workers=1 — однопоточный асинхронный режим.

**Full Runtime**: Цель — научные вычисления/масштабный параллелизм. Standard + балансировка нагрузки WorkStealer.

### 2. Разделение планировщика: обобщённые типы + инъекция

Основной принцип: VM напрямую не зависит от конкретного планировщика, использует обобщённый параметр `[S]` для вызова.

```yaoxiang
# Определение интерфейса планировщика
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# Однопоточный планировщик
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# Многопоточный планировщик
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VM использует планировщик через обобщённые типы
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**Ключевые моменты**:
- Компиляционный полиморфизм, нулевые накладные расходы во время выполнения
- Объекты trait не требуются
- Ограничение обобщённого типа `[S: Scheduler]` уже определено в RFC-011

### 3. Синхронность = частный случай планирования

```
❌ Заблуждение: отключить планировщик
✅ Правильно: использовать планировщик с одним worker'ом

num_workers = 1 → однопоточный асинхронный планировщик
num_workers > 1 → многопоточный параллельный планировщик

Тот же интерфейс планировщика, только разная конфигурация. Устраняем особые случаи.
```

### 4. Позиция DAG

> **Важное изменение**: Анализ DAG больше не применяется ко всей программе, а только внутри блока `spawn {}`. Обычный код (вне блока spawn) выполняется последовательно и не требует анализа DAG.

| Уровень | Поддержка spawn | Область анализа DAG | Описание |
|------|-----------|-------------|------|
| Core Runtime | ✅ | Внутри блока spawn | Ядро параллелизма |
| Standard Runtime | ✅ | Внутри блока spawn | spawn + DAG-планирование |
| Embedded Runtime | ❌ | Нет | Немедленное выполнение, без параллелизма |

### 5. Модель выполнения снизу вверх（внутри блока spawn）

> **Важное изменение**: Анализ DAG снизу вверх выполняется только внутри блока `spawn {}`, больше не применяется ко всей программе.

```
Пользовательский код（параллелизм внутри блока spawn）：
    (a, b) = spawn {
        fetch(url0),
        fetch(url1)
    }
    print(a)

Компиляционный анализ（снизу вверх внутри блока spawn）：
    fetch(url0) и fetch(url1) не имеют взаимозависимостей → можно выполнять параллельно
    print(a) вне блока spawn → последовательное выполнение, ожидание завершения spawn

Планирование во время выполнения（внутри блока spawn, начиная с листьев）：
    fetch(url0) ┐
                ├→ параллельное выполнение
    fetch(url1) ┘
    print(a)                       ← вне блока spawn, последовательное выполнение
```

**Ключевые моменты**:
- Анализ зависимостей снизу вверх ограничен только внутри блока `spawn {}`
- Задачи без зависимостей внутри блока spawn выполняются параллельно
- Код вне блока spawn выполняется последовательно, ожидая завершения блока spawn

### 6. Модель компиляции: двухбэкендная + статически линкуемый Runtime

#### 6.1 Два бэкенда, одно поведение

```
                      ┌─────────────────────┐
                      │   Компиляционный     │
                      │   фронтенд（единый）   │
                      │   Lexer → Parser     │
                      │   → TypeCheck        │
                      │   → анализ DAG       │
                      │   внутри блока spawn │
                      │   → анализ побега     │
                      │   → обнаружение циклов│
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │  VM бэкенд（разработка）│     │ LLVM бэкенд（производство）│
        │                    │     │                    │
        │  Генерация IR/     │     │  Генерация         │
        │  байткода          │     │  нативного кода    │
        │  VM интерпретирует  │     │  Линковка          │
        │  Поддержка          │     │  статической       │
        │  пошаговой отладки  │     │  библиотеки runtime│
        │  Быстрая итерация   │     │  Вывод .exe        │
        │                    │     │  Нулевые накладные │
        │                    │     │  расходы интерпретации│
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           Поведение полностью一致           Поведение полностью一致
```

**VM бэкенд**: Используется при разработке. Изменить код → немедленное выполнение → пошаговая отладка → быстрая итерация. Поведение полностью идентично итоговому exe.

**LLVM бэкенд**: Используется при выпуске. AOT-компиляция в нативный код, планировщик как статическая библиотека линкуется внутрь. Никакого интерпретатора, никакого JIT.

#### 6.2 Планировщик = статическая библиотека, а не виртуальная машина

```
Внутренняя структура итогового exe：

┌────────────────────────────────────────────┐
│  Ваш код（нативный машинный код）            │
│  ├── План выполнения DAG, определённый       │
│  │   при компиляции                         │
│  ├── Встроенные операции Move/ref/clone      │
│  └── Код освобождения RAII                  │
├────────────────────────────────────────────┤
│  Статическая библиотека runtime（~200-500KB）│
│  ├── Пул потоков（фиксированный размер =      │
│  │   num_workers）                          │
│  ├── Цикл событий（libuv / io_uring）       │
│  ├── Очереди work-stealing（только Full      │
│  │   Runtime）                              │
│  ├── Аллокатор памяти（jemalloc / mimalloc）│
│  └── Метаданные рефлексии（загрузка по        │
│      требованию, не постоянно в памяти）      │
├────────────────────────────────────────────┤
│  Нет：                                      │
│  ❌ Интерпретатор байткода                  │
│  ❌ JIT-компилятор                          │
│  ❌ GC                                      │
│  ❌ Виртуальная машина                      │
└────────────────────────────────────────────┘
```

Сравнение：

| Язык | Java | Go | YaoXiang |
|------|------|-----|-----------|
| Результат компиляции | Байткод | Нативный код | Нативный код |
| Способ выполнения | Интерпретация/JIT в JVM | Прямое выполнение | Прямое выполнение |
| Размер runtime | ~200MB（JVM） | ~1-2MB（с GC） | **~200-500KB（без GC）** |
| Управление памятью | GC | GC | **RAII（детерминировано）** |
| Рефлексия | Постоянно в памяти | Постоянно в памяти | **Хранится в exe, загрузка по требованию** |

#### 6.3 Почему производительность планировщика постоянна

**Ключевая идея**: Основная часть работы выполняется при компиляции, runtime только "выполняет".

```
Компиляция（однократно, не входит в runtime）：
    ├── Анализ DAG внутри блока spawn: кто от кого зависит
    ├── Топологическая сортировка: определение порядка
    │   выполнения внутри блока spawn
    ├── Идентификация параллельных задач: поддеревья
    │   без зависимостей внутри блока spawn
    ├── Анализ побега: ref → Rc или Arc
    ├── Обнаружение циклов: автоматическое понижение
    │   до Weak или ошибка
    └── Встраивание: мелкие функции просто раскрываются

Runtime（при каждом выполнении, структуры данных фиксированы）：
    ├── Распределение задач в пул потоков по порядку DAG,
    │   определённому при компиляции внутри блока spawn
    ├── При встрече I/O → приостановка текущей задачи,
    │   цикл событий берёт управление
    ├── Готовность задачи → возврат в очередь готовности
    └── Вот и всё.
```

**Сам планировщик — это структура данных фиксированного размера**: пул потоков, цикл событий, рабочие очереди. Нет динамического роста, нет адаптивной реоптимизации, нет GC-сканирования. Поведение полностью предсказуемо.

При компиляции уже вычислено, "что планировать" внутри блока spawn, runtime только "выполняет". Это отличается от tokio — tokio динамически строит цепочки Future во время выполнения. DAG YaoXiang статический и ограничен внутри блока spawn.

#### 6.4 Рефлексия: хранить, не резидентность

Метаданные рефлексии генерируются при компиляции и хранятся в отдельном сегменте (секции) exe. При запуске программы не загружаются. При первом запросе рефлексии загружаются в память по требованию через mmap. Аналогично:

```
Структура exe：
  .text     ← ваш код
  .rodata   ← константы
  .reflect  ← метаданные рефлексии（информация о типах,
              сигнатуры функций и т.д.）
              mmap загрузка по требованию, без доступа не занимает память
```

**Компромисс**: Увеличение размера exe (с данными рефлексии), но нулевые затраты памяти во время выполнения без обращения. При первом доступе есть задержка загрузки (как JIT-прогрев), далее нулевые затраты.

## Компромиссы

### Преимущества

- **Чёткое разделение уровней**: Embedded / Standard / Full три уровня
- **Переиспользование компиляции**: Фронтенд код полностью общий
- **Обобщённое разделение**: Компиляционный полиморфизм, нулевые накладные расходы
- **Согласованность**: Синхронность — это частный случай планирования
- **Дружелюбие к встраиваемым системам**: Высокая производительность + низкое потребление памяти + быстрый запуск

### Недостатки

- **Начальная сложность**: Требуется определить интерфейс планировщика и несколько вариантов runtime
- **Компиляционная привязка**: Тип планировщика определяется при компиляции

## Журнал принятия решений

| Решение | Принято | Дата |
|------|------|------|
| Схема разделения планировщика | Обобщённые типы + инъекция | 2025-01-05 |
| Однопоточный режим | Синхронность — частный случай планирования | 2025-01-05 |
| Реализация асинхронности | DAG естественно поддерживает | 2025-01-05 |
| WorkStealer | Расширенная функция Full Runtime | 2025-01-05 |
| Дизайн для встраиваемых систем | Немедленное выполнение, без DAG-планирования | 2025-01-05 |
| Этап компиляции | Все runtime используют общий фронтенд | 2025-01-05 |
| Разделение уровней выполнения | Embedded / Standard / Full | 2025-01-05 |
| Ограничения типов | Определено в RFC-011 | 2025-01-25 |
| Построение графа зависимостей | Статический граф зависимостей, определён при компиляции | 2025-01-05 |
| Двухбэкендная модель | VM（разработка и отладка）+ LLVM AOT（производство）， согласованное поведение | 2026-05-11 |
| Форма планировщика | Статическая библиотека, линкуется в exe, ~200-500KB, без GC | 2026-05-11 |

| Производительность планировщика | Анализ DAG выполняется при компиляции, runtime только выполняет | 2026-05-11 |
| Согласование области DAG | Анализ DAG только внутри блока spawn, согласовано с RFC-024 | 2026-06-05 |
| Обновление трёхуровневой архитектуры | Embedded без spawn, Standard с поддержкой spawn | 2026-06-05 |
