# RFC-008: Проект разделения модели параллелизма Runtime и планировщика

> **Статус**: Принято
> **Автор**: Чэнь Сюй (晨煦)
> **Дата создания**: 2025-01-05
> **Последнее обновление**: 2026-05-11 (обрезка + новая двух_backend модель, разделение компилятора и runtime, статическая библиотека планировщика, отложенная загрузка рефлексии)

> **См. также**:
> - [RFC-001: Модель 并作 и система обработки ошибок](./001-concurrent-model-error-handling.md)
> - [RFC-011: Проект системы обобщённых типов](./011-generic-type-system.md)

## Аннотация

В данном документе определяются ключевые аспекты архитектуры Runtime:

1. **Трёхуровневая архитектура Runtime**: Embedded (синхронное выполнение) → Standard (DAG-планирование) → Full (work-stealing)
2. **Разделение компиляции и выполнения**: Этапы компиляции полностью идентичны, различается только способ выполнения в runtime
3. **Двух_backend модель**: VM (разработка и отладка) и LLVM AOT (продакшен), поведение полностью идентично
4. **Планировщик = статическая библиотека**: При AOT-компиляции планировщик линкуется в exe, ~200-500KB, без GC
5. **Синхронность — частный случай планирования**: num_workers=1 означает синхронный режим

### Ключевое уточнение: это не Java

```
Java:   .java → .class → JVM (интерпретация/JIT + GC)      ← виртуальная машина всегда нужна
YaoXiang разработка: .yx → IR → выполнение в VM (быстрая итерация, пошаговая отладка)
YaoXiang продакшен: .yx → IR → LLVM → нативный exe (планировщик линкуется внутрь)

VM — инструмент разработки, а не суть runtime. По аналогии с Go: go run vs go build.
Итоговый exe = ваш нативный код + статическая библиотека планировщика + метаданные рефлексии. Без интерпретатора, без JIT, без GC.
```

## Мотивация

### Основное противоречие

| Противоречие | Описание |
|------|------|
| Прозрачность vs управляемость | Параллелизм должен быть поведением по умолчанию, но пользователь должен иметь возможность контроля |
| Ядро vs опциональность | DAG — ядро, но WorkStealing — продвинутая возможность при num_workers>1 |
| Однопоточность vs параллелизм | В однопоточном режиме 并作 проявляется как асинхронность, синхронность — лишь частный случай планирования |

## Предложение

### 1. Трёхуровневая архитектура Runtime

```
┌──────────────────────────────────────────────────────────────────┐
│                    Этап компиляции (одинаков для всех режимов)    │
│                                                                  │
│  Source Code → Lexer → Parser → TypeCheck → Codegen → IR        │
│                                                                  │
│  ⚠️ Единая система разбора, проверки типов, кодогенерации и IR    │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即时执行器         │ │ DAG 调度器     │ │ Full 调度器       │
│ Синхронное        │ │ Отложенное     │ │ Параллельная      │
│ выполнение        │ │ вычисление     │ │ оптимизация       │
│ Без DAG           │ │ Автоматический │ │ Work-stealing     │
│ планирования      │ │ 并作           │ │                   │
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| Уровень | Embedded | Standard | Full |
|------|----------|----------|------|
| Компиляция | Одинаковая | Одинаковая | Одинаковая |
| Режим выполнения | Синхронный | Отложенный+并作 | Параллельный |
| Потребление памяти | Низкое | Среднее | Высокое |
| Способность к 并作 | Нет | Автоматическая | Автоматическая+параллельная |
| Отложенное вычисление DAG | Нет | ✅ | ✅ |
| WorkStealer | Нет | Нет | ✅ |

**Embedded Runtime**: целевые платформы — WASM/скрипты игр/движки правил. 即时执行器 без DAG, высокая производительность при низком потреблении.

**Standard Runtime**: целевые платформы — веб-сервисы/конвейеры данных. Отложенное вычисление DAG + автоматический 并作. При num_workers=1 — однопоточный асинхронный режим.

**Full Runtime**: целевые платформы — научные вычисления/крупномасштабный параллелизм. Standard + WorkStealer для балансировки нагрузки.

### 2. Разделение планировщика: generics + инъекция

Основной принцип: VM напрямую не зависит от конкретного планировщика, а вызывает его через обобщённый параметр `[S]`.

```yaoxiang
# Определение интерфейса планировщика
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# Однопоточный планировщик
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# Многопоточный планировщик
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VM использует планировщик через generics
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**Ключевые моменты**:
- Компиляционный полиморфизм, нулевые накладные расходы в runtime
- Не требуется Trait-объект
- Ограничение обобщённого типа `[S: Scheduler]` определено в RFC-011

### 3. Синхронность = частный случай планирования

```
❌ Заблуждение: отключить планировщик
✅ Правильно: использовать планировщик с одним worker

num_workers = 1 → однопоточный асинхронный планировщик
num_workers > 1 → многопоточный параллельный планировщик

Один и тот же интерфейс планировщика, разная конфигурация. Никаких исключений.
```

### 4. Статус DAG

| Уровень | Содержит DAG | Описание |
|------|----------|------|
| Core Runtime | ✅ | Ядро отложенного вычисления |
| Standard Runtime | ✅ | DAG-планировщик |
| Embedded Runtime | ❌ | 即时 выполнение, без DAG |

### 5. Модель выполнения снизу вверх

```
Пользовательский код (синхронный синтаксис):
    a = fetch(url0)
    b = fetch(url1)
    print(a)

Анализ при компиляции (снизу вверх):
    print(a) требует a → зависит от fetch(url0)
    fetch(url1) никому не нужен → изолированный DAG

Планирование при выполнении (от листьев):
    fetch(url0) → print(a)    ← цепочка зависимостей, по порядку
    fetch(url1)                ← изолированный узел, независимый параллелизм
```

**Ключевые моменты**:
- Обратный анализ зависимостей от «мест, где нужен результат»
- Листовые узлы выполняются параллельно в первую очередь
- Изолированные DAG выполняются параллельно, не блокируя основной поток

### 6. Модель компиляции: два backend + статически линкуемый runtime

#### 6.1 Два backend, одно поведение

```
                      ┌─────────────────────┐
                      │   Фронтенд компилятора (единый) │
                      │   Lexer → Parser     │
                      │   → TypeCheck        │
                      │   → DAG-анализ       │
                      │   → Анализ побега     │
                      │   → Обнаружение циклов│
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │   VM backend (разработка)│     │  LLVM backend (продакшен) │
        │                   │     │                   │
        │  Генерация IR/байткода│     │  Генерация нативного кода│
        │  Интерпретация VM   │     │  Линковка статической библиотеки runtime│
        │  Поддержка пошаговой отладки││  Вывод .exe           │
        │  Быстрая итерация   │     │  Нулевые накладные расходы интерпретации│
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
          Поведение полностью一致         Поведение полностью一致
```

**VM backend**: используется при разработке. Изменение кода → мгновенный запуск → пошаговая отладка → быстрая итерация. Поведение полностью идентично финальному exe.

**LLVM backend**: используется при выпуске. AOT-компиляция в нативный код, планировщик как статическая библиотека линкуется внутрь. Без интерпретатора, без JIT.

#### 6.2 Планировщик = статическая библиотека, а не виртуальная машина

```
Внутренняя структура итогового exe:

┌────────────────────────────────────────────┐
│  Ваш код (нативный машинный код)            │
│  ├── DAG-план выполнения, определённый при компиляции│
│  ├── Инлайненные операции Move/ref/clone    │
│  └── Код освобождения RAII                  │
├────────────────────────────────────────────┤
│  Статическая библиотека runtime (~200-500KB) │
│  ├── Пул потоков (фиксированный размер = num_workers)│
│  ├── Цикл событий (libuv / io_uring)        │
│  ├── Очереди work-stealing (только Full Runtime)│
│  ├── Аллокатор памяти (jemalloc / mimalloc) │
│  └── Метаданные рефлексии (загружаются по требованию, не постоянно в памяти)│
├────────────────────────────────────────────┤
│  Отсутствует:                               │
│  ❌ Интерпретатор байткода                  │
│  ❌ JIT-компилятор                          │
│  ❌ GC                                     │
│  ❌ Виртуальная машина                      │
└────────────────────────────────────────────┘
```

Сравнение:

| | Java | Go | YaoXiang |
|------|------|-----|-----------|
| Результат компиляции | Байткод | Нативный код | Нативный код |
| Способ выполнения | Интерпретация/JIT в JVM | Прямое выполнение | Прямое выполнение |
| Размер runtime | ~200MB (JVM) | ~1-2MB (с GC) | **~200-500KB (без GC)** |
| Управление памятью | GC | GC | **RAII (детерминированное)** |
| Рефлексия | Постоянно в памяти | Постоянно в памяти | **Хранится в exe, загружается по требованию** |

#### 6.3 Почему производительность планировщика постоянна

**Ключевой инсайт**: основная работа выполняется на этапе компиляции, runtime только «исполняет».

```
Этап компиляции (однократно, не в runtime):
    ├── Построение глобального DAG: кто от кого зависит
    ├── Топологическая сортировка: определение порядка выполнения
    ├── Идентификация изолированных поддеревьев: что можно распараллелить
    ├── Анализ побега: ref → Rc или Arc
    ├── Обнаружение циклов: автоснижение до Weak или ошибка
    └── Инлайнинг: мелкие функции раскрываются напрямую

Этап выполнения (при каждом запуске, структуры данных фиксированы):
    ├── Диспетчеризация задач в пул потоков согласно DAG, определённому при компиляции
    ├── При I/O → приостановка текущей задачи, цикл событий берёт управление
    ├── Задача готова → возврат в очередь готовности
    └── Вот и всё.
```

**Сам планировщик — структура данных фиксированного размера**: пул потоков, цикл событий, рабочие очереди. Нет динамического роста, нет адаптивной реоптимизации, нет GC-сканирования. Поведение полностью предсказуемо.

При компиляции уже решено «что планировать», runtime только «исполняет». Это отличается от tokio — tokio динамически строит цепочки Future в runtime. DAG YaoXiang статичен.

#### 6.4 Рефлексия: хранится, не резидентна

Метаданные рефлексии генерируются при компиляции и хранятся в отдельном сегменте (section) exe. При запуске программы не загружаются. При первом запросе рефлексии загружаются в память по требованию через mmap. Аналогично:

```
Структура exe:
  .text     ← ваш код
  .rodata   ← константы
  .reflect  ← метаданные рефлексии (информация о типах, сигнатуры функций и т.д.)
              загрузка через mmap по требованию, без обращения не занимает память
```

**Компромисс**: увеличение размера exe (с данными рефлексии), но нулевые накладные расходы памяти при отсутствии обращений в runtime. Первое обращение имеет задержку загрузки (аналогично JIT warmup), впоследствии — нулевые накладные расходы.



```
src/
├── core/                    # Общий для всех runtime
│   ├── value.rs
│   ├── allocator.rs
│   └── ownership.rs
├── frontend/                # Общий для всех backend
│   ├── lexer/
│   ├── parser/
│   ├── typecheck/
│   └── dag/                 # ★ DAG-анализ (этап компиляции)
│       ├── builder.rs       #   Построение графа зависимостей
│       ├── escape.rs        #   Анализ побега (ref → Rc/Arc)
│       ├── cycle.rs         #   Обнаружение циклов + автоснижение
│       └── topology.rs      #   Топологическая сортировка
├── codegen/                 # Кодогенерация
│   ├── ir.rs                # Определение IR (общий для VM и LLVM)
│   ├── vm_backend/          # VM backend (разработка и отладка)
│   │   ├── bytecode.rs
│   │   └── compiler.rs
│   └── llvm_backend/        # LLVM backend (продакшен)
│       └── compiler.rs
├── embedded/                # 🟢 Embedded Runtime
│   └── executor.rs
├── runtime/                 # 🔵 Статическая библиотека runtime (линкуется в exe)
│   ├── thread_pool.rs       #   Пул потоков фиксированного размера
│   ├── event_loop.rs        #   Цикл событий I/O (libuv/io_uring)
│   ├── dag_executor.rs      #   Выполнение по DAG этапа компиляции
│   └── scheduler/
│       ├── single_thread.rs
│       └── multi_thread.rs
├── full/                    # 🟣 Full Runtime (опциональная линковка)
│   └── work_stealer.rs      #   Work-stealing
├── reflect/                 # Метаданные рефлексии
│   ├── metadata.rs          #   Генерация метаданных (этап компиляции)
│   └── loader.rs            #   Загрузка по требованию (runtime)
└── vm/                      # VM-интерпретатор (только для разработки)
    └── executor.rs
```

## Компромиссы

### Преимущества

- **Чёткая стратификация**: три уровня — Embedded / Standard / Full
- **Переиспользование компиляции**: код фронтенда полностью общий
- **Обобщённое разделение**: compile-time полиморфизм, нулевые накладные расходы
- **Согласованность**: синхронность — лишь частный случай планирования
- **Дружелюбность к встраиваемым системам**: высокая производительность + низкое потребление памяти + быстрый запуск

### Недостатки

- **Начальная сложность**: требуется определить интерфейс планировщика и несколько вариантов runtime
- **Compile-time связывание**: тип планировщика определяется на этапе компиляции
