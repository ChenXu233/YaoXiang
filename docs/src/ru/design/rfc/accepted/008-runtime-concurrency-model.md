# RFC-008：Модель параллелизма среды выполнения и развязка планировщика

> **Статус**: Принято
> **Автор**: Чэнь Сюй
> **Дата создания**: 2025-01-05
> **Последнее обновление**: 2026-05-11 (обрезка + новая двухбэкендная модель, разделение компилятора и среды выполнения, статическая библиотека планировщика, рефлексия по требованию)

> **Ссылки**:
> - [RFC-001: Модель 并作 и система обработки ошибок](./001-concurrent-model-error-handling.md)
> - [RFC-011: Проектирование системы generics](./011-generic-type-system.md)

## Аннотация

В данном документе определены ключевые аспекты архитектуры среды выполнения (Runtime):

1. **Трёхуровневая архитектура среды выполнения**: Embedded (немедленное выполнение) → Standard (DAG-планирование) → Full (work-stealing)
2. **Разделение компиляции и выполнения**: Этап компиляции полностью идентичен, различие только в способе выполнения в runtime
3. **Двухбэкендная модель**: VM (разработка и отладка) и LLVM AOT (продакшен), поведение полностью идентично
4. **Планировщик = статическая библиотека**: При AOT-компиляции планировщик линкуется в exe, ~200-500KB, без GC
5. **Синхронность — лишь частный случай планирования**: num_workers=1 означает синхронный режим

### Ключевое уточнение: это не Java

```
Java:   .java → .class → JVM (интерпретация/JIT + GC)      ← виртуальная машина всегда нужна
YaoXiang разработка: .yx → IR → выполнение в VM (быстрая итерация, пошаговая отладка)
YaoXiang продакшен: .yx → IR → LLVM → нативный exe (планировщик линкуется внутрь)

VM — инструмент разработки, а не сущность среды выполнения. Аналогично Go: go run vs go build.
Итоговый exe = ваш нативный код + статическая библиотека планировщика + метаданные рефлексии.
Никакого интерпретатора, никакого JIT, никакого GC.
```

## Мотивация

### Основное противоречие

| Противоречие | Описание |
|------|------|
| Прозрачность vs управляемость | Параллелизм должен быть поведением по умолчанию, но пользователь должен иметь контроль |
| Ядро vs опционально | DAG — ядро, но WorkStealing — продвинутая характеристика при num_workers>1 |
| Однопоточность vs параллелизм | В однопоточном режиме параллелизм проявляется как асинхронность, синхронность — лишь частный случай планирования |

## Предложение

### 1. Трёхуровневая архитектура среды выполнения

```
┌──────────────────────────────────────────────────────────────────┐
│                    Этап компиляции (одинаков для всех режимов)   │
│                                                                  │
│  Source Code → Lexer → Parser → TypeCheck → Codegen → IR        │
│                                                                  │
│  ⚠️ Единая система разбора, проверки типов, генерации кода и IR  │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即时执行器        │ │ DAG 调度器    │ │ Full 调度器      │
│ Синхронное       │ │ Ленивое       │ │ Параллельное     │
│ выполнение       │ │ вычисление    │ │ оптимизация      │
│ Без DAG          │ │ Автопараллел. │ │ Work-stealing    │
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| Уровень | Embedded | Standard | Full |
|------|----------|----------|------|
| Компиляция | одинакова | одинакова | одинакова |
| Режим выполнения | синхронный | ленивый+параллельный | параллельный |
| Потребление памяти | низкое | среднее | высокое |
| Возможности параллелизма | нет | авто | авто+параллельность |
| Ленивое вычисление DAG | нет | ✅ | ✅ |
| WorkStealer | нет | нет | ✅ |

**Embedded Runtime**: для WASM/скриптов игр/движков правил. Немедленный исполнитель, без DAG, высокопроизводительный и компактный.

**Standard Runtime**: для веб-сервисов/конвейеров данных. Ленивое вычисление DAG + автопараллелизм. num_workers=1 — однопоточный асинхронный режим.

**Full Runtime**: для научных вычислений/масштабного параллелизма. Standard + WorkStealer для балансировки нагрузки.

### 2. Развязка планировщика: generics + инъекция

Основной принцип: VM напрямую не зависит от конкретного планировщика, использует его через generics-параметр `[S]`.

```yaoxiang
# Определение интерфейса планировщика
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# Однопоточный планировщик
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# Многопоточный планировщик
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VM использует планировщик через generics
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**Ключевые моменты**:
- Полиморфизм на этапе компиляции, нулевые накладные расходы в runtime
- Не требуются объекты-Trait
- Ограничение generics-типа `[S: Scheduler]` уже определено в RFC-011

### 3. Синхронность = частный случай планирования

```
❌ Заблуждение: отключить планировщик
✅ Правильно: использовать планировщик с одним worker

num_workers = 1 → однопоточное асинхронное планирование
num_workers > 1 → многопоточное параллельное планирование

Единый интерфейс планировщика, разница только в конфигурации. Никаких особых случаев.
```

### 4. Место DAG

| Уровень | Содержит DAG | Описание |
|------|----------|------|
| Core Runtime | ✅ | Ядро ленивого вычисления |
| Standard Runtime | ✅ | DAG-планировщик |
| Embedded Runtime | ❌ | Немедленное выполнение, без DAG |

### 5. Модель выполнения снизу вверх

```
Пользовательский код (синхронный синтаксис):
    a = fetch(url0)
    b = fetch(url1)
    print(a)

Анализ на этапе компиляции (снизу вверх):
    print(a) требует a → зависит от fetch(url0)
    fetch(url1) никому не нужен → изолированный DAG

Планирование в runtime (от листьев):
    fetch(url0) → print(a)    ← цепочка зависимостей, по порядку
    fetch(url1)                ← изолированный узел, независимый параллелизм
```

**Ключевые моменты**:
- Обратный анализ зависимостей от «мест, где нужен результат»
- Листовые узлы выполняются параллельно в первую очередь
- Изолированные DAG выполняются параллельно, не блокируя основной поток

### 6. Модель компиляции: двухбэкендная + статически линкуемая среда выполнения

#### 6.1 Два бэкенда, одно поведение

```
                      ┌─────────────────────┐
                      │   Компиляционный    │
                      │   фронтенд (единый)  │
                      │   Lexer → Parser    │
                      │   → TypeCheck       │
                      │   → DAG-анализ      │
                      │   → Escape-анализ   │
                      │   → Цикло-детекция  │
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │  VM-бэкенд (разр.) │     │ LLVM-бэкенд       │
        │                    │     │ (продакшен)       │
        │  Генерация IR/     │     │  Генерация        │
        │  байткода          │     │  нативного кода   │
        │  Интерпретация     │     │  Линковка static   │
        │  в VM              │     │  library runtime   │
        │  Поддержка         │     │  Вывод .exe       │
        │  пошаговой отладки │     │  Нулевые наклад.  │
        │  Быстрая итерация  │     │  интерпретации    │
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           Поведение                  Поведение
           полностью                   полностью
           идентично                   идентично
```

**VM-бэкенд**: для разработки. Изменил код → сразу запустил → пошаговая отладка → быстрая итерация. Поведение полностью идентично финальному exe.

**LLVM-бэкенд**: для релиза. AOT-компиляция в нативный код, планировщик линкуется как статическая библиотека. Никакого интерпретатора, никакого JIT.

#### 6.2 Планировщик = статическая библиотека, а не виртуальная машина

```
Внутренняя структура итогового exe:

┌────────────────────────────────────────────┐
│  Ваш код (нативный машинный код)           │
│  ├── План выполнения DAG, определённый     │
│  │   на этапе компиляции                    │
│  ├── Инлайн-операции Move/ref/clone         │
│  └── Код освобождения RAII                 │
├────────────────────────────────────────────┤
│  Статическая библиотека runtime (~200-500KB)│
│  ├── Thread pool (фиксированный размер      │
│  │   = num_workers)                        │
│  ├── Event loop (libuv / io_uring)         │
│  ├── Очереди work-stealing (только Full)   │
│  ├── Аллокатор памяти (jemalloc/mimalloc) │
│  └── Метаданные рефлексии (по требованию,  │
│      не постоянно в памяти)                 │
├────────────────────────────────────────────┤
│  Отсутствует:                              │
│  ❌ Байткод-интерпретатор                  │
│  ❌ JIT-компилятор                          │
│  ❌ GC                                     │
│  ❌ Виртуальная машина                      │
└────────────────────────────────────────────┘
```

Сравнение:

| | Java | Go | YaoXiang |
|------|------|-----|-----------|
| Результат компиляции | байткод | нативный код | нативный код |
| Способ выполнения | интерпретация/JIT в JVM | прямое выполнение | прямое выполнение |
| Размер runtime | ~200MB (JVM) | ~1-2MB (с GC) | **~200-500KB (без GC)** |
| Управление памятью | GC | GC | **RAII (детерминированное)** |
| Рефлексия | постоянно в памяти | постоянно в памяти | **хранится в exe, загрузка по требованию** |

#### 6.3 Почему производительность планировщика постоянна

**Ключевой инсайт**: основная работа выполняется на этапе компиляции, в runtime происходит только «исполнение».

```
Этап компиляции (однократно, не в runtime):
    ├── Построение глобального DAG: кто от кого зависит
    ├── Топологическая сортировка: определение порядка выполнения
    ├── Идентификация изолированных поддеревьев: что можно параллелизовать
    ├── Escape-анализ: ref → Rc или Arc
    ├── Детекция циклов: автоснижение до Weak или ошибка
    └── Инлайн: мелкие функции напрямую раскрываются

Этап выполнения (при каждом запуске, структуры данных фиксированы):
    ├── Диспетчеризация задач в thread pool согласно DAG, определённому при компиляции
    ├── При I/O → приостановка текущей задачи, event loop берёт управление
    ├── Задача готова → возврат в очередь готовых
    └── Вот и всё.
```

**Планировщик сам по себе — структура данных фиксированного размера**: thread pool, event loop, рабочие очереди. Никакого динамического роста, никакой адаптивной реоптимизации, никакого GC-сканирования. Поведение полностью предсказуемо.

На этапе компиляции уже определено «что планировать», runtime только «исполняет». Это отличается от tokio — tokio динамически строит цепочки Future в runtime. DAG в YaoXiang статический.

#### 6.4 Рефлексия: хранится, не резидентна

Метаданные рефлексии генерируются при компиляции, хранятся в отдельном сегменте (section) exe. При запуске программы не загружаются. При первом запросе рефлексии загружаются в память по требованию через mmap. Примерно так:

```
Структура exe:
  .text     ← ваш код
  .rodata   ← константы
  .reflect  ← метаданные рефлексии (информация о типах, сигнатуры функций и т.д.)
              mmap загружает по требованию, без обращения не занимает память
```

**Компромисс**: увеличение размера exe (содержит данные рефлексии), но нулевые затраты памяти в runtime при отсутствии обращений. При первом доступе есть задержка загрузки (аналогично JIT warm-up), далее нулевые накладные расходы.



```
src/
├── core/                    # Общее для всех runtime
│   ├── value.rs
│   ├── allocator.rs
│   └── ownership.rs
├── frontend/                # Общее для всех бэкендов
│   ├── lexer/
│   ├── parser/
│   ├── typecheck/
│   └── dag/                 # ★ DAG-анализ (этап компиляции)
│       ├── builder.rs       #   Построение графа зависимостей
│       ├── escape.rs        #   Escape-анализ (ref → Rc/Arc)
│       ├── cycle.rs         #   Детекция циклов + автоснижение
│       └── topology.rs      #   Топологическая сортировка
├── codegen/                 # Генерация кода
│   ├── ir.rs                # Определение IR (общее для VM и LLVM)
│   ├── vm_backend/          # VM-бэкенд (разработка и отладка)
│   │   ├── bytecode.rs
│   │   └── compiler.rs
│   └── llvm_backend/        # LLVM-бэкенд (продакшен)
│       └── compiler.rs
├── embedded/                # 🟢 Embedded Runtime
│   └── executor.rs
├── runtime/                 # 🔵 Статическая библиотека runtime (линкуется в exe)
│   ├── thread_pool.rs       #   Thread pool фиксированного размера
│   ├── event_loop.rs        #   Event loop I/O (libuv/io_uring)
│   ├── dag_executor.rs      #   Исполнение по DAG этапа компиляции
│   └── scheduler/
│       ├── single_thread.rs
│       └── multi_thread.rs
├── full/                    # 🟣 Full Runtime (опциональная линковка)
│   └── work_stealer.rs      #   Work-stealing
├── reflect/                 # Метаданные рефлексии
│   ├── metadata.rs          #   Генерация метаданных (этап компиляции)
│   └── loader.rs            #   Загрузка по требованию (runtime)
└── vm/                      # VM-интерпретатор (только для разработки)
    └── executor.rs
```

## Компромиссы

### Преимущества

- **Чёткая стратификация**: три уровня — Embedded / Standard / Full
- **Переиспользование компиляции**: фронтенд-код полностью общий
- **Развязка через generics**: полиморфизм на этапе компиляции, нулевые накладные расходы
- **Согласованность**: синхронность — лишь частный случай планирования
- **Дружелюбность к встраиваемым системам**: высокая производительность + низкое потребление памяти + быстрый запуск

### Недостатки

- **Начальная сложность**: необходимо определить интерфейс планировщика и несколько вариантов runtime
- **Компиляционная привязка**: тип планировщика определяется на этапе компиляции
