# RFC-008: Архитектура модели параллелизма Runtime и разделение планировщика

> **⚠️ Замечание о соответствии**: Данный документ приведён в соответствие с [RFC-024 — новая модель параллелизма](/reference/language-spec/concurrency.md). Ставшийся полнопрограммный DAG-анализ, аннотации `@block`/`@eager`, модель уровней L1/L2/L3 заменены примитивом параллелизма `spawn {}`. DAG-анализ теперь выполняется только внутри блока `spawn {}`.

> **Справочные материалы**:
> - [RFC-011: Проектирование системы обобщений](./011-generic-type-system.md)
> - [Спецификация модели параллелизма (RFC-024)](/reference/language-spec/concurrency.md)

## Аннотация

Данный документ определяет ключевой дизайн архитектуры Runtime:

1. **Трёхуровневая архитектура Runtime**: Embedded (синхронное выполнение) → Standard (spawn + DAG-планирование) → Full (work-stealing)
2. **Разделение компиляции и выполнения**: Этапы компиляции полностью идентичны, различие только в способе выполнения во время исполнения
3. **Двойная модель бэкендов**: VM (разработка и отладка) и LLVM AOT (production-выпуск), поведение полностью идентично
4. **Планировщик = статическая библиотека**: При AOT-компиляции планировщик линкуется в exe, ~200-500 КБ, без GC
5. **Синхронность — частный случай планирования**: num_workers=1 означает синхронный режим

### Ключевое уточнение: это не Java

```
Java:   .java → .class → JVM (интерпретация/JIT + GC)        ← всегда требуется VM
YaoXiang разработка: .yx → IR → выполнение в VM (быстрая итерация, пошаговая отладка)
YaoXiang production: .yx → IR → LLVM → нативный exe (планировщик линкуется внутрь)

VM — инструмент разработки, не суть runtime. Как Go: go run vs go build.
Итоговый exe = ваш нативный код + статическая библиотека планировщика + метаданные рефлексии.
Никакого интерпретатора, никакого JIT, никакого GC.
```

## Мотивация

### Ключевое противоречие

| Противоречие | Описание |
|------|------|
| Прозрачность vs управляемость | Блоки spawn обеспечивают явный контроль параллелизма, обычный код выполняется последовательно |
| Ядро vs опциональность | spawn — базовый примитив параллелизма, WorkStealing — продвинутая возможность при num_workers>1 |
| Однопоточность vs параллелизм | В однопоточном режиме параллелизм проявляется как асинхронность, синхронность — частный случай планирования |

## Предложение

### 1. Трёхуровневая архитектура Runtime

```
┌──────────────────────────────────────────────────────────────────┐
│                    Этап компиляции (одинаков для всех режимов)   │
│                                                                  │
│  Исходный код → Lexer → Parser → TypeCheck → Codegen → IR       │
│                                                                  │
│  ⚠️ Единая система парсинга, проверки типов, генерации кода, IR  │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ Inline executor  │ │ spawn + DAG   │ │ Full scheduler    │
│ Синхронное       │ │ Параллелизм   │ │ Параллельная      │
│ выполнение       │ │ внутри блоков │ │ оптимизация       │
│ Без поддержки    │ │ spawn         │ │ Work-stealing     │
│ spawn            │ │               │ │                  │
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| Уровень | Embedded | Standard | Full |
|------|----------|----------|------|
| Компиляция | идентичная | идентичная | идентичная |
| Режим выполнения | синхронный | параллелизм внутри блоков spawn | параллельный |
| Потребление памяти | низкое | среднее | высокое |
| Возможности параллелизма | нет | внутри блоков spawn | внутри блоков spawn + параллелизм |
| Поддержка spawn | ❌ | ✅ | ✅ |
| DAG-анализ | нет | внутри блоков spawn | внутри блоков spawn |
| WorkStealer | нет | нет | ✅ |

**Embedded Runtime**: для WASM/игровых скриптов/движков правил. Inline executor, без поддержки spawn, высокопроизводительный, низкое потребление.

**Standard Runtime**: для веб-сервисов/конвейеров данных. Поддержка блоков `spawn {}`, DAG-анализ и автопараллелизм внутри блоков spawn. num_workers=1 — однопоточный асинхронный режим.

**Full Runtime**: для научных вычислений/масштабного параллелизма. Standard + WorkStealer для балансировки нагрузки.

### 2. Разделение планировщика: обобщения + инъекция

Основной принцип: VM напрямую не зависит от конкретного планировщика, вызывает его через обобщённый параметр `[S]`.

```yaoxiang
# Определение интерфейса планировщика
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# Однопоточный планировщик
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# Многопоточный планировщик
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VM использует планировщик через обобщения
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**Ключевые моменты**:
- Компиляционный полиморфизм, нулевые накладные расходы во время выполнения
- Никаких объектов trait
- Ограничение обобщённого типа `[S: Scheduler]` уже определено в RFC-011

### 3. Синхронность = частный случай планирования

```
❌ Заблуждение: отключить планировщик
✅ Правильно: использовать планировщик с одним worker

num_workers = 1 → однопоточный асинхронный планировщик
num_workers > 1 → многопоточный параллельный планировщик

Тот же интерфейс планировщика, только конфигурация разная. Никаких особых случаев.
```

### 4. Статус DAG

> **Важное изменение**: DAG-анализ более не применяется ко всей программе, а только внутри блоков `spawn {}`. Обычный код (вне блоков spawn) выполняется последовательно и не требует DAG-анализа.

| Уровень | Поддержка spawn | Область DAG-анализа | Описание |
|------|-----------|-------------|------|
| Core Runtime | ✅ | внутри блоков spawn | Ядро параллелизма |
| Standard Runtime | ✅ | внутри блоков spawn | spawn + DAG-планирование |
| Embedded Runtime | ❌ | нет | Inline execution, без параллелизма |

### 5. Модель выполнения снизу вверх (внутри блоков spawn)

> **Важное изменение**: DAG-анализ снизу вверх выполняется только внутри блоков `spawn {}`, больше не применяется ко всей программе.

```
Пользовательский код (параллелизм внутри блока spawn):
    (a, b) = spawn {
        fetch(url0),
        fetch(url1)
    }
    print(a)

Компиляционный анализ (снизу вверх внутри блока spawn):
    fetch(url0) и fetch(url1) не имеют взаимозависимости → могут выполняться параллельно
    print(a) вне блока spawn → последовательное выполнение, ожидание завершения spawn

Планирование во время выполнения (от листьев внутри блока spawn):
    fetch(url0) ┐
                ├→ параллельное выполнение
    fetch(url1) ┘
    print(a)                       ← вне блока spawn, последовательное выполнение
```

**Ключевые моменты**:
- Анализ зависимостей снизу вверх ограничен блоками `spawn {}`
- Независимые задачи внутри блока spawn выполняются параллельно
- Код вне блока spawn выполняется последовательно, ожидая завершения блока spawn

### 6. Модель компиляции: двойной бэкенд + статически связанный runtime

#### 6.1 Два бэкенда, одно поведение

```
                      ┌─────────────────────┐
                      │   Компиляционный    │
                      │   фронтенд (единый)  │
                      │   Lexer → Parser    │
                      │   → TypeCheck       │
                      │   → DAG-анализ      │
                      │     внутри блоков   │
                      │     spawn           │
                      │   → Анализ побега   │
                      │   → Обнаружение     │
                      │     циклов          │
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │  VM-бэкенд (dev)   │     │ LLVM-бэкенд (prod) │
        │                   │     │                   │
        │  Генерация IR/     │     │  Генерация        │
        │  байткода          │     │  нативного кода   │
        │  VM-интерпретация  │     │  Линковка         │
        │  Поддержка         │     │  статической      │
        │  пошаговой         │     │  библиотеки       │
        │  отладки           │     │  runtime          │
        │  Быстрая           │     │  Вывод .exe       │
        │  итерация          │     │  Нулевые накладые │
        │                    │     │  расходы           │
        │                    │     │  интерпретации     │
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           Поведение полностью一致          Поведение полностью一致
```

**VM-бэкенд**: используется при разработке. Изменил код → мгновенный запуск → пошаговая отладка → быстрая итерация. Поведение полностью идентично финальному exe.

**LLVM-бэкенд**: используется при выпуске. AOT-компиляция в нативный код, планировщик как статическая библиотека. Никакого интерпретатора, никакого JIT.

#### 6.2 Планировщик = статическая библиотека, не виртуальная машина

```
Внутренняя структура итогового exe:

┌────────────────────────────────────────────┐
│  Ваш код (нативный машинный код)            │
│  ├── DAG-план выполнения блоков spawn,      │
│  │   определённый на этапе компиляции        │
│  ├── Инлайненные операции Move/ref/clone    │
│  └── Код освобождения RAII                  │
├────────────────────────────────────────────┤
│  Статическая библиотека runtime (~200-500 КБ)│
│  ├── Thread pool (фиксированный размер       │
│  │   = num_workers)                         │
│  ├── Event loop (libuv / io_uring)          │
│  ├── Очереди work-stealing                  │
│  │   (только Full Runtime)                  │
│  ├── Memory allocator                       │
│  │   (jemalloc / mimalloc)                  │
│  └── Метаданные рефлексии                   │
│      (загружаются по запросу,                │
│       не резидентны в памяти)                │
├────────────────────────────────────────────┤
│  Отсутствует:                                │
│  ❌ Байткод-интерпретатор                    │
│  ❌ JIT-компилятор                          │
│  ❌ GC                                      │
│  ❌ Виртуальная машина                       │
└────────────────────────────────────────────┘
```

Сравнение:

| Язык | Java | Go | YaoXiang |
|------|------|-----|-----------|
| Результат компиляции | Байткод | Нативный код | Нативный код |
| Способ выполнения | JVM интерпретация/JIT | Прямое выполнение | Прямое выполнение |
| Размер runtime | ~200 МБ (JVM) | ~1-2 МБ (с GC) | **~200-500 КБ (без GC)** |
| Управление памятью | GC | GC | **RAII (детерминированное)** |
| Рефлексия | В памяти постоянно | В памяти постоянно | **Хранится в exe, загружается по запросу** |

#### 6.3 Почему производительность планировщика постоянна

**Ключевой инсайт**: Основная работа выполняется на этапе компиляции, runtime только «исполняет».

```
Время компиляции (разово, не в runtime):
    ├── Анализ DAG блоков spawn: кто от кого зависит
    ├── Топологическая сортировка: порядок выполнения
    │   внутри блоков spawn
    ├── Идентификация параллелизуемых задач:
    │   независимые поддеревья внутри блоков spawn
    ├── Анализ побега: ref → Rc или Arc
    ├── Обнаружение циклов: автоснижение до Weak или ошибка
    └── Инлайнинг: мелкие функции напрямую разворачиваются

Время выполнения (каждый запуск, структуры данных фиксированы):
    ├── Диспетчеризация задач в thread pool
    │   в порядке, определённом на этапе компиляции
    │   для блоков spawn
    ├── Встретился I/O → текущая задача приостановлена,
    │   event loop берёт управление
    ├── Задача готова → возврат в очередь готовых
    └── Вот и всё.
```

**Планировщик сам по себе — структура данных фиксированного размера**: thread pool, event loop, рабочие очереди. Никакого динамического роста, никакой адаптивной реоптимизации, никакого GC-сканирования. Поведение полностью предсказуемо.

На этапе компиляции уже вычислено, «что планировать» внутри блоков spawn, runtime только «исполняет». Это отличается от tokio — tokio динамически строит цепочки Future во время выполнения. DAG YaoXiang статический и ограничен блоками spawn.

#### 6.4 Рефлексия: хранится, не резидентна

Метаданные рефлексии генерируются на этапе компиляции, хранятся в отдельном сегменте (section) exe. При запуске программы не загружаются. При первом запросе рефлексии — mmap по требованию. Подобно:

```
Структура exe:
  .text     ← ваш код
  .rodata   ← константы
  .reflect  ← метаданные рефлексии
              (информация о типах, сигнатуры функций и т.д.)
              mmap загрузка по требованию,
              без доступа — без потребления памяти
```

**Компромисс**: увеличение размера exe (含反射数据), но нулевые затраты памяти во время выполнения, если не обращаться. Первое обращение имеет задержку загрузки (как JIT warmup), далее — нулевые накладные расходы.

## Компромиссы

### Преимущества

- **Чёткая стратификация**: три уровня — Embedded / Standard / Full
- **Переиспользование компиляции**: код фронтенда полностью общий
- **Обобщённое разделение**: компиляционный полиморфизм, нулевые накладные расходы
- **Согласованность**: синхронность — частный случай планирования
- **Дружелюбие к встраиваемым системам**: высокая производительность + низкое потребление памяти + быстрый запуск

### Недостатки

- **Начальная сложность**: требуется определение интерфейса планировщика и нескольких вариантов runtime
- **Компиляционная привязка**: тип планировщика определяется на этапе компиляции
