# RFC-008：Архитектура Runtime с разделением модели конкурентности и планировщика

> **Статус**: Принято
> **Автор**: Чэнь Сюй
> **Дата создания**: 2025-01-05
> **Последнее обновление**: 2026-06-05（приведение в соответствие с RFC-024: DAG-анализ ограничен внутри spawn блока, удалены старые ссылки L1/L2/L3）

> **⚠️ Пояснение о приведении в соответствие**: Данный документ приведён в соответствие с [RFC-024](/reference/language-spec/concurrency.md). Старый анализ DAG для всей программы, аннотации `@block`/`@eager`, модель уровней L1/L2/L3 заменены примитивом параллелизма `spawn {}` блока. DAG-анализ теперь применяется только внутри `spawn {}` блока.

> **Справочные материалы**:
> - [RFC-011: Проектирование системы обобщённых типов](./011-generic-type-system.md)
> - [Спецификация модели конкурентности（RFC-024）](/reference/language-spec/concurrency.md)

## Аннотация

Данный документ определяет ключевой дизайн архитектуры Runtime:

1. **Трёхуровневая архитектура runtime**: Embedded（немедленное выполнение） → Standard（spawn + DAG-планирование） → Full（work-stealing）
2. **Разделение компиляции и выполнения**: Этап компиляции полностью идентичен, разница только в способе выполнения во время 运行期
3. **Двойная модель backend**: VM（разработка и отладка） и LLVM AOT（продакшен），поведение полностью идентично
4. **Планировщик = статическая библиотека**: При AOT-компиляции планировщик линкуется в exe, ~200-500KB, без GC
5. **Синхронность — частный случай планирования**: num_workers=1 — синхронный режим

### Ключевое уточнение: это не Java

```
Java:   .java → .class → JVM（интерпретация/JIT + GC）        ← всегда требуется виртуальная машина
YaoXiang разработка: .yx → IR → VM выполнение（быстрая итерация, пошаговая отладка）
YaoXiang продакшен: .yx → IR → LLVM → нативный exe（планировщик линкуется внутрь）

VM — инструмент разработки, а не сущность runtime. Как go run vs go build в Go.
Итоговый exe = ваш нативный код + статическая библиотека планировщика + метаданные рефлексии. Никакого интерпретатора, никакого JIT, никакого GC.
```

## Мотивация

### Основное противоречие

| Противоречие | Описание |
|------|------|
| Прозрачность vs управляемость | spawn блок обеспечивает явный контроль параллелизма, обычный код выполняется последовательно |
| Ядро vs опциональность | spawn — базовый примитив параллелизма, WorkStealing — продвинутая возможность при num_workers>1 |
| Однопоточность vs конкурентность | В однопоточном режиме конкурентность проявляется как асинхронность, синхронность — частный случай планирования |

## Предложение

### 1. Трёхуровневая архитектура Runtime

```
┌──────────────────────────────────────────────────────────────────┐
│                    Этап компиляции（все режимы идентичны）        │
│                                                                  │
│  Source Code → Lexer → Parser → TypeCheck → Codegen → IR        │
│                                                                  │
│  ⚠️ Единая система разбора, проверки типов, кодогенерации, IR    │
└──────────────────────────────────────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
┌──────────────────┐ ┌───────────────┐ ┌──────────────────┐
│ 🟢 Embedded      │ │ 🔵 Standard   │ │ 🟣 Full          │
│ 即时执行器         │ │ spawn + DAG   │ │ Full 调度器       │
│ Синхронное        │ │ 并发 внутри    │ │ Параллельная      │
│ выполнение        │ │ spawn 块      │ │ оптимизация       │
│ Без spawn 支持    │ │ 自动并发       │ │ Work stealing    │
└──────────────────┘ └───────────────┘ └──────────────────┘
```

| Уровень | Embedded | Standard | Full |
|------|----------|----------|------|
| Компиляция | идентична | идентична | идентична |
| Режим выполнения | синхронный | 并发 внутри spawn 块 | параллельный |
| Потребление памяти | низкое | среднее | высокое |
| Возможности 并发 | нет | внутри spawn 块 | внутри spawn 块 + 并行 |
| spawn 支持 | ❌ | ✅ | ✅ |
| DAG 分析 | нет | внутри spawn 块 | внутри spawn 块 |
| WorkStealer | нет | нет | ✅ |

**Embedded Runtime**：Для WASM/игровых скриптов/движков правил. 即时执行器, без поддержки spawn, высокую производительность при низком потреблении.

**Standard Runtime**：Для веб-сервисов/конвейеров данных. Поддержка `spawn {}` блоков, DAG-анализ и автоматическая 并发 внутри spawn блоков. num_workers=1 — однопоточный асинхронный режим.

**Full Runtime**：Для научных вычислений/масштабного параллелизма. Standard + WorkStealer для балансировки нагрузки.

### 2. Разделение планировщика: Generics + Внедрение зависимостей

Основной принцип: VM не зависит напрямую от конкретного планировщика, использует его через обобщённый параметр `[S]`.

```yaoxiang
# Определение интерфейса планировщика
Scheduler: Type = {
    spawn: (Task) -> TaskId,
    await: (TaskId) -> Result,
    spawn_with_deps: (Task, List(TaskId)) -> TaskId,
    await_all: (List(TaskId)) -> List(Result),
    stats: () -> SchedulerStats,
}

# Однопоточный планировщик
SingleThreadScheduler: Scheduler = {
    spawn: (task) => { task_queue.push(task); generate_task_id() },
    await: (task_id) => { ... },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { queue_size: task_queue.len() },
}

# Многопоточный планировщик
MultiThreadScheduler: Scheduler = {
    spawn: (task) => { work_queue.push(task); generate_task_id() },
    await: (task_id) => { wait_for_completion(task_id) },
    spawn_with_deps: (task, deps) => { ... },
    await_all: (task_ids) => { ... },
    stats: () => { workers: get_worker_stats() },
}

# VM использует планировщик через обобщения
create_vm: [S: Scheduler](scheduler: S) -> VM = (scheduler) => {
    VM(scheduler: scheduler, memory: create_memory(), dag: create_dag())
}
```

**Ключевые моменты**:
- Компиляционная полиморфия, нулевые накладные расходы 运行期
- Не требуются Trait объекты
- Обобщённое ограничение типа `[S: Scheduler]` уже определено в RFC-011

### 3. Синхронность = частный случай планирования

```
❌ Заблуждение: отключить планировщик
✅ Правильно: использовать планировщик с одним worker

num_workers = 1 → однопоточный асинхронный планировщик
num_workers > 1 → многопоточный параллельный планировщик

Тот же интерфейс планировщика, только разная конфигурация. Никаких исключений.
```

### 4. Позиция DAG

> **Важное изменение**: DAG-анализ больше не применяется ко всей программе, а только внутри `spawn {}` блоков. Обычный код (вне spawn блоков) выполняется последовательно и не требует DAG-анализа.

| Уровень | spawn 支持 | Область DAG 分析 | 说明 |
|------|-----------|-------------|------|
| Core Runtime | ✅ | внутри spawn 块 | 并发 ядро |
| Standard Runtime | ✅ | внутри spawn 块 | spawn + DAG-планирование |
| Embedded Runtime | ❌ | нет | 即时执行, без 并发 |

### 5. Модель выполнения снизу вверх（внутри spawn блока）

> **Важное изменение**: DAG-анализ снизу вверх выполняется только внутри `spawn {}` блоков, а не для всей программы.

```
Пользовательский код（并发 внутри spawn 块）：
    (a, b) = spawn {
        fetch(url0),
        fetch(url1)
    }
    print(a)

Компиляционный анализ（снизу вверх внутри spawn 块）：
    fetch(url0) и fetch(url1) не зависят друг от друга → 可并行执行
    print(a) вне spawn 块 → последовательное выполнение, ожидание spawn 完成

运行期 планирование（от листьев внутри spawn 块）：
    fetch(url0) ┐
                ├→ 并行 выполнение
    fetch(url1) ┘
    print(a)                       ← вне spawn 块, последовательное выполнение
```

**Ключевые моменты**:
- Анализ зависимостей снизу вверх ограничен только внутри `spawn {}` блоков
- Задачи без зависимостей внутри spawn 块 выполняются 并行
- Код вне spawn 块 выполняется последовательно, ожидая 完成 spawn 块

### 6. Компиляционная модель: двойной backend + статически линкуемый runtime

#### 6.1 Два backend, одно поведение

```
                      ┌─────────────────────┐
                      │   Компиляционный frontend（единый）│
                      │   Lexer → Parser     │
                      │   → TypeCheck        │
                      │   → DAG分析 внутри spawn 块│
                      │   → 逃逸分析          │
                      │   → 环检测            │
                      └──────────┬──────────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
        ┌───────────────────┐     ┌───────────────────┐
        │   VM backend（开发） │     │  LLVM backend（生产）│
        │                   │     │                   │
        │  生成 IR/字节码      │     │  生成原生代码       │
        │  VM解释执行          │     │  链接运行时静态库    │
        │  支持步进调试         │     │  输出 .exe         │
        │  快速迭代           │     │  零解释开销         │
        └───────────────────┘     └───────────────────┘
                 │                         │
                 ▼                         ▼
           行为完全一致                 行为完全一致
```

**VM backend**：Используется при разработке. Изменение кода → немедленное выполнение → пошаговая отладка → быстрая итерация. Поведение полностью идентично финальному exe.

**LLVM backend**：Используется при 发布. AOT-компиляция в нативный код, планировщик как статическая библиотека линкуется внутрь. Никакого интерпретатора, никакого JIT.

#### 6.2 Планировщик = статическая библиотека, не виртуальная машина

```
Внутренняя структура итогового exe：

┌────────────────────────────────────────────┐
│  Ваш код（нативный машинный код）            │
│  ├── DAG план выполнения, определённый      │
│  │   при компиляции внутри spawn 块          │
│  ├── 内联的 Move/ref/clone операции         │
│  └── RAII код освобождения                  │
├────────────────────────────────────────────┤
│  运行时静态库（~200-500KB）                   │
│  ├── 线程池（фиксированный размер = num_workers）│
│  ├── 事件循环（libuv / io_uring）            │
│  ├── Work-stealing очереди（только Full）    │
│  ├── 内存分配器（jemalloc / mimalloc）       │
│  └── 反射元数据（по требованию, не 常驻）     │
├────────────────────────────────────────────┤
│  Нет：                                      │
│  ❌ 字节码解释器                             │
│  ❌ JIT компилятор                           │
│  ❌ GC                                      │
│  ❌ Виртуальная машина                       │
└────────────────────────────────────────────┘
```

Сравнение：

| Язык | Java | Go | YaoXiang |
|------|------|-----|-----------|
| Результат компиляции | 字节码 | 原生代码 | 原生代码 |
| Способ выполнения | JVM интерпретация/JIT | Прямое выполнение | Прямое выполнение |
| Размер 运行期 | ~200MB（JVM） | ~1-2MB（с GC） | **~200-500KB（без GC）** |
| Управление памятью | GC | GC | **RAII（детерминирован）** |
| Рефлексия | 常驻内存 | 常驻内存 | **储存在 exe, 按需加载** |

#### 6.3 Почему производительность планировщика постоянна

**Ключевой инсайт**: Основная работа выполняется при компиляции, 运行期 только "исполняет".

```
Компиляция（один раз, не в 运行期）：
    ├── Анализ DAG внутри spawn 块: кто от кого зависит
    ├── 拓扑排序: 确定 порядок выполнения внутри spawn 块
    ├── Определение 可并行 задач: поддеревья без зависимостей внутри spawn 块
    ├── 逃逸分析: ref → Rc или Arc
    ├── 环检测: авто降级 Weak или ошибка
    └── 内联: малые функции 直接展开

运行期（каждый запуск, структуры данных фиксированы）：
    ├── 分发 задач в 线程池按 компиляционному DAG порядку внутри spawn 块
    ├── При I/O → 挂起 текущую задачу, событийный цикл接管
    ├── Задача готова → 放回 в готовую очередь
    └── Вот и всё.
```

**Сам планировщик — фиксированная структура данных**: thread pool, event loop, work queue. Никакого динамического роста, никакой адаптивной реоптимизации, никакого GC-сканирования. Поведение полностью предсказуемо.

Компиляция уже вычислила, "что планировать" внутри spawn 块, 运行期 только "исполняет". Это отличается от tokio — tokio динамически строит цепочки Future 运行期. DAG в YaoXiang статический и ограничен внутри spawn 块.

#### 6.4 Рефлексия: хранить, не 常驻

Метаданные рефлексии генерируются при компиляции, хранятся в отдельном сегменте (section) exe. При запуске программы не загружаются. При первом запросе рефлексии загружаются по требованию через mmap. Аналогично:

```
exe layout:
  .text     ← ваш код
  .rodata   ← константы
  .reflect  ← метаданные рефлексии（информация о типах, сигнатуры функций и т.д.）
              mmap по требованию, не используется — не занимает память
```

**Компромисс**: увеличение размера exe (с данными рефлексии), но нулевые затраты памяти 运行期 если не обращаться. Первое обращение имеет задержку загрузки (аналогично JIT warm-up), далее нулевые накладные расходы.

## Компромиссы

### Преимущества

- **Чёткое разделение уровней**: Embedded / Standard / Full три уровня
- **Переиспользование компиляции**: Фронтенд код полностью общий
- **Generics разделение**: Компиляционный полиморфизм, нулевые накладные расходы
- **Согласованность**: Синхронность — частный случай планирования
- **Удобство для встраиваемых систем**: Высокая производительность + низкое потребление памяти + быстрый запуск

### Недостатки

- **Начальная сложность**: Требуется определить интерфейс планировщика и несколько вариантов 运行期
- **Компиляционная привязка**: Тип планировщика определяется при компиляции

| 反射元数据 | Компилируются в отдельный сегмент exe, mmap по требованию | 2026-05-11 |
| Производительность планировщика | DAG анализ при компиляции, 运行期 только исполняет | 2026-05-11 |
| Область DAG | DAG анализ ограничен внутри spawn 块, приведён в соответствие с RFC-024 | 2026-06-05 |
| Обновление трёхуровневой архитектуры | Embedded без spawn, Standard с поддержкой spawn | 2026-06-05 |
