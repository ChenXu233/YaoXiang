# RFC-004: Дизайн многопозиционного объединённого связывания для каррированных методов

## Аннотация

Данный RFC предлагает совершенно новый синтаксис **многопозиционного объединённого связывания**, позволяющий точно привязывать функцию к произвольным позициям параметров типа, поддерживающий как одно-позиционное, так и многопозиционное объединённое связывание, что фундаментально решает проблему «кто является вызывающим» при каррированном связывании, без необходимости введения ключевого слова `self`.

## Мотивация

### Зачем нужна эта возможность?

В текущем дизайне языка привязка свободной функции в качестве метода типа сталкивается со следующими проблемами:

1. **Негибкость позиции вызывающего**: традиционное связывание может зафиксировать только `obj` в `obj.method(args)` как первый параметр
2. **Сложность привязки нескольких параметров**: когда метод должен принимать несколько параметров одного типа, невозможно элегантно это выразить
3. **Семантическая неоднозначность каррирования**: при частичном применении трудно различить, к какой позиции происходит привязка

### Цель дизайна: объединение двух парадигм программирования

Данный дизайн направлен на **объединение функциональной и ООП парадигм**:

```yaoxiang
# Функциональный взгляд: явная передача всех аргументов
distance(p1, p2)

# ООП-взгляд: неявное this
p1.distance(p2)

# Синтаксический сахар [positions] делает обе записи эквивалентными, по сути это вызовы функций
Point.distance = distance[0]   # this привязывается к позиции 0
```

**Ключевые достоинства**:
- На нижнем уровне — функция, на верхнем — синтаксис метода
- Не вводится ключевое слово `self`, сохраняется лаконичность языка
- Полная функциональность: вызов метода по сути является передачей аргументов
- `[0]`, `[1]`, `[-1]` — гибкое управление позицией привязки this
- **Единый синтаксис**: определение функции использует формат `name: (params) -> Return = body`

### Текущие проблемы

```yaoxiang
# Проблемы существующего дизайна:
Point: Type = { x: Float, y: Float }
Vector: Type = { x: Float, y: Float, z: Float }

distance: (a: Point, b: Point) -> Float = { ... }
transform: (p: Point, v: Vector) -> Point = { ... }

# Можно привязать только к первому параметру
Point.distance = distance  # эквивалентно distance[0]
# p1.distance(p2) → distance(p1, p2) ✓

# Но если сигнатура transform имеет вид transform(Vector, Point)?
# Невозможно выразить семантику p1.transform(v1) → transform(v1, p1)
```

## Предложение

### Основной дизайн: явное указание позиции

**Ключевое правило: без `[n]` = без привязки.** `Point.name = func` — это лишь псевдоним пространства имён, не вызывающий никакой неявной привязки. Чтобы синтаксис вызова через `.` работал как `p.name(args)`, необходимо явное указание: `Point.name = func[n]`.

#### Однопозиционное связывание

```yaoxiang
# Явная привязка к позиции первого параметра Point (индексация с 0)
Point.distance = distance[0]
p1.distance(p2)                     # → distance(p1, p2)

# Привязка к позиции второго параметра Point
Point.compare = distance[1]         # привязка ко второму параметру Point
p1.compare(p2)                      # → distance(p2, p1)
```

**Без `[n]` = без привязки**:

```yaoxiang
# Нет [n] → чистый псевдоним пространства имён, синтаксис . не работает
Point.distance = distance            # доступно только Point.distance(p1, p2)
# p1.distance(p2)  ❌  привязки нет

# Фабричные функции естественно легальны, без специальной обработки
create_point: () -> Point = { ... }
Point.create = create_point          # Point.create()   ✅
```
- Типобезопасность: привязка происходит только при совпадении типов, что предотвращает ошибки
- Гибкое управление: `[n]` обеспечивает точный контроль позиции привязки

#### Каррированное связывание

Когда число параметров функции больше числа позиций привязки, автоматически генерируется каррированная функция. **Связывание всегда является явной операцией.**

```yaoxiang
Point: Type = { x: Float, y: Float }

# Базовая функция
scale: (p: Point, factor: Float) -> Point = {
    return Point(p.x * factor, p.y * factor)
}

# Явная привязка к позиции 0 → каррирование: оставшийся параметр factor предоставляется вызывающим
Point.scale = scale[0]

# Вызов
p1 = Point(2.0, 3.0)
scaled = p1.scale(2.0)       # → scale(p1, 2.0)

# Цепочка вызовов выглядит элегантнее
result = Point(2.0, 3.0).scale(2.0)  # → Point(4.0, 6.0)
```

### Синтаксис позиционно-индексной привязки

Вводится синтаксис `[position]` для точного управления связью между параметрами функции и типом:

```yaoxiang
# Формат синтаксиса: Type.method = function[positions]

# === Базовая привязка ===

# Однопозиционная привязка
Point.distance = distance[1]           # привязка к 1-му параметру (индексация с 0)
# Использование: p1.distance(p2) → distance(p2, p1)

# Многопозиционная объединённая привязка (деструктуризация кортежа)
Point.transform = transform[1, 2]      # привязка к 1-му и 2-му параметрам
# Использование: p1.transform(v1) → transform(v1, p1)
# Исходная сигнатура: transform(Point, Vector) → Point
# После привязки: Point.transform(Vector) → Point
```

### Подробное определение синтаксиса

```
Объявление_привязки ::= Тип '.' Идентификатор '=' Имя_функции '[' Список_позиций ']'

Список_позиций ::= Позиция (',' Позиция)*
Позиция        ::= Целое_число              # позиция
                | '_'                       # пропуск позиции (placeholder)
                | Целое_число '..' Целое_число  # диапазон позиций (будущее расширение)

Имя_функции    ::= Идентификатор
Тип            ::= Идентификатор (Дженерик_параметры)?
```

### Встроенные привязки

Привязка может быть записана непосредственно в теле определения типа, без отдельного оператора привязки:

```yaoxiang
# Способ 1: прямая привязка в теле определения типа
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]           # привязка к позиции 0
}

# Способ 2: анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
# Синтаксис: ((params) => body)[position]
```

**Семантика каррирования**:
- При привязке `distance = distance[0]` исходная сигнатура `(a: Point, b: Point) -> Float`
- Сгенерированная сигнатура метода: `b: Point -> Float` (позиция 0 заполняется вызывающим)

### Примеры использования

```yaoxiang
# === Полный пример ===

Point: Type = { x: Float, y: Float }
Vector: Type = { x: Float, y: Float, z: Float }

# 1. Базовая функция вычисления расстояния
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    return (dx * dx + dy * dy).sqrt()
}

# Привязка: Point.distance = distance[1]
# Вызов: p1.distance(p2) → distance(p2, p1)
# Но мы хотим p1.distance(p2) → distance(p1, p2), поэтому:
Point.distance = distance[0]

# 2. Операция трансформации (многопозиционная привязка)
transform: (p: Point, v: Vector) -> Point = {
    return Point(p.x + v.x, p.y + v.y)
}

# Привязка Point.transform = transform[1]
# Вызов: p.transform(v) → transform(v, p) ❌
# Привязка Point.transform = transform[0]
# Вызов: p.transform(v) → transform(p, v) ✓

# 3. Сложная функция с множеством параметров
multiply: (a: Point, s: Float) -> Point = {
    return Point(a.x * s, a.y * s)
}

# Привязка только к 1-му параметру (тип Point), 3-й параметр сохраняется
Point.scale = multiply[0, _]
# Вызов: p.scale(2.0) → multiply(p, 2.0)

# 4. Межтиповая привязка
Circle: Type = { center: Point, radius: Float }

distance: (a: Circle, b: Circle) -> Float = {
    return a.center.distance(b.center) - a.radius - b.radius
}

# Привязка метода distance к типу Circle
Circle.distance = distance[0, 1]
# Вызов: c1.distance(c2) → distance(c1, c2)
```

### Поддержка деструктуризации кортежей

```yaoxiang
# === Привязка с деструктуризацией кортежа ===

# Функция принимает параметр-кортеж
process_coordinates: (coord: (Float, Float)) -> String = {
    return match coord {
        (0.0, 0.0) -> "origin"
        (x, 0.0) -> "on x-axis at ${x}"
        (0.0, y) -> "on y-axis at ${y}"
        (x, y) -> "point at (${x}, ${y})"
    }
}

Coord: Type = { x: Float, y: Float }

# Автоматическая деструктуризация при привязке: Coord -> (Float, Float)
Coord.describe = process_coordinates[1]
# Использование: coord.describe() → process_coordinates((coord.x, coord.y))
```

### Привязка с несколькими возвращаемыми значениями

```yaoxiang
# === Привязка с несколькими возвращаемыми значениями ===

min_max: (list: List(Int)) -> (Int, Int) = {
    min = list.reduce(Int.MAX, (a, b) => if a < b then a else b)
    max = list.reduce(Int.MIN, (a, b) => if a > b then a else b)
    return (min, max)
}

List.range: (T:Type)->((self: List(T)) -> (T, T)) = min_max[1]
# Использование: (min_val, max_val) = list.range()
```

### Правила проверки типов

```rust
fn check_binding_type_compatibility(
    binding: &Binding,
    func: &Function
) -> Result<(), TypeError> {
    // 1. Если позиция ищется автоматически (явно не указана), проверить, найдено ли совпадение
    if binding.positions.is_empty() {
        return Err(TypeError::NoMatchingParameter(
            binding.type_name.clone(),
            func.name.clone()
        ));
    }

    // 2. Проверить валидность всех индексов позиций
    for pos in &binding.positions {
        if *pos >= func.params.len() {
            return Err(TypeError::InvalidBindingPosition(*pos));
        }
    }

    // 3. Проверить совместимость типов в позициях привязки
    for pos in &binding.positions {
        let param_type = &func.params[*pos].type_;
        let binding_type = &binding.type_name;

        if !isAssignable(binding_type, param_type) {
            return Err(TypeError::IncompatibleTypes(
                binding_type, param_type
            ));
        }
    }

    // 4. Проверить соответствие аргументов вызова метода оставшимся параметрам
    Ok(())
}
```

### Поведение во время выполнения

| Сценарий | Синтаксис привязки | Вызов | Преобразуется в |
|----------|---------------------|-------|----------------|
| Без привязки | `Point.distance = distance` | `Point.distance(p1, p2)` | `distance(p1, p2)` |
| Однопозиционная | `Point.distance = distance[0]` | `p1.distance(p2)` | `distance(p1, p2)` |
| Однопозиционная | `Point.distance = distance[1]` | `p1.distance(p2)` | `distance(p2, p1)` |
| Отрицательный индекс | `Point.test = func[-1]` | `p.test(a, b)` | `func(a, b, p)` |
| Многопозиционная (каррирование) | `Point.scale = scale[0]` | `p.scale(2.0)` | `scale(p, 2.0)` |
| Плейсхолдер | `Type.method = func[1]` | `obj.method(arg)` | `func(arg, obj)` |

**Пояснения**:
- **Без привязки**: `Point.name = func` — лишь псевдоним пространства имён, синтаксис вызова `.` не работает
- `[0]`: вызывающий привязывается к позиции 0 (первый параметр)
- `[1]`: вызывающий привязывается к позиции 1 (второй параметр)
- `[-1]`: вызывающий привязывается к последней позиции (отсчёт с конца)

## Компромиссы

### Преимущества

- **Явное связывание**: `[n]` — единственный механизм привязки, без `[n]` привязки нет, никакого неявного поведения
- **Точный контроль**: можно привязывать к произвольной позиции параметра, высокая гибкость
- **Типобезопасность**: полная проверка типов на этапе компиляции, привязка только при совпадении типов
- **Лаконичный синтаксис**: `[position]` интуитивно понятен
- **Отсутствие ключевого слова `self`**: сохраняется лаконичность языка
- **Каррирование-дружественность**: естественная поддержка частичного применения и цепочек вызовов
- **ООП-дружественность**: автоматическое каррирование упрощает миграцию ООП-программистов

### Недостатки

- **Порог входа**: требуется понимание концепции позиционного индекса
- **Сложность компиляции**: разбор привязок и проверка типов увеличивают сложность компилятора
- **Сложность отладки**: сообщения об ошибках должны чётко указывать на проблемы с позицией привязки

## Альтернативы

| Подход | Описание | Почему не выбран |
|--------|----------|------------------|
| Ключевое слово `self` | Введение `self` в стиле Python/Rust | Противоречит философии дизайна YaoXiang об отсутствии неявного `self` |
| Именованная привязка аргументов | Использование именованных аргументов `func(a=obj)` | Требует изменения определения сигнатуры функции, увеличивает сложность |
| Система макросов | Реализация привязки через макросы | Большие накладные расходы во время выполнения, снижение типобезопасности |
| Перегрузка операторов | Ограничение `self` на определённых позициях | Непоследовательный синтаксис, семантическая путаница |

## Стратегия реализации

### Этапы

1. **Фаза 1: Базовое связывание** (v0.3)
   - Реализация синтаксиса однопозиционной привязки `[n]` (n начинается с 0, поддержка отрицательных значений)
   - Базовая проверка типов и генерация кода
   - Покрытие модульными тестами

2. **Фаза 2: Расширенные возможности** (v0.5)
   - Поддержка синтаксиса диапазонов `[n..m]`
   - Оптимизация вычисления позиций во время компиляции

### Зависимости

- Внешних зависимостей нет
- Прямой связи с RFC-001 (обработка ошибок) нет
- Может быть реализован независимо

### Риски

- Обработка совместимости с существующим синтаксисом привязки
- Стратегия оптимизации производительности (раскрытие во время компиляции vs поиск во время выполнения)

## Приложения

### Приложение A: Запись о принятых дизайн-решениях

| Решение | Выбор | Обоснование |
|---------|-------|-------------|
| База индексации | С 0 | Согласуется с индексацией кортежей/списков параметров |
| Отрицательные индексы | Поддерживаются | Гибкость, отсчёт с конца |
| Плейсхолдер | `_` | Лаконичный, общепринятый символ |
| Синтаксис диапазонов | Реализуется | Пакетная привязка, например `[0..2]` |
| Стиль синтаксиса | Инфиксный `Type.method = func[positions]` | Единообразие с RFC-010 |
| **Правило привязки** | **Только явное `[n]` привязывает, без `[n]` — нет привязки** | **Отсутствие неявного поведения, ортогональность определения и привязки** |
| **Пространство имён** | **`Type.name` — лишь принадлежность к пространству имён, не вызывает привязку** | **Разделение определения и привязки** |
| **Синтаксис функций** | **Имена параметров в сигнатуре `name: (params) -> Return`** | **Единообразие с RFC-010** |

### Приложение B: Глоссарий

| Термин | Определение |
|--------|-------------|
| Позиция привязки | Индекс позиции в списке параметров функции |
| Объединённое связывание | Привязка типа к нескольким позициям параметров |
| Частичное применение | Предоставление только части аргументов, возврат функции, ожидающей завершения вызова |
| **Единый синтаксис** | **`name: (params) -> Return = body`, имена параметров объявляются в сигнатуре** |
| **Функция пространства имён** | **Синтаксис `Type.name`, функция принадлежит пространству имён Type, без подразумеваемой привязки** |
| **Явная привязка** | **`Type.name = func[n]`, единственный механизм привязки методов** |
