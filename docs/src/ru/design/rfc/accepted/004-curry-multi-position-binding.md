# RFC-004: Проектирование многопозиционной объединённой привязки для каррированных методов

## Аннотация

В данном RFC предлагается совершенно новый синтаксис **многопозиционной объединённой привязки**, позволяющий точно привязать функцию к произвольной позиции параметра типа, с поддержкой как однопозиционной, так и многопозиционной объединённой привязки. Это фундаментально решает проблему "кто является вызывающей стороной" в каррированных привязках без введения ключевого слова `self`.

## Мотивация

### Почему эта функция необходима?

При текущем дизайне языка связывание автономных функций с типами методов сталкивается со следующими проблемами:

1. **Неподвижная позиция вызывающей стороны**: традиционная привязка может зафиксировать только `obj` в `obj.method(args)` в качестве первого параметра
2. **Сложность привязки нескольких параметров**: когда методу необходимо принимать несколько параметров одного типа, нет элегантного способа выразить это
3. **Неоднозначность каррированной семантики**: при частичном применении сложно различить "к какой позиции привязка"

### Цели проектирования: унификация двух парадигм программирования

Данный дизайн направлен на **унификацию функциональной и ООП парадигм программирования**:

```yaoxiang
# Функциональный взгляд: явная передача всех параметров
distance(p1, p2)

# ООП взгляд: неявный this
p1.distance(p2)

# [positions] как синтаксический сахар делает обе записи эквивалентными,
# по сути это вызовы функций
Point.distance = distance[0]   # this привязан к позиции 0
```

**Ключевая ценность**:
- На нижнем уровне — функции, на верхнем — синтаксис методов
- Без введения ключевого слова `self`, сохранение простоты языка
- Полная функциональность: вызов метода по сути представляет собой передачу параметров
- `[0]`, `[1]`, `[-1]` для гибкого управления позицией привязки this
- **Унифицированный синтаксис**: определение функций использует формат `name: (params) -> Return = body`

### Текущие проблемы

```yaoxiang
# Проблемы существующего дизайна:
Point: Type = { x: Float, y: Float }
Vector: Type = { x: Float, y: Float, z: Float }

distance: (a: Point, b: Point) -> Float = { ... }
transform: (p: Point, v: Vector) -> Point = { ... }

# Можно привязать только к первому параметру
Point.distance = distance  # эквивалент distance[0]
# p1.distance(p2) → distance(p1, p2) ✓

# Но что если сигнатура transform — transform(Vector, Point)?
# Невозможно выразить семантику p1.transform(v1) → transform(v1, p1)
```

## Предложение

### Базовый дизайн: привязка по умолчанию + опциональное указание позиции

#### Привязка по умолчанию к первой позиции соответствующего типа

**Поведение по умолчанию**: `Type.method = function` автоматически находит первую позицию, соответствующую этому типу, и выполняет привязку

```yaoxiang
# Привязка по умолчанию к первой позиции соответствующего типа
Point.distance = distance           # компилятор автоматически ищет первую позицию параметра Point
p1.distance(p2)                     # → distance(p1, p2)

# Если функция имеет два параметра Point, привязка к первой совпадающей позиции
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    return (dx * dx + dy * dy).sqrt()
}
# Привязка: Point.distance = distance
# Вызов: p1.distance(p2) → distance(p1, p2) ✓

# Только когда требуется специальная позиция (не первая совпадающая),
# указываем явно
Point.compare = distance[1]        # привязка ко второму параметру Point
p1.compare(p2)                    # → distance(p2, p1)
```

**Обработка ошибок привязки**:
- **Не найден соответствующий тип**: если среди параметров функции нет этого типа, выдаётся ошибка или предупреждение
- **Режим фабричной функции**: если нет соответствующих параметров, возможно использование в качестве фабричной функции

```yaoxiang
# Ситуация 1: не найден соответствующий тип
create_point: () -> Point = { ... }
Point.create = create_point        # ошибка: нет параметра типа Point

# Ситуация 2: режим фабричной функции (опционально)
Point.create = create_point        # как фабричная функция, вызов: Point.create()
```

**Преимущества**:
- Интеллектуальная привязка: автоматическое сопоставление по типу, соответствует интуиции
- Типобезопасность: привязка выполняется только при совпадении типов, предотвращает ошибки
- Гибкость управления: когда привязка по умолчанию не соответствует ожиданиям,
  можно явно указать позицию

#### Автоматическая каррированная привязка

Когда количество параметров функции превышает количество позиций привязки, автоматически генерируется каррированная функция:

```yaoxiang
Point: Type = { x: Float, y: Float }

# Базовая функция: 3 параметра
scale: (p: Point, factor: Float) -> Point = {
    return Point(p.x * factor, p.y * factor)
}

# При привязке автоматическое каррирование
Point.scale = scale[0, 1]   # Point привязан к позициям 0, 1, позиция 2 сохранена

# При вызове автоматическое частичное применение
p1 = Point(2.0, 3.0)
scaled = p1.scale(2.0)       # → scale(p1, 2.0) прямой вызов
result = scaled              # → Point(4.0, 6.0)

# Более элегантные цепочечные вызовы
result = Point(2.0, 3.0).scale(2.0)  # → Point(4.0, 6.0)
```

### Синтаксис позиционной индексной привязки

Вводится синтаксис `[position]` для точного управления соотношением между параметрами функции и типами:

```yaoxiang
# Формат синтаксиса: Type.method = function[positions]

# === Базовая привязка ===

# Однопозиционная привязка
Point.distance = distance[1]           # привязка к параметру 1 (индекс начинается с 0)
# Использование: p1.distance(p2) → distance(p2, p1)

# Многопозиционная объединённая привязка (деструктуризация кортежа)
Point.transform = transform[1, 2]      # привязка к параметрам 1, 2
# Использование: p1.transform(v1) → transform(v1, p1)
# Оригинальная сигнатура функции: transform(Point, Vector) → Point
# После привязки: Point.transform(Vector) → Point
```

### Детальное определение синтаксиса

```
Привязка ::= Тип '.' Идентификатор '=' ИмяФункции '[' СписокПозиций ']'

СписокПозиций ::= Позиция (',' Позиция)*
Позиция     ::= ЦелоеЧисло                    # заполнитель
           | '_'                           # пропуск этой позиции (заполнитель)
           | ЦелоеЧисло '..' ЦелоеЧисло    # диапазон позиций (расширение в будущем)

ИмяФункции  ::= Идентификатор
Тип         ::= Идентификатор (ОбобщённыеПараметры)?
```

### Встроенные привязки

Привязки можно записывать непосредственно в теле определения типа без отдельного оператора привязки:

```yaoxiang
# Способ 1: непосредственная привязка в теле определения типа
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]           # привязка к позиции 0
}

# Способ 2: анонимная функция + позиционная привязка
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
# Синтаксис: ((params) => body)[position]
```

**Каррированная семантика**:
- При привязке `distance = distance[0]`, оригинальная сигнатура функции `(a: Point, b: Point) -> Float`
- Генерируемая сигнатура метода: `b: Point -> Float` (позиция 0 заполняется вызывающей стороной)

### Примеры использования

```yaoxiang
# === Полный пример ===

Point: Type = { x: Float, y: Float }
Vector: Type = { x: Float, y: Float, z: Float }

# 1. Базовый расчёт расстояния
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    return (dx * dx + dy * dy).sqrt()
}

# Привязка: Point.distance = distance[1]
# Вызов: p1.distance(p2) → distance(p2, p1)
# Но мы хотим p1.distance(p2) → distance(p1, p2), поэтому:
Point.distance = distance[0]

# 2. Операция трансформации (многопозиционная привязка)
transform: (p: Point, v: Vector) -> Point = {
    return Point(p.x + v.x, p.y + v.y)
}

# Привязка Point.transform = transform[1]
# Вызов: p.transform(v) → transform(v, p) ❌
# Привязка Point.transform = transform[0]
# Вызов: p.transform(v) → transform(p, v) ✓

# 3. Сложная функция с несколькими параметрами
multiply: (a: Point, s: Float) -> Point = {
    return Point(a.x * s, a.y * s)
}

# Привязка только параметра 1 (тип Point), сохранение параметра 3
Point.scale = multiply[0, _]
# Вызов: p.scale(2.0) → multiply(p, 2.0)

# 4. Привязка между типами
Circle: Type = { center: Point, radius: Float }

distance: (a: Circle, b: Circle) -> Float = {
    return a.center.distance(b.center) - a.radius - b.radius
}

# Привязка метода расстояния к типу Circle
Circle.distance = distance[0, 1]
# Вызов: c1.distance(c2) → distance(c1, c2)
```

### Поддержка деструктуризации кортежей

```yaoxiang
# === Привязка с деструктуризацией кортежа ===

# Функция принимает параметр-кортеж
process_coordinates: (coord: (Float, Float)) -> String = {
    return match coord {
        (0.0, 0.0) -> "origin"
        (x, 0.0) -> "on x-axis at ${x}"
        (0.0, y) -> "on y-axis at ${y}"
        (x, y) -> "point at (${x}, ${y})"
    }
}

Coord: Type = { x: Float, y: Float }

# Автоматическая деструктурирующая привязка: Coord -> (Float, Float)
Coord.describe = process_coordinates[1]
# Использование: coord.describe() → process_coordinates((coord.x, coord.y))
```

### Привязка с множественными возвращаемыми значениями

```yaoxiang
# === Привязка с множественными возвращаемыми значениями ===

min_max: (list: List(Int)) -> (Int, Int) = {
    min = list.reduce(Int.MAX, (a, b) => if a < b then a else b)
    max = list.reduce(Int.MIN, (a, b) => if a > b then a else b)
    return (min, max)
}

List.range: (T:Type)->((self: List(T)) -> (T, T)) = min_max[1]
# Использование: (min_val, max_val) = list.range()
```

### Правила проверки типов

```rust
fn check_binding_type_compatibility(
    binding: &Binding,
    func: &Function
) -> Result<(), TypeError> {
    // 1. Если это автоматический поиск позиции (не указано явно),
    // проверить, найдено ли совпадение
    if binding.positions.is_empty() {
        return Err(TypeError::NoMatchingParameter(
            binding.type_name.clone(),
            func.name.clone()
        ));
    }

    // 2. Проверить, что все индексы позиций валидны
    for pos in &binding.positions {
        if *pos >= func.params.len() {
            return Err(TypeError::InvalidBindingPosition(*pos));
        }
    }

    // 3. Проверить совместимость типов в позициях привязки
    for pos in &binding.positions {
        let param_type = &func.params[*pos].type_;
        let binding_type = &binding.type_name;

        if !isAssignable(binding_type, param_type) {
            return Err(TypeError::IncompatibleTypes(
                binding_type, param_type
            ));
        }
    }

    // 4. Проверить соответствие параметров вызова метода оставшимся параметрам
    Ok(())
}
```

### Поведение во время выполнения

| Сценарий | Синтаксис привязки | Вызов | Преобразование |
|------|---------|------|--------|
| Привязка по умолчанию | `Point.distance = distance` | `p1.distance(p2)` | `distance(p1, p2)` |
| Автоматическое сопоставление | `Point.transform = transform` | `p.transform(v)` | `transform(p, v)` |
| Однопозиционная | `Point.distance = distance[1]` | `p1.distance(p2)` | `distance(p2, p1)` |
| Однопозиционная | `Point.test = func[-1]` | `p.test(a, b)` | `func(a, b, p)` |
| Автоматическое каррирование | `Point.scale = scale[0, _]` | `p.scale(2.0)` | `scale(p, 2.0)` |
| Заполнитель | `Type.method = func[1, _]` | `obj.method(arg)` | `func(arg, obj)` |

**Пояснения**:
- **Привязка по умолчанию**: автоматический поиск первой позиции соответствующего типа
- `[0]`: this привязан к позиции 0 (первый параметр)
- `[1]`: this привязан к позиции 1 (второй параметр)
- `[-1]`: this привязан к последней позиции (отсчёт с конца)

## Компромиссы

### Преимущества

- **Интеллектуальная привязка по умолчанию**: привязка по умолчанию к первой позиции соответствующего типа без явного указания `[positions]`
- **Точное управление**: можно привязать к любой позиции параметра, высокая гибкость
- **Типобезопасность**: полная проверка типов во время компиляции, привязка выполняется только при совпадении типов
- **Компактный синтаксис**: синтаксис `[position]` интуитивно понятен
- **Без ключевого слова `self`**: сохранение простоты языка
- **Дружественность к каррированию**: естественная поддержка частичного применения и цепочечных вызовов
- **Дружественность к ООП**: автоматическое каррирование упрощает миграцию для ООП-программистов

### Недостатки

- **Затраты на обучение**: требуется понимание концепции позиционных индексов
- **Сложность компиляции**:解析 привязок и проверка типов увеличивают сложность компилятора
- **Сложность отладки**: сообщения об ошибках должны чётко указывать на проблемы с позициями привязки

## Альтернативные решения

| Решение | Описание | Почему не выбрано |
|------|------|-----------|
| Ключевое слово `self` | Введение стиля `self` из Python/Rust | Нарушает философию дизайна YaoXiang без неявного `self` |
| Именованная привязка параметров | Использование именованных параметров `func(a=obj)` | Требует изменения определений сигнатур функций, увеличивает сложность |
| Макросистема | Реализация привязки через макросы | Накладные расходы во время выполнения, снижение типобезопасности |
| Перегрузка операторов | Ограничение `self` определёнными позициями | Синтаксис не унифицирован, семантика запутанная |

## Стратегия реализации

### Фазы разделения

1. **Фаза 1: Базовая привязка** (v0.3)
   - Реализация синтаксиса однопозиционной привязки `[n]` (n начинается с 0, поддержка отрицательных чисел)
   - Базовая проверка типов и кодогенерация
   - Покрытие юнит-тестами

2. **Фаза 2: Продвинутые возможности** (v0.5)
   - Поддержка синтаксиса диапазонов `[n..m]`
   - Оптимизация вычисления позиций во время компиляции

### Зависимости

- Нет внешних зависимостей
- Нет прямой связи с RFC-001 (обработка ошибок)
- Может быть реализовано независимо

### Риски

- Совместимость с существующим синтаксисом привязок
- Стратегии оптимизации производительности (разворачивание во время компиляции vs поиск во время выполнения)

## Приложения

### Приложение A: Журнал решений по дизайну

| Решение | Решение | Обоснование |
|------|------|-----------|
| База индексов | Начинается с 0 | Согласованность с индексацией кортежей/списков параметров |
| Отрицательные индексы | Поддерживаются | Гибкость, отсчёт с конца |
| Заполнитель | `_` | Лаконичность, универсальный символ |
| Синтаксис диапазонов | Реализовать | Пакетная привязка, например `[0..2]` |
| Стиль синтаксиса | Инфиксный `Type.method = func[positions]` | Унификация с RFC-010 |
| **Логика привязки по умолчанию** | **Привязка к первой позиции соответствующего типа** | **Более интеллектуально, безопасно, соответствует интуиции** |
| **Обработка ошибок привязки** | **При невозможности найти совпадение — ошибка/предупреждение/фабричная функция** | **Гибкая обработка в зависимости от контекста** |
| **Синтаксис функций** | **Имя параметра в сигнатуре `name: (params) -> Return`** | **Унификация с RFC-010** |

### Приложение B: Глоссарий

| Термин | Определение |
|------|------|
| Позиция привязки | Индекс позиции в списке параметров функции |
| Объединённая привязка | Привязка типа к нескольким позициям параметров |
| Частичное применение | Предоставление только части параметров, возвращается функция для завершения вызова |
| **Унифицированный синтаксис** | **`name: (params) -> Return = body`, имя параметра объявляется в сигнатуре** |
| **Привязка с сопоставлением типов** | **Логика привязки по умолчанию: автоматический поиск первой позиции, соответствующей типу вызывающей стороны** |
| **Привязка фабричной функции** | **Когда среди параметров функции нет соответствующего типа, использование в качестве конструктора** |
