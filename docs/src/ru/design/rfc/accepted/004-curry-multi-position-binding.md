# RFC-004: Дизайн мультипозиционного объединённого связывания для каррированных методов

## Аннотация

Данный RFC предлагает совершенно новый синтаксис **мультипозиционного объединённого связывания**, позволяющий точно связывать функции с произвольными позициями параметров типа, поддерживающий как однопозиционное, так и мультипозиционное объединённое связывание. Это фундаментально решает проблему «кто является вызывающим» при каррированном связывании, не требуя введения ключевого слова `self`.

## Мотивация

### Зачем нужна эта возможность?

В текущем дизайне языка привязка независимой функции в качестве метода типа сталкивается со следующими проблемами:

1. **Негибкость позиции вызывающего**: традиционное связывание может зафиксировать `obj` только как первый параметр в `obj.method(args)`
2. **Сложность связывания нескольких параметров**: когда метод должен принимать несколько параметров одного типа, невозможно выразить это элегантно
3. **Двусмысленность семантики каррирования**: при частичном применении трудно отличить «к какой позиции выполнено связывание»

### Цель дизайна: унификация двух парадигм программирования

Данный дизайн направлен на **унификацию функциональной и ООП парадигм**:

```yaoxiang
# Функциональная перспектива: явная передача всех аргументов
distance(p1, p2)

# ООП перспектива: неявный this
p1.distance(p2)

# Синтаксический сахар [positions] делает обе записи эквивалентными — по сути это вызовы функций
Point.distance = distance[0]   # this связывается с позицией 0
```

**Ключевые преимущества**:
- Внизу — функция, наверху — синтаксис метода
- Без введения ключевого слова `self`, сохраняя краткость языка
- Полная функциональность: вызов метода по сути является передачей аргументов
- `[0]`, `[1]`, `[-1]` — гибкое управление позицией связывания this
- **Единый синтаксис**: определение функции использует формат `name: (params) -> Return = body`

### Текущая проблема

```yaoxiang
# Проблема существующего дизайна:
Point: Type = { x: Float, y: Float }
Vector: Type = { x: Float, y: Float, z: Float }

distance: (a: Point, b: Point) -> Float = { ... }
transform: (p: Point, v: Vector) -> Point = { ... }

# Можно связать только с первым параметром
Point.distance = distance  # эквивалентно distance[0]
# p1.distance(p2) → distance(p1, p2) ✓

# Но что если сигнатура transform имеет вид transform(Vector, Point)?
# Невозможно выразить семантику p1.transform(v1) → transform(v1, p1)
```

## Предложение

### Основной дизайн: явное указание позиции

**Ключевое правило: отсутствие `[n]` означает отсутствие связывания.** `Point.name = func` — это лишь псевдоним пространства имён, не вызывающий никакого неявного связывания. Чтобы синтаксис вызова через `.` вида `p.name(args)` работал, необходимо явное указание: `Point.name = func[n]`.

#### Однопозиционное связывание

```yaoxiang
# Явное связывание с первой позицией параметра типа Point (индексация с 0)
Point.distance = distance[0]
p1.distance(p2)                     # → distance(p1, p2)

# Связывание со второй позицией параметра типа Point
Point.compare = distance[1]         # связывание со вторым параметром Point
p1.compare(p2)                      # → distance(p2, p1)
```

**Без `[n]` = без связывания**:

```yaoxiang
# Без [n] → чистый псевдоним пространства имён, синтаксис вызова через . не работает
Point.distance = distance            # доступно только Point.distance(p1, p2)
# p1.distance(p2)  ❌  связывание отсутствует

# Фабричные функции естественно легальны, без специальной обработки
create_point: () -> Point = { ... }
Point.create = create_point          # Point.create()   ✅
```
- Типобезопасность: связывание происходит только при совпадении типов, что предотвращает ошибки
- Гибкое управление: `[n]` обеспечивает точный контроль позиции связывания

#### Каррированное связывание

Когда количество параметров функции превышает количество позиций связывания, автоматически генерируется каррированная функция. **Связывание всегда является явной операцией.**

```yaoxiang
Point: Type = { x: Float, y: Float }

# Базовая функция
scale: (p: Point, factor: Float) -> Point = {
    return Point(p.x * factor, p.y * factor)
}

# Явное связывание с позицией 0 → каррирование: оставшийся параметр factor предоставляется вызывающим
Point.scale = scale[0]

# Вызов
p1 = Point(2.0, 3.0)
scaled = p1.scale(2.0)       # → scale(p1, 2.0)

# Цепочечный вызов становится элегантнее
result = Point(2.0, 3.0).scale(2.0)  # → Point(4.0, 6.0)
```

### Синтаксис позиционного индексированного связывания

Вводится синтаксис `[position]` для точного управления связью между параметрами функции и типом:

```yaoxiang
# Формат синтаксиса: Type.method = function[positions]

# === Базовое связывание ===

# Однопозиционное связывание
Point.distance = distance[1]           # связывание с 1-м параметром (индексация с 0)
# Использование: p1.distance(p2) → distance(p2, p1)

# Мультипозиционное объединённое связывание (деструктуризация кортежа)
Point.transform = transform[1, 2]      # связывание с 1-м и 2-м параметрами
# Использование: p1.transform(v1) → transform(v1, p1)
# Исходная сигнатура: transform(Point, Vector) → Point
# После связывания: Point.transform(Vector) → Point
```

### Подробное определение синтаксиса

```
Объявление_связывания ::= Тип '.' Идентификатор '=' Имя_функции '[' Список_позиций ']'

Список_позиций ::= Позиция (',' Позиция)*
Позиция         ::= Целое_число               # плейсхолдер
                  | '_'                        # пропуск позиции (плейсхолдер)
                  | Целое_число '..' Целое_число  # диапазон позиций (будущее расширение)

Имя_функции     ::= Идентификатор
Тип             ::= Идентификатор (Дженерик_параметры)?
```

### Встроенные связывания

Связывание может быть записано непосредственно в теле определения типа, без отдельного оператора связывания:

```yaoxiang
# Способ 1: связывание непосредственно в теле определения типа
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]           # связывание с позицией 0
}

# Способ 2: анонимная функция + позиционное связывание
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
# Синтаксис: ((params) => body)[position]
```

**Семантика каррирования**:
- При связывании `distance = distance[0]` исходная сигнатура `(a: Point, b: Point) -> Float`
- Сгенерированная сигнатура метода: `b: Point -> Float` (позиция 0 заполняется вызывающим)

### Примеры использования

```yaoxiang
# === Полный пример ===

Point: Type = { x: Float, y: Float }
Vector: Type = { x: Float, y: Float, z: Float }

# 1. Базовый расчёт расстояния
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    return (dx * dx + dy * dy).sqrt()
}

# Связывание: Point.distance = distance[1]
# Вызов: p1.distance(p2) → distance(p2, p1)
# Но мы хотим p1.distance(p2) → distance(p1, p2), поэтому:
Point.distance = distance[0]

# 2. Операция трансформации (мультипозиционное связывание)
transform: (p: Point, v: Vector) -> Point = {
    return Point(p.x + v.x, p.y + v.y)
}

# Связывание Point.transform = transform[1]
# Вызов: p.transform(v) → transform(v, p) ❌
# Связывание Point.transform = transform[0]
# Вызов: p.transform(v) → transform(p, v) ✓

# 3. Сложная функция с множеством параметров
multiply: (a: Point, s: Float) -> Point = {
    return Point(a.x * s, a.y * s)
}

# Связывание только с 1-м параметром (тип Point), сохранение 3-го параметра
Point.scale = multiply[0, _]
# Вызов: p.scale(2.0) → multiply(p, 2.0)

# 4. Межтиповое связывание
Circle: Type = { center: Point, radius: Float }

distance: (a: Circle, b: Circle) -> Float = {
    return a.center.distance(b.center) - a.radius - b.radius
}

# Связывание метода distance с типом Circle
Circle.distance = distance[0, 1]
# Вызов: c1.distance(c2) → distance(c1, c2)
```

### Поддержка деструктуризации кортежей

```yaoxiang
# === Связывание с деструктуризацией кортежа ===

# Функция принимает параметр-кортеж
process_coordinates: (coord: (Float, Float)) -> String = {
    return match coord {
        (0.0, 0.0) -> "origin"
        (x, 0.0) -> "on x-axis at ${x}"
        (0.0, y) -> "on y-axis at ${y}"
        (x, y) -> "point at (${x}, ${y})"
    }
}

Coord: Type = { x: Float, y: Float }

# Автоматическое деструктурирующее связывание: Coord -> (Float, Float)
Coord.describe = process_coordinates[1]
# Использование: coord.describe() → process_coordinates((coord.x, coord.y))
```

### Связывание с множественным возвращаемым значением

```yaoxiang
# === Связывание с множественным возвращаемым значением ===

min_max: (list: List(Int)) -> (Int, Int) = {
    min = list.reduce(Int.MAX, (a, b) => if a < b then a else b)
    max = list.reduce(Int.MIN, (a, b) => if a > b then a else b)
    return (min, max)
}

List.range: (T:Type)->((self: List(T)) -> (T, T)) = min_max[1]
# Использование: (min_val, max_val) = list.range()
```

### Правила проверки типов

```rust
fn check_binding_type_compatibility(
    binding: &Binding,
    func: &Function
) -> Result<(), TypeError> {
    // 1. Если позиция определяется автоматически (явно не указана), проверить, найдено ли совпадение
    if binding.positions.is_empty() {
        return Err(TypeError::NoMatchingParameter(
            binding.type_name.clone(),
            func.name.clone()
        ));
    }

    // 2. Проверить валидность всех индексов позиций
    for pos in &binding.positions {
        if *pos >= func.params.len() {
            return Err(TypeError::InvalidBindingPosition(*pos));
        }
    }

    // 3. Проверить совместимость типов в позициях связывания
    for pos in &binding.positions {
        let param_type = &func.params[*pos].type_;
        let binding_type = &binding.type_name;

        if !isAssignable(binding_type, param_type) {
            return Err(TypeError::IncompatibleTypes(
                binding_type, param_type
            ));
        }
    }

    // 4. Проверить соответствие аргументов вызова метода оставшимся параметрам
    Ok(())
}
```

### Поведение во время выполнения

| Сценарий | Синтаксис связывания | Вызов | Преобразуется в |
|----------|---------------------|-------|-----------------|
| Без связывания | `Point.distance = distance` | `Point.distance(p1, p2)` | `distance(p1, p2)` |
| Однопозиционное | `Point.distance = distance[0]` | `p1.distance(p2)` | `distance(p1, p2)` |
| Однопозиционное | `Point.distance = distance[1]` | `p1.distance(p2)` | `distance(p2, p1)` |
| Отрицательный индекс | `Point.test = func[-1]` | `p.test(a, b)` | `func(a, b, p)` |
| Мультипозиционное (каррирование) | `Point.scale = scale[0]` | `p.scale(2.0)` | `scale(p, 2.0)` |
| Плейсхолдер | `Type.method = func[1]` | `obj.method(arg)` | `func(arg, obj)` |

**Пояснения**:
- **Без связывания**: `Point.name = func` — лишь псевдоним пространства имён, без синтаксиса вызова через `.`
- `[0]`: вызывающий связывается с позицией 0 (первый параметр)
- `[1]`: вызывающий связывается с позицией 1 (второй параметр)
- `[-1]`: вызывающий связывается с последней позицией (отсчёт с конца)

## Компромиссы

### Преимущества

- **Явное связывание**: `[n]` — единственный механизм связывания, отсутствие записи означает отсутствие связывания, никакого неявного поведения
- **Точный контроль**: возможность связывания с произвольной позицией параметра, высокая гибкость
- **Типобезопасность**: полная проверка типов во время компиляции, связывание только при совпадении типов
- **Краткость синтаксиса**: синтаксис `[position]` интуитивно понятен
- **Отсутствие ключевого слова `self`**: сохраняет краткость языка
- **Дружелюбность к каррированию**: естественная поддержка частичного применения и цепочечных вызовов
- **Дружелюбность к ООП**: автоматическое каррирование позволяет ООП-программистам легко мигрировать

### Недостатки

- **Стоимость обучения**: требуется понимание концепции позиционного индексирования
- **Сложность компиляции**: разрешение связывания и проверка типов усложняют компилятор
- **Сложность отладки**: сообщения об ошибках должны чётко указывать на проблемы с позициями связывания

## Альтернативы

| Подход | Описание | Почему не выбран |
|--------|----------|------------------|
| Ключевое слово `self` | Введение `self` в стиле Python/Rust | Противоречит философии дизайна YaoXiang об отсутствии неявного `self` |
| Связывание по именованным параметрам | Использование именованных параметров `func(a=obj)` | Требует изменения определения сигнатуры функции, увеличивает сложность |
| Система макросов | Реализация связывания через макросы | Большие накладные расходы во время выполнения, снижение типобезопасности |
| Перегрузка операторов | Ограничение `self` определёнными позициями | Непоследовательный синтаксис, семантическая путаница |

## Стратегия реализации

### Этапы

1. **Фаза 1: Базовое связывание** (v0.3)
   - Реализация синтаксиса однопозиционного связывания `[n]` (n с 0, поддержка отрицательных значений)
   - Базовая проверка типов и генерация кода
   - Покрытие модульными тестами

2. **Фаза 2: Расширенные возможности** (v0.5)
   - Поддержка синтаксиса диапазонов `[n..m]`
   - Оптимизация вычисления позиций во время компиляции

### Зависимости

- Нет внешних зависимостей
- Нет прямой связи с RFC-001 (обработка ошибок)
- Может быть реализовано независимо

### Риски

- Обработка совместимости с существующим синтаксисом связывания
- Стратегии оптимизации производительности (раскрытие во время компиляции vs поиск во время выполнения)

## Приложения

### Приложение A: Записи решений по дизайну

| Решение | Выбор | Обоснование |
|---------|-------|-------------|
| Базис индексации | С 0 | Соответствует индексации кортежей/списков параметров |
| Отрицательные индексы | Поддерживаются | Гибкость, отсчёт с конца |
| Плейсхолдер | `_` | Краткий, общепринятый символ |
| Синтаксис диапазонов | Реализуется | Пакетное связывание, например `[0..2]` |
| Стиль синтаксиса | Инфиксный `Type.method = func[positions]` | Унификация с RFC-010 |
| **Правило связывания** | **Только явное `[n]` вызывает связывание, без записи — без связывания** | **Нет неявного поведения, определение и связывание ортогональны** |
| **Пространство имён** | **`Type.name` — лишь принадлежность к пространству имён, не вызывает связывания** | **Разделение определения и связывания** |
| **Синтаксис функций** | **Имена параметров в сигнатуре `name: (params) -> Return`** | **Унификация с RFC-010** |

### Приложение B: Глоссарий

| Термин | Определение |
|--------|-------------|
| Позиция связывания | Индекс позиции в списке параметров функции |
| Объединённое связывание | Связывание типа с несколькими позициями параметров |
| Частичное применение | Предоставление только части параметров, возврат функции, ожидающей завершения вызова |
| **Единый синтаксис** | **`name: (params) -> Return = body`, имена параметров объявляются в сигнатуре** |
| **Функция в пространстве имён** | **Синтаксис `Type.name`, функция принадлежит пространству имён Type, без неявного связывания** |
| **Явное связывание** | **`Type.name = func[n]`, единственный механизм связывания методов** |
