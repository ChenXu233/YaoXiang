> **Устаревшее**:

## Аннотация

Данный документ описывает дизайн LLVM AOT (Ahead-of-Time) компилятора для языка YaoXiang. LLVM-бэкенд и VM-бэкенд (интерпретатор) используют общий компиляторный фронтенд, образуя архитектуру с двумя бэкендами, определённую в [RFC-008](../accepted/008-runtime-concurrency-model.md): VM применяется для разработки и отладки, LLVM — для production-сборок.

**Основные обязанности**:

```
Исходный код → Фронтенд (общий) → IR → LLVM Codegen → .o → Линковка со статической библиотекой планировщика → exe
```

Компилятор преобразует исходный код YaoXiang в нативный машинный код, при этом:

| Возможность языка | Стратегия компиляции |
|----------|----------|
| Обычный код | Последовательный машинный код, нулевые накладные расходы на планирование |
| `spawn { }` блок | Прямое подвыражение → распределение задач + синхронное ожидание (согласуется с [RFC-024](../accepted/024-concurrency-model.md)) |
| `native("symbol")` | LLVM `declare external` + marshalling аргументов (согласуется с [RFC-026](./026-ffi-core-mechanism.md)) |
| `.drop` деструктор | Вставка RAII-кода очистки (согласуется с [RFC-009](../accepted/009-ownership-model.md)) |
| `&T` / `&mut T` токены | Типы нулевого размера, исчезают после компиляции |
| `ref T` общий доступ | Толстый указатель `{ refcount_ptr, data_ptr }`, компилятор автоматически выбирает Rc/Arc |

**Связь с RFC-024**: RFC-024 определяет **пользовательскую семантику** spawn блоков (создание задач прямыми подвыражениями, синхронное блокирующее ожидание). Настоящий документ определяет, **как эта семантика компилируется в машинный код**.

**Связь с RFC-026**: RFC-026 определяет **пользовательский синтаксис** FFI (`native()`, привязка методов через `[0]`, `.drop`). Настоящий документ определяет, **как FFI-вызовы генерируют LLVM IR**.

## Мотивация

### Зачем нужен LLVM AOT компилятор?

В настоящее время YaoXiang имеет только интерпретатор в качестве исполнительного бэкенда:

| Проблема | Влияние |
|------|------|
| Узкое место производительности | Интерпретация в 10-100x медленнее машинного кода |
| Сложность развёртывания | Требуется тащить интерпретатор и runtime |
| Продакшн-среда | Интерпретатор не подходит для сценариев, чувствительных к производительности |

### LLVM в модели с двумя бэкендами

[RFC-008](../accepted/008-runtime-concurrency-model.md) §6 определяет архитектуру с двумя бэкендами:

```
                    ┌─────────────────────┐
                    │  Фронтенд (унифицированный) │
                    │  Lexer → Parser      │
                    │  → TypeCheck         │
                    │  → spawn анализ      │
                    │  → escape анализ     │
                    └──────────┬──────────┘
                               │
                  ┌────────────┴────────────┐
                  ▼                         ▼
      ┌───────────────────┐     ┌───────────────────┐
      │  VM бэкенд (разработка) │     │ LLVM бэкенд (продакшн) │
      │  IR → интерпретация │     │  IR → нативный код      │
      │  Пошаговая отладка  │     │  Линковка со статической │
      │  Быстрая итерация   │     │  библиотекой планировщика│
      └───────────────────┘     └───────────────────┘
```

**Поведение обоих бэкендов полностью идентично** — различается только способ исполнения. Один и тот же исходный код, один и тот же набор проверки типов, один и тот же результат анализа spawn блоков.

## Предложение

### 1. Архитектура компилятора

LLVM-бэкенд располагается на последнем этапе конвейера компиляции, принимая IR от фронтенда и генерируя нативный код:

```
Исходный код
  → Lexer / Parser (frontend/core/)
  → TypeCheck + анализ spawn блоков (frontend/core/typecheck/)
  → Генерация IR (middle/core/ir_gen.rs)
  → LLVM Codegen (backends/llvm/)
      ├── Отображение типов: типы YaoXiang → типы LLVM IR
      ├── Трансляция функций: инструкции IR → инструкции LLVM IR
      ├── Развёртывание spawn: прямые подвыражения → функции задач + вызовы планировщика
      ├── Развёртывание FFI: вызовы native() → declare + marshalling
      └── Вставка деструкторов: конец области видимости → вызов .drop()
  → LLVM-оптимизации + генерация целевого кода
  → Линковка со статической библиотекой runtime → исполняемый файл
```

### 2. Процесс компиляции

```
Фаза 1: Фронтенд (общий с VM бэкендом)
  - Парсинг, проверка типов, анализ spawn блоков, escape анализ
  - Выход: IR с аннотациями типов

Фаза 2: Генерация LLVM IR
  - Отображение типов, объявления функций, трансляция инструкций
  - Выход: LLVM Module

Фаза 3: LLVM-оптимизации
  - Стандартный конвейер LLVM-оптимизаций (O0/O1/O2/O3)
  - Встраивание, свёртка констант, удаление мёртвого кода

Фаза 4: Генерация целевого кода
  - LLVM TargetMachine → .o файлы
  - Платформы: Linux (ELF), macOS (Mach-O), Windows (COFF)

Фаза 5: Линковка
  - Линковка со статической библиотекой runtime (планировщик, аллокатор)
  - Выход: исполняемый файл
```

### 3. Отображение типов

#### 3.1 Отображение типов YaoXiang → LLVM IR

| Тип YaoXiang | Тип LLVM IR | Пояснение |
|---------------|-------------|------|
| `Int` | `i64` | По умолчанию 64-битное знаковое целое |
| `Int32` | `i32` | Явное 32-битное целое (главным образом для FFI) |
| `Float` | `f64` | По умолчанию 64-битное с плавающей запятой |
| `Float32` | `f32` | Явное 32-битное с плавающей запятой (главным образом для FFI) |
| `Bool` | `i1` | Логическое значение |
| `Char` | `i32` | Unicode-кодпоинт |
| `String` | `{ i8*, i64 }` | Указатель + длина в байтах |
| `Void` | `{}` | Пустой тип нулевого размера |
| `&T` | — | Токен нулевого размера, исчезает после компиляции, не порождает IR |
| `&mut T` | — | Токен нулевого размера, исчезает после компиляции, не порождает IR |
| `ref T` | `{ i64*, T* }` | Толстый указатель (указатель счётчика ссылок + указатель на данные) |
| `*T` | `T*` | Сырой указатель |
| `[T; N]` | `[N x T]` | Массив фиксированной длины |
| `List(T)` | `{ T*, i64, i64 }` | Указатель на данные + длина + ёмкость |
| Структура | Соответствующий LLVM struct | Поля располагаются в порядке объявления |
| Запись-перечисление | `{ i64, [max_payload_size] }` | Тег + union максимального размера payload |
| `?T` | `{ i1, T }` | Флаг наличия значения + данные (универсальное представление) |
| Непрозрачный тип FFI | `{ i8* }` | Обёртка над C-указателем |
| Указатель на функцию | `T (...)*` | Тип указателя на функцию |

> **Нулевые накладные расходы для `&T` / `&mut T`**: [RFC-009](../accepted/009-ownership-model.md) §2.7 определяет, что компилятор внутренне назначает токенам уникальный идентификатор бренда (уникальное целое в compile-time), после мономорфизации и встраивания бренд полностью исчезает — в сгенерированном машинном коде не остаётся никаких следов токенов.

### 4. Нормализация IR и трансляция инструкций

#### 4.0 Нормализация IR (стек → регистры)

Текущий IR (`src/middle/core/ir.rs`) содержит стековые инструкции (`Push`/`Pop`/`Dup`/`Swap`), спроектированные для байт-код VM. LLVM IR имеет SSA-форму и не принимает стековые операции.

**Стратегия обработки**: путь LLVM перед трансляцией инструкций проходит через облегчённый проход нормализации:

| Стековая инструкция | Стратегия нормализации |
|--------|-----------|
| `Push(r)` | Запись `stack.push(r)`, IR не порождается |
| `Pop(r)` | `r = stack.pop()`, порождается `load` (из стекового слота) |
| `Dup` | `stack.push(stack.top())`, IR не порождается |
| `Swap` | Обмен двух верхних элементов стека, IR не порождается |

После нормализации все операнды становятся ссылками на регистры/локальные переменные, стековые операции полностью устраняются. Этот проход выполняется первым шагом в `translator.rs`.

> **Почему не устранить стековые инструкции на уровне IR?** Потому что VM-бэкенду нужна стековая семантика. Нормализация на входе в LLVM-трансляцию сохраняет IR общим для обоих бэкендов — каждый бэкенд потребляет один и тот же IR согласно своим нуждам.
>
> **Предусловие**: фаза генерации IR гарантирует баланс стека — на всех путях управления в одной и той же программной точке глубина стека одинакова (байт-код VM бэкенда зависит от того же предусловия, иначе выполнение байт-кода будет ошибочным). Проход нормализации это предусловие не проверяет; при нарушении LLVM бэкенд порождает неопределённое поведение.

|---------|---------|------|
| `StringLength { dst, src }` | `%len = extractvalue { i8*, i64 } %src, 1` | String — это `{ ptr, len }`, длина в поле 1 |
| `StringConcat { dst, lhs, rhs }` | `call String @yx_string_concat(%lhs, %rhs)` | Runtime-функция-помощник |
| `StringGetChar { dst, src, index }` | `getelementptr` + `load i32` | С проверкой границ |
| `StringFromInt { dst, src }` | `call String @yx_string_from_int(%src)` | Runtime-функция-помощник |
| `StringFromFloat { dst, src }` | `call String @yx_string_from_f64(%src)` | Runtime-функция-помощник |

**Инструкции для замыканий**:

| IR-инструкция | LLVM IR | Пояснение |
|---------|---------|------|
| `MakeClosure { dst, func: String, env }` | Аллокация структуры замыкания + заполнение указателя на функцию (по имени функции) и окружения | `{ fn_ptr, env_fields... }` |
| `LoadUpvalue { dst, upvalue_idx }` | `%v = extractvalue %env, upvalue_idx` | Чтение upvalue из окружения замыкания |
| `StoreUpvalue { src, upvalue_idx }` | `%env = insertvalue %env, %src, upvalue_idx` | Запись upvalue в окружение замыкания |
| `CloseUpvalue(val)` | Копирует стековый upvalue в кучу | |

**Прочие инструкции**:

| IR-инструкция | LLVM IR | Пояснение |
|---------|---------|------|
| `HeapAlloc { dst, type_id }` | `call i8* @malloc(i64 size)` + запись тега типа | Аллокация в куче + информация о типе |
| `NewDict { dst, keys, values }` | `call Dict @yx_dict_new(%keys, %values)` | Runtime-функция-помощник |

> **Примечание**: `Push`/`Pop`/`Dup`/`Swap` уже устранены на этапе нормализации в §4.0 и не появляются в таблице трансляции. `Borrow`/`Release` — это токены нулевого размера в compile-time, машинного кода не порождают.

### 5. Генерация кода для spawn блоков

Согласуется с [RFC-024](../accepted/024-concurrency-model.md), компиляция spawn блока разбивается на следующие шаги.

#### 5.1 Напоминание семантики

```yaoxiang
(r1, r2) = spawn {
    t1 = fetch("url1"),   // прямое подвыражение → задача 1
    t2 = fetch("url2"),   // прямое подвыражение → задача 2
    return (t1, t2)       // синхронное ожидание, сборка результата
}
```

**Правила** (RFC-024 §2.1):
- **Прямые подвыражения** spawn блока (верхнеуровневые операторы, разделённые запятыми) порождают параллельные задачи
- Выражения внутри вложенных `{}` не считаются прямыми подвыражениями и не становятся отдельными задачами
- Весь spawn блок синхронно блокирует, ожидая завершения всех задач, после чего возвращает результат

#### 5.2 Шаги компиляции

```
Шаг 1: Идентификация прямых подвыражений
  Обход тела spawn блока, сбор верхнеуровневых операторов

Шаг 2: Анализ зависимостей
  Для каждого прямого подвыражения анализируется, на какие переменные,
  порождённые предыдущими задачами, оно ссылается
  Нет зависимостей → может быть диспетчеризовано немедленно параллельно
  Есть зависимости → поставлено в очередь на ожидание задач-зависимостей

Шаг 3: Обнаружение конфликтов ресурсов (RFC-024 §2.5)
  Проверка, не используется ли один и тот же экземпляр ресурса несколькими задачами
  Конфликт по тому же экземпляру → пометить как сериализуемый

Шаг 4: Генерация функций задач
  Каждое прямое подвыражение порождает отдельную LLVM-функцию (замыкание)

Шаг 5: Генерация кода диспетчеризации
  Вызов runtime-планировщика task_spawn / task_wait

Шаг 6: Сборка результата
  Сбор выходных данных всех задач, формирование возвращаемого кортежа
```

#### 5.3 Шаблон генерации LLVM IR

```llvm
; вход в spawn блок
%task_count = 2
%tasks = alloca [2 x %TaskHandle]

; создание задачи 1: fetch("url1")
%task1_fn = @spawn_closure_1
call @runtime_task_spawn(%tasks[0], %task1_fn, ...)

; создание задачи 2: fetch("url2")
%task2_fn = @spawn_closure_2
call @runtime_task_spawn(%tasks[1], %task2_fn, ...)

; синхронное ожидание всех задач
call @runtime_task_wait_all(%tasks, %task_count)

; сборка возвращаемого значения
%r1 = call @runtime_task_result(%tasks[0])
%r2 = call @runtime_task_result(%tasks[1])
ret { %r1, %r2 }
```

#### 5.4 Зависимые задачи

```yaoxiang
result = spawn {
    data = fetch("url"),       // задача 1: без зависимостей
    processed = parse(data),   // задача 2: зависит от data из задачи 1
    return processed
}
```

Компилятор обнаруживает, что `parse(data)` ссылается на `data`, порождённое задачей 1, и при генерации кода диспетчеризации помечает зависимость:

```llvm
; задача 2 создаётся с зависимостью от задачи 1
call @runtime_task_spawn_with_dep(%tasks[1], %task2_fn, %tasks[0])
;                                                              ↑
;                                              зависит от задачи 0 (fetch)
```

#### 5.5 Автоматическая сериализация по типу-ресурсу

Типы-ресурсы ([RFC-024 §2.5](../accepted/024-concurrency-model.md)) — `FilePath`, `HttpUrl`, `DBUrl`, `Console`, а также определённые пользователем — в spawn блоке автоматически сериализуются:

```yaoxiang
(a, b) = spawn {
    r1 = db.exec("SELECT ..."),   // использует SqliteDb (тип-ресурс)
    r2 = db.exec("INSERT ...")    // тот же экземпляр → автоматическая сериализация
}
```

Компилятор обнаруживает, что один и тот же экземпляр ресурса используется двумя задачами, и порождает сериализующую зависимость:

```llvm
; задача 2 зависит от задачи 1 (автосериализация по тому же ресурсу)
call @runtime_task_spawn_with_dep(%tasks[1], %task2_fn, %tasks[0])
```

#### 5.6 spawn for для data parallelism

```yaoxiang
results = spawn for item in items {
    process(item)
}
```

Компилятор разворачивает в N независимых задач (N = длина items), ограниченных максимальным уровнем конкурентности.

### 6. Генерация кода для FFI

> ⚠️ **Замечание о зависимостях**: **Архитектура** генерации FFI-кода, определённая в данном разделе (`native("x")` → `declare external @x` → функция-обёртка marshalling → call) стабильна и не меняется при изменении синтаксиса RFC-026. Конкретные таблицы правил marshalling аргументов (§6.2) и раскладки непрозрачных типов (§6.3) ссылаются на определения RFC-026 — если синтаксис `native()` или правила marshalling в RFC-026 изменятся, потребуется обновить лишь соответствующие таблицы отображения в данном документе, архитектурный уровень не пострадает. Текущий статус RFC-026: **На рассмотрении**, находится в каталоге `review/` вместе с настоящим документом.
>
> **Предусловие принятия**: до принятия настоящего RFC части RFC-026, относящиеся к §6 данного документа (синтаксис объявлений `native()`, правила marshalling аргументов, раскладка непрозрачного типа `{ i8* }`, соглашение о привязке `.drop`) должны быть заморожены или приняты вместе с 026. Иначе таблицы отображения §6.2/§6.3/§7 могут устареть до реализации.

Согласуется с [RFC-026](./026-ffi-core-mechanism.md), в данном разделе определяется стратегия генерации LLVM IR для FFI-вызовов.

#### 6.1 Объявление функций native()

```yaoxiang
sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")
```

Компилируется в LLVM IR:

```llvm
; объявление внешней C-функции
declare i8* @sqlite3_open(i8*)

; функция-обёртка YaoXiang (выполняет marshalling)
define { i8* } @__yx_sqlite3_open({ i8*, i64 } %filename) {
    ; marshalling: YaoXiang String → C-строка
    %c_str = extractvalue { i8*, i64 } %filename, 0
    ; вызов C-функции
    %raw = call i8* @sqlite3_open(i8* %c_str)
    ; unmarshalling: C-указатель → непрозрачный тип
    %result = insertvalue { i8* } undef, i8* %raw, 0
    ret { i8* } %result
}
```

**Ключевые моменты**:
- `native("sqlite3_open")` → `declare external @sqlite3_open`
- Компилятор автоматически генерирует функцию-обёртку для marshalling
- Сигнатура обёртки использует типы YaoXiang, внутри выполняется преобразование к C-типам

#### 6.2 Marshalling аргументов

| Направление | Преобразование |
|------|------|
| YaoXiang `String` → C `char*` | Передаётся поле `.ptr` |
| YaoXiang `Int32` → C `int` | Прямая передача (`i32`) |
| YaoXiang `*Void` → C `void*` | Прямая передача (`i8*`) |
| YaoXiang `T` (прозрачный тип) → C `struct T*` | Передаётся адрес |
| YaoXiang `T` (непрозрачный тип) → C `struct T*` | Передаётся указатель, извлечённый из `{ i8* }` |

#### 6.3 LLVM-раскладка непрозрачного типа

Непрозрачный тип, определённый в [RFC-026](./026-ffi-core-mechanism.md) §4.1:

```yaoxiang
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void
    }
    return SqliteDb
}
```

LLVM-раскладка: `{ i8* }` — структура, содержащая C-указатель.

**Оптимизация раскладки**: если непрозрачный тип имеет единственное поле `handle: *Void`, его можно оптимизировать до прямого использования `i8*` (без обрамляющей структуры). Оптимизированный ABI полностью совместим с C-указателем, накладные расходы на marshalling равны нулю. Компилятор включает эту оптимизацию по умолчанию, пользователь её не замечает.

#### 6.4 LLVM-представление `?T` (Nullable) для возвращаемых значений FFI

Nullable-возвращаемые значения FFI, определённые в [RFC-026](./026-ffi-core-mechanism.md) §7.6:

```yaoxiang
sqlite3_open: (filename: String) -> ?SqliteDb = native("sqlite3_open")
```

Универсальное LLVM-представление: `{ i1, { i8* } }` — флаг наличия значения + данные.

**Оптимизация для FFI null-указателя**: если `T` в `?T` — непрозрачный тип (внутри указатель), компилятор использует оптимизацию **null-указатель = None**:

```llvm
; оптимизированное LLVM-представление: прямое использование nullable-указателя
define i8* @__yx_sqlite3_open(...) {
    %raw = call i8* @sqlite3_open(...)
    ; null → None, не null → Some(обёрнутый в непрозрачный тип)
    ret i8* %raw
}
```

Со стороны вызывающего:
```llvm
%raw = call i8* @__yx_sqlite3_open(...)
%is_null = icmp eq i8* %raw, null
br i1 %is_null, label %none_branch, label %some_branch
```

Эта оптимизация делает FFI-вызовы `?SqliteDb` **полностью без дополнительных накладных расходов** — эквивалентны по стоимости проверке null в C.

#### 6.5 Интеграция с yx-bindgen

Файлы привязок, автоматически сгенерированные [yx-bindgen](./026-ffi-core-mechanism.md) §6, обрабатываются компилятором как обычный исходный код YaoXiang. Компилятору не нужно знать, что код получен из bindgen — обработка объявлений `native()` и определений типов `unsafe {}` совершенно одинакова.

### 7. Генерация кода для деструкторов

Согласуется с семантикой RAII из [RFC-009](../accepted/009-ownership-model.md) и соглашением о `.drop` из [RFC-026](./026-ffi-core-mechanism.md) §7.

#### 7.1 Распознавание привязки `.drop`

```yaoxiang
SqliteDb.drop = sqlite3_close[0]
```

Компилятор распознаёт привязку `.drop` и помечает указатель на функцию-деструктор в метаданных типа.

#### 7.2 Вставка cleanup в конце области видимости

```
Пользовательский код:
{
    db = SqliteDb.open("test.db")
    stmt = db.prepare("SELECT ...")
    stmt.step()
    // ← конец области видимости
}

Вставленный компилятором cleanup (в обратном порядке):
    call @sqlite3_finalize(%stmt)    // stmt.drop()
    call @sqlite3_close(%db)          // db.drop()
```

**Точки вставки**:
- Нормальное окончание области видимости (`}`)
- Досрочный возврат (перед `return`)
- Путь распространения ошибки `?` (перед `?`)
- Окончание spawn блока (деструкторы переменных внутри задачи)

#### 7.3 Move и деструкторы

```yaoxiang
db = SqliteDb.open("test.db")
db2 = db                // Move: ownership передаётся db2
// db уже недействителен, drop для db здесь не вставляется
// ← конец области видимости: drop вставляется только для db2
```

Компилятор отслеживает семантику Move ([RFC-009](../accepted/009-ownership-model.md) §1) и вставляет вызов деструктора только в точке последнего владельца переменной.

#### 7.4 Обработка ошибок деструктора

```llvm
; debug режим: проверяется возвращаемое значение деструктора
%ret = call i32 @sqlite3_close(i8* %handle)
%ok = icmp eq i32 %ret, 0
br i1 %ok, label %done, label %panic
panic:
  call @__yx_panic("destructor failed")
  unreachable
done:
  ret void

; release режим: возвращаемое значение игнорируется
call i32 @sqlite3_close(i8* %handle)
ret void
```

### 8. Структура артефактов компиляции

Артефакты компиляции включают следующие составляющие (конкретные определения struct будут определены на этапе реализации):

- **Машинный код**: целевой файл LLVM (`.o`), содержащий результаты трансляции всех функций
- **Метаданные spawn**: указатели функций задач для каждого spawn блока, отношения зависимостей, пары сериализации по конфликтам ресурсов
- **Таблица FFI-символов**: ссылки на внешние C-символы (имя символа + признак слабой ссылки)
- **Таблица точек входа**: список функций входа в исполняемый файл
- **Информация о типах**: метаданные рефлексии, записываются в секцию `.reflect`, mmap по запросу в runtime

### 9. Runtime-библиотека

Согласуется с [RFC-008 §6.2](../accepted/008-runtime-concurrency-model.md), runtime линкуется в финальный exe в виде **статической библиотеки**.

```
Внутренняя структура финального exe:

┌────────────────────────────────────────────┐
│  Пользовательский код (нативный машинный код)        │
│  ├── Обычные функции (последовательное выполнение)    │
│  ├── Развёрнутые spawn блоки (функции задач + вызовы  │
│  │   планировщика)                                    │
│  ├── Функции-обёртки FFI marshalling                  │
│  └── RAII-код деструкторов                            │
├────────────────────────────────────────────┤
│  Статическая библиотека runtime (около 500KB-1MB,     │
│  в зависимости от платформы и выбора функциональности) │
│  ├── Пул потоков (num_workers)                        │
│  ├── Цикл обработки событий (libuv / io_uring)        │
│  ├── Очереди work-stealing (только Full Runtime)      │
│  ├── Аллокатор памяти (jemalloc / mimalloc)           │
│  └── Метаданные рефлексии (секция .reflect, mmap по   │
│     запросу)                                          │
│                                                        │
│  Отсутствует:                                          │
│  ❌ Байт-код интерпретатор                             │
│  ❌ JIT-компилятор                                      │
│  ❌ GC                                                  │
│  ❌ Виртуальная машина                                  │
└────────────────────────────────────────────┘
```

**Ключевой принцип проектирования**: идентификация задач и анализ зависимостей spawn блоков выполняются в compile-time, в runtime происходит лишь «создание задачи → распределение в пул потоков → ожидание завершения» — структуры данных фиксированы, поведение предсказуемо.

> **Расхождение с оценкой размера из RFC-008**: в RFC-008 §4 размер планировщика оценивается в 200-500KB и включает только ядро диспетчеризации задач. Оценка 500KB-1MB в данном документе дополнительно включает аллокатор памяти (jemalloc/mimalloc), цикл обработки событий (libuv/io_uring) и секцию метаданных рефлексии. Реальный размер зависит от платформы и выбора функциональности, точные цифры будут даны на этапе реализации.

**Три уровня runtime и их связь с LLVM** (согласуется с RFC-008 §1):

| Runtime | Поведение LLVM AOT |
|--------|---------------|
| **Embedded** | Без поддержки spawn, прямой последовательный машинный код |
| **Standard** | Поддержка spawn блоков, DAG внутри spawn блока + однопоточный планировщик (num_workers=1) |
| **Full** | Поддержка spawn блоков, DAG внутри spawn блока + многопоточный планировщик (num_workers>1), поддержка WorkStealing |

```
│
├── backends/
│   ├── common/                        # общие значения / куча / опкоды
│   ├── interpreter/                   # древовидный интерпретатор (VM бэкенд)
│   ├── llvm/                          # [! планируется] генерация кода LLVM бэкенда (см. список файлов ниже)
│   │   ├── mod.rs                     # [! планируется] точка входа LLVM бэкенда
│   │   ├── context.rs                 # [! планируется] управление контекстом LLVM
│   │   ├── types.rs                   # [! планируется] отображение типов (YaoXiang → LLVM IR)
│   │   ├── values.rs                  # [! планируется] отображение значений
│   │   ├── func.rs                    # [! планируется] трансляция функций
│   │   ├── spawn.rs                   # [! планируется] развёртывание spawn блоков
│   │   ├── ffi.rs                     # [! планируется] генерация кода FFI-вызовов
│   │   └── drop.rs                    # [! планируется] вставка деструкторов
│   └── runtime/                       # компилируемый runtime (статическая библиотека, линкуется в exe)
│       ├── engine.rs                  # движок диспетчеризации задач
│       ├── facade.rs                  # внешний интерфейс
│       └── task.rs                    # представление задачи
│
└── util/
    └── diagnostic/                    # диагностика ошибок (общая)
```

> **Ключевое изменение**: анализ spawn блоков (идентификация задач, анализ зависимостей, обнаружение конфликтов ресурсов) будет реализован в `frontend/core/spawn/` (общий фронтенд). Существующий `frontend/core/typecheck/passes/spawn_placement.rs` (проверка расположения spawn) будет перенесён в `frontend/core/spawn/placement.rs`, подробности см. в RFC-024. LLVM бэкенд лишь потребляет результаты анализа, порождая соответствующий код диспетчеризации.
>
> **Текущее состояние**: файлы `middle/passes/codegen/` — `buffer.rs`, `emitter.rs`, `bytecode.rs`, `flow.rs`, `operand.rs` — обслуживают генерацию байт-кода VM бэкенда (`CodegenContext::generate()` → `BytecodeFile`). LLVM бэкенд будет реализован в `backends/llvm/`, на одном уровне с interpreter бэкендом и runtime — оба потребляют один и тот же вход `ModuleIR`, выдавая разные целевые форматы (байт-код или нативный код).

### Поддержка платформенных ABI

| Платформа | Целевой тройной | Выходной формат | Соглашение о вызовах (FFFI по умолчанию) |
|------|-----------|----------|---------------------|
| Linux x86_64 | `x86_64-unknown-linux-gnu` | ELF | System V AMD64 |
| macOS x86_64 | `x86_64-apple-darwin` | Mach-O | System V AMD64 |
| macOS ARM64 | `aarch64-apple-darwin` | Mach-O | ARM64 AAPCS |
| Windows x86_64 | `x86_64-pc-windows-msvc` | COFF | Microsoft x64 |

Для FFI-вызовов по умолчанию используется C-соглашение о вызовах платформы. Пользователь может переопределить его опциями вида `native("symbol", cc = "stdcall")` (согласуется с будущими расширениями [RFC-026](./026-ffi-core-mechanism.md)).

### Консистентность семантики с плавающей точкой (VM ↔ LLVM)

Ключевое обещание архитектуры с двумя бэкендами — идентичность поведения VM (отладочная сборка) и LLVM (продакшн-сборка). Операции с плавающей точкой в обоих режимах исполнения несут потенциальные точки рассогласования:

| Сценарий | Риск | Стратегия |
|------|------|------|
| Распространение NaN | VM и LLVM могут по-разному обрабатывать знак и payload NaN | Компилятор нормализует представление NaN на уровне IR, сравнения NaN унифицированно используют `fcmp uno` |
| Режим округления | LLVM по умолчанию — round-to-nearest-even, VM зависит от CPU хоста | Не экспонировать нестандартные режимы округления, VM и LLVM унифицированно используют RTNE |
| Деление на ноль | IEEE 754 определяет ±Inf, но некоторые платформы могут ловушку | debug-режим проверяет деление на ноль и сообщает диагностику; release-режим следует IEEE 754 |
| `-0.0` против `+0.0` | Операции сравнения могут быть неэквивалентны | Унифицированное использование правил IEEE 754: `+0.0 == -0.0` |
| Денормализованные числа | Некоторые платформы используют flush-to-zero | LLVM не включает атрибут `denormal-fp-math`, сохраняется полная семантика IEEE 754 |

> **Стратегия тестирования**: реализуется кросс-бэкендный набор тестов на консистентность операций с плавающей точкой — один и тот же исходный код YaoXiang выполняется в VM и LLVM бэкендах, выходные значения сравниваются поэлементно. Эта группа тестов является обязательным гейтом CI.

## Компромиссы

### Преимущества

1. **Производительность**: AOT-компиляция в 10-100x быстрее интерпретации
2. **Унифицированный фронтенд**: VM и LLVM используют общий фронтенд, поведение полностью идентично
3. **Нулевые накладные расходы планирования**: обычный код напрямую компилируется в последовательный машинный код, за пределами spawn блоков нет накладных расходов на DAG
4. **Статическая линковка**: нет внешних зависимостей от runtime, для развёртывания достаточно одного exe
5. **Нулевой GC**: детерминированные деструкторы по RAII, без пауз
6. **Нулевые накладные расходы FFI**: оптимизация null-указателя для `?T`, оптимизация раскладки непрозрачного типа, стоимость FFI-вызова эквивалентна C
7. **Анализ в compile-time**: идентификация задач и анализ зависимостей spawn блоков выполняются в compile-time, в runtime остаётся только исполнение

### Недостатки

1. **Сложность интеграции с LLVM**: требуется глубокое понимание inkwell API и LLVM IR
2. **Время компиляции**: AOT-компиляция медленнее интерпретатора (одноразовая цена)
3. **Опыт отладки**: отладка нативного кода требует поддержки символов DWARF/PDB (компилятор должен генерировать отладочную информацию)
4. **Инкрементальная компиляция**: для крупных проектов требуется дополнительное проектирование инкрементальной компиляции
5. **Консистентность семантики с плавающей точкой**: VM и LLVM могут различаться в граничных случаях (распространение NaN, режим округления, деление на ноль) — для обеспечения идентичного поведения двух бэкендов применяются стратегии нормализации (см. §10)

### Согласованность со связанными RFC

| RFC | Согласованность |
|-----|--------|
| RFC-024 модель конкурентности spawn блоков | ✅ прямые подвыражения spawn блока → диспетчеризация задач |
| RFC-008 архитектура runtime | ✅ два бэкенда + статическая библиотека планировщика + структура каталогов |
| RFC-009 модель ownership v9 | ✅ токены `&T`/`&mut T` (нулевой размер), `ref T` (толстый указатель), `?T` (Option) |
| RFC-026 основной механизм FFI | ✅ `native()` → declare + marshalling, `.drop` → RAII cleanup |

## Альтернативы

| Подход | Описание | Почему не выбран |
|------|------|------|
| Только интерпретатор | AOT не нужен | Недостаточная производительность |
| Чисто статическая компиляция (без runtime) | Планировщик не линкуется | Spawn блокам требуется диспетчеризация задач в runtime |
| Cranelift в качестве бэкенда | Более быстрая компиляция | Производительность в runtime уступает LLVM, может рассматриваться как будущий опциональный бэкенд |
| Линковка с внешним LLVM runtime | Использование встроенного runtime LLVM | Вводит ненужные зависимости |

## Связанные работы

### Lazy Task Creation (1990)[^1]

| Атрибут | Описание |
|------|------|
| Организация | MIT |
| Авторы | James R. Larus, Robert H. Halstead Jr. |
| Ядро | Отложенное создание подзадач, создание по требованию |
| Ценность для референса | Теоретическая база для диспетчеризации задач в spawn блоках по требованию |

**Ключевая идея**: вместо немедленного создания задачи создание откладывается. Когда родительской задаче требуется значение подзадачи, подзадача только тогда и создаётся. Это решает проблему накладных расходов мелкозернистых параллельных задач[^1]. Планирование spawn блоков в YaoXiang перенимает эту идею — задачи идентифицируются в compile-time, но в runtime распределяются в пул потоков по требованию.

### Lazy Scheduling (2014)[^2]

| Атрибут | Описание |
|------|------|
| Организация | University of Maryland |
| Авторы | Tzannes, Caragea |
| Ядро | Адаптивное планирование в runtime, без дополнительного состояния |
| Ценность для референса | Референс для дизайна планировщика с WorkStealing в Full Runtime |

### Язык SISAL[^3]

| Атрибут | Описание |
|------|------|
| Организация | Lawrence Livermore National Laboratory (LLNL) |
| Ядро | Язык с единичным присваиванием, dataflow-граф, неявный параллелизм |
| Ценность для референса | Доказательство жизнеспособности модели dataflow в промышленном применении |

**Ключевое отличие**: параллелизм в SISAL — **неявный** — язык имеет семантику единичного присваивания, и компилятор автоматически анализирует граф зависимостей по всей программе для определения параллелизма. Параллелизм в YaoXiang — **явный** — пользователь размечает параллельные области блоками `spawn {}`, компилятор анализирует зависимости только внутри spawn блока. Это избавляет от сложности полнопрограммного анализа SISAL, сохраняя контроль пользователя над параллельным поведением.

### Mul-T параллельный Scheme[^4]

| Атрибут | Описание |
|------|------|
| Организация | MIT |
| Ядро | Конструкция Future, реализация Lazy Task Creation |
| Ценность для референса | Референс по конкретной реализации |

### Сводное сравнение

| Технология | Отложенное создание | Маркировка параллелизма | Область анализа | Ownership |
|------|----------|----------|----------|--------|
| Lazy Task Creation[^1] | ✅ | Неявная | Вся программа | N/A |
| Lazy Scheduling[^2] | ✅ | Неявная | Вся программа | N/A |
| SISAL[^3] | ✅ | Неявная (единичное присваивание) | Вся программа | N/A |
| Mul-T[^4] | ✅ | Явная (future) | Точка вызова | N/A |
| **YaoXiang** | ✅ | **Явная (spawn блок)** | **Внутри spawn блока** | **✅ (Move + токены + ref)** |

**Инновация YaoXiang**: маркировка параллелизма повышена с уровня «каждый вызов функции» (future) до уровня «структурный блок» (spawn). Пользователь пишет обычный код, помещая spawn блок в местах, где нужен параллелизм. Область анализа ограничена spawn блоком — компиляция эффективна, а поведение управляемо.

### Приложение А: Сравнение с async в Rust

| Характеристика | Rust async | YaoXiang LLVM AOT |
|------|-----------|-------------------|
| Артефакт компиляции | Конечный автомат + машинный код | Машинный код + метаданные задач spawn |
| Runtime | tokio | Статически слинкованный планировщик (около 500KB-1MB) |
| Маркировка конкурентности | ключевые слова async/await | блок `spawn { }` |

| Цветные функции | распространение async | **без раскраски функций** |
| Синхронное ожидание | `.await` | spawn блок автоматически синхронно блокирует |
| Управление памятью | GC (в runtime) | **RAII (детерминированное)** |
| Механизм разделения | `Arc::new()` + ручной Weak | **ключевое слово `ref` (компилятор автоматически выбирает Rc/Arc)** |
