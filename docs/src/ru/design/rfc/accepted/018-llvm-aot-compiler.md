## Аннотация

Данный документ описывает дизайн LLVM AOT (Ahead-of-Time) компилятора для языка YaoXiang. Бэкенд LLVM и бэкенд VM (интерпретатор) совместно используют единый компиляторный фронтенд, образуя архитектуру с двумя бэкендами, определённую в [RFC-008](../accepted/008-runtime-concurrency-model.md): VM используется для разработки и отладки, LLVM — для промышленных релизов.

**Основные обязанности**:

```
Исходный код → Фронтенд (общий) → IR → LLVM Codegen → .o → Линковка статической библиотеки планировщика → exe
```

Компилятор транслирует исходный код YaoXiang в нативный машинный код, где:

| Языковая возможность | Стратегия компиляции |
|----------|----------|
| Обычный код | Последовательный машинный код, нулевые накладные расходы на планирование |
| `spawn { }`-блок | Прямые подвыражения → распределение задач + синхронное ожидание (согласовано с [RFC-024](../accepted/024-concurrency-model.md)) |
| `native("symbol")` | LLVM `declare external` + marshalling аргументов (согласовано с [RFC-026](./026-ffi-core-mechanism.md)) |
| Деструктор `.drop` | Вставка кода RAII cleanup (согласовано с [RFC-009](../accepted/009-ownership-model.md)) |
| Токены `&T` / `&mut T` | Типы нулевого размера, исчезают после компиляции |
| Разделяемое владение `ref T` | «Толстый» указатель `{ refcount_ptr, data_ptr }`, компилятор автоматически выбирает Rc/Arc |

**Связь с RFC-024**: RFC-024 определяет **пользовательскую семантику** spawn-блоков (создание задач через прямые подвыражения, синхронное блокирующее ожидание). Данный документ определяет, как эта семантика **транслируется в машинный код**.

**Связь с RFC-026**: RFC-026 определяет **пользовательский синтаксис** FFI (`native()`, привязка методов через `[0]`, `.drop`). Данный документ определяет, как FFI-вызовы **генерируют LLVM IR**.

## Мотивация

### Зачем нужен LLVM AOT компилятор?

В настоящее время YaoXiang имеет только интерпретатор в качестве исполнительного бэкенда:

| Проблема | Влияние |
|------|------|
| Узкое место производительности | Интерпретация в 10-100 раз медленнее машинного кода |
| Сложность развёртывания | Необходимо поставлять интерпретатор и runtime |
| Продакшн-среда | Интерпретатор не подходит для сценариев, чувствительных к производительности |

### LLVM в модели с двумя бэкендами

[RFC-008](../accepted/008-runtime-concurrency-model.md) §6 определяет архитектуру с двумя бэкендами:

```
                    ┌─────────────────────┐
                    │  Фронтенд (единый)    │
                    │  Lexer → Parser      │
                    │  → TypeCheck         │
                    │  → spawn-анализ      │
                    │  → escape-анализ     │
                    └──────────┬──────────┘
                               │
                  ┌────────────┴────────────┐
                  ▼                         ▼
      ┌───────────────────┐     ┌───────────────────┐
      │  Бэкенд VM (dev)   │     │ Бэкенд LLVM (prod) │
      │  IR → Интерпретация │     │  IR → Нативный код │
      │  Пошаговая отладка  │     │  Линковка стат.    │
      │  Быстрая итерация   │     │  библиотеки        │
      │                     │     │  Вывод .exe        │
      └───────────────────┘     └───────────────────┘
```

Поведение обоих бэкендов **полностью идентично** — разница только в способе исполнения. Один и тот же исходный код, одна и та же проверка типов, один и тот же результат spawn-анализа.

## Предложение

### 1. Архитектура компилятора

Бэкенд LLVM располагается на последней стадии конвейера компиляции, принимает IR от фронтенда и генерирует нативный код:

```
Исходный код
  → Lexer / Parser (frontend/core/)
  → TypeCheck + spawn-анализ (frontend/core/typecheck/)
  → Генерация IR (middle/core/ir_gen.rs)
  → LLVM Codegen (backends/llvm/)
      ├── Маппинг типов: типы YaoXiang → типы LLVM IR
      ├── Трансляция функций: инструкции IR → инструкции LLVM IR
      ├── Развёртывание spawn: прямые подвыражения → функция задачи + вызов планировщика
      ├── Развёртывание FFI: вызовы native() → declare + marshalling
      └── Вставка деструкторов: конец области видимости → вызов .drop()
  → Оптимизации LLVM + генерация целевого кода
  → Линковка статической библиотеки runtime → исполняемый файл
```

### 2. Процесс компиляции

```
Фаза 1: Фронтенд (общий с бэкендом VM)
  - Парсинг, проверка типов, анализ spawn-блоков, escape-анализ
  - Выход: IR с аннотациями типов

Фаза 2: Генерация LLVM IR
  - Маппинг типов, объявления функций, трансляция инструкций
  - Выход: LLVM Module

Фаза 3: Оптимизации LLVM
  - Стандартный конвейер оптимизаций LLVM (O0/O1/O2/O3)
  - Инлайнинг, свёртка констант, удаление мёртвого кода

Фаза 4: Генерация целевого кода
  - LLVM TargetMachine → файл .o
  - Платформы: Linux (ELF), macOS (Mach-O), Windows (COFF)

Фаза 5: Линковка
  - Линковка статической библиотеки runtime (планировщик, аллокатор)
  - Выход: исполняемый файл
```

### 3. Маппинг типов

#### 3.1 Маппинг типов YaoXiang → LLVM IR

| Тип YaoXiang | Тип LLVM IR | Пояснение |
|---------------|-------------|------|
| `Int` | `i64` | По умолчанию 64-битное знаковое целое |
| `Int32` | `i32` | Явное 32-битное целое (в основном для FFI) |
| `Float` | `f64` | По умолчанию 64-битное число с плавающей точкой |
| `Float32` | `f32` | Явное 32-битное число с плавающей точкой (в основном для FFI) |
| `Bool` | `i1` | Логическое значение |
| `Char` | `i32` | Unicode-кодпоинт |
| `String` | `{ i8*, i64 }` | Указатель + длина в байтах |
| `Void` | `{}` | Пустой тип нулевого размера |
| `&T` | — | Токен нулевого размера, исчезает после компиляции, не порождает IR |
| `&mut T` | — | Токен нулевого размера, исчезает после компиляции, не порождает IR |
| `ref T` | `{ i64*, T* }` | «Толстый» указатель (указатель счётчика ссылок + указатель на данные) |
| `*T` | `T*` | Голый указатель |
| `[T; N]` | `[N x T]` | Массив фиксированной длины |
| `List(T)` | `{ T*, i64, i64 }` | Указатель на данные + длина + ёмкость |
| Структура | Соответствующий struct в LLVM | Поля упорядочены по определению |
| Регистрируемый enum | `{ i64, [max_payload_size] }` | Тег + union максимального payload |
| `?T` | `{ i1, T }` | Флаг наличия значения + данные (универсальное представление) |
| Непрозрачный FFI-тип | `{ i8* }` | Обёртка для C-указателя |
| Указатель на функцию | `T (...)*` | Тип указателя на функцию |

> **Нулевые накладные расходы `&T` / `&mut T`**: [RFC-009](../accepted/009-ownership-model.md) §2.7 определяет, что компилятор внутренне назначает токенам уникальный идентификатор бренда (уникальное целое в компайл-тайме), и после мономорфизации и инлайнинга бренд полностью исчезает — в сгенерированном машинном коде не остаётся никаких следов токенов.

### 4. Нормализация IR и трансляция инструкций

#### 4.0 Нормализация IR (стек → регистры)

Текущий IR (`src/middle/core/ir.rs`) содержит стековые инструкции (`Push`/`Pop`/`Dup`/`Swap`), спроектированные для байткодовой VM. LLVM IR имеет SSA-форму и не принимает стековые операции.

**Стратегия обработки**: путь LLVM перед трансляцией инструкций проходит через облегчённый нормализационный проход:

| Стековая инструкция | Стратегия нормализации |
|--------|-----------|
| `Push(r)` | Запись `stack.push(r)`, не порождает IR |
| `Pop(r)` | `r = stack.pop()`, порождает `load` (из слота стека) |
| `Dup` | `stack.push(stack.top())`, не порождает IR |
| `Swap` | Обмен двух верхних элементов стека, не порождает IR |

После нормализации все операнды становятся ссылками на регистры/локальные переменные, стековые операции полностью устраняются. Данный проход выполняется как первый шаг `translator.rs`.

> **Почему не устранить стековые инструкции на уровне IR?** Потому что бэкенду VM нужна стековая семантика. Нормализация на входе в LLVM-трансляцию сохраняет общий IR для обоих бэкендов — каждый бэкенд потребляет один и тот же IR по своим потребностям.
>
> **Предусловие**: стадия генерации IR гарантирует баланс стека — все пути потока управления, достигающие одной и той же программной точки, имеют одинаковую глубину стека (от этого же предусловия зависит байткодовый бэкенд VM, иначе исполнение байткода будет некорректным). Нормализационный проход не проверяет данное предусловие; при его нарушении бэкенд LLVM порождает неопределённое поведение.

| `UnsafeBlockEnd` | — | **Метка компайл-тайма, не порождает IR** |
| `PtrFromRef { dst, src }` | `%p = ptrtoint T* %src to i64` (или прямое копирование указателя) | |
| `PtrDeref { dst, src }` | `%v = load T, T* %src` | |
| `PtrStore { dst, src }` | `store T %src, T* %dst` | |
| `PtrLoad { dst, src }` | `%v = load T, T* %src` | |

**Строковые инструкции**:

| IR-инструкция | LLVM IR | Пояснение |
|---------|---------|------|
| `StringLength { dst, src }` | `%len = extractvalue { i8*, i64 } %src, 1` | String — это `{ ptr, len }`, длина в поле 1 |
| `StringConcat { dst, lhs, rhs }` | `call String @yx_string_concat(%lhs, %rhs)` | Вспомогательная функция runtime |
| `StringGetChar { dst, src, index }` | `getelementptr` + `load i32` | С проверкой границ |
| `StringFromInt { dst, src }` | `call String @yx_string_from_int(%src)` | Вспомогательная функция runtime |
| `StringFromFloat { dst, src }` | `call String @yx_string_from_f64(%src)` | Вспомогательная функция runtime |

**Инструкции замыканий**:

| IR-инструкция | LLVM IR | Пояснение |
|---------|---------|------|
| `MakeClosure { dst, func: String, env }` | Аллокация структуры замыкания + заполнение указателя на функцию (по имени функции) и среды | `{ fn_ptr, env_fields... }` |
| `LoadUpvalue { dst, upvalue_idx }` | `%v = extractvalue %env, upvalue_idx` | Чтение upvalue из окружения замыкания |
| `StoreUpvalue { src, upvalue_idx }` | `%env = insertvalue %env, %src, upvalue_idx` | Запись upvalue в окружение замыкания |
| `CloseUpvalue(val)` | Копирование стекового upvalue в кучу | |

**Прочие инструкции**:

| IR-инструкция | LLVM IR | Пояснение |
|---------|---------|------|
| `HeapAlloc { dst, type_id }` | `call i8* @malloc(i64 size)` + запись тега типа | Аллокация в куче + информация о типе |
| `NewDict { dst, keys, values }` | `call Dict @yx_dict_new(%keys, %values)` | Вспомогательная функция runtime |

> **Примечание**: `Push`/`Pop`/`Dup`/`Swap` уже устранены на стадии нормализации (§4.0) и не появляются в таблице трансляции. `Borrow`/`Release` — токены нулевого размера в компайл-тайме, не порождают никакого машинного кода.

### 5. Генерация кода для spawn-блоков

Согласовано с [RFC-024](../accepted/024-concurrency-model.md), компиляция spawn-блоков состоит из следующих шагов.

#### 5.1 Повторение семантики

```yaoxiang
(r1, r2) = spawn {
    t1 = fetch("url1"),   // прямое подвыражение → задача 1
    t2 = fetch("url2"),   // прямое подвыражение → задача 2
    return (t1, t2)       // синхронное ожидание, сборка результата
}
```

**Правила** (RFC-024 §2.1):
- **Прямые подвыражения** spawn-блока (инструкции верхнего уровня, разделённые запятыми) порождают параллельные задачи
- Выражения внутри вложенных `{}` не являются прямыми подвыражениями и не становятся отдельными задачами
- Весь spawn-блок синхронно блокирует, ожидая завершения всех задач перед возвратом

#### 5.2 Шаги компиляции

```
Шаг 1: Идентификация прямых подвыражений
  Обход тела spawn-блока, сбор инструкций верхнего уровня

Шаг 2: Анализ зависимостей
  Для каждого прямого подвыражения анализ, какие переменные, порождённые
  предыдущими задачами, оно использует
  Без зависимостей → немедленное параллельное планирование
  С зависимостями → постановка в очередь с ожиданием зависимых задач

Шаг 3: Обнаружение конфликтов ресурсов (RFC-024 §2.5)
  Проверка, используется ли один и тот же экземпляр типа-ресурса
  несколькими задачами
  Конфликт по одному экземпляру → пометка последовательным порядком исполнения

Шаг 4: Генерация функций задач
  Каждое прямое подвыражение порождает отдельную функцию LLVM (замыкание)

Шаг 5: Генерация кода планирования
  Вызов task_spawn / task_wait планировщика runtime

Шаг 6: Сборка результатов
  Сбор выходных данных всех задач, формирование возвращаемого кортежа
```

#### 5.3 Шаблон генерации LLVM IR

```llvm
; Вход в spawn-блок
%task_count = 2
%tasks = alloca [2 x %TaskHandle]

; Создание задачи 1: fetch("url1")
%task1_fn = @spawn_closure_1
call @runtime_task_spawn(%tasks[0], %task1_fn, ...)

; Создание задачи 2: fetch("url2")
%task2_fn = @spawn_closure_2
call @runtime_task_spawn(%tasks[1], %task2_fn, ...)

; Синхронное ожидание всех задач
call @runtime_task_wait_all(%tasks, %task_count)

; Сборка возвращаемого значения
%r1 = call @runtime_task_result(%tasks[0])
%r2 = call @runtime_task_result(%tasks[1])
ret { %r1, %r2 }
```

#### 5.4 Зависимые задачи

```yaoxiang
result = spawn {
    data = fetch("url"),       // задача 1: без зависимостей
    processed = parse(data),   // задача 2: зависит от data из задачи 1
    return processed
}
```

Компилятор обнаруживает, что `parse(data)` ссылается на `data`, порождённое задачей 1, и при генерации кода планирования помечает зависимость:

```llvm
; Задача 2 создаётся с зависимостью от задачи 1
call @runtime_task_spawn_with_dep(%tasks[1], %task2_fn, %tasks[0])
;                                                              ↑
;                                                 зависимость от задачи 0 (fetch)
```

#### 5.5 Автоматическая сериализация типов-ресурсов

Типы-ресурсы, определённые в [RFC-024 §2.5](../accepted/024-concurrency-model.md) (`FilePath`, `HttpUrl`, `DBUrl`, `Console` и пользовательские типы-ресурсы), автоматически сериализуются в spawn-блоках:

```yaoxiang
(a, b) = spawn {
    r1 = db.exec("SELECT ..."),   // использует SqliteDb (тип-ресурс)
    r2 = db.exec("INSERT ...")    // тот же экземпляр → автоматическая сериализация
}
```

Компилятор обнаруживает, что один и тот же экземпляр ресурса используется двумя задачами, и порождает сериализующую зависимость:

```llvm
; Задача 2 зависит от задачи 1 (автосериализация для того же ресурса)
call @runtime_task_spawn_with_dep(%tasks[1], %task2_fn, %tasks[0])
```

#### 5.6 spawn for — параллелизм по данным

```yaoxiang
results = spawn for item in items {
    process(item)
}
```

Компилятор разворачивает в N независимых задач (N = длина items), с ограничением на максимальную конкурентность.

### 6. Генерация FFI-кода

> ⚠️ **Замечание о зависимостях**: определённая в данном разделе **архитектура** генерации FFI-кода (`native("x")` → `declare external @x` → функция-обёртка с marshalling → call) стабильна и не меняется при изменении синтаксиса RFC-026. Конкретные таблицы правил marshalling аргументов (§6.2) и раскладки непрозрачных типов (§6.3) ссылаются на определения RFC-026 — если синтаксис `native()` или правила marshalling в RFC-026 изменятся, достаточно обновить соответствующие таблицы маппинга в данном документе, архитектурный уровень не пострадает. Текущий статус RFC-026: **на рассмотрении**, находится в каталоге `review/` вместе с данным документом.
>
> **Предусловие принятия**: до принятия данного RFC, части RFC-026, относящиеся к §6 данного документа (синтаксис объявления `native()`, правила marshalling аргументов, раскладка непрозрачного типа `{ i8* }`, соглашение о привязке `.drop`), должны быть заморожены или приняты вместе с 026. Иначе таблицы маппинга в §6.2/§6.3/§7 могут устареть ещё до реализации.

Согласовано с [RFC-026](./026-ffi-core-mechanism.md), данный раздел определяет стратегию генерации LLVM IR для FFI-вызовов.

#### 6.1 Объявление функций native()

```yaoxiang
sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")
```

Транслируется в LLVM IR:

```llvm
; Объявление внешней C-функции
declare i8* @sqlite3_open(i8*)

; Функция-обёртка YaoXiang (выполняет marshalling)
define { i8* } @__yx_sqlite3_open({ i8*, i64 } %filename) {
    ; marshalling: YaoXiang String → C-строка
    %c_str = extractvalue { i8*, i64 } %filename, 0
    ; Вызов C-функции
    %raw = call i8* @sqlite3_open(i8* %c_str)
    ; unmarshalling: C-указатель → непрозрачный тип
    %result = insertvalue { i8* } undef, i8* %raw, 0
    ret { i8* } %result
}
```

**Ключевые моменты**:
- `native("sqlite3_open")` → `declare external @sqlite3_open`
- Компилятор автоматически генерирует функцию-обёртку с marshalling
- Сигнатура обёртки использует типы YaoXiang, внутри — преобразование к C-типам

#### 6.2 Marshalling параметров

| Направление | Преобразование |
|------|------|
| YaoXiang `String` → C `char*` | Извлечение поля `.ptr` и передача |
| YaoXiang `Int32` → C `int` | Прямая передача (`i32`) |
| YaoXiang `*Void` → C `void*` | Прямая передача (`i8*`) |
| YaoXiang `T` (прозрачный тип) → C `struct T*` | Передача адреса |
| YaoXiang `T` (непрозрачный тип) → C `struct T*` | Извлечение указателя из `{ i8* }` и передача |

#### 6.3 Раскладка непрозрачных типов в LLVM

Непрозрачный тип, определённый в [RFC-026](./026-ffi-core-mechanism.md) §4.1:

```yaoxiang
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void
    }
    return SqliteDb
}
```

Раскладка в LLVM: `{ i8* }` — структура, содержащая C-указатель.

**Оптимизация раскладки**: когда непрозрачный тип имеет единственное поле `handle: *Void`, его можно оптимизировать до прямого использования `i8*` (без оборачивающей struct). Оптимизированный ABI полностью совпадает с C-указателем, нулевые накладные расходы на marshalling. Данная оптимизация включена по умолчанию и незаметна для пользователя.

#### 6.4 LLVM-представление обнуляемых возвращаемых значений `?T`

FFI-обнуляемые возвращаемые значения, определённые в [RFC-026](./026-ffi-core-mechanism.md) §7.6:

```yaoxiang
sqlite3_open: (filename: String) -> ?SqliteDb = native("sqlite3_open")
```

Универсальное представление в LLVM: `{ i1, { i8* } }` — флаг наличия значения + данные.

**Оптимизация для FFI null-указателя**: если `T` в `?T` — непрозрачный тип (внутри указатель), компилятор использует **null-указатель = None** оптимизацию:

```llvm
; Оптимизированное представление в LLVM: прямое использование обнуляемого указателя
define i8* @__yx_sqlite3_open(...) {
    %raw = call i8* @sqlite3_open(...)
    ; null → None, не null → Some (оборачивается как непрозрачный тип)
    ret i8* %raw
}
```

Сторона вызова:
```llvm
%raw = call i8* @__yx_sqlite3_open(...)
%is_null = icmp eq i8* %raw, null
br i1 %is_null, label %none_branch, label %some_branch
```

Данная оптимизация делает FFI-вызов `?SqliteDb` **полностью без дополнительных накладных расходов** — эквивалентен проверке null в C.

#### 6.5 Интеграция с yx-bindgen

Файлы привязок, автоматически генерируемые [yx-bindgen](./026-ffi-core-mechanism.md) §6, обрабатываются при компиляции как обычный исходный код YaoXiang. Компилятору не нужно знать, что код получен из bindgen — обработка объявлений `native()` и определений типов `unsafe {}` полностью одинакова.

### 7. Генерация кода деструкторов

Согласовано с семантикой RAII из [RFC-009](../accepted/009-ownership-model.md) и соглашением `.drop` из [RFC-026](./026-ffi-core-mechanism.md) §7.

#### 7.1 Идентификация привязки .drop

```yaoxiang
SqliteDb.drop = sqlite3_close[0]
```

Компилятор распознаёт привязку `.drop` и помечает указатель на деструктор в метаданных типа.

#### 7.2 Вставка Cleanup в конце области видимости

```
Код пользователя:
{
    db = SqliteDb.open("test.db")
    stmt = db.prepare("SELECT ...")
    stmt.step()
    // ← конец области видимости
}

Cleanup, вставленный компилятором (в обратном порядке):
    call @sqlite3_finalize(%stmt)    // stmt.drop()
    call @sqlite3_close(%db)          // db.drop()
```

**Точки вставки**:
- Нормальное завершение области видимости (`}`)
- Досрочный возврат (перед `return`)
- Пути распространения ошибок `?` (перед `?`)
- Конец spawn-блока (деструкторы переменных внутри задачи)

#### 7.3 Move и деструкторы

```yaoxiang
db = SqliteDb.open("test.db")
db2 = db                // Move: владение передаётся в db2
// db больше не валиден, drop для db не вставляется
// ← конец области видимости: drop вставляется только для db2
```

Компилятор отслеживает семантику Move ([RFC-009](../accepted/009-ownership-model.md) §1) и вставляет вызов деструктора только у финального владельца переменной.

#### 7.4 Обработка ошибок деструктора

```llvm
; debug-режим: проверка возвращаемого значения деструктора
%ret = call i32 @sqlite3_close(i8* %handle)
%ok = icmp eq i32 %ret, 0
br i1 %ok, label %done, label %panic
panic:
  call @__yx_panic("destructor failed")
  unreachable
done:
  ret void

; release-режим: игнорирование возвращаемого значения
call i32 @sqlite3_close(i8* %handle)
ret void
```

### 8. Структура артефактов компиляции

Артефакт компиляции состоит из следующих компонентов (конкретные определения struct уточняются на этапе реализации):

- **Машинный код**: целевой файл LLVM (`.o`), содержащий результаты трансляции всех функций
- **Метаданные spawn**: указатели на функции задач, отношения зависимостей, пары сериализации конфликтов ресурсов для каждого spawn-блока
- **Таблица FFI-символов**: ссылки на внешние C-символы (имя символа + признак слабой ссылки)
- **Таблица точек входа**: список функций входа исполняемого файла
- **Информация о типах**: метаданные рефлексии, записываемые в секцию `.reflect`, по необходимости mmap в runtime

### 9. Библиотека runtime

Согласовано с [RFC-008 §6.2](../accepted/008-runtime-concurrency-model.md), runtime линкуется в виде **статической библиотеки** в финальный exe.

```
Внутренняя структура финального exe:

┌────────────────────────────────────────────┐
│  Код пользователя (нативный машинный код)    │
│  ├── Обычные функции (последовательное исп.) │
│  ├── Развёртывание spawn-блоков             │
│  │   (функции задач + вызовы планировщика)  │
│  ├── Функции-обёртки FFI с marshalling      │
│  └── Код деструкторов RAII                 │
├────────────────────────────────────────────┤
│  Статическая библиотека runtime             │
│  (около 500KB-1MB, зависит от платформы      │
│   и выбранного набора функций)              │
│  ├── Пул потоков (num_workers)              │
│  ├── Цикл событий (libuv / io_uring)        │
│  ├── Очереди WorkStealing (только Full)     │
│  ├── Аллокатор памяти (jemalloc / mimalloc) │
│  └── Метаданные рефлексии (секция .reflect,  │
│      mmap по необходимости)                 │
│                                              │
│  Отсутствует:                               │
│  ❌ Байткодовый интерпретатор                │
│  ❌ JIT-компилятор                           │
│  ❌ GC                                       │
│  ❌ Виртуальная машина                       │
└────────────────────────────────────────────┘
```

**Ключевой дизайн**: идентификация задач и анализ зависимостей в spawn-блоках завершаются в компайл-тайме, runtime выполняет только «создание задачи → распределение в пул потоков → ожидание завершения» — структуры данных фиксированы, поведение предсказуемо.

> **Различие с оценкой размера в RFC-008**: RFC-008 §4 оценивает планировщик в 200-500KB, только ядро диспетчеризации задач. Оценка 500KB-1MB в данном документе дополнительно включает аллокатор памяти (jemalloc/mimalloc), цикл событий (libuv/io_uring) и секцию метаданных рефлексии. Фактический размер зависит от платформы и выбранного набора функций, точные цифры будут даны на этапе реализации.

**Отношение трёх уровней runtime к LLVM** (согласовано с RFC-008 §1):

| Runtime | Поведение LLVM AOT |
|--------|---------------|
| **Embedded** | Без поддержки spawn, прямой последовательный машинный код |
| **Standard** | Поддержка spawn-блоков, DAG внутри блока + однопоточное планирование (num_workers=1) |
| **Full** | Поддержка spawn-блоков, DAG внутри блока + многопоточное планирование (num_workers>1), поддержка WorkStealing |

```
│       │   ├── buffer.rs              # Пул констант + буфер байткода (для VM)

│       │   ├── flow.rs                # Распределение регистров + генерация меток + таблица символов (для VM)
│       │   └── operand.rs             # Разбор операндов (для VM)
│       ├── lifetime/                  # Анализ времени жизни / активности токенов
│       └── mono/                      # Мономорфизация
│
├── backends/
│   ├── common/                        # Общие значения/куча/опкоды
│   ├── interpreter/                   # Древовидный интерпретатор (бэкенд VM)
│   ├── llvm/                          # [! планируется] Генерация кода для бэкенда LLVM (см. список файлов ниже)
│   │   ├── mod.rs                     # [! планируется] Точка входа бэкенда LLVM
│   │   ├── context.rs                 # [! планируется] Управление контекстом LLVM
│   │   ├── types.rs                   # [! планируется] Маппинг типов (YaoXiang → LLVM IR)
│   │   ├── values.rs                  # [! планируется] Маппинг значений
│   │   ├── func.rs                    # [! планируется] Трансляция функций
│   │   ├── spawn.rs                   # [! планируется] Развёртывание spawn-блоков
│   │   ├── ffi.rs                     # [! планируется] Генерация кода FFI-вызовов
│   │   └── drop.rs                    # [! планируется] Вставка деструкторов
│   └── runtime/                       # Компилируемый runtime (статическая библиотека, линкуется в exe)
│       ├── engine.rs                  # Движок планирования задач
│       ├── facade.rs                  # Внешний интерфейс
│       └── task.rs                    # Представление задачи
│
└── util/
    └── diagnostic/                    # Диагностика ошибок (общая)
```

> **Ключевое изменение**: анализ spawn-блоков (идентификация задач, анализ зависимостей, обнаружение конфликтов ресурсов) будет реализован в `frontend/core/spawn/` (общий для фронтендов). Существующий `frontend/core/typecheck/passes/spawn_placement.rs` (проверка мест размещения spawn) будет перенесён в `frontend/core/spawn/placement.rs`, подробности см. в RFC-024. Бэкенд LLVM только потребляет результаты анализа и генерирует соответствующий код планирования.
>
> **Текущее состояние**: существующие `buffer.rs`, `emitter.rs`, `bytecode.rs`, `flow.rs`, `operand.rs` в `middle/passes/codegen/` обслуживают генерацию байткода для бэкенда VM (`CodegenContext::generate()` → `BytecodeFile`). Бэкенд LLVM будет реализован в `backends/llvm/`, на одном уровне с бэкендом interpreter и runtime — оба потребляют один и тот же вход `ModuleIR` и выводят разные целевые форматы (байткод vs нативный код).

### Поддержка платформенных ABI

| Платформа | Целевой триплет | Формат вывода | Соглашение о вызовах (по умолчанию для FFI) |
|------|-----------|----------|---------------------|
| Linux x86_64 | `x86_64-unknown-linux-gnu` | ELF | System V AMD64 |
| macOS x86_64 | `x86_64-apple-darwin` | Mach-O | System V AMD64 |
| macOS ARM64 | `aarch64-apple-darwin` | Mach-O | ARM64 AAPCS |
| Windows x86_64 | `x86_64-pc-windows-msvc` | COFF | Microsoft x64 |

FFI-вызовы по умолчанию используют платформенное C-соглашение о вызовах. Пользователь может переопределить через `native("symbol", cc = "stdcall")` и т.п. (согласовано с будущими расширениями [RFC-026](./026-ffi-core-mechanism.md)).

### Согласованность семантики чисел с плавающей точкой (VM ↔ LLVM)

Ключевое обещание архитектуры с двумя бэкендами — идентичность поведения VM (для разработки и отладки) и LLVM (для промышленных релизов). Операции с плавающей точкой в двух режимах исполнения потенциально могут иметь несоответствия:

| Сценарий | Риск | Стратегия |
|------|------|------|
| Распространение NaN | VM и LLVM могут по-разному обрабатывать знаковый бит и payload NaN | Компилятор нормализует представление NaN на уровне IR, для сравнений NaN единообразно используется `fcmp uno` |
| Режим округления | LLVM по умолчанию round-to-nearest-even, VM зависит от CPU хоста | Не экспонируются режимы округления, отличные от дефолтного, VM и LLVM единообразно используют RTNE |
| Деление на ноль | IEEE 754 определяет ±Inf, но некоторые платформы могут генерировать trap | В debug-режиме проверяется деление на ноль с выдачей диагностики; в release-режиме следование IEEE 754 |
| `-0.0` vs `+0.0` | Операции сравнения могут быть неэквивалентны | Единообразно используются правила IEEE 754: `+0.0 == -0.0` |
| Денормализованные числа | Некоторые платформы используют flush-to-zero | LLVM не включает атрибут `denormal-fp-math`, сохраняется полная семантика IEEE 754 |

> **Стратегия тестирования**: реализация кросс-бэкендового набора тестов на согласованность плавающей точки — один и тот же исходный код YaoXiang исполняется на бэкендах VM и LLVM с побайтовым сравнением выходов. Данный набор тестов является обязательным шлюзом CI.

## Компромиссы

### Преимущества

1. **Производительность**: AOT-компиляция в 10-100 раз быстрее интерпретации
2. **Единый фронтенд**: VM и LLVM используют один фронтенд, поведение полностью идентично
3. **Нулевые накладные расходы планирования**: обычный код напрямую компилируется в последовательный машинный код, за пределами spawn-блоков нет накладных расходов на DAG
4. **Статическая линковка**: нет внешних зависимостей от runtime, достаточно одного exe для развёртывания
5. **Нулевой GC**: детерминированные деструкторы RAII, без пауз
6. **Нулевые накладные расходы FFI**: оптимизация null-указателя для `?T`, оптимизация раскладки непрозрачного типа — стоимость FFI-вызова эквивалентна C
7. **Анализ в компайл-тайме**: идентификация задач и анализ зависимостей в spawn-блоках завершаются в компайл-тайме, runtime только исполняет

### Недостатки

1. **Сложность интеграции с LLVM**: требуется глубокое понимание inkwell API и LLVM IR
2. **Время компиляции**: AOT-компиляция медленнее интерпретатора (разовая цена)
3. **Опыт отладки**: отладка нативного кода требует поддержки символов DWARF/PDB (компилятор должен генерировать отладочную информацию)
4. **Инкрементальная компиляция**: для крупных проектов требуется дополнительный дизайн инкрементальной компиляции
5. **Согласованность семантики плавающей точки**: VM и LLVM могут различаться в граничных случаях (распространение NaN, режимы округления, деление на ноль и т.д.) — требуется стратегия нормализации для обеспечения идентичного поведения (см. §10)

### Согласованность со связанными RFC

| RFC | Согласованность |
|-----|--------|
| RFC-024 — модель конкурентности на основе spawn-блоков | ✅ Прямые подвыражения spawn-блока → диспетчеризация задач |
| RFC-008 — архитектура runtime | ✅ Два бэкенда + статическая библиотека планировщика + структура каталогов модулей |
| RFC-009 — модель владения v9 | ✅ Токены `&T`/`&mut T` (нулевой размер), `ref T` («толстый» указатель), `?T` (Option) |
| RFC-026 — базовый механизм FFI | ✅ `native()` → declare + marshalling, `.drop` → RAII cleanup |

## Связанные работы

### Lazy Task Creation (1990)[^1]

| Атрибут | Описание |
|------|------|
| Организация | MIT |
| Авторы | James R. Larus, Robert H. Halstead Jr. |
| Ядро | Отложенное создание подзадач, создание по требованию |
| Ценность для справки | Теоретический фундамент для планирования задач в spawn-блоках по требованию |

**Ключевая идея**: вместо немедленного создания задач — отложенное создание. Когда родительской задаче требуется значение подзадачи, подзадача создаётся в этот момент. Это решает проблему накладных расходов на мелкозернистые параллельные задачи[^1]. Планирование spawn-блоков YaoXiang заимствует эту идею — задачи идентифицируются в компайл-тайме, но в runtime распределяются в пул потоков по требованию.

### Lazy Scheduling (2014)[^2]

| Атрибут | Описание |
|------|------|
| Организация | University of Maryland |
| Авторы | Tzannes, Caragea |
| Ядро | Адаптивное планирование в runtime без дополнительного состояния |
| Ценность для справки | Ссылка для дизайна WorkStealing-планировщика в Full Runtime |

### Язык SISAL[^3]

| Атрибут | Описание |
|------|------|
| Организация | Lawrence Livermore National Laboratory (LLNL) |
| Ядро | Язык с единичным присваиванием, Dataflow-граф, неявный параллелизм |
| Ценность для справки | Доказательство осуществимости модели Dataflow в промышленных приложениях |

**Ключевое различие**: параллелизм в SISAL **неявный** — язык имеет семантику единичного присваивания, компилятор автоматически анализирует граф зависимостей данных по всей программе для определения параллелизма. В YaoXiang параллелизм **явный** — пользователь помечает параллельные области блоками `spawn {}`, компилятор анализирует зависимости только внутри spawn-блока. Это избегает сложности полнопрограммного анализа SISAL, сохраняя контроль пользователя над параллельным поведением.

### Mul-T — параллельный Scheme[^4]

| Атрибут | Описание |
|------|------|
| Организация | MIT |
| Ядро | Конструкция future, реализация Lazy Task Creation |
| Ценность для справки | Ссылка на конкретную реализацию |

### Сравнительная сводка

| Технология | Отложенное создание | Маркер параллелизма | Область анализа | Владение |
|------|----------|----------|----------|--------|
| Lazy Task Creation[^1] | ✅ | Неявный | Вся программа | N/A |
| Lazy Scheduling[^2] | ✅ | Неявный | Вся программа | N/A |
| SISAL[^3] | ✅ | Неявный (единичное присваивание) | Вся программа | N/A |
| Mul-T[^4] | ✅ | Явный (future) | Точка вызова | N/A |
| **YaoXiang** | ✅ | **Явный (spawn-блок)** | **Внутри spawn-блока** | **✅ (Move + токены + ref)** |

**Инновация YaoXiang**: подъём маркера параллелизма с уровня «каждый вызов функции» (future) до уровня «структурного блока» (spawn). Пользователь пишет обычный код и помещает spawn-блок в местах, где нужен параллелизм. Область анализа ограничена spawn-блоком — компиляция эффективна, поведение контролируемо.

### Приложение A: Сравнение с Rust async

| Характеристика | Rust async | YaoXiang LLVM AOT |
|------|-----------|-------------------|
| Артефакт компиляции | Конечный автомат + машинный код | Машинный код + метаданные задач spawn |
| Runtime | tokio | Статически линкуемый планировщик (около 500KB-1MB) |

| Цветные функции | «Заражающий» async | **Нет окрашивания функций** |
| Синхронное ожидание | `.await` | spawn-блок автоматически синхронно блокирует |
| Управление памятью | GC (в runtime) | **RAII (детерминированное)** |
| Механизм разделения | `Arc::new()` + ручной Weak | **Ключевое слово `ref` (компилятор автоматически выбирает Rc/Arc)** |
