# RFC-007: Единая схема синтаксиса определения функций

> **Статус**: Принято
> **Автор**: 沫郁酱
> **Дата создания**: 2025-01-05
> **Последнее обновление**: 2026-03-21 (выравнивание правил конструктора типа и семантики возврата из блоков кода)

## Аннотация

Данный RFC определяет финальный вариант **синтаксиса определения функций** для языка YaoXiang. Используется унифицированный синтаксис `name: (params) -> Return = body`, полностью соответствующий модели `name: type = value` из RFC-010.

Во избежание неоднозначностей: при наличии входных параметров типы параметров должны быть явно указаны либо в «сигнатуре», либо в «заголовке lambda», как минимум в одном месте; пропуск типов с обеих сторон не допускается.

Последнее выражение в блоке кода `{ ... }` используется как возвращаемое значение; пустой блок `{}` возвращает `Void`.

## Мотивация

### Зачем нужна эта функция?

1. **Синтаксическая согласованность**: устранение исторического багажа старого синтаксиса, унификация стиля
2. **Краткость**: алгоритм HM автоматически выводит типы, сокращая шаблонный код
3. **Типовая безопасность**: алгоритм HM гарантирует типобезопасность, явная аннотация требуется только при невозможности вывода
4. **Зрелость языка**: алгоритм HM — проверенное решение современных функциональных языков

### Унифицированная модель синтаксиса

**Основной принцип**: `name: Signature = LambdaBody`

- **Полная форма**: сигнатура (с именами параметров + типы + `->` + тип возврата) + заголовок lambda (с именами параметров)
- **Правила сокращения**: пропуск допускается везде, где это не приводит к неоднозначности
  - `->` нельзя пропускать (маркер типа функции, иначе будет распознано как кортеж)
  - **При наличии входных параметров** типы параметров должны быть явно указаны хотя бы в одном месте — в сигнатуре или в заголовке lambda
  - Заголовок lambda можно опустить → если сигнатура уже объявляет имена и типы параметров
  - Тип возврата можно указать явно или опустить при возможности вывода

```yaoxiang
# Полная форма (полная сигнатура + полный заголовок lambda)
add: (a: Int, b: Int) -> Int = (a, b) => a + b

# Сокращение: пропуск заголовка lambda (сигнатура уже объявляет параметры)
add: (a: Int, b: Int) -> Int = a + b

# Сокращение: пропуск сигнатуры (заголовок lambda аннотирует типы параметров)
add = (a: Int, b: Int) => a + b

# ❌ Ошибка: типы параметров не указаны с обеих сторон
# add = (a, b) => a + b
```

### Цели проектирования

```yaoxiang
# === Полная форма ===
add: (a: Int, b: Int) -> Int = (a, b) => { a + b }

# === Сокращённая форма ===
add: (a: Int, b: Int) -> Int = a + b                 # пропуск заголовка lambda
add = (a: Int, b: Int) => a + b                      # пропуск сигнатуры

# === Функции без параметров ===
main: () -> Void = () => { println("Hello") }          # полная форма
main: () -> Void = { println("Hello") }                # пропуск заголовка lambda
main = { println("Hello") }                            # минимальная форма (выводится как () -> Void)

# === Обобщённые функции (используется унифицированный синтаксис RFC-010)===
identity: (T: Type) -> ((x: T) -> T) = (x) => x         # полная форма
identity: (T: Type) -> ((x: T) -> T) = x                # пропуск заголовка lambda
identity = (x: T) => x                                  # пропуск сигнатуры (заголовок lambda аннотирует тип)

# === Рекурсивные функции ===
factorial: (n: Int) -> Int = (n) => {
    if n <= 1 { 1 } else { n * factorial(n - 1) }
}
```

### Синтаксические правила

| Сценарий | Синтаксис | Описание |
|------|------|------|
| **Полная форма** | `name: (a: Type, b) -> Ret = (a, b) => { return ... }` | Полная сигнатура + полный заголовок lambda |
| **Пропуск заголовка lambda** | `name: (a: Type, b: Type) -> Ret = { ... }` | Сигнатура объявляет параметры |
| **Пропуск сигнатуры** | `name = (a: Type, b: Type) => { ... }` | Заголовок lambda аннотирует типы параметров |
| **Полная форма без параметров** | `name: () -> Void = () => { return ... }` | Функция без параметров, полная форма |
| **Сокращение без параметров** | `name: () -> Void = { return ... }` | Пропуск заголовка lambda |
| **Минимум без параметров** | `name = { return ... }` | Минимальная форма без параметров и возврата |

**Примечание**: последнее выражение в блоке `{ ... }` используется как возвращаемое значение; для досрочного завершения используется `return`. Пустой блок `{}` выводится как `Void`.

**Примечание**: `->` является маркером типа функции и не может быть опущен (иначе будет распознано как кортеж).

**Важно**: выражение `if` использует фигурные скобки `{}` для веток и не поддерживает ключевые слова `then/else`:
```yaoxiang
# Правильно: использовать фигурные скобки
if n <= 1 { return 1 } else { return n * factorial(n - 1) }

# Ошибка: ключевые слова then/else не поддерживаются
# if n <= 1 then return 1 else return n * factorial(n - 1)
```

## Предложение

### Алгоритм HM и поддержка полиморфизма высших порядков

**Основная возможность**: алгоритм HM поддерживает полиморфизм высших порядков (Higher-rank polymorphism) через аннотации обобщённых типов

**Принцип проектирования**:
- **Функции высших порядков**: при передаче функций в качестве параметров необходимо ограничить их типы обобщёнными параметрами
- **Форма аннотации типа**: `(T: Type) -> ((f: (T) -> T, x: T) -> T)` — обобщённый параметр ограничивает тип функции
- **Принцип работы HM**: вывод типов функций через обобщённые параметры для реализации полиморфной композиции функций

**Пояснение примеров**:
```yaoxiang
# ✅ Поддержка полиморфизма высших порядков: обобщённое ограничение параметра типа функции
call_twice: (T: Type) -> ((f: (T) -> T, x: T) -> T) = {
    return f(f(x))
}
# Использование: call_twice((x) => x + 1, 5)  # выводится T=Int

compose: (A: Type, B: Type, C: Type) -> ((f: (B) -> C, g: (A) -> B, x: A) -> C) = {
    return f(g(x))
}
# Использование: compose((x) => x * 2, (x) => x + 1, 5)  # выводится A=Int, B=Int, C=Int

# ❌ Не поддерживается: функция высших порядков без обобщённого ограничения
# bad_hof: (f, x) => f(f(x))  # HM не может вывести, отсутствует обобщённый параметр
```

**Процесс вывода по HM**:
1. Распознавание параметра функции высшего порядка: `f: (T) -> T`
2. Создание обобщённого ограничения: `(T: Type)`
3. Вывод конкретного типа через обобщённую инстанциацию
4. Реализация полиморфной композиции функций

### Правила синтаксиса lambda-выражений

**Важное правило**: блок кода `{ ... }` возвращает значение последнего выражения; для досрочного завершения используется `return`. Пустой блок `{}` возвращает `Void`.

| Синтаксическая форма | Синтаксис | Способ возврата |
|---------|------|----------|
| **Форма блока кода** | `{ statements }` | Последнее выражение как возвращаемое значение; можно использовать `return` для досрочного возврата |
| **Выражение** | `expression` | Непосредственное возвращение значения выражения |

**Примеры**:
```yaoxiang
main: () -> Void = { println("Hello") }         # возвращает Void (последнее выражение — println)
add: (a: Int, b: Int) -> Int = { a + b }        # возвращает Int (последнее выражение — a + b)
empty: () -> Void = {}                          # пустой блок возвращает Void

# Досрочный возврат: использование return
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    n * factorial(n - 1)
}

# Форма выражения: непосредственный возврат значения (без return)
add: (a: Int, b: Int) -> Int = a + b            # правильно: форма выражения
main: () -> Void = println("Hello")               # правильно: форма выражения
```

**Ключевая идея**:
1. Определения функций используют алгоритм HM для вывода типов, с выводом по возможности и явными ошибками при невозможности
2. **Принцип работы алгоритма HM**: автоматический вывод типов через информацию контекста — ограничения типов операторов, отношения вызовов функций и т.д.
3. **Поддержка обобщений**: полиморфные функции используют обобщённый синтаксис `(T: Type)` для явного ограничения параметров типа (RFC-010/011)
4. **Границы вывода**: тип возврата и локальные переменные выводятся; типы параметров функций с параметрами требуют явной аннотации (в сигнатуре или заголовке lambda — хотя бы в одном месте)
5. Функции без параметров без возврата используют `name: () -> Void = { ... }`, унифицировано с RFC-010
6. Старый синтаксис выводится из эксплуатации, предоставляется инструмент миграции

**Примеры вывода типов**:
```yaoxiang
# Обобщённые функции: явные параметры типа (используется унифицированный синтаксис RFC-010)
identity: (T: Type) -> ((x: T) -> T) = x
map: (T: Type, R: Type) -> ((f: (T) -> R, list: List(T)) -> List(R)) = {
    result = List(R)()
    for item in list { result.push(f(item)) }
    return result
}

# Полиморфные функции: определение через явные обобщённые ограничения (RFC-010/011)
add: (T: Add) -> ((a: T, b: T) -> T) = a + b
print_sum: (a: Int, b: Int) -> Void = { println(a + b) }  # выводится как (Int, Int) -> Void

# Полиморфизм высших порядков: реализация поддержки HM через обобщённые аннотации типов
call_twice: (T: Type) -> ((f: (T) -> T, x: T) -> T) = { return f(f(x)) }
compose: (A: Type, B: Type, C: Type) -> ((f: (B) -> C, g: (A) -> B, x: A) -> C) = { return f(g(x)) }
```

```yaoxiang
# === Определение функций: вывод типов по алгоритму HM ===

# Стандартные функции: алгоритм HM выводит тип возврата (типы параметров требуют явного указания)
add = (a: Int, b: Int) => a + b            # выводится как (a: Int, b: Int) -> Int
main = { println("Hello") }                # выводится как () -> Void

# Частично явные параметры: алгоритм HM выводит оставшуюся часть
print_sum: (a: Int, b: Int) -> Void = { println(a + b) }  # выводится как (Int, Int) -> Void
greet: (name: String) -> Void = { println("Hello " + name) }  # выводится как (String) -> Void

# Обобщённые функции: явное ограничение параметров полиморфного типа (используется унифицированный синтаксис RFC-010)
identity: (T: Type) -> ((x: T) -> T) = x
map: (T: Type, R: Type) -> ((f: (T) -> R, list: List(T)) -> List(R)) = {
    # реализация функции map
    return List(R)()
}

# Рекурсивные функции: вывод через алгоритм HM и рекурсивные ограничения
factorial: (n: Int) -> Int = {
    if n <= 1 { 1 } else { n * factorial(n - 1) }
}

# === Присваивание переменных: вывод типов по алгоритму HM ===

# Явный тип
x: Int = 42

# Алгоритм HM автоматически выводит Int
y = 42                               # выводится как Int

# Алгоритм HM автоматически выводит String
name = "YaoXiang"                    # выводится как String

# Алгоритм HM автоматически выводит Float
pi = 3.14159                         # выводится как Float
```

**Правила вывода типов по HM**:

| Сценарий | Синтаксис | Можно опустить | Пример | Описание |
|------|------|----------|------|
| **Полная форма** | `name: (a: Type, b: Type) -> Ret = (a, b) => ...` | Ничего | Полная сигнатура + полный заголовок lambda |
| **Пропуск заголовка lambda** | `name: (a: Type, b: Type) -> Ret = ...` | Заголовок lambda | Сигнатура объявляет параметры |
| **Пропуск сигнатуры** | `name = (a: Type, b: Type) => ...` | Сигнатура | Заголовок lambda предоставляет типы параметров |
| **Пропуск возврата Ret** | `name: (a: Type, b: Type) -> = ...` | Тип возврата | HM выводит тип возврата |
| **Полная форма без параметров** | `name: () -> Void = () => { ... }` | Ничего | Функция без параметров, полная форма |
| **Сокращение без параметров** | `name: () -> Void = { ... }` | Заголовок lambda | Пропуск `() =>` |
| **Минимум без параметров** | `name = { ... }` | Всё | Минимальная форма без параметров и возврата |
| **Присваивание переменной** | `name = value` | Тип | HM выводит тип |
| **Явная переменная** | `name: Type = value` | Ничего | Явная типовая аннотация |

**Основные принципы**:
- `->` является маркером типа функции и не может быть опущен (иначе будет распознано как кортеж)
- Тип возврата `Ret` может быть опущен, выводится HM по телу функции
- При наличии входных параметров типы параметров должны быть явно указаны (в сигнатуре или заголовке lambda — хотя бы в одном месте)
- Остальные части могут быть опущены при возможности вывода и отсутствии неоднозначности
- Нет неявных преобразований типов, избегая хаоса как в JavaScript

## Детальное проектирование

### Раскрытие синтаксического сахара

Независимо от степени сокращения, всё最終规范化到 унифицированного промежуточного представления:

```rust
// Полная форма
add: (a: Int, b: Int) -> Int = (a, b) => a + b

// Раскрытие в IR
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    return a + b
};

// Пропуск заголовка lambda
add: (a: Int, b: Int) -> Int = a + b

// Раскрытие в IR (идентично полной форме)
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    return a + b
};

// Пропуск сигнатуры (заголовок lambda аннотирует типы параметров)
add = (a: Int, b: Int) => a + b

// Раскрытие в IR
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    a + b
};
```

### Определение синтаксиса

```bnf
function_def ::= identifier ':' type_expr '=' expression
               | identifier '=' expression
               | identifier '=' block                    # Минимальная форма: без параметров и возврата

identifier ::= [a-zA-Z_][a-zA-Z0-9_]*

type_expr ::= identifier                     # Ссылка на тип
       | '()'                          # Пустой тип
       | '(' parameters ')' '->' type_expr   # Функциональный тип (имена параметров в сигнатуре)
       | type_expr '->' type_expr            # Простой функциональный тип
       | identifier '(' type_expr (',' type_expr)* ')'  # Применение типа

expression ::= '(' parameters ')' '=>' block
             | '(' ')' '=>' block
             | '(' parameters ')' '=>' expression

parameters ::= parameter (',' parameter)*
parameter ::= identifier                # Вывод типа
            | identifier ':' type_expr      # Частично явный тип

block ::= '{' statement (',' statement)* '}'
        | expression

statement ::= identifier ':' expression  # Оператор присваивания
           | expression                  # Выражение (выполняется, но не возвращает)
           | 'return' expression         # Оператор возврата (возвращает указанное значение)

# Примечание: блок кода возвращает значение последнего выражения; пустой блок {} выводится как Void
# Например: { 1 + 1 } возвращает Int; { println("Hello") } возвращает Void
# Примечание: обобщённые параметры используют синтаксис (T: Type), как часть функционального типа, без отдельного BNF-правила
```

### Обработка ошибок

```yaoxiang
# === Примеры ошибок компиляции ===

# Ошибка 1: Несовпадение типа возврата блока кода
add: (a: Int, b: Int) -> Int = { println(a + b) }
// Ошибка: последнее выражение блока — println(...), возвращает Void, но сигнатура ожидает Int
// Правильно: add: (a: Int, b: Int) -> Int = a + b
// Или: add: (a: Int, b: Int) -> Int = { a + b }

# Ошибка 2: Использование необъявленного параметра типа
identity: (x: T) -> T = x
// Ошибка: T не объявлен; требуется явный обобщённый параметр (RFC-010)
// Правильно: identity: (T: Type) -> ((x: T) -> T) = x

# Правильно: алгоритм HM выводит тип возврата
double = (x: Int) => x + x

# Полная форма (постепенное сокращение)
double: (x: Int) -> Int = (x) => x + x                # полная
double: (x: Int) -> Int = x + x                       # пропуск заголовка lambda
double = (x: Int) => x + x                            # пропуск типа возврата (HM выводит возвращаемый тип)
# double = (x) => x + x                               # ❌ Типы параметров нельзя опускать с обеих сторон
```

## Компромиссы

### Преимущества

- **Унификация синтаксиса**: модель `name: Signature = LambdaBody` охватывает все сценарии
- **Гибкое сокращение**: любая часть может быть опущена при возможности вывода по HM
- **Типовая безопасность**: алгоритм HM гарантирует типовую безопасность, отсутствие неявных преобразований типов
- **Поддержка рекурсии**: алгоритм HM и рекурсивные ограничения автоматически выводят типы
- **Нулевая нагрузка**: плавный переход от полной к минимальной форме

### Недостатки

- **Стоимость миграции**: старый код требует инструмента конвертации
- **Стоимость обучения**: требуется понимание модели «полная форма + произвольное сокращение»

## Альтернативные решения

| Решение | Описание | Почему не выбрано |
|------|------|-----------|
| Вывод типов по алгоритму HM | Использование алгоритма Hindley-Milner для вывода типов | ✅ **Принято**, стандарт современных функциональных языков |
| Явные объявления типов | Все типы должны быть явно записаны | Противоречит принципу упрощения синтаксиса, увеличивает шаблонный код |
| Сохранение старого синтаксиса | Поддержка старого и нового синтаксиса одновременно | Разделение синтаксиса, высокая стоимость поддержки |
| Ключевое слово fn | Введение fn для различия функций и переменных | Противоречит дизайну «функция — это lambda» |

### Инструмент миграции

```bash
# Миграция одного файла
yaoxiang-migrate --old-to-new src/main.yaoxiang

# Миграция всего проекта
yaoxiang-migrate --old-to-new --recursive src/

# Предпросмотр миграции (без изменения файлов)
yaoxiang-migrate --old-to-new --dry-run src/main.yaoxiang
```

Правила миграции:
```yaoxiang
# Старый синтаксис
add(Int, Int) -> Int = (a, b) => { a + b }
main() -> Int = { println("Hello"); 0 }
main() = { println("Hello") }

# === Новый синтаксис: полная форма (полная сигнатура + полный заголовок lambda)===
add: (a: Int, b: Int) -> Int = (a, b) => a + b
main: () -> Void = () => { println("Hello") }

# === Сокращение: пропуск заголовка lambda ===
add: (a: Int, b: Int) -> Int = a + b
main: () -> Void = { println("Hello") }

# === Сокращение: вывод по HM ===
add = (a: Int, b: Int) => a + b              # выводится как (a: Int, b: Int) -> Int
main = { println("Hello") }                  # выводится как () -> Void

# === Минимальная форма ===
main = {                                      # эквивалентно main: () -> Void = { ... }
    println("Hello")
}
```

### Зависимости

- Нет внешних зависимостей
- Может быть реализовано независимо

### Риски

| Риск | Влияние | Меры снижения |
|------|------|---------| 
| Пропуск при миграции | Сбой компиляции старого кода | Предоставление инструмента миграции, охватывающего все паттерны старого синтаксиса |
| Ошибки парсера | Нестабильный синтаксический анализ | Достаточное тестовое покрытие |

## Открытые вопросы

> Следующие вопросы уже решены в проектировании, записаны в Приложении A.

- ~~Q1: Следует ли сохранить сверхкраткую запись `main() = body`?~~ → Решено: сохранено как `main = { ... }`
- ~~Q2: Сохраняется ли `:` после имени функции?~~ → Решено: необязательно; но для функций с параметрами типы параметров всё ещё должны быть аннотированы в сигнатуре или заголовке lambda
- ~~Q3: Поддерживает ли алгоритм HM вывод типов параметров?~~ → Решено: возвращаемое значение/локальные переменные выводятся; типы параметров функций с параметрами требуют явной аннотации
- ~~Q4: Вводится ли ключевое слово `fn`?~~ → Решено: не вводится, функция — это lambda
- ~~Q5: Какова стратегия миграции старого кода?~~ → Решено: предоставляется инструмент `yaoxiang-migrate`
- ~~Q6: Как использовать обобщённые функции?~~ → Решено: используется унифицированный синтаксис RFC-010 `(T: Type)`

## Приложения

### Приложение A: Справочник синтаксиса определения функций в разных языках

| Язык | Стиль синтаксиса | Особенности |
|------|---------|------|
| Rust | `fn add(a: i32, b: i32) -> i32 { ... }` | Ключевое слово + типовые аннотации |
| Haskell | `add a b = ...` / `add :: Int -> Int -> Int` | Сигнатура типа отделена |
| OCaml | `let add a b = ...` | Типы параметров можно опускать |
| MoonBit | `fn add(a: Int, b: Int): Int { ... }` | Краткие типовые аннотации |
| TypeScript | `const add = (a: number, b: number): number => ...` | Стиль lambda |
| Scala | `def add(a: Int, b: Int): Int = { ... }` | Ключевое слово def |
| **YaoXiang** | `name = (a: Int, b: Int) => a + b` | **Функция = lambda, HM выводит возвращаемое значение** |

### Приложение C: Глоссарий

| Термин | Определение |
|------|------|
| Алгоритм HM | Алгоритм вывода типов Hindley-Milner, автоматически выводит типы функций и переменных |
| Обобщения | Использование параметров типа `(T: Type)` для ограничения полиморфных функций, например `identity: (T: Type) -> ((x: T) -> T) = x` (RFC-010) |
| Заполнение типа по умолчанию | Функции без параметров без возврата опускают `-> Void`, компилятор заполняет автоматически |
| Синтаксический сахар | Синтаксические упрощения, делающие код более читаемым |
| Нормализация | Преобразование синтаксических форм в унифицированное внутреннее представление |
| Функция — это lambda | Функция по сути является lambda-переменной, тип выводится автоматически по алгоритму HM |
