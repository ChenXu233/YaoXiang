# RFC-007: Унифицированная схема синтаксиса определения функций

> **Статус**: Принято
> **Автор**: 沫郁酱
> **Дата создания**: 2025-01-05
> **Последнее обновление**: 2026-03-21 (выравнивание правил конструктора типов с семантикой возврата из блоков кода)

## Аннотация

Настоящий RFC определяет финальный вариант **синтаксиса определения функций** языка YaoXiang. Используется унифицированный синтаксис `name: (params) -> Return = body`, полностью соответствующий модели `name: type = value` из RFC-010.

Во избежание неоднозначности: при наличии входных параметров у функции типы параметров должны быть явно указаны либо в «сигнатуре», либо в «заголовке lambda» — хотя бы в одном из них; одновременное опущение обеих частей не допускается.

Внутри блока кода `{ ... }` необходимо использовать `return` для возврата значения; при отсутствии `return` по умолчанию возвращается `Void`. Форма выражения `= expr` возвращает значение напрямую.

## Мотивация

### Зачем нужна эта функциональность?

1. **Синтаксическая согласованность**: устранение исторического багажа старого синтаксиса, унификация стиля
2. **Краткость**: алгоритм HM автоматически выводит типы, сокращая шаблонный код
3. **Типовая безопасность**: алгоритм HM гарантирует типобезопасность, явное указание требуется только при невозможности вывода
4. **Зрелость языка**: алгоритм HM — проверенное решение современных функциональных языков

### Унифицированная модель синтаксиса

**Основной принцип**: `name: Signature = LambdaBody`

- **Полная форма**: сигнатура (с именами параметров + типы + `->` + тип возврата) + заголовок Lambda (с именами параметров)
- **Правила сокращения**: опускание допускается, если не вносит неоднозначность
  - `->` нельзя опускать (маркер типа функции, иначе будет распознано как кортеж)
  - **При наличии входных параметров** типы параметров должны быть явно указаны хотя бы в сигнатуре или в заголовке lambda
  - Заголовок lambda можно опустить → если в сигнатуре уже объявлены имена и типы параметров
  - Тип возврата можно указать явно, а можно опустить при возможности вывода

```yaoxiang
# Полная форма (полная сигнатура + полный заголовок lambda)
add: (a: Int, b: Int) -> Int = (a, b) => a + b

# Сокращение: опущен заголовок Lambda (сигнатура уже объявляет параметры)
add: (a: Int, b: Int) -> Int = a + b

# Сокращение: опущена сигнатура (заголовок lambda указывает типы параметров)
add = (a: Int, b: Int) => a + b

# ❌ Ошибка: типы параметров не указаны ни с одной стороны
# add = (a, b) => a + b
```

### Цели проектирования

```yaoxiang
# === Полная форма ===
add: (a: Int, b: Int) -> Int = (a, b) => { a + b }

# === Сокращённые формы ===
add: (a: Int, b: Int) -> Int = a + b                 # опущен заголовок Lambda
add = (a: Int, b: Int) => a + b                      # опущена сигнатура

# === Функции без параметров ===
main: () -> Void = () => { println("Hello") }          # полная форма
main: () -> Void = { println("Hello") }                # опущен заголовок Lambda
main = { println("Hello") }                            # минимальная форма (выводится как () -> Void)

# === Обобщённые функции (с использованием унифицированного синтаксиса RFC-010) ===
identity: (T: Type) -> ((x: T) -> T) = (x) => x         # полная форма
identity: (T: Type) -> ((x: T) -> T) = x                # опущен заголовок Lambda
identity = (x: T) => x                                  # опущена сигнатура (заголовок lambda указывает тип)

# === Рекурсивные функции ===
factorial: (n: Int) -> Int = (n) => {
    if n <= 1 { return 1 } else { return n * factorial(n - 1) }
}
```

### Синтаксические правила

| Сценарий | Синтаксис | Описание |
|------|------|----------|
| **Полная форма** | `name: (a: Type, b) -> Ret = (a, b) => { return ... }` | Полная сигнатура + полный заголовок Lambda |
| **Опущен заголовок Lambda** | `name: (a: Type, b: Type) -> Ret = { ... }` | Сигнатура объявляет параметры |
| **Опущена сигнатура** | `name = (a: Type, b: Type) => { ... }` | Заголовок lambda указывает типы параметров |
| **Полная форма без параметров** | `name: () -> Void = () => { return ... }` | Полная форма для функций без параметров |
| **Сокращённая форма без параметров** | `name: () -> Void = { return ... }` | Опущен заголовок Lambda |
| **Минимальная форма без параметров** | `name = { return ... }` | Минимальная форма без параметров и возврата |

**Примечание**: внутри блока кода `{ ... }` необходимо использовать `return` для возврата значения; при отсутствии `return` по умолчанию возвращается `Void`. Форма выражения `= expr` возвращает значение напрямую.

**Примечание**: `->` является маркером типа функции и не может быть опущен (иначе будет распознано как кортеж).

**Важно**: выражение `if` использует фигурные скобки `{}` для обёртки веток и не поддерживает ключевые слова `then/else`:

```yaoxiang
# Правильно: использование фигурных скобок
if n <= 1 { return 1 } else { return n * factorial(n - 1) }

# Неправильно: ключевые слова then/else не поддерживаются
# if n <= 1 then return 1 else return n * factorial(n - 1)
```

## Предложение

### Алгоритм HM и поддержка полиморфизма высших порядков

**Основная функциональность**: алгоритм HM поддерживает полиморфизм высших порядков (Higher-rank polymorphism) через параметрические типы.

**Принцип проектирования**:
- **Функции высших порядков**: при передаче функций в качестве параметров необходимо параметрически ограничить их типы
- **Форма аннотации типа**: `(T: Type) -> ((f: (T) -> T, x: T) -> T)` — параметрический параметр ограничивает тип функции
- **Рабочий процесс HM**: выведение типов функций через параметрические экземпляры, реализация полиморфной композиции функций

**Пояснение примеров**:

```yaoxiang
# ✅ Поддержка полиморфизма высших порядков: параметрическое ограничение типа функции
call_twice: (T: Type) -> ((f: (T) -> T, x: T) -> T) = {
    return f(f(x))
}
# Использование: call_twice((x) => x + 1, 5)  # выводится T=Int

compose: (A: Type, B: Type, C: Type) -> ((f: (B) -> C, g: (A) -> B, x: A) -> C) = {
    return f(g(x))
}
# Использование: compose((x) => x * 2, (x) => x + 1, 5)  # выводится A=Int, B=Int, C=Int

# ❌ Не поддерживается: функции высших порядков без параметрического ограничения
# bad_hof: (f, x) => f(f(x))  # HM не может вывести, отсутствует параметрический параметр
```

**Процесс вывода по HM**:

1. Распознавание параметров функций высших порядков: `f: (T) -> T`
2. Создание параметрического ограничения: `(T: Type)`
3. Выведение конкретных типов через параметрическую инстанциацию
4. Реализация полиморфной композиции функций

### Синтаксические правила выражений Lambda

**Важное правило**: внутри блока кода `{ ... }` необходимо использовать `return` для возврата значения; при отсутствии `return` по умолчанию возвращается `Void`. Форма выражения `= expr` возвращает значение напрямую.

| Форма синтаксиса | Синтаксис | Способ возврата |
|---------|------|----------|
| **Форма блока** | `{ statements }` | Обязательно использовать `return`; при отсутствии `return` по умолчанию `Void` |
| **Форма выражения** | `expression` | Возврат значения выражения напрямую |

**Примеры**:

```yaoxiang
main: () -> Void = { println("Hello") }         # возвращает Void (нет return)
add: (a: Int, b: Int) -> Int = { return a + b }  # возвращает Int (явный return)
empty: () -> Void = {}                          # пустой блок по умолчанию возвращает Void

# Ранний возврат: использование return
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Форма выражения: возврат значения напрямую (без return)
add: (a: Int, b: Int) -> Int = a + b            # правильно: форма выражения
main: () -> Void = println("Hello")               # правильно: форма выражения
```

**Ключевая идея**:

1. Определение функций через вывод типов алгоритмом HM, по возможности выведение, при невозможности — явная ошибка
2. **Принцип работы алгоритма HM**: автоматический вывод типов через контекстную информацию (ограничения типов операторов, отношения вызовов функций и т.д.)
3. **Поддержка обобщений**: полиморфные функции используют параметрический синтаксис `(T: Type)` для явного ограничения параметров типа (RFC-010/011)
4. **Границы вывода**: тип возврата и локальные переменные выводятся; типы параметров функций с параметрами требуют явной аннотации (в сигнатуре или в заголовке lambda — хотя бы в одном)
5. Функции без параметров без возврата используют `name: () -> Void = { ... }`, унифицировано с RFC-010
6. Старый синтаксис выводится из эксплуатации, предоставляется инструмент миграции

**Примеры вывода типов**:

```yaoxiang
# Обобщённые функции: явные параметры типа (с использованием унифицированного синтаксиса RFC-010)
identity: (T: Type) -> ((x: T) -> T) = x
map: (T: Type, R: Type) -> ((f: (T) -> R, list: List(T)) -> List(R)) = {
    result = List(R)()
    for item in list { result.push(f(item)) }
    return result
}

# Полиморфные функции: определение через явные параметрические ограничения (RFC-010/011)
add: (T: Add) -> ((a: T, b: T) -> T) = a + b
print_sum: (a: Int, b: Int) -> Void = { println(a + b) }  # выводится как (Int, Int) -> Void

# Полиморфизм высших порядков: реализация HM через параметрические аннотации типов
call_twice: (T: Type) -> ((f: (T) -> T, x: T) -> T) = { return f(f(x)) }
compose: (A: Type, B: Type, C: Type) -> ((f: (B) -> C, g: (A) -> B, x: A) -> C) = { return f(g(x)) }
```

```yaoxiang
# === Определение функций: вывод типов алгоритмом HM ===

# Стандартные функции: HM выводит тип возврата (типы параметров требуют явного указания)
add = (a: Int, b: Int) => a + b            # выводится как (a: Int, b: Int) -> Int
main = { println("Hello") }                # выводится как () -> Void

# Частично явные параметры: HM выводит остальное
print_sum: (a: Int, b: Int) -> Void = { println(a + b) }  # выводится как (Int, Int) -> Void
greet: (name: String) -> Void = { println("Hello " + name) }  # выводится как (String) -> Void

# Обобщённые функции: явное ограничение параметров полиморфного типа (с использованием унифицированного синтаксиса RFC-010)
identity: (T: Type) -> ((x: T) -> T) = x
map: (T: Type, R: Type) -> ((f: (T) -> R, list: List(T)) -> List(R)) = {
    # Реализация функции map
    return List(R)()
}

# Рекурсивные функции: вывод через HM и рекурсивные ограничения
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 } else { return n * factorial(n - 1) }
}

# === Присваивание переменных: вывод типов алгоритмом HM ===

# Явный тип
x: Int = 42

# HM автоматически выводит как Int
y = 42                               # выводится как Int

# HM автоматически выводит как String
name = "YaoXiang"                    # выводится как String

# HM автоматически выводит как Float
pi = 3.14159                         # выводится как Float
```

**Правила вывода типов по HM**:

| Сценарий | Синтаксис | Опускаемые части | Пример |
|------|------|----------|------|
| **Полная форма** | `name: (a: Type, b: Type) -> Ret = (a, b) => ...` | нет | полная сигнатура + полный заголовок Lambda |
| **Опущен заголовок Lambda** | `name: (a: Type, b: Type) -> Ret = ...` | заголовок Lambda | сигнатура объявляет параметры |
| **Опущена сигнатура** | `name = (a: Type, b: Type) => ...` | сигнатура | заголовок lambda предоставляет типы параметров |
| **Опущен возвращаемый Ret** | `name: (a: Type, b: Type) -> = ...` | тип возврата | HM выводит тип возврата |
| **Полная без параметров** | `name: () -> Void = () => { ... }` | нет | полная форма для функций без параметров |
| **Сокращённая без параметров** | `name: () -> Void = { ... }` | заголовок Lambda | опущено `() =>` |
| **Минимальная без параметров** | `name = { ... }` | всё | минимальная форма без параметров и возврата |
| **Присваивание переменной** | `name = value` | тип | HM выводит тип |
| **Явная переменная** | `name: Type = value` | нет | явная аннотация типа |

**Основные принципы**:

- `->` является маркером типа функции и не может быть опущен (иначе будет распознано как кортеж)
- Тип возврата `Ret` можно опустить, HM выведет его из тела функции
- При наличии входных параметров типы параметров должны быть явно указаны (в сигнатуре или в заголовке lambda — хотя бы в одном)
- Остальные части можно опустить при возможности вывода и отсутствии неоднозначности
- Нет неявных преобразований типов, избегание хаоса как в JavaScript

## Детальное проектирование

### Раскрытие синтаксического сахара

Независимо от степени сокращения, всё最终规范化到统一中间表示：

```rust
// Полная форма
add: (a: Int, b: Int) -> Int = (a, b) => a + b

// Раскрытый IR
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    return a + b
};

// Опущен заголовок Lambda
add: (a: Int, b: Int) -> Int = a + b

// Раскрытый IR (идентичен полной форме)
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    return a + b
};

// Опущена сигнатура (заголовок lambda указывает типы параметров)
add = (a: Int, b: Int) => a + b

// Раскрытый IR
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    a + b
};
```

### Синтаксическое определение

```bnf
function_def ::= identifier ':' type_expr '=' expression
               | identifier '=' expression
               | identifier '=' block                    # минимальная форма: без параметров и возврата

identifier ::= [a-zA-Z_][a-zA-Z0-9_]*

type_expr ::= identifier                     # ссылка на тип
       | '()'                          # пустой тип
       | '(' parameters ')' '->' type_expr   # функциональный тип (имена параметров в сигнатуре)
       | type_expr '->' type_expr            # простой функциональный тип
       | identifier '(' type_expr (',' type_expr)* ')'  # применение типа

expression ::= '(' parameters ')' '=>' block
             | '(' ')' '=>' block
             | '(' parameters ')' '=>' expression

parameters ::= parameter (',' parameter)*
parameter ::= identifier                # вывод типа
            | identifier ':' type_expr      # частично явный тип

block ::= '{' statement (',' statement)* '}'
        | expression

statement ::= identifier ':' expression  # оператор присваивания
           | expression                  # выражение (выполняется, но не возвращает)
           | 'return' expression         # оператор возврата (возвращает указанное значение)

# Примечание: в блоке кода обязателен return для возврата значения; без return по умолчанию Void
# Например: { return 1 + 1 } возвращает Int; { println("Hello") } возвращает Void
# Примечание: параметрические параметры используют синтаксис (T: Type), как часть функционального типа,
# отдельное BNF-правило не требуется
```

### Обработка ошибок

```yaoxiang
# === Примеры ошибок компиляции ===

# Ошибка 1: несоответствие типа возврата блока
add: (a: Int, b: Int) -> Int = { println(a + b) }
// Ошибка: в блоке нет return, по умолчанию Void, но сигнатура ожидает Int
// Правильно: add: (a: Int, b: Int) -> Int = a + b
// или: add: (a: Int, b: Int) -> Int = { return a + b }

# Ошибка 2: использование необъявленного параметра типа
identity: (x: T) -> T = x
// Ошибка: T не объявлен; требуется явный параметрический параметр (RFC-010)
// Правильно: identity: (T: Type) -> ((x: T) -> T) = x

# Правильно: HM выводит тип возврата
double = (x: Int) => x + x

# Полная форма (пошаговое сокращение)
double: (x: Int) -> Int = (x) => x + x                # полная
double: (x: Int) -> Int = x + x                       # опущен заголовок Lambda
double = (x: Int) => x + x                            # опущен тип возврата (HM выводит)
# double = (x) => x + x                               # ❌ Типы параметров нельзя опускать с обеих сторон
```

## Компромиссы

### Преимущества

- **Единый синтаксис**: модель `name: Signature = LambdaBody` охватывает все сценарии
- **Гибкое сокращение**: любая часть может быть опущена при возможности вывода по HM
- **Типовая безопасность**: алгоритм HM гарантирует типобезопасность, отсутствие неявных преобразований типов
- **Поддержка рекурсии**: HM и рекурсивные ограничения автоматически выводят типы
- **Нулевая нагрузка**: плавный переход от полной к минимальной форме

### Недостатки

- **Стоимость миграции**: старый код требует инструмента преобразования
- **Порог изучения**: необходимо понимать модель «полная форма + произвольное сокращение»

## Альтернативные решения

| Решение | Описание | Почему не выбрано |
|------|------|-----------|
| Вывод типов по алгоритму HM | Использование алгоритма Hindley-Milner для вывода типов | ✅ **Принято**, стандарт современных функциональных языков |
| Явные объявления типов | Все типы должны быть явно указаны | Противоречит принципу упрощения синтаксиса, увеличивает шаблонный код |
| Сохранение старого синтаксиса | Поддержка старого и нового синтаксиса одновременно | Расщепление синтаксиса, высокая стоимость сопровождения |
| Ключевое слово fn | Введение fn для различения функций и переменных | Противоречит проектированию «функция = lambda» |

### Инструмент миграции

```bash
# Миграция одного файла
yaoxiang-migrate --old-to-new src/main.yaoxiang

# Миграция всего проекта
yaoxiang-migrate --old-to-new --recursive src/

# Предпросмотр миграции (без изменения файлов)
yaoxiang-migrate --old-to-new --dry-run src/main.yaoxiang
```

Правила миграции:

```yaoxiang
# Старый синтаксис
add(Int, Int) -> Int = (a, b) => { a + b }
main() -> Int = { println("Hello"); 0 }
main() = { println("Hello") }

# === Новый синтаксис: полная форма (полная сигнатура + полный заголовок Lambda) ===
add: (a: Int, b: Int) -> Int = (a, b) => a + b
main: () -> Void = () => { println("Hello") }

# === Сокращение: опущен заголовок Lambda ===
add: (a: Int, b: Int) -> Int = a + b
main: () -> Void = { println("Hello") }

# === Сокращение: вывод по HM ===
add = (a: Int, b: Int) => a + b              # выводится как (a: Int, b: Int) -> Int
main = { println("Hello") }                  # выводится как () -> Void

# === Минимальная форма ===
main = {                                      # эквивалентно main: () -> Void = { ... }
    println("Hello")
}
```

### Зависимости

- Нет внешних зависимостей
- Может быть реализовано независимо

### Риски

| Риск | Влияние | Меры снижения |
|------|------|---------|
| Пропуск при миграции | Ошибка компиляции старого кода | Предоставление инструмента миграции, охватывающего все паттерны старого синтаксиса |
| Ошибки парсера | Нестабильный синтаксический разбор | Достаточное тестовое покрытие |

## Открытые вопросы

> Следующие вопросы решены в процессе проектирования, записаны в Приложении A.

- ~~Q1: Следует ли сохранить предельно краткую запись `main() = body`?~~ → Решено: сохранена как `main = { ... }`
- ~~Q2: Сохраняется ли `:` после имени функции?~~ → Решено: опционально; но для функций с параметрами всё ещё требуется аннотация типов параметров в сигнатуре или в заголовке lambda
- ~~Q3: Поддерживает ли алгоритм HM вывод типов параметров?~~ → Решено: возвращаемое значение/локальные переменные выводятся; типы параметров функций с параметрами требуют явной аннотации
- ~~Q4: Вводится ли ключевое слово `fn`?~~ → Решено: не вводится, функция — это lambda
- ~~Q5: Какова стратегия миграции старого кода?~~ → Решено: предоставлен инструмент `yaoxiang-migrate`
- ~~Q6: Как использовать обобщённые функции?~~ → Решено: использовать унифицированный синтаксис RFC-010 `(T: Type)`

### Приложение A: Справочник синтаксиса определения функций в различных языках

| Язык | Стиль синтаксиса | Особенности |
|------|---------|------|
| Rust | `fn add(a: i32, b: i32) -> i32 { ... }` | Ключевое слово + аннотации типов |
| Haskell | `add a b = ...` / `add :: Int -> Int -> Int` | Типовая сигнатура отдельно |
| OCaml | `let add a b = ...` | Типы параметров можно опускать |
| MoonBit | `fn add(a: Int, b: Int): Int { ... }` | Краткие аннотации типов |
| TypeScript | `const add = (a: number, b: number): number => ...` | Стиль Lambda |
| Scala | `def add(a: Int, b: Int): Int = { ... }` | Ключевое слово def |
| **YaoXiang** | `name = (a: Int, b: Int) => a + b` | **Функция = lambda, HM выводит возвращаемое значение** |

### Приложение C: Глоссарий

| Термин | Определение |
|------|------|
| Алгоритм HM | Алгоритм вывода типов Hindley-Milner, автоматически выводит типы функций и переменных |

| Нормализация | Преобразование формы синтаксиса в унифицированное внутреннее представление |
| Функция как lambda | Функция по сути является переменной lambda, тип автоматически выводится алгоритмом HM |
