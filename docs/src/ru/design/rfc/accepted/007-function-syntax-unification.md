# RFC-007: Унифицированный синтаксис определения функций

## Резюме

Настоящий RFC определяет окончательное решение для синтаксиса **определения функций** языка YaoXiang. Используется унифицированный синтаксис `name: (params) -> Return = body`, полностью согласующийся с моделью `name: type = value` из RFC-010.

Во избежание неоднозначности: если функция имеет входные параметры, типы параметров должны быть явно указаны как минимум в одном из мест — в «сигнатуре» или в «заголовке lambda»; отсутствие указаний в обоих местах будет отклонено.

Внутри блока кода `{ ... }` для возврата значения обязательно использование `return`; при отсутствии `return` по умолчанию возвращается `Void`. Выражение в форме `= expr` возвращает значение напрямую.

## Мотивация

### Зачем нужна эта возможность?

1. **Согласованность синтаксиса**: устранить историческое наследие старого синтаксиса, унифицировать стиль
2. **Краткость**: алгоритм HM автоматически выводит типы, сокращая шаблонный код
3. **Безопасность типов**: алгоритм HM гарантирует безопасность типов, явное указание требуется только когда вывод невозможен
4. **Зрелость языка**: алгоритм HM — зрелое решение современных функциональных языков

### Модель унифицированного синтаксиса

**Базовый принцип**: `name: Signature = LambdaBody`

- **Полная форма**: сигнатура (содержит имена параметров + типы + `->` + тип возврата) + заголовок lambda (содержит имена параметров)
- **Правила сокращения**: опускать всё, что не приводит к неоднозначности
  - `->` нельзя опускать (признак функционального типа, иначе будет разобрано как кортеж)
  - **При наличии входных параметров** их типы должны быть явно указаны как минимум в одном из мест: в сигнатуре или в заголовке lambda
  - Заголовок lambda можно опустить → если имена и типы параметров уже объявлены в сигнатуре
  - Тип возврата можно указать явно, либо опустить, если он выводим

```yaoxiang
# Полная форма (сигнатура полностью + заголовок lambda полностью)
add: (a: Int, b: Int) -> Int = (a, b) => a + b

# Сокращение: опускаем заголовок lambda (параметры объявлены в сигнатуре)
add: (a: Int, b: Int) -> Int = a + b

# Сокращение: опускаем сигнатуру (типы параметров указаны в заголовке lambda)
add = (a: Int, b: Int) => a + b

# ❌ Ошибка: типы параметров не указаны ни в одном месте
# add = (a, b) => a + b
```

### Цели проектирования

```yaoxiang
# === Полная форма ===
add: (a: Int, b: Int) -> Int = (a, b) => { a + b }

# === Сокращённые формы ===
add: (a: Int, b: Int) -> Int = a + b                 # Опускаем заголовок lambda
add = (a: Int, b: Int) => a + b                      # Опускаем сигнатуру

# === Функции без параметров ===
main: () -> Void = () => { println("Hello") }          # Полная форма
main: () -> Void = { println("Hello") }                # Опускаем заголовок lambda
main = { println("Hello") }                            # Минимальная форма (выводится как () -> Void)

# === Обобщённые (generic) функции (используется унифицированный синтаксис RFC-010) ===
identity: (T: Type) -> ((x: T) -> T) = (x) => x         # Полная форма
identity: (T: Type) -> ((x: T) -> T) = x                # Опускаем заголовок lambda
identity = (x: T) => x                                  # Опускаем сигнатуру (типы указаны в заголовке lambda)

# === Рекурсивные функции ===
factorial: (n: Int) -> Int = (n) => {
    if n <= 1 { return 1 } else { return n * factorial(n - 1) }
}
```

### Правила синтаксиса

| Сценарий | Синтаксис | Пояснение |
|------|------|----------|
| **Полная форма** | `name: (a: Type, b) -> Ret = (a, b) => { return ... }` | Сигнатура + заголовок lambda полностью |
| **Опускаем заголовок lambda** | `name: (a: Type, b: Type) -> Ret = { ... }` | Параметры объявлены в сигнатуре |
| **Опускаем сигнатуру** | `name = (a: Type, b: Type) => { ... }` | Типы параметров указаны в заголовке lambda |
| **Без параметров, полная** | `name: () -> Void = () => { return ... }` | Полная форма для функции без параметров |
| **Без параметров, сокращённая** | `name: () -> Void = { return ... }` | Опускаем заголовок lambda |
| **Без параметров, минимальная** | `name = { return ... }` | Минимальная: без параметров и без возврата |

**Внимание**: внутри блока кода `{ ... }` для возврата значения обязательно использование `return`; при отсутствии `return` по умолчанию возвращается `Void`. Выражение в форме `= expr` возвращает значение напрямую.

**Внимание**: `->` — это признак функционального типа, его нельзя опускать (иначе будет разобрано как кортеж).

**Важно**: выражение `if` использует фигурные скобки `{}` для обрамления ветвей, ключевые слова `then/else` не поддерживаются:
```yaoxiang
# Правильно: используем фигурные скобки
if n <= 1 { return 1 } else { return n * factorial(n - 1) }

# Ошибка: ключевые слова then/else не поддерживаются
# if n <= 1 then return 1 else return n * factorial(n - 1)
```

## Предложение

### Алгоритм HM и поддержка полиморфизма высшего ранга

**Ключевая возможность**: алгоритм HM поддерживает полиморфизм высшего ранга (Higher-rank polymorphism) посредством аннотаций обобщённых типов.

**Принцип проектирования**:
- **Функции высшего порядка**: при передаче функции в качестве аргумента для ограничения её типа требуются обобщённые параметры
- **Форма аннотации типа**: `(T: Type) -> ((f: (T) -> T, x: T) -> T)` — обобщённый параметр ограничивает функциональный тип
- **Рабочий процесс HM**: вывод функциональных типов через инстанциацию обобщённых параметров, реализация композиции полиморфных функций

**Поясняющие примеры**:
```yaoxiang
# ✅ Поддержка полиморфизма высшего ранга: обобщённый параметр ограничивает функциональный аргумент
call_twice: (T: Type) -> ((f: (T) -> T, x: T) -> T) = {
    return f(f(x))
}
# Использование: call_twice((x) => x + 1, 5)  # Выводится T=Int

compose: (A: Type, B: Type, C: Type) -> ((f: (B) -> C, g: (A) -> B, x: A) -> C) = {
    return f(g(x))
}
# Использование: compose((x) => x * 2, (x) => x + 1, 5)  # Выводится A=Int, B=Int, C=Int

# ❌ Не поддерживается: функция высшего порядка без обобщённого ограничения
# bad_hof: (f, x) => f(f(x))  # HM не может выполнить вывод, отсутствует обобщённый параметр
```

**Процесс вывода HM**:
1. Распознать аргументы-функции высшего порядка: `f: (T) -> T`
2. Создать обобщённое ограничение: `(T: Type)`
3. Вывести конкретные типы путём инстанциации обобщённых параметров
4. Реализовать композицию полиморфных функций

### Правила синтаксиса lambda-выражений

**Важное правило**: внутри блока кода `{ ... }` для возврата значения обязательно использование `return`; при отсутствии `return` по умолчанию возвращается `Void`. Выражение в форме `= expr` возвращает значение напрямую.

| Синтаксическая форма | Синтаксис | Способ возврата |
|---------|------|----------|
| **Блочная форма** | `{ statements }` | Обязательно использование `return`; без `return` по умолчанию `Void` |
| **Выражение** | `expression` | Возвращает значение выражения напрямую |

**Примеры**:
```yaoxiang
main: () -> Void = { println("Hello") }         # Возвращает Void (без return)
add: (a: Int, b: Int) -> Int = { return a + b }  # Возвращает Int (явный return)
empty: () -> Void = {}                          # Пустой блок по умолчанию возвращает Void

# Досрочный возврат: используем return
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Форма выражения: возвращает значение напрямую (без return)
add: (a: Int, b: Int) -> Int = a + b            # Правильно: форма выражения
main: () -> Void = println("Hello")               # Правильно: форма выражения
```

**Ключевые идеи**:
1. Определение функций использует алгоритм HM для вывода типов, максимально автоматически; при невозможности вывода выдаётся явная ошибка
2. **Принцип работы алгоритма HM**: автоматический вывод типов на основе контекстной информации — типов операторов, отношений вызовов функций и т.д.
3. **Поддержка обобщений (generics)**: полиморфные функции используют синтаксис `(T: Type)` для явного ограничения параметров типа (RFC-010/011)
4. **Границы вывода**: тип возврата и локальные переменные выводимы; для функций с параметрами типы параметров требуют явного указания (в сигнатуре или в заголовке lambda)
5. Функции без параметров и без возврата используют форму `name: () -> Void = { ... }`, унифицированную с RFC-010
6. Старый синтаксис выводится из употребления, предоставляется инструмент миграции

**Примеры вывода типов**:
```yaoxiang
# Обобщённые функции: явные параметры типов (используется унифицированный синтаксис RFC-010)
identity: (T: Type) -> ((x: T) -> T) = x
map: (T: Type, R: Type) -> ((f: (T) -> R, list: List(T)) -> List(R)) = {
    result = List(R)()
    for item in list { result.push(f(item)) }
    return result
}

# Полиморфные функции: определяются посредством явных обобщённых ограничений (RFC-010/011)
add: (T: Add) -> ((a: T, b: T) -> T) = a + b
print_sum: (a: Int, b: Int) -> Void = { println(a + b) }  # Выводится как (Int, Int) -> Void

# Полиморфизм высшего ранга: реализуется через обобщённые аннотации типов, поддерживаемые HM
call_twice: (T: Type) -> ((f: (T) -> T, x: T) -> T) = { return f(f(x)) }
compose: (A: Type, B: Type, C: Type) -> ((f: (B) -> C, g: (A) -> B, x: A) -> C) = { return f(g(x)) }
```

```yaoxiang
# === Определение функций: вывод типов алгоритмом HM ===

# Стандартная функция: алгоритм HM выводит тип возврата (типы параметров требуют явного указания)
add = (a: Int, b: Int) => a + b            # Выводится как (a: Int, b: Int) -> Int
main = { println("Hello") }                # Выводится как () -> Void

# Частично явные параметры: алгоритм HM выводит остальное
print_sum: (a: Int, b: Int) -> Void = { println(a + b) }  # Выводится как (Int, Int) -> Void
greet: (name: String) -> Void = { println("Hello " + name) }  # Выводится как (String) -> Void

# Обобщённые функции: явные ограничения для полиморфных параметров типа (унифицированный синтаксис RFC-010)
identity: (T: Type) -> ((x: T) -> T) = x
map: (T: Type, R: Type) -> ((f: (T) -> R, list: List(T)) -> List(R)) = {
    # Реализация функции map
    return List(R)()
}

# Рекурсивная функция: выводится посредством алгоритма HM и рекурсивного ограничения
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 } else { return n * factorial(n - 1) }
}

# === Присваивание переменных: вывод типов алгоритмом HM ===

# Явный тип
x: Int = 42

# Алгоритм HM автоматически выводит Int
y = 42                               # Выводится как Int

# Алгоритм HM автоматически выводит String
name = "YaoXiang"                    # Выводится как String

# Алгоритм HM автоматически выводит Float
pi = 3.14159                         # Выводится как Float
```

**Правила вывода типов HM**:

| Сценарий | Синтаксис | Что можно опустить | Пример |
|------|------|----------|------|
| **Полная форма** | `name: (a: Type, b: Type) -> Ret = (a, b) => ...` | Ничего | Сигнатура + заголовок lambda полностью |
| **Опускаем заголовок lambda** | `name: (a: Type, b: Type) -> Ret = ...` | Заголовок lambda | Параметры объявлены в сигнатуре |
| **Опускаем сигнатуру** | `name = (a: Type, b: Type) => ...` | Сигнатура | Типы параметров указаны в заголовке lambda |
| **Опускаем тип возврата Ret** | `name: (a: Type, b: Type) -> = ...` | Тип возврата | HM выводит тип возврата |
| **Без параметров, полная** | `name: () -> Void = () => { ... }` | Ничего | Полная форма для функции без параметров |
| **Без параметров, сокращённая** | `name: () -> Void = { ... }` | Заголовок lambda | Опускаем `() =>` |
| **Без параметров, минимальная** | `name = { ... }` | Всё | Минимальная форма без параметров и возврата |
| **Присваивание переменной** | `name = value` | Тип | HM выводит тип |
| **Явная переменная** | `name: Type = value` | Ничего | Явная аннотация типа |

**Базовые принципы**:
- `->` — это признак функционального типа, его нельзя опускать (иначе будет разобрано как кортеж)
- Тип возврата `Ret` можно опустить, HM выведёт его из тела функции
- При наличии входных параметров их типы должны быть указаны явно (в сигнатуре или в заголовке lambda)
- Остальные части можно опускать, когда они выводимы и это не приводит к неоднозначности
- Неявные преобразования типов отсутствуют, что исключает путаницу в стиле JavaScript

## Подробное проектирование

### Раскрытие синтаксического сахара

Независимо от того, что опущено, в итоге всё нормализуется к единому промежуточному представлению:

```rust
// Полная форма
add: (a: Int, b: Int) -> Int = (a, b) => a + b

// Раскрытие в IR
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    return a + b
};

// Опускаем заголовок lambda
add: (a: Int, b: Int) -> Int = a + b

// Раскрытие в IR (идентично полной форме)
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    return a + b
};

// Опускаем сигнатуру (типы параметров указаны в заголовке lambda)
add = (a: Int, b: Int) => a + b

// Раскрытие в IR
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    a + b
};
```

### Определение синтаксиса

```bnf
function_def ::= identifier ':' type_expr '=' expression
               | identifier '=' expression
               | identifier '=' block                    # Минимальная форма: без параметров и без возврата

identifier ::= [a-zA-Z_][a-zA-Z0-9_]*

type_expr ::= identifier                     # Ссылка на тип
       | '()'                          # Пустой тип
       | '(' parameters ')' '->' type_expr   # Функциональный тип (имена параметров в сигнатуре)
       | type_expr '->' type_expr            # Простой функциональный тип
       | identifier '(' type_expr (',' type_expr)* ')'  # Применение типа

expression ::= '(' parameters ')' '=>' block
             | '(' ')' '=>' block
             | '(' parameters ')' '=>' expression

parameters ::= parameter (',' parameter)*
parameter ::= identifier                # Вывод типа
            | identifier ':' type_expr      # Частично явный тип

block ::= '{' statement (',' statement)* '}'
        | expression

statement ::= identifier ':' expression  # Оператор присваивания
           | expression                  # Выражение-инструкция (выполняется, не возвращает)
           | 'return' expression         # Оператор возврата (возвращает указанное значение)

# Внимание: внутри блока кода обязательно использование return; без return по умолчанию возвращается Void
# Например: { return 1 + 1 } возвращает Int; { println("Hello") } возвращает Void
# Внимание: обобщённые параметры используют синтаксис (T: Type) как часть функционального типа, отдельное правило BNF не требуется
```

### Обработка ошибок

```yaoxiang
# === Примеры ошибок компиляции ===

# Ошибка 1: тип возврата блока не совпадает
add: (a: Int, b: Int) -> Int = { println(a + b) }
// Ошибка: в блоке нет return, по умолчанию возвращается Void, но сигнатура ожидает Int
// Правильно: add: (a: Int, b: Int) -> Int = a + b
// Или: add: (a: Int, b: Int) -> Int = { return a + b }

# Ошибка 2: использование необъявленного параметра типа
identity: (x: T) -> T = x
// Ошибка: T не объявлен; требуется явный обобщённый параметр (RFC-010)
// Правильно: identity: (T: Type) -> ((x: T) -> T) = x

# Правильно: алгоритм HM выводит тип возврата
double = (x: Int) => x + x

# Полная форма (пошаговое сокращение)
double: (x: Int) -> Int = (x) => x + x                # Полная
double: (x: Int) -> Int = x + x                       # Опускаем заголовок lambda
double = (x: Int) => x + x                            # Опускаем тип возврата (HM выводит)
// double = (x) => x + x                               # ❌ Типы параметров нельзя опускать в обоих местах
```

## Компромиссы

### Преимущества

- **Согласованность синтаксиса**: модель `name: Signature = LambdaBody` покрывает все сценарии
- **Гибкие сокращения**: любую часть можно опустить, если она выводима посредством HM
- **Безопасность типов**: алгоритм HM гарантирует безопасность типов, исключая неявные преобразования
- **Поддержка рекурсии**: алгоритм HM и рекурсивные ограничения автоматически выводят типы
- **Отсутствие накладных расходов**: плавный переход от полной формы к минимальной

### Недостатки

- **Стоимость миграции**: старый код требует конвертации с помощью инструмента миграции
- **Стоимость обучения**: необходимо понять модель «полная форма + произвольные сокращения»

## Альтернативы

| Решение | Описание | Почему не выбрано |
|------|------|-----------|
| Вывод типов алгоритмом HM | Использование алгоритма Хиндли-Милнера для вывода типов | ✅ **Принято**, стандарт современных функциональных языков |
| Явные объявления типов | Все типы должны быть записаны явно | Нарушает принцип упрощения синтаксиса, увеличивает объём шаблонного кода |
| Сохранение старого синтаксиса | Одновременная поддержка старого и нового синтаксиса | Разделение синтаксиса, высокие затраты на сопровождение |
| Ключевое слово `fn` | Введение `fn` для различения функций и переменных | Нарушает принцип «функция — это lambda» |

## Стратегия реализации

### Этапы

1. **Фаза 1: Синтаксический разбор и алгоритм HM** (v0.3)
   - Реализация нового синтаксиса `name = lambda` + вывод типов алгоритмом HM
   - Реализация автоматического заполнения для функций без параметров и без возврата

2. **Фаза 2: Инструмент миграции** (v0.3)
   - Разработка инструмента `yaoxiang-migrate --old-to-new`
   - Автоматическая конвертация кода со старым синтаксисом

3. **Фаза 3: Валидация и документация** (v0.3)
   - Верификация завершения миграции старого кода
   - Обновление документации

### Инструмент миграции

```bash
# Миграция одного файла
yaoxiang-migrate --old-to-new src/main.yaoxiang

# Миграция всего проекта
yaoxiang-migrate --old-to-new --recursive src/

# Предварительный просмотр миграции (без изменения файлов)
yaoxiang-migrate --old-to-new --dry-run src/main.yaoxiang
```

Правила миграции:
```yaoxiang
# Старый синтаксис
add(Int, Int) -> Int = (a, b) => { a + b }
main() -> Int = { println("Hello"); 0 }
main() = { println("Hello") }

# === Новый синтаксис: полная форма (сигнатура полностью + заголовок lambda полностью) ===
add: (a: Int, b: Int) -> Int = (a, b) => a + b
main: () -> Void = () => { println("Hello") }

# === Сокращение: опускаем заголовок lambda ===
add: (a: Int, b: Int) -> Int = a + b
main: () -> Void = { println("Hello") }

# === Сокращение: вывод HM ===
add = (a: Int, b: Int) => a + b              # Выводится как (a: Int, b: Int) -> Int
main = { println("Hello") }                  # Выводится как () -> Void

# === Минимальная форма ===
main = {                                      # Эквивалентно main: () -> Void = { ... }
    println("Hello")
}
```

### Зависимости

- Внешние зависимости отсутствуют
- Может быть реализовано независимо

### Риски

| Риск | Влияние | Меры по снижению |
|------|------|---------|
| Пропуски при миграции | Сбой компиляции старого кода | Предоставить инструмент миграции, покрывающий все шаблоны старого синтаксиса |
| Ошибки парсера | Нестабильность синтаксического разбора | Достаточное тестовое покрытие |

## Открытые вопросы

> Перечисленные ниже вопросы решены в рамках проектирования и зафиксированы в Приложении A.

- ~~В1: Следует ли сохранить максимально краткую форму записи `main() = body`?~~ → Решено: сохранено как `main = { ... }`
- ~~В2: Сохранять ли `:` после имени функции?~~ → Решено: сохранение опционально; но для функций с параметрами типы параметров по-прежнему требуется указать в сигнатуре или в заголовке lambda
- ~~В3: Поддерживает ли алгоритм HM вывод типов параметров?~~ → Решено: тип возврата / локальные переменные выводимы; для функций с параметрами типы параметров требуют явного указания
- ~~В4: Вводить ли ключевое слово `fn`?~~ → Решено: не вводится, функция — это lambda
- ~~В5: Какова стратегия миграции старого кода?~~ → Решено: предоставляется инструмент `yaoxiang-migrate`
- ~~В6: Как использовать обобщённые функции?~~ → Решено: используется унифицированный синтаксис RFC-010 `(T: Type)`

## Приложения

### Приложение A: Справка по синтаксису определения функций в различных языках

| Язык | Стиль синтаксиса | Особенности |
|------|---------|------|
| Rust | `fn add(a: i32, b: i32) -> i32 { ... }` | Ключевое слово + аннотации типов |
| Haskell | `add a b = ...` / `add :: Int -> Int -> Int` | Сигнатура типа отделена от определения |
| OCaml | `let add a b = ...` | Типы параметров можно опускать |
| MoonBit | `fn add(a: Int, b: Int): Int { ... }` | Краткие аннотации типов |
| TypeScript | `const add = (a: number, b: number): number => ...` | Стиль lambda |
| Scala | `def add(a: Int, b: Int): Int = { ... }` | Ключевое слово def |
| **YaoXiang** | `name = (a: Int, b: Int) => a + b` | **Функция = lambda, возвращаемый тип выводится HM** |

### Приложение C: Глоссарий

| Термин | Определение |
|------|------|
| Алгоритм HM | Алгоритм вывода типов Хиндли-Милнера, автоматически выводит типы функций и переменных |
| Обобщения (generics) | Использование параметров типа `(T: Type)` для ограничения полиморфных функций, например `identity: (T: Type) -> ((x: T) -> T) = x` (RFC-010) |
| Заполнение типа по умолчанию | Для функций без параметров и без возврата опускается `-> Void`, компилятор заполняет автоматически |
| Синтаксический сахар | Упрощённая запись синтаксиса, облегчающая чтение кода |
| Нормализация | Преобразование синтаксических форм в единое внутреннее представление |
| Функция есть lambda | Функция по своей сути является переменной-lambda, тип выводится алгоритмом HM |
