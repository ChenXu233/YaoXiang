# RFC-007: Унифицированная схема синтаксиса определения функций

> **Статус**: Принято
> **Автор**: Мо Юй Цзян (沫郁酱)
> **Дата создания**: 2025-01-05
> **Последнее обновление**: 2026-03-21 (выравнивание правил конструктора типов с семантикой возврата из блоков кода)

## Аннотация

Данный RFC определяет окончательный вариант **синтаксиса определения функций** в языке YaoXiang. Используется унифицированный синтаксис `name: (params) -> Return = body`, полностью соответствующий модели `name: type = value` из RFC-010.

Для предотвращения неоднозначности: при наличии входных параметров у функции тип параметра должен быть явно указан либо в «сигнатуре», либо в «заголовке lambda», как минимум в одном из этих мест; одновременное опускание в обоих местах не допускается.

Последнее выражение в блоке кода `{ ... }` используется в качестве возвращаемого значения; пустой блок `{}` возвращает `Void`.

## Мотивация

### Зачем нужна эта функция?

1. **Синтаксическая согласованность**: устранение исторического багажа старого синтаксиса, унификация стиля
2. **Лаконичность**: алгоритм HM автоматически выводит типы, сокращая шаблонный код
3. **Типобезопасность**: алгоритм HM гарантирует типобезопасность, явное указание требуется только когда вывод невозможен
4. **Зрелость языка**: алгоритм HM — это зрелое решение современных функциональных языков

### Унифицированная модель синтаксиса

**Основной принцип**: `name: Signature = LambdaBody`

- **Полная форма**: сигнатура (с именами параметров + типы + `->` + тип возврата) + заголовок lambda (с именами параметров)
- **Правила сокращения**: опускание допускается везде, где это не приводит к неоднозначности
  - `->` нельзя опускать (это маркер функционального типа, иначе будет распознано как кортеж)
  - **При наличии входных параметров** тип параметра должен быть явно указан хотя бы в одном месте — в сигнатуре или в заголовке lambda
  - Заголовок lambda можно опустить → если в сигнатуре уже объявлены имена и типы параметров
  - Тип возврата можно указать явно или опустить, если он выводится

```yaoxiang
# Полная форма (полная сигнатура + полный заголовок lambda)
add: (a: Int, b: Int) -> Int = (a, b) => a + b

# Сокращение: опущен заголовок lambda (сигнатура уже объявляет параметры)
add: (a: Int, b: Int) -> Int = a + b

# Сокращение: опущена сигнатура (заголовок lambda указывает типы параметров)
add = (a: Int, b: Int) => a + b

# ❌ Ошибка: типы параметров не указаны нигде
# add = (a, b) => a + b
```

### Цели проектирования

```yaoxiang
# === Полная форма ===
add: (a: Int, b: Int) -> Int = (a, b) => { a + b }

# === Сокращённые формы ===
add: (a: Int, b: Int) -> Int = a + b                 # опущен заголовок lambda
add = (a: Int, b: Int) => a + b                      # опущена сигнатура

# === Функции без параметров ===
main: () -> Void = () => { println("Hello") }          # полная форма
main: () -> Void = { println("Hello") }                # опущен заголовок lambda
main = { println("Hello") }                            # минимальная форма (выводится как () -> Void)

# === Обобщённые функции (с использованием унифицированного синтаксиса RFC-010) ===
identity: (T: Type) -> ((x: T) -> T) = (x) => x         # полная форма
identity: (T: Type) -> ((x: T) -> T) = x                # опущен заголовок lambda
identity = (x: T) => x                                  # опущена сигнатура (заголовок lambda указывает тип)

# === Рекурсивные функции ===
factorial: (n: Int) -> Int = (n) => {
    if n <= 1 { 1 } else { n * factorial(n - 1) }
}
```

### Синтаксические правила

| Ситуация | Синтаксис | Описание |
|------|------|----------|
| **Полная форма** | `name: (a: Type, b) -> Ret = (a, b) => { return ... }` | Полная сигнатура + полный заголовок lambda |
| **Опущен заголовок lambda** | `name: (a: Type, b: Type) -> Ret = { ... }` | Сигнатура объявляет параметры |
| **Опущена сигнатура** | `name = (a: Type, b: Type) => { ... }` | Заголовок lambda указывает типы параметров |
| **Пустые параметры, полная** | `name: () -> Void = () => { return ... }` | Полная форма для функции без параметров |
| **Пустые параметры, сокращённая** | `name: () -> Void = { return ... }` | Опущен заголовок lambda |
| **Пустые параметры, минимальная** | `name = { return ... }` | Минимальная форма без параметров и возврата |

**Примечание**: последнее выражение в блоке `{ ... }` используется как возвращаемое значение; для досрочного завершения используется `return`. Пустой блок `{}` выводится как `Void`.

**Примечание**: `->` является маркером функционального типа и не может быть опущен (иначе будет распознано как кортеж).

**Важно**: выражение `if` использует фигурные скобки `{}` для ветвей и не поддерживает ключевые слова `then/else`:

```yaoxiang
# Правильно: использование фигурных скобок
if n <= 1 { return 1 } else { return n * factorial(n - 1) }

# Неправильно: ключевые слова then/else не поддерживаются
# if n <= 1 then return 1 else return n * factorial(n - 1)
```

## Предложение

### Алгоритм HM и поддержка полиморфизма высших порядков

**Основная возможность**: алгоритм HM поддерживает полиморфизм высших порядков (Higher-rank polymorphism) через аннотации обобщённых типов

**Принцип проектирования**:
- **Функции высшего порядка**: при передаче функций в качестве параметров необходимо ограничить их функциональный тип обобщениями
- **Форма аннотации типа**: `(T: Type) -> ((f: (T) -> T, x: T) -> T)` — обобщённый параметр ограничивает функциональный тип
- **Рабочий процесс HM**: вывод функционального типа через обобщённые параметры, реализация полиморфной композиции функций

**Пояснение примеров**:

```yaoxiang
# ✅ Поддержка полиморфизма высших порядков: обобщённое ограничение параметра функционального типа
call_twice: (T: Type) -> ((f: (T) -> T, x: T) -> T) = {
    return f(f(x))
}
# Использование: call_twice((x) => x + 1, 5)  # выводится T=Int

compose: (A: Type, B: Type, C: Type) -> ((f: (B) -> C, g: (A) -> B, x: A) -> C) = {
    return f(g(x))
}
# Использование: compose((x) => x * 2, (x) => x + 1, 5)  # выводится A=Int, B=Int, C=Int

# ❌ Не поддерживается: функция высшего порядка без обобщённого ограничения
# bad_hof: (f, x) => f(f(x))  # HM не может вывести, отсутствует обобщённый параметр
```

**Процесс вывода по HM**:

1. Идентификация параметров функции высшего порядка: `f: (T) -> T`
2. Создание обобщённого ограничения: `(T: Type)`
3. Вывод конкретного типа через обобщённую инстанциацию
4. Реализация полиморфной композиции функций

### Синтаксические правила для Lambda-выражений

**Важное правило**: блок кода `{ ... }` возвращает значение последнего выражения; для досрочного завершения используется `return`. Пустой блок `{}` возвращает `Void`.

| Синтаксическая форма | Синтаксис | Способ возврата |
|---------|------|----------|
| **Форма блока кода** | `{ statements }` | Последнее выражение как возвращаемое значение; можно использовать `return` для досрочного возврата |
| **Форма выражения** | `expression` | Непосредственный возврат значения выражения |

**Примеры**:

```yaoxiang
main: () -> Void = { println("Hello") }         # возвращает Void (последнее выражение — println)
add: (a: Int, b: Int) -> Int = { a + b }        # возвращает Int (последнее выражение — a + b)
empty: () -> Void = {}                          # пустой блок возвращает Void

# Досрочный возврат: использование return
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    n * factorial(n - 1)
}

# Форма выражения: непосредственный возврат (без return)
add: (a: Int, b: Int) -> Int = a + b            # правильно: форма выражения
main: () -> Void = println("Hello")               # правильно: форма выражения
```

**Ключевая идея**:

1. Определения функций используют алгоритм HM для вывода типов, стараясь вывести автоматически, при невозможности — явная ошибка
2. **Принцип работы алгоритма HM**: автоматический вывод типов через контекстную информацию — ограничения типов операторов, связи вызовов функций и т.д.
3. **Поддержка обобщений**: полиморфные функции используют обобщённый синтаксис `(T: Type)` для явного ограничения параметров типа (RFC-010/011)
4. **Границы вывода**: тип возврата и локальные переменные выводятся; типы параметров функций с параметрами должны быть явно указаны (в сигнатуре или заголовке lambda — хотя бы в одном)
5. Функции без параметров и без возврата используют `name: () -> Void = { ... }`, унифицировано с RFC-010
6. Старый синтаксис выводится из эксплуатации, предоставляется инструмент миграции

**Примеры вывода типов**:

```yaoxiang
# Обобщённые функции: явные параметры типа (использование унифицированного синтаксиса RFC-010)
identity: (T: Type) -> ((x: T) -> T) = x
map: (T: Type, R: Type) -> ((f: (T) -> R, list: List(T)) -> List(R)) = {
    result = List(R)()
    for item in list { result.push(f(item)) }
    return result
}

# Полиморфные функции: определение через явные обобщённые ограничения (RFC-010/011)
add: (T: Add) -> ((a: T, b: T) -> T) = a + b
print_sum: (a: Int, b: Int) -> Void = { println(a + b) }  # выводится как (Int, Int) -> Void

# Полиморфизм высших порядков: реализация поддержки HM через обобщённые аннотации типов
call_twice: (T: Type) -> ((f: (T) -> T, x: T) -> T) = { return f(f(x)) }
compose: (A: Type, B: Type, C: Type) -> ((f: (B) -> C, g: (A) -> B, x: A) -> C) = { return f(g(x)) }
```

```yaoxiang
# === Определение функций: вывод типов по алгоритму HM ===

# Стандартная функция: алгоритм HM выводит тип возврата (типы параметров требуют явного указания)
add = (a: Int, b: Int) => a + b            # выводится как (a: Int, b: Int) -> Int
main = { println("Hello") }                # выводится как () -> Void

# Частично явные параметры: алгоритм HM выводит остальное
print_sum: (a: Int, b: Int) -> Void = { println(a + b) }  # выводится как (Int, Int) -> Void
greet: (name: String) -> Void = { println("Hello " + name) }  # выводится как (String) -> Void

# Обобщённые функции: явное ограничение параметров полиморфного типа (использование унифицированного синтаксиса RFC-010)
identity: (T: Type) -> ((x: T) -> T) = x
map: (T: Type, R: Type) -> ((f: (T) -> R, list: List(T)) -> List(R)) = {
    # реализация функции map
    return List(R)()
}

# Рекурсивные функции: вывод через алгоритм HM и рекурсивные ограничения
factorial: (n: Int) -> Int = {
    if n <= 1 { 1 } else { n * factorial(n - 1) }
}

# === Присваивание переменных: вывод типов по алгоритму HM ===

# Явный тип
x: Int = 42

# Алгоритм HM автоматически выводит Int
y = 42                               # выводится как Int

# Алгоритм HM автоматически выводит String
name = "YaoXiang"                    # выводится как String

# Алгоритм HM автоматически выводит Float
pi = 3.14159                         # выводится как Float
```

**Правила вывода типов по HM**:

| Ситуация | Синтаксис | Опускаемые части | Пример |
|------|------|----------|------|
| **Полная форма** | `name: (a: Type, b: Type) -> Ret = (a, b) => ...` | нет | Полная сигнатура + полный заголовок lambda |
| **Опущен заголовок lambda** | `name: (a: Type, b: Type) -> Ret = ...` | Заголовок lambda | Сигнатура объявляет параметры |
| **Опущена сигнатура** | `name = (a: Type, b: Type) => ...` | Сигнатура | Заголовок lambda предоставляет типы параметров |
| **Опущен возвращаемый тип Ret** | `name: (a: Type, b: Type) -> = ...` | Тип возврата | HM выводит тип возврата |
| **Пустые параметры, полная** | `name: () -> Void = () => { ... }` | нет | Полная форма для функции без параметров |
| **Пустые параметры, сокращённая** | `name: () -> Void = { ... }` | Заголовок lambda | Опущено `() =>` |
| **Пустые параметры, минимальная** | `name = { ... }` | всё | Минимальная форма без параметров и возврата |
| **Присваивание переменной** | `name = value` | тип | HM выводит тип |
| **Явная переменная** | `name: Type = value` | нет | Явная аннотация типа |

**Основные принципы**:

- `->` является маркером функционального типа и не может быть опущен (иначе будет распознано как кортеж)
- Тип возврата `Ret` можно опустить, HM выведет его из тела функции
- При наличии входных параметров тип параметра должен быть явно указан (хотя бы в сигнатуре или в заголовке lambda)
- Остальные части можно опускать, если они выводятся и это не приводит к неоднозначности
- Нет неявных преобразований типов, избегая хаоса как в JavaScript

## Детальное проектирование

### Раскрытие синтаксического сахара

Независимо от степени сокращения, всё в итоге нормализуется к унифицированному промежуточному представлению:

```rust
// Полная форма
add: (a: Int, b: Int) -> Int = (a, b) => a + b

// Раскрытие в IR
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    return a + b
};

// Опущен заголовок lambda
add: (a: Int, b: Int) -> Int = a + b

// Раскрытие в IR (идентично полной форме)
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    return a + b
};

// Опущена сигнатура (заголовок lambda указывает типы параметров)
add = (a: Int, b: Int) => a + b

// Раскрытие в IR
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    a + b
};
```

### Определение синтаксиса

```bnf
function_def ::= identifier ':' type_expr '=' expression
               | identifier '=' expression
               | identifier '=' block                    # минимальная форма: без параметров и возврата

identifier ::= [a-zA-Z_][a-zA-Z0-9_]*

type_expr ::= identifier                     # ссылка на тип
       | '()'                          # пустой тип
       | '(' parameters ')' '->' type_expr   # функциональный тип (имена параметров в сигнатуре)
       | type_expr '->' type_expr            # простой функциональный тип
       | identifier '(' type_expr (',' type_expr)* ')'  # применение типа

expression ::= '(' parameters ')' '=>' block
             | '(' ')' '=>' block
             | '(' parameters ')' '=>' expression

parameters ::= parameter (',' parameter)*
parameter ::= identifier                # вывод типа
            | identifier ':' type_expr      # частично явный тип

block ::= '{' statement (',' statement)* '}'
        | expression

statement ::= identifier ':' expression  # оператор присваивания
           | expression                  # выражение-оператор (выполняется, но не возвращает)
           | 'return' expression         # оператор возврата (возвращает указанное значение)

# Примечание: блок кода возвращает значение последнего выражения; пустой блок {} выводится как Void
# Например: { 1 + 1 } возвращает Int; { println("Hello") } возвращает Void
# Примечание: обобщённые параметры используют синтаксис (T: Type), как часть функционального типа,
# не требуют отдельного BNF-правила
```

### Обработка ошибок

```yaoxiang
# === Примеры ошибок компиляции ===

# Ошибка 1: несоответствие типа возврата блока кода
add: (a: Int, b: Int) -> Int = { println(a + b) }
// Ошибка: последнее выражение блока — println(...), возвращает Void, но сигнатура ожидает Int
// Правильно: add: (a: Int, b: Int) -> Int = a + b
// Или: add: (a: Int, b: Int) -> Int = { a + b }

# Ошибка 2: использование необъявленного параметра типа
identity: (x: T) -> T = x
// Ошибка: T не объявлен; требуется явный обобщённый параметр (RFC-010)
// Правильно: identity: (T: Type) -> ((x: T) -> T) = x

# Правильно: алгоритм HM выводит тип возврата
double = (x: Int) => x + x

# Полная форма (пошапеное сокращение)
double: (x: Int) -> Int = (x) => x + x                # полная
double: (x: Int) -> Int = x + x                       # опущен заголовок lambda
double = (x: Int) => x + x                            # опущен тип возврата (выводится HM)
# double = (x) => x + x                               # ❌ Типы параметров нельзя опускать с обеих сторон
```

## Компромиссы

### Преимущества

- **Единый синтаксис**: модель `name: Signature = LambdaBody` охватывает все сценарии
- **Гибкие сокращения**: любая часть может быть опущена при возможности вывода по HM
- **Типобезопасность**: алгоритм HM гарантирует типобезопасность, отсутствие неявных преобразований типов
- **Поддержка рекурсии**: алгоритм HM и рекурсивные ограничения автоматически выводят типы
- **Нулевая нагрузка**: плавный переход от полной формы к минимальной

### Недостатки

- **Стоимость миграции**: старый код требует инструмента конвертации
- **Стоимость обучения**: требуется понимание модели «полная форма + произвольные сокращения»

## Альтернативные решения

| Решение | Описание | Почему не выбрано |
|------|------|-----------|
| Вывод типов по алгоритму HM | Использование алгоритма Hindley-Milner для вывода типов | ✅ **Принято**, стандарт современных функциональных языков |
| Явные объявления типов | Все типы должны быть явно указаны | Противоречит принципу упрощения синтаксиса, увеличивает шаблонный код |
| Сохранение старого синтаксиса | Поддержка старого и нового синтаксиса одновременно | Раскол синтаксиса, высокая стоимость поддержки |
| Ключевое слово fn | Введение fn для различения функций и переменных | Противоречит дизайну «функция = lambda» |

   - Реализация нового синтаксиса `name = lambda` + вывод типов по алгоритму HM
   - Реализация заполнения по умолчанию для функций без параметров и без возврата

2. **Фаза 2: Инструмент миграции** (v0.3)
   - Разработка инструмента `yaoxiang-migrate --old-to-new`
   - Автоматическая конвертация кода со старым синтаксисом

3. **Фаза 3: Верификация и документация** (v0.3)
   - Верификация завершения миграции старого кода
   - Обновление документации

### Инструмент миграции

```bash
# Миграция одного файла
yaoxiang-migrate --old-to-new src/main.yaoxiang

# Миграция всего проекта
yaoxiang-migrate --old-to-new --recursive src/

# Предпросмотр миграции (без изменения файлов)
yaoxiang-migrate --old-to-new --dry-run src/main.yaoxiang
```

Правила миграции:

```yaoxiang
# Старый синтаксис
add(Int, Int) -> Int = (a, b) => { a + b }
main() -> Int = { println("Hello"); 0 }
main() = { println("Hello") }

# === Новый синтаксис: полная форма (полная сигнатура + полный заголовок lambda) ===
add: (a: Int, b: Int) -> Int = (a, b) => a + b
main: () -> Void = () => { println("Hello") }

# === Сокращение: опущен заголовок lambda ===
add: (a: Int, b: Int) -> Int = a + b
main: () -> Void = { println("Hello") }

# === Сокращение: вывод по HM ===
add = (a: Int, b: Int) => a + b              # выводится как (a: Int, b: Int) -> Int
main = { println("Hello") }                  # выводится как () -> Void

# === Минимальная форма ===
main = {                                      # эквивалентно main: () -> Void = { ... }
    println("Hello")
}
```

### Зависимости

- Нет внешних зависимостей
- Может быть реализовано независимо

### Риски

| Риск | Влияние | Меры снижения |
|------|------|---------|
| Пропуск при миграции | Ошибка компиляции старого кода | Предоставление инструмента миграции, покрывающего все паттерны старого синтаксиса |
| Ошибки парсера | Нестабильность синтаксического анализа | Достаточное тестовое покрытие |

## Открытые вопросы

> Следующие вопросы уже решены в процессе проектирования, записаны в Приложении A.

- ~~Q1: Следует ли сохранить минимальную форму записи `main() = body`?~~ → Решено: сохранено как `main = { ... }`
- ~~Q2: Сохраняется ли `:` после имени функции?~~ → Решено: опционально; но для функций с параметрами тип параметра всё ещё должен быть указан в сигнатуре или заголовке lambda
- ~~Q3: Поддерживает ли алгоритм HM вывод типов параметров?~~ → Решено: тип возврата/локальные переменные выводятся; типы параметров функций с параметрами требуют явного указания
- ~~Q4: Вводится ли ключевое слово `fn`?~~ → Решено: не вводится, функция = lambda
- ~~Q5: Какова стратегия миграции старого кода?~~ → Решено: предоставляется инструмент `yaoxiang-migrate`
- ~~Q6: Как использовать обобщённые функции?~~ → Решено: используется унифицированный синтаксис RFC-010 `(T: Type)`

## Приложения

### Приложение A: Справочник синтаксиса определения функций в различных языках

| Язык | Стиль синтаксиса | Особенности |
|------|---------|------|
| Rust | `fn add(a: i32, b: i32) -> i32 { ... }` | Ключевое слово + аннотации типов |
| Haskell | `add a b = ...` / `add :: Int -> Int -> Int` | Разделение сигнатуры типа |
| OCaml | `let add a b = ...` | Типы параметров можно опускать |
| MoonBit | `fn add(a: Int, b: Int): Int { ... }` | Компактные аннотации типов |
| TypeScript | `const add = (a: number, b: number): number => ...` | Стиль lambda |
| Scala | `def add(a: Int, b: Int): Int = { ... }` | Ключевое слово def |
| **YaoXiang** | `name = (a: Int, b: Int) => a + b` | **Функция = lambda, HM выводит возвращаемое значение** |

### Приложение C: Глоссарий

| Термин | Определение |
|------|------|
| Алгоритм HM | Алгоритм вывода типов Hindley-Milner, автоматически выводит типы функций и переменных |
| Обобщения | Использование параметров типа `(T: Type)` для ограничения полиморфных функций, например `identity: (T: Type) -> ((x: T) -> T) = x` (RFC-010) |
| Заполнение типа по умолчанию | Функции без параметров и без возврата опускают `-> Void`, компилятор автоматически заполняет |
| Синтаксический сахар | Синтаксическое упрощение для улучшения читаемости кода |
| Нормализация | Преобразование синтаксических форм в унифицированное внутреннее представление |
| Функция = lambda | Функция по сути является lambda-переменной, тип автоматически выводится по алгоритму HM |
