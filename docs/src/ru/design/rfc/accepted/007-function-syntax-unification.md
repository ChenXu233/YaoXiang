# RFC-007: Унифицированная схема синтаксиса определения функций

## Аннотация

Данный RFC определяет финальный вариант **синтаксиса определения функций** для языка YaoXiang. Используется унифицированный синтаксис `name: (params) -> Return = body`, полностью соответствующий модели `name: type = value` из RFC-010.

Во избежание неоднозначности: при наличии входных параметров у функции тип параметра должен быть явно указан либо в «сигнатуре», либо в «заголовке lambda» (как минимум в одном из этих мест); пропуск типа с обеих сторон не допускается.

Внутри блока кода `{ ... }` необходимо использовать `return` для возврата значения; при отсутствии `return` по умолчанию возвращается `Void`. Форма выражения `= expr` возвращает значение напрямую.

## Мотивация

### Зачем нужна эта функциональность?

1. **Синтаксическая согласованность**: устранение исторического багажа старого синтаксиса, унификация стиля
2. **Краткость**: алгоритм HM автоматически выводит типы, сокращая шаблонный код
3. **Типовая безопасность**: алгоритм HM гарантирует типобезопасность; явное указание типов требуется только когда вывод невозможен
4. **Зрелость языка**: алгоритм HM — проверенное решение современных функциональных языков

### Унифицированная модель синтаксиса

**Основной принцип**: `name: Signature = LambdaBody`

- **Полная форма**: сигнатура (с именем параметра + тип + `->` + тип возврата) + заголовок lambda (с именем параметра)
- **Правила сокращения**: опускание допускается везде, где это не создаёт неоднозначности
  - `->` нельзя опускать (маркер типа функции; иначе будет распознано как кортеж)
  - **При наличии входных параметров** тип параметра должен быть явно указан хотя бы в сигнатуре или в заголовке lambda
  - Заголовок lambda можно опустить → если в сигнатуре уже объявлены имя и тип параметра
  - Тип возврата можно указать явно или опустить при возможности вывода

```yaoxiang
# Полная форма (сигнатура полная + заголовок lambda полный)
add: (a: Int, b: Int) -> Int = (a, b) => a + b

# Сокращение: опущен заголовок lambda (сигнатура уже объявляет параметры)
add: (a: Int, b: Int) -> Int = a + b

# Сокращение: опущена сигнатура (заголовок lambda указывает типы параметров)
add = (a: Int, b: Int) => a + b

# ❌ Ошибка: тип параметра не указан нигде
# add = (a, b) => a + b
```

### Цели проектирования

```yaoxiang
# === Полная форма ===
add: (a: Int, b: Int) -> Int = (a, b) => { a + b }

# === Сокращённые формы ===
add: (a: Int, b: Int) -> Int = a + b                 # опущен заголовок lambda
add = (a: Int, b: Int) => a + b                      # опущена сигнатура

# === Функции без параметров ===
main: () -> Void = () => { println("Hello") }          # полная форма
main: () -> Void = { println("Hello") }                # опущен заголовок lambda
main = { println("Hello") }                            # минимальная форма (выводится как () -> Void)

# === Обобщённые функции (с использованием унифицированного синтаксиса RFC-010) ===
identity: (T: Type) -> ((x: T) -> T) = (x) => x         # полная форма
identity: (T: Type) -> ((x: T) -> T) = x                # опущен заголовок lambda
identity = (x: T) => x                                  # опущена сигнатура (заголовок lambda указывает тип)

# === Рекурсивные функции ===
factorial: (n: Int) -> Int = (n) => {
    if n <= 1 { return 1 } else { return n * factorial(n - 1) }
}
```

### Правила синтаксиса

| Сценарий | Синтаксис | Описание |
|------|------|----------|
| **Полная форма** | `name: (a: Type, b) -> Ret = (a, b) => { return ... }` | Сигнатура + полный заголовок lambda |
| **Опущен заголовок lambda** | `name: (a: Type, b: Type) -> Ret = { ... }` | Сигнатура уже объявляет параметры |
| **Опущена сигнатура** | `name = (a: Type, b: Type) => { ... }` | Заголовок lambda указывает типы параметров |
| **Без параметров, полная** | `name: () -> Void = () => { return ... }` | Функция без параметров, полная форма |
| **Без параметров, сокращённая** | `name: () -> Void = { return ... }` | Опущен заголовок lambda |
| **Без параметров, минимальная** | `name = { return ... }` | Минимальная форма без параметров и возврата |

**Примечание**: внутри блока кода `{ ... }` необходимо использовать `return` для возврата значения; при отсутствии `return` по умолчанию возвращается `Void`. Форма выражения `= expr` возвращает значение напрямую.

**Примечание**: `->` является маркером типа функции, его нельзя опускать (иначе будет распознано как кортеж).

**Важно**: выражение `if` использует фигурные скобки `{}` для веток и не поддерживает ключевые слова `then/else`:

```yaoxiang
# Правильно: использовать фигурные скобки
if n <= 1 { return 1 } else { return n * factorial(n - 1) }

# Неправильно: ключевые слова then/else не поддерживаются
# if n <= 1 then return 1 else return n * factorial(n - 1)
```

## Предложение

### Алгоритм HM и поддержка полиморфизма высших порядков

**Основная функциональность**: алгоритм HM обеспечивает поддержку полиморфизма высших порядков (Higher-rank polymorphism) через аннотации обобщённых типов.

**Принцип проектирования**:
- **Функции высших порядков**: при передаче функции в качестве параметра необходимо ограничить её тип обобщённым параметром
- **Форма аннотации типа**: `(T: Type) -> ((f: (T) -> T, x: T) -> T)` — обобщённый параметр ограничивает тип функции
- **Рабочий процесс HM**: вывод типа функции через обобщённые параметры, реализация полиморфной композиции функций

**Пояснение примеров**:
```yaoxiang
# ✅ Поддержка полиморфизма высших порядков: обобщённое ограничение параметра типа функции
call_twice: (T: Type) -> ((f: (T) -> T, x: T) -> T) = {
    return f(f(x))
}
# Использование: call_twice((x) => x + 1, 5)  # выводится T=Int

compose: (A: Type, B: Type, C: Type) -> ((f: (B) -> C, g: (A) -> B, x: A) -> C) = {
    return f(g(x))
}
# Использование: compose((x) => x * 2, (x) => x + 1, 5)  # выводится A=Int, B=Int, C=Int

# ❌ Не поддерживается: функция высших порядков без обобщённого ограничения
# bad_hof: (f, x) => f(f(x))  # HM не может вывести, отсутствует обобщённый параметр
```

**Процесс вывода по HM**:
1. Распознавание параметра функции высшего порядка: `f: (T) -> T`
2. Создание обобщённого ограничения: `(T: Type)`
3. Вывод конкретного типа через обобщённую инстанциацию
4. Реализация полиморфной композиции функций

### Правила синтаксиса lambda-выражений

**Важное правило**: внутри блока кода `{ ... }` необходимо использовать `return` для возврата значения; при отсутствии `return` по умолчанию возвращается `Void`. Форма выражения `= expr` возвращает значение напрямую.

| Форма синтаксиса | Синтаксис | Способ возврата |
|---------|------|----------|
| **Форма блока кода** | `{ statements }` | Обязательно использовать `return`; при отсутствии `return` — `Void` по умолчанию |
| **Форма выражения** | `expression` | Значение выражения возвращается напрямую |

**Примеры**:
```yaoxiang
main: () -> Void = { println("Hello") }         # возвращает Void (нет return)
add: (a: Int, b: Int) -> Int = { return a + b }  # возвращает Int (явный return)
empty: () -> Void = {}                          # пустой блок возвращает Void по умолчанию

# Ранний возврат: использование return
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Форма выражения: возврат значения напрямую (без return)
add: (a: Int, b: Int) -> Int = a + b            # правильно: форма выражения
main: () -> Void = println("Hello")               # правильно: форма выражения
```

**Ключевая идея**:
1. Определение функций выполняется через вывод типов по алгоритму HM; при невозможности вывода выдаётся явная ошибка
2. **Принцип работы алгоритма HM**: автоматический вывод типов через контекстную информацию (ограничения типов операторов, связи вызовов функций и т.д.)
3. **Поддержка обобщений**: полиморфные функции используют обобщённый синтаксис `(T: Type)` для явного ограничения параметров типа (RFC-010/011)
4. **Границы вывода**: тип возврата и локальные переменные могут быть выведены; типы параметров функций с параметрами должны быть явно указаны (в сигнатуре или заголовке lambda — хотя бы в одном)
5. Функции без параметров без возврата используют `name: () -> Void = { ... }`, унифицировано с RFC-010
6. Старый синтаксис выводится из эксплуатации, предоставляется инструмент миграции

**Примеры вывода типов**:
```yaoxiang
# Обобщённые функции: явные параметры типа (использование унифицированного синтаксиса RFC-010)
identity: (T: Type) -> ((x: T) -> T) = x
map: (T: Type, R: Type) -> ((f: (T) -> R, list: List(T)) -> List(R)) = {
    result = List(R)()
    for item in list { result.push(f(item)) }
    return result
}

# Полиморфные функции: определение через явные обобщённые ограничения (RFC-010/011)
add: (T: Add) -> ((a: T, b: T) -> T) = a + b
print_sum: (a: Int, b: Int) -> Void = { println(a + b) }  # выводится как (Int, Int) -> Void

# Полиморфизм высших порядков: реализация поддержки HM через обобщённые аннотации типов
call_twice: (T: Type) -> ((f: (T) -> T, x: T) -> T) = { return f(f(x)) }
compose: (A: Type, B: Type, C: Type) -> ((f: (B) -> C, g: (A) -> B, x: A) -> C) = { return f(g(x)) }
```

```yaoxiang
# === Определение функций: вывод типов по алгоритму HM ===

# Стандартные функции: HM выводит тип возврата (типы параметров требуют явного указания)
add = (a: Int, b: Int) => a + b            # выводится как (a: Int, b: Int) -> Int
main = { println("Hello") }                # выводится как () -> Void

# Частично явные параметры: HM выводит остальное
print_sum: (a: Int, b: Int) -> Void = { println(a + b) }  # выводится как (Int, Int) -> Void
greet: (name: String) -> Void = { println("Hello " + name) }  # выводится как (String) -> Void

# Обобщённые функции: явное ограничение параметров полиморфного типа (использование унифицированного синтаксиса RFC-010)
identity: (T: Type) -> ((x: T) -> T) = x
map: (T: Type, R: Type) -> ((f: (T) -> R, list: List(T)) -> List(R)) = {
    # Реализация функции map
    return List(R)()
}

# Рекурсивные функции: вывод через HM и рекурсивные ограничения
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 } else { return n * factorial(n - 1) }
}

# === Присваивание переменных: вывод типов по алгоритму HM ===

# Явный тип
x: Int = 42

# HM автоматически выводит Int
y = 42                               # выводится как Int

# HM автоматически выводит String
name = "YaoXiang"                    # выводится как String

# HM автоматически выводит Float
pi = 3.14159                         # выводится как Float
```

**Правила вывода типов по HM**:

| Сценарий | Синтаксис | Опускаемые части | Пример |
|------|------|----------|------|
| **Полная форма** | `name: (a: Type, b: Type) -> Ret = (a, b) => ...` | Нет | Сигнатура + полный заголовок lambda |
| **Опущен заголовок lambda** | `name: (a: Type, b: Type) -> Ret = ...` | Заголовок lambda | Сигнатура уже объявляет параметры |
| **Опущена сигнатура** | `name = (a: Type, b: Type) => ...` | Сигнатура | Заголовок lambda предоставляет типы параметров |
| **Опущен тип возврата Ret** | `name: (a: Type, b: Type) -> = ...` | Тип возврата | HM выводит тип возврата |
| **Без параметров, полная** | `name: () -> Void = () => { ... }` | Нет | Функция без параметров, полная форма |
| **Без параметров, сокращённая** | `name: () -> Void = { ... }` | Заголовок lambda | Опущен `() =>` |
| **Без параметров, минимальная** | `name = { ... }` | Всё | Минимальная форма без параметров и возврата |
| **Присваивание переменной** | `name = value` | Тип | HM выводит тип |
| **Явная переменная** | `name: Type = value` | Нет | Явная аннотация типа |

**Основные принципы**:
- `->` является маркером типа функции, его нельзя опускать (иначе будет распознано как кортеж)
- Тип возврата `Ret` можно опустить; HM выведет его из тела функции
- При наличии входных параметров тип параметра должен быть явно указан (в сигнатуре или заголовке lambda — хотя бы в одном)
- Остальные части можно опускать при возможности вывода и отсутствии неоднозначности
- Нет неявных преобразований типов, избегая хаоса как в JavaScript

## Детальное проектирование

### Раскрытие синтаксического сахара

Независимо от степени сокращения, всё最终规范化到统一内部表示：

```rust
// Полная форма
add: (a: Int, b: Int) -> Int = (a, b) => a + b

// Раскрытый IR
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    return a + b
};

// Опущен заголовок lambda
add: (a: Int, b: Int) -> Int = a + b

// Раскрытый IR (идентичен полной форме)
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    return a + b
};

// Опущена сигнатура (заголовок lambda указывает типы параметров)
add = (a: Int, b: Int) => a + b

// Раскрытый IR
let add: (Int, Int) -> Int = |a: Int, b: Int| -> Int {
    a + b
};
```

### Определение синтаксиса

```bnf
function_def ::= identifier ':' type_expr '=' expression
               | identifier '=' expression
               | identifier '=' block                    # Минимальная форма: без параметров и возврата

identifier ::= [a-zA-Z_][a-zA-Z0-9_]*

type_expr ::= identifier                     # Ссылка на тип
       | '()'                          # Пустой тип
       | '(' parameters ')' '->' type_expr   # Функциональный тип (имена параметров в сигнатуре)
       | type_expr '->' type_expr            # Простой функциональный тип
       | identifier '(' type_expr (',' type_expr)* ')'  # Применение типа

expression ::= '(' parameters ')' '=>' block
             | '(' ')' '=>' block
             | '(' parameters ')' '=>' expression

parameters ::= parameter (',' parameter)*
parameter ::= identifier                # Вывод типа
            | identifier ':' type_expr      # Частично явный тип

block ::= '{' statement (',' statement)* '}'
        | expression

statement ::= identifier ':' expression  # Оператор присваивания
           | expression                  # Оператор-выражение (выполняется, но не возвращает)
           | 'return' expression         # Оператор возврата (возвращает указанное значение)

# Примечание: внутри блока кода обязателен return для возврата; при отсутствии return по умолчанию Void
# Например: { return 1 + 1 } возвращает Int; { println("Hello") } возвращает Void
# Примечание: обобщённые параметры используют синтаксис (T: Type), являются частью функционального типа, не требуют отдельного BNF-правила
```

### Обработка ошибок

```yaoxiang
# === Примеры ошибок компиляции ===

# Ошибка 1: Несовпадение типа возврата блока кода
add: (a: Int, b: Int) -> Int = { println(a + b) }
// Ошибка: в блоке нет return, по умолчанию Void, но сигнатура ожидает Int
// Правильно: add: (a: Int, b: Int) -> Int = a + b
// Или: add: (a: Int, b: Int) -> Int = { return a + b }

# Ошибка 2: Использование необъявленного параметра типа
identity: (x: T) -> T = x
// Ошибка: T не объявлен; требуется явный обобщённый параметр (RFC-010)
// Правильно: identity: (T: Type) -> ((x: T) -> T) = x

# Правильно: HM выводит тип возврата
double = (x: Int) => x + x

# Полная форма (постепенное сокращение)
double: (x: Int) -> Int = (x) => x + x                # полная
double: (x: Int) -> Int = x + x                       # опущен заголовок lambda
double = (x: Int) => x + x                            # опущен тип возврата (HM выводит возвращаемое)
// double = (x) => x + x                               # ❌ Недопустимо: тип параметра нельзя опускать с обеих сторон
```

## Компромиссы

### Преимущества

- **Единый синтаксис**: модель `name: Signature = LambdaBody` охватывает все сценарии
- **Гибкие сокращения**: любую часть можно опустить при возможности вывода по HM
- **Типовая безопасность**: алгоритм HM гарантирует типобезопасность, отсутствие неявных преобразований
- **Поддержка рекурсии**: HM и рекурсивные ограничения автоматически выводят типы
- **Нулевая нагрузка**: плавный переход от полной к минимальной форме

### Недостатки

- **Стоимость миграции**: старый код требует преобразования инструментом миграции
- **Кривая обучения**: необходимо понимать модель «полная форма + произвольные сокращения»

## Альтернативные решения

| Решение | Описание | Почему не выбрано |
|------|------|-----------|
| Вывод типов по алгоритму HM | Использование алгоритма Hindley-Milner для вывода типов | ✅ **Принято**, стандарт современных функциональных языков |
| Явные типы везде | Все типы должны быть явно указаны | Противоречит принципу упрощения синтаксиса, увеличивает шаблонный код |
| Сохранение старого синтаксиса | Одновременная поддержка старого и нового синтаксиса | Расщепление синтаксиса, высокая стоимость поддержки |
| Ключевое слово fn | Введение fn для различения функций и переменных | Противоречит дизайну «функция — это lambda» |

### Инструмент миграции

```bash
# Миграция одного файла
yaoxiang-migrate --old-to-new src/main.yaoxiang

# Миграция всего проекта
yaoxiang-migrate --old-to-new --recursive src/

# Предпросмотр миграции (без изменения файлов)
yaoxiang-migrate --old-to-new --dry-run src/main.yaoxiang
```

Правила миграции:
```yaoxiang
# Старый синтаксис
add(Int, Int) -> Int = (a, b) => { a + b }
main() -> Int = { println("Hello"); 0 }
main() = { println("Hello") }

# === Новый синтаксис: полная форма (сигнатура полная + заголовок lambda полный)===
add: (a: Int, b: Int) -> Int = (a, b) => a + b
main: () -> Void = () => { println("Hello") }

# === Сокращение: опущен заголовок lambda ===
add: (a: Int, b: Int) -> Int = a + b
main: () -> Void = { println("Hello") }

# === Сокращение: вывод по HM ===
add = (a: Int, b: Int) => a + b              # выводится как (a: Int, b: Int) -> Int
main = { println("Hello") }                  # выводится как () -> Void

# === Минимальная форма ===
main = {                                      # эквивалентно main: () -> Void = { ... }
    println("Hello")
}
```

### Зависимости

- Нет внешних зависимостей
- Может быть реализовано независимо

### Риски

| Риск | Влияние | Меры снижения |
|------|------|---------|
| Пропуск при миграции | Ошибка компиляции старого кода | Предоставление инструмента миграции, охватывающего все паттерны старого синтаксиса |
| Ошибки парсера | Нестабильный синтаксический анализ | Достаточное тестовое покрытие |

## Открытые вопросы

> Следующие вопросы решены в процессе проектирования, записаны в Приложении A.

- ~~Q1: Следует ли сохранить сверхкраткую запись `main() = body`?~~ → Решено: сохранена как `main = { ... }`
- ~~Q2: Сохраняется ли `:` после имени функции?~~ → Решено: опционально; функции с параметрами всё равно должны указывать тип параметра в сигнатуре или заголовке lambda
- ~~Q3: Поддерживает ли HM вывод типов параметров?~~ → Решено: тип возврата/локальные типы выводятся; типы параметров функций с параметрами требуют явного указания
- ~~Q4: Вводится ли ключевое слово `fn`?~~ → Решено: не вводится, функция — это lambda
- ~~Q5: Какова стратегия миграции старого кода?~~ → Решено: предоставляется инструмент `yaoxiang-migrate`
- ~~Q6: Как использовать обобщённые функции?~~ → Решено: используется унифицированный синтаксис RFC-010 `(T: Type)`

### Приложение A: Справочник синтаксиса определения функций в различных языках

| Язык | Стиль синтаксиса | Особенности |
|------|---------|------|
| Rust | `fn add(a: i32, b: i32) -> i32 { ... }` | Ключевое слово + аннотации типов |
| Haskell | `add a b = ...` / `add :: Int -> Int -> Int` | Сигнатура типа отделена |
| OCaml | `let add a b = ...` | Типы параметров можно опускать |
| MoonBit | `fn add(a: Int, b: Int): Int { ... }` | Краткие аннотации типов |
| TypeScript | `const add = (a: number, b: number): number => ...` | Стиль Lambda |
| Scala | `def add(a: Int, b: Int): Int = { ... }` | Ключевое слово def |
| **YaoXiang** | `name = (a: Int, b: Int) => a + b` | **Функция = lambda, HM выводит возвращаемое** |

### Приложение C: Глоссарий

| Термин | Определение |
|------|------|
| Алгоритм HM | Алгоритм вывода типов Hindley-Milner, автоматически выводит типы функций и переменных |
| Обобщения | Использование параметров типа `(T: Type)` для ограничения полиморфных функций, например `identity: (T: Type) -> ((x: T) -> T) = x` (RFC-010) |
| Заполнение типа по умолчанию | Для функций без параметров без возврата можно опустить `-> Void`, компилятор заполняет автоматически |
| Синтаксический сахар | Синтаксическое упрощение для улучшения читаемости кода |
| Нормализация | Преобразование форм синтаксиса в унифицированное внутреннее представление |
| Функция — это lambda | Функция по сути является lambda-переменной, тип автоматически выводится по алгоритму HM |
