# RFC-009a: Анализ времени жизни токенов — конвейер доказательств на основе логики Хоара

> **Родительский RFC**: [RFC-009: Проект модели владения](../accepted/009-ownership-model.md)
>
> **Зависимости**: [RFC-027: Предикаты времени компиляции и унифицированная статическая верификация](../accepted/027-compile-time-evaluation-types.md)
>
> **Предусловие**: RFC-027 должен быть принят. Все механизмы настоящего RFC (конвейер доказательств, откат к SMT, сбор путевых условий) зависят от реализации RFC-027.
>
> **Настоящий RFC исправляет и заменяет раздел RFC-009 §«Обнаружение конфликтов токенов: потоково-чувствительный анализ живости» (строки 663–684).**

## Резюме

В строке 684 RFC-009 утверждается, что обнаружение конфликтов токенов «не требует……NLL». Вывод верен, аргументация ошибочна.

Дело не в том, что «токены — это значения, и линейного отслеживания достаточно». Дело в том, что **живость токенов — это утверждение логики Хоара, а не специальный потоково-чувствительный анализ.**

`{все conflicting_tokens мертвы} op {WriteToken безопасно получен}` — то же самое `{P} op {Q}`, разделяющее конвейер доказательств RFC-027 с проверкой типов и верификацией предикатов. Никаких новых фреймворков анализа. Один конвейер, множество утверждений.

---

## Мотивация

### Путаница в RFC-009

RFC-009 смешивает две задачи:

1. **Линейное отслеживание** (после Move недоступно) — `{v не перемещён} use(v) {тип совпадает}`. Уже есть в проверщике типов.
2. **Взаимодействие времён жизни токенов** (дочерний токен жив → родительский токен приостановлен → дочерний токен мёртв → родительский токен возрождается) — `{все conflicting_tokens мертвы} write(data) {безопасно}`. Требуется **анализ живости**, а не линейное отслеживание.

### Текущее состояние кода

| Компонент | Состояние |
|------|------|
| `BorrowChecker` | Линейное сканирование IR, пассивная реакция на явные инструкции `Borrow`/`Release` |
| `ControlFlowAnalyzer::analyze_instruction` | Пустая реализация (`control_flow.rs:145-153`) |
| `liveness_analysis` | Существует, но используется только для вставки Drop, не подключён к конфликтам токенов |
| Вставка Release | Жёстко закодирована после инструкции Call — чисто лексическая область видимости (`ir_gen.rs:2734-2736`) |

**Видимые пользователю последствия**:

```yaoxiang
data = vec![1, 2, 3]
view = &data              # создание ReadToken
x = view.total_count      # последнее использование view
data.push(4)              # ❌ Release(view) ещё не выполнен, ReadToken "жив"
```

### Зачем нужна переработка

Предыдущая версия (009a v1) использовала нарратив «DAG вместо NLL», вводя ненужные новые концепции (консервативные правила ветвления, специальную обработку циклов). Основное противоречие не было прояснено: **проверка заимствований — не отдельная система — это одно из утверждений Хоара.**

## Основной проект

### Всё есть Хоар

```
Проверка типов:     { x: Int }        x + 1        { result: Int }
Проверка заимствований: { view мёртв }  data.push(4)  { WriteToken успешно получен }
Верификация предикатов: { y > 0 }      divide(x, y)  { result: Int }
Обрезка обратных рёбер: { i == n }     следующая итерация цикла { cond == false }
```

Одна и та же форма `{P} op {Q}`. Компилятор генерирует предусловие P для каждой операции и отправляет его в конвейер доказательств для верификации.

**Проверка заимствований и пользовательские предикаты разделяют один конвейер.** Разница лишь в том, кто генерирует утверждение и что происходит при невозможности доказать.

### Два вида предикатов, один конвейер

| | Пользовательский предикат | Системный предикат (заимствования) |
|---|---|---|
| Генерация утверждений | Программист (типовые аннотации) | Компилятор (дерево брендов + правила владения) |
| Предоставление доказательства | Компилятор + программист | **Полностью автоматически компилятором** |
| При невозможности доказать | Написать функцию-доказательство или рефакторить | Рефакторить код (двери остаются, но нужны крайне редко) |
| Видимость | Видна в сигнатуре | Неявная, не загрязняет типовые сигнатуры |
| Стоимость обучения | Изучается при необходимости | Нулевая |

**Доказательство системных предикатов не открывает программисту функцию-доказательство — компилятор делает это полностью автоматически.** При невозможности доказать пользователь рефакторит код.

**Три режима отказа, один движок верификации.** Невозможно доказать утверждение типа → ошибка компиляции (нельзя обойти). Невозможно доказать утверждение заимствования → ошибка компиляции, рефакторинг кода (нельзя обойти). Невозможно доказать пользовательский предикат → ошибка компиляции, можно написать функцию-доказательство (можно обойти). Стратегии отказа различаются, но движок верификации один и тот же — SMT-решатель + правила вывода ядра компилятора. Разница лишь в том, «кто отвечает за дополнение доказательства при неудаче» — компилятор отказывается писать за программиста доказательство заимствования (стратегия доказательства утверждений заимствований — структурный анализ + SMT, вмешательство программиста не требуется), но принимает функции-доказательства, написанные программистом для пользовательских предикатов. Это не несоответствие конвейера — это разные границы ответственности для разных категорий утверждений.

Это отличается от Rust `'a`: `'a` — обязательный курс, функция-доказательство — факультативный — подавляющее большинство пользователей никогда в жизни не столкнётся с дверью факультатива.

### Утверждения заимствований: автоматическая генерация компилятором

Пользователь пишет `data.push(4)`. Компилятор автоматически генерирует утверждение:

```
WriteToken(data, node) может быть получен
  = forall t в conflicting_tokens(data): t в node уже мёртв
  = forall t в brand_tree.children(data): forward_reachable(node) ∩ consumers(t) == ∅
```

**Три правила, ноль особых случаев:**

1. **Дерево брендов** (RFC-009 §2.7) отвечает на вопрос «кто с кем конфликтует»: сопоставление префиксов, O(глубина), глубина ≤ 3
2. **Список потребителей** (автоматически собирается при построении DAG) отвечает на вопрос «кто последним использовал токен»
3. **Прямая достижимость** отвечает на вопрос «может ли потребитель быть выполнен»: структурная обрезка + логическая обрезка

### Прямая достижимость: обратное прохождение от потребителя

Для каждого потребителя C токена T:

```
Из C — обратный BFS по DAG.
Ребро обрезается, если:
  1. Это break (структурная обрезка)
  2. Условие пути ⇒ !loop_cond доказано SMT как истинное (логическая обрезка, конвейер RFC-027)

Распространяем активность в обратном направлении по всем необрезанным рёбрам
(включая обратные рёбра, которые переносят активность в предыдущую итерацию).
Помечаем все достижимые узлы → unsafe.
```

Запрос: операция записи в узле W → W ∉ unsafe → безопасно.

**Не нужно изобретать «консервативные правила ветвления». Не нужно «консервативной живости в цикле». Один обратный BFS + два правила обрезки.**

### Стратегия доказательства: быстрый канал в первую очередь, SMT как страховка

```
Каждая операция записи, требующая токен
  │
  ├→ Быстрый канал: структурный анализ DAG (покрывает 95%+ сценариев)
  │     │
  │     ├→ Сопоставление префиксов в дереве брендов → поиск конфликтующих токенов (O(глубина))
  │     ├→ Обратный BFS, break обрезает обратные рёбра
  │     └→ Нет проходимых обратных рёбер → непосредственный вердикт Proved / Disproved
  │
  └→ Медленный канал: логическая обрезка SMT (только когда быстрый канал встречает проходимое обратное ребро)
        │
        ├→ Начало обратного ребра имеет путевое условие → SMT судит path_cond ⇒ !loop_cond
        │     ├→ Proved → логическая обрезка → возврат к быстрому каналу
        │     └→ Disproved / Unproven → обратное ребро проходимо → пометить unsafe
        │
        └→ Начало обратного ребра не имеет путевого условия → обратное ребро проходимо напрямую
```

**Покрытие быстрого канала**: линейный код, if/else, loop + break, while без путевых условий.
**Покрытие медленного канала**: тело цикла while, когда путевое условие указывает на выход из цикла.
**Непокрытое**: условия времени выполнения, которые не могут быть статически доказаны → обратное ребро проходимо → unsafe → ошибка компиляции (пользователь рефакторит).

SMT — не основная сила, а страховочная сетка. В отличие от пользовательских предикатов RFC-027: пользовательские предикаты используют SMT как основную силу; системные предикаты заимствований используют структурный анализ как основную силу, SMT лишь дополняет угловые случаи, недоступные структурному анализу.

## Анализ вариантов использования

### Линейный код

```yaoxiang
data = vec![1, 2, 3]        # узел 1
view = &data                # узел 2: потребляет data, производит ReadToken(#1)
x = view.total_count        # узел 3: потребляет view (= последний потребитель #1)
data.push(4)                # узел 4: требует WriteToken(data)
```

Обратный BFS из `view.total_count` (узел 3) → узел 3 — последний потребитель #1 → узел 4 > узел 3 → узел 4 не в unsafe → ✅

### if/else: без специальных правил

```yaoxiang
view = &data
if cond {
    use(view)               # then-ветка потребляет view
} else {
    do_something_else()     # не трогает view
}
data.push(4)                # последний потребитель view в if → после if потребителей нет → ✅
```

if/else — это составной узел DAG. Внутреннее потребление приписывается этому узлу. Состояния веток не объединяются. Никакого консервативного голосования. **Есть ли потребитель после — это целочисленное сравнение.**

### if/else с утечкой возвращаемого значения

```yaoxiang
view = &data
result = if cond {
    view                     # view утекает в result
} else {
    something_else
}
use(result)                  # косвенное потребление view
data.push(4)                 # у view остался потребитель (use(result))
                             # → push в unsafe → ❌ правильная ошибка
```

view утекает через возвращаемое значение → `use(result)` — потребитель view → обратный обход от `push` достигает `use(result)` → unsafe.

### Цикл: break обрезает обратное ребро

```yaoxiang
view = &data
loop {
    use(view)                # consumer
    if is_last {
        data.push(4)         # операция записи
        break                # ← структурная обрезка
    }
}
```

Обратный BFS из `use(view)` → обратное ребро → вперёд к `data.push(4)` → встречает `break` → **обрезано** → `data.push(4)` не в unsafe → ✅

Без break:

```yaoxiang
view = &data
loop {
    use(view)
    data.push(4)             # нет break для обрезки → обратное ребро проходимо → следующий use(view) достижим
                             # → push в unsafe → ❌ правильная ошибка
}
```

### while: логическая обрезка SMT

```yaoxiang
view = &data
mut i: UpTo(n) = 0
while i < n {
    use(view)                # consumer
    i += 1
    if i == n {
        data.push(4)         # путевое условие: i == n
    }
}
```

Обратный BFS из `use(view)` → обратное ребро → к `data.push(4)` → проверка путевого условия `i == n` → SMT-запрос: `i == n ⇒ !(i < n)`? → Proved → **логическая обрезка** → `data.push(4)` не в unsafe → ✅

## Суть: ID бренда — это и есть `'a`

Не говорим «нам не нужен `'a`». Говорим «`#42` — это и есть `'42`».

| Rust | YaoXiang | Эквивалентность |
|------|----------|--------|
| `'a` | `#42` | Идентификатор времени жизни на этапе компиляции |
| Ограничение outlives `'a: 'b` | `#42` — префикс `#42.field_x` | Сравнение строковых префиксов = отношение частичного порядка |
| Распространение живости NLL (неподвижная точка CFG) | Обратный BFS (DAG) | Оба — вычисление достижимости |
| Факты Polonius | Логическая обрезка SMT | Оба — рассуждение о путевых условиях |
| Решение неподвижной точки системы ограничений | Сопоставление префиксов дерева брендов + BFS | Разная кодировка, одна задача |

**Мы не изобрели новый анализ. Мы лишь опустили `'a` с уровня типовых сигнатур на уровень доказательств.** ID бренда делает в точности то же, что `'a` — маркирует идентичность заимствования, отслеживает отношение порождения, определяет конфликт. Разница одна: `'a` находится в типовых сигнатурах, написанных пользователем; `#42` — внутри компилятора.

В этом нет ничего постыдного. Карри-Ховард говорит, что типы — это утверждения, программы — доказательства. `'a` — не часть утверждения — это часть стратегии доказательства. Rust вписывает стратегию доказательства в сигнатуру утверждения. Мы возвращаем её туда, где ей следует быть.

### Что устраняют ограничения дизайна языка

| Источник сложности | Устранён? | Причина |
|---|---|---|
| Затенение переменных | ✅ | Язык запрещает — одно имя всегда указывает на одно и то же |
| Заимствование через итерации for | ✅ | Каждая итерация — новое связывание — естественная изоляция между итерациями |
| Аннотация времени жизни `'a` | ✅ | Путь бренда = `#42.field_x`, выводится компилятором |
| Именованные времена жизни + распространение ограничений | ✅ | Сравнение префиксов путей брендов заменяет явное множество ограничений |
| Решение ограничений графа заимствований (Polonius) | ✅ | Сопоставление префиксов дерева брендов + запрос потребителей DAG |
| Распространение живости заимствований в теле цикла | ❌ | Нужно обрабатывать, как в Rust — обратным BFS + логической обрезкой |
| Консервативность условных ветвей | ❌ | Как в Rust — SMT покрывает доказуемое, остальное консервативно отвергается |

### Почему DAG осуществим

Три ограничения дизайна YaoXiang делают анализ DAG осуществимым:

- **Отсутствие затенения переменных** — одно имя всегда указывает на одно и то же, не нужно отслеживать переприсваивания
- **for создаёт новое связывание каждую итерацию** — естественная изоляция между итерациями, нет заимствований через итерации
- **Структурированная конкурентность** — чёткие границы задач, не нужно распространять живость через задачи

Эти ограничения устраняют основные источники сложности итерации неподвижной точки CFG в Rust. DAG не «более продвинут», чем CFG — более простой дизайн языка позволяет более простой анализ.

## Детальный проект

### Реестр системных предикатов

Компилятор автоматически генерирует следующие утверждения, отправляемые в конвейер доказательств RFC-027:

| Системный предикат | Момент срабатывания | Форма утверждения |
|---|---|---|
| `borrow_conflict` | Требуется WriteToken(v) | `forall t ∈ conflicting(v): dead_at(t, node)` |
| `use_after_move` | Использование переменной v | `¬moved(v)` |
| `use_after_drop` | Использование переменной v | `¬dropped(v)` |
| `double_drop` | Drop(v) | `¬dropped(v)` |
| `mut_violation` | Запись в неизменяемую переменную v | `is_mut(v)` |

Существующие `BorrowChecker`, `MoveChecker`, `DropChecker`, `MutChecker` **становятся генераторами утверждений** — не исчезают, меняют роль. Они генерируют утверждения, конвейер их верифицирует.

### Дерево брендов

Механизм брендов из RFC-009 §2.7 формализуется как дерево брендов.

**Семантика токенов — заморозка первична, не копирование первично**:

Сущностная разница между `&T` и `&mut T` — не «можно ли копировать», а «допускается ли одновременная запись»:

```
ReadToken(T):  Предоставляет право только на чтение, одновременно замораживая
               исходные данные T — любой WriteToken(T) в этот период
               не может быть получен. Заморозка — первичная семантика ReadToken.
               Dup (копируемость) — следствие заморозки: поскольку данные
               заморожены (мутация невозможна), множественные представления
               только для чтения по природе безопасны.

WriteToken(T): Предоставляет исключительное право на чтение и запись. Поскольку
               запись существует, никакой другой токен (чтение или запись)
               не может сосуществовать. Отсутствие Dup (линейный тип) —
               следствие исключительности.
```

**Причинно-следственная связь**:
```
ReadToken существует → исходные данные заморожены → множественное безопасное чтение → Dup
                                 ↓
                    WriteToken отвергнут (системный предикат borrow_conflict)
```

А не:
```
ReadToken имеет Dup → может быть несколько → заодно проверим конфликт  ← перевёрнутая причинность
```

```
BrandTree:
  nodes: Map<BrandId, BrandNode>

BrandNode:
  id: BrandId               # "#42", "#42.field_x"
  kind: ReadToken | WriteToken
  source_var: Operand
  parent: Option<BrandId>   # родительский узел в отношении порождения
  children: Set<BrandId>    # порождённые дочерние токены
  consumers: Set<NodeId>    # узлы DAG, потребляющие этот токен
  ref_count: usize          # количество безопасных копий во время заморозки ReadToken
```

**Определение конфликта — механизм обеспечения через заморозку**:

```rust
fn conflicts(a: &BrandId, b: &BrandId) -> bool {
    // Условие конфликта: один источник + хотя бы одна запись + перекрытие путей брендов
    // Это означает:
    //   1. ReadToken vs ReadToken → нет конфликта (оба только для чтения, мутация невозможна)
    //   2. WriteToken vs ReadToken → конфликт (запись нарушает гарантию заморозки чтения)
    //   3. WriteToken vs WriteToken → конфликт (две записи не могут сосуществовать)
    a.source() == b.source()
        && (a.is_write() || b.is_write())
        && (a.is_prefix_of(b) || b.is_prefix_of(a))
}
```

Сравнение строковых префиксов O(глубина), глубина ≤ 3. Константный уровень.

### Анализ живости обратным BFS

```
Алгоритм: check_borrow(token, node, dag, brand_tree)

Вход:
  token: проверяемый WriteToken
  node:  узел DAG, содержащий операцию записи

Выход: Proved | Disproved

Алгоритм:
  # Быстрый канал: обратный BFS
  unsafe = empty_set
  queue = brand_tree.consumers(token)

  while queue не пуста:
    cur = queue.pop()
    unsafe.add(cur)

    for each pred in dag.predecessors(cur):
      # Структурная обрезка: break не проходим
      if pred является ребром break:
        continue

      # Обратное ребро → проверить необходимость отката к SMT
      if pred является обратным ребром:
        path_cond = путевое условие в pred
        loop_cond = условие цикла
        # Сначала проверить, обрезано ли структурно (соответствующий break уже обрезал путь → сюда не дойдём)
        # Затем проверить путевое условие
        if path_cond не пусто:
          result = smt_fallback(path_cond, loop_cond)   # ← медленный канал
          if result == Proved:
            continue                    # логическая обрезка
        # Нет путевого условия или SMT не смог → проходим обратное ребро
        # fall through

      if pred ∉ unsafe:
        queue.push(pred)

  # Вердикт
  if node ∈ unsafe:
    return Disproved
  else:
    return Proved


smt_fallback(path_cond, loop_cond):
  # Вызывается только при обратном ребре с путёвым условием
  # Использует конвейер доказательств RFC-027, разделяя тот же SMT-решатель, тот же бюджет
  return smt.prove(path_cond ⇒ !loop_cond)
  # Proved → логическая обрезка
  # Disproved / Unproven → не обрезаем, обратное ребро проходимо
```

O(N), где количество вызовов SMT = количество обратных рёбер × доля обратных рёбер с путевыми условиями. В реальном коде вызовы SMT крайне редки — только в теле цикла `while` с переменными уточняющего типа в путевом условии.

### Сбор путевых условий

Предоставляется существующими механизмами RFC-027 §3.2-3.3:

- **if guard**: `if y > 0` → в true-ветку помещается `y > 0`
- **match-паттерн**: `if let Some(v) = opt` → в ветку помещается `opt == Some(v)`
- **Присваивание**: `i += 1`, компилятор поддерживает информацию о домене значений переменной
- **while cond**: внутри тела цикла помещается `cond == true`

Каждый узел DAG несёт множество путевых условий. При прохождении обратного BFS через обратное ребро берётся путевое условие в начале обратного ребра, SMT определяет, исключает ли оно условие входа в следующую итерацию.

### Интерфейс с RFC-027

Системные предикаты заимствований и пользовательские предикаты разделяют один конвейер доказательств — разница лишь в **основной стратегии доказательства**:

| Тип запроса | Источник утверждения | Основная стратегия | Откат |
|---|---|---|---|
| Равенство типов | Проверщик типов | Структурная эквивалентность | — |
| Пользовательский предикат | Типовая аннотация программиста | SMT | Функция-доказательство программиста |
| Конфликт заимствований | Автогенерация компилятором | **Структурный анализ DAG (быстрый канал)** | Логическая обрезка SMT |

Роль SMT-решателя в проверке заимствований: **не основная сила, а страховочная сетка.** Вызывается только при необходимости логической обрезки обратного ребра while. Подавляющее большинство проверок заимствований завершается на быстром канале — обратный BFS за O(N), нулевые накладные расходы SMT.

### Связь с существующим кодом

| Существующий компонент | Обработка |
|----------|------|
| `BorrowChecker` | Становится `BorrowPredicateEmitter` — генерирует утверждения Хоара для заимствований |
| `MoveChecker` | Становится `MovePredicateEmitter` — генерирует утверждения `¬moved(v)` |
| `DropChecker` | То же — генерирует утверждения, связанные с Drop |
| `MutChecker` | То же — генерирует утверждения `is_mut(v)` |
| `ControlFlowAnalyzer` | Больше не нужен — конвейер обрабатывает единообразно |
| `liveness_analysis` | Сохраняется — вставка Drop по-прежнему нуждается в информации о живости переменных |
| Жёстко закодированный Release в `ir_gen.rs` | Удаляется — позиция Release определяется анализом потребителей DAG |

### NLL и границы итераций

**Момент смерти токена = точка последнего использования (NLL), а не конец лексической области видимости.**

Это естественное следствие анализа потребителей: позиция потребителя определяет последнее использование токена. `use(v)` — потребитель `v` → `v` умирает сразу после `use(v)`. Не нужны дополнительные `{}` или `drop()` для досрочного завершения жизни токена.

**Граница итерации цикла — линия смерти копий токенов.** Три правила:

```
Правило 1: Переменные, объявленные внутри цикла, автоматически умирают в конце каждой итерации.
           Каждая итерация for — новое связывание (гарантия дизайна языка), то же для loop.

Правило 2: ref_count дерева брендов в заголовке цикла учитывает только копии, созданные вне цикла.
           Новые копии, созданные Dup внутри цикла, обнуляются по ref_count на границе итерации.

Правило 3: При прохождении обратного BFS через обратное ребро информация о живости
           текущей итерации не переносится. Переносится только ref_count в заголовке цикла
           (т.е. копии вне цикла).
```

Пример:

```yaoxiang
view = &data                          # заголовок цикла: ref_count = 1, consumer = use(view)
loop {
    v2: &Point = view                 # Dup внутри цикла → ref_count = 2
    use(v2)                           # consumer: последнее использование v2 → v2 умирает → ref_count = 1
    data.push(4)                      # ✅ безопасно! v2 мёртв, остался только view (ref_count = 1, без конфликта записи)
    # граница итерации: правило 3 — v2 не переносится в следующую итерацию. В начале следующей итерации v2 пересоздаётся.
}
```

Этот дизайн не требует дополнительного правила «консервативной живости в цикле». Обратный BFS стартует от потребителя, потребитель внутри тела цикла → живость ограничена текущей итерацией → обратное ребро не проходимо. Полностью согласуется с примером цикла из §Анализа вариантов использования RFC-009a.

### Распространение ошибок `?` и Release под управлением области видимости

`?` — досрочный возврат — помимо нормального выхода из области видимости существует дополнительный путь выхода. Токен должен быть освобождён на этом пути, ошибка в порядке освобождения означает UB.

**Инструкции Release генерируются анализом областей видимости, а не жёстко закодированы после Call.**

Компилятор поддерживает список точек выхода для каждой области видимости:
- `}` (нормальное окончание области видимости)
- `?` (распространение ошибки, досрочный return)
- явный `return`

В каждой точке выхода в обратном порядке объявления (LIFO) вставляются инструкции Release для всех активных токенов этой области видимости. Отношения родитель-потомок в дереве брендов автоматически обрабатывают каскадное освобождение порождённых токенов:

```yaoxiang
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)    # возврат дочернего токена &Float + родительского токена &Point
}

fn use_case(p: Point) -> Result<(), Error> = {
    (x_ref, p_ref) = p.get_x()?   # если ? распространяет:
    # дерево брендов знает, что x_ref — порождение p_ref (#42.field_x — префикс #42)
    # порядок освобождения: x_ref (дочерний) → p_ref (родительский) → LIFO соблюдается автоматически
    p.modify()                     # WriteToken — все ReadToken освобождены
    Ok(())
}
```

Место реализации: остаётся в `ir_gen.rs`, изменяется на управляемое областью видимости — без нового прохода компилятора.

| Операция | Сложность | Частота срабатывания |
|------|--------|---------|
| Определение конфликта в дереве брендов | O(1) | Каждый запрос токена |
| Запрос потребителей DAG | O(1) | Каждый запрос токена |
| Обратный BFS (быстрый канал) | O(N) | Каждый запрос токена, N = количество узлов в блоке |
| Логическая обрезка SMT (fallback) | ~1ms | **Крайне редко** — только while + путевое условие |

**Условия срабатывания SMT fallback крайне жёсткие**: одновременно (1) цикл while (2) тело цикла содержит операцию записи (3) после операции записи есть путевое условие, по которому можно судить о завершении цикла (4) компилятору нужно полагаться на это условие для обрезки обратного ребра. В реальном коде доля составляет значительно менее 1%. Остальные проверки заимствований полностью завершаются на быстром канале.

Связь с пользовательскими предикатами RFC-027: пользовательские предикаты используют SMT как основную силу, системные предикаты заимствований — структурный анализ как основную силу. Оба разделяют один SMT-решатель и верхний предел бюджета (RFC-027 §8), но системные предикаты заимствований почти не расходуют бюджет SMT.

Линейный код → нет обратных рёбер → уровень 1 O(N) мгновенно. Цикл + путевое условие → вызов SMT, линейная арифметика за миллисекунды (бюджет RFC-027 100ms). Результат одного BFS может кэшироваться для повторного использования при нескольких запросах одного токена.

### Дизайн сообщений об ошибках

**Основной принцип: в сообщениях об ошибках появляются только символы, написанные пользователем.**

В Rust ошибки, связанные с заимствованиями, делятся на две категории:

**Ошибки на уровне переменных**: E0597 (живёт недостаточно долго), E0502 (изменяемое + неизменяемое заимствование одновременно), E0499 (множественные изменяемые заимствования). Rust уже является эталоном — имя переменной + номер строки, без `'a`. YaoXiang достигает той же точности. Вся информация содержится в дереве брендов: точка создания токена, позиция потребителя, точка запроса.

**Ошибки на уровне сигнатур**: E0623 (lifetime mismatch), E0106 (missing lifetime specifier), E0477 (не выполнено required lifetime). Построены вокруг `'a`. YaoXiang **не имеет этого класса ошибок** — в сигнатурах нет `'a`. Не «нельзя сообщить», а то, что пользователь не писал, не нужно сообщать.

Пример конфликта внутри функции:

```
Ошибка: `data` заморожен, невозможно получить изменяемый доступ
 --> src/main.yx:5:9
2 |     view = &data
  |            ----- `data` заморожен (токен только для чтения создан здесь)
4 |         use(view)
  |             ---- `view` всё ещё используется здесь, заморозка не снята
5 |         data.push(4)
  |         ^^^^ здесь требуется изменяемый доступ
```

(Точность на уровне Rust E0499 — имя переменной + номер строки, без ID бренда.)

Пример межфункциональной утечки:

```
Ошибка: один из источников данных, удерживаемых `num` (строка 4), — это `default_str` (строка 3),
но `default_str` становится недействительным в строке 6, тогда как `num` всё ещё используется в строке 5.

Рекомендация: переместите объявление `default_str` в вызывающую функцию, или используйте `ref default_str` для совместного владения.
```

(Точность на уровне Rust E0597. Сводка брендов знает, что у `num` два пути источника — уже есть в компиляторе, формулировка ошибки применима.)

## Исправления в основном тексте RFC-009

Раздел RFC-009 §«Обнаружение конфликтов токенов: потоково-чувствительный анализ живости» обновлён:

1. Удалено «не нужно:……NLL» — не потому что вывод неверен, а потому что аргументация ошибочна («токены — это значения, линейного отслеживания достаточно»)
2. Переходное решение «уровень 1/уровень 2» сохранено, полное решение указывает на настоящий RFC
3. Уточнено: ID бренда (`#42`) — это и есть `'a` — информация идентична, кодировка различна. Не изобретён новый анализ — время жизни перенесено с уровня типов на уровень доказательств

## Компромиссы

### Преимущества

1. **Сигнатуры типов не содержат времён жизни**: `#42` — это и есть `'42` — та же информация, закодированная в дереве брендов, не выставленная в типовой сигнатуре. Этот пункт неопровержим: посчитайте, сколько параметров `'a` требуется в Rust для обобщённого типа с 3 параметрами-ссылками, и сколько требуется в YaoXiang. Ответ: 3 против 0.

2. **Концептуальное единство**: проверка заимствований и пользовательские предикаты разделяют один конвейер доказательств — `{P} op {Q}`, конвейер верифицирует P. Согласованность с Карри-Ховард.

3. **Нулевой новый фреймворк анализа**: не вводится новый фреймворк анализа. Пользователь не воспринимает существование «проверщика заимствований» — как не воспринимает деталей реализации «проверщика типов».

4. **Сообщения об ошибках содержат только символы, написанные пользователем**: устранено целое измерение категорий ошибок (E0623, E0106, E0477 — все вокруг `'a`). Точность ошибок на уровне переменных соответствует Rust.

5. **Алгоритм не консервативен**: обратный BFS + обрезка break + логическая обрезка SMT. Не требуется «консервативная живость в цикле». Не требуется «консервативное объединение ветвей».

### Недостатки

1. **Не новое изобретение**: ID бренда делает в точности то же, что `'a` — сложность решения ограничений внутри компилятора не исчезла, лишь кодировка изменилась с «имя переменной + множество ограничений» на «путь бренда + сопоставление префиксов». Для конечного пользователя разница лишь в том, что в сигнатуре не пишется `'a`.

2. **Полностью новая реализация**: дерево брендов в коде существует лишь как концепция, требует реализации с нуля. BorrowChecker, ControlFlowAnalyzer заменяются.

3. **Зависимость от SMT**: логическая обрезка зависит от Z3 (уже введён в RFC-027, новых зависимостей нет). Однако проверка заимствований почти не срабатывает — только при while + путевом условии.

4. **Крайне редкие паттерны требуют рефакторинга**: межветвевые заимствования, которые компилятор не может автоматически доказать, требуют от пользователя рефакторинга кода. В отличие от резервного варианта `'a` в Rust: в Rust есть `'a` как инструмент (аннотация — и проходит); резервный вариант YaoXiang (функция-доказательство) не входит в MVP.
