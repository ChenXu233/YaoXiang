# RFC-009a: Анализ времени жизни токенов — на основе конвейера доказательств Хоара

> **Родительский RFC**: [RFC-009: Проект модели владения](../accepted/009-ownership-model.md)
>
> **Зависимость**: [RFC-027: Предикаты времени компиляции и унифицированная статическая верификация](../accepted/027-compile-time-evaluation-types.md)
>
> **Предусловие**: RFC-027 должен быть принят. Все механизмы данного RFC (конвейер доказательств, SMT-fallback, сбор путевых условий) зависят от реализации RFC-027.
>
> **Настоящий RFC исправляет и заменяет RFC-009 §"Обнаружение конфликтов токенов: потоково-чувствительный анализ живучести" (строки 663–684).**

## Резюме

В строке 684 RFC-009 утверждается, что обнаружение конфликтов токенов "не требует……NLL". Вывод верен, аргумент — нет.

Не потому что "токены — это значения, и линейного отслеживания достаточно". А потому что: **живучесть токенов — это утверждение логики Хоара, а не специальный потоково-чувствительный анализ.**

`{конфликтующие_токены все мертвы} op {WriteToken безопасно получен}` — то же самое `{P} op {Q}`, которое разделяет конвейер доказательств RFC-027 с проверкой типов и верификацией предикатов. Никаких новых аналитических фреймворков. Один конвейер — множество утверждений.

---

## Мотивация

### Путаница в RFC-009

RFC-009 смешивает две задачи:

1. **Линейное отслеживание** (после Move недоступно) — `{v не move-нут} use(v) {тип совпадает}`. Уже есть в проверщике типов.
2. **Взаимодействие времён жизни токенов** (дочерний токен жив → родительский токен приостановлен → дочерний токен умер → родительский токен возрождается) — `{все конфликтующие токены мертвы} write(data) {безопасно}`. Требует **анализа живучести**, а не линейного отслеживания.

### Текущее состояние кода

| Компонент | Состояние |
|------|------|
| `BorrowChecker` | Линейное сканирование IR, пассивная реакция на явные инструкции `Borrow`/`Release` |
| `ControlFlowAnalyzer::analyze_instruction` | Пустая реализация (`control_flow.rs:145-153`) |
| `liveness_analysis` | Существует, но используется только для вставки Drop, не подключён к конфликтам токенов |
| Вставка Release | Жёстко закодирована после инструкции Call — чисто лексическая область видимости (`ir_gen.rs:2734-2736`) |

**Видимые пользователю последствия**:

```yaoxiang
data = vec![1, 2, 3]
view = &data              # создание ReadToken
x = view.total_count      # последнее использование view
data.push(4)              # ❌ Release(view) ещё не выполнен, ReadToken "жив"
```

### Почему требуется переработка

Предыдущая версия (009a v1) использовала нарратив "DAG вместо NLL", вводя ненужные новые концепции (консервативные правила ветвления, специальную обработку циклов). Основное противоречие не было прояснено: **проверка заимствований — не отдельная система — это один из видов утверждений Хоара.**

## Основной проект

### Всё суть Хоар

```
Проверка типов:    { x: Int }        x + 1        { result: Int }
Проверка заимств.: { view мёртв }    data.push(4)  { WriteToken успешно получен }
Верификация пред.: { y > 0 }         divide(x, y)  { result: Int }
Разрыв обратной дуги: { i == n }     следующая итерация  { cond == false }
```

Одна и та же форма `{P} op {Q}`. Компилятор генерирует предусловие P для каждой операции и отправляет его в конвейер доказательств на верификацию.

**Проверка заимствований и пользовательские предикаты разделяют один конвейер.** Разница только в том, кто порождает утверждение и что происходит при невозможности доказать.

### Два вида предикатов — один конвейер

| | Пользовательский предикат | Системный предикат (заимствование) |
|---|---|---|
| Генерация утверждений | Программист (аннотации типов) | Компилятор (дерево брендов + правила владения) |
| Предоставление доказательства | Компилятор + программист | **Полностью автоматически компилятором** |
| Не доказуемо | Написать функцию-доказательство или рефакторить | Рефакторить код (двери оставлены, но нужны крайне редко) |
| Видимость | Видно в сигнатуре | Неявное, не загрязняет сигнатуры типов |
| Кривая обучения | Учишь только если хочешь | Ноль |

**Доказательства для системных предикатов не открывают программисту функций-доказательств — компилятор делает это полностью автоматически.** Когда доказать не удаётся, пользователь рефакторит код.

**Три режима отказа — один движок верификации.** Утверждение типа не доказуемо → ошибка компиляции (не обойти). Утверждение заимствования не доказуемо → ошибка компиляции, рефакторинг (не обойти). Пользовательский предикат не доказуем → ошибка компиляции, можно написать функцию-доказательство (обходимо). Стратегии отказа различаются, но движок верификации один и тот же — SMT-решатель + ядро компилятора с правилами вывода. Разница только в том, "кто отвечает за восполнение доказательства при неудаче" — компилятор отказывается писать за программиста доказательства заимствования (стратегия доказательства утверждений заимствования — структурный анализ + SMT, вмешательство программиста не требуется), но принимает написанные программистом функции-доказательства для пользовательских предикатов. Это не рассогласование конвейера — это разные границы ответственности для разных категорий утверждений.

Это отличается от Rust `'a`: `'a` — обязательный предмет, функция-доказательство — факультативный — подавляющее большинство пользователей за всю жизнь ни разу не столкнётся с дверью факультатива.

### Утверждения заимствования: автоматическая генерация компилятором

Пользователь пишет `data.push(4)`. Компилятор автоматически генерирует утверждение:

```
WriteToken(data, node) может быть получен
  = forall t в conflicting_tokens(data): t в node уже мёртв
  = forall t в brand_tree.children(data): forward_reachable(node) ∩ consumers(t) == ∅
```

**Три правила, ноль особых случаев:**

1. **Дерево брендов** (RFC-009 §2.7) отвечает на вопрос "кто с кем конфликтует": сопоставление префиксов, O(глубина), глубина ≤ 3
2. **Список потребителей** (автоматически собирается при построении DAG) отвечает на вопрос "кто последним потребил токен"
3. **Прямая достижимость** отвечает на вопрос "может ли потребитель ещё быть выполнен": структурный разрыв + логический разрыв

### Прямая достижимость: обратный обход от потребителей

Для каждого потребителя C токена T:

```
Из C выполняется обратный BFS по DAG.
Ребро разрывается, если:
  1. Оно является break (структурный разрыв)
  2. Условие пути ⇒ !loop_cond доказано SMT как истинное (логический разрыв, конвейер RFC-027)

Распространяем в обратном направлении по всем неразорванным рёбрам (включая обратные дуги,
обратная дуга распространяет живучесть на предыдущую итерацию).
Помечаем все достижимые узлы → unsafe.
```

Запрос: операция записи в узле W → W ∉ unsafe → безопасно.

**Не нужно изобретать "консервативные правила ветвления". Не нужно "консервативной живучести в циклах". Один обратный BFS + два правила разрыва.**

### Стратегия доказательства: быстрый канал в приоритете, SMT как страховка

```
Каждая операция записи, требующая токен
  │
  ├→ Быстрый канал: структурный анализ DAG (покрывает 95%+ сценариев)
  │     │
  │     ├→ Сопоставление префиксов дерева брендов → найти конфликтующие токены (O(глубина))
  │     ├→ Обратный BFS, break разрывает обратные дуги
  │     └→ Нет проходимых обратных дуг → прямое решение Proved / Disproved
  │
  └→ Медленный канал: логический разрыв SMT (только когда быстрый канал встречает проходимую обратную дугу)
        │
        ├→ Начало обратной дуги имеет условие пути → SMT проверяет path_cond ⇒ !loop_cond
        │     ├→ Proved → логический разрыв → откат к быстрому каналу
        │     └→ Disproved / Unproven → обратная дуга проходима → пометить unsafe
        │
        └→ Начало обратной дуги не имеет условия пути → обратная дуга прямо проходима
```

**Покрытие быстрого канала**: линейный код, if/else, loop + break, while без условий пути.
**Покрытие медленного канала**: тело цикла while, когда условие пути указывает на выход из цикла.
**Непокрытое**: условия времени выполнения, которые нельзя статически доказать → обратная дуга проходима → unsafe → ошибка компиляции (пользователь рефакторит).

SMT — не главная сила, а страховочная сетка. В отличие от пользовательских предикатов RFC-027, где SMT — главная сила; для системных предикатов заимствования главная сила — структурный анализ, а SMT лишь дополняет углы, куда структурный анализ не достаёт.

## Анализ вариантов использования

### Линейный код

```yaoxiang
data = vec![1, 2, 3]        # узел 1
view = &data                # узел 2: потребляет data, производит ReadToken(#1)
x = view.total_count        # узел 3: потребляет view (= последний потребитель #1)
data.push(4)                # узел 4: требует WriteToken(data)
```

Обратный BFS от `view.total_count` (узел 3) → узел 3 — последний потребитель #1 → узел 4 > узел 3 → узел 4 не в unsafe → ✅

### if/else: без специальных правил

```yaoxiang
view = &data
if cond {
    use(view)               # then-ветвь потребляет view
} else {
    do_something_else()     # не трогает view
}
data.push(4)                # последний потребитель view внутри if → после if потребителей нет → ✅
```

if/else — составной узел DAG. Внутреннее потребление приписывается этому узлу. Состояния ветвей не объединяются. Консервативного голосования нет. **Есть ли потребитель после — целочисленное сравнение.**

### if/else с утечкой возвращаемого значения

```yaoxiang
view = &data
result = if cond {
    view                     # view утекает в result
} else {
    something_else
}
use(result)                  # косвенно потребляет view
data.push(4)                 # у view всё ещё есть потребитель (use(result))
                             # → push в unsafe → ❌ корректная ошибка
```

view утекает через возвращаемое значение → `use(result)` — потребитель view → обратный обход от `push` достигает `use(result)` → unsafe.

### Цикл: break разрывает обратную дугу

```yaoxiang
view = &data
loop {
    use(view)                # consumer
    if is_last {
        data.push(4)         # операция записи
        break                # ← структурный разрыв
    }
}
```

Обратный BFS от `use(view)` → обратная дуга → вперёд к `data.push(4)` → встречает `break` → **разрыв** → `data.push(4)` не в unsafe → ✅

Без break:

```yaoxiang
view = &data
loop {
    use(view)
    data.push(4)             # нет break-разрыва → обратная дуга проходима → следующая итерация use(view) достижима
                             # → push в unsafe → ❌ корректная ошибка
}
```

### while: логический разрыв SMT

```yaoxiang
view = &data
mut i: UpTo(n) = 0
while i < n {
    use(view)                # consumer
    i += 1
    if i == n {
        data.push(4)         # условие пути: i == n
    }
}
```

Обратный BFS от `use(view)` → обратная дуга → к `data.push(4)` → проверка условия пути `i == n` → SMT-запрос: `i == n ⇒ !(i < n)`? → Proved → **логический разрыв** → `data.push(4)` не в unsafe → ✅

## Суть: Brand ID — это и есть `'a`

Не говорим "нам не нужен `'a`". Говорим "`#42` — это и есть `'42`".

| Rust | YaoXiang | Эквивалентность |
|------|----------|--------|
| `'a` | `#42` | Идентификатор времени жизни времени компиляции |
| Ограничение outlives `'a: 'b` | `#42` — префикс для `#42.field_x` | Сравнение строковых префиксов = отношение частичного порядка |
| Распространение живучести NLL (неподвижная точка CFG) | Обратный BFS (DAG) | Обе — вычисления достижимости |
| Факты Polonius | Логический разрыв SMT | Оба — рассуждения об условиях пути |
| Решение системы ограничений неподвижной точки | Сопоставление префиксов дерева брендов + BFS | Разные кодировки, одна задача |

**Мы не изобрели новый анализ. Мы просто опустили `'a` с уровня сигнатур типов на уровень доказательств.** Brand ID делает ровно то же, что `'a` — маркирует идентичность заимствования, отслеживает отношения порождения, разрешает конфликты. Разница одна: `'a` находится в сигнатуре типа, написанной пользователем; `#42` — внутри компилятора.

В этом нет ничего постыдного. Curry-Howard говорит: типы — это утверждения, программы — доказательства. `'a` не является частью утверждения — это часть стратегии доказательства. Rust записывает стратегию доказательства в сигнатуру утверждения. Мы возвращаем её туда, где ей место.

### Что устраняют ограничения дизайна языка

| Источник сложности | Устранено? | Причина |
|---|---|---|
| Затенение переменных | ✅ | Язык запрещает — одно имя всегда ссылается на одно и то же |
| Заимствование через итерации for | ✅ | Каждая итерация — новое связывание — естественная изоляция между итерациями |
| Аннотации времени жизни `'a` | ✅ | Путь бренда = `#42.field_x`, выводится компилятором |
| Именованные времена жизни + распространение ограничений | ✅ | Сравнение префиксов путей брендов заменяет явные множества ограничений |
| Решение системы ограничений графа заимствований (Polonius) | ✅ | Сопоставление префиксов дерева брендов + запрос потребителей DAG |
| Распространение живучести заимствований в теле цикла | ❌ | Требует обработки как и в Rust — через обратный BFS + логический разрыв |
| Консервативность условных ветвлений | ❌ | Как и в Rust — SMT покрывает доказуемое, остальное консервативно отвергается |

### Почему DAG возможен

Три ограничения дизайна языка YaoXiang делают анализ DAG возможным:

- **Нет затенения переменных** — одно имя всегда указывает на одно и то же, не нужно отслеживать через пересвязывания
- **for создаёт новое связывание на каждой итерации** — естественная изоляция между итерациями, нет заимствований между итерациями
- **Структурированная конкурентность** — чёткие границы задач, не нужно распространение живучести между задачами

Эти ограничения устранили основные источники сложности итераций неподвижной точки CFG в Rust. DAG не "более продвинутый" чем CFG — более простой дизайн языка позволяет более простой анализ.

## Детальный проект

### Реестр системных предикатов

Компилятор автоматически генерирует следующие утверждения и отправляет их в конвейер доказательств RFC-027:

| Системный предикат | Момент срабатывания | Форма утверждения |
|---|---|---|
| `borrow_conflict` | Требуется WriteToken(v) | `forall t ∈ conflicting(v): dead_at(t, node)` |
| `use_after_move` | Использование переменной v | `¬moved(v)` |
| `use_after_drop` | Использование переменной v | `¬dropped(v)` |
| `double_drop` | Drop(v) | `¬dropped(v)` |
| `mut_violation` | Запись в неизменяемую переменную v | `is_mut(v)` |

Существующие `BorrowChecker`, `MoveChecker`, `DropChecker`, `MutChecker` **становятся генераторами утверждений** — не исчезают, меняют роль. Они генерируют утверждения, конвейер их верифицирует.

### Дерево брендов

Механизм брендов из RFC-009 §2.7 формализован как дерево брендов.

**Семантика токенов — заморозка первична, не копирование первично**:

Суть различия между `&T` и `&mut T` не в "можно ли копировать", а в "допускается ли одновременная запись":

```
ReadToken(T):  Предоставляет право только на чтение, одновременно замораживая
               исходные данные T — любой WriteToken(T) в этот период не может
               быть получен. Заморозка — первичная семантика ReadToken. Dup
               (копируемость) — следствие заморозки: поскольку данные заморожены
               (мутация невозможна), множественные представления только для чтения
               по своей природе безопасны.

WriteToken(T): Предоставляет исключительное право на чтение и запись. Поскольку
               запись существует, никакой другой токен (чтения или записи)
               не может сосуществовать. Реализация без Dup (линейный тип) —
               следствие исключительности.
```

**Причинно-следственная связь**:
```
ReadToken существует → исходные данные заморожены → множественное только-чтение безопасно → Dup
                                        ↓
                          WriteToken отвергнут (принуждается системным предикатом borrow_conflict)
```

А не:
```
ReadToken имеет Dup → может быть несколько → заодно проверить конфликт  ← обратная причинность
```

```
BrandTree:
  nodes: Map<BrandId, BrandNode>

BrandNode:
  id: BrandId               # "#42", "#42.field_x"
  kind: ReadToken | WriteToken
  source_var: Operand
  parent: Option<BrandId>   # родительский узел в отношении порождения
  children: Set<BrandId>    # порождённые дочерние токены
  consumers: Set<NodeId>    # узлы DAG, потребляющие этот токен
  ref_count: usize          # число безопасных копий в период заморозки ReadToken
```

**Определение конфликта** — механизм обеспечения, гарантированный заморозкой:

```rust
fn conflicts(a: &BrandId, b: &BrandId) -> bool {
    // Условие конфликта: один источник + хотя бы одна запись + перекрытие путей брендов
    // Это означает:
    //   1. ReadToken vs ReadToken → нет конфликта (оба только-чтение, мутации нет)
    //   2. WriteToken vs ReadToken → конфликт (запись нарушает гарантию заморозки чтения)
    //   3. WriteToken vs WriteToken → конфликт (две записи не могут сосуществовать)
    a.source() == b.source()
        && (a.is_write() || b.is_write())
        && (a.is_prefix_of(b) || b.is_prefix_of(a))
}
```

Сравнение строковых префиксов O(глубина), глубина ≤ 3. Константный уровень.

### Анализ живучести обратным BFS

```
Алгоритм: check_borrow(token, node, dag, brand_tree)

Вход:
  token: WriteToken для проверки
  node:  узел DAG операции записи

Выход: Proved | Disproved

Алгоритм:
  # Быстрый канал: обратный BFS
  unsafe = empty_set
  queue = brand_tree.consumers(token)

  while queue not empty:
    cur = queue.pop()
    unsafe.add(cur)

    for each pred in dag.predecessors(cur):
      # Структурный разрыв: break не пересекается
      if pred является break-ребром:
        continue

      # Обратная дуга → проверка необходимости SMT-fallback
      if pred является обратной дугой:
        path_cond = условие пути в pred
        loop_cond = условие цикла
        # Сначала проверим, можно ли разорвать структурно
        # (соответствующий break уже разорвал путь → сюда не дойдём)
        # Затем проверим условие пути
        if path_cond не пусто:
          result = smt_fallback(path_cond, loop_cond)   # ← медленный канал
          if result == Proved:
            continue                    # логический разрыв
        # Нет условия пути или SMT не доказал → пересекаем обратную дугу
        # fall through

      if pred ∉ unsafe:
        queue.push(pred)

  # Решение
  if node ∈ unsafe:
    return Disproved
  else:
    return Proved


smt_fallback(path_cond, loop_cond):
  # Вызывается только при обратной дуге + наличии условия пути
  # Использует конвейер доказательств RFC-027, разделяет тот же SMT-решатель, тот же бюджет
  return smt.prove(path_cond ⇒ !loop_cond)
  # Proved → логический разрыв
  # Disproved / Unproven → не разрываем, обратная дуга проходима
```

O(N), где число вызовов SMT = число обратных дуг × доля обратных дуг с условиями пути. В реальном коде вызовы SMT крайне редки — только в теле циклов `while` с переменными уточнённых типов, имеющими условия пути.

### Сбор условий пути

Предоставляется существующими механизмами RFC-027 §3.2-3.3:

- **if guard**: `if y > 0` → в true-ветвь помещается `y > 0`
- **match-паттерн**: `if let Some(v) = opt` → в ветвь помещается `opt == Some(v)`
- **Присваивание**: `i += 1`, компилятор поддерживает информацию о домене значений
- **условие while**: внутри тела цикла помещается `cond == true`

Каждый узел DAG несёт множество условий пути. При обратном BFS обратной дуги берётся условие пути в начале обратной дуги, SMT определяет, исключает ли оно условие входа в следующую итерацию.

### Интерфейс с RFC-027

Системные предикаты заимствования и пользовательские предикаты разделяют один конвейер доказательств — разница в **главной стратегии доказательства**:

| Тип запроса | Источник утверждения | Главная стратегия | Fallback |
|---|---|---|---|
| Равенство типов | Проверщик типов | Структурная эквивалентность | — |
| Пользовательский предикат | Аннотации типов программиста | SMT | Функция-доказательство программиста |
| Конфликт заимствования | Автогенерация компилятором | **Структурный анализ DAG (быстрый канал)** | Логический разрыв SMT |

Роль SMT-решателя в проверке заимствований: **не главная сила, а страховочная сетка.** Вызывается только когда обратная дуга while требует логического разрыва. Подавляющее большинство проверок заимствований завершается на быстром канале — обратный BFS O(N), нулевые накладные расходы SMT.

### Связь с существующим кодом

| Существующий компонент | Обработка |
|----------|------|
| `BorrowChecker` | Становится `BorrowPredicateEmitter` — генерирует утверждения Хоара для заимствований |
| `MoveChecker` | Становится `MovePredicateEmitter` — генерирует утверждения `¬moved(v)` |
| `DropChecker` | То же — генерирует утверждения, связанные с Drop |
| `MutChecker` | То же — генерирует утверждения `is_mut(v)` |
| `ControlFlowAnalyzer` | Больше не нужен — конвейер обрабатывает унифицированно |
| `liveness_analysis` | Сохраняется — вставка Drop по-прежнему нуждается в информации о живучести переменных |
| Жёстко закодированный Release в `ir_gen.rs` | Удаляется — позиция Release определяется анализом потребителей DAG |

### NLL и границы итераций

**Момент смерти токена = точка последнего использования (NLL), а не конец лексической области видимости.**

Это естественное следствие анализа потребителей: позиция потребителя определяет последнее использование токена. `use(v)` — потребитель `v` → `v` умирает сразу после `use(v)`. Никаких дополнительных `{}` или `drop()` для преждевременного завершения жизни токена не требуется.

**Граница итерации цикла — линия смерти копий токена.** Три правила:

```
Правило 1: Переменные, объявленные в цикле, автоматически умирают в конце каждой итерации.
           Каждая итерация for — новое связывание (гарантия дизайна языка), то же для loop.

Правило 2: ref_count дерева брендов на заголовке цикла учитывает только копии,
           созданные снаружи цикла. Копии, созданные Dup внутри цикла, обнуляются на границе итерации.

Правило 3: При пересечении обратной дуги в обратном BFS информация о живучести текущей итерации
           не переносится. Переносится только ref_count на заголовке цикла (т.е. копии снаружи цикла).
```

Пример:

```yaoxiang
view = &data                          # заголовок цикла: ref_count = 1, consumer = use(view)
loop {
    v2: &Point = view                 # Dup внутри цикла → ref_count = 2
    use(v2)                           # consumer: последнее использование v2 → v2 умирает → ref_count = 1
    data.push(4)                      # ✅ безопасно! v2 мёртв, остался только view (ref_count = 1, нет конфликта записи)
    # граница итерации: правило 3 — v2 не переносится в следующую итерацию. В начале следующей итерации v2 пересоздаётся новым связыванием.
}
```

Этот проект не требует дополнительного правила "консервативной живучести в цикле". Обратный BFS исходит из потребителей, потребитель находится в теле цикла → живучесть ограничена текущей итерацией → обратная дуга не пересекается. Полностью согласуется с примером цикла из §Анализ вариантов использования RFC-009a.

### `?` распространение ошибок и Release, управляемый областью видимости

`?` — преждевременный возврат — за пределами нормального выхода из области видимости появляется дополнительный путь выхода. Токены должны быть освобождены на этом пути; неправильный порядок освобождения — UB.

**Инструкции Release генерируются анализом области видимости, а не жёстко закодированы после Call.**

Компилятор поддерживает список точек выхода для каждой области видимости:
- `}` (нормальное завершение области видимости)
- `?` (распространение ошибки, преждевременный return)
- явный `return`

В каждой точке выхода в обратном порядке объявления (LIFO) вставляются инструкции Release для всех активных токенов этой области видимости. Отношения родитель-потомок дерева брендов автоматически обрабатывают каскадное освобождение порождённых токенов:

```yaoxiang
Point.get_x: (self: &Point) -> (&Float, &Point) = {
    return (&self.x, self)    # возвращает дочерний токен &Float + родительский токен &Point
}

fn use_case(p: Point) -> Result<(), Error> = {
    (x_ref, p_ref) = p.get_x()?   # если ? распространяет:
    # дерево брендов знает, что x_ref — порождение p_ref (#42.field_x — префикс #42)
    # порядок освобождения: x_ref (потомок) → p_ref (родитель) → LIFO выполняется автоматически
    p.modify()                     # WriteToken — все ReadToken уже освобождены
    Ok(())
}
```

Место реализации: остаётся в `ir_gen.rs`, но меняется на управляемый областью видимости — никакого нового прохода компилятора.

| Операция | Сложность | Частота срабатывания |
|------|--------|---------|
| Проверка конфликта в дереве брендов | O(1) | Каждый запрос токена |
| Запрос потребителей DAG | O(1) | Каждый запрос токена |
| Обратный BFS (быстрый канал) | O(N) | Каждый запрос токена, N = число узлов в блоке |
| Логический разрыв SMT (fallback) | ~1ms | **Крайне редко** — только while + условия пути |

**Условия срабатывания SMT-fallback крайне строгие**: должны одновременно выполняться (1) цикл while (2) в теле цикла есть операция записи (3) после операции записи есть условие пути, позволяющее судить о завершении цикла (4) компилятору нужно полагаться на это условие для разрыва обратной дуги. В реальном коде доля значительно ниже 1%. Все остальные проверки заимствований завершаются на быстром канале.

Связь с пользовательскими предикатами RFC-027: пользовательские предикаты используют SMT как главную силу, системные предикаты заимствований — структурный анализ как главную силу. Оба разделяют один и тот же SMT-решатель и верхний предел бюджета (RFC-027 §8), но системные предикаты заимствований почти не расходуют бюджет SMT.

Линейный код → нет обратных дуг → уровень 1 O(N) за секунды. Цикл + условия пути → вызов SMT, линейная арифметика за миллисекунды (бюджет RFC-027 100ms). Результат одного BFS может кэшироваться для повторных запросов того же токена.

### Проектирование сообщений об ошибках

**Ключевой принцип: в сообщениях об ошибках появляются только символы, написанные пользователем.**

Rust разделяет ошибки, связанные с заимствованиями, на две категории:

**Ошибки уровня переменных**: E0597 (живёт недостаточно долго), E0502 (одновременное изменяемое + неизменяемое заимствование), E0499 (множественные изменяемые заимствования). Rust уже является эталоном — имя переменной + номер строки, без `'a`. Точность YaoXiang на том же уровне. Вся информация в дереве брендов: точка создания токена, позиция потребителя, точка запроса.

**Ошибки уровня сигнатур**: E0623 (несовпадение времён жизни), E0106 (отсутствует спецификатор времени жизни), E0477 (не выполнено required lifetime). Построены вокруг `'a`. YaoXiang **не имеет таких ошибок** — в сигнатурах нет `'a`. Не "не может сообщить", а то, что пользователь не писал, не нужно сообщать.

Пример конфликта внутри функции:

```
ошибка: `data` заморожен, невозможно получить изменяемое право
 --> src/main.yx:5:9
2 |     view = &data
  |            ----- `data` заморожен (токен только-для-чтения создан здесь)
4 |         use(view)
  |             ---- `view` всё ещё используется здесь, заморозка не снята
5 |         data.push(4)
  |         ^^^^ здесь требуется изменяемое право
```

(Точность на уровне Rust E0499 — имя переменной + номер строки, без Brand ID.)

Пример утечки между функциями:

```
ошибка: один из источников данных `num` (строка 4) — `default_str` (строка 3),
но `default_str` становится недействительным в строке 6, а `num` всё ещё
используется в строке 5.

Предложение: поднимите объявление `default_str` в вызывающую сторону или используйте
`ref default_str` для совместного владения.
```

(Точность на уровне Rust E0597. Сводка брендов знает, что у `num` два пути источника — уже есть в компиляторе, формулировка ошибки доступна.)

## Компромиссы

### Преимущества

1. **Сигнатуры типов не содержат времён жизни**: `#42` — это и есть `'42` — та же информация, закодированная в дереве брендов, не выставленная в сигнатуре типа. Это нефальсифицируемо: посчитайте, сколько параметров `'a` нужно в Rust для обобщённого типа с 3 ссылочными параметрами, и сколько нужно в YaoXiang. Ответ: 3 против 0.

2. **Концептуальное единство**: проверка заимствований и пользовательские предикаты разделяют один конвейер доказательств — `{P} op {Q}`, конвейер верифицирует P. Согласовано с Curry-Howard.

3. **Ноль новых аналитических фреймворков**: никаких новых аналитических фреймворков не вводится. Пользователь не ощущает существования "проверщика заимствований" — как не ощущает деталей реализации "проверщика типов".

4. **Сообщения об ошибках содержат только символы, написанные пользователем**: устранено целое измерение категорий ошибок (E0623, E0106, E0477 — все вокруг `'a`). Ошибки уровня переменных — на уровне точности Rust.

5. **Алгоритм не консервативен**: обратный BFS + разрыв break + логический разрыв SMT. Не требуется "консервативная живучесть в циклах". Не требуется "консервативное слияние ветвей".

### Недостатки

1. **Не новое изобретение**: Brand ID делает ровно то же, что `'a` — сложность решения ограничений внутри компилятора не исчезла, только кодировка изменилась с "имя переменной + множество ограничений" на "путь бренда + сопоставление префиксов". Для конечного пользователя разница только в том, что в сигнатуре не пишется `'a`.

2. **Полностью новая реализация**: дерево брендов в коде существует только как концепция, требует реализации с нуля. `BorrowChecker`, `ControlFlowAnalyzer` заменяются.

3. **Зависимость от SMT**: логический разрыв зависит от Z3 (уже введён в RFC-027, новых зависимостей нет). Но проверка заимствований почти не запускает его — только при while + условиях пути.

4. **Крайне редкие паттерны требуют рефакторинга**: межветочные заимствования, которые компилятор не может автоматически доказать, требуют от пользователя рефакторинга кода. В отличие от подстраховки Rust `'a`: в Rust есть `'a` как ручка (аннотация — и проходит); подстраховка YaoXiang (функция-доказательство) не входит в MVP.

| Альтернатива | Почему не выбрана |

## Открытые вопросы

- [x] **Семантика `ref_count` дерева брендов между итерациями при развёртке цикла** — выбран NLL: токен умирает после последнего использования. Копии, связанные в цикле, умирают на границе итерации, обратный BFS не переносит живучесть между итерациями. Подробности см. в §NLL и границы итераций.
- [x] **Порядок освобождения токенов на пути распространения ошибки `?`** — Release управляется анализом области видимости (остаётся в `ir_gen.rs`). В каждой точке выхода области видимости (`}`, `?`, явный `return`) активные токены освобождаются в порядке LIFO. Отношения родитель-потомок дерева брендов автоматически обрабатывают каскадное освобождение. Подробности см. в §`?` распространение ошибок и Release, управляемый областью видимости.
- [ ] Синтаксис функции-доказательства (далёкая перспектива, не MVP — не блокирует ни одну фазу)
