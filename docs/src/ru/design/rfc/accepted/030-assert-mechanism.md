# RFC-030: Механизм assert-утверждений

## Резюме

В YaoXiang вводится механизм `assert`-утверждений для тестирования, проверки предусловий и runtime panic. `assert` и компиляционный уточняющий тип `Assert(C)` (см. RFC-011 §4.3) — это **две стороны одного и того же уточняющего примитива** — dispatch автоматически распределяет его в компиляционное доказательство или runtime-проверку в зависимости от того, доступны ли свободные переменные предиката во время компиляции. `assert(false, "msg")` эквивалентно `raise` и не требует отдельного ключевого слова `throw`/`raise`.

## Мотивация

### Зачем нужна эта возможность?

В настоящее время E2E-тестирование в YaoXiang может только имитировать assert-утверждения с помощью `if` + `io.println` + `return`:

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

Такой подход имеет три проблемы:

1. **Много шаблонного кода**: каждое утверждение занимает 4 строки, тестовые файлы раздуваются
2. **Слабые сообщения об ошибках**: ручная конкатенация строк, отсутствует позиция в исходном коде
3. **Некомпозируемость**: невозможно массово регистрировать утверждения, нельзя передавать их как аргументы в тестовый фреймворк

### Текущие проблемы

- Нет единого механизма assert-утверждений
- Тестовый код заполнен паттерном `if` + печать + `return`
- На уровне байт-кода уже есть инструкция `Throw`, но она не доступна на уровне языка
- RFC-011 определяет компиляционный условный тип `Assert(C)`, но runtime `assert()` ещё не реализован

### Принципы проектирования

`assert` — это единственный пользовательский механизм panic в YaoXiang. `assert(false, "msg")` эквивалентно `raise`, и не требуется отдельного ключевого слова `throw`/`raise`. Сама функция `assert` является лучшей инкапсуляцией `if raise`.

**Никаких новых ключевых слов, никакого нового синтаксиса. Всё — вызовы функций.**

## Вариант A: нативная функция

Реализовать `assert` как нативную функцию, не вводя нового ключевого слова.

```yaoxiang
use std.assert.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Сигнатуры перегрузки

`assert` имеет две перегрузки:

```
// Основная сигнатура: assert — это value-universe-введение для Assert
assert: (cond: Bool, ?msg: String | Error) -> Assert(IsTrue(cond))
//                                       ^^^^^^^^^^^^^^^^^^^^^^^^
//                                       Возвращает уточняющий тип, а не ()
//
// IsTrue: Bool -> Type — мост от значения истинности к типу:
//   IsTrue(true)  = Void   (⊤, программа продолжается)
//   IsTrue(false) = Never  (⊥, расходимость/ошибка компиляции)
```

Фактическое поведение `assert` определяется распределением dispatch:
- Все свободные переменные известны на этапе компиляции → **CompileTime**: компилятор вычисляет cond, true → стирается в Void, false → ошибка компиляции (Never не может быть населена)
- Есть runtime-свободные переменные → **Runtime**: вставляется проверка, в потокочувствительное множество предположений Γ впрыскивается уточняющий факт

Необязательное сообщение `?msg` и перегрузка Result (см. ниже) сохраняются как payload для runtime raise.

#### Перегрузка 1: условное утверждение `(Bool, ?String | Error)`

`Bool` + необязательное сообщение. Сообщение может быть значением `String` или `Error`:

```yaoxiang
assert(1 + 1 == 2)                    // Без сообщения, panic с информацией по умолчанию
assert(1 + 1 == 2, "math is broken")  // Строковое сообщение
assert(x > 0, my_error)                // Прямой бросок значения Error
```

`assert(false, "msg")` — это эквивалент `raise`/`throw` в YaoXiang — отдельное ключевое слово не требуется.

### Преимущества

- **Никаких изменений синтаксиса**: чистая функция, не нужно новое ключевое слово
- **Никаких новых концепций**: повторное использование существующего механизма регистрации нативных функций
- **Высокая расширяемость**: перегрузка функций естественно поддерживает несколько сигнатур
- **Самодокументируемость**: пространство имён `std.assert` само по себе является документацией

### Недостатки

- Их нет. При корректной сигнатуре типа assert компилятор может выводить мёртвый код через анализ достижимости функций. Дополнительные pass не нужны.

### Поведение во время выполнения

1. Вычислить первый параметр `condition: Bool`
2. Если `true`, вернуть `Unit`
3. Если `false`, инициировать runtime panic:
   - Вывести содержимое `message` (если есть)
   - Вывести стек вызовов (в debug-режиме)
   - Завершить текущее выполнение

#### Поведение при сбое для каждой перегрузки

| Сигнатура | Поведение при сбое |
|-----------|--------------------|
| `assert(false)` | Panic с информацией по умолчанию |
| `assert(false, "msg")` | Вывод строкового сообщения с последующим panic |
| `assert(false, error_val)` | Бросок значения Error |
| `assert(Err(x))` | Извлечение содержимого Err и panic |

### Связь с компиляционным Assert

`assert` и `Assert` — **две стороны одного и того же уточняющего примитива** — распределяются автоматически конвейером dispatch в зависимости от того, доступны ли свободные переменные предиката на этапе компиляции:

| Условие | Распределение | Поведение |
|---------|---------------|-----------|
| Все свободные переменные известны на этапе компиляции | CompileTime → конвейер доказательств | Proved → стирается, Disproved → ошибка компиляции, Unknown → требуется доказательство |
| Есть runtime-свободные переменные | Runtime → вставка check | Bool-проверка + впрыскивание уточняющего факта в потокочувствительное множество предположений Γ |

```yaoxiang
use std.assert

# Известно на этапе компиляции (параметр дженерика) — идёт через CompileTime, нулевые runtime-затраты
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: assert.Assert(N > 0),   # N — параметр дженерика, вычисляется на этапе компиляции
}

# Runtime-значение — идёт через Runtime, вставляется Bool-проверка
x = read_int()
assert.assert(x > 0, "expected positive")  # runtime check
```

> **2026-07-12 Унифицированное решение**: предыдущий вывод о «полной независимости» заменён. `assert()` — это value-введение для `Assert`, dispatch выполняет автоматическое распределение.

### Изменения в компиляторе

**Изменения в parser, AST, typecheck, IR gen не требуются.**

Достаточно добавить регистрацию нативной функции в `src/std/`:

1. Добавить `src/std/assert.rs`
2. Зарегистрировать `std.assert.assert` и `std.assert.Assert` (последний — компиляционный условный тип, см. #155)
3. Внутри вызвать существующую инструкцию `BytecodeInstr::Throw`

### Преимущества

- **Никаких изменений синтаксиса**: чистая функция, не нужно новое ключевое слово
- **Никаких новых концепций**: повторное использование существующего механизма регистрации нативных функций
- **Высокая расширяемость**: сигнатура функции может быть расширена до вариантов `assert_eq` и т.п. (в будущем)
- **Самодокументируемость**: пространство имён `std.assert` само по себе является документацией

### Недостатки

- ~~Невозможность компиляционного анализа: в отличие от варианта B (ключевое слово), нельзя выполнить устранение мёртвого кода на этапе компиляции~~ → **В рамках унифицированного решения это неверно**. assert в режиме CompileTime проходит через конвейер доказательств: cond, известный на этапе компиляции → стирается или приводит к ошибке компиляции (`assert(false)` → Never → мёртвый код).
- Стек вызовов доступен только в debug-режиме

## Вариант B: встроенное ключевое слово (заменён унифицированным решением)

> Устарело. Противопоставление вариантов A и B снимается конвейером распределения dispatch — assert является value-введением для Assert: cond, известный на этапе компиляции, проходит через конвейер доказательств (нулевые runtime-затраты), cond во время выполнения — через check. Не нужно выбирать между «функцией» и «ключевым словом». Ниже приведена историческая запись.

```yaoxiang
assert(1 + 1 == 2, "math is broken")
```

### Сигнатура типа

Нет отдельной сигнатуры типа — ключевое слово обрабатывается парсером.

### Поведение во время выполнения

Аналогично варианту A.

### Изменения в компиляторе

Требуются изменения в parser, AST, typecheck, IR gen:
1. parser: новый вариант `Expr::Assert`
2. AST: новый узел `Expr::Assert`
3. typecheck: проверка типов аргументов
4. IR gen: генерация `BytecodeInstr::Throw`

### Преимущества

- Позиция в исходном коде известна на этапе компиляции (не зависит от debug-информации)
- Возможно компиляционное свёртывание констант: `assert(true)` → пустая операция, `assert(false)` → ошибка компиляции

### Недостатки

| Недостаток | Влияние |
|------------|---------|
| Требуются изменения в парсере | Введение нового узла синтаксиса, увеличение стоимости поддержки |
| Ключевое слово нерасширяемо | Варианты вроде `assert_eq` всё равно требуют функций |
| Преимущество на этапе компиляции не реализуемо | См. анализ ниже |

### Сравнение

| Измерение | Вариант A (функция) | Вариант B (ключевое слово) |
|-----------|---------------------|---------------------------|
| Стоимость реализации | ~20 строк | parser + AST + typecheck + IR gen |
| Изменения синтаксиса | Нет | Новое ключевое слово |
| Расширяемость | Перегрузка функций | Требуется сопутствующий макрос |
| Позиция в исходном коде | Debug-информация | Доступна на этапе компиляции |
| Свёртывание констант | Требует поддержки pass | Доступно на этапе компиляции |
| Runtime-затраты | Вызов функции | Минимальны |

### Реалистичные ограничения компиляционного анализа

Главное преимущество варианта B — компиляционный анализ — требует **pass свёртывания констант**, чтобы стать реальным. То есть компилятору нужно на этапе компиляции вычислить `false` в `assert(false)`, чтобы узнать, что это мёртвый код.

В настоящее время у YaoXiang нет pass свёртывания констант. Даже при использовании варианта B `assert(x > 0)` — типичная запись — по-прежнему не может быть проанализирована на этапе компиляции. Анализу поддаются только литералы вроде `assert(true)` / `assert(false)`.

Таким образом, преимущество варианта B на этапе компиляции **на текущем этапе теоретическое, а не практическое**.

### 2026-07-05: Выбор варианта A (заменён унифицированным решением)

Реализация варианта A в 20 строк победила по соотношению ценности и стоимости. После определения унифицированного решения 2026-07-12 противопоставление A/B снимается конвейером распределения dispatch — assert является value-введением для Assert, и больше не нужно выбирать между «функцией» и «ключевым словом».

### 2026-07-12: Определено унифицированное решение (заменяет вывод от 2026-07-11 о «полной независимости»)

**Вывод**: `assert` и `Assert` — не два независимых механизма. `assert: (Bool) -> Assert(IsTrue(cond))` — dispatch выполняет автоматическое распределение:

- Известно на этапе компиляции → вход в конвейер доказательств (Proved стирается / Disproved — ошибка / Unknown — требуется доказательство)
- Runtime-вход → вставка check + впрыскивание предположения в Γ

**Структура модуля**: `std.assert` единообразно несёт runtime-утверждения (`assert`) и компиляционные уточняющие типы (`Assert`, `IsTrue`). Больше не «отдельные реализации», а две стороны одного примитива.

### 2026-07-11: Проектирование перегрузок assert

**Вопрос**: Почему `assert` требует двух перегрузок, а не единой `(Bool, ?String)`?

**Ответ**:

Runtime `assert()` — это единственный пользовательский механизм panic в YaoXiang. `assert(false, "msg")` эквивалентно `raise`/`throw` в других языках. Поэтому он должен покрывать три сценария:
1. Условие + простое сообщение: `assert(cond, "msg")`
2. Условие + пользовательский Error: `assert(cond, my_error)`
3. Проверка Result: `assert(result)` — самый краткий `if is_err { panic }`

Обоснованность перегрузки Result состоит в том, что это кратчайший путь распространения ошибки — «Result должен быть Ok, иначе смерть». Не нужно сначала вызывать `.is_ok()`, а затем отдельно обрабатывать ошибку.
