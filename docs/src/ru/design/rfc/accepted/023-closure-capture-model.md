# RFC-023: Модель захвата замыканий

> **Статус**: Принято
> **Автор**: Чэнь Сюй
> **Дата создания**: 2026-05-29
> **Последнее обновление**: 2026-05-29

> **Ссылки**:
> - [RFC-007: Унификация синтаксиса функций](./accepted/007-function-syntax-unification.md)
> - [RFC-009: Модель владения v9](./accepted/009-ownership-model.md)

## Резюме

Данный RFC определяет **модель захвата замыканий** языка YaoXiang. Компилятор автоматически анализирует внешние переменные, на которые ссылается тело замыкания, и в зависимости от типа переменной (Dup/не-Dup) и от того, является ли замыкание неутекающим, автоматически выбирает способ захвата — для типов Dup выполняется прямое копирование, для не-Dup без утечки — заимствование, для не-Dup с утечкой — Move. Пользователю не требуются аннотации; используется единый набор правил с автоматическим выбором заимствования при вызове функций.

## Мотивация

### Зачем это нужно?

В настоящее время захват замыканий **не реализован** — поле `env` инструкции `MakeClosure` всегда пустое, и lambda не может ссылаться на внешние переменные. Система токенов заимствования требует, чтобы замыкания могли захватывать токены `&T` (нуль-стоимостное копирование), что является ключевым сценарием использования.

### Текущая проблема

```yaoxiang
# Такой код сейчас не компилируется — lambda не может ссылаться на threshold
filter_by: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)  # ❌ threshold не может быть захвачен
}
```

## Предложение

### Основной дизайн

Захват замыканий полностью автоматически определяется компилятором. Правила **полностью идентичны** правилам автоматического выбора заимствования при вызове функций:

```
Тип переменной    Замыкание утекает    Способ захвата
───────────────────────────────────────────────────────
Dup               всегда               Копирование (побитовое копирование или нуль-стоимость)
не-Dup            не утекает           Автоматическое заимствование (&T или &mut T)
не-Dup            утекает              Move (передача владения)
```

**Определение утечки**:

```
spawn { || ... }           → утекает
return || ...              → утекает
let x = || ... ;  x  в поле → утекает
items.filter(|p| ...)      → не утекает (вызов синхронной функции высшего порядка)
||.method()                → не утекает (вызов на месте)
```

Консервативный принцип: при невозможности определения считать утекающим.

### Примеры

```yaoxiang
# 1. Токен Dup — прямое копирование (нуль-стоимость)
filter_by: (items: List(Point), threshold: &Float) -> List(Point) = {
    # threshold: &Float → Dup → компилятор копирует токен в замыкание
    # Токен нулевого размера, нулевые накладные расходы во время выполнения
    items.filter(|p| p.x > threshold)
}

# 2. Не-Dup + не утекает — автоматическое заимствование
process: (buf: Buffer) -> Void = {
    # buf не Dup, filter не утекает → автоматическое создание токена &Buffer
    transform(|b| b.read())
    # После возврата замыкания токен освобождается, buf снова доступен
}

# 3. Замыкание утекает — Move
spawn_worker: (data: Data) -> Void = {
    # data не Dup, spawn → утекает → Move
    spawn { use(data) }
}

# 4. Смешанный захват
complex: (items: List(Point), config: &Config, buf: Buffer) -> List(Point) = {
    # config: &Config → Dup → копирование токена
    # buf: Buffer → не Dup, не утекает → заимствование &mut Buffer
    items.filter(|p| {
        let threshold = config.get_threshold()
        buf.update(p)
        p.x > threshold
    })
}

# 5. Обнаружение конфликтов заимствования
bad: (buf: Buffer) -> Void = {
    closure = |b| b.write()
    buf.read()  # ❌ buf уже заимствован замыканием, конфликт
}
```

### Синтаксические изменения

**Нуль-синтаксических изменений**. Способ захвата определяется компилятором автоматически, пользователю не требуются аннотации.

## Детальный дизайн

### Влияние на систему типов

Сигнатура типа Lambda остаётся без изменений: `(params) -> Return`. Захваченные переменные не отражаются в сигнатуре типа; они обрабатываются компилятором на этапе генерации IR.

### Изменения в компиляторе

| Компонент | Изменение | Описание |
|----------|-----------|----------|
| `capture.rs` (создать) | Анализ захвата + анализ утечки + выбор шаблона | ~150 строк |
| `expressions.rs` | Вызов анализа захвата при выводе типа lambda | ~10 строк |
| `ir_gen.rs` | Заполнение env в MakeClosure; пропуск ZST | ~80 строк |
| `ir.rs` | Возможно, потребуется корректировка типа env в MakeClosure | ~5 строк |

**Процесс анализа захвата**:

```
1. Обход AST тела lambda
2. Сбор всех ссылок Expr::Var(name)
3. Фильтрация: только переменные из внешней области видимости замыкания
4. Классификация: Read (только чтение) / Write (чтение-запись) / Move (перемещён)
5. Проверка атрибутов типа: является ли Dup
6. Определение утечки: способ использования замыкания
7. Выбор режима захвата:
   Dup → Copy
   не-Dup + не утекает + Read → Borrow (&T)
   не-Dup + не утекает + Write → BorrowMut (&mut T)
   не-Dup + утекает → Move
```

**Генерация IR**:

```rust
// Текущее (пусто)
Instruction::MakeClosure { dst, func, env: Vec::new() }

// Изменить на
Instruction::MakeClosure { dst, func, env: captured_env }

// Логика генерации captured_env:
for captured in captures {
    match captured.mode {
        Copy if is_zst(captured.ty) => {
            // Тип нулевого размера — не генерируется никаких инструкций
            // Замыкание напрямую ссылается на внешнюю переменную (устранение на этапе компиляции)
        }
        Copy => {
            // Генерируется Move dst, src (поверхностное копирование типа Dup)
        }
        Borrow => {
            // Генерируется Borrow dst, src (создание ReadToken)
        }
        BorrowMut => {
            // Генерируется Borrow dst, src (создание WriteToken)
        }
        Move => {
            // Генерируется Move dst, src (передача владения)
        }
    }
}
```

### Поведение во время выполнения

Способ захвата не влияет на производительность во время выполнения:

- **Dup + ZST** (например, токены `&T`) → нуль-инструкций, замыкание напрямую ссылается на внешнюю переменную
- **Dup + не-ZST** (например, Int) → одно копирование в регистр
- **Borrow/BorrowMut** → создание токена (концепция времени компиляции, нулевые накладные расходы)
- **Move** → та же стоимость, что и обычное Move

### Обратная совместимость

Полностью совместим. В настоящее время все lambda не могут захватывать внешние переменные; данный RFC только добавляет выразительность, не нарушая существующий код.

## Компромиссы

### Преимущества

1. **Нуль-аннотаций**: пользователю не нужно писать никаких аннотаций захвата
2. **Единообразие с вызовами функций**: правила захвата = правила автоматического заимствования при вызове функций
3. **Нуль-стоимость**: захват токенов Dup полностью устраняется на этапе компиляции
4. **Безопасность**: анализ утечки предотвращает use-after-free

### Недостатки

1. **Консервативный анализ утечки**: при невозможности определения выполняется Move, что может быть избыточным
2. **Неявность**: способ захвата не отражён в коде; при отладке необходимо смотреть вывод компилятора

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Явное ключевое слово `move` в стиле Rust | Добавление нового синтаксиса, увеличение когнитивной нагрузки на пользователя |
| Полное Move | Невозможно выразить нуль-стоимостное заимствование токенов |
| Полное заимствование | Утечка замыкания приведёт к висячим ссылкам |
| Ручные аннотации пользователем | Противоречит философии проектирования «полностью автоматический компилятор» |

## Стратегия реализации

### Разделение на этапы

1. **Phase 1**: Анализ захвата (только идентификация ссылок на внешние переменные, без различения способов захвата)
2. **Phase 2**: Анализ утечки + выбор шаблона
3. **Phase 3**: Генерация IR + оптимизация ZST
4. **Phase 4**: Интеграция обнаружения конфликтов заимствования

### Зависимости

- Зависит от RFC-011 (система обобщённых типов, раздел 2.4 маркерный trait Dup/Clone) — необходим trait Dup для определения возможности копирования переменной
- Зависит от RFC-009 v9 (токены заимствования) — режимы захвата Borrow/BorrowMut требуют типов токенов
- После реализации RFC-023 и данного RFC можно приступать к реализации системы токенов заимствования (RFC-009 v9)

### Риски

- Анализ утечки может быть слишком консервативным, что приводит к избыточному Move; можно оптимизировать позже
- Анализ захвата обобщённых замыканий может потребовать дополнительной обработки

## Журнал решений по дизайну

| Решение | Решение | Причина | Дата |
|---------|---------|---------|------|
| Выбор способа захвата | Полностью автоматический | Единые правила с вызовами функций | 2026-05-29 |
| Анализ утечки | Консервативный принцип | При невозможности определения — утечка, безопасность важнее | 2026-05-29 |
