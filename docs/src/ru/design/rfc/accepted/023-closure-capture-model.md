# RFC-023: Модель захвата замыканий

> **Справочные материалы**:
> - [RFC-007: Унификация синтаксиса функций](./accepted/007-function-syntax-unification.md)
> - [RFC-009: Модель владения v9](./accepted/009-ownership-model.md)
> - [RFC-011: Проектирование системы обобщённых типов](./accepted/011-generic-type-system.md) — Раздел 2.4: Встроенный marker trait Dup/Clone

## Аннотация

Настоящий RFC определяет **модель захвата замыканий** языка YaoXiang. Компилятор автоматически анализирует внешние переменные, на которые ссылается тело замыкания, и в зависимости от типа переменной (Dup/не-Dup) и от того, замыкание ускользает или нет, автоматически выбирает способ захвата — для типов Dup выполняется прямое копирование, для не-Dup без ускользания выполняется заимствование, для не-Dup с ускользанием выполняется Move. Пользователю не требуется вводить аннотации, при этом используется единый набор правил совместно с автоматическим выбором заимствования при вызове функций.

## Мотивация

### Зачем это нужно?

В текущей реализации захват замыканий **не реализован** — поле `env` инструкции `MakeClosure` всегда пустое, лямбда-выражения не могут ссылаться ни на какие внешние переменные. Система токенов заимствования требует, чтобы замыкания могли захватывать токены `&T` (нуль-стоимостное копирование), что является ключевым сценарием использования.

### Текущие проблемы

```yaoxiang
# Данный код сейчас не компилируется — лямбда-выражение не может ссылаться на threshold
filter_by: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)  # ❌ threshold не может быть захвачен
}
```

## Предложение

### Основной дизайн

Захват замыканий полностью автоматически определяется компилятором. Правила **полностью идентичны** правилам автоматического выбора заимствования при вызове функций:

```
Тип переменной    Ускользает ли замыкание    Способ захвата
──────────────────────────────────────────────────────────
Dup               Любой                      Копирование (побитовое копирование или нуль-стоимостное)
Не Dup            Не ускользает              Автоматическое заимствование (&T или &mut T)
Не Dup            Ускользает                 Move (передача владения)
```

**Определение ускользания**:

```
spawn { || ... }           → Ускользание
return || ...              → Ускользание
let x = || ... ;  x 存字段 → Ускользание
items.filter(|p| ...)      → Не ускользает (синхронный вызов функции высшего порядка)
||.method()                → Не ускользает (вызов на месте)
```

Консервативный принцип: при невозможности определения считается, что замыкание ускользает.

### Примеры

```yaoxiang
# 1. Токен Dup — прямое копирование (нуль-стоимостное)
filter_by: (items: List(Point), threshold: &Float) -> List(Point) = {
    # threshold: &Float → Dup → компилятор копирует токен в замыкание
    # Токен нулевого размера, нулевые накладные расходы во время выполнения
    items.filter(|p| p.x > threshold)
}

# 2. Не Dup + не ускользает — автоматическое заимствование
process: (buf: Buffer) -> Void = {
    # buf не Dup, filter не ускользает → автоматическое создание токена &Buffer
    transform(|b| b.read())
    # После возврата замыкания токен освобождается, buf снова доступен
}

# 3. Замыкание ускользает — Move
spawn_worker: (data: Data) -> Void = {
    # data не Dup, spawn → ускользание → Move
    spawn { use(data) }
}

# 4. Смешанный захват
complex: (items: List(Point), config: &Config, buf: Buffer) -> List(Point) = {
    # config: &Config → Dup → копирование токена
    # buf: Buffer → не Dup, не ускользает → заимствование &mut Buffer
    items.filter(|p| {
        let threshold = config.get_threshold()
        buf.update(p)
        p.x > threshold
    })
}

# 5. Обнаружение конфликтов заимствования
bad: (buf: Buffer) -> Void = {
    closure = |b| b.write()
    buf.read()  # ❌ buf уже заимствован замыканием, конфликт здесь
}
```

### Синтаксические изменения

**Нуль-синтаксических изменений**. Способ захвата определяется компилятором автоматически, от пользователя не требуются аннотации.

## Детальный дизайн

### Влияние на систему типов

Сигнатура типа лямбда-выражения остаётся неизменной: `(params) -> Return`. Захваченные переменные не отражаются в сигнатуре типа, их обработка выполняется на этапе генерации IR компилятором.

### Изменения в компиляторе

| Компонент | Изменение | Описание |
|-----------|-----------|----------|
| `capture.rs` (новый) | Анализ захвата + анализ ускользания + выбор режима | ~150 строк |
| `expressions.rs` | Вывод типа лямбда вызывает анализ захвата | ~10 строк |
| `ir_gen.rs` | Заполнение env в MakeClosure; пропуск ZST | ~80 строк |
| `ir.rs` | Возможно, потребуется корректировка типа env в MakeClosure | ~5 строк |

**Процесс анализа захвата**:

```
1. Обход AST тела лямбда
2. Сбор всех ссылок Expr::Var(name)
3. Фильтрация: только переменные из внешней области видимости замыкания
4. Классификация: Read (только чтение) / Write (чтение-запись) / Move (перемещён)
5. Проверка атрибутов типа: является ли Dup
6. Определение ускользания: способ использования замыкания
7. Выбор режима захвата:
   Dup → Copy
   Не Dup + не ускользает + Read → Borrow (&T)
   Не Dup + не ускользает + Write → BorrowMut (&mut T)
   Не Dup + ускользает → Move
```

**Генерация IR**:

```rust
// Текущее состояние (пусто)
Instruction::MakeClosure { dst, func, env: Vec::new() }

// После изменений
Instruction::MakeClosure { dst, func, env: captured_env }

// Логика генерации captured_env:
for captured in captures {
    match captured.mode {
        Copy if is_zst(captured.ty) => {
            // Тип нулевого размера — не генерируется никаких инструкций
            // Тело замыкания напрямую ссылается на внешнюю переменную (устраняется на этапе компиляции)
        }
        Copy => {
            // Генерируется Move dst, src (поверхностное копирование для типов Dup)
        }
        Borrow => {
            // Генерируется Borrow dst, src (создание ReadToken)
        }
        BorrowMut => {
            // Генерируется Borrow dst, src (создание WriteToken)
        }
        Move => {
            // Генерируется Move dst, src (передача владения)
        }
    }
}
```

### Поведение во время выполнения

Способ захвата не влияет на производительность во время выполнения:

- **Dup + ZST** (например, токен `&T`) → Нуль инструкций, тело замыкания напрямую ссылается на внешнюю переменную
- **Dup + не ZST** (например, Int) → Одно копирование в регистр
- **Borrow/BorrowMut** → Создание токена (концепция времени компиляции, нулевые накладные расходы)
- **Move** → Та же стоимость, что и обычный Move

### Обратная совместимость

Полностью совместимо. В настоящее время все лямбда-выражения не могут захватывать внешние переменные, данный RFC только расширяет выразительные возможности и не нарушает существующий код.

## Компромиссы

### Преимущества

1. **Нуль-аннотаций**: пользователю не нужно писать никаких аннотаций захвата
2. **Единообразие с вызовами функций**: правила захвата = правила автоматического заимствования функций
3. **Нуль-стоимостность**: захват токенов Dup полностью устраняется на этапе компиляции
4. **Безопасность**: анализ ускользания предотвращает use-after-free

### Недостатки

1. **Консервативный анализ ускользания**: при невозможности определения выполняется Move, что может привести к ненужным перемещениям
2. **Неявность**: способ захвата не отражён в коде, при отладке требуется просмотр вывода компилятора

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Явное ключевое слово `move` в стиле Rust | Вводит новый синтаксис, увеличивает когнитивную нагрузку на пользователя |
| Полный Move | Невозможно выразить нуль-стоимостное заимствование токенов |
| Полное заимствование | Ускользание замыкания приведёт к висячим ссылкам |
| Ручные аннотации пользователем способа захвата | Противоречит философии проектирования «полностью автоматический компилятор» |

## Стратегия реализации

### Разделение на этапы

1. **Phase 1**: Анализ захвата (только идентификация ссылок на внешние переменные, без различения способов захвата)
2. **Phase 2**: Анализ ускользания + выбор режима
3. **Phase 3**: Генерация IR + оптимизация ZST
4. **Phase 4**: Интеграция обнаружения конфликтов заимствования

### Зависимости

- Зависит от RFC-011 (система обобщённых типов, раздел 2.4 trait Dup/Clone) — требуется trait Dup для определения возможности копирования переменной
- Зависит от RFC-009 v9 (токены заимствования) — режимы захвата Borrow/BorrowMut требуют типов токенов
- После реализации RFC-023 можно приступать к реализации системы токенов заимствования (реализация RFC-009 v9)

### Риски

- Анализ ускользания может быть слишком консервативным, что приводит к ненужным Move; возможна последующая оптимизация
- Анализ захвата обобщённых замыканий может потребовать дополнительной обработки
