# RFC-011: Дизайн системы обобщений — абстракции с нулевой стоимостью и замена макросов

## Резюме

Настоящий документ определяет **дизайн системы обобщений** языка YaoXiang: абстракции с нулевой стоимостью реализуются за счёт мощных возможностей обобщений, оптимизации на этапе compile-time уменьшают зависимость от макросов, а также предоставляется механизм устранения мёртвого кода.

**Основные положения дизайна**:
- **Унифицированный синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — параметры обобщений и обычные параметры унифицированы
- **Механизм самоописания Type**: `Type` — это языковой особый объект, позиции `Type` в сигнатуре автоматически распознаются и заполняются путём вывода
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения на типы функций
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Обобщения этапа compile-time**: `N: Int` — параметры обобщений-значения, инстанциация констант на этапе compile-time
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Абстракции с нулевой стоимостью: мономорфизация на этапе compile-time, отсутствие накладных расходов в runtime
- Устранение мёртвого кода: анализ графа инстанциации + оптимизации LLVM
- Замена макросов: обобщения покрывают 90% сценариев использования макросов
- Безопасность типов: проверка на этапе compile-time, дружелюбность к IDE
- **Явное лучше неявного**: `Type` самоописывающий, компилятор выполняет автоматический вывод

## Идея «вселенной типов» и типы, зависящие от значений

Система обобщений YaoXiang опирается на **идею «вселенной типов»** — ментальную модель, в которой все концепции языка объединяются в многоуровневую структуру; ключевая инновация — возведение **типов, зависящих от значений** (value-dependent types) в полноправные элементы слоя Type2.

### Что такое тип, зависящий от значений?

**Тип, зависящий от значений** — это тип, который зависит от одного или нескольких **значений** (а не только от других типов). Эти значения могут быть вычислены на этапе compile-time, что обеспечивает гарантии безопасности типов уже на стадии компиляции.

```yaoxiang
# Традиционные обобщения: параметры-типы
List: (T: Type) -> Type

# Тип, зависящий от значений: параметры-значения
Vec: (n: Int) -> Type  # тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # тип матрицы зависит от числа строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционными обобщениями, типы, зависящие от значений в YaoXiang, обладают следующими ключевыми преимуществами:

| Свойство | Традиционные обобщения (C++/Rust) | Типы, зависящие от значений, в YaoXiang |
|------|-------------------|---------------------|
| Значения, от которых зависит тип | Только параметры-типы | Любые значения, в том числе результаты вызова функций |
| Вычисление на этапе compile-time | C++ — ручная специализация шаблонов, в Rust — отсутствует | Автоматическое вычисление на этапе compile-time с гарантией завершимости |
| Вычисления на уровне типов | Шаблонное метапрограммирование (сложное/опасное) | Унифицированный движок вычислений на уровне типов |
| Безопасность типов | В C++ отсутствует, в Rust ограничена | Полная безопасность типов, проверка на этапе compile-time |
| Проверка размерностей | Проверка в runtime или ручная специализация | Проверка размерностей на этапе compile-time, без накладных расходов в runtime |

### Уровни «вселенной типов» и типы, зависящие от значений

Идея «вселенной типов» разделяет концепции языка по семантической роли на разные уровни; типы, зависящие от значений, располагаются на **слое Type2**:

| Уровень | Роль | Примеры |
|------|------|------|
| Type-1 | Значения | `42`, `factorial(5)`, сами функции |
| Type0 | Ключевые слова-метатипы | `Type` |
| Type1 | Конкретные типы | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функции/конструкторы типов/типы, зависящие от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевой принцип дизайна**: функции, конструкторы типов и типы, зависящие от значений на слое Type2 используют **унифицированный синтаксис** формы `(params) -> result`:
- Обычная функция: `(Int, Int) -> Int` → результат — значение
- Конструктор типов: `(T: Type) -> Type` → результат — тип
- Тип, зависящий от значений: `(n: Int) -> Type` → результат — тип, но зависящий от параметра-значения

> **Изоморфизм Карри–Ховарда**: это единство — не совпадение. Изоморфизм Карри–Ховарда гласит: «типы — это propositions, программы — это доказательства» — тип функции `A → B` соответствует логической импликации «если A, то B», обобщение `(T: Type) -> Type` соответствует квантору всеобщности «для всех типов T», а тип, зависящий от значений, `(n: Int) -> Type` соответствует утверждению «для каждого целого n существует тип». YaoXiang объединяет функции, конструкторы типов и типы, зависящие от значений, на слое Type2 — тем самым «доказательство» и «вычисление» сводятся к одному и тому же понятию — **конструктивному доказательству**. Это прямое воплощение изоморфизма Карри–Ховарда в дизайне языка: единая форма `(params) -> result` одновременно несёт в себе логическое propositions и вычислительный процесс.

### Гарантия детерминированности на этапе compile-time

Идея «вселенной типов» YaoXiang требует: **всё на слое Type детерминировано на этапе compile-time**.

```yaoxiang
# Пример проверки размерности на этапе compile-time
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка на этапе compile-time: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется на этапе compile-time
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Вычисление на этапе compile-time: factorial(3) = 6, размер вектора известен на этапе compile-time
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов.
2. Проверяет, размечена ли функция спецификацией `decreases` (см. механизм проверки завершимости ниже).
3. Выполняет вычисление на этапе compile-time.
4. Встраивает результат в генерируемый тип.

### Сценарии применения типов, зависящих от значений

#### Проверка размерностей на этапе compile-time
```yaoxiang
# Умножение матриц: проверка соответствия размерностей на этапе compile-time
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка на этапе compile-time: a.Cols == b.Rows, иначе — ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка обнаруживается на этапе compile-time:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива — константа этапа compile-time
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — константа этапа compile-time, может использоваться в вычислениях на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно на этапе compile-time)
```

#### Условные типы
```yaoxiang
# Уровневый If для типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейства типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры-типы T, R фиксируются на этапе compile-time
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    return result
}

# При использовании всё прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # выводится как map[Int, Int]
```

### Механизм проверки завершимости (интеграция с RFC-022)

Вычисление типов, зависящих от значений, на этапе compile-time должно **гарантированно завершаться**, иначе система типов попадёт в бесконечный цикл. YaoXiang обеспечивает это посредством **спецификации decreases**, бесшовно интегрируясь с RFC-022.

#### Спецификация завершимости для рекурсивных функций
```yaoxiang
# Факториал на этапе compile-time: необходимо доказать завершимость
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # на каждом шаге рекурсии n строго уменьшается
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # вычисление на этапе compile-time: factorial(5) = 120
```

#### Спецификация завершимости для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Порядок работы проверки завершимости

```
┌─────────────────────────────────────────────────────────────┐
│  Этап проверки типов                                        │
│  Обнаружен вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка спецификации decreases                          │
│     - есть decreases: проверка выполнения условия убывания   │
│       на всех путях рекурсии                                │
│     - нет decreases, но очевидная завершимость:              │
│       вычислить напрямую                                    │
│     - нет decreases и возможна незавершимость: ошибка        │
│       компиляции                                            │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисление на этапе compile-time (встроенный интерпретатор) │
│     - Чистая функция: прямое вычисление                     │
│     - Побочные эффекты: ошибка компиляции (в позиции типа   │
│       побочные эффекты запрещены)                           │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                            │
│     - Vec(factorial(5)) → Vec(120)                          │
│     - Matrix(Float, 3, 3) → конкретный тип                  │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантия того, что вычисление на этапе compile-time всегда завершится, и система типов не попадёт в бесконечный цикл.
- **Единство**: проверка завершимости и верификация частичной корректности используют единый механизм спецификаций.
- **Постепенное усиление**: можно постепенно переходить от проверок в runtime к полному статическому доказательству.

## Мотивация

### Зачем нужна мощная система обобщений?

Существующие mainstream-языки имеют ограничения в обобщениях:

| Язык | Возможности обобщений | Проблема |
|------|----------|------|
| Java | Граничные типы | Мономорфизация на этапе compile-time, без специализации |
| C# | Ограничения обобщений | Проверка типов в runtime, накладные расходы |
| Rust | Обобщения + Trait | Сложная система Trait, крутая кривая обучения |
| C++ | Шаблоны | Сложная специализация шаблонов, плохие сообщения об ошибках |
| **YaoXiang** | **Типы, зависящие от значений** | **Типы могут зависеть от значений, проверка размерностей на этапе compile-time, гарантия завершимости** |

### Ключевые противоречия

1. **Производительность vs гибкость**: гибкость в runtime vs оптимизации на этапе compile-time
2. **Сложность vs простота**: мощная система типов vs удобство использования
3. **Макросы vs обобщения**: кодогенерация макросами vs типобезопасные обобщения
4. **Зависимость от значений vs безопасность типов**: традиционные обобщения не позволяют проверять размерности на этапе compile-time

### Ключевые преимущества типов, зависящих от значений

**Типы, зависящие от значений** — центральное преимущество YaoXiang по сравнению с традиционными обобщениями:

| Преимущество | Пояснение |
|------|------|
| **Типы зависят от значений** | `Vec: (n: Int) -> Type` позволяет типу зависеть от конкретного значения |
| **Вычисление на этапе compile-time** | Вызовы функций в позициях типов вычисляются на этапе compile-time, результат непосредственно встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` — проверка размерностей матрицы на этапе compile-time |
| **Вычисления на уровне типов** | Условные типы `If`, `Match` поддерживают вычисления на уровне типов |
| **Гарантия завершимости** | Спецификация decreases обеспечивает гарантированное завершение вычислений на этапе compile-time |

```yaoxiang
# То, что невозможно в C++/Rust — проверка на этапе compile-time
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Вычисление на этапе compile-time: factorial(3) = 6, factorial(2) = 2
# Тип — Matrix(Float, 6, 2)

# Несоответствие размерностей обнаруживается на этапе compile-time
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщений

```yaoxiang
# Пример: унифицированный дизайн API
# Операция map для разных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Подход с обобщениями: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели дизайна

### Основные цели

1. **Абстракции с нулевой стоимостью** — вызов обобщённой функции эквивалентен вызову конкретной
2. **Устранение мёртвого кода** — анализ на этапе compile-time, инстанциация только используемых обобщений
3. **Замена макросов** — обобщения покрывают 90% сценариев использования макросов
4. **Безопасность типов** — проверка на этапе compile-time, без накладных расходов в runtime
5. **Дружелюбность к IDE** — интеллектуальные подсказки, ясные сообщения об ошибках
6. **Типы, зависящие от значений** — типы могут зависеть от значений, поддерживается проверка размерностей на этапе compile-time
7. **Безопасность вычислений на этапе compile-time** — спецификация decreases гарантирует завершимость вычислений на этапе compile-time

### Принципы дизайна

- **Детерминированность на этапе compile-time** — параметры обобщений фиксируются на этапе compile-time
- **Приоритет мономорфизации** — генерация конкретного кода, без вызовов виртуальных функций
- **Ограничения как движущая сила** — ограничения типов направляют инстанциацию
- **Платформенные оптимизации** — специализация с поддержкой платформенно-специфичных оптимизаций
- **Единство «вселенной типов»** — функции, конструкторы типов и типы, зависящие от значений, объединены на слое Type2
- **Гарантия завершимости** — вызовы функций в позициях типов должны иметь доказательство завершимости

### 1. Базовые обобщения

#### 1.1 Параметры-типы обобщений

> **Ключевое правило**: определение обобщённого типа **должно явно аннотироваться `: Type`**, иначе HM-вывод интерпретирует его как функцию.
>
> | Запись | Значение |
> |------|------|
> | `List: (T: Type) -> Type = {...}` | ✅ конструктор типа |
> | `List = {...}` | ❌ HM-вывод интерпретирует как функцию, а не как тип |

```yaoxiang
# Определение обобщённого типа (обязательно : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — лишь имя по соглашению, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM-вывод интерпретирует как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Ограничение обобщений (прямое выражение, в одну строку return можно опустить)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров-типов
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 2.4 Встроенные marker-трейты: Dup и Clone

**Три вида семантики копирования**:

| Тип | Значение | Способ срабатывания | Сценарий применения |
|------|------|----------|----------|
| **Копирование примитивного значения** | При присваивании автоматическое копирование значения, оба значения полностью независимы | Автоматически при присваивании/передаче параметров | Int, Float, Bool, Char |
| **Dup** | Поверхностное копирование: копируется дескриптор/токен, базовые данные разделяются | Автоматически при присваивании/передаче параметров | токены `&T`, `ref T`, String/Bytes |
| **Clone** | Глубокое копирование: создаётся полностью независимая копия | `value.clone()` | любой тип, реализующий Clone |

**Семантика Dup**: для типов, реализующих Dup, при присваивании/передаче параметров владение не передаётся — компилятор копирует дескриптор/токен, и несколько владельцев указывают на одни и те же базовые данные. Это дополняет Move-семантику по умолчанию из модели владения RFC-009.

**Dup и Clone — ортогональные концепции**:

```
Dup  = копирование дескриптора, разделяемые данные (изменения влияют друг на друга)
Clone = копирование данных, копия независима (изменения не влияют друг на друга)
```

**Правила**:

```
1. Примитивные типы значений (Int, Float, Bool, Char) — встроенное в компилятор копирование значений, не относятся к Dup
2. Dup  — применимо только к ссылочным/токеновым типам и к типам с внутренним подсчётом ссылок
3. Clone — явное глубокое копирование, может реализовать любой тип
4. Move по умолчанию — остальные типы сохраняют Move-семантику по умолчанию
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|------|-----|------|
| `&T` (токен заимствования) | ✅ | Токен нулевого размера, копирование токена = несколько представлений указывают на одни данные |
| `ref T` | ✅ | Копирование Rc/Arc = счётчик ссылок+1, разделяемые данные в куче |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, копирование дескриптора разделяет базовый буфер |
| `&mut T` (токен изменяемого заимствования) | ❌ | Линейная эксклюзивность, копировать нельзя |
| struct | производный | Все поля Dup → struct Dup |
| enum | производный | Все поля всех вариантов Dup → enum Dup |
| tuple | производный | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захваченное окружение может не быть Dup |
| `*T` (сырой указатель) | ❌ | unsafe, не участвует в системе владения |

**Int/Float/Bool/Char не являются Dup** — это типы значений, при присваивании компилятор автоматически копирует значение (оба значения полностью независимы). Это не «поверхностное копирование», а встроенная в компилятор обработка примитивов; её не нужно и не следует выражать через атрибут Dup.

```yaoxiang
# Примитивные типы значений: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x          # копирование значения, x и y полностью независимы
print(x)       # ✅

# Dup: поверхностное копирование, копия дескриптора разделяет данные
view: &Point = &point
view2 = view    # ✅ Dup: копия токена, оба указывают на один и тот же point
print(view.x)   # ✅

# Clone: явное глубокое копирование, создаётся независимая копия
backup = big_struct.clone()  # явный вызов

# Ограничения обобщений
dup_use: (T: Dup) -> T = x         # T: Dup → можно поверхностно копировать
clone_use: (T: Clone) -> T = x.clone()  # T: Clone → можно глубоко копировать
```

> **Замечание**: `Send`/`Sync` не предоставляются как видимые пользователю trait-ы. Безопасность передачи между задачами полностью обеспечивается ключевым словом `ref` и компилятором — `ref` автоматически выбирает Rc или Arc, и пользователю не нужно понимать Send/Sync.

}

        item = self.data[self.index]
        self.index = self.index + 1
        return Option.some(item)
    } else {
        return Option.none()
    }
}

    return arr.data[0]
}

### 5. Условные типы

> **Изоморфизм Карри–Ховарда**: с точки зрения Карри–Ховарда условные типы — это **case-анализ** в логике. Тип `Bool` соответствует propositions с двумя возможными значениями (True/False), а `If` в зависимости от истинности/ложности этого propositions выбирает различные результаты — это в точности логическая дизъюнкция случаев. `match C { True => T, False => E }` фактически выражает: «если propositions C истинно, заключение — T; если ложно — E».

#### 5.1 Условный тип If

```yaoxiang
# Уровневый If для типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление на этапе compile-time
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Проверка на этапе compile-time
Assert: (C: Bool) -> Type = match C {
    True => Void,
    False => compile_error("Assertion failed"),
}

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри–Ховарда**: семейства типов — наиболее прямое воплощение «propositions как типов». `Add: (A: Type, B: Type) -> Type` — это не «функция сложения, записанная на уровне типов», а **построение propositions о сложении натуральных чисел**. `(Zero, B) => B` утверждает: «propositions Add(Zero, B) эквивалентно B»; `(Succ(A'), B) => Succ(Add(A', B))` утверждает: «если Add(A', B) истинно, то и Add(Succ(A'), B) истинно». Это и есть определение сложения в аксиомах Пеано. Проверка типов подтверждает корректность этого match-выражения, что эквивалентно проверке логической согласованности данного определения.

```yaoxiang
# Преобразование типов на этапе compile-time
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # по умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Карри–Ховард: это индуктивное определение сложения натуральных чисел)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: вычисление 2 + 3 на этапе compile-time
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

struct InstantiationGraph {
    // Узлы: инстанциации обобщений
    nodes: HashMap<InstanceKey, InstanceNode>,

    // Рёбра: отношения использования
    edges: HashMap<InstanceKey, Vec<InstanceKey>>,

    generic: FunctionId,  // ID обобщённой функции
    type_args: Vec<TypeId>,  // аргументы-типы
    const_args: Vec<ConstId>,  // аргументы-const
}

// Алгоритм: анализ достижимости
fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Не посещённые инстанциации — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}

# Используется синтаксис (T: Type) -> Type
Result: (T: Type, E: Type) -> Type = {

    data: Array(T),
    length: Int,

    # Обобщённые методы (T автоматически попадает в область видимости из внешнего List(T))
    push: (self: List(T), item: T) -> Void,
    pop: (self: List(T)) -> Option(T),
    map: (R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)),
    filter: (self: List(T), predicate: (T) -> Bool) -> List(T),
    fold: (U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U),
}

# ======== 2. Реализация обобщённых методов ========
# Функция определена в пространстве имён List (префикс List. = принадлежность пространству имён)
# Чтобы синтаксис вызова через . работал (например, list.push(item)), требуется явная привязка: List.push = push[0]
# self — лишь имя параметра по соглашению, компилятор смотрит на тип, а не на имя

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

List.pop: (T: Type) -> ((self: List(T)) -> Option(T)) = {
    if self.length > 0 {
        self.length = self.length - 1
        return Option.some(self.data[self.length])
    } else {
        return Option.none()
    }
}

List.map: (T: Type, R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)) = {
    result = List(R)()
    for i in 0..self.length {
        result.push(f(self.data[i]))
    }
    return result
}

List.filter: (T: Type) -> ((self: List(T), predicate: (T) -> Bool) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        if predicate(self.data[i]) {
            result.push(self.data[i])
        }
    }
    return result
}

List.fold: (T: Type, U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U) = {
    result = initial
    for i in 0..self.length {
        result = f(result, self.data[i])
    }
    return result
}

# ======== 3. Использование ограничений типов ========
# Реализация Clone для List

    for i in 0..self.length {

    }
    return result
}

# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённая быстрая сортировка
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Реализуется через перегрузку функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========

# ======== 1. Тип матрицы, известной на этапе compile-time ========
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),

    # Проверка размерностей на этапе compile-time: используется тип Assert из стандартной библиотеки
    _assert: Assert[Rows > 0],  # Rows > 0, иначе — ошибка компиляции
    _assert: Assert[Cols > 0],  # Cols > 0, иначе — ошибка компиляции

    # Операции над матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {
            for j in 0..M {
                sum = Zero::zero()
                for k in 0..Cols {
                    sum = sum + self.data[i][k] * other.data[k][j]
                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    return matrix
}

matrix_2x3.data[0][1] = 2.0
matrix_2x3.data[0][2] = 3.0
matrix_2x3.data[1][0] = 4.0
matrix_2x3.data[1][1] = 5.0
matrix_2x3.data[1][2] = 6.0

# Матрица 3x2
matrix_3x2 = Matrix(Float, 3, 2)()
matrix_3x2.data[0][0] = 7.0
matrix_3x2.data[0][1] = 8.0
matrix_3x2.data[1][0] = 9.0
matrix_3x2.data[1][1] = 10.0

matrix_3x2.data[2][1] = 12.0

# Умножение матриц: 2x3 * 3x2 = 2x2
result = matrix_2x3.multiply(matrix_3x2)

# Проверка на этапе compile-time: тип result — Matrix(Float, 2, 2)
# Единичная матрица 3x3
identity_3x3 = identity(Float, 3)()

## Компромиссы

### Преимущества

1. **Абстракции с нулевой стоимостью**
   - Мономорфизация на этапе compile-time, отсутствие накладных расходов в runtime
   - Не нужны виртуальные функции, не нужен RTTI

2. **Устранение мёртвого кода**
   - Анализ на этапе compile-time, инстанциация только используемых обобщений
   - Раздувание кода контролируемо

3. **Замена макросов**
   - Типобезопасная кодогенерация
   - Дружелюбность к IDE, ясные сообщения об ошибках

4. **Вычисления на этапе compile-time**
   - Обобщения этапа compile-time поддерживают вычисления на этапе compile-time
   - Такие свойства, как проверка размерностей
   - Не нужно ключевое слово `const`, только ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанциация обобщений увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Потребление памяти**
   - Возрастает потребление памяти компилятором
   - Механизмы кэширования требуют памяти

3. **Сложность реализации**
   - Сложный решатель ограничений
   - Сложный движок вычислений на уровне типов

4. **Диагностика ошибок**
   - Ошибки обобщений могут быть сложными
   - Нужны ясные подсказки

### Меры по смягчению

1. **Стратегии кэширования**
   - Кэширование результатов инстанциации
   - LRU-кэш для ограничения памяти

2. **Инкрементальная компиляция**
   - Кэширование результатов компиляции
   - Инкрементальная инстанциация

3. **Подсказки при ошибках**
   - Ясные сообщения об ошибках
   - Подсказки по выводу параметров обобщений

4. **Параллельная компиляция**
   - Параллельная инстанциация обобщений
   - Многопоточное решение ограничений

## Альтернативы

| Решение | Почему не выбрано |
|------|--------------|
| Только базовые обобщения | Не могут заменить сложные макросы |
| Чистая система макросов | Нет безопасности типов, плохие сообщения об ошибках |
| Только ограничения-зависимости | Недостаточная гибкость |
| Обобщения в runtime | Накладные расходы на производительность |

### Риски

| Риск | Влияние | Мера по смягчению |
|------|------|------|
| Сложность решения ограничений | Слишком долгая компиляция | Инкрементальное решение + кэширование |
| Раздувание кода | Чрезмерно большой бинарник | DCE + контроль порогов |
| Сложность реализации | Удлинение цикла разработки | Поэтапная реализация |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

## Приложение

### Синтаксис BNF

```bnf
# Параметры обобщений используют унифицированный синтаксис () как часть типа функции
# Например: map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в параметрах обобщений)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры обобщений — первая группа параметров в типе функции: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённый тип, например: List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Параметры Type в обобщениях автоматически заполняются компилятором из типов фактических аргументов
# Например, в map(numbers, f) T извлекается из numbers: List(Int), R — из f: (Int) -> String
```
