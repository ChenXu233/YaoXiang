# RFC-011: Система обобщений — абстракции с нулевой стоимостью и замена макросов

## Резюме

Настоящий документ определяет **проект системы обобщений** языка YaoXiang — мощные возможности обобщённого программирования для достижения абстракций с нулевой стоимостью, снижения зависимости от макросов за счёт оптимизаций на этапе компиляции, а также механизм устранения мёртвого кода.

**Ключевые решения**:
- **Единый синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — параметры обобщений и обычные параметры унифицированы
- **Механизм самоописания Type**: `Type` — особая сущность языкового уровня, позиции `Type` в сигнатуре заполняются автоматическим выводом
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения на функции
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Обобщения на этапе компиляции**: `N: Int` — параметры-значения обобщений, инстанцирование константами на этапе компиляции
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Абстракции с нулевой стоимостью: мономорфизация на этапе компиляции, без накладных расходов во время выполнения
- Устранение мёртвого кода: анализ графа инстанцирования + оптимизации LLVM
- Замена макросов: обобщения покрывают 90% сценариев, где раньше требовались макросы
- Безопасность типов: проверки на этапе компиляции, удобство для IDE
- **Явное лучше неявного**: самоописание `Type`, автоматический вывод компилятора

## Идея вселенной типов и типы, зависящие от значений

Система обобщений YaoXiang построена на **идее вселенной типов** — ментальной модели, которая унифицирует все концепции языка в многоуровневую структуру. Ключевая инновация — возведение **типов, зависящих от значений** (value-dependent types), в полноправные сущности уровня Type2.

### Что такое типы, зависящие от значений?

**Тип, зависящий от значений** — это тип, который зависит от одного или нескольких **значений** (а не только от других типов). Эти значения могут быть вычислены на этапе компиляции, обеспечивая гарантии безопасности типов уже к моменту компиляции.

```yaoxiang
# Традиционные обобщения: параметры-типы
List: (T: Type) -> Type

# Тип, зависящий от значений: параметр-значение
Vec: (n: Int) -> Type  # тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # тип матрицы зависит от числа строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционными обобщениями, типы, зависящие от значений, в YaoXiang обладают следующими ключевыми преимуществами:

| Свойство | Традиционные обобщения (C++/Rust) | Типы, зависящие от значений в YaoXiang |
|------|-------------------|---------------------|
| Значения, от которых зависит тип | Только параметры-типы | Любые значения, включая результаты вызова функций |
| Вычисление на этапе компиляции | C++ — ручная специализация шаблонов, Rust — нет | Автоматическое вычисление на этапе компиляции с гарантией завершимости |
| Вычисления на уровне типов | Шаблонное метапрограммирование (сложное/опасное) | Унифицированный движок вычислений на уровне типов |
| Безопасность типов | C++ — нет, Rust — ограниченная | Полная безопасность типов, проверки на этапе компиляции |
| Проверка размерностей | Проверка во время выполнения или ручная специализация | Проверка размерностей на этапе компиляции, без накладных расходов |

### Уровни вселенной типов и типы, зависящие от значений

Идея вселенной типов разделяет концепции языка по семантической роли на разные уровни. Типы, зависящие от значений, расположены на **уровне Type2**:

| Уровень | Роль | Примеры |
|------|------|------|
| Type-1 | Значения | `42`, `factorial(5)`, сами функции |
| Type0 | Ключевые слова метатипов | `Type` |
| Type1 | Конкретные типы | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функции / конструкторы типов / типы, зависящие от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевое решение**: функции, конструкторы типов и типы, зависящие от значений, на уровне Type2 имеют **единый синтаксис** — форму `(params) -> result`:
- Обычная функция: `(Int, Int) -> Int` → результат — значение
- Конструктор типов: `(T: Type) -> Type` → результат — тип
- Тип, зависящий от значений: `(n: Int) -> Type` → результат — тип, но зависящий от параметра-значения

> **Изоморфизм Карри-Ховарда**: это единство — не случайность. Изоморфизм Карри-Ховарда утверждает «типы суть propositions, программы суть доказательства» — тип функции `A → B` соответствует логической импликации «если A, то B», обобщение `(T: Type) -> Type` соответствует квантору всеобщности «для всех типов T», тип, зависящий от значений `(n: Int) -> Type`, соответствует «для каждого целого n существует тип». YaoXiang объединяет функции, конструкторы типов и типы, зависящие от значений, на уровне Type2, по сути объединяя «доказательство» и «вычисление» в одно понятие — **конструктивное доказательство**. Это прямое воплощение изоморфизма Карри-Ховарда в проектировании языка: одна форма (`(params) -> result`) одновременно несёт логическую propositions и вычислительный процесс.

### Гарантия детерминированности на этапе компиляции

Идея вселенной типов YaoXiang требует: **всё на уровне Type определено на этапе компиляции**.

```yaoxiang
# Пример проверки размерностей на этапе компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка на этапе компиляции: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется на этапе компиляции
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Вычисление на этапе компиляции: factorial(3) = 6, размер вектора известен на этапе компиляции
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов
2. Проверяет, помечена ли функция спецификацией `decreases` (см. механизм проверки завершимости ниже)
3. Выполняет вычисление на этапе компиляции
4. Встраивает результат в сгенерированный тип

### Сценарии применения типов, зависящих от значений

#### Проверка размерностей на этапе компиляции
```yaoxiang
# Умножение матриц: проверка соответствия размерностей на этапе компиляции
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка на этапе компиляции: a.Cols == b.Rows, иначе — ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка перехватывается на этапе компиляции:
# multiply(matrix_2x3, matrix_4x2)  # ошибка компиляции: 2 != 4
```

#### Размер массива с гарантией типовой безопасности
```yaoxiang
# Размер массива — константа на этапе компиляции
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — константа на этапе компиляции, может использоваться в вычислениях на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно на этапе компиляции)
```

#### Условные типы
```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейства типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры-типы T, R определяются на этапе компиляции
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    return result
}

# При использовании всё прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # выводится как map[Int, Int]
```

### Механизм проверки завершимости (интеграция с RFC-022)

Вычисление на этапе компиляции для типов, зависящих от значений, должно быть **гарантированно завершимым**, иначе система типов уйдёт в бесконечный цикл. YaoXiang обеспечивает это с помощью **спецификации decreases**, бесшовно интегрированной с RFC-022.

#### Спецификация завершимости для рекурсивных функций
```yaoxiang
# Факториал на этапе компиляции: необходимо доказать завершимость
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # при каждой рекурсии n строго уменьшается
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # вычисление на этапе компиляции: factorial(5) = 120
```

#### Спецификация завершимости для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершимости

```
┌─────────────────────────────────────────────────────────────┐
│  Этап проверки типов                                         │
│  Встречается вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка спецификации decreases                            │
│     - decreases указана: проверить, что условие убывания       │
│       выполняется на всех путях рекурсии                       │
│     - decreases отсутствует, но завершимость очевидна:         │
│       вычислить напрямую                                       │
│     - decreases отсутствует и завершимость не очевидна:         │
│       ошибка компиляции                                        │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисление на этапе компиляции (встроенным интерпретатором) │
│     - Чистая функция: вычислить напрямую                       │
│     - Побочные эффекты: ошибка компиляции                      │
│       (в позиции типа побочных эффектов быть не должно)         │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                               │
│     - Vec(factorial(5)) → Vec(120)                            │
│     - Matrix(Float, 3, 3) → конкретный тип                     │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантия завершимости вычислений на этапе компиляции, защита системы типов от бесконечных циклов
- **Единообразие**: проверка завершимости и верификация частичной корректности используют единый механизм спецификаций
- **Постепенное усиление**: можно переходить от проверок во время выполнения к полному статическому доказательству

## Мотивация

### Зачем нужна мощная система обобщений?

Существующие mainstream-языки имеют ограничения в обобщениях:

| Язык | Возможности обобщений | Проблемы |
|------|----------|------|
| Java | Ограниченные типы | Мономорфизация на этапе компиляции, без специализации обобщений |
| C# | Ограничения обобщений | Проверка типов во время выполнения, накладные расходы |
| Rust | Обобщения + Trait | Сложная система Trait, крутая кривая обучения |
| C++ | Шаблоны | Сложная специализация шаблонов, плохие сообщения об ошибках |
| **YaoXiang** | **Типы, зависящие от значений** | **Типы могут зависеть от значений, проверка размерностей на этапе компиляции, гарантия завершимости** |

### Ключевые противоречия

1. **Производительность vs гибкость**: гибкость во время выполнения vs оптимизации на этапе компиляции
2. **Сложность vs простота**: мощная система типов vs удобство использования
3. **Макросы vs обобщения**: кодогенерация макросами vs безопасность типов обобщений
4. **Зависимость от значений vs безопасность типов**: традиционные обобщения не позволяют проверять размерности на этапе компиляции

### Ключевые преимущества типов, зависящих от значений

**Типы, зависящие от значений** в YaoXiang — главное преимущество перед традиционными обобщениями:

| Преимущество | Описание |
|------|------|
| **Типы зависят от значений** | `Vec: (n: Int) -> Type` позволяет типу зависеть от конкретного значения |
| **Вычисление на этапе компиляции** | Вызов функции в позиции типа вычисляется на этапе компиляции, результат встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` — проверка размерностей матрицы на этапе компиляции |
| **Вычисления на уровне типов** | Условные типы `If`, `Match` поддерживают вычисления на уровне типов |
| **Гарантия завершимости** | Спецификация decreases обеспечивает завершимость вычислений на этапе компиляции |

```yaoxiang
# Проверка на этапе компиляции, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Вычисление на этапе компиляции: factorial(3) = 6, factorial(2) = 2
# Тип — Matrix(Float, 6, 2)

# Несовпадение размерностей перехватывается на этапе компиляции
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # ошибка компиляции: 2 != 3
```

### Ценность системы обобщений

```yaoxiang
# Пример: унифицированный дизайн API
# Операция map для разных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Подход с обобщениями: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели проектирования

### Основные цели

1. **Абстракции с нулевой стоимостью** — вызов обобщений эквивалентен вызову конкретного типа
2. **Устранение мёртвого кода** — анализ на этапе компиляции, инстанцируются только используемые обобщения
3. **Замена макросов** — обобщения покрывают 90% сценариев, где раньше требовались макросы
4. **Безопасность типов** — проверки на этапе компиляции, без накладных расходов на типы во время выполнения
5. **Удобство для IDE** — интеллектуальные подсказки, понятные сообщения об ошибках
6. **Типы, зависящие от значений** — типы могут зависеть от значений, поддержка проверки размерностей на этапе компиляции
7. **Безопасность вычислений на этапе компиляции** — спецификация decreases гарантирует завершимость вычислений на этапе компиляции

### Принципы проектирования

- **Определённость на этапе компиляции**: параметры обобщений определяются на этапе компиляции
- **Приоритет мономорфизации**: генерация конкретного кода, без вызовов виртуальных функций
- **Ограничения как движущая сила**: ограничения типов направляют инстанцирование
- **Платформенные оптимизации**: специализация для платформенно-специфичных оптимизаций
- **Единая вселенная типов**: функции, конструкторы типов и типы, зависящие от значений, унифицированы на уровне Type2
- **Гарантия завершимости**: вызовы функций в позиции типа должны доказывать завершимость

### 1. Базовые обобщения

#### 1.1 Параметры обобщённых типов

> **Ключевое правило**: определение обобщённого типа **должно явно содержать `: Type`**, иначе HM-вывод интерпретирует его как функцию.
>
> | Запись | Значение |
> |------|------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типов |
> | `List = {...}` | ❌ HM выводит функцию, а не тип |

```yaoxiang
# Определение обобщённого типа (обязательно : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — лишь соглашение об имени, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM выводит функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Ограничение обобщений (прямое выражение, в одну строку return можно опустить)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров-типов
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 2.4 Встроенные marker trait: Dup и Clone

**Три вида семантики копирования**:

| Тип | Значение | Способ срабатывания | Сценарий применения |
|------|------|----------|----------|
| **Копирование значения-примитива** | При присваивании автоматическое копирование значения, два значения полностью независимы | Автоматически при присваивании/передаче параметра | Int, Float, Bool, Char |
| **Dup** | Поверхностное копирование: копирование дескриптора/токена, базовые данные разделяются | Автоматически при присваивании/передаче параметра | Токены `&T`, `ref T`, String/Bytes |
| **Clone** | Глубокое копирование: создание полной независимой копии | `value.clone()` | Любой тип, реализующий Clone |

**Семантика Dup**: типы, реализующие Dup, при присваивании/передаче параметра не передают владение — компилятор копирует дескриптор/токен, и несколько владельцев указывают на одни и те же базовые данные. Это дополняет семантику Move по умолчанию из модели владения RFC-009.

**Dup и Clone — ортогональные концепции**:

```
Dup = копирование дескриптора, разделение данных (изменения влияют друг на друга)
Clone = копирование данных, независимые копии (изменения не влияют друг на друга)
```

**Правила**:

```
1. Типы-примитивы значений (Int, Float, Bool, Char) — встроенное копирование значений компилятором, не относятся к Dup
2. Dup — применим только к ссылочным/токенным типам и типам с внутренним подсчётом ссылок
3. Clone — явное глубокое копирование, может реализовать любой тип
4. Move по умолчанию — остальные типы сохраняют семантику Move по умолчанию
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|------|-----|------|
| `&T` (токен заимствования) | ✅ | Токен нулевого размера, копирование токена = несколько представлений на одни данные |
| `ref T` | ✅ | Копирование Rc/Arc = счётчик ссылок +1, разделение данных в куче |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, копирование дескриптора разделяет базовый буфер |
| `&mut T` (токен изменяемости) | ❌ | Линейная исключительность, копирование невозможно |
| struct | Производное | Все поля Dup → struct Dup |
| enum | Производное | Все поля всех вариантов Dup → enum Dup |
| tuple | Производное | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захватываемая среда может быть не Dup |
| `*T` (сырой указатель) | ❌ | unsafe, не участвует в системе владения |

**Int/Float/Bool/Char не являются Dup** — это типы значений, при присваивании компилятор автоматически копирует значение (два значения полностью независимы). Это не «поверхностное копирование», а встроенная обработка примитивов компилятором, которую не нужно и не следует выражать через свойство типа Dup.

```yaoxiang
# Типы-примитивы значений: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x          # копирование значения, x и y полностью независимы
print(x)       # ✅

# Dup: поверхностное копирование, копирование дескриптора разделяет данные
view: &Point = &point
view2 = view    # ✅ Dup: копирование токена, оба указывают на один и тот же point
print(view.x)   # ✅

# Clone: явное глубокое копирование, создание независимой копии
backup = big_struct.clone()  # явный вызов

# Ограничения обобщений
dup_use: (T: Dup) -> T = x         # T: Dup → можно поверхностно копировать
clone_use: (T: Clone) -> T = x.clone()  # T: Clone → можно глубоко копировать
```

> **Примечание**: `Send`/`Sync` не предоставляются как видимые пользователю trait. Гарантии безопасности межзадачного взаимодействия обрабатываются полностью автоматически ключевым словом `ref` и компилятором — `ref` автоматически выбирает Rc или Arc, и пользователю не нужно понимать Send/Sync.

### Never и Void: ⊥ и ⊤ в системе типов

Система типов YaoXiang в изоморфизме Карри-Ховарда одновременно обладает ⊥ (ложь/пустой тип) и ⊤ (истина/Unit), представленными двумя встроенными именами типов — `Never` и `Void`:

**Never (⊥)** — три непреложных ядерных свойства:

1. **Нулевое число конструкторов**: ни один литерал или выражение не может породить значение типа `Never`. Это мета-свойство, оно должно быть встроенным.
2. **Принцип взрыва**: `Never <: T` выполняется для любого типа `T`. Значение `Never` может использоваться как значение любого типа — именно поэтому код после `assert(false)` проходит проверку типов (хотя никогда не выполняется).
3. **Маркер расходимости**: `f: (...) -> Never` означает, что `f` гарантированно не возвращается. Компилятор использует это для анализа мёртвого кода.

`Never` — встроенное имя типа, а не ключевое слово, парсер ничего о нём не знает. Синтаксис пустых литералов и литералов типа не предоставляется.

**Void (⊤, т.е. Unit)** — ровно один обитатель (значение `void` по умолчанию), носитель истинной propositions «всегда истинно». `Void` — единица нуль-полевого произведения типов, `Never` — единица нуль-вариантного суммирования типов — они двойственны друг другу. `x: Void = <по умолчанию>` допустимо, `x: Never = ...` — справа писать нечего.

#### 4.3 Верификация на этапе компиляции (реализация стандартной библиотеки)

```yaoxiang
# ════════════════════════════════════════════════════════
# Реализация стандартной библиотеки: использование условных типов
# ════════════════════════════════════════════════════════

# Определения стандартной библиотеки
# IsTrue: мост от вселенной значений к вселенной типов — отображение Bool на тип
IsTrue: (b: Bool) -> Type = match b {
    true => Void,      # ⊤, имеет значение, программа продолжается
    false => Never,    # ⊥, нет значения, расходимость
}

# Assert: типовой примитив уточнения на этапе компиляции — ти́повое выражение propositions Bool
Assert: (cond: Bool) -> Type = IsTrue(cond)
#
# cond == true  → Assert(true)  = Void    (тождественно истинно, стирается)
# cond == false → Assert(false) = Never   (тождественно ложно, ошибка компиляции / расходимость)
# cond неразрешимо → канал доказательства решает по шаблону dispatch:
#                    CompileTime → Unknown, требуется prove
#                    Runtime     → вставить check, внедрить предположение Γ

# Применение 1: использование в определении типа как ограничение
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    # Проверка на этапе компиляции: N должно быть больше 0 (Assert в позиции типа)
    length: Assert(N > 0),
}

# Применение 2: использование в выражении
IntArray: (N: Int) -> Type = StaticArray(Int, N)
# Проверка: размер IntArray(10) равен sizeof(Int) * 10
Assert(size_of(IntArray(10)) == sizeof(Int) * 10)
```

```yaoxiang
# Общая реализация

    result = Zero::zero()
    for item in arr.data {
        result = result + item
    }
    return result
}

# Специализация для N=2

    # Оптимизация компилятора: развёртка цикла
    return arr.data[0] + arr.data[1] + arr.data[2] + arr.data[3]
}
```

### 5. Условные типы

> **Изоморфизм Карри-Ховарда**: условные типы с точки зрения Карри-Ховарда — это **case-анализ** в логике. Тип `Bool` соответствует propositions с двумя возможными значениями (True/False), `If` выбирает разные результаты в зависимости от истинности этой propositions — это и есть логический case-дизъюнкт. `match C { True => T, False => E }` фактически выражает: «если propositions C истинно, заключение — T; если ложно — E».

#### 5.1 Условный тип If

```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Верификация на этапе компиляции (унифицирована с определением Assert в §4.3)
# Assert: (cond: Bool) -> Type = IsTrue(cond)

# Использование
# Вычисление типов: If(True, Int, String) => Int
# Вычисление типов: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри-Ховарда**: семейства типов — самое прямое воплощение принципа «propositions как типы». `Add: (A: Type, B: Type) -> Type` — это не «функция сложения на уровне типов», а **конструирование propositions о сложении натуральных чисел**. `(Zero, B) => B` утверждает «propositions Add(Zero, B) эквивалентно B», `(Succ(A'), B) => Succ(Add(A', B))` утверждает «если Add(A', B) истинно, то и Add(Succ(A'), B) истинно». Это и есть определение сложения в аксиомах Пеано. То, что проверка типов пропускает это выражение match, эквивалентно проверке логической непротиворечивости этого определения.

```yaoxiang
# Преобразование типов на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # по умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Карри-Ховард: case-анализ + рекурсивный вызов, требует проверки завершимости для полной индукции)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: вычисление 2 + 3 на этапе компиляции
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

#### 7.1 Анализ графа инстанцирования

```rust
// Внутри компилятора: построение графа зависимостей инстанцирований обобщений
struct InstantiationGraph {
    // Узлы: инстанцирования обобщений
    nodes: HashMap<InstanceKey, InstanceNode>,

    // Рёбра: отношения использования
    edges: HashMap<InstanceKey, Vec<InstanceKey>>,
}

struct InstanceKey {
    generic: FunctionId,  // ID обобщённой функции
    type_args: Vec<TypeId>,  // аргументы-типы
    const_args: Vec<ConstId>,  // аргументы-константы
}

// Алгоритм: анализ достижимости
fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.п.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Не посещённые инстанцирования — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}
```

#### 9.1 Полный пример обобщённого контейнера

```yaoxiang
# ======== 1. Определение обобщённого контейнера ========
# Использование синтаксиса (T: Type) -> Type
Result: (T: Type, E: Type) -> Type = {

    err: (E) -> Self,
}

}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,

    # Обобщённые методы (T подставляется внешним List(T) в область видимости)
    push: (self: List(T), item: T) -> Void,
    pop: (self: List(T)) -> Option(T),
    map: (R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)),
    filter: (self: List(T), predicate: (T) -> Bool) -> List(T),
    fold: (U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U),
}

# ======== 2. Реализация обобщённых методов ========
# Определение функций в пространстве имён List (List. префикс = принадлежность к пространству имён)
# Чтобы вызов через . работал (например, list.push(item)), требуется явная привязка: List.push = push[0]
# self — лишь соглашение об имени параметра, компилятор смотрит на тип, а не на имя

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

List.pop: (T: Type) -> ((self: List(T)) -> Option(T)) = {
    if self.length > 0 {
        self.length = self.length - 1
        return Option.some(self.data[self.length])
    } else {
        return Option.none()
    }
}

List.map: (T: Type, R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)) = {
    result = List(R)()
    for i in 0..self.length {
        result.push(f(self.data[i]))
    }
    return result
}

List.filter: (T: Type) -> ((self: List(T), predicate: (T) -> Bool) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        if predicate(self.data[i]) {
            result.push(self.data[i])
        }
    }
    return result
}

List.fold: (T: Type, U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U) = {
    result = initial
    for i in 0..self.length {
        result = f(result, self.data[i])
    }
    return result
}

# ======== 3. Использование ограничений типов ========
# Реализация Clone для List
List.clone: (T: Clone) -> ((self: List(T)) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        result.push(self.data[i].clone())
    }
    return result
}
```

#### 9.2 Пример обобщённых алгоритмов

```yaoxiang
# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённая быстрая сортировка
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Использование перегрузки функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========
```

#### 9.3 Пример обобщений на этапе компиляции

```yaoxiang
# ======== 1. Тип матрицы на этапе компиляции ========
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),

    # Проверка размерностей на этапе компиляции: используем тип Assert из стандартной библиотеки
    _assert: Assert[Rows > 0],  # Rows > 0, иначе — ошибка компиляции
    _assert: Assert[Cols > 0],  # Cols > 0, иначе — ошибка компиляции

    # Операции над матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {
            for j in 0..M {
                sum = Zero::zero()
                for k in 0..Cols {
                    sum = sum + self.data[i][k] * other.data[k][j]
                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

# ======== 2. Создание матрицы на этапе компиляции ========
identity: (T: Add + Multiply + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    for i in 0..N {
        for j in 0..N {
            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    return matrix
}

# ======== 3. Примеры использования ========
# Создание матрицы с размером, известным на этапе компиляции
# Матрица 2x3
matrix_2x3 = Matrix(Float, 2, 3)()

matrix_2x3.data[0][1] = 2.0
matrix_2x3.data[0][2] = 3.0
matrix_2x3.data[1][0] = 4.0
matrix_2x3.data[1][1] = 5.0
matrix_2x3.data[1][2] = 6.0

# Матрица 3x2
matrix_3x2 = Matrix(Float, 3, 2)()
matrix_3x2.data[0][0] = 7.0
matrix_3x2.data[0][1] = 8.0
matrix_3x2.data[1][0] = 9.0
matrix_3x2.data[1][1] = 10.0
matrix_3x2.data[2][0] = 11.0
matrix_3x2.data[2][1] = 12.0

# Умножение матриц: 2x3 * 3x2 = 2x2
result = matrix_2x3.multiply(matrix_3x2)

# Проверка на этапе компиляции: тип result — Matrix(Float, 2, 2)
# Единичная матрица 2x2
identity_3x3 = identity(Float, 3)()
```

## Компромиссы

### Преимущества

1. **Абстракции с нулевой стоимостью**
   - Мономорфизация на этапе компиляции, без накладных расходов во время выполнения
   - Без виртуальных функций, без RTTI

2. **Устранение мёртвого кода**
   - Анализ на этапе компиляции, инстанцируются только используемые обобщения
   - Раздувание кода контролируемо

3. **Замена макросов**
   - Типобезопасная кодогенерация
   - Удобство для IDE, понятные сообщения об ошибках

4. **Вычисления на этапе компиляции**
   - Обобщения с константами поддерживают вычисления на этапе компиляции
   - Возможности вроде проверки размерностей
   - Ключевое слово `const` не требуется, только ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанцирование обобщений увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Потребление памяти**
   - Увеличивается потребление памяти компилятором
   - Механизмам кэширования требуется память

3. **Сложность реализации**
   - Сложный решатель ограничений
   - Сложный движок вычислений на уровне типов

4. **Диагностика ошибок**
   - Ошибки обобщений могут быть сложными
   - Нужны понятные сообщения об ошибках

### Меры по смягчению

1. **Стратегии кэширования**
   - Кэширование результатов инстанцирования
   - LRU-кэш с ограничением памяти

2. **Инкрементальная компиляция**
   - Кэширование результатов компиляции
   - Инкрементальное инстанцирование

3. **Сообщения об ошибках**
   - Понятные сообщения об ошибках
   - Подсказки по выводу параметров обобщений

4. **Параллельная компиляция**
   - Параллельное инстанцирование обобщений
   - Многопоточное решение ограничений

## Альтернативы

| Альтернатива | Почему не выбрана |
|------|--------------|
| Только базовые обобщения | Не позволяют заменить сложные макросы |
| Чистая система макросов | Нет безопасности типов, плохие сообщения об ошибках |
| Только ограничения зависимостей | Недостаточная гибкость |
| Обобщения во время выполнения | Накладные расходы на производительность |

### Риски

| Риск | Влияние | Мера по смягчению |
|------|------|------|
| Сложность решения ограничений | Чрезмерное время компиляции | Инкрементальное решение + кэширование |
| Раздувание кода | Чрезмерный размер бинарника | DCE + контроль порогов |
| Сложность реализации | Удлинение цикла разработки | Поэтапная реализация |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

### BNF грамматики

```bnf
# Параметры обобщений используют единый синтаксис () как часть типа функции
# Например, map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в параметрах обобщений)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры обобщений — первая группа параметров в типе функции: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённый тип вроде List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Параметр Type в обобщениях автоматически заполняется компилятором из типов фактических аргументов
# Например, в map(numbers, f) T извлекается из numbers: List(Int), R — из f: (Int) -> String
```
