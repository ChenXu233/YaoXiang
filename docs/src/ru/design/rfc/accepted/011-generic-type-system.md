# RFC-011: Дизайн системы обобщённых типов — Абстракции с нулевой стоимостью и замена макросов

> **Статус**: Принято
> **Автор**: Чэнь Сюй
> **Дата создания**: 2025-01-25
> **Последнее обновление**: 2026-04-22 (обновлено в части самоописательного механизма Type, унифицирован синтаксис вызова обобщений)

## Аннотация

В данном документе определяется **дизайн системы обобщённых типов** языка YaoXiang. Система реализует абстракции с нулевой стоимостью, использует оптимизацию на этапе компиляции для снижения зависимости от макросов и обеспечивает механизм устранения мёртвого кода.

**Основные принципы проектирования**:
- **Унифицированный синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — обобщённые параметры и обычные параметры унифицированы
- **Самоописательный механизм Type**: `Type` является языковой конструкцией особого рода; позиции `Type` в сигнатурах автоматически заполняются выводом типов
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения на тип функций
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Обобщения на этапе компиляции**: `N: Int` — параметры-значения обобщённых типов, инстанциирование на этапе компиляции
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Абстракции с нулевой стоимостью: мономорфизация на этапе компиляции, отсутствие накладных расходов во время выполнения
- Устранение мёртвого кода: анализ графа инстанциирования + оптимизация LLVM
- Замена макросов: обобщения заменяют 90% сценариев использования макросов
- Типовая безопасность: проверка на этапе компиляции, дружелюбие к IDE
- **Явное лучше неявного**: `Type` самоописателен, компилятор выводит автоматически

## Идея типовой вселенной и типы, зависящие от значений

Система обобщённых типов YaoXiang построена на основе **идеи типовой вселенной** — ментальной модели, объединяющей все концепции языка в иерархическую структуру. Ключевая инновация заключается в продвижении **типов, зависящих от значений**, до статуса полноправных граждан на уровне Type2.

### Что такое типы, зависящие от значений?

**Типы, зависящие от значений** — это типы, которые зависят от одного или нескольких **значений** (а не только от других типов). Эти значения могут вычисляться на этапе компиляции, обеспечивая гарантии типовой безопасности уже на этапе компиляции.

```yaoxiang
# Традиционные обобщения: параметры-типы
List: (T: Type) -> Type

# Типы, зависящие от значений: параметры-значения
Vec: (n: Int) -> Type  # Тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # Тип матрицы зависит от количества строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционными обобщениями, типы YaoXiang, зависящие от значений, обладают следующими преимуществами:

| Характеристика | Традиционные обобщения (C++/Rust) | YaoXiang: типы, зависящие от значений |
|------|-------------------|---------------------|
| Значения, от которых зависят типы | Только параметры-типы | Любые значения, включая результаты вызовов функций |
| Вычисления на этапе компиляции | Ручная специализация шаблонов C++, Rust — нет | Автоматические вычисления на этапе компиляции с гарантией завершения |
| Вычисления на уровне типов | Метапрограммирование шаблонов (сложно/опасно) | Единый механизм вычислений на уровне типов |
| Типовая безопасность | C++ — нет, Rust — ограниченная | Полная типовая безопасность с проверкой на этапе компиляции |
| Проверка размерностей | Проверка во время выполнения или ручная специализация | Проверка размерностей на этапе компиляции, без накладных расходов |

### Уровни типовой вселенной и типы, зависящие от значений

Идея типовой вселенной распределяет концепции языка по различным уровням согласно их семантическим ролям. Типы, зависящие от значений, находятся на **уровне Type2**:

| Уровень | Роль | Пример |
|------|------|------|
| Type-1 | Значения | `42`, `factorial(5)`, сами функции |
| Type0 | Ключевое слово метатипа | `Type` |
| Type1 | Конкретные типы | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функции/конструкторы типов/типы, зависящие от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевой дизайн**: функции, конструкторы типов и типы, зависящие от значений, на уровне Type2 имеют **унифицированный синтаксис** — все они представлены в форме `(params) -> result`:
- Обычные функции: `(Int, Int) -> Int` → возвращают значение
- Конструкторы типов: `(T: Type) -> Type` → возвращают тип
- Типы, зависящие от значений: `(n: Int) -> Type` → возвращают тип, но зависят от параметра-значения

> **Изоморфизм Карри — Ховарда**: Это единство не случайно. Изоморфизм Карри — Ховарда утверждает, что «типы — это утверждения, а программы — это доказательства». Функциональный тип `A → B` соответствует логическому импликации «если A, то B», обобщение `(T: Type) -> Type` соответствует универсальному квантификатору «для всех типов T», а тип, зависящий от значений, `(n: Int) -> Type` соответствует «для каждого целого n существует тип». Объединение функций, конструкторов типов и типов, зависящих от значений, на уровне Type2 в YaoXiang по сути унифицирует «доказательства» и «вычисления» в одну концепцию — **конструктивные доказательства**. Это прямое воплощение изоморфизма Карри — Ховарда в дизайне языка: одна форма (`(params) -> result`) одновременно выражает логические утверждения и вычислительные процессы.

### Гарантии детерминированности на этапе компиляции

Типовая вселенная YaoXiang требует: **всё на уровне Type должно быть определено на этапе компиляции**.

```yaoxiang
# Пример проверки размерностей на этапе компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка на этапе компиляции: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется на этапе компиляции
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Вычисление на этапе компиляции: factorial(3) = 6, размер вектора определяется на этапе компиляции
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов
2. Проверяет, помечены ли функции уменьшающим инвариантом `decreases` (см. механизм проверки завершения ниже)
3. Выполняет вычисления на этапе компиляции
4. Встраивает результат в генерируемый тип

### Сценарии применения типов, зависящих от значений

#### Проверка размерностей на этапе компиляции
```yaoxiang
# Умножение матриц: проверка соответствия размерностей на этапе компиляции
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка на этапе компиляции: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка перехватывается на этапе компиляции:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива является константой этапа компиляции
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — константа этапа компиляции, может использоваться для вычислений на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно на этапе компиляции)
```

#### Условные типы
```yaoxiang
# Типовый If
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейство типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры типа T, R определяются на этапе компиляции
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    return result
}

# При использовании полностью прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # Выводится как map[Int, Int]
```

### Механизм проверки завершения (интеграция с RFC-022)

Вычисления на уровне типов должны **гарантированно завершаться**, иначе типовая система попадёт в бесконечный цикл. YaoXiang обеспечивает это через **уменьшающие инварианты** (`decreases`), полностью интегрированные с RFC-022.

#### Уменьшающие инварианты для рекурсивных функций
```yaoxiang
# Факториал на этапе компиляции: необходимо доказать завершение
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # При каждой рекурсии n строго уменьшается
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # factorial(5) = 120 вычисляется на этапе компиляции
```

#### Decreases规фикация для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершения

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза проверки типов                                        │
│  Встречен вызов функции в позиции типа (например, Vec(factorial(5)))
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка decreases-инварианта                            │
│     - Есть decreases: верификация убывания на всех путях    │
│     - Нет decreases, но очевидно завершается: прямоое вычисление
│     - Нет decreases и возможно не завершается: ошибка компиляции
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисления на этапе компиляции (выполняются встроенным интерпретатором)
│     - Чистые функции: прямое вычисление                     │
│     - Побочные эффекты: ошибка компиляции (позиция типа должна быть чистой)
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                            │
│     - Vec(factorial(5)) → Vec(120)                          │
│     - Matrix(Float, 3, 3) → конкретный тип                  │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантирует завершение вычислений на этапе компиляции, избегая бесконечных циклов в системе типов
- **Единство**: проверка завершения и верификация частичной корректности используют общий механизм спецификаций
- **Постепенное усиление**: можно переходить от проверок во время выполнения к полностью статическому доказательству

## Мотивация

### Зачем нужна мощная система обобщений?

Текущие主流语言 имеют ограничения в системах обобщений:

| Язык | Возможности обобщений | Проблемы |
|------|----------|------|
| Java | Ограниченные типы | Компиляция в один тип, без специализации обобщений |
| C# | Ограничения обобщений | Проверка типов во время выполнения, накладные расходы |
| Rust | Обобщения + Trait | Система Trait сложна, крутая кривая обучения |
| C++ | Шаблоны | Специализация шаблонов сложна, плохие сообщения об ошибках |
| **YaoXiang** | **Типы, зависящие от значений** | **Типы могут зависеть от значений, проверка размерностей на этапе компиляции, гарантия завершения** |

### Ключевые противоречия

1. **Производительность vs гибкость**: гибкость во время выполнения vs оптимизация на этапе компиляции
2. **Сложность vs простота**: мощная система типов vs удобство использования
3. **Макросы vs обобщения**: генерация кода макросами vs типобезопасные обобщения
4. **Зависимость от значений vs безопасность типов**: традиционные обобщения не могут проверять размерности на этапе компиляции

### Ключевые преимущества типов, зависящих от значений

**Типы, зависящие от значений**, YaoXiang — ключевое преимущество перед традиционными обобщениями:

| Преимущество | Описание |
|------|------|
| **Типы зависят от значений** | `Vec: (n: Int) -> Type` позволяет типу зависеть от конкретного значения |
| **Вычисления на этапе компиляции** | Вызовы функций в позициях типов вычисляются на этапе компиляции, результат встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` верифицирует размерности матрицы на этапе компиляции |
| **Вычисления на уровне типов** | Условные типы `If`, `Match` поддерживают вычисления на уровне типов |
| **Гарантия завершения** | Decreases-инварианты гарантируют обязательное завершение вычислений на этапе компиляции |

```yaoxiang
# Проверка на этапе компиляции, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Вычисление на этапе компиляции: factorial(3) = 6, factorial(2) = 2
# Тип: Matrix(Float, 6, 2)

# Несоответствие размерностей перехватывается на этапе компиляции
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщений

```yaoxiang
# Пример: унифицированный дизайн API
# Операция map для разных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Подход с обобщениями: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели проектирования

### Основные цели

1. **Абстракции с нулевой стоимостью** — вызов обобщённой функции эквивалентен вызову конкретной функции
2. **Устранение мёртвого кода** — анализ на этапе компиляции, инстанциируются только используемые обобщения
3. **Замена макросов** — обобщения заменяют 90% сценариев использования макросов
4. **Типовая безопасность** — проверка на этапе компиляции, без накладных расходов на проверку типов во время выполнения
5. **Дружелюбие к IDE** — интеллектуальные подсказки, чёткие сообщения об ошибках
6. **Типы, зависящие от значений** — типы могут зависеть от значений, поддержка проверки размерностей на этапе компиляции
7. **Безопасность вычислений на этапе компиляции** — гарантия завершения вычислений на этапе компиляции через decreases-инварианты

### Принципы проектирования

- **Детерминированность на этапе компиляции**: параметры обобщений определяются на этапе компиляции
- **Приоритет мономорфизации**: генерация конкретного кода, избегание виртуальных вызовов функций
- **Ограничения управляют инстанциированием**: ограничения типов направляют инстанциирование
- **Оптимизация под платформу**: специализация поддерживает оптимизацию под конкретную платформу
- **Единство типовой вселенной**: функции/конструкторы типов/типы, зависящие от значений, унифицированы на уровне Type2
- **Гарантия завершения**: вызовы функций в позициях типов должны доказать завершение

### 1. Базовые обобщения

#### 1.1 Параметры обобщённых типов

> **Ключевое правило**: Определения обобщённых типов **должны явно содержать аннотацию `: Type`**, иначе HM-вывод определит их как функции.
>
> | Запись | Значение |
> |------|------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM-вывод определяет как функцию, не тип |

```yaoxiang
# Определение обобщённого типа (должна быть аннотация : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM-вывод определяет как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Обобщённое ограничение (прямое выражение, return можно опустить в однострочном)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров-типов
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 2.4 Встроенные marker trait: Dup и Clone

**Три вида семантики копирования**:

| Тип | Значение | Способ активации | Сценарий использования |
|------|------|----------|------|
| **Примитивное копирование значения** | При присваивании автоматическое копирование значения, два значения полностью независимы | Автоматически при присваивании/передаче параметров | Int, Float, Bool, Char |
| **Dup** | Неполная копия: копирование дескриптора/токена, базовые данные разделяются | Автоматически при присваивании/передаче параметров | Токены `&T`, `ref T`, String/Bytes |
| **Clone** | Глубокая копия: создаётся полностью независимая копия | `value.clone()` | Любой тип, реализующий Clone |

**Семантика Dup**: типы, реализующие Dup, при присваивании/передаче параметров не передают владение — компилятор копирует дескриптор/токен, несколько владельцев указывают на одни и те же базовые данные. Это дополнение к стандартной семантике Move из RFC-009.

**Dup и Clone — ортогональные концепции**:

```
Dup = копирование дескриптора, разделённые данные (изменения влияют друг на друга)
Clone = копирование данных, независимые копии (изменения не влияют друг на друга)
```

**Правила**:

```
1. Типы примитивных значений (Int, Float, Bool, Char) — встроенное компилятором копирование значения, не принадлежат Dup
2. Dup  — применимо только к ссылочным/токенным типам и типам с внутренним подсчётом ссылок
3. Clone — явная глубокая копия, любой тип может реализовать
4. Move по умолчанию — остальные типы сохраняют стандартную семантику Move
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|------|-----|------|
| `&T` (заимствованный токен) | ✅ | Токен нулевого размера, копирование токена = несколько представлений на одни данные |
| `ref T` | ✅ | Rc/Arc copy = счётчик ссылок +1, разделённые данные в куче |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, копирование дескриптора разделяет базовый буфер |
| `&mut T` (изменяемый токен) | ❌ | Линейный эксклюзив, не может копироваться |
| struct | Производный | Все поля Dup → struct Dup |
| enum | Производный | Все поля всех вариантов Dup → enum Dup |
| tuple | Производный | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захваченное окружение может не быть Dup |
| `*T` (сырой указатель) | ❌ | unsafe, не участвует в системе владения |

**Int/Float/Bool/Char НЕ являются Dup** — это типы значений, при присваивании компилятор автоматически копирует значение (два значения полностью независимы). Это не «неполная копия», а встроенная обработка примитивов компилятором; не нужно и не следует выражать через типовое свойство Dup.

```yaoxiang
# Типы примитивных значений: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x          # Копирование значения, x и y полностью независимы
print(x)       # ✅

# Dup: неполная копия, копирование дескриптора разделяет данные
view: &Point = &point
view2 = view    # ✅ Dup: копирование токена, оба указывают на один point
print(view.x)   # ✅

# Clone: явная глубокая копия, создание независимой копии
backup = big_struct.clone()  # Явный вызов

# Обобщённые ограничения
dup_use: (T: Dup) -> T = x         # T: Dup → можно неполное копирование
clone_use: (T: Clone) -> T = x.clone()  # T: Clone → можно глубокое копирование
```

> **Примечание**: `Send`/`Sync` не являются видимыми для пользователя trait. Гарантии безопасности при межзадачном взаимодействии полностью автоматически обеспечиваются ключевым словом `ref` и компилятором — `ref` автоматически выбирает Rc или Arc, пользователю не нужно понимать Send/Sync.

```yaoxiang
        self.index = self.index + 1

    } else {
        return Option.none()
    }
}
```

### 5. Условные типы

> **Изоморфизм Карри — Ховарда**: С точки зрения Карри — Ховарда условные типы представляют собой **case-анализ** в логике. Тип `Bool` соответствует утверждению с двумя возможными значениями (True/False), `If` выбирает различные результаты в зависимости от истинности утверждения — это именно дизъюнктивный case-анализ в логике. `match C { True => T, False => E }` фактически выражает: «когда известно, что утверждение C истинно, заключение T; когда C ложно, заключение E».

#### 5.1 Условный тип If

```yaoxiang
# Типовый If
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Верификация на этапе компиляции
Assert: (C: Bool) -> Type = match C {
    True => Void,
    False => compile_error("Assertion failed"),
}

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри — Ховарда**: Семейства типов — наиболее прямое воплощение «утверждения являются типами». `Add: (A: Type, B: Type) -> Type` — это не «написанная на уровне типов функция сложения», а **построение утверждения о сложении натуральных чисел**. `(Zero, B) => B` означает «утверждение Add(Zero, B) эквивалентно B», `(Succ(A'), B) => Succ(Add(A', B))` означает «если Add(A', B) верно, то Add(Succ(A'), B) тоже верно». Это само определение сложения в аксиомах Пеано. Проверка типами того, что это match-выражение проходит, эквивалентна верификации логической согласованности данного определения.

```yaoxiang
# Преобразование типов на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # По умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Curry-Howard: это также индуктивное определение сложения натуральных чисел)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: вычисление 2 + 3 на этапе компиляции
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

```rust
    // Узлы: инстанциирования обобщений

    edges: HashMap<InstanceKey, Vec<InstanceKey>>,

    const_args: Vec<ConstId>,  // Параметры-константы

fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Непосещённые инстанциирования — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}
```

```yaoxiang
# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённый quicksort
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Использование перегрузки функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Пример использования ========
```

```yaoxiang
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),

    # Проверка размерностей на этапе компиляции: использование типа Assert из стандартной библиотеки
    _assert: Assert[Rows > 0],  # Rows > 0, иначе ошибка компиляции
    _assert: Assert[Cols > 0],  # Cols > 0, иначе ошибка компиляции

    # Операции с матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {
            for j in 0..M {
                sum = Zero::zero()
                for k in 0..Cols {
                    sum = sum + self.data[i][k] * other.data[k][j]
                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    return matrix
}

matrix_2x3.data[1][2] = 6.0

# Матрица 3x2
matrix_3x2 = Matrix(Float, 3, 2)()
matrix_3x2.data[0][0] = 7.0
matrix_3x2.data[0][1] = 8.0
matrix_3x2.data[1][0] = 9.0
matrix_3x2.data[1][1] = 10.0
matrix_3x2.data[2][0] = 11.0
matrix_3x2.data[2][1] = 12.0

# Умножение матриц: 2x3 * 3x2 = 2x2
result = matrix_2x3.multiply(matrix_3x2)

# Верификация на этапе компиляции: тип результата Matrix(Float, 2, 2)
# Единичная матрица 3x3
identity_3x3 = identity(Float, 3)()

# Несоответствие размерностей: ошибка компиляции
# bad_multiply = matrix_2x3.multiply(identity_3x3)  # Ошибка компиляции: 3x3 != 2x3
```

## Компромиссы

### Преимущества

1. **Абстракции с нулевой стоимостью**
   - Мономорфизация на этапе компиляции, отсутствие накладных расходов во время выполнения
   - Не нужны виртуальные функции, нет RTTI

2. **Устранение мёртвого кода**
   - Анализ на этапе компиляции, инстанциируются только используемые обобщения
   - Разбухание кода под контролем

3. **Замена макросов**
   - Типобезопасная генерация кода
   - Дружелюбие к IDE, чёткие сообщения об ошибках

4. **Вычисления на этапе компиляции**
   - Обобщения на этапе компиляции поддерживают вычисления на этапе компиляции
   - Характеристики вроде проверки размерностей
   - Не требуется ключевое слово `const`, только ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанциирование обобщений увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Использование памяти**
   - Увеличение использования памяти компилятором
   - Механизм кэширования требует памяти

3. **Сложность реализации**
   - Сложность решателя ограничений
   - Сложность механизма вычислений на уровне типов

4. **Диагностика ошибок**
   - Ошибки обобщений могут быть сложными
   - Требуются чёткие подсказки об ошибках

### Меры по смягчению

1. **Стратегия кэширования**
   - Кэширование результатов инстанциирования
   - Ограничение использования памяти через LRU-кэш

2. **Инкрементная компиляция**
   - Кэширование результатов компиляции
   - Инкрементное инстанциирование

3. **Подсказки об ошибках**
   - Чёткие сообщения об ошибках
   - Подсказки по выводу параметров обобщений

4. **Параллельная компиляция**
   - Параллельное инстанциирование обобщений
   - Многопоточное решение ограничений

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| Только базовые обобщения | Не может заменить сложные макросы |
| Чистая макросистема | Нет типовой безопасности, плохие сообщения об ошибках |
| Только ограничения | Недостаточная гибкость |
| Обобщения во время выполнения | Есть накладные расходы |

### Риски

| Риск | Влияние | Меры по смягчению |
|------|------|------|
| Сложность решения ограничений | Слишком долгое время компиляции | Инкрементное решение + кэширование |
| Разбухание кода | Слишком большой бинарный файл | DCE + контроль порогов |
| Сложность реализации | Увеличение срока разработки | Фазовость реализации |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

## Приложение

### БНФ-грамматика

```bnf
# Параметры обобщений используют унифицированный синтаксис (), как часть типа функции
# Например: map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничения типов (в параметрах обобщений)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры обобщений — первая группа параметров в типе функции: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённые типы, например List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Параметры Type в параметрах обобщений автоматически заполняются компилятором из типов фактических аргументов
# Например, map(numbers, f), T извлекается из numbers: List(Int), R из f: (Int) -> String
```
