# RFC-011: Проектирование системы обобщённого программирования — Абстракции с нулевой стоимостью и замена макросов

> **Статус**: Принято
> **Автор**: Чэнь Сюй
> **Дата создания**: 2025-01-25
> **Последнее обновление**: 2026-04-22 (обновление механизма самодокументирования Type, унификация синтаксиса вызова обобщений)

## Аннотация

В данном документе определяется **система проектирования обобщённого программирования** языка YaoXiang, реализующая абстракции с нулевой стоимостью через мощные возможности обобщений, сокращая зависимость от макросов посредством оптимизации на этапе компиляции и обеспечивая механизм устранения мёртвого кода.

**Основные проектные решения**:
- **Унифицированный синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — параметры обобщений и обычные параметры унифицированы
- **Механизм самодокументирования Type**: `Type` является языковой конструкцией особого рода, позиции `Type` в сигнатурах заполняются автоматически
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения типа function
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Обобщения на этапе компиляции**: `N: Int` — параметры-значения на этапе компиляции, инстанцирование констант
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Абстракции с нулевой стоимостью: мономорфизация на этапе компиляции, отсутствие накладных расходов во время выполнения
- Устранение мёртвого кода: анализ графа инстанцирования + оптимизация LLVM
- Замена макросов: обобщения заменяют 90% сценариев использования макросов
- Типовая безопасность: проверка на этапе компиляции, удобство для IDE
- **Явное лучше неявного**: `Type` самодокументируется, компилятор выводит автоматически

## Концепция типовой вселенной и типы, зависящие от значений

Система обобщённого программирования YaoXiang строится на **концепции типовой вселенной** — ментальной модели, объединяющей все концепции языка в иерархическую структуру. Ключевая инновация заключается в том, что **типы, зависящие от значений**, подняты до уровня первоклассных граждан Type2.

### Что такое типы, зависящие от значений?

**Тип, зависящий от значений**, — это тип, который зависит от одного или нескольких **значений** (а не только от других типов). Эти значения могут вычисляться на этапе компиляции, обеспечивая гарантии типовой безопасности уже на этапе компиляции.

```yaoxiang
# Традиционные обобщения: параметры типа
List: (T: Type) -> Type

# Типы, зависящие от значений: параметры-значения
Vec: (n: Int) -> Type  # Тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # Тип матрицы зависит от числа строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционным обобщённым программированием, типы YaoXiang, зависящие от значений, обладают следующими ключевыми преимуществами:

| Характеристика | Традиционные обобщения (C++/Rust) | YaoXiang типы, зависящие от значений |
|------|-------------------|---------------------|
| Зависимость типа от значений | Только параметры типа | Зависимость от любых значений, включая результаты вызовов функций |
| Вычисление на этапе компиляции | Ручная специализация шаблонов C++, отсутствует в Rust | Автоматическое вычисление на этапе компиляции с гарантией завершения |
| Вычисления на уровне типов | Метапрограммирование шаблонов (сложно/опасно) | Унифицированный механизм вычислений на уровне типов |
| Типовая безопасность | C++ отсутствует, Rust ограничена | Полная типовая безопасность, проверка на этапе компиляции |
| Проверка размерностей | Проверка во время выполнения или ручная специализация | Проверка размерностей на этапе компиляции, отсутствие накладных расходов |

### Уровни типовой вселенной и типы, зависящие от значений

Концепция типовой вселенной распределяет концепции языка по семантическим ролям на различные уровни. Типы, зависящие от значений, располагаются на **уровне Type2**:

| Уровень | Роль | Пример |
|------|------|------|
| Type-1 | Значение | `42`, `factorial(5)`, сама функция |
| Type0 | Ключевое слово метаязыка | `Type` |
| Type1 | Конкретный тип | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функция/Конструктор типа/Типы, зависящие от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевое проектное решение**: функции, конструкторы типов и типы, зависящие от значений, на уровне Type2 имеют **унифицированный синтаксис** — все представляют собой форму `(params) -> result`:
- Обычные функции: `(Int, Int) -> Int` → возвращают значение
- Конструкторы типов: `(T: Type) -> Type` → возвращают тип
- Типы, зависящие от значений: `(n: Int) -> Type` → возвращают тип, но зависят от параметров-значений

> **Изоморфизм Карри — Ховарда**: Это единство не случайно. Изоморфизм Карри — Ховарда устанавливает, что «типы — это утверждения, программы — это доказательства». Функциональный тип `A → B` соответствует логическому следованию «если A, то B», обобщение `(T: Type) -> Type` соответствует универсальному квантору «для всех типов T», а тип, зависящий от значений, `(n: Int) -> Type` соответствует «для каждого целого n существует тип». Унификация функций, конструкторов типов и типов, зависящих от значений, на уровне Type2 по сути объединяет «доказательства» и «вычисления» в единую концепцию — **конструктивные доказательства**. Это прямое воплощение изоморфизма Карри — Ховарда в дизайне языка: одна форма (`(params) -> result`) одновременно выражает логические утверждения и вычислительные процессы.

### Гарантии детерминированности на этапе компиляции

Концепция типовой вселенной YaoXiang требует: **всё на уровне Type является детерминированным на этапе компиляции**.

```yaoxiang
# Пример проверки размерностей на этапе компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка на этапе компиляции: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется на этапе компиляции
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Вычисление на этапе компиляции: factorial(3) = 6, размер вектора определяется на этапе компиляции
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов
2. Проверяет, помечены ли функции уменьшающим вариантом (см. механизм проверки завершения ниже)
3. Выполняет вычисления на этапе компиляции
4. Встраивает результат в генерируемый тип

### Сценарии применения типов, зависящих от значений

#### Проверка размерностей на этапе компиляции
```yaoxiang
# Умножение матриц: проверка соответствия размерностей на этапе компиляции
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка на этапе компиляции: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка перехватывается на этапе компиляции:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива является константой этапа компиляции
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — константа этапа компиляции, может использоваться для вычислений на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно на этапе компиляции)
```

#### Условные типы
```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейство типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры типа T, R определяются на этапе компиляции
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    result
}

# Использование полностью прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # Выводится как map[Int, Int]
```

### Механизм проверки завершения (интеграция с RFC-022)

Вычисления на этапе компиляции для типов, зависящих от значений, должны **гарантированно завершаться**, иначе типовая система войдёт в бесконечный цикл. YaoXiang обеспечивает это через **уменьшающий вариант**, полностью интегрированный с RFC-022.

#### Уменьшающий вариант для рекурсивных функций
```yaoxiang
# Факториал на этапе компиляции: необходимо доказать завершение
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # При каждой рекурсии n строго уменьшается
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # Вычисление factorial(5) = 120 на этапе компиляции
```

#### Уменьшающий вариант для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершения

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза проверки типов                                        │
│  Встречен вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка уменьшающего варианта                           │
│     - Есть decreases: проверяется выполнение условия убывания │
│       на всех путях рекурсии                                 │
│     - Нет decreases, но завершение очевидно: прямое вычисление │
│     - Нет decreases и возможно не завершение: ошибка компиляции │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисление на этапе компиляции (встроенный интерпретатор) │
│     - Чистая функция: прямое вычисление                      │
│     - Побочные эффекты: ошибка компиляции (в позиции типа побочных эффектов быть не может) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                            │
│     - Vec(factorial(5)) → Vec(120)                          │
│     - Matrix(Float, 3, 3) → конкретный тип                   │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантия завершения вычислений на этапе компиляции, предотвращение бесконечных циклов в типовой системе
- **Единство**: проверка завершения и верификация частичной корректности используют общий механизм спецификаций
- **Постепенное улучшение**: можно постепенно перейти от проверок во время выполнения к полной статической верификации

## Мотивация

### Почему нужна мощная система обобщённого программирования?

Современные主流ные языки имеют ограничения в обобщениях:

| Язык | Возможности обобщений | Проблема |
|------|----------|------|
| Java | Ограниченные типы | Мономорфизация на этапе компиляции, без специализации обобщений |
| C# | Ограничения обобщений | Проверка типов во время выполнения, накладные расходы |
| Rust | Обобщения + Trait | Система Trait сложна, крутая кривая обучения |
| C++ | Шаблоны | Сложная специализация шаблонов, плохие сообщения об ошибках |
| **YaoXiang** | **Типы, зависящие от значений** | **Типы могут зависеть от значений, проверка размерностей на этапе компиляции, гарантия завершения** |

### Ключевые противоречия

1. **Производительность vs гибкость**: гибкость во время выполнения vs оптимизация на этапе компиляции
2. **Сложность vs простота**: мощная типовая система vs удобство использования
3. **Макросы vs обобщения**: генерация кода макросами vs типобезопасность обобщений
4. **Зависимость от значений vs типовая безопасность**: традиционные обобщения не могут проверять размерности на этапе компиляции

### Ключевые преимущества типов, зависящих от значений

**Типы, зависящие от значений**, YaoXiang — это ключевое преимущество перед традиционными обобщениями:

| Преимущество | Описание |
|------|------|
| **Типы зависят от значений** | `Vec: (n: Int) -> Type` делает тип зависящим от конкретного значения |
| **Вычисление на этапе компиляции** | Вызовы функций в позициях типов вычисляются на этапе компиляции, результат встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` проверяет размерности матрицы на этапе компиляции |
| **Вычисления на уровне типов** | `If`, `Match` и другие условные типы поддерживают вычисления на уровне типов |
| **Гарантия завершения** | Уменьшающий вариант гарантирует завершение вычислений на этапе компиляции |

```yaoxiang
# Проверка на этапе компиляции, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Вычисление на этапе компиляции: factorial(3) = 6, factorial(2) = 2
# Тип: Matrix(Float, 6, 2)

# Несоответствие размерностей перехватывается на этапе компиляции
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщённого программирования

```yaoxiang
# Пример: унифицированный дизайн API
# Операция map для различных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Обобщённый подход: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Проектные цели

### Основные цели

1. **Абстракции с нулевой стоимостью** — вызов обобщённой функции эквивалентен вызову конкретной функции
2. **Устранение мёртвого кода** — анализ на этапе компиляции, инстанцируются только используемые обобщения
3. **Замена макросов** — обобщения заменяют 90% сценариев использования макросов
4. **Типовая безопасность** — проверка на этапе компиляции, без накладных расходов на проверку типов во время выполнения
5. **Удобство для IDE** — интеллектуальные подсказки, чёткие сообщения об ошибках
6. **Типы, зависящие от значений** — типы могут зависеть от значений, поддержка проверки размерностей на этапе компиляции
7. **Безопасность вычислений на этапе компиляции** — гарантия завершения вычислений на этапе компиляции через уменьшающий вариант

### Проектные принципы

- **Детерминированность на этапе компиляции**: параметры обобщений определяются на этапе компиляции
- **Приоритет мономорфизации**: генерация конкретного кода, избегание виртуальных вызовов функций
- **Ограничения направляют инстанцирование**: ограничения типов направляют инстанцирование
- **Оптимизация под платформу**: специализация поддерживает оптимизацию, специфичную для платформы
- **Единство типовой вселенной**: функции/конструкторы типов/типы, зависящие от значений, унифицированы на уровне Type2
- **Гарантия завершения**: вызовы функций в позициях типов должны доказать завершение

## Предложение

### 1. Базовые обобщения

#### 1.1 Параметры обобщённого типа

> **Ключевое правило**: Определения обобщённых типов **должны явно содержать пометку `: Type`**, иначе HM выведет их как функции.
>
> | Запись | Значение |
> |------|------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM выводит как функцию, не тип |

```yaoxiang
# Определение обобщённого типа (обязательно : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM выводит как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Ограничение обобщения (прямое выражение, return можно опустить в однострочном)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров типа
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 1.2 Механизм самодокументирования Type

`Type` является языковой конструкцией особого рода. Компилятор естественным образом распознаёт позиции `Type` в сигнатурах и автоматически выводит и заполняет их из типов фактических аргументов.

```yaoxiang

#         ^^^^^^^^   ^^^^^^
#         объявление типа   вызов конструктора: Int заполняет T

# Вывод при вызове функции

f: (x: Int) -> String = (x) => x.to_string()

# Компилятор выводит: T=Int, R=String
```

#### 2.4 Встроенные marker trait: Dup и Clone

**Две семантики копирования**:

| Атрибут | Значение | Способ触发 | Стоимость |
|------|------|----------|------|
| **Dup** | Неявное поверхностное копирование | Автоматически при присваивании/передаче аргументов | Нулевая (побитовое копирование или тип нулевого размера) |
| **Clone** | Явное глубокое копирование | `value.clone()` | Зависит от типа |

**Семантика Dup**: типы, реализующие Dup, при присваивании/передаче аргументов не передают владение — компилятор автоматически создаёт копию. Это дополнение к семантике Move по умолчанию из RFC-009.

**Правила**:

```
1. Dup → Clone     Все типы Dup автоматически реализуют Clone (клонирование полей поэтапно)
2. Clone ↛ Dup     Наличие Clone не означает наличие Dup
3. По умолчанию Move     Типы без реализации Dup сохраняют семантику Move по умолчанию
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|------|-----|------|
| Int, Float, Bool, Char | ✅ | Примитивы, встроены в компилятор |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, поверхностное копирование безопасно |
| `&T` (заёмный токен) | ✅ | Токен нулевого размера с доказательством на этапе компиляции, копирование без побочных эффектов |
| `&mut T` (изменяемый токен) | ❌ | Линейный эксклюзив, копирование невозможно |
| struct | Выводится | Все поля Dup → struct Dup |
| enum | Выводится | Все поля всех вариантов Dup → enum Dup |
| tuple | Выводится | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захваченное окружение может не быть Dup |
| Arc(T) | ✅ | Arc сам может быть поверхностно скопирован |

**Dup и Clone — это marker trait** — trait без методов. Для примитивов компилятор автоматически регистрирует impl. Для struct/enum/tuple компилятор рекурсивно проверяет все поля/элементы: если все реализуют Dup/Clone, автоматически генерируется impl (автоматический вывод).

```yaoxiang
# Dup: неявное поверхностное копирование
x: Int = 42
y = x          # Int: Dup → копирование
print(x)       # ✅

# Clone: явное глубокое копирование
backup = big_struct.clone()  # Явный вызов

# Ограничения обобщений
dup_use: (x: T: Dup) -> T = x         # T: Dup → можно неявно копировать
clone_use: (x: T: Clone) -> T = x.clone()  # T: Clone → явное глубокое копирование

# Токен Dup — передача с нулевой стоимостью
view: &Point = &point
view2 = view    # ✅ &T: Dup
print(view.x)   # ✅ Нулевая стоимость — токен нулевого размера
```

> **Примечание**: `Send`/`Sync` не являются видимыми для пользователя trait. Гарантии безопасности при межзадачном взаимодействии обеспечиваются ключевым словом `ref` и полностью автоматической работой компилятора — `ref` автоматически выбирает Rc или Arc, пользователю не нужно разбираться в Send/Sync.

# Iterator trait (использование синтаксиса (Item: Type) -> Type)
Iterator: (Item: Type) -> Type = {
    next: (Self) -> Option(Item),
    has_next: (Self) -> Bool,

collect_all: (T: Type, I: Iterator(T))(iter: I) -> List(T) = {
    result: List(T) = List(T)
    while iter.has_next() {
        if let Some(item) = iter.next() {
            result.push(item)
        }
    }
    return result
}

# Реализация Iterator для Array
# Синтаксический сахар для методов: Array.Item, Array.next, Array.has_next
Array.has_next: (T: Type)(self: Array(T)) -> Bool = {
    return self.index < self.length
}

Array.next: (T: Type)(self: Array(T)) -> Option(T) = {
    if has_next(self) {
        item = self.data[self.index]
        self.index = self.index + 1
        return Option.some(item)
    } else {
        return Option.none()
    }
}

Array.Item: (T: Type)(arr: Array(T)) -> T = {
    return arr.data[0]
}
```

#### 4.1 Параметры-константы на этапе компиляции

**Основное проектное решение**: `Type` в сигнатуре обобщения помечает параметры типа этапа компиляции. Параметры-значения типа `Int` по умолчанию определяются на этапе компиляции в контексте обобщений. Ключевое слово `const` не требуется.

# ════════════════════════════════════════════════════════

# Факториал на этапе компиляции: N должен быть литералом, известным на этапе компиляции
factorial: (N: Int) -> (n: N) -> Int = {
    match n {
        0 => 1,
        _ => n * factorial(n - 1)

}

# Сложение на этапе компиляции
add: (a: Int, b: Int) -> (a: a, b: b) -> Int = a + b

### 5. Условные типы

> **Изоморфизм Карри — Ховарда**: Условные типы с точки зрения изоморфизма Карри — Ховарда представляют собой **case-анализ** в логике. Тип `Bool` соответствует утверждению с двумя возможными значениями (True/False), `If` выбирает различные результаты в зависимости от истинности этого утверждения — это именно дизъюнктивный case в логике. `match C { True => T, False => E }` фактически выражает: «если известно, что утверждение C истинно, заключение — T; если C ложно, заключение — E».

#### 5.1 Условный тип If

```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Проверка на этапе компиляции
Assert: (C: Bool) -> Type = match C {
    True => Void,
    False => compile_error("Assertion failed"),
}

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри — Ховарда**: Семейства типов — наиболее прямое воплощение «утверждения — это типы». `Add: (A: Type, B: Type) -> Type` — это не «написанная на уровне типов функция сложения», а **конструкция утверждения о сложении натуральных чисел**. `(Zero, B) => B` означает «утверждение Add(Zero, B) эквивалентно B», `(Succ(A'), B) => Succ(Add(A', B))` означает «если Add(A', B) верно, то Add(Succ(A'), B) тоже верно». Это сама формулировка сложения по Пеано. Проверка типами этого match-выражения эквивалентна проверке логической согласованности данного определения.

```yaoxiang
# Преобразование типов на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # По умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Curry-Howard: это также индуктивное определение сложения натуральных чисел)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

```

struct InstantiationGraph {
    // Узлы: инстанцирование обобщений

    const_args: Vec<ConstId>,  // Параметры Const

fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Непосещённые инстанцирования — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение ёмкости
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

    }

    return result
}

# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 if a < b, 0 if a == b, 1 if a > b
}

# Обобщённый quicksort
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Использование перегрузки функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========

   - Раздувание кода под контролем

3. **Замена макросов**
   - Типобезопасная генерация кода
   - Удобство для IDE, чёткие сообщения об ошибках

4. **Вычисления на этапе компиляции**
   - Обобщения на этапе компиляции поддерживают вычисления на этапе компиляции
   - Такие характеристики, как проверка размерностей
   - Без ключевого слова `const`, только ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанцирование обобщений увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Использование памяти**
   - Увеличение использования памяти компилятором
   - Для кэша требуется память

3. **Сложность реализации**
   - Решатель ограничений сложен
   - Механизм вычислений на уровне типов сложен

4. **Диагностика ошибок**
   - Ошибки обобщений могут быть сложными
   - Требуются чёткие подсказки об ошибках

### Меры по смягчению

1. **Стратегии кэширования**
   - Кэширование результатов инстанцирования
   - Ограничение использования памяти через LRU-кэш

2. **Инкрементальная компиляция**
   - Кэширование результатов компиляции
   - Инкрементальное инстанцирование

3. **Подсказки об ошибках**
   - Чёткие сообщения об ошибках
   - Подсказки при выводе параметров обобщений

4. **Параллельная компиляция**
   - Параллельное инстанцирование обобщений
   - Многопоточное решение ограничений

## Альтернативные варианты

| Вариант | Почему не выбран |
|------|--------------|
| Только базовые обобщения | Не способны заменить сложные макросы |
| Чистая макросистема | Нет типовой безопасности, плохие сообщения об ошибках |
| Только ограничения | Недостаточная гибкость |
| Обобщения во время выполнения | Есть накладные расходы на производительность |

### Риски

| Риск | Влияние | Меры по смягчению |
|------|------|------|
| Сложность решения ограничений | Слишком долгое время компиляции | Инкрементальное решение + кэширование |
| Раздувание кода | Слишком большой бинарный файл | DCE + контроль пороговых значений |
| Сложность реализации | Увеличение срока разработки | Реализация поэтапно |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

## Приложение

### БНФ синтаксиса

```bnf
# Параметры обобщений используют унифицированный синтаксис (), как часть типа функции
# Например map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в параметрах обобщений)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры обобщений — первая группа параметров в типе функции: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённые типы, такие как List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Type в параметрах обобщений автоматически заполняется компилятором из типов фактических аргументов
# Например map(numbers, f), T извлекается из numbers: List(Int), R из f: (Int) -> String
```
