# RFC-011: Система обобщений (Generics) - Абстракции с нулевой стоимостью и замена макросов

## Резюме

Настоящий документ определяет **дизайн системы обобщений (generics)** языка YaoXiang, реализующий абстракции с нулевой стоимостью благодаря мощным возможностям generics, уменьшающий зависимость от макросов за счёт оптимизаций на этапе компиляции, и предоставляющий механизмы устранения мёртвого кода.

**Основные положения дизайна**:
- **Унифицированный синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — параметры generics унифицированы с обычными параметрами
- **Механизм Type-самоописания**: `Type` — особая сущность языкового уровня, позиции `Type` в сигнатуре автоматически выводятся и заполняются
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения на типы функций
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Generics на этапе компиляции**: `N: Int` — параметры-значения generics, инстанциация константами на этапе компиляции
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценности**:
- Абстракции с нулевой стоимостью: мономорфизация на этапе компиляции, без накладных расходов во время выполнения
- Устранение мёртвого кода: анализ графа инстанциации + оптимизации LLVM
- Замена макросов: generics заменяют 90% сценариев использования макросов
- Безопасность типов: проверки на этапе компиляции, дружелюбно к IDE
- **Явное лучше неявного**: `Type` самоописывается, компилятор автоматически выводит

## Идея вселенной типов и типы, зависящие от значений

Система обобщений YaoXiang построена на **идее вселенной типов**, концептуальной модели, которая унифицирует все концепции языка в многоуровневую структуру; ключевое новшество — возведение **типов, зависящих от значений** в первоклассных граждан уровня Type2.

### Что такое тип, зависящий от значений?

**Тип, зависящий от значений** — это тип, который зависит от одного или нескольких **значений** (а не только от других типов). Эти значения могут быть вычислены на этапе компиляции, обеспечивая гарантии безопасности типов уже на стадии компиляции.

```yaoxiang
# Традиционные generics: параметры-типы
List: (T: Type) -> Type

# Тип, зависящий от значений: параметры-значения
Vec: (n: Int) -> Type  # Тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # Тип матрицы зависит от количества строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционными generics, типы YaoXiang, зависящие от значений, обладают следующими ключевыми преимуществами:

| Свойство | Традиционные generics (C++/Rust) | Типы YaoXiang, зависящие от значений |
|------|-------------------|---------------------|
| Значения, от которых зависит тип | Только параметры-типы | Может зависеть от любых значений, включая результаты вызова функций |
| Вычисление на этапе компиляции | C++ — ручная специализация шаблонов, Rust — отсутствует | Автоматическое вычисление на этапе компиляции, гарантия завершимости |
| Вычисления на уровне типов | Шаблонное метапрограммирование (сложное/опасное) | Унифицированный движок вычислений на уровне типов |
| Безопасность типов | C++ — отсутствует, Rust — ограничена | Полная безопасность типов, проверки на этапе компиляции |
| Проверка размерностей | Проверка в runtime или ручная специализация | Проверка размерностей на этапе компиляции, без накладных расходов в runtime |

### Иерархия вселенной типов и типы, зависящие от значений

Идея вселенной типов разделяет концепции языка по семантическим ролям на разные уровни; типы, зависящие от значений, располагаются на **уровне Type2**:

| Уровень | Роль | Примеры |
|------|------|------|
| Type-1 | Значения | `42`, `factorial(5)`, сама функция |
| Type0 | Ключевые слова метатипов | `Type` |
| Type1 | Конкретные типы | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функции/конструкторы типов/типы, зависящие от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевое решение**: функции, конструкторы типов и типы, зависящие от значений, на уровне Type2 имеют **унифицированный синтаксис** — все имеют форму `(params) -> result`:
- Обычная функция: `(Int, Int) -> Int` → возвращает значение
- Конструктор типа: `(T: Type) -> Type` → возвращает тип
- Тип, зависящий от значений: `(n: Int) -> Type` → возвращает тип, но зависит от параметра-значения

> **Изоморфизм Карри-Ховарда**: это единство не случайно. Изоморфизм Карри-Ховарда утверждает, что «типы суть propositions, программы суть доказательства» — тип функции `A → B` соответствует логической импликации «если A, то B», generics `(T: Type) -> Type` соответствуют квантору всеобщности «для всех типов T», тип, зависящий от значений, `(n: Int) -> Type` соответствует «для каждого целого n существует тип». YaoXiang объединяет функции, конструкторы типов и типы, зависящие от значений, на уровне Type2, что по сути объединяет «доказательство» и «вычисление» в одно понятие — **конструктивное доказательство**. Это и есть прямое воплощение изоморфизма Карри-Ховарда в дизайне языка: одна форма (`(params) -> result`) одновременно несёт логические propositions и процесс вычисления.

### Гарантия детерминированности на этапе компиляции

Идея вселенной типов YaoXiang требует: **всё на уровне Type должно быть определено на этапе компиляции**.

```yaoxiang
# Пример проверки размерности на этапе компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка на этапе компиляции: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 - выполняется на этапе компиляции
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Вычисление на этапе компиляции: factorial(3) = 6, размер вектора определяется на этапе компиляции
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов
2. Проверяет, что функция помечена спецификацией `decreases` (см. механизм проверки завершимости ниже)
3. Выполняет вычисление на этапе компиляции
4. Встраивает результат в порождаемый тип

### Сценарии применения типов, зависящих от значений

#### Проверка размерности на этапе компиляции
```yaoxiang
# Умножение матриц: проверка совпадения размерностей на этапе компиляции
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка на этапе компиляции: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка перехватывается на этапе компиляции:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива — константа этапа компиляции
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — константа этапа компиляции, может использоваться в вычислениях на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно на этапе компиляции)
```

#### Условные типы
```yaoxiang
# Условный тип If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейство типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры-типы T, R определяются на этапе компиляции
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    return result
}

# Использование полностью прозрачно, тип выводится автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # Выводится как map[Int, Int]
```

### Механизм проверки завершимости (интеграция с RFC-022)

Вычисление на этапе компиляции для типов, зависящих от значений, должно **гарантировать завершимость**, иначе система типов попадёт в бесконечный цикл. YaoXiang обеспечивает это посредством **спецификации decreases**, бесшовно интегрируясь с RFC-022.

#### Спецификация завершимости для рекурсивных функций
```yaoxiang
# Факториал на этапе компиляции: необходимо доказать завершимость
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # При каждой рекурсии n строго убывает
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # Вычисление на этапе компиляции factorial(5) = 120
```

#### Спецификация завершимости для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершимости

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза проверки типов                                         │
│  Встречается вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка спецификации decreases                           │
│     - Есть decreases: проверка выполнения условия убывания    │
│       на всех путях рекурсии                                  │
│     - Нет decreases, но очевидно завершается: прямое вычисление│
│     - Нет decreases, и может не завершиться: ошибка компиляции│
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисление на этапе компиляции (встроенным интерпретатором)│
│     - Чистая функция: прямое вычисление                       │
│     - Побочные эффекты: ошибка компиляции                     │
│       (в позиции типа побочные эффекты недопустимы)           │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                              │
│     - Vec(factorial(5)) → Vec(120)                            │
│     - Matrix(Float, 3, 3) → конкретный тип                   │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантия неизбежной завершимости вычислений на этапе компиляции, защита системы типов от бесконечных циклов
- **Унификация**: проверка завершимости и верификация частичной корректности используют единый механизм спецификаций
- **Постепенное усиление**: можно перейти от runtime-проверок к полному статическому доказательству

## Мотивация

### Зачем нужна мощная система обобщений?

Существующие mainstream-языки имеют ограничения в generics:

| Язык | Возможности generics | Проблемы |
|------|----------|------|
| Java | Граничные типы | Мономорфизация на этапе компиляции, без специализации generics |
| C# | Ограничения generics | Проверка типов в runtime, есть накладные расходы на производительность |
| Rust | Generics + Trait | Сложная система Trait, крутая кривая обучения |
| C++ | Шаблоны | Сложная специализация шаблонов, плохие сообщения об ошибках |
| **YaoXiang** | **Типы, зависящие от значений** | **Тип может зависеть от значений, проверка размерности на этапе компиляции, гарантия завершимости** |

### Ключевые противоречия

1. **Производительность vs гибкость**: гибкость в runtime vs оптимизации на этапе компиляции
2. **Сложность vs простота**: мощная система типов vs удобство использования
3. **Макросы vs generics**: кодогенерация макросами vs типобезопасность generics
4. **Зависимость от значений vs безопасность типов**: традиционные generics не могут проверять размерности на этапе компиляции

### Ключевые преимущества типов, зависящие от значений

**Типы, зависящие от значений** YaoXiang — ключевое преимущество по сравнению с традиционными generics:

| Преимущество | Описание |
|------|------|
| **Тип зависит от значения** | `Vec: (n: Int) -> Type` позволяет типу зависеть от конкретного значения |
| **Вычисление на этапе компиляции** | Вызовы функций в позициях типов вычисляются на этапе компиляции, результат встраивается в тип |
| **Проверка размерности** | `Matrix(Float, 3, 3)` проверяет размерности матрицы на этапе компиляции |
| **Вычисления на уровне типов** | Условные типы `If`, `Match` поддерживают вычисления на уровне типов |
| **Гарантия завершимости** | Спецификация decreases гарантирует завершимость вычислений на этапе компиляции |

```yaoxiang
# Проверка на этапе компиляции, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Вычисление на этапе компиляции: factorial(3) = 6, factorial(2) = 2
# Тип: Matrix(Float, 6, 2)

# Несовпадение размерностей перехватывается на этапе компиляции
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщений

```yaoxiang
# Пример: унифицированный дизайн API
# Операция map для разных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Подход с generics: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели дизайна

### Основные цели

1. **Абстракции с нулевой стоимостью** — вызов generics эквивалентен вызову с конкретным типом
2. **Устранение мёртвого кода** — анализ на этапе компиляции, инстанциация только используемых generics
3. **Замена макросов** — generics заменяют 90% сценариев использования макросов
4. **Безопасность типов** — проверка на этапе компиляции, без накладных расходов на типы в runtime
5. **Дружелюбность к IDE** — интеллектуальные подсказки, понятные сообщения об ошибках
6. **Типы, зависящие от значений** — тип может зависеть от значений, поддержка проверки размерности на этапе компиляции
7. **Безопасность вычислений на этапе компиляции** — спецификация decreases гарантирует завершимость вычислений на этапе компиляции

### Принципы дизайна

- **Определённость на этапе компиляции**: параметры generics определяются на этапе компиляции
- **Приоритет мономорфизации**: генерация конкретного кода, без виртуальных вызовов
- **Управление через ограничения**: ограничения типов руководят инстанциацией
- **Платформенные оптимизации**: специализация для поддержки платформенно-специфичных оптимизаций
- **Единство вселенной типов**: функции/конструкторы типов/типы, зависящие от значений, объединены на уровне Type2
- **Гарантия завершимости**: вызовы функций в позициях типов должны доказывать завершимость

### 1. Базовые generics

#### 1.1 Параметры-типы в generics

> **Ключевое правило**: определение обобщённого типа **должно явно аннотироваться `: Type`**, иначе HM выведет его как функцию.
>
> | Запись | Значение |
> |------|------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM выводит как функцию, а не как тип |

```yaoxiang
# Определение обобщённого типа (должно иметь : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — просто имя по соглашению, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM выводит как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Ограничение generics (прямое выражение, return в одну строку можно опустить)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров-типов
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 2.4 Встроенные маркерные trait: Dup и Clone

**Три типа семантики копирования**:

| Тип | Значение | Способ срабатывания | Сценарий применения |
|------|------|----------|----------|
| **Копирование примитивных значений** | При присваивании автоматическое копирование значения, два значения полностью независимы | Автоматически при присваивании/передаче параметра | Int, Float, Bool, Char |
| **Dup** | Поверхностное копирование: копируется дескриптор/токен, нижележащие данные разделяются | Автоматически при присваивании/передаче параметра | токены `&T`, `ref T`, String/Bytes |
| **Clone** | Глубокое копирование: создание полностью независимой копии | `value.clone()` | Любой тип, реализующий Clone |

**Семантика Dup**: типы, реализующие Dup, при присваивании/передаче параметра не передают владение — компилятор копирует дескриптор/токен, несколько владельцев указывают на одни и те же нижележащие данные. Это дополнение к семантике Move по умолчанию в модели владения из RFC-009.

**Dup и Clone — ортогональные понятия**:

```
Dup = копирование дескриптора, разделяемые данные (изменения влияют друг на друга)
Clone = копирование данных, независимые копии (изменения не влияют друг на друга)
```

**Правила**:

```
1. Примитивные типы значений (Int, Float, Bool, Char) — встроенное в компилятор копирование значений, не относится к Dup
2. Dup — применимо только к ссылочным/токеновым типам и типам с внутренним подсчётом ссылок
3. Clone — явное глубокое копирование, может быть реализовано любым типом
4. Move по умолчанию — остальные типы сохраняют семантику Move по умолчанию
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|------|-----|------|
| `&T` (токен заимствования) | ✅ | Токен нулевого размера, копирование токена = несколько точек зрения на одни данные |
| `ref T` | ✅ | Копирование Rc/Arc = счётчик ссылок +1, разделяемые данные в куче |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, копирование дескриптора разделяет буфер |
| `&mut T` (токен изменяемого заимствования) | ❌ | Линейно эксклюзивный, нельзя копировать |
| struct | Производный | Все поля Dup → struct Dup |
| enum | Производный | Все поля всех вариантов Dup → enum Dup |
| tuple | Производный | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захватываемая среда может быть не Dup |
| `*T` (сырой указатель) | ❌ | unsafe, не участвует в системе владения |

**Int/Float/Bool/Char не являются Dup** — это типы значений, при присваивании компилятор автоматически копирует значение (два значения полностью независимы). Это не «поверхностное копирование», а встроенная в компилятор обработка примитивов, не требующая и не должны выражаться через атрибут типа Dup.

```yaoxiang
# Примитивные типы значений: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x          # Копирование значения, x и y полностью независимы
print(x)       # ✅

# Dup: поверхностное копирование, копирование дескриптора разделяет данные
view: &Point = &point
view2 = view    # ✅ Dup: копирование токена, оба указывают на одну и ту же point
print(view.x)   # ✅

# Clone: явное глубокое копирование, создание независимой копии
backup = big_struct.clone()  # Явный вызов

# Ограничения generics
dup_use: (T: Dup) -> T = x         # T: Dup → можно поверхностно копировать
clone_use: (T: Clone) -> T = x.clone()  # T: Clone → можно глубоко копировать
```

> **Примечание**: `Send`/`Sync` не являются видимыми пользователю trait. Безопасность межзадачных операций обеспечивается ключевым словом `ref` и полностью автоматической обработкой компилятора — `ref` автоматически выбирает Rc или Arc, пользователю не нужно понимать Send/Sync.

}

        self.index = self.index + 1
        return Option.some(item)
    } else {
        return Option.none()
    }
}

}

### 5. Условные типы

> **Изоморфизм Карри-Ховарда**: условные типы с точки зрения Карри-Ховарда — это **case-анализ** в логике. Тип `Bool` соответствует proposition с двумя возможными значениями (True/False), `If` в зависимости от истинности выбирает разные результаты — это в точности логическая дизъюнкция case. `match C { True => T, False => E }` фактически выражает: «когда proposition C истинно, заключение — T; когда ложно — E».

#### 5.1 Условный тип If

```yaoxiang
# Условный If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Верификация на этапе компиляции
Assert: (C: Bool) -> Type = match C {
    True => Void,
    False => compile_error("Assertion failed"),
}

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри-Ховарда**: семейства типов — наиболее прямое воплощение «propositions как типов». `Add: (A: Type, B: Type) -> Type` — это не «функция сложения, написанная на уровне типов», а **построение proposition о сложении натуральных чисел**. `(Zero, B) => B` говорит «proposition Add(Zero, B) эквивалентно B», `(Succ(A'), B) => Succ(Add(A', B))` говорит «если Add(A', B) истинно, то и Add(Succ(A'), B) тоже истинно». Это и есть определение сложения из аксиом Пеано. То, что проверка типов проходит для этого match-выражения, эквивалентно проверке логической согласованности этого определения.

```yaoxiang
# Преобразование типов на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # по умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Карри-Ховард: это также индуктивное определение сложения натуральных чисел)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: вычисление 2 + 3 на этапе компиляции
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

struct InstantiationGraph {
    // Узлы: инстанциации generics
    nodes: HashMap<InstanceKey, InstanceNode>,

    // Рёбра: отношения использования
    edges: HashMap<InstanceKey, Vec<InstanceKey>>,
}

    generic: FunctionId,  // ID обобщённой функции
    type_args: Vec<TypeId>,  // Параметры-типы
    const_args: Vec<ConstId>,  // Параметры-константы
}

// Алгоритм: анализ достижимости
fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Не посещённые инстанциации — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}

    push: (self: List(T), item: T) -> Void,
    pop: (self: List(T)) -> Option(T),
    map: (R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)),
    filter: (self: List(T), predicate: (T) -> Bool) -> List(T),

# Определения функций находятся в пространстве имён List (префикс List. = принадлежность пространству имён)
# Чтобы вызов через . работал (например, list.push(item)), требуется явная привязка: List.push = push[0]
# self — просто имя параметра по соглашению, компилятор смотрит на тип, а не на имя

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

List.pop: (T: Type) -> ((self: List(T)) -> Option(T)) = {
    if self.length > 0 {
        self.length = self.length - 1
        return Option.some(self.data[self.length])
    } else {
        return Option.none()
    }
}

List.map: (T: Type, R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)) = {
    result = List(R)()
    for i in 0..self.length {
        result.push(f(self.data[i]))
    }
    return result
}

List.filter: (T: Type) -> ((self: List(T), predicate: (T) -> Bool) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        if predicate(self.data[i]) {
            result.push(self.data[i])
        }
    }
    return result
}

List.fold: (T: Type, U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U) = {
    result = initial
    for i in 0..self.length {
        result = f(result, self.data[i])
    }
    return result
}

# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённая быстрая сортировка
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Реализация с использованием перегрузки функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========

# ======== 1. Тип матрицы, известный на этапе компиляции ========
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),

    # Проверка размерности на этапе компиляции: использование стандартного типа библиотеки Assert
    _assert: Assert[Rows > 0],  # Rows > 0, иначе ошибка компиляции
    _assert: Assert[Cols > 0],  # Cols > 0, иначе ошибка компиляции

    # Операции над матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {
            for j in 0..M {
                sum = Zero::zero()
                for k in 0..Cols {
                    sum = sum + self.data[i][k] * other.data[k][j]
                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

identity: (T: Add + Multiply + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    for i in 0..N {
        for j in 0..N {
            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    return matrix
}

matrix_2x3.data[1][1] = 5.0
matrix_2x3.data[1][2] = 6.0

# Матрица 3x2
matrix_3x2 = Matrix(Float, 3, 2)()
matrix_3x2.data[0][0] = 7.0
matrix_3x2.data[0][1] = 8.0
matrix_3x2.data[1][0] = 9.0
matrix_3x2.data[1][1] = 10.0

matrix_3x2.data[2][1] = 12.0

# Умножение матриц: 2x3 * 3x2 = 2x2
result = matrix_2x3.multiply(matrix_3x2)

# Единичная матрица 2x2
identity_3x3 = identity(Float, 3)()

## Компромиссы

### Преимущества

1. **Абстракции с нулевой стоимостью**
   - Мономорфизация на этапе компиляции, без накладных расходов в runtime
   - Виртуальные функции не нужны, RTTI не нужен

2. **Устранение мёртвого кода**
   - Анализ на этапе компиляции, инстанциация только используемых generics
   - Раздувание кода контролируемо

3. **Замена макросов**
   - Типобезопасная кодогенерация
   - Дружелюбно к IDE, понятные сообщения об ошибках

4. **Вычисления на этапе компиляции**
   - Generics на этапе компиляции поддерживают compile-time вычисления
   - Возможности вроде проверки размерности
   - Ключевое слово `const` не требуется, чистые ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанциация generics увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Потребление памяти**
   - Увеличивается потребление памяти компилятором
   - Механизмы кеширования требуют памяти

3. **Сложность реализации**
   - Сложный решатель ограничений
   - Сложный движок вычислений на уровне типов

4. **Диагностика ошибок**
   - Ошибки generics могут быть сложными
   - Требуются понятные подсказки об ошибках

### Меры по смягчению

1. **Стратегии кеширования**
   - Кеширование результатов инстанциации
   - LRU-кеш для ограничения памяти

2. **Инкрементальная компиляция**
   - Кеширование результатов компиляции
   - Инкрементальная инстанциация

3. **Подсказки об ошибках**
   - Понятные сообщения об ошибках
   - Подсказки по выводу параметров generics

4. **Параллельная компиляция**
   - Параллельная инстанциация generics
   - Многопоточное решение ограничений

## Альтернативы

| Подход | Почему не выбран |
|------|--------------|
| Только базовые generics | Невозможно заменить сложные макросы |
| Чистая система макросов | Нет типобезопасности, плохие сообщения об ошибках |
| Только ограничения-зависимости | Недостаточная гибкость |
| Generics в runtime | Есть накладные расходы на производительность |

### Риски

| Риск | Влияние | Меры по смягчению |
|------|------|----------|
| Сложность решения ограничений | Слишком долгая компиляция | Инкрементальное решение + кеширование |
| Раздувание кода | Слишком большой бинарный файл | DCE + пороговый контроль |
| Сложность реализации | Увеличение сроков разработки | Поэтапная реализация |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

## Приложение

### Синтаксис BNF

```bnf
# Параметры generics используют унифицированный синтаксис () как часть типа функции
# Например, map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в параметрах generics)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры generics — первая группа параметров в типе функции: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённый тип, например List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Type в параметрах generics автоматически заполняется компилятором из типов фактических аргументов
# Например, map(numbers, f): T извлекается из numbers: List(Int), R — из f: (Int) -> String
```
