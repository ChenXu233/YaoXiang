# RFC-011: Система обобщённого программирования — абстракции с нулевой стоимостью и замена макросов

## Аннотация

Данный документ определяет **дизайн системы обобщённого программирования (generics)** языка YaoXiang, реализующий абстракции с нулевой стоимостью за счёт мощных возможностей обобщений, использующий оптимизации времени компиляции для снижения зависимости от макросов и предоставляющий механизм устранения мёртвого кода.

**Ключевые решения**:
- **Унифицированный синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — параметры обобщений унифицированы с обычными параметрами
- **Механизм самоописания Type**: `Type` — особая сущность уровня языка; позиции `Type` в сигнатуре автоматически заполняются путём вывода
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения на типы функций
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Обобщения времени компиляции**: `N: Int` — параметры-значения обобщений, инстанцирование константами времени компиляции
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Абстракции с нулевой стоимостью: мономорфизация на этапе компиляции, без накладных расходов во время выполнения
- Устранение мёртвого кода: анализ графа инстанцирования + оптимизации LLVM
- Замена макросов: обобщения заменяют 90% сценариев использования макросов
- Типобезопасность: проверка на этапе компиляции, дружественность к IDE
- **Явное лучше неявного**: `Type` — самоописательный, компилятор выполняет автоматический вывод

## Идея вселенной типов и типы, зависящие от значений

Система обобщений YaoXiang построена на **идее вселенной типов** — ментальной модели, которая унифицирует все концепции языка в многоуровневую структуру. Ключевая инновация — возведение **типов, зависящих от значений**, в полноправные сущности уровня Type2.

### Что такое тип, зависящий от значений?

**Тип, зависящий от значений** — это тип, который зависит от одного или нескольких **значений** (а не только от других типов). Эти значения могут быть вычислены на этапе компиляции, обеспечивая гарантии типобезопасности уже на стадии компиляции.

```yaoxiang
# Традиционные обобщения: параметры типов
List: (T: Type) -> Type

# Тип, зависящий от значений: параметр-значение
Vec: (n: Int) -> Type  # тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # тип матрицы зависит от количества строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционными обобщениями, типы, зависящие от значений в YaoXiang, обладают следующими ключевыми преимуществами:

| Характеристика | Традиционные обобщения (C++/Rust) | Типы, зависящие от значений в YaoXiang |
|------|-------------------|---------------------|
| Значения, от которых зависит тип | Только параметры типов | Может зависеть от любых значений, включая результаты вызова функций |
| Вычисление на этапе компиляции | C++: ручная специализация шаблонов, Rust: отсутствует | Автоматическое вычисление на этапе компиляции с гарантией завершимости |
| Вычисления на уровне типов | Метапрограммирование шаблонов (сложное/опасное) | Унифицированный движок вычислений на уровне типов |
| Типобезопасность | C++: отсутствует, Rust: ограничена | Полная типобезопасность, проверка на этапе компиляции |
| Проверка размерностей | Проверка во время выполнения или ручная специализация | Проверка размерностей на этапе компиляции, без накладных расходов в рантайме |

### Уровни вселенной типов и типы, зависящие от значений

Идея вселенной типов разделяет концепции языка по семантическим ролям на различные уровни. Типы, зависящие от значений, располагаются на **уровне Type2**:

| Уровень | Роль | Примеры |
|------|------|------|
| Type-1 | Значения | `42`, `factorial(5)`, сама функция |
| Type0 | Ключевые слова метатипов | `Type` |
| Type1 | Конкретные типы | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функции/конструкторы типов/типы, зависящие от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевое решение**: функции, конструкторы типов и типы, зависящие от значений на уровне Type2 имеют **унифицированный синтаксис** — форму `(params) -> result`:
- Обычная функция: `(Int, Int) -> Int` → результат — значение
- Конструктор типа: `(T: Type) -> Type` → результат — тип
- Тип, зависящий от значений: `(n: Int) -> Type` → результат — тип, но зависящий от параметра-значения

> **Изоморфизм Карри-Ховарда**: это единство — не совпадение. Изоморфизм Карри-Ховарда утверждает «тип — это утверждение, программа — это доказательство»: тип функции `A → B` соответствует логической импликации «если A, то B»; обобщение `(T: Type) -> Type` соответствует квантору всеобщности «для всех типов T»; тип, зависящий от значений `(n: Int) -> Type`, соответствует «для каждого целого n существует тип». YaoXiang объединяет функции, конструкторы типов и типы, зависящие от значений, на уровне Type2, по сути, объединяя «доказательство» и «вычисление» в одну концепцию — **конструктивное доказательство**. Это прямое воплощение изоморфизма Карри-Ховарда в дизайне языка: одна форма `(params) -> result` одновременно несёт логические утверждения и процесс вычисления.

### Гарантия детерминированности на этапе компиляции

Идея вселенной типов в YaoXiang требует: **всё на уровне Type должно быть определено на этапе компиляции**.

```yaoxiang
# Пример проверки размерностей на этапе компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка на этапе компиляции: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется на этапе компиляции
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Вычисление на этапе компиляции: factorial(3) = 6, размер вектора определяется на этапе компиляции
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позиции типа
2. Проверяет, помечена ли функция спецификацией `decreases` (см. механизм проверки завершимости ниже)
3. Выполняет вычисление на этапе компиляции
4. Встраивает результат в сгенерированный тип

### Сценарии применения типов, зависящих от значений

#### Проверка размерностей на этапе компиляции
```yaoxiang
# Умножение матриц: проверка совпадения размерностей на этапе компиляции
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка на этапе компиляции: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка перехватывается на этапе компиляции:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива — константа времени компиляции
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — константа времени компиляции, может использоваться в вычислениях на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно на этапе компиляции)
```

#### Условные типы
```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейства типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры типов T, R определяются на этапе компиляции
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    return result
}

# При использовании полностью прозрачно, тип выводится автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # выводится как map[Int, Int]
```

### Механизм проверки завершимости (интеграция с RFC-022)

Вычисление на этапе компиляции для типов, зависящих от значений, должно **гарантировать завершимость**, иначе система типов попадёт в бесконечный цикл. YaoXiang обеспечивает это через **спецификацию decreases**, бесшовно интегрируясь с RFC-022.

#### Спецификация завершимости для рекурсивных функций
```yaoxiang
# Факториал на этапе компиляции: необходимо доказать завершимость
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # при каждой рекурсии n строго уменьшается
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # вычисление на этапе компиляции factorial(5) = 120
```

#### Спецификация завершимости для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершимости

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза проверки типов                                         │
│  Встречается вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка спецификации decreases                          │
│     - Есть decreases: проверить, что условие убывания        │
│       выполняется на всех путях рекурсии                      │
│     - Нет decreases, но явно завершима: вычислить напрямую   │
│     - Нет decreases и может не завершиться: ошибка компиляции│
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисление на этапе компиляции (встроенным интерпретатором)│
│     - Чистая функция: вычислить напрямую                      │
│     - Побочные эффекты: ошибка компиляции                     │
│       (в позиции типа побочные эффекты запрещены)            │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                             │
│     - Vec(factorial(5)) → Vec(120)                           │
│     - Matrix(Float, 3, 3) → конкретный тип                   │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантия, что вычисление на этапе компиляции обязательно завершится, предотвращает бесконечные циклы в системе типов
- **Унифицированность**: проверка завершимости и верификация частичной корректности используют единый механизм спецификаций
- **Прогрессивное усиление**: возможен постепенный переход от проверок в рантайме к полному статическому доказательству

## Мотивация

### Зачем нужна мощная система обобщений?

Современные mainstream-языки имеют ограничения в обобщениях:

| Язык | Возможности обобщений | Проблема |
|------|----------|------|
| Java | Граничные типы | Мономорфизация на этапе компиляции, без специализации обобщений |
| C# | Ограничения обобщений | Проверка типов в рантайме, есть потеря производительности |
| Rust | Обобщения + Trait | Сложная система Trait, крутая кривая обучения |
| C++ | Шаблоны | Сложная специализация шаблонов, плохие сообщения об ошибках |
| **YaoXiang** | **Типы, зависящие от значений** | **Типы могут зависеть от значений, проверка размерностей на этапе компиляции, гарантия завершимости** |

### Ключевые противоречия

1. **Производительность vs гибкость**: гибкость в рантайме vs оптимизации на этапе компиляции
2. **Сложность vs простота**: мощная система типов vs удобство использования
3. **Макросы vs обобщения**: кодогенерация макросами vs типобезопасность обобщений
4. **Зависимость от значений vs типобезопасность**: традиционные обобщения не могут проверять размерности на этапе компиляции

### Ключевые преимущества типов, зависящих от значений

**Типы, зависящие от значений** в YaoXiang — это ключевое преимущество по сравнению с традиционными обобщениями:

| Преимущество | Описание |
|------|------|
| **Типы зависят от значений** | `Vec: (n: Int) -> Type` позволяет типам зависеть от конкретных значений |
| **Вычисление на этапе компиляции** | Вызовы функций в позиции типа вычисляются на этапе компиляции, результат встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` проверяет размерности матрицы на этапе компиляции |
| **Вычисления на уровне типов** | Условные типы `If`, `Match` поддерживают вычисления на уровне типов |
| **Гарантия завершимости** | Спецификация decreases гарантирует, что вычисление на этапе компиляции обязательно завершится |

```yaoxiang
# Проверка на этапе компиляции, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Вычисление на этапе компиляции: factorial(3) = 6, factorial(2) = 2
# Тип — Matrix(Float, 6, 2)

# Несовпадение размерностей перехватывается на этапе компиляции
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщений

```yaoxiang
# Пример: унифицированный дизайн API
# операция map для разных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Подход с обобщениями: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели дизайна

### Основные цели

1. **Абстракции с нулевой стоимостью** — вызов обобщённой функции эквивалентен вызову для конкретного типа
2. **Устранение мёртвого кода** — анализ на этапе компиляции, инстанцирование только используемых обобщений
3. **Замена макросов** — обобщения заменяют 90% сценариев использования макросов
4. **Типобезопасность** — проверка на этапе компиляции, без накладных расходов в рантайме
5. **Дружественность к IDE** — интеллектуальные подсказки, понятные сообщения об ошибках
6. **Типы, зависящие от значений** — типы могут зависеть от значений, поддержка проверки размерностей на этапе компиляции
7. **Безопасность вычислений на этапе компиляции** — спецификация decreases гарантирует завершимость

### Принципы дизайна

- **Определённость на этапе компиляции** — параметры обобщений определяются на этапе компиляции
- **Приоритет мономорфизации** — генерация конкретного кода, избегание вызовов виртуальных функций
- **Управление через ограничения** — ограничения типов направляют инстанцирование
- **Платформенные оптимизации** — специализация для платформенно-специфичных оптимизаций
- **Единство вселенной типов** — функции/конструкторы типов/типы, зависящие от значений, объединены на уровне Type2
- **Гарантия завершимости** — вызовы функций в позиции типа должны доказывать завершимость

### 1. Базовые обобщения

#### 1.1 Параметры обобщённых типов

> **Ключевое правило**: определение обобщённого типа **должно явно аннотироваться `: Type`**, иначе будет выведено HM как функция.
>
> | Запись | Значение |
> |------|------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM выводит как функцию, а не тип |

```yaoxiang
# Определение обобщённого типа (должно содержать : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — только имя-конвенция, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM выводит как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Ограничение обобщений (прямое выражение, для одной строки return можно опустить)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров типов
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 2.4 Встроенные marker trait: Dup и Clone

**Три вида семантики копирования**:

| Тип | Значение | Способ срабатывания | Сценарий применения |
|------|------|----------|----------|
| **Копирование примитивных значений** | При присваивании автоматическое копирование значения, два значения полностью независимы | Автоматически при присваивании/передаче параметров | Int, Float, Bool, Char |
| **Dup** | Поверхностное копирование: копирование дескриптора/токена, базовые данные разделяются | Автоматически при присваивании/передаче параметров | `&T` токены, `ref T`, String/Bytes |
| **Clone** | Глубокое копирование: создание полностью независимой копии | `value.clone()` | Любой тип, реализующий Clone |

**Семантика Dup**: для типов, реализующих Dup, при присваивании/передаче параметров владение не передаётся — компилятор копирует дескриптор/токен, и несколько владельцев указывают на одни и те же базовые данные. Это дополняет семантику Move по умолчанию в модели владения из RFC-009.

**Dup и Clone — ортогональные концепции**:

```
Dup = копирование дескриптора, разделяемые данные (изменения влияют друг на друга)
Clone = копирование данных, независимая копия (изменения не влияют друг на друга)
```

**Правила**:

```
1. Примитивные типы значений (Int, Float, Bool, Char) — встроенное в компилятор копирование значений, не относятся к Dup
2. Dup — применимо только к ссылочным/токенным типам и типам с внутренним подсчётом ссылок
3. Clone — явное глубокое копирование, любой тип может реализовать
4. Move по умолчанию — остальные типы сохраняют семантику Move по умолчанию
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|------|-----|------|
| `&T` (токен заимствования) | ✅ | Токен нулевого размера, копирование токена = несколько точек зрения на одни данные |
| `ref T` | ✅ | Копирование Rc/Arc = счётчик ссылок +1, разделяемые данные в куче |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, копирование дескриптора разделяет базовый буфер |
| `&mut T` (токен изменяемого заимствования) | ❌ | Линейная эксклюзивность, нельзя копировать |
| struct | Производный | Все поля Dup → struct Dup |
| enum | Производный | Все поля всех variant Dup → enum Dup |
| tuple | Производный | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захваченное окружение может быть не Dup |
| `*T` (сырой указатель) | ❌ | unsafe, не участвует в системе владения |

**Int/Float/Bool/Char не являются Dup** — это типы значений; при присваивании компилятор автоматически копирует значение (два значения полностью независимы). Это не «поверхностное копирование», а встроенная в компилятор обработка примитивов, которая не нуждается и не должна выражаться через атрибут типа Dup.

```yaoxiang
# Примитивные типы значений: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x          # Копирование значения, x и y полностью независимы
print(x)       # ✅

# Dup: поверхностное копирование, дескриптор копирует — данные разделяются
view: &Point = &point
view2 = view    # ✅ Dup: токен скопирован, оба указывают на один и тот же point
print(view.x)   # ✅

# Clone: явное глубокое копирование, создание независимой копии
backup = big_struct.clone()  # Явный вызов

# Ограничения обобщений
dup_use: (T: Dup) -> T = x         # T: Dup → можно поверхностно копировать
clone_use: (T: Clone) -> T = x.clone()  # T: Clone → можно глубоко копировать
```

> **Примечание**: `Send`/`Sync` не являются видимыми для пользователя trait. Гарантии безопасности между задачами полностью обрабатываются ключевым словом `ref` и компилятором автоматически — `ref` автоматически выбирает Rc или Arc, пользователю не нужно понимать Send/Sync.

```yaoxiang
Iterator: (Item: Type) -> Type = {

        }
    }
    return result
}

# Реализация Iterator для Array
# Использование синтаксического сахара: Array.Item, Array.next, Array.has_next
Array.has_next: (T: Type)(self: Array(T)) -> Bool = {
    return self.index < self.length
}

Array.next: (T: Type)(self: Array(T)) -> Option(T) = {
    if has_next(self) {
        item = self.data[self.index]
        self.index = self.index + 1
        return Option.some(item)
    } else {
        return Option.none()
    }
}

Array.Item: (T: Type)(arr: Array(T)) -> T = {
    return arr.data[0]
}
```

### 5. Условные типы

> **Изоморфизм Карри-Ховарда**: условные типы с точки зрения Карри-Ховарда представляют **case-анализ** в логике. Тип `Bool` соответствует утверждению с двумя возможными значениями (True/False), `If` выбирает разные результаты в зависимости от истинности этого утверждения — это и есть логическая дизъюнкция case. `match C { True => T, False => E }` фактически выражает: «когда известно, что утверждение C истинно, заключение — T; когда C ложно, заключение — E».

#### 5.1 Условный тип If

```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Проверка на этапе компиляции
Assert: (C: Bool) -> Type = match C {
    True => Void,
    False => compile_error("Утверждение не выполнено"),
}

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри-Ховарда**: семейства типов — наиболее прямое воплощение принципа «утверждение как тип». `Add: (A: Type, B: Type) -> Type` — это не «функция сложения на уровне типов», а **конструирование утверждения о сложении натуральных чисел**. `(Zero, B) => B` означает «утверждение Add(Zero, B) эквивалентно B»; `(Succ(A'), B) => Succ(Add(A', B))` означает «если Add(A', B) верно, то Add(Succ(A'), B) тоже верно». Это само определение сложения в аксиомах Пеано. То, что проверка типов подтверждает прохождение этого match-выражения, эквивалентно проверке логической согласованности этого определения.

```yaoxiang
# Преобразование типа на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # по умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Карри-Ховард: это индуктивное определение сложения натуральных чисел)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: вычисление 2 + 3 на этапе компиляции
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

```rust
struct InstantiationGraph {
    // Узлы: инстанцирования обобщений
    nodes: HashMap<InstanceKey, InstanceNode>,

    // Рёбра: отношения использования
    edges: HashMap<InstanceKey, Vec<InstanceKey>>,

    generic: FunctionId,  // ID обобщённой функции
    type_args: Vec<TypeId>,  // аргументы типов
    const_args: Vec<ConstId>,  // аргументы Const
}

// Алгоритм: анализ достижимости
fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Не посещённые инстанцирования — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}
```

```yaoxiang
# Определения функций находятся в пространстве имён List (префикс List. = принадлежность пространству имён)
# Для работы синтаксиса . вызова (list.push(item)) требуется явная привязка: List.push = push[0]
# self — только имя-конвенция для параметра, компилятор смотрит на тип, а не на имя

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

List.pop: (T: Type) -> ((self: List(T)) -> Option(T)) = {
    if self.length > 0 {
        self.length = self.length - 1
        return Option.some(self.data[self.length])
    } else {
        return Option.none()
    }
}

List.map: (T: Type, R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)) = {
    result = List(R)()
    for i in 0..self.length {
        result.push(f(self.data[i]))
    }
    return result
}

List.filter: (T: Type) -> ((self: List(T), predicate: (T) -> Bool) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        if predicate(self.data[i]) {
            result.push(self.data[i])
        }
    }
    return result
}

List.fold: (T: Type, U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U) = {
    result = initial
    for i in 0..self.length {
        result = f(result, self.data[i])

    return result
}
```

```yaoxiang
# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённая быстрая сортировка
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Реализация через перегрузку функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========
# Сортировка массива Int
```

```yaoxiang
# ======== 1. Тип матрицы времени компиляции ========
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),

    # Проверка размерностей на этапе компиляции: использование типа Assert из стандартной библиотеки
    _assert: Assert[Rows > 0],  # Rows > 0, иначе ошибка компиляции
    _assert: Assert[Cols > 0],  # Cols > 0, иначе ошибка компиляции

    # Операции над матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {
            for j in 0..M {
                sum = Zero::zero()
                for k in 0..Cols {
                    sum = sum + self.data[i][k] * other.data[k][j]
                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

# ======== 2. Создание матрицы на этапе компиляции ========
identity: (T: Add + Multiply + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    for i in 0..N {
        for j in 0..N {
            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    return matrix
}

matrix_2x3.data[0][0] = 1.0

matrix_3x2 = Matrix(Float, 3, 2)()

result = matrix_2x3.multiply(matrix_3x2)
```

## Компромиссы

### Преимущества

1. **Абстракции с нулевой стоимостью**
   - Мономорфизация на этапе компиляции, без накладных расходов в рантайме
   - Виртуальные функции не нужны, RTTI не используется

2. **Устранение мёртвого кода**
   - Анализ на этапе компиляции, инстанцирование только используемых обобщений
   - Раздувание кода контролируемо

3. **Замена макросов**
   - Типобезопасная кодогенерация
   - Дружественность к IDE, понятные сообщения об ошибках

4. **Вычисления на этапе компиляции**
   - Обобщения времени компиляции поддерживают вычисления на этапе компиляции
   - Возможности вроде проверки размерностей
   - Ключевое слово `const` не требуется, чистые ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанцирование обобщений увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Потребление памяти**
   - Увеличивается потребление памяти компилятором
   - Механизмы кэширования требуют памяти

3. **Сложность реализации**
   - Решатель ограничений сложен
   - Движок вычислений на уровне типов сложен

4. **Диагностика ошибок**
   - Ошибки обобщений могут быть сложными
   - Требуются понятные подсказки об ошибках

### Меры по смягчению

1. **Стратегии кэширования**
   - Кэширование результатов инстанцирования
   - Ограничение памяти LRU-кэша

2. **Инкрементальная компиляция**
   - Кэширование результатов компиляции
   - Инкрементальное инстанцирование

3. **Подсказки об ошибках**
   - Понятные сообщения об ошибках
   - Подсказки для вывода параметров обобщений

4. **Параллельная компиляция**
   - Параллельное инстанцирование обобщений
   - Многопоточное решение ограничений

## Альтернативы

| Альтернатива | Почему не выбрана |
|------|--------------|
| Только базовые обобщения | Не могут заменить сложные макросы |
| Чистая система макросов | Нет типобезопасности, плохие сообщения об ошибках |
| Только ограничения зависимостей | Недостаточная гибкость |
| Обобщения времени выполнения | Есть потеря производительности |

### Риски

| Риск | Влияние | Мера по смягчению |
|------|------|------|
| Сложность решения ограничений | Слишком долгая компиляция | Инкрементальное решение + кэширование |
| Раздувание кода | Слишком большой бинарный файл | DCE + пороговый контроль |
| Сложность реализации | Увеличение цикла разработки | Поэтапная реализация |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

## Приложение

### БНФ грамматики

```bnf
# Параметры обобщений используют унифицированный синтаксис () как часть типа функции
# Например, map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в параметрах обобщений)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры обобщений — первая группа параметров в типе функции: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённый тип, например List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Type в параметрах обобщений автоматически заполняется компилятором из типов фактических аргументов
# Например, map(numbers, f): T извлекается из numbers: List(Int), R — из f: (Int) -> String
```
