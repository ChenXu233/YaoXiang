# RFC-011: Проект системы обобщений — абстракции с нулевой стоимостью и замена макросов

## Резюме

Документ определяет **проект системы обобщений** языка YaoXiang, реализующий абстракции с нулевой стоимостью через мощные возможности обобщений, снижающий зависимость от макросов за счёт компайл-тайм оптимизаций и предоставляющий механизм удаления мёртвого кода.

**Ключевые решения**:
- **Единый синтаксис сигнатур**: `(T: Type, R: Type) -> ...` параметры обобщений унифицированы с обычными параметрами
- **Механизм самоописания Type**: `Type` — особая сущность уровня языка, позиции `Type` в сигнатурах автоматически заполняются путём вывода
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения на функции
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Компайл-тайм обобщения**: `N: Int` — параметры-значения обобщений, инстанциация компайл-тайм константами
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Абстракции с нулевой стоимостью: компайл-тайм мономорфизация, без накладных расходов в рантайме
- Удаление мёртвого кода: анализ графа инстанциации + оптимизации LLVM
- Замена макросов: обобщения покрывают 90% сценариев использования макросов
- Безопасность типов: компайл-тайм проверка, IDE-дружественность
- **Явное лучше неявного**: самоописание `Type`, автоматический вывод компилятора

## Идея вселенной типов и зависимые от значений типы

Система обобщений YaoXiang построена на **идее вселенной типов** — ментальной модели, унифицирующей все концепции языка в многоуровневую структуру, ключевая инновация которой состоит в возведении **зависимых от значений типов** в полноправные сущности уровня Type2.

### Что такое зависимые от значений типы?

**Зависимый от значения тип** — это тип, зависящий от одного или нескольких **значений** (а не только от других типов). Эти значения могут вычисляться в компайл-тайм, обеспечивая гарантии безопасности типов уже на этапе компиляции.

```yaoxiang
# Традиционные обобщения: параметры-типы
List: (T: Type) -> Type

# Зависимый от значения тип: параметр-значение
Vec: (n: Int) -> Type  # тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # тип матрицы зависит от числа строк и столбцов
```

### Ключевые преимущества зависимых от значений типов

По сравнению с традиционными обобщениями, зависимые от значений типы YaoXiang обладают следующими ключевыми преимуществами:

| Свойство | Традиционные обобщения (C++/Rust) | Зависимые от значений типы YaoXiang |
|------|-------------------|---------------------|
| Зависимость типов от значений | Только от параметров-типов | От любых значений, включая результаты вызовов функций |
| Компайл-тайм вычисление | C++ — ручная специализация шаблонов, Rust — отсутствует | Автоматическое компайл-тайм вычисление, гарантия завершимости |
| Вычисления на уровне типов | Метапрограммирование шаблонов (сложное/опасное) | Единый движок вычислений на уровне типов |
| Безопасность типов | C++ — отсутствует, Rust — ограничена | Полная безопасность типов, компайл-тайм проверка |
| Проверка размерностей | Рантайм-проверка или ручная специализация | Компайл-тайм проверка размерностей, без накладных расходов |

### Уровни вселенной типов и зависимые от значений типы

Идея вселенной типов разделяет концепции языка по семантической роли на разные уровни; зависимые от значений типы располагаются на **уровне Type2**:

| Уровень | Роль | Примеры |
|------|------|------|
| Type-1 | Значения | `42`, `factorial(5)`, сами функции |
| Type0 | Ключевые слова метатипов | `Type` |
| Type1 | Конкретные типы | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функции/конструкторы типов/зависимые от значений типы** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевое решение**: функции, конструкторы типов и зависимые от значений типы на уровне Type2 имеют **единый синтаксис** — форму `(params) -> result`:
- Обычная функция: `(Int, Int) -> Int` → результат — значение
- Конструктор типа: `(T: Type) -> Type` → результат — тип
- Зависимый от значения тип: `(n: Int) -> Type` → результат — тип, но зависит от параметра-значения

> **Изоморфизм Карри-Ховарда**: это единство — не совпадение. Изоморфизм Карри-Ховарда утверждает, что «типы суть propositions, программы суть доказательства» — тип функции `A → B` соответствует логической импликации «если A, то B», обобщение `(T: Type) -> Type` соответствует квантору всеобщности «для всех типов T», зависимый от значения тип `(n: Int) -> Type` соответствует «для каждого целого n существует тип». YaoXiang унифицирует функции, конструкторы типов и зависимые от значений типы на уровне Type2, по сути объединяя «доказательство» и «вычисление» в одну концепцию — **конструктивное доказательство**. Это прямое воплощение изоморфизма Карри-Ховарда в дизайне языка: одна форма (`(params) -> result`) несёт в себе и логические propositions, и вычислительный процесс.

### Гарантия компайл-тайм детерминированности

Идея вселенной типов YaoXiang требует: **всё на уровне Type детерминировано в компайл-тайм**.

```yaoxiang
# Пример компайл-тайм проверки размерностей
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Компайл-тайм проверка: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется в компайл-тайм
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Компайл-тайм вычисление: factorial(3) = 6, размер вектора определён в компайл-тайм
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов
2. Проверяет наличие у функции спецификации `decreases` (см. механизм проверки завершимости ниже)
3. Выполняет вычисление в компайл-тайм
4. Встраивает результат в порождаемый тип

### Сценарии применения зависимых от значений типов

#### Компайл-тайм проверка размерностей
```yaoxiang
# Умножение матриц: компайл-тайм проверка соответствия размерностей
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Компайл-тайм проверка: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка обнаруживается в компайл-тайм:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива — компайл-тайм константа
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — компайл-тайм константа, может использоваться в вычислениях на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно в компайл-тайм)
```

#### Условные типы
```yaoxiang
# Уровневый If
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейство типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры типов T, R определены в компайл-тайм
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    return result
}

# Использование полностью прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # выводится как map[Int, Int]
```

### Механизм проверки завершимости (интеграция с RFC-022)

Компайл-тайм вычисление зависимых от значений типов должно **гарантировать завершимость**, иначе система типов попадёт в бесконечный цикл. YaoXiang обеспечивает это с помощью **спецификации decreases**, бесшовно интегрируясь с RFC-022.

#### Спецификация завершимости для рекурсивных функций
```yaoxiang
# Компайл-тайм факториал: необходимо доказать завершимость
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # на каждом шаге рекурсии n строго убывает
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # компайл-тайм вычисление factorial(5) = 120
```

#### Спецификация завершимости для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершимости

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза проверки типов                                         │
│  Встречается вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка спецификации decreases                           │
│     - Есть decreases: проверить, что условие убывания        │
│       выполняется на всех путях рекурсии                      │
│     - Нет decreases, но очевидно завершимо: прямое вычисление │
│     - Нет decreases и может не завершиться: ошибка компиляции │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Компайл-тайм вычисление (исполняется встроенным         │
│     интерпретатором)                                         │
│     - Чистая функция: прямое вычисление                      │
│     - Побочные эффекты: ошибка компиляции (позиция типа     │
│       должна быть без побочных эффектов)                     │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                             │
│     - Vec(factorial(5)) → Vec(120)                          │
│     - Matrix(Float, 3, 3) → конкретный тип                  │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантируется завершимость компайл-тайм вычислений, исключается зацикливание системы типов
- **Единообразие**: проверка завершимости и верификация частичной корректности используют единый механизм спецификаций
- **Постепенное усиление**: можно переходить от рантайм-проверок к полному статическому доказательству

## Мотивация

### Зачем нужна мощная система обобщений?

Существующие mainstream-языки имеют ограничения в обобщениях:

| Язык | Возможности обобщений | Проблема |
|------|----------|------|
| Java | Граничные типы | Компайл-тайм мономорфизация, нет специализации обобщений |
| C# | Ограничения обобщений | Рантайм-проверка типов, накладные расходы |
| Rust | Обобщения + Trait | Сложная система Trait, крутая кривая обучения |
| C++ | Шаблоны | Сложная специализация шаблонов, плохие сообщения об ошибках |
| **YaoXiang** | **Зависимые от значений типы** | **Типы могут зависеть от значений, компайл-тайм проверка размерностей, гарантия завершимости** |

### Ключевые противоречия

1. **Производительность vs гибкость**: гибкость рантайма vs компайл-тайм оптимизации
2. **Сложность vs простота**: мощная система типов vs удобство использования
3. **Макросы vs обобщения**: кодогенерация макросами vs типобезопасность обобщений
4. **Зависимость от значений vs безопасность типов**: традиционные обобщения не могут проверять размерности в компайл-тайм

### Ключевые преимущества зависимых от значений типов

**Зависимые от значений типы** YaoXiang — это ключевое преимущество по сравнению с традиционными обобщениями:

| Преимущество | Описание |
|------|------|
| **Зависимость типов от значений** | `Vec: (n: Int) -> Type` позволяет типам зависеть от конкретных значений |
| **Компайл-тайм вычисление** | Вызовы функций в позициях типов вычисляются в компайл-тайм, результат непосредственно встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` проверяет размерности матрицы в компайл-тайм |
| **Вычисления на уровне типов** | `If`, `Match` и другие условные типы поддерживают вычисления на уровне типов |
| **Гарантия завершимости** | Спецификация decreases обеспечивает завершимость компайл-тайм вычислений |

```yaoxiang
# Компайл-тайм проверка, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Компайл-тайм вычисление: factorial(3) = 6, factorial(2) = 2
# Тип — Matrix(Float, 6, 2)

# Несоответствие размерностей обнаруживается в компайл-тайм
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщений

```yaoxiang
# Пример: унифицированный дизайн API
# Операция map для различных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Подход с обобщениями: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели проектирования

### Основные цели

1. **Абстракции с нулевой стоимостью** — вызов обобщённой функции эквивалентен вызову конкретной
2. **Удаление мёртвого кода** — компайл-тайм анализ, инстанциация только используемых обобщений
3. **Замена макросов** — обобщения покрывают 90% сценариев использования макросов
4. **Безопасность типов** — компайл-тайм проверка, без рантайм-затрат на типы
5. **IDE-дружественность** — интеллектуальные подсказки, понятные сообщения об ошибках
6. **Зависимые от значений типы** — типы могут зависеть от значений, поддержка компайл-тайм проверки размерностей
7. **Безопасность компайл-тайм вычислений** — гарантия завершимости через спецификацию decreases

### Принципы проектирования

- **Компайл-тайм детерминированность** — параметры обобщений определены в компайл-тайм
- **Приоритет мономорфизации** — генерация конкретного кода, избежание виртуальных вызовов
- **Управление через ограничения** — ограничения типов направляют инстанциацию
- **Платформенные оптимизации** — специализация для платформенно-специфичных оптимизаций
- **Единство вселенной типов** — функции/конструкторы типов/зависимые от значений типы унифицированы на уровне Type2
- **Гарантия завершимости** — вызовы функций в позициях типов должны доказывать завершимость

### 1. Базовые обобщения

#### 1.1 Параметры-типы в обобщениях

> **Ключевое правило**: определения обобщённых типов **должны явно аннотироваться `: Type`**, иначе они будут выведены HM как функции.
>
> | Запись | Значение |
> |------|------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM выводит как функцию, не тип |

```yaoxiang
# Определение обобщённого типа (должно иметь : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — лишь соглашение об именовании, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM выводит как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Ограничение обобщения (прямое выражение, однострочное может опускать return)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров-типов
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 2.4 Встроенные marker trait: Dup и Clone

**Три вида семантики копирования**:

| Тип | Значение | Способ срабатывания | Сценарий применения |
|------|------|----------|----------|
| **Копирование значения-примитива** | При присваивании автоматическое копирование значения, два значения полностью независимы | Автоматически при присваивании/передаче параметров | Int, Float, Bool, Char |
| **Dup** | Поверхностное копирование: копирование дескриптора/токена, базовые данные разделяются | Автоматически при присваивании/передаче параметров | `&T` токены, `ref T`, String/Bytes |
| **Clone** | Глубокое копирование: создание полной независимой копии | `value.clone()` | Любой тип, реализующий Clone |

**Семантика Dup**: для типов, реализующих Dup, при присваивании/передаче параметров владение не передаётся — компилятор копирует дескриптор/токен, несколько владельцев указывают на одни и те же базовые данные. Это дополняет семантику Move по умолчанию из модели владения RFC-009.

**Dup и Clone — ортогональные концепции**:

```
Dup = копирование дескриптора, разделение данных (изменения влияют друг на друга)
Clone = копирование данных, копии независимы (изменения не влияют друг на друга)
```

**Правила**:

```
1. Типы-примитивы значений (Int, Float, Bool, Char) — встроенное в компилятор копирование значений, не относится к Dup
2. Dup — применим только к ссылочным/токен-типам и типам с внутренним подсчётом ссылок
3. Clone — явное глубокое копирование, любой тип может его реализовать
4. Move по умолчанию — остальные типы сохраняют семантику Move по умолчанию
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|------|-----|------|
| `&T` (токен заимствования) | ✅ | Токен нулевого размера, копирование токена = несколько представлений к одним данным |
| `ref T` | ✅ | Копирование Rc/Arc = счётчик ссылок +1, разделение кучи |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, копирование дескриптора разделяет базовый буфер |
| `&mut T` (токен изменяемого заимствования) | ❌ | Линейная исключительность, нельзя копировать |
| struct | Производное | Все поля Dup → struct Dup |
| enum | Производное | Все поля всех вариантов Dup → enum Dup |
| tuple | Производное | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захватываемая среда может быть не Dup |
| `*T` (сырой указатель) | ❌ | unsafe, не участвует в системе владения |

**Int/Float/Bool/Char не являются Dup** — это типы значений, при присваивании компилятор автоматически копирует значение (два значения полностью независимы). Это не «поверхностное копирование», а встроенная обработка примитивов компилятором, не требующая и не должна выражаться через атрибут типа Dup.

```yaoxiang
# Типы-примитивы значений: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x          # Копирование значения, x и y полностью независимы
print(x)       # ✅

# Dup: поверхностное копирование, копирование дескриптора разделяет данные
view: &Point = &point
view2 = view    # ✅ Dup: копирование токена, оба указывают на один и тот же point
print(view.x)   # ✅

# Clone: явное глубокое копирование, создание независимой копии
backup = big_struct.clone()  # Явный вызов

# Ограничения обобщений
dup_use: (T: Dup) -> T = x         # T: Dup → можно поверхностно копировать
clone_use: (T: Clone) -> T = x.clone()  # T: Clone → можно глубоко копировать
```

> **Примечание**: `Send`/`Sync` не являются видимыми пользователю trait. Гарантии безопасности между задачами полностью автоматизированы через ключевое слово `ref` и компилятор — `ref` автоматически выбирает Rc или Arc, пользователю не нужно понимать Send/Sync.

### Never и Void: ⊥ и ⊤ в системе типов

Система типов YaoXiang в изоморфизме Карри-Ховарда одновременно обладает ⊥ (ложь/пустой тип) и ⊤ (истина/Unit), представленными двумя встроенными именами типов: `Never` и `Void`:

**Never (⊥)** — три не подлежащих обсуждению ядерных свойства:

1. **Нулевой набор конструкторов**: ни один литерал или выражение не может породить значение типа `Never`. Это мета-свойство, оно должно быть встроенным.
2. **Принцип взрыва**: `Never <: T` выполняется для любого типа `T`. Значение `Never` может использоваться как значение любого типа — именно поэтому код после `assert(false)` проходит проверку типов (хотя никогда не выполнится).
3. **Метка расходимости**: `f: (...) -> Never` означает, что `f` гарантированно не возвращается. Компилятор использует это для анализа мёртвого кода.

`Never` — встроенное имя типа, не ключевое слово, парсер ничего о нём не знает. Синтаксис пустого объединения и литералов типа не открывается.

**Void (⊤, т.е. Unit)** — имеет ровно одного обитателя (значение void по умолчанию), является носителем proposition «всегда истинно». `Void` — единица нуль-полевого произведения типов, `Never` — единица нуль-вариантного объединения типов — они двойственны. `x: Void = <по умолчанию>` корректно, `x: Never = ...` не имеет правой части для записи.

#### 4.3 Компайл-тайм верификация (реализация в стандартной библиотеке)

```yaoxiang
# ════════════════════════════════════════════════════════
# Реализация стандартной библиотеки: использование условных типов
# ════════════════════════════════════════════════════════

# Определения стандартной библиотеки
# IsTrue: мост из вселенной значений во вселенную типов — отображение истинности Bool в тип
IsTrue: (b: Bool) -> Type = match b {
    true => Void,      # ⊤, имеет значение, программа продолжается
    false => Never,    # ⊥, нет значения, расходимость
}

# Assert: примитив уточнения типа на уровне компиляции — типовый уровень proposition для Bool
Assert: (cond: Bool) -> Type = IsTrue(cond)
#
# cond = true  → Assert(true)  = Void    (всегда истинно, стирается)
# cond = false → Assert(false) = Never   (всегда ложно, ошибка компиляции/расходимость)
# cond неразрешим → конвейер доказательств решает по dispatch-паттерну:
#                   CompileTime → Unknown, требует prove
#                   Runtime     → вставка check, инъекция предположения Γ

# Использование 1: в качестве ограничения в определении типа
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    # Компайл-тайм проверка: N должно быть больше 0 (Assert в позиции типа)
    length: Assert(N > 0),
}

# Использование 2: в выражениях
IntArray: (N: Int) -> Type = StaticArray(Int, N)
# Проверка: размер IntArray(10) равен sizeof(Int) * 10
Assert(size_of(IntArray(10)) == sizeof(Int) * 10)
```

```yaoxiang
sum: (T: Type) -> ((arr: Array(T, 1)) -> T) = arr.data[0]

# Специализация для N=2
sum: (T: Type) -> ((arr: Array(T, 2)) -> T) = arr.data[0] + arr.data[1]

# Развёртка цикла для малых массивов (N <= 4)
sum: (T: Type, N: Int) -> ((arr: Array(T, N)) -> T) = {
    # Компиляторная оптимизация: развёртка цикла
    return arr.data[0] + arr.data[1] + arr.data[2] + arr.data[3]
}
```

### 5. Условные типы

> **Изоморфизм Карри-Ховарда**: условные типы с точки зрения Карри-Ховарда представляют логический **case-анализ**. Тип `Bool` соответствует proposition с двумя возможными значениями (True/False), `If` в зависимости от истинности proposition выбирает различные результаты — это и есть логический case-дизъюнкт. `match C { True => T, False => E }` фактически выражает: «зная, что proposition C истинно, заключаем T; зная, что ложно — E».

#### 5.1 Условный тип If

```yaoxiang
# Уровневый If
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: компайл-тайм ветвление
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Компайл-тайм верификация (унифицирована с определением Assert из §4.3)
# Assert: (cond: Bool) -> Type = IsTrue(cond)

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри-Ховарда**: семейства типов — наиболее прямое воплощение «propositions как типы». `Add: (A: Type, B: Type) -> Type` — это не «функция сложения, записанная на уровне типов», а **конструирование proposition о сложении натуральных чисел**. `(Zero, B) => B` говорит «proposition Add(Zero, B) эквивалентно B», `(Succ(A'), B) => Succ(Add(A', B))` говорит «если Add(A', B) истинно, то и Add(Succ(A'), B) истинно». Это и есть определение сложения из аксиом Пеано. То, что выражение match проходит проверку типов, эквивалентно проверке логической согласованности этого определения.

```yaoxiang
# Компайл-тайм преобразование типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # по умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Карри-Ховард: case-анализ + рекурсивный вызов, для полной индукции требуется проверка завершимости)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: компайл-тайм вычисление 2 + 3
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

### 7. Механизм удаления мёртвого кода

#### 7.1 Анализ графа инстанциации

```rust
// Внутри компилятора: построение графа зависимостей инстанциаций обобщений
struct InstantiationGraph {
    // Узлы: инстанциации обобщений
    nodes: HashMap<InstanceKey, InstanceNode>,

    // Рёбра: отношения использования
    edges: HashMap<InstanceKey, Vec<InstanceKey>>,
}

struct InstanceKey {
    generic: FunctionId,  // ID обобщённой функции
    type_args: Vec<TypeId>,  // параметры-типы
    const_args: Vec<ConstId>,  // параметры-Const
}

// Алгоритм: анализ достижимости
fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Непосещённые инстанциации — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}
```

#### 9.1 Полный пример обобщённого контейнера

```yaoxiang
# ======== 1. Определение обобщённого контейнера ========
# Используется синтаксис (T: Type) -> Type
Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,
}

Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self,
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,

    # Обобщённые методы (T автоматически вводится в область видимости внешним List(T))
    push: (self: List(T), item: T) -> Void,
    pop: (self: List(T)) -> Option(T),
    map: (R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)),
    filter: (self: List(T), predicate: (T) -> Bool) -> List(T),
    fold: (U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U),
}

# ======== 2. Реализация обобщённых методов ========
# Определения функций находятся в пространстве имён List (префикс List. = принадлежность к пространству имён)
# Чтобы . вызов вроде list.push(item) работал, требуется явная привязка: List.push = push[0]
# self — лишь соглашение об именовании параметра, компилятор смотрит на тип, а не на имя

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

List.pop: (T: Type) -> ((self: List(T)) -> Option(T)) = {
    if self.length > 0 {
        self.length = self.length - 1
        return Option.some(self.data[self.length])
    } else {
        return Option.none()
    }
}

List.map: (T: Type, R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)) = {
    result = List(R)()
    for i in 0..self.length {
        result.push(f(self.data[i]))
    }
    return result
}

List.filter: (T: Type) -> ((self: List(T), predicate: (T) -> Bool) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        if predicate(self.data[i]) {
            result.push(self.data[i])
        }
    }
    return result
}

List.fold: (T: Type, U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U) = {
    result = initial
    for i in 0..self.length {
        result = f(result, self.data[i])
    }
    return result
}

# ======== 3. Использование ограничений типов ========
# Реализация Clone для List
List.clone: (T: Clone) -> ((self: List(T)) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        result.push(self.data[i].clone())
    }
    return result
}
```

#### 9.2 Пример обобщённого алгоритма

```yaoxiang
# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённая быстрая сортировка
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Реализуется через перегрузку функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========
# Сортировка массива Int
numbers = Array(Int)(3, 1, 4, 1, 5, 9, 2, 6)
sorted = quicksort(numbers, Comparator(Int)())

# Сортировка массива String (требуется StringComparator)
strings = Array(String)("hello", "world", "foo", "bar")
sorted_strings = quicksort(strings, Comparator(String)())
```

#### 9.3 Пример компайл-тайм обобщений

```yaoxiang
# ======== 1. Тип матрицы с компайл-тайм размерностями ========
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),

    # Компайл-тайм проверка размерностей: используется тип Assert из стандартной библиотеки
    _assert: Assert[Rows > 0],  # Rows > 0, иначе ошибка компиляции
    _assert: Assert[Cols > 0],  # Cols > 0, иначе ошибка компиляции

    # Операции над матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {
            for j in 0..M {
                sum = Zero::zero()
                for k in 0..Cols {
                    sum = sum + self.data[i][k] * other.data[k][j]
                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

# ======== 2. Создание матрицы в компайл-тайм ========
identity: (T: Add + Multiply + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    for i in 0..N {
        for j in 0..N {
            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    return matrix
}

# ======== 3. Примеры использования ========
# Создание матрицы с размером, известным в компайл-тайм
# Матрица 2x3
matrix_2x3 = Matrix(Float, 2, 3)()
matrix_2x3.data[0][0] = 1.0
matrix_2x3.data[0][1] = 2.0
matrix_2x3.data[0][2] = 3.0
matrix_2x3.data[1][0] = 4.0
matrix_2x3.data[1][1] = 5.0
matrix_2x3.data[1][2] = 6.0

# Матрица 3x2
matrix_3x2 = Matrix(Float, 3, 2)()
matrix_3x2.data[0][0] = 7.0
matrix_3x2.data[0][1] = 8.0
matrix_3x2.data[1][0] = 9.0
matrix_3x2.data[1][1] = 10.0
matrix_3x2.data[2][0] = 11.0
matrix_3x2.data[2][1] = 12.0

# Умножение матриц: 2x3 * 3x2 = 2x2
result = matrix_2x3.multiply(matrix_3x2)

# Компайл-тайм проверка: тип result — Matrix(Float, 2, 2)
# Единичная матрица 3x3
identity_3x3 = identity(Float, 3)()

# Несоответствие размерностей: ошибка компиляции
# bad_multiply = matrix_2x3.multiply(identity_3x3)  # Ошибка компиляции: 3x3 != 2x3
```

## Компромиссы

### Преимущества

1. **Абстракции с нулевой стоимостью**
   - Компайл-тайм мономорфизация, без накладных расходов в рантайме
   - Нет необходимости в виртуальных функциях, нет RTTI

2. **Удаление мёртвого кода**
   - Компайл-тайм анализ, инстанциация только используемых обобщений
   - Раздувание кода контролируемо

3. **Замена макросов**
   - Типобезопасная кодогенерация
   - IDE-дружественность, понятные сообщения об ошибках

4. **Компайл-тайм вычисления**
   - Компайл-тайм обобщения поддерживают компайл-тайм вычисления
   - Проверка размерностей и другие возможности
   - Нет необходимости в ключевом слове `const`, чистые ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанциация обобщений увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Потребление памяти**
   - Увеличивается потребление памяти компилятором
   - Механизмы кеширования требуют памяти

3. **Сложность реализации**
   - Решатель ограничений сложен
   - Движок вычислений на уровне типов сложен

4. **Диагностика ошибок**
   - Ошибки обобщений могут быть сложными
   - Требуются понятные сообщения об ошибках

### Меры по смягчению

1. **Стратегия кеширования**
   - Кеширование результатов инстанциации
   - LRU-кеш для ограничения памяти

2. **Инкрементальная компиляция**
   - Кеширование результатов компиляции
   - Инкрементальная инстанциация

3. **Сообщения об ошибках**
   - Понятные сообщения об ошибках
   - Подсказки по выводу параметров обобщений

4. **Параллельная компиляция**
   - Параллельная инстанциация обобщений
   - Многопоточное решение ограничений

## Альтернативы

| Альтернатива | Почему не выбрана |
|------|--------------|
| Только базовые обобщения | Не могут заменить сложные макросы |
| Чисто макросистема | Нет безопасности типов, плохие сообщения об ошибках |
| Только ограничения | Недостаточная гибкость |
| Рантайм-обобщения | Накладные расходы на производительность |

### Риски

| Риск | Влияние | Меры по смягчению |
|------|------|------|
| Сложность решения ограничений | Чрезмерное время компиляции | Инкрементальное решение + кеш |
| Раздувание кода | Чрезмерный размер бинарника | DCE + контроль порогов |
| Сложность реализации | Удлинение цикла разработки | Поэтапная реализация |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

### БНФ синтаксиса

```bnf
# Параметры обобщений используют единый синтаксис () как часть типа функции
# Например, map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в параметрах обобщений)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры обобщений — первая группа параметров типа функции: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (единый синтаксис Binding)
# Обобщённый тип вроде List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Параметры Type в обобщениях компилятор автоматически заполняет из типов фактических аргументов
# Например, map(numbers, f): T извлекается из numbers: List(Int), R — из f: (Int) -> String
```
