# RFC-011: Система обобщений — абстракции с нулевой стоимостью и замена макросов

## Резюме

Данный документ определяет **дизайн системы обобщений** языка YaoXiang, реализующий абстракции с нулевой стоимостью посредством мощных возможностей обобщений, использующий оптимизации compile-time для снижения зависимости от макросов и предоставляющий механизм устранения мёртвого кода.

**Основные принципы дизайна**:
- **Унифицированный синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — параметры обобщений и обычные параметры унифицированы
- **Механизм самодокументирования Type**: `Type` — особая сущность языкового уровня; позиции `Type` в сигнатуре автоматически заполняются путём вывода
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения на функции
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Обобщения compile-time**: `N: Int` — параметры-значения обобщений, инстанцирование константами compile-time
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, type family

**Ценность**:
- Абстракции с нулевой стоимостью: мономорфизация compile-time, без затрат runtime
- Устранение мёртвого кода: анализ графа инстанцирования + оптимизации LLVM
- Замена макросов: обобщения заменяют 90% сценариев использования макросов
- Безопасность типов: проверка compile-time, удобство для IDE
- **Явное лучше неявного**: `Type` самодокументируется, компилятор выполняет автоматический вывод

## Идея вселенной типов и типы, зависящие от значений

Система обобщений YaoXiang построена на **идее вселенной типов** — ментальной модели, унифицирующей все концепции языка в многоуровневую структуру, ключевой инновацией которой является возведение **типов, зависящих от значений**, в полноправные сущности первого класса на уровне Type2.

### Что такое тип, зависящий от значений?

**Тип, зависящий от значений** — это тип, который зависит от одного или нескольких **значений** (а не только от других типов). Эти значения могут вычисляться compile-time, тем самым предоставляя гарантии безопасности типов уже на этапе компиляции.

```yaoxiang
# Традиционные обобщения: параметры-типы
List: (T: Type) -> Type

# Типы, зависящие от значений: параметры-значения
Vec: (n: Int) -> Type  # тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # тип матрицы зависит от количества строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционными обобщениями, типы, зависящие от значений, в YaoXiang обладают следующими ключевыми преимуществами:

| Характеристика | Традиционные обобщения (C++/Rust) | Типы, зависящие от значений, в YaoXiang |
|------|-------------------|---------------------|
| Значения, от которых зависит тип | Только параметры-типы | Может зависеть от любых значений, включая результаты вызова функций |
| Вычисление compile-time | Ручная специализация шаблонов C++, отсутствует в Rust | Автоматическое вычисление compile-time с гарантией завершаемости |
| Вычисления на уровне типов | Метапрограммирование шаблонов (сложное/опасное) | Унифицированный движок вычислений на уровне типов |
| Безопасность типов | Отсутствует в C++, ограничена в Rust | Полная безопасность типов, проверка compile-time |
| Проверка размерностей | Проверка runtime или ручная специализация | Проверка размерностей compile-time, без затрат runtime |

### Уровни вселенной типов и типы, зависящие от значений

Идея вселенной типов разделяет концепции языка по семантическим ролям на различные уровни; типы, зависящие от значений, расположены на **уровне Type2**:

| Уровень | Роль | Примеры |
|------|------|------|
| Type-1 | Значения | `42`, `factorial(5)`, сама функция |
| Type0 | Ключевые слова метаязыка типов | `Type` |
| Type1 | Конкретные типы | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функции/конструкторы типов/типы, зависящие от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевой дизайн**: функции, конструкторы типов и типы, зависящие от значений, на уровне Type2 имеют **унифицированный синтаксис** — все имеют форму `(params) -> result`:
- Обычная функция: `(Int, Int) -> Int` → возвращает значение
- Конструктор типа: `(T: Type) -> Type` → возвращает тип
- Тип, зависящий от значений: `(n: Int) -> Type` → возвращает тип, но зависит от параметра-значения

> **Изоморфизм Карри-Ховарда**: это единство — не совпадение. Изоморфизм Карри-Ховарда утверждает «типы суть propositions, программы суть доказательства» — функциональный тип `A → B` соответствует логической импликации «если A, то B», обобщение `(T: Type) -> Type` соответствует квантору всеобщности «для всех типов T», тип, зависящий от значений, `(n: Int) -> Type` соответствует «для каждого целого n существует тип». YaoXiang унифицирует функции, конструкторы типов и типы, зависящие от значений, на уровне Type2, по сути объединяя «доказательство» и «вычисление» в одну концепцию — **конструктивное доказательство**. Это прямое воплощение изоморфизма Карри-Ховарда в дизайне языка: одна форма (`(params) -> result`) одновременно несёт логические propositions и вычислительный процесс.

### Гарантия детерминированности compile-time

Идея вселенной типов в YaoXiang требует: **всё на уровне Type должно быть определено compile-time**.

```yaoxiang
# Пример проверки размерностей compile-time
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка compile-time: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется compile-time
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Вычисление compile-time: factorial(3) = 6, размер вектора определяется compile-time
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов
2. Проверяет, помечена ли функция спецификацией `decreases` (см. механизм проверки завершаемости ниже)
3. Выполняет вычисление compile-time
4. Встраивает результат в генерируемый тип

### Сценарии применения типов, зависящих от значений

#### Проверка размерностей compile-time
```yaoxiang
# Умножение матриц: проверка соответствия размерностей compile-time
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка compile-time: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка обнаруживается compile-time:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Размер массива с безопасностью типов
```yaoxiang
# Размер массива — константа compile-time
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — константа compile-time, может использоваться в вычислениях на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно compile-time)
```

#### Условные типы
```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Type family
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры-типы T, R определяются compile-time
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    return result
}

# Использование полностью прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # Выводится как map[Int, Int]
```

### Механизм проверки завершаемости (интеграция с RFC-022)

Вычисление compile-time для типов, зависящих от значений, должно **гарантировать завершаемость**, иначе система типов попадёт в бесконечный цикл. YaoXiang обеспечивает это посредством **спецификации decreases**, бесшовно интегрируясь с RFC-022.

#### Спецификация завершаемости рекурсивных функций
```yaoxiang
# Факториал compile-time: необходимо доказать завершаемость
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # На каждом рекурсивном вызове n строго убывает
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # Вычисление compile-time factorial(5) = 120
```

#### Спецификация завершаемости циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершаемости

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза проверки типов                                        │
│  Встречается вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка спецификации decreases                          │
│     - Есть decreases: проверяется выполнение условия убывания │
│       на всех путях рекурсии                                  │
│     - Нет decreases, но очевидно завершается: прямое вычисление │
│     - Нет decreases и может не завершиться: ошибка компиляции │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисление compile-time (встроенным интерпретатором)    │
│     - Чистая функция: прямое вычисление                       │
│     - Побочные эффекты: ошибка компиляции (позиция типа      │
│       не должна иметь побочных эффектов)                     │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                            │
│     - Vec(factorial(5)) → Vec(120)                          │
│     - Matrix(Float, 3, 3) → конкретный тип                   │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантия завершаемости вычислений compile-time, защита системы типов от бесконечных циклов
- **Унификация**: проверка завершаемости разделяет единый механизм спецификаций с верификацией частичной корректности
- **Постепенное усиление**: возможность перехода от проверок runtime к полному статическому доказательству

## Мотивация

### Почему необходима мощная система обобщений?

Существующие mainstream языки имеют ограничения в обобщениях:

| Язык | Возможности обобщений | Проблемы |
|------|----------|------|
| Java | Ограниченные типы | Мономорфизация compile-time, без специализации обобщений |
| C# | Ограничения обобщений | Проверка типов runtime, затраты производительности |
| Rust | Обобщения + Trait | Сложная система Trait, крутая кривая обучения |
| C++ | Шаблоны | Сложная специализация шаблонов, плохие сообщения об ошибках компиляции |
| **YaoXiang** | **Типы, зависящие от значений** | **Типы могут зависеть от значений, проверка размерностей compile-time, гарантия завершаемости** |

### Ключевые противоречия

1. **Производительность vs гибкость**: гибкость runtime vs оптимизации compile-time
2. **Сложность vs простота**: мощная система типов vs удобство использования
3. **Макросы vs обобщения**: генерация кода макросами vs безопасность типов обобщений
4. **Зависимость от значений vs безопасность типов**: традиционные обобщения не могут проверять размерности compile-time

### Ключевые преимущества типов, зависящих от значений

**Типы, зависящие от значений**, в YaoXiang — ключевое преимущество по сравнению с традиционными обобщениями:

| Преимущество | Описание |
|------|------|
| **Тип зависит от значения** | `Vec: (n: Int) -> Type` позволяет типу зависеть от конкретного значения |
| **Вычисление compile-time** | Вызов функции в позиции типа вычисляется compile-time, результат встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` проверяет размерности матрицы compile-time |
| **Вычисления на уровне типов** | Условные типы `If`, `Match` поддерживают вычисления на уровне типов |
| **Гарантия завершаемости** | Спецификация decreases гарантирует завершаемость вычислений compile-time |

```yaoxiang
# Проверка compile-time, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Вычисление compile-time: factorial(3) = 6, factorial(2) = 2
# Тип: Matrix(Float, 6, 2)

# Несоответствие размерностей обнаруживается compile-time
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщений

```yaoxiang
# Пример: унифицированный дизайн API
# Операция map для различных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Подход с обобщениями: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели дизайна

### Ключевые цели

1. **Абстракции с нулевой стоимостью** — вызов обобщений эквивалентен вызову конкретного типа
2. **Устранение мёртвого кода** — анализ compile-time, инстанцирование только используемых обобщений
3. **Замена макросов** — обобщения заменяют 90% сценариев использования макросов
4. **Безопасность типов** — проверка compile-time, без затрат типов runtime
5. **Удобство для IDE** — интеллектуальные подсказки, понятные сообщения об ошибках
6. **Типы, зависящие от значений** — типы могут зависеть от значений, поддержка проверки размерностей compile-time
7. **Безопасность вычислений compile-time** — гарантия завершаемости через спецификацию decreases

### Принципы дизайна

- **Детерминированность compile-time** — параметры обобщений определяются compile-time
- **Приоритет мономорфизации** — генерация конкретного кода, избегание вызовов виртуальных функций
- **Управление через ограничения** — ограничения типов направляют инстанцирование
- **Платформенная оптимизация** — специализация для платформенно-специфичных оптимизаций
- **Унификация вселенной типов** — функции/конструкторы типов/типы, зависящие от значений, унифицированы на уровне Type2
- **Гарантия завершаемости** — вызовы функций в позициях типов должны доказывать завершаемость

## Предложение

### 1. Базовые обобщения

#### 1.1 Параметры типов обобщений

> **Ключевое правило**: определение обобщённого типа **должно явно аннотироваться `: Type`**, иначе будет выведено HM как функция.
>
> | Запись | Значение |
> |------|------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM выводит как функцию, не тип |

```yaoxiang
# Определение обобщённого типа (должно иметь : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — лишь соглашение об именовании, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM выводит как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Ограничение обобщения (прямое выражение, в одной строке можно опустить return)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров-типов
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 2.4 Встроенные marker trait: Dup и Clone

**Три типа семантики копирования**:

| Тип | Значение | Способ запуска | Сценарий применения |
|------|------|----------|----------|
| **Копирование примитивного значения** | При присваивании автоматическое копирование значения, два значения полностью независимы | Автоматически при присваивании/передаче параметров | Int, Float, Bool, Char |
| **Dup** | Поверхностное копирование: копирование дескриптора/токена, совместное использование базовых данных | Автоматически при присваивании/передаче параметров | Токены `&T`, `ref T`, String/Bytes |
| **Clone** | Глубокое копирование: создание полной независимой копии | `value.clone()` | Любой тип, реализующий Clone |

**Семантика Dup**: для типа, реализующего Dup, при присваивании/передаче параметров владение не передаётся — компилятор копирует дескриптор/токен, и несколько владельцев указывают на одни и те же базовые данные. Это дополняет семантику Move по умолчанию из модели владения RFC-009.

**Dup и Clone — ортогональные концепции**:

```
Dup = копирование дескриптора, совместное использование данных (изменения влияют друг на друга)
Clone = копирование данных, копия независима (изменения не влияют друг на друга)
```

**Правила**:

```
1. Примитивные типы значений (Int, Float, Bool, Char) — встроенное в компилятор копирование значений, не относятся к Dup
2. Dup  — применимо только к типам ссылок/токенов и типам со встроенным подсчётом ссылок
3. Clone — явное глубокое копирование, может быть реализовано любым типом
4. Move по умолчанию — остальные типы сохраняют семантику Move по умолчанию
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|------|-----|------|
| `&T` (токен заимствования) | ✅ | Токен нулевого размера, копирование токена = несколько представлений указывают на одни данные |
| `ref T` | ✅ | Копирование Rc/Arc = счётчик ссылок+1, совместное использование данных в куче |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, копирование дескриптора разделяет базовый буфер |
| `&mut T` (токен изменяемого заимствования) | ❌ | Линейная эксклюзивность, копирование невозможно |
| struct | Производный | Все поля Dup → struct Dup |
| enum | Производный | Все поля всех variant Dup → enum Dup |
| tuple | Производный | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захватываемая среда может быть не Dup |
| `*T` (сырой указатель) | ❌ | unsafe, не участвует в системе владения |

**Int/Float/Bool/Char не являются Dup** — это типы значений, при присваивании компилятор автоматически копирует значение (два значения полностью независимы). Это не «поверхностное копирование», а встроенная обработка компилятором примитивов, которая не нуждается и не должна выражаться через атрибут типа Dup.

```yaoxiang
# Примитивный тип значения: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x          # Копирование значения, x и y полностью независимы
print(x)       # ✅

# Dup: поверхностное копирование, копирование дескриптора разделяет данные
view: &Point = &point
view2 = view    # ✅ Dup: копирование токена, оба указывают на один и тот же point
print(view.x)   # ✅

# Clone: явное глубокое копирование, создание независимой копии
backup = big_struct.clone()  # Явный вызов

# Ограничения обобщений
dup_use: (T: Dup) -> T = x         # T: Dup → можно поверхностно копировать
clone_use: (T: Clone) -> T = x.clone()  # T: Clone → можно глубоко копировать
```

> **Примечание**: `Send`/`Sync` не являются видимыми пользователю trait. Гарантии безопасности межзадачного взаимодействия полностью обрабатываются ключевым словом `ref` и компилятором автоматически — `ref` автоматически выбирает Rc или Arc, и пользователю не нужно понимать Send/Sync.

#### 4.1 Параметры-константы compile-time

}

# Сложение compile-time
add: (a: Int, b: Int) -> (a: a, b: b) -> Int = a + b

### Never и Void: ⊥ и ⊤ в системе типов

Система типов YaoXiang в рамках изоморфизма Карри-Ховарда одновременно обладает ⊥ (ложь/пустой тип) и ⊤ (истина/Unit), представленными двумя встроенными именами типов: `Never` и `Void`.

**Never (⊥)** — три непересмотренных ядерных свойства:

1. **Нулевой набор конструкторов**: ни один литерал или выражение не может породить значение типа `Never`. Это метауровневое свойство, которое должно быть встроено.
2. **Принцип взрыва**: `Never <: T` выполняется для любого типа `T`. Значение `Never` может использоваться как значение любого типа — именно поэтому код после `assert(false)` проходит проверку типов (хотя никогда не достигается).
3. **Маркер расходимости**: `f: (...) -> Never` означает, что `f` гарантированно не возвращается. Компилятор использует это для анализа мёртвого кода.

`Never` — встроенное имя типа, а не ключевое слово, парсер не имеет о нём представления. Синтаксис пустого конструктора и литерала типа не открывается.

**Void (⊤, т.е. Unit)** — имеет ровно одного обитателя (значение void по умолчанию), является носителем истинной propositions «всегда истинно». `Void` — единица нульпольного произведения типов, `Never` — единица нульвариантного суммирования типов — они двойственны. `x: Void = <по умолчанию>` допустимо, `x: Never = ...` не имеет правой части для записи.

#### 4.3 Проверка compile-time (реализация стандартной библиотеки)

```yaoxiang
# ════════════════════════════════════════════════════════
# Реализация стандартной библиотеки: использование условных типов
# ════════════════════════════════════════════════════════

# Определения стандартной библиотеки
# IsTrue: мост от вселенной значений к вселенной типов — Bool-истина отображается в тип
IsTrue: (b: Bool) -> Type = match b {
    true => Void,      # ⊤, имеет значение, программа продолжается
    false => Never,    # ⊥, нет значения, расходимость
}

# Assert: примитив уточнения типа на compile-time — типовое выражение propositions для Bool
Assert: (cond: Bool) -> Type = IsTrue(cond)
#
# cond равно true  → Assert(true)  = Void    (всегда истинно, стирается)
# cond равно false → Assert(false) = Never   (всегда ложно, ошибка компиляции/расходимость)
# cond неразрешимо → по конвейеру доказательств в режиме dispatch:
#                  CompileTime → Unknown, требуется prove
#                  Runtime     → вставка check, инъекция предположения Γ

# Способ использования 1: как ограничение в определении типа
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    # Проверка compile-time: N должно быть больше 0 (Assert в позиции типа)
    length: Assert(N > 0),
}

# Способ использования 2: использование в выражении
IntArray: (N: Int) -> Type = StaticArray(Int, N)
# Проверка: размер IntArray(10) равен sizeof(Int) * 10
Assert(size_of(IntArray(10)) == sizeof(Int) * 10)
```

### 5. Условные типы

> **Изоморфизм Карри-Ховарда**: условные типы с точки зрения Карри-Ховарда представляют **case-анализ** в логике. Тип `Bool` соответствует propositions с двумя возможными значениями (True/False), `If` выбирает разные результаты в зависимости от истинности этой propositions — это и есть логический case-дизъюнкт. `match C { True => T, False => E }` фактически выражает: «зная, что propositions C истинно, заключение — T; зная, что C ложно, заключение — E».

#### 5.1 Условный тип If

```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление compile-time
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Проверка compile-time (унифицировано с определением Assert из §4.3)
# Assert: (cond: Bool) -> Type = IsTrue(cond)

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Type family

> **Изоморфизм Карри-Ховарда**: type family — наиболее прямое воплощение «propositions как типы». `Add: (A: Type, B: Type) -> Type` — это не «функция сложения, написанная на уровне типов», а **конструирование propositions о сложении натуральных чисел**. `(Zero, B) => B` говорит «propositions Add(Zero, B) эквивалентна B», `(Succ(A'), B) => Succ(Add(A', B))` говорит «если Add(A', B) истинно, то Add(Succ(A'), B) также истинно». Это и есть определение сложения в аксиомах Пеано. То, что проверка типов подтверждает прохождение этого match-выражения, эквивалентно проверке логической согласованности этого определения.

```yaoxiang
# Преобразование типа compile-time
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # По умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Карри-Ховард: case analysis + рекурсивный вызов, требует проверки завершаемости для полной индукции)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: вычисление compile-time 2 + 3
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

#### 7.1 Анализ графа инстанцирования

```rust
// Внутреннее устройство компилятора: построение графа зависимостей инстанцирования обобщений
struct InstantiationGraph {
    // Узлы: инстанцирования обобщений
    nodes: HashMap<InstanceKey, InstanceNode>,

    // Рёбра: отношения использования
    edges: HashMap<InstanceKey, Vec<InstanceKey>>,
}

struct InstanceKey {
    generic: FunctionId,  // ID обобщённой функции
    type_args: Vec<TypeId>,  // Аргументы-типы
    const_args: Vec<ConstId>,  // Аргументы Const
}

// Алгоритм: анализ достижимости
fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Непосещённые инстанцирования — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}
```

#### 9.1 Полный пример обобщённого контейнера

```yaoxiang
# ======== 1. Определение обобщённого контейнера ========
# Использование синтаксиса (T: Type) -> Type
Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,
}

    some: (T) -> Self,
    none: () -> Self,
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,

    # Обобщённые методы (T автоматически вводится в область видимости из внешнего List(T))
    push: (self: List(T), item: T) -> Void,
    pop: (self: List(T)) -> Option(T),
    map: (R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)),
    filter: (self: List(T), predicate: (T) -> Bool) -> List(T),
    fold: (U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U),
}

# ======== 2. Реализация обобщённых методов ========
# Определение функции в пространстве имён List (префикс List. = принадлежность пространству имён)
# Для работы синтаксиса вызова через . требуется явная привязка: List.push = push[0]
# self — лишь соглашение об именовании параметра, компилятор смотрит на тип, а не на имя

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

List.pop: (T: Type) -> ((self: List(T)) -> Option(T)) = {
    if self.length > 0 {
        self.length = self.length - 1
        return Option.some(self.data[self.length])
    } else {
        return Option.none()
    }
}

List.map: (T: Type, R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)) = {
    result = List(R)()
    for i in 0..self.length {
        result.push(f(self.data[i]))
    }
    return result
}

List.filter: (T: Type) -> ((self: List(T), predicate: (T) -> Bool) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        if predicate(self.data[i]) {
            result.push(self.data[i])
        }
    }
    return result
}

List.fold: (T: Type, U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U) = {
    result = initial
    for i in 0..self.length {
        result = f(result, self.data[i])
    }
    return result
}

# ======== 3. Использование ограничений типов ========
# Реализация Clone для List
List.clone: (T: Clone) -> ((self: List(T)) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        result.push(self.data[i].clone())
    }
    return result
}

```yaoxiang
# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённая быстрая сортировка
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Реализация с использованием перегрузки функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========
# Сортировка массива Int
numbers = Array(Int)(3, 1, 4, 1, 5, 9, 2, 6)
sorted = quicksort(numbers, Comparator(Int)())

# Сортировка массива String (требуется StringComparator)

    _assert: Assert[Cols > 0],  # Cols > 0, иначе ошибка компиляции

    # Операции с матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {
            for j in 0..M {
                sum = Zero::zero()
                for k in 0..Cols {
                    sum = sum + self.data[i][k] * other.data[k][j]
                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

# ======== 2. Создание матрицы compile-time ========
identity: (T: Add + Multiply + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    for i in 0..N {
        for j in 0..N {
            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    return matrix
}

## Компромиссы

### Преимущества

1. **Абстракции с нулевой стоимостью**
   - Мономорфизация compile-time, без затрат runtime
   - Без виртуальных функций, без RTTI

2. **Устранение мёртвого кода**
   - Анализ compile-time, инстанцирование только используемых обобщений
   - Контролируемое раздувание кода

3. **Замена макросов**
   - Генерация кода с безопасностью типов
   - Удобство для IDE, понятные сообщения об ошибках

4. **Вычисления compile-time**
   - Обобщения compile-time поддерживают вычисления compile-time
   - Возможности вроде проверки размерностей
   - Ключевое слово `const` не требуется, чистые ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанцирование обобщений увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Потребление памяти**
   - Увеличение потребления памяти компилятором
   - Механизмы кэширования требуют памяти

3. **Сложность реализации**
   - Сложный решатель ограничений
   - Сложный движок вычислений на уровне типов

4. **Диагностика ошибок**
   - Ошибки обобщений могут быть сложными
   - Необходимы понятные подсказки об ошибках

### Меры по смягчению

1. **Стратегии кэширования**
   - Кэширование результатов инстанцирования
   - LRU-кэш для ограничения памяти

2. **Инкрементальная компиляция**
   - Кэширование результатов компиляции
   - Инкрементальное инстанцирование

3. **Подсказки об ошибках**
   - Понятные сообщения об ошибках
   - Подсказки для вывода параметров обобщений

4. **Параллельная компиляция**
   - Параллельное инстанцирование обобщений
   - Многопоточное решение ограничений

## Альтернативы

| Альтернатива | Почему не выбрана |
|------|--------------|
| Только базовые обобщения | Невозможно заменить сложные макросы |
| Чистая система макросов | Нет безопасности типов, плохие сообщения об ошибках |
| Только ограничения | Недостаточная гибкость |
| Обобщения runtime | Затраты производительности |

### Риски

| Риск | Влияние | Мера по смягчению |
|------|------|----------|
| Сложность решения ограничений | Чрезмерное время компиляции | Инкрементальное решение + кэширование |
| Раздувание кода | Чрезмерный размер бинарного файла | DCE + контроль пороговых значений |
| Сложность реализации | Удлинение цикла разработки | Поэтапная реализация |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

## Приложение

### Синтаксис BNF

```bnf
# Параметры обобщений используют унифицированный синтаксис () как часть функционального типа
# Например, map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в параметрах обобщений)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры обобщений — первая группа параметров в функциональном типе: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённый тип, например, List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Type в параметрах обобщений автоматически заполняется компилятором из типов фактических аргументов
# Например, в map(numbers, f) T извлекается из numbers: List(Int), R — из f: (Int) -> String
```
