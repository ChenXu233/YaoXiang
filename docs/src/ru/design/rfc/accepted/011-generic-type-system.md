# RFC-011: Дизайн системы обобщённого программирования — Абстракции с нулевой стоимостью и замена макросов

## Резюме

В данном документе определён **дизайн системы обобщённого программирования** языка YaoXiang, реализующий абстракции с нулевой стоимостью посредством мощных возможностей обобщённого программирования, сокращающий зависимость от макросов через compile-time оптимизации и предоставляющий механизм устранения мёртвого кода.

**Ключевые аспекты дизайна**:
- **Унифицированный синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — обобщённые параметры и обычные параметры унифицированы
- **Механизм самоописания Type**: `Type` является языковой конструкцией особого рода, позиции `Type` в сигнатурах автоматически выводятся и заполняются
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения на типы функций
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Обобщения на этапе компиляции**: `N: Int` — параметры-значения обобщённых функций, константная инстанциация на этапе компиляции
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Абстракции с нулевой стоимостью: мономорфизация на этапе компиляции, отсутствие накладных расходов во время выполнения
- Устранение мёртвого кода: анализ графа инстанциации + оптимизация LLVM
- Замена макросов: обобщённые типы заменяют 90% сценариев использования макросов
- Типобезопасность: проверка на этапе компиляции, дружелюбность к IDE
- **Явное лучше неявного**: самоописание `Type`, автоматический вывод типов компилятором

## Концепция типовой вселенной и типозависимые типы

Система обобщённого программирования YaoXiang основана на **концепции типовой вселенной** — ментальной модели, объединяющей все концепции языка в иерархическую структуру, где ключевая инновация заключается в продвижении **типозависимых типов** в первоклассные гражданства уровня Type2.

### Что такое типозависимые типы?

**Типозависимый тип** — это тип, который зависит от одного или нескольких **значений** (а не только от других типов). Эти значения могут вычисляться на этапе компиляции, что обеспечивает гарантии типобезопасности уже на этапе компиляции.

```yaoxiang
# Традиционные обобщения: параметр типа
List: (T: Type) -> Type

# Типозависимые типы: параметр-значение
Vec: (n: Int) -> Type  # Тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # Тип матрицы зависит от числа строк и столбцов
```

### Ключевые преимущества типозависимых типов

По сравнению с традиционным обобщённым программированием, типозависимые типы YaoXiang обладают следующими ключевыми преимуществами:

| Характеристика | Традиционные обобщения (C++/Rust) | Типозависимые типы YaoXiang |
|------|-------------------|---------------------|
| Зависимость типа от значения | Только от параметров типа | От любого значения, включая результаты вызовов функций |
| Вычисление на этапе компиляции | Ручная специализация шаблонов C++, отсутствует в Rust | Автоматическое вычисление на этапе компиляции, гарантия завершения |
| Вычисления на уровне типов | Метапрограммирование шаблонов (сложно/опасно) | Унифицированный механизм вычислений на уровне типов |
| Типобезопасность | C++ отсутствует, Rust ограничена | Полная типобезопасность, проверка на этапе компиляции |
| Проверка размерностей | Проверка во время выполнения или ручная специализация | Compile-time проверка размерностей, без накладных расходов выполнения |

### Уровни типовой вселенной и типозависимые типы

Концепция типовой вселенной разделяет концепции языка по семантическим ролям на различные уровни, где типозависимые типы расположены на **уровне Type2**:

| Уровень | Роль | Пример |
|------|------|------|
| Type-1 | Значение | `42`, `factorial(5)`, сама функция |
| Type0 | Ключевое слово меатипа | `Type` |
| Type1 | Конкретный тип | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функция/конструктор типа/типозависимые типы** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевой аспект дизайна**: функции, конструкторы типов и типозависимые типы уровня Type2 **унифицированы синтаксически**, все используют форму `(params) -> result`:
- Обычная функция: `(Int, Int) -> Int` → возвращаемое значение является значением
- Конструктор типа: `(T: Type) -> Type` → возвращаемое значение является типом
- Типозависимый тип: `(n: Int) -> Type` → возвращаемое значение является типом, но зависит от параметра-значения

> **Изоморфизм Карри-Ховарда**: Это единство не случайно. Изоморфизм Карри-Ховарда устанавливает, что «типы — это утверждения, программы — это доказательства» — функциональный тип `A → B` соответствует логическому импликации «если A, то B», обобщённый тип `(T: Type) -> Type` соответствует универсальной квантификации «для всех типов T», типозависимый тип `(n: Int) -> Type` соответствует «для каждого целого n существует тип». Унификация функций, конструкторов типов и типозависимых типов в YaoXiang на уровне Type2, по сути, объединяет «доказательства» и «вычисления» в одну концепцию — **конструктивное доказательство**. Это прямое проявление изоморфизма Карри-Ховарда в дизайне языка: одна форма (`(params) -> result`) одновременно выражает логическое утверждение и вычислительный процесс.

### Гарантия compile-time детерминированности

Концепция типовой вселенной YaoXiang требует: **всё на уровне Type должно быть детерминировано на этапе компиляции**.

```yaoxiang
# Пример проверки размерностей на этапе компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Compile-time проверка: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется на этапе компиляции
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Compile-time вычисление: factorial(3) = 6, размер вектора определяется на этапе компиляции
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов
2. Проверяет, помечены ли функции атрибутом уменьшения (decreases) (см. механизм проверки завершения ниже)
3. Выполняет вычисления на этапе компиляции
4. Встраивает результат в генерируемый тип

### Сценарии применения типозависимых типов

#### Проверка размерностей на этапе компиляции
```yaoxiang
# Умножение матриц: compile-time проверка соответствия размерностей
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Compile-time проверка: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка обнаруживается на этапе компиляции:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива является compile-time константой
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N является compile-time константой, может использоваться для вычислений на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно на этапе компиляции)
```

#### Условные типы
```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейство типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры типа T, R определяются на этапе компиляции
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    return result
}

# Использование полностью прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # Выводится как map[Int, Int]
```

### Механизм проверки завершения (интеграция с RFC-022)

Вычисления на уровне типов должны **гарантированно завершаться**, иначе система типов войдёт в бесконечный цикл. YaoXiang обеспечивает это через **атрибут уменьшения (decreases)**, что полностью интегрируется с RFC-022.

#### Атрибут завершения для рекурсивных функций
```yaoxiang
# Compile-time факториал: необходимо доказать завершение
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # При каждой рекурсии n строго уменьшается
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # На этапе компиляции вычисляется factorial(5) = 120
```

#### Атрибут завершения для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершения

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза проверки типов                                        │
│  Встречен вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка атрибута decreases                              │
│     - Есть decreases: проверка условия уменьшения на всех путях рекурсии │
│     - Нет decreases, но очевидно завершается: прямоое вычисление │
│     - Нет decreases и возможно не завершается: ошибка компиляции │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисление на этапе компиляции (выполняется встроенным интерпретатором) │
│     - Чистая функция: прямое вычисление │
│     - Побочные эффекты: ошибка компиляции (позиция типа должна быть чистой) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                              │
│     - Vec(factorial(5)) → Vec(120)                          │
│     - Matrix(Float, 3, 3) → конкретный тип                   │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантирует, что вычисления на этапе компиляции обязательно завершатся, предотвращая бесконечные циклы в системе типов
- **Единство**: проверка завершения и верификация частичной корректности используют общий механизм атрибутов
- **Постепенное улучшение**: можно постепенно переходить от проверок во время выполнения к полностью статическому доказательству

## Мотивация

### Почему необходима мощная система обобщённого программирования?

Современные языки имеют ограничения в обобщённом программировании:

| Язык | Возможности обобщений | Проблема |
|------|----------|------|
| Java | Ограниченные типы | Compile-time мономорфизация, без специализации обобщений |
| C# | Ограничения обобщений | Проверка типов во время выполнения, накладные расходы производительности |
| Rust | Обобщения + Trait | Система Trait сложна, крутая кривая обучения |
| C++ | Шаблоны | Специализация шаблонов сложна, плохие сообщения об ошибках |
| **YaoXiang** | **Типозависимые типы** | **Типы могут зависеть от значений, compile-time проверка размерностей, гарантия завершения** |

### Основные противоречия

1. **Производительность vs Гибкость**: гибкость во время выполнения vs оптимизация на этапе компиляции
2. **Сложность vs Простота**: мощная система типов vs удобство использования
3. **Макросы vs Обобщения**: генерация кода макросами vs типобезопасность обобщений
4. **Зависимость от значений vs Типобезопасность**: традиционные обобщения не могут проверять размерности на этапе компиляции

### Ключевые преимущества типозависимых типов

**Типозависимые типы** YaoXiang — это основное преимущество по сравнению с традиционными обобщениями:

| Преимущество | Описание |
|------|------|
| **Тип зависит от значения** | `Vec: (n: Int) -> Type` позволяет типу зависеть от конкретного значения |
| **Вычисление на этапе компиляции** | Вызовы функций в позициях типов вычисляются на этапе компиляции, результат встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` проверяет размерности матрицы на этапе компиляции |
| **Вычисления на уровне типов** | Условные типы `If`, `Match` поддерживают вычисления на уровне типов |
| **Гарантия завершения** | Атрибут decreases обеспечивает гарантированное завершение вычислений на этапе компиляции |

```yaoxiang
# Compile-time проверка, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# На этапе компиляции вычисляется: factorial(3) = 6, factorial(2) = 2
# Тип: Matrix(Float, 6, 2)

# Несоответствие размерностей обнаруживается на этапе компиляции
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщённого программирования

```yaoxiang
# Пример: унифицированный дизайн API
# Операции map для разных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Подход с обобщениями: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели дизайна

### Основные цели

1. **Абстракции с нулевой стоимостью** - вызов обобщённой функции эквивалентен вызову конкретной функции
2. **Устранение мёртвого кода** - анализ на этапе компиляции, инстанцируются только используемые обобщения
3. **Замена макросов** - обобщения заменяют 90% сценариев использования макросов
4. **Типобезопасность** - проверка на этапе компиляции, без накладных расходов типов во время выполнения
5. **Дружелюбность к IDE** - интеллектуальные подсказки, чёткие сообщения об ошибках
6. **Типозависимые типы** - типы могут зависеть от значений, поддержка compile-time проверки размерностей
7. **Безопасность compile-time вычислений** - гарантия завершения compile-time вычислений через атрибут decreases

### Принципы дизайна

- **Детерминированность на этапе компиляции**: параметры обобщений определяются на этапе компиляции
- **Приоритет мономорфизации**: генерация конкретного кода, избегание виртуальных вызовов
- **Ограничения направляют инстанциацию**: ограничения типов направляют инстанциацию
- **Оптимизация под платформу**: специализация поддерживает оптимизацию под конкретную платформу
- **Единство типовой вселенной**: функции/конструкторы типов/типозависимые типы унифицированы на уровне Type2
- **Гарантия завершения**: вызовы функций в позициях типов должны доказать завершение

## Предложение

### 1. Базовые обобщения

#### 1.1 Параметры обобщённого типа

> **Ключевое правило**: Определения обобщённых типов **должны явно содержать аннотацию `: Type`**, иначе HM-вывод определит их как функции.
>
> | Запись | Значение |
> |------|------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM-вывод определяет как функцию, не тип |

```yaoxiang
# Определение обобщённого типа (обязательно с : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM-вывод определяет как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Обобщённое ограничение (прямое выражение, return можно опустить в однострочном)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров типа
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 2.4 Встроенные marker trait: Dup и Clone

**Три категории семантики копирования**:

| Тип | Значение | Способ активации | Сценарий применения |
|------|------|----------|------|
| **Примитивное значение** | При присваивании автоматическое копирование значения, два значения полностью независимы | Автоматически при присваивании/передаче параметров | Int, Float, Bool, Char |
| **Dup** | Поверхностное копирование: копирование дескриптора/токена, базовые данные разделяются | Автоматически при присваивании/передаче параметров | `&T` токены, `ref T`, String/Bytes |
| **Clone** | Глубокое копирование: создание полностью независимой копии | `value.clone()` | Любой тип, реализующий Clone |

**Семантика Dup**: типы, реализующие Dup, при присваивании/передаче параметров не передают владение — компилятор копирует дескриптор/токен, несколько владельцев указывают на одни и те же базовые данные. Это дополнение к семантике Move по умолчанию из модели владения RFC-009.

**Dup и Clone — ортогональные концепции**:

```
Dup = Копирование дескриптора, разделяемые данные (изменения взаимно влияют)
Clone = Копирование данных, независимые копии (изменения не влияют друг на друга)
```

**Правила**:

```
1. Типы примитивных значений (Int, Float, Bool, Char) — встроенное копирование значения компилятором, не принадлежат к Dup
2. Dup — применимо только к ссылочным/токенным типам и типам с внутренним подсчётом ссылок
3. Clone — явное глубокое копирование, любой тип может реализовать
4. Move по умолчанию — остальные типы сохраняют семантику Move по умолчанию
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|------|-----|------|
| `&T` (заимствованный токен) | ✅ | Токен нулевого размера, копирование токена = несколько представлений одних данных |
| `ref T` | ✅ | Rc/Arc copy = счётчик ссылок +1, разделяемые кучевов данные |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, копирование дескриптора разделяет базовый буфер |
| `&mut T` (изменяемый токен) | ❌ | Линейный эксклюзивный, нельзя копировать |
| struct | Производный | Все поля Dup → struct Dup |
| enum | Производный | Все поля всех вариантов Dup → enum Dup |
| tuple | Производный | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захваченное окружение может не быть Dup |
| `*T` (сырой указатель) | ❌ | unsafe, не участвует в системе владения |

**Int/Float/Bool/Char не являются Dup** — это типы значений, при присваивании компилятор автоматически копирует значение (два значения полностью независимы). Это не «поверхностное копирование», это встроенная обработка примитивов компилятором, не нуждающаяся и не требующая выражения через type attribute Dup.

```yaoxiang
# Типы примитивных значений: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x          # Копирование значения, x и y полностью независимы
print(x)       # ✅

# Dup: поверхностное копирование, копирование дескриптора разделяет данные
view: &Point = &point
view2 = view    # ✅ Dup: копирование токена, оба указывают на один point
print(view.x)   # ✅

# Clone: явное глубокое копирование, создание независимой копии
backup = big_struct.clone()  # Явный вызов

# Обобщённые ограничения
dup_use: (T: Dup) -> T = x         # T: Dup → можно поверхностное копирование
clone_use: (T: Clone) -> T = x.clone()  # T: Clone → можно глубокое копирование
```

> **Примечание**: `Send`/`Sync` не являются видимыми для пользователя trait. Гарантия безопасности межзадачного взаимодействия полностью автоматически обрабатывается компилятором через ключевое слово `ref` — `ref` автоматически выбирает Rc или Arc, пользователю не нужно понимать Send/Sync.

#### 4.1 Параметры compile-time констант

**Ключевой аспект дизайна**: `Type` в сигнатурах обобщений помечает параметры compile-time типа, параметры-значения типа `Int` по умолчанию определяемы на этапе компиляции в контексте обобщений. Ключевое слово `const` не требуется.

# Compile-time факториал: N должен быть известной на этапе компиляции литеральной константой

add: (a: Int, b: Int) -> (a: a, b: b) -> Int = a + b

### 5. Условные типы

> **Изоморфизм Карри-Ховарда**: условные типы с точки зрения Карри-Ховарда являются **case-анализом** в логике. Тип `Bool` соответствует утверждению с двумя возможными значениями (True/False), `If` выбирает разные результаты в зависимости от истинности этого утверждения — это именно case-дизъюнкция в логике. `match C { True => T, False => E }` фактически выражает: «когда известно, что утверждение C истинно, заключение T; когда C ложно, заключение E».

#### 5.1 Условный тип If

```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Проверка на этапе компиляции
Assert: (C: Bool) -> Type = match C {
    True => Void,
    False => compile_error("Assertion failed"),
}

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри-Ховарда**: семейства типов — наиболее прямое проявление «утверждения являются типами». `Add: (A: Type, B: Type) -> Type` — это не «написана функция сложения на уровне типов», а **конструирование утверждения о сложении натуральных чисел**. `(Zero, B) => B` означает «утверждение Add(Zero, B) эквивалентно B», `(Succ(A'), B) => Succ(Add(A', B))` означает «если Add(A', B) выполняется, то Add(Succ(A'), B) тоже выполняется». Это само определение сложения из аксиом Пеано. Проверка типами этого match-выражения эквивалентна проверке логической согласованности данного определения.

```yaoxiang
# Преобразование типов на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # По умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Curry-Howard: это также индуктивное определение сложения натуральных чисел)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: compile-time вычисление 2 + 3
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

struct InstantiationGraph {
    // Узлы: инстанциации обобщений
    nodes: HashMap<InstanceKey, InstanceNode>,

    // Рёбра: отношения использования
    edges: HashMap<InstanceKey, Vec<InstanceKey>>,

    type_args: Vec<TypeId>,  // Параметры типа
    const_args: Vec<ConstId>,  // Параметры Const
}

fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Непосещённые инстанциации — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение ёмкости
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

    if self.length > 0 {
        self.length = self.length - 1
        return Option.some(self.data[self.length])
    } else {
        return Option.none()
    }
}

    result = List(R)()
    for i in 0..self.length {
        result.push(f(self.data[i]))
    }
    return result
}

    result = List(T)()
    for i in 0..self.length {
        if predicate(self.data[i]) {
            result.push(self.data[i])

# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённый quicksort
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Использование перегрузки функций для реализации
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========

Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),

    # Проверка размерностей на этапе компиляции: использование типа Assert стандартной библиотеки
    _assert: Assert[Rows > 0],  # Rows > 0, иначе ошибка компиляции
    _assert: Assert[Cols > 0],  # Cols > 0, иначе ошибка компиляции

    # Операции с матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {
            for j in 0..M {
                sum = Zero::zero()
                for k in 0..Cols {
                    sum = sum + self.data[i][k] * other.data[k][j]
                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    return matrix
}

matrix_3x2 = Matrix(Float, 3, 2)()
matrix_3x2.data[0][0] = 7.0
matrix_3x2.data[0][1] = 8.0
matrix_3x2.data[1][0] = 9.0
matrix_3x2.data[1][1] = 10.0
matrix_3x2.data[2][0] = 11.0
matrix_3x2.data[2][1] = 12.0

# Умножение матриц: 2x3 * 3x2 = 2x2
result = matrix_2x3.multiply(matrix_3x2)

# На этапе компиляции проверяется: тип result — Matrix(Float, 2, 2)
# Единичная матрица 2x2
identity_3x3 = identity(Float, 3)()

# Несоответствие размерностей: ошибка компиляции
# bad_multiply = matrix_2x3.multiply(identity_3x3)  # Ошибка компиляции: 3x3 != 2x3

## Компромиссы

### Преимущества

1. **Абстракции с нулевой стоимостью**
   - Мономорфизация на этапе компиляции, отсутствие накладных расходов во время выполнения
   - Без виртуальных функций, без RTTI

2. **Устранение мёртвого кода**
   - Анализ на этапе компиляции, инстанцируются только используемые обобщения
   - Разбухание кода под контролем

3. **Замена макросов**
   - Генерация кода с типобезопасностью
   - Дружелюбность к IDE, чёткие сообщения об ошибках

4. **Вычисления на этапе компиляции**
   - Поддержка вычислений на этапе компиляции для обобщений
   - Такие характеристики как проверка размерностей
   - Без ключевого слова `const`, только ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанциация обобщений увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Использование памяти**
   - Увеличение использования памяти компилятором
   - Механизм кэширования требует памяти

3. **Сложность реализации**
   - Решатель ограничений сложен
   - Механизм вычислений на уровне типов сложен

4. **Диагностика ошибок**
   - Ошибки обобщений могут быть сложными
   - Требуются чёткие подсказки об ошибках

### Меры по смягчению

1. **Стратегия кэширования**
   - Кэширование результатов инстанциации
   - LRU-кэш для ограничения использования памяти

2. **Инкрементальная компиляция**
   - Кэширование результатов компиляции
   - Инкрементальная инстанциация

3. **Подсказки об ошибках**
   - Чёткие сообщения об ошибках
   - Подсказки при выводе обобщённых параметров

4. **Параллельная компиляция**
   - Параллельная инстанциация обобщений
   - Многопоточное решение ограничений

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| Только базовые обобщения | Не способны заменить сложные макросы |
| Чистая макросистема | Нет типобезопасности, плохие сообщения об ошибках |
| Только ограничения | Недостаточная гибкость |
| Обобщения во время выполнения | Есть накладные расходы производительности |

### Риски

| Риск | Влияние | Меры по смягчению |
|------|------|----------|
| Сложность решения ограничений | Слишком долгое время компиляции | Инкрементальное решение + кэширование |
| Разбухание кода | Слишком большой бинарный файл | DCE + контроль порогов |
| Сложность реализации | Увеличение срока разработки | Фазовость реализации |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

## Приложение

### БНФ грамматика

```bnf
# Параметры обобщений используют унифицированный синтаксис (), как часть типа функции
# Например map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в параметрах обобщений)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры обобщений — первая группа параметров в типе функции: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённые типы как List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Параметр Type в параметрах обобщений автоматически заполняется компилятором из фактических аргументов
# Например map(numbers, f), T извлекается из numbers: List(Int), R из f: (Int) -> String
```
