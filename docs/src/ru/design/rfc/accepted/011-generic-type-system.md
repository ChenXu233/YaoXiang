# RFC-011: Дизайн системы обобщённого программирования — Абстракции с нулевой стоимостью и замена макросов

> **Статус**: Принято
> **Автор**: Чэнь Сюй (晨煦)
> **Дата создания**: 2025-01-25
> **Последнее обновление**: 2026-04-22 (обновлено до механизма самодокументирования Type, унифицирован синтаксис вызова обобщённых функций)

## Аннотация

В данном документе определяется **дизайн системы обобщённого программирования** языка YaoXiang, реализующий абстракции с нулевой стоимостью через мощные возможности обобщённого программирования, сокращая зависимость от макросов посредством оптимизации на этапе компиляции и предоставляя механизм устранения мёртвого кода.

**Ключевые решения дизайна**:
- **Унифицированный синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — обобщённые параметры унифицированы с обычными параметрами
- **Механизм самодокументирования Type**: `Type` является языковой конструкцией особого рода, позиции `Type` в сигнатурах заполняются автоматически путём вывода
- **Ограничения типов**: `T: Clone + Add` — множественные ограничения, ограничения на типы функций
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Обобщённые типы на этапе компиляции**: `N: Int` — параметры-значения обобщённых типов, инстанцирование на этапе компиляции
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Абстракции с нулевой стоимостью: мономорфизация на этапе компиляции, отсутствие накладных расходов во время выполнения
- Устранение мёртвого кода: анализ графа инстанцирования + оптимизация LLVM
- Замена макросов: обобщённые типы покрывают 90% сценариев использования макросов
- Безопасность типов: проверка на этапе компиляции, удобство для IDE
- **Явное лучше неявного**: `Type` самодокументируется, компилятор автоматически выводит типы

## Концепция типовой вселенной и типы, зависящие от значений

Система обобщённого программирования YaoXiang построена на **концепции типовой вселенной** — ментальной модели, объединяющей все понятия языка в иерархическую структуру. Ключевая инновация заключается в том, что **типы, зависящие от значений**, подняты до статуса полноправных граждан на уровне Type2.

### Что такое типы, зависящие от значений?

**Типы, зависящие от значений** — это типы, которые зависят от одного или нескольких **значений** (а не только от других типов). Эти значения могут вычисляться на этапе компиляции, обеспечивая гарантии безопасности типов уже на этапе компиляции.

```yaoxiang
# Традиционные обобщённые типы: параметры-типы
List: (T: Type) -> Type

# Типы, зависящие от значений: параметры-значения
Vec: (n: Int) -> Type  # Тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # Тип матрицы зависит от количества строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционным обобщённым программированием, типы YaoXiang, зависящие от значений, обладают следующими преимуществами:

| Характеристика | Традиционные обобщённые типы (C++/Rust) | YaoXiang: типы, зависящие от значений |
|----------------|----------------------------------------|---------------------------------------|
| Значения, от которых зависит тип | Только параметры-типы | Любые значения, включая результаты вызовов функций |
| Вычисления на этапе компиляции | Ручная специализация шаблонов C++, отсутствует в Rust | Автоматические вычисления на этапе компиляции с гарантией завершения |
| Вычисления на уровне типов | Метапрограммирование шаблонов (сложно/опасно) | Единый механизм вычислений на уровне типов |
| Безопасность типов | C++ отсутствует, Rust ограничена | Полная безопасность типов, проверка на этапе компиляции |
| Проверка размерностей | Проверка во время выполнения или ручная специализация | Проверка размерностей на этапе компиляции, без накладных расходов во время выполнения |

### Уровни типовой вселенной и типы, зависящие от значений

Концепция типовой вселенной разделяет понятия языка по семантическим ролям на разные уровни. Типы, зависящие от значений, располагаются на **уровне Type2**:

| Уровень | Роль | Пример |
|---------|------|--------|
| Type-1 | Значения | `42`, `factorial(5)`, сами функции |
| Type0 | Ключевое слово метакласса | `Type` |
| Type1 | Конкретные типы | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функции/конструкторы типов/типы, зависящие от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевое решение дизайна**: Функции, конструкторы типов и типы, зависящие от значений, на уровне Type2 имеют **унифицированный синтаксис** — всё это форма `(params) -> result`:
- Обычные функции: `(Int, Int) -> Int` → возвращают значение
- Конструкторы типов: `(T: Type) -> Type` → возвращают тип
- Типы, зависящие от значений: `(n: Int) -> Type` → возвращают тип, но зависят от параметров-значений

> **Изоморфизм Карри — Ховарда**: Это единство — не случайность. Изоморфизм Карри — Ховарда утверждает, что «типы — это высказывания, программы — это доказательства». Функциональный тип `A → B` соответствует логическому импликации «если A, то B», обобщённый тип `(T: Type) -> Type` соответствует универсальному квантору «для всех типов T», а тип, зависящий от значений, `(n: Int) -> Type` соответствует «для каждого целого n существует тип». Унификация функций, конструкторов типов и типов, зависящих от значений, на уровне Type2 в YaoXiang по сути объединяет «доказательства» и «вычисления» в единое понятие — **конструктивное доказательство**. Это прямое воплощение изоморфизма Карри — Ховарда в дизайне языка: одна форма (`(params) -> result`) одновременно выражает логическое высказывание и вычислительный процесс.

### Гарантии детерминированности на этапе компиляции

Концепция типовой вселенной YaoXiang требует: **всё на уровне Type определяется на этапе компиляции**.

```yaoxiang
# Пример проверки размерностей на этапе компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка на этапе компиляции: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется на этапе компиляции
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Вычисление на этапе компиляции: factorial(3) = 6, размер вектора определяется на этапе компиляции
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов
2. Проверяет, помечены ли функции уменьшающими вариантами (decreases) (см. механизм проверки завершения ниже)
3. Выполняет вычисления на этапе компиляции
4. Встраивает результат в генерируемый тип

### Сценарии применения типов, зависящих от значений

#### Проверка размерностей на этапе компиляции
```yaoxiang
# Умножение матриц: проверка соответствия размерностей на этапе компиляции
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> (((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка на этапе компиляции: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка перехватывается на этапе компиляции:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива является константой этапа компиляции
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — константа этапа компиляции, может использоваться для вычислений на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно на этапе компиляции)
```

#### Условные типы
```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейство типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры типов T, R определяются на этапе компиляции
map: (T: Type, R: Type) -> ((
    list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    result
}

# Использование полностью прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # Выводится как map[Int, Int]
```

### Механизм проверки завершения (интеграция с RFC-022)

Вычисления типов, зависящих от значений, на этапе компиляции должны **гарантированно завершаться**, иначе система типов войдёт в бесконечный цикл. YaoXiang обеспечивает это через **уменьшающие варианты** (decreases), полностью интегрированные с RFC-022.

#### Decreases-规ла для рекурсивных функций
```yaoxiang
# Факториал на этапе компиляции: необходимо доказать завершение
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # При каждой рекурсии n строго уменьшается
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # Вычисление factorial(5) = 120 на этапе компиляции
```

#### Decreases-规ла для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершения

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза проверки типов                                        │
│  Встречен вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка decreases-规ла                                  │
│     - Есть decreases: верификация убывания на всех путях рекурсии │
│     - Нет decreases, но завершение очевидно: прямое вычисление    │
│     - Нет decreases и возможно незавершение: ошибка компиляции     │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисление на этапе компиляции (встроенный интерпретатор)      │
│     - Чистые функции: прямое вычисление                         │
│     - Побочные эффекты: ошибка компиляции (в позиции типа побочных эффектов быть не может) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                              │
│     - Vec(factorial(5)) → Vec(120)                           │
│     - Matrix(Float, 3, 3) → конкретный тип                   │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: Гарантирует обязательное завершение вычислений на этапе компиляции, предотвращая зацикливание системы типов
- **Единство**: Проверка завершения и верификация частичной корректности используют единый механизм спецификаций
- **Постепенное улучшение**: Можно постепенно перейти от проверок во время выполнения к полной статической верификации

## Мотивация

### Зачем нужна мощная система обобщённого программирования?

Текущие языки имеют ограничения в обобщённом программировании:

| Язык | Возможности обобщённого программирования | Проблема |
|------|-------------------------------------------|----------|
| Java | Ограниченные типы | Компиляция с мономорфизацией, без специализации обобщённых типов |
| C# | Обобщённые типы с ограничениями | Проверка типов во время выполнения, накладные расходы |
| Rust | Обобщённые типы + trait | Сложная система trait, крутая кривая обучения |
| C++ | Шаблоны | Сложная специализация шаблонов, плохие сообщения об ошибках |
| **YaoXiang** | **Типы, зависящие от значений** | **Типы зависят от значений, проверка размерностей на этапе компиляции, гарантия завершения** |

### Основные противоречия

1. **Производительность vs гибкость**: Гибкость во время выполнения vs оптимизация на этапе компиляции
2. **Сложность vs простота**: Мощная система типов vs удобство использования
3. **Макросы vs обобщённые типы**: Генерация кода макросами vs безопасность типов обобщённых типов
4. **Зависимость от значений vs безопасность типов**: Традиционные обобщённые типы не могут проверять размерности на этапе компиляции

### Ключевые преимущества типов, зависящих от значений

**Типы YaoXiang, зависящие от значений** — это ключевое преимущество перед традиционным обобщённым программированием:

| Преимущество | Описание |
|--------------|----------|
| **Типы зависят от значений** | `Vec: (n: Int) -> Type` делает тип зависящим от конкретного значения |
| **Вычисления на этапе компиляции** | Вызовы функций в позициях типов вычисляются на этапе компиляции, результат встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` проверяет размерности матрицы на этапе компиляции |
| **Вычисления на уровне типов** | `If`, `Match` и другие условные типы поддерживают вычисления на уровне типов |
| **Гарантия завершения** | Decreases-规ла обеспечивает обязательное завершение вычислений на этапе компиляции |

```yaoxiang
# Проверка на этапе компиляции, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Вычисление на этапе компиляции: factorial(3) = 6, factorial(2) = 2
# Тип: Matrix(Float, 6, 2)

# Несоответствие размерностей перехватывается на этапе компиляции
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщённого программирования

```yaoxiang
# Пример: Унифицированный дизайн API
# Операция map для разных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Обобщённый подход: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели дизайна

### Основные цели

1. **Абстракции с нулевой стоимостью** — Вызов обобщённой функции эквивалентен вызову конкретной типизированной функции
2. **Устранение мёртвого кода** — Анализ на этапе компиляции, инстанциируются только используемые обобщённые типы
3. **Замена макросов** — Обобщённые типы заменяют 90% сценариев использования макросов
4. **Безопасность типов** — Проверка на этапе компиляции, отсутствие накладных расходов на проверку типов во время выполнения
5. **Удобство для IDE** — Интеллектуальные подсказки, чёткие сообщения об ошибках
6. **Типы, зависящие от значений** — Типы могут зависеть от значений, поддержка проверки размерностей на этапе компиляции
7. **Безопасность вычислений на этапе компиляции** — Гарантия завершения вычислений на этапе компиляции через decreases-规ла

### Принципы дизайна

- **Определённость на этапе компиляции**: Параметры обобщённых типов определяются на этапе компиляции
- **Приоритет мономорфизации**: Генерация конкретного кода, избегание виртуальных вызовов
- **Ограничения как драйверы**: Ограничения типов направляют инстанцирование
- **Оптимизация под платформу**: Специализация поддерживает оптимизацию под конкретную платформу
- **Единство типовой вселенной**: Функции/конструкторы типов/типы, зависящие от значений, унифицированы на уровне Type2
- **Гарантия завершения**: Вызовы функций в позициях типов должны доказать завершение

## Предложение

### 1. Базовые обобщённые типы

#### 1.1 Параметры обобщённых типов

> **Ключевое правило**: Определения обобщённых типов **должны явно помечаться `: Type`**, иначе HM-вывод определит их как функции.
>
> | Запись | Значение |
> |--------|----------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM-вывод определяет как функцию, не тип |

```yaoxiang
# Определение обобщённого типа (обязательно : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM-вывод определяет как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Обобщённое ограничение (прямое выражение, в одну строку return можно опустить)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров типов
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

# Использование синтаксиса методов: Array.Item, Array.next, Array.has_next

    return self.index < self.length
}

Array.next: (T: Type)(self: Array(T)) -> Option(T) = {
    if has_next(self) {
        item = self.data[self.index]
        self.index = self.index + 1
        return Option.some(item)
    } else {
        return Option.none()
    }
}

Array.Item: (T: Type)(arr: Array(T)) -> T = {
    return arr.data[0]
}

### 5. Условные типы

> **Изоморфизм Карри — Ховарда**: Условные типы с точки зрения Карри — Ховарда представляют собой **case-анализ** в логике. Тип `Bool` соответствует высказыванию с двумя возможными значениями (True/False), а `If` выбирает разные результаты в зависимости от истинности этого высказывания — это именно case-дизъюнкция в логике. `match C { True => T, False => E }` фактически выражает: «когда известно, что высказывание C истинно, заключение T; когда C ложно, заключение E».

#### 5.1 Условный тип If

```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Проверка на этапе компиляции
Assert: (C: Bool) -> Type = match C {
    True => Void,
    False => compile_error("Assertion failed"),
}

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри — Ховарда**: Семейства типов — наиболее прямое воплощение «высказывания как тип». `Add: (A: Type, B: Type) -> Type` — это не «написанная на уровне типов функция сложения», а **построение высказывания о сложении натуральных чисел**. `(Zero, B) => B` означает «высказывание Add(Zero, B) эквивалентно B», а `(Succ(A'), B) => Succ(Add(A', B))` означает «если Add(A', B) верно, то Add(Succ(A'), B) тоже верно». Это само определение сложения из аксиом Пеано. Проверка типами этого match-выражения эквивалентна проверке логической согласованности данного определения.

```yaoxiang
# Преобразование типов на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # По умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Curry-Howard: это также индуктивное определение сложения натуральных чисел)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: вычисление 2 + 3 на этапе компиляции
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

struct InstantiationGraph {
    // Узлы: инстанцирования обобщённых типов
    nodes: HashMap<InstanceKey, InstanceNode>,

    // Рёбра: отношения использования
    edges: HashMap<InstanceKey, Vec<InstanceKey>>,
}

struct InstanceKey {
    generic: FunctionId,  // ID обобщённой функции
    type_args: Vec<TypeId>,  // Параметры-типы
    const_args: Vec<ConstId>,  // Параметры-константы
}

// Алгоритм: анализ достижимости
fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Непосещённые инстанцирования — это мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}

    # Обобщённые методы (T автоматически вводится в область видимости из внешнего List(T))
    push: (self: List(T), item: T) -> Void,
    pop: (self: List(T)) -> Option(T),
    map: (R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)),
    filter: (self: List(T), predicate: (T) -> Bool) -> List(T),

# ======== 2. Реализация обобщённых методов ========
# Использование синтаксиса Type.method: автоматически ассоциируется с типом List

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение ёмкости
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

List.pop: (T: Type) -> ((self: List(T)) -> Option(T)) = {
    if self.length > 0 {
        self.length = self.length - 1
        return Option.some(self.data[self.length])
    } else {
        return Option.none()
    }
}

List.map: (T: Type, R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)) = {
    result = List(R)()
    for i in 0..self.length {
        result.push(f(self.data[i]))
    }
    return result
}

List.filter: (T: Type) -> ((self: List(T), predicate: (T) -> Bool) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        if predicate(self.data[i]) {
            result.push(self.data[i])
        }
    }
    return result
}

List.fold: (T: Type, U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U) = {
    result = initial
    for i in 0..self.length {
        result = f(result, self.data[i])
    }
    return result
}

# ======== 3. Использование ограничений типов ========
# Реализация Clone для List
List.clone: (T: Clone) -> ((self: List(T)) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        result.push(self.data[i].clone())
    }
    return result
}

# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённый quicksort
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Использование перегрузки функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========
# Сортировка массива Int

sorted = quicksort(numbers, Comparator(Int)())

# Сортировка массива String (требуется StringComparator)

    _assert: Assert[Rows > 0],  # Rows > 0, иначе ошибка компиляции

## Компромиссы

### Преимущества

1. **Абстракции с нулевой стоимостью**
   - Мономорфизация на этапе компиляции, отсутствие накладных расходов во время выполнения
   - Без виртуальных функций, без RTTI

2. **Устранение мёртвого кода**
   - Анализ на этапе компиляции, инстанциируются только используемые обобщённые типы
   - Разбухание кода под контролем

3. **Замена макросов**
   - Типобезопасная генерация кода
   - Удобство для IDE, чёткие сообщения об ошибках

4. **Вычисления на этапе компиляции**
   - Поддержка вычислений на этапе компиляции через обобщённые типы
   - Такие возможности, как проверка размерностей
   - Ключевое слово `const` не требуется, только ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанцирование обобщённых типов увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Использование памяти**
   - Увеличение использования памяти компилятором
   - Механизм кэширования требует память

3. **Сложность реализации**
   - Сложный решатель ограничений
   - Сложный механизм вычислений на уровне типов

4. **Диагностика ошибок**
   - Ошибки, связанные с обобщёнными типами, могут быть сложными
   - Требуются чёткие подсказки при ошибках

### Меры по смягчению

1. **Стратегия кэширования**
   - Кэширование результатов инстанцирования
   - Ограничение использования памяти LRU-кэшем

2. **Инкрементная компиляция**
   - Кэширование результатов компиляции
   - Инкрементное инстанцирование

3. **Подсказки при ошибках**
   - Чёткие сообщения об ошибках
   - Подсказки по выводу обобщённых параметров

4. **Параллельная компиляция**
   - Параллельное инстанцирование обобщённых типов
   - Многопоточное решение ограничений

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Только базовые обобщённые типы | Не способны заменить сложные макросы |
| Чистая макросистема | Без безопасности типов, плохие сообщения об ошибках |
| Только ограничения | Недостаточная гибкость |
| Обобщённые типы во время выполнения | Накладные расходы на производительность |

### Риски

| Риск | Влияние | Меры по смягчению |
|------|--------|-------------------|
| Сложность решения ограничений | Слишком долгое время компиляции | Инкрементное решение + кэширование |

## Приложение

### BNF-синтаксис

```bnf
# Параметры обобщённых типов используют унифицированный синтаксис (), как часть типа функции
# Например: map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничения типов (в параметрах обобщённых типов)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры обобщённых типов — первая группа параметров в типе функции: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённые типы, такие как List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Type в параметрах обобщённых типов автоматически заполняется компилятором из фактических типов аргументов
# Например: map(numbers, f), T извлекается из numbers: List(Int), R из f: (Int) -> String
```
