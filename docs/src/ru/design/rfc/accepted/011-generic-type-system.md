# RFC-011: Дизайн системы обобщений — нулевая стоимость абстракции и замена макросов

> **Статус**: Принято
> **Автор**: Чэнь Сюй
> **Дата создания**: 2025-01-25
> **Последнее обновление**: 2026-04-22 (обновлено до механизма самодокументирования Type, унифицирован синтаксис обобщённых вызовов)

## Аннотация

В данном документе определяется **дизайн системы обобщений** языка YaoXiang, реализующий нулевую стоимость абстракции через мощные возможности обобщений, сокращая зависимость от макросов посредством оптимизации на этапе компиляции и предоставляя механизм устранения мёртвого кода.

**Основной дизайн**:
- **Унифицированный синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — обобщённые параметры унифицированы с обычными параметрами
- **Механизм самодокументирования Type**: `Type` является языковой сущностью особого рода; позиции `Type` в сигнатурах автоматически заполняются выводом
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения функционального типа
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Компилируемые обобщения**: `N: Int` — параметры-значения обобщений, инстанцирование на этапе компиляции
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Нулевая стоимость абстракции: мономорфизация на этапе компиляции, отсутствие накладных расходов во время выполнения
- Устранение мёртвого кода: анализ графа инстанцирования + оптимизация LLVM
- Замена макросов: обобщения заменяют 90% сценариев использования макросов
- Типобезопасность: проверка на этапе компиляции, дружественность к IDE
- **Явное лучше неявного**: `Type` самодокументируется, компилятор выводит автоматически

## Концепция типа-вселенной и типы, зависящие от значений

Система обобщений YaoXiang построена на **концепции типа-вселенной** — ментальной модели, унифицирующей все концепции языка в иерархическую структуру, где ключевая инновация заключается в продвижении **типов, зависящих от значений**, до первоклассных граждан уровня Type2.

### Что такое типы, зависящие от значений?

**Тип, зависящий от значения**, — это тип, который зависит от одного или нескольких **значений** (а не только от других типов). Эти значения могут вычисляться на этапе компиляции, обеспечивая гарантии типобезопасности уже на этапе компиляции.

```yaoxiang
# Традиционные обобщения: параметры типа
List: (T: Type) -> Type

# Типы, зависящие от значений: параметры-значения
Vec: (n: Int) -> Type  # Тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # Тип матрицы зависит от числа строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционными обобщениями, типы YaoXiang, зависящие от значений, обладают следующими преимуществами:

| Характеристика | Традиционные обобщения (C++/Rust) | YaoXiang — типы, зависящие от значений |
|----------------|-----------------------------------|----------------------------------------|
| Значения, от которых зависит тип | Только параметры типа | Любые значения, включая результаты вызовов функций |
| Вычисления на этапе компиляции | Ручная специализация шаблонов C++, в Rust отсутствует | Автоматические вычисления на этапе компиляции с гарантией завершения |
| Вычисления на уровне типов | Метапрограммирование шаблонов (сложно/опасно) | Унифицированный движок вычислений на уровне типов |
| Типобезопасность | C++ отсутствует, Rust ограничена | Полная типобезопасность, проверки на этапе компиляции |
| Проверка размерностей | Проверка во время выполнения или ручная специализация | Проверка размерностей на этапе компиляции, без накладных расходов |

### Уровни типа-вселенной и типы, зависящие от значений

Концепция типа-вселенной распределяет концепции языка по семантическим ролям на разные уровни, где типы, зависящие от значений, находятся на **уровне Type2**:

| Уровень | Роль | Пример |
|---------|------|--------|
| Type-1 | Значение | `42`, `factorial(5)`, сама функция |
| Type0 | Ключевое слово меатипа | `Type` |
| Type1 | Конкретный тип | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функция/конструктор типа/тип, зависящий от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевой дизайн**: Функции, конструкторы типов и типы, зависящие от значений, на уровне Type2 **унифицированы синтаксически** — все имеют форму `(params) -> result`:
- Обычная функция: `(Int, Int) -> Int` → возвращает значение
- Конструктор типа: `(T: Type) -> Type` → возвращает тип
- Тип, зависящий от значений: `(n: Int) -> Type` → возвращает тип, но зависит от параметра-значения

> **Изоморфизм Карри — Ховарда**: Данная унификация — не совпадение. Изоморфизм Карри — Ховарда устанавливает, что «типы — это утверждения, программы — это доказательства». Функциональный тип `A → B` соответствует логическому импликации «если A, то B», обобщение `(T: Type) -> Type` соответствует универсальному квантору «для всех типов T», а тип, зависящий от значений, `(n: Int) -> Type` соответствует «для каждого целого n существует тип». Унификация YaoXiang функций, конструкторов типов и типов, зависящих от значений, на уровне Type2 по сути унифицирует «доказательства» и «вычисления» в единую концепцию — **конструктивное доказательство**. Это прямое воплощение изоморфизма Карри — Ховарда в дизайне языка: одна форма (`(params) -> result`) одновременно выражает логическое утверждение и вычислительный процесс.

### Гарантии компилируемости

Концепция типа-вселенной YaoXiang требует: **всё на уровне Type должно быть определено на этапе компиляции**.

```yaoxiang
# Пример проверки размерностей на этапе компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка на этапе компиляции: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется на этапе компиляции
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Вычисление на этапе компиляции: factorial(3) = 6, размер вектора определяется на этапе компиляции
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типа
2. Проверяет наличие редукции decreases для функций (см. механизм проверки завершения ниже)
3. Выполняет вычисление на этапе компиляции
4. Встраивает результат в генерируемый тип

### Сценарии применения типов, зависящих от значений

#### Проверка размерностей на этапе компиляции
```yaoxiang
# Умножение матриц: проверка соответствия размерностей на этапе компиляции
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка на этапе компиляции: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка перехватывается на этапе компиляции:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива — компилируемая константа
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — компилируемая константа, может использоваться для вычислений на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно на этапе компиляции)
```

#### Условные типы
```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейство типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры типа T, R определяются на этапе компиляции
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    result
}

# При использовании полностью прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # Выводится как map[Int, Int]
```

### Механизм проверки завершения (интеграция с RFC-022)

Вычисления на этапе компиляции для типов, зависящих от значений, должны **гарантированно завершаться**, иначе система типов войдёт в бесконечный цикл. YaoXiang обеспечивает это через **редукцию decreases**, полностью интегрируясь с RFC-022.

#### Редукция завершения для рекурсивных функций
```yaoxiang
# Факториал на этапе компиляции: необходимо доказать завершение
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # При каждой рекурсии n строго уменьшается
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # Вычисление factorial(5) = 120 на этапе компиляции
```

#### Редукция завершения для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершения

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза проверки типов                                         │
│  Встречен вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка редукции decreases                               │
│     - Есть decreases: проверяется выполнение условия убывания на всех │
│       рекурсивных путях                                       │
│     - Нет decreases, но очевидно завершается: прямое вычисление │
│     - Нет decreases и возможно не завершается: ошибка компиляции │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисление на этапе компиляции (выполняется встроенным интерпретатором) │
│     - Чистая функция: прямое вычисление                       │
│     - Побочные эффекты: ошибка компиляции (в позиции типа побочных эффектов быть не может) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                              │
│     - Vec(factorial(5)) → Vec(120)                           │
│     - Matrix(Float, 3, 3) → конкретный тип                   │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантирует обязательное завершение вычислений на этапе компиляции, предотвращая бесконечные циклы в системе типов
- **Унификация**: проверка завершения и верификация частичной корректности используют общий механизм редукций
- **Постепенное усиление**: можно постепенно перейти от проверок во время выполнения к полностью статическому доказательству

## Мотивация

### Зачем нужна мощная система обобщений?

Возможности обобщений в текущих основных языках ограничены:

| Язык | Возможности обобщений | Проблема |
|------|----------------------|----------|
| Java | Ограниченные типы | Мономорфизация на этапе компиляции, без специализации обобщений |
| C# | Ограничения обобщений | Проверка типов во время выполнения, накладные расходы производительности |
| Rust | Обобщения + Trait | Система Trait сложна, крутая кривая обучения |
| C++ | Шаблоны | Специализация шаблонов сложна, плохие сообщения об ошибках компиляции |
| **YaoXiang** | **Типы, зависящие от значений** | **Типы могут зависеть от значений, проверка размерностей на этапе компиляции, гарантия завершения** |

### Основное противоречие

1. **Производительность vs Гибкость**: гибкость во время выполнения vs оптимизация на этапе компиляции
2. **Сложность vs Простота**: мощная система типов vs удобство использования
3. **Макросы vs Обобщения**: генерация кода макросами vs типобезопасность обобщений
4. **Зависимость от значений vs Типобезопасность**: традиционные обобщения не могут проверять размерности на этапе компиляции

### Ключевые преимущества типов, зависящих от значений

**Типы, зависящие от значений**, YaoXiang — ключевое преимущество перед традиционными обобщениями:

| Преимущество | Описание |
|--------------|----------|
| **Типы зависят от значений** | `Vec: (n: Int) -> Type` позволяет типу зависеть от конкретного значения |
| **Вычисления на этапе компиляции** | Вызовы функций в позициях типа вычисляются на этапе компиляции, результат встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` проверяет размерности матрицы на этапе компиляции |
| **Вычисления на уровне типов** | `If`, `Match` и другие условные типы поддерживают вычисления на уровне типов |
| **Гарантия завершения** | Редукция decreases обеспечивает обязательное завершение вычислений на этапе компиляции |

```yaoxiang
# Проверка на этапе компиляции, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Вычисление на этапе компиляции: factorial(3) = 6, factorial(2) = 2
# Тип: Matrix(Float, 6, 2)

# Несоответствие размерностей перехватывается на этапе компиляции
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщений

```yaoxiang
# Пример: унифицированный дизайн API
# Операция map для разных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Подход с обобщениями: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели дизайна

### Основные цели

1. **Нулевая стоимость абстракции** — обобщённые вызовы эквивалентны вызовам конкретных типов
2. **Устранение мёртвого кода** — анализ на этапе компиляции, инстанцируются только используемые обобщения
3. **Замена макросов** — обобщения заменяют 90% сценариев использования макросов
4. **Типобезопасность** — проверки на этапе компиляции, без накладных расходов типов во время выполнения
5. **Дружественность к IDE** — интеллектуальные подсказки, чёткие сообщения об ошибках
6. **Типы, зависящие от значений** — типы могут зависеть от значений, поддержка проверки размерностей на этапе компиляции
7. **Безопасность вычислений на этапе компиляции** — гарантия завершения вычислений на этапе компиляции через редукцию decreases

### Принципы дизайна

- **Определённость на этапе компиляции**: параметры обобщений определяются на этапе компиляции
- **Приоритет мономорфизации**: генерация конкретного кода, избегание виртуальных вызовов
- **Ограничения как направляющие**: ограничения типов направляют инстанцирование
- **Оптимизация под платформу**: специализация поддерживает оптимизацию под конкретную платформу
- **Унификация типа-вселенной**: функции/конструкторы типов/типы, зависящие от значений, унифицированы на уровне Type2
- **Гарантия завершения**: вызовы функций в позициях типа должны доказать завершение

### 1. Базовые обобщения

#### 1.1 Параметры обобщённого типа

> **Ключевое правило**: Определения обобщённых типов **должны явно помечаться `: Type`**, иначе HM выведет их как функции.
>
> | Запись | Значение |
> |--------|----------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM выводит как функцию, не как тип |

```yaoxiang
# Определение обобщённого типа (必须有 : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM выводит как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Обобщённое ограничение (прямое выражение, для однострочных можно опустить return)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров типа
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 2.4 Встроенные marker trait: Dup и Clone

**Три категории семантики копирования**:

| Тип | Значение | Способ активации | Область применения |
|-----|----------|------------------|-------------------|
| **Примитивное значение** | При присваивании автоматическое копирование значения, два значения полностью независимы | Автоматически при присваивании/передаче параметров | Int, Float, Bool, Char |
| **Dup** | Поверхностное копирование: копирование дескриптора/токена, базовые данные разделяются | Автоматически при присваивании/передаче параметров | Токены `&T`, `ref T`, String/Bytes |
| **Clone** | Глубокое копирование: создание полностью независимой копии | `value.clone()` | Любой тип, реализующий Clone |

**Семантика Dup**: Типы, реализующие Dup, при присваивании/передаче параметров не передают владение — компилятор копирует дескриптор/токен, несколько владельцев указывают на одни и те же базовые данные. Это дополнение к стандартной семантике Move из RFC-009.

**Dup и Clone — ортогональные концепции**:

```
Dup = копирование дескриптора, разделяемые данные (изменения влияют друг на друга)
Clone = копирование данных, независимые копии (изменения не влияют друг на друга)
```

**Правила**:

```
1. Типы примитивных значений (Int, Float, Bool, Char) — встроенное копирование значения компилятором, не являются Dup
2. Dup — применимо только к ссылочным/токенным типам и типам с внутренним подсчётом ссылок
3. Clone — явное глубокое копирование, любой тип может реализовать
4. По умолчанию Move — остальные типы сохраняют стандартную семантику Move
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|-----|-----|---------|
| `&T` (заимствованный токен) | ✅ | Токен нулевого размера, копирование токена = несколько представлений одних данных |
| `ref T` | ✅ | Rc/Arc copy = счётчик ссылок +1, разделяемые данные в куче |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, копирование дескриптора разделяет базовый буфер |
| `&mut T` (изменяемый токен) | ❌ | Линейный эксклюзив, не может копироваться |
| struct | Выводится | Все поля Dup → struct Dup |
| enum | Выводится | Все варианты всех полей Dup → enum Dup |
| tuple | Выводится | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захваченное окружение может не быть Dup |
| `*T` (сырой указатель) | ❌ | unsafe, не участвует в системе владения |

**Int/Float/Bool/Char не являются Dup** — это типы значений, при присваивании компилятор автоматически копирует значение (два значения полностью независимы). Это не «поверхностное копирование», это встроенная обработка примитивов компилятором; не нужно и не следует выражать через атрибут типа Dup.

```yaoxiang
# Типы примитивных значений: компилятор автоматически копирует значение (не Dup)
x: Int = 42
y = x          # Копирование значения, x и y полностью независимы
print(x)       # ✅

# Dup: поверхностное копирование, копирование дескриптора разделяет данные
view: &Point = &point
view2 = view    # ✅ Dup: копирование токена, оба указывают на один point
print(view.x)   # ✅

# Clone: явное глубокое копирование, создание независимой копии
backup = big_struct.clone()  # Явный вызов

# Обобщённые ограничения
dup_use: (T: Dup) -> T = x         # T: Dup → можно поверхностно копировать
clone_use: (T: Clone) -> T = x.clone()  # T: Clone → можно глубоко копировать
```

> **Примечание**: `Send`/`Sync` не являются видимыми для пользователя trait. Гарантия безопасности при межзадачном взаимодействии обеспечивается ключевым словом `ref` и полностью автоматической работой компилятора — `ref` автоматически выбирает Rc или Arc, пользователю не нужно понимать Send/Sync.

    }
}

### 5. Условные типы

> **Изоморфизм Карри — Ховарда**: Условные типы с точки зрения Карри — Ховарда представляют собой **case-анализ** в логике. Тип `Bool` соответствует утверждению с двумя возможными значениями (True/False), `If` выбирает разные результаты в зависимости от истинности этого утверждения — это именно дизъюнкция case в логике. `match C { True => T, False => E }` фактически выражает: «когда утверждение C истинно, заключение T; когда C ложно, заключение E».

#### 5.1 Условный тип If

```yaoxiang
# If на уровне типов
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Проверка на этапе компиляции
Assert: (C: Bool) -> Type = match C {
    True => Void,
    False => compile_error("Assertion failed"),
}

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри — Ховарда**: Семейства типов — наиболее прямое воплощение «утверждения — это тип». `Add: (A: Type, B: Type) -> Type` — это не «написанная на уровне типов функция сложения», а **конструирование утверждения о сложении натуральных чисел**. `(Zero, B) => B` означает «утверждение Add(Zero, B) эквивалентно B», `(Succ(A'), B) => Succ(Add(A', B))` означает «если Add(A', B) верно, то Add(Succ(A'), B) тоже верно». Это сама индуктивное определение сложения из аксиом Пеано. Проверка типами этого match-выражения эквивалентна проверке логической согласованности данного определения.

```yaoxiang
# Преобразование типов на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # По умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Curry-Howard: это также индуктивное определение сложения натуральных чисел)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: вычисление 2 + 3 на этапе компиляции
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

struct InstantiationGraph {
    // Узлы: инстанцирования обобщений

    const_args: Vec<ConstId>,  // Параметры Const

fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Непосещённые инстанцирования — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}

    if self.length >= self.data.length {

        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]

    return result
}

# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённый quicksort
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Использование перегрузки функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========

Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),

    # Проверка размерностей на этапе компиляции: использование типа Assert стандартной библиотеки
    _assert: Assert[Rows > 0],  # Rows > 0, иначе ошибка компиляции
    _assert: Assert[Cols > 0],  # Cols > 0, иначе ошибка компиляции

    # Операции с матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {

                sum = Zero::zero()
                for k in 0..Cols {
                    sum = sum + self.data[i][k] * other.data[k][j]
                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

matrix_3x2 = Matrix(Float, 3, 2)()
matrix_3x2.data[0][0] = 7.0
matrix_3x2.data[0][1] = 8.0
matrix_3x2.data[1][0] = 9.0
matrix_3x2.data[1][1] = 10.0
matrix_3x2.data[2][0] = 11.0
matrix_3x2.data[2][1] = 12.0

# Умножение матриц: 2x3 * 3x2 = 2x2
result = matrix_2x3.multiply(matrix_3x2)

# Проверка на этапе компиляции: тип результата Matrix(Float, 2, 2)
# Единичная матрица 2x2
identity_3x3 = identity(Float, 3)()

## Компромиссы

### Преимущества

1. **Нулевая стоимость абстракции**
   - Мономорфизация на этапе компиляции, без накладных расходов во время выполнения
   - Не требуются виртуальные вызовы, нет RTTI

2. **Устранение мёртвого кода**
   - Анализ на этапе компиляции, инстанцируются только используемые обобщения
   - Раздувание кода контролируемо

3. **Замена макросов**
   - Генерация кода с типобезопасностью
   - Дружественность к IDE, чёткие сообщения об ошибках

4. **Вычисления на этапе компиляции**
   - Поддержка вычислений на этапе компиляции в обобщениях
   - Такие характеристики, как проверка размерностей
   - Не требуется ключевое слово `const`, только ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанцирование обобщений увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Использование памяти**
   - Увеличение использования памяти компилятором
   - Механизм кэширования требует памяти

3. **Сложность реализации**
   - Сложность решателя ограничений
   - Сложность движка вычислений на уровне типов

4. **Диагностика ошибок**
   - Ошибки обобщений могут быть сложными
   - Требуются чёткие подсказки об ошибках

### Меры по смягчению

1. **Стратегия кэширования**
   - Кэширование результатов инстанцирования
   - Ограничение памяти через LRU-кэш

2. **Инкрементальная компиляция**
   - Кэширование результатов компиляции
   - Инкрементальное инстанцирование

3. **Подсказки об ошибках**
   - Чёткие сообщения об ошибках
   - Подсказки для вывода параметров обобщений

4. **Параллельная компиляция**
   - Параллельное инстанцирование обобщений
   - Многопоточное решение ограничений

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Только базовые обобщения | Не может заменить сложные макросы |
| Чистая макросистема | Нет типобезопасности, плохие сообщения об ошибках |
| Только ограничения | Недостаточная гибкость |
| Обобщения во время выполнения | Накладные расходы производительности |

### Риски

| Риск | Влияние | Меры по смягчению |
|------|---------|-------------------|
| Сложность решения ограничений | Слишком долгое время компиляции | Инкрементальное решение + кэширование |
| Раздувание кода | Слишком большой бинарный файл | DCE + контроль порогов |
| Сложность реализации | Увеличение срока разработки | Реализация по этапам |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

## Приложение

### БНФ грамматика

```bnf
# Параметры обобщений используют унифицированный синтаксис (), как часть функционального типа
# Например: map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в обобщённых параметрах)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Обобщённые параметры — первая группа параметров в функциональном типе: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённые типы, такие как List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Type в обобщённых параметрах автоматически заполняется компилятором из фактических аргументов
# Например: map(numbers, f), T извлекается из numbers: List(Int), R из f: (Int) -> String
```
