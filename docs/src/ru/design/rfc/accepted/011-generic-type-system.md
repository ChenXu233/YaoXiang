# RFC-011: Проектирование системы дженериков — Абстракции с нулевой стоимостью и замена макросов

## Аннотация

Настоящий документ определяет **проектирование системы дженериков** языка YaoXiang, реализующей абстракции с нулевой стоимостью за счёт мощных возможностей дженериков, использующей оптимизации времени компиляции для снижения зависимости от макросов и предоставляющей механизм устранения мёртвого кода.

**Ключевые решения**:
- **Единый синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — параметры дженериков и обычные параметры унифицированы
- **Механизм самоописания `Type`**: `Type` — особая сущность уровня языка; позиции `Type` в сигнатуре автоматически выводятся и заполняются
- **Ограничения типов**: `T: Dup + Add` — множественные ограничения, ограничения на функции
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Дженерики времени компиляции**: `N: Int` — параметризация значениями, инстанцирование константами времени компиляции
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Абстракции с нулевой стоимостью: мономорфизация во время компиляции, отсутствие накладных расходов во время выполнения
- Устранение мёртвого кода: анализ графа инстанцирования + оптимизации LLVM
- Замена макросов: дженерики покрывают 90% сценариев использования макросов
- Типобезопасность: проверки времени компиляции, дружелюбность к IDE
- **Явное лучше неявного**: самоописание `Type`, автоматический вывод компилятором

## Идея типовой вселенной и типы, зависящие от значений

Система дженериков YaoXiang опирается на **идею типовой вселенной** — ментальную модель, унифицирующую все концепции языка в иерархическую структуру, ключевая инновация которой состоит в возведении **типов, зависящих от значений**, в первоклассных граждан уровня Type2.

### Что такое типы, зависящие от значений?

**Тип, зависящий от значений** — это тип, зависящий от одного или нескольких **значений** (а не только от других типов). Эти значения могут быть вычислены во время компиляции, обеспечивая гарантии типобезопасности уже на этом этапе.

```yaoxiang
# Традиционные дженерики: параметр-тип
List: (T: Type) -> Type

# Тип, зависящий от значений: параметр-значение
Vec: (n: Int) -> Type  # тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # тип матрицы зависит от числа строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционными дженериками, типы, зависящие от значений, в YaoXiang обладают следующими ключевыми преимуществами:

| Свойство | Традиционные дженерики (C++/Rust) | YaoXiang: типы, зависящие от значений |
|----------|-----------------------------------|--------------------------------------|
| Значения, от которых зависит тип | Только параметры-типы | Любые значения, включая результаты вызовов функций |
| Вычисление во время компиляции | C++ — ручная специализация шаблонов; Rust — отсутствует | Автоматическое вычисление во время компиляции с гарантией завершаемости |
| Вычисления на уровне типов | Шаблонное метапрограммирование (сложно/опасно) | Единый движок вычислений на уровне типов |
| Типобезопасность | C++ — отсутствует; Rust — ограничена | Полная типобезопасность, проверки времени компиляции |
| Проверка размерностей | Проверка во время выполнения или ручная специализация | Проверка размерностей во время компиляции, без накладных расходов |

### Уровни типовой вселенной и типы, зависящие от значений

Идея типовой вселенной разделяет концепции языка по смысловым ролям на разные уровни. Типы, зависящие от значений, расположены на **уровне Type2**:

| Уровень | Роль | Примеры |
|---------|------|---------|
| Type-1 | Значения | `42`, `factorial(5)`, сама функция |
| Type0 | Ключевое слово метаязыка | `Type` |
| Type1 | Конкретные типы | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функции / конструкторы типов / типы, зависящие от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевое решение**: функции, конструкторы типов и типы, зависящие от значений, на уровне Type2 имеют **единый синтаксис** — форму `(params) -> result`:
- Обычная функция: `(Int, Int) -> Int` → результат — значение
- Конструктор типов: `(T: Type) -> Type` → результат — тип
- Тип, зависящий от значений: `(n: Int) -> Type` → результат — тип, но зависящий от параметров-значений

> **Изоморфизм Карри–Ховарда**: это единство — не совпадение. Изоморфизм Карри–Ховарда утверждает «тип — это утверждение, программа — это доказательство» — функциональный тип `A → B` соответствует логической импликации «если A, то B»; дженерик `(T: Type) -> Type` соответствует квантору всеобщности «для всех типов T»; тип, зависящий от значений, `(n: Int) -> Type` соответствует «для каждого целого n существует тип». YaoXiang объединяет функции, конструкторы типов и типы, зависящие от значений, на уровне Type2 — по сути, это объединение «доказательства» и «вычисления» в одно понятие — **конструктивное доказательство**. Это и есть прямое воплощение изоморфизма Карри–Ховарда в дизайне языка: единая форма `(params) -> result` одновременно несёт логическое утверждение и процесс вычисления.

### Гарантия детерминированности во время компиляции

Идея типовой вселенной YaoXiang требует: **всё на уровне Type детерминировано во время компиляции**.

```yaoxiang
# Пример проверки размерности во время компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка времени компиляции: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется во время компиляции
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Вычисление времени компиляции: factorial(3) = 6, размер вектора определяется во время компиляции
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов
2. Проверяет, помечена ли функция спецификацией `decreases` (см. механизм проверки завершаемости ниже)
3. Выполняет вычисление во время компиляции
4. Встраивает результат в генерируемый тип

### Сценарии применения типов, зависящих от значений

#### Проверка размерностей во время компиляции
```yaoxiang
# Умножение матриц: проверка соответствия размерностей во время компиляции
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка времени компиляции: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка перехватывается во время компиляции:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива — константа времени компиляции
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — константа времени компиляции, может использоваться в вычислениях на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно во время компиляции)
```

#### Условные типы
```yaoxiang
# Уровень типов: If
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейство типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры типов T, R определяются во время компиляции
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    return result
}

# При использовании полностью прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # выводится как map[Int, Int]
```

### Механизм проверки завершаемости (интеграция с RFC-022)

Вычисление во время компиляции для типов, зависящих от значений, должно быть **гарантированно завершаемым**, иначе система типов зациклится. YaoXiang обеспечивает это с помощью спецификации **`decreases`**, бесшовно интегрирующейся с RFC-022.

#### Спецификация завершаемости для рекурсивных функций
```yaoxiang
# Факториал времени компиляции: необходимо доказать завершаемость
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # при каждой рекурсии n строго убывает
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # вычисление времени компиляции: factorial(5) = 120
```

#### Спецификация завершаемости для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершаемости

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза проверки типов                                        │
│  Встречается вызов функции в позиции типа (например, Vec(factorial(5))) │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверить спецификацию `decreases`                       │
│     - Есть `decreases`: проверить выполнение условия убывания│
│       на всех путях рекурсии                                │
│     - Нет `decreases`, но очевидно завершается: вычислить напрямую│
│     - Нет `decreases` и может не завершиться: ошибка компиляции│
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисление во время компиляции (встроенным интерпретатором)│
│     - Чистая функция: вычислить напрямую                     │
│     - С побочными эффектами: ошибка компиляции (позиция типа │
│       должна быть без побочных эффектов)                    │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                            │
│     - Vec(factorial(5)) → Vec(120)                          │
│     - Matrix(Float, 3, 3) → конкретный тип                  │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантия завершения вычислений времени компиляции, защита от зацикливания системы типов
- **Единообразие**: проверка завершаемости и верификация частичной корректности используют единый механизм спецификаций
- **Постепенное усиление**: возможен плавный переход от проверок во время выполнения к полному статическому доказательству

## Мотивация

### Зачем нужна мощная система дженериков?

Существующие системы дженериков в主流 языках имеют ограничения:

| Язык | Возможности дженериков | Проблемы |
|------|------------------------|----------|
| Java | Ограниченные типы | Мономорфизация во время компиляции, без специализации |
| C# | Ограничения дженериков | Проверка типов во время выполнения, накладные расходы |
| Rust | Дженерики + Trait | Сложная система Trait, крутая кривая обучения |
| C++ | Шаблоны | Сложная специализация шаблонов, плохие сообщения об ошибках |
| **YaoXiang** | **Типы, зависящие от значений** | **Типы зависят от значений, проверка размерностей во время компиляции, гарантия завершаемости** |

### Ключевые противоречия

1. **Производительность vs гибкость**: гибкость во время выполнения vs оптимизация во время компиляции
2. **Сложность vs простота**: мощная система типов vs удобство использования
3. **Макросы vs дженерики**: кодогенерация макросами vs типобезопасность дженериков
4. **Зависимость от значений vs типобезопасность**: традиционные дженерики не могут проверять размерности во время компиляции

### Ключевые преимущества типов, зависящих от значений

**Типы, зависящие от значений** в YaoXiang — ключевое преимущество по сравнению с традиционными дженериками:

| Преимущество | Описание |
|--------------|----------|
| **Типы зависят от значений** | `Vec: (n: Int) -> Type` позволяет типу зависеть от конкретного значения |
| **Вычисление во время компиляции** | Вызовы функций в позициях типов вычисляются во время компиляции, результат встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` проверяет размерности матрицы во время компиляции |
| **Вычисления на уровне типов** | Условные типы `If`, `Match` поддерживают вычисления на уровне типов |
| **Гарантия завершаемости** | Спецификация `decreases` гарантирует завершение вычислений времени компиляции |

```yaoxiang
# Проверка во время компиляции, невозможная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# Вычисление времени компиляции: factorial(3) = 6, factorial(2) = 2
# Тип: Matrix(Float, 6, 2)

# Несоответствие размерностей перехватывается во время компиляции
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы дженериков

```yaoxiang
# Пример: единообразный дизайн API
# Операция map для разных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Подход с дженериками: одна функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели проектирования

### Ключевые цели

1. **Абстракции с нулевой стоимостью** — вызов дженерика эквивалентен вызову конкретного типа
2. **Устранение мёртвого кода** — анализ во время компиляции, инстанцирование только используемых дженериков
3. **Замена макросов** — дженерики покрывают 90% сценариев использования макросов
4. **Типобезопасность** — проверки во время компиляции, без накладных расходов на типы во время выполнения
5. **Дружелюбность к IDE** — интеллектуальные подсказки, ясные сообщения об ошибках
6. **Типы, зависящие от значений** — типы могут зависеть от значений, поддержка проверки размерностей во время компиляции
7. **Безопасность вычислений времени компиляции** — гарантия завершаемости через спецификацию `decreases`

### Принципы проектирования

- **Детерминированность во время компиляции** — параметры дженериков определяются во время компиляции
- **Приоритет мономорфизации** — генерация конкретного кода, отсутствие виртуальных вызовов
- **Управление через ограничения** — ограничения типов направляют инстанцирование
- **Платформенные оптимизации** — специализация для платформо-специфичных оптимизаций
- **Единство типовой вселенной** — функции / конструкторы типов / типы, зависящие от значений, объединены на уровне Type2
- **Гарантия завершаемости** — вызовы функций в позициях типов должны доказывать завершаемость

### 1. Базовые дженерики

#### 1.1 Параметры обобщённых типов

> **Ключевое правило**: определение обобщённого типа **должно явно аннотироваться `: Type`**, иначе оно будет выведено HM как функция.
>
> | Запись | Значение |
> |--------|----------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM выводит как функцию, не тип |

```yaoxiang
# Определение обобщённого типа (обязательно : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,   # self — лишь соглашение об имени, не ключевое слово
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM выводит как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Ограничение дженерика (прямое выражение, в одной строке return можно опустить)
clone: (T: Clone)(value: T) -> T = value.clone()

# Несколько параметров-типов
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 2.4 Встроенные marker-трейты: Dup и Clone

**Три категории семантики копирования**:

| Тип | Значение | Способ срабатывания | Сценарий применения |
|-----|----------|---------------------|---------------------|
| **Копирование примитивных значений** | При присваивании автоматическое копирование значения, два значения полностью независимы | Автоматически при присваивании/передаче параметра | Int, Float, Bool, Char |
| **Dup** | Поверхностное копирование: копирование дескриптора/токена, общие нижележащие данные | Автоматически при присваивании/передаче параметра | Токены `&T`, `ref T`, String/Bytes |
| **Clone** | Глубокое копирование: создание полной независимой копии | `value.clone()` | Любой тип, реализующий Clone |

**Семантика Dup**: типы, реализующие Dup, при присваивании/передаче не передают владение — компилятор копирует дескриптор/токен, и несколько владельцев указывают на одни и те же нижележащие данные. Это дополняет семантику Move по умолчанию из модели владения RFC-009.

**Dup и Clone — ортогональные понятия**:

```
Dup  = копирование дескриптора, общие данные (изменения влияют друг на друга)
Clone = копирование данных, копии независимы (изменения не влияют друг на друга)
```

**Правила**:

```
1. Примитивные типы значений (Int, Float, Bool, Char) — встроенное в компилятор копирование значений, не относятся к Dup
2. Dup  — применимо только к ссылочным/токен-типам и типам со встроенным подсчётом ссылок
3. Clone — явное глубокое копирование, может реализовать любой тип
4. Move по умолчанию — остальные типы сохраняют семантику Move по умолчанию
```

**Какие типы являются Dup**:

| Тип | Dup | Причина |
|-----|-----|---------|
| `&T` (токен заимствования) | ✅ | Токен нулевого размера, копирование токена = несколько представлений, указывающих на одни данные |
| `ref T` | ✅ | Копирование Rc/Arc = счётчик ссылок +1, общие данные в куче |
| String, Bytes | ✅ | Внутренний подсчёт ссылок, копирование дескриптора разделяет нижележащий буфер |
| `&mut T` (токен изменяемого заимствования) | ❌ | Линейно-исключительный, копировать нельзя |
| struct | Выводится | Все поля Dup → struct Dup |
| enum | Выводится | Все поля всех вариантов Dup → enum Dup |
| tuple | Выводится | Все элементы Dup → tuple Dup |
| Fn (замыкание) | ❌ | Захватываемая среда может быть не Dup |
| `*T` (сырой указатель) | ❌ | unsafe, не участвует в системе владения |

**Int/Float/Bool/Char не являются Dup** — это типы значений, при присваивании компилятор автоматически копирует значение (два значения полностью независимы). Это не «поверхностное копирование», а встроенная в компилятор обработка примитивов, которая не требует и не должна выражаться через атрибут типа Dup.

```yaoxiang
# Примитивные типы значений: автоматическое копирование значений компилятором (не Dup)
x: Int = 42
y = x          # Копирование значения, x и y полностью независимы
print(x)       # ✅

# Dup: поверхностное копирование, копирование дескриптора разделяет данные
view: &Point = &point
view2 = view    # ✅ Dup: копирование токена, оба указывают на одну и ту же point
print(view.x)   # ✅

# Clone: явное глубокое копирование, создание независимой копии
backup = big_struct.clone()  # Явный вызов

# Ограничения дженериков
dup_use: (T: Dup) -> T = x         # T: Dup → можно поверхностно копировать
clone_use: (T: Clone) -> T = x.clone()  # T: Clone → можно глубоко копировать
```

> **Примечание**: `Send`/`Sync` не являются видимыми пользователю трейтами. Безопасность передачи между задачами полностью обеспечивается ключевым словом `ref` и компилятором автоматически — `ref` автоматически выбирает Rc или Arc, пользователю не нужно понимать Send/Sync.

```yaoxiang
    }
}
```

### Never и Void: ⊥ и ⊤ в системе типов

Система типов YaoXiang в рамках изоморфизма Карри–Ховарда обладает одновременно ⊥ (ложь/пустой тип) и ⊤ (истина/Unit), представленными двумя встроенными именами типов: `Never` и `Void`:

**Never (⊥)** — три не подлежащих обсуждению свойства ядра:

1. **Нулевое число конструкторов**: ни один литерал или выражение не может породить значение типа `Never`. Это мета-свойство, оно должно быть встроенным.
2. **Принцип взрыва**: `Never <: T` выполняется для любого типа `T`. Значение `Never` может использоваться как значение любого типа — именно поэтому код после `assert(false)` всё ещё проходит проверку типов (хотя никогда не выполнится).
3. **Маркер расходимости**: `f: (...) -> Never` означает, что `f` гарантированно не возвращается. Компилятор использует это для анализа мёртвого кода.

`Never` — встроенное имя типа, не ключевое слово, парсер не замечает его как таковое. Синтаксис пустого типа и литералов типа не открывается.

**Void (⊤, т.е. Unit)** — ровно один обитатель (значение void по умолчанию), носитель истинного утверждения «всегда истинно». `Void` — единичный элемент пустого поля произведения, `Never` — единичный элемент пустого варианта суммы — они двойственны. `x: Void = <по умолчанию>` — допустимо, `x: Never = ...` — справа писать нечего.

#### 4.3 Верификация времени компиляции (реализация стандартной библиотеки)

```yaoxiang
# ════════════════════════════════════════════════════════
# Реализация стандартной библиотеки: использование условных типов
# ════════════════════════════════════════════════════════

# Определения стандартной библиотеки
# IsTrue: мост от вселенной значений к вселенной типов — истинность Bool отображается в тип
IsTrue: (b: Bool) -> Type = match b {
    true => Void,      # ⊤, имеет значение, программа продолжается
    false => Never,    # ⊥, нет значения, расходимость
}

# Assert: примитив уточнения типа во время компиляции — типовое выражение утверждения о Bool
Assert: (cond: Bool) -> Type = IsTrue(cond)
#
# cond = true  → Assert(true)  = Void    (всегда истинно, стирается)
# cond = false → Assert(false) = Never   (всегда ложно, ошибка компиляции / расходимость)
# cond неразрешимо → конвейер доказательства решает по режиму dispatch:
#                    CompileTime → Unknown, требуется prove
#                    Runtime     → вставка check, инъекция предположения в Γ

# Способ использования 1: в качестве ограничения в определении типа
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    # Проверка времени компиляции: N должно быть больше 0 (Assert в позиции типа)
    length: Assert(N > 0),
}

# Способ использования 2: в выражении
IntArray: (N: Int) -> Type = StaticArray(Int, N)
# Верификация: размер IntArray(10) равен sizeof(Int) * 10
Assert(size_of(IntArray(10)) == sizeof(Int) * 10)
```

### 5. Условные типы

> **Изоморфизм Карри–Ховарда**: с точки зрения Карри–Ховарда условные типы — это **case-анализ** в логике. Тип `Bool` соответствует утверждению с двумя возможными значениями (True/False), `If` в зависимости от истинности утверждения выбирает разные результаты — это и есть case-анализ дизъюнкции в логике. `match C { True => T, False => E }` фактически выражает: «утверждение C истинно — заключение T, утверждение C ложно — заключение E».

#### 5.1 Условный тип If

```yaoxiang
# Уровень типов: If
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление во время компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Верификация во время компиляции (унифицирована с определением Assert в §4.3)
# Assert: (cond: Bool) -> Type = IsTrue(cond)

# Использование
# Вычисление типа: If(True, Int, String) => Int
# Вычисление типа: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри–Ховарда**: семейства типов — наиболее прямое воплощение принципа «утверждение как тип». `Add: (A: Type, B: Type) -> Type` — это не «функция сложения на уровне типов», а **конструирование утверждения о сложении натуральных чисел**. `(Zero, B) => B` говорит: «утверждение Add(Zero, B) эквивалентно B»; `(Succ(A'), B) => Succ(Add(A', B))` говорит: «если Add(A', B) истинно, то Add(Succ(A'), B) также истинно». Это и есть само определение сложения в аксиомах Пеано. То, что проверка типов одобряет это выражение `match`, эквивалентно проверке логической согласованности этого определения.

```yaoxiang
# Преобразование типа во время компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # по умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Карри–Ховард: case-анализ + рекурсивный вызов, для полноты как
# полной индукции требуется проверка завершаемости)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: вычисление во время компиляции 2 + 3
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

#### 7.1 Анализ графа инстанцирования

```rust
struct InstantiationGraph {
    // Узлы: инстанцирования дженериков
    nodes: HashMap<InstanceKey, InstanceNode>,

    // Рёбра: отношения использования
    edges: HashMap<InstanceKey, Vec<InstanceKey>>,
}

    generic: FunctionId,  // ID обобщённой функции
    type_args: Vec<TypeId>,  // Параметры-типы
    const_args: Vec<ConstId>,  // Параметры-константы
}

// Алгоритм: анализ достижимости
fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Не посещённые инстанцирования — мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}
```

#### 9.1 Полный пример обобщённого контейнера

```yaoxiang
# ======== 1. Определение обобщённого контейнера ========
# Используется синтаксис (T: Type) -> Type
Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,
}

Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self,
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,

    # Обобщённые методы (T автоматически вводится во внешний List(T))
    push: (self: List(T), item: T) -> Void,
    pop: (self: List(T)) -> Option(T),
    map: (R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)),
    filter: (self: List(T), predicate: (T) -> Bool) -> List(T),
    fold: (U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U),
}

# ======== 2. Реализация обобщённых методов ========
# Определение функций в пространстве имён List (префикс List. = принадлежность пространству имён)
# Чтобы вызов через . (например, list.push(item)) работал, требуется явная привязка: List.push = push[0]
# self — лишь соглашение об имени параметра, компилятор смотрит на тип, а не на имя

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

List.pop: (T: Type) -> ((self: List(T)) -> Option(T)) = {
    if self.length > 0 {
        self.length = self.length - 1
        return Option.some(self.data[self.length])
    } else {
        return Option.none()
    }
}

List.map: (T: Type, R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)) = {
    result = List(R)()
    for i in 0..self.length {
        result.push(f(self.data[i]))
    }
    return result
}

List.filter: (T: Type) -> ((self: List(T), predicate: (T) -> Bool) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        if predicate(self.data[i]) {
            result.push(self.data[i])
        }
    }
    return result
}

List.fold: (T: Type, U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U) = {
    result = initial
    for i in 0..self.length {
        result = f(result, self.data[i])
    }
    return result
}

# ======== 3. Использование ограничений типов ========
# Реализация Clone для List
List.clone: (T: Clone) -> ((self: List(T)) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        result.push(self.data[i].clone())
    }
    return result
}
```

#### 9.2 Пример обобщённого алгоритма

```yaoxiang
# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённая быстрая сортировка
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Реализация через перегрузку функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========
# Сортировка массива Int
numbers = Array(Int)(3, 1, 4, 1, 5, 9, 2, 6)
sorted = quicksort(numbers, Comparator(Int)())

# Сортировка массива String (требуется StringComparator)
strings = Array(String)("hello", "world", "foo", "bar")
sorted_strings = quicksort(strings, Comparator(String)())
```

#### 9.3 Пример дженериков времени компиляции

```yaoxiang
# ======== 1. Тип матрицы времени компиляции ========
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),

    # Проверка размерностей во время компиляции: с использованием типа Assert из стандартной библиотеки
    _assert: Assert[Rows > 0],  # Rows > 0, иначе ошибка компиляции
    _assert: Assert[Cols > 0],  # Cols > 0, иначе ошибка компиляции

    # Операции над матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {
            for j in 0..M {
                sum = Zero::zero()
                for k in 0..Cols {
                    sum = sum + self.data[i][k] * other.data[k][j]
                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

# ======== 2. Создание матрицы во время компиляции ========
identity: (T: Add + Multiply + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    for i in 0..N {
        for j in 0..N {
            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    return matrix
}

# ======== 3. Примеры использования ========
# Создание матрицы с размером, известным во время компиляции
# Матрица 2x3
matrix_2x3 = Matrix(Float, 2, 3)()
matrix_2x3.data[0][0] = 1.0
matrix_2x3.data[0][1] = 2.0
matrix_2x3.data[0][2] = 3.0
matrix_2x3.data[1][0] = 4.0
matrix_2x3.data[1][1] = 5.0
matrix_2x3.data[1][2] = 6.0

# Матрица 3x2
matrix_3x2 = Matrix(Float, 3, 2)()
matrix_3x2.data[0][0] = 7.0
matrix_3x2.data[0][1] = 8.0
matrix_3x2.data[1][0] = 9.0
matrix_3x2.data[1][1] = 10.0
matrix_3x2.data[2][0] = 11.0
matrix_3x2.data[2][1] = 12.0

# Умножение матриц: 2x3 * 3x2 = 2x2
result = matrix_2x3.multiply(matrix_3x2)

# Проверка во время компиляции: тип result — Matrix(Float, 2, 2)
# Единичная матрица 2x2
identity_3x3 = identity(Float, 3)()

# Несоответствие размерностей: ошибка компиляции
# bad_multiply = matrix_2x3.multiply(identity_3x3)  # Ошибка компиляции: 3x3 != 2x3
```

## Компромиссы

### Преимущества

1. **Абстракции с нулевой стоимостью**
   - Мономорфизация во время компиляции, отсутствие накладных расходов во время выполнения
   - Не требуются виртуальные функции или RTTI

2. **Устранение мёртвого кода**
   - Анализ во время компиляции, инстанцирование только используемых дженериков
   - Управляемое раздувание кода

3. **Замена макросов**
   - Типобезопасная кодогенерация
   - Дружелюбность к IDE, ясные сообщения об ошибках

4. **Вычисления во время компиляции**
   - Дженерики времени компиляции поддерживают вычисления во время компиляции
   - Возможности вроде проверки размерностей
   - Не требуется ключевое слово `const`, чистые ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанцирование дженериков увеличивает время компиляции
   - Разрешение ограничений может быть медленным

2. **Потребление памяти**
   - Увеличение потребления памяти компилятором
   - Механизмы кэширования требуют памяти

3. **Сложность реализации**
   - Сложный решатель ограничений
   - Сложный движок вычислений на уровне типов

4. **Диагностика ошибок**
   - Ошибки дженериков могут быть сложными
   - Необходимы ясные подсказки об ошибках

### Меры по смягчению

1. **Стратегия кэширования**
   - Кэширование результатов инстанцирования
   - Ограничение памяти LRU-кэшем

2. **Инкрементальная компиляция**
   - Кэширование результатов компиляции
   - Инкрементальное инстанцирование

3. **Сообщения об ошибках**
   - Ясные сообщения об ошибках
   - Подсказки по выводу параметров дженериков

4. **Параллельная компиляция**
   - Параллельное инстанцирование дженериков
   - Многопоточное разрешение ограничений

## Альтернативы

| Подход | Почему не выбран |
|--------|------------------|
| Только базовые дженерики | Не может заменить сложные макросы |
| Чисто макросистема | Нет типобезопасности, плохие сообщения об ошибках |
| Только ограничения | Недостаточная гибкость |
| Дженерики времени выполнения | Накладные расходы на производительность |

### Риски

| Риск | Влияние | Меры по смягчению |
|------|---------|-------------------|
| Сложность разрешения ограничений | Чрезмерное время компиляции | Инкрементальное разрешение + кэширование |
| Раздувание кода | Чрезмерный размер бинарника | DCE + управление пороговыми значениями |
| Сложность реализации | Удлинение цикла разработки | Поэтапная реализация |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

```bnf
# Параметры дженерика используют единый синтаксис () как часть функционального типа
# Например, map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в параметрах дженерика)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Параметры дженерика — первая группа параметров в функциональном типе: (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (единый синтаксис Binding)
# Обобщённый тип, например List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Параметр Type в дженериках автоматически заполняется компилятором из типов фактических аргументов
# Например, map(numbers, f), T извлекается из numbers: List(Int), R — из f: (Int) -> String
```
