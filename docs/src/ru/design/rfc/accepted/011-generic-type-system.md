# RFC-011: Проектирование системы обобщённых типов — Абстракции с нулевой стоимостью и замена макросов

> **Статус**: Принято
> **Автор**: Чэнь Сюй
> **Дата создания**: 2025-01-25
> **Последнее обновление**: 2026-04-22 (обновлено до механизма самодокументирования Type, унифицирован синтаксис обобщённых вызовов)

## Краткое содержание

В данном документе определяется **проектирование системы обобщённых типов** языка YaoXiang, обеспечивающей нулевую стоимость абстракций через мощные возможности обобщённого программирования, использование оптимизаций на этапе компиляции для снижения зависимости от макросов и предоставление механизма устранения мёртвого кода.

**Основные проектные решения**:
- **Унифицированный синтаксис сигнатур**: `(T: Type, R: Type) -> ...` — обобщённые параметры унифицированы с обычными параметрами
- **Механизм самодокументирования Type**: `Type` является языковой конструкцией особого рода; позиции `Type` в сигнатурах автоматически заполняются выводом типов
- **Ограничения типов**: `T: Clone + Add` — множественные ограничения, ограничения на типы функций
- **Ассоциированные типы**: `Iterator: (Item: Type) -> Type = { next: () -> Option(Item), has_next: () -> Bool }`
- **Обобщённые типы на этапе компиляции**: `N: Int` — обобщённые параметры-значения, инстанцирование на этапе компиляции
- **Условные типы**: `If: (C: Bool, T: Type, E: Type) -> Type` — вычисления на уровне типов, семейства типов

**Ценность**:
- Нулевая стоимость абстракций: мономорфизация на этапе компиляции, отсутствие накладных расходов во время выполнения
- Устранение мёртвого кода: анализ графа инстанцирования + оптимизация LLVM
- Замена макросов: обобщённые типы покрывают 90% сценариев использования макросов
- Типобезопасность: проверка на этапе компиляции, дружественность к IDE
- **Явное лучше неявного**: самодокументирование `Type`, автоматический вывод компилятором

## Концепция тип-универсума и типы, зависящие от значений

Система обобщённых типов YaoXiang построена на основе **концепции тип-универсума** — ментальной модели, объединяющей все концепции языка в иерархическую структуру. Ключевое нововведение заключается в повышении **типов, зависящих от значений**, до статуса полноправных граждан на уровне Type2.

### Что такое типы, зависящие от значений?

**Тип, зависящий от значений** — это тип, который зависит от одного или нескольких **значений** (а не только от других типов). Эти значения могут вычисляться на этапе компиляции, что обеспечивает гарантии типобезопасности уже на этапе компиляции.

```yaoxiang
# Традиционные обобщённые типы: параметры-типы
List: (T: Type) -> Type

# Типы, зависящие от значений: параметры-значения
Vec: (n: Int) -> Type  # Тип вектора зависит от значения длины n
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type  # Тип матрицы зависит от числа строк и столбцов
```

### Ключевые преимущества типов, зависящих от значений

По сравнению с традиционными обобщёнными типами, типы YaoXiang, зависящие от значений, обладают следующими ключевыми преимуществами:

| Характеристика | Традиционные обобщённые типы (C++/Rust) | YaoXiang: типы, зависящие от значений |
|----------------|----------------------------------------|---------------------------------------|
| Значения, от которых зависит тип | Только параметры-типы | Любые значения, включая результаты вызовов функций |
| Вычисления на этапе компиляции | Ручная специализация шаблонов C++, отсутствует в Rust | Автоматические вычисления на этапе компиляции с гарантией завершения |
| Вычисления на уровне типов | Метапрограммирование шаблонов (сложно/опасно) | Единый движок вычислений на уровне типов |
| Типобезопасность | C++ — отсутствует, Rust — ограниченная | Полная типобезопасность, проверки на этапе компиляции |
| Проверка размерностей | Проверки во время выполнения или ручная специализация | Проверка размерностей на этапе компиляции, без накладных расходов во время выполнения |

### Уровни тип-универсума и типы, зависящие от значений

Концепция тип-универсума разделяет концепции языка по семантическим ролям на различные уровни; типы, зависящие от значений, находятся на **уровне Type2**:

| Уровень | Роль | Пример |
|---------|------|--------|
| Type-1 | Значения | `42`, `factorial(5)`, сами функции |
| Type0 | Ключевое слово метатипа | `Type` |
| Type1 | Конкретные типы | `Int`, `String`, `Vec(3)` |
| **Type2** | **Функции/конструкторы типов/типы, зависящие от значений** | `add: (Int, Int) -> Int`, `Vec: (n: Int) -> Type`, `Matrix: (T: Type, Rows: Int, Cols: Int) -> Type` |

**Ключевое проектное решение**: функции, конструкторы типов и типы, зависящие от значений, на уровне Type2 имеют **унифицированный синтаксис** — все они представляют собой форму `(params) -> result`:
- Обычные функции: `(Int, Int) -> Int` → возвращают значение
- Конструкторы типов: `(T: Type) -> Type` → возвращают тип
- Типы, зависящие от значений: `(n: Int) -> Type` → возвращают тип, но зависят от параметров-значений

> **Изоморфизм Карри — Ховарда**: это единство не случайно. Изоморфизм Карри — Ховарда устанавливает, что «типы — это предложения, программы — это доказательства». Функциональный тип `A → B` соответствует логическому импликации «если A, то B», обобщённый тип `(T: Type) -> Type` соответствует универсальной квантификации «для всех типов T», а тип, зависящий от значений, `(n: Int) -> Type` соответствует «для каждого целого n существует тип». Унификация функций, конструкторов типов и типов, зависящих от значений, на уровне Type2 в YaoXiang по сути объединяет «доказательства» и «вычисления» в одну концепцию — **конструктивные доказательства**. Это прямое воплощение изоморфизма Карри — Ховарда в дизайне языка: одна форма (`(params) -> result`) одновременно выражает логическое предложение и вычислительный процесс.

### Гарантии компиляционной определённости

Концепция тип-универсума YaoXiang требует: **всё на уровне Type является компиляционно определённым**.

```yaoxiang
# Пример проверки размерностей на этапе компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
    # Проверка на этапе компиляции: размерности должны быть положительными
    _assert: Assert[Rows > 0],
    _assert: Assert[Cols > 0],
}

# Создание единичной матрицы 3x3 — выполняется на этапе компиляции
identity: (T: Add + Zero + One, N: Int) -> ((size: N) -> Matrix(T, N, N)) = {
    matrix = Matrix(T, N, N)()
    # ...
}

# Компиляционные вычисления: factorial(3) = 6, размер вектора определяется на этапе компиляции
vec: Vec(factorial(3)) = Vec(6)()
```

Компилятор автоматически:
1. Обнаруживает вызовы функций в позициях типов
2. Проверяет наличие редукции decreases для функций (см. механизм проверки завершения ниже)
3. Выполняет вычисления на этапе компиляции
4. Встраивает результаты в генерируемые типы

### Сценарии применения типов, зависящих от значений

#### Проверка размерностей на этапе компиляции
```yaoxiang
# Умножение матриц: проверка соответствия размерностей на этапе компиляции
multiply: (T: Add + Multiply + Zero,
           Rows: Int, Cols: Int, M: Int) -> ((
    a: Matrix(T, Rows, Cols),
    b: Matrix(T, Cols, M)
) -> Matrix(T, Rows, M)) = {
    # Проверка на этапе компиляции: a.Cols == b.Rows, иначе ошибка компиляции
    result = Matrix(T, Rows, M)()
    # ...
}

# Ошибка перехватывается на этапе компиляции:
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: 2 != 4
```

#### Типобезопасный размер массива
```yaoxiang
# Размер массива является компиляционной константой
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: N,
}

# N — компиляционная константа, может использоваться для вычислений на уровне типов
first_three: Array(Int, 3) = Array(Int, 3)(1, 2, 3)
# first_three.length == 3 (известно на этапе компиляции)
```

#### Условные типы
```yaoxiang
# Типовый If
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Семейство типов
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,
}
```

#### Обобщённые функции
```yaoxiang
# map: обобщённая функция, параметры-типы T, R определяются на этапе компиляции
map: (T: Type, R: Type) -> (
    (list: List(T), f: (x: T) -> R) -> List(R)
) = (list, f) => {
    result = List(R)()
    for x in list {
        result.push(f(x))
    }
    result
}

# Использование полностью прозрачно, типы выводятся автоматически
numbers = List(1, 2, 3)
doubled = map(numbers, (x) => x * 2)  # Выводится как map[Int, Int]
```

### Механизм проверки завершения (интеграция с RFC-022)

Вычисления на этапе компиляции для типов, зависящих от значений, должны **гарантированно завершаться**, иначе система типов войдёт в бесконечный цикл. YaoXiang обеспечивает это через **редукцию decreases**, которая бесшовно интегрирована с RFC-022.

#### Редукция завершения для рекурсивных функций
```yaoxiang
# Компиляционный факториал: необходимо доказать завершение
factorial: (n: Int) -> Int = {
    //! requires: n >= 0
    //! ensures: result == n!
    //! decreases: n    # При каждой рекурсии n строго убывает
    if n <= 1 { return 1 }
    return n * factorial(n - 1)
}

# Использование: вызов в позиции типа
vec: Vec(factorial(5)) = Vec(120)()  # factorial(5) = 120 вычисляется на этапе компиляции
```

#### Редукция завершения для циклов
```yaoxiang
sum: (arr: Array(Int, n)) -> Int = {
    s = 0; i = 0
    while i < n {
        /*! invariant: s == sum(arr[0..i]) && 0 <= i <= n !*/
        /*! decreases: n - i !*/
        s += arr[i]; i += 1
    }
    return s
}
```

#### Рабочий процесс проверки завершения

```
┌─────────────────────────────────────────────────────────────┐
│  Этап проверки типов                                         │
│  Встречен вызов функции в позиции типа (например, Vec(factorial(5)))│
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Проверка редукции decreases                              │
│     - Есть decreases: верификация убывания на всех путях      │
│     - Нет decreases, но завершение очевидно: прямое вычисление│
│     - Нет decreases и возможно незавершение: ошибка компиляции│
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Вычисления на этапе компиляции (выполняются встроенным   │
│     интерпретатором)                                          │
│     - Чистые функции: прямое вычисление                      │
│     - Побочные эффекты: ошибка компиляции (позиции типов      │
│       должны быть без побочных эффектов)                      │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Встраивание результата в тип                               │
│     - Vec(factorial(5)) → Vec(120)                           │
│     - Matrix(Float, 3, 3) → конкретный тип                   │
└─────────────────────────────────────────────────────────────┘
```

#### Преимущества

- **Безопасность**: гарантия завершения вычислений на этапе компиляции, предотвращение бесконечных циклов в системе типов
- **Единство**: проверка завершения и верификация частичной корректности используют общий механизм спецификаций
- **Постепенное улучшение**: возможность перехода от проверок во время выполнения к полностью статическому доказательству

## Мотивация

### Почему нужна мощная система обобщённых типов?

Текущие mainstream-языки имеют ограничения в обобщённом программировании:

| Язык | Возможности обобщённых типов | Проблема |
|------|------------------------------|----------|
| Java | Ограниченные типы | Компиляционная мономорфизация, без специализации обобщённых типов |
| C# | Ограничения обобщённых типов | Проверка типов во время выполнения, накладные расходы |
| Rust | Обобщённые типы + Trait | Сложная система Trait, крутая кривая обучения |
| C++ | Шаблоны | Сложная специализация шаблонов, плохие сообщения об ошибках |
| **YaoXiang** | **Типы, зависящие от значений** | **Типы могут зависеть от значений, проверка размерностей на этапе компиляции, гарантия завершения** |

### Ключевые противоречия

1. **Производительность vs гибкость**: гибкость во время выполнения vs оптимизации на этапе компиляции
2. **Сложность vs простота**: мощная система типов vs удобство использования
3. **Макросы vs обобщённые типы**: генерация кода макросами vs типобезопасность обобщённых типов
4. **Зависимость от значений vs типобезопасность**: традиционные обобщённые типы не могут проверять размерности на этапе компиляции

### Ключевые преимущества типов, зависящих от значений

**Типы, зависящие от значений** — ключевое преимущество YaoXiang перед традиционными обобщёнными типами:

| Преимущество | Описание |
|--------------|----------|
| **Типы зависят от значений** | `Vec: (n: Int) -> Type` позволяет типу зависеть от конкретного значения |
| **Вычисления на этапе компиляции** | Вызовы функций в позициях типов вычисляются на этапе компиляции, результат встраивается в тип |
| **Проверка размерностей** | `Matrix(Float, 3, 3)` проверяет размерности матрицы на этапе компиляции |
| **Типовые вычисления** | `If`, `Match` и другие условные типы поддерживают вычисления на уровне типов |
| **Гарантия завершения** | Редукция decreases обеспечивает обязательное завершение вычислений на этапе компиляции |

```yaoxiang
# Проверка на этапе компиляции, недоступная в C++/Rust
matrix: Matrix(Float, factorial(3), factorial(2)) = ...
# factorial(3) = 6, factorial(2) = 2 вычисляются на этапе компиляции
# Тип: Matrix(Float, 6, 2)

# Несоответствие размерностей перехватывается на этапе компиляции
identity: Matrix(Float, 3, 3) = ...
# multiply(matrix_2x3, identity_3x3)  # Ошибка компиляции: 2 != 3
```

### Ценность системы обобщённых типов

```yaoxiang
# Пример: унифицированный дизайн API
# Операция map для различных типов контейнеров

# Традиционный подход: отдельная реализация для каждого типа
map_int_array: (array: Array(Int), f: Fn(Int) -> Int) -> Array(Int) = ...
map_string_array: (array: Array(String), f: Fn(String) -> String) -> Array(String) = ...
map_int_list: (list: List(Int), f: Fn(Int) -> Int) -> List(Int) = ...
map_string_list: (list: List(String), f: Fn(String) -> String) -> List(String) = ...

# Обобщённый подход: одна обобщённая функция покрывает все типы
map: (T: Type, R: Type)(container: Container(T), f: Fn(T) -> R) -> Container(R) = {
    for item in container {
        result.push(f(item))
    }
    result
}
```

## Цели проектирования

### Основные цели

1. **Нулевая стоимость абстракций** — обобщённые вызовы эквивалентны вызовам конкретных типов
2. **Устранение мёртвого кода** — анализ на этапе компиляции, инстанцируются только используемые обобщённые типы
3. **Замена макросов** — обобщённые типы покрывают 90% сценариев использования макросов
4. **Типобезопасность** — проверки на этапе компиляции, без накладных расходов на проверку типов во время выполнения
5. **Дружественность к IDE** — интеллектуальные подсказки, чёткие сообщения об ошибках
6. **Типы, зависящие от значений** — типы могут зависеть от значений, поддержка проверки размерностей на этапе компиляции
7. **Безопасность вычислений на этапе компиляции** — гарантия завершения вычислений на этапе компиляции через редукцию decreases

### Принципы проектирования

- **Компиляционная определённость**: параметры обобщённых типов определяются на этапе компиляции
- **Приоритет мономорфизации**: генерация конкретного кода, избегание вызовов виртуальных функций
- **Ограничения驱动**: ограничения типов направляют инстанцирование
- **Оптимизация под платформу**: специализация поддерживает оптимизацию под конкретную платформу
- **Единство тип-универсума**: функции/конструкторы типов/типы, зависящие от значений, унифицированы на уровне Type2
- **Гарантия завершения**: вызовы функций в позициях типов должны доказать завершение

## Предложение

### 1. Базовые обобщённые типы

#### 1.1 Параметры обобщённых типов

> **Ключевое правило**: определения обобщённых типов **должны явно содержать аннотацию `: Type`**, иначе HM выведет их как функции.
>
> | Запись | Значение |
> |--------|----------|
> | `List: (T: Type) -> Type = {...}` | ✅ Конструктор типа |
> | `List = {...}` | ❌ HM выводит как функцию, не тип |

```yaoxiang
# Определение обобщённого типа (обязательно : Type)
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self
}

List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (self: List(T), item: T) -> Void,
    get: (self: List(T), index: Int) -> Option(T),
}

# Обобщённая функция (без : Type, HM выводит как функцию)
map: (T: Type, R: Type) -> ((opt: Option(T), f: Fn(T) -> R) -> Option(R)) = {
    return match opt {
        some => Option.some(f(some)),
        none => Option.none(),
    }
}

# Обобщённое ограничение (прямое выражение, return можно опустить в однострочном)
clone: (T: Clone)(value: T) -> T = value.clone()

# Множественные параметры типов
combine: (T: Type, U: Type) -> ((a: T, b: U) -> (T, U)) = (a, b)
```

#### 1.2 Механизм самодокументирования Type

`Type` является языковой конструкцией особого рода; компилятор естественным образом распознаёт позиции `Type` в сигнатурах и автоматически заполняет их выводом из фактических типов аргументов.

```yaoxiang
# Компилятор автоматически выводит обобщённые параметры
numbers: List(Int) = List(Int)
#         ^^^^^^^^   ^^^^^^
#         объявление типа   вызов конструктора: Int заполняет T

# Вывод при вызове функции
numbers: List(Int) = List(Int)
f: (x: Int) -> String = (x) => x.to_string()
strings: List(String) = map(numbers, f)
# Компилятор выводит: T=Int, R=String
```

#### 1.3 Мономорфизация

```yaoxiang
# Исходный код
map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R)) = {
    result: List(R) = List(R)()
    for x in list {
        result.push(f(x))
    }
    return result
}

# Точки использования
int_list: List(Int) = List(Int)
doubled: List(Int) = map(int_list, (x: Int) => x * 2)  # Инстанцирование map[Int, Int]

string_list: List(String) = List(String)
uppercased: List(String) = map(string_list, (s: String) => s.to_uppercase())  # Инстанцирование map[String, String]

# После компиляции (эквивалентный код)
map_Int_Int: (list: List(Int), f: (Int) -> Int) -> List(Int) = {
    result: List(Int) = List(Int)
    for x in list {
        result.push(f(x))
    }
    return result
}

map_String_String: (list: List(String), f: (String) -> String) -> List(String) = {
    result: List(String) = List(String)
    for s in list {
        result.push(f(s))
    }
    return result
}
```

### 3. Ассоциированные типы

#### 3.1 Определение ассоциированных типов

```yaoxiang
# Iterator trait (использует синтаксис (Item: Type) -> Type)
Iterator: (Item: Type) -> Type = {
    next: (Self) -> Option(Item),
    has_next: (Self) -> Bool,
    collect: (T: Type)(Self) -> List(T),
}

# Использование
collect_all: (T: Type, I: Iterator(T))(iter: I) -> List(T) = {
    result: List(T) = List(T)
    while iter.has_next() {
        if let Some(item) = iter.next() {
            result.push(item)
        }
    }
    return result
}

# Реализация Iterator для Array
# Используется синтаксический сахар методов: Array.Item, Array.next, Array.has_next
Array.has_next: (T: Type)(self: Array(T)) -> Bool = {
    return self.index < self.length
}

Array.next: (T: Type)(self: Array(T)) -> Option(T) = {
    if has_next(self) {
        item = self.data[self.index]
        self.index = self.index + 1
        return Option.some(item)
    } else {
        return Option.none()
    }
}

Array.Item: (T: Type)(arr: Array(T)) -> T = {
    return arr.data[0]
}
```

### 4. Обобщённые типы на этапе компиляции

#### 4.1 Параметры-константы на этапе компиляции

**Основное проектное решение**: маркер `Type` в обобщённых сигнатурах обозначает параметры типов на этапе компиляции, параметры-значения типа `Int` по умолчанию определяются на этапе компиляции в обобщённом контексте. Ключевое слово `const` не требуется.

```yaoxiang
# ════════════════════════════════════════════════════════
# Параметры-константы на этапе компиляции: Int по умолчанию
# определяется на этапе компиляции в обобщённом контексте
# ════════════════════════════════════════════════════════

# Компиляционный факториал: N должен быть литералом, известным
# на этапе компиляции
factorial: (N: Int) -> (n: N) -> Int = {
    match n {
        0 => 1,
        _ => n * factorial(n - 1)
    }
}

# Компиляционное сложение
add: (a: Int, b: Int) -> (a: a, b: b) -> Int = a + b

# ════════════════════════════════════════════════════════
# Массивы констант на этапе компиляции
# ════════════════════════════════════════════════════════
StaticArray: (T: Type, N: Int) -> Type = {
    data: Array(T, N),  # Массив с размером, известным на этапе компиляции
    length: N,
}

# Способ использования
arr: StaticArray(Int, factorial(5))  # StaticArray(Int, 120), компилятор вычисляет на этапе компиляции
```

identity_matrix: (T: Add + Zero + One, N: Int)(size: N) -> Matrix(T, N, N) = {
    matrix: Matrix(T, N, N) = Matrix(T, N, N)()
    for i in 0..size {
        for j in 0..size {
            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    matrix
}

identity_3x3: Matrix(Float, 3, 3) = identity_matrix(Float, 3)(3)

### 5. Условные типы

> **Изоморфизм Карри — Ховарда**: условные типы с точки зрения Карри — Ховарда представляют собой **case-анализ** в логике. Тип `Bool` соответствует утверждению с двумя возможными значениями (True/False), `If` выбирает различные результаты в зависимости от истинности этого утверждения — это именно дизъюнктивный case в логике. `match C { True => T, False => E }` фактически выражает: «когда известно, что утверждение C истинно, заключение T; когда C ложно, заключение E».

#### 5.1 Условный тип If

```yaoxiang
# Типовый If
If: (C: Bool, T: Type, E: Type) -> Type = match C {
    True => T,
    False => E,
}

# Пример: ветвление на этапе компиляции
NonEmpty: (T: Type) -> Type = If(T != Void, T, Never)

Optional: (T: Type) -> Type = If(T != Void, T, Void)

# Верификация на этапе компиляции
Assert: (C: Bool) -> Type = match C {
    True => Void,
    False => compile_error("Assertion failed"),
}

# Использование
# Типовое вычисление: If(True, Int, String) => Int
# Типовое вычисление: If(False, Int, String) => String
```

#### 5.2 Семейства типов

> **Изоморфизм Карри — Ховарда**: семейства типов — наиболее прямое воплощение «предложения как типы». `Add: (A: Type, B: Type) -> Type` — это не «функция сложения, записанная на уровне типов», а **конструирование утверждения о сложении натуральных чисел**. `(Zero, B) => B` означает «утверждение Add(Zero, B) эквивалентно B», `(Succ(A'), B) => Succ(Add(A', B))` означает «если Add(A', B) верно, то Add(Succ(A'), B) тоже верно». Это само определение сложения из аксиом Пеано. Проверка типами этого match-выражения эквивалентна верификации логической согласованности данного определения.

```yaoxiang
# Преобразование типов на этапе компиляции
AsString: (T: Type) -> Type = match T {
    Int => String,
    Float => String,
    Bool => String,
    _ => String,  # По умолчанию
}

# Вычисления на уровне типов
Length: (T: Type) -> Type = match T.length {
    0 => Zero,
    1 => Succ(Zero),
    2 => Succ(Succ(Zero)),
    _ => TooLong,
}

# Сложение на уровне типов (Curry-Howard: это также индуктивное
# определение сложения натуральных чисел)
Add: (A: Type, B: Type) -> Type = match (A, B) {
    (Zero, B) => B,
    (Succ(A'), B) => Succ(Add(A', B)),
}

# Пример: вычисление 2 + 3 на этапе компиляции
Two: Type = Succ(Succ(Zero))
Three: Type = Succ(Succ(Succ(Zero)))
Five: Type = Add[Two, Three]  # Succ(Succ(Succ(Succ(Succ(Zero)))))
```

### 6. Специализация через перегрузку функций

#### 6.1 Базовая специализация

#### 6.3 Идеальное сочетание перегрузки функций и встраивания

**Ключевая характеристика**: перегрузка функций естественно сочетается с оптимизацией встраивания, реализуя абстракции с нулевой стоимостью.

```yaoxiang
# ======== Исходный код ========
sum: (arr: Array(Int)) -> Int = {
    return native_sum_int(arr.data, arr.length)
}

sum: (arr: Array(Float)) -> Float = {
    return simd_sum_float(arr.data, arr.length)
}

sum: (T: Type) -> ((arr: Array(T)) -> T) = {
    result = Zero::zero()
    for item in arr {
        result = result + item
    }
    return result
}

# Использование
int_arr = Array(Int)(1, 2, 3, 4, 5)
result = sum(int_arr)

# ======== После компиляции (эквивалентный код) ========
# Компилятор автоматически выбирает оптимальную специализацию,
# затем выполняет встраивание
result = native_sum_int(int_arr.data, int_arr.length)

# Полностью эквивалентно оптимизированному коду, написанному вручную!
```

**Ключевые преимущества**:

### 7. Механизм устранения мёртвого кода

#### 7.1 Анализ графа инстанцирования

```rust
// Внутреннее представление компилятора: построение графа зависимостей
// инстанцирования обобщённых типов
struct InstantiationGraph {
    // Узлы: инстанцирования обобщённых типов
    nodes: HashMap<InstanceKey, InstanceNode>,

    // Рёбра: связи использования
    edges: HashMap<InstanceKey, Vec<InstanceKey>>,
}

struct InstanceKey {
    generic: FunctionId,  // ID обобщённой функции
    type_args: Vec<TypeId>,  // Параметры-типы
    const_args: Vec<ConstId>,  // Параметры Const
}

// Алгоритм: анализ достижимости
fn eliminate_dead_instantiations(graph: &InstantiationGraph) {
    let mut reachable = HashSet::new();

    // Начинаем с точек входа (main, экспортируемые функции и т.д.)
    let entry_points = find_entry_points();
    for entry in entry_points {
        dfs_visit(entry, &graph, &mut reachable);
    }

    // Непосещённые инстанцирования — это мёртвый код
    for node in &graph.nodes {
        if !reachable.contains(node.key) {
            eliminate(node);
        }
    }
}
```

    length: Int,

    # Обобщённые методы (T автоматически вводится из внешнего List(T))
    push: (self: List(T), item: T) -> Void,
    pop: (self: List(T)) -> Option(T),
    map: (R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)),
    filter: (self: List(T), predicate: (T) -> Bool) -> List(T),
    fold: (U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U),
}

# ======== 2. Реализация обобщённых методов ========
# Использование синтаксического сахара Type.method:
# автоматическая ассоциация с типом List

List.push: (T: Type) -> ((self: List(T), item: T) -> Void) = {
    if self.length >= self.data.length {
        # Расширение ёмкости
        new_data = Array(T)(self.data.length * 2)
        for i in 0..self.length {
            new_data[i] = self.data[i]
        }
        self.data = new_data
    }
    self.data[self.length] = item
    self.length = self.length + 1
}

List.pop: (T: Type) -> ((self: List(T)) -> Option(T)) = {
    if self.length > 0 {
        self.length = self.length - 1
        return Option.some(self.data[self.length])
    } else {
        return Option.none()
    }
}

List.map: (T: Type, R: Type) -> ((self: List(T), f: (T) -> R) -> List(R)) = {
    result = List(R)()
    for i in 0..self.length {
        result.push(f(self.data[i]))
    }
    return result
}

List.filter: (T: Type) -> ((self: List(T), predicate: (T) -> Bool) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        if predicate(self.data[i]) {
            result.push(self.data[i])
        }
    }
    return result
}

List.fold: (T: Type, U: Type) -> ((self: List(T), initial: U, f: (U, T) -> U) -> U) = {
    result = initial
    for i in 0..self.length {
        result = f(result, self.data[i])
    }
    return result
}

# ======== 3. Использование ограничений типов ========
# Реализация Clone для List
List.clone: (T: Clone) -> ((self: List(T)) -> List(T)) = {
    result = List(T)()
    for i in 0..self.length {
        result.push(self.data[i].clone())
    }
    return result
}

# ======== 1. Обобщённый алгоритм сортировки ========
Comparator: (T: Type) -> Type = {
    compare: (T, T) -> Int,  # -1 если a < b, 0 если a == b, 1 если a > b
}

# Обобщённый быстрый qsort
quicksort: (T: Clone) -> ((array: Array(T), cmp: Comparator(T)) -> Array(T)) = {
    if array.length <= 1 {
        return array.clone()
    }

    pivot = array[array.length / 2]
    left = Array(T)()
    right = Array(T)()

    for i in 0..array.length {
        if i == array.length / 2 {
            continue
        }
        item = array[i]
        comparison = cmp.compare(item, pivot)
        if comparison < 0 {
            left.push(item)
        } else {
            right.push(item)
        }
    }

    sorted_left = quicksort(left, cmp)
    sorted_right = quicksort(right, cmp)

    result = sorted_left.clone()
    result.push(pivot)
    result.extend(sorted_right)
    return result
}

# ======== 2. Реализация IntComparator ========
# Использование перегрузки функций
compare: (a: Int, b: Int) -> Int = {
    if a < b {
        return -1
    } else if a > b {
        return 1
    } else {
        return 0
    }
}

# ======== 3. Примеры использования ========
# Сортировка массива Int
numbers = Array(Int)(3, 1, 4, 1, 5, 9, 2, 6)
sorted = quicksort(numbers, Comparator(Int)())

# Сортировка массива String (требуется StringComparator)

Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),

    # Проверка размерностей на этапе компиляции:
    # использование типа Assert стандартной библиотеки
    _assert: Assert[Rows > 0],  # Rows > 0, иначе ошибка компиляции
    _assert: Assert[Cols > 0],  # Cols > 0, иначе ошибка компиляции

    # Операции с матрицами
    multiply: (M: Int) -> ((self: Matrix(T, Rows, Cols), other: Matrix(T, Cols, M)) -> Matrix(T, Rows, M)) = {
        result = Matrix(T, Rows, M)()
        for i in 0..Rows {

                for k in 0..Cols {

                }
                result.data[i][j] = sum
            }
        }
        return result
    }
}

            if i == j {
                matrix.data[i][j] = One::one()
            } else {
                matrix.data[i][j] = Zero::zero()
            }
        }
    }
    return matrix
}

## Компромиссы

### Преимущества

1. **Нулевая стоимость абстракций**
   - Мономорфизация на этапе компиляции, отсутствие накладных расходов во время выполнения
   - Без виртуальных функций, без RTTI

2. **Устранение мёртвого кода**
   - Анализ на этапе компиляции, инстанцируются только используемые обобщённые типы
   - Раздувание кода под контролем

3. **Замена макросов**
   - Типобезопасная генерация кода
   - Дружественность к IDE, чёткие сообщения об ошибках

4. **Вычисления на этапе компиляции**
   - Поддержка обобщённых типов на этапе компиляции
   - Характеристики типа проверки размерностей
   - Без ключевого слова `const`, только ограничения типов

### Недостатки

1. **Время компиляции**
   - Инстанцирование обобщённых типов увеличивает время компиляции
   - Решение ограничений может быть медленным

2. **Использование памяти**
   - Увеличение использования памяти компилятором
   - Для кэша требуется память

3. **Сложность реализации**
   - Сложный решатель ограничений
   - Сложный движок вычислений на уровне типов

4. **Диагностика ошибок**
   - Ошибки обобщённых типов могут быть сложными
   - Требуются чёткие подсказки об ошибках

### Меры по смягчению

1. **Стратегии кэширования**
   - Кэширование результатов инстанцирования
   - Ограничение памяти LRU-кэшем

2. **Инкрементальная компиляция**
   - Кэширование результатов компиляции
   - Инкрементальное инстанцирование

3. **Подсказки об ошибках**
   - Чёткие сообщения об ошибках
   - Подсказки для вывода обобщённых параметров

4. **Параллельная компиляция**
   - Параллельное инстанцирование обобщённых типов
   - Многопоточное решение ограничений

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Только базовые обобщённые типы | Не может заменить сложные макросы |
| Чистая макросистема | Без типобезопасности, плохие сообщения об ошибках |
| Только ограничения | Недостаточная гибкость |
| Обобщённые типы во время выполнения | Накладные расходы на производительность |

### Риски

| Риск | Влияние | Меры по смягчению |
|------|---------|-------------------|
| Сложность решения ограничений | Слишком долгое время компиляции | Инкрементальное решение + кэширование |
| Раздувание кода | Слишком большой бинарный файл | DCE + контроль порогов |
| Сложность реализации | Увеличение сроков разработки | Реализация по этапам |
| Диагностика ошибок | Плохой пользовательский опыт | Подробные сообщения об ошибках |

## Приложение

### БНФ-грамматика синтаксиса

```bnf
# Параметры обобщённых типов используют унифицированный синтаксис (),
# являясь частью функционального типа
# Например: map: (T: Type, R: Type) -> ((list: List(T), f: (T) -> R) -> List(R))

# Ограничение типа (в параметрах обобщённого типа)
type_bound ::= identifier
             | identifier '+' identifier ('+' identifier)*

# Объявление параметра (тип + имя)
parameter ::= identifier ':' type

parameters ::= parameter (',' parameter)*

# Объявление функции: name: type = expression
# Обобщённые параметры — первая группа параметров в функциональном типе:
# (T: Type) -> ((params) -> return)
function ::= identifier ':' type '=' (expression | block)

# Объявление метода: Type.method: type = expression
method ::= identifier '.' identifier ':' type '=' (expression | block)

# Определение типа (унифицированный синтаксис Binding)
# Обобщённые типы, такие как List: (T: Type) -> Type = { ... }
generic_type ::= identifier ':' type '=' type_expression

# Параметры Type в обобщённых параметрах автоматически заполняются
# компилятором из типов фактических аргументов
# Например, map(numbers, f), T извлекается из numbers: List(Int),
# R — из f: (Int) -> String
```
