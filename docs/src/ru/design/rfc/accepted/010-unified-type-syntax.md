# RFC-010: Унифицированный синтаксис типов - модель name: type = value

> **Статус**: Принято
>
> **Автор**: Чэнь Сюй
>
> **Дата создания**: 2025-01-20
>
> **Последнее обновление**: 2026-06-05 (обновлены правила возврата и семантика {})

## Аннотация

Данный RFC предлагает предельно минималистичную унифицированную модель синтаксиса типов: **всё есть `name: type = value`**.

В YaoXiang существует только одна форма объявления:

```
identifier : type = expression
```

где `type` может быть произвольным тип-выражением, а `expression` — произвольным значением-выражением.
**Нет `fn`, нет `struct`, нет `trait`, нет `impl`, нет ключевого слова `type` в нижнем регистре (но есть `Type` как метатип-ключевое слово)**.

> **Основной дизайн**: `Type` сам по себе является обобщённым типом. `(T: Type) -> Type` означает «тип, принимающий параметр типа T».

| Концепция     | Код                                        |
|---------------|--------------------------------------------|
| Переменная    | `x: Int = 42`                             |
| Функция       | `add: (a: Int, b: Int) -> Int = a + b`    |
| Запись        | `Point: Type = { x: Float, y: Float }`    |
| Интерфейс     | `Drawable: Type = { draw: (Surface) -> Void }` |
| Обобщённый тип| `List: (T: Type) -> Type = { data: Array(T), length: Int }` |
| Обобщённый тип| `Map: (K: Type, V: Type) -> Type = { keys: Array(K), values: Array(V) }` |
| Метод         | `Point.draw: (self: Point, s: Surface) -> Void = ...` |
| Обобщённая функция| `map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R))` |

**`Type` — единственное метатип-ключевое слово в языке**.
Оно используется для разметки тип-уровня, компилятор автоматически обрабатывает различение Type0, Type1, Type2... прозрачно для пользователя.

```yaoxiang
// Основной синтаксис: унификация + различение

// Переменная
x: Int = 42

// Функция (имена параметров в сигнатуре)
add: (a: Int, b: Int) -> Int = a + b

// Запись
Point: Type = {
    x: Float,
    y: Float,
    draw: (Surface) -> Void,
    serialize: () -> String
}

// Интерфейс (по сути — запись, все поля которой являются функциями)
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Определение методов (синтаксис Type.method)
Point.draw: (self: Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}

Point.serialize: (self: Point) -> String = {
    return "Point(${self.x}, ${self.y})"
}

// Обобщённый тип ((T: Type) -> Type = обобщённый тип, принимающий параметр типа)
List: (T: Type) -> Type = {
    data: Array(T),
    length: Int
}

Map: (K: Type, V: Type) -> Type = {
    keys: Array(K),
    values: Array(V)
}

// Использование
p: Point = Point(1.0, 2.0)
p.draw(screen)           // синтаксический сахар → Point.draw(p, screen)
s: Drawable = p           // структурное подтипирование: Point реализует Drawable
drawables: List(Drawable) = [p, r]
process_all(drawables)
```

## Мотивация

### Зачем нужна эта функция?

Текущая система типов содержит несколько обособленных концепций:
- Синтаксис объявления переменных
- Синтаксис определения функций
- Синтаксис определения типов (различные формы)
- Синтаксис определения интерфейсов
- Синтаксис привязки методов

Отсутствие унификации между этими концепциями приводит к фрагментации синтаксиса и высоким затратам на изучение.

### Цели проектирования

1. **Предельная унификация**: одно синтаксическое правило покрывает все случаи
2. **Простота и элегантность**: симметричная эстетика `name: type = value`
3. **Без новых ключевых слов**: повторное использование существующих синтаксических элементов
4. **Теоретическая элегантность**: типы сами по себе являются значениями типа Type
5. **Дружелюбие к обобщениям**: бесшовная интеграция с системой обобщений (RFC-011)

### Интеграция с системой обобщений

Унифицированная модель синтаксиса RFC-010 **естественно согласуется** с дизайном системы обобщений RFC-011, параметры обобщений органично вписываются в унифицированную модель:

```yaoxiang
// Базовые обобщения (RFC-011 Фаза 1)
List: (T: Type) -> Type = { data: Array(T), length: Int }

// Обобщённая функция (RFC-023 синтаксис: Type параметр в сигнатуре можно опустить, выводится при вызове)
map: (: Type, R: Type) -> (( list: List(T), f: (T) -> R) -> List(R)) = ...

// Ограничения типов (RFC-011 Фаза 2)
clone: (value: T) -> T = value.clone()  // Ограничение T: Clone переносится типом параметра

// Const-обобщения (RFC-011 Фаза 4)
Array: (T: Type, N: Int) -> Type = { data: Array(T, N), length: N }
```

**Зависимости**:
- RFC-011 Фаза 1 (базовые обобщения) является **сильной зависимостью** RFC-010
- Без базовых обобщений обобщённые примеры RFC-010 не скомпилируются
- Рекомендация: RFC-011 Фаза 1 и RFC-010 реализовать синхронно

## Предложение

### Основной принцип: конструктор типа vs функция/переменная

**Это ключевой выбор дизайна, определяющий правила устранения синтаксической неоднозначности:**

| Запись     | Значение              | Правило |
|------------|----------------------|---------|
| **`x: Type = ...`** | Конструктор типа   | `: Type` явно объявлен → принудительно тип |
| **`f = ...`**       | Функция или переменная | Нет `: Type` → HM активно выводит как функцию/переменную |

**Почему так сделано?**

Синтаксис `{ ... }` сам по себе неоднозначен:
- `{ x: Float, y: Float }` может быть **литералом типа** (запись)
- `{ a = 1 + 1 }` может быть **блоком кода** (выполняемые операторы, возвращает Void)

**Правила устранения неоднозначности**:
- **Есть** `: Type` → принудительная интерпретация как конструктор типа, `{ ... }` — литерал типа
- **Нет** `: Type` → HM активно интерпретирует `{ ... }` как блок кода, выводит как функциональный тип

```yaoxiang
# ✅ Конструктор типа: есть : Type
Point: Type = { x: Float, y: Float }

# ✅ Функция: нет : Type, HM выводит как () -> Void
main = { println("Hello") }

# ❌ Ошибка: нет : Type, компилятор не может интерпретировать { ... } как тип
Point = { x: Float, y: Float }  // HM выводит как функцию, не как тип!
```

---

**Унифицированная модель: identifier : type = expression**

```
├── Переменная
│   └── x: Int = 42
│
├── Функция
│   └── add: (a: Int, b: Int) -> Int = a + b  # Нет : Type, HM выводит как функцию
│
├── Запись
│   └── Point: Type = { x: Float, y: Float }  # Обязательно возвращает: Type
│
├── Интерфейс
│   └── Drawable: Type = { draw: (Surface) -> Void }  # Обязательно возвращает: Type
│
├── Обобщённый тип
│   └── List: (T: Type) -> Type = { data: Array(T), length: Int }  # Обязательно возвращает: Type
│
├── Обобщённый тип (несколько параметров)
│   └── Map: (K: Type, V: Type) -> Type = { keys: Array(K), values: Array(V) }  # Обязательно возвращает: Type
│
├── Метод
│   └── Point.draw: (self: Point, surface: Surface) -> Void = ...
│
└── Обобщённая функция
    └── map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R))  # Не возвращает Type, HM выводит как функцию
```

### Уровни метаязыка (внутренне компилятора)

**Компилятор внутренне поддерживает вселенную уровней `level: selfpointnum`** (хранится как строка, теоретически бесконечно extensible).

| Level  | Описание |
|--------|----------|
| `Type0`| Повседневные типы (`Int`, `Float`, `Point`) |
| `Type1`| Конструкторы типов (`List`, `Maybe`) |
| `Type2+`| Высшие конструкторы |

**Пользователь никогда не видит эти числа**, только `: Type`.

> **Изоморфизм Карри-Ховарда**: существование вселенных уровней — не деталь реализации, а необходимое условие логической согласованности. Изоморфизм Карри-Ховарда отождествляет типы с высказываниями; если позволить `Type: Type` («тип типов тоже является типом»), возникнет парадокс Рассела, аналогичный «это высказывание ложно» — в типовой системе это проявится как парадокс Жирара. Многоуровневая структура YaoXiang `Type0 / Type1 / Type2…` (накапливающиеся вселенные в типовой теории Мартина-Лёфа) гарантирует, что каждый тип принадлежит определённому уровню, а `Typeₙ : Typeₙ₊₁` образует никогда не замыкающуюся восходящую цепочку, фундаментально предотвращая парадоксы. Это означает, что типовая система YaoXiang в смысле Карри-Ховарда является **логически согласованной**.

### Синтаксические определения

#### 1. Объявление переменных

```yaoxiang
// Базовый синтаксис
x: Int = 42
name: String = "Alice"
flag: Bool = true

// Вывод типа (можно опустить)
y = 100  // Выводится как Int
```

#### 2. Определение функций

```yaoxiang
// Одно-выражений форма (прямое возвращение значения, без return)
add: (a: Int, b: Int) -> Int = a + b
greet: (name: String) -> String = "Hello, ${name}!"

// Форма с блоком (обязательно использовать return для возврата значения)
process: (x: Int) -> Int = {
    a = x * 2
    b = a + 1
    return b
}

// Многострочный блок
calc: (x: Float, y: Float, op: String) -> Float = {
    return match op {
        "+" -> x + y,
        "-" -> x - y,
        _ -> 0.0
    }
}

// Void функция (return не нужен внутри блока)
print: (msg: String) -> Void = {
    console.write(msg)
}
```

#### Правила возврата

Возвращаемое значение зависит от формы правой части `=`:

| Запись     | Возвращаемое значение |
|------------|----------------------|
| `= expr` (без фигурных скобок) | Прямое возвращение `expr` |
| `= { ... }` (с фигурными скобками) | Обязательно использовать `return`, иначе возвращается `Void` |

```yaoxiang
# Одно выражение: прямое возвращение, return не нужен
add: (a: Int, b: Int) -> Int = a + b

# Блок кода: обязательно return для возврата значения
process: (x: Int) -> Int = {
    a = x * 2
    b = a + 1
    return b
}

# Void функция: return не нужен
print: (msg: String) -> Void = {
    console.write(msg)
}
```

> **Обоснование дизайна**: `{ ... }` — это зависимо-управляемая вычислительная единица (см. далее), её семантика возврата отличается от одно-выражений формы. Фигурные скобки вводят контекст с несколькими операторами, поэтому требуется явный `return` для устранения неоднозначности «является ли последнее выражение возвращаемым значением».

#### Семантика `{}`: зависимо-управляемая вычислительная единица

`{ ... }` в YaoXiang — это не просто блок кода, это **зависимо-управляемая вычислительная единица**. Эта семантика остаётся консистентной в теле функции, инициализации переменных и `spawn`:

**Основные правила**:
- Операторы присваивания внутри `{}` автоматически упорядочиваются по зависимостям, а не по порядку записи
- Операции с готовыми зависимостями выполняются немедленно, отсутствующие — блокируются до готовности
- Для явного возврата значения используется `return` (см. правила возврата)

```yaoxiang
# Зависимо-управляемый: b зависит от a, компилятор автоматически упорядочивает
result: Int = {
    b = a + 1      # Зависит от a → автоматически после a
    a = 10         # Нет зависимостей → может выполниться первым
    return b       # Возвращает 11
}
```

> **Отличие от одно-выражений**: `= expr` (без фигурных скобок) — простое связывание с прямым возвратом значения; `= { ... }` (с фигурными скобками) — вводит контекст зависимо-управляемых вычислений с несколькими операторами и явным `return`.

#### Блок `spawn`

`spawn { ... }` — единственный примитив параллелизма в YaoXiang. Он использует зависимо-управляемую семантику `{}` для автоматического распараллеливания:

- Прямые дочерние присваивания внутри `spawn { ... }` автоматически создают параллельные задачи
- Задачи с готовыми зависимостями выполняются конкурентно немедленно
- Вызывающая сторона блокируется до завершения всех дочерних задач

```yaoxiang
result = spawn {
    a = fetch_data("url1")    # Задача 1
    b = fetch_data("url2")    # Задача 2 (нет зависимости от a, выполняется параллельно)
    c = process(a, b)        # Зависит от a и b → выполняется после обоих
    return c
}
// Вызывающая сторона блокируется здесь, пока все задачи внутри spawn не завершатся
```

> **Подробное определение**: Полная семантика `spawn`, правила создания задач и модель блокировки описаны в `008-runtime-concurrency-model.md`.

#### 3. Определение типов

Определение типов — ядро унифицированного синтаксиса YaoXiang, включает поля, значения по умолчанию, привязанные методы, реализации интерфейсов:

##### Базовые типы

**Запись**: список полей, типы полей могут быть произвольными тип-выражениями.

```yaoxiang
Point: Type = {
    x: Float,
    y: Float
}
```

**Поля со значениями по умолчанию**: поля могут иметь значения по умолчанию, необязательны при конструировании.

```yaoxiang
Point: Type = {
    x: Float = 0,
    y: Float = 0
}
```

Использование:

```yaoxiang
Point() → Point(x=0, y=0)
Point(x=1) → Point(x=1, y=0)
Point(x=1, y=2) → Point(x=1, y=2)
```

**Поля без значений по умолчанию**: обязательны при конструировании.

```yaoxiang
Point2: Type = {
    x: Float,
    y: Float
}
```

Использование:
```yaoxiang
Point2(x=1, y=2) //✓
Point2() //✗
Point2(x=1) //✗
```

##### Привязка методов

**Способ 1: прямая привязка внешней функции внутри определения типа**

```yaoxiang
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]           # Привязка к позиции 0, каррирование: method: (b: Point) -> Float
}
// Вызов: p1.distance(p2) → distance(p1, p2)
```

**Способ 2: анонимная функция + позиционная привязка**

```yaoxiang
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) → distance(p1, p2)
```

##### Реализация интерфейсов

**Имя интерфейса пишется в теле типа, компилятор автоматически проверяет реализацию**

```yaoxiang
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

Point: Type = {
    x: Float,
    y: Float,
    Drawable,          # Реализует интерфейс Drawable

}
```

##### Определение интерфейсов

**Интерфейс = запись, все поля которой являются функциями**

```yaoxiang
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой тип/пустой интерфейс

```

##### Определение методов (внешнее)

**Типовые методы**: ассоциированы с конкретным типом (синтаксис Type.method)

```yaoxiang
Point.draw: (self: Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}

Point.serialize: (self: Point) -> String = {
    return "Point(${self.x}, ${self.y})"
}
```

##### Привязка методов (внешняя)

Обычные методы можно привязать к типу синтаксисом `[position]` (подробно о синтаксисе — RFC-004).

**Ручная привязка**:

```yaoxiang
// Явная привязка
Point.distance = distance[0]

// Указание позиции привязки
Point.transform = transform[1]  # this привязан к позиции 1
```

**Мульти-позиционная привязка**:

```yaoxiang
// Привязка нескольких позиций (автоматическое каррирование)
Point.transform = transform_points[0, 1]
// Вызов: p1.transform(p2)(2.0) → transform_points(p1, p2, 2.0)
```

**Обратная привязка** (тип-метод → обычная функция):

```yaoxiang
// Типовой метод в обычную функцию
draw_point: (p: Point, surface: Surface) -> Void = Point.draw
```

#### 5. Обобщённые типы

```yaoxiang
// Базовые обобщения (RFC-011 Фаза 1)
List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (T:Type)-((self: List(T), item: T) -> Void),
    get: (T:Type)->((self: List(T), index: Int) -> Maybe(T))
}

// Конкретная реализация (RFC-023 синтаксис)
IntList: Type = List(Int)

IntList.push = {
    self.data.append(item)
    self.length = self.length + 1
}

List.push = (type: Type) -> {
    return (self: List(type), item: type) -> {
        self.data.append(item)
        self.length = self.length + 1
    }
}

IntList.push(Int)(self, item)  // Пример вызова

// Обобщённые методы (RFC-023 синтаксис: параметры типа автоматически выводятся при вызове)
List.push: (self: List(T), item: T) -> Void = {
    self.data.append(item)
    self.length = self.length + 1
}

List.get: (self: List(T), index: Int) -> Maybe(T) = {
    if index >= 0 && index < self.length {
        return Maybe.Just(self.data[index])
    } else {
        return Maybe.Nothing
    }
}
```

#### 6. Синтаксис вызова обобщений

Вызов обобщённых типов и обобщённых функций унифицирован через синтаксис `()`. `[]` не используется ни в каком обобщённом контексте.

**Основные правила**:

1. **`()` делает всё**: применение типа, вызов функции, конструирование значения — всё через `()`

```yaoxiang
# Аннотация типа
numbers: List(Int) = List(1, 2, 3)

# Пустой контейнер: T приходит слева
empty: List(Int) = List()

# Вызов обобщённой функции — тип автоматически перетекает из параметров
strings = map(numbers, f)
// T=Int приходит из numbers: List(Int)
// R=String приходит из f: (Int) -> String
```

2. **Type слева, значение справа**: `name: type = value` — параметры типа объявляются слева, справа всегда конкретное значение. Пустой контейнер `List()` должен получить `T` из левой аннотации типа.

3. **Информация о типе пишется один раз** — при объявлении параметра, компилятор несёт её дальше:

```yaoxiang
numbers: List(Int) = List(1, 2, 3)  # Int пишется один раз слева
f: (Int) -> String = (x) => x.to_string()
strings = map(numbers, f)   # T=Int, R=String автоматически из типов numbers и f
```

4. **Конструирование значения выводит тип из элементов**:

```yaoxiang
x = List(1, 2, 3)       # Выводится как List(Int)
y = List("a", "b")      # Выводится как List(String)
z = List()              # ❌ Ошибка компиляции: невозможно вывести T
z: List(Int) = List()   # ✅ T=Int приходит из левой аннотации
```

5. **Псевдонимы типов**:

```yaoxiang
IntList: Type = List(Int)
StringToInt: Type = (String) -> Int
Matrix3x3: Type = Matrix(Float, 3, 3)
```

> **Сравнение со старым синтаксисом**: `List[Int]` → `List(Int)`, `List[Int]()` → `List()`, `List[Int](1,2,3)` → `List(1,2,3)`.
> Старый `[]` синтаксис для обобщений полностью удалён. `[]` используется только для литералов массивов/списков и доступа по индексу.

#### Полный пример

    draw: (self: Self, surface: Surface) -> Void,
    bounding_box: (self: Self) -> Rect
}

    serialize: (self: Self) -> String
}

    translate: (self: Self, dx: Float, dy: Float) -> Self,
    scale: (self: Self, factor: Float) -> Self
}

Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable,
    Transformable
}

Rect: Type = {
    x: Float,
    y: Float,
    width: Float,
    height: Float,
    Drawable,
    Serializable,
    Transformable
}

    surface.plot(self.x, self.y)

}

}

    return Point(self.x + dx, self.y + dy)
}

}

    dx = p1.x - p2.x
    dy = p1.y - p2.y
    return (dx * dx + dy * dy).sqrt()
}

// Методы Rect

    surface.draw_rect(self.x, self.y, self.width, self.height)

}

    return Rect(self.x + dx, self.y + dy, self.width, self.height)
}

}

// ======== 4. Привязка методов ========

Point.distance = distance[0]  // Привязка к позиции 0, каррирование: method: (p2: Point) -> Float
Point.transform = transform[1]  // Привязка к позиции 1, каррирование: method: (dx: Float, dy: Float) -> Point
Rect.transform = transform[1]  // Привязка к позиции 1, каррирование: method: (dx: Float, dy: Float) -> Rect

// ...аналогично, привязать остальные методы...

// ======== 5. Использование ========

// Вызов методов (синтаксический сахар)

// Вызов обычного метода (прямой вызов)

// Цепочечный вызов

## Детальный дизайн

### Алгоритм проверки интерфейсов

```rust
fn check_type_implements_interface(
    typ: &Type,
    iface: &Type
) -> Result<(), TypeError> {
    // Для каждого поля интерфейса (функциональное поле)
    for (field_name, iface_field) in &iface.fields {
        // Проверяем, есть ли у типа метод с таким именем
        if let Some(method) = typ.methods.get(field_name) {
            // Проверяем совместимость сигнатуры метода
            // Поле интерфейса: (Surface) -> Void
            // Сигнатура метода: (Point, Surface) -> Void
            // Сравнение: после удаления параметра self должно совпадать
            if !method_signature_matches(method, iface_field.type_) {
                return Err(TypeError::MethodSignatureMismatch {
                    type_name: typ.name,
                    interface_name: iface.name,
                    method_name: field_name,
                });
            }
        } else {
            return Err(TypeError::MissingMethod {
                type_name: typ.name,
                interface_name: iface.name,
                method_name: field_name,
            });
        }
    }
    Ok(())
}
```

### Прямое присваивание интерфейсу и оптимизация компиляции

Интерфейсные типы поддерживают прямое присваивание, компилятор автоматически выбирает оптимальную стратегию вызова на основе типа правой части:

```yaoxiang
// Прямое присваивание конкретного типа → тип известен на этапе компиляции, вызов без накладных расходов
d: Drawable = Circle(1)
d.draw(screen)  // После компиляции: прямой вызов circle_draw(screen), без vtable

// Возврат из функции → тип неизвестен на этапе компиляции, используется vtable
d: Drawable = get_shape()
d.draw(screen)  // Поиск метода через vtable

// Разнородная коллекция → используется vtable
shapes: List(Drawable) = [Circle(1), Rect(2, 3)]
for s in shapes {
    s.draw(screen)  // Поиск метода через vtable
}
```

**Стратегии оптимизации компиляции**:

| Сценарий                              | Результат вывода      | Способ вызова |
|---------------------------------------|----------------------|---------------|
| `d: Drawable = Circle(1)`             | Конкретный тип Circle| Прямой вызов (нулевые накладные расходы) |
| `d: Drawable = get_shape()`           | Неизвестный          | vtable |
| `shapes: List(Drawable) = [...]`      | Разнородный          | vtable |

**Правила**:
1. Когда правая часть — конкретный конструктор типа, известный на этапе компиляции, генерируется прямой вызов IR
2. Когда тип правой части невозможно определить на этапе компиляции, используется механизм vtable
3. vtable как fallback гарантирует корректность динамического полиморфизма

### Поддержка утиной типизации

```yaoxiang
// Если есть тот же метод — можно присвоить типу интерфейса
CustomPoint: Type = {

### Изменения синтаксиса

| Раньше                                   | Теперь                                              |
|------------------------------------------|-----------------------------------------------------|
| `type Point = Point(x: Float, y: Float)`| `type Point = { x: Float, y: Float }`              |
| `type Result(T, E) = ok(T) \| err(E)`   | `Result: (T: Type, E: Type) -> Type = { ok: (T) -> Self, err: (E) -> Self }` |
| Требовалось ключевое слово `impl`       | Ключевое слово не нужно, имя интерфейса пишется в теле типа |

## Синтаксический дизайн: именованные функции — по сути синтаксический сахар для Lambda

### Ключевое понимание

**Именованные функции и лямбда-выражения — это одно и то же!** Единственное различие: именованная функция просто даёт имя лямбде.

```yaoxiang
// По сути полностью идентичны
add: (a: Int, b: Int) -> Int = a + b           # Именованная функция (рекомендуется)
add: (a: Int, b: Int) -> Int = (a, b) => a + b        # Лямбда-форма (полностью эквивалентно)
```

### Модель синтаксического сахара

```
// Именованная функция = Lambda + имя
name: (Params) -> ReturnType = body

// По сути
name: (Params) -> ReturnType = (params) => body
```

**Ключевой момент**: когда сигнатура полностью объявляет типы параметров, имена параметров в голове лямбды становятся избыточными и могут быть опущены.

### Правила области видимости параметров

**Параметры перекрывают внешние переменные**: область видимости параметров в сигнатуре перекрывает тело функции, приоритет у внутренней области.

```yaoxiang
x = 10  // Внешняя переменная

double: (x: Int) -> Int = x * 2  // ✅ Параметр x перекрывает внешний x, результат 20
```

### Гибкость позиционирования аннотаций

Аннотации типа можно ставить в любом из следующих мест, **достаточно аннотировать хотя бы в одном**:

| Позиция аннотации | Форма                           | Примечание |
|-------------------|--------------------------------|------------|
| Только в сигнатуре| `double: (x: Int) -> Int = x * 2` | ✅ Рекомендуется |
| Только в голове Lambda| `double = (x: Int) => x * 2` | ✅ Допустимо |
| Обе стороны        | `double: (x: Int) -> Int = (x) => x * 2` | ✅ Избыточно, но допустимо |

### Полный пример

```yaoxiang
// ✅ Рекомендуется: полная сигнатура, голова лямбды опущена
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1
main: () -> Void = { print("hi") }

// ✅ Допустимо: тип указан в голове лямбды
double = (x: Int) => x * 2

// ✅ Допустимо: обе стороны аннотированы
double: (x: Int) -> Int = (x) => x * 2
```

### Преимущества дизайна

| Характеристика | Преимущество |
|----------------|--------------|
| **Краткость**  | При полной сигнатуре не нужно повторно писать имена параметров |
| **Гибкость**   | Сохранена лямбда-форма, используй какую удобнее |
| **Согласованность** | С единой моделью объявления переменных `x: Int = 42` |
| **Наглядность** | `name: Type = body` напрямую соответствует «имя name, тип Type, значение body» |

## Приложение

### БНФ синтаксиса

```bnf
program ::= statement*

statement ::= declaration | expression

# Унифицированное объявление: name: Type = expression
declaration ::= identifier ':' type_expr '=' expression

# Типовое выражение
type_expr ::= identifier
       | identifier '(' type_expr (',' type_expr)* ')'      # Применение типа
       | '(' type_expr (',' type_expr)* ')' '->' type_expr       # Функциональный тип
       | '{' type_field* '}'                       # Запись/интерфейсный тип
       | 'Type'                                    # Метатип

type_field ::= identifier ':' type_expr
             | identifier                           # Ограничение интерфейса

# Обобщённые параметры: как часть функционального типа, например (T: Type, R: Type) -> (...)
# Не требуется отдельное БНФ-правило — параметр : Type это обычный параметр функции

# Выражение
expression ::= literal
              | identifier
              | identifier '(' expression (',' expression)* ')'  # Вызов функции / вызов конструктора
              | '(' expression (',' expression)* ')'              # Кортеж
              | expression '.' identifier '(' arguments? ')'    # Вызов метода
              | lambda
              | '{' field ':' expression (',' field ':' expression)* '}'

arguments ::= expression (',' expression)*

lambda ::= '(' parameter_list? ')' '=>' block

block ::= expression | '{' expression* '}'
```

### Глоссарий

| Термин       | Определение |
|--------------|-------------|
| Объявление   | Оператор присваивания формы `name: type = value` |
| Запись       | Тип `{ ... }` с именованными полями |
| Интерфейс    | Запись, все поля которой являются функциональными типами |
| Обобщённый тип | Тип, определённый как `Name: (T: Type) -> Type = { ... }`, принимающий параметры типа |
| Типовой метод| Метод формы `Type.method`, ассоциированный с конкретным типом |
| Обобщённая функция | Функция, использующая синтаксис `(T: Type)`, параметры типа передаются как первая группа параметров |
| Метатип      | `Type`, единственный маркер тип-уровня в языке |
