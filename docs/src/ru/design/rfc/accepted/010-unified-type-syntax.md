# RFC-010: Унифицированный синтаксис типов — модель name: type = value


## Резюме

Настоящий RFC предлагает предельно унифицированную модель синтаксиса типов: **всё есть `name: type = value`**.

В YaoXiang существует только одна форма объявления:

```
identifier : type = expression
```

где `type` может быть любым выражением типа, а `expression` — любым выражением значения.
**Нет `fn`, нет `struct`, нет `trait`, нет `impl`, нет строчного ключевого слова `type` (но есть `Type` как ключевое слово метатипа)**.

> **Ключевая идея**: `Type` сам по себе является дженерик-типом. `(T: Type) -> Type` означает «тип, принимающий параметр типа T».

| Понятие       | Запись в коде                                      |
|---------------|---------------------------------------------------|
| Переменная    | `x: Int = 42`                                |
| Функция       | `add: (a: Int, b: Int) -> Int = a + b`       |
| Тип записи    | `Point: Type = { x: Float, y: Float }`       |
| Интерфейс     | `Drawable: Type = { draw: (Surface) -> Void }` |
| Дженерик-тип  | `List: (T: Type) -> Type = { data: Array(T), length: Int }` |
| Дженерик-тип  | `Map: (K: Type, V: Type) -> Type = { keys: Array(K), values: Array(V) }` |
| Метод         | `Point.draw: (p: Point, s: Surface) -> Void = ...`<br>`Point.draw = draw[0]` |
| Дженерик-функция | `map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R))` |

**`Type` — единственное ключевое слово метатипа в языке**.

> **Пространство имён vs привязка метода**: префикс `Type.name` обозначает лишь **принадлежность к пространству имён**, и ничего более.
> Он не вызывает никакой неявной привязки. Чтобы синтаксис вызова через `.` вроде `p.draw(screen)` работал,
> требуется явная привязка: `Point.draw = draw[0]`.
> Подробности см. в разделе «Пространства имён и привязка методов» ниже.
Оно используется для маркировки уровня типов; компилятор автоматически различает Type0, Type1, Type2... — это прозрачно для пользователя.

```yaoxiang
// Базовый синтаксис: унифицированный и различаемый

// Переменная
x: Int = 42

// Функция (имена параметров указываются в сигнатуре)
add: (a: Int, b: Int) -> Int = a + b

// Тип записи
Point: Type = {
    x: Float,
    y: Float,
    draw: (Surface) -> Void,
    serialize: () -> String
}

// Интерфейс (по сути — тип записи, все поля которого являются функциями)
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Определение метода (используется синтаксис Type.method)
Point.draw: (self: Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}

Point.serialize: (self: Point) -> String = {
    return "Point(${self.x}, ${self.y})"
}

// Дженерик-тип ((T: Type) -> Type = дженерик-тип, принимающий параметр типа)
List: (T: Type) -> Type = {
    data: Array(T),
    length: Int
}

Map: (K: Type, V: Type) -> Type = {
    keys: Array(K),
    values: Array(V)
}

// Использование
p: Point = Point(1.0, 2.0)
p.draw(screen)           // синтаксический сахар → Point.draw(p, screen)
s: Drawable = p           // структурная подтипизация: Point реализует Drawable
drawables: List(Drawable) = [p, r]
process_all(drawables)
```

## Мотивация

### Зачем нужна эта возможность?

В текущей системе типов существует несколько разрозненных понятий:
- Синтаксис объявления переменных
- Синтаксис определения функций
- Синтаксис определения типов (другой синтаксис)
- Синтаксис определения интерфейсов
- Синтаксис привязки методов

Этим понятиям не хватает единства, что приводит к фрагментации синтаксиса и высокой стоимости обучения.

### Цели проектирования

1. **Предельное единство**: одно правило синтаксиса покрывает все случаи
2. **Простота и элегантность**: симметричная эстетика `name: type = value`
3. **Без новых ключевых слов**: повторное использование существующих элементов синтаксиса
4. **Теоретическая элегантность**: сам тип является значением типа Type
5. **Дружественность к дженерикам**: бесшовная интеграция с системой дженериков (RFC-011)

### Интеграция с системой дженериков

Унифицированная синтаксическая модель RFC-010 **органично сочетается** с дизайном системы дженериков RFC-011, параметры дженериков могут бесшовно вписаться в единую модель:

```yaoxiang
// Базовые дженерики (RFC-011 Phase 1)
List: (T: Type) -> Type = { data: Array(T), length: Int }

// Дженерик-функция (синтаксис RFC-023: позиция Type в сигнатуре может опускаться, тип выводится автоматически при вызове)
map: (: Type, R: Type) -> (( list: List(T), f: (T) -> R) -> List(R)) = ...

// Ограничения типов (RFC-011 Phase 2)
clone: (value: T) -> T = value.clone()  // Ограничение T: Clone несётся в типе параметра

// Const-дженерики (RFC-011 Phase 4)
Array: (T: Type, N: Int) -> Type = { data: Array(T, N), length: N }
```

**Зависимости**:
- RFC-011 Phase 1 (базовые дженерики) — **сильная зависимость** для RFC-010
- Без базовых дженериков примеры дженериков в RFC-010 не скомпилируются
- Рекомендация: реализовать RFC-011 Phase 1 одновременно с RFC-010

## Предложение

### Ключевой принцип: конструктор типа vs функция/переменная

**Это ключевой выбор проектирования, определяющий правила устранения неоднозначности синтаксиса:**

| Запись | Значение | Правило |
|------|------|------|
| **`x: Type = ...`** | Конструктор типа | `: Type` явно объявлен → принудительно тип |
| **`f = ...`** | Функция или переменная | Без `: Type` → HM самостоятельно выводит как функцию/переменную |

**Почему так спроектировано?**

Сам синтаксис `{ ... }` неоднозначен:
- `{ x: Float, y: Float }` может быть **литералом типа** (тип записи)
- `{ a = 1 + 1 }` может быть **блоком кода** (выполняемый оператор, возвращающий Void)

**Правила устранения неоднозначности**:
- **Есть** `: Type` → принудительно интерпретируется как конструктор типа, `{ ... }` — литерал типа
- **Нет** `: Type` → HM интерпретирует `{ ... }` как блок кода и выводит как функцию

```yaoxiang
# ✅ Конструктор типа: есть : Type
Point: Type = { x: Float, y: Float }

# ✅ Функция: нет : Type, HM выводит () -> Void
main = { println("Hello") }

# ❌ Ошибка: нет : Type, компилятор не может интерпретировать { ... } как тип
Point = { x: Float, y: Float }  // HM выводит как функцию, а не как тип!
```

---

**Унифицированная модель: identifier : type = expression**

```
├── Переменная
│   └── x: Int = 42
│
├── Функция
│   └── add: (a: Int, b: Int) -> Int = a + b  # Без : Type, HM выводит функцию
│
├── Тип записи
│   └── Point: Type = { x: Float, y: Float }  # Должен возвращать: Type
│
├── Интерфейс
│   └── Drawable: Type = { draw: (Surface) -> Void }  # Должен возвращать: Type
│
├── Дженерик-тип
│   └── List: (T: Type) -> Type = { data: Array(T), length: Int }  # Должен возвращать: Type
│
├── Дженерик-тип (много параметров)
│   └── Map: (K: Type, V: Type) -> Type = { keys: Array(K), values: Array(V) }  # Должен возвращать: Type
│
├── Функция пространства имён
│   └── draw: (p: Point, surface: Surface) -> Void = ...
│       Point.draw = draw[0]  # Только после явной привязки появляется синтаксис точечного вызова
│
└── Дженерик-функция
    └── map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R))  # Не возвращает Type, HM выводит функцию
```

### Уровни метатипов (внутри компилятора)

**Внутри компилятора** поддерживается иерархия вселенной `level: selfpointnum` (хранится строкой, теоретически бесконечно расширяемая).

| Level | Описание |
|-------|------|
| `Type0` | Обычные типы (`Int`, `Float`, `Point`) |
| `Type1` | Конструкторы типов (`List`, `Maybe`) |
| `Type2+` | Конструкторы высшего порядка |

**Пользователь никогда не видит этих чисел**, только `: Type`.

### Изоморфизм Карри-Ховарда: тип — это утверждение, программа — это доказательство

Унифицированный синтаксис YaoXiang `name: type = value` выбран не случайно — он представляет собой прямое отображение изоморфизма Карри-Ховарда (Curry-Howard correspondence). Этот изоморфизм раскрывает глубокий факт: **система типов и логическая система — это две стороны одной и той же сущности**.

| Логика (утверждение) | Система типов (YaoXiang) | Пример |
|---|---|---|
| Утверждение P | Тип T | `Int`, `Bool` |
| Доказательство P | Значение типа T | `42: Int`, `true: Bool` |
| P → Q (импликация) | Тип функции `(P) -> Q` | `(x: Int) -> Bool` |
| P ∧ Q (конъюнкция) | Тип записи `{ p: P, q: Q }` | `{ x: Int, y: Bool }` |
| ∀x.P(x) (квантор всеобщности) | Дженерик-функция `(T: Type) -> ...` | `map: (T: Type, R: Type) -> ...` |
| P ⊕ Q (дизъюнкция) | Перечисление / tagged union | `Maybe: (T: Type) -> Type = { ... }` |

**Значение `name: type = value` в рамках Карри-Ховарда**:

```yaoxiang
// "x: Int = 42" читается как: "существует доказательство типа Int, с именем x, со значением 42"
x: Int = 42

// "add: (a: Int, b: Int) -> Int = a + b" читается как:
// "существует доказательство импликации: дано доказательство a и b типа Int, можно построить доказательство типа Int"
add: (a: Int, b: Int) -> Int = a + b

// "Point: Type = { x: Float, y: Float }" читается как:
// "Point — это утверждение, для доказательства которого требуется одновременно предоставить доказательство x типа Float и доказательство y типа Float"
Point: Type = { x: Float, y: Float }
```

**Почему это важно?**

1. **Логическая непротиворечивость = безопасность типов**: если система типов позволяет конструировать значение типа `T` без какого-либо допустимого представления во время выполнения, это подобно тому, как в логике разрешают доказывать ложное утверждение — система рушится. Карри-Ховард говорит нам: **язык с безопасной типизацией по своей сути является непротиворечивой логической системой**.

2. **Иерархия вселенной — необходимое условие**: как подробно описано ниже, если разрешить `Type: Type` (то есть «тип типа тоже является типом»), возникает парадокс Рассела (в теории типов проявляющийся как парадокс Жирара). Расслоение YaoXiang `Type₀ : Type₁ : Type₂ : ...` гарантирует, что каждый тип принадлежит только одному уровню, формируя никогда не замыкающуюся восходящую цепочку, что фундаментально избегает парадоксов. Это означает, что система типов YaoXiang является **логически непротиворечивой** в смысле Карри-Ховарда.

3. **Теоретическая основа унифицированного синтаксиса**: причина, по которой `name: type = value` может одним синтаксисом покрыть переменные, функции, типы, интерфейсы и дженерики, состоит в том, что все они в рамках Карри-Ховарда представляют собой одно и то же — **предоставление доказательства для утверждения**. Переменная — это свидетельство утверждения, функция — это свидетельство импликации, запись — это свидетельство конъюнкции, дженерик — это свидетельство квантора всеобщности. Унифицированный синтаксис — не искусственное совпадение, а естественное следствие изоморфизма Карри-Ховарда.

> **Дополнительное чтение**: Wadler, P. (2015). *"Propositions as Types."* Communications of the ACM, 58(12), 75–84. Эта статья доступно объясняет историю и значение изоморфизма Карри-Ховарда.

### Определение синтаксиса

#### 1. Объявление переменных

```yaoxiang
// Базовый синтаксис
x: Int = 42
name: String = "Alice"
flag: Bool = true

// Вывод типа (можно опустить)
y = 100  // выводится как Int
```

#### 2. Определение функций

```yaoxiang
// Одно-выражение (возвращает значение напрямую, без return)
add: (a: Int, b: Int) -> Int = a + b
greet: (name: String) -> String = "Hello, ${name}!"

// Блочная форма (необходимо использовать return)
process: (x: Int) -> Int = {
    a = x * 2
    b = a + 1
    return b
}

// Многострочный блок
calc: (x: Float, y: Float, op: String) -> Float = {
    return match op {
        "+" -> x + y,
        "-" -> x - y,
        _ -> 0.0
    }
}

// Функция Void (внутри блока return не нужен)
print: (msg: String) -> Void = {
    console.write(msg)
}
```

#### Правила возврата

Возвращаемое значение зависит от формы правой части `=`:

| Запись | Возвращаемое значение |
|------|--------|
| `= expr` (без фигурных скобок) | Возвращает `expr` напрямую |
| `= { ... }` (с фигурными скобками) | Требуется `return`, иначе возвращается `Void` |

```yaoxiang
# Одно выражение: возвращает значение напрямую, return не нужен
add: (a: Int, b: Int) -> Int = a + b

# Блок кода: требуется return для возврата значения
process: (x: Int) -> Int = {
    a = x * 2
    b = a + 1
    return b
}

# Функция Void: return не нужен
print: (msg: String) -> Void = {
    console.write(msg)
}
```

> **Обоснование проектирования**: `{ ... }` — это вычислительная единица, управляемая зависимостями (см. ниже), её семантика возврата отличается от одно-выражения. Фигурные скобки вводят контекст нескольких операторов, поэтому требуется явный `return` для устранения неоднозначности «является ли последнее выражение возвращаемым значением».

#### Семантика `{}`: вычислительная единица, управляемая зависимостями

`{ ... }` в YaoXiang — не просто блок кода, а **вычислительная единица, управляемая зависимостями**. Эта семантика остаётся согласованной в телах функций, инициализации переменных и `spawn`:

**Основные правила**:
- Операторы присваивания внутри `{}` автоматически сортируются по зависимостям, а не по порядку записи
- Когда зависимости готовы, выполнение происходит немедленно; при отсутствии — блокировка и ожидание
- Используйте `return` для явного возврата значения (см. правила возврата)

```yaoxiang
# Управление зависимостями: b зависит от a, компилятор автоматически упорядочивает
result: Int = {
    b = a + 1      # зависит от a → автоматически после a
    a = 10         # без зависимостей → может выполниться первым
    return b       # возвращает 11
}
```

> **Отличие от одно-выражения**: `= expr` (без фигурных скобок) — простая привязка, возвращающая значение напрямую; `= { ... }` (с фигурными скобками) вводит контекст вычислений, управляемый зависимостями, допускающий несколько операторов и явный `return`.

#### Блок `spawn`

`spawn { ... }` — единственная параллельная примитива YaoXiang. Он использует семантику зависимостей `{}` для автоматического распараллеливания:

- Прямые дочерние присваивания внутри `spawn { ... }` автоматически создают параллельные задачи
- Задачи с готовыми зависимостями выполняются конкурентно немедленно
- Вызывающий блокируется в ожидании завершения всех дочерних задач

```yaoxiang
result = spawn {
    a = fetch_data("url1")    # задача 1
    b = fetch_data("url2")    # задача 2 (без зависимостей от a, выполняется параллельно)
    c = process(a, b)         # зависит от a, b → ждёт завершения обоих
    return c
}
// Вызывающий блокируется здесь до завершения всех задач в spawn-блоке
```

> **Подробное определение**: полная семантика `spawn`, правила создания задач и модель блокировки описаны в `008-runtime-concurrency-model.md`.

#### Блок `unsafe`

`unsafe { ... }` используется для определения непрозрачных типов и операций с сырыми указателями. Он использует семантику `return` блока `{}` для возврата определения типа во внешнюю область видимости:

**Основные правила**:
- В `unsafe {}` можно определять типы и работать с сырыми указателями
- `return` возвращает определение типа во внешнюю область видимости
- Возвращённый тип доступен за пределами `unsafe {}`
- Доступ к полям типа требует разрешения unsafe

```yaoxiang
# Определение непрозрачного типа в unsafe-блоке
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void  # сырой указатель
    }
    return SqliteDb
}

# SqliteDb доступен за пределами unsafe-блока
db = sqlite3_open("test.db")

# ❌ Ошибка компиляции: для доступа к полю handle требуется разрешение unsafe
handle = db.handle

# ✅ Через вызов метода
db.close()
```

> **Подробное определение**: полная семантика `unsafe`, определение FFI-типов и привязка методов описаны в `ffi.md`.

#### 3. Определение типов

Определение типов — ядро унифицированного синтаксиса YaoXiang, включающее поля, значения по умолчанию, привязанные методы и реализацию интерфейсов:

##### Базовые типы

**Тип записи**: список полей, типы полей могут быть любыми выражениями типов.

```yaoxiang
Point: Type = {
    x: Float,
    y: Float
}
```

**Поля со значениями по умолчанию**: поля могут иметь значения по умолчанию, необязательные при конструировании.

```yaoxiang
Point: Type = {
    x: Float = 0,
    y: Float = 0
}
```

Использование:

```yaoxiang
Point() → Point(x=0, y=0)
Point(x=1) → Point(x=1, y=0)
Point(x=1, y=2) → Point(x=1, y=2)
```

**Поля без значений по умолчанию**: должны быть предоставлены при конструировании.

```yaoxiang
Point2: Type = {
    x: Float,
    y: Float
}
```

Использование:
```yaoxiang
Point2(x=1, y=2) //✓
Point2() //✗
Point2(x=1) //✗
```

##### Привязка методов

**Способ 1: прямая привязка внешней функции внутри тела определения типа**

```yaoxiang
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]           // привязка к позиции 0, после каррирования method: (b: Point) -> Float
}
// Вызов: p1.distance(p2) → distance(p1, p2)
```

**Способ 2: анонимная функция + позиционная привязка**

```yaoxiang
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) → distance(p1, p2)
```

##### Определение функций пространства имён

**Префикс `Type.name` обозначает лишь принадлежность к пространству имён**, и ничего более. Он не вызывает никакой неявной привязки.

```yaoxiang
// Функция пространства имён: обычная функция в пространстве имён Point
Point.draw: (p: &Point, surface: Surface) -> Void = {
    surface.plot(p.x, p.y)
}

Point.serialize: (p: &Point) -> String = {
    return "Point(${p.x}, ${p.y})"
}

// Вызов: это обычный вызов функции
Point.draw(p, screen)
Point.serialize(p)
```

> **Примечание**: `self` — не ключевое слово, а лишь устоявшееся имя параметра. Запись в виде `p`, `this`, `x` даёт абсолютно тот же эффект.
> Компилятор смотрит не на имя параметра, а на его тип.

##### Привязка методов (единственный способ)

Чтобы синтаксис вызова метода через `.` вроде `p.draw(screen)` работал, **требуется явная привязка**.
Синтаксис `[position]` — единственный механизм привязки функции в качестве «метода» (подробный синтаксис см. в RFC-004).

```yaoxiang
// Определение функции
draw: (p: &Point, surface: Surface) -> Void = {
    surface.plot(p.x, p.y)
}

// Явная привязка — только после этого доступен синтаксис p.draw(screen)
Point.draw = draw[0]   // параметр в позиции 0 (&Point) заполняется вызывающим

// Использование
p.draw(screen)          // синтаксический сахар → draw(&p, screen)
Point.draw(p, screen)   // оба способа вызова эквивалентны

// Без [0] — без привязки. Point.draw — просто псевдоним обычной функции, синтаксис . недоступен
Point.draw = draw       // без привязки: только Point.draw(p, screen)
```

**Поведение по умолчанию**: без `[n]` — ни один параметр не привязан. Пользователь должен явно решить, какие параметры заполняются вызывающим.

**Множественная позиционная привязка**:

```yaoxiang
// Привязка нескольких позиций (автоматическое каррирование)
Point.transform = transform_points[0, 1]
// Вызов: p1.transform(p2)(2.0) → transform_points(p1, p2, 2.0)
```

**Обратная операция** (метод → обычная функция):

```yaoxiang
// Извлечение функции из привязки
draw_point: (p: &Point, surface: Surface) -> Void = Point.draw
```

#### 5. Дженерик-типы

```yaoxiang
// Базовые дженерики (RFC-011 Phase 1)
List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (T:Type)-((self: List(T), item: T) -> Void),
    get: (T:Type)->((self: List(T), index: Int) -> Maybe(T))
}

// Конкретизация (синтаксис RFC-023)
IntList: Type = List(Int)

IntList.push = {
    self.data.append(item)
    self.length = self.length + 1
}

List.push = (type: Type) -> {
    return (self: List(type), item: type) -> {
        self.data.append(item)
        self.length = self.length + 1
    }
}

IntList.push(Int)(self, item)  // Пример вызова

// Дженерик-метод (синтаксис RFC-023: параметры типа выводятся автоматически в точке вызова)
List.push: (self: List(T), item: T) -> Void = {
    self.data.append(item)
    self.length = self.length + 1
}

List.get: (self: List(T), index: Int) -> Maybe(T) = {
    if index >= 0 && index < self.length {
        return Maybe.Just(self.data[index])
    } else {
        return Maybe.Nothing
    }
}
```

#### 6. Синтаксис вызова дженериков

Для вызова дженерик-типов и дженерик-функций единообразно используется синтаксис `()`. `[]` ни в каком дженерик-контексте не используется.

**Основные правила**:

1. **`()` для всего применения**: применение типа, вызов функции, конструирование значения — всё через `()`

```yaoxiang
# Аннотация типа
numbers: List(Int) = List(1, 2, 3)

# Пустой контейнер: T берётся слева
empty: List(Int) = List()

# Вызов дженерик-функции — типы автоматически «перетекают» из параметров
strings = map(numbers, f)
// T=Int из numbers: List(Int)
// R=String из f: (Int) -> String
```

2. **Тип слева, значение справа**: `name: type = value` — параметр типа объявляется слева, справа всегда конкретное значение. `T` для пустого контейнера `List()` должен браться из аннотации типа слева.

3. **Информацию о типе нужно написать только один раз** — в объявлении параметра, и компилятор «протащит» её:

```yaoxiang
numbers: List(Int) = List(1, 2, 3)  // Int написан слева один раз
f: (Int) -> String = (x) => x.to_string()
strings = map(numbers, f)   // T=Int, R=String автоматически из типов numbers и f
```

4. **Конструирование значения выводит тип из элементов**:

```yaoxiang
x = List(1, 2, 3)       // выводится как List(Int)
y = List("a", "b")      // выводится как List(String)
z = List()              // ❌ Ошибка компиляции: T не может быть выведен
z: List(Int) = List()   // ✅ T=Int берётся из аннотации слева
```

5. **Псевдонимы типов**:

```yaoxiang
IntList: Type = List(Int)
StringToInt: Type = (String) -> Int
Matrix3x3: Type = Matrix(Float, 3, 3)
```

> **Сравнение со старым синтаксисом**: `List[Int]` → `List(Int)`, `List[Int]()` → `List()`, `List[Int](1,2,3)` → `List(1,2,3)`.
> Старый синтаксис дженериков с `[]` полностью удалён. `[]` используется только для литералов массивов/списков и индексированного доступа.

## Детальное проектирование

### Алгоритм проверки интерфейсов

```rust
fn check_type_implements_interface(
    typ: &Type,
    iface: &Type
) -> Result<(), TypeError> {
    // Для каждого поля интерфейса (поля-функции)
    for (field_name, iface_field) in &iface.fields {
        // Проверяем, есть ли в типе метод с тем же именем
        if let Some(method) = typ.methods.get(field_name) {
            // Проверяем совместимость сигнатуры метода
            // Поле интерфейса: (Surface) -> Void
            // Сигнатура метода: (Point, Surface) -> Void
            // Сравнение: после отбрасывания self должно совпасть
            if !method_signature_matches(method, iface_field.type_) {
                return Err(TypeError::MethodSignatureMismatch {
                    type_name: typ.name,
                    interface_name: iface.name,
                    method_name: field_name,
                });
            }
        } else {
            return Err(TypeError::MissingMethod {
                type_name: typ.name,
                interface_name: iface.name,
                method_name: field_name,
            });
        }
    }
    Ok(())
}
```

### Прямое присваивание интерфейсу и оптимизации времени компиляции

Типы интерфейсов поддерживают прямое присваивание; компилятор автоматически выберет оптимальную стратегию вызова на основе типа правой части:

```yaoxiang
// Прямое присваивание конкретного типа → конкретный тип может быть определён во время компиляции, вызов без накладных расходов
d: Drawable = Circle(1)
d.draw(screen)  // После компиляции: прямой вызов circle_draw(screen), без vtable

// Возвращаемое значение функции → конкретный тип не может быть определён во время компиляции, используется vtable
d: Drawable = get_shape()
d.draw(screen)  // Поиск метода через vtable

// Гетерогенная коллекция → используется vtable
shapes: List(Drawable) = [Circle(1), Rect(2, 3)]
for s in shapes {
    s.draw(screen)  // Поиск метода через vtable
}
```

**Стратегии оптимизации времени компиляции**:

| Сценарий | Результат вывода | Способ вызова |
|------|----------|----------|
| `d: Drawable = Circle(1)` | Конкретный тип Circle | Прямой вызов (без накладных расходов) |
| `d: Drawable = get_shape()` | Неизвестно | vtable |
| `shapes: List(Drawable) = [...]` | Гетерогенный | vtable |

**Правила**:
1. Когда правая часть является конструктором конкретного типа и может быть определена во время компиляции, генерируется IR с прямым вызовом
2. Когда тип правой части не может быть определён во время компиляции, откат на механизм vtable
3. vtable служит страховкой для корректности полиморфизма во время выполнения

### Изменения синтаксиса

| До | После |
|------|------|
| `type Point = Point(x: Float, y: Float)` | `Point: Type = { x: Float, y: Float }` |
| `type Result(T, E) = ok(T) \| err(E)` | `Result: (T: Type, E: Type) -> Type = { ok: (T) -> Result(T, E), err: (E) -> Result(T, E) }` |
| Требуется ключевое слово `impl` | Ключевое слово не нужно, имя интерфейса указывается в теле типа |

## Пояснение к проектированию синтаксиса: именованные функции по сути являются синтаксическим сахаром для лямбд

### Ключевое понимание

**Именованные функции и лямбда-выражения — это одно и то же!** Единственное отличие: именованная функция даёт лямбде имя.

```yaoxiang
// Эти две записи по сути полностью идентичны
add: (a: Int, b: Int) -> Int = a + b           // именованная функция (рекомендуется)
add: (a: Int, b: Int) -> Int = (a, b) => a + b        // лямбда-форма (полностью эквивалентна)
```

### Модель синтаксического сахара

```
// Именованная функция = лямбда + имя
name: (Params) -> ReturnType = body

// По сути это
name: (Params) -> ReturnType = (params) => body
```

**Ключевой момент**: когда в сигнатуре полностью объявлены типы параметров, заголовок лямбды с именами параметров становится избыточным и может быть опущен.

### Правила областей видимости параметров

**Параметры перекрывают внешние переменные**: область видимости параметров в сигнатуре перекрывает тело функции, внутренняя область видимости имеет более высокий приоритет.

```yaoxiang
x = 10  // внешняя переменная

double: (x: Int) -> Int = x * 2  // ✅ Параметр x перекрывает внешний x, результат равен 20
```

### Гибкость расположения аннотаций

Аннотации типов могут располагаться в любом из следующих мест, **достаточно указать хотя бы в одном**:

| Расположение аннотации | Форма | Описание |
|----------|------|------|
| Только в сигнатуре | `double: (x: Int) -> Int = x * 2` | ✅ Рекомендуется |
| Только в заголовке лямбды | `double = (x: Int) => x * 2` | ✅ Допустимо |
| В обоих местах | `double: (x: Int) -> Int = (x) => x * 2` | ✅ Избыточно, но разрешено |

### Полные примеры

```yaoxiang
// ✅ Рекомендуется: сигнатура полная, заголовок лямбды опущен
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1
main: () -> Void = { print("hi") }

// ✅ Допустимо: типы указаны в заголовке лямбды
double = (x: Int) => x * 2

// ✅ Допустимо: аннотации в обоих местах
double: (x: Int) -> Int = (x) => x * 2
```

### Преимущества проектирования

| Свойство | Преимущество |
|------|------|
| **Краткость** | При полной сигнатуре имена параметров не нужно дублировать |
| **Гибкость** | Сохранена лямбда-форма, используйте ту, что нравится |
| **Согласованность** | Сохранён единый шаблон с объявлением переменных `x: Int = 42` |
| **Интуитивность** | `name: Type = body` напрямую соответствует «имя name, тип Type, значение body» |

## Компромиссы

### Преимущества

| Преимущество | Описание |
|------|------|
| Предельное единство | Одно правило синтаксиса покрывает все случаи |
| Теоретическая элегантность | Идеально симметричное `name: type = value` |
| Без новых ключевых слов | Повторное использование существующих элементов синтаксиса |
| Простота реализации | Компилятору нужно обрабатывать только одну форму объявления |

| Простота расширения | Новые возможности естественно вписываются в эту модель |

| Недостаток | Описание |

| Сложность разбора | Унифицированный синтаксис может усложнить разбор | Использовать рекурсивно-нисходящий парсер |
| Накладные расходы производительности | Поиск в vtable может нести дополнительные расходы | Оптимизация мономорфизацией во время компиляции |

## Пасхалка 🎮: Исток языка

> ✨ **Type: Type = Type** ✨

```yaoxiang
// Попытка определить тип типа...
Type: Type = Type
```

**Внимание**: это **невыразимая** вещь!

```
╔══════════════════════════════════════════════════════════════╗
║                                                              ║
║   Одно порождает два, два порождает три, три порождает всё сущее.║
║   Предел порождает два, два — образы.                            ║
║                                                              ║
║   Type: Type = Type                                          ║
║   Это исток ЯоСяна, граница языка.                          ║
║   Здесь компилятор безмолвствует, философия останавливается.    ║
║                                                              ║
║   Спасибо, что достигли философской границы языка.             ║
║                                                              ║
╚══════════════════════════════════════════════════════════════╝
```

> **Примечание**: компилятор не может корректно обработать `Type: Type = Type` (это приведёт к парадоксу вселенных Type0/Type1), но мы намеренно сохранили этот «пасхальный яйцо» — когда вы попытаетесь его скомпилировать, вы получите дзенское послание от создателя языка. Это не только техническая граница, но и дань уважения YaoXiang философии типов.

## Приложение

### БНФ синтаксиса

```bnf
program ::= statement*

statement ::= declaration | expression

# Унифицированное объявление: name: Type = expression
declaration ::= identifier ':' type_expr '=' expression

# Выражение типа
type_expr ::= identifier
       | identifier '(' type_expr (',' type_expr)* ')'      # Применение типа
       | '(' type_expr (',' type_expr)* ')' '->' type_expr       # Тип функции
       | '{' type_field* '}'                       # Тип записи/интерфейса
       | 'Type'                                    # Метатип

type_field ::= identifier ':' type_expr
             | identifier                           # Ограничение интерфейса

# Параметр дженерика: как часть типа функции, например (T: Type, R: Type) -> (...)
# Не требует отдельного правила БНФ — параметр : Type является обычным параметром функции

# Выражение
expression ::= literal
              | identifier
              | identifier '(' expression (',' expression)* ')'  # Вызов функции / вызов конструктора
              | '(' expression (',' expression)* ')'              # Кортеж
              | expression '.' identifier '(' arguments? ')'    # Вызов метода
              | lambda
              | '{' field ':' expression (',' field ':' expression)* '}'

arguments ::= expression (',' expression)*

lambda ::= '(' parameter_list? ')' '=>' block

block ::= expression | '{' expression* '}'
```

### Глоссарий

| Термин | Определение |
|------|------|
| Объявление | Оператор присваивания в форме `name: type = value` |
| Тип записи | Тип `{ ... }`, содержащий именованные поля |
| Интерфейс | Тип записи, все поля которого имеют функциональный тип |
| Дженерик-тип | Тип, определённый как `Name: (T: Type) -> Type = { ... }`, принимающий параметры типа |
| Функция пространства имён | Функция в форме `Type.name`, принадлежащая пространству имён Type. Не подразумевает никакой привязки |
| Привязка метода | `Type.name = func[n]`, привязывает позицию n функции func как вызывающего, делая доступным синтаксис `obj.name(args)` |
| Дженерик-функция | Функция, использующая синтаксис `(T: Type)`, параметры типа являются первой группой параметров |
| Метатип | `Type`, единственная метка уровня типов в языке |
