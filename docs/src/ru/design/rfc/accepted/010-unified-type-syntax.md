# RFC-010: Унифицированный синтаксис типов — модель name: type = value


## Резюме

Настоящий RFC предлагает минималистичную унифицированную модель синтаксиса типов: **всё есть `name: type = value`**.

В YaoXiang существует только одна форма объявления:

```
identifier : type = expression
```

где `type` может быть любым выражением типа, а `expression` — любым выражением значения.
**Нет `fn`, нет `struct`, нет `trait`, нет `impl`, нет ключевого слова `type` в нижнем регистре (но есть `Type` как ключевое слово меta-типа)**.

> **Ключевая идея**: `Type` сам по себе является типовым параметром. `(T: Type) -> Type` означает «тип, принимающий параметр типа T».

| Понятие          | Запись в коде                                          |
|------------------|--------------------------------------------------------|
| Переменная       | `x: Int = 42`                                          |
| Функция          | `add: (a: Int, b: Int) -> Int = a + b`                 |
| Тип записи       | `Point: Type = { x: Float, y: Float }`                 |
| Интерфейс        | `Drawable: Type = { draw: (Surface) -> Void }`         |
| Типовой параметр | `List: (T: Type) -> Type = { data: Array(T), length: Int }` |
| Типовой параметр | `Map: (K: Type, V: Type) -> Type = { keys: Array(K), values: Array(V) }` |
| Метод            | `Point.draw: (p: Point, s: Surface) -> Void = ...`<br>`Point.draw = draw[0]` |
| Типовая функция  | `map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R))` |

**`Type` — единственное ключевое слово меta-типа в языке**.

> **Пространство имён vs привязка метода**: префикс `Type.name` обозначает **принадлежность к пространству имён**, и ничего более.
> Он не вызывает никакой неявной привязки. Чтобы синтаксис вызова через `.`, такой как `p.draw(screen)`, работал,
> требуется явная привязка: `Point.draw = draw[0]`.
> Подробности см. в разделе «Пространства имён и привязка методов».
Он используется для обозначения уровня типов; компилятор автоматически обрабатывает различие между Type0, Type1, Type2... и делает это прозрачным для пользователя.

```yaoxiang
// Основной синтаксис: унификация + различение

// Переменная
x: Int = 42

// Функция (имена параметров в сигнатуре)
add: (a: Int, b: Int) -> Int = a + b

// Тип записи
Point: Type = {
    x: Float,
    y: Float,
    draw: (Surface) -> Void,
    serialize: () -> String
}

// Интерфейс (по сути — тип записи, все поля которого являются функциями)
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Определение метода (используется синтаксис Type.method)
Point.draw: (self: Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}

Point.serialize: (self: Point) -> String = {
    return "Point(${self.x}, ${self.y})"
}

// Типовые параметры ((T: Type) -> Type = типовой параметр, принимающий аргумент типа)
List: (T: Type) -> Type = {
    data: Array(T),
    length: Int
}

Map: (K: Type, V: Type) -> Type = {
    keys: Array(K),
    values: Array(V)
}

// Использование
p: Point = Point(1.0, 2.0)
p.draw(screen)           // синтаксический сахар → Point.draw(p, screen)
s: Drawable = p           // структурное подтипирование: Point реализует Drawable
drawables: List(Drawable) = [p, r]
process_all(drawables)
```

## Мотивация

### Зачем нужна эта возможность?

Текущая система типов содержит несколько разрозненных концепций:
- Синтаксис объявления переменных
- Синтаксис определения функций
- Синтаксис определения типов (различный)
- Синтаксис определения интерфейсов
- Синтаксис привязки методов

Между этими концепциями отсутствует единство, что приводит к фрагментации синтаксиса и высокой стоимости обучения.

### Цели проектирования

1. **Предельное единство**: одно правило синтаксиса покрывает все случаи
2. **Простота и элегантность**: симметричная эстетика `name: type = value`
3. **Без новых ключевых слов**: повторное использование существующих элементов синтаксиса
4. **Теоретическая элегантность**: сам тип также является значением типа Type
5. **Типовая дружественность**: бесшовная интеграция с системой обобщений (RFC-011)

### Интеграция с системой обобщений

Унифицированная синтаксическая модель RFC-010 **органично сочетается** с дизайном системы обобщений RFC-011: параметры обобщений естественно вписываются в унифицированную модель:

```yaoxiang
// Базовые обобщения (RFC-011 Фаза 1)
List: (T: Type) -> Type = { data: Array(T), length: Int }

// Типовая функция (синтаксис RFC-023: позицию Type в сигнатуре можно опустить, вывод происходит в точке вызова)
map: (: Type, R: Type) -> (( list: List(T), f: (T) -> R) -> List(R)) = ...

// Ограничения типов (RFC-011 Фаза 2)
clone: (value: T) -> T = value.clone()  // ограничение T: Clone передаётся через тип параметра

// Const-обобщения (RFC-011 Фаза 4)
Array: (T: Type, N: Int) -> Type = { data: Array(T, N), length: N }
```

**Зависимости**:
- Фаза 1 RFC-011 (базовые обобщения) — **сильная зависимость** для RFC-010
- Без базовых обобщений примеры обобщений в RFC-010 не скомпилируются
- Рекомендация: реализовывать Фазу 1 RFC-011 и RFC-010 одновременно

## Предложение

### Ключевой принцип: конструктор типа vs функция/переменная

**Это ключевой выбор дизайна, определяющий правила устранения неоднозначности синтаксиса:**

| Запись | Значение | Правило |
|--------|----------|---------|
| **`x: Type = ...`** | Конструктор типа | `: Type` объявлено явно → принудительно тип |
| **`f = ...`** | Функция или переменная | Без `: Type` → HM активно выводит функцию/переменную |

**Почему такое решение?**

Сам синтаксис `{ ... }` неоднозначен:
- `{ x: Float, y: Float }` может быть **литералом типа** (тип записи)
- `{ a = 1 + 1 }` может быть **блоком кода** (исполняемый оператор, возвращающий Void)

**Правила устранения неоднозначности**:
- **Есть** `: Type` → принудительно интерпретируется как конструктор типа, `{ ... }` — литерал типа
- **Нет** `: Type` → HM активно интерпретирует `{ ... }` как блок кода, выводится тип функции

```yaoxiang
# ✅ Конструктор типа: есть : Type
Point: Type = { x: Float, y: Float }

# ✅ Функция: нет : Type, HM выводит () -> Void
main = { println("Hello") }

# ❌ Ошибка: нет : Type, компилятор не может интерпретировать { ... } как тип
Point = { x: Float, y: Float }  // HM выводит функцию, а не тип!
```

---

**Унифицированная модель: identifier : type = expression**

```
├── Переменная
│   └── x: Int = 42
│
├── Функция
│   └── add: (a: Int, b: Int) -> Int = a + b  # Нет : Type, HM выводит функцию
│
├── Тип записи
│   └── Point: Type = { x: Float, y: Float }  # Должно возвращать: Type
│
├── Интерфейс
│   └── Drawable: Type = { draw: (Surface) -> Void }  # Должно возвращать: Type
│
├── Типовой параметр
│   └── List: (T: Type) -> Type = { data: Array(T), length: Int }  # Должно возвращать: Type
│
├── Типовой параметр (несколько)
│   └── Map: (K: Type, V: Type) -> Type = { keys: Array(K), values: Array(V) }  # Должно возвращать: Type
│
├── Функция пространства имён
│   └── draw: (p: Point, surface: Surface) -> Void = ...
│       Point.draw = draw[0]  # Только после явной привязки доступен синтаксис вызова через точку
│
└── Типовая функция
    └── map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R))  # Не возвращает Type, HM выводит функцию
```

### Уровни меta-типов (внутри компилятора)

**Внутри компилятора** поддерживается иерархия вселенных `level: selfpointnum` (хранится в виде строки, теоретически бесконечно расширяемая).

| Уровень | Описание |
|---------|----------|
| `Type0` | Обычные типы (`Int`, `Float`, `Point`) |
| `Type1` | Конструкторы типов (`List`, `Maybe`) |
| `Type2+` | Конструкторы высшего порядка |

**Пользователь никогда не видит этих чисел**, только `: Type`.

### Изоморфизм Карри-Ховарда: тип — это утверждение, программа — это доказательство

Унифицированный синтаксис YaoXiang `name: type = value` выбран не произвольно — он представляет собой прямое отображение изоморфизма Карри-Ховарда (Curry-Howard correspondence). Этот изоморфизм раскрывает глубокий факт: **система типов и логическая система — это две стороны одной и той же сущности**.

| Логика (утверждение) | Система типов (YaoXiang) | Пример |
|---|---|---|
| Утверждение P | Тип T | `Int`, `Bool` |
| Доказательство P | Значение типа T | `42: Int`, `true: Bool` |
| P → Q (импликация) | Тип функции `(P) -> Q` | `(x: Int) -> Bool` |
| P ∧ Q (конъюнкция) | Тип записи `{ p: P, q: Q }` | `{ x: Int, y: Bool }` |
| ∀x.P(x) (квантор всеобщности) | Типовая функция `(T: Type) -> ...` | `map: (T: Type, R: Type) -> ...` |
| P ⊕ Q (дизъюнкция) | Перечисление / tagged union | `Maybe: (T: Type) -> Type = { ... }` |

**Значение `name: type = value` в контексте Карри-Ховарда**:

```yaoxiang
// "x: Int = 42" читается как: "существует доказательство типа Int по имени x со значением 42"
x: Int = 42

// "add: (a: Int, b: Int) -> Int = a + b" читается как:
// "существует доказательство импликации: даны доказательства a и b типа Int, можно построить доказательство типа Int"
add: (a: Int, b: Int) -> Int = a + b

// "Point: Type = { x: Float, y: Float }" читается как:
// "Point — это утверждение, доказательство которого требует одновременного предоставления доказательств x и y типа Float"
Point: Type = { x: Float, y: Float }
```

**Почему это важно?**

1. **Логическая непротиворечивость = типобезопасность**: если система типов позволяет создать значение типа `T` без какого-либо легитимного представления во время выполнения, это подобно тому, как в логике разрешают доказать ложное утверждение — система рушится. Карри-Ховард говорит нам: **типобезопасный язык по своей сути является непротиворечивой логической системой**.

2. **Иерархия вселенных — необходимое условие**: как подробно описано ниже, если разрешить `Type: Type` (т.е. «тип типа тоже является типом»), возникнет парадокс Рассела (в теории типов он проявляется как парадокс Жирара). Многоуровневая структура YaoXiang `Type₀ : Type₁ : Type₂ : ...` гарантирует, что каждый тип принадлежит только одному уровню, образуя бесконечно восходящую цепочку, что фундаментально предотвращает парадоксы. Это означает, что система типов YaoXiang **логически непротиворечива** в смысле Карри-Ховарда.

3. **Теоретическая основа унифицированного синтаксиса**: причина, по которой `name: type = value` может покрыть одной формой синтаксиса переменные, функции, типы, интерфейсы и обобщения, заключается в том, что все они в контексте Карри-Ховарда суть одно и то же — **предоставление доказательства для утверждения**. Переменная — это свидетельство утверждения, функция — свидетельство импликации, запись — свидетельство конъюнкции, обобщение — свидетельство квантора всеобщности. Унифицированный синтаксис — не случайность дизайна, а естественное следствие изоморфизма Карри-Ховарда.

> **Дополнительное чтение**: Wadler, P. (2015). *"Propositions as Types."* Communications of the ACM, 58(12), 75–84. Эта статья доступно объясняет историю и значение изоморфизма Карри-Ховарда.

### Определение синтаксиса

#### 1. Объявление переменных

```yaoxiang
// Базовый синтаксис
x: Int = 42
name: String = "Alice"
flag: Bool = true

// Вывод типа (можно опустить)
y = 100  // выводится как Int
```

#### 2. Определение функций

```yaoxiang
// Форма с одним выражением (возвращает значение напрямую, return не нужен)
add: (a: Int, b: Int) -> Int = a + b
greet: (name: String) -> String = "Hello, ${name}!"

// Форма с блоком кода (необходимо использовать return для возврата значения)
process: (x: Int) -> Int = {
    a = x * 2
    b = a + 1
    return b
}

// Многострочный блок кода
calc: (x: Float, y: Float, op: String) -> Float = {
    return match op {
        "+" -> x + y,
        "-" -> x - y,
        _ -> 0.0
    }
}

// Функция Void (внутри блока кода return не требуется)
print: (msg: String) -> Void = {
    console.write(msg)
}
```

#### Правила возврата

Возвращаемое значение зависит от формы правой части `=`:

| Запись | Возвращаемое значение |
|--------|----------------------|
| `= expr` (без фигурных скобок) | Возвращает `expr` напрямую |
| `= { ... }` (с фигурными скобками) | Требуется `return`, иначе возвращается `Void` |

```yaoxiang
# Одно выражение: возвращает значение напрямую, return не нужен
add: (a: Int, b: Int) -> Int = a + b

# Блок кода: для возврата значения необходим return
process: (x: Int) -> Int = {
    a = x * 2
    b = a + 1
    return b
}

# Функция Void: return не нужен
print: (msg: String) -> Void = {
    console.write(msg)
}
```

> **Обоснование дизайна**: `{ ... }` — это вычислительный блок, управляемый зависимостями (см. ниже), и его семантика возврата отличается от семантики одиночного выражения. Фигурные скобки вводят контекст нескольких операторов, поэтому требуется явный `return` для устранения неоднозначности «является ли последнее выражение возвращаемым значением».

#### Семантика `{}`: вычислительный блок, управляемый зависимостями

`{ ... }` в YaoXiang — это не просто блок кода, а **вычислительный блок, управляемый зависимостями**. Эта семантика остаётся единообразной в телах функций, инициализации переменных и `spawn`:

**Ключевые правила**:
- Операторы присваивания внутри `{}` автоматически упорядочиваются по зависимостям, а не по порядку записи
- Если зависимости удовлетворены, выполнение происходит немедленно; если нет — блокируется до готовности
- Используйте `return` для явного возврата значения (см. правила возврата)

```yaoxiang
# Управление зависимостями: b зависит от a, компилятор автоматически упорядочивает
result: Int = {
    b = a + 1      # зависит от a → автоматически после a
    a = 10         # без зависимостей → может выполняться первым
    return b       # возвращает 11
}
```

> **Отличие от одиночного выражения**: `= expr` (без фигурных скобок) — это простое связывание, возвращающее значение напрямую; `= { ... }` (с фигурными скобками) вводит контекст вычислений, управляемый зависимостями, допускающий несколько операторов и явный `return`.

#### Блок `spawn`

`spawn { ... }` — единственная параллельная примитивная конструкция в YaoXiang. Она использует семантику `{}`, управляемую зависимостями, для автоматического распараллеливания:

- Прямые присваивания внутри `spawn { ... }` автоматически создают параллельные задачи
- Задачи с удовлетворёнными зависимостями выполняются конкурентно
- Вызывающая сторона блокируется до завершения всех подзадач

```yaoxiang
result = spawn {
    a = fetch_data("url1")    # задача 1
    b = fetch_data("url2")    # задача 2 (нет зависимости от a, выполняется параллельно)
    c = process(a, b)         # зависит от a, b → ожидает завершения обоих
    return c
}
// вызывающая сторона блокируется здесь до завершения всех задач внутри spawn
```

> **Подробное определение**: полная семантика `spawn`, правила создания задач и модель блокировки описаны в `008-runtime-concurrency-model.md`.

#### Блок `unsafe`

`unsafe { ... }` используется для определения непрозрачных типов и операций с сырыми указателями. Он использует семантику `return` блока `{}` для возврата определения типа во внешнюю область видимости:

**Ключевые правила**:
- Внутри `unsafe {}` можно определять типы и работать с сырыми указателями
- Используйте `return` для возврата определения типа во внешнюю область видимости
- Возвращённый тип доступен за пределами `unsafe {}`
- Для доступа к полям типа требуются права unsafe

```yaoxiang
# Определение непрозрачного типа в блоке unsafe
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void  # сырой указатель
    }
    return SqliteDb
}

# SqliteDb доступен за пределами блока unsafe
db = sqlite3_open("test.db")

# ❌ Ошибка компиляции: для доступа к полю handle требуются права unsafe
handle = db.handle

# ✅ Через вызов метода
db.close()
```

> **Подробное определение**: полная семантика `unsafe`, определения FFI-типов и привязка методов описаны в `ffi.md`.

#### 3. Определение типов

Определение типов — ядро унифицированного синтаксиса YaoXiang, включающее поля, значения по умолчанию, привязанные методы и реализации интерфейсов:

##### Базовые типы

**Тип записи**: список полей, типы полей могут быть любыми выражениями типов.

```yaoxiang
Point: Type = {
    x: Float,
    y: Float
}
```

**Поля со значениями по умолчанию**: поля могут иметь значения по умолчанию, необязательные при конструировании.

```yaoxiang
Point: Type = {
    x: Float = 0,
    y: Float = 0
}
```

Использование:

```yaoxiang
Point() → Point(x=0, y=0)
Point(x=1) → Point(x=1, y=0)
Point(x=1, y=2) → Point(x=1, y=2)
```

**Поля без значений по умолчанию**: должны быть предоставлены при конструировании.

```yaoxiang
Point2: Type = {
    x: Float,
    y: Float
}
```

Использование:
```yaoxiang
Point2(x=1, y=2) //✓
Point2() //✗
Point2(x=1) //✗
```

##### Встроенные типы

Система идентификаторов YaoXiang разделена на три уровня, распознаваемых различными фазами компилятора:

1. **Ключевые слова** (отдельные токены парсера) — управляющие конструкции и ключевые слова объявления, такие как `if`, `match`, `pub`, `return`
2. **Зарезервированные слова литералов** (отдельные токены парсера) — `true`, `false`, `void`, `Type`, не могут использоваться как обычные идентификаторы
3. **Имена встроенных типов** (предварительно зарегистрированные в проверщике типов) — парсер рассматривает их как обычные идентификаторы, а проверщик типов отвечает за разрешение. **Не являются зарезервированными словами, могут быть затенены (не рекомендуется)**

Разница между `void` (в нижнем регистре, зарезервированное слово литерала) и `Void` (в верхнем регистре, имя встроенного типа): `void` — это литерал значения (равен единственному значению Unit), `Void` — это имя типа (равен типу Unit, логически ⊤). `let x: Void = void` — допустимая запись.

Предопределённые имена встроенных типов:

| Тип | Логический аналог | Описание |
|-----|-------------------|----------|
| `Never` | ⊥ (ложь/пустой тип) | Ноль конструкторов, ни одно значение не может населять этот тип. Означает «невозможно» — расходимость, паника, мёртвый код. `Never <: T` выполняется для любого `T` (принцип взрыва). Функция, возвращающая `Never`, никогда не возвращается нормально. **Не ключевое слово, а имя встроенного типа.** |
| `Void` | ⊤ (истина/Unit) | Ровно один обитатель (по умолчанию значение void). `x: Void = <default>` допустимо. Единичный элемент как для типа суммы, так и для типа произведения — `Void` — это тип произведения с нулём полей (Unit), `Never` — это тип суммы с нулём вариантов. |
| `Int` | — | Знаковое целое |
| `Float` | — | Число с плавающей точкой |
| `Bool` | — | Логическое значение: `true` / `false` |
| `Char` | — | Символ Unicode |
| `String` | — | Строка |

##### Привязка методов

**Способ 1: прямая привязка внешней функции внутри определения типа**

```yaoxiang
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]           // привязка к позиции 0, после каррирования method: (b: Point) -> Float
}
// Вызов: p1.distance(p2) → distance(p1, p2)
```

**Способ 2: анонимная функция + позиционная привязка**

```yaoxiang
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) → distance(p1, p2)
```

##### Определение функций пространства имён

**Префикс `Type.name` обозначает принадлежность к пространству имён**, и ничего более. Он не вызывает никакой неявной привязки.

```yaoxiang
// Функция пространства имён: обычная функция в пространстве имён Point
Point.draw: (p: &Point, surface: Surface) -> Void = {
    surface.plot(p.x, p.y)
}

Point.serialize: (p: &Point) -> String = {
    return "Point(${p.x}, ${p.y})"
}

// Вызов: это обычный вызов функции
Point.draw(p, screen)
Point.serialize(p)
```

> **Примечание**: `self` — не ключевое слово, а просто конвенциональное имя параметра. Запись `p`, `this`, `x` абсолютно эквивалентна.
> Компилятор смотрит не на имя параметра, а на его тип.

##### Привязка метода (единственный способ)

Чтобы синтаксис вызова метода через `.`, такой как `p.draw(screen)`, работал, **требуется явная привязка**.
Синтаксис `[position]` — единственный механизм привязки функции в качестве «метода» (подробный синтаксис см. в RFC-004).

```yaoxiang
// Определение функции
draw: (p: &Point, surface: Surface) -> Void = {
    surface.plot(p.x, p.y)
}

// Явная привязка — только после этого доступен синтаксис p.draw(screen)
Point.draw = draw[0]   // параметр на позиции 0 (&Point) заполняется вызывающей стороной

// Использование
p.draw(screen)          // синтаксический сахар → draw(&p, screen)
Point.draw(p, screen)   // оба способа вызова эквивалентны

// Без [0] = без привязки. Point.draw — просто псевдоним обычной функции, без синтаксиса .
Point.draw = draw       // без привязки: только Point.draw(p, screen)
```

**Поведение по умолчанию**: без `[n]` = ни один параметр не привязан. Пользователь должен явно решить, какие параметры заполняются вызывающей стороной.

**Множественная позиционная привязка**:

```yaoxiang
// Привязка нескольких позиций (автоматическое каррирование)
Point.transform = transform_points[0, 1]
// Вызов: p1.transform(p2)(2.0) → transform_points(p1, p2, 2.0)
```

**Обратная операция** (метод обратно в обычную функцию):

```yaoxiang
// Извлечение функции из привязки
draw_point: (p: &Point, surface: Surface) -> Void = Point.draw
```

#### 5. Типовые параметры

```yaoxiang
// Базовые обобщения (RFC-011 Фаза 1)
List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (T:Type)-((self: List(T), item: T) -> Void),
    get: (T:Type)->((self: List(T), index: Int) -> Maybe(T))
}

// Конкретная инстанциация (синтаксис RFC-023)
IntList: Type = List(Int)

IntList.push = {
    self.data.append(item)
    self.length = self.length + 1
}

List.push = (type: Type) -> {
    return (self: List(type), item: type) -> {
        self.data.append(item)
        self.length = self.length + 1
    }
}

IntList.push(Int)(self, item)  // пример вызова

// Обобщённый метод (синтаксис RFC-023: параметры типа выводятся в точке вызова)
List.push: (self: List(T), item: T) -> Void = {
    self.data.append(item)
    self.length = self.length + 1
}

List.get: (self: List(T), index: Int) -> Maybe(T) = {
    if index >= 0 && index < self.length {
        return Maybe.Just(self.data[index])
    } else {
        return Maybe.Nothing
    }
}
```

#### 6. Синтаксис вызова обобщений

Для вызова обобщённых типов и обобщённых функций единообразно используется синтаксис `()`. `[]` не используется ни в одном контексте обобщений.

**Ключевые правила**:

1. **`()` делает всё**: применение типа, вызов функции, конструирование значения — всё через `()`

```yaoxiang
# Аннотация типа
numbers: List(Int) = List(1, 2, 3)

# Пустой контейнер: T берётся слева
empty: List(Int) = List()

# Вызов обобщённой функции — типы автоматически «перетекают» из аргументов
strings = map(numbers, f)
// T=Int приходит из numbers: List(Int)
// R=String приходит из f: (Int) -> String
```

2. **Тип слева, значение справа**: `name: type = value` — параметры типа объявляются слева, справа всегда конкретные значения. T в пустом контейнере `List()` должен быть получен из аннотации типа слева.

3. **Информацию о типе нужно записать только один раз** — в объявлении параметра компилятор «проносит» её:

```yaoxiang
numbers: List(Int) = List(1, 2, 3)  // Int записан слева один раз
f: (Int) -> String = (x) => x.to_string()
strings = map(numbers, f)   // T=Int, R=String автоматически из типов numbers и f
```

4. **Тип значения-конструктора выводится из элементов**:

```yaoxiang
x = List(1, 2, 3)       // выводится как List(Int)
y = List("a", "b")      // выводится как List(String)
z = List()              // ❌ ошибка компиляции: невозможно вывести T
z: List(Int) = List()   // ✅ T=Int берётся из аннотации слева
```

5. **Псевдонимы типов**:

```yaoxiang
IntList: Type = List(Int)
StringToInt: Type = (String) -> Int
Matrix3x3: Type = Matrix(Float, 3, 3)
```

> **Сравнение со старым синтаксисом**: `List[Int]` → `List(Int)`, `List[Int]()` → `List()`, `List[Int](1,2,3)` → `List(1,2,3)`.
> Старый синтаксис обобщений с `[]` полностью удалён. `[]` используется только для литералов массивов/списков и доступа по индексу.

## Детальное проектирование

### Алгоритм проверки интерфейса

```rust
fn check_type_implements_interface(
    typ: &Type,
    iface: &Type
) -> Result<(), TypeError> {
    // Для каждого поля интерфейса (поля-функции)
    for (field_name, iface_field) in &iface.fields {
        // Проверяем, есть ли у типа метод с тем же именем
        if let Some(method) = typ.methods.get(field_name) {
            // Проверяем совместимость сигнатур методов
            // Поле интерфейса: (Surface) -> Void
            // Сигнатура метода: (Point, Surface) -> Void
            // Сравнение: должно совпадать после удаления self-параметра
            if !method_signature_matches(method, iface_field.type_) {
                return Err(TypeError::MethodSignatureMismatch {
                    type_name: typ.name,
                    interface_name: iface.name,
                    method_name: field_name,
                });
            }
        } else {
            return Err(TypeError::MissingMethod {
                type_name: typ.name,
                interface_name: iface.name,
                method_name: field_name,
            });
        }
    }
    Ok(())
}
```

### Прямое присваивание интерфейсу и оптимизация на этапе компиляции

Интерфейсные типы поддерживают прямое присваивание; компилятор автоматически выберет оптимальную стратегию вызова на основе типа правой части:

```yaoxiang
// Прямое присваивание конкретного типа → конкретный тип известен на этапе компиляции, вызов с нулевыми накладными расходами
d: Drawable = Circle(1)
d.draw(screen)  // после компиляции: прямой вызов circle_draw(screen), без vtable

// Возвращаемое значение функции → конкретный тип не может быть определён на этапе компиляции, используется vtable
d: Drawable = get_shape()
d.draw(screen)  // поиск метода через vtable

// Гетерогенная коллекция → используется vtable
shapes: List(Drawable) = [Circle(1), Rect(2, 3)]
for s in shapes {
    s.draw(screen)  // поиск метода через vtable
}
```

**Стратегии оптимизации на этапе компиляции**:

| Сценарий | Результат вывода | Способ вызова |
|----------|------------------|---------------|
| `d: Drawable = Circle(1)` | Конкретный тип Circle | Прямой вызов (нулевые накладные расходы) |
| `d: Drawable = get_shape()` | Неизвестно | vtable |
| `shapes: List(Drawable) = [...]` | Гетерогенный | vtable |

**Правила**:
1. Когда правая часть является конкретным конструктором типа и может быть определена на этапе компиляции, генерируется IR с прямым вызовом
2. Когда тип правой части не может быть определён на этапе компиляции, происходит откат к механизму vtable
3. vtable в качестве запасного варианта гарантирует корректность полиморфизма во время выполнения

### Изменения синтаксиса

| До | После |
|----|-------|
| `type Point = Point(x: Float, y: Float)` | `type Point = { x: Float, y: Float }` |

| Требуется ключевое слово `impl` | Ключевое слово не нужно, имена интерфейсов записываются в теле типа |

## Пояснение дизайна синтаксиса: именованные функции по сути являются синтаксическим сахаром для лямбда-выражений

### Ключевое понимание

**Именованные функции и лямбда-выражения — это одно и то же!** Единственное отличие: именованная функция дала лямбда-выражению имя.

```yaoxiang
// Эти две записи по сути полностью идентичны
add: (a: Int, b: Int) -> Int = a + b           // именованная функция (рекомендуется)
add: (a: Int, b: Int) -> Int = (a, b) => a + b        // лямбда-форма (полностью эквивалентна)
```

### Модель синтаксического сахара

```
// Именованная функция = лямбда + имя
name: (Params) -> ReturnType = body

// По сути это
name: (Params) -> ReturnType = (params) => body
```

**Ключевой момент**: когда сигнатура полностью объявляет типы параметров, имена параметров в заголовке лямбды становятся избыточными и могут быть опущены.

### Правила области видимости параметров

**Параметры затеняют внешние переменные**: область видимости параметров в сигнатуре перекрывает тело функции; приоритет внутренней области выше.

```yaoxiang
x = 10  // внешняя переменная

double: (x: Int) -> Int = x * 2  // ✅ параметр x затеняет внешний x, результат равен 20
```

### Гибкость расположения аннотаций

Аннотации типов могут находиться в любом из следующих мест, **достаточно указать хотя бы в одном**:

| Расположение аннотации | Форма | Пояснение |
|------------------------|-------|-----------|
| Только сигнатура | `double: (x: Int) -> Int = x * 2` | ✅ рекомендуется |
| Только заголовок лямбды | `double = (x: Int) => x * 2` | ✅ допустимо |
| С обеих сторон | `double: (x: Int) -> Int = (x) => x * 2` | ✅ избыточно, но разрешено |

### Полный пример

```yaoxiang
// ✅ Рекомендуется: сигнатура полная, заголовок лямбды опущен
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1
main: () -> Void = { print("hi") }

// ✅ Допустимо: типы указаны в заголовке лямбды
double = (x: Int) => x * 2

// ✅ Допустимо: аннотации с обеих сторон
double: (x: Int) -> Int = (x) => x * 2
```

### Преимущества дизайна

| Свойство | Преимущество |
|----------|--------------|
| **Краткость** | При полной сигнатуре не нужно повторять имена параметров |
| **Гибкость** | Сохранена лямбда-форма, используйте ту, которая нравится |
| **Единообразие** | Сохранён единый шаблон с объявлением переменных `x: Int = 42` |
| **Интуитивность** | `name: Type = body` напрямую соответствует «имя — name, тип — Type, значение — body» |

## Компромиссы

| Преимущество | Описание |

## Пасхалка 🎮: Исток языка

> ✨ **Type: Type = Type** ✨

```yaoxiang
// Попробуем определить тип типа...
Type: Type = Type
```

**Внимание**: это **невыразимая** вещь!

```
╔══════════════════════════════════════════════════════════════╗
║                                                              ║
║   Одно порождает два, два порождает три, три порождает тьму вещей.   ║
║   Предел порождает два образа.                                ║
║                                                              ║
║   Type: Type = Type                                          ║
║   Это исток YaoXiang, граница языка.                          ║
║   Здесь компилятор безмолвствует, философия останавливается.   ║
║                                                              ║
║   Спасибо, что достигли философской границы языка.            ║
║                                                              ║
╚══════════════════════════════════════════════════════════════╝
```

> **Примечание**: компилятор не может корректно обработать `Type: Type = Type` (это привело бы к парадоксу вселенных Type0/Type1), но мы намеренно сохранили этот «пасхальный яйцо» — при попытке его скомпилировать вы получите дзен-сообщение от создателя языка. Это не только техническая граница, но и дань уважения YaoXiang к философии типов.

## Приложение

### BNF-грамматика

```bnf
program ::= statement*

statement ::= declaration | expression

# Унифицированное объявление: name: Type = expression
declaration ::= identifier ':' type_expr '=' expression

# Выражение типа
type_expr ::= identifier
       | identifier '(' type_expr (',' type_expr)* ')'      # применение типа
       | '(' type_expr (',' type_expr)* ')' '->' type_expr       # тип функции
       | '{' type_field* '}'                       # тип записи/интерфейса
       | 'Type'                                    # меta-тип

type_field ::= identifier ':' type_expr
             | identifier                           # ограничение интерфейса

# Параметры обобщений: как часть типа функции, например (T: Type, R: Type) -> (...)
# Отдельное правило BNF не требуется — параметры : Type — это обычные параметры функции

# Выражение
expression ::= literal
              | identifier
              | identifier '(' expression (',' expression)* ')'  # вызов функции / вызов конструктора
              | '(' expression (',' expression)* ')'              # кортеж
              | expression '.' identifier '(' arguments? ')'    # вызов метода
              | lambda
              | '{' field ':' expression (',' field ':' expression)* '}'

arguments ::= expression (',' expression)*

lambda ::= '(' parameter_list? ')' '=>' block

block ::= expression | '{' expression* '}'
```

### Глоссарий

| Термин | Определение |
|--------|-------------|
| Объявление | Оператор присваивания в форме `name: type = value` |
| Тип записи | Тип `{ ... }`, содержащий именованные поля |
| Интерфейс | Тип записи, все поля которого являются типами функций |
| Типовой параметр | Тип, определённый как `Name: (T: Type) -> Type = { ... }`, принимающий аргументы типов |
| Функция пространства имён | Функция в форме `Type.name`, принадлежащая пространству имён Type. Не подразумевает никакой привязки |
| Привязка метода | `Type.name = func[n]`, привязывает позицию n функции func как вызывающую сторону, делая доступным синтаксис `obj.name(args)` |
| Типовая функция | Функция, использующая синтаксис `(T: Type)`, параметры типа передаются как первая группа параметров |
| Меta-тип | `Type`, единственный маркер уровня типов в языке |
