# RFC-010: Унифицированный синтаксис типов - модель name: type = value


## Резюме

Настоящий RFC предлагает предельно простую унифицированную модель синтаксиса типов: **всё есть `name: type = value`**.

YaoXiang имеет только одну форму объявления:

```
identifier : type = expression
```

где `type` может быть любым выражением типа, а `expression` — любым выражением значения.
**Нет `fn`, нет `struct`, нет `trait`, нет `impl`, нет ключевого слова `type` в нижнем регистре (но есть `Type` как ключевое слово метатипа)**.

> **Ключевое решение**: `Type` сам по себе является泛型 типом. `(T: Type) -> Type` означает «тип, принимающий параметр типа T».

| Понятие       | Запись в коде                                      |
|---------------|----------------------------------------------------|
| Переменная    | `x: Int = 42`                                      |
| Функция       | `add: (a: Int, b: Int) -> Int = a + b`             |
| Тип записи    | `Point: Type = { x: Float, y: Float }`             |
| Интерфейс     | `Drawable: Type = { draw: (Surface) -> Void }`     |
| 泛型 тип      | `List: (T: Type) -> Type = { data: Array(T), length: Int }` |
| 泛型 тип      | `Map: (K: Type, V: Type) -> Type = { keys: Array(K), values: Array(V) }` |
| Метод         | `Point.draw: (p: Point, s: Surface) -> Void = ...`<br>`Point.draw = draw[0]` |
| 泛型 функция  | `map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R))` |

**`Type` — единственное ключевое слово метатипа в языке**.

> **Пространство имён vs привязка метода**: префикс `Type.name` означает **принадлежность к пространству имён**, и ничего более.
> Он не вызывает никакой неявной привязки. Чтобы синтаксис вызова через `.` (например, `p.draw(screen)`) работал,
> необходима явная привязка: `Point.draw = draw[0]`.
> Подробности см. в разделе «Пространства имён и привязка методов» ниже.
Он используется для обозначения уровня типа, компилятор автоматически обрабатывает различие между Type0, Type1, Type2... прозрачно для пользователя.

```yaoxiang
// Основной синтаксис: унификация + различие

// Переменная
x: Int = 42

// Функция (имена параметров в сигнатуре)
add: (a: Int, b: Int) -> Int = a + b

// Тип записи
Point: Type = {
    x: Float,
    y: Float,
    draw: (Surface) -> Void,
    serialize: () -> String
}

// Интерфейс (по сути тип записи, все поля которой — функции)
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Определение метода (синтаксис Type.method)
Point.draw: (self: Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}

Point.serialize: (self: Point) -> String = {
    return "Point(${self.x}, ${self.y})"
}

// 泛型 тип ((T: Type) -> Type = 泛型 тип, принимающий параметры типа)
List: (T: Type) -> Type = {
    data: Array(T),
    length: Int
}

Map: (K: Type, V: Type) -> Type = {
    keys: Array(K),
    values: Array(V)
}

// Использование
p: Point = Point(1.0, 2.0)
p.draw(screen)           // синтаксический сахар → Point.draw(p, screen)
s: Drawable = p           // структурная подтипизация: Point реализует Drawable
drawables: List(Drawable) = [p, r]
process_all(drawables)
```

## Мотивация

### Зачем нужна эта возможность?

Текущая система типов содержит несколько разрозненных концепций:
- синтаксис объявления переменных;
- синтаксис определения функций;
- синтаксис определения типов (различный);
- синтаксис определения интерфейсов;
- синтаксис привязки методов.

Между этими концепциями отсутствует единство, что приводит к фрагментации синтаксиса и высокой стоимости обучения.

### Цели проектирования

1. **Предельное единство**: одно синтаксическое правило покрывает все случаи.
2. **Простота и элегантность**: симметричная эстетика `name: type = value`.
3. **Без новых ключевых слов**: повторное использование существующих элементов синтаксиса.
4. **Теоретическая элегантность**: сам тип является значением типа Type.
5. **Дружелюбность к 泛型**: бесшовная интеграция с системой 泛型 (RFC-011).

### Интеграция с системой 泛型

Унифицированная синтаксическая модель RFC-010 **органично согласуется** с дизайном системы 泛型 из RFC-011: параметры 泛型 естественно вписываются в унифицированную модель:

```yaoxiang
// Базовая 泛型 (RFC-011 Phase 1)
List: (T: Type) -> Type = { data: Array(T), length: Int }

// 泛型 функция (синтаксис RFC-023: позиция Type в сигнатуре может быть опущена, вывод на месте вызова)
map: (: Type, R: Type) -> (( list: List(T), f: (T) -> R) -> List(R)) = ...

// Ограничение типа (RFC-011 Phase 2)
clone: (value: T) -> T = value.clone()  // ограничение T: Clone несётся типом параметра

// Const 泛型 (RFC-011 Phase 4)
Array: (T: Type, N: Int) -> Type = { data: Array(T, N), length: N }
```

**Зависимости**:
- RFC-011 Phase 1 (базовая 泛型) является **сильной зависимостью** для RFC-010;
- без базовой 泛型 примеры 泛型 в RFC-010 не скомпилируются;
- рекомендация: реализовывать RFC-011 Phase 1 и RFC-010 синхронно.

## Предложение

### Ключевой принцип: конструктор типа vs функция/переменная

**Это ключевой выбор дизайна, определяющий правила устранения неоднозначности синтаксиса:**

| Запись | Значение | Правило |
|------|------|------|
| **`x: Type = ...`** | Конструктор типа | Явное объявление `: Type` → принудительно тип |
| **`f = ...`** | Функция или переменная | Без `: Type` → HM самостоятельно выводит функцию/переменную |

**Почему именно так?**

Синтаксис `{ ... }` сам по себе неоднозначен:
- `{ x: Float, y: Float }` может быть **литералом типа** (тип записи);
- `{ a = 1 + 1 }` может быть **блоком кода** (выполняемый оператор, возвращающий Void).

**Правила устранения неоднозначности**:
- **Есть** `: Type` → принудительно интерпретируется как конструктор типа, `{ ... }` — литерал типа;
- **Нет** `: Type` → HM активно интерпретирует `{ ... }` как блок кода, выводится тип функции.

```yaoxiang
# ✅ Конструктор типа: есть : Type
Point: Type = { x: Float, y: Float }

# ✅ Функция: нет : Type, HM выводит () -> Void
main = { println("Hello") }

# ❌ Ошибка: нет : Type, компилятор не может интерпретировать { ... } как тип
Point = { x: Float, y: Float }  // HM выводит функцию, а не тип!
```

---

**Унифицированная модель: identifier : type = expression**

```
├── Переменная
│   └── x: Int = 42
│
├── Функция
│   └── add: (a: Int, b: Int) -> Int = a + b  # нет : Type, HM выводит функцию
│
├── Тип записи
│   └── Point: Type = { x: Float, y: Float }  # должен возвращать: Type
│
├── Интерфейс
│   └── Drawable: Type = { draw: (Surface) -> Void }  # должен возвращать: Type
│
├── 泛型 тип
│   └── List: (T: Type) -> Type = { data: Array(T), length: Int }  # должен возвращать: Type
│
├── 泛型 тип (несколько параметров)
│   └── Map: (K: Type, V: Type) -> Type = { keys: Array(K), values: Array(V) }  # должен возвращать: Type
│
├── Функция пространства имён
│   └── draw: (p: Point, surface: Surface) -> Void = ...
│       Point.draw = draw[0]  # только после явной привязки доступен синтаксис вызова через точку
│
└── 泛型 функция
    └── map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R))  # не возвращает Type, HM выводит функцию
```

### Иерархия метатипов (внутри компилятора)

**Внутри компилятора** поддерживается иерархия вселенной `level: selfpointnum` (хранится в виде строки, теоретически может быть бесконечно расширяемой).

| Level | Описание |
|-------|------|
| `Type0` | Обычные типы (`Int`, `Float`, `Point`) |
| `Type1` | Конструкторы типов (`List`, `Maybe`) |
| `Type2+` | Конструкторы высшего порядка |

**Пользователь никогда не видит этих чисел**, он видит только `: Type`.

### Изоморфизм Карри-Ховарда: тип есть утверждение, программа есть доказательство

Унифицированный синтаксис YaoXiang `name: type = value` выбран не произвольно — он представляет собой прямое отображение изоморфизма Карри-Ховарда (Curry-Howard correspondence). Этот изоморфизм раскрывает глубокий факт: **система типов и логическая система — это две стороны одного и того же**.

| Логика (утверждение) | Система типов (YaoXiang) | Пример |
|---|---|---|
| Утверждение P | Тип T | `Int`, `Bool` |
| Доказательство истинности P | Значение типа T | `42: Int`, `true: Bool` |
| P → Q (импликация) | Тип функции `(P) -> Q` | `(x: Int) -> Bool` |
| P ∧ Q (конъюнкция) | Тип записи `{ p: P, q: Q }` | `{ x: Int, y: Bool }` |
| ∀x.P(x) (квантор всеобщности) | 泛型 функция `(T: Type) -> ...` | `map: (T: Type, R: Type) -> ...` |
| P ⊕ Q (дизъюнкция) | Перечисление / tagged union | `Maybe: (T: Type) -> Type = { ... }` |

**Значение `name: type = value` в свете Карри-Ховарда**:

```yaoxiang
// "x: Int = 42" читается как: "существует доказательство типа Int с именем x, значение которого 42"
x: Int = 42

// "add: (a: Int, b: Int) -> Int = a + b" читается как:
// "существует доказательство импликации: даны доказательства a и b типа Int, можно построить доказательство типа Int"
add: (a: Int, b: Int) -> Int = a + b

// "Point: Type = { x: Float, y: Float }" читается как:
// "Point — это утверждение, для доказательства которого необходимо одновременно предоставить доказательство x типа Float и доказательство y типа Float"
Point: Type = { x: Float, y: Float }
```

**Почему это важно?**

1. **Логическая непротиворечивость = безопасность типов**: если система типов позволяет создать значение типа `T` без какого-либо допустимого представления во время выполнения, это подобно тому, как в логике разрешают доказать ложное утверждение — система рушится. Карри-Ховард говорит нам: **язык с безопасной системой типов по своей сути является непротиворечивой логической системой**.

2. **Иерархия вселенных — необходимое условие**: как подробно описано ниже, если разрешить `Type: Type` (то есть «тип типа также является типом»), возникнет парадокс Рассела (в теории типов проявляется как парадокс Жирара). Слоистая структура YaoXiang `Type₀ : Type₁ : Type₂ : ...` гарантирует, что каждый тип принадлежит строго определённому уровню, формируя бесконечно восходящую цепочку, что фундаментально предотвращает парадоксы. Это означает, что система типов YaoXiang **логически непротиворечива** в смысле Карри-Ховарда.

3. **Теоретическая основа унифицированного синтаксиса**: то, что `name: type = value` способен одним синтаксисом покрыть переменные, функции, типы, интерфейсы и 泛型, объясняется тем, что все они в свете Карри-Ховарда суть одно и то же — **предоставление доказательства для утверждения**. Переменная — свидетельство утверждения, функция — свидетельство импликации, запись — свидетельство конъюнкции, 泛型 — свидетельство квантора всеобщности. Унифицированный синтаксис — не случайность дизайна, а естественное следствие изоморфизма Карри-Ховарда.

> **Дополнительная литература**: Wadler, P. (2015). *"Propositions as Types."* Communications of the ACM, 58(12), 75–84. Эта статья объясняет доступным языком историю и значение изоморфизма Карри-Ховарда.

### Определение синтаксиса

#### 1. Объявление переменных

```yaoxiang
// Базовый синтаксис
x: Int = 42
name: String = "Alice"
flag: Bool = true

// Вывод типа (можно опустить)
y = 100  // выводится как Int
```

#### 2. Определение функций

```yaoxiang
// Форма одного выражения (возвращает значение напрямую, без return)
add: (a: Int, b: Int) -> Int = a + b
greet: (name: String) -> String = "Hello, ${name}!"

// Форма блока кода (необходимо использовать return для возврата значения)
process: (x: Int) -> Int = {
    a = x * 2
    b = a + 1
    return b
}

// Многострочный блок кода
calc: (x: Float, y: Float, op: String) -> Float = {
    return match op {
        "+" -> x + y,
        "-" -> x - y,
        _ -> 0.0
    }
}

// Функция Void (return внутри блока не требуется)
print: (msg: String) -> Void = {
    console.write(msg)
}
```

#### Правила возврата

Возвращаемое значение зависит от формы правой части `=`:

| Запись | Возвращаемое значение |
|------|--------|
| `= expr` (без фигурных скобок) | Возвращается `expr` напрямую |
| `= { ... }` (с фигурными скобками) | Обязателен `return`, иначе возвращается `Void` |

```yaoxiang
# Одно выражение: возвращается напрямую, return не нужен
add: (a: Int, b: Int) -> Int = a + b

# Блок кода: для возврата значения обязателен return
process: (x: Int) -> Int = {
    a = x * 2
    b = a + 1
    return b
}

# Функция Void: return не нужен
print: (msg: String) -> Void = {
    console.write(msg)
}
```

> **Обоснование дизайна**: `{ ... }` — это зависимо-управляемая вычислительная единица (см. ниже), семантика возврата которой отличается от одиночного выражения. Фигурные скобки вводят контекст нескольких операторов, поэтому требуется явный `return` для устранения неоднозначности «является ли последнее выражение возвращаемым значением».

#### Семантика `{}`: зависимо-управляемая вычислительная единица

В YaoXiang `{ ... }` — не просто блок кода, а **зависимо-управляемая вычислительная единица**. Эта семантика остаётся единообразной в телах функций, инициализаторах переменных и `spawn`:

**Ключевые правила**:
- Операторы присваивания внутри `{}` автоматически упорядочиваются по зависимостям, а не по порядку записи;
- при готовности зависимостей выполнение происходит немедленно, при их отсутствии — блокировка с ожиданием;
- для явного возврата значения используется `return` (см. правила возврата).

```yaoxiang
# Управление по зависимостям: b зависит от a, компилятор автоматически упорядочит
result: Int = {
    b = a + 1      # зависит от a → автоматически после a
    a = 10         # нет зависимостей → может выполняться первым
    return b       # возвращает 11
}
```

> **Отличие от одиночного выражения**: `= expr` (без фигурных скобок) — простое связывание с прямым возвратом значения; `= { ... }` (с фигурными скобками) вводит контекст зависимо-управляемого вычисления, допускающий несколько операторов и явный `return`.

#### Блок `spawn`

`spawn { ... }` — единственная примитивная конструкция параллелизма в YaoXiang. Она использует зависимо-управляемую семантику `{}` для автоматического распараллеливания:

- прямые дочерние присваивания внутри `spawn { ... }` автоматически создают параллельные задачи;
- задачи с готовыми зависимостями немедленно выполняются конкурентно;
- вызывающий блокируется до завершения всех подзадач.

```yaoxiang
result = spawn {
    a = fetch_data("url1")    # задача 1
    b = fetch_data("url2")    # задача 2 (нет зависимостей от a, выполняется параллельно)
    c = process(a, b)         # зависит от a, b → ожидает их завершения
    return c
}
// вызывающий блокируется здесь до завершения всех задач в spawn-блоке
```

> **Подробное определение**: полная семантика `spawn`, правила создания задач и модель блокировки описаны в `008-runtime-concurrency-model.md`.

#### Блок `unsafe`

`unsafe { ... }` используется для определения непрозрачных типов и работы с сырыми указателями. Он использует семантику `return` блока `{}` для возврата определения типа во внешнюю область видимости:

**Ключевые правила**:
- внутри `unsafe {}` можно определять типы и работать с сырыми указателями;
- `return` возвращает определение типа во внешнюю область видимости;
- возвращённый тип доступен за пределами `unsafe {}`;
- доступ к полям типа требует разрешения unsafe.

```yaoxiang
# Определение непрозрачного типа внутри unsafe-блока
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void  # сырой указатель
    }
    return SqliteDb
}

# SqliteDb доступен за пределами unsafe-блока
db = sqlite3_open("test.db")

# ❌ Ошибка компиляции: для доступа к полю handle требуется разрешение unsafe
handle = db.handle

# ✅ Через вызов метода
db.close()
```

> **Подробное определение**: полная семантика `unsafe`, определение типов FFI и привязка методов описаны в `ffi.md`.

#### 3. Определение типов

Определение типа — ядро унифицированного синтаксиса YaoXiang, включающее поля, значения по умолчанию, привязанные методы и реализации интерфейсов:

##### Базовые типы

**Тип записи**: список полей, типы полей могут быть любыми выражениями типа.

```yaoxiang
Point: Type = {
    x: Float,
    y: Float
}
```

**Поля со значениями по умолчанию**: поля могут иметь значения по умолчанию, при конструировании они необязательны.

```yaoxiang
Point: Type = {
    x: Float = 0,
    y: Float = 0
}
```

Использование:

```yaoxiang
Point() → Point(x=0, y=0)
Point(x=1) → Point(x=1, y=0)
Point(x=1, y=2) → Point(x=1, y=2)
```

**Поля без значений по умолчанию**: должны быть предоставлены при конструировании.

```yaoxiang
Point2: Type = {
    x: Float,
    y: Float
}
```

Использование:
```yaoxiang

```

##### Встроенные типы

Система идентификаторов YaoXiang состоит из трёх слоёв, распознаваемых на разных фазах компилятора:

1. **Ключевые слова** (отдельные токены парсера) — управляющие конструкции и ключевые слова объявлений, такие как `if`, `match`, `pub`, `return`.
2. **Зарезервированные слова литералов** (отдельные токены парсера) — `true`, `false`, `void`, `Type`, не могут использоваться как обычные идентификаторы.
3. **Имена встроенных типов** (предварительно зарегистрированные в проверщике типов) — парсер обрабатывает их как обычные идентификаторы, разбор выполняет проверщик типов. **Не являются зарезервированными словами и могут быть затенены (не рекомендуется)**.

Различие между `void` (в нижнем регистре, зарезервированное слово литерала) и `Void` (в верхнем регистре, имя встроенного типа): `void` — литерал значения (равен единственному значению Unit), `Void` — имя типа (равен типу Unit, логически ⊤). `let x: Void = void` — допустимая запись.

Предопределённые имена встроенных типов:

| Тип | Логический аналог | Описание |
|------|---------|------|
| `Never` | ⊥ (ложь/пустой тип) | Ноль конструкторов, ни одно значение не может населять этот тип. Означает «невозможно» — расходимость, паника, мёртвый код. `Never <: T` выполняется для любого `T` (принцип взрыва). Функция, возвращающая `Never`, никогда не возвращается нормально. **Не ключевое слово, а имя встроенного типа.** |
| `Void` | ⊤ (истина/Unit) | Ровно один обитатель (по умолчанию значение void). `x: Void = <по умолчанию>` допустимо. Аналог единицы для типов-сумм и единицы для типов-произведений — `Void` — это произведение нуля полей (Unit), `Never` — сумма нуля вариантов. |
| `Int` | — | Знаковое целое |
| `Float` | — | Число с плавающей точкой |
| `Bool` | — | Логическое значение: `true` / `false` |
| `Char` | — | Символ Unicode |
| `String` | — | Строка |

##### Привязка методов

**Способ 1: прямая привязка внешней функции внутри определения типа**

```yaoxiang
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]           // привязка к позиции 0, после каррирования method: (b: Point) -> Float
}
// Вызов: p1.distance(p2) → distance(p1, p2)
```

**Способ 2: анонимная функция + позиционная привязка**

```yaoxiang
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) → distance(p1, p2)
```

**Префикс `Type.name` означает принадлежность к пространству имён**, и ничего более. Он не вызывает никакой неявной привязки.

```yaoxiang
// Функция пространства имён: обычная функция в пространстве имён Point

}

Point.serialize: (p: &Point) -> String = {

}

// Вызов: это обычный вызов функции
Point.draw(p, screen)
Point.serialize(p)
```

> **Примечание**: `self` не является ключевым словом, это лишь устоявшееся имя параметра. Запись в виде `p`, `this`, `x` полностью эквивалентна.
> Компилятор смотрит на тип, а не на имя параметра.

##### Привязка методов (единственный способ)

Чтобы синтаксис вызова метода через `.` (например, `p.draw(screen)`) работал, **необходима явная привязка**.
Синтаксис `[position]` — единственный механизм привязки функции в качестве «метода» (подробный синтаксис см. в RFC-004).

```yaoxiang
// Определение функции
draw: (p: &Point, surface: Surface) -> Void = {
    surface.plot(p.x, p.y)
}

// Явная привязка — только после неё доступен синтаксис p.draw(screen)
Point.draw = draw[0]   // параметр на позиции 0 (&Point) заполняется вызывающим

// Использование
p.draw(screen)          // синтаксический сахар → draw(&p, screen)
Point.draw(p, screen)   // обе формы вызова эквивалентны

// Без [0] = без привязки. Point.draw — просто псевдоним обычной функции, синтаксис . недоступен
Point.draw = draw       // без привязки: только Point.draw(p, screen)
```

**Поведение по умолчанию**: без `[n]` = ни один параметр не привязан. Пользователь должен явно решить, какие параметры заполняются вызывающим.

**Многопозиционная привязка**:

```yaoxiang
// Привязка нескольких позиций (автоматическое каррирование)
Point.transform = transform_points[0, 1]
// Вызов: p1.transform(p2)(2.0) → transform_points(p1, p2, 2.0)
```

**Обратная операция** (преобразование метода в обычную функцию):

```yaoxiang
// Извлечение функции из привязки
draw_point: (p: &Point, surface: Surface) -> Void = Point.draw
```

#### 5. 泛型 типы

```yaoxiang
// Базовая 泛型 (RFC-011 Phase 1)
List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (T:Type)-((self: List(T), item: T) -> Void),
    get: (T:Type)->((self: List(T), index: Int) -> Maybe(T))
}

// Конкретизация (синтаксис RFC-023)
IntList: Type = List(Int)

IntList.push = {
    self.data.append(item)
    self.length = self.length + 1
}

List.push = (type: Type) -> {
    return (self: List(type), item: type) -> {
        self.data.append(item)
        self.length = self.length + 1
    }
}

IntList.push(Int)(self, item)  // пример вызова

// 泛型 метод (синтаксис RFC-023: параметры типа выводятся автоматически в месте вызова)
List.push: (self: List(T), item: T) -> Void = {
    self.data.append(item)
    self.length = self.length + 1
}

List.get: (self: List(T), index: Int) -> Maybe(T) = {
    if index >= 0 && index < self.length {
        return Maybe.Just(self.data[index])
    } else {
        return Maybe.Nothing
    }
}
```

#### 6. Синтаксис вызова 泛型

Для вызова 泛型 типов и 泛型 функций единообразно используется синтаксис `()`. `[]` не используется ни в каком контексте 泛ка.

**Ключевые правила**:

1. **`()` для всего**: применение типа, вызов функции, конструирование значения — всё через `()`

```yaoxiang
# Аннотация типа
numbers: List(Int) = List(1, 2, 3)

# Пустой контейнер: T берётся слева
empty: List(Int) = List()

# Вызов 泛型 функции — типы выводятся из параметров автоматически
strings = map(numbers, f)
// T=Int берётся из numbers: List(Int)
// R=String берётся из f: (Int) -> String
```

2. **Type слева, значение справа**: `name: type = value` — параметры типа объявляются слева, справа всегда конкретные значения. Для пустого контейнера `List()` значение `T` должно браться из аннотации типа слева.

3. **Информацию о типе нужно записать лишь однажды** — при объявлении параметра компилятор проносит её далее:

```yaoxiang
numbers: List(Int) = List(1, 2, 3)  // Int записан один раз слева
f: (Int) -> String = (x) => x.to_string()
strings = map(numbers, f)   // T=Int, R=String выводятся автоматически из типов numbers и f
```

4. **Конструирование значения выводит тип из элементов**:

```yaoxiang
x = List(1, 2, 3)       // выводится как List(Int)
y = List("a", "b")      // выводится как List(String)
z = List()              // ❌ ошибка компиляции: невозможно вывести T
z: List(Int) = List()   // ✅ T=Int берётся из аннотации слева
```

5. **Псевдонимы типов**:

```yaoxiang
IntList: Type = List(Int)
StringToInt: Type = (String) -> Int
Matrix3x3: Type = Matrix(Float, 3, 3)
```

> **Сравнение со старым синтаксисом**: `List[Int]` → `List(Int)`, `List[Int]()` → `List()`, `List[Int](1,2,3)` → `List(1,2,3)`.
> Старый синтаксис 泛型 с `[]` полностью удалён. `[]` используется только для литералов массивов/списков и доступа по индексу.

## Детальный дизайн

### Алгоритм проверки интерфейсов

```rust
fn check_type_implements_interface(
    typ: &Type,
    iface: &Type
) -> Result<(), TypeError> {
    // Для каждого поля интерфейса (поля-функции)
    for (field_name, iface_field) in &iface.fields {
        // Проверяем, есть ли в типе метод с тем же именем
        if let Some(method) = typ.methods.get(field_name) {
            // Проверяем совместимость сигнатур методов
            // Поле интерфейса: (Surface) -> Void
            // Сигнатура метода: (Point, Surface) -> Void
            // Сравнение: после удаления параметра self должно совпадать
            if !method_signature_matches(method, iface_field.type_) {
                return Err(TypeError::MethodSignatureMismatch {
                    type_name: typ.name,
                    interface_name: iface.name,
                    method_name: field_name,
                });
            }
        } else {
            return Err(TypeError::MissingMethod {
                type_name: typ.name,
                interface_name: iface.name,
                method_name: field_name,
            });
        }
    }
    Ok(())
}
```

### Прямое присваивание интерфейсу и оптимизации времени компиляции

Интерфейсный тип поддерживает прямое присваивание. Компилятор автоматически выбирает оптимальную стратегию вызова на основе типа правой части:

```yaoxiang
// Прямое присваивание конкретного типа → конкретный тип известен в момент компиляции, вызов без накладных расходов
d: Drawable = Circle(1)
d.draw(screen)  // после компиляции: прямой вызов circle_draw(screen), без vtable

// Возвращаемое значение функции → конкретный тип не известен в момент компиляции, используется vtable
d: Drawable = get_shape()
d.draw(screen)  // поиск метода через vtable

// Гетерогенная коллекция → используется vtable
shapes: List(Drawable) = [Circle(1), Rect(2, 3)]
for s in shapes {
    s.draw(screen)  // поиск метода через vtable
}
```

**Стратегии оптимизации времени компиляции**:

| Сценарий | Результат вывода | Способ вызова |
|------|----------|----------|
| `d: Drawable = Circle(1)` | Конкретный тип Circle | Прямой вызов (без накладных расходов) |
| `d: Drawable = get_shape()` | Неизвестно | vtable |
| `shapes: List(Drawable) = [...]` | Гетерогенный | vtable |

**Правила**:
1. Когда правая часть — конструктор конкретного типа и известен в момент компиляции, генерируется IR с прямым вызовом.
2. Когда тип правой части невозможно определить в момент компиляции, выполняется откат на механизм vtable.
3. vtable служит страховкой для корректности полиморфизма во время выполнения.

## Объяснение дизайна синтаксиса: именованная функция по сути является синтаксическим сахаром для лямбды

### Ключевое понимание

**Именованная функция и лямбда-выражение — это одно и то же!** Единственное отличие: именованная функция даёт лямбде имя.

```yaoxiang
// Эти две записи по сути полностью идентичны
add: (a: Int, b: Int) -> Int = a + b           // именованная функция (рекомендуется)
add: (a: Int, b: Int) -> Int = (a, b) => a + b        // лямбда-форма (полностью эквивалентна)
```

### Модель синтаксического сахара

```
// Именованная функция = лямбда + имя
name: (Params) -> ReturnType = body

// По сути это
name: (Params) -> ReturnType = (params) => body
```

**Ключевой момент**: когда сигнатура полностью объявляет типы параметров, имена параметров в заголовке лямбды становятся избыточными и могут быть опущены.

### Правила области видимости параметров

**Параметры перекрывают внешние переменные**: область видимости параметров в сигнатуре перекрывает тело функции, внутренняя область видимости имеет более высокий приоритет.

```yaoxiang
x = 10  // внешняя переменная

double: (x: Int) -> Int = x * 2  // ✅ параметр x перекрывает внешний x, результат равен 20
```

### Гибкость позиции аннотации

Аннотации типов могут находиться в любой из следующих позиций, **достаточно аннотации хотя бы в одной**:

| Позиция аннотации | Форма | Описание |
|----------|------|------|
| Только в сигнатуре | `double: (x: Int) -> Int = x * 2` | ✅ рекомендуется |
| Только в заголовке лямбды | `double = (x: Int) => x * 2` | ✅ допустимо |
| Аннотации в обоих местах | `double: (x: Int) -> Int = (x) => x * 2` | ✅ избыточно, но разрешено |

### Полный пример

```yaoxiang
// ✅ Рекомендуется: сигнатура полная, заголовок лямбды опускается
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1
main: () -> Void = { print("hi") }

// ✅ Допустимо: типы аннотированы в заголовке лямбды
double = (x: Int) => x * 2

// ✅ Допустимо: аннотации в обоих местах
double: (x: Int) -> Int = (x) => x * 2
```

### Преимущества дизайна

| Свойство | Преимущество |
|------|------|
| **Краткость** | При полной сигнатуре нет необходимости повторять имена параметров |
| **Гибкость** | Сохранена лямбда-форма, используйте ту, которая нравится |
| **Единообразие** | Совпадение с шаблоном объявления переменной `x: Int = 42` |
| **Интуитивность** | `name: Type = body` напрямую соответствует «имя name, тип Type, значение body» |

**Предупреждение**: это **невыразимая** вещь!

## Приложение

### BNF синтаксиса

```bnf
program ::= statement*

statement ::= declaration | expression

# Унифицированное объявление: name: Type = expression
declaration ::= identifier ':' type_expr '=' expression

# Выражение типа
type_expr ::= identifier
       | identifier '(' type_expr (',' type_expr)* ')'      # Применение типа
       | '(' type_expr (',' type_expr)* ')' '->' type_expr       # Тип функции
       | '{' type_field* '}'                       # Тип записи / интерфейса
       | 'Type'                                    # Метатип

type_field ::= identifier ':' type_expr
             | identifier                           # Ограничение интерфейса

# Параметры 泛ка: как часть типа функции, например (T: Type, R: Type) -> (...)
# Отдельное правило BNF не требуется — параметры : Type — это обычные параметры функции

# Выражение
expression ::= literal
              | identifier
              | identifier '(' expression (',' expression)* ')'  # Вызов функции / вызов конструктора
              | '(' expression (',' expression)* ')'              # Кортеж
              | expression '.' identifier '(' arguments? ')'    # Вызов метода
              | lambda
              | '{' field ':' expression (',' field ':' expression)* '}'

arguments ::= expression (',' expression)*

lambda ::= '(' parameter_list? ')' '=>' block

block ::= expression | '{' expression* '}'
```

### Глоссарий

| Термин | Определение |
|------|------|
| Объявление | Оператор присваивания в форме `name: type = value` |
| Тип записи | Тип `{ ... }`, содержащий именованные поля |
| Интерфейс | Тип записи, все поля которой имеют тип функции |
| 泛型 тип | Тип, определённый как `Name: (T: Type) -> Type = { ... }`, принимающий параметры типа |
| Функция пространства имён | Функция в форме `Type.name`, принадлежащая пространству имён Type. Не подразумевает никакой привязки |
| Привязка метода | `Type.name = func[n]`, привязывающая позицию n функции func в качестве вызывающего, что делает доступным синтаксис `obj.name(args)` |
| 泛型 функция | Функция, использующая синтаксис `(T: Type)`, параметры типа являются первой группой параметров |
| Метатип | `Type`, единственный маркер уровня типа в языке |
