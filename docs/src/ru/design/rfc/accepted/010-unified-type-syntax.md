# RFC-010: Унифицированный синтаксис типов - модель name: type = value


## Аннотация

Данный RFC предлагает предельно минималистичную унифицированную модель синтаксиса типов: **всё есть `name: type = value`**.

В YaoXiang существует только одна форма объявления:

```
identifier : type = expression
```

Здесь `type` может быть любым тип-выражением, а `expression` — любым выражением-значением.
**Нет `fn`, нет `struct`, нет `trait`, нет `impl`, нет ключевого слова `type` в нижнем регистре (но есть `Type` как ключевое слово метакласса)**.

> **Основной дизайн**: `Type` сам по себе является обобщённым типом. `(T: Type) -> Type` означает «тип, принимающий параметр-тип T».

| Концепция       | Код                                      |
|-----------------|------------------------------------------|
| Переменная       | `x: Int = 42`                            |
| Функция       | `add: (a: Int, b: Int) -> Int = a + b`   |
| Записьный тип   | `Point: Type = { x: Float, y: Float }`   |
| Интерфейс       | `Drawable: Type = { draw: (Surface) -> Void }` |
| Обобщённый тип   | `List: (T: Type) -> Type = { data: Array(T), length: Int }` |
| Обобщённый тип   | `Map: (K: Type, V: Type) -> Type = { keys: Array(K), values: Array(V) }` |
| Метод       | `Point.draw: (p: Point, s: Surface) -> Void = ...`<br>`Point.draw = draw[0]` |
| Обобщённая функция   | `map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R))` |

**`Type` — единственное ключевое слово метакласса в языке**.

> **Пространство имён vs привязка методов**: префикс `Type.name` обозначает **принадлежность к пространству имён**, и только это.
> Он не вызывает никаких неявных привязок. Чтобы синтаксис вызова `.` вроде `p.draw(screen)` заработал,
> необходима явная привязка: `Point.draw = draw[0]`.
> Подробности см. в разделе «Пространства имён и привязка методов» ниже.
Оно используется для разметки уровня типовой иерархии, компилятор автоматически обрабатывает различение Type0, Type1, Type2... , прозрачно для пользователя.

```yaoxiang
// Основной синтаксис: унификация + различение

// Переменная
x: Int = 42

// Функция (имена параметров в сигнатуре)
add: (a: Int, b: Int) -> Int = a + b

// Записной тип
Point: Type = {
    x: Float,
    y: Float,
    draw: (Surface) -> Void,
    serialize: () -> String
}

// Интерфейс (по сути записной тип, все поля которого являются функциями)
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Определение метода (синтаксис Type.method)
Point.draw: (self: Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}

Point.serialize: (self: Point) -> String = {
    return "Point(${self.x}, ${self.y})"
}

// Обобщённый тип ((T: Type) -> Type = обобщённый тип, принимающий параметры-типы)
List: (T: Type) -> Type = {
    data: Array(T),
    length: Int
}

Map: (K: Type, V: Type) -> Type = {
    keys: Array(K),
    values: Array(V)
}

// Использование
p: Point = Point(1.0, 2.0)
p.draw(screen)           // Синтаксический сахар → Point.draw(p, screen)
s: Drawable = p           // Структурный подтип: Point реализует Drawable
drawables: List(Drawable) = [p, r]
process_all(drawables)
```

## Мотивация

### Зачем нужен этот функционал?

Текущая система типов содержит несколько разрозненных концепций:
- Синтаксис объявления переменных
- Синтаксис определения функций
- Синтаксис определения типов (различные формы)
- Синтаксис определения интерфейсов
- Синтаксис привязки методов

Отсутствие унификации между этими концепциями приводит к фрагментации синтаксиса и высокой стоимости обучения.

### Цели дизайна

1. **Предельная унификация**: одно синтаксическое правило покрывает все случаи
2. **Простота и элегантность**: симметричная эстетика `name: type = value`
3. **Без новых ключевых слов**: переиспользование существующих синтаксических элементов
4. **Теоретическая элегантность**: типы сами по себе являются значениями типа Type
5. **Дружелюбие к обобщениям**: бесшовная интеграция с системой обобщений (RFC-011)

### Интеграция с системой обобщений

Унифицированная модель синтаксиса RFC-010 **естественно согласуется** с дизайном системы обобщений RFC-011, параметры обобщений органично вписываются в унифицированную модель:

```yaoxiang
// Базовые обобщения (RFC-011 Phase 1)
List: (T: Type) -> Type = { data: Array(T), length: Int }

// Обобщённые функции (RFC-023 синтаксис: параметр Type в сигнатуре можно опустить, выводится при вызове)
map: (: Type, R: Type) -> (( list: List(T), f: (T) -> R) -> List(R)) = ...

// Ограничения типов (RFC-011 Phase 2)
clone: (value: T) -> T = value.clone()  // Ограничение T: Clone переносится через тип параметра

// Const-обобщения (RFC-011 Phase 4)
Array: (T: Type, N: Int) -> Type = { data: Array(T, N), length: N }
```

**Зависимости**:
- RFC-011 Phase 1 (базовые обобщения) является **сильной зависимостью** RFC-010
- Без базовых обобщений обобщённые примеры RFC-010 не скомпилируются
- Рекомендация: RFC-011 Phase 1 и RFC-010 реализовать синхронно

## Предложение

### Основной принцип: конструкторы типов vs функции/переменные

**Это ключевой выбор дизайна, определяющий правила устранения синтаксической неоднозначности:**

| Запись | Значение | Правило |
|--------|----------|---------|
| **`x: Type = ...`** | Конструктор типа | `: Type` явно объявляет → принудительно тип |
| **`f = ...`** | Функция или переменная | Нет `: Type` → HM активно выводит как функцию/переменную |

**Почему такой дизайн?**

Синтаксис `{ ... }` сам по себе неоднозначен:
- `{ x: Float, y: Float }` может быть **тип-литералом** (записной тип)
- `{ a = 1 + 1 }` может быть **блоком кода** (выполняемые операторы, возвращает Void)

**Правила устранения неоднозначности**:
- **Есть** `: Type` → принудительное разбор как конструктор типа, `{ ... }` — тип-литерал
- **Нет** `: Type` → HM активно парсит `{ ... }` как блок кода, выводит как функциональный тип

```yaoxiang
# ✅ Конструктор типа: есть : Type
Point: Type = { x: Float, y: Float }

# ✅ Функция: нет : Type, HM выводит как () -> Void
main = { println("Hello") }

# ❌ Ошибка: нет : Type, компилятор не может распознать { ... } как тип
Point = { x: Float, y: Float }  // HM выводит как функцию, не как тип!
```

---

**Унифицированная модель: identifier : type = expression**

```
├── Переменная
│   └── x: Int = 42
│
├── Функция
│   └── add: (a: Int, b: Int) -> Int = a + b  # Нет : Type, HM выводит как функцию
│
├── Записной тип
│   └── Point: Type = { x: Float, y: Float }  # Обязательный возврат: Type
│
├── Интерфейс
│   └── Drawable: Type = { draw: (Surface) -> Void }  # Обязательный возврат: Type
│
├── Обобщённый тип
│   └── List: (T: Type) -> Type = { data: Array(T), length: Int }  # Обязательный возврат: Type
│
├── Обобщённый тип (несколько параметров)
│   └── Map: (K: Type, V: Type) -> Type = { keys: Array(K), values: Array(V) }  # Обязательный возврат: Type
│
├── Функция пространства имён
│   └── draw: (p: Point, surface: Surface) -> Void = ...
│       Point.draw = draw[0]  # Явная привязка — только после неё появляется синтаксис вызова .
│
└── Обобщённая функция
    └── map: (T: Type, R: Type) -> ((list: List(T), f: (x: T) -> R) -> List(R))  # Не возвращает Type, HM выводит как функцию
```

### Уровни метакласса (внутреннее устройство компилятора)

**Внутренне** компилятор поддерживает вселенную-уровень `level: selfpointnum` (хранится как строка, теоретически бесконечно расширяемая).

| Уровень | Описание |
|---------|----------|
| `Type0` | Повседневные типы (`Int`, `Float`, `Point`) |
| `Type1` | Конструкторы типов (`List`, `Maybe`) |
| `Type2+` | Конструкторы высших порядков |

**Пользователь никогда не видит эти числа**, только `: Type`.

> **Изоморфизм Карри — Ховарда**: существование вселенных-уровней — не деталь реализации, а необходимое условие логической согласованности. Изоморфизм Карри — Ховарда отождествляет типы с высказываниями; если допустить `Type: Type` (т.е. «тип типа тоже является типом»), возникнет парадокс, подобный «это высказывание ложно» — парадокс Рассела в логике, проявляющийся как парадокс Жирара в теории типов. Иерархия `Type0 / Type1 / Type2…` YaoXiang (накопительные вселенные теории типов Мартина-Лёфа) гарантирует, что каждый тип принадлежит ровно одному уровню, а `Typeₙ : Typeₙ₊₁` образует никогда не замыкающуюся восходящую цепочку, что фундаментально предотвращает парадоксы. Это означает, что система типов YaoXiang в смысле Карри — Ховарда является **логически согласованной**.

### Синтаксические определения

#### 1. Объявление переменных

```yaoxiang
// Базовая форма
x: Int = 42
name: String = "Alice"
flag: Bool = true

// Вывод типа (можно опустить)
y = 100  // Выводится как Int
```

#### 2. Определение функций

```yaoxiang
// Форма с одним выражением (непосредственный возврат, без return)
add: (a: Int, b: Int) -> Int = a + b
greet: (name: String) -> String = "Hello, ${name}!"

// Форма с блоком кода (обязателен return для возврата значения)
process: (x: Int) -> Int = {
    a = x * 2
    b = a + 1
    return b
}

// Многострочный блок кода
calc: (x: Float, y: Float, op: String) -> Float = {
    return match op {
        "+" -> x + y,
        "-" -> x - y,
        _ -> 0.0
    }
}

// Void-функция (return не нужен в блоке кода)
print: (msg: String) -> Void = {
    console.write(msg)
}
```

#### Правила возврата

Возвращаемое значение зависит от формы правой части `=`:

| Запись | Возвращаемое значение |
|--------|-----------------------|
| `= expr` (без фигурных скобок) | Непосредственный возврат `expr` |
| `= { ... }` (с фигурными скобками) | Обязателен `return`, иначе возврат `Void` |

```yaoxiang
# Одно выражение: непосредственный возврат, return не нужен
add: (a: Int, b: Int) -> Int = a + b

# Блок кода: return обязателен
process: (x: Int) -> Int = {
    a = x * 2
    b = a + 1
    return b
}

# Void-функция: return не нужен
print: (msg: String) -> Void = {
    console.write(msg)
}
```

> **Обоснование дизайна**: `{ ... }` — это вычислительная единица с зависимостями (см. ниже), её семантика возврата отличается от одно-выражений. Фигурные скобки вводят контекст с несколькими операторами, поэтому нужен явный `return` для устранения неоднозначности «является ли последнее выражение возвращаемым значением».

#### Семантика `{}`: Вычислительная единица с зависимостями

`{ ... }` в YaoXiang — это не просто блок кода — это **вычислительная единица с зависимостями**. Эта семантика едина для тела функции, инициализации переменных и `spawn`:

**Основные правила**:
- Операторы присваивания внутри `{}` автоматически упорядочиваются по зависимостям, а не по порядку записи
- Операции с готовыми зависимостями выполняются немедленно, при отсутствии — блокируются до готовности
- `return` используется для явного возврата значения (см. правила возврата)

```yaoxiang
# Зависимости: b зависит от a, компилятор автоматически упорядочивает
result: Int = {
    b = a + 1      # Зависит от a → автоматически после a
    a = 10         # Без зависимостей → может выполниться первым
    return b       # Возвращает 11
}
```

> **Отличие от одно-выражений**: `= expr` (без скобок) — простое связывание с непосредственным возвратом значения; `= { ... }` (со скобками) — контекст вычислений с зависимостями, допускающий несколько операторов и явный `return`.

#### Блок `spawn`

`spawn { ... }` — единственный примитив параллелизма YaoXiang. Он использует семантику `{}` с зависимостями для автоматического распараллеливания:

- Прямые дочерние присваивания внутри `spawn { ... }` автоматически создают параллельные задачи
- Операции с готовыми зависимостями выполняются параллельно немедленно
- Вызывающая сторона блокируется до завершения всех дочерних задач

```yaoxiang
result = spawn {
    a = fetch_data("url1")    # Задача 1
    b = fetch_data("url2")    # Задача 2 (не зависит от a, выполняется параллельно)
    c = process(a, b)         # Зависит от a, b → ждёт обоих
    return c
}
// Вызывающая сторона блокируется здесь, пока все задачи внутри spawn не завершатся
```

> **Подробное определение**: Полная семантика `spawn`, правила создания задач и модель блокировки описаны в `008-runtime-concurrency-model.md`.

#### Блок `unsafe`

`unsafe { ... }` используется для определения непрозрачных типов и операций с сырыми указателями. Он использует семантику `return` фигурных скобок для возврата определения типа во внешнюю область видимости:

**Основные правила**:
- Внутри `unsafe {}` можно определять типы и работать с сырыми указателями
- `return` возвращает определение типа во внешнюю область видимости
- Возвращённый тип доступен за пределами `unsafe {}`
- Доступ к полям типа требует unsafe-разрешения

```yaoxiang
# Определение непрозрачного типа в unsafe-блоке
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void  # Сырой указатель
    }
    return SqliteDb
}

# SqliteDb доступен за пределами unsafe-блока
db = sqlite3_open("test.db")

# ❌ Ошибка компиляции: поле handle требует unsafe-разрешения
handle = db.handle

# ✅ Через вызов метода
db.close()
```

> **Подробное определение**: Полная семантика `unsafe`, определения FFI-типов и привязка методов описаны в `ffi.md`.

#### 3. Определение типов

Определение типа — ключевая часть унифицированного синтаксиса YaoXiang, включает поля, значения по умолчанию, привязанные методы, реализации интерфейсов:

##### Базовые типы

**Записной тип**: список полей, тип поля может быть любым тип-выражением.

```yaoxiang
Point: Type = {
    x: Float,
    y: Float
}
```

**Поля со значениями по умолчанию**: поля могут иметь значения по умолчанию, необязательны при конструировании.

```yaoxiang
Point: Type = {
    x: Float = 0,
    y: Float = 0
}
```

Использование:

```yaoxiang
Point() → Point(x=0, y=0)
Point(x=1) → Point(x=1, y=0)
Point(x=1, y=2) → Point(x=1, y=2)
```

**Поля без значений по умолчанию**: обязательны при конструировании.

```yaoxiang
Point2: Type = {
    x: Float,
    y: Float
}
```

Использование:
```yaoxiang
Point2(x=1, y=2) //✓
Point2() //✗
Point2(x=1) //✗
```

##### Привязка методов

**Способ 1: Непосредственная привязка внешней функции внутри определения типа**

```yaoxiang
distance: (a: Point, b: Point) -> Float = { ... }
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance = distance[0]           // Привязка к позиции 0, после каррирования method: (b: Point) -> Float
}
// Вызов: p1.distance(p2) → distance(p1, p2)
```

**Способ 2: Анонимная функция + позиционная привязка**

```yaoxiang
Point: Type = {
    x: Float = 0,
    y: Float = 0,
    distance: ((a: Point, b: Point) -> Float)[0] = ((a, b) => {
        dx = a.x - b.x
        dy = a.y - b.y
        return (dx * dx + dy * dy).sqrt()
    })
}
// Синтаксис: ((params) => body)[position]
// Вызов: p1.distance(p2) → distance(p1, p2)
```

##### Реализация интерфейсов

**Имя интерфейса записывается в теле типа, компилятор автоматически проверяет реализацию**

```yaoxiang
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

Point: Type = {
    x: Float,
    y: Float,
    Drawable,          // Реализует интерфейс Drawable
    Serializable      // Реализует интерфейс Serializable
}
```

##### Определение интерфейсов

**Интерфейс = записной тип, все поля которого являются функциями**

```yaoxiang
Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

// Пустой тип / пустой интерфейс
EmptyType: Type = {}
Empty: Type = {}
```

##### Определение функций пространства имён

**Префикс `Type.name` обозначает принадлежность к пространству имён**, и только это. Он не вызывает никаких неявных привязок.

```yaoxiang
// Функция пространства имён: обычная функция в пространстве Point
Point.draw: (p: &Point, surface: Surface) -> Void = {
    surface.plot(p.x, p.y)
}

Point.serialize: (p: &Point) -> String = {
    return "Point(${p.x}, ${p.y})"
}

// Вызов: это обычный вызов функции
Point.draw(p, screen)
Point.serialize(p)
```

> **Примечание**: `self` не является ключевым словом, это просто договорённость об именовании параметра. Запись `p`, `this`, `x` даёт полностью идентичный эффект.
> Компилятор смотрит на тип, а не на имя параметра.

##### Привязка методов (единственный способ)

Чтобы синтаксис вызова метода `.` вроде `p.draw(screen)` заработал, **обязательна явная привязка**.
Синтаксис `[position]` — единственный механизм привязки функции в качестве «метода» (подробный синтаксис см. RFC-004).

```yaoxiang
// Определение функции
draw: (p: &Point, surface: Surface) -> Void = {
    surface.plot(p.x, p.y)
}

// Явная привязка — только после неё появляется синтаксис p.draw(screen)
Point.draw = draw[0]   // Параметр позиции 0 (&Point) заполняется вызывающей стороной

// Использование
p.draw(screen)          // Синтаксический сахар → draw(&p, screen)
Point.draw(p, screen)   // Оба вызова эквивалентны

// Без [0] = нет привязки. Point.draw — просто псевдоним функции, без синтаксиса .
Point.draw = draw       // Без привязки: только Point.draw(p, screen)
```

**Поведение по умолчанию**: без `[n]` = нет привязки. Пользователь должен явно решить, какие параметры заполняются вызывающей стороной.

**Привязка нескольких позиций**:

```yaoxiang
// Привязка нескольких позиций (автоматическое каррирование)
Point.transform = transform_points[0, 1]
// Вызов: p1.transform(p2)(2.0) → transform_points(p1, p2, 2.0)
```

**Обратная операция** (метод → обычная функция):

```yaoxiang
// Извлечение функции из привязки
draw_point: (p: &Point, surface: Surface) -> Void = Point.draw
```

#### 5. Обобщённые типы

```yaoxiang
// Базовые обобщения (RFC-011 Phase 1)
List: (T: Type) -> Type = {
    data: Array(T),
    length: Int,
    push: (T:Type)-((self: List(T), item: T) -> Void),
    get: (T:Type)->((self: List(T), index: Int) -> Maybe(T))
}

// Конкретная конкретизация (RFC-023 синтаксис)
IntList: Type = List(Int)

IntList.push = {
    self.data.append(item)
    self.length = self.length + 1
}

List.push = (type: Type) -> {
    return (self: List(type), item: type) -> {
        self.data.append(item)
        self.length = self.length + 1
    }
}

IntList.push(Int)(self, item)  // Пример вызова

// Обобщённые методы (RFC-023 синтаксис: параметр типа автоматически выводится из контекста вызова)
List.push: (self: List(T), item: T) -> Void = {
    self.data.append(item)
    self.length = self.length + 1
}

List.get: (self: List(T), index: Int) -> Maybe(T) = {
    if index >= 0 && index < self.length {
        return Maybe.Just(self.data[index])
    } else {
        return Maybe.Nothing
    }
}
```

#### 6. Синтаксис вызова обобщений

Вызов обобщённых типов и обобщённых функций унифицирован и использует синтаксис `()`. `[]` не используется ни в каком контексте обобщений.

**Основные правила**:

1. **`()` выполняет всё**: применение типа, вызов функции, конструирование значения — всё через `()`

```yaoxiang
# Аннотация типа
numbers: List(Int) = List(1, 2, 3)

# Пустой контейнер: T берётся слева
empty: List(Int) = List()

# Вызов обобщённой функции — тип автоматически «перетекает» из параметров
strings = map(numbers, f)
// T=Int приходит из numbers: List(Int)
// R=String приходит из f: (Int) -> String
```

2. **Type слева, значение справа**: `name: type = value` — параметры Type объявляются слева, справа всегда конкретное значение. Для пустого контейнера `List()` параметр `T` должен браться из аннотации типа слева.

3. **Информация о типе пишется один раз** — при объявлении параметра, компилятор несёт её вместе:

```yaoxiang
numbers: List(Int) = List(1, 2, 3)  // Int пишется один раз слева
f: (Int) -> String = (x) => x.to_string()
strings = map(numbers, f)   // T=Int, R=String автоматически из типов numbers и f
```

4. **Конструирование значения выводит тип из элементов**:

```yaoxiang
x = List(1, 2, 3)       // Выводится как List(Int)
y = List("a", "b")      // Выводится как List(String)
z = List()              // ❌ Ошибка компиляции: невозможно вывести T
z: List(Int) = List()   // ✅ T=Int из аннотации слева
```

5. **Псевдонимы типов**:

```yaoxiang
IntList: Type = List(Int)
StringToInt: Type = (String) -> Int
Matrix3x3: Type = Matrix(Float, 3, 3)
```

> **Сравнение со старым синтаксисом**: `List[Int]` → `List(Int)`, `List[Int]()` → `List()`, `List[Int](1,2,3)` → `List(1,2,3)`.
> Старый `[]` синтаксис обобщений полностью удалён. `[]` используется только для литералов массивов/списков и индексного доступа.

#### Полный пример

```yaoxiang
// ======== 1. Определение интерфейсов ========
// Интерфейс = записной тип, все поля которого являются функциональными типами
// В интерфейсе параметр self не нужен — интерфейс определяет только «сигнатуру функции без позиции вызывающего»

Drawable: Type = {
    draw: (surface: Surface) -> Void,
    bounding_box: () -> Rect
}

Serializable: Type = {
    serialize: () -> String
}

Transformable: Type = {
    translate: (dx: Float, dy: Float) -> Transformable,  // Возвращает тип интерфейса, конкретная реализация возвращает свой тип
    scale: (factor: Float) -> Transformable
}

// ======== 2. Определение типов ========

Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable,
    Transformable
}

Rect: Type = {
    x: Float,
    y: Float,
    width: Float,
    height: Float,
    Drawable,
    Serializable,
    Transformable
}

// ======== 3. Реализация методов (обычные функции + явная привязка)========

// Определение функции (self — просто договорённость, не ключевое слово)
draw: (p: &Point, surface: Surface) -> Void = {
    surface.plot(p.x, p.y)
}

bounding_box: (p: &Point) -> Rect = {
    return Rect(p.x - 1, p.y - 1, 2, 2)
}

serialize: (p: &Point) -> String = {
    return "Point(${p.x}, ${p.y})"
}

translate: (p: &Point, dx: Float, dy: Float) -> Point = {
    return Point(p.x + dx, p.y + dy)
}

scale: (p: &Point, factor: Float) -> Point = {
    return Point(p.x * factor, p.y * factor)
}

distance: (p1: &Point, p2: &Point) -> Float = {
    dx = p1.x - p2.x
    dy = p1.y - p2.y
    return (dx * dx + dy * dy).sqrt()
}

// Явная привязка — только после неё появляется синтаксис вызова метода
Point.draw = draw[0]
Point.bounding_box = bounding_box[0]
Point.serialize = serialize[0]
Point.translate = translate[0]
Point.scale = scale[0]
Point.distance = distance[0]

// Методы Rect аналогично
draw: (r: &Rect, surface: Surface) -> Void = {
    surface.draw_rect(r.x, r.y, r.width, r.height)
}
Rect.draw = draw[0]

bounding_box: (r: &Rect) -> Rect = r
Rect.bounding_box = bounding_box[0]

serialize: (r: &Rect) -> String = {
    return "Rect(${r.x}, ${r.y}, ${r.width}, ${r.height})"
}
Rect.serialize = serialize[0]

translate: (r: &Rect, dx: Float, dy: Float) -> Rect = {
    return Rect(r.x + dx, r.y + dy, r.width, r.height)
}
Rect.translate = translate[0]

scale: (r: &Rect, factor: Float) -> Rect = {
    return Rect(r.x * factor, r.y * factor, r.width * factor, r.height * factor)
}
Rect.scale = scale[0]

// ======== 4. Использование ========

// Создание экземпляров
p: Point = Point(1.0, 2.0)
r: Rect = Rect(0.0, 0.0, 10.0, 20.0)

// Вызов методов (синтаксический сахар)
p.draw(screen)
r.draw(screen)

// Обычный вызов метода (прямой вызов)
d: Float = distance(p, Point(0.0, 0.0))

// Цепочка вызовов
p2: Point = p.translate(1.0, 1.0).scale(2.0)

// Присваивание интерфейсу
drawables: List(Drawable) = [p, r]
for d in drawables {
    d.draw(screen)
}

// Обобщённые функции (RFC-023 синтаксис: параметры типа опускаются при вызове, автоматический вывод)
process_all: (items: List(T)) -> Void = {
    for item in items {
        print(item.serialize())
    }
}

process_all([p, r])
```

## Детальный дизайн

### Алгоритм проверки интерфейсов

```rust
fn check_type_implements_interface(
    typ: &Type,
    iface: &Type
) -> Result<(), TypeError> {
    // Для каждого поля интерфейса (функционального поля)
    for (field_name, iface_field) in &iface.fields {
        // Проверка: есть ли у типа метод с таким именем
        if let Some(method) = typ.methods.get(field_name) {
            // Проверка совместимости сигнатуры метода
            // Поле интерфейса: (Surface) -> Void
            // Сигнатура метода: (Point, Surface) -> Void
            // Сравнение: после удаления параметра self должно совпадать
            if !method_signature_matches(method, iface_field.type_) {
                return Err(TypeError::MethodSignatureMismatch {
                    type_name: typ.name,
                    interface_name: iface.name,
                    method_name: field_name,
                });
            }
        } else {
            return Err(TypeError::MissingMethod {
                type_name: typ.name,
                interface_name: iface.name,
                method_name: field_name,
            });
        }
    }
    Ok(())
}
```

### Прямое присваивание интерфейсу и оптимизация на этапе компиляции

Интерфейсный тип поддерживает прямое присваивание; компилятор автоматически выбирает оптимальную стратегию вызова по типу правого значения:

```yaoxiang
// Прямое присваивание конкретного типа → конкретный тип известен на этапе компиляции, вызов без накладных расходов
d: Drawable = Circle(1)
d.draw(screen)  // После компиляции: прямой вызов circle_draw(screen), без vtable

// Возврат из функции → конкретный тип неизвестен на этапе компиляции, используется vtable
d: Drawable = get_shape()
d.draw(screen)  // Поиск метода через vtable

// Гетерогенная коллекция → используется vtable
shapes: List(Drawable) = [Circle(1), Rect(2, 3)]
for s in shapes {
    s.draw(screen)  // Поиск метода через vtable
}
```

**Стратегии оптимизации на этапе компиляции**:

| Ситуация | Вывод | Способ вызова |
|----------|-------|---------------|
| `d: Drawable = Circle(1)` | Конкретный тип Circle | Прямой вызов (нулевые накладные) |
| `d: Drawable = get_shape()` | Неизвестен | vtable |
| `shapes: List(Drawable) = [...]` | Гетерогенный | vtable |

**Правила**:
1. Когда правое значение является конкретным конструктором типа и тип известен на этапе компиляции, генерируется прямой вызов IR
2. Когда тип правого значения невозможно определить на этапе компиляции, используется механизм vtable
3. vtable гарантирует корректность полиморфизма во время выполнения

| Раньше | Теперь |
|--------|--------|
| `type Point = Point(x: Float, y: Float)` | `type Point = { x: Float, y: Float }` |
| `type Result(T, E) = ok(T) \| err(E)` | `Result: (T: Type, E: Type) -> Type = { ok: (T) -> Result(T, E), err: (E) -> Result(T, E) }` |
| Требовалось ключевое слово `impl` | Ключевое слово не нужно, имя интерфейса пишется после тела типа |

## Замечания о дизайне синтаксиса: именованные функции по сути являются синтаксическим сахаром для Lambda

### Ключевое понимание

**Именованные функции и Lambda-выражения — это одно и то же!** Единственное различие: именованная функция даёт Lambda имя.

```yaoxiang
// Оба по сути полностью идентичны
add: (a: Int, b: Int) -> Int = a + b           // Именованная функция (рекомендуется)
add: (a: Int, b: Int) -> Int = (a, b) => a + b        // Lambda-форма (полностью эквивалентно)
```

### Модель синтаксического сахара

```
// Именованная функция = Lambda + имя
name: (Params) -> ReturnType = body

// По сути эквивалентно
name: (Params) -> ReturnType = (params) => body
```

**Ключевой момент**: когда сигнатура полностью объявляет типы параметров, имена параметров в голове Lambda становятся избыточными и могут быть опущены.

### Правила области видимости параметров

**Параметры перекрывают внешние переменные**: область видимости параметров в сигнатуре перекрывает тело функции, приоритет у внутренней области выше.

```yaoxiang
x = 10  // Внешняя переменная

double: (x: Int) -> Int = x * 2  // ✅ Параметр x перекрывает внешний x, результат 20
```

### Гибкость позиционирования аннотации типа

Аннотация типа может располагаться в любом из следующих мест, **достаточно аннотировать хотя бы в одном**:

| Позиция аннотации | Форма | Описание |
|-------------------|-------|----------|
| Только в сигнатуре | `double: (x: Int) -> Int = x * 2` | ✅ Рекомендуется |
| Только в голове Lambda | `double = (x: Int) => x * 2` | ✅ Допустимо |
| Обе стороны | `double: (x: Int) -> Int = (x) => x * 2` | ✅ Избыточно, но допустимо |

### Полный пример

```yaoxiang
// ✅ Рекомендуется: сигнатура полная, голова Lambda опущена
add: (a: Int, b: Int) -> Int = a + b
inc: (x: Int) -> Int = x + 1
main: () -> Void = { print("hi") }

// ✅ Допустимо: тип в голове Lambda
double = (x: Int) => x * 2

// ✅ Допустимо: обе стороны аннотированы
double: (x: Int) -> Int = (x) => x * 2
```

### Преимущества дизайна

| Характеристика | Преимущество |
|----------------|--------------|
| **Краткость** | При полной сигнатуре не нужно повторять имена параметров |
| **Гибкость** | Lambda-форма сохранена, используй какую удобно |
| **Единство** | Единообразие с объявлением переменных `x: Int = 42` |
| **Наглядность** | `name: Type = body` непосредственно соответствует «называется name, тип Type, значение body» |

## Приложения

### BNF грамматика

```bnf
program ::= statement*

statement ::= declaration | expression

# Унифицированное объявление: name: Type = expression
declaration ::= identifier ':' type_expr '=' expression

# Типовое выражение
type_expr ::= identifier
       | identifier '(' type_expr (',' type_expr)* ')'      # Применение типа
       | '(' type_expr (',' type_expr)* ')' '->' type_expr       # Функциональный тип
       | '{' type_field* '}'                       # Записной / интерфейсный тип
       | 'Type'                                    # Метатип

type_field ::= identifier ':' type_expr
             | identifier                           # Ограничение-интерфейс

# Обобщённые параметры: как часть функционального типа, например (T: Type, R: Type) -> (...)
# Отдельное BNF-правило не требуется — параметр : Type это обычный параметр функции

# Выражение
expression ::= literal
              | identifier
              | identifier '(' expression (',' expression)* ')'  # Вызов функции / вызов конструктора
              | '(' expression (',' expression)* ')'              # Кортеж
              | expression '.' identifier '(' arguments? ')'    # Вызов метода
              | lambda
              | '{' field ':' expression (',' field ':' expression)* '}'

arguments ::= expression (',' expression)*

lambda ::= '(' parameter_list? ')' '=>' block

block ::= expression | '{' expression* '}'
```

### Глоссарий

| Термин | Определение |
|--------|-------------|
| Объявление | Оператор присваивания вида `name: type = value` |
| Записной тип | Тип `{ ... }` с именованными полями |
| Интерфейс | Записной тип, все поля которого являются функциональными типами |
| Обобщённый тип | Тип, определённый как `Name: (T: Type) -> Type = { ... }`, принимающий параметры-типы |
| Функция пространства имён | Функция формы `Type.name`, принадлежащая пространству имён Type. Не влечёт никакой привязки |
| Привязка метода | `Type.name = func[n]`, привязка позиции n функции func как вызывающей стороны, делающая синтаксис `obj.name(args)` доступным |
| Обобщённая функция | Функция, использующая синтаксис `(T: Type)`, параметры-типы идут как первая группа параметров |
| Метатип | `Type`, единственный маркер уровня типовой иерархии в языке |
