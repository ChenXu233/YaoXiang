# RFC-015: Проектирование системы конфигурации YaoXiang

> **Дата принятия**: 2026-02-15

> **Предшествующий RFC**: [RFC-014: Проектирование системы управления пакетами](014-package-manager.md)

## Резюме

Проектирование унифицированной системы конфигурации для языка YaoXiang, поддерживающей пользовательский и проектный уровни, обеспечивающей общую инфраструктуру конфигурации для менеджера пакетов, компилятора, REPL, LSP и других компонентов.

## Мотивация

### Зачем нужен эта функциональность/изменение?

Инструментарий YaoXiang содержит несколько компонентов:

- Менеджер пакетов (читает конфигурацию зависимостей)
- Фронтенд компилятора (читает конфигурацию i18n)
- REPL (читает интерактивную конфигурацию)
- LSP (читает конфигурацию fmt/lint/test)
- Система сборки (читает конфигурацию сборки)

Каждому компоненту нужна унифицированная инфраструктура конфигурации.

### Текущие проблемы

- Конфигурации компонентов разрозненны, нет единого стандарта
- Пользователи не могут унифицированно управлять настройками предпочтений
- Нет чёткого разделения уровней между проектной и пользовательской конфигурацией

## Предложение

### Основной дизайн

**Многоуровневая архитектура**:
```
Приоритет конфигурации (высокий → низкий):
┌─────────────────────────────────────────────┐
│ 1. Проектный yaoxiang.toml                   │ ← Контроль проектной команды
├─────────────────────────────────────────────┤
│ 2. Пользовательский ~/.config/yaoxiang/config.toml │ ← Пользовательские предпочтения
├─────────────────────────────────────────────┤
│ 3. Значения компилятора по умолчанию        │ ← Разумные начальные значения
└─────────────────────────────────────────────┘
```

**Правило**: верхний уровень переопределяет нижний, ненастроенные опции возвращаются к нижнему уровню.

### Ограничения уровней конфигурации

| Секция конфигурации | Пользовательский | Проектный | Потребитель |
|---------------------|------------------|-----------|-------------|
| `[package].*` | ❌ | ✅ | Менеджер пакетов |
| `[yaoxiang]` | ❌ | ✅ | Компилятор |
| `[dependencies]` | ❌ | ✅ | Менеджер пакетов |
| `[dev-dependencies]` | ❌ | ✅ | Менеджер пакетов |
| `[bin]` | ❌ | ✅ | Менеджер пакетов |
| `[lib]` | ❌ | ✅ | Менеджер пакетов |
| `[build]` | ✅ | ✅ | Система сборки |
| `[profile.*]` | ✅ | ✅ | Система сборки |
| `[install]` | ✅ | ❌ | Менеджер пакетов |
| `[i18n]` | ✅ | ✅ | Компилятор |
| `[repl]` | ✅ | ✅ | REPL |
| `[fmt]` | ✅ | ✅ | LSP |
| `[lint]` | ✅ | ✅ | LSP |
| `[test]` | ✅ | ✅ | LSP |
| `[tasks]` | ✅ | ✅ | CLI |

### Примеры

**Проектная конфигурация**:
```toml
# yaoxiang.toml
[package]
name = "my-package"
version = "0.1.0"

[yaoxiang]
version = ">=0.1.0, <1.0.0"

[dependencies]
foo = "^1.0.0"

[build]
output = "dist/"

[tasks]
build = "yaoxiang build"
test = "yaoxiang test"
```

**Пользовательская конфигурация**:
```toml
# ~/.config/yaoxiang/config.toml
[install]
dir = "~/.local/share/yaoxiang"

[i18n]
lang = "zh"
fallback = "en"

[repl]
history-size = 1000
prompt = "yx> "
colors = true

[fmt]
line-width = 120
indent-width = 4

[lint]
rules = ["recommended"]
```

## Детальный проект

### Конфигурация исключительно для проекта

```toml
[package]
name = "my-package"
version = "0.1.0"
description = "A short description"
authors = ["Alice <alice@example.com>"]
license = "MIT"
repository = "https://github.com/alice/my-project"

[yaoxiang]
version = ">=0.1.0, <1.0.0"

[dependencies]
foo = "^1.0.0"

[dev-dependencies]
test-utils = "0.1.0"

[lib]
path = "src/lib.yx"

[[bin]]
name = "my-cli"
path = "src/cli.yx"

[exports]
"." = "src/lib.yx"
"./foo" = "src/foo.yx"

[build]
script = "build.yx"
output = "dist/"

[profile.release]
optimize = true
lto = true

[run]
main = "src/main.yx"
args = ["--quiet"]

[tasks]
build = "yaoxiang build"
test = "yaoxiang test"
lint = "yaoxiang fmt && yaoxiang check"
```

### Конфигурация исключительно для пользователя

```toml
[install]
dir = "~/.local/share/yaoxiang"
```

### Конфигурация для обоих уровней

| Поле | Тип | Значение по умолчанию | Описание |
|------|------|----------------------|----------|
| `[i18n].lang` | String | "en" | Язык |
| `[i18n].fallback` | String | "en" | Резервный язык |
| `[repl].history-size` | Number | 1000 | Размер истории |
| `[repl].history-file` | Path | ~ | Файл истории |
| `[repl].prompt` | String | "yx> " | Приглашение |
| `[repl].colors` | Boolean | true | Подсветка синтаксиса |
| `[repl].auto-imports` | [String] | [] | Автоматический импорт |
| `[fmt].line-width` | Number | 120 | Ширина строки |
| `[fmt].indent-width` | Number | 4 | Ширина отступа |
| `[fmt].use-tabs` | Boolean | false | Табуляция для отступов |
| `[fmt].single-quote` | Boolean | false | Одиночные кавычки |
| `[lint].rules` | [String] | ["recommended"] | Набор правил |
| `[lint].strict` | Boolean | false | Строгий режим |
| `[test].report` | String | "console" | Отчёт о тестировании |
| `[build].output` | String | "dist/" | Выходной каталог |

### Переопределение командной строкой и переменными окружения

```bash
# Переопределение командной строки
yaoxiang run main.yx --lang zh
yaoxiang fmt --config-indent-width=2

# Переменные окружения
export YAOXIANG_LANG=zh
export YAOXIANG_FMT_INDENT_WIDTH=2
```

**Приоритет**: `Командная строка > Переменные окружения > Файл конфигурации`

### Команда yaoxiang config

Предоставление CLI-команды для управления конфигурацией:

```bash
# Инициализация пользовательской конфигурации (генерация по умолчанию)
yaoxiang config init

# Редактирование пользовательской конфигурации (открыть редактор)
yaoxiang config edit

# Просмотр текущей конфигурации (эффективная конфигурация после слияния)
yaoxiang config show

# Просмотр источника конфигурации
yaoxiang config show --source

# Сброс к конфигурации по умолчанию
yaoxiang config reset
```

**При первом запуске**: При первом запуске любой команды `yaoxiang` автоматически проверяется наличие пользовательской конфигурации. Если её нет, она автоматически генерируется с параметрами по умолчанию.

**Расположение файлов конфигурации**:

- Проектная: `./yaoxiang.toml` (корневой каталог проекта)
- Пользовательская: `~/.config/yaoxiang/config.toml`

### Семантика слияния конфигураций

Конфигурации разных уровней сливаются по следующим правилам:

| Тип | Стратегия | Описание |
|-----|-----------|----------|
| Скаляр (String/Number/Boolean) | Замена | Проектный уровень переопределяет пользовательский |
| Массив (Array) | Замена | Проектный уровень полностью заменяет пользовательский |
| Объект (Object) | Глубокое слияние | Полевое слияние, неопределённые поля наследуют нижний уровень |

**Пример — глубокое слияние объектов**:
```toml
# Пользовательский уровень
[lint]
rules = ["recommended"]
severity = "warn"

# Проектный уровень
[lint]
strict = true

# Результат слияния
[lint]
rules = ["recommended"]    # Из пользовательского уровня
severity = "warn"          # Из пользовательского уровня
strict = true             # Из проектного уровня
```

### Обратная совместимость

- ✅ Существующий режим без файла конфигурации продолжает поддерживаться (все компоненты используют встроенные значения по умолчанию)
- ✅ Все новые пункты конфигурации имеют разумные значения по умолчанию
- ✅ При первом запуске команды пользователем автоматически генерируется конфигурация с параметрами по умолчанию
- ✅ При сбое разбора конфигурации отображается понятная ошибка с указанием конкретного номера строки и причины ошибки

## Компромиссы

### Преимущества

- Унифицированная инфраструктура конфигурации, сокращение дублирующегося кода
- Пользовательские предпочтения согласованы между проектами
- LSP/REPL/Компилятор используют одну и ту же конфигурацию
- Поэтапная конфигурация, объявление по необходимости

### Недостатки

- Большое количество пунктов конфигурации, несколько увеличенная кривая обучения
- Требуется единый парсер конфигурации

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Независимая конфигурация каждого компонента | Дублирующийся код, фрагментированный пользовательский опыт |
| Только поддержка аргументов командной строки | Невозможность сохранения пользовательских предпочтений |
| Только поддержка переменных окружения | Сложность версионирования проектной конфигурации |

## Стратегия реализации

### Разделение на этапы

| Этап | Содержание |
|------|------------|
| **Phase 1** | Базовый парсер конфигурации, поддержка toml, проектная конфигурация, `yaoxiang config init` |
| **Phase 2** | Пользовательская конфигурация, логика слияния конфигураций, `yaoxiang config edit/show` |
| **Phase 3** | Переопределение командной строки/переменными окружения, ограничения платформы `platform`, расширение `[tool.*]` |

### Зависимости

- Зависит от RFC-014 системы управления пакетами

### Риски

| Риск | Меры смягчения |
|------|----------------|
| Слишком много пунктов конфигурации | Разумные значения по умолчанию, невидимость для пользователя |
| Сложность парсера | Использование существующей библиотеки toml |

### Принятые функции (третий этап)

#### Ограничения платформы `platform`

> **Внимание**: Следующий синтаксис используется в файле конфигурации `yaoxiang.toml`, **не** в исходном коде YaoXiang (файлы `.yx`). Пользователям не нужно писать `cfg(...)` в коде.

Поддержка конфигурации, зависящей от целевой операционной системы/архитектуры:

```toml
# yaoxiang.toml (файл конфигурации)

[target.'cfg(windows)'.build]
output = "dist/win32"

[target.'cfg(unix)'.build]
output = "dist/unix"

[target.'cfg(target_arch = "x86_64")'.build]
rustflags = ["-C target-cpu=native"]
```

**Синтаксис**: `[target.'<условие>'.<секция конфигурации>]`

**Описание**:

- Этот синтаксис появляется только в файле конфигурации `yaoxiang.toml`
- При сборке выбор соответствующей конфигурации осуществляется по параметру `--target`
- Пользователям **не нужно** и **не следует** писать синтаксис `cfg(...)` в исходном коде `.yx`

**Поддерживаемые условия**:

- `cfg(os = "windows")` — Система Windows
- `cfg(os = "linux")` — Система Linux
- `cfg(os = "macos")` — Система macOS
- `cfg(target_arch = "x86_64")` — 64-битная архитектура x86
- `cfg(target_arch = "aarch64")` — 64-битная архитектура ARM

#### Расширение конфигурации сторонних инструментов `[tool.*]`

Разрешение сторонним инструментам хранить конфигурацию под `[tool.<название>]`:

```toml
[tool.eslint]
extension = ["yx", "yxp"]
ignore = ["node_modules/", "dist/"]

[tool.prettier]
semi = false
singleQuote = true
```

**Поведение**:

- YaoXiang игнорирует неизвестные секции `[tool.*]`, но сохраняет их в файле конфигурации
- Сторонние инструменты могут быть интегрированы через `yaoxiang tool run <название>` или прямой доступ
- Конфигурация, специфичная для инструментов, не проверяется
