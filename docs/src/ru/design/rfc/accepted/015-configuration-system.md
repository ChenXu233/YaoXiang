# RFC-015：Дизайн системы конфигурации YaoXiang

> **Статус**: Принято
> **Автор**: Чэнь Сюй (晨煦)
> **Дата создания**: 2026-02-12
> **Дата принятия**: 2026-02-15
> **Последнее обновление**: 2026-02-15

> **Предшествующий RFC**: [RFC-014: Дизайн системы управления пакетами](014-package-manager.md)

## Аннотация

Проектирование унифицированной системы конфигурации для языка YaoXiang с поддержкой двух уровней — пользовательского и проектного, предоставляющей общую инфраструктуру конфигурации для таких компонентов, как менеджер пакетов, компилятор, REPL и LSP.

## Мотивация

### Зачем нужна эта функция/изменение?

Инструментарий YaoXiang включает несколько компонентов:

- Менеджер пакетов (читает конфигурацию зависимостей)
- Интерфейсный компилятор (читает конфигурацию i18n)
- REPL (читает интерактивную конфигурацию)
- LSP (читает конфигурацию fmt/lint/test)
- Система сборки (читает конфигурацию сборки)

Всем компонентам необходима унифицированная инфраструктура конфигурации.

### Текущие проблемы

- Конфигурации компонентов разрозненны, отсутствует единый стандарт
- Пользователи не могут централизованно управлять настройками предпочтений
- Отсутствует чёткое разделение уровней между проектной и пользовательской конфигурацией

## Предложение

### Основной дизайн

**Многоуровневая архитектура**:
```
Приоритет конфигурации (высокий → низкий):
┌─────────────────────────────────────────────┐
│ 1. Проектный уровень yaoxiang.toml          │ ← Контроль команды проекта
├─────────────────────────────────────────────┤
│ 2. Пользовательский уровень ~/.config/yaoxiang/config.toml │ ← Предпочтения пользователя
├─────────────────────────────────────────────┤
│ 3. Значения по умолчанию компилятора        │ ← Разумные начальные значения
└─────────────────────────────────────────────┘
```

**Правило**: Верхний уровень переопределяет нижний; для ненастроенных параметров происходит возврат к нижнему уровню.

### Ограничения уровней конфигурации

| Секция конфигурации | Пользовательский | Проектный | Потребитель |
|----------------------|-------------------|-----------|-------------|
| `[package].*` | ❌ | ✅ | Менеджер пакетов |
| `[yaoxiang]` | ❌ | ✅ | Компилятор |
| `[dependencies]` | ❌ | ✅ | Менеджер пакетов |
| `[dev-dependencies]` | ❌ | ✅ | Менеджер пакетов |
| `[bin]` | ❌ | ✅ | Менеджер пакетов |
| `[lib]` | ❌ | ✅ | Менеджер пакетов |
| `[build]` | ✅ | ✅ | Система сборки |
| `[profile.*]` | ✅ | ✅ | Система сборки |
| `[install]` | ✅ | ❌ | Менеджер пакетов |
| `[i18n]` | ✅ | ✅ | Компилятор |
| `[repl]` | ✅ | ✅ | REPL |
| `[fmt]` | ✅ | ✅ | LSP |
| `[lint]` | ✅ | ✅ | LSP |
| `[test]` | ✅ | ✅ | LSP |
| `[tasks]` | ✅ | ✅ | CLI |

### Примеры

**Проектная конфигурация**:
```toml
# yaoxiang.toml
[package]
name = "my-package"
version = "0.1.0"

[yaoxiang]
version = ">=0.1.0, <1.0.0"

[dependencies]
foo = "^1.0.0"

[build]
output = "dist/"

[tasks]
build = "yaoxiang build"
test = "yaoxiang test"
```

**Пользовательская конфигурация**:
```toml
# ~/.config/yaoxiang/config.toml
[install]
dir = "~/.local/share/yaoxiang"

[i18n]
lang = "zh"
fallback = "en"

[repl]
history-size = 1000
prompt = "yx> "
colors = true

[fmt]
line-width = 120
indent-width = 4

[lint]
rules = ["recommended"]
```

## Детальный дизайн

### Конфигурация уровня проекта

```toml
[package]
name = "my-package"
version = "0.1.0"
description = "Краткое описание"
authors = ["Алиса <alice@example.com>"]
license = "MIT"
repository = "https://github.com/alice/my-project"

[yaoxiang]
version = ">=0.1.0, <1.0.0"

[dependencies]
foo = "^1.0.0"

[dev-dependencies]
test-utils = "0.1.0"

[lib]
path = "src/lib.yx"

[[bin]]
name = "my-cli"
path = "src/cli.yx"

[exports]
"." = "src/lib.yx"
"./foo" = "src/foo.yx"

[build]
script = "build.yx"
output = "dist/"

[profile.release]
optimize = true
lto = true

[run]
main = "src/main.yx"
args = ["--quiet"]

[tasks]
build = "yaoxiang build"
test = "yaoxiang test"
lint = "yaoxiang fmt && yaoxiang check"
```

### Конфигурация уровня пользователя

```toml
[install]
dir = "~/.local/share/yaoxiang"
```

### Конфигурация обоих уровней

| Поле | Тип | Значение по умолчанию | Описание |
|------|------|----------------------|----------|
| `[i18n].lang` | String | "en" | Язык |
| `[i18n].fallback` | String | "en" | Резервный язык |
| `[repl].history-size` | Number | 1000 | Размер истории |
| `[repl].history-file` | Path | ~ | Файл истории |
| `[repl].prompt` | String | "yx> " | Приглашение |
| `[repl].colors` | Boolean | true | Подсветка синтаксиса |
| `[repl].auto-imports` | [String] | [] | Автоимпорты |
| `[fmt].line-width` | Number | 120 | Ширина строки |
| `[fmt].indent-width` | Number | 4 | Отступ |
| `[fmt].use-tabs` | Boolean | false | Табуляция для отступа |
| `[fmt].single-quote` | Boolean | false | Одинарные кавычки |
| `[lint].rules` | [String] | ["recommended"] | Набор правил |
| `[lint].strict` | Boolean | false | Строгий режим |
| `[test].report` | String | "console" | Отчёт о тестировании |
| `[build].output` | String | "dist/" | Директория вывода |

### Переопределение через командную строку и переменные окружения

```bash
# Переопределение через командную строку
yaoxiang run main.yx --lang zh
yaoxiang fmt --config-indent-width=2

# Переменные окружения
export YAOXIANG_LANG=zh
export YAOXIANG_FMT_INDENT_WIDTH=2
```

**Приоритет**: `Командная строка > Переменные окружения > Файл конфигурации`

### Команда yaoxiang config

Предоставление CLI-команд для управления конфигурацией:

```bash
# Инициализация пользовательской конфигурации (генерация по умолчанию)
yaoxiang config init

# Редактирование пользовательской конфигурации (открытие редактора)
yaoxiang config edit

# Просмотр текущей конфигурации (объединённой действующей)
yaoxiang config show

# Просмотр источников конфигурации
yaoxiang config show --source

# Сброс к конфигурации по умолчанию
yaoxiang config reset
```

**При первом запуске**: При первом запуске любой команды `yaoxiang` автоматически проверяется наличие пользовательской конфигурации. Если она отсутствует, конфигурация генерируется автоматически с параметрами по умолчанию.

**Расположение файлов конфигурации**:

- Проектный: `./yaoxiang.toml` (корневой каталог проекта)
- Пользовательский: `~/.config/yaoxiang/config.toml`

### Семантика слияния конфигураций

Конфигурации разных уровней сливаются по следующим правилам:

| Тип | Стратегия | Описание |
|-----|-----------|----------|
| Скаляр (String/Number/Boolean) | Замена | Проектный уровень переопределяет пользовательский |
| Массив (Array) | Замена | Проектный уровень полностью заменяет пользовательский |
| Объект (Object) | Глубокое слияние | Поэлементное слияние, неопределённые поля наследуются с нижнего уровня |

**Пример — глубокое слияние объектов**:
```toml
# Пользовательский уровень
[lint]
rules = ["recommended"]
severity = "warn"

# Проектный уровень
[lint]
strict = true

# Результат слияния
[lint]
rules = ["recommended"]    # из пользовательского
severity = "warn"          # из пользовательского
strict = true              # из проектного
```

### Обратная совместимость

- ✅ Режим без файла конфигурации продолжает поддерживаться (все компоненты используют встроенные значения по умолчанию)
- ✅ Все новые параметры конфигурации имеют разумные значения по умолчанию
- ✅ При первом запуске команды конфигурация генерируется автоматически с параметрами по умолчанию
- ✅ При ошибке разбора конфигурации отображается понятное сообщение об ошибке с указанием конкретной строки и причины

## Компромиссы

### Преимущества

- Унифицированная инфраструктура конфигурации, сокращение дублирующего кода
- Единообразие пользовательских предпочтений между проектами
- LSP/REPL/Компилятор используют единый набор конфигураций
- Постепенная настройка конфигурации, объявление по необходимости

### Недостатки

- Большое количество параметров конфигурации, несколько повышенная кривая обучения
- Необходим унифицированный парсер конфигураций

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Независимая конфигурация каждого компонента | Дублирование кода, разрозненный пользовательский опыт |
| Только параметры командной строки | Невозможность сохранения пользовательских предпочтений |
| Только переменные окружения | Сложность версионирования проектных конфигураций |

## Стратегия реализации

### Фазы

| Фаза | Содержание |
|------|-----------|
| **Фаза 1** | Базовый парсер конфигурации, поддержка toml, проектная конфигурация, `yaoxiang config init` |
| **Фаза 2** | Пользовательская конфигурация, логика слияния конфигураций, `yaoxiang config edit/show` |
| **Фаза 3** | Переопределение через командную строку/переменные окружения, ограничения `platform`, расширение `[tool.*]` |

### Зависимости

- Зависит от RFC-014 системы управления пакетами

### Риски

| Риск | Меры снижения |
|------|---------------|
| Слишком много параметров конфигурации | Разумные значения по умолчанию, невидимость для пользователя |
| Сложность парсера | Использование существующей библиотеки toml |

### Принятые функции (третья фаза)

#### Ограничения `platform` для целевой платформы

> **Примечание**: Следующий синтаксис используется в файле конфигурации `yaoxiang.toml` **исключительно**, **не** в исходном коде YaoXiang (файлы `.yx`). Пользователям не нужно писать `cfg(...)` в коде.

Поддержка платформенно-специфичных конфигураций на основе целевой операционной системы/архитектуры:

```toml
# yaoxiang.toml (файл конфигурации)

[target.'cfg(windows)'.build]
output = "dist/win32"

[target.'cfg(unix)'.build]
output = "dist/unix"

[target.'cfg(target_arch = "x86_64")'.build]
rustflags = ["-C target-cpu=native"]
```

**Синтаксис**: `[target.'<условие>'.<секция конфигу>]`

**Описание**:

- Этот синтаксис появляется только в файле конфигурации `yaoxiang.toml`
- При сборке выбор соответствующей конфигурации осуществляется на основе параметра `--target`
- Пользователям **не нужно** и **не следует** писать синтаксис `cfg(...)` в исходном коде `.yx`

**Поддерживаемые условия**:

- `cfg(os = "windows")` — Система Windows
- `cfg(os = "linux")` — Система Linux
- `cfg(os = "macos")` — Система macOS
- `cfg(target_arch = "x86_64")` — 64-битная архитектура x86
- `cfg(target_arch = "aarch64")` — 64-битная архитектура ARM

#### Расширение `[tool.*]` для сторонних инструментов

Разрешение сторонним инструментам хранить конфигурацию в секции `[tool.<название>]`:

```toml
[tool.eslint]
extension = ["yx", "yxp"]
ignore = ["node_modules/", "dist/"]

[tool.prettier]
semi = false
singleQuote = true
```

**Поведение**:

- YaoXiang игнорирует неизвестные секции `[tool.*]`, но сохраняет их в файле конфигурации
- Сторонние инструменты могут быть интегрированы через `yaoxiang tool run <название>` или напрямую обращаться к конфигурации
- Конфигурация, специфичная для инструментов, не проверяется
