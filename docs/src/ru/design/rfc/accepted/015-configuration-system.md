# RFC-015: Дизайн системы конфигурации YaoXiang

> **Дата принятия**: 2026-02-15

> **Предшествующий RFC**: [RFC-014: Дизайн системы управления пакетами](014-package-manager.md)

## Резюме

Проектирование единой системы конфигурации языка YaoXiang с поддержкой двух уровней — пользовательского и проектного — для обеспечения общей инфраструктуры конфигурации для таких компонентов, как пакетный менеджер, компилятор, REPL, LSP и др.

## Мотивация

### Зачем нужна эта возможность/изменение?

Инструментальная цепочка YaoXiang включает несколько компонентов:
- Пакетный менеджер (читает конфигурацию зависимостей)
- Фронтенд компилятора (читает конфигурацию i18n)
- REPL (читает конфигурацию интерактивного режима)
- LSP (читает конфигурацию fmt/lint/test)
- Система сборки (читает конфигурацию сборки)

Каждому компоненту требуется единая инфраструктура конфигурации.

### Текущая проблема

- Конфигурации компонентов разрозненны, нет единого стандарта
- Пользователь не может централизованно управлять предпочтениями
- Нет чёткого разделения уровней между проектной и пользовательской конфигурацией

## Предложение

### Основной дизайн

**Многоуровневая архитектура**:
```
Приоритет конфигурации (высокий → низкий):
┌─────────────────────────────────────────────┐
│ 1. Проектный yaoxiang.toml                   │ ← Управляется командой проекта
├─────────────────────────────────────────────┤
│ 2. Пользовательский ~/.config/yaoxiang/config.toml │ ← Предпочтения пользователя
├─────────────────────────────────────────────┤
│ 3. Значения по умолчанию компилятора          │ ← Разумные начальные значения
└─────────────────────────────────────────────┘
```

**Правило**: верхний уровень перекрывает нижний; неуказанные опции откатываются к нижнему уровню.

### Ограничения уровней конфигурации

| Секция конфигурации | Пользовательский | Проектный | Потребитель |
|--------|--------|--------|--------|
| `[package].*` | ❌ | ✅ | Пакетный менеджер |
| `[yaoxiang]` | ❌ | ✅ | Компилятор |
| `[dependencies]` | ❌ | ✅ | Пакетный менеджер |
| `[dev-dependencies]` | ❌ | ✅ | Пакетный менеджер |
| `[bin]` | ❌ | ✅ | Пакетный менеджер |
| `[lib]` | ❌ | ✅ | Пакетный менеджер |
| `[build]` | ✅ | ✅ | Система сборки |
| `[profile.*]` | ✅ | ✅ | Система сборки |
| `[install]` | ✅ | ❌ | Пакетный менеджер |
| `[i18n]` | ✅ | ✅ | Компилятор |
| `[repl]` | ✅ | ✅ | REPL |
| `[fmt]` | ✅ | ✅ | LSP |
| `[lint]` | ✅ | ✅ | LSP |
| `[test]` | ✅ | ✅ | LSP |
| `[tasks]` | ✅ | ✅ | CLI |

### Примеры

**Проектная конфигурация**:
```toml
# yaoxiang.toml
[package]
name = "my-package"
version = "0.1.0"

[yaoxiang]
version = ">=0.1.0, <1.0.0"

[dependencies]
foo = "^1.0.0"

[build]
output = "dist/"

[tasks]
build = "yaoxiang build"
test = "yaoxiang test"
```

**Пользовательская конфигурация**:
```toml
# ~/.config/yaoxiang/config.toml
[install]
dir = "~/.local/share/yaoxiang"

[i18n]
lang = "zh"
fallback = "en"

[repl]
history-size = 1000
prompt = "yx> "
colors = true

[fmt]
line-width = 120
indent-width = 4

[lint]
rules = ["recommended"]
```

## Детальный дизайн

### Специфичная проектная конфигурация

```toml
[package]
name = "my-package"
version = "0.1.0"
description = "A short description"
authors = ["Alice <alice@example.com>"]
license = "MIT"
repository = "https://github.com/alice/my-project"

[yaoxiang]
version = ">=0.1.0, <1.0.0"

[dependencies]
foo = "^1.0.0"

[dev-dependencies]
test-utils = "0.1.0"

[lib]
path = "src/lib.yx"

[[bin]]
name = "my-cli"
path = "src/cli.yx"

[exports]
"." = "src/lib.yx"
"./foo" = "src/foo.yx"

[build]
script = "build.yx"
output = "dist/"

[profile.release]
optimize = true
lto = true

[run]
main = "src/main.yx"
args = ["--quiet"]

[tasks]
build = "yaoxiang build"
test = "yaoxiang test"
lint = "yaoxiang fmt && yaoxiang check"
```

### Специфичная пользовательская конфигурация

```toml
[install]
dir = "~/.local/share/yaoxiang"
```

### Конфигурация для обоих уровней

| Поле | Тип | Значение по умолчанию | Описание |
|------|------|--------|------|
| `[i18n].lang` | String | "en" | Язык |
| `[i18n].fallback` | String | "en" | Резервный язык |
| `[repl].history-size` | Number | 1000 | Количество записей в истории |
| `[repl].history-file` | Path | ~ | Файл истории |
| `[repl].prompt` | String | "yx> " | Приглашение |
| `[repl].colors` | Boolean | true | Подсветка синтаксиса |
| `[repl].auto-imports` | [String] | [] | Автоимпорты |
| `[fmt].line-width` | Number | 120 | Ширина строки |
| `[fmt].indent-width` | Number | 4 | Ширина отступа |
| `[fmt].use-tabs` | Boolean | false | Отступы табами |
| `[fmt].single-quote` | Boolean | false | Одинарные кавычки |
| `[lint].rules` | [String] | ["recommended"] | Набор правил |
| `[lint].strict` | Boolean | false | Строгий режим |
| `[test].report` | String | "console" | Отчёт тестов |
| `[build].output` | String | "dist/" | Каталог вывода |

### Переопределение через командную строку и переменные окружения

```bash
# Переопределение через командную строку
yaoxiang run main.yx --lang zh
yaoxiang fmt --config-indent-width=2

# Переменные окружения
export YAOXIANG_LANG=zh
export YAOXIANG_FMT_INDENT_WIDTH=2
```

**Приоритет**: `командная строка > переменные окружения > файл конфигурации`

### Команда `yaoxiang config`

Предоставление CLI-команды для управления конфигурацией:

```bash
# Инициализация пользовательской конфигурации (генерация с параметрами по умолчанию)
yaoxiang config init

# Редактирование пользовательской конфигурации (открытие в редакторе)
yaoxiang config edit

# Просмотр текущей конфигурации (объединённый эффективный результат)
yaoxiang config show

# Просмотр источника конфигурации
yaoxiang config show --source

# Сброс к конфигурации по умолчанию
yaoxiang config reset
```

**Первый запуск**: при первом запуске любой команды `yaoxiang` выполняется автоматическая проверка наличия пользовательской конфигурации. Если она отсутствует — автоматически генерируется с параметрами по умолчанию.

**Расположение файлов конфигурации**:
- Проектный: `./yaoxiang.toml` (корень проекта)
- Пользовательский: `~/.config/yaoxiang/config.toml`

### Семантика слияния конфигурации

Конфигурации разных уровней сливаются по следующим правилам:

| Тип | Стратегия | Описание |
|------|------|------|
| Скаляр (String/Number/Boolean) | Замена | Проектный перекрывает пользовательский |
| Массив (Array) | Замена | Проектный полностью заменяет пользовательский |
| Объект (Object) | Глубокое слияние | Полевое слияние; неопределённые поля наследуются с нижнего уровня |

**Пример — глубокое слияние объектов**:
```toml
# Пользовательский уровень
[lint]
rules = ["recommended"]
severity = "warn"

# Проектный уровень
[lint]
strict = true

# Результат слияния
[lint]
rules = ["recommended"]    # Из пользовательского
severity = "warn"          # Из пользовательского
strict = true             # Из проектного
```

### Обратная совместимость

- ✅ Существующий режим без файла конфигурации продолжает поддерживаться (все компоненты используют встроенные значения по умолчанию)
- ✅ Для всех новых параметров конфигурации предусмотрены разумные значения по умолчанию
- ✅ При первом запуске команды пользовательская конфигурация автоматически генерируется с параметрами по умолчанию
- ✅ При ошибке разбора конфигурации отображается понятное сообщение с указанием конкретной строки и причины ошибки

## Компромиссы

### Преимущества

- Единая инфраструктура конфигурации, уменьшение дублирования кода
- Предпочтения пользователя согласованы между проектами
- LSP/REPL/компилятор используют общую конфигурацию
- Прогрессивная конфигурация, декларация по необходимости

### Недостатки

- Множество параметров конфигурации, немного повышается порог входа
- Требуется единый парсер конфигурации

## Альтернативы

| Альтернатива | Почему не выбрана |
|------|-----------|
| Независимая конфигурация каждого компонента | Дублирование кода, фрагментированный пользовательский опыт |
| Поддержка только параметров командной строки | Невозможно сохранить предпочтения пользователя |
| Поддержка только переменных окружения | Затруднено версионирование проектной конфигурации |

## Стратегия реализации

### Этапы

| Этап | Содержание |
|------|------|
| **Phase 1** | Базовый парсер конфигурации, поддержка toml, проектная конфигурация, `yaoxiang config init` |
| **Phase 2** | Пользовательская конфигурация, логика слияния, `yaoxiang config edit/show` |
| **Phase 3** | Переопределение через CLI/переменные окружения, ограничения `platform`, расширение `[tool.*]` |

### Зависимости

- Зависит от RFC-014 "Система управления пакетами"

### Риски

| Риск | Меры по снижению |
|------|----------|
| Слишком много параметров конфигурации | Разумные значения по умолчанию, незаметно для пользователя |
| Сложность парсера | Использование существующей библиотеки toml |

## Открытые вопросы

- [x] Синтаксис условной компиляции `features`? → **Вынесено в отдельный RFC**, зависит от RFC-011 "Система generics"
- [x] Дизайн `workspace`? → **Вынесено в отдельный RFC**, высокая сложность, требует отдельного проектирования

### Принятые возможности (третий этап)

#### Платформенные ограничения `platform`

> **Внимание**: приведённый ниже синтаксис используется в **файле конфигурации** `yaoxiang.toml`, а **не** в исходном коде YaoXiang (файлы `.yx`). Пользователю не нужно писать `cfg(...)` в коде.

Поддержка платформенно-специфичной конфигурации на основе целевой ОС/архитектуры:

```toml
# yaoxiang.toml (файл конфигурации)

[target.'cfg(windows)'.build]
output = "dist/win32"

[target.'cfg(unix)'.build]
output = "dist/unix"

[target.'cfg(target_arch = "x86_64")'.build]
rustflags = ["-C target-cpu=native"]
```

**Синтаксис**: `[target.'<условие>'.<секция конфигурации>]`

**Пояснения**:
- Этот синтаксис встречается только в файле конфигурации `yaoxiang.toml`
- При сборке соответствующая конфигурация выбирается по параметру `--target`
- В исходном коде `.yx` пользователю **не нужно** и **не следует** писать синтаксис `cfg(...)`

**Поддерживаемые условия**:
- `cfg(os = "windows")` — система Windows
- `cfg(os = "linux")` — система Linux
- `cfg(os = "macos")` — система macOS
- `cfg(target_arch = "x86_64")` — 64-битная архитектура x86
- `cfg(target_arch = "aarch64")` — 64-битная архитектура ARM

#### Расширение конфигурации сторонних инструментов `[tool.*]`

Разрешить сторонним инструментам хранить свою конфигурацию в секции `[tool.<имя>]`:

```toml
[tool.eslint]
extension = ["yx", "yxp"]
ignore = ["node_modules/", "dist/"]

[tool.prettier]
semi = false
singleQuote = true
```

**Поведение**:
- YaoXiang игнорирует неизвестные секции `[tool.*]`, но сохраняет их в файле конфигурации
- Сторонние инструменты могут интегрироваться через `yaoxiang tool run <имя>` или получать к ним прямой доступ
- Специфичная для инструмента конфигурация не валидируется
