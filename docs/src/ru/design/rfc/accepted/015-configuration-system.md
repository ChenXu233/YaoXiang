# RFC-015: Проектирование системы конфигурации YaoXiang

> **Статус**: Принято
> **Автор**: Чэнь Сюй (晨煦)
> **Дата создания**: 2026-02-12
> **Дата принятия**: 2026-02-15
> **Последнее обновление**: 2026-02-15

> **Предшествующий RFC**: [RFC-014: Проектирование системы управления пакетами](014-package-manager.md)

## Аннотация

Проектирование единой системы конфигурации для языка YaoXiang с поддержкой двух уровней: пользовательского и проектного. Система предоставляет общую инфраструктуру конфигурации для менеджера пакетов, компилятора, REPL, LSP и других компонентов.

## Мотивация

### Зачем нужна эта функция/изменение?

Цепочка инструментов YaoXiang включает несколько компонентов:

- Менеджер пакетов (чтение конфигурации зависимостей)
- Интерфейсный компилятор (чтение конфигурации i18n)
- REPL (чтение интерактивной конфигурации)
- LSP (чтение конфигурации fmt/lint/test)
- Система сборки (чтение конфигурации сборки)

Всем компонентам необходима единая инфраструктура конфигурации.

### Текущие проблемы

- Конфигурации компонентов разрозненны, отсутствует единый стандарт
- Пользователи не могут унифицированно управлять настройками предпочтений
- Отсутствует чёткое разделение между конфигурацией проекта и пользовательской конфигурацией

## Предложение

### Основной дизайн

**Многоуровневая архитектура**:

```
Приоритет конфигурации (высокий → низкий):
┌─────────────────────────────────────────────┐
│ 1. Проектный yaoxiang.toml                   │ ← Управляется командой проекта
├─────────────────────────────────────────────┤
│ 2. Пользовательский ~/.config/yaoxiang/config.toml │ ← Пользовательские предпочтения
├─────────────────────────────────────────────┤
│ 3. Значения по умолчанию компилятора         │ ← Разумные начальные значения
└─────────────────────────────────────────────┘
```

**Правило**: вышестоящий уровень переопределяет нижестоящий, а параметры без конфигурации наследуются от нижестоящего уровня.

### Ограничения уровней конфигурации

| Секция конфигурации | Пользовательский | Проектный | Потребитель |
|----------------------|------------------|-----------|-------------|
| `[package].*` | ❌ | ✅ | Менеджер пакетов |
| `[yaoxiang]` | ❌ | ✅ | Компилятор |
| `[dependencies]` | ❌ | ✅ | Менеджер пакетов |
| `[dev-dependencies]` | ❌ | ✅ | Менеджер пакетов |
| `[bin]` | ❌ | ✅ | Менеджер пакетов |
| `[lib]` | ❌ | ✅ | Менеджер пакетов |
| `[build]` | ✅ | ✅ | Система сборки |
| `[profile.*]` | ✅ | ✅ | Система сборки |
| `[install]` | ✅ | ❌ | Менеджер пакетов |
| `[i18n]` | ✅ | ✅ | Компилятор |
| `[repl]` | ✅ | ✅ | REPL |
| `[fmt]` | ✅ | ✅ | LSP |
| `[lint]` | ✅ | ✅ | LSP |
| `[test]` | ✅ | ✅ | LSP |
| `[tasks]` | ✅ | ✅ | CLI |

### Примеры

**Проектная конфигурация**:

```toml
# yaoxiang.toml
[package]
name = "my-package"
version = "0.1.0"

[yaoxiang]
version = ">=0.1.0, <1.0.0"

[dependencies]
foo = "^1.0.0"

[build]
output = "dist/"

[tasks]
build = "yaoxiang build"
test = "yaoxiang test"
```

**Пользовательская конфигурация**:

```toml
# ~/.config/yaoxiang/config.toml
[install]
dir = "~/.local/share/yaoxiang"

[i18n]
lang = "zh"
fallback = "en"

[repl]
history-size = 1000
prompt = "yx> "
colors = true

[fmt]
line-width = 120
indent-width = 4

[lint]
rules = ["recommended"]
```

## Детальный дизайн

### Конфигурация, специфичная для проекта

```toml
[package]
name = "my-package"
version = "0.1.0"
description = "Краткое описание"
authors = ["Алиса <alice@example.com>"]
license = "MIT"
repository = "https://github.com/alice/my-project"

[yaoxiang]
version = ">=0.1.0, <1.0.0"

[dependencies]
foo = "^1.0.0"

[dev-dependencies]
test-utils = "0.1.0"

[lib]
path = "src/lib.yx"

[[bin]]
name = "my-cli"
path = "src/cli.yx"

[exports]
"." = "src/lib.yx"
"./foo" = "src/foo.yx"

[build]
script = "build.yx"
output = "dist/"

[profile.release]
optimize = true
lto = true

[run]
main = "src/main.yx"
args = ["--quiet"]

[tasks]
build = "yaoxiang build"
test = "yaoxiang test"
lint = "yaoxiang fmt && yaoxiang check"
```

### Конфигурация, специфичная для пользователя

```toml
[install]
dir = "~/.local/share/yaoxiang"
```

### Конфигурация для обоих уровней

| Поле | Тип | Значение по умолчанию | Описание |
|------|------|-----------------------|----------|
| `[i18n].lang` | Строка | "en" | Язык |
| `[i18n].fallback` | Строка | "en" | Резервный язык |
| `[repl].history-size` | Число | 1000 | Количество записей истории |
| `[repl].history-file` | Путь | ~ | Файл истории |
| `[repl].prompt` | Строка | "yx> " | Приглашение |
| `[repl].colors` | Булево | true | Подсветка синтаксиса |
| `[repl].auto-imports` | [Строка] | [] | Автоимпорт |
| `[fmt].line-width` | Число | 120 | Ширина строки |
| `[fmt].indent-width` | Число | 4 | Отступ |
| `[fmt].use-tabs` | Булево | false | Табуляция для отступов |
| `[fmt].single-quote` | Булево | false | Одинарные кавычки |
| `[lint].rules` | [Строка] | ["recommended"] | Набор правил |
| `[lint].strict` | Булево | false | Строгий режим |
| `[test].report` | Строка | "console" | Отчёт о тестировании |
| `[build].output` | Строка | "dist/" | Выходной каталог |

### Переопределение через командную строку и переменные окружения

```bash
# Переопределение через командную строку
yaoxiang run main.yx --lang zh
yaoxiang fmt --config-indent-width=2

# Переменные окружения
export YAOXIANG_LANG=zh
export YAOXIANG_FMT_INDENT_WIDTH=2
```

**Приоритет**: `Командная строка > Переменные окружения > Файл конфигурации`

### Команда yaoxiang config

Предоставляется CLI-команда для управления конфигурацией:

```bash
# Инициализация пользовательской конфигурации (генерация по умолчанию)
yaoxiang config init

# Редактирование пользовательской конфигурации (открыть редактор)
yaoxiang config edit

# Просмотр текущей конфигурации (объединённой действующей)
yaoxiang config show

# Просмотр источника конфигурации
yaoxiang config show --source

# Сброс к конфигурации по умолчанию
yaoxiang config reset
```

**Первый запуск**: При первом запуске любой команды `yaoxiang` автоматически проверяется наличие пользовательской конфигурации. Если её нет, она автоматически генерируется с параметрами по умолчанию.

**Расположение файлов конфигурации**:

- Проектная: `./yaoxiang.toml` (в корне проекта)
- Пользовательская: `~/.config/yaoxiang/config.toml`

### Семантика слияния конфигураций

Конфигурации разных уровней объединяются по следующим правилам:

| Тип | Стратегия | Описание |
|-----|-----------|----------|
| Скаляр (Строка/Число/Булево) | Замена | Проектная переопределяет пользовательскую |
| Массив (Array) | Замена | Проектная полностью заменяет пользовательскую |
| Объект (Object) | Глубокое слияние | Поля объединяются построчно, неопределённые поля наследуются |

**Пример — глубокое слияние объектов**:

```toml
# Пользовательская
[lint]
rules = ["recommended"]
severity = "warn"

# Проектная
[lint]
strict = true

# Результат слияния
[lint]
rules = ["recommended"]    # Из пользовательской
severity = "warn"          # Из пользовательской
strict = true              # Из проектной
```

### Обратная совместимость

- ✅ Существующий режим без файла конфигурации продолжает поддерживаться (все компоненты используют встроенные значения по умолчанию)
- ✅ Все новые параметры конфигурации имеют разумные значения по умолчанию
- ✅ При первом запуске команды автоматически генерируется конфигурация с параметрами по умолчанию
- ✅ При ошибке разбора конфигурации отображается понятное сообщение об ошибке с указанием конкретной строки и причины

## Компромиссы

### Преимущества

- Единая инфраструктура конфигурации, сокращение дублирующегося кода
- Пользовательские предпочтения согласованы между проектами
- LSP/REPL/Компилятор используют единый набор конфигураций
- Постепенная конфигурация, объявление по необходимости

### Недостатки

- Большое количество параметров конфигурации, несколько увеличена кривая обучения
- Требуется единый парсер конфигурации

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Независимая конфигурация каждого компонента | Дублирующийся код, фрагментированный пользовательский опыт |
| Только параметры командной строки | Невозможность сохранять пользовательские предпочтения |
| Только переменные окружения | Сложность версионирования конфигурации проекта |

## Стратегия реализации

### Фазы

| Фаза | Содержание |
|------|------------|
| **Фаза 1** | Базовый парсер конфигурации, поддержка toml, проектная конфигурация, `yaoxiang config init` |
| **Фаза 2** | Пользовательская конфигурация, логика слияния, `yaoxiang config edit/show` |
| **Фаза 3** | Переопределение через командную строку/переменные окружения, ограничения `platform`, расширение `[tool.*]` |

### Зависимости

- Зависит от RFC-014 системы управления пакетами

### Риски

| Риск | Меры снижения |
|------|---------------|
| Слишком много параметров конфигурации | Разумные значения по умолчанию, прозрачность для пользователя |
| Сложность парсера | Использование существующей библиотеки toml |

### Принятые функции (третья фаза)

#### Ограничения `platform` для целевой платформы

> **Внимание**: Приведённый ниже синтаксис используется в файле конфигурации `yaoxiang.toml` **и только в нём**, **не** в исходном коде YaoXiang (`.yx` файлах). Пользователям не нужно писать `cfg(...)` в коде.

Поддержка конфигурации, специфичной для целевой операционной системы/архитектуры:

```toml
# yaoxiang.toml (файл конфигурации)

[target.'cfg(windows)'.build]
output = "dist/win32"

[target.'cfg(unix)'.build]
output = "dist/unix"

[target.'cfg(target_arch = "x86_64")'.build]
rustflags = ["-C target-cpu=native"]
```

**Синтаксис**: `[target.'<условие>'.<секция_конфигу>]`

**Описание**:

- Этот синтаксис появляется только в файле конфигурации `yaoxiang.toml`
- При сборке выбирается соответствующая конфигурация на основе параметра `--target`
- В исходном коде `.yx` пользователям **не нужно** и **не следует** писать синтаксис `cfg(...)`

**Поддерживаемые условия**:

- `cfg(os = "windows")` — Система Windows
- `cfg(os = "linux")` — Система Linux
- `cfg(os = "macos")` — Система macOS
- `cfg(target_arch = "x86_64")` — 64-битная архитектура x86
- `cfg(target_arch = "aarch64")` — 64-битная архитектура ARM

#### Расширение `[tool.*]` для сторонних инструментов

Позволяет сторонним инструментам хранить конфигурацию в `[tool.<название>]`:

```toml
[tool.eslint]
extension = ["yx", "yxp"]
ignore = ["node_modules/", "dist/"]

[tool.prettier]
semi = false
singleQuote = true
```

**Поведение**:

- YaoXiang игнорирует неизвестные секции `[tool.*]`, но сохраняет их в файле конфигурации
- Сторонние инструменты могут быть интегрированы через `yaoxiang tool run <название>` или напрямую обращаться к конфигурации
- Конфигурация инструментов не проверяется
