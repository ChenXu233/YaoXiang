# RFC-015: Дизайн системы конфигурации YaoXiang

> **Дата принятия**: 2026-02-15

> **Предшествующий RFC**: [RFC-014: Дизайн системы управления пакетами](014-package-manager.md)

## Резюме

Проектирование единой системы конфигурации языка YaoXiang с поддержкой двух уровней — пользовательского и проектного, предоставляющей общую инфраструктуру конфигурации для таких компонентов, как менеджер пакетов, компилятор, REPL, LSP и других.

## Мотивация

### Зачем нужна эта возможность/изменение?

Инструментальная цепочка YaoXiang включает несколько компонентов:
- Менеджер пакетов (чтение конфигурации зависимостей)
- Фронтенд компилятора (чтение конфигурации i18n)
- REPL (чтение интерактивной конфигурации)
- LSP (чтение конфигурации fmt/lint/test)
- Система сборки (чтение конфигурации сборки)

Каждому компоненту необходима единая инфраструктура конфигурации.

### Текущие проблемы

- Конфигурации компонентов разрозненны и не имеют единой спецификации
- Пользователи не могут централизованно управлять предпочтениями
- Между проектной и пользовательской конфигурациями нет чёткой иерархии

## Предложение

### Основной дизайн

**Многоуровневая архитектура**:
```
Приоритет конфигурации (от высокого к низкому):
┌─────────────────────────────────────────────┐
│ 1. Проектный yaoxiang.toml                   │ ← Управляется командой проекта
├─────────────────────────────────────────────┤
│ 2. Пользовательский ~/.config/yaoxiang/config.toml │ ← Предпочтения пользователя
├─────────────────────────────────────────────┤
│ 3. Значения по умолчанию компилятора         │ ← Разумные начальные значения
└─────────────────────────────────────────────┘
```

**Правило**: верхний уровень перекрывает нижний; неуказанные опции откатываются на уровень ниже.

### Ограничения уровней конфигурации

| Секция конфигурации | Пользовательский | Проектный | Потребитель |
|---------------------|------------------|-----------|-------------|
| `[package].*` | ❌ | ✅ | Менеджер пакетов |
| `[yaoxiang]` | ❌ | ✅ | Компилятор |
| `[dependencies]` | ❌ | ✅ | Менеджер пакетов |
| `[dev-dependencies]` | ❌ | ✅ | Менеджер пакетов |
| `[bin]` | ❌ | ✅ | Менеджер пакетов |
| `[lib]` | ❌ | ✅ | Менеджер пакетов |
| `[build]` | ✅ | ✅ | Система сборки |
| `[profile.*]` | ✅ | ✅ | Система сборки |
| `[install]` | ✅ | ❌ | Менеджер пакетов |
| `[i18n]` | ✅ | ✅ | Компилятор |
| `[repl]` | ✅ | ✅ | REPL |
| `[fmt]` | ✅ | ✅ | LSP |
| `[lint]` | ✅ | ✅ | LSP |
| `[test]` | ✅ | ✅ | LSP |
| `[tasks]` | ✅ | ✅ | CLI |

### Примеры

**Проектная конфигурация**:
```toml
# yaoxiang.toml
[package]
name = "my-package"
version = "0.1.0"

[yaoxiang]
version = ">=0.1.0, <1.0.0"

[dependencies]
foo = "^1.0.0"

[build]
output = "dist/"

[tasks]
build = "yaoxiang build"
test = "yaoxiang test"
```

**Пользовательская конфигурация**:
```toml
# ~/.config/yaoxiang/config.toml
[install]
dir = "~/.local/share/yaoxiang"

[i18n]
lang = "zh"
fallback = "en"

[repl]
history-size = 1000
prompt = "yx> "
colors = true

[fmt]
line-width = 120
indent-width = 4

[lint]
rules = ["recommended"]
```

## Детальный дизайн

### Конфигурация, относящаяся только к проекту

```toml
[package]
name = "my-package"
version = "0.1.0"
description = "A short description"
authors = ["Alice <alice@example.com>"]
license = "MIT"
repository = "https://github.com/alice/my-project"

[yaoxiang]
version = ">=0.1.0, <1.0.0"

[dependencies]
foo = "^1.0.0"

[dev-dependencies]
test-utils = "0.1.0"

[lib]
path = "src/lib.yx"

[[bin]]
name = "my-cli"
path = "src/cli.yx"

[exports]
"." = "src/lib.yx"
"./foo" = "src/foo.yx"

[build]
script = "build.yx"
output = "dist/"

[profile.release]
optimize = true
lto = true

[run]
main = "src/main.yx"
args = ["--quiet"]

[tasks]
build = "yaoxiang build"
test = "yaoxiang test"
lint = "yaoxiang fmt && yaoxiang check"
```

### Конфигурация, относящаяся только к пользователю

```toml
[install]
dir = "~/.local/share/yaoxiang"
```

### Конфигурация, доступная на обоих уровнях

| Поле | Тип | Значение по умолчанию | Описание |
|------|-----|----------------------|----------|
| `[i18n].lang` | Строка | "en" | Язык |
| `[i18n].fallback` | Строка | "en" | Язык отката |
| `[repl].history-size` | Число | 1000 | Количество записей в истории |
| `[repl].history-file` | Путь | ~ | Файл истории |
| `[repl].prompt` | Строка | "yx> " | Приглашение |
| `[repl].colors` | Логический | true | Подсветка синтаксиса |
| `[repl].auto-imports` | [Строка] | [] | Автоматический импорт |
| `[fmt].line-width` | Число | 120 | Ширина строки |
| `[fmt].indent-width` | Число | 4 | Ширина отступа |
| `[fmt].use-tabs` | Логический | false | Отступ табами |
| `[fmt].single-quote` | Логический | false | Одинарные кавычки |
| `[lint].rules` | [Строка] | ["recommended"] | Набор правил |
| `[lint].strict` | Логический | false | Строгий режим |
| `[test].report` | Строка | "console" | Отчёт о тестировании |
| `[build].output` | Строка | "dist/" | Директория вывода |

### Переопределение через командную строку и переменные окружения

```bash
# Переопределение через командную строку
yaoxiang run main.yx --lang zh
yaoxiang fmt --config-indent-width=2

# Переменные окружения
export YAOXIANG_LANG=zh
export YAOXIANG_FMT_INDENT_WIDTH=2
```

**Приоритет**: `командная строка > переменные окружения > файл конфигурации`

### Команда yaoxiang config

Предоставление CLI-команд для управления конфигурацией:

```bash
# Инициализация пользовательской конфигурации (генерация с параметрами по умолчанию)
yaoxiang config init

# Редактирование пользовательской конфигурации (открытие в редакторе)
yaoxiang config edit

# Просмотр текущей конфигурации (объединённой эффективной конфигурации)
yaoxiang config show

# Просмотр источника конфигурации
yaoxiang config show --source

# Сброс к конфигурации по умолчанию
yaoxiang config reset
```

**Первый запуск**: при первом запуске любой команды `yaoxiang` выполняется автоматическая проверка наличия пользовательской конфигурации. Если она отсутствует — автоматически генерируется файл со значениями по умолчанию.

**Расположение файлов конфигурации**:
- Проектный: `./yaoxiang.toml` (корень проекта)
- Пользовательский: `~/.config/yaoxiang/config.toml`

### Семантика слияния конфигураций

Конфигурации разных уровней объединяются по следующим правилам:

| Тип | Стратегия | Описание |
|-----|-----------|----------|
| Скаляр (Строка/Число/Логический) | Замена | Проектный перекрывает пользовательский |
| Массив (Array) | Замена | Проектный полностью заменяет пользовательский |
| Объект (Object) | Глубокое слияние | Полевое слияние, неопределённые поля наследуются с нижнего уровня |

**Пример — глубокое слияние объектов**:
```toml
# Пользовательский
[lint]
rules = ["recommended"]
severity = "warn"

# Проектный
[lint]
strict = true

# Результат слияния
[lint]
rules = ["recommended"]    # Из пользовательского
severity = "warn"          # Из пользовательского
strict = true              # Из проектного
```

### Обратная совместимость

- ✅ Продолжается поддержка существующего режима без файла конфигурации (все компоненты используют встроенные значения по умолчанию)
- ✅ Все новые опции конфигурации имеют разумные значения по умолчанию
- ✅ При первом запуске команды пользовательская конфигурация автоматически генерируется со значениями по умолчанию
- ✅ При ошибке разбора конфигурации отображается понятное сообщение с указанием конкретной строки и причины ошибки

## Компромиссы

### Преимущества

- Единая инфраструктура конфигурации, сокращение дублирования кода
- Предпочтения пользователя единообразны между проектами
- LSP/REPL/компилятор используют общий набор конфигурации
- Прогрессивная конфигурация с декларацией по необходимости

### Недостатки

- Множество опций конфигурации — небольшое увеличение порога входа
- Требуется единый парсер конфигурации

## Альтернативы

| Вариант | Почему не выбран |
|---------|------------------|
| Независимые конфигурации для каждого компонента | Дублирование кода, разрозненный пользовательский опыт |
| Поддержка только параметров командной строки | Невозможно сохранить пользовательские предпочтения |
| Поддержка только переменных окружения | Проектную конфигурацию сложно версионировать |

## Стратегия реализации

### Этапы

| Этап | Содержание |
|------|------------|
| **Фаза 1** | Базовый парсер конфигурации, поддержка toml, проектная конфигурация, `yaoxiang config init` |
| **Фаза 2** | Пользовательская конфигурация, логика слияния конфигураций, `yaoxiang config edit/show` |
| **Фаза 3** | Переопределение через командную строку/переменные окружения, ограничения `platform`, расширения `[tool.*]` |

### Зависимости

- Зависит от RFC-014 "Система управления пакетами"

### Риски

| Риск | Меры по снижению |
|------|------------------|
| Слишком много опций конфигурации | Разумные значения по умолчанию, пользователь не замечает |
| Сложность парсера | Использование существующих библиотек toml |

## Открытые вопросы

- [x] Синтаксис условной компиляции `features`? → **Перенесён в отдельный RFC**, зависит от RFC-011 "Система generics"

### Принятые возможности (третья фаза)

#### Платформенные ограничения `platform`

> **Внимание**: приведённый ниже синтаксис используется в файле конфигурации **yaoxiang.toml**, а **не** в исходном коде YaoXiang (файлы `.yx`). Пользователю **не нужно** писать `cfg(...)` в коде.

Поддержка платформо-специфичной конфигурации на основе целевой ОС/архитектуры:

```toml
# yaoxiang.toml (файл конфигурации)

[target.'cfg(windows)'.build]
output = "dist/win32"

[target.'cfg(unix)'.build]
output = "dist/unix"

[target.'cfg(target_arch = "x86_64")'.build]
rustflags = ["-C target-cpu=native"]
```

**Синтаксис**: `[target.'<условие>'.<секция конфигурации>]`

**Пояснение**:
- Данный синтаксис используется только в файле конфигурации `yaoxiang.toml`
- Во время сборки соответствующая конфигурация выбирается по параметру `--target`
- В исходном коде `.yx` пользователю **не нужно** и **не следует** писать синтаксис `cfg(...)`

**Поддерживаемые условия**:
- `cfg(os = "windows")` — система Windows
- `cfg(os = "linux")` — система Linux
- `cfg(os = "macos")` — система macOS
- `cfg(target_arch = "x86_64")` — 64-битная архитектура x86
- `cfg(target_arch = "aarch64")` — 64-битная архитектура ARM

#### Расширения конфигурации сторонних инструментов `[tool.*]`

Сторонним инструментам разрешается хранить конфигурацию в секции `[tool.<имя>]`:

```toml
[tool.eslint]
extension = ["yx", "yxp"]
ignore = ["node_modules/", "dist/"]

[tool.prettier]
semi = false
singleQuote = true
```

**Поведение**:
- YaoXiang игнорирует неизвестные секции `[tool.*]`, но сохраняет их в файле конфигурации
- Сторонние инструменты могут интегрироваться через `yaoxiang tool run <имя>` или обращаться напрямую
- Конфигурация конкретных инструментов не валидируется
