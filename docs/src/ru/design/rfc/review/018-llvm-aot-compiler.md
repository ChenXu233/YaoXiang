# RFC-018: Проект LLVM AOT-компилятора

> **Ссылки**:
> - [RFC-024: Модель параллелизма на основе блоков spawn](../accepted/024-concurrency-model.md)
> - [RFC-008: Разделение модели параллелизма среды выполнения и планировщика](../accepted/008-runtime-concurrency-model.md)
> - [RFC-009: Модель владения](../accepted/009-ownership-model.md)
> - [RFC-026: Основной механизм FFI](./026-ffi-core-mechanism.md)
> - [RFC-010: Единый синтаксис типов](../accepted/010-unified-type-syntax.md)

> **Устарело**:
> - Прежняя модель «автоматического восходящего DAG-анализа» — заменена моделью прямых подвыражений блоков spawn из RFC-024
> - Неявный вывод побочных эффектов `@IO`/`@Pure` — заменён механизмом типов-ресурсов из RFC-024
> - Отображение типа `Arc(T)` — заменено ключевым словом `ref` из RFC-009 v9

## Резюме

В данном документе описывается LLVM AOT (Ahead-of-Time) компилятор языка YaoXiang. Бэкенд LLVM и бэкенд VM (интерпретатор) используют общий компиляционный фронтенд, образуя архитектуру с двумя бэкендами, определённую в [RFC-008](../accepted/008-runtime-concurrency-model.md): VM применяется для разработки и отладки, LLVM — для промышленных релизов.

**Основные обязанности**:

```
Исходный код → Фронтенд (общий) → IR → LLVM Codegen → .o → Линковка статической библиотеки планировщика → exe
```

Компилятор преобразует исходный код YaoXiang в нативный машинный код, где:

| Возможность языка | Стратегия компиляции |
|------------------|---------------------|
| Обычный код | Последовательный машинный код, нулевые накладные расходы на планирование |
| Блок `spawn { }` | Прямые подвыражения → распределение задач + синхронное ожидание (согласно [RFC-024](../accepted/024-concurrency-model.md)) |
| `native("symbol")` | LLVM `declare external` + marshalling аргументов (согласно [RFC-026](./026-ffi-core-mechanism.md)) |
| Деструктор `.drop` | Вставка RAII cleanup-кода (согласно [RFC-009](../accepted/009-ownership-model.md)) |
| Токены `&T` / `&mut T` | Типы нулевого размера, исчезают после компиляции |
| Общее владение `ref T` | «Толстый» указатель `{ refcount_ptr, data_ptr }`, компилятор автоматически выбирает Rc/Arc |

**Связь с RFC-024**: RFC-024 определяет **пользовательскую семантику** блоков spawn (создание задач из прямых подвыражений, синхронное блокирующее ожидание). Данный документ определяет, как эта семантика **компилируется в машинный код**.

**Связь с RFC-026**: RFC-026 определяет **пользовательский синтаксис** FFI (`native()`, привязка методов через `[0]`, `.drop`). Данный документ определяет, как FFI-вызовы **генерируются в LLVM IR**.

## Мотивация

### Зачем нужен LLVM AOT-компилятор?

В настоящее время YaoXiang имеет только интерпретатор в качестве исполнительного бэкенда:

| Проблема | Влияние |
|----------|---------|
| Узкое место производительности | Интерпретация в 10-100x медленнее машинного кода |
| Сложность развёртывания | Требует наличия интерпретатора и среды выполнения |
| Промышленные сценарии | Интерпретатор не подходит для задач, чувствительных к производительности |

### LLVM в модели с двумя бэкендами

[RFC-008](../accepted/008-runtime-concurrency-model.md) §6 определяет архитектуру с двумя бэкендами:

```
                    ┌──────────────────────┐
                    │  Фронтенд (единый)   │
                    │  Lexer → Parser      │
                    │  → TypeCheck         │
                    │  → Анализ spawn      │
                    │  → Анализ побегов    │
                    └──────────┬───────────┘
                               │
                  ┌────────────┴─────────────┐
                  ▼                          ▼
      ┌────────────────────┐    ┌────────────────────┐
      │  Бэкенд VM (dev)   │    │ Бэкенд LLVM (prod)  │
      │  IR → интерпретация│    │  IR → нативный код  │
      │  пошаговая отладка │    │  линковка стат.     │
      │  быстрая итерация  │    │  библиотеки         │
      │                    │    │  планировщика → .exe│
      └────────────────────┘    └────────────────────┘
```

Поведение **полностью идентично** у обоих бэкендов — различается лишь способ исполнения. Те же исходные коды, тот же вывод типов, тот же результат анализа spawn.

## Предложение

### 1. Архитектура компилятора

LLVM-бэкенд расположен на последней стадии конвейера компиляции: получает IR от фронтенда и генерирует нативный код.

```
Исходный код
  → Lexer / Parser (frontend/core/)
  → TypeCheck + анализ spawn (frontend/core/typecheck/)
  → Генерация IR (middle/core/ir_gen.rs)
  → LLVM Codegen (middle/passes/codegen/llvm/)
      ├── Отображение типов: типы YaoXiang → типы LLVM IR
      ├── Трансляция функций: инструкции IR → инструкции LLVM IR
      ├── Развёртка spawn: прямые подвыражения → функция задачи + вызов планировщика
      ├── Развёртка FFI: вызовы native() → declare + marshalling
      └── Вставка деструкторов: конец области видимости → вызов .drop()
  → Оптимизации LLVM + генерация кода для целевой платформы
  → Линковка статической библиотеки среды выполнения → исполняемый файл
```

### 2. Процесс компиляции

```
Фаза 1: Фронтенд (общий с VM-бэкендом)
  - Разбор, проверка типов, анализ блоков spawn, анализ побегов
  - Выход: IR с аннотациями типов

Фаза 2: Генерация LLVM IR
  - Отображение типов, объявления функций, трансляция инструкций
  - Выход: LLVM Module

Фаза 3: Оптимизация LLVM
  - Стандартный конвейер оптимизаций LLVM (O0/O1/O2/O3)
  - Встраивание, свёртывание констант, удаление мёртвого кода

Фаза 4: Генерация кода для целевой платформы
  - LLVM TargetMachine → объектные файлы .o
  - Платформы: Linux (ELF), macOS (Mach-O), Windows (COFF)

Фаза 5: Линковка
  - Линковка статической библиотеки среды выполнения (планировщик, аллокатор)
  - Выход: исполняемый файл
```

### 3. Отображение типов

#### 3.1 Отображение типов YaoXiang → LLVM IR

| Тип YaoXiang | Тип LLVM IR | Описание |
|--------------|-------------|----------|
| `Int` | `i64` | Целое со знаком по умолчанию, 64 бита |
| `Int32` | `i32` | Явное 32-битное целое (в основном для FFI) |
| `Float` | `f64` | 64-битное число с плавающей точкой по умолчанию |
| `Float32` | `f32` | Явное 32-битное число с плавающей точкой (в основном для FFI) |
| `Bool` | `i1` | Логическое значение |
| `Char` | `i32` | Unicode-кодпоинт |
| `String` | `{ i8*, i64 }` | Указатель + длина в байтах |
| `Void` | `{}` | Пустой тип нулевого размера |
| `&T` | — | Токен нулевого размера, исчезает после компиляции, не порождает IR |
| `&mut T` | — | Токен нулевого размера, исчезает после компиляции, не порождает IR |
| `ref T` | `{ i32*, T* }` | Толстый указатель (указатель на счётчик ссылок + указатель на данные) |
| `*T` | `T*` | Голый указатель |
| `[T; N]` | `[N x T]` | Массив фиксированной длины |
| `List(T)` | `{ T*, i64, i64 }` | Указатель на данные + длина + ёмкость |
| Структура | Соответствующий struct LLVM | Поля располагаются в порядке определения |
| Record enum | `{ i64, [max_payload_size] }` | Тег + union максимального размера payload |
| `?T` | `{ i1, T }` | Флаг наличия значения + данные (универсальное представление) |
| Непрозрачный тип FFI | `{ i8* }` | Обёртка C-указателя |
| Указатель на функцию | `T (...)*` | Тип указателя на функцию |

> **Нулевые накладные расходы `&T` / `&mut T`**: [RFC-009](../accepted/009-ownership-model.md) §2.7 определяет, что компилятор внутренне присваивает токенам брендовые идентификаторы (уникальные целые числа на этапе компиляции). После мономорфизации и встраивания бренды полностью исчезают — в сгенерированном машинном коде нет никаких следов токенов.

#### 3.2 Отображение типов аргументов FFI

Согласно [RFC-026](./026-ffi-core-mechanism.md) §2.2, с добавлением колонки LLVM IR:

| Тип C | Тип YaoXiang | LLVM IR | Описание |
|-------|--------------|---------|----------|
| `int` | `Int32` | `i32` | |
| `long` | `Int64` | `i64` | |
| `float` | `Float32` | `f32` | |
| `double` | `Float64` | `f64` | |
| `char` | `Char` | `i32` | C `char` → YaoXiang `Char` (Unicode-совместимо) |
| `char*` | `String` | `{ i8*, i64 }` | marshalling: C-строка → YaoXiang `String` |
| `bool` | `Bool` | `i1` | |
| `size_t` | `Uint` | `i64` | |
| `void*` | `*Void` | `i8*` | |
| `struct T*` | `T` (прозрачный тип) | `T*` | Передаётся указатель |
| `typedef struct T T` | `T` (непрозрачный тип) | `{ i8* }` | Обёртка C-указателя |

### 4. Трансляция инструкций

Каждая инструкция IR напрямую отображается в соответствующую инструкцию LLVM IR, краткая таблица отображения:

| Инструкция IR | LLVM IR |
|---------------|---------|
| `BinaryOp { add }` | `add` |
| `BinaryOp { sub }` | `sub` |

| `Compare { eq }` | `icmp eq` / `fcmp oeq` |
| `CallStatic` | `call` |
| `CallIndirect` | `call` (через указатель на функцию) |
| `Load` | `load` |
| `Store` | `store` |

Подробная трансляция инструкций находится в реализации в `middle/passes/codegen/llvm/`.

### 5. Генерация кода для блоков spawn

Согласно [RFC-024](../accepted/024-concurrency-model.md), компиляция блоков spawn выполняется в следующие этапы.

#### 5.1 Краткое повторение семантики

```yaoxiang
(r1, r2) = spawn {
    t1 = fetch("url1"),   // прямое подвыражение → задача 1
    t2 = fetch("url2"),   // прямое подвыражение → задача 2
    return (t1, t2)       // синхронное ожидание, сборка результата
}
```

**Правила** (RFC-024 §2.1):
- **Прямые подвыражения** блока spawn (верхнеуровневые инструкции, разделённые запятыми) порождают параллельные задачи
- Выражения внутри вложенных `{}` не являются прямыми подвыражениями и не становятся отдельными задачами
- Весь блок spawn блокирует синхронно, ожидая завершения всех задач, после чего возвращает результат

#### 5.2 Шаги компиляции

```
Шаг 1: Идентификация прямых подвыражений
  Обход тела блока spawn, сбор инструкций верхнего уровня

Шаг 2: Анализ зависимостей
  Для каждого прямого подвыражения анализ, какие переменные,
  порождённые предыдущими задачами, в нём используются
  Нет зависимостей → может быть запланировано немедленно параллельно
  Есть зависимости → постановка в очередь до завершения зависимых задач

Шаг 3: Обнаружение конфликтов ресурсов (RFC-024 §2.5)
  Проверка, используется ли один и тот же экземпляр типа-ресурса
  несколькими задачами
  Конфликт по одному экземпляру → пометить порядком последовательного исполнения

Шаг 4: Генерация функций задач
  Каждое прямое подвыражение порождает отдельную функцию LLVM (замыкание)

Шаг 5: Генерация кода планирования
  Вызов task_spawn / task_wait среды выполнения

Шаг 6: Сборка результатов
  Сбор выходных данных всех задач, формирование return-кортежа
```

#### 5.3 Шаблон генерации LLVM IR

```llvm
; вход в блок spawn
%task_count = 2
%tasks = alloca [2 x %TaskHandle]

; создание задачи 1: fetch("url1")
%task1_fn = @spawn_closure_1
call @runtime_task_spawn(%tasks[0], %task1_fn, ...)

; создание задачи 2: fetch("url2")
%task2_fn = @spawn_closure_2
call @runtime_task_spawn(%tasks[1], %task2_fn, ...)

; синхронное ожидание всех задач
call @runtime_task_wait_all(%tasks, %task_count)

; сборка возвращаемого значения
%r1 = call @runtime_task_result(%tasks[0])
%r2 = call @runtime_task_result(%tasks[1])
ret { %r1, %r2 }
```

#### 5.4 Зависимые задачи

```yaoxiang
result = spawn {
    data = fetch("url"),       // задача 1: нет зависимостей
    processed = parse(data),   // задача 2: зависит от data из задачи 1
    return processed
}
```

Компилятор обнаруживает, что `parse(data)` ссылается на `data`, порождённое задачей 1, и при генерации кода планирования помечает зависимость:

```llvm
; задача 2 создаётся с зависимостью от задачи 1
call @runtime_task_spawn_with_dep(%tasks[1], %task2_fn, %tasks[0])
;                                                              ↑
;                                            зависит от задачи 0 (fetch)
```

#### 5.5 Автоматическая сериализация по типам-ресурсам

Типы-ресурсы, определённые в [RFC-024 §2.5](../accepted/024-concurrency-model.md) (`FilePath`, `HttpUrl`, `DBUrl`, `Console` и пользовательские типы-ресурсы), автоматически сериализуются внутри блоков spawn:

```yaoxiang
(a, b) = spawn {
    r1 = db.exec("SELECT ..."),   // использует SqliteDb (тип-ресурс)
    r2 = db.exec("INSERT ...")    // тот же экземпляр → автоматическая сериализация
}
```

Компилятор обнаруживает, что один и тот же экземпляр ресурса используется двумя задачами, и генерирует последовательную зависимость:

```llvm
; задача 2 зависит от задачи 1 (автосериализация по тому же ресурсу)
call @runtime_task_spawn_with_dep(%tasks[1], %task2_fn, %tasks[0])
```

#### 5.6 spawn for — параллелизм по данным

```yaoxiang
results = spawn for item in items {
    process(item)
}
```

Компилятор разворачивает это в N независимых задач (N = длина items), с ограничением максимального параллелизма.

### 6. Генерация кода FFI

Согласно [RFC-026](./026-ffi-core-mechanism.md), в данном разделе определяется стратегия генерации LLVM IR для FFI-вызовов.

#### 6.1 Объявление функции native()

```yaoxiang
sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")
```

Компилируется в LLVM IR:

```llvm
; объявление внешней C-функции
declare i8* @sqlite3_open(i8*)

; обёртка YaoXiang (выполняет marshalling)
define { i8* } @__yx_sqlite3_open({ i8*, i64 } %filename) {
    ; marshalling: YaoXiang String → C-строка
    %c_str = extractvalue { i8*, i64 } %filename, 0
    ; вызов C-функции
    %raw = call i8* @sqlite3_open(i8* %c_str)
    ; unmarshalling: C-указатель → непрозрачный тип
    %result = insertvalue { i8* } undef, i8* %raw, 0
    ret { i8* } %result
}
```

**Ключевые моменты**:
- `native("sqlite3_open")` → `declare external @sqlite3_open`
- Компилятор автоматически генерирует обёртку с marshalling
- Сигнатура обёртки использует типы YaoXiang, внутри происходит преобразование к C-типам

#### 6.2 Marshalling аргументов

| Направление | Преобразование |
|-------------|----------------|
| YaoXiang `String` → C `char*` | Извлечение поля `.ptr` и передача |
| YaoXiang `Int32` → C `int` | Прямая передача (`i32`) |
| YaoXiang `*Void` → C `void*` | Прямая передача (`i8*`) |
| YaoXiang `T` (прозрачный тип) → C `struct T*` | Передача адреса |
| YaoXiang `T` (непрозрачный тип) → C `struct T*` | Извлечение указателя из `{ i8* }` и передача |

#### 6.3 LLVM-раскладка непрозрачных типов

Непрозрачные типы, определённые в [RFC-026](./026-ffi-core-mechanism.md) §4.1:

```yaoxiang
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void
    }
    return SqliteDb
}
```

LLVM-раскладка: `{ i8* }` — структура с одним C-указателем.

**Оптимизация раскладки**: когда непрозрачный тип имеет единственное поле `handle: *Void`, возможна оптимизация до прямого использования `i8*` (без внешней struct). Оптимизированный ABI полностью совпадает с C-указателем — нулевые накладные расходы на marshalling. Оптимизация включена по умолчанию и невидима для пользователя.

#### 6.4 LLVM-представление nullable-возвращаемого значения ?T

Nullable-возвращаемые значения FFI, определённые в [RFC-026](./026-ffi-core-mechanism.md) §7.6:

```yaoxiang
sqlite3_open: (filename: String) -> ?SqliteDb = native("sqlite3_open")
```

Универсальное LLVM-представление: `{ i1, { i8* } }` — флаг наличия значения + данные.

**Оптимизация для null-указателя FFI**: если `T` в `?T` — непрозрачный тип (внутри указатель), компилятор применяет оптимизацию **null-указатель = None**:

```llvm
; оптимизированное LLVM-представление: напрямую используется nullable-указатель
define i8* @__yx_sqlite3_open(...) {
    %raw = call i8* @sqlite3_open(...)
    ; null → None, не-null → Some (обёрнутый в непрозрачный тип)
    ret i8* %raw
}
```

На стороне вызывающего:
```llvm
%raw = call i8* @__yx_sqlite3_open(...)
%is_null = icmp eq i8* %raw, null
br i1 %is_null, label %none_branch, label %some_branch
```

Эта оптимизация делает FFI-вызовы `?SqliteDb` **без дополнительных накладных расходов** — эквивалентно проверке null в C.

#### 6.5 Интеграция с yx-bindgen

Файлы привязок, автоматически сгенерированные [yx-bindgen](./026-ffi-core-mechanism.md) §6, обрабатываются при компиляции как обычный исходный код YaoXiang. Компилятору не требуется знать, что код получен из bindgen — обработка объявлений `native()` и определений типов в `unsafe {}` полностью идентична.

### 7. Генерация кода деструкторов

Согласно RAII-семантике из [RFC-009](../accepted/009-ownership-model.md) и соглашению `.drop` из [RFC-026](./026-ffi-core-mechanism.md) §7.

#### 7.1 Распознавание привязки .drop

```yaoxiang
SqliteDb.drop = sqlite3_close[0]
```

Компилятор распознаёт привязку `.drop` и помечает указатель на деструктор в метаданных типа.

#### 7.2 Вставка cleanup в конце области видимости

```
Код пользователя:
{
    db = SqliteDb.open("test.db")
    stmt = db.prepare("SELECT ...")
    stmt.step()
    // ← конец области видимости
}

Вставленный компилятором cleanup (в обратном порядке):
    call @sqlite3_finalize(%stmt)    // stmt.drop()
    call @sqlite3_close(%db)         // db.drop()
```

**Места вставки**:
- Нормальное завершение области видимости (`}`)
- Досрочный возврат (перед `return`)
- Путь распространения ошибки `?` (перед `?`)
- Завершение блока spawn (деструкторы переменных внутри задачи)

#### 7.3 Move и деструктор

```yaoxiang
db = SqliteDb.open("test.db")
db2 = db                // Move: владение передаётся db2
// db недействителен, для db drop не вставляется
// ← конец области видимости: drop вставляется только для db2
```

Компилятор отслеживает семантику Move ([RFC-009](../accepted/009-ownership-model.md) §1) и вставляет вызов деструктора только у конечного держателя переменной.

#### 7.4 Обработка ошибок деструктора

```llvm
; режим debug: проверка возвращаемого значения деструктора
%ret = call i32 @sqlite3_close(i8* %handle)
%ok = icmp eq i32 %ret, 0
br i1 %ok, label %done, label %panic
panic:
  call @__yx_panic("destructor failed")
  unreachable
done:
  ret void

; режим release: возврат игнорируется
call i32 @sqlite3_close(i8* %handle)
ret void
```

### 9. Библиотека среды выполнения

Согласно [RFC-008 §6.2](../accepted/008-runtime-concurrency-model.md), среда выполнения линкуется в финальный exe в виде **статической библиотеки**.

```
Внутренняя структура финального exe:

┌────────────────────────────────────────────┐
│  Пользовательский код (нативный машинный)  │
│  ├── Обычные функции (последовательные)     │
│  ├── Развёртка блоков spawn                │
│  │   (функции задач + вызовы планировщика)  │
│  ├── Обёртки marshalling для FFI           │
│  └── RAII-код деструкторов                 │
├────────────────────────────────────────────┤
│  Статическая библиотека среды выполнения    │
│  (~200-500KB)                              │
│  ├── Пул потоков (num_workers)             │
│  ├── Цикл событий (libuv / io_uring)       │
│  ├── Work-stealing очереди                 │
│  │   (только Full Runtime)                 │
│  ├── Аллокатор памяти (jemalloc / mimalloc)│
│  └── Метаданные рефлексии (секция .reflect,│
│      подгружаются по требованию через mmap) │
│                                              │
│  Отсутствует:                              │
│  ❌ Интерпретатор байткода                  │
│  ❌ JIT-компилятор                          │
│  ❌ GC                                       │
│  ❌ Виртуальная машина                       │
└────────────────────────────────────────────┘
```

**Ключевой замысел**: распознавание задач и анализ зависимостей блоков spawn выполняются на этапе компиляции; среда выполнения лишь «создаёт задачу → распределяет в пул потоков → ожидает завершения» — структуры данных фиксированы, поведение предсказуемо.

**Связь трёх уровней среды выполнения с LLVM** (согласно RFC-008 §1):

| Среда выполнения | Поведение LLVM AOT |
|------------------|---------------------|
| **Embedded** | Без поддержки spawn, напрямую генерируется последовательный машинный код |
| **Standard** | Поддержка блоков spawn, DAG внутри блока spawn + однопоточное планирование (num_workers=1) |
| **Full** | Поддержка блоков spawn, DAG внутри блока spawn + многопоточное планирование (num_workers>1), поддержка WorkStealing |

## Компромиссы

### Преимущества

1. **Производительность**: AOT-компиляция в 10-100x быстрее интерпретации
2. **Единый фронтенд**: VM и LLVM используют общий фронтенд, поведение полностью идентично
3. **Нулевые накладные расходы на планирование**: обычный код напрямую компилируется в последовательный машинный код, за пределами блоков spawn нет DAG-накладных расходов
4. **Статическая линковка**: нет внешних зависимостей от среды выполнения, для развёртывания достаточно одного exe
5. **Нулевой GC**: детерминированные деструкторы RAII, без пауз
6. **Нулевые накладные расходы FFI**: оптимизация `?T` с null-указателем, оптимизация раскладки непрозрачных типов — стоимость FFI-вызовов эквивалентна C
7. **Анализ на этапе компиляции**: распознавание задач и анализ зависимостей блоков spawn выполняются на этапе компиляции, среда выполнения только исполняет

### Недостатки

1. **Сложность интеграции с LLVM**: требуется глубокое понимание API inkwell и LLVM IR
2. **Время компиляции**: AOT-компиляция медленнее интерпретатора (разовая цена)
3. **Опыт отладки**: отладка нативного кода требует поддержки символов DWARF/PDB (компилятор должен генерировать отладочную информацию)
4. **Инкрементальная компиляция**: для крупных проектов требуется дополнительное проектирование инкрементальной компиляции

### Согласованность с родственными RFC

| RFC | Согласованность |
|-----|-----------------|
| RFC-024 модель параллелизма на блоках spawn | ✅ Прямые подвыражения блока spawn → распределение задач |
| RFC-008 архитектура среды выполнения | ✅ Двойной бэкенд + статическая библиотека планировщика + структура каталогов модулей |
| RFC-009 v9 модель владения | ✅ Токены `&T`/`&mut T` (нулевого размера), `ref T` (толстый указатель), `?T` (Option) |
| RFC-026 основной механизм FFI | ✅ `native()` → declare + marshalling, `.drop` → RAII cleanup |

## Родственные работы

### Lazy Task Creation (1990)[^1]

| Атрибут | Описание |
|---------|----------|
| Организация | MIT |
| Авторы | James R. Larus, Robert H. Halstead Jr. |
| Ядро | Отложенное создание подзадач, создание по требованию |
| Ценность для нас | Теоретический фундамент диспетчеризации задач по требованию внутри блоков spawn |

**Основная идея**: задачи не создаются немедленно, а отложенно. Когда родительской задаче требуется значение дочерней, дочерняя задача создаётся в этот момент. Это решает проблему накладных расходов на мелкозернистый параллелизм[^1]. Планирование блоков spawn в YaoXiang заимствует этот подход — задачи распознаются на этапе компиляции, но на этапе выполнения распределяются в пул потоков по требованию.

### Lazy Scheduling (2014)[^2]

| Атрибут | Описание |
|---------|----------|
| Организация | University of Maryland |
| Авторы | Tzannes, Caragea |
| Ядро | Адаптивное планирование на этапе выполнения, без дополнительного состояния |
| Ценность для нас | Референс для проектирования планировщика WorkStealing в Full Runtime |

### Язык SISAL[^3]

| Атрибут | Описание |
|---------|----------|
| Организация | Lawrence Livermore National Laboratory (LLNL) |
| Ядро | Язык с единым присваиванием, dataflow-граф, неявный параллелизм |
| Ценность для нас | Доказательство осуществимости модели dataflow в индустриальных приложениях |

**Ключевое отличие**: в SISAL параллелизм **неявный** — язык использует одноприсваиваемую семантику, и компилятор автоматически строит граф зависимостей по всей программе для определения параллелизма. В YaoXiang параллелизм **явный** — пользователь отмечает параллельные регионы блоками `spawn {}`, а компилятор анализирует зависимости только внутри блока spawn. Это позволяет избежать сложности полнопрограммного анализа SISAL, сохраняя контроль пользователя над параллельным поведением.

### Параллельный Scheme Mul-T[^4]

| Атрибут | Описание |
|---------|----------|
| Организация | MIT |
| Ядро | Конструкция future, реализация Lazy Task Creation |
| Ценность для нас | Референс конкретной реализации |

### Сравнительная сводка

| Технология | Отложенное создание | Маркер параллелизма | Область анализа | Владение |
|------------|---------------------|---------------------|-----------------|----------|
| Lazy Task Creation[^1] | ✅ | Неявный | Вся программа | N/A |
| Lazy Scheduling[^2] | ✅ | Неявный | Вся программа | N/A |
| SISAL[^3] | ✅ | Неявный (single assignment) | Вся программа | N/A |
| Mul-T[^4] | ✅ | Явный (future) | Точка вызова | N/A |
| **YaoXiang** | ✅ | **Явный (блок spawn)** | **Внутри блока spawn** | **✅ (Move + токены + ref)** |

**Инновация YaoXiang**: подъём маркера параллелизма с уровня «каждый вызов функции» (future) на уровень «структурированного блока» (spawn). Пользователь пишет обычный код, а в местах, где нужен параллелизм, помещает блок spawn. Область анализа ограничена блоком spawn — компиляция эффективна, а поведение предсказуемо.

### Приложение А: Сравнение с Rust async

| Возможность | Rust async | YaoXiang LLVM AOT |
|-------------|-----------|-------------------|
| Артефакт компиляции | Конечный автомат + машинный код | Машинный код + метаданные задач spawn |
| Среда выполнения | tokio | Статически линкуемый планировщик (~200-500KB) |
| Маркер параллелизма | Ключевые слова async/await | Блок `spawn { }` |
| Создание задач | Генерация конечного автомата на этапе компиляции | Распознавание прямых подвыражений на этапе компиляции → функции задач |
| Цветные функции | Заражение async | **Без окрашивания функций** |
| Синхронное ожидание | `.await` | Блок spawn автоматически блокирует синхронно |
| Управление памятью | GC (среда выполнения) | **RAII (детерминированное)** |
| Механизм разделения владения | `Arc::new()` + ручной Weak | **`ref` ключевое слово (компилятор автоматически выбирает Rc/Arc)** |
