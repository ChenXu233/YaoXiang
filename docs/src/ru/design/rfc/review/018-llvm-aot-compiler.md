## Краткое содержание

В данном документе проектируется LLVM AOT-компилятор для языка YaoXiang, цель которого — генерация машинного кода + DAG-метаданных через предварительную компиляцию, с планированием DAG в **spawn-блоках** во время выполнения на основе **восходящего** анализа зависимостей.

**Ключевые инновации**:
- Не "при встрече с вызовом функции генерировать Future", а **обратный анализ зависимостей от "места, где нужен результат"**
- **Листовые узлы выполняются параллельно в первую очередь**, цепочки зависимостей обходятся последовательно
- **Изолированные DAG выполняются независимо и параллельно**: узлы без потребителей не блокируют основной процесс
- **Бесконечные циклы как фоновые DAG**: планировщик выполняет их частями, не зависая
- **Анализ DAG ограничен spawn-блоками**: эффективная компиляция, предсказуемое поведение

Данный подход принципиально отличается от модели Rust async/await + tokio runtime:
- Rust: пользователь пишет `async fn`, компилятор генерирует конечный автомат
- YaoXiang: пользователь пишет обычные функции, **компилятор автоматически анализирует DAG в spawn-блоках**, планировщик выполняет восходяще

Соответствует модели параллелизма на основе spawn-блоков из RFC-024: обычный код выполняется последовательно, внутри spawn-блоков — параллельное планирование DAG.

## Мотивация

### Зачем нужен LLVM AOT-компилятор?

В настоящее время YaoXiang имеет только интерпретатор в качестве серверной части выполнения, что создаёт следующие проблемы:

| Проблема | Влияние |
|------|------|
| Узкое место производительности | Интерпретируемое выполнение в 10-100 раз медленнее машинного кода |
| Сложность развёртывания | Требуется переносить интерпретатор и runtime |
| Проблема цветных функций | Синхронные функции не могут вызывать параллельные функции |

### Проблема цветных функций и параллелизм spawn-блоков

**Традиционное проектирование**:
- Синхронные функции (синие) → не могут вызывать → параллельные функции (красные)
- Синхронность — по умолчанию, параллельность требует пометки `spawn`
- Цвета "заразны": однажды использовав параллельность, вся цепочка вызовов становится параллельной

**Параллелизм spawn-блоков по RFC-024 (цель)**:
- Обычный код выполняется последовательно, без раскраски функций
- Внутри `spawn { ... }` — анализ DAG, параллельное выполнение
- Вызывающая сторона блокируется синхронно, без колбэков и `await`

**Инвертированное проектирование (RFC-018)**:
- Обычный код напрямую генерирует последовательный машинный код без накладных расходов DAG
- Внутри spawn-блоков компилятор автоматически анализирует зависимости DAG, runtime выполняет восходяще
- Решает проблему цветных функций: все функции унифицированы, без разделения на синхронные/параллельные

### Ключевая инновация: восходящее выполнение + DAG внутри spawn-блоков

Ключевая инновация данного проектирования — **модель восходящего выполнения** (ограниченная внутри spawn-блоков):

```
Традиционный вызов (сверху вниз):
  call fetch(url) → выполнение → возврат результата

Восходящее выполнение:
  print(a) ← начинаем с "места, где нужен результат"
       ↑
  fetch(url0) ← анализируем зависимости, ищем в обратном направлении

  fetch(url1) ← изолированный, независимое параллельное выполнение
```

**Ключевые различия**:
- Не "при встрече с вызовом функции генерировать Future"
- Анализ зависимостей в обратном направлении от "конечного результата"
- Узлы без потребителей (изолированные) не выполняются или выполняются независимо
- Бесконечные циклы как фоновые DAG, планировщик выполняет их частями

### DAG-планировщик внутри spawn-блоков

```
Представление DAG внутри spawn-блока:

        print(result) ─────────────────────────┐
           │                                    │
    ┌──────┴──────┐                             │
    │             │                             │
process(a)   process(b)                        │
    │             │                             │
compute(x)   compute(y)  ←── изолированный DAG ─┤
    │                                           │
fetch(url0)  fetch(url1)  fetch(url2)          │
    (выполнен)                                  │

Параллельно существует фоновый DAG (while True):
    ┌─────────────────────────────────────────┐ │
    │  while True:                            │ │
    │      update_ui()                        │ │
    │      fetch_new() ──→ process(data)      │ │
    └─────────────────────────────────────────┘ │
```

**Принцип работы планировщика**:
```
1. Обратный анализ от "конечного результата":
   print(result) → зависит от process → зависит от fetch

2. Построение DAG внутри spawn-блока:
   - Листовые узлы: fetch (без зависимостей)
   - Внутренние узлы: process, compute
   - Корневой узел: print

3. Выполнение:
   - fetch выполняются параллельно
   - process ожидает завершения fetch
   - print ожидает завершения process
   - изолированный compute выполняется независимо

4. Пропуск уже выполненных:
   - Если узел уже выполнен, зависимые от него узлы могут
     использовать результат повторно
```

### Обработка бесконечных циклов

```
Сценарий 1: единственный while/for (без накладных расходов планирования)
──────────────────────────────────────────────
main: () -> () = {
    while True {
        update_ui()
        fetch_data()
    }
}
→ только один бесконечный цикл
→ прямое синхронное выполнение, никакой разницы с обычным кодом

Сценарий 2: несколько while (автоматическое разбиение на срезы)
──────────────────────────────────────────────
main: () -> () = {
    while True { update_ui() }      # фоновая задача 1
    while True { network_poll() }  # фоновая задача 2
    server_loop()                   # основная задача
}
→ 3 независимые задачи
→ планировщик переключается между срезами
→ настоящий параллелизм

Адаптивный планировщик:
──────────────────────────────────────────────
if количество_задач == 1:
    прямое выполнение (синхронно)
else:
    планирование срезами (параллельно)
```

**Обработка фоновых DAG**:
```
Основной DAG (имеет завершение):
    fetch → process → print → завершение

Фоновый DAG (бесконечный цикл):
    while True → update_ui → fetch_new → process → возврат в начало

Планировщик:
    - основной DAG завершается после выполнения
    - фоновый DAG работает вечно, но планировщик выполняет его
      "срезами"
    - не зависает в цикле
```

## Предложение

### Основное проектирование

```
┌─────────────────────────────────────────────────────┐
│  Компиляция                                         │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐           │
│  │ Parser  │→│Анализ   │→│LLVM     │→ Машинный │
│  │         │ │DAG      │→│Codegen  │   код     │
│  └─────────┘  └─────────┘  └─────────┘           │
│                      ↓                           │
│              Генерация: DAG-метаданные            │
└─────────────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────┐
│  Runtime                                            │
│  ┌─────────────────────────────────────────────┐ │
│  │  Библиотека DAG-планировщика                │ │
│  │  • Загрузка машинного кода                  │ │
│  │  • Чтение DAG-метаданных                    │ │
│  │  • Ленивое планирование: приостановка       │ │
│  │    вызовов, выполнение по требованию        │ │
│  │  • Поддержка параллельного/последовательного │
│  │    выполнения                               │ │
│  └─────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### Процесс восходящего выполнения

```
Пользовательский код:
    main: () -> () = {
        pages = urls.map(|url| fetch(url))
        results = pages.map(|page| parse_page(page))
        save_results(results)
    }

Фаза 1: Восходящий анализ (компиляция)
─────────────────────────────────────────
Начиная с save_results(results):
    "нужен results" → зависит от parse_page(results)
    "нужен page0" → зависит от fetch(url0)
    "нужен page1" → зависит от fetch(url1)
    ...

Построение DAG внутри spawn-блока:
    fetch(url0), fetch(url1), fetch(url2) ← листовые узлы
           ↓
    parse_page(page0), parse_page(page1)   ← зависят от листовых
           ↓
    save_results                          ← корневой узел

Фаза 2: Параллельное выполнение листовых (runtime)
─────────────────────────────────────────
Планировщик находит все листовые узлы:
    - fetch(url0), fetch(url1), fetch(url2) без зависимостей → параллельно
    - контроль параллелизма (например, 16 одновременных)

Фаза 3: Обход вверх
─────────────────────────────────────────
Когда parse_page нужен page0:
    - проверяем готовность page0
    - готов → выполняем parse_page
    - не готов → ждём, затем продолжаем

Фаза 4: Изолированные выполняются независимо и параллельно
─────────────────────────────────────────
Если результат какого-то fetch никому не нужен:
    - выполняется как "изолированный DAG"
    - может использовать другое ядро, не влияет на основной процесс
```

### Структура результатов компиляции

```rust
/// Результат компиляции: машинный код + DAG-метаданные
pub struct CompiledArtifact {
    /// Машинный код, скомпилированный LLVM (ELF/Mach-O/COFF)
    machine_code: Vec<u8>,

    /// DAG-метаданные: описание зависимостей функций
    dag: DAGMetadata,

    /// Таблица точек входа
    entries: Vec<EntryPoint>,

    /// Информация о типах (для FFI)
    type_info: TypeInfo,
}

/// DAG-метаданные
pub struct DAGMetadata {
    /// Узлы: вызовы функций
    nodes: Vec<DAGNode>,
    /// Рёбра: зависимости (from, to)
    edges: Vec<(usize, usize)>,
}

/// Одиночный DAG-узел
pub struct DAGNode {
    /// ID функции
    pub function_id: usize,
    /// Зависимые узлы ID
    pub deps: Vec<usize>,
    /// Метка побочного эффекта (@IO / @Pure)
    pub effect: EffectTag,
}
```

### Интерфейс runtime-планировщика

```rust
/// Trait DAG-планировщика
pub trait DAGScheduler: Send + Sync {
    /// Плановое выполнение
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue;

    /// Выполнение одиночной функции
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}

/// Реализация планировщика
pub struct DefaultDAGScheduler {
    /// Пул потоков
    thread_pool: ThreadPool,
    /// Результат компиляции
    artifact: CompiledArtifact,
    /// Максимальный параллелизм
    max_parallelism: usize,
}

impl DefaultDAGScheduler {
    pub fn new(artifact: CompiledArtifact, num_workers: usize) -> Self {
        Self {
            thread_pool: ThreadPool::new(num_workers),
            artifact,
            max_parallelism: num_workers * 2, // адаптивный контроль зернистости
        }
    }
}

impl DAGScheduler for DefaultDAGScheduler {
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue {
        // 1. Обход тела функции, приостановка всех вызовов
        // 2. Построение списка задач для выполнения
        // 3. Планирование выполнения по порядку зависимостей (контроль параллелизма)
        // 4. Триггер выполнения при запросе значения
        // 5. Возврат результата
    }
}
```

### Стратегии планирования

| Сценарий | Стратегия планирования |
|------|----------|
| Обычный код (не spawn-блок) | Последовательное выполнение, без накладных расходов DAG |
| Внутри spawn-блока | DAG с ленивым планированием, выполнение без зависимостей параллельно |
| Циклическая зависимость | Обнаружение во время выполнения, ошибка |

### Обработка побочных эффектов: неявная Effect System

Пользователь не замечает обработку побочных эффектов, компилятор выводит автоматически:

```
Пользовательский код:
  print("a")
  print("b")
  x = compute(1)
  y = compute(2)

Вывод компилятора:
  print → @IO (внешний вызов)
  compute → @Pure (чистая функция)

Выполнение планировщиком:
  print("a") ──→ последовательно (все @IO)
  print("b") ──→ последовательно
  compute(1) ─┬─→ параллельно (DAG-планирование внутри spawn-блока)
  compute(2) ─┘
```

### Связь с трёхуровневым runtime

RFC-008 определяет архитектуру Embedded / Standard / Full runtime. Соответствие LLVM AOT-компилятора трём уровням runtime (выравнивание с RFC-024):

| Runtime | Поведение LLVM AOT |
|--------|---------------|
| **Embedded** | Без поддержки spawn, прямая генерация последовательного машинного кода |
| **Standard** | Поддержка spawn-блоков, DAG внутри spawn-блоков + однопоточное планирование (num_workers=1) |
| **Full** | Поддержка spawn-блоков, DAG внутри spawn-блоков + многопоточное планирование (num_workers>1), поддержка WorkStealing |

### Проектирование интерфейса планировщика

```rust
/// Метка побочного эффекта
pub enum EffectTag {
    /// Чистая функция, без побочных эффектов
    Pure,
    /// Имеет побочные эффекты I/O
    IO,
}

/// Trait DAG-планировщика
pub trait DAGScheduler: Send + Sync {
    /// Плановое выполнение DAG внутри spawn-блока
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue;

    /// Выполнение одиночной функции (обычный код, последовательное выполнение)
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}
```

## Компромиссы

### Преимущества

1. **Повышение производительности**: AOT-компиляция в 10-100 раз быстрее интерпретируемого выполнения
2. **Решение проблемы цветных функций**: без раскраски функций, параллелизм внутри spawn-блоков
3. **Унифицированный runtime**: интерпретатор и LLVM используют один планировщик
5. **Неявные побочные эффекты**: пользователь не замечает, компилятор обрабатывает автоматически
6. **Безопасность владения**: зависит от модели владения в стиле Rust, без гонок данных

### Недостатки

1. **Сложность реализации**: требуется опыт интеграции LLVM
2. **Время компиляции**: AOT-компиляция медленнее интерпретатора
3. **Сложность отладки**: отладка AOT-кода сложнее, чем интерпретатора

### Согласованность с дизайном RFC

| RFC | Согласованность |
|-----|--------|
| RFC-024 модель параллелизма на основе spawn-блоков | ✅ Анализ DAG ограничен spawn-блоками |
| RFC-008 архитектура runtime | ✅ Дизайн runtime-планировщика согласован |
| RFC-009 модель владения | ✅ ARC runtime корректно реализован |

## Альтернативные решения

| Решение | Описание | Почему не выбрано |
|------|------|-----------|
| Только интерпретатор | AOT не нужен | Недостаточная производительность, нет поддержки параллелизма spawn-блоков |
| Чистая статическая компиляция | Без runtime-планирования | Ленивое планирование требуется во время выполнения |
| Связывание с внешним LLVM runtime | Использовать LLVM runtime | Требуются дополнительные зависимости |

## Связанные работы

### Lazy Task Creation (1990)[^1]

| Атрибут | Описание |
|------|------|
| Организация | MIT |
| Автор | James R. Larus, Robert H. Halstead Jr. |
| Суть | Ленивое создание подзадач, создание по требованию |
| Ценность | Техническая основа, истоки концепции ленивого планирования |

**Ключевая идея**: задачи создаются не сразу, а отложенно. Когда родительской задаче нужен результат дочерней, тогда и создаётся дочерняя задача. Это решает проблему накладных расходов на задачи с мелкой зернистостью[^1].

### Lazy Scheduling (2014)[^2]

| Атрибут | Описание |
|------|------|
| Организация | University of Maryland |
| Автор | Tzannes, Caragea |
| Суть | Адаптивное планирование во время выполнения, без дополнительного состояния |
| Ценность | Дизайн планировщика, адаптивный контроль зернистости |

**Ключевая идея**: через "ленивое выполнение" автоматически контролируется зернистость, не нужно поддерживать сложное состояние. Когда система занята, задачи автоматически объединяются, когда свободна — разделяются[^2].

### Язык SISAL[^3]

| Атрибут | Описание |
|------|------|
| Организация | Lawrence Livermore National Laboratory (LLNL) |
| Суть | Язык с одной присваиванием, Dataflow-графы, неявный параллелизм |
| Ценность | Доказательство осуществимости, производительность близка к Fortran |

**Ключевой вклад**: SISAL доказал, что модель Dataflow в промышленных приложениях может достигать производительности, близкой к Fortran[^3].

### Параллельная схема Mul-T[^4]

| Атрибут | Описание |
|------|------|
| Организация | MIT |
| Суть | Конструкция Future, реализация Lazy Task Creation |
| Ценность | Ссылка на конкретную реализацию |

**Ключевой механизм**:
```scheme
;; Синтаксис Multilisp / Mul-T
(let ((a (future compute-a))      ;; немедленный возврат future
      (b (future compute-b)))      ;; немедленный возврат future
  (join a b))                      ;; ожидание завершения
```

### Сравнительная сводка

| Технология | Ленивое создание | Анализ DAG | Обработка побочных эффектов | Владение |
|------|----------|----------|------------|--------|
| Lazy Task Creation[^1] | ✅ | ❌ | ❌ | N/A |
| Lazy Scheduling[^2] | ✅ | ❌ | ❌ | N/A |
| SISAL[^3] | ✅ | ✅ (глобальный) | N/A (одна присваивание) | N/A |
| Mul-T[^4] | ✅ | ❌ | ❌ | N/A |
| **YaoXiang** | ✅ | ✅ (внутри spawn-блоков) | ✅ (неявный) | ✅ (ARC) |

**Инновация YaoXiang**: использование современных языковых возможностей (владение + неявные побочные эффекты) упрощает традиционный дизайн, ограничение DAG spawn-блоками снижает сложность.

## Сравнение с традиционными методами автоматического распараллеливания

### Традиционные компиляторы: параллелизация на уровне циклов

Коммерческие компиляторы (Intel Fortran, Oracle Fortran) используют **автоматическую параллелизацию на уровне циклов**[^5]:

**Основной процесс**:
```
1. Определение параллелизуемых циклов
2. Анализ зависимостей для обращений к массивам внутри циклов
3. Определение наличия зависимостей между итерациями цикла
4. При отсутствии зависимостей — генерация многопоточного кода
```

**Технологии анализа зависимостей**:

| Технология | Описание |
|------|------|
| **Зависимость данных** | Два обращения обращаются к одной области памяти |
| **Use-Def** | Отношение определения и использования переменных |
| **Анализ псевдонимов** | Указывают ли указатели на одну область памяти |

**Условия параллелизуемости циклов**:
```fortran
! Можно параллелизовать
DO I = 1, N
  A(I) = C(I)
END DO

! Нельзя параллелизовать (зависимость от предыдущей итерации)
DO I = 2, N
  A(I) = A(I-1) + B(I)
END DO
```

### Haskell: механизм Spark

GHC (Glasgow Haskell Compiler) использует **механизм Spark** для реализации параллелизма чистых функций[^6]:

```haskell
-- rpar: параллельное выполнение, создание spark
-- rseq: последовательное выполнение, ожидание завершения

example = do
  a <- rpar (f x)   -- создание spark, параллельное выполнение f x
  b <- rpar (g y)   -- создание spark, параллельное выполнение g y
  rseq a            -- ожидание завершения a
  rseq b            -- ожидание завершения b
  return (a, b)
```

**Механизм пула Spark**:
- Spark берётся из пула и分配ляется свободным процессорным ядрам
- Если spark не используется (никто не ждёт результата), он удаляется GC
- Это решает проблему зернистости: слишком мелкие spark отбрасываются

### Язык Clean: типы уникальности

Язык Clean реализует безопасность параллелизма через **типы уникальности (Uniqueness Types)**[^7]:

```clean
-- *Array означает уникальность, можно безопасно изменять
modify :: *Array Int -> *Array Int
```

**Ключевая идея**: если значение имеет единственную ссылку, его можно безопасно изменять в параллельной среде, так как другие ссылки не увидят промежуточное состояние.

### Программные срезы и графы зависимостей

**Граф программных зависимостей (PDG)** — основа обнаружения параллелизма:

```
Узлы: операторы
Рёбра: зависимость по данным + управляющая зависимость

Обнаружение параллелизма:
  Если между двумя узлами нет достижимого пути → можно выполнять параллельно
```

### Комплексное сравнение

| Метод | Анализ зависимостей | Зернистость | Обработка побочных эффектов | Типичный сценарий |
|------|----------|------|------------|----------|
| Intel/Oracle Fortran[^5] | Сложный анализ массивов | Итерации цикла | N/A | Научные вычисления |
| GHC Spark[^6] | Предположение о чистых функциях | Выражения | N/A | Функциональное программирование |
| Clean[^7] | Типы уникальности | Переписывание графов | N/A | Функциональное программирование |
| **YaoXiang** | Гарантия владения | Вызовы функций | Неявный вывод | Общего назначения |

### Приложение A: Подробное сравнение с Rust async

| Характеристика | Rust async | YaoXiang LLVM AOT |
|------|-----------|-------------------|

| Runtime | tokio | DAG Scheduler |

| Цветные функции | Заражение async | **Без раскраски функций, параллелизм внутри spawn-блоков** |
| Аннотации | async/await | Нет (spawn-блок — единственный примитив параллелизма) |

### Приложение B: Примеры оптимизации планировщика

**Сценарий 1: Планировщик обнаружил возможность объединения**

```
Исходный DAG:
  compute_a() ──┐
  compute_b() ──┼──→ compute_c()

После оптимизации планировщика:
  compute_a + compute_b объединены в одну задачу
  → снижение накладных расходов планирования
```

**Сценарий 2: Зависимость не используется**

```
a = expensive_compute()  // вычислено
b = other_thing()        // a не нужен
print(b)                 // напрямую возвращаем b, пропускаем a
```
