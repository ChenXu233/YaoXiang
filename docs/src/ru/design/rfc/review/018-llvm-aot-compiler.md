# RFC-018：LLVM AOT компилятор и проект L3 прозрачного параллелизма

> **Статус**: Черновик
> **Автор**: Чэнь Сюй

> **Последнее обновление**: 2026-03-10

## Аннотация

В данном документе представлен дизайн LLVM AOT компилятора для языка YaoXiang, цель которого — через предварительную компиляцию генерировать машинный код + DAG метаданные, а runtime **глобальный DAG планировщик** выполняет на основе **bottom-up** анализа зависимостей.

**Ключевые инновации**:
- Не "при встрече с вызовом функции генерировать Future", а **от "места, где нужен результат" обратный анализ зависимостей**
- **Листовые узлы выполняются параллельно в первую очередь**, цепочка зависимостей обходится снизу вверх
- **Изолированные DAG выполняются независимо параллельно**: узлы без потребителей не блокируют основной поток
- **Бесконечные циклы как фоновые DAG**: планировщик выполняет их по частям, не зависая

Данный дизайн принципиально отличается от модели Rust async/await + tokio runtime:
- Rust: пользователь пишет `async fn`, компилятор генерирует конечный автомат
- YaoXiang: пользователь пишет обычные функции, **компилятор автоматически анализирует DAG**, планировщик выполняет снизу вверх

Соответствует L3 прозрачному 并发 дизайну RFC-001: по умолчанию @auto (автоматический параллелизм), @block синхронный — это частный случай, решающий проблему цветных функций.

## Мотивация

### Зачем нужен LLVM AOT компилятор?

В настоящее время YaoXiang имеет только интерпретатор как execution backend, что создаёт следующие проблемы:

| Проблема | Влияние |
|----------|---------|
| Узкое место производительности | Интерпретируемое выполнение медленнее машинного кода в 10-100x |
| Сложность развёртывания | Необходимо поставлять интерпретатор и runtime |
| Проблема цветных функций | Синхронные функции не могут вызывать 并发 функции |

### Проблема цветных функций и L3 прозрачный 并发

**Традиционный дизайн (текущий)**:
- Синхронные функции (синие) → не могут вызывать → 并发 функции (красные)
- Синхронный режим по умолчанию, 并发 требует отметки `spawn`
- Цвет "заразен": если используешь 并发, вся цепочка вызовов становится 并发

**RFC-001 L3 прозрачный 并发 (цель)**:
- L3: прозрачный 并发 по умолчанию (@auto)
- L2: явный spawn 并发
- L1: @block синхронный режим

**Инвертированный дизайн (RFC-018)**:
- L3 прозрачный 并发 по умолчанию, автоматический анализ DAG зависимостей при компиляции
- Решает проблему цветных функций: синхронные функции могут напрямую вызывать код "по умолчанию 并发"
- @block только как частный случай принудительного последовательного выполнения

### Ключевая инновация: выполнение снизу вверх + глобальный DAG

Ключевая инновация данного дизайна — **модель выполнения снизу вверх**:

```
Традиционный вызов (top-down):
  call fetch(url) → выполнение → возврат результата

Выполнение снизу вверх:
  print(a) ← начинаем с "места, где нужен результат"
       ↑
  fetch(url0) ← анализируем зависимости, ищем в обратном направлении

  fetch(url1) ← изолированный, независимое параллельное выполнение
```

**Ключевые различия**:
- Не "при встрече с вызовом функции генерировать Future"
- А начинаем с "конечного результата" и анализируем зависимости в обратном направлении
- Узлы без потребителей (изолированные) не выполняются или выполняются независимо параллельно
- Бесконечные циклы как фоновые DAG, планировщик выполняет их по частям

### Глобальный DAG планировщик

```
Вид DAG всей программы:

        print(result) ─────────────────────────┐
           │                                    │
    ┌──────┴──────┐                             │
    │             │                             │
process(a)   process(b)                        │
    │             │                             │
compute(x)   compute(y)  ←── изолированный DAG ──┤
    │                                           │
fetch(url0)  fetch(url1)  fetch(url2)          │
    (выполнено)                                 │

При этом есть ещё фоновый DAG (while True):
    ┌─────────────────────────────────────────┐ │
    │  while True:                            │ │
    │      update_ui()                        │ │
    │      fetch_new() ──→ process(data)      │ │
    └─────────────────────────────────────────┘ │
```

**Принцип работы планировщика**:
```
1. Обратный анализ от "конечного результата":
   print(result) → зависит от process → зависит от fetch

2. Построение глобального DAG:
   - Листовые узлы: fetch (нет зависимостей)
   - Внутренние узлы: process, compute
   - Корневой узел: print

3. Выполнение:
   - fetch выполняются параллельно
   - process ожидает завершения fetch
   - print ожидает завершения process
   - изолированные compute выполняются независимо параллельно

4. Пропуск уже выполненного:
   - Если узел уже выполнен, зависимые от него последующие узлы
     могут использовать результат повторно
```

### Обработка бесконечных циклов

```
Сценарий 1: один while/for (без накладных расходов планирования)
──────────────────────────────────────────────
main: () -> () = {
    while True {
        update_ui()
        fetch_data()
    }
}
→ только один бесконечный цикл
→ прямое синхронное выполнение, без разницы с обычным кодом

Сценарий 2: несколько while (автоматическое разбиение)
──────────────────────────────────────────────
main: () -> () = {
    while True { update_ui() }      # фоновая задача 1
    while True { network_poll() }  # фоновая задача 2
    server_loop()                   # основная задача
}
→ 3 независимые задачи
→ планировщик переключает их по частям
→ настоящий параллелизм

Адаптивный планировщик:
──────────────────────────────────────────────
if количество_задач == 1:
    прямое выполнение (синхронное)
else:
    планирование по частям (параллельное)
```

**Обработка фоновых DAG**:
```
Основной DAG (с окончанием):
    fetch → process → print → конец

Фоновый DAG (бесконечный цикл):
    while True → update_ui → fetch_new → process → возврат в начало

Планировщик:
    - основной DAG завершается после выполнения
    - фоновый DAG работает вечно, но планировщик выполняет его
      "по частям"
    - не зависает в цикле
```

### Основной дизайн

```
┌─────────────────────────────────────────────────────┐
│  Компиляция                                          │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐           │
│  │ Parser  │→│DAG анализ│→│LLVM Codegen│→ машинный код │
│  └─────────┘  └─────────┘  └─────────┘           │
│                      ↓                           │
│              Генерация: DAG метаданные                │
└─────────────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────┐
│  Runtime                                             │
│  ┌─────────────────────────────────────────────┐ │
│  │  DAG Scheduler библиотека                    │ │
│  │  • Загрузка машинного кода                   │ │
│  │  • Чтение DAG метаданных                    │ │
│  │  • Lazy планирование: приостановка вызовов,  │ │
│  │    выполнение по требованию                 │ │
│  │  • Поддержка параллельного/последовательного  │ │
│  │    выполнения                               │ │
│  └─────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### Процесс выполнения снизу вверх

```
Пользовательский код:
    main: () -> () = {
        pages = urls.map(|url| fetch(url))
        results = pages.map(|page| parse_page(page))
        save_results(results)
    }

Фаза 1: Обратный анализ снизу вверх (компиляция)
─────────────────────────────────────────
Начиная с save_results(results):
    "нужен results" → зависит от parse_page(results)
    "нужен page0" → зависит от fetch(url0)
    "нужен page1" → зависит от fetch(url1)
    ...

Построение глобального DAG:
    fetch(url0), fetch(url1), fetch(url2) ← листовые узлы
           ↓
    parse_page(page0), parse_page(page1)   ← зависят от листьев
           ↓
    save_results                          ← корневой узел

Фаза 2: Параллельное выполнение листьев (runtime)
─────────────────────────────────────────
Планировщик находит все листовые узлы:
    - fetch(url0), fetch(url1), fetch(url2) без зависимостей → параллельное выполнение
    - контроль количества 并发 (например, 16)

Фаза 3: Обход вверх
─────────────────────────────────────────
Когда parse_page нужен page0:
    - проверка готовности page0
    - готов → выполнить parse_page
    - не готов → ожидание, после завершения продолжение

Фаза 4: Изолированные параллельно
─────────────────────────────────────────
Если результат какого-то fetch никому не нужен:
    - выполняется как "изолированный DAG" независимо
    - может использовать другой核心, не влияет на основной поток
```

### Структура артефактов компиляции

```rust
/// Артефакт компиляции: машинный код + DAG метаданные
pub struct CompiledArtifact {
    /// Машинный код, скомпилированный LLVM (ELF/Mach-O/COFF)
    machine_code: Vec<u8>,

    /// DAG метаданные: описывают зависимости функций
    dag: DAGMetadata,

    /// Таблица точек входа
    entries: Vec<EntryPoint>,

    /// Информация о типах (для FFI)
    type_info: TypeInfo,
}

/// DAG метаданные
pub struct DAGMetadata {
    /// Узлы: вызовы функций
    nodes: Vec<DAGNode>,
    /// Рёбра: зависимости (from, to)
    edges: Vec<(usize, usize)>,
}

/// Один DAG узел
pub struct DAGNode {
    /// ID функции
    pub function_id: usize,
    /// Зависимые узлы ID
    pub deps: Vec<usize>,
    /// Метка побочного эффекта (@IO / @Pure)
    pub effect: EffectTag,
}
```

### Интерфейс runtime планировщика

```rust
/// DAG Scheduler trait
pub trait DAGScheduler: Send + Sync {
    /// Планирование выполнения
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue;

    /// Выполнение одной функции
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}

/// Реализация планировщика
pub struct DefaultDAGScheduler {
    /// Пул потоков
    thread_pool: ThreadPool,
    /// Артефакт компиляции
    artifact: CompiledArtifact,
    /// Максимальная степень параллелизма
    max_parallelism: usize,
}

impl DefaultDAGScheduler {
    pub fn new(artifact: CompiledArtifact, num_workers: usize) -> Self {
        Self {
            thread_pool: ThreadPool::new(num_workers),
            artifact,
            max_parallelism: num_workers * 2, // адаптивный контроль гранулярности
        }
    }
}

impl DAGScheduler for DefaultDAGScheduler {
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue {
        // 1. Обход тела функции, приостановка всех вызовов
        // 2. Построение списка задач для выполнения
        // 3. Планирование выполнения по порядку зависимостей (контроль 并发)
        // 4. Запуск выполнения при запросе значения
        // 5. Возврат результата
    }
}
```

### Стратегия планирования

| Аннотация | Сценарий | Стратегия планирования |
|-----------|----------|------------------------|
| `@auto` (по умолчанию, L3) | Прозрачный 并发 | Lazy DAG планирование, выполнение без зависимостей параллельно |
| `@block` (L1) | Принудительная синхронизация | Без DAG, чистое последовательное выполнение |
| Циклическая зависимость | Обнаружена runtime | Ошибка |

### Обработка побочных эффектов: неявная Effect System

Пользователь не замечает обработку побочных эффектов, компилятор выводит автоматически:

```
Пользовательский код:
  print("a");
  print("b");
  let x = compute(1);
  let y = compute(2);

Вывод компилятора:
  print → @IO (внешний вызов)
  compute → @Pure (чистая функция)

Выполнение планировщиком:
  print("a") ──→ последовательно (все @IO)
  print("b") ──→ последовательно
  compute(1) ─┬─→ параллельно (DAG планирование)
  compute(2) ─┘
```

### Отношение к трёхуровневому runtime

RFC-008 определяет архитектуру Embedded / Standard / Full трёхуровневого runtime. Соответствие LLVM AOT компилятора и трёхуровневого runtime:

| Runtime | LLVM AOT поведение |
|---------|-------------------|
| **Embedded** | Без DAG планирования, прямо генерирует последовательный машинный код |
| **Standard** | DAG + однопоточное планирование (num_workers=1) |
| **Full** | DAG + многопоточное планирование (num_workers>1), поддержка WorkStealing |

### Дизайн интерфейса планировщика

```rust
/// Стратегия планирования
pub enum ScheduleStrategy {
    /// @block: принудительное последовательное, без DAG
    Serial,
    /// @eager: eager evaluation, ожидание завершения зависимостей
    Eager,
    /// @auto (по умолчанию): lazy планирование, автоматическое DAG планирование
    Lazy,
}

/// Метка побочного эффекта
pub enum EffectTag {
    /// Чистая функция, без побочных эффектов
    Pure,
    /// Имеет I/O побочные эффекты
    IO,
}

/// DAG Scheduler trait
pub trait DAGScheduler: Send + Sync {
    /// Планирование выполнения (с параметром стратегии)
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint], strategy: ScheduleStrategy) -> RuntimeValue;

    /// Выполнение одной функции
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}
```

## Компромиссы

### Преимущества

1. **Повышение производительности**: AOT компиляция быстрее интерпретируемого выполнения в 10-100x
2. **Решение проблемы цветных функций**: 默认并发, синхронный — это частный случай
3. **Единый runtime**: интерпретатор и LLVM используют общий планировщик
5. **Неявные побочные эффекты**: пользователь не замечает, компилятор обрабатывает автоматически
6. **Безопасность владения**: зависит от модели владения в стиле Rust, без гонок данных

### Недостатки

1. **Сложность реализации**: требуется опыт работы с LLVM
2. **Время компиляции**: AOT компиляция медленнее интерпретатора
3. **Сложность отладки**: отладка AOT кода сложнее, чем интерпретатора

### Согласованность с дизайном RFC

| RFC | Согласованность |
|-----|-----------------|
| RFC-001 并作 модель | ✅ DAG анализ зависимостей — это основа |
| RFC-008 Runtime архитектура | ✅ Дизайн runtime планировщика согласован |
| RFC-009 Модель владения | ✅ ARC runtime реализован корректно |

| Решение | Описание | Почему не выбрано |
|---------|----------|-------------------|

| Связывание с внешним LLVM runtime | Использование LLVM runtime | Требуются дополнительные зависимости |

## Связанные работы

### Lazy Task Creation (1990)[^1]

| Атрибут | Описание |
|---------|----------|
| Организация | MIT |
| Авторы | James R. Larus, Robert H. Halstead Jr. |
| Суть | Lazy создание подзадач, создание по требованию |
| Ценность | Техническая база, источник концепции lazy планирования |

**Основная идея**: задачи создаются не сразу, а откладываются. Когда родительской задаче нужен результат дочерней, тогда и создаётся дочерняя задача. Это решает проблему накладных расходов при выполнении задач с мелкой гранулярностью[^1].

### Lazy Scheduling (2014)[^2]

| Атрибут | Описание |
|---------|----------|
| Организация | University of Maryland |
| Авторы | Tzannes, Caragea |
| Суть | Адаптивное планирование runtime, без дополнительного состояния |
| Ценность | Дизайн планировщика, адаптивный контроль гранулярности |

**Основная идея**: "lazy выполнение" автоматически контролирует гранулярность без поддержки сложного состояния. Когда система занята, задачи автоматически объединяются, когда свободна — разделяются[^2].

### Язык SISAL[^3]

| Атрибут | Описание |
|---------|----------|
| Организация | Lawrence Livermore National Laboratory (LLNL) |
| Суть | Single-assignment язык, Dataflow граф, неявный параллелизм |
| Ценность | Доказательство осуществимости, производительность близка к Fortran |

**Основной вклад**: SISAL доказал, что модель Dataflow в промышленных приложениях может достичь производительности, близкой к Fortran[^3].

### Параллельная схема Mul-T[^4]

| Атрибут | Описание |
|---------|----------|
| Организация | MIT |
| Суть | Конструкция Future, реализация Lazy Task Creation |
| Ценность | Конкретная реализация для справки |

**Основной механизм**:
```scheme
;; Синтаксис Multilisp / Mul-T
(let ((a (future compute-a))      ;; немедленный возврат future
      (b (future compute-b)))      ;; немедленный возврат future
  (join a b))                      ;; ожидание завершения
```

### Сравнительное резюме

| Технология | Lazy создание | DAG анализ | Обработка побочных эффектов | Владение |
|------------|---------------|-------------|-----------------------------|----------|
| Lazy Task Creation[^1] | ✅ | ❌ | ❌ | N/A |
| Lazy Scheduling[^2] | ✅ | ❌ | ❌ | N/A |
| SISAL[^3] | ✅ | ✅ (глобальный) | N/A (single assignment) | N/A |
| Mul-T[^4] | ✅ | ❌ | ❌ | N/A |
| **YaoXiang** | ✅ | ✅ (внутри функций) | ✅ (неявный) | ✅ (ARC) |

**Инновация YaoXiang**: использование современных языковых возможностей (владение + неявные побочные эффекты) упрощает традиционный дизайн, ограничение DAG внутри функциональных блоков снижает сложность.

## Сравнение с традиционными методами автоматического параллелизма

### Традиционный компилятор: параллелизм уровня циклов

Коммерческие компиляторы (Intel Fortran, Oracle Fortran) используют **автоматический параллелизм уровня циклов**[^5]:

**Основной процесс**:
```
1. Определение параллелизуемых циклов
2. Анализ зависимостей обращений к массивам внутри цикла
3. Определение наличия зависимостей между итерациями цикла
4. Если зависимостей нет, генерация многопоточного кода
```

**Технологии анализа зависимостей**:

| Технология | Описание |
|------------|----------|
| **Зависимости данных** | Обращаются ли два доступа к одной памяти |
| **Use-Def** | Отношения определения и использования переменных |
| **Анализ алиасов** | Указывают ли указатели на одну память |

**Условия параллелизуемости цикла**:
```fortran
! Можно параллелизовать
DO I = 1, N
  A(I) = C(I)
END DO

! Нельзя параллелизовать (зависимость от предыдущей итерации)
DO I = 2, N
  A(I) = A(I-1) + B(I)
END DO
```

### Haskell: механизм Spark

GHC (Glasgow Haskell Compiler) использует **механизм Spark** для реализации 并发 чистых функций[^6]:

```haskell
-- rpar: параллельное выполнение, создание spark
-- rseq: последовательное выполнение, ожидание завершения

example = do
  a <- rpar (f x)   -- создание spark, параллельное выполнение f x
  b <- rpar (g y)   -- создание spark, параллельное выполнение g y
  rseq a            -- ожидание завершения a
  rseq b            -- ожидание завершения b
  return (a, b)
```

**Механизм пула Spark**:
- Spark берутся из пула и分配给空闲ные процессоры
- Если spark не использован (никто не ждёт результат), он перерабатывается GC
- Это решает проблему гранулярности: слишком мелкие spark отбрасываются

### Язык Clean: уникальные типы

Язык Clean реализует 并发 безопасность через **уникальные типы (Uniqueness Types)**[^7]:

```clean
-- *Array означает уникальность, можно безопасно изменять
modify :: *Array Int -> *Array Int
```

**Основная идея**: если значение имеет唯一ственную ссылку, его можно безопасно изменять в параллельной среде, потому что其他ные ссылки не увидят промежуточное состояние.

### Программные срезы и графы зависимостей

**Граф программных зависимостей (PDG)** является основой обнаружения параллелизма:

```
Узлы: операторы
Рёбра: зависимости по данным + управления

Обнаружение параллелизма:
  Если между двумя узлами нет достижимого пути → можно параллелизовать
```

### Комплексное сравнение

| Метод | Анализ зависимостей | Гранулярность | Обработка побочных эффектов | Типичный сценарий |
|-------|---------------------|---------------|-----------------------------|-------------------|
| Intel/Oracle Fortran[^5] | Сложный анализ массивов | Итерации цикла | N/A | Научные вычисления |
| GHC Spark[^6] | Предположение о чистых функциях | Выражения | N/A | Функциональное программирование |
| Clean[^7] | Уникальные типы | Перезапись графов | N/A | Функциональное программирование |
| **YaoXiang** | Гарантия владения | Вызовы функций | Неявный вывод | Универсальный |

### Приложение B: Примеры оптимизации планировщика

**Сценарий 1: Планировщик обнаружил возможность объединения выполнения**

```
Исходный DAG:
  compute_a() ──┐
  compute_b() ──┼──→ compute_c()

После оптимизации планировщика:
  Объединение compute_a + compute_b в одну задачу
  → уменьшение накладных расходов планирования
```

**Сценарий 2: Зависимость не используется**

```
let a = expensive_compute(); // вычислено
let b = other_thing();       // a не нужна
print(b);                    // напрямую возвращает b, пропускает a
```
