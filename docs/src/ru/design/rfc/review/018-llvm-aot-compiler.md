# RFC-018: Проектирование LLVM AOT компилятора

> ** Ссылки**:
> - [RFC-024: Модель параллелизма на основе блоков spawn](./accepted/024-concurrency-model.md)
> - [RFC-008: Деклаплинг модели параллелизма и планировщика Runtime](./accepted/008-runtime-concurrency-model.md)
> - [RFC-009: Проектирование модели владения](./accepted/009-ownership-model.md)

## Аннотация

В данном документе проектируется LLVM AOT компилятор для языка YaoXiang, целью которого является генерация машинного кода + метаданных DAG посредством предварительной компиляции, с последующим планированием DAG планировщиком **внутри блоков spawn**, на основе **анализа зависимостей снизу вверх**.

**Основные инновации**:
- Не "при встрече с вызовом функции генерировать Future", а **анализ зависимостей в обратном направлении от места, где требуется результат**
- **Листовые узлы выполняются параллельно в первую очередь**, цепочки зависимостей обходятся последовательно снизу вверх
- **Изолированные DAG выполняются параллельно независимо**: узлы без потребителей не блокируют основной поток
- **Бесконечные циклы выполняются как фоновые DAG**: планировщик выполняет их по частям, не зависая
- **Анализ DAG ограничен блоками spawn**: эффективная компиляция, предсказуемое поведение

Данная архитектура фундаментально отличается от модели Rust async/await + runtime tokio:
- Rust: пользователь пишет `async fn`, компилятор генерирует конечный автомат
- YaoXiang: пользователь пишет обычные функции, **компилятор автоматически анализирует DAG внутри блоков spawn**, планировщик выполняет снизу вверх

Соблюдается модель параллелизма на основе блоков spawn из RFC-024: обычный код выполняется последовательно, внутри блоков spawn выполняется параллельное планирование DAG.

## Мотивация

### Зачем нужен LLVM AOT компилятор?

В настоящее время YaoXiang имеет только интерпретатор в качестве исполняющего backend, что создаёт следующие проблемы:

| Проблема | Влияние |
|----------|---------|
| Проблема производительности | Интерпретируемое выполнение медленнее машинного кода в 10-100 раз |
| Сложность развёртывания | Необходимо поставлять интерпретатор и runtime |
| Проблема цветных функций | Синхронные функции не могут вызывать параллельные функции |

### Проблема цветных функций и параллелизм блоков spawn

**Традиционное проектирование**:
- Синхронные функции (синие) → не могут вызывать → параллельные функции (красные)
- Синхронность используется по умолчанию, параллелизм требует пометки `spawn`
- Цвет "заразен": однажды использовав параллелизм, вся цепочка вызовов становится параллельной

**Параллелизм блоков spawn RFC-024 (цель)**:
- Обычный код выполняется последовательно, без окрашивания функций
- Внутри блока `spawn { ... }` анализируется DAG, выполняется параллельно
- Вызывающий код блокируется синхронно, без коллбэков и `await`

**Инвертированное проектирование (RFC-018)**:
- Обычный код напрямую генерирует последовательный машинный код без накладных расходов DAG
- Внутри блоков spawn компилятор автоматически анализирует зависимости DAG во время компиляции, runtime выполняет снизу вверх
- Решение проблемы цветных функций: все функции унифицированы, без разделения на синхронные/параллельные

### Основная инновация: выполнение снизу вверх + DAG внутри блоков spawn

Ключевая инновация данного проекта — **модель выполнения снизу вверх** (ограниченная блоками spawn):

```
Традиционный вызов (сверху вниз):
  call fetch(url) → выполнение → возврат результата

Выполнение снизу вверх:
  print(a) ← начинаем с "места, где требуется результат"
       ↑
  fetch(url0) ← анализируем зависимости в обратном направлении

  fetch(url1) ← изолированный, выполняется параллельно независимо
```

**Ключевые различия**:
- Не "при встрече с вызовом функции генерировать Future"
- Анализ зависимостей в обратном направлении от конечного требуемого результата
- Узлы без потребителей (изолированные) не выполняются или выполняются параллельно независимо
- Бесконечные циклы выполняются как фоновые DAG, планировщик выполняет их по частям

### Планировщик DAG внутри блоков spawn

```
Представление DAG внутри блока spawn:

        print(result) ─────────────────────────┐
           │                                    │
    ┌──────┴──────┐                             │
    │             │                             │
process(a)   process(b)                        │
    │             │                             │
compute(x)   compute(y)  ←── изолированный DAG ─┤
    │                                           │
fetch(url0)  fetch(url1)  fetch(url2)          │
    (выполнен)                                    │

В то же время существует фоновый DAG (while True):
    ┌─────────────────────────────────────────┐ │
    │  while True:                            │ │
    │      update_ui()                        │ │
    │      fetch_new() ──→ process(data)      │ │
    └─────────────────────────────────────────┘ │
```

**Принцип работы планировщика**:
```
1. Анализ в обратном направлении от "конечного результата":
   print(result) → зависит от process → зависит от fetch

2. Построение DAG внутри блока spawn:
   - Листовые узлы: fetch (без зависимостей)
   - Внутренние узлы: process, compute
   - Корневой узел: print

3. Выполнение:
   - fetch выполняются параллельно
   - process ожидает завершения fetch
   - print ожидает завершения process
   - изолированный compute выполняется параллельно независимо

4. Пропуск уже выполненных:
   - Если какой-либо узел уже выполнен, зависимые от него последующие
     узлы могут использовать результат повторно
```

### Обработка бесконечных циклов

```
Сценарий 1: единственный while/for (без накладных расходов планирования)
──────────────────────────────────────────────────────────────────────
main: () -> () = {
    while True {
        update_ui()
        fetch_data()
    }
}
→ единственный бесконечный цикл
→ прямое синхронное выполнение, ничем не отличается от обычного кода

Сценарий 2: несколько while (автоматическое разделение)
────────────────────────────────────────────────────────
main: () -> () = {
    while True { update_ui() }      # фоновый процесс 1
    while True { network_poll() }  # фоновый процесс 2
    server_loop()                   # основной процесс
}
→ 3 независимых процесса
→ планировщик переключается между ними
→ настоящий параллелизм

Адаптивный планировщик:
──────────────────────────────────────────────────────────────────────
if количество_задач == 1:
    прямое выполнение (синхронно)
else:
    планирование по частям (параллельно)
```

**Обработка фоновых DAG**:
```
Основной DAG (с окончанием):
    fetch → process → print → завершение

Фоновый DAG (бесконечный цикл):
    while True → update_ui → fetch_new → process → возврат в начало

Планировщик:
    - основной DAG завершается после выполнения
    - фоновый DAG работает вечно, но планировщик выполняет его "по частям"
    - не зависает в цикле
```

### Процесс выполнения снизу вверх

```
Пользовательский код:
    main: () -> () = {
        pages = urls.map(|url| fetch(url))
        results = pages.map(|page| parse_page(page))
        save_results(results)
    }

Фаза 1: Анализ снизу вверх (время компиляции)
──────────────────────────────────────────────────────────────────────
Начиная с save_results(results):
    "требуется results" → зависит от parse_page(results)
    "требуется page0" → зависит от fetch(url0)
    "требуется page1" → зависит от fetch(url1)
    ...

Построение DAG внутри блока spawn:
    fetch(url0), fetch(url1), fetch(url2) ← листовые узлы
           ↓
    parse_page(page0), parse_page(page1)   ← зависят от листьев
           ↓
    save_results                          ← корневой узел

Фаза 2: Параллельное выполнение листьев (время выполнения)
──────────────────────────────────────────────────────────────────────
Планировщик находит все листовые узлы:
    - fetch(url0), fetch(url1), fetch(url2) без зависимостей → параллельно
    - контроль параллелизма (например, 16 одновременных)

Фаза 3: Обход снизу вверх
──────────────────────────────────────────────────────────────────────
Когда parse_page требуется page0:
    - проверка готовности page0
    - готово → выполнить parse_page
    - не готово → ожидание, затем продолжение

Фаза 4: Изолированные параллельно независимо
──────────────────────────────────────────────────────────────────────
Если результат某个 fetch никем не используется:
    - выполняется как "изолированный DAG"
    - может использовать другой процессор, не влияет на основной поток
```

### Структура результатов компиляции

```rust
/// Результат компиляции: машинный код + метаданные DAG
pub struct CompiledArtifact {
    /// Машинный код, скомпилированный LLVM (ELF/Mach-O/COFF)
    machine_code: Vec<u8>,

    /// Метаданные DAG: описание зависимостей функций
    dag: DAGMetadata,

    /// Таблица точек входа
    entries: Vec<EntryPoint>,

    /// Информация о типах (для FFI)
    type_info: TypeInfo,
}

/// Метаданные DAG
pub struct DAGMetadata {
    /// Узлы: вызовы функций
    nodes: Vec<DAGNode>,
    /// Ребра: зависимости (from, to)
    edges: Vec<(usize, usize)>,
}

/// Отдельный узел DAG
pub struct DAGNode {
    /// ID функции
    pub function_id: usize,
    /// Зависимые узлы
    pub deps: Vec<usize>,
    /// Метка побочного эффекта (@IO / @Pure)
    pub effect: EffectTag,
}
```

### Интерфейс планировщика runtime

```rust
/// Трейт планировщика DAG
pub trait DAGScheduler: Send + Sync {
    /// Планирование выполнения
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue;

    /// Выполнение одной функции
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}

/// Реализация планировщика
pub struct DefaultDAGScheduler {
    /// Пул потоков
    thread_pool: ThreadPool,
    /// Результат компиляции
    artifact: CompiledArtifact,
    /// Максимальный параллелизм
    max_parallelism: usize,
}

impl DefaultDAGScheduler {
    pub fn new(artifact: CompiledArtifact, num_workers: usize) -> Self {
        Self {
            thread_pool: ThreadPool::new(num_workers),
            artifact,
            max_parallelism: num_workers * 2, // адаптивный контроль зернистости
        }
    }
}

impl DAGScheduler for DefaultDAGScheduler {
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue {
        // 1. Обход тела функции, приостановка всех вызовов
        // 2. Построение списка задач для выполнения
        // 3. Планирование выполнения по порядку зависимостей (контроль параллелизма)
        // 4. Триггер выполнения при запросе значения
        // 5. Возврат результата
    }
}
```

### Стратегии планирования

| Сценарий | Стратегия планирования |
|----------|------------------------|
| Обычный код (вне блоков spawn) | Последовательное выполнение, без накладных расходов DAG |
| Внутри блока spawn | Отложенное планирование DAG, выполнение без зависимостей параллельно |
| Циклические зависимости | Обнаружение во время выполнения, ошибка |

### Обработка побочных эффектов: неявная система Effect

Неявная обработка побочных эффектов без ведома пользователя, компилятор выводит автоматически:

```
Пользовательский код:
  print("a")
  print("b")
  x = compute(1)
  y = compute(2)

Вывод компилятора:
  print → @IO (внешний вызов)
  compute → @Pure (чистая функция)

Выполнение планировщиком:
  print("a") ──→ последовательно (все @IO)
  print("b") ──→ последовательно
  compute(1) ─┬─→ параллельно (планирование DAG внутри spawn)
  compute(2) ─┘
```

### Отношение к трёхуровневой архитектуре runtime

RFC-008 определяет трёхуровневую архитектуру Embedded / Standard / Full runtime. Соответствие LLVM AOT компилятора трём уровням runtime (выравнивание с RFC-024):

| Runtime | Поведение LLVM AOT |
|---------|-------------------|
| **Embedded** | Без поддержки spawn, напрямую генерируется последовательный машинный код |
| **Standard** | Поддержка блоков spawn, DAG внутри spawn + однопоточное планирование (num_workers=1) |
| **Full** | Поддержка блоков spawn, DAG внутри spawn + многопоточное планирование (num_workers>1), поддержка WorkStealing |

### Проектирование интерфейса планировщика

```rust
/// Метка побочного эффекта
pub enum EffectTag {
    /// Чистая функция без побочных эффектов
    Pure,
    /// Имеет побочный эффект I/O
    IO,
}

/// Трейт планировщика DAG
pub trait DAGScheduler: Send + Sync {
    /// Планирование выполнения DAG внутри блока spawn
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue;

    /// Выполнение одной функции (обычный код, последовательное выполнение)
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}
```

## Компромиссы

### Преимущества

1. **Повышение производительности**: AOT компиляция быстрее интерпретируемого выполнения в 10-100 раз
2. **Решение проблемы цветных функций**: без окрашивания функций, параллелизм внутри блоков spawn
3. **Унифицированный runtime**: интерпретатор и LLVM используют общий планировщик
5. **Неявные побочные эффекты**: без ведома пользователя, компилятор обрабатывает автоматически
6. **Безопасность владения**: зависит от модели владения в стиле Rust, без гонок данных

### Недостатки

1. **Сложность реализации**: требуется опыт интеграции LLVM
2. **Время компиляции**: AOT компиляция медленнее интерпретатора
3. **Сложность отладки**: отладка AOT кода сложнее, чем интерпретатора

### Согласованность с дизайном RFC

| RFC | Согласованность |
|-----|-----------------|
| RFC-024 модель параллелизма блоков spawn | ✅ Анализ DAG ограничен блоками spawn |
| RFC-008 архитектура runtime | ✅ Дизайн планировщика runtime согласован |
| RFC-009 модель владения | ✅ ARC runtime корректно реализован |

## Альтернативные решения

| Решение | Описание | Почему не выбрано |
|---------|----------|-------------------|
| Только интерпретатор | Без AOT | Недостаточная производительность, без поддержки параллелизма блоков spawn |
| Чистая статическая компиляция | Без планирования runtime | Отложенное планирование требуется во время выполнения |
| Связывание с внешним LLVM runtime | Использование LLVM runtime | Требуются дополнительные зависимости |

## Связанные работы

### Lazy Task Creation (1990)[^1]

| Атрибут | Описание |
|---------|----------|
| Организация | MIT |
| Авторы | James R. Larus, Robert H. Halstead Jr. |
| Суть | Отложенное создание подзадач, создание по требованию |
| Ценность ссылки | Техническая основа, истоки концепции отложенного планирования |

**Основная идея**: задачи создаются не сразу, а отложенно. Когда родительской задаче требуется значение дочерней задачи, тогда и создаётся дочерняя задача. Это решает проблему накладных расходов на производительность при мелкозернистом параллелизме задач[^1].

### Lazy Scheduling (2014)[^2]

| Атрибут | Описание |
|---------|----------|
| Организация | University of Maryland |
| Авторы | Tzannes, Caragea |
| Суть | Адаптивное планирование runtime, без дополнительного состояния |
| Ценность ссылки | Дизайн планировщика, адаптивный контроль зернистости |

**Основная идея**: автоматический контроль зернистости через "отложенное выполнение", без поддержки сложного состояния. Когда система занята, задачи автоматически объединяются; когда простаивает — автоматически разделяются[^2].

### Язык SISAL[^3]

| Атрибут | Описание |
|---------|----------|
| Организация | Lawrence Livermore National Laboratory (LLNL) |
| Суть | Язык с единственным присваиванием, Dataflow граф, неявный параллелизм |
| Ценность ссылки | Доказательство осуществимости, производительность близка к Fortran |

**Основной вклад**: SISAL доказал, что модель Dataflow в промышленных приложениях может достигать производительности, близкой к Fortran[^3].

### Параллельная схема Mul-T[^4]

| Атрибут | Описание |
|---------|----------|
| Организация | MIT |
| Суть | Конструкция Future, реализация Lazy Task Creation |
| Ценность ссылки | Ссылка на конкретную реализацию |

**Основной механизм**:
```scheme
;; Синтаксис Multilisp / Mul-T
(let ((a (future compute-a))      ;; немедленный возврат future
      (b (future compute-b)))      ;; немедленный возврат future
  (join a b))                      ;; ожидание завершения
```

### Сравнительная сводка

| Технология | Отложенное создание | Анализ DAG | Обработка побочных эффектов | Владение |
|------------|---------------------|------------|-----------------------------|----------|
| Lazy Task Creation[^1] | ✅ | ❌ | ❌ | N/A |
| Lazy Scheduling[^2] | ✅ | ❌ | ❌ | N/A |
| SISAL[^3] | ✅ | ✅ (глобально) | N/A (единственное присваивание) | N/A |
| Mul-T[^4] | ✅ | ❌ | ❌ | N/A |
| **YaoXiang** | ✅ | ✅ (внутри блоков spawn) | ✅ (неявно) | ✅ (ARC) |

**Инновация YaoXiang**: использование современных языковых возможностей (владение + неявные побочные эффекты) для упрощения традиционного дизайна, ограничение DAG блоками spawn для снижения сложности.

## Сравнение с традиционными методами автоматического параллелизма

### Традиционные компиляторы: параллелизм на уровне циклов

Коммерческие компиляторы (Intel Fortran, Oracle Fortran) используют **автоматический параллелизм на уровне циклов**[^5]:

**Основной процесс**:
```
1. Идентификация распараллеливаемых циклов
2. Анализ зависимостей обращений к массивам внутри цикла
3. Определение наличия зависимостей между итерациями цикла
4. При отсутствии зависимостей генерация многопоточного кода
```

**Технологии анализа зависимостей**:

| Технология | Описание |
|------------|----------|
| **Зависимость данных** | Обращаются ли два обращения к одной и той же области памяти |
| **Use-Def** | Отношение определения и использования переменных |
| **Анализ псевдонимов** | Указывают ли указатели на одну и ту же область памяти |

**Условия распараллеливаемости циклов**:
```fortran
! Можно распараллелить
DO I = 1, N
  A(I) = C(I)
END DO

! Нельзя распараллелить (зависимость от предыдущей итерации)
DO I = 2, N
  A(I) = A(I-1) + B(I)
END DO
```

### Haskell: механизм Spark

GHC (Glasgow Haskell Compiler) использует **механизм Spark** для реализации параллелизма чистых функций[^6]:

```haskell
-- rpar: параллельное выполнение, создание spark
-- rseq: последовательное выполнение, ожидание завершения

example = do
  a <- rpar (f x)   -- создание spark, параллельное выполнение f x
  b <- rpar (g y)   -- создание spark, параллельное выполнение g y
  rseq a            -- ожидание завершения a
  rseq b            -- ожидание завершения b
  return (a, b)
```

**Механизм пула Spark**:
- Извлечение spark из пула и分配空闲处理核
- Если spark не использован (никто не ожидает результат), он перерабатывается GC
- Это решает проблему зернистости: слишком мелкие spark отбрасываются

### Язык Clean: типы уникальности

Язык Clean реализует безопасность параллелизма через **типы уникальности (Uniqueness Types)**[^7]:

```clean
-- *Array означает уникальность, можно безопасно изменять
modify :: *Array Int -> *Array Int
```

**Основная идея**: если значение имеет唯一ственную ссылку, его можно безопасно изменять в параллельной среде, поскольку никакая другая ссылка не увидит промежуточное состояние.

### Программные срезы и графы зависимостей

**Граф зависимостей программы (PDG)** является основой для обнаружения параллелизма:

```
Узлы: операторы
Ребра: зависимости по данным + зависимости управления

Обнаружение параллелизма:
  Если между двумя узлами нет достижимого пути → можно выполнять параллельно
```

### Комплексное сравнение

| Метод | Анализ зависимостей | Зернистость | Обработка побочных эффектов | Типичный сценарий |
|-------|---------------------|-------------|-----------------------------|-------------------|
| Intel/Oracle Fortran[^5] | Сложный анализ массивов | Итерации цикла | N/A | Научные вычисления |
| GHC Spark[^6] | Предположение о чистых функциях | Выражения | N/A | Функциональное программирование |
| Clean[^7] | Типы уникальности | Перезапись графа | N/A | Функциональное программирование |
| **YaoXiang** | Гарантия владения | Вызовы функций | Неявный вывод | Общего назначения |

### Приложение B: Примеры оптимизации планировщика

**Сценарий 1: Планировщик обнаруживает возможность объединения**

```
Исходный DAG:
  compute_a() ──┐
  compute_b() ──┼──→ compute_c()

После оптимизации планировщиком:
  Объединить compute_a + compute_b в одну задачу
  → уменьшение накладных расходов планирования
```

**Сценарий 2: Зависимость не используется**

```
a = expensive_compute()  // вычислено
b = other_thing()        // не требует a
print(b)                 // напрямую вернуть b, пропустить a
```
