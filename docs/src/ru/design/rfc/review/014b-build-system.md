# RFC-014b: Система сборки и распространение бинарных пакетов

> Данный RFC является под-RFC к [RFC-014: Проектирование системы управления пакетами](../accepted/014-package-manager.md).

## Резюме

Определяет механизмы сборки системы управления пакетами YaoXiang: декларативная конфигурация сборки, стратегии сборки (cargo/cmake/custom/none), распространение предкомпилированных бинарных пакетов, проверка системных зависимостей.

## Мотивация

Некоторые пакеты представляют собой чистый код `.yx` и не требуют сборки. Другие требуют компиляции FFI-привязок (вызов Cargo, CMake и т.д.). Необходим единый механизм, позволяющий авторам пакетов декларировать требования к сборке, а пакетному менеджеру — автоматически их обрабатывать.

### Текущие проблемы

- Отсутствует декларирование конфигурации сборки (в `yaoxiang.toml` нет секции `[build]`)
- Отсутствует механизм распространения предкомпилированных бинарных пакетов
- Сборка FFI-пакетов полностью зависит от ручных действий пользователя
- Отсутствует проверка системных зависимостей

## Предложение

### Основной замысел: декларативная сборка + приоритет предкомпиляции

Автор пакета декларирует требования к сборке в `yaoxiang.toml`, пакетный менеджер принимает решения автоматически на основании декларации.

### Стратегии сборки

```rust
enum BuildStrategy {
    None,          // Чистый .yx пакет, сборка не требуется
    Cargo,         // Вызов cargo build, чтение конфигурации [build.cargo]
    Cmake,         // Вызов cmake
    Custom,        // Выполнение скрипта build.yx
}
```

Примечание: вариант `Precompiled` удалён. Наличие секции `[binaries]` автоматически активирует поведение с приоритетом предкомпиляции, явное объявление strategy не требуется.

### Декларация сборки в yaoxiang.toml

```toml
[package]
name = "native-foo"
version = "1.0.0"

[build]
strategy = "cargo"              # Стратегия сборки
headers = ["include/sqlite3.h"] # Опционально: заголовочные файлы C, автоматически обрабатываемые yx-bindgen

[build.cargo]
features = ["ffi"]             # cargo build --features ffi
target = "release"             # cargo build --release

[build.requirements]
cargo = ">= 1.70"              # Инструменты, необходимые во время сборки
cmake = ">= 3.20"

[build.platforms]              # Переопределения для конкретных платформ
"x86_64-unknown-linux-gnu" = { cargo-features = ["linux-ffi"] }
"x86_64-pc-windows-msvc" = { cargo-features = ["win-ffi"] }
"aarch64-apple-darwin" = { cargo-features = ["mac-ffi"] }
```

### Дерево принятия решений при установке

```
yaoxiang install foo
    │
    ├─ 1. Есть запись для текущей платформы в [binaries]?
    │     → Да: загрузить, проверить SHA-256, установить напрямую (пропустить сборку)
    │     → Нет: продолжить
    │
    ├─ 2. Загрузить пакет с исходным кодом
    │
    ├─ 3. [build].headers содержит значения?
    │     → Да: автоматически запустить yx-bindgen для генерации файлов привязок
    │
    ├─ 4. Прочитать [build].strategy
    │     → "none": установить напрямую
    │     → "cargo": прочитать конфигурацию [build.cargo], сформировать команду cargo build
    │     → "cmake": вызвать cmake
    │     → "custom": выполнить скрипт build.yx
    │
    └─ 5. Установить в vendor/
```

**Приоритет предкомпиляции, исходный код как запасной вариант.** Наличие секции `[binaries]` автоматически активирует проверку предкомпиляции, явное указание strategy не требуется.

### Подробности стратегии cargo

При `strategy = "cargo"` считывается конфигурация `[build.cargo]` для формирования команды:

```toml
[build]
strategy = "cargo"

[build.cargo]
features = ["ffi"]             # → cargo build --features ffi
target = "release"             # → cargo build --release

[build.platforms]              # Переопределения платформы
"x86_64-unknown-linux-gnu" = { cargo-features = ["linux-ffi"] }
"x86_64-pc-windows-msvc" = { cargo-features = ["win-ffi"] }
"aarch64-apple-darwin" = { cargo-features = ["mac-ffi"] }
```

Фактически выполняемые команды:

```bash
# Базовый вариант
cargo build --release --features ffi

# При наличии переопределения платформы (на примере linux)
cargo build --release --features ffi,linux-ffi
```

### Декларация предкомпилированных бинарных пакетов

```toml
# yaoxiang.toml
[binaries]
"x86_64-unknown-linux-gnu" = { url = "releases/download/v1.0.0/foo-linux-x86_64.tar.gz", sha256 = "abc123" }
"x86_64-pc-windows-msvc" = { url = "https://example.com/foo-win-x86_64.tar.gz", sha256 = "def456" }
"aarch64-apple-darwin" = { url = "releases/download/v1.0.0/foo-macos-aarch64.tar.gz", sha256 = "ghi789" }
```

**Формат URL:** Поддерживаются абсолютные URL и относительные пути. Относительные пути задаются относительно адреса репозитория пакета (URL GitHub-репозитория или корневой URL реестра).

**Условия пропуска сборки:**
1. В `[binaries]` имеется запись для текущей платформы
2. Проверка SHA-256 прошла успешно
3. Загрузка выполнена успешно

При выполнении всех трёх условий → сборка пропускается. Иначе → возврат к сборке из исходного кода.

### Скрипт сборки build.yx

При `strategy = "custom"` выполняется `build.yx`.

**Модель выполнения (минимальная спецификация):**
- Скрипт является обычным кодом `.yx` с полным доступом к `std`
- Рабочая директория: корень пакета (`vendor/<pkg>-<ver>/`)
- Успех: код возврата 0
- Неудача: ненулевой код возврата, установка прерывается
- Пакетный менеджер не ограничивает поведение скрипта, проверяется только код возврата

```yx
# build.yx — скрипт сборки пакета
use std.os
use std.io

fn main() {
    let platform = os.platform()
    let arch = os.arch()

    if os.file_exists("Cargo.toml") {
        io.println("Building native extension via Cargo...")
        let result = os.exec("cargo build --release")
        if result.exit_code != 0 {
            io.println("Build failed!")
            os.exit(1)
        }
    }

    io.println("Build complete!")
}
```

### Проверка системных зависимостей

Перед установкой автоматически проверяются все `[build.requirements]`, при неудовлетворении выдаётся ошибка:

```
Error: Build requirement not satisfied
  cargo >= 1.70 required, but cargo is not installed
  Install: https://rustup.rs
```

### Интеграция с yx-bindgen (поле headers)

Поле `[build].headers` декларирует заголовочные файлы C, которые должен обработать yx-bindgen. Система сборки автоматически запускает yx-bindgen для генерации файлов привязок `.yx`.

```toml
[build]
strategy = "cargo"
headers = ["include/sqlite3.h", "include/json.h"]
```

Процесс сборки:

```
1. Есть предкомпилированный пакет в [binaries]? → пропустить всю сборку
2. [build].headers содержит значения? → yx-bindgen автоматически генерирует привязки
3. Выполнить [build].strategy (cargo/cmake/custom)
4. Установить
```

yx-bindgen разбирает сигнатуры функций и определения типов из заголовочных файлов C (`.h`) и автоматически генерирует декларации привязок `.yx`. Пользователю не нужно запускать его вручную — система сборки обрабатывает это автоматически при обнаружении конфигурации `headers`.

**Связь с RFC-026:** RFC-026 определяет языковую семантику `yx-bindgen` (синтаксис `native("symbol")`, типы unsafe). RFC-014b определяет способ его интеграции в процесс сборки (конфигурация `headers`). Они дополняют друг друга.

### Интеграция с Cargo Workspace

Если в пакете есть FFI-код, можно одновременно определить Cargo workspace:

```
my-package/
├── yaoxiang.toml          # Конфигурация пакета YaoXiang
├── Cargo.toml             # Cargo workspace (FFI-часть)
├── src/
│   └── lib.yx             # Код YaoXiang
└── native/
    ├── Cargo.toml          # Rust FFI-код
    └── src/
        └── lib.rs
```

`yaoxiang build` автоматически обнаруживает это и вызывает `cargo build` для компиляции нативной части.

## Детальное проектирование

### Идентификаторы платформ

Используется формат target triple из Rust (`arch-vendor-os-env`):

| Платформа | Идентификатор |
|------|------|
| Linux x86_64 (glibc) | `x86_64-unknown-linux-gnu` |
| Linux x86_64 (musl) | `x86_64-unknown-linux-musl` |
| Linux ARM64 | `aarch64-unknown-linux-gnu` |
| Windows x86_64 (MSVC) | `x86_64-pc-windows-msvc` |
| Windows x86_64 (MinGW) | `x86_64-pc-windows-gnu` |
| macOS ARM64 | `aarch64-apple-darwin` |
| macOS x86_64 | `x86_64-apple-darwin` |

Используется Rust target triple вместо упрощённого формата, потому что:
1. Различаются разные ABI в одной ОС (gnu vs musl, msvc vs gnu)
2. Совместимость с экосистемой Rust/Cargo, снижает вероятность ошибок маппинга
3. Будущее расширение не потребует изменения формата

### Структура каталогов артефактов сборки

```
build/
└── native/
    ├── x86_64-unknown-linux-gnu/
    │   └── libfoo.so
    ├── x86_64-pc-windows-msvc/
    │   └── foo.dll
    └── aarch64-apple-darwin/
        └── libfoo.dylib
```

### Полный жизненный цикл предкомпилированного пакета

```
Разработчик:
  1. Пишет код .yx + FFI-привязки
  2. Декларирует [build] + [binaries] в yaoxiang.toml
  3. yaoxiang publish
     → Автоматически собирает мультиплатформенные бинарники в CI
     → Загружает исходный код + предкомпилированные артефакты

Пользователь:
  yaoxiang add native-foo
    → Обнаружены предкомпилированные артефакты → загрузить напрямую (секунды)
    → Нет предкомпилированных артефактов → загрузить исходный код + выполнить сборку (минуты)
```

## Компромиссы

### Преимущества

- Декларативная конфигурация, пользователю не нужно понимать детали сборки
- Приоритет предкомпиляции, установка очень быстрая
- Поддержка нескольких платформ с автоматическим выбором
- Бесшовная интеграция с экосистемой Cargo

### Недостатки

- Предкомпилированные артефакты требуют поддержки CI
- Сборка для нескольких платформ усложняет выпуск
- Скрипты build.yx требуют механизма песочницы

## Альтернативы

| Вариант | Почему отклонён |
|------|-----------|
| Распространение только исходного кода | Пользователю нужно установить инструментарий сборки, высокий порог входа |
| Бинарный формат наподобие Python wheel | Слишком сложно, экосистеме YaoXiang на начальном этапе не нужно |
| Отсутствие поддержки FFI-сборки | Ограничивает расширяемость языка |

## Стратегия реализации

### Разбивка на этапы

| Этап | Содержание |
|------|------|
| Phase 5a | Разбор конфигурации `[build]` + enum `BuildStrategy` |
| Phase 5b | Проверка системных зависимостей |
| Phase 5c | Интеграция сборки Cargo (чтение `[build.cargo]` для формирования команды) |
| Phase 5d | Загрузка и проверка предкомпилированных бинарников |
| Phase 5e | Выполнение скрипта build.yx |
| Phase 5f | Интеграция yx-bindgen (поле `headers`) |

### Зависимости

- Зависит от RFC-014a (протокол реестра, для загрузки предкомпилированных артефактов)
- Зависит от crate `sha2` (проверка целостности)
