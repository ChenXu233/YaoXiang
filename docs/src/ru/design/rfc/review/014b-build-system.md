# RFC-014b: Система сборки и распространения бинарных файлов

> Настоящий RFC является под-RFC к [RFC-014: Проектирование системы управления пакетами](../accepted/014-package-manager.md).

## Аннотация

Определяет механизмы сборки системы управления пакетами YaoXiang: декларативная конфигурация сборки, стратегии сборки (cargo/cmake/custom/none), распространение предкомпилированных бинарных файлов, проверка системных зависимостей.

## Мотивация

Некоторые пакеты представляют собой чистый код `.yx`, не требующий сборки. Некоторым требуется компиляция FFI-связывания (вызов Cargo, CMake и т.д.). Необходим единый механизм, позволяющий авторам пакетов декларировать требования к сборке, а менеджеру пакетов — автоматически их обрабатывать.

### Текущие проблемы

- Отсутствует декларирование конфигурации сборки (в `yaoxiang.toml` нет секции `[build]`)
- Отсутствует механизм распространения предкомпилированных бинарных файлов
- Сборка FFI-пакетов полностью зависит от ручных действий пользователя
- Отсутствует проверка системных зависимостей

## Предложение

### Основной дизайн: декларативная сборка + приоритет предкомпиляции

Автор пакета декларирует требования к сборке в `yaoxiang.toml`, менеджер пакетов автоматически принимает решения на основе декларации.

### Стратегии сборки

```rust
enum BuildStrategy {
    None,          // Чистый .yx-пакет, сборка не требуется
    Cargo,         // Вызов cargo build, чтение конфигурации [build.cargo]
    Cmake,         // Вызов cmake
    Custom,        // Выполнение скрипта build.yx
}
```

Примечание: вариант `Precompiled` удалён. Наличие секции `[binaries]` автоматически активирует поведение приоритета предкомпиляции, явное указание strategy не требуется.

### Декларация сборки в yaoxiang.toml

```toml
[package]
name = "native-foo"
version = "1.0.0"

[build]
strategy = "cargo"              # Стратегия сборки
headers = ["include/sqlite3.h"] # Необязательно: C-заголовки, автоматически обрабатываемые yx-bindgen

[build.cargo]
features = ["ffi"]             # cargo build --features ffi
target = "release"             # cargo build --release

[build.requirements]
cargo = ">= 1.70"              # Инструменты, необходимые во время сборки
cmake = ">= 3.20"

[build.platforms]              # Переопределения для конкретных платформ
"x86_64-unknown-linux-gnu" = { cargo-features = ["linux-ffi"] }
"x86_64-pc-windows-msvc" = { cargo-features = ["win-ffi"] }
"aarch64-apple-darwin" = { cargo-features = ["mac-ffi"] }
```

### Дерево решений при установке

```
yaoxiang install foo
    │
    ├─ 1. Есть ли в [binaries] запись для текущей платформы?
    │     → Да: скачать, проверить SHA-256, установить сразу (пропустить сборку)
    │     → Нет: продолжить
    │
    ├─ 2. Скачать пакет с исходным кодом
    │
    ├─ 3. Есть ли значение в [build].headers?
    │     → Да: автоматически запустить yx-bindgen для генерации файлов связывания
    │
    ├─ 4. Прочитать [build].strategy
    │     → "none": установить сразу
    │     → "cargo": прочитать конфигурацию [build.cargo], сформировать команду cargo build
    │     → "cmake": вызвать cmake
    │     → "custom": выполнить скрипт build.yx
    │
    └─ 5. Установить в vendor/
```

**Приоритет предкомпиляции, исходный код как fallback.** Наличие секции `[binaries]` автоматически активирует проверку предкомпиляции, явное указание strategy не требуется.

### Подробности стратегии cargo

При `strategy = "cargo"` читается конфигурация `[build.cargo]` для формирования команды:

```toml
[build]
strategy = "cargo"

[build.cargo]
features = ["ffi"]             # → cargo build --features ffi
target = "release"             # → cargo build --release

[build.platforms]              # Переопределение платформы
"x86_64-unknown-linux-gnu" = { cargo-features = ["linux-ffi"] }
"x86_64-pc-windows-msvc" = { cargo-features = ["win-ffi"] }
"aarch64-apple-darwin" = { cargo-features = ["mac-ffi"] }
```

Фактически выполняемые команды:

```bash
# Базовая
cargo build --release --features ffi

# При наличии переопределения платформы (на примере linux)
cargo build --release --features ffi,linux-ffi
```

### Декларация предкомпилированных бинарных файлов

```toml
# yaoxiang.toml
[binaries]
"x86_64-unknown-linux-gnu" = { url = "releases/download/v1.0.0/foo-linux-x86_64.tar.gz", sha256 = "abc123" }
"x86_64-pc-windows-msvc" = { url = "https://example.com/foo-win-x86_64.tar.gz", sha256 = "def456" }
"aarch64-apple-darwin" = { url = "releases/download/v1.0.0/foo-macos-aarch64.tar.gz", sha256 = "ghi789" }
```

**Формат URL:** Поддерживаются абсолютные URL и относительные пути. Относительные пути задаются относительно адреса репозитория пакета (URL GitHub-репозитория или корневой URL Registry).

**Условия пропуска сборки:**
1. В `[binaries]` имеется запись для текущей платформы
2. Проверка SHA-256 прошла успешно
3. Загрузка успешно завершена

При выполнении всех трёх условий → сборка пропускается. Иначе → fallback к сборке из исходного кода.

### Скрипт сборки build.yx

При `strategy = "custom"` выполняется `build.yx`.

**Модель выполнения (минимальная спецификация):**
- Скрипт — это обычный код `.yx` с полным доступом к `std`
- Рабочая директория: корень пакета (`vendor/<pkg>-<ver>/`)
- Успех: код выхода 0
- Неудача: ненулевой код выхода, установка прерывается
- Менеджер пакетов не ограничивает поведение скрипта, проверяется только код выхода

```yx
# build.yx — скрипт сборки пакета
use std.os
use std.io

fn main() {
    let platform = os.platform()
    let arch = os.arch()

    if os.file_exists("Cargo.toml") {
        io.println("Сборка native-расширения через Cargo...")
        let result = os.exec("cargo build --release")
        if result.exit_code != 0 {
            io.println("Сборка не удалась!")
            os.exit(1)
        }
    }

    io.println("Сборка завершена!")
}
```

### Проверка системных зависимостей

Перед установкой автоматически проверяются все `[build.requirements]`, при неудовлетворении выдаётся ошибка:

```
Error: Build requirement not satisfied
  cargo >= 1.70 required, but cargo is not installed
  Install: https://rustup.rs
```

### Интеграция с yx-bindgen (поле headers)

Поле `[build].headers` декларирует C-заголовки, которые должен обработать yx-bindgen. Система сборки автоматически запускает yx-bindgen для генерации файлов связывания `.yx`.

```toml
[build]
strategy = "cargo"
headers = ["include/sqlite3.h", "include/json.h"]
```

Процесс сборки:

```
1. Есть предкомпилированный артефакт в [binaries]? → пропустить всю сборку
2. Есть значение в [build].headers? → автоматически сгенерировать связывания через yx-bindgen
3. Выполнить [build].strategy (cargo/cmake/custom)
4. Установить
```

yx-bindgen парсит сигнатуры функций и определения типов из C-заголовков (`.h`) и автоматически генерирует декларации связывания `.yx`. Пользователю не нужно запускать его вручную — система сборки автоматически обрабатывает это при обнаружении конфигурации `headers`.

**Связь с RFC-026:** RFC-026 определяет языковую семантику `yx-bindgen` (синтаксис `native("symbol")`, unsafe-типы). RFC-014b определяет способ его интеграции в процесс сборки (конфигурация `headers`). Они дополняют друг друга.

### Интеграция с Cargo Workspace

Если в пакете есть FFI-код, можно одновременно определить Cargo workspace:

```
my-package/
├── yaoxiang.toml          # Конфигурация пакета YaoXiang
├── Cargo.toml             # Cargo workspace (FFI-часть)
├── src/
│   └── lib.yx             # Код YaoXiang
└── native/
    ├── Cargo.toml          # Rust FFI-код
    └── src/
        └── lib.rs
```

`yaoxiang build` автоматически обнаруживает и вызывает `cargo build` для компиляции native-части.

## Детальное проектирование

### Идентификаторы платформ

Используется формат Rust target triple (`arch-vendor-os-env`):

| Платформа | Идентификатор |
|------|------|
| Linux x86_64 (glibc) | `x86_64-unknown-linux-gnu` |
| Linux x86_64 (musl) | `x86_64-unknown-linux-musl` |
| Linux ARM64 | `aarch64-unknown-linux-gnu` |
| Windows x86_64 (MSVC) | `x86_64-pc-windows-msvc` |
| Windows x86_64 (MinGW) | `x86_64-pc-windows-gnu` |
| macOS ARM64 | `aarch64-apple-darwin` |
| macOS x86_64 | `x86_64-apple-darwin` |

Использование Rust target triple вместо упрощённого формата обусловлено:
1. Различением ABI в пределах одной ОС (gnu vs musl, msvc vs gnu)
2. Совпадением с экосистемой Rust/Cargo, что снижает количество ошибок маппинга
3. Отсутствием необходимости менять формат при будущем расширении

### Структура каталога артефактов сборки

```
build/
└── native/
    ├── x86_64-unknown-linux-gnu/
    │   └── libfoo.so
    ├── x86_64-pc-windows-msvc/
    │   └── foo.dll
    └── aarch64-apple-darwin/
        └── libfoo.dylib
```

### Полный жизненный цикл предкомпилированного пакета

```
Разработчик:
  1. Пишет .yx-код + FFI-связывания
  2. Декларирует [build] + [binaries] в yaoxiang.toml
  3. yaoxiang publish
     → Автоматическая сборка мультиплатформенных бинарных файлов в CI
     → Загрузка исходного кода + предкомпилированных артефактов

Пользователь:
  yaoxiang add native-foo
    → Обнаружены предкомпилированные артефакты → скачать сразу (секунды)
    → Нет предкомпилированных артефактов → скачать исходный код + выполнить сборку (минуты)
```

## Компромиссы

### Преимущества

- Декларативная конфигурация, пользователю не нужно понимать детали сборки
- Приоритет предкомпиляции, очень быстрая установка
- Поддержка нескольких платформ, автоматический выбор
- Бесшовная интеграция с экосистемой Cargo

### Недостатки

- Для предкомпилированных артефактов требуется поддержка CI
- Мультиплатформенная сборка усложняет публикацию
- Для скриптов build.yx требуется механизм песочницы

## Альтернативы

| Решение | Почему не выбрано |
|------|-----------|
| Распространение только исходного кода | Пользователю нужно установить инструментарий сборки, высокий порог входа |
| Бинарный формат наподобие Python wheel | Слишком сложно, экосистеме YaoXiang на ранней стадии не нужно |
| Отсутствие поддержки FFI-сборки | Ограничивает возможности расширения языка |

## Стратегия реализации

### Разбиение на этапы

| Этап | Содержание |
|------|------|
| Phase 5a | Разбор конфигурации `[build]` + enum `BuildStrategy` |
| Phase 5b | Проверка системных зависимостей |
| Phase 5c | Интеграция сборки Cargo (формирование команды по `[build.cargo]`) |
| Phase 5d | Загрузка + проверка предкомпилированных бинарных файлов |
| Phase 5e | Выполнение скрипта build.yx |
| Phase 5f | Интеграция yx-bindgen (поле `headers`) |

### Зависимости

- Зависит от RFC-014a (протокол Registry, для загрузки предкомпилированных артефактов)
- Зависит от crate `sha2` (проверка целостности)
