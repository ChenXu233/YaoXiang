# RFC-030: механизм assert-утверждений

## Резюме

В YaoXiang вводится механизм `assert`-утверждений для тестирования, проверки предусловий и runtime panic. `assert` и компилируемый уточняющий тип `Assert(C)` (см. RFC-011 §4.3) — это **две стороны одного и того же уточняющего примитива** — dispatch автоматически распределяет их на компилируемое доказательство или runtime-проверку в зависимости от того, доступны ли свободные переменные предиката во время компиляции. `assert(false, "msg")` эквивалентно `raise`, отдельные ключевые слова `throw`/`raise` не требуются.

## Мотивация

### Зачем нужна эта возможность?

В текущей реализации YaoXiang сквозное (E2E) тестирование может имитировать утверждения только через `if` + `io.println` + `return`:

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

Такой способ записи имеет три проблемы:

1. **Много шаблонного кода**: каждое утверждение требует 4 строки, файлы тестов раздуваются
2. **Слабые сообщения об ошибках**: ручная конкатенация строк, отсутствует информация о позиции в исходном коде
3. **Некомпозируемость**: невозможно пакетно регистрировать утверждения, невозможно передать их как аргументы в тестовый фреймворк

### Текущие проблемы

- Отсутствует единый механизм утверждений
- Тестовый код засорен шаблоном `if` + печать + `return`
- На уровне байткода уже есть инструкция `Throw`, но на уровне языка она не доступна
- RFC-011 определяет компилируемый условный тип `Assert(C)`, но runtime-функция `assert()` ещё не реализована

### Принципы проектирования

`assert` — это единственный пользовательский механизм panic в YaoXiang. `assert(false, "msg")` эквивалентно `raise`, отдельные ключевые слова `throw`/`raise` не требуются. Функция `assert` сама по себе является наилучшей инкапсуляцией `if raise`.

**Никаких новых ключевых слов, никакого нового синтаксиса. Всё — вызов функции.**

## Вариант A: native-функция

Реализовать `assert` как native-функцию без введения новых ключевых слов.

```yaoxiang
use std.assert.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Перегруженные сигнатуры

`assert` имеет две перегрузки:

```
// Основная сигнатура: assert — это вводящий оператор значения для Assert
assert: (cond: Bool, ?msg: String | Error) -> Assert(IsTrue(cond))
//                                       ^^^^^^^^^^^^^^^^^^^^^^^^
//                                       Возвращает уточняющий тип, а не ()
//
// IsTrue: Bool -> Type — это мост от значения истинности к типу:
//   IsTrue(true)  = Void   (⊤, программа продолжается)
//   IsTrue(false) = Never  (⊥, расходимость/ошибка компиляции)
```

Фактическое поведение `assert` определяется механизмом dispatch:
- Все свободные переменные известны на этапе компиляции → **CompileTime**: компилятор вычисляет cond, true → стирается до Void, false → ошибка компиляции (Never нельзя населить)
- Есть свободные переменные времени выполнения → **Runtime**: вставляется проверка, в flow-sensitive набор допущений Γ добавляется уточняющий факт

Необязательное сообщение `?msg` и перегрузки для Result (см. ниже) сохраняются как полезная нагрузка для runtime-raise.

#### Перегрузка 1: утверждение по условию `(Bool, ?String | Error)`

`Bool` + необязательное сообщение. Сообщение может быть значением `String` или `Error`:

```yaoxiang
assert(1 + 1 == 2)                    // Без сообщения, panic-информация по умолчанию
assert(1 + 1 == 2, "math is broken")   // Строковое сообщение
assert(x > 0, my_error)                // Прямой выброс значения Error
```

`assert(false, "msg")` — это эквивалент `raise`/`throw` в YaoXiang — отдельное ключевое слово не требуется.

#### Перегрузка 2: утверждение для `Result` `(Result)`

Один аргумент `Result`, автоматическая проверка на `Err`:

### Достоинства

- **Нулевые изменения синтаксиса**: чистая функция, новые ключевые слова не нужны
- **Нулевые новые концепции**: повторное использование существующего механизма регистрации native-функций
- **Высокая расширяемость**: перегрузка функций естественно поддерживает несколько сигнатур
- **Самодокументируемость**: пространство имён `std.assert` само по себе является документацией

### Недостатки

- Отсутствуют. Когда сигнатура типа `assert` корректна, компилятор может вывести мёртвый код через анализ достижимости функций. Дополнительные проходы не требуются.

### Поведение во время выполнения

1. Вычислить первый аргумент `condition: Bool`
2. Если `true` — вернуть `Unit`
3. Если `false` — инициировать runtime panic:
   - Вывести содержимое `message` (если задано)
   - Вывести стек вызовов (в debug-режиме)
   - Завершить текущее выполнение

#### Поведение при сбое для каждой перегрузки

| Сигнатура | Поведение при сбое |
|-----------|--------------------|
| `assert(false)` | Сообщение panic по умолчанию |
| `assert(false, "msg")` | Вывод строкового сообщения, затем panic |
| `assert(false, error_val)` | Выброс значения Error |
| `assert(Err(x))` | Извлечение содержимого Err и panic |

### Связь с компилируемым Assert

`assert` и `Assert` — это **две стороны одного и того же уточняющего примитива** — конвейер dispatch автоматически выбирает одну из них в зависимости от того, доступны ли свободные переменные предиката на этапе компиляции:

| Условие | Распределение | Поведение |
|---------|---------------|-----------|
| Все свободные переменные известны на этапе компиляции | CompileTime → конвейер доказательств | Proved → стирается, Disproved → ошибка компиляции, Unknown → требуется доказательство |
| Есть свободные переменные времени выполнения | Runtime → вставка check | Проверка Bool + добавление уточняющего факта в flow-sensitive набор допущений Γ |

```yaoxiang
use std.assert

# Известно на этапе компиляции (параметр дженерика) — путь CompileTime, нулевые runtime-затраты
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: assert.Assert(N > 0),   # N — параметр дженерика, вычисляется на этапе компиляции
}

# Значение времени выполнения — путь Runtime, вставка проверки Bool
x = read_int()
assert.assert(x > 0, "expected positive")  # runtime check
```

> **Унифицированное решение от 2026-07-12**: прежний вывод о «полной независимости» заменён. `assert()` — это вводящий оператор значения для `Assert`, dispatch выполняет автоматическое распределение.

### Изменения в компиляторе

**Изменения в parser, AST, typecheck, IR gen не требуются.**

Достаточно добавить регистрацию native-функций в `src/std/`:

1. Добавить `src/std/assert.rs`
2. Зарегистрировать `std.assert.assert` и `std.assert.Assert` (последний — компилируемый условный тип, см. #155)
3. Внутри вызвать существующую инструкцию `BytecodeInstr::Throw`

### Достоинства

- **Нулевые изменения синтаксиса**: чистая функция, новые ключевые слова не нужны
- **Нулевые новые концепции**: повторное использование существующего механизма регистрации native-функций
- **Высокая расширяемость**: сигнатуры функций можно расширить до вариантов `assert_eq` и т.п. (в будущем)
- **Самодокументируемость**: пространство имён `std.assert` само по себе является документацией

### Недостатки

- ~~Неизвестно на этапе компиляции: в отличие от варианта B (ключевое слово), невозможно выполнить удаление мёртвого кода на этапе компиляции~~ → **Больше не актуально в рамках унифицированного решения**. `assert` в режиме CompileTime идёт через конвейер доказательств; cond, известный на этапе компиляции → стирается или приводит к ошибке компиляции (`assert(false)` → Never → мёртвый код).
- Стек вызовов доступен только в debug-режиме

## Вариант B: встроенное ключевое слово (заменён унифицированным решением)

> Устарело. Противоречие между вариантами A и B снимается конвейером dispatch — `assert` является вводящим оператором значения для `Assert`, известное на этапе компиляции идёт через конвейер доказательств (нулевые runtime-затраты), время выполнения идёт через check. Нет необходимости выбирать между «функцией» и «ключевым словом». Ниже — историческая запись.

```yaoxiang
assert(1 + 1 == 2, "math is broken")
```

### Сигнатура типа

Нет отдельной сигнатуры типа — ключевое слово обрабатывается парсером.

### Поведение во время выполнения

Аналогично варианту A.

### Изменения в компиляторе

Требуются изменения в parser, AST, typecheck, IR gen:
1. parser: добавить вариант `Expr::Assert`
2. AST: добавить узел `Expr::Assert`
3. typecheck: проверить типы аргументов
4. IR gen: сгенерировать `BytecodeInstr::Throw`

### Достоинства

- Позиция в исходном коде известна на этапе компиляции (не зависит от debug-информации)
- Возможна свёртка констант на этапе компиляции: `assert(true)` → пустая операция, `assert(false)` → ошибка компиляции

### Недостатки

| Недостаток | Влияние |
|------------|---------|
| Требуются изменения в парсере | Введение нового узла синтаксиса, увеличение стоимости поддержки |
| Ключевое слово не расширяемо | Для вариантов типа `assert_eq` всё равно нужны функции |
| Преимущества на этапе компиляции не практичны | См. анализ ниже |

### Сравнение

| Аспект | Вариант A (функция) | Вариант B (ключевое слово) |
|--------|---------------------|-----------------------------|
| Стоимость реализации | ~20 строк | parser + AST + typecheck + IR gen |
| Изменения синтаксиса | Нет | Новое ключевое слово |
| Расширяемость | Перегрузка функций | Требуются сопутствующие макросы |
| Позиция в исходном коде | debug-информация | Доступна на этапе компиляции |
| Свёртка констант | Требуется поддержка pass | Доступна на этапе компиляции |
| Runtime-затраты | Вызов функции | Минимальны |

### Реалистичные ограничения анализа на этапе компиляции

Ключевое преимущество варианта B — анализ на этапе компиляции — требует **прохода свёртки констант**, чтобы начать работать. То есть компилятору нужно вычислить `false` в `assert(false)` на этапе компиляции, чтобы узнать, что это мёртвый код.

В текущей версии YaoXiang нет прохода свёртки констант. Даже при использовании варианта B частые случаи вроде `assert(x > 0)` всё равно не могут быть проанализированы на этапе компиляции. Анализу поддаются только литералы вроде `assert(true)` / `assert(false)`.

Таким образом, преимущества варианта B на этапе компиляции **на текущем этапе являются теоретическими, а не практическими**.

### 2026-07-05: выбор варианта A (заменён унифицированным решением)

Реализация варианта A в 20 строк побеждает по соотношению ценности и стоимости. После определения унифицированного решения 2026-07-12 противоречие между A и B снимается конвейером dispatch — `assert` является вводящим оператором значения для `Assert`, и выбор между «функцией» и «ключевым словом» больше не стоит.

### 2026-07-12: определено унифицированное решение (заменяет вывод от 2026-07-11 о «полной независимости»)

**Вывод**: `assert` и `Assert` — не два независимых механизма. `assert: (Bool) -> Assert(IsTrue(cond))` — автоматическое распределение через dispatch:

- Известно на этапе компиляции → конвейер доказательств (Proved стирается / Disproved ошибка / Unknown требует доказательства)
- Входные данные времени выполнения → вставка check + добавление допущения в Γ

**Структура модуля**: `std.assert` единообразно несёт runtime-утверждения (`assert`) и компилируемые уточняющие типы (`Assert`, `IsTrue`). Больше не «раздельная реализация», а две стороны одного примитива.

### 2026-07-11: проектирование перегрузок `assert`

**Вопрос**: зачем `assert` нужны две перегрузки, а не единая `(Bool, ?String)`?

**Ответ**:

Runtime-функция `assert()` — единственный пользовательский механизм panic в YaoXiang. `assert(false, "msg")` эквивалентно `raise`/`throw` в других языках. Поэтому она должна покрывать три сценария:
1. Условие + простое сообщение: `assert(cond, "msg")`
2. Условие + пользовательский Error: `assert(cond, my_error)`
3. Проверка Result: `assert(result)` — самая краткая запись `if is_err { panic }`

Разумность перегрузки для Result в том, что это кратчайший путь распространения ошибки — «Result должен быть Ok, иначе смерть». Не нужно сначала вызывать `.is_ok()`, а затем отдельно обрабатывать ошибку.
