# RFC-030: Механизм утверждений assert

## Резюме

Ввести в YaoXiang механизм утверждений `assert` для тестирования и проверки предусловий. Данный RFC предлагает два варианта реализации — **нативная функция** и **ключевое слово** — анализирует компромиссы каждого и выносит их на решение.

## Мотивация

### Зачем нужна эта функциональность?

В настоящее время E2E-тесты YaoXiang могут имитировать утверждения только через связку `if` + `io.println` + `return`:

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

Такой способ записи имеет три проблемы:

1. **Много шаблонного кода**: каждое утверждение занимает 4 строки, тестовые файлы раздуваются
2. **Слабые сообщения об ошибках**: ручная конкатенация строк, отсутствие позиции в исходном коде
3. **Некомпозируемость**: нет возможности массово регистрировать утверждения или передавать их в качестве аргументов в тестовый фреймворк

### Текущая проблема

- Отсутствует единый механизм утверждений
- Тестовый код наводнён шаблоном `if` + печать + `return`
- На уровне байт-кода уже есть инструкция `Throw`, но на уровне языка она не доступна

## Вариант A: нативная функция

Реализовать `assert` как нативную функцию, не вводя новое ключевое слово.

```yaoxiang
use std.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Сигнатура типа

```
assert: (Bool, String) -> ()
```

### Поведение во время выполнения

1. Вычислить первый аргумент `condition: Bool`
2. Если `true`, вернуть `Unit`
3. Если `false`, вызвать панику времени выполнения:
   - Вывести содержимое `message`
   - Вывести стек вызовов (в режиме отладки)
   - Прервать текущее выполнение

### Изменения в компиляторе

**Не требуется менять parser, AST, typecheck, IR gen.**

Достаточно добавить регистрацию нативной функции в каталоге `src/std/`:

1. Добавить файл `src/std/assert.rs`
2. Зарегистрировать нативную функцию `std.assert`
3. Внутри задействовать уже имеющийся механизм паники времени выполнения

### Преимущества

- **Нулевые изменения синтаксиса**: не нужно менять parser, AST, IR, codegen
- **Простота реализации**: около 20 строк кода регистрации нативной функции
- **Полноправный гражданин**: можно передавать в качестве аргумента, присваивать переменной, использовать в функциях высшего порядка
- **Постепенное наращивание**: в будущем можно добавить оптимизации встраивания на этапе компиляции, не ломая существующий код

### Недостатки

- **Нет анализа на этапе компиляции**: компилятор не может выполнить удаление мёртвого кода после `assert(false)`
- **Ограниченные сообщения об ошибках**: позиция в исходном коде извлекается из стека вызовов во время выполнения, что менее точно, чем у ключевого слова
- **Возможность затенения**: пользователь может определить одноимённую функцию, перекрывающую `assert`
- **Требуется импорт**: необходимо писать `use std.assert`, чтобы воспользоваться

## Вариант B: ключевое слово

Ввести `assert` как ключевое слово языка, обрабатываемое компилятором на уровне IR особым образом.

```yaoxiang
main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Синтаксис

```
assert '(' Expr ',' StringLit ')'
```

`assert` является ключевым словом и не требует импорта через `use`.

### Изменения в компиляторе

Необходимо изменить всю цепочку:

| Этап | Изменение |
|------|-----------|
| Lexer | Добавить токен `KwAssert` |
| Parser | Добавить синтаксис `assert(expr, "msg")` |
| AST | Добавить `StmtKind::Assert { condition, message }` |
| IR Gen | Компилировать Assert в условный переход + Throw |
| Codegen | Генерировать последовательность инструкций `JmpIfNot` + `Throw` |

### Преимущества

- **Доступно из коробки**: не требует импорта, доступно в любом месте
- **Точная позиция в исходном коде**: компилятор знает позицию assert на уровне AST, сообщения об ошибках точные до строки и столбца
- **Невозможность затенения**: ключевое слово нельзя перекрыть пользовательским кодом
- **Расширяемость в будущем**: можно выполнять анализ на этапе компиляции (в связке с проходом свёртки констант)
- **Однозначная семантика**: `assert` имеет явную семантику на уровне синтаксиса, что исключает неправильное использование

### Недостатки

- **Высокая стоимость реализации**: требуется изменить parser, AST, IR, codegen во всей цепочке
- **Не полноправный гражданин**: нельзя передавать в качестве аргумента, нельзя присваивать переменной
- **Раздувание ключевых слов**: каждое новое ключевое слово увеличивает сложность языка
- **Плохая расширяемость**: добавление новых вариантов сигнатуры (например, `assert_eq`) требует изменения синтаксиса

## Сравнение

| Измерение | Вариант A: функция | Вариант B: ключевое слово |
|-----------|--------------------|---------------------------|
| Стоимость реализации | Низкая (~20 строк) | Высокая (~200 строк, 4 модуля) |
| Изменения синтаксиса | Нет | Да (новое ключевое слово + синтаксис) |
| Требуется import | Да | Нет |
| Полноправный гражданин | Да | Нет |
| Точность позиции в исходном коде | Стек вызовов в runtime | Точно до строки и столбца на этапе компиляции |
| Невозможность затенения | Нет | Да |
| Анализ на этапе компиляции | Невозможен (нужна свёртка констант) | Возможен (в связке со свёрткой констант) |
| Расширяемость | Высокая (перегрузка, функции высшего порядка) | Низкая (нужно менять синтаксис) |

### Практические ограничения анализа на этапе компиляции

Ключевое преимущество варианта B — анализ на этапе компиляции — требует наличия **прохода свёртки констант** (constant folding pass). То есть компилятор должен уметь вычислять `false` в `assert(false)` на этапе компиляции, чтобы определить, что это мёртвый код.

В YaoXiang в настоящее время нет прохода свёртки констант. Даже при выборе варианта B распространённые случаи вроде `assert(x > 0)` всё равно не могут быть проанализированы на этапе компиляции. Анализу поддаются только литералы вроде `assert(true)` / `assert(false)`.

Поэтому преимущество варианта B в плане анализа на этапе компиляции **на текущей стадии теоретическое, а не практическое**.
