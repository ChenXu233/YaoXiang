# RFC-030: механизм assert утверждений

## Резюме

В YaoXiang вводится механизм утверждений `assert` для тестирования, проверки предусловий и panic во время выполнения. `assert` и тип-精化 на этапе компиляции `Assert(C)` (см. RFC-011 §4.3) представляют собой **две стороны одного и того же 精化-примитива** — диспетчеризация автоматически направляет их в доказательство на этапе компиляции или в проверку на этапе выполнения в зависимости от того, доступны ли свободные переменные предиката на этапе компиляции. `assert(false, "msg")` эквивалентно `raise` и не требует отдельного ключевого слова `throw`/`raise`.

## Мотивация

### Зачем нужна эта возможность?

В настоящее время E2E-тесты YaoXiang могут лишь имитировать утверждения через `if` + `io.println` + `return`:

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

Такой подход имеет три проблемы:

1. **Много шаблонного кода**: каждое утверждение требует 4 строки, файлы тестов раздуваются
2. **Слабые сообщения об ошибках**: ручная конкатенация строк, отсутствие позиции в исходном коде
3. **Невозможность композиции**: нельзя зарегистрировать утверждения пакетно, нельзя передать их как аргумент в тестовый фреймворк

### Текущие проблемы

- Нет единого механизма утверждений
- Тестовый код наводнён шаблоном `if` + печать + `return`
- На уровне байт-кода уже есть инструкция `Throw`, но на уровне языка она не暴露
- RFC-011 определяет условный тип `Assert(C)` на этапе компиляции, но `assert()` на этапе выполнения ещё не реализован

### Принципы проектирования

`assert` — это единственный пользовательский механизм panic в YaoXiang. `assert(false, "msg")` эквивалентно `raise`, отдельное ключевое слово `throw`/`raise` не требуется. Сама функция `assert` — это наилучшая инкапсуляция `if raise`.

**Никаких новых ключевых слов, никакого нового синтаксиса. Всё — вызовы функций.**

## Вариант A: нативная функция

Реализовать `assert` как нативную функцию, не вводя новых ключевых слов.

```yaoxiang
use std.assert.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Перегруженные сигнатуры

`assert` имеет две перегрузки:

```
// Основная сигнатура: assert — это введение значения для Assert
assert: (cond: Bool, ?msg: String | Error) -> Assert(IsTrue(cond))
//                                       ^^^^^^^^^^^^^^^^^^^^^^^^
//                                       Возвращает 精化-тип, а не ()
//
// IsTrue: Bool -> Type — мост от значения истинности к типу:
//   IsTrue(true)  = Void   (⊤, программа продолжается)
//   IsTrue(false) = Never  (⊥, расходимость / ошибка компиляции)
```

Фактическое поведение `assert` определяется диспетчеризацией:
- Все свободные переменные известны на этапе компиляции → **CompileTime**: компилятор вычисляет cond, true → стирается в Void, false → ошибка компиляции (Never необитаем)
- Есть свободные переменные этапа выполнения → **Runtime**: вставляется проверка, в потоко-чувствительное множество предположений Γ注入ляется 精化-факт

Необязательное сообщение `?msg` и перегрузка Result (см. ниже) сохраняются как полезная нагрузка raise на этапе выполнения.

#### Перегрузка 1: утверждение условия `(Bool, ?String | Error)`

`Bool` + необязательное сообщение. Сообщение может быть значением `String` или `Error`:

```yaoxiang
assert(1 + 1 == 2)                    // без сообщения, информация о panic по умолчанию
assert(1 + 1 == 2, "math is broken")   // строковое сообщение
assert(x > 0, my_error)                // непосредственно бросает значение Error
```

`assert(false, "msg")` — это эквивалент `raise`/`throw` в YaoXiang — отдельное ключевое слово не требуется.

#### Перегрузка 2: утверждение Result `(Result)`

Один аргумент `Result`, автоматически проверяется, не является ли он `Err`:

### Достоинства

- **Нулевые изменения синтаксиса**: чистая функция, без новых ключевых слов
- **Нулевые новые концепции**: повторное использование существующего механизма регистрации нативных функций
- **Высокая расширяемость**: перегрузка функций естественно поддерживает несколько сигнатур
- **Самодокументирование**: пространство имён `std.assert` само по себе является документацией

### Недостатки

- Отсутствуют. Когда сигнатура типа `assert` корректна, компилятор может вывести мёртвый код через анализ достижимости функций. Дополнительные проходы не требуются.

### Поведение во время выполнения

1. Вычислить первый аргумент `condition: Bool`
2. Если `true`, вернуть `Unit`
3. Если `false`, вызвать panic на этапе выполнения:
   - Вывести содержимое `message` (если задано)
   - Вывести стек вызовов (в отладочном режиме)
   - Завершить текущее выполнение

#### Поведение при сбое для каждой перегрузки

| Сигнатура | Поведение при сбое |
|------|-----------|
| `assert(false)` | Информация о panic по умолчанию |
| `assert(false, "msg")` | Вывести строковое сообщение, затем panic |
| `assert(false, error_val)` | Бросить значение Error |
| `assert(Err(x))` | Извлечь содержимое Err и выполнить panic |

### Связь с компиляционным Assert

`assert` и `Assert` — это **две стороны одного и того же 精化-примитива** — конвейер диспетчеризации автоматически выбирает один из них по критерию «доступны ли свободные переменные предиката на этапе компиляции»:

| Условие | Диспетчеризация | Поведение |
|------|------|------|
| Все свободные переменные известны на этапе компиляции | CompileTime → конвейер доказательств | Proved → стереть, Disproved → ошибка компиляции, Unknown → требуется доказательство |
| Есть свободные переменные этапа выполнения | Runtime → вставить проверку | Проверка Bool +注入ление 精化-факта в потоко-чувствительное множество предположений Γ |

```yaoxiang
use std.assert

# Известно на этапе компиляции (泛型-параметр) — идёт по CompileTime, нулевые накладные расходы
Array: (T: Type, N: Int) -> Type = {
    data: Array(T, N),
    length: assert.Assert(N > 0),   # N — это 泛型-параметр, вычисляется на этапе компиляции
}

# Значение этапа выполнения — идёт по Runtime, вставляется проверка Bool
x = read_int()
assert.assert(x > 0, "expected positive")  # проверка во время выполнения
```

> **2026-07-12 Единое решение**: прежний вывод о «полной независимости» заменён. `assert()` — это введение значения для `Assert`, диспетчеризация выполняется автоматически.

### Изменения в компиляторе

**Изменения в parser, AST, typecheck, IR gen не требуются.**

Достаточно лишь добавить регистрацию нативной функции в `src/std/`:

1. Добавить `src/std/assert.rs`
2. Зарегистрировать `std.assert.assert` и `std.assert.Assert` (последний — это условный тип на этапе компиляции, см. #155)
3. Внутри вызвать уже существующую инструкцию `BytecodeInstr::Throw`

### Достоинства

- **Нулевые изменения синтаксиса**: чистая функция, без новых ключевых слов
- **Нулевые новые концепции**: повторное использование существующего механизма регистрации нативных функций
- **Высокая расширяемость**: сигнатуру функции можно расширить до вариантов `assert_eq` и т.п. (в будущем)
- **Самодокументирование**: пространство имён `std.assert` само по себе является документацией

### Недостатки

- ~~Неизвестно на этапе компиляции: в отличие от варианта B (ключевое слово), невозможно выполнить устранение мёртвого кода на этапе компиляции~~ → **В рамках единого решения более неактуально**. assert в режиме CompileTime идёт по конвейеру доказательств, cond, известный на этапе компиляции → стирается или вызывает ошибку компиляции (`assert(false)` → Never → мёртвый код).
- Стек вызовов доступен только в отладочном режиме

## Вариант B: встроенное ключевое слово (заменено единым решением)

> Устарело. Противоречие между вариантами A и B снято конвейером диспетчеризации — assert является введением значения для Assert, при известности на этапе компиляции идёт по конвейеру доказательств (нулевые накладные расходы), во время выполнения — по check. Выбирать между «функцией» и «ключевым словом» не нужно. Ниже приведена историческая запись.

```yaoxiang
assert(1 + 1 == 2, "math is broken")
```

### Сигнатура типа

Нет отдельной сигнатуры типа — ключевое слово обрабатывается парсером.

### Поведение во время выполнения

Аналогично варианту A.

### Изменения в компиляторе

Требуются изменения в parser, AST, typecheck, IR gen:
1. parser: добавить вариант `Expr::Assert`
2. AST: добавить узел `Expr::Assert`
3. typecheck: проверить типы аргументов
4. IR gen: сгенерировать `BytecodeInstr::Throw`

### Достоинства

- Позиция в исходном коде известна на этапе компиляции (не зависит от отладочной информации)
- Возможна свёртка констант на этапе компиляции: `assert(true)` → пустая операция, `assert(false)` → ошибка компиляции

### Недостатки

| Недостаток | Влияние |
|------|------|
| Требуются изменения в парсере | Вводится новый синтаксический узел, возрастают затраты на сопровождение |
| Ключевое слово нерасширяемо | Варианты вроде `assert_eq` всё равно требуют функций |
| Преимущества этапа компиляции нереалистичны | См. анализ ниже |

### Сравнение

| Измерение | Вариант A (функция) | Вариант B (ключевое слово) |
|------|---------------|-----------------|
| Стоимость реализации | ~20 строк | parser + AST + typecheck + IR gen |
| Изменения синтаксиса | Нет | Новое ключевое слово |
| Расширяемость | Перегрузка функций | Требуется сопутствующий макрос |
| Позиция в исходном коде | отладочная информация | доступна на этапе компиляции |
| Свёртка констант | требуется поддержка прохода | доступна на этапе компиляции |
| Накладные расходы выполнения | вызов функции | минимальны |

### Реалистичные ограничения анализа на этапе компиляции

Ключевое преимущество варианта B — анализ на этапе компиляции — требует **прохода свёртки констант**, чтобы начать работать. То есть компилятору необходимо вычислить `false` в `assert(false)` на этапе компиляции, чтобы распознать это как мёртвый код.

В YaoXiang в настоящее время нет прохода свёртки констант. Даже при использовании варианта B типичная запись `assert(x > 0)` по-прежнему не может быть проанализирована на этапе компиляции. Анализу поддаются только литералы вида `assert(true)` / `assert(false)`.

Таким образом, преимущества этапа компиляции у варианта B **на текущем этапе теоретические, а не практические**.

### 2026-07-05: выбор варианта A (заменён единым решением)

Реализация варианта A в 20 строк победила по соотношению ценности и стоимости. После определения единого решения 2026-07-12 противостояние вариантов A/B снято конвейером диспетчеризации — assert является введением значения для Assert, выбор между «функцией» и «ключевым словом» более не стоит.

### 2026-07-12: определено единое решение (заменяет вывод от 2026-07-11 о «полной независимости»)

**Заключение**: `assert` и `Assert` — не два независимых механизма. `assert: (Bool) -> Assert(IsTrue(cond))` — диспетчеризация выполняется автоматически:

- Известно на этапе компиляции → попадает в конвейер доказательств (Proved стирается / Disproved — ошибка / Unknown — требуется доказательство)
- Входные данные этапа выполнения → вставляется check +注入假设 в Γ

**Структура модуля**: `std.assert` единообразно несёт утверждения времени выполнения (`assert`) и компиляционные 精化-типы (`Assert`, `IsTrue`). Больше не «раздельная реализация», а две стороны одного примитива.

### 2026-07-11: проектирование перегрузок assert

**Вопрос**: зачем `assert` нужны две перегрузки вместо единой `(Bool, ?String)`?

**Ответ**:

`assert()` на этапе выполнения — это единственный пользовательский механизм panic в YaoXiang. `assert(false, "msg")` эквивалентно `raise`/`throw` в других языках. Поэтому он должен покрывать три сценария:
1. Условие + простое сообщение: `assert(cond, "msg")`
2. Условие + пользовательский Error: `assert(cond, my_error)`
3. Проверка Result: `assert(result)` — самая короткая форма `if is_err { panic }`

Обоснованность перегрузки Result состоит в том, что это кратчайший путь распространения ошибки — «Result должен быть Ok, иначе смерть». Не нужно сначала вызывать `.is_ok()`, а затем отдельно обрабатывать ошибку.
