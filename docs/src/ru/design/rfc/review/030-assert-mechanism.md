# RFC-030: механизм утверждений assert

## Резюме

Введение в YaoXiang механизма утверждений `assert` для тестирования и проверки предусловий. Настоящий RFC предлагает два варианта реализации — **native функцию** и **ключевое слово** — с анализом компромиссов каждого из них для принятия решения.

## Мотивация

### Зачем нужна эта возможность?

В настоящее время E2E-тесты YaoXiang могут имитировать утверждения только через комбинацию `if` + `io.println` + `return`:

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

Такой способ записи порождает три проблемы:

1. **Много шаблонного кода**: каждое утверждение требует 4 строк, файлы тестов раздуваются.
2. **Слабые сообщения об ошибках**: ручная конкатенация строк, отсутствует позиция в исходном коде.
3. **Некомпозируемость**: невозможно пакетно регистрировать утверждения, нельзя передавать их в качестве аргумента фреймворку тестирования.

### Текущая проблема

- Отсутствует единый механизм утверждений.
- Тестовый код наводнён шаблонами `if` + печать + `return`.
- На уровне байт-кода уже есть инструкция `Throw`, но на уровне языка она не доступна.

## Вариант A: native функция

Реализовать `assert` как native функцию, без введения нового ключевого слова.

```yaoxiang
use std.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Сигнатура типа

```
assert: (Bool, String) -> ()
```

### Поведение во время выполнения

1. Вычислить первый аргумент `condition: Bool`.
2. Если `true` — вернуть `Unit`.
3. Если `false` — вызвать panic во время выполнения:
   - вывести содержимое `message`;
   - вывести стек вызовов (в режиме отладки);
   - завершить текущее исполнение.

### Изменения в компиляторе

**Не требуется менять parser, AST, typecheck, IR gen.**

Требуется лишь добавить регистрацию native функции в `src/std/`:

1. Добавить файл `src/std/assert.rs`.
2. Зарегистрировать native функцию `std.assert`.
3. Внутри вызвать уже существующий механизм panic времени выполнения.

### Достоинства

- **Нулевые изменения синтаксиса**: не нужно менять parser, AST, IR, codegen.
- **Простая реализация**: около 20 строк кода регистрации native функции.
- **Полноправный объект первого класса**: можно передавать в качестве аргумента, присваивать переменной, использовать в функциях высшего порядка.
- **Постепенное улучшение**: в будущем можно добавить оптимизацию инлайнинга на этапе компиляции, не нарушая существующий код.

### Недостатки

- **Отсутствие анализа на этапе компиляции**: компилятор не может выполнить удаление мёртвого кода после `assert(false)`.
- **Ограниченные сообщения об ошибках**: позиция в исходном коде получается через стек вызовов во время выполнения, что менее точно, чем у ключевого слова.
- **Возможность затенения**: пользователь может определить одноимённую функцию, перекрывающую `assert`.
- **Требуется import**: для использования необходимо писать `use std.assert`.

## Вариант B: ключевое слово

Ввести `assert` как ключевое слово языка; компилятор выполняет специальную обработку на уровне IR.

```yaoxiang
main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Синтаксис

```
assert '(' Expr ',' StringLit ')'
```

`assert` — ключевое слово, не требующее импорта через `use`.

### Изменения в компиляторе

Требуются изменения по всей цепочке:

| Этап | Изменения |
|------|------|
| Lexer | Добавить токен `KwAssert` |
| Parser | Добавить синтаксис `assert(expr, "msg")` |
| AST | Добавить `StmtKind::Assert { condition, message }` |
| IR Gen | Компилировать Assert в условный переход + Throw |
| Codegen | Генерировать последовательность инструкций `JmpIfNot` + `Throw` |

### Достоинства

- **Встроенная доступность**: не требуется import, доступно из любого места.
- **Точная позиция в исходном коде**: компилятор знает позицию assert на уровне AST, сообщения об ошибках точно указывают строку и столбец.
- **Невозможность затенения**: ключевое слово не может быть перекрыто пользовательским кодом.
- **Возможность расширения в будущем**: можно выполнять анализ на этапе компиляции (в связке с pass свёртки констант).
- **Ясная семантика**: `assert` имеет явную семантику на уровне синтаксиса, что исключает неправильное использование.

### Недостатки

- **Высокая стоимость реализации**: требуется менять parser, AST, IR, codegen по всей цепочке.
- **Не полноправный объект первого класса**: нельзя передавать в качестве аргумента, нельзя присваивать переменной.
- **Разрастание ключевых слов**: каждое новое ключевое слово увеличивает сложность языка.
- **Плохая расширяемость**: для добавления вариантов сигнатур (например, `assert_eq`) необходимо менять синтаксис.

## Сравнение

| Измерение | Вариант A: функция | Вариант B: ключевое слово |
|------|-------------|---------------|
| Стоимость реализации | Низкая (~20 строк) | Высокая (~200 строк, 4 модуля) |
| Изменения синтаксиса | Нет | Да (новое ключевое слово + синтаксис) |
| Требуется import | Да | Нет |
| Полноправный объект первого класса | Да | Нет |
| Точность позиции в исходном коде | Стек вызовов во время выполнения | Точно до строки и столбца на этапе компиляции |
| Невозможность затенения | Нет | Да |
| Анализ на этапе компиляции | Невозможен (требуется свёртка констант) | Возможен (со свёрткой констант) |
| Расширяемость | Высокая (перегрузка, функции высшего порядка) | Низкая (требуется менять синтаксис) |

### Практические ограничения анализа на этапе компиляции

Главное преимущество варианта B — анализ на этапе компиляции — требует **pass свёртки констант** для активации. Иными словами, компилятор должен вычислить `false` в `assert(false)` на этапе компиляции, чтобы распознать это как мёртвый код.

В настоящее время в YaoXiang нет pass свёртки констант. Даже при использовании варианта B распространённая запись `assert(x > 0)` всё равно не может быть проанализирована на этапе компиляции. Анализу поддаются только литералы вида `assert(true)` / `assert(false)`.

Поэтому преимущества варианта B на этапе компиляции **на текущем этапе носят теоретический, а не практический характер**.
