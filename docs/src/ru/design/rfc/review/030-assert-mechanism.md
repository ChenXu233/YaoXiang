# RFC-030: Механизм утверждений assert

## Резюме

В YaoXiang вводится механизм утверждений `assert`, предназначенный для тестирования и проверки предусловий. `assert` — это единственный в YaoXiang механизм panic времени выполнения, не требующий отдельных ключевых слов `raise`/`throw`. Условный тип времени компиляции `Assert(C)` — это отдельная функциональность (см. RFC-011 §4.3), размещённая в том же модуле `std.assert`, но имеющая полностью независимую семантику и реализацию.

## Мотивация

### Зачем нужна эта функциональность?

В настоящее время E2E-тесты YaoXiang могут имитировать утверждения только через `if` + `io.println` + `return`:

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

Такой подход имеет три проблемы:

1. **Много шаблонного кода**: каждое утверждение требует 4 строк, файлы тестов разрастаются
2. **Слабые сообщения об ошибках**: ручная конкатенация строк, отсутствует информация об исходной позиции
3. **Не комбинируется**: невозможно пакетно регистрировать утверждения, нельзя передать их в качестве аргумента в тестовый фреймворк

### Текущие проблемы

- Отсутствует единый механизм утверждений
- Код тестов наводнён шаблоном `if` + печать + `return`
- На уровне байт-кода уже существует инструкция `Throw`, но на уровне языка она не доступна
- RFC-011 определяет условный тип времени компиляции `Assert(C)`, однако `assert()` времени выполнения ещё не реализован

### Принципы проектирования

`assert` — это единственный пользовательский механизм panic в YaoXiang. `assert(false, "msg")` эквивалентно `raise`, отдельные ключевые слова `throw`/`raise` не нужны. Сама функция `assert` представляет собой оптимальную инкапсуляцию `if raise`.

**Никаких новых ключевых слов, никакого нового синтаксиса. Всё — вызов функции.**

## Вариант A: native-функция

`assert` реализуется как native-функция без введения новых ключевых слов.

```yaoxiang
use std.assert.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Сигнатуры перегрузки

`assert` имеет две перегрузки:

```
// Перегрузка 1: утверждение по условию
assert(cond: Bool, ?msg: String | Error)

// Перегрузка 2: утверждение по Result
assert(cond: Result)
```

#### Перегрузка 1: утверждение по условию `(Bool, ?String | Error)`

`Bool` + опциональное сообщение. Сообщение может быть значением `String` или `Error`:

```yaoxiang
assert(1 + 1 == 2)                    // без сообщения, panic с информацией по умолчанию
assert(1 + 1 == 2, "math is broken")   // строковое сообщение
assert(x > 0, my_error)                // выброс значения Error напрямую
```

`assert(false, "msg")` — это эквивалент `raise`/`throw` в YaoXiang — отдельное ключевое слово не требуется.

#### Перегрузка 2: утверждение по Result `(Result)`

Единственный аргумент `Result` — функция автоматически проверяет, что значение не `Err`:

### Достоинства

- **Нулевые изменения синтаксиса**: чистая функция, не нужны новые ключевые слова
- **Нулевые новые концепции**: повторно используется существующий механизм регистрации native-функций
- **Высокая расширяемость**: перегрузка функций естественно поддерживает несколько сигнатур
- **Самодокументируемость**: пространство имён `std.assert` само по себе является документацией

### Недостатки

Отсутствуют. При корректной сигнатуре типа `assert` компилятор может выявлять мёртвый код посредством анализа достижимости функций. Дополнительные проходы не требуются.

### Поведение во время выполнения

1. Вычисляется первый аргумент `condition: Bool`
2. Если `true`, возвращается `Unit`
3. Если `false`, инициируется panic времени выполнения:
   - Выводится содержимое `message` (если задано)
   - Выводится стек вызова (в режиме отладки)
   - Текущее выполнение прерывается

#### Поведение при сбое для каждой перегрузки

| Сигнатура | Поведение при сбое |
|------|-----------|
| `assert(false)` | panic с информацией по умолчанию |
| `assert(false, "msg")` | вывод строкового сообщения, затем panic |
| `assert(false, error_val)` | выброс значения Error |
| `assert(Err(x))` | извлечение содержимого Err и panic |

### Связь с `Assert` времени компиляции

Модуль `std.assert` содержит две полностью независимые функции:

| Функция | Тип | Имя | Момент | Поведение при сбое | Накладные расходы времени выполнения |
|------|------|------|------|----------|-----------|
| Утверждение времени выполнения | функция | `assert()` | время выполнения | panic + стек вызова | всегда присутствуют |
| Условный тип времени компиляции | тип | `Assert(C)` | время компиляции | ошибка компиляции | нулевые |

```yaoxiang
use std.assert

// assert времени выполнения: для тестов, проверка значений в рантайме
assert.assert(result == 42, "expected 42")

// Assert времени компиляции: уточняющий тип, валидация параметров дженериков
length: assert.Assert(N > 0)
```

Они размещены в одном модуле, поскольку обе представляют разные проявления концепции «утверждения», но реализация полностью независима. `assert()` не требует `Assert(C)`, и наоборот.

### Изменения в компиляторе

**Изменения в parser, AST, typecheck, IR gen не требуются.**

Достаточно добавить регистрацию native-функции в `src/std/`:

1. Создать `src/std/assert.rs`
2. Зарегистрировать `std.assert.assert` и `std.assert.Assert` (последний — условный тип времени компиляции, см. #155)
3. Внутри вызывается уже существующая инструкция `BytecodeInstr::Throw`

### Достоинства

- **Нулевые изменения синтаксиса**: чистая функция, не нужны новые ключевые слова
- **Нулевые новые концепции**: повторно используется существующий механизм регистрации native-функций
- **Высокая расширяемость**: сигнатура функции может быть расширена до вариантов вроде `assert_eq` (в будущем)
- **Самодокументируемость**: пространство имён `std.assert` само по себе является документацией

### Недостатки

- Неизвестно на этапе компиляции: в отличие от варианта B (ключевое слово), невозможно выполнить устранение мёртвого кода на этапе компиляции
- Стек вызова доступен только в режиме отладки

## Вариант B: встроенное ключевое слово

> Устаревший вариант, приведён исключительно для истории.

Реализация через ключевое слово `assert`: на этапе компиляции доступны позиция в исходном коде и свёртка констант.

```yaoxiang
assert(1 + 1 == 2, "math is broken")
```

### Сигнатура типа

Самостоятельной сигнатуры типа нет — ключевое слово обрабатывается парсером.

### Поведение во время выполнения

Аналогично варианту A.

### Изменения в компиляторе

Требуются изменения в parser, AST, typecheck, IR gen:
1. parser: добавление варианта `Expr::Assert`
2. AST: добавление узла `Expr::Assert`
3. typecheck: проверка типов аргументов
4. IR gen: генерация `BytecodeInstr::Throw`

### Достоинства

- На этапе компиляции известна позиция в исходном коде (не зависит от debug-информации)
- Возможна свёртка констант на этапе компиляции: `assert(true)` → пустая операция, `assert(false)` → ошибка компиляции

### Недостатки

| Недостаток | Влияние |
|------|------|
| Требует изменений в парсере | введение нового синтаксического узла, рост стоимости поддержки |
| Ключевое слово не расширяемо | варианты вроде `assert_eq` всё равно требуют функций |
| Преимущества времени компиляции не реализуемы | см. анализ ниже |

### Сравнение

| Аспект | Вариант A (функция) | Вариант B (ключевое слово) |
|------|---------------|-----------------|
| Стоимость реализации | ~20 строк | parser + AST + typecheck + IR gen |
| Изменения синтаксиса | нет | новое ключевое слово |
| Расширяемость | перегрузка функций | требуются сопутствующие макросы |
| Позиция в исходном коде | debug info | доступна на этапе компиляции |
| Свёртка констант | требуется поддержка pass | доступна на этапе компиляции |
| Накладные расходы времени выполнения | вызов функции | минимальные |

### Реалистичные ограничения анализа времени компиляции

Основное преимущество варианта B — анализ времени компиляции — требует наличия **прохода свёртки констант**. То есть компилятор должен вычислить `false` в `assert(false)` на этапе компиляции, чтобы распознать это как мёртвый код.

В YaoXiang в настоящее время нет прохода свёртки констант. Даже при использовании варианта B `assert(x > 0)` и подобные распространённые конструкции по-прежнему не могут быть проанализированы на этапе компиляции. Анализу поддаются только литералы вида `assert(true)` / `assert(false)`.

Следовательно, преимущества варианта B на этапе компиляции на **текущей стадии являются теоретическими, а не практическими**.

## Открытые вопросы

- [x] ~~Выбрать вариант A или B?~~ → **Вариант A: native-функция**
- [x] ~~Нужна ли `assert` упрощённая форма `assert(cond)` без `message`?~~ → **Поддерживается. `assert(cond, ?msg)`, message опционально.**
- [x] ~~Нужны ли варианты `assert_eq`, `assert_ne` и т.д.?~~ → **Не нужны. YAGNI. Дождаться формирования тестового фреймворка.**
- [x] ~~Включает ли вывод panic позицию в исходном коде?~~ → Вариант A зависит от debug-информации (стек вызова).
- [ ] ~~Вопрос объединения assert / Assert~~ → **Вынесен в отдельное обсуждение #156**. Идеал — `assert: (cond: Bool) -> Assert(cond)` для их объединения, но `Assert(C)` как уточняющий тип требует, чтобы `C` было известно на этапе компиляции, что конфликтует с утверждением времени выполнения. Подробности см. в [#156](https://github.com/ChenXu233/YaoXiang/issues/156).

## Приложение А: Записи обсуждения дизайна

### 2026-07-05: Выбор варианта A

Реализация варианта A в 20 строк побеждает по соотношению ценности и стоимости. Преимущества варианта B на этапе компиляции при отсутствии прохода свёртки констант остаются теоретическими.

### 2026-07-11: Дизайн перегрузки assert

**Вопрос**: Почему `assert` нужны две перегрузки, а не единая `(Bool, ?String)`?

**Ответ**:

`assert()` времени выполнения — единственный пользовательский механизм panic в YaoXiang. `assert(false, "msg")` эквивалентно `raise`/`throw` в других языках. Следовательно, он должен покрывать три сценария:
1. Условие + простое сообщение: `assert(cond, "msg")`
2. Условие + пользовательский Error: `assert(cond, my_error)`
3. Проверка Result: `assert(result)` — кратчайшая запись `if is_err { panic }`

Обоснованность перегрузки Result в том, что это кратчайший путь распространения ошибки — «Result должен быть Ok, иначе смерть». Не нужно сначала вызывать `.is_ok()`, а затем отдельно обрабатывать ошибку.

### 2026-07-11: Структура модуля std.assert

`std.assert` содержит две функции:

- `assert()`: функция утверждения времени выполнения (данный RFC)
- `Assert(C)`: условный тип времени компиляции (RFC-011 §4.3, #155)

Они размещены в одном модуле, поскольку обе представляют разные проявления концепции «утверждения». Как пространство имён, `std.assert` самодокументируемо — разработчик, видя `use std.assert`, понимает, что здесь подключается функциональность, связанная с утверждениями.
