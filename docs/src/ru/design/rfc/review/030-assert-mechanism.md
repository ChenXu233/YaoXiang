# RFC-030: Механизм assert

## Резюме

Ввести в YaoXiang механизм `assert`, предназначенный для тестирования и проверки предусловий. Настоящий RFC предлагает два варианта реализации — **native-функцию** и **ключевое слово** — анализирует компромиссы каждого из них и выносит решение.

## Мотивация

### Зачем нужна эта возможность?

В настоящее время E2E-тесты YaoXiang могут имитировать assert только через связку `if` + `io.println` + `return`:

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

Такой способ записи имеет три проблемы:

1. **Много шаблонного кода**: каждый assert требует 4 строки, тестовые файлы раздуваются
2. **Слабые сообщения об ошибках**: ручная конкатенация строк, отсутствует позиция в исходном коде
3. **Некомпозируемость**: невозможно пакетно регистрировать assert'ы, невозможно передавать их в качестве аргументов тестовому фреймворку

### Текущие проблемы

- Отсутствует единый механизм assert
- Тестовый код насыщен шаблоном `if` + печать + `return`
- На уровне байт-кода уже есть инструкция `Throw`, но на уровне языка она не экспонирована

## Вариант A: native-функция

Реализовать `assert` как native-функцию, не вводя нового ключевого слова.

```yaoxiang
use std.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Сигнатура типа

```
assert: (Bool, String) -> ()
```

### Поведение во время выполнения

1. Вычислить первый аргумент `condition: Bool`
2. Если значение `true`, вернуть `Unit`
3. Если значение `false`, инициировать runtime panic:
   - вывести содержимое `message`
   - вывести стек вызовов (в debug-режиме)
   - завершить текущее выполнение

### Изменения в компиляторе

**Не требуется изменений в parser, AST, typecheck, IR gen.**

Достаточно добавить регистрацию native-функции в `src/std/`:

1. Добавить `src/std/assert.rs`
2. Зарегистрировать native-функцию `std.assert`
3. Внутри вызывать уже существующий механизм runtime panic

### Преимущества

- **Нулевые изменения синтаксиса**: не нужно править parser, AST, IR, codegen
- **Простая реализация**: порядка 20 строк кода регистрации native-функции
- **Гражданин первого класса**: можно передавать как аргумент, присваивать переменной, использовать в функциях высшего порядка
- **Постепенное улучшение**: в дальнейшем можно добавить оптимизацию compile-time инлайнинга, не ломая существующий код

### Недостатки

- **Нет анализа на этапе компиляции**: компилятор не может выполнить устранение мёртвого кода для кода после `assert(false)`
- **Ограниченные сообщения об ошибках**: позиция в исходном коде должна извлекаться из стека вызовов во время выполнения — менее точно, чем у ключевого слова
- **Возможность затенения**: пользователь может определить одноимённую функцию, перекрывающую `assert`
- **Требуется импорт**: для использования необходимо писать `use std.assert`

## Вариант B: ключевое слово

Ввести `assert` как ключевое слово языка, со специальной обработкой компилятором на уровне IR.

```yaoxiang
main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Синтаксис

```
assert '(' Expr ',' StringLit ')'
```

`assert` — ключевое слово, его не нужно импортировать через `use`.

### Изменения в компиляторе

Требуются изменения во всей цепочке:

| Этап | Изменения |
|------|-----------|
| Lexer | Добавить токен `KwAssert` |
| Parser | Добавить синтаксис `assert(expr, "msg")` |
| AST | Добавить `StmtKind::Assert { condition, message }` |
| IR Gen | Скомпилировать Assert в условный переход + Throw |
| Codegen | Сгенерировать последовательность инструкций `JmpIfNot` + `Throw` |

### Преимущества

- **Встроенная доступность**: не требует импорта, доступен в любом месте
- **Точная позиция в исходном коде**: компилятор знает позицию assert на уровне AST, сообщения об ошибках точные до строки и столбца
- **Невозможность затенения**: ключевое слово не может быть перекрыто пользовательским кодом
- **Расширяемость в будущем**: возможен анализ на этапе компиляции (в связке с проходом свёртки констант)
- **Ясная семантика**: `assert` обладает явной семантикой на уровне синтаксиса и не может быть использован неправильно

### Недостатки

- **Высокая стоимость реализации**: требует изменений parser, AST, IR, codegen по всей цепочке
- **Не гражданин первого класса**: нельзя передать как аргумент, нельзя присвоить переменной
- **Разрастание ключевых слов**: каждое новое ключевое слово увеличивает сложность языка
- **Низкая расширяемость**: добавление новых вариантов сигнатуры (например, `assert_eq`) требует изменения синтаксиса

## Сравнение

| Аспект | Вариант A: функция | Вариант B: ключевое слово |
|--------|-------------------|---------------------------|
| Стоимость реализации | Низкая (~20 строк) | Высокая (~200 строк, 4 модуля) |
| Изменения синтаксиса | Нет | Да (новое ключевое слово + синтаксис) |
| Требуется импорт | Да | Нет |
| Гражданин первого класса | Да | Нет |
| Точность позиции в исходном коде | Стек вызовов во время выполнения | Точно до строки/столбца на этапе компиляции |
| Невозможность затенения | Нет | Да |
| Анализ на этапе компиляции | Невозможен (требуется свёртка констант) | Возможен (совместно со свёрткой констант) |
| Расширяемость | Высокая (перегрузка, функции высшего порядка) | Низкая (требуется изменение синтаксиса) |

### Реалистичные ограничения анализа на этапе компиляции

Ключевое преимущество варианта B — анализ на этапе компиляции — требует для своей работы **проход свёртки констант**. То есть компилятор должен вычислить `false` в `assert(false)` на этапе компиляции, чтобы распознать этот код как мёртвый.

В YaoXiang в настоящее время нет прохода свёртки констант. Даже при использовании варианта B распространённые варианты записи вроде `assert(x > 0)` не могут быть проанализированы на этапе компиляции. Анализу поддаются только литералы вида `assert(true)` / `assert(false)`.

Следовательно, преимущество варианта B на этапе компиляции **на текущем этапе является теоретическим, а не практическим**.
