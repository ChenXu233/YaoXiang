## Краткое описание

В данном документе определяются основные механизмы FFI (Foreign Function Interface — внешний интерфейс функций) для YaoXiang, включая:

1. **Определение FFI-типов**: использование блока `unsafe {}` для определения непрозрачных типов, возврат в вышестоящую область видимости через `return`
2. **Объявление FFI-функций**: использование синтаксиса `native("symbol")` для объявления внешних функций
3. **Привязка методов**: использование синтаксиса `[0]` для указания позиции параметра self
4. **Обработка непрозрачных типов**: компилятор автоматически определяет непрозрачные типы и пустые типы
5. **Поведение FFI в блоках spawn**: типы-ресурсы автоматически сериализуются, типы-нересурсы могут выполняться параллельно

**Основной принцип проектирования — одно правило, унифицированная семантика**:

```
Все return внутри {} возвращают содержимое в вышестоящую область видимости
По умолчанию, при отсутствии return, возвращается Void
```

## Мотивация

### Зачем нужен данный дизайн?

RFC-020 и RFC-021 определяют различные аспекты FFI:
- RFC-020：интеграция динамических модулей и FFI
- RFC-021：расширение FFI на основе библиотек

Оба документа имеют перекрывающиеся части, поэтому их необходимо объединить в единую спецификацию FFI.

### Цели проектирования

1. **Унификация**: унифицированная семантика return для всех блоков `{}`
2. **Безопасность**: доступ к полям непрозрачных типов требует права unsafe
3. **Простота**: не требуются новые ключевые слова или специальные标记и
4. **Практичность**: автоматическая генерация привязок через yx-bindgen

## Предложение

### 1. Определение FFI-типов

#### 1.1 Блок unsafe + семантика return

Определение непрозрачных типов в блоке unsafe с возвратом в вышестоящую область видимости:

```yaoxiang
// Определение непрозрачного типа в блоке unsafe
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void  // Голый указатель
    }
    return SqliteDb
}

// SqliteDb доступен за пределами блока unsafe
db = sqlite3_open("test.db")

// ❌ Ошибка компиляции: доступ к полю handle требует права unsafe
handle = db.handle

// ✅ Через вызов метода
db.close()
```

#### 1.2 Прозрачные типы

Прозрачные типы определяются напрямую, без блока unsafe:

```yaoxiang
// Прозрачный тип
Point: Type = {
    x: Int32,
    y: Int32
}

// Пользователь может создавать напрямую
p: Point = Point { x: 1, y: 2 }
```

#### 1.3 Определение непрозрачных типов

Компилятор автоматически определяет непрозрачные типы и пустые типы:

```yaoxiang
// Непрозрачный тип (ссылается на него native функция)
SqliteDb: Type = {}
sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")
// → SqliteDb ссылается на него native функция → непрозрачный тип

// Пустой тип (не ссылается на него ни одна native функция)
MyType: Type = {}
// → MyType не ссылается на него ни одна native функция → пустой тип
```

**Правила определения**:
- Если тип используется функцией `native` → непрозрачный тип
- В противном случае → пустой тип

---

### 2. Объявление FFI-функций

#### 2.1 Синтаксис native

Использование синтаксиса `native("symbol")` для объявления внешних функций:

```yaoxiang
// Объявление FFI-функций
sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")
sqlite3_close: (db: SqliteDb) -> Int32 = native("sqlite3_close")
sqlite3_exec: (db: SqliteDb, sql: String) -> Int32 = native("sqlite3_exec")
```

#### 2.2 Маппинг типов параметров

Типы параметров FFI-функций используют типы YaoXiang напрямую, компилятор автоматически обрабатывает маппинг типов C:

| Тип C | Тип YaoXiang |
|-------|--------------|
| `int` | `Int32` |
| `long` | `Int64` |
| `float` | `Float32` |
| `double` | `Float64` |
| `char` | `Char` |
| `char*` | `String` |
| `bool` | `Bool` |
| `size_t` | `Uint` |
| `void*` | `*Void` |
| `struct T*` | `T` (прозрачный тип)|
| `typedef struct T T` | `T` (непрозрачный тип)|

#### 2.3 Возвращаемые типы

Возвращаемые типы FFI-функций используют типы YaoXiang напрямую:

```yaoxiang
// Возврат непрозрачного типа
sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")

// Возврат прозрачного типа
get_point: () -> Point = native("get_point")

// Возврат базового типа
get_value: () -> Int32 = native("get_value")
```

### 3. Привязка методов

#### 3.1 Синтаксис [0]

Использование синтаксиса `[0]` для указания позиции параметра self в кортеже параметров функции:

```yaoxiang
// FFI-функции
sqlite3_close: (db: SqliteDb) -> Int32 = native("sqlite3_close")
sqlite3_exec: (db: SqliteDb, sql: String) -> Int32 = native("sqlite3_exec")

// Привязка методов (self на позиции 0)
SqliteDb.close = sqlite3_close[0]
SqliteDb.exec = sqlite3_exec[0]
```

**Способ вызова**:
```yaoxiang
db = sqlite3_open("test.db")

// Вызов метода
db.close()  // эквивалентно sqlite3_close(db)
db.exec("SELECT * FROM users")  // эквивалентно sqlite3_exec(db, "SELECT * FROM users")
```

#### 3.2 Привязка конструкторов

Конструкторы не используют `[0]`, привязываются как обычные функции:

```yaoxiang
// FFI-функция
sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")

// Привязка конструктора (обычная функция)
SqliteDb.open = sqlite3_open
```

**Способ вызова**:
```yaoxiang
// Создание через конструктор
db = SqliteDb.open("test.db")
```

#### 3.3 Позиция привязки

Привязка методов может быть в любом месте, поскольку типы являются контейнерами данных:

```yaoxiang
// Привязка после определения типа
SqliteDb.close = sqlite3_close[0]

// Привязка в другом файле
SqliteDb.exec = sqlite3_exec[0]

// Компилятор в итоге проверяет всё
```

### 4. Обработка непрозрачных типов

#### 4.1 Автоматическая обработка компилятором

Компилятор автоматически определяет непрозрачные типы и внутренне обрабатывает хранение указателей C:

```text
Анализ компилятора:
    SqliteDb: Type = {}
    sqlite3_open: ... -> SqliteDb = native("sqlite3_open")

Вывод:
    SqliteDb — непрозрачный тип
    Внутри автоматически добавлено @internal handle: *Void
    Запрещено создание пользователем напрямую
```

#### 4.2 Код пользователя

```yaoxiang
import sqlite3_bindings

// ✅ Создание через конструктор
db = SqliteDb.open("test.db")

// ❌ Ошибка компиляции: нельзя создавать непрозрачный тип напрямую
db: SqliteDb = {}

// ✅ Через вызов метода
result = db.exec("SELECT * FROM users")
db.close()
```

### 5. Поведение FFI в блоках spawn

#### 5.1 Автоматическая сериализация типов-ресурсов

Если FFI-тип является типом-ресурсом, в блоке spawn происходит автоматическая сериализация:

```yaoxiang
// SqliteDb — тип-ресурс
(a, b) = spawn {
    db1 = SqliteDb.open("db1.sqlite"),  // SqliteDb — ресурс
    db2 = SqliteDb.open("db2.sqlite")   // Разные экземпляры, можно параллельно
}

(a, b) = spawn {
    result1 = db.exec("SELECT ..."),  // Тот же SqliteDb
    result2 = db.exec("INSERT ...")   // Автоматическая сериализация
}
```

#### 5.2 Параллельное выполнение типов-нересурсов

Если FFI-тип не является типом-ресурсом, в блоке spawn возможно параллельное выполнение:

```yaoxiang
// Float не является типом-ресурсом
(a, b) = spawn {
    result1 = sin(1.0),  // Можно параллельно
    result2 = cos(1.0)   // Можно параллельно
}
```

---

### 6. Инструментарий yx-bindgen

#### 6.1 Что генерируется

yx-bindgen генерирует следующее:
- Определения FFI-типов (блок unsafe + return)
- Объявления FFI-функций (синтаксис native)
- Привязки методов (синтаксис [0])

#### 6.2 Пример генерации

```bash
yx-bindgen --header /usr/include/sqlite3.h --output sqlite3_bindings.yx
```

Результат генерации:

```yaoxiang
// sqlite3_bindings.yx
// Автоматически сгенерировано, не редактировать вручную

// ============================================================================
// Определения типов
// ============================================================================

SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void
    }
    return SqliteDb
}

SqliteStmt = unsafe {
    SqliteStmt: Type = {
        handle: *Void
    }
    return SqliteStmt
}

// ============================================================================
// Объявления FFI-функций
// ============================================================================

sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")
sqlite3_close: (db: SqliteDb) -> Int32 = native("sqlite3_close")
sqlite3_exec: (db: SqliteDb, sql: String) -> Int32 = native("sqlite3_exec")
sqlite3_prepare_v2: (db: SqliteDb, sql: String) -> SqliteStmt = native("sqlite3_prepare_v2")
sqlite3_step: (stmt: SqliteStmt) -> Int32 = native("sqlite3_step")
sqlite3_finalize: (stmt: SqliteStmt) -> Int32 = native("sqlite3_finalize")

// ============================================================================
// Привязки методов
// ============================================================================

// Конструктор (обычная функция)
SqliteDb.open = sqlite3_open

// Методы (self на позиции 0)
SqliteDb.close = sqlite3_close[0]
SqliteDb.exec = sqlite3_exec[0]
SqliteDb.prepare = sqlite3_prepare_v2[0]

// Методы SqliteStmt
SqliteStmt.step = sqlite3_step[0]
SqliteStmt.finalize = sqlite3_finalize[0]
```

## Компромиссы

### Преимущества

1. **Унифицированная семантика**: унифицированная семантика return для всех блоков `{}`
2. **Не требуются новые ключевые слова**: используются существующие unsafe и return
3. **Не требуются специальные标记и**: компилятор автоматически определяет непрозрачные типы
4. **Безопасность**: доступ к полям непрозрачных типов требует права unsafe
5. **Практичность**: автоматическая генерация привязок через yx-bindgen

### Недостатки

1. **Область видимости блока unsafe**: необходимо изменить семантику return для блоков `{}`
2. **Сложность компилятора**: требуется автоматическое определение непрозрачных типов
3. **Поддержка yx-bindgen**: требуется постоянное обновление для поддержки новых C-библиотек
