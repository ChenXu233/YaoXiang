## Резюме

Данный документ определяет основные механизмы FFI (интерфейса внешних функций) в YaoXiang, включая:

1. **Определение FFI типов**: использование блоков `unsafe {}` для определения непрозрачных типов, возврат результата во внешнюю область видимости через `return`
2. **Объявление FFI функций**: использование синтаксиса `native("symbol")` для объявления внешних функций
3. **Привязка методов**: использование синтаксиса `[0]` для указания позиции параметра self
4. **Обработка непрозрачных типов**: явное определение непрозрачных типов в блоках `unsafe {}`, пустое тело `Type = {}` обозначает вакуумный тип
5. **Жизненный цикл непрозрачных типов**: привязка деструктора через `.drop`, автоматическое освобождение по RAII, безопасная обработка Null
6. **Поведение FFI в spawn блоках**: ресурсные типы автоматически сериализуются, нересурсные типы могут выполняться параллельно

**Ключевой дизайн — один принцип, единая семантика**:

```
Все return внутри {} возвращают содержимое во внешнюю область видимости
По умолчанию отсутствие return означает возврат Void
```

## Мотивация

### Зачем нужен этот дизайн?

RFC-020 и RFC-021 определяли разные аспекты FFI:
- RFC-020: Динамические модули и интеграция FFI
- RFC-021: Расширение FFI на основе библиотек

Они пересекаются, и их необходимо объединить в единую спецификацию FFI.

### Цели дизайна

1. **Единообразие**: семантика return во всех блоках `{}` согласована
2. **Безопасность**: доступ к полям непрозрачных типов требует привилегий unsafe
3. **Простота**: не требуются новые ключевые слова или специальные маркеры
4. **Практичность**: yx-bindgen автоматически генерирует привязки

## Предложение

### 1. Определение FFI типов

#### 1.1 Блок unsafe + семантика return

В блоке unsafe определяется непрозрачный тип, который возвращается во внешнюю область видимости через return:

```yaoxiang
// Определение непрозрачного типа в блоке unsafe
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void  // Голый указатель
    }
    return SqliteDb
}

// SqliteDb доступен за пределами блока unsafe
db = sqlite3_open("test.db")

// ❌ Ошибка компиляции: доступ к полю handle требует привилегий unsafe
handle = db.handle

// ✅ Через вызов метода
db.close()
```

#### 1.2 Прозрачные типы

Прозрачные типы определяются напрямую, без блока unsafe:

```yaoxiang
// Прозрачный тип
Point: Type = {
    x: Int32,
    y: Int32
}

// Пользователь может создавать экземпляры напрямую
p: Point = Point { x: 1, y: 2 }
```

#### 1.3 Непрозрачные типы, прозрачные типы и вакуумные типы

Различие между тремя типами определяется в момент **объявления** и не требует от компилятора вывода между файлами:

```yaoxiang
// Прозрачный тип: имеет поля
Point: Type = { x: Int32, y: Int32 }
// Пользователь может создавать экземпляры и обращаться к полям
p: Point = Point { x: 1, y: 2 }

// Вакуумный тип: пустое тело, не в блоке unsafe
MyMarker: Type = {}
// Тип нулевого размера, свободное создание
x: MyMarker = {}

// Непрозрачный тип: возвращается из блока unsafe
SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void
    }
    return SqliteDb
}
// SqliteDb является непрозрачным типом, нельзя создавать напрямую, нельзя обращаться к полям
```

**Правила**:
- **Есть поля** → прозрачный тип
- **Пустое тело + не в блоке unsafe** → вакуумный тип (нулевого размера)
- **Возвращается из блока unsafe** → непрозрачный тип

Функции `native` могут ссылаться только на явно определённые типы, не изменяя их свойства. Свойства типа определяются в момент объявления, а не выводятся в момент использования.

### 2. Объявление FFI функций

#### 2.1 Синтаксис native

Для объявления внешних функций используется синтаксис `native("symbol")`:

```yaoxiang
// Объявление FFI функции
sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")
sqlite3_close: (db: SqliteDb) -> Int32 = native("sqlite3_close")
sqlite3_exec: (db: SqliteDb, sql: String) -> Int32 = native("sqlite3_exec")
```

#### 2.2 Сопоставление типов параметров

Типы параметров FFI функций напрямую используют типы YaoXiang, а компилятор автоматически обрабатывает отображение в C-типы:

| C тип | Тип YaoXiang |
|--------|---------------|
| `int` | `Int32` |
| `long` | `Int64` |
| `float` | `Float32` |
| `double` | `Float64` |
| `char` | `Char` |
| `char*` | `String` |
| `bool` | `Bool` |
| `size_t` | `Uint` |
| `void*` | `*Void` |
| `struct T*` | `T` (прозрачный тип) |
| `typedef struct T T` | `T` (непрозрачный тип) |

#### 2.3 Тип возвращаемого значения

Тип возвращаемого значения FFI функции напрямую использует тип YaoXiang:

```yaoxiang
// Возвращает непрозрачный тип
sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")

// Возвращает прозрачный тип
get_point: () -> Point = native("get_point")

// Возвращает базовый тип
get_value: () -> Int32 = native("get_value")
```

### 3. Привязка методов

#### 3.1 Синтаксис [0]

Синтаксис `[0]` используется для указания позиции параметра self в кортеже параметров функции:

```yaoxiang
// FFI функция
sqlite3_close: (db: SqliteDb) -> Int32 = native("sqlite3_close")
sqlite3_exec: (db: SqliteDb, sql: String) -> Int32 = native("sqlite3_exec")

// Привязка методов (self на позиции 0)
SqliteDb.close = sqlite3_close[0]
SqliteDb.exec = sqlite3_exec[0]
```

**Способ вызова**:
```yaoxiang
db = sqlite3_open("test.db")

// Вызов метода
db.close()  // Эквивалентно sqlite3_close(db)
db.exec("SELECT * FROM users")  // Эквивалентно sqlite3_exec(db, "SELECT * FROM users")
```

#### 3.2 Привязка конструктора

Конструкторы привязываются без `[0]` как обычные функции:

```yaoxiang
// FFI функция
sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")

// Привязка конструктора (обычная функция)
SqliteDb.open = sqlite3_open
```

**Способ вызова**:
```yaoxiang
// Создание через конструктор
db = SqliteDb.open("test.db")
```

#### 3.3 Расположение привязок

Привязки методов могут располагаться в любом месте, поскольку типы являются контейнерами данных:

```yaoxiang
// Привязка после определения типа
SqliteDb.close = sqlite3_close[0]

// Привязка в другом файле
SqliteDb.exec = sqlite3_exec[0]

// Компилятор в любом случае выполнит проверку
```

### 4. Обработка непрозрачных типов

#### 4.1 Внутреннее хранение непрозрачного типа

Поле `handle: *Void`, определённое в блоке `unsafe {}` типа, автоматически управляется компилятором:

```text
Обработка компилятором:
    SqliteDb = unsafe {
        SqliteDb: Type = {
            handle: *Void         // ← Компилятор внутренне хранит C-указатель
        }
        return SqliteDb
    }

Результат:
    SqliteDb является непрозрачным типом (явно определён блоком unsafe)
    Доступ к полям требует привилегий unsafe
    Запрещено прямое создание пользователем (только через native конструктор)
```

Обратный вывод компилятора не требуется — непрозрачность типа определяется способом объявления, что ясно и предсказуемо.

#### 4.2 Пользовательский код

```yaoxiang
import sqlite3_bindings

// ✅ Создание через конструктор
db = SqliteDb.open("test.db")

// ❌ Ошибка компиляции: нельзя создать непрозрачный тип напрямую
db: SqliteDb = {}

// ✅ Через вызов метода
result = db.exec("SELECT * FROM users")
db.close()
```

### 5. Поведение FFI в spawn блоках

#### 5.1 Автоматическая сериализация ресурсных типов

Если FFI тип является ресурсным, в spawn блоке он автоматически сериализуется:

```yaoxiang
// SqliteDb является ресурсным типом
(a, b) = spawn {
    db1 = SqliteDb.open("db1.sqlite"),  // Ресурс SqliteDb
    db2 = SqliteDb.open("db2.sqlite")   // Разные экземпляры, могут выполняться параллельно
}

(a, b) = spawn {
    result1 = db.exec("SELECT ..."),  // Тот же SqliteDb
    result2 = db.exec("INSERT ...")   // Автоматическая сериализация
}
```

#### 5.2 Параллельное выполнение для нересурсных типов

Если FFI тип не является ресурсным, в spawn блоке он может выполняться параллельно:

```yaoxiang
// Float не является ресурсным типом
(a, b) = spawn {
    result1 = sin(1.0),  // Могут выполняться параллельно
    result2 = cos(1.0)   // Могут выполняться параллельно
}
```

### 6. Инструментарий yx-bindgen

#### 6.1 Генерируемое содержимое

yx-bindgen генерирует следующее:
- Определения FFI типов (блок unsafe + return)
- Объявления FFI функций (синтаксис native)
- Привязки методов (синтаксис [0])

#### 6.2 Пример генерации

```bash
yx-bindgen --header /usr/include/sqlite3.h --output sqlite3_bindings.yx
```

Результат:

```yaoxiang
// sqlite3_bindings.yx
// Сгенерировано автоматически, не редактировать вручную

// ============================================================================
// Определения типов
// ============================================================================

SqliteDb = unsafe {
    SqliteDb: Type = {
        handle: *Void
    }
    return SqliteDb
}

SqliteStmt = unsafe {
    SqliteStmt: Type = {
        handle: *Void
    }
    return SqliteStmt
}

// ============================================================================
// Объявления FFI функций
// ============================================================================

sqlite3_open: (filename: String) -> SqliteDb = native("sqlite3_open")
sqlite3_close: (db: SqliteDb) -> Int32 = native("sqlite3_close")
sqlite3_exec: (db: SqliteDb, sql: String) -> Int32 = native("sqlite3_exec")
sqlite3_prepare_v2: (db: SqliteDb, sql: String) -> SqliteStmt = native("sqlite3_prepare_v2")
sqlite3_step: (stmt: SqliteStmt) -> Int32 = native("sqlite3_step")
sqlite3_finalize: (stmt: SqliteStmt) -> Int32 = native("sqlite3_finalize")

// ============================================================================
// Привязки методов
// ============================================================================

// Конструктор (обычная функция)
SqliteDb.open = sqlite3_open

// Методы (self на позиции 0)
SqliteDb.close = sqlite3_close[0]
SqliteDb.exec = sqlite3_exec[0]
SqliteDb.prepare = sqlite3_prepare_v2[0]

// Методы SqliteStmt
SqliteStmt.step = sqlite3_step[0]
SqliteStmt.finalize = sqlite3_finalize[0]

// ============================================================================
// Деструкторы
// ============================================================================

SqliteDb.drop = sqlite3_close[0]
SqliteStmt.drop = sqlite3_finalize[0]
```

### 7. Управление жизненным циклом непрозрачных типов

Непрозрачные типы следуют модели владения из RFC-009, без введения новых концепций.

#### 7.1 Основные принципы

- **Семантика Move**: непрозрачные типы по умолчанию используют Move, присваивание/передача параметров/возврат = передача владения, копирование запрещено
- **Освобождение по RAII**: автоматический вызов деструктора при выходе из области видимости
- **Отслеживание потребления**: после явного вызова деструктора переменная считается потреблённой и не может использоваться повторно

#### 7.2 Привязка деструктора

Для привязки деструктора используется соглашение `.drop`, синтаксис полностью совпадает с обычной привязкой метода:

```yaoxiang
// Привязка деструктора (self на позиции 0)
SqliteDb.drop = sqlite3_close[0]
SqliteStmt.drop = sqlite3_finalize[0]
```

Компилятор распознаёт привязку `.drop` и автоматически вызывает её при выходе из области видимости. **Не вводятся новые ключевые слова, не вводится система trait** — это привязка метода + RAII, семантика, уже обещанная в RFC-009.

#### 7.3 Автоматический вызов деструктора

```yaoxiang
{
    db = SqliteDb.open("test.db")
    stmt = db.prepare("SELECT * FROM users")
    stmt.step()
    // ← Выход из области видимости, автоматический вызов деструкторов в обратном порядке:
    //   stmt.drop()  → sqlite3_finalize(stmt)
    //   db.drop()    → sqlite3_close(db)
}
```

**Порядок вызова деструкторов**: обратный порядку объявления (последний объявленный — первый уничтожается), что соответствует семантике RAII.

#### 7.4 Явный вызов деструктора

```yaoxiang
db = SqliteDb.open("test.db")
db.close()              // Явный вызов деструктора. close эквивалентен drop — используется имя, под которым выполнена привязка
db.exec("...")          // ❌ Ошибка компиляции: db уже потреблён (нельзя читать после Move)
```

Нет отдельной концепции "close vs drop". После `SqliteDb.drop = sqlite3_close[0]` вызовы `db.close()` и `db.drop()` обращаются к одной и той же функции.

#### 7.5 Деструктор и Move

```yaoxiang
db = SqliteDb.open("test.db")
db2 = db                // Move: владение передаётся db2
// db здесь уже недействителен
// ← Выход из области видимости, автоматический вызов db2.drop()

// Потребление в функции
process_db: (db: SqliteDb) -> Void = {
    result = db.exec("...")
    // ← Конец функции, db уничтожается здесь
}

db = SqliteDb.open("test.db")
process_db(db)          // Move в функцию, уничтожение при завершении функции
// db здесь уже недействителен
```

#### 7.6 Обработка Null

```yaoxiang
// Может вернуть null → маркер `?` для опционального типа, пользователь обязан обработать
sqlite3_open: (filename: String) -> ?SqliteDb = native("sqlite3_open")

db = SqliteDb.open("test.db")
match db {
    Some(db) => {
        db.exec("SELECT * FROM users")
        // ← Выход из области видимости, автоматический вызов db.drop()
    }
    None => print("Ошибка открытия")
}

// Не может вернуть null → без маркера, panic при null
// Используется, когда по контракту C-функция никогда не возвращает null
sqlite3_get_global: () -> SqliteDb = native("sqlite3_get_global")
```

**Принцип дизайна**: возврат null из C либо требует обработки пользователем (`?`), либо вызывает panic с раскрытием проблемы. Третьего варианта "молчаливого игнорирования" не существует.

#### 7.7 Обработка ошибок деструктора

```yaoxiang
// Деструктор может возвращать код ошибки
sqlite3_close: (db: SqliteDb) -> Int32 = native("sqlite3_close")
SqliteDb.drop = sqlite3_close[0]

// Поведение компилятора:
//   debug режим: возврат деструктора != 0 → panic (выявление проблемы)
//   release режим: возврат игнорируется (стандарт C допускает ошибку close без влияния на безопасность памяти)
```

#### 7.8 Деструкторы в spawn блоках

```yaoxiang
// Ресурсные типы в spawn блоках автоматически сериализуются, уничтожение безопасно по умолчанию
{
    db = SqliteDb.open("test.db")
    result = db.exec("...")
}  // ← Гарантия сериализации: drop после завершения exec, без гонок

// Move через границу spawn
db = SqliteDb.open("test.db")
spawn {
    use(db)             // Move в spawn
    // ← Конец spawn, автоматическое уничтожение
}
```

#### 7.9 Типы без деструктора

Непрозрачные типы не обязаны иметь привязку `.drop`. Типы без привязки деструктора при выходе из области видимости не выполняют никаких действий — это подходит для обёрток над статическими данными, глобальными хэндлами и т.п., не требующими очистки.

Компилятор в debug режиме выдаёт lint-предупреждение (warn по умолчанию) для непрозрачных типов без привязки `.drop`, чтобы напомнить пользователю о необходимости подтверждения.

#### 7.10 Сводка правил жизненного цикла

| Сценарий | Поведение | Источник |
|------|------|------|
| Присваивание непрозрачного типа | Move (копирование запрещено) | RFC-009 |
| Привязка `.drop` | Синтаксис привязки метода `[0]` | Данный документ §3 |
| Выход из области видимости | Автоматический вызов `.drop()` в обратном порядке | RAII из RFC-009 |
| Явный `.close()` | Потребление переменной, недоступна после | Семантика Move из RFC-009 |
| Возврат Null | Опциональный `?T` / прямой panic | Данный документ §7.6 |
| В spawn блоках | Автоматическая сериализация, безопасное уничтожение | RFC-024 |

## Компромиссы

### Преимущества

1. **Единая семантика**: семантика return во всех блоках `{}` согласована
2. **Не требуются новые ключевые слова**: используются существующие unsafe и return
3. **Явное определение**: свойства типа определяются в момент объявления — возврат из unsafe блока → непрозрачный, пустое тело → вакуумный, без вывода
4. **Жизненный цикл без новых концепций**: `.drop` = привязка метода + RAII, без системы trait, без новых ключевых слов
5. **Безопасность**: доступ к полям непрозрачных типов требует привилегий unsafe, переменная недоступна после вызова деструктора
6. **Практичность**: yx-bindgen автоматически генерирует привязки (включая деструкторы)

### Недостатки

1. **Область видимости блока unsafe**: необходимо изменить семантику return блоков `{}`
2. **Поддержка yx-bindgen**: требуется постоянное обновление для поддержки новых C-библиотек

## Записи решений дизайна

| Решение | Определение | Причина | Дата |
|------|------|------|------|
| Определение FFI типов | Блок unsafe + return | Единая семантика, без новых ключевых слов | 2026-06-05 |
| Определение непрозрачного типа | Явное определение в блоке unsafe | Свойства типа определяются в момент объявления, не зависят от внешнего вывода | 2026-06-05 |
| Привязка методов | Синтаксис [0] | Явное указание позиции self | 2026-06-05 |
| Конструктор | Привязка как обычная функция | Не требуется специальный синтаксис | 2026-06-05 |
| Поведение в spawn блоках | Автоматическая сериализация ресурсных типов | Безопасность, соответствие модели конкурентности | 2026-06-05 |
| Деструктор | `.drop = native_fn[0]` | Привязка метода + RAII, ноль новых концепций | 2026-06-10 |
| Обработка Null | Опциональный `?T` / прямой panic | Проблемы C не скрываются | 2026-06-10 |
