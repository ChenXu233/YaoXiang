# RFC 022: Поддержка статической верификации на основе логики Хора (спецификационные комментарии и спецификационные типы)

## Аннотация

В данном документе предлагается внедрение в язык YaoXiang механизма **статической верификации на основе логики Хора**, позволяющего разработчикам записывать предусловия, постусловия и инварианты циклов в комментариях в форме `//!` или `/*! ... !*/`. При отладочной сборке (Debug Build) выполняется обязательная статическая верификация, и только после её успешного прохождения возможна релизная сборка (Release Build); при релизной сборке спецификационные комментарии игнорируются (нулевые накладные расходы), а кэш верификации очищается. Сами спецификации рассматриваются как часть системы типов, формируя «спецификационные типы» (такие как `Requires(P)`, `Ensures(P)`), и могут быть расширены пользователем. Данная архитектура направлена на сохранение лаконичности языка при обеспечении высокой надёжности критического кода и идеально интегрируется с унифицированной моделью типов YaoXiang.

## Мотивация

### Зачем нужна данная возможность/изменение?

Язык YaoXiang уже обеспечивает безопасность памяти и потоков через модель владения (RFC-009) и модель параллелизма (RFC-001), однако логическая корректность по-прежнему зависит от тестирования. В системном программировании и критически важных областях (аэрокосмическая отрасль, финансы, ядра операционных систем) логические ошибки могут привести к катастрофическим последствиям. Существующие механизмы (такие как проверка заимствований в Rust) не способны обнаруживать подобные ошибки. Логика Хора предоставляет математический аппарат доказательства, однако традиционные инструменты формальной верификации зачастую требуют отдельного языка спецификаций и сложного порога вхождения.

### Текущие проблемы

- Логическая корректность может быть проверена только тестированием, без гарантий на этапе компиляции
- Критически важный системный код лишён средств формальной верификации
- Существующие инструменты формальной верификации имеют крутой порог вхождения и отделены от мейнстримных языков программирования

## Предложение

### Основная архитектура

Наша цель — спроектировать **лёгкое, интегрированное с языком** решение статической верификации:

- **Обязательная верификация при Debug Build**: разработчики записывают спецификации в модулях, требующих высокой надёжности; при отладочной сборке верификация обязательна, и только после её успеха возможна релизная сборка
- **Элегантный синтаксис**: используются комментарии `//!`, без введения новых ключевых слов; поддерживается подсветка синтаксиса в редакторах
- **Интеграция с системой типов**: спецификации становятся частью типов, участвуют в проверке типов, поддерживают пользовательские спецификационные типы
- **Доказуемость и тестируемость**: возможна как статическое доказательство, так и понижение до runtime-ассертов; удобен для постепенного внедрения

### 1. Синтаксис спецификационных комментариев

В начале тела функции или цикла используются комментарии `//!` (однострочные) или `/*! ... !*/` (многострочные) для записи спецификаций.

#### 1.1 Унифицированный синтаксис спецификаций

Спецификации используют унифицированную в YaoXiang модель синтаксиса `name: Type = expression`, полностью интегрированную с системой типов:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0
    //! ensures: GreaterOrEqual(result, arr[0..n])
    //! ensures: ExistsMax(result, arr[0..n])
    // Реализация...
}
```

- Спецификация по сути является **объявлением типа**, справа — булево выражение
- Слева — экземпляр спецификационного типа (с возможными параметрами типа)
- Доступна специальная переменная `result` для обозначения возвращаемого значения

#### 1.2 Спецификации циклов

```yaoxiang
while i < n {
    /*! invariant: Bounds[i, n] = 0 <= i <= n
                  && SumInvariant[s, arr[0..i]] !*/
    s = s + arr[i]
    i = i + 1
}
```

#### 1.3 Спецификационные выражения

Булевы выражения справа от спецификаций используют синтаксис выражений YaoXiang, поддерживая:
- Арифметические, сравнительные и логические операции
- Кванторы: `forall i in 0..n: P(i)`, `exists i in 0..n: P(i)` — встроенные в язык логические конструкции
- Вызовы функций (только чистые функции)

### 2. Система спецификационных типов

Спецификационные типы по своей сути являются обычными типами YaoXiang, полностью соответствующими унифицированной модели синтаксиса.

#### 2.1 Встроенные спецификационные типы

Компилятор предоставляет следующие встроенные спецификационные типы (используемые непосредственно в спецификациях):

```yaoxiang
// Определения встроенных спецификационных типов
NonEmpty: (T: Type) -> Type = { len: T; len > 0 }
Positive: Type = { x: Int; x > 0 }
GreaterOrEqual: (T: Type) -> Type = { result: T, arr: Array(T); result >= arr[0] && forall i in 1..arr.len: result >= arr[i] }
Bounds: (T: Type) -> Type = { i: T, n: T; 0 <= i && i <= n }
SumInvariant: (T: Type) -> Type = { s: T, arr: Array(T); s == sum(arr[0..i]) }

// Конструкции кванторов (встроены в язык, не являются функциями)
forall: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
exists: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
```

#### 2.2 Пользовательские спецификационные типы

Полностью аналогично обычным определениям типов, пользователи могут определять собственные спецификационные типы:

```yaoxiang
// Определение спецификации положительного целого
Positive: Type = { x: Int; x > 0 }

// Определение спецификации отсортированного массива
Sorted: (T: Ord) -> Type = {
    arr: Array(T);
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

// Определение спецификации существования максимума
ExistsMax: (T: Ord) -> Type = {
    result: T, arr: Array(T);
    exists i in 0..arr.len: result == arr[i]
    && forall j in 0..arr.len: result >= arr[j]
}
```

Использование пользовательских спецификаций:

```yaoxiang
sqrt: (x: Positive) -> Float = {
    //! ensures: SquareRootResult(result, x) = result * result <= x && (result+1)*(result+1) > x
    // Реализация...
}

binary_search: (T: Ord) -> ((arr: Sorted(Array(T)), key: T) -> Option(Index)) = {
    //! ensures: SearchResult(result, arr, key)
    // Реализация...
}
```

Спецификационные типы, как и другие типы, поддерживают параметры-типы, ограничения типов и участвуют в выводе типов.

### 3. Режимы компиляции

| Режим | Поведение | Опция |
|------|------|------|
| **Debug Build** | Парсинг спецификаций, генерация условий верификации, вызов SMT-решателя для доказательства; верификация обязательна перед релизной сборкой | `yaoxiangc --debug source.yx` |
| **Release Build** | Игнорирование всех комментариев `//!`, отсутствие генерации кода; очистка всех кэшей верификации; включение агрессивной оптимизации | `yaoxiangc --release source.yx` |
| **Runtime-проверки** | Преобразование спецификаций в runtime-ассерты, при нарушении — panic | `yaoxiangc --enable-runtime-checks source.yx` |

В режиме верификации при неудаче доказательства компилятор сообщает об ошибке и предоставляет возможные контрпримеры (например, входные значения).

### 4. Механизм верификации

Компилятор преобразует спецификации в условия верификации (Verification Conditions) и отправляет их встроенному SMT-решателю (например, Z3). Процесс верификации в общих чертах выглядит следующим образом:

1. Сбор `requires` и `ensures` функции, `invariant` цикла
2. Генерация доказательственных обязательств для каждого цикла: инвариант истинен при входе в цикл, сохраняется после каждой итерации, а при выходе из цикла влечёт постусловие
3. Преобразование тела функции в логическую формулу, объединение со спецификациями для формирования условий верификации
4. Вызов SMT-решателя для проверки выполнимости

Если решатель возвращает `unsat` (невыполнима), спецификация считается верной; в противном случае сообщается контрпример.

### 5. Интеграция с тестированием

Режим runtime-проверок преобразует спецификации в ассерты, используемые при тестировании. Совместно со средствами покрытия спецификаций можно оценить степень покрытия тестами. В перспективе возможны инструменты извлечения спецификаций, автоматически выводящие候选ные спецификации из тестов.

### 6. Поддержка в редакторах

`//!` и `/*! ... !*/` могут распознаваться редакторами как специальные комментарии, выделяемые другим цветом (например, фиолетовым), что отличает их от обычных комментариев. Language Server способен предоставлять подсказки при наведении курсора, автодополнение и сообщения об ошибках верификации.

## Детальное проектирование

### Синтаксические изменения

| До | После |
|------|------|
| Синтаксис спецификационных комментариев отсутствовал | Разрешены комментарии-спецификации `//!` и `/*! ... !*/` |

### 7.1 Синтаксическое расширение

На основе существующего синтаксиса (RFC-010) разрешено появление нуля или более комментариев `//!` или `/*! ... !*/` в начале тела функции или цикла. Синтаксис спецификаций согласован с унифицированным синтаксисом типов:

```
spec_comment     ::= ('//!' spec_line) | ('/*!' spec_block '!*/')
spec_line        ::= spec_name ':' type_expr '=' expr
spec_name        ::= 'requires' | 'ensures' | 'invariant'
spec_block       ::= (spec_name ':' type_expr '=' expr ';')*
```

- Спецификация по сути является объявлением типа: `имя_спецификации: спецификационный_тип = булево_выражение`
- `type_expr` — выражение спецификационного типа, может содержать параметры типа
- `expr` использует синтаксис выражений YaoXiang, поддерживает кванторы

### 7.2 Проверка типов

В режиме верификации компилятор преобразует спецификационные комментарии в соответствующие экземпляры спецификационных типов и записывает их в метаданные функции или цикла.

### 7.3 Генерация условий верификации

Используется исчисление наиweakest precondition или наиболееstrongest postcondition совместно с инвариантами циклов для генерации формул первого порядка. Генерируемые VC используют формат SMT-LIB с вызовом внешнего решателя.

### 7.4 Сообщения об ошибках

При неудаче доказательства решатель может предоставить модель (контрпример). Компилятор должен преобразовать эти контрпримеры в читаемую форму, например в виде конкретных входных значений, помогая пользователю в отладке.

### 7.5 Runtime-проверки

В режиме `--enable-runtime-checks` компилятор преобразует спецификации в операторы `assert`:
- `requires`: вставляется `assert(cond)` при входе в функцию
- `ensures`: вставляется `assert(cond)` перед каждой точкой возврата, где `result` заменён на фактическое возвращаемое значение
- `invariant`: вставляется `assert(cond)` в начале тела цикла

### 7.6 Интеграция с существующим проектированием

- **Модель владения**: выражения в спецификациях подчиняются правилам владения, доступны только для чтения (чистые функции), без побочных эффектов
- **Обобщённая система**: спецификационные типы поддерживают параметры-типы (например, `Requires(P)`), могут сочетаться с обобщёнными функциями/типами
- **Зависимые типы**: значения зависимых типов в спецификациях (например, длина массива `n`) естественно доступны

### Влияние на систему типов

- Спецификационные типы являются обычными типами YaoXiang, соответствующими унифицированной модели синтаксиса
- Компилятор предоставляет встроенные распространённые спецификационные типы (`Positive`, `NonEmpty`, `GreaterOrEqual` и др.)
- Пользователи могут определять собственные спецификационные типы через обычные определения типов
- Спецификационные типы поддерживают параметры-типы и ограничения типов

### Поведение во время выполнения

- **Debug Build**: вызов SMT-решателя для статической верификации, увеличение времени компиляции; после успешной верификации кэширование результатов
- **Release Build**: спецификационные комментарии игнорируются, нулевые накладные расходы; очистка всех кэшей верификации; включение агрессивных оптимизаций и очистки Span-кэша
- **Режим runtime-проверок**: генерация операторов assert, обнаружение нарушений во время выполнения

### Изменения в компиляторе

- Парсер: распознавание синтаксиса спецификационных комментариев
- Семантический анализ: сбор спецификаций, преобразование в спецификационные типы
- Верификационный backend: генерация условий верификации, вызов SMT-решателя
- Генерация кода: поддержка режима runtime-проверок

### Обратная совместимость

- ✅ Полная обратная совместимость
- Спецификационные комментарии игнорируются при обычной компиляции, не затрагивают существующий код
- При релизной сборке спецификации игнорируются, отсутствуют дополнительные накладные расходы

## Компромиссы

### Преимущества

- **Верификация при Debug Build**: обязательная верификация при отладочной сборке обеспечивает логическую корректность
- **Элегантный синтаксис**: исключительно комментарии, без новых ключевых слов, дружелюбно к редакторам
- **Интеграция с системой типов**: спецификации являются типами, расширяемы
- **Постепенное внедрение**: возможен переход от runtime-проверок к статической верификации
- **Повышение надёжности**: обнаружение логических ошибок, трудно выявляемых тестированием

### Недостатки

- **Время компиляции**: режим верификации может существенно увеличить время компиляции
- **Порог вхождения**: требуется изучение эффективного написания спецификаций и кванторов
- **Ограничения SMT-решателей**: некоторые сложные свойства могут не поддаваться автоматическому доказательству

## Альтернативные решения

| Решение | Преимущества | Недостатки |
|------|------|------|
| Новые ключевые слова (например, `requires`) | Интуитивный синтаксис | Введение новых ключевых слов, нарушение лаконичности |
| Отдельные файлы спецификаций (например, CVL) | Разделение спецификаций и кода | Увеличение количества файлов, сложность синхронизации |
| Только runtime-ассерты | Простая реализация | Отсутствие статических гарантий |
| **Наше решение (комментарии + спецификационные типы)** | Баланс лаконичности и функциональности | Требуется поддержка редакторов |

## Стратегия реализации

### Фазы

| Фаза | Содержание |
|------|------|
| **Фаза 1: Базовая поддержка** | Расширение парсера, распознавание комментариев `//!` и `/*! ... !*/`, присоединение к AST-узлам; в режиме верификации сбор спецификаций, генерация простых условий верификации (только арифметические сравнения); интеграция Z3-решателя |
| **Фаза 2: Поддержка кванторов** | Поддержка кванторных выражений, трансляция в `forall`/`exists` формата SMT-LIB; подсветка спецификаций в IDE и подсказки при наведении курсора |
| **Фаза 3: Оптимизация и инструментарий** | Инкрементальная верификация, кэширование уже верифицированных модулей; отчёт о покрытии спецификаций; инструменты извлечения спецификаций (генерация候选ных спецификаций из тестов) |

### Зависимости

- RFC-009: Модель владения — выражения в спецификациях требуют семантики чистых функций
- RFC-010: Унифицированный синтаксис типов — система спецификационных типов построена на основе системы типов
- RFC-011: Проектирование обобщённой системы — спецификационные типы поддерживают параметры-типы

### Риски

1. **Сложность интеграции SMT-решателей**: интеграция Z3 и других решателей может столкнуться с техническими трудностями
   - Смягчение: использование зрелых Rust-привязок к Z3, постепенное расширение支持的 типов выражений

2. **Сложность отладки при неудаче верификации**: когда SMT-решатель не может доказать спецификацию, пользователю может быть трудно понять причину
   - Смягчение: предоставление понятных сообщений об ошибках и объяснений контрпримеров

3. **Накладные расходы по производительности**: режим верификации может существенно увеличить время компиляции
   - Смягчение: реализация инкрементальной верификации и механизма кэширования

## Жизненный цикл и судьба

```
┌─────────────┐
│   Черновик   │  ← Создано автором
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ На рассмотрении │  ← Обсуждение сообществом
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  Принято    │    │  Отклонено  │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│ accepted/  │    │    rfc/     │
│ (официальный дизайн) │    │ (оставлено на месте) │
└─────────────┘    └─────────────┘
```

### Пояснение статусов

| Статус | Расположение | Пояснение |
|------|------|------|
| **Черновик** | `docs/design/rfc/` | Черновик автора, ожидает подачи на рассмотрение |
| **На рассмотрении** | `docs/design/rfc/` | Открыто обсуждение и обратная связь сообщества |
| **Принято** | `docs/design/accepted/` | Становится официальным документом проектирования, переходит в фазу реализации |
| **Отклонено** | `docs/design/rfc/` | Остаётся в каталоге RFC, статус обновлён |

### Действия после принятия

1. Переместить RFC в каталог `docs/design/accepted/`
2. Обновить имя файла на описательное (например, `hoare-logic-static-verification.md`)
3. Обновить статус на «Официальный»
4. Обновить статус на «Принято», добавить дату принятия

### Действия после отклонения

1. Оставить в каталоге `docs/design/rfc/`
2. Добавить в начало файла причину отклонения и дату
3. Обновить статус на «Отклонено»
