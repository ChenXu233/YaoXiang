# RFC-032: единый модификатор выражений spawn

> **Настоящий документ определяет синтаксис `spawn`, рефакторизацию AST/IR и расширения системы типов**.
> Семантика поведения во время выполнения (гранулярность разбиения задач, владение, области видимости, распространение ошибок, типы ресурсов, вложенность) приведена в [RFC-024: семантика параллельного времени выполнения на основе spawn](./024-concurrency-model.md).
>
> Два RFC совместно определяют `spawn` — 024 отвечает на вопрос «что делать», 032 — «как представлять».

> **Ключевое наблюдение**: `spawn` не должен модифицировать только блок `{}`. Он может модифицировать **произвольное выражение**. `spawn for` — это не специальный синтаксис, а естественная комбинация `spawn` + выражения `for`.

## Резюме

`spawn` расширяется с `spawn { }` (модифицирует только блок) до `spawn <expr>` (модифицирует произвольное выражение). `Expr::SpawnFor` удаляется из AST и естественным образом заменяется на `Expr::Spawn { body: Expr::For { .. } }`. Типы структур выражений (Block, For, While, If и т. д.) добавляются в систему типов в виде новых вариантов `MonoType`. `Spawn<T>` оборачивает структуру параллельно исполняемого вычисления, помечается на этапе компиляции и стирается после проверки типов.


## Мотивация

### Зачем нужны эти изменения?

В настоящее время `spawn for x in items { body }` является самостоятельной комбинацией ключевых слов, а в AST для её представления существует специальный вариант `Expr::SpawnFor`. Это нарушает ортогональность языка:

1. **Неединообразный синтаксис**: `spawn` может модифицировать только блок `{}`, а `spawn for` — жёстко закодированное исключение.
2. **Отсутствие ортогональности**: комбинации `spawn while`, `spawn if` и т. д. не выражаются естественно.
3. **Неполная система типов**: spawn невидим в системе типов, невозможно через отражение получить параллельную структуру.

### Текущие проблемы

```rust
// В AST два варианта spawn
Spawn { body: Box<Block>, span: Span },         // spawn { ... }
SpawnFor { var, var_mut, iterable, body, span },  // spawn for x in items { ... }

// MonoType содержит только типы значений, без типов вычислительных структур
// тип spawn { a, b } = Tuple(T_a, T_b)  ← информация «это spawn» потеряна
// тип spawn for      = List(T)          ← информация «это параллелизм по данным» потеряна
```

## Предложение

### Основная идея

`spawn <expr>`: `spawn` модифицирует произвольное выражение. Форма выражения определяет, как DAG разбивает задачи.

**Всё есть тип**: `MonoType` расширяется с «тип значений» до «тип значения + тип вычислительной структуры». Каждая ключевая структура выражения получает соответствующий вариант в системе типов. `Spawn<T>` оборачивает структуру параллельно исполняемого вычисления.

### Ментальная модель пользователя

`spawn` = «выполни это выражение параллельно». Форма выражения определяет способ разбиения:

| Форма выражения | Поведение при параллелизме | Тип |
|-----------|---------|------|
| `spawn { a, b, c }` | `a`, `b`, `c` выполняются независимо и параллельно | `Spawn(Block(Tuple(T_a, T_b, T_c)))` |
| `spawn for x in items { f(x) }` | N итераций выполняются независимо и параллельно | `Spawn(ForExpr { body_ty: List(T) })` |
| `spawn while cond { step() }` | Каждая итерация — отдельная задача | `Spawn(WhileExpr { body_ty: List(T) })` |
| `spawn if c { a } else { b }` | Выбранная ветвь целиком — домен spawn | `Spawn(IfExpr { then_ty: T_a, else_ty: Some(T_b) })` |
| `spawn call(x)` | Сам вызов — одна задача | `Spawn(Call { fn_ty: Fn(A→R), result_ty: R })` |
| `spawn 42` | Одна задача | `Spawn(Int)` |

Компилятор отвечает за анализ DAG для определения зависимостей, а среда выполнения планирует по модели GMP — задачи без зависимостей помещаются в рабочую очередь, воркеры захватывают их. Общая синхронная блокировка — ожидание завершения всех задач.

**Отличие от Go**: `go` в Go — «запустил и забыл», `spawn` в YaoXiang — «разбить, выполнить параллельно, подождать завершения всех».

### Ортогональность потока управления

| Комбинация | Семантика | Различие |
|------|------|------|
| `spawn for x in items { body }` | Параллелизм по данным: каждая итерация — независимая задача | DAG анализирует зависимости между итерациями |
| `for x in items spawn { body }` | На каждой итерации создаётся домен spawn | Без анализа между итерациями |
| `spawn while cond { body }` | Условный параллелизм: каждая итерация — независимая задача | Межитерационные зависимости гарантируются условием |
| `while cond spawn { body }` | На каждой итерации создаётся домен spawn | Семантика отличается, но не требует специальной обработки |
| `spawn if c { a } else { b }` | Весь if-else — один домен spawn | При выполнении выбирается ветвь по условию |
| `if c spawn { a } else { b }` | Только одна ветвь — spawn | Внутри выражения if оборачивается spawn |

### Устраняемая сложность

- ❌ `Expr::SpawnFor` удаляется из AST
- ❌ `SpawnForAnalysis` удаляется из DAG-анализа
- ❌ `spawn for` больше не обрабатывается в парсере как специальная комбинация ключевых слов
- ❌ `Ir::SpawnFor` удаляется из IR

## Детальное проектирование

### 1. Уровень AST

**До:**

```rust
Spawn { body: Box<Block>, span: Span },         // spawn { ... }
SpawnFor { var, var_mut, iterable, body, span },  // spawn for x in items { ... }
```

**После:**

```rust
Spawn { body: Box<Expr>, span: Span },           // spawn <произвольное выражение>
```

`Expr::SpawnFor` удаляется. AST-представление `spawn for x in items { body }`:

```rust
Expr::Spawn {
    body: Box::new(Expr::For {
        var: "x",
        iterable: items,
        body: body_block,
        ..
    })
}
```

**Особый случай IF**:

| Запись | Структура AST |
|------|---------|
| `spawn if cond { a } else { b }` | `Spawn { body: Expr::If { ... } }` |
| `if cond spawn { a } else { b }` | `Expr::If { then: Spawn { body: {a} }, else: {b} }` |

Семантика различается, но обе формы — естественные комбинации, не требующие специальных правил.

### 2. Уровень парсера

`spawn` имеет самый низкий приоритет связывания (аналог `return`) и захватывает всё последующее выражение:

```
spawn a + b        →  spawn (a + b)         ≠  (spawn a) + b
spawn f(x).y       →  spawn (f(x).y)
```

Изменения в парсере: в `pratt/nud.rs` `spawn` больше не требует `{`, а вызывает общий парсер выражений:

```
token spawn → parse_expr(min_precedence) → Expr::Spawn { body: expr }
```

`spawn for` больше не обрабатывается как комбинация ключевых слов — `for` обрабатывается общим парсером выражений и порождает `Expr::For`, а `spawn` только оборачивает.

### 3. Система типов

**Новые варианты `MonoType`:**

```rust
// ========== Типы вычислительных структур ==========

/// Выражение-блок {}
Block(Box<MonoType>),

/// Выражение цикла for
ForExpr { body_ty: Box<MonoType> },

/// Выражение цикла while
WhileExpr { body_ty: Box<MonoType> },

/// Выражение ветвления if-else
IfExpr {
    then_ty: Box<MonoType>,
    else_ty: Option<Box<MonoType>>,
},

/// Выражение вызова функции
Call {
    fn_ty: Box<MonoType>,
    result_ty: Box<MonoType>,
},

/// Параллельная обёртка spawn: внутреннее выражение исполняется параллельно
/// Помечается на этапе компиляции, стирается после проверки типов
Spawn(Box<MonoType>),
```

**Правила вывода типов**: вывод типа каждого выражения возвращает «тип вычислительной структуры». Без обёртки `Spawn` — последовательное выполнение, с обёрткой `Spawn` — параллельное. После проверки типов `Spawn` стирается, тип сводится к внутреннему типу значения.

**Процесс проверки типов**:
1. Вывести тип T выражения body (тип вычислительной структуры)
2. Если body обёрнут в spawn, обернуть в `Spawn(T)`
3. При выводе типа присваивания — деструктурировать: `results: List(Data) = spawn for ... {}` — из `Spawn(ForExpr { body_ty: List(Data) })` извлекается `List(Data)`

`Spawn<T>` стирается после проверки типов, среде выполнения не нужно знать, получены ли данные параллельно или последовательно. Но при отражении на этапе компиляции (`type_of(x)`) доступна полная топология параллелизма.

### 4. Уровень анализа DAG

Два текущих входа объединяются в один:

```rust
/// Единый вход: диспетчеризация по типу выражения body
fn analyze_spawn_expr(body: &Expr, ...) -> SpawnAnalysis {
    match body {
        Expr::Block(block)       => analyze_block_tasks(block, ...),
        Expr::For { .. }         => analyze_iter_tasks(IterKind::For, body, ...),
        Expr::While { .. }       => analyze_iter_tasks(IterKind::While, body, ...),
        Expr::If { .. }          => analyze_if_task(body, ...),
        _                        => single_task(body, ...),
    }
}
```

**Единая структура результата**:

```rust
struct SpawnAnalysis {
    source: TaskSource,
    plan: ExecutionPlan,
}

enum TaskSource {
    /// spawn { a, b, c } — N заранее известных прямых подвыражений
    Explicit(Vec<TaskInfo>),
    /// spawn for/while — N задач порождается итерациями во время выполнения
    Iterate {
        kind: IterKind,
        iter_var: String,
        iterable: Option<Expr>,      // есть у for, нет у while
        condition: Option<Expr>,     // есть у while, нет у for
        body: Block,
        reads: HashSet<String>,
        writes: HashSet<String>,
        resource_vars: HashSet<String>,
    },
}

enum IterKind { For, While }
```

Структура `SpawnForAnalysis` удаляется.

| Вид body | Как разбивается на задачи |
|-----------|--------------|
| `Expr::Block` | Прямые подвыражения → список задач |
| `Expr::For` | Каждая итерация → одна задача (параллелизм по данным) |
| `Expr::While` | Каждая итерация → одна задача |
| `Expr::If` | Выбранная ветвь целиком → одна задача |
| `Expr::Call` / другое | Само выражение → одна задача |

После завершения DAG-анализа среда выполнения планирует по модели GMP — задачи без зависимостей помещаются в рабочую очередь, воркеры захватывают их.

### 5. Уровень IR / Codegen

`Ir::SpawnFor` удаляется. Унифицирован в `Ir::Spawn`, несущий информацию `TaskSource`.

Трансляция HIR → IR на основе `SpawnAnalysis.source` генерирует вызовы среды выполнения:
- `TaskSource::Explicit(tasks)` → заранее известный на этапе компиляции список задач
- `TaskSource::Iterate { .. }` → разворачивается во время выполнения (управляется компилятором, по аналогии с par_iter, но с нулевыми накладными расходами)

### 6. Уровень Placement

Две текущие ветви объединяются в одну:

```rust
// До
Expr::Spawn { body, .. } => self.check_block(body),
Expr::SpawnFor { body, iterable, .. } => {
    self.check_expr(iterable);
    self.check_block(body);
}

// После
Expr::Spawn { body, .. } => self.check_expr(body),   // body — это Expr, рекурсия
```

### 7. Обратная совместимость

Семантика существующего кода `spawn for` не меняется, парсер автоматически разбирает `spawn for x in items { body }` как `Expr::Spawn { body: Expr::For }`. Внутреннее представление меняется, видимое пользователем поведение остаётся прежним.

Новый синтаксис получается естественно:
```yx
spawn while has_next() {
    item = next()
    process(item)
}

spawn if use_cache {
    load_from_cache(key)
} else {
    fetch(key)
}
```

## Компромиссы

### Преимущества

1. **Ортогональность синтаксиса**: `spawn` + произвольный поток управления = естественная параллельная комбинация.
2. **Всё есть тип**: система типов полностью фиксирует вычислительную структуру, отражение на этапе компиляции даёт топологию параллелизма.
3. **Устранение особых случаев**: удаление `Expr::SpawnFor` и связанного специального кода.
4. **Расширяемость**: новые структуры потока управления в будущем автоматически комбинируются с `spawn` без модификации логики spawn.

### Недостатки

1. **Разрастание системы типов**: 6 новых вариантов `MonoType` увеличивают сложность проверки типов.
2. **Критическое изменение**: меняется внутреннее представление AST/IR, требуется обновить весь код, потребляющий `Expr::SpawnFor`.
3. **Вывод типов выражений**: каждое выражение теперь должно возвращать тип вычислительной структуры — большой радиус воздействия.

## Альтернативы

| Альтернатива | Почему не выбрана |
|------|-------------|
| Сохранить `spawn for` как самостоятельный синтаксис | Нарушает ортогональность, создаёт единственный в языке специальный случай комбинации ключевых слов. |
| `spawn` модифицирует только `{}`, параллелизм по данным — через библиотечный `par_iter` | Первичная возможность языка опускается в библиотеку, теряется DAG-анализ и обнаружение конфликтов ресурсов на уровне компилятора. |
| Удалить `SpawnFor`, но не вводить типы вычислительных структур | Система типов теряет рефлексию, spawn становится невидимым на уровне типов. |


## Связь с RFC-019

6 вариантов `MonoType`, вводимых этим RFC (Block/ForExpr/WhileExpr/IfExpr/Call/Spawn), являются **встроенным подмножеством компилятора** для [RFC-019: типоуровневая гомиконичность](./019-typed-homoiconicity.md). Ключевая идея RFC-019 «синтаксические структуры попадают в систему типов» реализуется здесь как: 6 вычислительных структур, понимаемых компилятором изначально, получают соответствующее типовое представление. Пользователь не может через `SyntaxRule` определять новые типы вычислительных структур, но эти 6 встроенных покрывают все ключевые потоки управления.

## Интеграция с пайплайном доказательств

Причина существования 6 вариантов `MonoType`: они сообщают [пайплайну доказательств этапа компиляции из RFC-027](../accepted/027-compile-time-evaluation-types.md) **какую форму имеет доказываемое утверждение**. Сам пайплайн отвечает за фактическое доказательство (анализ свободных переменных, классификация эффектов, анализ алиасов, обнаружение конфликтов), а `MonoType` выполняет лишь одну задачу — предоставляет структурированный интерфейс ввода.

### Отображение вариантов в утверждения

| Тип | Форма утверждения | Стратегия доказательства |
|------|---------|---------|
| `Spawn(ForExpr { body_ty })` | Параллелизм по данным: N итерационных задач без межитерационных конфликтов | Извлечь свободные переменные body → классификация эффектов → проверить отсутствие Write(Shared) / `&mut`(Shared) |
| `Spawn(WhileExpr { body_ty })` | Условный параллелизм: каждая итерация независима + отсутствие межитерационных причинно-следственных зависимостей | То же + проверка наличия межитерационных побочных эффектов в условии |
| `Spawn(Block(T))` | Явная группа задач: зависимости между задачами задаёт DAG | Проверить граф зависимостей из DAG-анализа — входные данные каждой задачи готовы к моменту её запуска |
| `Spawn(IfExpr { then_ty, else_ty })` | Ветвевой spawn: выбранная ветвь целиком — один домен spawn | Выбор ветви без конфликтов, рекурсивная проверка внутри body |
| `Spawn(Call { fn_ty, result_ty })` | Вызов spawn: вызываемая функция — отдельная задача | Проверка чистоты или изолированности функции |
| `Spawn(T)` (значение, например `spawn 42`) | Одиночный spawn: без параллелизма | Тривиально проходит |

### Сценарии доказательств

**Сценарий 1 — чистый параллелизм по данным (успех):**

```yaoxiang
items = [1, 2, 3, 4, 5]
results = spawn for item in items { item * 2 }
// Тип: Spawn(ForExpr { body_ty: List(Int) })
```

1. Свободные переменные: `item` (локальная цикла, независимая копия на итерацию), `items` (внешняя, только чтение в body)
2. Классификация эффектов: все Read(Local) или Read(Shared), записей нет
3. Доказано ✓

**Сценарий 2 — только чтение разделяемого (успех):**

```yaoxiang
config = load_config()
results = spawn for item in items { process(item, config) }
// Тип: Spawn(ForExpr { body_ty: List(Result) })
```

1. Свободные переменные: `item` (Read(Local)), `config` (внешняя, в body нет путей записи → Read(Shared))
2. Классификация эффектов: всё только для чтения
3. Доказано ✓

**Сценарий 3 — конфликт записи (отказ):**

```yaoxiang
mut counter = 0
spawn for item in items { counter += 1 }
```

1. Свободные переменные: `item` (Read(Local)), `counter` (внешняя, `+=` десахарируется в запись)
2. Классификация эффектов: `counter` — Write(Shared), запись в одну и ту же память из разных итераций
3. Инстанциация конфликта: `Write(task_0, counter) ∧ Write(task_1, counter) = True`
4. Не доказано ✗ → ошибка компиляции: `Ошибка: в body spawn for обнаружен межитерационный конфликт записи. Переменная counter записывается несколькими параллельными задачами.`

**Сценарий 4 — while + итератор с состоянием (предупреждение/отказ):**

```yaoxiang
spawn while iter.has_next() {
    item = iter.next()
    process(item)
}
// Тип: Spawn(WhileExpr { body_ty: List(Processed) })
```

1. Свободные переменные: `iter` (внешняя, `next()` → `&mut self` → `&mut`(Shared))
2. `next()` изменяет состояние итератора, итерация N+1 зависит от побочного эффекта итерации N
3. Это не независимые задачи → нарушение ограничения независимости `Spawn(WhileExpr)`
4. Компилятор сообщает о межитерационной причинно-следственной зависимости, предлагает заменить на `spawn for`

**Сценарий 5 — spawn if (успех):**

```yaoxiang
result = spawn if use_cache { load(key) } else { fetch(key) }
// Тип: Spawn(IfExpr { then_ty: T, else_ty: Option(T) })
```

1. Выполняется только одна ветвь, межзадачных конфликтов нет
2. Если в body есть вложенный spawn — рекурсивная проверка
3. Доказано ✓

**Сценарий 6 — межзадачные зависимости в spawn-блоке (DAG + верификация пайплайном):**

```yaoxiang
spawn {
    a = fetch_user(id)
    b = fetch_orders(a.user_id)  // зависит от a
    c = compute_stats()           // независимая
}
// Тип: Spawn(Block(Tuple(User, Orders, Stats)))
```

1. DAG-анализ: `a` и `c` независимы (можно параллельно), `b` зависит от `a` (планируется после `a`)
2. Верификация пайплайном: вход `b` (`a.user_id`) вычислен к моменту запуска `b`
3. Доказано ✓

### Чего MonoType не делает

| Что делает | Чего не делает |
|--------|---------|
| Идентифицирует форму утверждения | Не выполняет доказательство |
| Фиксирует вычислительную структуру на уровне типов | Не заменяет DAG-анализ |
| Предоставляет пайплайну RFC-027 типизированный ввод | Не заменяет анализ свободных переменных, анализ алиасов, обнаружение конфликтов |

Фактическая работа по доказательству выполняется стандартными проходами анализа компилятора. Ценность `MonoType` в том, что эти проходы диспетчеризируются в единой типовой структуре — пайплайну доказательств не нужны специальные ветви для каждого вида узла AST.

## Стратегия реализации

### Этапы

1. **AST + парсер**: `Spawn { body: Box<Expr> }`, удаление `SpawnFor`
2. **Система типов**: 6 новых вариантов `MonoType`, все выражения возвращают тип вычислительной структуры при выводе
3. **Унификация DAG-анализа**: объединение входов, слияние `TaskSource` в одно перечисление
4. **Адаптация IR / Codegen**: удаление `Ir::SpawnFor`, унификация путей обработки
5. **Упрощение Placement**: удаление ветви `SpawnFor`
6. **Тестирование**: все существующие тесты `spawn for` проходят

### Радиус воздействия

| Файл/каталог | Изменения |
|-----------|------|
| `frontend/core/parser/ast.rs` | `Spawn` body заменён на `Box<Expr>`, удалён `SpawnFor` |
| `frontend/core/parser/pratt/nud.rs` | Обработчик `spawn` упрощён до общего парсера выражений |
| `frontend/core/types/mono.rs` | Добавлены варианты `Block`/`ForExpr`/`WhileExpr`/`IfExpr`/`Call`/`Spawn` |
| `frontend/core/spawn/analysis.rs` | Единый вход, `TaskSource` объединяет Explicit + Iterate |
| `frontend/core/spawn/placement.rs` | Удалена ветвь `SpawnFor` |
| `frontend/core/typecheck/` | Все узлы выражений адаптированы к выводу типов вычислительной структуры |
| `middle/core/ir.rs` | Удалён `Ir::SpawnFor` |
| `middle/` (IR gen, codegen) | Унификация пути spawn, стирание типа Spawn |
| `tests/yaoxiang/04-concurrency/spawn_for.yx` | Семантика не меняется, проверяется прохождение |

### Зависимости

- RFC-024 (модель параллелизма на основе spawn-блоков) — этот RFC является её ортогональным расширением
- RFC-010 (унифицированный синтаксис типов) — основа изменений системы типов
- RFC-027 (пайплайн доказательств этапа компиляции) — варианты `MonoType` предоставляют пайплайну ввод о форме утверждений
- RFC-019 (типоуровневая гомиконичность) — варианты `MonoType` — его встроенное подмножество компилятора

## Записи о проектных решениях

| Решение | Выбор | Причина | Дата |
|------|------|------|------|
| Область модификации spawn | Произвольное выражение | Устранение особого случая `spawn for` | 2026-06-16 |
| Поддержка `spawn while` | Поддерживается | Ортогональность синтаксиса, низкая стоимость реализации | 2026-06-16 |
| Семантика `spawn if` | Модифицирует весь if-else | Различие с `if spawn { }` | 2026-06-16 |
| Система типов | Вводятся типы вычислительных структур | «Всё есть тип», поддержка отражения этапа компиляции | 2026-06-16 |
| Стирание типа spawn | Стирается после проверки типов | Среде выполнения не нужна информация о структуре параллелизма | 2026-06-16 |
| Приоритет связывания spawn | Самый низкий (как return) | Захватывает всё последующее выражение | 2026-06-16 |
| DAG для внутренностей for | Не раскрывать подвыражения внутри for | Правило прямых подвыражений не меняется, for целиком — источник задачи | 2026-06-16 |
| Интеграция с пайплайном доказательств | Варианты `MonoType` отображаются в утверждения пайплайна RFC-027 | Пайплайну нужно знать форму проверяемого утверждения, `MonoType` предоставляет структурированный ввод | 2026-07-03 |
| Связь с RFC-019 | Встроенное подмножество компилятора | Пользователь не может расширять, но разделяет идею «синтаксис = тип» | 2026-07-03 |
| Границы доказательства | 6 сценариев: чистый параллелизм / Read(Shared) / конфликт записи / зависимость в while / spawn if / spawn block | Явные обязательства и условия отказа для каждого варианта `MonoType` | 2026-07-03 |
