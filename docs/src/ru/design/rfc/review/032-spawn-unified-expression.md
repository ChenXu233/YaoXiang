# RFC-032: унифицированный модификатор выражений spawn

> **Ключевое прозрение**: `spawn` не должен модифицировать только блок `{}`. Он может модифицировать **произвольное выражение**. `spawn for` — это не особый синтаксис, а естественная комбинация `spawn` + выражения `for`.

## Резюме

Расширить `spawn` от `spawn { }` (модифицирует только блок) до `spawn <expr>` (модифицирует произвольное выражение). `Expr::SpawnFor` удаляется из AST и естественно заменяется на `Expr::Spawn { body: Expr::For { .. } }`. Структурные типы выражений (Block, For, While, If и т. д.) входят в систему типов как новые варианты `MonoType`, `Spawn<T>` оборачивает параллельно исполняемую вычислительную структуру, помечается на этапе compile-time и стирается после проверки типов.

## Мотивация

### Зачем нужно это изменение?

В настоящее время `spawn for x in items { body }` представляет собой самостоятельную комбинацию ключевых слов, для которой в AST выделен специальный узел `Expr::SpawnFor`. Это нарушает ортогональность языка:

1. **Неединообразие синтаксиса**: `spawn` может модифицировать только блок `{}`, а `spawn for` — жёстко закодированное исключение.
2. **Отсутствие ортогональности**: комбинации вроде `spawn while`, `spawn if` и т. п. невозможно выразить естественно.
3. **Неполнота системы типов**: spawn невидим в системе типов, поэтому невозможно получить параллельную структуру через рефлексию типов.

### Текущая проблема

```rust
// Два варианта spawn в AST
Spawn { body: Box<Block>, span: Span },         // spawn { ... }
SpawnFor { var, var_mut, iterable, body, span },  // spawn for x in items { ... }

// MonoType содержит только типы значений, но не типы вычислительных структур
// Тип spawn { a, b } = Tuple(T_a, T_b)  ← информация «это spawn» потеряна
// Тип spawn for     = List(T)             ← информация «это data-parallel» потеряна
```

## Предложение

### Основной замысел

`spawn <expr>`: `spawn` модифицирует произвольное выражение. Форма выражения определяет, как DAG разбивает задачи.

**Всё есть тип**: `MonoType` расширяется с «типов значений» до «типов значений + типов вычислительных структур». Каждой ключевой структуре выражения соответствует вариант типа. `Spawn<T>` оборачивает параллельно исполняемую вычислительную структуру.

### Ментальная модель пользователя

`spawn` = «выполни это выражение параллельно». Форма выражения определяет способ разбиения:

| Форма выражения | Параллельное поведение | Тип |
|-----------------|----------------------|-----|
| `spawn { a, b, c }` | `a`, `b`, `c` независимо и параллельно | `Spawn(Block(Tuple(T_a, T_b, T_c)))` |
| `spawn for x in items { f(x) }` | N итераций независимо и параллельно | `Spawn(ForExpr { body_ty: List(T) })` |
| `spawn while cond { step() }` | Каждая итерация — отдельная задача | `Spawn(WhileExpr { body_ty: List(T) })` |
| `spawn if c { a } else { b }` | Выбранная ветвь целиком — область spawn | `Spawn(IfExpr { then_ty: T_a, else_ty: Some(T_b) })` |
| `spawn call(x)` | Сам вызов — одна задача | `Spawn(Call { fn_ty: Fn(A→R), result_ty: R })` |
| `spawn 42` | Одна отдельная задача | `Spawn(Int)` |

Компилятор отвечает за DAG-анализ для определения зависимостей, а среда выполнения планирует по модели GMP — задачи без зависимостей попадают в очередь работ, и воркеры разбирают их. Общая синхронизация — блокирующая: ожидается завершение всех задач.

**Отличие от Go**: `go` в Go — «бросил и забыл», тогда как `spawn` в YaoXiang — «разбить на параллельное выполнение, подождать завершения всех частей и продолжить».

### Ортогональность управляющих конструкций

| Комбинация | Семантика | Различие |
|------------|-----------|----------|
| `spawn for x in items { body }` | Параллелизм по данным: каждая итерация = отдельная задача | DAG-анализ зависимостей между итерациями |
| `for x in items spawn { body }` | На каждой итерации создаётся область spawn | Без межитерационного анализа |
| `spawn while cond { body }` | Параллелизм по условию: каждая итерация = отдельная задача | Межитерационные зависимости гарантируются условием |
| `while cond spawn { body }` | На каждой итерации создаётся область spawn | Семантически отличается от вышеописанной, специальной обработки не требуется |
| `spawn if c { a } else { b }` | Весь if-else — единая область spawn | При выполнении выбирается ветвь по условию |
| `if c spawn { a } else { b }` | spawn только в одной ветви | Внутри if-выражения вложен spawn |

### Устраняемая сложность

- ❌ `Expr::SpawnFor` удаляется из AST
- ❌ `SpawnForAnalysis` удаляется из DAG-анализа
- ❌ `spawn for` больше не обрабатывается в парсере как комбинация ключевых слов
- ❌ `Ir::SpawnFor` удаляется из IR

## Детальное проектирование

### 1. Уровень AST

**Ранее:**

```rust
Spawn { body: Box<Block>, span: Span },         // spawn { ... }
SpawnFor { var, var_mut, iterable, body, span },  // spawn for x in items { ... }
```

**После:**

```rust
Spawn { body: Box<Expr>, span: Span },           // spawn <произвольное выражение>
```

`Expr::SpawnFor` удаляется. AST-представление `spawn for x in items { body }`:

```rust
Expr::Spawn {
    body: Box::new(Expr::For {
        var: "x",
        iterable: items,
        body: body_block,
        ..
    })
}
```

**Особый случай с IF**:

| Запись | Структура AST |
|--------|---------------|
| `spawn if cond { a } else { b }` | `Spawn { body: Expr::If { ... } }` |
| `if cond spawn { a } else { b }` | `Expr::If { then: Spawn { body: {a} }, else: {b} }` |

Семантика различается, но обе записи — естественные комбинации, специальных правил не требуется.

### 2. Уровень парсера

`spawn` имеет наименьший приоритет связывания (как `return`) и поглощает всё последующее выражение:

```
spawn a + b        →  spawn (a + b)         ≠  (spawn a) + b
spawn f(x).y       →  spawn (f(x).y)
```

Изменения в парсере: в `pratt/nud.rs` `spawn` больше не требует `{`, а вызывает общий разбор выражений:

```
token spawn → parse_expr(min_precedence) → Expr::Spawn { body: expr }
```

`spawn for` больше не обрабатывается как комбинация ключевых слов — `for` обрабатывается общим парсером выражений и порождает `Expr::For`, а `spawn` только выполняет оборачивание.

### 3. Система типов

**Новые варианты `MonoType`:**

```rust
// ========== Типы вычислительных структур ==========

/// Выражение-блок {}
Block(Box<MonoType>),

/// Выражение цикла for
ForExpr { body_ty: Box<MonoType> },

/// Выражение цикла while
WhileExpr { body_ty: Box<MonoType> },

/// Выражение ветвления if-else
IfExpr {
    then_ty: Box<MonoType>,
    else_ty: Option<Box<MonoType>>,
},

/// Выражение вызова функции
Call {
    fn_ty: Box<MonoType>,
    result_ty: Box<MonoType>,
},

/// Параллельная обёртка spawn: внутреннее выражение исполняется параллельно
/// Помечается на compile-time, стирается после проверки типов
Spawn(Box<MonoType>),
```

**Правила вывода типов**: вывод типа каждого выражения возвращает «тип вычислительной структуры». Без обёртки `Spawn` — последовательное выполнение, с обёрткой `Spawn` — параллельное. После проверки типов `Spawn` стирается, и тип понижается до внутреннего типа значения.

**Ход проверки типов**:
1. Вывести тип T тела выражения (тип вычислительной структуры)
2. Если обёрнуто в spawn, обернуть в `Spawn(T)`
3. При выводе типа присваивания деструктурировать: `results: List(Data) = spawn for ... {}` — извлечь `List(Data)` из `Spawn(ForExpr { body_ty: List(Data) })`

`Spawn<T>` стирается после проверки типов: среде выполнения не нужно знать, поступили ли данные параллельно или последовательно. Однако compile-time рефлексия (`type_of(x)`) может получить полную параллельную топологию.

### 4. Уровень DAG-анализа

Два входных точки сливаются в одну:

```rust
/// Единая точка входа: диспетчеризация по типу тела выражения
fn analyze_spawn_expr(body: &Expr, ...) -> SpawnAnalysis {
    match body {
        Expr::Block(block)       => analyze_block_tasks(block, ...),
        Expr::For { .. }         => analyze_iter_tasks(IterKind::For, body, ...),
        Expr::While { .. }       => analyze_iter_tasks(IterKind::While, body, ...),
        Expr::If { .. }          => analyze_if_task(body, ...),
        _                        => single_task(body, ...),
    }
}
```

**Единая структура результата**:

```rust
struct SpawnAnalysis {
    source: TaskSource,
    plan: ExecutionPlan,
}

enum TaskSource {
    /// spawn { a, b, c } — N прямых подвыражений, известных на этапе компиляции
    Explicit(Vec<TaskInfo>),
    /// spawn for/while — N задач порождаются итерациями в среде выполнения
    Iterate {
        kind: IterKind,
        iter_var: String,
        iterable: Option<Expr>,      // есть у for, нет у while
        condition: Option<Expr>,     // есть у while, нет у for
        body: Block,
        reads: HashSet<String>,
        writes: HashSet<String>,
        resource_vars: HashSet<String>,
    },
}

enum IterKind { For, While }
```

Структура `SpawnForAnalysis` удаляется.

| Вид тела | Как разбивается на задачи |
|----------|---------------------------|
| `Expr::Block` | Прямые подвыражения → список задач |
| `Expr::For` | Каждая итерация → одна задача (параллелизм по данным) |
| `Expr::While` | Каждая итерация → одна задача |
| `Expr::If` | Выбранная ветвь целиком → одна задача |
| `Expr::Call` / прочее | Само выражение → одна задача |

После завершения DAG-анализа среда выполнения планирует по модели GMP — задачи без зависимостей попадают в очередь работ, воркеры разбирают их.

### 5. Уровень IR / Codegen

`Ir::SpawnFor` удаляется. Всё объединяется в `Ir::Spawn`, несущий информацию `TaskSource`.

Трансляция HIR → IR на основе `SpawnAnalysis.source` генерирует вызовы среды выполнения:
- `TaskSource::Explicit(tasks)` → список задач, известный на этапе компиляции
- `TaskSource::Iterate { .. }` → раскрытие в среде выполнения (управляется компилятором, аналогично par_iter, но с нулевыми накладными расходами)

### 6. Уровень Placement

Две ветви сливаются в одну:

```rust
// Ранее
Expr::Spawn { body, .. } => self.check_block(body),
Expr::SpawnFor { body, iterable, .. } => {
    self.check_expr(iterable);
    self.check_block(body);
}

// После
Expr::Spawn { body, .. } => self.check_expr(body),   // body — это Expr, достаточно рекурсии
```

### 7. Обратная совместимость

Семантика существующего кода `spawn for` остаётся неизменной: парсер автоматически разбирает `spawn for x in items { body }` в `Expr::Spawn { body: Expr::For }`. Внутреннее представление меняется, видимое пользователю поведение — нет.

Новый синтаксис получается естественно:
```yx
spawn while has_next() {
    item = next()
    process(item)
}

spawn if use_cache {
    load_from_cache(key)
} else {
    fetch(key)
}
```

## Компромиссы

### Преимущества

1. **Синтаксическая ортогональность**: `spawn` + любая управляющая конструкция = естественная параллельная комбинация.
2. **Всё есть тип**: система типов полностью фиксирует вычислительную структуру, compile-time рефлексия даёт параллельную топологию.
3. **Устранение особых случаев**: удаляется `Expr::SpawnFor` и связанный с ним специальный код.
4. **Расширяемость**: будущие управляющие конструкции автоматически комбинируются с `spawn` без изменения логики spawn.

### Недостатки

1. **Разрастание системы типов**: 6 новых вариантов `MonoType`, усложнение проверки типов.
2. **Разрушающее изменение**: меняется внутреннее представление AST/IR, требуется обновить весь код, потребляющий `Expr::SpawnFor`.
3. **Вывод типов выражений**: каждое выражение теперь должно возвращать тип вычислительной структуры — широкое влияние.

## Альтернативы

| Альтернатива | Почему отклонена |
|--------------|------------------|
| Сохранить `spawn for` как самостоятельный синтаксис | Нарушает ортогональность, остаётся единственной в языке комбинацией ключевых слов |
| `spawn` модифицирует только `{}`, параллелизм по данным — через стандартную библиотеку `par_iter` | Базовая возможность языка уходит в библиотеку, теряется DAG-анализ и обнаружение конфликтов ресурсов на уровне компилятора |
| Удалить `SpawnFor`, но не вводить в систему типов типы вычислительных структур | Система типов теряет рефлексию, spawn становится невидим на уровне типов |

## Связь с RFC-019

Шесть введённых вариантов `MonoType` (Block/ForExpr/WhileExpr/IfExpr/Call/Spawn) представляют собой **встроенное подмножество компилятора** для [RFC-019: гомоиконичность на уровне типов](./019-typed-homoiconicity.md). Основная идея RFC-019 — «синтаксические структуры входят в систему типов» — реализуется здесь так: 6 вычислительных структур, понимаемых компилятором изначально, получают соответствующие типовые представления. Пользователь не может определять новые типы вычислительных структур через `SyntaxRule`, но эти 6 встроенных покрывают все ключевые управляющие конструкции.

## Интеграция с конвейером доказательств

Причина существования 6 вариантов `MonoType`: они указывают [конвейеру доказательств compile-time из RFC-027](../accepted/027-compile-time-evaluation-types.md) **какую форму имеет доказываемое утверждение**. Сам конвейер выполняет фактическую работу по доказательству (анализ свободных переменных, классификация эффектов, анализ алиасов, обнаружение конфликтов), а `MonoType` делает только одно — предоставляет структурированный интерфейс ввода.

### Сопоставление вариантов с утверждениями

| Тип | Форма утверждения | Стратегия доказательства |
|-----|-------------------|-------------------------|
| `Spawn(ForExpr { body_ty })` | Параллелизм по данным: N итерационных задач без межитерационных конфликтов | Извлечь свободные переменные body → классифицировать эффекты → проверить отсутствие Write(Shared) / `&mut`(Shared) |
| `Spawn(WhileExpr { body_ty })` | Параллелизм по условию: каждая итерация независима + нет межитерационных причинно-следственных зависимостей | То же + проверка, что условие итерации не имеет межитерационных побочных эффектов |
| `Spawn(Block(T))` | Явная группа задач: зависимости между задачами задаются DAG | Проверить DAG-граф зависимостей — входы каждой задачи готовы к её запуску |
| `Spawn(IfExpr { then_ty, else_ty })` | spawn в ветви: выбранная ветвь — единая область spawn | Выбор ветви бесконфликтен, рекурсивная проверка внутри body |
| `Spawn(Call { fn_ty, result_ty })` | spawn вызова: вызываемая функция — отдельная задача | Проверить чистоту или изолированность функции |
| `Spawn(T)` (значение, например `spawn 42`) | spawn одного значения: параллелизма нет | Тривиально проходит |

### Сценарии доказательства

**Сценарий 1 — чистый параллелизм по данным (проходит):**

```yaoxiang
items = [1, 2, 3, 4, 5]
results = spawn for item in items { item * 2 }
// Тип: Spawn(ForExpr { body_ty: List(Int) })
```

1. Свободные переменные: `item` (локальная для цикла, независимая копия на каждую итерацию), `items` (внешняя, в body — только чтение)
2. Классификация эффектов: только Read(Local) или Read(Shared), записей нет
3. Proved ✓

**Сценарий 2 — только чтение разделяемого (проходит):**

```yaoxiang
config = load_config()
results = spawn for item in items { process(item, config) }
// Тип: Spawn(ForExpr { body_ty: List(Result) })
```

1. Свободные переменные: `item` (Read(Local)), `config` (внешняя, в body нет путей записи → Read(Shared))
2. Классификация эффектов: только чтение
3. Proved ✓

**Сценарий 3 — конфликт записи (отклоняется):**

```yaoxiang
mut counter = 0
spawn for item in items { counter += 1 }
```

1. Свободные переменные: `item` (Read(Local)), `counter` (внешняя, `+=` сводится к записи)
2. Классификация эффектов: `counter` — Write(Shared), межитерационная запись в одну и ту же память
3. Инстанциация конфликта: `Write(task_0, counter) ∧ Write(task_1, counter) = True`
4. Disproved ✗ → ошибка компиляции: `Ошибка: в теле spawn for обнаружен межитерационный конфликт записи. Переменная counter записывается несколькими параллельными задачами.`

**Сценарий 4 — while + итератор с состоянием (предупреждение/отклонение):**

```yaoxiang
spawn while iter.has_next() {
    item = iter.next()
    process(item)
}
// Тип: Spawn(WhileExpr { body_ty: List(Processed) })
```

1. Свободные переменные: `iter` (внешняя, `next()` → `&mut self` → `&mut`(Shared))
2. `next()` изменяет состояние итератора, итерация N+1 зависит от побочного эффекта итерации N
3. Это не независимые задачи → нарушает ограничение независимости `Spawn(WhileExpr)`
4. Компилятор сообщает о межитерационной причинно-следственной зависимости и предлагает заменить на `spawn for`

**Сценарий 5 — spawn if (проходит):**

```yaoxiang
result = spawn if use_cache { load(key) } else { fetch(key) }
// Тип: Spawn(IfExpr { then_ty: T, else_ty: Option(T) })
```

1. Выполняется только одна ветвь, межзадачных конфликтов нет
2. При наличии вложенного spawn в body — рекурсивная проверка
3. Proved ✓

**Сценарий 6 — межзадачные зависимости в spawn-блоке (DAG + проверка конвейером):**

```yaoxiang
spawn {
    a = fetch_user(id)
    b = fetch_orders(a.user_id)  // зависит от a
    c = compute_stats()           // независим
}
// Тип: Spawn(Block(Tuple(User, Orders, Stats)))
```

1. DAG-анализ: `a` и `c` независимы (могут выполняться параллельно), `b` зависит от `a` (планируется после `a`)
2. Проверка конвейером: вход `b` (`a.user_id`) готов к моменту запуска `b`
3. Proved ✓

### Чего `MonoType` не делает

| Делает | Не делает |
|--------|-----------|
| Идентифицирует форму утверждения | Не выполняет доказательство |
| Фиксирует вычислительную структуру на уровне типов | Не заменяет DAG-анализ |
| Предоставляет конвейеру RFC-027 типизированный ввод | Не заменяет анализ свободных переменных, анализ алиасов, обнаружение конфликтов |

Фактическую работу по доказательству выполняют стандартные проходы анализа компилятора. Ценность `MonoType` в том, что эти проходы планируются в едином типовом каркасе — конвейеру доказательств не нужны специальные ветви для каждого узла AST.

## Стратегия реализации

### Фазы

1. **AST + Parser**: `Spawn { body: Box<Expr> }`, удаление `SpawnFor`
2. **Система типов**: 6 новых вариантов `MonoType`, вывод типов выражений возвращает тип вычислительной структуры
3. **Унификация DAG-анализа**: слияние входных точек, объединение `TaskSource` в Explicit + Iterate
4. **Адаптация IR / Codegen**: удаление `Ir::SpawnFor`, единый путь обработки
5. **Упрощение Placement**: удаление ветви `SpawnFor`
6. **Тестирование**: все существующие тесты `spawn for` проходят

### Область воздействия

| Файл / каталог | Изменения |
|----------------|-----------|
| `frontend/core/parser/ast.rs` | `body` у `Spawn` становится `Box<Expr>`, удаляется `SpawnFor` |
| `frontend/core/parser/pratt/nud.rs` | Обработчик `spawn` упрощается до общего разбора выражений |
| `frontend/core/types/mono.rs` | Добавляются варианты `Block` / `ForExpr` / `WhileExpr` / `IfExpr` / `Call` / `Spawn` |
| `frontend/core/spawn/analysis.rs` | Единая точка входа, слияние `TaskSource` (Explicit + Iterate) |
| `frontend/core/spawn/placement.rs` | Удаление ветви `SpawnFor` |
| `frontend/core/typecheck/` | Адаптация всех узлов выражений к выводу типов вычислительных структур |
| `middle/core/ir.rs` | Удаление `Ir::SpawnFor` |
| `middle/` (IR gen, codegen) | Единый путь spawn, стирание типа Spawn |
| `tests/yaoxiang/04-concurrency/spawn_for.yx` | Семантика неизменна, проверка прохождения |

### Зависимости

- RFC-024 (модель параллелизма на основе блоков spawn) — данный RFC является её ортогональным расширением
- RFC-010 (унифицированный синтаксис типов) — основа для изменений системы типов
- RFC-027 (конвейер доказательств compile-time) — варианты `MonoType` задают форму утверждений для конвейера
- RFC-019 (гомоиконичность на уровне типов) — варианты `MonoType` — его встроенное подмножество

## Журнал проектных решений

| Решение | Выбор | Причина | Дата |
|---------|-------|---------|------|
| Область модификации spawn | Произвольное выражение | Устранение особого случая `spawn for` | 2026-06-16 |
| Поддержка `spawn while` | Поддерживается | Синтаксическая ортогональность, низкая стоимость реализации | 2026-06-16 |
| Семантика `spawn if` | Модифицирует весь if-else | Разграничение с `if spawn { }` | 2026-06-16 |
| Система типов | Ввести типы вычислительных структур | «Всё есть тип», поддержка compile-time рефлексии | 2026-06-16 |
| Стирание типа spawn | Стирается после проверки типов | Среде выполнения не нужна информация о параллельной структуре | 2026-06-16 |
| Приоритет связывания spawn | Наименьший (как у return) | Поглощает всё последующее выражение | 2026-06-16 |
| Поведение DAG для внутренности for | Не раскрывать подвыражения внутри for | Правило прямых подвыражений не меняется, for целиком — один источник задач | 2026-06-16 |
| Интеграция с конвейером доказательств | Варианты `MonoType` отображаются на утверждения конвейера RFC-027 | Конвейеру нужно знать форму проверяемого утверждения, `MonoType` предоставляет структурированный ввод | 2026-07-03 |
| Связь с RFC-019 | Встроенное подмножество компилятора | Пользователь не может определять свои, но разделяется идея «синтаксис есть тип» | 2026-07-03 |
| Границы доказательства | 6 сценариев: чистый параллелизм / только чтение разделяемого / конфликт записи / зависимость while / spawn if / spawn block | Уточнение обязанностей по доказательству и условий отказа для каждого варианта `MonoType` | 2026-07-03 |
