# RFC-032: унифицированный модификатор выражений spawn

> **Настоящий документ определяет синтаксис `spawn`, рефакторизацию AST/IR, расширения системы типов**.
> Семантика поведения во время выполнения (гранулярность разбиения задач, владение, области видимости, распространение ошибок, типы ресурсов, вложенность) приведена в [RFC-024: семантика параллельной среды выполнения на основе spawn](./024-concurrency-model.md).
>
> Два RFC совместно определяют `spawn` — 024 отвечает на вопрос «что делать», 032 — «как представить».

> **Ключевая идея**: `spawn` не должен модифицировать только блок `{}`. Он может модифицировать **произвольное выражение**. `spawn for` — не особый синтаксис, а естественная комбинация `spawn` + выражения `for`.

## Резюме

Расширение `spawn` с `spawn { }` (модифицирует только блок) до `spawn <expr>` (модифицирует произвольное выражение). Вариант `Expr::SpawnFor` удаляется из AST и естественно заменяется на `Expr::Spawn { body: Expr::For { .. } }`. Типы структур выражений (Block, For, While, If и т. д.) входят в систему типов как новые варианты `MonoType`, `Spawn<T>` оборачивает параллельно выполняемую вычислительную структуру, помечается на этапе компиляции и стирается после проверки.


## Мотивация

### Зачем нужно это изменение?

В настоящее время `spawn for x in items { body }` является самостоятельной комбинацией ключевых слов, и в AST имеется специальный вариант `Expr::SpawnFor` для его представления. Это нарушает ортогональность языка:

1. **Неунифицированный синтаксис**: `spawn` может модифицировать только блок `{}`, а `spawn for` — жёстко закодированное исключение.
2. **Отсутствие ортогональности**: комбинации вроде `spawn while`, `spawn if` и т. д. невозможно выразить естественно.
3. **Неполная система типов**: `spawn` невидим в системе типов, невозможно получить параллельную структуру через рефлексию типов.

### Текущие проблемы

```rust
// Два варианта spawn в AST
Spawn { body: Box<Block>, span: Span },         // spawn { ... }
SpawnFor { var, var_mut, iterable, body, span },  // spawn for x in items { ... }

// MonoType содержит только типы значений, без типов вычислительных структур
// тип spawn { a, b } = Tuple(T_a, T_b)  ← теряется информация «это spawn»
// тип spawn for      = List(T)          ← теряется информация «это параллелизм по данным»
```

## Предложение

### Основной замысел

`spawn <expr>`: `spawn` модифицирует произвольное выражение. Форма выражения определяет, как DAG разбивает задачи.

**Всё есть тип**: `MonoType` расширяется с «типов значений» до «типов значений + типов вычислительных структур». Каждой ключевой структуре выражения соответствует вариант типа в системе типов. `Spawn<T>` оборачивает параллельно выполняемую вычислительную структуру.

### Ментальная модель пользователя

`spawn` = «выполнить это выражение параллельно». Форма выражения определяет способ разбиения:

| Форма выражения | Поведение при параллелизме | Тип |
|----------------|---------------------------|-----|
| `spawn { a, b, c }` | `a`, `b`, `c` выполняются независимо и параллельно | `Spawn(Block(Tuple(T_a, T_b, T_c)))` |
| `spawn for x in items { f(x) }` | N итераций выполняются независимо и параллельно | `Spawn(ForExpr { body_ty: List(T) })` |
| `spawn while cond { step() }` | Каждая итерация — отдельная задача | `Spawn(WhileExpr { body_ty: List(T) })` |
| `spawn if c { a } else { b }` | Выбранная ветвь в целом — домен spawn | `Spawn(IfExpr { then_ty: T_a, else_ty: Some(T_b) })` |
| `spawn call(x)` | Сам вызов — одна задача | `Spawn(Call { fn_ty: Fn(A→R), result_ty: R })` |
| `spawn 42` | Одна задача | `Spawn(Int)` |

Компилятор отвечает за анализ DAG для определения зависимостей, среда выполнения планирует по модели GMP — задачи без зависимостей помещаются в очередь работ, worker'ы их подхватывают. Синхронизация блокирующая — ожидается завершение всех задач.

**Отличие от Go**: `go` в Go — «запустил и забыл», `spawn` в YaoXiang — «разбить на параллельное выполнение, дождаться завершения всех частей и продолжить».

### Ортогональность потока управления

| Комбинация | Семантика | Различие |
|------------|-----------|----------|
| `spawn for x in items { body }` | Параллелизм по данным: каждая итерация — отдельная задача | Анализ зависимостей между итерациями в DAG |
| `for x in items spawn { body }` | Каждая итерация создаёт домен spawn | Без анализа между итерациями |
| `spawn while cond { body }` | Условный параллелизм: каждая итерация — отдельная задача | Межитерационные зависимости обеспечиваются условием |
| `while cond spawn { body }` | Каждая итерация создаёт домен spawn | Семантика отличается от предыдущего, но не требует особой обработки |
| `spawn if c { a } else { b }` | Весь if-else — один домен spawn | При выполнении выбирается ветвь по условию |
| `if c spawn { a } else { b }` | spawn только в одной ветви | Внутри выражения if оборачивается spawn |

### Устраняемая сложность

- ❌ `Expr::SpawnFor` удаляется из AST
- ❌ `SpawnForAnalysis` удаляется из анализа DAG
- ❌ `spawn for` больше не обрабатывается особо в парсере как составное ключевое слово
- ❌ `Ir::SpawnFor` удаляется из IR

## Детальное проектирование

### 1. Уровень AST

**До:**

```rust
Spawn { body: Box<Block>, span: Span },         // spawn { ... }
SpawnFor { var, var_mut, iterable, body, span },  // spawn for x in items { ... }
```

**После:**

```rust
Spawn { body: Box<Expr>, span: Span },           // spawn <произвольное выражение>
```

`Expr::SpawnFor` удаляется. AST-представление `spawn for x in items { body }`:

```rust
Expr::Spawn {
    body: Box::new(Expr::For {
        var: "x",
        iterable: items,
        body: body_block,
        ..
    })
}
```

**Особый случай IF**:

| Запись | Структура AST |
|--------|---------------|
| `spawn if cond { a } else { b }` | `Spawn { body: Expr::If { ... } }` |
| `if cond spawn { a } else { b }` | `Expr::If { then: Spawn { body: {a} }, else: {b} }` |

Семантика различается, но обе являются естественными комбинациями, не требующими особых правил.

### 2. Уровень парсера

`spawn` имеет наименьший приоритет связывания (аналогично `return`), поглощая всё последующее выражение:

```
spawn a + b        →  spawn (a + b)         ≠  (spawn a) + b
spawn f(x).y       →  spawn (f(x).y)
```

Изменения в парсере: в `pratt/nud.rs` `spawn` больше не требует `{`, а вызывает общий парсер выражений:

```
token spawn → parse_expr(min_precedence) → Expr::Spawn { body: expr }
```

`spawn for` больше не обрабатывается как составное ключевое слово — `for` обрабатывается общим парсером выражений, порождая `Expr::For`, а `spawn` лишь выполняет оборачивание.

### 3. Система типов

**Новые варианты `MonoType`:**

```rust
// ========== Типы вычислительных структур ==========

/// Выражение-блок {}
Block(Box<MonoType>),

/// Цикл for
ForExpr { body_ty: Box<MonoType> },

/// Цикл while
WhileExpr { body_ty: Box<MonoType> },

/// Ветвь if-else
IfExpr {
    then_ty: Box<MonoType>,
    else_ty: Option<Box<MonoType>>,
},

/// Вызов функции
Call {
    fn_ty: Box<MonoType>,
    result_ty: Box<MonoType>,
},

/// Параллельная обёртка spawn: внутреннее выражение выполняется параллельно
/// Помечается на этапе компиляции, стирается после проверки типов
Spawn(Box<MonoType>),
```

**Правила вывода типов**: вывод типа каждого выражения возвращает «тип вычислительной структуры». Без обёртки `Spawn` — последовательное выполнение, с обёрткой `Spawn` — параллельное выполнение. После завершения проверки типов `Spawn` стирается, тип понижается до внутреннего типа значения.

**Поток проверки типов**:
1. Вывод типа T выражения body (тип вычислительной структуры)
2. Если обёрнут в spawn, обернуть в `Spawn(T)`
3. При деструктурирующем присваивании: `results: List(Data) = spawn for ... {}` — извлечь `List(Data)` из `Spawn(ForExpr { body_ty: List(Data) })`

`Spawn<T>` стирается после проверки типов, среде выполнения не нужно знать, получены ли данные параллельно или последовательно. Однако рефлексия на этапе компиляции (`type_of(x)`) позволяет получить полную топологию параллелизма.

### 4. Уровень анализа DAG

Две текущие точки входа объединяются в одну:

```rust
/// Единая точка входа: диспетчеризация по типу выражения body
fn analyze_spawn_expr(body: &Expr, ...) -> SpawnAnalysis {
    match body {
        Expr::Block(block)       => analyze_block_tasks(block, ...),
        Expr::For { .. }         => analyze_iter_tasks(IterKind::For, body, ...),
        Expr::While { .. }       => analyze_iter_tasks(IterKind::While, body, ...),
        Expr::If { .. }          => analyze_if_task(body, ...),
        _                        => single_task(body, ...),
    }
}
```

**Единая структура результата**:

```rust
struct SpawnAnalysis {
    source: TaskSource,
    plan: ExecutionPlan,
}

enum TaskSource {
    /// spawn { a, b, c } — N прямых подвыражений, известных на этапе компиляции
    Explicit(Vec<TaskInfo>),
    /// spawn for/while — N задач порождается итерациями в среде выполнения
    Iterate {
        kind: IterKind,
        iter_var: String,
        iterable: Option<Expr>,      // есть у for, нет у while
        condition: Option<Expr>,     // есть у while, нет у for
        body: Block,
        reads: HashSet<String>,
        writes: HashSet<String>,
        resource_vars: HashSet<String>,
    },
}

enum IterKind { For, While }
```

Структура `SpawnForAnalysis` удаляется.

| Тип body | Как разбивается на задачи |
|----------|--------------------------|
| `Expr::Block` | Прямые подвыражения → список задач |
| `Expr::For` | Каждая итерация → одна задача (параллелизм по данным) |
| `Expr::While` | Каждая итерация → одна задача |
| `Expr::If` | Выбранная ветвь целиком → одна задача |
| `Expr::Call` / прочее | Само выражение → одна задача |

После анализа DAG среда выполнения планирует по модели GMP — задачи без зависимостей помещаются в очередь работ, worker'ы их подхватывают.

### 5. Уровень IR / Codegen

`Ir::SpawnFor` удаляется. Всё унифицируется до `Ir::Spawn`, несущего информацию `TaskSource`.

Трансляция HIR → IR генерирует вызовы среды выполнения на основе `SpawnAnalysis.source`:
- `TaskSource::Explicit(tasks)` → список задач, известных на этапе компиляции
- `TaskSource::Iterate { .. }` → раскрытие во время выполнения (управляется компилятором, аналогично par_iter, но с нулевыми накладными расходами)

### 6. Уровень Placement

Две текущие ветви объединяются в одну:

```rust
// До
Expr::Spawn { body, .. } => self.check_block(body),
Expr::SpawnFor { body, iterable, .. } => {
    self.check_expr(iterable);
    self.check_block(body);
}

// После
Expr::Spawn { body, .. } => self.check_expr(body),   // body — это Expr, достаточно рекурсии
```

### 7. Обратная совместимость

Семантика существующего кода `spawn for` остаётся неизменной, парсер автоматически разбирает `spawn for x in items { body }` как `Expr::Spawn { body: Expr::For }`. Внутреннее представление меняется, видимое пользователю поведение не меняется.

Новый синтаксис получается естественно:
```yx
spawn while has_next() {
    item = next()
    process(item)
}

spawn if use_cache {
    load_from_cache(key)
} else {
    fetch(key)
}
```

## Компромиссы

### Преимущества

1. **Ортогональность синтаксиса**: `spawn` + любой поток управления = естественная параллельная комбинация.
2. **Всё есть тип**: система типов полностью фиксирует вычислительную структуру, рефлексия на этапе компиляции позволяет получить топологию параллелизма.
3. **Устранение особых случаев**: удаляется `Expr::SpawnFor` и связанный с ним особый код.
4. **Расширяемость**: будущие новые структуры потока управления автоматически комбинируются с `spawn` без изменения логики spawn.

### Недостатки

1. **Разрастание системы типов**: 6 новых вариантов `MonoType`, повышение сложности проверки типов.
2. **Критическое изменение**: меняется внутреннее представление AST/IR, требуется обновить весь код, использующий `Expr::SpawnFor`.
3. **Вывод типов выражений**: каждое выражение теперь должно возвращать тип вычислительной структуры, что затрагивает большую область кода.

## Альтернативы

| Альтернатива | Почему не выбрана |
|--------------|-------------------|
| Сохранить `spawn for` как самостоятельный синтаксис | Нарушает ортогональность, становится единственным исключением комбинированного ключевого слова в языке |
| `spawn` модифицирует только `{}`, параллелизм по данным — через стандартную библиотеку `par_iter` | Базовая возможность языка уходит в библиотеку, теряется анализ DAG и обнаружение конфликтов ресурсов на уровне компилятора |
| Удалить `SpawnFor`, но не вводить типы вычислительных структур в систему типов | Система типов теряет возможности рефлексии, `spawn` становится невидимым на уровне типов |


## Связь с RFC-019

6 вариантов `MonoType`, вводимых настоящим RFC (Block/ForExpr/WhileExpr/IfExpr/Call/Spawn), составляют **встроенное подмножество компилятора** для [RFC-019: типоуровневая гомоиконичность](./019-typed-homoiconicity.md). Ключевая идея RFC-019 «синтаксические структуры входят в систему типов» реализуется здесь так: 6 вычислительных структур, которые компилятор понимает изначально, получают соответствующие типовые представления. Пользователь не может определять новые типы вычислительных структур через `SyntaxRule`, но эти 6 встроенных вариантов покрывают все ключевые потоки управления.

## Интеграция с конвейером доказательств

Причина существования 6 вариантов `MonoType`: они указывают [конвейеру доказательств на этапе компиляции RFC-027](../accepted/027-compile-time-evaluation-types.md), **какую форму имеет проверяемое утверждение**. Сам конвейер выполняет реальную работу по доказательству (анализ свободных переменных, классификация эффектов, анализ алиасов, обнаружение конфликтов), `MonoType` делает только одно — предоставляет структурированный интерфейс ввода.

### Отображение вариантов → утверждений

| Тип | Форма утверждения | Стратегия доказательства |
|-----|-------------------|--------------------------|
| `Spawn(ForExpr { body_ty })` | Параллелизм по данным: N задач итераций без межитерационных конфликтов | Извлечь свободные переменные body → классифицировать эффекты → проверить отсутствие Write(Shared) / `&mut`(Shared) |
| `Spawn(WhileExpr { body_ty })` | Условный параллелизм: каждая итерация независима + нет межитерационных причинно-следственных зависимостей | То же + проверка, что условие итерации не имеет межитерационных побочных эффектов |
| `Spawn(Block(T))` | Явная группа задач: зависимости между задачами задаются DAG | Проверить граф зависимостей из анализа DAG — входные данные каждой задачи готовы к моменту её запуска |
| `Spawn(IfExpr { then_ty, else_ty })` | spawn ветви: выбранная ветвь — один домен spawn целиком | Выбор ветви без конфликтов, рекурсивная проверка внутри body |
| `Spawn(Call { fn_ty, result_ty })` | spawn вызова: вызываемая функция — отдельная задача | Проверить чистоту или изолированность функции |
| `Spawn(T)` (значение, например `spawn 42`) | spawn одиночного значения: параллелизма нет | Тривиально проходит |

### Сценарии доказательства

**Сценарий 1 — чистый параллелизм по данным (проходит):**

```yaoxiang
items = [1, 2, 3, 4, 5]
results = spawn for item in items { item * 2 }
// Тип: Spawn(ForExpr { body_ty: List(Int) })
```

1. Свободные переменные: `item` (локальная в цикле, независимая копия в каждой итерации), `items` (внешняя, в body только чтение)
2. Классификация эффектов: все Read(Local) или Read(Shared), записей нет
3. Proved ✓

**Сценарий 2 — только чтение общего (проходит):**

```yaoxiang
config = load_config()
results = spawn for item in items { process(item, config) }
// Тип: Spawn(ForExpr { body_ty: List(Result) })
```

1. Свободные переменные: `item` (Read(Local)), `config` (внешняя, в body нет путей записи → Read(Shared))
2. Классификация эффектов: все только чтение
3. Proved ✓

**Сценарий 3 — конфликт записи (отклоняется):**

```yaoxiang
mut counter = 0
spawn for item in items { counter += 1 }
```

1. Свободные переменные: `item` (Read(Local)), `counter` (внешняя, `+=` десахарируется в запись)
2. Классификация эффектов: `counter` — Write(Shared), запись в одну и ту же память из разных итераций
3. Инстанциация конфликта: `Write(task_0, counter) ∧ Write(task_1, counter) = True`
4. Disproved ✗ → ошибка компиляции: `Ошибка: в теле spawn for существует межитерационный конфликт записи. Переменная counter записывается несколькими параллельными задачами.`

**Сценарий 4 — while + итератор с состоянием (предупреждение/отклонение):**

```yaoxiang
spawn while iter.has_next() {
    item = iter.next()
    process(item)
}
// Тип: Spawn(WhileExpr { body_ty: List(Processed) })
```

1. Свободные переменные: `iter` (внешний, `next()` → `&mut self` → `&mut`(Shared))
2. `next()` изменяет состояние итератора, итерация N+1 зависит от побочного эффекта итерации N
3. Это не независимые задачи → нарушение ограничения независимости `Spawn(WhileExpr)`
4. Компилятор сообщает о межитерационной причинно-следственной зависимости, предлагает перейти на `spawn for`

**Сценарий 5 — spawn if (проходит):**

```yaoxiang
result = spawn if use_cache { load(key) } else { fetch(key) }
// Тип: Spawn(IfExpr { then_ty: T, else_ty: Option(T) })
```

1. Выполняется только одна ветвь, межзадачных конфликтов нет
2. Если внутри body есть вложенный spawn — рекурсивная проверка
3. Proved ✓

**Сценарий 6 — зависимости между задачами в spawn-блоке (DAG + проверка конвейером):**

```yaoxiang
spawn {
    a = fetch_user(id)
    b = fetch_orders(a.user_id)  // зависит от a
    c = compute_stats()          // независима
}
// Тип: Spawn(Block(Tuple(User, Orders, Stats)))
```

1. Анализ DAG: `a` и `c` независимы (могут выполняться параллельно), `b` зависит от `a` (планируется после a)
2. Проверка конвейером: входные данные `b` (`a.user_id`) вычислены к моменту запуска `b`
3. Proved ✓

### Что `MonoType` НЕ делает

| Что делает | Чего НЕ делает |
|------------|----------------|
| Идентифицирует форму утверждения | Не выполняет доказательство |
| Фиксирует вычислительную структуру на уровне типов | Не заменяет анализ DAG |
| Предоставляет конвейеру RFC-027 ввод типов | Не заменяет анализ свободных переменных, анализ алиасов, обнаружение конфликтов |

Реальная работа по доказательству выполняется стандартными проходами анализа компилятора. Ценность `MonoType` в том, что эти проходы планируются в единой типовой среде — конвейеру доказательств не нужны особые ветви для каждого узла AST.

## Стратегия реализации

### Этапы

1. **AST + парсер**: `Spawn { body: Box<Expr> }`, удалить `SpawnFor`.
2. **Система типов**: добавить 6 вариантов `MonoType`, вывод типов всех выражений возвращает тип вычислительной структуры.
3. **Унификация анализа DAG**: объединить точки входа, объединить `TaskSource` в один enum.
4. **Адаптация IR / Codegen**: удалить `Ir::SpawnFor`, унифицировать путь обработки.
5. **Упрощение Placement**: удалить ветвь `SpawnFor`.
6. **Тестирование**: все существующие тесты `spawn for` проходят.

### Область воздействия

| Файл/каталог | Изменение |
|--------------|-----------|
| `frontend/core/parser/ast.rs` | `body` у `Spawn` меняется на `Box<Expr>`, удалить `SpawnFor` |
| `frontend/core/parser/pratt/nud.rs` | Обработчик `spawn` упрощается до вызова общего парсера выражений |
| `frontend/core/types/mono.rs` | Добавить варианты `Block`/`ForExpr`/`WhileExpr`/`IfExpr`/`Call`/`Spawn` |
| `frontend/core/spawn/analysis.rs` | Единая точка входа, объединение `TaskSource` (Explicit + Iterate) |
| `frontend/core/spawn/placement.rs` | Удалить ветвь `SpawnFor` |
| `frontend/core/typecheck/` | Все узлы выражений адаптируются к выводу типа вычислительной структуры |
| `middle/core/ir.rs` | Удалить `Ir::SpawnFor` |
| `middle/` (IR gen, codegen) | Унифицированный путь spawn, стирание типа Spawn |
| `tests/yaoxiang/04-concurrency/spawn_for.yx` | Семантика неизменна, проверка проходит |

### Зависимости

- RFC-024 (модель параллелизма на основе spawn-блока) — настоящий RFC является её ортогональным расширением
- RFC-010 (унифицированный синтаксис типов) — основа изменений системы типов
- RFC-027 (конвейер доказательств на этапе компиляции) — варианты `MonoType` предоставляют конвейеру ввод форм утверждений
- RFC-019 (типоуровневая гомоиконичность) — варианты `MonoType` — его встроенное подмножество компилятора

| Решение | Выбор | Причина | Дата |
|---------|-------|---------|------|
| Область модификации spawn | Произвольное выражение | Устранение особого случая `spawn for` | 2026-06-16 |
| Поддержка `spawn while` | Поддерживается | Ортогональность синтаксиса, низкая стоимость реализации | 2026-06-16 |
| Семантика `spawn if` | Модифицирует весь if-else | Различение с `if spawn { }` | 2026-06-16 |
| Система типов | Ввод типов вычислительных структур | «Всё есть тип», поддержка рефлексии на этапе компиляции | 2026-06-16 |
| Стирание типа spawn | Стирается после проверки типов | Среде выполнения не нужна информация о структуре параллелизма | 2026-06-16 |
| Приоритет связывания spawn | Наименьший (как return) | Поглощает всё последующее выражение | 2026-06-16 |
| DAG для внутренних выражений for | Не раскрывать внутренние подвыражения for | Правило прямых подвыражений не меняется, for целиком — один источник задач | 2026-06-16 |
| Интеграция с конвейером доказательств | Варианты `MonoType` отображаются на утверждения конвейера RFC-027 | Конвейеру нужна форма проверяемого утверждения, `MonoType` предоставляет структурированный ввод | 2026-07-03 |

| Границы доказательства | 6 сценариев: чистый параллелизм / только чтение общего / конфликт записи / зависимость while / spawn if / spawn block | Уточнить обязательства и условия отказа для каждого варианта `MonoType` | 2026-07-03 |
