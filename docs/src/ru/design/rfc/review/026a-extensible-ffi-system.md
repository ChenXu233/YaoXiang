# RFC-026a: Расширяемая система механизмов FFI

> **Родительский RFC**: [RFC-026: Основной механизм FFI](../accepted/026-ffi-core-mechanism.md)
>
> Данный RFC определяет раздел расширяемости RFC-026 — как подключать механизмы FFI помимо C ABI (Wasm, Python, пользовательский ABI) в виде плагинов, а также режимы динамической загрузки.

## Резюме

RFC-026 определяет основной механизм FFI; `Native.c("lib")` использует встроенный C ABI. Настоящий RFC абстрагирует механизмы ABI в подключаемый `FfiMechanism`, чтобы ядро не жёстко кодировало какой-либо конкретный ABI:

1. **Абстракция `FfiMechanism`**: определяет четыре операции, которые должен реализовать механизм (загрузка библиотеки, разрешение символов, маршалинг, вызов)
2. **Тег механизма как выбор механизма**: `Native.c` / `Native.wasm` / `Native.python` соответственно выбирают зарегистрированный механизм
3. **Реестр механизмов compile-time**: теги механизмов проверяются на этапе компиляции; незарегистрированный тег вызывает ошибку компиляции
4. **Статическая vs динамическая загрузка**: оба режима сохраняют границы безопасности RFC-026

## Мотивация

RFC-026 встраивает только C ABI (`Native.c`). Однако в будущем YaoXiang может потребоваться:
- Вызов Wasm-модулей (`Native.wasm`)
- Встраивание расширений Python (`Native.python`)
- Пользовательский ABI (проприетарное оборудование, мосты RPC)

Вместо жёсткого кодирования этих ABI в компиляторе лучше абстрагировать «как загружать библиотеку, как разрешать символы, как маршалировать, как вызывать» в trait, и каждая реализация становится плагином. Ядро знает только `FfiMechanism` и не знает ни одного конкретного ABI.

### Проектные ограничения

1. **Проверка тегов механизмов compile-time**: `xxx` в `Native.xxx(...)` должен быть зарегистрированным механизмом, иначе ошибка компиляции
2. **Без жёсткого кодирования механизмов**: компилятор не имеет встроенного списка механизмов (кроме `.c` как эталонной реализации); механизмы регистрируются плагинами
3. **Сохранение границ безопасности RFC-026**: любой механизм должен соблюдать деление типов на две группы, изоляцию временной области маршалинга, Move + RAII
4. **Совместимость с самобутстрапом**: реестр механизмов деградирует до `Dict`/`Set` YaoXiang

## Предложение

### 1. Абстракция `FfiMechanism`

Каждый механизм FFI реализует четыре операции. Это ключ к тому, чтобы ядро не жёстко кодировало ABI, — компилятор вызывает только этот интерфейс и не знает, что за ним стоит C, Wasm или что-то ещё:

```rust
trait FfiMechanism {
    /// Тег механизма, например "c" / "wasm" / "python"
    fn tag(&self) -> &str;

    /// Загрузка библиотеки. C: dlopen/статическая линковка; Wasm: инстанциация модуля; Python: import.
    /// Возвращает внутренний дескриптор библиотеки механизма.
    fn load_library(&self, id: &str) -> Result<LibraryHandle>;

    /// Разрешение символа. Может вызываться compile-time для проверки наличия символа.
    /// C: dlsym/поиск в таблице символов; Wasm: поиск в таблице экспорта.
    fn resolve(&self, lib: &LibraryHandle, symbol: &str) -> Result<SymbolHandle>;

    /// Вызов. Маршалит аргументы по сигнатуре YaoXiang, выполняет, маршалит возвращаемое значение.
    /// Должен соблюдать правила маршалинга из RFC-026 §3 (изоляция временной области).
    fn invoke(
        &self,
        sym: &SymbolHandle,
        args: &[RuntimeValue],
        sig: &Signature,
    ) -> Result<RuntimeValue>;
}
```

**Ключевой момент**: реализация `invoke` должна соблюдать RFC-026 §3 — входные параметры копируются во временную область, возврат через memcpy, ограничение заимствования на один вызов. Механизм может выбирать свои детали ABI, но **не может нарушать границы безопасности**. Это обязательство плагина.

### 2. Тег механизма как выбор механизма

```yaoxiang
// .c → механизм C ABI (встроенная эталонная реализация RFC-026)
sqlite3 = Native.c("libsqlite3")
SqliteDb.open: (f: String) -> ?SqliteDb = sqlite3("sqlite3_open")

// .wasm → механизм Wasm (регистрируется плагином yx_wasm_ffi)
wasm_mod = Native.wasm("mymodule.wasm")
process: (input: String) -> String = wasm_mod("process")

// .python → механизм Python (регистрируется плагином yx_python_ffi)
np = Native.python("numpy")
```

`.c` / `.wasm` в `Native.c` / `Native.wasm` — это **теги механизма**, выбирающие, какой зарегистрированный `FfiMechanism` использовать. Ядро встраивает `.c` как эталонную реализацию; остальные предоставляются плагинами.

### 3. Регистрация механизмов и проверка compile-time

Плагин через `.so` объявляет в реестре механизмов compile-time, какие теги он предоставляет:

```text
use yx_wasm_ffi
  → загружает libyx_wasm_ffi.so
  → вызывает yx_register_mechanism()
  → регистрирует FfiMechanism { tag: "wasm", ... }
  → в реестре механизмов появляется "wasm"

// далее:
Native.wasm("mod.wasm")    // ✅ компилируется, "wasm" зарегистрирован
Native.foo("x")            // ❌ ошибка компиляции: Unknown FFI mechanism 'foo'
                           //    Try: `use yx_foo_ffi`
```

Реестр механизмов compile-time **хранит только теги механизмов** (строки) + указатель на соответствующий экземпляр `FfiMechanism`. При компиляции `Native.xxx(...)` идёт поиск по таблице; отсутствие тега приводит к ошибке компиляции.

### 4. Статическая vs динамическая загрузка

Реализация `load_library` определяет момент загрузки; оба режима сохраняют границы безопасности RFC-026:

| Режим | Поведение `load_library` | Проверка символов | Доверие |
|-------|--------------------------|-------------------|---------|
| **Статический** (по умолчанию, C ABI) | `-llib` на этапе компиляции, библиотека попадает в таблицу символов | Чтение таблицы символов compile-time | Полное |
| **Динамический** | dlopen/инстанциация при первом вызове в runtime | Проверка при первой загрузке, отсутствие — fail-fast | Декларативное, проверка при загрузке |

```yaoxiang
// Статический: C-библиотека линкуется compile-time
sqlite3 = Native.c("libsqlite3")           // -lsqlite3 compile-time

// Динамический: плагин, обнаруженный в runtime
plugin = Native.c.dynamic("./plugins/foo.so")   // dlopen в runtime
```

Независимо от статического или динамического режима, маршалинг подчиняется правилу изоляции временной области из RFC-026 §3. В динамическом режиме отсутствие символа — **чистая ошибка runtime** (fail-fast), а не крах.

### 5. Полный поток информации

```
use yx_wasm_ffi                     ← регистрирует механизм "wasm"
       │
       ▼
wasm_mod = Native.wasm("mod.wasm")
  compile-time: поиск в реестре "wasm" существует ✅
         → вызов load_library("mod.wasm") механизма wasm
         → инстанциация Wasm-модуля, возврат дескриптора библиотеки
       │
       ▼
process: (input: String) -> String = wasm_mod("process")
  compile-time: вызов resolve(lib, "process") механизма wasm для проверки наличия экспорта ✅
         → генерация CallNative { mechanism: "wasm", lib, symbol: "process", sig }
       │
       ▼  runtime
  Выполнение CallNative
  → invoke(sym, args, sig) механизма
  → маршалинг по sig (изоляция временной области) → выполнение Wasm → маршалинг возврата
```

### 6. Деградация после самобутстрапа

Trait `FfiMechanism` и реестр механизмов периода хостинга Rust после самобутстрапа деградируют до обычных структур YaoXiang:

```yaoxiang
// После самобутстрапа реестр механизмов — это Dict
let mechanisms: Dict(String, FfiMechanism) = {}
mechanisms["c"] = c_mechanism
mechanisms["wasm"] = wasm_mechanism

// FfiMechanism — это интерфейс в YaoXiang (динамическая диспетчеризация RFC-011a)
// Native.c("lib") → mechanisms["c"].load_library("lib")
```

В период Rust используется trait object (`Box<dyn FfiMechanism>`), после самобутстрапа — интерфейс YaoXiang (RFC-011a). Интерфейс единообразен: загрузка, разрешение, маршалинг, вызов.

## Компромиссы

### Преимущества

1. **Нулевое жёсткое кодирование ABI**: ядро знает только `FfiMechanism`; новый ABI = новый плагин
2. **Единые границы безопасности**: все механизмы обязаны соблюдать правила маршалинга RFC-026 §3
3. **Проверка механизмов compile-time**: отсутствующий тег механизма вызывает ошибку компиляции, а не обнаруживается в runtime
4. **Единая абстракция статики и динамики**: детали реализации `load_library` скрыты внутри механизма

### Недостатки

1. **Порог входа в написание плагинов**: реализация `FfiMechanism` требует понимания целевого ABI + контракта маршалинга
2. **Обязательства механизма держатся на соглашении**: изоляция временной области маршалинга держится на добросовестности плагина; ядро не может принудительно верифицировать реализацию плагина

## Связь с другими RFC

- **RFC-026** (родительский): основной механизм FFI — `FfiMechanism` должен соблюдать его правила маршалинга и границы безопасности
- **RFC-011a**: интерфейсы и динамическая диспетчеризация — после самобутстрапа `FfiMechanism` деградирует до интерфейса YaoXiang
- **RFC-014**: система управления пакетами — обнаружение и загрузка плагинов `.so` зависят от пакетного менеджера
- **RFC-021** (устаревший): FFI-расширения на основе библиотек — настоящий RFC опускает его API `ffi.load_library` на уровень плагинов механизма
