# RFC-026a: Расширяемая система механизмов FFI

> **Родительский RFC**: [RFC-026: Основной механизм FFI](../accepted/026-ffi-core-mechanism.md)
>
> Настоящий RFC определяет расширяемую часть RFC-026 — как подключать механизмы FFI помимо C ABI (Wasm, Python, пользовательский ABI) в виде плагинов, а также модель динамической загрузки.

## Аннотация

RFC-026 определяет основной механизм FFI, при этом `Native.c("lib")` идёт через встроенный C ABI. Настоящий RFC абстрагирует механизмы ABI в подключаемый `FfiMechanism`, так что ядро не жёстко кодирует какой-либо конкретный ABI:

1. **Абстракция `FfiMechanism`**: определяет четыре операции, которые должен реализовать механизм (загрузка библиотеки, разрешение символов, маршалинг, вызов)
2. **Тег механизма есть выбор механизма**: `Native.c` / `Native.wasm` / `Native.python` выбирают соответственно зарегистрированный механизм
3. **Реестр механизмов compile-time**: теги механизмов проверяются в compile-time, незарегистрированные теги порождают ошибку компиляции
4. **Статическая vs динамическая загрузка**: обе модели сохраняют границы безопасности RFC-026

## Мотивация

RFC-026 встраивает только C ABI (`Native.c`). Однако YaoXiang в будущем может нуждаться в:
- вызове Wasm-модулей (`Native.wasm`)
- встраивании расширений Python (`Native.python`)
- пользовательском ABI (проприетарное оборудование, RPC-мост)

Вместо того чтобы жёстко кодировать эти ABI в компиляторе, лучше абстрагировать «как загружать библиотеку, как разрешать символы, как маршалировать, как вызывать» в trait, причём каждый механизм реализуется как плагин. Ядро знает только `FfiMechanism` и не знает ни одного конкретного ABI.

### Ограничения проектирования

1. **Проверка тегов механизма в compile-time**: `xxx` в `Native.xxx(...)` должен быть зарегистрированным механизмом, иначе ошибка компиляции
2. **Без жёсткого кодирования механизмов**: компилятор не содержит встроенного списка механизмов (кроме `.c` как референтной реализации), механизмы регистрируются плагинами
3. **Сохранение границ безопасности RFC-026**: любой механизм должен соблюдать разделение типов, изоляцию временной области маршалинга, Move + RAII
4. **Совместимость с самобутстрапом**: реестр механизмов вырождается в `Dict`/`Set` YaoXiang

## Предложение

### 1. Абстракция `FfiMechanism`

Каждый механизм FFI реализует четыре операции. Это ключ к тому, что ядро не жёстко кодирует ABI — компилятор вызывает только этот интерфейс и не знает, что за ним стоит C, Wasm или что-то ещё:

```rust
trait FfiMechanism {
    /// Тег механизма, например "c" / "wasm" / "python"
    fn tag(&self) -> &str;

    /// Загрузка библиотеки. C: dlopen/статическая линковка; Wasm: инстанциация модуля; Python: import.
    /// Возвращает дескриптор библиотеки внутри механизма.
    fn load_library(&self, id: &str) -> Result<LibraryHandle>;

    /// Разрешение символа. Может вызываться в compile-time для проверки наличия символа.
    /// C: dlsym/поиск в таблице символов; Wasm: поиск в таблице экспорта.
    fn resolve(&self, lib: &LibraryHandle, symbol: &str) -> Result<SymbolHandle>;

    /// Вызов. Маршалирует параметры по сигнатуре YaoXiang, выполняет, маршалирует возвращаемое значение.
    /// Должен соблюдать правила маршалинга из RFC-026 §3 (изоляция временной области).
    fn invoke(
        &self,
        sym: &SymbolHandle,
        args: &[RuntimeValue],
        sig: &Signature,
    ) -> Result<RuntimeValue>;
}
```

**Ключевое**: реализация `invoke` должна соблюдать RFC-026 §3 — копирование входных параметров во временную область, возврат через memcpy, ограничение заимствования единственным вызовом. Механизм может выбирать свои детали ABI, но **не может нарушать границы безопасности**. Это обязательство плагина.

### 2. Тег механизма есть выбор механизма

```yaoxiang
// .c → механизм C ABI (встроенная референтная реализация RFC-026)
sqlite3 = Native.c("libsqlite3")
SqliteDb.open: (f: String) -> ?SqliteDb = sqlite3("sqlite3_open")

// .wasm → механизм Wasm (регистрируется плагином yx_wasm_ffi)
wasm_mod = Native.wasm("mymodule.wasm")
process: (input: String) -> String = wasm_mod("process")

// .python → механизм Python (регистрируется плагином yx_python_ffi)
np = Native.python("numpy")
```

`.c` / `.wasm` в `Native.c` / `Native.wasm` — это **теги механизма**, выбирающие, какой зарегистрированный `FfiMechanism` использовать. Ядро встраивает `.c` как референтную реализацию; остальные поставляются плагинами.

### 3. Регистрация механизма и проверка в compile-time

Плагин через `.so` объявляет в compile-time реестру механизмов, какие теги механизмов он предоставляет:

```text
use yx_wasm_ffi
  → загрузить libyx_wasm_ffi.so
  → вызвать yx_register_mechanism()
  → зарегистрировать FfiMechanism { tag: "wasm", ... }
  → в реестре механизмов появляется "wasm"

// далее:
Native.wasm("mod.wasm")    // ✅ компилируется, "wasm" зарегистрирован
Native.foo("x")            // ❌ ошибка компиляции: Неизвестный механизм FFI 'foo'
                           //    Попробуйте: `use yx_foo_ffi`
```

Реестр механизмов в compile-time **хранит только теги механизмов** (строки) + указатели на соответствующие экземпляры `FfiMechanism`. При компиляции `Native.xxx(...)` идёт поиск в таблице; отсутствие тега вызывает ошибку компиляции.

### 4. Статическая vs динамическая загрузка

Реализация `load_library` определяет момент загрузки; обе модели сохраняют границы безопасности RFC-026:

| Модель | Поведение `load_library` | Проверка символов | Тип |
|------|-------------------|---------|------|
| **Статическая** (по умолчанию, C ABI) | `-llib` в compile-time, библиотека попадает в таблицу символов | Чтение таблицы символов в compile-time | Полностью реализован |
| **Динамическая** | dlopen/инстанциация при первом вызове в runtime | Проверка при первой загрузке, отсутствие — fail-fast | Декларативно доверенный, проверяется при загрузке |

```yaoxiang
// Статическая: C-библиотека линкуется в compile-time
sqlite3 = Native.c("libsqlite3")           // -lsqlite3 в compile-time

// Динамическая: плагин, обнаруживаемый в runtime
plugin = Native.c.dynamic("./plugins/foo.so")   // dlopen в runtime
```

Как при статической, так и при динамической загрузке маршалинг идёт через изоляцию временной области из RFC-026 §3. В динамическом режиме отсутствие символа — это **чистая ошибка runtime** (fail-fast), а не падение.

### 5. Полный поток информации

```
use yx_wasm_ffi                     ← регистрирует механизм "wasm"
       │
       ▼
wasm_mod = Native.wasm("mod.wasm")
  compile-time: поиск в реестре механизмов "wasm" существует ✅
         → вызвать load_library("mod.wasm") механизма wasm
         → инстанцировать Wasm-модуль, вернуть дескриптор библиотеки
       │
       ▼
process: (input: String) -> String = wasm_mod("process")
  compile-time: вызвать resolve(lib, "process") механизма wasm для проверки наличия экспорта ✅
         → сгенерировать CallNative { mechanism: "wasm", lib, symbol: "process", sig }
       │
       ▼  runtime
  Выполнение CallNative
  → invoke(sym, args, sig) механизма
  → маршалинг по sig (изоляция временной области) → выполнение Wasm → маршалинг возврата
```

### 6. Вырождение после самобутстрапа

Trait `FfiMechanism` + реестр механизмов, находящиеся под управлением Rust, после самобутстрапа вырождаются в обычные структуры YaoXiang:

```yaoxiang
// После самобутстрапа реестр механизмов — это Dict
let mechanisms: Dict(String, FfiMechanism) = {}
mechanisms["c"] = c_mechanism
mechanisms["wasm"] = wasm_mechanism

// FfiMechanism — это интерфейс в YaoXiang (динамическая диспетчеризация RFC-011a)
// Native.c("lib") → mechanisms["c"].load_library("lib")
```

В период Rust используется trait object (`Box<dyn FfiMechanism>`), после самобутстрапа — интерфейс YaoXiang (RFC-011a). Интерфейс един: загрузка, разрешение, маршалинг, вызов.

## Компромиссы

### Преимущества

1. **Нулевое жёсткое кодирование ABI**: ядро знает только `FfiMechanism`, новый ABI = новый плагин
2. **Единая граница безопасности**: все механизмы принудительно соблюдают правила маршалинга RFC-026 §3
3. **Проверка механизма в compile-time**: при отсутствии тега механизма — ошибка компиляции, а не обнаружение в runtime
4. **Единая абстракция статического и динамического**: детали реализации `load_library` скрыты внутри механизма

### Недостатки

1. **Порог входа для написания плагинов**: реализация `FfiMechanism` требует понимания целевого ABI + контракта маршалинга
2. **Обязательства механизма основаны на соглашении**: изоляция временной области маршалинга зависит от соблюдения плагином; ядро не может принудительно проверить реализацию плагина

## Связь с другими RFC

- **RFC-026** (родительский): основной механизм FFI — `FfiMechanism` должен соблюдать его правила маршалинга и границы безопасности
- **RFC-011a**: интерфейсы и динамическая диспетчеризация — после самобутстрапа `FfiMechanism` вырождается в интерфейс YaoXiang
- **RFC-014**: система управления пакетами — обнаружение и загрузка плагинов `.so` зависит от пакетного менеджера
- **RFC-021** (устарел): расширение FFI через драйверы библиотек — настоящий RFC опускает его API `ffi.load_library` на уровень плагинов механизма
