## Аннотация

В данном документе предлагается ввести в YaoXiang **компиляторные предикаты** как первоклассные сущности и унифицировать все проверки статической верификации на этапе компиляции в единый **конвейер вычисления Bool-значений**. Компиляторные предикаты — это не внешние аннотации спецификаций, это функции. Функция, возвращающая Bool, может использоваться в позиции типа; компилятор вызывает её во время компиляции и проверяет возвращаемое значение. Тип — это утверждение, вычисление на этапе компиляции — это доказательство. SMT-решатель встроен как внутренний компонент компиляторного вычислителя; программист лишь заботится о том, «может ли компилятор доказать выполнение условия», без необходимости изучать отдельный язык спецификаций.

**Основной тезис**: единственная задача проверки типов на этапе компиляции — это вычисление булевых значений. Равенство типов, конфликты токенов, редукция зависимых типов, вычисление компиляторных предикатов, импликации в логике Хоара — всё это компиляторное вычисление булевых значений, выполняемое по единому конвейеру.

## Мотивация

### Почему упраздняется RFC-022?

RFC-022 проектирует спецификации в форме комментариев `//!`:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0          ← это комментарий, независимый от типа
    //! ensures: ExistsMax(result, arr[0..n])   ← это комментарий, независимый от типа
}
```

Это фундаментальная ошибка соответствия Curry-Howard: **спецификации и типы разделены на два слоя**. Комментарий — не тип. Комментарий не участвует в проверке типов. Комментарий — это лишь ментальная модель для «внешнего инструмента».

Белая книга выражается ясно:

> «Нет комментариев `//!`. Нет отдельного языка спецификаций. Всё находится внутри системы типов».

### Текущие проблемы

- Комментарии `//!` в RFC-022 — это внешний синтаксис, не входящий в систему типов
- Типы спецификаций и обычные типы — это два разных набора, создающих концептуальную избыточность
- Разделение на Debug Build с проверкой и Release Build без проверки нарушает унификацию
- SMT-решатель позиционируется как внешний инструмент, а не как стандартный компонент компилятора
- Проверка типов, проверка заимствований, проверка компиляторных предикатов и макросы следуют разными путями

### Правильная ментальная модель

Проверка типов может быть абстрагирована как чистая функция:

```
isWellTyped : Program → Bool
```

Все проверки на этапе компиляции — простое сопоставление типов, обнаружение конфликтов заимствований, проверка компиляторных предикатов — это подзадачи этой функции. Они используют общий конвейер вычислений, различаясь лишь сложностью выражений и стратегией вычисления.

## Предложение

### 1. `{}` — пространство проверки решателя: тип — это утверждение, верификация — это проверка типов

`{}` в YaoXiang — это пространство проверки решателя. Всё внутри — это утверждения, и решатель гарантирует истинность каждого из них.

```
Point: Type = { x: Float, y: Float }
#               ^^^^^^^^^^^^^^^^^^^^^  решатель гарантирует, что x — это Float, а y — это Float

List: (T: Type) -> Type = { data: Array(T) }
#                           ^^^^^^^^^^^^^^^  решатель гарантирует, что data — это Array(T)
```

**Обобщённые типы — это частный случай компиляторных предикатов.**

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }
#          ^^^^^^              ^^^^^^
#          параметр в сигнатуре    внутри {} только утверждение
#          решатель проверяет x > 0 при вызове во время компиляции

List: (T: Type) -> Type = { data: Array(T) }
#      ^^^^^^^^              ^^^^^^^^^^^^^^^
#      параметр в сигнатуре  решатель проверяет type_of(T) == Type, type_of(data) == Array(T)
```

Один и тот же паттерн: `name: (params) -> Type = { утверждение }`. Решатель не различает «утверждения типа» и «утверждения значения».

**Инварианты цикла не нужно записывать отдельно. Аннотация типа на переменной — это инвариант цикла по Флойду-Хоару.**

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0   # решатель проверяет: 0 == sum(arr[0..0])  → True
    mut i: UpTo(arr.len) = 0     # решатель проверяет: 0 <= 0 <= arr.len   → True
    while i < arr.len {
        s += arr[i]  # решатель проверяет: s_new == sum(arr[0..i+1])
        i += 1       # решатель проверяет: i+1 <= arr.len сохраняется
    }
    return s  # s: SumUpTo(arr, arr.len) = sum(arr[0..arr.len])
}
```

Компилятор генерирует условия верификации для тела цикла один раз — индуктивная гипотеза (аннотация типа) → операция присваивания → удовлетворяет ли новое значение аннотации типа. После того как SMT-решатель докажет индуктивный шаг, все итерации автоматически покрываются. Не нужны `: decreases`, не нужны `: Invariant`, не нужно индуктивное доказательство — компилятор разлагает индукцию на локальные VC для каждого присваивания.

### 2. Предусловия/постусловия: компиляторные предикаты в типах параметров и возвращаемого типа

Отказываемся от `//! requires`/`//! ensures` из RFC-022. Компиляторные предикаты используются как аннотации типа параметра или возвращаемого типа:

```yaoxiang
# Предусловие: тип параметра содержит компиляторный предикат
divide: (a: Int, b: Positive) -> Int = a / b
#                       ^^^^^^^^  решатель проверяет на сайте вызова { b > 0 }

# Постусловие: возвращаемый тип — компиляторный предикат
IsMax: (T: Ord, arr: Array(T), result: T) -> Type = {
    forall j in 0..arr.len: result >= arr[j]
}

max: (T: Ord) -> ((arr: NonEmpty(Array(T))) -> IsMax(T, arr)) = {
    #                                           ^^^^^^^^^^^^^^
    #  IsMax(T, arr) частично применён к arr, возвращает (result: T) -> Type
    #  решатель проверяет на точке возврата { forall j: result >= arr[j] }
    result = arr[0]
    for i in 1..arr.len {
        if arr[i] > result { result = arr[i] }
    }
    return result
}
```

### 3. Конвейер компиляторного вычисления Bool-значений

Все проверки на этапе компиляции используют единый конвейер:

```
Компилятор встречает Bool-выражение, которое нужно вычислить
        │
        ├── Равенство типов (T1 == T2)
        │   → компилятор определяет напрямую
        │
        ├── Условия конфликта токенов (!conflicting(tokens))
        │   → анализ активности с учётом потока (отслеживание атрибутов Dup/Linear)
        │
        ├── Редукция зависимых типов (n + m упрощается)
        │   → система переписывания термов на этапе компиляции
        │
        ├── Компиляторные предикаты (x > 0, forall...)
        │   → компилятор + SMT-решатель
        │
        └── Импликации в логике Хоара (P ⇒ Q)
            → SMT-решатель
                    │
                    ▼
             ┌──────────┐
             │ True     │  → компиляция успешна
             │ False    │  → ошибка компиляции + контрпример
             │ Превышен │  → ошибка компиляции + позиция предиката + потребление
             │ бюджет   │
             └──────────┘
```

Результат компиляторного вычисления Bool-значений может быть только трёх видов — это **три возможных результата одной операции вычисления**:

```
eval_compile_time : BoolExpr → True | False | Превышен_бюджет
```

Это не вопрос дизайна, а неизбежный вывод из проблемы остановки. Компилятор вычисляет произвольное Bool-выражение — теорема Райса утверждает неразрешимость. Компиляторный вычислитель тоже является программой, и перед ним та же проблема остановки. Поэтому результатов может быть три:

- `True` → за halting, ответ истина
- `False` → за halting, ответ ложь
- `Превышен бюджет` → не остановился заданного ресурсного上限а

Жёсткое ограничение бюджета — это инженерное решение проблемы остановки. Никаких ручек — потому что дать их означает спрашивать пользователя «как вы думаете, остановится ли ваша программа», на что не может ответить ни пользователь, ни компилятор.

**Ключевые аспекты дизайна**:
- **Три результата из одного источника**: `True`/`False`/`Превышен_бюджет` — это три выхода единого конвейера вычислений
- **Жёсткое ограничение бюджета**: количество шагов, время, глубина инстанцирования кванторов — фиксированы внутри компилятора, ручек нет
- **Автоматический выбор стратегии**: компилятор автоматически выбирает стратегию вычисления по сложности выражения; если может решить сам — решает, если нет — вызывает SMT
- **Инженерная реальность**: подавляющее большинство предикатов (`x > 0`, `arr.len > 0`, `0 <= idx < arr.len`) — это линейная арифметика, SMT возвращает True/False за миллисекунды. Превышение бюджета произойдёт только когда пользователь написал сложный предикат за пределами разрешимого фрагмента

**Многоуровневые зависимости внутри конвейера**: перечисленные вычислители используют общий интерфейс, но имеют порядок вычисления. Равенство типов — предпосылка для всего последующего анализа; проверка владения/токенов зависит от информации о типах; проверка уточняющих предикатов зависит от результатов первых двух уровней. Компилятор вычисляет уровнями; выражение, не прошедшее нижний уровень, не попадает на верхний — чтобы не тратить бюджет SMT-решателя на программу с ошибками типов.

```
Порядок вычисления (единый конвейер, многоуровневое планирование)
├── Уровень 0: Равенство типов (T1 == T2)
│   └── Структурная унификация → при неудаче дальнейшее бессмысленно, возврат False
├── Уровень 1: Владение/конфликт токенов
│   └── Анализ активности с учётом потока → при неудаче безопасность памяти не гарантирована, возврат False
└── Уровень 2: Уточняющие предикаты/Импликации Хоара
    └── Компилятор + SMT → False (контрпример) или Превышен_бюджет
```

Каждый уровень по-прежнему возвращает `True/False/Превышен_бюджет`, используя общий интерфейс и общую систему бюджетирования.

### 4. Унификация трёх уровней функций

| Уровень | Время выполнения | Вход | Выход | Пример |
|---------|------------------|------|-------|--------|
| Функция уровня значения | Runtime | Значение | Значение | `add: (a: Int, b: Int) -> Int = a + b` |
| Конструктор типа | Компиляция | Тип/значение | Type | `List: (T: Type) -> Type = { data: Array(T) }` |
| Компиляторный предикат | Компиляция | Значение | Type | `Positive: (x: Int) -> Type = { x > 0 }` |

Все используют одинаковый синтаксис `name: type = value`. Компиляторные предикаты и конструкторы типов проходят через один и тот же конвейер компиляторных вычислений — `{}` — это пространство проверки решателя.

### 5. Циклы: генерация условий верификации по Флойду-Хоару

Для циклов не нужны отдельные аннотации `: Invariant(...)` или `: decreases(...)`. Аннотации компиляторных предикатов на переменных определяют утверждения в стиле Флойда-Хоара — компилятор генерирует условия верификации из аннотаций типов, а SMT-решатель проверяет, сохраняет ли каждое присваивание тип.

Основной механизм: каждая операция присваивания соответствует тройке Хоара `{P} x := e {Q}`, условие верификации — это `P ⇒ Q[e/x]`. Компилятор генерирует условие верификации для тела цикла один раз — после того как SMT-решатель докажет индуктивный шаг, все итерации автоматически покрываются.

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0   # проверка: 0 == sum(arr[0..0])  → True
    mut i: UpTo(arr.len) = 0     # проверка: 0 <= 0 <= arr.len   → True
    while i < arr.len {
        # Компилятор генерирует VC для тела цикла один раз:
        #
        # s += arr[i]:
        #   Посылка (индуктивная гипотеза): s == sum(arr[0..i])
        #   Условие верификации: s_new == sum(arr[0..i+1])
        #   Подстановка s_new = s_old + arr[i]:
        #     s_old + arr[i] == sum(arr[0..i]) + arr[i] == sum(arr[0..i+1])
        #   SMT-решатель: линейная арифметика, миллисекунды → True
        #
        # i += 1:
        #   Посылка: i < arr.len (условие цикла) + i >= 0 (типовая граница)
        #   Условие верификации: i+1 >= 0 и i+1 <= arr.len
        #   SMT: прямо выводится из посылки → True
        s += arr[i]
        i += 1
    }
    return s  #此时 s: SumUpTo(arr, arr.len), 即 s == sum(arr[0..arr.len])
}
```

Инвариант цикла — это аннотация типа на переменной. Программист пишет тип, компилятор проверяет индуктивный шаг. Компилятору не нужно «обнаруживать» инвариант и не нужно «автоматически делать индукцию» — он разлагает индуктивное доказательство на локальные условия верификации для каждой операции присваивания и передаёт их SMT-решателю для покорения.

### 6. Проверка завершаемости

Полностью автоматическая на этапе компиляции. Циклы, которые компилятор может доказать завершающимися, проходят; те, которые не может доказать, выдают ошибку компиляции — программист обязан сделать цикл анализируемым для компилятора. Никаких полуавтоматических аннотаций.

#### 6.1 Принципы проектирования

Компилятор автоматически извлекает информацию для доказательства завершаемости из двух источников:

1. **Аннотации типов переменных**: граничные ограничения в уточняющих типах (например, `UpTo(n)` даёт верхнюю границу `n` и нижнюю границу `0`)
2. **Операции в теле цикла**: операции, применяемые к переменной при каждой итерации

Компилятор пытается четыре стратегии синтеза меры по приоритету и останавливается, найдя подходящую.

#### 6.2 Стратегия 1: Автоматический синтез линейной ранк-функции

Когда у переменной есть линейная граничная аннотация, компилятор перечисляет кандидатов в линейные меры и верифицирует их через SMT.

```
Вход:
  Переменные v₁: UpTo(u₁), v₂: UpTo(u₂), ... (переменные с верхней и нижней границами)
  Условие цикла cond
  Множество присваиваний в теле цикла

Алгоритм:
  1. Извлечение границ каждой переменной из аннотаций типа: [low_i, high_i]
  2. Перечисление кандидатов в меры: v_i, u_i - v_i, v_i - v_j и другие линейные комбинации
  3. Для каждого кандидата m:
     - SMT-проверка m ≥ 0 (из типovых границ)
     - Для каждого пути выполнения в теле цикла SMT-проверка m' < m (строгое убывание)
  4. Найдена подходящая линейная комбинация → завершаемость доказана
```

Область покрытия: произвольная переменная, которой присваивается линейное выражение (`v = a·v + b`), с ограниченной типовой аннотацией. Включает `i += const`, `i -= const`, а также сжатие интервала в стиле бинарного поиска:

```yaoxiang
# Бинарный поиск: low = mid + 1 или high = mid
# Мера high - low строго убывает на обоих путях
binary_search: (arr: Sorted(Array(Int)), key: Int) -> Option(Int) = {
    mut low: UpTo(arr.len) = 0
    mut high: UpTo(arr.len) = arr.len
    while low < high {
        let mid = (low + high) / 2
        if arr.data[mid] < key { low = mid + 1 }
        else if arr.data[mid] > key { high = mid }
        else { return Some(mid) }
    }
    return None
}
```

#### 6.3 Стратегия 2: Подсчёт нарушений предиката — автоматическое извлечение меры из целевого типа

Когда возвращаемый тип цикла — это `forall`-уточняющий предикат, а тело цикла содержит локальные модификации данных, компилятор автоматически генерирует меру из определения предиката.

Ключевая идея: **пользовательская спецификация — это материал для рассуждений компилятора**. Компилятору не нужно встраивать понятие «что такое сортировка» — он читает определение `Sorted`, автоматически извлекает меру из определения.

```
Вход:
  Целевой тип: Sorted(arr) = { forall i in 0..arr.len-1: arr[i] <= arr[i+1] }
  Операции в теле цикла: обмен соседних элементов

Алгоритм:
  1. Разбор определения предиката: forall i in range: cond(i, arr)
  2. Автоматическая генерация меры: violation_count = |{ i | ¬cond(i, arr) }|
  3. Анализ влияния операций на меру:
     - Обмен соседних элементов arr[j], arr[j+1] = arr[j+1], arr[j]
     - Затрагивает только три пары индексов: j-1, j, j+1
     - Если arr[j] > arr[j+1] (нарушение предиката), после обмена эта пара удовлетворяет предикату
     - violation_count уменьшается как минимум на 1
  4. Верхняя граница: n·(n-1)/2 (максимальное число соседних инверсий), нижняя граница: 0
  → завершаемость доказана
```

```yaoxiang
sort: (arr: Array(Int)) -> Sorted(arr) = {
    mut i: UpTo(arr.len) = 0
    while i < arr.len - 1 {
        mut j: UpTo(arr.len - i - 1) = 0
        while j < arr.len - i - 1 {
            if arr.data[j] > arr.data[j+1] {
                arr.data[j], arr.data[j+1] = arr.data[j+1], arr.data[j]
            }
            j += 1
        }
        i += 1
    }
    return arr
}
```

#### 6.4 Стратегия 3: Паттерны ограниченного возрастания/убывания

`v += const` (константа), переменная с верхней граничной типовой аннотацией → мера `upper_bound - v` убывает на `const` каждую итерацию, нижняя граница 0. Это вырожденный случай стратегии 1, компилятор обрабатывает его в первую очередь.

#### 6.5 Стратегия 4: Шаблон меры с мультипликативным масштабированием

`v *= const` (const > 1), переменная с верхней и нижней граничными типовыми аннотациями. Компилятор имеет встроенный шаблон логарифмической меры `ceil(log_const(upper/v))`, каждое умножение на const уменьшает меру на 1.

```yaoxiang
mut i: Positive = 1
while i < n {
    # Компилятор автоматически выводит: мера ceil(log₂(n/i)), каждое умножение на 2 уменьшает меру на 1
    i *= 2
}
```

#### 6.6 Разделение завершаемости и корректности

Доказательство завершаемости и доказательство корректности независимы:
- **Завершаемость**: четыре стратегии автоматически доказывают, что цикл завершится за конечное число шагов
- **Корректность**: продвигает ли тело цикла состояние к целевому типу, проверяется SMT-решателем через условия верификации

Оба проходят → компиляция успешна. Завершаемость доказана, но корректность не прошла → ошибка компиляции + контрпример. Завершаемость не доказана → ошибка компиляции с указанием неанализируемой переменной или операции.

#### 6.7 Проверка завершаемости рекурсивных функций

Для рекурсивных функций, которые должны вычисляться на этапе компиляции, компилятор проверяет убывание параметров:

```yaoxiang
factorial: (n: Int) -> Int = {
    if n <= 1 { return 1 }
    return n * factorial(n - 1)  # Компилятор анализирует: n-1 < n → убывает → завершается
}

# Использование на этапе компиляции — компилятор гарантирует завершение factorial при компиляции
vec: Vec(factorial(5)) = Vec(120)()  # 5! = 120, вычислено на этапе компиляции
```

| Сценарий | Поведение |
|----------|-----------|
| Компилятор может проанализировать рекурсивное убывание (например, `n-1`) | Вычисление на этапе компиляции |
| Не убывает / невозможно определить убывание | Ошибка компиляции |
| Вызов в runtime (не в типовой позиции) | Проверка завершаемости не требуется |

#### 6.8 Жёсткие границы

`i = f(i)` где `f` необратима, не замкнута, не сохраняет никакой монотонности — математически невозможно автоматически доказать завершаемость. Ошибка компиляции:

> Данный цикл невозможно автоматически доказать завершающимся. Цикловая переменная зависит от неанализируемой функции `f`. Используйте итеративный паттерн, анализируемый компилятором.

Это не провал компилятора. Любой код, безопасность которого невозможно статически доказать, не должен проходить компиляцию.

### 7. SMT-решатель: внутренний компонент компиляторного вычислителя

SMT-решатель в традиционных языках — это внешний инструмент (как F\* вызывает Z3, Dafny вызывает Z3). В YaoXiang это внутренний компонент конвейера компиляторных вычислений — вызывается только когда компилятор сам не может напрямую определить результат.

```
Bool-выражение на этапе компиляции
        │
        ├── Компилятор может определить напрямую (простые сравнения, простая арифметика,
        │   тривиальные формулы после constant folding)
        │   → возврат True/False напрямую
        │
        └── Компилятор не может определить напрямую (кванторы, символьные переменные)
            → предварительная редукция типов (factorial(5) → 120)
            → перевод в формат SMT-LIB
            → отправка в Z3/CVC5 (с ограничением бюджета)
            → возврат: unsat (выполнено) │ sat (контрпример) │ budget_exceeded
```

**Бюджет решения — жёсткое ограничение, как глубина стека**:

| Измерение бюджета | Описание | Жёсткий上限 |
|-------------------|----------|------------|
| Количество шагов | Шаги переписывания/резольюции | Фиксированное значение внутри компилятора |
| Время | Время решения одного предиката | Фиксированное значение внутри компилятора |
| Глубина инстанцирования кванторов | Глубина раскрытия forall/exists | Фиксированное значение внутри компилятора |

Превышение бюджета — ошибка компиляции. Никакой деградации, никаких runtime-проверок, никакого silent pass. Сообщение об ошибке точно указывает позицию предиката в исходном коде и потреблённый бюджет.

**Почему это практически работает**: в инженерной практике 95% реальных предикатов — это линейная арифметика — `x > 0`, `arr.len > 0`, `0 <= idx < arr.len` — всё это находится в разрешимом фрагменте, и SMT-решатель возвращает True/False за миллисекунды. При встрече с редкими сложными предикатами, превышающими бюджет, пользователь просто разбивает предикат на более простые.

Зависимые типы перед вызовом SMT проходят предварительную редукцию: `factorial(5)` вычисляется на этапе компиляции и даёт `120`, `append([1,2], [3])` вычисляется и даёт `[1,2,3]`. Эти детерминированные вычисления значений не потребляют бюджет SMT.

Программисту не нужно знать о существовании SMT. Ментальная модель: **компилятор доказывает — проходит, не может — ошибка** — полностью согласуется с проверкой типов.

### 8. Композиция компиляторных предикатов

Компиляторные предикаты — это функции, возвращающие Type, их композиция естественно реализуется через функциональную композицию:

```yaoxiang
SortedNonEmpty: (T: Ord, arr: Array(T)) -> Type = {
    Sorted(T, arr) && NonEmpty(T, arr)
}
```

### 9. Примеры кода

#### 9.1 Безопасное деление

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

divide: (a: Int, b: Positive) -> Int = a / b

result = divide(10, 2)   # ✅ решатель проверяет { 2 > 0 } → True
# result = divide(10, 0)  # ❌ решатель проверяет { 0 > 0 } → False
```

#### 9.2 Безопасный доступ к массиву

```yaoxiang
InBounds: (T: Type, idx: Int, arr: Array(T)) -> Type = { 0 <= idx && idx < arr.len }

get: (T: Type) -> ((arr: Array(T), idx: InBounds(T, arr)) -> T) = arr.data[idx]

arr = Array(Int)(1, 2, 3)
x = get(arr, 1)   # ✅ решатель проверяет { 0 <= 1 && 1 < 3 } → True
# y = get(arr, 5)  # ❌ решатель проверяет { 0 <= 5 && 5 < 3 } → False
```

#### 9.3 Корректность сортировки

```yaoxiang
Sorted: (T: Ord, arr: Array(T)) -> Type = {
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

sort: (T: Ord) -> ((arr: Array(T)) -> Sorted(T)) = {
    result = arr.clone()
    # ... реализация алгоритма сортировки ...
    return result
}
```

#### 9.4 Циклы: генерация VC компилятором

```yaoxiang
SumUpTo: (arr: Array(Int), i: Int) -> Type = { s: Int; s == sum(arr[0..i]) }
UpTo: (n: Int) -> Type = { i: Int; 0 <= i <= n }

sum: (arr: Array(Int)) -> Int = {
    mut s: SumUpTo(arr, 0) = 0
    mut i: UpTo(arr.len) = 0
    while i < arr.len {
        s += arr[i]
        i += 1
    }
    return s
}
```

## Детальное проектирование

### Синтаксические изменения

| До (RFC-022) | После (этот RFC) |
|---|---|
| `//! requires: NonEmpty(n) = n > 0` | Компиляторный предикат как тип параметра `(b: Positive)` |
| `//! ensures: ExistsMax(result, arr)` | Возвращаемый тип — компиляторный предикат `-> IsMax(T, arr)` |
| `/*! invariant: ... !*/` | Аннотация типа компиляторного предиката на переменной — инвариант Флойда-Хоара |
| `//! decreases: n` | Полностью автоматический вывод меры функцией компилятором |
| Спецификация — комментарий | Спецификация — типовая система |

### Синтаксис

**Для компиляторных предикатов нет нового синтаксиса.** `{}` — это пространство проверки решателя, полностью совместимое с существующим синтаксисом определения типов. Компиляторный предикат — это функция, возвращающая Type — `name: (params) -> Type = { утверждение }`.

```bnf
# Компиляторный предикат = функция, возвращающая Type, {} содержит утверждения решателя
# Используется существующий синтаксис функций/типов, новые BNF-правила не требуются
predicate ::= identifier ':' params '->' 'Type' '=' '{' assertions '}'
```

### Влияние на систему типов

- **Вселенная типов**: компиляторные предикаты находятся на уровне Type₂ — функции, принимающие значения и возвращающие Type, на том же уровне, что и конструкторы типов
- **Взаимодействие с обобщениями**: компиляторные предикаты могут иметь обобщённые параметры, например `NonEmpty: (T: Type) -> (arr: Array(T)) -> Type`
- **Взаимодействие с владением**: выражения в компиляторных предикатах подчиняются правилам владения, могут только читать, не могут писать
- **Вывод типов**: параметры компиляторных предикатов участвуют в HM-выводе типов

### Представление во время выполнения

Компиляторные предикаты во время выполнения **полностью стираются**. `Positive: (x: Int) -> Type = { x > 0 }` — параметр `b: Positive` во время выполнения представлен просто как `Int`. Уточняющее условие `{ x > 0 }` проверено SMT-решателем на этапе компиляции, после проверки не оставляет никаких следов во время выполнения.

- `Positive(5)` → во время выполнения представлен как `Int(5)`, уточняющее условие `{ 5 > 0 }` проверено, стёрто
- `SumUpTo(arr, 0)` → во время выполнения представлен как `Int(0)`, равенство `0 == sum(arr[0..0])` проверено, стёрто
- Помещение компиляторного предиката в типовую позицию (например, `f(x: Positive)`) не создаёт никакого обёрточного типа, не выделяет дополнительной памяти, не вставляет runtime-проверок

Стирание обобщённых типов и стирание уточняющих типов — это один и тот же принцип: оба являются функциями этапа компиляции, которые полностью исчезают после вычисления на этапе компиляции. В Release Build компиляторные предикаты имеют нулевые накладные расходы во время выполнения — это прямое следствие того, что «предикаты — это функции».

**Ограничение взаимодействия с `ref`**: компиляторные предикаты могут ссылаться только на неизменяемые заимствования или значения, владение которыми уже передано. Попытка использовать изменяемое заимствование в компиляторном предикате — компилятор не может гарантировать во время компиляции, что результат верификации останется верным во время выполнения — такое использование вызывает ошибку компиляции.

### Изменения в компиляторе

1. **Парсер**: компиляторные предикаты используют стандартный синтаксис функций, дополнительные правила парсинга не требуются
2. **Конвейер компиляторных вычислений**: унифицированный интерфейс возврата Bool, автоматический выбор стратегии
3. **SMT-бэкенд**: интеграция Z3/CVC5, трансляция в формат SMT-LIB
4. **Генерация условий верификации**: исчисление WP/SP + обязательства по доказательству инвариантов цикла
5. **Сообщения об ошибках**: форматирование контрпримеров, привязка к позициям в исходном коде

### Обратная совместимость

- ✅ Код, не использующий компиляторные предикаты, остаётся полностью неизменным
- ✅ Компиляторные предикаты имеют нулевые накладные расходы во время выполнения в Release Build
- ⚠️ Синтаксис `//!` из RFC-022 больше не поддерживается — но 022 никогда не был реализован, затраты на миграцию отсутствуют

## Компромиссы

### Преимущества

- **Полная реализация соответствия Curry-Howard**: тип — это утверждение, программа — это доказательство, `name: Proposition = Proof`
- **Унификация**: компиляторные предикаты и обычные функции используют полностью идентичный синтаксис, нет концептуального разделения
- **Прозрачность SMT**: программисту не нужно знать о существовании SMT, ментальная модель согласована с проверкой типов
- **Постепенное внедрение**: можно начать с одного компиляторного предиката и постепенно расширять покрытие
- **Нулевые накладные расходы во время выполнения**: все вычисления происходят на этапе компиляции, в Release Build не вставляются никакие runtime-проверки

### Недостатки

- **Время компиляции**: SMT-решение увеличивает время компиляции, но жёсткое ограничение бюджета гарантирует контролируемый上限
- **Границы возможностей SMT**: сложные предикаты за пределами линейной арифметики могут превысить бюджет, пользователю потребуется разбивать предикаты
- **Кривая обучения**: написание эффективных компиляторных предикатов требует определённого опыта
- **Сложность реализации**: унификация конвейера компиляторных вычислений требует тщательного проектирования

### Снижение рисков

- Жёсткое ограничение бюджета SMT-решения (шаги/время/глубина инстанцирования), превышение бюджета — ошибка компиляции
- Предварительная редукция зависимых типов: детерминированные вычисления значений выполняются заранее, SMT получает только недетерминированную часть
- Инкрементная верификация: проверяются только изменённые модули
- Чёткие сообщения об ошибках +展示 контрпримеров + отчёт о потреблении бюджета

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| RFC-022: спецификации в форме комментариев `//!` | Спецификации и типы разделены, нарушение соответствия Curry-Howard |
| Отдельные файлы спецификаций (например, CVL) | Спецификации отделены от кода, увеличение затрат на сопровождение |
| Только runtime-проверки | Невозможна статическая гарантия корректности |
| Внешние пруверы (например, Coq) | Крутая кривая обучения, разделение с компилятором |
| **Наше решение: компиляторные предикаты как первоклассные сущности** | ✅ |
