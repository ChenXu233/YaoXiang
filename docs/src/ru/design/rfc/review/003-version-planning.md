## ⚠️ Причина отклонения

**Данный RFC не соответствует спецификации RFC и был отклонён.**

### Анализ проблем

| Проблема | Описание |
|----------|----------|
| **Неправильный тип документа** | RFC — это документ решений о том, «почему именно так спроектировано», а данная статья — документ управления проектом о том, «где мы сейчас и куда идём» |
| **Отсутствие проектных решений** | Нет анализа компромиссов при выборе технологий, нет сравнения альтернативных вариантов, нет обоснования дизайна |
| **Чрезмерное планирование** | Запланировано 9 версий (v0.1–v1.0), но выполнено только 1.5, всё после v0.3 —推测 |
| **Высокая стоимость поддержки** | Таблица показателей успеха содержит 50 состояний, что затрудняет поддержку |
| **Путаница с номерами версий** | Пропущен v0.8 без объяснений |

### Правильный тип документа

Данный документ следует вести как **дорожную карту проекта (Roadmap)**, а не как RFC. Дорожная карта фиксирует факты и кракосрочные планы, а RFC фиксирует проектные решения.

## Резюме

План выпуска версий YaoXiang — дорожная карта от v0.1 до v1.0.

**Основные цели**:
1. **Компиляция в байт-код**: Поддержка REPL и пошаговой компиляции
2. **Самокомпиляция (Bootstrap)**: Использование YaoXiang для написания компилятора YaoXiang
3. **AOT-компиляция**: Компиляция байт-кода в машинный код для целевой платформы

## 一、Мотивация

### Зачем нужно планирование версий?

1. **Управление проектом**: Разбиение целей на выполнимые этапы
2. **Ожидания пользователей**: Позволить пользователям понять этапы развития языка
3. **Распределение ресурсов**: Определение приоритетов на каждом этапе
4. **Контроль рисков**: Своевременное выявление проблем и корректировка направления

### Ключевые проектные решения

- **Приоритет байт-кода**: Сначала реализовать интерпретацию, затем AOT
- **Поэтапная поставка**: Каждая версия имеет работающую функциональность
- **Обратная совместимость**: До v1.0 API может изменяться, но с предварительным уведомлением
- **Валидация через самокомпиляцию**: Доказать выразительность языка через самокомпиляцию
- **Многоуровневая оптимизация производительности**: Сначала заставить работать, затем оптимизировать

## 四、Трёхуровневая стратегия компиляции

| Уровень | Версия | Входные данные | Выходные данные | Описание |
|---------|--------|----------------|-----------------|----------|
| L1: Байт-код | v0.1+ | Исходный код (.yx) | Байт-код (.yxb) | VM интерпретирует |
| L2: Самокомпиляция | v0.9+ | Исходный код YaoXiang | Байт-код | Компилирует сам себя |
| L3: AOT | v1.0+ | Исходный код/байт-код | Машинный код | Нативная производительность |

**Обоснование приоритета байт-кода**:
1. **Поддержка REPL**: Незамедлительная компиляция введённого кода, интерактивная разработка
2. **Поэтапная компиляция**: При изменении одной функции достаточно перекомпилировать только её
3. **Кроссплатформенность**: Файлы .yxb работают на разных платформах, нужна только VM для соответствующей платформы

## 五、Стратегия зависимостей

**Краткосрочная перспектива**: Вызов библиотек Rust для повторного использования crates.io (Cargo-паразитизм)

**Текущие зависимости**:
- Параллелизм: parking_lot, crossbeam, rayon
- Структуры данных: indexmap, hashbrown, smallvec
- Сети: tokio
- Сериализация: serde, ron

**Долгосрочная перспектива**: Собственная стандартная библиотека и менеджер пакетов
