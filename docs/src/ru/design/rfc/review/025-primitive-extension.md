# RFC-025: Расширяемый механизм примитивных типов

## Резюме

Настоящий документ определяет **расширяемый механизм примитивных типов** (Extensible Primitive Type Mechanism) компилятора YaoXiang (`Primitive::Extension`). Механизм позволяет внешнему коду регистрировать в компиляторе пользовательские примитивные типы, что даёт компилятору возможность поддерживать предметно-специфичные типы (кубиты, буферы GPU, SIMD-векторы, аппаратные регистры и т. д.) без их жёсткого кодирования.

## Мотивация

### Зачем нужен этот механизм?

В настоящее время компилятор жёстко кодирует все примитивные типы: `Int`, `Float`, `String`, `Bool`, `Unit`. Добавление каждого нового примитивного типа требует изменения нескольких мест в исходном коде компилятора — проверщика типов, генератора кода, анализатора владения, оптимизатора.

Это нарушает принцип открытости/закрытости: открыт для расширения, закрыт для модификации.

### Границы проектирования

```
Жёстко закодированные базовые типы (основа языка): Int, Float, String, Bool, Unit
Динамически расширяемые типы (предметные плагины):  регистрируются через Primitive::Extension
```

Базовые типы жёстко кодируются, поскольку компилятор глубоко зависит от их семантики (арифметические операции, условные переходы, хеширование, сравнение). Расширяемые типы являются для компилятора **непрозрачными значениями** — компилятору известны только их размер, выравнивание и свойства владения, но не их внутренняя семантика.

Это не попытка «объединить все типы в динамически загружаемые». Базовые типы и расширяемые типы — это две разные сущности.

## Предложение

### Основной дизайн

#### 1. Атрибуты типа Extension

Каждый регистрируемый расширяемый примитивный тип должен объявлять следующие атрибуты:

```rust
pub struct PrimitiveExtension {
    /// Имя типа, например "Qubit", "Buffer", "Vec128"
    pub name: String,

    /// Размер в байтах (тип с фиксированным размером)
    /// None означает, что размер неизвестен на этапе компиляции (определяется во время выполнения)
    pub size: Option<usize>,

    /// Требование к выравниванию
    pub align: Option<usize>,

    /// Допускается ли неявное копирование
    /// false = семантика Move (присваивание перемещает значение, например Qubit)
    /// true = семантика Copy (присваивание копирует значение, например Vec128)
    pub is_copy: bool,

    /// Допускается ли пустое значение (zero-sized type)
    pub allow_zst: bool,
}
```

#### 2. Интерфейс регистрации

```rust
// Внутренний API компилятора
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".to_string(),
    size: Some(0),           // Логический размер равен 0, физическое состояние находится в квантовом процессоре
    align: Some(1),
    is_copy: false,          // Семантика Move, соответствует no-cloning
    allow_zst: true,
});
```

После регистрации `Qubit` становится допустимым примитивным типом в системе типов и может использоваться в объявлениях переменных, параметрах функций и полях структур.

#### 3. Поведение проверщика типов

Расширяемые примитивные типы в проверке типов подчиняются следующим правилам:

| Сценарий | Поведение |
|------|------|
| Объявление переменной `q: Qubit = ...` | ✅ Допустимо |
| Параметр функции `fn(q: Qubit)` | ✅ Допустимо |
| Поле структуры `{ q: Qubit }` | ✅ Допустимо |
| Присваивание при `is_copy == false` | Семантика Move, исходная переменная становится недействительной |
| Присваивание при `is_copy == true` | Семантика Copy, исходная переменная сохраняется |
| Неявное копирование (многократное использование в функции) | Зависит от `is_copy` |
| Сравнение `==`, `!=` | ❌ Ошибка компиляции (нет встроенного сравнения) |
| Арифметика `+`, `-` | ❌ Ошибка компиляции (нет встроенных операций) |
| Ограничение дженерика `T: Copy` | Выполняется только при `is_copy == true` |

#### 4. Поведение генератора кода

Расширяемые примитивные типы при генерации кода обрабатываются как **непрозрачные значения**:

- LLVM IR: генерируется как `{size} x i8` или структура соответствующего размера
- Никакие специальные инструкции не генерируются — семантику обеспечивает бэкенд или библиотека
- Если бэкенду требуется специальная обработка (например, квантовые гейты QIR), она реализуется через механизм регистрации бэкенда (выходит за рамки данного RFC)

### Примеры

#### Регистрация типа с семантикой Move

```rust
// Кубит: не копируется, размер 0 (физическое состояние находится в QPU)
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".into(),
    size: Some(0),
    align: Some(1),
    is_copy: false,
    allow_zst: true,
});
```

```yaoxiang
# Пользовательский код
q: Qubit = qubit(0)
q2 = q          # ❌ Ошибка компиляции: Qubit — тип с семантикой Move, q становится недействительным
q = H(q)        # ✅ q потребляется, возвращается новое значение q
```

#### Регистрация типа с семантикой Copy

```rust
// SIMD-вектор: копируемый, фиксированный размер
compiler.register_primitive(PrimitiveExtension {
    name: "Vec128".into(),
    size: Some(16),
    align: Some(16),
    is_copy: true,
    allow_zst: false,
});
```

```yaoxiang
# Пользовательский код
a: Vec128 = load_vec128(data)
b = a           # ✅ Семантика Copy, a остаётся действительным
c = add_vec128(a, b)  # ✅ a и b оба доступны
```

## Детальное проектирование

### Изменения в компиляторе

| Компонент | Изменение |
|------|------|
| Система типов | Добавить вариант `Ty::Extension`, хранящий метаданные `PrimitiveExtension` |
| Проверщик типов | Расширяемые типы не участвуют в разрешении встроенных операций, не удовлетворяют встроенным ограничениям trait (если явно не реализованы) |
| Анализатор владения | Решает, Move или Copy семантика, на основе `is_copy` |
| Генератор кода | Генерирует непрозрачные значения в соответствии с `size`/`align`, без специальных инструкций |
| Сообщения об ошибках | Сообщения об ошибках для расширяемых типов ссылаются на `name`, заданное при регистрации |

### Связь с FFI

`Primitive::Extension` и RFC-021 (FFI) ортогональны:

| | Primitive::Extension | FFI |
|---|---|---|
| Назначение | Регистрация нового **типа** | Вызов внешней **функции** |
| Уровень | Система типов | Среда выполнения |
| Пример | `Qubit` — это тип | `native("sin")` — это вызов функции |

Одна предметная область может нуждаться в обоих механизмах одновременно: тип `Qubit` регистрируется через Extension, а функции квантовых гейтов — через FFI.

### Обратная совместимость

- ✅ Полностью обратно совместимо
- Семантика существующих типов не изменяется
- Расширяемые типы — это новая возможность, не затрагивающая существующий код

## Компромиссы

### Преимущества

- ✅ Компилятору не нужно изменять исходный код для каждой новой предметной области
- ✅ Предметные эксперты могут самостоятельно регистрировать типы, не полагаясь на команду компилятора
- ✅ Базовые типы остаются жёстко закодированными, что не ухудшает глубокую оптимизацию компилятором базовых типов
- ✅ Простой интерфейс — одна структура определяет все атрибуты

### Недостатки

- ⚠️ Расширяемые типы не поддерживают встроенные операции — для реализации семантики требуются дополнительные функции или механизмы бэкенда
- ⚠️ При отладке расширяемые типы отображаются как непрозрачные значения, что менее наглядно, чем базовые типы

## Альтернативы

| Подход | Почему не выбран |
|------|--------------|
| Динамическая загрузка всех типов | Базовые типы (Int/Float/Bool) требуют глубокой оптимизации компилятором, динамическая загрузка лишит этих возможностей |
| Жёсткое кодирование для каждой предметной области | Каждая новая область требует изменений в компиляторе, не масштабируется |
| Чисто библиотечный подход (без регистрации типов) | Невозможно гарантировать семантику на уровне системы типов (например, no-cloning), проверка только во время выполнения |

## Стратегия реализации

### Этап 1: Базовый интерфейс

- [ ] Добавить вариант `Ty::Extension` в систему типов
- [ ] Реализовать API `register_primitive`
- [ ] Расширить проверщик типов для обработки семантики `is_copy`
- [ ] Расширить генератор кода для обработки непрозрачных значений
- [ ] Модульные тесты

### Этап 2: Момент регистрации

- [ ] Поддержка пакетной регистрации при инициализации компилятора (файл конфигурации или builder API)
- [ ] Поддержка предварительной регистрации в стандартной библиотеке (модуль `std.primitive` экспортирует определения расширяемых типов)

### Зависимости

- Нет жёстких зависимостей. Может быть реализован независимо от других RFC.

## Открытые вопросы

- [ ] Должны ли расширяемые типы поддерживать реализацию trait (например, позволить `Qubit` реализовать пользовательский trait `QuantumGate`)?
- [ ] Нужны ли хуки жизненного цикла (например, `on_drop`) для поддержки расширяемых типов с семантикой RAII?
- [ ] Формат файла конфигурации: TOML, YAML или собственный синтаксис конфигурации YaoXiang (см. RFC-015)?
