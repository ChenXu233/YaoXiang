# RFC-011a: Реализация интерфейсов и динамическая диспетчеризация

> **Родительский RFC**: [RFC-011: Проект системы дженериков](../accepted/011-generic-type-system.md)
>
> **Настоящий RFC дополняет и заменяет разделы §2.1–2.4 RFC-011, касающиеся ограничений интерфейсов.**

## Аннотация

RFC-011 определяет систему дженериков, но не детализирует механизм реализации интерфейсов. Настоящий документ дополняет его:

1. **Объявление интерфейса**: имя интерфейса пишется прямо в определении типа, без ключевого слова `impl`.
2. **Реализация метода**: поддерживается и внутреннее, и внешнее объявление.
3. **Правила перегрузки**: при разных сигнатурах допускается перегрузка, при одинаковых — ошибка (переопределение запрещено).
4. **Значения по умолчанию**: после поля сразу пишется `= значение`.
5. **Динамическая диспетчеризация**: сбор типов на этапе компиляции + сопоставление интерфейсов, без vtable.

**Ключевая конструкция**:

```yaoxiang
# 接口定义
Animal: Type = {
    speak: (Self) -> String,
}

# 类型定义（内部声明）
Dog: Type = {
    x: Int = 10,
    Animal,  # 接口声明
    speak: (Self) -> String = "Woof",
}

# 外部声明（重载）
Dog.speak: (Self, volume: Int) -> String = "WOOF"

# 异构容器（动态分发）
animals: List(Animal) = [Dog.new(), Cat.new()]
animals[0].speak()  # "Woof"
```

**Устранённая сложность**:
- ❌ Нет ключевого слова `impl`.
- ❌ Нет аннотации `dyn Trait + 'a`.
- ❌ Нет vtable (сбор типов на этапе компиляции + упаковка в перечисление).
- ❌ Нет переопределения (единые правила перегрузки).

## Мотивация

### Недостатки RFC-011

RFC-011 определяет систему дженериков, но не раскрывает:

| Проблема | Описание |
|----------|----------|
| Синтаксис объявления интерфейса | Как объявить, что тип реализует интерфейс? |
| Расположение реализации метода | Внутреннее или внешнее объявление? |
| Правила перегрузки | Как обрабатывать одноимённые методы? |
| Синтаксис значений по умолчанию | Как задать значение по умолчанию для поля? |
| Динамическая диспетчеризация | Как реализовать гетерогенный контейнер? |

### Цели проектирования

1. **Простота**: ключевое слово `impl` не требуется.
2. **Гибкость**: реализация метода возможна и внутри, и снаружи.
3. **Единообразие**: согласованные правила перегрузки.
4. **Удобство**: лаконичный синтаксис значений по умолчанию.
5. **Нулевые накладные расходы**: без vtable, сбор типов на этапе компиляции.

### Сравнение с Rust

| Возможность | Rust | YaoXiang |
|-------------|------|----------|
| Объявление интерфейса | `impl Animal for Dog { ... }` | `Dog: Type = { Animal, ... }` |
| Реализация метода | В блоке `impl` | Внутри или снаружи |
| Перегрузка | Не поддерживается | Поддерживается (при разных сигнатурах) |
| Значение по умолчанию | Требуется `#[default]` | Пишется `= значение` |
| Гетерогенный контейнер | `Vec<Box<dyn Animal + 'a>>` | `List(Animal)` |
| Динамическая диспетчеризация | Поиск по vtable | Сбор типов на этапе компиляции |

## Предложение

### 1. Объявление интерфейса

**Основное правило**: в определении типа имя интерфейса пишется напрямую, ключевое слово `impl` не нужно.

```yaoxiang
# 接口定义
Animal: Type = {
    speak: (Self) -> String,
}

# 类型声明实现接口
Dog: Type = {
    x: Int,
    Animal,  # 接口声明
}
```

**Обработка компилятором**:
1. Распознать, что `Animal` — тип интерфейса.
2. Проверить наличие в `Dog` всех методов, требуемых интерфейсом `Animal`.
3. При успехе → сгенерировать доказательство реализации.
4. При неудаче → ошибка компиляции.

**Синтаксический сахар (эквивалентность)**:

```yaoxiang
Dog: Type = {
    x: Int,
    Animal,  # 等价于展开 Animal 的方法，但保留来源标记
}

# 等价于（但保留来源信息）
Dog: Type = {
    x: Int,
    speak: (Self) -> String,  # 来自 Animal
}
```

**Зачем нужен маркер источника**:
- Прямое раскрытие теряет информацию об источнике.
- Маркер источника используется для генерации доказательства реализации.
- Во время выполнения по доказательству находится корректный метод.

### 2. Реализация метода

**Основное правило**: поддерживается и внутреннее, и внешнее объявление метода.

#### 2.1 Внутреннее объявление

```yaoxiang
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",  # 方法实现在内部
}
```

#### 2.2 Внешнее объявление

```yaoxiang
Dog: Type = {
    x: Int,
    Animal,
}

# 方法实现在外部
Dog.speak: (Self) -> String = "Woof"
```

#### 2.3 Смешанное объявление

```yaoxiang
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",  # 部分方法在内部
}

# 部分方法在外部
Dog.play: (Self) -> Void = { ... }
```

**Обработка компилятором**:
1. Собрать все определения (внутренние и внешние).
2. Сгруппировать по сигнатуре (перегрузка).
3. Проверить наличие переопределений (сообщить об ошибке).
4. Проверить полноту интерфейса.
5. Сгенерировать доказательство реализации.

### 3. Перегрузка и переопределение

**Основное правило**:
- Разные сигнатуры → перегрузка → разрешена.
- Одинаковые сигнатуры → переопределение → ошибка.

#### 3.1 Перегрузка (допускается)

```yaoxiang
# 参数类型不同，允许重载
Dog.speak: (Self) -> String = "Woof"
Dog.speak: (Self, volume: Int) -> String = "WOOF"
```

#### 3.2 Переопределение (запрещено)

```yaoxiang
# 签名完全相同，禁止覆盖
Dog.speak: (Self) -> String = "Woof"
Dog.speak: (Self) -> String = "Bark"  # ❌ 报错：覆盖不允许
```

**Сообщение об ошибке**:

```
Ошибка: повторное определение Dog.speak(Self) -> String
  --> файл2:5:1
  |
5 | Dog.speak: (Self) -> String = "Bark"
  | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ повторное определение
  |
  --> файл1:3:1
  |
3 | Dog.speak: (Self) -> String = "Woof"
  | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ первое определение
```

#### 3.3 Единые правила

**Внутренние и внешние объявления подчиняются одним и тем же правилам перегрузки/переопределения**:

```yaoxiang
# 内部声明
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

# 外部声明（重载，允许）
Dog.speak: (Self, volume: Int) -> String = "WOOF"

# 外部声明（覆盖，禁止）
Dog.speak: (Self) -> String = "Bark"  # ❌ 报错
```

### 4. Значения по умолчанию

**Основное правило**: после поля сразу пишется `= значение`, что избавляет от необходимости в конструкторах.

```yaoxiang
Dog: Type = {
    x: Int = 10,  # 默认值
    y: Int = 20,  # 默认值
    Animal,
}
```

**Генерация конструкторов компилятором**:

```yaoxiang
# 所有字段都有默认值 → 生成无参构造函数
Dog.new: () -> Dog = { x: 10, y: 20 }

# 部分字段有默认值 → 生成部分参数构造函数
Dog.new: (x: Int) -> Dog = { x: x, y: 20 }
Dog.new: (y: Int) -> Dog = { x: 10, y: y }

# 全参数构造函数
Dog.new: (x: Int, y: Int) -> Dog = { x: x, y: y }
```

**Внешнее объявление значений по умолчанию**:

```yaoxiang
Dog: Type = {
    x: Int,
    y: Int,
    Animal,
}

# 外部声明默认值
Dog.x: Int = 10
Dog.y: Int = 20
```

**Эквивалентно внутреннему объявлению**.

### 5. Реализация в компиляторе

#### 5.1 Дескриптор интерфейса

```rust
// 编译器内部：接口描述符
struct InterfaceDescriptor {
    name: String,
    methods: Vec<MethodSignature>,
}
```

#### 5.2 Определение типа

```rust
// 编译器内部：类型定义
struct TypeDefinition {
    name: String,
    fields: Vec<Field>,
    interface_implementations: Vec<InterfaceImplementation>,
}

// 接口实现（保留来源信息）
struct InterfaceImplementation {
    interface: InterfaceId,
    methods: HashMap<MethodId, FunctionBody>,
}
```

#### 5.3 Доказательство реализации

```rust
// 编译器内部：实现证明
struct ImplementationProof {
    type_id: TypeId,
    interface_id: InterfaceId,
    methods: Vec<MethodPointer>,
}
```

#### 5.4 Процесс компиляции

```
1. 解析类型定义，收集接口声明
2. 收集所有方法定义（内部和外部）
3. 按签名分组（重载）
4. 检查覆盖（报错）
5. 检查接口完整性
6. 生成实现证明
7. 运行时，值携带实现证明
```

### 6. Динамическая диспетчеризация

**Ключевая идея**: сбор типов на этапе компиляции + сопоставление интерфейсов, без vtable.

#### 6.1 Гетерогенный контейнер

```yaoxiang
# 接口定义
Animal: Type = {
    speak: (Self) -> String,
}

# 类型定义
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

Cat: Type = {
    y: Int,
    Animal,
    speak: (Self) -> String = "Meow",
}

# 异构容器
animals: List(Animal) = [Dog.new(), Cat.new()]
animals[0].speak()  # "Woof"
animals[1].speak()  # "Meow"
```

#### 6.2 Сбор типов на этапе компиляции

**Основная стратегия: отслеживание владения и инкрементальное построение.** Вместо сканирования на этапе компиляции всех типов, реализующих интерфейс, выполняется инкрементальный сбор в каждой **точке операции владения** контейнера `List(Animal)`:

```yaoxiang
// 构造点
animals: List(Animal) = [Dog.new()]       // AnimalGroup = { Dog(Dog) }

// append 点
animals.append(Cat.new())                  // 编译器在 append 处看到 Cat → 扩展为 { Dog, Cat }
animals.append(Bird.new())                 // 再扩展 { Dog, Cat, Bird }
```

**Обработка компилятором (инкрементально)**:

1. При первом конструировании `List(I)` → сгенерировать начальное перечисление (все известные в текущей компиляционной единице типы-конструкторы).
2. При каждом `append` / `push` / присваивании по индексу → проверить, присутствует ли тип значения в перечислении; если нет — расширить набор вариантов.
3. Для итогового перечисления сгенерировать мономорфизированный `match` для диспетчеризации.
4. Между компиляционными единицами: на этапе компоновки объединить наборы вариантов перечислений из разных единиц.

**Автоматически сгенерированное перечисление**:

```yaoxiang
# 编译器自动生成（用户不感知）
AnimalGroup: Type = {
    Dog(Dog),
    Cat(Cat),
    Bird(Bird),    # ← append(Bird.new()) 触发增量扩展
}

# List(Animal) 内部等价于 List(AnimalGroup)
```

#### 6.3 Проверка соответствия интерфейсу

**Ключевое наблюдение**: проверка соответствия интерфейсу выполняется на этапе компиляции, даже если типы поступают из динамически загружаемых плагинов.

```yaoxiang
# 插件系统
plugin = load_plugin("bird.so")

# 编译器检查：plugin.create_bird() 返回类型必须实现 Animal
bird: Animal = plugin.create_bird()  # 编译期检查

# 放入异构容器 —— append 点触发枚举扩展
animals: List(Animal) = [Dog.new(), Cat.new()]
animals.append(bird)                 # 编译器：(1) 验证 bird 实现了 Animal (2) 扩展枚举
```

**Обработка компилятором**:
1. Проверить тип возвращаемого значения параметра `append`.
2. Убедиться, что этот тип реализует целевой интерфейс.
3. При успехе → расширить перечисление, разрешить размещение.
4. При неудаче → ошибка компиляции.

#### 6.4 Диспетчеризация во время выполнения

**Поток вызова (match по перечислению, сгенерированный компилятором; доказательство реализации стёрто)**:

```
animals[0].speak()
  ↓
match, сгенерированный компилятором:
  match animals[0] {
    AnimalGroup.Dog(d) => d.speak(),
    AnimalGroup.Cat(c) => c.speak(),
    AnimalGroup.Bird(b) => b.speak(),
  }
```

**Сравнение с vtable**:

| | vtable (Rust) | Перечисление на этапе компиляции (YaoXiang) |
|---|---|---|
| Способ поиска | Указатель на vtable → указатель на метод | match по перечислению → прямой вызов |
| Накладные расходы во время выполнения | Одно косвенное обращение | Сравнение строк/ветвление (оптимизируется предсказанием ветвлений CPU) |
| Генерация на этапе компиляции | vtable | Перечисление + match |
| Пользовательская аннотация | Требуется `dyn Trait + 'a` | Не требуется |
| ImplementationProof | Неприменимо | Концепция чисто компиляционного времени, нулевые накладные расходы во время выполнения |

**Преимущества YaoXiang**:
- Не требуется аннотация бренда.
- Типобезопасность на этапе компиляции.
- Прозрачно для пользователя (не нужно писать `dyn Animal`).
- ImplementationProof — чисто компиляционная концепция, нулевые накладные расходы во время выполнения.

#### 6.5 Ограничения и рамки

**Текущий этап (в пределах одной компиляционной единицы):** полная поддержка. Отслеживание владения покрывает все точки `append`/конструирования, перечисление строится инкрементально.

**Между компиляционными единицами:** на этапе компоновки наборы вариантов перечислений из разных единиц объединяются. Механизм общий с мономорфизацией при компоновке (каждая единица генерирует часть перечисления, компоновщик объединяет).

**Не поддерживается:** динамические типы во время выполнения (полная утиная типизация). Набор типов должен быть полностью известен на этапе компиляции.

## Анализ вариантов использования

### Базовая реализация интерфейса

```yaoxiang
# 接口定义
Animal: Type = {
    speak: (Self) -> String,
}

# 类型定义
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",
}

# 使用
dog = Dog.new()
dog.speak()  # "Woof"
```

### Реализация нескольких интерфейсов

```yaoxiang
# 多个接口
Animal: Type = {
    speak: (Self) -> String,
}

Pet: Type = {
    name: (Self) -> String,
}

# 类型实现多个接口
Dog: Type = {
    x: Int = 10,
    Animal,
    Pet,
    speak: (Self) -> String = "Woof",
    name: (Self) -> String = "Buddy",
}

# 使用
dog = Dog.new()
dog.speak()  # "Woof"
dog.name()   # "Buddy"
```

### Дженерик-интерфейсы

```yaoxiang
# 泛型接口
Container: (T: Type) -> Type = {
    add: (self: &mut Self, item: T) -> Void,
    get: (self: &Self, index: Int) -> T,
}

# 实现泛型接口
IntList: Type = {
    data: Array(Int),
    Container(Int),
    add: (self: &mut Self, item: Int) -> Void = ...,
    get: (self: &Self, index: Int) -> Int = ...,
}
```

### Гетерогенный контейнер

```yaoxiang
# 接口定义
Animal: Type = {
    speak: (Self) -> String,
}

# 类型定义
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

Cat: Type = {
    y: Int,
    Animal,
    speak: (Self) -> String = "Meow",
}

# 异构容器
animals: List(Animal) = [Dog.new(), Cat.new()]

# 使用
for animal in animals {
    print(animal.speak())
}
# 输出：
# Woof
# Meow
```

### Система плагинов

```yaoxiang
# 接口定义
Plugin: Type = {
    name: (Self) -> String,
    execute: (Self) -> Void,
}

# 主程序
main: () -> Void = {
    # 加载插件
    plugin1 = load_plugin("plugin1.so")
    plugin2 = load_plugin("plugin2.so")

    # 编译器检查：plugin1 和 plugin2 必须实现 Plugin 接口
    plugins: List(Plugin) = [plugin1, plugin2]

    # 执行所有插件
    for plugin in plugins {
        print(plugin.name())
        plugin.execute()
    }
}
```

## Компромиссы

### Преимущества

1. **Простота**: ключевое слово `impl` не требуется.
2. **Гибкость**: реализация метода возможна и внутри, и снаружи.
3. **Единообразие**: согласованные правила перегрузки.
4. **Удобство**: лаконичный синтаксис значений по умолчанию.
5. **Нулевые накладные расходы**: без vtable, сбор типов на этапе компиляции.
6. **Типобезопасность**: проверка соответствия интерфейсу на этапе компиляции.
7. **Прозрачность для пользователя**: не нужно писать `dyn Animal + 'a`.

### Недостатки

1. **Ограничение**: не поддерживаются динамические типы во время выполнения (полная утиная типизация).
2. **Затраты на этапе компиляции**: для каждого интерфейса необходимо генерировать варианты перечисления и код диспетчеризации `match`.
3. **Набор типов**: должен быть полностью известен на этапе компиляции (в пределах одной компиляционной единицы).

### Меры по смягчению

1. **Система плагинов**: поддерживается за счёт проверки соответствия интерфейсу на этапе компиляции.
2. **Набор типов**: отслеживание владения с инкрементальным построением — сбор в каждой точке `append`/конструирования, а не глобальное сканирование.
3. **Между компиляционными единицами**: объединение наборов вариантов перечислений на этапе компоновки — общий механизм с мономорфизацией при компоновке.

## Альтернативы

| Вариант | Почему не выбран |
|---------|------------------|
| Ключевое слово `impl` | Увеличивает сложность синтаксиса |
| vtable (`dyn Trait`) | Требуется аннотация бренда (`'a`) |
| Полная утиная типизация | Накладные расходы во время выполнения, отсутствие типобезопасности |
| Упаковка в перечисление вручную | Высокая нагрузка на пользователя |

## Связь с RFC-009

**Бренды и реализация интерфейсов**:
- Реализация интерфейсов находится на уровне типа и не затрагивает бренды.
- Бренды относятся к уровню доказательства заимствования (RFC-009a).
- Эти два механизма ортогональны и не влияют друг на друга.

**Динамическая диспетчеризация и бренды**:
- Динамическая диспетчеризация использует доказательство реализации и не требует аннотации бренда.
- Доказательство реализации генерируется на этапе компиляции, поиск во время выполнения отсутствует.
- Устраняется сложность, связанная с `dyn Trait + 'a`.


## Наследование интерфейсов

Интерфейс может включать другой интерфейс. **Новый синтаксис не вводится** — используется то же синтаксическое положение, что и при объявлении интерфейса у типа:

```yaoxiang
Animal: Type = {
    speak: (Self) -> String,
}

Pet: Type = {
    Animal,                       # Pet 继承 Animal — 无新关键字
    name: (Self) -> String,
}

# Dog 实现 Pet 时，必须同时满足 Animal 和 Pet 的所有方法
Dog: Type = {
    x: Int,
    Pet,
    speak: (Self) -> String = "Woof",  # 来自 Animal
    name: (Self) -> String = "Buddy",  # 来自 Pet
}
```

**Принцип проектирования:** наследование существует, но его злоупотребление не поощряется. Предпочтительный способ композиции — множественное объявление интерфейсов (`Dog: Type = { Animal, Pet, ... }`). Тип может напрямую объявить все интерфейсы, которые он удовлетворяет, без построения дерева наследования. Наследование интерфейсов применяется только при наличии явной иерархии «is-a».

**Обработка компилятором:** раскрытие цепочки наследования. `Pet` раскрывается в `{ все методы Animal, name: ... }`. Когда `Dog` объявляет `Pet`, компилятор проверяет, что `Dog` удовлетворяет всем методам как `Animal`, так и `Pet`.

## Реализация метода по умолчанию

Интерфейс может предоставлять реализацию метода по умолчанию. Реализующий тип может её переопределить или унаследовать:

```yaoxiang
fmt: Type = {
    display: (Self) -> String,                      # 必须实现
    debug: (Self) -> String = Self.display(),       # ✅ 引用同接口方法
    summary: (Self) -> String = f"<{Self.name}>",   # ❌ 编译错误：Self.name 不在 fmt 里
}
```

**Ключевое ограничение: интерфейс не может предполагать реализацию вышестоящего типа.** Метод по умолчанию может ссылаться только на методы, объявленные в том же интерфейсе. Поля конкретных типов и методы других интерфейсов для метода по умолчанию недоступны — интерфейс является замкнутым контрактом и не может «заглядывать в карман» реализующего типа. Нарушение этого правила приводит к ошибке **непосредственно при определении интерфейса**.

**Наследование может предполагать реализацию нижестоящего:** когда интерфейс `Pet` наследует `Animal`, методы по умолчанию `Pet` могут использовать методы, объявленные в `Animal` — это гарантировано самим фактом наследования.

```yaoxiang
Animal: Type = {
    speak: (Self) -> String,
}

Pet: Type = {
    Animal,                                              # 继承
    name: (Self) -> String,
    introduce: (Self) -> String = Self.name() + " says " + Self.speak(),  # ✅ speak 来自继承的 Animal
}
```

**Поведение на этапе компиляции:** когда тип реализует интерфейс, для каждого метода:
1. Тип предоставляет метод → используется метод типа.
2. Тип не предоставляет, у интерфейса есть значение по умолчанию → компилятор встраивает реализацию по умолчанию в тип (нулевые накладные расходы на vtable).
3. Тип не предоставляет, у интерфейса нет значения по умолчанию → ошибка компиляции.

**Принцип проектирования:** метод по умолчанию подобен механизму автоматического вывода `Copy`/`Clone` — компилятор при необходимости генерирует реализацию автоматически, пользователь может её переопределить. Ключевые слова `virtual`/`override`/`super` не вводятся.

## Записи проектных решений

| Решение | Выбор | Причина | Дата |
|---------|-------|---------|------|
| Синтаксис объявления интерфейса | Имя интерфейса пишется прямо в теле типа | Устраняет ключевое слово `impl`; объявление интерфейса — естественная часть определения типа | 2026-06-14 |
| Динамическая диспетчеризация | Сбор типов на этапе компиляции + автоматическая генерация перечисления | Без vtable, нулевой поиск во время выполнения, прозрачно для пользователя | 2026-06-14 |
| Внешнее объявление метода | Поддерживается | Гибкость, эквивалентная внутреннему объявлению; компилятор отвечает за сбор между файлами | 2026-06-14 |
| Переопределение | Запрещено (ошибка при совпадении сигнатур) | Переопределение ведёт к непредсказуемому поведению; перегрузка покрывает все случаи | 2026-06-14 |
| Наследование интерфейсов | Поддерживается, без нового синтаксиса | То же синтаксическое положение, что и объявление интерфейса у типа. Поощряется композиция (множественные объявления интерфейсов), не поощряются глубокие деревья наследования | 2026-07-03 |
| Реализация метода по умолчанию | Поддерживается, подобно автоматическому выводу `Copy`/`Clone` | Интерфейс предоставляет тело, компилятор при необходимости встраивает его в реализующий тип; пользователь может переопределить. Ключевые слова `virtual`/`override` не вводятся | 2026-07-03 |
| Ограничения методов по умолчанию | Проверка при определении интерфейса: можно ссылаться только на методы того же интерфейса, нельзя предполагать реализацию вышестоящего | Интерфейс — замкнутый контракт. Наследование может предполагать реализацию подтипов, но интерфейс не может предполагать наличие полей/методов реализующего типа | 2026-07-03 |
| Стратегия сбора типов | Отслеживание владения и инкрементальное построение — сбор в каждой точке `append`/конструирования | Не глобальное сканирование всех реализаторов, а инкрементальное расширение перечисления по точкам операций владения | 2026-07-03 |
| ImplementationProof | Чисто компиляционная концепция, стирается во время выполнения | Во время выполнения используется диспетчеризация через `match` по перечислению; доказательство служит только для проверки на этапе компиляции | 2026-07-03 |
| Между компиляционными единицами | Объединение наборов вариантов перечислений на этапе компоновки | Общий механизм с мономорфизацией при компоновке; каждая единица генерирует часть перечисления, компоновщик объединяет | 2026-07-03 |
