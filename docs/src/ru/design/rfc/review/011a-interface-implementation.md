# RFC-011a: Реализация интерфейсов и динамическая диспетчеризация

> **Родительский RFC**: [RFC-011: Проект системы обобщений (generics)](../accepted/011-generic-type-system.md)
>
> **Настоящий RFC дополняет и заменяет разделы 2.1-2.4 RFC-011, посвящённые ограничениям интерфейса.**

## Краткое содержание

RFC-011 определяет систему обобщений, но не детализирует механизм реализации интерфейсов. Настоящий документ дополняет:

1. **Объявление интерфейса**: имя интерфейса пишется прямо в определении типа, без ключевого слова `impl`
2. **Реализация методов**: поддерживаются как внутренние, так и внешние объявления
3. **Правила перегрузки**: при разных сигнатурах — перегрузка разрешена, при одинаковых — ошибка (перекрытие запрещено)
4. **Значения по умолчанию**: после поля сразу пишется `= value`
5. **Динамическая диспетчеризация**: сбор типов на этапе компиляции + сопоставление интерфейсов, без таблиц виртуальных методов

**Ключевые решения проектирования**:

```yaoxiang
# Определение интерфейса
Animal: Type = {
    speak: (Self) -> String,
}

# Определение типа (внутреннее объявление)
Dog: Type = {
    x: Int = 10,
    Animal,  # объявление интерфейса
    speak: (Self) -> String = "Woof",
}

# Внешнее объявление (перегрузка)
Dog.speak: (Self, volume: Int) -> String = "WOOF"

# Гетерогенный контейнер (динамическая диспетчеризация)
animals: List(Animal) = [Dog.new(), Cat.new()]
animals[0].speak()  # "Woof"
```

**Устранённая сложность**:
- ❌ Нет ключевого слова `impl`
- ❌ Нет аннотации `dyn Trait + 'a`
- ❌ Нет таблиц виртуальных методов (сбор типов на этапе компиляции + перечисление-обёртка)
- ❌ Нет перекрытия (единые правила перегрузки)

## Мотивация

### Недостатки RFC-011

RFC-011 определяет систему обобщений, но не детализирует:

| Проблема | Пояснение |
|------|------|
| Синтаксис объявления интерфейса | Как объявить, что тип реализует интерфейс? |
| Расположение реализации методов | Внутри типа или снаружи? |
| Правила перегрузки | Как обрабатывать одноимённые методы? |
| Синтаксис значений по умолчанию | Как задать значение поля по умолчанию? |
| Динамическая диспетчеризация | Как реализовать гетерогенный контейнер? |

### Цели проектирования

1. **Простота**: ключевое слово `impl` не требуется
2. **Гибкость**: реализация методов поддерживается как внутри, так и снаружи
3. **Единообразие**: согласованные правила перегрузки
4. **Удобство**: лаконичный синтаксис значений по умолчанию
5. **Нулевые накладные расходы**: нет таблиц виртуальных методов, сбор типов на этапе компиляции

### Сравнение с Rust

| Возможность | Rust | YaoXiang |
|------|------|----------|
| Объявление интерфейса | `impl Animal for Dog { ... }` | `Dog: Type = { Animal, ... }` |
| Реализация методов | В блоке `impl` | Внутри или снаружи |
| Перегрузка | Не поддерживается | Поддерживается (при разных сигнатурах) |
| Значения по умолчанию | Требуется `#[default]` | Сразу пишется `= value` |
| Гетерогенный контейнер | `Vec<Box<dyn Animal + 'a>>` | `List(Animal)` |
| Динамическая диспетчеризация | Поиск по таблице виртуальных методов | Сбор типов на этапе компиляции |

## Предложение

### 1. Объявление интерфейса

**Основное правило**: в определении типа имя интерфейса пишется напрямую, без ключевого слова `impl`.

```yaoxiang
# Определение интерфейса
Animal: Type = {
    speak: (Self) -> String,
}

# Объявление реализации интерфейса типом
Dog: Type = {
    x: Int,
    Animal,  # объявление интерфейса
}
```

**Обработка компилятором**:
1. Распознать, что `Animal` — это тип интерфейса
2. Проверить, что у `Dog` есть все методы, требуемые `Animal`
3. Если проверка пройдена → сгенерировать доказательство реализации
4. Если не пройдена → ошибка компиляции

**Синтаксический сахар — эквивалентность**:

```yaoxiang
Dog: Type = {
    x: Int,
    Animal,  # эквивалентно раскрытию методов Animal, но с сохранением метки источника
}

# Эквивалентно (но сохраняет информацию об источнике)
Dog: Type = {
    x: Int,
    speak: (Self) -> String,  # из Animal
}
```

**Зачем нужна метка источника**:
- Прямое раскрытие привело бы к потере информации об источнике
- Метка источника используется для генерации доказательства реализации
- Время выполнения находит правильный метод через доказательство

### 2. Реализация методов

**Основное правило**: реализация методов поддерживается как внутри типа, так и снаружи.

#### 2.1 Внутреннее объявление

```yaoxiang
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",  # реализация метода внутри
}
```

#### 2.2 Внешнее объявление

```yaoxiang
Dog: Type = {
    x: Int,
    Animal,
}

# Реализация метода снаружи
Dog.speak: (Self) -> String = "Woof"
```

#### 2.3 Смешанное объявление

```yaoxiang
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",  # часть методов внутри
}

# Часть методов снаружи
Dog.play: (Self) -> Void = { ... }
```

**Обработка компилятором**:
1. Собрать все определения (внутренние и внешние)
2. Сгруппировать по сигнатуре (перегрузка)
3. Проверить наличие перекрытия (сообщить об ошибке)
4. Проверить полноту интерфейса
5. Сгенерировать доказательство реализации

### 3. Перегрузка и перекрытие

**Основное правило**:
- Разные сигнатуры → перегрузка → разрешена
- Одинаковые сигнатуры → перекрытие → ошибка

#### 3.1 Перегрузка (разрешена)

```yaoxiang
# Типы параметров различаются — перегрузка разрешена
Dog.speak: (Self) -> String = "Woof"
Dog.speak: (Self, volume: Int) -> String = "WOOF"
```

#### 3.2 Перекрытие (запрещено)

```yaoxiang
# Сигнатуры полностью совпадают — перекрытие запрещено
Dog.speak: (Self) -> String = "Woof"
Dog.speak: (Self) -> String = "Bark"  # ❌ ошибка: перекрытие не разрешено
```

**Сообщение об ошибке**:

```
ошибка: повторное определение Dog.speak(Self) -> String
  --> файл2:5:1
  |
5 | Dog.speak: (Self) -> String = "Bark"
  | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ повторное определение
  |
  --> файл1:3:1
  |
3 | Dog.speak: (Self) -> String = "Woof"
  | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ первое определение
```

#### 3.3 Единые правила

**Внутренние и внешние объявления подчиняются одним и тем же правилам перегрузки/перекрытия**:

```yaoxiang
# Внутреннее объявление
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

# Внешнее объявление (перегрузка, разрешена)
Dog.speak: (Self, volume: Int) -> String = "WOOF"

# Внешнее объявление (перекрытие, запрещено)
Dog.speak: (Self) -> String = "Bark"  # ❌ ошибка
```

### 4. Значения по умолчанию

**Основное правило**: после поля сразу пишется `= value`, что избавляет от необходимости в конструкторе.

```yaoxiang
Dog: Type = {
    x: Int = 10,  # значение по умолчанию
    y: Int = 20,  # значение по умолчанию
    Animal,
}
```

**Конструкторы, генерируемые компилятором**:

```yaoxiang
# У всех полей есть значения по умолчанию → генерируется конструктор без аргументов
Dog.new: () -> Dog = { x: 10, y: 20 }

# У части полей есть значения по умолчанию → генерируются частичные конструкторы
Dog.new: (x: Int) -> Dog = { x: x, y: 20 }
Dog.new: (y: Int) -> Dog = { x: 10, y: y }

# Полный конструктор
Dog.new: (x: Int, y: Int) -> Dog = { x: x, y: y }
```

**Внешнее объявление значений по умолчанию**:

```yaoxiang
Dog: Type = {
    x: Int,
    y: Int,
    Animal,
}

# Внешнее объявление значений по умолчанию
Dog.x: Int = 10
Dog.y: Int = 20
```

**Эквивалентно внутреннему объявлению**.

### 5. Реализация в компиляторе

#### 5.1 Дескриптор интерфейса

```rust
// Внутри компилятора: дескриптор интерфейса
struct InterfaceDescriptor {
    name: String,
    methods: Vec<MethodSignature>,
}
```

#### 5.2 Определение типа

```rust
// Внутри компилятора: определение типа
struct TypeDefinition {
    name: String,
    fields: Vec<Field>,
    interface_implementations: Vec<InterfaceImplementation>,
}

// Реализация интерфейса (с сохранением информации об источнике)
struct InterfaceImplementation {
    interface: InterfaceId,
    methods: HashMap<MethodId, FunctionBody>,
}
```

#### 5.3 Доказательство реализации

```rust
// Внутри компилятора: доказательство реализации
struct ImplementationProof {
    type_id: TypeId,
    interface_id: InterfaceId,
    methods: Vec<MethodPointer>,
}
```

#### 5.4 Процесс компиляции

```
1. Разобрать определения типов, собрать объявления интерфейсов
2. Собрать все определения методов (внутренние и внешние)
3. Сгруппировать по сигнатуре (перегрузка)
4. Проверить на перекрытие (сообщить об ошибке)
5. Проверить полноту интерфейса
6. Сгенерировать доказательство реализации
7. В рантайме значение несёт с собой доказательство реализации
```

### 6. Динамическая диспетчеризация

**Ключевое решение**: сбор типов на этапе компиляции + сопоставление интерфейсов, без таблиц виртуальных методов.

#### 6.1 Гетерогенный контейнер

```yaoxiang
# Определение интерфейса
Animal: Type = {
    speak: (Self) -> String,
}

# Определения типов
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

Cat: Type = {
    y: Int,
    Animal,
    speak: (Self) -> String = "Meow",
}

# Гетерогенный контейнер
animals: List(Animal) = [Dog.new(), Cat.new()]
animals[0].speak()  # "Woof"
animals[1].speak()  # "Meow"
```

#### 6.2 Сбор типов на этапе компиляции

**Ключевая стратегия: отслеживание владения с инкрементальным построением.** Вместо того чтобы сканировать на этапе компиляции все типы, реализующие интерфейс, мы инкрементально собираем их в каждой **точке операции владения** для `List(Animal)`:

```yaoxiang
// Точка конструирования
animals: List(Animal) = [Dog.new()]       // AnimalGroup = { Dog(Dog) }

// Точка append
animals.append(Cat.new())                  // компилятор видит Cat в append → расширяет до { Dog, Cat }
animals.append(Bird.new())                 // снова расширяет { Dog, Cat, Bird }
```

**Обработка компилятором (инкрементальная)**:

1. При первом конструировании `List(I)` → сгенерировать начальное перечисление (все известные на данный момент типы конструкторов в текущей единице компиляции)
2. При каждом `append` / `push` / присваивании по индексу → проверить, есть ли тип значения в перечислении; если нет — расширить варианты перечисления
3. Для итогового перечисления сгенерировать мономорфизированный код диспетчеризации через `match`
4. Между единицами компиляции: на этапе линковки объединить наборы вариантов перечисления из разных единиц

**Автоматически сгенерированное перечисление**:

```yaoxiang
# Генерируется компилятором автоматически (пользователь не видит)
AnimalGroup: Type = {
    Dog(Dog),
    Cat(Cat),
    Bird(Bird),    # ← расширено из append(Bird.new())
}

# List(Animal) внутри эквивалентно List(AnimalGroup)
```

#### 6.3 Проверка соответствия интерфейсу

**Ключевое наблюдение**: проверка соответствия интерфейсу выполняется на этапе компиляции, даже если тип приходит из динамически загружаемого плагина.

```yaoxiang
# Система плагинов
plugin = load_plugin("bird.so")

# Проверка компилятора: тип возврата plugin.create_bird() должен реализовывать Animal
bird: Animal = plugin.create_bird()  # проверка на этапе компиляции

# Помещаем в гетерогенный контейнер — точка append запускает расширение перечисления
animals: List(Animal) = [Dog.new(), Cat.new()]
animals.append(bird)                 # компилятор: (1) проверить, что bird реализует Animal (2) расширить перечисление
```

**Обработка компилятором**:
1. Проверить тип возврата параметра `append`
2. Убедиться, что этот тип реализует целевой интерфейс
3. Если проверка пройдена → расширить перечисление, разрешить помещение
4. Если не пройдена → ошибка компиляции

#### 6.4 Диспетчеризация в рантайме

**Поток вызова (на этапе компиляции генерируется `match` по перечислению, ImplementationProof стёрт):**

```
animals[0].speak()
  ↓
сгенерированный компилятором match:
  match animals[0] {
    AnimalGroup.Dog(d) => d.speak(),
    AnimalGroup.Cat(c) => c.speak(),
    AnimalGroup.Bird(b) => b.speak(),
  }
```

**Сравнение с таблицей виртуальных методов**:

| | Таблица виртуальных методов (Rust) | Перечисление на этапе компиляции (YaoXiang) |
|---|---|---|
| Способ поиска | указатель на таблицу → указатель на метод | match по перечислению → прямой вызов |
| Накладные расходы в рантайме | одно косвенное обращение | сравнение/ветвление (оптимизируется предсказателем переходов CPU) |
| Генерация на этапе компиляции | таблица виртуальных методов | перечисление + match |
| Аннотации пользователя | требуется `dyn Trait + 'a` | не требуется |
| ImplementationProof | неприменимо | стёрт на этапе компиляции, в рантайме отсутствует |

**Преимущества YaoXiang**:
- Не нужны аннотации с временем жизни
- Безопасность типов на этапе компиляции
- Прозрачно для пользователя (не нужно писать `dyn Animal`)
- ImplementationProof — исключительно концепция этапа компиляции, нулевые накладные расходы в рантайме

#### 6.5 Ограничения и область применения

**В пределах одной единицы компиляции:** полная поддержка. Отслеживание владения покрывает все точки `append`/конструирования, перечисление строится инкрементально.

**Между единицами компиляции:** на этапе линковки происходит объединение наборов вариантов перечисления из разных единиц. Проект разделяет один и тот же механизм с линковочной мономорфизацией (каждая единица генерирует часть перечисления, линковщик объединяет).

**Не поддерживается:** полностью динамические типы в рантайме (полный утиный тип). Набор типов должен быть полностью известен на этапе компиляции.

## Анализ вариантов использования

### Базовая реализация интерфейса

```yaoxiang
# Определение интерфейса
Animal: Type = {
    speak: (Self) -> String,
}

# Определение типа
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",
}

# Использование
dog = Dog.new()
dog.speak()  # "Woof"
```

### Реализация нескольких интерфейсов

```yaoxiang
# Несколько интерфейсов
Animal: Type = {
    speak: (Self) -> String,
}

Pet: Type = {
    name: (Self) -> String,
}

# Тип реализует несколько интерфейсов
Dog: Type = {
    x: Int = 10,
    Animal,
    Pet,
    speak: (Self) -> String = "Woof",
    name: (Self) -> String = "Buddy",
}

# Использование
dog = Dog.new()
dog.speak()  # "Woof"
dog.name()   # "Buddy"
```

### Обобщённый интерфейс

```yaoxiang
# Обобщённый интерфейс
Container: (T: Type) -> Type = {
    add: (self: &mut Self, item: T) -> Void,
    get: (self: &Self, index: Int) -> T,
}

# Реализация обобщённого интерфейса
IntList: Type = {
    data: Array(Int),
    Container(Int),
    add: (self: &mut Self, item: Int) -> Void = ...,
    get: (self: &Self, index: Int) -> Int = ...,
}
```

### Гетерогенный контейнер

```yaoxiang
# Определение интерфейса
Animal: Type = {
    speak: (Self) -> String,
}

# Определения типов
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

Cat: Type = {
    y: Int,
    Animal,
    speak: (Self) -> String = "Meow",
}

# Гетерогенный контейнер
animals: List(Animal) = [Dog.new(), Cat.new()]

# Использование
for animal in animals {
    print(animal.speak())
}
# Вывод:
# Woof
# Meow
```

### Система плагинов

```yaoxiang
# Определение интерфейса
Plugin: Type = {
    name: (Self) -> String,
    execute: (Self) -> Void,
}

# Основная программа
main: () -> Void = {
    # Загрузка плагинов
    plugin1 = load_plugin("plugin1.so")
    plugin2 = load_plugin("plugin2.so")

    # Проверка компилятора: plugin1 и plugin2 должны реализовывать интерфейс Plugin
    plugins: List(Plugin) = [plugin1, plugin2]

    # Выполнение всех плагинов
    for plugin in plugins {
        print(plugin.name())
        plugin.execute()
    }
}
```

## Компромиссы

### Преимущества

1. **Простота**: ключевое слово `impl` не требуется
2. **Гибкость**: реализация методов поддерживается как внутри, так и снаружи
3. **Единообразие**: согласованные правила перегрузки
4. **Удобство**: лаконичный синтаксис значений по умолчанию
5. **Нулевые накладные расходы**: нет таблиц виртуальных методов, сбор типов на этапе компиляции
6. **Безопасность типов**: проверка соответствия интерфейсу выполняется на этапе компиляции
7. **Прозрачность для пользователя**: не нужно писать `dyn Animal + 'a`

### Недостатки

1. **Ограничение**: не поддерживаются полностью динамические типы в рантайме (полный утиный тип)
2. **Затраты на этапе компиляции**: для каждого интерфейса необходимо сгенерировать варианты перечисления и код диспетчеризации через `match`
3. **Набор типов**: должен быть полностью известен на этапе компиляции (в пределах одной единицы компиляции)

### Меры по смягчению

1. **Система плагинов**: поддерживается через проверку соответствия интерфейсу на этапе компиляции
2. **Набор типов**: отслеживание владения с инкрементальным построением — сбор в каждой точке `append`/конструирования, а не глобальное сканирование
3. **Между единицами компиляции**: объединение наборов вариантов перечисления на этапе линковки, общий механизм с линковочной мономорфизацией

---

## Альтернативные варианты

| Вариант | Почему не выбран |
|------|--------------|
| Ключевое слово `impl` | Увеличивает сложность синтаксиса |
| Таблица виртуальных методов (`dyn Trait`) | Требует аннотации со временем жизни (`'a`) |
| Полный утиный тип | Накладные расходы в рантайме, отсутствие безопасности типов |
| Перечисление-обёртка (ручная) | Высокая нагрузка на пользователя |

---

## Связь с RFC-009

**Бренды и реализация интерфейса**:
- Реализация интерфейса находится на уровне типа, не затрагивает бренды
- Бренды относятся к уровню доказательства заимствования (RFC-009a)
- Эти механизмы ортогональны и не влияют друг на друга

**Динамическая диспетчеризация и бренды**:
- Динамическая диспетчеризация использует доказательство реализации и не требует аннотаций с брендами
- Доказательство реализации генерируется на этапе компиляции, нулевой поиск в рантайме
- Позволяет избежать сложности `dyn Trait + 'a`


## Наследование интерфейсов

Интерфейс может содержать другой интерфейс. **Новый синтаксис не вводится** — используется та же самая синтаксическая позиция, что и для объявления интерфейса типом:

```yaoxiang
Animal: Type = {
    speak: (Self) -> String,
}

Pet: Type = {
    Animal,                       # Pet наследует Animal — без новых ключевых слов
    name: (Self) -> String,
}

# Когда Dog реализует Pet, он должен одновременно удовлетворять всем методам Animal и Pet
Dog: Type = {
    x: Int,
    Pet,
    speak: (Self) -> String = "Woof",  # из Animal
    name: (Self) -> String = "Buddy",  # из Pet
}
```

**Принцип проектирования:** наследование существует, но его чрезмерное использование не поощряется. Основной способ композиции — объявление нескольких интерфейсов (`Dog: Type = { Animal, Pet, ... }`). Тип может напрямую объявить все интерфейсы, которые он удовлетворяет, без необходимости выражать это через дерево наследования. Наследование интерфейсов используется только при наличии явной иерархии «is-a».

**Обработка компилятором:** раскрытие цепочки наследования. `Pet` раскрывается в `{ все методы Animal, name: ... }`. Когда `Dog` объявляет `Pet`, компилятор проверяет, что `Dog` удовлетворяет всем методам `Animal` и `Pet`.

## Реализация методов по умолчанию

Интерфейс может предоставлять реализации методов по умолчанию. Реализующий тип может выбрать: переопределить их или унаследовать:

```yaoxiang
fmt: Type = {
    display: (Self) -> String,                      # должен быть реализован
    debug: (Self) -> String = Self.display(),       # ✅ ссылается на метод того же интерфейса
    summary: (Self) -> String = f"<{Self.name}>",   # ❌ ошибка компиляции: Self.name отсутствует в fmt
}
```

**Ключевое ограничение: интерфейс не может предполагать реализацию вышестоящего уровня.** Метод по умолчанию может ссылаться только на методы, уже объявленные в этом же интерфейсе. Поля конкретного типа или методы других интерфейсов невидимы для метода по умолчанию — интерфейс является замкнутым контрактом и не может залезать в карман реализующего типа. Нарушение этого ограничения вызывает **немедленную ошибку на этапе определения интерфейса**.

**Наследование может предполагать реализацию нижестоящего уровня:** когда интерфейс `Pet` наследует `Animal`, методы по умолчанию `Pet` могут использовать методы, объявленные в `Animal` — поскольку наследование гарантирует их наличие.

```yaoxiang
Animal: Type = {
    speak: (Self) -> String,
}

Pet: Type = {
    Animal,                                              # наследование
    name: (Self) -> String,
    introduce: (Self) -> String = Self.name() + " says " + Self.speak(),  # ✅ speak приходит из унаследованного Animal
}
```

**Поведение на этапе компиляции:** когда тип реализует интерфейс, для каждого метода:
1. Тип предоставил реализацию → используется метод типа
2. Тип не предоставил, интерфейс имеет реализацию по умолчанию → компилятор встраивает реализацию по умолчанию в тип (нулевые накладные расходы, без таблицы виртуальных методов)
3. Тип не предоставил, интерфейс не имеет реализации по умолчанию → ошибка компиляции

**Принцип проектирования:** методы по умолчанию похожи на механизм автоматического вывода `Copy`/`Clone` — компилятор при необходимости генерирует их автоматически, пользователь может переопределить. Ключевые слова `virtual`/`override`/`super` не вводятся.

| Решение | Выбор | Причина | Дата |
|------|------|------|------|
| Синтаксис объявления интерфейса | Имя интерфейса пишется прямо в теле типа | Устранение ключевого слова `impl`, объявление интерфейса — естественная часть определения типа | 2026-06-14 |
| Динамическая диспетчеризация | Сбор типов на этапе компиляции + автогенерация перечисления | Без таблиц виртуальных методов, нулевой поиск в рантайме, прозрачно для пользователя | 2026-06-14 |
| Внешнее объявление методов | Поддерживается | Гибкость, эквивалентная внутреннему объявлению, компилятор отвечает за межфайловое собирание | 2026-06-14 |
| Перекрытие | Запрещено (ошибка при совпадении сигнатур) | Перекрытие ведёт к непредсказуемому поведению, перегрузка покрывает все случаи | 2026-06-14 |
| Наследование интерфейсов | Поддерживается, без нового синтаксиса | Та же синтаксическая позиция, что и при объявлении интерфейса типом. Поощряется композиция (объявление нескольких интерфейсов), глубокие деревья наследования не поощряются | 2026-07-03 |
| Реализация методов по умолчанию | Поддерживается, похоже на автогенерацию `Copy`/`Clone` | Интерфейс предоставляет тело, компилятор встраивает его в реализующий тип при необходимости; пользователь может переопределить. Ключевые слова `virtual`/`override` не вводятся | 2026-07-03 |
| Ограничения методов по умолчанию | Проверяется на этапе определения интерфейса: можно ссылаться только на методы того же интерфейса, нельзя предполагать реализацию вышестоящего уровня | Интерфейс — замкнутый контракт. Наследование может предполагать реализацию нижестоящего уровня, но интерфейс не может предполагать поля/методы реализующего типа | 2026-07-03 |
| Стратегия сбора типов | Отслеживание владения с инкрементальным построением — сбор в каждой точке `append`/конструирования | Не глобальное сканирование всех реализаторов, а инкрементальное расширение перечисления по точкам операций владения | 2026-07-03 |
| ImplementationProof | Чисто компиляционная концепция, стирается в рантайме | В рантайме работает диспетчеризация через `match` по перечислению, доказательство используется только для проверок на этапе компиляции | 2026-07-03 |
| Между единицами компиляции | Объединение наборов вариантов перечисления на этапе линковки | Общий механизм с линковочной мономорфизацией: каждая единица генерирует часть перечисления, линковщик объединяет | 2026-07-03 |
