## Аннотация

В данном документе определяется схема **поддержки квантовых вычислений** и **интеграции с множественными backend-ами** для языка YaoXiang. Ключевая идея: **существующий дизайн YaoXiang (по умолчанию Move, возврат владения, непрозрачные типы, DAG-планировщик, параметры 泛型-констант) естественным образом формирует полноценную основу для квантового языка программирования, не требуя введения какого-либо нового квантового специфического синтаксиса**. Мы добавляем небольшое количество встроенных типов (`Qubit`, `Complex`, `Topology`) и встроенных функций (квантовые вентили, измерения, топологические ограничения), используя существующие механизмы языка для реализации квантовой семантики, автоматического параллелизма для максимизации использования квантовых ресурсов, гибридного классического программирования и поддержки множественных backend-ов.

## Мотивация

### Зачем нужна поддержка квантовых вычислений?

Текущая экосистема квантового программирования имеет серьёзные разрывы:

- **Низкоуровневые языки (QCIS, OpenQASM)**: напрямую управляют физическими квантовыми вентилями, но лишены системы типов и механизмов абстракции, что затрудняет написание сложных алгоритмов.
- **Высокоуровневые фреймворки (Qiskit, Cirq, Q#)**: расширяют классические языки (Python, C#), квантовая семантика реализуется через библиотеки, что приводит к:
  - Правило невозможности клонирования квантового состояния требует ручного соблюдения пользователем (или зависит от линейной системы типов).
  - Операции квантовых вентилей синтаксически отделены от классического кода, высокая кривая обучения.
- **Гибридные вычисления**: квантовая и классическая части требуют явного разделения, отсутствует единая модель потока данных.

### Текущие проблемы

Существующий дизайн YaoXiang恰好提供了解决这些问题的完备基础:

| Потребность квантовых вычислений | Существующий дизайн YaoXiang | Пояснение |
|-------------|-------------------|------|
| Невозможность клонирования квантового состояния | **Семантика Move по умолчанию** | Присваивание передаёт владение, без неявного копирования, естественно соответствует теореме о неклонировании |
| Квантовые вентили как унитарные преобразования | **Возврат владения** | `q = H(q)` потребляет исходный qubit, возвращает новый qubit, точно соответствует семантике вентиля |
| Запутанные состояния | **Непрозрачные типы** | `BellPair` можно использовать только целиком, компилятор отслеживает время жизни, предотвращает неправильное разложение |
| Физические топологические ограничения | **Параметры 泛型-констант** | `Qubit(Topology, N)` проверка смежности на этапе компиляции |
| Коллапс при измерении | **Повторное использование пустого состояния** | После измерения qubit становится пустым, может быть повторно инициализирован, имитируя квантовый коллапс |
| Автоматический параллелизм квантовых схем | **DAG-планировщик** | Операторы внутри функции автоматически распараллеливаются по зависимостям, независимые вентили естественно параллельны |
| Гибридный классическо-квантовый поток управления | **Унифицированный синтаксис** | Квантовые и классические операции используют одинаковую форму `name: type = value` |

**YaoXiang 不是"添加量子支持"，而是发现自己的设计已经量子原生。**

> **Пояснение семантики**: В данном документе используется **семантика владения** YaoXiang для выражения квантовых операций. На уровне компилятора обеспечивается неклонирование (безопасность владения). "Потребление-создание" на уровне языка является **синтаксическим выражением передачи владения** — потребление = получение владения, возврат = передача владения. Базовая реализация может представлять собой реальные обратимые квантовые вентили (изменяющие квантовое состояние на месте), а не "создание нового квантового состояния".

### Цели проектирования

1. **Ноль нового синтаксиса**: не вводить ключевые слова `quantum`, `circuit` и др., все квантовые возможности выражаются через существующие механизмы языка.
2. **Типовая безопасность**: компилятор гарантирует, что квантовое состояние не копируется и не используется нелегально.
3. **Проверка топологических ограничений на этапе компиляции**: через параметры 泛型-констант `Qubit(T, N)` верификация двухкубитных вентилей на соответствие физической топологии.
4. **Прозрачность для множественных backend-ов**: один код компилируется в QIR (универсальная экосистема) или QCIS (отечественный квантовый набор инструкций), переключение через командную строку.
5. **Бесшовное смешение с классикой**: квантовые вычисления могут свободно вызывать классические функции, классический код также может оперировать квантовыми данными (через `ref` для совместного использования, но с ограничениями владения).

## Предложение

### Базовый дизайн

#### 1. Маппинг квантовой системы типов

**Базовые типы**:
```yaoxiang
Qubit: Type0 = primitive_qubit
Complex: Type0 = { re: Float, im: Float }
```
- `Qubit` является типом первого класса, следующим правилам владения (Move, RAII).
- `Complex` используется для представления амплитуд, компилятор может выполнять встраивание и оптимизацию.

**Квантовые вентили как функции**:
```yaoxiang
# Сигнатуры встроенных функций
H: (Qubit) -> Qubit = builtin_hadamard
X: (Qubit) -> Qubit = builtin_pauli_x
Y: (Qubit) -> Qubit = builtin_pauli_y
Z: (Qubit) -> Qubit = builtin_pauli_z
CNOT: (control: Qubit, target: Qubit) -> { Qubit, Qubit } = builtin_cnot
```
- Все вентили потребляют входной qubit, возвращают новый qubit (или пару запутанных кубитов). Синтаксис возврата владения `q = H(q)` напрямую соответствует математической семантике.
- Многокубитные вентили возвращают структуры, результат получается через 模式匹配 или доступ к полям.

**Измерение**:
```yaoxiang
measure: (Qubit) -> Int = builtin_measure   # Потребляет qubit, возвращает классический бит
measure_all: (List(Qubit)) -> List(Int) = builtin_measure_all
```
- После измерения qubit потребляется (становится пустым), пользователь может повторно инициализировать через повторное использование пустого состояния.

**Инициализация**:
```yaoxiang
qubit: (Int) -> Qubit = builtin_qubit   # Инициализация базовым состоянием 0 или 1
```

#### 2. Запутанность и инкапсуляция через непрозрачные типы

Запутанные пары инкапсулируются в непрозрачных типах, предоставляются только комбинированные операции, разложение запрещено:

```yaoxiang
# Встроенный непрозрачный тип
BellPair: Type0 = primitive_bell_pair

# Встроенные функции - только целостные операции
CNOT: (Qubit, Qubit) -> BellPair
measure_bell: (BellPair) -> { Int, Int }
split_bell: (BellPair) -> { Qubit, Qubit }  # Разделение запутанной пары (следует использовать осторожно)
apply_cnot_to_bell: (BellPair, Qubit) -> BellPair
```

**Ключевой дизайн**:
- Нет доступа к полям, только целостные операции через встроенные функции
- `measure_bell(bp)` потребляет всю запутанную пару за раз, возвращает классические биты
- Компилятор может отслеживать полное время жизни запутанной пары

**Сравнение с Python/Qiskit**:
```
Python (Qiskit): Схема строится во время выполнения, ошибки могут быть обнаружены только после отправки
YaoXiang:       Большинство логических ошибок перехватываются на этапе компиляции
```

**Оставшиеся 10%** (такие как физическая декогеренция, ошибки вентилей) являются проблемами оборудования, которые язык не может решить.

#### 3. Физические топологические ограничения

Квантовый чип является ограниченным топологическим графом, **не произвольные два qubit могут выполнять двухкубитный вентиль**, они должны быть смежными. YaoXiang использует **параметры 泛型-констант** для обеспечения топологических ограничений на этапе компиляции.

**Определение типа топологии**:
```yaoxiang
# Топология как тип, содержит матрицу смежности
Topology: Type0 = primitive_topology

# Встроенные топологические константы
Linear8: Topology = topology(8)          # Линейная 8-битная: 0-1-2-3-4-5-6-7
Grid3x3: Topology = topology(3, 3)        # Сетка 3x3
Ring16: Topology = topology(16, ring)    # Кольцевая 16-битная
```

**Qubit привязывается к топологии и позиции**:
```yaoxiang
# Qubit(T, N) - T это тип топологии, N это параметр константы позиции
q0: Qubit(Grid3x3, 0)   # Топология Grid3x3, позиция (0,0)
q1: Qubit(Grid3x3, 1)   # Топология Grid3x3, позиция (0,1)
q2: Qubit(Grid3x3, 2)   # Топология Grid3x3, позиция (0,2)
q3: Qubit(Grid3x3, 3)   # Топология Grid3x3, позиция (1,0)
```

**Автоматическое ограничение операций вентилей**:
```yaoxiang
# Сигнатура типа CNOT с топологическим ограничением
CNOT: (T: Topology, I: Int, J: Int) -> (
    (Qubit(T, I), Qubit(T, J)) -> { Qubit(T, I), Qubit(T, J) }
) when adjacent(T, I, J)

# Проверка на этапе компиляции
CNOT(q0, q1)  # ✅ В Grid3x3 (0,0) и (0,1) смежны
CNOT(q0, q2)  # ❌ Ошибка компиляции: (0,0) и (0,2) не смежны
```

**Ограничение `adjacent` на этапе компиляции**:
- `adjacent` является функцией этапа компиляции, использующей матрицу смежности топологии для статической проверки
- При константных индексах — 100% верификация на этапе компиляции
- При динамических индексах — генерация кода проверки во время выполнения

**Маппинг виртуальных на физические кубиты**:
```yaoxiang
# Физическая позиция неизвестна на этапе компиляции? Используйте вывод типов
q = qubit(Grid3x3)  # Автоматическое распределение позиции 0, последующий вывод
```

#### 4. Владение и линейный поток квантовых состояний

Все квантовые операции следуют семантике Move, гарантируя, что qubit не копируется:
```yaoxiang
q = qubit(0)
q2 = q          # ❌ Ошибка компиляции: q уже перемещён, нельзя использовать повторно
q = H(q)        # ✅ Потребляет q, возвращает новый q
measure(q)      # ✅ Потребляет q, после чего q становится пустым
q = qubit(0)    # ✅ Повторное использование пустого состояния
```

#### 4. Автоматический параллелизм и DAG-планировщик

В Standard или Full Runtime DAG-планировщик автоматически анализирует квантовые программы:
```yaoxiang
apply_two_qubit_gates: () -> {Qubit, Qubit} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    # Вышеуказанные две строки не имеют зависимостей по данным, DAG автоматически выполняет их параллельно
    CNOT(q1, q2)   # Зависит от q1 и q2, автоматическое ожидание
}
```
- Планировщик использует конфигурацию `num_workers` (количество физических квантовых процессоров) для реального параллелизма.
- Пользователю не нужно вручную упорядочивать вентили, достаточно описать поток данных.

#### 5. Гибридные классические вычисления

Классический и квантовый код полностью слиты:
```yaoxiang
grover_search: (target: Int) -> Int = () => {
    n = 4
    qubits = List(Qubit)()
    for i in 0..n {
        qubits.append(H(qubit(0)))
    }
    # Классический цикл смешан с квантовыми операциями
    oracle(qubits, target)   # oracle — последовательность квантовых вентилей
    qubits = diffusion(qubits)
    results = measure_all(qubits)
    return decode_result(results)   # Классическая постобработка
}
```
- В одной функции можно произвольно смешивать квантовые вентили и классический поток управления.
- Система владения гарантирует, что квантовые переменные не будут ошибочно скопированы в классических ветвлениях.

#### 6. Архитектура поддержки множественных backend-ов

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   YaoXiang源码   │     │   Проверка типов │     │   DAG промежуточное│
│  (Унифицированный │────▶│  + Анализ владения│────▶│   представление    │
│   синтаксис)     │     │                  │     │   (Граф потока данных)│
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                          │
                                                          ▼
                          ┌─────────────────────────────────────────────┐
                          │        Backend-ы генерации кода (плагины)    │
                          ├─────────────────┬───────────────────────────┤
                          │  QIR Backend    │  QCIS Backend             │
                          │  (Универсальная │  (Отечественный квантовый │
                          │   экосистема)   │   набор инструкций)       │
                          ├─────────────────┼───────────────────────────┤
                          │  - Вывод .ll    │  - Вывод .qcis текста     │
                          │  - Адаптер QPU  │  - Адаптер CAS/Гуодун     │
                          └─────────────────┴───────────────────────────┘
```

- **Процесс компиляции**: унифицированный frontend → построение DAG → выбор backend → генерация целевого кода.
- **QIR Backend**: маппит узлы DAG на квантовые intrinsic QIR, генерирует LLVM bitcode, может дополнительно использовать оптимизации LLVM.
- **QCIS Backend**: сериализует DAG в QCIS-инструкции (например, `H q0`), поддерживает прямую отправку в консоль управления квантовым чипом.

### Примеры

#### Приготовление и измерение состояния Белла
```yaoxiang
bell_measure: () -> {Int, Int} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    bell = CNOT(q1, q2)  # Возвращает непрозрачный тип BellPair
    result = measure_bell(bell)  # Измерение двух битов за раз
    return result
}
```

#### Квантовая телепортация (упрощённая)
```yaoxiang
teleport: (msg: Qubit, bell: BellPair) -> Qubit = (msg, bell) => {
    # Разделение запутанной пары для получения двух независимых кубитов
    (alice_qubit, bob_qubit) = split_bell(bell)

    # Действия Алисы
    (msg, alice_qubit) = CNOT(msg, alice_qubit)
    msg = H(msg)
    a1 = measure(msg)
    a2 = measure(alice_qubit)

    # Передача классической информации (автоматически обрабатывается планировщиком)
    # Действия Боба
    if a2 == 1 { bob_qubit = X(bob_qubit) }
    if a1 == 1 { bob_qubit = Z(bob_qubit) }
    return bob_qubit
}
```

## Детальной дизайн

### Встроенные типы и определения функций

В модуле `compiler/builtins` добавляются:
```rust
builtins.insert("Qubit", Ty::Primitive(Primitive::Qubit));
builtins.insert("Complex", Ty::Record(vec![
    ("re", Ty::Primitive(Primitive::Float)),
    ("im", Ty::Primitive(Primitive::Float)),
]));

// Квантовые вентили
for (name, sig) in GATES {
    builtins.insert(name, Ty::Function(vec![Ty::Qubit], Ty::Qubit));
}
builtins.insert("CNOT", Ty::Function(
    vec![Ty::Qubit, Ty::Qubit],
    Ty::Record(vec![
        ("q0", Ty::Qubit),
        ("q1", Ty::Qubit)
    ])
));
builtins.insert("measure", Ty::Function(vec![Ty::Qubit], Ty::Primitive(Primitive::Int)));
builtins.insert("qubit", Ty::Function(vec![Ty::Primitive(Primitive::Int)], Ty::Qubit));
```

### Специальная обработка Qubit в проверщике владения

- `Qubit` помечен как `!Copy` (по умолчанию Move), неявное копирование запрещено.
- Параметр функции измерения `measure` имеет тип `Qubit` (передача по значению), потребляет владение.
- В записях, возвращаемых многокубитными вентилями, все поля являются `Qubit`, необходимо соблюдать правила владения.

### Оптимизация квантовых вентилей DAG-планировщиком

- Узлы квантовых вентилей рассматриваются как чистые функции (без побочных эффектов), планировщик может произвольно переупорядочивать независимые вентили.
- При выводе "последовательности квантовых инструкций" планировщик сохраняет зависимости по данным и группирует параллельные вентили (применимо для многопроцессорных квантовых систем).
- Поддержка конфигурации `--target-num-qubits` и `--target-topology` для последующей раскладки и маршрутизации (будущее расширение).

### Детальной маппинг QIR Backend

| Операция YaoXiang | Инструкция QIR |
|---------------|----------|
| `H(q)` | `call void @__quantum__qis__h__body(%Qubit* %q)` |
| `CNOT(q1, q2)` | `call void @__quantum__qis__cnot__body(%Qubit* %q1, %Qubit* %q2)` |
| `measure(q)` | `%result = call i1 @__quantum__qis__mz__body(%Qubit* %q)` |
| `qubit(0)` | `%q = call %Qubit* @__quantum__rt__qubit_allocate()` |

QIR Backend использует LLVM `-O2` для дальнейшей оптимизации и выводит bitcode, совместимый со спецификацией QIR Alliance.

### Детальной маппинг QCIS Backend

| Операция YaoXiang | Инструкция QCIS |
|---------------|-----------|
| `H(q)` (q соответствует физическому биту 2) | `H 2` |
| `CNOT(q1,q2)` (q1→бит 0, q2→бит 1) | `CNOT 0 1` |
| `measure(q)` (бит 0) | `M 0` |
| `qubit(0)` инициализация | Подразумевается в первой используемой инструкции, не требует дополнительной инструкции |

- Необходимо вести таблицу маппинга виртуальных кубитов (переменных YaoXiang) на физические биты.
- Поддержка проверки топологических ограничений (будущая реализация).

### Генерация гибридного классического кода

- Классические части (циклы, условия, целочисленные операции) генерируют нативный код (x86/ARM), взаимодействуют с квантовым backend через FFI или встроенные вызовы.
- В QIR Backend классические части могут быть понижены до LLVM IR и скомпилированы совместно с QIR.

### Влияние на систему типов

- Новые примитивные типы `Qubit` и `Complex`.
- `Qubit` автоматически имеет семантику Move, копирование запрещено.
- Сигнатуры функций квантовых вентилей должны быть зарегистрированы в системе типов.

### Обратная совместимость

- ✅ Полная обратная совместимость
- Новые встроенные типы и функции не влияют на существующий код
- Квантовые возможности опциональны, при отключении нет дополнительных накладных расходов

## Компромиссы

### Преимущества

- **Нет нового синтаксиса**: разработчикам нужно выучить лишь несколько встроенных функций для написания квантовых программ.
- **Типовая безопасность**: система владения автоматически предотвращает копирование кубитов, избегая распространённых ошибок квантового программирования.
- **Автоматический параллелизм**: DAG-планировщик бесплатно предоставляет параллелизм на уровне вентилей без дополнительных оптимизаций компилятора.
- **Совместимость с экосистемой**: QIR Backend позволяет YaoXiang работать на множестве квантовых облачных платформ; QCIS Backend обеспечивает независимость и контроль.
- **Гибридные возможности**: естественное слияние классического и квантового кода подходит для написания сложных квантовых алгоритмов (классическое управление в Shor, Grover).

### Недостатки

- **Статическое количество Qubit**: текущий дизайн предполагает, что количество кубитов известно на этапе компиляции, динамическое распределение через `List(Qubit)`, но кучность `List` может вызвать дополнительные накладные расходы (можно смягчить оптимизацией).
- **Повторное использование после измерения**: повторное использование пустого состояния позволяет повторно инициализировать кубит, но физические квантовые биты могут иметь время релаксации, требуется обработка системой выполнения (в настоящее время ответственность пользователя).
- **Динамическое маппинг топологии**: когда физическая топология становится известна только во время выполнения, проверка на этапе компиляции не работает, необходимо генерировать код проверки во время выполнения (текущая версия поддерживает только статическую проверку).

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| Введение ключевого слова `quantum` и типа `circuit` | Увеличение нового синтаксиса, высокая стоимость обучения, противоречит принципу простого дизайна YaoXiang |
| Реализация квантовой поддержки только как библиотеки | Невозможность использования компилятора для обеспечения безопасности квантовых состояний, невозможность глубокой интеграции с DAG-планировщиком |
| Ожидание созревания квантового оборудования | Упущение критического окна для проектирования квантовых языков программирования |
| Повторное использование существующих квантовых фреймворков (например, Qiskit) | Квантовая семантика через библиотеку, невозможность получить безопасность системы типов и владения |
| Отдельное проектирование квантового подъязыка | Увеличение сложности языка, высокие затраты на поддержку |

## Стратегия реализации

### Фазы

| Фаза | Время | Содержание |
|------|------|------|
| Phase 1 | 1 месяц | Базовые квантовые типы и встроенные функции: добавление типов `Qubit`, `Complex` в компилятор, реализация проверки типов встроенных функций, расширение проверщика владения |
| Phase 2 | 1 месяц | DAG-планировщик распознаёт квантовые вентили: модификация логики построения DAG, пометка квантовых вентилей как чистых функций, реализация группового вывода параллельных вентилей |
| Phase 3 | 2 месяца | Прототип QIR Backend: реализация генератора кода DAG→QIR, интеграция LLVM, подключение симулятора QIR для верификации |
| Phase 4 | 2 месяца | Прототип QCIS Backend: реализация транслятора DAG→QCIS-инструкции, проектирование маппинга виртуальных-физических битов, подключение отечественной квантовой платформы для верификации |
| Phase 5 | 2 месяца | Усиление гибридной классики: обеспечение корректной генерации кода при пересечении классического потока управления и квантовых вентилей, поддержка `List(Qubit)`, добавление примеров программ |
| Phase 6 | 2 месяца | Оптимизация и документация: реализация базовой раскладки и маршрутизации, написание пользовательского руководства и учебника по квантовому программированию, выпуск preview-версии |

### Риски

1. **Доступность квантового оборудования**: зависимость от доступности внешних квантовых симуляторов и реальных QPU.
   - **Смягчение**: приоритетное подключение к открытым симуляторам (QIR runner, Qiskit Aer), реальные QPU как долгосрочная цель.

2. **Сложность реализации backend-ов**: спецификации QIR и QCIS могут изменяться.
   - **Смягчение**: абстракция интерфейса генерации кода, изоляция различий backend-ов для удобства последующей адаптации.

3. **Неопределённость производительности**: характеристики производительности квантовых программ отличаются от классических.
   - **Смягчение**: предоставление инструментов профилирования, чтобы пользователи понимали эффект параллелизма на уровне вентилей.

## Открытые вопросы

- [x] **Топологические ограничения**: реализованы через параметры 泛型-констант `Qubit(Topology, N)` с проверкой на этапе компиляции.
- [ ] **Динамический квантовый регистр**: как `List(Qubit)` маппится в QCIS Backend? Можно генерировать соответствующее количество физических битов, но требуется механизм распределения во время выполнения.
- [ ] **Смягчение ошибок**: следует ли предоставлять встроенные конструкции смягчения ошибок (например, динамическая декогеренция)? Можно сначала реализовать как библиотеку.
- [ ] **Взаимодействие с существующими квантовыми SDK**: можно ли импортировать модули QASM или QIR? В будущем можно рассмотреть FFI.
- [ ] **Автоматическая раскладка и маршрутизация**: когда количество виртуальных кубитов превышает количество физических, как автоматически маппить?
