## Аннотация

В данном документе определяется **квантовая нативная поддержка** языка YaoXiang и схема **интеграции с несколькими бэкендами**. Основная идея: **существующий дизайн YaoXiang (по умолчанию Move, возврат владения, непрозрачные типы, DAG-планировщик, параметры-константы в generics) естественным образом формирует полноценную основу для квантового языка программирования без введения какого-либо нового квантового специфического синтаксиса**. Мы добавляем небольшое количество встроенных типов (`Qubit`, `Complex`, `Topology`) и встроенных функций (квантовые вентили, измерения, топологические ограничения), используем существующие языковые механизмы для реализации квантовой нативной семантики, автоматического параллелизма для максимизации использования квантовых ресурсов, гибридного классического программирования и поддержки нескольких бэкендов.

## Мотивация

### Почему необходима квантовая нативная поддержка?

Текущая экосистема квантового программирования имеет серьёзные разрывы:

- **Низкоуровневые языки (QCIS, OpenQASM)**: напрямую управляют физическими квантовыми вентилями, но не имеют системы типов и механизмов абстракции, что затрудняет написание сложных алгоритмов.
- **Высокоуровневые фреймворки (Qiskit, Cirq, Q#)**: построены на расширении классических языков (Python, C#), квантовая семантика реализуется через библиотеки, что приводит к:
  - Правило о невозможности клонирования квантового состояния требует ручного соблюдения пользователем (или линейная система типов добавляется задним числом).
  - Синтаксис операций квантовых вентилей отделен от классического кода, высокая стоимость обучения.
- **Гибридные вычисления**: квантовая и классическая части требуют явного разделения, отсутствует единая модель потока данных.

### Текущие проблемы

Существующий дизайн YaoXiang предоставляет полноценную основу для решения этих проблем:

| Потребность квантовых вычислений | Существующий дизайн YaoXiang | Пояснение |
|-------------|-------------------|------|
| Невозможность клонирования квантового состояния | **Семантика Move по умолчанию** | Присваивание передаёт владение, без скрытого копирования, естественно соответствует теореме о неклонируемости |
| Квантовые вентили как унитарные преобразования | **Возврат владения** | `q = H(q)` потребляет исходный qubit, возвращает новый qubit, точно соответствует семантике вентиля |
| Состояния запутывания | **Непрозрачные типы** | `BellPair` можно оперировать только как единое целое, компилятор отслеживает время жизни, предотвращает неправильное разложение |
| Физические топологические ограничения | **Параметры-константы в generics** | `Qubit(Topology, N)` проверка соседства на этапе компиляции |
| Коллапс при измерении | **Повторное использование пустого состояния** | После измерения qubit становится пустым, может быть повторно инициализирован, имитируя квантовый коллапс |
| Автоматический параллелизм квантовых цепей | **DAG-планировщик** | Операторы внутри функции автоматически параллельны согласно зависимостям по данным, независимые вентили естественно параллельны |
| Гибридный классическо-квантовый поток управления | **Единый синтаксис** | Квантовые и классические операции используют одинаковую форму `name: type = value` |

**YaoXiang не «добавляет квантовую поддержку», а обнаруживает, что его дизайн уже является квантовым нативным.**

> **Пояснение семантики**: В данном документе используется **семантика владения** YaoXiang для выражения квантовых операций. На уровне компилятора обеспечивается неклонируемость (безопасность владения). На уровне языка «потребление-создание» является **синтаксическим выражением передачи владения** — потребление = получение владения, возврат = передача владения. Базовая реализация может представлять собой реальный обратимый квантовый вентиль (модификация квантового состояния на месте), а не真正的 «создание нового квантового состояния».

### Цели проектирования

1. **Ноль нового синтаксиса**: не вводятся ключевые слова вроде `quantum`, `circuit`; все квантовые возможности выражаются через существующие языковые механизмы.
2. **Безопасность типов**: компилятор гарантирует, что квантовые состояния не копируются и не используются неправомерно.
3. **Проверка топологических ограничений на этапе компиляции**: через параметры-константы generics `Qubit(T, N)` проверяется соответствие операций двухбитовых вентилей физической топологии.
4. **Прозрачная поддержка нескольких бэкендов**: один код может компилироваться в QIR (универсальная экосистема) или QCIS (отечественный набор квантовых инструкций), переключение через параметры командной строки.
5. **Бесшовное гибридное классическое**: квантовые вычисления могут свободно вызывать классические функции, классический код также может оперировать квантовыми данными (через `ref` совместно, но с ограничениями владения).

## Предложение

### Основной дизайн

#### 1. Маппинг квантовой системы типов

**Базовые типы**:
```yaoxiang
Qubit: Type0 = primitive_qubit
Complex: Type0 = { re: Float, im: Float }
```
- `Qubit` — тип первого класса, следует правилам владения (Move, RAII).
- `Complex` используется для представления амплитуды, компилятор может выполнить встраивание и оптимизацию.

**Квантовые вентили как функции**:
```yaoxiang
# Встроенные сигнатуры функций
H: (Qubit) -> Qubit = builtin_hadamard
X: (Qubit) -> Qubit = builtin_pauli_x
Y: (Qubit) -> Qubit = builtin_pauli_y
Z: (Qubit) -> Qubit = builtin_pauli_z
CNOT: (control: Qubit, target: Qubit) -> { Qubit, Qubit } = builtin_cnot
```
- Все вентили потребляют входной qubit и возвращают новый qubit (или пару запутывания). Синтаксис возврата владения `q = H(q)` напрямую соответствует математической семантике.
- Многокубитные вентили возвращают структуры, результаты получаются через сопоставление с образцом или доступ к полям.

**Измерение**:
```yaoxiang
measure: (Qubit) -> Int = builtin_measure   # Потребляет qubit, возвращает классический бит
measure_all: (List(Qubit)) -> List(Int) = builtin_measure_all
```
- После измерения qubit потребляется (становится пустым), пользователь может выполнить повторную инициализацию через повторное использование пустого состояния.

**Инициализация**:
```yaoxiang
qubit: (Int) -> Qubit = builtin_qubit   # Инициализация базового состояния 0 или 1
```

#### 2. Запутывание и инкапсуляция через непрозрачные типы

Пара запутывания инкапсулируется как непрозрачный тип, предоставляются только составные операции, разбор запрещён:

```yaoxiang
# Встроенный непрозрачный тип
BellPair: Type0 = primitive_bell_pair

# Встроенные функции — только целостные операции
CNOT: (Qubit, Qubit) -> BellPair
measure_bell: (BellPair) -> { Int, Int }
split_bell: (BellPair) -> { Qubit, Qubit }  # Разделение пары запутывания (следует использовать осторожно)
apply_cnot_to_bell: (BellPair, Qubit) -> BellPair
```

**Ключевой дизайн**:
- Доступ к полям не предоставляется, допускаются только встроенные функции целостной обработки
- `measure_bell(bp)` потребляет всю пару запутывания за один раз, возвращает классические биты
- Компилятор может отслеживать полный жизненный цикл пары запутывания

**Сравнение с Python/Qiskit**:
```
Python (Qiskit): Построение цепи во время выполнения, ошибки могут быть обнаружены только после отправки
YaoXiang:       Большинство логических ошибок перехватываются на этапе компиляции
```

**Оставшиеся 10%** (такие как физическая декогеренция, погрешность вентилей) — это проблемы оборудования, которые язык не может решить.

#### 3. Физические топологические ограничения

Квантовый чип — это ограниченный топологический граф, **не любые два qubit могут выполнять двухбитовый вентиль**, они должны быть соседними. YaoXiang использует **параметры-константы generics** для обеспечения топологических ограничений на этапе компиляции.

**Определение типа топологии**:
```yaoxiang
# Топология как тип, содержит матрицу смежности
Topology: Type0 = primitive_topology

# Встроенные константы топологии
Linear8: Topology = topology(8)          # Линейные 8 бит: 0-1-2-3-4-5-6-7
Grid3x3: Topology = topology(3, 3)        # Сетка 3x3
Ring16: Topology = topology(16, ring)    # Кольцо из 16 бит
```

**Qubit связывает топологию и позицию**:
```yaoxiang
# Qubit(T, N) - T это тип топологии, N - параметр позиции-константы
q0: Qubit(Grid3x3, 0)   # Топология Grid3x3, позиция (0,0)
q1: Qubit(Grid3x3, 1)   # Топология Grid3x3, позиция (0,1)
q2: Qubit(Grid3x3, 2)   # Топология Grid3x3, позиция (0,2)
q3: Qubit(Grid3x3, 3)   # Топология Grid3x3, позиция (1,0)
```

**Автоматические ограничения операций вентилей**:
```yaoxiang
# Сигнатура типа CNOT с топологическим ограничением
CNOT: (T: Topology, I: Int, J: Int) -> (
    (Qubit(T, I), Qubit(T, J)) -> { Qubit(T, I), Qubit(T, J) }
) when adjacent(T, I, J)

# Проверка на этапе компиляции
CNOT(q0, q1)  # ✅ В Grid3x3 (0,0) и (0,1) соседние
CNOT(q0, q2)  # ❌ Ошибка компиляции: (0,0) и (0,2) не соседние
```

**`adjacent` — ограничение на этапе компиляции**:
- `adjacent` — функция этапа компиляции, использующая матрицу смежности топологии для статической проверки
- При константных индексах — 100% проверка на этапе компиляции
- При динамических индексах — генерация кода проверки во время выполнения

**Виртуальное в физическое отображение**:
```yaoxiang
# Конкретная физическая позиция неизвестна во время компиляции? Использовать вывод типов
q = qubit(Grid3x3)  # Автоматическое распределение позиции 0, последующий вывод
```

#### 4. Владение и линейный поток квантовых состояний

Все квантовые операции следуют семантике Move, обеспечивая невозможность копирования qubit:
```yaoxiang
q = qubit(0)
q2 = q          # ❌ Ошибка компиляции: q уже перемещён, нельзя использовать повторно
q = H(q)        # ✅ Потребляет q, возвращает новый q
measure(q)      # ✅ Потребляет q, после чего q становится пустым
q = qubit(0)    # ✅ Повторное использование пустого состояния
```

#### 4. Автоматический параллелизм и DAG-планировка

В Standard или Full Runtime планировщик DAG автоматически анализирует квантовые программы:
```yaoxiang
apply_two_qubit_gates: () -> {Qubit, Qubit} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    # Вышеуказанные две строки не имеют зависимости по данным, DAG автоматически выполняет параллельно
    CNOT(q1, q2)   # Зависит от q1 и q2, автоматическое ожидание
}
```
- Планировщик использует конфигурацию `num_workers` (количество физических квантовых процессоров) для реального параллельного выполнения.
- Пользователю не нужно вручную упорядочивать вентили, достаточно описать поток данных.

#### 5. Гибридные классические вычисления

Классический и квантовый код полностью интегрированы:
```yaoxiang
grover_search: (target: Int) -> Int = () => {
    n = 4
    qubits = List(Qubit)()
    for i in 0..n {
        qubits.append(H(qubit(0)))
    }
    # Классический цикл смешан с квантовыми операциями
    oracle(qubits, target)   # oracle — это последовательность квантовых вентилей
    qubits = diffusion(qubits)
    results = measure_all(qubits)
    return decode_result(results)   # Классическая постобработка
}
```
- В одной функции можно произвольно смешивать квантовые вентили и классический поток управления.
- Система владения обеспечивает, что квантовые переменные не будут ошибочно скопированы в классических ветвлениях.

#### 6. Архитектура поддержки нескольких бэкендов

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   YaoXiang Source│     │   Type Checking │     │   DAG IR        │
│   (Единый синтаксис)│────▶│   + Ownership Analysis│────▶│   (Поток данных)│
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                          │
                                                          ▼
                          ┌─────────────────────────────────────────────┐
                          │           Генерация кода бэкенды (вставляемые) │
                          ├─────────────────┬───────────────────────────┤
                          │  QIR Бэкенд    │  QCIS Бэкенд             │
                          │  (Универсальная экосистема)│  (Отечественный набор инструкций)│
                          ├─────────────────┼───────────────────────────┤
                          │  - Вывод .ll файла │  - Вывод .qcis текста  │
                          │  - Адаптация под несколько QPU│  - Адаптация под оборудование CAS/国盾│
                          └─────────────────┴───────────────────────────┘
```

- **Процесс компиляции**: унифицированный фронтенд → построение DAG → выбор бэкенда → генерация целевого кода.
- **QIR Бэкенд**: маппинг узлов DAG на квантовые вентили intrinsic QIR, генерация LLVM bitcode с возможностью дальнейшей оптимизации LLVM.
- **QCIS Бэкенд**: сериализация DAG в инструкции QCIS (например, `H q0`), поддержка прямой отправки на консоль квантового чипа.

### Примеры

#### Приготовление и измерение состояния Белла
```yaoxiang
bell_measure: () -> {Int, Int} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    bell = CNOT(q1, q2)  # Возвращает непрозрачный тип BellPair
    result = measure_bell(bell)  # Одновременное измерение двух битов
    return result
}
```

#### Квантовая телепортация (упрощённая)
```yaoxiang
teleport: (msg: Qubit, bell: BellPair) -> Qubit = (msg, bell) => {
    # Разделение пары запутывания для получения двух независимых qubit
    (alice_qubit, bob_qubit) = split_bell(bell)

    # Операции Алисы
    (msg, alice_qubit) = CNOT(msg, alice_qubit)
    msg = H(msg)
    a1 = measure(msg)
    a2 = measure(alice_qubit)

    # Передача классической информации (автоматически обрабатывается планировщиком)
    # Операции Боба
    if a2 == 1 { bob_qubit = X(bob_qubit) }
    if a1 == 1 { bob_qubit = Z(bob_qubit) }
    return bob_qubit
}
```

## Детальное проектирование

### Встроенные типы и определения функций

В модуле `compiler/builtins` добавляется:
```rust
builtins.insert("Qubit", Ty::Primitive(Primitive::Qubit));
builtins.insert("Complex", Ty::Record(vec![
    ("re", Ty::Primitive(Primitive::Float)),
    ("im", Ty::Primitive(Primitive::Float)),
]));

// Квантовые вентили
for (name, sig) in GATES {
    builtins.insert(name, Ty::Function(vec![Ty::Qubit], Ty::Qubit));
}
builtins.insert("CNOT", Ty::Function(
    vec![Ty::Qubit, Ty::Qubit],
    Ty::Record(vec![
        ("q0", Ty::Qubit),
        ("q1", Ty::Qubit)
    ])
));
builtins.insert("measure", Ty::Function(vec![Ty::Qubit], Ty::Primitive(Primitive::Int)));
builtins.insert("qubit", Ty::Function(vec![Ty::Primitive(Primitive::Int)], Ty::Qubit));
```

### Специальная обработка Qubit в проверщике владения

- `Qubit` помечен как `!Copy` (по умолчанию Move), скрытое копирование запрещено.
- Параметр функции измерения `measure` — `Qubit` (передача по значению), потребляет владение.
- В записях, возвращаемых многокубитными вентилями, все поля — `Qubit`, должны соблюдать правила владения.

### Оптимизация квантовых вентилей DAG-планировщиком

- Узлы квантовых вентилей рассматриваются как чистые функции (без побочных эффектов), планировщик может произвольно переупорядочивать независимые вентили.
- При выводе «последовательности квантовых инструкций» планировщик сохраняет зависимости по данным и группирует параллельные вентили (применимо для многопроцессорных квантовых систем).
- Поддержка конфигурации `--target-num-qubits` и `--target-topology` для последующей раскладки и маршрутизации (будущее расширение).

### Детальный маппинг QIR бэкенда

| Операция YaoXiang | Инструкция QIR |
|---------------|----------|
| `H(q)` | `call void @__quantum__qis__h__body(%Qubit* %q)` |
| `CNOT(q1, q2)` | `call void @__quantum__qis__cnot__body(%Qubit* %q1, %Qubit* %q2)` |
| `measure(q)` | `%result = call i1 @__quantum__qis__mz__body(%Qubit* %q)` |
| `qubit(0)` | `%q = call %Qubit* @__quantum__rt__qubit_allocate()` |

QIR бэкенд использует LLVM `-O2` для дальнейшей оптимизации и выводит совместимый с QIR Alliance bitcode.

### Детальный маппинг QCIS бэкенда

| Операция YaoXiang | Инструкция QCIS |
|---------------|-----------|
| `H(q)` (q соответствует физическому биту 2) | `H 2` |
| `CNOT(q1,q2)` (q1→бит 0, q2→бит 1) | `CNOT 0 1` |
| `measure(q)` (бит 0) | `M 0` |
| `qubit(0)` инициализация | Подразумевается в первой использующей инструкции, дополнительная инструкция не нужна |

- Необходимо вести таблицу отображения виртуальных qubit (переменные YaoXiang) в физические биты.
- Поддержка проверки топологических ограничений (будущая реализация).

### Генерация гибридного классического кода

- Классическая часть (циклы, условия, целочисленные операции) обычно генерирует нативный код (x86/ARM), через FFI или встроенные вызовы взаимодействует с квантовым бэкендом.
- В QIR бэкенде классическая часть может быть понижена до LLVM IR и скомпилирована вместе с QIR.

### Влияние на систему типов

- Добавлены примитивные типы `Qubit` и `Complex`.
- `Qubit` автоматически имеет семантику Move, копирование запрещено.
- Сигнатуры функций квантовых вентилей необходимо зарегистрировать в системе типов.

### Обратная совместимость

- ✅ Полная обратная совместимость
- Новые встроенные типы и функции не влияют на существующий код
- Квантовые возможности опциональны, без дополнительных накладных расходов при отключении

## Компромиссы

### Преимущества

- **Нет нового синтаксиса**: разработчикам нужно выучить только небольшое количество встроенных функций для написания квантовых программ.
- **Безопасность типов**: система владения автоматически предотвращает копирование qubit, избегая типичных ошибок квантового программирования.
- **Автоматический параллелизм**: DAG-планировщик бесплатно предоставляет параллелизм на уровне вентилей без дополнительной оптимизации компилятора.
- **Совместимость с экосистемой**: QIR бэкенд позволяет YaoXiang работать на нескольких облачных квантовых платформах; QCIS бэкенд обеспечивает автономность и контроль.
- **Гибридные возможности**: естественное слияние классического и квантового подходит для написания сложных квантовых алгоритмов (классическое управление в Shor, Grover).

### Недостатки

- **Статическое количество Qubit**: текущий дизайн предполагает, что количество qubit известно на этапе компиляции, динамическое распределение через `List(Qubit)`, но кучевое выделение `List` может вносить дополнительные накладные расходы (может быть смягчено оптимизацией).
- **Повторное использование после измерения**: повторное использование пустого состояния позволяет повторно инициализировать qubit, но физические квантовые биты могут иметь время релаксации, требуется обработка системой выполнения (в настоящее время ответственность пользователя).
- **Динамическое топологическое отображение**: когда физическая топология становится известна только во время выполнения, проверка на этапе компиляции не может работать, требуется генерация кода проверки во время выполнения (в текущей версии только статическая проверка).

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| Введение ключевого слова `quantum` и типа `circuit` | Увеличение нового синтаксиса, высокая стоимость обучения, противоречит принципу простоты дизайна YaoXiang |
| Реализация квантовой поддержки только как библиотеки | Невозможно использовать компилятор для гарантии безопасности квантовых состояний, невозможна глубокая интеграция с DAG-планировщиком |
| Ожидание созревания квантового оборудования перед поддержкой | Упускается критическое окно для проектирования языков квантового программирования |
| Повторное использование существующих квантовых фреймворков (например, Qiskit) | Квантовая семантика реализуется через библиотеку, невозможно получить гарантии безопасности системы типов и владения |
| Отдельное проектирование квантового подъязыка | Увеличение сложности языка, высокие затраты на поддержку |

## Стратегия реализации

### Фазы

| Фаза | Время | Содержание |
|------|------|------|
| Фаза 1 | 1 месяц | Базовые квантовые типы и встроенные функции: добавление типов `Qubit`, `Complex` в компилятор, реализация проверки типов встроенных функций, расширение проверщика владения |
| Фаза 2 | 1 месяц | DAG-планировщик распознаёт квантовые вентили: модификация логики построения DAG, пометка квантовых вентилей как чистых функций, реализация группового вывода параллельных вентилей |
| Фаза 3 | 2 месяца | Прототип QIR бэкенда: реализация генератора кода DAG→QIR, интеграция LLVM, подключение QIR симулятора для верификации |
| Фаза 4 | 2 месяца | Прототип QCIS бэкенда: реализация перевода DAG→инструкции QCIS, проектирование отображения виртуальных-физических битов, подключение отечественной квантовой платформы для верификации |
| Фаза 5 | 2 месяца | Усиление гибридного классического: обеспечение корректной генерации кода при пересечении классического потока управления и квантовых вентилей, поддержка `List(Qubit)`, добавление примеров программ |
| Фаза 6 | 2 месяца | Оптимизация и документация: реализация базовой раскладки и маршрутизации, написание руководства пользователя и учебника по квантовому программированию, выпуск превью-версии |

### Риски

1. **Доступность квантового оборудования**: зависимость от доступности внешних квантовых симуляторов и реальных QPU.
   - **Смягчение**: приоритетное подключение к открытым симуляторам (QIR runner, Qiskit Aer), реальные QPU как долгосрочная цель.

2. **Сложность реализации бэкендов**: спецификации QIR и QCIS могут изменяться.
   - **Смягчение**: абстракция интерфейса генерации кода, изоляция различий бэкендов для удобства последующей адаптации.

3. **Неопределённость производительности**: характеристики производительности квантовых программ отличаются от классических.
   - **Смягчение**: предоставление инструментов профилирования, чтобы пользователи понимали эффект параллелизма на уровне вентилей.

## Открытые вопросы

- [x] **Топологические ограничения**: реализованы через параметры-константы generics `Qubit(Topology, N)` с проверкой на этапе компиляции.
- [ ] **Динамические квантовые регистры**: как `List(Qubit)` отображается в QCIS бэкенде? Можно генерировать соответствующее количество физических битов, но требуется механизм распределения во время выполнения.
- [ ] **Смягчение ошибок**: предоставлять ли встроенные конструкции смягчения ошибок (например, динамическую развязку)? Можно сначала реализовать как библиотеку.
- [ ] **Взаимодействие с существующими квантовыми SDK**: можно ли импортировать модули QASM или QIR? В будущем можно рассмотреть FFI.
- [ ] **Автоматическая раскладка и маршрутизация**: когда количество виртуальных qubit превышает количество физических qubit, как автоматически отображать?
