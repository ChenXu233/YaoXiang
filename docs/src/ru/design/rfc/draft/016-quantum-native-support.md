## Аннотация

В данном документе определяется **квантовая нативная поддержка** языка YaoXiang и схема **интеграции с множественными backends**. Ключевая идея: **существующий дизайн YaoXiang (по умолчанию Move, возврат владения, непрозрачные типы, DAG-планировщик, обобщённые константные параметры) естественным образом составляет полную основу для квантового языка программирования, без необходимости введения какого-либо нового квантового специализированного синтаксиса**. Мы добавляем несколько встроенных типов (`Qubit`, `Complex`, `Topology`) и встроенных функций (квантовые вентили, измерения, топологические ограничения), используем существующие механизмы языка для реализации квантовой нативной семантики, автоматического параллелизма для максимизации использования квантовых ресурсов, гибридного классического программирования, а также поддержки множественных backends.

## Мотивация

### Зачем нужна квантовая нативная поддержка?

Текущая экосистема квантового программирования имеет серьёзные разрывы:

- **Низкоуровневые языки (QCIS, OpenQASM)**: напрямую управляют физическими квантовыми вентилями, но лишены системы типов и механизмов абстракции, что затрудняет написание сложных алгоритмов.
- **Высокоуровневые фреймворки (Qiskit, Cirq, Q#)**: построены на расширении классических языков (Python, C#), где квантовая семантика реализована через библиотеки, что приводит к:
  - Правило неклонирования квантового состояния должно соблюдаться пользователем вручную (или с помощью линейной системы типов, добавленной задним числом).
  - Синтаксис операций квантовых вентилей оторван от классического кода, высокий порог вхождения.
- **Гибридные вычисления**: квантовая и классическая части требуют явного разделения, отсутствует единая модель потока данных.

### Текущие проблемы

Существующий дизайн YaoXiang恰好提供了完备的基础来解决 эти проблемы:

| Потребность квантовых вычислений | Имеющийся дизайн YaoXiang | Пояснение |
|-------------|-------------------|------|
| Неклонируемость квантового состояния | **Семантика Move по умолчанию** | Присваивание перемещает владение, нет неявного копирования, естественно соответствует теореме о неклонировании |
| Квантовые вентили как унитарные преобразования | **Возврат владения** | `q = H(q)` потребляет исходный qubit, возвращает новый qubit, точно соответствует семантике вентиля |
| Запутанные состояния | **Непрозрачные типы** | `BellPair` можно оперировать только целиком, компилятор отслеживает жизненный цикл, предотвращая ошибочное расщепление |
| Ограничения физической топологии | **Обобщённые константные параметры** | `Qubit(Topology, N)` — проверка соседства в время компиляции |
| Коллапс при измерении | **Повторное использование пустого состояния** | После измерения qubit становится пустым, может быть повторно инициализирован, имитируя коллапс квантового состояния |
| Автоматический параллелизм квантовых цепей | **DAG-планировщик** | Операторы внутри функции автоматически распараллеливаются по информационным зависимостям, независимые вентили естественно параллельны |
| Гибридный классическо-квантовый управляющий поток | **Унифицированный синтаксис** | Квантовые и классические операции используют одинаковую форму `name: type = value` |

**YaoXiang 不是"添加量子支持"，而是发现自己的设计已经量子原生。**

> **Пояснение семантики**: В данном документе используется **семантика владения** YaoXiang для выражения квантовых операций. На уровне компилятора обеспечивается неклонирование (безопасность владения). «Потребление-создание» на уровне языка является **синтаксическим выражением передачи владения** — потребление = получение владения, возврат = передача владения. Базовая реализация может представлять собой реальный обратимый квантовый вентиль (модификация квантового состояния на месте), а не действительное «создание нового квантового состояния».

### Цели проектирования

1. **Ноль нового синтаксиса**: не вводить ключевые слова `quantum`, `circuit` и др., все квантовые возможности выражаются через существующие механизмы языка.
2. **Типовая безопасность**: компилятор гарантирует, что квантовые состояния не копируются и не используются неправомерно.
3. **Проверка топологических ограничений в время компиляции**: через обобщённые константные параметры `Qubit(T, N)` обеспечивается проверка допустимости двухкубитных вентилей относительно физической топологии.
4. **Прозрачность для множественных backends**: один код для квантовых вычислений может компилироваться в QIR (универсальная экосистема) или QCIS (отечественный квантовый набор команд), переключение через параметры командной строки.
5. **Бесшовная гибридизация классики**: квантовые вычисления могут свободно вызывать классические функции, классический код также может оперировать квантовыми данными (через `ref` для разделяемого доступа, но с ограничениями владения).

## Предложение

### Основной дизайн

#### 1. Маппинг квантовой системы типов

**Базовые типы**:
```yaoxiang
Qubit: Type0 = primitive_qubit
Complex: Type0 = { re: Float, im: Float }
```
- `Qubit` — тип первого класса, следует правилам владения (Move, RAII).
- `Complex` используется для представления амплитуд, компилятор может выполнить встраивание и оптимизацию.

**Квантовые вентили как функции**:
```yaoxiang
# Сигнатуры встроенных функций
H: (Qubit) -> Qubit = builtin_hadamard
X: (Qubit) -> Qubit = builtin_pauli_x
Y: (Qubit) -> Qubit = builtin_pauli_y
Z: (Qubit) -> Qubit = builtin_pauli_z
CNOT: (control: Qubit, target: Qubit) -> { Qubit, Qubit } = builtin_cnot
```
- Все вентили потребляют входной qubit, возвращают новый qubit (или пару запутанных кубитов). Синтаксис возврата владения `q = H(q)` напрямую соответствует математической семантике.
- Многокубитные вентили возвращают структуры, результаты извлекаются через сопоставление с образцом или доступ к полям.

**Измерение**:
```yaoxiang
measure: (Qubit) -> Int = builtin_measure   # Потребляет qubit, возвращает классический бит
measure_all: (List(Qubit)) -> List(Int) = builtin_measure_all
```
- После измерения qubit потребляется (становится пустым), пользователь может выполнить повторную инициализацию через повторное использование пустого состояния.

**Инициализация**:
```yaoxiang
qubit: (Int) -> Qubit = builtin_qubit   # Инициализация базисного состояния 0 или 1
```

#### 2. Запутанность и инкапсуляция через непрозрачные типы

Запутанные пары инкапсулируются в непрозрачный тип, предоставляются только операции объединения, запрещается расщепление:

```yaoxiang
# Встроенный непрозрачный тип
BellPair: Type0 = primitive_bell_pair

# Встроенные функции — только целостные операции
CNOT: (Qubit, Qubit) -> BellPair
measure_bell: (BellPair) -> { Int, Int }
split_bell: (BellPair) -> { Qubit, Qubit }  # Разделение запутанной пары (следует использовать осторожно)
apply_cnot_to_bell: (BellPair, Qubit) -> BellPair
```

**Ключевой дизайн**:
- Нет доступа к полям, только встроенные функции позволяют целостные операции
- `measure_bell(bp)` потребляет всю запутанную пару за раз, возвращает классические биты
- Компилятор может отслеживать полный жизненный цикл запутанной пары

**Сравнение с Python/Qiskit**:
```
Python (Qiskit): Цепь строится во время выполнения, ошибки могут быть обнаружены только после отправки
YaoXiang:       Большинство логических ошибок перехватываются при компиляции
```

**Оставшиеся 10%** (такие как физическая декогеренция, погрешности вентилей) — это проблемы аппаратного обеспечения, которые язык решить не в силах.

#### 3. Ограничения физической топологии

Квантовый чип — это ограниченный топологический граф, **не любые два кубита могут выполнять двухкубитный вентиль**, они должны быть соседними. YaoXiang использует **обобщённые константные параметры** для обеспечения топологических ограничений во время компиляции.

**Определение топологического типа**:
```yaoxiang
# Топология как тип, содержит матрицу смежности
Topology: Type0 = primitive_topology

# Встроенные топологические константы
Linear8: Topology = topology(8)          # Линейный 8-кубитный: 0-1-2-3-4-5-6-7
Grid3x3: Topology = topology(3, 3)        # Сетка 3x3
Ring16: Topology = topology(16, ring)    # Кольцевой 16-кубитный
```

**Qubit связывается с топологией и позицией**:
```yaoxiang
# Qubit(T, N) - T это тип топологии, N - константный позиционный параметр
q0: Qubit(Grid3x3, 0)   # Топология Grid3x3, позиция (0,0)
q1: Qubit(Grid3x3, 1)   # Топология Grid3x3, позиция (0,1)
q2: Qubit(Grid3x3, 2)   # Топология Grid3x3, позиция (0,2)
q3: Qubit(Grid3x3, 3)   # Топология Grid3x3, позиция (1,0)
```

**Автоматические ограничения для вентильных операций**:
```yaoxiang
# Сигнатура типа CNOT с топологическим ограничением
CNOT: (T: Topology, I: Int, J: Int) -> (
    (Qubit(T, I), Qubit(T, J)) -> { Qubit(T, I), Qubit(T, J) }
) when adjacent(T, I, J)

# Проверка во время компиляции
CNOT(q0, q1)  # ✅ В Grid3x3 (0,0) и (0,1) соседние
CNOT(q0, q2)  # ❌ Ошибка компиляции: (0,0) и (0,2) не соседние
```

**`adjacent` — ограничение времени компиляции**:
- `adjacent` — функция времени компиляции, использующая матрицу смежности топологии для статической проверки
- При константных индексах — 100% проверка во время компиляции
- При динамических индексах — генерация кода проверки во время выполнения

**Виртуально-физическое отображение**:
```yaoxiang
# Физическая позиция неизвестна во время компиляции? Используем вывод типов
q = qubit(Grid3x3)  # Автоматическое распределение позиции 0, дальнейший вывод
```

#### 4. Владение и линейный поток квантовых состояний

Все квантовые операции следуют семантике Move, что гарантирует отсутствие копирования кубита:
```yaoxiang
q = qubit(0)
q2 = q          # ❌ Ошибка компиляции: q уже перемещён, нельзя использовать повторно
q = H(q)        # ✅ Потребляет q, возвращает новый q
measure(q)      # ✅ Потребляет q, после чего q становится пустым
q = qubit(0)    # ✅ Повторное использование пустого состояния
```

#### 4. Автоматический параллелизм и DAG-планирование

В Standard или Full Runtime планировщик DAG автоматически анализирует квантовую программу:
```yaoxiang
apply_two_qubit_gates: () -> {Qubit, Qubit} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    # Выше два оператора без информационной зависимости, DAG автоматически распараллеливает
    CNOT(q1, q2)   # Зависит от q1 и q2, автоматическое ожидание
}
```
- Планировщик использует конфигурацию `num_workers` (количество физических квантовых процессоров) для реального параллелизма.
- Пользователю не нужно вручную упорядочивать вентили, достаточно описать поток данных.

#### 5. Гибридные классические вычисления

Классический и квантовый код полностью интегрированы:
```yaoxiang
grover_search: (target: Int) -> Int = () => {
    n = 4
    qubits = List(Qubit)()
    for i in 0..n {
        qubits.append(H(qubit(0)))
    }
    # Классический цикл смешан с квантовыми операциями
    oracle(qubits, target)   # oracle — последовательность квантовых вентилей
    qubits = diffusion(qubits)
    results = measure_all(qubits)
    return decode_result(results)   # Классическая постобработка
}
```
- В одной функции можно произвольно смешивать квантовые вентили и классический управляющий поток.
- Система владения гарантирует, что квантовые переменные не будут ошибочно скопированы в классических ветвлениях.

#### 6. Архитектура поддержки множественных backends

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   YaoXiang源     │     │   类型检查      │     │   DAG 中间表示  │
│   (统一语法)     │────▶│   + 所有权分析   │────▶│   (数据流图)    │
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                          │
                                                          ▼
                          ┌─────────────────────────────────────────────┐
                          │           代码生成后端 (可插拔)              │
                          ├─────────────────┬───────────────────────────┤
                          │  QIR 后端       │  QCIS 后端                │
                          │  (通用生态)      │  (国产量子指令集)         │
                          ├─────────────────┼───────────────────────────┤
                          │  - 输出 .ll 文件 │  - 输出 .qcis 文本        │
                          │  - 适配多种 QPU  │  - 适配中科院/国盾硬件    │
                          └─────────────────┴───────────────────────────┘
```

- **Процесс компиляции**: унифицированный frontend → построение DAG → выбор backend → генерация целевого кода.
- **QIR backend**: отображает узлы DAG в квантовые вентили intrinsic QIR, генерирует LLVM bitcode, может использовать дальнейшую оптимизацию LLVM.
- **QCIS backend**: сериализует DAG в команды QCIS (например, `H q0`), поддерживает прямую отправку на консоль управления квантовым чипом.

### Примеры

#### Приготовление и измерение состояния Белла
```yaoxiang
bell_measure: () -> {Int, Int} = () => {
    q1 = H(qubit(0))
    q2 = H(qubit(0))
    bell = CNOT(q1, q2)  # Возвращает непрозрачный тип BellPair
    result = measure_bell(bell)  # Измеряет два бита за раз
    return result
}
```

#### Квантовая телепортация (упрощённая)
```yaoxiang
teleport: (msg: Qubit, bell: BellPair) -> Qubit = (msg, bell) => {
    # Разделение запутанной пары для получения двух независимых кубитов
    (alice_qubit, bob_qubit) = split_bell(bell)

    # Операции Алисы
    (msg, alice_qubit) = CNOT(msg, alice_qubit)
    msg = H(msg)
    a1 = measure(msg)
    a2 = measure(alice_qubit)

    # Передача классической информации (автоматически обрабатывается планировщиком)
    # Операции Боба
    if a2 == 1 { bob_qubit = X(bob_qubit) }
    if a1 == 1 { bob_qubit = Z(bob_qubit) }
    return bob_qubit
}
```

## Детальное проектирование

### Встроенные типы и определения функций

В модуле `compiler/builtins` добавляются:
```rust
builtins.insert("Qubit", Ty::Primitive(Primitive::Qubit));
builtins.insert("Complex", Ty::Record(vec![
    ("re", Ty::Primitive(Primitive::Float)),
    ("im", Ty::Primitive(Primitive::Float)),
]));

// Квантовые вентили
for (name, sig) in GATES {
    builtins.insert(name, Ty::Function(vec![Ty::Qubit], Ty::Qubit));
}
builtins.insert("CNOT", Ty::Function(
    vec![Ty::Qubit, Ty::Qubit],
    Ty::Record(vec![
        ("q0", Ty::Qubit),
        ("q1", Ty::Qubit)
    ])
));
builtins.insert("measure", Ty::Function(vec![Ty::Qubit], Ty::Primitive(Primitive::Int)));
builtins.insert("qubit", Ty::Function(vec![Ty::Primitive(Primitive::Int)], Ty::Qubit));
```

### Специальная обработка Qubit в проверке владения

- `Qubit` помечен как `!Copy` (по умолчанию Move), неявное копирование запрещено.
- Параметр функции измерения `measure` имеет тип `Qubit` (передача по значению), потребляется владение.
- В записях, возвращаемых многокубитными вентилями, поля имеют тип `Qubit`, по-прежнему требуется соблюдение правил владения.

### Оптимизация квантовых вентилей DAG-планировщиком

- Узлы квантовых вентилей рассматриваются как чистые функции (без побочных эффектов), планировщик может произвольно переупорядочивать независимые вентили.
- При выводе «последовательности квантовых команд» планировщик сохраняет информационные зависимости и группирует параллельные вентили (применимо для многопроцессорных квантовых систем).
- Поддержка настройки `--target-num-qubits` и `--target-topology` для последующего размещения и маршрутизации (будущее расширение).

### Детальное отображение QIR backend

| Операция YaoXiang | Команда QIR |
|---------------|----------|
| `H(q)` | `call void @__quantum__qis__h__body(%Qubit* %q)` |
| `CNOT(q1, q2)` | `call void @__quantum__qis__cnot__body(%Qubit* %q1, %Qubit* %q2)` |
| `measure(q)` | `%result = call i1 @__quantum__qis__mz__body(%Qubit* %q)` |
| `qubit(0)` | `%q = call %Qubit* @__quantum__rt__qubit_allocate()` |

QIR backend использует LLVM `-O2` для дальнейшей оптимизации и выводит bitcode, совместимый с QIR Alliance.

### Детальное отображение QCIS backend

| Операция YaoXiang | Команда QCIS |
|---------------|-----------|
| `H(q)` (q соответствует физическому биту 2) | `H 2` |
| `CNOT(q1,q2)` (q1→бит 0, q2→бит 1) | `CNOT 0 1` |
| `measure(q)` (бит 0) | `M 0` |
| `qubit(0)` инициализация | Подразумевается в первой команде использования, дополнительные команды не нужны |

- Необходимо вести таблицу отображения виртуальных кубитов (переменных YaoXiang) на физические биты.
- Поддержка проверки топологических ограничений (будущая реализация).

### Генерация гибридного классического кода

- Классическая часть (циклы, условия, целочисленные операции) генерирует нативный код (x86/ARM) штатно, взаимодействие с квантовым backend через FFI или встроенные вызовы.
- В QIR backend классическая часть может быть понижена до LLVM IR, совместная компиляция с QIR.

### Влияние на систему типов

- Добавлены примитивные типы `Qubit` и `Complex`.
- `Qubit` автоматически имеет семантику Move, копирование запрещено.
- Сигнатуры функций квантовых вентилей должны быть зарегистрированы в системе типов.

### Обратная совместимость

- ✅ Полная обратная совместимость
- Новые встроенные типы и функции не влияют на существующий код
- Квантовые возможности являются опциональными, без дополнительных накладных расходов при отключении

## Компромиссы

### Преимущества

- **Нет нового синтаксиса**: разработчикам достаточно изучить несколько встроенных функций для написания квантовых программ.
- **Типовая безопасность**: система владения автоматически предотвращает копирование кубитов, избегая распространённых ошибок квантового программирования.
- **Автоматический параллелизм**: DAG-планировщик бесплатно предоставляет параллелизм на уровне вентилей без дополнительных оптимизаций компилятора.
- **Совместимость с экосистемой**: QIR backend позволяет YaoXiang работать на множестве квантовых облачных платформ; QCIS backend обеспечивает технологический суверенитет.
- **Гибридные возможности**: естественное слияние классического и квантового кода подходит для написания сложных квантовых алгоритмов (классическое управление в алгоритмах Шора, Гровера и др.).

### Недостатки

- **Статическое количество кубитов**: текущий дизайн предполагает, что количество кубитов известно во время компиляции, динамическое распределение возможно через `List(Qubit)`, но кучное распределение `List` может вносить дополнительные накладные расходы (может быть смягчено оптимизацией).
- **Повторное использование после измерения**: повторное использование пустого состояния позволяет повторно инициализировать кубит, но физические кубиты могут иметь время релаксации, требуется обработка в runtime (в настоящее время на ответственности пользователя).
- **Динамическое топологическое отображение**: когда физическая топология становится известна только во время выполнения, проверка во время компиляции не работает, требуется генерация кода проверки во время выполнения (текущая версия поддерживает только статическую проверку).

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| Введение ключевого слова `quantum` и типа `circuit` | Добавление нового синтаксиса, высокий порог вхождения, противоречит принципу простоты дизайна YaoXiang |
| Реализация квантовой поддержки только через библиотеку | Невозможно использовать гарантии компилятора для безопасности квантовых состояний, невозможна глубокая интеграция с DAG-планировщиком |
| Ожидание зрелости квантового оборудования перед поддержкой | Пропуск ключевого окна для проектирования языков квантового программирования |
| Повторное использование существующих квантовых фреймворков (например, Qiskit) | Квантовая семантика реализуется через библиотеку, невозможно получить гарантии системы типов и владения |
| Разработка отдельного квантового подъязыка | Увеличивает сложность языка, высокие затраты на поддержку |

## Стратегия реализации

### Фазы

| Фаза | Время | Содержание |
|------|------|------|
| Фаза 1 | 1 месяц | Базовые квантовые типы и встроенные функции: добавление типов `Qubit`, `Complex` в компилятор, реализация проверки типов встроенных функций, расширение проверки владения |
| Фаза 2 | 1 месяц | DAG-планировщик распознаёт квантовые вентили: модификация логики построения DAG, маркировка квантовых вентилей как чистых функций, реализация группового вывода параллельных вентилей |
| Фаза 3 | 2 месяца | Прототип QIR backend: реализация генератора кода из DAG в QIR, интеграция LLVM, подключение QIR-симулятора для верификации |
| Фаза 4 | 2 месяца | Прототип QCIS backend: реализация транслятора из DAG в команды QCIS, проектирование таблицы отображения виртуальных-физических битов, подключение отечественной квантовой платформы для верификации |
| Фаза 5 | 2 месяца | Усиление гибридной классики: обеспечение корректной кодогенерации при пересечении классического управляющего потока и квантовых вентилей, поддержка `List(Qubit)`, добавление примеров программ |
| Фаза 6 | 2 месяца | Оптимизация и документация: реализация базового размещения и маршрутизации, написание руководства пользователя и учебника по квантовому программированию, выпуск preview-версии |

### Риски

1. **Доступность квантового оборудования**: зависимость от доступности внешних квантовых симуляторов и реальных QPU.
   - **Смягчение**: приоритет — подключение к открытым симуляторам (QIR runner, Qiskit Aer), реальные QPU — долгосрочная цель.

2. **Сложность реализации backend**: спецификации QIR и QCIS могут изменяться.
   - **Смягчение**: абстракция интерфейса кодогенерации, изоляция различий backends для удобства последующей адаптации.

3. **Неопределённость производительности**: характеристики производительности квантовых программ отличаются от классических.
   - **Смягчение**: предоставление инструментов профилирования, чтобы пользователи понимали эффект параллелизма на уровне вентилей.

## Открытые вопросы

- [x] **Топологические ограничения**: реализованы через обобщённые константные параметры `Qubit(Topology, N)`, проверка во время компиляции.
- [ ] **Динамические квантовые регистры**: как `List(Qubit)` отображается в QCIS backend? Можно генерировать соответствующее количество физических битов, но требуется механизм распределения во время выполнения.
- [ ] **Смягчение ошибок**: следует ли предоставлять встроенные конструкции смягчения ошибок (например, динамическая декогеренция)? Можно начать с реализации в виде библиотеки.
- [ ] **Взаимодействие с существующими квантовыми SDK**: можно ли импортировать модули QASM или QIR? В будущем можно рассмотреть FFI.
- [ ] **Автоматическое размещение и маршрутизация**: когда количество виртуальных кубитов превышает количество физических, как выполнить автоматическое отображение?
