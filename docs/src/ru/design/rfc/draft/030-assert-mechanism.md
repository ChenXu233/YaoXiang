# RFC-030: механизм assert-утверждений

## Резюме

Ввести в YaoXiang механизм утверждений `assert` для тестирования и проверки предусловий. Настоящий RFC предлагает два варианта реализации — **native-функция** и **ключевое слово** — с анализом компромиссов каждого подхода для принятия решения.

## Мотивация

### Зачем нужна эта возможность?

В текущей версии YaoXiang сквозные (E2E) тесты могут имитировать утверждения только через связку `if` + `io.println` + `return`:

```yaoxiang
val = some_func()
if val != 42 {
    io.println("FAIL: expected 42")
    return
}
```

Такой способ записи вызывает три проблемы:

1. **Много шаблонного кода**: каждое утверждение занимает 4 строки, файлы тестов раздуваются.
2. **Слабые сообщения об ошибках**: ручная конкатенация строк, отсутствует позиция в исходном коде.
3. **Невозможность композиции**: нельзя пакетно зарегистрировать утверждения, нельзя передать их как аргумент в тестовый фреймворк.

### Текущие проблемы

- Отсутствует единый механизм утверждений
- Тестовый код наводнён шаблоном `if` + печать + `return`
- На уровне байткода уже есть инструкция `Throw`, но на уровне языка она не доступна

## Вариант A: native-функция

Реализовать `assert` как native-функцию без введения нового ключевого слова.

```yaoxiang
use std.assert

main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Сигнатура типа

```
assert: (Bool, String) -> ()
```

### Поведение во время выполнения

1. Вычислить первый аргумент `condition: Bool`
2. Если `true`, вернуть `Unit`
3. Если `false`, вызвать runtime panic:
   - Вывести содержимое `message`
   - Вывести стек вызовов (в debug-режиме)
   - Завершить текущее выполнение

### Изменения в компиляторе

**Не требуется менять parser, AST, typecheck, IR gen.**

Достаточно добавить регистрацию native-функции в `src/std/`:

1. Добавить `src/std/assert.rs`
2. Зарегистрировать native-функцию `std.assert`
3. Внутри вызвать уже существующий механизм runtime panic

### Достоинства

- **Нулевые изменения синтаксиса**: не нужно править parser, AST, IR, codegen
- **Простота реализации**: около 20 строк кода регистрации native-функции
- **Объект первого класса**: можно передавать как аргумент, присваивать переменной, использовать в функциях высшего порядка
- **Постепенное усиление**: в будущем можно добавить оптимизацию инлайнинга на этапе компиляции, не ломая код

### Недостатки

- **Нет анализа на этапе компиляции**: компилятор не может выполнить устранение мёртвого кода после `assert(false)`
- **Ограниченные сообщения об ошибках**: позиция в исходном коде извлекается из стека вызовов в runtime, менее точно, чем у ключевого слова
- **Возможность затенения**: пользователь может определить одноимённую функцию, перекрывающую `assert`
- **Требуется import**: необходимо писать `use std.assert` для использования

## Вариант B: ключевое слово

Ввести `assert` как ключевое слово языка, компилятор выполняет специальную обработку на уровне IR.

```yaoxiang
main = {
    assert(1 + 1 == 2, "math is broken")
    assert(get_name() == "YaoXiang", "name mismatch")
}
```

### Синтаксис

```
assert '(' Expr ',' StringLit ')'
```

`assert` является ключевым словом, `use`-импорт не требуется.

### Изменения в компиляторе

Требуется сквозная правка:

| Этап | Изменения |
|------|-----------|
| Lexer | Добавить токен `KwAssert` |
| Parser | Добавить синтаксис `assert(expr, "msg")` |
| AST | Добавить `StmtKind::Assert { condition, message }` |
| IR Gen | Скомпилировать Assert в условный переход + Throw |
| Codegen | Сгенерировать последовательность инструкций `JmpIfNot` + `Throw` |

### Достоинства

- **Доступно из коробки**: не требует import, можно использовать в любом месте
- **Точная позиция в исходном коде**: компилятор знает позицию assert на уровне AST, сообщения об ошибках точные до строки и столбца
- **Невозможно затенить**: ключевое слово нельзя перекрыть пользовательским кодом
- **Расширяемость в будущем**: возможен анализ на этапе компиляции (в связке с проходом свёртки констант)
- **Однозначная семантика**: `assert` имеет чёткий смысл на уровне синтаксиса и не может быть использован неправильно

### Недостатки

- **Высокая стоимость реализации**: требуется правка parser, AST, IR, codegen по всей цепочке
- **Не является объектом первого класса**: нельзя передать как аргумент, нельзя присвоить переменной
- **Раздувание ключевых слов**: каждое новое ключевое слово увеличивает сложность языка
- **Слабая расширяемость**: добавление новых вариантов сигнатуры (например, `assert_eq`) требует изменения синтаксиса

## Сравнение

| Аспект | Вариант A: функция | Вариант B: ключевое слово |
|--------|--------------------|---------------------------|
| Стоимость реализации | Низкая (~20 строк) | Высокая (~200 строк, 4 модуля) |
| Изменения синтаксиса | Нет | Да (новое ключевое слово + синтаксис) |
| Требуется import | Да | Нет |
| Объект первого класса | Да | Нет |
| Точность позиции в исходном коде | Стек вызовов в runtime | Точно до строки и столбца на этапе компиляции |
| Невозможно затенить | Нет | Да |
| Анализ на этапе компиляции | Невозможен (нужна свёртка констант) | Возможен (в связке со свёрткой констант) |
| Расширяемость | Высокая (перегрузка, функции высшего порядка) | Низкая (нужно менять синтаксис) |

### Реалистичные ограничения анализа на этапе компиляции

Ключевое преимущество варианта B — анализ на этапе компиляции — требует наличия **прохода свёртки констант (constant folding)**. То есть компилятор должен вычислить `false` в `assert(false)` на этапе компиляции, чтобы распознать это как мёртвый код.

В YaoXiang в данный момент проход свёртки констант отсутствует. Даже при выборе варианта B такие распространённые формы записи, как `assert(x > 0)`, по-прежнему не могут быть проанализированы на этапе компиляции. Анализу поддаются только литералы вида `assert(true)` / `assert(false)`.

Таким образом, преимущество варианта B в плане компиляции **на текущем этапе является теоретическим, а не практическим**.
