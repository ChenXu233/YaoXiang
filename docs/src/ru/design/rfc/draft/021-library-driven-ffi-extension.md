## Аннотация

В данном документе предлагается схема расширения FFI (внешний интерфейс функций), управляемая **библиотеками**. Единственной точкой входа в FFI является объявление `native("symbol")` + таблица регистрации `FfiRegistry` во время выполнения, при этом в ядре не вводится второй набор механизмов. Поверх этого стандартная библиотека обеспечивает возможность динамической загрузки библиотек, межъязыковые вызовы привязки и др. Привязки вызовов для конкретных языков (таких как C, Python, JavaScript) автоматически генерируются официальным инструментарием или разрабатываются проектами по мере необходимости.

## Мотивация

### Недостатки текущей реализации

Текущая реализация FFI уже обладает следующими возможностями:
- Синтаксис объявления `native("symbol")` для внешних функций
- Таблица регистрации функций `FfiRegistry`

Однако функциональность относительно ограничена:
- Отсутствует поддержка динамической загрузки библиотек
- Нет инфраструктуры для межъязыковых вызовов
- Отсутствуют инструменты автоматической генерации привязок

### Философия дизайна

YaoXiang следует принципу **«Ядро просто, сложность опускается в библиотеки»**:

> **Хороший вкус (Good Taste)**: Назначение языка — предоставлять атомарные возможности, а не универсальный функциональный набор. Сложность должна решаться через библиотеки, а не накапливаться в компиляторе.

Таким образом, данная схема:
- ✅ **Нулевые изменения синтаксиса** — полная обратная совместимость, единственной точкой входа в FFI является `native("symbol")`
- ✅ **Библиотека как язык** — функциональность расширяется через стандартную библиотеку
- ✅ **Автоматизация инструментарием** — привязки автоматически генерируются `yx-bindgen`, а не поддерживаются вручную
- ✅ **Постепенное улучшение** — разработчики добавляют функциональность по необходимости

## Предложение

### 1. Расширение основной библиотеки FFI

Расширение модуля `std.ffi`. Примечание: все вызовы внешних функций по-прежнему осуществляются через объявление `native("symbol")`, `std.ffi` только обеспечивает вспомогательные возможности.

#### 1.1 Динамическая загрузка библиотек

```yaoxiang
import ffi

# Загрузка динамической библиотеки (.so/.dll/.dylib)
lib = ffi.load_library("./libmyext.so")

# Получение символа функции из библиотеки, возврат имени symbol для native
ffi.register_library_symbols(lib, [
    "my_function",
    "another_func",
])
```

`load_library` возвращает дескриптор `DynamicLibrary`, а `register_library_symbols` регистрирует имена символов в известной таблице `FfiRegistry`. После этого пользователь по-прежнему использует объявление `native`:

```yaoxiang
my_func: (a: Int, b: Int) -> Int = native("my_function")
```

Нет второго набора синтаксиса вызова, нет обёрток `try_call`.

#### 1.2 Управление библиотеками

```yaoxiang
# Список загруженных библиотек
loaded = ffi.loaded_libraries()

# Выгрузка библиотеки
ffi.unload_library(lib)

# Проверка версии библиотеки
ffi.check_version(lib, "1.0.0")
```

#### 1.3 Разрешение символов

```yaoxiang
# Поиск символа по имени (возврат структуры Symbol)
sin_sym = ffi.dlsym("libm.so", "sin")
```

Соглашения о межъязыковых вызовах и преобразование типов не обрабатываются во время выполнения через универсальные обёртки, а генерируются `yx-bindgen` во время компиляции.

### 2. Реализация динамической загрузки библиотек

#### 2.1 Основные структуры данных

```rust
pub struct DynamicLibrary {
    handle: *mut std::ffi::c_void,
    path: String,
}

impl DynamicLibrary {
    pub fn load(path: &str) -> Result<Self, FfiError>;
    pub fn get_symbol(&self, name: &str) -> Result<*mut std::ffi::c_void, FfiError>;
    pub fn unload(self) -> Result<(), FfiError>;
}
```

#### 2.2 Типы ошибок

```rust
pub enum FfiError {
    LibraryNotFound { name: String, os_error: Option<OsError> },
    SymbolNotFound { name: String, os_error: Option<OsError> },
    CallFailed { message: String, os_error: Option<OsError> },
    Timeout,
}

pub struct OsError {
    pub code: i32,
    pub message: String,
}
```

`OsError` содержит нативные коды ошибок платформы (`dlerror()` в Linux, `GetLastError()` в Windows), обеспечивая отлаживаемость.

### 3. Мультиязыковые привязки: схема инструментария

Отказ от «фантазий о поддержке привязок сообществом для всех языков». Переход к автоматической генерации официальным инструментарием.

#### 3.1 Архитектурный дизайн

```
┌───────────────────────────────────────────────┐
│  YaoXiang код                                 │
│                                               │
│  // Пользователь пишет только объявление native│
│  my_func: (a: Int) -> Int = native("my_func") │
└───────────────────────────────────────────────┘
         ↑                          ↑
         |  Время компиляции        | Время выполнения
┌──────────────────┐   ┌────────────────────────┐
│  yx-bindgen       │   │  std.ffi + FfiRegistry  │
│  (C header → .yx) │   │  - dlopen/dlsym         │
│                   │   │  - LoadLibrary/GetProc  │
└──────────────────┘   └────────────────────────┘
```

#### 3.2 Генератор привязок (`yx-bindgen`)

`yx-bindgen` — это автономный инструмент CLI, который генерирует код привязок FFI YaoXiang из заголовочных файлов C:

```bash
yx-bindgen --header /usr/include/sqlite3.h --output sqlite3.yx
```

Пример сгенерированного результата:

```yaoxiang
# Автоматически сгенерировано, не редактировать вручную
# Source: /usr/include/sqlite3.h

sqlite3_open: (filename: *const u8, ppDb: *mut *mut opaque) -> Int
    = native("sqlite3_open")

sqlite3_close: (db: *mut opaque) -> Int
    = native("sqlite3_close")

sqlite3_exec: (
    db: *mut opaque,
    sql: *const u8,
    callback: *mut opaque,
    arg: *mut opaque,
    errmsg: *mut *mut u8,
) -> Int
    = native("sqlite3_exec")
```

`yx-bindgen` поддерживается официально, что гарантирует:
- Полноту маппинга типов (`int` → `Int`, `char*` → `*const u8`, `void*` → `*mut opaque`)
- Выравнивание layout структур (автоматическая эквивалентность `#[repr(C)]`)
- Преобразование сигнатур callback

#### 3.3 Официально поддерживаемые пакеты привязок

Команда ядра YaoXiang не обязуется поддерживать универсальные пакеты привязок для всех языков, но предоставляет официальный пакет привязок `libc` (подмножество POSIX + Windows API) в качестве примера лучших практик FFI и базовых возможностей.

Привязки для других языков и библиотек:
- Генерируются самостоятельно с помощью `yx-bindgen`
- Могут публиковаться как пакеты YaoXiang (например, `libsqlite3`, `libcurl`, `libsdl2`)
- Команда ядра не несёт ответственности за поддержку, но предоставляет механизм публикации и управления версиями пакетов

### 4. Уровень преобразования типов

#### 4.1 Компиляционный маппинг типов

Преобразование типов осуществляется не через обёртки времени выполнения, а статически определяется при генерации `yx-bindgen`:

| Тип C | Тип YaoXiang | Способ преобразования |
|--------|---------------|----------|
| `int` | `Int` | Прямая передача значения |
| `char*` | `*const u8` | Передача указателя |
| `void*` | `*mut opaque` | Непрозрачный указатель |
| `struct T` | `extern struct T` | Соответствие layout памяти |
| `int*` | `*mut Int` | Передача указателя (изменяемый) |
| `const int*` | `*const Int` | Передача указателя (только чтение) |

#### 4.2 Ручное преобразование (вспомогательные средства стандартной библиотеки)

```yaoxiang
# Явное преобразование
raw_ptr = ffi.to_pointer(my_bytes)
c_string = ffi.to_c_string(my_string)
```

### 5. Модель владения памятью

#### 5.1 Основные принципы

При каждом выделении памяти через границу FFI необходимо чётко ответить на два вопроса:
1. **Кто выделяет?** (C-сторона `malloc` / YaoXiang сторона времени выполнения)
2. **Кто освобождает?** (C-сторона `free` / YaoXiang сторона времени выполнения)

При генерации `yx-bindgen` добавляет аннотации для типичных паттернов:

```yaoxiang
# Выделено C, вызывающая сторона должна освободить через sqlite3_free
sqlite3_exec: (...) -> Int
    = native("sqlite3_exec")
    # memory: C-allocated, caller must free errmsg via sqlite3_free

# Вызывающая сторона выделяет указатель
read: (fd: Int, buf: *mut u8, count: Int) -> Int
    = native("read")
    # memory: caller-allocated buf
```

Время выполнения не выполняет автоматическое управление памятью для ссылок на указатели через границу FFI — ответственность за владение лежит на вызывающей стороне.

#### 5.2 Обработка строк

`char*`, возвращённый C-функцией, немедленно копируется при преобразовании в `String` YaoXiang. Владение исходным указателем определяется соглашением C-функции (объявляется через аннотации), автоматическое освобождение не выполняется.

### 6. Вопросы безопасности

#### 6.1 Безопасность 并发

Вызовы FFI **по умолчанию не участвуют в планировании DAG**, рассматриваются как блокирующие операции. C-функции, подтверждённые как reentrant, могут быть помечены как `@concurrent`:

```yaoxiang
# Чистая функция, без глобального состояния, может быть concurrent
sin_safe: (x: Float) -> Float = native("sin")
    # reentrant: true

# Имеет глобальное состояние, не может быть concurrent
strtok: (s: *const u8, delim: *const u8) -> *const u8 = native("strtok")
    # reentrant: false
```

`yx-bindgen` по возможности аннотирует информацию о reentrancy для стандартных C-функций (варианты `_r` для `strtok` и т.д.).

**Требования к асинхронным вызывающим:** Перед вызовом FFI функции вызывающая сторона должна убедиться, что целевая функция является reentrant. Время выполнения не выполняет автоматическое обнаружение — это статически неразрешимая задача.

#### 6.2 Изоляция ошибок

- Ошибки FFI вызовов распространяются через тип `Result` (если функция объявила возвращаемый тип Result)
- Механизм тайм-аута предотвращает deadlock внешних функций

```yaoxiang
# Вызов с тайм-аутом (реализуется на уровне FfiRegistry)
result = ffi.call_with_timeout("blocking_func", 5000)  # Тайм-аут 5 секунд
```

#### 6.3 Безопасность указателей

- Параметры-указатели требуют маркировки `unsafe` со стороны YaoXiang
- Жизненный цикл указателей через границу FFI гарантируется вызывающей стороной

### 7. Изменения компилятора

**Нулевые изменения синтаксиса** — требуется только объявление `native("symbol")`, которое уже реализовано в текущем компиляторе.

В интерпретаторе/среде выполнения добавляется:
- Инструкции динамической загрузки библиотек (привязки FFI для `DynamicLibrary`)
- Механизм тайм-аута

### 8. Функции, отклонённые к рассмотрению

Следующие функции после рассмотрения явно исключены и больше не включены в RFC:

- **`ffi.try_call`**: Избыточно, уже есть `native` + возвращаемый тип `Result`
- **`ffi.verify_signature`**: Время выполнения делает работу компилятора, неправильный уровень абстракции
- **`ffi.async_call`**: Требует уточнения модели контракта reentrancy перед рассмотрением
- **Таблица привязок, поддерживаемая сообществом**: Неосуществимо, переход к схеме инструментария `yx-bindgen`

## Компромиссы

### Преимущества

- ✅ **Нулевые изменения синтаксиса** — единственной точкой входа в FFI является `native("symbol")`, полная обратная совместимость
- ✅ **Библиотека как язык** — функциональность постепенно добавляется через стандартную библиотеку
- ✅ **Управление инструментарием** — `yx-bindgen` автоматически обрабатывает генерацию привязок
- ✅ **Безопасность памяти** — модель владения чёткая, нет use-after-free из-за автоматической сборки мусора
- ✅ **Отлаживаемость** — ошибки содержат нативные коды ошибок ОС

### Недостатки

- ⚠️ Безопасность типов ограничена выразительностью C заголовочных файлов (`void*` невозможно статически различить)
- ⚠️ `yx-bindgen` требует постоянной поддержки для отслеживания эволюции стандарта C
- ⚠️ Привязки для не-C языков (Python/JS/Java) требуют индивидуальной обработки проектами, нет унифицированной схемы

## Приложение B: Примеры кода

### Полный пример: Использование C-библиотеки

```yaoxiang
# Загрузка C математической библиотеки
libm = ffi.load_library("libm.so")

# Регистрация C символов в таблице времени выполнения (yx-bindgen делает это во время компиляции)
ffi.register_library_symbols(libm, ["sin", "cos", "sqrt"])

# Использование через объявление native
sin_f: (x: Float) -> Float = native("sin")
cos_f: (x: Float) -> Float = native("cos")

# Прямой вызов
result = sin_f(3.14159 / 2)

# Для C функций, которые могут завершиться с ошибкой, используется Result
file_open: (path: *const u8, mode: *const u8) -> Result(*mut opaque, Int)
    = native("fopen")
```

### Использование yx-bindgen

```bash
# Автоматическая генерация всех объявлений, без ручного написания
yx-bindgen --header /usr/include/math.h --output math_bindings.yx

# Подключение в YaoXiang
import "math_bindings.yx"
# sin_f / cos_f и др. уже автоматически объявлены как native("sin") / native("cos")
```
