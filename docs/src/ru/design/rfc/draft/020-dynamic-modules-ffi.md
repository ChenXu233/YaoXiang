## Аннотация

Настоящий документ на основе RFC-001, 008 и 018 далее уточняет и расширяет модель 并发 языка YaoXiang для решения практических сценариев использования, таких как **динамическая загрузка модулей**, **внешний функциональный интерфейс (FFI)** и **более тонкая оптимизация планирования**. Основные проектные решения включают:

1. **Метаданные динамических модулей**: предоставление описаний зависимостей на этапе компиляции для динамических библиотек, написанных на том же языке, что позволяет главной программе статически строить DAG с сохранением прозрачного 并发.
2. **Семантика планирования FFI**: внешние функции по умолчанию представлены как узлы `@block` в DAG, могут быть интегрированы в параллельное планирование посредством аннотаций (подробнее о инструментарии FFI в [RFC-021](../review/021-library-driven-ffi-extension.md)).
3. **Оптимизация на основе контекста вызова**: вместо статических пороговых fallback компилятор интеллектуально решает, следует ли встроить функцию или запланировать её как независимый узел, основываясь на реальной роли функции в DAG (количество потребителей, побочные эффекты и т.д.).
4. **Механизм слияния потока управления и DAG**: посредством Phi-узлов и динамического развёртывания динамические конструкции `if`, `loop` и другие естественно интегрируются в граф потока данных.
5. **Оптимизация памяти и производительности планировщика во время выполнения**: чёткое определение управления жизненным циклом узлов, аллокация регионов, блокирово-free очереди и другие low-cost абстракции.

Настоящий документ направлен на совершенствование спецификации языка, обеспечивая способность модели 并发 YaoXiang обрабатывать как статический анализ всей программы, так и гибко управлять динамикой и внешним взаимодействием, сохраняя высокую производительность и опыт разработки.

## Мотивация

### Недостатки существующего дизайна

RFC-001/008/018 построили элегантную прозрачную модель 并发, однако при столкновении с реальными потребностями всё ещё существуют слепые зоны:

- **Динамические модули**: когда программа поддерживает плагины или динамически связываемые библиотеки, главная программа не может получить информацию о внутренних вызовах модуля и зависимостях на этапе компиляции, что приводит к сбою построения глобального DAG.
- **FFI вызовы**: внешние функции (такие как библиотеки C) являются полностью «чёрным ящиком», их внутренняя реализация может содержать 并发, блокировку или побочные эффекты, рассмотрение их как обычных узлов нарушит безопасность 并发.
- **Накладные расходы планирования мелких функций**: «L1 автоматический fallback» в RFC-001 использует статический порог (количество инструкций <50), такое неявное правило затрудняет предсказуемость поведения разработчиками и не может адаптироваться к сложному контексту вызова.
- **Слияние потока управления и DAG**: представление динамических конструкций `if`, `loop` в DAG остаётся неопределённым, что может повлиять на точность анализа зависимостей.
- **Контроль накладных расходов во время выполнения**: с ростом количества узлов в DAG управление памятью планировщика и оптимизация производительности требуют чёткого проектирования во избежание узких мест.

### Цели

- При сохранении основной философии прозрачного 并发 обеспечить чёткую, безопасную и постепенную поддержку динамических модулей и FFI.
- Перейти от «неявных глобальных правил» оптимизации планирования к «интеллектуальным решениям на основе контекста», повышая предсказуемость и производительность.
- Усовершенствовать представление динамического потока управления в DAG, обеспечивая естественную интеграцию всех конструкций программы в модель потока данных.
- Чётко определить стратегии управления памятью и оптимизации производительности планировщика для реализации low-cost абстракций.

## Предложение

### 1. Метаданные динамических модулей

#### 1.1 Содержание контракта

Каждая динамическая библиотека (`.yxo` / платформо-зависимая динамическая библиотека), скомпилированная YaoXiang, должна сопровождаться **файлом метаданных** (`.yxmeta`), содержащим:

- **Список экспортируемых функций**: полные сигнатуры типов каждой функции (параметры, возвращаемое значение, маркеры ресурсов).
- **Маркеры побочных эффектов**: автоматически выводимые компилятором `@pure` / `@io` (разработчик также может явно переопределить).
- **Зависимости ресурсов**: является ли каждый параметр типом ресурса (например, `File`), содержит ли возвращаемое значение новые ресурсы.
- **Резюме графа вызовов** (опционально): возможные вызовы других экспортируемых функций по ID для обнаружения циклических зависимостей между модулями.
- **Информация о владении**: семантика владения параметров (заимствование/перемещение), владение возвращаемого значения.
- **并发安全ность**: автоматически выводимый результат соответствия `Send`/`Sync`.

Формат метаданных использует бинарный или структурированный текст (например, MessagePack) для обеспечения эффективности парсинга.

#### 1.2 Обработка на этапе компиляции

При компиляции главной программы при встрече с вызовом функции динамического модуля:

1. Считать соответствующий файл `.yxmeta`.
2. Создать в глобальном DAG **placeholder-узел**, записывающий зависимости ввода-вывода, маркеры побочных эффектов и прочее, полученные из метаданных.
3. Placeholder-узел участвует в анализе зависимостей наравне с обычными узлами, планировщик может заранее спланировать порядок выполнения.

#### 1.3 Привязка во время выполнения

При загрузке динамического модуля:

- Среда выполнения проверяет соответствие фактических сигнатур функций метаданным (предотвращение несоответствия версий).
- Привязывает placeholder-узел к фактическому указателю на функцию.

**О семантике планирования подграфа**: если динамический модуль содержит внутренний независимый под-DAG (например, модуль сам содержит логику 并发), этот подграф выполняется как **независимая единица планирования**. Его границы определяются экспортируемыми функциями модуля: при вызове экспортируемой функции подграф начинает выполнение как единое целое и продолжается до возврата функции. Внутреннее планирование узлов подграфа осуществляется собственным планировщиком подграфа (модуль может продолжать использовать стандартный планировщик), однако взаимодействие подграфа с главным DAG ограничено вводом-выводом данных — placeholder-узел в главном DAG заботится только о начале и конце подграфа, не вмешиваясь во внутреннее планирование. Данный дизайн гарантирует инкапсуляцию модуля, сохраняя при этом статическую целостность главного DAG.

#### 1.4 Гарантии безопасности

- Если динамический модуль нарушает контракт (например, заявляет `@pure`, но модифицирует глобальное состояние), последствия ложатся на разработчика (подобно границам unsafe в FFI). Однако поскольку это тот же язык, возможно усиление безопасности посредством проверок во время выполнения (например, изоляция памяти), но это увеличит накладные расходы.
- Циклические зависимости между модулями: если модуль A вызывает B, а B вызывает A, и отношения вызовов объявлены в метаданных, компилятор может обнаружить и сообщить об ошибке; если не объявлены, возможна взаимоблокировка во время выполнения, обнаруживаемая планировщиком с panic.

### 2. Семантика планирования FFI в DAG

Полный инструментарий поддержки FFI (загрузка динамических библиотек, генерация привязок, преобразование типов, владение памятью) определён в [RFC-021](../review/021-library-driven-ffi-extension.md). В данном разделе описано только поведение FFI вызовов в планировании DAG.

#### 2.1 Поведение планирования по умолчанию

Внешние функции (объявленные через `native("symbol")`) в DAG по умолчанию представлены как **узлы `@block`**:

- Не участвуют в параллельном планировании DAG, выполняются синхронно в текущем потоке.
- Во время выполнения планировщик не вмешивается во внутренний 并发.
- Возвращаемое значение доступно, но сам вызов не порождает рёбер зависимости.

#### 2.2 Опциональные аннотации 并发

Разработчик может посредством аннотаций интегрировать FFI вызовы в планирование DAG (подробнее в [RFC-021 §2.2](../review/021-library-driven-ffi-extension.md)):

- `@pure`: рассматривается как обычный узел DAG, может выполняться параллельно с другими узлами без зависимостей.
- `@io`: участвует в анализе зависимостей ресурсов, множественные вызовы к одному ресурсу автоматически сериализуются.

#### 2.3 Влияние на планировщик

Узлы FFI в планировщике используют ту же структуру `TaskNode`, что и обычные узлы, отличие только в том, что `effect` помечен как `Block`. При встрече с узлом `Block` планировщик пропускает параллельное планирование и выполняет синхронно.

### 3. Оптимизация на основе контекста вызова

Вместо «L1 автоматический fallback» со статическим порогом в RFC-001, переход к **интеллектуальным решениям компилятора на основе реального контекста каждой точки вызова в DAG**.

#### 3.1 Основа для принятия решений оптимизации

Компилятор анализирует каждый узел вызова функции:

- **Количество потребителей**: сколько downstream-узлов используют результат данного узла. Если равен 1, узел является кандидатом на встраивание; если больше 1, должен остаться независимым узлом для разделения результата.
- **Побочные эффекты**: если узел имеет побочный эффект `@io`, должен остаться независимым узлом для обеспечения порядка.
- **Оценка объёма вычислений**: по-прежнему может использоваться эвристика на основе количества инструкций и т.д., но не как жёсткий порог, а только для оценки выгоды от встраивания.
- **Зависимости ресурсов**: если узел связан с переменными ресурса (например, `File`), и переменная ресурса передаётся между upstream и downstream, встраивание может нарушить цепь зависимостей, требуется осторожность.

#### 3.2 Операция встраивания

Если решение — встроить:

- Логика вычислений узла непосредственно встраивается в код его единственного downstream-узла.
- Узел удаляется из DAG, его входные данные напрямую становятся входными данными downstream-узла.
- При генерации итогового кода встроенная функция не порождает независимую единицу планирования.

#### 3.3 Ограничения встраивания

- Рекурсивные функции или вызовы внутри циклов обычно не встраиваются во избежание бесконечного развёртывания.
- Функции через границы модулей (динамические модули, FFI) не встраиваются.
- Разработчик может принудительно запретить встраивание аннотацией `@noinline` или подсказать компилятору попробовать встроить аннотацией `@forceinline`.

#### 3.4 Наблюдаемость

Компилятор должен генерировать отчёт об оптимизации (включаемый через `--emit-optimization-report`), содержащий следующую информацию:

- **Каждая точка встраивания**: список имён встроенных функций, местоположение вызова, причина встраивания (например, «единственный потребитель и чистая функция»).
- **Причины сохранения независимого узла**: например, «несколько потребителей», «побочные эффекты», «вызов через модуль» и т.д.
- **Статистика решений**: общее количество встраиваний, количество сохранённых узлов, помощь разработчику в оценке эффекта оптимизации.

Формат вывода отчёта может быть текстовым или JSON для удобства парсинга инструментами.

### 4. Слияние потока управления и DAG

#### 4.1 Обработка условных ветвлений (if)

Вводится **Phi-узел** (заимствование из формы SSA) для представления точки слияния ветвей:

- При компиляции для каждого выражения `if` строится:
  - Два под-DAG ветвей (соответственно `then` и `else`).
  - Phi-узел, чьи входы включают условную переменную и выходы двух ветвей.
- Семантика Phi-узла: когда условная переменная готова, на основе значения условия выбирается выход соответствующей ветви как собственный выход.
- Во время выполнения Phi-узел зависит от условной переменной; когда условие готово, он динамически добавляет себя в downstream-список выбранной ветви и ожидает результат этой ветви.

Пример DAG:
```
        cond
       /    \
  then DAG  else DAG
       \    /
        Phi
         |
      последующие узлы
```

#### 4.2 Обработка циклов (loop/while)

Цикл рассматривается как под-DAG с обратными рёбрами, **развёртываемый по требованию** во время выполнения:

- При компиляции распознаётся тело цикла, строится **шаблон цикла**, включающий:
  - Узел условия.
  - Под-DAG тела цикла.
  - Переменные состояния, передаваемые между итерациями.
- Во время выполнения, когда требуется результат цикла (например, использование накопленного значения после завершения цикла), планировщик начинает динамическое развёртывание итераций:
  1. Первое планирование узла условия, если истинно — создаётся первая итерация под-DAG, входы включают начальное состояние и внешние переменные.
  2. После завершения итерации генерируется новое состояние, снова планируется узел условия (зависит от нового состояния) для решения о продолжении.
  3. Повторение до ложности условия, выход последней итерации является результатом цикла.

**Сложный пример: условие цикла зависит от обновления внутри тела цикла**
```yaoxiang
let mut x = 0
while x < 10 {
    x = compute(x)  // x обновляется внутри тела цикла
}
```

В данном паттерне узел условия `x < 10` зависит от обновлённого `x` после каждой итерации. DAG представлен следующим образом:
- Шаблон цикла содержит переменную состояния `x` с начальным значением 0.
- Каждая итерация: сначала выполняется узел условия (зависит от текущего `x`), если истинно — выполняется `x = compute(x)` и генерируется новый `x`, затем снова вход в узел условия.
- Во время выполнения динамическое развёртывание по описанному процессу до ложности условия.

Зависимости между итерациями естественно формируются через переменные состояния, итерации с зависимостями автоматически сериализуются, итерации без зависимостей могут выполняться параллельно (например, `map`).

#### 4.3 Специальная обработка бесконечных циклов

Единственный бесконечный цикл выполняется синхронно как часть главного DAG напрямую (без накладных расходов планирования); несколько бесконечных циклов выполняются как background DAG с time-slice 并发 планировщиком.

### 5. Оптимизация памяти и производительности планировщика во время выполнения

#### 5.1 Управление жизненным циклом узлов

- Каждый узел поддерживает **счётчик ссылок** (атомарная переменная), представляющий количество потребителей, зависящих от его результата.
- После выполнения узла и передачи результата всем downstream-узлам счётчик ссылок обнуляется, память узла может быть освобождена.
- Сами результаты значений также используют подсчёт ссылок (`Arc<T>`), но возможна оптимизация: если результат используется только одним потребителем, владение перемещается напрямую, избегая накладных расходов подсчёта.

#### 5.2 Регионная аллокация памяти (Arena)

Для динамически генерируемых узлов с коротким жизненным циклом (например, итерации цикла) используется **аллокатор регионов**:

- Для одного развёртывания цикла выделяется один регион памяти.
- Узлы внутри региона распределяются непрерывно, освобождение происходит целиком, уменьшая фрагментацию и накладные расходы освобождения.
- При завершении региона вся память узлов回收一次.

#### 5.3 Блокирово-free структуры данных

- Счётчики зависимостей: используется `AtomicUsize` с атомарным декрементом через `fetch_sub`.
- Очередь готовности: применяется двусторонняя очередь Chase-Lev (локальная очередь каждого потока + work-stealing), уменьшающая конфликт блокировок.
- Downstream-список: после создания только для чтения, избегая конкурентных модификаций.

#### 5.4 Адаптивное планирование

- Если в системе только один бесконечный цикл, выполняется синхронно напрямую, нулевые накладные расходы планирования.
- На основе гранулярности задач и нагрузки системы динамически регулируется степень параллелизма (например, мониторинг длины очереди для регулировки количества рабочих потоков).

#### 5.5 Принцип low-cost абстракций

Все накладные расходы планирования пропорциональны количеству задач, дополнительные накладные расходы на каждую задачу (создание, помещение в очередь, обработка зависимостей) контролируются на уровне десятков наносекунд. Для сверхмелкозернистых задач накладные расходы планирования избегаются посредством оптимизации встраивания (см. раздел 3).

### 6.1 Формат метаданных динамических модулей (черновик)

```rust
struct Metadata {

```

### 6.2 Основные структуры данных планировщика

```rust
struct TaskNode {
    id: TaskId,
    deps: Vec<TaskId>,                // Upstream зависимости
    remaining_deps: AtomicUsize,
    inputs: Vec<Option<Value>>,
    result: Option<Value>,
    func: Executable,
    downstream: Vec<TaskId>,           // Downstream узлы (только чтение после создания)
    effect: EffectTag,

}

    arenas: ArenaAllocator,               // Аллокатор регионов
}
```

### 6.3 Анализ оптимизации на основе контекста

Компилятор на уровне MIR выполняет следующие шаги:

1. Построение глобального графа вызовов и графа зависимостей данных.
2. Для каждого узла вызова функции вычисление out-degree (количество потребителей).
3. Если out-degree равен 1, функция не имеет побочных эффектов (`@pure`) и нерекурсивна, узел помечается как «кандидат на встраивание».
4. С учётом эвристики (например, количество инструкций) оценка выгоды встраивания, решение о встраивании.
5. При встраивании код узла вызова встраивается в downstream, зависимости обновляются.

### 6.4 Представление узлов потока управления

```rust
enum NodeKind {
    Normal(FuncId),
    Phi { cond: TaskId, then_branch: TaskId, else_branch: TaskId },
    LoopTemplate { cond: FuncId, body: FuncId, state_var: VarId },
    // ...
}
```

При динамическом развёртывании во время выполнения `LoopTemplate` генерирует серию экземпляров узлов `Normal`.

## Компромиссы

### Преимущества

- **Безопасная интеграция динамических модулей**: контракт метаданных позволяет динамическим библиотекам бесшовно разделять модель 并发 с главной программой, сохраняя статическую целостность DAG.
- **Постепенная интеграция FFI**: разработчик может постепенно добавлять аннотации к внешним функциям, переходя от безопасного fallback к эффективному 并发.
- **Предсказуемость оптимизации**: решения на основе контекста заменяют неявные пороги, поведение прозрачно, разработчик может понять оптимизацию через инструменты.
- **Естественная интеграция потока управления**: Phi-узлы и динамическое развёртывание позволяют DAG представлять все конструкции программы без специального синтаксиса.
- **Масштабируемость производительности**: аллокация регионов, блокирово-free очереди и прочий дизайн обеспечивают способность планировщика справляться с крупномасштабным 并发.

### Недостатки

- **Увеличение сложности компиляции контрактом метаданных**: требуется генерация и парсинг метаданных для динамических библиотек, инструментарий должен поддерживать.
- **Зависимость правильности аннотаций FFI от разработчика**: некорректные аннотации могут привести к гонке данных, необходимо снижать риски через документацию и подсказки инструментов.
- **Времязатратность анализа оптимизации контекста**: глобальный анализ может увеличить время компиляции, но может быть смягчено инкрементной компиляцией.
- **Увеличение накладных расходов выполнения при динамическом развёртывании**: развёртывание циклов требует динамического создания узлов, но аллокация регионов может смягчить.

## Стратегия реализации

### Фазы разделения (с рекомендациями приоритетов)

> **Рекомендации по приоритету реализации**: на начальном этапе не обязательно стремиться к идеальной реализации, можно сначала использовать простые решения для запуска системы, затем постепенно оптимизировать. Например:
> - Подсчёт ссылок можно напрямую использовать через `Arc`.
> - Блокирово-free очередь можно использовать через зрелые библиотеки (например, deque crossbeam).
> - Аллокация регионов может сначала использовать простой bump allocator, оптимизировать позднее.

#### Фаза 1: Базовая поддержка (v0.7)
- [ ] Реализовать FFI по умолчанию как `@block`.
- [ ] Добавить аннотации `@pure`, `@io` для FFI.
- [ ] Реализовать типы-обёртки ресурсов (например, `File`) и их базовые методы.

#### Фаза 2: Метаданные динамических модулей (v0.8)
- [ ] Спроектировать формат метаданных, модифицировать компилятор для генерации `.yxmeta` для динамических библиотек.
- [ ] Реализовать чтение метаданных при компиляции главной программы и создание placeholder-узлов.
- [ ] Реализовать механизм привязки во время выполнения.

#### Фаза 3: Оптимизация контекста (v0.9)
- [ ] Реализовать анализ графа вызовов, вычисление out-degree узлов.
- [ ] Добавить поддержку решений о встраивании и генерации кода.
- [ ] Реализовать вывод отчётов об оптимизации (точки встраивания, причины и т.д.).

#### Фаза 4: Слияние потока управления и DAG (v0.10)
- [ ] Реализовать Phi-узлы и представление условных ветвлений при компиляции.
- [ ] Реализовать шаблоны циклов и динамическое развёртывание во время выполнения.
- [ ] Усовершенствовать background-планирование бесконечных циклов.

#### Фаза 5: Оптимизация производительности (v1.0)
- [ ] Реализовать аллокатор регионов.
- [ ] Оптимизировать блокирово-free очереди и work-stealing.
- [ ] Бенчмаркинг и настройка.

## Связь с другими RFC

- **RFC-001**: расширяет обработку побочных эффектов и уровни 并发, заменяя автоматический fallback контекстной оптимизацией.

- **RFC-018**: уточняет построение DAG и реализацию планировщика, добавляя Phi-узлы и динамическое развёртывание.
