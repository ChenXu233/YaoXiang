## Аннотация

Данный документ на основе RFC-001, 008 и 018 дополнительно детализирует и расширяет модель параллелизма YaoXiang для решения практических задач: **динамическая загрузка модулей**, **внешний интерфейс функций (FFI)** и **более тонкая оптимизация планирования**. Ключевые аспекты дизайна включают:

1. **Контракт метаданных динамических модулей**: предоставление описаний зависимостей на этапе компиляции для динамических библиотек, написанных на том же языке, что позволяет главной программе статически строить DAG с сохранением прозрачного параллелизма.
2. **Многоуровневая обработка FFI**: внешний код по умолчанию понижается до синхронного (`@block`), разработчики могут предоставлять метаданные через аннотации для постепенной интеграции в планирование параллелизма, внешний параллелизм управляется внешним языком.
3. **Оптимизация на основе контекста вызова**: вместо статических пороговых возвратов компилятор интеллектуально решает, следует ли встраивать функцию или планировать её как независимый узел, основываясь на реальной роли функции в DAG (количество потребителей, побочные эффекты и т.д.).
4. **Механизм объединения потока управления и DAG**: через Phi-узлы и динамическое разворачивание динамические конструкции вроде `if`, `loop` естественно интегрируются в граф потока данных.
5. **Оптимизация памяти и производительности планировщика времени выполнения**: чёткое определение управления жизненным циклом узлов, распределения по областям, безблокировочных очередей и других низкозатратных абстракций.

Данный документ призван усовершенствовать спецификацию языка, обеспечивая способность модели параллелизма YaoXiang справляться как со статическим анализом всей программы, так и с динамичностью и внешним взаимодействием, сохраняя высокую производительность и качество разработки.

## Мотивация

### Недостатки существующего дизайна

RFC-001/008/018 построили элегантную модель прозрачного параллелизма, но при столкновении с реальными потребностями всё ещё существуют слепые зоны:

- **Динамические модули**: когда программа поддерживает плагины или динамически связываемые библиотеки, главная программа не может на этапе компиляции узнать внутренние вызовы модуля и зависимости, что приводит к невозможности построения глобального DAG.
- **Вызовы FFI**: внешние функции (например, из C-библиотек) являются полностью чёрным ящиком, их внутренняя часть может содержать параллелизм, блокировки или побочные эффекты, прямое рассмотрение их как обычных узлов разрушит безопасность параллелизма.
- **Накладные расходы планирования малых функций**: предложенное в RFC-001 "автоматическое возвращение L1" использует статический порог (количество инструкций <50), это неявное правило затрудняет разработчикам предсказание поведения и не может адаптироваться к сложному контексту вызовов.
- **Объединение потока управления и DAG**: представление динамических структур вроде `if`, `loop` в DAG пока не определено, что может повлиять на точность анализа зависимостей.
- **Контроль накладных расходов времени выполнения**: с увеличением количества узлов DAG управление памятью планировщика и оптимизация производительности требуют чёткого дизайна во избежание узких мест.

### Цели

- Сохраняя основную философию прозрачного параллелизма, обеспечить чёткую, безопасную и постепенную поддержку динамических модулей и FFI.
- Превратить оптимизацию планирования из "неявных глобальных правил" в "интеллектуальные решения на основе контекста", повышая предсказуемость и производительность.
- Усовершенствовать представление динамического потока управления в DAG, обеспечивая естественную интеграцию всех конструкций программы в модель потока данных.
- Чётко определить стратегии управления памятью и оптимизации производительности планировщика для достижения низкозатратных абстракций.

## Предложение

### 1. Контракт метаданных динамических модулей

#### 1.1 Содержание контракта

Каждая динамическая библиотека (`.yxo` / платформо-зависимая динамическая библиотека), скомпилированная YaoXiang, должна поставляться с **файлом описания метаданных** (`.yxmeta`), содержащим:

- **Список экспортируемых функций**: полные сигнатуры типов каждой функции (параметры, возвращаемое значение, маркеры ресурсов).
- **Маркеры побочных эффектов**: автоматически выводимые компилятором `@pure` / `@io` (разработчик также может явно переопределить).
- **Зависимости ресурсов**: является ли каждый параметр типом ресурса (например, `File`), содержит ли возвращаемое значение новые ресурсы.
- **Резюме графа вызовов** (опционально): идентификаторы других экспортируемых функций, которые данная функция может вызывать, для обнаружения циклических зависимостей между модулями.
- **Информация о владении**: семантика владения параметрами (заимствование/перемещение), владение возвращаемым значением.
- **Параллельная безопасность**: автоматически выводимые результаты соответствия `Send`/`Sync`.

Формат метаданных использует бинарный или структурированный текст (например, MessagePack) для обеспечения эффективности парсинга.

#### 1.2 Обработка на этапе компиляции

При компиляции главной программы при обнаружении вызова функции динамического модуля:

1. Читается файл `.yxmeta` соответствующего модуля.
2. В глобальном DAG создаётся **узел-заполнитель**, записывающий зависимости ввода-вывода, маркеры побочных эффектов и т.д., полученные из метаданных.
3. Узел-заполнитель участвует в анализе зависимостей наравне с обычными узлами, планировщик может заранее спланировать порядок выполнения.

#### 1.3 Связывание во время выполнения

При загрузке динамического модуля:

- Среда выполнения проверяет соответствие фактической сигнатуры функции метаданным (предотвращение несовпадения версий).
- Узел-заполнитель связывается с фактическим указателем на функцию.

**О семантике планирования подграфа**: если внутри динамического модуля есть независимый под-DAG (например, модуль содержит внутреннюю логику параллелизма), этот подграф выполняется как **независимая единица планирования**. Его границы определяются экспортируемыми функциями модуля: при вызове экспортируемой функции подграф начинает выполняться как единое целое до возврата этой функции. Внутреннее планирование узлов подграфа управляется собственным планировщиком подграфа (модуль может продолжать использовать стандартный планировщик), но взаимодействие подграфа с главным DAG ограничено только потоком данных ввода-вывода — узел-заполнитель в главном DAG заботится только о начале и конце подграфа, не вмешиваясь в его внутреннее планирование. Этот дизайн обеспечивает инкапсуляцию модуля, одновременно сохраняя статическую целостность главного DAG.

#### 1.4 Гарантии безопасности

- Если динамический модуль нарушает контракт (например, заявляет `@pure`, но модифицирует глобальное состояние), последствия несёт разработчик (аналогично небезопасным границам FFI). Однако, поскольку это тот же язык, возможно усиление безопасности через проверки времени выполнения (например, изоляция памяти), но это увеличит накладные расходы.
- Циклические зависимости между модулями: если модуль A вызывает B, B вызывает A, и эти отношения объявлены в метаданных, компилятор может обнаружить и сообщить об ошибке; если не объявлены, возможна взаимоблокировка во время выполнения, которую обнаружит планировщик и вызовет panic.

### 2. Многоуровневая обработка FFI

#### 2.1 Понижение по умолчанию

Внешние функции (введённые через `extern`) по умолчанию рассматриваются как синхронные вызовы `@block`:

- Вызов не попадает в планирование DAG, выполняется непосредственно в текущем потоке.
- Во время выполнения планировщик YaoXiang не вмешивается во внутренний параллелизм.
- Возвращаемое значение доступно, но сам вызов не участвует в анализе зависимостей.

> **⚠️ Для разработчиков**: вызовы FFI по умолчанию не участвуют в планировании DAG, поэтому существующий внутри них параллелизм (например, потоки, запущенные C-библиотекой) полностью управляется внешним языком, YaoXiang не может его отслеживать. Если требуется интеграция вызова FFI в планирование параллелизма, необходимо предоставить метаданные через аннотации (см. раздел 2.2). Не следует предполагать, что "синхронный код автоматически станет параллельным" — вызовы FFI не будут автоматически распараллеливаться.

#### 2.2 Опциональная аннотация метаданных

Разработчики могут через аннотации предоставить более точные метаданные, позволяя вызовам FFI постепенно интегрироваться в планирование параллелизма:

```yaoxiang
@pure   // Чистая функция, без побочных эффектов, свободное планирование
extern "C" fn sin(x: Double) -> Double

@io     // Ввод-вывод, требуется сериализация по зависимостям ресурсов
extern "C" fn write(fd: i32, buf: *u8, count: usize) -> isize

@block  // Явное объявление синхронным (по умолчанию, можно опустить)
extern "C" fn old_api() -> i32
```

- Если помечен как `@pure` и параметры не затрагивают переменные ресурсов, рассматривается как обычный узел DAG, может выполняться параллельно.
- Если помечен как `@io` и параметры содержат типы ресурсов (например, обёрнутый `File`), автоматически устанавливаются рёбра зависимостей.

#### 2.3 Обёртывание ресурсов

Рекомендуется, чтобы стандартная библиотека предоставляла типы-обёртки для распространённых FFI-ресурсов (файловые дескрипторы, подключения к базам данных), инкапсулируя сырые ресурсы в типы ресурсов YaoXiang для использования анализа зависимостей DAG. Например:

```yaoxiang
struct File { fd: i32 }  // Помечен как тип ресурса
impl File {
    @io
    fn write(self: &mut File, data: &[u8]) -> Result { ... }
}
```

Таким образом, множественные вызовы `write` для одной и той же переменной `File` будут автоматически сериализованы.

#### 2.4 Управление внешним параллелизмом

Если FFI-функция запускает потоки или асинхронные операции внутри, это полностью управляется внешним языком. YaoXiang заботится только о начале и конце вызова, не отслеживая внутренние детали. Разработчики должны убедиться, что эти параллельные операции не конфликтуют с моделью памяти YaoXiang (например, при использовании callback учитывать владение и ограничения Send).

### 3. Оптимизация на основе контекста вызова

Вместо "автоматического возврата L1" со статическим порогом в RFC-001, теперь **компилятор интеллектуально принимает решения на основе реального контекста каждой точки вызова в DAG**.

#### 3.1 Основа для принятия решений об оптимизации

Компилятор анализирует каждый узел вызова функции:

- **Количество потребителей**: сколько нижестоящих узлов используют результат этого узла. Если равен 1, кандидат на встраивание; если больше 1, должен остаться независимым узлом для совместного использования результата.
- **Побочные эффекты**: если узел имеет побочный эффект `@io`, должен остаться независимым узлом для обеспечения порядка.
- **Оценка вычислительной сложности**: всё ещё можно использовать эвристику на основе количества инструкций и т.д., но не как жёсткий порог, а только для оценки выгоды от встраивания.
- **Зависимости ресурсов**: если узел затрагивает переменные ресурсов (например, `File`) и эти переменные передаются между выше- и нижестоящими уровнями, встраивание может нарушить цепочку зависимостей, требуется осторожность.

#### 3.2 Операция встраивания

Если принято решение о встраивании:

- Логика вычисления этого узла непосредственно встраивается в код его единственного нижестоящего узла.
- Узел удаляется из DAG, его входы напрямую становятся входами нижестоящего узла.
- При генерации конечного кода встроенная функция не создаёт независимой единицы планирования.

#### 3.3 Ограничения встраивания

- Рекурсивные функции или вызовы внутри циклов обычно не встраиваются во избежание бесконечного разворачивания.
- Функции через границы модулей (динамические модули, FFI) не встраиваются.
- Разработчики могут через аннотацию `@noinline` принудительно запретить встраивание или через `@forceinline` подсказать компилятору попробовать встроить.

#### 3.4 Наблюдаемость

Компилятор должен генерировать отчёт об оптимизации (можно включить через `--emit-optimization-report`), содержащий следующую информацию:

- **Каждая точка встраивания**: список встроенных функций, места вызовов, причины встраивания (например, "единственный потребитель и чистая функция").
- **Причины сохранения независимых узлов**: например, "несколько потребителей", "содержит побочные эффекты", "вызов через модуль" и т.д.
- **Статистика решений**: общее количество встраиваний, количество сохранённых узлов, помощь разработчикам в оценке эффекта оптимизации.

Формат вывода отчёта может быть текстовым или JSON для удобства парсинга инструментами.

### 4. Объединение потока управления и DAG

#### 4.1 Обработка условных ветвлений (if)

Вводится **Phi-узел** (заимствование из формы SSA) для представления точки слияния ветвлений:

- На этапе компиляции для каждого выражения `if` строится:
  - Два под-DAG ветвей (соответственно для `then` и `else`).
  - Phi-узел, чьи входы включают условную переменную и выходы двух ветвей.
- Семантика Phi-узла: когда условная переменная готова, согласно значению условия выбирается выход соответствующей ветви в качестве собственного выхода.
- Во время выполнения Phi-узел зависит от условной переменной; после готовности условия он динамически добавляет себя в список нижестоящих выбранной ветви и ожидает результат этой ветви.

Пример DAG:
```
        cond
       /    \
  then DAG  else DAG
       \    /
        Phi
         |
      последующие узлы
```

#### 4.2 Обработка циклов (loop/while)

Цикл рассматривается как под-DAG с обратными рёбрами, **разворачиваемый по требованию** во время выполнения:

- На этапе компиляции распознаётся тело цикла, строится **шаблон цикла**, включающий:
  - Условный узел.
  - Под-DAG тела цикла.
  - Переменные состояния, передаваемые между итерациями.
- Во время выполнения, когда требуется результат цикла (например, значение накопления после завершения цикла), планировщик начинает динамически разворачивать итерации:
  1. Первое планирование условного узла, если истинно — создаётся экземпляр под-DAG первой итерации, входы включают начальное состояние и внешние переменные.
  2. После завершения итерации создаётся новое состояние, снова планируется условный узел (зависит от нового состояния) для решения о продолжении.
  3. Повторение до ложности условия, выход последней итерации является результатом цикла.

**Сложный пример: условие цикла зависит от обновления внутри тела цикла**
```yaoxiang
let mut x = 0
while x < 10 {
    x = compute(x)  // x обновляется внутри тела цикла
}
```

В этом паттерне условный узел `x < 10` зависит от обновлённого `x` после каждой итерации. DAG представляется следующим образом:
- Шаблон цикла содержит переменную состояния `x` с начальным значением 0.
- Каждая итерация: сначала выполняется условный узел (зависит от текущего `x`), если истинно — выполняется `x = compute(x)` и создаётся новый `x`, затем снова входим в условный узел.
- Во время выполнения динамически разворачивается по описанному процессу, пока условие не станет ложным.

Зависимости между итерациями естественно формируют поток данных через переменные состояния, итерации с зависимостями автоматически сериализуются, итерации без зависимостей могут выполняться параллельно (например, `map`).

#### 4.3 Особая обработка бесконечных циклов

Одиночный бесконечный цикл выполняется синхронно как часть главного DAG напрямую (нулевые накладные расходы планирования); несколько бесконечных циклов выполняются как фоновые DAG, планировщик разделяет их по временным срезам для параллельного выполнения.

### 5. Оптимизация памяти и производительности планировщика времени выполнения

#### 5.1 Управление жизненным циклом узлов

- Каждый узел поддерживает **счётчик ссылок** (атомарная переменная), показывающий количество потребителей, зависящих от его результата.
- После завершения выполнения узла и передачи результата всем нижестоящим счётчик ссылок становится равным нулю, память узла может быть освобождена.
- Сами результаты значений также используют подсчёт ссылок (`Arc<T>`), но возможна оптимизация: если результат используется только одним потребителем, права владения передаются напрямую, избегая накладных расходов на подсчёт.

#### 5.2 Поканальное распределение памяти (Arena)

Для динамически генерируемых узлов с коротким жизненным циклом (например, итерации циклов) используется **поканальный аллокатор**:

- Для одного разворачивания цикла выделяется область памяти.
- Узлы внутри области распределяются непрерывно, освобождаются все вместе, уменьшая фрагментацию и накладные расходы на освобождение.
- При завершении области вся память узлов возвращается одним махом.

#### 5.3 Безблокировочные структуры данных

- Счётчики зависимостей: используется `AtomicUsize` с атомарным декрементом через `fetch_sub`.
- Очередь готовности: используется двухсторонняя очередь Chase-Lev (локальная очередь каждого потока + work-stealing), уменьшающая конфликты блокировок.
- Списки нижестоящих: после создания только для чтения, избегая конкурентных модификаций.

#### 5.4 Адаптивное планирование

- Если в системе только один бесконечный цикл, выполняется синхронно напрямую, нулевые накладные расходы планирования.
- Согласно гранулярности задач и нагрузке системы динамически регулируется степень параллелизма (например, через мониторинг длины очереди регулируется количество рабочих потоков).

#### 5.5 Принцип низкозатратных абстракций

Все накладные расходы планирования пропорциональны количеству задач, дополнительные накладные расходы на каждую задачу (создание, постановка в очередь, обработка зависимостей) контролируются на уровне десятков наносекунд. Для задач сверхмалой гранулярности через оптимизацию встраивания (см. раздел 3) избегается планирование.

## Детальный дизайн

### 6.1 Формат метаданных динамических модулей (черновик)

```rust
// Структура файла метаданных (упрощённо)
struct Metadata {
    version: u32,
    functions: Vec<FuncMeta>,
}

struct FuncMeta {
    name: String,
    signature: TypeSignature,
    effects: EffectTag,      // Pure | IO | Block
    resource_params: Vec<usize>, // Список индексов параметров, указывающих какие параметры являются типами ресурсов
    calls: Vec<String>,       // Имена других экспортируемых функций, которые вызывает данная (опционально)
    ownership: OwnershipInfo,
    send_sync: SendSync,      // Соответствует ли Send/Sync
}
```

### 6.2 Основные структуры данных планировщика

```rust
struct TaskNode {
    id: TaskId,
    deps: Vec<TaskId>,                // Вышестоящие зависимости
    remaining_deps: AtomicUsize,
    inputs: Vec<Option<Value>>,
    result: Option<Value>,
    func: Executable,
    downstream: Vec<TaskId>,           // Нижестоящие узлы (после создания только для чтения)
    effect: EffectTag,
    arena_id: Option<ArenaId>,         // Принадлежащая область (опционально)
}

struct Scheduler {
    ready_queues: PerThreadQueue<TaskId>, // Локальная очередь каждого потока
    global_work_stealer: WorkStealer,
    arenas: ArenaAllocator,               // Поканальный аллокатор
}
```

### 6.3 Анализ оптимизации на основе контекста

Компилятор на уровне MIR выполняет следующие шаги:

1. Строится глобальный граф вызовов и граф зависимостей данных.
2. Для каждого узла вызова функции вычисляется его исходящая степень (количество потребителей).
3. Если исходящая степень равна 1, функция без побочных эффектов (`@pure`) и нерекурсивна, узел помечается как "кандидат на встраивание".
4. С учётом эвристики (например, количество инструкций) оценивается выгода от встраивания, принимается решение о встраивании.
5. При встраивании код узла вызова встраивается в его нижестоящий узел, зависимости обновляются.

### 6.4 Представление узлов потока управления

```rust
enum NodeKind {
    Normal(FuncId),
    Phi { cond: TaskId, then_branch: TaskId, else_branch: TaskId },
    LoopTemplate { cond: FuncId, body: FuncId, state_var: VarId },
    // ...
}
```

При динамическом разворачивании во время выполнения `LoopTemplate` генерирует серию экземпляров `Normal`.

## Компромиссы

### Преимущества

- **Безопасная интеграция динамических модулей**: контракт метаданных позволяет динамическим библиотекам бесшовно разделять модель параллелизма с главной программой, сохраняя целостность статического DAG.
- **Постепенная интеграция FFI**: разработчики могут постепенно добавлять аннотации для внешних функций, переходя от безопасного понижения к эффективному параллелизму.
- **Предсказуемая оптимизация**: решения на основе контекста заменяют неявные пороги, поведение прозрачно, разработчики могут понимать оптимизацию через инструменты.
- **Естественная интеграция потока управления**: Phi-узлы и динамическое разворачивание позволяют DAG представлять все конструкции программы без специального синтаксиса.
- **Масштабируемая производительность**: поканальное распределение, безблокировочные очереди и другие решения обеспечивают способность планировщика справляться с крупномасштабным параллелизмом.

### Недостатки

- **Контракт метаданных увеличивает сложность компиляции**: требуется генерация и парсинг метаданных для динамических библиотек, инструментарий должен поддерживать.
- **Аннотации FFI зависят от корректности разработчика**: неправильная маркировка может привести к гонке данных, необходимо снижать риски через документацию и подсказки инструментов.
- **Анализ оптимизации контекста затратен по времени**: глобальный анализ может увеличить время компиляции, но может быть смягчён через инкрементную компиляцию.
- **Динамическое разворачивание увеличивает накладные расходы выполнения**: разворачивание циклов требует динамического создания узлов, но поканальное распределение может смягчить.

## Стратегия реализации

### Фазы с приоритетами

> **Рекомендации по приоритетам реализации**: на начальном этапе не обязательно стремиться к идеальной реализации, можно сначала использовать простые решения для запуска системы, затем постепенно оптимизировать. Например:
> - Подсчёт ссылок можно напрямую использовать через `Arc`.
> - Безблокировочную очередь можно использовать через зрелые библиотеки (например, deque от crossbeam).
> - Поканальное распределение можно сначала реализовать через простой bump allocator, оптимизировать позже.
