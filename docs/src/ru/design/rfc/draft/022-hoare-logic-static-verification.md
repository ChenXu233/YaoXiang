# RFC 022: Поддержка статической верификации по Хоару (аннотации спецификаций и типы спецификаций)

> **Статус**: На рассмотрении
> **Автор**: Чэнь Сюй
> **Дата создания**: 2026-03-16
> **Последнее обновление**: 2026-03-16

> **Ссылки**:
> - [RFC-010: Единый синтаксис типов](./010-unified-type-syntax.md)
> - [RFC-011: Проектирование системы обобщённых типов](./011-generic-type-system.md)
> - [RFC-009: Модель владения](./009-ownership-model.md)

## Аннотация

В данном документе предлагается ввести в язык YaoXiang **механизм статической верификации на основе логики Хоара**, позволяющий разработчикам записывать предусловия, постусловия и инварианты циклов в виде комментариев с использованием `//!` или `/*! ... !*/`. При отладочной сборке (Debug Build) выполняется обязательная статическая верификация; сборка в режиме выпуска (Release Build) возможна только после успешной верификации. При Release Build аннотации спецификаций игнорируются (нулевые накладные расходы), а кэш верификации очищается. Спецификации рассматриваются как часть системы типов, формируя «типы спецификаций» (такие как `Requires(P)`, `Ensures(P)`), и могут быть расширены пользователем. Данный подход обеспечивает высокую надёжность критического кода, сохраняя простоту языка, и идеально интегрируется с единой моделью типов YaoXiang.

## Мотивация

### Зачем нужен данный функционал?

YaoXiang уже обеспечивает безопасность памяти и потоков через модель владения (RFC-009) и модель конкурентности (RFC-001), однако логическая корректность по-прежнему зависит от тестирования. В системном программировании и критически важных областях (аэрокосмическая отрасль, финансы, ядра операционных систем) логические ошибки могут привести к катастрофическим последствиям. Существующие решения (например, проверка заимствований в Rust) не способны обнаружить такие ошибки. Логика Хоара предоставляет математический аппарат доказательства, однако традиционные инструменты формальной верификации обычно требуют отдельного языка спецификаций и имеют крутую кривую обучения.

### Текущие проблемы

- Логическая корректность может быть проверена только тестированием, без гарантий на этапе компиляции
- Для критически важного системного кода отсутствуют средства формальной верификации
- Существующие инструменты формальной верификации имеют крутую кривую обучения и отделены от主流ных языков программирования

## Предложение

### Основная концепция

Наша цель — спроектировать **лёгкое, интегрированное в язык** решение статической верификации:

- **Обязательная верификация при Debug Build**: Разработчики пишут спецификации в модулях, требующих высокой надёжности; при Debug Build верификация обязательна, и только после её успеха возможен переход к Release Build
- **Элегантный синтаксис**: Используются комментарии `//!`, без введения новых ключевых слов; поддерживается подсветка синтаксиса в редакторах
- **Интеграция с системой типов**: Спецификации становятся частью типов, участвуют в проверке типов и поддерживают пользовательские типы спецификаций
- **Доказуемость и тестируемость**: Возможна как статическое доказательство, так и понижение до运行时ных утверждений, что упрощает постепенное внедрение

### 1. Синтаксис аннотаций спецификаций

В начале тела функции или цикла можно записывать спецификации с помощью `//!` (однострочный) или `/*! ... !*/` (многострочный).

#### 1.1 Единый синтаксис спецификаций

Спецификации используют единую модель синтаксиса YaoXiang `name: Type = expression`, полностью интегрированную с системой типов:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0
    //! ensures: GreaterOrEqual(result, arr[0..n])
    //! ensures: ExistsMax(result, arr[0..n])
    // реализация...
}
```

- Спецификация по сути является **объявлением типа**; правая часть — булево выражение
- Левая часть — экземпляр типа спецификации (может содержать параметры типа)
- Доступна специальная переменная `result` для обозначения возвращаемого значения

#### 1.2 Спецификации циклов

```yaoxiang
while i < n {
    /*! invariant: Bounds[i, n] = 0 <= i <= n
                  && SumInvariant[s, arr[0..i]] !*/
    s = s + arr[i]
    i = i + 1
}
```

#### 1.3 Выражения спецификаций

Булевы выражения справа от спецификаций используют синтаксис выражений YaoXiang, поддерживая:

- Арифметические, сравнительные и логические операции
- Кванторы: `forall i in 0..n: P(i)`, `exists i in 0..n: P(i)` — встроенные в язык логические конструкции
- Вызовы функций (только чистые функции)

### 2. Типы спецификаций

Типы спецификаций по сути являются обычными типами YaoXiang, полностью соответствующими единой модели синтаксиса.

#### 2.1 Встроенные типы спецификаций

Компилятор содержит следующие часто используемые типы спецификаций (доступны для прямого использования):

```yaoxiang
// Определения встроенных типов спецификаций
NonEmpty: (T: Type) -> Type = { len: T; len > 0 }
Positive: Type = { x: Int; x > 0 }
GreaterOrEqual: (T: Type) -> Type = { result: T, arr: Array(T); result >= arr[0] && forall i in 1..arr.len: result >= arr[i] }
Bounds: (T: Type) -> Type = { i: T, n: T; 0 <= i && i <= n }
SumInvariant: (T: Type) -> Type = { s: T, arr: Array(T); s == sum(arr[0..i]) }

// Конструкции кванторов (встроены в язык, не являются функциями)
forall: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
exists: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
```

#### 2.2 Пользовательские типы спецификаций

Полностью аналогично обычным определениям типов, пользователи могут определять собственные типы спецификаций:

```yaoxiang
// Определение спецификации положительного целого
Positive: Type = { x: Int; x > 0 }

// Определение спецификации отсортированного массива
Sorted: (T: Ord) -> Type = {
    arr: Array(T);
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

// Определение спецификации существования максимума
ExistsMax: (T: Ord) -> Type = {
    result: T, arr: Array(T);
    exists i in 0..arr.len: result == arr[i]
    && forall j in 0..arr.len: result >= arr[j]
}
```

Использование пользовательских спецификаций:

```yaoxiang
sqrt: (x: Positive) -> Float = {
    //! ensures: SquareRootResult(result, x) = result * result <= x && (result+1)*(result+1) > x
    // реализация...
}

binary_search: (T: Ord) -> ((arr: Sorted(Array(T)), key: T) -> Option(Index)) = {
    //! ensures: SearchResult(result, arr, key)
    // реализация...
}
```

Типы спецификаций, как и другие типы, поддерживают параметры обобщений, ограничения типов и участвуют в выводе типов.

### 3. Режимы компиляции

| Режим | Поведение | Опция |
|-------|-----------|-------|
| **Debug Build** | Разбор спецификаций, генерация условий верификации, вызов SMT-решателя для доказательства; верификация обязательна для перехода к Release Build | `yaoxiangc --debug source.yx` |
| **Release Build** | Игнорирование всех комментариев `//!`, отсутствие генерации кода; очистка всего кэша верификации; включение агрессивных оптимизаций | `yaoxiangc --release source.yx` |
| **Проверка во время выполнения** | Преобразование спецификаций в утверждения во время выполнения, при нарушении — panic | `yaoxiangc --enable-runtime-checks source.yx` |

В режиме верификации, при неудаче доказательства, компилятор сообщает об ошибке и предоставляет возможные контрпримеры (например, входные значения).

### 4. Механизм верификации

Компилятор преобразует спецификации в условия верификации (Verification Conditions) и отправляет их в интегрированный SMT-решатель (например, Z3). Процесс верификации приблизительно следующий:

1. Сбор `requires` и `ensures` функции и `invariant` циклов
2. Для каждого цикла генерация обязательств по доказательству инварианта: выполнение до входа в цикл, сохранение после каждой итерации, при выходе из цикла — импликация постусловия
3. Преобразование тела функции в логическую формулу с объединением спецификаций для формирования условий верификации
4. Вызов SMT-решателя для проверки выполнимости

Если решатель возвращает `unsat` (невыполнимо), спецификация верна; иначе сообщается контрпример.

### 5. Интеграция с тестированием

Режим проверки во время выполнения преобразует спецификации в утверждения для использования в тестах. Совместно с инструментами покрытия спецификаций можно оценить степень покрытия тестами. В будущем возможно рассмотрение инструментов извлечения спецификаций для автоматического вывода候选ных спецификаций из тестов.

### 6. Поддержка редакторами

`//!` и `/*! ... !*/` могут распознаваться редакторами как специальные комментарии, выделяемые другим цветом (например, фиолетовым), для отличия от обычных комментариев. LSP-сервер может предоставлять всплывающие подсказки, автодополнение и отчёты об ошибках верификации для спецификаций.

## Детальное проектирование

### Синтаксические изменения

| До | После |
|----|-------|
| Отсутствует синтаксис аннотаций спецификаций | Разрешены аннотации спецификаций `//!` и `/*! ... !*/` |

### 7.1 Синтаксическое расширение

На основе существующего синтаксиса (RFC-010) допускается появление нуля или более комментариев `//!` или `/*! ... !*/` в начале тела функции или цикла. Синтаксис спецификаций соответствует единому синтаксису типов:

```
spec_comment     ::= ('//!' spec_line) | ('/*!' spec_block '!*/')
spec_line        ::= spec_name ':' type_expr '=' expr
spec_name        ::= 'requires' | 'ensures' | 'invariant'
spec_block       ::= (spec_name ':' type_expr '=' expr ';')*
```

- Спецификация по сути является объявлением типа: `имя_спецификации: тип_спецификации = булево_выражение`
- `type_expr` — выражение типа спецификации, может содержать параметры типа
- `expr` использует синтаксис выражений YaoXiang, поддерживая кванторы

### 7.2 Проверка типов

В режиме верификации компилятор преобразует аннотации спецификаций в соответствующие экземпляры типов спецификаций и записывает их в метаданные функции или цикла.

### 7.3 Генерация условий верификации

Используется исчисление weakest precondition или strongest postcondition совместно с инвариантами циклов для генерации формул первого порядка. Сгенерированные VC используют формат SMT-LIB, вызывается внешний решатель.

### 7.4 Сообщения об ошибках

При неудаче доказательства решатель может предоставить модель (контрпример). Компилятор должен преобразовать эти контрпримеры в читаемую форму, например конкретные значения входных данных, для помощи пользователю в отладке.

### 7.5 Проверка во время выполнения

В режиме `--enable-runtime-checks` компилятор преобразует спецификации в операторы `assert`:

- `requires`: вставка `assert(cond)` в начале функции
- `ensures`: вставка `assert(cond)` перед всеми точками возврата функции, где `result` заменяется фактическим возвращаемым значением
- `invariant`: вставка `assert(cond)` в начале тела цикла

### 7.6 Интеграция с существующим проектированием

- **Модель владения**: Выражения в спецификациях подчиняются правилам владения — могут только читать, но не записывать (чистые функции), что исключает побочные эффекты
- **Система обобщений**: Типы спецификаций поддерживают параметры обобщений (например, `Requires(P)`), могут сочетаться с обобщёнными функциями/типами
- **Зависимые типы**: Значение-зависимые типы в спецификациях (например, длина массива `n`) естественным образом применимы

### Влияние на систему типов

- Типы спецификаций являются обычными типами YaoXiang, соответствующими единой модели синтаксиса
- Компилятор содержит встроенные часто используемые типы спецификаций (`Positive`, `NonEmpty`, `GreaterOrEqual` и др.)
- Пользователи могут определять собственные типы спецификаций через обычные определения типов
- Типы спецификаций могут иметь параметры обобщений и поддерживать ограничения типов

### Поведение во время выполнения

- **Debug Build**: Вызов SMT-решателя для статической верификации, увеличение времени компиляции; после успешной верификации — кэширование результата
- **Release Build**: Аннотации спецификаций игнорируются, нулевые накладные расходы; очистка всего кэша верификации; включение агрессивных оптимизаций, таких как очистка кэша Span
- **Режим проверки во время выполнения**: Генерация операторов assert, обнаружение нарушений во время выполнения

### Изменения в компиляторе

- Парсер: распознавание синтаксиса аннотаций спецификаций
- Семантический анализ: сбор спецификаций, преобразование в типы спецификаций
- Верификационный backend: генерация условий верификации, вызов SMT-решателя
- Генерация кода: поддержка режима проверки во время выполнения

### Обратная совместимость

- ✅ Полная обратная совместимость
- Аннотации спецификаций при обычной компиляции игнорируются, не влияют на существующий код
- При Release Build спецификации игнорируются, отсутствуют дополнительные накладные расходы

## Компромиссы

### Преимущества

- **Верификация при Debug Build**: Обязательная верификация при Debug Build обеспечивает логическую корректность
- **Элегантный синтаксис**: Чистые комментарии, без новых ключевых слов, удобство в редакторах
- **Интеграция с системой типов**: Спецификации — это типы, расширяемость
- **Постепенное внедрение**: Возможен постепенный переход от проверок во время выполнения к статической верификации
- **Повышение надёжности**: Обнаружение логических ошибок, труднодоступных для тестирования

### Недостатки

- **Время компиляции**: Режим верификации может значительно увеличить время компиляции
- **Кривая обучения**: Необходимость изучения написания эффективных спецификаций и кванторов
- **Ограничения SMT-решателя**: Некоторые сложные свойства могут не поддаваться автоматическому доказательству

## Альтернативные решения

| Решение | Преимущества | Недостатки |
|---------|--------------|------------|
| Новые ключевые слова (например, `requires`) | Интуитивный синтаксис | Введение новых ключевых слов, нарушение простоты |
| Отдельные файлы спецификаций (например, CVL) | Разделение спецификаций и кода | Увеличение количества файлов, сложность синхронизации |
| Только утверждения во время выполнения | Простая реализация | Невозможность статической гарантии |
| **Наше решение (комментарии + типы спецификаций)** | Баланс простоты и функциональности | Требуется поддержка редакторами |

## Стратегия реализации

### Фазы

| Фаза | Содержание |
|------|------------|
| **Фаза 1: Базовая поддержка** | Расширение парсера для распознавания комментариев `//!` и `/*! ... !*/`, прикрепление их к AST-узлам; в режиме верификации — сбор спецификаций, генерация простых условий верификации (только арифметические сравнения); интеграция решателя Z3 |
| **Фаза 2: Поддержка кванторов** | Поддержка выражений с кванторами, трансляция в `forall`/`exists` SMT-LIB; подсветка и всплывающие подсказки для спецификаций в IDE |
| **Фаза 3: Оптимизация и инструментарий** | Инкрементная верификация, кэширование уже проверенных модулей; отчёт о покрытии спецификаций; инструмент извлечения спецификаций (генерация候选ных спецификаций из тестов) |

### Зависимости

- RFC-009: Модель владения — выражения в спецификациях требуют семантики чистых функций
- RFC-010: Единый синтаксис типов — система типов спецификаций основана на системе типов
- RFC-011: Проектирование системы обобщённых типов — типы спецификаций поддерживают параметры обобщений

### Риски

1. **Сложность интеграции SMT-решателя**: Интеграция решателей типа Z3 может столкнуться с техническими трудностями
   - Смягчение: Использование зрелых Rust-привязок к Z3, постепенное расширение支持的 типов выражений

2. **Сложность отладки при неудаче верификации**: Когда SMT-решатель не может доказать спецификацию, пользователю может быть трудно понять причину
   - Смягчение: Предоставление понятных сообщений об ошибках и объяснений контрпримеров

3. **Накладные расходы на производительность**: Режим верификации может значительно увеличить время компиляции
   - Смягчение: Реализация инкрементной верификации и механизмов кэширования

```
┌─────────────┐
│   Черновик  │  ← Создано автором
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ На рассмотрении │  ← Обсуждение сообществом
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  Принято    │    │  Отклонено  │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│ accepted/   │    │    rfc/    │
│ (официальный│    │(остаётся на │
│  дизайн)    │    │   месте)    │
└─────────────┘    └─────────────┘
```

### Описание статусов

| Статус | Расположение | Описание |
|--------|--------------|----------|
| **Черновик** | `docs/design/rfc/` | Черновик автора, ожидает подачи на рассмотрение |
| **На рассмотрении** | `docs/design/rfc/` | Открыто обсуждение и обратная связь сообщества |
| **Принято** | `docs/design/accepted/` | Становится официальным документом дизайна, переходит в фазу реализации |
| **Отклонено** | `docs/design/rfc/` | Остаётся в каталоге RFC, статус обновлён |

### Действия после принятия

1. Переместить RFC в каталог `docs/design/accepted/`
2. Обновить имя файла на описательное название (например, `hoare-logic-static-verification.md`)
3. Обновить статус на «Официальный»
4. Обновить статус на «Принято», добавить дату принятия

### Действия после отклонения

1. Оставить в каталоге `docs/design/rfc/`
2. В начале файла добавить причину отклонения и дату
3. Обновить статус на «Отклонено»
