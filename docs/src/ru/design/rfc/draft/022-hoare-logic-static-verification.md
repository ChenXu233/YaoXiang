# RFC 022: Статическая верификация на основе логики Хора (спецификационные комментарии и спецификационные типы)

> **Статус**: На рассмотрении

> **Последнее обновление**: 2026-03-16

> - [RFC-011: Проектирование системы обобщённых типов](./011-generic-type-system.md)

## Аннотация

В данном документе предлагается внедрить в язык YaoXiang **механизм статической верификации на основе логики Хора**, позволяющий разработчикам записывать предусловия, постусловия и инварианты циклов в комментариях в форме `//!` или `/*! ... !*/`. При сборке в режиме Debug верификация выполняется принудительно, и только после её успешного завершения возможна сборка в режиме Release; при сборке в режиме Release спецификационные комментарии игнорируются (нулевые накладные расходы), а кэш верификации очищается. Сами спецификации рассматриваются как часть системы типов, формируя «спецификационные типы» (такие как `Requires(P)`, `Ensures(P)`), и могут расширяться пользователем. Данный подход призван сохранить лаконичность языка, обеспечивая при этом высокую надёжность критически важного кода, и完美но интегрируется с унифицированной моделью типов YaoXiang.

## Мотивация

### Зачем нужна данная возможность/изменение?

YaoXiang уже обеспечивает безопасность памяти и потоков через модель владения (RFC-009) и модель конкурентности (RFC-001), однако логическая корректность по-прежнему зависит от тестирования. В системном программировании и областях с критическими требованиями к безопасности (аэрокосмическая отрасль, финансы, ядра операционных систем) логические ошибки могут привести к катастрофическим последствиям. Существующие решения (например, проверка заимствований в Rust) не способны обнаруживать подобные ошибки. Логика Хора предоставляет математический инструмент доказательства, однако традиционные средства формальной верификации, как правило, требуют отдельного языка спецификаций и связаны со сложной кривой обучения.

### Текущие проблемы

- Логическая корректность может быть проверена только тестированием, без гарантий на этапе компиляции
- Для критически важного системного кода отсутствуют средства формальной верификации
- Существующие инструменты формальной верификации имеют крутую кривую обучения и отделены от мейнстримных языков программирования

## Предложение

### Основная концепция

Наша цель — спроектировать **лёгкое, интегрированное с языком** решение для статической верификации:

- **Верификация обязательна при Debug Build**: разработчики пишут спецификации в модулях, требующих высокой надёжности; при Debug Build верификация выполняется принудительно, и только после её успеха возможна сборка в режиме Release
- **Элегантный синтаксис**: используются комментарии `//!`, без введения новых ключевых слов; поддерживается подсветка синтаксиса в редакторах
- **Интеграция с системой типов**: спецификации становятся частью типа, участвуют в проверке типов и поддерживают пользовательские спецификационные типы
- **Доказуемость и тестируемость**: возможна как статическая проверка, так и понижение до runtime-утверждений для постепенного внедрения

### 1. Синтаксис спецификационных комментариев

В начале тела функции или цикла можно записывать спецификации с помощью `//!` (однострочный) или `/*! ... !*/` (многострочный).

#### 1.1 Унифицированный синтаксис спецификаций

Спецификации используют унифицированную модель синтаксиса YaoXiang `name: Type = expression`, полностью согласованную с системой типов:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0
    //! ensures: GreaterOrEqual(result, arr[0..n])
    //! ensures: ExistsMax(result, arr[0..n])
    // реализация...
}
```

- Спецификация по своей сути является **объявлением типа**, справа — булево выражение
- Слева находится экземпляр спецификационного типа (может иметь параметры типа)
- Можно использовать специальную переменную `result` для обозначения возвращаемого значения

#### 1.2 Спецификации циклов

```yaoxiang
while i < n {
    /*! invariant: Bounds[i, n] = 0 <= i <= n
                  && SumInvariant[s, arr[0..i]] !*/
    s = s + arr[i]
    i = i + 1
}
```

#### 1.3 Спецификационные выражения

Булевы выражения справа от спецификаций используют синтаксис выражений YaoXiang и поддерживают:
- Арифметические, сравнительные и логические операции
- Кванторы: `forall i in 0..n: P(i)`, `exists i in 0..n: P(i)` — встроенные в язык логические конструкции
- Вызовы функций (только чистые функции)

### 2. Система спецификационных типов

Спецификационные типы по своей сути являются обычными типами YaoXiang, полностью соответствующими унифицированной модели синтаксиса.

#### 2.1 Встроенные спецификационные типы

Компилятор предоставляет следующие часто используемые спецификационные типы (можно использовать напрямую в спецификациях):

```yaoxiang
// Встроенные спецификационные типы
NonEmpty: (T: Type) -> Type = { len: T; len > 0 }
Positive: Type = { x: Int; x > 0 }
GreaterOrEqual: (T: Type) -> Type = { result: T, arr: Array(T); result >= arr[0] && forall i in 1..arr.len: result >= arr[i] }
Bounds: (T: Type) -> Type = { i: T, n: T; 0 <= i && i <= n }
SumInvariant: (T: Type) -> Type = { s: T, arr: Array(T); s == sum(arr[0..i]) }

// Конструкции кванторов (встроены в язык, не являются функциями)
forall: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
exists: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
```

#### 2.2 Пользовательские спецификационные типы

Полностью аналогично обычным определениям типов, пользователи могут определять собственные спецификационные типы:

```yaoxiang
// Определение спецификационного типа для положительного целого
Positive: Type = { x: Int; x > 0 }

// Определение спецификационного типа для отсортированного массива
Sorted: (T: Ord) -> Type = {
    arr: Array(T);
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

// Определение спецификационного типа для существования максимума
ExistsMax: (T: Ord) -> Type = {
    result: T, arr: Array(T);
    exists i in 0..arr.len: result == arr[i]
    && forall j in 0..arr.len: result >= arr[j]
}
```

Использование пользовательских спецификаций:

```yaoxiang
sqrt: (x: Positive) -> Float = {
    //! ensures: SquareRootResult(result, x) = result * result <= x && (result+1)*(result+1) > x
    // реализация...
}

binary_search: (T: Ord) -> ((arr: Sorted(Array(T)), key: T) -> Option(Index)) = {
    //! ensures: SearchResult(result, arr, key)
    // реализация...
}
```

Спецификационные типы, как и другие типы, поддерживают обобщённые параметры, ограничения типов и участвуют в выводе типов.

### 3. Режимы компиляции

| Режим | Поведение | Опция |
|------|------|------|
| **Debug Build** | Разбор спецификаций, генерация условий верификации, вызов SMT-решателя для доказательства; верификация должна пройти успешно перед Release Build | `yaoxiangc --debug source.yx` |
| **Release Build** | Игнорирование всех комментариев `//!`, отсутствие генерации кода; очистка всего кэша верификации; включение агрессивных оптимизаций | `yaoxiangc --release source.yx` |
| **Runtime-проверки** | Преобразование спецификаций в runtime-утверждения, при нарушении — panic | `yaoxiangc --enable-runtime-checks source.yx` |

В режиме верификации, если доказательство не удалось, компилятор сообщит об ошибке и предоставит возможный контрпример (например, входные значения).

### 4. Механизм верификации

Компилятор преобразует спецификации в условия верификации (Verification Conditions) и отправляет их в интегрированный SMT-решатель (например, Z3). Процесс верификации大致如下:

1. Сбор `requires` и `ensures` для функций, `invariant` для циклов
2. Для каждого цикла генерируются обязательства по доказательству инварианта цикла: выполняется перед входом в цикл, сохраняется после каждой итерации, при выходе из цикла влечёт постусловие
3. Тело функции преобразуется в логические формулы, объединяется со спецификациями, формируя условия верификации
4. Вызов SMT-решателя для проверки выполнимости

Если решатель возвращает `unsat` (невыполнимо), спецификация верна; иначе сообщается контрпример.

### 5. Интеграция с тестированием

Режим runtime-проверок преобразует спецификации в утверждения для тестирования. В сочетании с инструментами покрытия спецификаций можно оценить степень покрытия тестами. В будущем возможно рассмотрение инструментов вывода спецификаций, автоматически извлекающих candidate-спецификации из тестов.

### 6. Поддержка в редакторах

`//!` и `/*! ... !*/` могут распознаваться редакторами как особые комментарии с отличной подсветкой (например, фиолетовый цвет), отличающейся от обычных комментариев. Language Server может предоставлять hover-подсказки, автодополнение и отчёты об ошибках верификации для спецификаций.

## Детальное проектирование

### Синтаксические изменения

| До | После |
|------|------|
| Нет синтаксиса спецификационных комментариев | Разрешены комментарии-спецификации `//!` и `/*! ... !*/` |

### 7.1 Расширение синтаксиса

На основе существующего синтаксиса (RFC-010) разрешается появление нуля или более комментариев `//!` или `/*! ... !*/` в начале тела функции и цикла. Синтаксис спецификаций согласован с унифицированным синтаксисом типов:

```
spec_comment     ::= ('//!' spec_line) | ('/*!' spec_block '!*/')
spec_line        ::= spec_name ':' type_expr '=' expr
spec_name        ::= 'requires' | 'ensures' | 'invariant'
spec_block       ::= (spec_name ':' type_expr '=' expr ';')*
```

- Спецификация по сути является объявлением типа: `имя_спецификации: спецификационный_тип = булево_выражение`
- `type_expr` — выражение спецификационного типа, может содержать параметры типа
- `expr` использует синтаксис выражений YaoXiang, поддерживает кванторы

### 7.2 Проверка типов

В режиме верификации компилятор преобразует спецификационные комментарии в соответствующие экземпляры спецификационных типов и записывает их в метаданные функции или цикла.

### 7.3 Генерация условий верификации

Используется исчисление weakest precondition или strongest postcondition совместно с инвариантами циклов для генерации формул первого порядка. Сгенерированные VC используют формат SMT-LIB, вызывается внешний решатель.

### 7.4 Сообщения об ошибках

Если доказательство не удалось, решатель может предоставить модель (контрпример). Компилятор должен преобразовать эти контрпримеры в читаемую форму, например конкретные значения входных данных, помогая пользователю в отладке.

### 7.5 Runtime-проверки

В режиме `--enable-runtime-checks` компилятор преобразует спецификации в операторы `assert`:
- `requires`: вставка `assert(cond)` в начале функции
- `ensures`: вставка `assert(cond)` перед всеми точками возврата функции, где `result` заменён на фактическое возвращаемое значение
- `invariant`: вставка `assert(cond)` в начале тела цикла

### 7.6 Интеграция с существующим дизайном

- **Модель владения**: выражения в спецификациях подчиняются правилам владения — только чтение, без записи (чистые функции), без побочных эффектов
- **Обобщённая система**: спецификационные типы поддерживают обобщённые параметры (например, `Requires(P)`), могут сочетаться с обобщёнными функциями/типами
- **Зависимые типы**: значения-зависимые типы в спецификациях (например, длина массива `n`) используются естественно

### Влияние на систему типов

- Спецификационные типы являются обычными типами YaoXiang, согласованными с унифицированной моделью синтаксиса
- Компилятор предоставляет встроенные часто используемые спецификационные типы (`Positive`, `NonEmpty`, `GreaterOrEqual` и др.)
- Пользователи могут определять собственные спецификационные типы обычным определением типов
- Спецификационные типы могут иметь обобщённые параметры и поддерживают ограничения типов

### Поведение во время выполнения

- **Debug Build**: вызов SMT-решателя для статической верификации, увеличение времени компиляции; после успешной верификации кэширование результатов
- **Release Build**: спецификационные комментарии игнорируются, нулевые накладные расходы во время выполнения; очистка всего кэша верификации; включение агрессивных оптимизаций, таких как очистка кэша Span
- **Режим runtime-проверок**: генерация операторов assert, обнаружение нарушений во время выполнения

### Изменения в компиляторе

- Парсер: распознавание синтаксиса спецификационных комментариев
- Семантический анализ: сбор спецификаций, преобразование в спецификационные типы
- Backend верификации: генерация условий верификации, вызов SMT-решателя
- Генерация кода: поддержка режима runtime-проверок

### Обратная совместимость

- ✅ Полная обратная совместимость
- Спецификационные комментарии игнорируются при обычной компиляции, не влияют на существующий код
- В режиме Release Build спецификации игнорируются, отсутствуют дополнительные накладные расходы

## Компромиссы

### Преимущества

- **Верификация при Debug Build**: принудительная верификация в режиме Debug обеспечивает логическую корректность
- **Элегантный синтаксис**: только комментарии, без новых ключевых слов, удобно для редакторов
- **Интеграция с системой типов**: спецификации являются типами, расширяемы
- **Постепенное внедрение**: возможен переход от runtime-проверок к статической верификации
- **Повышение надёжности**: способность обнаруживать логические ошибки, которые трудно найти тестированием

### Недостатки

- **Время компиляции**: режим верификации может значительно увеличить время компиляции
- **Кривая обучения**: требуется изучение эффективного написания спецификаций и кванторов
- **Ограничения SMT-решателей**: некоторые сложные свойства могут не поддаваться автоматическому доказательству

## Альтернативные решения

| Решение | Преимущества | Недостатки |
|------|------|------|
| Новые ключевые слова (например, `requires`) | Интуитивный синтаксис | Введение новых ключевых слов, нарушение лаконичности |
| Отдельные файлы спецификаций (например, CVL) | Разделение спецификаций и кода | Увеличение количества файлов, сложность синхронизации |
| Только runtime-утверждения | Простая реализация | Отсутствие статических гарантий |
| **Наше решение (комментарии + спецификационные типы)** | Баланс лаконичности и функциональности | Требуется поддержка редакторов |

## Стратегия реализации

### Фазы

| Фаза | Содержание |
|------|------|
| **Фаза 1: Базовая поддержка** | Расширение парсера для распознавания комментариев `//!` и `/*! ... !*/`, прикрепление их к узлам AST; в режиме верификации сбор спецификаций, генерация простых условий верификации (только арифметические сравнения); интеграция Z3-решателя |
| **Фаза 2: Поддержка кванторов** | Поддержка выражений с кванторами, трансляция в `forall`/`exists` в формате SMT-LIB; подсветка спецификаций в IDE и hover-подсказки |
| **Фаза 3: Оптимизация и инструментарий** | Инкрементальная верификация, кэширование уже проверенных модулей; отчёты о покрытии спецификаций; инструменты вывода спецификаций (генерация candidate-спецификаций из тестов) |

### Зависимости

- RFC-009: Модель владения — выражения в спецификациях требуют семантики чистых функций
- RFC-010: Унифицированный синтаксис типов — система спецификационных типов основана на системе типов
- RFC-011: Проектирование системы обобщённых типов — спецификационные типы поддерживают обобщённые параметры

### Риски

1. **Сложность интеграции SMT-решателя**: интеграция Z3 и подобных решателей может столкнуться с техническими трудностями
   - Смягчение: использование зрелых Rust-привязок к Z3, постепенное расширение支持的 выражений

2. **Сложность отладки при сбое верификации**: когда SMT-решатель не может доказать спецификацию, пользователю может быть трудно понять причину
   - Смягчение: предоставление чётких сообщений об ошибках и пояснений к контрпримерам

3. **Производительность**: режим верификации может значительно увеличить время компиляции
   - Смягчение: реализация инкрементальной верификации и механизма кэширования

### Пояснение к статусам

| Статус | Расположение | Описание |
|------|------|------|
| **Черновик** | `docs/design/rfc/` | Черновик автора, ожидает подачи на рассмотрение |
| **На рассмотрении** | `docs/design/rfc/` | Открыто обсуждение и обратная связь сообщества |
| **Принято** | `docs/design/accepted/` | Становится официальным документом дизайна, переходит в фазу реализации |
| **Отклонено** | `docs/design/rfc/` | Остаётся в директории RFC, статус обновлён |

### Действия после принятия

1. Переместить RFC в директорию `docs/design/accepted/`
2. Обновить имя файла на описательное (например, `hoare-logic-static-verification.md`)
3. Обновить статус на «Официальный»
4. Обновить статус на «Принято», добавить дату принятия

### Действия после отклонения

1. Оставить в директории `docs/design/rfc/`
2. Добавить в начало файла причину отклонения и дату
3. Обновить статус на «Отклонено»
