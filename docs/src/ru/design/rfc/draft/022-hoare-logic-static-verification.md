# RFC 022: Поддержка статической верификации на основе логики Хоара (спецификационные комментарии и спецификационные типы)

## Аннотация

В данном документе предлагается введение в язык YaoXiang **механизма статической верификации на основе логики Хоара**, позволяющего разработчикам записывать предусловия, постусловия и инварианты циклов в комментариях с использованием `//!` или `/*! ... !*/`. При сборке в режиме Debug выполняется обязательная статическая верификация; переход к Release Build возможен только после успешной верификации. При Release Build спецификационные комментарии игнорируются (нулевые накладные расходы), а кэш верификации очищается. Спецификации рассматриваются как часть системы типов, формируя «спецификационные типы» (такие как `Requires(P)`, `Ensures(P)`), и могут быть расширены пользователем. Данный подход обеспечивает высокую надёжность критически важного кода, сохраняя при этом простоту языка, и идеально интегрируется с унифицированной моделью типов YaoXiang.

## Мотивация

### Зачем нужен данный функционал?

Язык YaoXiang уже обеспечивает безопасность памяти и потоков через модель владения (RFC-009) и модель параллелизма (RFC-001), однако логическая корректность по-прежнему зависит от тестирования. В системном программировании и критически важных областях (аэрокосмическая отрасль, финансы, ядра операционных систем) логические ошибки могут привести к катастрофическим последствиям. Существующие механизмы (например, проверка заимствований в Rust) не способны обнаружить подобные ошибки. Логика Хоара предоставляет математический аппарат доказательства, однако традиционные инструменты формальной верификации требуют отдельного языка спецификаций и имеют крутую кривую обучения.

### Текущие проблемы

- Логическая корректность может быть проверена только тестированием, без гарантий на этапе компиляции
- Для критически важного системного кода отсутствуют средства формальной верификации
- Существующие инструменты формальной верификации имеют крутую кривую обучения и отделены от主流ных языков программирования

## Предложение

### Основная концепция

Наша цель — спроектировать **лёгкое, интегрированное в язык** решение статической верификации:

- **Обязательная верификация в Debug Build**: Разработчики пишут спецификации в модулях, требующих высокой надёжности; при Debug Build верификация обязательна для перехода к Release Build
- **Элегантный синтаксис**: Использование `//!` комментариев без введения новых ключевых слов; поддержка подсветки синтаксиса в редакторах
- **Интеграция с системой типов**: Спецификации становятся частью типа, участвуют в проверке типов; поддержка пользовательских спецификационных типов
- **Доказуемость и тестируемость**: Как статическое доказательство, так и понижение до runtime-ассертов для постепенного внедрения

### 1. Синтаксис спецификационных комментариев

В начале тела функции или цикла используются `//!` (однострочный) или `/*! ... !*/` (многострочный) комментарии для записи спецификаций.

#### 1.1 Унифицированный синтаксис спецификаций

Спецификации используют унифицированную модель синтаксиса YaoXiang `name: Type = expression`, полностью интегрированную с системой типов:

```yaoxiang
max: (T: Ord) -> ((arr: Array(T, n)) -> T) = {
    //! requires: NonEmpty(n) = n > 0
    //! ensures: GreaterOrEqual(result, arr[0..n])
    //! ensures: ExistsMax(result, arr[0..n])
    // Реализация...
}
```

- Спецификация по сути является **объявлением типа** с булевым выражением справа
- Слева — экземпляр спецификационного типа (с параметрами типа)
- Доступна специальная переменная `result` для обозначения возвращаемого значения

#### 1.2 Спецификации циклов

```yaoxiang
while i < n {
    /*! invariant: Bounds[i, n] = 0 <= i <= n
                  && SumInvariant[s, arr[0..i]] !*/
    s = s + arr[i]
    i = i + 1
}
```

#### 1.3 Выражения спецификаций

Булевы выражения справа от спецификаций используют синтаксис выражений YaoXiang, поддерживая:
- Арифметические, сравнительные и логические операции
- Кванторы: `forall i in 0..n: P(i)`, `exists i in 0..n: P(i)` — встроенные в язык логические конструкции
- Вызовы функций (только чистые функции)

### 2. Система спецификационных типов

Спецификационные типы по сути являются обычными типами YaoXiang, полностью соответствующими унифицированной модели синтаксиса.

#### 2.1 Встроенные спецификационные типы

Компилятор предоставляет следующие часто используемые спецификационные типы (доступные непосредственно в спецификациях):

```yaoxiang
// Определения встроенных спецификационных типов
NonEmpty: (T: Type) -> Type = { len: T; len > 0 }
Positive: Type = { x: Int; x > 0 }
GreaterOrEqual: (T: Type) -> Type = { result: T, arr: Array(T); result >= arr[0] && forall i in 1..arr.len: result >= arr[i] }
Bounds: (T: Type) -> Type = { i: T, n: T; 0 <= i && i <= n }
SumInvariant: (T: Type) -> Type = { s: T, arr: Array(T); s == sum(arr[0..i]) }

// Кванторные конструкции (встроены в язык, не являются функциями)
forall: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
exists: (start: Int, end: Int, pred: (Int) -> Bool) -> Bool
```

#### 2.2 Пользовательские спецификационные типы

Полностью аналогично обычным определениям типов, пользователи могут определять собственные спецификационные типы:

```yaoxiang
// Определение спецификации положительного числа
Positive: Type = { x: Int; x > 0 }

// Определение спецификации отсортированного массива
Sorted: (T: Ord) -> Type = {
    arr: Array(T);
    forall i in 0..arr.len-1: arr[i] <= arr[i+1]
}

// Определение спецификации существования максимума
ExistsMax: (T: Ord) -> Type = {
    result: T, arr: Array(T);
    exists i in 0..arr.len: result == arr[i]
    && forall j in 0..arr.len: result >= arr[j]
}
```

Использование пользовательских спецификаций:

```yaoxiang
sqrt: (x: Positive) -> Float = {
    //! ensures: SquareRootResult(result, x) = result * result <= x && (result+1)*(result+1) > x
    // Реализация...
}

binary_search: (T: Ord) -> ((arr: Sorted(Array(T)), key: T) -> Option(Index)) = {
    //! ensures: SearchResult(result, arr, key)
    // Реализация...
}
```

Спецификационные типы, как и другие типы, поддерживают параметры дженериков, ограничения типов и участвуют в выводе типов.

### 3. Режимы компиляции

| Режим | Поведение | Опция |
|------|------|------|
| **Debug Build** | Разбор спецификаций, генерация условий верификации, вызов SMT-солвера для доказательства; верификация обязательна для перехода к Release Build | `yaoxiangc --debug source.yx` |
| **Release Build** | Игнорирование всех `//!` комментариев, отсутствие генерации кода; очистка всего кэша верификации; агрессивная оптимизация | `yaoxiangc --release source.yx` |
| **Runtime-проверки** | Преобразование спецификаций в runtime-ассерты, нарушение вызывает panic | `yaoxiangc --enable-runtime-checks source.yx` |

В режиме верификации, если доказательство не удалось, компилятор сообщает об ошибке и предоставляет возможный контрпример (например, входные значения).

### 4. Механизм верификации

Компилятор преобразует спецификации в условия верификации (Verification Conditions) и отправляет их встроенному SMT-солверу (например, Z3). Процесс верификации大致如下:

1. Сбор `requires` и `ensures` функции и `invariant` цикла
2. Генерация обязательств по доказательству инварианта цикла: верность перед входом в цикл, сохранение после каждой итерации, импликация постусловия при выходе из цикла
3. Преобразование тела функции в логические формулы, объединение со спецификациями, формирование условий верификации
4. Вызов SMT-солвера для проверки выполнимости

Если солвер возвращает `unsat` (невыполнимо), спецификация верна; иначе сообщается контрпример.

### 5. Интеграция с тестированием

Режим runtime-проверок преобразует спецификации в ассерты для использования в тестах. В сочетании с инструментами покрытия спецификаций можно оценить степень покрытия тестами. В будущем возможно рассмотрение инструментов извлечения спецификаций, автоматически выводящихcandidate-спецификации из тестов.

### 6. Поддержка редакторами

`//!` и `/*! ... !*/` могут распознаваться редакторами как специальные комментарии с отличной подсветкой (например, фиолетовым цветом). Language Server может предоставлять hover-подсказки, автодополнение и отчёты об ошибках верификации для спецификаций.

## Детальное проектирование

### Синтаксические изменения

| До | После |
|------|------|
| Нет синтаксиса спецификационных комментариев | Разрешены `//!` и `/*! ... !*/` спецификационные комментарии |

### 7.1 Синтаксическое расширение

На основе существующего синтаксиса (RFC-010) разрешается появление нуля или более `//!` или `/*! ... !*/` комментариев в начале тела функции или цикла. Синтаксис спецификаций соответствует унифицированному синтаксису типов:

```
spec_comment     ::= ('//!' spec_line) | ('/*!' spec_block '!*/')
spec_line        ::= spec_name ':' type_expr '=' expr
spec_name        ::= 'requires' | 'ensures' | 'invariant'
spec_block       ::= (spec_name ':' type_expr '=' expr ';')*
```

- Спецификация по сути является объявлением типа: `имя_спецификации: спецификационный_тип = булево_выражение`
- `type_expr` — выражение спецификационного типа с параметрами
- `expr` использует синтаксис выражений YaoXiang с поддержкой кванторов

### 7.2 Проверка типов

В режиме верификации компилятор преобразует спецификационные комментарии в соответствующие экземпляры спецификационных типов и записывает их в метаданные функции или цикла.

### 7.3 Генерация условий верификации

Используется исчисление weakest precondition или strongest postcondition совместно с инвариантами циклов для генерации формул первого порядка. Сгенерированные VC используют формат SMT-LIB с вызовом внешнего солвера.

### 7.4 Сообщения об ошибках

Если доказательство не удалось, солвер может предоставить модель (контрпример). Компилятор должен преобразовать эти контрпримеры в читаемую форму, например конкретные входные значения, для помощи пользователю в отладке.

### 7.5 Runtime-проверки

В режиме `--enable-runtime-checks` компилятор преобразует спецификации в операторы `assert`:
- `requires`: вставка `assert(cond)` в начале функции
- `ensures`: вставка `assert(cond)` перед всеми точками возврата функции с заменой `result` на фактическое возвращаемое значение
- `invariant`: вставка `assert(cond)` в начале тела цикла

### 7.6 Интеграция с существующим дизайном

- **Модель владения**: Выражения в спецификациях подчиняются правилам владения, допускается только чтение (чистые функции), без побочных эффектов
- **Система дженериков**: Спецификационные типы поддерживают параметры дженериков (например, `Requires(P)`), могут комбинироваться с обобщёнными функциями/типами
- **Зависимые типы**: Значения, зависящие от типов в спецификациях (например, длина массива `n`), естественно применимы

### Влияние на систему типов

- Спецификационные типы являются обычными типами YaoXiang, соответствующими унифицированной модели синтаксиса
- Компилятор предоставляет встроенные часто используемые спецификационные типы (`Positive`, `NonEmpty`, `GreaterOrEqual` и др.)
- Пользователи могут определять собственные спецификационные типы через обычные определения типов
- Спецификационные типы поддерживают параметры дженериков и ограничения типов

### Поведение во время выполнения

- **Debug Build**: Вызов SMT-солвера для статической верификации, увеличение времени компиляции; после успешной верификации кэширование результатов
- **Release Build**: Спецификационные комментарии игнорируются, нулевые накладные расходы во время выполнения; очистка всего кэша верификации; агрессивные оптимизации, включая очистку кэша Span
- **Режим runtime-проверок**: Генерация assert-операторов, обнаружение нарушений во время выполнения

### Изменения в компиляторе

- Парсер: распознавание синтаксиса спецификационных комментариев
- Семантический анализ: сбор спецификаций, преобразование в спецификационные типы
- Backend верификации: генерация условий верификации, вызов SMT-солвера
- Генерация кода: поддержка режима runtime-проверок

### Обратная совместимость

- ✅ Полная обратная совместимость
- Спецификационные комментарии игнорируются при обычной компиляции, не влияют на существующий код
- В Release Build спецификации игнорируются, отсутствуют дополнительные накладные расходы

## Компромиссы

### Преимущества

- **Верификация в Debug Build**: Обязательная верификация при Debug Build обеспечивает логическую корректность
- **Элегантный синтаксис**: Чистые комментарии без новых ключевых слов, дружелюбность к редакторам
- **Интеграция с системой типов**: Спецификации как типы, расширяемость
- **Постепенное внедрение**: Переход от runtime-проверок к статической верификации
- **Повышение надёжности**: Обнаружение логических ошибок, трудновыявляемых тестированием

### Недостатки

- **Время компиляции**: Режим верификации может значительно увеличить время компиляции
- **Кривая обучения**: Необходимость изучения эффективного написания спецификаций и кванторов
- **Ограничения SMT-солверов**: Некоторые сложные свойства могут не поддаваться автоматическому доказательству

## Альтернативные решения

| Решение | Преимущества | Недостатки |
|------|------|------|
| Новые ключевые слова (например, `requires`) | Интуитивный синтаксис | Введение новых ключевых слов, нарушение простоты |
| Отдельные файлы спецификаций (например, CVL) | Разделение спецификаций и кода | Увеличение количества файлов, сложность синхронизации |
| Только runtime-ассерты | Простота реализации | Отсутствие статических гарантий |
| **Наше решение (комментарии + спецификационные типы)** | Баланс простоты и функциональности | Требуется поддержка редакторами |

## Стратегия реализации

### Фазы

| Фаза | Содержание |
|------|------|
| **Фаза 1: Базовая поддержка** | Расширение парсера для распознавания `//!` и `/*! ... !*/` комментариев, присоединение к AST-узлам; в режиме верификации сбор спецификаций, генерация простых условий верификации (только арифметические сравнения); интеграция Z3-солвера |
| **Фаза 2: Поддержка кванторов** | Поддержка кванторных выражений, трансляция в `forall`/`exists` формата SMT-LIB; подсветка IDE и hover-подсказки для спецификаций |
| **Фаза 3: Оптимизация и инструментарий** | Инкрементальная верификация, кэширование уже верифицированных модулей; отчёты о покрытии спецификаций; инструменты извлечения спецификаций (генерация candidate-спецификаций из тестов) |

### Зависимости

- RFC-009: Модель владения — выражения спецификаций требуют семантики чистых функций
- RFC-010: Унифицированный синтаксис типов — система спецификационных типов основана на системе типов
- RFC-011: Проектирование системы дженериков — спецификационные типы поддерживают параметры дженериков

### Риски

1. **Сложность интеграции SMT-солверов**: Интеграция Z3 и других солверов может столкнуться с техническими трудностями
   - Смягчение: использование зрелых Rust-привязок к Z3, постепенное расширение поддерживаемых типов выражений

2. **Сложность отладки при неудаче верификации**: Когда SMT-солвер не может доказать спецификацию, пользователю может быть трудно понять причину
   - Смягчение: предоставление чётких сообщений об ошибках и объяснений контрпримеров

3. **Накладные расходы на производительность**: Режим верификации может значительно увеличить время компиляции
   - Смягчение: реализация инкрементальной верификации и механизмов кэширования

## Жизненный цикл и судьба

```
┌─────────────┐
│   Черновик   │  ← Создано автором
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ На рассмотрении │  ← Обсуждение сообщества
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  Принято    │    │  Отклонено  │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│ accepted/  │    │    rfc/     │
│ (официальный дизайн) │    │ (сохраняется на месте) │
└─────────────┘    └─────────────┘
```

### Описание статусов

| Статус | Расположение | Описание |
|------|------|------|
| **Черновик** | `docs/design/rfc/` | Черновик автора, ожидает подачи на рассмотрение |
| **На рассмотрении** | `docs/design/rfc/` | Открыто обсуждение и обратная связь сообщества |
| **Принято** | `docs/design/accepted/` | Становится официальным документом дизайна, переходит в фазу реализации |
| **Отклонено** | `docs/design/rfc/` | Сохраняется в каталоге RFC, обновляется статус |

### Действия после принятия

1. Переместить RFC в каталог `docs/design/accepted/`
2. Обновить имя файла на описательное (например, `hoare-logic-static-verification.md`)
3. Обновить статус на "Официальный"
4. Обновить статус на "Принято", добавить дату принятия

### Действия после отклонения

1. Сохранить в каталоге `docs/design/rfc/`
2. Добавить в начало файла причину отклонения и дату
3. Обновить статус на "Отклонено"
