# RFC-034: Унифицированный инструментарий отладки

## Резюме

Введение в YaoXiang унифицированного инструментария отладки. Ключевая идея проекта — **один источник, три потребителя**: компиляторный фронтенд встраивает позиции исходного кода, имена переменных и информацию о типах как сущности первого класса в YaoXiang IR; три бэкенда — интерпретатор, JIT и LLVM — каждый потребляют один и тот же набор метаданных. Пользователь запускает DAP-сервер (Debug Adapter Protocol) командой `yaoxiang run --debug`, VS Code подключается через stdio, и получает единообразный опыт точек останова, пошагового выполнения, просмотра переменных, стека вызовов, вычисления выражений и отладки конкурентности — вне зависимости от того, какой движок выполнения работает под капотом.

## Мотивация

### Зачем нужна эта возможность?

Текущие средства диагностики ошибок в программах на YaoXiang крайне примитивны:

```yaoxiang
io.println("DEBUG: x = " + x.to_string())
io.println("DEBUG: entered branch A")
```

Три смертельных проблемы:

1. **Блокировка самостоятельной компиляции (bootstrapping) компилятора**: компилятор YaoXiang пишется на самом YaoXiang, но написавший компилятор человек не может отлаживать свой собственный код. Отсутствие интерактивной отладки на стадии bootstrapping — это тупик.
2. **Три движка — ноль отладки**: интерпретатор, JIT и LLVM работают каждый по-своему, и при возникновении проблемы пользователь может лишь смотреть, появилось ли `ALL TESTS PASSED` в stdout. Сбой утверждения? Неизвестно, в какой строке и каковы значения переменных.
3. **Конкурентность — чёрный ящик**: `spawn` создаёт несколько задач — какая из них зависла? Кто переместил переменную? Всё угадывается наугад.

### Цели проектирования

- **Единый опыт**: точка останова, срабатывающая в интерпретаторе, должна срабатывать и в JIT, и в LLVM иметь согласованное отображение в исходный код. Пользователь не должен ощущать разницы между движками.
- **Один источник**: метаданные отладки следуют вместе с IR, без дублирования определений и без поддержки двух систем отображений.
- **Нулевое вмешательство**: один параметр `yaoxiang run --debug`; без него поведение компиляции и выполнения полностью не меняется.
- **Стандарт DAP**: прямое подключение к экосистеме VS Code, без изобретения нового протокола редактора.

## Предложение

### Основной проект

Обзор архитектуры:

```
┌──────────────────────────────────────────────────────┐
│                    VS Code / редактор                  │
│              DAP-клиент (launch.json)                 │
└────────────────────────┬─────────────────────────────┘
                         │ stdio
┌────────────────────────▼─────────────────────────────┐
│                 DAP-сервер (yx-core)                   │
│  ┌─────────┐  ┌──────────┐  ┌───────────────────┐    │
│  │ упр. сес.│  │ упр. точек│  │ движок вычисл.   │    │
│  └─────────┘  └──────────┘  └───────────────────┘    │
└────────────────────────┬─────────────────────────────┘
                         │ запросы/управление
┌────────────────────────▼─────────────────────────────┐
│               Интерфейс отладки runtime (trait)        │
│  pause / resume / step / get_frames / eval / ...    │
└────┬──────────────────┬──────────────────┬──────────┘
     │                  │                  │
┌────▼────┐    ┌───────▼───────┐    ┌─────▼──────┐
│Интерпр. │    │ JIT (RFC-028)│    │  LLVM AOT  │
│ читает  │    │ генерирует   │    │ метаданные │
│ метадан.│    │ лёгкую табл. │    │ IR → DWARF │
│ IR      │    │ отладки      │    │            │
└─────────┘    └───────────────┘    └────────────┘
```

**Ключевые проектные решения**:

1. **DAP-сервер и runtime разделены через trait**. Серверу безразлично, что внизу — интерпретатор или JIT: он лишь отдаёт команды через trait `DebugEngine`. Каждый движок независимо реализует один и тот же trait.
2. **`yaoxiang run --debug` принудительно использует интерпретатор**. Отладка требует управляемости, а не производительности. В режиме LLVM генерируется лишь DWARF для пост-мортем анализа (core dump / crash report), без интерактивной отладки.
3. **Повторное использование логики обнаружения точки входа из `yaoxiang run`**. Новая подкоманда не вводится — ментальная модель остаётся «запустить мою программу в режиме отладки».

### Метаданные отладки в IR

К существующему YaoXiang IR добавляются метаданные; новые виды IR не вводятся. Все метаданные генерируются в **одном месте — компиляторном фронтенде**, а для бэкендов они доступны только для чтения:

| Метаданные | Точка прикрепления | Описание |
|--------|----------------|------|
| `SourceLocation` | Каждый узел IR | Исходный файл:номер_строки:номер_столбца |
| `VarName` | Узлы объявления/связывания переменной | Имя переменной в исходном коде |
| `TypeAnnotation` | Узлы переменных/выражений | Выведенный тип (включая compile-time предикаты) |
| `ScopeBoundary` | Вход/выход блока/функции | Жизненный цикл области видимости переменной |
| `SpanInfo` | Узлы spawn | Границы задач внутри spawn-блока |

### Процесс запуска

```
yaoxiang run --debug file.yx
    │
    ├── Фаза 1: Компиляция (с метаданными отладки)
    │   ├── Синтаксический анализ → AST
    │   ├── Проверка типов + валидация compile-time предикатов
    │   └── Понижение до IR (с прикреплением метаданных отладки)
    │
    ├── Фаза 2: Использование движка интерпретатора
    │   └── В режиме --debug независимо от --release используется интерпретатор
    │
    ├── Фаза 3: Запуск DAP-сервера
    │   ├── Инициализация канала передачи stdio
    │   ├── Ожидание подключения VS Code
    │   ├── После attach — приостановка на точке входа в программу
    │   └── Переход в интерактивный цикл отладки
    │
    └── Фаза 4: Завершение программы / завершение сессии отладки → выход
```

### Различия между режимами

| Режим | Способ отладки |
|------|---------|
| `yaoxiang run --debug` | Принудительный интерпретатор, полнофункциональная интерактивная отладка через DAP |
| `yaoxiang run --release` | Генерация DWARF для пост-мортем анализа (core dump / crash report), DAP не запускается |
| `yaoxiang run` (обычный) | Без метаданных отладки, без поддержки отладки |

## Детальный проект

### 1. Точки останова

```
Типы точек останова:
├── Точки останова на строке исходного кода  → фронтенд генерирует метаданные позиций, бэкенд выполняет сопоставление
├── Точки останова на входе в функцию        → срабатывает при вызове функции (фаза 2)
├── Условные точки останова                  → срабатывает, если выражение вычисляется в true
└── Точки останова по данным                 → срабатывает при изменении переменной (фаза 2)
```

**Ядро логики точки останова на строке исходного кода**:

```
VS Code отправляет: "Установить точку останова в file.yx:42"
    │
DAP-сервер:
    ├── Запрашивает в IR все узлы с SourceLocation == (file.yx, 42)
    ├── Передаёт runtime: "Приостановить выполнение на этих IR-адресах"
    └── Runtime возвращает: ID точки останова

Выполнение программы доходит до IR-узла → runtime проверяет: этот узел в списке точек останова?
    ├── Обычная точка останова → приостановить, уведомить DAP-сервер
    └── Условная точка останова → вычислить условное выражение → приостановить, только если true
```

**Реализации в трёх движках**:

| | Интерпретатор | JIT | LLVM |
|---|---|---|---|
| Способ вставки точек останова | Проверка ID IR-узла в цикле выполнения | JIT вставляет `int3` в машинный код | Использование LLVM DWARF + аппаратных точек останова |
| Вычисление условной точки останова | Прямая интерпретация выражения | Временная JIT-компиляция условного выражения | Стек выражений DWARF + вычисление |
| Накладные расходы | Дополнительный поиск по таблице на каждый IR-узел | Только в точках останова | Почти нулевые (аппаратные точки останова) |

### 2. Пошаговое выполнение

```
Step Over    → Выполнить текущую строку, пропустить тело вызываемой функции, остановиться на следующей строке
Step Into    → Войти внутрь вызываемой функции на текущей строке
Step Out     → Выполнять до возврата из текущей функции
Continue     → Возобновить выполнение до следующей точки останова или завершения программы
```

**Логика реализации**: пошаговое выполнение по сути является **временной точкой останова**. Она разделяет один механизм с явно установленными пользователем точками останова — это не две системы, а два варианта использования одной.

```
Step Over:
    Текущий номер строки исходного кода = query_line(frame)
    → Установить временную точку останова на следующей строке
    → Если текущая строка — вызов функции: установить временную точку останова после точки вызова
    → Continue → попадание во временную точку → удалить → приостановить

Step Into:
    Первая исполнимая строка цели вызова
    → Найти позицию исходного кода первого IR-узла тела функции
    → Установить временную точку останова → Continue → попадание → приостановить

Step Out:
    Адрес возврата текущего кадра стека
    → Найти следующую строку вызывающей стороны
    → Установить временную точку останова → Continue → попадание → приостановить
```

**Обработка четырёх граничных случаев временных точек останова**:

1. **Принадлежность задаче при конкурентном выполнении**: временная точка останова привязана к ID текущей задачи; попадания из других задач просто игнорируются.
2. **Step Over над spawn-блоком**: Step Over вне spawn-блока означает выполнение всего spawn-блока и переход дальше. Для отладки внутри spawn-блока следует использовать Step Into.
3. **Временная точка останова не сработала**: установить тайм-аут сторожевого таймера (30 секунд без срабатывания какой-либо точки останова) → принудительная приостановка → уведомление VS Code. Параллельно слушать событие завершения программы → немедленная очистка.
4. **Несколько IR-узлов на одной строке**: временная точка останова Step Over помечается флагом `ignore_current_line`; при попадании, если номер строки исходного кода равен текущему номеру строки → игнорировать, продолжить.

### 3. Просмотр переменных и области видимости

```
VS Code запрашивает: "Список переменных текущего кадра"
    │
DAP-сервер:
    ├── Запрашивает IR-узел текущей точки приостановки
    ├── Обходит все связывания переменных внутри текущего ScopeBoundary
    │   └── Каждое связывание возвращает: (имя, тип, ссылка на runtime-значение)
    └── Собирает VariablesResponse → VS Code
```

**Слои областей видимости**:

```
┌─ Globals ───────────────────────────┐
│  Связывания уровня модуля: константы, псевдонимы типов, глобальные │
├─ Locals ────────────────────────────┤
│  Локальные переменные, видимые в текущей функции │
│  ├── Аргументы (параметры функции)            │
│  └── Локальные связывания (let / присваивание) │
├─ Captured ──────────────────────────┤
│  Внешние переменные, захваченные spawn-блоком / замыканием │
│  Показывают состояние владения: перемещено / разделяемая ссылка │
└─────────────────────────────────────┘
```

**Различия между движками**:

| Движок | Получение значения переменной |
|------|---------------|
| Интерпретатор | Прямое чтение кадра стека и кучи VM. Каждое значение имеет чёткое представление в памяти |
| JIT | Значения в регистрах и на стеке → требуется таблица отображения «переменная → регистр/слот стека», генерируемая при JIT-компиляции |
| LLVM | Секция `.debug_info` DWARF → `DW_AT_location` → встроенная поддержка LLDB |

**Отображение специальных типов**: compile-time предикаты, уточняющие типы, отображают ценную для отладки информацию:

```
x: Positive(x)  →  Отображается как "Int (x > 0 = True)"
y: Sorted(y)    →  Отображается как "Array(Int) (гарантия сортировки)"
result: T       →  Отображается конкретный runtime-тип
```

### 4. Стек вызовов

```
Запрос DAP: StackTrace
    │
Ответ:
┌──────────────────────────────────┐
│ #0  process_item()  file.yx:42  │  ← Текущая точка приостановки
│     locals: item = "hello"       │
│     ID spawn-задачи: task-3      │
├──────────────────────────────────┤
│ #1  main()          file.yx:67  │  ← Вызывающий
│     locals: data = ["hello", ...]│
├──────────────────────────────────┤
│ #2  <entry>         file.yx:1   │  ← Корень
└──────────────────────────────────┘
```

Каждый кадр хранит: сигнатуру функции, позицию вызова (исходный файл + номер строки), локальные переменные (ленивое вычисление), контекст spawn (ID задачи).

Интерпретатор и JIT каждый поддерживают свой список кадров. Получение кадров не бесплатно — но режим отладки не преследует цель нулевых накладных расходов.

### 5. Вычисление выражений (Watch / REPL)

Пользователь вводит произвольное выражение YaoXiang в точке останова:

```
Watch: x + y         → Возвращает результат вычисления
Watch: items[2].name → Доступ к сложной структуре
Watch: f(x)          → Вызов функции (риск побочных эффектов)
```

**Стратегия вычисления**:

```
Пользователь вводит выражение
    │
├── Фронтенд компилятора разбирает выражение
├── Проверка типов в контексте текущего кадра
├── Значения переменных берутся из текущего кадра (только для чтения)
├── Выражение выполняется как отдельная микропрограмма
│   └── Изменение внешних переменных запрещено
│   └── spawn запрещён
│   └── IO запрещено (или опционально разрешено)
└── Возврат результирующего значения → состояние исходного кадра полностью не меняется
```

**Интерпретатор как естественная песочница**: вычисление выражений — это не создание новой песочницы, а сам интерпретатор и есть песочница. Вычисление выражений лишь временно помещает кадр в стек и уничтожает его после использования. Одна и та же VM используется совместно с обычным выполнением, но никакие побочные эффекты не фиксируются.

**Вычисление вызовов функций**: разрешено по умолчанию, но с предупреждением пользователя «это выражение может иметь побочные эффекты», требующим подтверждения.

**Различия между движками**:

| Движок | Вычисление выражений |
|------|---------------|
| Интерпретатор | Повторное использование существующего пути eval, инъекция окружения текущего кадра |
| JIT | Временная компиляция выражения → линковка с текущим кадром → выполнение → выброс временного кода |
| LLVM | Не поддерживается — режим LLVM не предоставляет интерактивной отладки |

### 6. Отладка конкурентности

**Видимость модели задач**:

Концепция `threads` в DAP отображается на spawn-задачи YaoXiang. Каждая задача имеет свой список кадров стека и состояние выполнения.

```
┌─ Threads ───────────────────────────┐
│  ● task-1  main()     file.yx:10   │ ← Текущий фокус
│  ▶ task-2  fetch()    file.yx:34   │ ← Выполняется
│  ⏸ task-3  process()  file.yx:56   │ ← Приостановлена точкой останова
│  ◼ task-4  write()    Завершена      │
└─────────────────────────────────────┘
```

**Точки останова в контексте конкурентности**:

| Режим приостановки | Поведение | Сценарий использования |
|----------|------|---------|
| `stop-all` (по умолчанию) | Одна задача попала → все задачи приостановлены | Отладка гонок данных, глобального состояния |
| `stop-this-only` | Приостановить только попавшую задачу, остальные продолжают | Отладка логики независимой задачи |

**Семантика пошагового выполнения для spawn-блока**:

```
spawn {          // Step Over → выполнить весь spawn-блок
    task_a()     // Step Into → войти в task_a
    task_b()     // Выполняется параллельно, пошаговое выполнение не действует
}
```

### 7. Отображение протокола DAP

#### Фаза 1: основные запросы

| Запрос DAP | Семантика YaoXiang |
|----------|---------------|
| `initialize` | Согласование возможностей: поддержка точек останова, пошагового выполнения, переменных, кадров стека |
| `launch` / `attach` | Запуск/присоединение к программе на YaoXiang (в режиме `--debug` используется attach) |
| `setBreakpoints` | Установка точек останова на строках исходного кода |
| `configurationDone` | Точки останова готовы, начать выполнение |
| `threads` | Возвращает список всех активных spawn-задач |
| `stackTrace` | Возвращает список кадров стека указанной задачи |
| `scopes` | Возвращает области видимости переменных текущего кадра |
| `variables` | Возвращает список переменных указанной области видимости |
| `continue` | Возобновить выполнение |
| `next` | Step Over |
| `stepIn` | Step Into |
| `stepOut` | Step Out |
| `pause` | Прервать все задачи |
| `evaluate` | Вычислить выражение в текущем кадре |
| `disconnect` | Завершить сессию отладки |

#### Фаза 2: расширенные запросы

| Запрос DAP | Семантика YaoXiang |
|----------|---------------|
| `setFunctionBreakpoints` | Точки останова по имени функции |
| `setExceptionBreakpoints` | Приостановка при ошибке/панике |
| `dataBreakpointInfo` | Точки останова по данным (срабатывают при изменении переменной) |

## Стратегия реализации

### Фаза нулевая: инфраструктура (предшествует всем остальным)

**Цель**: прикрепление метаданных отладки к IR в компиляторном фронтенде.

| Компонент | Изменения |
|------|---------|
| Определение IR | Добавление полей метаданных: `SourceLocation`, `VarName`, `TypeAnnotation` и др. |
| Parser | Запись позиций исходного кода для каждого узла AST |
| TypeChecker | Прикрепление информации о типах к узлам IR |
| Тесты | Проверка того, что дамп IR содержит информацию о позициях и переменных |

**Runtime не затрагивается.**

### Фаза первая: DAP MVP на интерпретаторе

**Цель**: `yaoxiang run --debug file.yx` позволяет устанавливать точки останова, выполнять пошагово и просматривать переменные.

| Компонент | Изменения |
|------|---------|
| DAP-сервер (новый модуль yx-core) | Транспортный уровень stdio, обработка основных запросов, менеджер точек останова (отображение строка исходного кода → IR-узел) |
| Trait отладки runtime (yx-core) | Определение trait `DebugEngine` (pause, resume, step, get_frames, eval, get_variables) |
| Интерпретатор | Проверка точек останова в цикле выполнения, механизм приостановки/возобновления, поддержание списка кадров, реализация `InterpreterDebugEngine` |
| CLI | Параметр `yaoxiang run --debug` |

**Критерии приёмки**: для любого файла `.yx` в `tests/yaoxiang/` в VS Code должны работать установка точек останова, Step Over и просмотр значений переменных.

### Фаза вторая: расширенные возможности отладки

**Цель**: вычисление выражений, точки останова по функциям, отладка конкурентности, точки останова по исключениям.

| Компонент | Изменения |
|------|---------|
| Движок вычисления выражений | Компиляция микропрограммы (повторное использование parser + typechecker), помещение временного кадра в VM, изоляция побочных эффектов |
| Отладка конкурентности | Отображение списка spawn-задач, привязка точек останова к ID задачи, стратегии приостановки stop-all / stop-this-only |
| Точки останова по функциям/исключениям | Отображение `setFunctionBreakpoints`, `setExceptionBreakpoints` |
| Расширение VS Code | Предоставление шаблона `launch.json` по умолчанию |

### Фаза третья: отладка JIT и DWARF в LLVM

**Цель**: движок JIT повторно использует DAP, LLVM порождает DWARF для анализа крашей.

| Компонент | Изменения |
|------|---------|
| JIT | Реализация trait `DebugEngine`, генерация таблицы отображения переменная→регистр во время компиляции, список кадров runtime, временная компиляция выражений |
| LLVM | Метаданные отладки IR → LLVM `DILocation` / `DISubprogram` → DWARF (без интерактивного DAP) |

### Зависимости

```
Фаза 0 (метаданные IR)
    ↓
Фаза 1 (DAP MVP на интерпретаторе)  ← отсюда уже можно пользоваться
    ↓
Фаза 2 (расширенные возможности)
    ↓
Фаза 3 (JIT + DWARF для LLVM)
```

### Риски

| Риск | Смягчение |
|------|------|
| Сложность механизма приостановки интерпретатора | Использовать простые channel/signal вместо сложного конечного автомата; приостановка — это просто запрет на выборку следующей инструкции |
| Типобезопасность вычисления выражений | Повторное использование существующего typechecker, только чтение, без фиксации побочных эффектов |
| Тонкости протокола DAP | Ссылаться на реализации debugpy / delve — протокол уже устоялся |
| Livelock при stop-all в конкурентной отладке | Механизм тайм-аута + принудительная приостановка |

## Компромиссы

### Преимущества

- **Один источник**: метаданные отладки генерируются один раз, три движка делят их. Не возникнет ситуации «в интерпретаторе отладочная информация верна, а в LLVM — нет»
- **Нулевое вмешательство**: один параметр `--debug`; без него поведение полностью не меняется
- **Стандарт DAP**: прямое подключение к экосистеме VS Code, без необходимости в собственном протоколе редактора или UI отладчика
- **Приоритет интерпретатора**: отладка естественно ложится на интерпретатор — гибкость, управляемость, простое вычисление выражений. Отказ от интерактивной отладки в режиме LLVM — наиболее прагматичный выбор

### Недостатки

- **Низкая производительность режима отладки**: интерпретатор значительно медленнее JIT/LLVM. Но отладка не требует производительности — никто не ожидает запуска production-нагрузки в режиме отладки
- **Ограниченная отладка LLVM**: AOT-компиляция не поддерживает интерактивную отладку, только GDB/LLDB + DWARF. Но это компромисс: в режиме LLVM и не должно быть различий в поведении отладки
- **Сложность приостановки при конкурентности**: реализация семантики stop-all в интерпретаторе требует обхода всех активных задач

### Альтернативы

| Альтернатива | Почему не выбрана |
|------|-------------|
| Каждый из трёх движков реализует DAP самостоятельно | Тройной объём работы, тройной набор багов. Нарушает «хороший вкус» |
| Использовать только DWARF, без собственного DAP | У интерпретатора и JIT нет концепции DWARF, LLDB не проникнет внутрь VM |
| Сделать консольный отладчик по образцу Python pdb | Опыт VS Code на порядок превосходит консольные отладчики |
| Встроить DAP в процесс LSP | Жизненные циклы принципиально различны — LSP следует за проектом, DAP — за сессией отладки. Изоляция процессов — жёсткое требование |
