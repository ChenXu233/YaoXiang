# RFC-034: Унифицированный инструментарий отладки

## Резюме

Введение единого инструментария отладки в YaoXiang. Основной замысел — **один источник, три потребителя**: компиляторный фронтенд встраивает позиции исходного кода, имена переменных и информацию о типах в YaoXiang IR как полноправных граждан, а три бэкенда — интерпретатор, JIT и LLVM — каждый потребляют один и тот же набор метаданных. Пользователь запускает DAP-сервер (Debug Adapter Protocol) командой `yaoxiang run --debug`, VS Code подключается через stdio, после чего доступен унифицированный опыт отладки: точки останова, пошаговое выполнение, просмотр переменных, стек вызовов, вычисление выражений, отладка конкурентных задач — независимо от нижележащего исполнительного движка.

## Мотивация

### Зачем нужна эта функциональность?

Текущие средства диагностики ошибок в программах на YaoXiang крайне примитивны:

```yaoxiang
io.println("DEBUG: x = " + x.to_string())
io.println("DEBUG: entered branch A")
```

Три фатальные проблемы:

1. **Самоприменяющаяся разработка компилятора затруднена**: компилятор YaoXiang пишется на YaoXiang, но автор компилятора не может отлаживать собственный код. Отсутствие интерактивной отладки на этапе самоприменения — это тупик.
2. **Три движка — ноль отладки**: интерпретатор, JIT и LLVM работают каждый по-своему; при сбое пользователь может лишь проверить, появилась ли `ALL TESTS PASSED` в stdout. Сбой утверждения? Неизвестно, на какой строке и каковы значения переменных.
3. **Конкурентность — чёрный ящик**: `spawn` порождает несколько задач — какая зависла? Кто переместил (`move`) переменную? Всё на уровне догадок.

### Цели проектирования

- **Единый опыт**: точка останова, работающая в интерпретаторе, работает и в JIT, и в LLVM с согласованным отображением на исходный код. Пользователь не ощущает разницы между движками.
- **Один источник**: отладочные метаданные текут вместе с IR, не дублируются, не требуется поддерживать два набора отображений.
- **Нулевая интрузивность**: один аргумент `yaoxiang run --debug`; без этого аргумента поведение компиляции и выполнения полностью идентично.
- **Стандарт DAP**: прямое подключение к экосистеме VS Code, без изобретения собственного протокола редактора.

## Предложение

### Основной замысел

Архитектурный обзор:

```
┌──────────────────────────────────────────────────────┐
│                    VS Code / Редактор                  │
│              DAP Client (launch.json)                 │
└────────────────────────┬─────────────────────────────┘
                         │ stdio
┌────────────────────────▼─────────────────────────────┐
│                 DAP-сервер (yx-core)                   │
│  ┌─────────┐  ┌──────────┐  ┌───────────────────┐    │
│  │ Упр. сеансами│ Упр. точками останова │ Движок вычисления выражений│
│  └─────────┘  └──────────┘  └───────────────────┘    │
└────────────────────────┬─────────────────────────────┘
                         │ Запросы/Управление
┌────────────────────────▼─────────────────────────────┐
│            Интерфейс отладки среды выполнения (trait)   │
│  pause / resume / step / get_frames / eval / ...    │
└────┬──────────────────┬──────────────────┬──────────┘
     │                  │                  │
┌────▼────┐    ┌───────▼───────┐    ┌─────▼──────┐
│Интерпретатор│   │ JIT (RFC-028)│    │  LLVM AOT  │
│Прямое     │    │ Генерация     │    │ Метаданные │
│использование│  │ лёгких отладочн.│   │ IR → DWARF │
│метаданных IR│  │ таблиц        │    │            │
└─────────┘    └───────────────┘    └────────────┘
```

**Ключевые проектные решения**:

1. **DAP-сервер и среда выполнения разделены через trait**. Серверу безразлично, что внизу — интерпретатор или JIT — он лишь отдаёт команды через trait `DebugEngine`. Каждый движок реализует один и тот же trait самостоятельно.
2. **`yaoxiang run --debug` принудительно использует интерпретатор**. Отладка требует контролируемости, а не производительности. В режиме LLVM генерируется лишь DWARF для посмертного анализа (дампы ядра / отчёты о сбоях); интерактивная отладка не выполняется.
3. **Логика обнаружения точки входа заимствована у `yaoxiang run`**. Новая подкоманда не вводится — ментальная модель: «запустить мою программу в режиме отладки».

### Отладочные метаданные IR

К существующему YaoXiang IR добавляются метаданные без введения новых разновидностей IR. Все метаданные генерируются в **одном месте — компиляторном фронтенде**; для бэкендов они доступны только для чтения:

| Метаданные | Точка прикрепления | Описание |
|------------|--------------------|----------|
| `SourceLocation` | Каждый узел IR | Файл:строка:столбец |
| `VarName` | Узлы объявления/связывания переменных | Имя переменной в исходном коде |
| `TypeAnnotation` | Узлы переменных/выражений | Выведенный тип (включая предикаты времени компиляции) |
| `ScopeBoundary` | Вход/выход блока/функции | Жизненный цикл области видимости переменной |
| `SpanInfo` | Узлы spawn | Граница задач внутри spawn-блока |

### Последовательность запуска

```
yaoxiang run --debug file.yx
    │
    ├── Фаза 1: Компиляция (с отладочными метаданными)
    │   ├── Разбор → AST
    │   ├── Проверка типов + верификация предикатов компиляции
    │   └── Понижение до IR (с прикреплением отладочных метаданных)
    │
    ├── Фаза 2: Использование движка-интерпретатора
    │   └── В режиме --debug всегда применяется интерпретатор, независимо от --release
    │
    ├── Фаза 3: Запуск DAP-сервера
    │   ├── Инициализация транспортного канала stdio
    │   ├── Ожидание attach от VS Code
    │   ├── После attach — пауза на входе в программу
    │   └── Переход в интерактивный цикл отладки
    │
    └── Фаза 4: Завершение программы / сеанса отладки → выход
```

### Различия режимов

| Режим | Способ отладки |
|-------|----------------|
| `yaoxiang run --debug` | Принудительный интерпретатор, полнофункциональная интерактивная отладка по DAP |
| `yaoxiang run --release` | Генерация DWARF для посмертного анализа (дампы ядра / отчёты о сбоях), DAP не запускается |
| `yaoxiang run` (обычный) | Без отладочных метаданных, отладка не поддерживается |

## Подробное проектирование

### 1. Точки останова

```
Типы точек останова:
├── Точка останова на строке исходного кода → фронтенд генерирует метаданные позиций, бэкенд запрашивает совпадения
├── Точка останова на входе в функцию      → срабатывает при вызове функции (вторая фаза)
├── Условная точка останова                 → срабатывает, когда выражение-условие истинно
└── Точка останова по данным                → срабатывает при изменении переменной (вторая фаза)
```

**Базовая логика точки останова на строке**:

```
VS Code отправляет: "Установить точку останова в file.yx:42"
    │
DAP-сервер:
    ├── Запрашивает в IR все узлы, у которых SourceLocation == (file.yx, 42)
    ├── Передаёт среде выполнения: "Приостановить на этих адресах IR"
    └── Среда выполнения возвращает: ID точки останова

Выполнение программы доходит до IR-узла → среда выполнения проверяет: этот узел в списке точек останова?
    ├── Обычная точка → пауза, уведомление DAP-сервера
    └── Условная → вычисление выражения-условия → истина → пауза
```

**Реализация в трёх движках**:

|                  | Интерпретатор | JIT | LLVM |
|------------------|---------------|-----|------|
| Способ вставки точки | Проверка ID IR-узла в цикле выполнения | JIT вставляет `int3` в машинный код | LLVM DWARF + аппаратные точки останова |
| Вычисление условия | Прямая интерпретация выражения | Временная JIT-компиляция выражения-условия | Стек выражений DWARF + вычисление |
| Накладные расходы | Доп. поиск по таблице на каждый IR-узел | Только в точках останова | Почти нулевые (аппаратные) |

### 2. Пошаговое выполнение

```
Step Over    → Выполнить текущую строку, пропустить внутренность вызова, остановиться на следующей
Step Into    → Войти внутрь вызова функции на текущей строке
Step Out     → Выполнить до возврата из текущей функции
Continue     → Возобновить выполнение до следующей точки останова или завершения программы
```

**Логика реализации**: пошаговые операции по сути являются **временными точками останова**. Они разделяют единый механизм с явно заданными пользователем точками — это не две системы, а два применения одной.

```
Step Over:
    Текущий номер строки = query_line(frame)
    → Установить временную точку останова на следующей строке
    → Если текущая строка содержит вызов: установить временную точку после точки вызова
    → Continue → срабатывание временной точки → удаление → пауза

Step Into:
    Первая исполнимая строка цели вызова
    → Найти позицию в исходном коде первого IR-узла тела функции
    → Установить временную точку → Continue → срабатывание → пауза

Step Out:
    Адрес возврата текущего кадра стека
    → Найти следующую строку вызывающего кода
    → Установить временную точку → Continue → срабатывание → пауза
```

**Обработка четырёх граничных случаев временных точек останова**:

1. **Принадлежность к конкурентной задаче**: временная точка привязана к ID текущей задачи; срабатывания в других задачах просто игнорируются.
2. **Step Over spawn-блока**: Step Over снаружи spawn-блока эквивалентен выполнению всего spawn-блока целиком с переходом дальше. Для входа внутрь spawn для отладки следует использовать Step Into.
3. **Несрабатывание временной точки**: устанавливается таймаут-сторож (30 секунд без срабатывания) → принудительная пауза → уведомление VS Code. Параллельно слушается событие завершения программы → немедленная очистка.
4. **Несколько IR-узлов на одной строке**: временная точка Step Over помечается `ignore_current_line`; при срабатывании, если номер строки источника совпадает с текущим → игнорировать, продолжить.

### 3. Просмотр переменных и областей видимости

```
VS Code запрашивает: "Список переменных текущего кадра"
    │
DAP-сервер:
    ├── Запрашивает IR-узел текущей точки паузы
    ├── Обходит связывания переменных внутри текущего ScopeBoundary
    │   └── Для каждого связывания возвращает: (имя, тип, ссылка на значение среды выполнения)
    └── Формирует VariablesResponse → VS Code
```

**Слои областей видимости**:

```
┌─ Глобальные (Globals) ─────────────────────┐
│  Связывания уровня модуля: константы, псевдонимы типов, глобальные переменные │
├─ Локальные (Locals) ──────────────────────┤
│  Локальные переменные, видимые в текущей функции │
│  ├── Параметры (аргументы функции)            │
│  └── Локальные связывания (let / присваивание) │
├─ Захваченные (Captured) ───────────────────┐
│  Внешние переменные, захваченные spawn-блоком / замыканием │
│  Отображается состояние владения: перемещено / разделяемая ссылка │
└──────────────────────────────────────────────┘
```

**Различия движков**:

| Движок | Получение значения переменной |
|--------|-------------------------------|
| Интерпретатор | Прямое чтение стека кадров и кучи VM. Каждое значение имеет явное представление в памяти |
| JIT | Значения в регистрах и на стеке → требуется таблица отображения «переменная → регистр/слот стека», формируемая при JIT-компиляции |
| LLVM | Секция `.debug_info` DWARF → `DW_AT_location` → встроенная поддержка LLDB |

**Отображение специальных типов**: предикаты компиляции, уточняющие тип, дают полезную для отладки информацию:

```
x: Positive(x)  →  Отобразить "Int (x > 0 = True)"
y: Sorted(y)    →  Отобразить "Array(Int) (гарантия сортировки)"
result: T       →  Отобразить конкретный тип времени выполнения
```

### 4. Стек вызовов

```
DAP-запрос: StackTrace
    │
Ответ:
┌──────────────────────────────────────┐
│ #0  process_item()  file.yx:42       │  ← Текущая точка паузы
│     locals: item = "hello"            │
│     ID spawn-задачи: task-3           │
├──────────────────────────────────────┤
│ #1  main()          file.yx:67       │  ← Вызывающий код
│     locals: data = ["hello", ...]     │
├──────────────────────────────────────┤
│ #2  <entry>         file.yx:1        │  ← Корень
└──────────────────────────────────────┘
```

Каждый кадр содержит: сигнатуру функции, позицию вызова (файл+строка), локальные переменные (ленивое вычисление), контекст spawn (ID задачи).

Интерпретатор и JIT поддерживают собственные списки кадров. Получение кадров — не нулевая стоимость, но в режиме отладки нулевая стоимость и не требуется.

### 5. Вычисление выражений (Watch / REPL)

Пользователь вводит произвольное выражение YaoXiang в точке останова:

```
Watch: x + y         → Возвращает результат вычисления
Watch: items[2].name → Доступ к сложной структуре
Watch: f(x)          → Вызов функции (риск побочных эффектов)
```

**Стратегия вычисления**:

```
Пользователь вводит выражение
    │
├── Парсинг выражения компиляторным фронтендом
├── Проверка типов в контексте текущего кадра
├── Значения переменных берутся из текущего кадра (только для чтения)
├── Выражение выполняется как отдельная микропрограмма
│   └── Изменение внешних переменных запрещено
│   └── spawn запрещён
│   └── Ввод-вывод запрещён (или опционально разрешён)
└── Возврат значения результата → исходное состояние кадра полностью неизменно
```

**Интерпретатор — естественная песочница**: вычисление выражения не создаёт новой песочницы — сам интерпретатор ей является. Вычисление лишь временно помещает кадр в стек, а затем уничтожает его. Используется та же VM, что и при обычном выполнении, но побочные эффекты не фиксируются.

**Вычисление вызовов функций**: разрешено по умолчанию, но с предупреждением «это выражение может иметь побочные эффекты», требующим подтверждения пользователя.

**Различия движков**:

| Движок | Вычисление выражений |
|--------|----------------------|
| Интерпретатор | Переиспользование существующего пути eval с инъекцией окружения текущего кадра |
| JIT | Временная компиляция выражения → линковка с текущим кадром → выполнение → выброс временного кода |
| LLVM | Не поддерживается — в режиме LLVM интерактивная отладка не выполняется |

### 6. Отладка конкурентности

**Видимость модели задач**:

Понятие DAP `threads` отображается на spawn-задачи YaoXiang. Каждая задача имеет собственный список кадров стека и состояние выполнения.

```
┌─ Threads ─────────────────────────────┐
│  ● task-1  main()     file.yx:10       │ ← Текущий фокус
│  ▶ task-2  fetch()    file.yx:34       │ ← Выполняется
│  ⏸ task-3  process()  file.yx:56       │ ← Пауза на точке останова
│  ◼ task-4  write()    Завершена         │
└────────────────────────────────────────┘
```

**Точки останова в конкурентном контексте**:

| Режим паузы | Поведение | Применимо |
|-------------|-----------|-----------|
| `stop-all` (по умолчанию) | Одна задача сработала → все задачи приостановлены | Отладка гонок данных, глобального состояния |
| `stop-this-only` | Пауза только сработавшей задачи, остальные продолжают | Отладка логики отдельной задачи |

**Семантика пошагового выполнения для spawn-блоков**:

```
spawn {          // Step Over → выполнить весь spawn-блок целиком
    task_a()     // Step Into → войти в task_a
    task_b()     // Выполняется параллельно, не зависит от отдельных шагов
}
```

### 7. Отображение протокола DAP

#### Первая фаза: базовые запросы

| DAP-запрос | Семантика YaoXiang |
|------------|--------------------|
| `initialize` | Согласование возможностей: поддержка точек останова, пошагового выполнения, переменных, кадров стека |
| `launch` / `attach` | Запуск/присоединение к программе YaoXiang (--debug работает в режиме attach) |
| `setBreakpoints` | Установка точек останова на строках исходного кода |
| `configurationDone` | Точки останова готовы, начать выполнение |
| `threads` | Список всех активных spawn-задач |
| `stackTrace` | Список кадров стека указанной задачи |
| `scopes` | Области видимости переменных текущего кадра |
| `variables` | Список переменных в указанной области видимости |
| `continue` | Возобновить выполнение |
| `next` | Step Over |
| `stepIn` | Step Into |
| `stepOut` | Step Out |
| `pause` | Прервать все задачи |
| `evaluate` | Вычислить выражение в текущем кадре |
| `disconnect` | Завершить сеанс отладки |

#### Вторая фаза: расширенные запросы

| DAP-запрос | Семантика YaoXiang |
|------------|--------------------|
| `setFunctionBreakpoints` | Точки останова по имени функции |
| `setExceptionBreakpoints` | Пауза при ошибке/панике |
| `dataBreakpointInfo` | Точки останова по данным (при изменении переменной) |

## Стратегия реализации

### Фаза ноль: инфраструктура (до всех остальных фаз)

**Цель**: компиляторный фронтенд прикрепляет отладочные метаданные к IR.

| Компонент | Изменения |
|-----------|-----------|
| Определение IR | Добавление полей метаданных `SourceLocation`, `VarName`, `TypeAnnotation` и т. д. |
| Parser | Каждый узел AST записывает свою позицию в исходном коде |
| TypeChecker | Информация о типах прикрепляется к узлам IR |
| Тесты | Проверка, что дамп IR содержит информацию о позициях и переменных |

**Среда выполнения не затрагивается.**

### Первая фаза: DAP-MVP для интерпретатора

**Цель**: `yaoxiang run --debug file.yx` позволяет устанавливать точки останова, пошагово выполнять и просматривать переменные.

| Компонент | Изменения |
|-----------|-----------|
| DAP-сервер (новый модуль yx-core) | Транспортный уровень stdio, обработка базовых запросов, менеджер точек останова (отображение строк исходного кода на IR-узлы) |
| Trait отладки среды выполнения (yx-core) | Определение trait `DebugEngine` (pause, resume, step, get_frames, eval, get_variables) |
| Интерпретатор | Проверка точек останова в цикле выполнения, механизм паузы/возобновления, ведение списка кадров, реализация `InterpreterDebugEngine` |
| CLI | Аргумент `yaoxiang run --debug` |

**Критерий приёмки**: для любого `.yx`-файла из `tests/yaoxiang/` в VS Code можно установить точку останова, выполнить Step Over и увидеть значения переменных.

### Вторая фаза: расширенные возможности отладки

**Цель**: вычисление выражений, функциональные точки останова, отладка конкурентности, точки останова на исключениях.

| Компонент | Изменения |
|-----------|-----------|
| Движок вычисления выражений | Компиляция микропрограмм (переиспользование parser + typechecker), помещение временного кадра в VM, изоляция побочных эффектов |
| Отладка конкурентности | Отображение списка spawn-задач, привязка точек останова к ID задач, стратегии паузы stop-all / stop-this-only |
| Функциональные точки останова и точки останова на исключениях | Отображение `setFunctionBreakpoints`, `setExceptionBreakpoints` |
| Расширение VS Code | Шаблон `launch.json` по умолчанию |

### Третья фаза: отладка JIT и DWARF для LLVM

**Цель**: повторное использование DAP движком JIT, генерация LLVM DWARF для посмертного анализа сбоев.

| Компонент | Изменения |
|-----------|-----------|
| JIT | Реализация trait `DebugEngine`, генерация таблицы отображения «переменная → регистр» при компиляции, ведение списка кадров в среде выполнения, временная компиляция выражений |
| LLVM | Метаданные отладки IR → LLVM `DILocation` / `DISubprogram` → DWARF (без интерактивного DAP) |

### Зависимости

```
Фаза ноль (метаданные IR)
    ↓
Фаза первая (DAP-MVP для интерпретатора)  ← отсюда уже пригодно к использованию
    ↓
Фаза вторая (расширенные возможности)
    ↓
Фаза третья (JIT + DWARF для LLVM)
```

### Риски

| Риск | Смягчение |
|------|-----------|
| Сложность механизма паузы в интерпретаторе | Простые каналы/сигналы вместо сложного конечного автомата: пауза — это просто блокировка выборки следующей инструкции |
| Типобезопасность вычисления выражений | Переиспользование существующего typechecker, ссылки только для чтения, без фиксации побочных эффектов |
| Детали протокола DAP | Опора на реализации debugpy / delve — протокол зрелый |
| Возможный livelock stop-all в конкурентной отладке | Механизм таймаута + принудительная пауза |

## Компромиссы

### Преимущества

- **Один источник**: отладочные метаданные генерируются один раз и разделяются тремя движками. Не возникнет ситуации «в интерпретаторе отладочная информация верна, а в LLVM — нет»
- **Нулевая интрузивность**: один аргумент `--debug`; без него поведение полностью идентично
- **Стандарт DAP**: прямое подключение к экосистеме VS Code, без необходимости в собственном протоколе редактора или UI отладчика
- **Приоритет интерпретатора**: отладка естественно ложится на интерпретатор — гибкость, контролируемость, простое вычисление выражений. Отсутствие интерактивной отладки в режиме LLVM — наиболее прагматичный выбор

### Недостатки

- **Низкая производительность в режиме отладки**: интерпретатор значительно медленнее JIT/LLVM. Но отладка не требует производительности — никто не запускает продуктовую нагрузку в режиме отладки
- **Ограниченная отладка LLVM**: AOT-компиляция не поддерживает интерактивную отладку, только GDB/LLDB + DWARF. Это компромисс: в режиме LLVM не должно быть разницы в поведении отладки
- **Сложность конкурентной паузы**: реализация семантики stop-all в интерпретаторе требует обхода всех активных задач

### Альтернативы

| Альтернатива | Почему не выбрана |
|--------------|-------------------|
| Каждый из трёх движков реализует DAP самостоятельно | Тройной объём работы, тройной набор багов. Противоречит «хорошему вкусу» |
| Только DWARF, без собственного DAP | Интерпретатор и JIT не имеют концепции DWARF; LLDB не способен заглянуть внутрь VM |
| Сделать консольный отладчик по образцу Python pdb | Опыт VS Code полностью превосходит консольный отладчик |
| Поместить DAP в процесс LSP | Жизненные циклы принципиально разные — LSP привязан к проекту, DAP — к сеансу отладки. Изоляция процессов — жёсткое требование |
