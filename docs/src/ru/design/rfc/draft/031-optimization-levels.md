# RFC-031: Уровни оптимизации и менеджер Pass'ов

> **Ссылки**:
> - [RFC-011: Дизайн системы дженериков](../accepted/011-generic-type-system.md)
> - [RFC-028: JIT-компилятор](./028-jit-compiler.md)
> - [RFC-018: AOT-компилятор на базе LLVM](../accepted/018-llvm-aot-compiler.md)

## Резюме

Настоящий документ предлагает ввести в YaoXiang **систему уровней оптимизации** и **менеджер Pass'ов**, чтобы перевести компиляторскую оптимизацию из модели «всё или ничего» в настраиваемые пакеты оптимизаций. Уровни оптимизации (O0–O3) определяют различные комбинации стратегий оптимизации, а менеджер Pass'ов отвечает за выполнение проходов оптимизации (Pass'ов) в порядке зависимостей. Документ также определяет стандартный интерфейс Pass'ов оптимизации, обеспечивающий архитектурную основу для последующих расширений (мономорфизация, инлайнинг, свёртка констант и т. д.).

**Ключевая цель: дать пользователю возможность явного выбора между скоростью компиляции, размером бинарника и производительностью во время выполнения.**

## Мотивация

### Зачем нужны уровни оптимизации?

В текущем компиляторе нет конфигурации оптимизаций, и весь код проходит через один и тот же конвейер обработки. Это приводит к следующим проблемам:

1. **Плохой опыт отладки**: при отладке оптимизация не нужна, но её нельзя отключить
2. **Невозможно контролировать размер бинарника**: мономорфизация дженериков раздувает бинарник, но отключить её нельзя
3. **Неконтролируемая скорость компиляции**: нельзя выбрать «быструю компиляцию» или «глубокую оптимизацию» в зависимости от сценария
4. **Pass'ы оптимизации не упорядочены**: в будущем у нескольких Pass'ов оптимизации будут зависимости, и ими нужно управлять централизованно

### Текущие проблемы

```yaoxiang
# В текущей реализации: весь код проходит одинаковую обработку
# - При отладке: оптимизация не нужна, но её нельзя отключить
# - В продакшене: оптимизация нужна, но глубину не настроить
# - Дженерик-функции: генерируется несколько копий, но управлять этим нельзя

identity: (T: Type) -> (x: T) -> T = (x) => x
x = identity(42)        # Будет сгенерирована identity_Int
s = identity("hello")   # Будет сгенерирована identity_String
# Пользователь не может выбрать «без мономорфизации» (режим стирания типов)
```

### Ценность уровней оптимизации

| Сценарий | Потребность | Уровень оптимизации |
|------|------|----------|
| Разработка и отладка | Быстрая компиляция, сохранение отладочной информации | O0 |
| Повседневная разработка | Базовая оптимизация, баланс скорости компиляции | O1 |
| Тестирование / CI | Стандартная оптимизация, проверка поведения в продакшене | O2 |
| Релиз в продакшен | Глубокая оптимизация, максимальная производительность | O3 |
| Скрипты / быстрые прототипы | Автоматический выбор (на основе целевой платформы) | Auto |

## Предложение

### Основной дизайн

#### 1. Определение уровней оптимизации

```rust
/// Уровень оптимизации
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize, Default)]
pub enum OptLevel {
    /// O0: без оптимизации (режим отладки)
    /// - Сохраняет всю отладочную информацию
    /// - Не выполняет никаких оптимизирующих преобразований
    /// - Максимальная скорость компиляции
    /// - Применение: разработка и отладка, быстрая итерация
    O0,

    /// O1: базовая оптимизация (по умолчанию)
    /// - Мономорфизация по требованию (без генерации неиспользуемых специализаций)
    /// - Базовая свёртка констант
    /// - Базовое удаление мёртвого кода
    /// - Применение: повседневная разработка
    #[default]
    O1,

    /// O2: стандартная оптимизация
    /// - Мономорфизация по требованию
    /// - Полная свёртка констант
    /// - Полное удаление мёртвого кода
    /// - Инлайнинг малых функций
    /// - Оптимизация хвостовых вызовов
    /// - Применение: тестирование, CI, релиз в продакшен
    O2,

    /// O3: агрессивная оптимизация
    /// - Полная мономорфизация (предварительная генерация всех возможных комбинаций типов)
    /// - Агрессивный инлайнинг
    /// - Все Pass'ы оптимизации
    /// - Возможно увеличение времени компиляции и размера бинарника
    /// - Применение: задачи с требованиями максимальной производительности
    O3,

    /// Auto: автоматический выбор
    /// - Стратегия оптимизации выбирается автоматически на основе целевой платформы и доступных ресурсов
    /// - Применение: скрипты, быстрые прототипы
    Auto,
}
```

#### 2. Интерфейс Pass'а оптимизации

```rust
/// Интерфейс Pass'а оптимизации
pub trait OptimizationPass {
    /// Имя Pass'а (для логов и объявления зависимостей)
    fn name(&self) -> &str;

    /// Запуск Pass'а
    fn run(&self, module: &mut ModuleIR, config: &PassConfig) -> PassResult;

    /// Какие другие Pass'ы должны быть выполнены перед этим
    fn dependencies(&self) -> Vec<&str> {
        vec![]
    }

    /// Должен ли этот Pass выполняться при текущей конфигурации
    fn should_run(&self, config: &PassConfig) -> bool {
        true
    }
}

/// Конфигурация Pass'а
#[derive(Debug, Clone)]
pub struct PassConfig {
    /// Уровень оптимизации
    pub opt_level: OptLevel,
    /// Включена ли отладочная информация
    pub debug_info: bool,
    /// Целевая платформа
    pub target_platform: TargetPlatform,
}

/// Результат выполнения Pass'а
#[derive(Debug, Default)]
pub struct PassResult {
    /// Было ли модифицировано IR
    pub changed: bool,
    /// Статистика
    pub stats: PassStats,
}

/// Статистика Pass'а
#[derive(Debug, Default)]
pub struct PassStats {
    /// Количество инлайненных функций
    pub functions_inlined: usize,
    /// Количество мономорфизированных функций
    pub functions_monomorphized: usize,
    /// Количество удалённого мёртвого кода
    pub dead_code_removed: usize,
    /// Количество свёрнутых констант
    pub constants_folded: usize,
}
```

#### 3. Менеджер Pass'ов

```rust
/// Оптимизатор
pub struct Optimizer {
    /// Список зарегистрированных Pass'ов (упорядочен по зависимостям)
    passes: Vec<Box<dyn OptimizationPass>>,
}

impl Optimizer {
    /// Создать оптимизатор для указанного уровня оптимизации
    pub fn for_opt_level(level: OptLevel) -> Self {
        let passes = Self::create_passes_for_level(level);
        Self { passes }
    }

    /// Создать список Pass'ов для указанного уровня
    fn create_passes_for_level(level: OptLevel) -> Vec<Box<dyn OptimizationPass>> {
        match level {
            OptLevel::O0 => {
                vec![
                    // Режим отладки: минимум оптимизаций, только необходимая очистка
                    Box::new(ConstFoldPass::minimal()),
                ]
            }
            OptLevel::O1 => {
                vec![
                    // Базовая оптимизация
                    Box::new(ConstFoldPass::basic()),
                    Box::new(MonomorphizePass::on_demand()),
                    Box::new(DcePass::basic()),
                ]
            }
            OptLevel::O2 => {
                vec![
                    // Стандартная оптимизация
                    Box::new(ConstFoldPass::full()),
                    Box::new(MonomorphizePass::on_demand()),
                    Box::new(DcePass::full()),
                    Box::new(InlinePass::small_functions()),
                    Box::new(TcoPass::new()),
                ]
            }
            OptLevel::O3 => {
                vec![
                    // Агрессивная оптимизация
                    Box::new(ConstFoldPass::full()),
                    Box::new(MonomorphizePass::full()),
                    Box::new(InlinePass::aggressive()),
                    Box::new(DcePass::full()),
                    Box::new(TcoPass::new()),
                    // Дополнительные агрессивные оптимизации...
                ]
            }
            OptLevel::Auto => {
                // Автоматический выбор: определяется целевой платформой
                Self::create_passes_for_level(OptLevel::O1)
            }
        }
    }

    /// Выполнить все Pass'ы оптимизации
    pub fn run(&self, module: &mut ModuleIR, config: &PassConfig) -> OptimizerResult {
        let mut total_stats = OptimizerStats::default();

        for pass in &self.passes {
            if !pass.should_run(config) {
                continue;
            }

            let result = pass.run(module, config);
            total_stats.merge(result.stats);
        }

        OptimizerResult {
            module: module.clone(),
            stats: total_stats,
        }
    }
}
```

### Примеры

#### Использование из командной строки

```bash
# Режим отладки: без оптимизации
yaoxiang build --opt-level O0

# Повседневная разработка: базовая оптимизация (по умолчанию)
yaoxiang build

# Релиз в продакшен: стандартная оптимизация
yaoxiang build --opt-level O2

# Максимальная производительность: агрессивная оптимизация
yaoxiang build --opt-level O3

# Автоматический выбор
yaoxiang build --opt-level Auto
```

### Изменения синтаксиса

Изменений синтаксиса нет. Уровень оптимизации — это настройка компилятора, не влияющая на синтаксис языка.

## Детальный дизайн

### Соответствие уровней оптимизации и Pass'ов

| Pass | O0 | O1 | O2 | O3 | Описание |
|------|----|----|----|----|----|
| **Свёртка констант** | Минимальная | Базовая | Полная | Полная | Вычисление константных выражений в compile-time |
| **Мономорфизация** | ❌ | По требованию | По требованию | Полная | Специализация дженерик-функций |
| **Удаление мёртвого кода** | ❌ | Базовое | Полное | Полное | Удаление неиспользуемого кода |
| **Инлайнинг функций** | ❌ | ❌ | Малые функции | Агрессивный | Подстановка тела функции в точку вызова |
| **Оптимизация хвостовых вызовов** | ❌ | ❌ | ✅ | ✅ | Преобразование хвостовой рекурсии в цикл |
| **Анализ побегов** | ❌ | ❌ | ❌ | ✅ | Определение стекового/кучного размещения |
| **Оптимизация циклов** | ❌ | ❌ | ❌ | ✅ | Развёртка циклов, вынос инвариантов |

### Стратегии мономорфизации

```rust
/// Стратегия мономорфизации
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize, Default)]
pub enum MonoStrategy {
    /// Без мономорфизации — стирание типов, дженерик-функция существует в одном экземпляре
    /// Плюсы: компактный бинарник, быстрая компиляция
    /// Минусы: накладные расходы на динамическую диспетчеризацию во время выполнения
    Erased,

    /// Мономорфизация по требованию — код генерируется только для реально используемых комбинаций типов
    /// Плюсы: абстракции с нулевой стоимостью, без накладных расходов в runtime
    /// Минусы: возможно раздувание бинарника
    #[default]
    OnDemand,

    /// Полная мономорфизация — предварительная генерация всех возможных комбинаций типов
    /// Плюсы: все вызовы определяются в compile-time
    /// Минусы: медленная компиляция, большой бинарник
    Full,
}

/// Конфигурация мономорфизации
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub struct MonoConfig {
    /// Включена ли мономорфизация
    #[serde(default = "default_true")]
    pub enabled: bool,

    /// Стратегия мономорфизации
    #[serde(default)]
    pub strategy: MonoStrategy,

    /// Включено ли DCE (удаление мёртвого кода)
    #[serde(default = "default_true")]
    pub dce_enabled: bool,

    /// Максимальная глубина специализации (защита от бесконечной рекурсии в дженериках)
    #[serde(default = "default_max_mono_depth")]
    pub max_depth: usize,
}

impl Default for MonoConfig {
    fn default() -> Self {
        Self {
            enabled: true,
            strategy: MonoStrategy::OnDemand,
            dce_enabled: true,
            max_depth: 100,
        }
    }
}
```

### Интеграция в процесс компиляции

```rust
// src/frontend/pipeline.rs

impl Pipeline {
    fn run_ir_generation(
        &mut self,
        source_name: &str,
        source: &str,
        ast: &Module,
        type_result: &TypeCheckResult,
        phase_durations: &mut Vec<(CompilationPhase, u64)>,
    ) -> IRResult {
        let start = Instant::now();

        // 1. Генерация базового IR
        let mut ir = middle::generate_ir(ast, type_result)?;

        // 2. Запуск Pass'ов оптимизации в соответствии с уровнем оптимизации
        let optimizer = Optimizer::for_opt_level(self.config.optimization_level);
        let pass_config = PassConfig {
            opt_level: self.config.optimization_level,
            debug_info: self.config.generate_debug_info,
            target_platform: TargetPlatform::detect(),
        };

        let result = optimizer.run(&mut ir, &pass_config);

        let duration = start.elapsed().as_millis() as u64;
        phase_durations.push((CompilationPhase::Optimization, duration));

        IRResult::success(result.module)
    }
}
```

### Влияние на систему типов

Прямого влияния нет. Pass'ы оптимизации работают на уровне IR и не затрагивают систему типов.

### Поведение в runtime

| Уровень оптимизации | Поведение в runtime |
|----------|-----------|
| O0 | Без оптимизации, сохраняется вся отладочная информация |
| O1 | Базовая оптимизация, сохраняется базовая отладочная информация |
| O2 | Стандартная оптимизация, без отладочной информации |
| O3 | Агрессивная оптимизация, без отладочной информации |

**Ключевой момент: изменения в runtime не требуются**. Pass'ы оптимизации влияют только на уровень IR и кодогенерацию; во время выполнения поиск функций ведётся по имени/идентификатору, и процесс оптимизации прозрачен для runtime.

### Изменения в компиляторе

| Компонент | Изменения |
|------|------|
| `frontend/config.rs` | Добавить перечисление `OptLevel` и `MonoConfig` |

### Обратная совместимость

- ✅ Полная обратная совместимость
- Уровень оптимизации по умолчанию — O1, поведение совпадает с текущим
- Пользователь может явно указать уровень оптимизации для переопределения поведения по умолчанию

## Компромиссы

### Преимущества

- **Гибкость**: пользователь может выбирать стратегию оптимизации в зависимости от сценария
- **Расширяемость**: стандартный интерфейс Pass'ов упрощает добавление новых оптимизаций
- **Предсказуемость**: поведение каждого уровня оптимизации определено явно
- **Удобство отладки**: режим O0 сохраняет полную отладочную информацию

### Недостатки

- **Рост сложности**: требуется сопровождать несколько уровней оптимизации
- **Увеличение матрицы тестирования**: нужно тестировать поведение на каждом уровне оптимизации
- **Нагрузка на документацию**: нужно пояснять значение каждого уровня

## Альтернативные варианты

| Вариант | Почему не выбран |
|------|--------------|
| Только два состояния: вкл./выкл. | Невозможна тонкая настройка глубины оптимизации |
| Использовать числовые `-O` в стиле GCC/LLVM | Не соответствует системе конфигурации YaoXiang |
| Каждый Pass оптимизации включается отдельно | Пользователь должен разбираться в деталях каждого Pass'а, использование усложняется |
| Отложить до v2.0 | Мономорфизация уже реализована, но не интегрирована — сначала нужно решить архитектурный вопрос |

## Стратегия реализации

### Этапы

1. **Этап 1 (текущий)**: определение уровней оптимизации и интерфейса Pass'ов
2. **Этап 2**: реализация Pass'а мономорфизации (на базе существующего модуля `mono/`)
3. **Этап 3**: реализация Pass'ов свёртки констант и удаления мёртвого кода
4. **Этап 4**: реализация Pass'ов инлайнинга функций и оптимизации хвостовых вызовов
5. **Этап 5**: реализация Pass'ов агрессивной оптимизации (анализ побегов, оптимизация циклов)

### Зависимости

- Зависит от RFC-011 (система дженериков), модуль мономорфизации
- Зависит от RFC-028 (JIT-компилятор), интерфейс Pass'ов оптимизации
- Общий дизайн Pass'ов оптимизации с RFC-018 (LLVM AOT)

### Риски

- **Регрессия производительности**: Pass'ы оптимизации могут вносить ошибки, приводящие к падению производительности
- **Увеличение времени компиляции**: Pass'ы оптимизации увеличивают время компиляции
- **Раздувание бинарника**: мономорфизация может значительно увеличить размер бинарника

| Термин | Определение |
|------|------|
| **Pass оптимизации** | Самостоятельный модуль, выполняющий одно преобразование IR |
| **Мономорфизация** | Стратегия кодогенерации, при которой дженерик-функции специализируются под конкретные типы |
| **Свёртка констант** | Вычисление константных выражений в compile-time |
| **Удаление мёртвого кода** | Удаление недостижимого или неиспользуемого кода в программе |
| **Инлайнинг функций** | Подстановка тела функции в точку вызова для устранения накладных расходов на вызов |
| **Оптимизация хвостовых вызовов** | Преобразование хвостовой рекурсии в цикл во избежание переполнения стека |
| **Анализ побегов** | Анализ того, выходит ли переменная за пределы области видимости, для выбора стекового или кучного размещения |

## Жизненный цикл и дальнейшая судьба

Настоящий RFC определяет архитектуру уровней оптимизации и формирует единый каркас для последующих Pass'ов оптимизации.

**Связь с мономорфизацией**: мономорфизация — один из Pass'ов оптимизации; после принятия настоящего RFC она станет первым реализованным Pass'ом.
