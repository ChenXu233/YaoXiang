# RFC-031: Уровни оптимизации и менеджер Pass'ов

> **Ссылки**:
> - [RFC-011: Дизайн системы обобщённых типов](../accepted/011-generic-type-system.md)
> - [RFC-028: JIT-компилятор](./028-jit-compiler.md)
> - [RFC-018: LLVM AOT-компилятор](../accepted/018-llvm-aot-compiler.md)

## Резюме

Настоящий документ предлагает ввести в YaoXiang **систему уровней оптимизации** и **менеджер Pass'ов**, превращая компиляторные оптимизации из режима «всё или ничего» в настраиваемый набор оптимизаций. Уровни оптимизации (O0–O3) определяют различные комбинации стратегий оптимизации, а менеджер Pass'ов отвечает за выполнение оптимизационных Pass'ов в порядке зависимостей. Документ также определяет стандартный интерфейс для оптимизационных Pass'ов, создавая архитектурный фундамент для будущих расширений (мономорфизация, инлайнинг, свёртка констант и т. д.).

**Ключевая цель: дать пользователю возможность явного выбора между скоростью компиляции, размером бинарника и производительностью в рантайме.**

## Мотивация

### Зачем нужны уровни оптимизации?

В текущем компиляторе нет конфигурации оптимизаций — весь код проходит через один и тот же конвейер обработки. Это приводит к следующим проблемам:

1. **Плохой опыт отладки**: при отладке оптимизация не нужна, но её нельзя отключить.
2. **Невозможно контролировать размер бинарника**: мономорфизация обобщений раздувает бинарник, но отключить её нельзя.
3. **Скорость компиляции неуправляема**: нельзя выбрать «быструю компиляцию» или «глубокую оптимизацию» в зависимости от сценария.
4. **Оптимизационные Pass'ы выполняются бессистемно**: в будущем между несколькими Pass'ами возникнут зависимости, требующие единого управления.

### Текущие проблемы

```yaoxiang
# В текущей реализации: весь код обрабатывается одинаково
# — при отладке: оптимизация не нужна, но её нельзя отключить
# — в продакшене: оптимизация нужна, но глубину нельзя настроить
# — обобщённые функции: генерируются в нескольких экземплярах, но это нельзя контролировать

identity: (T: Type) -> (x: T) -> T = (x) => x
x = identity(42)        # сгенерируется identity_Int
s = identity("hello")   # сгенерируется identity_String
# Пользователь не может выбрать «без мономорфизации» (режим стирания типов)
```

### Ценность уровней оптимизации

| Сценарий | Потребность | Уровень оптимизации |
|----------|-------------|---------------------|
| Разработка и отладка | Быстрая компиляция, сохранение отладочной информации | O0 |
| Повседневная разработка | Базовая оптимизация, баланс скорости компиляции | O1 |
| Тестирование / CI | Стандартная оптимизация, проверка продакшен-поведения | O2 |
| Продакшен-релиз | Глубокая оптимизация, максимальная производительность | O3 |
| Скрипты / быстрые прототипы | Автовыбор (на основе целевой платформы) | Auto |

## Предложение

### Основной дизайн

#### 1. Определение уровней оптимизации

```rust
/// Уровень оптимизации
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize, Default)]
pub enum OptLevel {
    /// O0: без оптимизации (режим отладки)
    /// — сохраняется вся отладочная информация
    /// — никаких оптимизирующих трансформаций
    /// — максимальная скорость компиляции
    /// — применение: отладка, быстрая итерация
    O0,

    /// O1: базовая оптимизация (по умолчанию)
    /// — мономорфизация по требованию (без неиспользуемых специализаций)
    /// — базовая свёртка констант
    /// — базовое устранение мёртвого кода
    /// — применение: повседневная разработка
    #[default]
    O1,

    /// O2: стандартная оптимизация
    /// — мономорфизация по требованию
    /// — полная свёртка констант
    /// — полное устранение мёртвого кода
    /// — инлайнинг малых функций
    /// — оптимизация хвостовых вызовов
    /// — применение: тестирование, CI, продакшен-релиз
    O2,

    /// O3: агрессивная оптимизация
    /// — полная мономорфизация (предварительная генерация всех возможных комбинаций типов)
    /// — агрессивный инлайнинг
    /// — все оптимизационные Pass'ы
    /// — возможно увеличение времени компиляции и размера бинарника
    /// — применение: требования максимальной производительности
    O3,

    /// Auto: автоматический выбор
    /// — стратегия выбирается автоматически на основе целевой платформы и доступных ресурсов
    /// — применение: скрипты, быстрые прототипы
    Auto,
}
```

#### 2. Интерфейс оптимизационного Pass'а

```rust
/// Интерфейс оптимизационного Pass'а
pub trait OptimizationPass {
    /// Имя Pass'а (для логов и объявления зависимостей)
    fn name(&self) -> &str;

    /// Запуск Pass'а
    fn run(&self, module: &mut ModuleIR, config: &PassConfig) -> PassResult;

    /// Какие Pass'ы должны быть выполнены до этого Pass'а
    fn dependencies(&self) -> Vec<&str> {
        vec![]
    }

    /// Должен ли этот Pass выполняться при текущей конфигурации
    fn should_run(&self, config: &PassConfig) -> bool {
        true
    }
}

/// Конфигурация Pass'а
#[derive(Debug, Clone)]
pub struct PassConfig {
    /// Уровень оптимизации
    pub opt_level: OptLevel,
    /// Включена ли отладочная информация
    pub debug_info: bool,
    /// Целевая платформа
    pub target_platform: TargetPlatform,
}

/// Результат выполнения Pass'а
#[derive(Debug, Default)]
pub struct PassResult {
    /// Было ли изменено IR
    pub changed: bool,
    /// Статистика
    pub stats: PassStats,
}

/// Статистика Pass'а
#[derive(Debug, Default)]
pub struct PassStats {
    /// Количество инлайненных функций
    pub functions_inlined: usize,
    /// Количество мономорфизированных функций
    pub functions_monomorphized: usize,
    /// Количество единиц устранённого мёртвого кода
    pub dead_code_removed: usize,
    /// Количество свёрнутых констант
    pub constants_folded: usize,
}
```

#### 3. Менеджер Pass'ов

```rust
/// Оптимизатор
pub struct Optimizer {
    /// Список зарегистрированных Pass'ов (упорядочен по зависимостям)
    passes: Vec<Box<dyn OptimizationPass>>,
}

impl Optimizer {
    /// Создание оптимизатора по уровню оптимизации
    pub fn for_opt_level(level: OptLevel) -> Self {
        let passes = Self::create_passes_for_level(level);
        Self { passes }
    }

    /// Создание списка Pass'ов для указанного уровня
    fn create_passes_for_level(level: OptLevel) -> Vec<Box<dyn OptimizationPass>> {
        match level {
            OptLevel::O0 => {
                vec![
                    // Режим отладки: минимум оптимизаций, только необходимая очистка
                    Box::new(ConstFoldPass::minimal()),
                ]
            }
            OptLevel::O1 => {
                vec![
                    // Базовая оптимизация
                    Box::new(ConstFoldPass::basic()),
                    Box::new(MonomorphizePass::on_demand()),
                    Box::new(DcePass::basic()),
                ]
            }
            OptLevel::O2 => {
                vec![
                    // Стандартная оптимизация
                    Box::new(ConstFoldPass::full()),
                    Box::new(MonomorphizePass::on_demand()),
                    Box::new(DcePass::full()),
                    Box::new(InlinePass::small_functions()),
                    Box::new(TcoPass::new()),
                ]
            }
            OptLevel::O3 => {
                vec![
                    // Агрессивная оптимизация
                    Box::new(ConstFoldPass::full()),
                    Box::new(MonomorphizePass::full()),
                    Box::new(InlinePass::aggressive()),
                    Box::new(DcePass::full()),
                    Box::new(TcoPass::new()),
                    // Дополнительные агрессивные оптимизации...
                ]
            }
            OptLevel::Auto => {
                // Автовыбор: определяется целевой платформой
                Self::create_passes_for_level(OptLevel::O1)
            }
        }
    }

    /// Запуск всех оптимизационных Pass'ов
    pub fn run(&self, module: &mut ModuleIR, config: &PassConfig) -> OptimizerResult {
        let mut total_stats = OptimizerStats::default();

        for pass in &self.passes {
            if !pass.should_run(config) {
                continue;
            }

            let result = pass.run(module, config);
            total_stats.merge(result.stats);
        }

        OptimizerResult {
            module: module.clone(),
            stats: total_stats,
        }
    }
}
```

### Примеры

#### Использование из командной строки

```bash
# Режим отладки: без оптимизации
yaoxiang build --opt-level O0

# Повседневная разработка: базовая оптимизация (по умолчанию)
yaoxiang build

# Продакшен-релиз: стандартная оптимизация
yaoxiang build --opt-level O2

# Максимальная производительность: агрессивная оптимизация
yaoxiang build --opt-level O3

# Автовыбор
yaoxiang build --opt-level Auto
```

### Изменения синтаксиса

Изменений синтаксиса нет. Уровень оптимизации — это конфигурация компилятора, не затрагивающая синтаксис языка.

## Детальный дизайн

### Соответствие уровней оптимизации и Pass'ов

| Pass | O0 | O1 | O2 | O3 | Описание |
|------|----|----|----|----|----------|
| **Свёртка констант** | Минимальная | Базовая | Полная | Полная | Вычисление константных выражений в compile-time |
| **Мономорфизация** | ❌ | По требованию | По требованию | Полная | Специализация обобщённых функций |
| **Устранение мёртвого кода** | ❌ | Базовое | Полное | Полное | Удаление недостижимого или неиспользуемого кода |
| **Инлайнинг функций** | ❌ | ❌ | Малые функции | Агрессивный | Вставка тела функции в точку вызова |
| **Оптимизация хвостовых вызовов** | ❌ | ❌ | ✅ | ✅ | Преобразование хвостовой рекурсии в цикл |
| **Анализ побегов** | ❌ | ❌ | ❌ | ✅ | Определение аллокации в стеке/куче |
| **Оптимизация циклов** | ❌ | ❌ | ❌ | ✅ | Развёртка циклов, вынос инвариантов |

### Стратегии мономорфизации

```rust
/// Стратегия мономорфизации
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize, Default)]
pub enum MonoStrategy {
    /// Без мономорфизации — стирание типов, обобщённая функция существует в одном экземпляре
    /// Плюсы: маленький бинарник, быстрая компиляция
    /// Минусы: накладные расходы на динамическую диспетчеризацию в рантайме
    Erased,

    /// Мономорфизация по требованию — код генерируется только для фактически используемых комбинаций типов
    /// Плюсы: абстракции с нулевыми накладными расходами, без рантайм-затрат
    /// Минусы: возможно раздувание бинарника
    #[default]
    OnDemand,

    /// Полная мономорфизация — предварительная генерация всех возможных комбинаций типов
    /// Плюсы: все вызовы известны в compile-time
    /// Минусы: медленная компиляция, большой бинарник
    Full,
}

/// Конфигурация мономорфизации
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub struct MonoConfig {
    /// Включена ли мономорфизация
    #[serde(default = "default_true")]
    pub enabled: bool,

    /// Стратегия мономорфизации
    #[serde(default)]
    pub strategy: MonoStrategy,

    /// Включено ли устранение мёртвого кода (DCE)
    #[serde(default = "default_true")]
    pub dce_enabled: bool,

    /// Максимальная глубина специализации (для предотвращения бесконечной рекурсии обобщений)
    #[serde(default = "default_max_mono_depth")]
    pub max_depth: usize,
}

impl Default for MonoConfig {
    fn default() -> Self {
        Self {
            enabled: true,
            strategy: MonoStrategy::OnDemand,
            dce_enabled: true,
            max_depth: 100,
        }
    }
}
```

### Интеграция в процесс компиляции

```rust
// src/frontend/pipeline.rs

impl Pipeline {
    fn run_ir_generation(
        &mut self,
        source_name: &str,
        source: &str,
        ast: &Module,
        type_result: &TypeCheckResult,
        phase_durations: &mut Vec<(CompilationPhase, u64)>,
    ) -> IRResult {
        let start = Instant::now();

        // 1. Генерация базового IR
        let mut ir = middle::generate_ir(ast, type_result)?;

        // 2. Запуск оптимизационных Pass'ов согласно уровню оптимизации
        let optimizer = Optimizer::for_opt_level(self.config.optimization_level);
        let pass_config = PassConfig {
            opt_level: self.config.optimization_level,
            debug_info: self.config.generate_debug_info,
            target_platform: TargetPlatform::detect(),
        };

        let result = optimizer.run(&mut ir, &pass_config);

        let duration = start.elapsed().as_millis() as u64;
        phase_durations.push((CompilationPhase::Optimization, duration));

        IRResult::success(result.module)
    }
}
```

### Влияние на систему типов

Прямого влияния нет. Оптимизационные Pass'ы работают на уровне IR и не затрагивают систему типов.

### Поведение в рантайме

| Уровень оптимизации | Поведение в рантайме |
|---------------------|----------------------|
| O0 | Без оптимизации, сохраняется вся отладочная информация |
| O1 | Базовая оптимизация, сохраняется базовая отладочная информация |
| O2 | Стандартная оптимизация, без отладочной информации |
| O3 | Агрессивная оптимизация, без отладочной информации |

**Ключевой момент: изменения в рантайме не требуются.** Оптимизационные Pass'ы влияют только на уровень IR и кодогенерацию, в рантайме выполнение идёт через поиск по имени/ID функции и не учитывает процесс оптимизации.

### Изменения в компиляторе

| Компонент | Изменения |
|-----------|-----------|
| `frontend/config.rs` | Добавить перечисление `OptLevel` и `MonoConfig` |
| `frontend/pipeline.rs` | Интегрировать менеджер Pass'ов |
| `middle/passes/optimizer/` | Новый модуль оптимизационных Pass'ов |
| `middle/passes/mono/` | Рефакторинг под стандартный интерфейс Pass'а |
| CLI | Добавить параметр `--opt-level` |

### Обратная совместимость

- ✅ Полностью обратно совместимо
- Уровень оптимизации по умолчанию — O1, поведение совпадает с текущим
- Пользователь может явно указать уровень оптимизации для переопределения поведения по умолчанию

## Компромиссы

### Преимущества

- **Гибкость**: пользователь выбирает стратегию оптимизации под сценарий
- **Расширяемость**: стандартный интерфейс Pass'а упрощает добавление новых оптимизаций
- **Предсказуемость**: поведение каждого уровня оптимизации явно определено
- **Дружелюбность к отладке**: режим O0 сохраняет полную отладочную информацию

### Недостатки

- **Рост сложности**: требуется поддержка нескольких уровней оптимизации
- **Расширение матрицы тестирования**: нужно тестировать поведение каждого уровня
- **Нагрузка на документацию**: требуется объяснять смысл каждого уровня

## Альтернативы

| Решение | Почему не выбрано |
|---------|-------------------|
| Только два состояния — вкл/выкл | Невозможно точно контролировать глубину оптимизации |
| Стиль `-O` из GCC/LLVM | Не консистентно с системой конфигурации YaoXiang |
| Независимый переключатель для каждого Pass'а | Пользователю нужно разбираться в деталях Pass'ов, использование усложняется |
| Отложить до v2.0 | Мономорфизация уже реализована, но не интегрирована — нужно сначала решить архитектурный вопрос |

## Стратегия реализации

### Этапы

1. **Этап 1 (текущий)**: определение уровней оптимизации и интерфейса Pass'а
2. **Этап 2**: реализация Pass'а мономорфизации (на основе существующего модуля `mono/`)
3. **Этап 3**: реализация Pass'ов свёртки констант и устранения мёртвого кода
4. **Этап 4**: реализация Pass'ов инлайнинга функций и оптимизации хвостовых вызовов
5. **Этап 5**: реализация агрессивных оптимизаций (анализ побегов, оптимизация циклов)

### Зависимости

- Зависит от модуля мономорфизации в RFC-011 (система обобщений)
- Зависит от интерфейса оптимизационных Pass'ов в RFC-028 (JIT-компилятор)
- Делит дизайн оптимизационных Pass'ов с RFC-018 (LLVM AOT)

### Риски

- **Регрессия производительности**: Pass'ы могут содержать баги, ведущие к падению производительности
- **Увеличение времени компиляции**: Pass'ы увеличивают время компиляции
- **Раздувание бинарника**: мономорфизация может значительно увеличить размер бинарника

## Приложение Б: Глоссарий

| Термин | Определение |
|--------|-------------|
| **Оптимизационный Pass** | Самостоятельный модуль, выполняющий одну трансформацию IR |
| **Мономорфизация** | Стратегия кодогенерации, при которой обобщённая функция специализируется под конкретные типы |
| **Свёртка констант** | Вычисление константных выражений на этапе compile-time |
| **Устранение мёртвого кода** | Удаление недостижимого или неиспользуемого кода в программе |
| **Инлайнинг функций** | Вставка тела функции в точку вызова для устранения накладных расходов на вызов |
| **Оптимизация хвостовых вызовов** | Преобразование хвостовой рекурсии в цикл для предотвращения переполнения стека |
| **Анализ побегов** | Анализ того, выходит ли переменная за пределы области видимости, для решения о стековой/кучевой аллокации |

## Жизненный цикл и дальнейшая судьба

Настоящий RFC определяет архитектуру уровней оптимизации и предоставляет единый фреймворк для будущих оптимизационных Pass'ов.
