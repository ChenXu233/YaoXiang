# RFC-025: Расширяемый механизм примитивных типов

> **Статус**: Черновик
> **Автор**: Чэнь Сюй
> **Дата создания**: 2026-06-05
> **Последнее обновление**: 2026-06-05

## Аннотация

В данном документе определяется **расширяемый механизм примитивных типов** (`Primitive::Extension`) компилятора YaoXiang. Он позволяет внешнему коду регистрировать пользовательские примитивные типы, чтобы компилятор мог поддерживать предметно-ориентированные типы (кубиты, GPU-буферы, SIMD-векторы, аппаратные регистры и т.д.) без жёсткого кодирования.

## Мотивация

### Зачем нужен этот механизм?

В текущий момент компилятор содержит жёстко закодированные примитивные типы: `Int`, `Float`, `String`, `Bool`, `Unit`. Каждый раз при добавлении нового примитивного типа требуется изменять несколько мест в исходном коде компилятора — проверщик типов, генератор кода, анализатор собственности, оптимизатор.

Это нарушает принцип открытости/закрытости: открытость для расширения, закрытость для модификации.

### Границы проектирования

```
Жёстко закодированные базовые типы (основа языка): Int, Float, String, Bool, Unit
Динамически расширяемые типы (предметно-ориентированные плагины): через Primitive::Extension
```

Базовые типы жёстко закодированы, поскольку компилятор глубоко зависит от их семантики (арифметические операции, условные переходы, хэширование, сравнения). Расширяемые типы являются для компилятора **непрозрачными значениями** — компилятор знает только их размер, выравнивание и атрибуты собственности, но не внутреннюю семантику.

Это не «превращение всех типов в динамически загружаемые». Базовые и расширяемые типы — это две разные вещи.

## Предложение

### Основной дизайн

#### 1. Атрибуты типа Extension

При регистрации каждого расширяемого примитивного типа необходимо объявить следующие атрибуты:

```rust
pub struct PrimitiveExtension {
    /// Имя типа, например "Qubit", "Buffer", "Vec128"
    pub name: String,

    /// Размер в байтах (фиксированный размер типа)
    /// None означает, что размер неизвестен на этапе компиляции (требует определения в runtime)
    pub size: Option<usize>,

    /// Требования к выравниванию
    pub align: Option<usize>,

    /// Допускается ли неявное копирование
    /// false = семантика Move (присваивание означает перемещение, например Qubit)
    /// true = семантика Copy (присваивание означает копирование, например Vec128)
    pub is_copy: bool,

    /// Допускается ли пустое значение (zero-sized type)
    pub allow_zst: bool,
}
```

#### 2. Интерфейс регистрации

```rust
// Внутренний API компилятора
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".to_string(),
    size: Some(0),           // Логический размер 0, физическое состояние на квантовом процессоре
    align: Some(1),
    is_copy: false,          // Семантика Move, соответствует no-cloning
    allow_zst: true,
});
```

После регистрации `Qubit` становится легальным примитивным типом в системы типов и может использоваться для объявления переменных, параметров функций, полей структур.

#### 3. Поведение проверщика типов

Расширяемые примитивные типы следуют следующим правилам при проверке типов:

| Сценарий | Поведение |
|------|------|
| Объявление переменной `q: Qubit = ...` | ✅ Легально |
| Параметр функции `fn(q: Qubit)` | ✅ Легально |
| Поле структуры `{ q: Qubit }` | ✅ Легально |
| Присваивание при `is_copy == false` | Семантика Move, исходная переменная становится недействительной |
| Присваивание при `is_copy == true` | Семантика Copy, исходная переменная остаётся действительной |
| Неявное копирование (использование в нескольких местах функции) | Зависит от `is_copy` |
| Сравнение `==`, `!=` | ❌ Ошибка компиляции (нет встроенного сравнения) |
| Арифметика `+`, `-` | ❌ Ошибка компиляции (нет встроенных операций) |
| Обобщённое ограничение `T: Copy` | Удовлетворяется только если `is_copy == true` |

#### 4. Поведение генератора кода

Расширяемые примитивные типы обрабатываются генератором кода как **непрозрачные значения**:

- LLVM IR: генерируются как `{size} x i8` или структура соответствующего размера
- Никаких специальных инструкций не генерируется — семантику обеспечивает backend или библиотека
- Если backend требует специальной обработки (например, квантовые вентили QIR), это реализуется через механизм регистрации backend (не входит в данный RFC)

### Примеры

#### Регистрация типа с семантикой Move

```rust
// Квантовый бит: не копируется, размер 0 (физическое состояние на QPU)
compiler.register_primitive(PrimitiveExtension {
    name: "Qubit".into(),
    size: Some(0),
    align: Some(1),
    is_copy: false,
    allow_zst: true,
});
```

```yaoxiang
# Код пользователя
q: Qubit = qubit(0)
q2 = q          # ❌ Ошибка компиляции: Qubit — тип Move, q уже недействителен
q = H(q)        # ✅ Потребляет q, возвращает новый q
```

#### Регистрация типа с семантикой Copy

```rust
// SIMD-вектор: копируемый, фиксированный размер
compiler.register_primitive(PrimitiveExtension {
    name: "Vec128".into(),
    size: Some(16),
    align: Some(16),
    is_copy: true,
    allow_zst: false,
});
```

```yaoxiang
# Код пользователя
a: Vec128 = load_vec128(data)
b = a           # ✅ Семантика Copy, a всё ещё действителен
c = add_vec128(a, b)  # ✅ a и b оба доступны
```

## Детальное проектирование

### Изменения в компиляторе

| Компонент | Изменение |
|------|------|
| Система типов | Добавлен новый вариант `Ty::Extension`, хранящий метаданные `PrimitiveExtension` |
| Проверщик типов | Расширяемые типы не участвуют в разрешении встроенных операций, не удовлетворяют встроенным trait-ограничениям (если не реализованы явно) |
| Анализатор собственности | На основе `is_copy` определяет семантику Move или Copy |
| Генератор кода | Генерирует непрозрачные значения согласно `size`/`align`, без специальных инструкций |
| Сообщения об ошибках | Ошибки расширяемых типов ссылаются на `name` в момент регистрации |

### Связь с FFI

`Primitive::Extension` ортогонален RFC-021 (FFI):

| | Primitive::Extension | FFI |
|---|---|---|
| Назначение | Регистрация нового **типа** | Вызов внешней **функции** |
| Уровень | Система типов | Runtime |
| Пример | `Qubit` — это тип | `native("sin")` — это вызов функции |

Предметная область может одновременно нуждаться в обоих: тип `Qubit` через Extension, а функции квантовых вентилей через FFI.

### Обратная совместимость

- ✅ Полная обратная совместимость
- Семантика существующих типов не изменяется
- Расширяемые типы — это новая возможность, не влияющая на существующий код

## Компромиссы

### Преимущества

- ✅ Компилятору не нужно модифицировать исходный код для каждой новой предметной области
- ✅ Эксперты в предметной области могут самостоятельно регистрировать типы, не завися от команды компилятора
- ✅ Базовые типы остаются жёстко закодированными, не жертвуя глубокой оптимизацией компилятором базовых типов
- ✅ Простой интерфейс — одна структура определяет все атрибуты

### Недостатки

- ⚠️ Расширяемые типы не поддерживают встроенные операции — требуются дополнительные функции или механизмы backend для реализации семантики
- ⚠️ При отладке расширяемые типы отображаются как непрозрачные значения, менее наглядно, чем базовые типы

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| Все типы с динамической загрузкой | Базовые типы (Int/Float/Bool) требуют глубокой оптимизации компилятором, динамическая загрузка лишит этих возможностей |
| Жёсткое кодирование для каждой предметной области | Каждая новая предметная область требует изменений в компиляторе, не масштабируется |
| Чисто библиотечное решение (без регистрации типов) | Невозможно гарантировать семантику на уровне системы типов (например, no-cloning), только проверка в runtime |

## Стратегия реализации

### Этап 1: Основной интерфейс

- [ ] Добавить вариант `Ty::Extension` в систему типов
- [ ] Реализовать API `register_primitive`
- [ ] Расширить проверщик типов для обработки семантики `is_copy`
- [ ] Расширить генератор кода для обработки непрозрачных значений
- [ ] Модульные тесты

### Этап 2: Момент регистрации

- [ ] Поддержка пакетной регистрации при инициализации компилятора (конфигурационный файл или builder API)
- [ ] Поддержка предварительной регистрации стандартной библиотекой (модуль `std.primitive` экспортирует определения расширяемых типов)

### Зависимости

- Нет жёстких зависимостей. Может быть реализовано независимо от других RFC.

## Открытые вопросы

- [ ] Должны ли расширяемые типы поддерживать реализацию trait (например, позволить `Qubit` реализовать пользовательский trait `QuantumGate`)?
- [ ] Нужны ли хуки жизненного цикла (например, `on_drop`) для поддержки расширяемых типов с RAII-семантикой?
- [ ] Формат конфигурационного файла: TOML, YAML или собственный синтаксис конфигурации YaoXiang (см. RFC-015)?
