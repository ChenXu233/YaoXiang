# RFC-028: JIT-компилятор — многоуровневый исполнительный движок в VM

> **Ссылки**:
> - [RFC-018: Проект LLVM AOT-компилятора](../review/018-llvm-aot-compiler.md)
> - [RFC-024: Модель конкурентности на основе spawn-блоков](../accepted/024-concurrency-model.md)
> - [RFC-008: Разделение модели конкурентности Runtime и планировщика](../accepted/008-runtime-concurrency-model.md)

## Резюме

Настоящий документ предлагает ввести Cranelift JIT-компилятор в бэкенд VM языка YaoXiang, обновив VM от чистого интерпретатора до **многоуровневого исполнительного движка**: холодный код выполняется интерпретатором, горячие функции компилируются Cranelift в нативный код. Путь JIT разделяет нормализующие IR-проходы с путём LLVM AOT из RFC-018: Cranelift отвечает за быструю компиляцию JIT, а LLVM — за глубокую оптимизацию AOT. Каждый инструмент решает свою задачу.

**Ключевое позиционирование: JIT обслуживает VM, а не заменяет её.**

## Мотивация

### Зачем нужен JIT?

Текущий бэкенд VM — это чистый интерпретатор, скорость выполнения в 10–100 раз ниже нативного кода. При разработке часто запускаются тесты, скрипты, локальная отладка — для этих сценариев не нужна предельная оптимизация AOT, но требуется заметно более быстрое выполнение, чем у интерпретатора.

### Почему не использовать только LLVM AOT?

LLVM AOT-компиляция занимает много времени (секунды), что не подходит для итераций разработки. Разработке нужен опыт «изменил — запустил»: изменил строку кода → перезапустил → почти мгновенно увидел результат. Cranelift JIT компилирует одну функцию за 1–5 мс, пользователь не ощущает задержки компиляции.

### Почему Cranelift, а не LLVM ORC JIT?

| Параметр | Cranelift JIT | LLVM ORC JIT |
|------|--------------|--------------|
| Скорость компиляции | 1–5 мс/функция | 10–100 мс/функция |
| Размер зависимостей | Небольшой | Большой (требуется полный LLVM) |
| Качество кода | 70–80% от LLVM -O2 | Очень высокое |
| Сценарий применения | Разработка и отладка, быстрые итерации | Не подходит (см. компромиссы в документе) |

Cranelift компилирует быстро, качества кода достаточно. LLVM остаётся для глубокой офлайн-оптимизации AOT. Один инструмент — одна задача хорошо.

## Предложение

### Ключевая архитектура

```
Исполнительный движок VM
├── Слой интерпретатора
│   ├── Выполняет байткод-инструкции
│   ├── Собирает данные о горячих точках (invocation count + loop backedge count)
│   └── При достижении порога → отправляет задачу компиляции
│
├── Слой JIT-компиляции (Cranelift Backend)
│   ├── Очередь компиляции (фоновый поток, не блокирует интерпретатор)
│   ├── IR → нормализация → Cranelift IR → нативный код
│   └── Повторно использует IR-нормализующий проход из RFC-018 §4.0 (стек → SSA)
│
├── Кэш кода
│   ├── Таблица функций: ID функции → {вход в интерпретатор, вход в JIT (опц.)}
│   ├── Атомарная замена точки входа скомпилированной функции
│   └── Группировка по модулям (резерв интерфейса горячей перезагрузки)
│
└── Профилирование горячих точек
    ├── Счётчик вызовов на функцию + счётчик обратных рёбер циклов
    ├── Периодическое затухание (предотвращает разовый прогрев)
    └── Три уровня: Cold → Warm → Hot → Compiled
```

### Интеграция с существующей архитектурой

```
Исходный код → Фронтенд (общий) → IR → ┬→ codegen байткода → Интерпретатор VM → [горячие функции] → Cranelift JIT
                                         │
                                         └→ codegen LLVM AOT → .o → Линковка → exe (продакшн)
```

JIT и AOT разделяют **IR-нормализующий проход** (`middle/passes/ir_normalize.rs`), низкоуровневый codegen меняется с LLVM на Cranelift.

### Поток выполнения

```
Вызов функции
  → fn_entry.code_ptr.load()
  → ┬─ stub интерпретатора (холодное состояние): интерпретирует байткод по инструкциям
    └─ Нативный код JIT (горячее состояние): выполняет машинный код напрямую
  → Возврат
```

## Детальный проект

### 1. Структура каталогов

```
src/
├── backends/
│   ├── interpreter/              # Существующий — интерпретатор VM
│   │   └── executor/
│   │       ├── engine.rs         # Изменён — точка входа вызова: с прямой интерпретации на диспетчеризацию через FunctionEntry
│   │       └── ...
│   │
│   ├── jit/                      # Новый — слой JIT-компиляции
│   │   ├── mod.rs                # Вход в модуль JIT, инициализация контекста Cranelift
│   │   ├── profiler.rs           # Счётчики горячих точек + затухание + пороговые решения
│   │   ├── entry.rs              # FunctionEntry + управление AtomicPtr
│   │   ├── cache.rs              # Кэш кода (управление исполняемыми страницами mmap)
│   │   ├── compiler.rs           # IR → Cranelift IR → нативный код
│   │   ├── types.rs              # Соответствие типов YaoXiang → типы Cranelift
│   │   └── abi.rs                # Соглашение о вызовах функций (System V / Microsoft x64)
│   │
│   ├── llvm/                     # В планах — LLVM AOT (RFC-018)
│   ├── common/                   # Существующий
│   └── runtime/                  # Существующий
│
└── middle/
    └── passes/
        └── ir_normalize.rs       # Новый — общая IR-нормализация (стек → SSA)
                                  #   Совместно используется JIT и LLVM AOT
```

**Ключевые ограничения**:
- `backends/jit/` зависит только от `middle/` (определения IR, проход нормализации), стандартной библиотеки и крейта Cranelift
- `backends/jit/` не зависит от `backends/llvm/`, оба являются равноправными бэкендами
- `backends/jit/` не зависит от `backends/interpreter/`, взаимодействие через интерфейс `FunctionEntry`

### 2. Профилирование горячих точек и многоуровневое срабатывание

#### 2.1 Конечный автомат горячих точек

```
Cold ──(invocation > 50 или backedge > 500)──→ Warm
Warm ──(invocation > 200)────────────────────→ Hot
Hot ──(отправка в очередь компиляции, завершение компиляции)──────→ Compiled
```

> Пороги являются настраиваемыми параметрами, приведённые выше — значения по умолчанию. Диапазон взят из практических значений LuaJIT, JVM C1, V8 Sparkplug (50–1000).

#### 2.2 Счётчики

Каждая функция хранит в `FunctionEntry` (подробности в §4.1) два атомарных счётчика:

```rust
// Поля горячих точек FunctionEntry (полное определение см. в §4.1)
invocation_count: AtomicU32,   // Количество вызовов функции
backedge_count: AtomicU32,     // Количество обратных переходов цикла
state: AtomicU8,               // Cold | Warm | Hot | Compiled
```

#### 2.3 Механизм затухания

Каждые 5 секунд все счётчики сдвигаются вправо на 1 бит (умножение на 0.5). Это предотвращает JIT-компиляцию кода, который при запуске срабатывает часто, но выполняется лишь однажды (например, инициализирующий обход).

```rust
fn decay(entry: &FunctionEntry) {
    entry.invocation_count.fetch_update(Ordering::Relaxed, Ordering::Relaxed, |v| Some(v >> 1));
    entry.backedge_count.fetch_update(Ordering::Relaxed, Ordering::Relaxed, |v| Some(v >> 1));
}
```

Использование побитовых операций — нулевые накладные расходы на деление.

#### 2.4 Очередь компиляции

```
Поток интерпретатора                  Фоновый поток JIT
    │                                       │
    ├─ Достигнут порог Hot                  │
    ├─ Отправка запроса на компиляцию ──────→  │
    │  (не блокирует интерпретатор)            ├─ Извлечение IR функции
    │                                        ├─ IR-нормализация (стек → SSA)
    │                                        ├─ Компиляция Cranelift
    │                                        ├─ Запись в кэш кода
    │                                        └─ Атомарное обновление указателя входа
    │  Следующий вызов этой функции ←────────┘
    │  Прямой переход на нативный код
```

Во время компиляции функция продолжает выполняться через интерпретатор. После завершения компиляции следующий вызов атомарно переключается на JIT-код.

### 3. Конвейер компиляции IR → Cranelift

#### 3.1 Конвейер

```
YaoXiang IR (стековая форма)
  → Проход IR-нормализации (стек → регистры/SSA)    ← Повторно использует RFC-018 §4.0
  → Построение Cranelift IR
  → Оптимизация Cranelift + генерация машинного кода
  → Запись в кэш кода
```

#### 3.2 Соответствие типов YaoXiang → Cranelift

| Тип YaoXiang | Тип Cranelift | Пояснение |
|---------------|---------------|------|
| `Int` | `i64` | |
| `Int32` | `i32` | |
| `Float` | `f64` | |
| `Float32` | `f32` | |
| `Bool` | `i8` | В Cranelift нет `i1`, используется `i8` |
| `Char` | `i32` | Кодовая точка Unicode |
| `String` | `{ i64, i64 }` | Указатель + длина |
| `Void` | Пустой кортеж | |
| `&T` | — | Нулевой размер, исчезает после компиляции |
| `&mut T` | — | Нулевой размер, исчезает после компиляции |
| `ref T` | `{ i64, i64 }` | Указатель со счётчиком ссылок + указатель на данные |
| `*T` | `i64` | Голый указатель |
| `List(T)` | `{ i64, i64, i64 }` | Указатель на данные + длина + ёмкость |
| Структура | Cranelift struct | |
| Record enum | `{ i64, [max_payload] }` | Тег + union |
| `?T` | `{ i8, T }` | Флаг наличия значения + данные |

> Сравнение с таблицей типов LLVM из RFC-018 §3: Cranelift не различает типы указателей, не имеет `i1`, в целом проще.

#### 3.3 Трансляция ключевых инструкций

| IR-инструкция | Cranelift IR |
|---------|-------------|
| `Add { dst, lhs, rhs }` | `iadd` (целое) / `fadd` (с плавающей точкой) |
| `Sub { dst, lhs, rhs }` | `isub` / `fsub` |
| `Mul { dst, lhs, rhs }` | `imul` / `fmul` |
| `Div { dst, lhs, rhs }` | `sdiv` / `udiv` / `fdiv` |
| `Eq { dst, lhs, rhs }` | `icmp eq` / `fcmp eq` |
| `Jmp(label)` | `jump` |
| `JmpIf(cond, label)` | `brnz` |
| `Ret(Some(v))` | `return` |
| `Call { dst, func, args }` | `call` |
| `Load { dst, src }` | `load` |
| `Store { dst, src }` | `store` |
| `Spawn { ... }` | Вызов `task_spawn` рантайма + `task_wait_all` |

> Полная таблица трансляции приведена в основном тексте RFC. Ключевой принцип: набор инструкций Cranelift покрывает все операции IR YaoXiang, семантических пробелов нет.

#### 3.4 Сосуществование двух нормализаций

Интерпретатору VM требуется стековая семантика (`Push`/`Pop`/`Dup`/`Swap`), а Cranelift JIT и LLVM AOT требуют регистровую/SSA. Проход IR-нормализации выполняет преобразование один раз (RFC-018 §4.0), JIT и AOT используют его совместно, не меняя само представление IR. Каждый бэкенд потребляет один и тот же IR по своим потребностям.

### 4. Таблица входов функций и атомарная замена

#### 4.1 FunctionEntry

```rust
struct FunctionEntry {
    /// Атомарно заменяемая цель исполнения
    code_ptr: AtomicPtr<u8>,
    /// Неизменяемые метаданные
    bytecode: &'static [u8],        // Fallback интерпретатора
    ir: &'static FunctionIR,        // Входные данные для JIT-компиляции
    /// Статистика времени выполнения
    invocation_count: AtomicU32,
    backedge_count: AtomicU32,
    state: AtomicU8,                // Cold | Warm | Hot | Compiled
}
```

#### 4.2 Диспетчеризация входа

```
Вызывающий код
  → fn_entry.code_ptr.load(Ordering::Acquire)
  → ┬─ Адрес stub интерпретатора → выполнение интерпретатора, байткод по инструкциям
    └─ Адрес JIT-кода             → прямой переход на нативный код
```

Однократное разыменование указателя. Обработка косвенного перехода предиктором ветвлений современного CPU: первое предсказание ошибочно, далее — всегда верно. Накладные расходы ~1 такт.

#### 4.3 Атомарное переключение

После завершения компиляции — один CAS:

```rust
fn install_jit_code(entry: &FunctionEntry, jit_code: *mut u8) -> bool {
    entry.code_ptr.compare_exchange(
        INTERPRETER_STUB,      // Ожидание: всё ещё указывает на интерпретатор
        jit_code,              // Заменить на: JIT-код
        Ordering::AcqRel,
        Ordering::Acquire,
    ).is_ok()
}
```

Без приостановки интерпретатора, без ожидания на safe-point, без обхода точек вызова. Одна атомарная операция завершает переключение.

### 5. Кэш кода

#### 5.1 Структура

```
CodeCache:
  modules:
    "main.yao":
      functions:
        "compute"    → FunctionEntry (state: Compiled)
        "process"    → FunctionEntry (state: Cold)
        "init"       → FunctionEntry (state: Compiled)
      native_pages:   [ исполняемые страницы mmap ]
    "lib.yao":
      functions:
        "helper"     → FunctionEntry (state: Compiled)
      native_pages:   [ исполняемые страницы mmap ]
```

#### 5.2 Управление исполняемой памятью

```rust
struct NativePage {
    ptr: *mut u8,
    size: usize,
    used: AtomicUsize,     // Использованные байты
    remaining: usize,       // Оставшаяся ёмкость
}

impl CodeCache {
    fn allocate(&self, code_size: usize) -> *mut u8;
    fn deallocate(&self, ptr: *mut u8, code_size: usize);  // Вызывается только при инвалидации модуля
}
```

Каждому модулю выделяется непрерывная исполняемая страница mmap, все JIT-функции модуля размещаются на одной странице. При инвалидации модуля страница освобождается целиком, без необходимости освобождать функции по одной.

### 6. Зарезервированные точки расширения для горячей перезагрузки

Следующие интерфейсы компилируются, но не вызываются до реализации горячей перезагрузки. Принцип проектирования интерфейсов: **при реализации JIT достаточно `insert` и `compare_exchange` для одной функции, модульные операции остаются на горячую перезагрузку**.

```rust
/// Расширенный интерфейс кэша кода (зарезервирован, не реализован)
trait CodeCacheExt {
    /// Инвалидировать весь JIT-код модуля, откатиться на интерпретатор
    fn invalidate_module(&self, module_path: &str);

    /// Инвалидировать функции по диапазону позиций в исходном коде
    fn invalidate_range(&self, file: &str, start: u32, end: u32);

    /// Атомарно заменить таблицу функций всего модуля
    fn swap_module(&self, module_path: &str, new_functions: HashMap<String, FunctionEntry>);
}

/// Расширенный интерфейс очереди компиляции (зарезервирован, не реализован)
trait CompileQueueExt {
    /// Приоритетная вставка (компиляция горячей перезагрузки выше обычной JIT-компиляции)
    fn submit_priority(&self, task: CompileTask);
}
```

**Почему группировка по модулям?** Самому JIT нужны только функции. Организация по модулям существует исключительно ради горячей перезагрузки: после перекомпиляции модуля можно атомарно заменить весь набор функций модуля, а не делать CAS по функциям по одной — последнее при наличии циклических зависимостей между функциями привело бы к несогласованному состоянию.

## Компромиссы

### Преимущества

1. **Нулевая воспринимаемая задержка компиляции**: Cranelift 1–5 мс/функция, фоновый поток, интерпретатор не приостанавливается
2. **Общая инфраструктура**: JIT и AOT разделяют IR-нормализующий проход (RFC-018 §4.0), не изобретаем велосипед
3. **Без разрушительных изменений**: чисто инкрементальная функциональность. VM не меняется, интерпретатор не меняется, просто появляется более быстрый горячий путь
4. **Без зависимости от LLVM**: VM не вводит LLVM, остаётся лёгкой
5. **Естественная мультиплатформенность**: Cranelift изначально поддерживает x86_64 и ARM64, покрывает все целевые платформы
6. **Резерв для горячей перезагрузки**: кэш кода сгруппирован по модулям + косвенные переходы через входы функций — структурная основа для будущей горячей перезагрузки

### Недостатки

1. **Новая зависимость Cranelift**: вводится новый внешний крейт, требуется изучить его API
2. **Сложность отладки**: кадры стека JIT-кода должны быть совместимы с кадрами стека интерпретатора, требуется дополнительная обработка отображения отладочной информации
3. **Задержка прогрева на холодном старте**: в первые секунды после запуска программы ускорения JIT нет, требуется накопление горячих точек
4. **ABI платформ**: для разных платформ (Linux/macOS/Windows) требуется отдельная адаптация mmap и соглашений о вызовах

### Совместимость со связанными RFC

| RFC | Совместимость |
|-----|--------|
| RFC-018 LLVM AOT | ✅ Общий IR-нормализующий проход, JIT и AOT — равноправные бэкенды |
| RFC-024 spawn-блоки | ✅ spawn-блоки компилируются в вызовы функций рантайма |
| RFC-008 архитектура рантайма | ✅ Все три слоя рантайма (Embedded/Standard/Full) поддерживают JIT |

## Альтернативы

| Вариант | Почему не выбран |
|------|--------|
| Только LLVM AOT, без JIT | При разработке требуется перекомпиляция всей программы, теряется опыт быстрых итераций |
| LLVM ORC JIT | Высокая задержка компиляции (10–100 мс), большая зависимость от LLVM, не подходит для встраивания в VM |
| Самодельный лёгкий JIT (dynasm) | Высокая стоимость поддержки ручного бэкенда, менее зрелый, чем Cranelift |
| Шаблонный JIT | Нулевая оптимизация, плохое качество кода, напрасная трата времени JIT-компиляции |
| Whole-program JIT (без интерпретатора) | Медленный холодный старт, простые скрипты не стоят компилировать |

## Зависимости
