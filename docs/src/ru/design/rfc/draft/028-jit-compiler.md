# RFC-028: JIT-компилятор — многоуровневый исполнительный движок внутри VM

> **Ссылки**:
> - [RFC-018: Проект LLVM AOT-компилятора](../review/018-llvm-aot-compiler.md)
> - [RFC-024: Модель параллелизма на основе блоков spawn](../accepted/024-concurrency-model.md)
> - [RFC-008: Разделение модели параллелизма и планировщика в Runtime](../accepted/008-runtime-concurrency-model.md)

## Резюме

Настоящий документ предлагает внедрить в серверную часть VM языка YaoXiang JIT-компилятор на базе Cranelift и перевести VM из чистого интерпретатора в **многоуровневый исполнительный движок**: холодный код исполняется интерпретатором, горячие функции компилируются Cranelift в машинный код. Путь JIT и путь LLVM AOT из RFC-018 совместно используют проход нормализации IR; Cranelift отвечает за быструю компиляцию в JIT, LLVM — за глубокую оптимизацию в AOT. Каждый инструмент решает свою задачу.

**Ключевая позиция: JIT обслуживает VM, а не заменяет её.**

## Мотивация

### Зачем нужен JIT?

Серверная часть VM в настоящее время является чистым интерпретатором и работает в 10–100 раз медленнее машинного кода. Во время разработки тесты, скрипты и локальная отладка запускаются часто — в этих сценариях не нужна предельная оптимизация AOT, но требуется заметно более высокая скорость по сравнению с интерпретатором.

### Почему не только LLVM AOT?

AOT-компиляция LLVM занимает много времени (порядка секунд) и не подходит для итераций разработки. Разработчику нужна возможность «изменил и сразу запустил»: изменил строку кода → перезапустил → почти мгновенно увидел результат. Cranelift компилирует отдельную функцию за 1–5 мс, и пользователь не ощущает задержки компиляции.

### Почему Cranelift, а не LLVM ORC JIT?

| Аспект | Cranelift JIT | LLVM ORC JIT |
|------|--------------|--------------|
| Скорость компиляции | 1–5 мс/функция | 10–100 мс/функция |
| Размер зависимостей | Небольшой | Большой (требуется полный LLVM) |
| Качество кода | 70–80% от LLVM -O2 | Превосходное |
| Применимые сценарии | Разработка и отладка, быстрые итерации | Не применим (см. компромиссы в данном документе) |

Cranelift компилирует быстро, а качества кода достаточно. LLVM остаётся для глубокой автономной оптимизации в AOT. Один инструмент — одна задача.

## Предложение

### Ключевая архитектура

```
Исполнительный движок VM
├── Слой интерпретатора
│   ├── Исполняет байткод-инструкции
│   ├── Собирает данные о горячих точках (invocation count + loop backedge count)
│   └── При достижении порога → ставит задачу компиляции
│
├── Слой JIT-компиляции (бэкенд Cranelift)
│   ├── Очередь компиляции (фоновый поток, не блокирует интерпретатор)
│   ├── IR → нормализация → Cranelift IR → машинный код
│   └── Совместное использование прохода нормализации IR из RFC-018 §4.0 (стек → SSA)
│
├── Кэш кода
│   ├── Таблица функций: ID функции → {вход в интерпретатор, вход в JIT (опц.)}
│   ├── Атомарная замена входа функции после компиляции
│   └── Группировка по модулям (зарезервировано для интерфейса горячей перезагрузки)
│
└── Профилирование горячих точек
    ├── Счётчик вызовов и счётчик обратных рёбер цикла для каждой функции
    ├── Периодическое затухание (чтобы одноразовый прогрев не вызывал компиляцию)
    └── Четыре уровня: Cold → Warm → Hot → Compiled
```

### Сопряжение с существующей архитектурой

```
Исходный код → Фронтенд (общий) → IR → ┬→ codegen байткода → Интерпретатор VM → [горячие функции] → Cranelift JIT
                                         │
                                         └→ codegen LLVM AOT → .o → Линковка → exe (продакшн)
```

JIT и AOT совместно используют **проход нормализации IR** (`middle/passes/ir_normalize.rs`); низкоуровневый codegen заменяется с LLVM на Cranelift.

### Поток исполнения

```
Вызов функции
  → fn_entry.code_ptr.load()
  → ┬─ Заглушка интерпретатора (холодное состояние): интерпретация байткода по одной инструкции
    └─ Машинный код JIT (горячее состояние): прямой переход на машинный код
  → Возврат
```

## Детальное проектирование

### 1. Структура каталогов

```
src/
├── backends/
│   ├── interpreter/              # Существующий — интерпретатор VM
│   │   └── executor/
│   │       ├── engine.rs         # Изменения — точка входа: диспетчеризация через FunctionEntry вместо прямой интерпретации
│   │       └── ...
│   │
│   ├── jit/                      # Новый — слой JIT-компиляции
│   │   ├── mod.rs                # Вход в модуль JIT, инициализация контекста Cranelift
│   │   ├── profiler.rs           # Счётчики горячих точек, затухание, пороговые решения
│   │   ├── entry.rs              # FunctionEntry + управление AtomicPtr
│   │   ├── cache.rs              # Кэш кода (управление исполняемыми страницами через mmap)
│   │   ├── compiler.rs           # IR → Cranelift IR → машинный код
│   │   ├── types.rs              # Сопоставление типов YaoXiang → Cranelift
│   │   └── abi.rs                # Соглашение о вызовах (System V / Microsoft x64)
│   │
│   ├── llvm/                     # В планах — LLVM AOT (RFC-018)
│   ├── common/                   # Существующий
│   └── runtime/                  # Существующий
│
└── middle/
    └── passes/
        └── ir_normalize.rs       # Новый — общая нормализация IR (стек → SSA)
                                  #   Совместно используется JIT и LLVM AOT
```

**Ключевые ограничения**:
- `backends/jit/` зависит только от `middle/` (определения IR, проходы нормализации), стандартной библиотеки и крейта Cranelift
- `backends/jit/` не зависит от `backends/llvm/` — это два параллельных бэкенда
- `backends/jit/` не зависит от `backends/interpreter/` и взаимодействует через интерфейс `FunctionEntry`

### 2. Профилирование горячих точек и многоуровневое срабатывание

#### 2.1 Конечный автомат состояний

```
Cold ──(invocation > 50 или backedge > 500)──→ Warm
Warm ──(invocation > 200)────────────────────→ Hot
Hot ──(постановка в очередь компиляции, завершение компиляции)──→ Compiled
```

> Пороговые значения являются настраиваемыми; выше указаны значения по умолчанию. Диапазон (50–1000) основан на реальных порогах LuaJIT, JVM C1 и V8 Sparkplug.

#### 2.2 Счётчики

Каждая функция поддерживает в `FunctionEntry` (см. §4.1) два атомарных счётчика:

```rust
// Поля горячих точек в FunctionEntry (полное определение см. в §4.1)
invocation_count: AtomicU32,   // Сколько раз функция была вызвана
backedge_count: AtomicU32,     // Сколько раз было пройдено обратное ребро цикла
state: AtomicU8,              // Cold | Warm | Hot | Compiled
```

#### 2.3 Механизм затухания

Каждые 5 секунд все счётчики сдвигаются вправо на 1 бит (умножение на 0,5). Это предотвращает бессмысленную JIT-компиляцию кода, который при запуске выполняется часто, но в дальнейшем вызывается лишь однократно (например, обход инициализации).

```rust
fn decay(entry: &FunctionEntry) {
    entry.invocation_count.fetch_update(Ordering::Relaxed, Ordering::Relaxed, |v| Some(v >> 1));
    entry.backedge_count.fetch_update(Ordering::Relaxed, Ordering::Relaxed, |v| Some(v >> 1));
}
```

Используются побитовые операции — без накладных расходов на деление.

#### 2.4 Очередь компиляции

```
Поток интерпретатора                    Фоновый поток JIT
    │                                        │
    ├─ Достигнут порог Hot                   │
    ├─ Отправка запроса на компиляцию ───→  │
    │  (не блокирует интерпретатор)           ├─ Извлечение IR функции
    │                                        ├─ Нормализация IR (стек → SSA)
    │                                        ├─ Компиляция Cranelift
    │                                        ├─ Запись в кэш кода
    │                                        └─ Атомарное обновление указателя входа
    │  Следующий вызов этой функции ←──────  │
    │  Напрямую использует машинный код       │
```

Во время компиляции функция продолжает исполняться через интерпретатор. После завершения компиляции следующий вызов атомарно переключается на JIT-код.

### 3. Конвейер IR → Cranelift

#### 3.1 Конвейер

```
YaoXiang IR (стековая форма)
  → Проход нормализации IR (стек → регистры/SSA)    ← повторное использование RFC-018 §4.0
  → Построение Cranelift IR
  → Оптимизации Cranelift + генерация машинного кода
  → Запись в кэш кода
```

#### 3.2 Сопоставление типов YaoXiang → Cranelift

| Тип YaoXiang | Тип Cranelift | Пояснение |
|---------------|---------------|-----------|
| `Int` | `i64` | |
| `Int32` | `i32` | |
| `Float` | `f64` | |
| `Float32` | `f32` | |
| `Bool` | `i8` | В Cranelift нет `i1`, используется `i8` |
| `Char` | `i32` | Кодовая точка Unicode |
| `String` | `{ i64, i64 }` | Указатель + длина |
| `Void` | Пустой кортеж | |
| `&T` | — | Нулевой размер, исчезает после компиляции |
| `&mut T` | — | Нулевой размер, исчезает после компиляции |
| `ref T` | `{ i64, i64 }` | Указатель со счётчиком ссылок + указатель на данные |
| `*T` | `i64` | Голый указатель |
| `List(T)` | `{ i64, i64, i64 }` | Указатель на данные + длина + ёмкость |
| Структура | Cranelift struct | |
| Запись-перечисление | `{ i64, [max_payload] }` | Тег + union |
| `?T` | `{ i8, T }` | Флаг наличия значения + данные |

> По сравнению с таблицей типов LLVM из RFC-018 §3: Cranelift не различает типы указателей и не имеет `i1`; в целом всё проще.

#### 3.3 Трансляция ключевых инструкций

| IR-инструкция | Cranelift IR |
|---------|-------------|
| `Add { dst, lhs, rhs }` | `iadd` (целое) / `fadd` (вещественное) |
| `Sub { dst, lhs, rhs }` | `isub` / `fsub` |
| `Mul { dst, lhs, rhs }` | `imul` / `fmul` |
| `Div { dst, lhs, rhs }` | `sdiv` / `udiv` / `fdiv` |
| `Eq { dst, lhs, rhs }` | `icmp eq` / `fcmp eq` |
| `Jmp(label)` | `jump` |
| `JmpIf(cond, label)` | `brnz` |
| `Ret(Some(v))` | `return` |
| `Call { dst, func, args }` | `call` |
| `Load { dst, src }` | `load` |
| `Store { dst, src }` | `store` |
| `Spawn { ... }` | Вызов рантайм-функций `task_spawn` + `task_wait_all` |

> Полная таблица трансляции приведена в основном тексте RFC. Главный принцип: набор инструкций Cranelift покрывает все операции IR YaoXiang, семантических пробелов нет.

#### 3.4 Сосуществование двух форм нормализации

Интерпретатору VM нужна стековая семантика (`Push`/`Pop`/`Dup`/`Swap`), а Cranelift JIT и LLVM AOT нужны регистры/SSA. Проход нормализации IR выполняет одно преобразование (RFC-018 §4.0); JIT и AOT используют его совместно, не изменяя само представление IR. Каждый бэкенд потребляет один и тот же IR по-своему.

### 4. Таблица входов функций и атомарная замена

#### 4.1 FunctionEntry

```rust
struct FunctionEntry {
    /// Атомарно заменяемая цель исполнения
    code_ptr: AtomicPtr<u8>,
    /// Неизменяемые метаданные
    bytecode: &'static [u8],        // Резервный путь в интерпретатор
    ir: &'static FunctionIR,        // Входные данные для JIT-компиляции
    /// Статистика рантайма
    invocation_count: AtomicU32,
    backedge_count: AtomicU32,
    state: AtomicU8,                // Cold | Warm | Hot | Compiled
}
```

#### 4.2 Диспетчеризация входа

```
Вызывающий код
  → fn_entry.code_ptr.load(Ordering::Acquire)
  → ┬─ Адрес заглушки интерпретатора → интерпретация байткода по одной инструкции
    └─ Адрес JIT-кода                  → прямой переход на машинный код
```

Одно разыменование указателя. Поведение предсказателя переходов современного CPU при косвенных переходах: первое предсказание ошибочно, далее — безошибочно. Накладные расходы — порядка 1 такта.

#### 4.3 Атомарное переключение

После завершения компиляции — одна CAS-операция:

```rust
fn install_jit_code(entry: &FunctionEntry, jit_code: *mut u8) -> bool {
    entry.code_ptr.compare_exchange(
        INTERPRETER_STUB,      // Ожидание: всё ещё указывает на интерпретатор
        jit_code,              // Заменить на: JIT-код
        Ordering::AcqRel,
        Ordering::Acquire,
    ).is_ok()
}
```

Без приостановки интерпретатора, без ожидания в safepoint, без обхода точек вызова. Переключение выполняется одной атомарной операцией.

### 5. Кэш кода

#### 5.1 Структура

```
CodeCache:
  modules:
    "main.yao":
      functions:
        "compute"    → FunctionEntry (state: Compiled)
        "process"    → FunctionEntry (state: Cold)
        "init"       → FunctionEntry (state: Compiled)
      native_pages:   [ исполняемые страницы, выделенные через mmap ]
    "lib.yao":
      functions:
        "helper"     → FunctionEntry (state: Compiled)
      native_pages:   [ исполняемые страницы, выделенные через mmap ]
```

#### 5.2 Управление исполняемой памятью

```rust
struct NativePage {
    ptr: *mut u8,
    size: usize,
    used: AtomicUsize,     // Использованные байты
    remaining: usize,       // Оставшаяся ёмкость
}

impl CodeCache {
    fn allocate(&self, code_size: usize) -> *mut u8;
    fn deallocate(&self, ptr: *mut u8, code_size: usize);  // Вызывается только при инвалидации модуля
}
```

Каждому модулю выделяется непрерывная последовательность исполняемых страниц mmap; все JIT-функции модуля размещаются на одной странице. При инвалидации модуля страница освобождается целиком — не нужно освобождать функции по одной.

### 6. Зарезервированные точки расширения для горячей перезагрузки

Перечисленные ниже интерфейсы компилируются, но до реализации горячей перезагрузки не вызываются. Принцип проектирования интерфейсов: **при реализации JIT достаточно `insert` и `compare_exchange` для одной функции; операции уровня модуля остаются для горячей перезагрузки.**

```rust
/// Расширенный интерфейс кэша кода (зарезервирован, не реализован)
trait CodeCacheExt {
    /// Инвалидировать весь JIT-код модуля, откатиться на интерпретатор
    fn invalidate_module(&self, module_path: &str);

    /// Инвалидировать конкретные функции по диапазону позиций в исходном коде
    fn invalidate_range(&self, file: &str, start: u32, end: u32);

    /// Атомарно заменить таблицу функций всего модуля
    fn swap_module(&self, module_path: &str, new_functions: HashMap<String, FunctionEntry>);
}

/// Расширенный интерфейс очереди компиляции (зарезервирован, не реализован)
trait CompileQueueExt {
    /// Вставка с приоритетом (компиляция горячей перезагрузки выше обычной JIT-компиляции)
    fn submit_priority(&self, task: CompileTask);
}
```

**Зачем группировать по модулям?** Самому JIT нужны только функции. Организация по модулям существует исключительно ради горячей перезагрузки: после перекомпиляции модуля можно атомарно заменить весь набор его функций, а не делать CAS по одной — иначе при наличии циклических зависимостей между функциями возможно рассогласованное состояние.

## Компромиссы

### Преимущества

1. **Незаметная задержка компиляции**: Cranelift — 1–5 мс/функция, компиляция в фоновом потоке, интерпретатор не приостанавливается
2. **Совместная инфраструктура**: JIT и AOT совместно используют проход нормализации IR (RFC-018 §4.0), колесо не изобретается заново
3. **Без разрушительных изменений**: чисто инкрементальная функциональность. VM не меняется, интерпретатор не меняется — появляется лишь более быстрый горячий путь
4. **Без зависимости от LLVM**: VM не подтягивает LLVM и остаётся лёгкой
5. **Естественная мультиплатформенность**: Cranelift изначально поддерживает x86_64 и ARM64, что покрывает все целевые платформы
6. **Задел под горячую перезагрузку**: группировка кэша кода по модулям + косвенные переходы на вход функций формируют структурную основу для будущей горячей перезагрузки

### Недостатки

1. **Новая зависимость от Cranelift**: вводится новый внешний крейт, требуется изучение его API
2. **Сложность отладки**: кадры стека, сгенерированные JIT, должны быть совместимы с кадрами стека интерпретатора, отображение отладочной информации требует дополнительной обработки
3. **Задержка прогрева при холодном старте**: в первые несколько секунд после запуска программы JIT-ускорения нет — нужно накопление статистики горячих точек
4. **ABI платформ**: разные платформы (Linux/macOS/Windows) требуют отдельной адаптации для mmap и соглашений о вызовах

### Согласованность с другими RFC

| RFC | Согласованность |
|-----|--------|
| RFC-018 LLVM AOT | ✅ Совместное использование прохода нормализации IR, JIT и AOT — параллельные бэкенды |
| RFC-024 блоки spawn | ✅ Блоки spawn компилируются в вызовы рантайм-функций |
| RFC-008 архитектура Runtime | ✅ Все три уровня Runtime (Embedded/Standard/Full) поддерживают JIT |

## Альтернативы

| Альтернатива | Почему не выбрана |
|------|--------|
| Только LLVM AOT, без JIT | При разработке потребуется перекомпилировать всю программу — теряется опыт быстрых итераций |
| LLVM ORC JIT | Высокая задержка компиляции (10–100 мс), большая зависимость от LLVM, не подходит для встраивания в VM |
| Собственный лёгкий JIT (dynasm) | Высокие затраты на поддержку самописного бэкенда, Cranelift гораздо зрелее |
| Шаблонный JIT | Нулевая оптимизация, низкое качество кода, время JIT-компиляции тратится впустую |
| Полностью программный JIT (без интерпретатора) | Медленный холодный старт; простые скрипты не оправдывают компиляцию |

## Зависимости

- RFC-018 (LLVM AOT) → совместно используемый проход нормализации IR
- RFC-024 (блоки spawn) → JIT-компиляция блоков spawn
- RFC-008 (архитектура Runtime) → поддержка JIT на всех трёх уровнях Runtime
- Крейт Cranelift → бэкенд JIT
