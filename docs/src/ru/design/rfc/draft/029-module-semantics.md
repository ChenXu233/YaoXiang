# RFC-029: Семантическая система модулей

## Резюме

Интеграция модульной системы в конвейер компиляции для реализации многофайловой компиляции, управления видимостью на уровне модулей и горячей перезагрузки. **Не вводятся правило сирот или проверка согласованности** — трейты в YaoXiang являются структурными типами (RFC-011 §2.1) и не требуют отслеживания номинального `impl` в стиле Rust.

## Мотивация

### Текущая проблема

**Физический уровень** модульной системы (загрузка, разбор, кэширование, граф зависимостей, горячая перезагрузка) полностью реализован (`frontend/module/`), но **не интегрирован в конвейер компиляции**:

- `pipeline.rs` принимает только одну исходную строку и не поддерживает многофайловые проекты
- Инструкции `use` не могут фактически загружать модули при проверке типов
- `ModuleCache`, `HotReloader`, `VendorLoader` реализованы, но не имеют вызывающих сторон
- Встроенные функции стандартной библиотеки жёстко зарегистрированы через `ModuleRegistry::with_std()` и не проходят по общему пути загрузки модулей

### Почему правило сирот не требуется

RFC-011 определяет трейт как структурный тип:

```yaoxiang
Clone: Type = { clone: (Self) -> Self }
```

- **Нет `impl Trait for Type`** — методы определяются непосредственно на типе
- **Нет правила сирот** — любой модуль может добавлять методы к своим типам
- **Нет проверки согласованности** — методы являются частью структуры типа, без номинального сопоставления

Следовательно, `TraitImplementation` не нуждается в полях `defined_in` или `module`. Связанные issue #46 и #73 закрыты.

## Предложение

### Основной дизайн

**Интеграция в два слоя**:

```text
┌─────────────────────────────────────────────────┐
│  Интеграция конвейера (Pipeline Integration)    │  ← Новое: многофайловая компиляция, загрузка модулей
├─────────────────────────────────────────────────┤
│  Видимость (Visibility)                         │  ← Новое: pub / по умолчанию (видимость в модуле)
└─────────────────────────────────────────────────┘
```

### 1. Многофайловая компиляция

Точка входа компилятора расширяется с одного файла до директории проекта:

```rust
/// Компилирует проект (а не отдельный файл)
pub fn compile_project(&mut self, project_root: &Path) -> Result<Vec<ModuleIR>, CompileError> {
    // 1. Прочитать yaoxiang.toml для получения точки входа
    // 2. Рекурсивно загрузить зависимые модули из точки входа
    // 3. Топологически отсортировать граф зависимостей
    // 4. Скомпилировать каждый модуль по порядку
    // 5. Межмодульная проверка типов (разрешение инструкций use)
}
```

Точка интеграции: новый метод `compile_project` в `compiler.rs`, внутри используется `ModuleLoader` для загрузки модулей.

### 2. Разрешение модулей в инструкции `use`

В настоящее время `statements.rs` содержит `ModuleRegistry`, но использует его только для запросов регистрации. Необходимо расширить его до фактической загрузки:

```yaoxiang
# Текущее поведение: инструкция use не может найти модуль при проверке типов
use math.geometry.Point  # ❌ В ModuleRegistry отсутствует math.geometry

# Целевое поведение: инструкция use инициирует загрузку модуля
use math.geometry.Point  # ✅ ModuleLoader загружает math/geometry.yx, извлекает экспорт Point
```

Путь реализации:
1. Инструкция `use` инициирует `ModuleLoader::load()`
2. Результат загрузки регистрируется в `ModuleRegistry`
3. Проверщик типов запрашивает экспортируемые типы из `ModuleRegistry`

### 3. Система видимости

```yaoxiang
# math/geometry.yx
pub type Point = { x: Int, y: Int }       # pub = доступен из других модулей
type InternalState = { cache: Int }        # по умолчанию = виден только внутри модуля geometry

pub Point.distance: (self: Point, other: Point) -> Float = {
    # ...
}
```

```rust
/// Уровень видимости
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum Visibility {
    /// Публичный — доступен из всех модулей
    Public,
    /// По умолчанию — виден только в определяющем модуле
    Module,
}
```

Проверщик типов проверяет видимость при межмодульных ссылках.

### 4. Кэш модулей

`ModuleCache` уже реализует стратегии кэширования LRU/TTL. После интеграции в конвейер компиляции:
- Первая компиляция: загрузка + компиляция + кэширование
- Последующие компиляции: при попадании в кэш — пропуск
- Изменения файлов: `HotReloader` автоматически инвалидирует устаревший кэш

### 5. Интеграция горячей перезагрузки

`HotReloader` полностью реализован (`frontend/module/hot_reload.rs`), необходимо интегрировать его в конвейер компиляции:

```rust
// При запуске конвейера компиляции
let mut reloader = HotReloader::new(project_root, config, cache.clone());
let mut event_rx = reloader.start()?;

// В асинхронном главном цикле
tokio::spawn(async move {
    while let Some(event) = event_rx.recv().await {
        for module in &event.affected_modules {
            pipeline.recompile_module(module).await;
        }
    }
});
```

## Изменения в компиляторе

| Компонент | Изменения |
|------|------|
| `compiler.rs` | Новый метод `compile_project` |
| `pipeline.rs` | Поддержка компиляции нескольких модулей, запросов к кэшу модулей |
| `typecheck/inference/statements.rs` | Инструкция `use` инициирует загрузку модуля |
| `typecheck/mod.rs` | Регистрация встроенных функций из путей общих модулей (вместо жёсткого кодирования) |
| `frontend/module/loader.rs` | Уже реализован, изменений не требуется |
| `frontend/module/cache.rs` | Уже реализован, изменений не требуется |
| `frontend/module/hot_reload.rs` | Уже реализован, изменений не требуется |
| Уровень AST | Аннотация видимости типов ключевым словом `pub` (если ещё не поддерживается) |

## Стратегия реализации

### Разбивка на этапы

**Фаза 1: Точка входа многофайловой компиляции**
1. Добавить метод `compile_project(project_root)` в `compiler.rs`
2. Использовать `ModuleLoader` для рекурсивной загрузки зависимостей из точки входа
3. Использовать `ModuleDependencyGraph` для топологической сортировки
4. Последовательно вызывать существующий поток компиляции отдельных файлов

**Фаза 2: Разрешение модулей в инструкции `use`**
5. В `statements.rs` инструкция `use` инициирует `ModuleLoader::load()`
6. Результат загрузки регистрируется в `ModuleRegistry`
7. Экспортируемые типы доступны при проверке типов

**Фаза 3: Видимость**
8. Разбор аннотации `pub` на уровне AST для типов
9. Проверщик типов проверяет видимость при межмодульных ссылках

**Фаза 4: Кэширование и горячая перезагрузка**
10. Интеграция `ModuleCache` в `pipeline.rs`
11. Интеграция `HotReloader` в `pipeline.rs`
12. Инкрементальная перекомпиляция обрабатывает только затронутые модули

### Зависимости

- RFC-014 (Менеджер пакетов) — имена пакетов берутся из `yaoxiang.toml`
- RFC-011 (Система дженериков) — трейты являются структурными типами, не затрагивают принадлежность модулей
