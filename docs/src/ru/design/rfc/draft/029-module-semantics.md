# RFC-029: Семантическая система модулей

## Аннотация

Подключить модульную систему к конвейеру компиляции, реализовать многофайловую компиляцию и управление видимостью на уровне пакета.

**Основной принцип**: проверщик типов только запрашивает предварительно построенный реестр модулей и не обращается к диску. Граф модулей полностью строится до проверки типов.

**Не входит**: кэширование, наблюдение за файлами, горячая перезагрузка, инкрементальная перекомпиляция. Это оптимизации жизненного цикла компиляции и относятся к последующим отдельным RFC.

## Мотивация

### Текущие проблемы

1. **Компилятор поддерживает только один файл**: `Compiler::compile(name, source)` не может обрабатывать межфайловые зависимости
2. **Правила экспорта конфликтуют друг с другом**: автоматический экспорт типов, автоматический экспорт констант, автоматический экспорт методов, проверка `pub` для функций — четыре набора исключений
3. **Два резолвера модулей**: `frontend/module/resolver.rs` и `package/source/module_resolver.rs` имеют разный порядок поиска
4. **Проверщик типов связан с загрузкой файлов**: в черновике требовалось, чтобы `use` запускал `ModuleLoader::load()` во время проверки типов

### Цели проектирования

- Один проект может компилировать несколько файлов `.yx`
- Семантика операторов `use` должна быть ясной и недвусмысленной
- Правила видимости унифицированы в одно
- Однофайловый режим продолжает работать без обязательного `yaoxiang.toml`
- Проверщик типов является чистой логикой, не выполняет файловый ввод-вывод

## Предложение

### 1. Идентичность модулей и разрешение путей

#### Определение модуля

**Модуль** — это файл `.yx`. Путь модуля представляет собой разделённый точками именованный путь, соответствующий расположению в файловой системе.

```
math.geometry → src/math/geometry.yx
             → src/math/geometry/mod.yx
             → src/math/geometry/index.yx
```

**Пакет** — это проект, содержащий файл `yaoxiang.toml` и включающий несколько модулей. Пакет является единственной границей инкапсуляции.

#### Правила разрешения путей

Порядок поиска (единственное правило, заменяющее существующие два резолвера):

1. **Стандартная библиотека**: `std` или `std.*` → встроенные модули, запрашиваются из `ModuleRegistry`
2. **Каталог vendor**: `.yaoxiang/vendor/<pkg>-*/src/` → пакеты зависимостей
3. **Относительный путь от текущего файла**: относительно каталога, в котором находится текущий файл `.yx`
4. **Каталог src проекта**: `<project_root>/src/`

Порядок попыток обнаружения файла:

```
base/name.yx
base/name/mod.yx
base/name/index.yx
```

Поиск останавливается на первом существующем файле. Если `name.yx` и `name/mod.yx` существуют одновременно, выдаётся ошибка:

```
Неоднозначность пути модуля: `math.geometry` соответствует одновременно:
  src/math/geometry.yx
  src/math/geometry/mod.yx
Пожалуйста, удалите один из них.
```

#### Унифицированный резолвер

Упразднить два существующих `ModuleResolver`. Сохранить `frontend/module/resolver.rs` как единственную реализацию, удалить `package/source/module_resolver.rs`. Поддержка переменной окружения `YXPATH` объединяется в единый резолвер.

### 2. Семантика импорта

#### Синтаксические формы

```yaoxiang
use math.geometry                          # пространство имён модуля
use math.geometry as geo                   # псевдоним пространства имён модуля
use math.geometry.{Point}                  # выборочный импорт
use math.geometry.{Point, distance}        # множественный выборочный импорт
use math.geometry.{Point as P}             # выборочный импорт с псевдонимом
use math.geometry.{Point as P, distance as dist}  # множественный с псевдонимами
```

#### Семантика

Все формы импорта — это **правила разрешения имён на этапе компиляции**, а не копии ссылок во время выполнения. Импортируемые имена указывают на идентичность объявлений в таблице экспорта модуля.

| Синтаксис | Привязка в текущей области видимости | Способ использования |
|------|-----------------|----------|
| `use path` | последний сегмент path как пространство имён модуля | `geometry.Point` |
| `use path as alias` | alias как пространство имён модуля | `alias.Point` |
| `use path.{item}` | сам item | `item` |
| `use path.{item as alias}` | сам alias | `alias` |

#### Удаляемые синтаксические формы

- ~~`from path use item`~~: не принимается форма Python from-import
- ~~`use path.*`~~: импорт с подстановочным знаком создаёт риск конфликтов, импорт пространства имён модуля достаточен
- ~~`use path.{a, b} as c, d`~~: параллельные списки, сопоставляемые по позиции, являются хрупкой структурой данных, псевдонимы должны следовать за каждым объявлением: `use path.{a as c, b as d}`

#### Семантика пути

`path` в `use path` всегда является **путём модуля**, а не объявлением. Если модуль не найден, сразу выдаётся ошибка:

```
Модуль `math.geometry.Point` не найден.
Если `Point` является объявлением в модуле `math.geometry`, используйте:
use math.geometry.{Point}
```

Не выполняется fallback «сначала искать полный модуль, при неудаче считать последний сегмент объявлением».

#### Конфликты импорта

Импорт одноимённых объектов сразу выдаёт ошибку, без тихого перезаписания:

```
Конфликт импорта имени `Point`:
  math.geometry.Point
  graphics.geometry.Point
Пожалуйста, используйте выборочный импорт или псевдоним пространства имён модуля.
```

### 3. Видимость

#### Правило

Пакет является единственной границей инкапсуляции. Модуль не несёт границу полномочий.

| Запись | Внутри текущего пакета | Другие пакеты |
|------|:--------:|:------:|
| По умолчанию (без `pub`) | ✅ | ❌ |
| `pub` | ✅ | ✅ |

**Одно правило, применяемое ко всем объявлениям верхнего уровня**: типам, функциям, константам, методам.

Упраздняются четыре набора исключений в существующем коде:

- ~~Определения типов всегда экспортируются~~ → то же правило
- ~~Константы автоматически экспортируются~~ → то же правило
- ~~Методы автоматически экспортируются~~ → то же правило
- ~~Только функции проверяют `pub`~~ → то же правило

#### Структура данных

Заменить `is_pub: bool` в AST на:

```rust
pub enum Visibility {
    Package,  // по умолчанию: видимо в текущем пакете
    Public,   // pub: видимо во всех пакетах
}
```

#### Таблица экспорта

Каждый модуль поддерживает две таблицы:

- **PackageSymbols**: полная таблица символов внутри пакета, включающая все объявления верхнего уровня
- **PublicExports**: подмножество объявлений с `pub`, предоставляемых другим пакетам

`use` внутри одного пакета запрашивает `PackageSymbols`; `use` между пакетами может запрашивать только `PublicExports`.

Ссылка на объявление без `pub` из другого пакета сразу выдаёт ошибку:

```
`internalHelper` в модуле `math.geometry` невидим.
Это не объявление с pub, его можно использовать только в пакете `math`.
```

### 4. Процесс компиляции проекта

#### Конвейер компиляции

```
Точка входа проекта
  → прочитать yaoxiang.toml для получения файла точки входа
  → рекурсивно разобрать операторы use от точки входа, обнаружить все зависимые модули
  → построить граф зависимостей модулей (ModuleDependencyGraph)
  → обнаружить циклические зависимости
  → выполнить топологическую сортировку
  → для каждого модуля в порядке сортировки: лексический анализ → синтаксический анализ → извлечение экспорта
  → построить ModuleRegistry (содержащий таблицы экспорта всех модулей)
  → выполнить проверку типов для каждого модуля в топологическом порядке (запрашивая ModuleRegistry)
  → сгенерировать несколько ModuleIR
  → агрегировать диагностику
```

Проверщик типов **только запрашивает** предварительно построенный `ModuleRegistry`, не выполняет загрузку файлов, не обращается к диску.

#### Выбор файла точки входа

Приоритет:

1. `[run].main` (если существует)
2. `path` первой записи `[[bin]]`
3. `path` из `[lib]`
4. `src/main.yx` (соглашение по умолчанию)

Однофайловый режим не требует `yaoxiang.toml`, компилируется указанный файл напрямую.

#### Циклические зависимости

```
Обнаружена циклическая зависимость:
  math.geometry → math.transform → math.geometry
```

Циклическая зависимость является ошибкой компиляции, никакой специальной обработки не выполняется.

#### Агрегация ошибок

Ошибки многофайловой компиляции агрегируются в топологическом порядке модулей. Каждая ошибка помечается исходным модулем и позицией в файле:

```
Ошибка: в модуле `math.geometry`:
  src/math/geometry.yx:12:5
  Тип `Circle` не определён

Ошибка: в модуле `app.main`:
  src/main.yx:3:1
  Модуль `math.geometry` невидим
```

### 5. Изменения в компиляторе

| Компонент | Изменение |
|------|------|
| `compiler.rs` | добавить метод `compile_project(project_root)` |
| `pipeline.rs` | сохранить обязанности компиляции одного модуля, не превращать в объект-бог |
| `typecheck/checker.rs` | операторы `use` запрашивают `ModuleRegistry`, не запускают загрузку файлов |
| `typecheck/inference/statements.rs` | то же самое, `process_use_stmt` только запрашивает, не загружает |
| `frontend/module/resolver.rs` | объединить поддержку YXPATH из `package/source/module_resolver.rs`, стать единственным резолвером |
| `frontend/module/loader.rs` | расширить: поддержка рекурсивного обнаружения, построения полного графа модулей |
| `frontend/module/dep_graph.rs` | уже реализован, повторно использовать топологическую сортировку и обнаружение циклов |
| `frontend/module/registry.rs` | уже реализован, повторно использовать запросы к таблице экспорта |
| `frontend/module/cache.rs` | уже реализован, в этом RFC не подключается к конвейеру компиляции |
| `frontend/module/hot_reload.rs` | уже реализован, в этом RFC не подключается к конвейеру компиляции |
| AST `is_pub: bool` | заменить на перечисление `Visibility` |
| `package/source/module_resolver.rs` | удалить, обязанности объединяются в `frontend/module/resolver.rs` |

## Стратегия реализации

### Разбиение на этапы

**Фаза 1: Унификация разрешения модулей**
1. Объединить два `ModuleResolver`, удалить `package/source/module_resolver.rs`
2. Поддержать переменную окружения `YXPATH`
3. Обнаружение неоднозначности путей модулей

**Фаза 2: Структура данных видимости**
4. AST `is_pub: bool` → перечисление `Visibility`
5. Парсер поддерживает отображение ключевого слова `pub` в `Visibility::Public`
6. `ModuleLoader::extract_exports` единообразно использует `Visibility` для определения экспорта

**Фаза 3: Точка входа компиляции проекта**
7. Добавить метод `compile_project(project_root)` в `compiler.rs`
8. Рекурсивно обнаруживать модули от точки входа, построить `ModuleDependencyGraph`
9. Топологическая сортировка, последовательная загрузка модулей и извлечение экспорта
10. Построить полный `ModuleRegistry`
11. Выполнить проверку типов для каждого модуля в топологическом порядке
12. Сгенерировать несколько `ModuleIR`, агрегировать диагностику

**Фаза 4: Синтаксис импорта**
13. Реализовать синтаксис `use path.{item as alias}`
14. Устранить догадки fallback в конце пути

### Зависимости

- RFC-014 (менеджер пакетов) — имена берутся из `yaoxiang.toml`, структура каталога vendor
- RFC-011 (система обобщений) — trait является структурным типом, не затрагивает принадлежность модулей
- RFC-009 (модель владения) — импорт модуля — это разрешение имён на этапе компиляции, не затрагивает копирование ссылок во время выполнения

## Планирование дочерних RFC

Следующие дочерние RFC находятся **в предполагаемом планировании**, работа над их черновиками ещё не начата:

| Дочерний RFC | Возможность (предполагается) | Предусловие (предполагается) |
|--------|-------------|-----------------|
| 029a | Кэширование модулей и инкрементальная перекомпиляция | граф модулей и таблицы экспорта стабилизированы |
| 029b | Наблюдение за файлами и горячая перезагрузка | механизм инвалидации кэша из 029a |
| 029c | Реэкспорт (`pub use`) | таблица экспорта и правила видимости реализованы |
| 029d | Аргумент CLI `--entry` для переопределения выбора точки входа | точка входа компиляции проекта доступна |
| 029e | Формат вывода многофайловой диагностики `--json` | механизм агрегации диагностики доступен |
| — | `pub(package)` — видимость только в пакете | в настоящее время нет реальной потребности, временно не включается |
| — | Компиляция мультипакетного рабочего пространства | возлагается на RFC-014c |
