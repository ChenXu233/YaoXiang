# RFC-019: Типовой гомоиконичность (Typed Homoiconicity) - Синтаксис как тип

>

>

>
> **⚠️ Постоянное экспериментальное уведомление**: Это **исследовательский эксперимент**, предназначенный для проверки осуществимости языковой конструкции "синтаксис как тип". **Данный RFC никогда не будет объединён**, независимо от результатов он не попадёт в ветку dev/main. Экспериментальная ветка будет заброшена или архивирована после завершения.
>
> - **Цель эксперимента**: Проверить сложность реализации и потенциальную ценность типового гомоиконичности
> - **Критическая точка**: Отказ при отсутствии прогресса в течение 6 месяцев
> - **Критерий успеха**: Возможность запустить хотя бы одно пользовательское ключевое слово (полный цикл: парсинг → компиляция → выполнение)
>
> **Не гарантируется объединение в main ветку**, в будущем проект может быть отклонён или заброшен по различным причинам. Не используйте эту функциональность в производственной среде.
>
> **⚠️ Пояснение позиционирования**: Данный RFC является мысленным экспериментом в области дизайна языка и не предоставляет инженерного решения. Для практического расширяемого парсерного паттерна см. Rust `syn::Parse` или Haskell `parsec`.

## Аннотация

Данный RFC предлагает радикальный эксперимент в дизайне языка: **сделать синтаксическую структуру языка частью системы типов**.

Ключевая идея происходит из лисповского "код как данные" (гомоиконичность), но реализуется через **статическую систему типов**:
- Синтаксическое дерево (AST) является типом
- Ключевые слова — это предопределённые экземпляры типов
- Пользователи могут расширять синтаксис языка через определение типов

Это означает: сам язык становится компонуемыми, расширяемыми "кубиками".

---

## Мотивация

### Зачем проводить этот эксперимент?

1. **Стремление к унификации**: Устранить "ключевые слова" как особый синтаксический элемент, сделать всё типами и функциями
2. **Расширяемость языка**: Пользователи могут определять новые синтаксические структуры так же, как определяют функции
3. **Типобезопасные макросы**: Традиционные макросы (текстовая замена) опасны; типовая гомоиконичность может обеспечить проверку на этапе компиляции
4. **Образовательная цель**: Глубокое понимание сущности дизайна языков

### Связь с Lisp

Lisp давно реализует "код как данные":
```lisp
; Lisp код сам по себе является S-expression
(if (> x 0) "positive" "negative")
```

Разница данного эксперимента: **усиление этой идеи через статическую систему типов**.

## Предложение

### Основные концепции

#### 1. Синтаксическое дерево как тип (AST as Type)

```yaoxiang
// Узлы синтаксического дерева являются типами
If: Type = { condition: Expr, then: Block, else: Block }
While: Type = { condition: Expr, body: Block }
Return: Type = { value: Expr }
Block: Type = { statements: Array[Expr] }
Let: Type = { name: String, value: Expr, body: Expr }
Function: Type = { params: Array[Param], body: Expr }
Call: Type = { func: Expr, args: Array[Expr] }

// Базовые типы
Literal: Type = { value: Int }
StringLiteral: Type = { value: String }
Variable: Type = { name: String }
```

#### 2. Ключевые слова = функции обработки типов

```yaoxiang
// Вычислитель — это функция обработки этих типов
eval_if: (node: If, env: Env) -> Value = ...
eval_while: (node: While, env: Env) -> Value = ...
eval_return: (node: Return, env: Env) -> Value = ...
eval_block: (node: Block, env: Env) -> Value = ...

// Компилятор тоже может быть функцией
compile_if: (node: If, ctx: CompileContext) -> IR = ...
compile_while: (node: While, ctx: CompileContext) -> IR = ...
```

#### 3. Типы несут правила парсинга (ключевая инновация)

Это ключевой момент эксперимента: **типы не только описывают данные, но и несут правила парсинга кода**.

```yaoxiang
// Тип синтаксического правила
SyntaxRule: Type = {
    // Как парсить код этого типа
    parse: (token_stream: TokenStream) -> (Self, remaining_tokens)

    // Как компилировать/вычислять экземпляр типа
    compile: (node: Self, ctx: CompileContext) -> IR
    eval: (node: Self, env: Env) -> Value
}

// Синтаксическое правило для типа IF
IF: SyntaxRule = {
    // Парсинг "if (cond) { then } else { else }"
    parse: (tokens: TokenStream) -> (If, remaining) = {
        consume("if")
        cond = parse_expression(tokens)
        consume("{")
        then_block = parse_block(tokens)
        consume("}")
        consume("else")
        consume("{")
        else_block = parse_block(tokens)
        consume("}")
        return If(cond, then_block, else_block), tokens
    }

    eval: (node: If, env: Env) -> Value = {
        if eval(node.condition, env) != 0 {
            return eval(node.then, env)
        } else {
            return eval(node.else, env)
        }
    }
}
```

#### 4. Расширение синтаксиса пользователем

Пользователи могут определять свои собственные "ключевые слова":

```yaoxiang
// Пользователь определяет новую синтаксическую структуру: unless
Unless: SyntaxRule = {
    parse: (tokens: TokenStream) -> (If, remaining) = {
        consume("unless")
        cond = parse_expression(tokens)
        consume("{")
        body = parse_block(tokens)
        consume("}")
        // unless эквивалентен if (!cond) { body }
        return If(Not(cond), body, Block([])), tokens
    }
}

// Использование
unless x > 0 {
    print("x is not positive")
}

// Раскрывается в
if !(x > 0) {
    print("x is not positive")
}
```

### Примеры

#### Полный пример: пользовательский синтаксис цикла

```yaoxiang
// Определяем цикл "times": n.times { ... } выполняется n раз
TimesLoop: SyntaxRule = {
    parse: (tokens: TokenStream) -> (While, remaining) = {
        receiver = parse_expression(tokens)  // Получаем число
        consume(".times")
        consume("{")
        body = parse_block(tokens)
        consume("}")
        // Преобразуем в цикл while
        counter_var = gensym("i")
        return While(
            Less(Variable(counter_var), receiver),
            Block([
                body,
                Assign(counter_var, Add(Variable(counter_var), Literal(1)))
            ])
        ), tokens
    }
}

// Использование
5.times {
    print("Hello!")
}

// Раскрывается в
i = 0
while i < 5 {
    print("Hello!")
    i = i + 1
}
```

#### Пример: Синтаксис сопоставления с образцом

```yaoxiang
// Пользователь определяет сопоставление с образцом
Match: SyntaxRule = {
    parse: (tokens: TokenStream) -> (MatchNode, remaining) = {
        subject = parse_expression(tokens)
        consume("{")
        cases = []
        while !check("}") {
            pattern = parse_pattern(tokens)
            consume("=>")
            body = parse_expression(tokens)
            cases.push((pattern, body))
        }
        consume("}")
        return MatchNode(subject, cases), tokens
    }
}

// Использование
match x {
    0 => "zero",
    1 => "one",
    n if n > 10 => "big",
    _ => "other"
}
```

### Ключевые технические проблемы

#### 1. Функционализация управления потоком

Проблема: `if` должен вычислять только одну ветку, нельзя использовать обычный вызов функции.

Решение: передача thunk (отложенное вычисление)

```yaoxiang
// Внутреннее представление после компиляции
If: Type = {
    condition: Expr,
    then: () -> Value,  // thunk, отложенное вычисление
    else: () -> Value
}
```

#### 2. Нелокальный возврат return

Проблема: `return` должен выходить из нескольких уровней функций.

Решение:
- Вариант A: CPS-трансформация на этапе компиляции
- Вариант B: Использование Result/Either монады
- Вариант C: Ограничение области видимости return

#### 3. Синтаксическая неоднозначность

Проблема: Как отличить `if(x > 0) { 1 }` — это вызов функции или ключевое слово?

Решение:
- Ключевые слова используют особый синтаксис (например, `if ... { } else { }`)
- Или через ограничения системы типов

#### 4. Бесконечная рекурсия

Проблема: Пользователь может определить самореферентные синтаксические правила.

Решение: Обнаружение циклических зависимостей на этапе компиляции

## Связь с существующими системами

### Связь с RFC-010 (Единый синтаксис типов)

RFC-010 реализовал унифицированный синтаксис `name: type = value`, данный RFC является его расширением:

| RFC-010 | Данный RFC |
|----------|------------|
| Переменные, функции, типы — всё `name: type = value` | Ключевые слова тоже `name: type = value` |
| Типы являются значениями | Синтаксические правила тоже значения |
| `Type` — это метатип | `SyntaxRule` — это метатип синтаксиса |

### Сравнение с Lisp/Макросами

| Характеристика | Лисп-макросы | Данный эксперимент |
|-----------------|--------------|---------------------|
| Представление кода | S-expression (списки) | Экземпляры типов |
| Способ расширения | defmacro | Определение типа SyntaxRule |
| Типобезопасность | Слабая (текстовая замена) | Сильная (проверка типов) |
| Время парсинга | Время выполнения/компиляции | Этап компиляции |
| Поддержка IDE | Слабая | Сильная (информация о типах) |

### Преимущества

- **Предельная унификация**: Устранение границы между ключевыми словами и обычным кодом

- **Типобезопасность**: Безопаснее традиционных макросов

- **Сложность реализации**: Требуется значительная переработка компилятора

- **Крутая кривая обучения**: Абстрактные концепции, требуется понимание системы типов

- Сложность реализации может превысить ожидания

## Приложения

### Глоссарий терминов

| Термин | Определение |
|--------|-------------|
| Гомоиконичность (Homoiconicity) | Код и данные используют одно и то же представление |
| Синтаксическое дерево (AST) | Представление программы в виде абстрактного синтаксического дерева |
| SyntaxRule | Тип, несущий правила синтаксического парсинга |
| Thunk | Обёртка функции для отложенного вычисления |
| CPS | Continuation Passing Style, стиль передачи продолжений |
