# RFC-019: Типоуровневая гомоиконичность (Typed Homoiconicity) — синтаксис как тип

> **Статус**: Черновик
>
> **Автор**: Чэнь Сюй (晨煦)
>
> **Дата создания**: 2026-02-20
>
> **⚠️ Постоянное экспериментальное уведомление**: Это **исследовательский эксперимент**, предназначенный для проверки осуществимости языковой конструкции "синтаксис как тип". **Данный RFC никогда не будет объединён**, независимо от результатов, и не попадёт в ветки dev/main. Экспериментальная ветка будет удалена или архивирована после завершения.
>
> - **Цель эксперимента**: проверить сложность реализации и потенциальную ценность типоуровневой гомоиконичности
> - **Критическая точка**: отказ при отсутствии прогресса в течение 6 месяцев
> - **Критерий успеха**: возможность запустить хотя бы одно пользовательское ключевое слово (полный цикл: парсинг → компиляция → выполнение)
>
> **Не гарантируется слияние в main-ветку**, в будущем по разным причинам возможен отказ или забвение. Не используйте эту функциональность в производственной среде.

## Аннотация

Настоящий RFC предлагает радикальный эксперимент в дизайне языка: **сделать синтаксическую структуру языка частью системы типов**.

Ключевая идея заимствована из Lisp — "код как данные" (гомоиконичность), но реализована через **статическую систему типов**:
- Синтаксическое дерево (AST) является типом
- Ключевые слова — это предопределённые экземпляры типов
- Пользователи могут расширять синтаксис языка через определение типов

Это означает: язык сам становится компонуемым, расширяемым "конструктором".

---

## Мотивация

### Зачем проводить этот эксперимент?

1. **Стремление к унификации**: устранить "ключевые слова" как особый синтаксический элемент, сделать всё типами и функциями
2. **Расширяемость языка**: пользователи могут определять новые синтаксические структуры так же, как определяют функции
3. **Типобезопасные макросы**: традиционные макросы (текстовые подстановки) опасны; типоуровневая гомоиконичность обеспечивает проверки на этапе компиляции
4. **Образовательная цель**: глубокое понимание сущности дизайна языков

### Связь с Lisp

Lisp давно реализует "код как данные":
```lisp
; Код на Lisp — это S-expression
(if (> x 0) "positive" "negative")
```

Различие данного эксперимента: **усиление этой идеи статической системой типов**.

## Предложение

### Ключевые концепции

#### 1. Синтаксическое дерево как тип (AST as Type)

```yaoxiang
// Узлы синтаксического дерева — это типы
If: Type = { condition: Expr, then: Block, else: Block }
While: Type = { condition: Expr, body: Block }
Return: Type = { value: Expr }
Block: Type = { statements: Array[Expr] }
Let: Type = { name: String, value: Expr, body: Expr }
Function: Type = { params: Array[Param], body: Expr }
Call: Type = { func: Expr, args: Array[Expr] }

// Базовые типы
Literal: Type = { value: Int }
StringLiteral: Type = { value: String }
Variable: Type = { name: String }
```

#### 2. Ключевые слова = функции обработки типов

```yaoxiang
// Вычислители — это функции, обрабатывающие эти типы
eval_if: (node: If, env: Env) -> Value = ...
eval_while: (node: While, env: Env) -> Value = ...
eval_return: (node: Return, env: Env) -> Value = ...
eval_block: (node: Block, env: Env) -> Value = ...

// Компилятор тоже может быть функцией
compile_if: (node: If, ctx: CompileContext) -> IR = ...
compile_while: (node: While, ctx: CompileContext) -> IR = ...
```

#### 3. Типы несут правила парсинга (ключевая инновация)

Это ключевой момент эксперимента: **типы не только описывают данные, но и несут правила разбора кода**.

```yaoxiang
// Тип правил синтаксиса
SyntaxRule: Type = {
    // Как разобрать код этого типа
    parse: (token_stream: TokenStream) -> (Self, remaining_tokens)

    // Как скомпилировать/вычислить экземпляр типа
    compile: (node: Self, ctx: CompileContext) -> IR
    eval: (node: Self, env: Env) -> Value
}

// Синтаксическое правило для типа IF
IF: SyntaxRule = {
    // Разбор "if (cond) { then } else { else }"
    parse: (tokens: TokenStream) -> (If, remaining) = {
        consume("if")
        cond = parse_expression(tokens)
        consume("{")
        then_block = parse_block(tokens)
        consume("}")
        consume("else")
        consume("{")
        else_block = parse_block(tokens)
        consume("}")
        return If(cond, then_block, else_block), tokens
    }

    eval: (node: If, env: Env) -> Value = {
        if eval(node.condition, env) != 0 {
            return eval(node.then, env)
        } else {
            return eval(node.else, env)
        }
    }
}
```

#### 4. Расширение синтаксиса пользователем

Пользователи могут определять свои "ключевые слова":

```yaoxiang
// Пользователь определяет новую синтаксическую конструкцию: unless
Unless: SyntaxRule = {
    parse: (tokens: TokenStream) -> (If, remaining) = {
        consume("unless")
        cond = parse_expression(tokens)
        consume("{")
        body = parse_block(tokens)
        consume("}")
        // unless эквивалентен if (!cond) { body }
        return If(Not(cond), body, Block([])), tokens
    }
}

// Использование
unless x > 0 {
    print("x is not positive")
}

// Раскрывается в
if !(x > 0) {
    print("x is not positive")
}
```

### Примеры

#### Полный пример: пользовательский синтаксис цикла

```yaoxiang
// Определяем цикл "times": n.times { ... } выполняется n раз
TimesLoop: SyntaxRule = {
    parse: (tokens: TokenStream) -> (While, remaining) = {
        receiver = parse_expression(tokens)  // Получаем число
        consume(".times")
        consume("{")
        body = parse_block(tokens)
        consume("}")
        // Преобразуем в цикл while
        counter_var = gensym("i")
        return While(
            Less(Variable(counter_var), receiver),
            Block([
                body,
                Assign(counter_var, Add(Variable(counter_var), Literal(1)))
            ])
        ), tokens
    }
}

// Использование
5.times {
    print("Hello!")
}

// Раскрывается в
i = 0
while i < 5 {
    print("Hello!")
    i = i + 1
}
```

#### Пример: синтаксис сопоставления с образцом

```yaoxiang
// Пользовательское сопоставление с образцом
Match: SyntaxRule = {
    parse: (tokens: TokenStream) -> (MatchNode, remaining) = {
        subject = parse_expression(tokens)
        consume("{")
        cases = []
        while !check("}") {
            pattern = parse_pattern(tokens)
            consume("=>")
            body = parse_expression(tokens)
            cases.push((pattern, body))
        }
        consume("}")
        return MatchNode(subject, cases), tokens
    }
}

// Использование
match x {
    0 => "zero",
    1 => "one",
    n if n > 10 => "big",
    _ => "other"
}
```

### Архитектура системы

```

│                    Исходный код                     │

                  │
                  ▼

│              Синтаксический парсер (Parser)          │
│  - Распознавание ключевых слов                      │
│  - Поиск соответствующего типа SyntaxRule           │
│  - Вызов метода parse типа                          │
└─────────────────┬───────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────────────┐
│              AST (экземпляры типов)                 │
│  If, While, Match, TimesLoop...                     │
└─────────────────┬───────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────────────┐
│              Компилятор / Интерпретатор             │
│  - Вызов методов compile/eval типа                  │
│  - Генерация целевого кода или выполнение          │
└─────────────────────────────────────────────────────┘
```

### Ключевые технические проблемы

#### 1. Функционализация потока управления

Проблема: `if` требует вычисления только одной ветки, нельзя использовать обычный вызов функции.

Решение: передача thunk (отложенное вычисление)

```yaoxiang
// Внутреннее представление после компиляции
If: Type = {
    condition: Expr,
    then: () -> Value,  // thunk, отложенное вычисление
    else: () -> Value
}
```

#### 2. Нелокальный возврат return

Проблема: `return` должен выходить из нескольких уровней функций.

Решение:
- Вариант A: CPS-преобразование на этапе компиляции
- Вариант B: использование Result/Either-монады
- Вариант C: ограничение области действия return

#### 3. Синтаксическая неоднозначность

Проблема: как отличить `if(x > 0) { 1 }` как вызов функции от ключевого слова?

Решение:
- Ключевые слова используют особый синтаксис (например, `if ... { } else { }`)
- Или ограничение через систему типов

#### 4. Бесконечная рекурсия

Проблема: пользователь может определить самоссылающиеся синтаксические правила.

Решение: обнаружение циклических зависимостей на этапе компиляции

## Связь с существующими системами

### Связь с RFC-010 (Унифицированный синтаксис типов)

RFC-010 реализовал унифицированный синтаксис `name: type = value`, настоящий RFC является его развитием:

| RFC-010 | Настоящий RFC |
|----------|--------|
| Переменные, функции, типы — всё `name: type = value` | Ключевые слова тоже `name: type = value` |
| Типы являются значениями | Синтаксические правила тоже значения |
| `Type` — это метатип | `SyntaxRule` — это метатип синтаксиса |

### Сравнение с Lisp/макросами

| Характеристика | Макросы Lisp | Наш эксперимент |
|------|---------|--------|
| Представление кода | S-expression (списки) | Экземпляры типа |
| Способ расширения | defmacro | Определение типа SyntaxRule |
| Типобезопасность | Слабая (текстовая подстановка) | Сильная (проверка типов) |
| Время парсинга | Runtime/compile-time | Compile-time |
| Поддержка IDE | Слабая | Сильная (информация о типах) |

## Компромиссы

### Преимущества

- **Предельная унификация**: устранение границы между ключевыми словами и обычным кодом
- **Расширяемость языка**: пользователи могут определять собственный синтаксис
- **Типобезопасность**: безопаснее традиционных макросов
- **Образовательная ценность**: глубокое понимание природы языков

### Недостатки

- **Сложность реализации**: требуется значительная переработка компилятора
- **Опасения по производительности**: интерпретация на runtime может быть медленной
- **Кривая обучения**: абстрактные концепции, требуется понимание системы типов
- **Сомнительная практичность**: возможен овер-инжиниринг

### Риски

- Эксперимент может провалиться, не найдётся практических сценариев использования
- Сложность реализации превысит ожидания
- Конфликт с существующими возможностями

### Глоссарий терминов
