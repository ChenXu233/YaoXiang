# RFC-011a: Реализация interface и динамическая диспетчеризация

> **Родительский RFC**: [RFC-011: Проектирование generics системы](../accepted/011-generic-type-system.md)
>
> **Данный RFC дополняет и заменяет разделы 2.1–2.4 RFC-011, посвящённые ограничениям interface.**

## Аннотация

RFC-011 определяет generics систему, но не детализирует механизм реализации interface. Настоящий документ дополняет:

1. **Объявление interface**: имя interface записывается непосредственно в определении типа, ключевое слово `impl` не требуется
2. **Реализация методов**: поддерживается как внутреннее, так и внешнее объявление
3. **Правила перегрузки**: разные сигнатуры разрешают перегрузку, одинаковые сигнатуры приводят к ошибке (переопределение запрещено)
4. **Значения по умолчанию**: записываются сразу после поля в виде `= значение`
5. **Динамическая диспетчеризация**: compile-time сбор типов + сопоставление interface, без виртуальной таблицы

**Ключевая конструкция**:

```yaoxiang
# Определение interface
Animal: Type = {
    speak: (Self) -> String,
}

# Определение типа (внутреннее объявление)
Dog: Type = {
    x: Int = 10,
    Animal,  # объявление interface
    speak: (Self) -> String = "Woof",
}

# Внешнее объявление (перегрузка)
Dog.speak: (Self, volume: Int) -> String = "WOOF"

# Гетерогенный контейнер (динамическая диспетчеризация)
animals: List(Animal) = [Dog.new(), Cat.new()]
animals[0].speak()  # "Woof"
```

**Устранённая сложность**:
- ❌ Нет ключевого слова `impl`
- ❌ Нет аннотации `dyn Trait + 'a`
- ❌ Нет виртуальной таблицы (compile-time сбор типов + обёртка в enum)
- ❌ Нет переопределения (единые правила перегрузки)

## Мотивация

### Недостатки RFC-011

RFC-011 определяет generics систему, но не детализирует:

| Проблема | Описание |
|------|------|
| Синтаксис объявления interface | Как объявить, что тип реализует interface? |
| Расположение реализации методов | Внутреннее или внешнее объявление? |
| Правила перегрузки | Как обрабатывать одноимённые методы? |
| Синтаксис значений по умолчанию | Как задать значение поля по умолчанию? |
| Динамическая диспетчеризация | Как реализовать гетерогенный контейнер? |

### Цели проектирования

1. **Простота**: ключевое слово `impl` не требуется
2. **Гибкость**: реализация методов поддерживается как внутри, так и снаружи
3. **Единообразие**: согласованные правила перегрузки
4. **Удобство**: лаконичный синтаксис значений по умолчанию
5. **Нулевые накладные расходы**: нет виртуальной таблицы, compile-time сбор типов

### Сравнение с Rust

| Возможность | Rust | YaoXiang |
|------|------|----------|
| Объявление interface | `impl Animal for Dog { ... }` | `Dog: Type = { Animal, ... }` |
| Реализация метода | В блоке `impl` | Внутри или снаружи |
| Перегрузка | Не поддерживается | Поддерживается (разные сигнатуры) |
| Значение по умолчанию | Требуется `#[default]` | Записывается сразу как `= значение` |
| Гетерогенный контейнер | `Vec<Box<dyn Animal + 'a>>` | `List(Animal)` |
| Динамическая диспетчеризация | Поиск в виртуальной таблице | Compile-time сбор типов |

## Предложение

### 1. Объявление interface

**Основное правило**: имя interface записывается непосредственно в определении типа, ключевое слово `impl` не требуется.

```yaoxiang
# Определение interface
Animal: Type = {
    speak: (Self) -> String,
}

# Объявление типа, реализующего interface
Dog: Type = {
    x: Int,
    Animal,  # объявление interface
}
```

**Обработка компилятором**:
1. Распознать, что `Animal` — это interface type.
2. Проверить наличие у `Dog` всех методов, требуемых `Animal`.
3. При успехе → сгенерировать доказательство реализации.
4. При неудаче → ошибка компиляции.

**Синтаксический сахар — эквивалентность**:

```yaoxiang
Dog: Type = {
    x: Int,
    Animal,  # эквивалентно раскрытию методов Animal, но с сохранением метки источника
}

# Эквивалентно (но с сохранением информации об источнике)
Dog: Type = {
    x: Int,
    speak: (Self) -> String,  # из Animal
}
```

**Зачем нужна метка источника**:
- Прямое раскрытие теряет информацию об источнике.
- Метка источника используется для генерации доказательства реализации.
- В runtime по доказательству находится корректный метод.

### 2. Реализация методов

**Основное правило**: реализация методов поддерживается как внутренним, так и внешним объявлением.

#### 2.1 Внутреннее объявление

```yaoxiang
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",  # реализация метода внутри
}
```

#### 2.2 Внешнее объявление

```yaoxiang
Dog: Type = {
    x: Int,
    Animal,
}

# Реализация метода снаружи
Dog.speak: (Self) -> String = "Woof"
```

#### 2.3 Смешанное объявление

```yaoxiang
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",  # часть методов внутри
}

# часть методов снаружи
Dog.play: (Self) -> Void = { ... }
```

**Обработка компилятором**:
1. Собрать все определения (внутренние и внешние).
2. Сгруппировать по сигнатуре (перегрузка).
3. Проверить наличие переопределений (ошибка).
4. Проверить полноту interface.
5. Сгенерировать доказательство реализации.

### 3. Перегрузка и переопределение

**Основные правила**:
- Разные сигнатуры → перегрузка → разрешена
- Одинаковые сигнатуры → переопределение → ошибка

#### 3.1 Перегрузка (разрешена)

```yaoxiang
# Типы параметров различаются, перегрузка разрешена
Dog.speak: (Self) -> String = "Woof"
Dog.speak: (Self, volume: Int) -> String = "WOOF"
```

#### 3.2 Переопределение (запрещено)

```yaoxiang
# Сигнатуры полностью совпадают, переопределение запрещено
Dog.speak: (Self) -> String = "Woof"
Dog.speak: (Self) -> String = "Bark"  # ❌ ошибка: переопределение недопустимо
```

**Сообщение об ошибке**:

```
ошибка: повторное определение Dog.speak(Self) -> String
  --> файл2:5:1
  |
5 | Dog.speak: (Self) -> String = "Bark"
  | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ повторное определение
  |
  --> файл1:3:1
  |
3 | Dog.speak: (Self) -> String = "Woof"
  | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ первое определение
```

#### 3.3 Единые правила

**Внутренние и внешние объявления подчиняются одним и тем же правилам перегрузки/переопределения**:

```yaoxiang
# Внутреннее объявление
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

# Внешнее объявление (перегрузка, разрешена)
Dog.speak: (Self, volume: Int) -> String = "WOOF"

# Внешнее объявление (переопределение, запрещено)
Dog.speak: (Self) -> String = "Bark"  # ❌ ошибка
```

### 4. Значения по умолчанию

**Основное правило**: после поля сразу пишется `= значение`, что избавляет от необходимости в конструкторе.

```yaoxiang
Dog: Type = {
    x: Int = 10,  # значение по умолчанию
    y: Int = 20,  # значение по умолчанию
    Animal,
}
```

**Генерация конструктора компилятором**:

```yaoxiang
# Все поля имеют значения по умолчанию → генерируется конструктор без параметров
Dog.new: () -> Dog = { x: 10, y: 20 }

# Часть полей имеет значения по умолчанию → генерируются конструкторы с частичными параметрами
Dog.new: (x: Int) -> Dog = { x: x, y: 20 }
Dog.new: (y: Int) -> Dog = { x: 10, y: y }

# Конструктор со всеми параметрами
Dog.new: (x: Int, y: Int) -> Dog = { x: x, y: y }
```

**Внешнее объявление значений по умолчанию**:

```yaoxiang
Dog: Type = {
    x: Int,
    y: Int,
    Animal,
}

# Внешнее объявление значений по умолчанию
Dog.x: Int = 10
Dog.y: Int = 20
```

**Эквивалентно внутреннему объявлению**.

### 5. Реализация в компиляторе

#### 5.1 Дескриптор interface

```rust
// Внутреннее представление компилятора: дескриптор interface
struct InterfaceDescriptor {
    name: String,
    methods: Vec<MethodSignature>,
}
```

#### 5.2 Определение типа

```rust
// Внутреннее представление компилятора: определение типа
struct TypeDefinition {
    name: String,
    fields: Vec<Field>,
    interface_implementations: Vec<InterfaceImplementation>,
}

// Реализация interface (с сохранением информации об источнике)
struct InterfaceImplementation {
    interface: InterfaceId,
    methods: HashMap<MethodId, FunctionBody>,
}
```

#### 5.3 Доказательство реализации

```rust
// Внутреннее представление компилятора: доказательство реализации
struct ImplementationProof {
    type_id: TypeId,
    interface_id: InterfaceId,
    methods: Vec<MethodPointer>,
}
```

#### 5.4 Процесс компиляции

```
1. Разобрать определения типов, собрать объявления interface.
2. Собрать все определения методов (внутренние и внешние).
3. Сгруппировать по сигнатуре (перегрузка).
4. Проверить переопределения (ошибка).
5. Проверить полноту interface.
6. Сгенерировать доказательство реализации.
7. В runtime значение несёт доказательство реализации.
```

### 6. Динамическая диспетчеризация

**Ключевая конструкция**: compile-time сбор типов + сопоставление interface, без виртуальной таблицы.

#### 6.1 Гетерогенный контейнер

```yaoxiang
# Определение interface
Animal: Type = {
    speak: (Self) -> String,
}

# Определения типов
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

Cat: Type = {
    y: Int,
    Animal,
    speak: (Self) -> String = "Meow",
}

# Гетерогенный контейнер
animals: List(Animal) = [Dog.new(), Cat.new()]
animals[0].speak()  # "Woof"
animals[1].speak()  # "Meow"
```

#### 6.2 Compile-time сбор типов

**Обработка компилятором**:

```
1. Сканировать все типы, помещаемые в List(Animal).
2. Собрать: Dog, Cat.
3. Автоматически сгенерировать enum AnimalGroup.
4. Сгенерировать мономорфизированный код для AnimalGroup.
5. В runtime диспетчеризация выполняется через сопоставление по enum.
```

**Автоматически сгенерированный enum**:

```yaoxiang
# Сгенерировано компилятором автоматически (невидимо для пользователя)
AnimalGroup: Type = {
    Dog(Dog),
    Cat(Cat),
}

# List(Animal) эквивалентно List(AnimalGroup)
animals: List(AnimalGroup) = [
    AnimalGroup.Dog(Dog.new()),
    AnimalGroup.Cat(Cat.new()),
]
```

#### 6.3 Проверка соответствия interface

**Ключевое наблюдение**: проверка соответствия interface выполняется в compile-time, даже если типы поступают из динамически загружаемых плагинов.

```yaoxiang
# Система плагинов
plugin = load_plugin("bird.so")

# Проверка компилятора: возвращаемый тип plugin.create_bird() должен реализовывать interface Animal
bird: Animal = plugin.create_bird()  # проверка в compile-time

# Помещение в гетерогенный контейнер
animals: List(Animal) = [Dog.new(), Cat.new(), bird]
```

**Обработка компилятором**:
1. Проверить тип возвращаемого значения `plugin.create_bird()`.
2. Убедиться, что этот тип реализует interface `Animal`.
3. При успехе → разрешить помещение в `List(Animal)`.
4. При неудаче → ошибка компиляции.

#### 6.4 Диспетчеризация в runtime

**Последовательность вызова**:

```
animals[0].speak()
  ↓
найти доказательство реализации для animals[0] (interface Animal)
  ↓
из доказательства получить указатель на метод speak
  ↓
вызвать метод
```

**Сравнение с виртуальной таблицей**:

| | Виртуальная таблица (Rust) | Доказательство реализации (YaoXiang) |
|---|---|---|
| Способ поиска | указатель на vtable → указатель на метод | доказательство реализации → указатель на метод |
| Накладные расходы в runtime | одно косвенное обращение | одно косвенное обращение |
| Генерация в compile-time | виртуальная таблица | доказательство реализации |
| Аннотация времени жизни | требуется `dyn Trait + 'a` | не требуется |

**Преимущества YaoXiang**:
- Не требуется аннотация времени жизни (доказательству реализации не нужен `'a`).
- Compile-time безопасность типов (соответствие interface проверяется в compile-time).
- Прозрачно для пользователя (не нужно писать `dyn Animal`).

#### 6.5 Ограничения

**Не поддерживается полностью динамическая типизация в runtime**:
- Набор типов должен быть полностью известен в compile-time.
- Система плагинов требует проверки соответствия interface в compile-time.
- Не поддерживается полностью утиная типизация (проверка наличия метода в runtime).

**Комбинаторный взрыв набора типов не является проблемой**:
- Достаточно линейного сбора типов.
- Для каждого типа генерируется вариант enum.
- Не требуется генерация кода для каждой комбинации.

## Анализ вариантов использования

### Базовая реализация interface

```yaoxiang
# Определение interface
Animal: Type = {
    speak: (Self) -> String,
}

# Определение типа
Dog: Type = {
    x: Int = 10,
    Animal,
    speak: (Self) -> String = "Woof",
}

# Использование
dog = Dog.new()
dog.speak()  # "Woof"
```

### Множественная реализация interface

```yaoxiang
# Несколько interface
Animal: Type = {
    speak: (Self) -> String,
}

Pet: Type = {
    name: (Self) -> String,
}

# Тип реализует несколько interface
Dog: Type = {
    x: Int = 10,
    Animal,
    Pet,
    speak: (Self) -> String = "Woof",
    name: (Self) -> String = "Buddy",
}

# Использование
dog = Dog.new()
dog.speak()  # "Woof"
dog.name()   # "Buddy"
```

### Обобщённый interface

```yaoxiang
# Обобщённый interface
Container: (T: Type) -> Type = {
    add: (self: &mut Self, item: T) -> Void,
    get: (self: &Self, index: Int) -> T,
}

# Реализация обобщённого interface
IntList: Type = {
    data: Array(Int),
    Container(Int),
    add: (self: &mut Self, item: Int) -> Void = ...,
    get: (self: &Self, index: Int) -> Int = ...,
}
```

### Гетерогенный контейнер

```yaoxiang
# Определение interface
Animal: Type = {
    speak: (Self) -> String,
}

# Определения типов
Dog: Type = {
    x: Int,
    Animal,
    speak: (Self) -> String = "Woof",
}

Cat: Type = {
    y: Int,
    Animal,
    speak: (Self) -> String = "Meow",
}

# Гетерогенный контейнер
animals: List(Animal) = [Dog.new(), Cat.new()]

# Использование
for animal in animals {
    print(animal.speak())
}
# Вывод:
# Woof
# Meow
```

### Система плагинов

```yaoxiang
# Определение interface
Plugin: Type = {
    name: (Self) -> String,
    execute: (Self) -> Void,
}

# Основная программа
main: () -> Void = {
    # Загрузка плагинов
    plugin1 = load_plugin("plugin1.so")
    plugin2 = load_plugin("plugin2.so")

    # Проверка компилятора: plugin1 и plugin2 должны реализовывать interface Plugin
    plugins: List(Plugin) = [plugin1, plugin2]

    # Выполнение всех плагинов
    for plugin in plugins {
        print(plugin.name())
        plugin.execute()
    }
}
```

## Компромиссы

### Преимущества

1. **Простота**: ключевое слово `impl` не требуется
2. **Гибкость**: реализация методов поддерживается как внутри, так и снаружи
3. **Единообразие**: согласованные правила перегрузки
4. **Удобство**: лаконичный синтаксис значений по умолчанию
5. **Нулевые накладные расходы**: нет виртуальной таблицы, compile-time сбор типов
6. **Безопасность типов**: соответствие interface проверяется в compile-time
7. **Прозрачность для пользователя**: не нужно писать `dyn Animal + 'a`

### Недостатки

1. **Ограничение**: не поддерживается полностью динамическая типизация в runtime (полная утиная типизация).
2. **Compile-time накладные расходы**: для каждого interface требуется сгенерировать доказательство реализации.
3. **Набор типов**: должен быть полностью известен в compile-time.

### Меры по смягчению

1. **Система плагинов**: поддерживается через проверку соответствия interface в compile-time.
2. **Compile-time накладные расходы**: доказательство реализации — легковесная структура данных.
3. **Набор типов**: линейный сбор, без комбинаторного взрыва.

## Альтернативы

| Подход | Почему не выбран |
|------|------------------|
| Ключевое слово `impl` | Увеличивает сложность синтаксиса |
| Виртуальная таблица (`dyn Trait`) | Требует аннотации времени жизни (`'a`) |
| Полная утиная типизация | Накладные расходы в runtime, небезопасность типов |
| Обёртка в enum (ручная) | Высокая нагрузка на пользователя |

## Связь с RFC-009

**Бренды и реализация interface**:
- Реализация interface находится на уровне типов и не затрагивает бренды.
- Бренды находятся на уровне доказательств заимствования (RFC-009a).
- Эти два механизма ортогональны и не влияют друг на друга.

**Динамическая диспетчеризация и бренды**:
- Для динамической диспетчеризации используется доказательство реализации, аннотация времени жизни не требуется.
- Доказательство реализации генерируется в compile-time, поиск в runtime равен нулю.
- Устраняется сложность `dyn Trait + 'a`.
