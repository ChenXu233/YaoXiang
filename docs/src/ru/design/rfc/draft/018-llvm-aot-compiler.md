## Аннотация

В данном документе проектируется LLVM AOT компилятор для языка YaoXiang, цель которого — генерация машинного кода + DAG метаданных при предварительной компиляции, с последующим выполнением **глобальным DAG планировщиком** на основе **восходящего** анализа зависимостей.

**Ключевые инновации**:
- Не "при встрече с вызовом функции генерировать Future", а **обратный анализ зависимостей от "места, где нужен результат"**
- **Листовые узлы выполняются параллельно в первую очередь**, цепочки зависимостей обходятся последовательно снизу вверх
- **Изолированные DAG выполняются независимо и параллельно**: узлы без потребителей не блокируют основной поток
- **Бесконечные циклы как фоновые DAG**: планировщик выполняет их по частям, не зависая

Данный подход принципиально отличается от модели Rust async/await + runtime tokio:
- Rust: пользователь пишет `async fn`, компилятор генерирует конечный автомат
- YaoXiang: пользователь пишет обычные функции, **компилятор автоматически анализирует DAG**, планировщик выполняет восходящий обход

Соответствует L3 прозрачному 并行 дизайну RFC-001: по умолчанию @auto (автоматический параллелизм), @block синхронный — это особый случай, решающий проблему "цветных функций".

## Мотивация

### Зачем нужен LLVM AOT компилятор?

В настоящее время YaoXiang имеет только интерпретатор в качестве исполняющего backend, что создаёт следующие проблемы:

| Проблема | Влияние |
|------|------|
| Узкое место производительности | Интерпретируемое выполнение медленнее машинного кода в 10-100x |
| Сложность развёртывания | Необходимо поставлять интерпретатор и runtime |
| Проблема "цветных функций" | Синхронные функции не могут вызывать 并行 функции |

### Проблема "цветных функций" и L3 прозрачный 并行

**Традиционный подход (текущий)**:
- Синхронные функции (синие) → не могут вызывать → 并行 функции (красные)
- Синхронность по умолчанию, 并行 требует пометки `spawn`
- Цвет "заразен": однажды использовав 并行, вся цепочка вызовов становится 并行ной

**L3 прозрачный 并行 из RFC-001 (цель)**:
- L3: прозрачный 并行 по умолчанию (@auto)
- L2: явный spawn 并行
- L1: @block синхронный режим

**Инвертированный подход (RFC-018)**:
- По умолчанию L3 прозрачный 并行, автоматический анализ DAG зависимостей при компиляции
- Решает проблему "цветных функций": синхронные функции могут напрямую вызывать код "по умолчанию 并行ный"
- @block только как особый случай принудительного последовательного выполнения

### Ключевая инновация: восходящее выполнение + глобальный DAG

Ключевая инновация данного подхода — **модель восходящего выполнения**:

```
Традиционный вызов (сверху вниз):
  call fetch(url) → выполнение → возврат результата

Восходящее выполнение:
  print(a) ← начинаем с "места, где нужен результат"
       ↑
  fetch(url0) ← анализируем зависимости, идём в обратном направлении

  fetch(url1) ← изолированный, независимое параллельное выполнение
```

**Ключевое различие**:
- Не "при встрече с вызовом функции генерируем Future"
- А от "конечного нужного результата" в обратном направлении анализируем зависимости
- Узлы без потребителей (изолированные) не выполняются или выполняются независимо
- Бесконечные циклы как фоновые DAG, планировщик выполняет их по частям

### Глобальный DAG планировщик

```
Представление DAG всей программы:

        print(result) ─────────────────────────┐
           │                                    │
    ┌──────┴──────┐                             │
    │             │                             │
process(a)   process(b)                        │
    │             │                             │
compute(x)   compute(y)  ←── изолированный DAG ─┤
    │                                           │
fetch(url0)  fetch(url1)  fetch(url2)          │
    (выполнен)                                   │

Параллельно существует фоновый DAG (while True):
    ┌─────────────────────────────────────────┐ │
    │  while True:                            │ │
    │      update_ui()                        │ │
    │      fetch_new() ──→ process(data)      │ │
    └─────────────────────────────────────────┘ │
```

**Принцип работы планировщика**:
```
1. Обратный анализ от "конечного результата":
   print(result) → зависит от process → зависит от fetch

2. Построение глобального DAG:
   - Листовые узлы: fetch (нет зависимостей)
   - Внутренние узлы: process, compute
   - Корневой узел: print

3. Выполнение:
   - fetch выполняются параллельно
   - process ожидает завершения fetch
   - print ожидает завершения process
   - изолированный compute выполняется независимо параллельно

4. Пропуск уже выполненного:
   - Если какой-то узел уже выполнялся, зависимые от него
   последующие узлы могут использовать результат повторно
```

### Обработка бесконечных циклов

```
Сценарий 1: одиночный while/for (без накладных расходов планирования)
──────────────────────────────────────────────
main: () -> () = {
    while True {
        update_ui()
        fetch_data()
    }
}
→ только один бесконечный цикл
→ прямое синхронное выполнение, никакой разницы от обычного кода

Сценарий 2: несколько while (автоматическое разбиение)
──────────────────────────────────────────────
main: () -> () = {
    while True { update_ui() }      # фоновая задача 1
    while True { network_poll() }  # фоновая задача 2
    server_loop()                   # основная задача
}
→ 3 независимые задачи
→ планировщик переключается между срезами
→ настоящий 并行

Адаптивный планировщик:
──────────────────────────────────────────────
if количество_задач == 1:
    прямое выполнение (синхронно)
else:
    планирование срезами (параллельно)
```

**Обработка фоновых DAG**:
```
Основной DAG (с завершением):
    fetch → process → print → завершение

Фоновый DAG (бесконечный цикл):
    while True → update_ui → fetch_new → process → возврат в начало

Планировщик:
    - основной DAG завершается после выполнения
    - фоновый DAG работает вечно, но планировщик выполняет его
    "срезами"
    - не зависает в цикле
```

### Процесс восходящего выполнения

```
Пользовательский код:
    main: () -> () = {
        pages = urls.map(|url| fetch(url))
        results = pages.map(|page| parse_page(page))
        save_results(results)
    }

Фаза 1: Восходящий анализ (время компиляции)
─────────────────────────────────────────
Начиная с save_results(results):
    "нужен results" → зависит от parse_page(results)
    "нужен page0" → зависит от fetch(url0)
    "нужен page1" → зависит от fetch(url1)
    ...

Построение глобального DAG:
    fetch(url0), fetch(url1), fetch(url2) ← листовые узлы
           ↓
    parse_page(page0), parse_page(page1)   ← зависят от листьев
           ↓
    save_results                          ← корневой узел

Фаза 2: Параллельное выполнение листьев (время выполнения)
─────────────────────────────────────────
Планировщик находит все листовые узлы:
    - fetch(url0), fetch(url1), fetch(url2) без зависимостей → параллельное выполнение
    - контроль количества 并行 (например, 16)

Фаза 3: Обход снизу вверх
─────────────────────────────────────────
Когда parse_page нужен page0:
    - проверяем, готов ли page0
    - готов → выполняем parse_page
    - не готов → ждём, после готовности продолжаем

Фаза 4: Изолированные независимо параллельно
─────────────────────────────────────────
Если результат какого-то fetch никому не нужен:
    - выполняется как "изолированный DAG"
    - может использовать другой核心, не влияя на основной поток
```

### Структура артефактов компиляции

```rust
/// Артефакт компиляции: машинный код + DAG метаданные
pub struct CompiledArtifact {
    /// Машинный код, скомпилированный LLVM (ELF/Mach-O/COFF)
    machine_code: Vec<u8>,

    /// DAG метаданные: описывают зависимости функций
    dag: DAGMetadata,

    /// Таблица точек входа
    entries: Vec<EntryPoint>,

    /// Информация о типах (для FFI)
    type_info: TypeInfo,
}

/// DAG метаданные
pub struct DAGMetadata {
    /// Узлы: вызовы функций
    nodes: Vec<DAGNode>,
    /// Рёбра: зависимости (from, to)
    edges: Vec<(usize, usize)>,
}

/// Одиночный DAG узел
pub struct DAGNode {
    /// ID функции
    pub function_id: usize,
    /// Зависимые ID узлов
    pub deps: Vec<usize>,
    /// Метка побочного эффекта (@IO / @Pure)
    pub effect: EffectTag,
}
```

### Интерфейс runtime планировщика

```rust
/// DAG планировщик trait
pub trait DAGScheduler: Send + Sync {
    /// Планирование выполнения
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue;

    /// Выполнение одной функции
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}

/// Реализация планировщика
pub struct DefaultDAGScheduler {
    /// Пул потоков
    thread_pool: ThreadPool,
    /// Артефакт компиляции
    artifact: CompiledArtifact,
    /// Максимальное количество параллельных задач
    max_parallelism: usize,
}

impl DefaultDAGScheduler {
    pub fn new(artifact: CompiledArtifact, num_workers: usize) -> Self {
        Self {
            thread_pool: ThreadPool::new(num_workers),
            artifact,
            max_parallelism: num_workers * 2, // адаптивный контроль зернистости
        }
    }
}

impl DAGScheduler for DefaultDAGScheduler {
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue {
        // 1. Обход тела функции, приостановка всех вызовов
        // 2. Построение списка задач для выполнения
        // 3. Планирование выполнения по порядку зависимостей
        //    (контроль количества параллельных задач)
        // 4. Триггер выполнения при запросе значения
        // 5. Возврат результата
    }
}
```

### Стратегии планирования

| Аннотация | Сценарий | Стратегия планирования |
|------|------|----------|
| `@auto` (по умолчанию, L3) | Прозрачный 并行 | DAG ленивое планирование, выполнение без зависимостей параллельно |
| `@block` (L1) | Принудительная синхронизация | Без DAG, чистое последовательное выполнение |
| Циклическая зависимость | Обнаружено в runtime | Ошибка |

### Обработка побочных эффектов: неявная Effect System

Пользователь не замечает обработку побочных эффектов, компилятор выводит автоматически:

```
Пользовательский код:
  print("a");
  print("b");
  let x = compute(1);
  let y = compute(2);

Вывод компилятора:
  print → @IO (внешний вызов)
  compute → @Pure (чистая функция)

Выполнение планировщиком:
  print("a") ──→ последовательно (все @IO)
  print("b") ──→ последовательно
  compute(1) ─┬─→ параллельно (DAG планирование)
  compute(2) ─┘
```

### Отношение к трёхуровневому runtime

RFC-008 определяет трёхуровневую архитектуру Embedded / Standard / Full runtime. Соответствие LLVM AOT компилятора и трёхуровневого runtime:

| Runtime | Поведение LLVM AOT |
|--------|---------------|
| **Embedded** | Без DAG планирования, напрямую генерируется последовательный машинный код |
| **Standard** | DAG + однопоточное планирование (num_workers=1) |
| **Full** | DAG + многопоточное планирование (num_workers>1), поддержка WorkStealing |

### Дизайн интерфейса планировщика

```rust
/// Стратегия планирования
pub enum ScheduleStrategy {
    /// @block: принудительное последовательное, без DAG
    Serial,
    /// @eager: энергичное вычисление, ожидание завершения зависимостей
    Eager,
    /// @auto (по умолчанию): ленивое планирование, автоматическое DAG планирование
    Lazy,
}

/// Метка побочного эффекта
pub enum EffectTag {
    /// Чистая функция, без побочных эффектов
    Pure,
    /// Имеет побочные эффекты I/O
    IO,
}

/// DAG планировщик trait
pub trait DAGScheduler: Send + Sync {
    /// Планирование выполнения (с параметром стратегии)
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint], strategy: ScheduleStrategy) -> RuntimeValue;

    /// Выполнение одной функции
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}
```

## Компромиссы

### Преимущества

1. **Повышение производительности**: AOT компиляция быстрее интерпретируемого выполнения в 10-100x
2. **Решение проблемы "цветных функций"**: по умолчанию 并行, синхронность — особый случай
3. **Единый runtime**: интерпретатор и LLVM разделяют один планировщик
5. **Неявные побочные эффекты**: пользователь не замечает, компилятор обрабатывает автоматически
6. **Безопасность владения**: зависит от модели владения в стиле Rust, нет гонок данных

### Недостатки

1. **Сложность реализации**: требуется опыт интеграции с LLVM
2. **Время компиляции**: AOT компиляция медленнее интерпретатора
3. **Сложность отладки**: отладка AOT кода сложнее, чем интерпретатора

### Соответствие дизайну RFC

| RFC | Соответствие |
|-----|--------|
| RFC-001 并作 модель | ✅ Анализ DAG зависимостей — основа |
| RFC-008 Архитектура runtime | ✅ Дизайн runtime планировщика согласован |
| RFC-009 Модель владения | ✅ ARC runtime корректно реализован |

## Альтернативные решения

| Решение | Описание | Почему не выбрано |
|------|------|-----------|
| Только интерпретатор | AOT не нужен | Недостаточно производительности, проблема цветных функций |
| Чистая статическая компиляция | Без runtime планирования | Ленивое планирование требует runtime |
| Внешний LLVM runtime | Использовать LLVM runtime | Требует дополнительных зависимостей |

## Связанные работы

### Lazy Task Creation (1990)[^1]

| Атрибут | Описание |
|------|------|
| Организация | MIT |
| Авторы | James R. Larus, Robert H. Halstead Jr. |
| Суть | Отложенное создание подзадач, создание по требованию |
| Ценность | Техническая основа, источник концепции ленивого планирования |

**Ключевая идея**: задачи создаются не сразу, а отложенно. Когда родительской задаче нужны значения дочерних задач — тогда и создаются дочерние задачи. Это решает проблему накладных расходов на выполнение задач с мелкой зернистостью[^1].

### Lazy Scheduling (2014)[^2]

| Атрибут | Описание |
|------|------|
| Организация | University of Maryland |
| Авторы | Tzannes, Caragea |
| Суть | Адаптивное runtime планирование, без дополнительного состояния |
| Ценность | Дизайн планировщика, адаптивный контроль зернистости |

**Ключевая идея**: через "отложенное выполнение" автоматически контролируется зернистость, не требуется поддерживать сложное состояние. Когда система занята — задачи автоматически объединяются, когда свободна — автоматически разделяются[^2].

### Язык SISAL[^3]

| Атрибут | Описание |
|------|------|
| Организация | Lawrence Livermore National Laboratory (LLNL) |
| Суть | Single-assignment язык, Dataflow граф, неявный параллелизм |
| Ценность | Доказательство осуществимости, производительность близка к Fortran |

**Ключевой вклад**: SISAL доказал, что модель Dataflow в промышленных приложениях может достигать производительности, близкой к Fortran[^3].

### Параллельная схема Mul-T[^4]

| Атрибут | Описание |
|------|------|
| Организация | MIT |
| Суть | Конструкция Future, реализация Lazy Task Creation |
| Ценность | Ссылка на конкретную реализацию |

**Ключевой механизм**:
```scheme
;; Синтаксис Multilisp / Mul-T
(let ((a (future compute-a))      ;; немедленный возврат future
      (b (future compute-b)))      ;; немедленный возврат future
  (join a b))                      ;; ожидание завершения
```

### Сравнительное резюме

| Технология | Отложенное создание | DAG анализ | Обработка побочных эффектов | Владение |
|------|----------|----------|------------|----------|
| Lazy Task Creation[^1] | ✅ | ❌ | ❌ | N/A |
| Lazy Scheduling[^2] | ✅ | ❌ | ❌ | N/A |
| SISAL[^3] | ✅ | ✅ (глобальный) | N/A (single-assignment) | N/A |
| Mul-T[^4] | ✅ | ❌ | ❌ | N/A |
| **YaoXiang** | ✅ | ✅ (внутри функций) | ✅ (неявный) | ✅ (ARC) |

**Инновация YaoXiang**: использование современных языковых возможностей (владение + неявные побочные эффекты) упрощает традиционный дизайн, ограничение DAG областью блока функции снижает сложность.

## Сравнение с традиционными методами автоматического параллелизма

### Традиционный компилятор: параллелизм на уровне циклов

Коммерческие компиляторы (Intel Fortran, Oracle Fortran) применяют **автоматическую параллелизацию на уровне циклов**[^5]:

**Основной процесс**:
```
1. Идентификация циклов, готовых к параллелизму
2. Анализ зависимостей обращений к массивам внутри цикла
3. Определение наличия зависимостей между итерациями цикла
4. Если зависимостей нет — генерация многопоточного кода
```

**Технологии анализа зависимостей**:

| Технология | Описание |
|------|------|
| **Зависимость данных** | Два обращения обращаются к одной области памяти |
| **Use-Def** | Отношения определения и использования переменных |
| **Анализ алиасов** | Указывают ли указатели на одну область памяти |

**Условия параллелизуемости цикла**:
```fortran
! Можно параллелизовать
DO I = 1, N
  A(I) = C(I)
END DO

! Нельзя параллелизовать (зависимость от предыдущей итерации)
DO I = 2, N
  A(I) = A(I-1) + B(I)
END DO
```

### Haskell: механизм Spark

GHC (Glasgow Haskell Compiler) применяет **механизм Spark** для реализации 并行 чистых функций[^6]:

```haskell
-- rpar: параллельное выполнение, создание spark
-- rseq: последовательное выполнение, ожидание завершения

example = do
  a <- rpar (f x)   -- создать spark, параллельно выполнить f x
  b <- rpar (g y)   -- создать spark, параллельно выполнить g y
  rseq a            -- ожидать завершения a
  rseq b            -- ожидать завершения b
  return (a, b)
```

**Механизм пула Spark**:
- Spark берутся из пула и分配ются свободным процессорным ядрам
- Если spark не используется (никто не ждёт результата) — он удаляется GC
- Это решает проблему зернистости: слишком мелкие spark отбрасываются

### Язык Clean: уникальные типы

Язык Clean реализует 并行ную безопасность через **уникальные типы (Uniqueness Types)**[^7]:

```clean
-- *Array означает уникальность, можно безопасно изменять
modify :: *Array Int -> *Array Int
```

**Ключевая идея**: если значение имеет唯一ственную ссылку, его можно безопасно изменять в параллельной среде, потому что другие ссылки не увидят промежуточное состояние.

### Программные срезы и графы зависимостей

**Граф программных зависимостей (PDG)** — основа обнаружения параллелизма:

```
Узлы: операторы
Рёбра: зависимость по данным + управляющая зависимость

Обнаружение параллелизма:
  Если между двумя узлами нет достижимого пути → можно параллелизовать
```

### Комплексное сравнение

| Метод | Анализ зависимостей | Зернистость | Обработка побочных эффектов | Типичный сценарий |
|------|----------|------|------------|----------|
| Intel/Oracle Fortran[^5] | Сложный анализ массивов | Итерации цикла | N/A | Научные вычисления |
| GHC Spark[^6] | Предположение о чистых функциях | Выражения | N/A | Функциональное программирование |
| Clean[^7] | Уникальные типы | Перезапись графов | N/A | Функциональное программирование |
| **YaoXiang** | Гарантии владения | Вызовы функций | Неявный вывод | Общий |

### Приложение A: Подробное сравнение с Rust async

| Характеристика | Rust async | YaoXiang LLVM AOT |
|------|-----------|-------------------|

| Аннотации | async/await | @auto/@eager/@block |

### Приложение B: Пример оптимизации планировщика

**Сценарий 1: Планировщик обнаружил возможность объединения выполнения**

```
Исходный DAG:
  compute_a() ──┐
  compute_b() ──┼──→ compute_c()

После оптимизации планировщика:
  Объединяем compute_a + compute_b в одну задачу
  → Уменьшаются накладные расходы планирования
```

**Сценарий 2: Зависимость не используется**

```
let a = expensive_compute(); // вычислено
let b = other_thing();       // a не нужен
print(b);                    // напрямую возвращаем b, пропускаем a
```
