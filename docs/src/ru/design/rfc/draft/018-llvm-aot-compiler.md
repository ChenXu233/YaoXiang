# RFC-018：LLVM AOT-компилятор и прозрачная параллельная модель L3

> **Последнее обновление**: 2026-03-10

## Аннотация

В данном документе описывается дизайн LLVM AOT-компилятора для языка YaoXiang, цель которого — генерация машинного кода и DAG-метаданных на этапе предварительной компиляции, с последующим выполнением **глобальным DAG-планировщиком** на основе **восходящего** анализа зависимостей.

**Ключевые инновации**:
- Не «генерация Future при встрече с вызовом функции», а **обратный анализ зависимостей от «места, где нужен результат»**
- **Листовые узлы выполняются параллельно первыми**, цепочки зависимостей обходятся снизу вверх
- **Изолированные DAG выполняются независимо**: узлы без потребителей не блокируют основной поток
- **Бесконечные циклы как фоновые DAG**: планировщик выполняет их пошагово, без зависания

Данный дизайн принципиально отличается от модели Rust async/await + tokio:
- Rust: пользователь пишет `async fn`, компилятор генерирует конечный автомат
- YaoXiang: пользователь пишет обычные функции, **компилятор автоматически анализирует DAG**, планировщик выполняет восходящее планирование

Соответствует прозрачной параллельной модели L3 из RFC-001: по умолчанию @auto (автоматический параллелизм), @block (синхронный) — это особый случай, решающий проблему «цветных функций».

## Мотивация

### Зачем нужен LLVM AOT-компилятор?

В текущей реализации YaoXiang есть только интерпретатор, что создаёт следующие проблемы:

| Проблема | Влияние |
|----------|---------|
| Узкое место производительности | Интерпретируемое выполнение в 10-100 раз медленнее машинного кода |
| Сложность развёртывания | Требуется переносить интерпретатор и runtime |
| Проблема «цветных функций» | Синхронные функции не могут вызывать параллельные |

### Проблема «цветных функций» и прозрачный параллелизм L3

**Традиционный дизайн (текущий)**:
- Синхронные функции (синие) → не могут вызывать → параллельные функции (красные)
- Синхронность — по умолчанию, параллелизм требует пометки `spawn`
- «Цвет» «заразен»: однажды использовав параллелизм, вся цепочка вызовов становится параллельной

**Прозрачный параллелизм L3 из RFC-001 (цель)**:
- L3: прозрачный параллелизм по умолчанию (@auto)
- L2: явный spawn параллелизма
- L1: режим синхронизации @block

**Инвертированный дизайн (RFC-018)**:
- По умолчанию L3 — прозрачный параллелизм, компилятор автоматически анализирует DAG-зависимости
- Решает проблему «цветных функций»: синхронные функции могут напрямую вызывать код «по умолчанию параллельный»
- @block используется только как особый случай для принудительного последовательного выполнения

### Ключевая инновация: восходящее выполнение + глобальный DAG

Ключевая инновация данного дизайна — **модель восходящего выполнения**:

```
Традиционный вызов (сверху вниз):
  call fetch(url) → выполнение → возврат результата

Восходящее выполнение:
  print(a) ← начинаем с «места, где нужен результат»
       ↑
  fetch(url0) ← анализируем зависимости, идём в обратном направлении

  fetch(url1) ← изолированная ветвь, независимое параллельное выполнение
```

**Ключевые отличия**:
- Не «генерация Future при встрече с вызовом функции»
- Анализ зависимостей в обратном направлении от «конечного результата»
- Узлы без потребителей (изолированные) не выполняются или выполняются независимо
- Бесконечные циклы — фоновые DAG, планировщик выполняет пошагово

### Глобальный DAG-планировщик

```
Представление DAG всей программы:

        print(result) ─────────────────────────┐
           │                                    │
    ┌──────┴──────┐                             │
    │             │                             │
process(a)   process(b)                        │
    │             │                             │
compute(x)   compute(y)  ←── изолированный DAG ──────────┤
    │                                           │
fetch(url0)  fetch(url1)  fetch(url2)          │
    (выполнен)                                    │

Параллельно существует фоновый DAG (while True):
    ┌─────────────────────────────────────────┐ │
    │  while True:                            │ │
    │      update_ui()                        │ │
    │      fetch_new() ──→ process(data)      │ │
    └─────────────────────────────────────────┘ │
```

**Принцип работы планировщика**:
```
1. Обратный анализ от «конечного результата»:
   print(result) → зависит от process → зависит от fetch

2. Построение глобального DAG:
   - Листовые узлы: fetch (без зависимостей)
   - Внутренние узлы: process, compute
   - Корневой узел: print

3. Выполнение:
   - fetch выполняются параллельно
   - process ожидает завершения fetch
   - print ожидает завершения process
   - изолированные compute выполняются независимо

4. Пропуск уже выполненных:
   - Если узел уже выполнен, зависимые от него узлы могут
     использовать результат повторно
```

### Обработка бесконечных циклов

```
Сценарий 1: один while/for (без накладных расходов планирования)
──────────────────────────────────────────────
main: () -> () = {
    while True {
        update_ui()
        fetch_data()
    }
}
→ только один бесконечный цикл
→ прямое синхронное выполнение, как обычный код

Сценарий 2: несколько while (автоматическое разбиение)
──────────────────────────────────────────────
main: () -> () = {
    while True { update_ui() }      # фоновая задача 1
    while True { network_poll() }  # фоновая задача 2
    server_loop()                   # основная задача
}
→ 3 независимые задачи
→ планировщик переключается между ними
→ настоящий параллелизм

Адаптивный планировщик:
──────────────────────────────────────────────
if количество_задач == 1:
    прямое выполнение (синхронно)
else:
    пошаговое планирование (параллельно)
```

**Обработка фоновых DAG**:
```
Основной DAG (завершается):
    fetch → process → print → конец

Фоновый DAG (бесконечный цикл):
    while True → update_ui → fetch_new → process → возврат в начало

Планировщик:
    - Основной DAG завершается после выполнения
    - Фоновый DAG работает вечно, но планировщик выполняет его
      пошагово
    - Не зависает в цикле
```

### Процесс восходящего выполнения

```
Код пользователя:
    main: () -> () = {
        pages = urls.map(|url| fetch(url))
        results = pages.map(|page| parse_page(page))
        save_results(results)
    }

Фаза 1: Восходящий анализ (этап компиляции)
─────────────────────────────────────────
Начинаем с save_results(results):
    "нужен results" → зависит от parse_page(results)
    "нужен page0" → зависит от fetch(url0)
    "нужен page1" → зависит от fetch(url1)
    ...

Построение глобального DAG:
    fetch(url0), fetch(url1), fetch(url2) ← листовые узлы
           ↓
    parse_page(page0), parse_page(page1)   ← зависят от листьев
           ↓
    save_results                          ← корневой узел

Фаза 2: Параллельное выполнение листьев (runtime)
─────────────────────────────────────────
Планировщик находит все листовые узлы:
    - fetch(url0), fetch(url1), fetch(url2) без зависимостей → параллельное выполнение
    - контроль параллелизма (например, 16 одновременных)

Фаза 3: Обход вверх
─────────────────────────────────────────
Когда parse_page нужен page0:
    - проверяем, готов ли page0
    - готов → выполняем parse_page
    - не готов → ждём, затем продолжаем

Фаза 4: Независимое параллельное выполнение изолированных
─────────────────────────────────────────
Если результат какого-то fetch никому не нужен:
    - выполняется как «изолированный DAG»
    - может использовать другой поток, не влияет на основной поток
```

### Структура результата компиляции

```rust
/// Результат компиляции: машинный код + DAG-метаданные
pub struct CompiledArtifact {
    /// Машинный код, скомпилированный LLVM (ELF/Mach-O/COFF)
    machine_code: Vec<u8>,

    /// DAG-метаданные: описание зависимостей функций
    dag: DAGMetadata,

    /// Таблица точек входа
    entries: Vec<EntryPoint>,

    /// Информация о типах (для FFI)
    type_info: TypeInfo,
}

/// DAG-метаданные
pub struct DAGMetadata {
    /// Узлы: вызовы функций
    nodes: Vec<DAGNode>,
    /// Рёбра: зависимости (от, к)
    edges: Vec<(usize, usize)>,
}

/// Отдельный узел DAG
pub struct DAGNode {
    /// ID функции
    pub function_id: usize,
    /// Зависимые узлы
    pub deps: Vec<usize>,
    /// Метка побочного эффекта (@IO / @Pure)
    pub effect: EffectTag,
}
```

### Интерфейс runtime-планировщика

```rust
/// Trait планировщика DAG
pub trait DAGScheduler: Send + Sync {
    /// Планирование выполнения
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue;

    /// Выполнение одной функции
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}

/// Реализация планировщика
pub struct DefaultDAGScheduler {
    /// Пул потоков
    thread_pool: ThreadPool,
    /// Результат компиляции
    artifact: CompiledArtifact,
    /// Максимальный параллелизм
    max_parallelism: usize,
}

impl DefaultDAGScheduler {
    pub fn new(artifact: CompiledArtifact, num_workers: usize) -> Self {
        Self {
            thread_pool: ThreadPool::new(num_workers),
            artifact,
            max_parallelism: num_workers * 2, // адаптивный контроль гранулярности
        }
    }
}

impl DAGScheduler for DefaultDAGScheduler {
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue {
        // 1. Обход тела функции, приостановка всех вызовов
        // 2. Построение списка задач для выполнения
        // 3. Планирование по порядку зависимостей (контроль параллелизма)
        // 4. Запуск выполнения при запросе значения
        // 5. Возврат результата
    }
}
```

### Стратегии планирования

| Аннотация | Сценарий | Стратегия планирования |
|-----------|----------|------------------------|
| `@auto` (по умолчанию, L3) | Прозрачный параллелизм | Ленивое DAG-планирование, выполнение независимых узлов параллельно |
| `@block` (L1) | Принудительная синхронизация | Без DAG, чисто последовательное выполнение |
| Циклическая зависимость | Обнаружена в runtime | Ошибка |

### Обработка побочных эффектов: неявная Effect System

Обработка побочных эффектов невидима для пользователя, компилятор выводит автоматически:

```
Код пользователя:
  print("a");
  print("b");
  let x = compute(1);
  let y = compute(2);

Вывод компилятора:
  print → @IO (внешний вызов)
  compute → @Pure (чистая функция)

Выполнение планировщиком:
  print("a") ──→ последовательно (все @IO)
  print("b") ──→ последовательно
  compute(1) ─┬─→ параллельно (DAG-планирование)
  compute(2) ─┘
```

### Отношение к трёхуровневой runtime-архитектуре

RFC-008 определяет трёхуровневую архитектуру Embedded / Standard / Full runtime. Соответствие LLVM AOT-компилятора трём уровням runtime:

| Runtime | Поведение LLVM AOT |
|---------|-------------------|
| **Embedded** | Без DAG-планирования, генерация последовательного машинного кода |
| **Standard** | DAG + однопоточное планирование (num_workers=1) |
| **Full** | DAG + многопоточное планирование (num_workers>1), поддержка WorkStealing |

### Дизайн интерфейса планировщика

```rust
/// Стратегия планирования
pub enum ScheduleStrategy {
    /// @block: принудительная последовательность, без DAG
    Serial,
    /// @eager: энергичное вычисление, ожидание завершения зависимостей
    Eager,
    /// @auto (по умолчанию): ленивое планирование, автоматическое DAG-планирование
    Lazy,
}

/// Метка побочного эффекта
pub enum EffectTag {
    /// Чистая функция без побочных эффектов
    Pure,
    /// Функция с I/O-побочными эффектами
    IO,
}

/// Trait DAG-планировщика
pub trait DAGScheduler: Send + Sync {
    /// Планирование выполнения (с параметром стратегии)
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint], strategy: ScheduleStrategy) -> RuntimeValue;

    /// Выполнение одной функции
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}
```

## Компромиссы

### Преимущества

1. **Повышение производительности**: AOT-компиляция в 10-100 раз быстрее интерпретируемого выполнения
2. **Решение проблемы «цветных функций»**: параллелизм по умолчанию, синхронность — особый случай
3. **Единый runtime**: интерпретатор и LLVM используют общий планировщик
5. **Неявные побочные эффекты**: невидимы для пользователя, автоматическая обработка компилятором
6. **Безопасность владения**: модель владения в стиле Rust, без гонок данных

### Недостатки

1. **Сложность реализации**: требуется опыт работы с LLVM
2. **Время компиляции**: AOT-компиляция медленнее интерпретатора
3. **Сложность отладки**: отладка AOT-кода сложнее, чем интерпретатора

## Связанные работы

### Lazy Task Creation (1990)[^1]

| Атрибут | Описание |
|---------|----------|
| Организация | MIT |
| Авторы | James R. Larus, Robert H. Halstead Jr. |
| Суть | Отложенное создание подзадач, создание по требованию |
| Ценность | Техническая основа, происхождение концепции ленивого планирования |

**Ключевая идея**: задачи создаются не сразу, а отложенно. Когда родительской задаче нужен результат дочерней, тогда и создаётся дочерняя задача. Это решает проблему накладных расходов на задачи с мелкой гранулярностью[^1].

### Lazy Scheduling (2014)[^2]

| Атрибут | Описание |
|---------|----------|
| Организация | University of Maryland |
| Авторы | Tzannes, Caragea |
| Суть | Адаптивное планирование в runtime, без дополнительного состояния |
| Ценность | Дизайн планировщика, адаптивный контроль гранулярности |

**Ключевая идея**: автоматический контроль гранулярности через «отложенное выполнение», без поддержки сложного состояния. Когда система занята, задачи автоматически объединяются; когда простаивает — автоматически разделяются[^2].

### Язык SISAL[^3]

| Атрибут | Описание |
|---------|----------|
| Организация | Lawrence Livermore National Laboratory (LLNL) |
| Суть | Single-assignment язык, Dataflow графы, неявный параллелизм |
| Ценность | Доказательство осуществимости, производительность близка к Fortran |

**Основной вклад**: SISAL доказал, что модель Dataflow в промышленных приложениях может достичь производительности, близкой к Fortran[^3].

### Параллельная схема Mul-T[^4]

| Атрибут | Описание |
|---------|----------|
| Организация | MIT |
| Суть | Конструкция Future, реализация Lazy Task Creation |
| Ценность | Конкретная реализация для справки |

**Основной механизм**:
```scheme
;; Синтаксис Multilisp / Mul-T
(let ((a (future compute-a))      ;; немедленный возврат future
      (b (future compute-b)))      ;; немедленный возврат future
  (join a b))                      ;; ожидание завершения
```

### Сравнительное резюме

| Технология | Отложенное создание | DAG-анализ | Обработка побочных эффектов | Владение |
|------------|---------------------|-------------|----------------------------|----------|
| Lazy Task Creation[^1] | ✅ | ❌ | ❌ | N/A |
| Lazy Scheduling[^2] | ✅ | ❌ | ❌ | N/A |
| SISAL[^3] | ✅ | ✅ (глобальный) | N/A (single-assignment) | N/A |
| Mul-T[^4] | ✅ | ❌ | ❌ | N/A |
| **YaoXiang** | ✅ | ✅ (внутри функций) | ✅ (неявный) | ✅ (ARC) |

**Инновация YaoXiang**: использование современных языковых возможностей (владение + неявные побочные эффекты) для упрощения традиционного дизайна, ограничение DAG областью блока функций для снижения сложности.

## Сравнение с традиционными методами автоматического параллелизма

### Традиционные компиляторы: параллелизм на уровне циклов

Коммерческие компиляторы (Intel Fortran, Oracle Fortran) используют **автоматический параллелизм на уровне циклов**[^5]:

**Основной процесс**:
```
1. Определение параллелизуемых циклов
2. Анализ зависимостей обращений к массивам внутри цикла
3. Определение наличия зависимостей между итерациями цикла
4. При отсутствии зависимостей — генерация многопоточного кода
```

**Технологии анализа зависимостей**:

| Технология | Описание |
|------------|----------|
| **Зависимости данных** | Два обращения идут к одной области памяти |
| **Use-Def** | Отношения определения и использования переменных |
| **Анализ алиасов** | Указывают ли указатели на одну область памяти |

**Условия параллелизуемости цикла**:
```fortran
! Можно параллелизовать
DO I = 1, N
  A(I) = C(I)
END DO

! Нельзя параллелизовать (зависимость от предыдущей итерации)
DO I = 2, N
  A(I) = A(I-1) + B(I)
END DO
```

### Haskell: механизм Spark

GHC (Glasgow Haskell Compiler) использует **механизм Spark** для реализации параллелизма чистых функций[^6]:

```haskell
-- rpar: параллельное выполнение, создание spark
-- rseq: последовательное выполнение, ожидание завершения

example = do
  a <- rpar (f x)   -- создание spark, параллельное выполнение f x
  b <- rpar (g y)   -- создание spark, параллельное выполнение g y
  rseq a            -- ожидание завершения a
  rseq b            -- ожидание завершения b
  return (a, b)
```

**Механизм пула Spark**:
- Spark извлекаются из пула и分配给空闲ным процессорам
- Если spark не используется (никто не ждёт результат), он удаляется GC
- Это решает проблему гранулярности: слишком мелкие spark отбрасываются

### Язык Clean: типы уникальности

Язык Clean реализует параллельную безопасность через **типы уникальности (Uniqueness Types)**[^7]:

```clean
-- *Array означает уникальность, можно безопасно изменять
modify :: *Array Int -> *Array Int
```

**Ключевая идея**: если значение имеет единственную ссылку, его можно безопасно изменять в параллельной среде, так как другие ссылки не увидят промежуточное состояние.

### Программные срезы и графы зависимостей

**Граф зависимостей программы (PDG)** — основа обнаружения параллелизма:

```
Узлы: операторы
Рёбра: зависимости по данным + управления

Обнаружение параллелизма:
  Если между двумя узлами нет достижимого пути → можно выполнять параллельно
```

### Комплексное сравнение

| Метод | Анализ зависимостей | Гранулярность | Обработка побочных эффектов | Типичный сценарий |
|-------|---------------------|---------------|-----------------------------|-------------------|
| Intel/Oracle Fortran[^5] | Сложный анализ массивов | Итерации цикла | N/A | Научные вычисления |
| GHC Spark[^6] | Предположение о чистых функциях | Выражения | N/A | Функциональное программирование |
| Clean[^7] | Типы уникальности | Перезапись графов | N/A | Функциональное программирование |
| **YaoXiang** | Гарантии владения | Вызовы функций | Неявный вывод | Общего назначения |

### Приложение B: Пример оптимизации планировщика

**Сценарий 1: Планировщик обнаружил возможность объединения**

```
Исходный DAG:
  compute_a() ──┐
  compute_b() ──┼──→ compute_c()

После оптимизации планировщика:
  compute_a + compute_b объединены в одну задачу
  → уменьшение накладных расходов планирования
```

**Сценарий 2: Зависимость не используется**

```
let a = expensive_compute(); // вычислено
let b = other_thing();       // a не нужна
print(b);                    // возврат b напрямую, пропуск a
```
