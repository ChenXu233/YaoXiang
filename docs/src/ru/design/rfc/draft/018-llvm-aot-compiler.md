# RFC-018：Проектирование LLVM AOT-компилятора и прозрачного параллелизма L3

> **Последнее обновление**: 2026-03-10

## Аннотация

В данном документе описывается проектирование LLVM AOT-компилятора для языка YaoXiang, целью которого является генерация машинного кода + DAG-метаданных через предварительную компиляцию, а выполнение осуществляется **глобальным DAG-планировщиком** среды выполнения на основе анализа зависимостей **снизу вверх**.

**Ключевые инновации**:
- Не «при встрече с вызовом функции генерировать Future», а **обратный анализ зависимостей от «места, где нужен результат»**
- **Листовые узлы выполняются параллельно первыми**, цепочки зависимостей обходятся последовательно снизу вверх
- **Изолированные DAG выполняются независимо параллельно**: узлы без потребителей не блокируют основной поток
- **Бесконечные циклы как фоновые DAG**: планировщик выполняет их по частям, не зависая

Данная архитектура существенно отличается от модели Rust async/await + runtime tokio:
- Rust: пользователь пишет `async fn`, компилятор генерирует конечный автомат
- YaoXiang: пользователь пишет обычные функции, **компилятор автоматически анализирует DAG**, планировщик выполняет снизу вверх

Соответствует проектированию прозрачного 并发 L3 из RFC-001: по умолчанию @auto (автоматический параллелизм), @block синхронный — это особый случай, решающий проблему «окрашенных функций».

## Мотивация

### Зачем нужен LLVM AOT-компилятор?

В настоящее время YaoXiang имеет только интерпретатор в качестве исполняющего backend, что создаёт следующие проблемы:

| Проблема | Влияние |
|----------|---------|
| Узкое место производительности | Интерпретируемое выполнение медленнее машинного кода в 10-100 раз |
| Сложность развёртывания | Необходимо поставлять интерпретатор и среду выполнения |
| Проблема «окрашенных функций» | Синхронные функции не могут вызывать 并发-функции |

### Проблема «окрашенных функций» и прозрачный 并发 L3

**Традиционное проектирование (текущее)**:
- Синхронные функции (синие) → не могут вызывать → 并发-функции (красные)
- Синхронность — по умолчанию, 并发 требует маркера `spawn`
- Цвет «заразен»: однажды использовав 并发, вся цепочка вызовов становится 并发

**Прозрачный 并发 L3 из RFC-001 (цель)**:
- L3: прозрачный 并发 по умолчанию (@auto)
- L2: явный spawn 并发
- L1: режим @block синхронного выполнения

**Инвертированное проектирование (RFC-018)**:
- По умолчанию L3 прозрачный 并发, автоматический анализ DAG-зависимостей при компиляции
- Решает проблему «окрашенных функций»: синхронные функции могут напрямую вызывать код «по умолчанию 并发»
- @block используется только как особый случай для принудительного последовательного выполнения

### Ключевая инновация: выполнение снизу вверх + глобальный DAG

Ключевая инновация данного проектирования — **модель выполнения снизу вверх**:

```
Традиционный вызов (сверху вниз):
  call fetch(url) → выполнение → возврат результата

Выполнение снизу вверх:
  print(a) ← начинаем с «места, где нужен результат»
       ↑
  fetch(url0) ← анализируем зависимости, ищем в обратном направлении

  fetch(url1) ← изолированный, выполняется параллельно независимо
```

**Ключевые различия**:
- Не «при встрече с вызовом функции генерировать Future»
- Анализ зависимостей в обратном направлении от «конечного результата»
- Узлы без потребителей (изолированные) не выполняются или выполняются независимо параллельно
- Бесконечные циклы как фоновые DAG, планировщик выполняет их по частям

### Глобальный DAG-планировщик

```
Представление DAG всей программы:

        print(result) ─────────────────────────┐
           │                                    │
    ┌──────┴──────┐                             │
    │             │                             │
process(a)   process(b)                        │
    │             │                             │
compute(x)   compute(y)  ←── изолированный DAG ─┤
    │                                           │
fetch(url0)  fetch(url1)  fetch(url2)          │
    (выполнен)                                  │

Параллельно существует фоновый DAG (while True):
    ┌─────────────────────────────────────────┐ │
    │  while True:                            │ │
    │      update_ui()                        │ │
    │      fetch_new() ──→ process(data)      │ │
    └─────────────────────────────────────────┘ │
```

**Принцип работы планировщика**:
```
1. Обратный анализ от «конечного результата»:
   print(result) → зависит от process → зависит от fetch

2. Построение глобального DAG:
   - Листовые узлы: fetch (без зависимостей)
   - Внутренние узлы: process, compute
   - Корневой узел: print

3. Выполнение:
   - fetch выполняются параллельно
   - process ожидает завершения fetch
   - print ожидает завершения process
   - изолированный compute выполняется независимо параллельно

4. Пропуск уже выполненного:
   - Если узел уже выполнялся, зависимые от него узлы
     могут использовать готовый результат
```

### Обработка бесконечных циклов

```
Сценарий 1: одиночный while/for (без накладных расходов调度)
──────────────────────────────────────────────
main: () -> () = {
    while True {
        update_ui()
        fetch_data()
    }
}
→ только один бесконечный цикл
→ прямое синхронное выполнение, никакой разницы от обычного кода

Сценарий 2: несколько while (автоматическое разделение)
──────────────────────────────────────────────
main: () -> () = {
    while True { update_ui() }      # фоновая задача 1
    while True { network_poll() }  # фоновая задача 2
    server_loop()                   # основная задача
}
→ 3 независимые задачи
→ планировщик переключается между ними
→ настоящий параллелизм

Адаптивный планировщик:
──────────────────────────────────────────────
if количество_задач == 1:
    выполнение напрямую (синхронно)
else:
    планирование по частям (параллельно)
```

**Обработка фоновых DAG**:
```
Основной DAG (имеет завершение):
    fetch → process → print → завершение

Фоновый DAG (бесконечный цикл):
    while True → update_ui → fetch_new → process → возврат в начало

Планировщик:
    - основной DAG завершается после выполнения
    - фоновый DAG работает вечно, но планировщик выполняет его
      «по частям»
    - не зависает в цикле
```

## Предложение

### Основное проектирование

```
┌─────────────────────────────────────────────────────┐
│  Compile-time                                        │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐            │
│  │ Parser  │→│DAG-анализ│→│LLVM Codegen│→ машинный код│
│  └─────────┘  └─────────┘  └─────────┘            │
│                      ↓                           │
│              Генерация: DAG-метаданные              │
└─────────────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────┐
│  Runtime                                             │
│  ┌─────────────────────────────────────────────┐ │
│  │  Библиотека DAG-планировщика                 │ │
│  │  • Загрузка машинного кода                   │ │
│  │  • Чтение DAG-метаданных                    │ │
│  │  • Ленивое планирование: приостановка        │ │
│  │    вызовов, выполнение по требованию        │ │
│  │  • Поддержка параллельного/последовательного  │ │
│  │    выполнения                                │ │
│  └─────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### Процесс выполнения снизу вверх

```
Код пользователя:
    main: () -> () = {
        pages = urls.map(|url| fetch(url))
        results = pages.map(|page| parse_page(page))
        save_results(results)
    }

Фаза 1: Анализ снизу вверх (compile-time)
─────────────────────────────────────────
Начинаем с save_results(results):
    "нужен results" → зависит от parse_page(results)
    "нужен page0" → зависит от fetch(url0)
    "нужен page1" → зависит от fetch(url1)
    ...

Построение глобального DAG:
    fetch(url0), fetch(url1), fetch(url2) ← листовые узлы
           ↓
    parse_page(page0), parse_page(page1)   ← зависят от листьев
           ↓
    save_results                          ← корневой узел

Фаза 2: Параллельное выполнение листьев (runtime)
─────────────────────────────────────────
Планировщик находит все листовые узлы:
    - fetch(url0), fetch(url1), fetch(url2) без зависимостей → параллельно
    - контролирует 并发数 (например, 16)

Фаза 3: Обход снизу вверх
─────────────────────────────────────────
Когда parse_page нуждается в page0:
    - проверяет, готов ли page0
    - готов → выполняет parse_page
    - не готов → ожидает, после завершения продолжает

Фаза 4: Изолированные выполняются независимо параллельно
─────────────────────────────────────────
Если результат某个 fetch никому не нужен:
    - выполняется как «изолированный DAG»
    - может использовать другой核心, не влияет на основной поток
```

### Структура результата компиляции

```rust
/// Результат компиляции: машинный код + DAG-метаданные
pub struct CompiledArtifact {
    /// Машинный код, скомпилированный LLVM (ELF/Mach-O/COFF)
    machine_code: Vec<u8>,

    /// DAG-метаданные: описание зависимостей функций
    dag: DAGMetadata,

    /// Таблица точек входа
    entries: Vec<EntryPoint>,

    /// Информация о типах (для FFI)
    type_info: TypeInfo,
}

/// DAG-метаданные
pub struct DAGMetadata {
    /// Узлы: вызовы функций
    nodes: Vec<DAGNode>,
    /// Ребра: зависимости (from, to)
    edges: Vec<(usize, usize)>,
}

/// Одиночный DAG-узел
pub struct DAGNode {
    /// ID функции
    pub function_id: usize,
    /// ID зависимых узлов
    pub deps: Vec<usize>,
    /// Маркер побочных эффектов (@IO / @Pure)
    pub effect: EffectTag,
}
```

### Интерфейс планировщика runtime

```rust
/// trait DAG-планировщика
pub trait DAGScheduler: Send + Sync {
    /// Планирование выполнения
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue;

    /// Выполнение одиночной функции
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}

/// Реализация планировщика
pub struct DefaultDAGScheduler {
    /// Пул потоков
    thread_pool: ThreadPool,
    /// Результат компиляции
    artifact: CompiledArtifact,
    /// Максимальный 并发数
    max_parallelism: usize,
}

impl DefaultDAGScheduler {
    pub fn new(artifact: CompiledArtifact, num_workers: usize) -> Self {
        Self {
            thread_pool: ThreadPool::new(num_workers),
            artifact,
            max_parallelism: num_workers * 2, // адаптивный контроль粒度
        }
    }
}

impl DAGScheduler for DefaultDAGScheduler {
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint]) -> RuntimeValue {
        // 1. Обход тела функции, приостановка всех вызовов
        // 2. Построение списка задач для выполнения
        // 3. Планирование выполнения по порядку зависимостей (контроль 并发数)
        // 4. Триггер выполнения при необходимости значения
        // 5. Возврат результата
    }
}
```

### Стратегии планирования

| Аннотация | Сценарий | Стратегия планирования |
|-----------|----------|------------------------|
| `@auto` (по умолчанию, L3) | Прозрачный 并发 | Ленивое DAG-планирование, выполнение без зависимостей параллельно |
| `@block` (L1) | Принудительная синхронизация | Без DAG, чистое последовательное выполнение |
| Циклическая зависимость | Обнаружена при runtime | Сообщение об ошибке |

### Обработка побочных эффектов: неявная Effect System

Пользователь не замечает обработки побочных эффектов, компилятор выводит автоматически:

```
Код пользователя:
  print("a");
  print("b");
  let x = compute(1);
  let y = compute(2);

Вывод компилятора:
  print → @IO (внешний вызов)
  compute → @Pure (чистая функция)

Выполнение планировщиком:
  print("a") ──→ последовательно (все @IO)
  print("b") ──→ последовательно
  compute(1) ─┬─→ параллельно (DAG-планирование)
  compute(2) ─┘
```

### Отношение к трёхуровневому runtime

RFC-008 определяет трёхуровневую архитектуру Embedded / Standard / Full runtime. Соответствие LLVM AOT-компилятора трём уровням runtime:

| Runtime | Поведение LLVM AOT |
|---------|-------------------|
| **Embedded** | Без DAG-планирования, прямо генерирует последовательный машинный код |
| **Standard** | DAG + однопоточное планирование (num_workers=1) |
| **Full** | DAG + многопоточное планирование (num_workers>1), поддержка WorkStealing |

### Проектирование интерфейса планировщика

```rust
/// Стратегия планирования
pub enum ScheduleStrategy {
    /// @block: принудительное последовательное выполнение, без DAG
    Serial,
    /// @eager: энергичное вычисление, ожидание завершения зависимостей
    Eager,
    /// @auto (по умолчанию): ленивое планирование, автоматическое DAG-планирование
    Lazy,
}

/// Маркер побочных эффектов
pub enum EffectTag {
    /// Чистая функция, без побочных эффектов
    Pure,
    /// Имеет побочные эффекты I/O
    IO,
}

/// trait DAG-планировщика
pub trait DAGScheduler: Send + Sync {
    /// Планирование выполнения (с параметром стратегии)
    fn schedule(&self, dag: &DAGMetadata, entries: &[EntryPoint], strategy: ScheduleStrategy) -> RuntimeValue;

    /// Выполнение одиночной функции
    fn execute(&self, func: &CompiledFunction, args: &[RuntimeValue]) -> RuntimeValue;
}
```

## Компромиссы

### Преимущества

1. **Повышение производительности**: AOT-компиляция быстрее интерпретируемого выполнения в 10-100 раз
2. **Решение проблемы «окрашенных функций»**: по умолчанию 并发, синхронность — особый случай
3. **Единая среда выполнения**: интерпретатор и LLVM совместно используют один планировщик
5. **Неявные побочные эффекты**: пользователь не замечает, компилятор обрабатывает автоматически
6. **Безопасность владения**: опирается на модель владения в стиле Rust, нет гонок данных

### Недостатки

1. **Сложность реализации**: требуется опыт интеграции LLVM
2. **Время компиляции**: AOT-компиляция медленнее интерпретатора
3. **Сложность отладки**: отладка AOT-кода сложнее, чем интерпретатора

### Согласованность с дизайном RFC

| RFC | Согласованность |

| RFC-009 Модель владения | ✅ Корректная реализация ARC runtime |

## Связанные работы

### Lazy Task Creation (1990)[^1]

| Атрибут | Описание |
|---------|----------|
| Организация | MIT |
| Авторы | James R. Larus, Robert H. Halstead Jr. |
| Суть | Ленивое создание подзадач, создание по требованию |
| Ценность | Техническая база, источник концепции ленивого планирования |

**Ключевая идея**: задачи создаются не сразу, а отложенно. Когда родительской задаче нужен результат дочерней, только тогда дочерняя задача и создаётся. Это решает проблему накладных расходов на выполнение задач с мелкой粒度ностью[^1].

### Lazy Scheduling (2014)[^2]

| Атрибут | Описание |
|---------|----------|
| Организация | University of Maryland |
| Авторы | Tzannes, Caragea |
| Суть | Адаптивное планирование runtime, без дополнительного состояния |
| Ценность | Дизайн планировщика, адаптивный контроль粒度 |

**Ключевая идея**: «ленивое выполнение» автоматически контролирует粒度ность, не требует поддержания сложного состояния. Когда система занята, задачи автоматически объединяются; когда простаивает — автоматически разделяются[^2].

### Язык SISAL[^3]

| Атрибут | Описание |
|---------|----------|
| Организация | Lawrence Livermore National Laboratory (LLNL) |
| Суть | Язык с единственным присваиванием, Dataflow-графы, неявный параллелизм |
| Ценность | Доказательство осуществимости, производительность близка к Fortran |

**Ключевой вклад**: SISAL доказал, что модель Dataflow в промышленных приложениях может достигать производительности, близкой к Fortran[^3].

### Параллельная схема Mul-T[^4]

| Атрибут | Описание |
|---------|----------|
| Организация | MIT |
| Суть | Конструкция Future, реализация Lazy Task Creation |
| Ценность | Ссылка на конкретную реализацию |

**Ключевой механизм**:
```scheme
;; Синтаксис Multilisp / Mul-T
(let ((a (future compute-a))      ;; немедленный возврат future
      (b (future compute-b)))      ;; немедленный возврат future
  (join a b))                      ;; ожидание завершения
```

### Сравнительная сводка

| Технология | Ленивое создание | DAG-анализ | Обработка побочных эффектов | Владение |
|------------|------------------|-------------|----------------------------|----------|
| Lazy Task Creation[^1] | ✅ | ❌ | ❌ | N/A |
| Lazy Scheduling[^2] | ✅ | ❌ | ❌ | N/A |
| SISAL[^3] | ✅ | ✅ (глобальный) | N/A (единственное присваивание) | N/A |
| Mul-T[^4] | ✅ | ❌ | ❌ | N/A |
| **YaoXiang** | ✅ | ✅ (внутри функций) | ✅ (неявный) | ✅ (ARC) |

**Инновация YaoXiang**: использование современных языковых возможностей (владение + неявные побочные эффекты) упрощает традиционный дизайн, ограничение DAG внутри блока функций снижает сложность.

## Сравнение с традиционными методами автоматического параллелизма

### Традиционные компиляторы: параллелизм на уровне циклов

Коммерческие компиляторы (Intel Fortran, Oracle Fortran) применяют **автоматическую параллелизацию на уровне циклов**[^5]:

**Основной процесс**:
```
1. Идентификация подлежащих параллелизации циклов
2. Анализ зависимостей массивов внутри цикла
3. Определение наличия зависимостей между итерациями цикла
4. При отсутствии зависимостей генерация многопоточного кода
```

**Технологии анализа зависимостей**:

| Технология | Описание |
|------------|----------|
| **Зависимости данных** | Обращаются ли два доступа к одной и той же области памяти |
| **Use-Def** | Отношения определения и использования переменных |
| **Анализ алиасов** | Указывают ли указатели на одну область памяти |

**Условия параллелизуемости цикла**:
```fortran
! Можно параллелизовать
DO I = 1, N
  A(I) = C(I)
END DO

! Нельзя параллелизовать (зависимость от предыдущей итерации)
DO I = 2, N
  A(I) = A(I-1) + B(I)
END DO
```

### Haskell: механизм Spark

GHC (Glasgow Haskell Compiler) применяет **механизм Spark** для реализации параллелизма чистых функций[^6]:

```haskell
-- rpar: параллельное выполнение, создание spark
-- rseq: последовательное выполнение, ожидание завершения

example = do
  a <- rpar (f x)   -- создание spark, параллельное выполнение f x
  b <- rpar (g y)   -- создание spark, параллельное выполнение g y
  rseq a            -- ожидание завершения a
  rseq b            -- ожидание завершения b
  return (a, b)
```

**Механизм пула Spark**:
- Spark берутся из пула и分配ются свободным процессорным ядрам
- Если spark не используется (никто не ждёт результата), он удаляется GC
- Это решает проблему粒度: слишком мелкие spark отбрасываются

### Язык Clean: типы уникальности

Язык Clean реализует безопасный параллелизм через **типы уникальности (Uniqueness Types)**[^7]:

```clean
-- *Array означает уникальность, можно безопасно изменять
modify :: *Array Int -> *Array Int
```

**Ключевая идея**: если значение имеет единственную ссылку, его можно безопасно изменять в параллельной среде, так как другие ссылки не увидят промежуточное состояние.

### Программные срезы и графы зависимостей

**Граф программных зависимостей (PDG)** — основа обнаружения параллелизма:

```
Узлы: операторы
Ребра: зависимости по данным + управляющие зависимости

Обнаружение параллелизма:
  Если между двумя узлами нет достижимого пути → можно параллелизовать
```

### Комплексное сравнение

| Метод | Анализ зависимостей | Granularity | Обработка побочных эффектов | Типичный сценарий |
|-------|---------------------|-------------|----------------------------|-------------------|
| Intel/Oracle Fortran[^5] | Сложный анализ массивов | Итерации цикла | N/A | Научные вычисления |
| GHC Spark[^6] | Предположение о чистых функциях | Выражения | N/A | Функциональное программирование |
| Clean[^7] | Типы уникальности | Перезапись графов | N/A | Функциональное программирование |
| **YaoXiang** | Гарантии владения | Вызовы функций | Неявный вывод | Универсальный |

| Характеристика | Rust async | YaoXiang LLVM AOT |

| Аннотации | async/await | @auto/@eager/@block |

### Приложение B: Примеры оптимизации планировщика

**Сценарий 1: Планировщик обнаружил возможность合并执行**

```
Исходный DAG:
  compute_a() ──┐
  compute_b() ──┼──→ compute_c()

После оптимизации планировщика:
  Объединение compute_a + compute_b в одну задачу
  → сокращение накладных расходов调度
```

**Сценарий 2: Зависимость не используется**

```
let a = expensive_compute(); // вычислено
let b = other_thing();       // не нуждается в a
print(b);                    // возврат b напрямую, пропуск a
```
