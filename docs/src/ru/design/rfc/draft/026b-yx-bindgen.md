# RFC-026b: Инструментарий yx-bindgen

> **Родительский RFC**: [RFC-026: Основной механизм FFI](../accepted/026-ffi-core-mechanism.md)
>
> **Зависимости**: Реализация данного RFC зависит от первоочередного принятия основного механизма FFI из RFC-026.

## Резюме

yx-bindgen выполняет механическое преобразование заголовочных файлов C в файлы FFI-привязок `.yx`, генерируя привязки библиотек, разделение типов (непрозрачные дескрипторы / прозрачные типы), объявления функций и привязки методов.

**Два ключевых принципа**:
1. **Механический вывод черновика без предположений о владении** — владение задаётся типом YaoXiang, пользователь подтверждает его согласно документации C
2. **Платформо-корректные гарантии размещения** — размеры полей и выравнивание прозрачных типов вычисляются согласно целевой платформе, что гарантирует бинарное соответствие с C `struct`

## Мотивация

Ручное написание FFI-привязок утомительно и чревато ошибками — C-библиотеки могут содержать десятки и сотни функций. Ещё опаснее **размещение прозрачных типов**: при ручном написании `Timespec: Type = { tv_sec: Int64, tv_nsec: Int64 }` размеры полей, выравнивание и отступы должны точно соответствовать `struct timespec` C на целевой платформе, иначе чтение/запись C по некорректному размещению приведёт к выходу за границы (гарантия доверия из RFC-026 §2.2). yx-bindgen механически вычисляет размещение из `.h`, устраняя риск ручного предоставления таких гарантий.

## Предложение

### 1. Использование

```bash
yx-bindgen --header /usr/include/sqlite3.h --lib sqlite3 --output sqlite3_bindings.yx
```

`--lib` задаёт имя линкуемой библиотеки и генерирует заголовок `Native.c("libsqlite3")`.

### 2. Содержимое генерации

yx-bindgen генерирует четыре категории содержимого:
- **Заголовок привязки библиотеки**: `lib = Native.c("libxxx")`
- **Разделение типов**: чёрный ящик-указатель → непрозрачный дескриптор; структура данных → прозрачный тип (с размещением)
- **Объявления функций**: привязки `lib("symbol")`
- **Привязки методов**: `Type.method` или `[N]`

### 3. Автоматическое определение при разделении типов

yx-bindgen определяет принадлежность типа по способу его использования в C (RFC-026 §2):

| Тип C | Определение | Вывод yx-bindgen |
|--------|------|----------------|
| `typedef struct T T;` (неполный тип / используется только как указатель) | Чёрный ящик → непрозрачный дескриптор | `T: Type = lib("T")` |
| `struct T { fields };` (поля видимы, читаются и пишутся) | Данные → прозрачный тип | `T: Type = { ...размещение }` |
| `int/long/float/double` | Значение | `Int32/Int64/Float32/Float64` |
| `char*` (параметр/возврат) | Значение (копия) | `String` |
| `void*` | Без типа | `*Void` (системный уровень) |

Неполный тип (`typedef struct sqlite3 sqlite3;` без тела определения) → неизбежно чёрный ящик-дескриптор. Полностью определённый `struct` → прозрачный тип, размещение полей вычисляется механически.

### 4. Вычисление размещения (ключ для прозрачных типов)

Для полностью определённого `struct` yx-bindgen вычисляет смещение, размер и выравнивание каждого поля согласно ABI целевой платформы:

```c
struct timespec {
    time_t tv_sec;    // зависит от платформы: Linux x86_64 = 8 байт
    long   tv_nsec;   // 8 байт
};
```

```yaoxiang
// Целевая платформа Linux x86_64
Timespec: Type = {
    tv_sec: Int64,    // смещение 0, размер 8
    tv_nsec: Int64    // смещение 8, размер 8
}
// Общий размер 16, выравнивание 8 — бинарно совпадает с C struct
```

**Платформенные различия**: размеры `time_t`, `long`, `size_t` варьируются в зависимости от платформы. yx-bindgen выбирает корректное отображение согласно `--target`, гарантируя побайтовое соответствие сгенерированного прозрачного типа C-структуре на целевой платформе. Это ключевая ценность для устранения риска ручного предоставления гарантий размещения из RFC-026 §2.2.

### 5. Пример генерации

Вход (абстракция `sqlite3.h`):

```c
typedef struct sqlite3 sqlite3;          // неполный → чёрный ящик-дескриптор
typedef struct sqlite3_stmt sqlite3_stmt;

int sqlite3_open(const char *filename, sqlite3 **ppDb);
int sqlite3_close(sqlite3 *db);
int sqlite3_exec(sqlite3 *db, const char *sql, ...);
```

Выход (`sqlite3_bindings.yx`):

```yaoxiang
// sqlite3_bindings.yx — сгенерировано автоматически, не редактировать вручную

// ============================================================================
// Привязка библиотеки
// ============================================================================
sqlite3 = Native.c("libsqlite3")

// ============================================================================
// Типы (неполные типы → непрозрачные дескрипторы)
// ============================================================================
SqliteDb: Type = sqlite3("sqlite3")
SqliteStmt: Type = sqlite3("sqlite3_stmt")

// ============================================================================
// Функции + привязки методов
// ============================================================================
SqliteDb.open: (filename: String) -> ?SqliteDb = sqlite3("sqlite3_open")
SqliteDb.exec: (sql: String) -> Int32 = sqlite3("sqlite3_exec")
SqliteDb.close: () -> Int32 = sqlite3("sqlite3_close")

// ============================================================================
// Деструктор (опционально, подтверждается пользователем)
// ============================================================================
SqliteDb.drop = SqliteDb.close
```

### 6. Пользовательская корректировка

Сгенерированные привязки являются черновиком; пользователь подтверждает семантику владения согласно документации C-библиотеки (yx-bindgen не делает предположений):

```yaoxiang
// Сгенерировано: String по умолчанию (копия) — корректно в большинстве случаев
SqliteDb.errmsg: () -> String = sqlite3("sqlite3_errmsg")

// getenv возвращает указатель на статическую область, не следует копировать
// и не следует принимать владение — пользователь меняет на голый указатель
getenv: (name: String) -> *const U8 = Native.c("libc")("getenv")

// Дескриптор, принадлежащий библиотеке — пользователь подтверждает,
// привязывать ли .drop
SqliteDb.drop = SqliteDb.close   // сгенерированное предложение, подтверждается пользователем
```

**Ключевой момент**: размещение гарантируется yx-bindgen механически (платформо-корректно), но семантика владения (наличие `.drop`, является ли `char*` копией или голым указателем) подтверждается пользователем согласно документации C. Инструмент отвечает за механическую корректность, пользователь — за семантическую.

## Компромиссы

### Преимущества

1. **Автоматизация гарантий размещения**: размещение прозрачных типов вычисляется механически согласно платформе, что устраняет ошибки ручного задания отступов/выравнивания
2. **Автоматическое определение при разделении типов**: неполный тип → дескриптор, полный `struct` → прозрачный тип
3. **Аудитируемость**: вывод представляет собой обычный `.yx`, который пользователь может читать, изменять и коммитить в систему контроля версий

### Недостатки

1. **Владение по-прежнему требует ручного подтверждения**: yx-bindgen не предполагает `.drop`, не предполагает семантику `char*`
2. **Зависимость от парсинга C-заголовков**: требуется libclang или tree-sitter-c
3. **Платформенная специализация**: разные `--target` генерируют разное размещение; кроссплатформенные пакеты требуют нескольких версий или выбора во время выполнения

## Стратегия реализации

- [ ] Парсинг заголовочных файлов C (libclang)
- [ ] Определение при разделении типов (неполный тип vs полный `struct`)
- [ ] Вычисление размещения по ABI платформы (offset/size/align, согласно --target)
- [ ] Генерация кода (привязка библиотеки + типы + функции + методы)
- [ ] Интеграционное тестирование (sqlite3, libcurl, верификация размещения на нескольких платформах)

## Связь с другими RFC

- **RFC-026** (родительский): Основной механизм FFI — сгенерированные привязки используют его синтаксис `Native.c("lib")("sym")` и разделение типов
- **RFC-026a**: Расширяемый механизм FFI — в будущем может быть расширен для генерации привязок других механизмов, таких как `Native.wasm`
