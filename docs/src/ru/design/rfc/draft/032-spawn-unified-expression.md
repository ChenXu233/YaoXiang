# RFC-032: унифицированный модификатор выражений spawn

> **Ключевое понимание**: `spawn` не должен модифицировать только блок `{}`. Он может модифицировать **произвольное выражение**. `spawn for` — не особый синтаксис — это естественная комбинация `spawn` + выражения `for`.

## Аннотация

Расширение `spawn` от `spawn { }` (модификация только блока) до `spawn <expr>` (модификация произвольного выражения). `Expr::SpawnFor` удаляется из AST и естественно заменяется на `Expr::Spawn { body: Expr::For { .. } }`. Структурные типы выражений (Block, For, While, If и т.д.) входят в систему типов как новые варианты `MonoType`, `Spawn<T>` оборачивает структуру параллельно исполняемого вычисления, помечается на этапе компиляции и стирается после проверки.

## Мотивация

### Почему необходимо это изменение?

В настоящее время `spawn for x in items { body }` представляет собой независимую комбинацию ключевых слов, и в AST имеется специальный `Expr::SpawnFor` для её представления. Это нарушает ортогональность языка:

1. **Неунифицированный синтаксис**: `spawn` может модифицировать только блок `{}`, `spawn for` — жёстко закодированное исключение.
2. **Отсутствие ортогональности**: комбинации `spawn while`, `spawn if` и т.п. не могут быть выражены естественно.
3. **Неполная система типов**: `spawn` невидим в системе типов, невозможно получить структуру параллелизма через отражение типов.

### Текущая проблема

```rust
// В AST два варианта spawn
Spawn { body: Box<Block>, span: Span },         // spawn { ... }
SpawnFor { var, var_mut, iterable, body, span },  // spawn for x in items { ... }

// MonoType содержит только типы значений, без типов структуры вычислений
// тип spawn { a, b } = Tuple(T_a, T_b)  ← информация "это spawn" потеряна
// тип spawn for       = List(T)         ← информация "это параллелизм данных" потеряна
```

## Предложение

### Основной дизайн

`spawn <expr>`: `spawn` модифицирует произвольное выражение. Форма выражения определяет, как DAG разбивает задачи.

**Всё есть тип**: `MonoType` расширяется с «типов значений» до «типов значений + типов структуры вычислений». Каждой ключевой структуре выражения соответствует вариант типа в системе типов. `Spawn<T>` оборачивает структуру параллельно исполняемого вычисления.

### Ментальная модель пользователя

`spawn` = «выполнить это выражение параллельно». Форма выражения определяет способ разбиения:

| Форма выражения | Поведение параллелизма | Тип |
|-----------|---------|------|
| `spawn { a, b, c }` | `a`, `b`, `c` исполняются независимо параллельно | `Spawn(Block(Tuple(T_a, T_b, T_c)))` |
| `spawn for x in items { f(x) }` | N итераций исполняются независимо параллельно | `Spawn(ForExpr { body_ty: List(T) })` |
| `spawn while cond { step() }` | каждая итерация — независимая задача | `Spawn(WhileExpr { body_ty: List(T) })` |
| `spawn if c { a } else { b }` | выбранная ветвь целиком — область spawn | `Spawn(IfExpr { then_ty: T_a, else_ty: Some(T_b) })` |
| `spawn call(x)` | сам вызов — одна задача | `Spawn(Call { fn_ty: Fn(A→R), result_ty: R })` |
| `spawn 42` | единственная задача | `Spawn(Int)` |

Компилятор отвечает за DAG-анализ для определения зависимостей, рантайм планирует по модели GMP — задачи без зависимостей помещаются в рабочую очередь, воркеры захватывают их. Общая синхронизация блокирует, ожидая завершения всех задач.

**Отличие от Go**: `go` в Go означает «запустить и забыть», тогда как `spawn` в YaoXiang означает «разбить на параллельное выполнение, подождать завершения и затем продолжить».

### Ортогональность потока управления

| Комбинация | Семантика | Различие |
|------|------|------|
| `spawn for x in items { body }` | параллелизм по данным: каждая итерация = независимая задача | DAG-анализ зависимостей между итерациями |
| `for x in items spawn { body }` | каждая итерация создаёт область spawn | без анализа между итерациями |
| `spawn while cond { body }` | условный параллелизм: каждая итерация = независимая задача | межитерационные зависимости обеспечиваются условием |
| `while cond spawn { body }` | каждая итерация создаёт область spawn | семантика отличается от вышеописанной, но не требует специальной обработки |
| `spawn if c { a } else { b }` | весь if-else — одна область spawn | при выполнении ветвь выбирается по условию |
| `if c spawn { a } else { b }` | spawn только в одной ветви | внутри выражения if вложен spawn |

### Устраняемая сложность

- ❌ `Expr::SpawnFor` удаляется из AST
- ❌ `SpawnForAnalysis` удаляется из DAG-анализа
- ❌ `spawn for` больше не обрабатывается парсером как комбинация ключевых слов
- ❌ `Ir::SpawnFor` удаляется из IR

## Подробный дизайн

### 1. Слой AST

**До:**

```rust
Spawn { body: Box<Block>, span: Span },         // spawn { ... }
SpawnFor { var, var_mut, iterable, body, span },  // spawn for x in items { ... }
```

**После:**

```rust
Spawn { body: Box<Expr>, span: Span },           // spawn <произвольное выражение>
```

`Expr::SpawnFor` удаляется. AST-представление `spawn for x in items { body }`:

```rust
Expr::Spawn {
    body: Box::new(Expr::For {
        var: "x",
        iterable: items,
        body: body_block,
        ..
    })
}
```

**Особый случай IF**:

| Запись | Структура AST |
|------|---------|
| `spawn if cond { a } else { b }` | `Spawn { body: Expr::If { ... } }` |
| `if cond spawn { a } else { b }` | `Expr::If { then: Spawn { body: {a} }, else: {b} }` |

Семантика различается, но обе формы — естественные комбинации, не требующие специальных правил.

### 2. Слой парсера

`spawn` имеет наименьший приоритет связывания (как `return`) и поглощает всё последующее выражение:

```
spawn a + b        →  spawn (a + b)         ≠  (spawn a) + b
spawn f(x).y       →  spawn (f(x).y)
```

Изменение в парсере: в `pratt/nud.rs` `spawn` больше не требует `{`, а вызывает общий парсер выражений:

```
token spawn → parse_expr(min_precedence) → Expr::Spawn { body: expr }
```

`spawn for` больше не обрабатывается как комбинация ключевых слов — `for` обрабатывается общим парсером выражений и порождает `Expr::For`, а `spawn` только выполняет оборачивание.

### 3. Система типов

**Новые варианты `MonoType`:**

```rust
// ========== Типы структуры вычислений ==========

/// Выражение блока {}
Block(Box<MonoType>),

/// Выражение цикла for
ForExpr { body_ty: Box<MonoType> },

/// Выражение цикла while
WhileExpr { body_ty: Box<MonoType> },

/// Выражение ветвления if-else
IfExpr {
    then_ty: Box<MonoType>,
    else_ty: Option<Box<MonoType>>,
},

/// Выражение вызова функции
Call {
    fn_ty: Box<MonoType>,
    result_ty: Box<MonoType>,
},

/// Параллельная обёртка spawn: внутреннее выражение исполняется параллельно
/// Помечается на этапе компиляции, стирается после проверки типов
Spawn(Box<MonoType>),
```

**Правила вывода типов**: вывод типа каждого выражения возвращает «тип структуры вычислений». Без обёртки `Spawn` = последовательное исполнение, с обёрткой `Spawn` = параллельное исполнение. После проверки типов `Spawn` стирается, тип сводится к внутреннему типу значения.

**Поток проверки типов**:
1. Вывести тип T выражения body (тип структуры вычислений)
2. Если обёрнут в spawn, обернуть как `Spawn(T)`
3. При присваивании выполнить деструктуризацию: `results: List(Data) = spawn for ... {}` — извлечь `List(Data)` из `Spawn(ForExpr { body_ty: List(Data) })`

`Spawn<T>` стирается после проверки типов, рантайму не нужно знать, получены ли данные параллельно или последовательно. Однако отражение на этапе компиляции (`type_of(x)`) позволяет получить полную топологию параллелизма.

### 4. Слой DAG-анализа

Два текущих входа объединяются в один:

```rust
/// Унифицированный вход: диспетчеризация по типу выражения body
fn analyze_spawn_expr(body: &Expr, ...) -> SpawnAnalysis {
    match body {
        Expr::Block(block)       => analyze_block_tasks(block, ...),
        Expr::For { .. }         => analyze_iter_tasks(IterKind::For, body, ...),
        Expr::While { .. }       => analyze_iter_tasks(IterKind::While, body, ...),
        Expr::If { .. }          => analyze_if_task(body, ...),
        _                        => single_task(body, ...),
    }
}
```

**Унифицированная структура результата:**

```rust
struct SpawnAnalysis {
    source: TaskSource,
    plan: ExecutionPlan,
}

enum TaskSource {
    /// spawn { a, b, c } — N прямых подвыражений, известных на этапе компиляции
    Explicit(Vec<TaskInfo>),
    /// spawn for/while — N задач порождаются итерациями в рантайме
    Iterate {
        kind: IterKind,
        iter_var: String,
        iterable: Option<Expr>,      // есть у for, нет у while
        condition: Option<Expr>,     // есть у while, нет у for
        body: Block,
        reads: HashSet<String>,
        writes: HashSet<String>,
        resource_vars: HashSet<String>,
    },
}

enum IterKind { For, While }
```

Структура `SpawnForAnalysis` удаляется.

| Вид body | Как разбивается на задачи |
|-----------|--------------|
| `Expr::Block` | прямые подвыражения → список задач |
| `Expr::For` | каждая итерация → одна задача (параллелизм по данным) |
| `Expr::While` | каждая итерация → одна задача |
| `Expr::If` | выбранная ветвь целиком → одна задача |
| `Expr::Call` / прочее | само выражение → одна задача |

После завершения DAG-анализа рантайм планирует по модели GMP — задачи без зависимостей помещаются в рабочую очередь, воркеры захватывают их.

### 5. Слой IR / Codegen

`Ir::SpawnFor` удаляется. Всё унифицируется до `Ir::Spawn` с информацией `TaskSource`.

Трансляция HIR → IR генерирует вызовы рантайма в соответствии с `SpawnAnalysis.source`:
- `TaskSource::Explicit(tasks)` → список задач, известный на этапе компиляции
- `TaskSource::Iterate { .. }` → раскрытие в рантайме (управляется компилятором, аналог par_iter с нулевой стоимостью)

### 6. Слой размещения (Placement)

Две текущие ветви объединяются в одну:

```rust
// До
Expr::Spawn { body, .. } => self.check_block(body),
Expr::SpawnFor { body, iterable, .. } => {
    self.check_expr(iterable);
    self.check_block(body);
}

// После
Expr::Spawn { body, .. } => self.check_expr(body),   // body — это Expr, достаточно рекурсии
```

### 7. Обратная совместимость

Семантика существующего кода `spawn for` не изменяется: парсер автоматически разбирает `spawn for x in items { body }` как `Expr::Spawn { body: Expr::For }`. Внутреннее представление меняется, видимое пользователю поведение остаётся прежним.

Новый синтаксис становится доступным естественно:
```yx
spawn while has_next() {
    item = next()
    process(item)
}

spawn if use_cache {
    load_from_cache(key)
} else {
    fetch(key)
}
```

## Компромиссы

### Преимущества

1. **Синтаксическая ортогональность**: `spawn` + произвольный поток управления = естественная комбинация параллелизма.
2. **Всё есть тип**: система типов полностью фиксирует структуру вычислений, отражение на этапе компиляции даёт топологию параллелизма.
3. **Устранение особых случаев**: удаление `Expr::SpawnFor` и связанного специального кода обработки.
4. **Расширяемость**: новые структуры потока управления в будущем автоматически комбинируются с `spawn` без изменения логики spawn.

### Недостатки

1. **Раздувание системы типов**: 6 новых вариантов `MonoType`, увеличение сложности проверки типов.
2. **Разрушающее изменение**: изменение внутреннего представления AST/IR, требуется обновление всего кода, потребляющего `Expr::SpawnFor`.
3. **Вывод типов выражений**: теперь каждое выражение должно возвращать тип структуры вычислений, большая область воздействия.

## Альтернативы

| Подход | Почему не выбран |
|------|-------------|
| Сохранить `spawn for` как независимый синтаксис | нарушает ортогональность, остаётся единственной комбинацией ключевых слов-исключением в языке |
| `spawn` модифицирует только `{}`, параллелизм по данным — в стандартной библиотеке `par_iter` | базовая возможность языка переносится в библиотеку, теряется DAG-анализ и обнаружение конфликтов ресурсов на уровне компилятора |
| Удалить `SpawnFor`, но не вводить типы структуры вычислений | система типов теряет способность к отражению, spawn становится невидим на уровне типов |

## Стратегия реализации

### Разбивка по этапам

1. **AST + Parser**: `Spawn { body: Box<Expr> }`, удаление `SpawnFor`.
2. **Система типов**: добавить 6 вариантов `MonoType`, все выражения возвращают типы структуры вычислений.
3. **Унификация DAG-анализа**: объединить входы, объединить перечисление `TaskSource`.
4. **Адаптация IR / Codegen**: удалить `Ir::SpawnFor`, унифицировать пути обработки.
5. **Упрощение Placement**: удалить ветвь `SpawnFor`.
6. **Тестирование и верификация**: все существующие тесты `spawn for` должны проходить.

### Область воздействия

| Файл/каталог | Изменения |
|-----------|------|
| `frontend/core/parser/ast.rs` | `Spawn` body изменён на `Box<Expr>`, удалён `SpawnFor` |
| `frontend/core/parser/pratt/nud.rs` | обработчик `spawn` упрощён до общего парсера выражений |
| `frontend/core/types/mono.rs` | новые варианты `Block`/`ForExpr`/`WhileExpr`/`IfExpr`/`Call`/`Spawn` |
| `frontend/core/spawn/analysis.rs` | унифицированный вход, объединение `TaskSource` Explicit + Iterate |
| `frontend/core/spawn/placement.rs` | удалена ветвь `SpawnFor` |
| `frontend/core/typecheck/` | все узлы выражений адаптированы к выводу типов структуры вычислений |
| `middle/core/ir.rs` | удалён `Ir::SpawnFor` |
| `middle/` (IR gen, codegen) | унификация пути spawn, стирание типа `Spawn` |
| `tests/yaoxiang/04-concurrency/spawn_for.yx` | семантика не изменяется, верификация проходит |

- RFC-010 (унифицированный синтаксис типов) — основа для изменения системы типов.
