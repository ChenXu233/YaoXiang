# RFC-033: Оператор рефлексии `^^`

> **Ссылки**:
>
> - [RFC-010: Унифицированный синтаксис типов — модель name: type = value](../accepted/010-unified-type-syntax.md)
> - [RFC-011: Проектирование системы обобщённых типов](../accepted/011-generic-type-system.md)
> - [RFC-027: Предикаты времени компиляции и унифицированная статическая верификация](../accepted/027-compile-time-evaluation-types.md)
> - [RFC-011a: Реализация интерфейсов и динамическая диспетчеризация](../draft/011a-interface-implementation.md)

## Резюме

В данном документе предлагается ввести оператор `^^` как точку входа в рефлексию, используемую для получения метаданных типов и значений. `^^T` возвращает объект статических метаданных типа `T`, а `^^obj` возвращает метаданные динамического типа значения `obj`. Объект метаданных представляет собой обычный record-тип, содержащий такую информацию, как имя, параметры, поля, и может использоваться как во время компиляции, так и во время выполнения.

## Мотивация

### Зачем нужна эта возможность?

1. **Сериализация/десериализация**: необходим доступ к информации о полях типов для автоматической генерации кода сериализации.
2. **Метапрограммирование на этапе компиляции**: необходим доступ к структуре типов во время компиляции для генерации кода или проверки ограничений.
3. **Отладка и инструментарий во время выполнения**: необходима возможность вывода информации о типах во время выполнения для облегчения отладки.
4. **Проверка типов во время выполнения**: необходимо определять отношения между типами во время выполнения, например: «Каков тип obj?»

### Текущая проблема

В текущей версии YaoXiang отсутствует механизм рефлексии, и нет возможности обращаться к метаданным типов во время компиляции или выполнения. Если использовать прямой доступ вроде `.name`, `.fields` для метаданных типа, это будет конфликтовать с пользовательскими полями:

```yaoxiang
Person: Type = { name: String, age: Int }

# Если Person.name — это имя в метаданных типа или поле name?
# Это вызовет трудности разбора и семантическую путаницу
```

Необходим синтаксис доступа к метаданным типа, **не вторгающийся в пространство имён обычных полей**.

## Предложение

### Основная конструкция

Предлагается ввести оператор `^^` как точку входа в рефлексию, чётко разделяющую обычный код и запросы метаданных.

**Два способа использования**:

1. **Статическая рефлексия (作用于 тип)**: `^^T` возвращает объект статических метаданных типа `T`.
2. **Динамическая рефлексия (作用于 значение)**: `^^obj` возвращает метаданные динамического типа значения `obj`.

**Структура метаданных**:

```yaoxiang
TypeMeta: Type = {
    name: String,
    params: Array(ParamMeta),
    fields: Array(FieldMeta),
    return_type: Type,
    refinement: Option(Expr)  # во время компиляции Some(Expr), во время выполнения None
}

ParamMeta: Type = {
    name: String,
    type: Type
}

FieldMeta: Type = {
    name: String,
    type: Type
}
```

**Иерархия вселенных (Universe)**: если `T: Type_n`, то `^^T: Type_{n+1}`, что соответствует стандартным правилам подъёма в теории типов.

**Приоритет**: `^^` — это унарный префиксный оператор с наивысшим приоритетом. `^^T.name` эквивалентно `(^^T).name`.

### Примеры

#### Базовое использование

```yaoxiang
Point: Type = { x: Float, y: Float }

# Статическая рефлексия
meta = ^^Point
print(meta.name)           # "Point"
print(meta.fields.len)     # 2
print(meta.fields[0].name) # "x"
print(fields[0].type)      # Float

# Динамическая рефлексия (требуется включить рефлексию во время выполнения)
obj = Point(1.0, 2.0)
meta = ^^obj
print(meta.name)           # "Point"
```

#### Обобщённые типы

```yaoxiang
List: (T: Type) -> Type = { data: Array(T), length: Int }

# Рефлексия самого обобщённого типа
meta = ^^List
print(meta.name)           # "List"
print(meta.params)         # [{ name: "T", type: Type }]

# Рефлексия конкретного инстанциированного типа
meta = ^^List(Int)
print(meta.name)           # "List(Int)"
print(meta.params)         # []
```

#### Функции

```yaoxiang
add: (a: Int, b: Int) -> Int = a + b

meta = ^^add
print(meta.name)           # "add"
print(meta.params)         # [{ name: "a", type: Int }, { name: "b", type: Int }]
print(meta.return_type)    # Int
```

#### Уточняющие (refinement) типы

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

# Во время компиляции: refinement имеет значение Some(Expr)
meta = ^^Positive
print(meta.name)           # "Positive"
print(meta.refinement)     # Some(AST(x > 0))

# Во время выполнения: refinement имеет значение None (стирается)
```

#### Использование в предикатах времени компиляции

```yaoxiang
# Проверка наличия полей у типа
HasFields: (T: Type) -> Type = { ^^T.fields.len > 0 }

# Проверка типа поля
HasFloatField: (T: Type) -> Type = {
    exists field in ^^T.fields: field.type == Float
}

# Использование
obj: HasFields(Point) = Point(1.0, 2.0)  # ✅ Проверка пройдена
# obj: HasFields(Int) = 42  # ❌ Проверка не пройдена
```

#### Пример сериализации

```yaoxiang
# Чистая функция времени компиляции: генерация JSON-строки
to_json: (T: Type) -> ((obj: T) -> String) = {
    meta = ^^T
    parts: Array(String) = []
    for field in meta.fields {
        # Генерация кода доступа к полю во время компиляции
        parts.push("\"${field.name}\": ${obj.${field.name}}")
    }
    return "{" + parts.join(", ") + "}"
}

# Использование
point_to_json = to_json(Point)
print(point_to_json(Point(1.0, 2.0)))  # '{"x": 1.0, "y": 2.0}'
```

### Синтаксические изменения

| До | После |
|------|------|
| Нет механизма рефлексии | `^^T` для получения метаданных типа |
| Нет механизма рефлексии | `^^obj` для получения метаданных динамического типа значения |

## Детальное проектирование

### Влияние на систему типов

- **Новые типы**: `TypeMeta`, `ParamMeta`, `FieldMeta`.
- **Иерархия вселенных**: тип, возвращаемый `^^T`, находится на один уровень выше, чем `T`.
- **Взаимодействие с обобщёнными типами**: поддерживаются как `^^List`, так и `^^List(Int)`.
- **Взаимодействие с функциями**: `^^add` возвращает метаданные функции (включая параметры и тип возврата).
- **Взаимодействие с уточняющими типами**: `^^Positive` возвращает метаданные уточняющего типа (включая выражение уточнения).

### Поведение во время выполнения

**Рефлексия во время компиляции**:
- `^^T` полностью вычисляется на этапе компиляции, результат встраивается как константа.
- Выражение уточнения доступно во время компиляции.

**Рефлексия во время выполнения**:
- По умолчанию отключена — нулевые накладные расходы.
- Включается флагом компиляции `--enable-runtime-reflection`.
- После включения `^^obj` возвращает метаданные динамического типа.
- Выражение уточнения стирается до `None` во время выполнения.

**Генерация по необходимости + treeshake**:
- Метаданные генерируются только для типов, фактически использующих `^^`.
- Для неиспользуемых типов метаданные не генерируются (treeshake).

### Изменения в компиляторе

1. **Лексический анализатор**: распознавание `^^` как единого токена.
2. **Синтаксический анализатор**: добавление правила префиксного выражения `^^`.
3. **Система типов**: определение типов `TypeMeta`, `ParamMeta`, `FieldMeta`.
4. **Проверщик типов**: генерация экземпляров метаданных для каждого типа.
5. **Вычислитель времени компиляции**: поддержка вычисления `^^T` на этапе компиляции.
6. **Среда выполнения (опционально)**: генерация RTTI для типов, подвергаемых рефлексии.

### Обратная совместимость

- ✅ Существующий синтаксис не затрагивается: `^^` — новый оператор, не конфликтует с существующим синтаксисом.
- ✅ Существующие типы не затрагиваются: все типы автоматически поддерживают `^^`.
- ✅ Существующие функции не затрагиваются: функции могут, но не обязаны использовать `^^`.
- ✅ Предикаты времени компиляции не затрагиваются: `^^T` в предикатах ведёт себя так же, как обычное содержимое.
- ✅ Среда выполнения не затрагивается: рефлексия во время выполнения по умолчанию отключена — нулевые накладные расходы.

## Компромиссы

### Преимущества

- **Единообразие**: функции, обобщённые типы и уточняющие типы обрабатываются единообразно.
- **Нулевые накладные расходы**: рефлексия времени компиляции полностью стирается; рефлексия времени выполнения опциональна.
- **Интеграция с существующими системами**: бесшовная интеграция с предикатами времени компиляции (RFC-027).
- **Простота**: `^^` — чистый символ, не конфликтует с пользовательскими идентификаторами.
- **Генерация по необходимости**: оптимизация treeshake, нулевые накладные расходы для неиспользуемых типов.

### Недостатки

- **Кривая обучения**: необходимо понимать семантику `^^` и структуру метаданных.
- **Накладные расходы во время выполнения**: включение рефлексии времени выполнения увеличивает расход памяти (один указатель на каждый экземпляр).
- **Сложность реализации**: требуется модификация нескольких компонентов компилятора.

## Альтернативы

| Вариант | Почему не выбран |
|------|--------------|
| Функция `reflect(T)` | Вводит дополнительный идентификатор в область видимости, может быть перекрыт пользователем |
| Функция `type_info(T)` | То же, что и выше |
| Одинарный оператор `^` | Возможен конфликт с побитовыми операциями; кроме того, именно из-за конфликта C++26 выбрал `^^` |
| Символы `@@`, `##` и т. п. | Нет прецедента, менее очевидны, чем `^^` |

## Этапы реализации

| Этап | Содержание | Зависимости |
|------|------|------|
| Фаза 1 | Разбор оператора `^^` на этапе компиляции | Нет |
| Фаза 2 | Определение структуры данных `TypeMeta` | Фаза 1 |
| Фаза 3 | Генерация метаданных на этапе компиляции | Фаза 2 |
| Фаза 4 | Поддержка рефлексии во время выполнения (опционально) | Фаза 3 |
| Фаза 5 | Интеграция с предикатами времени компиляции | Фаза 3 |

### Зависимости

```
Фаза 1 (разбор)
    ↓
Фаза 2 (структуры данных)
    ↓
Фаза 3 (метаданные времени компиляции)
    ↓
    ├────────────┐
    ↓            ↓
Фаза 4        Фаза 5
(рефлексия    (предикаты
 выполнения)  компиляции)
```

### Риски

- **Конфликты разбора**: `^^` потенциально может конфликтовать с существующим синтаксисом (анализ показал отсутствие конфликтов).
- **Влияние на производительность**: генерация метаданных на этапе компиляции может увеличить время компиляции (можно оптимизировать с помощью treeshake).
- **Накладные расходы во время выполнения**: включение рефлексии увеличивает расход памяти (смягчается генерацией по необходимости).

## Открытые вопросы

- [x] Область применения `^^`: только для типов и значений, не для выражений
- [x] Цепочечный доступ: поддерживается, к объекту метаданных, возвращаемому `^^T`, можно обращаться как к обычному
- [x] Сопоставление с образцом: поддерживается, `TypeMeta` — обычный record-тип, допускающий сопоставление с образцом
- [x] Сравнение: поддерживается, объекты метаданных одного типа равны
- [x] Расход памяти: генерация по необходимости + оптимизация treeshake
