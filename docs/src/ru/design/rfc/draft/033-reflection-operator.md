# RFC-033: Оператор рефлексии `^^`

> **Ссылки**:
>
> - [RFC-010: Единый синтаксис типов — модель name: type = value](../accepted/010-unified-type-syntax.md)
> - [RFC-011: Проект системы обобщённых типов](../accepted/011-generic-type-system.md)
> - [RFC-027: Предикаты времени компиляции и унифицированная статическая верификация](../accepted/027-compile-time-evaluation-types.md)
> - [RFC-011a: Реализация интерфейсов и динамическая диспетчеризация](../draft/011a-interface-implementation.md)

## Резюме

Настоящий документ предлагает ввести оператор `^^` в качестве точки входа для рефлексии, предназначенной для получения метаданных типов и значений. `^^T` возвращает объект статических метаданных типа `T`, а `^^obj` — метаданные динамического типа значения `obj`. Объекты метаданных представляют собой обычные записи (record type), содержащие имя, параметры, поля и прочую информацию, и могут использоваться как во время компиляции, так и во время выполнения.

## Мотивация

### Зачем нужна эта возможность?

1. **Сериализация/десериализация**: необходим доступ к информации о полях типа для автоматической генерации кода сериализации.
2. **Метапрограммирование во время компиляции**: требуется доступ к структуре типов на этапе компиляции для генерации кода или проверки ограничений.
3. **Отладка и инструментарий во время выполнения**: необходимо выводить информацию о типах в процессе работы программы.
4. **Проверка типов во время выполнения**: требуется возможность определять отношения между типами в рантайме, например: «Какой тип у obj?».

### Текущая проблема

В текущей версии YaoXiang отсутствует механизм рефлексии, поэтому получить метаданные типов во время компиляции или выполнения невозможно. Прямой доступ к метаданным типа через `.name`, `.fields` конфликтовал бы с пользовательскими полями:

```yaoxiang
Person: Type = { name: String, age: Int }

# Если Person.name — это имя в метаданных типа или поле name?
# Это привело бы к сложностям разбора и семантической путанице
```

Необходим синтаксис, который **не вторгается в пространство имён обычных полей** для доступа к метаданным типа.

## Предложение

### Базовая конструкция

Ввести оператор `^^` в качестве точки входа в рефлексию, чётко разделяющий обычный код и запросы метаданных.

**Две формы использования**:

1. **Статическая рефлексия (над типом)**: `^^T` возвращает объект статических метаданных типа `T`.
2. **Динамическая рефлексия (над значением)**: `^^obj` возвращает метаданные динамического типа значения `obj`.

**Структура метаданных**:

```yaoxiang
TypeMeta: Type = {
    name: String,
    params: Array(ParamMeta),
    fields: Array(FieldMeta),
    return_type: Type,
    refinement: Option(Expr)  # во время компиляции Some(Expr), во время выполнения None
}

ParamMeta: Type = {
    name: String,
    type: Type
}

FieldMeta: Type = {
    name: String,
    type: Type
}
```

**Иерархия вселенных (universes)**: если `T: Type_n`, то `^^T: Type_{n+1}`, что соответствует стандартному правилу повышения уровня в теории типов.

**Приоритет**: `^^` — унарный префиксный оператор с наивысшим приоритетом. `^^T.name` эквивалентно `(^^T).name`.

### Примеры

#### Базовое использование

```yaoxiang
Point: Type = { x: Float, y: Float }

# Статическая рефлексия
meta = ^^Point
print(meta.name)           # "Point"
print(meta.fields.len)     # 2
print(meta.fields[0].name) # "x"
print(fields[0].type)      # Float

# Динамическая рефлексия (требуется включённая рефлексия рантайма)
obj = Point(1.0, 2.0)
meta = ^^obj
print(meta.name)           # "Point"
```

#### Обобщённые типы

```yaoxiang
List: (T: Type) -> Type = { data: Array(T), length: Int }

# Рефлексия самого обобщённого типа
meta = ^^List
print(meta.name)           # "List"
print(meta.params)         # [{ name: "T", type: Type }]

# Рефлексия конкретной инстанциации типа
meta = ^^List(Int)
print(meta.name)           # "List(Int)"
print(meta.params)         # []
```

#### Функции

```yaoxiang
add: (a: Int, b: Int) -> Int = a + b

meta = ^^add
print(meta.name)           # "add"
print(meta.params)         # [{ name: "a", type: Int }, { name: "b", type: Int }]
print(meta.return_type)    # Int
```

#### Уточнённые (refinement) типы

```yaoxiang
Positive: (x: Int) -> Type = { x > 0 }

# Во время компиляции: refinement содержит Some(Expr)
meta = ^^Positive
print(meta.name)           # "Positive"
print(meta.refinement)     # Some(AST(x > 0))

# Во время выполнения: refinement содержит None (стирается)
```

#### Использование в предикатах времени компиляции

```yaoxiang
# Проверка наличия полей у типа
HasFields: (T: Type) -> Type = { ^^T.fields.len > 0 }

# Проверка типа поля
HasFloatField: (T: Type) -> Type = {
    exists field in ^^T.fields: field.type == Float
}

# Применение
obj: HasFields(Point) = Point(1.0, 2.0)  # ✅ Проверка пройдена
# obj: HasFields(Int) = 42  # ❌ Проверка не пройдена
```

#### Пример сериализации

```yaoxiang
# Чистая функция времени компиляции: генерирует JSON-строку
to_json: (T: Type) -> ((obj: T) -> String) = {
    meta = ^^T
    parts: Array(String) = []
    for field in meta.fields {
        # Генерация кода доступа к полю во время компиляции
        parts.push("\"${field.name}\": ${obj.${field.name}}")
    }
    return "{" + parts.join(", ") + "}"
}

# Применение
point_to_json = to_json(Point)
print(point_to_json(Point(1.0, 2.0)))  # '{"x": 1.0, "y": 2.0}'
```

### Синтаксические изменения

| До | После |
|------|------|
| Механизм рефлексии отсутствует | `^^T` — получение метаданных типа |
| Механизм рефлексии отсутствует | `^^obj` — получение метаданных динамического типа значения |

## Детальное проектирование

### Влияние на систему типов

- **Новые типы**: `TypeMeta`, `ParamMeta`, `FieldMeta`.
- **Иерархия вселенных**: тип, возвращаемый `^^T`, на один уровень выше, чем `T`.
- **Взаимодействие с обобщёнными типами**: поддерживается и `^^List`, и `^^List(Int)`.
- **Взаимодействие с функциями**: `^^add` возвращает метаданные функции (включая параметры и возвращаемый тип).
- **Взаимодействие с уточнёнными типами**: `^^Positive` возвращает метаданные уточнённого типа (включая уточняющее выражение).

### Поведение во время выполнения

**Рефлексия во время компиляции**:
- `^^T` полностью вычисляется на этапе компиляции, результат встраивается как константа.
- Уточняющее выражение доступно на этапе компиляции.

**Рефлексия во время выполнения**:
- По умолчанию отключена, нулевые накладные расходы.
- Включается опцией компилятора `--enable-runtime-reflection`.
- После включения `^^obj` возвращает метаданные динамического типа.
- Уточняющее выражение стирается до `None` во время выполнения.

**Генерация по запросу + treeshake**:
- Метаданные генерируются только для тех типов, для которых фактически используется `^^`.
- Для неиспользуемых типов метаданные не генерируются (treeshake).

### Изменения в компиляторе

1. **Лексический анализатор**: распознавание `^^` как единого токена.
2. **Синтаксический анализатор**: добавление правила для префиксного выражения `^^`.
3. **Система типов**: добавление определений типов `TypeMeta`, `ParamMeta`, `FieldMeta`.
4. **Проверщик типов**: генерация экземпляра метаданных для каждого типа.
5. **Вычислитель времени компиляции**: поддержка вычисления `^^T` на этапе компиляции.
6. **Среда выполнения (опционально)**: генерация RTTI для типов, к которым применяется рефлексия.

### Обратная совместимость

- ✅ На существующий синтаксис влияния нет: `^^` — новый оператор, не конфликтующий с существующим синтаксисом.
- ✅ На существующие типы влияния нет: все типы автоматически поддерживают `^^`.
- ✅ На существующие функции влияния нет: функции могут использовать `^^`, но не обязаны.
- ✅ На предикаты времени компиляции влияния нет: `^^T` в предикатах ведёт себя так же, как обычное содержимое.
- ✅ На время выполнения влияния нет: рефлексия рантайма по умолчанию отключена, нулевые накладные расходы.

## Компромиссы

### Преимущества

- **Единообразие**: единая обработка функций, обобщённых и уточнённых типов.
- **Нулевые накладные расходы**: рефлексия времени компиляции полностью стирается, рефлексия рантайма опциональна.
- **Интеграция с существующими системами**: бесшовная интеграция с предикатами времени компиляции (RFC-027).
- **Краткость**: `^^` — чистый символ, не конфликтующий с пользовательскими идентификаторами.
- **Генерация по запросу**: оптимизация treeshake обеспечивает нулевые накладные расходы для неиспользуемых типов.

### Недостатки

- **Кривая обучения**: требуется понимание семантики `^^` и структуры метаданных.
- **Накладные расходы рантайма**: включение рефлексии рантайма увеличивает расход памяти (по одному указателю на каждый экземпляр).
- **Сложность реализации**: требуется изменение нескольких компонентов компилятора.

## Альтернативы

| Вариант | Почему не выбран |
|------|--------------|
| Функция `reflect(T)` | Вводит дополнительный идентификатор в область видимости, который может быть перекрыт пользователем |
| Функция `type_info(T)` | То же самое |
| Одинарный оператор `^` | Возможен конфликт с побитовыми операциями; кроме того, в C++26 выбран `^^` именно из-за такого конфликта |
| Символы `@@`, `##` и т. п. | Нет прецедентов, сложнее объяснить, чем `^^` |

### Риски

- **Конфликт разбора**: `^^` потенциально может конфликтовать с существующим синтаксисом (анализ показал отсутствие конфликтов).
- **Влияние на производительность**: генерация метаданных во время компиляции может увеличить время компиляции (смягчается оптимизацией treeshake).
- **Накладные расходы рантайма**: включение рефлексии рантайма увеличивает расход памяти (смягчается генерацией по запросу).

## Открытые вопросы

- [x] Область применения `^^`: действует только над типами и значениями, не над произвольными выражениями.
- [x] Цепочечный доступ: поддерживается, к свойствам объекта метаданных `^^T` можно обращаться обычным образом.
- [x] Сопоставление с образцом: поддерживается, поскольку `TypeMeta` — обычный record type.
- [x] Сравнение: поддерживается, объекты метаданных одного типа считаются равными.
- [x] Расход памяти: генерация по запросу + оптимизация treeshake.
