## Жизненный цикл и судьба

RFC проходит через следующие состояния:

```
┌─────────────┐
│  черновик   │  ← создаётся автором
└──────┬──────┘
       │
       ▼
┌─────────────┐
│на рассмот-  │  ← общественное обсуждение
│   рении     │
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  принято    │    │  отклонено  │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  accepted/  │    │    rfc/     │
│ (официаль-  │    │ (остаётся   │
│  ный дизайн)│    │   на месте) │
└─────────────┘    └─────────────┘
```

### Описание статусов

| Статус | Расположение | Описание |
|------|------|------|
| **Черновик** | `docs/design/rfc/` | Черновик автора, ожидает подачи на рассмотрение |
| **На рассмотрении** | `docs/design/rfc/` | Открыто для общественного обсуждения и обратной связи |
| **Принято** | `docs/design/accepted/` | Становится официальным проектным документом, переходит в фазу реализации |
| **Отклонено** | `docs/design/rfc/` | Сохраняется в каталоге RFC, статус обновляется |

### Действия после принятия

1. Переместить RFC в каталог `docs/design/accepted/`
2. Обновить имя файла на описательное (например, `enhanced-pattern-matching.md`)
3. Обновить статус на «официальный»
4. Обновить статус на «принято», добавить дату принятия

### Действия после отклонения

1. Сохранить в каталоге `docs/design/rfc/`
2. Добавить в начало файла причину и дату отклонения
3. Обновить статус на «отклонено»

### Действия после определения результатов обсуждения

Когда по открытому вопросу достигнут консенсус:

1. **Обновить Приложение А**: заполнить «Решение» в теме обсуждения
2. **Обновить основной текст**: синхронизировать решение с основным текстом документа
3. **Зафиксировать решение**: добавить запись в «Приложение Б: Записи проектных решений»
4. **Отметить вопрос**: поставить отметку `[x]` в списке «Открытые вопросы»
