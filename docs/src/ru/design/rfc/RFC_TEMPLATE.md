## Жизненный цикл и судьба

RFC имеет следующие статусы переходов:

```
┌─────────────┐
│  Черновик   │  ← Создаётся автором
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  На review  │  ← Обсуждение сообществом
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  Принят     │    │  Отклонён   │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│   accepted/ │    │    rfc/     │
│ (финальный  │    │(остаётся на │
│   дизайн)   │    │   месте)    │
└─────────────┘    └─────────────┘
```

### Описание статусов

| Статус | Расположение | Описание |
|------|------|------|
| **Черновик** | `docs/design/rfc/` | Черновик автора, ожидает отправки на review |
| **На review** | `docs/design/rfc/` | Открыто для обсуждения и обратной связи сообщества |
| **Принят** | `docs/design/accepted/` | Становится официальным дизайн-документом, переходит в стадию реализации |
| **Отклонён** | `docs/design/rfc/` | Остаётся в директории RFC, статус обновляется |

### Действия после принятия

1. Переместить RFC в директорию `docs/design/accepted/`
2. Обновить имя файла на описательное (например, `enhanced-pattern-matching.md`)
3. Обновить статус на «Финальный»
4. Обновить статус на «Принят», добавить дату принятия

### Действия после отклонения

1. Сохранить в директории `docs/design/rfc/`
2. Добавить в начало файла причину и дату отклонения
3. Обновить статус на «Отклонён»

### Действия после определения open question

Когда консенсус по какому-либо открытому вопросу достигнут:

1. **Обновить Приложение А**: Заполнить «Решение» в теме обсуждения
2. **Обновить основную часть**: Синхронизировать решение с документом
3. **Записать решение**: Добавить в «Приложение Б: Журнал дизайнерских решений»
4. **Отметить вопрос**: Поставить галочку `[x]` в списке «Открытые вопросы»
