## Жизненный цикл и исход

RFC проходит следующие состояния:

```
┌─────────────┐
│  Черновик   │  ← Автор создаёт
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ На рассмот- │  ← Обсуждение сообществом
│   рении     │
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  Принято    │    │  Отклонено  │
└──────┬──────┘    └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│   accepted/ │    │    rfc/     │
│  (офици-    │    │ (остаётся   │
│  альный     │    │  на месте)  │
│  дизайн)    │    │             │
└─────────────┘    └─────────────┘
```

### Описание статусов

| Статус | Расположение | Описание |
|------|------|------|
| **Черновик** | `docs/design/rfc/` | Черновик автора, ожидает отправки на рассмотрение |
| **На рассмотрении** | `docs/design/rfc/` | Открыто для обсуждения сообществом и обратной связи |
| **Принято** | `docs/design/accepted/` | Становится официальным проектным документом, переходит в стадию реализации |
| **Отклонено** | `docs/design/rfc/` | Остаётся в каталоге RFC, статус обновляется |

### Действия после принятия

1. Переместить RFC в каталог `docs/design/accepted/`
2. Обновить имя файла на описательное (например, `enhanced-pattern-matching.md`)
3. Обновить статус на "Официальный"
4. Обновить статус на "Принято", добавить дату принятия

### Действия после отклонения

1. Оставить в каталоге `docs/design/rfc/`
2. Добавить причину отклонения и дату в начало файла
3. Обновить статус на "Отклонено"

### Действия после достижения консенсуса по обсуждению

Когда по какому-либо открытому вопросу достигнут консенсус:

1. **Обновить Приложение A**: Заполнить «Решение» в теме обсуждения
2. **Обновить основной текст**: Синхронизировать решение с основным текстом документа
3. **Записать решение**: Добавить в «Приложение B: Записи решений по дизайну»
4. **Отметить вопрос**: Отметить `[x]` в списке «Открытые вопросы»
