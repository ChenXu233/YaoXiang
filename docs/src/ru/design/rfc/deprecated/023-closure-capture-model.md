# RFC-023: Модель захвата замыканий

> **Ссылки**:
> - [RFC-007: Унификация синтаксиса функций](./accepted/007-function-syntax-unification.md)
> - [RFC-009: Модель владения v9](./accepted/009-ownership-model.md)
> - [RFC-011: Дизайн системы обобщений](./accepted/011-generic-type-system.md) — раздел 2.4: встроенные marker-trait'ы Dup/Clone

## Резюме

Данный RFC определяет **модель захвата замыканий** языка YaoXiang. Компилятор автоматически анализирует внешние переменные, на которые ссылается тело замыкания, и в зависимости от типа переменной (Dup/не-Dup) и того, является ли замыкание «убегающим» (escape), автоматически выбирает способ захвата — для Dup-типов выполняется прямое копирование, для не-Dup без escape — заимствование, для не-Dup с escape — Move. Пользователь не указывает аннотации; правила едины с правилами автоматического выбора заимствования при вызове функций.

## Мотивация

### Зачем это нужно?

В настоящее время захват замыканий — это **пустая реализация**: поле `env` инструкции `MakeClosure` всегда пустое, и lambda не может ссылаться ни на какие внешние переменные. Система токенов заимствования требует, чтобы замыкание могло захватывать токены `&T` (копирование с нулевой стоимостью) — это один из ключевых сценариев использования.

### Текущая проблема

```yaoxiang
# Этот код в данный момент не компилируется — lambda не может захватить threshold
filter_by: (items: List(Point), threshold: &Float) -> List(Point) = {
    items.filter(|p| p.x > threshold)  # ❌ threshold не может быть захвачен
}
```

## Предложение

### Основной дизайн

Захват замыкания определяется компилятором полностью автоматически. Правила **полностью совпадают** с правилами автоматического выбора заимствования при вызове функций:

```
Тип переменной    Убегание замыкания    Способ захвата
──────────────────────────────────────────────────────────
Dup               Любое                 Копирование (побитовое или с нулевой стоимостью)
Не Dup            Без escape            Автоматическое заимствование (&T или &mut T)
Не Dup            С escape              Move (передача владения)
```

**Определение escape**:

```
spawn { || ... }           → escape
return || ...              → escape
let x = || ... ;  x сохранён в поле → escape
items.filter(|p| ...)      → без escape (синхронный вызов функции высшего порядка)
||.method()                → без escape (немедленный вызов)
```

Принцип консервативности: если невозможно определить — обрабатывать как escape.

### Примеры

```yaoxiang
# 1. Dup-токен — прямое копирование (нулевая стоимость)
filter_by: (items: List(Point), threshold: &Float) -> List(Point) = {
    # threshold: &Float → Dup → компилятор копирует токен в замыкание
    # Токен нулевого размера, нулевые накладные расходы в рантайме
    items.filter(|p| p.x > threshold)
}

# 2. Не-Dup + без escape — автоматическое заимствование
process: (buf: Buffer) -> Void = {
    # buf не Dup, filter без escape → автоматически создаётся токен &Buffer
    transform(|b| b.read())
    # После возврата замыкания токен освобождается, buf снова доступен
}

# 3. Замыкание с escape — Move
spawn_worker: (data: Data) -> Void = {
    # data не Dup, spawn → escape → Move
    spawn { use(data) }
}

# 4. Смешанный захват
complex: (items: List(Point), config: &Config, buf: Buffer) -> List(Point) = {
    # config: &Config → Dup → копирование токена
    # buf: Buffer → не Dup, без escape → заимствование &mut Buffer
    items.filter(|p| {
        let threshold = config.get_threshold()
        buf.update(p)
        p.x > threshold
    })
}

# 5. Обнаружение конфликта заимствований
bad: (buf: Buffer) -> Void = {
    closure = |b| b.write()
    buf.read()  # ❌ buf уже заимствован замыканием, здесь конфликт
}
```

### Изменения в синтаксисе

**Никаких изменений в синтаксисе**. Способ захвата определяется компилятором автоматически, пользователю не нужны аннотации.

## Детальный дизайн

### Влияние на систему типов

Сигнатура типа lambda остаётся без изменений: `(params) -> Return`. Захваченные переменные не отражаются в сигнатуре типа; они обрабатываются компилятором на этапе генерации IR.

### Изменения в компиляторе

| Компонент | Изменения | Описание |
|------|------|------|
| `capture.rs` (новый) | Анализ захвата + escape-анализ + выбор режима | ~150 строк |
| `expressions.rs` | Вывод типа lambda вызывает анализ захвата | ~10 строк |
| `ir_gen.rs` | Заполнение env в MakeClosure; пропуск ZST | ~80 строк |
| `ir.rs` | Возможно, потребуется корректировка типа env в MakeClosure | ~5 строк |

**Процесс анализа захвата**:

```
1. Обход AST тела lambda
2. Сбор всех ссылок Expr::Var(name)
3. Фильтрация: только переменные из внешней области замыкания
4. Классификация: Read (только чтение) / Write (чтение-запись) / Move (передаются)
5. Проверка свойств типа: является ли Dup
6. Определение escape: способ использования замыкания
7. Выбор режима захвата:
   Dup → Copy
   Не-Dup + без escape + Read → Borrow (&T)
   Не-Dup + без escape + Write → BorrowMut (&mut T)
   Не-Dup + escape → Move
```

**Генерация IR**:

```rust
// Текущее (пустое)
Instruction::MakeClosure { dst, func, env: Vec::new() }

// Изменено на
Instruction::MakeClosure { dst, func, env: captured_env }

// Логика генерации captured_env:
for captured in captures {
    match captured.mode {
        Copy if is_zst(captured.ty) => {
            // Тип нулевого размера — никакие инструкции не генерируются
            // Тело замыкания напрямую ссылается на внешний контекст (устраняется в compile-time)
        }
        Copy => {
            // Генерация Move dst, src (поверхностное копирование Dup-типа)
        }
        Borrow => {
            // Генерация Borrow dst, src (создание ReadToken)
        }
        BorrowMut => {
            // Генерация Borrow dst, src (создание WriteToken)
        }
        Move => {
            // Генерация Move dst, src (передача владения)
        }
    }
}
```

### Поведение в рантайме

Способ захвата не влияет на производительность в рантайме:

- **Dup + ZST** (например, токены `&T`) → нулевые инструкции, тело замыкания напрямую ссылается на внешнюю переменную
- **Dup + не ZST** (например, Int) → одно копирование в регистре
- **Borrow/BorrowMut** → создание токена (концепция compile-time, нулевые накладные расходы)
- **Move** → та же стоимость, что и обычный Move

### Обратная совместимость

Полностью совместимо. В данный момент ни одна lambda не может захватывать внешние переменные; данный RFC только расширяет выразительные возможности и не нарушает работу существующего кода.

## Компромиссы

### Преимущества

1. **Нулевые аннотации**: пользователю не нужно указывать какие-либо аннотации захвата
2. **Единообразие с вызовом функций**: правила захвата = правила автоматического заимствования при вызове функций
3. **Нулевая стоимость**: захват Dup-токенов полностью устраняется в compile-time
4. **Безопасность**: escape-анализ предотвращает use-after-free

### Недостатки

1. **Консервативность escape-анализа**: при невозможности определения обрабатывается как escape, что может приводить к ненужным Move
2. **Неявность**: способ захвата не отражён в коде; при отладке нужно смотреть вывод компилятора

## Альтернативы

| Вариант | Почему не выбран |
|------|--------------|
| Явное ключевое слово `move` в стиле Rust | Вводит новый синтаксис, увеличивает когнитивную нагрузку на пользователя |
| Всегда Move | Невозможно выразить заимствование токенов с нулевой стоимостью |
| Всегда заимствование | Escape замыкания приведёт к висячим ссылкам |
| Ручная аннотация способа захвата | Противоречит философии дизайна «полная автоматизация компилятора» |

## Стратегия реализации

### Этапы

1. **Фаза 1**: Анализ захвата (только распознавание ссылок на внешние переменные, без различия способов захвата)
2. **Фаза 2**: Escape-анализ + выбор режима
3. **Фаза 3**: Генерация IR + оптимизация ZST
4. **Фаза 4**: Интеграция обнаружения конфликтов заимствования

### Зависимости

- Зависит от RFC-011 (система обобщений, раздел 2.4 trait Dup/Clone) — требуется trait Dup для определения возможности копирования переменной
- Зависит от RFC-009 v9 (токены заимствования) — режимы захвата Borrow/BorrowMut требуют типов токенов
- После реализации RFC-023 и данного RFC можно приступать к реализации системы токенов заимствования (реализация RFC-009 v9)

### Риски

- Escape-анализ может быть слишком консервативным, приводя к ненужным Move; можно оптимизировать впоследствии
- Анализ захвата для обобщённых замыканий может потребовать дополнительной обработки
