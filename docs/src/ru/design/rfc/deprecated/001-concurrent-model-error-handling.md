## Аннотация

Предлагается модель 并作 для YaoXiang: синхронный синтаксис для описания логики с автоматическим параллельным выполнением во время 运行时间. Ключевые механизмы: трёхуровневая архитектура параллелизма + DAG-анализ зависимостей + система типов Result.

## Быстрая справка

| Сценарий | Способ написания | Описание |
|------|------|------|
| Автоматический параллелизм | Без аннотаций (по умолчанию) | Максимальный параллелизм |
| Синхронное ожидание | `@eager` | Ожидание завершения зависимостей |
| Полностью последовательно | `@block` | Без параллелизма, для отладки |
| Локальный параллелизм | `spawn` | Параллелизм внутри области видимости @block |

## Мотивация

Текущие модели параллелизма в популярных языках имеют явные недостатки:

| Язык | Модель параллелизма | Проблема |
|------|----------|------|
| Rust | async/await + tokio | Распространение асинхронности, крутая кривая обучения |
| Go | goroutine | Нет безопасности типов |
| Python | asyncio | Ограничение GIL |
| JavaScript | Promise/async | Сложные колбэки |

### Ключевое противоречие

1. **Прозрачность vs Управляемость**: полная прозрачность без контроля vs полный контроль без прозрачности
2. **Параллелизм vs Отладка**: параллельные программы сложно отлаживать vs отлаживаемые программы сложно распараллелить

## Предложение

### 1. Модель 并作: трёхуровневая архитектура параллелизма

> **Примечание**: L1/L2/L3 — это ментальная модель, помогающая пользователям понять различные сценарии. Реальная реализация представляет собой единый механизм: автоматический DAG-анализ + контроль через аннотации.

| Уровень | Ментальная модель | Синтаксис | Способ выполнения | Степень параллелизма |
|------|----------|------|----------|--------|
| **L1** | Запрет параллелизма | `@block` | Чисто последовательное выполнение | ❌ Нет |
| **L2** | 并作 внутри @block | `spawn` | Управляемый параллелизм в области видимости @block | ⚠️ Частичный |
| **L3** | Полный параллелизм | По умолчанию (без аннотаций) | Автоматический анализ DAG | ✅ Полный |

#### L1: Синхронный режим @block

```yaoxiang
main: () -> Void @block = {
    data1 = fetch_sync("api1")
    data2 = fetch_sync("api2")
    process(data1, data2)    # Строго последовательно, без параллелизма
}
```

#### L2: Управляемый параллелизм внутри @block

```yaoxiang
# spawn можно использовать только внутри функций @block
main: () -> Void @block = {
    spawn { data1 = fetch_data("api1") }
    spawn { data2 = fetch_data("api2") }
    # Ожидание завершения всех spawn (контроль стандартной библиотеки)
    process(data1, data2)
}
```

#### L3: Полная прозрачность (по умолчанию)

```yaoxiang
# Не нужны никакие аннотации, компилятор автоматически анализирует DAG
heavy_calc: (n: Int) -> Int = fibonacci(n)

auto_parallel: (n: Int) -> Int = {
    a = heavy_calc(1)    # Автоматический параллелизм
    b = heavy_calc(2)    # Автоматический параллелизм
    c = heavy_calc(3)    # Автоматический параллелизм
    a + b + c            # При запросе значения ожидаем все результаты
}
```

### 2. Полное сравнение аннотаций

| Измерение | По умолчанию (без аннотаций) | `@eager` | `@block` | `spawn` |
|------|---------------|----------|----------|---------|
| **Способ выполнения** | Автоматический DAG-анализ | Синхронное ожидание зависимостей | Чисто последовательно | 并作 внутри @block |
| **Степень параллелизма** | ✅ Полный | ⚠️ В порядке зависимостей | ❌ Нет | ⚠️ Частичный |
| **Построение DAG** | ✅ | ✅ | ❌ | ✅ |

**Руководство по выбору**:
- Максимальный параллелизм → Без аннотаций (по умолчанию)
- Нужны упорядоченные побочные эффекты → `@eager`
- Отладка/новички/критический код → `@block`
- Нужен параллелизм внутри @block → `spawn`

```yaoxiang
# По умолчанию: максимальный параллелизм
calc_all: () -> Int = {
    a = heavy_calc(1)    # Автоматический параллелизм
    b = heavy_calc(2)    # Автоматический параллелизм
    a + b
}

# @eager: синхронное ожидание
calc_seq: () -> Int @eager = {
    a = heavy_calc(1)    # Синхронное выполнение
    b = heavy_calc(2)    # Синхронное выполнение
    a + b
}

# @block: чисто последовательно
calc_simple: () -> Int @block = {
    a = heavy_calc(1)    # Принудительно синхронно
    b = heavy_calc(2)    # Синхронно
    a + b
}

# spawn: 并作 внутри @block
calc_mixed: () -> Int @block = {
    spawn { heavy_calc(1) }
    spawn { heavy_calc(2) }
    heavy_calc(3)        # Синхронно
}
```

### 3. Анализ DAG-зависимостей

#### 3.1 Ключевой принцип: выполнение снизу вверх

```
Код пользователя (синхронный синтаксис):
    a = fetch(url0)
    b = fetch(url1)
    print(a)

Анализ при компиляции (снизу вверх):
    print(a) требует a → зависит от fetch(url0)
    fetch(url1) никому не нужен → изолированный DAG

Планирование во время 运行时间 (от листьев):
    fetch(url0) → print(a)    ← цепочка зависимостей, по порядку
    fetch(url1)                ← изолированный, параллельно
```

**Ключевая интуиция**: не "сверху вниз" генерировать Future, а "снизу вверх" анализировать зависимости от результата.

#### 3.2 Изолированный DAG: независимый параллелизм

```
Основной поток: fetch(url0) → process → print
Изолированный:  fetch(url1)  ← результат никому не нужен, параллельно

Планировщик: основной поток выполняется по цепочке зависимостей, изолированный — на другом ядре параллельно
```

#### 3.3 Типы ресурсов и побочные эффекты

**Ключевая идея**: операции с ресурсами помечаются через типы, DAG автоматически строится. Операции с одним ресурсом автоматически сериализуются, с разными ресурсами — автоматически параллелятся.

**Границы типов ресурсов — чёткое определение**:

Resource type — это встроенные в компилятор типы-маркеры. Следующие типы распознаются компилятором как ресурсы:

| Тип ресурса | Описание | Поведение компилятора |
|----------|------|-----------|
| `FilePath` | Путь в файловой системе | Операции с одним путём автоматически сериализуются |
| `HttpUrl` | HTTP endpoint | Операции с одним URL автоматически сериализуются |
| `DBUrl` | Подключение к БД | Операции с одним подключением автоматически сериализуются |
| `Console` | Стандартный вывод | Все операции Console автоматически сериализуются |

Пользовательские типы ресурсов требуют явного маркирования:
```yaoxiang
Database: Resource              # Явное маркирование как тип ресурса
query: (Database, String) -> Result(Row, Error)
# Параметр Database — это Resource, автоматически распознаётся как операция с ресурсом
```

Типы без маркера Resource не отслеживаются компилятором для зависимостей по ресурсам.

**Правила использования**:
- Передавайте дескрипторы ресурсов через переменные, DAG автоматически управляет порядком
- Использование 字面量 для одного ресурса — проблема дизайна пользователя, не ответственность языка

```yaoxiang
# ✅ Правильно: передача через переменную, DAG автоматически сериализует
filename: String = "data.txt"
File.write(filename, x)
File.write(filename, y)    # DAG сериализует

# ⚠️ Ответственность пользователя: 字面量
File.write("data.txt", x)
File.write("data.txt", y)  # Возможен параллелизм, пользователь сам отвечает
```

#### 3.4 Обработка бесконечных циклов

```
1 цикл → Прямое синхронное выполнение, нулевые накладные расходы на планирование
Несколько циклов → Планировщик разбивает на срезы и переключает, настоящий параллелизм
```

### 4. Тип Result и обработка ошибок

```yaoxiang
Result: (T: Type, E: Type) -> Type = { ok: (T) -> Self, err: (E) -> Self }

# Оператор ? прозрачно распространяет ошибки
process: () -> Result(Data, Error) = {
    data = fetch_data()?
    processed = transform(data)?
    save(processed)?
}
```

### 5. Дизайн узлов DAG

```rust
enum NodeKind {
    Task,      // Узел задачи
    Value,     // Узел значения
    Control,   // Узел потока управления
}

struct Node {
    id: NodeId,
    kind: NodeKind,
    inputs: Vec<ValueNodeId>,   // Входные зависимости
    outputs: Vec<ValueNodeId>,  // Выходные значения
    span: Span,                 // Позиция в исходном коде
}
```

| Тип ребра | Символ | Семантика |
|--------|------|------|
| DataEdge | → | Зависимость по данным (поток значений) |
| ControlEdge | ● | Зависимость по управлению (последовательное выполнение) |
| SpawnEdge | ◎ | 并作-вход (точка возможного параллелизма) |

### 6. Система типов

```
Send → Безопасная передача между потоками
Sync → Безопасное совместное использование между потоками
Arc(T) реализует Send + Sync (потокобезопасный подсчёт ссылок)
```

## Компромиссы

### Преимущества

1. **Постепенное внедрение**: трёхуровневая модель адаптируется к разному уровню навыков
2. **Естественный синтаксис**: синхронный код получает преимущества параллелизма
3. **Компиляционная безопасность**: ограничения Send/Sync устраняют гонки данных
4. **Отладка**: граф ошибок предоставляет чёткое представление о распространении ошибок

### Недостатки

1. **Кривая обучения**: необходимо понимать концепцию DAG-зависимостей
2. **Время компиляции**: полный DAG-анализ всей программы может быть медленным
3. **Сложность инструментария**: требуются принципиально новые инструменты отладки и визуализации

## Альтернативные решения

| Решение | Почему не выбрано |
|------|--------------|
| Только явный async/await | Невозможно реализовать прозрачный параллелизм |
| Только полностью прозрачный параллелизм | Пользователь теряет контроль |
| Goroutine в стиле Go | Нет безопасности типов, невозможна проверка при компиляции |
| Только режим L1 | Отказ от ключевой ценности модели 并作 |

## Стратегия реализации

### Фазы

1. **Фаза 1 (v0.1)**: Синхронный режим @block, базовые типы
2. **Фаза 2 (v0.2)**: Планировщик FlowScheduler
3. **Фаза 3 (v0.3)**: Блоки spawn, явный параллелизм
4. **Фаза 4 (v0.5)**: L3 полная прозрачность, автоматический DAG-анализ
5. **Фаза 5 (v0.6)**: Граф ошибок, графический отладчик
6. **Фаза 6 (v1.0)**: Оптимизация для производственного использования

### Зависимости

- RFC-001 без внешних зависимостей (базовое ядро)
- RFC-008 (Runtime-модель параллелизма) → Дизайн завершён
- RFC-011 (Обобщённая система) → Дизайн завершён

### Риски

1. **Производительность DAG-анализа**: полный анализ программы может быть O(n²), требуется оптимизация
2. **Отсутствие инструментария**: отладчик необходимо разрабатывать с нуля
3. **Принятие пользователями**: прозрачный параллелизм требует качественной документации

## Приложение: Глоссарий терминов

| Термин | Определение |
|------|------|
| 并作 модель | Парадигма параллелизма YaoXiang: синхронный синтаксис, асинхронная суть |
| DAG | Направленный ациклический граф, описывающий зависимости вычислений |
