# RFC-001: 并作 模型 и система обработки ошибок

## Источники дизайна

| Документ | Связь |
|----------|-------|
| [async-whitepaper](/src/archive/async-whitepaper) | Источник дизайна, теоретическая база |
| [language-spec](/src/design/language-spec) | Цель спецификации |

## Аннотация

Предлагается модель 并作 для YaoXiang: синхронный синтаксис для описания логики с автоматическим параллельным выполнением во время исполнения. Ключевой механизм: трёхуровневая архитектура параллелизма + DAG-анализ зависимостей + система типов Result.

## Быстрый выбор

| Сценарий | Запись | Описание |
|----------|--------|----------|
| Автоматический параллелизм | Без аннотаций (по умолчанию) | Максимальный параллелизм |
| Синхронное ожидание | `@eager` | Ожидание завершения зависимостей |
| Полностью последовательно | `@block` | Без параллелизма, для отладки |
| Частичный параллелизм | `spawn` | Параллелизм внутри области видимости @block |

## Мотивация

Текущие модели параллелизма в популярных языках имеют очевидные недостатки:

| Язык | Модель параллелизма | Проблема |
|------|---------------------|----------|
| Rust | async/await + tokio | Распространение асинхронности, крутая кривая обучения |
| Go | goroutine | Нет типобезопасности |
| Python | asyncio | Ограничения GIL |
| JavaScript | Promise/async | Сложные коллбэки |

### Ключевое противоречие

1. **Прозрачность vs управляемость**: полная прозрачность без контроля vs полный контроль без прозрачности
2. **Параллелизм vs отладка**: параллельные программы сложно отлаживать vs отлаживаемые программы сложно распараллелить

## Предложение

### 1. Модель 并作: трёхуровневая архитектура параллелизма

> **Примечание**: L1/L2/L3 — это ментальная модель, помогающая пользователям понять различные сценарии. Реализация представляет собой единый механизм: автоматический DAG-анализ + контроль через аннотации.

| Уровень | Ментальная модель | Синтаксис | Способ выполнения | Степень параллелизма |
|---------|-------------------|-----------|-------------------|----------------------|
| **L1** | Запрет параллелизма | `@block` | Чисто последовательное выполнение | ❌ Нет |
| **L2** | Параллелизм внутри @block | `spawn` | Управляемый параллелизм в области видимости @block | ⚠️ Частичный |
| **L3** | Полный параллелизм | По умолчанию (без аннотаций) | Автоматический DAG-анализ | ✅ Полный |

#### L1: Синхронный режим @block

```yaoxiang
main: () -> Void @block = {
    data1 = fetch_sync("api1")
    data2 = fetch_sync("api2")
    process(data1, data2)    # Строго последовательно, без параллелизма
}
```

#### L2: Управляемый параллелизм внутри @block

```yaoxiang
# spawn можно использовать только внутри функций @block
main: () -> Void @block = {
    spawn { data1 = fetch_data("api1") }
    spawn { data2 = fetch_data("api2") }
    # Ожидание завершения всех spawn (контроль стандартной библиотеки)
    process(data1, data2)
}
```

#### L3: Полная прозрачность (по умолчанию)

```yaoxiang
# Не требуется никаких аннотаций, компилятор автоматически анализирует DAG
heavy_calc: (n: Int) -> Int = fibonacci(n)

auto_parallel: (n: Int) -> Int = {
    a = heavy_calc(1)    # Автоматически параллельно
    b = heavy_calc(2)    # Автоматически параллельно
    c = heavy_calc(3)    # Автоматически параллельно
    a + b + c            # При запросе значения ожидаем всех результатов
}
```

### 2. Полное сравнение аннотаций

| Измерение | По умолчанию (без аннотаций) | `@eager` | `@block` | `spawn` |
|-----------|------------------------------|----------|----------|---------|
| **Способ выполнения** | Автоматический DAG-анализ | Синхронное ожидание зависимостей | Чисто последовательно | Параллелизм внутри @block |
| **Степень параллелизма** | ✅ Полный | ⚠️ По порядку зависимостей | ❌ Нет | ⚠️ Частичный |
| **Построение DAG** | ✅ | ✅ | ❌ | ✅ |

**Руководство по выбору**:
- Максимальный параллелизм → Без аннотаций (по умолчанию)
- Требуется упорядоченность побочных эффектов → `@eager`
- Отладка/новички/критический код → `@block`
- Нужен параллелизм внутри @block → `spawn`

```yaoxiang
# По умолчанию: максимальный параллелизм
calc_all: () -> Int = {
    a = heavy_calc(1)    # Автоматически параллельно
    b = heavy_calc(2)    # Автоматически параллельно
    a + b
}

# @eager: синхронное ожидание
calc_seq: () -> Int @eager = {
    a = heavy_calc(1)    # Синхронное выполнение
    b = heavy_calc(2)    # Синхронное выполнение
    a + b
}

# @block: чисто последовательно
calc_simple: () -> Int @block = {
    a = heavy_calc(1)    # Принудительно синхронно
    b = heavy_calc(2)    # Синхронно
    a + b
}

# spawn: параллелизм внутри @block
calc_mixed: () -> Int @block = {
    spawn { heavy_calc(1) }
    spawn { heavy_calc(2) }
    heavy_calc(3)        # Синхронно
}
```

### 3. Анализ DAG-зависимостей

#### 3.1 Ключевой принцип: выполнение снизу вверх

```
Пользовательский код (синхронный синтаксис):
    a = fetch(url0)
    b = fetch(url1)
    print(a)

Анализ при компиляции (снизу вверх):
    print(a) требует a → зависит от fetch(url0)
    fetch(url1) никому не нужен → изолированный DAG

Планирование во время выполнения (от листьев):
    fetch(url0) → print(a)    ← цепочка зависимостей, по порядку
    fetch(url1)                ← изолированный, параллельно
```

**Ключевая интуиция**: не "сверху вниз" генерировать Future, а "снизу вверх" анализировать зависимости от результата.

#### 3.2 Изолированный DAG: независимый параллелизм

```
Основной поток: fetch(url0) → process → print
Изолированный:  fetch(url1)  ← результат никому не нужен, параллельно

Планировщик: основной поток выполняется по цепочке зависимостей, изолированный использует другой ресурс параллельно
```

#### 3.3 Типы ресурсов и побочные эффекты

**Ключевая идея**: операции с ресурсами помечаются через типы, DAG строится автоматически. Операции с одним ресурсом автоматически сериализуются, с разными ресурсами — автоматически распараллеливаются.

**Границы типов ресурсов — чёткое определение**:

Resource type — это встроенные в компилятор типы-маркеры. Следующие типы распознаются компилятором как ресурсы:

| Тип ресурса | Описание | Поведение компилятора |
|-------------|----------|----------------------|
| `FilePath` | Путь в файловой системе | Операции с тем же путём автоматически сериализуются |
| `HttpUrl` | HTTP- endpoint | Операции с тем же URL автоматически сериализуются |
| `DBUrl` | Подключение к БД | Операции с тем же подключением автоматически сериализуются |
| `Console` | Стандартный вывод | Все операции с Console автоматически сериализуются |

Пользовательские типы ресурсов требуют явного маркирования:
```yaoxiang
Database: Resource              # Явное маркирование как тип ресурса
query: (Database, String) -> Result(Row, Error)
# Параметр Database — Resource, автоматически распознаётся как операция с ресурсом
```

Типы без маркера Resource не отслеживаются компилятором для зависимостей по ресурсам.

**Правила использования**:
- Передавайте дескрипторы ресурсов через переменные, DAG автоматически управляет порядком
- Использование литералов для одного ресурса — проблема дизайна пользователя, не ответственность языка

```yaoxiang
# ✅ Правильно: передача через переменную, DAG автоматически сериализует
filename: String = "data.txt"
File.write(filename, x)
File.write(filename, y)    # DAG сериализует

# ⚠️ Ответственность пользователя: литералы
File.write("data.txt", x)
File.write("data.txt", y)  # Возможно параллельно, пользователь сам отвечает
```

#### 3.4 Обработка бесконечных циклов

```
1 цикл → Прямое синхронное выполнение, нулевые накладные расходы планировщика
Несколько циклов → Планировщик разбивает на срезы и переключается, настоящий параллелизм
```

### 4. Result тип и обработка ошибок

```yaoxiang
Result: (T: Type, E: Type) -> Type = { ok: (T) -> Self, err: (E) -> Self }

# Оператор ? прозрачно распространяет ошибки
process: () -> Result(Data, Error) = {
    data = fetch_data()?
    processed = transform(data)?
    save(processed)?
}
```

### 5. Дизайн узлов DAG

```rust
enum NodeKind {
    Task,      // Узел задачи
    Value,     // Узел значения
    Control,   // Узел управления потоком
}

struct Node {
    id: NodeId,
    kind: NodeKind,
    inputs: Vec<ValueNodeId>,   // Входные зависимости
    outputs: Vec<ValueNodeId>,  // Выходные значения
    span: Span,                 // Позиция в исходном коде
}
```

| Тип ребра | Обозначение | Семантика |
|-----------|-------------|-----------|
| DataEdge | → | Зависимость по данным (поток значений) |
| ControlEdge | ● | Зависимость по управлению (последовательное выполнение) |
| SpawnEdge | ◎ | Точка входа параллелизма (возможная точка параллелизма) |

### 6. Система типов

```
Send → Можно безопасно передавать между потоками
Sync → Можно безопасно разделять между потоками
Arc(T) реализует Send + Sync (потокобезопасный подсчёт ссылок)
```

## Компромиссы

### Преимущества

1. **Постепенное принятие**: трёхуровневая модель адаптируется к разному уровню навыков
2. **Естественный синтаксис**: синхронный код получает преимущества параллелизма
3. **Компиляционная безопасность**: ограничения Send/Sync устраняют гонки данных
4. **Отлаживаемость**: граф ошибок обеспечивает чёткое представление распространения ошибок

### Недостатки

1. **Кривая обучения**: требуется понимание концепции DAG-зависимостей
2. **Время компиляции**: полный DAG-анализ программы может быть медленным
3. **Сложность инструментария**: требуются принципиально новые инструменты отладки и визуализации

## Альтернативные решения

| Решение | Почему не выбрано |
|---------|-------------------|
| Только явный async/await | Невозможно реализовать прозрачный параллелизм |
| Только полностью прозрачный параллелизм | Пользователь теряет контроль |
| Go-style goroutine | Нет типобезопасности, невозможна компиляционная проверка |
| Только режим L1 | Потеря ключевой ценности модели 并作 |

## Стратегия реализации

### Разделение на этапы

1. **Этап 1 (v0.1)**: Синхронный режим @block, базовые типы
2. **Этап 2 (v0.2)**: Планировщик FlowScheduler
3. **Этап 3 (v0.3)**: Блоки spawn, явный параллелизм
4. **Этап 4 (v0.5)**: L3 полная прозрачность, автоматический DAG-анализ
5. **Этап 5 (v0.6)**: Граф ошибок, графический отладчик
6. **Этап 6 (v1.0)**: Производственная оптимизация

### Зависимости

- RFC-001 не имеет внешних зависимостей (базовое ядро)
- RFC-008 (Runtime модель параллелизма) → Дизайн завершён
- RFC-011 (Обобщённая система) → Дизайн завершён

### Риски

1. **Производительность DAG-анализа**: полный анализ программы может быть O(n²), требуется оптимизация
2. **Отсутствие инструментария**: отладчик нужно разрабатывать с нуля
3. **Принятие пользователями**: прозрачный параллелизм требует качественной документации

## Приложение: Глоссарий

| Термин | Определение |
|--------|-------------|
| 并作 模型 | Парадигма параллелизма YaoXiang: синхронный синтаксис, асинхронная суть |
| DAG | Направленный ациклический граф, описывающий зависимости вычислений |
| spawn | 并作 в области видимости @block с контролируемым параллелизмом |
| @block | Синхронная аннотация, отключает оптимизацию параллелизма |
| @eager | Срочное вычисление, ожидание завершения зависимостей |
| Resource | Маркер типа ресурса, операции автоматически строят DAG-зависимости |
| Граф ошибок | Визуализированный путь распространения ошибок |
