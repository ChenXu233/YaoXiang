## 一、为何 создан YaoXiang？

### 1.1 Восполняемый пробел в языках

В долгой истории языков программирования мы стали свидетелями рождения и эволюции множества прекрасных языков: C принёс революцию эффективности системного программирования, Python создал доступный всем опыт обучения программированию, Rust доказал, что безопасность памяти и производительность совместимы, TypeScript сделал большие фронтенд-проекты поддерживаемыми. Однако, рассматривая современную экосистему языков, мы обнаруживаем очевидный пробел — **ни один язык не способен одновременно удовлетворить три ключевых требования**:

| Требование | Проблемы существующих решений |
|------------|-------------------------------|
| **Безопасность типов** | Rust слишком строг, крутая кривая обучения; TypeScript с опциональными типами не может обеспечить compile-time гарантии |
| **Естественный синтаксис** | Синтаксис Rust сложен и запутан; Haskell слишком высокий порог входа в функциональное программирование; традиционные статические языки многословны |
| **Дружелюбие к ИИ** | Существующие языки имеют синтаксические неоднозначности, сложный AST, непредсказуемое скрытое поведение, что ограничивает точность генерации и модификации кода ИИ |

Создание YaoXiang призвано заполнить этот пробел. Мы верим: **язык программирования должен быть одновременно мощным и доступным, безопасным и эффективным, строгим и элегантным**.

### 1.2 Решаемые практические проблемы

**Проблема первая: фрагментация type system**

Современные языки программирования демонстрируют серьёзную фрагментацию в системах типов. Статически типизированные языки стремятся к абсолютной правильности во время компиляции, но часто за счёт снижения эффективности разработки; динамически типизированные языки предоставляют гибкость, но выявляют трудноустранимые дефекты в больших проектах. YaoXiang предлагает единую абстракцию «всё есть тип», делая типы сквозной темой дизайна языка, а не запоздалым дополнением.

**Проблема вторая: выбор между безопасностью памяти и производительностью**

Долгое время разработчикам приходилось делать сложный выбор между безопасностью памяти и производительностью выполнения. GC (сборка мусора) освобождает разработчика, но приносит задержки и накладные расходы памяти; ручное управление памятью эффективно, но опасно как хождение по канату. YaoXiang использует модель ownership в стиле Rust, устраняя гонки данных и утечки памяти на этапе компиляции, сохраняя zero-cost abstraction без необходимости GC для высокой производительности.

**Проблема третья: когнитивная нагрузка асинхронного программирования**

Современные приложения невозможно представить без сетей и параллелизма, а асинхронное программирование всегда было кошмаром для программистов. Вложенные callback-функции, цепочки Promise, синтаксис async/await — каждый подход увеличивал сложность кода. YaoXiang переработал асинхронную модель: достаточно добавить маркер `spawn` после сигнатуры функции, и компилятор автоматически обрабатывает все асинхронные детали, делая concurrent programming таким же естественным, как синхронный код.

**Проблема четвёртая: узкое место AI-assisted программирования**

Когда ИИ начинает помогать разработчикам писать код, выбор дизайна языка становится критически важным. Нечёткие правила синтаксиса, неявные преобразования типов, сложные синтаксические конструкции — эти привычные для программистов особенности становятся препятствием для понимания и генерации ИИ. YaoXiang с самого начала сделал «дружелюбие к ИИ» одной из ключевых целей: строгие правила отступов, чёткие границы блоков кода, однозначный синтаксис позволяют ИИ точно понимать, генерировать и модифицировать код.

### 1.3 Философские корни языка

Имя YaoXiang происходит от «Яо» и «Сян» в «И Цзин». «Яо» — базовый символ, составляющий гексаграммы, олицетворяющий изменения инь и ян, движение и покой; «Сян» — внешнее проявление сущности вещей, представляющее все явления и всё сущее.

Эта философия отражена в каждой детали дизайна языка:

- **Единство**: Как простые символы яо составляют сложные гексаграммы, YaoXiang использует несколько ключевых понятий (типы, функции, конструкторы) для построения полной программной модели
- **Иерархичность**: Как сян делится на врождённый и приобретённый, type system YaoXiang имеет чёткую иерархическую структуру — от primitive type до generics, от значений до meta type
- **Изменчивость**: Как инь и ян чередуются бесконечно, YaoXiang поддерживает зависимые типы, позволяя типам эволюционировать вместе со значениями
- **Распознаваемость**: Как гексаграммы поддаются толкованию, YaoXiang обеспечивает полную возможность рефлексии типов, полная информация о типах доступна в runtime
- **Доказуемость**: Как гексаграммы раскрывают законы вещей, type system YaoXiang следует изоморфизму Карри-Ховарда (типы — это утверждения, программы — доказательства), процесс проверки типов — это верификация логического доказательства

## 二、Ключевая философия и принципы

Следующие проектные установки — фундамент YaoXiang, **бескомпромиссный и нерушимый**. Каждое предложение特性必须经过这些原则的检验。

### 2.1 Принцип первый: Всё есть тип

В мировоззрении YaoXiang тип — высший уровень абстракции, сквозная концепция языка.

**Конкретное воплощение**:

- **Значения — экземпляры типов**: `42` — экземпляр типа `Int`, `"hello"` — экземпляр типа `String`
- **Типы сами являются типами**: meta type (`type`) — тип всех типов, `Int` — экземпляр `type`
- **Функции — это type mapping**: `fn add(Int, Int) -> Int` описывает type mapping из `Int × Int` в `Int`
- **Модули — это комбинации типов**: модуль — это именованное пространство имён, содержащее функции и типы

**Бескомпромиссная причина**: Единая абстракция типа упрощает семантику языка, устраняет дуализм значений и типов, делает type system охранником правильности кода, а не препятствием.

### 2.2 Принцип второй: Строгая структурированность

Синтаксический дизайн YaoXiang преследует цели «однозначность, предсказуемость, лёгкость解析».

**Конкретные правила**:

- **Обязательные 4 пробела отступа**: использование Tab запрещено, границы блоков кода一目了然
- **Скобки нельзя опускать**: параметры функций должны быть в скобках, элементы списка должны быть разделены запятыми
- **Блоки кода должны быть в фигурных скобках**: управление потоком (`if`, `while`, `for` и т.д.) должно использовать `{ }`
- **Минимальное количество ключевых слов**: только 17 основных ключевых слов, отказ от синтаксического сахара

**Бескомпромиссная причина**: Строгая структурированность даёт три ключевых преимущества — (1) более точная подсветка синтаксиса и сворачивание кода в IDE; (2) значительно повышенная точность генерации и модификации кода ИИ; (3) новички могут быстро понять структуру кода.

### 2.3 Принцип третий: Zero-cost abstraction

Высокоуровневые абстракции не должны приводить к накладным расходам runtime.

**Конкретные гарантии**:

- **Monomorphization**: generic функции во время компиляции展开为конкретные версии, без накладных расходов на поиск в виртуальной таблице
- **Inline оптимизация**: простые функции автоматически встраиваются, устраняя накладные расходы на вызов функций
- **Приоритет распределения в стеке**: небольшие объекты по умолчанию размещаются в стеке, куча используется только при необходимости
- **Без GC**: модель ownership гарантирует безопасность памяти без накладных расходов runtime от сборщика мусора

**Бескомпромиссная причина**: Производительность — это вопрос生存能力языка программирования. Любой дизайн, жертвующий производительностью ради удобства, — это предательство программистов.

### 2.4 Принцип четвёртый: По умолчанию иммутабельность

Изменчивость неразрывно связана со сложностью. YaoXiang выбирает иммутабельность по умолчанию, делая код более понятным.

**Конкретные правила**:

- Переменные по умолчанию иммутабельны, после присваивания их нельзя изменять
- Для изменчивости необходимо явно объявить `mut`
- Ссылки по умолчанию иммутабельны, изменчивые ссылки требуют маркера `mut`
- Transfer ownership означает, что исходная привязка прекращает существование

**Бескомпромиссная причина**: Иммутабельность — основа безопасности并发安全ности, гарантия читаемости кода, плод мудрости функционального программирования.

### 2.5 Принцип пятый: Тип как данные

Информация о типах не должна существовать только во время компиляции, а должна быть полностью доступна в runtime.

**Конкретные возможности**:

- Запрос типа в runtime: любой значение может получить свою информацию о типе
- Рефлексия типов: можно конструировать и манипулировать самими типами
- Деструктуризация через pattern matching: конструкторы типов можно напрямую использовать в pattern matching
- Специализация generics: в runtime можно получить конкретизированные типы generic параметров

**Бескомпромиссная причина**: Полная возможность рефлексии типов — основа метапрограммирования, краеугольный камень высокопроизводительных фреймворков и инструментов.

## 三、Ключевые инновации и特性

YaoXiang, впитывая лучшие特性существующих языков, предлагает следующие инновационные решения.

### 3.1 Инновация первая: Унифицированный синтаксис типов

**Традиционные языки** часто требуют нескольких ключевых слов для определения типов:

```rust
// Rust
struct Point { x: f64, y: f64 }
enum Result<T, E> { Ok(T), Err(E) }
enum Color { Red, Green, Blue }
union IntOrFloat { i: i32, f: f32 }
```

**Унифицированный синтаксис YaoXiang**: только ключевое слово `type`, `{}` для определения структур данных, `[]` для определения interface type.

```yaoxiang
# === Типы данных (фигурные скобки) ===

# Структура
type Point = { x: Float, y: Float }

# Перечисление (несколько конструкторов)
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# Конструктор без параметров
type Color = { red | green | blue }

# Смешанный тип
type Shape = { circle(Float) | rect(Float, Float) }

# === Interface type (квадратные скобки) ===

# Определение интерфейса: набор сигнатур методов
type Serializable = [
    serialize() -> String
]

type Drawable = [
    draw(Surface) -> Void,
    bounding_box() -> Rect
]

# === Generics ===

Option: (T: Type) -> Type = { some(T) | none }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }
```

**Инновационная ценность**: Унифицированный синтаксис типов, устранение фрагментации ключевых слов `enum`/`struct`/`union`/`trait`.

### 3.2 Инновация вторая: Конструкторы как типы

**Конструкция значений и pattern matching полностью идентичны**:

```yaoxiang
# Определение типа
type Point = { x: Float, y: Float }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

# Конструкция значений: аналогично вызову функции
p: Point = Point(3.0, 4.0)
r: Result[Int, String] = ok(42)
err_msg: Result[Int, String] = err("not found")
c: Color = green

# Pattern matching: прямая деструктуризация
match point {
    Point(0.0, 0.0) -> "origin"
    Point(x, y) -> "point at ({x}, {y})"
}
```

### 3.3 Инновация третья: Каррированное связывание методов

YaoXiang использует чистый функциональный дизайн, реализуя синтаксический сахар вызова методов через каррирование без введения ключевых слов `class` и `method`.

```yaoxiang
# === Point.yx ===

# Определение типа
type Point = { x: Float, y: Float }

# Основная функция: евклидово расстояние
distance(Point, Point) -> Float = (a, b) => {
    dx = a.x - b.x
    dy = a.y - b.y
    (dx * dx + dy * dy).sqrt()
}

# Связывание синтаксиса метода
Point.distance = distance  # Привязка по умолчанию к позиции 0
```

```yaoxiang
# === main.yx ===

use Point

main() -> Void = () => {
    p1 = Point(3.0, 4.0)
    p2 = Point(1.0, 2.0)

    # Два способа вызова полностью эквивалентны
    d1 = distance(p1, p2)      # Прямой вызов основной функции
    d2 = p1.distance(p2)       # Синтаксис метода

    # Функциональное использование: предварительная привязка первого параметра
    dist_from_origin = Point.distance(Point(0.0, 0.0))
    result = dist_from_origin(p1)   # 5.0

    # Каррирование: отложенное вычисление
    get_dist_to_p2 = p2.distance(_)
    d3 = get_dist_to_p2(p1)         # 2.828
}
```

**Инновационная ценность**: Чистый функциональный дизайн без скрытого параметра `self`, функции как значения могут свободно передаваться и комбинироваться.

### 3.4 Инновация четвёртая: Модель spawn

> «Все сущее действует сообща, я наблюдаю их возвращение.» — «И.Цзин.Гексаграмма Фу»
>
> Модель spawn заимствует эту идею и описывает парадигму программирования: разработчик описывает логику синхронным, последовательным мышлением, а runtime языка автоматически, эффективно выполняет вычислительные единицы параллельно, как все сущее действует сообща, и в итоге объединяет их координацию.

**Три основных принципа**:

| Принцип | Описание |
|---------|----------|
| **Синхронный синтаксис** | То, что видишь, то и получаешь — последовательный код |
| **Параллельная сущность** | Runtime автоматически извлекает параллелизм |
| **Единая координация** | Результаты автоматически собираются при необходимости, гарантируя логическую корректность |

**Терминология**:

| Официальный термин | Соответствующий синтаксис | Объяснение |
|--------------------|---------------------------|------------|
| **Spawn функция** | `spawn (params) => body` | Определение вычислительной единицы, участвующей в spawn выполнении |
| **Spawn блок** | `spawn { a(), b() }` | Явно объявленная параллельная область, задачи внутри блока выполняются параллельно |
| **Spawn цикл** | `spawn for x in xs { ... }` | Data parallelism, тело цикла выполняется параллельно для всех элементов |
| **Spawn значение** | `Async(T)` | Значение в процессе spawn, автоматически ожидает при использовании |
| **Spawn граф** | Lazy вычислительный граф (DAG) | Сцена, где происходит spawn, описывает зависимости и отношения параллелизма |
| **Spawn планировщик** | Runtime планировщик задач | Координирует всё сущее, позволяя ему действовать параллельно в правильный момент |

> **Подробнее**: [RFC-001 Модель spawn](./rfc/001-concurrent-model-error-handling.md)

```yaoxiang
# === Spawn функция ===
# Функция с маркером spawn (синтаксис RFC-003)
fetch_data: String -> JSON = spawn (url) => {
    HTTP.get(url).json()
}

# === Spawn блок ===
# Выражения внутри spawn { } принудительно выполняются параллельно
compute_all: () -> (Int, Int, Int) = spawn () => {
    (a, b, c) = spawn {
        heavy_calc(1),    # Задача 1
        heavy_calc(2),    # Задача 2
        another_calc(3)   # Задача 3
    }
    (a, b, c)
}

# === Spawn цикл ===
# Цикл с маркером spawn for автоматически параллелизуется
parallel_sum: Int -> Int = spawn (n) => {
    total = spawn for i in 0..n {
        fibonacci(i)  # Каждая итерация выполняется параллельно
    }
    total
}

# === Автоматическое ожидание ===
main() -> Void = () => {
    # Два независимых запроса автоматически выполняются параллельно
    users = fetch_data("https://api.example.com/users")
    posts = fetch_data("https://api.example.com/posts")

    # Точка ожидания автоматически вставляется при необходимости результата
    print(users.length + posts.length)  # Автоматическое ожидание users и posts
}
```

**Потокобезопасность**:

```yaoxiang
# Ограничения Send/Sync гарантируют compile-time потокобезопасность
type SafeCounter = SafeCounter(mutex: Mutex(Int))

main: () -> Void = () => {
    counter: Arc[SafeCounter] = Arc.new(SafeCounter(Mutex.new(0)))

    # spawn автоматически проверяет ограничение Send
    spawn () => {
        guard = counter.mutex.lock()
        guard.value = guard.value + 1
    }
}
```

**Техническая документация**:
- Подробнее в [RFC-001 Модель spawn](./rfc/001-concurrent-model-error-handling.md)

**Инновационная ценность**: Когнитивная нагрузка асинхронного программирования снижается до нуля, читаемость кода полностью идентична синхронному коду, при этом достигается высокопроизводительное параллельное выполнение.

### 3.5 Инновация пятая: Поддержка зависимых типов (будущая特性)

> **Статус**: Рассматривается реализация после v1.0

Типы могут зависеть от значений, реализуя настоящую разработку через типы.

```yaoxiang
# Вектор фиксированной длины (будущий синтаксис)
Vector: (T: Type, n: Int) -> Type = {
    data: [T; n]
    length: n
}

# Использование
vec: Vector[Int, 3] = Vector([1, 2, 3], 3)

# Проверка типов
# vec: Vector[Int, 3] = Vector([1, 2], 2)  # Ошибка компиляции! Длина не совпадает
```

**Инновационная ценность**: Больше ошибок обнаруживается во время компиляции, достигаются более точные гарантии типов.

### 3.6 Инновация шестая: Минималистичный дизайн ключевых слов

YaoXiang определяет только 18 основных ключевых слов, что значительно меньше, чем в основных языках:

```
type   pub    use    spawn
ref    mut    if     elif
else   match  while  for    return
break  continue as     in     unsafe
```

| Сравнение языков | Количество ключевых слов |
|------------------|---------------------------|
| YaoXiang | **18** |
| Rust | 51+ |
| Python | 35 |
| TypeScript | 64+ |
| Go | 25 |

**Инновационная ценность**: Меньшая нагрузка на память, более последовательный стиль синтаксиса, более легко解析мая синтаксическая структура.

### 4.2 Определение типов и функции

```yaoxiang
# Унифицированный синтаксис типов
type Point = { x: Float, y: Float }
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }
```

### 4.3 Pattern matching

```yaoxiang
# Мощный pattern matching
classify(Int) -> String = (n) => {
    match n {
        0 -> "zero"
        1 -> "one"
        _ if n < 0 -> "negative"
        _ -> "positive"
    }
}

# Деструктуризация
type Point = { x: Float, y: Float }
match point {
    Point(0.0, 0.0) -> "origin"
    Point(x, y) -> "point at ({x}, {y})"
}
```

```yaoxiang
# List comprehension в стиле Python
evens = [x * 2 for x in 0..10]          # [0, 4, 8, 12, 16]

# Вложенные comprehension
matrix = [[i * j for j in 1..4] for i in 1..3]
# [[1, 2, 3], [2, 4, 6], [3, 6, 9]]
```

### 4.5 Модель ownership

```yaoxiang
type Point = { x: Float, y: Float }

# По умолчанию Move (нулевое копирование)
p = Point(1.0, 2.0)
p2 = p              # Move, передача ownership, p становится недействительным

# Явный ref = Arc (безопасное разделение)
shared = ref p      # Arc, потокобезопасный

spawn(() => print(shared.x))   # ✅ Безопасно

# Явный clone() = копирование
p3 = p.clone()      # p и p3 независимы
```

**Основные правила**:
- По умолчанию Move (нулевое копирование)
- Разделение через `ref` (Arc)
- Копия через `clone()`
- Системный уровень через `unsafe` + `*T`

### 4.6 Обработка ошибок

```yaoxiang
# Тип Result
Result: (T: Type, E: Type) -> Type = { ok(T) | err(E) }

divide: (Float, Float) -> Result[Float, String] = (a, b) => {
    if b == 0.0 {
        err("Division by zero")
    } else {
        ok(a / b)
    }
}

# Использование оператора ?
process: () -> Result[Int, String] = () => {
    a = read_number()?
    b = read_number()?
    c = divide(a, b)?
    ok(c * 2)
}
```

### 4.7 Concurrent programming (SeamlessAsync)

```yaoxiang
# spawn маркирует асинхронную функцию
fetch_api: String -> JSON spawn = (url) => {
    response = HTTP.get(url)
    JSON.parse(response.body)
}

# Параллельный блок: явный параллелизм
process_all: () -> (JSON, JSON, JSON) spawn = () => {
    (a, b, c) = spawn {
        fetch_api("https://api1.com/data"),
        fetch_api("https://api2.com/data"),
        fetch_api("https://api3.com/data")
    }
    (a, b, c)
}

# Параллельный цикл данных
parallel_process: Int -> Int spawn = (n) => {
    total = spawn for i in 0..n {
        compute(i)
    }
    total
}

# Пример потокобезопасности
type ThreadSafeCounter = { value: Mutex(Int) }

main: () -> Void = () => {
    counter = ThreadSafeCounter(Mutex.new(0))

    # spawn автоматически проверяет ограничение Send
    spawn () => {
        guard = counter.value.lock()
        guard.value = guard.value + 1
    }
}
```

### 5.1 Принятые проектные решения

Следующие решения прошли достаточное обсуждение и проверку, **изменения больше не принимаются**:

| Модуль | Решение | Описание |
|--------|---------|----------|
| **Type system** | Всё есть тип | Значения, функции, модули, generics — всё это типы |
| **Синтаксис типов** | Унифицированный синтаксис с фигурными скобками | `{}` для типов данных, `[]` для interface type |

| **Файл как модуль** | Система модулей | Каждый файл `.yx` — это модуль |
| **Главная функция** | `main: () -> Void` | Точка входа программы |

4. **Найти задачи**: Посмотреть метку `good first issue` в GitHub Issues
5. **Отправить PR**: Следовать стандартам коммитов, написать тесты
6. **Участвовать в рецензировании**: Рецензировать чужой код, участвовать в дискуссиях

**Стандарты коммитов**:

```bash
# Формат сообщения коммита
<type>(<scope>): <subject>

# Типы
feat: новая特性
fix: исправление бага
docs: обновление документации
style: форматирование кода (без влияния на функциональность)
refactor: рефакторинг
perf: оптимизация производительности
test: тесты
chore: инструменты сборки или вспомогательные инструменты

# Примеры
feat(typecheck): add generic type inference
fix(parser): fix infinite loop on invalid input
docs(readme): update installation instructions
```

**Стиль кода**:

- Следовать спецификации `rustfmt.toml`
- Убедиться, что `cargo clippy` не выдаёт предупреждений
- Писать необходимые unit тесты
- Обновлять связанную документацию

## Приложение C: Часто задаваемые вопросы

**Q: Какие преимущества YaoXiang перед Rust?**

A: YaoXiang сохраняет безопасность памяти и zero-cost abstraction Rust, но использует более простой синтаксис и меньшую когнитивную нагрузку. **Модель spawn** проще, чем `async/await` в Rust — достаточно одного маркера `spawn`, не нужно вручную управлять Future и Pin. «Всё сущее действует сообща» — concurrent programming становится таким же интуитивным, как описание законов природы. Ограничения Send/Sync обеспечивают同等потокобезопасность. Унифицированный синтаксис типов устраняет концептуальную фрагментацию `enum`/`struct`/`union`.

**Q: Для чего подходит YaoXiang?**

A: Системное программирование, прикладная разработка, веб-сервисы, скриптовые инструменты, AI-assisted программирование. Цель — стать универсальным языком программирования.

**Q: Почему выбран 4-пробельный отступ?**

A: 4 пробела обеспечивают чёткое визуальное разделение блоков кода, уменьшают путаницу от вложенности. Это тщательно продуманное решение для «дружелюбия к ИИ».

**Q: Когда выйдет версия 1.0?**

A: Цель v1.0: production ready. Дата выхода зависит от прогресса реализации, подробнее в [RFC планирования версий](./rfc/003-version-planning.md).

**Q: Как связаться с основной командой?**

A: Через GitHub Discussions или канал сообщества Discord. Основные члены команды регулярно отвечают.
