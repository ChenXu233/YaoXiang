## I. Зачем создавать YaoXiang?

### 1.1 Заполнение пробела в языках

На протяжении истории языков программирования мы стали свидетелями рождения и эволюции бесчисленных выдающихся языков: язык C произвёл революцию эффективности в системном программировании, Python создал опыт программирования, доступный каждому, Rust доказал, что безопасность памяти и производительность могут сосуществовать, TypeScript сделал крупные фронтенд-проекты поддерживаемыми. Однако, рассматривая современную экосистему языков, мы по-прежнему обнаруживаем очевидный разрыв — **ни один язык не способен одновременно удовлетворить следующие три ключевые потребности**:

| Потребность | Проблемы существующих решений |
|------|-------------------|
| **Безопасность типов** | Rust слишком строг, крутая кривая обучения; TypeScript использует опциональные типы и не может гарантировать проверку на этапе компиляции |
| **Естественный синтаксис** | Синтаксис Rust сложен и непонятен; функциональный порог Haskell слишком высок; традиционные статические языки многословны и громоздки |
| **AI-дружественность** | Существующие языки имеют неоднозначный синтаксис, сложный AST и непредсказуемое скрытое поведение, что ограничивает точность генерации и модификации кода ИИ |

YaoXiang рождён именно для того, чтобы заполнить этот пробел. Мы верим: **язык программирования должен быть одновременно мощным и доступным, безопасным и эффективным, строгим и элегантным**.

### 1.2 Решаемые практические проблемы

**Проблема первая: фрагментация систем типов**

Современные языки программирования демонстрируют серьёзную фрагментацию в системах типов. Статически типизированные языки стремятся к абсолютной корректности на этапе компиляции, но часто жертвуют эффективностью разработки; динамически типизированные языки обеспечивают гибкость, но обнаруживают трудно поддерживаемые дефекты в крупных проектах. YaoXiang предлагает унифицированную абстрактную структуру «всё есть тип» (everything is a type), делающую типы основной линией проектирования языка, а не запоздалой заплаткой.

**Проблема вторая: дилемма между безопасностью памяти и производительностью**

Долгое время разработчики были вынуждены делать трудный выбор между безопасностью памяти и производительностью. GC (сборка мусора), хотя и освобождает разработчика, привносит колебания задержек и накладные расходы памяти; ручное управление памятью, хотя и эффективно, опасно как хождение по канату. YaoXiang использует модель владения в стиле Rust, устраняя гонки данных и утечки памяти на этапе компиляции, сохраняя при этом абстракции с нулевой стоимостью, и обеспечивает высокую производительность без GC.

**Проблема третья: когнитивная нагрузка асинхронного программирования**

Современные приложения не могут обойтись без сетей и параллелизма, а асинхронное программирование всегда было кошмаром программистов. Вложенные callback-функции, цепочки Promise, синтаксис async/await — каждое решение увеличивает сложность кода. YaoXiang переработал модель асинхронности: достаточно добавить маркер `spawn` после сигнатуры функции, и компилятор автоматически обработает все асинхронные детали, делая параллельное программирование столь же естественным, как синхронный код.

**Проблема четвёртая: узкие места ИИ-ассистированного программирования**

Когда ИИ начинает помогать разработчикам писать код, выбор дизайна языка становится критически важным. Размытые синтаксические правила, неявные преобразования типов, сложный синтаксический сахар — эти особенности, к которым привыкли люди-программисты, становятся препятствием для понимания и генерации кода ИИ. YaoXiang с самого начала ставит «AI-дружественность» в качестве ключевой цели: строгие правила отступов, явные границы блоков кода, недвусмысленные синтаксические структуры позволяют ИИ точно понимать, генерировать и модифицировать код.

### 1.3 Философские корни языка

Название YaoXiang происходит от «яо» (爻) и «сян» (象) в «И цзин». «Яо» — базовые символы, составляющие триграммы и гексаграммы, символизирующие изменения инь и ян, взаимное порождение движения и покоя; «сян» — внешние проявления сути вещей, олицетворяющие все явления и все существа.

Эта философская идея воплощена в каждой детали дизайна языка:

- **Единство**: подобно тому, как простые символы яо образуют сложные гексаграммы, YaoXiang использует несколько ключевых концепций (типы, функции, конструкторы) для построения полной модели программирования
- **Иерархичность**: подобно тому, как у сян есть различие между врождённым и приобретённым, система типов YaoXiang обладает ясной иерархической структурой — от primitive type к generics, от значений к meta type
- **Изменчивость**: подобно непрерывному круговороту инь и ян, бесконечности перемен, YaoXiang поддерживает зависимые типы (dependent types), позволяя типам эволюционировать вместе со значениями
- **Идентифицируемость**: подобно тому, как гексаграммы можно интерпретировать, а все вещи можно представить, YaoXiang обеспечивает полную способность отражения типов (type reflection), делая информацию о типах полностью доступной во время выполнения
- **Доказуемость**: подобно тому, как гексаграммы раскрывают законы вещей, система типов YaoXiang следует изоморфизму Карри-Ховарда (тип есть утверждение, программа есть доказательство); процесс проверки типов есть проверка логического доказательства

## II. Ключевая философия и принципы

Следующие проектные заповеди — краеугольный камень YaoXiang, **не подлежащий обсуждению и нарушению**. Любое предложение по функциональности должно пройти проверку этими принципами.

### 2.1 Принцип первый: всё есть тип

В мировоззрении YaoXiang тип — высшая абстрактная единица, ключевая концепция, пронизывающая язык.

**Конкретные проявления**:

- **Значения суть экземпляры типов**: `42` — экземпляр типа `Int`, `"hello"` — экземпляр типа `String`
- **Сами типы также являются типами**: `Type` — единственное ключевое слово meta type в языке; тип `Int` есть `Type`
- **Функции суть отображения типов**: `add: (a: Int, b: Int) -> Int` описывает отображение из `Int × Int` в `Int`
- **Модули суть комбинации типов**: модуль — это комбинация пространств имён, содержащих функции и типы

**Причина непреклонности**: унифицированная абстракция типов упрощает семантику языка, устраняет дуализм значения и типа, превращая систему типов в хранителя корректности кода, а не в препятствие.

### 2.2 Принцип второй: строгая структурированность

Синтаксис YaoXiang стремится к «однозначности, предсказуемости, лёгкости разбора».

**Конкретные правила**:

- **Обязательный отступ в 4 пробела**: символы табуляции запрещены, границы блоков кода очевидны
- **Скобки не могут быть опущены**: параметры функций должны быть в скобках, элементы списка должны разделяться запятыми
- **Блоки кода должны быть в фигурных скобках**: `if`, `while`, `for` и другие управляющие конструкции должны быть обёрнуты в `{ }`
- **Минимальное количество ключевых слов**: только 17 ключевых слов ядра, отказ от разрастания синтаксического сахара

**Причина непреклонности**: строгая структурированность даёт три ключевых преимущества — (1) более точная подсветка синтаксиса и свёртывание кода в IDE; (2) значительное повышение точности генерации и модификации кода ИИ; (3) новички могут быстро понять структуру кода.

### 2.3 Принцип третий: абстракции с нулевой стоимостью

Высокоуровневые абстракции не должны привносить накладных расходов во время выполнения.

**Конкретные гарантии**:

- **Мономорфизация**: обобщённые функции разворачиваются в конкретные версии на этапе компиляции, без затрат на поиск в виртуальных таблицах
- **Оптимизация встраивания (inlining)**: простые функции автоматически встраиваются, устраняя затраты на вызовы функций
- **Приоритет стекового размещения**: мелкие объекты по умолчанию размещаются в стеке, куча используется только при необходимости
- **Без GC**: модель владения гарантирует безопасность памяти без затрат на сборщик мусора во время выполнения

**Причина непреклонности**: производительность — это底线 (красная линия) выживания языка программирования. Любой дизайн, жертвующий производительностью ради удобства, — предательство по отношению к программистам.

### 2.4 Принцип четвёртый: неизменяемость по умолчанию

Изменяемость неразрывно связана со сложностью. YaoXiang выбирает неизменяемость по умолчанию, делая код более лёгким для рассуждения и понимания.

**Конкретные правила**:

- Переменные по умолчанию неизменяемы, после присваивания не могут быть изменены
- При необходимости в изменяемости требуется явное объявление `mut`
- Ссылки по умолчанию неизменяемы, изменяемые ссылки требуют маркировки `mut`
- Передача ownership означает, что исходная привязка становится недействительной

**Причина непреклонности**: неизменяемость — основа безопасности параллелизма, гарантия читаемости кода, кристаллизация мудрости функционального программирования.

### 2.5 Принцип пятый: типы суть данные

Информация о типах не должна существовать только на этапе компиляции, она должна быть полностью доступна во время выполнения.

**Конкретные возможности**:

- Запрос типа во время выполнения: для любого значения можно получить информацию о его типе
- Отражение типов (type reflection): можно конструировать и оперировать самими типами
- Деконструкция через pattern matching: конструкторы типов могут непосредственно использоваться в pattern matching
- Специализация generics: во время выполнения можно получить конкретизированные типы параметров generics

**Причина непреклонности**: полная способность отражения типов — это основа метапрограммирования, краеугольный камень высокопроизводительных фреймворков и инструментов.

## III. Ключевые инновации и особенности

YaoXiang, впитывая лучшие черты существующих языков, предлагает следующие инновационные дизайнерские решения.

### 3.1 Инновация первая: унифицированный синтаксис типов

**Традиционные языки** часто требуют нескольких ключевых слов для определения типов:

```rust
// Rust
struct Point { x: f64, y: f64 }
enum Result<T, E> { Ok(T), Err(E) }
enum Color { Red, Green, Blue }
trait Drawable { fn draw(&self, s: &Surface); }
```

**Унифицированный синтаксис YaoXiang**: всё есть `name: type = value`, `Type` — единственное ключевое слово meta type.

```yaoxiang
# === Record type ===

Point: Type = {
    x: Float,
    y: Float,
}

# Поля со значениями по умолчанию
Point3D: Type = {
    x: Float = 0,
    y: Float = 0,
    z: Float = 0,
}

# === Generic type ===

Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self,
}

Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,
}

# === Interface type (запись, все поля которой — типы функций) ===

Drawable: Type = {
    draw: (Surface) -> Void,
    bounding_box: () -> Rect,
}

Serializable: Type = {
    serialize: () -> String,
}

# === Реализация интерфейса (имя интерфейса записывается в теле типа) ===

Point: Type = {
    x: Float,
    y: Float,
    Drawable,
    Serializable,
}

# === Метод (синтаксис Type.method) ===

Point.draw: (self: &Point, surface: Surface) -> Void = {
    surface.plot(self.x, self.y)
}
```

**Ценность инновации**: нет фрагментации ключевых слов `fn`, `struct`, `enum`, `trait`, `impl` — один унифицированный синтаксис покрывает все объявления.

### 3.2 Инновация вторая: конструктор как тип

**Конструирование значения полностью идентично вызову функции**:

```yaoxiang
# Определение типа
Point: Type = { x: Float, y: Float }
Option: (T: Type) -> Type = {
    some: (T) -> Self,
    none: () -> Self,
}

# Конструирование значения: то же, что вызов функции
p: Point = Point(3.0, 4.0)
opt: Option(Int) = Option.some(42)
none: Option(Int) = Option.none()

# Pattern matching: прямая деконструкция
match opt {
    Option.some(value) -> print(value)
    Option.none -> print("nothing")
}
```

### 3.3 Инновация третья: каррирование для привязки методов

YaoXiang использует чисто функциональный дизайн, реализуя синтаксический сахар для вызова методов объектов через каррирование, без введения ключевых слов `class` и `method`.

```yaoxiang
# === Определение типа ===

Point: Type = {
    x: Float,
    y: Float,
}

# Основная функция: евклидово расстояние
distance: (a: Point, b: Point) -> Float = {
    dx = a.x - b.x
    dy = a.y - b.y
    return (dx * dx + dy * dy).sqrt()
}

# Привязка метода через синтаксический сахар ([0] означает привязку к позиции 0-го параметра)
Point.distance = distance[0]

# === Использование ===

p1 = Point(3.0, 4.0)
p2 = Point(1.0, 2.0)

# Оба способа вызова полностью эквивалентны
d1 = distance(p1, p2)     # Прямой вызов основной функции
d2 = p1.distance(p2)      # Синтаксический сахар метода

# Использование каррирования
dist_from_p1 = p1.distance  # Частичное применение, ожидает второй параметр
d3 = dist_from_p1(p2)       # 2.828
```

**Ценность инновации**: чистый функциональный дизайн, без скрытого параметра `self`, функции — это значения, которые можно свободно передавать и комбинировать.

### 3.4 Инновация четвёртая: модель spawn (并作)

> «Все существа действуют совместно, и я наблюдаю их возвращение.»
> — «И цзин», гексаграмма «Фу» (Возврат)
>
> Модель spawn (并作) берёт начало в этом, описывая парадигму программирования: разработчик описывает логику синхронно, последовательно, а среда выполнения языка заставляет вычислительные единицы действовать совместно, подобно всем существам, автоматически и эффективно выполняясь параллельно, и в конечном счёте объединяясь согласованно.

**Три ключевые принципы**:

| Принцип | Описание |
|------|------|
| **Синхронный синтаксис** | Код «что видишь, то и получаешь», последовательный |
| **Параллельная сущность** | Среда выполнения автоматически извлекает параллелизм |
| **Единая координация** | Результаты автоматически собираются при необходимости, гарантируя логическую корректность |

**Терминология**:

| Официальный термин | Соответствующий синтаксис | Пояснение |
|----------|----------|------|
| **spawn function** | `spawn (params) => body` | Определяет вычислительную единицу, способную участвовать в параллельном выполнении |
| **spawn block** | `spawn { a(), b() }` | Явно объявленная область параллелизма, задачи внутри блока выполняются параллельно |
| **spawn loop** | `spawn for x in xs { ... }` | Параллелизм данных, тело цикла выполняется параллельно на всех элементах |
| **spawn value** | `Async(T)` | Будущее значение, находящееся в процессе параллельного выполнения, ожидание происходит автоматически при использовании |
| **spawn graph** | Ленивый вычислительный граф (DAG) | Сцена параллельного выполнения, описывающая зависимости и параллельные отношения |
| **spawn scheduler** | Планировщик задач среды выполнения | Интеллектуальный центр, координирующий все существа, чтобы они действовали совместно в правильный момент |

> **Подробнее**: [RFC-001 Модель spawn](./rfc/001-concurrent-model-error-handling.md)

```yaoxiang
# === spawn function ===
# Функция, помеченная spawn
fetch_data: (url: String) -> JSON spawn = {
    return HTTP.get(url).json()
}

# === spawn block ===
# Выражения внутри spawn { } принудительно выполняются параллельно
compute_all: () -> (Int, Int, Int) spawn = {
    (a, b, c) = spawn {
        heavy_calc(1),    # Задача 1
        heavy_calc(2),    # Задача 2
        another_calc(3)   # Задача 3
    }
    return (a, b, c)
}

# === Автоматическое ожидание ===
main: () -> Void = {
    # Два независимых запроса автоматически выполняются параллельно
    users = fetch_data("https://api.example.com/users")
    posts = fetch_data("https://api.example.com/posts")

    # Точки ожидания автоматически вставляются при необходимости результата
    print(users.length + posts.length)  # Автоматическое ожидание users и posts
}
```

**Потокобезопасность**:

```yaoxiang
# Ключевое слово ref автоматически обрабатывает потокобезопасность (компилятор автоматически выбирает Rc/Arc)
main: () -> Void = {
    counter = ref SafeCounter(0)

    # Разделяемое владение между задачами: компилятор автоматически выбирает Arc
    spawn {
        counter.increment()
    }
    spawn {
        counter.increment()
    }
}
```

**Техническая документация**:
- Подробности см. в [RFC-001 Модель spawn](./rfc/accepted/001-concurrent-model-error-handling.md)

**Ценность инновации**: когнитивная нагрузка асинхронного программирования сводится к нулю, читаемость кода полностью идентична синхронному коду, при этом достигается высокопроизводительное параллельное выполнение.

### 3.5 Инновация пятая: зависимые от значений типы (RFC-011)

> **Статус**: в разработке, частично реализовано

Типы могут зависеть от значений, реализуя подлинное type-driven development.

```yaoxiang
# Тип матрицы: размеры определяются на этапе компиляции
Matrix: (T: Type, Rows: Int, Cols: Int) -> Type = {
    data: Array(Array(T, Cols), Rows),
}

# Вычисление на этапе компиляции: factorial(3) = 6
vec: Vec(factorial(3)) = Vec(6)()

# Проверка размеров на этапе компиляции
identity_3x3: Matrix(Float, 3, 3) = identity(Float, 3)(3)
# multiply(matrix_2x3, matrix_4x2)  # Ошибка компиляции: несоответствие размеров
```

**Ценность инновации**: перехват большего числа ошибок на этапе компиляции, более точные гарантии типов.

### 3.6 Инновация шестая: минималистичный дизайн ключевых слов

YaoXiang определяет только 17 ключевых слов ядра, что значительно меньше, чем в mainstream языках:

```
pub    use    spawn
ref    mut    if     elif
else   match  while  for    return
break  continue as     in     unsafe
```

| Сравниваемый язык | Количество ключевых слов |
|----------|-----------|
| YaoXiang | **17** |
| Rust | 51+ |
| Python | 35 |
| TypeScript | 64+ |
| Go | 25 |

**Ценность инновации**: меньшая нагрузка на запоминание, более согласованный стиль синтаксиса, более лёгкая для разбора синтаксическая структура.

### 4.4 Модель владения (RFC-009 v9)

```yaoxiang
Point: Type = { x: Float, y: Float }

# По умолчанию Move (без копирования)
p1 = Point(1.0, 2.0)
p2 = p1              # Move, p1 больше нельзя читать

# &T / &mut T токены (нулевые затраты на этапе компиляции)
p2.print()           # Компилятор автоматически создаёт токен &Point
p2.shift(1.0, 1.0)  # Компилятор автоматически создаёт токен &mut Point

# ref: разделяемое владение (компилятор автоматически выбирает Rc/Arc)
shared = ref p2      # Разделение между областями видимости

# clone(): явное глубокое копирование
backup = p2.clone()

# unsafe + сырые указатели: системный уровень
unsafe {
    ptr: *Point = &p2
    (*ptr).x = 0.0
}
```

**Градиент владения**:
```
&T / &mut T    Move       ref        clone()    unsafe
    |             |          |           |          |
Токен заимствования  По умолчанию  Разделяемое  Глубокое  Сырой указатель
Нулевая стоимость    Без копий   Авто Rc/Arc   Явное     Системный уровень
```

### 4.5 Обработка ошибок

```yaoxiang
# Тип Result
Result: (T: Type, E: Type) -> Type = {
    ok: (T) -> Self,
    err: (E) -> Self,
}

divide: (a: Float, b: Float) -> Result(Float, String) = {
    if b == 0.0 {
        return Result.err("Division by zero")
    }
    return Result.ok(a / b)
}

# Использование match для обработки
result = divide(10.0, 2.0)
match result {
    Result.ok(value) -> print(value),
    Result.err(msg) -> print("Error: ${msg}"),
}
```

### 4.6 Параллельное программирование (модель spawn)

```yaoxiang
# spawn маркирует асинхронную функцию
fetch_api: (url: String) -> JSON spawn = {
    response = HTTP.get(url)
    return JSON.parse(response.body)
}

# spawn block: явный параллелизм
process_all: () -> (JSON, JSON, JSON) spawn = {
    (a, b, c) = spawn {
        fetch_api("https://api1.com/data"),
        fetch_api("https://api2.com/data"),
        fetch_api("https://api3.com/data")
    }
    return (a, b, c)
}
```

## V. Дорожная карта и открытые вопросы

### 5.1 Принятые дизайнерские решения

Следующие решения были всесторонне обсуждены и рассмотрены, **изменения больше не принимаются**:

| Модуль | Решение | Пояснение |
|------|------|------|
| **Система типов** | Всё есть тип | Значения, функции, модули, generics — всё суть типы |
| **Синтаксис типов** | Унифицированный `name: type = value` | Одна форма объявления покрывает все случаи, `Type` — единственное ключевое слово meta type |
| **Ключевые слова** | 17 ключевых слов ядра | Не включает `type`/`fn`/`struct`/`enum`/`trait`/`impl` |
| **Синтаксис функций** | Сигнатура + выражение | `name: (params) -> ReturnType = body` |
| **Привязка методов** | Каррирование (RFC-004) | `Type.method = function[position]` |
| **Модель асинхронности** | Модель spawn (并作) | Маркер `spawn`, ленивое вычисление, автоматический параллелизм |
| **Управление памятью** | Модель владения (RFC-009 v9) | Move + токены &T/&mut T + ref + clone + unsafe, без GC |
| **Файл как модуль** | Модульная система | Каждый файл `.yx` — модуль |
| **Главная функция** | `main: () -> Void` | Точка входа в программу |
| **Потокобезопасность** | ref автоматически выбирает Rc/Arc | Escape-анализ компилятора, невидим для пользователя |

## Приложение C: Часто задаваемые вопросы

**В: Какие преимущества YaoXiang имеет по сравнению с Rust?**

О: YaoXiang сохраняет безопасность памяти и абстракции с нулевой стоимостью Rust, но использует более простой синтаксис и меньшую когнитивную нагрузку. **Модель spawn** (并作) проще, чем `async/await` в Rust — нужен лишь один маркер `spawn`, без ручного управления Future и Pin. «Все существа действуют совместно, и я наблюдаю их возвращение» — параллельное программирование становится столь же интуитивным, как описание законов природы. **Модель владения** (RFC-009 v9) заменяет аннотации времени жизни на Move + токены &T/&mut T, заменяет проверяющий заимствование на атрибуты типов (Dup/Linear). Унифицированный синтаксис типов устраняет фрагментацию концепций `enum`/`struct`/`trait`/`impl`.

**В: Для каких типов разработки подходит YaoXiang?**

О: Системное программирование, разработка приложений, веб-сервисы, скриптовые инструменты, ИИ-ассистированное программирование. Цель — стать универсальным языком программирования.

**В: Почему выбран отступ в 4 пробела?**

О: 4 пробела обеспечивают ясное визуальное разделение блоков кода, уменьшают путаницу от глубины вложенности. Это хорошо продуманное дизайнерское решение для «AI-дружественности».

**В: Когда выйдет версия 1.0?**

О: Цель v1.0: готовность к промышленному использованию. Время выпуска зависит от прогресса реализации, подробности см. в [RFC планирования версий](./rfc/003-version-planning.md).

**В: Как связаться с основной командой?**

О: Через GitHub Discussions или канал сообщества Discord. Члены основной команды регулярно отвечают.
